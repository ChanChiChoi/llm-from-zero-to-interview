# 第十章：上线运营与反馈闭环

大模型产品上线不是终点，而是持续迭代的开始。模型会遇到训练时没见过的问题，用户会用意想不到的方式提问，知识库会过期，成本会变化，模型版本会更新，安全风险会暴露。没有运营和反馈闭环的大模型产品，很快会从“看起来智能”变成“不稳定、不可信、没人用”。

本章系统讲上线运营与反馈闭环：灰度发布、质量监控、用户反馈、A/B test、失败样本分析、数据闭环、模型和 prompt 迭代、成本监控、安全监控和持续运营。

## 10.1 上线为什么不是终点

传统软件上线后，主要关注 bug 和性能。大模型产品上线后，还要关注模型行为。

例如：

1. 用户问法是否符合预期。
2. 模型是否幻觉。
3. RAG 是否召回正确文档。
4. Agent 是否误用工具。
5. 成本是否超预算。
6. 拒答是否过多或过少。
7. 用户是否真的采纳输出。

面试回答：

```text
大模型产品上线后必须持续运营，因为模型行为会受到用户输入、知识库更新、模型版本、prompt、工具和业务环境影响。需要监控质量、成本、延迟、安全和用户反馈，把失败案例沉淀为评估集或优化数据，形成持续迭代闭环。
```

## 10.2 灰度发布

大模型产品不应该一次全量上线。

灰度发布可以按：

1. 用户群体。
2. 业务线。
3. 地区。
4. 功能模块。
5. 风险等级。
6. 流量比例。

灰度阶段要观察：

1. 使用率。
2. 成功率。
3. 用户反馈。
4. 错误类型。
5. 成本。
6. 延迟。
7. 安全事件。

灰度不是形式，而是为了在风险可控范围内发现真实问题。

## 10.3 质量监控

质量监控要覆盖模型输出和任务结果。

常见指标：

1. 答案正确率。
2. 用户满意度。
3. 任务完成率。
4. 引用准确率。
5. 无法回答率。
6. 转人工率。
7. 重试率。
8. 重新生成率。
9. 投诉率。
10. 人工审核通过率。

不同产品指标不同。客服看解决率，RAG 看引用准确率，Agent 看任务完成率，代码助手看采纳率和测试通过率。

## 10.4 成本监控

大模型产品成本可能悄悄上涨。

需要监控：

1. 平均 token。
2. 输出长度。
3. 模型调用次数。
4. RAG 检索次数。
5. Agent 步数。
6. 工具调用次数。
7. 重试次数。
8. 单用户成本。
9. 单任务成本。
10. 总成本趋势。

成本异常可能来自 prompt 变长、RAG top-k 过大、Agent 无限循环、用户滥用或模型升级。

## 10.5 延迟监控

延迟影响体验。

要监控：

1. 首 token 延迟。
2. 总生成时间。
3. 检索耗时。
4. 工具调用耗时。
5. 排队时间。
6. 超时率。
7. P50、P95、P99 延迟。

平均延迟不够，长尾延迟更影响用户体验。尤其是企业工作流中，P95 和 P99 很重要。

## 10.6 用户反馈

用户反馈是优化的重要来源。

反馈形式：

1. 点赞点踩。
2. 选择错误类型。
3. 输入期望答案。
4. 标记引用错误。
5. 请求转人工。
6. 手动编辑结果。
7. 投诉和客服记录。

简单点赞点踩信息量有限。更好的反馈要能定位问题：是检索错、理解错、生成错、引用错、权限错，还是用户期望不同。

## 10.7 失败样本分析

失败样本是产品改进的燃料。

失败类型可以分为：

1. 意图理解错误。
2. 检索失败。
3. 引用错误。
4. 生成幻觉。
5. 工具调用失败。
6. 权限问题。
7. 拒答不当。
8. 格式不符合要求。
9. 延迟过高。
10. 用户需求不清。

每类失败对应不同优化手段。不能把所有失败都归因于“模型不够强”。

## 10.8 A/B Test

A/B test 用于比较不同方案。

可以测试：

1. 不同模型。
2. 不同 prompt。
3. 不同 RAG 策略。
4. 不同交互设计。
5. 不同拒答策略。
6. 不同排序或推荐方案。

指标要提前定义。例如测试 RAG 策略时，看引用准确率、自助解决率和延迟，而不是只看主观感觉。

## 10.9 离线评估和在线评估

离线评估：用固定评估集测试。

优点：

1. 可重复。
2. 成本低。
3. 适合回归测试。

缺点：

1. 覆盖不了真实用户长尾。
2. 可能过拟合评估集。

在线评估：看真实用户使用数据。

优点：

1. 贴近真实场景。
2. 能发现新问题。

缺点：

1. 风险更高。
2. 归因复杂。
3. 需要灰度和监控。

成熟产品需要离线和在线结合。

## 10.10 反馈到数据闭环

用户反馈要进入数据闭环。

流程：

```text
线上日志 -> 失败样本 -> 人工标注 -> 评估集更新 -> prompt/RAG/模型优化 -> 回归测试 -> 灰度上线
```

反馈闭环的关键是：反馈不能停留在表格里，要真正驱动系统改进。

## 10.11 Prompt 迭代

Prompt 是大模型产品的重要配置。

迭代时要注意：

1. 记录版本。
2. 说明修改原因。
3. 跑回归评估。
4. 灰度上线。
5. 监控线上变化。
6. 能快速回滚。

不要在生产环境里随手改 prompt。Prompt 变化可能影响大量行为。

## 10.12 模型版本迭代

模型升级可能带来能力提升，也可能带来回归。

要评估：

1. 任务成功率是否提升。
2. 延迟是否变化。
3. 成本是否变化。
4. 风格是否变化。
5. 拒答策略是否变化。
6. 安全表现是否变化。
7. 结构化输出是否稳定。

模型升级要像软件版本升级一样管理，而不是直接替换。

## 10.13 知识库运营

RAG 产品要运营知识库。

工作包括：

1. 新文档入库。
2. 过期文档下线。
3. 文档负责人维护。
4. 权限同步。
5. 缺失问题补文档。
6. 高访问文档优化。
7. 冲突文档合并。

知识库不运营，RAG 效果会逐渐下降。

## 10.14 安全监控

上线后要持续监控安全。

包括：

1. 敏感信息泄露。
2. 越权访问。
3. Prompt injection。
4. 高风险请求。
5. 不合规输出。
6. 工具误用。
7. 异常调用量。
8. 滥用行为。

安全不是上线前测一次，而是持续监控和响应。

## 10.15 运营看板

大模型产品需要运营看板。

看板可以包括：

1. DAU 和 MAU。
2. 任务量。
3. 成功率。
4. 满意度。
5. 成本。
6. 延迟。
7. 错误类型分布。
8. 热门问题。
9. 低满意度样本。
10. 安全事件。

看板的目标不是好看，而是帮助团队发现问题和决定下一步优化。

## 10.16 持续迭代优先级

不是所有问题都要立刻修。

优先级可以看：

1. 影响用户数。
2. 业务价值。
3. 安全风险。
4. 发生频率。
5. 修复成本。
6. 是否阻塞核心流程。

高频低风险问题和低频高风险问题都要关注，只是处理方式不同。

## 10.17 常见失败模式

1. 上线后没人看数据。
2. 只收集点赞点踩，不分析原因。
3. Prompt 随意改，没有版本管理。
4. 模型升级不做回归。
5. 知识库不维护。
6. 成本异常没人发现。
7. 安全事件没有响应流程。
8. A/B test 指标不清。
9. 反馈没有进入评估集。
10. 只追求模型能力，不运营产品。

大模型产品的长期质量来自持续运营，而不是一次上线。

## 10.18 面试题：大模型产品上线后如何运营

回答要点：

```text
上线后要监控质量、成本、延迟、安全和用户反馈。质量上看任务成功率、答案正确率、引用准确率和满意度；成本上看 token、模型调用、工具调用和单任务成本；安全上看越权、敏感信息和不合规输出。用户反馈和失败样本要进入评估集，驱动 prompt、RAG、模型和产品交互迭代。
```

## 10.19 面试题：如何做大模型产品 A/B Test

回答要点：

```text
先明确实验目标和主指标，例如提升自助解决率、降低转人工率或提高引用准确率。然后控制变量，只比较一个关键差异，比如模型、prompt、检索策略或交互设计。实验要灰度上线，监控质量、延迟、成本和安全指标，避免只看单一点击率。最后结合统计结果和失败案例决定是否全量。
```

## 10.20 本章小结

大模型产品上线后，需要持续监控、反馈、评估和迭代。灰度发布、质量监控、成本监控、用户反馈、失败样本分析、A/B test、知识库运营、安全监控和版本管理，都是产品长期成功的关键。
