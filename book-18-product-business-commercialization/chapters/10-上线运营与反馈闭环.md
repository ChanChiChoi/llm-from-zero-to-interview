# 第十章：上线运营与反馈闭环

## 0. 本讲资料边界与第二轮精修口径

本讲按 `WRITING_PLAN.md` 的第二轮要求做公式和 demo 精修。联网资料主要核对七类口径：OpenAI Evals / graders 资料提醒我们，线上反馈要回流为可复现评估、打分器和回归报告；OpenAI 生产最佳实践和 latency optimization 资料提醒我们，生产系统要持续监控错误、延迟、重试、限流、成本和用户体验；OpenAI Agents tracing 资料提醒我们，Agent 和工具型产品要保留可审计 trace；OpenAI safety best practices 提醒我们，上线后仍要做安全监控、滥用检测和事故响应；Google SRE 的 SLI / SLO / error budget 口径提醒我们，平均值不足以描述可用性和长尾体验；NIST AI RMF Generative AI Profile 提醒我们，生成式 AI 上线后要持续治理、监控、度量和人类监督；OWASP LLM Top 10 提醒我们，敏感信息泄露、提示注入、过度代理和资源消耗风险会在线上持续出现。

本章不替代企业 SRE、数据安全、隐私合规或真实增长实验设计。这里聚焦产品和工程面试口径：如何把大模型产品上线后的质量、成本、延迟、安全、反馈、回归评估和版本治理做成可观测、可回滚、可迭代的闭环。

大模型产品上线不是终点，而是持续迭代的开始。模型会遇到训练时没见过的问题，用户会用意想不到的方式提问，知识库会过期，成本会变化，模型版本会更新，安全风险会暴露。没有运营和反馈闭环的大模型产品，很快会从“看起来智能”变成“不稳定、不可信、没人用”。

本章系统讲上线运营与反馈闭环：灰度发布、质量监控、用户反馈、A/B test、失败样本分析、数据闭环、模型和 prompt 迭代、成本监控、安全监控和持续运营。

## 10.1 上线为什么不是终点

传统软件上线后，主要关注 bug 和性能。大模型产品上线后，还要关注模型行为。

例如：

1. 用户问法是否符合预期。
2. 模型是否幻觉。
3. RAG 是否召回正确文档。
4. Agent 是否误用工具。
5. 成本是否超预算。
6. 拒答是否过多或过少。
7. 用户是否真的采纳输出。

面试回答：

```text
大模型产品上线后必须持续运营，因为模型行为会受到用户输入、知识库更新、模型版本、prompt、工具和业务环境影响。需要监控质量、成本、延迟、安全和用户反馈，把失败案例沉淀为评估集或优化数据，形成持续迭代闭环。
```

## 10.2 灰度发布

大模型产品不应该一次全量上线。

灰度发布可以按：

1. 用户群体。
2. 业务线。
3. 地区。
4. 功能模块。
5. 风险等级。
6. 流量比例。

灰度阶段要观察：

1. 使用率。
2. 成功率。
3. 用户反馈。
4. 错误类型。
5. 成本。
6. 延迟。
7. 安全事件。

灰度不是形式，而是为了在风险可控范围内发现真实问题。

## 10.3 质量监控

质量监控要覆盖模型输出和任务结果。

常见指标：

1. 答案正确率。
2. 用户满意度。
3. 任务完成率。
4. 引用准确率。
5. 无法回答率。
6. 转人工率。
7. 重试率。
8. 重新生成率。
9. 投诉率。
10. 人工审核通过率。

不同产品指标不同。客服看解决率，RAG 看引用准确率，Agent 看任务完成率，代码助手看采纳率和测试通过率。

## 10.4 成本监控

大模型产品成本可能悄悄上涨。

需要监控：

1. 平均 token。
2. 输出长度。
3. 模型调用次数。
4. RAG 检索次数。
5. Agent 步数。
6. 工具调用次数。
7. 重试次数。
8. 单用户成本。
9. 单任务成本。
10. 总成本趋势。

成本异常可能来自 prompt 变长、RAG top-k 过大、Agent 无限循环、用户滥用或模型升级。

## 10.5 延迟监控

延迟影响体验。

要监控：

1. 首 token 延迟。
2. 总生成时间。
3. 检索耗时。
4. 工具调用耗时。
5. 排队时间。
6. 超时率。
7. P50、P95、P99 延迟。

平均延迟不够，长尾延迟更影响用户体验。尤其是企业工作流中，P95 和 P99 很重要。

## 10.6 用户反馈

用户反馈是优化的重要来源。

反馈形式：

1. 点赞点踩。
2. 选择错误类型。
3. 输入期望答案。
4. 标记引用错误。
5. 请求转人工。
6. 手动编辑结果。
7. 投诉和客服记录。

简单点赞点踩信息量有限。更好的反馈要能定位问题：是检索错、理解错、生成错、引用错、权限错，还是用户期望不同。

## 10.7 失败样本分析

失败样本是产品改进的燃料。

失败类型可以分为：

1. 意图理解错误。
2. 检索失败。
3. 引用错误。
4. 生成幻觉。
5. 工具调用失败。
6. 权限问题。
7. 拒答不当。
8. 格式不符合要求。
9. 延迟过高。
10. 用户需求不清。

每类失败对应不同优化手段。不能把所有失败都归因于“模型不够强”。

## 10.8 A/B Test

A/B test 用于比较不同方案。

可以测试：

1. 不同模型。
2. 不同 prompt。
3. 不同 RAG 策略。
4. 不同交互设计。
5. 不同拒答策略。
6. 不同排序或推荐方案。

指标要提前定义。例如测试 RAG 策略时，看引用准确率、自助解决率和延迟，而不是只看主观感觉。

## 10.9 离线评估和在线评估

离线评估：用固定评估集测试。

优点：

1. 可重复。
2. 成本低。
3. 适合回归测试。

缺点：

1. 覆盖不了真实用户长尾。
2. 可能过拟合评估集。

在线评估：看真实用户使用数据。

优点：

1. 贴近真实场景。
2. 能发现新问题。

缺点：

1. 风险更高。
2. 归因复杂。
3. 需要灰度和监控。

成熟产品需要离线和在线结合。

## 10.10 反馈到数据闭环

用户反馈要进入数据闭环。

流程：

```text
线上日志 -> 失败样本 -> 人工标注 -> 评估集更新 -> prompt/RAG/模型优化 -> 回归测试 -> 灰度上线
```

反馈闭环的关键是：反馈不能停留在表格里，要真正驱动系统改进。

## 10.11 Prompt 迭代

Prompt 是大模型产品的重要配置。

迭代时要注意：

1. 记录版本。
2. 说明修改原因。
3. 跑回归评估。
4. 灰度上线。
5. 监控线上变化。
6. 能快速回滚。

不要在生产环境里随手改 prompt。Prompt 变化可能影响大量行为。

## 10.12 模型版本迭代

模型升级可能带来能力提升，也可能带来回归。

要评估：

1. 任务成功率是否提升。
2. 延迟是否变化。
3. 成本是否变化。
4. 风格是否变化。
5. 拒答策略是否变化。
6. 安全表现是否变化。
7. 结构化输出是否稳定。

模型升级要像软件版本升级一样管理，而不是直接替换。

## 10.13 知识库运营

RAG 产品要运营知识库。

工作包括：

1. 新文档入库。
2. 过期文档下线。
3. 文档负责人维护。
4. 权限同步。
5. 缺失问题补文档。
6. 高访问文档优化。
7. 冲突文档合并。

知识库不运营，RAG 效果会逐渐下降。

## 10.14 安全监控

上线后要持续监控安全。

包括：

1. 敏感信息泄露。
2. 越权访问。
3. Prompt injection。
4. 高风险请求。
5. 不合规输出。
6. 工具误用。
7. 异常调用量。
8. 滥用行为。

安全不是上线前测一次，而是持续监控和响应。

## 10.15 运营看板

大模型产品需要运营看板。

看板可以包括：

1. DAU 和 MAU。
2. 任务量。
3. 成功率。
4. 满意度。
5. 成本。
6. 延迟。
7. 错误类型分布。
8. 热门问题。
9. 低满意度样本。
10. 安全事件。

看板的目标不是好看，而是帮助团队发现问题和决定下一步优化。

## 10.16 持续迭代优先级

不是所有问题都要立刻修。

优先级可以看：

1. 影响用户数。
2. 业务价值。
3. 安全风险。
4. 发生频率。
5. 修复成本。
6. 是否阻塞核心流程。

高频低风险问题和低频高风险问题都要关注，只是处理方式不同。

## 10.16.1 关键公式与上线运营指标速查

可以把一次线上请求或任务样本写成：

```math
o_i=(u_i,x_i,y_i,z_i,f_i,b_i,v_i,c_i,l_i,r_i)
```

其中 `u_i` 是用户或租户，`x_i` 是输入，`y_i` 是模型输出或 Agent 轨迹结果，`z_i` 是任务是否成功，`f_i` 是用户反馈或人工审核标签，`b_i` 是 bad case 类别，`v_i` 是模型 / prompt / RAG / 工具版本，`c_i` 是成本，`l_i` 是延迟，`r_i` 是安全、隐私或回滚风险标记。

**1. 灰度覆盖率**

灰度发布要明确当前流量占比：

```math
q_{\mathrm{roll}}=\frac{N_{\mathrm{rollout}}}{N_{\mathrm{eligible}}+\epsilon}
```

其中 `N_rollout` 是进入新版本或新策略的请求数，`N_eligible` 是符合实验条件的请求数。灰度覆盖率太高会放大风险，太低则统计信号不足。

**2. 线上任务成功率与采纳率**

上线后要看真实任务是否完成，而不是只看模型输出是否流畅：

```math
A_{\mathrm{live}}=\frac{N_{\mathrm{success}}}{N_{\mathrm{task}}+\epsilon},\qquad
R_{\mathrm{adopt}}=\frac{N_{\mathrm{adopted}}}{N_{\mathrm{success}}+\epsilon}
```

其中 `N_success` 是真实任务成功数，`N_adopted` 是用户采纳、保存、提交或继续使用模型结果的次数。低采纳率可能说明答案虽然正确，但不符合工作流、格式、信任或责任要求。

**3. 反馈采集率与反馈行动率**

反馈闭环要区分“收到了反馈”和“真的处理了反馈”：

```math
C_{\mathrm{fb}}=\frac{N_{\mathrm{feedback}}}{N_{\mathrm{task}}+\epsilon},\qquad
R_{\mathrm{fb}}=\frac{N_{\mathrm{feedback\_actioned}}}{N_{\mathrm{feedback}}+\epsilon}
```

其中 `N_feedback` 是点赞点踩、错误类型、编辑、投诉、转人工或人工审核反馈数，`N_feedback_actioned` 是已经进入评估集、prompt 修复、知识库修复、产品改动或安全回归的反馈数。

**4. Bad case 分析覆盖率**

失败样本不能只堆在表格里：

```math
C_{\mathrm{bad}}=\frac{N_{\mathrm{bad\_triaged}}}{N_{\mathrm{bad}}+\epsilon}
```

其中 `N_bad` 是线上失败样本数，`N_bad_triaged` 是已经标注失败原因、责任模块、影响范围、修复方式和复测状态的样本数。

**5. 回归评估通过率与评估切片覆盖率**

每次改模型、prompt、检索策略、工具 schema 或安全策略前，都要跑回归：

```math
R_{\mathrm{reg}}=\frac{N_{\mathrm{reg\_pass}}}{N_{\mathrm{reg}}+\epsilon},\qquad
C_{\mathrm{slice}}=\frac{N_{\mathrm{slice\_covered}}}{N_{\mathrm{slice}}+\epsilon}
```

其中 `N_reg` 是回归样本数，`N_reg_pass` 是通过样本数；`N_slice` 是预期覆盖的业务、风险、语言、长度、权限、知识库和高价值用户切片数。

**6. 延迟 SLO 与成本预算门禁**

线上体验看长尾，不只看平均值：

```math
G_{\mathrm{lat}}=\mathbf{1}[L_{95}\leq L_{\mathrm{slo}}],\qquad
G_{\mathrm{cost}}=\mathbf{1}[C_{\mathrm{unit}}\leq C_{\mathrm{budget}}]
```

其中 `L_95` 是 P95 延迟，`L_slo` 是延迟目标，`C_unit` 是单任务成本，`C_budget` 是单任务成本预算。模型升级、prompt 变长、RAG top-k 增大、Agent 步数增加和重试率上升都可能让这两个门禁失效。

**7. 安全事故率与事故处理覆盖率**

上线后要持续监控安全、隐私和滥用事件：

```math
R_{\mathrm{incident}}=\frac{N_{\mathrm{incident}}}{N_{\mathrm{task}}+\epsilon},\qquad
C_{\mathrm{incident}}=\frac{N_{\mathrm{incident\_resolved}}}{N_{\mathrm{incident}}+\epsilon}
```

其中 `N_incident` 是敏感信息泄露、越权访问、提示注入、工具误用、高风险输出、异常调用或滥用事件数，`N_incident_resolved` 是已经完成止血、定位、修复、通知和复测的事件数。

**8. Trace 覆盖率与回滚就绪**

Agent、RAG、工具调用和企业应用上线后，光存输入输出不够，还要能复盘链路：

```math
C_{\mathrm{trace}}=\frac{N_{\mathrm{trace}}}{N_{\mathrm{trace\_required}}+\epsilon},\qquad
G_{\mathrm{rollback}}=\mathbf{1}[\mathrm{rollback\_plan\_ready}]
```

其中 `N_trace` 是实际保留了可审计 trace、版本、检索证据、工具调用和关键指标的请求数，`N_trace_required` 是按风险策略要求必须保留 trace 的请求数。`G_rollback` 表示是否已经准备 feature flag、旧版本、数据回滚、配置回滚、人工兜底和事故沟通预案。没有 trace 就很难定位失败，没有回滚就不适合扩大灰度。

**9. 上线运营门禁**

一个简化上线运营门禁可以写成：

```math
G_{\mathrm{ops}}=\mathbf{1}[A_{\mathrm{live}}\geq a_0]\mathbf{1}[R_{\mathrm{adopt}}\geq d_0]\mathbf{1}[R_{\mathrm{fb}}\geq f_0]\mathbf{1}[C_{\mathrm{bad}}\geq b_0]\mathbf{1}[R_{\mathrm{reg}}\geq r_0]\mathbf{1}[C_{\mathrm{slice}}\geq s_0]\mathbf{1}[G_{\mathrm{lat}}=1]\mathbf{1}[G_{\mathrm{cost}}=1]\mathbf{1}[R_{\mathrm{incident}}\leq i_0]\mathbf{1}[C_{\mathrm{incident}}\geq h_0]\mathbf{1}[C_{\mathrm{trace}}\geq t_0]\mathbf{1}[G_{\mathrm{rollback}}=1]
```

真实项目还要加入 canary、shadow traffic、feature flag、回滚预案、模型 / prompt / RAG / 工具版本记录、trace、人工复核、异常告警、SLO burn rate 和事故复盘。运营门禁强调：上线不是“评估通过一次”，而是“能持续发现问题、修复问题并证明没有引入严重回归”。

```python
        "regression": rate(row["reg_pass"], row["reg_total"]),
        "slice": rate(row["slices_covered"], row["slices"]),
        "trace": rate(row["traces"], row["traces_required"]),
        "incident_rate": rate(row["incidents"], row["tasks"]),
        "incident_resolved": (
            1.0
            if row["incidents"] == 0
            else rate(row["incidents_resolved"], row["incidents"])
        ),
    }
    gates = {
        "quality_ok": (
            metrics["live_success"] >= thresholds["live_success"]
            and metrics["adoption"] >= thresholds["adoption"]
        ),
        "feedback_ok": (
            metrics["feedback_action"] >= thresholds["feedback_action"]
            and metrics["bad_triage"] >= thresholds["bad_triage"]
        ),
        "regression_ok": (
            metrics["regression"] >= thresholds["regression"]
            and metrics["slice"] >= thresholds["slice"]
        ),
        "latency_ok": row["p95_latency_ms"] <= row["latency_slo_ms"],
        "cost_ok": row["unit_cost"] <= row["cost_budget"],
        "incident_ok": (
            metrics["incident_rate"] <= thresholds["incident_rate"]
            and metrics["incident_resolved"] >= thresholds["incident_resolved"]
        ),
        "trace_ok": metrics["trace"] >= thresholds["trace"],
        "rollback_ok": row["rollback_ready"],
    }
    failed = [name for name, ok in gates.items() if not ok]
    score_parts = [
        metrics["live_success"],
        metrics["adoption"],
        metrics["feedback_action"],
        metrics["bad_triage"],
        metrics["regression"],
        metrics["slice"],
        metrics["trace"],
        1.0 if gates["latency_ok"] else 0.0,
        1.0 if gates["cost_ok"] else 0.0,
        1.0 if gates["incident_ok"] else 0.0,
        1.0 if gates["rollback_ok"] else 0.0,
    ]
    return {
        "name": row["name"],
        "score": round(sum(score_parts) / len(score_parts), 3),
        "gate_pass": not failed,
        "failed_gates": failed,
        "metrics": metrics,
    }


reports = [audit(row) for row in products]
ranked = [
    (r["name"], r["score"], r["gate_pass"])
    for r in sorted(reports, key=lambda x: x["score"], reverse=True)
]
ops_pass = [r["name"] for r in reports if r["gate_pass"]]
needs_rework = {r["name"]: r["failed_gates"] for r in reports if not r["gate_pass"]}
reason_counts = Counter(reason for row in products for reason in row["failure_reasons"])

sample = reports[0]["metrics"]
sample_metrics = {
    "live_success": round(sample["live_success"], 3),
    "adoption": round(sample["adoption"], 3),
    "feedback_action": round(sample["feedback_action"], 3),
    "bad_triage": round(sample["bad_triage"], 3),
    "regression": round(sample["regression"], 3),
    "trace": round(sample["trace"], 3),
}

print("ranked=", ranked)
print("ops_pass=", ops_pass)
print("sample_metrics=", sample_metrics)
print("needs_rework=", needs_rework)
print("top_failure_reasons=", reason_counts.most_common(5))
```

## 10.17 常见失败模式

1. 上线后没人看数据。
2. 只收集点赞点踩，不分析原因。
3. Prompt 随意改，没有版本管理。
4. 模型升级不做回归。
5. 知识库不维护。
6. 成本异常没人发现。
7. 安全事件没有响应流程。
8. A/B test 指标不清。
9. 反馈没有进入评估集。
10. 只追求模型能力，不运营产品。

大模型产品的长期质量来自持续运营，而不是一次上线。

## 10.18 面试题：大模型产品上线后如何运营

回答要点：

```text
上线后要监控质量、成本、延迟、安全和用户反馈。质量上看任务成功率、答案正确率、引用准确率和满意度；成本上看 token、模型调用、工具调用和单任务成本；安全上看越权、敏感信息和不合规输出。用户反馈和失败样本要进入评估集，驱动 prompt、RAG、模型和产品交互迭代。
```

## 10.19 面试题：如何做大模型产品 A/B Test

回答要点：

```text
先明确实验目标和主指标，例如提升自助解决率、降低转人工率或提高引用准确率。然后控制变量，只比较一个关键差异，比如模型、prompt、检索策略或交互设计。实验要灰度上线，监控质量、延迟、成本和安全指标，避免只看单一点击率。最后结合统计结果和失败案例决定是否全量。
```

## 10.20 本章小结

大模型产品上线后，需要持续监控、反馈、评估和迭代。灰度发布、质量监控、成本监控、用户反馈、失败样本分析、A/B test、知识库运营、安全监控和版本管理，都是产品长期成功的关键。
