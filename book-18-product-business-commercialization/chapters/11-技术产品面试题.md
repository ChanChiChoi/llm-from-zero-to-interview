# 第十一章：技术产品面试题

本章是第十八册的综合面试题。前面章节已经讲过产品化总览、用户需求、产品体验、ROI、企业级应用、RAG、Agent、多模态、隐私合规和上线运营。本章把这些内容整理成面试中可以直接使用的回答。

技术产品类问题的关键，不是把自己包装成产品经理，而是证明你理解技术如何产生业务价值。一个成熟的算法工程师，应该既能讲模型、RAG、Agent、评估，也能讲用户、场景、成本、风险、上线和反馈闭环。

本讲按 `WRITING_PLAN.md` 的第二轮要求做公式和 demo 精修。联网资料主要核对七类口径：OpenAI Evals / graders 用于把线上反馈和失败样本转成可重复评估；OpenAI 生产最佳实践和 latency optimization 用于支撑成本、延迟、版本和生产稳定性回答；OpenAI Agents SDK tracing 用于支撑 Agent trace、工具调用和可审计性；Google SRE SLI / SLO / error budget 用于支撑 P95/P99、SLO 和回滚口径；NIST AI RMF Generative AI Profile 用于支撑生成式 AI 上线后的治理、监控和人类监督；OWASP LLM Top 10 用于支撑提示注入、敏感信息泄露、过度代理和资源消耗等线上风险边界；前面第十八册各章的审计 demo 用于把面试回答落到可计算指标和门禁。

本章不替代真实产品经理面试、商业计划或法务合规审查。这里聚焦算法工程师的技术产品表达：把“我会做模型”升级成“我能证明模型方案在真实用户、成本、风险和运营闭环中值得上线”。

## 11.1 大模型从 Demo 到产品差在哪里

回答要点：

```text
Demo 主要证明技术能跑通，通常样例少、输入可控、失败可接受。产品要面对真实用户和长尾输入，需要稳定性、成本控制、安全合规、权限、监控、反馈闭环和明确业务指标。大模型产品化不是把模型接进 UI，而是把模型能力转化为用户价值，并保证它在真实环境中可靠、可控、可持续。
```

## 11.2 如何判断一个 LLM 场景值不值得做

回答要点：

```text
我会从用户痛点、任务频率、单次价值、数据可得性、可验证性、容错率、成本和风险几个维度评估。优先选择高频或高价值、数据可得、错误可控、指标可衡量、能接入现有工作流的场景。对于高风险场景，不会一开始全自动，而是先做人机协同和试点验证。
```

## 11.3 如何判断需求是否真实

回答要点：

```text
我不会只听用户说想要什么，而会看他们现在是否已经花时间或成本解决这个问题，任务发生频率如何，现有方案痛点在哪里，成功后能提升什么指标，以及用户是否愿意改变工作流。如果没有明确用户、没有当前成本、没有成功指标，很可能是伪需求或展示型需求。
```

## 11.4 如何把模型能力转成产品体验

回答要点：

```text
我会先明确用户任务和成功标准，再把模型能力映射到体验指标。例如理解能力对应意图识别和任务完成率，生成能力对应草稿质量和采纳率，检索能力对应引用准确率，推理能力对应复杂任务成功率。同时要设计延迟、结构化输出、引用、不确定性提示、错误恢复和反馈入口，保证用户能信任和控制结果。
```

## 11.5 为什么高分模型产品体验可能不好

回答要点：

```text
因为 benchmark 只覆盖部分能力，而产品体验还受延迟、稳定性、可控性、交互设计、引用、拒答、风格、成本和错误恢复影响。一个模型离线分数高，但如果线上响应慢、输出格式不稳定、不能解释来源、错误时无法恢复，用户体验仍然会差。
```

## 11.6 如何估算大模型项目 ROI

回答要点：

```text
我会先明确收益来源，是节省人工、提升效率、增加收入还是降低风险。然后估算任务量、自动化比例、单次节省时间或增量收入，再计算模型调用、RAG、工具、人审、研发、运维和合规成本。最后看单位经济账和整体 ROI，并通过试点和线上数据校准假设。
```

## 11.7 如何降低大模型产品成本

回答要点：

```text
可以从模型、上下文、调用策略和系统架构优化。简单任务用小模型或规则，大模型只处理复杂任务；控制 prompt 和 RAG 上下文长度；缓存高频问题；减少无效重试；对 Agent 设置最大步骤和工具调用预算；离线任务用批处理；自部署场景可以考虑量化和推理加速。但优化要保证质量、安全和用户体验不被明显损害。
```

## 11.8 企业级 LLM 应用和普通应用有什么区别

回答要点：

```text
企业级 LLM 应用更强调数据、权限、系统集成、稳定性、审计和 ROI。普通应用可能只关注用户体验和模型效果，但企业应用需要接入 SSO、知识库、工单、CRM、数据仓库等系统，保证用户只能访问有权限的数据，并记录审计日志。它还要能衡量业务指标，例如节省人工、提高解决率或降低成本。
```

## 11.9 如何落地企业知识库问答

回答要点：

```text
我会先做数据治理，整理文档来源、版本、权限和元数据，再构建 RAG 流程，包括文档解析、分块、embedding、检索、rerank、答案生成和引用。权限过滤必须贯穿检索和生成。上线前要建立评估集，评估召回率、答案正确率、引用准确率和无法回答率。上线后收集用户反馈，持续更新文档和优化检索。
```

## 11.10 RAG 产品落地最难的是什么

回答要点：

```text
RAG 落地最难的不只是检索算法，而是企业文档治理、权限控制、知识更新、引用可信和效果评估。文档过期、版本冲突、权限不清、引用不支持答案，都会让用户失去信任。一个可靠 RAG 产品需要文档解析、分块、混合检索、rerank、权限过滤、引用校验、评估集和用户反馈闭环。
```

## 11.11 如何评估企业 RAG

回答要点：

```text
我会分层评估。检索层看召回率和 rerank 命中率；生成层看答案正确率、引用准确率和证据支持率；产品层看自助解决率、用户满意度、无法回答准确率、延迟和成本。评估集要来自真实用户问题，并标注标准答案、支持文档、关键段落和权限信息。
```

## 11.12 Agent 产品落地最重要的是什么

回答要点：

```text
Agent 产品落地最重要的是控制任务边界和行动风险。Agent 能执行工具和真实动作，所以要明确适用场景、工具权限、状态管理、人工确认、失败恢复和审计日志。早期不建议直接全自动，应从低风险、可验证、人机协同的任务开始，用 trace 和评估指标持续改进。
```

## 11.13 如何设计企业 Agent 产品

回答要点：

```text
我会先选择高价值但风险可控的任务，定义完成标准。系统上设计 planner、tool registry、executor、state manager、permission checker、human confirmation、logger 和 evaluator。只给 Agent 当前任务需要的最小工具权限，高风险动作必须确认。上线后评估任务完成率、工具调用成功率、错误恢复、人工接管、安全违规、成本和用户满意度。
```

## 11.14 多模态产品落地难点是什么

回答要点：

```text
多模态产品落地难点包括输入质量不稳定、模型对细节理解可能出错、生成可控性不足、延迟和成本较高、版权和隐私风险更复杂，以及评估标准更主观。产品上需要输入质量引导、引用或证据展示、人审机制、安全审核、版权治理和明确的体验指标。
```

## 11.15 如何落地图像理解产品

回答要点：

```text
我会先明确图像理解服务哪个任务，例如文档解析、质检、拍照问答或设计评审。然后评估输入质量、目标识别难度、OCR 或 layout 需求、错误容忍度和人审流程。系统上需要图像预处理、模型理解、结构化输出、置信度提示和反馈闭环。高风险场景不能让模型独立决策，而应作为辅助工具。
```

## 11.16 大模型产品如何做隐私保护

回答要点：

```text
我会从数据全链路做隐私保护。首先梳理数据流，明确用户输入、检索文档、工具结果、日志和反馈会流向哪里。然后做数据最小化、PII 检测和脱敏、权限隔离、日志治理、加密存储、用户删除机制和审计。对于外部模型 API，要明确数据是否保存、是否用于训练、是否出境。隐私保护不能只靠提示词，要靠系统设计。
```

## 11.17 企业 RAG 如何避免数据泄露

回答要点：

```text
企业 RAG 要在检索和生成全链路做权限控制。检索时只能召回用户有权限的文档，引用也必须是用户可访问的来源；不能把无权限文档摘要给用户。日志要脱敏，向量索引要按租户或权限隔离，权限变更要及时同步。还要通过审计记录谁访问了哪些内容。
```

## 11.18 大模型产品上线后如何运营

回答要点：

```text
上线后要监控质量、成本、延迟、安全和用户反馈。质量上看任务成功率、答案正确率、引用准确率和满意度；成本上看 token、模型调用、工具调用和单任务成本；安全上看越权、敏感信息和不合规输出。用户反馈和失败样本要进入评估集，驱动 prompt、RAG、模型和产品交互迭代。
```

## 11.19 如何做大模型产品 A/B Test

回答要点：

```text
先明确实验目标和主指标，例如提升自助解决率、降低转人工率或提高引用准确率。然后控制变量，只比较一个关键差异，比如模型、prompt、检索策略或交互设计。实验要灰度上线，监控质量、延迟、成本和安全指标，避免只看单一点击率。最后结合统计结果和失败案例决定是否全量。
```

## 11.20 关键公式与面试评分速查

技术产品面试不是背诵产品词汇，而是用指标和门禁证明判断。一次面试回答可以抽象为：

```math
a_i=(q_i,T_i,F_i,D_i,R_i,H_i)
```

其中 `q_i` 是问题，`T_i` 是覆盖的主题集合，`F_i` 是给出的公式或指标集合，`D_i` 是能落地的 demo 或项目证据集合，`R_i` 是风险边界集合，`H_i` 是 trade-off 集合。

**1. 主题覆盖率**

```math
C_{\mathrm{topic}}=\frac{|T_{\mathrm{ans}}\cap T_{\mathrm{req}}|}{|T_{\mathrm{req}}|+\epsilon}
```

`T_req` 至少应覆盖 demo vs product、真实需求、产品指标、ROI、RAG、Agent、多模态、隐私、上线运营和反馈闭环。只讲模型结构而不讲用户、指标和风险，主题覆盖通常不够。

**2. 公式覆盖率**

```math
C_{\mathrm{formula}}=\frac{|F_{\mathrm{ans}}\cap F_{\mathrm{req}}|}{|F_{\mathrm{req}}|+\epsilon}
```

高频公式包括 monthly benefit、unit cost、ROI、retrieval recall、citation accuracy、task success、unauthorized rate、input quality rate、ops gate、trace coverage、feedback action rate 和 privacy gate。公式不要求每题都写满，但要能在关键题上把判断量化。

**3. Demo 证据覆盖率**

```math
C_{\mathrm{demo}}=\frac{|D_{\mathrm{ans}}\cap D_{\mathrm{req}}|}{|D_{\mathrm{req}}|+\epsilon}
```

这里的 demo 指本册前面可运行的审计脚本或项目证据，例如 ROI audit、RAG product audit、Agent product audit、multimodal product audit、privacy audit 和 ops audit。面试中能说出“我会用什么表和什么脚本验证”，比只说“持续优化”更可信。

**4. 风险与 trade-off 覆盖率**

```math
C_{\mathrm{risk}}=\frac{|R_{\mathrm{ans}}\cap R_{\mathrm{req}}|}{|R_{\mathrm{req}}|+\epsilon},\qquad
C_{\mathrm{trade}}=\frac{|H_{\mathrm{ans}}\cap H_{\mathrm{req}}|}{|H_{\mathrm{req}}|+\epsilon}
```

风险至少包括数据泄露、过度代理、隐私、版权、事故响应和采用率不确定性；trade-off 至少包括质量 vs 成本、召回 vs 延迟、自动化 vs 控制、延迟 vs 质量、上线速度 vs 安全。

**5. 技术产品面试门禁**

```math
G_{\mathrm{prod\_int}}=\mathbf{1}[C_{\mathrm{topic}}\geq t_0]\mathbf{1}[C_{\mathrm{formula}}\geq f_0]\mathbf{1}[C_{\mathrm{demo}}\geq d_0]\mathbf{1}[C_{\mathrm{risk}}\geq r_0]\mathbf{1}[C_{\mathrm{trade}}\geq h_0]\mathbf{1}[N_{\mathrm{weak}}\leq w_0]
```

其中 `N_weak` 是低分问题数。这个门禁的用途不是给自己制造焦虑，而是避免面试回答停留在“我知道这些词”，帮助你定位缺公式、缺 demo、缺风险边界或缺 trade-off 的题。

def union(field):
    values = set()
    for answer in answers:
        values.update(answer[field])
    return values


def coverage(field):
    got = union(field)
    need = required[field]
    return round(len(got & need) / len(need), 3), sorted(need - got)


summary = {}
missing = {}
for field in required:
    summary[field + "_coverage"], missing[field] = coverage(field)

question_scores = {}
for answer in answers:
    topic_hit = len(answer["topics"] & required["topics"]) / max(1, len(answer["topics"]))

weak_questions = [q for q, score in question_scores.items() if score < 0.75]
gates = {

}
interview_ready = all(gates.values())
revision_plan = []

if missing["demos"]:

if weak_questions:

print("summary=", summary)

print("question_scores=", question_scores)
print("weak_questions=", weak_questions)
print("gates=", gates)
print("interview_ready=", interview_ready)
print("revision_plan=", revision_plan)

## 11.22 技术产品面试回答模板

通用模板：

```text
我会从用户价值、技术可行性、成本收益、风险控制和上线运营五个方面看。先确认用户痛点和成功指标，再判断大模型是否是合适方案；然后设计模型、RAG 或 Agent 架构，并评估数据、权限、延迟和成本；上线时通过试点和灰度降低风险；上线后用监控和反馈闭环持续迭代。
```

用于 RAG：

```text
RAG 项目我会重点看文档治理、权限、检索、引用、评估和反馈闭环。因为企业 RAG 的难点通常不是接一个向量库，而是让答案基于正确、最新、有权限、可追溯的知识。
```

用于 Agent：

```text
Agent 项目我会重点看任务边界、工具权限、人工确认、失败恢复和审计。Agent 能执行真实动作，所以产品化时不能只追求自主性，而要保证可控、可验证、可回滚。
```

用于 ROI：

```text
ROI 我会把收益和成本都拆开。收益看节省人工、提升效率、增加收入或降低风险；成本看模型、检索、工具、人审、研发、运维和合规。再用试点数据校准假设，而不是只凭主观估计。
```

## 11.23 高频追问

追问：什么时候不该用大模型？

```text
当规则或传统系统能稳定低成本解决，任务强确定、低延迟、低容错，或者数据和权限无法治理时，不应强行用大模型。大模型适合自然语言、多样表达、长尾、生成和复杂上下文场景。
```

追问：产品上线后指标下降怎么办？

```text
先定位是模型问题、检索问题、数据变化、prompt 变化、用户流量变化还是成本延迟问题。需要看日志、失败样本、版本变更和用户反馈，再决定回滚、修复知识库、调整 prompt 或重新评估模型。
```

追问：如何平衡安全和体验？

```text
安全和体验不是简单对立。低风险场景可以更自动化，高风险场景需要引用、人审和确认。好的拒答要说明原因并提供替代帮助，权限和审计应由系统层实现，避免把安全负担完全交给用户或模型。
```

## 11.24 本章小结

技术产品面试考察的是你是否理解大模型从能力到价值的完整链路。好的回答不只讲模型，也讲用户需求、场景选择、产品体验、成本收益、企业系统、RAG/Agent 落地、隐私合规和上线运营。

到这里，第十八册《大模型产品化、商业化与落地》的正文第一版完成，第二轮也已经把产品化、场景选择、体验、ROI、企业应用、RAG、Agent、多模态、隐私治理、上线运营和技术产品面试串成了可计算、可复盘、可面试表达的闭环。本册的主线是：从“技术能做”走向“产品值得做、用户愿意用、业务能受益、风险可控制、系统可运营”。
