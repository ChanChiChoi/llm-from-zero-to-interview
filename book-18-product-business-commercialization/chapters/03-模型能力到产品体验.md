# 第三章：模型能力到产品体验

## 0. 本讲资料边界与第二轮精修口径

本讲按 `WRITING_PLAN.md` 的第二轮要求做公式和 demo 精修。联网资料主要核对四类口径：OpenAI evals / evaluation best practices 强调上线前要把任务、样本、指标和回归集定义清楚；OpenAI latency optimization 和 structured outputs 资料提示延迟、流式返回、缓存、结构化格式会直接影响体验；Google HEART 框架强调从用户目标拆出可观测 signals 和 metrics；Google SRE 的 SLI / SLO / error budget 口径提醒我们不能只看平均延迟，要看尾延迟和可接受失败率；NIST AI RMF / Generative AI Profile 提醒产品体验还要包含透明性、风险提示、人审和治理边界。

本章不展开具体 prompt 模板、RAG 检索算法、Agent 工具协议、前端设计系统或线上 A/B 实验平台。这里的核心问题是：如何把模型的理解、生成、检索、推理和工具能力，转成用户能感知的“快、准、稳、可控、可信、可恢复”的产品体验。

模型能力不等于产品体验。一个模型 benchmark 很高，但如果响应太慢、回答不稳定、无法引用来源、格式不可控、拒答策略混乱、用户不知道怎么追问，产品体验依然会很差。大模型产品化的关键，是把理解、生成、检索、推理和工具调用能力，转化为用户可感知、可信任、可持续使用的体验。

本章系统讲模型能力到产品体验的转化：延迟、准确性、可控性、解释性、拒答、风格、交互设计、引用、置信度、用户反馈、体验指标和常见失败模式。

## 3.1 为什么模型能力不等于体验

模型能力是技术侧概念，产品体验是用户侧感受。

例如模型有很强的总结能力，但用户体验可能受这些因素影响：

1. 等待时间太长。
2. 摘要太长或太短。
3. 重点不符合用户工作流。
4. 无法追溯原文。
5. 风格不符合业务场景。
6. 结果每次不一致。
7. 用户不知道如何修改输出。

面试回答：

```text
模型能力需要通过产品设计转化为体验。用户感知到的不是 benchmark，而是任务是否更快完成、答案是否可信、交互是否顺畅、错误是否可恢复、成本是否合理。产品化时要把准确率、延迟、可控性、解释性、拒答和反馈机制一起设计，而不是只替换更强模型。
```

## 3.2 延迟体验

延迟是用户最直接感知的指标之一。

大模型产品的延迟来自：

1. 模型推理。
2. RAG 检索。
3. reranker。
4. 工具调用。
5. 多轮 Agent 循环。
6. 网络传输。
7. 排队和限流。

延迟不是越低越好，而是要和任务价值匹配。闲聊需要快；复杂报告可以慢一点；高价值分析可以等待，但要给用户进度反馈。

## 3.3 如何改善延迟感知

产品上可以降低用户对延迟的负面感受。

方法：

1. 流式输出。
2. 显示进度。
3. 先返回摘要，再补充细节。
4. 后台执行长任务。
5. 给用户可取消选项。
6. 缓存常见结果。
7. 对简单问题走快速路径。

例如 Agent 执行复杂任务时，可以展示“正在检索资料、正在生成草稿、正在检查引用”，而不是让用户空等。

## 3.4 准确性体验

用户感知的准确性不只是最终答案对不对。

还包括：

1. 是否理解了用户意图。
2. 是否覆盖关键约束。
3. 是否引用正确来源。
4. 是否承认不确定性。
5. 是否避免编造。
6. 是否能被用户验证。
7. 是否在追问中保持一致。

对于企业应用，准确性往往比“回答得流畅”更重要。流畅但错误的答案会损害信任。

## 3.5 可控性

可控性指用户能否影响输出结果。

常见控制维度：

1. 长度。
2. 风格。
3. 格式。
4. 语气。
5. 受众。
6. 数据范围。
7. 引用来源。
8. 是否保守或发散。

大模型默认输出可能不符合用户预期。产品可以提供模板、参数、选项、示例和编辑入口，让用户更容易控制结果。

## 3.6 结构化输出

结构化输出能显著提升体验和稳定性。

例如：

```text
结论：...
依据：...
风险：...
下一步建议：...
```

结构化输出适合：

1. 报告生成。
2. 工单总结。
3. 合同审查。
4. 客服回复。
5. 数据分析。
6. 面试反馈。

结构化输出的价值是降低用户理解成本，也便于后续系统处理。

## 3.7 解释性

用户需要知道答案为什么可信。

解释可以包括：

1. 依据来源。
2. 推理摘要。
3. 关键假设。
4. 不确定点。
5. 反例或限制。
6. 工具验证结果。

但解释不是越长越好。产品中更重要的是可验证解释，而不是完整内部推理。对于 RAG 产品，引用原文通常比长篇自我解释更有价值。

## 3.8 引用和来源

引用是建立信任的重要机制。

好的引用应该：

1. 指向具体文档或段落。
2. 支持对应结论。
3. 标明时间或版本。
4. 区分事实和推断。
5. 允许用户跳转查看原文。

坏引用包括：

1. 引用不相关文档。
2. 引用只支持部分结论。
3. 编造来源。
4. 用过期文档支持新结论。

引用准确率是很多企业 RAG 产品的重要指标。

## 3.9 拒答体验

拒答不是简单说“不行”。好的拒答应该说明原因，并给出可行替代方案。

例如：

```text
我不能直接给出医疗诊断，但可以帮你整理症状、列出需要咨询医生的问题，并提醒你尽快就医。
```

拒答体验要处理：

1. 安全限制。
2. 权限不足。
3. 信息不足。
4. 任务超出能力。
5. 数据不可访问。
6. 高风险建议。

拒答太少会有风险；拒答太多会降低可用性。产品需要平衡安全和帮助性。

## 3.10 风格一致性

风格是产品体验的一部分。

不同产品需要不同风格：

1. 客服：礼貌、简洁、可执行。
2. 法务：谨慎、严谨、保留条件。
3. 编程助手：直接、技术准确。
4. 教育产品：循序渐进、有引导。
5. 销售助手：有说服力但不能夸大。

风格一致性可以通过系统 prompt、模板、示例、后处理和评估来保证。

## 3.11 交互设计

大模型产品不一定只有聊天框。

常见交互形态：

1. 聊天式问答。
2. 文档侧边栏助手。
3. 表单自动填写。
4. 一键生成草稿。
5. 批量处理。
6. 工作流节点。
7. IDE 插件。
8. 浏览器插件。
9. 报告生成器。

聊天框灵活，但不一定最高效。很多企业场景更适合把模型嵌入现有流程，而不是让用户专门打开一个聊天页面。

## 3.12 用户引导

用户需要知道如何使用 AI 功能。

可以提供：

1. 示例问题。
2. 推荐任务。
3. Prompt 模板。
4. 输入字段。
5. 可选参数。
6. 结果编辑入口。
7. 反馈按钮。

不要假设用户会写好 prompt。好的产品应该把 prompt 工程隐藏在体验设计里。

## 3.13 错误恢复

模型错误不可避免，关键是能否恢复。

恢复方式：

1. 允许用户追问。
2. 提供重新生成。
3. 支持修改约束。
4. 显示引用让用户检查。
5. 给出“哪里不对”的反馈入口。
6. 对低置信答案提醒用户。
7. 高风险场景转人工。

产品体验中，错误恢复比假装永远正确更重要。

## 3.14 信任校准

信任校准是让用户既不过度信任，也不过度怀疑。

方法：

1. 标注不确定性。
2. 展示来源。
3. 区分事实和建议。
4. 对高风险内容提醒复核。
5. 不夸大能力。
6. 对无法完成的任务明确说明。

如果产品总是自信地给错答案，用户会失去信任；如果产品总是保守拒答，用户也不会继续使用。

## 3.15 个性化体验

大模型可以根据用户角色和偏好调整体验。

例如：

1. 给新手更多解释。
2. 给专家更短结论。
3. 给销售输出客户化话术。
4. 给工程师输出代码和命令。
5. 根据企业规范调整格式。

个性化要注意隐私和可控性。用户应该知道哪些偏好被记住，并能修改或删除。

## 3.16 体验指标

常见体验指标：

1. 任务完成率。
2. 首次成功率。
3. 平均完成时间。
4. 用户满意度。
5. 追问率。
6. 重新生成率。
7. 人工接管率。
8. 结果采纳率。
9. 引用点击率。
10. 留存率。

技术指标要和体验指标关联。例如延迟下降是否提升留存，引用准确率是否提升采纳率。

## 3.16.1 关键公式与能力到体验指标速查

为了避免只说“模型更强所以体验更好”，可以把一次产品交互记成结构化样本：

```math
e_i=(x_i,u_i,m_i,\hat y_i,y_i,z_i,\ell_i,c_i,r_i,h_i)
```

其中，$x_i$ 是用户输入，$u_i$ 是用户角色或场景，$m_i$ 是产品形态，$\hat y_i$ 是模型输出，$y_i$ 是期望结果或人工标注，$z_i$ 是可用证据或工具结果，$\ell_i$ 是端到端延迟，$c_i$ 是成本，$r_i$ 是风险标签，$h_i$ 是用户反馈或人工审核结果。

模型能力可以先抽象成一个能力向量：

```math
a_i=(U_i,G_i,R_i,Q_i,T_i,F_i,S_i)
```

其中，$U_i$ 是意图理解，$G_i$ 是生成质量，$R_i$ 是检索和引用能力，$Q_i$ 是复杂推理能力，$T_i$ 是工具调用能力，$F_i$ 是格式稳定性，$S_i$ 是安全和拒答能力。这个向量本身还不是体验，它必须映射到用户可观测指标。

任务成功率是最基础的体验指标：

```math
R_{\mathrm{succ}}=\frac{1}{N}\sum_{i=1}^{N}\mathbb{1}(\hat y_i\ \mathrm{passes}\ V_i)
```

其中，$V_i$ 是任务验收函数，可以是单元测试、人工 rubric、引用检查、业务规则或用户采纳。面试中要强调：验收函数必须贴近真实任务，而不是只看模型输出是否流畅。

延迟满意度可以按 SLO 粗略归一化：

```math
L_i=\max(0,1-\frac{\ell_{p95,i}}{\tau_i})
```

其中，$\ell_{p95,i}$ 是 P95 端到端延迟，$\tau_i$ 是该产品形态的可接受延迟阈值。闲聊、客服、IDE 补全和长报告的 $\tau_i$ 不一样；复杂任务可以慢，但需要进度反馈和可取消机制。

引用支持率衡量“用户为什么敢信”：

```math
C_{\mathrm{cite}}=\frac{\sum_i n_{\mathrm{sup},i}}{\sum_i n_{\mathrm{claim},i}}
```

其中，$n_{\mathrm{sup},i}$ 是被证据支持的关键结论数，$n_{\mathrm{claim},i}$ 是需要证据支持的关键结论数。对 RAG 和企业知识助手来说，引用支持率通常比“回答很像正确答案”更重要。

结构化输出稳定率衡量产品能否接入工作流：

```math
F_{\mathrm{fmt}}=\frac{N_{\mathrm{valid}}}{N}
```

其中，$N_{\mathrm{valid}}$ 是符合 JSON、表格、字段或业务模板要求的输出数。结构化输出失败会导致后续系统无法消费，即使文本内容看起来不错，也会破坏体验。

错误恢复率衡量用户遇到错误后能否继续完成任务：

```math
R_{\mathrm{rec}}=\frac{N_{\mathrm{recovered}}}{N_{\mathrm{failed}}}
```

其中，$N_{\mathrm{failed}}$ 是用户发现或系统检测到的失败次数，$N_{\mathrm{recovered}}$ 是通过追问、重新生成、修改约束、人工接管或工具校验恢复成功的次数。好产品不是永远不犯错，而是错误可见、可解释、可恢复。

信任校准可以用预测置信度和实际正确率的差距表示：

```math
K_{\mathrm{cal}}=1-\frac{1}{B}\sum_{b=1}^{B}|p_b-a_b|
```

其中，$p_b$ 是第 $b$ 个置信度桶的平均预测置信度，$a_b$ 是该桶真实正确率。过度自信会让用户误信错误答案；过度保守会造成过度拒答和低采用率。

综合体验分可以写成：

```math
S_{\mathrm{ux}}=0.25R_{\mathrm{succ}}+0.15L+0.15C_{\mathrm{cite}}+0.15F_{\mathrm{fmt}}+0.10C_{\mathrm{ctrl}}+0.10R_{\mathrm{rec}}+0.10K_{\mathrm{cal}}-0.15R_{\mathrm{over}}-0.20R_{\mathrm{unsafe}}
```

其中，$C_{\mathrm{ctrl}}$ 是用户可控性满意度，$R_{\mathrm{over}}$ 是过度拒答率，$R_{\mathrm{unsafe}}$ 是不安全或不合规输出率。权重可以按产品调整，核心思想是：体验必须同时看质量、延迟、可信、格式、可控、恢复和风险。

上线门禁可以写成：

```math
G_{\mathrm{ux}}=\mathbb{1}(R_{\mathrm{succ}}\ge \tau_s,\ \ell_{p95}\le \tau_l,\ C_{\mathrm{cite}}\ge \tau_c,\ F_{\mathrm{fmt}}\ge \tau_f,\ R_{\mathrm{unsafe}}\le \tau_u)
```

这个门禁提醒面试回答不要只说“换更强模型”。如果 P95 延迟、引用支持、格式稳定、安全和恢复不过线，高分模型也不该直接进入生产体验。

## 3.17 常见失败模式

1. 模型能力强，但交互入口复杂。
2. 回答准确，但太慢。
3. 输出格式不稳定，无法接入流程。
4. 拒答策略过度保守或过度宽松。
5. 没有引用，用户不敢信。
6. 错误后没有反馈入口。
7. 风格不符合业务场景。
8. 用户不会写 prompt。
9. 产品只追求炫技，不解决任务。
10. 体验指标和模型指标脱节。

产品体验是模型、系统、交互和业务流程共同作用的结果。

## 3.18 面试题：如何把模型能力转成产品体验

回答要点：

```text
我会先明确用户任务和成功标准，再把模型能力映射到体验指标。例如理解能力对应意图识别和任务完成率，生成能力对应草稿质量和采纳率，检索能力对应引用准确率，推理能力对应复杂任务成功率。同时要设计延迟、结构化输出、引用、不确定性提示、错误恢复和反馈入口，保证用户能信任和控制结果。
```

## 3.19 面试题：为什么一个高分模型产品体验可能不好

回答要点：

```text
因为 benchmark 只覆盖部分能力，而产品体验还受延迟、稳定性、可控性、交互设计、引用、拒答、风格、成本和错误恢复影响。一个模型离线分数高，但如果线上响应慢、输出格式不稳定、不能解释来源、错误时无法恢复，用户体验仍然会差。
```

```python
    calibration = calibration_score(product["confidence_bins"])
    over_refusal = safe_div(product["over_refusals"], n)
    unsafe_rate = safe_div(product["unsafe_outputs"], n)
    ux_score = (
        0.25 * success
        + 0.15 * latency_score
        + 0.15 * citation
        + 0.15 * format_stability
        + 0.10 * controllability
        + 0.10 * recovery
        + 0.10 * calibration
        + 0.05 * adoption
        - 0.15 * over_refusal
        - 0.20 * unsafe_rate
    )
    metrics = {
        "success_rate": round(success, 3),
        "adoption_rate": round(adoption, 3),
        "latency_score": round(latency_score, 3),
        "citation_support": round(citation, 3),
        "format_stability": round(format_stability, 3),
        "controllability": round(controllability, 3),
        "recovery_rate": round(recovery, 3),
        "calibration": calibration,
        "over_refusal_rate": round(over_refusal, 3),
        "unsafe_rate": round(unsafe_rate, 3),
        "ux_score": round(clamp01(ux_score), 3),
    }

    failed = []
    if metrics["success_rate"] < 0.72:
        failed.append("task_success")
    if product["p95_latency_ms"] > product["latency_slo_ms"]:
        failed.append("p95_latency")
    if metrics["citation_support"] < 0.80:
        failed.append("citation")
    if metrics["format_stability"] < 0.82:
        failed.append("format")
    if metrics["controllability"] < 0.65:
        failed.append("control")
    if metrics["recovery_rate"] < 0.55:
        failed.append("recovery")
    if metrics["calibration"] < 0.82:
        failed.append("calibration")
    if metrics["over_refusal_rate"] > 0.08:
        failed.append("over_refusal")
    if metrics["unsafe_rate"] > 0.0:
        failed.append("unsafe")

    metrics["experience_gate"] = not failed and metrics["ux_score"] >= 0.72
    metrics["failed_gates"] = failed
    return metrics


audits = {product["name"]: audit(product) for product in PRODUCTS}
ranked = sorted(
    ((name, result["ux_score"], result["experience_gate"]) for name, result in audits.items()),
    key=lambda item: item[1],
    reverse=True,
)

print("ranked=", ranked)
print("ship_experience=", [name for name, result in audits.items() if result["experience_gate"]])
print("needs_rework=", {name: result["failed_gates"] for name, result in audits.items() if not result["experience_gate"]})
for name in [item[0] for item in ranked]:
    print(name, audits[name])
```

这段 demo 体现了本章核心判断：`support_rag_sidebar` 不是因为模型最强，而是因为成功率、引用、格式、恢复和延迟都过线，所以体验门禁通过；`code_copilot_inline` 延迟很好，但成功率还不够；`analyst_agent_report` 价值可能高，但 P95 延迟、引用和校准不足；`generic_chat_entry` 看起来灵活，但缺少明确任务、引用、控制和恢复，体验分最低。

## 3.21 本章小结

大模型产品体验不是模型能力的简单外显，而是模型、系统和交互设计共同塑造的结果。延迟、准确性、可控性、解释性、引用、拒答、风格、错误恢复和用户引导，都会决定用户是否愿意持续使用。
