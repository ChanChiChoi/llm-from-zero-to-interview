# 第二章：用户需求与场景选择

## 0. 本讲资料边界与第二轮精修口径

本讲按 `WRITING_PLAN.md` 的第二轮要求做公式和 demo 精修。联网资料主要核对四类口径：OpenAI evals 与生产评估强调先定义任务、样本、指标和反馈闭环；Google HEART 框架强调把用户目标拆成 goals、signals、metrics；Google SRE 的 SLI / SLO / error budget 口径提醒我们把可靠性和可接受失败率放进上线门禁；NIST AI RMF / Generative AI Profile 提醒我们在场景筛选阶段就评估风险、可控性和治理责任。

本章不讨论具体模型选型、prompt 模板、RAG 架构、Agent 工具协议或企业销售合同细节。这里的核心问题只有一个：在动手做大模型应用前，如何判断一个候选场景是否有真实用户、真实痛点、可衡量收益、可控风险和可试点路径。

大模型落地的第一步不是选模型，而是选场景。场景选错，再强的模型也很难产生价值；场景选对，即使模型方案不复杂，也可能带来明显收益。很多失败的大模型项目，不是技术完全做不出来，而是需求不真实、用户不用、成本不划算或风险不可控。

本章系统讲用户需求与场景选择：如何判断需求是否真实，如何筛选高价值场景，如何评估任务频率、容错率、数据可得性、自动化空间、ROI、组织阻力和落地优先级。对算法工程师来说，这一章能帮助你把项目从“技术可行”讲到“业务值得做”。

## 2.1 场景选择为什么重要

同样的大模型能力，在不同场景中的价值完全不同。

例如“文档总结”：

1. 给用户总结一篇普通新闻，价值有限。
2. 给律师总结合同风险，价值较高。
3. 给客服总结用户历史工单，能节省时间。
4. 给医生总结病历，价值高但风险也高。

技术能力相似，但用户、价值、风险、合规和验收标准完全不同。

面试回答：

```text
大模型项目首先要选对场景。判断一个场景是否值得做，要看用户痛点是否真实、任务频率是否足够、人工成本是否高、模型是否比传统方法更合适、数据是否可得、错误是否可控，以及收益是否覆盖成本。场景选择决定了后续模型、产品和运营设计。
```

## 2.2 真实需求和伪需求

真实需求通常有几个特征：

1. 用户现在已经在花时间或花钱解决。
2. 问题高频或高价值。
3. 用户愿意改变工作流。
4. 解决后有明确收益。
5. 失败后用户会明显不满。

伪需求常见表现：

1. 只是觉得“大模型很酷”。
2. 用户说想要，但实际不用。
3. 没有明确使用场景。
4. 没有可衡量指标。
5. 传统规则已经足够。
6. 只是内部展示项目。

判断需求时，不要只听用户说什么，还要看用户现在怎么做、痛点出现频率、愿意为解决方案付出什么成本。

## 2.3 用户是谁

同一个产品，不同用户的需求不同。

例如企业知识库问答：

1. 一线客服关心快速查到答案。
2. 客服主管关心解决率和转人工率。
3. IT 团队关心权限和系统集成。
4. 法务关心合规。
5. 管理层关心成本收益。

场景选择必须明确主用户和决策者。很多项目失败，是因为使用者觉得麻烦，决策者只看概念，技术团队只看模型指标，三方目标不一致。

## 2.4 用户任务链路

不要只看单点功能，要看完整任务链路。

例如“帮销售写方案”不是一个单步任务。它可能包括：

1. 了解客户行业。
2. 查询历史案例。
3. 选择产品卖点。
4. 生成方案初稿。
5. 人工修改。
6. 审批。
7. 发给客户。

大模型可以介入哪个环节？哪个环节最耗时？哪个环节最容易出错？哪个环节最能衡量收益？这些问题比“模型能不能写方案”更重要。

## 2.5 高频场景和高价值场景

高频场景：每天大量发生，单次价值可能不高，但总收益大。

例如：

1. 客服问答。
2. 工单分类。
3. 文档检索。
4. 会议纪要。
5. 邮件草稿。

高价值场景：发生频率不一定高，但单次价值大。

例如：

1. 合同风险审查。
2. 重大故障排查。
3. 投研报告分析。
4. 代码迁移。
5. 企业知识沉淀。

优先级不只看频率，也看单次收益、风险和落地难度。

## 2.6 容错率

容错率决定产品形态。

高容错场景：

1. 头脑风暴。
2. 文案初稿。
3. 会议摘要草稿。
4. 内部知识探索。

低容错场景：

1. 医疗诊断。
2. 法律结论。
3. 财务报表。
4. 权限修改。
5. 自动下单或支付。

高容错场景可以更自动化；低容错场景需要人审、引用、验证、权限和审计。不能把低容错场景做成完全自动化黑盒。

## 2.7 数据可得性

大模型应用离不开数据。

要评估：

1. 是否有知识库。
2. 数据是否结构化。
3. 数据是否最新。
4. 数据质量如何。
5. 是否有权限访问。
6. 是否包含隐私。
7. 是否能用于评估。
8. 是否有历史反馈。

很多 RAG 项目失败，不是模型不会答，而是知识库混乱、文档过期、权限不清、没有标准答案。

## 2.8 任务是否可验证

可验证性决定落地难度。

容易验证：

1. 代码测试是否通过。
2. 表单字段是否填对。
3. 答案是否来自文档。
4. 分类是否匹配标签。
5. 数据分析结果是否符合计算。

难验证：

1. 创意是否好。
2. 战略建议是否正确。
3. 用户是否满意。
4. 开放式分析是否全面。

可验证场景更适合自动化和规模化；难验证场景更需要人工反馈、评价体系和灰度上线。

## 2.9 大模型是否必要

不是所有智能需求都需要大模型。

可以先问：

1. 规则能否解决？
2. 搜索能否解决？
3. 传统分类模型能否解决？
4. 模板能否解决？
5. 工作流优化能否解决？

如果规则和搜索已经足够，大模型可能增加成本和不确定性。大模型适合处理自然语言、多样表达、长尾问题、生成式任务和复杂上下文。

## 2.10 自动化空间

大模型不一定要完全自动化。可以分层：

1. Assist：辅助用户生成草稿。
2. Recommend：给出建议和候选。
3. Review：帮助检查和发现问题。
4. Automate with approval：自动执行但需确认。
5. Fully automate：完全自动执行。

很多场景适合先做辅助，而不是一开始就全自动。随着数据和信任积累，再逐步提高自动化程度。

## 2.11 ROI 初步估算

场景选择要做粗略 ROI 估算。

可以这样估：

```text
月收益 = 月任务量 * 单次节省时间 * 人力成本 或 单次增量收入
月成本 = 模型调用 + 检索 + 工具 + 人审 + 运维 + 研发摊销
```

如果收益远小于成本，就算技术可行也不值得优先做。

ROI 不是只算钱，也可以算风险降低、客户满意度、交付速度。但最终要能连接到业务目标。

## 2.12 落地难度

落地难度来自多个方面：

1. 数据治理难度。
2. 系统集成难度。
3. 用户习惯改变。
4. 权限和合规要求。
5. 评估标准不清。
6. 延迟要求。
7. 成本约束。
8. 组织协作成本。

一个高价值但落地极难的场景，不一定适合作为第一个项目。早期可以选择价值中等但可快速验证的场景建立信任。

## 2.13 场景优先级矩阵

可以用二维矩阵评估：

```text
业务价值高 + 落地难度低：优先做
业务价值高 + 落地难度高：拆分试点
业务价值低 + 落地难度低：可做 demo 或辅助功能
业务价值低 + 落地难度高：不做
```

更细一点，可以打分：

1. 用户痛点：1 到 5 分。
2. 任务频率：1 到 5 分。
3. 单次价值：1 到 5 分。
4. 数据可得性：1 到 5 分。
5. 可验证性：1 到 5 分。
6. 风险可控性：1 到 5 分。
7. 集成难度：反向 1 到 5 分。
8. 成本可控性：1 到 5 分。

评分不是为了假装精确，而是帮助团队把隐含判断显性化。

## 2.13.1 关键公式与场景选择指标速查

为了让场景选择不只停留在主观讨论，可以把每个候选场景记成一个结构化样本：

```math
s_i=(u_i,j_i,p_i,f_i,v_i,d_i,e_i,l_i,k_i,w_i,a_i,h_i)
```

其中，$u_i$ 是用户和利益相关方，$j_i$ 是任务链路，$p_i$ 是痛点强度，$f_i$ 是月任务频次，$v_i$ 是单次任务价值，$d_i$ 是数据可得性，$e_i$ 是结果可验证性，$l_i$ 是大模型适配度，$k_i$ 是剩余风险，$w_i$ 是工作流适配度，$a_i$ 是自动化层级，$h_i$ 是落地难度。面试中你不一定要背这个 tuple，但要体现“我会把场景拆成用户、任务、价值、数据、验证、风险和工作流”。

真实需求不是一句“用户说想要”，而是痛点、当前替代方案和组织承诺共同成立。可以把痛点粗略归一化为：

```math
P_i=0.4p_i+0.3q_i+0.3r_i
```

其中，$p_i$ 是问题严重度，$q_i$ 是任务紧急度，$r_i$ 是当前方案的不满意程度，三者都归一化到 $[0,1]$。如果用户没有当前替代方案，也没有愿意投入的时间、数据或预算，即使口头兴趣很强，也要降低真实需求判断。

月度收益可以先做粗估：

```math
B_i=f_i(h_i c_i+v_i g_i)
```

其中，$f_i$ 是月任务量，$h_i$ 是单次节省小时数，$c_i$ 是小时成本或机会成本，$v_i$ 是单次任务价值，$g_i$ 是成功率、质量或转化率提升带来的增量比例。这个公式的直觉是：收益来自“省时间”和“提升结果”两部分。

大模型适配度可以写成一个加权分：

```math
L_i=0.35n_i+0.25c_i+0.20g_i+0.20o_i-0.30r_i
```

其中，$n_i$ 表示自然语言表达变化程度，$c_i$ 表示上下文依赖，$g_i$ 表示生成、总结、改写或推理需求，$o_i$ 表示长尾输入比例，$r_i$ 表示规则、搜索或模板已经足够的程度。这个分数提醒你：不是所有智能需求都该上 LLM；如果规则和搜索稳定解决，LLM 可能只是增加成本和不确定性。

为了把“业务价值”和“落地可行性”放在一起，可以把收益压到 $[0,1]$：

```math
V_i=\min(1,\frac{B_i}{B_{\mathrm{ref}}})
```

其中，$B_{\mathrm{ref}}$ 是团队约定的参考收益阈值。然后给候选场景一个优先级分：

```math
S_i=0.25P_i+0.20V_i+0.15L_i+0.15D_i+0.10E_i+0.10W_i+0.05A_i-0.15H_i-0.10K_i
```

其中，$D_i$ 是数据准备度，$E_i$ 是可验证性，$W_i$ 是工作流适配度，$A_i$ 是自动化层级合理性，$H_i$ 是落地难度，$K_i$ 是剩余风险。权重不是固定真理，真实项目可以按行业调整；关键是把“为什么优先做这个场景”讲成可讨论、可复盘的判断。

试点门禁可以写成：

```math
G_{\mathrm{pilot},i}=\mathbb{1}(P_i\ge \tau_p,\ B_i\ge \tau_b,\ L_i\ge \tau_l,\ D_i\ge \tau_d,\ E_i\ge \tau_e,\ K_i\le \tau_k)
```

其中，$\tau_p,\tau_b,\tau_l,\tau_d,\tau_e,\tau_k$ 是团队预先设定的阈值。它表达的不是“分数高就一定成功”，而是“痛点、收益、LLM 适配度、数据、验证和风险至少要同时过基本线，才适合进入试点”。

## 2.14 试点场景

大模型产品通常适合先试点。

好的试点场景：

1. 用户明确。
2. 数据可得。
3. 任务边界清楚。
4. 风险可控。
5. 指标可衡量。
6. 能在短周期看到效果。
7. 成功后可以扩展。

试点不是做玩具 demo，而是用受控范围验证真实价值。

## 2.15 和业务方沟通的问题清单

和业务方沟通时，可以问：

1. 现在这个任务谁在做？
2. 每天或每月发生多少次？
3. 单次大约耗时多久？
4. 当前错误率或投诉点是什么？
5. 成功后希望提升哪个指标？
6. 可接受的错误是什么，不可接受的错误是什么？
7. 是否有历史数据和标准答案？
8. 是否需要审批或合规评估？
9. 用户是否愿意改变工作流？
10. 如果只能辅助一部分，最希望辅助哪一步？

这些问题比直接问“你想要一个 AI 功能吗”更有效。

## 2.16 常见失败模式

1. 没有真实用户。
2. 痛点不够痛。
3. 任务频率太低。
4. 数据不可用或质量差。
5. 风险不可控。
6. 评估指标不清。
7. 用户工作流接不进去。
8. 成本高于收益。
9. 一开始就追求全自动。
10. 技术团队和业务团队目标不一致。

场景选择阶段发现问题，是好事。越早发现不值得做，越能节省成本。

## 2.17 面试题：如何选择大模型落地场景

回答要点：

```text
我会从用户痛点、任务频率、单次价值、数据可得性、可验证性、容错率、成本和风险几个维度评估。优先选择高频或高价值、数据可得、错误可控、指标可衡量、能接入现有工作流的场景。对于高风险场景，不会一开始全自动，而是先做人机协同和试点验证。
```

## 2.18 面试题：如何判断需求是否真实

回答要点：

```text
我不会只听用户说想要什么，而会看他们现在是否已经花时间或成本解决这个问题，任务发生频率如何，现有方案痛点在哪里，成功后能提升什么指标，以及用户是否愿意改变工作流。如果没有明确用户、没有当前成本、没有成功指标，很可能是伪需求或展示型需求。
```

```python
        "data_readiness": 0.35,
        "verifiability": 0.30,
        "risk_control": 0.62,
        "workflow_fit": 0.34,
        "automation_level": "assist",
        "implementation_difficulty": 0.42,
        "current_workaround": False,
        "stakeholder_alignment": 0.35,
    },
]


def reality_score(s):
    observed = 1.0 if s["current_workaround"] else 0.0
    freq_score = min(1.0, s["monthly_frequency"] / 1000.0)
    score = (
        0.35 * s["user_pain"]
        + 0.25 * freq_score
        + 0.20 * s["workflow_fit"]
        + 0.10 * observed
        + 0.10 * s["stakeholder_alignment"]
    )
    return round(score, 3)


def monthly_benefit(s):
    time_value = s["time_saved_hours"] * s["hourly_cost"]
    quality_value = s["value_per_task"] * s["quality_uplift"]
    return round(s["monthly_frequency"] * (time_value + quality_value), 2)


def llm_fit_score(s):
    score = (
        0.35 * s["language_variability"]
        + 0.25 * s["context_need"]
        + 0.20 * s["generation_need"]
        + 0.20 * s["long_tail"]
        - 0.30 * s["rule_sufficiency"]
    )
    return round(clamp01(score), 3)


def audit(s, benefit_ref=20000.0):
    benefit = monthly_benefit(s)
    benefit_score = clamp01(benefit / benefit_ref)
    llm_fit = llm_fit_score(s)
    automation = AUTOMATION_SCORE[s["automation_level"]]
    residual_risk = 1.0 - s["risk_control"]
    priority = (
        0.25 * s["user_pain"]
        + 0.20 * benefit_score
        + 0.15 * llm_fit
        + 0.15 * s["data_readiness"]
        + 0.10 * s["verifiability"]
        + 0.10 * s["workflow_fit"]
        + 0.05 * automation
        - 0.15 * s["implementation_difficulty"]
        - 0.10 * residual_risk
    )
    priority = round(clamp01(priority), 3)
    real_demand = (
        reality_score(s) >= 0.60
        and s["current_workaround"]
        and s["monthly_frequency"] * s["value_per_task"] > 0
    )

    rejected_reasons = []
    if not real_demand:
        rejected_reasons.append("real_demand")
    if s["user_pain"] < 0.60:
        rejected_reasons.append("pain")
    if benefit_score < 0.18:
        rejected_reasons.append("business_value")
    if llm_fit < 0.55:
        rejected_reasons.append("llm_fit")
    if s["data_readiness"] < 0.55:
        rejected_reasons.append("data")
    if s["verifiability"] < 0.55:
        rejected_reasons.append("verifiability")
    if s["risk_control"] < 0.60:
        rejected_reasons.append("risk_control")
    if s["workflow_fit"] < 0.55:
        rejected_reasons.append("workflow")
    if s["implementation_difficulty"] > 0.70:
        rejected_reasons.append("difficulty")

    if not real_demand or s["risk_control"] < 0.45:
        recommendation = "reject"
    elif not rejected_reasons and priority >= 0.70:
        recommendation = "pilot"
    elif priority >= 0.58 and s["user_pain"] >= 0.65 and benefit_score >= 0.35:
        recommendation = "split_pilot"
    elif priority >= 0.43:
        recommendation = "defer"
    else:
        recommendation = "reject"

    return {
        "real_demand": real_demand,
        "monthly_benefit": benefit,
        "benefit_score": round(benefit_score, 3),
        "llm_fit": llm_fit,
        "priority_score": priority,
        "recommendation": recommendation,
        "rejected_reasons": rejected_reasons,
    }


audits = {s["name"]: audit(s) for s in SCENARIOS}
ranked = sorted(
    ((name, a["priority_score"], a["recommendation"]) for name, a in audits.items()),
    key=lambda item: item[1],
    reverse=True,
)

print("ranked=", ranked)
print("pilot_candidates=", [name for name, a in audits.items() if a["recommendation"] == "pilot"])
print("split_pilots=", [name for name, a in audits.items() if a["recommendation"] == "split_pilot"])
print("rejected=", {name: a["rejected_reasons"] for name, a in audits.items() if a["recommendation"] == "reject"})
for name in [item[0] for item in ranked]:
    print(name, audits[name])
```

这段 demo 的关键不是某个权重一定正确，而是训练你形成面试表达：先证明需求真实，再估算收益，再判断 LLM 是否必要，最后把数据、验证、风险、工作流和试点边界合起来做决策。输出中 `support_rag` 适合直接试点，`contract_review` 价值高但风险和难度较高，适合拆成受控试点；`meeting_notes` 暂缓；`refund_automation` 和 `generic_chatbot` 不能直接推进。

## 2.20 本章小结

大模型产品化首先要选对场景。好的场景通常有真实用户、明确痛点、高频或高价值、数据可得、风险可控、结果可验证，并且能接入现有工作流。场景选择不是拍脑袋，而是对用户、任务、数据、成本、风险和 ROI 的综合判断。

下一章会进入“模型能力到产品体验”，讨论如何把模型的理解、生成、检索、推理和工具调用能力，转化为用户真正感受到的产品体验。
