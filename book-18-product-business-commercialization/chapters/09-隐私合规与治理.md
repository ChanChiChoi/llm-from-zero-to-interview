# 第九章：隐私、合规与治理

大模型产品一旦进入真实业务，就会接触用户输入、企业文档、客户资料、对话日志、代码、图片、语音和业务数据。隐私、合规与治理不是上线前补一条免责声明，而是产品架构、数据流、权限系统、日志系统和组织流程的一部分。做不好这些，模型效果再好也难以进入企业核心场景。

本章系统讲大模型产品的隐私合规与治理：PII、数据最小化、权限隔离、日志治理、数据跨境、训练数据使用、模型输出责任、审计、合规流程和治理机制。

## 9.1 为什么隐私合规重要

大模型产品的数据流复杂。

一次简单对话可能涉及：

1. 用户输入。
2. 历史上下文。
3. 检索到的企业文档。
4. 工具调用结果。
5. 模型输出。
6. 日志记录。
7. 反馈数据。
8. 人工审核。

任何一个环节都可能泄露敏感信息。

面试回答：

```text
大模型产品必须重视隐私合规，因为模型会处理用户输入、企业知识、日志、工具结果和生成内容。风险不仅是模型训练泄露数据，还包括 RAG 越权召回、日志保存敏感信息、跨用户上下文混淆、输出不合规内容和外部 API 数据流出。治理要覆盖数据采集、存储、检索、推理、日志、反馈和删除全链路。
```

## 9.2 PII 是什么

PII 是 personally identifiable information，也就是可以识别个人身份的信息。

常见 PII：

1. 姓名。
2. 手机号。
3. 邮箱。
4. 身份证号。
5. 地址。
6. 银行卡号。
7. 医疗记录。
8. 人脸图像。
9. 声纹。
10. 精确位置。

大模型产品要识别 PII，并根据场景做脱敏、拒绝、权限控制或用户确认。

## 9.3 企业敏感数据

企业敏感数据不一定是个人隐私，也可能是商业机密。

例如：

1. 客户名单。
2. 合同价格。
3. 财务数据。
4. 研发代码。
5. 产品路线图。
6. 内部会议纪要。
7. 安全漏洞信息。
8. 供应商协议。

企业 LLM 应用要区分不同数据密级，并和权限系统、审计系统结合。

## 9.4 数据最小化

数据最小化原则是：只收集、使用和保存完成任务所必要的数据。

具体做法：

1. 不需要的字段不传给模型。
2. 能脱敏就脱敏。
3. 能本地处理就不外发。
4. 日志只保存必要摘要。
5. 长期 memory 不保存临时敏感信息。
6. 工具调用只传必要参数。

很多隐私问题来自“为了方便，把所有上下文都传给模型”。产品化时要控制数据流。

## 9.5 数据流梳理

治理前先画清楚数据流。

需要知道：

1. 数据从哪里来。
2. 经过哪些系统。
3. 传给哪个模型或服务。
4. 是否出境或出企业边界。
5. 谁能访问。
6. 保存多久。
7. 是否用于训练或评估。
8. 如何删除。

没有数据流图，就很难做权限、合规和审计。

## 9.6 权限隔离

权限隔离是企业 LLM 的底线。

要求：

1. 用户只能访问有权限的数据。
2. 不同租户隔离。
3. 不同项目隔离。
4. RAG 检索时做权限过滤。
5. Agent 工具调用按用户身份执行。
6. 日志访问也要权限控制。

一个常见错误是只在 UI 层做权限，后端检索和模型上下文仍然混入无权限内容。这会导致隐性数据泄露。

## 9.7 日志治理

日志对调试和评估很重要，但日志也可能泄露信息。

日志可能包含：

1. 用户输入。
2. 模型输出。
3. 检索文档。
4. 工具参数。
5. 错误堆栈。
6. API 返回。
7. 用户反馈。

日志治理要考虑：

1. 是否脱敏。
2. 保存多久。
3. 谁能访问。
4. 是否加密。
5. 是否可删除。
6. 是否用于训练。
7. 是否记录审计。

不要为了排查方便，把完整敏感上下文长期存储。

## 9.8 训练数据使用

用户输入和企业数据是否能用于训练，必须明确。

要回答：

1. 用户是否授权。
2. 数据是否脱敏。
3. 是否用于模型训练、评估或日志分析。
4. 是否会被第三方模型服务使用。
5. 是否能删除。
6. 是否能追踪来源。

企业客户通常非常关心“我的数据会不会被拿去训练其他模型”。产品条款和技术实现必须一致。

## 9.9 外部模型和数据出境

调用外部模型 API 时，要关注数据是否离开企业边界。

需要评估：

1. 服务商是否保存请求。
2. 是否用于训练。
3. 数据存储地区。
4. 是否满足行业合规。
5. 是否支持企业隔离。
6. 是否有审计和删除机制。

有些企业会要求私有化部署或专有云部署，原因往往不是模型效果，而是数据合规。

## 9.10 模型输出责任

模型输出可能产生责任问题。

例如：

1. 错误法律建议。
2. 错误医疗解释。
3. 财务分析误导。
4. 生成侵权内容。
5. 生成不合规营销话术。
6. 错误操作建议导致损失。

产品应明确：模型是辅助工具还是自动决策系统；哪些场景需要人工复核；输出是否有来源和限制说明。

## 9.11 审计

审计是治理的重要部分。

审计记录包括：

1. 谁发起请求。
2. 访问了哪些数据。
3. 使用了哪些模型。
4. 调用了哪些工具。
5. 输出了什么。
6. 是否有人审。
7. 是否触发安全策略。
8. 是否发生异常。

审计不仅用于事后追责，也用于发现系统性风险。

## 9.12 数据保留和删除

产品要定义数据保留策略。

问题包括：

1. 对话保存多久。
2. 日志保存多久。
3. 上传文件是否保留。
4. 用户能否删除。
5. 删除是否影响备份。
6. memory 如何清除。
7. 企业管理员如何管控。

没有删除机制的大模型产品，很难满足隐私和企业治理要求。

## 9.13 合规不是只靠法务

合规需要技术、产品、法务、安全和业务共同参与。

技术团队负责：

1. 数据流控制。
2. 权限隔离。
3. 脱敏。
4. 日志治理。
5. 审计。
6. 安全策略实现。

产品团队负责：

1. 用户告知。
2. 授权入口。
3. 删除入口。
4. 人工复核流程。
5. 风险提示。

法务和安全负责审查要求，但不能替代工程实现。

## 9.14 治理机制

大模型治理机制包括：

1. 数据分级分类。
2. 模型使用规范。
3. Prompt 和工具变更审批。
4. 高风险场景评审。
5. 红队测试。
6. 评估集和回归测试。
7. 日志审计。
8. 事故响应流程。

治理不是为了阻碍创新，而是让大模型应用能进入更高价值、更高风险的业务场景。

## 9.15 常见失败模式

1. 把所有用户输入都长期存日志。
2. RAG 越权召回文档。
3. 多租户数据隔离不彻底。
4. 外部 API 数据使用条款不清。
5. Memory 保存敏感信息。
6. 用户无法删除数据。
7. 模型输出没有风险提示。
8. 高风险建议没有人审。
9. 合规只停留在文档，没有工程实现。
10. 安全事件无法追溯。

隐私合规问题一旦出事，影响通常比模型回答错误更严重。

## 9.16 面试题：大模型产品如何做隐私保护

回答要点：

```text
我会从数据全链路做隐私保护。首先梳理数据流，明确用户输入、检索文档、工具结果、日志和反馈会流向哪里。然后做数据最小化、PII 检测和脱敏、权限隔离、日志治理、加密存储、用户删除机制和审计。对于外部模型 API，要明确数据是否保存、是否用于训练、是否出境。隐私保护不能只靠提示词，要靠系统设计。
```

## 9.17 面试题：企业 RAG 如何避免数据泄露

回答要点：

```text
企业 RAG 要在检索和生成全链路做权限控制。检索时只能召回用户有权限的文档，引用也必须是用户可访问的来源；不能把无权限文档摘要给用户。日志要脱敏，向量索引要按租户或权限隔离，权限变更要及时同步。还要通过审计记录谁访问了哪些内容。
```

## 9.18 本章小结

隐私、合规与治理是大模型产品进入企业和高风险行业的基础。它覆盖 PII、企业敏感数据、数据最小化、权限隔离、日志治理、训练数据使用、外部 API、输出责任、审计、删除和组织治理。

下一章会进入上线运营与反馈闭环，讨论大模型产品上线后如何监控、收集反馈、分析失败、持续迭代和稳定运营。
