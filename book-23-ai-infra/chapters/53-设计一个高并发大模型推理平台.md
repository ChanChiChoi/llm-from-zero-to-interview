# 第 53 章 设计一个高并发大模型推理平台

上一章设计了大模型训练平台。本章继续系统设计题：设计一个高并发大模型推理平台。

这是 AI Infra 面试中最常见的开放题之一。面试官想看你是否理解在线服务、大模型 runtime、KV cache、路由、缓存、限流、扩缩容、灰度发布、可观测性和成本治理之间的关系。

先记住一句话：

> 高并发大模型推理平台不是一个模型 API 服务，而是围绕低延迟、高吞吐、高可用、低成本和多模型治理构建的在线服务平台。

## 53.1 题目理解

题目可能这样问：

1. 请设计一个支持高并发的 ChatGPT 类推理平台。
2. 请设计一个多模型、多租户的大模型服务平台。
3. 请设计一个支持 streaming、自动扩缩容和灰度发布的 LLM serving 系统。
4. 请设计一个企业内部统一模型推理网关。

核心都是：如何把大模型稳定、高效、可控地服务线上请求。

## 53.2 需求澄清

先问清楚：

1. 支持哪些模型？chat、embedding、多模态、rerank？
2. 模型规模多大？7B、70B、MoE？
3. QPS 和并发目标是多少？
4. 输入和输出 token 分布是什么？
5. TTFT、TPOT、p99 SLO 是多少？
6. 是否支持 streaming？
7. 是否多租户？是否需要配额和权限？
8. 是否支持多模型路由？
9. 是否需要缓存？
10. 是否需要灰度、A/B 测试和回滚？
11. 是否有安全策略和审计要求？
12. 成本预算是什么？

推理平台设计必须从 token 分布和 SLO 开始，而不是从 QPS 开始。

## 53.3 核心目标

平台目标：

1. 支持多模型、多版本、多 runtime。
2. 支持高并发和 streaming。
3. 控制 TTFT、TPOT 和 p99。
4. 提高 GPU 利用率和 tokens/s。
5. 支持模型路由、灰度和回滚。
6. 支持缓存、限流、熔断和降级。
7. 支持自动扩缩容和 warm pool。
8. 支持多租户权限、配额和审计。
9. 支持可观测性和故障定位。
10. 支持成本归因和成本优化。

高并发不只是“能抗 QPS”，还要抗 token 负载和长上下文。

## 53.4 总体架构

可以设计如下：

```text
Client / App / Agent
  -> API Gateway
  -> Auth / Quota / Rate Limit
  -> Safety Input Check
  -> Model Router
  -> Cache Layer
  -> Admission Controller
  -> Inference Runtime Cluster
      -> vLLM / TGI / Triton / SGLang / TensorRT-LLM
      -> Runtime Scheduler
      -> Continuous Batching
      -> KV Cache Manager
  -> Model Registry
  -> Deployment Controller
  -> Autoscaler
  -> Observability
  -> Cost / Audit
```

架构要体现两层：平台治理层和 runtime 执行层。

平台治理层负责鉴权、路由、限流、缓存、发布、观测和成本。

Runtime 执行层负责模型加载、prefill、decode、batching 和 KV cache。

## 53.5 API Gateway

API Gateway 负责统一入口。

能力包括：

1. API 认证。
2. 协议适配。
3. 请求大小限制。
4. 基础限流。
5. request ID 注入。
6. streaming 连接管理。
7. 超时控制。
8. 初步日志和审计。

Gateway 不应该做复杂模型选择。

复杂路由应交给 Model Router。

## 53.6 多租户鉴权和配额

多租户推理平台要控制：

1. 谁能调用哪个模型。
2. 每个租户 QPS 上限。
3. 每个租户 token/s 上限。
4. 每个租户并发上限。
5. 每个租户成本预算。
6. 最大上下文长度。
7. 最大输出长度。

LLM 平台限流不能只按 QPS。

一个长上下文请求可能比几十个短请求更贵。

## 53.7 Model Router

Model Router 决定请求去哪。

决策依据：

1. 请求指定模型。
2. 任务能力。
3. 租户权限。
4. 成本预算。
5. 延迟 SLO。
6. 灰度规则。
7. endpoint 健康状态。
8. queue length。
9. KV cache 水位。
10. fallback chain。

Router 输出 route decision，并记录 route trace。

路由不是负载均衡，它还要选模型、版本和降级路径。

## 53.8 Cache Layer

缓存层可以包括：

1. prompt cache。
2. KV cache。
3. semantic cache。
4. result cache。

缓存收益：

1. 降低 TTFT。
2. 减少 prefill 成本。
3. 降低重复请求成本。
4. 提升用户体验。

缓存风险：

1. 权限误命中。
2. 过期答案。
3. 模型版本不一致。
4. 安全策略不一致。

cache key 必须包含模型版本、参数、租户、权限域和数据版本。

## 53.9 Admission Controller

Admission Controller 判断系统当前是否接得住请求。

检查：

1. 队列长度。
2. queue wait time。
3. KV cache 剩余空间。
4. endpoint 健康。
5. 当前 p99。
6. 租户配额。
7. 请求 token 数。
8. timeout budget。

如果接不住，可以：

1. 排队。
2. 降级。
3. 转发到其他 endpoint。
4. 返回限流。

及时拒绝比无限排队更健康。

## 53.10 Runtime Cluster

Runtime Cluster 负责真正执行推理。

可选 runtime：

1. vLLM。
2. TGI。
3. Triton。
4. SGLang。
5. TensorRT-LLM。
6. 自研 runtime。

Runtime 负责：

1. 模型加载。
2. tokenizer。
3. prefill。
4. decode。
5. continuous batching。
6. KV cache。
7. streaming。
8. runtime metrics。

平台层不要把所有逻辑塞进 runtime。

## 53.11 Prefill、Decode 和 KV Cache

系统设计中一定要讲：

1. Prefill 处理输入 prompt，影响 TTFT。
2. Decode 逐 token 生成，影响 TPOT。
3. KV cache 避免重复计算，但占显存。
4. 长上下文和高并发会放大 KV cache 压力。
5. 调度不能只看请求数，要看 token budget。

这是大模型推理平台区别于普通 Web 服务的核心。

## 53.12 Continuous Batching

Continuous batching 让请求在 decode step 动态加入和退出。

收益：

1. 提高 GPU 利用率。
2. 提高 output tokens/s。
3. 更好处理不同输出长度。
4. 降低短请求被长请求拖住的问题。

代价：

1. 调度复杂。
2. KV cache 管理复杂。
3. 指标和 trace 更复杂。

高并发推理平台通常离不开 continuous batching。

## 53.13 队列调度

Runtime queue 不能简单 FIFO。

要考虑：

1. token budget。
2. 输入长度。
3. 预计输出长度。
4. 优先级。
5. 租户公平性。
6. prefill/decode 平衡。
7. 长短请求分队列。

长请求和短请求混跑会导致 p99 变差。

可以对长上下文请求使用专门队列或专门实例。

## 53.14 自动扩缩容

Autoscaler 不能只看 CPU 或 QPS。

应看：

1. QPS。
2. input tokens/s。
3. output tokens/s。
4. queue length。
5. queue wait time。
6. TTFT。
7. TPOT。
8. p99。
9. KV cache usage。
10. GPU utilization。
11. timeout rate。

还要考虑冷启动：镜像、权重、runtime 初始化和 kernel 预热。

因此需要 warm pool 和预测扩容。

## 53.15 稳定性保护

稳定性机制包括：

1. 多维限流。
2. admission control。
3. 熔断。
4. timeout budget。
5. 谨慎重试。
6. 降级路由。
7. fallback chain。
8. endpoint health check。

重试要非常谨慎。

如果请求已经开始生成或已经返回部分 token，重试会导致成本增加和体验不一致。

## 53.16 灰度发布和回滚

推理平台要支持：

1. 模型版本灰度。
2. prompt 灰度。
3. runtime 灰度。
4. 路由策略灰度。
5. 安全策略灰度。
6. 一键回滚。

观察指标：

1. 质量。
2. 安全。
3. TTFT。
4. TPOT。
5. p99。
6. 错误率。
7. 成本。
8. 工具调用成功率。

模型发布不能只看 HTTP 错误率。

## 53.17 可观测性设计

必须记录 metrics、logs、traces、events。

核心 metrics：

1. QPS。
2. TTFT。
3. TPOT。
4. p95/p99。
5. input/output tokens/s。
6. queue length。
7. active sequences。
8. KV cache usage。
9. GPU utilization。
10. cache hit rate。
11. fallback rate。
12. cost per 1k tokens。

Trace 要拆分 gateway、auth、router、cache、queue、prefill、decode、safety 和 streaming。

## 53.18 安全和审计

安全设计包括：

1. API key 和身份认证。
2. 模型访问权限。
3. 租户配额。
4. 输入输出安全检查。
5. 日志脱敏。
6. trace 权限控制。
7. RAG 权限过滤。
8. tool call 权限控制。
9. 审计日志。

推理请求可能包含用户隐私和企业知识，不能把 prompt 和输出无保护地长期保存。

## 53.19 成本治理

成本治理包括：

1. 按租户统计 input/output tokens。
2. 按模型统计 cost per 1k tokens。
3. 记录 cache saved cost。
4. 小模型优先。
5. 分层路由。
6. 输出长度控制。
7. 量化。
8. autoscaling。
9. warm pool 优化。
10. 预算和告警。

高并发推理平台如果没有成本治理，很快会不可持续。

## 53.20 高可用设计

高可用手段：

1. 多副本 runtime。
2. 多 endpoint。
3. health check。
4. 熔断和摘除。
5. fallback model。
6. 多可用区部署。
7. 灰度和回滚。
8. 自动扩缩容。
9. 降级策略。

多地域部署要考虑数据合规、模型权重分发和成本。

## 53.21 状态和数据模型

核心对象包括：

1. Model。
2. ModelVersion。
3. Endpoint。
4. Deployment。
5. RoutePolicy。
6. QuotaPolicy。
7. CachePolicy。
8. ScalingPolicy。
9. SafetyPolicy。
10. RuntimeInstance。

这些对象让平台从“部署服务”升级为“治理模型服务”。

## 53.22 关键 trade-off

需要讲清楚：

1. 吞吐 vs 延迟。
2. GPU 利用率 vs p99。
3. 缓存收益 vs 一致性和安全。
4. 小模型成本低 vs 质量可能下降。
5. warm pool 延迟低 vs 空闲成本高。
6. 灰度速度 vs 发布风险。
7. 完整 trace vs 隐私和成本。
8. 自动重试 vs 重试风暴。

系统设计题中，trade-off 比组件名更重要。

## 53.23 面试回答模板

可以这样回答：

```text
我会先澄清模型规模、QPS、token 分布、TTFT/TPOT/p99 SLO、是否 streaming、多租户、安全和成本要求。

整体架构分为平台治理层和 runtime 执行层。请求先进入 API Gateway，做认证、限流和 request ID 注入，再经过安全检查和 Model Router。Router 根据模型能力、租户权限、成本、SLO、灰度规则和 endpoint 实时负载选择模型版本和 endpoint。缓存层尝试命中 prompt cache、语义缓存或结果缓存，Admission Controller 根据队列、KV cache 和配额判断是否接收请求。

执行层使用 vLLM/TGI/Triton/SGLang 等 runtime，支持 prefill、decode、continuous batching、KV cache 和 streaming。调度上按 token budget、优先级、长短请求和租户公平性做队列管理。

扩缩容不只看 QPS，而结合 input/output tokens/s、queue wait、TTFT、TPOT、KV cache、GPU 和 timeout。稳定性上做限流、熔断、timeout budget、fallback 和降级。发布上支持模型、prompt、runtime 和路由策略灰度与回滚。最后通过 metrics、logs、traces、events、成本归因和审计保证可运营。
```

这段回答覆盖需求、架构、runtime、调度、稳定性、发布和治理。

## 53.24 常见扣分点

扣分点一：只说 Nginx + 模型服务。

问题：忽略模型路由、runtime、KV cache、batching 和治理。

扣分点二：只按 QPS 扩容。

问题：不理解 token 负载和长上下文差异。

扣分点三：不提 TTFT/TPOT。

问题：无法解释用户体验和推理阶段瓶颈。

扣分点四：不提缓存权限。

问题：企业场景中缓存误命中可能导致数据泄露。

扣分点五：发布只看错误率。

问题：模型发布还要看质量、安全、成本和工具调用行为。

## 53.26 本章小结

本章系统设计了一个高并发大模型推理平台。

你需要记住：

1. 推理平台是在线模型服务平台，不只是模型 HTTP API。
2. 高并发设计要围绕 TTFT、TPOT、tokens/s、KV cache、queue 和 p99 展开。
3. 平台治理层负责鉴权、路由、缓存、限流、发布、观测和成本，runtime 层负责 prefill、decode、batching 和 KV cache。
4. 自动扩缩容、稳定性保护、灰度回滚和成本治理是生产推理平台的核心能力。
5. 面试时要讲清请求链路、核心对象、关键指标和 trade-off。
