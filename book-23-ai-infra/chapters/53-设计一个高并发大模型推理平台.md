# 第 53 章 设计一个高并发大模型推理平台

上一章设计了大模型训练平台。本章继续系统设计题：设计一个高并发大模型推理平台。

这是 AI Infra 面试中最常见的开放题之一。面试官想看你是否理解在线服务、大模型 runtime、KV cache、路由、缓存、限流、扩缩容、灰度发布、可观测性和成本治理之间的关系。

先记住一句话：

> 高并发大模型推理平台不是一个模型 API 服务，而是围绕低延迟、高吞吐、高可用、低成本和多模型治理构建的在线服务平台。

## 53.0 本讲资料边界与第二轮精修口径

本讲按 `WRITING_PLAN.md` 的第二轮要求做过资料校准。重点参考的是 Google SRE 对 SLI / SLO、长尾延迟、错误预算和发布风险控制的系统设计口径，OpenTelemetry 对 traces、metrics、logs 和事件关联的观测抽象，Triton Inference Server 对 request、queue、compute、batch 和 GPU 指标的推理服务指标口径，vLLM 对 PagedAttention、KV cache、OpenAI-compatible serving、observability 和 metrics 的公开文档，以及前文第 26 到第 35 章已经校准过的 runtime、prefill / decode、KV cache、continuous batching、模型路由、缓存、扩缩容、保护策略和发布治理口径。

这些资料共同指向一个稳定结论：高并发大模型推理平台不是把模型封装成 HTTP API，也不是把所有逻辑塞进 runtime，而是用平台控制面把需求澄清、SLO、请求链路、模型版本、路由策略、KV 容量、调度、缓存隔离、扩缩容、限流保护、发布回滚、多租户安全、观测、成本和审计统一管理。

本章只抽象截至 2026-06 仍稳定的大模型推理平台系统设计方法，不把某个 runtime、云厂商、Kubernetes 发行版、service mesh、模型仓库或内部平台写成通用标准。正文公式用于面试表达、容量估算和系统设计自查；真实上线仍要用目标模型、硬件、runtime、流量分布、SLO、租户协议、合规要求和历史事故复盘来校准。

## 53.1 题目理解

题目可能这样问：

1. 请设计一个支持高并发的 ChatGPT 类推理平台。
2. 请设计一个多模型、多租户的大模型服务平台。
3. 请设计一个支持 streaming、自动扩缩容和灰度发布的 LLM serving 系统。
4. 请设计一个企业内部统一模型推理网关。

核心都是：如何把大模型稳定、高效、可控地服务线上请求。

## 53.2 需求澄清

先问清楚：

1. 支持哪些模型？chat、embedding、多模态、rerank？
2. 模型规模多大？7B、70B、MoE？
3. QPS 和并发目标是多少？
4. 输入和输出 token 分布是什么？
5. TTFT、TPOT、p99 SLO 是多少？
6. 是否支持 streaming？
7. 是否多租户？是否需要配额和权限？
8. 是否支持多模型路由？
9. 是否需要缓存？
10. 是否需要灰度、A/B 测试和回滚？
11. 是否有安全策略和审计要求？
12. 成本预算是什么？

推理平台设计必须从 token 分布和 SLO 开始，而不是从 QPS 开始。

## 53.3 核心目标

平台目标：

1. 支持多模型、多版本、多 runtime。
2. 支持高并发和 streaming。
3. 控制 TTFT、TPOT 和 p99。
4. 提高 GPU 利用率和 tokens/s。
5. 支持模型路由、灰度和回滚。
6. 支持缓存、限流、熔断和降级。
7. 支持自动扩缩容和 warm pool。
8. 支持多租户权限、配额和审计。
9. 支持可观测性和故障定位。
10. 支持成本归因和成本优化。

高并发不只是“能抗 QPS”，还要抗 token 负载和长上下文。

## 53.4 总体架构

可以设计如下：

```text
Client / App / Agent
  -> API Gateway
  -> Auth / Quota / Rate Limit
  -> Safety Input Check
  -> Model Router
  -> Cache Layer
  -> Admission Controller
  -> Inference Runtime Cluster
      -> vLLM / TGI / Triton / SGLang / TensorRT-LLM
      -> Runtime Scheduler
      -> Continuous Batching
      -> KV Cache Manager
  -> Model Registry
  -> Deployment Controller
  -> Autoscaler
  -> Observability
  -> Cost / Audit
```

架构要体现两层：平台治理层和 runtime 执行层。

平台治理层负责鉴权、路由、限流、缓存、发布、观测和成本。

Runtime 执行层负责模型加载、prefill、decode、batching 和 KV cache。

## 53.5 API Gateway

API Gateway 负责统一入口。

能力包括：

1. API 认证。
2. 协议适配。
3. 请求大小限制。
4. 基础限流。
5. request ID 注入。
6. streaming 连接管理。
7. 超时控制。
8. 初步日志和审计。

Gateway 不应该做复杂模型选择。

复杂路由应交给 Model Router。

## 53.6 多租户鉴权和配额

多租户推理平台要控制：

1. 谁能调用哪个模型。
2. 每个租户 QPS 上限。
3. 每个租户 token/s 上限。
4. 每个租户并发上限。
5. 每个租户成本预算。
6. 最大上下文长度。
7. 最大输出长度。

LLM 平台限流不能只按 QPS。

一个长上下文请求可能比几十个短请求更贵。

## 53.7 Model Router

Model Router 决定请求去哪。

决策依据：

1. 请求指定模型。
2. 任务能力。
3. 租户权限。
4. 成本预算。
5. 延迟 SLO。
6. 灰度规则。
7. endpoint 健康状态。
8. queue length。
9. KV cache 水位。
10. fallback chain。

Router 输出 route decision，并记录 route trace。

路由不是负载均衡，它还要选模型、版本和降级路径。

## 53.8 Cache Layer

缓存层可以包括：

1. prompt cache。
2. KV cache。
3. semantic cache。
4. result cache。

缓存收益：

1. 降低 TTFT。
2. 减少 prefill 成本。
3. 降低重复请求成本。
4. 提升用户体验。

缓存风险：

1. 权限误命中。
2. 过期答案。
3. 模型版本不一致。
4. 安全策略不一致。

cache key 必须包含模型版本、参数、租户、权限域和数据版本。

## 53.9 Admission Controller

Admission Controller 判断系统当前是否接得住请求。

检查：

1. 队列长度。
2. queue wait time。
3. KV cache 剩余空间。
4. endpoint 健康。
5. 当前 p99。
6. 租户配额。
7. 请求 token 数。
8. timeout budget。

如果接不住，可以：

1. 排队。
2. 降级。
3. 转发到其他 endpoint。
4. 返回限流。

及时拒绝比无限排队更健康。

## 53.10 Runtime Cluster

Runtime Cluster 负责真正执行推理。

可选 runtime：

1. vLLM。
2. TGI。
3. Triton。
4. SGLang。
5. TensorRT-LLM。
6. 自研 runtime。

Runtime 负责：

1. 模型加载。
2. tokenizer。
3. prefill。
4. decode。
5. continuous batching。
6. KV cache。
7. streaming。
8. runtime metrics。

平台层不要把所有逻辑塞进 runtime。

## 53.11 Prefill、Decode 和 KV Cache

系统设计中一定要讲：

1. Prefill 处理输入 prompt，影响 TTFT。
2. Decode 逐 token 生成，影响 TPOT。
3. KV cache 避免重复计算，但占显存。
4. 长上下文和高并发会放大 KV cache 压力。
5. 调度不能只看请求数，要看 token budget。

这是大模型推理平台区别于普通 Web 服务的核心。

## 53.12 Continuous Batching

Continuous batching 让请求在 decode step 动态加入和退出。

收益：

1. 提高 GPU 利用率。
2. 提高 output tokens/s。
3. 更好处理不同输出长度。
4. 降低短请求被长请求拖住的问题。

代价：

1. 调度复杂。
2. KV cache 管理复杂。
3. 指标和 trace 更复杂。

高并发推理平台通常离不开 continuous batching。

## 53.13 队列调度

Runtime queue 不能简单 FIFO。

要考虑：

1. token budget。
2. 输入长度。
3. 预计输出长度。
4. 优先级。
5. 租户公平性。
6. prefill/decode 平衡。
7. 长短请求分队列。

长请求和短请求混跑会导致 p99 变差。

可以对长上下文请求使用专门队列或专门实例。

## 53.14 自动扩缩容

Autoscaler 不能只看 CPU 或 QPS。

应看：

1. QPS。
2. input tokens/s。
3. output tokens/s。
4. queue length。
5. queue wait time。
6. TTFT。
7. TPOT。
8. p99。
9. KV cache usage。
10. GPU utilization。
11. timeout rate。

还要考虑冷启动：镜像、权重、runtime 初始化和 kernel 预热。

因此需要 warm pool 和预测扩容。

## 53.15 稳定性保护

稳定性机制包括：

1. 多维限流。
2. admission control。
3. 熔断。
4. timeout budget。
5. 谨慎重试。
6. 降级路由。
7. fallback chain。
8. endpoint health check。

重试要非常谨慎。

如果请求已经开始生成或已经返回部分 token，重试会导致成本增加和体验不一致。

## 53.16 灰度发布和回滚

推理平台要支持：

1. 模型版本灰度。
2. prompt 灰度。
3. runtime 灰度。
4. 路由策略灰度。
5. 安全策略灰度。
6. 一键回滚。

观察指标：

1. 质量。
2. 安全。
3. TTFT。
4. TPOT。
5. p99。
6. 错误率。
7. 成本。
8. 工具调用成功率。

模型发布不能只看 HTTP 错误率。

## 53.17 可观测性设计

必须记录 metrics、logs、traces、events。

核心 metrics：

1. QPS。
2. TTFT。
3. TPOT。
4. p95/p99。
5. input/output tokens/s。
6. queue length。
7. active sequences。
8. KV cache usage。
9. GPU utilization。
10. cache hit rate。
11. fallback rate。
12. cost per 1k tokens。

Trace 要拆分 gateway、auth、router、cache、queue、prefill、decode、safety 和 streaming。

## 53.18 安全和审计

安全设计包括：

1. API key 和身份认证。
2. 模型访问权限。
3. 租户配额。
4. 输入输出安全检查。
5. 日志脱敏。
6. trace 权限控制。
7. RAG 权限过滤。
8. tool call 权限控制。
9. 审计日志。

推理请求可能包含用户隐私和企业知识，不能把 prompt 和输出无保护地长期保存。

## 53.19 成本治理

成本治理包括：

1. 按租户统计 input/output tokens。
2. 按模型统计 cost per 1k tokens。
3. 记录 cache saved cost。
4. 小模型优先。
5. 分层路由。
6. 输出长度控制。
7. 量化。
8. autoscaling。
9. warm pool 优化。
10. 预算和告警。

高并发推理平台如果没有成本治理，很快会不可持续。

## 53.20 高可用设计

高可用手段：

1. 多副本 runtime。
2. 多 endpoint。
3. health check。
4. 熔断和摘除。
5. fallback model。
6. 多可用区部署。
7. 灰度和回滚。
8. 自动扩缩容。
9. 降级策略。

多地域部署要考虑数据合规、模型权重分发和成本。

核心对象包括：

## 53.22 推理平台系统设计指标和最小 demo

面试中讲高并发推理平台，最好把“组件架构”收束成几个可计算指标。一个推理平台系统设计样本可以抽象为：

```math
D_i=(q_i,a_i,l_i,r_i,k_i,c_i,s_i,p_i,g_i,o_i,u_i,h_i,z_i)
```

其中，`q_i` 是需求和流量画像，`a_i` 是入口、鉴权和配额，`l_i` 是请求生命周期，`r_i` 是模型路由和发布，`k_i` 是 runtime、prefill / decode 和 KV cache，`c_i` 是缓存策略，`s_i` 是准入和调度，`p_i` 是限流、熔断、超时和降级，`g_i` 是多租户安全治理，`o_i` 是观测链路，`u_i` 是成本容量，`h_i` 是高可用和回滚，`z_i` 是未关闭风险。

某个设计维度的覆盖率可以写成：

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(D_i)=1]
```

其中，`g_j` 是第 `j` 个检查函数，例如是否给出 token 分布、是否区分 prefill / decode、是否估算 KV 容量、是否给出 trace 和成本归因。这个公式的直觉是：系统设计不是“讲到了一个词”，而是每个关键维度都能被样本化检查。

端到端延迟可以拆成：

```math
T_{\mathrm{e2e}}=
T_{\mathrm{gateway}}+
T_{\mathrm{auth}}+
T_{\mathrm{route}}+
T_{\mathrm{cache}}+
T_{\mathrm{queue}}+
T_{\mathrm{prefill}}+
T_{\mathrm{decode}}+
T_{\mathrm{safety}}+
T_{\mathrm{stream}}
```

TTFT 主要到首 token 生成前为止：

```math
T_{\mathrm{ttft}}=
T_{\mathrm{gateway}}+
T_{\mathrm{auth}}+
T_{\mathrm{route}}+
T_{\mathrm{cache}}+
T_{\mathrm{queue}}+
T_{\mathrm{prefill}}
```

Decode 阶段可以用输出 token 数近似：

```math
T_{\mathrm{decode}}\approx y_iT_{\mathrm{tpot}}
```

其中，`y_i` 是第 `i` 个请求输出 token 数，`T_{\mathrm{tpot}}` 是每个输出 token 的平均或分位延迟。这个近似提醒你：长输出会把总延迟拉长，即使 TTFT 看起来很好，p99 也可能失败。

token 负载要拆成输入和输出：

```math
\Lambda_{\mathrm{in}}=\lambda_{\mathrm{req}}\mathbb{E}[x_i],
\qquad
\Lambda_{\mathrm{out}}=\lambda_{\mathrm{req}}\mathbb{E}[y_i],
\qquad
\Lambda_{\mathrm{tok}}=\Lambda_{\mathrm{in}}+\Lambda_{\mathrm{out}}
```

其中，`\lambda_{\mathrm{req}}` 是请求到达率，`x_i` 是输入 token 数，`y_i` 是输出 token 数。推理平台按 QPS 扩容会漏掉长上下文和长输出的真实负载。

KV cache 单请求显存可以粗略估算为：

```math
M_{\mathrm{kv,req}}=2LH_{\mathrm{kv}}D_hB_{\mathrm{elem}}S_{\mathrm{ctx}}
```

其中，`L` 是层数，`H_{\mathrm{kv}}` 是 KV head 数，`D_h` 是 head dimension，`B_{\mathrm{elem}}` 是每个元素字节数，`S_{\mathrm{ctx}}` 是输入加已生成输出的上下文 token 数。前面的 `2` 表示 key 和 value 两份缓存。

KV 容量门禁可以写成：

```math
G_{\mathrm{kv}}=\mathbf{1}\left[N_{\mathrm{active}}M_{\mathrm{kv,req}}\le \eta_{\mathrm{kv}}M_{\mathrm{gpu}}\right]
```

其中，`N_{\mathrm{active}}` 是活跃请求数，`M_{\mathrm{gpu}}` 是可用显存，`\eta_{\mathrm{kv}}` 是留给 KV 的比例。真实平台还要扣除权重、workspace、fragmentation、runtime overhead 和不同请求长度造成的 block waste。

多指标副本需求可以用最大值收束：

```math
N_{\mathrm{need}}=\max\left(
\left\lceil\frac{\lambda_{\mathrm{req}}}{\mu_{\mathrm{req}}}\right\rceil,
\left\lceil\frac{\Lambda_{\mathrm{in}}}{\mu_{\mathrm{in}}}\right\rceil,
\left\lceil\frac{\Lambda_{\mathrm{out}}}{\mu_{\mathrm{out}}}\right\rceil,
\left\lceil\frac{N_{\mathrm{kv}}}{N_{\mathrm{kv,one}}}\right\rceil
\right)
```

其中，`\mu_{\mathrm{req}}`、`\mu_{\mathrm{in}}`、`\mu_{\mathrm{out}}` 分别是单实例请求、输入 token 和输出 token 能力，`N_{\mathrm{kv}}` 是需要承载的 KV 并发，`N_{\mathrm{kv,one}}` 是单实例 KV 并发上限。这个公式比“QPS 除以单机 QPS”更接近 LLM serving 的真实容量规划。

最终系统设计门禁可以写成：

```math
G_{\mathrm{design}}=\mathbf{1}\left[
\min_j C_j\ge\tau_j \land
T_{\mathrm{ttft,p95}}\le B_{\mathrm{ttft}} \land
T_{\mathrm{tpot,p95}}\le B_{\mathrm{tpot}} \land
G_{\mathrm{kv}}=1 \land
R_{\mathrm{trace}}\ge\rho \land
P_0=0
\right]
```

其中，`C_j` 是各设计维度覆盖率，`R_{\mathrm{trace}}` 是请求链路 trace 覆盖率，`P_0` 是未关闭 P0 风险。这个门禁强调：系统设计不是组件列表，而是可验收的工程闭环。

下面这个 0 依赖 Python demo 演示一个“推理平台系统设计答案审计器”：输入 toy design answers，检查是否覆盖需求、请求链路、SLO、模型注册、路由、runtime、KV、缓存、扩缩容、保护、发布、多租户、观测、成本和 trade-off，并给出容量估算样例。

```python
from math import ceil


class MiniInferencePlatformSystemDesignAudit:
    CHECKS = [
        "requirement_clarification",
        "request_lifecycle_coverage",
        "slo_token_contract",
        "model_registry_release",
        "router_policy_fit",
        "runtime_prefill_decode",
        "kv_scheduler_capacity",
        "cache_isolation_strategy",
        "autoscaling_capacity_plan",
        "protection_policy_design",
        "release_governance_design",
        "tenant_security_governance",
        "observability_trace_readiness",
        "cost_capacity_governance",
        "tradeoff_boundary_reasoning",
        "inference_platform_design_gate",
    ]

    def __init__(self):
        self.traffic = {
            "peak_qps": 42,
            "avg_input_tokens": 1400,
            "avg_output_tokens": 180,
            "p95_input_tokens": 6000,
            "p95_output_tokens": 640,
            "active_sequences": 96,
            "ttft_budget_s": 1.8,
            "tpot_budget_ms": 45,
            "p99_budget_s": 12.0,
        }
        self.instance = {
            "req_capacity": 7,
            "input_tps": 9000,
            "output_tps": 900,
            "kv_concurrency": 18,
            "gpu_memory_gib": 80,
            "kv_memory_fraction": 0.30,
        }
        self.model = {
            "layers": 40,
            "kv_heads": 8,
            "head_dim": 128,
            "bytes_per_element": 2,
            "context_tokens": self.traffic["p95_input_tokens"] + self.traffic["p95_output_tokens"],
        }
        self.latency = {
            "gateway_s": 0.03,
            "auth_s": 0.02,
            "route_s": 0.04,
            "cache_s": 0.03,
            "queue_s": 0.45,
            "prefill_s": 0.78,
            "tpot_ms": 34,
            "safety_s": 0.08,
            "stream_s": 0.15,
        }

    def kv_per_request_mib(self):
        total_bytes = (
            2
            * self.model["layers"]
            * self.model["kv_heads"]
            * self.model["head_dim"]
            * self.model["bytes_per_element"]
            * self.model["context_tokens"]
        )
        return round(total_bytes / (1024 ** 2), 1)

    def latency_summary(self):
        decode_s = self.traffic["p95_output_tokens"] * self.latency["tpot_ms"] / 1000
        ttft_s = (
            self.latency["gateway_s"]
            + self.latency["auth_s"]
            + self.latency["route_s"]
            + self.latency["cache_s"]
            + self.latency["queue_s"]
            + self.latency["prefill_s"]
        )
        e2e_s = ttft_s + decode_s + self.latency["safety_s"] + self.latency["stream_s"]
        return {"ttft_s": round(ttft_s, 3), "decode_s": round(decode_s, 3), "e2e_s": round(e2e_s, 3)}

    def capacity_plan(self):
        input_tps = self.traffic["peak_qps"] * self.traffic["avg_input_tokens"]
        output_tps = self.traffic["peak_qps"] * self.traffic["avg_output_tokens"]
        by_req = ceil(self.traffic["peak_qps"] / self.instance["req_capacity"])
        by_input = ceil(input_tps / self.instance["input_tps"])
        by_output = ceil(output_tps / self.instance["output_tps"])
        by_kv = ceil(self.traffic["active_sequences"] / self.instance["kv_concurrency"])
        return {
            "input_tps": input_tps,
            "output_tps": output_tps,
            "by_req": by_req,
            "by_input": by_input,
            "by_output": by_output,
            "by_kv": by_kv,
            "replicas_needed": max(by_req, by_input, by_output, by_kv),
        }

    def kv_gate(self, active_sequences_per_replica):
        kv_mib = self.kv_per_request_mib()
        available_mib = self.instance["gpu_memory_gib"] * 1024 * self.instance["kv_memory_fraction"]
        used_mib = kv_mib * active_sequences_per_replica
        return {
            "kv_per_request_mib": kv_mib,
            "available_kv_mib": round(available_mib, 1),
            "used_kv_mib": round(used_mib, 1),
            "kv_gate_pass": used_mib <= available_mib,
        }

    def build_design_cases(self):
        complete = {"name": "complete_inference_platform_design"}
        complete.update({check: True for check in self.CHECKS})
        cases = [complete]
        bad_names = [
            "requirements_missing_bad",
            "request_lifecycle_missing_bad",
            "slo_token_contract_missing_bad",
            "model_registry_release_missing_bad",
            "router_policy_missing_bad",
            "runtime_phase_missing_bad",
            "kv_scheduler_missing_bad",
            "cache_isolation_missing_bad",
            "autoscaling_single_metric_bad",
            "protection_policy_missing_bad",
            "release_governance_missing_bad",
            "tenant_security_missing_bad",
            "observability_trace_missing_bad",
            "cost_capacity_missing_bad",
            "tradeoff_boundary_missing_bad",
            "design_gate_missing_bad",
        ]
        for bad_name, failed_check in zip(bad_names, self.CHECKS):
            case = {"name": bad_name}
            case.update({check: True for check in self.CHECKS})
            case[failed_check] = False
            cases.append(case)
        return cases

    def audit_cases(self, cases):
        metrics = {
            check: round(sum(1 for case in cases if case.get(check)) / len(cases), 3)
            for check in self.CHECKS
        }
        failed_cases = [case["name"] for case in cases if not all(case.get(check) for check in self.CHECKS)]
        failed_gates = [check for check, value in metrics.items() if value < 1.0]
        hard_blockers = sum(1 for case in cases if case["name"].endswith("_bad"))
        gate_pass = min(metrics.values()) >= 0.95 and hard_blockers == 0
        return metrics, failed_cases, failed_gates, hard_blockers, gate_pass

    def run(self):
        latency = self.latency_summary()
        capacity = self.capacity_plan()
        active_per_replica = ceil(self.traffic["active_sequences"] / capacity["replicas_needed"])
        kv = self.kv_gate(active_sequences_per_replica=active_per_replica)
        metrics, failed_cases, failed_gates, hard_blockers, gate_pass = self.audit_cases(
            self.build_design_cases()
        )
        slo_pass = {
            "ttft": latency["ttft_s"] <= self.traffic["ttft_budget_s"],
            "tpot": self.latency["tpot_ms"] <= self.traffic["tpot_budget_ms"],
            "p99": latency["e2e_s"] <= self.traffic["p99_budget_s"],
        }
        return {
            "latency_summary": latency,
            "capacity_plan": capacity,
            "kv_gate": kv,
            "slo_pass": slo_pass,
            "metrics": metrics,
            "hard_blocker_count": hard_blockers,
            "failed_case_count": len(failed_cases),
            "failed_gate_count": len(failed_gates),
            "failed_case_sample": failed_cases[:4],
            "failed_gate_sample": failed_gates[:4],
            "inference_platform_design_gate_pass": gate_pass,
        }
```

## 53.23 关键 trade-off

需要讲清楚：

1. 吞吐 vs 延迟。
2. GPU 利用率 vs p99。
3. 缓存收益 vs 一致性和安全。
4. 小模型成本低 vs 质量可能下降。
5. warm pool 延迟低 vs 空闲成本高。
6. 灰度速度 vs 发布风险。
7. 完整 trace vs 隐私和成本。
8. 自动重试 vs 重试风暴。

系统设计题中，trade-off 比组件名更重要。

## 53.25 常见扣分点

扣分点一：只说 Nginx + 模型服务。

问题：忽略模型路由、runtime、KV cache、batching 和治理。

扣分点二：只按 QPS 扩容。

问题：不理解 token 负载和长上下文差异。

扣分点三：不提 TTFT/TPOT。

问题：无法解释用户体验和推理阶段瓶颈。

扣分点四：不提缓存权限。

问题：企业场景中缓存误命中可能导致数据泄露。

扣分点五：发布只看错误率。

问题：模型发布还要看质量、安全、成本和工具调用行为。

## 53.27 本章小结

本章系统设计了一个高并发大模型推理平台。

你需要记住：

1. 推理平台是在线模型服务平台，不只是模型 HTTP API。
2. 高并发设计要围绕 TTFT、TPOT、tokens/s、KV cache、queue 和 p99 展开。
3. 平台治理层负责鉴权、路由、缓存、限流、发布、观测和成本，runtime 层负责 prefill、decode、batching 和 KV cache。
4. 自动扩缩容、稳定性保护、灰度回滚和成本治理是生产推理平台的核心能力。
5. 面试时要讲清请求链路、核心对象、关键指标和 trade-off。
