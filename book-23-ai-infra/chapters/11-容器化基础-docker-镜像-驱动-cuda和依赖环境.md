# 第 11 章 容器化基础：Docker、镜像、驱动、CUDA 和依赖环境

前面我们讲了 GPU 集群、网络、存储和 checkpoint。本章讲一个训练平台和推理平台每天都会遇到的问题：环境。

大模型项目里，很多故障不是算法错了，而是环境不一致：CUDA 版本不对、驱动不匹配、PyTorch 版本不同、NCCL 配置变化、Python 依赖漂移、系统库缺失、镜像里混入了临时修改。

容器化就是为了解决这些问题的一种基础能力。

先记住一句话：

> 容器化的核心价值不是“把程序打包”，而是把训练和推理环境变成可复现、可分发、可调度、可审计的运行单元。

## 11.1 为什么 AI Infra 需要容器化

没有容器化时，训练任务可能直接跑在机器环境里。

这会带来很多问题：

1. 不同机器 Python 版本不同。
2. CUDA 版本不同。
3. PyTorch 版本不同。
4. 系统库版本不同。
5. 用户手工安装依赖互相污染。
6. 任务迁移到另一台机器后跑不起来。
7. 环境修改无法追踪。
8. 线上推理和离线训练环境不一致。

容器化可以把代码运行需要的用户态环境打包起来，让任务在不同机器上更一致地运行。

在 AI Infra 里，容器化支撑：

1. 训练任务提交。
2. 推理服务部署。
3. 评估任务运行。
4. 多租户隔离。
5. 依赖复现。
6. 镜像安全扫描。
7. 灰度发布和回滚。

## 11.2 Docker 是什么

Docker 是常见的容器技术。

你可以把 Docker 镜像理解成一个打包好的运行环境。

它通常包含：

1. 基础操作系统文件。
2. Python 或其他语言运行时。
3. CUDA 用户态库。
4. 深度学习框架。
5. 系统依赖。
6. Python 包。
7. 训练或推理代码。
8. 启动脚本。

容器运行时会从镜像启动容器。

容器不是虚拟机。它共享宿主机内核，但有自己的文件系统、进程空间、环境变量和资源限制。

## 11.3 镜像、容器和仓库

几个概念要区分。

镜像 image：静态包，包含运行环境。

容器 container：镜像运行起来后的进程环境。

镜像仓库 registry：存放镜像的服务。

tag：镜像标签，例如 `train:cuda12.1-pytorch2.2`。

digest：镜像内容哈希，比 tag 更可靠。

为什么 digest 重要？

因为 tag 可以被覆盖。

例如今天 `latest` 指向一个镜像，明天可能指向另一个镜像。为了复现训练，最好记录镜像 digest。

## 11.4 镜像分层

Docker 镜像由多层组成。

例如：

```text
base OS
  -> CUDA runtime
  -> PyTorch
  -> system dependencies
  -> python packages
  -> project code
  -> startup scripts
```

分层的好处：

1. 复用公共层。
2. 加快构建和分发。
3. 减少重复存储。
4. 便于组织基础镜像和业务镜像。

AI 平台常见做法：

1. 平台维护基础镜像。
2. 算法团队基于基础镜像构建项目镜像。
3. 推理平台维护 serving 镜像。
4. 安全团队扫描镜像漏洞。

不要让每个项目从零构建所有依赖，否则镜像会混乱且难维护。

## 11.5 GPU 容器和普通容器的区别

普通容器主要使用 CPU、内存和文件系统。

GPU 容器还需要访问宿主机 GPU 设备。

这涉及：

1. NVIDIA driver。
2. CUDA 用户态库。
3. GPU device file。
4. container runtime hook。
5. NCCL 和通信库。
6. MIG 或 GPU 分配策略。

一个关键点：驱动通常在宿主机上，容器里放的是用户态 CUDA 库和框架。

因此 GPU 容器不是完全自包含的，它仍然依赖宿主机驱动。

## 11.6 驱动、CUDA 和 PyTorch 的关系

这是 AI 环境最容易出错的地方。

可以这样理解：

1. NVIDIA Driver：宿主机驱动，负责和 GPU 硬件交互。
2. CUDA Driver API：驱动接口。
3. CUDA Runtime / Toolkit：用户态 CUDA 库和开发工具。
4. cuDNN、cuBLAS、NCCL：深度学习常用库。
5. PyTorch / TensorFlow：深度学习框架。

版本关系很重要。

常见问题：

1. 宿主机驱动太旧，不支持容器里的 CUDA。
2. PyTorch 编译的 CUDA 版本和运行环境不兼容。
3. NCCL 版本不匹配导致通信异常。
4. cuDNN 版本变化导致性能或行为差异。
5. 容器里有多个 CUDA 路径，实际加载了错误版本。

面试中可以用一句话总结：

```text
容器可以打包用户态依赖，但不能忽略宿主机 GPU 驱动和内核环境；CUDA、驱动、框架和通信库需要形成兼容矩阵。
```

## 11.7 CUDA 兼容性

CUDA 兼容性不是简单“版本越新越好”。

需要考虑：

1. GPU 架构是否支持。
2. 驱动版本是否支持。
3. PyTorch 是否支持。
4. 自定义 CUDA kernel 是否兼容。
5. Triton kernel 是否兼容。
6. NCCL 是否兼容。
7. 推理 runtime 是否兼容。

平台常见做法：

1. 维护少数标准 CUDA 版本。
2. 建立驱动、CUDA、PyTorch、NCCL 兼容矩阵。
3. 禁止随意使用未经验证的组合。
4. 新版本先灰度验证。
5. 保留旧版本镜像用于复现历史实验。

标准化比“大家自由装”更可靠。

## 11.8 依赖管理

AI 项目依赖很多。

包括：

1. Python 版本。
2. PyTorch / TensorFlow。
3. Transformers。
4. tokenizers。
5. DeepSpeed。
6. Megatron。
7. vLLM / TGI / SGLang。
8. Triton。
9. FlashAttention。
10. xFormers。
11. NCCL。
12. 系统库。

依赖管理常见问题：

1. 没有锁版本。
2. 依赖自动升级导致行为变化。
3. 开发环境和训练环境不同。
4. 训练环境和推理环境不同。
5. 镜像构建不可复现。
6. 私有包版本缺失。

建议：

1. 锁定关键依赖版本。
2. 记录镜像 digest。
3. 使用可复现构建流程。
4. 训练、评估、推理环境分别管理。
5. 对基础镜像做版本化。
6. 镜像构建后跑 smoke test。

## 11.9 训练镜像和推理镜像的区别

训练镜像和推理镜像不一定一样。

训练镜像通常包含：

1. 训练框架。
2. 分布式训练库。
3. 数据处理工具。
4. 实验追踪 SDK。
5. 调试工具。
6. profiler。
7. 编译工具。

推理镜像通常更关注：

1. serving runtime。
2. 模型加载。
3. 高性能推理库。
4. 安全加固。
5. 镜像体积。
6. 冷启动速度。
7. 稳定性。

训练镜像可以相对重一些，因为它需要调试和实验。

推理镜像应该尽量精简、可审计、启动快。

不要直接把实验训练镜像拿去线上推理。

## 11.10 镜像体积问题

AI 镜像很容易变得巨大。

原因：

1. CUDA 库大。
2. PyTorch 大。
3. 编译工具多。
4. Python 依赖多。
5. 缓存没有清理。
6. 模型权重误打进镜像。
7. 多版本依赖共存。

镜像过大带来的问题：

1. 拉取慢。
2. 任务启动慢。
3. 推理扩容慢。
4. 镜像仓库存储成本高。
5. 节点磁盘压力大。

优化方法：

1. 使用基础镜像分层。
2. 清理构建缓存。
3. 多阶段构建。
4. 不把大模型权重打进镜像。
5. 训练和推理镜像拆分。
6. 只保留必要依赖。
7. 节点预拉取常用镜像。

## 11.11 镜像构建流水线

成熟平台应该有镜像构建流水线。

典型流程：

```text
Code / Dockerfile
  -> Build
  -> Dependency Lock Check
  -> Unit Test / Smoke Test
  -> Security Scan
  -> Push Registry
  -> Record Digest
  -> Deploy / Submit Job
```

Smoke test 可以包括：

1. Python 能否启动。
2. CUDA 是否可用。
3. PyTorch 是否能看到 GPU。
4. NCCL 简单通信是否正常。
5. 关键包是否能 import。
6. 模型小样本推理是否成功。

镜像不是构建成功就能用，必须验证能在目标集群运行。

## 11.12 镜像安全

镜像是供应链安全的重要入口。

风险包括：

1. 基础镜像有漏洞。
2. Python 包有漏洞。
3. 镜像里包含密钥。
4. 镜像里包含敏感数据。
5. 使用未审核第三方包。
6. 镜像 tag 被覆盖。
7. 用户以 root 权限运行容器。
8. 容器逃逸风险。

安全措施：

1. 镜像漏洞扫描。
2. 禁止把密钥写进镜像。
3. 使用镜像签名。
4. 记录 digest。
5. 限制 root 权限。
6. 基础镜像定期更新。
7. 私有 registry 权限控制。
8. 构建日志审计。

AI 镜像经常包含大量第三方包，更需要供应链治理。

## 11.13 容器和多租户隔离

容器提供一定隔离，但不是强安全边界。

隔离包括：

1. 文件系统隔离。
2. 进程隔离。
3. 网络隔离。
4. 资源限制。
5. 环境变量隔离。

但 GPU、多租户和高权限挂载会带来额外风险。

例如：

1. 容器挂载了宿主机目录。
2. 容器以 privileged 模式运行。
3. 多租户共享同一 GPU。
4. 任务能访问不该访问的数据卷。
5. 环境变量泄露 token。

因此容器化要配合 Kubernetes、权限系统、密钥管理、网络策略和审计一起使用。

## 11.14 环境可复现性

训练复现需要记录环境。

至少记录：

1. 镜像 digest。
2. Python 版本。
3. CUDA 版本。
4. PyTorch 版本。
5. NCCL 版本。
6. 关键包版本。
7. 驱动版本。
8. GPU 型号。
9. 启动命令。
10. 环境变量。
11. 代码 commit。
12. 数据版本。

没有这些信息，几个月后很难复现实验。

实验追踪平台应该自动记录环境 metadata，而不是靠人手写。

## 11.15 常见环境问题

### 11.15.1 容器里看不到 GPU

可能原因：

1. 没有正确配置 GPU runtime。
2. Kubernetes 没有分配 GPU。
3. device plugin 异常。
4. 驱动问题。
5. 容器权限问题。

### 11.15.2 CUDA 版本不匹配

可能原因：

1. 宿主机驱动太旧。
2. 容器 CUDA 版本太新。
3. PyTorch 编译版本不匹配。
4. 系统加载了错误 CUDA 路径。

### 11.15.3 NCCL 通信失败

可能原因：

1. NCCL 版本不兼容。
2. 网卡配置不正确。
3. 容器网络限制。
4. RDMA 设备没有挂载进容器。
5. 环境变量配置错误。

### 11.15.4 本地能跑，集群不能跑

可能原因：

1. 本地环境和镜像不同。
2. 依赖没有写入镜像。
3. 使用了本地绝对路径。
4. 数据挂载路径不同。
5. 集群驱动和 CUDA 组合不同。
6. 启动命令或环境变量不同。

## 11.16 面试中如何回答容器化设计

如果面试官问：

```text
大模型训练平台为什么需要容器化？如何设计镜像体系？
```

可以这样回答：

```text
容器化的核心是让训练和推理环境可复现、可分发、可调度和可审计。大模型环境依赖复杂，包括 CUDA、cuDNN、NCCL、PyTorch、DeepSpeed、FlashAttention、Triton 和各种系统库，如果直接在宿主机安装，很容易出现环境漂移和多租户污染。

我会设计分层镜像体系：平台团队维护经过验证的基础镜像，包含 OS、CUDA、cuDNN、NCCL 和 PyTorch；算法团队基于基础镜像添加项目代码和依赖；推理平台使用更精简、安全、启动快的 serving 镜像。所有镜像记录 digest，而不是只依赖 tag。

镜像构建流程要包含依赖锁定、smoke test、GPU 可用性测试、NCCL 测试、安全扫描、签名和 registry 权限控制。训练任务提交时记录镜像 digest、驱动版本、CUDA 版本、代码 commit、数据版本和启动命令，保证后续可复现。
```

## 11.17 常见误区

误区一：用了 Docker 就完全可复现。

容器仍依赖宿主机驱动、GPU、内核和设备挂载。还需要记录镜像 digest、驱动版本、代码和数据版本。

误区二：镜像 tag 足够可靠。

Tag 可以被覆盖，复现时应该记录 digest。

误区三：训练镜像可以直接用于线上推理。

训练镜像通常很重，包含调试和编译工具。推理镜像应该更精简、安全、启动快。

误区四：CUDA 版本越新越好。

新版本要和驱动、框架、NCCL、自定义 kernel 和推理 runtime 兼容，必须经过验证。

误区五：容器就是安全边界。

容器不是绝对安全边界，多租户场景还需要权限、网络策略、密钥管理、审计和宿主机安全。

## 11.18 面试题

### 题 1：Docker 镜像和容器有什么区别？

答：镜像是静态的运行环境包，包含文件系统和依赖；容器是镜像运行后的进程环境。镜像可以存到 registry，容器在具体机器上启动运行。

### 题 2：为什么 GPU 容器仍然依赖宿主机驱动？

答：GPU 驱动通常安装在宿主机上，负责和硬件交互。容器里主要包含 CUDA 用户态库和框架。容器需要通过 GPU runtime 和设备挂载访问宿主机 GPU，因此驱动、CUDA、框架需要兼容。

### 题 3：为什么要记录镜像 digest？

答：Tag 可以被覆盖，不能保证长期指向同一内容。Digest 是镜像内容哈希，能唯一标识镜像版本，有利于实验复现、审计和回滚。

### 题 4：训练镜像和推理镜像有什么不同？

答：训练镜像通常包含训练框架、分布式库、调试工具、profiler 和编译工具；推理镜像更关注 serving runtime、启动速度、镜像体积、安全加固和稳定性。推理镜像应尽量精简。

### 题 5：容器里 NCCL 通信失败可能是什么原因？

答：可能是 NCCL 版本不兼容、网卡或 RDMA 设备没有挂载、容器网络策略限制、环境变量配置错误、驱动/CUDA 不匹配、GPU 到网卡亲和性或集群网络问题。

要求：包含构建、依赖锁定、GPU smoke test、安全扫描、签名、推送 registry 和记录 digest。

## 11.20 本章小结

本章讲了容器化基础。

你需要掌握：

1. 容器化让训练和推理环境可复现、可分发、可调度、可审计。
2. Docker 镜像是静态环境包，容器是镜像运行后的实例。
3. GPU 容器仍依赖宿主机驱动，CUDA、驱动、框架和 NCCL 需要兼容。
4. 镜像 tag 不够可靠，复现应记录 digest。
5. 训练镜像和推理镜像关注点不同，不能简单混用。
6. 镜像体积会影响任务启动和推理扩容。
7. 镜像构建应包含依赖锁定、smoke test、安全扫描、签名和审计。
8. 容器不是绝对安全边界，多租户还需要权限、网络、密钥和审计。
9. 实验追踪应记录镜像、驱动、CUDA、依赖、代码、数据和启动命令。
