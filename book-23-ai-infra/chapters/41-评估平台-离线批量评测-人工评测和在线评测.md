# 第 41 章 评估平台：离线批量评测、人工评测和在线评测

上一章讲了实验追踪。本章讲评估平台。

大模型研发中，训练出一个模型只是开始。它是否真的更好，是否更安全，是否能上线，是否比旧版本成本更低，都需要评估平台来回答。

先记住一句话：

> 评估平台不是跑 benchmark 的脚本，而是连接模型、数据、指标、人工判断、在线反馈和发布决策的质量控制系统。

## 41.1 为什么需要评估平台

小实验可以手动跑几个样例。

生产级模型需要回答：

1. 新模型比旧模型好吗？
2. 哪些能力提升，哪些能力退化？
3. 安全性是否达标？
4. 工具调用是否稳定？
5. 结构化输出是否可解析？
6. 长上下文是否可靠？
7. 延迟和成本是否可接受？
8. 是否能进入灰度发布？

这些问题不能靠主观感觉回答。

评估平台让评估流程标准化、可复现、可比较、可审计。

## 41.2 评估平台的核心目标

评估平台要做到：

1. 支持多模型评估。
2. 支持多数据集评估。
3. 支持多指标评估。
4. 支持离线批量评测。
5. 支持人工评测。
6. 支持在线评测。
7. 支持样本级诊断。
8. 支持评估报告版本化。
9. 支持发布门禁。
10. 支持评估结果对比。

一个成熟评估平台既服务研究，也服务生产发布。

## 41.3 评估类型总览

常见评估可以分为三类：

1. 离线批量评测。
2. 人工评测。
3. 在线评测。

离线批量评测：在固定数据集上批量跑模型，计算指标。

人工评测：由人类标注员或专家判断模型输出质量。

在线评测：在真实流量中观察模型表现和业务指标。

三者互补。

离线评测快、可复现，但不一定覆盖真实场景。

人工评测更接近人类偏好，但成本高。

在线评测最真实，但风险和噪声也最大。

## 41.4 离线批量评测

离线评测通常流程：

```text
ModelVersion
  -> Eval Dataset
  -> Inference Job
  -> Prediction Outputs
  -> Metric Computation
  -> Eval Report
```

关键是固定：

1. 模型版本。
2. eval 数据版本。
3. prompt 模板版本。
4. 推理参数。
5. 评估代码版本。
6. 指标定义。

否则不同评估结果不可比较。

## 41.5 Eval Dataset 管理

评估数据集必须版本化。

Eval dataset 应记录：

1. 数据集名称。
2. 版本。
3. 任务类型。
4. 样本数量。
5. 领域分布。
6. 难度分布。
7. 标签或参考答案。
8. 数据来源。
9. 构造方式。
10. 是否允许训练使用。

Eval 数据必须和训练数据隔离。

如果 eval 数据泄漏到训练集，评估结果会虚高。

## 41.6 指标定义

不同任务需要不同指标。

分类任务：

1. accuracy。
2. precision。
3. recall。
4. F1。

代码任务：

1. pass@k。
2. 编译成功率。
3. 单测通过率。

工具调用：

1. tool selection accuracy。
2. argument correctness。
3. execution success rate。

结构化输出：

1. JSON parse success rate。
2. schema validation pass rate。
3. field accuracy。

安全评估：

1. unsafe response rate。
2. over-refusal rate。
3. policy compliance rate。

不要用一个总分掩盖所有问题。

## 41.7 LLM-as-Judge

LLM-as-Judge 是用模型评估模型输出。

它适合：

1. 开放式问答。
2. 摘要质量。
3. 风格偏好。
4. 多轮对话质量。
5. 指令遵循。

但它有风险：

1. judge model 有偏好。
2. prompt 会影响评估。
3. 可能偏向更长答案。
4. 对事实性判断不可靠。
5. 不同 judge 版本不可直接比较。

因此要记录 judge model、judge prompt、评分规则和版本。

重要发布不能只依赖 LLM-as-Judge。

## 41.8 人工评测

人工评测适合评估：

1. 主观质量。
2. 风格。
3. 有用性。
4. 安全边界。
5. 复杂推理。
6. 多轮对话体验。
7. 业务可接受性。

人工评测流程：

```text
Sample Selection
  -> Annotation Guideline
  -> Human Labeling
  -> Review / QA
  -> Aggregation
  -> Report
```

人工评测不是随便找人打分，需要标注指南、质量控制和一致性检查。

## 41.9 人工评测质量控制

人工评测要控制质量。

常见方法：

1. 标注指南版本化。
2. 标注员培训。
3. gold set 检查。
4. 多人重复标注。
5. 一致性指标。
6. 仲裁机制。
7. 抽样复审。
8. 标注耗时监控。

如果人工评测本身不稳定，模型对比结论也不可靠。

## 41.10 Pairwise 评测

大模型常用 pairwise 评测。

给同一个输入，展示模型 A 和模型 B 的输出，让人或 judge 判断哪个更好。

结果可以统计 win rate。

Pairwise 的优点：

1. 比绝对打分更稳定。
2. 适合开放式生成。
3. 能直接比较新旧模型。

注意事项：

1. 隐藏模型身份。
2. 随机输出顺序。
3. 控制长度偏差。
4. 记录 tie。
5. 按任务类型分层统计。

Win rate 高不代表所有任务都好，要看分项结果。

## 41.11 在线评测

在线评测是在真实流量中观察模型表现。

常见形式：

1. A/B 测试。
2. 灰度发布指标监控。
3. 用户反馈。
4. 隐式行为指标。
5. 人工抽检线上样本。

在线评测最接近真实使用，但受噪声影响大。

例如用户群体、时间段、业务活动、路由策略变化都会影响指标。

## 41.12 在线指标

在线指标包括：

质量和体验：

1. 用户评分。
2. 点赞/点踩。
3. 追问率。
4. 人工接管率。
5. 任务完成率。
6. 投诉率。

性能：

1. TTFT。
2. TPOT。
3. p99 延迟。
4. 错误率。
5. 超时率。

成本：

1. 平均输入 token。
2. 平均输出 token。
3. cost per request。
4. cache hit rate。

安全：

1. 拦截率。
2. 违规输出率。
3. 过度拒答率。

## 41.13 发布门禁

评估平台应支持发布门禁。

例如模型要进入灰度，必须满足：

1. 核心能力不低于旧版本。
2. 安全评估通过。
3. 工具调用成功率达标。
4. JSON 解析成功率达标。
5. p99 延迟不超过阈值。
6. 成本不超过预算。
7. 人工评测 win rate 达标。

门禁不是为了阻碍发布，而是让发布风险显式化。

## 41.14 评估报告

Eval report 应包含：

1. model version。
2. eval dataset version。
3. eval code version。
4. prompt version。
5. inference config。
6. metric definitions。
7. aggregate metrics。
8. slice metrics。
9. sample-level results。
10. failure cases。
11. 与 baseline 对比。
12. 结论和建议。

评估报告应该作为 artifact 进入模型仓库和发布流程。

## 41.15 Slice Evaluation

只看总体指标很危险。

Slice evaluation 是按维度切片评估。

常见切片：

1. 语言。
2. 领域。
3. 难度。
4. 输入长度。
5. 输出格式。
6. 用户类型。
7. 安全类别。
8. 工具类型。
9. 多轮轮数。

新模型总分提升，可能只是在简单样本上提升，而复杂样本退化。

切片评估能暴露隐藏退化。

## 41.16 回归评测

回归评测用于防止旧能力退化。

每次新模型或新 prompt 发布前，都应跑回归集。

回归集通常包含：

1. 历史线上问题样本。
2. 关键业务样本。
3. 安全边界样本。
4. 工具调用样本。
5. 格式约束样本。
6. 高价值客户场景。

回归集应该持续更新。

线上出过问题的样本，修复后应加入回归集。

## 41.17 评估任务调度

评估平台也需要调度系统。

评估任务可能包括：

1. 批量推理。
2. 多模型对比。
3. 多数据集评测。
4. LLM judge。
5. 人工标注任务。
6. 安全扫描。

评估调度要考虑：

1. GPU 资源。
2. API 调用成本。
3. 优先级。
4. 依赖关系。
5. 结果缓存。
6. 失败重试。

评估平台本身也需要成本治理。

## 41.18 评估缓存

评估中很多推理可以复用。

例如同一个 model version、eval dataset、prompt version 和 inference config，如果已经跑过，可以复用 prediction outputs。

缓存 key 应包含：

1. model version。
2. eval dataset version。
3. prompt version。
4. inference config。
5. runtime version。

这样可以避免重复评估浪费成本。

但如果任何一个关键版本变化，就不能直接复用。

## 41.19 评估可观测性

评估任务也需要观测：

1. 评估任务状态。
2. 每秒处理样本数。
3. 推理错误率。
4. judge 错误率。
5. 成本消耗。
6. 样本失败原因。
7. 队列等待时间。
8. 结果写入状态。

如果评估任务失败，需要知道是模型推理失败、数据问题、评估代码问题，还是 judge 服务问题。

## 41.20 评估平台架构

一个评估平台可以这样设计：

```text
Eval API / UI
  -> Eval Dataset Registry
  -> Eval Job Scheduler
  -> Inference Runner
  -> Metric Engine
  -> LLM Judge Service
  -> Human Review System
  -> Eval Result Store
  -> Eval Report Generator
  -> Quality Gate Service
  -> Artifact Store
```

评估平台应和模型仓库、实验追踪、artifact 管理、发布系统打通。

## 41.21 常见误区

误区一：跑几个公开 benchmark 就够了。

公开 benchmark 不能覆盖企业业务场景，还需要内部数据、回归集和在线指标。

误区二：总分提升就能上线。

总分可能掩盖安全、工具调用、长上下文或特定业务退化。

误区三：LLM-as-Judge 完全可靠。

Judge model 有偏差，评估 prompt 会影响结论，重要场景需要人工和规则校验。

误区四：人工评测就是找人看看。

人工评测需要标注指南、质量控制、一致性检查和审计。

误区五：离线评估通过就代表线上好。

线上真实流量、用户行为和业务指标可能不同，还需要灰度和在线评测。

## 41.22 面试常见追问

问题一：如何设计一个大模型评估平台？

可以回答：设计 eval dataset registry、eval job scheduler、inference runner、metric engine、LLM judge、human review、result store、report generator 和 quality gate，并与模型仓库、实验追踪和发布系统打通。

问题二：离线评测、人工评测和在线评测有什么区别？

可以回答：离线评测可复现、速度快；人工评测适合主观质量和复杂场景；在线评测最接近真实用户，但风险和噪声更高。三者互补。

问题三：为什么要做 slice evaluation？

可以回答：总体指标可能掩盖局部退化。按语言、领域、难度、输入长度、安全类别和工具类型切片，能发现某些场景退化。

问题四：评估报告为什么要版本化？

可以回答：评估结果依赖模型、数据、prompt、推理参数、评估代码、judge model 和指标定义。版本化才能复现和比较。

## 41.24 本章小结

本章讲了评估平台。

你需要记住：

1. 评估平台是模型质量控制系统，不是单个 benchmark 脚本。
2. 离线批量评测、人工评测和在线评测各有优势和局限，需要组合使用。
3. Eval dataset、prompt、metric、judge model、评估代码和推理参数都要版本化。
4. 评估报告要包含总体指标、切片指标、样本级结果、失败案例和 baseline 对比。
5. 发布门禁把评估结果接入模型发布流程，是模型上线风险控制的关键。
