# 第 41 章 评估平台：离线批量评测、人工评测和在线评测

上一章讲了实验追踪。本章讲评估平台。

大模型研发中，训练出一个模型只是开始。它是否真的更好，是否更安全，是否能上线，是否比旧版本成本更低，都需要评估平台来回答。

先记住一句话：

> 评估平台不是跑 benchmark 的脚本，而是连接模型、数据、指标、人工判断、在线反馈和发布决策的质量控制系统。

## 41.0 本讲资料边界与第二轮精修口径

本讲按通用大模型评估平台抽象来写，不绑定某个内部平台、单一 benchmark、单一 LLM judge、单一标注系统或单一 A/B 实验工具。

资料校准时参考的公开口径包括：OpenAI 官方评估接口迁移说明、Hugging Face Evaluate 对 metric / comparison / measurement 的抽象、EleutherAI lm-evaluation-harness 对多任务离线评测的执行方式、HELM 对 scenario / metric / slice 的评估组织方式、Label Studio / HumanSignal 对标注质量与一致性的说明，以及 MLflow / Weave 这类实验评估工具对 scorer、trace、report 和 artifact 的工程抽象。

这些资料给本章一个共同边界：评估平台不是“把模型跑一遍数据集”，而是把模型版本、评估数据、prompt、推理配置、指标定义、judge、人评、在线指标、样本诊断、报告、门禁和发布证据统一成可复现对象。

本轮精修重点补三类内容：

1. 用稳定数学符号描述评估任务、数据覆盖、指标聚合、人评一致性、pairwise win rate、在线指标差异、切片退化、缓存命中和最终发布门禁。
2. 增加一个 0 依赖 `MiniEvaluationPlatformAudit` demo，用 toy case 演示评估平台如何检查 16 个关键审计维度。
3. 同步第四册百科、面试题、练习、术语表、项目任务和知识图谱，让“评估平台”成为 AI Infra 生命周期的一环，而不是孤立章节。

## 41.1 为什么需要评估平台

小实验可以手动跑几个样例。

生产级模型需要回答：

1. 新模型比旧模型好吗？
2. 哪些能力提升，哪些能力退化？
3. 安全性是否达标？
4. 工具调用是否稳定？
5. 结构化输出是否可解析？
6. 长上下文是否可靠？
7. 延迟和成本是否可接受？
8. 是否能进入灰度发布？

这些问题不能靠主观感觉回答。

评估平台让评估流程标准化、可复现、可比较、可审计。

可以把一次评估任务抽象成：

```math
J_i=(m_i,d_i,p_i,c_i,q_i,g_i,h_i,o_i,r_i,z_i)
```

其中 `m_i` 是模型版本，`d_i` 是评估数据版本，`p_i` 是 prompt / template 版本，`c_i` 是推理配置，`q_i` 是评估代码版本，`g_i` 是指标定义集合，`h_i` 是 judge 或人评配置，`o_i` 是样本级输出，`r_i` 是评估报告，`z_i` 是发布或回归状态。

如果这些字段没有一起记录，后续就只能看到一个分数，不能回答“这个分数从哪里来、能不能复现、能不能和旧版本比较”。

## 41.2 评估平台的核心目标

评估平台要做到：

1. 支持多模型评估。
2. 支持多数据集评估。
3. 支持多指标评估。
4. 支持离线批量评测。
5. 支持人工评测。
6. 支持在线评测。
7. 支持样本级诊断。
8. 支持评估报告版本化。
9. 支持发布门禁。
10. 支持评估结果对比。

一个成熟评估平台既服务研究，也服务生产发布。

因此它的最低可用标准不是“支持某个指标”，而是覆盖完整证据链：

```math
C_{\mathrm{eval}}=\frac{1}{N_{\mathrm{cap}}}\sum_{j=1}^{N_{\mathrm{cap}}}\mathbf{1}[a_j\ \mathrm{ready}]
```

其中 `a_j` 表示模型版本、数据版本、指标、样本诊断、报告、门禁、artifact、可观测性等平台能力。`C_eval` 越接近 1，说明评估平台越能支撑研发和发布闭环；如果只覆盖离线跑分，`C_eval` 会很低。

## 41.3 评估类型总览

常见评估可以分为三类：

1. 离线批量评测。
2. 人工评测。
3. 在线评测。

离线批量评测：在固定数据集上批量跑模型，计算指标。

人工评测：由人类标注员或专家判断模型输出质量。

在线评测：在真实流量中观察模型表现和业务指标。

三者互补。

离线评测快、可复现，但不一定覆盖真实场景。

人工评测更接近人类偏好，但成本高。

在线评测最真实，但风险和噪声也最大。

三类评估最好不要互相替代，而要做成分层证据：

```math
E_{\mathrm{all}}=(E_{\mathrm{offline}},E_{\mathrm{human}},E_{\mathrm{online}})
```

离线评估证明候选版本在固定口径下没有明显退化，人评证明开放式质量和偏好更可信，在线评估证明真实流量下的质量、延迟、成本和安全护栏没有失控。

## 41.4 离线批量评测

离线评测通常流程：

```text
ModelVersion
  -> Eval Dataset
  -> Inference Job
  -> Prediction Outputs
  -> Metric Computation
  -> Eval Report
```

关键是固定：

1. 模型版本。
2. eval 数据版本。
3. prompt 模板版本。
4. 推理参数。
5. 评估代码版本。
6. 指标定义。

否则不同评估结果不可比较。

离线批量评测的可复现门禁可以写成：

```math
G_{\mathrm{offline}}=\mathbf{1}[C_m=1 \land C_d=1 \land C_p=1 \land C_c=1 \land C_q=1 \land C_g=1]
```

其中 `C_m`、`C_d`、`C_p`、`C_c`、`C_q`、`C_g` 分别表示模型版本、数据版本、prompt 版本、推理配置、评估代码和指标定义是否固定。任何一项缺失，离线分数都不应直接用于发布判断。

## 41.5 Eval Dataset 管理

评估数据集必须版本化。

Eval dataset 应记录：

1. 数据集名称。
2. 版本。
3. 任务类型。
4. 样本数量。
5. 领域分布。
6. 难度分布。
7. 标签或参考答案。
8. 数据来源。
9. 构造方式。
10. 是否允许训练使用。

Eval 数据必须和训练数据隔离。

如果 eval 数据泄漏到训练集，评估结果会虚高。

数据覆盖率可以按必需切片统计：

```math
C_{\mathrm{dataset}}=\frac{1}{N_{\mathrm{slice}}}\sum_{s=1}^{N_{\mathrm{slice}}}\mathbf{1}[n_s\ge n_{\min}]
```

其中 `n_s` 是第 `s` 个切片的样本数，`n_min` 是该切片的最低样本量要求。覆盖率只看有没有样本还不够，还要看是否达到能支持比较的最小规模。

训练 / 评估污染率可以写成：

```math
R_{\mathrm{leak}}=\frac{N_{\mathrm{overlap}}}{N_{\mathrm{eval}}}
```

其中 `N_overlap` 是训练集和评估集 exact duplicate、near duplicate 或来源交叉的样本数。`R_leak` 越高，评估越可能虚高；重要 benchmark 或内部回归集要把这个值纳入门禁。

## 41.6 指标定义

不同任务需要不同指标。

分类任务：

1. accuracy。
2. precision。
3. recall。
4. F1。

代码任务：

1. pass@k。
2. 编译成功率。
3. 单测通过率。

工具调用：

1. tool selection accuracy。
2. argument correctness。
3. execution success rate。

结构化输出：

1. JSON parse success rate。
2. schema validation pass rate。
3. field accuracy。

安全评估：

1. unsafe response rate。
2. over-refusal rate。
3. policy compliance rate。

不要用一个总分掩盖所有问题。

如果必须给一个汇总分，也要把权重和方向写清楚：

```math
S_{\mathrm{agg}}=\frac{\sum_{k=1}^{K}w_k \tilde{M}_k}{\sum_{k=1}^{K}w_k}
```

其中 `M_k` 是第 `k` 个指标，`\tilde{M}_k` 是统一到“越大越好”后的归一化分数，`w_k` 是业务或安全权重。面试中要强调：汇总分只能做导航，真正决策仍要看分项指标、切片指标和失败样本。

候选模型和 baseline 的指标差异可以写成：

```math
\Delta_k=M_k(\mathrm{candidate})-M_k(\mathrm{baseline})
```

每个 `\Delta_k` 都应该记录统计口径、样本数、置信区间或至少稳定性说明，不能只写“提升 2 分”。

## 41.7 LLM-as-Judge

LLM-as-Judge 是用模型评估模型输出。

它适合：

1. 开放式问答。
2. 摘要质量。
3. 风格偏好。
4. 多轮对话质量。
5. 指令遵循。

但它有风险：

1. judge model 有偏好。
2. prompt 会影响评估。
3. 可能偏向更长答案。
4. 对事实性判断不可靠。
5. 不同 judge 版本不可直接比较。

因此要记录 judge model、judge prompt、评分规则和版本。

重要发布不能只依赖 LLM-as-Judge。

judge 和人工标注的一致率可以写成：

```math
A_{\mathrm{judge}}=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[\hat{y}^{\mathrm{judge}}_i=y^{\mathrm{human}}_i]
```

其中 `y_human` 是人工或专家标签，`\hat{y}_judge` 是 judge 输出。这个值低时，judge 只能用于探索性分析，不能作为发布硬门禁。

如果怀疑 judge 偏向长答案，可以记录长度偏差：

```math
B_{\mathrm{len}}=\mathrm{mean}(\ell_i\mid \hat{y}^{\mathrm{judge}}_i=1)-\mathrm{mean}(\ell_i\mid \hat{y}^{\mathrm{judge}}_i=0)
```

其中 `ell_i` 是回答长度。如果被判优的答案显著更长，要检查 rubric 是否把“啰嗦”误当成“质量高”。

## 41.8 人工评测

人工评测适合评估：

1. 主观质量。
2. 风格。
3. 有用性。
4. 安全边界。
5. 复杂推理。
6. 多轮对话体验。
7. 业务可接受性。

人工评测流程：

```text
Sample Selection
  -> Annotation Guideline
  -> Human Labeling
  -> Review / QA
  -> Aggregation
  -> Report
```

人工评测不是随便找人打分，需要标注指南、质量控制和一致性检查。

一次人工评测任务也应该有版本化对象：

```math
H_i=(s_i,u_i,v_i,a_i,g_i,r_i,t_i,z_i)
```

其中 `s_i` 是样本，`u_i` 是标注员或专家，`v_i` 是标注指南版本，`a_i` 是标注结果，`g_i` 是 gold set 命中情况，`r_i` 是复审或仲裁结果，`t_i` 是标注耗时，`z_i` 是质量状态。

## 41.9 人工评测质量控制

人工评测要控制质量。

常见方法：

1. 标注指南版本化。
2. 标注员培训。
3. gold set 检查。
4. 多人重复标注。
5. 一致性指标。
6. 仲裁机制。
7. 抽样复审。
8. 标注耗时监控。

如果人工评测本身不稳定，模型对比结论也不可靠。

gold set 准确率可以写成：

```math
A_{\mathrm{gold}}=\frac{1}{N_{\mathrm{gold}}}\sum_{i=1}^{N_{\mathrm{gold}}}\mathbf{1}[\hat{y}^{\mathrm{ann}}_i=y^{\mathrm{gold}}_i]
```

多人重复标注的一致率可以用简单 pair agreement 表达：

```math
A_{\mathrm{pair}}=\frac{2}{N_{\mathrm{ann}}(N_{\mathrm{ann}}-1)}\sum_{a<b}\mathbf{1}[y_a=y_b]
```

其中 `N_ann` 是同一样本上的标注人数。真实系统还可以使用更复杂的一致性指标，但面试里先把 gold set、重复标注、复审和仲裁讲清楚更重要。

## 41.10 Pairwise 评测

大模型常用 pairwise 评测。

给同一个输入，展示模型 A 和模型 B 的输出，让人或 judge 判断哪个更好。

结果可以统计 win rate。

Pairwise 的优点：

1. 比绝对打分更稳定。
2. 适合开放式生成。
3. 能直接比较新旧模型。

注意事项：

1. 隐藏模型身份。
2. 随机输出顺序。
3. 控制长度偏差。
4. 记录 tie。
5. 按任务类型分层统计。

Win rate 高不代表所有任务都好，要看分项结果。

带 tie 的 pairwise win rate 常写成：

```math
W=\frac{N_{\mathrm{win}}+0.5N_{\mathrm{tie}}}{N_{\mathrm{win}}+N_{\mathrm{loss}}+N_{\mathrm{tie}}}
```

为了避免把小样本波动误判成真实提升，可以同时给出近似标准误：

```math
\mathrm{SE}(W)=\sqrt{\frac{W(1-W)}{N}}
```

一个粗略 95% 区间可以写成 `W +- 1.96 SE(W)`。这不是严谨统计课里的全部细节，但足以提醒面试官：pairwise 评测必须关注样本量、随机顺序、盲评、tie 和切片稳定性。

## 41.11 在线评测

在线评测是在真实流量中观察模型表现。

常见形式：

1. A/B 测试。
2. 灰度发布指标监控。
3. 用户反馈。
4. 隐式行为指标。
5. 人工抽检线上样本。

在线评测最接近真实使用，但受噪声影响大。

例如用户群体、时间段、业务活动、路由策略变化都会影响指标。

在线指标差异可以写成：

```math
\Delta_{\mathrm{online}}=M_{\mathrm{treatment}}-M_{\mathrm{control}}
```

如果指标是比例，可以用近似标准误做基本 sanity check：

```math
\mathrm{SE}_{\Delta}=\sqrt{\frac{p_t(1-p_t)}{n_t}+\frac{p_c(1-p_c)}{n_c}}
```

其中 `p_t`、`n_t` 是 treatment 分桶的比例指标和样本数，`p_c`、`n_c` 是 control 分桶的比例指标和样本数。上线决策不能只看均值，还要看分桶稳定性、样本比例、护栏指标和回滚准备。

## 41.12 在线指标

在线指标包括：

质量和体验：

1. 用户评分。
2. 点赞/点踩。
3. 追问率。
4. 人工接管率。
5. 任务完成率。
6. 投诉率。

性能：

1. TTFT。
2. TPOT。
3. p99 延迟。
4. 错误率。
5. 超时率。

成本：

1. 平均输入 token。
2. 平均输出 token。
3. cost per request。
4. cache hit rate。

安全：

1. 拦截率。
2. 违规输出率。
3. 过度拒答率。

在线护栏门禁可以写成：

```math
G_{\mathrm{online}}=\mathbf{1}[\Delta_Q\ge -\epsilon_Q \land \Delta_L\le \epsilon_L \land \Delta_K\le \epsilon_K \land P_0=0]
```

其中 `Delta_Q` 是质量或任务完成率变化，`Delta_L` 是延迟变化，`Delta_K` 是成本变化，`P_0` 是 P0 风险事件数量。这个公式的直觉是：即使质量略升，只要延迟、成本或安全事件失控，也不能继续放量。

## 41.13 发布门禁

评估平台应支持发布门禁。

例如模型要进入灰度，必须满足：

1. 核心能力不低于旧版本。
2. 安全评估通过。
3. 工具调用成功率达标。
4. JSON 解析成功率达标。
5. p99 延迟不超过阈值。
6. 成本不超过预算。
7. 人工评测 win rate 达标。

门禁不是为了阻碍发布，而是让发布风险显式化。

发布门禁可以把离线、人评、在线、性能、成本和安全串起来：

```math
G_{\mathrm{release}}=\mathbf{1}[G_{\mathrm{offline}}=1 \land G_{\mathrm{human}}=1 \land G_{\mathrm{online}}=1 \land G_{\mathrm{safety}}=1 \land G_{\mathrm{rollback}}=1]
```

其中 `G_rollback` 表示回滚版本、回滚包、回滚路由和回滚监控都准备好。评估平台如果只负责产出分数，不负责把分数接到发布门禁，就仍然不是完整平台。

## 41.14 评估报告

Eval report 应包含：

1. model version。
2. eval dataset version。
3. eval code version。
4. prompt version。
5. inference config。
6. metric definitions。
7. aggregate metrics。
8. slice metrics。
9. sample-level results。
10. failure cases。
11. 与 baseline 对比。
12. 结论和建议。

评估报告应该作为 artifact 进入模型仓库和发布流程。

报告完整率可以写成：

```math
C_{\mathrm{report}}=\frac{1}{N_{\mathrm{field}}}\sum_{j=1}^{N_{\mathrm{field}}}\mathbf{1}[f_j\ \mathrm{present}]
```

其中 `f_j` 是报告必须字段。低完整率报告不能作为发布证据，因为它无法解释模型为什么被批准、拒绝或回滚。

## 41.15 Slice Evaluation

只看总体指标很危险。

Slice evaluation 是按维度切片评估。

常见切片：

1. 语言。
2. 领域。
3. 难度。
4. 输入长度。
5. 输出格式。
6. 用户类型。
7. 安全类别。
8. 工具类型。
9. 多轮轮数。

新模型总分提升，可能只是在简单样本上提升，而复杂样本退化。

切片评估能暴露隐藏退化。

切片退化可以按每个切片单独比较：

```math
D_s=M_s(\mathrm{candidate})-M_s(\mathrm{baseline})
```

然后定义切片回归门禁：

```math
G_{\mathrm{slice}}=\mathbf{1}[\min_s D_s\ge -\epsilon_s]
```

其中 `epsilon_s` 是允许的最大退化。这个门禁能防止“总体提升 1%，但高风险切片退化 10%”被平均数掩盖。

## 41.16 回归评测

回归评测用于防止旧能力退化。

每次新模型或新 prompt 发布前，都应跑回归集。

回归集通常包含：

1. 历史线上问题样本。
2. 关键业务样本。
3. 安全边界样本。
4. 工具调用样本。
5. 格式约束样本。
6. 高价值客户场景。

回归集应该持续更新。

线上出过问题的样本，修复后应加入回归集。

回归集通过率可以写成：

```math
C_{\mathrm{reg}}=\frac{N_{\mathrm{pass}}}{N_{\mathrm{reg}}}
```

回归集新鲜度可以写成：

```math
F_{\mathrm{reg}}=\frac{N_{\mathrm{recent}}}{N_{\mathrm{reg}}}
```

其中 `N_recent` 是来自最近线上事故、客户反馈、红队发现或高价值业务场景的样本数。只有固定老样本的回归集会逐渐失去覆盖能力。

## 41.17 评估任务调度

评估平台也需要调度系统。

评估任务可能包括：

1. 批量推理。
2. 多模型对比。
3. 多数据集评测。
4. LLM judge。
5. 人工标注任务。
6. 安全扫描。

评估调度要考虑：

1. GPU 资源。
2. API 调用成本。
3. 优先级。
4. 依赖关系。
5. 结果缓存。
6. 失败重试。

评估平台本身也需要成本治理。

评估任务成本可以粗略写成：

```math
K_{\mathrm{eval}}=K_{\mathrm{infer}}+K_{\mathrm{judge}}+K_{\mathrm{human}}+K_{\mathrm{store}}+K_{\mathrm{ops}}
```

其中 `K_infer` 是候选模型推理成本，`K_judge` 是 judge 调用成本，`K_human` 是人工标注成本，`K_store` 是样本和报告存储成本，`K_ops` 是调度、重试、监控和运维成本。评估任务也要有预算、优先级和失败重试策略，否则评估平台会变成隐形成本中心。

## 41.18 评估缓存

评估中很多推理可以复用。

例如同一个 model version、eval dataset、prompt version 和 inference config，如果已经跑过，可以复用 prediction outputs。

缓存 key 应包含：

1. model version。
2. eval dataset version。
3. prompt version。
4. inference config。
5. runtime version。

这样可以避免重复评估浪费成本。

但如果任何一个关键版本变化，就不能直接复用。

缓存 key 可以写成：

```math
k_{\mathrm{cache}}=H(m,d,p,c,q,r)
```

其中 `H` 是稳定 hash，`m` 是模型版本，`d` 是数据版本，`p` 是 prompt 版本，`c` 是推理配置，`q` 是评估代码版本，`r` 是 runtime 版本。

缓存命中率可以写成：

```math
R_{\mathrm{cache}}=\frac{N_{\mathrm{hit}}}{N_{\mathrm{request}}}
```

高缓存命中率能降低成本，但不能以牺牲正确性为代价。只要 key 少了一个版本字段，就可能复用错误输出。

## 41.19 评估可观测性

评估任务也需要观测：

1. 评估任务状态。
2. 每秒处理样本数。
3. 推理错误率。
4. judge 错误率。
5. 成本消耗。
6. 样本失败原因。
7. 队列等待时间。
8. 结果写入状态。

如果评估任务失败，需要知道是模型推理失败、数据问题、评估代码问题，还是 judge 服务问题。

可观测性覆盖率可以写成：

```math
C_{\mathrm{obs}}=\frac{1}{N_{\mathrm{signal}}}\sum_{j=1}^{N_{\mathrm{signal}}}\mathbf{1}[\sigma_j\ \mathrm{captured}]
```

其中 `sigma_j` 表示任务状态、吞吐、错误率、成本、样本失败原因、队列等待、结果写入、trace id 等信号。评估失败时，`C_obs` 越低，越难判断是模型差、数据坏、judge 不稳定还是平台故障。

## 41.20 评估平台架构

一个评估平台可以这样设计：

```text
Eval API / UI
  -> Eval Dataset Registry
  -> Eval Job Scheduler
  -> Inference Runner
  -> Metric Engine
  -> LLM Judge Service
  -> Human Review System
  -> Eval Result Store
  -> Eval Report Generator
  -> Quality Gate Service
  -> Artifact Store
```

评估平台应和模型仓库、实验追踪、artifact 管理、发布系统打通。

可以把关键对象连接成血缘图：

```math
G_{\mathrm{eval\_lineage}}=(V,E)
```

其中节点 `V` 包括 ModelVersion、EvalDataset、PromptVersion、EvalJob、PredictionArtifact、JudgeRun、HumanReviewTask、EvalReport、ReleaseDecision；边 `E` 表示“使用、生成、比较、批准、拒绝、发布、回滚”等关系。面试里讲评估平台架构时，要强调这张图能从线上模型反查到离线评估证据，也能从一个失败样本追到模型、prompt、judge 和发布决策。

## 41.21 评估平台审计指标和最小 demo

把前面的内容合在一起，评估平台可以用 16 个维度做最小审计：

1. eval dataset contract：数据集版本、样本量、切片、污染检查和使用权限是否完整。
2. offline batch reproducibility：模型、数据、prompt、推理配置、评估代码和指标定义是否固定。
3. metric definition versioning：指标名称、方向、权重、阈值和定义版本是否记录。
4. llm judge calibration：judge model、prompt、rubric、人类校准集和偏差检查是否完整。
5. human review quality control：标注指南、gold set、重复标注、复审和仲裁是否完整。
6. pairwise blinding randomization：pairwise 是否隐藏模型身份、随机顺序、记录 tie 并按切片统计。
7. online eval guardrail：A/B 或灰度是否有稳定分桶、样本比例、质量、延迟、成本和安全护栏。
8. release gate readiness：发布门禁是否连接离线、人评、在线、安全、性能、成本和回滚。
9. eval report completeness：评估报告是否含模型、数据、代码、prompt、配置、指标、切片、样本和结论。
10. slice regression detection：是否能发现语言、领域、难度、长度、安全、工具等切片退化。
11. regression suite coverage：回归集是否覆盖历史事故、关键业务、安全边界、工具和格式约束。
12. eval job scheduler reliability：评估调度是否有优先级、依赖、重试、预算和失败分类。
13. eval cache key integrity：缓存 key 是否包含所有会影响输出和指标的版本字段。
14. eval observability：评估任务是否有状态、吞吐、错误、成本、失败原因、trace 和结果写入信号。
15. artifact tracking linkage：prediction outputs、sample table、judge run、人评结果、报告和 release decision 是否进入 artifact / lineage。
16. evaluation platform gate：最终门禁是否能证明无 P0 风险、关键覆盖率达标、可复现、可比较、可发布或可回滚。

下面的 demo 不依赖外部库。它故意构造 1 个完整评估平台样本和 16 个只破坏一个维度的 bad case，帮助你理解“评估平台审计”不是看一个总分，而是逐项证明评估数据、指标、人评、judge、在线实验、报告、调度、缓存、可观测性和发布门禁都可信。

```python
from math import sqrt


class MiniEvaluationPlatformAudit:
    REQUIRED_DATASET_FIELDS = {
        "dataset_id",
        "version",
        "task_type",
        "sample_count",
        "slices",
        "source",
        "construction",
        "labels",
        "train_usage_allowed",
        "contamination_checked",
    }

    REQUIRED_OFFLINE_FIELDS = {
        "model_version",
        "dataset_version",
        "prompt_version",
        "inference_config_hash",
        "eval_code_version",
        "metric_definition_version",
        "runtime_version",
    }

    REQUIRED_REPORT_FIELDS = {
        "model_version",
        "dataset_version",
        "eval_code_version",
        "prompt_version",
        "inference_config",
        "metric_definitions",
        "aggregate_metrics",
        "slice_metrics",
        "sample_results",
        "failure_cases",
        "baseline_comparison",
        "decision",
    }

    REQUIRED_OBS_SIGNALS = {
        "state",
        "throughput",
        "inference_error_rate",
        "judge_error_rate",
        "cost",
        "sample_failure_reason",
        "queue_wait",
        "result_write_state",
        "trace_id",
    }

    CACHE_KEY_FIELDS = {
        "model_version",
        "dataset_version",
        "prompt_version",
        "inference_config_hash",
        "eval_code_version",
        "runtime_version",
    }

    def __init__(self, cases):
        self.cases = cases
        self.checks = [
            ("eval_dataset_contract", self.eval_dataset_contract),
            ("offline_batch_reproducibility", self.offline_batch_reproducibility),
            ("metric_definition_versioning", self.metric_definition_versioning),
            ("llm_judge_calibration", self.llm_judge_calibration),
            ("human_review_quality_control", self.human_review_quality_control),
            ("pairwise_blinding_randomization", self.pairwise_blinding_randomization),
            ("online_eval_guardrail", self.online_eval_guardrail),
            ("release_gate_readiness", self.release_gate_readiness),
            ("eval_report_completeness", self.eval_report_completeness),
            ("slice_regression_detection", self.slice_regression_detection),
            ("regression_suite_coverage", self.regression_suite_coverage),
            ("eval_job_scheduler_reliability", self.eval_job_scheduler_reliability),
            ("eval_cache_key_integrity", self.eval_cache_key_integrity),
            ("eval_observability", self.eval_observability),
            ("artifact_tracking_linkage", self.artifact_tracking_linkage),
            ("evaluation_platform_gate", self.evaluation_platform_gate),
        ]

    def eval_dataset_contract(self, case):
        dataset = case["dataset"]
        slices = dataset.get("slices", {})
        required_slices = {"language:zh", "language:en", "domain:code", "domain:math", "difficulty:hard"}
        enough_slice_samples = all(slices.get(name, 0) >= 20 for name in required_slices)
        required_fields_present = self.REQUIRED_DATASET_FIELDS <= set(dataset)
        return (
            required_fields_present
            and dataset["sample_count"] >= 100
            and enough_slice_samples
            and dataset["labels"]
            and dataset["contamination_checked"]
            and dataset["train_usage_allowed"] is False
        )

    def offline_batch_reproducibility(self, case):
        offline = case["offline"]
        return self.REQUIRED_OFFLINE_FIELDS <= set(offline)

    def metric_definition_versioning(self, case):
        metrics = case["metrics"]
        if not metrics:
            return False
        for metric in metrics:
            required = {"name", "version", "direction", "weight", "threshold"}
            if not required <= set(metric):
                return False
            if metric["direction"] not in {"higher_is_better", "lower_is_better"}:
                return False
            if metric["weight"] <= 0:
                return False
        return True

    def llm_judge_calibration(self, case):
        judge = case["judge"]
        required = {"model", "prompt_version", "rubric_version", "human_calibration_score", "bias_checks"}
        return (
            required <= set(judge)
            and judge["human_calibration_score"] >= 0.8
            and {"length_bias", "position_bias"} <= set(judge["bias_checks"])
        )

    def human_review_quality_control(self, case):
        human = case["human_review"]
        required = {"guideline_version", "gold_accuracy", "duplicate_annotation_rate", "review_rate", "arbitration"}
        return (
            required <= set(human)
            and human["gold_accuracy"] >= 0.9
            and human["duplicate_annotation_rate"] >= 0.2
            and human["review_rate"] >= 0.1
            and human["arbitration"]
        )

    def pairwise_blinding_randomization(self, case):
        pairwise = case["pairwise"]
        required = {"blind_model_ids", "randomized_order", "tie_recorded", "slice_breakdown", "wins", "losses", "ties"}
        return (
            required <= set(pairwise)
            and pairwise["blind_model_ids"]
            and pairwise["randomized_order"]
            and pairwise["tie_recorded"]
            and bool(pairwise["slice_breakdown"])
        )

    def online_eval_guardrail(self, case):
        online = case["online"]
        required = {
            "stable_bucket",
            "sample_ratio_health",
            "quality_delta",
            "latency_delta",
            "cost_delta",
            "safety_events",
            "rollback_ready",
        }
        return (
            required <= set(online)
            and online["stable_bucket"]
            and online["sample_ratio_health"] >= 0.95
            and online["quality_delta"] >= -0.01
            and online["latency_delta"] <= 0.05
            and online["cost_delta"] <= 0.08
            and online["safety_events"] == 0
            and online["rollback_ready"]
        )

    def release_gate_readiness(self, case):
        gate = case["release_gate"]
        required = {
            "offline_passed",
            "human_passed",
            "online_passed",
            "safety_passed",
            "performance_passed",
            "cost_passed",
            "rollback_ready",
            "approver",
        }
        return required <= set(gate) and all(gate[key] for key in required)

    def eval_report_completeness(self, case):
        report = case["report"]
        return self.REQUIRED_REPORT_FIELDS <= set(report)

    def slice_regression_detection(self, case):
        slices = case["slice_metrics"]
        if not slices:
            return False
        return all(item["candidate"] - item["baseline"] >= -0.02 for item in slices.values())

    def regression_suite_coverage(self, case):
        suite = case["regression_suite"]
        required = {"incident", "business", "safety", "tool", "format"}
        return required <= set(suite.get("categories", set())) and suite.get("fresh_case_ratio", 0.0) >= 0.2

    def eval_job_scheduler_reliability(self, case):
        scheduler = case["scheduler"]
        required = {"priority", "dependencies", "retry_policy", "budget_limit", "failure_classification", "owner"}
        return required <= set(scheduler) and scheduler["retry_policy"]["max_retries"] >= 1 and scheduler["budget_limit"] > 0

    def eval_cache_key_integrity(self, case):
        cache = case["cache"]
        return self.CACHE_KEY_FIELDS <= set(cache.get("key_fields", set())) and cache.get("hit_rate", 0.0) >= 0.2

    def eval_observability(self, case):
        observed = case["observability"].get("signals", set())
        return self.REQUIRED_OBS_SIGNALS <= observed

    def artifact_tracking_linkage(self, case):
        artifacts = case["artifacts"]
        required = {
            "prediction_outputs",
            "sample_table",
            "judge_run",
            "human_review",
            "eval_report",
            "release_decision",
            "lineage_edges",
        }
        return required <= set(artifacts) and len(artifacts["lineage_edges"]) >= 6

    def evaluation_platform_gate(self, case):
        gate = case["platform_gate"]
        return gate.get("enabled") and gate.get("min_coverage", 0.0) >= 0.95 and gate.get("p0_risks", 1) == 0

    def metric_scores(self):
        total = len(self.cases)
        return {
            name: round(sum(1 for case in self.cases if check(case)) / total, 3)
            for name, check in self.checks
        }

    def failed_cases(self):
        failures = []
        for case in self.cases:
            failed = [name for name, check in self.checks if not check(case)]
            if failed:
                failures.append((case["case_id"], failed))
        return failures

    def examples(self):
        case = self.cases[0]
        dataset = case["dataset"]
        pairwise = case["pairwise"]
        human = case["human_review"]
        judge = case["judge"]
        online = case["online"]
        report = case["report"]
        cache = case["cache"]

        required_slice_names = {"language:zh", "language:en", "domain:code", "domain:math", "difficulty:hard"}
        dataset_coverage = sum(
            1 for name in required_slice_names if dataset["slices"].get(name, 0) >= 20
        ) / len(required_slice_names)
        contamination_rate = dataset["overlap_count"] / dataset["sample_count"]
        offline_accuracy = report["aggregate_metrics"]["accuracy"]

        total_pairwise = pairwise["wins"] + pairwise["losses"] + pairwise["ties"]
        win_rate = (pairwise["wins"] + 0.5 * pairwise["ties"]) / total_pairwise
        win_rate_se = sqrt(win_rate * (1.0 - win_rate) / total_pairwise)

        online_pass = (
            online["quality_delta"] >= -0.01
            and online["latency_delta"] <= 0.05
            and online["cost_delta"] <= 0.08
            and online["safety_events"] == 0
        )

        return {
            "dataset_coverage": round(dataset_coverage, 3),
            "contamination_rate": round(contamination_rate, 4),
            "offline_accuracy": offline_accuracy,
            "pairwise_win_rate": round(win_rate, 3),
            "pairwise_win_rate_se": round(win_rate_se, 4),
            "human_gold_accuracy": human["gold_accuracy"],
            "human_pair_agreement": human["pair_agreement"],
            "judge_human_agreement": judge["human_calibration_score"],
            "judge_length_bias": judge["bias_checks"]["length_bias"],
            "online_guardrail_pass": online_pass,
            "release_ready": self.release_gate_readiness(case),
            "cache_hit_rate": cache["hit_rate"],
            "report_field_count": len(report),
            "lineage_edge_count": len(case["artifacts"]["lineage_edges"]),
            "estimated_eval_cost_usd": case["cost"]["total_usd"],
        }
```

## 41.22 常见误区

误区一：跑几个公开 benchmark 就够了。

公开 benchmark 不能覆盖企业业务场景，还需要内部数据、回归集和在线指标。

误区二：总分提升就能上线。

总分可能掩盖安全、工具调用、长上下文或特定业务退化。

误区三：LLM-as-Judge 完全可靠。

Judge model 有偏差，评估 prompt 会影响结论，重要场景需要人工和规则校验。

误区四：人工评测就是找人看看。

人工评测需要标注指南、质量控制、一致性检查和审计。

误区五：离线评估通过就代表线上好。

线上真实流量、用户行为和业务指标可能不同，还需要灰度和在线评测。

## 41.23 面试常见追问

问题一：如何设计一个大模型评估平台？

可以回答：设计 eval dataset registry、eval job scheduler、inference runner、metric engine、LLM judge、human review、result store、report generator 和 quality gate，并与模型仓库、实验追踪和发布系统打通。

问题二：离线评测、人工评测和在线评测有什么区别？

可以回答：离线评测可复现、速度快；人工评测适合主观质量和复杂场景；在线评测最接近真实用户，但风险和噪声更高。三者互补。

问题三：为什么要做 slice evaluation？

可以回答：总体指标可能掩盖局部退化。按语言、领域、难度、输入长度、安全类别和工具类型切片，能发现某些场景退化。

问题四：评估报告为什么要版本化？

可以回答：评估结果依赖模型、数据、prompt、推理参数、评估代码、judge model 和指标定义。版本化才能复现和比较。

问题五：评估平台为什么必须和实验追踪、artifact 管理和发布系统打通？

可以回答：实验追踪记录 run 参数和指标，artifact 管理保存 prediction outputs、sample table、eval report 和 release bundle，发布系统执行灰度、A/B、回滚和在线监控。评估平台如果不和这些系统打通，就无法从线上模型反查评估证据，也无法从失败样本追到数据、prompt、judge、指标和发布决策。

## 41.25 本章小结

本章讲了评估平台。

你需要记住：

1. 评估平台是模型质量控制系统，不是单个 benchmark 脚本。
2. 离线批量评测、人工评测和在线评测各有优势和局限，需要组合使用。
3. Eval dataset、prompt、metric、judge model、评估代码和推理参数都要版本化。
4. 评估报告要包含总体指标、切片指标、样本级结果、失败案例和 baseline 对比。
5. 发布门禁把评估结果接入模型发布流程，是模型上线风险控制的关键。
6. 成熟评估平台还要记录人评质量、judge 校准、pairwise 盲评、在线护栏、调度、缓存、可观测性、artifact 血缘和回滚证据。
