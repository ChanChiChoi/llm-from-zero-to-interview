# 第 52 章 设计一个大模型训练平台

前面六个部分讲了 AI Infra 的硬件、集群、训练平台、推理平台、数据模型实验平台、可观测性和治理。从本章开始进入第七部分：AI Infra 系统设计与开放题。

本章聚焦高频系统设计题：设计一个大模型训练平台。

这类题目常见于大模型平台、Infra、MLOps、LLMOps、训练系统和平台工程岗位。面试官不是想听你说“用 Kubernetes 跑训练”，而是考你是否理解训练任务从提交到调度、启动、运行、监控、checkpoint、容错、评估和产物注册的完整生命周期。

先记住一句话：

> 大模型训练平台不是脚本启动器，而是围绕 TrainingJob 生命周期构建的多租户、可观测、可恢复、可治理的训练任务管理系统。

## 52.0 本讲资料边界与第二轮精修口径

本章按 `WRITING_PLAN.md` 做第二轮精修时，主要参考 Kubernetes Job 对 run-to-completion workload 的通用控制器语义，参考 Kueue 对 queue、admission、quota、cohort 和 workload 调度的官方口径，参考 Kubeflow Trainer / Training Operator 对分布式训练任务 API 和 runtime 的抽象，参考 PyTorch `torchrun` / Elastic 对 rank、world size、rendezvous 和 worker restart 的说明，也参考 MLflow Tracking 对 experiment、run、params、metrics 和 artifacts 的实验追踪模型。

需要强调三点：

1. 本章讲的是系统设计题的稳定抽象，不绑定某个云厂商、内部平台、Kubernetes 发行版、训练 Operator 或单一 launcher。
2. 训练平台的核心不是“能创建 Pod”，而是把需求、TrainingJob 契约、准入、调度、分布式启动、数据、checkpoint、观测、容错、安全、成本和产物注册串成可审计生命周期。
3. 文中 demo 是教学级系统设计审计脚本，用于训练面试表达和设计自查，不代表生产级调度器、训练 Operator 或企业平台实现。

## 52.1 题目理解

题目可能这样问：

1. 请设计一个支持多机多卡的大模型训练平台。
2. 请设计一个公司内部训练平台，支持预训练、SFT、评估和 checkpoint。
3. 请设计一个多租户 GPU 训练平台。
4. 请设计一个支持 torchrun、DeepSpeed、Megatron、Ray 的训练平台。

这些题本质相同：设计 TrainingJob 生命周期管理系统。

## 52.2 先做需求澄清

面试时先问清楚：

1. 支持哪些训练类型？预训练、SFT、RLHF、DPO、评估？
2. 支持多大规模？单机多卡还是多机多卡？
3. GPU 集群规模多大？是否异构？
4. 是否多租户？是否有队列、配额、优先级？
5. 是否要求 checkpoint 和自动恢复？
6. 是否要求 experiment tracking？
7. 是否需要数据版本和模型仓库？
8. 是否支持抢占和低优先级队列？
9. 是否需要 Web Console、CLI、API？
10. SLO 是什么？调度等待、任务成功率、恢复时间？

需求澄清能体现你不是背模板，而是在做系统设计。

## 52.3 核心目标

训练平台核心目标：

1. 支持训练任务标准化提交。
2. 支持多机多卡分布式启动。
3. 支持 GPU 资源调度、队列和配额。
4. 支持数据、代码、镜像和配置版本化。
5. 支持日志、指标、事件和实验追踪。
6. 支持 checkpoint 保存和断点恢复。
7. 支持容错、重试和抢占恢复。
8. 支持多租户权限、安全和审计。
9. 支持模型产物注册和评估联动。
10. 支持成本归因和资源利用率治理。

这十点基本覆盖训练平台系统设计主干。

## 52.4 核心对象：TrainingJob

TrainingJob 是训练平台的核心抽象。

它不是普通 K8s Job。

TrainingJob 应该包含：

1. job name。
2. owner / tenant / project。
3. priority / queue。
4. resource request。
5. image。
6. code version。
7. entrypoint。
8. training config。
9. dataset version。
10. output artifact path。
11. checkpoint policy。
12. retry policy。
13. launcher type。
14. environment。
15. permissions。

TrainingJob 是平台理解训练任务的方式。

可以把一条训练任务抽象为：

$$
J_i=(u_i,q_i,r_i,l_i,d_i,c_i,k_i,o_i,s_i,z_i)
$$

其中 `u_i` 是 owner / tenant，`q_i` 是队列与优先级，`r_i` 是资源请求，`l_i` 是 launcher 配置，`d_i` 是 dataset / data manifest，`c_i` 是代码、镜像和最终配置，`k_i` 是 checkpoint policy，`o_i` 是 observability / experiment tracking，`s_i` 是 security / audit policy，`z_i` 是成本、产物和治理元数据。

## 52.5 TrainingJob 示例

可以用一个简化配置说明：

```yaml
apiVersion: ai.company.com/v1
kind: TrainingJob
metadata:
  name: chat-sft-v12
spec:
  owner: team-a
  queue: high-priority
  resources:
    gpuType: h100
    nodes: 8
    gpusPerNode: 8
  image: registry/train:pytorch-2.4-cuda12
  code:
    repo: git@example.com:llm/train.git
    commit: abc123
  launcher:
    type: deepspeed
  dataset:
    name: sft_mix
    version: 2026-05-30-v1
  checkpoint:
    intervalSteps: 1000
    keepLast: 3
  command: python train.py --config configs/sft.yaml
```

这个对象能让平台做校验、调度、追踪、恢复和审计。

## 52.6 总体架构

可以设计如下架构：

```text
User / CLI / Web Console
  -> Training API Server
  -> Auth / Quota / Validation
  -> TrainingJob Controller
  -> Queue / Scheduler
  -> Kubernetes / GPU Cluster
  -> Launcher Controller
  -> Training Workers
  -> Data Platform
  -> Checkpoint Store
  -> Experiment Tracker
  -> Model Registry
  -> Observability / Cost / Audit
```

各模块职责清晰：API 管提交，Controller 管生命周期，Scheduler 管资源，Launcher 管分布式启动，Observability 管可见性。

## 52.7 提交系统设计

提交入口可以包括：

1. Web Console。
2. CLI。
3. SDK。
4. REST/gRPC API。
5. CI/CD pipeline。

提交时平台要做校验：

1. 用户身份。
2. 队列权限。
3. GPU 配额。
4. 镜像合法性。
5. 数据集权限。
6. checkpoint 路径权限。
7. 配置合法性。
8. 资源规格合法性。

不合法任务应在提交阶段失败，而不是调度后才失败。

## 52.8 队列和配额

多租户训练平台必须有队列和配额。

队列维度：

1. team。
2. project。
3. priority。
4. training type。
5. GPU type。

配额维度：

1. 最大 GPU 数。
2. 最大并发任务数。
3. 最大单任务规模。
4. 存储配额。
5. 月度 GPU hour 预算。

配额既保护平台，也保护团队之间的公平性。

队列体验可以用 P95 等待时间描述：

$$
T_{\mathrm{queue,p95}}=Q_{0.95}(\{t_{\mathrm{start},i}-t_{\mathrm{submit},i}\}_{i=1}^{N})
$$

如果训练平台只说“有队列”，但不能按 team、priority、GPU type、job type 和时间窗口解释等待时间，就很难支撑资源治理。

## 52.9 调度系统设计

训练调度需要支持：

1. gang scheduling。
2. GPU 类型匹配。
3. 拓扑感知。
4. 队列优先级。
5. 配额限制。
6. 抢占。
7. 低优先级队列。
8. 资源碎片优化。
9. 数据本地性。

大模型训练不能只用普通 Pod 调度。

多机多卡任务需要所有 worker 同时启动，否则任务无法运行。

world size 和有效 batch 可以写成：

$$
W=n_{\mathrm{node}}n_{\mathrm{proc}}
$$

$$
B_{\mathrm{eff}}=B_{\mathrm{micro}}A_{\mathrm{grad}}W
$$

其中 `n_node` 是节点数，`n_proc` 是每节点进程数，`B_micro` 是每进程 micro batch，`A_grad` 是 gradient accumulation steps。系统设计题里说支持分布式训练，至少要能解释这几个量如何由平台配置和 launcher 一致生成。

## 52.10 分布式启动器

平台应封装 launcher。

支持：

1. torchrun。
2. DeepSpeed。
3. Megatron。
4. Ray。
5. MPI。
6. 自研 launcher。

Launcher 负责：

1. 计算 world size。
2. 设置 rank。
3. 设置 master address。
4. 注入环境变量。
5. 启动 worker。
6. 收集退出状态。
7. 处理失败和重试。

平台要屏蔽启动细节，让用户关注训练逻辑。

## 52.11 数据供给设计

训练平台要和数据平台打通。

TrainingJob 引用 dataset version，而不是裸路径。

平台负责：

1. 校验数据权限。
2. 解析 dataset manifest。
3. 注入数据路径。
4. 选择读取模式。
5. 配置缓存。
6. 记录数据版本。
7. 监控 data loader 吞吐。

数据供给慢会直接降低 GPU 利用率。

## 52.12 Checkpoint 设计

Checkpoint 是训练平台容错核心。

平台要支持：

1. 定期保存。
2. 手动保存。
3. 异步保存。
4. 保留策略。
5. checksum 校验。
6. manifest。
7. 跨节点恢复。
8. 从 checkpoint resume。

Checkpoint policy 应属于 TrainingJob 配置，而不是训练脚本临时决定。

一次故障最多丢失的 GPU 小时可以粗略写成：

$$
L_{\mathrm{lost}}=G\frac{t_{\mathrm{fail}}-t_{\mathrm{ckpt}}}{60}
$$

其中 `G` 是任务使用的 GPU 数，`t_fail - t_ckpt` 是距离最近可恢复 checkpoint 的分钟数。checkpoint 频率不是纯技术偏好，而是恢复目标、存储成本和 GPU 浪费之间的取舍。

## 52.13 容错和重试

训练任务失败可能来自：

1. 节点故障。
2. 网络故障。
3. 存储故障。
4. OOM。
5. 用户代码错误。
6. 数据错误。

平台要区分可重试和不可重试错误。

可重试：节点丢失、临时存储错误、可恢复通信失败。

不可重试：配置错误、权限错误、代码语法错误、数据 schema 错误。

不要无限重试用户错误。

## 52.14 可观测性设计

训练平台可观测性包括：

1. job status。
2. queue wait time。
3. scheduling events。
4. per-rank logs。
5. train loss。
6. eval loss。
7. learning rate。
8. gradient norm。
9. tokens/s。
10. GPU utilization。
11. data loader time。
12. checkpoint events。
13. cost。

用户需要一个 TrainingJob 页面，看状态、日志、指标、事件、checkpoint 和成本。

## 52.15 实验追踪联动

每个 TrainingJob 应自动创建 experiment run。

记录：

1. 参数。
2. 数据版本。
3. 代码版本。
4. 镜像版本。
5. 环境信息。
6. 指标曲线。
7. checkpoint artifact。
8. eval report。

这样训练结果才能复现和比较。

不要依赖用户手动登记实验结果。

trace / experiment / artifact 链接覆盖率可以写成：

$$
R_{\mathrm{trace}}=\frac{N_{\mathrm{linked}}}{N_{\mathrm{required}}}
$$

其中 required 至少包括 TrainingJob、final config、image digest、code commit、dataset version、checkpoint manifest、metrics、logs、eval report 和 model artifact。

## 52.16 模型仓库联动

训练完成后，平台可以把 checkpoint 注册到模型仓库。

流程：

```text
TrainingJob finished
  -> select checkpoint
  -> validate artifact
  -> register ModelVersion candidate
  -> trigger eval job
  -> attach eval report
```

这样训练产物才能进入标准发布流程。

## 52.17 安全和审计

安全设计包括：

1. 用户认证。
2. 队列权限。
3. 数据集权限。
4. secret 最小注入。
5. 镜像安全扫描。
6. 网络隔离。
7. artifact 访问控制。
8. 日志脱敏。
9. 审计记录。

训练任务运行用户代码，安全风险很高。

平台必须限制数据、密钥和网络访问。

## 52.18 成本治理

训练平台成本治理包括：

1. 提交时成本预估。
2. 运行中成本统计。
3. 按团队和项目归因。
4. failed job GPU hours。
5. idle GPU hours。
6. 低优先级队列。
7. 抢占任务。
8. checkpoint 存储成本。
9. 预算告警。

用户提交任务前应看到预估资源和成本。

平台负责人应看到集群利用率和浪费来源。

Web Console 页面可以包括：

3. 队列和配额页面。
4. TrainingJob 详情页。
5. 日志查看。
6. 指标曲线。

## 52.20 API / CLI / SDK 设计

平台应提供 API、CLI 和 SDK。

CLI 示例：

```bash
ai train submit -f job.yaml
ai train status chat-sft-v12
ai train logs chat-sft-v12 --rank 0
ai train stop chat-sft-v12
ai train resume chat-sft-v12 --checkpoint latest
```

SDK 用于自动化实验和 CI/CD。

API 用于平台集成。

三者应该共享同一套 TrainingJob schema。

## 52.21 状态机设计

TrainingJob 状态机可以是：

```text
Created -> Validating -> Queued -> Scheduling -> Running -> Succeeded
                                           -> Failed
                                           -> Preempted
                                           -> Retrying
                                           -> Cancelled
```

每次状态变化都应记录 event。

状态机让用户知道任务卡在哪一步。

## 52.22 训练平台系统设计指标和最小 demo

系统设计题不是只看架构图是否完整，也要能把设计拆成可验收门禁。可以用 16 个维度审计：

1. Requirement Clarification：是否问清训练类型、规模、多租户、抢占、恢复、发布和成本预算。
2. TrainingJob Contract：TrainingJob 是否绑定 owner、image digest、code commit、dataset version、final config、resources、distributed、checkpoint、queue 和 security。
3. Lifecycle State Machine：状态机是否覆盖 submitted、validated、queued、scheduled、starting、running、checkpointing、retrying、failed、completed 和 cancelled。
4. Admission Quota Policy：提交阶段是否做 schema、权限、配额、priority、checkpoint path 和 idempotency 校验。
5. Scheduler Resource Fit：是否支持 fair share、gang scheduling、拓扑感知、backfilling、preemption、fragmentation 和 starvation control。
6. Distributed Launcher Fit：launcher 是否能正确生成 rank、world size、rendezvous、GPU binding、NCCL 和 adapter 配置。
7. Data Config Lineage：数据、代码、镜像、环境、seed、tokenizer 和 final config 是否可追踪。
8. Checkpoint Recovery Coupling：checkpoint 是否绑定 manifest、checksum、restore drill、preemption hook 和最大丢失进度。
9. Observability Experiment Linkage：logs、metrics、events、traces、experiment run 和 artifact registry 是否互相链接。
10. Fault Recovery Flow：是否区分平台故障、用户错误、数据错误、OOM、通信失败和权限错误，并有 retry budget。
11. Security Audit Governance：身份、RBAC / ABAC、数据用途、secret、runtime 隔离、镜像供应链和审计是否覆盖。
12. Cost Capacity Governance：GPU-hours、存储、网络、失败成本、idle、预算和容量计划是否能归因。
13. Developer Self Service：Portal、CLI、SDK、模板、状态页、错误诊断和 checkpoint resume 是否可自助。
14. Interface Artifact Contract：API / CLI / SDK / Web、Model Registry、Eval Report 和 Deployment Package 是否共享契约。
15. Tradeoff Boundary Reasoning：是否能讲清资源利用率、等待时间、checkpoint、抢占、隔离、日志和自动重试的 trade-off。
16. Training Platform Design Gate：最终是否有 owner、dashboard、例外审批、审计导出和 P0 风险阻断。

综合门禁：

$$
G_{\mathrm{training\_platform\_design}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{queue,p95}}\le B_{\mathrm{queue}} \land L_{\mathrm{lost}}\le B_{\mathrm{loss}} \land R_{\mathrm{trace}}\ge \rho_{\mathrm{trace}} \land P_0=0\right]
$$

下面是一个 0 依赖 demo，用 toy design cases 检查训练平台系统设计是否覆盖关键闭环。

```python
from copy import deepcopy


class MiniTrainingPlatformSystemDesignAudit:
    GATES = [
        "requirement_clarification",
        "trainingjob_contract",
        "lifecycle_state_machine",
        "admission_quota_policy",
        "scheduler_resource_fit",
        "distributed_launcher_fit",
        "data_config_lineage",
        "checkpoint_recovery_coupling",
        "observability_experiment_linkage",
        "fault_recovery_flow",
        "security_audit_governance",
        "cost_capacity_governance",
        "developer_self_service",
        "interface_artifact_contract",
        "tradeoff_boundary_reasoning",
        "training_platform_design_gate",
    ]

    REQUIRED_JOB_FIELDS = [
        "owner", "project", "image_digest", "code_commit", "dataset_version",
        "final_config_hash", "resources", "distributed", "checkpoint_policy",
        "queue", "priority", "security_policy",
    ]
    REQUIRED_STATES = [
        "submitted", "validated", "queued", "scheduled", "starting", "running",
        "checkpointing", "retrying", "failed", "completed", "cancelled",
    ]
    REQUIRED_TRACE_LINKS = [
        "training_job", "final_config", "image_digest", "code_commit",
        "dataset_version", "checkpoint_manifest", "metrics", "logs",
        "eval_report", "model_artifact",
    ]

    @staticmethod
    def present(record, key):
        return key in record and record[key] is not None and record[key] != ""

    def coverage(self, record, fields):
        if not record:
            return 0.0
        return sum(1 for field in fields if self.present(record, field)) / len(fields)

    def world_size(self, case):
        resources = case["resources"]
        return resources["nodes"] * resources["gpus_per_node"]

    def effective_batch(self, case):
        batch = case["batch"]
        return batch["micro_batch"] * batch["grad_accum"] * self.world_size(case)

    def queue_p95_minutes(self, case):
        waits = sorted(case["queue"]["wait_minutes"])
        index = int(0.95 * (len(waits) - 1))
        return waits[index]

    def max_lost_gpu_hours(self, case):
        ckpt = case["checkpoint"]
        return round(self.world_size(case) * ckpt["max_stale_minutes"] / 60, 3)

    def trace_coverage(self, case):
        trace = case["observability"]
        return self.coverage(trace["links"], self.REQUIRED_TRACE_LINKS)

    def cost_attribution_rate(self, case):
        cost = case["cost"]
        return cost["attributed_gpu_hours"] / max(1e-12, cost["total_gpu_hours"])

    def requirement_clarification(self, case):
        req = case.get("requirements", {})
        required = ["train_types", "scale", "gpu_types", "multi_tenant", "preemption", "recovery_slo", "release_path", "cost_budget"]
        return self.coverage(req, required) == 1.0

    def trainingjob_contract(self, case):
        return self.coverage(case.get("training_job", {}), self.REQUIRED_JOB_FIELDS) == 1.0

    def lifecycle_state_machine(self, case):
        lifecycle = case.get("lifecycle", {})
        states = lifecycle.get("states", [])
        return all(state in states for state in self.REQUIRED_STATES) and lifecycle.get("legal_transitions") is True

    def admission_quota_policy(self, case):
        admission = case.get("admission", {})
        return (
            admission.get("schema_validation") is True
            and admission.get("quota_dry_run") is True
            and admission.get("dataset_permission_check") is True
            and admission.get("checkpoint_path_check") is True
            and admission.get("idempotency_key") is True
        )

    def scheduler_resource_fit(self, case):
        scheduler = case.get("scheduler", {})
        return (
            scheduler.get("gang") is True
            and scheduler.get("topology_aware") is True
            and scheduler.get("fair_share") is True
            and scheduler.get("preemption_policy") is True
            and scheduler.get("fragmentation_control") is True
            and self.queue_p95_minutes(case) <= scheduler.get("max_queue_p95_min", 0)
        )

    def distributed_launcher_fit(self, case):
        launcher = case.get("launcher", {})
        return (
            launcher.get("type") in {"torchrun", "deepspeed", "megatron", "ray", "mpi"}
            and launcher.get("world_size") == self.world_size(case)
            and launcher.get("rendezvous") is True
            and launcher.get("rank_assignment") is True
            and launcher.get("gpu_binding") is True
        )

    def data_config_lineage(self, case):
        lineage = case.get("lineage", {})
        required = ["dataset_version", "manifest_hash", "code_commit", "image_digest", "final_config_hash", "seed", "tokenizer_version"]
        return self.coverage(lineage, required) == 1.0

    def checkpoint_recovery_coupling(self, case):
        ckpt = case.get("checkpoint", {})
        return (
            ckpt.get("manifest_committed") is True
            and ckpt.get("checksum") is True
            and ckpt.get("restore_drill") is True
            and ckpt.get("preemption_hook") is True
            and self.max_lost_gpu_hours(case) <= ckpt.get("max_lost_gpu_hours", 0)
        )

    def observability_experiment_linkage(self, case):
        obs = case.get("observability", {})
        return (
            self.trace_coverage(case) >= obs.get("min_trace_coverage", 1.0)
            and obs.get("rank_logs") is True
            and obs.get("metrics") is True
            and obs.get("events") is True
            and obs.get("experiment_run") is True
        )

    def fault_recovery_flow(self, case):
        fault = case.get("fault_recovery", {})
        return (
            fault.get("taxonomy") is True
            and fault.get("retry_budget") is True
            and fault.get("backoff") is True
            and fault.get("fast_fail_user_error") is True
            and fault.get("diagnosis_report") is True
        )

    def security_audit_governance(self, case):
        security = case.get("security", {})
        return (
            security.get("identity_binding") is True
            and security.get("rbac_abac") is True
            and security.get("secret_scope") is True
            and security.get("runtime_isolation") is True
            and security.get("audit_log") is True
        )

    def cost_capacity_governance(self, case):
        cost = case.get("cost", {})
        return (
            self.cost_attribution_rate(case) >= cost.get("min_attribution_rate", 1.0)
            and cost.get("budget_alert") is True
            and cost.get("idle_tracking") is True
            and cost.get("capacity_plan") is True
        )

    def developer_self_service(self, case):
        dev = case.get("developer", {})
        return (
            dev.get("portal") is True
            and dev.get("cli") is True
            and dev.get("sdk") is True
            and dev.get("templates") is True
            and dev.get("status_page") is True
            and dev.get("resume_from_checkpoint") is True
        )

    def interface_artifact_contract(self, case):
        contract = case.get("interfaces", {})
        return (
            contract.get("shared_schema") is True
            and contract.get("api_cli_sdk_parity") is True
            and contract.get("model_registry_link") is True
            and contract.get("eval_report_link") is True
            and contract.get("artifact_contract") is True
        )

    def tradeoff_boundary_reasoning(self, case):
        tradeoff = case.get("tradeoff", {})
        required = ["utilization_wait", "checkpoint_cost", "preemption_ux", "isolation_sharing", "log_privacy", "retry_masking"]
        return self.coverage(tradeoff, required) == 1.0

    def training_platform_design_gate(self, case):
        gate = case.get("platform_gate", {})
        return (
            gate.get("enabled") is True
            and bool(gate.get("owner"))
            and gate.get("dashboard_ready") is True
            and gate.get("exception_approval") is True
            and gate.get("audit_export") is True
            and gate.get("p0_open") is False
        )

    def audit_case(self, case):
        return {gate: getattr(self, gate)(case) for gate in self.GATES}

    def run_all(self, cases):
        results = {case["case_id"]: self.audit_case(case) for case in cases}
        metrics = {}
        for gate in self.GATES:
            passed = sum(1 for result in results.values() if result[gate])
            metrics[gate] = round(passed / len(cases), 3)
        failed_cases = [case_id for case_id, result in results.items() if not all(result.values())]
        failed_gates = [gate for gate in self.GATES if any(not result[gate] for result in results.values())]
        return {
            "metrics": metrics,
            "hard_blocker_count": len(failed_cases),
            "failed_cases": failed_cases,
            "failed_gates": failed_gates,
            "training_platform_design_gate_pass": metrics["training_platform_design_gate"] == 1.0,
        }

    def example_outputs(self, case):
        return {
            "world_size": self.world_size(case),
            "effective_batch": self.effective_batch(case),
            "queue_p95_minutes": self.queue_p95_minutes(case),
            "max_lost_gpu_hours": self.max_lost_gpu_hours(case),
            "trace_coverage": round(self.trace_coverage(case), 3),
            "cost_attribution_rate": round(self.cost_attribution_rate(case), 3),
        }
```

## 52.23 关键 trade-off

系统设计中要讲 trade-off：

1. 资源利用率 vs 调度等待时间。
2. Checkpoint 频率 vs 存储和 I/O 成本。
3. 抢占能力 vs 用户体验。
4. 平台封装 vs 用户灵活性。
5. 多租户隔离 vs 资源共享效率。
6. 日志完整性 vs 隐私和成本。
7. 自动重试 vs 隐藏用户错误。

能讲 trade-off，说明你理解生产系统。

## 52.24 常见故障和应对

平台要支持常见故障定位：

1. 调度等待太久：看队列、配额、资源碎片。
2. 任务启动失败：看镜像、权限、entrypoint。
3. GPU 利用率低：看 data loader、通信、batch。
4. Loss NaN：看数据、学习率、precision、gradient。
5. OOM：看 batch、sequence、ZeRO/FSDP。
6. Hang：看 rank logs、NCCL、dataloader。
7. Checkpoint 慢：看存储吞吐和保存策略。

这些能力要体现在观测和 runbook 中。

## 52.26 常见扣分点

扣分点一：只说 Kubernetes Job。

问题：忽略 TrainingJob 抽象、分布式启动、checkpoint、数据版本和实验追踪。

扣分点二：不提 gang scheduling。

问题：多机多卡训练需要同时拿到资源，否则无法启动。

扣分点三：不提 checkpoint 和恢复。

问题：大模型训练时间长，没有容错会浪费大量 GPU 成本。

扣分点四：不提多租户和配额。

问题：内部平台一定会面对公平性和资源治理。

扣分点五：不提可观测性。

问题：训练故障复杂，没有日志、指标、事件和 trace 无法运维。

## 52.28 本章小结

本章系统设计了一个大模型训练平台。

你需要记住：

1. 大模型训练平台的核心对象是 TrainingJob。
2. 平台主链路是提交、校验、排队、调度、分布式启动、运行、观测、checkpoint、恢复、评估和模型注册。
3. 多机多卡训练需要 gang scheduling、拓扑感知和 launcher 抽象。
4. 生产训练平台必须支持数据版本、代码版本、实验追踪、checkpoint、容错、多租户、安全和成本治理。
