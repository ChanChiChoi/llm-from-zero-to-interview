# 第 40 章 实验追踪：参数、指标、日志、样本和代码版本

上一章讲了 Artifact 管理。本章讲实验追踪，也就是 experiment tracking。

大模型研发中，每天可能有大量训练、微调、评估、推理压测和 prompt 实验。如果没有实验追踪，团队很快会陷入“这个结果是哪次跑出来的”“为什么这次比上次好”“这个 checkpoint 对应哪个参数”的混乱。

先记住一句话：

> 实验追踪的目标，是把一次实验的输入、过程、输出和结论完整记录下来，让结果可比较、可复现、可解释。

## 40.0 本讲资料边界与第二轮精修口径

本讲按截至 2026-06 的稳定公开资料校准：MLflow Tracking 对 experiment、run、parameters、metrics、tags、artifacts、backend store 和 artifact store 的实验追踪抽象，Weights & Biases Experiments / Tables / Artifacts 对 metrics、hyperparameters、media、sample table 和 lineage 的记录口径，TensorBoard 对 scalar、histogram、image、text、graph 和 hparams 的可视化日志口径，DVC Experiments 对 Git commit、params、metrics、plots 和复现实验的工程边界，以及 OpenLineage 对 Job、Run、Dataset 和运行事件血缘的抽象。

本章只抽象大模型平台里的通用实验追踪能力，不绑定某个 tracking 产品、UI、时序库、日志系统、训练框架或内部平台字段名。

和上一章的分工是：

1. 第 39 章聚焦 Artifact 管理，回答“dataset、checkpoint、eval report 和 deployment package 这些产物如何被统一注册、校验和流转”。
2. 本章聚焦 Experiment Tracking，回答“一次训练、评估、prompt 实验或压测 run 的参数、指标、日志、样本、代码版本、成本和结论如何被记录、比较和复现”。

## 40.1 为什么需要实验追踪

在小实验里，你可能只看终端输出：

```text
loss=1.83 acc=0.72
```

但团队研发中会有很多变量：

1. 模型结构。
2. 数据版本。
3. 学习率。
4. batch size。
5. optimizer。
6. 训练步数。
7. seed。
8. prompt 模板。
9. eval 数据。
10. runtime 配置。
11. 代码版本。
12. 硬件环境。

如果这些没有记录，实验结果就很难复现和比较。

## 40.2 实验追踪记录什么

一次实验至少应该记录：

1. 实验 ID。
2. 实验名称。
3. 运行人。
4. 运行时间。
5. 代码版本。
6. 配置参数。
7. 数据版本。
8. 模型版本。
9. 环境信息。
10. 指标曲线。
11. 日志。
12. 输出 artifact。
13. 样本级结果。
14. 备注和结论。

实验追踪不是只保存最终分数，而是保存整个实验上下文。

## 40.3 Run 的概念

Experiment 是一组相关实验。

Run 是一次具体执行。

例如：

```text
Experiment: chat-sft-learning-rate-search
  Run 1: lr=1e-5
  Run 2: lr=2e-5
  Run 3: lr=5e-6
```

每个 run 都有自己的参数、指标、日志和 artifact。

对比实验时，比较的是多个 run。

一个 run 可以抽象为：

```math
R_i=(e_i,u_i,a_i,t_i,p_i,m_i,l_i,s_i,c_i,d_i,q_i,o_i,z_i)
```

其中 `e_i` 是 experiment id，`u_i` 是 run id，`a_i` 是 actor / owner，`t_i` 是时间与状态，`p_i` 是参数，`m_i` 是指标曲线，`l_i` 是日志，`s_i` 是样本级结果，`c_i` 是代码与环境版本，`d_i` 是数据和模型版本，`q_i` 是成本和资源，`o_i` 是输出 artifact，`z_i` 是 tags、notes 和 lineage 事件。

## 40.4 参数追踪

参数包括所有会影响结果的配置。

常见训练参数：

1. learning rate。
2. batch size。
3. global batch size。
4. optimizer。
5. weight decay。
6. warmup steps。
7. max steps。
8. gradient accumulation。
9. sequence length。
10. dropout。
11. seed。

大模型训练还要记录：

1. tensor parallel。
2. pipeline parallel。
3. data parallel。
4. ZeRO stage。
5. precision。
6. activation checkpointing。
7. flash attention。
8. tokenizer version。

没有参数追踪，就无法解释结果差异。

参数覆盖率可以写成：

```math
C_{\mathrm{param}}=\frac{1}{N_{\mathrm{param}}}\sum_{j=1}^{N_{\mathrm{param}}}\mathbf{1}[p_j\ \mathrm{recorded}]
```

这里 `N_param` 是当前任务必须记录的参数数量。训练 run 里，learning rate、global batch size、seed、optimizer、数据版本、并行策略和精度通常都应该进入这个集合。

## 40.5 指标追踪

指标包括训练过程指标和最终评估指标。

训练指标：

1. train loss。
2. eval loss。
3. learning rate。
4. gradient norm。
5. throughput。
6. tokens/s。
7. GPU utilization。
8. memory usage。
9. step time。

评估指标：

1. accuracy。
2. pass@k。
3. win rate。
4. reward score。
5. safety score。
6. tool call success rate。
7. JSON parse success rate。
8. human preference score。

指标要带 step 或 timestamp，否则无法画曲线和定位变化。

指标曲线可以表示为：

```math
M_k=\{(r_i,k,t,v)\mid r_i\in \mathcal{R},\ t\in \mathcal{T}\}
```

其中 `r_i` 是 run，`k` 是指标名，`t` 是 step 或 timestamp，`v` 是指标值。两个 run 的指标要可比，至少需要 metric definition、eval dataset、evaluator version 和 aggregation rule 一致。

## 40.6 日志追踪

日志用于排查问题。

常见日志包括：

1. training stdout/stderr。
2. framework 日志。
3. dataloader 日志。
4. distributed training 日志。
5. checkpoint 保存日志。
6. eval 日志。
7. 错误堆栈。
8. 系统事件。

日志要和 run ID 关联。

当某个 run loss 异常时，可以直接查看对应日志，而不是去集群里找已消失的 Pod。

## 40.7 样本级结果追踪

大模型实验不能只看聚合指标。

很多问题需要看样本级结果。

例如：

1. 哪些题答错了？
2. 哪些 prompt 触发拒答？
3. 哪些样本 JSON 解析失败？
4. 哪些代码题编译失败？
5. 哪些工具调用参数错了？
6. 哪些安全样本越界了？

样本级结果可以包括：

1. input。
2. prediction。
3. reference。
4. score。
5. error type。
6. model version。
7. prompt version。
8. evaluator version。

样本级追踪是模型诊断的基础。

样本级错误覆盖率可以写成：

```math
C_{\mathrm{sample}}=\frac{N_{\mathrm{samplediag}}}{N_{\mathrm{sample}}}
```

如果只存 aggregate score，不存 input hash、prediction、reference、score、error type、prompt version 和 evaluator version，就很难定位模型到底在哪类样本上退化。

## 40.8 代码版本追踪

代码版本非常重要。

实验应该记录：

1. Git commit。
2. branch。
3. 是否有未提交修改。
4. diff patch。
5. 依赖版本。
6. Docker image。
7. entrypoint。

如果代码没有固定，即使参数和数据一样，也可能无法复现。

生产平台通常要求训练任务使用镜像和 commit 绑定的代码版本。

代码可复现门禁可以写成：

```math
G_{\mathrm{code}}=\mathbf{1}[C_{\mathrm{commit}}=1 \land C_{\mathrm{dirty}}=0 \land C_{\mathrm{image}}=1 \land C_{\mathrm{entry}}=1]
```

其中 `C_commit` 表示 commit 已记录，`C_dirty` 表示运行时存在未提交改动，`C_image` 表示容器镜像 digest 已记录，`C_entry` 表示 entrypoint 和配置文件已记录。实验追踪里最危险的情况，是指标很好但无法证明当时跑的代码是什么。

## 40.9 数据版本追踪

实验必须记录使用的数据。

包括：

1. dataset name。
2. dataset version。
3. split。
4. manifest checksum。
5. sampling recipe。
6. shuffle seed。
7. tokenizer version。

数据版本变化是实验结果变化的常见原因。

如果只记录 `s3://bucket/train/`，而目录内容变了，就无法复现。

数据可复现门禁可以写成：

```math
G_{\mathrm{data}}=\mathbf{1}[C_{\mathrm{dataset}}=1 \land C_{\mathrm{manifest}}=1 \land C_{\mathrm{recipe}}=1 \land C_{\mathrm{seed}}=1]
```

其中 `C_dataset` 表示 dataset id / version 已记录，`C_manifest` 表示 manifest checksum 已记录，`C_recipe` 表示采样和清洗配方已记录，`C_seed` 表示 split、shuffle 和 sampling seed 已记录。

## 40.10 模型和 checkpoint 追踪

如果实验从某个模型继续训练，需要记录：

1. base model version。
2. init checkpoint。
3. adapter version。
4. tokenizer version。
5. resume step。
6. parent run。

输出也要记录：

1. checkpoint artifact。
2. final model artifact。
3. eval report artifact。
4. logs artifact。

实验追踪和 artifact 管理应该打通。

run 与 artifact 的血缘图可以写成：

```math
G_{\mathrm{run}}=(V_{\mathrm{run}}\cup V_{\mathrm{artifact}},E_{\mathrm{in}}\cup E_{\mathrm{out}})
```

其中输入边 `E_in` 连接 dataset、base model、checkpoint、config 和 code 到 run，输出边 `E_out` 连接 run 到 checkpoint、eval report、model artifact 和 deployment package。

## 40.11 环境追踪

环境也会影响实验。

应该记录：

1. Python 版本。
2. PyTorch 版本。
3. CUDA 版本。
4. driver 版本。
5. NCCL 版本。
6. transformers 版本。
7. deepspeed 版本。
8. GPU 类型。
9. GPU 数量。
10. 节点拓扑。
11. 容器镜像。

分布式训练中，环境差异可能导致性能和稳定性差异。

环境完整率可以写成：

```math
C_{\mathrm{env}}=\frac{N_{\mathrm{envrec}}}{N_{\mathrm{envreq}}}
```

对大模型训练，`N_envreq` 至少应覆盖 Python、框架、CUDA、driver、NCCL、容器镜像、GPU 类型、GPU 数量和并行拓扑。

## 40.12 实验对比

实验追踪平台必须支持对比多个 run。

常见对比维度：

1. 参数差异。
2. 数据版本差异。
3. 指标曲线。
4. 最终评估分数。
5. 训练耗时。
6. 资源消耗。
7. 成本。
8. 样本级错误差异。

例如你要回答：`lr=2e-5` 为什么比 `lr=1e-5` 好？

不仅要看最终分数，还要看 loss 曲线、eval 样本、训练稳定性和成本。

在 eval 口径一致时，两个 run 的指标差异可以写成：

```math
\Delta_k(a,b)=m_k(R_a)-m_k(R_b)
```

如果 eval dataset、metric definition、judge model、prompt template 或 aggregation rule 不一致，这个差异就只是一种提示，不能直接作为模型能力提升的证据。

## 40.13 实验搜索和筛选

实验多了以后，需要检索。

常见筛选条件：

1. 模型名称。
2. 数据版本。
3. 运行人。
4. 时间范围。
5. 参数范围。
6. 指标阈值。
7. 标签。
8. 状态。
9. 是否产出模型。
10. 是否进入发布候选。

如果找不到实验，记录再多也没用。

搜索可用性可以用结构化字段覆盖率近似：

```math
C_{\mathrm{index}}=\frac{N_{\mathrm{indexed}}}{N_{\mathrm{query}}}
```

常见 query field 包括 owner、experiment、model、dataset、code commit、metric threshold、tag、status、created time 和 artifact id。

## 40.14 Tags 和 Notes

实验平台通常支持 tags 和 notes。

Tags 用于结构化标记：

1. `sft`。
2. `dpo`。
3. `long-context`。
4. `debug`。
5. `release-candidate`。

Notes 用于记录人的判断：

1. 为什么跑这个实验。
2. 观察到什么现象。
3. 是否值得继续。
4. 下一步计划。

不要低估 notes 的价值。很多实验结论无法从指标自动推断。

实验结论完整率可以写成：

```math
C_{\mathrm{decision}}=\frac{N_{\mathrm{decision}}}{N_{\mathrm{candidate}}}
```

候选实验如果没有 hypothesis、观察、结论和 next action，后续团队只能重新读曲线和日志，协作效率会很低。

## 40.15 大模型实验的特殊性

大模型实验比传统 ML 实验更复杂。

原因：

1. 单次实验成本高。
2. 运行时间长。
3. 分布式环境复杂。
4. 数据版本巨大。
5. 评估指标多维。
6. 人工评估和自动评估并存。
7. prompt 和推理参数影响结果。
8. 样本级诊断很重要。

所以实验追踪必须和训练平台、评估平台、artifact 管理、成本系统打通。

大模型实验总成本可以粗略写成：

```math
K_{\mathrm{run}}=H_{\mathrm{gpu}}P_{\mathrm{gpu}}+K_{\mathrm{storage}}+K_{\mathrm{eval}}+K_{\mathrm{network}}+K_{\mathrm{ops}}
```

这里 `H_gpu` 是 GPU 小时，`P_gpu` 是单 GPU 小时价格或内部折算价，后几项分别表示存储、评估调用、网络和运维成本。成本必须和质量指标一起看，否则容易用巨大资源换很小提升。

## 40.16 Prompt 实验追踪

大模型应用中，很多实验不是训练，而是 prompt 改动。

Prompt 实验也要记录：

1. prompt template version。
2. system prompt。
3. tool description。
4. few-shot examples。
5. generation config。
6. eval dataset。
7. model version。
8. 输出结果。
9. 解析成功率。
10. 人工偏好。

Prompt 是模型行为的一部分，也要像代码和模型一样版本化。

## 40.17 Eval 实验追踪

评估实验要记录：

1. 被评模型。
2. eval dataset。
3. evaluator。
4. prompt 模板。
5. 推理参数。
6. judge model 版本。
7. metric 定义。
8. sample outputs。
9. aggregate metrics。

如果 judge model 或评估 prompt 变了，分数就可能不可比较。

评估实验追踪可以避免“分数涨了但评估口径变了”的误判。

## 40.18 成本追踪

大模型实验成本高，实验追踪应记录成本。

包括：

1. GPU 小时。
2. GPU 类型。
3. 节点数量。
4. 存储读写量。
5. 网络流量。
6. 训练 tokens。
7. 推理 tokens。
8. eval 调用成本。
9. 总成本估算。

这样团队才能比较“分数提升 0.2 是否值得多花 3 倍成本”。

成本收益比可以写成：

```math
E_{\mathrm{gain}}=\frac{\Delta Q}{K_{\mathrm{run}}-K_{\mathrm{baseline}}}
```

其中 `Delta Q` 是在同一 eval 口径下的质量提升。如果成本增加但质量提升落在噪声范围内，这个 run 不应该进入 release candidate。

## 40.19 实验状态管理

Run 可以有状态：

```text
created -> running -> succeeded -> failed -> killed -> archived
```

还可以有业务状态：

```text
debug -> candidate -> evaluated -> release_candidate -> rejected
```

状态管理帮助团队区分：

1. 哪些只是调试实验。
2. 哪些值得评估。
3. 哪些进入发布候选。
4. 哪些已经废弃。

状态转移应是受控的：

```math
G_{\mathrm{state}}=\mathbf{1}[\mathrm{transition}(s_{\mathrm{old}},s_{\mathrm{new}})\in \mathcal{A}]
```

其中 `A` 是允许的状态转移集合。比如 `debug -> release_candidate` 通常不应跳过 eval、safety 和 artifact integrity。

## 40.20 实验追踪系统架构

一个实验追踪系统可以这样设计：

```text
Experiment Tracking API
  -> Run Metadata Store
  -> Metrics Time Series Store
  -> Log Store
  -> Artifact Store
  -> Sample Result Store
  -> Search Index
  -> Lineage Graph
  -> UI / Dashboard
```

不同数据用不同存储：

1. 元数据适合关系型数据库。
2. 指标曲线适合时序存储。
3. 日志适合日志系统。
4. 大文件适合 artifact store。
5. 样本级结果适合对象存储或分析型数据库。

## 40.21 和训练平台的集成

训练平台提交任务时应自动创建 run。

运行过程中自动上报：

1. 参数。
2. 指标。
3. 日志。
4. checkpoint artifact。
5. 资源使用。
6. 错误事件。

任务完成后自动更新 run 状态。

不要依赖研究员手动复制结果到表格。

## 40.22 和评估平台的集成

评估平台应把每次 eval 也记录为 run，或作为训练 run 的子 run。

记录：

1. eval config。
2. eval dataset。
3. metrics。
4. sample outputs。
5. failure cases。
6. eval report artifact。

这样训练结果和评估结果可以关联。

实验可复现门禁可以写成：

```math
G_{\mathrm{repro}}=\mathbf{1}[G_{\mathrm{code}}=1 \land G_{\mathrm{data}}=1 \land C_{\mathrm{param}}\ge \tau_p \land C_{\mathrm{env}}\ge \tau_e \land C_{\mathrm{sample}}\ge \tau_s]
```

实验追踪总门禁可以写成：

```math
G_{\mathrm{track}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land G_{\mathrm{repro}}=1 \land G_{\mathrm{state}}=1 \land P_0=0\right]
```

其中 `C_j` 包括参数、指标、日志、样本、代码、数据、模型、环境、artifact、成本、搜索索引、lineage 和结论记录等覆盖率，`P0` 表示会导致不可复现、无法审计或错误发布的硬阻断问题数量。

## 40.23 实验追踪审计指标和最小 demo

下面这个 demo 不依赖第三方库，用一个 toy run 说明：实验追踪不是“保存几个分数”，而是要同时检查参数、指标、日志、样本、代码版本、数据版本、环境、artifact、成本、状态、搜索、血缘和实验结论。

```python
from copy import deepcopy
from pprint import pprint


class MiniExperimentTrackingAudit:
    def __init__(self, required):
        self.required = required

    @staticmethod
    def coverage(items, required):
        present = sum(1 for key in required if items.get(key))
        return round(present / len(required), 3)

    def metadata_completeness(self, run):
        return self.coverage(run, self.required["metadata"])

    def parameter_coverage(self, run):
        return self.coverage(run.get("params", {}), self.required["params"])

    @staticmethod
    def metric_step_coverage(run):
        metrics = run.get("metrics", [])
        if not metrics:
            return 0.0
        with_step = [m for m in metrics if "step" in m and "name" in m and "value" in m]
        return round(len(with_step) / len(metrics), 3)

    @staticmethod
    def log_linkage_coverage(run):
        logs = run.get("logs", [])
        if not logs:
            return 0.0
        linked = [log for log in logs if log.get("run_id") == run.get("run_id") and log.get("uri")]
        return round(len(linked) / len(logs), 3)

    def sample_result_coverage(self, run):
        samples = run.get("sample_results", [])
        if not samples:
            return 0.0
        good = []
        for sample in samples:
            if all(sample.get(key) is not None for key in self.required["sample"]):
                good.append(sample)
        return round(len(good) / len(samples), 3)

    @staticmethod
    def code_version_ready(run):
        code = run.get("code", {})
        return bool(
            code.get("commit")
            and not code.get("dirty")
            and code.get("image_digest", "").startswith("sha256:")
            and code.get("entrypoint")
        )

    @staticmethod
    def data_version_ready(run):
        data = run.get("data", {})
        return bool(
            data.get("dataset_version")
            and data.get("manifest_checksum")
            and data.get("sampling_recipe")
            and data.get("shuffle_seed") is not None
        )

    @staticmethod
    def model_checkpoint_linkage(run):
        model = run.get("model", {})
        artifacts = run.get("artifacts", [])
        artifact_types = {artifact.get("type") for artifact in artifacts}
        return bool(model.get("base_model_version") and model.get("parent_run") and "checkpoint" in artifact_types)

    def environment_coverage(self, run):
        return self.coverage(run.get("environment", {}), self.required["environment"])

    @staticmethod
    def prompt_eval_config_ready(run):
        prompt = run.get("prompt", {})
        evaluation = run.get("evaluation", {})
        return bool(
            prompt.get("template_version")
            and prompt.get("generation_config")
            and evaluation.get("dataset_version")
            and evaluation.get("metric_definitions")
            and evaluation.get("evaluator_version")
        )

    @staticmethod
    def artifact_linkage_ready(run):
        artifacts = run.get("artifacts", [])
        if not artifacts:
            return False
        for artifact in artifacts:
            if artifact.get("run_id") != run.get("run_id"):
                return False
            if not artifact.get("checksum") or not artifact.get("uri"):
                return False
        return True

    @staticmethod
    def cost_attribution_ready(run):
        cost = run.get("cost", {})
        return bool(cost.get("gpu_hours", 0) > 0 and cost.get("total_usd", 0) > 0 and cost.get("project"))

    @staticmethod
    def state_transition_ready(run):
        allowed = {
            ("created", "running"),
            ("running", "succeeded"),
            ("running", "failed"),
            ("succeeded", "evaluated"),
            ("evaluated", "release_candidate"),
            ("release_candidate", "rejected"),
        }
        transitions = run.get("state_transitions", [])
        return all(tuple(item) in allowed for item in transitions)

    @staticmethod
    def search_index_ready(run):
        indexed = set(run.get("indexed_fields", []))
        needed = {
            "owner",
            "experiment_id",
            "model_version",
            "dataset_version",
            "code_commit",
            "status",
            "tag",
            "metric",
        }
        return needed.issubset(indexed)

    @staticmethod
    def lineage_ready(run):
        nodes = set(run.get("lineage_nodes", []))
        edges = {tuple(edge) for edge in run.get("lineage_edges", [])}
        expected_edges = {
            ("dataset:sft-v4", "run:run-040-a"),
            ("code:9f3a12b", "run:run-040-a"),
            ("run:run-040-a", "artifact:ckpt-9200"),
            ("run:run-040-a", "artifact:eval-report-9200"),
        }
        return expected_edges.issubset(edges) and len(nodes) >= 6

    @staticmethod
    def decision_note_ready(run):
        return bool(run.get("hypothesis") and run.get("decision") and run.get("notes") and run.get("next_action"))

    def reproducibility_ready(self, run):
        return (
            self.code_version_ready(run)
            and self.data_version_ready(run)
            and self.parameter_coverage(run) == 1.0
            and self.environment_coverage(run) == 1.0
            and self.sample_result_coverage(run) >= 0.8
        )

    def audit_case(self, run):
        checks = {
            "run_metadata_contract": self.metadata_completeness(run) == 1.0,
            "parameter_config_capture": self.parameter_coverage(run) == 1.0,
            "metric_curve_step_capture": self.metric_step_coverage(run) == 1.0,
            "log_run_linkage": self.log_linkage_coverage(run) == 1.0,
            "sample_result_diagnosis": self.sample_result_coverage(run) == 1.0,
            "code_version_reproducibility": self.code_version_ready(run),
            "data_version_reproducibility": self.data_version_ready(run),
            "model_checkpoint_linkage": self.model_checkpoint_linkage(run),
            "environment_capture": self.environment_coverage(run) == 1.0,
            "prompt_eval_config_capture": self.prompt_eval_config_ready(run),
            "artifact_output_linkage": self.artifact_linkage_ready(run),
            "cost_attribution": self.cost_attribution_ready(run),
            "run_state_transition": self.state_transition_ready(run),
            "search_index_readiness": self.search_index_ready(run),
            "lineage_graph_readiness": self.lineage_ready(run),
            "decision_note_capture": self.decision_note_ready(run),
        }
        checks["experiment_tracking_gate"] = all(checks.values()) and self.reproducibility_ready(run)
        return checks

    def audit_suite(self, cases):
        metric_names = [name for name in self.audit_case(cases[0]) if name != "experiment_tracking_gate"]
        scores = {}
        for metric in metric_names:
            passed = sum(1 for case in cases if self.audit_case(case)[metric])
            scores[metric] = round(passed / len(cases), 3)

        failed_cases = []
        failed_gates = []
        for case in cases[1:]:
            checks = self.audit_case(case)
            if not checks["experiment_tracking_gate"]:
                failed_cases.append(case["case"])
            failed_gates.extend([name for name, passed in checks.items() if not passed])

        return {
            "scores": scores,
            "hard_blocker_count": len(failed_cases),
            "failed_cases": failed_cases,
            "failed_gates": sorted(set(failed_gates)),
            "experiment_tracking_gate_pass": all(
                self.audit_case(case)["experiment_tracking_gate"] for case in cases
            ),
        }
```

## 40.24 常见误区

误区一：实验追踪只记录最终指标。

最终指标不够，还要记录参数、数据、代码、环境、日志、样本和 artifact。

误区二：日志在训练机器上就行。

训练机器可能销毁，日志必须集中保存并和 run ID 关联。

误区三：prompt 实验不需要追踪。

Prompt 变化会显著影响模型行为，必须版本化和追踪。

误区四：实验分数可以直接横向比较。

只有 eval 数据、评估代码、prompt、judge model 和 metric 一致时，分数才可比。

误区五：成本不是实验指标。

大模型实验成本高，成本本身就是重要指标。

## 40.25 面试常见追问

问题一：实验追踪系统要记录哪些信息？

可以回答：记录参数、指标、日志、数据版本、代码版本、模型版本、环境、样本级结果、artifact、成本、运行状态和备注结论。

问题二：为什么样本级结果重要？

可以回答：聚合指标只能说明总体变化，样本级结果能定位具体失败类型，例如 JSON 解析失败、工具调用错误、安全越界或某类题退化。

问题三：如何保证实验可复现？

可以回答：固定代码 commit、镜像、依赖、数据版本、manifest checksum、模型版本、参数、seed、环境和输出 artifact，并记录完整 run metadata。

问题四：为什么 eval run 也要追踪？

可以回答：评估结果依赖 eval 数据、评估代码、prompt、judge model、推理参数和 metric 定义，任何一项变化都会影响分数。

## 40.27 本章小结

本章讲了实验追踪。

你需要记住：

1. 实验追踪的目标是让实验结果可比较、可复现、可解释。
2. 一次 run 应记录参数、指标、日志、样本、数据版本、模型版本、代码版本、环境和 artifact。
3. 大模型实验必须重视样本级结果、prompt 版本、eval 配置和成本。
4. 实验追踪系统要和训练平台、评估平台、artifact store、lineage graph 和成本系统打通。
5. 没有实验追踪，团队无法可靠地判断模型改进来自哪里。
