# 第 50 章 安全治理：密钥、权限、数据合规、供应链和模型访问控制

上一章讲了资源利用率优化。本章讲 AI Infra 安全治理。

AI Infra 的安全边界非常长：数据进入平台、训练任务读取数据、模型产物保存、推理服务暴露 API、RAG 检索企业知识、Agent 调用工具、日志和 trace 保存用户输入，每一处都可能出问题。

先记住一句话：

> AI Infra 安全治理不是给服务加一个鉴权中间件，而是覆盖身份、权限、密钥、数据、模型、供应链、运行时、审计和合规的系统工程。

## 50.0 本讲资料边界与第二轮精修口径

本章按 `WRITING_PLAN.md` 做第二轮精修时，主要参考 Kubernetes RBAC、Secret、Pod Security Standards、NetworkPolicy 和审计日志的官方口径，参考 NIST AI RMF / Generative AI Profile 对 AI 风险治理、数据治理、测量和管理的通用框架，参考 SLSA 对软件供应链 provenance、build integrity 和 artifact 可追溯的定义，也参考 OWASP LLM Top 10 对 prompt injection、sensitive information disclosure、supply chain、excessive agency 和 overreliance 等应用风险的分类。

需要强调三点：

1. 本章只讨论防御性安全治理、审计和平台门禁，不提供漏洞利用、越权访问、绕过审计或规避安全控制的方法。
2. 安全治理不能只落在推理 API，也必须进入训练任务、数据平台、模型仓库、RAG / Agent、运行时、供应链、日志和事故响应。
3. 文中 demo 是 0 依赖教学审计脚本，用于解释安全门禁如何串联，不代表生产级安全扫描器、合规判断或法律建议。

## 50.1 为什么 AI Infra 安全更复杂

AI Infra 安全复杂，原因包括：

1. 数据类型敏感。
2. 模型可能记忆训练数据。
3. prompt 和 trace 可能包含隐私。
4. 模型权重是高价值资产。
5. Agent 工具调用可能产生副作用。
6. RAG 可能越权检索知识。
7. 训练任务运行任意代码。
8. 镜像和依赖供应链复杂。
9. 多租户共享 GPU 集群。

传统 Web 安全仍然需要，但不够。

AI Infra 还要面对模型、数据和工具调用带来的新风险。

## 50.2 安全治理范围

AI Infra 安全治理至少包括：

1. 身份认证。
2. 权限控制。
3. 密钥管理。
4. 数据合规。
5. 多租户隔离。
6. 模型访问控制。
7. 供应链安全。
8. 运行时安全。
9. 日志和 trace 安全。
10. 审计和响应。

安全治理要贯穿训练、评估、推理、RAG、Agent 和平台管理。

## 50.3 身份认证

身份认证回答：你是谁。

常见身份包括：

1. 人类用户。
2. 服务账号。
3. 训练任务。
4. 推理服务。
5. Agent。
6. CI/CD 流水线。
7. 外部应用。

AI Infra 中不能只认证人，还要认证任务和服务。

例如训练任务读取数据时，应该使用任务绑定的 service account，而不是平台管理员的长期凭据。

可以把一次安全审计样本抽象为：

$$
S_i=(a_i,w_i,r_i,d_i,m_i,t_i,k_i,p_i,l_i,o_i,z_i)
$$

其中 `a_i` 是 actor，`w_i` 是 workload，`r_i` 是 resource，`d_i` 是 data classification，`m_i` 是 model / artifact，`t_i` 是 tool，`k_i` 是 secret，`p_i` 是 policy decision，`l_i` 是 log / trace，`o_i` 是 owner，`z_i` 是审计证据。

身份绑定覆盖率：

$$
C_{\mathrm{id}}=\frac{N_{\mathrm{bound}}}{N_{\mathrm{workload}}}
$$

其中 `N_bound` 是绑定了人类用户、service account、任务身份和租户的 workload 数，`N_workload` 是总 workload 数。

## 50.4 权限控制

权限控制回答：你能做什么。

权限对象包括：

1. Dataset。
2. Model。
3. ModelVersion。
4. Checkpoint。
5. Eval report。
6. Endpoint。
7. Tool。
8. Knowledge Base。
9. Trace。
10. Secret。

权限动作包括：

1. read。
2. write。
3. execute。
4. deploy。
5. delete。
6. approve。
7. export。

权限要细到资源和动作，不能只有“管理员”和“普通用户”。

访问决策可以写成门禁：

$$
G_{\mathrm{access}}=\mathbf{1}\left[I(a)=1 \land P(a,r,e)=\mathrm{allow} \land L(r)\le L(a) \land U(r)\subseteq U(a)\right]
$$

其中 `I(a)` 表示身份认证通过，`P(a,r,e)` 是在环境 `e` 下 actor `a` 对资源 `r` 的策略决策，`L(r)` 是资源敏感等级，`L(a)` 是 actor 可访问等级，`U(r)` 是资源允许用途集合，`U(a)` 是 actor 本次任务声明用途集合。

## 50.5 RBAC 和 ABAC

RBAC 是 role-based access control，基于角色的权限控制。

例如：

1. data scientist。
2. platform admin。
3. model reviewer。
4. security auditor。

ABAC 是 attribute-based access control，基于属性的权限控制。

例如根据：

1. 租户。
2. 数据等级。
3. 地域。
4. 项目。
5. 模型安全等级。
6. 用户部门。

做动态判断。

AI Infra 通常需要 RBAC + ABAC。

例如某用户是研究员，但只能访问自己项目下的低敏数据。

过度授权率：

$$
R_{\mathrm{over}}=\frac{N_{\mathrm{overprivileged}}}{N_{\mathrm{binding}}}
$$

其中 `N_overprivileged` 是拥有超出当前任务需要权限的绑定数，`N_binding` 是权限绑定总数。安全治理要让 `R_over` 趋近 0，而不是让“方便”覆盖最小权限原则。

## 50.6 密钥管理

密钥包括：

1. API key。
2. 数据库密码。
3. 对象存储凭据。
4. 外部工具 token。
5. SSH key。
6. 模型服务访问 token。
7. KMS key。

密钥管理原则：

1. 不写入代码。
2. 不写入镜像。
3. 不写入日志。
4. 使用 secret manager。
5. 最小权限。
6. 定期轮换。
7. 支持吊销。
8. 访问审计。

Agent 工具调用尤其要注意，不要把高权限密钥暴露给模型或 prompt。

密钥暴露率：

$$
R_{\mathrm{secret}}=\frac{N_{\mathrm{secret,exposed}}}{N_{\mathrm{secret,total}}}
$$

密钥轮换新鲜度可以写成：

$$
C_{\mathrm{rotate}}=\frac{N_{\mathrm{rotated\_within\_ttl}}}{N_{\mathrm{secret,total}}}
$$

其中 `N_secret,exposed` 是出现在代码、镜像、日志、trace、prompt 或无权限环境中的密钥数，`N_rotated_within_ttl` 是在 TTL 内轮换的密钥数。

## 50.7 训练任务中的密钥

训练任务可能需要访问数据和 artifact。

错误做法：

1. 在启动命令中传密钥。
2. 把密钥写进环境变量后打印日志。
3. 把密钥打进镜像。
4. 多租户共享同一个高权限 token。

更好的方式：

1. 使用短期凭证。
2. 使用 service account。
3. 按任务注入最小权限 secret。
4. 任务结束后自动失效。
5. 日志脱敏。

训练任务是运行任意代码的环境，密钥暴露风险很高。

训练任务密钥门禁：

$$
G_{\mathrm{job\_secret}}=\mathbf{1}\left[C_{\mathrm{short}}\ge \tau \land R_{\mathrm{secret}}=0 \land A_{\mathrm{secret}}=1 \land E_{\mathrm{log}}=1\right]
$$

其中 `C_short` 是短期凭证覆盖率，`A_secret` 是 secret 访问审计是否开启，`E_log` 是日志脱敏是否开启。

## 50.8 数据合规

数据合规关注数据是否允许被使用。

问题包括：

1. 数据是否包含个人信息。
2. 数据是否有使用授权。
3. 数据是否允许训练模型。
4. 数据是否允许跨地域处理。
5. 数据是否允许长期保存。
6. 数据是否允许用于评估或日志。
7. 数据删除请求如何处理。

数据平台要记录数据来源、授权、敏感等级、地域和使用限制。

不能把合规信息放在文档里靠人工记忆。

数据合规覆盖率：

$$
C_{\mathrm{compliance}}=\frac{N_{\mathrm{classified\_authorized}}}{N_{\mathrm{dataset}}}
$$

其中 `N_classified_authorized` 是同时有来源、授权、敏感等级、地域、用途和保留策略记录的数据集数。没有这些字段的数据，不应该进入训练、评估或日志闭环。

## 50.9 数据分级

数据可以分级：

1. Public。
2. Internal。
3. Confidential。
4. Restricted。

不同等级对应不同控制：

1. 谁能访问。
2. 是否能训练。
3. 是否能导出。
4. 是否能进入日志。
5. 是否能跨地域。
6. 是否需要脱敏。
7. 是否需要审批。

训练平台提交任务时，应检查 dataset 的数据等级和任务权限。

数据分级传播门禁：

$$
G_{\mathrm{class}}=\mathbf{1}\left[L_{\mathrm{artifact}}\ge \max_{d\in D_{\mathrm{input}}} L_d\right]
$$

其中 `L_artifact` 是训练产物、模型、checkpoint 或 eval report 的等级，`D_input` 是输入数据集合。产物等级不能低于输入数据最高等级。

## 50.10 PII 和敏感信息

PII 和敏感信息要在多个阶段处理：

1. 数据接入。
2. 数据清洗。
3. 训练数据构建。
4. 推理请求。
5. 日志和 trace。
6. 评估样本。
7. Agent 工具结果。

处理方式包括：

1. 检测。
2. 脱敏。
3. 删除。
4. 权限隔离。
5. 短期留存。
6. 审计。

敏感数据进入模型训练后，风险很难彻底消除，所以源头治理很重要。

PII 脱敏覆盖率：

$$
C_{\mathrm{pii}}=\frac{N_{\mathrm{pii,redacted}}}{N_{\mathrm{pii,detected}}}
$$

敏感数据阻断率：

$$
B_{\mathrm{sensitive}}=\frac{N_{\mathrm{sensitive,blocked}}}{N_{\mathrm{sensitive,total}}}
$$

其中 PII 和敏感信息要在数据、请求、日志、trace、评估样本和工具结果中统一处理。

## 50.11 多租户隔离

多租户隔离包括：

1. 身份隔离。
2. 数据隔离。
3. 模型隔离。
4. 资源隔离。
5. 网络隔离。
6. 日志隔离。
7. 缓存隔离。
8. trace 隔离。
9. 成本隔离。

不要只靠 Kubernetes namespace。

例如 RAG 缓存跨租户命中，或者向量索引没有权限过滤，都会造成严重数据泄露。

多租户越权命中率：

$$
R_{\mathrm{tenant\_leak}}=\frac{N_{\mathrm{cross\_tenant\_hit}}}{N_{\mathrm{access\_check}}}
$$

多租户隔离目标不是“命中后不展示”，而是让检索、缓存、日志、trace、工具和模型 artifact 在命中前就被权限和租户边界过滤。

## 50.12 模型访问控制

模型本身也是受保护资产。

需要控制：

1. 谁能调用模型。
2. 谁能下载权重。
3. 谁能部署模型。
4. 谁能查看 eval report。
5. 谁能访问敏感模型输出。
6. 哪些模型可以跨租户使用。
7. 哪些模型禁止导出。

企业自训模型可能包含内部知识、业务策略或专有能力，不能当普通文件公开。

模型访问门禁：

$$
G_{\mathrm{model\_access}}=\mathbf{1}\left[P(a,m,\mathrm{action})=\mathrm{allow} \land L(m)\le L(a) \land E_{\mathrm{download}}=1 \land E_{\mathrm{deploy}}=1\right]
$$

其中 `m` 是模型或模型版本，`action` 可以是 call、download、deploy、export 或 delete，`E_download` 和 `E_deploy` 是下载和部署审计证据。

## 50.13 模型输出安全

模型输出也要治理。

风险包括：

1. 泄露敏感信息。
2. 生成不安全内容。
3. 输出越权知识。
4. 错误调用工具。
5. 生成违规代码。
6. 幻觉导致错误决策。

治理手段：

1. 输出安全检查。
2. 高风险场景人工确认。
3. 工具调用权限校验。
4. 安全评估。
5. 线上监控。
6. 违规样本回流。

输出安全通过率：

$$
C_{\mathrm{output}}=\frac{N_{\mathrm{safe\_output}}}{N_{\mathrm{checked\_output}}}
$$

高风险输出人工确认覆盖率：

$$
C_{\mathrm{review}}=\frac{N_{\mathrm{high\_risk\_reviewed}}}{N_{\mathrm{high\_risk}}}
$$

输出安全不是只靠拒答；它还要支持越权知识阻断、工具动作确认、风险分级、监控和回归样本回流。

## 50.14 RAG 权限安全

RAG 安全的核心是检索阶段权限过滤。

要保证：

1. 文档有权限标签。
2. chunk 继承文档权限。
3. embedding 和索引带权限元数据。
4. 检索时强制过滤。
5. citation 不暴露无权限来源。
6. trace 存储脱敏。

错误做法是把无权限文档交给模型，再要求模型不要说。

权限必须由平台强制执行。

RAG 权限阻断率：

$$
B_{\mathrm{rag}}=\frac{N_{\mathrm{unauthorized\_chunk\_blocked}}}{N_{\mathrm{unauthorized\_chunk}}}
$$

引用安全覆盖率：

$$
C_{\mathrm{citation}}=\frac{N_{\mathrm{citation\_authorized}}}{N_{\mathrm{citation}}}
$$

其中权限过滤必须在检索和上下文组装前完成，不能把无权限 chunk 交给模型后再要求模型保密。

## 50.15 Agent 工具安全

Agent 工具调用有副作用风险。

工具可以分级：

1. 只读工具。
2. 低风险写工具。
3. 高风险写工具。
4. 外部支付或删除类工具。

高风险工具需要：

1. 用户确认。
2. 权限校验。
3. 参数校验。
4. 审计日志。
5. 限流。
6. 超时。
7. 回滚或补偿机制。

Agent 不能因为模型“想调用”就直接调用高权限工具。

高风险工具确认覆盖率：

$$
C_{\mathrm{tool\_confirm}}=\frac{N_{\mathrm{high\_risk\_confirmed}}}{N_{\mathrm{high\_risk\_tool\_call}}}
$$

工具越权率：

$$
R_{\mathrm{tool\_unauth}}=\frac{N_{\mathrm{unauthorized\_tool\_call}}}{N_{\mathrm{tool\_call}}}
$$

Agent 工具安全的目标是让工具调用满足权限、参数、确认、限流、超时、审计和补偿条件。

## 50.16 供应链安全

AI Infra 供应链包括：

1. 容器镜像。
2. Python 包。
3. CUDA 库。
4. 模型权重。
5. tokenizer 文件。
6. 数据集。
7. 第三方工具。
8. 开源 runtime。

风险包括：

1. 恶意依赖。
2. 漏洞镜像。
3. 被篡改权重。
4. 不安全反序列化。
5. 数据污染。
6. 未知许可证风险。

供应链安全要覆盖从构建到部署的全过程。

供应链完整性覆盖率：

$$
C_{\mathrm{supply}}=\frac{N_{\mathrm{artifact\_with\_provenance}}}{N_{\mathrm{artifact,total}}}
$$

高危依赖阻断率：

$$
B_{\mathrm{critical}}=\frac{N_{\mathrm{critical\_blocked}}}{N_{\mathrm{critical\_found}}}
$$

其中 provenance、checksum、签名、依赖锁定、漏洞扫描和许可证记录要从构建延续到部署。

## 50.17 镜像安全

镜像安全要求：

1. 基础镜像受控。
2. 依赖版本锁定。
3. 漏洞扫描。
4. 镜像签名。
5. 禁止 root 运行。
6. 最小化镜像。
7. 不包含密钥。
8. 镜像来源可追溯。

训练任务镜像尤其危险，因为用户可能提交自定义代码和依赖。

平台应对镜像来源和权限做控制。

镜像安全门禁：

$$
G_{\mathrm{image}}=\mathbf{1}\left[D_{\mathrm{digest}}=1 \land S_{\mathrm{signed}}=1 \land V_{\mathrm{critical}}=0 \land R_{\mathrm{root}}=0 \land K_{\mathrm{embedded}}=0\right]
$$

其中 `D_digest` 表示 digest 固定，`S_signed` 表示签名存在，`V_critical` 是未处理高危漏洞数，`R_root` 表示 root 运行风险，`K_embedded` 是镜像内嵌密钥数。

## 50.18 模型 artifact 安全

模型 artifact 安全包括：

1. 权重 checksum。
2. artifact 签名。
3. 安全格式。
4. tokenizer 文件校验。
5. model config 校验。
6. 下载权限。
7. 部署前校验。

不安全的模型文件可能在加载时触发风险。

生产环境应避免加载来源不明的模型 artifact。

模型 artifact 完整性门禁：

$$
G_{\mathrm{artifact}}=\mathbf{1}\left[C_{\mathrm{checksum}}=1 \land C_{\mathrm{signature}}=1 \land C_{\mathrm{format}}=1 \land C_{\mathrm{tokenizer}}=1 \land C_{\mathrm{permission}}=1\right]
$$

其中 checksum、signature、安全格式、tokenizer / config 绑定和下载权限必须同时满足。

## 50.19 运行时安全

运行时安全包括：

1. 容器隔离。
2. 非 root 用户。
3. 只读文件系统。
4. 网络策略。
5. syscall 限制。
6. GPU 设备隔离。
7. secret 最小注入。
8. egress 控制。
9. 资源限制。

训练任务和推理服务都应该有运行时安全边界。

多租户环境中尤其重要。

运行时加固覆盖率：

$$
C_{\mathrm{runtime}}=\frac{N_{\mathrm{hardened}}}{N_{\mathrm{workload}}}
$$

其中 hardened 至少包括非 root、只读文件系统、最小 secret 注入、egress 控制、网络策略、资源限制和 GPU 设备隔离。

## 50.20 日志和 Trace 安全

日志和 trace 可能包含：

1. prompt。
2. 用户输入。
3. 文档片段。
4. 工具参数。
5. 工具输出。
6. 模型输出。
7. 错误堆栈中的密钥。

治理要求：

1. 脱敏。
2. 字段级权限。
3. TTL。
4. 加密存储。
5. 访问审计。
6. 采样策略。

排障需要日志，但不能以牺牲隐私为代价。

日志 Trace 脱敏覆盖率：

$$
C_{\mathrm{log\_redact}}=\frac{N_{\mathrm{redacted\_field}}}{N_{\mathrm{sensitive\_field}}}
$$

留存合规率：

$$
C_{\mathrm{retention}}=\frac{N_{\mathrm{within\_ttl}}}{N_{\mathrm{log\_item}}}
$$

排障 trace 要能回答问题，但不能无限期保存原始 prompt、工具参数、文档片段或密钥。

## 50.21 审计

审计要记录关键动作：

1. 数据访问。
2. 模型下载。
3. 模型部署。
4. endpoint 调用。
5. 权限变更。
6. secret 访问。
7. 工具调用。
8. 高风险 Agent 操作。
9. 发布和回滚。
10. 数据删除和导出。

审计日志要不可随意篡改，并能按用户、租户、模型、数据集、时间查询。

审计覆盖率：

$$
C_{\mathrm{audit}}=\frac{N_{\mathrm{audited\_event}}}{N_{\mathrm{critical\_event}}}
$$

不可篡改覆盖率：

$$
C_{\mathrm{immut}}=\frac{N_{\mathrm{immutable\_event}}}{N_{\mathrm{audited\_event}}}
$$

审计日志至少要覆盖数据、模型、endpoint、secret、工具、权限、发布、回滚、删除和导出。

## 50.22 安全事件响应

安全事件包括：

1. 密钥泄露。
2. 越权访问。
3. 数据误用。
4. 模型权重泄露。
5. 供应链漏洞。
6. 高风险工具误调用。
7. 日志泄露敏感信息。

响应流程：

1. 识别事件。
2. 隔离影响。
3. 吊销密钥。
4. 停止相关服务。
5. 审计访问范围。
6. 修复根因。
7. 复盘和补充控制。

安全事件响应需要预案，不能等事故发生再想。

安全响应就绪门禁：

$$
G_{\mathrm{response}}=\mathbf{1}\left[C_{\mathrm{detect}}\ge \tau_d \land C_{\mathrm{revoke}}\ge \tau_r \land C_{\mathrm{scope}}\ge \tau_s \land C_{\mathrm{postmortem}}\ge \tau_p\right]
$$

其中 `C_detect` 是检测覆盖率，`C_revoke` 是吊销和隔离动作覆盖率，`C_scope` 是影响范围审计覆盖率，`C_postmortem` 是复盘行动项覆盖率。

## 50.23 安全治理系统架构

一个 AI Infra 安全治理架构：

```text
Identity Provider
  -> Policy Engine
  -> Secret Manager
  -> Data Catalog / Classification
  -> Model Registry Access Control
  -> Runtime Security
  -> Supply Chain Scanner
  -> Audit Log
  -> Security Monitoring
```

Policy Engine 负责统一判断权限。

Secret Manager 管密钥。

Data Catalog 管数据等级和合规信息。

Audit Log 支撑追溯。

安全治理总门禁：

$$
G_{\mathrm{security}}=\prod_{j=1}^{16}G_j
$$

其中 16 个 `G_j` 分别覆盖身份、权限、密钥、数据合规、PII、多租户、模型访问、输出安全、RAG、Agent、供应链、artifact、运行时、日志 trace、审计响应和最终平台门禁。

## 50.24 安全治理审计指标和最小 demo

把本章落到平台验收时，可以用 16 个门禁：

1. Identity Workload Binding：人类用户、service account、训练任务、推理服务、Agent、CI/CD 和外部应用身份是否绑定到 workload。
2. RBAC ABAC Policy Fit：RBAC、ABAC、deny by default、最小权限、用途和数据等级是否进入统一策略引擎。
3. Secret Lifecycle Management：密钥是否使用 secret manager、短期凭证、最小权限、轮换、吊销、访问审计和日志脱敏。
4. Data Compliance Classification：数据来源、授权、敏感等级、地域、用途、保留和删除请求是否结构化记录。
5. PII Sensitive Data Control：PII / 敏感信息是否在数据、请求、日志、trace、评估和工具结果中检测、脱敏、阻断和审计。
6. Tenant Isolation Boundary：租户的身份、数据、模型、资源、网络、缓存、日志、trace 和成本是否隔离。
7. Model Access Control：模型调用、权重下载、部署、导出、eval report 和敏感输出是否受权限与审计控制。
8. Model Output Safety Gate：模型输出是否经过安全策略、高风险人工确认、工具动作保护、监控和违规样本回流。
9. RAG Permission Enforcement：RAG 是否在检索和上下文组装前强制权限过滤，并保证 citation 和 trace 不泄露无权限来源。
10. Agent Tool Safety Gate：Agent 工具调用是否有权限、参数校验、高风险确认、限流、超时、审计和补偿机制。
11. Supply Chain Integrity：镜像、依赖、runtime、数据集、模型权重、tokenizer 和工具是否有 provenance、checksum、签名、扫描和许可证记录。
12. Model Artifact Integrity：模型权重、tokenizer、config、adapter、量化包和 deployment package 是否有 checksum、签名、安全格式和权限。
13. Runtime Security Boundary：训练 / 推理 workload 是否非 root、只读文件系统、网络策略、egress 控制、最小 secret、GPU 隔离和资源限制。
14. Log Trace Privacy Governance：日志和 trace 是否脱敏、字段级限权、TTL、加密、采样、访问审计和删除覆盖。
15. Audit Incident Response Readiness：审计日志是否覆盖关键事件且不可篡改，安全事件响应是否能检测、隔离、吊销、定界、修复和复盘。
16. Security Governance Gate：最终是否有 owner、安全策略、例外审批、审计证据、响应预案和 P0 风险阻断。

综合门禁：

$$
G_{\mathrm{security}}=\prod_{j=1}^{16}G_j
$$

## 50.25 常见误区

误区一：AI Infra 安全就是 API 鉴权。

还需要数据、模型、工具、trace、供应链和运行时安全。

误区二：embedding 不敏感。

embedding 可能泄露语义信息，仍需权限和隔离。

误区三：模型权重只是普通文件。

模型权重是高价值资产，可能包含专有能力和训练数据风险。

误区四：Agent 工具调用和普通 API 调用一样。

Agent 调用由模型决策，必须加强权限、确认、审计和副作用控制。

误区五：日志越完整越好。

日志和 trace 可能包含敏感信息，必须脱敏、限权和设置留存策略。

## 50.26 面试常见追问

问题一：AI Infra 安全治理包括哪些方面？

可以回答：包括身份认证、权限控制、密钥管理、数据合规、多租户隔离、模型访问控制、供应链安全、运行时安全、日志 trace 安全、审计和事件响应。

问题二：如何保护训练任务中的密钥？

可以回答：使用 service account 和短期凭证，按任务最小权限注入 secret，不写入代码、镜像、命令行或日志，任务结束后失效，并记录访问审计。

问题三：企业 RAG 如何防止越权？

可以回答：文档和 chunk 带权限标签，embedding 和索引保存权限元数据，检索时平台强制 metadata filter 或物理隔离，citation 和 trace 也要做权限控制。

问题四：模型供应链安全怎么做？

可以回答：控制模型来源，对权重、tokenizer、config 做 checksum 和签名校验，使用安全格式，扫描镜像和依赖漏洞，部署前做兼容和安全检查。

## 50.28 本章小结

本章讲了 AI Infra 安全治理。

你需要记住：

1. AI Infra 安全覆盖身份、权限、密钥、数据、模型、供应链、运行时、日志、trace 和审计。
2. 数据和模型都是高价值资产，必须有分级、权限和血缘。
3. 训练任务和 Agent 工具调用是高风险场景，需要最小权限、短期凭证和审计。
4. RAG 权限必须在检索阶段强制执行，不能依赖模型判断。
5. 供应链安全要覆盖镜像、依赖、模型 artifact、tokenizer、数据集和 runtime。
6. 安全治理必须产品化和平台化，不能只靠人工规范。
