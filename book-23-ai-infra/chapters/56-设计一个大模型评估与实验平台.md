# 第 56 章 设计一个大模型评估与实验平台

上一章设计了多租户 GPU 集群调度系统。本章继续系统设计题：设计一个大模型评估与实验平台。

评估与实验平台是大模型研发和发布的质量控制中心。训练得到一个模型、改了一个 prompt、更新了一个 RAG 配置、上线了一个 Agent，都需要评估和实验系统回答：它是否真的更好？是否更安全？是否更便宜？是否可以发布？

先记住一句话：

> 大模型评估与实验平台的核心，是把模型、prompt、数据、指标、样本、人工判断、线上反馈和发布门禁连接成可复现的质量决策系统。

## 56.0 本讲资料边界与第二轮精修口径

第二轮精修时，本章按“评估平台系统设计 + 质量决策闭环”校准，不绑定某个评估产品、实验平台或内部发布系统。

参考口径包括：

1. OpenAI Evals：把 eval 理解为评估 LLM 或 LLM 系统的框架，并强调可以构建自定义 eval 和私有 eval，用来衡量模型版本变化对具体用例的影响。
2. MLflow GenAI evaluation / monitoring：把 evaluation dataset、human feedback、LLM-as-a-Judge、systematic evaluation 和 production monitoring 放在 LLM / Agent 生命周期里理解。
3. OpenTelemetry 生成式 AI 语义约定：把模型调用、request / response、token、latency、span 属性和观测信号作为评估、trace 和线上监控可关联的基础。
4. Kubernetes Job / batch workload 口径：把 EvalJob 看成 run-to-completion 的批处理任务，需要状态、重试、失败分类、资源预算和 artifact 输出。

本章的目标不是教你背 benchmark 名称，而是把“模型是否可以发布”回答成一套可复现证据链：评估数据版本清楚，推理配置固定，指标定义有版本，LLM-as-Judge 有校准，人评有质量控制，样本级结果可回放，切片退化能发现，在线实验有 guardrail，评估报告能进入模型仓库和发布系统。

## 56.1 题目理解

面试题可能这样问：

1. 请设计一个大模型评估平台。
2. 请设计一个支持离线评测、人工评测和在线 A/B 的系统。
3. 请设计一个模型实验追踪与评估平台。
4. 如何设计模型发布前的质量门禁系统？

这些题的核心是：如何系统化评估模型和大模型应用，而不是手写 benchmark 脚本。

## 56.2 需求澄清

先问：

1. 评估对象是什么？模型、prompt、RAG、Agent 还是工具？
2. 支持哪些任务？问答、代码、工具调用、安全、长上下文？
3. 是否需要离线批量评测？
4. 是否需要人工评测？
5. 是否需要 LLM-as-Judge？
6. 是否需要在线 A/B 测试？
7. 是否需要和发布系统联动？
8. 是否需要样本级结果和失败分析？
9. 是否多租户？是否有权限和审计？
10. 评估成本和时延要求是什么？

评估平台的设计取决于评估对象和决策场景。

## 56.3 核心目标

平台目标：

1. 管理 eval dataset。
2. 管理 eval suite。
3. 支持多模型、多版本评估。
4. 支持 prompt/RAG/Agent 评估。
5. 支持离线批量评测。
6. 支持人工评测。
7. 支持 LLM-as-Judge。
8. 支持在线 A/B 测试。
9. 支持样本级结果和失败分析。
10. 支持评估报告、发布门禁和实验追踪。

一句话：评估平台要把“跑分”变成“质量决策”。

## 56.4 总体架构

可以设计如下：

```text
User / CI / Release System
  -> Eval API / Web Console
  -> Eval Dataset Registry
  -> Eval Suite Registry
  -> Eval Job Scheduler
  -> Inference Runner
  -> Metric Engine
  -> LLM Judge Service
  -> Human Review System
  -> Result Store
  -> Experiment Tracker
  -> Report Generator
  -> Quality Gate
  -> Model Registry / Release System
```

关键链路：选择模型和评估集，运行推理，计算指标，生成报告，决定是否进入发布。

## 56.5 Eval Dataset Registry

Eval dataset registry 管理评估数据集。

每个 eval dataset 记录：

1. 名称。
2. 版本。
3. 任务类型。
4. 样本数量。
5. 数据来源。
6. 标签或参考答案。
7. 难度分布。
8. 语言和领域分布。
9. 是否可用于训练。
10. 权限和敏感等级。

Eval 数据必须版本化，并与训练数据隔离。

## 56.6 Eval Suite

Eval suite 是一组评估任务集合。

例如：

```text
chat-release-gate
  - general-qa
  - safety-basic
  - tool-call-regression
  - json-format
  - long-context
```

Eval suite 用于标准化评估。

不同模型发布前跑同一套 suite，结果才可比较。

## 56.7 Eval Job

Eval Job 描述一次评估执行。

字段包括：

1. eval job ID。
2. target model version。
3. prompt version。
4. eval suite。
5. eval dataset versions。
6. inference config。
7. metric config。
8. judge model version。
9. runtime。
10. owner。
11. status。

Eval Job 和 TrainingJob 一样，也需要状态、日志、指标和 artifact。

## 56.8 Inference Runner

Inference Runner 负责批量调用模型。

它要支持：

1. 多模型 endpoint。
2. 并发控制。
3. 限流。
4. 重试。
5. 超时。
6. streaming 或非 streaming。
7. 结果缓存。
8. 失败样本记录。

评估时要固定 inference config。

temperature、max tokens、prompt 模板变化都会影响评估结果。

## 56.9 Metric Engine

Metric Engine 负责计算指标。

指标类型包括：

1. 规则指标。
2. 字符串匹配。
3. 结构化输出校验。
4. 单测执行。
5. 安全策略校验。
6. LLM-as-Judge 分数。
7. 人工标注结果。

Metric definition 必须版本化。

否则同一个结果在不同指标定义下可能分数不同。

## 56.10 LLM-as-Judge 服务

LLM-as-Judge 适合开放式生成评估。

Judge 服务要记录：

1. judge model version。
2. judge prompt version。
3. scoring rubric。
4. inference config。
5. calibration data。
6. bias checks。

还要防止：

1. 偏向长答案。
2. 偏向某种风格。
3. judge prompt 漂移。
4. judge model 升级后分数不可比。

重要结论最好结合人工评测或规则指标验证。

## 56.11 Human Review System

人工评测系统负责分发和收集人工判断。

能力包括：

1. 样本抽样。
2. 标注任务分发。
3. 标注指南版本。
4. 多人标注。
5. 仲裁。
6. gold set 检查。
7. 一致性统计。
8. 人工评测报告。

人工评测适合主观质量、安全边界和业务可接受性评估。

人工评测本身也要质量控制。

## 56.12 Sample-level Result Store

样本级结果非常重要。

每条样本应记录：

1. input。
2. reference。
3. prediction。
4. model version。
5. prompt version。
6. score。
7. error type。
8. latency。
9. token usage。
10. judge output。
11. human label。

样本级结果用于失败分析、回归集构建和模型对比。

只看平均分无法定位问题。

## 56.13 Slice Evaluation

平台要支持切片评估。

切片维度包括：

1. 语言。
2. 领域。
3. 难度。
4. 输入长度。
5. 输出格式。
6. 安全类别。
7. 工具类型。
8. 用户场景。

一个模型总体分数提升，但某个高价值业务场景退化，不能直接上线。

Slice evaluation 能发现隐藏退化。

## 56.14 Experiment Tracker 集成

每次 eval job 应关联 experiment run。

记录：

1. 参数。
2. 模型版本。
3. 数据版本。
4. prompt 版本。
5. 指标。
6. 样本级结果。
7. 报告 artifact。
8. 成本。

这样评估结果可以和训练实验、prompt 实验、RAG 实验关联。

## 56.15 Eval Report

Eval report 应包含：

1. 被评对象。
2. eval suite。
3. 数据版本。
4. 指标定义。
5. 总体指标。
6. 切片指标。
7. 与 baseline 对比。
8. 样本级失败案例。
9. 成本和延迟。
10. 结论和建议。

Eval report 应作为 artifact 进入模型仓库或发布系统。

## 56.16 Quality Gate

Quality Gate 是发布门禁。

示例规则：

1. 核心能力不低于 baseline。
2. 安全违规率低于阈值。
3. 工具调用成功率达标。
4. JSON parse success rate 达标。
5. p99 latency 不超过阈值。
6. cost per request 不超过预算。
7. 关键 slice 不退化。

Quality Gate 可以输出：

1. pass。
2. fail。
3. warning。
4. needs human review。

发布系统根据门禁结果决定是否允许灰度。

## 56.17 Online A/B 测试

在线 A/B 用真实流量比较版本。

平台要支持：

1. 稳定分桶。
2. 实验配置。
3. 指标采集。
4. guardrail 指标。
5. 实验停止。
6. 结果分析。
7. 回滚。

在线指标包括用户反馈、任务完成率、人工接管率、延迟、成本和安全指标。

在线 A/B 要和灰度发布区分：灰度偏风险控制，A/B 偏效果比较。

## 56.18 评估缓存

评估任务成本高，应支持缓存。

缓存 key 包括：

1. model version。
2. prompt version。
3. eval dataset version。
4. inference config。
5. runtime version。

如果这些都不变，可以复用 prediction outputs 或 metric results。

缓存能减少重复评估成本，但不能错误复用不同版本结果。

## 56.19 权限和安全

评估平台也需要安全治理。

要控制：

1. 谁能看 eval dataset。
2. 谁能运行评估。
3. 谁能查看 sample outputs。
4. 谁能查看人工标注结果。
5. 谁能修改 metric definition。
6. 谁能修改 release gate。

Eval 数据和 sample outputs 可能包含敏感信息，要做权限和脱敏。

## 56.20 成本治理

评估成本来自：

1. 批量推理。
2. LLM-as-Judge。
3. 人工评测。
4. 存储 sample outputs。
5. 运行单测或工具。

平台应记录：

1. cost per eval job。
2. cost per sample。
3. judge model cost。
4. human review cost。
5. 评估缓存节省成本。

评估不是免费操作，大规模评估需要预算控制。

## 56.21 可观测性

Eval 平台可观测性包括：

1. eval job status。
2. queue wait time。
3. samples processed per second。
4. inference error rate。
5. judge error rate。
6. metric computation error。
7. cost。
8. report generation latency。

如果评估失败，要知道是模型服务问题、数据问题、metric 问题还是 judge 服务问题。

## 56.22 核心数据模型

核心对象：

1. EvalDataset。
2. EvalSuite。
3. EvalJob。
4. MetricDefinition。
5. JudgeConfig。
6. HumanReviewTask。
7. SampleResult。
8. EvalReport。
9. QualityGate。
10. ABExperiment。

这些对象要有版本、权限、状态和血缘。

## 56.23 评估实验平台系统设计指标和最小 demo

系统设计题里，评估平台不能只讲“跑 benchmark 得分”。可以把一次评估实验抽象成：

```math
E_i=(a_i,m_i,d_i,p_i,c_i,k_i,j_i,h_i,o_i,r_i,s_i,g_i,z_i)
```

其中 `a_i` 是被评对象，可能是模型、prompt、RAG、Agent 或工具链；`m_i` 是模型版本，`d_i` 是 eval dataset 版本，`p_i` 是 prompt 版本，`c_i` 是推理配置，`k_i` 是指标定义，`j_i` 是 judge 配置，`h_i` 是人评配置，`o_i` 是在线实验配置，`r_i` 是样本级结果，`s_i` 是切片结果，`g_i` 是 release gate，`z_i` 是最终决策。

统一覆盖率可以写成：

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}\left[g_j(E_i)=1\right]
```

其中 `g_j` 是第 `j` 个审计条件，例如数据集契约、离线复现、指标版本、judge 校准、人评质量、在线 guardrail、报告完整性或 artifact 血缘。

数据集切片覆盖率可以写成：

```math
C_{\mathrm{slice}}=\frac{1}{S}\sum_{s=1}^{S}\mathbf{1}\left[n_s\ge n_{\min}\right]
```

训练 / 评估污染率可以写成：

```math
R_{\mathrm{leak}}=\frac{N_{\mathrm{overlap}}}{N_{\mathrm{eval}}}
```

多指标加权汇总可以写成：

```math
S_{\mathrm{agg}}=\frac{\sum_{k=1}^{K}w_kM_k}{\sum_{k=1}^{K}w_k}
```

其中 `M_k` 是已经同向归一化后的指标分数。注意它只适合做候选排序，不能替代切片退化和安全门禁。

Pairwise win rate 可以写成：

```math
W=\frac{N_{\mathrm{win}}+0.5N_{\mathrm{tie}}}{N_{\mathrm{win}}+N_{\mathrm{loss}}+N_{\mathrm{tie}}}
```

Judge 和人工标注一致率可以写成：

```math
A_{\mathrm{jh}}=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}\left[y_i^{\mathrm{judge}}=y_i^{\mathrm{human}}\right]
```

在线实验 guardrail 可以写成：

```math
G_{\mathrm{online}}=\mathbf{1}\left[\Delta_Q\ge -\epsilon_Q \land \Delta_L\le \epsilon_L \land \Delta_K\le \epsilon_K \land P_0=0\right]
```

其中 `Delta_Q` 是质量变化，`Delta_L` 是延迟变化，`Delta_K` 是成本变化，`P_0` 是 P0 级事故数量。

缓存命中率可以写成：

```math
R_{\mathrm{cache}}=\frac{N_{\mathrm{hit}}}{N_{\mathrm{request}}}
```

但缓存 key 必须包含模型版本、数据版本、prompt 版本、推理配置、评估代码版本和 runtime 版本，否则命中率越高，错误复用风险越大。

Artifact 血缘覆盖率可以写成：

```math
R_{\mathrm{lineage}}=\frac{|E_{\mathrm{observed}}|}{|E_{\mathrm{required}}|}
```

最终评估平台门禁可以写成：

```math
G_{\mathrm{eval}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land C_{\mathrm{slice}}\ge \alpha \land R_{\mathrm{leak}}\le \rho \land A_{\mathrm{jh}}\ge \beta \land G_{\mathrm{online}}=1 \land P_0=0\right]
```

下面的 0 依赖 demo 不实现完整评估系统，而是实现一个系统设计审计器：它给出 toy eval dataset、metric、pairwise、人评、judge、online guardrail、cache、lineage 和 design cases，输出可用于面试解释的质量决策指标。

```python
class MiniEvalExperimentPlatformSystemDesignAudit:
    def __init__(self):
        self.slice_counts = {
            "language:zh": 72,
            "language:en": 86,
            "domain:code": 64,
            "domain:math": 58,
            "difficulty:hard": 41,
        }
        self.sample_count = 500
        self.overlap_count = 1
        self.metrics = [
            {"name": "accuracy", "weight": 0.35, "score": 0.82},
            {"name": "safety_pass", "weight": 0.25, "score": 0.98},
            {"name": "tool_success", "weight": 0.20, "score": 0.76},
            {"name": "format_success", "weight": 0.10, "score": 0.93},
            {"name": "cost_budget", "weight": 0.10, "score": 0.80},
        ]
        self.pairwise = {"wins": 58, "losses": 36, "ties": 6}
        self.judge_labels = ["good", "bad", "good", "good", "bad"]
        self.human_labels = ["good", "bad", "bad", "good", "bad"]
        self.online = {
            "quality_delta": 0.025,
            "latency_delta": 0.03,
            "cost_delta": 0.04,
            "safety_events": 0,
        }
        self.cache = {"requests": 1200, "hits": 420}
        self.lineage = {"observed_edges": 13, "required_edges": 14}
        self.costs = {"inference": 18.2, "judge": 6.4, "human_review": 40.0}
        self.gates = [
            "eval_dataset_contract",
            "offline_batch_reproducibility",
            "metric_definition_versioning",
            "llm_judge_calibration",
            "human_review_quality_control",
            "pairwise_blinding_randomization",
            "online_eval_guardrail",
            "release_gate_readiness",
            "eval_report_completeness",
            "slice_regression_detection",
            "regression_suite_coverage",
            "eval_job_scheduler_reliability",
            "eval_cache_key_integrity",
            "eval_observability",
            "artifact_tracking_linkage",
            "evaluation_platform_gate",
        ]

    def dataset_slice_coverage(self, min_samples=40):
        covered = sum(1 for count in self.slice_counts.values() if count >= min_samples)
        return round(covered / len(self.slice_counts), 3)

    def contamination_rate(self):
        return round(self.overlap_count / self.sample_count, 4)

    def weighted_score(self):
        weight_sum = sum(metric["weight"] for metric in self.metrics)
        score = sum(metric["weight"] * metric["score"] for metric in self.metrics) / weight_sum
        return round(score, 3)

    def pairwise_win_rate(self):
        wins = self.pairwise["wins"]
        losses = self.pairwise["losses"]
        ties = self.pairwise["ties"]
        return round((wins + 0.5 * ties) / (wins + losses + ties), 3)

    def judge_human_agreement(self):
        matched = sum(
            1 for judge, human in zip(self.judge_labels, self.human_labels)
            if judge == human
        )
        return round(matched / len(self.human_labels), 3)

    def online_guardrail_pass(self):
        return (
            self.online["quality_delta"] >= -0.01
            and self.online["latency_delta"] <= 0.05
            and self.online["cost_delta"] <= 0.08
            and self.online["safety_events"] == 0
        )

    def cache_hit_rate(self):
        return round(self.cache["hits"] / self.cache["requests"], 3)

    def lineage_coverage(self):
        return round(self.lineage["observed_edges"] / self.lineage["required_edges"], 3)

    def estimated_eval_cost(self):
        return round(sum(self.costs.values()), 2)

    def eval_examples(self):
        return {
            "dataset_slice_coverage": self.dataset_slice_coverage(),
            "contamination_rate": self.contamination_rate(),
            "weighted_score": self.weighted_score(),
            "pairwise_win_rate": self.pairwise_win_rate(),
            "judge_human_agreement": self.judge_human_agreement(),
            "online_guardrail_pass": self.online_guardrail_pass(),
            "cache_hit_rate": self.cache_hit_rate(),
            "lineage_coverage": self.lineage_coverage(),
            "estimated_eval_cost": self.estimated_eval_cost(),
        }

    def design_cases(self):
        complete = {gate: True for gate in self.gates}
        bad_specs = [
            ("dataset_version_missing_bad", "eval_dataset_contract"),
            ("offline_config_unfrozen_bad", "offline_batch_reproducibility"),
            ("metric_version_missing_bad", "metric_definition_versioning"),
            ("judge_uncalibrated_bad", "llm_judge_calibration"),
            ("human_review_no_gold_bad", "human_review_quality_control"),
            ("pairwise_not_blinded_bad", "pairwise_blinding_randomization"),
            ("online_guardrail_missing_bad", "online_eval_guardrail"),
            ("release_gate_no_rollback_bad", "release_gate_readiness"),
            ("report_no_samples_bad", "eval_report_completeness"),
            ("slice_regression_hidden_bad", "slice_regression_detection"),
            ("regression_suite_thin_bad", "regression_suite_coverage"),
            ("eval_scheduler_unreliable_bad", "eval_job_scheduler_reliability"),
            ("cache_key_missing_prompt_bad", "eval_cache_key_integrity"),
            ("eval_observability_missing_bad", "eval_observability"),
            ("artifact_lineage_broken_bad", "artifact_tracking_linkage"),
            ("evaluation_gate_missing_bad", "evaluation_platform_gate"),
        ]
        cases = [{"name": "complete_eval_platform_design", "checks": complete}]
        for name, failed_gate in bad_specs:
            checks = {gate: True for gate in self.gates}
            checks[failed_gate] = False
            cases.append({"name": name, "checks": checks})
        return cases

    def evaluate_cases(self):
        cases = self.design_cases()
        metrics = {}
        failed_cases = []
        failed_gates = set()
        for gate in self.gates:
            passed = sum(1 for case in cases if case["checks"][gate])
            metrics[gate] = round(passed / len(cases), 3)
        for case in cases:
            missing = [gate for gate, ok in case["checks"].items() if not ok]
            if missing:
                failed_cases.append(case["name"])
                failed_gates.update(missing)
        return {
            "metrics": metrics,
            "failed_cases": failed_cases,
            "failed_gates": sorted(failed_gates),
            "hard_blocker_count": len(failed_cases),
            "gate_pass": len(failed_cases) == 0,
        }

    def run(self):
```

## 56.24 关键 trade-off

评估平台 trade-off：

1. 评估全面性 vs 成本。
2. 离线评估速度 vs 线上真实性。
3. LLM-as-Judge 低成本 vs 人工评测可信度。
4. 样本级结果完整保存 vs 隐私和存储成本。
5. 门禁严格 vs 发布效率。
6. 公开 benchmark vs 内部业务集。

成熟平台通常组合多种评估，而不是迷信单一分数。

## 56.26 常见扣分点

扣分点一：只说跑 benchmark。

评估平台还要管理数据、指标、样本、人工评测、报告、门禁和发布联动。

扣分点二：不提 eval 数据版本。

没有数据版本，评估不可复现。

扣分点三：只看总体分数。

总体分数会掩盖 slice 退化。

扣分点四：盲信 LLM-as-Judge。

Judge model 有偏差，需要版本化、校准和人工验证。

扣分点五：不提发布门禁。

评估结果应该进入发布决策，而不只是生成报告。

## 56.28 本章小结

本章系统设计了一个大模型评估与实验平台。

你需要记住：

1. 评估平台是质量决策系统，不是 benchmark 脚本。
2. Eval dataset、eval suite、metric、judge、prompt、模型和推理配置都要版本化。
3. 平台要支持离线批量评测、LLM-as-Judge、人工评测、在线 A/B 和用户反馈。
4. 样本级结果、slice evaluation 和 failure cases 是模型诊断关键。
5. Quality Gate 把评估结果接入发布流程，是企业级模型治理的重要环节。
6. 第二轮新增的公式和 demo 可以把切片覆盖、污染率、加权指标、pairwise、人评 / judge 一致率、在线 guardrail、缓存、血缘和发布门禁串成可验收的系统设计证据。
