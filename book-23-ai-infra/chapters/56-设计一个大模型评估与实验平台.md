# 第 56 章 设计一个大模型评估与实验平台

上一章设计了多租户 GPU 集群调度系统。本章继续系统设计题：设计一个大模型评估与实验平台。

评估与实验平台是大模型研发和发布的质量控制中心。训练得到一个模型、改了一个 prompt、更新了一个 RAG 配置、上线了一个 Agent，都需要评估和实验系统回答：它是否真的更好？是否更安全？是否更便宜？是否可以发布？

先记住一句话：

> 大模型评估与实验平台的核心，是把模型、prompt、数据、指标、样本、人工判断、线上反馈和发布门禁连接成可复现的质量决策系统。

## 56.1 题目理解

面试题可能这样问：

1. 请设计一个大模型评估平台。
2. 请设计一个支持离线评测、人工评测和在线 A/B 的系统。
3. 请设计一个模型实验追踪与评估平台。
4. 如何设计模型发布前的质量门禁系统？

这些题的核心是：如何系统化评估模型和大模型应用，而不是手写 benchmark 脚本。

## 56.2 需求澄清

先问：

1. 评估对象是什么？模型、prompt、RAG、Agent 还是工具？
2. 支持哪些任务？问答、代码、工具调用、安全、长上下文？
3. 是否需要离线批量评测？
4. 是否需要人工评测？
5. 是否需要 LLM-as-Judge？
6. 是否需要在线 A/B 测试？
7. 是否需要和发布系统联动？
8. 是否需要样本级结果和失败分析？
9. 是否多租户？是否有权限和审计？
10. 评估成本和时延要求是什么？

评估平台的设计取决于评估对象和决策场景。

## 56.3 核心目标

平台目标：

1. 管理 eval dataset。
2. 管理 eval suite。
3. 支持多模型、多版本评估。
4. 支持 prompt/RAG/Agent 评估。
5. 支持离线批量评测。
6. 支持人工评测。
7. 支持 LLM-as-Judge。
8. 支持在线 A/B 测试。
9. 支持样本级结果和失败分析。
10. 支持评估报告、发布门禁和实验追踪。

一句话：评估平台要把“跑分”变成“质量决策”。

## 56.4 总体架构

可以设计如下：

```text
User / CI / Release System
  -> Eval API / Web Console
  -> Eval Dataset Registry
  -> Eval Suite Registry
  -> Eval Job Scheduler
  -> Inference Runner
  -> Metric Engine
  -> LLM Judge Service
  -> Human Review System
  -> Result Store
  -> Experiment Tracker
  -> Report Generator
  -> Quality Gate
  -> Model Registry / Release System
```

关键链路：选择模型和评估集，运行推理，计算指标，生成报告，决定是否进入发布。

## 56.5 Eval Dataset Registry

Eval dataset registry 管理评估数据集。

每个 eval dataset 记录：

1. 名称。
2. 版本。
3. 任务类型。
4. 样本数量。
5. 数据来源。
6. 标签或参考答案。
7. 难度分布。
8. 语言和领域分布。
9. 是否可用于训练。
10. 权限和敏感等级。

Eval 数据必须版本化，并与训练数据隔离。

## 56.6 Eval Suite

Eval suite 是一组评估任务集合。

例如：

```text
chat-release-gate
  - general-qa
  - safety-basic
  - tool-call-regression
  - json-format
  - long-context
```

Eval suite 用于标准化评估。

不同模型发布前跑同一套 suite，结果才可比较。

## 56.7 Eval Job

Eval Job 描述一次评估执行。

字段包括：

1. eval job ID。
2. target model version。
3. prompt version。
4. eval suite。
5. eval dataset versions。
6. inference config。
7. metric config。
8. judge model version。
9. runtime。
10. owner。
11. status。

Eval Job 和 TrainingJob 一样，也需要状态、日志、指标和 artifact。

## 56.8 Inference Runner

Inference Runner 负责批量调用模型。

它要支持：

1. 多模型 endpoint。
2. 并发控制。
3. 限流。
4. 重试。
5. 超时。
6. streaming 或非 streaming。
7. 结果缓存。
8. 失败样本记录。

评估时要固定 inference config。

temperature、max tokens、prompt 模板变化都会影响评估结果。

## 56.9 Metric Engine

Metric Engine 负责计算指标。

指标类型包括：

1. 规则指标。
2. 字符串匹配。
3. 结构化输出校验。
4. 单测执行。
5. 安全策略校验。
6. LLM-as-Judge 分数。
7. 人工标注结果。

Metric definition 必须版本化。

否则同一个结果在不同指标定义下可能分数不同。

## 56.10 LLM-as-Judge 服务

LLM-as-Judge 适合开放式生成评估。

Judge 服务要记录：

1. judge model version。
2. judge prompt version。
3. scoring rubric。
4. inference config。
5. calibration data。
6. bias checks。

还要防止：

1. 偏向长答案。
2. 偏向某种风格。
3. judge prompt 漂移。
4. judge model 升级后分数不可比。

重要结论最好结合人工评测或规则指标验证。

## 56.11 Human Review System

人工评测系统负责分发和收集人工判断。

能力包括：

1. 样本抽样。
2. 标注任务分发。
3. 标注指南版本。
4. 多人标注。
5. 仲裁。
6. gold set 检查。
7. 一致性统计。
8. 人工评测报告。

人工评测适合主观质量、安全边界和业务可接受性评估。

人工评测本身也要质量控制。

## 56.12 Sample-level Result Store

样本级结果非常重要。

每条样本应记录：

1. input。
2. reference。
3. prediction。
4. model version。
5. prompt version。
6. score。
7. error type。
8. latency。
9. token usage。
10. judge output。
11. human label。

样本级结果用于失败分析、回归集构建和模型对比。

只看平均分无法定位问题。

## 56.13 Slice Evaluation

平台要支持切片评估。

切片维度包括：

1. 语言。
2. 领域。
3. 难度。
4. 输入长度。
5. 输出格式。
6. 安全类别。
7. 工具类型。
8. 用户场景。

一个模型总体分数提升，但某个高价值业务场景退化，不能直接上线。

Slice evaluation 能发现隐藏退化。

## 56.14 Experiment Tracker 集成

每次 eval job 应关联 experiment run。

记录：

1. 参数。
2. 模型版本。
3. 数据版本。
4. prompt 版本。
5. 指标。
6. 样本级结果。
7. 报告 artifact。
8. 成本。

这样评估结果可以和训练实验、prompt 实验、RAG 实验关联。

## 56.15 Eval Report

Eval report 应包含：

1. 被评对象。
2. eval suite。
3. 数据版本。
4. 指标定义。
5. 总体指标。
6. 切片指标。
7. 与 baseline 对比。
8. 样本级失败案例。
9. 成本和延迟。
10. 结论和建议。

Eval report 应作为 artifact 进入模型仓库或发布系统。

## 56.16 Quality Gate

Quality Gate 是发布门禁。

示例规则：

1. 核心能力不低于 baseline。
2. 安全违规率低于阈值。
3. 工具调用成功率达标。
4. JSON parse success rate 达标。
5. p99 latency 不超过阈值。
6. cost per request 不超过预算。
7. 关键 slice 不退化。

Quality Gate 可以输出：

1. pass。
2. fail。
3. warning。
4. needs human review。

发布系统根据门禁结果决定是否允许灰度。

## 56.17 Online A/B 测试

在线 A/B 用真实流量比较版本。

平台要支持：

1. 稳定分桶。
2. 实验配置。
3. 指标采集。
4. guardrail 指标。
5. 实验停止。
6. 结果分析。
7. 回滚。

在线指标包括用户反馈、任务完成率、人工接管率、延迟、成本和安全指标。

在线 A/B 要和灰度发布区分：灰度偏风险控制，A/B 偏效果比较。

## 56.18 评估缓存

评估任务成本高，应支持缓存。

缓存 key 包括：

1. model version。
2. prompt version。
3. eval dataset version。
4. inference config。
5. runtime version。

如果这些都不变，可以复用 prediction outputs 或 metric results。

缓存能减少重复评估成本，但不能错误复用不同版本结果。

## 56.19 权限和安全

评估平台也需要安全治理。

要控制：

1. 谁能看 eval dataset。
2. 谁能运行评估。
3. 谁能查看 sample outputs。
4. 谁能查看人工标注结果。
5. 谁能修改 metric definition。
6. 谁能修改 release gate。

Eval 数据和 sample outputs 可能包含敏感信息，要做权限和脱敏。

## 56.20 成本治理

评估成本来自：

1. 批量推理。
2. LLM-as-Judge。
3. 人工评测。
4. 存储 sample outputs。
5. 运行单测或工具。

平台应记录：

1. cost per eval job。
2. cost per sample。
3. judge model cost。
4. human review cost。
5. 评估缓存节省成本。

评估不是免费操作，大规模评估需要预算控制。

## 56.21 可观测性

Eval 平台可观测性包括：

1. eval job status。
2. queue wait time。
3. samples processed per second。
4. inference error rate。
5. judge error rate。
6. metric computation error。
7. cost。
8. report generation latency。

如果评估失败，要知道是模型服务问题、数据问题、metric 问题还是 judge 服务问题。

## 56.22 核心数据模型

核心对象：

1. EvalDataset。
2. EvalSuite。
3. EvalJob。
4. MetricDefinition。
5. JudgeConfig。
6. HumanReviewTask。
7. SampleResult。
8. EvalReport。
9. QualityGate。
10. ABExperiment。

这些对象要有版本、权限、状态和血缘。

## 56.23 关键 trade-off

评估平台 trade-off：

1. 评估全面性 vs 成本。
2. 离线评估速度 vs 线上真实性。
3. LLM-as-Judge 低成本 vs 人工评测可信度。
4. 样本级结果完整保存 vs 隐私和存储成本。
5. 门禁严格 vs 发布效率。
6. 公开 benchmark vs 内部业务集。

成熟平台通常组合多种评估，而不是迷信单一分数。

## 56.24 面试回答模板

可以这样回答：

```text
我会把大模型评估与实验平台设计成质量决策系统。核心模块包括 Eval Dataset Registry、Eval Suite Registry、Eval Job Scheduler、Inference Runner、Metric Engine、LLM Judge、Human Review、Result Store、Experiment Tracker、Report Generator 和 Quality Gate。

用户选择模型版本、prompt 版本或 RAG/Agent 配置，并选择 eval suite 发起 EvalJob。Scheduler 执行批量推理，Inference Runner 固定推理参数并记录输出。Metric Engine 计算规则指标、结构化指标、安全指标和任务指标，开放式任务可以调用 LLM-as-Judge，关键场景进入人工评测。

平台保存 sample-level results，支持按语言、领域、难度、输入长度、安全类别和工具类型做 slice evaluation。Eval report 作为 artifact 关联到模型仓库和实验追踪系统。Quality Gate 根据核心指标、guardrail、成本和延迟决定是否允许进入灰度或发布。

在线阶段支持 A/B 测试和用户反馈闭环。平台同时提供权限、安全、成本、缓存和可观测性，保证评估可复现、可比较、可审计。
```

## 56.25 常见扣分点

扣分点一：只说跑 benchmark。

评估平台还要管理数据、指标、样本、人工评测、报告、门禁和发布联动。

扣分点二：不提 eval 数据版本。

没有数据版本，评估不可复现。

扣分点三：只看总体分数。

总体分数会掩盖 slice 退化。

扣分点四：盲信 LLM-as-Judge。

Judge model 有偏差，需要版本化、校准和人工验证。

扣分点五：不提发布门禁。

评估结果应该进入发布决策，而不只是生成报告。

6. Quality Gate 应该包含哪些规则？
7. 在线 A/B 和灰度发布有什么区别？
8. 如何设计评估缓存 key？

## 56.27 本章小结

本章系统设计了一个大模型评估与实验平台。

你需要记住：

1. 评估平台是质量决策系统，不是 benchmark 脚本。
2. Eval dataset、eval suite、metric、judge、prompt、模型和推理配置都要版本化。
3. 平台要支持离线批量评测、LLM-as-Judge、人工评测、在线 A/B 和用户反馈。
4. 样本级结果、slice evaluation 和 failure cases 是模型诊断关键。
5. Quality Gate 把评估结果接入发布流程，是企业级模型治理的重要环节。
