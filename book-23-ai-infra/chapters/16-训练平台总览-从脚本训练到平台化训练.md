# 第 16 章 训练平台总览：从脚本训练到平台化训练

前面两部分讲了 AI Infra 总览、硬件、集群、网络、存储、调度和容量规划。从本章开始进入第三部分：大模型训练平台工程。

很多团队最开始训练模型，是在一台机器上手工运行脚本：登录机器、拉代码、装依赖、改配置、执行命令、看日志、手动保存 checkpoint。这种方式适合探索，但无法支撑多人、多任务、大规模、可复现的训练体系。

训练平台的目标，就是把“手工跑脚本”变成“可提交、可调度、可复现、可监控、可恢复、可审计”的平台能力。

先记住一句话：

> 大模型训练平台不是一个脚本启动器，而是训练任务生命周期管理系统。

## 16.0 本讲资料边界与第二轮精修口径

本讲按第二轮精修口径，只抽象训练平台的稳定工程问题，不绑定某个云厂商、某个公司内部平台或某一个 Kubernetes 发行版。

资料校准时可以把几类官方口径放在一起看：

1. Kubernetes Job 关注的是把一组 Pod 运行到完成，适合承载批处理任务，但它本身不理解大模型训练的代码版本、数据血缘、checkpoint、实验追踪和模型产物生命周期。
2. Kueue 这类批处理队列层关注 queue、quota、admission、resource flavor、workload 和抢占等资源治理问题，可以作为训练平台调度层的一部分。
3. Kubeflow Trainer / TrainJob 这类训练 operator 把训练任务抽象成面向机器学习的 API，对多 worker 启动、状态、runtime、训练函数和任务生命周期做了更高层封装。
4. PyTorch `torchrun` 关注分布式训练进程启动、rank、world size、rendezvous 和 elastic restart 等运行细节，训练平台通常要把这些细节封装在 launcher 里。
5. MLflow Tracking 这类实验追踪系统强调 parameter、metric、artifact、run 和 experiment 的记录，能支撑复现、比较和审计，但它不是完整的资源调度或训练作业平台。

所以本章的结论是：训练平台要把 Job / Queue / Launcher / Checkpoint / Tracking / Registry / Audit 串成一条训练生命周期，而不是把其中某个工具单独说成“平台”。

## 16.1 为什么需要训练平台

手工训练会遇到很多问题。

环境问题：

1. 每台机器依赖不同。
2. CUDA 和驱动版本不一致。
3. 代码修改没记录。
4. 训练命令靠聊天记录传播。

资源问题：

1. 谁占了 GPU 不清楚。
2. 任务排队靠人工协调。
3. 大任务和小任务互相影响。
4. 资源利用率低。

实验问题：

1. 数据版本不清楚。
2. 超参没有记录。
3. checkpoint 不知道对应哪个实验。
4. 结果无法复现。

运维问题：

1. 训练失败没人发现。
2. 节点故障后无法自动恢复。
3. 日志分散在各机器。
4. checkpoint 保存失败后才发现。

治理问题：

1. 权限不清楚。
2. 成本无法归因。
3. 数据访问无审计。
4. 模型产物混乱。

训练平台就是为了解决这些问题。

## 16.2 从脚本训练到平台化训练

脚本训练通常是：

```text
ssh 到机器
  -> git pull
  -> conda activate
  -> 修改配置
  -> bash train.sh
  -> tail log
  -> 手动保存 checkpoint
```

平台化训练应该是：

```text
提交 TrainingJob
  -> 校验配置
  -> 检查权限和配额
  -> 选择镜像、代码、数据和资源
  -> 调度 GPU
  -> 启动分布式训练
  -> 采集日志、指标和事件
  -> 保存 checkpoint
  -> 支持失败恢复
  -> 注册模型和 artifact
  -> 记录实验和成本
```

区别在于：平台化训练把训练过程中的关键状态和决策显式化、结构化、可追踪化。

## 16.3 训练平台的核心目标

训练平台要实现十个目标：

1. 任务提交标准化。
2. 资源申请结构化。
3. 环境可复现。
4. 数据和代码版本可追踪。
5. 分布式训练可启动。
6. 日志和指标可观测。
7. checkpoint 可管理。
8. 失败可恢复。
9. 权限和成本可治理。
10. 模型产物可注册和发布。

这些目标决定了训练平台的模块设计。

## 16.4 训练平台总体架构

一个典型训练平台架构：

```text
User / CLI / SDK / Portal
  -> Training Job API
  -> Config Validator
  -> Policy / Quota Engine
  -> Scheduler / Queue
  -> Launcher / Operator
  -> Kubernetes / GPU Cluster
  -> Data Access Layer
  -> Checkpoint Manager
  -> Metrics / Logs / Events
  -> Experiment Tracker
  -> Model Registry / Artifact Store
  -> Cost / Audit
```

模块解释：

1. User / CLI / SDK / Portal：用户入口。
2. Training Job API：统一提交训练任务。
3. Config Validator：校验配置合法性。
4. Policy / Quota Engine：检查权限、配额和安全策略。
5. Scheduler / Queue：排队、优先级、抢占和资源分配。
6. Launcher / Operator：启动 torchrun、DeepSpeed、Megatron、Ray 等。
7. Kubernetes / GPU Cluster：实际运行容器和任务。
8. Data Access Layer：挂载或访问训练数据。
9. Checkpoint Manager：保存、恢复和清理 checkpoint。
10. Metrics / Logs / Events：观测训练状态。
11. Experiment Tracker：记录参数、指标、代码、数据和环境。
12. Model Registry / Artifact Store：管理产物。
13. Cost / Audit：成本归因和审计。

## 16.5 TrainingJob 是什么

TrainingJob 是平台化训练的核心对象。

它描述一次训练任务。

一个 TrainingJob 至少应该包含：

1. job_id。
2. owner。
3. project。
4. image。
5. code_version。
6. entrypoint。
7. config。
8. dataset_version。
9. resource_request。
10. distributed_config。
11. checkpoint_config。
12. logging_config。
13. priority。
14. queue。
15. tags。

示例：

```yaml
job_name: sft-chat-7b-v2
project: customer-support-llm
image: registry/train:pytorch2.2-cuda12.1@sha256:abc
code:
  repo: git@example.com:llm/train.git
  commit: 8f31c2a
entrypoint: python train.py
config: configs/sft_7b.yaml
data:
  dataset_id: customer_support_sft
  version: 2026-05-30
resources:
  gpu_type: A100-80G
  gpu_count: 8
  gpu_per_node: 8
distributed:
  launcher: torchrun
  nnodes: 1
  nproc_per_node: 8
checkpoint:
  output_uri: s3://ckpt/customer-support/sft-chat-7b-v2/
  save_every_steps: 1000
queue: sft
priority: normal
```

TrainingJob 的价值在于把训练从命令行变成可管理对象。

## 16.6 任务提交入口

训练平台通常提供多种入口：

1. Web Portal。
2. CLI。
3. SDK。
4. API。
5. CI/CD 集成。
6. Notebook 集成。

不同用户偏好不同。

算法工程师可能喜欢 CLI 和 SDK。

平台管理员需要 Portal 查看任务和资源。

自动化流水线需要 API。

好的平台不应该强迫所有人写 Kubernetes YAML，而应该提供面向训练语义的接口。

## 16.7 配置校验

任务提交后，平台要先校验配置。

校验内容：

1. 镜像是否存在。
2. 镜像是否通过安全扫描。
3. 代码 commit 是否存在。
4. 数据集版本是否存在。
5. 用户是否有数据权限。
6. GPU 型号是否可用。
7. 资源申请是否超过配额。
8. checkpoint 路径是否可写。
9. 分布式参数是否一致。
10. 启动命令是否符合规范。

配置校验能把很多错误提前挡在排队之前。

否则任务排队几小时后才发现路径错了，非常浪费。

## 16.8 资源调度

训练平台通常不会直接自己管理 GPU，而是和调度层配合。

调度要考虑：

1. 队列。
2. 优先级。
3. 配额。
4. GPU 型号。
5. GPU 数量。
6. 每节点 GPU 数。
7. 网络和拓扑。
8. 是否可抢占。
9. checkpoint 支持情况。

对于大规模分布式训练，调度还要支持 gang scheduling。

训练平台需要把 TrainingJob 转换成调度系统能理解的资源请求。

## 16.9 分布式训练启动

启动分布式训练比单机脚本复杂。

需要处理：

1. master 地址。
2. rank 分配。
3. world size。
4. worker 启动顺序。
5. 环境变量。
6. 通信端口。
7. 失败重试。
8. 日志聚合。

常见启动方式：

1. torchrun。
2. DeepSpeed launcher。
3. Megatron 启动脚本。
4. MPI。
5. Ray。
6. 自研 launcher。

训练平台的价值是屏蔽这些细节，让用户专注训练配置和代码。

## 16.10 数据接入

训练平台要让任务可靠访问数据。

数据接入方式包括：

1. 挂载共享文件系统。
2. 从对象存储流式读取。
3. 本地缓存。
4. 分布式缓存。
5. 数据集服务。
6. 数据湖表。

平台要管理：

1. 数据版本。
2. 数据权限。
3. 数据路径。
4. 缓存策略。
5. 数据读取指标。
6. 数据血缘。

训练数据不是随便给一个路径。平台要知道这次训练用了什么数据、什么版本、谁授权。

## 16.11 Checkpoint 管理

训练平台必须内置 checkpoint 管理。

能力包括：

1. 自动保存。
2. 保存频率配置。
3. 分片 checkpoint。
4. 异步 checkpoint。
5. 恢复训练。
6. checkpoint manifest。
7. 生命周期管理。
8. 最优 checkpoint 标记。
9. checkpoint 成本统计。

平台还要在任务失败时自动判断：是否存在可恢复 checkpoint？从哪个 checkpoint 恢复？恢复配置是否兼容？

## 16.12 日志、指标和事件

训练平台要提供统一可观测性。

日志：

1. stdout / stderr。
2. rank 日志。
3. 错误堆栈。
4. 训练脚本日志。

指标：

1. loss。
2. learning rate。
3. tokens/s。
4. step time。
5. GPU utilization。
6. 显存占用。
7. MFU。
8. 通信时间。
9. dataloader time。
10. checkpoint time。

事件：

1. job submitted。
2. job scheduled。
3. container started。
4. checkpoint saved。
5. worker failed。
6. retry started。
7. job completed。

事件很重要，因为它记录任务生命周期。

## 16.13 实验追踪

训练平台要和实验追踪系统集成。

实验追踪记录：

1. 训练配置。
2. 超参。
3. 数据版本。
4. 代码 commit。
5. 镜像 digest。
6. GPU 类型。
7. 随机种子。
8. 指标曲线。
9. checkpoint。
10. eval report。
11. 最终模型。

没有实验追踪，团队就无法回答：这个模型是怎么训出来的？为什么比上一个好？能不能复现？

## 16.14 模型和 Artifact 注册

训练结束后，平台应该把产物注册到模型仓库或 artifact store。

产物包括：

1. final checkpoint。
2. best checkpoint。
3. tokenizer。
4. config。
5. LoRA / adapter。
6. eval report。
7. training log。
8. dataset lineage。
9. training config。

注册后模型才进入后续流程：评估、审核、发布、部署、回滚。

模型不能只是一个路径，而应该是带 metadata 的资产。

## 16.15 失败恢复

训练平台要处理失败。

常见失败：

1. OOM。
2. 节点故障。
3. 网络 timeout。
4. NCCL hang。
5. 数据读取失败。
6. checkpoint 保存失败。
7. 用户代码异常。
8. 镜像拉取失败。

平台可以做：

1. 自动重试。
2. 从 checkpoint 恢复。
3. 标记不可恢复错误。
4. 收集错误原因。
5. 通知用户。
6. 生成诊断报告。

不是所有失败都应该自动重试。

例如配置错误和代码 bug 重试也没用。节点故障和临时网络问题适合重试。

## 16.16 权限、安全和审计

训练平台要控制：

1. 谁能提交任务。
2. 谁能使用哪些 GPU。
3. 谁能访问哪些数据。
4. 谁能读取日志。
5. 谁能恢复或取消任务。
6. 谁能注册模型。
7. 谁能发布模型。

审计记录：

1. 用户。
2. 任务。
3. 数据集。
4. 镜像。
5. 代码版本。
6. checkpoint。
7. 模型产物。
8. 资源消耗。

训练平台是数据、算力和模型产物的交汇点，安全治理必须内置。

## 16.17 成本归因

训练平台要记录成本。

包括：

1. GPU-hours。
2. CPU / memory。
3. 存储。
4. checkpoint。
5. 网络流量。
6. 评估推理成本。
7. 日志和 trace。

归因维度：

1. team。
2. project。
3. user。
4. job。
5. model。
6. queue。

成本归因让团队知道哪些实验最贵，哪些任务低效，哪些资源池需要优化。

## 16.18 训练平台成熟度

可以把训练平台成熟度分成四级。

一级：脚本化。

1. 用户手工登录机器。
2. 手工运行脚本。
3. 日志和 checkpoint 分散。

二级：任务化。

1. 支持提交 Job。
2. 支持镜像和资源申请。
3. 基本日志和状态可见。

三级：平台化。

1. 支持队列、配额、checkpoint、实验追踪。
2. 支持分布式启动和自动恢复。
3. 支持 artifact 注册。

四级：治理化。

1. 支持权限、安全、审计、成本、质量门禁。
2. 支持容量规划和低效任务诊断。
3. 支持训练到评估再到发布的完整闭环。

目标不是一步到位，而是按团队规模逐步演进。

## 16.19 训练平台审计指标与最小 demo

训练平台是否可靠，不能只看“能提交任务”。更合理的方式是把一次训练任务看成审计样本：

```math
x_i=(j_i,c_i,r_i,e_i,d_i,l_i,k_i,o_i,m_i,a_i,s_i,u_i)
```

其中，`j_i` 是 TrainingJob 契约，`c_i` 是配置校验，`r_i` 是资源和配额，`e_i` 是镜像、代码和环境，`d_i` 是数据版本和权限，`l_i` 是 launcher 与分布式配置，`k_i` 是 checkpoint 策略，`o_i` 是观测信号，`m_i` 是实验追踪和模型产物，`a_i` 是权限审计，`s_i` 是失败恢复和状态机，`u_i` 是用户自助体验。

对每个检查维度 `g_j`，可以定义覆盖率：

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(x_i)=1]
```

训练平台的可复现字段覆盖率可以写成：

```math
R_{\mathrm{repr}}=\frac{K_{\mathrm{captured}}}{K_{\mathrm{required}}}
```

其中，`K_required` 至少包括 image digest、code commit、config version、dataset version、dependency lock、random seed、launcher version 和 environment metadata。

checkpoint 恢复的最大丢失进度可以写成：

```math
S_{\mathrm{lost}}\le S_{\mathrm{max}}
```

其中，`S_lost` 是故障后最多需要重跑的 step 数，`S_max` 是平台允许的最大丢失进度。这个约束比“有 checkpoint”更接近生产现实。

模型产物血缘覆盖率可以写成：

```math
A_{\mathrm{lineage}}=\frac{K_{\mathrm{linked}}}{K_{\mathrm{artifact}}}
```

其中，`K_linked` 是已经和训练任务、数据、代码、配置、eval report、owner、权限状态建立链接的 artifact 数量。

成本归因覆盖率可以写成：

```math
C_{\mathrm{cost}}=\frac{H_{\mathrm{attributed}}}{H_{\mathrm{total}}}
```

其中，`H_attributed` 是能归因到 team、project、user 或 job 的 GPU-hours，`H_total` 是总 GPU-hours。

综合门禁可以写成：

```math
G_{\mathrm{train\_platform}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{repr}}\ge \gamma \land A_{\mathrm{lineage}}\ge \eta \land C_{\mathrm{cost}}\ge \kappa \land P_0=0\right]
```

其中，`\tau_j` 是每个维度的最低覆盖率，`\gamma`、`\eta`、`\kappa` 分别是可复现、artifact 血缘和成本归因阈值，`P_0` 是 P0 级硬阻断数量。

## 16.20 面试中如何回答训练平台总览

如果面试官问：

```text
如何设计一个大模型训练平台？
```

可以这样回答：

```text
我会把训练平台看作训练任务生命周期管理系统，而不是脚本启动器。

用户通过 Portal、CLI 或 SDK 提交 TrainingJob，声明镜像、代码 commit、入口命令、训练配置、数据版本、资源需求、分布式配置、checkpoint 策略、队列和优先级。

平台先做配置校验、权限校验和配额校验，然后交给调度层做队列、gang scheduling、拓扑感知和资源分配。启动层负责 torchrun、DeepSpeed、Megatron 或 Ray 的分布式启动。

运行过程中，平台采集日志、指标和事件，记录 loss、tokens/s、step time、GPU 利用率、通信时间和 checkpoint 状态。Checkpoint Manager 负责保存、恢复和生命周期管理。

训练结束后，平台把 checkpoint、模型权重、tokenizer、config、eval report 和训练 metadata 注册到模型仓库或 artifact store，并把 GPU-hours、存储和网络成本归因到团队和项目。

横向还要有权限、安全、审计和失败恢复机制，保证训练可复现、可观测、可恢复、可治理。
```

## 16.21 常见误区

误区一：训练平台就是提交任务。

提交只是入口，真正的平台还要做调度、数据、checkpoint、监控、实验追踪、产物和治理。

误区二：有 Kubernetes 就有训练平台。

K8s 是底座，训练平台还要理解训练语义、分布式启动、checkpoint 和实验追踪。

误区三：日志够了，不需要事件。

事件记录任务生命周期，能帮助定位任务卡在哪个阶段。

误区四：训练成功就结束了。

训练产物还要注册、评估、审核、发布和成本归因。

误区五：失败自动重试越多越好。

配置错误和代码 bug 不适合重试，自动重试要区分错误类型。

误区六：TrainingJob YAML 字段越多越平台化。

字段多不等于治理强。高质量 TrainingJob 要能连接配置校验、权限、配额、launcher、checkpoint、实验追踪、artifact、审计和成本，而不是把所有参数堆进 YAML。

误区七：实验追踪可以替代训练平台。

实验追踪解决参数、指标和 artifact 记录问题，但不负责 GPU admission、队列、分布式 worker 生命周期、失败恢复、权限和容量治理。

## 16.22 面试题

### 题 1：训练平台和训练脚本的区别是什么？

答：训练脚本只负责执行训练逻辑。训练平台负责训练任务生命周期管理，包括任务提交、配置校验、资源调度、分布式启动、日志指标、checkpoint、失败恢复、实验追踪、产物注册、权限审计和成本归因。

### 题 2：TrainingJob 应该包含哪些字段？

答：应包含 job_id、owner、project、image、code_version、entrypoint、config、dataset_version、resource_request、distributed_config、checkpoint_config、logging_config、priority、queue 和 tags 等。

### 题 3：为什么训练平台需要实验追踪？

答：实验追踪记录代码、数据、配置、环境、超参、指标、checkpoint 和评估结果，帮助团队复现实验、比较模型、定位效果变化原因，并支撑后续发布和审计。

### 题 4：训练失败后平台应该如何处理？

答：平台应先分类错误。如果是节点故障、临时网络问题、存储短暂失败，可以自动重试或从 checkpoint 恢复。如果是配置错误、代码 bug、数据权限错误，应快速失败并给出诊断信息，而不是盲目重试。

### 题 5：为什么模型产物不能只是一个文件路径？

答：模型产物需要带 metadata，包括来源训练任务、数据版本、代码 commit、配置、checkpoint、eval report、权限、owner 和发布状态。只有这样才能评估、审核、部署、回滚和审计。

### 题 6：如何判断一个训练平台是否达到可上线门槛？

答：不能只看能否提交任务，而要看 TrainingJob 契约、配置校验、资源配额、环境复现、数据血缘和权限、分布式 launcher、checkpoint 恢复、日志指标事件、实验追踪、artifact 注册、失败分类、权限审计、成本归因、开发者自助和状态机是否合法。最后用 Training Platform Gate 收束：关键覆盖率过阈值、可复现字段完整、artifact lineage 完整、成本可归因且没有 P0 硬阻断。

## 16.24 本章小结

本章讲了训练平台总览。

你需要掌握：

1. 训练平台是训练任务生命周期管理系统，不是脚本启动器。
2. 平台化训练把环境、代码、数据、配置、资源、日志、checkpoint 和产物结构化。
3. TrainingJob 是训练平台的核心对象。
4. 训练平台需要任务提交、配置校验、权限配额、调度、launcher、数据接入、checkpoint、可观测性、实验追踪、产物注册、成本和审计。
5. 分布式训练启动需要处理 master、rank、world size、worker 和失败重试。
6. Checkpoint、实验追踪和模型仓库是训练可恢复、可复现和可发布的基础。
7. 成熟训练平台应从脚本化、任务化、平台化逐步走向治理化。
8. 第二轮精修后，本章新增 Training Platform Gate：用审计指标和 0 依赖 demo 把训练平台从功能清单升级成可验证门禁。
