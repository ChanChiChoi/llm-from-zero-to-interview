# 第 51 章 审计、日志留存、事故复盘和变更管理

上一章讲了安全治理。本章继续讲 AI Infra 的治理体系：审计、日志留存、事故复盘和变更管理。

如果说监控和告警帮助我们发现问题，审计和复盘帮助我们解释问题，变更管理帮助我们减少问题。成熟的 AI Infra 平台，不能只靠工程师临场排查，而要让关键操作、关键数据、关键事件都可追溯。

先记住一句话：

> 审计、留存、复盘和变更管理的目标，是让平台上的每一次重要操作都可追溯，每一次事故都能学习，每一次变更都能控制风险。

## 51.1 为什么 AI Infra 需要审计

AI Infra 中的关键资产很多：

1. 数据集。
2. 模型权重。
3. checkpoint。
4. endpoint。
5. prompt 模板。
6. 工具定义。
7. 知识库。
8. trace。
9. secret。
10. 发布配置。

任何一个资产被误改、误删、误用，都可能造成严重后果。

审计要回答：谁在什么时候，对什么资源，做了什么操作，结果是什么。

## 51.2 审计和普通日志的区别

普通日志用于排查系统运行问题。

审计日志用于追踪关键操作和安全责任。

普通日志可以比较详细，保留时间可能较短。

审计日志通常要求：

1. 结构化。
2. 不可随意篡改。
3. 保留更久。
4. 可按主体和资源查询。
5. 记录操作结果。
6. 支持合规和调查。

不要把审计日志和应用 debug 日志混为一谈。

## 51.3 审计日志应该记录什么

审计日志基本字段：

1. timestamp。
2. actor。
3. actor type。
4. tenant。
5. action。
6. resource type。
7. resource ID。
8. request ID。
9. source IP。
10. user agent。
11. decision。
12. reason。
13. before/after diff。
14. status。

actor 可以是人、服务账号、训练任务、Agent 或自动化系统。

decision 表示权限判断结果，例如 allow 或 deny。

## 51.4 需要审计的操作

AI Infra 中需要审计的操作包括：

数据相关：

1. 访问敏感 dataset。
2. 导出数据。
3. 删除数据。
4. 修改数据权限。

模型相关：

1. 下载模型权重。
2. 注册模型版本。
3. 发布模型。
4. 回滚模型。
5. 删除模型 artifact。

平台相关：

1. 修改队列配额。
2. 修改限流策略。
3. 修改路由规则。
4. 修改安全策略。
5. 访问 secret。

Agent/RAG 相关：

1. 调用高风险工具。
2. 修改工具定义。
3. 修改知识库权限。
4. 查看敏感 trace。

## 51.5 审计日志存储要求

审计日志存储要满足：

1. 可靠写入。
2. 防篡改。
3. 可查询。
4. 可导出。
5. 权限隔离。
6. 长期留存。
7. 加密存储。
8. 生命周期管理。

防篡改可以通过 append-only 存储、对象锁、签名、专用审计系统等方式实现。

审计日志本身也可能包含敏感信息，所以访问审计日志也要被审计。

## 51.6 日志留存策略

日志留存不是越久越好，也不是越短越好。

要按日志类型制定策略：

1. Debug 日志。
2. 训练日志。
3. 推理请求日志。
4. RAG/Agent trace。
5. 安全日志。
6. 审计日志。
7. 成本日志。
8. 发布事件日志。

不同日志有不同价值、成本和风险。

例如审计日志可能需要长期保存，而包含用户 prompt 的 trace 可能需要短 TTL 或脱敏保存。

## 51.7 日志留存的权衡

留存时间受多个因素影响：

1. 排障需求。
2. 安全调查需求。
3. 合规要求。
4. 存储成本。
5. 隐私风险。
6. 数据删除请求。
7. 业务价值。

完整保存所有请求内容很方便排障，但成本和隐私风险很高。

常见策略是：结构化摘要长期保存，敏感内容短期保存或采样保存。

## 51.8 日志分级

日志可以分级：

1. Operational logs：运行日志。
2. Security logs：安全日志。
3. Audit logs：审计日志。
4. Diagnostic traces：诊断 trace。
5. Business events：业务事件。

不同等级决定：

1. 谁能看。
2. 保存多久。
3. 是否脱敏。
4. 是否加密。
5. 是否允许导出。

AI 系统日志经常包含用户输入和模型输出，分级非常重要。

## 51.9 日志脱敏和最小化

日志留存应遵循最小化原则。

能记录摘要，就不要默认记录完整敏感内容。

常见脱敏对象：

1. API key。
2. access token。
3. 手机号。
4. 邮箱。
5. 身份证件号。
6. 私有文档片段。
7. 工具调用参数。
8. 用户 prompt。

脱敏要在写入前做，而不是写入后再清洗。

## 51.10 Trace 留存

RAG/Agent trace 很有价值，但也敏感。

Trace 留存策略可以是：

1. 错误 trace 保留更久。
2. 慢请求 trace 保留更久。
3. 普通成功 trace 采样。
4. 高敏请求只保留结构化元数据。
5. 工具参数脱敏。
6. 检索内容按权限控制。
7. 用户可请求删除。

Trace 是调试资产，也是隐私资产。

## 51.11 事故复盘的价值

事故复盘的目标不是追责，而是让系统变得更可靠。

复盘要回答：

1. 发生了什么？
2. 影响了谁？
3. 为什么发生？
4. 为什么没有更早发现？
5. 为什么没有自动恢复？
6. 如何防止再次发生？

如果每次事故只是临时修复，下次还会发生类似问题。

## 51.12 事故复盘模板

一个复盘模板可以包括：

1. 事故标题。
2. 事故等级。
3. 时间线。
4. 影响范围。
5. 用户影响。
6. 检测方式。
7. 响应过程。
8. 根因分析。
9. 触发因素。
10. 哪些防线失效。
11. 修复措施。
12. 后续行动项。
13. 负责人。
14. 截止时间。

AI Infra 复盘还应记录相关模型、数据、runtime、集群、路由和安全策略版本。

## 51.13 时间线很重要

事故时间线应该包含：

1. 首次异常发生时间。
2. 告警触发时间。
3. 值班响应时间。
4. 影响扩大时间。
5. 应急操作时间。
6. 服务恢复时间。
7. 根因确认时间。

时间线能暴露检测延迟和响应延迟。

如果问题发生 30 分钟后才告警，说明监控或 SLO 告警需要改进。

## 51.14 根因分析

根因分析不要停留在“某个服务挂了”。

要继续问：

1. 为什么服务挂了？
2. 为什么没有自动摘除？
3. 为什么 fallback 没生效？
4. 为什么发布前没测出来？
5. 为什么告警太晚？
6. 为什么值班 runbook 没覆盖？

常用方法是 5 Whys。

真正的根因往往是系统防线缺失，而不是某个人点错。

## 51.15 行动项管理

复盘没有行动项，就只是写报告。

行动项要具体：

1. 增加某个告警。
2. 修复某个自动回滚逻辑。
3. 增加发布前检查。
4. 改进 runbook。
5. 补充权限校验。
6. 增加数据质量门禁。
7. 改进 dashboard。

每个行动项要有 owner 和 deadline。

平台应跟踪行动项是否完成。

## 51.16 变更管理为什么重要

很多事故来自变更。

AI Infra 变更包括：

1. 模型版本发布。
2. prompt 修改。
3. runtime 升级。
4. 路由规则调整。
5. 安全策略变更。
6. 数据版本更新。
7. checkpoint 清理。
8. 集群驱动升级。
9. 网络配置变更。
10. 调度策略变更。

变更管理的目标是让高风险变更可审批、可灰度、可回滚、可审计。

## 51.17 变更记录应包含什么

变更记录包括：

1. 变更人。
2. 变更时间。
3. 变更对象。
4. 变更原因。
5. 变更内容。
6. 风险评估。
7. 回滚方案。
8. 审批记录。
9. 灰度计划。
10. 关联指标。

事故排查时，最近变更记录往往是第一入口。

## 51.18 高风险变更

高风险变更包括：

1. 生产模型全量发布。
2. 安全策略放宽。
3. RAG 权限规则修改。
4. 高风险工具启用。
5. 删除生产 artifact。
6. 调度器升级。
7. GPU driver 升级。
8. 网络策略修改。
9. Secret 轮换。
10. 跨地域数据策略变化。

高风险变更应需要审批、灰度和回滚验证。

## 51.19 变更冻结

在一些场景下需要变更冻结：

1. 大促期间。
2. 关键客户上线期间。
3. 错误预算耗尽。
4. 大规模事故恢复期间。
5. 安全事件调查期间。

冻结不是禁止所有变更，而是只允许低风险修复或经过审批的变更。

AI 平台中模型发布和 prompt 修改也应纳入冻结范围。

## 51.20 变更和 SLO

变更管理应和 SLO、错误预算联动。

例如：

1. 错误预算充足，正常灰度。
2. 错误预算快速消耗，放慢发布。
3. 错误预算耗尽，冻结高风险变更。
4. 变更后 SLO 破坏，自动回滚。

这能避免在系统已经不稳定时继续引入风险。

## 51.21 审计、复盘、变更的闭环

三者应该形成闭环：

```text
变更记录 -> 事件和告警 -> 事故复盘 -> 行动项 -> 改进变更流程
```

审计日志提供事实。

事故复盘提供分析。

变更管理减少未来风险。

如果三者割裂，治理体系会失效。

## 51.22 系统架构

一个治理平台可以包含：

```text
Audit Log Service
  -> Log Retention Policy Engine
  -> Change Management System
  -> Incident Management System
  -> Postmortem Repository
  -> Action Item Tracker
  -> Compliance Export
```

它应该和权限系统、发布系统、告警系统、模型仓库、数据平台和 RAG/Agent trace 平台打通。

## 51.23 常见误区

误区一：审计日志就是普通日志。

审计日志关注关键操作和责任追踪，要求更强完整性、留存和查询能力。

误区二：事故复盘是找人背锅。

复盘目标是修复系统防线，而不是追责个人。

误区三：Prompt 修改不是变更。

Prompt 修改会改变模型行为，必须纳入变更管理。

误区四：日志留存越久越好。

长期留存敏感内容会增加成本和隐私风险，要分级留存。

误区五：行动项写了就算完成。

行动项必须有 owner、deadline 和完成验证。

## 51.24 面试常见追问

问题一：审计日志应该记录什么？

可以回答：记录 actor、tenant、action、resource type、resource ID、timestamp、source、decision、reason、before/after diff 和 status，用于追踪关键操作。

问题二：AI Infra 哪些操作需要审计？

可以回答：数据访问和导出、模型下载和部署、权限变更、secret 访问、路由和安全策略修改、高风险工具调用、知识库权限修改、发布和回滚都需要审计。

问题三：事故复盘怎么做？

可以回答：记录时间线、影响范围、检测和响应过程、根因、防线失效点、修复措施和行动项，重点改进系统而不是追责个人。

问题四：为什么 prompt 修改也要变更管理？

可以回答：prompt 会影响模型行为、输出格式、安全策略和工具调用，可能造成线上质量和安全问题，因此需要版本、审批、灰度、回滚和审计。

## 51.26 本章小结

本章讲了审计、日志留存、事故复盘和变更管理。

你需要记住：

1. 审计用于追踪关键操作，普通日志用于排障，两者目标不同。
2. AI Infra 审计要覆盖数据、模型、权限、secret、发布、RAG/Agent 和高风险工具。
3. 日志和 trace 留存要在排障价值、合规、成本和隐私之间权衡。
4. 事故复盘要关注系统改进，不是追责个人。
5. 变更管理要覆盖模型、prompt、runtime、路由、数据、安全策略和基础设施配置。
6. 审计、复盘和变更管理应形成治理闭环。
