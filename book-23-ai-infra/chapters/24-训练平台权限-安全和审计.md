# 第 24 章 训练平台权限、安全和审计

上一章讲了训练数据供给。本章讲训练平台的权限、安全和审计。

训练平台连接了最贵的 GPU、最重要的数据、最敏感的模型权重和最复杂的代码执行环境。如果权限和审计没做好，轻则资源被滥用、成本失控，重则数据泄露、模型泄露、供应链攻击或合规事故。

先记住一句话：

> 训练平台安全不是给入口加登录，而是控制谁能用什么资源、读什么数据、运行什么镜像、产生什么模型、查看什么日志，并能事后追踪全过程。

## 24.1 训练平台为什么高风险

训练平台有几个天然风险。

第一，它能访问大量数据。

训练数据可能包含用户数据、业务机密、代码、文档、图片、音频、日志和标注信息。

第二，它能运行用户代码。

训练脚本本质上是任意代码执行。如果没有限制，用户代码可以访问网络、文件系统、密钥和其他服务。

第三，它能使用昂贵资源。

GPU 成本高，资源滥用会直接造成损失。

第四，它会产生模型产物。

模型权重、checkpoint、adapter、评估报告都可能继承训练数据的敏感性。

第五，它有复杂供应链。

镜像、Python 包、CUDA kernel、训练框架、数据处理脚本都可能引入风险。

因此训练平台必须从设计上内置权限、安全和审计。

## 24.2 安全目标

训练平台安全要实现：

1. 身份可信。
2. 权限最小化。
3. 数据访问可控。
4. 资源使用可控。
5. 镜像和代码可信。
6. 密钥不泄露。
7. 日志和产物不越权。
8. 高风险操作可审批。
9. 全流程可审计。
10. 事故可追溯。

这些目标分别对应身份、权限、数据、运行时、供应链、密钥、审计和治理模块。

## 24.3 身份模型

训练平台里至少有几类身份：

1. 用户身份。
2. 团队或项目身份。
3. Service Account。
4. TrainingJob 身份。
5. 平台服务身份。
6. 管理员身份。

用户身份用于提交任务和查看结果。

TrainingJob 身份用于任务运行时访问数据、checkpoint、日志和模型仓库。

平台服务身份用于调度、创建 Pod、写审计日志、注册 artifact。

不要让所有训练任务共享一个高权限平台身份。

更好的方式是为每个任务分配最小权限的运行身份。

## 24.4 RBAC 和 ABAC

RBAC 是基于角色的访问控制。

例如：

1. platform_admin。
2. project_admin。
3. trainer。
4. viewer。
5. auditor。

ABAC 是基于属性的访问控制。

属性可以包括：

1. project。
2. team。
3. environment。
4. data_classification。
5. resource_pool。
6. job_type。
7. model_stage。

训练平台通常需要 RBAC + ABAC。

例如：

```text
用户是 trainer，但只能在 project_a 下提交 normal priority 任务，且只能访问 internal 等级数据，不能访问 restricted 数据。
```

只用 RBAC 很难表达这种细粒度策略。

## 24.5 数据权限

训练平台必须控制数据访问。

数据权限要回答：

1. 用户能不能看到数据集？
2. 任务能不能读取数据？
3. 数据能不能用于训练？
4. 数据能不能跨区域读取？
5. 数据是否需要脱敏？
6. 训练出的模型是否继承数据等级？

数据权限不应该只靠文件系统 ACL。

平台需要理解 dataset_id、version、classification、owner、allowed_projects、allowed_purpose。

训练任务提交时必须校验数据权限。

运行时也要用任务身份访问数据，而不是用户把长期密钥写进脚本。

## 24.6 数据等级传播

使用敏感数据训练出的产物可能也敏感。

产物包括：

1. checkpoint。
2. final model。
3. LoRA / adapter。
4. eval report。
5. logs。
6. sample outputs。

规则示例：

```text
如果训练使用 restricted 数据，则 checkpoint、模型权重和 eval report 默认标记为 restricted。
```

这叫数据等级传播。

没有这条规则，用户可能无法读取原始数据，却可以下载用该数据训练出的模型。

## 24.7 资源权限

资源权限控制谁能用哪些 GPU 和队列。

常见策略：

1. 普通用户只能使用 normal 队列。
2. 高优先级队列需要审批。
3. H100 资源池只对关键项目开放。
4. 生产推理资源池禁止训练任务使用。
5. 可抢占资源允许实验任务使用。
6. 每个团队有 GPU-hour 预算。

资源权限和成本治理相关。

如果没有资源权限，昂贵 GPU 很容易被低价值任务占满。

## 24.8 镜像安全

训练任务运行镜像，因此镜像是安全入口。

风险：

1. 镜像包含漏洞。
2. 镜像包含恶意代码。
3. 镜像包含密钥。
4. 镜像来源不可信。
5. tag 被覆盖。
6. 基础镜像过旧。

平台应要求：

1. 只允许可信 registry。
2. 镜像扫描通过。
3. 记录 digest。
4. 高风险漏洞阻断。
5. 禁止密钥进入镜像。
6. 镜像签名。
7. 基础镜像白名单。

提交任务时就应该校验镜像，不要等 Pod 启动后才发现风险。

## 24.9 代码安全

训练平台运行用户代码。

代码风险包括：

1. 访问未授权数据。
2. 向外发送数据。
3. 读取宿主机路径。
4. 打印敏感信息。
5. 下载未知依赖。
6. 执行危险命令。

治理措施：

1. 限制运行时权限。
2. 禁止 privileged 容器。
3. 限制 hostPath。
4. 控制出网。
5. 代码仓库权限校验。
6. 高风险任务审批。
7. 日志脱敏。

训练平台不能假设所有训练代码都是安全的。

## 24.10 密钥管理

训练任务可能需要访问对象存储、数据服务、模型仓库和实验追踪系统。

不要把密钥写入：

1. 镜像。
2. 代码。
3. 配置文件。
4. 日志。
5. checkpoint。

更好的方式：

1. 使用 Secret Manager。
2. 按任务注入短期凭证。
3. 使用 Service Account。
4. 最小权限。
5. 自动轮换。
6. 访问审计。
7. 日志脱敏。

密钥泄露是训练平台重大事故。

## 24.11 网络安全

训练任务是否能访问外网、生产数据库、内部服务，需要严格控制。

网络策略包括：

1. 默认禁止跨租户访问。
2. egress 白名单。
3. 禁止访问生产敏感服务。
4. 只允许访问必要数据 endpoint。
5. 限制调试端口暴露。
6. 记录外部访问日志。

很多数据泄露不是从存储权限发生，而是训练代码把数据发到了外部 endpoint。

## 24.12 日志安全

训练日志可能包含敏感信息。

平台要控制：

1. 谁能看日志。
2. 日志保留多久。
3. 日志是否脱敏。
4. 是否允许下载。
5. 是否记录查看行为。

日志脱敏对象包括：

1. token。
2. 密钥。
3. 邮箱。
4. 手机号。
5. 用户输入。
6. 数据样本。
7. 连接串。

日志系统本身也是安全系统的一部分。

## 24.13 模型产物权限

模型产物包括：

1. checkpoint。
2. final model。
3. adapter。
4. tokenizer。
5. config。
6. eval report。
7. deployment package。

产物权限要根据：

1. 项目。
2. 数据等级。
3. 模型阶段。
4. owner。
5. 发布状态。
6. 审批结果。

例如训练中的 checkpoint 只能项目成员读，发布后的模型可以给更多应用调用，但下载权仍需限制。

模型权重是资产，不是普通文件。

## 24.14 高风险操作审批

高风险操作包括：

1. 使用 restricted 数据训练。
2. 导出模型权重。
3. 发布到生产。
4. 使用高优先级队列。
5. 关闭日志或 checkpoint。
6. 使用未审核镜像。
7. 访问外部网络。
8. 删除重要 checkpoint。

这些操作应该触发审批或策略阻断。

审批记录要进入审计日志。

## 24.15 审计日志

审计日志要记录谁在什么时候做了什么。

训练平台审计对象包括：

1. 任务提交。
2. 任务取消。
3. 数据访问。
4. 镜像使用。
5. checkpoint 创建和删除。
6. 模型注册和下载。
7. 权限修改。
8. 配额修改。
9. 高风险审批。
10. 日志查看。

审计日志字段：

1. actor。
2. action。
3. resource。
4. timestamp。
5. decision。
6. reason。
7. request_id。
8. ip / source。
9. metadata。

审计日志要防篡改，并有留存策略。

## 24.16 安全事件响应

训练平台要能响应安全事件。

例如：

1. 密钥泄露。
2. 未授权数据访问。
3. 镜像漏洞。
4. 模型权重误下载。
5. 日志泄露。
6. 异常外联。

响应流程：

1. 发现事件。
2. 冻结相关任务。
3. 撤销密钥。
4. 隔离产物。
5. 查询审计日志。
6. 通知 owner。
7. 修复策略。
8. 复盘。

没有审计，就无法做安全响应。

## 24.17 面试中如何回答训练平台安全

如果面试官问：

```text
大模型训练平台如何做权限、安全和审计？
```

可以这样回答：

```text
我会从身份、资源、数据、镜像、代码、密钥、网络、日志、模型产物和审计几个层面设计。

身份层接入统一 IdP，区分用户身份、TrainingJob 运行身份、Service Account 和平台服务身份。权限层用 RBAC + ABAC，控制谁能提交任务、使用哪些资源池、访问哪些数据、查看哪些日志、注册和发布哪些模型。

数据层按 dataset_id、version、classification 和 allowed_purpose 做授权，并做数据等级传播：使用敏感数据训练出的 checkpoint、模型和 eval report 继承敏感等级。资源层用队列、配额和优先级控制 GPU 使用。

运行层只允许可信镜像 registry，做镜像扫描、签名和 digest 记录；禁止 privileged、限制 hostPath、控制 egress，密钥通过 Secret Manager 注入短期凭证，避免写入镜像和日志。

审计层记录任务提交、数据访问、镜像使用、checkpoint 创建删除、模型注册下载、权限变更、高风险审批和日志查看。高风险操作如使用 restricted 数据、导出权重、生产发布和关闭日志需要审批。
```

## 24.18 常见误区

误区一：训练平台安全就是登录鉴权。

登录只是身份入口，还要控制数据、资源、镜像、密钥、网络、日志和模型产物。

误区二：用户能读数据，就能随便训练。

数据使用目的也要授权，训练产物可能继承数据敏感等级。

误区三：平台管理员可以看所有日志。

日志可能包含敏感数据，管理员访问也应最小权限和审计。

误区四：镜像能跑就行。

镜像必须可信、扫描、签名、记录 digest，并防止密钥泄露。

误区五：审计只在事故后需要。

审计是日常治理、合规、成本和安全响应的基础。

## 24.19 面试题

### 题 1：训练平台有哪些身份？

答：包括用户身份、团队或项目身份、TrainingJob 运行身份、Service Account、平台服务身份和管理员身份。任务运行时应使用最小权限的任务身份，而不是共享高权限身份。

### 题 2：为什么训练产物要继承数据等级？

答：模型权重、checkpoint、adapter 和评估报告可能包含或泄露训练数据的信息。如果使用敏感数据训练，产物也应继承相应敏感等级，避免用户绕过原始数据权限访问模型产物。

### 题 3：镜像安全要做哪些事？

答：使用可信 registry、镜像扫描、漏洞阻断、镜像签名、digest 记录、基础镜像白名单、禁止密钥进入镜像，并在任务提交时校验镜像策略。

### 题 4：哪些训练操作属于高风险操作？

答：使用敏感数据训练、导出模型权重、发布到生产、使用高优先级队列、关闭日志或 checkpoint、使用未审核镜像、访问外部网络、删除重要 checkpoint 等。

### 题 5：审计日志应该记录什么？

答：应记录 actor、action、resource、timestamp、decision、reason、request_id、source 和 metadata。覆盖任务提交、数据访问、镜像使用、checkpoint、模型产物、权限变更、审批和日志查看。

## 24.20 小练习

练习一：设计一个训练平台权限模型。

要求：包含 admin、project_admin、trainer、viewer、auditor 五类角色和对应权限。

练习二：设计数据等级传播规则。

要求：说明 public、internal、restricted 数据训练出的 checkpoint、model、eval report 应如何定级。

练习三：设计高风险操作审批流程。

要求：以“使用 restricted 数据训练并导出模型权重”为例，列出审批、审计和产物权限规则。

练习四：设计审计日志 schema。

要求：包含 actor、action、resource、decision、reason、timestamp、request_id 和 metadata。

## 24.21 本章小结

本章讲了训练平台权限、安全和审计。

你需要掌握：

1. 训练平台安全覆盖身份、资源、数据、镜像、代码、密钥、网络、日志、模型产物和审计。
2. 训练任务运行时应使用最小权限任务身份。
3. RBAC + ABAC 比单纯角色权限更适合训练平台。
4. 数据权限要控制使用目的，并处理产物敏感等级传播。
5. 资源权限和成本治理相关，高价值 GPU 和高优先级队列需要控制。
6. 镜像安全和代码运行时安全是供应链和执行安全的关键。
7. 密钥必须通过 Secret Manager 和短期凭证管理，不能写进镜像、代码或日志。
8. 日志和模型产物都需要权限和审计。
9. 高风险操作需要审批。
10. 审计日志是安全响应、合规和复盘的基础。
