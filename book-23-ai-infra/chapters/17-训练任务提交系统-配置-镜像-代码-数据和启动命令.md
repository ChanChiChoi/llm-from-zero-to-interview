# 第 17 章 训练任务提交系统：配置、镜像、代码、数据和启动命令

上一章讲了训练平台总览。本章深入训练平台的入口：训练任务提交系统。

训练任务提交系统看起来只是“把一个任务提交上去”，但它其实是训练平台治理的第一道关口。它决定了任务能否被复现、能否被调度、能否被审计、能否定位问题、能否恢复训练。

先记住一句话：

> 好的训练任务提交系统，不是让用户少写几行命令，而是把配置、镜像、代码、数据、资源、启动方式和权限变成结构化、可校验、可追踪的训练声明。

## 17.0 本讲资料边界与第二轮精修口径

本讲按第二轮精修口径，只抽象训练任务提交系统的稳定工程边界，不绑定某个云厂商、某个内部平台、某个训练 operator 或某个配置中心实现。

资料校准时可以把几类官方口径放在一起看：

1. Kubernetes Job 关注的是让一组 Pod 运行到完成，能承载批处理任务，但它本身不负责理解训练任务的镜像 digest、代码 commit、数据版本、最终配置和实验血缘。
2. Kubernetes ResourceQuota、PriorityClass 和 admission / dry-run 口径提醒我们：提交系统不能只接受 YAML，还要把资源配额、优先级和服务端策略校验放到权威控制面。
3. OCI / Docker 镜像引用中，tag 更像可读别名，digest 才能稳定指向镜像内容；训练任务如果只记录 `latest`，后续很难复现。
4. JSON Schema 这类 schema 校验工具适合表达 required fields、type、enum、pattern 和嵌套对象边界，但业务校验还要继续检查镜像、代码、数据权限、队列策略和 checkpoint 路径。
5. Kueue 这类队列系统把 Workload、LocalQueue、ClusterQueue、ResourceFlavor、admission 和 quota 作为调度治理对象；提交系统应该把 TrainingJob 转成可排队、可审计的工作负载，而不是绕过队列直接起容器。
6. PyTorch `torchrun` 关注 `nnodes`、`nproc_per_node`、rank、world size、rendezvous 和 elastic restart 等启动参数；提交系统要提前检查这些参数和 GPU 资源 shape 是否一致。

所以本章的结论是：训练任务提交系统是训练平台的控制面入口。它的核心价值不是“帮用户拼命令”，而是把训练声明变成可校验、可复现、可排队、可追踪、可幂等提交的对象。

## 17.1 为什么任务提交系统重要

如果训练任务只是一个命令：

```bash
python train.py --config config.yaml
```

平台很难知道：

1. 使用了哪个镜像？
2. 使用了哪个代码版本？
3. 使用了哪个数据版本？
4. 需要多少 GPU？
5. 是单机还是多机？
6. checkpoint 保存在哪里？
7. 用户是否有数据权限？
8. 任务失败能否恢复？
9. 训练产物属于哪个项目？
10. 成本应该归因给谁？

这些信息如果只存在用户脑子里或聊天记录里，训练就无法平台化。

任务提交系统的价值就是把这些信息变成平台可理解的结构化对象。

## 17.2 提交系统的核心职责

训练任务提交系统要做八件事：

1. 接收任务声明。
2. 校验配置合法性。
3. 校验镜像、代码和数据版本。
4. 检查权限、配额和策略。
5. 生成标准 TrainingJob。
6. 交给调度系统排队。
7. 记录任务 metadata。
8. 返回任务状态和可观测入口。

它不是直接跑训练，而是把训练请求转换成可管理的任务对象。

## 17.3 提交入口

常见提交入口包括：

1. Web Portal。
2. CLI。
3. Python SDK。
4. REST / gRPC API。
5. Notebook 插件。
6. CI/CD 流水线。

不同入口服务不同用户。

Portal 适合查看、编辑、对比和管理任务。

CLI 适合算法工程师日常使用。

SDK 适合自动化实验。

API 适合上层平台集成。

CI/CD 适合模型训练流水线。

但不管入口是什么，最终都应该落到同一种 TrainingJob schema。

## 17.4 TrainingJob Schema

TrainingJob schema 是提交系统的核心。

它应该覆盖：

1. 基本信息。
2. 镜像。
3. 代码。
4. 数据。
5. 启动命令。
6. 训练配置。
7. 分布式配置。
8. 资源配置。
9. checkpoint 配置。
10. 日志和指标配置。
11. 队列和优先级。
12. 权限和标签。

一个简化示例：

```yaml
apiVersion: ai.example.com/v1
kind: TrainingJob
metadata:
  name: sft-chat-7b-v3
  project: customer-support
  owner: alice
spec:
  image: registry/train:pytorch2.2-cuda12.1@sha256:abc
  code:
    repo: git@example.com:llm/train.git
    commit: 8f31c2a
  data:
    dataset_id: support_dialog_sft
    version: 2026-05-30
  command:
    - python
    - train.py
    - --config
    - configs/sft_7b.yaml
  resources:
    gpu_type: A100-80G
    gpu_count: 8
    gpu_per_node: 8
  distributed:
    launcher: torchrun
    nnodes: 1
    nproc_per_node: 8
  checkpoint:
    output_uri: s3://ckpts/customer-support/sft-chat-7b-v3/
    save_every_steps: 1000
  queue: sft
  priority: normal
```

Schema 的目标是让训练任务可解析、可校验、可复现。

## 17.5 基本信息

基本信息包括：

1. job_name。
2. owner。
3. project。
4. team。
5. description。
6. tags。
7. created_at。
8. experiment_id。
9. cost_center。

这些字段看起来不影响训练代码，但对平台非常重要。

它们用于：

1. 搜索任务。
2. 成本归因。
3. 权限判断。
4. 实验追踪。
5. 审计。
6. 运维通知。

一个没有 owner 和 project 的训练任务，在企业平台里是不合格的。

## 17.6 镜像字段

镜像字段决定任务运行环境。

至少要记录：

1. image registry。
2. image tag。
3. image digest。
4. base image。
5. CUDA 版本。
6. PyTorch 版本。
7. 构建时间。
8. 安全扫描状态。

推荐记录 digest，而不是只记录 tag。

因为 tag 可以被覆盖，digest 才能保证复现。

提交系统要校验：

1. 镜像是否存在。
2. 镜像是否来自可信 registry。
3. 镜像是否通过安全扫描。
4. 镜像 CUDA 是否和目标集群兼容。
5. 镜像是否允许在目标资源池运行。

## 17.7 代码字段

代码字段决定训练逻辑。

至少要记录：

1. repo。
2. commit。
3. branch。
4. diff 状态。
5. entrypoint。
6. subdirectory。
7. code package URI。

最推荐记录不可变 commit，而不是只记录 branch。

因为 branch 会移动。

如果允许用户提交未提交的本地 diff，平台应该把 diff 打包并记录 hash，否则后续无法复现。

提交系统要校验：

1. repo 是否存在。
2. commit 是否存在。
3. 用户是否有访问权限。
4. entrypoint 是否存在。
5. 是否使用了禁止路径或危险脚本。

## 17.8 数据字段

数据字段决定训练输入。

至少要记录：

1. dataset_id。
2. dataset_version。
3. data_uri。
4. split。
5. preprocessing_version。
6. tokenizer_version。
7. data_classification。
8. access_policy。

数据字段必须可版本化。

不能只写：

```text
/data/latest
```

因为 latest 会变化。

提交系统要校验：

1. 数据集是否存在。
2. 版本是否存在。
3. 用户是否有权限。
4. 数据等级是否允许该任务使用。
5. 数据是否允许跨区域或跨租户。
6. 训练产物是否需要继承数据等级。

## 17.9 启动命令

启动命令看似简单，但很容易出问题。

它包括：

1. entrypoint。
2. args。
3. environment variables。
4. working directory。
5. shell mode。
6. distributed launcher。

建议使用数组形式，而不是一整段 shell 字符串。

例如：

```yaml
command:
  - python
  - train.py
args:
  - --config
  - configs/sft.yaml
```

这样更容易校验和审计。

如果允许任意 shell 命令，平台很难判断命令做了什么，也更容易出现安全问题。

## 17.10 训练配置

训练配置包括：

1. model config。
2. optimizer config。
3. learning rate。
4. batch size。
5. sequence length。
6. precision。
7. gradient accumulation。
8. checkpoint frequency。
9. eval frequency。
10. seed。

配置可以来自：

1. YAML 文件。
2. JSON 文件。
3. 命令行参数。
4. 配置中心。
5. 实验模板。

关键是提交时要记录最终生效配置。

不能只记录“用了 configs/train.yaml”，还要记录这个文件的内容版本。

否则文件后续被修改，实验就无法复现。

## 17.11 分布式配置

分布式配置包括：

1. launcher。
2. nnodes。
3. nproc_per_node。
4. world_size。
5. master_addr。
6. master_port。
7. tensor_parallel_size。
8. pipeline_parallel_size。
9. data_parallel_size。
10. ZeRO / FSDP 配置。

提交系统要检查一致性。

例如：

```text
gpu_count = nnodes * nproc_per_node
world_size = gpu_count
```

如果用户申请 16 张 GPU，但配置里写 8 个进程，就会出问题。

## 17.12 资源配置

资源配置包括：

1. GPU 类型。
2. GPU 数量。
3. 每节点 GPU 数。
4. CPU。
5. 内存。
6. 本地盘。
7. 网络需求。
8. 存储挂载。
9. 是否需要完整节点。
10. 是否可抢占。

训练任务提交系统应该尽量让用户表达训练需求，而不是直接写底层 K8s 细节。

例如用户写：

```yaml
resources:
  profile: 8xA100-80G-single-node
```

平台展开成具体 GPU、CPU、内存、node selector 和调度约束。

## 17.13 Checkpoint 配置

Checkpoint 配置包括：

1. output_uri。
2. resume_from。
3. save_every_steps。
4. keep_last_n。
5. save_best。
6. async_save。
7. format。
8. shard_strategy。
9. retention_policy。

提交系统要校验：

1. output_uri 是否可写。
2. resume_from 是否存在。
3. resume checkpoint 是否 committed。
4. checkpoint 格式是否兼容。
5. 是否符合存储配额。

如果任务可抢占，checkpoint 配置应强制开启或至少给出警告。

## 17.14 日志和指标配置

日志和指标配置包括：

1. log level。
2. log destination。
3. metrics destination。
4. tensorboard uri。
5. experiment tracker。
6. sample logging。
7. rank log policy。
8. retention days。

注意日志可能包含敏感信息。

提交系统应支持：

1. 日志访问权限。
2. 日志脱敏。
3. 日志保留策略。
4. 关键事件采集。

训练平台不能只把所有日志丢进一个公共目录。

## 17.15 队列和优先级配置

提交时要指定：

1. queue。
2. priority。
3. preemptible。
4. max_runtime。
5. deadline。
6. cost_budget。

提交系统要校验：

1. 用户是否能使用该队列。
2. 用户是否能申请该优先级。
3. 队列配额是否足够。
4. 任务是否符合队列策略。
5. 可抢占任务是否配置 checkpoint。

优先级不是用户随便填的字符串，而是平台治理对象。

## 17.16 提交后的状态机

训练任务提交后，不是直接 running。

可以有如下状态：

```text
draft -> submitted -> validating -> rejected
                         -> queued -> scheduled -> starting -> running
                                                   -> failed
                                                   -> retrying
                                                   -> completed
                                                   -> cancelled
```

状态解释：

1. draft：用户编辑中。
2. submitted：已提交。
3. validating：平台校验中。
4. rejected：校验失败。
5. queued：进入队列等待资源。
6. scheduled：资源已分配。
7. starting：容器和 launcher 启动中。
8. running：训练中。
9. failed：失败。
10. retrying：重试或恢复中。
11. completed：完成。
12. cancelled：用户取消。

状态机让用户知道任务卡在哪里。

## 17.17 提交前校验和提交后校验

有些校验可以提交前完成：

1. YAML schema 是否正确。
2. 必填字段是否存在。
3. 镜像格式是否正确。
4. GPU 数量是否合法。
5. 分布式配置是否自洽。

有些校验必须提交后由平台完成：

1. 镜像是否存在。
2. 数据权限是否允许。
3. 配额是否足够。
4. checkpoint 路径是否可写。
5. 资源池是否支持目标 GPU。
6. 安全策略是否允许。

好的 CLI 可以做本地快速校验，平台 API 做权威校验。

## 17.18 幂等性和重复提交

提交系统要处理重复提交。

例如用户网络超时，不知道任务是否提交成功，又点了一次。

如果没有幂等设计，可能创建两个训练任务，浪费资源。

可以使用：

1. client_request_id。
2. job_name 唯一性。
3. submit token。
4. 幂等窗口。

提交 API 应该支持：同一个 client_request_id 重试时返回同一个 job_id。

这是平台工程里很实用但容易忽略的细节。

## 17.19 模板和复用

训练任务很多字段重复。

平台可以提供模板：

1. 7B SFT 模板。
2. 70B SFT 模板。
3. LoRA 微调模板。
4. 预训练模板。
5. Eval 模板。
6. RLHF rollout 模板。

模板好处：

1. 降低使用门槛。
2. 减少配置错误。
3. 统一最佳实践。
4. 便于治理。

但模板也要版本化。

否则模板更新后，历史任务无法解释。

## 17.20 训练任务提交审计指标与最小 demo

训练任务提交系统的设计，最终要能落到一张审计表：每个任务提交样本是否满足 schema、镜像、代码、数据、命令、配置、资源、分布式、checkpoint、队列、幂等、模板、状态机和 trace 的最低要求。

可以把第 `i` 个提交样本写成：

```math
z_i=(m_i,s_i,e_i,c_i,d_i,u_i,r_i,l_i,k_i,o_i,q_i,a_i,h_i,t_i,v_i)
```

其中，`m_i` 是 metadata，`s_i` 是 schema，`e_i` 是镜像环境，`c_i` 是代码版本，`d_i` 是数据版本和权限，`u_i` 是启动命令，`r_i` 是资源申请，`l_i` 是 launcher 和分布式参数，`k_i` 是 checkpoint，`o_i` 是日志指标，`q_i` 是队列和优先级，`a_i` 是 admission / dry run，`h_i` 是幂等信息，`t_i` 是模板版本，`v_i` 是状态机和审计 trace。

统一覆盖率可以写成：

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(z_i)=1]
```

其中，`g_j` 是第 `j` 个提交校验函数，`C_j` 是该维度在样本集上的通过率。

必填字段覆盖率可以写成：

```math
R_{\mathrm{required}}=\frac{K_{\mathrm{present}}}{K_{\mathrm{required}}}
```

其中，`K_present` 是已填写且非空的必填字段数量，`K_required` 是 schema 定义的必填字段数量。这个指标用于发现“看起来能提交，但缺 owner、project、queue、metrics destination 或 idempotency key”的灰色任务。

不可变输入绑定覆盖率可以写成：

```math
R_{\mathrm{binding}}=\frac{K_{\mathrm{immutable}}}{K_{\mathrm{input}}}
```

其中，`K_input` 至少包括 image digest、code commit、diff hash、dataset version、final config snapshot 和 template version，`K_immutable` 是其中已经被不可变标识绑定的输入数量。

分布式资源一致性可以写成：

```math
D_i=\mathbf{1}\left[g_i=n_i p_i \land w_i=g_i\right]
```

其中，`g_i` 是申请 GPU 数，`n_i` 是节点数，`p_i` 是每节点进程数，`w_i` 是 world size。很多训练任务不是代码坏了，而是提交时 `gpu_count`、`nnodes`、`nproc_per_node` 和 `world_size` 已经不一致。

重复提交率可以写成：

```math
R_{\mathrm{dup}}=\frac{N_{\mathrm{duplicate}}}{N_{\mathrm{retry}}}
```

其中，`N_retry` 是同一 `client_request_id` 的重试次数，`N_duplicate` 是重试后错误创建出的新任务数量。合格的提交 API 应该让这个值为 0。

综合门禁可以写成：

```math
G_{\mathrm{train\_submission}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{required}}\ge \gamma \land R_{\mathrm{binding}}\ge \eta \land R_{\mathrm{dup}}=0 \land P_0=0\right]
```

其中，`\tau_j` 是每个审计维度的最低通过率，`\gamma` 是必填字段覆盖率阈值，`\eta` 是不可变输入绑定阈值，`P_0` 是 P0 级硬阻断数量。

```python
def nested_get(mapping, path):
    cur = mapping
    for key in path:
        if not isinstance(cur, dict):
            return None
        cur = cur.get(key)
    return cur


def safe_ratio(numerator, denominator):
    return round(numerator / denominator, 3) if denominator else 0.0


def required_field_ratio(case):
    present = sum(1 for path in REQUIRED_PATHS if bool(nested_get(case, path)))
    return safe_ratio(present, len(REQUIRED_PATHS))
```

```text
我会把训练任务提交系统设计成 TrainingJob 声明入口，而不是简单命令提交器。

用户可以通过 Portal、CLI、SDK 或 API 提交 TrainingJob。TrainingJob schema 包含基本信息、镜像 digest、代码 repo 和 commit、数据集版本、启动命令、训练配置、分布式配置、资源需求、checkpoint 策略、日志指标配置、队列和优先级。

为了可复现，系统必须记录镜像 digest、代码 commit、数据版本、最终生效配置、启动命令、资源配置和环境 metadata。为了可靠性，提交 API 要支持幂等，避免重复提交。为了易用性，平台提供版本化模板，减少用户手写复杂配置。
```

## 17.22 常见误区

误区一：提交系统只是把命令发到集群。

真正的提交系统要做结构化声明、校验、权限、配额、审计和可复现记录。

误区二：记录 branch 就能复现代码。

Branch 会变化，应记录 commit；如果有本地 diff，也要打包和记录 hash。

误区三：数据路径足够描述数据。

路径可能变化，应记录 dataset_id、version、preprocessing_version 和权限策略。

误区四：启动命令越自由越好。

任意 shell 字符串难以校验和审计。结构化 command / args 更安全。

误区五：任务提交成功就等于能运行。

提交后还要经过校验、排队、调度、启动和运行多个阶段。

## 17.23 面试题

### 题 1：TrainingJob schema 为什么重要？

答：它把镜像、代码、数据、配置、资源、分布式启动、checkpoint、队列和权限变成结构化声明，方便校验、调度、复现、审计和排查。

### 题 2：为什么要记录镜像 digest 和代码 commit？

答：tag 和 branch 都可能变化，不能保证复现。Digest 唯一标识镜像内容，commit 唯一标识代码版本，是实验复现和审计的基础。

### 题 3：提交系统需要校验哪些内容？

答：需要校验 schema、镜像存在和安全状态、代码 commit、数据版本和权限、资源配额、GPU 类型、分布式配置一致性、checkpoint 路径、队列优先级权限和安全策略。

### 题 4：为什么提交 API 需要幂等？

答：用户或客户端可能因为网络超时重复提交。如果没有幂等，可能创建多个相同任务，浪费资源。使用 client_request_id 可以让重复提交返回同一个 job_id。

### 题 5：为什么训练配置要记录最终生效版本？

答：配置可能来自文件、命令行、模板和默认值。如果只记录配置文件路径，后续文件变化后无法复现。必须记录最终合并后的实际配置。

## 17.25 本章小结

本章讲了训练任务提交系统。

你需要掌握：

1. 任务提交系统是训练平台治理的第一道关口。
2. 好的提交系统把配置、镜像、代码、数据、资源和启动命令结构化。
3. TrainingJob schema 是核心对象。
4. 镜像要记录 digest，代码要记录 commit，数据要记录版本。
5. 启动命令应尽量结构化，方便校验和审计。
6. 分布式配置要和资源申请一致。
7. Checkpoint、日志、队列和优先级都应该是任务声明的一部分。
8. 提交系统需要状态机、校验、幂等和模板能力。
9. 可复现性来自完整 metadata，而不是单个训练脚本。
