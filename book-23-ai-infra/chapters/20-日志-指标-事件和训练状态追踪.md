# 第 20 章 日志、指标、事件和训练状态追踪

上一章讲了训练配置管理。本章讲训练平台的可观测性：日志、指标、事件和训练状态追踪。

大模型训练经常跑很久、资源很贵、链路很长。没有可观测性，任务失败时你只能看到“挂了”；任务变慢时你不知道是 GPU、网络、数据、checkpoint 还是代码问题；模型效果变化时你也不知道训练过程中发生过什么。

先记住一句话：

> 训练可观测性的目标不是收集更多日志，而是让平台能回答任务现在在哪、是否健康、为什么慢、为什么失败、是否可恢复、成本花在哪里。

## 20.1 为什么训练可观测性重要

训练任务不同于普通短任务。

它有几个特点：

1. 运行时间长。
2. 消耗 GPU 成本高。
3. 多节点多进程。
4. 故障原因复杂。
5. 中间状态重要。
6. 结果需要复现。
7. 失败恢复依赖 checkpoint。

没有可观测性，会出现：

1. 任务卡住没人知道。
2. GPU 空转没人发现。
3. loss 异常但继续烧钱。
4. checkpoint 保存失败很久才发现。
5. 某个 rank 慢拖住全部任务。
6. 训练失败后无法定位原因。
7. 实验对比缺少过程证据。

训练平台必须把可观测性作为核心能力，而不是上线后补日志。

## 20.2 日志、指标、事件、Trace 的区别

这几个概念经常混用，需要区分。

日志 logs：描述发生了什么，通常是文本或结构化记录。

指标 metrics：可聚合的数值时间序列，例如 loss、GPU utilization、tokens/s。

事件 events：任务生命周期中的关键状态变化，例如 job scheduled、checkpoint saved、worker failed。

Trace：跨组件、跨阶段的链路追踪，用来串起一次任务从提交到完成的全过程。

简单理解：

```text
Logs: 具体细节
Metrics: 数值趋势
Events: 状态变化
Traces: 因果链路
```

训练平台四者都需要。

## 20.3 训练日志

训练日志包括：

1. stdout / stderr。
2. rank 日志。
3. worker 日志。
4. launcher 日志。
5. dataloader 日志。
6. checkpoint 日志。
7. framework 日志。
8. 错误堆栈。

日志应该包含基本上下文：

1. job_id。
2. attempt_id。
3. node_id。
4. pod_id。
5. rank。
6. timestamp。
7. log_level。
8. component。

如果日志里没有 job_id 和 rank，排查分布式训练会非常痛苦。

## 20.4 Rank 维度日志

分布式训练有很多 rank。

常见问题：rank 0 日志正常，但 rank 17 已经报错。

所以日志系统要支持：

1. 按 job 查看所有日志。
2. 按 rank 过滤日志。
3. 按节点过滤日志。
4. 聚合错误堆栈。
5. 高亮第一个失败 rank。
6. 对 rank 0 主日志特殊展示。

一个实用设计是：默认展示 rank 0 关键日志，但错误排查时自动提示其他 rank 的异常。

## 20.5 日志脱敏和权限

训练日志可能包含敏感信息。

例如：

1. 数据样本片段。
2. 文件路径。
3. 用户输入。
4. API token。
5. 数据库连接串。
6. 模型输出。
7. 业务字段。

平台需要：

1. 日志访问权限。
2. 日志脱敏。
3. 日志保留策略。
4. 日志导出审计。
5. 敏感字段检测。

不要把所有训练日志默认暴露给所有平台用户。

## 20.6 训练指标

训练指标可以分成四类。

### 20.6.1 模型指标

包括：

1. loss。
2. eval loss。
3. perplexity。
4. accuracy。
5. reward。
6. safety score。
7. task-specific metric。

这些指标告诉我们模型是否在学习。

### 20.6.2 性能指标

包括：

1. tokens/s。
2. samples/s。
3. step time。
4. MFU。
5. HFU。
6. dataloader time。
7. forward time。
8. backward time。
9. optimizer time。
10. communication time。

这些指标告诉我们训练是否高效。

### 20.6.3 资源指标

包括：

1. GPU utilization。
2. GPU memory used。
3. GPU memory bandwidth。
4. CPU utilization。
5. host memory。
6. network throughput。
7. storage throughput。
8. disk usage。
9. power。
10. temperature。

这些指标告诉我们资源是否健康。

### 20.6.4 平台指标

包括：

1. queue wait time。
2. scheduling time。
3. image pull time。
4. startup time。
5. checkpoint save time。
6. checkpoint load time。
7. retry count。
8. preemption count。
9. cost。

这些指标告诉我们平台是否高效。

## 20.7 指标采集方式

指标来源包括：

1. 训练脚本主动上报。
2. Framework callback。
3. 日志解析。
4. GPU exporter。
5. Node exporter。
6. DCGM exporter。
7. Kubernetes metrics。
8. 存储和网络监控。
9. 平台控制面事件。

模型指标通常由训练脚本上报。

资源指标通常由 exporter 采集。

平台指标通常由训练平台控制面记录。

成熟平台要把这些指标用 job_id、rank、node、project 关联起来。

## 20.8 指标粒度

指标粒度要平衡成本和可诊断性。

粒度太粗：看不出问题。

粒度太细：存储成本高，查询慢。

常见维度：

1. job_id。
2. attempt_id。
3. rank。
4. node。
5. GPU id。
6. step。
7. timestamp。
8. component。

例如 GPU utilization 只按集群平均看没用，必须能下钻到 job、node、GPU。

loss 曲线要按 step 对齐，而不是只按时间。

## 20.9 事件 Events

事件记录任务生命周期关键变化。

常见事件：

1. JobSubmitted。
2. JobValidated。
3. JobQueued。
4. JobScheduled。
5. ImagePullStarted。
6. ContainerStarted。
7. DistributedInitStarted。
8. TrainingStarted。
9. CheckpointSaved。
10. EvalStarted。
11. WorkerFailed。
12. RetryStarted。
13. JobCompleted。
14. JobFailed。
15. JobCancelled。

事件应该是结构化的。

示例：

```json
{
  "event_type": "CheckpointSaved",
  "job_id": "job_123",
  "attempt_id": "attempt_2",
  "step": 12000,
  "checkpoint_uri": "s3://ckpts/job_123/step_12000",
  "duration_ms": 83000,
  "timestamp": "2026-05-30T12:00:00Z"
}
```

事件比日志更适合做状态追踪和自动化处理。

## 20.10 训练状态机

训练平台应该有状态机。

常见状态：

```text
submitted -> validating -> queued -> scheduled -> starting -> initializing -> running
                                                                  -> checkpointing
                                                                  -> evaluating
                                                                  -> retrying
                                                                  -> completed
                                                                  -> failed
                                                                  -> cancelled
```

状态机价值：

1. 用户知道任务在哪个阶段。
2. 平台知道是否可以重试。
3. 调度系统知道资源是否释放。
4. 运维可以定位卡点。
5. 审计可以复盘任务过程。

没有状态机，训练平台只是日志堆积。

## 20.11 Attempt 和重试追踪

一个 job 可能有多个 attempt。

例如第一次运行节点故障，第二次从 checkpoint 恢复。

需要区分：

1. job_id：逻辑任务。
2. attempt_id：某次运行尝试。
3. run_id：具体执行实例。

为什么重要？

因为指标和日志需要知道属于哪次尝试。

例如 loss 曲线应该能标注：step 8000 后任务失败，attempt 2 从 checkpoint 恢复。

## 20.12 Trace 设计

训练 trace 可以串起整个任务生命周期。

一个 trace 包含：

1. submit span。
2. validate span。
3. queue span。
4. schedule span。
5. image pull span。
6. startup span。
7. distributed init span。
8. training span。
9. checkpoint span。
10. eval span。
11. artifact registration span。

Trace 可以回答：

1. 任务为什么启动慢？
2. 排队花了多久？
3. 镜像拉取花了多久？
4. 分布式初始化卡了多久？
5. checkpoint 保存是否拖慢训练？

训练 trace 不一定要像微服务 trace 那样记录每个函数调用，但要记录关键阶段耗时。

## 20.13 Dashboard 设计

训练 dashboard 应该分层。

任务概览：

1. 当前状态。
2. 运行时长。
3. 队列等待时间。
4. 当前 step。
5. 最近 checkpoint。
6. 当前 loss。
7. GPU 数量。
8. 成本估计。

训练曲线：

1. loss。
2. learning rate。
3. tokens/s。
4. step time。
5. eval metrics。

资源面板：

1. GPU utilization。
2. 显存。
3. CPU。
4. 网络。
5. 存储。

诊断面板：

1. rank 差异。
2. 通信时间。
3. dataloader time。
4. checkpoint time。
5. 错误事件。

如果一个 dashboard 只有 loss 曲线，它不是训练平台 dashboard。

## 20.14 告警设计

训练平台需要告警。

常见告警：

1. loss NaN。
2. loss 长时间不下降。
3. GPU utilization 长时间过低。
4. step time 突然上升。
5. dataloader time 过高。
6. communication time 过高。
7. checkpoint 长时间未成功。
8. worker failed。
9. retry count 过多。
10. 存储空间不足。
11. 成本超过预算。

告警要避免过度打扰。

可以按严重程度分级：

1. info。
2. warning。
3. critical。

例如 loss NaN 是 critical，GPU 利用率略低可能只是 warning。

## 20.15 异常检测

平台可以做一些自动异常检测。

例如：

1. loss spike。
2. step time spike。
3. tokens/s 下降。
4. GPU utilization 异常低。
5. 某个 rank 明显慢。
6. checkpoint 保存时间异常长。
7. 数据读取吞吐下降。

异常检测可以基于：

1. 固定阈值。
2. 同任务历史。
3. 同类任务基线。
4. 滑动窗口统计。
5. 简单规则模型。

不需要一开始就复杂，很多问题用规则就能发现。

## 20.16 日志、指标和实验追踪的关系

实验追踪关注实验结果和配置。

可观测性关注运行过程。

二者要打通。

例如实验追踪中应该能跳转到：

1. 训练日志。
2. loss 曲线。
3. GPU 利用率。
4. checkpoint 事件。
5. 失败原因。
6. 评估报告。

这样分析模型效果时，不只看最终指标，还能看到训练过程是否健康。

## 20.17 数据留存和成本

日志和指标也有成本。

需要设计留存策略：

1. 高频指标保留多久？
2. 聚合指标保留多久？
3. 失败任务日志保留多久？
4. 成功任务日志保留多久？
5. 高敏日志是否更短留存？
6. 是否归档到低成本存储？

常见策略：

1. 原始高频指标保留 7 到 30 天。
2. 降采样指标保留更久。
3. 关键事件长期保留。
4. 发布模型关联日志长期保留。
5. 临时实验日志定期清理。

可观测性也需要生命周期管理。

## 20.18 面试中如何回答训练可观测性

如果面试官问：

```text
如何设计大模型训练平台的日志、指标和状态追踪？
```

可以这样回答：

```text
我会把训练可观测性分成 logs、metrics、events 和 traces。

Logs 用于排查具体错误，需要按 job、attempt、node、pod、rank、component 结构化采集，并支持 rank 维度查询和日志脱敏。

Metrics 分为模型指标、性能指标、资源指标和平台指标。模型指标包括 loss、eval score；性能指标包括 tokens/s、step time、MFU、communication time、dataloader time；资源指标包括 GPU utilization、显存、网络和存储；平台指标包括排队时间、启动时间、checkpoint 保存时间和重试次数。

Events 记录任务生命周期，比如 submitted、validated、queued、scheduled、started、checkpoint_saved、worker_failed、retry_started、completed。Trace 串起提交、校验、排队、调度、启动、初始化、训练、checkpoint 和 artifact 注册等阶段耗时。

平台需要训练状态机和 attempt 概念，区分逻辑 job 和多次运行尝试。Dashboard 要展示任务状态、训练曲线、资源使用、rank 差异、通信、数据加载和 checkpoint。告警覆盖 loss NaN、GPU 低利用、step time 上升、checkpoint 长时间失败和 worker failure。最后还要做日志权限、脱敏和留存成本管理。
```

## 20.19 常见误区

误区一：可观测性就是收集 stdout。

stdout 只是日志的一部分，还需要指标、事件、trace 和状态机。

误区二：只看 rank 0 日志就够。

分布式训练中错误可能发生在任意 rank。

误区三：有 loss 曲线就能判断训练健康。

还要看吞吐、step time、GPU、通信、数据加载和 checkpoint。

误区四：日志越多越好。

日志太多会增加成本和噪声，还可能泄露敏感信息。

误区五：成功任务不需要保留事件。

事件是审计、复现和性能分析的重要依据。

## 20.20 面试题

### 题 1：logs、metrics、events、traces 有什么区别？

答：logs 记录具体细节，metrics 是可聚合的数值时间序列，events 记录任务生命周期状态变化，traces 串起跨组件和阶段的因果链路。训练平台四者都需要。

### 题 2：训练平台应该采集哪些关键指标？

答：包括模型指标 loss、eval score；性能指标 tokens/s、step time、MFU、communication time、dataloader time；资源指标 GPU utilization、显存、CPU、网络、存储；平台指标 queue time、startup time、checkpoint time、retry count 和 cost。

### 题 3：为什么需要 attempt_id？

答：同一个 job 可能失败后重试或从 checkpoint 恢复。attempt_id 区分不同运行尝试，帮助关联日志、指标、事件和恢复过程，避免把多次运行混在一起。

### 题 4：训练 dashboard 应该包含什么？

答：应包含任务状态、运行时间、当前 step、最近 checkpoint、loss 和 learning rate 曲线、tokens/s、step time、GPU 和显存、rank 差异、通信时间、dataloader time、checkpoint time、错误事件和成本估计。

### 题 5：为什么日志需要权限和脱敏？

答：训练日志可能包含样本内容、用户输入、模型输出、路径、密钥片段和业务字段。如果不控制权限和脱敏，可能造成数据泄露。

## 20.22 本章小结

本章讲了日志、指标、事件和训练状态追踪。

你需要掌握：

1. 训练可观测性要回答任务在哪、是否健康、为什么慢、为什么失败、是否可恢复和成本花在哪里。
2. Logs、metrics、events、traces 各自解决不同问题。
3. 分布式训练日志要支持 rank、node、pod、attempt 维度查询。
4. 指标应覆盖模型、性能、资源和平台四类。
5. 事件和状态机用于追踪任务生命周期。
6. Attempt 用于区分同一 job 的多次运行尝试。
7. Trace 可以分析提交、排队、调度、启动、训练、checkpoint 等阶段耗时。
8. Dashboard 要同时展示训练曲线、资源、诊断和成本。
9. 告警和异常检测能减少 GPU 空烧和训练事故。
10. 日志、指标和事件也需要权限、脱敏和留存策略。
