# 第 9 章 存储体系：本地盘、共享文件系统、对象存储和数据湖

上一章讲了网络。网络决定多机 GPU 之间能不能高效同步，存储则决定数据、checkpoint、模型权重和各种 artifact 能不能稳定、高效、可追踪地被读写。

很多训练慢、训练抖动、checkpoint 保存失败、模型加载慢的问题，本质都是存储问题。

先记住一句话：

> AI Infra 的存储系统不只是放文件，而是支撑训练数据供给、checkpoint 恢复、模型版本管理、评估结果留存和多租户数据治理的基础设施。

## 9.0 本讲资料边界与第二轮精修口径

本讲第二轮精修时，资料口径按“AI 训练与推理存储体系的稳定抽象”处理，而不是按某个云产品、某个共享文件系统、某个对象存储 API 或某个数据湖表格式写死。高性能训练存储部分参考 FSx for Lustre / Lustre 类共享并行文件系统对高性能计算和机器学习 workload 的边界；对象存储部分参考 Amazon S3 对 bucket / object、容量、持久性、一致性和访问控制的通用抽象；GPU 数据路径部分参考 NVIDIA GPUDirect Storage 对 storage 和 GPU memory 直接 DMA、减少 CPU bounce buffer 的边界；数据格式和数据湖部分参考 Apache Parquet 的列式格式、Apache Iceberg 的 metadata / snapshot / manifest 这类表格式治理口径。

需要注意三点：

1. 本章只讲存储体系分层、访问模式、checkpoint、artifact、数据湖和治理，不把下一章专门的 checkpoint 生命周期细节提前展开。
2. 本地 NVMe、共享文件系统、对象存储和数据湖不是互相替代关系，而是分别承担热缓存、POSIX 易用性、权威源 / 归档和数据治理。
3. 存储瓶颈不能只看容量和平均吞吐，还要看 metadata、IOPS、小文件、缓存命中、恢复时间、一致性、权限、生命周期和成本。

## 9.1 大模型为什么依赖存储

大模型系统里，存储承载很多类型的数据。

包括：

1. 原始训练数据。
2. 清洗后的训练数据。
3. 数据索引和 metadata。
4. tokenizer 产物。
5. checkpoint。
6. 模型权重。
7. adapter / LoRA。
8. 量化模型。
9. eval report。
10. 训练日志。
11. 推理日志。
12. RAG 文档和向量索引。
13. Agent trace。

这些数据的访问模式不同。

例如：

1. 训练数据需要高吞吐顺序或随机读取。
2. checkpoint 需要大文件高吞吐写入和恢复读取。
3. 模型权重需要部署时快速加载。
4. 日志需要持续写入和查询。
5. RAG 文档需要更新、索引和权限过滤。

所以 AI 存储体系不能只用一种存储解决所有问题。

## 9.2 存储指标怎么看

常见指标包括：

1. 容量：能存多少数据。
2. 吞吐：单位时间能读写多少数据。
3. IOPS：每秒能处理多少 I/O 操作。
4. 延迟：一次读写要多久。
5. 并发：能支持多少任务同时读写。
6. 可用性：存储服务是否稳定。
7. 持久性：数据丢失概率。
8. 一致性：写入后读取能否看到最新数据。
9. 成本：每 TB 存储和访问成本。
10. 权限：谁能读写哪些数据。

不同任务关注指标不同。

训练数据读取更关注吞吐和并发。

小文件场景更关注 IOPS 和 metadata 性能。

checkpoint 更关注大文件写入吞吐、恢复速度和可靠性。

对象归档更关注容量和成本。

## 9.3 本地盘

本地盘通常指机器本地的 SSD / NVMe 存储。

优点：

1. 读写速度快。
2. 延迟低。
3. 不依赖远端存储网络。
4. 适合做缓存。
5. 可以缓解共享存储压力。

缺点：

1. 容量有限。
2. 节点故障后数据可能不可用。
3. 数据分布在不同机器上，管理复杂。
4. 不适合作为唯一长期存储。

适合场景：

1. 训练数据本地缓存。
2. 临时数据。
3. 解压后的数据 shard。
4. 中间结果。
5. 热点模型权重缓存。

本地盘常见用法是：远端对象存储或数据湖保存源数据，本地 NVMe 保存训练热数据。

## 9.4 共享文件系统

共享文件系统允许多台机器像访问本地目录一样访问同一套文件。

常见用途：

1. 共享代码。
2. 共享数据集。
3. 保存 checkpoint。
4. 保存日志和中间结果。
5. 多节点访问同一个路径。

优点：

1. 使用简单。
2. POSIX 语义友好。
3. 多节点共享方便。
4. 改造现有训练脚本成本低。

缺点：

1. 高并发下 metadata 压力大。
2. 大规模训练可能打爆吞吐。
3. 小文件过多性能差。
4. 故障影响范围大。
5. 成本和运维复杂度可能高。

共享文件系统适合易用性要求高的场景，但大规模预训练不能盲目依赖单一共享目录。

## 9.5 对象存储

对象存储以 object 为单位存储数据，常见形式是 bucket + key。

它不完全等同于文件系统。

优点：

1. 容量大。
2. 成本相对低。
3. 持久性强。
4. 易于跨服务访问。
5. 适合长期保存数据集、checkpoint 和 artifact。

缺点：

1. 延迟通常高于本地盘。
2. POSIX 语义不完整。
3. 小文件访问效率可能低。
4. 直接用于高频训练读取可能不理想。
5. 列目录和 metadata 操作可能昂贵或慢。

适合场景：

1. 原始数据保存。
2. 训练数据归档。
3. checkpoint 长期存储。
4. 模型权重仓库后端。
5. eval report 和 artifact 保存。

常见架构是：对象存储做权威源，本地缓存或分布式缓存做训练加速。

## 9.6 数据湖

数据湖是企业统一存储大规模结构化、半结构化和非结构化数据的平台。

它通常包含：

1. 原始数据层。
2. 清洗数据层。
3. 特征或训练数据层。
4. metadata。
5. 数据血缘。
6. 数据质量。
7. 权限和合规。

对于大模型，数据湖可以管理：

1. 网页文本。
2. 代码数据。
3. 文档数据。
4. 多模态数据。
5. 对话数据。
6. 评估数据。
7. RAG 知识库数据。

数据湖的价值不只是存储容量，而是数据治理。

它回答：

1. 这批数据从哪里来？
2. 是否有权限使用？
3. 是否包含敏感信息？
4. 清洗规则是什么？
5. 训练使用的是哪个版本？
6. 数据质量是否变化？

## 9.7 数据格式对训练性能的影响

训练数据不只是“存在那里”，格式也很重要。

常见格式包括：

1. JSON / JSONL。
2. Parquet。
3. Arrow。
4. WebDataset / tar shards。
5. TFRecord。
6. 自定义二进制格式。

JSONL 易读、调试方便，但解析成本高，体积可能较大。

Parquet 适合列式分析和数据工程，但训练读取要看访问模式。

WebDataset 或 shard 格式适合大规模顺序读取，减少小文件问题。

自定义二进制格式可能性能好，但可维护性和工具生态要评估。

训练系统常见优化是把原始数据离线预处理成训练友好的 shard，减少训练时动态解析。

## 9.8 小文件问题

小文件是训练存储中的常见坑。

如果一个数据集有几千万个小文件，会导致：

1. metadata 查询压力巨大。
2. 打开文件开销高。
3. 共享文件系统压力大。
4. 对象存储请求成本高。
5. dataloader 等待 I/O。

解决方法：

1. 合并成 shard。
2. 使用 tar / WebDataset。
3. 使用 Parquet / Arrow。
4. 建立索引文件。
5. 使用本地缓存。
6. 预取和顺序读取。

面试中如果提到训练数据供给，一定要想到小文件问题。

## 9.9 训练数据供给链路

一个典型训练数据供给链路：

```text
Raw Data
  -> Data Lake / Object Storage
  -> Cleaning / Dedup / Filtering
  -> Tokenization / Packing
  -> Training Shards
  -> Local Cache / Distributed Cache
  -> Dataloader
  -> GPU
```

每一层都可能成为瓶颈。

例如：

1. 清洗太慢，数据版本延迟。
2. tokenization 在线执行，CPU 成为瓶颈。
3. shard 太小，小文件问题严重。
4. 缓存命中率低，远端存储压力大。
5. dataloader worker 不够。
6. CPU 到 GPU 拷贝慢。

成熟平台会尽量把重预处理放到离线阶段，让训练阶段只做高效读取。

## 9.10 Checkpoint 存储需求

Checkpoint 是训练恢复的关键。

它通常包含：

1. 模型参数。
2. 优化器状态。
3. scheduler 状态。
4. random seed。
5. dataloader 状态。
6. 分布式 rank 信息。
7. 训练 step。
8. 配置和 metadata。

大模型 checkpoint 可能非常大。

存储系统需要支持：

1. 高吞吐写入。
2. 分片保存。
3. 快速恢复读取。
4. 元数据一致性。
5. 保留策略。
6. 权限控制。
7. 跨区域或跨集群复制。

checkpoint 不是随便把文件写出去就完了，后续恢复是否可靠同样重要。

## 9.11 模型权重存储和加载

推理平台上线模型时，需要加载模型权重。

如果模型很大，加载时间可能成为发布和扩容瓶颈。

常见问题：

1. 模型权重文件太大。
2. 多个实例同时拉取，打爆对象存储。
3. 冷启动慢。
4. 本地缓存失效。
5. 权重版本混乱。
6. tokenizer、config 和权重不匹配。

优化方法：

1. 节点本地缓存模型权重。
2. 分层缓存。
3. 预热模型。
4. 分片加载。
5. 模型仓库统一管理版本。
6. 发布前做完整性校验。

模型加载是推理平台 SLO 的一部分，不能只看请求处理时延。

## 9.12 Artifact 存储

大模型系统有很多 artifact。

包括：

1. dataset version。
2. training config。
3. checkpoint。
4. final model。
5. tokenizer。
6. LoRA / adapter。
7. quantized model。
8. eval report。
9. deployment package。
10. trace sample。

Artifact 存储要解决：

1. 版本管理。
2. 血缘关系。
3. 权限控制。
4. 生命周期管理。
5. 可复现性。
6. 审计。

如果 artifact 管理混乱，团队很快会不知道哪个模型来自哪个数据和配置。

## 9.13 存储一致性和元数据

存储系统不只是数据块，还包括 metadata。

metadata 包括：

1. 文件名。
2. 大小。
3. 修改时间。
4. 版本。
5. checksum。
6. owner。
7. 权限。
8. 数据血缘。

训练和部署非常依赖 metadata 正确性。

例如：

1. checkpoint 文件写了一半但 metadata 标记完成，会导致恢复失败。
2. 模型权重版本和 config 不匹配，会导致推理错误。
3. 数据集版本指向错了，会导致实验不可复现。

所以大文件写入通常需要 commit 语义：先写临时文件，校验完成后再标记为可用。

## 9.14 存储权限和合规

大模型数据可能包含敏感信息、版权内容、业务机密或用户数据。

存储系统需要支持：

1. 身份认证。
2. 访问控制。
3. 数据分类。
4. 加密存储。
5. 访问日志。
6. 数据留存策略。
7. 删除和过期策略。
8. 跨租户隔离。

AI Infra 中，数据权限不是训练脚本自己解决的，而应该由平台和存储治理统一支持。

## 9.15 常见存储瓶颈

### 9.15.1 数据读取慢

可能原因：

1. 远端对象存储延迟高。
2. 小文件太多。
3. 数据没有 shard。
4. 缓存命中率低。
5. dataloader worker 不够。
6. CPU 解码或 tokenization 太慢。
7. 共享文件系统 metadata 压力大。

### 9.15.2 Checkpoint 保存慢

可能原因：

1. checkpoint 太大。
2. 所有 rank 同时写。
3. 存储写入吞吐不足。
4. metadata 操作多。
5. 网络到存储路径拥塞。
6. 没有异步保存。

### 9.15.3 模型加载慢

可能原因：

1. 权重文件太大。
2. 冷启动无缓存。
3. 多副本同时下载。
4. 对象存储限流。
5. 权重分片不合理。
6. 校验和转换耗时。

### 9.15.4 存储成本失控

可能原因：

1. checkpoint 保留过多。
2. 重复数据集。
3. 中间 artifact 无生命周期管理。
4. 日志无限增长。
5. 冷数据没有降级存储。

## 9.16 存储优化策略

常见优化：

1. 数据预处理离线化。
2. 小文件合并成 shard。
3. 本地 NVMe 缓存。
4. 分布式缓存。
5. 顺序读取和预取。
6. 提高 dataloader 并发。
7. 使用异步 checkpoint。
8. checkpoint 分片保存。
9. 模型权重预热。
10. 设置 artifact 生命周期策略。
11. 对冷数据做归档降级。
12. 使用 checksum 和 commit 标记保证完整性。

优化时要注意：缓存能提升速度，但会带来一致性和容量管理问题。

## 9.17 存储体系审计指标与最小 demo

前面讲的是存储组件和常见瓶颈。真实平台设计里，还要把“该用哪种存储、够不够快、能不能恢复、成本是否可控”变成可检查指标。

可以把第 `i` 个存储工作负载样本写成：

```math
s_i=(u_i,c_i,q_i,m_i,h_i,k_i,r_i,a_i,g_i,l_i,z_i)
```

其中，`u_i` 是用途，例如 training data、checkpoint、model weights、artifact 或 RAG documents；`c_i` 是容量需求，`q_i` 是吞吐和 IOPS，`m_i` 是 metadata 压力，`h_i` 是 cache / hot data，`k_i` 是 checkpoint 或恢复需求，`r_i` 是可靠性和一致性，`a_i` 是 artifact lineage，`g_i` 是 governance，`l_i` 是 lifecycle / cost，`z_i` 是风险。

有效读写吞吐可以写成：

```math
B_{\mathrm{eff}}=\frac{D}{T}
```

其中，`D` 是实际读写数据量，`T` 是端到端耗时。它比纸面带宽更有用，因为包括协议、metadata、并发、网络和客户端实现的影响。

训练 I/O 占比可以写成：

```math
R_{\mathrm{io}}=\frac{T_{\mathrm{dataloader}}+T_{\mathrm{storage}}+T_{\mathrm{h2d}}}{T_{\mathrm{step}}}
```

如果 `R_io` 很高，GPU 可能不是算不动，而是在等数据。

小文件 metadata 放大可以写成：

```math
A_{\mathrm{meta}}=\frac{N_{\mathrm{file}}}{D_{\mathrm{TiB}}}
```

其中，`N_file` 是文件数，`D_TiB` 是数据集 TiB 数。文件数越多，metadata、open、stat、list 和对象请求开销越容易压垮共享文件系统或对象存储。

缓存命中率可以写成：

```math
H_{\mathrm{cache}}=\frac{D_{\mathrm{cache}}}{D_{\mathrm{total}}}
```

其中，`D_cache` 是从本地或分布式缓存命中的字节数，`D_total` 是总读取字节数。缓存命中率低时，远端对象存储或共享文件系统压力会上升。

Checkpoint 保存和恢复时间可以粗略写成：

```math
T_{\mathrm{ckpt}}=\frac{S_{\mathrm{ckpt}}}{\eta B_{\mathrm{write}}}+T_{\mathrm{meta}}+T_{\mathrm{commit}}
```

```math
T_{\mathrm{restore}}=\frac{S_{\mathrm{ckpt}}}{\eta B_{\mathrm{read}}}+T_{\mathrm{validate}}+T_{\mathrm{requeue}}
```

其中，`S_ckpt` 是 checkpoint 大小，`B_write` 和 `B_read` 是写入和读取带宽，`\eta` 是有效带宽系数，`T_meta` 是 metadata 开销，`T_commit` 是 checksum / commit 标记开销，`T_validate` 是恢复校验开销，`T_requeue` 是重新排队或重启开销。

存储成本可以拆成：

```math
K_{\mathrm{storage}}=K_{\mathrm{capacity}}+K_{\mathrm{request}}+K_{\mathrm{egress}}+K_{\mathrm{metadata}}+K_{\mathrm{ops}}
```

最后，可以把存储体系门禁写成：

```math
G_{\mathrm{storage}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{io}}\le \rho_{\mathrm{io}} \land T_{\mathrm{restore}}\le \tau_{\mathrm{restore}} \land P_0=0\right]
```

其中，`C_j` 是第 `j` 个存储审计指标覆盖率，`R_io` 是训练 I/O 占比，`T_restore` 是恢复耗时，`P_0` 是 P0 级风险数量。

下面这个 0 依赖 demo 演示如何把训练数据、checkpoint、模型权重和 artifact 存储写成审计规则。它故意构造 1 个完整样本和 16 个坏样本，让每个关键维度各失败一次。

```python
import copy


METRICS = [
    "capacity_tier_fit",
    "throughput_iops_fit",
    "local_nvme_cache_fit",
    "shared_fs_metadata_fit",
    "object_store_authority_fit",
    "data_lake_governance_fit",
    "training_shard_format_fit",
    "small_file_amplification_control",
    "dataloader_cache_hit_tracking",
    "checkpoint_write_recovery_fit",
    "model_weight_load_cache_fit",
    "artifact_lineage_metadata_fit",
    "consistency_commit_integrity",
    "security_compliance_fit",
    "lifecycle_cost_governance",
    "storage_system_gate",
]


def throughput_gib_s(gib, seconds):
    return gib / seconds


def transfer_time_s(gib, bandwidth_gib_s, efficiency):
    return gib / (bandwidth_gib_s * efficiency)


def checkpoint_time_s(size_gib, bandwidth_gib_s, efficiency, metadata_s, commit_s):
    return transfer_time_s(size_gib, bandwidth_gib_s, efficiency) + metadata_s + commit_s


def restore_time_s(size_gib, bandwidth_gib_s, efficiency, validate_s, requeue_s):
    return transfer_time_s(size_gib, bandwidth_gib_s, efficiency) + validate_s + requeue_s


def metadata_amplification(files, dataset_tib):
    return files / dataset_tib


def cache_hit_rate(cache_gib, total_gib):
    return cache_gib / total_gib


def monthly_storage_cost_usd(hot_tib, cold_tib, hot_per_tib, cold_per_tib, request_cost):
    return hot_tib * hot_per_tib + cold_tib * cold_per_tib + request_cost


def build_storage_cases():
    complete = {
        "name": "complete",
        "capacity": {"hot_tib": 500, "cold_tib": 1500, "growth_tib_per_month": 120, "tiered": True},
        "performance": {"read_gib_s": 120, "write_gib_s": 80, "iops": 800_000, "measured": True},
        "local_cache": {"nvme_tib": 64, "hit_rate": 0.88, "eviction_policy": "lru"},
        "shared_fs": {"metadata_ops": 40_000, "small_file_pressure": "controlled", "quota": True},
        "object_store": {"authoritative": True, "versioning": True, "direct_training_read": False},
        "data_lake": {"lineage": True, "quality_checks": True, "permission_tags": True},
        "format": {"sharded": True, "format": "webdataset_or_parquet", "avg_shard_gib": 2.0},
        "small_files": {"file_count": 50_000, "dataset_tib": 100, "sharded": True},
        "dataloader": {"cache_hit_tracked": True, "dataloader_wait_s": 0.25, "step_time_s": 4.0},

    def bad_case(name, mutator):

    bad_cases = [

def check_capacity(case):
    c = case["capacity"]
    return c["tiered"] and c["hot_tib"] > 0 and c["cold_tib"] > 0 and c["growth_tib_per_month"] > 0


def check_performance(case):
    p = case["performance"]
    return p["measured"] and p["read_gib_s"] >= 50 and p["write_gib_s"] >= 40 and p["iops"] >= 100_000


def check_local_cache(case):
    c = case["local_cache"]
    return c["nvme_tib"] > 0 and c["hit_rate"] >= 0.7 and bool(c["eviction_policy"])


def check_shared_fs(case):
    s = case["shared_fs"]
    return s["metadata_ops"] > 0 and s["small_file_pressure"] == "controlled" and s["quota"]


def check_object_store(case):
    o = case["object_store"]
    return o["authoritative"] and o["versioning"] and not o["direct_training_read"]


def check_data_lake(case):
    d = case["data_lake"]
    return d["lineage"] and d["quality_checks"] and d["permission_tags"]


def check_format(case):
    f = case["format"]
    return f["sharded"] and f["format"] in {"webdataset_or_parquet", "arrow"} and f["avg_shard_gib"] >= 0.5


def check_small_files(case):
    s = case["small_files"]
    return s["sharded"] and metadata_amplification(s["file_count"], s["dataset_tib"]) <= 10_000


def check_dataloader(case):
    d = case["dataloader"]
    return d["cache_hit_tracked"] and d["dataloader_wait_s"] / d["step_time_s"] <= 0.15


def check_checkpoint(case):
    c = case["checkpoint"]
    write_s = checkpoint_time_s(c["size_gib"], c["write_gib_s"], 0.75, 20, 10)
    restore_s = restore_time_s(c["size_gib"], c["restore_gib_s"], 0.75, 20, 10)
    return c["restore_tested"] and write_s <= 120 and restore_s <= 120


def check_model_load(case):
    m = case["model_load"]
    return m["local_weight_cache"] and m["warmup"] and m["max_load_s"] <= 300


def check_artifact(case):
    a = case["artifact"]
    return all(a[k] for k in ["dataset_version", "config", "model", "eval_report"])


def check_commit(case):
    c = case["commit"]
    return c["checksum"] and c["atomic_commit"] and c["manifest"]


def check_security(case):
    s = case["security"]
    return s["iam"] and s["encryption"] and s["audit_log"] and s["tenant_isolation"]


def check_lifecycle(case):
    l = case["lifecycle"]
    cost = monthly_storage_cost_usd(500, 1500, 23, 4, 500)
    return l["retention_policy"] and l["archive_cold_data"] and cost <= l["monthly_budget_usd"]


def check_gate(case):
    return case["gate"]["enabled"]

def audit_storage(cases):
    case_failures = {}
    for case in cases:
        failures = [name for name, check in CHECKS.items() if not check(case)]
        case_failures[case["name"]] = failures

    metrics = {}
    for name, check in CHECKS.items():
        metrics[name] = round(sum(int(check(case)) for case in cases) / len(cases), 3)

    failed_cases = [name for name, failures in case_failures.items() if failures]
    return {
        "metrics": metrics,
        "hard_blocker_count": len(failed_cases),
        "failed_cases": failed_cases,
        "storage_gate_pass": not failed_cases and min(metrics.values()) >= 0.95,
```

```text
对象存储或数据湖作为权威源，保存原始数据、清洗数据、checkpoint、模型权重和 artifact，提供容量、持久性、权限和数据治理。
```

## 9.19 常见误区

误区一：存储只是把文件放起来。

AI 存储还要支持高吞吐读取、checkpoint 恢复、模型版本、artifact 血缘、权限审计和成本治理。

误区二：共享文件系统最方便，所以所有任务都用它。

共享文件系统易用，但在大规模小文件和高并发场景可能成为瓶颈。

误区三：对象存储容量大，所以训练直接从对象存储读就行。

对象存储适合权威源和归档，但直接高频训练读取可能有延迟和请求开销，需要缓存或 shard 优化。

误区四：checkpoint 保存成功就没问题。

还要验证能否恢复，metadata 是否一致，版本是否完整，保留策略是否合理。

误区五：存储成本不重要。

大模型 checkpoint、数据集、日志和中间 artifact 很容易快速膨胀，没有生命周期管理会造成巨大浪费。

## 9.20 面试题

### 题 1：本地盘、共享文件系统、对象存储分别适合什么场景？

答：本地盘速度快、延迟低，适合热点数据缓存和临时数据；共享文件系统使用方便，适合多节点共享代码、数据和中间结果，但高并发和小文件场景可能瓶颈明显；对象存储容量大、持久性强、成本相对低，适合原始数据、checkpoint、模型权重和 artifact 的长期保存。

### 题 2：为什么小文件会拖慢训练？

答：小文件会带来大量 metadata 查询、打开文件和对象请求开销，导致共享文件系统或对象存储压力很大。dataloader 会等待 I/O，GPU 利用率下降。解决方法包括合并 shard、使用 WebDataset / Parquet / Arrow、建立索引和本地缓存。

### 题 3：训练数据供给链路有哪些瓶颈？

答：可能瓶颈包括数据清洗慢、在线 tokenization 慢、shard 设计不合理、小文件过多、远端存储延迟高、缓存命中率低、dataloader worker 不够、CPU 到 GPU 拷贝慢等。

### 题 4：checkpoint 存储要注意什么？

答：要关注保存吞吐、恢复速度、分片策略、异步保存、metadata 一致性、checksum 校验、保留策略、权限控制和恢复测试。checkpoint 不是只要写成功，还必须能可靠恢复。

### 题 5：模型加载慢怎么优化？

答：可以做本地缓存、分层缓存、模型预热、分片加载、对象存储限流保护、发布前完整性校验、避免多副本同时冷启动，并通过模型仓库统一管理权重、tokenizer 和 config 版本。

## 9.22 本章小结

本章讲了 AI Infra 中的存储体系。

你需要掌握：

1. AI 存储承载训练数据、checkpoint、模型权重、artifact、日志、RAG 文档和 trace。
2. 本地盘适合低延迟缓存，共享文件系统适合多节点共享，对象存储适合大容量长期保存，数据湖适合数据治理。
3. 存储指标包括容量、吞吐、IOPS、延迟、并发、可用性、持久性、一致性、成本和权限。
4. 数据格式、小文件、shard 和缓存会显著影响训练吞吐。
5. checkpoint 需要关注保存、加载、metadata、checksum、恢复测试和生命周期管理。
6. 模型权重加载会影响推理冷启动和扩容速度。
7. Artifact 管理要支持版本、血缘、权限、生命周期和可复现性。
8. 存储优化要结合本地缓存、分布式缓存、异步写入、预处理、生命周期策略和数据治理。
