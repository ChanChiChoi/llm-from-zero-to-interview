# 第 30 章 模型路由：模型选择、能力路由、成本路由和降级路由

上一章讲了 runtime 内部如何通过 batching、PagedAttention 和队列调度提升吞吐。本章把视角上移到平台层，讲模型路由。

在单模型服务里，请求只有一个去处。但在真实大模型平台里，通常同时存在多个模型、多个版本、多个 endpoint、多个租户和多种业务目标。此时，请求到底应该发给谁，就变成了核心问题。

先记住一句话：

> 模型路由不是简单负载均衡，而是在质量、能力、延迟、成本、稳定性和安全策略之间做选择。

## 30.0 本讲资料边界与第二轮精修口径

本讲按 `WRITING_PLAN.md` 的第二轮要求做过资料校准。重点参考的是 OpenAI 对模型能力、成本、延迟优化和生产检查项的公开文档，LiteLLM Router 对模型组、负载均衡、fallback、budget 和重试的工程抽象，OpenTelemetry 对 trace / span / attributes 的可观测性口径，以及 Google SRE 关于过载保护、配额、重试预算和 graceful degradation 的系统设计原则。

这些资料共同指向一个稳定事实：模型路由不是“把请求随机分给某个 endpoint”，也不是“永远使用最强模型”。生产路由要先做权限和能力过滤，再把质量、延迟 SLO、成本预算、实时健康度、灰度策略、fallback chain、降级策略和 route trace 放进同一张策略表。

本章只抽象截至 2026-06 仍稳定的多模型平台路由口径，不把某个闭源模型 ID、价格、上下文长度、SDK 字段名或第三方框架默认参数写成通用标准。正文中的公式用于面试表达、容量估算和策略审计；真实上线仍要用目标模型、真实 token 分布、在线评估、业务 SLO、成本账单和事故复盘数据校准。

## 30.1 为什么需要模型路由

如果平台只有一个模型、一个版本、一个实例，路由很简单。

```text
Request -> Model Server
```

但生产环境通常是这样：

```text
Request
  -> Router
      -> gpt-like-large-v3
      -> gpt-like-small-v2
      -> code-model-v1
      -> embedding-model-v4
      -> vision-language-model-v2
      -> fallback-model-v1
```

平台需要决定：

1. 用哪个模型？
2. 用哪个版本？
3. 用哪个 endpoint？
4. 是否走灰度版本？
5. 是否降级到小模型？
6. 是否拒绝请求？
7. 是否从缓存返回？

这就是模型路由。

## 30.2 模型路由和负载均衡的区别

传统负载均衡通常在一组等价实例之间选择一个实例。

例如：

```text
service-a: instance-1 / instance-2 / instance-3
```

这些实例能力基本相同。

模型路由面对的是不等价目标：

1. 模型能力不同。
2. 模型成本不同。
3. 模型延迟不同。
4. 上下文长度不同。
5. 安全等级不同。
6. 支持的任务类型不同。
7. 版本质量不同。
8. 当前负载不同。

所以模型路由先做“选模型和版本”，再做“选实例”。

## 30.3 模型路由的输入信息

Router 通常会使用这些信息：

1. 请求指定的模型名。
2. 业务场景。
3. 租户身份。
4. 用户等级。
5. 输入 token 数。
6. 预估输出 token 数。
7. 是否需要流式返回。
8. 是否需要工具调用。
9. 是否需要多模态能力。
10. 延迟 SLO。
11. 成本预算。
12. 安全策略。
13. 灰度规则。
14. endpoint 负载。
15. 健康状态。

路由不是只看一个字段，而是综合决策。

## 30.4 模型路由的输出结果

Router 的输出不应该只是一个 URL。

更完整的结果包括：

```text
RouteDecision
  model
  model_version
  endpoint
  runtime
  route_reason
  fallback_chain
  timeout_budget
  max_tokens
  safety_policy
  cost_policy
  trace_id
```

这样做的好处是：

1. 便于审计。
2. 便于 debug。
3. 便于成本归因。
4. 便于回放问题请求。
5. 便于分析路由策略效果。

线上系统里，“为什么这个请求路由到这个模型”必须可解释。

## 30.5 显式模型选择

最简单的路由方式是用户显式指定模型。

例如：

```json
{
  "model": "large-chat-v3",
  "messages": [...]
}
```

平台根据 `model` 字段找到对应 endpoint。

这种方式简单直接，适合：

1. 开发者 API。
2. 内部平台。
3. 用户明确知道模型差异的场景。
4. 需要稳定复现的实验。

但缺点也明显：

1. 用户要理解模型能力。
2. 成本优化依赖用户自觉。
3. 模型升级需要用户迁移。
4. 难以自动降级。
5. 难以按业务目标做智能选择。

所以多模型平台通常还需要能力路由和策略路由。

## 30.6 能力路由

能力路由是根据任务能力选择模型。

例如：

1. 普通对话 -> 通用 chat 模型。
2. 代码补全 -> code 模型。
3. 文档问答 -> 长上下文模型。
4. 图片理解 -> 多模态模型。
5. 向量检索 -> embedding 模型。
6. 函数调用 -> tool-use 能力强的模型。
7. 复杂推理 -> reasoning 模型。

能力路由的关键是维护模型能力画像。

Model Registry 中需要记录：

1. 支持任务类型。
2. 最大上下文长度。
3. 支持语言。
4. 支持多模态类型。
5. tool calling 能力。
6. 结构化输出能力。
7. 推理能力等级。
8. 安全等级。
9. 评估分数。

没有能力画像，能力路由只能靠硬编码。

## 30.7 成本路由

成本路由是根据成本目标选择模型。

不是所有请求都需要最大模型。

例如：

1. 简单分类用小模型。
2. 简单改写用中模型。
3. 复杂推理用大模型。
4. 高价值用户用更强模型。
5. 低优先级后台任务用便宜模型。

成本路由常见策略：

1. 小模型优先。
2. 失败再升级到大模型。
3. 根据置信度升级。
4. 根据用户等级选择模型。
5. 根据预算选择模型。
6. 根据时间段选择模型。

成本路由不是单纯省钱，而是在满足质量和 SLO 的前提下降低成本。

## 30.8 质量路由

质量路由是根据模型效果选择路径。

平台可以基于：

1. 离线评估结果。
2. 在线反馈。
3. A/B 测试结果。
4. 用户满意度。
5. 任务成功率。
6. 工具调用成功率。
7. 安全拦截率。

来调整路由策略。

例如某个新模型在数学题上更好，但在代码生成上退化，那么可以只让数学任务路由到新模型，而代码任务继续使用旧模型。

成熟平台不会把“新模型”直接替换“旧模型”，而是按任务和指标逐步迁移。

## 30.9 延迟路由

有些请求对延迟敏感，有些请求对质量更敏感。

例如：

1. 在线聊天要求低 TTFT。
2. 搜索改写要求几十毫秒到几百毫秒。
3. 后台报告生成可以等待更久。
4. Agent 长任务可以接受较长执行时间。

延迟路由可以根据 SLO 选择：

1. 更小模型。
2. 更近地域。
3. 更空闲 endpoint。
4. 更短上下文策略。
5. 非流式或流式返回。
6. 是否启用复杂推理。

延迟路由要结合实时负载，否则静态模型选择不够。

## 30.10 降级路由

降级路由用于系统压力过大、模型故障或成本超预算时。

常见降级路径：

1. 大模型 -> 小模型。
2. reasoning 模型 -> 普通 chat 模型。
3. 长上下文 -> 截断上下文。
4. 实时生成 -> 缓存结果。
5. 多模态模型 -> 文本模型。
6. 工具调用流程 -> 简化回答。
7. 高温采样 -> 确定性生成。

降级不能随便做，因为它可能影响质量和用户体验。

平台需要明确：

1. 哪些业务允许降级。
2. 哪些用户允许降级。
3. 降级后如何标记。
4. 降级指标如何统计。
5. 是否需要用户可见提示。

## 30.11 Fallback Chain

Fallback chain 是一组备用路径。

例如：

```text
primary: large-chat-v3
fallback-1: large-chat-v2
fallback-2: medium-chat-v3
fallback-3: cached-response
```

触发 fallback 的原因可能是：

1. endpoint 不健康。
2. 请求超时。
3. 队列过长。
4. 错误率过高。
5. 配额不足。
6. 成本预算耗尽。

Fallback 要避免无限重试。

每次 fallback 都要消耗时间预算和成本预算。

## 30.12 灰度路由

模型发布需要灰度。

灰度路由可以按：

1. 流量比例。
2. 租户。
3. 用户分桶。
4. 地域。
5. 业务场景。
6. prompt 类型。
7. 请求风险等级。

把部分请求导向新模型。

灰度期间要对比：

1. 质量指标。
2. 延迟指标。
3. 成本指标。
4. 错误率。
5. 安全指标。
6. 用户反馈。

灰度路由必须支持快速回滚。

## 30.13 多地域路由

如果平台跨地域部署，还要考虑地域路由。

目标包括：

1. 降低网络延迟。
2. 满足数据合规要求。
3. 提高可用性。
4. 均衡资源。
5. 支持灾备。

但大模型多地域部署成本高，因为模型权重巨大，GPU 资源昂贵，流量分布也不均匀。

多地域路由通常要考虑：

1. 用户所在地域。
2. 数据是否允许跨境或跨区。
3. 模型是否在该地域部署。
4. endpoint 当前负载。
5. 灾备策略。

## 30.14 Router 的实时信号

Router 不应该只看静态配置。

还应该接入实时信号：

1. endpoint 健康状态。
2. 当前 QPS。
3. queue length。
4. TTFT。
5. TPOT。
6. p95 / p99 延迟。
7. 错误率。
8. timeout rate。
9. GPU 显存水位。
10. KV cache 使用率。
11. admission reject rate。

如果某个 endpoint 队列已经很长，router 还继续导流，就会放大故障。

## 30.15 Router 的配置管理

Router 配置通常包括：

1. 模型到 endpoint 的映射。
2. 版本权重。
3. 灰度规则。
4. fallback chain。
5. 租户权限。
6. 成本策略。
7. 限流策略。
8. 降级策略。
9. 地域策略。
10. 安全策略。

这些配置必须：

1. 可版本化。
2. 可审计。
3. 可回滚。
4. 可灰度。
5. 支持 dry-run。

Router 配置错误可能导致大面积线上事故。

## 30.16 Router 的可观测性

模型路由必须记录 route trace。

至少包括：

1. 请求 ID。
2. 租户。
3. 输入 token 数。
4. 选择的模型。
5. 选择的版本。
6. 选择的 endpoint。
7. 路由原因。
8. 是否灰度。
9. 是否降级。
10. 是否 fallback。
11. 路由耗时。
12. 最终延迟和错误码。
13. 成本估算。

否则线上出现质量问题时，你很难回答：为什么这个用户被路由到了这个模型？

## 30.17 Router 的风险

模型路由会引入新的风险：

1. 路由规则过复杂，难以维护。
2. 静态规则和实时负载冲突。
3. 降级导致质量不可控。
4. fallback 造成重复成本。
5. 灰度分桶不稳定。
6. 缓存和路由策略不一致。
7. 多租户权限配置错误。
8. route trace 缺失导致问题无法复盘。

工程上要避免“一堆 if else 写死模型选择”。

更好的做法是将策略配置化、版本化、可观测化。

## 30.18 一个 Router 决策流程

一个完整路由流程可以是：

```text
1. 解析请求和租户身份
2. 检查模型访问权限
3. 识别任务类型和能力需求
4. 估算 input/output token
5. 读取成本和延迟 SLO
6. 筛选候选模型
7. 应用灰度规则
8. 结合实时负载选择 endpoint
9. 生成 fallback chain
10. 写入 route trace
11. 转发请求到 runtime
```

这个流程体现了模型路由的本质：先筛选可用能力，再按策略和实时状态做选择。

## 30.19 模型路由审计指标与最小 demo

第二轮精修时，需要把模型路由从“规则列表”升级成可解释、可回放、可门禁的决策模型。

可以把一次请求写成：

```math
r_i=(q_i,u_i,a_i,n_i,o_i,s_i,b_i,\ell_i,\rho_i,d_i,z_i)
```

其中，`q_i` 是任务类型，`u_i` 是租户或用户等级，`a_i` 是能力需求，`n_i` 是输入 token 数，`o_i` 是预估输出 token 数，`s_i` 是延迟 SLO，`b_i` 是成本预算，`\ell_i` 是实时负载信号，`\rho_i` 是风险等级，`d_i` 是是否允许降级，`z_i` 是 trace 和策略版本字段。

对一组模型候选 `\mathcal{M}`，路由器先做硬过滤：

```math
\mathcal{M}_i=\{m\in\mathcal{M}: A(m,u_i)=1 \land K(m,a_i)=1 \land H(m)=1 \land C(i,m)\le b_i \land T(i,m)\le s_i\}
```

这里 `A(m,u_i)` 表示租户是否有权限访问模型 `m`，`K(m,a_i)` 表示模型能力是否满足请求，`H(m)` 表示 endpoint 健康和负载是否可接受，`C(i,m)` 是预估成本，`T(i,m)` 是预估延迟。硬过滤不通过的模型不能靠打分“补回来”，否则会把权限、预算或 SLO 风险藏进平均分。

成本可以按输入和输出 token 估算：

```math
C(i,m)=\frac{n_i p_{\mathrm{in},m}+o_i p_{\mathrm{out},m}}{1000}
```

其中，`p_{\mathrm{in},m}` 和 `p_{\mathrm{out},m}` 分别是模型 `m` 的每千输入 token 和每千输出 token 单价或内部成本口径。线上成本还可能加入 GPU seconds、cache hit savings、region、priority 和 reserved capacity，但面试表达先把 token 成本讲清楚。

延迟可以拆成：

```math
T(i,m)=T_{\mathrm{route}}+T_{\mathrm{queue},m}+T_{\mathrm{ttft},m}+o_iT_{\mathrm{tpot},m}
```

其中，`T_{\mathrm{route}}` 是网关和路由开销，`T_{\mathrm{queue},m}` 是当前 endpoint 排队开销，`T_{\mathrm{ttft},m}` 是模型首 token 延迟基线，`T_{\mathrm{tpot},m}` 是每个输出 token 的平均生成时间。这个公式的重点不是精确预测每个请求，而是提醒面试时不要只按模型名字选路由。

硬过滤后，可以用 utility score 排序：

```math
U(i,m)=\alpha Q(i,m)-\beta \frac{C(i,m)}{b_i}-\gamma \max\left(0,\frac{T(i,m)}{s_i}-1\right)-\delta R(i,m)
```

其中，`Q(i,m)` 是任务质量或 eval 分数，`R(i,m)` 是安全、版本或业务风险惩罚，`\alpha,\beta,\gamma,\delta` 是策略权重。这个打分只应该在权限、能力、健康、预算和 SLO 已经通过之后使用。

Fallback chain 也要受预算约束：

```math
\sum_{k=1}^{K_i}T(i,m_k)\le s_i,\qquad \sum_{k=1}^{K_i}C(i,m_k)\le b_i
```

这表示一次请求最多尝试 `K_i` 个候选模型，总时间和总成本仍不能无限膨胀。否则 fallback 会把一次失败放大成多次排队、多次计费和更差的尾延迟。

灰度路由要稳定分桶：

```math
h_i=H(u_i,v)\bmod 10000,\qquad G(i,v)=\mathbf{1}[h_i<10000w_v]
```

其中，`H` 是稳定哈希，`v` 是候选版本，`w_v` 是灰度权重。稳定分桶可以避免同一用户在新旧模型之间来回跳，方便对比质量、延迟和安全指标。

Route trace 覆盖率可以写成：

```math
C_{\mathrm{trace}}=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[\mathrm{trace}(i)\supseteq F_{\mathrm{required}}]
```

其中，`F_{\mathrm{required}}` 是必填 trace 字段集合，例如 request id、tenant、task、input/output tokens、selected model、version、route reason、fallback、degrade、cost、latency 和 config version。没有 trace 的路由系统无法复盘质量回归和成本异常。

最终模型路由门禁可以写成：

```math
G_{\mathrm{route}}=\mathbf{1}\left[\min_j C_j\ge\tau_j \land C_{\mathrm{trace}}\ge\tau_{\mathrm{trace}} \land R_{\mathrm{fallback}}\le\rho_{\mathrm{fallback}} \land R_{\mathrm{degrade}}\le\rho_{\mathrm{degrade}} \land P_0=0\right]
```

其中，`C_j` 是各审计维度覆盖率，`R_{\mathrm{fallback}}` 是 fallback 率，`R_{\mathrm{degrade}}` 是降级率，`P_0` 是未关闭的 P0 风险数。这个门禁强调：路由策略要能解释、能回滚、能观测，而不是只在 demo 流量上看起来可用。

def stable_bucket(*parts):
    digest = sha256("|".join(map(str, parts)).encode()).hexdigest()
    return int(digest[:8], 16) % 10000


def estimate_cost(req, model):
    return (
        req["input_tokens"] * model["input_cost"]
        + req["expected_output_tokens"] * model["output_cost"]
    ) / 1000.0


def estimate_latency(req, model):
    route_overhead_ms = 24
    queue_penalty_ms = int(200 * model["load"])
    return route_overhead_ms + queue_penalty_ms + model["ttft_ms"] + req["expected_output_tokens"] * model["tpot_ms"]


def capability_match(req, model):
    context = req["input_tokens"] + req["expected_output_tokens"]
    return (
        req["task"] in model["tasks"]
        and req["modalities"].issubset(model["modalities"])
        and context <= model["max_context"]
        and (not req["needs_tool"] or model["tool_calling"])
    )


def trace_record(req, selected, decision, cost, latency):
    return {
        "request_id": req["id"],
        "tenant": req["tenant"],
        "task": req["task"],
        "input_tokens": req["input_tokens"],
        "expected_output_tokens": req["expected_output_tokens"],
        "selected_model": selected,
        "decision": decision,
        "cost": round(cost, 4),
        "latency_ms": int(latency),
        "config_version": "route-policy-2026-06-12",
        "reason": "highest_utility_after_policy_filter",
    }


def route_request(req):
    candidates = []
    rejected = []
    for name, model in MODELS.items():
        if "canary_weight" in model:
            bucket = stable_bucket(req["tenant"], req["user"], model["version"])
            if bucket >= int(model["canary_weight"] * 10000):
                rejected.append((name, "canary_bucket"))
                continue
        if req["tenant"] not in model["tenants"]:
            rejected.append((name, "permission"))
            continue
        if not capability_match(req, model):
            rejected.append((name, "capability"))
            continue
        if model["health"] < 0.90 or model["load"] > 0.90:
            rejected.append((name, "health_or_load"))
            continue
        cost = estimate_cost(req, model)
        if cost > req["budget"]:
            rejected.append((name, "budget"))
            continue
        latency = estimate_latency(req, model)
        if latency > req["slo_ms"]:
            rejected.append((name, "latency"))
            continue
        quality = model["quality"].get(req["task"], 0.0)
        cost_ratio = cost / max(req["budget"], 1e-9)
        risk_penalty = max(0.0, req["risk"] - model["safety_score"])
        score = 1.8 * quality - 0.55 * cost_ratio - 0.35 * risk_penalty
        candidates.append((score, name, cost, latency, quality))

    if not candidates and req.get("allow_degrade") and req.get("degrade_to_text"):
        text_req = dict(req)
        text_req["task"] = "summary"
        text_req["modalities"] = {"text"}
        text_req["needs_tool"] = False
        text_req["expected_output_tokens"] = min(req["expected_output_tokens"], 80)
        text_req["budget"] = max(req["budget"], 0.10)
        degraded_decision = route_request(text_req)
        degraded_decision["request_id"] = req["id"]
        degraded_decision["degraded"] = True
        degraded_decision["degrade_reason"] = "drop_image_and_answer_text_only"
        degraded_decision["rejected_candidates"] = rejected[:4]
        return degraded_decision

    if not candidates:
        return {
            "request_id": req["id"],
            "selected_model": None,
            "decision": "reject",
            "fallback_chain": [],
            "degraded": False,
            "rejected_candidates": rejected[:4],
            "trace": trace_record(req, None, "reject", 0.0, 0),
        }

    candidates.sort(reverse=True)
    score, selected, cost, latency, quality = candidates[0]
    chain = [selected]
    for fallback in ("small_chat", "fallback_chat"):
        if fallback != selected and req["tenant"] in MODELS[fallback]["tenants"]:
            if req["task"] in MODELS[fallback]["tasks"] and req["modalities"].issubset(MODELS[fallback]["modalities"]):
                chain.append(fallback)
    return {
        "request_id": req["id"],
        "selected_model": selected,
        "decision": "route",
        "fallback_chain": chain[:3],
        "score": round(score, 3),
        "estimated_cost": round(cost, 4),
        "estimated_latency_ms": int(latency),
        "quality": quality,
        "degraded": False,
        "rejected_candidates": rejected[:4],
        "trace": trace_record(req, selected, "route", cost, latency),
    }


def build_profile_cases():
    complete = {"name": "complete_model_routing_case"}
    complete.update({check: True for check in CHECKS})
    cases = [complete]
    bad_names = [
        "request_intent_missing_bad",
        "model_capability_missing_bad",
        "permission_filter_missing_bad",
        "cost_budget_missing_bad",
        "latency_slo_missing_bad",
        "quality_safety_missing_bad",
        "fallback_chain_unbounded_bad",
        "canary_bucket_unstable_bad",
        "health_load_ignored_bad",
        "route_trace_missing_bad",
        "degrade_policy_uncontrolled_bad",
        "config_version_missing_bad",
        "model_routing_gate_missing_bad",
    ]
    for bad_name, failed_check in zip(bad_names, CHECKS):
        case = {"name": bad_name}
        case.update({check: True for check in CHECKS})
        case[failed_check] = False
        cases.append(case)
    return cases


def audit_profiles(cases):
    metrics = {}
    for check in CHECKS:
        metrics[check] = round(sum(1 for case in cases if case.get(check)) / len(cases), 3)
    failed_cases = [case["name"] for case in cases if not all(case.get(check) for check in CHECKS)]
    failed_gates = [check for check, value in metrics.items() if value < 1.0]
    hard_blockers = sum(1 for case in cases if case["name"].endswith("_bad"))
    gate_pass = min(metrics.values()) >= 0.95 and hard_blockers == 0
    return metrics, failed_cases, failed_gates, hard_blockers, gate_pass

## 30.20 面试常见追问

问题一：模型路由和负载均衡有什么区别？

可以回答：负载均衡通常在等价实例之间分流；模型路由要先在不同能力、成本、延迟、版本和安全等级的模型之间选择，再选择具体 endpoint。

问题二：如何设计成本路由？

可以回答：可以建立模型成本和能力画像，对简单任务优先使用小模型，对复杂任务或低置信度结果升级到大模型，同时结合租户预算、SLO 和质量反馈做策略控制。

问题三：降级路由要注意什么？

可以回答：要明确哪些业务允许降级，降级到什么模型或缓存，是否影响用户体验，如何标记降级结果，并记录降级率、质量和成本影响。

问题四：Router 为什么需要实时指标？

可以回答：因为 endpoint 负载和健康状态动态变化。如果只按静态配置路由，可能继续把请求打到拥塞或故障实例，导致延迟和错误进一步扩大。

问题五：如何避免 fallback 变成雪崩放大器？

可以回答：fallback chain 要有最大尝试次数、总时间预算、总成本预算、错误类型分类、幂等控制和熔断规则。不能对所有错误无脑重试，也不能在主模型超时后继续走一串同样拥塞的模型。

问题六：模型灰度为什么要稳定分桶？

可以回答：稳定分桶能保证同一用户或同一租户在一段时间内稳定进入同一模型版本，方便比较质量、延迟和安全指标，也避免用户在新旧模型之间来回跳导致体验不一致。

## 30.22 本章小结

本章讲了模型路由。

你需要记住：

1. 模型路由不是简单负载均衡，而是模型、版本、endpoint 和策略的综合决策。
2. 路由依据包括能力、质量、成本、延迟、安全、租户、灰度和实时负载。
3. 成本路由的目标是在满足质量和 SLO 的前提下降低单位成本。
4. 降级和 fallback 可以提升可用性，但必须可控、可观测、可审计。
5. Router 配置必须版本化、可回滚，并记录完整 route trace。
6. 多模型平台的核心能力之一，就是把请求稳定地送到“最合适”而不是“最强”或“最便宜”的模型。
7. 模型路由门禁要同时检查请求画像、能力画像、权限过滤、成本预算、延迟 SLO、质量安全、fallback、灰度、实时健康、trace、降级策略和配置版本。
