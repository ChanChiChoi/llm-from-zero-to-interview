# 第 25 章 训练平台系统设计面试题

前面我们已经讲了训练平台总览、任务提交、分布式启动、配置管理、日志指标事件、checkpoint、容错、数据供给、权限安全和审计。本章把这些内容整合成一道系统设计题：如何设计一个大模型训练平台？

这是一道非常高频的 AI Infra 面试题。它不是问你会不会写训练脚本，而是考你是否理解大模型训练从提交、调度、启动、运行、监控、恢复到产物注册的完整生命周期。

先记住一句话：

> 设计大模型训练平台，要围绕 TrainingJob 生命周期展开：提交、校验、排队、调度、启动、运行、观测、checkpoint、恢复、评估、产物注册、审计和成本归因。

## 25.0 本讲资料边界与第二轮精修口径

本讲按 `WRITING_PLAN.md` 做第二轮精修时，重点参考 Kubernetes Job、Kueue、Kubeflow Trainer、PyTorch `torchrun`、OpenTelemetry 和 MLflow Tracking 等公开文档来校准系统边界：Kubernetes Job 更接近通用批任务控制器，Kueue 解决队列、配额、准入和资源共享，Kubeflow Trainer / Training Operator 更接近训练任务的高层语义，`torchrun` 负责把节点、rank、rendezvous 和进程数转成分布式启动参数，OpenTelemetry 把 logs、metrics、traces 等信号放进统一观测模型，MLflow Tracking 则帮助理解 experiment、run、params、metrics 和 artifacts 的记录边界。

这不意味着面试必须背某一个工具的 API。面试官真正想看的是你能不能把训练平台拆成稳定抽象：

1. 控制面：TrainingJob schema、准入校验、配额、状态机和审计。
2. 调度面：queue、priority、quota、gang scheduling、topology-aware placement、preemption 和 backfilling。
3. 启动面：launcher adapter、rank、local rank、world size、rendezvous、GPU binding 和 NCCL 环境。
4. 运行面：数据版本、checkpoint、日志指标事件、故障分类、重试和恢复。
5. 治理面：权限、安全、成本、artifact lineage、experiment tracking 和开发者自助。

所以本章中的 `TrainingJob` 是教学用抽象，不等同于某个厂商或开源项目的单一 CRD。你可以在实现中选择 Kubernetes + Kueue + Kubeflow Trainer，也可以选择 Slurm / Ray / 自研控制面，但系统设计回答必须证明这些模块之间的输入、输出、状态、指标和失败边界都清楚。

## 25.1 面试题描述

题目可能这样问：

```text
请设计一个企业级大模型训练平台。平台需要支持多个团队提交预训练、SFT、RLHF、评估等任务，能够管理 GPU 资源、分布式训练、数据版本、checkpoint、日志指标、失败恢复、权限安全、审计和成本统计。
```

这道题不要一上来就说“用 Kubernetes”。Kubernetes 是底座，不是完整答案。

面试官想听的是：你能不能把训练平台拆成模块，并解释关键 trade-off。

## 25.4 总体架构

一个合理架构：

```text
User / Portal / CLI / SDK
  -> Training Job API
  -> Config Validator
  -> Policy / Quota Engine
  -> Queue / Scheduler
  -> Launcher / Training Operator
  -> Kubernetes / GPU Cluster
      -> Worker Pods
      -> GPU / Network / Storage
  -> Data Access Layer
  -> Checkpoint Manager
  -> Observability: Logs / Metrics / Events / Trace
  -> Experiment Tracker
  -> Model Registry / Artifact Store
  -> Audit / Cost / Notification
```

模块说明：

1. Portal / CLI / SDK：用户提交和管理任务。
2. Training Job API：统一任务入口。
3. Config Validator：校验配置、镜像、代码、数据和资源。
4. Policy / Quota Engine：权限、配额、安全策略。
5. Queue / Scheduler：队列、优先级、gang scheduling、拓扑感知。
6. Launcher / Operator：启动 torchrun、DeepSpeed、Megatron、Ray。
7. Kubernetes / GPU Cluster：运行训练容器。
8. Data Access Layer：数据集、manifest、缓存和权限。
9. Checkpoint Manager：保存、恢复、保留和清理。
10. Observability：日志、指标、事件、trace、告警。
11. Experiment Tracker：参数、配置、数据、指标、环境。
12. Model Registry：模型权重、tokenizer、config、eval report。
13. Audit / Cost：审计日志和 GPU-hour 成本归因。

## 25.5 TrainingJob 设计

TrainingJob 是核心对象。

它应该包含：

1. metadata：job_name、owner、project、tags。
2. image：镜像 digest。
3. code：repo、commit。
4. data：dataset_id、version、manifest。
5. command：entrypoint、args。
6. config：最终生效配置。
7. resources：gpu_type、gpu_count、gpu_per_node、cpu、memory。
8. distributed：launcher、nnodes、nproc_per_node、parallelism。
9. checkpoint：save policy、resume_from、output_uri。
10. logging：logs、metrics、events。
11. scheduling：queue、priority、preemptible。
12. security：service_account、data_policy。

面试中要强调：TrainingJob 不是普通 Job，它是训练生命周期对象。

## 25.6 提交流程

任务提交流程：

1. 用户通过 Portal / CLI / SDK 提交 TrainingJob。
2. API 做 schema 校验。
3. 校验镜像 digest 和安全扫描状态。
4. 校验代码 commit。
5. 校验数据版本和访问权限。
6. 合并配置并生成最终生效配置。
7. 校验资源申请和配额。
8. 校验 checkpoint 路径。
9. 写入 metadata store。
10. 进入调度队列。

提交系统要支持幂等，避免重复创建任务。

## 25.7 调度设计

训练平台调度需要：

1. 队列。
2. 优先级。
3. 配额。
4. 公平性。
5. gang scheduling。
6. 拓扑感知。
7. 抢占。
8. backfilling。
9. GPU 碎片治理。

重点解释：

1. 分布式训练需要 worker 同时启动，所以需要 gang scheduling。
2. 大模型训练通信密集，所以需要拓扑感知。
3. 低优先级任务可以使用空闲资源，但要支持抢占和 checkpoint。
4. 多团队共用集群，需要配额和公平分享。

## 25.8 分布式启动设计

调度分配资源后，launcher 负责启动训练。

它要生成：

1. master_addr。
2. master_port。
3. rank。
4. local_rank。
5. world_size。
6. CUDA_VISIBLE_DEVICES。
7. NCCL 环境变量。
8. 启动命令。

平台应支持多种 launcher adapter：

1. torchrun。
2. DeepSpeed。
3. Megatron。
4. Ray。
5. MPI。

统一 adapter 可以降低用户使用复杂度。

## 25.9 数据层设计

数据层负责：

1. 数据集注册。
2. 数据版本。
3. dataset manifest。
4. 数据权限。
5. shard 管理。
6. 流式读取。
7. 本地缓存。
8. 分布式缓存。
9. shuffle 和 rank 切分。
10. 数据质量监控。

训练平台应保证：

1. 训练使用的数据版本可追踪。
2. 用户只能访问授权数据。
3. 数据读取吞吐足够。
4. 断点恢复时能恢复数据位置。

## 25.10 Checkpoint 设计

Checkpoint Manager 负责：

1. 保存频率。
2. resume checkpoint。
3. eval checkpoint。
4. release checkpoint。
5. 分片保存。
6. 异步保存。
7. manifest 和 checksum。
8. committed 状态。
9. 生命周期策略。
10. 从 checkpoint 恢复。

和调度联动：

1. 可抢占任务必须配置 checkpoint。
2. 抢占前尽量保存。
3. 重试时从最新 committed checkpoint 恢复。

## 25.12 容错设计

训练平台要做故障分类。

可重试：

1. 节点失败。
2. GPU 短暂故障。
3. 临时网络故障。
4. 存储短暂不可用。
5. Pod 被驱逐。

不可盲目重试：

1. 配置错误。
2. 权限错误。
3. 代码 bug。
4. 持续 OOM。
5. checkpoint 不兼容。
6. loss NaN。

容错流程：

1. 识别错误类型。
2. 生成诊断报告。
3. 判断是否可重试。
4. 选择健康节点。
5. 从 checkpoint 恢复。
6. 记录 attempt。
7. 通知用户。

完整流程如下：

1. 用户提交 TrainingJob。
2. 平台校验镜像、代码、数据、配置和权限。
3. 生成最终生效配置。

13. 评估平台运行评估。
14. Audit 和 Cost 系统记录全过程。

这个流程就是系统设计题的主线。

## 25.17 系统设计审计指标与最小 demo

系统设计题最容易失分的地方，是把答案讲成“模块名列表”。更好的方式是把设计转成一张审计表：每个模块有什么输入、输出、状态、指标、失败模式和门禁。

训练平台系统设计样本可以抽象成：

```math
p_i=(r_i,j_i,s_i,a_i,q_i,l_i,d_i,c_i,o_i,f_i,g_i,k_i,u_i,m_i,b_i,t_i)
```

其中，`r_i` 是需求澄清，`j_i` 是 TrainingJob 契约，`s_i` 是生命周期状态机，`a_i` 是准入和配额，`q_i` 是调度，`l_i` 是 launcher，`d_i` 是数据和配置血缘，`c_i` 是 checkpoint，`o_i` 是可观测性和实验追踪，`f_i` 是容错恢复，`g_i` 是安全审计，`k_i` 是成本容量，`u_i` 是开发者自助，`m_i` 是接口和 artifact 契约，`b_i` 是 trade-off 边界，`t_i` 是最终设计门禁。

第 `j` 个设计维度的覆盖率可以写成：

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(p_i)=1]
```

分布式训练的总进程数和 effective batch 可以先用两个简单公式验算：

```math
W=n_{\mathrm{node}}n_{\mathrm{proc}},\qquad B_{\mathrm{eff}}=B_{\mathrm{micro}}A_{\mathrm{grad}}W
```

其中，`W` 是 world size，`n_node` 是节点数，`n_proc` 是每节点进程数，`B_micro` 是每进程 micro batch，`A_grad` 是 gradient accumulation steps。系统设计回答如果连这些基本一致性都没有，就很难证明 launcher、资源申请和训练配置是一致的。

调度层可以用 P95 排队时间约束体验：

```math
T_{\mathrm{queue,p95}}=Q_{0.95}\left(\{t_{\mathrm{start},i}-t_{\mathrm{submit},i}\}_{i=1}^{N}\right)
```

checkpoint 和抢占的损失可以用最大丢失 GPU-hours 估算：

```math
L_{\mathrm{lost}}=G\frac{t_{\mathrm{fail}}-t_{\mathrm{ckpt}}}{60}
```

其中，`G` 是本次任务占用 GPU 数，时间单位是分钟。experiment、checkpoint、model registry 和 audit 的串联程度可以用 trace 覆盖率表示：

```math
R_{\mathrm{trace}}=\frac{N_{\mathrm{linked}}}{N_{\mathrm{required}}}
```

最终可以把设计门禁写成：

```math
G_{\mathrm{design}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{queue,p95}}\le B_{\mathrm{queue}} \land L_{\mathrm{lost}}\le B_{\mathrm{loss}} \land R_{\mathrm{trace}}\ge \rho_{\mathrm{trace}} \land P_0=0\right]
```

下面这个 demo 用 16 个门禁检查一个训练平台系统设计回答。它不是生产评审器，而是帮助你把面试回答从“我会用 Kubernetes”升级成“我能证明训练生命周期闭环可运行、可恢复、可复现、可治理”。

```python
from math import ceil


GATES = [
    "requirement_clarification",
    "trainingjob_contract",
    "lifecycle_state_machine",
    "admission_quota_policy",
    "scheduler_resource_fit",
    "distributed_launcher_fit",
    "data_config_lineage",
    "checkpoint_recovery_coupling",
    "observability_experiment_linkage",
    "fault_recovery_flow",
    "security_audit_governance",
    "cost_capacity_governance",
    "developer_self_service",
    "interface_artifact_contract",
    "tradeoff_boundary_reasoning",
    "training_platform_design_gate",
]


def p95(values):
    ordered = sorted(values)
    index = max(0, ceil(0.95 * len(ordered)) - 1)
    return ordered[index]


def all_present(mapping, names):
    return all(mapping.get(name) for name in names)


def evaluate_case(case):
    expected_states = [
        "submitted",
        "validated",
        "queued",
        "scheduled",
        "starting",
        "running",
        "checkpointing",
        "retrying",
        "failed",
        "completed",
        "cancelled",
    ]
    required_job_fields = [
        "owner",
        "project",
        "image_digest",
        "code_commit",
        "dataset_version",
        "final_config_hash",
        "resources",
        "distributed",
        "checkpoint_policy",
        "observability",
        "security",
        "queue",
    ]
    required_requirements = [
        "workload_scope",
        "cluster_scale",
        "multi_tenant",
        "data_sensitivity",
        "preemption",
        "recovery_slo",
        "artifact_release",
        "cost_budget",
    ]
    required_tradeoffs = [
        "kubernetes_vs_training_operator",
        "fairness_vs_utilization",
        "topology_vs_queue_time",
        "preemption_vs_lost_work",
        "self_service_vs_guardrail",
        "cost_vs_reliability",
    ]

    resources = case["resources"]
    world_size = resources["nodes"] * resources["processes_per_node"]
    effective_batch = resources["micro_batch"] * resources["grad_accum"] * world_size

    lineage = case["lineage"]
    checkpoint = case["checkpoint"]
    observability = case["observability"]
    cost = case["cost"]

    results = {}
    results["requirement_clarification"] = all_present(case["requirements"], required_requirements)
    results["trainingjob_contract"] = all_present(case["trainingjob"], required_job_fields)
    results["lifecycle_state_machine"] = (
        set(expected_states).issubset(case["states"])
        and case["state_transitions_valid"]
    )
    results["admission_quota_policy"] = all_present(
        case["admission"],
        ["schema", "quota_dry_run", "priority_approval", "image_policy", "dataset_permission", "idempotency"],
    )
    results["scheduler_resource_fit"] = all_present(
        case["scheduler"],
        ["queue", "quota", "fair_share", "gang", "topology", "backfilling", "preemption", "fragmentation"],
    )
    results["distributed_launcher_fit"] = (
        case["launcher"].get("world_size") == world_size
        and all_present(case["launcher"], ["rank_mapping", "rendezvous", "gpu_binding", "nccl", "adapter"])
    )
    results["data_config_lineage"] = all_present(
        lineage,
        ["image_digest", "code_commit", "dataset_version", "manifest", "final_config_hash", "env_versions", "seed", "experiment_run"],
    )
    results["checkpoint_recovery_coupling"] = (
        all_present(checkpoint, ["policy", "committed_manifest", "checksum", "restore_drill", "scheduler_preemption_hook"])
        and checkpoint["max_stale_minutes"] <= 30
    )
    results["observability_experiment_linkage"] = all_present(
        observability,
        ["logs", "metrics", "events", "traces", "rank_dimensions", "experiment_link", "dashboard", "alerts"],
    )
    results["fault_recovery_flow"] = all_present(

def clone_case(case, name):

    for key, value in case.items():

        elif isinstance(value, list):
            copied[key] = list(value)
        else:
            copied[key] = value
    copied["name"] = name
    return copied
```

## 25.18 常见扣分回答

扣分回答一：只说“用 Kubernetes 跑 Job”。

问题是没有讲训练语义、分布式启动、checkpoint、数据、实验和容错。

扣分回答二：没有 TrainingJob 抽象。

没有核心对象，系统会变成脚本堆积。

扣分回答三：没有 gang scheduling。

分布式训练不能只靠普通 Pod 调度。

扣分回答四：没有数据版本和配置复现。

训练平台必须能回答模型怎么训出来。

扣分回答五：没有 checkpoint 和恢复。

大模型训练长周期，没有恢复能力风险很高。

扣分回答六：没有安全和审计。

训练平台连接数据、GPU、代码和模型产物，安全必不可少。

## 25.19 面试题

### 题 1：训练平台的核心对象是什么？

答：TrainingJob。它描述一次训练任务的镜像、代码、数据、配置、资源、分布式启动、checkpoint、日志、队列、优先级、安全和 metadata，是训练生命周期管理的核心对象。

### 题 2：为什么不能只用 Kubernetes Job？

答：Kubernetes Job 只提供通用任务运行能力，不理解训练语义。大模型训练还需要分布式启动、gang scheduling、拓扑感知、checkpoint、实验追踪、数据版本、容错、安全审计和成本归因。

### 题 3：训练平台如何保证可复现？

答：记录镜像 digest、代码 commit、数据版本、最终生效配置、环境版本、随机种子、启动命令、checkpoint、评估配置和平台策略版本。

### 题 4：训练平台如何处理失败？

答：先分类故障。节点、临时网络、Pod 驱逐等可恢复故障可以从 checkpoint 自动重试；配置错误、权限错误、代码 bug、持续 OOM 和数值发散应快速失败并生成诊断报告。

### 题 5：训练平台如何做多租户治理？

答：通过身份认证、RBAC/ABAC、队列、配额、优先级、资源池、数据权限、镜像策略、密钥管理、日志权限、模型产物权限、审计和成本归因实现。

## 25.21 本章小结

本章用系统设计题串起了大模型训练平台。

你需要掌握：

1. 训练平台是 TrainingJob 生命周期管理系统。
2. 系统要覆盖提交、校验、调度、启动、运行、观测、checkpoint、恢复、产物、审计和成本。
3. TrainingJob 是核心抽象。
4. 调度要支持队列、优先级、配额、gang scheduling、拓扑感知和抢占。
5. Launcher 要统一封装 torchrun、DeepSpeed、Megatron、Ray。
6. 数据层要支持版本、manifest、缓存、shuffle 和权限。
7. Checkpoint 是容错、抢占和发布的基础。
8. 可观测性要包括 logs、metrics、events、traces。
9. 容错要做故障分类，而不是盲目重试。
10. 安全、审计和成本治理是企业训练平台必须能力。
