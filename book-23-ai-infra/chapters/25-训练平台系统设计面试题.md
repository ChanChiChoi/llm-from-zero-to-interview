# 第 25 章 训练平台系统设计面试题

前面我们已经讲了训练平台总览、任务提交、分布式启动、配置管理、日志指标事件、checkpoint、容错、数据供给、权限安全和审计。本章把这些内容整合成一道系统设计题：如何设计一个大模型训练平台？

这是一道非常高频的 AI Infra 面试题。它不是问你会不会写训练脚本，而是考你是否理解大模型训练从提交、调度、启动、运行、监控、恢复到产物注册的完整生命周期。

先记住一句话：

> 设计大模型训练平台，要围绕 TrainingJob 生命周期展开：提交、校验、排队、调度、启动、运行、观测、checkpoint、恢复、评估、产物注册、审计和成本归因。

## 25.1 面试题描述

题目可能这样问：

```text
请设计一个企业级大模型训练平台。平台需要支持多个团队提交预训练、SFT、RLHF、评估等任务，能够管理 GPU 资源、分布式训练、数据版本、checkpoint、日志指标、失败恢复、权限安全、审计和成本统计。
```

这道题不要一上来就说“用 Kubernetes”。Kubernetes 是底座，不是完整答案。

面试官想听的是：你能不能把训练平台拆成模块，并解释关键 trade-off。

## 25.2 需求澄清

可以先问：

1. 平台主要支持预训练，还是 SFT/微调为主？
2. GPU 集群规模多大？
3. 是否多租户？
4. 是否需要抢占式调度？
5. 是否需要支持多种 launcher？
6. 数据是否敏感？
7. 是否需要自动恢复？
8. 是否需要模型注册和发布？
9. 是否有成本预算要求？

如果要自己假设，可以说：

```text
我假设这是企业内部大模型训练平台，支持预训练、SFT、评估和 RLHF。平台服务多个团队，底层有 Kubernetes + GPU 集群。系统需要多租户、队列调度、分布式启动、数据版本、checkpoint、可观测性、自动恢复、模型 artifact 管理、权限审计和成本归因。
```

## 25.3 核心目标

训练平台要实现：

1. 标准化任务提交。
2. GPU 资源调度。
3. 分布式训练启动。
4. 数据版本和权限管理。
5. 配置和环境可复现。
6. 日志、指标、事件和 trace。
7. checkpoint 保存和恢复。
8. 失败自动诊断和重试。
9. 模型 artifact 注册。
10. 权限、安全、审计和成本治理。

这些目标对应平台的核心模块。

## 25.4 总体架构

一个合理架构：

```text
User / Portal / CLI / SDK
  -> Training Job API
  -> Config Validator
  -> Policy / Quota Engine
  -> Queue / Scheduler
  -> Launcher / Training Operator
  -> Kubernetes / GPU Cluster
      -> Worker Pods
      -> GPU / Network / Storage
  -> Data Access Layer
  -> Checkpoint Manager
  -> Observability: Logs / Metrics / Events / Trace
  -> Experiment Tracker
  -> Model Registry / Artifact Store
  -> Audit / Cost / Notification
```

模块说明：

1. Portal / CLI / SDK：用户提交和管理任务。
2. Training Job API：统一任务入口。
3. Config Validator：校验配置、镜像、代码、数据和资源。
4. Policy / Quota Engine：权限、配额、安全策略。
5. Queue / Scheduler：队列、优先级、gang scheduling、拓扑感知。
6. Launcher / Operator：启动 torchrun、DeepSpeed、Megatron、Ray。
7. Kubernetes / GPU Cluster：运行训练容器。
8. Data Access Layer：数据集、manifest、缓存和权限。
9. Checkpoint Manager：保存、恢复、保留和清理。
10. Observability：日志、指标、事件、trace、告警。
11. Experiment Tracker：参数、配置、数据、指标、环境。
12. Model Registry：模型权重、tokenizer、config、eval report。
13. Audit / Cost：审计日志和 GPU-hour 成本归因。

## 25.5 TrainingJob 设计

TrainingJob 是核心对象。

它应该包含：

1. metadata：job_name、owner、project、tags。
2. image：镜像 digest。
3. code：repo、commit。
4. data：dataset_id、version、manifest。
5. command：entrypoint、args。
6. config：最终生效配置。
7. resources：gpu_type、gpu_count、gpu_per_node、cpu、memory。
8. distributed：launcher、nnodes、nproc_per_node、parallelism。
9. checkpoint：save policy、resume_from、output_uri。
10. logging：logs、metrics、events。
11. scheduling：queue、priority、preemptible。
12. security：service_account、data_policy。

面试中要强调：TrainingJob 不是普通 Job，它是训练生命周期对象。

## 25.6 提交流程

任务提交流程：

1. 用户通过 Portal / CLI / SDK 提交 TrainingJob。
2. API 做 schema 校验。
3. 校验镜像 digest 和安全扫描状态。
4. 校验代码 commit。
5. 校验数据版本和访问权限。
6. 合并配置并生成最终生效配置。
7. 校验资源申请和配额。
8. 校验 checkpoint 路径。
9. 写入 metadata store。
10. 进入调度队列。

提交系统要支持幂等，避免重复创建任务。

## 25.7 调度设计

训练平台调度需要：

1. 队列。
2. 优先级。
3. 配额。
4. 公平性。
5. gang scheduling。
6. 拓扑感知。
7. 抢占。
8. backfilling。
9. GPU 碎片治理。

重点解释：

1. 分布式训练需要 worker 同时启动，所以需要 gang scheduling。
2. 大模型训练通信密集，所以需要拓扑感知。
3. 低优先级任务可以使用空闲资源，但要支持抢占和 checkpoint。
4. 多团队共用集群，需要配额和公平分享。

## 25.8 分布式启动设计

调度分配资源后，launcher 负责启动训练。

它要生成：

1. master_addr。
2. master_port。
3. rank。
4. local_rank。
5. world_size。
6. CUDA_VISIBLE_DEVICES。
7. NCCL 环境变量。
8. 启动命令。

平台应支持多种 launcher adapter：

1. torchrun。
2. DeepSpeed。
3. Megatron。
4. Ray。
5. MPI。

统一 adapter 可以降低用户使用复杂度。

## 25.9 数据层设计

数据层负责：

1. 数据集注册。
2. 数据版本。
3. dataset manifest。
4. 数据权限。
5. shard 管理。
6. 流式读取。
7. 本地缓存。
8. 分布式缓存。
9. shuffle 和 rank 切分。
10. 数据质量监控。

训练平台应保证：

1. 训练使用的数据版本可追踪。
2. 用户只能访问授权数据。
3. 数据读取吞吐足够。
4. 断点恢复时能恢复数据位置。

## 25.10 Checkpoint 设计

Checkpoint Manager 负责：

1. 保存频率。
2. resume checkpoint。
3. eval checkpoint。
4. release checkpoint。
5. 分片保存。
6. 异步保存。
7. manifest 和 checksum。
8. committed 状态。
9. 生命周期策略。
10. 从 checkpoint 恢复。

和调度联动：

1. 可抢占任务必须配置 checkpoint。
2. 抢占前尽量保存。
3. 重试时从最新 committed checkpoint 恢复。

## 25.11 可观测性设计

训练可观测性包括：

1. Logs。
2. Metrics。
3. Events。
4. Traces。
5. Dashboard。
6. Alerts。

关键指标：

1. loss。
2. learning rate。
3. tokens/s。
4. step time。
5. GPU utilization。
6. memory。
7. communication time。
8. dataloader time。
9. checkpoint time。
10. queue wait time。

关键事件：

1. JobSubmitted。
2. JobScheduled。
3. TrainingStarted。
4. CheckpointSaved。
5. WorkerFailed。
6. RetryStarted。
7. JobCompleted。

可观测性要支持 rank、node、attempt 维度。

## 25.12 容错设计

训练平台要做故障分类。

可重试：

1. 节点失败。
2. GPU 短暂故障。
3. 临时网络故障。
4. 存储短暂不可用。
5. Pod 被驱逐。

不可盲目重试：

1. 配置错误。
2. 权限错误。
3. 代码 bug。
4. 持续 OOM。
5. checkpoint 不兼容。
6. loss NaN。

容错流程：

1. 识别错误类型。
2. 生成诊断报告。
3. 判断是否可重试。
4. 选择健康节点。
5. 从 checkpoint 恢复。
6. 记录 attempt。
7. 通知用户。

## 25.13 安全和审计设计

训练平台安全包括：

1. 身份认证。
2. RBAC + ABAC。
3. 数据权限。
4. 数据等级传播。
5. 镜像扫描和签名。
6. 密钥管理。
7. 网络 egress 控制。
8. 日志脱敏。
9. 模型产物权限。
10. 审计日志。

审计要记录：

1. 谁提交任务。
2. 用了什么镜像。
3. 读了什么数据。
4. 生成了什么 checkpoint。
5. 谁下载了模型。
6. 谁修改了配额。
7. 谁审批了高风险操作。

## 25.14 成本治理

训练平台要记录：

1. GPU-hours。
2. CPU 和内存。
3. 存储。
4. checkpoint。
5. 网络。
6. 日志。
7. 评估推理成本。

成本维度：

1. team。
2. project。
3. user。
4. job。
5. queue。
6. model。

平台可以做：

1. 预算告警。
2. 低效任务告警。
3. 闲置 GPU 检测。
4. checkpoint 成本报告。
5. 队列利用率分析。

成本治理是训练平台生产化的重要标志。

## 25.15 一个完整流程

完整流程如下：

1. 用户提交 TrainingJob。
2. 平台校验镜像、代码、数据、配置和权限。
3. 生成最终生效配置。
4. 任务进入队列。
5. 调度器根据配额、优先级和拓扑分配 GPU。
6. Launcher 生成 rank、world size 和启动命令。
7. Worker Pod 启动训练。
8. Data Layer 提供训练数据。
9. Observability 收集日志、指标和事件。
10. Checkpoint Manager 周期保存。
11. 失败时故障分类并自动恢复。
12. 训练结束后注册模型和 artifact。
13. 评估平台运行评估。
14. Audit 和 Cost 系统记录全过程。

这个流程就是系统设计题的主线。

## 25.16 面试回答模板

可以这样回答：

```text
我会把大模型训练平台设计成 TrainingJob 生命周期管理系统。

入口层提供 Portal、CLI、SDK 和 API，用户提交 TrainingJob，包含镜像 digest、代码 commit、数据版本、训练配置、分布式配置、资源需求、checkpoint 策略、队列和优先级。

控制面先做 schema、镜像、代码、数据权限、资源配额和安全策略校验，生成最终生效配置并写入 metadata store。

调度层负责队列、优先级、配额、公平性、gang scheduling、拓扑感知、抢占和 backfilling。大模型训练需要 worker 同时启动，也需要考虑 GPU/NVLink/机柜网络拓扑。

启动层通过 launcher adapter 支持 torchrun、DeepSpeed、Megatron、Ray，自动生成 master_addr、rank、world_size、NCCL 配置和启动命令。

运行层接入数据平台，使用 dataset manifest、shard、缓存和权限控制保证数据高吞吐和可复现。Checkpoint Manager 负责分片保存、异步保存、manifest、checksum、保留策略和断点恢复。

可观测性层采集 logs、metrics、events、traces，展示 loss、tokens/s、step time、GPU、通信、dataloader、checkpoint 和 rank 级诊断。

容错层对节点失败、通信失败、OOM、数据错误、配置错误做分类，可恢复故障从 checkpoint 自动恢复，不可恢复错误快速失败并生成诊断报告。

安全治理层提供身份、RBAC/ABAC、数据权限、镜像安全、密钥管理、日志脱敏、模型产物权限和审计。成本层按 team/project/job 记录 GPU-hours、存储和网络成本。
```

## 25.17 常见扣分回答

扣分回答一：只说“用 Kubernetes 跑 Job”。

问题是没有讲训练语义、分布式启动、checkpoint、数据、实验和容错。

扣分回答二：没有 TrainingJob 抽象。

没有核心对象，系统会变成脚本堆积。

扣分回答三：没有 gang scheduling。

分布式训练不能只靠普通 Pod 调度。

扣分回答四：没有数据版本和配置复现。

训练平台必须能回答模型怎么训出来。

扣分回答五：没有 checkpoint 和恢复。

大模型训练长周期，没有恢复能力风险很高。

扣分回答六：没有安全和审计。

训练平台连接数据、GPU、代码和模型产物，安全必不可少。

## 25.18 面试题

### 题 1：训练平台的核心对象是什么？

答：TrainingJob。它描述一次训练任务的镜像、代码、数据、配置、资源、分布式启动、checkpoint、日志、队列、优先级、安全和 metadata，是训练生命周期管理的核心对象。

### 题 2：为什么不能只用 Kubernetes Job？

答：Kubernetes Job 只提供通用任务运行能力，不理解训练语义。大模型训练还需要分布式启动、gang scheduling、拓扑感知、checkpoint、实验追踪、数据版本、容错、安全审计和成本归因。

### 题 3：训练平台如何保证可复现？

答：记录镜像 digest、代码 commit、数据版本、最终生效配置、环境版本、随机种子、启动命令、checkpoint、评估配置和平台策略版本。

### 题 4：训练平台如何处理失败？

答：先分类故障。节点、临时网络、Pod 驱逐等可恢复故障可以从 checkpoint 自动重试；配置错误、权限错误、代码 bug、持续 OOM 和数值发散应快速失败并生成诊断报告。

### 题 5：训练平台如何做多租户治理？

答：通过身份认证、RBAC/ABAC、队列、配额、优先级、资源池、数据权限、镜像策略、密钥管理、日志权限、模型产物权限、审计和成本归因实现。

## 25.20 本章小结

本章用系统设计题串起了大模型训练平台。

你需要掌握：

1. 训练平台是 TrainingJob 生命周期管理系统。
2. 系统要覆盖提交、校验、调度、启动、运行、观测、checkpoint、恢复、产物、审计和成本。
3. TrainingJob 是核心抽象。
4. 调度要支持队列、优先级、配额、gang scheduling、拓扑感知和抢占。
5. Launcher 要统一封装 torchrun、DeepSpeed、Megatron、Ray。
6. 数据层要支持版本、manifest、缓存、shuffle 和权限。
7. Checkpoint 是容错、抢占和发布的基础。
8. 可观测性要包括 logs、metrics、events、traces。
9. 容错要做故障分类，而不是盲目重试。
10. 安全、审计和成本治理是企业训练平台必须能力。
