# 第 34 章 灰度发布、A/B 测试、回滚和模型版本管理

上一章讲了限流、熔断、重试、超时和降级。本章讲模型上线过程中的风险控制：灰度发布、A/B 测试、回滚和模型版本管理。

在传统后端服务里，新版本上线主要担心功能 bug、性能退化和错误率上升。大模型上线还要担心输出质量、风格、安全、成本、延迟、工具调用行为和业务指标变化。

先记住一句话：

> 模型发布不是把新权重部署上去，而是一个围绕版本、流量、指标、风险和回滚建立的发布控制系统。

## 34.1 为什么模型发布更复杂

模型版本变化可能带来多种影响：

1. 回答质量变化。
2. 语言风格变化。
3. 安全拒答率变化。
4. 幻觉率变化。
5. 工具调用格式变化。
6. 结构化输出稳定性变化。
7. 延迟变化。
8. 成本变化。
9. 下游业务转化率变化。
10. 用户满意度变化。

有些变化不是错误率能捕获的。

模型服务可能 200 OK，但答案质量下降，这就是模型发布的特殊风险。

## 34.2 模型版本管理的基本对象

一个成熟平台至少要管理这些对象：

1. Model。
2. ModelVersion。
3. ModelArtifact。
4. TokenizerVersion。
5. RuntimeConfig。
6. PromptTemplateVersion。
7. EvalReport。
8. Deployment。
9. Endpoint。
10. RoutePolicy。

不要只管理权重文件。

模型上线依赖的是一组版本：权重、tokenizer、prompt、runtime、推理参数、路由策略和安全策略。

## 34.3 ModelVersion 应该记录什么

ModelVersion 中通常记录：

1. 模型名称。
2. 版本号。
3. 权重路径。
4. 权重 checksum。
5. tokenizer 版本。
6. 基座模型。
7. 训练数据版本。
8. SFT / RLHF / DPO 信息。
9. 推理默认参数。
10. 最大上下文长度。
11. 支持能力。
12. 评估报告。
13. 安全报告。
14. 发布状态。
15. 创建人和审批人。

这些信息是可追溯性的基础。

当线上出现问题时，你要能回答：这个输出来自哪个模型、哪个版本、哪个 tokenizer、哪个 prompt、哪个 runtime。

## 34.4 发布状态机

模型版本可以有状态机：

```text
created -> validated -> staged -> canary -> production -> deprecated -> archived
```

状态说明：

1. created：版本已注册但未验证。
2. validated：通过基础校验和离线评估。
3. staged：部署到预发环境。
4. canary：小流量灰度。
5. production：正式生产。
6. deprecated：不再推荐使用，但仍可回滚或兼容。
7. archived：归档，不再服务线上请求。

状态机可以防止未经验证的模型直接上线。

## 34.5 发布前校验

模型发布前至少要校验：

1. 权重文件完整性。
2. tokenizer 是否匹配。
3. runtime 是否兼容。
4. 推理参数是否有效。
5. 上下文长度是否符合声明。
6. 基础 benchmark 是否通过。
7. 离线 eval 是否达标。
8. 安全 eval 是否达标。
9. 输出格式是否兼容。
10. 成本和延迟是否可接受。

很多事故来自“模型能加载”但“不符合线上契约”。

## 34.6 灰度发布是什么

灰度发布是让一小部分流量先使用新版本，观察指标后逐步扩大。

典型流程：

```text
0% -> 1% -> 5% -> 10% -> 25% -> 50% -> 100%
```

每一步都要观察指标。

如果指标异常，停止灰度或回滚。

灰度的核心不是“慢慢放量”，而是“在可控范围内暴露风险”。

## 34.7 灰度维度

灰度可以按多种维度：

1. 流量比例。
2. 租户。
3. 用户分桶。
4. 地域。
5. 业务场景。
6. prompt 类型。
7. 模型能力。
8. 风险等级。
9. 内部用户。
10. 白名单客户。

不同维度适合不同场景。

例如新模型先给内部员工、低风险 FAQ、非关键租户使用，比直接全量随机灰度更安全。

## 34.8 稳定分桶

灰度要使用稳定分桶。

稳定分桶指同一个用户或请求实体在同一实验中总是落到同一个版本。

常见方式：

```text
bucket = hash(user_id + experiment_id) % 100
```

稳定分桶的好处：

1. 用户体验一致。
2. 指标统计稳定。
3. 避免同一用户来回切版本。
4. 便于问题复现。

如果每次请求随机分流，用户可能一会儿看到旧模型，一会儿看到新模型，体验和评估都会变差。

## 34.9 A/B 测试是什么

A/B 测试是对比两个或多个版本在真实流量下的表现。

例如：

1. A：旧模型。
2. B：新模型。

比较它们的：

1. 任务成功率。
2. 用户满意度。
3. 点击率或转化率。
4. 安全拦截率。
5. 人工反馈。
6. 延迟。
7. 成本。
8. 工具调用成功率。

A/B 测试不只是模型质量评估，也是业务效果评估。

## 34.10 A/B 测试和灰度的区别

灰度发布关注风险控制。

A/B 测试关注效果比较。

灰度问的是：新版本能不能安全上线？

A/B 测试问的是：新版本是不是比旧版本更好？

它们经常一起使用，但目标不同。

一个模型可以通过灰度稳定性检查，但 A/B 测试发现业务效果没有提升。

## 34.11 模型发布指标

模型发布要观察多类指标。

质量指标：

1. 任务成功率。
2. 用户满意度。
3. 人工偏好胜率。
4. 工具调用成功率。
5. 结构化输出解析成功率。

安全指标：

1. 拒答率。
2. 违规输出率。
3. 敏感信息泄露率。
4. 越权回答率。

性能指标：

1. TTFT。
2. TPOT。
3. p95 / p99 延迟。
4. tokens/s。
5. 错误率。
6. 超时率。

成本指标：

1. cost per 1k tokens。
2. 平均输入 token。
3. 平均输出 token。
4. GPU 利用率。
5. cache 命中率。

业务指标：

1. 转化率。
2. 留存。
3. 工单解决率。
4. 用户投诉率。
5. 人工接管率。

模型发布不能只看一个指标。

## 34.12 Guardrail 指标

Guardrail 指标是发布过程中不能突破的红线。

例如：

1. p99 延迟不能上升超过 20%。
2. 成本不能上升超过 15%。
3. 安全违规率不能上升。
4. 工具调用失败率不能超过阈值。
5. 用户投诉不能显著增加。

即使主指标提升，只要 guardrail 破了，也要停止发布。

例如新模型回答更详细，但成本翻倍、延迟翻倍，就未必适合全量。

## 34.13 回滚机制

回滚是发布系统的生命线。

回滚不应该依赖人工临时改配置。

平台应该支持：

1. 一键回滚模型版本。
2. 回滚路由权重。
3. 回滚 prompt 模板。
4. 回滚 runtime 配置。
5. 回滚安全策略。
6. 回滚灰度实验。

回滚要快，且要可验证。

## 34.14 回滚触发条件

常见回滚触发条件：

1. 错误率上升。
2. 超时率上升。
3. p99 延迟突破阈值。
4. 安全指标异常。
5. 质量指标显著下降。
6. 成本异常上升。
7. 工具调用失败。
8. 结构化输出解析失败。
9. 用户投诉上升。
10. 人工审核发现严重问题。

回滚可以人工触发，也可以自动触发。

自动回滚要谨慎，避免指标波动导致频繁切换。

## 34.15 兼容性问题

模型版本升级可能破坏兼容性。

例如：

1. JSON 输出字段变化。
2. tool call 参数格式变化。
3. 拒答风格变化。
4. prompt 模板不再适配。
5. tokenizer 不兼容。
6. function calling 稳定性下降。
7. 下游 parser 解析失败。

所以模型发布要做契约测试。

契约测试检查模型输出是否满足下游依赖的格式和行为要求。

## 34.16 Prompt 版本也要管理

很多线上变化不是模型权重变了，而是 prompt 变了。

Prompt 模板也应该版本化：

1. system prompt。
2. instruction 模板。
3. tool 描述。
4. RAG 拼接模板。
5. 输出格式约束。
6. 安全提示词。

Prompt 版本变化同样需要灰度、A/B 测试和回滚。

不要把 prompt 当成代码里的字符串随便改。

## 34.17 Runtime 配置也要管理

Runtime 配置变化也可能影响输出和性能。

例如：

1. temperature。
2. top_p。
3. max_tokens。
4. stop sequence。
5. batch 参数。
6. KV cache 参数。
7. quantization。
8. speculative decoding。
9. runtime 版本。

这些变化都应该纳入发布管理。

否则你可能以为是模型质量变差，其实是推理参数或 runtime 变化导致。

## 34.18 发布审计

模型发布必须可审计。

审计内容包括：

1. 谁创建了版本。
2. 谁提交了发布。
3. 谁批准了发布。
4. 发布了什么模型和配置。
5. 灰度比例如何变化。
6. 何时全量。
7. 是否回滚。
8. 回滚原因是什么。
9. 发布期间指标如何变化。

对企业平台来说，模型发布审计和代码发布审计一样重要。

## 34.19 一个完整发布流程

一个完整模型发布流程可以是：

```text
1. 注册 ModelVersion
2. 校验权重、tokenizer、runtime 和配置
3. 运行离线 eval 和安全 eval
4. 部署到 staging
5. 执行契约测试和压测
6. 创建灰度发布计划
7. 对内部或低风险流量放量 1%
8. 观察质量、安全、延迟、成本和业务指标
9. 逐步扩大到 5%、10%、25%、50%
10. 达标后全量
11. 持续监控 guardrail
12. 异常时自动或人工回滚
13. 记录完整审计和复盘信息
```

这个流程看起来繁琐，但这是生产模型平台的必要复杂度。

## 34.20 常见误区

误区一：模型 eval 分数高就可以全量。

离线 eval 不能覆盖所有线上场景，还需要灰度和在线指标。

误区二：发布只管理权重。

权重、tokenizer、prompt、runtime、参数、安全策略和路由都要版本化。

误区三：HTTP 200 就代表发布成功。

模型可能稳定返回错误答案。质量指标和业务指标同样重要。

误区四：回滚就是切回旧模型。

有时需要回滚 prompt、runtime 参数、路由策略或安全策略。

误区五：A/B 测试只看满意度。

A/B 测试还要看延迟、成本、安全和下游业务指标。

## 34.21 面试常见追问

问题一：模型发布和普通服务发布有什么不同？

可以回答：模型发布不仅关注错误率和性能，还要关注输出质量、安全性、工具调用、结构化输出、成本、延迟、用户体验和业务指标，而且很多问题不会表现为 HTTP 错误。

问题二：如何设计模型灰度发布？

可以回答：先注册版本并完成离线 eval、安全 eval、契约测试和压测，再按租户、用户分桶、场景或流量比例小流量灰度，观察质量、安全、延迟、成本和业务指标，达标后逐步放量，异常则回滚。

问题三：为什么 prompt 也要版本管理？

可以回答：prompt 会影响模型行为、输出格式、安全策略和工具调用，改 prompt 可能像改模型一样影响线上结果，因此需要版本、灰度、审计和回滚。

问题四：回滚应该回滚什么？

可以回答：不仅包括模型权重版本，还包括 tokenizer、prompt 模板、runtime 配置、推理参数、路由权重、安全策略和灰度实验配置。

## 34.23 本章小结

本章讲了灰度发布、A/B 测试、回滚和模型版本管理。

你需要记住：

1. 模型发布管理的是一组版本，而不只是权重文件。
2. 模型上线风险包括质量、安全、延迟、成本、工具调用和业务指标变化。
3. 灰度发布用于控制风险，A/B 测试用于比较效果。
4. 发布指标要覆盖质量、安全、性能、成本和业务结果。
5. 回滚能力必须覆盖模型、prompt、runtime、路由和安全策略。
6. 企业级模型平台必须具备版本化、灰度、审计、回滚和复盘能力。
