# 第 34 章 灰度发布、A/B 测试、回滚和模型版本管理

上一章讲了限流、熔断、重试、超时和降级。本章讲模型上线过程中的风险控制：灰度发布、A/B 测试、回滚和模型版本管理。

在传统后端服务里，新版本上线主要担心功能 bug、性能退化和错误率上升。大模型上线还要担心输出质量、风格、安全、成本、延迟、工具调用行为和业务指标变化。

先记住一句话：

> 模型发布不是把新权重部署上去，而是一个围绕版本、流量、指标、风险和回滚建立的发布控制系统。

## 34.0 本讲资料边界与第二轮精修口径

本讲按 `WRITING_PLAN.md` 的第二轮要求做过资料校准。重点参考的是 Kubernetes Deployment 对 rolling update、revision history、rollout status 和 rollback 的稳定控制面口径，Argo Rollouts 对 canary steps、traffic weight、pause、analysis、abort 和 rollback 的发布治理抽象，MLflow Model Registry 对 model、model version、alias、tags、description 和 lineage metadata 的版本管理口径，Google SRE 对 canary release、发布风险控制和指标观测的工程经验，以及在线受控实验 / A/B testing 文献对随机分桶、样本比例、显著性、guardrail 指标和连续看数风险的实验口径。

这些资料共同指向一个稳定事实：模型发布不是“把新模型切成 100% 流量”。发布控制面要同时证明版本包完整、离线评估通过、契约兼容、灰度分桶稳定、A/B 实验设计可信、guardrail 不破、回滚路径可用、trace 可审计，并且任何一个硬门禁失败都能停止放量。

本章只抽象截至 2026-06 仍稳定的模型发布治理方法，不把某个发布平台、模型仓库、service mesh、实验平台或云厂商字段名写成通用标准。正文公式用于面试表达、发布策略审计和最小 demo；真实上线仍要结合业务风险、模型能力、历史事故、实验样本量、SLO、人工审核流程和合规要求校准阈值。

## 34.1 为什么模型发布更复杂

模型版本变化可能带来多种影响：

1. 回答质量变化。
2. 语言风格变化。
3. 安全拒答率变化。
4. 幻觉率变化。
5. 工具调用格式变化。
6. 结构化输出稳定性变化。
7. 延迟变化。
8. 成本变化。
9. 下游业务转化率变化。
10. 用户满意度变化。

有些变化不是错误率能捕获的。

模型服务可能 200 OK，但答案质量下降，这就是模型发布的特殊风险。

## 34.2 模型版本管理的基本对象

一个成熟平台至少要管理这些对象：

1. Model。
2. ModelVersion。
3. ModelArtifact。
4. TokenizerVersion。
5. RuntimeConfig。
6. PromptTemplateVersion。
7. EvalReport。
8. Deployment。
9. Endpoint。
10. RoutePolicy。

不要只管理权重文件。

模型上线依赖的是一组版本：权重、tokenizer、prompt、runtime、推理参数、路由策略和安全策略。

## 34.3 ModelVersion 应该记录什么

ModelVersion 中通常记录：

1. 模型名称。
2. 版本号。
3. 权重路径。
4. 权重 checksum。
5. tokenizer 版本。
6. 基座模型。
7. 训练数据版本。
8. SFT / RLHF / DPO 信息。
9. 推理默认参数。
10. 最大上下文长度。
11. 支持能力。
12. 评估报告。
13. 安全报告。
14. 发布状态。
15. 创建人和审批人。

这些信息是可追溯性的基础。

当线上出现问题时，你要能回答：这个输出来自哪个模型、哪个版本、哪个 tokenizer、哪个 prompt、哪个 runtime。

## 34.4 发布状态机

模型版本可以有状态机：

```text
created -> validated -> staged -> canary -> production -> deprecated -> archived
```

状态说明：

1. created：版本已注册但未验证。
2. validated：通过基础校验和离线评估。
3. staged：部署到预发环境。
4. canary：小流量灰度。
5. production：正式生产。
6. deprecated：不再推荐使用，但仍可回滚或兼容。
7. archived：归档，不再服务线上请求。

状态机可以防止未经验证的模型直接上线。

## 34.5 发布前校验

模型发布前至少要校验：

1. 权重文件完整性。
2. tokenizer 是否匹配。
3. runtime 是否兼容。
4. 推理参数是否有效。
5. 上下文长度是否符合声明。
6. 基础 benchmark 是否通过。
7. 离线 eval 是否达标。
8. 安全 eval 是否达标。
9. 输出格式是否兼容。
10. 成本和延迟是否可接受。

很多事故来自“模型能加载”但“不符合线上契约”。

## 34.6 灰度发布是什么

灰度发布是让一小部分流量先使用新版本，观察指标后逐步扩大。

典型流程：

```text
0% -> 1% -> 5% -> 10% -> 25% -> 50% -> 100%
```

每一步都要观察指标。

如果指标异常，停止灰度或回滚。

灰度的核心不是“慢慢放量”，而是“在可控范围内暴露风险”。

## 34.7 灰度维度

灰度可以按多种维度：

1. 流量比例。
2. 租户。
3. 用户分桶。
4. 地域。
5. 业务场景。
6. prompt 类型。
7. 模型能力。
8. 风险等级。
9. 内部用户。
10. 白名单客户。

不同维度适合不同场景。

例如新模型先给内部员工、低风险 FAQ、非关键租户使用，比直接全量随机灰度更安全。

## 34.8 稳定分桶

灰度要使用稳定分桶。

稳定分桶指同一个用户或请求实体在同一实验中总是落到同一个版本。

常见方式：

```text
bucket = hash(user_id + experiment_id) % 100
```

稳定分桶的好处：

1. 用户体验一致。
2. 指标统计稳定。
3. 避免同一用户来回切版本。
4. 便于问题复现。

如果每次请求随机分流，用户可能一会儿看到旧模型，一会儿看到新模型，体验和评估都会变差。

## 34.9 A/B 测试是什么

A/B 测试是对比两个或多个版本在真实流量下的表现。

例如：

1. A：旧模型。
2. B：新模型。

比较它们的：

1. 任务成功率。
2. 用户满意度。
3. 点击率或转化率。
4. 安全拦截率。
5. 人工反馈。
6. 延迟。
7. 成本。
8. 工具调用成功率。

A/B 测试不只是模型质量评估，也是业务效果评估。

## 34.10 A/B 测试和灰度的区别

灰度发布关注风险控制。

A/B 测试关注效果比较。

灰度问的是：新版本能不能安全上线？

A/B 测试问的是：新版本是不是比旧版本更好？

它们经常一起使用，但目标不同。

一个模型可以通过灰度稳定性检查，但 A/B 测试发现业务效果没有提升。

## 34.11 模型发布指标

模型发布要观察多类指标。

质量指标：

1. 任务成功率。
2. 用户满意度。
3. 人工偏好胜率。
4. 工具调用成功率。
5. 结构化输出解析成功率。

安全指标：

1. 拒答率。
2. 违规输出率。
3. 敏感信息泄露率。
4. 越权回答率。

性能指标：

1. TTFT。
2. TPOT。
3. p95 / p99 延迟。
4. tokens/s。
5. 错误率。
6. 超时率。

成本指标：

1. cost per 1k tokens。
2. 平均输入 token。
3. 平均输出 token。
4. GPU 利用率。
5. cache 命中率。

业务指标：

1. 转化率。
2. 留存。
3. 工单解决率。
4. 用户投诉率。
5. 人工接管率。

模型发布不能只看一个指标。

## 34.12 Guardrail 指标

Guardrail 指标是发布过程中不能突破的红线。

例如：

1. p99 延迟不能上升超过 20%。
2. 成本不能上升超过 15%。
3. 安全违规率不能上升。
4. 工具调用失败率不能超过阈值。
5. 用户投诉不能显著增加。

即使主指标提升，只要 guardrail 破了，也要停止发布。

例如新模型回答更详细，但成本翻倍、延迟翻倍，就未必适合全量。

## 34.13 回滚机制

回滚是发布系统的生命线。

回滚不应该依赖人工临时改配置。

平台应该支持：

1. 一键回滚模型版本。
2. 回滚路由权重。
3. 回滚 prompt 模板。
4. 回滚 runtime 配置。
5. 回滚安全策略。
6. 回滚灰度实验。

回滚要快，且要可验证。

## 34.14 回滚触发条件

常见回滚触发条件：

1. 错误率上升。
2. 超时率上升。
3. p99 延迟突破阈值。
4. 安全指标异常。
5. 质量指标显著下降。
6. 成本异常上升。
7. 工具调用失败。
8. 结构化输出解析失败。
9. 用户投诉上升。
10. 人工审核发现严重问题。

回滚可以人工触发，也可以自动触发。

自动回滚要谨慎，避免指标波动导致频繁切换。

## 34.15 兼容性问题

模型版本升级可能破坏兼容性。

例如：

1. JSON 输出字段变化。
2. tool call 参数格式变化。
3. 拒答风格变化。
4. prompt 模板不再适配。
5. tokenizer 不兼容。
6. function calling 稳定性下降。
7. 下游 parser 解析失败。

所以模型发布要做契约测试。

契约测试检查模型输出是否满足下游依赖的格式和行为要求。

## 34.16 Prompt 版本也要管理

很多线上变化不是模型权重变了，而是 prompt 变了。

Prompt 模板也应该版本化：

1. system prompt。
2. instruction 模板。
3. tool 描述。
4. RAG 拼接模板。
5. 输出格式约束。
6. 安全提示词。

Prompt 版本变化同样需要灰度、A/B 测试和回滚。

不要把 prompt 当成代码里的字符串随便改。

## 34.17 Runtime 配置也要管理

Runtime 配置变化也可能影响输出和性能。

例如：

1. temperature。
2. top_p。
3. max_tokens。
4. stop sequence。
5. batch 参数。
6. KV cache 参数。
7. quantization。
8. speculative decoding。
9. runtime 版本。

这些变化都应该纳入发布管理。

否则你可能以为是模型质量变差，其实是推理参数或 runtime 变化导致。

## 34.18 发布审计

模型发布必须可审计。

审计内容包括：

1. 谁创建了版本。
2. 谁提交了发布。
3. 谁批准了发布。
4. 发布了什么模型和配置。
5. 灰度比例如何变化。
6. 何时全量。
7. 是否回滚。
8. 回滚原因是什么。
9. 发布期间指标如何变化。

对企业平台来说，模型发布审计和代码发布审计一样重要。

## 34.19 一个完整发布流程

一个完整模型发布流程可以是：

```text
1. 注册 ModelVersion
2. 校验权重、tokenizer、runtime 和配置
3. 运行离线 eval 和安全 eval
4. 部署到 staging
5. 执行契约测试和压测
6. 创建灰度发布计划
7. 对内部或低风险流量放量 1%
8. 观察质量、安全、延迟、成本和业务指标
9. 逐步扩大到 5%、10%、25%、50%
10. 达标后全量
11. 持续监控 guardrail
12. 异常时自动或人工回滚
13. 记录完整审计和复盘信息
```

这个流程看起来繁琐，但这是生产模型平台的必要复杂度。

## 34.20 发布治理审计指标与最小 demo

灰度发布、A/B 测试、回滚和版本管理如果分散在不同系统里，线上事故时很难回答“到底哪一个版本、哪一个 prompt、哪一个 runtime 参数导致了问题”。因此需要把模型发布抽象成一个可审计的 release bundle。

一个发布样本可以写成：

```math
r_i=(m_i,w_i,t_i,p_i,c_i,s_i,e_i,g_i,b_i,k_i,l_i,a_i,o_i,z_i)
```

其中，`m_i` 是模型版本，`w_i` 是权重 artifact，`t_i` 是 tokenizer 版本，`p_i` 是 prompt 模板版本，`c_i` 是 runtime config，`s_i` 是安全策略版本，`e_i` 是离线 eval 报告，`g_i` 是灰度计划，`b_i` 是稳定分桶配置，`k_i` 是 guardrail 指标，`l_i` 是回滚目标，`a_i` 是审批信息，`o_i` 是上线观测结果，`z_i` 是 audit trace。

发布版本完整性可以写成覆盖率：

```math
C_{\mathrm{bundle}}=\frac{K_{\mathrm{captured}}}{K_{\mathrm{required}}}
```

其中，`K_{\mathrm{required}}` 包括权重 checksum、tokenizer、prompt、runtime、推理参数、安全策略、eval report、contract test、owner、approver、rollback target 和 policy version。只记录权重路径时，`C_{\mathrm{bundle}}` 远小于 1。

稳定分桶可以写成：

```math
b_u=h(u,e)\bmod 100
```

其中，`u` 是用户或租户实体，`e` 是 experiment id，`h` 是稳定 hash。若灰度权重为 `q`，则进入 treatment 的条件是：

```math
Z_u=\mathbf{1}[b_u<100q]
```

这能保证同一用户在同一实验中稳定看到同一版本，避免每次请求随机切换导致体验和指标污染。

灰度放量可以看成一个单调步骤序列：

```math
0=q_0<q_1<\cdots<q_K\le 1
```

第 `k` 步是否继续放量由指标门禁决定：

```math
G_k=\mathbf{1}\left[
\Delta Q_k\ge -\epsilon_q \land R_{\mathrm{safe},k}\le \rho_s \land L_{99,k}\le B_{99} \land C_k\le B_c \land P_{0,k}=0
\right]
```

其中，`\Delta Q_k` 是候选版本相对 baseline 的质量变化，`R_{\mathrm{safe},k}` 是安全风险率，`L_{99,k}` 是 p99 延迟，`C_k` 是单位成本，`P_{0,k}` 是 P0 风险数。只要某一步 `G_k=0`，发布就应该 pause、abort 或 rollback。

A/B 测试需要同时看主指标和 guardrail。对二值主指标，例如任务成功率，候选版本 `B` 和基线版本 `A` 的差异可以写成：

```math
\Delta_{\mathrm{success}}=\hat{p}_B-\hat{p}_A
```

一个常见的二比例近似统计量是：

```math
z=\frac{\hat{p}_B-\hat{p}_A}{\sqrt{\hat{p}(1-\hat{p})(1/n_A+1/n_B)}}
```

其中，`\hat{p}` 是 pooled success rate，`n_A` 和 `n_B` 是样本量。这个公式不是让面试中手算 p-value，而是提醒你：A/B 测试要有样本量、随机分桶、显著性和停止规则，不能看到一天数据好就全量。

样本比例异常可以用 sample ratio mismatch 的思想检查。若期望 treatment 占比为 `q`，实际 treatment 数为 `n_B`，总样本为 `n`，可以写成：

```math
\chi^2_{\mathrm{srm}}=\frac{(n_B-nq)^2}{nq}+\frac{(n_A-n(1-q))^2}{n(1-q)}
```

若这个值明显偏大，说明分流、日志或过滤可能有问题，A/B 结果不能直接信。

回滚准备度可以写成：

```math
R_{\mathrm{rollback}}=\mathbf{1}\left[
l_i\ne\varnothing \land T_{\mathrm{rollback}}\le B_{\mathrm{rollback}} \land C_{\mathrm{compat}}=1 \land D_{\mathrm{state}}=0
\right]
```

其中，`l_i` 是回滚目标，`T_{\mathrm{rollback}}` 是回滚生效时间，`C_{\mathrm{compat}}` 表示旧版本仍兼容当前路由、prompt、tokenizer 和安全策略，`D_{\mathrm{state}}` 表示是否存在无法回滚的状态迁移。模型发布如果没有已验证回滚目标，就不应该进入大流量灰度。

最终发布治理门禁可以写成：

```math
G_{\mathrm{release}}=\mathbf{1}\left[
\min_j C_j\ge\tau_j \land C_{\mathrm{bundle}}=1 \land G_{\mathrm{offline}}=1 \land G_{\mathrm{canary}}=1 \land G_{\mathrm{ab}}=1 \land R_{\mathrm{rollback}}=1 \land P_0=0
\right]
```

其中，`C_j` 是各审计维度覆盖率，`G_{\mathrm{offline}}` 是离线质量 / 安全 / 契约测试门禁，`G_{\mathrm{canary}}` 是灰度门禁，`G_{\mathrm{ab}}` 是在线实验门禁。这个门禁强调：模型发布系统不仅要能部署，还要能证明为什么放量、为什么停止、为什么回滚。

def stable_bucket(entity_id, experiment_id):
    raw = (entity_id + ":" + experiment_id).encode("utf-8")
    digest = hashlib.sha256(raw).hexdigest()
    return int(digest[:8], 16) % 100


def assign_variant(entity_id, experiment_id, treatment_percent):
    return "candidate" if stable_bucket(entity_id, experiment_id) < treatment_percent else "baseline"


def bundle_completeness(candidate):
    required = [
        "version",
        "weight_checksum",
        "tokenizer_version",
        "prompt_version",
        "runtime_config",
        "safety_policy",
        "eval_report",
        "owner",
        "approver",
        "rollback_target",
    ]
    captured = sum(1 for key in required if candidate.get(key))
    return round(captured / len(required), 3)


def guardrail_failures(step):
    failures = []
    if step["task_success"] - BASELINE["task_success"] < GUARDRAILS["min_task_success_delta"]:
        failures.append("task_success")
    if step["unsafe_rate"] > GUARDRAILS["max_unsafe_rate"]:
        failures.append("unsafe_rate")
    if step["parse_fail_rate"] > GUARDRAILS["max_parse_fail"]:
        failures.append("parse_fail_rate")
    if step["p99_latency_ms"] > GUARDRAILS["max_p99_latency_ms"]:
        failures.append("p99_latency_ms")
    if step["cost_per_1k"] > GUARDRAILS["max_cost_per_1k"]:
        failures.append("cost_per_1k")
    if step["complaint_rate"] > GUARDRAILS["max_complaint_rate"]:
        failures.append("complaint_rate")
    if step["p0"] > 0:
        failures.append("p0")
    return failures


def canary_decisions(steps):
    decisions = {}
    failures = {}
    for step in steps:
        failed = guardrail_failures(step)
        if failed:
            decisions[step["weight"]] = "rollback"
            failures[step["weight"]] = failed
            break
        decisions[step["weight"]] = "promote"
    return decisions, failures


def two_proportion_test(success_a, total_a, success_b, total_b):
    rate_a = success_a / total_a
    rate_b = success_b / total_b
    pooled = (success_a + success_b) / (total_a + total_b)
    se = sqrt(pooled * (1 - pooled) * (1 / total_a + 1 / total_b))
    z = (rate_b - rate_a) / se
    p_value = erfc(abs(z) / sqrt(2))
    return round(rate_b - rate_a, 4), round(z, 3), round(p_value, 6)


def sample_ratio_chi2(observed_treatment, observed_total, expected_ratio):
    observed_control = observed_total - observed_treatment
    expected_treatment = observed_total * expected_ratio
    expected_control = observed_total * (1 - expected_ratio)
    chi2 = ((observed_treatment - expected_treatment) ** 2) / expected_treatment
    chi2 += ((observed_control - expected_control) ** 2) / expected_control
    return round(chi2, 3)

## 34.21 常见误区

误区一：模型 eval 分数高就可以全量。

离线 eval 不能覆盖所有线上场景，还需要灰度和在线指标。

误区二：发布只管理权重。

权重、tokenizer、prompt、runtime、参数、安全策略和路由都要版本化。

误区三：HTTP 200 就代表发布成功。

模型可能稳定返回错误答案。质量指标和业务指标同样重要。

误区四：回滚就是切回旧模型。

有时需要回滚 prompt、runtime 参数、路由策略或安全策略。

误区五：A/B 测试只看满意度。

A/B 测试还要看延迟、成本、安全和下游业务指标。

## 34.22 面试常见追问

问题一：模型发布和普通服务发布有什么不同？

可以回答：模型发布不仅关注错误率和性能，还要关注输出质量、安全性、工具调用、结构化输出、成本、延迟、用户体验和业务指标，而且很多问题不会表现为 HTTP 错误。

问题二：如何设计模型灰度发布？

可以回答：先注册版本并完成离线 eval、安全 eval、契约测试和压测，再按租户、用户分桶、场景或流量比例小流量灰度，观察质量、安全、延迟、成本和业务指标，达标后逐步放量，异常则回滚。

问题三：为什么 prompt 也要版本管理？

可以回答：prompt 会影响模型行为、输出格式、安全策略和工具调用，改 prompt 可能像改模型一样影响线上结果，因此需要版本、灰度、审计和回滚。

问题四：回滚应该回滚什么？

可以回答：不仅包括模型权重版本，还包括 tokenizer、prompt 模板、runtime 配置、推理参数、路由权重、安全策略和灰度实验配置。

问题五：为什么 A/B 主指标显著提升也不能直接全量？

可以回答：因为主指标只代表一个目标，模型发布还必须满足安全、延迟、成本、结构化输出、工具调用、投诉率、样本比例健康、分桶稳定、回滚准备和审计 trace 等 guardrail。任何硬 guardrail 失败都应该暂停或回滚。

## 34.24 本章小结

本章讲了灰度发布、A/B 测试、回滚和模型版本管理。

你需要记住：

1. 模型发布管理的是一组版本，而不只是权重文件。
2. 模型上线风险包括质量、安全、延迟、成本、工具调用和业务指标变化。
3. 灰度发布用于控制风险，A/B 测试用于比较效果。
4. 发布指标要覆盖质量、安全、性能、成本和业务结果。
5. 回滚能力必须覆盖模型、prompt、runtime、路由和安全策略。
6. 企业级模型平台必须具备版本化、灰度、审计、回滚和复盘能力。
7. 发布治理要把 release bundle、稳定分桶、A/B 实验、guardrail、回滚目标和 audit trace 串成同一条证据链。
