# 第 35 章 推理平台系统设计面试题

前面第 26 到第 34 章讲了大模型推理平台工程：推理平台总览、runtime、prefill/decode、KV cache、continuous batching、模型路由、缓存、扩缩容、稳定性保护和模型发布。本章作为第四部分收尾，专门讲推理平台系统设计面试题。

面试中常见题目是：

> 请你设计一个大模型推理平台，支持多模型、多租户、高并发、低延迟、灰度发布和成本治理。

这类题不是让你背组件名，而是考你能否把在线服务、GPU runtime、大模型推理特性和平台治理串起来。

先记住一句话：

> 推理平台系统设计的核心，是在模型质量、延迟、吞吐、成本、安全和可用性之间建立一个可观测、可治理、可演进的在线服务体系。

## 35.1 面试题拆解

题目通常会这样问：

1. 如何部署多个大模型？
2. 如何支持高并发推理？
3. 如何控制 TTFT 和 TPOT？
4. 如何做模型路由？
5. 如何做缓存？
6. 如何自动扩缩容？
7. 如何限流和降级？
8. 如何灰度和回滚？
9. 如何做多租户隔离？
10. 如何控制成本？

如果你直接回答“K8s + vLLM + 网关”，是不够的。

你需要从需求、架构、核心链路、关键瓶颈、稳定性和治理几个层面展开。

## 35.2 先做需求澄清

系统设计题第一步是澄清需求。

可以问：

1. 支持哪些模型类型？文本、embedding、多模态、rerank？
2. 模型规模多大？7B、70B、MoE、长上下文？
3. QPS 和并发目标是多少？
4. 输入输出 token 分布是什么？
5. 延迟 SLO 是多少？TTFT、TPOT、p99？
6. 是否需要 streaming？
7. 是否多租户？租户之间是否隔离？
8. 是否需要 OpenAI-compatible API？
9. 是否需要灰度、A/B 测试和回滚？
10. 是否有成本预算？
11. 是否有安全和审计要求？

这些问题能体现你知道推理平台不是普通 HTTP 服务。

## 35.3 定义核心目标

一个生产级推理平台的核心目标可以定义为：

1. 支持多模型、多版本、多 runtime。
2. 支持高并发和流式生成。
3. 控制 TTFT、TPOT 和 p99 延迟。
4. 提高 GPU 利用率和 token 吞吐。
5. 支持模型路由、灰度和回滚。
6. 支持限流、熔断、超时和降级。
7. 支持多租户权限、配额和隔离。
8. 支持缓存和成本治理。
9. 支持完整指标、日志、trace 和审计。

面试回答时要把目标讲清楚，再讲架构。

## 35.4 总体架构

可以画出这样的架构：

```text
Client / App / Agent
  -> API Gateway
  -> Auth / Quota / Rate Limit
  -> Safety Filter
  -> Model Router
  -> Cache Layer
  -> Request Scheduler / Admission Controller
  -> Inference Runtime Cluster
      -> vLLM / TGI / Triton / SGLang / TensorRT-LLM
      -> GPU Workers
      -> KV Cache / Continuous Batching
  -> Model Registry
  -> Deployment Controller
  -> Autoscaler
  -> Observability Platform
  -> Cost / Audit / Billing
```

模块职责：

1. API Gateway：统一入口、协议转换、认证入口。
2. Auth / Quota / Rate Limit：身份、权限、配额和限流。
3. Safety Filter：输入输出安全策略。
4. Model Router：模型、版本和 endpoint 选择。
5. Cache Layer：prompt cache、语义缓存、结果缓存。
6. Admission Controller：判断系统当前是否接得住。
7. Runtime Cluster：执行 prefill、decode、batching、KV cache。
8. Model Registry：管理模型元数据和版本。
9. Deployment Controller：发布、灰度、回滚。
10. Autoscaler：基于 LLM 指标扩缩容。
11. Observability：指标、日志、trace、告警。
12. Cost / Audit：成本归因和审计。

## 35.5 请求链路

一个完整请求链路：

```text
1. Client 发送请求
2. Gateway 做认证、协议校验和基础限流
3. Quota Service 检查租户配额和 token 预算
4. Safety Filter 检查输入策略
5. Router 根据模型、能力、成本、SLO 和负载选择 endpoint
6. Cache Layer 尝试命中结果缓存或语义缓存
7. Admission Controller 检查队列、KV cache 和实例健康
8. Runtime Scheduler 将请求加入 waiting queue
9. Runtime 执行 prefill，生成 KV cache
10. Runtime 执行 decode，通过 streaming 返回 token
11. Safety Filter 检查输出策略
12. Observability 记录 TTFT、TPOT、tokens、trace 和错误码
13. Cost 模块按租户、模型和 token 做归因
14. Audit 模块记录访问事件
```

这条链路可以帮助面试官看到你对线上服务全流程的理解。

## 35.6 Runtime 设计重点

Runtime 是执行推理的核心。

需要讲清楚：

1. 使用 vLLM、TGI、Triton、SGLang 或 TensorRT-LLM。
2. 支持 continuous batching。
3. 支持 KV cache 管理。
4. 支持 streaming response。
5. 支持长上下文。
6. 暴露 runtime 指标。
7. 支持健康检查。
8. 支持多 GPU 或分布式推理。

关键指标：

1. TTFT。
2. TPOT。
3. input tokens/s。
4. output tokens/s。
5. queue length。
6. active sequences。
7. KV cache usage。
8. GPU utilization。
9. OOM count。

面试中要强调：runtime 不是普通 Python 服务，而是 GPU 执行引擎。

## 35.7 Prefill、Decode 和 KV Cache 怎么讲

可以这样讲：

1. Prefill 处理输入 prompt，主要影响 TTFT。
2. Decode 逐 token 生成，主要影响 TPOT 和总延迟。
3. KV cache 用于避免重复计算，但会占用大量显存。
4. 长上下文和高并发会放大 KV cache 压力。
5. 调度时不能只看 QPS，要看 input/output tokens、active sequences 和 KV cache。

这段是推理平台系统设计的核心区分点。

如果候选人完全不提 prefill/decode/KV cache，说明对 LLM serving 理解不足。

## 35.8 模型路由设计

Router 要做两层选择：

1. 选择模型和版本。
2. 选择 endpoint 和实例。

路由依据包括：

1. 请求指定模型。
2. 任务能力。
3. 租户权限。
4. 延迟 SLO。
5. 成本预算。
6. 灰度规则。
7. endpoint 健康状态。
8. queue length。
9. TTFT / TPOT。
10. KV cache 水位。

Router 输出要包含 route decision 和 route trace，方便审计和排障。

## 35.9 缓存设计

缓存层可以设计为多级缓存：

1. prompt cache：复用系统 prompt、工具说明、长文档前缀。
2. KV cache：由 runtime 管理，复用 attention KV。
3. 语义缓存：相似问题命中历史答案或中间结果。
4. 结果缓存：完全相同请求直接返回。

缓存 key 要包含：

1. 模型版本。
2. tokenizer 版本。
3. prompt hash。
4. 参数。
5. 租户和权限域。
6. 安全策略版本。
7. 数据版本。

要强调缓存不是纯性能问题，还涉及安全隔离、一致性和审计。

## 35.10 自动扩缩容设计

Autoscaler 不能只看 CPU 或 QPS。

应该看：

1. QPS。
2. input tokens/s。
3. output tokens/s。
4. waiting queue length。
5. queue wait time。
6. TTFT。
7. TPOT。
8. p95 / p99 latency。
9. GPU utilization。
10. GPU memory usage。
11. KV cache usage。
12. timeout rate。

还要考虑冷启动：

1. 镜像预拉取。
2. 权重本地缓存。
3. warm pool。
4. 模型预热。
5. 预测扩容。

缩容要有 draining，不再接收新请求，等待 active request 和 streaming 请求完成。

## 35.11 限流和稳定性设计

稳定性保护包括：

1. 租户级限流。
2. API key 限流。
3. token/s 限流。
4. 并发限制。
5. 最大上下文长度。
6. 最大输出长度。
7. admission control。
8. 熔断异常 endpoint。
9. 有预算的重试。
10. 分阶段超时。
11. 降级路由。

要强调重试不能无脑做。

如果请求已经开始生成或已经返回部分 token，重试可能导致重复成本和不一致体验。

## 35.12 灰度发布和回滚设计

模型发布要管理：

1. 权重版本。
2. tokenizer 版本。
3. prompt 模板版本。
4. runtime 配置。
5. 推理参数。
6. 路由策略。
7. 安全策略。

灰度方式：

1. 按流量比例。
2. 按租户。
3. 按用户分桶。
4. 按业务场景。
5. 按风险等级。

观察指标：

1. 质量。
2. 安全。
3. TTFT / TPOT。
4. p99 延迟。
5. 错误率。
6. 成本。
7. 工具调用成功率。
8. 业务指标。

回滚要一键化，并且能回滚模型、prompt、runtime、路由和安全策略。

## 35.13 多租户设计

多租户要考虑：

1. 身份认证。
2. 模型访问权限。
3. 租户级配额。
4. 租户级限流。
5. 租户级成本归因。
6. 数据隔离。
7. 缓存隔离。
8. 日志脱敏。
9. 审计记录。

不要只说 namespace 隔离。

推理平台多租户更关注模型访问权、token 配额、上下文数据和缓存边界。

## 35.14 可观测性设计

可观测性要覆盖请求全链路。

指标：

1. QPS。
2. TTFT。
3. TPOT。
4. p95 / p99 latency。
5. input/output tokens/s。
6. queue length。
7. GPU utilization。
8. KV cache usage。
9. error rate。
10. timeout rate。
11. cache hit rate。
12. cost per 1k tokens。

Trace 阶段：

1. gateway。
2. auth。
3. router。
4. cache。
5. queue。
6. prefill。
7. decode。
8. safety。
9. response streaming。

日志要脱敏，审计要记录谁在何时调用了哪个模型版本。

## 35.15 成本治理设计

成本治理包括：

1. 按租户统计 token 用量。
2. 按模型和版本统计成本。
3. 按 endpoint 统计 GPU 成本。
4. 按请求记录 input/output tokens。
5. 小模型优先路由。
6. 语义缓存和结果缓存。
7. prompt 压缩。
8. 输出长度限制。
9. 自动扩缩容。
10. 离峰批处理。

成本治理的目标不是单纯省钱，而是在满足质量和 SLO 的前提下降低单位 token 成本。

## 35.16 安全设计

安全设计至少包括：

1. API 鉴权。
2. 租户权限。
3. 模型访问控制。
4. 输入安全检查。
5. 输出安全检查。
6. 敏感信息脱敏。
7. 日志和 trace 脱敏。
8. 缓存隔离。
9. 数据留存策略。
10. 审计追踪。

对于企业平台，还要考虑模型输出是否可能泄露私有知识、缓存是否跨租户误命中、日志是否记录敏感 prompt。

## 35.17 系统瓶颈分析

常见瓶颈包括：

1. Prefill 计算瓶颈。
2. Decode 显存带宽瓶颈。
3. KV cache 显存瓶颈。
4. 队列排队瓶颈。
5. 长请求拖慢短请求。
6. 冷启动慢。
7. 路由不均衡。
8. 下游工具调用慢。
9. streaming 连接过多。
10. 缓存命中率低。

优化也要对应瓶颈：

1. Prefill 慢，考虑 prompt cache、长短请求分队列、prefill/decode 分离。
2. Decode 慢，考虑 continuous batching、KV cache 优化、speculative decoding。
3. KV cache 满，考虑限制上下文、PagedAttention、扩容、驱逐策略。
4. 冷启动慢，考虑 warm pool、权重缓存和预热。

## 35.18 面试回答模板

可以这样组织答案：

```text
我会先澄清需求，包括模型类型、QPS、token 分布、延迟 SLO、是否 streaming、多租户、安全和成本要求。

整体架构上，我会设计 API Gateway、鉴权限流、Model Router、Cache Layer、Admission Controller、Inference Runtime Cluster、Model Registry、Deployment Controller、Autoscaler、Observability 和 Cost/Audit 模块。

请求进来后，先做认证、配额和安全检查，再由 Router 根据模型能力、租户权限、成本、SLO、灰度规则和 endpoint 实时负载选择模型版本和实例。缓存层尝试命中 prompt cache、语义缓存或结果缓存。没有命中则进入 runtime 队列，由 runtime 执行 prefill 和 decode，并通过 continuous batching 和 KV cache 管理提高吞吐。

扩缩容上，我不会只看 QPS 或 GPU 利用率，而会结合 input/output tokens/s、TTFT、TPOT、queue length、KV cache usage、p99 和 timeout rate，并考虑大模型冷启动和 warm pool。

稳定性上，会做多维限流、admission control、熔断、分阶段超时、谨慎重试和降级路由。发布上支持模型、prompt、runtime 和路由策略的版本化、灰度、A/B 测试和一键回滚。最后通过指标、日志、trace、审计和成本归因保证平台可运营。
```

这个回答覆盖需求、架构、请求链路、runtime 特性、扩缩容、稳定性、发布和治理。

## 35.19 常见扣分回答

扣分回答一：只说 K8s 部署模型。

问题：忽略 runtime、KV cache、路由、缓存、SLO、扩缩容和成本治理。

扣分回答二：只看 QPS 扩容。

问题：不理解 token 负载、长上下文和 decode 瓶颈。

扣分回答三：失败就重试。

问题：不理解大模型请求成本、非幂等输出和 streaming 特性。

扣分回答四：灰度只看错误率。

问题：模型发布还要看质量、安全、成本、延迟和业务指标。

扣分回答五：缓存不考虑权限。

问题：语义缓存、结果缓存和 prompt cache 都可能引入跨租户数据泄露。

## 35.20 面试官可能继续追问

追问一：如果 p99 延迟突然升高，你怎么排查？

回答方向：先拆分 trace，看 gateway、router、queue、prefill、decode、streaming 哪一段变慢；再看 input/output token 分布、queue length、KV cache、GPU、runtime 错误和是否有长请求或租户突增。

追问二：如果 GPU 利用率不高但请求很慢，可能是什么原因？

回答方向：可能是队列调度、KV cache、显存带宽、长上下文、下游工具、streaming 网络、batch 配置或 admission control 问题，不一定是算力不足。

追问三：如何降低推理成本？

回答方向：小模型优先、能力路由、缓存、prompt 压缩、输出长度控制、量化、batching、自动扩缩容、离峰任务和按租户成本归因。

追问四：如何支持企业多租户？

回答方向：认证鉴权、模型访问控制、租户配额、限流、数据隔离、缓存隔离、日志脱敏、审计、成本归因和租户级 SLO。

## 35.22 本章小结

本章用系统设计面试题串起了整个推理平台工程。

你需要记住：

1. 推理平台不是一个模型 HTTP 服务，而是在线模型服务、GPU runtime 和平台治理的组合。
2. 系统设计要从需求、架构、请求链路、runtime、路由、缓存、扩缩容、稳定性、发布、安全、观测和成本展开。
3. Prefill、decode、KV cache、continuous batching 和 token 负载是 LLM serving 的核心差异点。
4. 多模型平台必须支持模型路由、灰度、回滚、缓存、多租户和成本归因。
5. 好的面试答案要体现 trade-off：质量、延迟、吞吐、成本、安全和可用性不能同时无条件最优。
