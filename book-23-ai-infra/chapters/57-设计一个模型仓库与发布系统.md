# 第 57 章 设计一个模型仓库与发布系统

上一章设计了大模型评估与实验平台。本章继续系统设计题：设计一个模型仓库与发布系统。

模型仓库与发布系统是训练平台、评估平台和推理平台之间的桥梁。训练产出 checkpoint，评估产出报告，发布系统决定哪些模型可以进入线上、如何灰度、如何回滚、如何审计。

先记住一句话：

> 模型仓库与发布系统不是存权重文件的目录，而是模型资产从训练产物到线上服务的版本化、评估、审批、发布和回滚控制系统。

## 57.0 本讲资料边界与第二轮精修口径

第二轮精修时，本章按“模型资产生命周期控制面 + 发布治理证据链”校准，不绑定某个模型仓库产品、对象存储、CI/CD 或推理平台实现。

参考口径包括：

1. MLflow Model Registry：把 model registry 看成集中模型仓库、API 和 UI，用来管理模型完整生命周期，强调版本、alias、metadata、lineage、stage / production workflow 和 governance。
2. Hugging Face Hub Model Cards：把模型卡看成随模型一起发布的 Markdown + metadata，记录模型用途、限制、训练参数、数据集和评估结果，说明模型仓库不能只有权重 URI。
3. Safetensors / 安全权重格式：用来校准权重格式安全、manifest、checksum 和加载风险边界，避免把任意 pickle 权重直接作为生产 artifact。
4. OpenTelemetry 生成式 AI 语义约定：用来校准线上模型调用 trace 如何反查 model id、tokenizer / config、release manifest、latency、token 和错误信号。

本章的目标不是教你搭一个文件目录，而是把“checkpoint 如何变成 production model”回答成可验证链路：ModelVersion 契约完整，权重 manifest 可校验，tokenizer / config 绑定，adapter / merge / quantization 有血缘，runtime 可加载，eval / safety 报告过线，发布状态机可审计，rollback alias 可用，release manifest 可复现。

## 57.1 题目理解

面试题可能这样问：

1. 请设计一个企业级模型仓库。
2. 请设计一个支持模型版本管理、评估门禁和发布回滚的系统。
3. 请设计一个从 checkpoint 到 production model 的发布流水线。
4. 如何管理模型权重、tokenizer、adapter、量化版本和发布状态？

核心是模型资产生命周期管理。

## 57.2 需求澄清

先问：

1. 支持哪些模型类型？LLM、embedding、rerank、多模态？
2. 模型 artifact 有哪些？权重、tokenizer、config、adapter、量化版本？
3. 是否需要关联训练任务和数据血缘？
4. 是否需要评估门禁？
5. 是否支持灰度发布和 A/B 测试？
6. 是否支持一键回滚？
7. 是否多租户和权限隔离？
8. 是否需要跨地域分发？
9. 是否需要 artifact 安全校验？
10. 生命周期和存储成本如何管理？

模型仓库和发布系统既是资产系统，也是生产变更系统。

## 57.3 核心目标

系统目标：

1. 管理模型和模型版本。
2. 管理权重、tokenizer、config、adapter、量化版本。
3. 记录模型血缘。
4. 关联 eval report。
5. 支持发布状态机。
6. 支持质量门禁和审批。
7. 支持灰度、A/B 和回滚。
8. 支持权限、安全和审计。
9. 支持 artifact 校验和分发。
10. 支持生命周期和成本治理。

## 57.4 总体架构

可以设计如下：

```text
Training Platform
  -> Model Registry API
  -> Artifact Store
  -> Metadata DB
  -> Lineage Graph
  -> Eval Report Store
  -> Quality Gate
  -> Release Controller
  -> Deployment Controller
  -> Inference Platform
  -> Audit / Cost / Security
```

训练平台注册候选模型，评估平台写入评估报告，发布系统控制 staged、canary、production 和 rollback。

## 57.5 核心对象

核心对象包括：

1. Model。
2. ModelVersion。
3. ModelArtifact。
4. TokenizerVersion。
5. AdapterVersion。
6. QuantizedVersion。
7. EvalReport。
8. ReleaseManifest。
9. Deployment。
10. ApprovalRecord。
11. AuditEvent。

这些对象要有版本、状态、权限和血缘。

## 57.6 Model 和 ModelVersion

Model 是逻辑模型系列。

例如：

```text
company-chat
company-code
company-embedding
```

ModelVersion 是具体版本。

例如：

```text
company-chat:v1.2.0
company-chat:v1.3.0-sft
company-chat:v1.4.0-dpo
```

线上部署必须指向具体 ModelVersion。

不要用 latest 作为生产发布依据。

## 57.7 ModelVersion 元数据

ModelVersion 应记录：

1. model name。
2. version。
3. model type。
4. architecture。
5. parameter size。
6. context length。
7. weight artifact。
8. tokenizer version。
9. model config。
10. training job ID。
11. dataset versions。
12. code version。
13. eval reports。
14. safety level。
15. release state。
16. owner。
17. created time。

这些元数据让模型可追溯、可部署、可审计。

## 57.8 Artifact Store

Artifact Store 保存大文件：

1. 权重分片。
2. tokenizer 文件。
3. config。
4. adapter。
5. 量化权重。
6. deployment package。
7. eval report artifact。

要求：

1. 支持大文件上传下载。
2. 支持 checksum。
3. 支持权限控制。
4. 支持跨地域复制。
5. 支持冷热分层。
6. 支持生命周期管理。

Artifact Store 和 Metadata DB 要分离。

## 57.9 Artifact 校验

模型 artifact 必须校验：

1. 文件是否完整。
2. checksum 是否匹配。
3. 权重格式是否合法。
4. tokenizer 是否存在。
5. config 是否匹配。
6. runtime 是否兼容。
7. 是否通过安全扫描。
8. 是否有许可证风险。

校验失败的模型不能进入发布流程。

不要等推理部署时才发现模型缺文件。

## 57.10 模型血缘

模型仓库要记录血缘：

```text
dataset version + training config + code version
  -> training job
  -> checkpoint
  -> model version
  -> eval report
  -> release manifest
  -> deployment
```

血缘用于：

1. 复现训练。
2. 追踪数据问题影响。
3. 审计模型来源。
4. 分析质量退化。
5. 判断 artifact 是否可删除。

## 57.11 Adapter 和 Merge 管理

LoRA adapter 依赖 base model。

系统要记录：

1. adapter version。
2. base model version。
3. training job。
4. dataset version。
5. merge status。
6. eval report。
7. compatibility constraints。

Merge 后模型要记录：

```text
base model + adapter -> merged model
```

否则后续无法判断某个线上模型如何产生。

## 57.12 量化版本管理

量化版本依赖原始模型。

要记录：

1. source model version。
2. quantization method。
3. calibration dataset。
4. quantization tool version。
5. target runtime。
6. quality eval。
7. latency benchmark。
8. cost improvement。

量化版本不能只看性能，也要看质量和兼容性。

## 57.13 发布状态机

ModelVersion 状态机可以是：

```text
registered -> validated -> evaluated -> approved -> staged -> canary -> production -> deprecated -> archived
```

状态含义：

1. registered：已注册。
2. validated：artifact 校验通过。
3. evaluated：评估完成。
4. approved：审批通过。
5. staged：部署到预发。
6. canary：小流量灰度。
7. production：正式生产。
8. deprecated：不推荐使用。
9. archived：归档。

状态变更必须记录审计。

## 57.14 Quality Gate

Quality Gate 决定模型是否能发布。

门禁规则：

1. eval suite 通过。
2. safety eval 通过。
3. latency benchmark 达标。
4. cost benchmark 达标。
5. tool call success rate 达标。
6. output format 兼容。
7. 关键 slice 不退化。
8. artifact 校验通过。

没有通过门禁的模型不能进入 canary。

## 57.15 审批流程

企业发布通常需要审批。

审批人可能包括：

1. 模型负责人。
2. 业务负责人。
3. 安全负责人。
4. 平台负责人。
5. 合规负责人。

审批记录包括：

1. 审批人。
2. 时间。
3. 结论。
4. 风险说明。
5. 关联 eval report。
6. 回滚方案。

高风险模型发布不能绕过审批。

## 57.16 Release Manifest

ReleaseManifest 描述一次发布用到的完整资产。

包括：

1. model version。
2. tokenizer version。
3. weight artifact。
4. runtime image。
5. runtime config。
6. generation config。
7. safety policy。
8. route policy。
9. eval report。
10. release owner。

回滚时回滚的是 release manifest，而不只是模型权重。

## 57.17 灰度发布

灰度发布支持：

1. 按流量比例。
2. 按租户。
3. 按用户分桶。
4. 按地域。
5. 按业务场景。

灰度过程：

```text
1% -> 5% -> 10% -> 25% -> 50% -> 100%
```

每一步观察质量、安全、延迟、错误率和成本。

异常则暂停或回滚。

## 57.18 A/B 测试

A/B 测试用于比较新旧模型效果。

要支持：

1. 稳定分桶。
2. 实验配置。
3. 指标采集。
4. guardrail。
5. 统计分析。
6. 结果报告。

灰度关注风险控制，A/B 关注效果对比。

两者可以同时使用。

## 57.19 回滚设计

回滚必须快速可靠。

回滚对象包括：

1. model version。
2. tokenizer。
3. runtime config。
4. prompt。
5. route policy。
6. safety policy。
7. generation config。

如果只回滚权重，其他配置不回滚，行为可能仍然异常。

回滚前要确保旧 artifact 仍然可用。

## 57.20 Deployment Controller

Deployment Controller 负责把 release manifest 部署到推理平台。

能力包括：

1. 创建 endpoint。
2. 加载模型。
3. health check。
4. warmup。
5. 注册到 router。
6. 灰度流量调整。
7. 回滚。
8. 摘除异常实例。

部署不是文件复制，而是生产流量接入过程。

## 57.21 权限控制

模型仓库权限包括：

1. 谁能注册模型。
2. 谁能下载权重。
3. 谁能查看 eval report。
4. 谁能发布模型。
5. 谁能回滚。
6. 谁能删除 artifact。
7. 哪些租户能调用模型。

模型权重是高价值资产，必须控制下载和导出。

## 57.22 安全和审计

审计操作包括：

1. 模型注册。
2. artifact 上传。
3. 权重下载。
4. eval report 修改。
5. 审批。
6. 发布。
7. 灰度调整。
8. 回滚。
9. 删除。

安全检查包括：

1. artifact checksum。
2. 权重格式安全。
3. 镜像扫描。
4. secret 检查。
5. 许可证信息。
6. 数据敏感等级继承。

## 57.23 生命周期管理

模型 artifact 很占存储。

生命周期策略：

1. production 版本长期保留。
2. 可回滚版本保留。
3. failed candidate 定期清理。
4. 中间 checkpoint 按策略清理。
5. archived 版本转冷存储。
6. 删除前检查 deployment 和 lineage。

删除模型前必须确认没有线上依赖和合规保留要求。

## 57.24 可观测性

系统要观察：

1. model registration count。
2. artifact validation failure。
3. eval gate pass rate。
4. deployment success rate。
5. canary error rate。
6. rollback count。
7. model load latency。
8. storage growth。
9. download audit events。

发布系统的问题往往会直接影响线上推理。

## 57.25 模型仓库与发布系统设计指标和最小 demo

模型仓库与发布系统可以把一次模型资产注册和发布抽象成：

```math
M_i=(m_i,v_i,w_i,t_i,c_i,a_i,q_i,e_i,s_i,r_i,d_i,l_i,z_i)
```

其中 `m_i` 是逻辑模型，`v_i` 是 ModelVersion，`w_i` 是权重 manifest，`t_i` 是 tokenizer，`c_i` 是 config，`a_i` 是 adapter / merge / quantization 关系，`q_i` 是 eval report，`e_i` 是安全和权限证据，`s_i` 是发布状态，`r_i` 是 release manifest，`d_i` 是 deployment，`l_i` 是 lineage，`z_i` 是最终治理状态。

统一覆盖率可以写成：

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}\left[g_j(M_i)=1\right]
```

Release manifest 完整率可以写成：

```math
C_{\mathrm{manifest}}=\frac{N_{\mathrm{present}}}{N_{\mathrm{required}}}
```

权重分片 checksum 覆盖率可以写成：

```math
C_{\mathrm{checksum}}=\frac{N_{\mathrm{checked}}}{N_{\mathrm{shard}}}
```

Tokenizer / config 绑定门禁可以写成：

```math
G_{\mathrm{tokcfg}}=\mathbf{1}\left[h_{\mathrm{tok}}=h_{\mathrm{tok}}^{\mathrm{record}} \land h_{\mathrm{cfg}}=h_{\mathrm{cfg}}^{\mathrm{record}}\right]
```

量化质量退化和延迟加速比可以写成：

```math
\Delta Q=Q_{\mathrm{source}}-Q_{\mathrm{quant}},\qquad S_{\mathrm{lat}}=\frac{L_{\mathrm{source}}}{L_{\mathrm{quant}}}
```

量化版本只有在 `Delta Q` 不超过阈值且 `S_lat` 确实带来收益时，才适合进入发布候选。

灰度发布 guardrail 可以写成：

```math
G_{\mathrm{canary}}=\mathbf{1}\left[\Delta_Q\ge -\epsilon_Q \land \Delta_L\le \epsilon_L \land \Delta_E\le \epsilon_E \land P_0=0\right]
```

回滚就绪门禁可以写成：

```math
G_{\mathrm{rollback}}=\mathbf{1}\left[A_{\mathrm{rollback}}=1 \land H_{\mathrm{old}}=1 \land T_{\mathrm{rollback}}\le \tau_{\mathrm{rollback}}\right]
```

其中 `A_rollback` 表示 rollback alias 存在，`H_old` 表示旧 release manifest 和 artifact 已通过健康检查。

Artifact 血缘覆盖率可以写成：

```math
R_{\mathrm{lineage}}=\frac{|E_{\mathrm{observed}}|}{|E_{\mathrm{required}}|}
```

生命周期存储成本可以写成：

```math
K_{\mathrm{life}}=\sum_{a=1}^{A}S_aP_aT_a
```

最终模型仓库发布门禁可以写成：

```math
G_{\mathrm{model\_release}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land C_{\mathrm{checksum}}=1 \land G_{\mathrm{tokcfg}}=1 \land G_{\mathrm{canary}}=1 \land G_{\mathrm{rollback}}=1 \land P_0=0\right]
```

下面的 0 依赖 demo 把模型仓库和发布系统合成一个 toy 审计器。它输出 manifest、checksum、tokenizer / config、adapter、量化、灰度、回滚、血缘和生命周期指标，并用 16 个 bad case 模拟面试中必须覆盖的风险点。

```python
class MiniModelRegistryReleaseSystemDesignAudit:
    def __init__(self):
        self.release_manifest = {
            "model_version": "company-chat:v1.4.0-dpo",
            "tokenizer_version": "tok-2026-06",
            "weight_manifest": "weights-manifest-sha256",
            "runtime_image": "runtime@sha256:abc",
            "runtime_config": "runtime-config-v7",
            "generation_config": "gen-config-v3",
            "safety_policy": "safety-policy-v12",
            "route_policy": "route-policy-v5",
            "eval_report": "eval-report-778",
            "rollback_target": "company-chat:v1.3.2",
        }
        self.weight_shards = [
            {"name": "model-00001.safetensors", "gib": 32.0, "checksum_ok": True},
            {"name": "model-00002.safetensors", "gib": 32.0, "checksum_ok": True},
            {"name": "model-00003.safetensors", "gib": 32.0, "checksum_ok": True},
            {"name": "model-00004.safetensors", "gib": 32.0, "checksum_ok": True},
        ]
        self.tokenizer_hash = "tok-hash-7"
        self.recorded_tokenizer_hash = "tok-hash-7"
        self.config_hash = "cfg-hash-9"
        self.recorded_config_hash = "cfg-hash-9"
        self.adapter = {"rank": 16, "hidden_size": 128, "base_model_ok": True}
        self.quant = {
            "source_quality": 0.842,
            "quant_quality": 0.830,
            "source_latency_ms": 145,
            "quant_latency_ms": 100,
        }
        self.canary = {"quality_delta": 0.006, "latency_delta": -0.12, "error_delta": 0.002, "p0": 0}
        self.rollback = {"alias_ready": True, "old_health_ok": True, "rollback_seconds": 45}
        self.lineage = {"observed_edges": 14, "required_edges": 15}
        self.lifecycle = [
            {"tier": "hot", "gib": 160, "usd_per_gib_month": 0.03, "months": 2},
            {"tier": "warm", "gib": 400, "usd_per_gib_month": 0.012, "months": 6},
            {"tier": "cold", "gib": 900, "usd_per_gib_month": 0.004, "months": 12},
        ]
        self.gates = [
            "model_version_contract",
            "weight_manifest_integrity",
            "safe_weight_format",
            "tokenizer_config_binding",
            "adapter_base_compatibility",
            "merge_lineage_completeness",
            "quantization_eval_gate",
            "runtime_compatibility_fit",
            "eval_report_linkage",
            "safety_permission_gate",
            "release_status_governance",
            "rollback_alias_readiness",
            "artifact_lineage_completeness",
            "load_cache_readiness",
            "lifecycle_retention_policy",
            "model_registry_gate",
        ]

    def release_manifest_completeness(self):
        required = {
            "model_version",
            "tokenizer_version",
            "weight_manifest",
            "runtime_image",
            "runtime_config",
            "generation_config",
            "safety_policy",
            "route_policy",
            "eval_report",
            "rollback_target",
        }
        present = sum(1 for field in required if self.release_manifest.get(field))
        return round(present / len(required), 3)

    def checksum_coverage(self):
        checked = sum(1 for shard in self.weight_shards if shard["checksum_ok"])
        return round(checked / len(self.weight_shards), 3)

    def weight_total_size_gib(self):
        return round(sum(shard["gib"] for shard in self.weight_shards), 1)

    def tokenizer_config_binding(self):
        return (
            self.tokenizer_hash == self.recorded_tokenizer_hash
            and self.config_hash == self.recorded_config_hash
        )

    def adapter_rank_ratio(self):
        return round(self.adapter["rank"] / self.adapter["hidden_size"], 3)

    def quant_quality_drop(self):
        return round(self.quant["source_quality"] - self.quant["quant_quality"], 3)

    def quant_latency_speedup(self):
        return round(self.quant["source_latency_ms"] / self.quant["quant_latency_ms"], 3)

    def canary_pass(self):
        return (
            self.canary["quality_delta"] >= -0.01
            and self.canary["latency_delta"] <= 0.05
            and self.canary["error_delta"] <= 0.01
            and self.canary["p0"] == 0
        )

    def rollback_ready(self):
        return (
            self.rollback["alias_ready"]
            and self.rollback["old_health_ok"]
            and self.rollback["rollback_seconds"] <= 60
        )

    def lineage_coverage(self):
        return round(self.lineage["observed_edges"] / self.lineage["required_edges"], 3)

    def lifecycle_monthly_cost(self):
        cost = 0.0
        for item in self.lifecycle:
            cost += item["gib"] * item["usd_per_gib_month"] * item["months"]
        return round(cost, 2)

    def registry_examples(self):
        return {
            "release_manifest_completeness": self.release_manifest_completeness(),
            "checksum_coverage": self.checksum_coverage(),
            "weight_total_size_gib": self.weight_total_size_gib(),
            "tokenizer_config_binding": self.tokenizer_config_binding(),
            "adapter_rank_ratio": self.adapter_rank_ratio(),
            "quant_quality_drop": self.quant_quality_drop(),
            "quant_latency_speedup": self.quant_latency_speedup(),
            "canary_pass": self.canary_pass(),
            "rollback_ready": self.rollback_ready(),
            "lineage_coverage": self.lineage_coverage(),
            "lifecycle_cost": self.lifecycle_monthly_cost(),
        }

    def design_cases(self):
        complete = {gate: True for gate in self.gates}
        bad_specs = [
            ("model_version_contract_missing_bad", "model_version_contract"),
            ("weight_manifest_checksum_missing_bad", "weight_manifest_integrity"),
            ("unsafe_weight_format_bad", "safe_weight_format"),
            ("tokenizer_config_mismatch_bad", "tokenizer_config_binding"),
            ("adapter_base_mismatch_bad", "adapter_base_compatibility"),
            ("merge_lineage_missing_bad", "merge_lineage_completeness"),
            ("quantization_quality_drop_bad", "quantization_eval_gate"),
            ("runtime_incompatible_bad", "runtime_compatibility_fit"),
            ("eval_report_missing_bad", "eval_report_linkage"),

    def evaluate_cases(self):
        cases = self.design_cases()
        metrics = {}
        failed_cases = []
        failed_gates = set()
        for gate in self.gates:
            passed = sum(1 for case in cases if case["checks"][gate])
            metrics[gate] = round(passed / len(cases), 3)
        for case in cases:
            missing = [gate for gate, ok in case["checks"].items() if not ok]
            if missing:
                failed_cases.append(case["name"])
                failed_gates.update(missing)
        return {
            "metrics": metrics,
            "failed_cases": failed_cases,
            "failed_gates": sorted(failed_gates),
            "hard_blocker_count": len(failed_cases),
            "gate_pass": len(failed_cases) == 0,
        }
```

## 57.26 核心 trade-off

Trade-off 包括：

1. 发布速度 vs 审批安全。
2. 模型版本保留 vs 存储成本。
3. 评估门禁严格 vs 迭代效率。
4. 权重下载便利 vs 模型资产安全。
5. 灰度慢 vs 新模型收益延迟。
6. 自动回滚 vs 指标波动误判。

企业平台要根据风险分层管理。

## 57.28 常见扣分点

扣分点一：模型仓库就是对象存储。

问题：忽略元数据、版本、血缘、评估、权限和发布状态。

扣分点二：只管理权重，不管理 tokenizer。

问题：tokenizer 不匹配会导致严重线上问题。

扣分点三：不提评估门禁。

问题：模型发布必须基于质量、安全、延迟和成本判断。

扣分点四：回滚只回滚模型文件。

问题：还要回滚 runtime、prompt、route 和 safety policy。

扣分点五：不提生命周期。

问题：模型和 checkpoint 存储成本非常高。

## 57.30 本章小结

本章系统设计了模型仓库与发布系统。

你需要记住：

1. 模型仓库管理的不只是权重，还有 tokenizer、config、adapter、量化版本、eval report、权限、血缘和发布状态。
2. 发布系统要通过 Quality Gate、审批、灰度、A/B 和回滚控制线上风险。
3. ReleaseManifest 是可复现发布和可靠回滚的关键。
4. 模型 artifact 要做 checksum、安全校验、runtime 兼容性检查和生命周期管理。
5. 企业级模型发布必须可审计、可追溯、可回滚、可治理。
6. 第二轮新增的公式和 demo 可以把 manifest 完整率、checksum 覆盖率、tokenizer / config 绑定、量化退化、灰度 guardrail、回滚就绪、血缘覆盖和生命周期成本统一成可验收的发布门禁。
