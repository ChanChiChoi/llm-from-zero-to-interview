# 第 19 章 训练配置管理：超参、版本、环境和可复现性

上一章讲了分布式训练启动器。本章讲训练平台里另一个容易被低估的问题：配置管理。

大模型训练配置非常多：模型结构、数据版本、batch size、learning rate、optimizer、precision、并行策略、checkpoint、评估频率、随机种子、环境变量。任何一个字段变化，都可能导致训练结果不同。

先记住一句话：

> 训练配置管理的核心不是把参数写进 YAML，而是记录、校验、版本化和冻结一次训练真正生效的全部条件。

## 19.0 本讲资料边界与第二轮精修口径

本讲第二轮精修时，按官方资料校准以下边界：

1. Hydra / OmegaConf 的稳定口径是配置组合、覆盖、结构化配置和命令行 override；本章借用它们的思想讲“来源、覆盖顺序和最终配置”，不绑定某个具体框架。
2. MLflow Tracking 的稳定口径是把一次 run 的 parameters、metrics、artifacts、code version 和 tags 等记录下来；本章借用它讲实验追踪和复现证据。
3. PyTorch reproducibility 文档强调，随机种子、确定性算法和环境记录能提升复现性，但跨平台、跨版本或不同算子路径不保证完全 bitwise 一致；本章因此区分“严格复现”和“统计复现”。
4. Pydantic / JSON Schema / typed config 的稳定口径是结构化校验和类型约束；本章只抽象 schema、字段类型、枚举、范围和跨字段一致性，不要求读者必须使用某个库。
5. 配置管理的工程目标不是保存一个 YAML 文件，而是冻结最终生效配置、记录字段来源、绑定代码 / 数据 / 镜像 / 环境 / 随机种子 / checkpoint / 实验追踪，并支持 diff、审批、恢复和发布。

因此，本章不把某个配置中心、实验平台、内部 SDK 或云厂商参数写成通用标准。这里讨论的是训练平台最稳定的抽象：配置来源、覆盖顺序、最终快照、版本绑定、schema 校验、跨字段一致性、资源策略、随机性边界、resume 兼容、发布耦合和审计门禁。

## 19.1 为什么训练配置管理重要

没有配置管理时，团队经常遇到这些问题：

1. 不知道某个模型是怎么训出来的。
2. 同一个配置重跑结果不一样。
3. 配置文件被改了，历史实验无法复现。
4. 命令行覆盖了 YAML，但没人记录。
5. 默认参数变化导致结果变化。
6. 训练代码和配置不兼容。
7. 分布式并行参数写错。
8. checkpoint 恢复时配置不一致。

配置管理的目标是让每次训练都能回答：

1. 用了什么配置？
2. 配置从哪里来？
3. 哪些字段被覆盖了？
4. 最终生效值是什么？
5. 配置和代码、数据、环境是否匹配？
6. 以后能否复现？

## 19.2 配置包含哪些内容

大模型训练配置可以分成十类。

### 19.2.1 模型配置

包括：

1. model_type。
2. hidden_size。
3. num_layers。
4. num_attention_heads。
5. vocab_size。
6. max_position_embeddings。
7. rope 配置。
8. activation。
9. normalization。
10. dropout。

模型配置决定模型结构。

如果模型结构和 checkpoint 不匹配，加载会失败或产生错误。

### 19.2.2 数据配置

包括：

1. dataset_id。
2. dataset_version。
3. train / eval split。
4. data sampling weight。
5. preprocessing_version。
6. tokenizer_version。
7. packing 策略。
8. max sequence length。

数据配置变化经常比模型配置变化更影响效果。

### 19.2.3 优化器配置

包括：

1. optimizer type。
2. learning rate。
3. beta1 / beta2。
4. weight decay。
5. epsilon。
6. gradient clipping。
7. warmup steps。
8. scheduler type。
9. min learning rate。

优化器配置影响收敛和稳定性。

### 19.2.4 训练批次配置

包括：

1. micro batch size。
2. global batch size。
3. gradient accumulation steps。
4. sequence length。
5. tokens per batch。
6. drop last。

这些字段要彼此一致。

例如：

```math
B_{\mathrm{global}}=B_{\mathrm{micro}}A_{\mathrm{grad}}D_{\mathrm{dp}}
```

其中，`B_micro` 是每个 data-parallel rank 上的 micro batch size，`A_grad` 是 gradient accumulation steps，`D_dp` 是 data parallel size。这个公式不只是数学关系，也是配置校验规则：如果用户声明的 global batch 和平台计算出的 global batch 不一致，学习率、吞吐、显存和 loss 曲线解释都会被误导。

### 19.2.5 精度配置

包括：

1. FP32。
2. FP16。
3. BF16。
4. FP8。
5. loss scaling。
6. mixed precision policy。

精度配置影响速度、显存和数值稳定性。

### 19.2.6 并行配置

包括：

1. data parallel size。
2. tensor parallel size。
3. pipeline parallel size。
4. sequence parallel。
5. ZeRO stage。
6. FSDP policy。
7. activation checkpointing。

并行配置要和 GPU 数量、模型结构、启动器配置匹配。

### 19.2.7 Checkpoint 配置

包括：

1. save_every_steps。
2. keep_last_n。
3. save_best。
4. async_save。
5. resume_from。
6. checkpoint_format。
7. shard_strategy。

Checkpoint 配置影响可靠性、成本和恢复能力。

### 19.2.8 评估配置

包括：

1. eval_every_steps。
2. eval_dataset_version。
3. metric list。
4. generation config。
5. judge model。
6. eval batch size。

评估配置决定指标是否可比较。

### 19.2.9 环境配置

包括：

1. image digest。
2. CUDA version。
3. PyTorch version。
4. NCCL version。
5. Python version。
6. environment variables。
7. driver version。

环境配置是复现训练的基础。

### 19.2.10 平台配置

包括：

1. queue。
2. priority。
3. resource profile。
4. node selector。
5. storage mount。
6. logging policy。
7. retry policy。
8. timeout。

平台配置决定任务如何运行在基础设施上。

## 19.3 配置来源

训练配置可能来自多个地方：

1. 默认值。
2. 基础 YAML。
3. 任务模板。
4. 命令行参数。
5. 环境变量。
6. 配置中心。
7. 代码内部默认值。
8. 平台自动注入字段。

这会带来一个问题：最终到底哪个值生效？

例如 learning rate 可能在 YAML 里是 `1e-5`，命令行覆盖成 `2e-5`，代码默认是 `3e-5`。

如果平台只记录 YAML，就会误导复现。

## 19.4 配置覆盖顺序

平台应该明确配置覆盖顺序。

例如：

```text
系统默认值
  < 基础模板
  < 项目模板
  < 用户 YAML
  < 命令行 override
  < 平台强制策略
```

平台强制策略可能包括：

1. 禁止使用未审核数据。
2. 强制 checkpoint。
3. 强制日志采集。
4. 强制安全环境变量。
5. 限制最大运行时长。

覆盖顺序必须可解释，否则用户不知道为什么配置变了。

## 19.5 最终生效配置

最终生效配置是训练真正使用的配置快照。

它应该在任务启动前生成并冻结。

包含：

1. 合并后的所有字段。
2. 每个字段来源。
3. 被覆盖字段记录。
4. 平台注入字段。
5. 配置 hash。
6. 生成时间。

示例：

```json
{
  "learning_rate": {
    "value": 0.00002,
    "source": "cli_override"
  },
  "global_batch_size": {
    "value": 512,
    "source": "user_yaml"
  },
  "checkpoint.save_every_steps": {
    "value": 1000,
    "source": "platform_policy"
  }
}
```

复现实验时，应使用最终生效配置，而不是原始配置文件。

抽象地说，最终生效配置可以写成：

```math
c^{\star}=M(c_{\mathrm{default}},c_{\mathrm{template}},c_{\mathrm{yaml}},c_{\mathrm{cli}},c_{\mathrm{env}},c_{\mathrm{policy}})
```

其中，`M` 是平台定义的 merge / override 函数。配置管理真正要记录的不是某一个输入文件，而是 `c_star` 的每个字段最终值、字段来源和生成时使用的 merge 规则版本。

## 19.6 配置版本化

配置也应该版本化。

版本化对象包括：

1. 训练模板。
2. YAML 配置文件。
3. 最终生效配置。
4. 默认参数集合。
5. 平台策略版本。

为什么平台策略版本也要记录？

因为平台可能自动注入或修改配置。

例如某天平台把默认 checkpoint 频率从 1000 step 改成 500 step，如果不记录策略版本，历史任务就难以解释。

## 19.7 配置校验

配置校验分为几类。

### 19.7.1 Schema 校验

检查字段类型和必填项。

例如：

1. learning_rate 必须是正数。
2. gpu_count 必须是整数。
3. precision 只能是允许值。
4. dataset_version 不能为空。

### 19.7.2 一致性校验

检查字段之间是否一致。

例如：

1. global_batch_size 是否等于 micro_batch * accumulation * data_parallel。
2. world_size 是否等于 GPU 总数。
3. tensor_parallel * pipeline_parallel 是否能整除 world_size。
4. checkpoint resume 是否和模型结构匹配。

### 19.7.3 资源校验

检查配置是否可能跑得动。

例如：

1. sequence length 太长是否会 OOM。
2. batch size 是否超过显存。
3. GPU 型号是否支持 BF16 / FP8。
4. 资源请求是否满足并行配置。

### 19.7.4 策略校验

检查是否符合平台规则。

例如：

1. 高敏数据是否允许训练。
2. 未审核镜像是否禁止使用。
3. 可抢占任务是否配置 checkpoint。
4. 高优先级是否需要审批。

## 19.8 超参搜索和配置管理

超参搜索会产生大量配置变体。

常见超参：

1. learning rate。
2. batch size。
3. warmup ratio。
4. weight decay。
5. dropout。
6. LoRA rank。
7. sequence length。
8. sampling ratio。

平台需要支持：

1. 配置模板。
2. 参数 sweep。
3. 实验分组。
4. 结果对比。
5. 失败任务过滤。
6. 最优配置记录。

超参搜索不能只生成一堆任务，还要能追踪每个任务对应的配置差异。

## 19.9 随机种子和可复现性

复现训练通常需要记录随机种子。

包括：

1. Python random seed。
2. NumPy seed。
3. PyTorch seed。
4. CUDA seed。
5. data shuffle seed。
6. dropout seed。

但记录 seed 不代表完全可复现。

原因：

1. 某些 GPU 算子非确定性。
2. 多机通信顺序可能变化。
3. 数据加载 worker 顺序可能变化。
4. 混合精度数值误差。
5. checkpoint 恢复位置差异。

所以要区分严格可复现和统计可复现。

严格可复现要求更高，成本也更高。

## 19.10 环境版本记录

配置管理不能只记录训练参数，还要记录环境。

至少包括：

1. image digest。
2. code commit。
3. Python version。
4. CUDA version。
5. PyTorch version。
6. NCCL version。
7. driver version。
8. GPU type。
9. launcher version。
10. platform version。

环境变化可能导致：

1. 性能变化。
2. 数值结果变化。
3. kernel 行为变化。
4. 通信行为变化。
5. checkpoint 兼容性变化。

所以环境版本是配置管理的一部分。

## 19.11 配置和 checkpoint 恢复

恢复 checkpoint 时，配置必须兼容。

需要检查：

1. 模型结构是否一致。
2. tokenizer 是否一致。
3. 并行策略是否兼容。
4. optimizer 配置是否一致。
5. scheduler 状态是否匹配。
6. precision 是否兼容。
7. dataset position 是否可恢复。

有些字段可以变化，例如日志路径。

有些字段不能随便变，例如 hidden size、layer 数、vocab size。

平台应该把配置字段分成：

1. immutable。
2. resumable-compatible。
3. runtime-only。

这样恢复时才能做自动校验。

## 19.12 配置和模型发布

模型发布也需要配置。

发布模型时要知道：

1. 模型结构。
2. tokenizer。
3. generation config。
4. safety config。
5. quantization config。
6. serving config。
7. eval config。
8. 训练数据 lineage。

训练配置和推理配置不完全一样，但它们有关联。

如果训练时使用了特殊 tokenizer 或 RoPE scaling，推理时必须匹配。

否则模型加载成功，也可能输出异常。

## 19.13 配置权限和审批

有些配置不是任何人都能改。

例如：

1. 使用高敏数据。
2. 使用生产资源池。
3. 申请高优先级。
4. 关闭日志采集。
5. 禁用 checkpoint。
6. 使用未审核镜像。
7. 调大最大运行时间。

这些配置应该触发策略检查或审批。

配置管理和权限系统要结合，而不是只做格式校验。

## 19.14 配置 diff 和实验对比

实验对比时，最重要的是知道差异在哪里。

平台应该支持配置 diff。

例如比较两个实验：

```text
learning_rate: 1e-5 -> 2e-5
global_batch_size: 256 -> 512
dataset_version: v3 -> v4
precision: fp16 -> bf16
```

配置 diff 可以帮助判断效果变化来自哪里。

没有配置 diff，实验对比会变成猜测。

## 19.15 常见配置事故

### 19.15.1 使用了错误数据版本

表现：效果突然变化，复现失败。

原因：使用了 latest 路径，数据被覆盖。

防线：强制 dataset_version。

### 19.15.2 命令行覆盖没人记录

表现：YAML 看起来一样，但结果不同。

原因：启动命令中加了 override。

防线：记录最终生效配置。

### 19.15.3 batch size 算错

表现：loss 不稳定或吞吐异常。

原因：micro batch、accumulation、data parallel 不一致。

防线：一致性校验。

### 19.15.4 恢复 checkpoint 后效果异常

表现：恢复后 loss 跳变。

原因：optimizer、scheduler、seed、数据位置或配置不一致。

防线：恢复兼容性校验。

## 19.16 训练配置管理审计指标与最小 demo

配置管理在平台里应该是一个审计对象，而不是一个文件上传功能。一个配置审计样本可以写成：

```math
q_i=(s_i,m_i,d_i,o_i,b_i,p_i,k_i,e_i,r_i,h_i,a_i,z_i)
```

其中，`s_i` 是配置来源和覆盖顺序，`m_i` 是模型配置，`d_i` 是数据配置，`o_i` 是优化器和 scheduler，`b_i` 是 batch / 并行配置，`p_i` 是 precision 和资源约束，`k_i` 是 checkpoint / resume，`e_i` 是环境版本，`r_i` 是随机性和复现边界，`h_i` 是超参 sweep 和配置 diff，`a_i` 是权限审批，`z_i` 是审计 trace。

第 `j` 个配置管理维度的覆盖率可以写成：

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(q_i)=1]
```

必填字段覆盖率可以写成：

```math
R_{\mathrm{required}}=\frac{K_{\mathrm{present}}}{K_{\mathrm{required}}}
```

可复现证据覆盖率可以写成：

```math
R_{\mathrm{repr}}=\frac{K_{\mathrm{captured}}}{K_{\mathrm{required}}}
```

checkpoint 恢复兼容性可以抽象为：

```math
G_{\mathrm{resume}}=\mathbf{1}\left[\forall k\in I_{\mathrm{immutable}},\ c^{\star}_{k}=c^{\mathrm{ckpt}}_{k}\right]
```

其中，`I_immutable` 是恢复时不能变化的字段集合，例如模型结构、tokenizer、vocab、并行切分、optimizer / scheduler 状态和数据游标等。训练配置门禁可以写成：

```math
G_{\mathrm{config}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{required}}\ge \gamma \land R_{\mathrm{repr}}\ge \eta \land G_{\mathrm{resume}}=1 \land P_0=0\right]
```

## 19.18 常见误区

误区一：配置管理就是保存 YAML。

YAML 只是来源之一，真正重要的是最终生效配置和字段来源。

误区二：记录随机种子就能完全复现。

多 GPU、非确定性算子、通信顺序和数据加载都可能影响严格复现。

误区三：默认参数不用记录。

默认参数变化也会影响结果，必须记录最终值。

误区四：环境不是配置。

镜像、CUDA、PyTorch、NCCL、驱动和 GPU 型号都会影响结果和性能。

误区五：配置自由度越高越好。

高风险配置需要策略校验和审批。

## 19.19 面试题

### 题 1：为什么只保存 YAML 不够？

答：配置可能来自默认值、模板、YAML、命令行、环境变量和平台策略。YAML 不一定等于最终生效配置。复现需要记录合并后的最终配置和每个字段来源。

### 题 2：训练配置应该包含哪些类型？

答：包括模型配置、数据配置、优化器配置、batch 配置、精度配置、并行配置、checkpoint 配置、评估配置、环境配置和平台配置。

### 题 3：配置校验有哪些层次？

答：包括 schema 校验、一致性校验、资源校验和策略校验。比如字段类型是否合法、batch size 是否一致、显存是否可能够用、是否有权限使用数据。

### 题 4：恢复 checkpoint 时为什么要校验配置？

答：如果模型结构、tokenizer、并行策略、optimizer、scheduler、precision 或数据位置不兼容，恢复后可能加载失败、loss 异常或训练不可复现。

### 题 5：配置 diff 有什么价值？

答：配置 diff 能明确两个实验的差异，帮助判断效果变化来自 learning rate、batch size、数据版本、precision 还是其他配置，避免凭感觉分析。

## 19.21 本章小结

本章讲了训练配置管理。

你需要掌握：

1. 配置管理的核心是记录、校验、版本化和冻结最终生效配置。
2. 训练配置包括模型、数据、优化器、batch、精度、并行、checkpoint、评估、环境和平台配置。
