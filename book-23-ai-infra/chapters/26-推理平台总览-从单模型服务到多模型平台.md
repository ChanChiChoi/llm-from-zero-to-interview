# 第 26 章 推理平台总览：从单模型服务到多模型平台

前面第三部分讲完了大模型训练平台。从本章开始进入第四部分：大模型推理平台工程。

训练平台解决“模型怎么训出来”，推理平台解决“模型怎么稳定、低延迟、低成本地服务线上请求”。推理平台和训练平台同样重要，因为模型效果再好，如果推理延迟高、吞吐低、成本失控、版本不可回滚，也无法真正落地。

先记住一句话：

> 大模型推理平台不是把模型包成一个 HTTP 服务，而是围绕延迟、吞吐、并发、成本、版本、安全和可用性构建的在线模型服务系统。

## 26.1 从单模型服务开始

最简单的推理服务长这样：

```text
Client -> HTTP API -> Model Server -> GPU -> Response
```

一个模型，一个服务，一个 GPU 或一组 GPU。

这种方式适合：

1. Demo。
2. 小规模内部服务。
3. 单一模型场景。
4. 流量很低的实验。

但生产环境很快会遇到问题：

1. 多个模型如何部署？
2. 请求如何路由到合适模型？
3. 高峰流量如何扩容？
4. 延迟如何控制？
5. GPU 利用率如何提高？
6. 模型版本如何灰度和回滚？
7. 成本如何统计？
8. 某个模型故障如何降级？

这就需要推理平台。

## 26.2 推理平台要解决什么问题

推理平台要解决十个核心问题：

1. 模型部署。
2. 请求接入。
3. 模型路由。
4. 推理运行时。
5. batching 和队列调度。
6. KV cache 管理。
7. 自动扩缩容。
8. 限流、熔断、超时和降级。
9. 灰度、A/B 测试和回滚。
10. 监控、安全和成本治理。

这些问题组合起来，才是完整推理平台。

## 26.3 推理和训练的差异

训练关注：

1. 长时间运行。
2. 吞吐。
3. checkpoint。
4. 分布式通信。
5. 收敛稳定性。
6. 实验可复现。

推理关注：

1. 在线延迟。
2. QPS。
3. 并发。
4. 可用性。
5. 弹性扩缩容。
6. 成本。
7. 版本发布。
8. 用户体验。

训练任务可以排队几小时，线上推理请求不行。

训练失败可以从 checkpoint 恢复，推理服务故障要快速切流或降级。

所以推理平台是在线服务系统，不是训练平台的附属品。

## 26.4 推理平台总体架构

一个典型推理平台架构：

```text
Client / App / Agent
  -> API Gateway
  -> Auth / Rate Limit / Safety
  -> Model Router
  -> Request Queue / Scheduler
  -> Inference Runtime
      -> Model Worker Pods
      -> GPU / KV Cache / Batching
  -> Model Registry
  -> Cache Layer
  -> Observability: Metrics / Logs / Traces
  -> Autoscaler
  -> Deployment Controller
  -> Cost / Audit
```

模块说明：

1. API Gateway：统一入口、认证、限流。
2. Safety：输入输出安全策略。
3. Model Router：选择模型、版本和实例。
4. Request Queue：请求排队和 batching。
5. Inference Runtime：执行 prefill、decode 和生成。
6. Model Registry：管理模型版本、权重、tokenizer 和配置。
7. Cache Layer：prompt cache、KV cache、语义缓存和结果缓存。
8. Observability：TTFT、TPOT、QPS、错误率、GPU、成本。
9. Autoscaler：自动扩缩容。
10. Deployment Controller：灰度、回滚、A/B 测试。
11. Cost / Audit：成本和访问审计。

## 26.5 推理平台的核心指标

推理平台常见指标：

1. QPS。
2. 并发数。
3. TTFT。
4. TPOT。
5. end-to-end latency。
6. p50 / p95 / p99 latency。
7. input tokens/s。
8. output tokens/s。
9. GPU utilization。
10. KV cache utilization。
11. queue length。
12. error rate。
13. timeout rate。
14. cost per 1k tokens。

TTFT 是 Time To First Token。

TPOT 是 Time Per Output Token。

这两个指标非常重要。用户体感通常受 TTFT 和后续 token 生成速度共同影响。

## 26.6 Prefill 和 Decode

大模型推理可以粗略分为两个阶段。

Prefill：处理输入 prompt，计算所有输入 token 的上下文表示，生成 KV cache。

Decode：逐 token 生成输出，每一步使用已有 KV cache 生成下一个 token。

两阶段资源画像不同：

1. Prefill 更偏计算密集。
2. Decode 更偏显存带宽和 KV cache 访问。
3. 长 prompt 会拉高 prefill 成本。
4. 长输出会拉高 decode 成本。

推理平台必须理解 prefill 和 decode，否则无法解释延迟和吞吐。

## 26.7 KV Cache 为什么关键

KV cache 保存历史 token 的 Key 和 Value。它能避免每次生成 token 都重新计算全部上下文。

但 KV cache 会占用大量显存，尤其在：

1. 长上下文。
2. 高并发。
3. 大 batch。
4. 多层大模型。
5. 多用户长对话。

KV cache 管不好，会导致：

1. 显存不足。
2. 请求被驱逐。
3. 延迟抖动。
4. GPU 利用率下降。
5. p99 延迟变差。

因此现代推理 runtime 会重点优化 KV cache，例如分页管理、复用、驱逐、前缀缓存和分离式 prefill/decode。

## 26.8 从单模型服务到多模型平台

单模型服务只需要回答：这个模型能不能跑起来？

多模型平台要回答更多问题：

1. 哪个业务能调用哪些模型？
2. 哪个模型版本是默认版本？
3. 哪些请求走大模型，哪些请求走小模型？
4. 哪些请求需要更强模型，哪些请求可以降级？
5. 模型服务异常时如何切换？
6. 不同模型的成本如何归因？
7. 模型更新是否影响线上质量？

多模型平台的核心抽象通常包括：

1. Model。
2. ModelVersion。
3. Endpoint。
4. Deployment。
5. Runtime。
6. RoutePolicy。
7. ScalingPolicy。
8. QuotaPolicy。
9. SafetyPolicy。
10. CostPolicy。

这些抽象让平台从“部署一个服务”升级为“管理一组模型能力”。

## 26.9 Model Registry 的作用

Model Registry 管理模型资产。

它通常记录：

1. 模型名称。
2. 模型版本。
3. 权重路径。
4. tokenizer。
5. 推理配置。
6. 上下文长度。
7. 支持的任务类型。
8. 量化方式。
9. 依赖 runtime。
10. 安全等级。
11. 评估结果。
12. 发布状态。

没有 Model Registry，推理平台很容易变成一堆没人说得清的服务。

## 26.10 Model Router 的作用

Model Router 负责决定请求去哪。

常见路由依据：

1. 模型名称。
2. 模型版本。
3. 用户或租户。
4. 业务线。
5. 请求类型。
6. prompt 长度。
7. 成本预算。
8. 延迟 SLO。
9. 安全策略。
10. 当前实例负载。

路由不是简单的负载均衡。大模型请求之间差异巨大，一个 200 token 请求和一个 100k token 请求不是同一种负载。

## 26.11 Runtime 的作用

Inference Runtime 是真正执行推理的组件。

常见 runtime 包括：

1. Triton。
2. vLLM。
3. TGI。
4. SGLang。
5. TensorRT-LLM。
6. 自研 runtime。

Runtime 负责：

1. 加载模型权重。
2. 管理显存。
3. 处理 tokenizer。
4. 执行 prefill。
5. 执行 decode。
6. 管理 KV cache。
7. 支持 batching。
8. 支持 streaming response。
9. 暴露指标。

平台不要把所有逻辑塞进 runtime。更合理的方式是：平台负责治理和编排，runtime 负责高性能执行。

## 26.12 Batching 为什么重要

大模型推理如果一个请求一个请求跑，GPU 利用率通常很低。

Batching 把多个请求合并执行，可以提高吞吐。

但 batching 会带来延迟权衡：

1. batch 越大，吞吐越高。
2. 等待 batch 的时间越长，TTFT 可能越差。
3. 请求长度差异越大，padding 或调度浪费越明显。
4. 长请求可能拖慢短请求。

所以推理平台常常需要 continuous batching，而不是静态 batch。

Continuous batching 允许新请求在 decode 过程中动态加入，也允许完成的请求动态退出，更适合生成式模型。

## 26.13 自动扩缩容不是只看 GPU 利用率

传统服务扩缩容经常看 CPU 或 QPS。

大模型推理不够简单。

常见扩缩容信号包括：

1. QPS。
2. 并发请求数。
3. 队列长度。
4. TTFT。
5. p95 / p99 延迟。
6. output tokens/s。
7. GPU 利用率。
8. 显存占用。
9. KV cache 占用。
10. 失败率和超时率。

只看 GPU 利用率可能误判。比如 decode 阶段可能 GPU 利用率不高，但显存带宽或 KV cache 已经成为瓶颈。

## 26.14 冷启动和热启动

大模型服务冷启动很慢，因为需要：

1. 拉取镜像。
2. 加载模型权重。
3. 初始化 runtime。
4. 分配显存。
5. 加载 tokenizer。
6. 预热 kernel。

冷启动可能耗时几十秒到数分钟。

常见优化方法：

1. 权重本地缓存。
2. 镜像预拉取。
3. 常用模型保留 warm pool。
4. runtime 预热。
5. 按业务峰谷提前扩容。
6. 权重格式优化。

推理平台的扩缩容不能只考虑“要不要扩”，还要考虑“扩出来多久能服务”。

## 26.15 灰度、A/B 测试和回滚

模型上线比普通服务上线更复杂。

因为模型版本变化会影响：

1. 输出质量。
2. 风格。
3. 安全性。
4. 延迟。
5. 成本。
6. 工具调用行为。
7. 下游业务指标。

推理平台需要支持：

1. 按流量比例灰度。
2. 按租户灰度。
3. 按用户分桶。
4. 按场景灰度。
5. 指标对比。
6. 一键回滚。

没有灰度和回滚，模型发布风险会非常高。

## 26.16 限流、熔断、超时和降级

推理平台必须保护自己。

常见保护机制：

1. 租户级限流。
2. 模型级限流。
3. endpoint 级限流。
4. 最大上下文长度限制。
5. 最大输出长度限制。
6. 排队超时。
7. 请求执行超时。
8. 熔断异常模型。
9. 降级到小模型。
10. 降级到缓存结果。

大模型推理成本高，一个异常客户端可能迅速打满 GPU 资源。所以限流和配额不是附加功能，而是平台基本能力。

## 26.17 安全和治理

推理平台安全至少包括：

1. 身份认证。
2. 租户鉴权。
3. 模型访问控制。
4. 输入内容策略。
5. 输出内容策略。
6. 敏感信息保护。
7. 日志脱敏。
8. 密钥保护。
9. 审计追踪。
10. 数据留存策略。

推理请求可能包含用户隐私、企业知识、代码、合同、财务数据和业务决策信息。平台必须明确哪些数据可以记录，哪些数据不能记录，哪些数据需要脱敏。

## 26.18 成本治理

推理成本通常由这些因素决定：

1. 模型大小。
2. 输入 token 数。
3. 输出 token 数。
4. 并发量。
5. GPU 单价。
6. GPU 利用率。
7. cache 命中率。
8. batch 效率。
9. 量化方式。
10. SLO 要求。

成本治理常见手段：

1. 小模型优先。
2. 分层模型路由。
3. 语义缓存。
4. prompt 压缩。
5. 输出长度控制。
6. 低优先级请求降级。
7. 离峰批处理。
8. endpoint 成本归因。

一个成熟推理平台必须能回答：哪个业务、哪个模型、哪个版本、哪类请求消耗了多少成本。

## 26.19 一个完整请求流程

一个线上请求的完整流程可能是：

1. Client 发起请求。
2. API Gateway 做认证和基础限流。
3. Safety 模块检查输入策略。
4. Router 根据模型、租户、成本和负载选择 endpoint。
5. Scheduler 将请求放入队列。
6. Runtime 将请求加入 continuous batching。
7. Worker 执行 prefill。
8. Worker 生成 KV cache。
9. Worker 执行 decode。
10. 服务端以 streaming 方式返回 token。
11. Safety 模块检查输出策略。
12. Metrics 记录 TTFT、TPOT、tokens 和错误码。
13. Trace 记录路由、排队、runtime 和返回阶段耗时。
14. Cost 模块做成本归因。
15. Audit 模块记录访问事件。

这才是线上推理平台的一次完整服务链路。

## 26.20 常见误区

误区一：推理平台就是 K8s 部署模型服务。

K8s 只是底座。推理平台还需要路由、runtime、batching、cache、扩缩容、灰度、限流、观测和成本治理。

误区二：QPS 越高越好。

推理平台不能只看 QPS，还要看 token 吞吐、TTFT、TPOT、p99 和成本。

误区三：GPU 利用率高就是好。

GPU 利用率高但 p99 延迟崩掉，也不是好系统。在线服务要在吞吐和延迟之间平衡。

误区四：大模型统一上最大模型。

很多请求不需要最大模型。分层路由、小模型优先和按需升级是成本治理的关键。

误区五：灰度只看错误率。

模型灰度还要看质量、安全、延迟、成本和业务指标。

## 26.21 面试回答模板

如果面试官问：“你怎么设计一个大模型推理平台？”

可以这样回答：

```text
我会先澄清业务需求，包括模型数量、QPS、上下文长度、延迟 SLO、流式返回、租户隔离和成本目标。

架构上，我会把平台拆成 API Gateway、鉴权限流、Model Router、Inference Runtime、Model Registry、缓存层、自动扩缩容、发布系统、观测系统和成本审计系统。

请求进入后先经过认证和限流，再由 Router 根据模型版本、租户、负载、SLO 和成本选择 endpoint。Runtime 负责模型加载、prefill、decode、KV cache 和 continuous batching。平台侧记录 TTFT、TPOT、p95/p99、tokens/s、队列长度、GPU 和 KV cache 使用率。

发布上支持灰度、A/B 测试和回滚。稳定性上支持超时、熔断、降级和缓存。成本上按租户、模型、版本和 token 用量做归因，并通过分层模型路由、缓存、量化和自动扩缩容降低成本。
```

这个回答覆盖了需求、架构、关键指标、稳定性和成本，比较完整。

## 26.23 本章小结

本章从单模型服务讲到多模型推理平台。

你需要记住：

1. 推理平台是在线服务系统，不只是模型 HTTP 包装。
2. 推理平台核心目标是低延迟、高吞吐、高可用和低成本。
3. TTFT、TPOT、p95/p99、tokens/s、队列长度和成本是关键指标。
4. Prefill、decode 和 KV cache 是理解大模型推理性能的基础。
5. 多模型平台需要 Model Registry、Model Router、Runtime、缓存、扩缩容、灰度和治理能力。
6. 成熟推理平台必须同时管理性能、稳定性、安全和成本。
