# 第 13 章 训练任务调度：队列、优先级、抢占、公平性和配额

上一章讲了 Kubernetes 与 GPU 资源管理。Kubernetes 能把 Pod 调度到节点上，但大模型训练平台不能只依赖默认调度。真实场景里，GPU 贵、任务多、团队多、优先级不同、训练时长不确定、大任务需要成组资源、小任务容易造成碎片。

训练任务调度就是解决这些问题的核心系统。

先记住一句话：

> 大模型训练调度的目标不是简单“谁先来谁先跑”，而是在稀缺 GPU 资源上平衡吞吐、等待时间、公平性、优先级、成本和任务成功率。

## 13.0 本讲资料边界与第二轮精修口径

本讲第二轮精修时，资料口径按“大模型训练调度治理的稳定抽象”处理，而不是绑定某个 Kubernetes 发行版、云厂商、内部平台或单一调度器实现。Kubernetes 侧参考 PriorityClass、Pod priority / preemption、ResourceQuota 和 scheduler framework 的稳定口径；Kueue 侧参考 LocalQueue、ClusterQueue、cohort 借用、preemption、admission fair sharing 和 topology-aware scheduling 的官方设计；Volcano 侧参考 queue、gang scheduling、priority、preemption、reclaimable、guarantee / deserved / weight 等调度治理概念；公平性部分参考 dominant resource fairness 的多资源公平思想。

需要注意三点：

1. 本章讲队列、优先级、配额、公平性、抢占、gang scheduling、backfilling、碎片治理、拓扑感知、checkpoint 和调度指标。第 12 章已经讲过 Kubernetes 与 GPU 资源表达，本章重点是训练任务调度策略和治理闭环。
2. Kueue、Volcano、YuniKorn、自研调度层和云厂商训练平台的 API 不完全相同。本章只抽象稳定工程问题：如何 admission、如何排队、如何公平、如何借用、如何抢占、如何恢复、如何观测。
3. 这里的 demo 是教学审计脚本，不调用真实 scheduler API。真实平台需要从 queue CRD、scheduler events、admission records、checkpoint metadata、cost ledger、DCGM / Prometheus 指标和训练平台数据库采集证据。

## 13.1 为什么训练调度很难

普通后端服务调度通常关注 CPU、内存和副本数。

大模型训练调度更难，原因包括：

1. GPU 昂贵且稀缺。
2. 任务资源需求差异大。
3. 大任务需要多节点同时可用。
4. 任务运行时间长。
5. 训练失败恢复依赖 checkpoint。
6. 不同团队有配额和优先级。
7. GPU 型号和拓扑影响性能。
8. 小任务会造成 GPU 碎片。
9. 低优先级任务希望利用空闲资源。
10. 线上推理不能被训练任务干扰。

因此训练调度不是 K8s 默认调度的简单应用，而是 AI Infra 的核心能力。

## 13.2 调度系统要解决什么问题

训练调度系统至少要回答：

1. 任务能不能运行？
2. 任务什么时候运行？
3. 任务应该用哪些 GPU？
4. 谁的任务优先？
5. 多个团队如何公平共享资源？
6. 高优先级任务来了怎么办？
7. 低优先级任务能不能被抢占？
8. 抢占后如何恢复？
9. 如何减少碎片？
10. 如何提高整体 GPU 利用率？

这些问题对应队列、优先级、配额、抢占、公平性和拓扑感知。

## 13.3 训练任务的资源请求

一个训练任务不应该只声明“我要 8 张 GPU”。

更完整的资源请求包括：

1. GPU 类型。
2. GPU 数量。
3. 每节点 GPU 数量。
4. 显存需求。
5. 网络需求。
6. 存储需求。
7. 预计运行时间。
8. 优先级。
9. 队列。
10. 是否可抢占。
11. checkpoint 路径。
12. 租户和项目。

示例：

```yaml
job:
  name: pretrain-7b-v3
  queue: pretrain
  priority: high
  resources:
    gpu_type: A100-80G
    gpu_count: 128
    gpu_per_node: 8
    network: ib
  preemptible: false
  checkpoint_uri: s3://ckpts/pretrain-7b-v3/
```

资源请求越结构化，调度系统越能做正确决策。

## 13.4 队列 Queue

队列是训练调度的基本组织方式。

常见队列：

1. pretrain queue。
2. sft queue。
3. eval queue。
4. research queue。
5. production queue。
6. low-priority queue。

队列可以有不同策略：

1. 最大 GPU 配额。
2. 最大并发任务数。
3. 优先级规则。
4. 是否允许抢占。
5. 是否允许使用空闲资源。
6. 默认 checkpoint 要求。

例如：

1. production queue 优先级最高，不允许被抢占。
2. research queue 有固定配额。
3. low-priority queue 可以使用空闲 GPU，但可被抢占。

队列让资源治理从“单个任务”提升到“团队和业务类型”。

## 13.5 优先级 Priority

优先级决定任务竞争资源时谁先运行。

优先级可以来自：

1. 业务重要性。
2. 截止时间。
3. 用户角色。
4. 项目等级。
5. 任务类型。
6. 是否线上故障修复。
7. 是否阻塞发布。

常见优先级：

```text
critical > high > normal > low > best-effort
```

但优先级不能无限滥用。

否则所有人都申请 high，系统就失去意义。

需要配套：

1. 优先级审批。
2. 每队列高优任务额度。
3. 使用审计。
4. 超额告警。
5. 成本归因。

## 13.6 配额 Quota

配额用于控制团队或项目可使用的资源。

配额可以按多种维度定义：

1. GPU 数量。
2. GPU 小时。
3. GPU 型号。
4. 队列。
5. 并发任务数。
6. 存储容量。
7. 推理实例数。
8. 月度成本预算。

例如：

```text
team_a:
  A100-80G: 128 GPUs
  H100: 32 GPUs
  monthly_budget: 100000 GPU-hours
```

配额的目标不是限制创新，而是避免某个团队占满集群，让资源使用可预期、可治理。

## 13.7 公平性 Fairness

公平性回答：多个团队同时需要 GPU 时，如何分配才合理？

常见策略：

1. FIFO。
2. Weighted fair sharing。
3. Dominant Resource Fairness。
4. 配额内优先。
5. 借用空闲资源。
6. 超额可抢占。

FIFO 最简单，但不一定公平。

如果一个超大任务排在前面，小任务可能都被堵住。

Weighted fair sharing 可以给不同团队设置权重。

例如核心项目权重更高，研究项目权重较低。

DRF 适合多资源场景，例如 GPU、CPU、内存、网络和存储同时考虑。

公平性不是平均分，而是按业务重要性、配额、历史使用和资源需求综合平衡。

## 13.8 抢占 Preemption

抢占是指高优先级任务需要资源时，中断低优先级任务并释放资源。

抢占的价值：

1. 提高高优先级任务响应速度。
2. 让低优先级任务利用空闲资源。
3. 提高集群整体利用率。
4. 支持弹性资源池。

抢占的代价：

1. 被抢占任务损失进度。
2. 需要 checkpoint 支持。
3. 可能增加存储压力。
4. 用户体验变差。
5. 调度逻辑复杂。

好的抢占策略需要：

1. 只抢占可抢占任务。
2. 尽量选择损失最小的任务。
3. 提前通知任务保存 checkpoint。
4. 记录抢占原因。
5. 支持恢复排队。
6. 避免频繁抖动。

没有 checkpoint 的任务不适合随便抢占。

## 13.9 抢占策略如何设计

选择被抢占任务时，可以考虑：

1. 任务优先级。
2. 是否可抢占。
3. 已运行时间。
4. 距离上次 checkpoint 多久。
5. 释放资源是否足够。
6. 是否影响同一团队公平性。
7. 是否会造成更多碎片。
8. 用户或项目权重。

一个合理策略：

```text
优先抢占低优先级、可抢占、最近有 checkpoint、释放后能满足高优任务、且损失最小的任务。
```

抢占不应该只看“谁优先级低”，还要看实际释放效果和恢复成本。

## 13.10 Gang Scheduling

分布式训练需要 gang scheduling。

一个任务可能有多个 worker，每个 worker 一个 Pod。

如果不能同时启动，任务无法正常运行。

Gang scheduling 要求：

1. 资源满足整体任务需求。
2. worker 要么全部调度成功。
3. 要么全部等待。
4. 避免部分 worker 占住资源。

例如 64 卡任务需要 8 个 8 卡 worker。调度系统必须确认 8 个 worker 都能放下，才启动任务。

这对大任务非常关键。

## 13.11 Backfilling

Backfilling 是利用大任务等待期间的空闲资源运行小任务。

例如一个大任务需要 128 卡，但当前只有 96 卡可用。它还要等一段时间。

此时可以运行一些短小任务，只要它们不会影响大任务预计启动时间。

Backfilling 的价值：

1. 提高 GPU 利用率。
2. 减少资源空转。
3. 改善小任务体验。

风险：

1. 小任务超时会阻塞大任务。
2. 预计运行时间不准。
3. 需要可抢占或严格截止。

适合配合低优先级队列和 checkpoint 使用。

## 13.12 GPU 碎片治理

GPU 碎片会导致明明有空闲 GPU，但大任务启动不了。

治理方法：

1. 小任务集中放置。
2. 大任务优先完整节点。
3. 按任务类型划分资源池。
4. bin packing。
5. 空间预留。
6. 抢占和重排。
7. 限制小任务随意占用整机关键资源。

例如：

1. 1 卡和 2 卡任务集中放到专门资源池。
2. 8 卡训练任务优先使用完整 8 卡节点。
3. 预训练资源池不接受零散小任务。

碎片治理直接影响大任务等待时间。

## 13.13 拓扑感知调度

训练调度不能只看 GPU 数量。

它还要看：

1. GPU 型号。
2. 显存大小。
3. NVLink / NVSwitch。
4. GPU 到网卡亲和性。
5. 节点所在机柜。
6. 网络拥塞。
7. 数据缓存位置。
8. 故障域。

例如 tensor parallel 任务需要高速 GPU 间通信，最好放同机 NVSwitch。

大规模 data parallel 可以跨机，但最好在同一高带宽 fabric 内。

拓扑感知调度可以提升 MFU 和扩展效率。

## 13.14 调度与 checkpoint 的关系

调度和 checkpoint 强相关。

抢占需要 checkpoint。

故障恢复需要 checkpoint。
低优先级队列需要 checkpoint。
弹性训练需要 checkpoint。

调度系统应该知道：

1. 任务是否支持 checkpoint。
2. 最近一次 checkpoint 时间。
3. checkpoint 路径。
4. 恢复预计耗时。
5. 抢占后损失多少进度。

没有这些信息，调度系统无法做理性抢占。

## 13.15 调度与成本治理

调度系统直接影响成本。

成本浪费来自：

1. GPU 空闲。
2. 大任务排队导致项目延期。
3. 低优先级任务占用昂贵 GPU。
4. 任务失败重跑。
5. 训练效率低。
6. checkpoint 太频繁。
7. 资源申请过大。

调度可以优化：

1. 提高利用率。
2. 使用低优先级队列。
3. 限制超额资源申请。
4. 让短任务利用碎片资源。
5. 按团队归因 GPU 小时。
6. 对低效任务告警。

调度不是只决定“跑不跑”，也决定“贵不贵”。

6. GPU 是否真正被有效使用？

## 13.17 训练调度治理审计指标与最小 demo

训练调度系统不能只看 GPU utilization。一个好的调度系统要证明：任务画像完整、队列规则清楚、优先级不滥用、配额不过载、公平性可解释、gang 能保证整体启动、backfilling 不阻塞保留任务、抢占有 checkpoint 兜底、碎片和拓扑可控、失败后能重排、成本可以归因。

可以把一个调度审计样本写成：

```math
s_i=(j_i,q_i,p_i,a_i,f_i,b_i,e_i,c_i,t_i,o_i,z_i)
```

其中，`j_i` 是训练任务资源画像，`q_i` 是队列配置，`p_i` 是优先级治理，`a_i` 是 admission / gang 状态，`f_i` 是公平性与配额，`b_i` 是 backfilling 策略，`e_i` 是抢占与恢复，`c_i` 是 checkpoint 耦合，`t_i` 是拓扑和碎片，`o_i` 是观测指标，`z_i` 是最终门禁。

统一覆盖率可以写成：

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(s_i)=1]
```

队列等待时间可以按队列统计 P95：

```math
W_{q,\mathrm{p95}}=\mathrm{P95}(t_{\mathrm{start}}-t_{\mathrm{sub}}\mid q)
```

租户配额占用率可以写成：

```math
Q_u=\frac{G_{u,\mathrm{used}}}{G_{u,\mathrm{quota}}}
```

其中，`G_u_used` 是租户 `u` 当前已用 GPU，`G_u_quota` 是租户 GPU 配额。

多资源公平性可以用 dominant share 近似：

```math
D_u=\max_r \frac{A_{u,r}}{C_r}
```

其中，`A_u,r` 是租户 `u` 使用的第 `r` 类资源，`C_r` 是该资源总容量。DRF 的直觉是让多个租户的 dominant share 尽量接近其权重，而不是只看单一 GPU 数。

抢占损失可以估算为：

```math
L_j=G_j(t_{\mathrm{now}}-t_{\mathrm{ckpt}})
```

其中，`G_j` 是被抢占任务占用的 GPU 数，`t_now - t_ckpt` 是距离上次 checkpoint 的时间。单位可以是 GPU-minute 或 GPU-hour。

Backfilling 的安全条件可以写成：

```math
B_j=\mathbf{1}\left[t_{\mathrm{now}}+T_j\le t_{\mathrm{reserve}}\right]
```

其中，`T_j` 是小任务预计运行时长，`t_reserve` 是大任务保留资源的预计启动时间。

GPU 碎片化比例仍然可以写成：

```math
R_{\mathrm{frag}}=1-\frac{\sum_n \lfloor g_n / g_{\mathrm{job}}\rfloor g_{\mathrm{job}}}{\sum_n g_n}
```

最后，可以把训练调度治理门禁写成：

```math
G_{\mathrm{train\_sched}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land W_{q,\mathrm{p95}}\le \omega_q \land Q_u\le 1 \land R_{\mathrm{frag}}\le \rho \land P_0=0\right]
```

其中，`\tau_j` 是每个检查项的覆盖率阈值，`\omega_q` 是队列等待时间 SLO，`\rho` 是碎片化阈值，`P_0` 是 P0 级调度风险数量。

def check_workload_shape(case):
    job = case["job"]
    required = ["gpu_type", "gpu_count", "gpu_per_node", "network", "queue", "priority", "checkpoint_uri"]
    return all(job.get(key) for key in required) and job["gpu_count"] % job["gpu_per_node"] == 0


def check_queue_policy(case):
    queue = case["queue"]
    return (
        queue["quota_gpu"] > 0
        and queue["max_running_jobs"] > 0
        and bool(queue["priority_classes"])
        and bool(queue["preemption_policy"])
        and queue["borrow_limit_gpu"] >= 0
        and queue["lending_enabled"]
    )


def check_priority(case):
    priority = case["priority"]
    job_priority = case["job"]["priority"]
    return (
        job_priority in priority["allowed_classes"]
        and priority["approved"]
        and priority["high_used"] <= priority["high_limit"]
        and priority["audit_log"]
    )


def check_quota(case):
    quota = case["quota"]
    return (
        quota_ratio(quota["used_gpu"], quota["quota_gpu"]) <= 1
        and quota["used_gpu_hours"] <= quota["budget_gpu_hours"]
        and quota["per_gpu_type_quota"]
    )


def check_fairness(case):
    fair = case["fairness"]
    return (
        fair["tracked"]
        and weighted_share_ratio(fair["tenant_gpu_used"], fair["cluster_gpu_total"], fair["tenant_weight"])
        <= fair["weighted_share_limit"]
        and dominant_share(fair["usage"], fair["capacity"]) <= fair["dominant_share_limit"]
    )


def check_gang(case):
    gang = case["gang"]
    return (
        gang["enabled"]
        and gang["min_available"] == gang["worker_count"]
        and gang["all_or_nothing"]
        and not gang["partial_allocation_allowed"]
    )


def check_backfill(case):
    backfill = case["backfill"]
    return (
        backfill["enabled"]
        and backfill_safe(backfill["now_min"], backfill["short_job_runtime_min"], backfill["reserved_start_min"])
        and backfill["deadline_enforced"]
        and backfill["short_job_preemptible"]
    )


def check_preemption(case):
    preemption = case["preemption"]
    loss = preemption_loss_gpu_min(
        preemption["now_min"],
        preemption["last_checkpoint_min"],
        preemption["gpu_count"],
    )
    return (
        preemption["candidate_preemptible"]
        and preemption["candidate_priority"] == "low"
        and preemption["higher_priority_waiting"]
        and preemption["checkpoint_supported"]
        and loss <= preemption["max_loss_gpu_min"]
        and preemption["reason_logged"]
        and preemption["restore_requeue"]
        and preemption["min_runtime_protection"]
    )


def check_fragmentation(case):
    frag = case["fragmentation"]
    return (
        fragmentation_ratio(frag["free_gpus_by_node"], frag["gpus_per_job"]) <= frag["threshold"]
        and frag["bin_packing"]
        and frag["large_job_pool"]
    )


def check_topology(case):
    topo = case["topology"]
    return (
        topology_score(topo["components"], topo["weights"]) >= topo["threshold"]
        and topo["topology_profile"]
        and topo["scheduler_uses_topology"]
    )


def check_checkpoint(case):
    checkpoint = case["checkpoint"]
    return (
        checkpoint["scheduler_knows_uri"]
        and checkpoint["last_checkpoint_age_min"] <= checkpoint["max_lost_min"]
        and checkpoint["restore_estimate_min"] <= checkpoint["resume_slo_min"]
    )


def check_cost(case):
    return all(case["cost"].values())


def check_observability(case):
    return all(case["observability"].values())


def check_failure_requeue(case):
    return all(case["failure"].values())


def check_starvation(case):
    return all(case["starvation"].values())


def check_gate(case):
    gate = case["gate"]
    return gate["enabled"] and gate["thresholds_ok"] and gate["p0_risks"] == 0

## 13.18 常见调度故障

### 13.18.1 大任务长期排队

可能原因：

1. 资源不足。
2. GPU 碎片化。
3. 配额不足。
4. 优先级太低。
5. 需要特定 GPU 型号。
6. 拓扑要求太严格。
7. gang scheduling 无法满足。

### 13.18.2 小任务体验差

可能原因：

1. FIFO 被大任务堵住。
2. 没有 backfilling。
3. 队列策略不合理。
4. 小任务和大任务混用资源池。
5. 交互式资源池不足。

### 13.18.3 抢占过多

可能原因：

1. 高优任务过多。
2. 低优先级资源池策略不合理。
3. 没有最小运行时间保护。
4. checkpoint 频率不匹配。
5. 资源预留不足。

### 13.18.4 GPU 利用率高但产出低

可能原因：

1. 低效任务占用资源。
2. 训练任务 MFU 低。
3. 大量失败重试。
4. 数据加载慢。
5. 通信拓扑差。
6. 用户申请资源过量。

## 13.19 面试中如何回答训练调度系统设计

如果面试官问：

```text
如何设计一个大模型训练任务调度系统？
```

可以这样回答：

```text
我会在 Kubernetes 之上设计面向 AI 任务的调度层。用户提交 TrainingJob，声明 GPU 类型、数量、每节点 GPU 数、网络需求、优先级、队列、是否可抢占和 checkpoint 路径。

调度层按队列管理任务，每个队列有配额、优先级和公平分享策略。分布式训练使用 gang scheduling，保证 worker 要么全部启动，要么全部等待，避免部分 Pod 占住资源。

资源选择时做拓扑感知，考虑 GPU 型号、NVLink/NVSwitch、GPU 到网卡亲和性、机柜网络、故障域和数据位置。为了提高利用率，支持 backfilling，让短任务使用大任务等待期间的空闲资源。

抢占方面，只抢占低优先级、可抢占、最近有 checkpoint 且释放资源有效的任务，并记录抢占原因和恢复状态。

治理方面，提供团队配额、公平性、成本归因、GPU 碎片监控、队列等待时间、GPU 小时统计和低效任务告警。
```

## 13.20 常见误区

误区一：训练调度就是 FIFO。

FIFO 简单但容易被大任务堵住，也无法表达优先级和公平性。

误区二：GPU 利用率高就说明调度好。

还要看任务产出、等待时间、公平性、MFU、失败率和成本。

误区三：抢占越多利用率越高。

抢占会带来进度损失和用户体验问题，需要 checkpoint 和策略控制。

误区四：配额就是固定不能超。

成熟系统可以允许借用空闲资源，但超额资源要可抢占。

误区五：调度不用理解网络拓扑。

大模型训练通信密集，拓扑会直接影响训练效率。

## 13.21 面试题

### 题 1：为什么大模型训练需要 gang scheduling？

答：分布式训练需要所有 worker 同时参与。如果只启动部分 worker，任务无法开始，但资源会被占住。Gang scheduling 保证任务所需的一组 Pod 要么全部调度成功，要么全部等待，避免资源浪费和任务卡死。

### 题 2：什么是 GPU 碎片化？如何缓解？

答：GPU 碎片化是指集群中有空闲 GPU，但分散在不同节点或拓扑位置，无法满足大任务需要完整节点或连续资源的需求。缓解方法包括 bin packing、小任务集中放置、大任务资源池、抢占、backfilling 和拓扑感知调度。

### 题 3：抢占策略应该考虑哪些因素？

答：应考虑任务优先级、是否可抢占、最近 checkpoint 时间、已运行时间、释放资源是否满足高优任务、恢复成本、团队公平性和碎片影响。不能只看优先级低就抢占。

### 题 4：配额和公平性有什么区别？

答：配额是资源使用上限或保障，例如某团队最多使用多少 GPU；公平性是多个团队同时竞争资源时如何分配，例如按权重、历史使用、dominant resource 或队列策略分配。配额是规则，公平性是调度目标。

### 题 5：为什么调度系统要知道 checkpoint 信息？

答：因为抢占、故障恢复和弹性调度都依赖 checkpoint。调度系统需要知道任务是否支持 checkpoint、最近 checkpoint 时间、恢复路径和恢复成本，才能选择损失最小的抢占对象。
