# 第 14 章 多租户隔离：资源隔离、权限隔离和数据隔离

前面我们讲了 Kubernetes、GPU 管理和训练任务调度。本章讲企业 AI Infra 里非常现实的问题：多个团队、多个项目、多个用户共用同一套 GPU 集群时，如何隔离？

如果没有多租户隔离，AI 平台很快会出现资源被抢占、数据被误读、权限失控、成本无法归因、日志混乱、镜像不可信、线上推理被实验任务干扰等问题。

先记住一句话：

> 多租户隔离不是只建几个 namespace，而是资源、身份、权限、数据、网络、镜像、密钥、审计和成本的系统性治理。

## 14.1 什么是多租户

租户可以是：

1. 一个业务团队。
2. 一个项目组。
3. 一个部门。
4. 一个外部客户。
5. 一个环境，例如 dev、staging、prod。

多租户就是多个租户共享同一套平台能力。

共享的好处：

1. 提高 GPU 利用率。
2. 降低重复建设成本。
3. 统一平台能力。
4. 统一监控和治理。
5. 统一安全和审计。

共享的风险：

1. 资源互相抢。
2. 数据互相看见。
3. 网络互相访问。
4. 密钥泄露。
5. 镜像污染。
6. 成本无法归因。
7. 故障影响扩大。

多租户隔离的目标，就是在共享资源的同时控制这些风险。

## 14.2 多租户隔离的层次

AI Infra 多租户隔离至少包括八层：

1. 资源隔离。
2. 权限隔离。
3. 数据隔离。
4. 网络隔离。
5. 运行时隔离。
6. 镜像和供应链隔离。
7. 密钥隔离。
8. 审计和成本隔离。

其中任何一层缺失，都可能出问题。

例如资源隔离做得好，但数据权限没做好，仍然可能泄露训练数据。

又比如数据权限做得好，但密钥以环境变量泄露到日志里，也会出事。

## 14.3 资源隔离

资源隔离回答：每个租户能使用多少 GPU、CPU、内存、存储和网络？

常见资源：

1. GPU 数量。
2. GPU 型号。
3. GPU 小时。
4. CPU。
5. 内存。
6. 本地盘。
7. 共享存储容量。
8. 网络带宽。
9. 推理 QPS。
10. 队列并发。

资源隔离手段：

1. Namespace。
2. ResourceQuota。
3. Queue quota。
4. PriorityClass。
5. LimitRange。
6. 独立资源池。
7. 调度队列。
8. 成本预算。

资源隔离不是所有资源完全物理隔开，而是明确租户的保障、上限和超额使用规则。

## 14.4 硬隔离和软隔离

资源隔离可以分成硬隔离和软隔离。

硬隔离：

1. 专属节点。
2. 专属 GPU 资源池。
3. 独立网络。
4. 独立存储 bucket。
5. 独立 Kubernetes 集群。

优点是隔离强，缺点是利用率可能低、成本高。

软隔离：

1. 共享集群。
2. namespace 隔离。
3. 配额限制。
4. 优先级队列。
5. 抢占策略。
6. 逻辑权限控制。

优点是利用率高，缺点是治理复杂。

企业常见做法是混合：生产推理、高敏数据和关键任务硬隔离；普通训练和实验软隔离。

## 14.5 权限隔离

权限隔离回答：谁可以提交任务、访问数据、使用模型、查看日志、修改配置、发布服务？

常见权限对象：

1. 用户。
2. 用户组。
3. Service Account。
4. 训练任务。
5. 推理服务。
6. Agent。
7. 平台管理员。

常见权限动作：

1. submit_job。
2. cancel_job。
3. read_logs。
4. read_dataset。
5. write_checkpoint。
6. deploy_model。
7. access_secret。
8. change_quota。
9. approve_release。

权限模型通常需要 RBAC 和 ABAC 结合。

RBAC 按角色授权，例如 admin、developer、viewer。

ABAC 按属性授权，例如项目、数据等级、环境、任务类型、资源池。

## 14.6 数据隔离

数据隔离是 AI Infra 多租户里最重要的一层。

需要隔离的数据包括：

1. 原始训练数据。
2. 清洗数据。
3. 评估数据。
4. RAG 文档。
5. checkpoint。
6. 模型权重。
7. 日志。
8. trace。
9. prompt。
10. 用户请求。

数据隔离要解决：

1. 谁能读？
2. 谁能写？
3. 谁能复制？
4. 谁能导出？
5. 谁能用于训练？
6. 谁能用于评估？
7. 是否允许跨租户共享？

尤其要注意：训练任务一旦能读数据，就可能把数据记进模型参数、checkpoint 或日志里。

所以数据权限不仅是读文件权限，还要考虑数据使用目的和下游产物权限。

## 14.7 数据分类与使用策略

企业数据通常需要分类。

例如：

1. public。
2. internal。
3. confidential。
4. restricted。
5. regulated。

不同分类对应不同策略：

1. 是否允许训练。
2. 是否允许导出。
3. 是否允许跨区域。
4. 是否需要脱敏。
5. 是否需要审批。
6. 产出的模型是否继承数据等级。

例如使用 restricted 数据训练出的模型，可能也应该被标记为 restricted，不能随便发布到所有租户。

这叫数据等级传播。

## 14.8 网络隔离

网络隔离回答：不同租户的任务能不能互相访问？能不能访问外网？能不能访问生产数据库？

常见手段：

1. Kubernetes NetworkPolicy。
2. VPC / subnet。
3. Service mesh policy。
4. 防火墙。
5. egress 控制。
6. 私有 endpoint。
7. DNS 限制。

AI 任务网络风险包括：

1. 训练任务访问不该访问的数据服务。
2. 容器向外发送敏感数据。
3. 不同租户 Pod 互相访问。
4. 调试端口暴露。
5. 推理服务被非授权调用。

默认策略应该是最小开放，而不是默认互通。

## 14.9 运行时隔离

运行时隔离包括容器、Pod、节点和 GPU 层面的隔离。

需要关注：

1. 容器是否 privileged。
2. 是否挂载宿主机路径。
3. 是否允许 root。
4. 是否共享 IPC / PID namespace。
5. 是否共享 GPU。
6. 是否使用 MIG。
7. 是否隔离本地缓存。
8. 是否限制系统调用。

GPU 多租户尤其复杂。

整卡分配隔离更强，但利用率可能低。

MIG 可以提供硬件级切分，适合部分推理和开发负载。

时间片共享利用率高，但性能隔离和安全隔离较弱。

选择哪种方式要看任务风险和稳定性要求。

## 14.10 镜像和供应链隔离

租户提交的镜像可能不可信。

风险包括：

1. 镜像包含恶意代码。
2. 镜像包含漏洞包。
3. 镜像里写入密钥。
4. 镜像 tag 被覆盖。
5. 镜像来源不明。
6. 基础镜像过旧。

治理措施：

1. 私有镜像仓库。
2. 镜像扫描。
3. 镜像签名。
4. 只允许可信 registry。
5. 基础镜像白名单。
6. 记录 digest。
7. 禁止高风险镜像运行。
8. 构建流水线审计。

多租户平台不能让任何用户随便拉任意镜像在 GPU 节点上运行。

## 14.11 密钥隔离

AI 任务经常需要访问：

1. 数据库。
2. 对象存储。
3. 模型仓库。
4. 外部 API。
5. 评估服务。
6. 私有代码仓库。

这需要密钥。

密钥隔离要避免：

1. 把密钥写进镜像。
2. 把密钥写进代码。
3. 把密钥打印到日志。
4. 多租户共享同一个高权限密钥。
5. 任务结束后密钥仍然有效。

更好的做法：

1. 使用短期凭证。
2. 使用 Service Account。
3. 按任务注入最小权限密钥。
4. 密钥自动轮换。
5. 访问审计。
6. 日志脱敏。

## 14.12 日志、Trace 和可见性隔离

日志和 trace 也可能包含敏感信息。

例如：

1. 训练样本片段。
2. 用户 prompt。
3. 模型输出。
4. 数据路径。
5. 密钥片段。
6. 错误堆栈中的参数。
7. Agent 工具调用结果。

因此日志隔离要控制：

1. 谁能看任务日志。
2. 谁能看推理请求日志。
3. 谁能看 Agent trace。
4. 日志保留多久。
5. 是否需要脱敏。
6. 是否允许导出。

不要默认所有平台管理员都能随便看所有用户数据日志。至少要有审计和最小权限。

## 14.13 成本隔离和归因

多租户平台必须能回答：谁用了多少资源，花了多少钱？

需要归因的资源：

1. GPU 小时。
2. CPU 和内存。
3. 存储容量。
4. 网络流量。
5. 推理 token。
6. checkpoint 存储。
7. 评估任务。
8. Agent 工具调用。

成本归因维度：

1. tenant。
2. team。
3. project。
4. user。
5. job。
6. model。
7. environment。

没有成本隔离，就无法做预算、配额和优化。

## 14.14 生产环境和实验环境隔离

生产推理和实验训练最好隔离。

原因：

1. 生产有 SLO。
2. 实验任务不稳定。
3. 实验镜像可能不安全。
4. 训练任务会占用网络和存储。
5. 生产数据权限更敏感。
6. 变更流程不同。

隔离方式：

1. 独立 namespace。
2. 独立节点池。
3. 独立资源池。
4. 独立网络策略。
5. 独立权限审批。
6. 独立发布流程。

关键生产服务可以使用硬隔离资源池。

## 14.15 跨租户共享

完全隔离并不总是最佳。

有些资源需要共享：

1. 基础模型。
2. 公共数据集。
3. 公共 embedding 服务。
4. 公共评估 benchmark。
5. 基础镜像。
6. 工具平台。

共享要有规则：

1. 谁维护？
2. 谁能使用？
3. 是否只读？
4. 是否可复制？
5. 使用后产物如何定级？
6. 成本如何分摊？

共享不是把权限全打开，而是把共享资源产品化、版本化、权限化。

## 14.16 多租户隔离架构示例

一个企业 AI 平台可以这样设计：

```text
User / Team
  -> Identity Provider
  -> AI Platform Portal / CLI
  -> Policy Engine
  -> Queue / Quota / Scheduler
  -> Kubernetes Namespace / Resource Pool
  -> Data Access Gateway
  -> Secret Manager
  -> Image Registry
  -> Observability / Audit / Cost
```

关键点：

1. 身份统一从 IdP 来。
2. Policy Engine 统一判断权限。
3. Scheduler 管资源配额和队列。
4. Data Access Gateway 管数据访问。
5. Secret Manager 管短期凭证。
6. Image Registry 管可信镜像。
7. Observability 管日志、审计和成本。

## 14.17 面试中如何回答多租户隔离

如果面试官问：

```text
如何设计一个多租户 GPU 训练平台？
```

可以这样回答：

```text
我会从资源、权限、数据、网络、运行时、镜像、密钥、审计和成本几个层面设计隔离。

资源层用 namespace、queue、quota、priority 和资源池隔离不同团队，关键生产推理和高敏任务使用独立节点池或硬隔离资源池。调度层支持公平分享、抢占和成本归因。

权限层接入统一身份系统，用 RBAC + ABAC 控制谁能提交任务、访问数据、读取日志、部署模型和修改配额。

数据层按数据分类和项目授权控制访问，训练数据、checkpoint、模型权重、RAG 文档、日志和 trace 都要有权限和审计。使用敏感数据训练出的模型也要继承相应数据等级。

网络层通过 NetworkPolicy、VPC、egress 控制和私有 endpoint 限制租户间互访和外部访问。运行时层限制 privileged、hostPath、root、GPU 共享等高风险配置。

镜像和密钥层使用私有 registry、镜像扫描、签名、digest、Secret Manager 和短期凭证，避免密钥写入镜像或日志。

最后通过审计和成本系统记录谁在什么时候用了什么资源、访问了什么数据、产生了什么模型和成本。
```

## 14.18 常见误区

误区一：namespace 就是多租户隔离。

Namespace 只是逻辑分组，还需要资源、权限、网络、数据、密钥和审计隔离。

误区二：容器就是强安全边界。

容器隔离有限，privileged、hostPath、共享 GPU 和高权限密钥都可能破坏隔离。

误区三：只隔离训练数据就够了。

Checkpoint、模型权重、日志、trace、prompt 和评估结果都可能包含敏感信息。

误区四：共享资源就是所有人可读。

共享需要 owner、版本、权限、审计、成本分摊和使用策略。

误区五：成本归因是财务问题。

成本归因直接影响配额、公平性、资源优化和平台治理，是 AI Infra 的核心能力。

## 14.19 面试题

### 题 1：AI Infra 多租户隔离包括哪些层？

答：包括资源隔离、权限隔离、数据隔离、网络隔离、运行时隔离、镜像和供应链隔离、密钥隔离、日志审计隔离和成本隔离。

### 题 2：为什么数据隔离不只是文件权限？

答：训练任务读取数据后，数据可能进入 checkpoint、模型权重、日志、trace 或评估结果。使用敏感数据训练出的模型也可能继承敏感等级。所以要控制数据使用目的和下游产物权限。

### 题 3：生产推理为什么要和实验训练隔离？

答：生产推理有 SLO 和稳定性要求，实验训练可能不稳定、占用网络和存储、使用未审核镜像或访问不同数据。混用会造成性能干扰和安全风险。

### 题 4：如何避免密钥泄露？

答：不要把密钥写进镜像、代码或日志。使用 Secret Manager、短期凭证、Service Account、最小权限、自动轮换、日志脱敏和访问审计。

### 题 5：硬隔离和软隔离如何选择？

答：高敏数据、生产推理、关键业务和强合规场景适合硬隔离，例如专属资源池或独立集群。普通训练、评估和研究任务可用软隔离，通过 namespace、quota、queue、policy 和审计提高资源利用率。

## 14.20 小练习

练习一：设计一个团队级租户模型。

要求：包含用户、项目、namespace、queue、quota、dataset permission、model permission 和 cost center。

练习二：设计一个数据等级传播规则。

要求：说明 restricted 数据训练出的 checkpoint、模型权重、eval report 应该如何定级和授权。

练习三：排查跨租户数据泄露风险。

假设某用户在日志里看到了其他团队的数据片段，请从日志权限、数据挂载、任务身份、共享目录、trace 和脱敏策略排查。

练习四：设计一个多租户成本报表。

要求：按团队、项目、用户、任务、GPU 类型、存储和推理 token 展示成本。

## 14.21 本章小结

本章讲了 AI Infra 多租户隔离。

你需要掌握：

1. 多租户隔离不是 namespace，而是系统性治理。
2. 资源隔离要覆盖 GPU、CPU、内存、存储、网络、队列、推理 QPS 和成本。
3. 权限隔离需要 RBAC + ABAC。
4. 数据隔离要覆盖训练数据、checkpoint、模型权重、日志、trace、RAG 文档和评估结果。
5. 敏感数据训练出的模型和 artifact 可能需要继承数据等级。
6. 网络隔离要控制租户互访、外部访问和生产数据访问。
7. 运行时隔离要限制 privileged、hostPath、root、共享 GPU 等风险。
8. 镜像、密钥、日志和审计都是多租户安全的重要部分。
9. 成本隔离和归因是配额、公平性和资源优化的基础。
