# 第 36 章 数据平台与训练数据供给链路

前面第四部分讲完了大模型推理平台工程。从本章开始进入第五部分：数据、模型与实验平台。

训练平台和推理平台解决的是“算力怎么用、服务怎么跑”。但大模型能力的底层基础仍然是数据。没有稳定的数据平台，训练任务会因为数据找不到、版本不清、质量不稳、吞吐不足和权限不明而反复失败。

先记住一句话：

> 大模型数据平台不是简单存放数据的目录，而是围绕数据采集、清洗、治理、版本、质量、权限、供给和血缘建立的工程系统。

## 36.1 为什么需要数据平台

在小实验中，数据可能只是一个本地文件：

```text
data/train.jsonl
```

但大模型训练中，数据规模和复杂度完全不同：

1. 数据量从 GB 到 TB、PB。
2. 数据来源包括网页、代码、文档、对话、图片、音频、视频、业务数据。
3. 数据需要清洗、去重、过滤、脱敏和标注。
4. 数据版本要可复现。
5. 数据质量要持续监控。
6. 数据权限要严格控制。
7. 训练时要高吞吐供给 GPU。

如果没有数据平台，训练会变成“谁知道这批数据从哪里来”的黑盒流程。

## 36.2 数据平台要解决什么问题

数据平台要解决八类问题：

1. 数据从哪里来。
2. 数据如何存。
3. 数据如何清洗。
4. 数据如何标注。
5. 数据如何版本化。
6. 数据质量如何监控。
7. 训练如何高吞吐读取。
8. 数据权限和血缘如何管理。

大模型训练中的很多问题看似是训练问题，本质上是数据问题。

比如 loss 异常，可能不是优化器坏了，而是某批数据重复、污染、格式错误或分布变化。

## 36.3 训练数据供给链路总览

一个典型训练数据供给链路：

```text
Raw Data Sources
  -> Ingestion
  -> Raw Data Lake
  -> Cleaning / Filtering
  -> Deduplication
  -> PII Redaction
  -> Quality Scoring
  -> Dataset Builder
  -> Dataset Version Registry
  -> Object Storage / File System
  -> Streaming / Sharding / Cache
  -> Training Job
```

每一步都可能影响最终模型质量。

数据平台的目标是让这条链路可复现、可观测、可治理。

## 36.4 数据来源

大模型数据来源常见包括：

1. 公开网页。
2. 书籍和百科。
3. 代码仓库。
4. 学术论文。
5. 对话数据。
6. 人工标注数据。
7. 合成数据。
8. 企业文档。
9. 产品日志。
10. 多模态数据。

不同来源的风险不同。

公开网页规模大但噪声多。

企业文档价值高但权限和隐私敏感。

合成数据可控但可能放大模型偏差。

代码数据结构清晰但许可证复杂。

## 36.5 数据接入 Ingestion

数据接入负责把外部数据进入平台。

接入系统要记录：

1. 数据来源。
2. 抓取时间。
3. 原始路径。
4. 数据格式。
5. 许可证或使用约束。
6. 所属租户或业务。
7. 权限等级。
8. 接入任务 ID。
9. 校验和。

不要只把文件丢进对象存储。

数据从进入平台开始就要有元数据。

## 36.6 Raw Data Lake

Raw Data Lake 保存原始数据。

原始数据通常不直接用于训练，但必须保留，因为后续清洗策略可能变化，需要回溯。

Raw Data Lake 的要求：

1. 原始数据不可随意覆盖。
2. 写入后有校验。
3. 支持按来源、时间、租户检索。
4. 支持权限控制。
5. 支持生命周期管理。
6. 支持审计。

保留 raw data 的价值是：当清洗规则或质量问题出现时，可以重新构建数据集。

## 36.7 数据清洗

数据清洗是把原始数据变成可训练数据的重要步骤。

常见清洗包括：

1. 格式解析。
2. 编码规范化。
3. HTML / boilerplate 去除。
4. 语言识别。
5. 低质量文本过滤。
6. 重复内容去除。
7. 乱码过滤。
8. 过短或过长样本过滤。
9. 敏感信息脱敏。
10. 不合规内容过滤。

清洗不是越狠越好。

过度清洗可能损失多样性；清洗不足会带来噪声和安全风险。

## 36.8 去重 Deduplication

去重对大模型非常关键。

重复数据会导致：

1. 训练效率下降。
2. 模型记忆重复内容。
3. 评估泄漏风险增加。
4. 数据分布被重复样本扭曲。
5. 成本浪费。

去重可以分为：

1. 精确去重。
2. 近似去重。
3. 文档级去重。
4. 段落级去重。
5. 样本级去重。
6. train/eval 跨集合去重。

跨集合去重非常重要，否则 eval 集出现在 train 集里，会让评估结果虚高。

## 36.9 PII 脱敏和安全过滤

PII 是 personally identifiable information，也就是个人可识别信息。

数据平台要检测和处理：

1. 手机号。
2. 邮箱。
3. 身份证件号。
4. 地址。
5. 银行卡。
6. 账号密钥。
7. 企业内部敏感信息。
8. 私有代码或凭据。

处理方式包括：

1. 删除样本。
2. 脱敏替换。
3. 权限隔离。
4. 标记敏感等级。
5. 禁止进入特定训练集。

敏感数据一旦进入模型训练，后续风险很难彻底消除。

## 36.10 数据质量评分

数据质量不是一个单一指标。

常见质量维度：

1. 可读性。
2. 信息密度。
3. 语言质量。
4. 格式完整性。
5. 重复率。
6. 毒性和安全风险。
7. 主题分布。
8. 领域覆盖。
9. 标注一致性。
10. 许可证风险。

平台可以为样本或文档打质量分，并在 dataset builder 中按策略采样。

质量评分的目标不是把数据分成好坏两类，而是支持可控的数据配方。

## 36.11 Dataset Builder

Dataset Builder 负责把清洗后的数据组合成训练数据集。

它通常支持：

1. 选择数据来源。
2. 设置采样比例。
3. 按质量分过滤。
4. 按语言过滤。
5. 按领域过滤。
6. 设置最大样本数。
7. 设置 train/val/test 切分。
8. 输出统一格式。
9. 生成 dataset manifest。

数据集不是一堆文件，而是一个有配方、有版本、有 manifest 的训练输入。

## 36.12 Dataset Manifest

Dataset manifest 是描述数据集内容的元数据文件。

它可以包含：

1. dataset 名称。
2. dataset 版本。
3. 数据来源列表。
4. 文件路径列表。
5. 样本数量。
6. token 数量估计。
7. 采样比例。
8. 清洗规则版本。
9. 去重规则版本。
10. 数据质量统计。
11. 权限等级。
12. checksum。

训练任务应该引用 dataset version 或 manifest，而不是随便引用一个目录。

## 36.13 数据格式

大模型训练常见格式：

1. JSONL。
2. Parquet。
3. Arrow。
4. WebDataset。
5. TFRecord。
6. 自定义二进制格式。

JSONL 易读易调试，但大规模读取效率不一定高。

Parquet 和 Arrow 更适合列式存储和大规模处理。

WebDataset 常用于多模态数据。

格式选择要考虑读取吞吐、压缩、随机访问、schema 演进和调试便利性。

## 36.14 Tokenization 在数据链路中的位置

Tokenization 可以在线做，也可以离线做。

在线 tokenization：训练时读取文本再分词。

优点：灵活，tokenizer 更新方便。

缺点：训练时 CPU 开销和延迟更高。

离线 tokenization：提前把文本转成 token IDs。

优点：训练读取更快。

缺点：tokenizer 版本绑定强，存储成本增加。

大规模预训练常会考虑离线 tokenization 或半离线 tokenization，以降低训练时数据处理瓶颈。

## 36.15 Sharding

Sharding 是把大数据集切成多个 shard。

它的作用：

1. 支持并行读取。
2. 降低单文件压力。
3. 提高分布式训练吞吐。
4. 支持 shuffle。
5. 便于断点恢复。

Shard 设计要避免：

1. shard 大小差异太大。
2. 样本分布不均。
3. 热点 shard。
4. 小文件过多。
5. shuffle 不充分。

训练数据供给不是只要能读，而是要持续喂饱 GPU。

## 36.16 数据读取链路

训练时的数据读取链路可能是：

```text
Object Storage / Data Lake
  -> Data Loader Workers
  -> Local Cache
  -> Shuffle Buffer
  -> Tokenizer / Decoder
  -> Batch Collator
  -> GPU Training Process
```

瓶颈可能出现在：

1. 对象存储吞吐。
2. 网络带宽。
3. 小文件过多。
4. 解压缩。
5. tokenization。
6. shuffle buffer。
7. CPU dataloader。
8. 本地缓存容量。

如果数据供给跟不上，GPU 会等待数据，训练效率下降。

## 36.17 数据缓存

训练数据缓存可以放在：

1. 本地 NVMe。
2. 分布式缓存。
3. 节点级缓存。
4. 任务级缓存。
5. 共享文件系统。

缓存目标：

1. 降低对象存储重复读取。
2. 提高 epoch 内读取速度。
3. 降低网络压力。
4. 支持任务重启。

但缓存要注意一致性和版本。

不要让训练任务误读旧版本数据。

## 36.18 Shuffle 设计

Shuffle 对训练很重要。

Shuffle 不充分会导致：

1. batch 分布偏移。
2. 训练不稳定。
3. loss 曲线异常。
4. 模型过度适应数据顺序。

大规模数据下，完全全局 shuffle 成本很高。

常见做法：

1. shard-level shuffle。
2. shard 内 shuffle。
3. sliding shuffle buffer。
4. 多 epoch 改变 shard 顺序。
5. 按 worker 分片后局部 shuffle。

Shuffle 是训练质量和数据吞吐之间的权衡。

## 36.19 数据权限

数据平台必须做权限控制。

权限维度包括：

1. 谁能看 raw data。
2. 谁能使用某个 dataset。
3. 哪些训练任务可以读取敏感数据。
4. 哪些模型可以用某类数据训练。
5. 数据是否允许导出。
6. 数据是否允许跨租户使用。

训练任务提交时，平台应该校验用户或服务账号是否有 dataset 访问权限。

不能因为训练跑在集群里，就默认能读所有数据。

## 36.20 数据血缘

数据血缘回答：某个模型是由哪些数据训练出来的。

至少记录：

1. 原始数据来源。
2. 清洗任务。
3. 清洗规则版本。
4. 去重规则版本。
5. dataset builder 配方。
6. dataset version。
7. 训练任务 ID。
8. checkpoint。
9. eval report。
10. deployment version。

当模型出现安全问题、版权问题或质量问题时，血缘可以帮助追溯原因。

## 36.21 数据供给和训练平台的接口

训练平台不应该直接拼文件路径。

更好的接口是：

```text
TrainingJob
  dataset:
    name: pretrain_mix
    version: 2026-05-30-v1
    split: train
    access_mode: streaming
    shuffle: true
```

训练平台通过 dataset registry 解析出实际文件、权限、格式和读取方式。

这样训练任务更可复现，也更容易审计。

## 36.22 常见故障

数据供给链路常见故障：

1. 数据路径错误。
2. 文件缺失。
3. schema 不一致。
4. tokenizer 版本不匹配。
5. 小文件太多导致读取慢。
6. 对象存储限流。
7. shuffle 不充分。
8. 数据权限失败。
9. 某个 shard 损坏。
10. 数据分布突然变化。
11. eval 数据泄漏到 train。

这些问题都需要指标、日志和数据血缘来定位。

## 36.23 面试常见追问

问题一：为什么训练任务不应该直接引用数据目录？

可以回答：直接引用目录缺少版本、权限、质量统计和血缘信息，难以复现和审计。应该引用 dataset version 或 manifest。

问题二：数据供给为什么会影响 GPU 利用率？

可以回答：如果数据读取、解压、tokenization、shuffle 或网络吞吐跟不上，GPU 会等待 batch，导致利用率下降。

问题三：为什么要保留 raw data？

可以回答：清洗规则和数据配方会变化，保留 raw data 才能重建数据集、追溯问题和验证数据血缘。

问题四：大模型数据平台为什么要做血缘？

可以回答：模型质量、安全、合规和版权问题都需要追溯训练数据来源、清洗规则、dataset version、训练任务和 checkpoint。

## 36.25 本章小结

本章讲了数据平台与训练数据供给链路。

你需要记住：

1. 大模型数据平台不是文件目录，而是数据采集、清洗、治理、版本、质量、权限和供给系统。
2. 训练数据供给链路包括 ingestion、raw data lake、清洗、去重、脱敏、质量评分、dataset builder、版本注册和训练读取。
3. 训练任务应该引用 dataset version 或 manifest，而不是直接引用路径。
4. 数据读取吞吐会直接影响 GPU 利用率和训练效率。
5. 数据权限、血缘和质量监控是企业级数据平台的核心能力。
