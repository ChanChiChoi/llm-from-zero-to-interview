# 第 33 章 限流、熔断、重试、超时和降级

上一章讲了自动扩缩容。本章继续讲推理平台的稳定性保护机制：限流、熔断、重试、超时和降级。

大模型推理服务很贵，也很容易被长请求、高并发、异常重试和下游拥塞拖垮。稳定性设计的核心不是让每个请求都尽量执行，而是在系统压力过大时保护整体服务质量。

先记住一句话：

> 推理平台的稳定性不是“永不失败”，而是能控制失败范围、避免故障扩散，并在压力下优先保证关键请求。

## 33.0 本讲资料边界与第二轮精修口径

本讲按 `WRITING_PLAN.md` 的第二轮要求做过资料校准。重点参考的是 Google SRE 对 overload、criticality、per-customer limits 和 retry budget 的系统设计口径，AWS Builders Library 对 timeout、retry、exponential backoff、jitter、token bucket 和 idempotency 的工程经验，Envoy 对 rate limit、circuit breaking、outlier detection、timeout、retry 和 retry budget 的代理层抽象，gRPC 对重试提交点、退避、重试节流和服务配置的说明，以及 Resilience4j 对 circuit breaker 状态机、滑动窗口、failure rate、slow call rate 和 half-open 探测的公开文档。

这些资料共同指向一个稳定事实：保护机制不能只靠某一个开关。限流解决“谁可以用多少”，admission control 解决“系统现在接不接得住”，熔断解决“坏依赖是否继续放大故障”，重试解决“哪些临时失败值得再试”，超时解决“请求最多占用多久”，降级解决“压力下保哪些能力、退哪些能力”。

本章只抽象截至 2026-06 仍稳定的推理平台保护策略，不把某个 API Gateway、service mesh、云厂商、SDK、runtime 或内部平台的字段名写成通用标准。正文公式用于面试表达、策略审计和最小 demo；真实上线仍要用目标模型、租户协议、流量分布、SLO、成本预算、runtime 行为和事故复盘来校准阈值。

## 33.1 为什么推理平台需要保护机制

大模型推理有几个特点：

1. 单请求成本高。
2. 长请求会占用资源很久。
3. streaming 连接持续时间长。
4. KV cache 和显存是硬约束。
5. 冷启动慢，不能瞬间扩容。
6. 多租户之间会相互影响。
7. 重试可能放大流量。

如果没有保护机制，一个异常业务、一个超长 prompt、一个重试风暴，就可能拖垮整个平台。

## 33.2 五类保护机制

本章讨论五类机制：

1. 限流：控制进入系统的请求量。
2. 熔断：发现某个依赖或模型异常时临时切断调用。
3. 重试：在可恢复错误下重新尝试。
4. 超时：限制请求最多占用多久。
5. 降级：在资源紧张或故障时使用更低成本或更简单路径。

它们不是孤立的，而是组合使用。

## 33.3 限流解决什么问题

限流是控制流量进入系统。

目标是：

1. 防止某个租户占满资源。
2. 防止突发流量压垮 runtime。
3. 防止长请求耗尽 KV cache。
4. 保证高优先级请求。
5. 控制成本。

限流不是为了拒绝用户，而是为了避免所有用户一起变慢或失败。

## 33.4 限流维度

推理平台限流可以按多个维度设计：

1. 租户级。
2. 用户级。
3. API key 级。
4. 模型级。
5. endpoint 级。
6. 任务类型级。
7. token 级。
8. 并发级。
9. 成本预算级。
10. 上下文长度级。

普通服务常按 QPS 限流。

大模型推理还必须按 token 和并发限流。

## 33.5 QPS 限流为什么不够

两个用户都是 10 QPS：

1. 用户 A 每次输入 100 token，输出 50 token。
2. 用户 B 每次输入 50000 token，输出 2000 token。

如果只看 QPS，它们一样。

但资源消耗完全不同。

所以推理平台需要：

1. request rate limit。
2. input token rate limit。
3. output token rate limit。
4. concurrent request limit。
5. max context length。
6. max output tokens。

这才是 LLM-aware 限流。

## 33.6 Token Bucket 和 Leaky Bucket

常见限流算法包括 token bucket 和 leaky bucket。

Token bucket：系统以固定速率生成令牌，请求消耗令牌。桶里有令牌就允许通过，没有就拒绝或排队。它允许一定突发。

Leaky bucket：请求进入桶后以固定速率流出，更平滑，但对突发更严格。

大模型平台可以把令牌从“请求数”扩展成“token 数”或“成本额度”。

例如：

```text
tenant_a: 100k input tokens/min, 50k output tokens/min
tenant_b: 10 concurrent requests, max 8k context
```

这样比单纯 QPS 更符合资源消耗。

## 33.7 Admission Control 和限流的区别

限流通常基于配额和速率。

Admission control 更关注当前系统是否有能力接收这个请求。

例如：

1. KV cache 是否足够？
2. 队列是否过长？
3. 当前 p99 是否已经超标？
4. endpoint 是否健康？
5. 请求是否过长？

限流回答：“这个租户是否允许这么用？”

Admission control 回答：“系统现在是否接得住？”

两者都需要。

## 33.8 熔断解决什么问题

熔断用于阻止故障扩散。

如果某个模型 endpoint 持续超时、错误率升高或队列爆炸，router 应该暂时停止向它发送请求。

熔断对象可以是：

1. 某个 runtime 实例。
2. 某个 endpoint。
3. 某个模型版本。
4. 某个下游工具。
5. 某个外部 API。
6. 某个地域。

熔断的目标是快速隔离坏节点或坏依赖。

## 33.9 熔断状态

经典熔断器有三种状态：

1. Closed：正常调用。
2. Open：熔断，直接拒绝或走 fallback。
3. Half-open：放少量探测请求，判断是否恢复。

在推理平台中，Half-open 很重要。

因为模型加载、GPU OOM、runtime 抖动可能是短暂问题，也可能是持续问题。

恢复时不能立刻把全部流量打回去，要逐步探测。

## 33.10 熔断触发条件

常见触发条件：

1. 错误率超过阈值。
2. 超时率超过阈值。
3. p99 延迟超过阈值。
4. queue length 持续过高。
5. OOM 次数过多。
6. health check 失败。
7. admission reject rate 过高。
8. runtime crash loop。

熔断要避免误伤。

比如某个租户发来大量超长请求导致超时，不能简单认为整个模型坏了，可能应该对该租户限流。

## 33.11 重试的危险

重试看似能提升成功率，但在大模型推理中很危险。

原因是：

1. 请求成本高。
2. 重试会放大流量。
3. 非幂等输出可能不同。
4. 原请求可能还在执行。
5. streaming 请求重试体验复杂。
6. 多层服务都重试会造成重试风暴。

所以推理平台不能无脑重试。

## 33.12 什么错误可以重试

适合重试的情况：

1. 连接建立失败。
2. endpoint 临时不可达。
3. 请求尚未进入 runtime 执行。
4. 明确的 transient error。
5. router 选到不健康实例但尚未开始生成。

不适合重试的情况：

1. 请求已经生成部分 token。
2. 用户取消请求。
3. 输入超过限制。
4. 权限失败。
5. 安全策略拒绝。
6. 配额不足。
7. 业务参数错误。

重试前必须知道请求执行到了哪个阶段。

## 33.13 重试策略

好的重试策略包括：

1. 最大重试次数。
2. 指数退避。
3. jitter 随机抖动。
4. 总超时预算。
5. 幂等键。
6. retry budget。
7. 只对特定错误重试。
8. 不跨越安全和权限边界。

retry budget 很重要。

它限制系统整体用于重试的额外流量，避免重试把平台打垮。

## 33.14 超时为什么重要

超时限制请求最多占用资源多久。

没有超时会导致：

1. 长请求长期占用 KV cache。
2. streaming 连接不释放。
3. 上游一直等待。
4. 队列持续堆积。
5. 成本不可控。

推理平台必须设计多层超时。

## 33.15 多层超时

常见超时包括：

1. 网关接收超时。
2. 排队超时。
3. prefill 超时。
4. decode 总超时。
5. 单 token 生成超时。
6. streaming idle 超时。
7. 下游工具调用超时。
8. 客户端总超时。

不同超时要有层次。

例如客户端总超时 60 秒，服务端不能设置 120 秒才返回，否则结果没有意义。

## 33.16 Timeout Budget

Timeout budget 是把总时间预算分配给各阶段。

例如：

```text
total timeout: 30s
gateway + auth: 0.5s
queue wait: 2s
prefill: 5s
decode: 20s
postprocess: 0.5s
buffer: 2s
```

如果队列阶段已经消耗太多预算，后面可能应该拒绝、降级或减少 max_tokens，而不是继续完整生成。

Timeout budget 能避免“请求虽然成功但用户已经不需要”的情况。

## 33.17 降级解决什么问题

降级是在资源不足、依赖异常或成本超预算时，用更低成本、更快或更稳定的路径返回。

降级目标是：

1. 保持核心功能可用。
2. 控制延迟。
3. 控制成本。
4. 避免故障扩散。
5. 给用户一个可接受结果。

降级不是失败，而是有控制的能力退让。

## 33.18 常见降级方式

常见降级方式：

1. 大模型降到小模型。
2. reasoning 模型降到普通模型。
3. 长上下文截断。
4. 降低最大输出长度。
5. 关闭工具调用。
6. 关闭多轮深度推理。
7. 返回缓存结果。
8. 返回模板化答案。
9. 后台异步处理。
10. 提示用户稍后再试。

降级策略必须按业务场景设计。

对客服 FAQ，降级到缓存可能可接受。

对金融决策，错误降级可能不可接受。

## 33.19 降级策略要可观测

每次降级都要记录：

1. 请求 ID。
2. 租户。
3. 原计划模型。
4. 降级后模型或路径。
5. 降级原因。
6. 降级耗时。
7. 用户是否可见。
8. 质量反馈。
9. 成本影响。

没有记录，降级会变成隐形质量问题。

## 33.20 保护机制之间的组合

一个请求可能经历这样的保护链路：

```text
1. API Gateway 做认证和基础限流
2. Quota Service 检查租户 token 配额
3. Admission Controller 检查队列和 KV cache
4. Router 选择健康 endpoint
5. Circuit Breaker 跳过异常实例
6. Timeout Budget 控制各阶段耗时
7. Retry Policy 只对安全错误重试
8. Degradation Policy 在压力过大时降级
9. Metrics 和 Trace 记录全过程
```

稳定性设计要从入口一直贯穿到 runtime。

## 33.21 保护策略审计指标与最小 demo

把限流、熔断、重试、超时和降级放到同一章，容易让人误以为它们只是五个配置项。真正上线时，更重要的是把它们变成一条可审计的请求保护链路。

一个请求保护样本可以写成：

```math
p_i=(r_i,u_i,m_i,x_i,y_i,c_i,q_i,k_i,e_i,t_i,s_i,a_i,d_i,z_i)
```

其中，`r_i` 是请求 ID，`u_i` 是租户或用户，`m_i` 是目标模型或 endpoint，`x_i` 是 input token 数，`y_i` 是计划 output token 数，`c_i` 是当前并发，`q_i` 是队列等待，`k_i` 是 KV pressure，`e_i` 是错误类型，`t_i` 是 timeout budget，`s_i` 是执行阶段，`a_i` 是 retry attempt，`d_i` 是降级动作，`z_i` 是 trace / audit 字段。

Token bucket 可以从请求数扩展成 token 和成本。一个简化的桶余额更新是：

```math
B_{u,t+1}=\min(B_{\max},B_{u,t}+\rho_u\Delta t)-w_i
```

其中，`B_{u,t}` 是租户 `u` 在时刻 `t` 的余额，`B_{\max}` 是桶容量，`\rho_u` 是补充速率，`\Delta t` 是时间间隔，`w_i` 是请求消耗的权重。对 LLM 来说，`w_i` 不应该只是 1，而可以写成：

```math
w_i=\alpha_x x_i+\alpha_y y_i+\alpha_c c_i+\alpha_k k_i
```

其中，`\alpha_x`、`\alpha_y`、`\alpha_c` 和 `\alpha_k` 分别控制输入 token、输出 token、并发占用和 KV 压力的权重。这样能解释为什么“同样 1 个请求”可能消耗完全不同的资源。

Admission control 可以写成一个即时门禁：

```math
A_i=\mathbf{1}\left[
B_{\mathrm{req}}\ge 1 \land B_{\mathrm{tok}}\ge x_i+y_i \land C_u<C_{u,\max} \land Q\le Q_{\max} \land R_{\mathrm{kv}}\le \rho_{\mathrm{kv}} \land T_{\mathrm{rem}}\ge T_{\min}
\right]
```

其中，`B_{\mathrm{req}}` 是请求数配额余额，`B_{\mathrm{tok}}` 是 token 配额余额，`C_u` 是租户当前并发，`Q` 是队列水位，`R_{\mathrm{kv}}` 是 KV pressure，`T_{\mathrm{rem}}` 是剩余 timeout budget。限流偏静态配额，admission 偏动态容量；两者缺一不可。

熔断要基于滑动窗口，而不是单个错误。对 endpoint `j`，可以定义：

```math
E_j=\frac{N_{\mathrm{error},j}}{N_{\mathrm{total},j}},\qquad
S_j=\frac{N_{\mathrm{slow},j}}{N_{\mathrm{total},j}}
```

其中，`E_j` 是错误率，`S_j` 是慢调用率。一个简化的熔断触发条件是：

```math
O_j=\mathbf{1}\left[
E_j\ge\theta_e \lor S_j\ge\theta_s \lor T_{j,\mathrm{p99}}\ge B_{99} \lor Q_j\ge Q_{\mathrm{open}} \lor R_{\mathrm{kv},j}\ge \rho_{\mathrm{open}}
\right]
```

其中，`O_j=1` 表示进入 open 状态。half-open 阶段不应该立刻放全量流量，而应该只允许 `H_{\max}` 个探测请求：

```math
H_j=\min(H_{\max},N_{\mathrm{probe},j})
```

重试要有预算。若初始请求数为 `N_{\mathrm{init}}`，重试请求数为 `N_{\mathrm{retry}}`，一个简单的 retry budget 是：

```math
R_{\mathrm{retry}}=\frac{N_{\mathrm{retry}}}{N_{\mathrm{init}}+N_{\mathrm{retry}}}\le \beta
```

其中，`\beta` 是允许的重试流量占比。指数退避加 jitter 可以写成：

```math
D_a=\min(D_{\max},D_0 2^{a-1})+\epsilon_a,\qquad 0\le\epsilon_a\le J_a
```

其中，`a` 是第几次 retry，`D_0` 是初始退避，`D_{\max}` 是最大退避，`\epsilon_a` 是随机抖动。重试还必须满足阶段和幂等条件：

```math
G_{\mathrm{retry}}=\mathbf{1}\left[
e_i\in\mathcal{E}_{\mathrm{transient}} \land s_i=\mathrm{not\_started} \land a_i<a_{\max} \land I_i=1 \land R_{\mathrm{retry}}\le\beta \land T_{\mathrm{rem}}\ge D_a+T_{\min}
\right]
```

其中，`\mathcal{E}_{\mathrm{transient}}` 是可重试错误集合，`I_i` 表示请求有幂等键或可去重。对已经开始 streaming、已经产生 side effect、权限失败、配额不足或安全拒绝的请求，`G_{\mathrm{retry}}` 必须为 0。

Timeout budget 要保证每一层都受总 deadline 约束：

```math
T_{\mathrm{total}}=T_{\mathrm{gateway}}+T_{\mathrm{queue}}+T_{\mathrm{prefill}}+T_{\mathrm{decode}}+T_{\mathrm{stream}}+T_{\mathrm{post}}+T_{\mathrm{buf}}
```

剩余预算为：

```math
T_{\mathrm{rem}}=T_{\mathrm{deadline}}-\sum_h T_h
```

如果 `T_{\mathrm{rem}}` 已经不足以完成完整 decode，就应该拒绝、减少输出长度或降级，而不是继续让请求占住 KV cache。

降级也要有约束。设可选降级集合为 `\mathcal{D}_i`，每个降级动作 `d` 有质量估计 `Q(d)`、成本 `K(d)` 和延迟 `L(d)`，可以写成：

```math
d_i^*=\arg\max_{d\in\mathcal{D}_i} Q(d)
```

并满足：

```math
L(d)\le B_{\mathrm{lat}} \land K(d)\le B_{\mathrm{cost}} \land P(d)=1
```

其中，`P(d)` 表示该业务、租户和风险等级允许这个降级动作。客服 FAQ 可以降级到缓存，金融决策可能只能拒绝或转人工；这不是技术偏好，而是业务风险边界。

最终保护策略门禁可以写成：

```math
G_{\mathrm{protect}}=\mathbf{1}\left[
\min_j C_j\ge\tau_j \land R_{\mathrm{retry}}\le\beta \land R_{\mathrm{timeout}}\le \omega \land R_{\mathrm{open\_miss}}=0 \land P_0=0
\right]
```

其中，`C_j` 是各审计维度覆盖率，`R_{\mathrm{timeout}}` 是超时率，`R_{\mathrm{open\_miss}}` 是应该熔断却继续放量的比例，`P_0` 是未关闭的 P0 风险数。这个门禁强调：保护机制不是单个中间件配置，而是入口、路由、runtime、缓存、降级、trace 和事故治理共同组成的控制面。

def estimate_cost(request):
    return round(
        request["input_tokens"] * POLICY["input_cost"]
        + request["max_output_tokens"] * POLICY["output_cost"],
        4,
    )


def endpoint_state(name):
    endpoint = ENDPOINTS[name]
    should_open = (
        endpoint["error_rate"] >= POLICY["error_open"]
        or endpoint["slow_rate"] >= POLICY["slow_open"]
        or endpoint["p99_s"] >= POLICY["p99_open_s"]
        or endpoint["queue"] >= POLICY["queue_open"]
        or endpoint["kv"] >= POLICY["kv_open"]
    )
    return "open" if should_open else endpoint["state"]


def rate_limit_decision(request):
    tenant = request["tenant"]
    limits = TENANT_LIMITS[tenant]
    usage = USAGE[tenant]
    if usage["requests"] + 1 > limits["requests_per_min"]:
        return False, "request_rate_limit"
    if usage["active"] + 1 > limits["concurrency"]:
        return False, "concurrency_limit"
    if request["input_tokens"] > limits["context_limit"]:
        return False, "context_limit"
    if usage["input_tokens"] + request["input_tokens"] > limits["input_tpm"]:
        return False, "input_token_limit"
    if usage["output_tokens"] + request["max_output_tokens"] > limits["output_tpm"]:
        return False, "output_token_limit"
    if usage["cost"] + estimate_cost(request) > limits["cost_budget"]:
        return False, "cost_budget_limit"
    return True, "pass"


def remaining_timeout(request):
    return round(request["timeout_budget_s"] - request["elapsed_s"], 3)


def retry_delay(attempt):
    base = min(POLICY["retry_max_delay_s"], POLICY["retry_base_delay_s"] * (2 ** attempt))
    return round(base + POLICY["retry_jitter_s"], 3)


def retry_budget_ok(request):
    used = request.get("retry_used_override", POLICY["retry_used"])
    initial = POLICY["retry_initial"]
    retry_ratio = (used + 1) / (initial + used + 1)
    return retry_ratio <= POLICY["retry_budget_ratio"], round(retry_ratio, 3)


def can_retry(request):
    error = request.get("error")
    if error is None:
        return False, "no_error", None
    if error in NON_RETRYABLE_ERRORS:
        return False, "non_retryable_error", None
    if error not in TRANSIENT_ERRORS:
        return False, "unknown_error_class", None
    if request["stage"] != "not_started" or request.get("partial_tokens", 0) > 0:
        return False, "already_started_or_streaming", None
    if request["attempt"] >= POLICY["retry_max_attempts"]:
        return False, "max_attempts", None
    if not request["idempotent"]:
        return False, "missing_idempotency", None
    budget_ok, retry_ratio = retry_budget_ok(request)
    if not budget_ok:
        return False, "retry_budget_exhausted", retry_ratio
    delay = retry_delay(request["attempt"])
    if remaining_timeout(request) < delay + POLICY["timeout_min_s"]:
        return False, "timeout_budget_too_small", retry_ratio
    return True, "retry_scheduled", retry_ratio


def admission_decision(request, endpoint_name):
    endpoint = ENDPOINTS[endpoint_name]
    observed_kv = request.get("observed_kv", endpoint["kv"])
    if endpoint["queue"] > POLICY["queue_limit"] or observed_kv > POLICY["kv_limit"]:
        return False, "capacity_pressure"
    if remaining_timeout(request) < POLICY["timeout_min_s"]:
        return False, "timeout_budget_too_small"
    return True, "pass"


def degrade(request, reason):
    tenant_policy = TENANT_LIMITS[request["tenant"]]
    if not tenant_policy["allow_degrade"]:
        return "reject_" + reason
    if reason == "capacity_pressure":
        reduced_output = max(64, request["max_output_tokens"] // 2)
        return "degrade_small_model_output_%d" % reduced_output
    if reason == "circuit_open" and request.get("fallback"):
        return "fallback_" + request["fallback"]
    return "degrade_cached_or_async"


def protect_request(request):
    trace = []
    rate_ok, rate_reason = rate_limit_decision(request)
    trace.append(("rate_limit", rate_reason))
    if not rate_ok:
        return "rate_limited_" + rate_reason, trace

    retry_ok, retry_reason, retry_ratio = can_retry(request)
    if request.get("error") is not None:
        trace.append(("retry", retry_reason if retry_ratio is None else "%s@%s" % (retry_reason, retry_ratio)))
        if retry_ok:
            return "retry_scheduled", trace
        return "no_retry_" + retry_reason, trace

    state = endpoint_state(request["endpoint"])
    trace.append(("circuit", state))
    if state == "open":
        return degrade(request, "circuit_open"), trace

    admission_ok, admission_reason = admission_decision(request, request["endpoint"])
    trace.append(("admission", admission_reason))
    if not admission_ok:
        return degrade(request, admission_reason), trace

    return "admitted_" + request["endpoint"], trace

## 33.22 常见误区

误区一：失败就重试。

大模型请求成本高，重试可能放大故障，必须限制重试条件和预算。

误区二：限流只按 QPS。

LLM 推理要按 token、并发、上下文长度和成本预算限流。

误区三：超时只设一个总超时。

推理链路很长，需要排队、prefill、decode、streaming 和工具调用等多层超时。

误区四：降级就是换小模型。

降级还可以是缓存、截断、减少输出、异步处理或模板化响应。

误区五：熔断只看错误率。

对推理服务，队列长度、超时率、OOM、KV cache 水位和 p99 也可能触发熔断。

## 33.23 面试常见追问

问题一：LLM 服务怎么做限流？

可以回答：不能只按 QPS，要按租户、API key、模型、并发、input tokens/s、output tokens/s、最大上下文、最大输出和成本预算做多维限流。

问题二：为什么推理请求不能无脑重试？

可以回答：因为请求成本高，可能已经在 runtime 执行或已返回部分 token，重试会放大流量且输出非幂等。应只对尚未执行的临时错误重试，并设置重试预算。

问题三：如何设计超时？

可以回答：使用 timeout budget，把总超时拆到网关、排队、prefill、decode、streaming、工具调用和后处理阶段，并根据剩余预算决定是否继续、降级或拒绝。

问题四：降级策略怎么设计？

可以回答：根据业务可接受范围设计降级路径，例如小模型、缓存、截断上下文、减少输出、关闭工具调用或异步处理，同时记录降级原因和质量影响。

问题五：如何把限流、熔断、重试、超时和降级串成一条链路？

可以回答：入口先做认证、租户限流、token / 并发 / 成本配额；admission control 再看队列、KV、剩余 timeout 和 runtime 健康；router 跳过 open endpoint；可重试错误必须满足 not started、幂等、retry budget 和剩余 deadline；资源不足或依赖异常时按业务策略降级；全程记录 request id、tenant、policy version、decision reason、fallback、retry attempt、timeout 和成本。

## 33.25 本章小结

本章讲了限流、熔断、重试、超时和降级。

你需要记住：

1. 推理平台稳定性设计的目标是控制故障范围，而不是保证每个请求都成功。
2. 限流要按请求、token、并发、上下文、租户和成本多维设计。
3. 熔断用于隔离异常模型、实例、工具或地域，防止故障扩散。
4. 重试必须谨慎，只对安全的临时错误重试，并设置 retry budget。
5. 超时要分阶段设计，使用 timeout budget 管理总耗时。
6. 降级是有控制的能力退让，必须可观测、可审计、可评估质量影响。
7. 五类保护机制要串成同一条 request protection trace，否则很难解释一次拒绝、重试或降级到底是否正确。
