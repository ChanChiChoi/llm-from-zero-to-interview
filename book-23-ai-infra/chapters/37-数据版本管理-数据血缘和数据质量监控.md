# 第 37 章 数据版本管理、数据血缘和数据质量监控

上一章讲了数据平台与训练数据供给链路。本章继续深入三个关键能力：数据版本管理、数据血缘和数据质量监控。

这三个能力决定了一个训练数据平台是否可复现、可追溯、可治理。没有它们，大模型训练就会变成“这次效果好像变差了，但没人知道为什么”。

先记住一句话：

> 数据平台的核心不是存更多数据，而是让每一次数据使用都能被复现、解释、追溯和监控。

## 37.1 为什么数据版本管理重要

大模型训练结果高度依赖数据。

如果数据发生变化，即使代码、模型结构和超参都不变，训练结果也可能变化。

常见变化包括：

1. 新增数据源。
2. 删除某批低质量数据。
3. 修改清洗规则。
4. 修改去重规则。
5. 修改采样比例。
6. 修改 train/val/test 切分。
7. 修改 tokenizer。
8. 修复数据格式错误。

如果没有数据版本，就无法判断模型效果变化来自代码、参数还是数据。

## 37.2 数据版本和代码版本的区别

代码版本通常是 Git commit。

数据版本更复杂，因为数据：

1. 体积巨大。
2. 来源多样。
3. 处理链路长。
4. 可能包含权限和合规约束。
5. 可能持续增量更新。
6. 不适合全部放进 Git。

所以数据版本通常由 manifest、元数据、文件 checksum、处理规则版本和存储路径共同定义。

## 37.3 一个 Dataset Version 应该包含什么

一个 dataset version 至少要记录：

1. dataset 名称。
2. 版本号。
3. 创建时间。
4. 创建人或任务。
5. 数据来源列表。
6. 清洗规则版本。
7. 去重规则版本。
8. 过滤规则版本。
9. 采样配方。
10. 文件 manifest。
11. 样本数量。
12. token 数量估计。
13. train/val/test split。
14. 数据质量统计。
15. 权限等级。
16. checksum。

训练任务引用 dataset version，而不是引用一个随时可能变化的目录。

## 37.4 Manifest 版本化

Manifest 是数据版本管理的核心。

它可以描述：

```text
dataset: pretrain_mix
version: 2026-05-30-v1
files:
  - path: s3://bucket/datasets/pretrain_mix/v1/shard-0001.parquet
    checksum: xxx
    samples: 1000000
    tokens: 850000000
  - path: s3://bucket/datasets/pretrain_mix/v1/shard-0002.parquet
    checksum: yyy
    samples: 980000
    tokens: 820000000
recipe:
  web: 40%
  code: 30%
  books: 20%
  qa: 10%
cleaning_rule_version: clean-v17
dedup_rule_version: dedup-v9
```

有了 manifest，训练任务才能精确知道自己读了哪些文件和规则。

## 37.5 不可变数据版本

生产级 dataset version 应该尽量不可变。

也就是说：一旦 `pretrain_mix:v1` 被发布，就不要悄悄修改它的文件内容。

如果需要修复，发布新版本：

```text
pretrain_mix:v1 -> pretrain_mix:v2
```

不可变版本的好处：

1. 训练可复现。
2. 问题可追溯。
3. 审计更清晰。
4. cache 更安全。
5. 多任务共享更可靠。

如果版本名不变但内容变了，后续所有分析都会混乱。

## 37.6 增量版本

有些数据是持续增长的，例如日志、对话、文档更新和用户反馈。

这时可以使用增量版本：

```text
dataset:v1 = base data
dataset:v2 = v1 + delta_2026_05
dataset:v3 = v2 + delta_2026_06 - bad_source_x
```

增量版本要记录：

1. base version。
2. 增加了哪些数据。
3. 删除了哪些数据。
4. 修改了哪些规则。
5. 新版本质量差异。

增量不是简单追加文件，而是受治理的版本演进。

## 37.7 数据血缘是什么

数据血缘是记录数据从来源到使用的完整链路。

它回答这些问题：

1. 某个 dataset 来自哪些原始数据？
2. 经过了哪些清洗和过滤任务？
3. 使用了哪些规则版本？
4. 被哪些训练任务使用？
5. 产生了哪些 checkpoint？
6. checkpoint 经过了哪些评估？
7. 最终部署到了哪些线上模型？

血缘把 raw data、dataset、training job、checkpoint、eval report 和 deployment 串起来。

## 37.8 数据血缘图

可以把血缘理解为一张图：

```text
RawSource
  -> IngestionJob
  -> RawDataset
  -> CleaningJob
  -> CleanDataset
  -> DedupJob
  -> DatasetVersion
  -> TrainingJob
  -> Checkpoint
  -> EvalReport
  -> ModelVersion
  -> Deployment
```

每个节点都有元数据，每条边表示依赖关系。

有了这张图，平台才能回答“这个线上模型到底吃过哪些数据”。

## 37.9 为什么血缘重要

血缘的价值包括：

1. 复现训练。
2. 定位质量问题。
3. 追踪安全问题。
4. 支持合规审计。
5. 评估数据删除影响。
6. 分析模型退化原因。
7. 做成本归因。
8. 支持数据资产管理。

例如发现某个数据源有许可证风险，就需要知道哪些 dataset、checkpoint 和线上模型使用了它。

没有血缘，只能人工翻日志，几乎不可维护。

## 37.10 数据删除和影响分析

企业场景中经常会遇到数据删除或撤回需求。

例如：

1. 某个文档不再允许训练。
2. 某批用户数据需要删除。
3. 某个数据源许可证变化。
4. 某个业务线撤回授权。

血缘可以帮助做影响分析：

```text
这个数据源 -> 哪些 dataset version -> 哪些 training job -> 哪些 checkpoint -> 哪些 deployment
```

注意：模型已经训练过的数据不一定能简单从权重中删除。血缘至少能帮助判断影响范围，并决定是否重训、下线或标记风险。

## 37.11 数据质量监控为什么重要

数据质量不是构建数据集时检查一次就结束。

数据源会变化，清洗规则会变化，业务数据分布会变化。

质量监控要持续进行。

常见问题：

1. 某个数据源突然乱码增多。
2. 某个采集任务重复抓取。
3. 某个清洗规则误删大量样本。
4. 某类语言比例异常变化。
5. 敏感信息过滤失效。
6. train/eval 泄漏。
7. 标注质量下降。
8. 合成数据比例过高。

这些都会影响训练结果。

## 37.12 数据质量指标

常见数据质量指标包括：

基础统计：

1. 样本数。
2. token 数。
3. 文件数。
4. shard 大小。
5. 平均样本长度。
6. 长度分布。

内容质量：

1. 语言分布。
2. 领域分布。
3. 重复率。
4. 低质量文本比例。
5. 乱码比例。
6. 毒性或风险内容比例。

结构质量：

1. schema 合法率。
2. JSON 解析成功率。
3. 空字段比例。
4. 标签缺失率。
5. 多模态文件缺失率。

安全合规：

1. PII 命中率。
2. 密钥或凭据命中率。
3. 许可证风险比例。
4. 敏感等级分布。

## 37.13 分布漂移监控

数据分布漂移是指数据分布和历史版本相比发生明显变化。

例如：

1. 英文比例从 50% 变成 80%。
2. 代码数据比例突然下降。
3. 某个领域数据突然增多。
4. 平均样本长度翻倍。
5. 合成数据比例过高。

漂移不一定是坏事，但必须被发现和解释。

平台可以对比当前 dataset version 和上一个 version 的统计差异，并生成 diff report。

## 37.14 Dataset Diff Report

Dataset diff report 用于比较两个数据版本。

它可以包含：

1. 新增样本数。
2. 删除样本数。
3. token 数变化。
4. 来源比例变化。
5. 语言分布变化。
6. 领域分布变化。
7. 重复率变化。
8. PII 命中率变化。
9. 质量分分布变化。
10. 采样配方变化。

训练效果变化时，diff report 是排查入口之一。

如果模型突然在代码任务上退化，先看代码数据比例、质量和去重策略有没有变化。

## 37.15 数据质量门禁

数据质量门禁是 dataset 发布前的检查。

例如：

1. schema 合法率必须大于 99.9%。
2. 重复率不能超过阈值。
3. PII 命中率不能超过阈值。
4. train/eval overlap 必须低于阈值。
5. 关键来源比例不能异常变化。
6. token 数必须达到预期。
7. shard checksum 必须通过。

不通过门禁的数据集不能进入训练。

这和代码 CI 类似，只不过检查对象变成数据。

## 37.16 Train/Eval 泄漏监控

Train/eval 泄漏是大模型评估中的严重问题。

如果 eval 样本出现在 train 数据中，评估结果会虚高。

泄漏形式包括：

1. 完全重复。
2. 近似重复。
3. 同一问题不同表述。
4. benchmark 数据被网页爬取后进入训练。
5. 标注数据同时进入训练和评估。

监控方法包括：

1. hash 去重。
2. MinHash / SimHash。
3. embedding 相似度。
4. benchmark contamination 检查。
5. 数据来源黑名单。

评估可信度依赖数据隔离。

## 37.17 标注数据质量监控

SFT、RLHF、偏好数据和人工评测都依赖标注质量。

标注数据要监控：

1. 标注员一致性。
2. 标签分布。
3. 标注耗时。
4. 低质量标注比例。
5. 冲突样本比例。
6. gold set 准确率。
7. 审核通过率。
8. 标注指南版本。

如果标注指南变化，也要记录版本。

否则模型行为变化可能来自标注标准变化，而不是算法改进。

## 37.18 合成数据质量监控

合成数据越来越常见，但也有风险。

需要监控：

1. 生成模型版本。
2. 生成 prompt 版本。
3. 采样参数。
4. 去重率。
5. 多样性。
6. 事实性。
7. 难度分布。
8. 与真实数据比例。
9. 是否引入偏差。
10. 是否被目标模型过度拟合。

合成数据不是免费质量提升，必须有质量评估和版本追踪。

## 37.19 数据质量告警

数据质量监控应该产生告警。

例如：

1. 样本量低于预期。
2. 某来源数据突然为 0。
3. PII 命中率升高。
4. JSON 解析失败率升高。
5. 重复率异常升高。
6. 语言分布漂移。
7. shard checksum 失败。
8. eval overlap 超阈值。

告警要能定位到：

1. 哪个 dataset。
2. 哪个 version。
3. 哪个数据源。
4. 哪个处理任务。
5. 哪个规则版本。

否则告警只会制造噪音。

## 37.20 数据版本和训练任务联动

训练任务提交时应记录：

1. dataset name。
2. dataset version。
3. manifest checksum。
4. tokenizer version。
5. data loader version。
6. sampling seed。
7. shuffle config。
8. split 信息。

这样才能复现一次训练。

如果训练只记录了路径，路径下内容又变化了，就无法复现。

## 37.21 数据版本和模型版本联动

模型版本应该记录训练所用数据。

例如：

```text
ModelVersion: chat-7b-sft-v12
base_model: base-7b-v3
sft_dataset: sft_mix:2026-05-30-v1
preference_dataset: pref_mix:2026-05-20-v2
eval_report: eval-2026-05-31
```

这样线上模型出问题时，可以沿模型版本追溯到 dataset version。

模型仓库和数据仓库不能割裂。

## 37.22 常见系统设计

一个数据版本、血缘和质量平台可以包含：

```text
Data Catalog
  -> Dataset Registry
  -> Manifest Store
  -> Lineage Graph
  -> Quality Metrics Store
  -> Quality Gate Service
  -> Diff Report Service
  -> Access Control
  -> Audit Log
```

模块职责：

1. Data Catalog：查找数据资产。
2. Dataset Registry：注册 dataset 和版本。
3. Manifest Store：保存文件列表和 checksum。
4. Lineage Graph：记录依赖关系。
5. Quality Metrics Store：保存质量指标。
6. Quality Gate Service：发布前门禁。
7. Diff Report Service：版本对比。
8. Access Control：权限控制。
9. Audit Log：记录访问和变更。

## 37.23 常见误区

误区一：数据版本就是目录名。

目录名不够，版本还要包含 manifest、规则版本、checksum、配方和质量统计。

误区二：数据血缘只是合规需要。

血缘也用于训练复现、质量排查、删除影响分析和成本归因。

误区三：质量检查只在构建时做一次。

数据源和规则会变化，质量需要持续监控和版本对比。

误区四：eval 分数可信。

如果 train/eval 泄漏，eval 分数可能严重虚高。

误区五：合成数据越多越好。

合成数据需要监控多样性、事实性、难度和偏差。

## 37.24 面试常见追问

问题一：如何保证训练数据可复现？

可以回答：训练任务引用不可变 dataset version，版本中包含 manifest、文件 checksum、清洗规则、去重规则、采样配方、split、tokenizer 和质量统计，并在训练任务中记录 manifest checksum 和随机种子。

问题二：数据血缘怎么设计？

可以回答：用 lineage graph 记录 raw source、ingestion、cleaning、dedup、dataset version、training job、checkpoint、eval report、model version 和 deployment 的依赖关系，支持从线上模型反查训练数据。

问题三：数据质量监控看哪些指标？

可以回答：看样本数、token 数、长度分布、语言/领域分布、重复率、schema 合法率、PII 命中率、许可证风险、train/eval overlap、标注一致性和版本 diff。

问题四：如何发现 train/eval 泄漏？

可以回答：使用 hash 精确匹配、MinHash/SimHash 近似匹配、embedding 相似度、benchmark contamination 检查和数据来源黑名单，检查 train 与 eval 之间的重复和近似重复。

## 37.26 本章小结

本章讲了数据版本管理、数据血缘和数据质量监控。

你需要记住：

1. 数据版本管理的目标是训练可复现，不是简单给目录起名字。
2. Dataset version 应包含 manifest、规则版本、checksum、采样配方、质量统计和权限信息。
3. 数据血缘把 raw data、dataset、training job、checkpoint、eval report、model version 和 deployment 串起来。
4. 数据质量监控要覆盖基础统计、内容质量、结构质量、安全合规、分布漂移和 train/eval 泄漏。
5. 数据质量门禁和 diff report 是数据平台工程化治理的重要工具。
