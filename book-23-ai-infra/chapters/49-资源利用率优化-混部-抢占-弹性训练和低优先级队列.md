# 第 49 章 资源利用率优化：混部、抢占、弹性训练和低优先级队列

上一章讲了成本治理。本章继续讲资源利用率优化。

AI Infra 中最昂贵的资源通常是 GPU。GPU 买了或租了以后，如果长期空闲，就是直接浪费。但为了保证训练任务和线上推理的稳定性，又不能简单把所有资源塞满。因此资源利用率优化的核心，是在稳定性、隔离、公平性和成本之间做工程权衡。

先记住一句话：

> GPU 利用率优化不是把集群打满，而是在满足优先级、SLO 和隔离约束下，减少空闲、减少碎片、减少失败和提高有效产出。

## 49.1 为什么资源利用率难

GPU 集群利用率难优化，原因包括：

1. 任务资源需求差异大。
2. 大模型训练需要 gang scheduling。
3. 多机多卡任务容易产生资源碎片。
4. 推理服务需要保留冗余和 warm pool。
5. 训练任务运行时间长。
6. 低优先级任务可能被抢占。
7. 不同 GPU 型号不可完全替代。
8. 数据、网络、存储也会限制 GPU 使用。

所以利用率优化不是单纯提高 GPU utilization，而是整体资源调度问题。

## 49.2 利用率指标

常见指标包括：

1. GPU allocation rate。
2. GPU utilization。
3. GPU memory utilization。
4. MFU。
5. tokens per GPU hour。
6. idle GPU hours。
7. pending jobs。
8. queue wait time。
9. preempted job count。
10. failed job GPU hours。
11. fragmentation rate。

GPU allocation rate 表示 GPU 是否被分配。

GPU utilization 表示 GPU 分配后是否真的忙。

两者都要看。

GPU 被分配但利用率很低，可能是数据、通信或配置问题。

## 49.3 有效利用率

有效利用率比表面利用率更重要。

例如：

1. GPU 满载跑失败任务，不是有效利用。
2. GPU 满载跑错误配置实验，不是有效利用。
3. GPU 满载但训练数据重复严重，产出价值低。
4. 推理 GPU 满载但 p99 崩掉，也不是好状态。

有效利用率要结合产出：

1. 成功训练 step。
2. 有效 tokens。
3. 成功请求。
4. 符合 SLO 的吞吐。
5. 可发布模型产物。

## 49.4 资源碎片

资源碎片是 GPU 集群常见问题。

例如集群中有 8 台机器，每台 8 卡，但空闲 GPU 分散在不同机器上：

```text
node1: 2 free
node2: 1 free
node3: 3 free
node4: 2 free
```

总共有 8 张空闲 GPU，但没有一台机器有 8 张连续 GPU，无法启动一个单机 8 卡任务。

多机多卡任务还需要网络拓扑匹配，碎片问题更严重。

## 49.5 减少碎片的方法

常见方法：

1. 拓扑感知调度。
2. bin packing。
3. 大任务优先或预留。
4. 队列重排。
5. 低优先级任务可抢占。
6. 任务资源规格标准化。
7. 避免小任务随意占用整机关键位置。
8. 定期 defragmentation。

调度器要知道 GPU 拓扑，而不是只看剩余卡数。

## 49.6 混部是什么

混部是把不同类型任务放在同一资源池或同一节点上运行，以提高资源利用率。

常见混部：

1. 训练和推理混部。
2. 在线和离线混部。
3. 高优先级和低优先级混部。
4. GPU 任务和 CPU 任务混部。
5. embedding 任务和训练任务混部。

混部目标是利用资源空隙。

但混部会带来干扰。

## 49.7 混部风险

混部风险包括：

1. GPU 显存竞争。
2. CPU 竞争。
3. 内存竞争。
4. 网络竞争。
5. 磁盘 I/O 竞争。
6. 推理 p99 延迟抖动。
7. 训练 step time 抖动。
8. 故障影响扩大。

在线推理任务和离线训练混部尤其要谨慎。

如果离线任务导致在线推理 p99 上升，节省的成本可能不值得。

## 49.8 安全混部策略

更安全的混部方式：

1. 按优先级隔离。
2. 按 GPU 型号隔离。
3. 在线服务保留资源。
4. 离线任务只使用可抢占资源。
5. 限制低优先级任务资源上限。
6. 使用 node taint/toleration。
7. 使用 QoS class。
8. 对在线服务设置 latency guardrail。
9. 异常时自动驱逐低优先级任务。

混部必须有监控和自动保护，不应靠人工观察。

## 49.9 抢占是什么

抢占是高优先级任务需要资源时，平台停止或迁移低优先级任务，释放资源。

抢占适合：

1. 低优先级训练。
2. 探索性实验。
3. 离线 embedding。
4. 批量评估。
5. 可恢复任务。

不适合：

1. 关键线上推理。
2. 无 checkpoint 的长任务。
3. 强实时任务。
4. 不可重复执行任务。

抢占的前提是任务能恢复。

## 49.10 抢占设计

抢占系统要考虑：

1. 谁可以抢占谁。
2. 抢占前是否保存 checkpoint。
3. 给任务多少 graceful shutdown 时间。
4. 被抢占任务如何重新排队。
5. 抢占次数是否有限制。
6. 是否补偿低优先级队列。
7. 抢占事件如何通知用户。

抢占不能粗暴 kill。

否则会浪费大量已完成训练进度。

## 49.11 低优先级队列

低优先级队列用于吸收闲置资源。

适合放：

1. 探索实验。
2. 批量评估。
3. embedding 生成。
4. 数据处理。
5. 非紧急微调。
6. 可中断任务。

低优先级队列的特点：

1. 等待时间不保证。
2. 可被抢占。
3. 成本较低。
4. 适合填充资源空洞。

这能提升集群整体利用率。

## 49.12 弹性训练是什么

弹性训练是训练任务可以在不同资源规模下运行。

例如一个任务可以从 64 卡扩到 128 卡，也可以从 128 卡缩到 64 卡。

弹性训练的价值：

1. 提高资源利用率。
2. 减少排队等待。
3. 支持抢占后继续运行。
4. 利用临时空闲 GPU。
5. 高峰期释放资源给更重要任务。

但弹性训练对框架和训练逻辑要求高。

## 49.13 弹性训练的难点

难点包括：

1. world size 变化。
2. global batch size 变化。
3. learning rate 调整。
4. optimizer state 重分片。
5. dataloader 分片变化。
6. checkpoint 格式兼容。
7. 通信拓扑变化。
8. 收敛稳定性影响。

弹性不是简单增加或减少 Pod。

如果 global batch 变化没有处理好，训练动态会改变。

## 49.14 弹性训练策略

常见策略：

1. 固定 global batch，改变 gradient accumulation。
2. 固定 learning rate schedule。
3. 支持 elastic dataloader。
4. checkpoint 支持不同 world size restore。
5. 只在安全边界内扩缩容。
6. 对弹性事件记录 trace。

弹性训练适合成熟平台，不适合一开始就强行支持所有任务。

## 49.15 推理资源利用率优化

推理资源优化方向：

1. continuous batching。
2. KV cache 管理。
3. 模型量化。
4. 多模型路由。
5. 小模型优先。
6. 自动扩缩容。
7. warm pool 优化。
8. cache 提升命中。
9. 低峰缩容。
10. 分离 prefill/decode。

推理资源利用率不能只追求 GPU 打满。

在线服务还要满足 TTFT、TPOT 和 p99。

## 49.16 Warm Pool 优化

Warm pool 可以降低冷启动，但会增加空闲成本。

优化方式：

1. 按模型热度设置最小副本。
2. 按时间预测扩容。
3. 热模型常驻。
4. 冷模型按需加载。
5. 权重本地缓存。
6. 多模型共享基础镜像。
7. 对低优先级模型允许冷启动。

不同业务需要不同 warm pool 策略。

高价值在线业务需要更高可用性，低频内部模型可以接受冷启动。

## 49.17 资源预留

资源预留用于保障关键任务。

常见预留：

1. 线上推理保留 GPU。
2. 关键训练项目保留配额。
3. 高优先级队列保留容量。
4. 故障恢复保留冗余。
5. 峰值流量前预留资源。

预留会降低表面利用率，但提高稳定性。

因此要平衡：预留太少影响 SLO，预留太多造成浪费。

## 49.18 配额和公平性

资源利用率优化不能牺牲公平性。

配额维度包括：

1. 团队。
2. 租户。
3. 项目。
4. 队列。
5. 优先级。
6. GPU 类型。

公平性策略：

1. dominant resource fairness。
2. weighted fair sharing。
3. quota borrowing。
4. priority-based scheduling。
5. preemption compensation。

允许借用空闲配额，可以提高利用率。

但借用资源要能在原 owner 需要时归还。

## 49.19 利用率优化和可靠性

利用率越高，系统余量越少。

风险包括：

1. 故障恢复空间不足。
2. 突发流量承接困难。
3. 训练任务排队增加。
4. 推理 p99 抖动。
5. 抢占频繁。

所以不能追求 100% 利用率。

平台要根据 SLO 保留合理 headroom。

Headroom 是为了应对波动和故障的安全余量。

## 49.20 调度器能力

高利用率需要强调度器能力：

1. gang scheduling。
2. 拓扑感知。
3. 队列和优先级。
4. 抢占。
5. 配额借用。
6. 资源碎片感知。
7. 多资源调度。
8. 弹性任务支持。
9. 调度模拟。
10. 调度事件可观测。

调度器是 AI Infra 利用率优化的核心组件之一。

## 49.21 利用率 Dashboard

Dashboard 应包括：

1. GPU allocation rate。
2. GPU utilization。
3. idle GPU hours。
4. queue wait time。
5. pending jobs。
6. fragmentation。
7. preemption count。
8. failed job GPU hours。
9. per-team usage。
10. per-priority usage。
11. tokens/GPU hour。

同时要能按 GPU 型号、集群、队列、团队和任务类型切分。

## 49.22 常见误区

误区一：GPU 利用率越高越好。

过高可能导致没有故障余量、p99 抖动和排队增加。

误区二：混部一定省钱。

如果混部影响在线 SLO 或导致训练抖动，收益可能小于风险。

误区三：抢占就是 kill 低优先级任务。

生产抢占需要 checkpoint、graceful shutdown、重新排队和事件通知。

误区四：弹性训练就是改副本数。

弹性训练涉及 world size、batch、学习率、optimizer state、dataloader 和 checkpoint。

误区五：资源预留是浪费。

合理预留是保证 SLO 和故障恢复能力的必要成本。

## 49.23 面试常见追问

问题一：如何提高 GPU 集群利用率？

可以回答：从减少碎片、拓扑感知调度、低优先级队列、抢占、配额借用、弹性训练、数据供给优化、checkpoint 策略、推理 batching 和自动扩缩容等方面综合优化。

问题二：训练和推理能混部吗？

可以回答：可以但要谨慎。要考虑 GPU、CPU、内存、网络和 I/O 干扰，在线推理要有 SLO guardrail，低优先级离线任务应可抢占，必要时物理或逻辑隔离。

问题三：抢占机制怎么设计？

可以回答：定义优先级和抢占关系，抢占前给任务保存 checkpoint 的时间，被抢占任务重新入队，限制抢占频率，记录事件并通知用户。

问题四：为什么不能追求 100% GPU 利用率？

可以回答：需要保留 headroom 应对突发流量、故障恢复和高优先级任务。过高利用率会增加排队、抢占和线上延迟风险。

## 49.25 本章小结

本章讲了资源利用率优化。

你需要记住：

1. GPU 利用率优化不是简单打满集群，而是提高有效产出。
2. 资源碎片、拓扑、gang scheduling 和多机多卡需求是 AI 集群调度的核心难点。
3. 混部、抢占、低优先级队列和配额借用可以提高利用率，但必须有隔离和 SLO guardrail。
4. 弹性训练能提升资源灵活性，但涉及 world size、batch、optimizer、dataloader 和 checkpoint 的复杂变化。
5. 利用率优化必须保留 headroom，不能牺牲可靠性和线上 SLO。
