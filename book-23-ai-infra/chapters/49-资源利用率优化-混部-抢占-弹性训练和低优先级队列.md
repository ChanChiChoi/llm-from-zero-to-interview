# 第 49 章 资源利用率优化：混部、抢占、弹性训练和低优先级队列

上一章讲了成本治理。本章继续讲资源利用率优化。

AI Infra 中最昂贵的资源通常是 GPU。GPU 买了或租了以后，如果长期空闲，就是直接浪费。但为了保证训练任务和线上推理的稳定性，又不能简单把所有资源塞满。因此资源利用率优化的核心，是在稳定性、隔离、公平性和成本之间做工程权衡。

先记住一句话：

> GPU 利用率优化不是把集群打满，而是在满足优先级、SLO 和隔离约束下，减少空闲、减少碎片、减少失败和提高有效产出。

## 49.0 本讲资料边界与第二轮精修口径

本章按 `WRITING_PLAN.md` 做第二轮精修时，主要参考 Kubernetes 对 Pod 资源请求 / 限制、调度、PriorityClass、preemption、ResourceQuota、bin packing 和 overprovisioning 的官方口径，参考 Kueue 对 queue、ClusterQueue、cohort borrowing、preemption 和 fair sharing 的调度治理模型，参考 PyTorch Elastic / torchrun 对弹性训练、rendezvous 和 worker 重启的说明，也参考 NVIDIA Kubernetes device plugin 对 GPU sharing、time-slicing 和 MPS 的边界说明。

需要强调三点：

1. 本章讨论的是 AI Infra 资源利用率治理的稳定工程问题，不绑定某个云厂商、调度器插件、Kubernetes 发行版、GPU 型号或内部平台实现。
2. 利用率指标不能脱离 SLO、隔离、公平性、checkpoint、恢复成本和业务价值单独优化。
3. 文中 demo 是教学级审计脚本，只用于解释资源治理门禁，不代表真实调度器、预测器或成本系统。

## 49.1 为什么资源利用率难

GPU 集群利用率难优化，原因包括：

1. 任务资源需求差异大。
2. 大模型训练需要 gang scheduling。
3. 多机多卡任务容易产生资源碎片。
4. 推理服务需要保留冗余和 warm pool。
5. 训练任务运行时间长。
6. 低优先级任务可能被抢占。
7. 不同 GPU 型号不可完全替代。
8. 数据、网络、存储也会限制 GPU 使用。

所以利用率优化不是单纯提高 GPU utilization，而是整体资源调度问题。

## 49.2 利用率指标

常见指标包括：

1. GPU allocation rate。
2. GPU utilization。
3. GPU memory utilization。
4. MFU。
5. tokens per GPU hour。
6. idle GPU hours。
7. pending jobs。
8. queue wait time。
9. preempted job count。
10. failed job GPU hours。
11. fragmentation rate。

GPU allocation rate 表示 GPU 是否被分配。

GPU utilization 表示 GPU 分配后是否真的忙。

两者都要看。

GPU 被分配但利用率很低，可能是数据、通信或配置问题。

可以先把一个时间窗口内的资源记录抽象为：

$$
R_i=(g_i,t_i,u_i,m_i,q_i,p_i,s_i,c_i,z_i)
$$

其中 `g_i` 是 GPU 数，`t_i` 是占用时长，`u_i` 是平均 GPU 利用率，`m_i` 是显存利用率，`q_i` 是队列，`p_i` 是优先级，`s_i` 是 SLO 通过状态，`c_i` 是 checkpoint / resume 状态，`z_i` 是标签和 owner 证据。

GPU 分配率：

$$
U_{\mathrm{alloc}}=\frac{G_{\mathrm{allocated}}}{G_{\mathrm{total}}}
$$

GPU 活跃利用率可以按 GPU-hours 加权：

$$
U_{\mathrm{active}}=\frac{\sum_i g_i t_i u_i}{\sum_i g_i t_i}
$$

空闲 GPU-hours：

$$
H_{\mathrm{idle}}=\sum_{n=1}^{N} g_{n,\mathrm{idle}} T
$$

其中 `T` 是统计窗口长度。面试中要特别说明：`U_alloc` 高只能说明资源被占用，`U_active` 高才说明占用期间较忙，但二者都不等于“产出有效”。

## 49.3 有效利用率

有效利用率比表面利用率更重要。

例如：

1. GPU 满载跑失败任务，不是有效利用。
2. GPU 满载跑错误配置实验，不是有效利用。
3. GPU 满载但训练数据重复严重，产出价值低。
4. 推理 GPU 满载但 p99 崩掉，也不是好状态。

有效利用率要结合产出：

1. 成功训练 step。
2. 有效 tokens。
3. 成功请求。
4. 符合 SLO 的吞吐。
5. 可发布模型产物。

可以用一个教学版有效利用率表达：

$$
U_{\mathrm{eff}}=\frac{\sum_i h_i v_i s_i}{\sum_i h_i}
$$

其中 `h_i=g_i t_i` 是任务 GPU-hours，`v_i` 是有效产出门禁，例如训练成功、评估可用、模型可发布或请求成功，`s_i` 是 SLO / 质量门禁。这个式子提醒你：失败任务、无效实验和破坏 SLO 的推理吞吐，都不能算成有效利用率。

## 49.4 资源碎片

资源碎片是 GPU 集群常见问题。

例如集群中有 8 台机器，每台 8 卡，但空闲 GPU 分散在不同机器上：

```text
node1: 2 free
node2: 1 free
node3: 3 free
node4: 2 free
```

总共有 8 张空闲 GPU，但没有一台机器有 8 张连续 GPU，无法启动一个单机 8 卡任务。

多机多卡任务还需要网络拓扑匹配，碎片问题更严重。

对一个需要每台节点至少 `g` 张同型 GPU 的任务，可以定义碎片比例：

$$
R_{\mathrm{frag}}(g)=1-\frac{\sum_{n=1}^{N}\left\lfloor f_n/g\right\rfloor g}{\sum_{n=1}^{N} f_n}
$$

其中 `f_n` 是第 `n` 个节点上的空闲 GPU 数。上面的例子里 `g=8` 时，没有任何节点能放下单机 8 卡任务，所以可用于该任务的连续空闲 GPU 为 0，`R_frag(8)=1`。

## 49.5 减少碎片的方法

常见方法：

1. 拓扑感知调度。
2. bin packing。
3. 大任务优先或预留。
4. 队列重排。
5. 低优先级任务可抢占。
6. 任务资源规格标准化。
7. 避免小任务随意占用整机关键位置。
8. 定期 defragmentation。

调度器要知道 GPU 拓扑，而不是只看剩余卡数。

一个简单的 bin packing 打分可以写成：

$$
S_{\mathrm{pack}}(n)=\sum_r w_r u_{n,r}
$$

其中 `u_{n,r}` 是节点 `n` 上资源 `r` 的已使用比例，`w_r` 是权重。Most-allocated 类策略会倾向把新任务放到已经较满但仍能满足约束的节点，从而保留更完整的空闲节点给大任务。AI Infra 里还要叠加拓扑和干扰风险：

$$
S_{\mathrm{node}}=\alpha S_{\mathrm{pack}}+\beta S_{\mathrm{topo}}-\gamma R_{\mathrm{interfere}}
$$

其中 `S_topo` 衡量 GPU、NIC、机柜和存储局部性，`R_interfere` 衡量混部干扰风险。

## 49.6 混部是什么

混部是把不同类型任务放在同一资源池或同一节点上运行，以提高资源利用率。

常见混部：

1. 训练和推理混部。
2. 在线和离线混部。
3. 高优先级和低优先级混部。
4. GPU 任务和 CPU 任务混部。
5. embedding 任务和训练任务混部。

混部目标是利用资源空隙。

但混部会带来干扰。

混部收益可以先用闲置资源填充率表示：

$$
F_{\mathrm{fill}}=\frac{H_{\mathrm{reclaimed}}}{H_{\mathrm{idle,before}}}
$$

其中 `H_reclaimed` 是被低优先级或可中断任务填充的 GPU-hours，`H_idle,before` 是混部前预计空闲 GPU-hours。这个指标只说明填洞能力，不说明混部是否安全。

## 49.7 混部风险

混部风险包括：

1. GPU 显存竞争。
2. CPU 竞争。
3. 内存竞争。
4. 网络竞争。
5. 磁盘 I/O 竞争。
6. 推理 p99 延迟抖动。
7. 训练 step time 抖动。
8. 故障影响扩大。

在线推理任务和离线训练混部尤其要谨慎。

如果离线任务导致在线推理 p99 上升，节省的成本可能不值得。

混部干扰可以按核心 SLO 指标估计：

$$
D_{\mathrm{lat}}=\frac{L_{\mathrm{p99,mix}}-L_{\mathrm{p99,base}}}{L_{\mathrm{p99,base}}}
$$

$$
D_{\mathrm{step}}=\frac{T_{\mathrm{step,mix}}-T_{\mathrm{step,base}}}{T_{\mathrm{step,base}}}
$$

其中 `D_lat` 是在线推理尾延迟放大，`D_step` 是训练 step time 放大。只要混部节省没有覆盖 SLO 风险、故障风险和排障成本，就不应该扩大使用。

## 49.8 安全混部策略

更安全的混部方式：

1. 按优先级隔离。
2. 按 GPU 型号隔离。
3. 在线服务保留资源。
4. 离线任务只使用可抢占资源。
5. 限制低优先级任务资源上限。
6. 使用 node taint/toleration。
7. 使用 QoS class。
8. 对在线服务设置 latency guardrail。
9. 异常时自动驱逐低优先级任务。

混部必须有监控和自动保护，不应靠人工观察。

安全混部门禁可以写成：

$$
G_{\mathrm{mix}}=\mathbf{1}\left[D_{\mathrm{lat}}\le \epsilon_{\mathrm{lat}} \land D_{\mathrm{step}}\le \epsilon_{\mathrm{step}} \land R_{\mathrm{evict}}\le \rho \land P_0=0\right]
$$

其中 `R_evict` 是驱逐或抢占导致的异常比例，`P0` 是高严重度风险数量。这个门禁强调：混部不是资源利用率技巧，而是带自动保护的调度策略。

## 49.9 抢占是什么

抢占是高优先级任务需要资源时，平台停止或迁移低优先级任务，释放资源。

抢占适合：

1. 低优先级训练。
2. 探索性实验。
3. 离线 embedding。
4. 批量评估。
5. 可恢复任务。

不适合：

1. 关键线上推理。
2. 无 checkpoint 的长任务。
3. 强实时任务。
4. 不可重复执行任务。

抢占的前提是任务能恢复。

抢占损失可以用 GPU-hours 估计：

$$
L_j=g_j(t_{\mathrm{now}}-t_{\mathrm{ckpt},j})
$$

其中 `g_j` 是被抢占任务的 GPU 数，`t_now` 是抢占时间，`t_ckpt,j` 是最近可恢复 checkpoint 时间。没有 checkpoint 或 checkpoint 太旧时，抢占的真实成本会非常高。

## 49.10 抢占设计

抢占系统要考虑：

1. 谁可以抢占谁。
2. 抢占前是否保存 checkpoint。
3. 给任务多少 graceful shutdown 时间。
4. 被抢占任务如何重新排队。
5. 抢占次数是否有限制。
6. 是否补偿低优先级队列。
7. 抢占事件如何通知用户。

抢占不能粗暴 kill。

否则会浪费大量已完成训练进度。

一次抢占是否值得，可以用收益减损失判断：

$$
B_{\mathrm{preempt}}=V_{\mathrm{high}}-L_{\mathrm{low}}-K_{\mathrm{resume}}-K_{\mathrm{noise}}
$$

其中 `V_high` 是高优先级任务按时启动的价值，`L_low` 是低优先级任务丢失 GPU-hours，`K_resume` 是恢复成本，`K_noise` 是用户通知、队列抖动和排障成本。生产系统通常只允许 `B_preempt>0` 且恢复证据完整的抢占。

## 49.11 低优先级队列

低优先级队列用于吸收闲置资源。

适合放：

1. 探索实验。
2. 批量评估。
3. embedding 生成。
4. 数据处理。
5. 非紧急微调。
6. 可中断任务。

低优先级队列的特点：

1. 等待时间不保证。
2. 可被抢占。
3. 成本较低。
4. 适合填充资源空洞。

这能提升集群整体利用率。

低优先级队列可用容量可以近似为：

$$
C_{\mathrm{low}}=\max(0,C_{\mathrm{total}}-C_{\mathrm{reserved}}-C_{\mathrm{high,pending}})
$$

其中 `C_total` 是总容量，`C_reserved` 是 SLO、故障恢复和关键任务预留，`C_high,pending` 是高优先级待满足需求。低优先级任务只能吃这个“可回收余量”，不能挤占硬预留。

## 49.12 弹性训练是什么

弹性训练是训练任务可以在不同资源规模下运行。

例如一个任务可以从 64 卡扩到 128 卡，也可以从 128 卡缩到 64 卡。

弹性训练的价值：

1. 提高资源利用率。
2. 减少排队等待。
3. 支持抢占后继续运行。
4. 利用临时空闲 GPU。
5. 高峰期释放资源给更重要任务。

但弹性训练对框架和训练逻辑要求高。

弹性训练的基本批量关系是：

$$
B_{\mathrm{global}}=b_{\mathrm{micro}} a_{\mathrm{grad}} w
$$

其中 `b_micro` 是单 worker micro batch，`a_grad` 是梯度累积步数，`w` 是 worker 数。如果希望 `B_global` 不变，worker 数变化后通常要调整 `a_grad`：

$$
a_{\mathrm{grad}}=\frac{B_{\mathrm{target}}}{b_{\mathrm{micro}}w}
$$

## 49.13 弹性训练的难点

难点包括：

1. world size 变化。
2. global batch size 变化。
3. learning rate 调整。
4. optimizer state 重分片。
5. dataloader 分片变化。
6. checkpoint 格式兼容。
7. 通信拓扑变化。
8. 收敛稳定性影响。

弹性不是简单增加或减少 Pod。

如果 global batch 变化没有处理好，训练动态会改变。

弹性事件的风险可以用批量漂移表示：

$$
\Delta B=\left|\frac{B_{\mathrm{global,new}}-B_{\mathrm{global,old}}}{B_{\mathrm{global,old}}}\right|
$$

如果 `Delta B` 超过阈值，就需要重新检查学习率、warmup、loss scale、数据分片和评估可比性。

## 49.14 弹性训练策略

常见策略：

1. 固定 global batch，改变 gradient accumulation。
2. 固定 learning rate schedule。
3. 支持 elastic dataloader。
4. checkpoint 支持不同 world size restore。
5. 只在安全边界内扩缩容。
6. 对弹性事件记录 trace。

弹性训练适合成熟平台，不适合一开始就强行支持所有任务。

弹性训练门禁可以写成：

$$
G_{\mathrm{elastic}}=\mathbf{1}\left[w_{\min}\le w\le w_{\max} \land \Delta B\le \epsilon_B \land C_{\mathrm{ckpt}}=1 \land C_{\mathrm{data}}=1\right]
$$

其中 `C_ckpt` 表示 checkpoint 能跨 world size 恢复，`C_data` 表示 dataloader 分片、shuffle 和去重不会因为 worker 数变化而破坏训练语义。

## 49.15 推理资源利用率优化

推理资源优化方向：

1. continuous batching。
2. KV cache 管理。
3. 模型量化。
4. 多模型路由。
5. 小模型优先。
6. 自动扩缩容。
7. warm pool 优化。
8. cache 提升命中。
9. 低峰缩容。
10. 分离 prefill/decode。

推理资源利用率不能只追求 GPU 打满。

在线服务还要满足 TTFT、TPOT 和 p99。

推理有效吞吐应扣掉失败请求和 SLO 失败请求：

$$
X_{\mathrm{good}}=\frac{N_{\mathrm{out,ok}}}{T} C_{\mathrm{slo}}
$$

其中 `N_out,ok` 是成功输出 token 数，`C_slo` 是请求级 SLO 通过率。对于在线推理，`X_good` 比单纯 tokens/s 更适合和资源利用率一起看。

## 49.16 Warm Pool 优化

Warm pool 可以降低冷启动，但会增加空闲成本。

优化方式：

1. 按模型热度设置最小副本。
2. 按时间预测扩容。
3. 热模型常驻。
4. 冷模型按需加载。
5. 权重本地缓存。
6. 多模型共享基础镜像。
7. 对低优先级模型允许冷启动。

不同业务需要不同 warm pool 策略。

高价值在线业务需要更高可用性，低频内部模型可以接受冷启动。

Warm pool 的取舍可以写成两类成本比较：

$$
K_{\mathrm{warm}}=r_{\min} c_{\mathrm{gpu}} T
$$

$$
K_{\mathrm{cold}}=\lambda_{\mathrm{cold}} p_{\mathrm{miss}} V_{\mathrm{lat}}
$$

其中 `r_min` 是常驻副本数，`c_gpu` 是单位时间成本，`lambda_cold` 是冷启动请求量，`p_miss` 是冷启动概率，`V_lat` 是一次冷启动延迟对业务或 SLO 的损失估计。当 `K_cold` 大于 `K_warm` 时，warm pool 更合理。

## 49.17 资源预留

资源预留用于保障关键任务。

常见预留：

1. 线上推理保留 GPU。
2. 关键训练项目保留配额。
3. 高优先级队列保留容量。
4. 故障恢复保留冗余。
5. 峰值流量前预留资源。

预留会降低表面利用率，但提高稳定性。

因此要平衡：预留太少影响 SLO，预留太多造成浪费。

资源预留可以按多个约束取最大值：

$$
C_{\mathrm{reserve}}=\max(C_{\mathrm{slo}},C_{\mathrm{recovery}},C_{\mathrm{priority}},C_{\mathrm{burst}})
$$

其中 `C_slo` 是在线 SLO 所需容量，`C_recovery` 是故障恢复容量，`C_priority` 是关键训练或客户预留，`C_burst` 是短期峰值容量。Headroom 可以写成：

$$
H_{\mathrm{headroom}}=1-\frac{C_{\mathrm{used}}}{C_{\mathrm{total}}}
$$

## 49.18 配额和公平性

资源利用率优化不能牺牲公平性。

配额维度包括：

1. 团队。
2. 租户。
3. 项目。
4. 队列。
5. 优先级。
6. GPU 类型。

公平性策略：

1. dominant resource fairness。
2. weighted fair sharing。
3. quota borrowing。
4. priority-based scheduling。
5. preemption compensation。

允许借用空闲配额，可以提高利用率。

但借用资源要能在原 owner 需要时归还。

Dominant resource share 可用于多资源公平性：

$$
D_u=\max_r \frac{A_{u,r}}{C_r}
$$

其中 `A_u,r` 是用户或租户 `u` 占用的资源 `r`，`C_r` 是集群总资源。借用量可以写成：

$$
B_u=\max(0,A_u-Q_u)
$$

其中 `Q_u` 是租户配额。允许 `B_u>0` 可以提高利用率，但必须配合可回收、可抢占和补偿策略。

## 49.19 利用率优化和可靠性

利用率越高，系统余量越少。

风险包括：

1. 故障恢复空间不足。
2. 突发流量承接困难。
3. 训练任务排队增加。
4. 推理 p99 抖动。
5. 抢占频繁。

所以不能追求 100% 利用率。

平台要根据 SLO 保留合理 headroom。

Headroom 是为了应对波动和故障的安全余量。

可靠性风险可以用尾延迟放大和抢占放大一起观察：

$$
A_{\mathrm{tail}}=\frac{L_{\mathrm{p99,current}}}{L_{\mathrm{p99,base}}}
$$

$$
A_{\mathrm{preempt}}=\frac{N_{\mathrm{preempt,current}}}{N_{\mathrm{preempt,base}}+1}
$$

当利用率上升伴随 `A_tail` 和 `A_preempt` 同时上升时，说明系统已经从“填洞”进入“挤压可靠性”的区域。

## 49.20 调度器能力

高利用率需要强调度器能力：

1. gang scheduling。
2. 拓扑感知。
3. 队列和优先级。
4. 抢占。
5. 配额借用。
6. 资源碎片感知。
7. 多资源调度。
8. 弹性任务支持。
9. 调度模拟。
10. 调度事件可观测。

调度器是 AI Infra 利用率优化的核心组件之一。

调度可执行性可以用覆盖率表达：

$$
C_{\mathrm{sched}}=\frac{1}{M}\sum_{j=1}^{M}\mathbf{1}[G_j=1]
$$

其中 `G_j` 是 gang、拓扑、抢占、配额、公平性、弹性、可观测性等调度能力门禁。只有 dashboard 指标而没有调度执行能力，利用率优化会停留在报表层。

## 49.21 利用率 Dashboard

Dashboard 应包括：

1. GPU allocation rate。
2. GPU utilization。
3. idle GPU hours。
4. queue wait time。
5. pending jobs。
6. fragmentation。
7. preemption count。
8. failed job GPU hours。
9. per-team usage。
10. per-priority usage。
11. tokens/GPU hour。

同时要能按 GPU 型号、集群、队列、团队和任务类型切分。

Dashboard 下钻覆盖率：

$$
C_{\mathrm{dash}}=\frac{N_{\mathrm{drill,ready}}}{N_{\mathrm{drill,total}}}
$$

其中下钻维度至少包括 cluster、GPU type、node、queue、tenant、team、priority、job type、model、endpoint 和 time window。没有下钻能力时，很难判断空闲、碎片、抢占和 SLO 抖动是谁造成的。

## 49.22 资源利用率治理审计指标和最小 demo

把本章落到平台验收时，可以用 16 个门禁：

1. Resource Metric Contract：GPU allocation、GPU utilization、memory、MFU、tokens、idle、pending、queue wait、preemption、failed GPU-hours 和 fragmentation 是否有统一口径。
2. Effective Utilization Accounting：是否把失败任务、无效实验、SLO 失败请求和不可发布产物从有效利用率中扣除。
3. Fragmentation Control：是否能按节点、GPU 型号、拓扑和任务规格计算碎片并执行 defragmentation。
4. Topology Binpacking Readiness：bin packing 是否同时考虑 GPU、NIC、rack、storage、gang 和 score trace。
5. Colocation SLO Guard：混部是否有 latency、step time、驱逐、低优先级上限和关键任务隔离保护。
6. Preemption Checkpoint Safety：抢占是否绑定 checkpoint 新鲜度、graceful shutdown、requeue、频率限制和用户通知。
7. Low Priority Reclaim Policy：低优先级队列是否只使用可回收余量，并能在高优先级需求出现时归还。
8. Elastic Training Safety：弹性训练是否控制 global batch 漂移、checkpoint world size、dataloader、rendezvous 和事件 trace。
9. Inference Capacity Utilization：推理利用率是否同时满足 TTFT、TPOT、p99、KV pressure、continuous batching 和 autoscaling。
10. Warm Pool Headroom Policy：warm pool 是否基于模型热度、冷启动损失、成本、headroom 策略和容量上限决策。
11. Quota Fairness Borrowing：配额借用是否受 dominant share、公平性、可回收和补偿机制约束。
12. Scheduler Observability：调度器是否暴露 gang、queue、priority、preemption、quota、topology、elastic 和 event trace。
13. Dashboard Drilldown Readiness：dashboard 是否能按 cluster、GPU type、node、queue、tenant、team、priority、job type、model、endpoint 和时间窗口下钻。
14. Reliability Headroom Guard：高利用率是否仍保留故障恢复、突发流量、高优任务和线上 SLO 余量。
15. Cost SLO Tradeoff Control：优化建议是否同时检查成本节省、质量、SLO、风险、owner 和回滚条件。
16. Resource Utilization Gate：最终是否有 owner、资源策略、例外审批、审计证据和 P0 风险阻断。

综合门禁：

$$
G_{\mathrm{resource\_utilization}}=\prod_{j=1}^{16}G_j
$$

下面是一个 0 依赖 demo，用 toy cases 检查资源利用率优化是否只是“把 GPU 打满”，还是能把指标、碎片、混部、抢占、低优先级队列、弹性训练、推理容量、warm pool、配额公平和可靠性余量连成闭环。

class MiniResourceUtilizationAudit:
    GATES = [
        "resource_metric_contract",
        "effective_utilization_accounting",
        "fragmentation_control",
        "topology_binpacking_readiness",
        "colocation_slo_guard",
        "preemption_checkpoint_safety",
        "low_priority_reclaim_policy",
        "elastic_training_safety",
        "inference_capacity_utilization",
        "warm_pool_headroom_policy",
        "quota_fairness_borrowing",
        "scheduler_observability",
        "dashboard_drilldown_readiness",
        "reliability_headroom_guard",
        "cost_slo_tradeoff_control",
        "resource_utilization_gate",
    ]

    METRIC_FIELDS = [
        "allocation",
        "utilization",
        "memory",
        "mfu",
        "tokens",
        "idle",
        "pending",
        "queue_wait",
        "preemption",
        "failed_gpu_hours",
        "fragmentation",
    ]
    DASHBOARD_FIELDS = [
        "cluster",
        "gpu_type",
        "node",
        "queue",
        "tenant",
        "team",
        "priority",
        "job_type",
        "model",
        "endpoint",
        "time_window",
    ]
    SCHEDULER_FIELDS = [
        "gang",
        "queue",
        "priority",
        "preemption",
        "quota",
        "topology",
        "elastic",
        "event_trace",
    ]

    @staticmethod
    def present(record, key):
        return key in record and record[key] is not None and record[key] != ""

    def coverage(self, record, fields):
        if not record:
            return 0.0
        return sum(1 for field in fields if self.present(record, field)) / len(fields)

    def allocation_rate(self, case):
        capacity = case["capacity"]
        return capacity["allocated_gpu"] / max(1e-12, capacity["total_gpu"])

    def active_utilization(self, case):
        records = case["utilization_records"]
        used = sum(item["gpu_hours"] * item["gpu_util"] for item in records)
        total = sum(item["gpu_hours"] for item in records)
        return used / max(1e-12, total)

    def effective_utilization(self, case):
        records = case["utilization_records"]
        useful = sum(
            item["gpu_hours"] * float(item["value_gate"]) * float(item["slo_gate"])
            for item in records
        )
        total = sum(item["gpu_hours"] for item in records)
        return useful / max(1e-12, total)

    def fragmentation_ratio(self, case):
        frag = case["fragmentation"]
        target = frag["target_per_node"]
        free_by_node = frag["free_by_node"]
        total_free = sum(free_by_node)
        packable = sum((free // target) * target for free in free_by_node)
        if total_free == 0:
            return 0.0
        return 1.0 - packable / total_free

    def colocation_latency_delta(self, case):
        coloc = case["colocation"]
        return (coloc["p99_ms_mix"] - coloc["p99_ms_base"]) / max(1e-12, coloc["p99_ms_base"])

    def colocation_step_delta(self, case):
        coloc = case["colocation"]
        return (coloc["step_ms_mix"] - coloc["step_ms_base"]) / max(1e-12, coloc["step_ms_base"])

    def preemption_loss_gpu_min(self, case):
        preempt = case["preemption"]
        return preempt["gpu_count"] * (preempt["now_min"] - preempt["last_checkpoint_min"])

    def global_batch(self, elastic):
        return elastic["micro_batch"] * elastic["grad_accum"] * elastic["workers"]

    def adjusted_grad_accum(self, case):
        elastic = case["elastic"]
        return elastic["target_global_batch"] / max(1e-12, elastic["micro_batch"] * elastic["new_workers"])

    def batch_delta(self, case):
        elastic = case["elastic"]
        old_batch = self.global_batch(elastic)
        new_batch = elastic["micro_batch"] * elastic["new_grad_accum"] * elastic["new_workers"]
        return abs(new_batch - old_batch) / max(1e-12, old_batch)

    def good_output_tokens_s(self, case):
        inference = case["inference"]
        return inference["successful_output_tokens"] / max(1e-12, inference["window_seconds"]) * inference["slo_pass_rate"]

    def warm_pool_cost(self, case):
        warm = case["warm_pool"]
        return warm["min_replicas"] * warm["gpu_cost_per_hour"] * warm["hours"]

    def cold_start_penalty(self, case):
        warm = case["warm_pool"]
        return warm["cold_requests"] * warm["cold_miss_prob"] * warm["latency_penalty_value"]

    def headroom(self, case):
        reliability = case["reliability"]
        return 1.0 - reliability["used_capacity"] / max(1e-12, reliability["total_capacity"])

    def dominant_share(self, case):
        fairness = case["fairness"]
        shares = []
        for resource, used in fairness["tenant_usage"].items():
            shares.append(used / max(1e-12, fairness["cluster_capacity"][resource]))
        return max(shares)

    def borrowed_gpu(self, case):
        fairness = case["fairness"]
        return max(0, fairness["tenant_usage"]["gpu"] - fairness["tenant_quota"]["gpu"])

    def resource_metric_contract(self, case):
        metrics = case.get("metrics", {})
        capacity = case.get("capacity", {})
        return (
            self.coverage(metrics, self.METRIC_FIELDS) == 1.0
            and metrics.get("freshness_minutes", 10**9) <= metrics.get("max_freshness_minutes", 0)
            and capacity.get("total_gpu", 0) > 0
            and 0.0 <= self.allocation_rate(case) <= 1.0
        )

    def effective_utilization_accounting(self, case):
        eff = case.get("effective", {})
        return (
            self.effective_utilization(case) >= eff.get("min_effective_utilization", 1.0)
            and eff.get("failed_gpu_hours", 10**9) <= eff.get("max_failed_gpu_hours", 0)
            and eff.get("invalid_experiments", 10**9) <= eff.get("max_invalid_experiments", 0)
            and eff.get("publishable_output_tracked") is True
        )

    def fragmentation_control(self, case):
        frag = case.get("fragmentation", {})
        return (
            self.fragmentation_ratio(case) <= frag.get("max_fragmentation_ratio", 0.0)
            and frag.get("defragmentation_plan") is True
            and frag.get("standard_shapes") is True
            and frag.get("gpu_type_aware") is True
        )

    def topology_binpacking_readiness(self, case):
        topo = case.get("topology", {})
        return (
            topo.get("binpacking_enabled") is True
            and topo.get("gpu_type") is True
            and topo.get("nic_locality") is True
            and topo.get("rack_locality") is True
            and topo.get("storage_locality") is True
            and topo.get("gang_aware") is True
            and topo.get("score_trace") is True
        )

    def colocation_slo_guard(self, case):
        coloc = case.get("colocation", {})
        return (
            self.colocation_latency_delta(case) <= coloc.get("max_latency_delta", 0.0)
            and self.colocation_step_delta(case) <= coloc.get("max_step_delta", 0.0)
            and coloc.get("low_priority_limit") is True
            and coloc.get("auto_evict") is True
            and coloc.get("critical_isolation") is True
        )

    def preemption_checkpoint_safety(self, case):
        preempt = case.get("preemption", {})
        return (
            preempt.get("enabled") is True
            and preempt.get("checkpoint_ready") is True
            and preempt.get("graceful_shutdown_seconds", 0) >= preempt.get("min_graceful_shutdown_seconds", 10**9)
            and preempt.get("requeue_ready") is True
            and preempt.get("notify_owner") is True
            and preempt.get("max_preemptions_per_day", 10**9) <= preempt.get("limit_preemptions_per_day", 0)
            and self.preemption_loss_gpu_min(case) <= preempt.get("max_loss_gpu_min", 0)
        )

    def low_priority_reclaim_policy(self, case):
        low = case.get("low_priority", {})
        capacity = case.get("capacity", {})
        reclaimable = max(0, capacity["total_gpu"] - capacity["reserved_gpu"] - capacity["high_pending_gpu"])
        return (
            low.get("enabled") is True
            and low.get("preemptible") is True
            and low.get("uses_reclaimable_only") is True
            and low.get("owner_notice") is True
            and low.get("compensation_policy") is True
            and low.get("current_gpu") <= reclaimable
            and low.get("current_gpu") <= low.get("max_gpu_share") * capacity["total_gpu"]
        )

    def elastic_training_safety(self, case):
        elastic = case.get("elastic", {})
        adjusted = self.adjusted_grad_accum(case)
        return (
            elastic.get("min_workers", 0) <= elastic.get("new_workers", -1) <= elastic.get("max_workers", 0)
            and self.batch_delta(case) <= elastic.get("max_batch_delta", 0.0)
            and abs(round(adjusted) - adjusted) <= 1e-9
            and elastic.get("checkpoint_world_size_safe") is True
            and elastic.get("elastic_dataloader") is True
            and elastic.get("rendezvous_ready") is True
            and elastic.get("event_trace") is True
        )

    def inference_capacity_utilization(self, case):
        inference = case.get("inference", {})
        return (
            self.good_output_tokens_s(case) >= inference.get("min_good_output_tokens_s", 10**9)
            and inference.get("ttft_p95_ms", 10**9) <= inference.get("ttft_slo_ms", 0)
            and inference.get("tpot_p95_ms", 10**9) <= inference.get("tpot_slo_ms", 0)
            and inference.get("p99_ms", 10**9) <= inference.get("p99_slo_ms", 0)
            and inference.get("kv_pressure", 1.0) <= inference.get("max_kv_pressure", 0.0)
            and inference.get("continuous_batching") is True
            and inference.get("autoscaling_ready") is True
        )

    def warm_pool_headroom_policy(self, case):
        warm = case.get("warm_pool", {})
        return (
            warm.get("policy_ready") is True
            and warm.get("heat_based") is True
            and warm.get("weight_cache_ready") is True
            and warm.get("cold_start_allowed_for_low_priority") is True
            and warm.get("max_replicas") >= warm.get("min_replicas")
            and self.warm_pool_cost(case) <= warm.get("max_warm_pool_cost", 0.0)
            and self.cold_start_penalty(case) >= self.warm_pool_cost(case)
        )

    def quota_fairness_borrowing(self, case):
        fairness = case.get("fairness", {})
        return (
            fairness.get("drf_ready") is True
            and fairness.get("weighted_share_ready") is True
            and fairness.get("borrow_returnable") is True
            and fairness.get("preemption_compensation") is True
            and self.dominant_share(case) <= fairness.get("max_dominant_share", 0.0)
            and self.borrowed_gpu(case) <= fairness.get("max_borrowed_gpu", 0)
        )

    def scheduler_observability(self, case):
        scheduler = case.get("scheduler", {})
        return self.coverage(scheduler, self.SCHEDULER_FIELDS) == 1.0 and scheduler.get("fresh") is True

    def dashboard_drilldown_readiness(self, case):
        dashboard = case.get("dashboard", {})
        return self.coverage(dashboard.get("drilldown", {}), self.DASHBOARD_FIELDS) == 1.0 and dashboard.get("fresh") is True

    def reliability_headroom_guard(self, case):
        reliability = case.get("reliability", {})
        return (

            and reliability.get("tail_amplification", 10**9) <= reliability.get("max_tail_amplification", 0.0)
            and reliability.get("preemption_amplification", 10**9) <= reliability.get("max_preemption_amplification", 0.0)
            and reliability.get("recovery_capacity") is True
            and reliability.get("p0_open") is False
        )

    def cost_slo_tradeoff_control(self, case):
        tradeoff = case.get("tradeoff", {})
        return (
            tradeoff.get("estimated_savings", 0.0) >= tradeoff.get("min_estimated_savings", 10**9)
            and tradeoff.get("quality_delta", -1.0) >= -tradeoff.get("max_quality_drop", 0.0)
            and tradeoff.get("slo_delta", -1.0) >= -tradeoff.get("max_slo_drop", 0.0)
            and tradeoff.get("risk_score", 1.0) <= tradeoff.get("max_risk_score", 0.0)
            and tradeoff.get("owner") is True
            and tradeoff.get("rollback_condition") is True
        )

    def resource_utilization_gate(self, case):
        gate = case.get("platform_gate", {})
        return (
            gate.get("enabled") is True
            and bool(gate.get("owner"))
            and gate.get("resource_policy") is True
            and gate.get("exception_approval") is True
            and gate.get("audit_ready") is True
            and gate.get("p0_open") is False
        )

    def audit_case(self, case):
        return {gate: getattr(self, gate)(case) for gate in self.GATES}

    def run_all(self, cases):
        results = {case["case_id"]: self.audit_case(case) for case in cases}
        metrics = {}
        for gate in self.GATES:
            passed = sum(1 for result in results.values() if result[gate])
            metrics[gate] = round(passed / len(cases), 3)
        failed_cases = [
            case_id
            for case_id, result in results.items()
            if not all(result.values())
        ]
        failed_gates = [
            gate
            for gate in self.GATES
            if any(not result[gate] for result in results.values())
        ]
        return {
            "metrics": metrics,
            "hard_blocker_count": len(failed_cases),
            "failed_cases": failed_cases,
            "failed_gates": failed_gates,
            "resource_utilization_gate_pass": metrics["resource_utilization_gate"] == 1.0,
        }

    def example_outputs(self, case):
        return {
            "allocation_rate": round(self.allocation_rate(case), 3),
            "active_utilization": round(self.active_utilization(case), 3),
            "effective_utilization": round(self.effective_utilization(case), 3),
            "fragmentation_ratio": round(self.fragmentation_ratio(case), 3),
            "colocation_latency_delta": round(self.colocation_latency_delta(case), 3),
            "colocation_step_delta": round(self.colocation_step_delta(case), 3),
            "preemption_loss_gpu_min": round(self.preemption_loss_gpu_min(case), 3),
            "global_batch": self.global_batch(case["elastic"]),
            "adjusted_grad_accum": round(self.adjusted_grad_accum(case), 3),
            "good_output_tokens_s": round(self.good_output_tokens_s(case), 3),
            "warm_pool_cost": round(self.warm_pool_cost(case), 3),
            "cold_start_penalty": round(self.cold_start_penalty(case), 3),
            "headroom": round(self.headroom(case), 3),
            "dominant_share": round(self.dominant_share(case), 3),
            "borrowed_gpu": self.borrowed_gpu(case),
        }

## 49.23 常见误区

误区一：GPU 利用率越高越好。

过高可能导致没有故障余量、p99 抖动和排队增加。

误区二：混部一定省钱。

如果混部影响在线 SLO 或导致训练抖动，收益可能小于风险。

误区三：抢占就是 kill 低优先级任务。

生产抢占需要 checkpoint、graceful shutdown、重新排队和事件通知。

误区四：弹性训练就是改副本数。

弹性训练涉及 world size、batch、学习率、optimizer state、dataloader 和 checkpoint。

误区五：资源预留是浪费。

合理预留是保证 SLO 和故障恢复能力的必要成本。

## 49.24 面试常见追问

问题一：如何提高 GPU 集群利用率？

可以回答：从减少碎片、拓扑感知调度、低优先级队列、抢占、配额借用、弹性训练、数据供给优化、checkpoint 策略、推理 batching 和自动扩缩容等方面综合优化。

问题二：训练和推理能混部吗？

可以回答：可以但要谨慎。要考虑 GPU、CPU、内存、网络和 I/O 干扰，在线推理要有 SLO guardrail，低优先级离线任务应可抢占，必要时物理或逻辑隔离。

问题三：抢占机制怎么设计？

可以回答：定义优先级和抢占关系，抢占前给任务保存 checkpoint 的时间，被抢占任务重新入队，限制抢占频率，记录事件并通知用户。

问题四：为什么不能追求 100% GPU 利用率？

可以回答：需要保留 headroom 应对突发流量、故障恢复和高优先级任务。过高利用率会增加排队、抢占和线上延迟风险。

## 49.26 本章小结

本章讲了资源利用率优化。

你需要记住：

1. GPU 利用率优化不是简单打满集群，而是提高有效产出。
2. 资源碎片、拓扑、gang scheduling 和多机多卡需求是 AI 集群调度的核心难点。
3. 混部、抢占、低优先级队列和配额借用可以提高利用率，但必须有隔离和 SLO guardrail。
4. 弹性训练能提升资源灵活性，但涉及 world size、batch、optimizer、dataloader 和 checkpoint 的复杂变化。
5. 利用率优化必须保留 headroom，不能牺牲可靠性和线上 SLO。
