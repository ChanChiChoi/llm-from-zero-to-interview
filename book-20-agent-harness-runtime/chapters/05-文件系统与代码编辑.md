## 5.1 本章目标

文件系统和代码编辑是 coding agent 最核心、也最容易出事故的能力。读错文件、改错位置、覆盖用户改动、重写无关内容、格式化整个文件、删除错误路径，都可能让用户失去对 agent 的信任。

本章专门讨论文件系统和编辑层。上一章讲了通用工具系统和 Tool Registry，本章聚焦 Read、List、Search、Edit、Apply Patch、Diff 这些和代码库直接交互的工具如何设计。

学完本章，你应该能回答：

1. Coding agent 需要哪些文件系统能力。
2. Read、Search、Edit、Apply Patch、Diff 的边界是什么。
3. 为什么优先小 patch，而不是 whole-file rewrite。
4. 如何避免覆盖用户并发修改。
5. 如何处理大文件、二进制文件、生成文件和隐藏文件。
6. 如何记录 diff、支持回滚和审计。
7. 面试中如何设计一个安全的代码编辑系统。

## 5.2 文件系统能力总览

Coding agent 通常需要以下文件系统能力：

1. 列目录。
2. 搜索文件名。
3. 搜索文件内容。
4. 读取文件片段。
5. 创建文件。
6. 精确编辑文件。
7. 删除文件。
8. 移动或重命名文件。
9. 比较 diff。
10. 检查文件状态。
11. 检测用户并发修改。
12. 回滚修改。

这些能力要分级。

只读能力：

1. list directory。
2. search。
3. read file。
4. git diff。

写入能力：

1. edit file。
2. apply patch。
3. create file。
4. delete file。
5. move file。

高风险写入：

1. 删除文件。
2. 大范围重命名。
3. 格式化整个项目。
4. 修改配置、锁文件、迁移文件。
5. 修改用户未授权路径。

文件系统工具的基本原则：默认只读，写入需要更强约束和 trace。

## 5.3 文件系统边界：Workspace Root

Coding agent 必须有明确 workspace root。

Workspace root 决定：

1. 哪些文件可以读。
2. 哪些文件可以写。
3. 哪些路径应该被拒绝。
4. trace 中如何展示相对路径。
5. sandbox 如何限制访问。

风险路径包括：

1. `../` 路径穿越。
2. 绝对路径。
3. 用户 home 目录。
4. 系统目录。
5. `.env`、密钥、凭证文件。
6. SSH key。
7. 浏览器、云服务或包管理器凭证。

路径校验原则：

1. 所有路径规范化。
2. 解析 symlink 后仍必须在 workspace 内。
3. 默认拒绝读取 secret pattern。
4. 写操作必须限制在允许目录内。
5. Trace 使用相对路径，避免泄露本机路径。

面试表达：

```text
我会先定义 workspace root，所有文件工具都必须基于规范化后的相对路径工作。路径穿越、workspace 外路径、secret 文件和系统目录默认拒绝。读操作和写操作分开授权，写操作还要记录 diff 和 trace。
```

## 5.4 List 和 Search：先定位，再读取

Agent 不应该一上来读取大量文件。更好的流程是先定位，再读取。

List 工具用于了解目录结构：

1. 项目有哪些顶层目录。
2. 源码、测试、配置在哪里。
3. 是否有 monorepo 子项目。
4. 是否有生成目录或 vendor 目录。

Search 工具有两类：

1. 文件名搜索。
2. 文件内容搜索。

Search 返回应该包含：

1. 路径。
2. 行号。
3. 匹配片段。
4. 匹配数量。
5. 是否截断。

好的 search 输出：

```text
status: success
matches: 3
items:
  - path: src/auth/login.ts
    line: 24
    snippet: export async function login(...)
  - path: tests/auth/login.test.ts
    line: 12
    snippet: describe("login", ...)
truncated: false
```

Search 的设计原则：

1. 默认排除 node_modules、.git、dist、build、vendor 等目录。
2. 支持 include/exclude pattern。
3. 返回片段而不是整文件。
4. 输出过多时截断并给 next offset。
5. 不把搜索结果当成完整事实，关键文件仍要 read。

## 5.5 Read：读取文件也有风险

Read 看似安全，但仍有风险。

风险包括：

1. 读取超大文件导致上下文污染。
2. 读取二进制文件产生乱码。
3. 读取密钥或隐私文件。
4. 读取生成文件误导模型。
5. 读取旧文件片段后基于过时内容编辑。

Read 工具应该支持：

1. offset。
2. limit。
3. 行号。
4. 文件大小提示。
5. 文件类型检测。
6. 截断标记。
7. secret 检测。

输出示例：

```text
status: success
path: src/auth/login.ts
offset: 1
limit: 120
total_lines: 260
truncated: true
content: ... with line numbers ...
```

Read 的关键是保留行号。没有行号，模型很难精确讨论和编辑。

## 5.6 编辑方式总览

常见编辑方式包括：

1. Whole-file rewrite。
2. Search/replace。
3. Patch-based edit。
4. AST-based edit。
5. Formatter-assisted edit。
6. Structured refactor。

不同方式适合不同场景。

Whole-file rewrite：

1. 适合小文件或新建文件。
2. 不适合大文件。
3. 容易引入无关 diff。
4. 容易覆盖用户改动。

Search/replace：

1. 适合唯一文本片段替换。
2. 风险是匹配到错误位置。
3. 需要检查匹配数量。

Patch-based edit：

1. 适合大多数代码修改。
2. 基于上下文 hunk。
3. 可以保留局部 diff。
4. 失败时可提示重新读取文件。

AST-based edit：

1. 适合语言结构明确的重构。
2. 对函数、类、import 修改更稳。
3. 实现复杂，需要语言解析器。

Formatter-assisted edit：

1. 用格式化工具整理局部或文件。
2. 风险是引入大量无关 diff。
3. 应谨慎控制范围。

一般原则：优先小 patch，其次 AST edit；谨慎 whole-file rewrite。

## 5.7 为什么优先 Patch-Based Edit

Patch-based edit 是 coding agent 最常用的编辑方式。

它的优势：

1. 修改范围小。
2. 便于 review。
3. 便于 trace。
4. 便于冲突检测。
5. 便于失败恢复。
6. 不容易重写无关内容。

Patch 的基本结构：

```text
file: src/auth/login.ts
old:
  if (!password) {
    throw new Error("invalid password")
  }
new:
  if (!password) {
    throw new ValidationError("password is required")
  }
```

或 unified diff：

```text
@@
-  if (!password) {
-    throw new Error("invalid password")
-  }
+  if (!password) {
+    throw new ValidationError("password is required")
+  }
```

Patch 应用前要检查：

1. 文件存在。
2. old context 唯一或足够明确。
3. 文件未被用户并发修改。
4. 修改路径在 workspace 内。
5. patch 不包含异常大范围删除。

Patch 应用后要记录：

1. 修改文件。
2. diff summary。
3. 行数变化。
4. 是否成功。
5. 错误类型。

## 5.8 Search/Replace 的坑

Search/replace 看起来简单，但风险很高。

常见坑：

1. 匹配到多个位置。
2. 匹配到注释或测试数据。
3. 匹配到相似但语义不同的代码。
4. 空白、格式和换行差异导致匹配失败。
5. 大文件截断后模型以为片段唯一。

安全策略：

1. 替换前统计匹配数量。
2. 多匹配时要求更长上下文。
3. 替换后展示 diff。
4. 不允许无上下文的全局替换。
5. 对大范围 replace 请求用户确认。

例子：

```text
replace all "timeout" with "deadline"
```

这可能会改变量名、文案、配置、测试快照和文档，风险很高。

更安全的做法是先搜索匹配点，再让模型选择具体文件和 hunk。

## 5.9 Whole-File Rewrite 的适用边界

Whole-file rewrite 不是完全不能用。

适合场景：

1. 新建小文件。
2. 重写很短的配置或示例文件。
3. 用户明确要求重写。
4. 文件由 agent 单独创建，且没有用户并发修改。

不适合场景：

1. 大文件。
2. 用户可能同时编辑的文件。
3. 含复杂格式、注释、生成片段的文件。
4. 只需要小改的现有源码。
5. 包含许可证、注释、格式细节的文件。

Whole-file rewrite 风险：

1. 删除用户代码。
2. 丢失注释。
3. 改变格式。
4. 引入无关 diff。
5. 破坏 import 顺序或 generated markers。

Runtime 可以设置策略：超过一定行数的文件禁止 whole-file rewrite，除非用户明确确认。

## 5.10 AST-Based Edit

AST-based edit 是更结构化的编辑方式。

适合：

1. 添加 import。
2. 修改函数签名。
3. 重命名符号。
4. 插入方法。
5. 批量迁移 API。
6. 保持格式和语法正确。

优势：

1. 不依赖脆弱字符串匹配。
2. 能理解语言结构。
3. 适合大规模重构。
4. 可以减少误改。

缺点：

1. 需要语言 parser。
2. 多语言支持成本高。
3. 对不完整代码或宏系统可能不稳。
4. 生成 diff 仍需要 review。

实践中，AST edit 和 patch edit 可以结合：AST 工具负责定位和生成变更，patch 工具负责应用和记录 diff。

## 5.11 Diff 是用户信任的核心

任何写操作后，都应该生成 diff。

Diff 的作用：

1. 用户 review。
2. Agent 自查。
3. Trace 记录。
4. 回滚基础。
5. 最终总结依据。

Diff 应该包含：

1. 修改文件列表。
2. 每个文件的 hunk。
3. 新增/删除行数。
4. 是否包含大文件或二进制文件。
5. 是否包含高风险文件。

Agent 最终总结不能只靠记忆，应基于 diff。

例如：

```text
修改文件：
1. src/auth/login.ts：增加空密码校验。
2. tests/auth/login.test.ts：补充空密码测试。
```

这些应该来自实际 diff，而不是模型猜测。

## 5.12 并发修改检测

用户和 agent 可能同时编辑文件。

并发修改检测方法：

1. 读取文件时记录 hash。
2. 编辑前再次检查 hash。
3. patch 应用时匹配 old context。
4. 检查 git diff 是否出现非 agent 改动。
5. 使用文件 mtime 作为辅助信号。

如果检测到变化：

1. 停止自动编辑。
2. 重新读取文件。
3. 告诉模型发生并发修改。
4. 必要时请求用户确认。
5. 不自动覆盖未知改动。

错误示例：

```text
Agent 读取文件 A，用户修改文件 A，agent 基于旧内容 whole-file rewrite，覆盖用户改动。
```

正确处理：

```text
File changed since last read. Re-read required before editing.
```

并发修改是 coding agent 和普通代码生成器的重要区别之一。

## 5.13 回滚和撤销

Agent 应该支持撤销自己的修改。

回滚方式：

1. 保存 before/after diff。
2. 用 reverse patch 回滚。
3. 利用 git working tree。
4. 对新建文件做删除回滚。
5. 对删除文件保存内容快照。

回滚要注意：

1. 只回滚 agent 自己的改动。
2. 不回滚用户在之后做的改动。
3. 回滚前检查文件是否又发生变化。
4. 回滚操作本身也要进 trace。
5. 高风险回滚需要用户确认。

没有回滚能力，agent 的自动编辑风险会显著上升。

## 5.14 文件类型和特殊文件

不同文件类型要区别处理。

源码文件：

1. 可以 read/search/edit。
2. 优先 patch。
3. 修改后运行测试或类型检查。

配置文件：

1. 小改为主。
2. 注意格式和注释。
3. 修改可能影响全局行为。

锁文件：

1. 不应手写修改。
2. 通常由包管理器生成。
3. 修改需要明确命令和验证。

生成文件：

1. 通常不直接编辑。
2. 应修改源文件或生成规则。
3. 需要识别 generated marker。

二进制文件：

1. 默认不读取为文本。
2. 默认不直接修改。
3. 需要专门工具处理。

隐藏文件和密钥文件：

1. 默认谨慎读取。
2. `.env`、密钥、凭证应默认拒绝。
3. 如果必须读取，应脱敏并请求确认。

迁移文件：

1. 修改可能影响数据库。
2. 需要更严格 review。
3. 不应随意重写历史 migration。

## 5.15 大文件处理

大文件会带来上下文和编辑风险。

策略：

1. 分页读取。
2. 先 search，再 read 局部。
3. 保留行号和 offset。
4. 不 whole-file rewrite。
5. 对编辑区域重新读取。
6. 修改后只返回相关 diff。

大文件中常见风险：

1. 模型看不到完整上下文。
2. 同名函数或变量多处出现。
3. search/replace 误改。
4. 输出截断导致模型以为文件结束。

工具输出必须明确 `truncated: true`，并提供继续读取方式。

## 5.16 代码编辑后的验证

编辑完成不是结束。

后续应做：

1. 生成 diff。
2. 检查是否有无关文件修改。
3. 检查语法或格式。
4. 运行相关测试。
5. 如果编辑失败，回滚或重新读取。
6. 最终总结基于 diff 和验证结果。

如果无法运行测试，也至少要做：

1. 读取修改后的相关片段。
2. 确认 patch 实际生效。
3. 检查 import、括号、缩进等明显问题。
4. 明确告诉用户未运行测试。

## 5.17 文件系统安全

文件系统安全要覆盖读取和写入。

读取风险：

1. 密钥泄露。
2. 隐私泄露。
3. 读取 workspace 外文件。
4. Prompt injection 隐藏在文件内容中。

写入风险：

1. 覆盖用户改动。
2. 修改无关文件。
3. 删除重要文件。
4. 写入恶意代码。
5. 写入 secret 到日志或代码。

安全策略：

1. workspace root 限制。
2. secret pattern denylist。
3. 写操作需要 diff。
4. 高风险文件需要确认。
5. 不自动删除文件。
6. 生成文件和锁文件谨慎处理。
7. 文件内容作为数据，不作为系统指令。

## 5.18 真实坑

常见真实坑包括：

1. 模型把用户并发修改覆盖掉。
2. Search/replace 匹配到错误位置。
3. 大文件上下文截断导致误改。
4. 自动格式化引入大量无关 diff。
5. Agent 修改生成文件，而不是源文件。
6. Agent 手写 lockfile。
7. Agent 删除看似无用但实际被动态加载的文件。
8. Agent 读取 `.env` 并把内容写进 trace。
9. Agent 总结的文件列表和真实 diff 不一致。
10. Agent patch 失败后继续假装已修改。

这些坑说明：文件编辑不是普通文本生成，而是需要强校验的工程操作。

## 5.19 文件编辑质量指标

文件编辑系统要能回答一个问题：这次改动是不是在正确路径、正确上下文、正确权限和可回滚边界内发生的。

可以把一次文件操作样本写成：

```math
e_i=(a_i,r_i,p_i,k_i,m_i,h_i,u_i,b_i,g_i,s_i,d_i,v_i,y_i)
```

其中 $a_i$ 是动作类型，例如 read、search、patch、replace、rewrite、delete；$r_i$ 是用户或模型给出的原始路径，$p_i$ 是规范化并解析后的路径，$k_i$ 是文件类型，$m_i$ 是 old context 匹配数量，$h_i$ 是读取时记录的文件 hash，$u_i$ 表示编辑前是否检测到用户并发修改，$b_i$ 表示是否为二进制文件，$g_i$ 表示是否为 generated / secret / lockfile 等特殊文件，$s_i$ 是 permission gate 决策，$d_i$ 是 diff summary，$v_i$ 是验证结果，$y_i$ 是最终 outcome。

Workspace 越界阻断率：

```math
C_{\mathrm{ws}}=
\frac{\sum_{i\in \mathcal{O}_{\mathrm{out}}}\mathbb{1}[\mathrm{block}(e_i)]}
{|\mathcal{O}_{\mathrm{out}}|}
```

其中 $\mathcal{O}_{\mathrm{out}}$ 是路径穿越、绝对路径、symlink 逃逸或 workspace 外路径的操作集合。这个指标必须接近 1，否则 workspace root 只是文档约定，不是真实边界。

Secret 读取阻断率：

```math
B_{\mathrm{secret}}=
\frac{\sum_{i\in \mathcal{O}_{\mathrm{secret}}}\mathbb{1}[\mathrm{block}(e_i)]}
{|\mathcal{O}_{\mathrm{secret}}|}
```

`.env`、SSH key、云服务凭证、package registry token、浏览器 cookie 和内部配置密钥，不应该直接进入模型上下文或 trace。

读取截断清晰率：

```math
C_{\mathrm{trunc}}=
\frac{1}{|\mathcal{R}_{\mathrm{succ}}|}
\sum_{i\in \mathcal{R}_{\mathrm{succ}}}
\mathbb{1}[\mathrm{has\_trunc\_flag}(e_i)]
```

只要 read / search 输出可能被截断，就必须显式返回 `truncated`、`offset`、`limit`、`total_lines` 或继续读取方式。否则模型会把局部片段误当成完整事实。

Patch 上下文唯一率：

```math
U_{\mathrm{ctx}}=
\frac{1}{|\mathcal{P}|}
\sum_{i\in \mathcal{P}}\mathbb{1}[m_i=1]
```

其中 $\mathcal{P}$ 是依赖 old context 的 replace / patch 操作集合。$m_i=0$ 说明上下文过期或找不到，$m_i>1$ 说明匹配不唯一，二者都不应直接写入。

Patch 应用成功率：

```math
S_{\mathrm{patch}}=
\frac{\sum_{i\in \mathcal{P}}\mathbb{1}[y_i=\mathrm{success}]}
{|\mathcal{P}|}
```

这个指标不能单独看。一个系统可以通过放宽匹配条件提高成功率，但会增加误改和无关 diff。因此它要和 $U_{\mathrm{ctx}}$、$R_{\mathrm{unrel}}$、$C_{\mathrm{conc}}$ 一起看。

并发修改保护率：

```math
C_{\mathrm{conc}}=
\frac{\sum_{i\in \mathcal{O}_{\mathrm{stale}}}\mathbb{1}[\mathrm{block}(e_i)\vee \mathrm{reread}(e_i)]}
{|\mathcal{O}_{\mathrm{stale}}|}
```

其中 $\mathcal{O}_{\mathrm{stale}}$ 是编辑前 hash、mtime 或 old context 已变化的操作集合。正确行为是停止写入、重新读取或请求用户确认，而不是基于旧内容覆盖。

无关 diff 率：

```math
R_{\mathrm{unrel}}=
\frac{\sum_{i\in \mathcal{P}}\mathbb{1}[\mathrm{unrelated\_diff}(e_i)]}
{|\mathcal{P}|}
```

无关 diff 包括顺手格式化、批量重命名、修改未要求文件、重排 import、改动 lockfile 或生成文件。这个指标越低越好。

高风险 whole-file rewrite 拒绝率：

```math
C_{\mathrm{whole}}=
\frac{\sum_{i\in \mathcal{W}_{\mathrm{risk}}}\mathbb{1}[\mathrm{block}(e_i)]}
{|\mathcal{W}_{\mathrm{risk}}|}
```

其中 $\mathcal{W}_{\mathrm{risk}}$ 是对大文件、用户正在编辑文件、生成文件、配置核心文件或未知格式文件做 whole-file rewrite 的请求集合。

回滚就绪率：

```math
C_{\mathrm{rb}}=
\frac{\sum_{i\in \mathcal{P}_{\mathrm{succ}}}\mathbb{1}[\mathrm{reverse\_patch}(e_i)]}
{|\mathcal{P}_{\mathrm{succ}}|}
```

成功写入后至少要能定位 agent 自己的改动、保存 before / after diff，并在回滚前再次检查用户是否又改过同一文件。

Diff 总结忠实率：

```math
F_{\mathrm{diff}}=
\frac{1}{|\mathcal{P}_{\mathrm{succ}}|}
\sum_{i\in \mathcal{P}_{\mathrm{succ}}}
\mathbb{1}[\mathrm{summary}(d_i)=\mathrm{actual\_diff}(e_i)]
```

最终回复用户时，修改文件、行数变化、验证命令和失败边界都应该来自真实 diff 和真实执行结果，而不是模型记忆。

文件编辑门禁：

```math
G_{\mathrm{fileedit}}=
\mathbb{1}[
C_{\mathrm{ws}}\ge \alpha_{\mathrm{ws}}
\wedge B_{\mathrm{secret}}\ge \alpha_{\mathrm{secret}}
\wedge C_{\mathrm{trunc}}\ge \alpha_{\mathrm{trunc}}
\wedge U_{\mathrm{ctx}}\ge \alpha_{\mathrm{ctx}}
\wedge S_{\mathrm{patch}}\ge \alpha_{\mathrm{patch}}
\wedge C_{\mathrm{conc}}\ge \alpha_{\mathrm{conc}}
\wedge R_{\mathrm{unrel}}\le \beta_{\mathrm{unrel}}
\wedge C_{\mathrm{rb}}\ge \alpha_{\mathrm{rb}}
\wedge F_{\mathrm{diff}}\ge \alpha_{\mathrm{diff}}
]
```

面试里可以这样总结：文件编辑系统不是把模型输出写进文件，而是把每一次 read、search、patch 和 diff 都放进 workspace、权限、上下文、并发修改、回滚和验证的约束里。

### 5.19.1 最小可运行文件编辑审计 demo

下面的 0 依赖脚本构造一个 toy workspace，模拟安全 read、路径穿越、secret 读取、二进制读取、generated 文件直改、ambiguous replace、大文件 whole-file rewrite、并发修改、无关 diff 和一次成功 patch。它不访问真实文件系统，只审计文件编辑 runtime 应该如何决策。

```python
import hashlib
import posixpath
from statistics import mean


MAX_READ_LINES = 3
MAX_WHOLE_REWRITE_LINES = 80
SECRET_NAMES = {".env", "id_rsa", "credentials.json"}


def digest(text):
    return hashlib.sha256(text.encode("utf-8")).hexdigest()[:12]


workspace = {
    "README.md": {
        "kind": "text",
        "generated": False,
        "content": "\n".join(
            [
                "# Demo",
                "This file explains the toy project.",
                "It has more lines than one bounded read.",
                "The edit agent must report truncation.",
                "End.",
            ]
        ),
    },
    "src/app.py": {
        "kind": "text",
        "generated": False,
        "content": "def normalize(name):\n    return name\n",
    },
    "src/util.py": {
        "kind": "text",
        "generated": False,
        "content": "timeout = 10\nretry_timeout = timeout + 5\n",
    },
    "src/concurrent.py": {
        "kind": "text",
        "generated": False,
        "content": "value = 2\n",
    },
    "docs/notes.md": {
        "kind": "text",
        "generated": False,
        "content": "internal notes\n",
    },
    "generated/client.py": {
        "kind": "text",
        "generated": True,
        "content": "# generated file\nAPI_VERSION = 'v1'\n",
    },
    "big/data.txt": {
        "kind": "text",
        "generated": False,
        "content": "\n".join(f"row {i}" for i in range(120)),
    },
    "assets/logo.bin": {
        "kind": "binary",
        "generated": False,
        "content": "\x00\x01PNG",
    },
    ".env": {
        "kind": "text",
        "generated": False,
        "content": "API_KEY=do-not-read",
    },
}

initial_hash = {path: digest(meta["content"]) for path, meta in workspace.items()}

ops = [
    {"id": "safe_read_truncated", "action": "read", "paths": ["README.md"], "limit": 3},
    {"id": "path_traversal", "action": "read", "paths": ["../outside.txt"], "limit": 3},
    {"id": "secret_read", "action": "read", "paths": [".env"], "limit": 3},
    {"id": "binary_read", "action": "read", "paths": ["assets/logo.bin"], "limit": 3},
    {"id": "generated_edit", "action": "patch", "paths": ["generated/client.py"], "old": "v1", "new": "v2"},
    {"id": "ambiguous_replace", "action": "replace", "paths": ["src/util.py"], "old": "timeout", "new": "deadline"},
    {"id": "whole_rewrite_big", "action": "whole_rewrite", "paths": ["big/data.txt"], "new": "small\n"},
    {
        "id": "stale_patch",
        "action": "patch",
        "paths": ["src/concurrent.py"],
        "old": "value = 1",
        "new": "value = 3",
        "expected_hash": "stale_hash",
    },
    {
        "id": "unrelated_patch",
        "action": "patch",
        "paths": ["src/app.py", "docs/notes.md"],
        "required_paths": {"src/app.py"},
        "old": "return name",
        "new": "return name.strip()",
    },
    {
        "id": "patch_ok",
        "action": "patch",
        "paths": ["src/app.py"],
        "old": "return name",
        "new": "return name.strip()",
        "validation": True,
    },
]


def safe_div(num, den, default=1.0):
    return default if den == 0 else num / den


def normalize_path(path):
    if path.startswith("/"):
        return None
    parts = [part for part in path.split("/") if part]
    if ".." in parts:
        return None
    normalized = posixpath.normpath(path)
    return None if normalized.startswith("../") else normalized


def is_secret(path):
    return posixpath.basename(path) in SECRET_NAMES or path.endswith(".pem")


def summarize_diff(path, old, new):
    before = workspace[path]["content"]
    after = before.replace(old, new, 1)
    removed = before.count("\n") - after.count("\n")
    added = after.count("\n") - before.count("\n")
    return after, {"path": path, "added_lines": max(0, added), "removed_lines": max(0, removed), "old": old, "new": new}


def audit_op(op):
    normalized_paths = [normalize_path(path) for path in op["paths"]]
    result = {
        "id": op["id"],
        "action": op["action"],
        "status": "blocked",
        "reasons": [],
        "outside_attempt": any(path is None for path in normalized_paths),
        "secret_attempt": False,
        "binary_attempt": False,
        "generated_attempt": False,
        "stale_attempt": False,
        "risky_whole_rewrite": False,
        "context_matches": None,
        "reverse_patch_ready": False,
        "diff_faithful": False,
        "truncated_field_present": False,
    }
    if result["outside_attempt"]:
        result["reasons"].append("path_outside_workspace")
        return result

    paths = normalized_paths
    missing = [path for path in paths if path not in workspace]
    if missing:
        result["reasons"].append("file_not_found")
        return result

    result["secret_attempt"] = any(is_secret(path) for path in paths)
    if result["secret_attempt"]:
        result["reasons"].append("secret_blocked")
        return result

    result["binary_attempt"] = any(workspace[path]["kind"] == "binary" for path in paths)
    if result["binary_attempt"] and op["action"] == "read":
        result["reasons"].append("binary_read_blocked")
        return result

    result["generated_attempt"] = any(workspace[path]["generated"] for path in paths)
    if result["generated_attempt"] and op["action"] != "read":
        result["reasons"].append("generated_file_direct_edit")
        return result

    if op["action"] == "read":
        lines = workspace[paths[0]]["content"].splitlines()
        limit = op.get("limit", MAX_READ_LINES)
        result["status"] = "success"
        result["reasons"] = ["pass"]
        result["truncated"] = len(lines) > limit
        result["truncated_field_present"] = True
        result["returned_lines"] = min(len(lines), limit)
        return result

    if op["action"] == "whole_rewrite":
        line_count = len(workspace[paths[0]]["content"].splitlines())
        result["risky_whole_rewrite"] = line_count > MAX_WHOLE_REWRITE_LINES
        if result["risky_whole_rewrite"]:
            result["reasons"].append("whole_file_rewrite_too_large")
            return result

    expected_hash = op.get("expected_hash", initial_hash.get(paths[0]))
    current_hash = digest(workspace[paths[0]]["content"])
    result["stale_attempt"] = expected_hash != current_hash
    if result["stale_attempt"]:
        result["reasons"].append("concurrent_modification")
        return result

    required_paths = op.get("required_paths", set(paths))
    if any(path not in required_paths for path in paths):
        result["reasons"].append("unrelated_diff")
        return result

    if "old" in op:
        matches = workspace[paths[0]]["content"].count(op["old"])
        result["context_matches"] = matches
        if matches != 1:
            result["reasons"].append("context_not_unique")
            return result

    if op["action"] in {"patch", "replace"}:
        after, diff = summarize_diff(paths[0], op["old"], op["new"])
        workspace[paths[0]]["content"] = after
        result["status"] = "success"
        result["reasons"] = ["pass"]
        result["diff"] = diff
        result["reverse_patch_ready"] = True
        result["diff_faithful"] = diff["path"] == paths[0] and diff["old"] == op["old"] and diff["new"] == op["new"]
        result["validation"] = bool(op.get("validation", False))
        return result

    result["reasons"].append("unsupported_action")
    return result


results = [audit_op(op) for op in ops]
context_results = [r for r in results if r["context_matches"] is not None]
edit_results = [r for r in results if r["action"] in {"patch", "replace", "whole_rewrite"}]
success_edits = [r for r in edit_results if r["status"] == "success"]
outside_attempts = [r for r in results if r["outside_attempt"]]
secret_attempts = [r for r in results if r["secret_attempt"]]
binary_attempts = [r for r in results if r["binary_attempt"]]
read_success = [r for r in results if r["action"] == "read" and r["status"] == "success"]
stale_attempts = [r for r in results if r["stale_attempt"]]
risky_whole = [r for r in results if r["risky_whole_rewrite"]]

metrics = {
    "workspace_containment": round(safe_div(sum("path_outside_workspace" in r["reasons"] for r in outside_attempts), len(outside_attempts)), 3),
    "secret_block": round(safe_div(sum("secret_blocked" in r["reasons"] for r in secret_attempts), len(secret_attempts)), 3),
    "binary_block": round(safe_div(sum("binary_read_blocked" in r["reasons"] for r in binary_attempts), len(binary_attempts)), 3),
    "read_truncation_clarity": round(mean(r["truncated_field_present"] for r in read_success), 3),
    "unique_context": round(mean(r["context_matches"] == 1 for r in context_results), 3),
    "patch_apply_success": round(mean(r["status"] == "success" for r in edit_results), 3),
    "concurrent_protection": round(safe_div(sum("concurrent_modification" in r["reasons"] for r in stale_attempts), len(stale_attempts)), 3),
    "whole_rewrite_reject": round(safe_div(sum("whole_file_rewrite_too_large" in r["reasons"] for r in risky_whole), len(risky_whole)), 3),
    "unrelated_diff_rate": round(mean("unrelated_diff" in r["reasons"] for r in edit_results), 3),
    "rollback_readiness": round(safe_div(sum(r["reverse_patch_ready"] for r in success_edits), len(success_edits)), 3),
    "diff_faithfulness": round(safe_div(sum(r["diff_faithful"] for r in success_edits), len(success_edits)), 3),
}

gates = {
    "workspace_ok": metrics["workspace_containment"] >= 1.0,
    "secret_ok": metrics["secret_block"] >= 1.0,
    "binary_ok": metrics["binary_block"] >= 1.0,
    "truncation_ok": metrics["read_truncation_clarity"] >= 1.0,
    "context_ok": metrics["unique_context"] >= 0.8,
    "patch_success_ok": metrics["patch_apply_success"] >= 0.7,
    "concurrent_ok": metrics["concurrent_protection"] >= 1.0,
    "whole_rewrite_ok": metrics["whole_rewrite_reject"] >= 1.0,
    "unrelated_diff_ok": metrics["unrelated_diff_rate"] == 0.0,
    "rollback_ok": metrics["rollback_readiness"] >= 1.0,
    "diff_ok": metrics["diff_faithfulness"] >= 1.0,
}

print("results=", {r["id"]: (r["status"], r["reasons"]) for r in results})
print("metrics=", metrics)
print("failed_gates=", [name for name, ok in gates.items() if not ok])
print("file_edit_gate_pass=", all(gates.values()))
```

典型输出会看到路径穿越、secret、二进制、generated 文件、ambiguous replace、大文件 whole-file rewrite、并发修改和无关 diff 都被阻断；只有 `patch_ok` 成功写入，并且有 reverse patch 和 diff summary。`file_edit_gate_pass=False` 是刻意保留的，因为 demo 里有上下文不唯一、patch 成功率不足和无关 diff 尝试，用来说明门禁不仅看“有没有一次改成功”，还要看失败是否被正确拦截。

## 5.20 面试题：如何设计 Coding Agent 的文件编辑系统

回答要点：

```text
我会把文件系统能力分成只读和写入两类。只读包括 list、search、read、diff，写入包括 apply patch、create、delete、move。所有路径必须限制在 workspace root 内，secret 文件默认拒绝。编辑上优先 patch-based edit，避免 whole-file rewrite；编辑前检查文件 hash 或上下文，防止覆盖用户并发修改；编辑后生成 diff、记录 trace，并运行相关测试。删除、大范围格式化、锁文件和迁移文件修改需要更严格确认。
```

## 5.21 面试题：为什么不建议 Whole-File Rewrite

回答要点：

```text
Whole-file rewrite 容易引入无关 diff、丢失注释、改变格式、覆盖用户并发修改，也让 review 和回滚更困难。它只适合新建小文件或用户明确要求重写的场景。对已有源码，更安全的是基于上下文的小 patch 或 AST-based edit，并在应用后生成 diff 和运行验证。
```

## 5.22 面试题：如何避免覆盖用户修改

回答要点：

```text
读取文件时记录版本信息，例如 hash、mtime 或 git 状态；编辑前重新校验文件是否变化；patch 应用时要求 old context 匹配；如果发现用户并发修改，就停止自动编辑、重新读取文件，并请求用户确认。回滚也只能回滚 agent 自己的改动，不能覆盖用户后续修改。
```

## 5.24 本章总结

本章讨论了 coding agent 的文件系统和代码编辑能力。

核心结论：

1. 文件系统能力要分只读和写入，默认只读，写入更严格。
2. Workspace root 是文件权限边界的基础。
3. Search 用于定位，Read 用于获取精确上下文，Edit 用于小范围修改。
4. Patch-based edit 通常比 whole-file rewrite 更安全。
5. Search/replace 必须检查匹配数量和上下文。
6. Diff 是用户信任、trace、回滚和总结的基础。
7. 并发修改检测是 coding agent 必须具备的能力。
8. 大文件、生成文件、锁文件、隐藏文件和迁移文件都要特殊处理。
9. 文件内容是数据，不是指令，要防 prompt injection 和 secret 泄露。
