# 第六章：终端执行与命令安全

## 6.1 本章目标

终端执行是 coding agent 最强的能力之一，也是最高风险的能力之一。它可以运行测试、构建项目、查看环境、复现 bug；也可以删除文件、上传数据、安装恶意依赖、修改系统配置、泄露密钥。

第五章讲文件系统和代码编辑，本章专门讲终端执行。终端工具的核心难点不是“能不能跑命令”，而是“如何让模型提出的命令在可控边界内执行，并把结果安全、结构化地返回给模型”。

学完本章，你应该能回答：

1. 为什么 terminal execution 是 harness 中最高风险的能力之一。
2. 如何对命令做风险分级。
3. 哪些命令可以自动执行，哪些必须确认，哪些应该禁止。
4. 如何设计 Bash 工具的输入、输出、超时和错误处理。
5. 如何处理 stdout/stderr 过长和日志中的敏感信息。
6. 如何用 sandbox 降低执行风险。
7. 面试中如何设计安全的命令执行系统。

## 6.2 为什么终端危险

终端工具可以做很多有用的事情：

1. 运行单元测试。
2. 运行 lint 和 type check。
3. 构建项目。
4. 查看 git 状态。
5. 检查依赖版本。
6. 复现用户报错。
7. 生成代码或文档。

但它也能做危险事情：

1. 删除文件。
2. 清空目录。
3. 修改数据库。
4. 安装不可信依赖。
5. 上传代码或数据。
6. 读取密钥。
7. 修改系统配置。
8. 启动长时间或高资源进程。
9. 执行仓库里的恶意脚本。

因此 terminal execution 不能被当成普通工具。它必须有命令分析、权限控制、超时、沙箱、输出脱敏和 trace。

面试表达：

```text
我会把终端执行看成 coding agent 中最高风险的工具之一。它既能运行测试验证修改，也能删除文件、访问网络、读取密钥或执行不可信脚本。所以 Bash 工具必须限制工作目录、设置超时、做命令风险分级、必要时请求用户确认，并对输出做截断和脱敏。
```

## 6.3 命令风险分级

命令不能只分允许和禁止。建议分级处理。

低风险只读命令：

1. 查看版本：`node --version`、`python --version`。
2. 查看状态：`git status`、`git diff --stat`。
3. 列出测试：部分测试框架支持 list。
4. 运行只读检查：lint、type check、unit test。

中风险写入命令：

1. 生成文件。
2. 格式化代码。
3. 更新 snapshot。
4. 生成 lockfile。
5. 本地构建产物写入 dist/build。

高风险命令：

1. 删除文件：`rm`、`del`。
2. 重置改动：`git reset`、`git checkout`、`git clean`。
3. 安装依赖：`npm install`、`pip install`、`curl | sh`。
4. 数据库迁移或清空。
5. 访问网络、上传下载。
6. 修改系统配置。
7. 使用 sudo 或管理员权限。
8. 读取 secret 文件或环境变量。

命令风险还取决于上下文。

例如：

```text
npm test
```

通常是低到中风险，但如果项目测试会启动容器、写数据库或调用外部 API，就变成更高风险。

所以风险分级不能只看命令名，也要看参数、工作目录、项目脚本内容和历史策略。

## 6.4 命令执行输入设计

Bash 工具输入不能只有 `command`。

更完整的输入 schema：

```text
command: string, required
working_dir: string, required, must be inside workspace
reason: string, required
expected_effect: enum(read_only, writes_files, installs_dependencies, network_access, modifies_database, unknown)
requires_network: boolean
```

字段含义：

1. command：要执行的命令。
2. working_dir：命令工作目录。
3. timeout_seconds：超时限制。
4. reason：为什么要执行。
5. expected_effect：模型声明预期副作用。
6. requires_network：是否需要网络访问。

为什么要 reason？

因为用户和 trace 需要知道命令目的。

例如：

```text
command: npm test -- auth
reason: verify auth tests after modifying login validation
expected_effect: read_only
```

如果模型给出的 expected_effect 和命令分析结果不一致，runtime 应该提高风险等级。

## 6.5 命令执行输出设计

命令输出必须结构化。

输出 schema 可以包括：

```text
status: success | failed | timeout | denied | cancelled
command: string
working_dir: string
exit_code: integer optional
stdout: string
stderr: string
duration_ms: integer
truncated: boolean
error_type: string optional
summary: string optional
```

成功不等于任务完成。

例如 `grep` 没匹配到内容可能 exit code 非零，但不是系统错误；测试命令 exit code 非零表示测试失败，是有价值反馈。

所以 runtime 要区分：

1. command execution failed：命令无法运行。
2. command completed with non-zero exit：命令运行了，但业务结果失败。
3. timeout：命令超时。
4. denied：权限系统拒绝。
5. cancelled：用户取消。

返回给模型时要保留关键 stdout/stderr，但避免把超长输出全部塞进上下文。

## 6.6 Allowlist 和 Denylist

命令安全常用 allowlist 和 denylist。

Allowlist 适合自动执行低风险命令：

1. `git status`。
2. `git diff`。
3. `npm test`。
4. `pytest`。
5. `go test`。
6. `cargo test`。
7. `npm run lint`。
8. `tsc --noEmit`。

Denylist 适合拦截明显危险命令：

1. `rm -rf`。
2. `sudo`。
3. `chmod -R 777`。
4. `curl ... | sh`。
5. `wget ... | bash`。
6. `git reset --hard`。
7. `git clean -fd`。
8. 读取 `.env`、SSH key、cloud credentials。

但只靠字符串匹配不够。

原因：

1. 命令可以通过 shell 拼接绕过。
2. 脚本内部可能执行危险动作。
3. 安全命令也可能带危险参数。
4. 不同平台命令不同。

更稳妥的做法是 allowlist、denylist、权限确认、沙箱和审计结合。

## 6.7 用户确认策略

并不是所有命令都需要用户确认，否则 agent 会很难用。

建议策略：

自动允许：

1. 明确只读的状态查询。
2. 相关测试命令。
3. lint 和 type check。
4. 无网络、无写入的构建检查。

需要确认：

1. 安装依赖。
2. 修改文件的格式化命令。
3. 更新 snapshot。
4. 执行项目脚本但副作用不明。
5. 数据库迁移。
6. 网络访问。

默认拒绝或强确认：

1. 删除文件或目录。
2. reset/clean 工作区。
3. 读取密钥。
4. 上传数据。
5. 修改系统目录。
6. 使用管理员权限。

确认内容应该包含：

1. 命令是什么。
2. 工作目录是什么。
3. 为什么要执行。
4. 预期副作用是什么。
5. 风险等级是什么。

用户确认本身也要进入 trace。

## 6.8 工作目录限制

命令必须在受控工作目录内执行。

风险包括：

1. 在错误目录执行删除或格式化。
2. 从子项目切换到父目录影响更多文件。
3. 使用绝对路径访问 workspace 外文件。
4. 通过 symlink 逃逸。

策略：

1. working_dir 必须在 workspace root 内。
2. 禁止隐式 `cd` 到未知路径。
3. 对路径做规范化和 symlink 解析。
4. 命令执行前展示工作目录。
5. Trace 记录 working_dir。

很多事故不是命令本身危险，而是在错误目录执行了命令。

## 6.9 超时和资源限制

Agent 运行命令必须有超时。

没有超时的问题：

1. 测试卡住。
2. dev server 常驻。
3. install 长时间等待。
4. 命令持续输出占满上下文或日志。
5. 多个任务堆积占用资源。

策略：

1. 默认超时。
2. 用户可确认延长超时。
3. 区分短命令和长命令。
4. 支持取消。
5. 捕获部分输出。
6. 记录 duration。

资源限制还包括：

1. CPU。
2. 内存。
3. 磁盘写入。
4. 子进程数量。
5. 网络访问。

生产级 runtime 最好把命令放在受控 sandbox 或容器里运行。

## 6.10 输出截断和摘要

命令输出可能非常长。

长输出来源：

1. 测试失败日志。
2. 构建日志。
3. 依赖安装日志。
4. 全量 lint 输出。
5. 运行时 stack trace。

处理策略：

1. 保留完整输出在 trace 或 artifact 中。
2. 返回给模型的是摘要和关键片段。
3. 优先保留错误行、失败测试名、stack trace 顶部和底部。
4. 标记 truncated。
5. 提供继续读取方式。

示例：

```text
status: failed
exit_code: 1
summary: 2 tests failed in auth/login.test.ts
stderr_excerpt: ...
truncated: true
full_output_ref: trace://cmd-123
```

不要让一段超长日志挤掉用户目标、当前计划和关键代码上下文。

## 6.11 环境变量和密钥脱敏

命令输出可能泄露敏感信息。

敏感信息包括：

1. API key。
2. token。
3. cookie。
4. 数据库连接串。
5. 云服务凭证。
6. 私钥路径。
7. 用户隐私数据。

风险来源：

1. `env` 命令。
2. 测试日志打印配置。
3. 失败堆栈包含连接串。
4. 工具把完整 stdout 写入 trace。
5. 模型在总结中复述敏感内容。

防护：

1. 默认禁止直接运行 `env`、`printenv` 等命令。
2. 对 stdout/stderr 做 secret masking。
3. Trace 中保存脱敏版本。
4. 高权限环境变量不暴露给子进程。
5. 最终回答禁止复述密钥。

注意：脱敏不是万能的。更好的方式是最小化环境变量暴露。

## 6.12 网络访问控制

网络命令风险很高。

风险包括：

1. 上传代码或数据。
2. 下载并执行未知脚本。
3. 安装不可信依赖。
4. 访问内部服务。
5. 泄露 prompt、trace 或文件内容。

策略：

1. 默认禁用网络或需要确认。
2. 区分 external read 和 external write。
3. 对允许域名做 allowlist。
4. 禁止 `curl | sh` 类命令。
5. 下载内容不自动执行。
6. Web 内容标记为不可信。
7. Trace 记录网络目标。

对 coding agent 来说，很多任务不需要网络。如果需要安装依赖或查文档，应该明确说明原因并请求确认。

## 6.13 测试命令并不总是安全

很多人以为测试命令一定安全，但并非如此。

测试命令可能：

1. 写数据库。
2. 启动容器。
3. 调用外部 API。
4. 写 snapshot。
5. 修改临时文件。
6. 读取环境变量。
7. 运行项目脚本。

策略：

1. 优先运行最小范围测试。
2. 使用测试环境变量。
3. 禁止生产数据库连接。
4. 对 snapshot update 单独确认。
5. 对集成测试和端到端测试提高风险等级。
6. 记录测试命令和结果。

运行测试是验证的关键，但仍要理解测试副作用。

## 6.14 安装依赖和包管理器命令

安装依赖是高风险操作。

风险：

1. 下载不可信包。
2. 执行 install scripts。
3. 修改 lockfile。
4. 改变项目依赖树。
5. 触发网络访问。
6. 引入供应链风险。

策略：

1. 默认请求用户确认。
2. 说明为什么必须安装。
3. 优先使用已有依赖。
4. 禁止自动执行任意 install scripts，或在 sandbox 中执行。
5. 修改 lockfile 后展示 diff。
6. 记录包名、版本和来源。

Agent 不应该因为测试缺依赖就自动安装一堆包，除非用户明确同意。

## 6.15 Git 命令安全

Git 命令有只读和高风险之分。

低风险：

1. `git status`。
2. `git diff`。
3. `git log`。
4. `git branch --show-current`。

高风险：

1. `git reset --hard`。
2. `git checkout -- <file>`。
3. `git clean -fd`。
4. `git rebase`。
5. `git push`。
6. `git commit`。
7. `git stash drop`。

策略：

1. 只读 Git 命令可自动执行。
2. 修改工作区、历史或远程的命令必须确认。
3. 不自动提交、推送或重写历史。
4. 回滚文件前确认不会覆盖用户改动。
5. Trace 记录 Git 操作。

Git 是保护用户改动的重要工具，但危险命令也可能直接破坏工作区。

## 6.16 沙箱执行

Sandbox 可以降低命令风险。

沙箱可以限制：

1. 文件系统访问。
2. 网络访问。
3. 环境变量。
4. CPU 和内存。
5. 子进程。
6. 系统调用。
7. 运行时间。

常见方式：

1. 容器。
2. 临时工作区副本。
3. 只读挂载。
4. 网络隔离。
5. 受限用户权限。
6. 远程 sandbox。

沙箱也有代价：

1. 启动慢。
2. 环境和本地不一致。
3. 文件同步复杂。
4. 调试成本高。
5. 某些测试依赖外部服务。

实际系统要根据任务风险决定是否必须 sandbox。

## 6.17 命令执行 Trace

每次命令执行都应进入 trace。

Trace 字段：

1. command。
2. working_dir。
3. reason。
4. risk_level。
5. permission decision。
6. start time / end time。
7. exit code。
8. stdout/stderr 摘要。
9. truncated 标记。
10. secret masking 标记。
11. artifacts。
12. error type。

Trace 的价值：

1. 用户审计 agent 做了什么。
2. Debug 命令失败原因。
3. 统计测试通过率和命令超时率。
4. 评估不同 agent 版本的工具使用效率。
5. 安全审计危险命令是否被拦截。

没有 trace 的命令执行不可审计。

## 6.18 真实坑

常见真实坑：

1. 测试命令实际会修改数据库。
2. install 命令引入不可信依赖。
3. 日志输出泄露密钥。
4. 模型建议执行危险清理命令。
5. 命令在错误目录运行。
6. 长时间 dev server 卡住 agent。
7. `git reset --hard` 清掉用户改动。
8. `curl | sh` 执行未知脚本。
9. 测试输出太长挤掉关键上下文。
10. 命令失败但 agent 误以为成功。

经验法则：只要命令能改变环境，就必须纳入权限、trace 和回滚思考。

## 6.19 面试题：如何设计安全的终端执行工具

回答要点：

```text
我会把终端工具设计成受控执行器，而不是直接把 shell 暴露给模型。输入要包含 command、working_dir、timeout、reason 和 expected_effect；执行前做命令风险分级、路径限制和权限判断；低风险测试和只读命令可以自动执行，高风险删除、安装、网络、数据库和 git reset 类命令必须确认或禁止；执行时设置超时和资源限制；输出要结构化、截断、脱敏并写入 trace。高风险任务可以放进 sandbox。
```

## 6.20 面试题：为什么测试命令也需要风险控制

回答要点：

```text
测试命令不一定纯只读。有些测试会写数据库、更新 snapshot、启动容器、调用外部 API 或读取敏感环境变量。因此 runtime 应该优先运行最小范围测试，并根据测试命令和项目脚本判断副作用。对于 snapshot update、集成测试、端到端测试或需要网络和数据库的测试，要提高风险等级并记录 trace。
```

## 6.21 面试题：命令输出太长怎么办

回答要点：

```text
完整输出可以保存在 trace 或 artifact 中，但返回给模型的应该是结构化摘要。对于测试失败，保留失败测试名、关键断言、相关 stack trace 和 exit code；对构建日志保留 first error 和 summary。输出要标记 truncated，并提供继续读取 full output 的引用。这样既不丢失可追踪性，也不会让长日志挤掉关键上下文。
```

## 6.22 小练习

1. 为 Bash 工具设计 input schema 和 output schema。
2. 给 10 条命令做风险分级：`git status`、`npm test`、`npm install`、`rm -rf dist`、`git reset --hard`、`pytest`、`curl URL | sh`、`docker compose up`、`npm run format`、`cat .env`。
3. 设计一个命令确认弹窗应该展示哪些字段。
4. 设计 stdout/stderr 的截断和摘要策略。
5. 思考如何检测日志中的密钥并脱敏。
6. 思考如何在 sandbox 中运行不可信测试。
7. 思考 agent 执行命令超时后应该如何反馈给模型。
8. 用 3 分钟回答“为什么 Bash 工具不能直接无约束暴露给模型”。

## 6.23 本章总结

本章讨论了终端执行与命令安全。

核心结论：

1. 终端执行是 coding agent 中最高风险的能力之一。
2. 命令要按只读、写入、破坏性、网络、权限等维度分级。
3. Bash 工具输入应包含 command、working_dir、timeout、reason 和 expected_effect。
4. 命令执行输出要结构化，区分 failed、timeout、denied、cancelled 和非零 exit code。
5. Allowlist、denylist、用户确认、沙箱和 trace 要结合使用。
6. 测试命令、安装依赖和 Git 命令都可能有副作用。
7. 长输出要截断、摘要，并保留完整 trace。
8. 环境变量、密钥和网络访问要默认谨慎。
9. 高风险命令不能只靠 prompt 约束，必须 runtime 硬控制。

下一章会进入上下文管理与记忆压缩，讨论 harness 如何在代码库、工具结果、终端输出、diff 和历史对话之间选择最有价值的信息放进模型上下文。
