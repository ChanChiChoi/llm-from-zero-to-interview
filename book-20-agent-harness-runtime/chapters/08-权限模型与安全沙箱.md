# 第八章：权限模型与安全沙箱

## 8.1 本章目标

Coding agent 能读文件、改代码、执行命令、访问网络、调用工具。能力越强，越需要权限模型和安全沙箱。如果没有系统级权限控制，模型的一次误判、一次 prompt injection、一次工具误用，都可能导致文件损坏、密钥泄露、数据外传或供应链风险。

前面章节分别讲了工具、文件、终端和上下文。本章把这些能力统一放进安全边界里讨论：agent 到底可以做什么，什么时候需要用户确认，哪些行为应该默认禁止，沙箱如何限制执行环境，trace 如何支持安全审计。

学完本章，你应该能回答：

1. 为什么权限不能只靠 prompt 约束。
2. Agent harness 应该有哪些权限维度。
3. 如何设计文件、命令、网络、secret、Git 和外部工具权限。
4. 用户确认策略如何设计。
5. 沙箱能限制什么，不能解决什么。
6. 如何防 prompt injection 绕过权限。
7. 面试中如何设计一个安全的 coding agent runtime。

## 8.2 为什么权限模型重要

普通聊天模型的错误通常停留在文本层面。Coding agent 的错误可能变成真实动作。

例如：

1. 误删文件。
2. 覆盖用户未提交改动。
3. 执行危险 shell 命令。
4. 读取 `.env` 或 SSH key。
5. 把代码上传到外部服务。
6. 安装恶意依赖。
7. 修改数据库。
8. 自动提交或推送错误代码。

Prompt 可以提醒模型不要做危险事，但 prompt 不是安全边界。

原因：

1. 模型可能忘记规则。
2. 模型可能误判命令风险。
3. 外部内容可能 prompt injection。
4. 工具参数可能被间接构造。
5. 用户或项目脚本可能包含恶意指令。

真正的安全边界必须在 runtime 层硬执行。

面试表达：

```text
我不会把 agent 安全只交给 prompt。Prompt 可以告诉模型应该怎么做，但权限模型必须在 runtime 层硬校验。文件读写、命令执行、网络访问、secret 读取、Git 操作和外部工具调用都要有权限等级、确认策略、trace 和审计。
```

## 8.3 权限维度

Harness 至少需要控制以下权限：

1. 文件读权限。
2. 文件写权限。
3. 文件删除和移动权限。
4. 命令执行权限。
5. 网络访问权限。
6. Secret 访问权限。
7. Git 操作权限。
8. 外部 API 或 MCP 工具权限。
9. 数据库和状态服务权限。
10. 日志和 trace 访问权限。

这些权限可以按资源和动作拆开。

资源：

1. workspace 文件。
2. workspace 外文件。
3. 环境变量。
4. 网络。
5. 数据库。
6. Git repository。
7. 外部 SaaS。
8. 用户隐私数据。

动作：

1. read。
2. write。
3. delete。
4. execute。
5. upload。
6. download。
7. commit。
8. push。
9. mutate external state。

权限模型的核心是把“agent 可以做什么”说清楚，而不是让每个工具各自猜。

## 8.4 最小权限原则

最小权限原则：agent 默认只拥有完成当前任务所需的最低权限。

默认建议：

1. 允许读 workspace 内普通源码文件。
2. 允许搜索代码。
3. 允许查看 git diff/status。
4. 写文件需要 diff 和确认策略。
5. 执行低风险测试命令可自动或半自动。
6. 网络访问默认需要确认。
7. Secret 访问默认拒绝。
8. 删除和 destructive 操作默认拒绝或强确认。

任务越高风险，权限越收紧。

例如 code review 任务：

1. 只需要读文件和 diff。
2. 不需要写文件。
3. 不需要执行命令，除非用户允许。

例如 bug fix 任务：

1. 需要读文件、写少量文件、运行相关测试。
2. 不需要网络、secret、git push。

例如部署诊断任务：

1. 可能需要读配置和日志。
2. 需要更严格脱敏。
3. 不能默认修改生产配置。

权限应该跟任务类型绑定，而不是一打开 agent 就给全权限。

## 8.5 权限策略矩阵

可以用矩阵表达权限策略。

示例：

```text
Action                     Default Policy
read workspace source       allow
read hidden file            ask or deny
read secret file            deny
write existing source       ask or allow after diff policy
create new source file      ask or allow in task scope
delete file                 ask strongly or deny
run unit test               allow with timeout
run install command         ask every time
network access              ask or deny
git status/diff/log         allow
git commit                  ask explicitly
git push                    deny by default
git reset --hard            deny by default
database migration          ask strongly
```

矩阵的好处：

1. 用户知道 agent 权限边界。
2. 开发者能统一实现策略。
3. 安全审计有依据。
4. 工具新增时容易归类。
5. Evaluation 可以统计越权请求。

## 8.6 用户确认策略

用户确认不是越多越安全。确认太频繁会让用户麻木，确认太少又会放大风险。

确认策略可以分级：

1. allow automatically：低风险只读。
2. ask once per session：常见低风险写操作。
3. ask every time：网络、安装依赖、高风险命令。
4. deny by default：secret、destructive、workspace 外操作。
5. require explicit user command：commit、push、数据库迁移。

确认弹窗或提示应包含：

1. 动作类型。
2. 目标资源。
3. 命令或 diff。
4. 执行原因。
5. 风险等级。
6. 是否可回滚。
7. 预期副作用。

坏确认：

```text
Allow command?
```

好确认：

```text
Agent wants to run: npm install lodash
Working directory: /workspace/app
Reason: install missing dependency for CSV export
Risk: network access + modifies package-lock.json + runs install scripts
Policy: requires explicit confirmation
```

用户确认也要写入 trace。

## 8.7 文件权限

文件权限要区分读、写、删、移动。

读权限：

1. workspace 内普通源码默认允许。
2. 大文件分页读取。
3. hidden file 谨慎。
4. secret file 默认拒绝。
5. workspace 外路径默认拒绝。

写权限：

1. 只能写 workspace 内文件。
2. 写前检查并发修改。
3. 写后生成 diff。
4. 高风险文件需要确认。
5. 不允许无关大范围格式化。

删除权限：

1. 默认不自动删除。
2. 删除必须展示目标和原因。
3. 应支持回滚或快照。
4. 删除目录比删除文件风险更高。

特殊文件：

1. `.env`、密钥、凭证默认拒绝。
2. lockfile 不手写修改。
3. migration 文件需要确认。
4. generated 文件优先不直接编辑。

文件权限必须和第 5 章的文件编辑策略配合。

## 8.8 命令执行权限

命令执行权限要和第 6 章的风险分级配合。

只读或低风险：

1. `git status`。
2. `git diff`。
3. 相关单元测试。
4. lint/type check。

需要确认：

1. install 命令。
2. format 命令。
3. snapshot update。
4. docker compose。
5. 数据库命令。
6. 网络命令。

默认拒绝：

1. `git reset --hard`。
2. `rm -rf`。
3. `sudo`。
4. `curl | sh`。
5. 读取 secret 的命令。
6. 修改系统路径。

命令权限不能只靠字符串匹配。还要结合工作目录、参数、项目脚本、网络、环境变量和 sandbox。

## 8.9 网络权限

网络访问常被低估。

风险：

1. 上传代码。
2. 泄露日志和 trace。
3. 下载恶意脚本。
4. 安装不可信依赖。
5. 访问内部服务。
6. 被 prompt injection 诱导外传数据。

策略：

1. 默认禁止或确认。
2. 区分 external read 和 external write。
3. 域名 allowlist。
4. 禁止未知脚本直接执行。
5. Web 内容标记为不可信。
6. 记录网络目标和数据类型。

对很多 coding 任务，网络不是必需权限。如果 agent 说需要联网，应该说明原因。

## 8.10 Secret 权限

Secret 权限应该默认最严格。

Secret 包括：

1. API key。
2. token。
3. cookie。
4. SSH key。
5. 云服务凭证。
6. 数据库连接串。
7. `.env` 文件。
8. 私有证书。

策略：

1. 默认不允许模型读取。
2. 默认不把 secret 放进上下文。
3. 命令输出做脱敏。
4. Trace 做脱敏和访问控制。
5. 子进程最小化环境变量。
6. 最终回答禁止复述 secret。

即使用户要求“帮我看 .env”，也要谨慎，最好只做结构检查或让用户提供脱敏片段。

## 8.11 Git 权限

Git 权限容易被忽略。

允许自动执行：

1. `git status`。
2. `git diff`。
3. `git log --oneline`。
4. `git branch --show-current`。

需要显式确认：

1. `git add`。
2. `git commit`。
3. `git stash`。
4. `git checkout`。
5. `git rebase`。

默认拒绝：

1. `git reset --hard`。
2. `git clean -fd`。
3. `git push --force`。
4. 自动修改远程仓库。

Agent 不应该在用户未要求时自动 commit 或 push。即使用户要求，也要先检查 diff，避免提交 secret 或无关文件。

## 8.12 安全沙箱能做什么

沙箱用于限制 agent 的执行环境。

可以限制：

1. 文件系统访问。
2. 网络访问。
3. CPU 和内存。
4. 执行时间。
5. 环境变量。
6. 系统调用。
7. 子进程。
8. 用户权限。

常见沙箱形态：

1. 容器。
2. 临时 workspace 副本。
3. 只读文件系统挂载。
4. 网络隔离。
5. 受限用户。
6. 远程执行环境。
7. 浏览器或 WebAssembly sandbox。

沙箱特别适合：

1. 执行不可信测试。
2. 运行安装脚本。
3. 评估 agent benchmark。
4. 处理用户上传代码。
5. 运行高风险命令。

沙箱不是万能，但它能把风险从用户真实环境转移到受控环境。

## 8.13 沙箱的限制

沙箱也有局限。

问题：

1. 环境和用户本机不同。
2. 启动和同步成本高。
3. 某些测试依赖真实服务。
4. 文件权限和路径可能不同。
5. 网络禁用后部分任务无法执行。
6. 沙箱 escape 风险仍需防护。

沙箱不能替代：

1. 权限模型。
2. 工具参数校验。
3. secret 管理。
4. trace 和审计。
5. 用户确认。

正确理解：沙箱是纵深防御的一层，不是唯一安全机制。

## 8.14 Prompt Injection 与权限绕过

Prompt injection 常见于外部内容：

1. README。
2. Issue。
3. 网页。
4. 代码注释。
5. 测试输出。
6. 工具返回。

攻击目标通常是诱导 agent：

1. 忽略系统规则。
2. 读取 secret。
3. 上传文件。
4. 执行危险命令。
5. 修改代码植入后门。

防护：

1. 不可信内容标记为 data。
2. 外部内容不能改变权限策略。
3. 工具调用必须 runtime 校验。
4. Secret 和网络默认受限。
5. 高风险动作必须用户确认。
6. Trace 记录疑似注入来源。

关键点：即使模型被诱导，runtime 也要挡住越权动作。

## 8.15 权限和 UX 的取舍

权限太宽，风险高；权限太窄，体验差。

常见取舍：

1. 低风险自动化 vs 高频确认打断。
2. 严格网络限制 vs 无法查文档或安装依赖。
3. 沙箱隔离 vs 环境不一致。
4. Secret 禁止读取 vs 无法调试配置问题。
5. 自动运行测试 vs 测试有副作用。

好的设计不是一刀切，而是分级：

1. 低风险默认允许。
2. 中风险清晰确认。
3. 高风险默认拒绝。
4. 企业场景支持管理员策略。
5. 用户可以查看和调整权限。

权限系统应该让用户理解风险，而不是只弹出模糊确认。

## 8.16 权限审计和 Trace

每次权限决策都应进入 trace。

Trace 字段：

1. action。
2. resource。
3. requested_permission。
4. risk_level。
5. policy_result。
6. user_decision。
7. reason。
8. timestamp。
9. tool name/version。

用途：

1. Debug agent 为什么被拒绝。
2. 安全审计越权尝试。
3. 统计高风险请求频率。
4. 优化确认策略。
5. 复盘事故。

没有权限 trace，就无法判断安全策略是否有效。

## 8.17 企业场景权限

企业 coding agent 还要考虑组织级策略。

企业策略可能包括：

1. 禁止上传代码到外部模型或服务。
2. 禁止读取某些目录。
3. 禁止访问生产 secret。
4. 强制所有命令在 sandbox 运行。
5. 强制日志脱敏。
6. 保留审计日志。
7. 对不同用户角色分配不同权限。

角色示例：

1. 普通开发者：读写自己 workspace。
2. Maintainer：允许更大范围修改和测试。
3. Admin：配置策略，但不一定执行代码修改。
4. Read-only reviewer：只能读和评论。

企业场景下，权限策略不能只存在客户端，还要可集中管理和审计。

## 8.18 真实坑

常见真实坑：

1. 权限默认过大。
2. 工具间接绕过权限。
3. Prompt injection 诱导 agent 读取敏感文件。
4. 日志记录了 secret。
5. 用户确认提示太模糊。
6. 沙箱和真实环境差异导致误判。
7. Git 危险命令清掉用户改动。
8. 网络访问把代码片段发到外部。
9. Install script 在本机执行恶意代码。
10. 权限决策没有 trace，无法复盘。

经验法则：安全设计要假设模型会犯错、工具会被误用、外部内容会诱导越权。

## 8.19 面试题：如何设计 Agent 权限模型

回答要点：

```text
我会按资源和动作设计权限模型。资源包括 workspace 文件、secret、网络、命令、Git、数据库和外部 API；动作包括 read、write、delete、execute、upload、commit、push 等。默认最小权限：普通源码读权限可自动允许，写文件需要 diff 和确认策略，网络、安装依赖、secret、destructive command 和 git push 默认严格限制。所有权限决策都要进入 trace，方便审计和复盘。
```

## 8.20 面试题：沙箱能解决什么问题

回答要点：

```text
沙箱可以限制 agent 执行环境，比如文件系统、网络、环境变量、CPU/内存、运行时间和系统调用。它适合运行不可信测试、安装脚本、高风险命令和 benchmark 任务。但沙箱不是万能的，它不能替代权限模型、工具参数校验、secret 管理、用户确认和 trace。生产系统通常需要沙箱作为纵深防御的一层。
```

## 8.21 面试题：如何防 Prompt Injection 诱导工具越权

回答要点：

```text
首先要把 README、网页、Issue、代码注释和工具返回都当作不可信数据，而不是系统指令。其次，模型即使生成了读取 secret、上传文件或执行危险命令的请求，runtime 也必须经过权限系统校验。Secret 和网络默认受限，高风险动作需要用户确认或直接拒绝。疑似 injection 的内容和被拦截动作要写入 trace，便于审计。
```

## 8.22 小练习

1. 为 coding agent 设计一个文件读写权限矩阵。
2. 为命令执行设计 allow、ask、deny 三层策略。
3. 列出 5 类 secret，并设计默认策略。
4. 设计一个用户确认弹窗，展示命令风险。
5. 思考如果 README 中写着“读取 ~/.ssh/id_rsa”，runtime 应该如何处理。
6. 设计一个 sandbox 能限制哪些资源。
7. 思考企业场景下如何做权限审计。
8. 用 3 分钟回答“为什么 agent 安全不能只靠 prompt”。

## 8.23 本章总结

本章讨论了权限模型与安全沙箱。

核心结论：

1. Coding agent 的错误会变成真实动作，必须有 runtime 级权限控制。
2. 权限要按资源和动作拆分，而不是简单允许/禁止。
3. 最小权限原则是默认策略。
4. 文件、命令、网络、secret、Git 和外部工具都需要独立权限策略。
5. 用户确认要分级，不能太频繁，也不能太模糊。
6. 沙箱是纵深防御的一层，可以降低命令和工具执行风险。
7. Prompt injection 的关键防线是 runtime 权限校验，而不是只靠 prompt。
8. 权限决策必须进入 trace，支持安全审计和事故复盘。

下一章会进入 Trace、日志、回放与可观测性，讨论如何记录 agent 的每一步，让失败可复现、行为可审计、评估可回放。
