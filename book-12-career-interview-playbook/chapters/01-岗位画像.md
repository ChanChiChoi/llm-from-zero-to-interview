# 第一章：岗位画像

很多人准备大模型算法岗面试时，会犯一个根本错误：把目标理解成“我要证明自己懂很多知识点”。于是疯狂背 Transformer、RLHF、LoRA、RAG、KV cache、MoE、FlashAttention、SFT、DPO、benchmark。知识当然重要，但真正的面试目标不是展示知识点数量，而是证明你能在一个高标准大模型团队里创造价值。

岗位画像就是回答一个问题：对方到底在招什么样的人？

如果你不知道岗位画像，就很容易准备错方向。比如你把大部分时间花在刷 LeetCode，但目标岗位更看重研究判断和训练经验。或者你拼命讲论文细节，但面试官更想知道你是否能把模型能力落到产品影响。又或者你只强调模型效果，却没有体现安全意识、实验严谨性和工程可靠性。

本章重点：OpenAI 当前大模型算法岗可能关注的能力：研究、训练、对齐、评估、多模态、系统、产品影响和安全意识。

说明：本章不代表任何公司的官方招聘标准，而是基于公开岗位描述、大模型行业通用要求和高水平 AI 团队的能力模型，总结一套求职准备视角。准备时应以目标岗位的最新 JD、公开博客、论文、产品和招聘沟通为准。

## 1.1 先建立直觉：大模型算法岗不是单一岗位

“大模型算法岗”这个词很宽。不同公司、不同团队、不同阶段，对它的定义完全不同。

有的岗位偏研究，希望你提出新方法、设计实验、写论文、推动模型能力边界。有的岗位偏训练工程，希望你能把万卡训练跑稳，定位 loss spike，优化吞吐和显存。有的岗位偏对齐，希望你理解人类偏好、安全、RLHF、DPO、红队和行为评估。有的岗位偏应用算法，希望你把模型能力转化为产品功能，比如代码助手、语音助手、企业知识库、Agent。有的岗位偏评估，希望你建立可靠 benchmark、自动评测和线上质量闭环。

所以准备面试之前，第一步不是打开资料开始学，而是判断目标岗位属于哪类画像。

可以把大模型相关岗位粗略分为八类：

| 岗位方向 | 核心价值 |
| --- | --- |
| 基础模型研究 | 提升模型能力边界，探索新架构、新训练方法、新对齐方法 |
| 预训练与训练系统 | 构建数据、训练、并行、稳定性、效率和大规模实验能力 |
| Post-training / 对齐 | 让模型更有用、更安全、更符合人类偏好和产品目标 |
| 评估与模型行为 | 定义、测量、解释和改进模型能力与风险 |
| 多模态 | 处理文本、图像、语音、视频和跨模态推理 |
| 推理与部署 | 降低延迟和成本，提高吞吐、可靠性和可扩展性 |
| Agent 与应用算法 | 把模型接入工具、记忆、规划、检索和业务工作流 |
| 数据与数据智能 | 构建高质量训练、微调、偏好、评估和安全数据闭环 |

这些方向有交集，但面试重点不同。你必须把自己的简历、项目、表达和复习时间对齐目标方向。

## 1.2 高水平大模型团队到底看什么

顶级大模型团队通常不只看“你会不会某个技术”。他们更看重你是否具备长期产生高质量技术判断的能力。

可以把能力拆成八个维度。

### 1.2.1 研究能力

研究能力不是“读过很多论文”，而是能提出有价值的问题、设计合理实验、解释结果，并从失败中获得信息。

面试中会观察：

1. 你能否讲清一个方法为什么被提出。
2. 你能否解释它解决了什么问题。
3. 你能否说出它的适用边界。
4. 你能否指出它和相邻方法的关系。
5. 你能否设计实验验证一个假设。
6. 你能否在结果不符合预期时提出排查路径。

一个强信号是：你不是背论文结论，而是能讲清“如果让我复现或改进这篇工作，我会怎么做”。

### 1.2.2 训练能力

训练能力包括数据、模型、优化器、并行、稳定性、监控、checkpoint、评估和故障排查。

面试中可能会问：

1. 训练 loss 突然 spike 怎么排查？
2. 数据重复率过高会有什么影响？
3. batch size、learning rate、warmup、weight decay 怎么影响训练？
4. 混合精度训练为什么可能不稳定？
5. ZeRO、tensor parallel、pipeline parallel、sequence parallel 分别解决什么问题？
6. 如何判断一个训练 run 是否值得继续？

高水平团队很重视训练直觉，因为大模型训练成本极高。一次错误实验可能浪费大量 GPU 时间。

### 1.2.3 对齐与安全意识

大模型不是只要聪明就够了，还要可控、有用、安全、诚实、稳健。

对齐与安全能力包括：SFT、RLHF、RLAIF、DPO、偏好数据、安全数据、拒答策略、红队、越狱鲁棒性、误杀控制、滥用防护、隐私保护和评估。

面试中会观察你是否能平衡两个目标：

1. 不能让模型轻易产生高风险输出。
2. 也不能让模型对正常请求过度拒答。

一个成熟候选人不会只说“加安全分类器”，而会讨论 policy、数据、评估、红队、人工复核、线上反馈和审计闭环。

### 1.2.4 评估能力

评估能力是大模型岗位的核心分水岭。

因为大模型很多能力没有简单准确率。写作质量、推理能力、工具使用、幻觉、安全、长上下文、多轮一致性，都需要设计合适评估。

面试中会关注：

1. 你能否区分离线 benchmark 和线上用户体验。
2. 你是否知道 benchmark 可能被污染或过拟合。
3. 你能否设计人工评审 rubric。
4. 你能否使用 LLM-as-judge，同时理解它的偏差。
5. 你能否把评估结果转化为下一轮改进。
6. 你能否处理指标提升但用户体验下降的情况。

强候选人会把评估视为研发闭环，而不是项目最后的打分步骤。

### 1.2.5 系统与工程能力

很多大模型岗位要求研究和工程兼备。

你不一定要是基础设施专家，但必须理解你的算法如何在真实系统中运行。

系统能力包括：推理延迟、吞吐、KV cache、batching、模型路由、缓存、RAG、数据管道、监控、灰度、回滚和可观测性。

面试中常见观察点：

1. 你是否知道模型质量不是唯一目标。
2. 你是否考虑成本和延迟。
3. 你是否能设计可复现实验和可上线流程。
4. 你是否懂线上失败、超时、降级和监控。
5. 你是否能和平台、产品、安全团队协作。

### 1.2.6 产品影响

顶级团队通常不只看技术炫技，还看技术是否产生真实影响。

产品影响可以是：用户满意度提升、任务成功率提升、幻觉下降、延迟下降、成本下降、覆盖更多语言、支持新场景、安全事故减少、人工审核量下降。

面试中讲项目时，最好不要只说“我实现了一个 RAG 系统”，而要说：

```text
我负责把企业知识库问答从纯 prompt stuffing 改成 hybrid retrieval + rerank + citation 的 RAG 链路。离线评测中引用准确率从 62% 提升到 81%，线上人工客服转接率下降 18%，p95 延迟控制在 3.2 秒以内。
```

数字不一定都要夸张，但要体现你知道如何定义和衡量影响。

### 1.2.7 协作与表达能力

大模型项目通常跨团队。一个模型能力改进可能涉及数据、训练、评估、产品、安全、基础设施和法律合规。

面试官会观察你是否能：

1. 讲清复杂问题。
2. 接受追问和挑战。
3. 承认不确定性。
4. 明确区分事实、假设和猜测。
5. 给出可执行下一步。
6. 和不同角色沟通 trade-off。

如果你在面试中把每个问题都讲成绝对结论，反而可能显得不成熟。大模型领域变化快，优秀候选人通常会表达边界和证据。

### 1.2.8 学习速度与适应能力

大模型领域变化非常快。两年前重要的技术栈，今天可能已经变成默认组件。今天热门的方法，明年可能被更简单的方法替代。

所以高水平团队会看你是否具备快速学习和迁移能力。

强信号包括：

1. 能快速理解新论文的核心贡献。
2. 能把一个领域的方法迁移到另一个领域。
3. 能从失败实验中更新判断。
4. 能持续跟踪技术变化但不盲目追热点。
5. 能把复杂问题拆成可验证的小实验。

## 1.3 不同岗位方向的画像

### 1.3.1 基础模型研究岗

基础模型研究岗关注模型能力边界。

可能涉及：新架构、预训练目标、优化方法、数据配比、长上下文、推理能力、工具使用、可解释性、多模态融合、世界模型等。

强候选人画像：

1. 有扎实数学、机器学习和深度学习基础。
2. 能读懂并批判前沿论文。
3. 有独立提出假设和设计实验的能力。
4. 能处理实验失败和不确定结果。
5. 有高质量论文、开源复现或研究型项目更好。

面试重点：论文深挖、研究品味、实验设计、结果解释、开放问题讨论。

常见弱点：只会复述论文，不会讲为什么重要；只会调参，不会形成假设；只追 SOTA，不关心机制。

### 1.3.2 预训练与训练工程岗

预训练与训练工程岗关注如何把大规模训练稳定、高效、可控地跑起来。

强候选人画像：

1. 熟悉 Transformer、优化器、学习率调度和训练稳定性。
2. 理解数据质量、去重、配比和污染检测。
3. 理解分布式训练和并行策略。
4. 能定位 loss spike、NaN、吞吐下降、checkpoint 异常。
5. 有大规模训练、微调或高性能训练经验。

面试重点：训练故障排查、并行策略、算力效率、实验监控、数据问题。

常见弱点：只知道算法公式，不知道训练系统；只会单卡实验，不理解大规模失败模式。

### 1.3.3 Post-training 与对齐岗

Post-training 关注如何把 base model 变成可用助手。

涉及 SFT、偏好数据、RLHF、DPO、reward model、安全数据、工具调用数据、多轮对话数据、风格控制和拒答行为。

强候选人画像：

1. 理解 base model、instruction tuning、preference tuning 的区别。
2. 理解偏好数据质量和标注一致性。
3. 能设计对齐实验和评估。
4. 关注 helpfulness、honesty、harmlessness 的平衡。
5. 能处理过度拒答、奖励黑客、风格退化等问题。

面试重点：RLHF/DPO 机制、数据设计、评估、安全、失败案例分析。

常见弱点：把对齐理解成单一算法；忽略数据和 policy；只看 win rate，不看副作用。

### 1.3.4 评估与模型行为岗

评估岗关注“我们怎么知道模型真的变好了”。

强候选人画像：

1. 能设计 benchmark 和线上指标。
2. 理解自动评测、人工评审和 LLM-as-judge 的优缺点。
3. 能分析模型行为变化。
4. 能发现数据污染、指标过拟合和评估偏差。
5. 能把评估结果变成训练、数据或产品改进建议。

面试重点：评估集构造、rubric、统计显著性、A/B 实验、错误分析。

常见弱点：只会跑公开榜单；不会定义业务指标；不会解释指标和用户体验不一致。

### 1.3.5 多模态岗

多模态岗关注文本、图像、语音、视频之间的理解和生成。

强候选人画像：

1. 理解视觉编码器、语言模型、跨模态对齐和生成模型。
2. 理解图文、语音、视频数据的清洗和标注。
3. 能处理 OCR、图表理解、空间关系、时序建模等问题。
4. 熟悉多模态评估和安全风险。
5. 有实际多模态项目或论文经验。

面试重点：跨模态表示、数据构造、训练策略、评估、产品场景。

常见弱点：只懂文本 LLM；不了解视觉或语音数据特性；忽略延迟和安全。

### 1.3.6 推理部署岗

推理部署岗关注如何让模型低成本、低延迟、高可靠地服务用户。

强候选人画像：

1. 理解 prefill、decode、KV cache、batching 和显存瓶颈。
2. 熟悉量化、蒸馏、推测解码、模型并行和 serving 框架。
3. 能分析 p95/p99 延迟和吞吐。
4. 能设计模型路由、缓存和降级。
5. 理解线上稳定性、监控和容量规划。

面试重点：性能瓶颈分析、成本优化、系统设计、线上故障处理。

常见弱点：只会说“用 vLLM”；不知道为什么快；不理解 workload 和 SLA。

### 1.3.7 Agent 与应用算法岗

Agent 与应用算法岗关注把模型接入真实任务。

强候选人画像：

1. 理解 RAG、工具调用、规划、记忆、状态管理和权限。
2. 能把复杂业务流程拆成模型可执行步骤。
3. 能设计失败恢复、重试、人工接管和审计。
4. 关注任务成功率而不是单轮回答质量。
5. 理解安全、成本和用户体验。

面试重点：系统设计、任务分解、工具安全、评估、线上闭环。

常见弱点：把 Agent 当成 prompt 技巧；忽略权限和失败恢复；只展示 demo，不衡量任务成功。

### 1.3.8 数据岗与数据智能岗

数据方向关注训练数据、微调数据、偏好数据、评估数据和安全数据。

强候选人画像：

1. 理解数据采集、清洗、去重、质量评分和污染检测。
2. 理解 data mixture 和 curriculum。
3. 能构建高质量 instruction、preference、reasoning、code、math 数据。
4. 能评估数据对模型能力的影响。
5. 关注版权、隐私、合规和治理。

面试重点：数据质量、数据实验、标注体系、数据闭环、治理。

常见弱点：只说“数据越多越好”；不理解数据分布和能力之间的关系。

## 1.4 OpenAI 风格岗位可能更强调什么

如果目标是 OpenAI 这类前沿大模型团队，需要特别关注几个倾向。

第一，研究和工程结合。

只会写论文但不能落地，或者只会工程实现但没有研究判断，都可能不够。前沿团队需要能把想法变成实验，把实验变成系统，把系统结果反过来推动研究判断的人。

第二，安全和社会影响意识。

大模型能力越强，风险越大。候选人需要理解模型可能被误用、可能产生偏见、可能泄露隐私、可能在高风险领域给出错误建议。安全不是面试附加题，而是核心能力之一。

第三，评估严谨性。

模型能力提升常常很微妙。没有严谨评估，就容易被 cherry-pick demo 误导。强团队会看重你是否能建立可靠证据。

第四，产品影响。

研究最终要服务模型能力和用户价值。你需要能把技术指标和真实用户问题连接起来。

第五，处理不确定性的能力。

前沿问题没有标准答案。面试官可能故意问开放问题，看你如何思考、如何分解、如何承认未知、如何提出实验。

第六，沟通清晰。

复杂项目需要跨团队协作。表达混乱会降低可信度，即使技术很强。

## 1.5 面试官如何从回答中判断你

同一个问题，不同回答暴露完全不同的层次。

例如面试官问：

```text
如果一个模型在数学推理 benchmark 上提升了，但线上用户反馈变差，你怎么分析？
```

弱回答：

```text
可能 benchmark 不准，再换一个 benchmark。
```

中等回答：

```text
我会看线上问题类型是否和 benchmark 分布不同，也会看模型是否变得更啰嗦或更容易拒答。
```

强回答：

```text
我会先确认变化是否显著，并分桶分析用户反馈：任务类型、语言、上下文长度、用户群体、模型版本、prompt 版本和路由策略。然后检查 benchmark 是否和线上分布一致，是否存在数据污染或过拟合。接着做错误分析，看是推理错误、格式错误、过度拒答、响应变慢、风格变差，还是某些高频场景退化。最后用线上样本构建回归集，设计 A/B 或 shadow test 验证修复方案。
```

强回答体现了实验、评估、产品、系统和排查能力。

## 1.6 简历和面试要围绕岗位画像组织

岗位画像不是只用于理解 JD，还要反向指导你的简历和面试表达。

如果你投训练方向，简历应该突出：训练规模、数据规模、并行策略、稳定性问题、吞吐优化、loss 曲线、checkpoint、实验管理。

如果你投对齐方向，简历应该突出：SFT/RLHF/DPO、偏好数据、reward model、安全评估、红队、拒答策略、人工标注和线上反馈。

如果你投 RAG/Agent 方向，简历应该突出：检索质量、工具调用、任务成功率、权限、安全、延迟、成本和产品指标。

如果你投评估方向，简历应该突出：benchmark、rubric、LLM-as-judge、人工评审一致性、A/B、回归测试和错误分析。

如果你投多模态方向，简历应该突出：数据类型、模型结构、跨模态对齐、生成质量、时序理解、评估和延迟。

一个项目可以有多个讲法。比如同一个企业知识库项目：

训练方向讲数据质量和微调。

系统方向讲索引、检索、延迟和缓存。

评估方向讲 benchmark、citation accuracy 和人工评审。

产品方向讲客服转接率下降和用户满意度提升。

面试准备的关键，就是把同一段经历重组为目标岗位最关心的证据。

## 1.7 候选人常见错位

### 1.7.1 把“知道”误认为“会做”

知道 RLHF 的流程，不等于会设计偏好数据实验。

知道 FlashAttention，不等于会分析推理瓶颈。

知道 RAG，不等于会处理权限、引用、更新和幻觉。

面试官通常会通过追问把“知道概念”和“做过事情”区分开。

### 1.7.2 只准备热门词，不准备底层逻辑

热门词会变化，但底层逻辑稳定。

例如 LoRA、QLoRA、DPO、RAG、MoE 都是具体方法。更底层的问题是：参数高效适配、偏好优化、知识注入、条件计算、数据效率、推理效率。

如果你能讲底层逻辑，就算遇到新方法也能现场推理。

### 1.7.3 只讲结果，不讲过程

“准确率提升 5%”不够。面试官想知道你怎么发现问题、怎么提出假设、怎么设计实验、怎么排除干扰、怎么确认结果可信。

过程体现能力，结果体现影响。两者都要讲。

### 1.7.4 只讲成功，不讲失败

真实项目都会失败。强候选人能讲清失败实验带来了什么信息。

例如：某个 reranker 离线提升但线上变慢，最后改成只对 top-50 候选重排；某个 DPO 数据提升 win rate 但导致过度拒答，后来重新分桶安全样本和 helpfulness 样本。

### 1.7.5 忽略安全和合规

在大模型岗位中，安全和合规不是“加分项”，而是底线。尤其是面向真实用户、企业数据、多模态输入、工具调用和自动化执行的系统。

## 1.8 如何读 JD

读 JD 时不要只看关键词，要拆成能力要求。

例如 JD 写：

```text
Experience with large-scale training and post-training of foundation models.
```

这背后可能意味着：

1. 你懂预训练和 post-training 的区别。
2. 你有训练实验经验。
3. 你理解大规模训练的稳定性和效率。
4. 你知道如何评估模型能力变化。
5. 你能在数据、算法和系统之间做 trade-off。

JD 写：

```text
Build evaluations for model behavior and safety.
```

这背后可能意味着：

1. 你能定义行为类别。
2. 你能构造评估数据。
3. 你能设计 rubric。
4. 你理解自动评估和人工评估的偏差。
5. 你能把评估接入模型迭代和上线门禁。

JD 写：

```text
Improve product experiences using frontier models.
```

这背后可能意味着：

1. 你能理解用户场景。
2. 你能把模型能力转化为功能。
3. 你能定义产品指标。
4. 你能处理延迟、成本、安全和质量。
5. 你能和产品、设计、工程协作。

读 JD 的目标，是把它翻译成你需要提供的证据。

## 1.10 准备优先级

如果时间有限，可以按优先级准备。

第一优先级：你的简历项目。

面试官一定会围绕你的项目深挖。你必须能讲清背景、目标、方法、指标、实验、失败、取舍和影响。

第二优先级：目标岗位核心能力。

投训练就重点准备训练。投对齐就重点准备 SFT/RLHF/DPO 和安全。投 RAG/Agent 就重点准备系统设计、检索、工具和评估。

第三优先级：大模型通用基础。

Transformer、attention、优化器、预训练、微调、推理、评估、安全，这些是底座。

第四优先级：coding 和系统设计。

不同岗位权重不同，但不能完全忽略。尤其是工程型和应用型岗位。

第五优先级：行为面试。

高水平团队非常重视协作、ownership、处理失败和价值观匹配。

## 1.11 本书后续章节如何使用

本书后续章节会围绕求职完整链路展开。

第二章讲简历策略，重点是如何把项目写成“岗位证据”。

第三章讲项目包装，重点是如何准备项目深挖和 STAR 技术叙事。

第四章讲 coding interview，重点是算法题、工程题和代码表达。

第五到第十章分别覆盖 ML/LLM 基础、training、deployment、alignment/safety、multimodal/research 和 system design 面试。

第十一章讲 behavioral interview。

第十二章讲 mock interview 与复盘。

第十三章给六个月冲刺计划。

你可以先用本章确定目标岗位画像，再决定后续章节的阅读顺序和投入比例。

## 1.12 小结

大模型算法岗不是一个单一岗位，而是一组围绕基础模型、训练、对齐、评估、多模态、部署、应用和数据的能力组合。

准备面试时，不要只问“我要学哪些知识点”，而要问：

1. 目标岗位要我证明什么能力？
2. 我的经历里哪些内容能作为证据？
3. 我在哪些追问上容易暴露短板？
4. 我的项目如何连接研究、工程、评估、安全和产品影响？
5. 我如何用清晰表达让面试官相信我能在真实团队里创造价值？

岗位画像不是求职包装术，而是准备方向盘。方向对了，后面的简历、项目、知识复习和 mock interview 才不会各自发散。
