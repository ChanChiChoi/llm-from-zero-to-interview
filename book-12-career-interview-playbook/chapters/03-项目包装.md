# 第三章：项目包装

简历让你进入面试，项目讲述决定面试官是否相信你真的做过、做懂了、做出了价值。

很多候选人的项目本身并不差，但讲出来很弱。常见表现是：从背景讲到技术栈，再讲到结果，中间没有问题意识，没有关键决策，没有失败和反思。面试官听完只能知道“你参与过一个项目”，但不知道你是不是核心贡献者，也不知道你是否具备独立解决复杂问题的能力。

项目包装不是夸大，也不是把普通项目讲成惊天动地的大项目。项目包装的本质是把真实经历组织成清晰、可信、可追问的技术故事。

本章重点：把项目讲成问题、方法、实验、结果、失败、反思和下一步。

## 3.1 什么是好的项目讲述

一个好的项目讲述，应该让面试官在 3 到 5 分钟内听懂七件事：

1. 你解决的是什么问题。
2. 这个问题为什么重要。
3. 难点和约束在哪里。
4. 你尝试了哪些方案。
5. 为什么最终选择这个方案。
6. 结果如何验证。
7. 你从中学到了什么。

如果只能讲清“用了什么技术”，说明项目还停留在工具层。如果能讲清“为什么这样做、为什么不是别的方案、失败在哪里、指标为什么可信”，才说明你真正理解项目。

大模型算法岗的项目讲述尤其看重三类能力：

1. 技术判断：能不能在很多方法中选择合适方案。
2. 实验能力：能不能设计实验、解释结果、排查异常。
3. 系统意识：能不能考虑数据、模型、评估、成本、安全和产品约束。

## 3.2 项目讲述的标准结构

推荐使用一个七段式结构：

1. 背景：业务、研究或工程场景是什么。
2. 问题：原系统或 baseline 有什么不足。
3. 难点：为什么这个问题不好解决。
4. 方法：你做了哪些关键设计。
5. 实验：你如何验证方法有效。
6. 结果：指标、现象和影响是什么。
7. 反思：失败、边界和下一步是什么。

这不是固定模板，而是一种组织逻辑。实际面试中，你可以先用 1 分钟讲概览，再根据面试官兴趣展开。

一个简洁开场可以这样说：

```text
我想讲一个 7B 模型在企业问答场景中的 post-training 项目。这个项目的目标不是单纯提高 benchmark 分数，而是降低幻觉和答非所问问题，让模型在真实企业知识库问答中更可用。我主要负责数据构建、SFT/DPO 训练和评估闭环。最终人评可用率从 71% 提升到 84%，幻觉类错误率从 18% 降到 11%。
```

这段开场包含场景、目标、角色、方法和结果。面试官可以立刻决定追问数据、训练、评估或失败案例。

## 3.3 背景：不要讲太久，但要讲清意义

背景的作用是让面试官知道项目为什么存在。背景不需要讲公司历史、团队架构或无关业务细节，只需要交代和技术问题相关的信息。

好的背景通常包括：

1. 使用场景：模型用于什么任务。
2. 用户或系统需求：为什么现有方案不够。
3. 约束条件：数据、算力、延迟、安全、成本或上线周期。
4. 项目目标：希望改善什么。

差的背景：

```text
我们公司有一个客服系统，业务比较复杂，有很多用户问题，所以我们做了一个大模型问答项目。
```

更好的背景：

```text
项目面向企业客服知识库问答。原系统基于检索和模板回复，覆盖率不足；直接接入通用大模型后，回答流畅但容易出现知识库外编造。项目目标是在不训练更大模型的前提下，通过领域数据和评估闭环提高回答可用率，并控制幻觉和误拒。
```

这个背景讲清了场景、原方案、问题、约束和目标。

## 3.4 问题：从“做了什么”转向“解决什么”

项目讲述最重要的转变，是从“我做了什么”转向“我解决了什么问题”。

同样是 LoRA 微调，背后的问题可能完全不同：

1. 领域知识不足。
2. 输出格式不稳定。
3. 多轮对话上下文利用差。
4. 安全拒答不稳定。
5. 训练成本受限。
6. 小样本数据下容易过拟合。

如果你只说“我用了 LoRA”，面试官无法判断技术含量。如果你说“我们在小样本领域数据下发现全参微调成本高、LoRA 容易过拟合，所以重点做了数据去重、rank 选择和验证集早停”，技术深度就出现了。

可以用下面的问题帮助自己提炼项目核心问题：

1. 如果没有这个项目，会有什么损失？
2. 原方法最主要的失败模式是什么？
3. 这个问题为什么不能用简单方案解决？
4. 你的方案针对的是哪个失败模式？
5. 面试官如果只记住一句话，应该记住什么？

项目的核心问题越清楚，后面的技术细节越容易展开。

## 3.5 难点：展示技术判断的入口

很多候选人不敢讲难点，担心暴露项目不完美。实际上，难点是项目最有价值的部分。没有难点的项目，很难体现候选人能力。

大模型项目常见难点包括：

| 方向 | 常见难点 |
| --- | --- |
| 数据 | 噪声、重复、覆盖不足、标注不一致、数据泄漏、PII 和版权边界 |
| 训练 | 显存不足、loss spike、收敛慢、过拟合、混合精度不稳定 |
| 对齐 | 偏好数据难构造、奖励黑客、安全和有用性冲突、误拒率上升 |
| 评估 | 自动指标不可信、人评成本高、样本选择偏差、线上离线不一致 |
| 推理 | 延迟高、吞吐低、KV cache 占用大、batching 策略复杂 |
| RAG | 召回不足、chunk 粒度、rerank 成本、上下文污染、引用不稳定 |
| Agent | 工具调用失败、规划不稳定、状态管理复杂、长任务评估困难 |

讲难点时，不要只列问题，要讲你如何判断优先级。

例如：

```text
我们最初以为主要问题是模型不懂领域知识，但错误分析发现，约一半错误来自检索召回不足和上下文冲突，只有一部分来自模型生成能力。所以后续没有盲目扩大微调数据，而是先改了检索切分和 rerank，再针对高频失败样本做 SFT。
```

这段话能体现你不是看到效果差就堆训练，而是做了错误分析和问题归因。

## 3.6 方法：讲决策链，不要只讲技术栈

方法部分最常见的问题是变成技术名词串联：用了 PyTorch、LoRA、DeepSpeed、RAG、FAISS、reranker、vLLM。工具本身不是重点，重点是你为什么这么设计。

好的方法讲述应该包含决策链：

1. 候选方案有哪些。
2. 选择标准是什么。
3. 为什么选当前方案。
4. 如何控制风险。
5. 如何验证是否有效。

例如，讲 SFT 和 DPO 项目时，可以这样组织：

```text
我们先做 SFT，因为基础模型在领域格式和任务指令上不稳定。SFT 后可用率提升，但模型仍然会在知识不足时强答，所以第二阶段构造偏好数据做 DPO，把“承认不知道并引用知识库”的回答作为 chosen，把编造答案作为 rejected。没有直接上 RLHF，是因为当时奖励模型数据不足，训练和调试成本也更高。
```

这段话回答了为什么 SFT、为什么 DPO、为什么不是 RLHF。

再比如讲推理优化：

```text
最初瓶颈不是模型计算，而是请求长度差异导致 batch padding 浪费。因此我们先按输入长度分桶并调整 continuous batching 策略，而不是直接做量化。量化后延迟下降有限，但输出质量在部分任务上变差，所以最终只对低风险场景启用 8-bit 权重。
```

这里体现了对瓶颈定位、方案排序和质量 trade-off 的理解。

## 3.7 实验：让故事从“我觉得”变成“我验证过”

项目讲述必须有实验或验证。不管是研究、训练、对齐、RAG、推理还是产品项目，都需要说明你如何知道方案有效。

实验部分要回答：

1. baseline 是什么。
2. 对比方案是什么。
3. 指标是什么。
4. 数据集或样本怎么来。
5. 是否做了消融。
6. 是否排除了数据泄漏和评估偏差。
7. 结果是否稳定。

大模型项目里，评估尤其容易出问题。比如自动指标可能无法反映真实质量，LLM-as-judge 可能偏向长答案，人评可能标注不一致，benchmark 可能和训练数据污染。

因此，讲实验时最好同时讲指标和误差分析。

差的说法：

```text
我们模型效果提升了 10%。
```

更好的说法：

```text
我们用 500 条人工构造的领域问答作为固定验证集，同时每周从线上日志抽样 200 条做人工评测。自动指标主要看引用命中和格式正确率，人评看答案正确性、完整性和是否幻觉。最终固定集可用率提升 13 个百分点，线上抽样提升 9 到 11 个百分点，说明不是只对固定集过拟合。
```

这个说法更可信，因为它说明了评估来源、指标和稳定性。

## 3.8 结果：讲清提升，也讲清代价

结果不是只报一个好看的数字。面试官会关心结果是否真实、是否稳定、是否有代价。

可以从四个层面讲结果：

1. 效果指标：准确率、任务完成率、win rate、人评可用率。
2. 效率指标：训练时间、吞吐、延迟、显存、成本。
3. 稳定性指标：失败率、中断次数、线上报警、回滚次数。
4. 负面影响：误拒、延迟上升、成本增加、某些子任务下降。

强候选人不会只讲“提升”，还会讲 trade-off。

例如：

```text
DPO 后模型在安全拒答上更稳定，有害请求拒答率提升明显，但早期版本误拒率也上升了。我们后续把安全数据按风险等级拆分，并加入正常边界问题作为 chosen，才把误拒率拉回可接受范围。
```

这个回答比单纯说“安全效果提升”更高级，因为它展示了对齐任务中常见的有用性和安全性冲突。

## 3.9 失败：失败经历是加分项，不是扣分项

很多面试官会问：“这个项目里最大的失败是什么？”

这不是陷阱题。面试官想看你是否真的做过项目，以及你如何从失败中学习。

一个好的失败案例应该包含：

1. 当时的假设是什么。
2. 为什么这个假设看起来合理。
3. 实验结果为什么不符合预期。
4. 你如何定位原因。
5. 后续如何调整。
6. 你从中得到什么原则。

差的回答：

```text
项目比较顺利，没有什么失败。
```

这个回答通常不可信。真实项目几乎一定有失败、反复和取舍。

更好的回答：

```text
我们一开始认为增加领域 SFT 数据一定能降低幻觉，所以扩充了很多问答对。但验证发现幻觉下降不明显，部分答案还变得更自信。后来做错误分析发现，很多问题本身需要检索外部知识，单靠 SFT 会让模型记住回答风格，却不能保证事实来源。后续我们把重点转到 RAG 召回和引用约束，再用 SFT 稳定输出格式。
```

这类失败案例能体现问题归因能力。

## 3.10 反思和下一步：展示成长性

项目讲完结果后，不要突然结束。可以用反思和下一步收尾。

反思可以包括：

1. 当前方案的边界是什么。
2. 哪些指标还不够可靠。
3. 哪些问题没有解决。
4. 如果资源更多，你会怎么做。
5. 如果重新开始，你会少走哪些弯路。

例如：

```text
这个项目最大的局限是评估集覆盖还不够，尤其是长尾问题和多轮上下文。下一步我会优先建设更系统的错误 taxonomy，并把线上失败样本按类型回流到数据和评估集中，而不是继续盲目增加通用 SFT 数据。
```

这个收尾能让面试官看到你不是只会执行任务，而是能从系统角度继续改进。

## 3.11 如何准备 1 分钟、3 分钟、10 分钟版本

同一个项目要准备三个版本。

### 3.11.1 1 分钟版本

用于自我介绍或面试官快速扫项目。

结构：场景、目标、你的角色、方法、结果。

示例：

```text
我做过一个企业知识库问答模型优化项目，目标是降低幻觉并提高答案可用率。我主要负责数据构建、SFT/DPO 训练和评估闭环。我们先通过错误分析发现主要问题来自知识库外编造和格式不稳定，然后用领域 SFT 稳定任务格式，再构造偏好数据让模型在证据不足时拒答或引用来源。最终人评可用率从 71% 提升到 84%，幻觉类错误率从 18% 降到 11%。
```

### 3.11.2 3 分钟版本

用于正式介绍项目。

结构：背景、问题、难点、方法、实验、结果、反思。

3 分钟版本要比 1 分钟版本多讲难点和实验，但不要展开所有实现细节。目标是给面试官留下追问入口。

### 3.11.3 10 分钟版本

用于深挖。

10 分钟版本要准备细节：

1. 数据怎么构造。
2. 模型和训练参数怎么选。
3. baseline 是什么。
4. 消融实验怎么做。
5. 失败案例是什么。
6. 指标如何定义。
7. 成本和延迟如何变化。
8. 线上和离线是否一致。
9. 安全和合规如何处理。
10. 如果重做会怎么改。

项目准备到 10 分钟版本，才算真正能扛住追问。

## 3.12 不同类型项目怎么讲

### 3.12.1 训练项目

训练项目要突出规模、稳定性、效率和实验判断。

重点问题包括：

1. 模型规模、数据规模和训练资源。
2. 训练目标和 tokenizer。
3. batch size、learning rate、warmup、weight decay 等超参。
4. 并行策略和显存优化。
5. loss spike、NaN、梯度异常如何处理。
6. checkpoint、恢复和监控体系。
7. 如何判断训练是否有效。

训练项目不要只说“我跑通了训练”。跑通只是起点，能解释训练过程中的异常和取舍才是重点。

### 3.12.2 Post-training 项目

Post-training 项目要突出数据、偏好、评估和模型行为。

重点问题包括：

1. SFT 数据从哪里来，质量如何控制。
2. 偏好数据如何构造，chosen 和 rejected 的标准是什么。
3. 为什么用 DPO、RLHF、RLAIF 或其他方法。
4. 如何平衡 helpfulness、harmlessness 和 honesty。
5. 如何评估幻觉、拒答、越狱鲁棒性和误拒。
6. 是否出现 reward hacking 或过度拒答。

讲 post-training 时，不能只讲训练 loss。模型行为变化和评估可信度更重要。

### 3.12.3 RAG 项目

RAG 项目要突出检索、上下文组织和答案可信度。

重点问题包括：

1. 文档如何切分和清洗。
2. embedding 模型和 reranker 如何选择。
3. 召回率、引用准确率和最终答案质量如何评估。
4. 如何处理检索不到、检索冲突和过期知识。
5. 如何控制上下文长度和延迟。
6. 是否做了 query rewrite、hybrid search 或多轮检索。

RAG 项目的常见误区是只讲向量数据库。真正难点通常在数据、评估和失败案例处理。

### 3.12.4 推理部署项目

推理部署项目要突出瓶颈定位、性能指标和质量 trade-off。

重点问题包括：

1. 请求模式是什么，输入输出长度分布如何。
2. 延迟和吞吐瓶颈在哪里。
3. batching、KV cache、量化、并行和 speculative decoding 如何选择。
4. 优化后质量是否变化。
5. 峰值流量、扩缩容和故障恢复如何处理。
6. 单位 token 成本如何下降。

推理项目中，面试官通常会追问具体数字和瓶颈分析。要准备好 p50、p95、QPS、显存和成本口径。

### 3.12.5 评估项目

评估项目要突出指标设计、可信度和闭环。

重点问题包括：

1. 评估集如何构建。
2. 自动评测和人评如何结合。
3. LLM-as-judge 如何校准。
4. 如何避免数据污染和 prompt 过拟合。
5. 如何做错误分类和回归测试。
6. 评估结果如何影响训练或产品决策。

评估项目看似不如训练项目“硬”，但在大模型团队中非常关键。能把评估讲深，是强信号。

## 3.13 面试官常见追问

准备项目时，可以提前模拟这些问题：

1. 这个项目里你的个人贡献是什么？
2. 你们的 baseline 是什么？
3. 为什么不用另一个方案？
4. 最大的失败是什么？
5. 指标是否可能被污染？
6. 线上效果和离线效果一致吗？
7. 如果数据规模扩大 10 倍，方案还成立吗？
8. 如果模型规模换成 70B，会有什么变化？
9. 如果算力减少一半，你会怎么取舍？
10. 如果安全要求更高，你会怎么改？
11. 这个方案的边界在哪里？
12. 你从这个项目中学到的最重要原则是什么？

这些问题的目的不是为难你，而是判断你是否真正理解项目。如果只能背准备稿，一追问就断，说明项目包装还不够。

## 3.14 如何把普通项目讲出深度

并不是每个人都有顶级实验室或万卡训练经历。普通项目也可以讲出深度，关键是挖掘真实问题。

比如一个简单的 LoRA 微调项目，可以从这些角度深化：

1. 数据质量：数据从哪里来，如何去重、过滤、标注。
2. 训练稳定性：学习率、rank、batch size 如何影响结果。
3. 泛化能力：是否只对验证集有效，是否有跨域测试。
4. 评估可信度：自动指标和人工评价是否一致。
5. 失败案例：模型在哪些问题上仍然失败。
6. 成本约束：为什么选择 LoRA 而不是全参微调。
7. 安全边界：是否出现不该回答的问题。

一个项目是否有深度，不完全取决于资源规模，而取决于你是否进行了认真归因和验证。

## 3.15 如何避免过度包装

项目包装的底线是真实。可以优化表达，但不能虚构事实。

常见过度包装包括：

1. 把参与写成主导。
2. 把 demo 写成线上系统。
3. 把阅读论文写成复现论文。
4. 把一次实验写成系统研究。
5. 把团队指标全部写成个人贡献。
6. 把未公开、不能披露的信息直接写出。
7. 把不确定结论写成确定结论。

过度包装的风险很高。面试官只要追问一个细节，比如数据规模、baseline、错误案例、训练配置、线上监控，就能看出你是否真实参与。

正确做法是诚实说明贡献边界：

```text
这个项目是团队合作，我主要负责数据清洗和评估部分，训练框架由同事负责。但我参与了训练结果分析，所以可以讲清数据变化对模型行为的影响。
```

这种回答不会减分，反而更可信。

## 3.17 本章小结

项目包装不是把项目吹大，而是把真实经历讲清楚。

好的项目讲述要从问题出发，而不是从工具出发。你需要讲清背景、问题、难点、方法、实验、结果、失败和反思。尤其要重视失败案例、评估可信度和 trade-off，因为这些地方最能体现真实经验和技术判断。

面试官真正想知道的不是你是否碰过某个热门技术，而是你能否在复杂约束下做出合理决策，并用实验验证自己的判断。
