# 第十二章：Mock Interview 与复盘

Mock Interview 的目标不是让你提前背答案，而是把知识、项目、系统设计、研究讨论和行为表达放进真实压力环境里检验。很多人平时看书能看懂，写笔记也完整，但一到面试就讲散、讲慢、讲不到重点、被追问就慌。这说明问题不在知识输入，而在面试输出没有训练。

真正有效的模拟面试必须包含三件事：真实流程、明确评分、严格复盘。只做题不复盘，等于重复错误；只复盘不再练，等于知道问题但没有形成新习惯。

本章重点：模拟面试流程、45 分钟脚本、评分表、错题归因、表达改进、项目深挖、系统设计训练、英文 mock、复盘记录和下一轮准备。

## 12.1 Mock Interview 到底训练什么

Mock Interview 训练的不是单点知识，而是五种综合能力：

1. 快速定位问题：听到题目后知道它属于基础、训练、部署、对齐、系统、研究还是行为题。
2. 结构化表达：能先给结论，再按层次展开。
3. 抗追问能力：面试官连续追问细节时不慌、不硬编。
4. 时间控制：能在 2 分钟、5 分钟、10 分钟版本之间切换。
5. 复盘迭代：能把一次失败面试转化为下一轮具体改进。

大模型算法岗的 mock 不能只问八股题。至少要覆盖：

1. 自我介绍。
2. 项目深挖。
3. ML/LLM 基础。
4. Training / Deployment / Alignment 专项。
5. System design。
6. Research discussion。
7. Behavioral。
8. 英文表达。

## 12.2 一次完整 Mock 的标准流程

建议用 45 分钟作为标准 mock，因为它接近真实单轮面试长度。

标准流程：

| 时间 | 模块 | 内容 |
| --- | --- | --- |
| 0-3 分钟 | 开场 | 自我介绍和岗位匹配 |
| 3-12 分钟 | 项目深挖 | 一个核心项目，追问数据、方法、实验、失败 |
| 12-22 分钟 | 技术基础 | Transformer、训练、对齐、部署等基础题 |
| 22-34 分钟 | 系统或研究 | 系统设计题或论文讨论题 |
| 34-40 分钟 | Behavioral | 失败、冲突、主动性、安全判断 |
| 40-43 分钟 | 反向提问 | 候选人提问 |
| 43-45 分钟 | 即时反馈 | 面试官给 1-2 条关键反馈 |

如果时间只有 30 分钟，可以压缩成：自我介绍 2 分钟，项目 8 分钟，技术 10 分钟，系统或行为 7 分钟，反馈 3 分钟。

如果是专项 mock，例如部署面试，可以把 30 分钟全部用于一个方向：5 分钟基础概念，15 分钟系统设计，5 分钟线上排障，5 分钟复盘。

## 12.3 Mock 前怎么准备

Mock 前不要临时乱翻资料。每次 mock 前准备四件事。

第一，确定目标。

本次 mock 只重点训练 1-2 个目标。例如：

1. 项目表达是否清楚。
2. Training 追问能否顶住。
3. System design 是否有主线。
4. Behavioral 是否具体。
5. 英文回答是否流畅。

第二，准备材料。

至少准备：

1. 2 分钟自我介绍。
2. 1 个 3 分钟项目讲稿。
3. 5 个项目可追问细节。
4. 10 道本轮方向题。
5. 1 个系统设计题或研究题。

第三，设置约束。

真实面试不会让你无限想。可以设置：

1. 基础题思考不超过 10 秒。
2. 开放题先用 20 秒组织结构。
3. 项目介绍不超过 3 分钟。
4. 不确定时必须明确说边界。

第四，录音或记录。

如果条件允许，录音很有价值。很多表达问题只有听回放才会发现，比如语速太快、口头禅太多、结构混乱、结论太晚出现。

## 12.4 面试官脚本：如何给自己做 Mock

如果没有真人面试官，也可以自问自答，但要模拟追问。

基础脚本：

```text
1. 请用 3 分钟介绍你自己，并说明为什么适合大模型算法岗。
2. 讲一个你最熟悉的大模型项目。
3. 这个项目最难的问题是什么？
4. 你怎么评估结果可信？
5. 有没有失败实验？你怎么解释？
6. 如果让你重新做，你会改什么？
7. Transformer 里 attention 解决什么问题？
8. SFT、RLHF、DPO 有什么区别？
9. KV Cache 为什么能加速推理？
10. 设计一个企业知识库 RAG 系统。
11. 讲一次你和别人有技术分歧的经历。
12. 你有什么问题想问我？
```

追问原则：

1. 每个项目回答至少追问 3 层。
2. 每个技术题至少问一个边界或失败模式。
3. 每个系统设计题至少问一次成本、一次安全、一次故障。
4. 每个 behavioral 题都追问“你具体做了什么”。

示例追问链：

```text
你说 RAG 幻觉下降了。
如何定义幻觉？
评估样本怎么来？
人评 rubric 是什么？
如果检索到了证据但模型仍然答错，你怎么归因？
这个方案上线后延迟增加了吗？
```

这种追问比单题问答更接近真实面试。

## 12.5 评分表：不要只凭感觉判断好坏

每次 mock 后都要打分。建议用 1 到 5 分。

| 维度 | 1 分 | 3 分 | 5 分 |
| --- | --- | --- | --- |
| 结构 | 没有主线 | 有基本结构 | 先结论后展开，层次清楚 |
| 准确性 | 概念错误多 | 主要正确但有模糊点 | 准确且能讲边界 |
| 深度 | 只会定义 | 能讲机制 | 能讲 trade-off、失败模式和实验 |
| 项目可信度 | 像旁观者 | 能讲自己贡献 | 能讲决策、失败、指标和反思 |
| 系统意识 | 只讲模型 | 能讲模块 | 能讲成本、安全、监控、降级 |
| 研究能力 | 背结论 | 能讲论文 | 能判断贡献、设计复现和 ablation |
| 表达 | 断裂混乱 | 基本可懂 | 简洁、有重点、可追问 |
| 诚实边界 | 硬编 | 偶尔承认不确定 | 主动区分事实、假设和待验证 |

一次 mock 的目标不是每项都 5 分，而是找到最低的 1-2 项优先改。

## 12.6 错题归因：不要只写“不会”

复盘时最常见的错误是只写：这题不会。

这没有用。必须做归因。

错题可以分为八类：

1. 知识缺口：确实不知道概念。
2. 机制不清：知道名词，但讲不清原理。
3. 边界缺失：只讲优点，不讲缺点和适用场景。
4. 表达混乱：知道内容，但组织不好。
5. 项目不熟：讲不清自己的数据、实验、指标或贡献。
6. 系统意识弱：忽略成本、安全、监控、失败场景。
7. 研究判断弱：不会评价论文、设计 ablation 或解释负结果。
8. 心态问题：被追问后紧张、硬编、急于辩解。

复盘模板：

```text
原题：
我的原回答：
问题类型：知识缺口 / 机制不清 / 边界缺失 / 表达混乱 / 项目不熟 / 系统意识弱 / 研究判断弱 / 心态问题
正确回答要点：
背后关联概念：
下一次 2 分钟版本答案：
需要补的资料或练习：
```

## 12.7 表达问题怎么改

很多人 mock 后发现不是不会，而是表达不稳定。

常见表达问题：

1. 开头太绕，迟迟不给结论。
2. 只讲细节，没有框架。
3. 讲太多背景，时间不够。
4. 每句话都加“可能、就是、然后”。
5. 被打断后回不到主线。
6. 不知道什么时候停。
7. 不确定时硬编。

改进方法：

1. 每题先用一句话给结论。
2. 用编号表达，例如“我会从三点回答”。
3. 开放题先讲框架，再讲细节。
4. 每个回答准备 30 秒、2 分钟、5 分钟版本。
5. 被打断后用“我先回到核心问题”收束。
6. 不确定时说“这个细节我不完全确定，但我会这样分析”。

示例：

差回答：

```text
KV Cache 就是那个推理的时候会缓存一些东西，然后可以避免重复计算，所以会快很多，但是也会占显存。
```

更好回答：

```text
KV Cache 的核心作用是加速自回归 decode。它缓存每层历史 token 的 Key 和 Value，使生成新 token 时不需要重新计算完整上下文。收益是降低重复计算，代价是显存随层数、上下文长度、并发和 KV head 数增长，因此长上下文高并发下 KV Cache 会成为主要瓶颈。
```

## 12.8 项目深挖怎么练

项目深挖是 mock 中最重要的模块。

每个项目至少准备十类问题：

1. 背景：为什么做这个项目？原问题是什么？
2. 目标：优化什么指标？为什么这个指标可信？
3. 数据：数据来源、清洗、标注、隐私和偏差。
4. 方法：候选方案有哪些？为什么选这个？
5. 训练：模型、超参、loss、训练稳定性。
6. 评估：baseline、指标、人评、统计显著性。
7. 失败：哪些实验没成功？为什么？
8. 系统：延迟、成本、监控、上线和回滚。
9. 安全：误拒、幻觉、隐私、权限、滥用。
10. 反思：如果重做会改什么？

项目复盘表：

| 问题 | 我的答案是否清楚 | 证据 | 需要补充 |
| --- | --- | --- | --- |
| 我解决的问题是什么 |  |  |  |
| 我的个人贡献是什么 |  |  |  |
| baseline 是什么 |  |  |  |
| 指标怎么定义 |  |  |  |
| 失败实验是什么 |  |  |  |
| 最大 trade-off 是什么 |  |  |  |
| 线上或系统约束是什么 |  |  |  |
| 安全和隐私怎么处理 |  |  |  |

## 12.9 系统设计 Mock 怎么练

系统设计 mock 的关键是流程稳定。

每次系统设计都按六步：

1. 澄清场景。
2. 列需求。
3. 画主链路。
4. 深入核心模块。
5. 讲 trade-off 和故障。
6. 讲演进和指标。

常用题库：

1. 设计 ChatGPT 服务。
2. 设计企业 RAG 系统。
3. 设计大模型推理平台。
4. 设计训练平台。
5. 设计 Agent 平台。
6. 设计多模态助手。
7. 设计实时语音助手。
8. 设计评估平台。
9. 设计安全审核系统。
10. 设计模型路由与缓存系统。

每道题必须主动讲四件事：

1. 指标：如何判断系统成功。
2. 成本：如何控制 token、GPU 和人工成本。
3. 安全：权限、隐私、滥用和审核。
4. 失败：故障、降级、回滚和监控。

## 12.10 Research Mock 怎么练

研究讨论 mock 要训练“判断力”，不是论文背诵。

常用题：

1. 最近哪篇论文重要？为什么？
2. 如何复现 DPO？
3. 如何判断一个新 attention 方法是否值得？
4. 如何设计 RAG 方法的 ablation？
5. 一个论文结果和你的直觉相反，你怎么办？
6. 如何解释负结果？
7. 如何提出一个研究改进点？
8. 如何把论文变成项目？

研究题回答必须包含：

1. 问题背景。
2. 方法机制。
3. 实验设计。
4. 贡献边界。
5. 复现路径。
6. 下一步。

如果没读过某篇论文，不要硬编。可以说：

```text
这篇我没有完整读过，但我会从问题是否重要、baseline 是否公平、实验是否支持核心假设、成本和边界是否清楚这几个角度判断。
```

## 12.11 Behavioral Mock 怎么练

Behavioral mock 要准备故事库。

至少练 8 个故事：

1. 成功项目。
2. 失败项目。
3. 技术判断错误。
4. 团队冲突。
5. 主动性。
6. 快速学习。
7. 压力管理。
8. 安全伦理判断。

每个故事用 STAR-L 讲：

1. Situation：背景。
2. Task：任务。
3. Action：行动。
4. Result：结果。
5. Learning：学习。

Mock 时要特别检查：

1. 是否讲清自己的贡献。
2. 是否推卸责任。
3. 是否有具体行动。
4. 是否有结果或证据。
5. 是否有反思。

## 12.12 英文 Mock 怎么练

英文 mock 不要求词汇华丽，要求结构清楚、稳定、不硬翻译中文长句。

训练顺序：

1. 先准备英文自我介绍。
2. 准备一个英文项目 2 分钟版本。
3. 准备一个英文失败经历。
4. 准备 5 个技术解释模板。
5. 练习不确定时的英文表达。

不确定时可以说：

```text
I am not fully certain about the exact detail, but I would reason about it from the objective, data, optimization, and system constraints.
```

项目开场可以说：

```text
One project I know well is an enterprise RAG system. The main goal was to reduce hallucination and improve answer traceability. My responsibility was to work on data construction, evaluation, and model improvement. The most important lesson was that many errors came from retrieval and chunking, not just from the language model itself.
```

英文复盘时重点听：

1. 是否句子太长。
2. 是否中英文混杂太多。
3. 是否术语准确。
4. 是否能自然表达 trade-off。
5. 是否能承认不确定。

## 12.13 Mock 频率怎么安排

如果还有 3 个月准备：

1. 第 1 个月：每周 1 次，重点项目和基础。
2. 第 2 个月：每周 2 次，加入系统设计和专项追问。
3. 第 3 个月：每周 2-3 次，按真实面试强度模拟。

如果还有 1 个月准备：

1. 每周至少 2 次完整 mock。
2. 每天 20 分钟口头回答训练。
3. 每两天复盘一个项目或系统题。
4. 每周至少一次英文 mock。

如果只有 1 周：

1. 不要再大面积学新知识。
2. 集中打磨自我介绍、项目、系统设计模板和高频错题。
3. 每天做一次 30 分钟 mock。
4. 每次只改 1-2 个关键问题。

## 12.14 如何判断自己准备好了

可以用下面标准自测：

1. 能稳定完成 2 分钟自我介绍。
2. 能把核心项目讲出 1、3、10 分钟版本。
3. 能回答 50 道高频基础题，且能讲边界。
4. 能完成 5 类系统设计题的主链路。
5. 能讲 3 篇论文的背景、方法、实验和边界。
6. 能讲 5 个 behavioral 故事。
7. 被追问不知道的问题时不会硬编。
8. 每次 mock 后能形成具体改进项。

如果你仍然出现以下问题，说明还没准备好：

1. 项目讲不清个人贡献。
2. 系统设计只会堆组件。
3. 论文只会背结论。
4. 被追问就慌或硬编。
5. 复盘记录总是“继续学习”，没有具体行动。

复盘必须具体到行动。例如：

差复盘：

```text

```

```text
设计 RAG 时漏讲权限和引用。明天补一版 RAG 系统 5 分钟答案，必须包含 permission filter、citation accuracy、faithfulness 和文档版本。
```

## 12.16 一周 Mock 闭环示例

一个高效的一周安排：

| 日期 | 训练内容 | 目标 |
| --- | --- | --- |
| 周一 | 项目 mock | 打磨 3 分钟项目讲述 |
| 周二 | 基础题口头训练 | 每题 2 分钟讲清机制和边界 |
| 周三 | 系统设计 mock | 练 RAG 或推理平台主链路 |
| 周四 | Research mock | 讲一篇论文和一个 ablation |
| 周五 | Behavioral mock | 练失败、冲突、安全判断 |
| 周六 | 综合 45 分钟 mock | 模拟真实面试 |
| 周日 | 深度复盘 | 整理错题、重录关键答案 |

周日不要只休息，也不要继续刷新内容。最有价值的是把一周暴露的问题转成下一周训练计划。

## 12.17 最后一页速记

Mock Interview 记住五句话：

1. 不 mock，不知道自己真实表达水平。
2. 不录音，很难发现表达习惯问题。
3. 不复盘，mock 只是重复错误。
4. 不重练，复盘不会自动变成能力。
5. 每次只改 1-2 个关键问题，进步最快。

面试前最后检查：

1. 自我介绍是否流畅。
2. 项目是否能被连续追问 5 层。
3. 基础题是否能先结论后机制。
4. 系统设计是否能主动讲指标、成本、安全和失败。
5. 研究题是否能讲背景、实验、边界和下一步。
6. Behavioral 是否有具体故事。
7. 英文回答是否结构清楚。
8. 不确定时是否能诚实表达。

这一章的核心是：面试能力不是看出来的，是练出来、录出来、复盘出来、重练出来的。
