# 第二章：简历策略

很多候选人写大模型算法岗简历时，会犯两个相反的错误。

第一种错误是把简历写成知识点清单：Transformer、RLHF、LoRA、RAG、MoE、FlashAttention、DeepSpeed、vLLM、DPO、Agent、multimodal 全部堆上去，看起来很热闹，但面试官看不出你到底做成了什么。第二种错误是把简历写成流水账：参与某项目，负责某模块，完成某功能，优化某指标，但缺少背景、难点、方法、结果和你的个人贡献。

大模型算法岗简历的核心不是“我知道什么”，而是“我在什么约束下，用什么判断和方法，解决了什么问题，产生了什么可验证影响”。

本章重点：如何写项目、论文、工程贡献、指标结果、影响力和技术关键词。

## 2.1 简历的本质：降低面试官判断成本

简历不是自传，也不是作品集目录。它的作用是让筛简历的人在很短时间内判断三件事：

1. 你是否和岗位方向匹配。
2. 你是否有真实、可追问的技术深度。
3. 你是否有足够强的产出信号，值得进入面试。

大模型算法岗尤其如此。因为岗位竞争激烈，很多候选人的关键词高度相似。如果你的简历只是写“熟悉 Transformer、熟悉大模型训练、熟悉 RAG、熟悉 LoRA 微调”，它并不能有效区分你和其他人。

更好的简历应该让面试官快速看到：

1. 你做过哪类问题。
2. 你的工作处在系统的哪个环节。
3. 你面对的主要约束是什么。
4. 你做了哪些关键决策。
5. 结果如何验证。
6. 如果追问，你能不能讲清细节。

所以，简历不是越长越好，也不是关键词越多越好。好的简历是高信息密度的证据链。

## 2.2 大模型算法岗简历应该突出什么

第 1 章讲过，大模型算法岗不是单一岗位。简历策略也必须跟岗位画像匹配。

如果目标是基础模型研究岗，简历应该突出研究问题、方法创新、实验设计、论文产出和对失败结果的理解。如果目标是训练工程岗，简历应该突出大规模训练经验、稳定性排查、吞吐优化、显存优化、分布式并行和监控体系。如果目标是对齐岗，简历应该突出 SFT、RLHF、DPO、偏好数据、安全评估、红队、拒答策略和模型行为分析。如果目标是推理部署岗，简历应该突出延迟、吞吐、显存、KV cache、batching、quantization、服务可靠性和成本。如果目标是 Agent 或应用算法岗，简历应该突出任务成功率、工具调用、检索、规划、评估闭环和产品影响。

可以用下面这张表快速对齐：

| 目标方向 | 简历重点 |
| --- | --- |
| 基础模型研究 | 问题定义、方法动机、实验设计、论文或复现深度 |
| 预训练与训练工程 | 数据、并行、优化、稳定性、吞吐、训练规模 |
| Post-training / 对齐 | SFT、偏好数据、DPO/RLHF、安全评估、模型行为 |
| 评估与模型行为 | benchmark、自动评测、人评、误差分析、评估可信度 |
| 多模态 | 跨模态数据、编码器、对齐、生成质量、任务指标 |
| 推理部署 | 延迟、吞吐、成本、显存、服务稳定性、线上流量 |
| Agent / RAG | 检索质量、工具调用、任务成功率、业务闭环 |
| 数据工程 | 数据采集、清洗、去重、质量评分、合规、安全数据 |

简历最忌讳“所有方向都想沾一点”。如果每个方向都写一点，但没有任何方向能讲深，反而会让面试官觉得你没有主线。

## 2.3 项目经历怎么写

项目经历是算法岗简历的核心。一个项目条目最好包含五个元素：背景、目标、方法、结果、个人贡献。

可以使用下面的结构：

1. 背景：为什么要做这个项目？原始问题是什么？
2. 目标：优化什么指标？解决什么瓶颈？满足什么约束？
3. 方法：你用了什么技术路线？为什么这么选？
4. 结果：效果提升多少？成本降低多少？稳定性改善多少？
5. 贡献：哪些是你独立负责，哪些是团队协作？

差的写法：

```text
参与大模型微调项目，使用 LoRA 对模型进行训练，提升模型效果。
```

这个写法的问题是信息太少。面试官不知道你微调什么模型、什么任务、什么数据、什么指标、提升多少、怎么评估、你负责哪部分。

更好的写法：

```text
负责企业问答场景的 7B 模型指令微调，构建 8 万条领域问答与拒答数据，基于 LoRA 进行 SFT；通过数据去重、难例补充和格式约束，将人工评测可用率从 71% 提升到 84%，同时将幻觉类错误率从 18% 降至 11%。
```

这个写法更好，因为它回答了几个关键问题：场景是什么，模型规模是多少，数据规模是多少，方法是什么，指标怎么变化，贡献在哪里。

再看一个训练工程方向的例子。

差的写法：

```text
参与大模型训练平台建设，使用 DeepSpeed 提升训练效率。
```

更好的写法：

```text
参与 13B 模型预训练稳定性优化，负责训练监控与异常定位模块；基于 loss、grad norm、数据 shard、吞吐和显存曲线建立告警规则，定位两类由坏样本和混合精度溢出导致的 loss spike，使长周期训练中断次数下降约 40%。
```

这里的重点不是“用了 DeepSpeed”，而是“你解决了什么训练问题”。

## 2.4 项目条目的黄金公式

对于大多数项目条目，可以套用一个简洁公式：

```text
在 [场景/约束] 下，为解决 [问题]，我负责 [关键工作]，采用 [方法/技术]，最终使 [指标] 从 [基线] 提升/降低到 [结果]。
```

例如：

```text
在客服多轮问答场景下，为解决模型答非所问和拒答不稳定问题，我负责构建偏好数据与 DPO 训练流程，结合人评和自动评测筛选 hard negative，使多轮任务完成率从 62% 提升到 76%，安全误拒率下降 9 个百分点。
```

这个公式的好处是，它强迫你把项目写成一个完整闭环，而不是堆技术名词。

但也要注意，不要机械套模板。真实项目里，有些结果不能公开，有些指标不能量化，有些项目偏研究探索，不一定有线上指标。这时可以改写为：

1. 用相对变化替代绝对数字，例如“降低约 30%”。
2. 用内部排名替代具体数值，例如“在同批方案中排名第一”。
3. 用实验结论替代线上指标，例如“验证了数据质量比参数量增加更影响该任务”。
4. 用可复现结果替代业务指标，例如“在三个公开 benchmark 上稳定优于 baseline”。

不要编造数字。虚假的指标很容易在追问中暴露，而且会直接伤害可信度。

## 2.5 如何写论文经历

如果你有论文经历，不要只写论文标题和会议。面试官更关心你在论文中真正贡献了什么。

论文经历可以从六个角度写：

1. 问题：论文解决什么问题？为什么这个问题重要？
2. 方法：核心方法是什么？和已有方法相比差异在哪里？
3. 实验：你设计了哪些关键实验？验证了哪些假设？
4. 结果：在哪些指标上提升？有没有失败或负结果？
5. 贡献：你负责方法、实验、数据、代码、写作还是分析？
6. 影响：是否开源、被引用、被复现、被产品采用或启发后续工作？

差的写法：

```text
发表大模型相关论文一篇，提出一种新的微调方法，在多个数据集上取得 SOTA。
```

这个写法过于抽象，而且“SOTA”如果没有具体任务和对比对象，价值有限。

更好的写法：

```text
一作完成参数高效微调论文，针对低资源领域适配中 LoRA 容易过拟合的问题，引入基于验证集不确定性的 rank 分配策略；负责方法设计、训练代码和消融实验，在 5 个领域分类任务上平均提升 2.3 个百分点，并通过消融验证 rank 分配对小样本场景更敏感。
```

这个版本更清楚：问题、方法、贡献、结果、实验结论都有。

如果论文还没发表，也可以写，但要谨慎表述：

```text
完成一项关于长上下文检索增强的研究项目，正在投稿；负责构建长文档问答评测集和错误分析，发现 chunk 粒度与 reranker 召回存在明显 trade-off，并提出基于问题类型的动态切分策略。
```

不要把“准备投稿”“计划投稿”包装成已经发表。面试官通常可以接受未发表研究，但不能接受夸大。

## 2.6 如何写工程贡献

很多候选人低估工程贡献的价值，以为算法岗简历只能写模型和论文。实际上，高水平大模型团队非常重视工程能力，因为大模型研发离不开大规模实验、训练系统、数据管线、评估平台和推理服务。

工程贡献不是“写了很多代码”，而是让系统更可靠、更快、更便宜、更可观测、更可复现。

常见高价值工程贡献包括：

1. 训练吞吐提升。
2. 显存占用降低。
3. 训练中断率下降。
4. checkpoint 恢复更可靠。
5. 数据处理速度提升。
6. 评估自动化覆盖更多任务。
7. 推理延迟降低。
8. 单位请求成本下降。
9. 服务可用性提升。
10. 实验复现和配置管理改善。

差的写法：

```text
搭建模型评估平台，支持自动化评测。
```

更好的写法：

```text
负责搭建大模型自动评估流水线，覆盖知识问答、代码生成、数学推理和安全拒答 4 类任务；统一 prompt 模板、采样参数和结果存储，使每次模型迭代评估从人工 2 天缩短到自动化 3 小时，并支持错误样本回流到训练数据。
```

工程贡献要尽量写出“前后对比”。没有前后对比，就很难体现价值。

## 2.7 指标结果怎么写

指标是简历的证据，但指标不是越多越好。好的指标应该和岗位目标相关，并且能经得起追问。

常见指标类型包括：

| 类型 | 示例 |
| --- | --- |
| 模型效果 | accuracy、F1、BLEU、pass@k、win rate、人评可用率 |
| 训练效率 | tokens/s、MFU、GPU 利用率、训练中断次数、收敛速度 |
| 推理效率 | latency、throughput、QPS、显存、单位 token 成本 |
| 数据质量 | 去重率、低质样本过滤率、PII 命中率、人工抽检通过率 |
| 产品影响 | 任务完成率、用户采纳率、留存、人工客服转接率 |
| 安全表现 | 有害输出率、越狱成功率、误拒率、合规命中率 |

写指标时要注意四件事。

第一，要有基线。只写“准确率 86%”不如写“从 78% 提升到 86%”。没有基线，面试官不知道这个结果强不强。

第二，要说明评估方式。自动评测、人评、线上 A/B、离线 benchmark 的可信度不同。比如“人评可用率提升 10 个百分点”必须能解释人评样本怎么抽、标注标准是什么、是否有一致性校验。

第三，要避免堆小数点。简历不是实验日志。通常写整数、百分比或近似值即可。过度精确反而显得不自然。

第四，要能解释代价。一个指标提升可能带来另一个指标下降。比如更强的安全拒答可能提高误拒率，更大的 reranker 可能增加延迟，更长上下文可能提高成本。能讲 trade-off 的候选人更可信。

## 2.8 技术关键词怎么放

关键词很重要，因为简历筛选系统和招聘人员会搜索关键词。但关键词应该服务于经历，而不是替代经历。

可以把关键词分为三类。

第一类是基础能力关键词：Transformer、attention、optimization、distributed training、evaluation、alignment、retrieval、inference。

第二类是具体方法关键词：LoRA、QLoRA、SFT、DPO、RLHF、RAG、reranker、MoE、FlashAttention、KV cache、quantization、speculative decoding。

第三类是工具和框架关键词：PyTorch、JAX、DeepSpeed、Megatron-LM、FSDP、vLLM、TensorRT-LLM、Ray、Kubernetes、Hugging Face。

关键词的正确写法是嵌入项目上下文：

```text
基于 PyTorch 和 FSDP 完成 7B 模型 SFT 训练，使用 LoRA 降低显存占用，并通过梯度累积和 sequence packing 提升训练吞吐。
```

关键词的错误写法是堆列表：

```text
熟悉 PyTorch、FSDP、DeepSpeed、LoRA、DPO、RLHF、RAG、MoE、vLLM、FlashAttention、Agent。
```

如果一定要写技能栏，可以短一些，并且按方向组织：

```text
技能：PyTorch / distributed training / SFT / DPO / RAG / evaluation / vLLM
```

不要把只是看过博客、跑过 demo 的技术写成“熟悉”。简历上的每个关键词都可能成为面试入口。

## 2.9 如何写影响力

影响力不等于“项目很大”。影响力是你的工作改变了什么。

影响力可以来自多个层面：

1. 研究影响：论文、开源、引用、被后续工作采用。
2. 工程影响：提高效率、降低成本、减少故障、提升复现性。
3. 产品影响：提升用户体验、转化率、任务完成率或满意度。
4. 团队影响：沉淀工具、规范、文档、平台，让其他人复用。
5. 数据影响：建立数据闭环，持续提升模型质量。

例如：

```text
沉淀数据质量评估脚本和人工抽检规范，被团队后续 3 个微调项目复用，减少重复清洗工作并提高数据版本可追溯性。
```

这个影响力不一定惊天动地，但真实、有价值、可追问。

如果是学生或刚入门候选人，没有线上产品影响，也可以写学习和研究影响：

```text
复现 DPO 训练流程并完成 6 组消融实验，整理训练配置、失败案例和调参经验，形成团队内部分享材料。
```

关键是不要把“参与过”写成“主导过”，也不要把“个人学习项目”包装成“生产系统”。诚实不等于弱，清晰表达真实贡献反而更容易获得信任。

## 2.10 不同背景候选人的简历策略

### 2.10.1 应届生

应届生通常缺少工业级项目，但可以用论文、课程项目、开源复现、竞赛和实习证明潜力。

应届生简历重点是：

1. 数学和机器学习基础是否扎实。
2. 是否能独立读论文、复现和做实验。
3. 是否有清晰的问题意识。
4. 是否能写可靠代码。
5. 是否有持续学习能力。

不要只写“阅读多篇大模型论文”。更好的写法是：

```text
系统复现 LoRA、QLoRA 和 DPO 在 7B 模型上的训练流程，比较不同 rank、学习率和数据规模对效果的影响，整理 20 余个失败实验并总结显存、收敛和过拟合问题。
```

这比空泛地写“熟悉大模型微调”强很多。

### 2.10.2 有算法经验但非大模型背景

这类候选人的优势是机器学习、实验方法和工程经验，劣势是大模型关键词和最新技术栈可能不够。

简历策略是把过去经验迁移到大模型语境：

1. 推荐系统经验可以对应召回、排序、评估、在线实验和数据闭环。
2. NLP 经验可以对应文本建模、预训练、信息抽取、问答和评估。
3. CV 经验可以对应多模态、表示学习、数据增强和生成模型。
4. 系统经验可以对应训练平台、推理服务和分布式优化。

例如，推荐系统背景可以这样写：

```text
负责推荐排序模型迭代，建立离线 AUC、线上 CTR 和用户分层分析闭环；该经验可迁移到大模型评估中的自动指标、人评校准和线上反馈闭环。
```

简历中不一定要直接写“可迁移到”，但你需要在项目表达里体现这种迁移能力。

### 2.10.3 工程背景转算法岗

工程背景候选人不要掩盖工程优势。大模型团队非常需要能把研究想法变成可靠系统的人。

你可以突出：

1. 分布式系统经验。
2. 高并发服务经验。
3. GPU 性能优化经验。
4. 数据平台经验。
5. 可观测性和稳定性经验。
6. 实验平台和自动化经验。

同时补上算法信号：

1. 至少一个完整的大模型训练或微调项目。
2. 至少一个评估和误差分析项目。
3. 能讲清 Transformer、SFT、DPO、RAG、推理优化等核心机制。

工程转算法的简历不能只证明“我会写代码”，还要证明“我理解模型行为和算法目标”。

### 2.10.4 已有大模型工业经验

如果你已经有大模型工业经验，简历要避免写得过于保守或过于笼统。

重点应该放在：

1. 规模：模型规模、数据规模、训练规模、线上流量。
2. 复杂度：稳定性、成本、安全、评估、协作约束。
3. 决策：你做过哪些关键技术选择。
4. 影响：你的工作如何影响模型能力、成本或产品。
5. 深度：你能否解释失败案例和 trade-off。

注意保密边界。不能公开的模型规模、业务数据和内部指标，可以用区间、相对变化或抽象描述。

## 2.11 一页还是两页

大多数候选人一页足够。如果经历较多、论文较多、工业项目较复杂，可以两页。但无论一页还是两页，都要保证第一页最强。

建议顺序如下：

1. 姓名和联系方式。
2. 一句话定位。
3. 教育或工作经历。
4. 最匹配目标岗位的 2 到 4 个项目。
5. 论文、开源、竞赛或技能。

“一句话定位”可以很有用，但不要写空话。

差的定位：

```text
热爱人工智能，具有扎实算法基础和强烈学习能力。
```

更好的定位：

```text
大模型 post-training 方向候选人，重点经历包括 7B 模型 SFT/DPO、偏好数据构建和模型安全评估。
```

或者：

```text
偏训练与推理工程方向，熟悉 PyTorch/FSDP/vLLM，做过模型微调、自动评估和推理延迟优化。
```

一句话定位的作用是告诉面试官“请用什么视角读我的简历”。

## 2.12 常见简历问题

### 2.12.1 只写职责，不写结果

```text
负责模型训练、数据处理和效果评估。
```

这句话几乎没有信息。它没有说明规模、难点、方法和结果。

### 2.12.2 只写结果，不写方法

```text
将模型准确率提升 15%。
```

这句话看起来强，但如果没有说明怎么做到，面试官会怀疑可信度。

### 2.12.3 技术词过密

一条经历里塞太多技术词，会显得像搜索引擎优化，而不是实际项目。

### 2.12.4 贡献边界不清

“参与”“负责”“主导”含义不同。不要把团队成果全部写成个人成果。

### 2.12.5 项目太多但都很浅

与其写 8 个浅项目，不如写 3 个能讲深的项目。面试通常会围绕你最强的项目追问。

### 2.12.6 没有目标岗位意识

投训练岗却把主要篇幅写成前端产品项目，投对齐岗却完全不提偏好数据和安全评估，都会降低匹配度。

## 2.13 简历和面试是一体的

写简历时要提前想清楚：面试官会怎么追问？

每个项目至少准备回答以下问题：

1. 为什么做这个项目？
2. 最初 baseline 是什么？
3. 你尝试过哪些方案？
4. 哪些方案失败了？为什么？
5. 最关键的改进来自哪里？
6. 指标是否可靠？有没有数据泄漏或评估偏差？
7. 如果再做一次，你会怎么改？
8. 这个项目和目标岗位有什么关系？

如果一个项目写在简历上，但你无法回答这些问题，就应该删掉或弱化。简历不是越满越好，而是每一项都能成为高质量对话的入口。

## 2.14 简历自检清单

投递前可以逐条检查：

1. 第一眼能否看出我的目标方向？
2. 最强的 2 到 3 个项目是否放在最显眼位置？
3. 每个项目是否包含背景、方法、结果和个人贡献？
4. 是否有可验证指标或清晰实验结论？
5. 技术关键词是否嵌入真实经历，而不是孤立堆砌？
6. 是否删除了和目标岗位无关的低价值内容？
7. 是否避免夸大论文、开源、指标和个人贡献？
8. 是否能对每个关键词进行 5 分钟以上追问？
9. 是否遵守保密和合规边界？
10. 是否针对不同岗位版本做过调整？

## 2.15 本章小结

大模型算法岗简历的核心是证据链。

你不需要证明自己懂所有热门概念，而要证明自己能在具体问题中做出有效判断。项目经历要讲清背景、目标、方法、结果和贡献。论文经历要讲清问题、方法、实验和你的真实角色。工程贡献要讲清效率、稳定性、成本和可复用影响。指标要有基线、评估方式和 trade-off。关键词要服务于经历，而不是替代经历。

一份好的简历会让面试官产生明确判断：这个候选人有主线，有真实经历，有可追问的技术深度，并且和岗位匹配。

下一章我们会继续讨论项目包装。简历解决的是“如何让别人愿意面试你”，项目包装解决的是“进入面试后，如何把一个项目讲成有深度、有逻辑、有说服力的技术故事”。
