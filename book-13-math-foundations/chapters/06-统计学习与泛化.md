# 第六章：统计学习与泛化

统计学习关心一个核心问题：模型在训练数据上表现好，为什么能在没见过的数据上也表现好？这就是泛化。对大模型算法岗来说，泛化不是抽象理论，而是每天都会遇到的问题：SFT loss 下降但线上效果差，benchmark 分数高但真实用户不满意，DPO 后 win rate 提升但过度拒答，RAG 在固定评测集好但换文档就失效。

大模型的泛化问题尤其有趣。传统直觉认为参数越多越容易过拟合，但大模型参数巨大，却能在很多任务上泛化。这不是说过拟合消失了，而是过拟合、记忆、数据规模、模型规模、优化、正则化和评估污染之间的关系更复杂。

本章目标：讲清 bias-variance、overfitting、underfitting、regularization、泛化误差、大模型泛化悖论、benchmark contamination，以及如何在面试中成熟地回答“模型是在背诵还是泛化”。

## 6.1 泛化到底是什么

泛化指模型在未见数据上的表现。

训练集表现好，只说明模型拟合了训练样本；验证集、测试集、线上真实样本表现好，才说明模型有一定泛化能力。

在 LLM 中，泛化可以表现为：

1. 能回答训练集中没有逐字出现的问题。
2. 能把已学模式迁移到新表述。
3. 能组合多个知识点解决新任务。
4. 能根据 prompt 中的示例进行 in-context learning。
5. 能在新领域数据上保持合理表现。

但泛化不是无限的。模型可能在训练分布附近泛化很好，一旦遇到分布外输入、对抗样本、长尾场景或新工具环境，就会失败。

面试回答：

```text
泛化是模型在未见数据上的表现。对 LLM 来说，泛化不只是测试集 accuracy，而是能否在新表述、新组合、新任务和真实用户分布上稳定输出。模型既可能记忆训练样本，也可能学习可迁移模式，关键要通过分布外、扰动和组合测试区分。
```

### 6.1.1 关键公式与泛化诊断速查

真实风险，也叫期望风险：

$$
R(f)=\mathbb{E}_{(x,y)\sim P}[\ell(f(x),y)]
$$

其中 `P` 是真实数据分布，`ell` 是损失函数。真实风险是我们真正关心的目标，但完整的 `P` 通常不可见。

经验风险：

$$
\hat R_S(f)=\frac{1}{n}\sum_{i=1}^{n}\ell(f(x_i),y_i)
$$

其中 `S={(x_i,y_i)}_{i=1}^n` 是训练样本。训练时最常见的做法是经验风险最小化：

$$
\hat f=\arg\min_{f\in\mathcal{F}}\hat R_S(f)
$$

泛化 gap 用验证集近似：

$$
G_{\mathrm{val}}=\hat R_{\mathrm{val}}(f)-\hat R_{\mathrm{train}}(f)
$$

如果指标是 accuracy、win rate 这类越大越好的分数，更常看：

$$
G_{\mathrm{score}}=S_{\mathrm{train}}(f)-S_{\mathrm{val}}(f)
$$

过拟合的常见信号是训练损失低、验证损失高，或者训练分数高、验证分数低；欠拟合的常见信号是训练和验证都差。

平方损失下的 bias-variance 分解可以写成：

$$
\mathbb{E}_{D,\epsilon}[(\hat f_D(x)-y)^2]
=B^2(x)+V(x)+\sigma^2
$$

其中：

$$
B^2(x)=(\mathbb{E}_{D}[\hat f_D(x)]-g(x))^2
$$

$$
V(x)=\mathbb{E}_{D}[(\hat f_D(x)-\mathbb{E}_{D}[\hat f_D(x)])^2]
$$

`B^2(x)` 表示模型平均预测和真实函数 `g(x)` 的偏差，`V(x)` 表示训练数据变化带来的预测波动，`sigma^2` 是不可约噪声。

L2 正则化：

$$
L_{\mathrm{reg}}(\theta)=\hat R_S(f_\theta)+\lambda\|\theta\|_2^2
$$

`lambda` 越大，参数被约束得越强；但正则过强会欠拟合。

Early stopping 的验证集选择：

$$
t^*=\arg\min_{t\in\mathcal{T}}\hat R_{\mathrm{val}}(f_{\theta_t}),\quad \theta^*=\theta_{t^*}
$$

其中 `t` 是 checkpoint step。不要用最终测试集来选 `t^*`，否则测试集会被开发过程污染。

分布偏移可以用验证分布和目标分布之间的性能差近似：

$$
\Delta_{\mathrm{shift}}=\hat R_{\mathrm{target}}(f)-\hat R_{\mathrm{val}}(f)
$$

如果用越大越好的分数，也可以看：

$$
\Delta_{\mathrm{score}}=S_{\mathrm{val}}(f)-S_{\mathrm{target}}(f)
$$

训练-评估 exact overlap 率：

$$
C_{\mathrm{exact}}=\frac{1}{m}\sum_{j=1}^{m}\mathbf{1}[\mathrm{hash}(e_j)\in H_{\mathrm{train}}]
$$

其中 `e_j` 是评估样本，`H_train` 是训练样本规范化后 hash 集合。`C_exact` 大于 0 就要解释来源；如果是最终测试集，通常要阻断或重建评估集。

泛化审计门禁可以写成：

$$
G_{\mathrm{gen}}=G_{\mathrm{gap}}\land G_{\mathrm{shift}}\land G_{\mathrm{leak}}\land G_{\mathrm{slice}}
$$

它不是一个新训练目标，而是上线或面试复盘时的 checklist：训练-验证 gap、分布偏移、污染和关键切片都不能只看平均分。

## 6.2 训练误差、验证误差和泛化误差

训练误差是在训练数据上的误差。

验证误差是在 held-out 数据上的误差。

泛化误差关心模型在真实数据分布上的期望误差。

实际中，我们无法直接看到完整真实分布，只能用验证集、测试集、线上抽样和人工评估近似。

如果训练误差低、验证误差高，通常说明过拟合。

如果训练误差和验证误差都高，通常说明欠拟合或训练不足。

如果验证集表现好但线上差，可能说明：

1. 验证集不代表线上分布。
2. 验证集被污染。
3. 评估指标不匹配用户体验。
4. 模型过拟合 benchmark。
5. 线上系统链路有问题。

## 6.3 Bias-Variance 直觉

Bias 和 variance 是传统统计学习中理解误差来源的经典框架。

Bias 可以理解为模型假设太强，学不出真实规律。

Variance 可以理解为模型对训练样本太敏感，换一批数据结果变化很大。

高 bias 的表现：

1. 训练集都学不好。
2. 模型太小或训练不足。
3. 特征或架构不合适。
4. Loss 长期很高。

高 variance 的表现：

1. 训练集很好，验证集差。
2. 小数据上过拟合。
3. 不同 seed 波动大。
4. 输出风格或能力不稳定。

在大模型中，这个框架仍然有用，但要小心。模型规模、数据规模和训练 recipe 会改变传统 bias-variance 直觉。大模型可能参数很多，但如果数据巨大、正则化合适、训练目标稳定，仍然能泛化。

面试回答：

```text
Bias-variance 是理解误差来源的框架。Bias 高表示模型或训练不足，训练集也学不好；variance 高表示模型对训练样本过度敏感，训练集好但验证集差。LLM 中这个框架仍有用，但大模型的过参数化和海量数据会让传统直觉更复杂。
```

## 6.4 Overfitting：模型学到了训练集的细节

Overfitting 指模型过度贴合训练数据，导致未见数据表现变差。

LLM 中过拟合可能出现在多个阶段。

预训练中过拟合：

1. 重复数据过多。
2. 模型记住训练样本。
3. Benchmark contamination。
4. 训练 loss 低但真实能力一般。

SFT 中过拟合：

1. 数据量小。
2. 指令模板单一。
3. 模型学到固定回答风格。
4. 真实用户问题泛化差。

DPO/RLHF 中过拟合：

1. 过拟合偏好数据风格。
2. Reward model 泛化差。
3. 学会迎合 judge。
4. 过度拒答或冗长回答。

RAG 中过拟合：

1. 只在固定评测文档上调 prompt。
2. 只优化固定 query set。
3. 换文档、换领域后召回下降。

面试回答：

```text
Overfitting 在 LLM 中不只发生在预训练，也会发生在 SFT、DPO、RAG prompt 和评估集上。典型表现是训练或固定 benchmark 表现很好，但真实用户分布、改写问题或新领域上表现变差。缓解需要数据多样性、去重、holdout、能力回归和线上抽样评估。
```

## 6.5 Underfitting：模型还没学够

Underfitting 指模型连训练数据都拟合不好。

LLM 中可能来自：

1. 模型太小。
2. 训练 token 不够。
3. Learning rate 或优化器配置不好。
4. 数据质量太差。
5. Loss mask 错误。
6. 训练时间太短。
7. LoRA rank 太小或可训练模块太少。

面试中如果 loss 高，不要只说“模型不够大”。要先判断训练是否正常：数据、label、mask、优化器、学习率、batch、精度和代码实现都要排查。

## 6.6 Regularization：限制模型过度贴合训练数据

Regularization 是降低过拟合风险的方法。

常见正则化包括：

1. Weight decay。
2. Dropout。
3. Early stopping。
4. Data augmentation。
5. Label smoothing。
6. 数据去重和质量过滤。
7. LoRA 低秩约束。
8. KL penalty。

在 LLM 中，正则化不只是数学项，也可以是数据和系统策略。

例如：

1. 去重降低记忆风险。
2. 混入通用数据降低 continued pretraining 的遗忘。
3. KL penalty 限制 RLHF policy 偏离 reference。
4. LoRA rank 限制微调自由度。
5. Early stopping 防止小数据 SFT 过拟合。

面试回答：

```text
Regularization 的目标是降低模型过度贴合训练数据的风险。LLM 中正则化不只包括 weight decay 和 dropout，也包括数据去重、early stopping、LoRA 低秩约束、通用数据混合和 RLHF 中的 KL penalty。核心是限制模型学到训练集噪声或过度偏离原有能力。
```

## 6.7 数据规模和模型规模

大模型泛化能力很大程度来自数据规模和模型规模共同作用。

如果模型大但数据少，可能 undertrain 或记忆训练数据。

如果数据多但模型小，模型容量不足，可能学不完复杂模式。

如果数据质量差，规模再大也会引入噪声、偏见、错误事实和安全风险。

这就是为什么 scaling law 和 Chinchilla 讨论模型参数、训练 token 和 compute 的匹配。

面试回答：

```text
大模型不是参数越大越好，也不是 token 越多越好。泛化取决于模型容量、数据规模、数据质量和计算预算的匹配。模型太大但训练 token 不够会 undertrain；数据很多但质量差也会损害泛化。因此要看 compute-optimal 配置和数据 mixture。
```

## 6.8 大模型为什么参数多但还能泛化

这是面试高频开放题。

传统直觉认为参数多容易过拟合。但大模型能泛化，可能来自多个因素：

1. 数据规模巨大。
2. 文本分布中有大量可复用结构。
3. 训练目标要求模型压缩统计规律。
4. Transformer 架构有归纳偏置。
5. 优化过程和正则化影响最终解。
6. 过参数化可能让优化更容易找到泛化较好的解。

更稳妥的回答不是“参数多也不会过拟合”，而是：

```text
大模型仍然会过拟合和记忆，但它在海量数据、合适训练目标、优化 recipe 和过参数化条件下，也能学习可迁移的统计结构。泛化能力必须通过 held-out、分布外、扰动、组合和真实用户评估验证，不能只看训练 loss 或公开榜单。
```

## 6.9 记忆和泛化不是二选一

LLM 既会记忆，也会泛化。

记忆更容易发生在：

1. 数据重复率高。
2. 样本独特且容易被识别。
3. 训练时间过长。
4. 模型容量足够大。
5. 数据包含代码、密钥、PII 或固定文本。

泛化更依赖：

1. 数据多样性。
2. 任务结构可迁移。
3. 模型容量和训练 token 匹配。
4. 评估覆盖改写和组合。
5. 训练没有只背固定模板。

面试回答：

```text
LLM 的能力来自记忆、模式学习、组合泛化和上下文学习的共同作用。不能简单说它只是背诵，也不能说它完全理解。更可操作的判断方式是设计扰动、改写、组合和分布外评估，看模型是否能在训练样本之外稳定解决问题。
```

## 6.10 Benchmark contamination

Benchmark contamination 指评测数据泄漏进训练数据，导致模型在 benchmark 上表现虚高。

常见来源：

1. 公开 benchmark 被爬入预训练数据。
2. 训练集中包含题目或答案的改写版本。
3. SFT 或偏好数据包含测试集风格。
4. 开发过程中反复调 prompt 过拟合固定评测集。

污染的风险是：模型看起来能力强，但真实泛化并没有提升。

缓解方法：

1. 数据去重和近重复检测。
2. 保密评测集。
3. 动态评测集。
4. 分布外测试。
5. 线上真实样本评估。
6. 只看榜单不下结论，要做 bad case 分析。

面试回答：

```text
Benchmark contamination 会让模型分数虚高，因为模型可能见过题目、答案或相似模板。判断真实泛化不能只看公开榜单，还要做污染检测、保密集、动态集、分布外评估和线上抽样。
```

## 6.11 分布外泛化和组合泛化

分布外泛化关注模型能否处理训练分布之外的输入。

组合泛化关注模型能否把已学组件重新组合到新场景。

例如模型见过：

```text
A 比 B 大，B 比 C 大
```

也见过简单比较任务，但没见过某个具体名字组合。它能否推出 A 比 C 大，就是组合泛化的一类例子。

LLM 评估中要特别关注：

1. 改写问题。
2. 新实体组合。
3. 多步推理。
4. 长上下文扰动。
5. 对抗提示。
6. 工具环境变化。

## 6.12 泛化和评估设计

如果评估集太窄，模型很容易“看起来泛化”。

好的泛化评估应包括：

1. In-distribution 测试。
2. Out-of-distribution 测试。
3. 改写和扰动测试。
4. 组合泛化测试。
5. 长尾样本。
6. 人工 bad case 分析。
7. 线上真实抽样。

对于 RAG，还要测试新文档、新 query、新权限和新领域。

对于 DPO，还要测试 helpfulness、安全拒答、over-refusal、格式稳定性和能力回归。

对于多模态，还要测试新图片风格、OCR、图表、计数和空间关系。

## 6.13 最小可运行泛化审计 demo

下面这个 demo 不调用任何机器学习库，只用 toy 数据模拟三类模型：`memorizer` 记住训练样本、`keyword_rule` 学到可迁移规则、`underfit` 基本没学到目标。它同时审计 train / validation / test gap、bias-variance、L2 正则、early stopping、线上分布偏移和训练-评估 exact overlap。

```python
from collections import defaultdict


train = [
    {"id": "tr_01", "text": "solve math triangle", "label": 1},
    {"id": "tr_02", "text": "debug code loop", "label": 1},
    {"id": "tr_03", "text": "reasoning chain puzzle", "label": 1},
    {"id": "tr_04", "text": "math proof induction", "label": 1},
    {"id": "tr_05", "text": "code bug stack", "label": 1},
    {"id": "tr_06", "text": "reasoning compare objects", "label": 1},
    {"id": "tr_07", "text": "weather small talk", "label": 0},
    {"id": "tr_08", "text": "movie preference chat", "label": 0},
    {"id": "tr_09", "text": "music playlist request", "label": 0},
    {"id": "tr_10", "text": "casual greeting", "label": 0},
]

validation = [
    {"id": "va_01", "text": "math word problem", "label": 1},
    {"id": "va_02", "text": "code task parsing", "label": 1},
    {"id": "va_03", "text": "reasoning chain with distractor", "label": 1},
    {"id": "va_04", "text": "math geometry proof", "label": 1},
    {"id": "va_05", "text": "weather forecast chat", "label": 0},
    {"id": "va_06", "text": "recipe suggestion", "label": 0},
]

test = [
    {"id": "te_01", "text": "math contest problem", "label": 1},
    {"id": "te_02", "text": "code parse json", "label": 1},
    {"id": "te_03", "text": "reasoning logic grid", "label": 1},
    {"id": "te_04", "text": "code unit test", "label": 1},
    {"id": "te_05", "text": "travel planning", "label": 0},
    {"id": "te_06", "text": "friendly greeting", "label": 0},
]

production = [
    {"id": "pr_01", "text": "math word problem", "label": 1, "slice": "in_domain"},
    {"id": "pr_02", "text": "code review task", "label": 1, "slice": "in_domain"},
    {"id": "pr_03", "text": "weather small talk", "label": 0, "slice": "in_domain"},
    {"id": "pr_04", "text": "algebra equation", "label": 1, "slice": "new_terms"},
    {"id": "pr_05", "text": "python bug report", "label": 1, "slice": "new_terms"},
    {"id": "pr_06", "text": "friendly greeting", "label": 0, "slice": "new_terms"},
]


def normalize(text):
    return " ".join(text.lower().split())


memorized_labels = {normalize(row["text"]): row["label"] for row in train}


def predict_memorizer(row):
    return memorized_labels.get(normalize(row["text"]), 0)


def predict_keyword_rule(row):
    words = set(normalize(row["text"]).split())
    return int(bool(words & {"math", "code", "reasoning"}))


def predict_underfit(row):
    return 0


def accuracy(predict_fn, rows):
    correct = sum(int(predict_fn(row) == row["label"]) for row in rows)
    return correct / len(rows)


def summarize_model(name, predict_fn):
    train_acc = accuracy(predict_fn, train)
    val_acc = accuracy(predict_fn, validation)
    test_acc = accuracy(predict_fn, test)
    return {
        "train": round(train_acc, 3),
        "val": round(val_acc, 3),
        "test": round(test_acc, 3),
        "gap": round(train_acc - val_acc, 3),
    }


models = {
    "memorizer": predict_memorizer,
    "keyword_rule": predict_keyword_rule,
    "underfit": predict_underfit,
}

score_summary = {name: summarize_model(name, fn) for name, fn in models.items()}

diagnosis = {
    "memorizer_overfit": score_summary["memorizer"]["train"] >= 0.9 and score_summary["memorizer"]["gap"] >= 0.25,
    "underfit_detected": score_summary["underfit"]["train"] < 0.7 and score_summary["underfit"]["val"] < 0.7,
    "rule_generalizes": score_summary["keyword_rule"]["val"] >= 0.9 and score_summary["keyword_rule"]["test"] >= 0.9,
}


def bias_variance(predictions, true_value, noise_var):
    mean_pred = sum(predictions) / len(predictions)
    bias_sq = (mean_pred - true_value) ** 2
    variance = sum((p - mean_pred) ** 2 for p in predictions) / len(predictions)
    return {
        "mean_prediction": round(mean_pred, 3),
        "bias_sq": round(bias_sq, 3),
        "variance": round(variance, 3),
        "noise": round(noise_var, 3),
        "expected_error": round(bias_sq + variance + noise_var, 3),
    }


weights = [3.0, 4.0]
task_loss = 0.42

l2_penalty = lambda_l2 * sum(w * w for w in weights)
regularized_loss = task_loss + l2_penalty

checkpoints = [
    {"step": 100, "train_loss": 0.82, "val_loss": 0.86},

def slice_scores(predict_fn, rows):
    grouped = defaultdict(list)
    for row in rows:
        grouped[row["slice"]].append(row)
    return {name: round(accuracy(predict_fn, subset), 3) for name, subset in sorted(grouped.items())}


prod_acc = accuracy(predict_keyword_rule, production)
shift = {
    "val_acc": score_summary["keyword_rule"]["val"],

    "score_drop": round(score_summary["keyword_rule"]["val"] - prod_acc, 3),

}

eval_for_leakage = validation + [{"id": "eval_copy", "text": "math proof induction", "label": 1}]

    for row in eval_for_leakage
    if normalize(row["text"]) in train_hash_to_id
]
overlap_rate = len(exact_overlaps) / len(eval_for_leakage)

checks = {
    "memorizer_overfits": diagnosis["memorizer_overfit"],
    "underfit_detected": diagnosis["underfit_detected"],
    "rule_generalizes_clean_sets": diagnosis["rule_generalizes"],
    "early_stopping_before_overfit": best_checkpoint["step"] == 300,
    "shift_detected": shift["score_drop"] > 0.2 and min(shift["slice_scores"].values()) < 0.7,
    "contamination_detected": overlap_rate > 0,

print("score_summary=", score_summary)
print("diagnosis=", diagnosis)
print("bias_variance=", bias_variance([1.2, 1.8, 2.4], true_value=2.0, noise_var=0.1))
print("regularization=", {"task_loss": task_loss, "l2_penalty": round(l2_penalty, 3), "total": round(regularized_loss, 3)})
```

## 6.14 面试高频问题

### 6.14.1 大模型是在背诵还是泛化

回答：

```text
两者都有。LLM 会记忆高频、重复或特殊训练样本，也会通过 next-token prediction 学习语言、知识和任务模式，从而在新表述和新组合上泛化。更成熟的判断不是二选一，而是用去重、污染检测、改写测试、组合泛化和分布外评估区分记忆和泛化。
```

### 6.14.2 如何判断 SFT 过拟合

回答：

```text
如果 SFT train loss 持续下降，但验证集、人评或真实用户样本变差，可能过拟合。还要看模型是否学到固定模板、回答变机械、对改写问题泛化差、误拒上升或能力回归。缓解可以用数据去重、多样化、early stopping、混入通用数据和能力回归评估。
```

### 6.14.3 如何缓解过拟合

回答：

```text
可以从数据、模型、优化和评估四层做。数据上去重、增加多样性和质量过滤；模型上用 LoRA rank、dropout、weight decay 等限制自由度；优化上 early stopping、降低学习率；评估上使用 held-out、分布外、改写和线上抽样，避免只过拟合固定 benchmark。
```

### 6.14.4 Benchmark contamination 为什么危险

回答：

```text
污染会让 benchmark 分数虚高，因为模型可能见过题目、答案或相似模板。这会误导模型比较和研究结论。需要做数据去重、近重复检测、保密评测集、动态评测和真实用户样本评估。
```

## 6.15 常见误区

1. 认为参数多一定过拟合。
2. 认为大模型不会过拟合。
3. 只看训练 loss 判断模型好坏。
4. 把 benchmark 高分等同于真实泛化。
5. 不区分记忆、模式学习和组合泛化。
6. SFT 只看 loss，不看真实用户问题。
7. DPO 只看 win rate，不看 over-refusal 和能力回归。
8. RAG 只看固定问题集，不测试新文档和新领域。

## 6.17 本章总结

统计学习与泛化帮助我们判断模型是否真的学到了可迁移能力。

本章最重要的结论：

1. 泛化是模型在未见数据上的表现。
2. 训练误差低不代表泛化好。
3. 经验风险是训练集上的平均损失，真实风险是目标分布上的期望损失，二者差距就是泛化问题的核心。
4. Bias 高通常训练集也差，variance 高通常训练集好但验证集差。
5. LLM 的过拟合可能发生在预训练、SFT、DPO、RAG 和评估集上。
6. Regularization 在 LLM 中包括 weight decay、dropout、early stopping、数据去重、LoRA 低秩约束和 KL penalty。
7. 大模型既会记忆，也会泛化。
8. Benchmark contamination 会让公开分数虚高。
9. 泛化评估要包含分布外、扰动、组合和真实用户样本。

面试中讲泛化，不要说“大模型参数多所以会过拟合”或“大模型规模大所以不会过拟合”。成熟回答应该区分训练拟合、记忆、泛化、数据污染、评估设计和真实任务表现。
