# 第五章：设计 RAG 系统

这一章讨论大模型系统设计中最常见、也最容易被问深的一类题：如何设计 RAG 系统。

RAG 是 Retrieval-Augmented Generation，也就是检索增强生成。它的目标不是让模型“凭记忆回答”，而是在回答前从外部知识库检索相关证据，再把证据交给模型生成答案。这样可以缓解知识过期、企业私有知识缺失、事实幻觉和可追溯性问题。

但真正的 RAG 系统不是“embedding 加向量库加 prompt”这么简单。生产级 RAG 要解决文档解析、chunk 切分、embedding、索引构建、权限过滤、混合检索、rerank、上下文拼接、引用、拒答、评估、监控、更新和安全问题。

本章重点：文档解析、embedding、向量库、reranker、prompt assembly、引用、评估。

## 5.1 先明确题目边界

面试官问“设计 RAG 系统”，可能有几种范围。

第一，设计企业知识库问答。

用户上传公司文档、制度、产品手册、FAQ，系统根据知识库回答问题，并给出引用来源。重点是权限、更新、引用和准确性。

第二，设计开放域问答。

系统从网页、百科、论文、新闻等公开知识源检索信息。重点是时效性、召回、证据质量和来源可信度。

第三，设计客服 RAG。

系统根据商品、订单、售后政策和历史工单回答用户问题。重点是实时业务数据、工单系统集成、人工接管和 SLA。

第四，设计代码或技术文档 RAG。

系统检索代码仓库、API 文档、设计文档和 issue。重点是代码结构、版本、符号检索、长上下文和权限。

面试开头可以这样澄清：

我先按企业知识库 RAG 系统设计，支持文档导入、解析、索引、权限控制、在线检索、rerank、答案生成、引用来源、拒答、评估和监控。第一版重点保证答案有据可查、权限正确、更新可控和延迟可接受。

## 5.2 RAG 解决什么问题

在系统设计面试中，先讲清 RAG 的价值很重要。

RAG 主要解决四类问题。

第一，模型知识过期。

模型训练完成后，参数里的知识不会自动更新。RAG 可以通过更新外部知识库，让模型使用最新资料。

第二，企业私有知识缺失。

模型没有见过公司的内部制度、产品文档、项目资料和客户合同。RAG 可以让模型在回答时读取授权范围内的内部文档。

第三，降低事实幻觉。

模型凭参数记忆回答容易编造。RAG 通过提供证据，让模型基于检索内容回答，并输出引用。

第四，提高可审计性。

用户和管理员可以看到答案来自哪些文档、哪些段落，方便验证和追责。

但 RAG 不是万能的。它不能自动保证答案正确。如果检索不到、检索错、证据冲突、模型不忠实使用证据，仍然会出错。

## 5.3 功能需求

一个企业级 RAG 系统通常需要以下功能。

| 模块 | 功能 |
| --- | --- |
| 文档接入 | 上传、同步、爬取、API 导入、权限同步 |
| 文档解析 | PDF、Word、HTML、Markdown、表格、图片 OCR |
| 数据治理 | 去重、清洗、版本、元数据、敏感信息处理 |
| Chunk 切分 | 按段落、标题、语义、表格和代码结构切分 |
| Embedding | 向量化、批处理、版本管理、重算 |
| 索引 | 向量索引、倒排索引、混合检索、增量更新 |
| 检索 | query rewrite、召回、权限过滤、rerank |
| 生成 | prompt assembly、引用、拒答、多轮上下文 |
| 评估 | 检索 recall、答案准确率、引用正确率、幻觉率 |
| 监控 | 延迟、命中率、失败率、成本、用户反馈 |

第一版可以先支持文本 PDF、Markdown 和网页文档，后续再扩展复杂表格、图片、扫描件、代码仓库和实时业务数据库。

## 5.4 非功能需求

RAG 系统的非功能需求包括准确性、权限、安全、延迟、更新和成本。

### 5.4.1 准确性

RAG 的核心不是“检索到东西”，而是“检索到正确证据，并让模型忠实使用证据”。

准确性要拆成：

1. 检索是否找到了相关文档。
2. Rerank 是否把正确证据排到前面。
3. Prompt 是否让模型看到了足够证据。
4. 生成是否忠实于证据。
5. 引用是否能支持答案。

### 5.4.2 权限正确性

企业 RAG 最重要的安全要求之一是权限。用户只能检索和看到自己有权限的文档。

权限不能靠 prompt 约束，必须在检索层和数据层实现。

### 5.4.3 延迟

在线问答链路包含 query rewrite、embedding、检索、rerank、模型生成和输出审核。每一步都会增加延迟。

要根据业务设定 P95 延迟目标。例如内部知识库可以接受几秒，在线客服可能要求更短。

### 5.4.4 更新及时性

文档更新后多久能被检索到？删除权限后多久生效？这些都影响系统设计。

如果是企业知识库，权限变更和文档删除通常要尽快生效。

### 5.4.5 成本

成本包括文档解析、embedding 计算、向量库存储、rerank 模型、生成模型 token 和评估成本。

## 5.5 容量估算

假设设计一个企业知识库 RAG：

1. 企业有 10 万员工。
2. 文档 1000 万篇。
3. 平均每篇 2000 token。
4. 切分后平均每篇 5 个 chunk。
5. 总 chunk 数约 5000 万。
6. 每个 embedding 维度 1024，使用 FP16 存储约 2KB。
7. 仅向量原始存储约 100GB，加索引和元数据可能数百 GB。

在线请求：

1. 峰值 1000 QPS。
2. 每次 query 做一次 embedding。
3. 向量召回 top 100。
4. Rerank top 100 到 top 10。
5. 最终拼接 5 到 10 个 chunk 给生成模型。

这个估算说明：RAG 系统有两个链路。

第一，离线或准实时索引链路，瓶颈是文档解析、embedding 批处理、索引构建和增量更新。

第二，在线问答链路，瓶颈是检索延迟、rerank 延迟、生成模型延迟和权限过滤。

## 5.6 高层架构

RAG 系统可以分成离线索引链路和在线问答链路。

### 5.6.1 离线索引链路

离线链路包括：

1. Document Connector：从文件系统、网页、数据库、企业网盘、Wiki、代码仓库同步文档。
2. Parser：解析 PDF、Word、HTML、Markdown、表格和图片。
3. Cleaner：清洗模板、页眉页脚、乱码、重复内容。
4. Chunker：按结构或语义切分文档。
5. Metadata Service：记录标题、作者、时间、权限、版本、来源。
6. Embedding Worker：批量生成 chunk embedding。
7. Index Builder：构建向量索引和倒排索引。
8. Index Store：存储索引、chunk、元数据和版本。

### 5.6.2 在线问答链路

在线链路包括：

1. API Gateway：鉴权、限流、请求入口。
2. Query Processor：query rewrite、意图识别、语言检测。
3. Retriever：向量检索、关键词检索、混合召回。
4. Permission Filter：按用户权限过滤文档。
5. Reranker：重排候选 chunk。
6. Context Builder：选择证据、去重、压缩、排序。
7. LLM Generator：基于证据生成答案。
8. Citation Verifier：检查引用是否支持答案。
9. Safety Filter：安全审核和敏感信息处理。
10. Feedback Logger：记录用户反馈和评估样本。

## 5.7 文档接入与解析

文档解析质量决定 RAG 上限。很多 RAG 系统失败，不是模型不行，而是文档解析和切分太差。

### 5.7.1 文档接入

常见来源包括：

1. 用户上传文件。
2. 企业网盘。
3. Wiki 和知识库。
4. 数据库。
5. 工单系统。
6. 代码仓库。
7. 网页和产品文档。

不同来源要有 connector，支持全量同步和增量同步。

### 5.7.2 文档解析

解析要保留结构信息。

例如 PDF 不是简单抽文本。要尽量保留标题层级、段落、表格、页码、脚注、图注和列表。表格要避免行列错位。扫描件需要 OCR，OCR 还要处理错误字符。

### 5.7.3 文档清洗

清洗包括：

1. 删除页眉页脚。
2. 删除导航栏和广告模板。
3. 处理乱码。
4. 去除重复段落。
5. 统一编码和标点。
6. 标记低质量内容。

清洗不能过度。过度清洗可能删掉关键信息。

### 5.7.4 文档版本

文档会更新。系统要记录 document_id、version、更新时间、来源和删除状态。

用户看到的答案最好能引用具体文档版本，否则文档更新后，历史答案可能无法复现。

## 5.8 Chunk 切分

Chunk 切分是 RAG 中最重要的工程细节之一。

### 5.8.1 为什么不能随便切

如果 chunk 太小，语义不完整，检索回来也无法回答问题。如果 chunk 太大，embedding 语义混杂，检索不准，也浪费上下文。

### 5.8.2 常见切分策略

1. 固定长度切分：实现简单，但可能切断语义。
2. 按段落切分：保留自然语义，但长度不均。
3. 按标题层级切分：适合文档和知识库。
4. 语义切分：根据语义边界切分，效果好但成本高。
5. 表格专门切分：保留表头、行列和说明。
6. 代码结构切分：按函数、类、文件和符号切分。

### 5.8.3 Overlap

Chunk overlap 可以缓解边界信息丢失。例如相邻 chunk 重叠 10% 到 20%。

但 overlap 会增加 chunk 数、embedding 成本和检索冗余。需要根据文档类型调参。

### 5.8.4 Parent-child chunk

一种常见设计是小 chunk 用于检索，大 chunk 或父段落用于生成。

因为小 chunk embedding 更聚焦，召回更准；生成时需要更完整上下文，所以取父节点或相邻段落。

## 5.9 Embedding 与索引

Embedding 把文本变成向量，向量库负责近似最近邻检索。

### 5.9.1 Embedding 模型选择

选择 embedding 模型要看：

1. 语言支持。
2. 领域适配。
3. 上下文长度。
4. 向量维度。
5. 速度和成本。
6. 是否支持 query/document 不同编码。
7. 是否需要微调。

企业中文知识库不能只看英文 benchmark。代码 RAG 也不能只用通用文本 embedding。

### 5.9.2 向量索引

向量索引可以使用 HNSW、IVF、PQ 等近似检索结构，也可以使用托管向量数据库。

系统设计面试中不需要深入实现算法，但要说明 trade-off：召回率、延迟、内存、构建时间、增量更新和过滤能力。

### 5.9.3 混合检索

纯向量检索不总是最好。关键词、数字、专有名词、错误码、函数名、政策编号等更适合 lexical search。

生产 RAG 常用混合检索：BM25 或倒排索引加向量检索，再融合候选。

### 5.9.4 Embedding 版本

Embedding 模型升级后，旧向量通常不能直接混用。平台要记录 embedding_model_version，并支持后台重算和索引版本切换。

## 5.10 权限过滤

企业 RAG 的权限设计非常重要。

### 5.10.1 权限为什么不能放到生成阶段

如果无权限文档已经被检索并放进 prompt，即使要求模型“不要泄露”，也已经发生了安全边界突破。

权限必须在检索前、检索中或检索后但生成前完成，绝不能只靠 prompt。

### 5.10.2 权限模型

常见权限模型包括：

1. 用户级权限。
2. 部门级权限。
3. 项目级权限。
4. 文档级 ACL。
5. 租户隔离。
6. 时间范围权限。

### 5.10.3 过滤方式

有两种主要方式。

第一，检索前过滤。根据用户权限只查询允许的索引分片或 collection。优点是安全和性能好，缺点是索引管理复杂。

第二，检索后过滤。先召回候选，再按权限过滤。实现简单，但如果过滤后候选不足，召回会下降，也可能浪费资源。

生产系统常用组合：租户和大权限域做检索前过滤，细粒度 ACL 做检索后过滤或索引内过滤。

## 5.11 Query 处理

用户 query 往往不适合直接检索。

### 5.11.1 Query rewrite

多轮对话中，用户可能问“那它怎么申请？”这类指代问题。Query rewrite 要结合上下文改写成完整问题。

例如把“那它怎么申请？”改成“员工如何申请年度培训预算报销？”

### 5.11.2 Multi-query

复杂问题可以生成多个检索 query，从不同角度召回证据。例如一个问题同时涉及政策、流程和例外情况。

### 5.11.3 HyDE

HyDE 的思路是先让模型生成一个假想答案，再用假想答案做检索。它有时能提升召回，但也可能引入偏差，需要评估。

### 5.11.4 查询分类

系统可以先判断问题类型：是否需要检索、是否是闲聊、是否需要实时数据库、是否不可回答、是否高风险。

不是所有问题都需要 RAG。强行检索可能增加延迟和噪声。

## 5.12 Reranker

Retriever 负责召回，reranker 负责精排。

### 5.12.1 为什么需要 reranker

向量检索擅长快速召回，但 top-k 顺序不一定准确。Reranker 可以用 cross-encoder 或 LLM 对 query 和候选 chunk 做更细粒度相关性判断。

### 5.12.2 Rerank 的 trade-off

Reranker 通常比向量检索慢，所以要控制候选数量。常见做法是召回 top 100，再 rerank 到 top 5 或 top 10。

### 5.12.3 Rerank 指标

要看 MRR、nDCG、Recall@k、最终答案准确率和延迟。Rerank 不能只看离线相关性，还要看端到端回答质量。

## 5.13 Prompt Assembly

Prompt assembly 是把系统指令、用户问题、对话历史和检索证据组织给模型。

### 5.13.1 证据选择

不能把所有候选都塞进 prompt。要根据 token budget 选择最相关、最多样、最可信的证据。

选择时考虑：

1. Rerank 分数。
2. 文档权威性。
3. 时间新鲜度。
4. 证据多样性。
5. 是否重复。
6. 用户权限。

### 5.13.2 证据排序

证据顺序会影响模型使用。可以按相关性、文档结构、时间或问题子任务排序。

### 5.13.3 指令设计

RAG prompt 通常要明确：

1. 只基于给定证据回答。
2. 不足以回答时说明不知道或请求补充信息。
3. 给出引用。
4. 不要编造来源。
5. 区分证据和模型常识。

### 5.13.4 上下文压缩

当证据太多时，可以用摘要、句子抽取或 rerank 后再压缩。压缩会带来信息丢失，因此要评估。

## 5.14 引用与可追溯

引用是 RAG 系统区别于普通聊天的重要能力。

### 5.14.1 引用粒度

引用可以到文档、章节、页码、段落或 chunk。

企业问答最好引用到具体段落或页码，方便用户核对。

### 5.14.2 引用验证

模型可能生成看似合理但不支持答案的引用。可以做 citation verifier，检查答案中的关键陈述是否被引用证据支持。

### 5.14.3 引用展示

前端要展示文档标题、位置、时间、摘要和跳转链接。用户点击引用时，要再次做权限检查。

### 5.14.4 历史答案可复现

如果文档更新，历史答案引用的内容可能变化。系统可以记录回答时使用的 document version 和 chunk id，以便审计。

## 5.15 拒答与不确定性

RAG 系统不能什么都答。

应该拒答或谨慎回答的情况包括：

1. 没有检索到相关证据。
2. 证据相关性很低。
3. 证据之间冲突。
4. 用户没有权限。
5. 问题需要实时业务系统但当前没有接入。
6. 问题属于高风险领域，需要人工确认。

拒答不是失败。正确拒答比编造答案更可靠。

可以设计策略：如果检索分数低于阈值，回答“知识库中没有找到足够依据”；如果证据冲突，列出冲突来源并建议人工确认；如果权限不足，提示用户没有访问权限。

## 5.16 评估体系

RAG 评估要分层，不然很难定位问题。

### 5.16.1 检索评估

指标包括 Recall@k、MRR、nDCG、hit rate。

需要构造 query 到 gold document 或 gold chunk 的标注集。

### 5.16.2 生成评估

指标包括答案准确率、faithfulness、引用支持率、完整性、简洁性、拒答正确率。

### 5.16.3 端到端评估

端到端评估看用户问题最终是否被正确回答。它最接近业务目标，但难定位是检索错、rerank 错还是生成错。

### 5.16.4 失败归因

RAG 失败可以归因到：

1. 文档不存在。
2. 文档解析失败。
3. Chunk 切分不合理。
4. Embedding 召回失败。
5. 关键词检索失败。
6. 权限过滤后无结果。
7. Rerank 排序错误。
8. Prompt 证据太多或太少。
9. 模型没有忠实使用证据。
10. 引用错误。

只有做失败归因，才能知道该优化哪一层。

## 5.17 监控与反馈

线上 RAG 监控包括性能、质量、安全和成本。

### 5.17.1 性能监控

包括 query embedding 延迟、检索延迟、rerank 延迟、生成延迟、端到端延迟、超时率。

### 5.17.2 质量监控

包括检索空结果率、低置信回答率、用户点踩率、引用点击率、重新提问率、人工抽检准确率。

### 5.17.3 安全监控

包括越权检索拦截、敏感信息命中、prompt injection 命中、违规输出、异常查询模式。

### 5.17.4 成本监控

包括 embedding 成本、rerank 成本、生成 token 成本、向量库存储成本和每个租户成本。

## 5.18 更新与删除

RAG 的优势之一是知识可更新，但更新系统并不简单。

### 5.18.1 增量更新

文档新增或修改后，只重算受影响 chunk 的 embedding，并更新索引。

### 5.18.2 删除

文档删除或权限撤销后，索引中对应 chunk 必须失效。不能只从前端隐藏文档。

### 5.18.3 索引版本切换

大规模重建索引时，可以先构建新版本索引，验证后切换流量，异常时回滚旧索引。

### 5.18.4 权限变更

权限变更要快速生效。可以把权限元数据与索引解耦，检索时动态检查权限，避免每次权限变化都重建索引。

## 5.19 安全设计

RAG 安全不仅是模型输出安全，还包括知识库安全。

### 5.19.1 Prompt injection

外部文档可能包含恶意指令，例如要求模型忽略系统提示、泄露上下文或执行非预期操作。

防护方式包括：

1. 把文档内容标记为不可信数据，而不是指令。
2. System prompt 明确区分指令和证据。
3. 工具和权限由系统控制，不交给模型决定。
4. 对检索文档做风险扫描。
5. 输出前做安全审核。

### 5.19.2 权限泄漏

权限过滤必须在生成前完成。引用跳转也要重新鉴权。

### 5.19.3 数据泄漏

日志、缓存、评估样本中可能包含敏感文档内容，要做脱敏、访问控制和保留期限。

### 5.19.4 合规边界

本章讨论的是合法授权范围内的企业知识库和公开知识检索系统设计，不提供绕过权限、抓取未授权内容、规避访问控制或泄露敏感数据的方法。

## 5.20 成本优化

RAG 成本优化可以从离线和在线两侧做。

### 5.20.1 离线成本

1. 增量解析和增量 embedding。
2. 批处理 embedding。
3. 对低价值文档降低更新频率。
4. 删除重复和低质量 chunk。
5. 索引冷热分层。

### 5.20.2 在线成本

1. 缓存 query embedding。
2. 缓存热门检索结果。
3. 用轻量 reranker 先筛，再用重 reranker 精排。
4. 控制最终证据数量。
5. 对简单问题不走 RAG。
6. 使用小模型生成简单答案。

成本优化不能牺牲权限和引用正确性。

## 5.21 面试答题模板

如果面试官问“设计 RAG 系统”，可以这样回答。

第一，澄清范围。

我先按企业知识库问答设计，支持文档导入、解析、索引、权限控制、检索、rerank、基于证据生成、引用、拒答、评估和监控。

第二，定义指标。

核心指标包括检索 Recall@k、答案准确率、faithfulness、引用支持率、拒答正确率、P95 延迟、权限违规率和单位查询成本。

第三，做容量估算。

根据文档数、平均 chunk 数、embedding 维度、索引大小、峰值 QPS 和 rerank 候选数估算离线索引和在线查询压力。

第四，画架构。

系统分为离线索引链路和在线问答链路。离线链路包括 connector、parser、cleaner、chunker、embedding worker、index builder；在线链路包括 query processor、retriever、permission filter、reranker、context builder、LLM generator、citation verifier 和 feedback logger。

第五，展开关键难点。

重点讲文档解析、chunk 切分、混合检索、权限过滤、rerank、prompt assembly、引用验证和拒答策略。

第六，讲评估和监控。

RAG 评估要分检索、生成和端到端，并做失败归因。线上监控要看空结果率、低置信率、引用错误、越权拦截、延迟和成本。

第七，讲安全和更新。

权限必须在检索层实现，文档删除和权限撤销要及时生效，日志和缓存要防泄漏，外部文档要防 prompt injection。

## 5.22 常见面试追问

### 5.22.1 RAG 能完全解决幻觉吗？

不能。RAG 可以降低事实幻觉，但如果检索错、证据不足、证据冲突或模型不忠实使用证据，仍然会幻觉。因此需要引用、拒答、faithfulness 评估和失败归因。

### 5.22.2 为什么不能只用向量检索？

向量检索擅长语义相似，但对专有名词、编号、错误码、函数名、数字和精确匹配不一定好。生产系统常用向量检索加 BM25 的混合检索。

### 5.22.3 Chunk 大小怎么选？

要根据文档类型和任务调参。太小语义不完整，太大检索不准且浪费上下文。常见做法是按结构切分，加适当 overlap，必要时用 parent-child chunk。

### 5.22.4 权限过滤应该在什么时候做？

必须在生成前做。可以检索前按租户或权限域过滤，检索中用元数据过滤，检索后做细粒度 ACL 过滤。不能把无权限文档放进 prompt 后再要求模型不要泄露。

### 5.22.5 Reranker 为什么有用？

Retriever 负责快速召回，reranker 对 query 和候选 chunk 做更精细相关性判断，能把真正有用的证据排到前面。但它增加延迟和成本，需要控制候选数。

### 5.22.6 如何判断 RAG 问题出在检索还是生成？

看分层评估。如果 gold chunk 没被召回，是检索问题；如果 gold chunk 召回了但没排前，是 rerank 问题；如果正确证据进了 prompt 但答案错，是生成或 prompt 问题；如果引用不支持答案，是 citation 问题。

### 5.22.7 文档更新后如何生效？

新增或修改文档触发增量解析、chunk、embedding 和索引更新。删除和权限撤销要让对应 chunk 立即失效。大规模重建可以用索引版本切换和回滚。

### 5.22.8 如何评估引用是否正确？

可以抽取答案中的关键 claim，检查每个 claim 是否被引用 chunk 支持。可以用规则、NLI 模型、LLM judge 和人工抽检结合，但高风险场景不能只依赖自动评估。

## 5.23 本章小结

RAG 系统的核心价值，是把大模型回答从“凭参数记忆”改造成“基于可更新、可授权、可引用证据”的生成流程。

本章要记住几句话：

1. 生产级 RAG 不是 embedding 加向量库，而是文档、索引、检索、生成、引用、权限和评估的完整系统。
2. RAG 有离线索引链路和在线问答链路，瓶颈不同。
3. 文档解析和 chunk 切分决定检索上限。
4. 混合检索通常比单纯向量检索更稳。
5. 权限必须在系统层和检索层实现，不能靠 prompt。
6. 引用要能支持答案，引用本身也需要验证。
7. 正确拒答比无依据编造更重要。
8. RAG 评估要分检索、rerank、生成、引用和端到端，并做失败归因。

如果面试中被问到 RAG 系统，最好的回答是先画出离线索引和在线问答两条链路，再围绕权限、引用、评估、更新和安全讲清生产级系统如何落地。
