# 第十章：设计评估平台

面试中如果让你“设计一个大模型评估平台”，考察的不是会不会调用几个 benchmark，也不是会不会画一个提交任务、跑脚本、生成报表的流水线。真正考察的是：你是否理解大模型评估为什么困难，能否把离线 benchmark、在线实验、人工评审、回归测试、安全评估、模型版本管理和报告体系组织成一个可长期演进的平台。

评估平台是大模型工程体系中的“仪表盘”和“刹车系统”。训练平台负责生产模型，推理平台负责服务模型，应用平台负责承载业务，而评估平台负责回答几个关键问题：这个模型到底好不好？相比上一版有没有退化？在哪些能力上提升，在哪些能力上变差？能不能上线？上线后是否仍然符合预期？

如果没有评估平台，模型迭代很容易变成“凭感觉”。某个 demo 看起来不错，不代表整体能力提升；某个 benchmark 分数上涨，不代表用户体验更好；某个安全样例通过，不代表真实攻击场景安全。大模型系统设计面试中，优秀答案要能把这些不确定性转化为可记录、可复现、可比较、可追责的工程流程。

本章重点讨论：benchmark 管理、模型版本、自动评估、人工评估、报告、回归测试。

合规边界：本章讨论合法合规的模型评估、质量治理和安全审核流程，不提供绕过平台安全策略、规避审计、伪造评估结果或攻击第三方系统的方法。

## 1. 题目理解：什么是大模型评估平台

大模型评估平台可以理解为一个统一系统，用来管理评测数据、模型版本、评测任务、评测指标、人工标注、结果报告和上线门禁。

它不是一个单独脚本，而是一个长期运行的平台。

它需要支持不同角色：

- 研究员：比较多个 checkpoint 的能力变化。
- 算法工程师：验证微调、数据清洗、推理策略是否有效。
- 产品经理：查看模型在业务场景中的可用性。
- 安全团队：检查模型是否存在越权、泄露、歧视、毒性和违规输出风险。
- 运维团队：追踪线上模型质量、延迟、成本和故障。
- 管理者：通过报告判断某个模型是否可以发布。

一个简化例子是：

公司训练了 `model-v12`，想替换线上 `model-v11`。评估平台需要自动拉取两个模型，在相同测试集上运行任务，计算准确率、胜率、拒答率、幻觉率、安全通过率、平均延迟和单位成本，再把结果和历史版本比较。如果某些关键指标低于阈值，平台应阻止上线或要求人工复核。

因此，评估平台的核心不是“跑一次分数”，而是“让模型质量评估成为可重复的生产流程”。

## 2. 为什么大模型评估比传统模型评估更难

传统分类模型评估通常比较简单：给定输入，模型输出一个类别，然后和标准答案比较，计算 accuracy、precision、recall、F1 等指标。

大模型评估复杂很多，主要原因有六个。

第一，输出是开放文本。

同一个问题可以有多个正确答案。比如“解释 Transformer 的注意力机制”，模型可能用公式解释，也可能用比喻解释。它们表达不同，但都可能正确。

第二，质量维度很多。

一个回答可能事实正确但啰嗦，也可能语言流畅但有幻觉，也可能技术上正确但不符合安全策略。评估不能只看一个分数。

第三，任务边界模糊。

大模型同时做问答、摘要、代码、推理、翻译、创作、对话、工具调用和多模态任务。不同任务的指标完全不同。

第四，模型输出不稳定。

温度、top-p、随机种子、系统提示词、上下文长度、工具状态都会影响结果。评估平台必须记录运行环境，否则结果不可复现。

第五，自动指标不总是可靠。

BLEU、ROUGE 等指标适合部分任务，但无法完整衡量事实性、逻辑性和用户偏好。LLM-as-a-judge 又存在偏置、位置偏好、长度偏好和自我偏好。

第六，线上真实体验和离线 benchmark 有差距。

Benchmark 分数高，不代表真实用户满意。真实用户问题更脏、更长、更模糊、更有业务上下文，也更容易包含异常输入。

所以，大模型评估平台通常需要混合以下几类评估：

- 规则评估。
- 精确匹配评估。
- 传统文本指标评估。
- 代码执行评估。
- 工具调用轨迹评估。
- LLM 裁判评估。
- 人工偏好评估。
- 安全红队评估。
- 线上 A/B 实验。
- 长期回归测试。

面试回答中要强调：没有单一指标能代表模型能力，评估平台要服务于多维度、可复现、可追踪的决策。

## 3. 需求分析

设计系统前，先明确需求。系统设计面试最怕一上来画架构图，不问场景。评估平台的需求可以分为功能需求和非功能需求。

### 3.1 功能需求

核心功能包括：

- Benchmark 管理：支持创建、导入、版本化、分组、抽样和权限控制。
- 模型版本管理：支持注册模型、checkpoint、推理配置、prompt 模板和部署 endpoint。
- 评估任务管理：支持创建评估任务、选择模型、选择数据集、选择指标、配置并发和预算。
- 自动评估：支持准确率、匹配类指标、文本相似度、代码执行、工具调用检查和 LLM 裁判。
- 人工评估：支持人工打分、pairwise 偏好比较、一致性检查和仲裁。
- 安全评估：支持有害内容、隐私泄露、越权意图、幻觉、偏见和合规风险检测。
- 报告生成：支持版本对比、能力维度雷达图、样例分析、失败案例聚类和上线建议。
- 回归测试：支持每次模型、prompt、RAG 索引或推理策略变更后自动运行关键测试集。
- 结果追踪：支持历史趋势、实验对比、可复现配置和审计日志。
- 门禁控制：支持上线前质量阈值和人工审批流程。

这些功能不要只列名词，要说明它们之间的关系。

一个典型流程是：

1. 用户注册待评估模型。
2. 用户选择 benchmark suite。
3. 平台生成评估任务。
4. Worker 调用模型 endpoint 或推理集群。
5. 平台收集输出和运行元数据。
6. 自动评估器计算指标。
7. 高风险或低置信样例进入人工评审。
8. 报告服务汇总结果。
9. Gatekeeper 判断是否允许进入下一阶段。

### 3.2 非功能需求

评估平台的非功能需求同样重要。

- 可复现性：同一次评估必须能记录模型版本、数据版本、prompt 版本、推理参数、评估器版本和运行时间。
- 可比较性：不同模型要在相同数据和相同设置下比较，避免“拿 A 的强项和 B 的弱项比”。
- 可扩展性：支持新增任务类型、指标和评估器。
- 稳定性：评估任务可能运行数小时甚至数天，系统要支持断点续跑和失败重试。
- 成本可控：大规模评估会消耗大量 GPU、API 调用和人工标注成本。
- 安全性：测试集、用户数据、模型输出和人工评审内容都可能敏感。
- 审计性：上线决策要能回溯依据，尤其是金融、医疗、教育、政企等场景。
- 公平性：人工评审任务分发要避免长期偏向某些评审员或某类样例。

如果面试官追问“哪个最重要”，可以回答：离线研究阶段最重要的是可复现和可比较；生产上线阶段最重要的是门禁、审计和回归；规模扩大后最重要的是成本和平台化扩展。

## 4. 容量估算

容量估算不需要追求绝对准确，重点是展示思考方法。

假设一个中型大模型团队：

- 每天新增模型或 prompt 版本：50 个。
- 每个版本平均跑 5 个 benchmark suite。
- 每个 suite 包含 5,000 条样例。
- 每条样例平均输入输出合计 3,000 tokens。
- 每天评估样例数：50 * 5 * 5,000 = 1,250,000 条。
- 每天 token 量：1,250,000 * 3,000 = 3,750,000,000 tokens。

这说明评估平台不是“小脚本系统”。它可能每天产生十亿级 token 调用，并存储大量输入、输出、评分、日志和中间结果。

如果每条样例保存：

- 输入文本：2 KB。
- 模型输出：4 KB。
- 评分结果：1 KB。
- 元数据和日志：1 KB。

那么每天结果存储大约是：

1,250,000 * 8 KB = 10 GB。

如果保留一年，单纯结果数据就是数 TB 级别。再考虑多模态样例、音视频评估、人工标注和日志，存储会更大。

调度侧也要估算：

- 评估任务提交 QPS 可能不高。
- 样例执行 QPS 可能很高。
- LLM 裁判调用成本可能接近被评估模型本身。
- 人工评估吞吐量远低于自动评估。

这些估算会影响系统设计：需要队列、批处理、缓存、采样、优先级和成本预算。

## 5. 高层架构

一个典型大模型评估平台可以拆成以下模块：

```text
User / CI / Release Pipeline
        |
        v
Evaluation API Gateway
        |
        v
Experiment Service ------ Model Registry
        |                         |
        |                         v
        |                  Model Endpoint / Inference Cluster
        |
        v
Benchmark Registry ------ Dataset Store
        |
        v
Task Scheduler ------ Queue ------ Evaluation Workers
        |                              |
        |                              v
        |                       Model Invocation Layer
        |                              |
        |                              v
        |                       Raw Output Store
        |
        v
Metric Engine ------ Judge Model Service
        |
        v
Human Review Service
        |
        v
Report Service ------ Dashboard
        |
        v
Gatekeeper / Release Decision
        |
        v
Audit Log / Metadata Store
```

各模块职责如下。

- Evaluation API Gateway：提供创建任务、查询结果、下载报告等接口。
- Experiment Service：管理一次评估实验的配置、状态和生命周期。
- Model Registry：记录模型版本、checkpoint、endpoint、prompt、推理参数和权限。
- Benchmark Registry：管理评测集、任务分类、数据版本和指标绑定关系。
- Dataset Store：存储评测样例，可以是对象存储、数据湖或专用数据仓库。
- Task Scheduler：把评估任务拆成样例级或 batch 级子任务。
- Queue：削峰填谷，隔离不同优先级任务。
- Evaluation Workers：执行模型调用、收集输出、运行部分轻量指标。
- Model Invocation Layer：统一封装不同模型 endpoint、内部推理集群和外部 API。
- Raw Output Store：保存模型原始输出和运行日志。
- Metric Engine：计算自动指标，并调用裁判模型。
- Judge Model Service：提供 LLM-as-a-judge 能力。
- Human Review Service：管理人工评审任务、标注规范和一致性检查。
- Report Service：生成对比报告、趋势图、失败案例和上线建议。
- Gatekeeper：根据阈值和审批流程决定是否放行。
- Audit Log：记录谁在何时用什么数据评估了什么模型，得出了什么结论。

这个架构的关键是：数据、模型、评估器、报告和上线门禁都被版本化，而不是一次性脚本。

## 6. 数据模型设计

评估平台的核心实体包括 `ModelVersion`、`Benchmark`、`EvalRun`、`EvalSampleResult`、`MetricResult`、`HumanReview` 和 `Report`。

### 6.1 ModelVersion

`ModelVersion` 表示一个可评估的模型版本。

关键字段包括：

- `model_id`：模型标识。
- `version_id`：版本标识。
- `checkpoint_uri`：checkpoint 或镜像地址。
- `endpoint_uri`：在线服务地址。
- `prompt_template_id`：系统 prompt 或模板版本。
- `tokenizer_version`：分词器版本。
- `inference_config`：温度、top-p、最大输出长度、停止词等。
- `training_data_snapshot`：训练数据快照，可选。
- `created_by`：创建人。
- `created_at`：创建时间。

为什么 prompt 也要记录？因为对大模型应用来说，模型行为不仅由权重决定，也由 system prompt、RAG 模板、工具说明、解码参数决定。如果只记录 checkpoint，不记录这些配置，评估结果就不可复现。

### 6.2 Benchmark

`Benchmark` 表示一组评测样例及其评估规则。

关键字段包括：

- `benchmark_id`：评测集标识。
- `version_id`：数据版本。
- `task_type`：问答、代码、摘要、RAG、工具调用、安全、数学、多模态等。
- `sample_count`：样例数量。
- `schema`：输入输出格式。
- `metric_config`：绑定哪些评估指标。
- `split`：dev、test、hidden、regression 等。
- `license`：数据授权信息。
- `sensitivity_level`：敏感等级。
- `owner`：负责人。

评测集必须版本化。否则如果 benchmark 被悄悄修改，历史分数就无法比较。

### 6.3 EvalRun

`EvalRun` 表示一次评估实验。

关键字段包括：

- `run_id`：评估运行标识。
- `model_version_id`：待评估模型。
- `baseline_model_version_id`：对照模型，可选。
- `benchmark_suite_id`：评测套件。
- `run_config`：并发、采样比例、随机种子、预算等。
- `status`：pending、running、paused、failed、completed、cancelled。
- `started_at`：开始时间。
- `finished_at`：结束时间。
- `created_by`：创建人。

### 6.4 EvalSampleResult

`EvalSampleResult` 表示单条样例的模型输出和执行信息。

关键字段包括：

- `run_id`：所属评估运行。
- `sample_id`：样例标识。
- `input_hash`：输入哈希。
- `raw_input_uri`：输入内容地址。
- `raw_output_uri`：输出内容地址。
- `latency_ms`：延迟。
- `input_tokens`：输入 token 数。
- `output_tokens`：输出 token 数。
- `cost`：调用成本。
- `error_code`：错误码。
- `retry_count`：重试次数。
- `seed`：随机种子，可选。

不要把所有长文本都塞进关系型数据库。常见做法是：元数据放数据库，长文本和多媒体放对象存储，数据库保存 URI 和 hash。

### 6.5 MetricResult

`MetricResult` 表示自动评估结果。

字段包括：

- `metric_name`：指标名称。
- `metric_version`：指标实现版本。
- `score`：分数。
- `confidence`：置信度。
- `explanation`：裁判模型解释，可选。
- `evidence_uri`：证据或日志地址。

指标也要版本化。一个指标实现升级后，历史分数可能变化。平台要么保留旧指标版本，要么支持重新计算并明确标记。

## 7. Benchmark 管理

Benchmark 管理是评估平台的基础。

很多团队一开始直接把测试集放在仓库或表格里，后面会遇到很多问题：数据被修改但没人知道，样例重复，标签质量不稳定，训练集和测试集污染，敏感数据权限失控，不同团队用不同版本比较模型。

一个成熟的 Benchmark Registry 至少要支持以下能力。

### 7.1 数据导入

支持多种来源：

- 手工构造测试集。
- 历史线上问题抽样。
- 专家标注样例。
- 公开 benchmark。
- 红队攻击样例。
- 回归 bug 样例。
- 合成数据生成的测试样例。

导入时要做 schema 校验、重复检测、敏感信息扫描和授权检查。

### 7.2 数据分层

评测集不应该只有一个总分，而应按能力维度分层。

例如通用助手可以分为：

- 知识问答。
- 数学推理。
- 代码生成。
- 中文写作。
- 长上下文。
- 指令遵循。
- 安全拒答。
- 工具调用。
- RAG 事实性。
- 多轮对话。

每个维度还可以继续细分。比如代码评估可以分为 Python、SQL、前端、算法题、单元测试修复、代码解释等。

这样的好处是：模型总分下降时，可以定位是哪类能力退化；模型总分上升时，也能看出是否只是某个维度拉高。

### 7.3 隐藏测试集

如果测试集被模型训练或 prompt 调优反复看到，就会发生 benchmark overfitting。平台需要支持 hidden set。

Hidden set 的设计原则是：

- 普通用户不能查看答案。
- 评估 worker 可以读取但不暴露标签。
- 报告只展示聚合结果或脱敏样例。
- 访问需要审计。
- 定期刷新。

这类似考试题库。如果考生提前背过答案，分数就不能代表真实能力。

### 7.4 回归测试集

回归测试集来自历史问题。

例如某个线上模型曾经错误回答“退款政策”，后来修复了。这个样例就应该进入回归测试集。之后每次模型升级都要验证：不能重新犯同样的错误。

回归测试集通常不大，但优先级很高。它代表真实业务事故和已知风险。

### 7.5 数据污染检测

如果评测样例出现在训练集中，模型可能只是记住答案，而不是具备能力。平台可以做污染检测：

- 通过 n-gram 或 embedding 检索训练语料中的相似片段。
- 比较输入、标准答案和训练数据的重合度。
- 标记高风险样例。
- 在报告中区分 clean score 和 contaminated score。

污染检测无法完全解决问题，但可以提高评估可信度。

## 8. 模型版本管理

模型版本管理决定评估是否可复现。

大模型应用中的“版本”不是单一权重文件，而是一组组合配置。

一个生产模型版本可能包括：

- Base model 权重。
- SFT checkpoint。
- RLHF 或 DPO checkpoint。
- LoRA adapter。
- tokenizer。
- system prompt。
- few-shot examples。
- RAG 检索索引版本。
- 工具 schema。
- 安全策略版本。
- 解码参数。
- 后处理规则。

如果其中任何一个变化，模型行为都可能变化。

所以评估平台应该把“待评估对象”抽象为 `EvaluationTarget`，而不是只叫 `Model`。

例如：

```text
EvaluationTarget = ModelWeights
                 + PromptTemplate
                 + InferenceConfig
                 + RetrievalConfig
                 + ToolConfig
                 + SafetyPolicy
                 + PostProcessConfig
```

这样做的好处是：

- 可以评估纯模型 checkpoint。
- 可以评估 prompt 变更。
- 可以评估 RAG 索引更新。
- 可以评估工具调用策略。
- 可以评估安全规则升级。

面试中如果能指出这一点，会比只说“模型版本管理”更有工程深度。

## 9. 评估任务调度

评估任务通常很耗时，需要调度系统。

### 9.1 任务拆分

一次评估可以拆成三层：

- Run：一次完整评估。
- Shard：一个 benchmark 分片。
- Sample Task：单条或小 batch 样例执行。

这样可以并行执行，也方便失败重试。

### 9.2 队列设计

建议使用多队列：

- release queue：上线前评估，优先级最高。
- regression queue：关键回归测试，优先级高。
- research queue：研究实验，优先级中。
- full-benchmark queue：完整大规模评估，优先级低。
- human-review queue：人工评审任务。

调度器需要考虑：

- 优先级。
- 预算。
- 模型 endpoint 限流。
- GPU 资源。
- 评估任务 deadline。
- 数据权限。
- 失败重试。

### 9.3 幂等与断点续跑

评估任务常见失败包括：模型服务超时、GPU 节点故障、外部 API 限流、裁判模型异常、对象存储写入失败。

系统应保证：

- 单条样例可以重试。
- 已成功样例不重复计费或重复覆盖。
- 中断后可以从未完成样例继续。
- 每次重试记录 retry count。
- 如果模型输出非确定性，要记录重试策略。

实现上可以用 `(run_id, sample_id, evaluator_version)` 作为幂等键。

## 10. 自动评估

自动评估是平台吞吐量最高的部分。

### 10.1 精确匹配与规则评估

适合有明确答案的任务：

- 选择题。
- 数学题最终答案。
- JSON schema 校验。
- 工具调用参数校验。
- 分类任务。
- 安全策略是否拒答。

优点是便宜、稳定、可解释。缺点是无法处理开放文本质量。

### 10.2 文本相似度指标

常见指标包括 BLEU、ROUGE、METEOR、BERTScore 等。

它们适合摘要、翻译和复述类任务，但不适合完全开放的对话质量评估。

例如两个回答语义相同但措辞不同，BLEU 可能很低。一个回答复制了参考答案中的关键词，但事实逻辑错误，ROUGE 也可能不低。

所以这些指标可以作为辅助，不能单独作为上线门禁。

### 10.3 代码执行评估

代码生成评估通常要运行测试用例。

流程如下：

1. 模型生成代码。
2. 平台把代码放入沙箱环境。
3. 执行单元测试。
4. 收集通过率、运行时间、内存占用和错误日志。
5. 清理环境。

代码执行评估要特别注意安全隔离：

- 禁止访问宿主机敏感目录。
- 限制网络访问。
- 限制 CPU、内存和运行时间。
- 使用临时容器或沙箱。
- 记录执行日志。

这里讨论的是安全执行评估代码，不涉及绕过沙箱或攻击系统。

### 10.4 工具调用评估

Agent 和工具调用类模型不能只看最终回答，还要看调用轨迹。

评估维度包括：

- 是否选择了正确工具。
- 工具参数是否符合 schema。
- 是否按正确顺序调用。
- 是否错误重复调用。
- 是否使用工具结果生成最终答案。
- 是否在工具失败时合理降级。

例如用户问“查一下北京明天温度并提醒我带伞”，模型需要先调用天气工具，再根据结果决定是否创建提醒。只看最终文本是不够的。

### 10.5 RAG 评估

RAG 系统评估要拆成检索和生成两部分。

检索指标包括：

- Recall@k。
- Precision@k。
- MRR。
- nDCG。
- 覆盖率。

生成指标包括：

- 答案是否被证据支持。
- 是否引用正确文档。
- 是否编造不存在的信息。
- 是否在证据不足时拒答。
- 是否遵守业务口径。

RAG 评估平台要保存检索结果、引用片段、最终回答和证据链。否则出现幻觉时很难判断是检索没召回，还是生成模型没利用证据。

### 10.6 LLM-as-a-judge

LLM 裁判常用于开放文本评估。

它可以判断：

- 回答是否正确。
- 是否遵循指令。
- 是否有幻觉。
- 是否更偏好 A 还是 B。
- 是否安全合规。
- 是否对用户有帮助。

常见模式有两种：

- 单答案打分：给一个回答打 1 到 5 分。
- Pairwise 比较：比较两个模型回答哪个更好。

Pairwise 通常比绝对打分更稳定，因为“哪个更好”比“应该给几分”更容易判断。

但 LLM 裁判有明显缺点：

- 可能偏向更长的回答。
- 可能偏向位置靠前的回答。
- 可能偏向和自己风格相似的回答。
- 对事实性问题也会犯错。
- 成本较高。
- 裁判 prompt 改了以后分数不可直接比较。

因此平台要做校准：

- 随机交换 A/B 顺序。
- 使用多个裁判模型投票。
- 对部分样例引入人工 gold label。
- 记录 judge prompt 版本。
- 输出置信度和解释。
- 定期评估裁判模型本身。

## 11. 人工评估

人工评估成本高，但在很多关键场景不可替代。

### 11.1 什么时候需要人工评估

适合人工评估的场景包括：

- 开放式对话质量。
- 复杂推理正确性。
- 业务口径判断。
- 安全边界判断。
- 品牌语气和用户体验。
- 高价值上线前抽检。
- 自动评估低置信样例。

人工评估不是替代自动评估，而是补足自动评估不能可靠判断的部分。

### 11.2 标注方式

常见标注方式包括：

- Likert 打分：1 到 5 分评价质量。
- Pairwise 偏好：比较 A 和 B 哪个更好。
- 多维度打分：正确性、完整性、可读性、安全性分别打分。
- 错误类型标注：事实错误、拒答错误、格式错误、幻觉、偏见等。
- 仲裁标注：多个评审员不一致时由专家裁决。

对大模型评估来说，pairwise 往往比绝对打分更稳定。因为不同评审员对“4 分”和“5 分”的理解可能不同，但比较两个答案通常更一致。

### 11.3 质量控制

人工评估平台要控制标注质量。

常见方法：

- Gold questions：混入已知答案样例。
- Overlap review：同一条样例分给多个评审员。
- Inter-annotator agreement：计算评审员一致性。
- Reviewer calibration：评审前做培训和校准。
- Expert audit：专家抽查普通评审结果。
- Time anomaly detection：发现过快或异常规律的标注。

人工评估如果没有质量控制，会制造新的噪声。

### 11.4 标注规范

标注规范要具体，不能只写“判断回答好不好”。

一个好的规范要说明：

- 什么算事实错误。
- 什么算未遵循指令。
- 什么算过度拒答。
- 什么算安全风险。
- 多轮对话如何考虑上下文。
- RAG 场景是否必须引用证据。
- 不确定时如何选择。
- 是否允许使用外部资料核查。

评估平台应把标注规范版本和评审结果绑定。规范变化后，历史数据不能直接混用。

## 12. 报告系统

报告不是简单展示平均分。优秀报告要帮助团队做决策。

报告至少包括：

- 总体结论：是否建议上线。
- 与 baseline 对比：提升、持平、退化。
- 分能力维度结果。
- 统计显著性或置信区间。
- 失败案例样例。
- 回归测试结果。
- 安全评估结果。
- 成本和延迟变化。
- 人工评审摘要。
- 风险和建议。

### 12.1 版本对比

最常见报告是 `candidate` 对比 `baseline`。

不要只给一个总分，例如：

```text
model-v12: 82.1
model-v11: 81.7
```

这个信息几乎无法决策。

更好的报告是：

```text
总体：v12 相比 v11 小幅提升，但长上下文和安全拒答存在退化风险。

知识问答：+2.3
数学推理：+1.1
代码生成：+0.4
中文写作：+3.8
长上下文：-2.6
RAG 事实性：-1.9
安全拒答：-3.2
延迟：+8%
单位成本：+5%

建议：不直接全量上线，可先灰度到低风险流量，并修复安全拒答回归。
```

### 12.2 失败案例分析

失败案例比平均分更能推动改进。

平台可以按错误类型聚类：

- 事实幻觉。
- 数学推理错误。
- 没有遵循格式。
- 工具参数错误。
- 证据引用错误。
- 过度拒答。
- 应拒未拒。
- 多轮上下文遗忘。

报告应能点击进入具体样例，查看输入、模型输出、参考答案、评分解释、历史版本对比和人工备注。

### 12.3 趋势报告

长期来看，平台要展示趋势：

- 每个模型版本的能力变化。
- 每个 benchmark 的分数变化。
- 回归 bug 修复率。
- 安全通过率变化。
- 人工偏好胜率变化。
- 线上用户满意度变化。

趋势报告能帮助团队发现系统性问题。例如最近三周中文写作一直提升，但工具调用持续下降，说明训练数据或 prompt 可能偏向写作而牺牲了 agent 能力。

## 13. 回归测试与上线门禁

评估平台的一个重要作用是防止“修一个地方，坏另一个地方”。

### 13.1 触发条件

回归测试可以由以下事件触发：

- 新模型 checkpoint。
- 新 prompt 模板。
- RAG 索引更新。
- 工具 schema 变更。
- 安全策略变更。
- 后处理规则变更。
- 线上事故修复。
- 发布流程触发。

### 13.2 分层回归

不是每次变更都跑全量评估。可以分层：

- Smoke test：几十到几百条，几分钟内完成。
- Critical regression：关键业务和安全样例，几十分钟内完成。
- Standard suite：常规能力评估，数小时完成。
- Full suite：完整评估，可能需要一天或更久。

这样可以平衡速度和覆盖率。

### 13.3 上线门禁

Gatekeeper 可以设置硬门槛和软门槛。

硬门槛示例：

- 安全通过率不能低于 99%。
- 关键回归样例必须全部通过。
- P95 延迟不能超过 baseline 20%。
- 单位成本不能超过预算上限。
- 隐私泄露测试不能有高危失败。

软门槛示例：

- 总体胜率低于 baseline 1% 时需要人工审批。
- 某个非关键能力下降超过 3% 时需要评估负责人确认。
- LLM 裁判和人工评审冲突时进入仲裁。

门禁的核心不是阻碍发布，而是让风险显性化。

## 14. 在线评估与离线评估的关系

评估平台不能只做离线 benchmark。

离线评估适合上线前发现明显问题，但真实用户环境更复杂。上线后还需要在线评估。

在线评估包括：

- A/B 实验。
- 灰度发布。
- 用户反馈。
- 会话成功率。
- 人工客服转接率。
- 任务完成率。
- 追问率。
- 投诉率。
- 安全拦截率。
- 成本和延迟监控。

离线和在线结果可能不一致。

例如模型在 benchmark 上更强，但线上用户满意度下降，可能原因是：

- 回答变长，用户不喜欢。
- 延迟上升。
- 业务口径不一致。
- 对真实脏输入鲁棒性差。
- 安全策略过度拒答。
- RAG 检索在真实数据上表现差。

所以平台应把线上反馈样例回流到离线 benchmark，形成闭环：

```text
线上问题 -> 样例采样 -> 脱敏与标注 -> 回归测试集 -> 模型改进 -> 再评估 -> 灰度上线
```

## 15. 成本优化

评估平台很容易变贵。

成本来源包括：

- 被评估模型推理成本。
- 裁判模型成本。
- GPU 资源。
- 对象存储。
- 人工评审。
- 多版本重复评估。

常见优化手段：

- 分层评估，先跑小集再跑大集。
- 对低风险变更只跑相关 benchmark。
- 缓存相同输入、相同模型配置的输出。
- 对长 benchmark 抽样评估。
- 只对不确定样例调用 LLM 裁判。
- 自动评估通过后减少人工评审量。
- 夜间低峰运行低优先级任务。
- 对高成本任务设置预算和审批。

缓存要谨慎。缓存键必须包含模型版本、prompt 版本、推理参数、benchmark 样例版本和评估器版本。否则可能复用错误结果。

## 16. 安全与权限

评估平台会处理大量敏感内容。

安全设计包括：

- 数据集权限：不同团队只能访问授权 benchmark。
- Hidden set 保护：防止测试集泄露。
- PII 脱敏：线上样例进入评估集前要脱敏。
- 审计日志：记录数据访问、任务创建、结果下载和审批。
- 沙箱执行：代码评估和工具调用必须隔离。
- 结果脱敏：报告中展示失败样例时过滤敏感信息。
- 最小权限：评估 worker 只访问必要数据。
- 加密存储：敏感样例和结果加密。

还要防止评估结果被人为操纵。例如某个团队为了上线模型，故意选择有利 benchmark 或删除失败样例。平台可以通过审批、审计、固定发布套件和 hidden set 降低风险。

## 17. 可观测性

评估平台本身也需要监控。

关键指标包括：

- 任务提交量。
- 任务排队时间。
- 样例执行成功率。
- 评估运行耗时。
- 模型调用失败率。
- 裁判模型失败率。
- 人工评审积压量。
- 平均评估成本。
- 报告生成延迟。
- 门禁失败原因分布。

常见告警包括：

- release 评估任务排队过久。
- 某个模型 endpoint 大量超时。
- 裁判模型输出异常。
- 人工评审 SLA 超时。
- 对象存储写入失败。
- benchmark 数据版本缺失。

评估平台如果不可观测，最终会变成“评估任务卡住了但没人知道”。

## 18. 典型接口设计

可以在面试中给出几个核心 API。

```http

Content-Type: application/json

{
  "target_id": "model-v12-prod-candidate",
  "baseline_id": "model-v11-prod",
```

## 19. 面试答题模板

如果面试中遇到“设计一个大模型评估平台”，可以按以下顺序回答：

1. 先澄清场景：评估的是基础模型、RAG 应用、Agent，还是多模态模型？用于研究对比还是上线门禁？
2. 说明难点：开放文本、多维指标、复现困难、自动评估不可靠、人工评估成本高、离线和在线不一致。
3. 定义核心需求：benchmark 管理、模型版本、评估任务、自动评估、人工评估、报告、回归测试、门禁。
4. 做容量估算：模型版本数、benchmark 大小、token 量、存储量、人工评审量。
5. 画高层架构：API、实验服务、模型注册、benchmark 注册、调度队列、worker、指标引擎、人工评审、报告、门禁、审计。
6. 深入关键模块：benchmark 版本化、EvaluationTarget、LLM judge 校准、回归测试、hidden set、幂等重试。
7. 讨论 trade-off：自动评估便宜但不够准，人工评估准但贵；全量评估覆盖好但慢，分层评估更实用；hidden set 可信但维护成本高。
8. 补充安全和权限：PII 脱敏、hidden set 保护、沙箱执行、审计日志。
9. 最后说明上线闭环：离线评估通过后灰度上线，线上反馈回流为回归测试集。

## 20. 常见追问

### 20.1 如果 LLM 裁判不可靠怎么办

可以回答：

- 不把 LLM 裁判作为唯一依据。
- 使用 pairwise 而不是纯绝对打分。
- 随机交换答案顺序降低位置偏差。
- 使用多个裁判模型投票。
- 对关键样例引入人工评审。
- 定期用 gold set 校准裁判。
- 记录 judge prompt 和模型版本。

### 20.2 如何防止 benchmark 被刷分

可以回答：

- 使用 hidden set。
- 控制测试集访问权限。
- 定期刷新样例。
- 检测训练数据污染。
- 发布套件固定并审计。
- 在线指标和离线指标结合。
- 关注分维度能力，而不是单一总分。

### 20.3 如何评估 RAG 系统

可以回答：

- 把检索和生成拆开评估。
- 检索看 Recall@k、MRR、nDCG。
- 生成看证据支持、引用正确性、幻觉率、拒答合理性。
- 保存检索结果、引用片段和最终回答。
- 线上问题回流到回归测试集。

### 20.4 如何做上线门禁

可以回答：

- 定义 release suite。
- 设置硬门槛和软门槛。
- 安全、隐私、关键回归用硬门槛。
- 非关键能力退化进入人工审批。
- 灰度上线后继续监控在线指标。
- 所有决策保留审计记录。

### 20.5 如何减少评估成本

可以回答：

- 分层评估。
- 低风险变更只跑相关测试集。
- 缓存确定性结果。
- 抽样评估大规模 benchmark。
- 只对疑难样例调用裁判模型。
- 自动评估过滤后再人工评审。
- 设置预算和优先级队列。

## 21. 本章小结

大模型评估平台的核心价值，是把模型质量从“主观感觉”变成“可复现、可比较、可审计、可决策”的工程流程。

设计时要抓住几条主线：

- Benchmark 要版本化、分层管理，并保护 hidden set。
- 模型版本要扩展为 EvaluationTarget，覆盖权重、prompt、RAG、工具和安全策略。
- 自动评估适合规模化，但不能完全替代人工评估。
- LLM-as-a-judge 有用，但必须校准和审计。
- 报告要服务上线决策，而不是只展示平均分。
- 回归测试和门禁是生产级评估平台的关键。
- 线上反馈要回流到离线测试集，形成质量闭环。

面试中，一个强答案不是列出很多指标，而是能说明：为什么大模型评估难，如何保证评估可信，如何支撑模型持续迭代，以及如何把评估结果转化为上线决策。
