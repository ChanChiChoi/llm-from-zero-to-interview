# 第二章：设计 ChatGPT 服务

这一章讨论一个最经典的大模型系统设计题：如何设计一个 ChatGPT 类服务。

很多同学听到这个题，会马上想到“前端、后端、模型 API、数据库”。这只覆盖了表层。真正的 ChatGPT 服务不是简单把用户输入转发给模型，而是一个端到端系统：要支持多轮会话、上下文管理、流式输出、模型路由、安全审核、日志审计、用户反馈、成本控制、灰度发布和持续评估。

本章重点：API、会话、模型服务、streaming、上下文管理、安全过滤、日志和监控。

## 2.1 先明确题目边界

面试官说“设计 ChatGPT 服务”，可能有三种含义。

第一，设计面向用户的聊天产品。

这类题关注用户登录、会话列表、多轮对话、流式输出、历史记录、用户反馈、内容安全和高可用。

第二，设计 OpenAI API 类模型服务。

这类题关注 API key、多租户、配额、模型路由、推理集群、限流、计费、SLA 和开发者体验。

第三，设计企业内部智能助手。

这类题会额外关注企业知识库、权限隔离、审计、数据不出域、合规和管理员控制台。

面试开头可以这样澄清：

我先按面向 C 端或企业用户的 ChatGPT 类聊天服务设计，核心功能包括登录、创建会话、多轮对话、流式返回、历史管理、安全审核、模型推理、日志监控和反馈。如果需要，我可以再扩展到开发者 API、多租户计费或企业知识库。

这个回答的好处是：你给了默认范围，也留了扩展空间。

## 2.2 功能需求

一个基础 ChatGPT 服务至少需要这些功能。

| 模块 | 功能 |
| --- | --- |
| 用户系统 | 注册登录、身份认证、用户等级、配额 |
| 会话系统 | 新建会话、会话列表、消息历史、删除和归档 |
| 聊天接口 | 发送消息、流式返回、停止生成、重试、重新生成 |
| 上下文管理 | 多轮上下文拼接、历史摘要、token budget 控制 |
| 模型服务 | 模型选择、推理调度、batching、版本管理 |
| 安全审核 | 输入审核、输出审核、敏感信息处理、违规策略 |
| 日志反馈 | 请求日志、用户反馈、错误追踪、质量样本采集 |
| 运营管理 | 灰度发布、A/B 测试、公告、限流和降级 |

第一版可以不做插件、联网搜索、文件上传、语音、多模态和 Agent，但架构上要保留扩展点。

## 2.3 非功能需求

非功能需求决定系统设计的取舍。

### 2.3.1 延迟需求

ChatGPT 类服务通常采用流式输出，所以要重点关注：

1. 首 token 延迟，也就是 TTFT。
2. token 生成速度。
3. 端到端完成时间。
4. 流式连接稳定性。

用户对 TTFT 很敏感。如果点击发送后 5 秒没有任何输出，体验会很差；如果 1 秒内开始输出，即使完整回答需要 10 秒，用户也更容易接受。

### 2.3.2 可用性需求

聊天服务可以按 99.9% 或更高可用性设计。要考虑模型服务故障、GPU 资源不足、安全审核服务超时、数据库故障和网络抖动。

### 2.3.3 成本需求

大模型服务成本主要来自 GPU 推理和 token。需要设计配额、模型分层、缓存、长度限制和降级策略。

### 2.3.4 安全需求

ChatGPT 服务必须防止有害输出、隐私泄漏、越权访问、未成年人风险、prompt injection、垃圾请求和滥用。

安全不是上线后再补的模块，而是贯穿输入、上下文、模型、输出和日志。

## 2.4 容量估算

面试中可以做一个简化估算，展示你知道瓶颈在哪里。

假设：

1. DAU 为 100 万。
2. 每个用户每天平均 10 次对话请求。
3. 每日请求 1000 万。
4. 平均分摊 QPS 约 116。
5. 峰值按 10 倍估算，峰值 QPS 约 1160。

再估算 token：

1. 平均输入 1000 token。
2. 平均输出 500 token。
3. 每次请求平均 1500 token。
4. 峰值 QPS 1160。
5. 峰值 token 吞吐约 174 万 token/s。

这个数字不一定要精确，但能说明：ChatGPT 服务的核心瓶颈不是普通 Web QPS，而是模型推理 token 吞吐、GPU 调度和长上下文成本。

存储也要估算。

如果每次请求保存 prompt、response、元数据和审核结果，平均 5KB 到 20KB，1000 万请求每天就是几十 GB 到数百 GB。若还保存完整 trace、检索证据和模型日志，规模会更大。因此日志需要冷热分层、采样、脱敏和保留策略。

## 2.5 高层架构

一个 ChatGPT 服务可以分成以下组件。

| 组件 | 职责 |
| --- | --- |
| Client | Web、App、桌面端、SDK |
| API Gateway | 鉴权、限流、路由、协议转换 |
| User Service | 用户、套餐、配额、权限 |
| Conversation Service | 会话、消息、历史、状态 |
| Chat Orchestrator | 编排一次聊天请求的完整流程 |
| Context Manager | 构造上下文、摘要、token budget |
| Safety Service | 输入审核、输出审核、风险策略 |
| Model Router | 选择模型、区域、版本和降级路径 |
| Inference Service | 模型推理、batching、KV cache、streaming |
| Storage | 用户库、会话库、消息库、日志库 |
| Event Bus | 异步日志、反馈、评估、训练数据流 |
| Monitoring | 性能、质量、安全、成本监控 |

核心链路可以概括为：

用户请求进入 API Gateway，完成鉴权和限流；Chat Orchestrator 读取会话历史，调用 Safety Service 做输入审核；Context Manager 构造 prompt；Model Router 选择模型；Inference Service 流式生成；输出经过安全审核和格式处理后返回用户；日志和反馈异步写入事件系统。

## 2.6 API 设计

面试中不需要写完整 OpenAPI，但要说清关键接口。

### 2.6.1 创建会话

接口可以是：

```text
POST /v1/conversations
```

请求包含用户选择的模型、标题、系统配置或场景类型。返回 conversation_id。

### 2.6.2 发送消息

接口可以是：

```text
POST /v1/conversations/{conversation_id}/messages
```

请求包含用户消息、是否流式返回、模型选择、temperature、max_tokens 等参数。

响应可以有两种模式。

非流式模式一次性返回完整答案，适合短任务和 API 调用。流式模式通过 SSE 或 WebSocket 持续返回增量 token，适合聊天产品。

### 2.6.3 停止生成

接口可以是：

```text
POST /v1/responses/{response_id}/cancel
```

用户点击停止时，服务端要取消推理任务或停止向客户端继续发送。如果底层推理引擎无法立即取消，也要停止下游流式传输并释放可释放资源。

### 2.6.4 重新生成

重新生成不是简单覆盖旧消息。更合理的是保留同一个 user message 下的多个 assistant candidate，便于用户切换，也便于后续做偏好数据。

### 2.6.5 用户反馈

接口可以是：

```text
POST /v1/messages/{message_id}/feedback
```

反馈包括点赞、点踩、原因标签、用户文字说明。反馈不应直接进入训练集，需要经过采样、脱敏、审核和质量筛选。

## 2.7 会话和消息模型

ChatGPT 服务的核心数据对象包括 user、conversation、message、response 和 feedback。

### 2.7.1 Conversation

Conversation 可以包含：

1. conversation_id。
2. user_id。
3. title。
4. created_at、updated_at。
5. model 默认设置。
6. system prompt 或场景配置。
7. 状态，例如 active、archived、deleted。

### 2.7.2 Message

Message 可以包含：

1. message_id。
2. conversation_id。
3. role，例如 system、user、assistant、tool。
4. content。
5. parent_message_id。
6. token_count。
7. safety_status。
8. created_at。

为什么需要 parent_message_id？因为用户可能对同一个问题重新生成多个回答，或者从某个历史节点分叉继续对话。严格线性列表无法表达分支。

### 2.7.3 Response

Response 可以记录一次模型生成的元数据：

1. response_id。
2. model_name 和 model_version。
3. decoding 参数。
4. prompt token 和 completion token。
5. latency、TTFT、tokens/s。
6. finish_reason，例如 stop、length、cancel、safety。
7. trace_id。

这些信息对计费、调试、评估和回滚非常重要。

## 2.8 上下文管理

上下文管理是 ChatGPT 服务的核心难点之一。

模型不能无限记住所有历史。每次请求都要在上下文窗口内选择哪些内容传给模型。

### 2.8.1 基础策略

最简单策略是保留最近 N 轮对话。但这会丢失早期重要信息，也可能浪费 token。

更好的策略是混合使用：

1. 固定 system prompt。
2. 当前用户问题。
3. 最近若干轮原文。
4. 更早历史的摘要。
5. 用户长期偏好或记忆。
6. 必要的外部知识或工具结果。

### 2.8.2 Token budget

Context Manager 要给不同部分分配 token budget。

例如总上下文 16k token，可以粗略分配：

1. system prompt 1k。
2. 当前问题 1k。
3. 最近历史 6k。
4. 历史摘要 2k。
5. 外部证据 4k。
6. 预留输出 2k。

不同任务分配不同。普通聊天可以多留历史，RAG 问答要给证据更多预算，代码任务可能要给代码上下文更多预算。

### 2.8.3 历史摘要

当会话很长时，可以把早期对话总结成摘要。

摘要要保留：

1. 用户目标。
2. 关键事实。
3. 已做决定。
4. 未解决问题。
5. 用户偏好。

摘要也可能出错，所以重要事实最好有结构化记录或引用原文。

### 2.8.4 长期记忆

长期记忆可以提升体验，但风险更高。

需要考虑：

1. 用户是否明确授权。
2. 用户能否查看和删除记忆。
3. 哪些信息不能记。
4. 记忆如何更新和过期。
5. 记忆是否会导致隐私泄漏或错误个性化。

面试中可以说：第一版先不做复杂长期记忆，只做会话内上下文和可控摘要；长期记忆作为后续扩展，并提供用户可见和可删除机制。

## 2.9 模型服务设计

模型服务是 ChatGPT 系统成本和延迟的核心。

### 2.9.1 Model Router

Model Router 根据请求选择模型。

选择因素包括：

1. 用户套餐。
2. 任务类型。
3. 上下文长度。
4. 延迟要求。
5. 成本预算。
6. 模型健康状态。
7. 灰度实验配置。

简单问题可以用小模型，复杂推理用大模型，代码任务用代码模型，高风险请求走更严格安全模型。

### 2.9.2 Inference Service

Inference Service 要支持：

1. 模型加载和热更新。
2. continuous batching。
3. prefill 和 decode 调度。
4. KV cache 管理。
5. 流式 token 输出。
6. 请求取消。
7. 超时和重试。
8. 多模型多版本部署。

### 2.9.3 长短请求隔离

长上下文请求会占用更多 prefill 时间和 KV cache，可能拖慢短请求。可以设计不同队列：短请求队列、长请求队列、高优先级队列、批处理队列。

这样可以避免一个超长请求影响大量普通聊天请求。

### 2.9.4 推理优化

常见优化包括：

1. Continuous batching 提高 GPU 利用率。
2. Paged attention 管理 KV cache。
3. Prefix cache 复用相同 system prompt 或历史前缀。
4. Speculative decoding 降低 decode 延迟。
5. 量化降低显存和成本。
6. 模型蒸馏或小模型路由降低简单请求成本。
7. 输出长度限制防止成本失控。

## 2.10 Streaming 设计

ChatGPT 类产品通常必须支持 streaming。

### 2.10.1 SSE 还是 WebSocket

SSE 简单，适合服务端单向推送 token，浏览器支持较好。WebSocket 支持双向通信，更适合复杂交互，例如语音、实时协作或多路事件。

普通文本聊天第一版可以用 SSE。移动端或复杂实时场景可以考虑 WebSocket。

### 2.10.2 流式链路

流式链路大致是：

1. 客户端发送请求并建立流式连接。
2. 网关转发请求到 Chat Orchestrator。
3. Orchestrator 创建 trace_id 和 response_id。
4. 推理服务生成 token。
5. token 增量通过流式通道返回客户端。
6. 服务端周期性 flush。
7. 完成后发送 finish event。
8. 异步写完整回答和日志。

### 2.10.3 流式输出中的安全问题

输出审核在 streaming 中更难。如果等完整回答生成后再审核，用户就不能实时看到；如果边生成边发，可能有风险内容已经发出。

常见策略包括：

1. 输入先审核，降低明显风险请求。
2. 模型侧使用安全对齐和系统策略。
3. 对生成 token 做轻量实时检测。
4. 对完整输出做最终审核。
5. 高风险场景不直接流式展示，先生成后审核再返回。

不同业务风险不同。普通闲聊可以边生成边审核；高风险医疗、金融、未成年人或企业敏感信息场景要更保守。

### 2.10.4 客户端断开

客户端断开时，服务端要检测连接状态，停止下游传输，并尽可能取消推理任务。如果推理已经无法取消，也要避免继续写大量无用输出。

## 2.11 安全过滤

ChatGPT 服务的安全可以分成输入、上下文、输出和行为四层。

### 2.11.1 输入审核

输入审核识别有害请求、垃圾请求、恶意 prompt、PII、越权意图和异常频率。

输入审核结果不一定只有通过和拒绝，也可以分级：

1. 低风险：正常处理。
2. 中风险：加安全系统提示或限制模型能力。
3. 高风险：拒绝回答。
4. 不确定：走更强审核模型或人工队列。

### 2.11.2 上下文安全

上下文里可能包含用户历史、系统提示、工具结果和外部文档。要避免把不该给当前请求看的内容塞进 prompt。

如果系统支持企业数据或文件上传，还要做权限过滤、文件安全扫描、敏感字段脱敏和租户隔离。

### 2.11.3 输出审核

输出审核检查有害建议、隐私泄漏、仇恨歧视、色情暴力、诈骗、危险操作、医疗金融法律高风险建议等。

输出审核可能带来延迟，所以可以分层：轻量规则和小模型做实时过滤，高风险场景用更强审核模型。

### 2.11.4 滥用防护

滥用防护包括账号风控、IP 限流、API key 配额、异常模式检测、爬虫识别和黑名单。

这里要注意合规边界：系统防护的目标是保护服务稳定性、用户安全和数据安全，不是为了帮助绕过第三方平台限制。

## 2.12 数据存储设计

ChatGPT 服务常见存储包括：

1. 用户数据库。
2. 会话和消息数据库。
3. 配额和计费数据库。
4. 日志和 trace 存储。
5. 反馈数据存储。
6. 模型配置和版本元数据。

### 2.12.1 会话存储

会话和消息可以用关系型数据库或文档数据库。

核心访问模式是：按 user_id 查会话列表，按 conversation_id 查消息历史，按 message_id 查单条消息。

要考虑分页、归档、删除、软删除和用户数据导出。

### 2.12.2 热冷分层

近期会话是热数据，需要低延迟读取。很久以前的历史可以归档到低成本存储。

对于模型上下文构造，不一定每次读取完整历史。可以读取最近消息加摘要。

### 2.12.3 日志存储

日志和 trace 用于排障、评估、风控、成本分析和训练数据候选。

日志要遵守最小化原则：能不存敏感内容就不存；必须存时要脱敏、加密、访问控制和保留期限。

## 2.13 日志、反馈与数据闭环

ChatGPT 服务上线后，要靠数据闭环持续改进。

### 2.13.1 请求日志

请求日志包括：

1. trace_id。
2. user_id 或匿名化用户标识。
3. model_version。
4. prompt token 和 completion token。
5. latency、TTFT、tokens/s。
6. safety status。
7. finish_reason。
8. error code。

### 2.13.2 反馈日志

反馈包括点赞、点踩、重新生成、复制、继续追问、用户举报、人工标注。

这些信号噪声很大。比如用户复制不一定代表答案正确，继续追问可能代表感兴趣，也可能代表没回答清楚。所以反馈需要结合上下文和抽样人工审核。

### 2.13.3 训练数据闭环

用户数据不能直接进入训练集。更安全的流程是：采样、脱敏、去重、风险过滤、质量筛选、授权检查、人工标注或偏好构造，再进入训练或评估数据池。

面试中主动说这一点很重要，因为它体现了隐私和合规意识。

## 2.14 监控与告警

ChatGPT 服务的监控要分四类。

### 2.14.1 性能监控

包括 QPS、并发连接数、TTFT、端到端延迟、tokens/s、错误率、超时率、队列长度、GPU 利用率、显存使用和 batch size。

### 2.14.2 质量监控

包括用户反馈率、点踩率、重试率、重新生成率、自动评估分、人工抽检分、事实性错误率、上下文遗忘率。

### 2.14.3 安全监控

包括输入违规率、输出违规率、拒答率、过度拒答率、PII 检测命中、滥用账号数、prompt injection 命中。

### 2.14.4 成本监控

包括每个模型的 token 用量、每个租户成本、每个功能成本、缓存命中率、平均每请求成本和异常高成本请求。

## 2.15 高可用与降级

ChatGPT 服务常见故障包括：

1. 推理集群过载。
2. 某个模型版本异常。
3. 安全审核服务超时。
4. 会话数据库变慢。
5. 流式连接大量断开。
6. 某个区域 GPU 不可用。
7. 日志系统积压。

对应策略包括：

1. 多模型副本和多可用区部署。
2. 限流和排队。
3. 降级到小模型或旧版本模型。
4. 临时缩短最大输出长度。
5. 对免费用户降低优先级，对付费用户保障 SLA。
6. 安全服务失败时采用保守策略。
7. 日志异步化，避免阻塞主链路。
8. 快速回滚模型版本和 prompt 配置。

要注意：降级不能牺牲安全底线。不能因为审核服务慢，就完全跳过高风险审核。

## 2.16 成本优化

成本优化可以从请求前、请求中、请求后三个阶段做。

### 2.16.1 请求前

1. 用户配额和套餐限制。
2. 最大输入长度限制。
3. 重复请求识别。
4. 简单问题路由到小模型。

### 2.16.2 请求中

1. Prompt 压缩。
2. 历史摘要。
3. Prefix cache。
4. Continuous batching。
5. Speculative decoding。
6. 动态 max_tokens。

### 2.16.3 请求后

1. 热门问答缓存。
2. 用户反馈驱动路由策略。
3. 失败请求分析。
4. 高成本请求审计。

成本优化的目标不是简单省钱，而是在质量、延迟和安全可接受的前提下降低单位成功请求成本。

## 2.17 灰度发布和 A/B 测试

ChatGPT 服务变化频繁，包括模型版本、system prompt、安全策略、路由策略、上下文策略和 UI 交互。

不能直接全量上线。

灰度发布需要：

1. 按用户、租户、地区或流量比例灰度。
2. 保留旧版本回滚。
3. 监控质量、延迟、安全和成本。
4. 对关键指标设置自动熔断。
5. 对高风险变更先做离线评估。

A/B 测试要避免只看用户停留时间。大模型产品还要看事实性、安全、拒答率、用户投诉和成本。

## 2.18 面试答题模板

如果面试官问“设计 ChatGPT 服务”，可以这样回答。

第一，澄清需求。

我先按面向用户的多轮聊天服务设计，支持登录、会话管理、发送消息、流式输出、历史记录、安全审核、模型推理、日志反馈和监控。第一版不展开插件、联网搜索和多模态，但架构保留扩展点。

第二，定义指标。

核心指标包括 TTFT、tokens/s、端到端延迟、QPS、可用性、用户反馈、违规率、拒答率和单请求成本。

第三，容量估算。

根据 DAU、每人请求数、峰值倍数、平均输入输出 token 估算峰值 QPS 和 token 吞吐，再估算会话和日志存储。

第四，画架构。

系统包括 Client、API Gateway、User Service、Conversation Service、Chat Orchestrator、Context Manager、Safety Service、Model Router、Inference Service、Storage、Event Bus 和 Monitoring。

第五，讲请求链路。

用户发送消息后，网关鉴权限流，编排服务读取历史，输入审核，上下文管理构造 prompt，模型路由选择模型，推理服务流式生成，输出审核后返回，日志和反馈异步处理。

第六，展开关键难点。

重点展开上下文管理、streaming、推理调度、安全过滤、日志闭环和成本优化。

第七，讲高可用和扩展。

通过多副本、限流、降级、队列隔离、模型版本回滚、异步日志、灰度发布和 A/B 测试保证稳定迭代。

## 2.19 常见面试追问

### 2.19.1 为什么 ChatGPT 服务要流式输出？

因为模型生成可能需要较长时间。流式输出能降低用户感知延迟，让用户尽快看到首 token，同时支持中途停止生成。它不一定降低总计算量，但能显著改善交互体验。

### 2.19.2 SSE 和 WebSocket 怎么选？

普通文本聊天中，服务端向客户端单向推 token，SSE 更简单。需要双向实时交互、语音、多路事件或复杂状态同步时，WebSocket 更合适。

### 2.19.3 如何处理超长对话？

不能无限塞历史。可以保留最近若干轮，早期历史做摘要，重要事实结构化存储，并根据 token budget 选择上下文。必要时让用户开启新会话或明确选择相关历史。

### 2.19.4 用户点击停止生成时服务端怎么做？

客户端发送 cancel 请求或断开连接，服务端停止向客户端推送，并尽可能取消推理任务、释放资源和记录 finish_reason。如果底层任务无法立即取消，也要避免继续占用上层连接。

### 2.19.5 如何避免模型泄露用户隐私？

要从多层做：输入和日志脱敏，用户数据访问控制，训练数据授权和过滤，输出 PII 检测，会话隔离，租户隔离，敏感操作审计。不能只靠模型自己“不泄露”。

### 2.19.6 如何降低首 token 延迟？

可以优化网关和编排链路，减少同步依赖；使用 prompt cache；缩短输入上下文；优化 prefill；做好模型 warmup；使用高优先级队列；必要时用更小模型或区域就近路由。

### 2.19.7 如何做模型版本升级？

先离线评估，再小流量灰度，监控质量、延迟、安全和成本指标。保留旧版本回滚。会话中最好记录 model_version，便于复现问题和解释不同回答。

### 2.19.8 日志能不能直接用于训练？

不能直接用。日志可能包含隐私、噪声、有害内容和版权风险。需要授权检查、脱敏、去重、质量筛选、安全过滤和人工标注，才能进入训练或评估数据池。

## 2.20 本章小结

设计 ChatGPT 服务，本质上是设计一条稳定、低延迟、可控、可监控的多轮模型交互链路。

本章要记住几句话：

1. ChatGPT 服务不是简单转发模型 API，而是会话、上下文、推理、安全、日志和反馈的完整系统。
2. 指标要同时覆盖 TTFT、tokens/s、质量、安全、可用性和成本。
3. 容量估算要看 token 吞吐，而不只是 QPS。
4. 会话模型要支持历史、分支、重新生成和模型版本记录。
5. 上下文管理要控制 token budget，不能无限塞历史。
6. Streaming 改善用户感知延迟，但会让输出审核更复杂。
7. 安全要覆盖输入、上下文、输出、日志和滥用防护。
8. 数据闭环要经过授权、脱敏、筛选和评估，不能直接把用户日志拿去训练。

如果面试中被问到这道题，最好的回答不是堆组件，而是讲清：一次用户消息从进入系统到流式返回，中间经过哪些控制点，每个控制点如何保证体验、质量、安全和成本。
