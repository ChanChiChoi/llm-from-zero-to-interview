# 第十四章：系统设计面试题

系统设计面试是大模型算法岗、平台岗、应用岗和研究工程岗都绕不开的一类题。它考察的不是你能不能背出某个架构图，而是你能不能在一个开放问题里，主动澄清边界、拆解需求、识别核心矛盾、提出可落地方案，并能解释方案背后的取舍。

大模型系统设计题尤其容易答散。因为它同时涉及模型、推理、训练、数据、RAG、Agent、安全、评估、成本、延迟和产品体验。很多候选人一听到“设计 ChatGPT”就开始堆模块：网关、模型、数据库、缓存、消息队列、向量库。这样看似完整，实际缺少主线。

本章的目标是给你一套可迁移的答题方法，并用高频题练习如何组织答案。

高频问题：设计 ChatGPT、设计推理平台、设计 RAG、设计实时语音助手、设计评估平台。

合规边界：本章讨论合法合规的大模型系统设计、平台架构、工程治理和面试表达，不提供绕过安全审核、规避权限控制、窃取数据、滥用模型能力或攻击线上系统的方法。

## 14.1 系统设计面试到底考什么

系统设计面试不是让你画最复杂的架构，而是考察五类能力。

第一，抽象能力。

面试官给你的题通常很宽。例如“设计一个 ChatGPT”“设计一个 RAG 系统”。你需要把模糊题目变成明确系统：面向谁、解决什么问题、什么规模、什么 SLA、哪些功能先做、哪些暂不做。

第二，拆解能力。

一个大系统不能直接回答，必须拆成模块。例如 ChatGPT 可以拆成会话层、模型推理层、上下文管理、安全审核、工具调用、存储、监控、评估和反馈闭环。

第三，权衡能力。

系统设计没有唯一答案。大模型系统尤其充满 trade-off：质量 vs 成本、延迟 vs 准确率、缓存命中 vs 隐私、召回率 vs 噪声、安全拦截 vs 误杀、强模型 vs 小模型、同步体验 vs 异步可靠性。

第四，工程落地能力。

你不能只说“用一个模型解决”。要讲清数据怎么流、服务怎么调用、状态怎么存、失败怎么处理、如何监控、如何灰度、如何回滚。

第五，演进能力。

真实系统不会一步到位。一个优秀答案通常会分阶段：第一版先打通核心链路，第二版补齐质量和观测，第三版做成本优化和自动化闭环。

## 14.2 标准答题框架

遇到任何大模型系统设计题，都可以用下面框架。

### 14.2.1 第一步：澄清需求

先问边界，不要急着画图。

可以问：

1. 面向 C 端用户、企业内部用户，还是开发者 API？
2. 支持文本，还是还要支持图片、语音、视频？
3. 是否需要 RAG、工具调用、多轮对话？
4. 规模大概是多少 QPS、多少 DAU、多少文档量？
5. 更关注质量、延迟、成本、安全，还是可用性？
6. 是否有多租户、权限隔离和合规要求？
7. 是设计 MVP，还是设计可支撑大规模上线的平台？

如果面试官不补充，你可以自己合理假设。

### 14.2.2 第二步：列功能需求和非功能需求

功能需求描述系统要做什么。

非功能需求描述系统要做到什么程度。

例如设计 RAG：

功能需求包括文档上传、解析、切分、embedding、检索、rerank、生成、引用、反馈。

非功能需求包括低延迟、高召回、权限隔离、可更新、可观测、可评估、可扩展。

### 14.2.3 第三步：画高层架构

不要一开始陷入数据库字段。先给出高层数据流。

常见结构是：

```text
Client -> Gateway -> Orchestrator -> Core Services -> Model/Storage -> Observability
```

对于大模型系统，可以进一步展开为：

```text
Client
  -> API Gateway
  -> Request Analyzer
  -> Safety / Policy
  -> Orchestrator
  -> Retrieval / Tool / Cache
  -> Model Serving
  -> Output Guard
  -> Response
  -> Logs / Metrics / Evaluation
```

### 14.2.4 第四步：深入核心模块

每道题都有不同核心。

设计 ChatGPT，核心是会话、上下文、模型推理、安全、流式输出。

设计推理平台，核心是调度、批处理、KV cache、模型版本、GPU 利用率、SLA。

设计 RAG，核心是文档处理、检索、rerank、上下文拼接、引用和权限。

设计语音助手，核心是 ASR、流式理解、低延迟、打断、TTS 和状态管理。

设计评估平台，核心是数据集、评测任务、指标、裁判模型、人工评审和回归测试。

### 14.2.5 第五步：讲扩展、失败和权衡

面试官通常会追问：

1. 流量变大怎么办？
2. 模型变慢怎么办？
3. 结果不准怎么办？
4. 数据泄露怎么办？
5. 成本太高怎么办？
6. 如何灰度新模型？
7. 如何评估效果？

你应该主动覆盖这些点，而不是等面试官不断提示。

## 14.3 大模型系统设计的通用架构积木

很多题看起来不同，底层积木相似。

### 14.3.1 接入层

接入层负责认证、限流、路由、租户识别、请求大小限制、基础日志。

常见组件：API Gateway、Load Balancer、Rate Limiter、Auth Service。

大模型系统里，接入层还要关注 token 级成本，因为一个请求的成本不是只看 QPS，还要看输入输出 token 数。

### 14.3.2 编排层

编排层决定一次请求怎么执行。

它可能要调用安全审核、RAG、工具、模型、缓存和后处理。

编排层是很多大模型应用的核心，因为大模型不再只是一个单函数调用，而是一条动态工作流。

### 14.3.3 模型服务层

模型服务层负责推理。

需要考虑：模型版本、GPU 调度、batching、KV cache、量化、并发、流式输出、超时、fallback。

### 14.3.4 数据与存储层

不同系统需要不同存储。

会话系统需要消息存储。

RAG 需要对象存储、元数据数据库、向量索引、倒排索引。

评估平台需要数据集版本、实验结果、模型输出和人工标注。

### 14.3.5 安全与策略层

包括输入审核、输出审核、权限控制、PII 检测、工具调用审核、策略引擎和审计日志。

大模型系统的安全不是外挂模块，而是贯穿请求全链路。

### 14.3.6 观测与评估层

包括 metrics、logs、traces、成本统计、用户反馈、离线评测、线上 A/B、红队回归。

没有评估和观测，就无法判断模型变更、prompt 变更和路由策略是否真的有效。

## 14.4 高频题一：设计 ChatGPT 类服务

### 14.4.1 题目澄清

可以先这样说：

```text
我先按一个文本为主的 ChatGPT 类服务设计，支持多轮对话、流式输出、会话管理、基础 RAG、工具调用、安全审核、用户反馈和模型版本灰度。目标是支持 C 端或企业用户的在线问答，重点关注低延迟、高可用、安全和成本控制。
```

### 14.4.2 需求

功能需求：

1. 用户注册登录和会话管理。
2. 多轮对话。
3. 流式输出。
4. 上下文管理和历史压缩。
5. 模型推理。
6. 安全审核。
7. 可选 RAG 和工具调用。
8. 用户反馈。
9. 管理后台和模型灰度。

非功能需求：

1. p95 延迟可控。
2. 服务高可用。
3. 多租户隔离。
4. 成本可观测。
5. 数据隐私保护。
6. 支持模型迭代和回滚。

### 14.4.3 高层架构

```text
Web/App
  -> API Gateway
  -> Auth / Rate Limit
  -> Conversation Service
  -> Request Orchestrator
  -> Safety Input Guard
  -> Context Builder
  -> Retrieval / Tool Router / Cache
  -> Model Router
  -> Model Serving
  -> Safety Output Guard
  -> Stream Response
  -> Logs / Metrics / Evaluation
```

### 14.4.4 核心模块

会话服务负责保存 messages、session metadata、用户偏好、上下文摘要。

上下文构建器负责从历史消息中选择哪些内容进入 prompt。不能无限塞历史，因为上下文窗口有限、成本有限。

模型路由根据任务难度、用户等级、上下文长度和实时负载选择模型。

安全审核在输入和输出两侧都要做。输入审核拦截高风险请求，输出审核避免模型生成违规、隐私泄露或业务不允许内容。

流式输出提升体感延迟。即使完整回答要 10 秒，用户看到首 token 在 1 秒内出现，体验会好很多。

反馈系统记录点赞、点踩、重试、复制、追问和人工标注，用于后续模型和 prompt 优化。

### 14.4.5 关键权衡

上下文越长，模型理解越充分，但成本和延迟越高。需要摘要、检索和重要性选择。

安全审核越严格，风险越低，但误杀可能增加。需要策略分级和申诉反馈。

模型越强，质量越好，但成本越高。需要大小模型路由和缓存。

流式输出体验好，但输出审核更复杂。可以做分段审核、风险类别预判和高风险场景非流式。

### 14.4.6 高频追问

如何处理超长历史？

可以结合滑动窗口、会话摘要、重要消息保留、用户显式 pin、RAG 检索历史和分层记忆。不要简单截断最后 N 条，因为早期关键约束可能丢失。

如何降低成本？

使用大小模型路由、prompt cache、response cache、输出长度控制、上下文压缩、批处理、量化和热点问题预缓存。

如何灰度新模型？

按用户、租户、任务类型或流量比例灰度。对比满意度、追问率、延迟、成本、安全拦截和人工评审结果，异常时回滚。

## 14.5 高频题二：设计大模型推理平台

### 14.5.1 题目澄清

```text
我先按一个服务多业务、多模型的在线推理平台设计。平台支持模型部署、版本管理、流式推理、动态 batching、GPU 调度、路由、限流、监控、灰度和弹性伸缩。目标是提高 GPU 利用率，同时满足不同业务 SLA。
```

### 14.5.2 需求

功能需求：

1. 模型注册和版本管理。
2. 在线部署和下线。
3. 推理 API。
4. 流式输出。
5. 多模型路由。
6. 动态 batching。
7. 资源调度。
8. 灰度发布和回滚。
9. 监控和告警。

非功能需求：

1. 高吞吐。
2. 低尾延迟。
3. GPU 高利用率。
4. 故障隔离。
5. 多租户配额。
6. 成本可控。

### 14.5.3 高层架构

```text
Client / Business Service
  -> Inference Gateway
  -> Auth / Quota / Rate Limit
  -> Model Router
  -> Scheduler
  -> Model Runtime Workers
  -> GPU Cluster
  -> Metrics / Logs / Traces
  -> Model Registry / Config Center
```

### 14.5.4 核心设计

模型注册中心保存 model_id、version、artifact path、tokenizer、runtime config、resource requirement、supported features。

调度器根据模型、队列长度、SLA、GPU 资源和 batch 策略把请求分配给 worker。

动态 batching 把多个请求合并推理，提高吞吐，但会增加等待时间。需要按 SLA 设定最大等待窗口。

KV cache 管理影响长上下文和多轮对话性能。需要考虑显存占用、会话生命周期、淘汰策略和模型版本绑定。

模型服务 worker 负责加载权重、执行推理、流式返回 token、上报 metrics。

### 14.5.5 关键权衡

batch 越大，吞吐越高，但单请求等待更久。

模型副本越多，可用性越高，但显存成本越高。

量化可以降低成本和延迟，但可能影响质量。

多租户共享 GPU 提高利用率，但隔离和公平性更难。

### 14.5.6 高频追问

如何处理热点模型？

增加副本、独立队列、优先级调度、缓存热门 prompt、限制低优先级流量，并根据 p95 延迟自动扩缩容。

如何保证高优先级业务？

使用租户配额、优先级队列、资源预留、限流和降级策略。关键业务可以独占模型副本。

如何定位延迟问题？

拆分网关延迟、排队延迟、prefill 延迟、decode 延迟、网络延迟和输出审核延迟，用 trace 串起来分析。

## 14.6 高频题三：设计 RAG 系统

### 14.6.1 题目澄清

```text
我先按一个企业知识库 RAG 系统设计。系统支持文档上传、解析、切分、索引、权限控制、混合检索、rerank、答案生成、引用溯源、反馈和知识库更新。目标是让用户基于企业私有文档问答，并保证答案可追溯、权限安全和更新及时。
```

### 14.6.2 需求

功能需求：

1. 文档上传和管理。
2. 文档解析和清洗。
3. chunk 切分。
4. embedding 和索引。
5. 向量检索和关键词检索。
6. rerank。
7. prompt 构造。
8. 答案生成和引用。
9. 权限过滤。
10. 反馈和评估。

非功能需求：

1. 检索召回高。
2. 答案事实性强。
3. 引用可追溯。
4. 文档更新及时。
5. 多租户隔离。
6. 延迟可控。

### 14.6.3 高层架构

```text
Document Pipeline:
Upload -> Parse -> Clean -> Chunk -> Embed -> Index -> Version

Query Pipeline:
Query -> Rewrite -> Permission Filter -> Hybrid Retrieve -> Rerank -> Context Builder -> LLM -> Citation -> Feedback
```

### 14.6.4 核心设计

文档解析要处理 PDF、Word、HTML、Markdown、表格、图片 OCR。解析质量直接影响检索质量。

chunk 切分不能只按固定长度。需要考虑标题层级、段落、表格、代码块、语义边界和引用位置。

混合检索通常比纯向量检索更稳。向量检索擅长语义相似，BM25 擅长关键词、编号、术语和精确匹配。

rerank 用更强模型对候选文档重排，提高最终上下文质量。

权限过滤必须在检索链路中生效。用户不能看到没有权限的文档片段，也不能通过答案间接获得私有信息。

引用溯源让用户知道答案来自哪些文档、哪一段、哪个版本。

### 14.6.5 关键权衡

chunk 太小，上下文不完整；chunk 太大，检索噪声高且占用 prompt。

top-k 太小容易漏召回，top-k 太大增加噪声和成本。

强 reranker 质量好但延迟高。

答案生成越自由，可读性越好，但越容易幻觉；引用约束越强，事实性越好但表达可能受限。

### 14.6.6 高频追问

如何减少幻觉？

提高检索质量，要求答案基于引用生成，对无证据问题回答不知道，做引用一致性检查，加入离线评估和用户反馈。

文档更新后怎么办？

维护知识库版本，增量解析和索引，失效 retrieval cache 和 response cache，保留历史版本用于审计。

如何评估 RAG？

评估检索 recall、MRR、NDCG，生成答案的 faithfulness、answer correctness、citation accuracy，以及端到端用户满意度。

## 14.7 高频题四：设计实时语音助手

### 14.7.1 题目澄清

```text
我先按一个实时语音助手设计，支持用户语音输入、流式 ASR、对话理解、LLM 生成、工具调用、流式 TTS、打断和多轮上下文。目标是低延迟、自然交互和稳定可用。
```

### 14.7.2 需求

功能需求：

1. 音频采集和传输。
2. VAD 判断用户是否在说话。
3. 流式 ASR。
4. 对话状态管理。
5. LLM 生成。
6. 工具调用。
7. 流式 TTS。
8. 支持用户打断。
9. 安全审核和日志。

非功能需求：

1. 低首响延迟。
2. ASR 准确率高。
3. TTS 自然。
4. 支持弱网。
5. 会话状态可靠。
6. 隐私保护。

### 14.7.3 高层架构

```text
Client Audio
  -> Streaming Gateway
  -> VAD
  -> Streaming ASR
  -> Dialogue Manager
  -> LLM / Tool Orchestrator
  -> Streaming TTS
  -> Audio Playback
  -> Monitoring / Evaluation
```

### 14.7.4 核心设计

VAD 决定什么时候开始和结束一轮用户输入。VAD 太敏感会误触发，太迟钝会增加延迟。

流式 ASR 边听边转写，可以提前把 partial transcript 送给对话管理器。

LLM 可以在用户话音结束前做意图预判，但最终回答需要等 ASR 稳定结果。

TTS 要支持流式合成，让用户尽快听到回复。

打断机制很关键。当用户开始说话时，系统要停止当前 TTS，取消或暂停 LLM 生成，并更新对话状态。

### 14.7.5 关键权衡

越早开始生成，延迟越低，但 ASR 尚未稳定，误解风险更高。

TTS 分块越小，首响越快，但语调自然度可能下降。

保留完整音频有利于质检，但隐私风险更高，需要合规策略。

### 14.7.6 高频追问

如何降低端到端延迟？

使用流式 ASR、意图预判、短 prompt、模型路由、流式 TTS、并行安全检查、边生成边播报和就近部署。

如何处理用户打断？

客户端检测到新语音后发送 interruption event，服务端停止 TTS 播放、取消生成任务、记录已播放内容，并把新输入接入当前会话状态。

如何评估语音助手？

看 ASR WER、首响延迟、端到端延迟、打断成功率、任务完成率、用户满意度、TTS 自然度和安全指标。

## 14.8 高频题五：设计评估平台

### 14.8.1 题目澄清

```text
我先按一个大模型评估平台设计，支持数据集管理、模型输出生成、自动指标、LLM-as-judge、人工评审、红队评测、回归测试、A/B 实验和报告生成。目标是支撑模型、prompt、RAG、路由策略和安全策略的持续迭代。
```

### 14.8.2 需求

功能需求：

1. 评测数据集管理。
2. 数据集版本控制。
3. 模型和 prompt 版本管理。
4. 批量生成。
5. 自动指标计算。
6. 裁判模型评分。
7. 人工评审。
8. 红队和安全评测。
9. 回归测试。
10. 报告和看板。

非功能需求：

1. 可复现。
2. 可扩展。
3. 成本可控。
4. 评审一致性高。
5. 支持多业务线。
6. 支持审计。

### 14.8.3 高层架构

```text
Dataset Registry
  -> Evaluation Job Scheduler
  -> Model Runner
  -> Metric Calculator
  -> Judge Model Service
  -> Human Review Tool
  -> Result Store
  -> Report Dashboard
  -> Regression Gate
```

### 14.8.4 核心设计

数据集要有版本、来源、任务类型、标签、难度、领域、语言和权限信息。

评测任务要记录模型版本、prompt 版本、解码参数、工具版本、知识库版本和运行时间，否则结果不可复现。

自动指标适合有标准答案的任务，例如分类准确率、BLEU、ROUGE、EM、F1、代码测试通过率。

开放生成任务需要 LLM-as-judge 和人工评审结合。裁判模型要有 rubric，不能只给一个模糊分数。

回归测试用于防止新模型在旧能力上倒退。上线前可以设置门禁，例如安全分不得下降、核心任务胜率不得低于基线。

### 14.8.5 关键权衡

自动评测便宜快，但可能和人类偏好不一致。

人工评审质量高，但成本高、速度慢、一致性需要控制。

裁判模型可扩展，但可能有偏差，需要校准和抽样复核。

评测集太固定会被过拟合，评测集太动态又难以比较历史结果。

### 14.8.6 高频追问

如何避免评测污染？

评测集分公开集、内部保密集、动态集和线上真实样本集。训练数据和评测数据做污染检测，核心评测集限制访问，报告只展示聚合结果。

如何评估 LLM-as-judge 是否可靠？

用人工标注样本校准，计算和人工偏好的一致率，检查位置偏差、长度偏差、模型偏好，并定期抽样复核。

如何支持上线门禁？

把评测平台接入发布流程。模型或 prompt 变更必须跑核心回归集，达到质量、安全、延迟和成本阈值后才能灰度。

## 14.9 其他常见系统设计题简表

| 题目 | 核心考点 |
| --- | --- |
| 设计 Agent 平台 | 工具注册、规划执行、权限、状态、失败恢复、审计 |

| 设计多模态助手 | 图片理解、文档解析、跨模态检索、安全、延迟 |

## 14.10 面试中的表达技巧

### 14.10.1 不要只堆组件

很多候选人会说：

```text
用 API Gateway、Redis、Kafka、MySQL、向量数据库、Kubernetes、Prometheus。
```

这不是系统设计，只是组件列表。

更好的表达是：

```text
我把系统分成在线请求链路和离线优化链路。在线链路负责低延迟响应，所以核心是网关、编排、检索、模型推理和安全审核。离线链路负责文档索引、评估、反馈分析和策略更新。缓存放在检索、embedding 和模型响应几个位置，但会按租户和版本隔离。
```

### 14.10.2 主动说明假设

如果面试官没有给规模，你可以说：

```text
我先假设日活 100 万，峰值 1000 QPS，主要是文本请求，p95 延迟目标 5 秒以内。如果规模更大，后面可以扩展调度和缓存设计。
```

假设不是拍脑袋，而是为了让设计有边界。

### 14.10.3 先 MVP，后演进

一个优秀答案可以分阶段：

第一阶段，打通核心链路。

第二阶段，补齐监控、评估、安全、灰度。

第三阶段，做成本优化、自动路由、数据闭环。

这种表达比一次性画终极架构更真实。

### 14.10.4 用指标支撑设计

不要只说“提高质量”“降低成本”。要说具体指标。

例如：

1. 延迟：p50、p95、p99、首 token 延迟。
2. 成本：每请求成本、每百万 token 成本、每用户日成本。
3. 质量：任务成功率、人工胜率、用户满意度、追问率。
4. RAG：检索 recall、citation accuracy、faithfulness。
5. 安全：拦截率、误杀率、红队通过率。
6. 稳定性：错误率、超时率、可用性。

## 14.11 常见扣分点

第一，只讲模型，不讲系统。

大模型系统设计不是“调用一个 GPT API”。需要考虑会话、缓存、存储、安全、评估、监控、成本和发布。

第二，只讲在线链路，不讲离线链路。

RAG 需要文档索引，评估需要数据集，路由需要日志分析，安全需要红队数据。离线链路决定系统能否长期变好。

第三，忽略权限和隐私。

企业 RAG、会话缓存、日志、评估样本都可能包含敏感信息。权限隔离和数据治理必须主动讲。

第四，忽略失败场景。

模型超时、检索为空、向量库故障、ASR 错误、缓存误命中、安全误杀都要有兜底。

第五，不讲评估。

没有评估，就不知道系统是否变好。大模型系统尤其需要离线评测、线上反馈和 A/B。

第六，不讲成本。

大模型系统的成本通常是核心约束。token、GPU、embedding、rerank、存储、人工审核都要考虑。

## 14.12 一套万能收尾

系统设计题最后可以用一段总结收束答案：

```text
总结一下，我会先用一个 MVP 打通核心链路，保证用户请求能经过认证、编排、模型推理、安全审核和日志记录。然后补齐缓存、监控、评估、灰度和回滚，保证系统可运营。最后通过模型路由、RAG 优化、离线评测和用户反馈闭环持续降低成本、提升质量。整个设计的核心权衡是质量、延迟、成本、安全和可维护性之间的平衡。
```

这段话适合多数大模型系统设计题，因为它强调了分阶段、可运营和 trade-off。

## 14.13 最后一页速记

面试前可以记住这张清单。

澄清问题：

1. 用户是谁？
2. 输入输出是什么？
3. 是否多轮、RAG、工具、多模态？
4. 规模和 SLA 是什么？
5. 安全、隐私和成本要求是什么？

架构主线：

```text
Gateway -> Analyzer -> Policy/Safety -> Orchestrator -> Retrieval/Tool/Cache -> Model Serving -> Guard -> Telemetry/Eval
```

核心 trade-off：

1. 质量 vs 成本。
2. 延迟 vs 准确率。
3. 缓存命中率 vs 隐私和正确性。
4. 召回率 vs 噪声。
5. 自动化 vs 人工审核。
6. 强模型 vs 小模型。
7. 快速迭代 vs 稳定回归。

必讲能力：

1. 安全审核。
2. 权限隔离。
3. 监控指标。
4. 评估体系。
5. 灰度回滚。
6. 成本优化。
7. 故障降级。

如果你能在每道题中都自然覆盖这些点，系统设计面试就不会停留在“背架构图”，而会体现出你真正理解大模型系统如何上线、运行和持续演进。
