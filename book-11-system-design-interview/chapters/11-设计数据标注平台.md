# 第十一章：设计数据标注平台

面试中如果让你“设计一个数据标注平台”，千万不要只回答一个表单系统：用户上传数据，标注员打标签，管理员导出结果。这个答案太浅，因为大模型时代的数据标注已经不只是给图片框框、给文本分类，而是要支撑 SFT、RLHF、DPO、RAG 评测、多模态理解、安全对齐、工具调用和模型评估等多个环节。

一个好的数据标注平台，本质上是大模型数据闭环里的“生产车间”和“质检系统”。它负责把原始数据、模型输出、人工偏好、专家判断和质量审计组织成可追踪、可复用、可迭代的数据资产。

本章重点：任务分发、标注规范、质量控制、一致性检查、数据版本和隐私。

合规边界：本章讨论合法合规的数据标注、隐私保护、质量管理和平台架构，不提供规避隐私保护、泄露个人敏感信息、绕过审核或伪造标注结果的方法。

## 11.1 先明确题目边界

“数据标注平台”这个题目很宽。面试时第一步要澄清它服务什么业务。

常见范围包括：

1. 传统监督学习标注：文本分类、实体识别、图片分类、目标检测、语音转写。
2. 大模型 SFT 标注：指令、回答、对话、工具调用轨迹、多轮改写。
3. 偏好标注：两个或多个模型回答之间做 pairwise ranking 或打分。
4. 安全标注：判断输入或输出是否涉及毒性、隐私、越权、违法、歧视、危险建议。
5. 评估标注：人工评审模型输出的正确性、有用性、事实性、风格和安全性。
6. 多模态标注：图像描述、视频片段标注、语音情绪、OCR、视觉问答。

面试开头可以这样限定：

```text
我先按一个面向大模型训练和评估的数据标注平台来设计，支持文本、多轮对话、偏好比较和安全审核。平台要支持任务创建、样本管理、标注规范、任务分发、人工标注、模型预标注、质量控制、数据版本、权限隔离和隐私保护。图片、语音、视频标注作为后续扩展。
```

这个边界比较合适，因为它覆盖大模型算法岗最常见的数据闭环，又不会一上来陷入所有模态、所有行业、所有权限系统的无限复杂度。

## 11.2 为什么大模型标注平台比传统标注平台更难

传统标注平台通常处理的是“输入到标签”的映射。例如一张图片是不是猫，一句话属于什么情感，一个实体是不是人名。这类任务虽然也有难度，但标签空间相对固定，评判标准也相对明确。

大模型标注平台复杂很多。

第一，标注对象更复杂。

一个样本可能是一段多轮对话、一个工具调用轨迹、一段代码、一个 RAG 检索结果、一组候选回答，甚至是输入、输出、引用文档和安全策略的组合。

第二，标签不是简单类别。

标注员可能要判断帮助性、真实性、无害性、完整性、风格、拒答是否合理、是否遵循系统指令、是否引用了正确证据。这些维度之间还可能冲突。

第三，主观性更强。

“哪个回答更好”往往没有唯一答案。不同标注员可能有不同偏好，所以平台必须做一致性检查、仲裁和标注规范迭代。

第四，质量要求更高。

用于 SFT 或偏好训练的数据会直接影响模型行为。如果标注存在系统性偏差，模型就会学到错误偏好。例如过度偏好长答案，模型会变啰嗦；过度偏好拒答，模型会变保守；过度偏好迎合用户，模型可能更容易产生幻觉或安全风险。

第五，隐私和安全风险更高。

真实用户对话、企业文档、代码片段和业务数据可能包含 PII、商业秘密、密钥、客户信息和内部策略。平台必须在采样、脱敏、权限、审计和导出环节都做控制。

因此，优秀的答案要强调：标注平台不是简单 CRUD，而是一个围绕“数据质量”和“可追溯性”设计的生产系统。

## 11.3 功能需求

一个大模型数据标注平台至少需要以下功能。

| 模块 | 功能 |
| --- | --- |
| 数据接入 | 上传原始样本、导入线上日志、导入模型输出、导入评测样例 |
| 数据预处理 | 去重、脱敏、格式校验、采样、任务切分 |
| 标注项目管理 | 创建项目、定义任务类型、配置标签体系、配置标注规范 |
| 标注界面 | 单样本标注、对话标注、pairwise 比较、多维打分、评论和证据填写 |
| 任务分发 | 队列、优先级、标注员匹配、负载均衡、盲标、重复标注 |
| 质量控制 | 金标题、交叉标注、一致性检查、抽检、仲裁、标注员评分 |
| 模型辅助 | 模型预标注、主动学习、低置信样本推荐、自动质检 |
| 数据版本 | 数据集版本、标注版本、规范版本、导出版本、血缘追踪 |
| 权限隐私 | 角色权限、字段脱敏、敏感样本隔离、访问审计、导出审批 |
| 报表监控 | 进度、吞吐、成本、一致性、错误率、标注员质量 |

一个典型流程是：

1. 算法工程师创建标注项目。
2. 上传或选择一批候选样本。
3. 平台对样本做格式校验、去重、脱敏和抽样。
4. 项目负责人配置标签体系、评分维度和标注规范。
5. 平台把任务分发给合适的标注员。
6. 标注员在界面中完成标注并提交。
7. 平台对样本做一致性检查、金标校验和抽检。
8. 有争议样本进入仲裁。
9. 通过质检的数据进入版本化数据集。
10. 数据集被训练平台、评估平台或安全审核系统消费。

## 11.4 非功能需求

数据标注平台的非功能需求通常比功能列表更能体现系统设计能力。

### 11.4.1 可追溯性

平台必须知道每条数据从哪里来、谁标的、什么时候标的、按哪个规范标的、经过哪些质检、最终进入了哪个数据集版本。

如果模型上线后出现问题，例如模型学会了错误拒答策略，团队需要追溯是哪批偏好数据导致的。如果没有 lineage，数据问题很难定位。

### 11.4.2 质量稳定性

标注质量不能只靠标注员自觉。平台要通过机制保证稳定性，例如金标题、重复标注、交叉验证、抽检和仲裁。

### 11.4.3 可扩展性

不同任务需要不同标注界面。文本分类、pairwise ranking、多轮对话、安全审核、RAG 事实核查和工具调用轨迹评估的界面差别很大。平台要支持任务模板和插件化配置。

### 11.4.4 隐私安全

标注平台经常处理真实数据。必须支持脱敏、最小权限、租户隔离、水印、访问审计、导出审批和数据保留策略。

### 11.4.5 成本效率

人工标注很贵。平台要提高单位成本产出的有效数据量，例如通过模型预标注、主动学习、任务路由和低质量标注员识别降低浪费。

### 11.4.6 可运营性

标注平台不是一次性工具，而是长期运营系统。它要能监控标注进度、吞吐、积压、返工率、争议率、成本和 SLA。

## 11.5 核心架构

一个可落地的数据标注平台可以拆成下面几层。

```text
                 ┌──────────────────────────┐
                 │        Web / Admin        │
                 │ 项目管理 标注界面 报表 审批 │
                 └────────────┬─────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
┌───────▼───────┐     ┌───────▼───────┐     ┌───────▼───────┐
│ Project Service│     │ Task Service   │     │ Quality Service│
│ 项目/规范/标签 │     │ 分发/队列/状态 │     │ 质检/仲裁/金标 │
└───────┬───────┘     └───────┬───────┘     └───────┬───────┘
        │                     │                     │
┌───────▼───────┐     ┌───────▼───────┐     ┌───────▼───────┐
│ Dataset Service│     │ Annotation DB │     │ Model Assist   │
│ 数据集/版本/血缘│     │ 标注结果/事件 │     │ 预标注/质检/采样│
└───────┬───────┘     └───────┬───────┘     └───────┬───────┘
        │                     │                     │
        └─────────────┬───────┴─────────────┬───────┘
                      │                     │
              ┌───────▼───────┐     ┌───────▼───────┐
              │ Object Storage │     │ Audit / IAM    │
              │ 原始数据和附件 │     │ 权限 审计 脱敏 │
              └───────────────┘     └───────────────┘
```

主要组件包括：

1. Project Service：管理项目、任务类型、标签 schema、标注规范和验收规则。
2. Dataset Service：管理原始样本、数据集版本、样本状态、数据血缘和导出。
3. Task Service：负责任务切分、队列、分发、领取、超时、回收和重试。
4. Annotation Service：保存标注结果、修改历史、评论、证据和提交事件。
5. Quality Service：负责金标校验、一致性计算、抽检、仲裁和标注员评分。
6. Model Assist Service：提供模型预标注、自动质检、主动学习和风险样本推荐。
7. IAM / Audit：负责权限控制、访问审计、导出审批、敏感字段控制和合规日志。
8. Reporting Service：展示进度、质量、成本、效率和项目风险。

面试时不需要把每个服务都画得很细，但必须讲清三件事：样本如何流动，标注如何分发，质量如何闭环。

## 11.6 数据模型设计

数据标注平台的数据模型要支持版本化和审计。一个简化设计如下。

### 11.6.1 Project

Project 表示一个标注项目。

关键字段包括：

| 字段 | 含义 |
| --- | --- |
| project_id | 项目 ID |
| name | 项目名称 |
| task_type | 任务类型，如分类、排序、对话打分 |
| label_schema_id | 标签体系版本 |
| guideline_id | 标注规范版本 |
| owner | 项目负责人 |
| status | 草稿、运行中、暂停、完成 |
| created_at | 创建时间 |

### 11.6.2 Sample

Sample 表示待标注样本。

| 字段 | 含义 |
| --- | --- |
| sample_id | 样本 ID |
| dataset_id | 所属数据集 |
| source | 来源，如线上日志、评测集、模型生成 |
| content_uri | 原文或附件存储位置 |
| metadata | 语言、领域、风险级别等元数据 |
| pii_status | 是否已脱敏 |
| hash | 去重和追踪用哈希 |
| version | 样本版本 |

### 11.6.3 Task

Task 表示分发给标注员的工作单元。

| 字段 | 含义 |
| --- | --- |
| task_id | 任务 ID |
| project_id | 项目 ID |
| sample_id | 样本 ID |
| assignee | 标注员 |
| status | 待领取、进行中、已提交、返工、仲裁、完成 |
| priority | 优先级 |
| due_at | 截止时间 |
| attempt | 第几轮标注 |

### 11.6.4 Annotation

Annotation 表示一次标注提交。

| 字段 | 含义 |
| --- | --- |
| annotation_id | 标注 ID |
| task_id | 任务 ID |
| sample_id | 样本 ID |
| annotator_id | 标注员 ID |
| result | 标注结果 JSON |
| confidence | 标注员置信度 |
| comment | 解释或备注 |
| guideline_version | 使用的规范版本 |
| submitted_at | 提交时间 |

### 11.6.5 QualityRecord

QualityRecord 表示质检结果。

| 字段 | 含义 |
| --- | --- |
| record_id | 质检记录 ID |
| sample_id | 样本 ID |
| check_type | 金标、一致性、抽检、自动质检 |
| score | 质量分 |
| decision | 通过、返工、仲裁、丢弃 |
| reviewer | 质检人或系统 |
| reason | 原因 |

这个数据模型的重点不是表有多全，而是要体现：一条最终训练数据不是凭空出现的，它有来源、规范、标注、质检和版本。

## 11.7 任务分发设计

任务分发是标注平台的核心。简单的做法是把任务平均分给标注员，但真实系统要考虑很多因素。

### 11.7.1 分发目标

任务分发要同时满足：

1. 吞吐：尽快完成标注项目。
2. 质量：复杂任务分给能力更强的标注员。
3. 公平：避免少数人被过度分配。
4. 一致性：需要重复标注的样本要分给不同标注员。
5. 隐私：敏感样本只能给有权限的人。
6. 成本：专家标注贵，普通样本不应全部交给专家。

### 11.7.2 分发策略

常见策略包括：

| 策略 | 适用场景 |
| --- | --- |
| FIFO | 简单项目，按进入顺序处理 |
| Priority Queue | 高优先级项目或即将到期任务 |
| Skill-based Routing | 医疗、法律、代码等专家标注 |
| Redundant Assignment | 同一样本分给多人，用于一致性检查 |
| Gold Injection | 混入金标题，评估标注员质量 |
| Active Learning | 优先标注模型低置信或高价值样本 |
| Risk-based Routing | 高风险样本分给高级审核员 |

在大模型偏好标注中，常见设计是：普通样本先由两名标注员独立判断，如果结果一致则通过；如果不一致，进入第三人仲裁或专家复核。

### 11.7.3 任务状态机

任务要有明确状态机，否则很容易出现重复标注、任务丢失和无法追踪。

```text
Created -> Assigned -> InProgress -> Submitted -> QCReview -> Accepted
                                      │              │
                                      │              ├-> Rework -> InProgress
                                      │              │
                                      │              └-> Arbitration -> Accepted / Rejected
                                      │
                                      └-> Expired -> Reassigned
```

关键机制包括：

1. 领取锁：防止同一任务被多人同时领取。
2. 超时回收：标注员长时间不提交，任务重新入队。
3. 幂等提交：避免重复点击导致多次提交。
4. 版本校验：如果规范或样本更新，旧任务不能直接提交。
5. 操作日志：记录领取、修改、提交、返工、仲裁全过程。

## 11.8 标注规范设计

标注平台能不能产出高质量数据，很大程度上取决于标注规范。

标注规范不是一段说明文字，而是把模糊判断转化为可执行标准的产品文档。它至少要包括：

1. 任务目标：为什么标注这批数据，用于训练、评估还是安全审核。
2. 标签定义：每个标签是什么意思，不是什么意思。
3. 正例和反例：给出典型样本。
4. 边界案例：处理模糊、冲突、多标签、缺信息的情况。
5. 打分标准：如果是 1 到 5 分，每一档代表什么。
6. 冲突优先级：事实性、帮助性、安全性冲突时谁优先。
7. 提交要求：是否必须写理由，是否必须引用证据。
8. 更新记录：规范变更后如何影响旧标注。

以大模型回答评估为例，如果只让标注员判断“回答好不好”，结果会非常不稳定。更好的方式是拆成多个维度：

| 维度 | 说明 |
| --- | --- |
| 正确性 | 是否事实正确、逻辑正确 |
| 完整性 | 是否覆盖用户问题的关键点 |
| 有用性 | 是否能帮助用户完成目标 |
| 简洁性 | 是否无不必要冗余 |
| 指令遵循 | 是否遵守格式、角色和约束 |
| 安全性 | 是否包含风险内容或不当建议 |
| 可验证性 | 是否给出依据或引用证据 |

对于 pairwise preference，可以要求标注员选择 A 更好、B 更好、差不多、都不好，并填写主要原因：事实错误、遗漏关键点、格式不对、过度拒答、幻觉、安全问题等。

## 11.9 质量控制机制

质量控制是本题最重要的部分。面试官通常会追问：“你怎么保证标注质量？”

一个成熟平台不会只靠人工抽检，而是多层质量控制。

### 11.9.1 金标题

金标题是已经由专家确认答案的样本。平台把金标题混入普通任务，用来衡量标注员是否认真、是否理解规范。

注意金标题不能太明显，否则标注员会专门识别金标题。金标也要定期更新，因为规范变化后，旧金标可能不再适用。

### 11.9.2 重复标注

同一样本分给多个标注员。如果结果一致，说明样本比较明确；如果不一致，说明样本存在歧义、规范不清或标注员质量问题。

重复标注适合高价值任务，但成本较高。可以对高风险样本、低置信样本、新标注员样本使用更高重复率。

### 11.9.3 一致性指标

平台可以计算标注员之间的一致性。

常见指标包括：

1. 简单一致率：多人标签相同的比例。
2. Cohen's Kappa：两个标注员的一致性，考虑随机一致。
3. Fleiss' Kappa：多个标注员的一致性。
4. Kendall Tau 或 Spearman：用于排序任务的一致性。
5. Pairwise agreement：用于偏好比较。

一致性低不一定说明标注员差，也可能说明任务本身模糊。系统要把低一致性样本聚类分析，反过来改进规范。

### 11.9.4 抽检和仲裁

抽检适合常规质量监控。平台可以按项目、标注员、风险等级、时间段抽样。

仲裁适合处理冲突样本。仲裁员通常是高级标注员、专家或项目负责人。仲裁结果要反向更新规范和金标库。

### 11.9.5 自动质检

自动质检不能完全替代人工，但可以捕获明显问题。

例如：

1. 必填字段为空。
2. 标签和理由矛盾。
3. 标注耗时异常短。
4. 同一标注员长时间总选同一选项。
5. 回答被标为“正确”，但引用证据中没有相关信息。
6. 安全风险样本被标为完全安全，但命中了高风险规则。

### 11.9.6 标注员质量画像

平台可以维护标注员画像：

| 指标 | 用途 |
| --- | --- |
| 金标准确率 | 判断是否理解规范 |
| 一致性 | 判断和群体或专家是否一致 |
| 返工率 | 判断提交质量 |
| 仲裁败诉率 | 判断争议样本中谁更可靠 |
| 平均耗时 | 识别过快或过慢问题 |
| 领域能力 | 用于任务路由 |

这个画像不能用于粗暴惩罚，而应服务于培训、路由和质量改进。

## 11.10 模型辅助标注

大模型时代的标注平台不应该完全依赖人工。模型可以参与预标注、筛选、质检和数据选择。

### 11.10.1 模型预标注

模型先给出候选标签、候选理由或摘要，标注员只需确认或修改。这可以提高效率。

风险是标注员可能被模型锚定，盲目接受模型答案。因此界面设计要谨慎，例如对高风险任务隐藏模型答案，或要求标注员先独立判断再展示模型建议。

### 11.10.2 主动学习

主动学习的核心思想是：不要随机标注所有样本，而是优先标注对模型最有价值的样本。

高价值样本可能包括：

1. 模型置信度低的样本。
2. 多个模型分歧大的样本。
3. 线上高频失败样本。
4. 安全风险高但覆盖不足的样本。
5. 新领域、新语言、新格式样本。

这样可以用更少标注成本获得更大训练收益。

### 11.10.3 自动质检和异常检测

模型可以检查标注理由是否和标签一致，回答是否引用了正确文档，是否存在明显安全风险。但自动质检也会误判，所以它更适合做风险排序，而不是直接决定最终标签。

### 11.10.4 人机协同原则

一个务实原则是：

```text
低风险、低价值、规则明确的样本更多交给模型；高风险、高价值、主观性强的样本更多交给人工和专家。
```

这能在成本和质量之间取得平衡。

## 11.11 数据版本和血缘

数据标注平台最终要产出可训练、可评估、可审计的数据集。如果没有版本管理，后续模型训练很难复现。

### 11.11.1 为什么要版本化

假设团队训练了一个 `sft-v7` 模型，效果很好。三个月后想复现，却发现：原始样本变了，标注规范更新了，一部分标注被返工覆盖，过滤规则也变了。这时就无法知道当初模型到底用的是什么数据。

所以平台必须记录：

1. 原始数据版本。
2. 样本过滤规则版本。
3. 标注规范版本。
4. 标注结果版本。
5. 质检规则版本。
6. 导出脚本版本。
7. 数据集 manifest。

### 11.11.2 数据集 Manifest

每个导出的数据集可以生成一个 manifest 文件，记录：

```json
{
  "dataset_id": "sft_customer_support_v3",
  "created_at": "2026-05-26T10:00:00Z",
  "source_datasets": ["online_logs_2026w18", "failure_cases_v2"],
  "sample_count": 120000,
  "schema_version": "chat_sft_schema_v4",
  "guideline_version": "support_sft_guideline_v6",
  "quality_policy": "qc_policy_v3",
  "filters": ["pii_redaction_v5", "dedup_v2", "safety_filter_v4"],
  "export_format": "jsonl",
  "owner": "data-team"
}
```

训练平台只消费带 manifest 的数据集，评估平台也记录模型对应的数据集版本。这样才能把模型行为和数据版本关联起来。

### 11.11.3 不可变版本和增量版本

一个重要设计原则是：已发布的数据集版本不可变。如果要修复错误，创建新版本，而不是原地覆盖。

可以支持增量版本，例如：

1. `dataset-v1`：初始 10 万条。
2. `dataset-v2`：在 v1 基础上修复 2000 条错误标注。
3. `dataset-v3`：新增 3 万条失败案例。

这样训练结果才可复现。

## 11.12 隐私和权限设计

数据标注平台是隐私风险很高的系统。系统设计面试中，如果完全不提隐私和权限，会显得不成熟。

### 11.12.1 数据脱敏

平台接入样本时要尽量先脱敏，再进入标注队列。

常见处理包括：

1. 替换姓名、手机号、邮箱、身份证号、地址等 PII。
2. 屏蔽密钥、token、密码、内部 URL。
3. 对企业客户数据做租户隔离。
4. 对附件和图片做 OCR 后敏感信息检测。
5. 对无法脱敏但必须标注的样本启用更严格权限。

### 11.12.2 最小权限

标注员只应看到完成任务所需的信息。

例如做回答偏好标注时，不一定需要看到用户 ID、地理位置、订单号和联系方式。做安全审核时，可能需要看到更多上下文，但应限制导出和复制。

### 11.12.3 访问审计

平台要记录：

1. 谁看过哪些样本。
2. 谁导出了哪些数据。
3. 谁修改了标注结果。
4. 谁改变了标注规范。
5. 谁审批了敏感数据访问。

这些日志用于排查泄露、追责和合规审计。

### 11.12.4 导出控制

训练数据导出应该有审批和水印机制。对敏感数据集，可以限制只能导出到训练平台的受控路径，不能下载到个人机器。

## 11.14 存储设计

标注平台通常需要多种存储。

| 存储 | 内容 |
| --- | --- |
| 关系型数据库 | 项目、任务、用户、权限、状态机、标注元数据 |
| 对象存储 | 原始文本、大文件、图片、音频、视频、导出数据集 |
| 搜索引擎 | 按关键词、标签、错误类型检索样本 |
| OLAP / 数仓 | 统计报表、质量趋势、成本分析 |
| 缓存 | 任务领取、项目配置、热点样本 |
| 消息队列 | 导入、预处理、分发、质检、导出异步任务 |

关系型数据库适合保存强一致的任务状态。对象存储适合保存大文本和附件。OLAP 适合做项目报表。不要把所有内容都塞进一张大表。

## 11.15 扩展性设计：不同任务类型如何支持

标注平台如果只支持一种表单，很快就会被新任务打爆。更好的方式是用 schema 驱动界面。

一个 Label Schema 可以描述：

1. 输入展示方式：单文本、多轮对话、两个回答对比、图片加文本。
2. 标注字段：单选、多选、打分、排序、文本理由、证据引用。
3. 校验规则：必填、范围、互斥、条件显示。
4. 质检规则：是否需要重复标注、是否需要专家仲裁。
5. 导出映射：标注结果如何转换成训练格式。

例如偏好标注 schema：

```json
{
  "task_type": "pairwise_preference",
  "inputs": ["prompt", "answer_a", "answer_b"],
  "fields": [
    {"name": "preferred", "type": "single_choice", "options": ["A", "B", "tie", "both_bad"]},
    {"name": "reason", "type": "text", "required": true},
    {"name": "safety_issue", "type": "boolean"}
  ],
  "quality": {
    "redundant_labels": 2,
    "arbitration_on_disagreement": true
  }
}
```

这样新增任务时主要新增配置和少量前端组件，而不是重写整个平台。

## 11.16 和训练、评估、安全系统的关系

数据标注平台不是孤立系统。

它和训练平台的关系是：

1. 训练平台提供失败样本和需要增强的能力方向。
2. 标注平台产出 SFT、偏好、拒答、安全和工具调用数据。
3. 训练平台记录使用的数据集版本。
4. 新模型结果再回流到标注平台做偏好和失败分析。

它和评估平台的关系是：

1. 评估平台发现模型弱点。
2. 弱点样本进入标注平台补充高质量标签。
3. 标注结果形成新的评测集或训练集。
4. 评估平台用新数据验证修复是否有效。

它和安全审核系统的关系是：

1. 安全系统发现高风险样本。
2. 标注平台组织安全专家标注风险类别和期望响应。
3. 标注结果用于安全训练和红队评估。
4. 安全策略更新后，标注规范同步更新。

这就是大模型数据闭环：线上数据、评估失败、安全风险和人工偏好不断回流，形成下一轮训练和评估的数据资产。

## 11.17 关键指标

标注平台上线后，要用指标运营。

### 11.17.1 进度指标

1. 待标注样本数。
2. 已完成样本数。
3. 每日标注吞吐。
4. 项目预计完成时间。
5. 队列积压。

### 11.17.2 质量指标

1. 金标准确率。
2. 多人一致率。
3. 仲裁率。
4. 返工率。
5. 抽检通过率。
6. 低质量标注占比。

### 11.17.3 成本指标

1. 单样本成本。
2. 单有效样本成本。
3. 专家标注占比。
4. 返工成本。
5. 模型预标注节省成本。

### 11.17.4 数据价值指标

1. 标注数据进入训练后的指标提升。
2. 失败案例修复率。
3. 安全违规率下降。
4. 用户反馈改善。
5. 评估集覆盖提升。

最后一类最重要。因为标注平台的目标不是“标了多少条”，而是“产生了多少有效数据价值”。

## 11.18 常见追问

### 11.18.1 如果标注员意见不一致怎么办？

先判断不一致来源。

如果是样本本身模糊，要进入仲裁并更新规范。如果是标注员理解错误，要培训和返工。如果是任务主观性强，可以保留分布标签，而不是强行压成唯一答案。

偏好数据中，有时“不一致”本身也是信息。它说明两个回答差异不明显，训练时可以降低样本权重或不纳入偏好训练。

### 11.18.2 如何防止标注员乱标？

综合使用金标题、重复标注、耗时异常检测、选项分布检测、抽检、仲裁败诉率和标注员画像。对新标注员提高抽检比例，对稳定标注员降低重复标注比例。

### 11.18.3 如何降低成本？

可以从四方面入手：

1. 用模型预标注减少人工输入。
2. 用主动学习优先标注高价值样本。
3. 用任务路由让专家只处理难样本。
4. 用自动质检减少返工和低质量数据进入训练。

### 11.18.4 如何处理隐私数据？

先脱敏再标注；无法脱敏的样本进入受控队列；使用最小权限、访问审计、导出审批和水印；敏感数据尽量只在受控环境中访问，不允许随意下载。

### 11.18.5 如何支持多语言标注？

需要语言识别、语言技能画像、不同语言的标注规范、本地化界面和分语言质量统计。不能把所有语言用同一批标注员处理，也不能只用中文或英文规范套所有语言。

### 11.18.6 如何做大模型偏好数据？

核心是 pairwise 或 listwise 比较。每个样本包含 prompt、多个候选回答、上下文和必要证据。标注员判断哪个回答更好，并给出原因。平台做重复标注和一致性检查，最终导出为 DPO、RLHF 或 reward model 所需格式。

## 11.19 面试回答模板

面试中可以按这个顺序回答：

1. 澄清范围：面向大模型训练和评估，支持 SFT、偏好、安全和评估标注。
2. 给出需求：数据接入、项目管理、任务分发、标注界面、质量控制、版本、权限和报表。
3. 画架构：Project、Dataset、Task、Annotation、Quality、Model Assist、IAM、Reporting。
4. 讲主流程：导入样本、脱敏、配置规范、分发、标注、质检、仲裁、导出数据集。
5. 深入质量：金标、重复标注、一致性、抽检、仲裁、自动质检和标注员画像。
6. 深入版本：样本、规范、标注、质检、导出 manifest，全链路可追溯。
7. 深入隐私：脱敏、最小权限、审计、导出审批、敏感样本隔离。
8. 讲扩展：schema 驱动支持多任务，模型辅助降低成本，和训练评估系统形成闭环。

一个简洁回答可以是：

```text
我会把数据标注平台设计成围绕数据质量闭环的平台，而不是简单表单系统。核心对象包括项目、样本、任务、标注结果、质检记录和数据集版本。样本导入后先做脱敏、去重和格式校验，然后按项目规范生成任务，通过队列分发给具备对应权限和技能的标注员。提交后进入金标校验、重复标注一致性检查、抽检和仲裁。通过质检的数据形成不可变的数据集版本，带 manifest 导出给训练和评估平台。平台还要支持模型预标注、主动学习、自动质检、权限审计和隐私保护。这样才能保证标注数据可用、可追踪、可复现、可合规。
```

## 11.20 本章小结

数据标注平台是大模型工程体系中的关键基础设施。它不只是把人工标签录入数据库，而是把原始样本、标注规范、人工判断、质量控制、数据版本和隐私治理连接起来。

本章要点如下：

1. 大模型标注任务比传统标注更复杂，因为样本结构复杂、判断维度多、主观性强、质量影响大。
2. 标注平台的核心功能包括数据接入、项目管理、任务分发、标注界面、质量控制、模型辅助、版本管理和权限隐私。
3. 任务分发要考虑吞吐、质量、公平、技能、隐私和成本。
4. 标注规范决定了标注质量的上限，必须包含定义、示例、边界案例、打分标准和更新记录。
5. 质量控制要结合金标题、重复标注、一致性检查、抽检、仲裁、自动质检和标注员画像。
6. 数据版本和血缘是训练复现和问题追踪的基础。
7. 隐私保护要贯穿数据接入、展示、标注、质检、导出和审计全过程。
8. 模型辅助可以提升效率，但高风险和高价值样本仍需要人工和专家把关。

面试中最重要的一句话是：

```text
数据标注平台的核心不是让人点标签，而是稳定生产可训练、可评估、可追溯、可合规的高质量数据。
```
