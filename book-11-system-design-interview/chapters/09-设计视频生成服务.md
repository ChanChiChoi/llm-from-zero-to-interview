# 第九章：设计视频生成服务

这一章讨论如何设计视频生成服务。

视频生成服务和前面几类系统很不一样。ChatGPT 服务追求流式低延迟，推理平台追求高吞吐在线服务，实时语音助手追求交互自然；视频生成服务通常是长耗时、重 GPU、强异步、强安全审核、强队列调度的系统。

用户输入一段 prompt、一张参考图、一个视频片段或一组控制条件，系统生成数秒到数十秒的视频。这个过程可能需要几十秒到数分钟甚至更久，不能按普通同步 API 设计。生产级视频生成服务要处理任务队列、GPU 调度、进度反馈、内容安全、结果存储、失败重试、配额计费、版权和滥用风险。

本章重点：任务队列、长耗时生成、GPU 调度、内容安全、结果存储、用户体验。

## 9.1 先明确题目边界

面试官问“设计视频生成服务”，可能有几种范围。

第一，设计文本生成视频服务。

用户输入文本 prompt，系统生成视频。重点是任务队列、推理耗时、GPU 调度、内容安全和结果展示。

第二，设计图像生成视频服务。

用户上传首帧、参考图或产品图，系统生成动态视频。重点是图片审核、图像条件、身份和版权风险。

第三，设计视频编辑或视频扩展服务。

用户上传视频，系统做局部重绘、风格化、扩展、补帧或转场。重点是输入视频解析、分段处理、时序一致性和存储成本。

第四，设计企业视频生成平台。

面向营销、广告、教育、游戏或影视预览，支持模板、批量生成、审批、团队协作和资产管理。

面试开头可以这样澄清：

我先按文本和图片条件的视频生成服务设计，支持用户提交 prompt 或参考图，异步生成短视频，提供排队、进度、取消、内容安全、结果存储、重新生成、配额计费和审核。视频编辑、长视频和企业协作作为后续扩展。

## 9.2 视频生成服务的核心难点

视频生成比图片生成更复杂，因为它多了时间维度。

核心难点包括：

1. 生成耗时长：不能同步阻塞等待。
2. GPU 成本高：需要细粒度调度和配额。
3. 结果体积大：视频文件、预览、缩略图和中间产物都要存储。
4. 安全风险高：视频可用于误导、深度伪造、色情暴力、版权侵权。
5. 用户体验难：用户要知道排队进度、生成进度、失败原因和预计等待时间。
6. 质量不稳定：同一 prompt 可能生成多种结果，需要重试、种子、候选和反馈。
7. 任务不可逆成本：一旦开始生成，取消也可能已经消耗大量 GPU。

所以视频生成服务的设计重点不是“如何实现模型”，而是“如何把长耗时高成本模型变成可靠产品”。

## 9.3 功能需求

一个视频生成服务通常需要以下功能。

| 模块 | 功能 |
| --- | --- |
| 用户入口 | prompt 输入、参考图上传、参数选择、模板 |
| 任务服务 | 创建任务、查询状态、取消、重试、重新生成 |
| 安全审核 | 输入 prompt、图片、视频输出和元数据审核 |
| 队列调度 | 优先级、配额、公平性、预计等待时间 |
| GPU 执行 | 模型加载、采样、分辨率、时长、批处理 |
| 进度反馈 | 排队中、生成中、后处理中、完成、失败 |
| 结果存储 | 视频文件、缩略图、预览、元数据、生命周期 |
| 后处理 | 编码、压缩、水印、转码、封面、审核 |
| 计费配额 | 按时长、分辨率、模型、GPU 时间计费 |
| 反馈评估 | 用户评分、重生成、举报、质量抽检 |

第一版可以先支持短视频，例如 4 到 10 秒、固定分辨率、少量模型参数。不要一开始就支持任意长视频、复杂编辑和高并发实时生成。

## 9.4 非功能需求

### 9.4.1 可用性

视频生成是异步任务，服务可用性不仅是 API 可用，还包括队列、GPU worker、对象存储、审核系统和通知系统可用。

### 9.4.2 公平性

GPU 资源有限。系统要防止少数大用户占满队列，也要避免免费用户无限排队影响付费用户 SLA。

### 9.4.3 可解释状态

用户不能只看到“处理中”。系统要尽量提供排队位置、预计等待时间、生成阶段和失败原因。

### 9.4.4 安全性

视频生成风险高，输入和输出都要审核。对人脸、名人、未成年人、暴力、色情、政治敏感、深度伪造和版权内容要有策略。

### 9.4.5 成本可控

每个任务可能消耗大量 GPU 秒。系统要有配额、预算、取消策略、失败重试上限和异常成本告警。

## 9.5 容量估算

假设设计一个公开视频生成服务：

1. 日活用户 100 万。
2. 每个用户每天平均提交 0.2 个视频任务。
3. 每日任务 20 万。
4. 峰值提交 QPS 约 50。
5. 每个任务平均生成 8 秒视频。
6. 平均 GPU 生成耗时 60 秒。
7. 平均输出视频 20MB。

任务吞吐估算：

如果一张 GPU 同时只能跑 1 个任务，且每任务 60 秒，那么 1 张 GPU 每小时约 60 个任务。20 万任务每天需要约 3333 GPU 小时，平均约 139 张 GPU 持续工作。考虑峰值、失败重试、空闲和不同模型，实际可能需要更多。

存储估算：

20 万任务每天，每个结果 20MB，仅输出视频每天约 4TB。如果保存原始输入、预览、缩略图、中间 latent 或多候选结果，存储更高。

这个估算说明：视频生成系统的核心瓶颈是 GPU 时间和对象存储，不是普通 API QPS。

## 9.6 高层架构

视频生成服务可以分成前台请求、任务编排、队列调度、GPU 执行、后处理、安全审核和结果管理。

| 组件 | 职责 |
| --- | --- |
| Client | prompt 输入、参数选择、任务状态、结果展示 |
| API Gateway | 鉴权、限流、请求入口 |
| User/Quota Service | 用户等级、配额、计费、优先级 |
| Task Service | 创建任务、状态机、取消、重试、查询 |
| Safety Service | 输入审核、输出审核、风险策略 |
| Prompt Processor | prompt 改写、翻译、模板、负面词处理 |
| Queue Manager | 多队列、优先级、公平调度、ETA |
| GPU Scheduler | GPU 资源分配、模型实例、任务放置 |
| Generation Worker | 视频模型推理、采样、分辨率、时长控制 |
| Post Processor | 转码、压缩、水印、缩略图、封面 |
| Object Store | 输入文件、输出视频、预览、缩略图 |
| Metadata Store | 任务状态、参数、模型版本、结果元数据 |
| Notification Service | 完成通知、失败通知、回调 |
| Monitoring | 队列、GPU、任务成功率、成本、安全事件 |

核心链路是：用户提交任务，API 完成鉴权和配额检查；Safety Service 审核输入；Task Service 创建任务；Queue Manager 排队；GPU Scheduler 分配 worker；Generation Worker 执行生成；Post Processor 做转码、水印和缩略图；Safety Service 审核输出；结果写入 Object Store；用户收到通知并查看结果。

## 9.7 任务状态机

视频生成任务必须有清晰状态机。

常见状态包括：

1. created：任务已创建。
2. input_reviewing：输入审核中。
3. rejected：输入不通过。
4. queued：排队中。
5. running：生成中。
6. post_processing：后处理中。
7. output_reviewing：输出审核中。
8. succeeded：完成。
9. failed：失败。
10. cancelled：取消。
11. expired：结果过期。

状态转换要幂等。用户重复点击提交或网络重试时，不能重复创建多个收费任务。可以使用 idempotency key。

### 9.7.1 取消任务

取消分两种情况。

如果任务还在 queued，可以直接取消并释放配额。如果任务已经 running，取消可能只能停止后续生成，但已经消耗的 GPU 成本不一定能完全退还。

### 9.7.2 重试任务

失败重试要区分原因。基础设施错误可以自动重试，输入不合规不能重试，模型生成失败可以有限重试。

### 9.7.3 重新生成

重新生成通常创建新任务，但复用原 prompt、参数和参考图。系统可以保留多个候选，便于用户比较。

## 9.8 队列与调度

队列设计是视频生成服务的核心。

### 9.8.1 多队列

可以按任务类型和用户等级分队列：

1. 免费用户队列。
2. 付费用户队列。
3. 企业 SLA 队列。
4. 高分辨率队列。
5. 批量任务队列。
6. 安全复核队列。

### 9.8.2 优先级

优先级可以由用户等级、任务年龄、成本、SLA 和公平性共同决定。

不能简单让付费用户永远插队，否则免费用户永远等不到。可以使用配额、权重公平队列或老化机制。

### 9.8.3 ETA 估算

用户很关心还要等多久。ETA 可以根据队列长度、平均任务耗时、GPU 可用数和优先级估算。

ETA 不一定精确，但比完全没有反馈好。

### 9.8.4 任务分桶

不同分辨率、时长、模型和参考条件的任务耗时不同。调度时可以按任务规格分桶，避免短任务被长任务阻塞。

## 9.9 GPU 调度

视频生成对 GPU 资源要求高。

### 9.9.1 调度目标

GPU Scheduler 要考虑：

1. 模型类型。
2. 分辨率。
3. 视频时长。
4. GPU 显存。
5. 多 GPU 并行需求。
6. 当前负载。
7. 任务优先级。
8. 模型 warmup 状态。

### 9.9.2 模型实例池

不同模型可以有不同实例池。例如快速低清模型、高质量模型、图生视频模型、视频编辑模型。

热门模型保留 warm pool，冷门模型按需加载。加载大模型慢，所以要平衡显存占用和冷启动。

### 9.9.3 批处理

某些视频生成模型可以批处理相同规格任务，提高吞吐。但批处理会增加等待时间，也要求任务规格接近。

### 9.9.4 抢占

视频生成任务通常不适合频繁抢占，因为中间状态复杂。低优先级长任务可以在分阶段边界支持抢占和恢复，但第一版可以先避免抢占，靠队列控制。

## 9.10 生成 Worker

Generation Worker 负责真正调用模型生成视频。

### 9.10.1 输入参数

参数包括：

1. prompt。
2. negative prompt。
3. 参考图片或视频。
4. 分辨率。
5. 时长。
6. 帧率。
7. seed。
8. guidance scale。
9. 采样步数。
10. 模型版本。

参数要被记录，方便复现和问题定位。

### 9.10.2 随机性和 seed

视频生成有随机性。保存 seed 可以让部分结果可复现。重新生成时可以选择保留 seed 或换 seed。

### 9.10.3 进度上报

Worker 应周期性上报进度，例如模型加载、采样步数、后处理、上传结果。即使进度是估计值，也能改善用户体验。

### 9.10.4 中间结果

某些系统可以生成低清预览或关键帧预览，让用户提前判断是否继续。这样可以减少无效高成本生成。

## 9.11 后处理与结果存储

生成模型输出后，还要后处理。

### 9.11.1 后处理

后处理包括：

1. 视频编码。
2. 压缩。
3. 格式转换。
4. 水印。
5. 缩略图。
6. 封面抽帧。
7. 质量检查。
8. 元数据写入。

### 9.11.2 结果存储

输出视频放对象存储，Metadata Store 保存 task_id、user_id、model_version、prompt、参数、文件 URL、大小、时长、审核状态和过期时间。

### 9.11.3 生命周期

视频结果可能很大，不能无限保存。

可以设计：免费用户结果保存 7 天，付费用户保存更久，用户收藏或下载后按策略保留，违规结果删除或隔离。

### 9.11.4 CDN

用户播放和下载视频需要 CDN。生成完成后，可以生成多码率版本，支持不同网络条件播放。

## 9.12 内容安全

视频生成安全非常关键。

### 9.12.1 输入审核

输入 prompt 和参考图要审核，识别色情、暴力、自伤、仇恨、违法、未成年人风险、名人冒充、隐私、版权、政治敏感等。

### 9.12.2 输出审核

输出视频也要审核，因为模型可能生成输入中没有明确要求的风险内容。

输出审核可以抽帧加视频分类，结合音频审核和文本元数据审核。

### 9.12.3 人脸和身份风险

图生视频可能涉及真实人脸、名人、普通人的肖像权和深度伪造风险。

系统可以限制真实人物生成、要求授权证明、加水印和元数据标识。

### 9.12.4 水印和溯源

视频生成结果可以加入可见或不可见水印，并记录生成元数据，方便后续溯源。

### 9.12.5 合规边界

本章讨论合法授权范围内的视频生成服务设计，不提供制作深度伪造、冒充真实个人、规避内容审核、生成违法有害内容或移除溯源标识的方法。

## 9.13 用户体验设计

视频生成等待时间长，用户体验很重要。

### 9.13.1 任务进度

展示排队中、预计等待、生成中、后处理中、审核中、完成或失败。

### 9.13.2 失败原因

失败要给出可理解原因。例如输入不合规、参考图格式错误、生成失败、资源繁忙、审核未通过。

### 9.13.3 重新生成和参数调整

用户常常需要多次尝试。可以提供保留 prompt、修改参数、换 seed、生成多个候选的能力。

### 9.13.4 通知

长任务完成后，通过站内通知、邮件、Webhook 或客户端推送提醒用户。

### 9.13.5 预览

低清预览、关键帧预览和生成中的阶段性预览能减少用户焦虑，但会增加系统复杂度。

## 9.14 计费与配额

视频生成成本高，必须设计配额和计费。

### 9.14.1 计费维度

可以按：

1. 视频时长。
2. 分辨率。
3. 模型等级。
4. 生成次数。
5. GPU 时间。
6. 是否加速队列。
7. 存储时长。

### 9.14.2 配额

免费用户可以限制每日次数、分辨率、时长和队列优先级。企业用户可以有团队配额和预算控制。

### 9.14.3 退款和失败任务

输入审核拒绝通常不计费。基础设施失败可以退还额度。用户取消 running 任务是否退还，要按产品策略明确。

## 9.15 监控与评估

视频生成服务监控包括系统、质量、安全和成本。

### 9.15.1 系统指标

包括任务提交 QPS、队列长度、平均等待时间、P95 完成时间、任务成功率、失败率、取消率、GPU 利用率、后处理延迟、对象存储错误。

### 9.15.2 质量指标

包括用户评分、重新生成率、下载率、举报率、人工抽检质量、视频时序一致性、prompt adherence。

### 9.15.3 安全指标

包括输入拒绝率、输出拒绝率、人工复核率、违规漏检率、深度伪造风险命中、水印成功率。

### 9.15.4 成本指标

包括每任务 GPU 秒、每秒视频成本、失败任务成本、存储成本、CDN 成本和不同用户等级成本。

## 9.16 高可用与故障恢复

常见故障包括：

1. GPU worker 崩溃。
2. 模型加载失败。
3. 队列积压。
4. 后处理失败。
5. 对象存储上传失败。
6. 安全审核服务不可用。
7. 任务状态写入失败。

恢复策略包括：

1. 任务状态持久化。
2. Worker 心跳和超时回收。
3. 基础设施错误自动重试。
4. 后处理阶段可单独重试。
5. 输出上传失败可从临时文件恢复。
6. 安全服务失败时保守处理，不直接发布。
7. 队列积压时限流或关闭低优先级入口。

## 9.17 成本优化

视频生成成本优化包括：

1. 分层模型：草稿模型和高质量模型分开。
2. 低清预览：先生成低清，用户满意再高清。
3. 队列批处理：同规格任务合批。
4. Warm pool：热门模型常驻，冷门模型按需。
5. 限制默认时长和分辨率。
6. 失败早停：安全或参数错误尽早拒绝。
7. 结果生命周期管理。
8. 对异常高成本任务告警。

成本优化不能绕过内容安全和溯源要求。

## 9.18 面试答题模板

如果面试官问“设计视频生成服务”，可以这样回答。

第一，澄清范围。

我先按文本和图片条件的短视频生成服务设计，支持用户提交 prompt 或参考图，异步生成视频，提供队列、进度、取消、重试、内容安全、结果存储和配额计费。

第二，定义指标。

核心指标包括任务成功率、平均等待时间、P95 完成时间、GPU 利用率、输入输出审核通过率、用户评分、重新生成率、每任务成本和存储成本。

第三，做容量估算。

根据每日任务数、平均生成耗时、平均输出大小和峰值提交 QPS，估算需要的 GPU 小时、队列长度和对象存储规模。

第四，画架构。

系统包括 Client、API Gateway、User/Quota Service、Task Service、Safety Service、Prompt Processor、Queue Manager、GPU Scheduler、Generation Worker、Post Processor、Object Store、Metadata Store、Notification Service 和 Monitoring。

第五，讲核心链路。

用户提交任务后，先鉴权和配额检查，输入审核通过后创建任务并入队，调度器分配 GPU worker，worker 生成视频并上报进度，后处理转码和水印，输出审核通过后写入对象存储并通知用户。

第六，展开关键难点。

重点讲任务状态机、队列公平性、GPU 调度、进度反馈、输入输出安全审核、结果生命周期、计费配额和失败恢复。

第七，讲安全和成本。

视频生成风险高，要做 prompt、参考图和输出视频审核，加入水印和溯源；成本通过配额、分层模型、低清预览、批处理和生命周期管理控制。

## 9.19 常见面试追问

### 9.19.1 视频生成为什么要异步？

因为生成耗时长、GPU 成本高、任务可能排队和后处理，使用同步请求容易超时且体验差。异步任务可以查询状态、取消、通知和重试。

### 9.19.2 如何估算需要多少 GPU？

根据每日任务数、平均每任务 GPU 耗时、峰值系数和利用率估算 GPU 小时。例如 20 万任务、每任务 60 秒，需要约 3333 GPU 小时，再考虑峰值和冗余。

### 9.19.3 任务队列如何保证公平？

可以按用户等级和任务类型分队列，用权重公平调度、配额、老化机制和最大并发限制，避免大用户或长任务占满资源。

### 9.19.4 输出视频为什么也要审核？

因为模型可能生成输入中没有明确要求的违规内容，或者参考图引发身份、色情、暴力、版权和深度伪造风险。只审核输入不够。

### 9.19.5 用户取消任务如何处理？

如果任务还在队列中，直接取消并释放配额；如果已经运行，要尽力停止 worker 或标记结果丢弃，但已消耗的 GPU 成本可能无法完全回收。

### 9.19.6 如何设计生成进度？

可以按状态展示排队、生成、后处理、审核、完成；生成中可以根据采样步数、阶段和历史平均耗时估算百分比，但要避免承诺过度精确。

### 9.19.7 如何处理失败重试？

基础设施错误、worker 崩溃、对象存储失败可以自动重试；输入不合规不能重试；模型生成失败可以有限重试，并记录失败原因和成本。

### 9.19.8 如何防止视频生成被滥用？

通过输入审核、输出审核、人脸和身份风险控制、水印、溯源、配额、风控、举报、人工复核和审计来防控。高风险内容不能为了体验绕过审核。

## 9.20 本章小结

视频生成服务的核心价值，是把长耗时、高成本、高风险的视频模型能力，包装成用户可提交、可等待、可追踪、可审核、可计费的异步服务。

本章要记住几句话：

1. 视频生成服务通常要异步设计，不能当普通同步 API。
2. 核心瓶颈是 GPU 时间、队列等待、结果存储和内容安全。
3. 任务状态机要清楚，支持排队、生成、后处理、审核、完成、失败和取消。
4. 队列调度要在付费优先、资源利用率和公平性之间取舍。
5. 输出视频必须审核，因为风险可能在生成结果中出现。
6. 结果存储要有生命周期，视频文件不能无限保存。
7. 用户体验要提供 ETA、进度、失败原因、通知和重新生成。
8. 成本优化不能牺牲内容安全、水印和溯源。

如果面试中被问到视频生成服务，最好的回答是强调异步任务系统：从任务提交、审核、排队、GPU 调度、生成、后处理、输出审核、存储到通知和计费，完整讲清长耗时高成本服务如何稳定落地。
