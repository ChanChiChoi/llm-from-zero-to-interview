# 第八章：设计实时语音助手

这一章讨论如何设计实时语音助手。

上一章的多模态助手把语音作为一种输入模态，重点是短音频上传、ASR 转写和后续文本处理。本章不同：实时语音助手强调低延迟、连续对话、边听边识别、边想边说、可打断、可恢复。它更像电话助手、车载助手、智能音箱、语音客服或实时会议助手。

实时语音助手不是“ASR 转文本，加 LLM，再 TTS 播放”这么简单。真正的系统要处理麦克风音频流、语音活动检测、端点检测、流式 ASR、增量理解、LLM 低延迟生成、流式 TTS、播放队列、用户打断、噪声、回声、网络抖动、状态管理、安全审核和质量评估。

本章重点：流式 ASR、LLM、TTS、低延迟、打断机制、语音活动检测。

## 8.1 先明确题目边界

面试官问“设计实时语音助手”，可能有几种范围。

第一，设计智能音箱或车载语音助手。

用户说一句话，系统实时识别、理解、回复。重点是唤醒、低延迟、噪声鲁棒、打断和设备端能力。

第二，设计电话客服助手。

用户通过电话实时对话，系统识别语音、查业务系统、生成回答并合成语音。重点是稳定性、SLA、人工转接、录音合规和任务完成率。

第三，设计实时会议助手。

系统实时转写、总结、提取待办，也可以语音问答。重点是多说话人、长会话、低打扰和隐私。

第四，设计通用实时语音 API。

为上层应用提供 ASR、LLM、TTS、打断、状态管理和流式传输能力。

面试开头可以这样澄清：

我先按通用实时语音助手设计，支持用户通过麦克风连续说话，系统流式 ASR、调用 LLM、流式 TTS 播放回答，并支持用户打断、噪声处理、会话状态、安全审核和监控。电话客服、车载和会议助手可以作为场景化扩展。

## 8.2 实时语音助手的核心难点

实时语音助手和文本聊天最大的区别是时间。

文本聊天中，用户输入是完整文本，系统可以等模型完整生成后再返回。语音助手中，用户正在说话，系统要判断什么时候开始听、什么时候认为一句话结束、什么时候开始回复、用户打断时如何停止当前回复。

核心难点包括：

1. 低延迟：用户说完后要很快听到回应。
2. 流式处理：音频、ASR、LLM、TTS 都要增量处理。
3. 打断机制：用户说话时要能打断系统正在播放的语音。
4. 噪声鲁棒：环境可能有背景声、回声、口音和多人说话。
5. 状态管理：对话是连续的，不能每句话都孤立处理。
6. 安全审核：输出在播放前或播放中要控制风险。
7. 成本控制：实时系统可能长连接、长时间占用资源。

## 8.3 功能需求

一个实时语音助手通常需要以下功能。

| 模块 | 功能 |
| --- | --- |
| 音频接入 | 麦克风采集、编码、上传、网络传输 |
| VAD | 检测用户是否在说话、端点判断 |
| ASR | 流式识别、增量 transcript、置信度、标点 |
| NLU/LLM | 意图理解、对话生成、工具调用、状态更新 |
| TTS | 流式语音合成、音色、语速、情感控制 |
| 播放控制 | 播放队列、暂停、恢复、打断、回声处理 |
| 会话管理 | 多轮上下文、用户状态、任务状态 |
| 安全审核 | 输入审核、输出审核、敏感信息处理 |
| 监控评估 | 延迟、ASR 准确率、打断成功率、任务成功率 |
| 降级转人工 | 超时、失败、高风险或用户要求时转人工 |

第一版可以先支持单用户、单语言、云端 ASR/LLM/TTS；后续再扩展多语言、端侧唤醒、多人分离和离线模式。

## 8.4 非功能需求

### 8.4.1 低延迟

实时语音的延迟可以拆成：

1. 音频采集和上传延迟。
2. VAD 端点检测延迟。
3. ASR 首字和最终结果延迟。
4. LLM 首 token 延迟。
5. TTS 首音频帧延迟。
6. 播放缓冲延迟。

用户体验最敏感的是“用户说完到助手开始说”的时间。通常要尽量控制在一两秒以内，越短越自然。

### 8.4.2 稳定性

长连接可能断开，音频包可能丢失，ASR 或 TTS 服务可能抖动。系统要支持重连、缓冲、超时和降级。

### 8.4.3 自然性

自然性包括语音流畅、不要过长停顿、不要抢话、能被打断、语速合适、回答长度适合语音场景。

### 8.4.4 安全与隐私

语音包含生物特征、环境信息、个人隐私和敏感对话。系统要有录音授权、加密、保留期限、访问控制和审计。

### 8.4.5 成本

实时语音可能长期保持连接，ASR、LLM、TTS 都持续消耗资源。要按连接时长、音频时长、token 和 TTS 字符数统计成本。

## 8.5 容量估算

假设设计一个语音客服助手：

1. 峰值同时在线通话 1 万路。
2. 每路平均通话 5 分钟。
3. 用户说话占 40% 时间，助手说话占 30% 时间，其余是静音和等待。
4. ASR 实际处理音频约 4000 路并发。
5. TTS 实际合成约 3000 路并发。
6. 每通电话平均 10 轮对话，每轮 1 次 LLM 调用。

需要估算：

1. 长连接数量。
2. 音频流带宽。
3. ASR 并发。
4. LLM 请求峰值。
5. TTS 并发。
6. 录音和日志存储。

如果每路音频压缩后约几十 kbps，1 万路并发就是数百 Mbps 级别带宽。录音如果长期保存，存储也会很大。因此实时语音系统必须考虑流媒体基础设施和生命周期管理。

## 8.6 高层架构

实时语音助手可以分成客户端、流媒体接入、语音处理、对话编排、模型服务、播放控制和监控评估。

| 组件 | 职责 |
| --- | --- |
| Client | 音频采集、播放、VAD 辅助、打断事件 |
| Realtime Gateway | WebSocket/WebRTC、鉴权、会话路由 |
| Audio Stream Service | 音频分片、缓冲、重连、时序管理 |
| VAD Service | 语音活动检测、端点检测、静音判断 |
| ASR Service | 流式转写、partial/final transcript |
| Dialogue Orchestrator | 管理对话状态、调用 LLM、处理打断 |
| LLM Service | 生成文本回复、工具调用、状态更新 |
| TTS Service | 流式语音合成、音色和播放格式 |
| Playback Controller | 播放队列、暂停、取消、barge-in |
| Safety Service | 输入输出审核、隐私和风险控制 |
| Session Store | 会话状态、转写、上下文、事件 |
| Monitoring | 延迟、错误、质量、成本、安全事件 |

核心链路是：客户端采集音频并通过 WebSocket 或 WebRTC 发送；网关建立实时会话；VAD 判断用户说话段落；ASR 增量输出 transcript；Orchestrator 根据 partial 或 final transcript 调用 LLM；LLM 生成文本；TTS 流式合成音频；客户端播放；如果用户中途说话，触发打断并取消当前 TTS 或 LLM。

## 8.7 音频接入

### 8.7.1 WebSocket 还是 WebRTC

WebSocket 实现简单，适合应用内实时音频上传和服务端返回音频流。WebRTC 更适合低延迟音视频通信，内置抖动缓冲、回声消除和网络适应能力。

如果是浏览器或移动端实时语音助手，可以优先考虑 WebRTC；如果是后端 API 或简单语音流，WebSocket 也可以。

### 8.7.2 音频格式

常见参数包括采样率、声道、编码格式和帧大小。

帧太小会增加网络包和服务端处理开销；帧太大会增加延迟。常见做法是 20ms 到 100ms 一帧。

### 8.7.3 网络抖动

网络会抖动、丢包和重连。Audio Stream Service 要按时间戳重排音频帧，处理丢包和缓冲，避免 ASR 输入混乱。

### 8.7.4 客户端预处理

客户端可以做降噪、回声消除、自动增益控制和本地 VAD，但服务端不能完全依赖客户端结果，因为不同设备质量差异很大。

## 8.8 VAD 与端点检测

VAD 是 Voice Activity Detection，判断用户是否在说话。

### 8.8.1 为什么 VAD 重要

VAD 决定什么时候开始 ASR、什么时候认为用户说完、什么时候触发 LLM。VAD 太敏感会把噪声当语音，VAD 太保守会漏掉用户说话。

### 8.8.2 端点检测

端点检测判断一句话是否结束。它影响响应速度。

如果等待太久，助手反应慢；如果过早截断，用户没说完系统就开始回答。

可以结合静音时长、ASR 语义完整性、语气和标点来判断。

### 8.8.3 Barge-in

Barge-in 指用户在助手说话时插话打断。

系统要检测到用户语音，停止当前播放，取消或暂停 TTS 和 LLM，并把用户新输入作为更高优先级处理。

这是实时语音体验的关键。

## 8.9 流式 ASR

ASR 要支持 partial 和 final 两类结果。

### 8.9.1 Partial transcript

Partial transcript 是增量识别结果，可能会变化。它可以用于提前理解用户意图，但不能完全信任。

### 8.9.2 Final transcript

Final transcript 是端点后确认的结果，更稳定。正式调用业务工具或生成关键回答时，最好基于 final transcript。

### 8.9.3 标点和大小写

ASR 输出如果没有标点，会影响 LLM 理解。可以使用标点恢复或直接使用支持标点的流式 ASR。

### 8.9.4 专有词和热词

客服、医疗、金融、企业内部场景有大量专有词。ASR 要支持热词、词表或领域适配。

### 8.9.5 多语言和口音

多语言助手要做语言检测和模型路由。口音和噪声场景要有鲁棒性评估。

## 8.10 LLM 对话编排

### 8.10.1 什么时候调用 LLM

最简单策略是等 final transcript 后调用 LLM。这样稳定，但延迟较高。

更激进策略是基于 partial transcript 预判意图，提前检索或准备回答。这样降低延迟，但可能误判。

生产中可以折中：对简单明显意图做预取，对高风险或不确定意图等 final。

### 8.10.2 回复长度控制

语音回复不宜太长。用户听长答案比读长文本更累。

Prompt 中要要求模型使用短句、先给结论、必要时询问是否继续详细解释。

### 8.10.3 多轮状态

语音对话中用户常说“对，就这个”“帮我改一下”“再说一遍”。系统要维护上下文、指代关系和当前任务状态。

### 8.10.4 工具调用

语音客服或助手常需要查订单、建工单、设日程。工具调用要复用 Agent 平台的权限、审计和人工确认机制。

高风险动作不能仅凭语音误识别结果直接执行。需要确认，例如“我将取消订单 123，是否确认？”

## 8.11 流式 TTS

### 8.11.1 TTS 首包延迟

TTS 不应等完整回答生成后才合成。可以把 LLM 输出按句子或短片段流式送给 TTS，尽快开始播放。

### 8.11.2 分句策略

过早分句可能导致语音不自然，过晚分句增加延迟。可以按标点、语义边界和最大等待时间切分。

### 8.11.3 音色和情绪

不同业务需要不同音色和语气。客服需要稳定清晰，陪伴助手可能需要更自然，车载助手需要简短明确。

### 8.11.4 TTS 取消

用户打断时，TTS 任务要能取消，播放队列要清空或暂停。

## 8.12 打断机制

打断机制是实时语音助手的核心体验。

### 8.12.1 打断链路

1. 客户端或服务端 VAD 检测到用户开始说话。
2. Playback Controller 立即停止播放当前音频。
3. Orchestrator 标记当前回复被打断。
4. 取消后续 TTS 合成。
5. 如果 LLM 仍在生成，取消或忽略后续 token。
6. 新的用户语音进入 ASR 和对话链路。

### 8.12.2 回声问题

助手正在播放时，麦克风可能收进助手自己的声音。需要回声消除，否则系统会把自己的回复当成用户说话。

### 8.12.3 打断误触发

背景声、咳嗽、键盘声可能误触发打断。可以结合音量、持续时间、声纹或 ASR 置信度过滤。

### 8.12.4 打断后的上下文

被打断的回答是否进入上下文，要谨慎。如果用户没听完，系统不应假设用户已接收完整信息。可以记录已播放到哪个片段。

## 8.13 安全审核

实时语音安全比文本更难，因为输出是边合成边播放。

### 8.13.1 输入审核

ASR 转写文本进入安全审核。高风险请求可以拒答、转人工或要求确认。

### 8.13.2 输出审核

如果等完整文本审核后再 TTS，延迟高；如果边生成边播，风险高。

可以分层处理：

1. 对低风险场景做流式轻量审核。
2. 对高风险场景先生成完整文本并审核后播放。
3. 对敏感领域使用更保守 prompt 和策略。
4. 对工具写操作强制确认。

### 8.13.3 语音隐私

录音、转写和声纹都属于敏感数据。要有用户授权、加密存储、访问控制、保留期限和删除机制。

### 8.13.4 合规边界

本章讨论合法授权范围内的实时语音助手设计，不提供窃听、绕过录音授权、规避通话告知、声纹追踪或泄露敏感语音内容的方法。

## 8.14 监控与评估

实时语音助手要监控性能、质量、体验、安全和成本。

### 8.14.1 延迟指标

包括：

1. 音频上行延迟。
2. VAD 端点延迟。
3. ASR partial 延迟。
4. ASR final 延迟。
5. LLM 首 token 延迟。
6. TTS 首包延迟。
7. 用户说完到助手开口延迟。

### 8.14.2 质量指标

包括 ASR WER、意图识别准确率、任务成功率、TTS 自然度、用户满意度、人工转接率。

### 8.14.3 体验指标

包括打断成功率、误打断率、抢话率、长停顿率、重复追问率、用户提前挂断率。

### 8.14.4 安全指标

包括违规输入率、违规输出率、隐私命中、未经确认高风险动作拦截率。

### 8.14.5 成本指标

包括每分钟 ASR 成本、每轮 LLM token 成本、TTS 字符成本、连接占用成本和失败会话成本。

## 8.15 高可用与降级

常见故障包括：

1. ASR 服务延迟升高。
2. LLM 服务超时。
3. TTS 服务不可用。
4. 网络断开。
5. VAD 误判。
6. 工具调用失败。
7. 安全审核服务超时。

降级策略包括：

1. ASR 慢时提示用户稍等或转文本输入。
2. LLM 慢时播放短等待语，例如“我查一下”。
3. TTS 不可用时返回文本。
4. 网络断开后支持重连和会话恢复。
5. 工具失败时解释失败并转人工。
6. 安全服务失败时采用保守策略。
7. 电话客服场景可转人工坐席。

## 8.16 成本优化

实时语音成本优化包括：

1. VAD 减少静音段 ASR 计算。
2. 简单意图使用小模型。
3. 短回复减少 TTS 字符数。
4. 缓存常见话术音频。
5. 对长时间静默连接自动断开。
6. ASR、LLM、TTS 分层模型路由。
7. 批量或并发优化 TTS。
8. 限制会话最长时长。

成本优化不能牺牲通话告知、隐私保护和安全审核。

## 8.17 面试答题模板

如果面试官问“设计实时语音助手”，可以这样回答。

第一，澄清范围。

我先按通用实时语音助手设计，支持用户通过麦克风连续对话，系统流式 ASR、LLM 生成、流式 TTS 播放，并支持打断、会话状态、安全审核和监控。

第二，定义指标。

核心指标包括用户说完到助手开口延迟、ASR WER、LLM 首 token、TTS 首包、打断成功率、误打断率、任务成功率、安全违规率和每分钟成本。

第三，做容量估算。

根据同时在线连接数、用户说话占比、助手说话占比、平均轮数，估算 ASR 并发、LLM QPS、TTS 并发、带宽和录音存储。

第四，画架构。

系统包括 Client、Realtime Gateway、Audio Stream Service、VAD Service、ASR Service、Dialogue Orchestrator、LLM Service、TTS Service、Playback Controller、Safety Service、Session Store 和 Monitoring。

第五，讲核心链路。

客户端采集音频并通过 WebRTC 或 WebSocket 上传，VAD 做语音活动和端点检测，ASR 输出 partial/final transcript，Orchestrator 调用 LLM，LLM 文本流进入 TTS，TTS 音频流返回客户端播放。

第六，展开关键难点。

重点讲低延迟拆解、流式 ASR、流式 TTS、打断机制、回声消除、多轮状态、安全审核和降级。

第七，讲安全和隐私。

录音和转写需要授权、加密、访问控制和保留期限；高风险工具动作必须确认；输出审核要根据风险选择流式或完整审核。

## 8.18 常见面试追问

### 8.18.1 实时语音助手和普通语音转文字有什么区别？

语音转文字只负责 ASR；实时语音助手还要做对话状态、LLM 生成、TTS 播放、打断、低延迟、工具调用、安全审核和体验控制。

### 8.18.2 为什么 VAD 很重要？

VAD 决定什么时候听、什么时候认为用户说完、什么时候打断系统回复。它直接影响延迟、抢话、误打断和成本。

### 8.18.3 如何降低用户说完到助手开口的延迟？

可以优化端点检测、使用流式 ASR、基于 partial transcript 预取、降低 LLM TTFT、分句流式 TTS、减少播放缓冲，并避免不必要的同步安全或工具调用。

### 8.18.4 用户打断时系统怎么处理？

检测到用户语音后立即停止播放，取消 TTS，取消或忽略 LLM 后续输出，记录已播放内容，把新语音作为高优先级输入进入 ASR 和对话链路。

### 8.18.5 如何避免助手把自己的声音识别成用户？

需要回声消除、播放音频参考、VAD 过滤和 ASR 置信度判断。客户端和服务端可以结合处理。

### 8.18.6 高风险业务能不能边生成边播？

要谨慎。低风险闲聊可以流式审核边播，高风险医疗、金融、法律或写操作场景最好先生成完整文本，审核或确认后再播放。

### 8.18.7 电话客服场景如何转人工？

当用户要求、连续失败、情绪升级、权限不足、高风险操作或系统故障时转人工。转人工时要把摘要、已验证身份、历史对话和失败原因传给坐席。

### 8.18.8 录音和转写如何保存？

要基于用户授权和业务合规设置保留期限。录音、转写、摘要和日志都要加密、访问控制、审计，并支持删除或脱敏。

## 8.19 本章小结

实时语音助手的核心价值，是把大模型能力变成自然、低延迟、可打断的语音交互。

本章要记住几句话：

1. 实时语音助手不是 ASR 加 LLM 加 TTS 的简单串联，而是流式、低延迟、可打断的交互系统。
2. 延迟要拆成音频上传、VAD、ASR、LLM、TTS 和播放缓冲分别优化。
3. VAD 和端点检测决定助手是否抢话、慢半拍或误打断。
4. ASR partial 可以降低延迟，但 final transcript 更可靠。
5. TTS 要流式合成，并支持取消。
6. Barge-in 是实时语音体验的核心能力。
7. 录音、转写和声纹都属于敏感数据，需要授权、加密、保留期限和审计。
8. 高风险场景不能为了低延迟牺牲安全审核和人工确认。

如果面试中被问到实时语音助手，最好的回答是先拆延迟链路，再讲流式 ASR、LLM、TTS 如何协同，最后重点讲打断、回声、状态、安全和降级。
