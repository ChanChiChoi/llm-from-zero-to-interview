# 第十二章：设计安全审核系统

面试中如果让你“设计一个大模型安全审核系统”，考察的不是会不会在模型前面加一个关键词黑名单，也不是会不会调用一个内容安全 API。真正考察的是：你是否理解大模型安全风险的复杂性，能否把输入审核、输出审核、策略引擎、模型分类器、人工复核、红队数据、误杀控制、审计追踪和产品体验组织成一个可长期演进的系统。

安全审核系统是大模型产品的“安全门”和“刹车系统”。它既要拦截明显违规、越权、隐私泄露和高风险内容，也要避免把正常用户请求大量误杀。它既要足够严格，又不能让产品变得不可用。它既要覆盖已知风险，又要能通过红队、线上反馈和人工审计不断发现新风险。

本章重点：输入输出审核、policy engine、red team 数据、误杀控制、审计。

合规边界：本章讨论合法合规的大模型安全治理、内容审核、隐私保护、风险识别和系统架构，不提供绕过安全审核、构造攻击提示词、规避平台策略或滥用模型能力的方法。

## 12.1 先明确题目边界

“安全审核系统”这个题目很宽。它可能指内容审核、模型安全、企业权限、未成年人保护、广告合规、金融合规、医疗合规、代码安全、数据泄露检测，也可能指红队平台。

面试时可以先限定范围：

```text
我先按一个面向大模型应用的安全审核系统来设计。系统部署在大模型服务链路中，对用户输入、模型输出、工具调用、检索内容和附件进行风险识别，根据 policy engine 决定放行、拒绝、改写、降级、人工复核或记录审计。系统要支持低延迟在线审核、异步深度审核、策略版本管理、误杀控制、红队数据回流和审计追踪。
```

这个范围适合大模型系统设计面试，因为它覆盖 ChatGPT 类服务、RAG、Agent、企业助手、多模态应用和开放 API 的核心安全问题。

## 12.2 为什么大模型安全审核比传统内容审核更难

传统内容审核主要处理文本、图片或视频是否违规。例如是否涉黄、涉暴、辱骂、广告、垃圾内容。这类问题虽然也复杂，但审核对象通常是用户直接发布的内容。

大模型安全审核更难，原因有几个。

第一，大模型是交互式系统。

用户不是提交一段静态内容，而是在多轮对话中逐步引导模型。某句话单看可能正常，结合上下文后可能变成高风险请求。

第二，风险不仅来自输入，也来自输出。

用户输入可能正常，但模型输出可能泄露隐私、生成错误建议、违反业务规则、引用未授权内容或产生危险指导。

第三，模型可以调用工具。

Agent 系统可能访问数据库、发邮件、下订单、调用代码执行环境或修改文件。安全审核不再只是文本分类，还要审核意图、权限、动作和后果。

第四，风险和业务场景强相关。

同一句话在不同场景下风险不同。医学、金融、法律、教育、招聘、未成年人场景对安全策略要求完全不同。

第五，误杀成本很高。

如果安全系统过于保守，正常问题被拒绝，用户体验会变差，企业工作流会被打断。安全系统不能只追求拦截率，还要控制误杀率。

第六，攻击和滥用会不断变化。

平台上线后，用户会发现新的绕开方式、新的上下文组合、新的工具滥用路径。安全系统必须是持续学习和迭代的系统。

所以面试回答要强调：安全审核不是单点模型，而是规则、分类器、策略、人工、审计和数据闭环组成的治理系统。

## 12.3 需求分析

### 12.3.1 功能需求

一个大模型安全审核系统通常需要支持：

| 模块 | 功能 |
| --- | --- |
| 输入审核 | 对用户 prompt、上下文、附件、图片、音频、代码进行风险识别 |
| 输出审核 | 对模型回答、引用内容、生成文件、图片、视频进行风险识别 |
| 工具调用审核 | 审核 tool call 的意图、参数、权限和外部影响 |
| Policy Engine | 根据风险类别、用户等级、场景、地区、模型和策略版本做决策 |
| 风险分类器 | 毒性、隐私、越权、违法违规、危险建议、歧视、垃圾内容等识别 |
| PII 检测 | 检测和脱敏姓名、电话、邮箱、地址、证件号、密钥等敏感信息 |
| 人工复核 | 对高风险、低置信、争议样本进入人工审核或专家复核 |
| 红队数据 | 管理红队样本、线上攻击样本、失败样本和回归测试集 |
| 误杀申诉 | 支持用户反馈、业务申诉、审核复盘和策略调整 |
| 审计追踪 | 记录输入、输出、风险判断、策略版本、决策和操作日志 |

一个典型流程是：

1. 用户提交请求。
2. 输入审核服务检查 prompt、附件和上下文。
3. Policy Engine 决定放行、拒绝、改写、降级或人工复核。
4. 请求进入模型推理链路。
5. 如果模型要调用工具，工具调用审核检查权限和风险。
6. 模型生成输出。
7. 输出审核服务检查回答内容、引用、文件和多模态结果。
8. Policy Engine 再次决策：直接返回、部分遮盖、替换为安全回复、人工复核或阻断。
9. 记录完整审计日志。
10. 高风险和误杀样本回流到红队、标注和评估系统。

### 12.3.2 非功能需求

安全审核系统的非功能需求非常关键。

| 需求 | 说明 |
| --- | --- |
| 低延迟 | 在线聊天不能因为审核增加过多延迟 |
| 高可用 | 审核系统故障不能导致全站不可用，也不能默认完全放开 |
| 可解释 | 审核决策要能说明风险类别、策略和原因 |
| 可配置 | 不同业务、地区、用户群和模型版本使用不同策略 |
| 可追溯 | 每次放行、拒绝、改写和复核都能追踪策略版本 |
| 可迭代 | 新风险出现后能快速加规则、更新分类器、回放测试 |
| 误杀可控 | 能监控 false positive，支持申诉和策略复盘 |
| 安全隔离 | 高风险数据、隐私数据和审计日志要严格权限控制 |

面试中要特别强调低延迟和误杀控制。安全系统如果太慢，用户体验崩；如果太严，业务不可用；如果太松，风险不可控。

## 12.4 风险分类体系

设计安全审核系统前，必须先定义风险 taxonomy。没有清晰分类，策略和报表都会混乱。

一个通用大模型风险分类可以包括：

| 风险类别 | 说明 |
| --- | --- |
| 有害内容 | 暴力、仇恨、骚扰、自伤等高风险内容 |
| 隐私泄露 | PII、账号、密钥、企业机密、内部文档泄露 |
| 越权请求 | 要求访问无权限数据、绕过权限、执行未授权操作 |
| 危险建议 | 可能造成现实伤害、财产损失或安全事故的指导 |
| 非法违规 | 违法交易、欺诈、作弊、恶意滥用等 |
| 偏见歧视 | 基于敏感属性的不公平或歧视性输出 |
| 幻觉风险 | 医疗、金融、法律等高风险领域的无依据结论 |
| 儿童安全 | 涉及未成年人保护的高风险内容 |
| 版权和合规 | 未授权复现受保护内容、违规改写或分发 |
| 垃圾和滥用 | 批量广告、刷量、钓鱼、垃圾生成 |
| Prompt 注入 | 试图让模型忽略系统指令、泄露上下文或操控工具 |
| 数据外泄 | 通过 RAG、工具或上下文泄露不该展示的信息 |

这个分类要支持层级化。例如隐私泄露可以细分为普通 PII、敏感 PII、密钥凭证、商业秘密。危险建议可以按风险等级分为低风险科普、中风险需要免责声明、高风险必须拒绝。

风险分类不是纯技术问题，还需要法务、安全、产品、政策和业务共同维护。

## 12.5 核心架构

一个可落地的大模型安全审核系统可以设计成下面的结构。

```text
              ┌───────────────────────────┐
              │        Client / API        │
              └─────────────┬─────────────┘
                            │
                    ┌───────▼───────┐
                    │ Safety Gateway │
                    │ 输入审核/限流/鉴权│
                    └───────┬───────┘
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
┌───────▼───────┐   ┌───────▼───────┐   ┌───────▼───────┐
│ Classifier Svc │   │ Policy Engine │   │ PII / DLP Svc  │
│ 风险分类器     │   │ 策略决策       │   │ 隐私和数据防泄漏│
└───────┬───────┘   └───────┬───────┘   └───────┬───────┘
        │                   │                   │
        └───────────┬───────┴───────────┬───────┘
                    │                   │
            ┌───────▼───────┐   ┌───────▼───────┐
            │ Model Runtime  │   │ Tool Firewall │
            │ LLM/RAG/Agent  │   │ 工具调用审核   │
            └───────┬───────┘   └───────┬───────┘
                    │                   │
            ┌───────▼───────────────────▼───────┐
            │         Output Safety Filter        │
            │ 输出审核 改写 遮盖 拒绝 人工复核       │
            └───────┬───────────────────────────┘
                    │
        ┌───────────▼───────────┐
        │ Audit / Review / Data  │
        │ 审计 人工复核 红队数据  │
        └───────────────────────┘
```

核心组件包括：

1. Safety Gateway：入口层，负责鉴权、限流、输入审核和请求上下文构造。
2. Classifier Service：风险分类器集合，包括文本、多模态、PII、越权、prompt 注入等分类器。
3. Policy Engine：根据分类器结果、业务场景、用户等级、策略版本和地区规则做决策。
4. PII / DLP Service：负责敏感信息检测、脱敏和数据防泄漏。
5. Tool Firewall：审核 Agent 工具调用的权限、参数、风险和外部副作用。
6. Output Safety Filter：审核模型输出，必要时拒绝、改写、遮盖或转人工。
7. Human Review System：对低置信、高风险和申诉样本做人工复核。
8. Red Team Data Platform：管理红队样本、失败案例、回归测试集和策略评估。
9. Audit Log Service：记录每次安全决策的证据、策略、版本和操作人。

## 12.6 输入审核设计

输入审核发生在模型推理之前。它的目标是识别用户请求是否高风险，以及是否需要特殊处理。

输入审核不能只看当前 prompt，还要看：

1. 当前用户输入。
2. 历史多轮上下文。
3. 系统提示词和开发者指令。
4. 用户身份、租户、权限和年龄分层。
5. 附件、图片、音频、文件和代码。
6. 当前业务场景，例如客服、医疗、企业知识库、代码助手。
7. 请求的工具和外部动作。

输入审核可以输出结构化结果：

```json
{
  "risk_level": "medium",
  "categories": ["privacy", "potential_data_exfiltration"],
  "confidence": 0.87,
  "signals": {
    "pii_detected": true,
    "tool_requested": "database_search",
    "contains_sensitive_terms": false
  },
  "recommended_action": "allow_with_redaction"
}
```

输入审核的常见决策包括：

| 决策 | 含义 |
| --- | --- |
| Allow | 正常放行 |
| Allow with Warning | 放行但提示风险或限制范围 |
| Redact | 对输入中的敏感信息脱敏后再进入模型 |
| Rewrite | 将请求改写为安全、明确的形式 |
| Refuse | 直接拒绝处理高风险请求 |
| Escalate | 转人工或专家复核 |
| Rate Limit | 对疑似滥用用户限流 |

输入审核的关键是低延迟。可以用轻量规则和小模型做快速判断，对高风险或低置信请求再走异步深度审核。

## 12.7 输出审核设计

输出审核发生在模型生成之后、返回用户之前。它非常重要，因为模型可能生成不应返回的内容。

输出审核要检查：

1. 是否泄露系统提示词、开发者指令或内部策略。
2. 是否包含 PII、密钥、账号、内部链接或客户信息。
3. 是否给出危险、违法或不当建议。
4. 是否在高风险领域给出无依据结论。
5. 是否违反版权、合规或业务规则。
6. 是否含有歧视、仇恨、骚扰或不适当内容。
7. 是否引用了用户无权限访问的 RAG 文档。
8. 是否包含工具调用结果中的敏感字段。

输出审核有一个工程难点：流式输出。

ChatGPT 类产品通常边生成边返回。如果等完整回答生成后再审核，延迟高；如果边生成边放行，可能已经把风险内容返回给用户。

常见方案有三种：

1. Chunk 审核：模型每生成一小段，审核后再发送。
2. 延迟缓冲：先缓存若干 token 或句子，审核通过后再流出。
3. 双层审核：输入审核严格控制高风险请求，输出审核做快速流式扫描，完整输出后再做异步审计。

对高风险场景，例如医疗、金融、未成年人、企业数据查询，可以牺牲一些流式体验，采用更严格的输出审核和延迟返回。

输出审核的处理动作包括：

| 动作 | 说明 |
| --- | --- |
| Pass | 原样返回 |
| Mask | 遮盖敏感字段 |
| Safe Completion | 替换为安全回复 |
| Regenerate | 要求模型按安全约束重新生成 |
| Partial Return | 返回安全部分，删除高风险部分 |
| Human Review | 暂缓返回，进入人工审核 |
| Block | 阻断返回 |

## 12.8 Policy Engine 设计

Policy Engine 是安全审核系统的决策中心。分类器只告诉你“可能有什么风险”，Policy Engine 决定“应该怎么处理”。

为什么不能只靠分类器？因为同样的风险分在不同业务下动作不同。

例如：

1. 普通闲聊中出现电话号码，可能需要遮盖。
2. 用户在自己的客服工单中查询电话号码，可能允许展示部分信息。
3. 标注员在受控环境下审核隐私样本，可能允许完整展示但禁止导出。
4. 未成年人场景中，同样内容可能需要更严格限制。

### 12.8.1 Policy 输入

Policy Engine 的输入可以包括：

1. 风险分类结果。
2. 风险置信度。
3. 用户身份、角色、租户和权限。
4. 业务场景。
5. 地区和合规要求。
6. 模型版本。
7. 工具调用类型。
8. 数据来源和文档权限。
9. 历史行为和滥用信号。
10. 策略版本和实验分组。

### 12.8.2 Policy 输出

输出通常是结构化决策：

```json
{
  "decision": "redact_and_allow",
  "reason_codes": ["PII_DETECTED", "USER_HAS_PARTIAL_ACCESS"],
  "redaction_policy": "mask_phone_email",
  "audit_level": "high",
  "message_template": "privacy_safe_response_v2",
  "policy_version": "policy_2026_05_26"
}
```

### 12.8.3 Policy 表达方式

Policy 可以用几种方式实现：

| 方式 | 优点 | 缺点 |
| --- | --- | --- |
| 硬编码规则 | 简单快速 | 难维护，发布风险高 |
| 配置化规则 | 可热更新，适合业务策略 | 表达复杂逻辑时容易膨胀 |
| 规则引擎 | 支持复杂条件和优先级 | 学习成本和调试成本高 |
| 模型决策 | 能处理模糊语义 | 可解释性和稳定性较弱 |
| 混合方式 | 工程上最常用 | 需要良好治理和测试 |

实际系统通常采用混合方式：高风险硬规则兜底，中低风险由分类器加策略规则决策，复杂灰区进入人工或异步复核。

### 12.8.4 策略版本管理

策略必须版本化。每次决策都要记录 policy version，否则后续无法解释为什么当时放行或拒绝。

策略上线前要做：

1. 离线回放：用历史流量和红队集测试拦截率和误杀率。
2. 小流量灰度：先对部分用户或内部流量生效。
3. 指标监控：观察拒绝率、申诉率、人工复核率和用户满意度。
4. 快速回滚：发现误杀或漏杀异常时能回退策略。

## 12.9 分类器和规则的组合

安全审核系统通常不会只用一种方法。

### 12.9.1 规则

规则适合处理确定性强的问题，例如：

1. API key、token、身份证号、手机号等格式化 PII。
2. 内部域名、密钥前缀、特定业务字段。
3. 已知违规词、黑名单 URL、垃圾广告模板。
4. 频率异常、批量请求、短时间重复提交。

规则优点是快、可解释、成本低。缺点是容易漏掉变体，也容易误杀。

### 12.9.2 传统分类器

传统分类器适合稳定类别，例如毒性、辱骂、垃圾内容、成人内容、低质量文本。它们成本低、延迟小，适合在线链路。

### 12.9.3 LLM Judge

LLM Judge 适合判断复杂语义，例如是否越权、是否遵循安全策略、是否存在隐含风险、是否引用证据。缺点是成本高、延迟高、稳定性需要评估。

### 12.9.4 多阶段审核

常见设计是多阶段：

```text
快速规则 -> 轻量分类器 -> 策略引擎 -> 大模型复核 -> 人工复核
```

大多数低风险请求只走前两层。只有高风险、低置信或高价值请求才进入大模型复核或人工复核。

这样可以平衡延迟、成本和安全性。

## 12.10 工具调用安全

Agent 场景下，安全审核系统必须覆盖工具调用。否则模型可能生成看似正常的文本，却执行了危险动作。

工具调用安全要检查：

1. 用户是否有权限调用该工具。
2. 工具参数是否包含敏感信息。
3. 工具结果是否包含用户无权访问的数据。
4. 动作是否有外部副作用，例如发邮件、付款、删除数据、修改配置。
5. 是否需要用户二次确认。
6. 是否超过频率、金额、范围或租户边界。
7. 是否受 prompt 注入影响。

可以把工具分成不同风险等级：

| 工具类型 | 风险 | 处理方式 |
| --- | --- | --- |
| 只读公开查询 | 低 | 普通鉴权和日志 |
| 企业知识库查询 | 中 | 权限过滤、引用审核、DLP |
| 代码执行 | 高 | 沙箱、资源限制、网络隔离 |
| 发邮件/发消息 | 高 | 用户确认、收件人校验、内容审核 |
| 支付/下单/删除 | 极高 | 强认证、二次确认、人工或规则审批 |

Tool Firewall 可以在工具调用前后都工作：

1. 调用前审核意图和参数。
2. 调用中限制权限和资源。
3. 调用后审核返回结果是否可展示。

## 12.11 RAG 场景的数据外泄防护

RAG 系统有特殊风险：模型可能把检索到的内部文档泄露给无权限用户，或受到文档中的恶意指令影响。

安全设计包括：

1. 检索前鉴权：只检索用户有权限访问的文档。
2. 检索后过滤：再次检查 chunk 权限和敏感字段。
3. 引用级审计：回答中记录引用了哪些文档和段落。
4. 输出 DLP：检查回答是否泄露敏感字段。
5. Prompt 注入防护：把检索文档视为不可信内容，不允许其覆盖系统指令。
6. 最小必要原则：只把回答所需片段放入上下文，不把整篇敏感文档塞给模型。
7. 租户隔离：向量索引、缓存和日志都要隔离租户。

面试中要强调：RAG 权限不能只在前端做，也不能只依赖模型“不要泄露”。权限过滤必须发生在检索和生成链路中。

## 12.12 误杀控制

安全系统最容易犯的错误是只讲拦截，不讲误杀。实际产品里误杀会直接影响用户体验、业务收入和内部效率。

### 12.12.1 误杀来源

常见误杀来源包括：

1. 关键词规则过粗。
2. 分类器没有理解上下文。
3. 高风险领域的正常科普被拒绝。
4. 企业内部合法查询被当成数据外泄。
5. 多语言、方言、缩写和行业术语识别错误。
6. 策略变更没有充分回放测试。

### 12.12.2 误杀指标

需要监控：

1. 总拒绝率。
2. 各风险类别拒绝率。
3. 用户申诉率。
4. 申诉成功率。
5. 人工复核改判率。
6. 正常任务中断率。
7. 不同业务、语言、地区的误杀差异。

### 12.12.3 降低误杀的方法

可以采取：

1. 多级风险分层，不要把所有风险都直接拒绝。
2. 对中低风险请求采用改写、脱敏、提示边界，而不是拒绝。
3. 使用上下文感知分类器，而不是只看关键词。
4. 对高价值企业场景提供人工复核和申诉。
5. 策略上线前做历史流量回放。
6. 分业务、分语言、分地区设定阈值。
7. 保留安全替代回答，让用户知道可以怎么问。

一个成熟系统应该有“安全可用”的理念：尽量把不安全请求转化为安全帮助，而不是简单说不。

## 12.13 红队数据和回归测试

红队数据是安全系统持续进化的燃料。

红队数据来源包括：

1. 内部安全团队构造的测试样本。
2. 外部红队活动样本。
3. 线上被拦截的高风险样本。
4. 线上漏拦后复盘样本。
5. 用户举报和申诉样本。
6. 评估平台发现的安全失败样本。
7. 新策略灰度中的争议样本。

这些样本要进入专门的数据集，并打上标签：风险类别、严重程度、是否应拒绝、期望安全回复、触发策略、模型版本和复盘结论。

红队数据有两个用途。

第一，用于离线评估。

每次安全分类器、policy、模型或 prompt 变更前，都要在红队集上回归测试，防止旧问题复发。

第二，用于训练和对齐。

高质量红队样本可以进入 SFT、安全偏好训练或拒答训练，让模型本身更安全。

要注意，红队数据本身可能包含敏感内容，必须严格权限控制、脱敏和审计。

## 12.14 人工复核系统

不是所有安全决策都适合自动完成。人工复核适合：

1. 高风险但分类器置信度低的样本。
2. 用户申诉样本。
3. 企业客户重要业务被拒绝的样本。
4. 新风险类别样本。
5. 政策边界模糊样本。
6. 需要专家判断的医疗、法律、金融、未成年人场景。

人工复核系统要支持：

1. 样本脱敏展示。
2. 风险分类和策略原因展示。
3. 原始上下文和模型输出查看。
4. 审核员决策：维持、改判、升级、加入红队集。
5. 审核规范和案例库。
6. 审核员一致性和质量评估。
7. 结果回流到策略和训练数据。

人工复核不是兜底垃圾桶。它必须和策略、标注、评估系统形成闭环，否则只会堆积大量无法处理的 case。

## 12.15 审计和可追溯

安全审核系统必须记录完整审计链路。

一次请求至少应记录：

| 字段 | 说明 |
| --- | --- |
| request_id | 请求 ID |
| user_id / tenant_id | 用户和租户，必要时脱敏 |
| input_hash | 输入哈希，避免明文长期保存 |
| risk_scores | 各分类器风险分 |
| categories | 命中的风险类别 |
| policy_version | 策略版本 |
| decision | 放行、拒绝、脱敏、复核等 |
| model_version | 模型版本 |
| tool_calls | 工具调用和审批结果 |
| output_hash | 输出哈希或脱敏内容 |
| reviewer | 人工复核人 |
| timestamp | 时间 |

日志要注意两点。

第一，审计日志本身也可能包含敏感信息。不能为了审计而无限制保存原文。可以采用哈希、脱敏、分级存储和保留周期。

第二，审计日志要防篡改。高风险系统可以使用 append-only 日志、访问审计和权限隔离。

审计的价值不仅是合规，也是工程排障：当用户反馈“为什么我的请求被拒绝”，团队可以追踪到具体分类器、策略版本和规则。

## 12.16 可用性和故障策略

安全审核系统在主链路上，故障策略很重要。

如果审核服务挂了怎么办？

不能简单回答“默认放行”或“默认拒绝”。更合理的是按风险分层：

1. 低风险业务：可以降级到基础规则和日志审计。
2. 中风险业务：可以限制部分功能，保留核心问答。
3. 高风险业务：默认拒绝或转人工。
4. 工具调用和外部副作用：默认阻断高风险动作。
5. 企业敏感数据查询：默认收紧权限。

系统要支持：

1. 多副本部署。
2. 熔断和超时。
3. 本地规则缓存。
4. 策略配置缓存。
5. 异步补审。
6. 降级模式。
7. 故障审计。

安全系统的可用性目标不是“永远不拒绝”，而是“在故障时仍然保持可控风险”。

## 12.17 延迟和性能优化

安全审核会增加链路延迟。优化方法包括：

1. 先用规则和轻量模型过滤大部分请求。
2. 分类器并行调用，而不是串行调用。
3. 对低风险请求跳过重型审核。
4. 对高风险场景使用更严格但可接受的审核链路。
5. 缓存用户权限、策略配置和低风险模板结果。
6. 对长文本分块审核。
7. 流式输出使用缓冲和 chunk 审核。
8. 异步审计不阻塞正常返回。

面试中可以给一个目标：普通聊天输入审核增加几十毫秒到一两百毫秒，复杂输出审核和 LLM Judge 只用于高风险或低置信场景。

## 12.18 多租户和企业场景

企业大模型产品里，安全审核要支持多租户。

不同企业可能有不同策略：

1. 金融客户要求更严格的投资建议审核。
2. 医疗客户要求更严格的诊断和处方边界。
3. 教育客户要求未成年人保护。
4. 代码助手客户要求密钥泄露和许可证风险检测。
5. 企业知识库要求文档权限和租户隔离。

多租户设计要支持：

1. 租户级策略配置。
2. 租户级红队和误杀统计。
3. 租户级数据隔离和审计。
4. 租户级管理员申诉和策略审批。
5. 默认平台策略兜底，租户策略只能更严格或在允许范围内配置。

不能让租户随意关闭核心安全策略，否则平台风险不可控。

## 12.19 指标体系

安全审核系统需要同时看安全效果和用户体验。

### 12.19.1 安全指标

1. 漏拦率。
2. 高风险样本拦截率。
3. 红队集通过率。
4. PII 泄露率。
5. 越权工具调用阻断率。
6. 安全事故数。

### 12.19.2 体验指标

1. 拒绝率。
2. 误杀率。
3. 用户申诉率。
4. 申诉成功率。
5. 平均审核延迟。
6. 对话完成率。

### 12.19.3 运营指标

1. 人工复核队列长度。
2. 人工复核 SLA。
3. 策略变更次数。
4. 回滚次数。
5. 红队样本新增量。
6. 分类器版本覆盖率。

指标要分业务、分语言、分地区、分用户类型看。总体拒绝率正常，不代表某个语言或客户没有被严重误杀。

## 12.20 常见追问

### 12.20.1 输入审核和输出审核哪个更重要？

两者都重要。输入审核可以提前拦截高风险请求，降低模型生成风险；输出审核可以防止模型自己产生不该返回的内容。只做输入审核会漏掉模型幻觉、隐私泄露和工具结果外泄；只做输出审核会增加成本和延迟，也可能在工具执行前拦不住危险动作。

### 12.20.2 如何处理流式输出审核？

可以使用缓冲加 chunk 审核。低风险请求快速流式返回，高风险场景延迟更多 token 或句子进行审核。若中途发现风险，停止输出并替换为安全回复。同时完整输出进入异步审计，用于后续复盘。

### 12.20.3 如何降低误杀？

用多级动作替代一刀切拒绝。低风险放行，中风险脱敏、改写、提示边界，高风险拒绝或人工复核。上线前用历史流量回放，线上监控申诉成功率和人工改判率，按业务和语言调阈值。

### 12.20.4 如何防止 RAG 泄露企业数据？

检索前做权限过滤，检索后再次校验 chunk 权限，生成时只放入最小必要上下文，输出时做 DLP 审核，回答中记录引用文档，并保证向量索引、缓存和日志租户隔离。

### 12.20.5 如何设计策略灰度？

先离线回放红队集和历史流量，再内部灰度，然后小流量线上灰度。监控拒绝率、误杀率、申诉率、人工复核率和安全漏拦样本。如果异常，快速回滚到上一策略版本。

### 12.20.6 如何让安全审核系统持续变好？

建立数据闭环。线上拦截、漏拦、申诉、人工复核、红队测试和评估失败样本都进入安全数据集。安全团队定期更新策略、分类器、标注规范和回归测试集。每次变更都通过离线回放和灰度验证。

## 12.21 面试回答模板

面试中可以按这个顺序回答：

1. 澄清范围：面向大模型应用，覆盖输入、输出、工具调用、RAG 和多租户策略。
2. 定义风险：隐私、越权、有害内容、危险建议、违法违规、偏见、数据外泄、prompt 注入。
3. 给出架构：Safety Gateway、Classifier、PII/DLP、Policy Engine、Tool Firewall、Output Filter、Human Review、Audit、Red Team Data。
4. 讲主流程：输入审核、策略决策、模型推理、工具审核、输出审核、审计记录、数据回流。
5. 深入 Policy Engine：分类器只产出信号，策略引擎结合用户、业务、地区、权限和风险等级做动作。
6. 深入误杀控制：分级动作、上下文判断、申诉、人工复核、历史回放和灰度发布。
7. 深入红队闭环：红队样本、线上失败、人工复核进入安全数据集，用于回归测试和训练。
8. 深入审计和隐私：记录策略版本和决策原因，同时保护审计日志本身。

一个简洁回答可以是：

```text
我会把大模型安全审核系统设计成主链路上的安全网，而不是单个关键词过滤器。请求进入后先经过 Safety Gateway 做鉴权、限流和输入审核，规则、PII 检测和轻量分类器并行产出风险信号，再由 Policy Engine 结合用户权限、业务场景、地区和策略版本决定放行、脱敏、拒绝、降级或人工复核。模型推理过程中，Tool Firewall 审核工具调用权限和参数；模型输出后，Output Filter 做隐私、危险内容、越权引用和合规检查。所有决策记录 policy version、risk score、model version 和审计日志。线上漏拦、误杀、申诉和红队样本回流到标注、评估和策略系统，用于持续迭代。
```

## 12.22 本章小结

安全审核系统是大模型产品能否长期上线运营的关键基础设施。

本章要点如下：

1. 大模型安全审核比传统内容审核更复杂，因为它涉及多轮上下文、模型输出、工具调用、RAG、权限和业务场景。
2. 安全审核系统要覆盖输入审核、输出审核、工具调用审核、PII/DLP、Policy Engine、人工复核、红队数据和审计。
3. Policy Engine 是决策中心，分类器只提供风险信号，最终动作要结合用户、权限、业务、地区和策略版本。
4. 输入审核负责提前识别风险，输出审核负责防止模型生成或泄露不该返回的内容。
5. Agent 和 RAG 场景必须特别关注工具副作用、权限校验和数据外泄。
6. 误杀控制和安全拦截同样重要，成熟系统应采用分级动作和申诉复盘，而不是一刀切拒绝。
7. 红队数据、线上失败、人工复核和评估结果要形成安全数据闭环。
8. 审计日志要记录完整决策链路，同时保护日志中的敏感信息。

面试中最重要的一句话是：

```text
大模型安全审核系统不是一个过滤器，而是围绕风险识别、策略决策、误杀控制、审计追踪和数据闭环构建的长期治理平台。
```
