# 第三章：设计训练平台

这一章讨论大模型系统设计中的另一类核心题：如何设计一个训练平台。

训练平台和“写一个训练脚本”不是一回事。训练脚本解决的是单次实验如何跑起来；训练平台解决的是大量团队、数据、模型、GPU、实验和产物如何稳定、可复现、可审计、可扩展地运转。

一个成熟训练平台要支持数据管道、任务提交、资源调度、分布式训练、checkpoint、实验追踪、评估、模型注册、权限控制、成本统计和故障恢复。对大模型来说，还要特别关注 GPU 集群、网络通信、长时间作业、断点续训、数据版本、训练稳定性和安全合规。

本章重点：数据管道、任务调度、分布式训练、checkpoint、实验追踪、资源管理。

## 3.1 先明确题目边界

面试官问“设计训练平台”，可能有几种范围。

第一，设计通用机器学习训练平台。

这类平台支持数据集、训练任务、实验记录、模型产物和部署，类似企业内部 MLOps 平台。

第二，设计大模型预训练平台。

这类平台关注海量数据、数百到数千 GPU、长时间训练、checkpoint、容错、通信性能和稳定性。

第三，设计大模型微调平台。

这类平台关注多用户提交 SFT、LoRA、DPO、RLHF 等任务，关注多租户、数据权限、资源配额、自动评估和模型注册。

第四，设计研究实验平台。

这类平台关注快速试验、配置管理、实验对比、复现、日志可视化和结果归档。

面试开头可以这样澄清：

我先按面向大模型团队的训练平台设计，既支持预训练和继续训练，也支持 SFT、LoRA、DPO 等微调任务。核心目标是让用户提交训练任务后，平台能完成数据准备、资源调度、分布式启动、日志监控、checkpoint 管理、评估和模型注册。如果时间有限，我会重点展开数据版本、GPU 调度、分布式训练、checkpoint 和实验追踪。

## 3.2 功能需求

训练平台的功能可以分成十类。

| 模块 | 功能 |
| --- | --- |
| 用户与权限 | 用户、团队、项目、角色、配额 |
| 数据管理 | 数据集注册、版本、血缘、预处理、权限 |
| 任务提交 | 配置训练任务、镜像、代码、超参、资源 |
| 资源调度 | GPU、CPU、内存、网络、存储、队列优先级 |
| 分布式训练 | 多机多卡启动、rank 管理、通信配置 |
| 日志监控 | loss、吞吐、显存、GPU 利用率、错误日志 |
| Checkpoint | 保存、恢复、清理、转换、最佳 checkpoint |
| 实验追踪 | 参数、代码版本、数据版本、指标、产物 |
| 自动评估 | 训练中评估、训练后评估、回归测试 |
| 模型注册 | 模型版本、元数据、审批、发布到推理平台 |

第一版平台可以先支持离线训练和手动评估，后续再扩展自动 HPO、RLHF pipeline、弹性训练和训练数据闭环。

## 3.3 非功能需求

训练平台的非功能需求很关键，因为训练作业贵、慢、容易失败。

### 3.3.1 可复现性

平台必须记录代码版本、镜像版本、数据版本、训练配置、随机种子、依赖环境、模型初始化、checkpoint 和评估配置。

如果一次实验效果很好但无法复现，平台价值就很低。

### 3.3.2 可扩展性

平台要能从单机单卡扩展到多机多卡，从小模型微调扩展到大模型继续训练。

扩展性不只是加机器，还包括数据加载、存储带宽、网络通信、调度队列和日志系统能否承受规模增长。

### 3.3.3 容错性

大模型训练可能跑几天到几周，任何节点、网络、存储或代码问题都可能导致失败。平台必须支持 checkpoint、自动重试、故障定位和断点续训。

### 3.3.4 多租户隔离

多个团队共享 GPU 集群时，要有资源配额、优先级、数据权限、镜像隔离、日志访问控制和成本归属。

### 3.3.5 成本可控

训练成本很高。平台要记录每个任务消耗的 GPU 小时、存储、网络和失败重试成本，并支持预算、配额和空闲资源回收。

## 3.4 容量估算

训练平台容量估算要看用户、作业、GPU、存储和网络。

假设一个中型大模型团队：

1. 100 名算法工程师和研究员。
2. 每人每天提交 3 个训练或评估任务。
3. 每日 300 个任务。
4. 其中 70% 是小规模实验，20% 是中等微调，10% 是大规模训练。
5. 集群有 512 张 GPU。

需要估算：

1. 同时运行多少任务。
2. 每类任务平均占用几张 GPU。
3. 平均运行时长。
4. 队列等待时间。
5. checkpoint 和日志每天产生多少数据。

例如一个 7B 模型 SFT 任务可能占用 8 张 GPU 跑数小时；一个 70B 继续训练任务可能占用 64 到 256 张 GPU 跑数天；一个预训练任务可能占用更多 GPU 跑数周。

存储估算也很重要。训练平台会存：

1. 原始数据。
2. 预处理数据。
3. tokenized 数据。
4. checkpoint。
5. optimizer state。
6. 日志和 tensorboard event。
7. 评估输出。

Checkpoint 往往是大头。模型参数、optimizer state 和训练状态加起来可能是模型权重大小的数倍。如果每隔一小时保存一次，不做清理，很快会占满存储。

## 3.5 高层架构

训练平台可以分成以下组件。

| 组件 | 职责 |
| --- | --- |
| Web Console / CLI / SDK | 用户提交任务、查看实验和管理模型 |
| API Gateway | 鉴权、限流、请求入口 |
| Project Service | 项目、团队、权限、配额 |
| Dataset Service | 数据集注册、版本、权限、血缘 |
| Job Service | 训练任务定义、状态管理、重试 |
| Scheduler | 资源调度、队列、优先级、抢占 |
| Cluster Manager | Kubernetes、Slurm 或自研集群管理 |
| Training Launcher | 生成分布式启动命令和环境变量 |
| Artifact Store | checkpoint、模型权重、日志、评估产物 |
| Metadata Store | 实验参数、指标、版本、血缘 |
| Monitoring | GPU、任务、日志、loss、吞吐监控 |
| Evaluation Service | 自动评估、回归测试、报告生成 |
| Model Registry | 模型注册、审批、发布、回滚 |

核心数据流是：用户通过 Web、CLI 或 SDK 提交训练任务；平台校验权限和配额；解析代码、镜像、数据和资源配置；调度器分配 GPU；Launcher 启动分布式训练；训练过程上报日志、指标和 checkpoint；任务完成后触发评估；合格模型进入 Model Registry，后续可发布到推理平台。

## 3.6 数据管道设计

训练平台最容易被低估的是数据管道。大模型训练效果很大程度取决于数据，而平台要保证数据可追踪、可复现、可授权。

### 3.6.1 数据集注册

Dataset Service 要记录：

1. dataset_id。
2. 数据来源。
3. 数据版本。
4. 数据 schema。
5. 数据大小。
6. 许可和授权信息。
7. 质量报告。
8. PII 和安全扫描结果。
9. 创建者和更新时间。

训练任务不应直接引用一个随时变化的目录，而应引用明确的数据版本。

### 3.6.2 数据预处理

预处理流程包括清洗、去重、过滤、格式转换、分词、打包、混合比例和 train/valid/test 划分。

平台要记录每一步处理配置和输出版本。否则数据一变，实验结果就无法解释。

### 3.6.3 Tokenized 数据

大模型训练通常会把文本预先 tokenize 并打包成固定长度序列，提高训练吞吐。

Tokenized 数据与 tokenizer 强绑定。如果 tokenizer 版本变化，旧 tokenized 数据可能不能复用。因此数据版本要记录 tokenizer 名称、版本和 special token 配置。

### 3.6.4 数据权限

不同团队的数据可能不能互相访问。企业数据、用户日志、医疗金融数据、代码仓库数据都可能有权限限制。

训练平台必须在数据集注册、任务提交和运行时都检查权限，不能只在前端隐藏数据集名称。

### 3.6.5 数据血缘

数据血缘回答：某个模型到底用了哪些数据训练？这些数据来自哪里？经过哪些过滤？是否包含后来要求删除的数据？

这对审计、复现、版权、隐私和 unlearning 都很重要。

## 3.7 任务提交设计

训练任务需要包含足够完整的配置。

一个任务配置可以包括：

1. 任务名称和项目。
2. 代码仓库和 commit id。
3. Docker 镜像。
4. 启动命令。
5. 模型初始化 checkpoint。
6. 数据集版本。
7. 训练超参。
8. 资源需求。
9. 分布式策略。
10. checkpoint 策略。
11. 评估配置。
12. 环境变量和 secrets 引用。

不要把 secrets 明文写进任务配置。应该使用 secret manager 注入，并控制访问权限。

### 3.7.1 Web、CLI 和 SDK

研究员常用 CLI 或 SDK 提交任务，平台管理员和产品化用户可能用 Web Console。

CLI 适合自动化，Web 适合查看状态和比较实验。SDK 适合把平台能力集成到 notebook 或内部工具。

### 3.7.2 配置模板

为了减少错误，平台可以提供任务模板，例如：

1. SFT 模板。
2. LoRA 模板。
3. DPO 模板。
4. 继续预训练模板。
5. 评估任务模板。

模板不是为了限制研究，而是提供默认正确的配置，减少重复踩坑。

## 3.8 资源调度

训练平台最核心的系统问题之一是 GPU 调度。

### 3.8.1 调度目标

调度器要同时考虑：

1. GPU 数量和类型。
2. 节点拓扑。
3. 网络带宽。
4. 存储带宽。
5. 任务优先级。
6. 队列公平性。
7. 资源碎片。
8. 抢占和恢复。

大模型训练不是任意 GPU 拼起来就行。多机训练对网络拓扑敏感，跨机通信慢会显著降低 MFU 和吞吐。

### 3.8.2 队列和优先级

可以设计多队列：

1. 交互实验队列：小任务、快速反馈、低资源。
2. 微调队列：中等资源，常规优先级。
3. 大训练队列：大规模长任务，需要审批。
4. 评估队列：可延迟，但要保障发布前执行。
5. 紧急队列：线上问题修复或高优先级任务。

队列要防止两种问题：大任务长期霸占集群，小任务永远排不上；或者小任务太多导致大任务永远凑不齐资源。

### 3.8.3 资源配额

资源配额可以按团队、项目和用户设置。

配额包括 GPU 小时、最大并发任务、最大 GPU 数、存储容量和预算。超配额任务可以排队、降级或需要审批。

### 3.8.4 抢占

抢占可以提高集群利用率，但会影响训练任务。

如果支持抢占，必须配套 checkpoint。低优先级任务被抢占后，应能从最近 checkpoint 恢复，而不是从头开始。

## 3.9 分布式训练启动

训练平台要屏蔽一部分分布式训练复杂性。

### 3.9.1 Launcher 做什么

Training Launcher 负责：

1. 分配 master address 和 port。
2. 设置 world size、rank、local rank。
3. 注入 NCCL 和网络环境变量。
4. 挂载数据和 checkpoint 存储。
5. 启动每个节点上的训练进程。
6. 收集日志和退出码。

### 3.9.2 支持的训练框架

平台可以支持 PyTorch DDP、FSDP、DeepSpeed、Megatron-LM、Ray Train 或内部训练框架。

面试中不需要实现这些框架，但要说明平台如何抽象：用户选择分布式策略，平台生成对应启动配置和资源需求。

### 3.9.3 通信和网络

大模型训练常常受通信影响。平台要暴露或监控：

1. NCCL 错误。
2. all-reduce 时间。
3. pipeline bubble。
4. 跨节点带宽。
5. 节点拓扑。

当训练吞吐低时，不能只看 GPU 利用率，还要分析数据加载、通信和存储。

## 3.10 Checkpoint 设计

Checkpoint 是训练平台的生命线。

### 3.10.1 保存什么

完整 checkpoint 通常包括：

1. 模型参数。
2. Optimizer state。
3. Scheduler state。
4. Grad scaler state。
5. Random state。
6. 当前 step 和 epoch。
7. 数据读取位置。
8. 分布式切分元数据。

只保存模型权重不一定能严格断点续训，因为 optimizer 和随机状态丢失后训练轨迹会变化。

### 3.10.2 保存频率

保存太频繁会影响训练吞吐和存储成本；保存太少则故障损失大。

可以按时间和 step 结合，例如每 N step 保存轻量 checkpoint，每 M 小时保存完整 checkpoint，关键评估点保存 best checkpoint。

### 3.10.3 Checkpoint 清理

平台要自动清理：

1. 保留最近 K 个。
2. 保留 best K 个。
3. 保留里程碑 checkpoint。
4. 清理失败任务的临时文件。
5. 对长期归档做压缩或转冷存储。

否则 checkpoint 会迅速耗尽存储。

### 3.10.4 格式转换

训练 checkpoint 和推理权重格式可能不同。平台可以支持从训练格式转换成推理格式，例如合并分片、去掉 optimizer state、转换 dtype、生成 tokenizer 和 config 元数据。

## 3.11 实验追踪

实验追踪回答四个问题：

1. 这个结果是怎么来的？
2. 和上一个实验差在哪里？
3. 能不能复现？
4. 产物在哪里？

### 3.11.1 记录什么

实验系统要记录：

1. 参数配置。
2. 代码 commit。
3. 镜像版本。
4. 数据版本。
5. 初始模型。
6. 训练指标。
7. 评估指标。
8. 资源消耗。
9. checkpoint 列表。
10. 日志和错误。

### 3.11.2 指标可视化

常见训练指标包括 loss、learning rate、gradient norm、吞吐、MFU、显存、数据加载时间、通信时间、eval loss。

大模型训练中，吞吐和稳定性与 loss 一样重要。一个 loss 正常但 MFU 很低的任务，可能在浪费大量 GPU。

### 3.11.3 实验对比

平台要支持按实验组对比。例如同一数据版本下不同 learning rate、不同 LoRA rank、不同 beta、不同模型初始化的效果。

没有实验对比能力，研究员只能靠表格和截图，容易出错。

## 3.12 自动评估和模型注册

训练完成不是结束，还要评估和注册。

### 3.12.1 自动评估

自动评估可以包括：

1. 通用能力评估。
2. 领域任务评估。
3. 安全评估。
4. 偏好评估。
5. 回归测试。
6. 成本和延迟评估。

不同模型进入不同评估套件。SFT 模型需要指令遵循和安全评估，代码模型需要编译和单测，RAG 相关模型需要检索或引用评估。

### 3.12.2 评估触发

评估可以在训练中周期性触发，也可以训练完成后触发。

训练中评估帮助早停和发现崩溃；训练后评估决定是否进入模型注册。

### 3.12.3 模型注册

Model Registry 记录：

1. model_id 和 version。
2. 来源训练任务。
3. 基座模型。
4. 数据版本。
5. 训练配置。
6. 评估报告。
7. 安全报告。
8. 权重位置。
9. 推理配置。
10. 审批状态。

模型注册后，才可以被推理平台灰度发布。训练平台和推理平台之间要通过模型注册解耦。

## 3.13 监控与告警

训练平台监控分为集群、任务、训练质量和成本四类。

### 3.13.1 集群监控

包括 GPU 利用率、显存、温度、ECC 错误、节点健康、网络带宽、存储吞吐、队列长度。

### 3.13.2 任务监控

包括任务状态、运行时长、重启次数、日志错误、数据加载速度、checkpoint 时间。

### 3.13.3 训练质量监控

包括 loss 是否 NaN、loss 是否异常震荡、gradient norm 是否爆炸、eval 指标是否退化、吞吐是否低于预期。

### 3.13.4 成本监控

包括每个任务 GPU 小时、每个团队资源消耗、失败任务浪费、checkpoint 存储成本和空闲 GPU 时间。

告警要分级。一个小实验失败不应半夜叫醒全组；一个千卡训练 loss NaN 或 checkpoint 连续失败就必须高优先级告警。

## 3.14 故障恢复

训练平台常见故障包括：

1. 代码 bug。
2. OOM。
3. 数据坏样本。
4. NCCL 通信失败。
5. GPU 硬件错误。
6. 节点重启。
7. 存储写入失败。
8. Checkpoint 损坏。
9. Loss NaN。

恢复策略包括：

1. 自动重试短暂基础设施错误。
2. OOM 时提示降低 batch size 或启用 checkpointing。
3. 节点故障时重新调度并从 checkpoint 恢复。
4. Checkpoint 写入采用临时文件加原子提交。
5. 保存多个 checkpoint 防止最后一个损坏。
6. NaN 检测后停止任务并保留现场。
7. 日志和核心指标保留，方便定位。

不要对所有失败都盲目重试。代码 bug 和数据 bug 重试只会浪费 GPU。

## 3.15 安全与合规

训练平台的安全合规包括数据安全、代码安全、模型安全和访问控制。

### 3.15.1 数据安全

训练数据可能包含隐私、商业机密、版权内容或敏感领域数据。平台要支持授权检查、脱敏、访问控制、数据保留期限和删除请求追踪。

### 3.15.2 代码和镜像安全

用户提交的训练代码运行在集群上，可能访问数据和网络。平台要做镜像扫描、权限隔离、网络策略、secret 管理和最小权限。

### 3.15.3 模型产物安全

模型权重可能是核心资产。要控制下载权限、发布权限、外部导出审批和水印或签名机制。

### 3.15.4 审计

平台要记录谁用了哪些数据，训练了哪个模型，导出了哪些 checkpoint，发布了哪个版本。

审计日志不能随意删除。

## 3.16 成本优化

训练平台成本优化可以从资源、作业和产物三方面做。

### 3.16.1 资源层

1. 提高 GPU 利用率。
2. 减少资源碎片。
3. 使用队列和配额。
4. 空闲任务自动回收。
5. 低优先级任务使用可抢占资源。

### 3.16.2 作业层

1. 先小规模 smoke test，再大规模训练。
2. 自动检测明显错误配置。
3. 训练早停。
4. 失败任务快速终止。
5. 超参搜索设置预算。

### 3.16.3 产物层

1. Checkpoint 生命周期管理。
2. 日志采样和压缩。
3. 冷热存储分层。
4. 删除无用中间产物。

一个很实用的设计是：大规模训练任务提交前，平台自动要求先通过小数据和少量 step 的 smoke test，避免配置错误直接浪费几百张 GPU。

## 3.17 面试答题模板

如果面试官问“设计训练平台”，可以这样回答。

第一，澄清范围。

我先按大模型团队内部训练平台设计，支持预训练、继续训练、SFT、LoRA、DPO 和评估任务。目标是让用户可复现、可监控、可恢复地使用 GPU 集群完成训练。

第二，定义指标。

核心指标包括任务成功率、队列等待时间、GPU 利用率、训练吞吐、checkpoint 成功率、实验可复现性、资源成本和故障恢复时间。

第三，画架构。

系统包括 Web/CLI/SDK、API Gateway、Project Service、Dataset Service、Job Service、Scheduler、Cluster Manager、Training Launcher、Artifact Store、Metadata Store、Monitoring、Evaluation Service 和 Model Registry。

第四，讲数据流。

用户提交任务后，平台校验权限和配额，解析代码、镜像、数据和资源配置，调度 GPU，启动分布式训练，训练中上报日志和 checkpoint，完成后自动评估并注册模型。

第五，展开关键难点。

重点讲数据版本和血缘、GPU 调度、分布式启动、checkpoint、实验追踪、故障恢复和成本控制。

第六，讲安全合规。

数据、代码、镜像、secret、checkpoint 和模型权重都要有访问控制和审计。

第七，讲扩展。

后续可以支持弹性训练、自动超参搜索、RLHF pipeline、多集群调度、模型自动发布和训练评估闭环。

## 3.18 常见面试追问

### 3.18.1 训练平台和推理平台最大的区别是什么？

训练平台面向长时间、资源密集、可中断恢复的离线作业，重点是数据版本、资源调度、分布式训练、checkpoint 和实验追踪。推理平台面向在线请求，重点是低延迟、高可用、batching、KV cache、模型路由和自动扩缩容。

### 3.18.2 为什么数据版本这么重要？

因为模型效果高度依赖数据。如果训练任务引用的是一个会变化的目录，后续无法复现实验，也无法解释模型用了哪些数据、是否包含敏感数据或污染数据。

### 3.18.3 Checkpoint 为什么不能只保存模型权重？

只保存模型权重可以用于推理或微调初始化，但不能严格恢复训练。断点续训还需要 optimizer state、scheduler state、随机状态、step、数据读取位置和分布式切分信息。

### 3.18.4 如何提高 GPU 利用率？

可以优化调度减少碎片，改善数据加载，使用合适的并行策略，提高 batch size，减少通信瓶颈，监控 MFU，自动发现空闲任务，并用队列和抢占提高整体利用率。

### 3.18.5 如何处理大任务和小任务的公平性？

可以设置多队列、配额、优先级和资源预留。小任务需要快速反馈，大任务需要凑齐资源。调度器要避免大任务长期霸占，也避免大任务永远排不上。

### 3.18.6 训练任务失败后如何恢复？

先判断失败类型。基础设施故障可以自动重试并从 checkpoint 恢复；代码 bug、数据 bug、配置错误不应盲目重试。平台要保留日志、错误、checkpoint 和环境信息，帮助定位。

### 3.18.7 如何防止训练平台被滥用？

通过用户权限、项目隔离、资源配额、镜像扫描、网络限制、secret 管理、数据访问控制、审计日志和预算告警来控制。

### 3.18.8 模型训练完后如何进入线上？

训练完成后先触发自动评估和安全评估，生成报告。通过审批后进入 Model Registry，再由推理平台按灰度策略加载和发布。训练平台不应直接覆盖线上模型。

## 3.19 本章小结

训练平台的核心价值，是把不稳定、昂贵、难复现的训练过程，变成可提交、可调度、可监控、可恢复、可追踪的工程系统。

本章要记住几句话：

1. 训练平台不是训练脚本，而是数据、资源、实验、产物和评估的完整系统。
2. 大模型训练平台最重要的难点是数据版本、GPU 调度、分布式训练、checkpoint 和实验追踪。
3. 数据必须可追踪，任务必须可复现，模型产物必须可审计。
4. Checkpoint 是长时间训练的生命线，要支持保存、恢复、清理和格式转换。
5. 调度器要在 GPU 利用率、队列公平性、大任务资源需求和抢占恢复之间取舍。
6. 训练指标不只看 loss，还要看吞吐、MFU、显存、通信和数据加载。
7. 用户日志和敏感数据进入训练前必须经过授权、脱敏、过滤和审计。
8. 训练完成后要通过评估和 Model Registry，再进入推理平台灰度发布。

如果面试中被问到训练平台，最好的回答不是讲“怎么写 PyTorch 代码”，而是讲清楚：一个训练任务从数据选择、配置提交、资源调度、分布式启动、运行监控、checkpoint、评估到模型注册，平台如何保证稳定、复现、安全和成本可控。
