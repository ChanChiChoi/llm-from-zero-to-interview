# 第十三章：设计模型路由与缓存系统

面试中如果让你“设计一个模型路由与缓存系统”，考察的不是会不会在多个模型之间随机选一个，也不是会不会给 API 前面加一层 Redis。真正考察的是：你是否理解大模型服务的核心矛盾，能否在质量、成本、延迟、吞吐、稳定性、隐私和可观测性之间做系统化取舍。

模型路由与缓存系统是大模型平台降本增效的关键层。它位于业务请求和底层推理服务之间，负责判断一个请求应该走哪个模型、是否可以复用已有结果、是否需要降级、是否要拆分、是否要异步处理、是否要触发安全和质量兜底。

本章重点：大小模型路由、prompt cache、response cache、embedding cache、成本优化。

合规边界：本章讨论合法合规的大模型服务架构、缓存策略、成本优化和工程治理，不讨论规避服务计费、绕过访问控制、泄露用户数据或利用缓存获取未授权内容的方法。

## 13.1 先明确题目边界

“模型路由与缓存系统”可以有很多解释。面试时不要一上来就画 Redis + 模型 API，而要先限定范围。

一个清晰的范围可以这样表达：

```text
我先按一个面向多模型大模型平台的在线路由与缓存系统来设计。系统接收业务请求，根据任务类型、用户等级、上下文长度、SLA、成本预算、模型能力、实时负载和历史质量表现，将请求路由到合适的模型或模型组合。同时系统支持 prompt cache、response cache、embedding cache、检索缓存和会话级缓存，以降低延迟和成本。系统还需要支持降级、灰度、A/B 实验、缓存隔离、隐私保护、监控告警和质量回流。
```

这个定义适合系统设计面试，因为它把问题从“怎么省钱”提升到了“怎么构建一个可持续运营的模型服务控制面”。

## 13.2 为什么需要模型路由与缓存

大模型产品上线后，很快会遇到几个现实问题。

第一，所有请求都走最强模型太贵。

一个通用聊天产品里，有些问题是复杂推理、代码生成、长文档理解，有些只是改写一句话、摘要一段文本、分类一个意图、抽取几个字段。如果所有请求都调用最大模型，质量可能不错，但成本会失控。

第二，所有请求都走小模型质量不稳定。

小模型在简单任务上很划算，但在复杂推理、多跳检索、专业知识、长上下文和高风险场景中容易出错。如果为了省成本盲目使用小模型，用户体验和业务可信度会下降。

第三，很多请求具有重复性。

企业知识库问答中，同一个政策、同一个产品说明、同一个报错、同一个 FAQ 会被反复问。编程助手中，同一段依赖安装错误、同一个 API 用法也会重复出现。如果每次都重新检索、重新 embedding、重新调用模型，就是浪费。

第四，推理服务有动态负载。

模型集群可能因为流量高峰、GPU 故障、批处理任务、版本发布而负载变化。路由层需要根据实时健康状况做切换、限流和降级。

第五，业务目标不同。

免费用户、付费用户、内部员工、企业客户、低风险任务、高风险任务的 SLA 和质量要求不同。系统不能只按一个固定模型处理所有请求。

所以，模型路由与缓存系统解决的是一个核心问题：在满足质量和安全底线的前提下，用尽可能低的成本和延迟完成请求。

## 13.3 先建立直觉：像机场调度系统

可以把大模型平台想象成一个机场。

不同模型像不同类型的飞机。小模型像短途支线飞机，便宜、起飞快、适合短距离任务。大模型像大型远程客机，贵、资源占用高，但能处理复杂任务。专用模型像货机、医疗专机、代码专机，适合特定场景。

模型路由层像塔台。它要决定：这趟任务该用哪架飞机、是否要转机、是否要等待、是否要改降备用机场、是否可以复用已有航线。

缓存像机场的常用航线和预处理能力。热门路线不需要每次重新规划，常见物资可以提前备好，常用乘客信息可以安全复用。

这个类比能帮助理解三个要点：

1. 路由不是简单选择最大或最小模型，而是动态调度。
2. 缓存不是只存最终答案，还可以存 prompt 前缀、embedding、检索结果、中间工具结果。
3. 成本优化不能牺牲安全、隐私和质量底线。

## 13.4 需求分析

### 13.4.1 功能需求

一个模型路由与缓存系统通常需要支持：

| 模块 | 功能 |
| --- | --- |
| 请求解析 | 解析任务类型、语言、上下文长度、用户等级、业务场景、SLA |
| 模型目录 | 维护可用模型、能力标签、价格、延迟、上下文长度、版本和状态 |
| 路由决策 | 根据规则、打分模型或策略引擎选择模型 |
| 大小模型协同 | 小模型先判别、改写、分类，大模型处理复杂请求 |
| Response Cache | 对可复用回答做缓存 |
| Prompt Cache | 复用长 prompt 或系统提示词的 KV cache / 前缀计算 |
| Embedding Cache | 缓存文本向量，避免重复 embedding |
| Retrieval Cache | 缓存检索结果、rerank 结果和文档片段 |
| 降级与熔断 | 模型超时、错误率升高、成本超预算时切换策略 |
| 实验平台 | 支持灰度、A/B、模型对比和策略回滚 |
| 监控审计 | 记录路由原因、缓存命中、成本、延迟、质量和错误 |

### 13.4.2 非功能需求

系统还要满足：

| 指标 | 要求 |
| --- | --- |
| 低延迟 | 路由层不能成为新的瓶颈 |
| 高可用 | 单个模型故障不影响整体服务 |
| 可解释 | 能解释为什么某请求走某模型、为什么命中缓存 |
| 可隔离 | 不同用户、租户、权限和隐私域的缓存必须隔离 |
| 可扩展 | 新模型、新缓存类型、新策略能快速接入 |
| 可观测 | 能按模型、业务、用户层级统计成本和质量 |
| 一致性 | 缓存更新、模型版本变更、策略变更要可控 |
| 安全性 | 不能因缓存泄露用户输入、企业知识或敏感信息 |

系统设计面试中，非功能需求往往决定方案深度。只讲功能，很容易像普通 CRUD；讲清 SLA、隔离、版本和观测，才能体现工程成熟度。

## 13.5 总体架构

一个典型架构可以分成八层：

1. API Gateway：接收请求，做认证、限流、租户识别、基础日志。
2. Request Analyzer：解析任务类型、上下文长度、语言、风险等级、实时性要求。
3. Cache Manager：查询 response cache、prompt cache、embedding cache、retrieval cache。
4. Router / Policy Engine：根据策略选择模型、工作流或降级路径。
5. Model Serving Layer：对接不同模型服务，包括自研模型、开源模型、第三方模型。
6. Quality Guard：对输出做基础质量、安全、格式和业务规则检查。
7. Telemetry Pipeline：采集成本、延迟、命中率、质量反馈、错误码和 trace。
8. Offline Optimizer：基于日志训练路由模型、更新缓存策略、分析成本和质量。

一个在线请求流程如下：

1. 用户提交请求。
2. Gateway 完成认证、租户识别和限流。
3. Request Analyzer 提取任务特征。
4. Cache Manager 先检查是否能命中安全可复用的缓存。
5. 如果命中 response cache，返回缓存答案并记录命中原因。
6. 如果未命中，Router 根据策略选择模型。
7. 如果是 RAG 请求，查询 embedding cache 和 retrieval cache。
8. 调用模型服务生成回答。
9. Quality Guard 做安全、格式、引用、长度和业务规则检查。
10. 结果写入合适的缓存，并把日志送入观测系统。

注意：缓存不一定在路由之前或之后固定发生。某些缓存应该在路由前查询，例如 response cache；某些缓存发生在路由后，例如模型专属 prompt cache；某些缓存贯穿整个链路，例如 embedding cache。

## 13.6 模型目录：路由系统的基础元数据

路由系统不能凭感觉选模型，必须有一个模型目录。模型目录记录每个模型的能力、成本、限制和运行状态。

常见字段包括：

| 字段 | 示例 |
| --- | --- |
| model_id | gpt-large-v3、qwen-coder-xx、small-classifier-v2 |
| version | 2026-05-20 |
| capability_tags | chat、code、math、vision、tool-use、long-context |
| context_window | 8k、32k、128k、1M |
| input_price | 每百万输入 token 成本 |
| output_price | 每百万输出 token 成本 |
| latency_p50/p95 | 延迟统计 |
| throughput | 每秒可处理 token 或请求数 |
| quality_score | 离线评测和线上反馈综合得分 |
| safety_score | 安全评测结果 |
| availability | 当前可用性和健康状态 |
| tenant_policy | 哪些租户或业务可使用 |
| deprecation_status | 是否即将下线 |

面试中可以强调：模型目录不是静态配置文件，而是路由系统的控制平面。它需要和监控、评测、发布系统联动。

例如，一个模型刚发布时只允许 5% 流量灰度；某个模型 p95 延迟上升时，路由层自动降低它的权重；某个模型在安全评测中回归时，高风险业务暂时禁用它。

## 13.7 路由策略一：规则路由

最简单、最稳定的方式是规则路由。

例如：

| 条件 | 路由策略 |
| --- | --- |
| prompt 长度小于 500 且任务是分类 | 小模型 |
| 任务是代码生成 | 代码专用模型 |
| 上下文超过 32k | 长上下文模型 |
| 用户是企业高级客户 | 高质量模型优先 |
| 请求是高风险医疗/法律建议 | 强模型 + 安全审核 + 引用要求 |
| 低峰期批处理任务 | 成本最低模型 |
| 主模型 p95 超时 | 备用模型 |

规则路由的优点是：

1. 简单可靠，容易上线。
2. 可解释性强，方便和业务沟通。
3. 出问题容易回滚。
4. 适合冷启动阶段。

缺点是：

1. 规则会越来越多，维护困难。
2. 无法细腻判断任务难度。
3. 容易出现边界条件。
4. 不能自动适应模型质量变化。

所以实际系统通常从规则路由起步，再逐步引入打分模型和在线学习。

## 13.8 路由策略二：任务难度分级

大小模型路由的关键不是“用小模型替代大模型”，而是判断请求难度。

常见难度特征包括：

| 特征 | 含义 |
| --- | --- |
| prompt 长度 | 长上下文通常更难 |
| 输出长度预估 | 长文生成更贵，也更容易出错 |
| 任务类型 | 分类、抽取通常简单；推理、代码、规划更难 |
| 是否需要工具 | 工具调用意味着状态和权限更复杂 |
| 是否需要多跳检索 | 多文档综合通常更难 |
| 是否有结构化约束 | 严格 JSON、表格、函数调用需要格式稳定性 |
| 历史失败率 | 某类请求之前经常失败，应升级模型 |
| 用户价值 | 高价值用户或关键业务更偏向强模型 |
| 风险等级 | 高风险场景需要更强模型和安全兜底 |

可以设计一个 difficulty_score：

```text
difficulty_score = f(task_type, prompt_len, context_len, retrieval_need, tool_need, risk_level, historical_failure)
```

然后按分数选择模型：

| 难度 | 策略 |
| --- | --- |
| 低 | 小模型直接处理 |
| 中 | 中模型处理，必要时失败升级 |
| 高 | 大模型处理 |
| 极高 | 大模型 + 检索 + 工具 + 人工或异步流程 |

这里的重点不是公式本身，而是方法论：先把请求可解释地分层，再把不同层映射到不同成本和质量的处理路径。

## 13.9 路由策略三：级联路由

级联路由是大小模型协同的常见方案。

它的流程是：

1. 小模型先判断任务类型、难度和风险。
2. 如果任务简单，小模型直接回答。
3. 如果小模型置信度低，升级到大模型。
4. 如果大模型回答后质量检查失败，可以重试、换模型或人工处理。

例如客服问答场景：

1. 小模型判断用户是否在问 FAQ。
2. 如果是标准 FAQ，走检索 + 模板化回答。
3. 如果是复杂投诉，走大模型总结上下文并生成回复建议。
4. 如果涉及退款、法律风险或敏感用户，转人工复核。

级联路由的优点是成本低，简单请求不用占用大模型。缺点是链路变长，错误可能在前面传播。

面试中可以说清一个关键点：级联系统必须有升级机制，不能让小模型的错误分类直接决定最终质量。常见升级信号包括低置信度、输出格式错误、安全风险、用户追问、历史差评、规则校验失败。

## 13.10 路由策略四：多模型投票与仲裁

对于高价值或高风险任务，可以使用多模型策略。

常见方式包括：

1. 并行调用多个模型，再由 judge 模型或规则选择答案。
2. 一个模型生成，另一个模型审查。
3. 小模型生成草稿，大模型修订。
4. 专用模型负责事实检索，通用模型负责表达。
5. 多个模型分别解决子问题，最后聚合。

这种策略能提高可靠性，但成本和延迟较高，所以不适合所有请求。

适用场景包括：

1. 企业合同摘要。
2. 金融研报分析。
3. 医疗知识解释。
4. 代码迁移和自动修复。
5. 高价值客户的关键工单。

不适用场景包括：

1. 普通闲聊。
2. 低价值批量改写。
3. 对实时性极高的简单请求。
4. 已有稳定模板可处理的请求。

这体现了系统设计中的基本原则：可靠性不是免费获得的，它要用成本和延迟购买。

## 13.11 Response Cache：缓存最终回答

Response Cache 是最容易理解的一类缓存：相同或近似相同的问题，直接返回已有答案。

适合缓存的请求包括：

1. FAQ 问答。
2. 静态文档解释。
3. 产品政策说明。
4. 标准错误码解释。
5. 固定格式转换。
6. 非个性化摘要。

不适合缓存的请求包括：

1. 包含个人隐私或企业机密的请求。
2. 强依赖实时状态的请求。
3. 需要个性化上下文的请求。
4. 高风险建议。
5. 创作型、强随机性或需要新颖性的请求。

Response Cache 的关键设计点是 cache key。

最简单的 key 是：

```text
hash(model_id + prompt + system_prompt + parameters)
```

但真实系统要复杂得多，通常还要加入：

1. 租户 ID。
2. 用户权限范围。
3. 知识库版本。
4. 模型版本。
5. prompt 模板版本。
6. 安全策略版本。
7. 输出语言。
8. temperature 等采样参数。

一个更安全的 key 可以表示为：

```text
hash(tenant_id, permission_scope, model_version, prompt_template_version, knowledge_base_version, normalized_query, generation_params)
```

注意，cache key 不是越短越好。过短会错误复用，导致数据泄露或答案过期；过长会降低命中率。系统设计的难点就是在安全和命中率之间找到平衡。

## 13.12 Exact Match 与 Semantic Cache

Response Cache 可以分为精确匹配和语义匹配。

精确匹配要求请求完全相同或规范化后相同。例如去掉多余空格、统一大小写、标准化标点。

优点是安全、可解释、误命中少。缺点是命中率有限。

语义缓存则允许“意思相近”的问题复用答案。例如：

```text
如何重置密码？
我忘记密码了怎么办？
密码找回在哪里？
```

它们可能都能命中同一条答案。

语义缓存通常使用 embedding 检索：

1. 对新 query 计算 embedding。
2. 在缓存库中检索相似 query。
3. 如果相似度超过阈值，且权限、版本、场景一致，则复用答案。
4. 如果低于阈值，则调用模型并写入缓存。

语义缓存的问题是误命中风险。两个问题看起来相似，但答案可能不同。

例如：

```text
如何关闭自动续费？
如何开启自动续费？
```

它们语义空间可能接近，但业务动作相反。如果错误复用，后果很严重。

因此语义缓存通常需要：

1. 高相似度阈值。
2. 意图分类一致。
3. 实体和槽位一致。
4. 权限和租户一致。
5. 知识版本一致。
6. 对高风险任务禁用或只做候选。
7. 必要时用轻量模型做 cache hit verification。

面试中讲到这里，会明显比“加 Redis 缓存”更成熟。

## 13.13 Prompt Cache：缓存前缀计算

Prompt Cache 通常指复用 prompt 前缀的计算结果，尤其是 Transformer 推理中的 KV cache。

大模型生成时，输入 prompt 会先经过模型计算得到中间状态。对于很长且重复的前缀，例如系统提示词、工具说明、企业知识库固定说明、多轮会话历史，如果每次都重新计算，会浪费大量算力。

典型场景包括：

1. 同一个 Agent 的系统 prompt 很长。
2. 同一个企业助手每次都带相同政策和工具说明。
3. 多轮会话中前面的历史不变，只追加新消息。
4. 长文档问答中多个问题共享同一文档前缀。
5. 批量任务使用相同模板处理不同输入。

Prompt Cache 的收益主要体现在：

1. 降低首 token 延迟。
2. 降低重复前缀计算成本。
3. 提升长上下文服务吞吐。
4. 改善多轮对话体验。

但它也有边界：

1. 只对重复前缀有用。
2. 通常和模型版本、权重、推理参数强绑定。
3. KV cache 占用显存或高速存储，容量有限。
4. 多租户隔离更复杂。
5. 前缀一旦变化，缓存可能失效。

Prompt Cache 的 key 通常包含：

```text
hash(model_id, model_version, tokenizer_version, prefix_tokens, inference_config, tenant_scope)
```

如果模型版本或 tokenizer 变化，旧 prompt cache 通常不能安全复用。

## 13.14 Embedding Cache：缓存向量计算

Embedding Cache 在 RAG、语义搜索、语义缓存和推荐系统中非常常见。

它缓存的是文本到向量的结果。

适合缓存的内容包括：

1. 知识库文档 chunk 的 embedding。
2. 用户高频 query 的 embedding。
3. FAQ 问题的 embedding。
4. 工单标题和摘要的 embedding。
5. 代码片段、图片描述或多模态特征。

Embedding Cache 的收益是避免重复调用 embedding 模型，降低延迟和成本。

但它有几个容易被忽视的问题。

第一，embedding 和模型版本强绑定。

如果 embedding 模型升级，旧向量和新向量可能不在同一个空间里，不能混用。索引需要重建或分版本维护。

第二，文本规范化会影响命中率。

同一段文本如果多了空格、换行、HTML 标签、大小写变化，hash 可能不同。需要设计 canonicalization，例如去除无意义空白、统一 Unicode、清理 HTML。

第三，权限隔离仍然重要。

企业 A 的私有文档 embedding 不能被企业 B 使用。即使向量看起来不可读，也可能通过相似检索泄露信息。

第四，缓存不等于索引。

Embedding Cache 解决“文本是否已经算过向量”；向量索引解决“给定 query 找哪些向量相似”。两者可以共用存储，也可以分开设计。

## 13.15 Retrieval Cache：缓存检索结果

RAG 系统里，除了 embedding，还可以缓存检索结果。

例如用户问：

```text
公司年假政策是什么？
```

系统可能需要：

1. 计算 query embedding。
2. 向量检索 top-k 文档。
3. BM25 检索候选。
4. reranker 重排。
5. 拼接上下文。

如果这个问题高频出现，而且知识库版本没有变化，就可以缓存最终 top-k 文档和 rerank 结果。

Retrieval Cache 的 key 通常包含：

```text
hash(tenant_id, permission_scope, query_normalized, embedding_model_version, index_version, reranker_version, top_k, filters)
```

知识库更新时，retrieval cache 要失效。否则系统可能引用过期政策。

常见失效策略包括：

1. TTL：超过时间自动过期。
2. 版本号：知识库版本变化后整体失效。
3. 分片失效：只让相关文档集合的缓存失效。
4. 事件驱动：文档更新后发布 invalidation event。
5. 懒失效：命中时检查版本，不一致再刷新。

如果面试题包含 RAG，讲清 retrieval cache 会很加分，因为很多候选人只讲 response cache。

## 13.16 会话缓存与多轮对话

多轮对话中，缓存设计更复杂。

一个用户连续提问时，系统可能复用：

1. 会话摘要。
2. 历史消息压缩结果。
3. 已检索文档。
4. 工具调用结果。
5. prompt 前缀 KV cache。
6. 用户偏好和上下文状态。

例如用户先问：

```text
帮我分析这份合同。
```

随后追问：

```text
第二条有什么风险？
```

第二个问题不能按独立 query 缓存，因为“第二条”依赖会话上下文。此时 cache key 必须包含会话状态版本，或者只缓存中间结果，例如合同解析结果和章节索引，而不是缓存最终回答。

会话缓存常见原则：

1. 缓存中间结构比缓存最终回答更安全。
2. 对依赖上下文的回答，必须绑定 session_id 和 state_version。
3. 会话结束或超过 TTL 后清理敏感缓存。
4. 用户删除对话时，相关缓存也要删除或不可再访问。
5. 多端同步时要避免旧状态覆盖新状态。

## 13.17 缓存一致性与失效

缓存系统最难的问题不是写入，而是失效。

在大模型系统中，以下变化都可能导致缓存失效：

1. 模型版本变化。
2. prompt 模板变化。
3. 系统提示词变化。
4. 安全策略变化。
5. 知识库文档变化。
6. 用户权限变化。
7. 业务规则变化。
8. 采样参数变化。
9. 输出格式要求变化。
10. 法规或合规要求变化。

因此缓存 key 要包含关键版本，缓存 value 也要带元数据。

一个缓存 value 可以包含：

| 字段 | 含义 |
| --- | --- |
| answer | 缓存回答 |
| model_id/version | 生成模型和版本 |
| prompt_template_version | 模板版本 |
| policy_version | 安全策略版本 |
| kb_version | 知识库版本 |
| created_at | 创建时间 |
| expire_at | 过期时间 |
| quality_score | 质量评分 |
| safety_score | 安全评分 |
| hit_count | 命中次数 |
| tenant_scope | 可访问范围 |

常见失效策略：

1. TTL 过期。
2. LRU/LFU 淘汰。
3. 版本号不一致自动失效。
4. 管理后台手动失效。
5. 风险事件触发批量失效。
6. 用户删除数据触发隐私删除。

这里要强调：大模型缓存不能只按技术缓存理解，还要按业务和合规缓存理解。

## 13.18 成本模型

模型路由的目标之一是成本优化，所以要能估算成本。

一次请求的成本通常包括：

1. 输入 token 成本。
2. 输出 token 成本。
3. embedding 成本。
4. 检索和 rerank 成本。
5. 工具调用成本。
6. 安全审核成本。
7. 缓存存储和网络成本。
8. 重试和失败成本。

可以用一个简化表达：

```text
request_cost = input_tokens * input_unit_price
             + output_tokens * output_unit_price
             + embedding_cost
             + retrieval_cost
             + safety_cost
             + retry_cost
             + cache_cost
```

路由系统可以在请求前估算成本，在请求后记录真实成本。

常见成本优化手段包括：

1. 简单任务走小模型。
2. 高频问题走 response cache。
3. 长系统提示词使用 prompt cache。
4. RAG 查询使用 embedding cache 和 retrieval cache。
5. 批量离线任务使用低峰资源。
6. 输出长度设置合理上限。
7. 对低价值请求限制最大上下文。
8. 对失败率高的路径减少无效重试。

但成本优化必须有质量约束。一个更合理的目标不是最低成本，而是：

```text
minimize cost subject to quality >= Q, latency <= L, safety_risk <= R
```

这句话在面试中很有用，因为它表达了系统设计的本质：不是单指标优化，而是约束优化。

## 13.19 延迟优化

缓存和路由也直接影响延迟。

一次大模型请求的延迟通常包括：

1. 网关和认证延迟。
2. 路由决策延迟。
3. 缓存查询延迟。
4. embedding 延迟。
5. 检索和 rerank 延迟。
6. 模型排队延迟。
7. 首 token 延迟。
8. 解码生成延迟。
9. 输出审核延迟。
10. 网络传输延迟。

优化方法包括：

1. 路由规则本地化，避免每次远程查复杂配置。
2. 缓存查询并行化，例如 response cache、embedding cache、权限检查部分并行。
3. 热门 prompt 预热。
4. 热点 FAQ 预生成答案。
5. 对长回答使用流式输出。
6. 对非关键安全检查异步补充，但关键风险仍需在线拦截。
7. 对超时模型设置快速 fallback。
8. 使用 batch 和 continuous batching 提升推理吞吐。

延迟优化要关注 p95、p99，而不仅是平均值。用户体感通常由尾延迟决定。

## 13.20 隐私与安全

缓存是降本工具，也是数据泄露风险点。

常见风险包括：

1. A 用户命中 B 用户的私有回答。
2. 企业租户之间缓存串用。
3. 用户删除数据后缓存仍保留。
4. 缓存 key 中明文包含敏感 prompt。
5. 日志记录了完整隐私内容。
6. 语义缓存误命中导致泄露业务信息。
7. 调试后台可直接查看敏感缓存。

安全设计原则：

1. cache key 使用 hash，不直接暴露原始 prompt。
2. 缓存 value 按租户、权限、数据域隔离。
3. 私有和敏感请求默认不进入全局缓存。
4. PII、高风险、企业机密场景设置更短 TTL 或禁用 response cache。
5. 管理后台访问缓存需要权限、审计和脱敏。
6. 用户删除数据时触发缓存删除或密钥销毁。
7. 对语义缓存增加意图和实体校验。

一个重要原则是：全局缓存只适合公共、稳定、低风险的信息；私有上下文只能在严格隔离范围内缓存。

## 13.21 可观测性

没有可观测性，路由系统会变成黑盒。

需要监控的指标包括：

| 类别 | 指标 |
| --- | --- |
| 流量 | QPS、请求类型分布、租户分布 |
| 路由 | 各模型流量占比、升级率、降级率、fallback 率 |
| 缓存 | 命中率、误命中率、过期率、淘汰率、写入率 |
| 成本 | 每请求成本、每用户成本、每业务成本、token 消耗 |
| 延迟 | p50/p95/p99、首 token 延迟、生成延迟 |
| 质量 | 用户点赞/点踩、人工评审分、任务成功率 |
| 安全 | 拦截率、误杀率、高风险输出率 |
| 稳定性 | 错误率、超时率、重试率、模型健康状态 |

除了指标，还需要 trace。一次请求应该能追踪：

1. 请求被识别为什么任务。
2. 查询了哪些缓存。
3. 是否命中缓存。
4. 为什么选择某个模型。
5. 模型调用耗时和 token 数。
6. 是否触发重试或降级。
7. 输出是否通过质量和安全检查。

这对排查线上问题非常关键。例如用户投诉“回答变差了”，你需要知道是模型版本变了、路由策略变了、缓存命中了旧答案，还是知识库版本没更新。

## 13.22 离线优化与反馈闭环

成熟系统不会长期依赖手写规则。

离线优化可以做几件事：

1. 分析哪些请求适合小模型。
2. 找出高成本低价值流量。
3. 挖掘高频 query，建立预缓存。
4. 评估语义缓存误命中。
5. 训练路由分类器或打分模型。
6. 分析不同模型在不同任务上的胜率。
7. 更新 prompt 模板和安全策略。
8. 发现应该进入专用模型训练的数据。

一个典型闭环是：

1. 在线系统记录请求特征、路由选择、模型输出、成本、延迟和反馈。
2. 离线评估系统对样本打质量分。
3. 分析不同策略的 cost-quality frontier。
4. 生成新的路由策略或模型打分表。
5. 灰度发布到线上。
6. 监控质量、成本和投诉变化。
7. 如果异常，快速回滚。

这就是为什么模型路由本质上不是简单中间件，而是一个持续优化系统。

## 13.23 灰度、A/B 与回滚

路由策略一旦改错，可能大面积影响用户。所以必须支持灰度。

常见发布方式：

1. 按租户灰度。
2. 按用户比例灰度。
3. 按请求类型灰度。
4. 按模型版本灰度。
5. 按地域或业务线灰度。

A/B 实验可以比较：

1. 新旧模型。
2. 新旧路由策略。
3. 是否启用语义缓存。
4. 不同缓存阈值。
5. 不同 fallback 策略。

需要关注的实验指标不只是成本下降，还包括：

1. 用户满意度。
2. 任务完成率。
3. 追问率。
4. 人工客服转接率。
5. 安全拦截和误杀。
6. 延迟分布。
7. 投诉和退款。

回滚机制也很重要。策略配置、模型目录、缓存开关、阈值都应该可快速回滚，不能每次靠重新发版。

## 13.24 容灾与降级

模型服务可能失败，缓存系统也可能失败。

常见故障包括：

1. 主模型超时。
2. GPU 集群负载过高。
3. 第三方模型 API 不可用。
4. Redis 或向量库故障。
5. 路由配置错误。
6. 某模型输出质量突然下降。
7. 成本预算触发限制。

降级策略包括：

1. 大模型降到中模型。
2. 实时回答改为异步任务。
3. 关闭多模型投票。
4. 暂停低价值或批处理请求。
5. 只返回检索结果和引用，不生成长答案。
6. 使用模板化 FAQ。
7. 对免费用户限流，保障高优先级业务。
8. 缓存不可用时直接走模型，但限制高成本路径。

设计降级时要注意：降级结果必须对用户诚实。如果系统无法完成复杂任务，可以说明暂时不可用或能力受限，而不是输出看似自信但低质量的答案。

## 13.27 面试答题结构

遇到这个题，可以按下面顺序回答。

第一步，澄清范围：

```text
是面向多模型在线服务，还是只做单模型缓存？是否包含 RAG？是否有多租户和企业数据隔离？目标是降成本、降延迟，还是提升可用性？
```

第二步，列需求：

1. 路由到不同模型。
2. 支持 response/prompt/embedding/retrieval cache。
3. 支持降级、灰度、监控、审计。
4. 保证多租户隔离和隐私。

第三步，画架构：

```text
Gateway -> Request Analyzer -> Cache Manager -> Router/Policy Engine -> Model Serving -> Quality Guard -> Telemetry -> Offline Optimizer
```

第四步，展开核心设计：

1. 模型目录。
2. 路由策略。
3. 缓存 key 和失效。
4. 成本和延迟优化。
5. 安全隔离。

第五步，讲扩展和权衡：

1. 规则路由 vs 学习型路由。
2. 精确缓存 vs 语义缓存。
3. 命中率 vs 误命中风险。
4. 成本下降 vs 质量下降。
5. 缓存复用 vs 隐私隔离。

第六步，讲监控和回滚。

这套结构比直接背架构图更稳，因为它能覆盖面试官追问。

## 13.28 高频追问

### 追问一：如何判断一个请求能不能走小模型？

可以从任务类型、上下文长度、风险等级、历史失败率、输出格式要求、用户等级和置信度判断。初期用规则，后期用离线评测数据训练难度分类器。关键是要有升级机制，小模型低置信或质量检查失败时升级到大模型。

### 追问二：语义缓存误命中怎么办？

提高相似度阈值，加入意图分类、实体槽位、权限、知识库版本和场景一致性检查。对高风险任务禁用语义缓存，或者只把语义命中作为候选，再用轻量模型验证是否等价。

### 追问三：模型升级后缓存怎么处理？

缓存 key 和 value 都带 model_version、prompt_template_version、policy_version。模型升级后可以让旧缓存自然过期，也可以分批失效。对于安全或事实敏感业务，应强制失效；对于低风险 FAQ，可以保留但标记版本。

### 追问四：如何避免跨租户数据泄露？

缓存 key 加 tenant_id 和 permission_scope，缓存 value 做租户隔离。私有数据不进入全局缓存。管理后台脱敏和审计。用户删除数据时清理相关缓存。语义缓存也必须按租户和权限范围检索。

### 追问五：如果缓存系统挂了怎么办？

缓存应该是优化层，不应该是强依赖。response cache 挂了可以直接走模型，但要限流防止成本暴涨；embedding cache 挂了可以重新计算 embedding；retrieval cache 挂了可以重新检索。若缓存故障导致延迟上升，可临时关闭语义缓存或降低复杂路径流量。

### 追问六：如何证明路由策略真的省钱而没有伤害质量？

通过 A/B 实验比较成本、延迟、用户满意度、任务完成率、追问率、人工转接率、安全指标和投诉率。不能只看 token 成本下降，因为小模型质量下降可能导致用户多轮追问，最终总成本更高。

### 追问七：prompt cache 和 response cache 有什么区别？

Response cache 缓存最终答案，命中后可以不调用模型。Prompt cache 缓存输入前缀的中间计算，仍然需要模型继续生成。Response cache 更省成本但适用范围窄，prompt cache 对长上下文、多轮会话和固定系统提示词更有用。

### 追问八：路由系统是否应该实时调用一个大模型来决定路由？

通常不建议每次都用大模型做路由，因为会增加成本和延迟。可以用规则、小分类器、轻量模型或离线训练的打分模型。只有在高价值、复杂、低频场景中，才考虑更重的路由判断。

## 13.29 常见错误回答

错误一：只说“用 Redis 缓存”。

问题是没有说明缓存什么、key 怎么设计、如何失效、如何隔离、如何处理语义相似和版本变化。

错误二：只按价格选模型。

问题是忽略质量、安全、延迟和用户价值。最便宜模型不一定让总成本最低。

错误三：所有请求都用语义缓存。

问题是误命中可能造成严重后果，特别是权限、金融、医疗、法律和企业私有知识场景。

错误四：忽略模型版本。

模型、tokenizer、prompt 模板、知识库和安全策略变化都会影响缓存有效性。

错误五：没有观测和回滚。

路由策略一旦上线，必须能知道它影响了哪些请求、节省了多少成本、质量是否下降，并且能快速回滚。

## 13.30 小结

模型路由与缓存系统的本质，是在大模型服务前面建立一个智能控制层。

它不是简单的代理，也不是简单的缓存，而是把模型能力、请求难度、业务价值、实时负载、缓存复用、成本预算、安全策略和质量反馈组织到同一套决策系统中。

面试中要抓住五个核心点：

1. 路由要基于任务难度、模型能力、SLA、成本和风险，而不是固定模型。
2. 缓存分为 response cache、prompt cache、embedding cache、retrieval cache 和会话缓存。
3. 缓存 key 必须包含租户、权限、模型版本、prompt 版本、知识库版本和策略版本。
4. 成本优化必须受质量、延迟和安全约束。
5. 系统必须有监控、灰度、回滚和离线反馈闭环。

如果能把这五点讲清楚，这道题基本就能从“会用缓存”上升到“会设计大模型平台控制面”。
