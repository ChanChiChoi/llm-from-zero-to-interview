# 第六章：设计 Agent 平台

这一章讨论大模型系统设计中非常热门、也非常容易被问到风险边界的一类题：如何设计 Agent 平台。

Agent 不是“让模型自己想办法”。生产级 Agent 平台的核心，不是炫酷 demo，而是让模型在受控环境中规划、调用工具、观察结果、更新状态、处理错误，并且所有高风险动作都能被权限、审计和人工确认约束。

如果 ChatGPT 服务主要是“模型回答问题”，RAG 系统主要是“模型基于证据回答问题”，Agent 平台则进一步变成“模型可以使用工具完成任务”。一旦模型能调用工具，它就可能读写数据库、发送邮件、改代码、创建工单、下单、操作云资源。因此 Agent 平台的系统设计重点是可控性、权限、安全、状态管理和可观测性。

本章重点：工具注册、权限、状态管理、计划执行、观察反馈、审计和安全。

## 6.1 先明确题目边界

面试官问“设计 Agent 平台”，可能有几种范围。

第一，设计个人助手 Agent。

它能查日历、写邮件、总结文档、安排会议。重点是用户授权、隐私、确认机制和多应用集成。

第二，设计企业办公 Agent。

它能查知识库、建工单、更新 CRM、生成报表、调用内部系统。重点是企业权限、审计、工作流和数据隔离。

第三，设计开发者 Agent。

它能读代码、改代码、运行测试、提交 PR。重点是沙箱、代码权限、测试、回滚和人工 review。

第四，设计通用 Agent 平台。

它为多个业务提供工具注册、任务编排、状态管理、权限控制、执行环境、监控和安全策略。

面试开头可以这样澄清：

我先按企业内部通用 Agent 平台设计，支持多个业务注册工具，用户提交自然语言任务，Agent 在权限控制下规划步骤、调用工具、观察结果、更新状态，并在高风险动作前请求人工确认。重点展开工具注册、权限、状态管理、执行循环、审计、安全和评估。

## 6.2 Agent 平台解决什么问题

传统聊天机器人只能回答。Agent 平台希望让模型完成任务。

例如：

1. “帮我查上季度华东区销售下降的原因，并生成分析报告。”
2. “帮我找出这个服务最近报错的根因，并创建修复工单。”
3. “帮我阅读这个仓库，修复一个单测失败。”
4. “帮我根据会议纪要创建项目计划和待办。”

这些任务不能只靠一次模型生成。它们需要读取数据、检索文档、调用 API、执行代码、写入系统、处理中间失败。

Agent 平台的价值是把这些能力从一次性 prompt trick 变成可复用、可控、可审计的系统能力。

但 Agent 也带来新的风险：错误工具调用、越权访问、成本失控、循环执行、状态污染、错误写入、prompt injection、敏感数据泄漏。

因此生产级 Agent 平台的设计目标不是“尽量自主”，而是“在明确边界内自主”。

## 6.3 功能需求

Agent 平台通常需要以下功能。

| 模块 | 功能 |
| --- | --- |
| 用户入口 | 自然语言任务、模板任务、API 调用 |
| Agent Runtime | 规划、执行、观察、反思、终止 |
| Tool Registry | 工具注册、schema、权限、版本、描述 |
| Permission Service | 用户授权、工具权限、数据权限、动作权限 |
| State Store | 任务状态、步骤记录、上下文、变量、结果 |
| Execution Sandbox | 代码执行、文件操作、网络访问、资源限制 |
| Human Approval | 高风险动作确认、人工接管、审批流 |
| Audit Log | 工具调用、输入输出、权限决策、修改记录 |
| Monitoring | 成功率、失败率、步数、成本、延迟、安全事件 |
| Evaluation | 任务成功率、工具准确率、错误类型、回归测试 |

第一版可以只支持有限工具集和固定任务类型，不要一开始就做完全开放通用 Agent。

## 6.4 非功能需求

Agent 平台的非功能需求和普通聊天系统不同。

### 6.4.1 可控性

Agent 可以行动，所以必须可控。平台要限制工具范围、参数、调用次数、执行时间、资源消耗和可写操作。

### 6.4.2 可审计性

每一步为什么调用某个工具、传了什么参数、拿到什么结果、做了什么决策，都要记录。出了事故后必须能还原。

### 6.4.3 安全性

安全包括权限安全、数据安全、工具安全、执行环境安全和输出安全。

### 6.4.4 鲁棒性

工具可能失败，外部 API 可能超时，模型可能选择错误步骤，任务可能进入循环。平台要支持重试、降级、人工接管和终止。

### 6.4.5 成本可控

Agent 可能一个任务调用几十次模型和工具。平台要设置最大步数、token budget、工具预算、超时和异常成本告警。

## 6.5 容量估算

Agent 平台估算不能只看外部任务 QPS，因为一个任务会放大为多次模型调用和工具调用。

假设企业 Agent 平台：

1. 峰值任务请求 100 QPS。
2. 每个任务平均 6 个步骤。
3. 每个步骤平均 1 次模型调用。
4. 每个任务平均 4 次工具调用。
5. 平均每次模型调用输入 2000 token，输出 500 token。

那么内部模型调用峰值约 600 QPS，工具调用峰值约 400 QPS，token 吞吐约 150 万 token/s。

还要估算状态和日志：

1. 每个任务记录 plan、step、tool input、tool output、observation、决策和最终答案。
2. 如果平均每个任务日志 50KB，100 QPS 一天就是数百 GB 级别。
3. 高风险工具还要保存审计和审批记录。

这个估算说明：Agent 平台的瓶颈通常在任务放大、工具稳定性、状态存储、审计日志和成本控制。

## 6.6 高层架构

Agent 平台可以分成以下组件。

| 组件 | 职责 |
| --- | --- |
| Client / API | 用户提交任务、查看进度、确认动作 |
| API Gateway | 鉴权、限流、租户隔离 |
| Task Service | 创建任务、状态机、任务生命周期 |
| Agent Orchestrator | 规划、步骤调度、终止判断 |
| LLM Service | 规划模型、执行模型、总结模型 |
| Tool Registry | 工具定义、schema、版本、风险等级 |
| Tool Executor | 调用工具、参数校验、超时、重试 |
| Permission Service | 用户、工具、数据、动作权限决策 |
| State Store | 任务状态、上下文、变量、步骤结果 |
| Sandbox | 代码执行、文件系统、网络隔离 |
| Approval Service | 高风险动作审批、人工确认 |
| Audit Service | 工具调用、权限、写操作、审批记录 |
| Monitoring | 成功率、延迟、成本、安全事件 |
| Evaluation Service | 任务评估、回归测试、失败归因 |

核心在线链路是：用户提交任务，Task Service 创建任务；Agent Orchestrator 生成计划；每一步由模型决定工具调用或下一步动作；Permission Service 检查权限；Tool Executor 执行工具；结果写入 State Store；Orchestrator 基于观察结果决定继续、结束、重试或请求人工确认；最终生成答案并记录审计。

## 6.7 工具注册

Tool Registry 是 Agent 平台的基础。

### 6.7.1 工具描述

每个工具要有清晰 schema，包括：

1. tool_id。
2. 名称和自然语言描述。
3. 输入参数 schema。
4. 输出 schema。
5. 是否只读。
6. 风险等级。
7. 权限要求。
8. 超时和重试策略。
9. 版本。
10. owner 和审计要求。

工具描述要准确。描述含糊会让模型误用工具。

### 6.7.2 只读工具和写工具

工具要区分只读和写操作。

只读工具如查询文档、搜索数据库、读取工单。写工具如发送邮件、修改数据库、创建工单、提交代码、重启服务。

写工具风险更高，需要更严格的权限、参数校验、审批和回滚方案。

### 6.7.3 工具版本

工具 API 会变化。平台要记录工具版本，并让任务执行时绑定版本，避免执行中工具 schema 改变导致任务失败。

### 6.7.4 工具测试

工具上线前要做测试：schema 是否正确、参数校验是否生效、错误码是否清晰、超时是否可控、权限是否正确。

## 6.8 权限设计

Agent 平台的权限不能交给模型判断。

### 6.8.1 权限层级

权限包括：

1. 用户是否能使用某个 Agent。
2. 用户是否能调用某个工具。
3. 用户是否能访问某类数据。
4. 用户是否能执行写操作。
5. 某个动作是否需要二次确认。
6. 某个租户是否允许数据出域。

### 6.8.2 权限决策点

权限检查要发生在：

1. 任务创建时。
2. 工具候选集生成前。
3. 工具调用前。
4. 工具返回结果写入状态前。
5. 最终答案返回前。
6. 引用或文件下载时。

### 6.8.3 最小权限

Agent 只能看到完成当前任务需要的工具和数据，不应该默认拿到所有工具。

例如一个会议助手不应该有重启生产服务的工具；一个代码修复 Agent 不应该能访问 HR 系统。

### 6.8.4 授权和审批

高风险动作要人工确认。例如发送外部邮件、删除文件、修改生产配置、执行支付、提交 PR、重启服务。

审批界面要展示 Agent 准备做什么、为什么做、会影响什么、是否可回滚。

## 6.9 状态管理

Agent 任务通常不是一次模型调用，而是多步状态机。

### 6.9.1 状态内容

State Store 要记录：

1. task_id。
2. 用户目标。
3. 当前状态。
4. 计划。
5. 已执行步骤。
6. 每步工具输入输出。
7. 中间变量。
8. 错误和重试。
9. 审批状态。
10. 最终结果。

### 6.9.2 状态机

一个简单任务状态机可以是：

1. created。
2. planning。
3. running。
4. waiting_approval。
5. paused。
6. succeeded。
7. failed。
8. cancelled。

状态机比“while 循环调用模型”更可靠，因为它能暂停、恢复、审计和失败处理。

### 6.9.3 幂等性

工具调用尤其是写操作要考虑幂等性。Agent 重试时不能重复发送邮件、重复扣款、重复创建工单。

可以使用 idempotency key，或者在写操作前检查已有结果。

### 6.9.4 长任务恢复

长任务可能跨分钟、小时甚至天。平台要能从状态中恢复，而不是依赖内存中的上下文。

## 6.10 计划执行循环

Agent 常见执行循环是 plan、act、observe、update。

### 6.10.1 Plan

模型根据用户目标生成计划。计划可以是一次性完整计划，也可以逐步规划。

生产系统中，计划不应完全自由。可以给模型提供任务模板、工具约束和输出 schema。

### 6.10.2 Act

模型选择工具和参数。平台必须做参数校验、权限检查和风险判断。

模型输出的工具调用只是候选动作，不是最终动作。最终是否执行由系统决定。

### 6.10.3 Observe

工具返回结果后，平台把 observation 写入状态，并摘要给模型。工具输出可能很长，要做截断、摘要或结构化处理。

### 6.10.4 Update

模型根据 observation 决定下一步：继续调用工具、修改计划、请求人工确认、结束任务或失败退出。

### 6.10.5 Termination

必须有终止条件，包括最大步数、最大时间、最大 token、最大工具调用次数、连续失败次数和目标完成判断。

没有终止条件的 Agent 很容易循环和烧钱。

## 6.11 工具执行

Tool Executor 是模型和外部世界之间的安全边界。

### 6.11.1 参数校验

所有工具参数都要按 schema 校验。不能因为模型生成了一个字段，就直接传给下游系统。

### 6.11.2 超时和重试

每个工具要有超时、重试和熔断策略。

只读查询可以重试，写操作重试要谨慎，必须保证幂等。

### 6.11.3 结果标准化

不同工具返回格式不同。平台可以把结果标准化为状态码、结构化数据、错误信息和可展示摘要。

### 6.11.4 错误处理

工具失败时，Agent 可以换工具、修改参数、请求用户补充信息或结束任务。不能无限重试。

## 6.12 执行沙箱

开发者 Agent 或数据分析 Agent 经常需要执行代码。

### 6.12.1 为什么需要沙箱

模型生成的代码可能有 bug、删除文件、访问网络、消耗大量资源或泄露数据。必须在隔离环境执行。

### 6.12.2 沙箱限制

沙箱可以限制：

1. CPU、内存、磁盘。
2. 执行时间。
3. 网络访问。
4. 文件系统目录。
5. 环境变量和 secrets。
6. 系统调用。

### 6.12.3 文件和数据权限

Agent 只能访问任务授权范围内的文件。临时文件要隔离，任务结束后按策略清理。

### 6.12.4 代码变更回滚

如果 Agent 修改代码或配置，要能生成 diff、运行测试、提交给人工 review，而不是直接改生产。

## 6.13 人工确认与接管

生产级 Agent 不能所有动作都自动执行。

### 6.13.1 需要确认的动作

常见需要人工确认的动作包括：

1. 发送外部消息。
2. 修改或删除数据。
3. 触发支付或订单。
4. 修改生产配置。
5. 提交代码或发布服务。
6. 访问高敏感数据。

### 6.13.2 确认界面

确认界面要展示：

1. Agent 准备执行什么。
2. 参数是什么。
3. 为什么要执行。
4. 影响范围。
5. 风险等级。
6. 是否可回滚。

### 6.13.3 人工接管

当 Agent 不确定、连续失败或风险升高时，应能转人工。人工接管后，Agent 可以提供上下文摘要和已执行步骤。

## 6.14 审计与可观测性

Agent 平台必须强审计。

### 6.14.1 审计日志

审计日志包括：

1. 谁发起任务。
2. 任务目标是什么。
3. Agent 看到哪些工具。
4. 每次工具调用参数。
5. 权限决策结果。
6. 工具返回摘要。
7. 人工审批记录。
8. 最终输出。
9. 写操作影响对象。

### 6.14.2 Trace

Trace 用来调试一次任务。它应该串起模型调用、工具调用、状态变更、审批和错误。

### 6.14.3 指标

Agent 指标包括：

1. 任务成功率。
2. 平均步数。
3. 平均模型调用次数。
4. 平均工具调用次数。
5. 平均任务成本。
6. 人工确认率。
7. 工具失败率。
8. 循环终止率。
9. 安全拦截率。

## 6.15 安全设计

Agent 安全比普通聊天和 RAG 更复杂，因为 Agent 能行动。

### 6.15.1 Prompt injection

如果 Agent 会读取网页、邮件、文档或工单，这些外部内容可能包含恶意指令。

防护原则是：外部内容是数据，不是指令。系统指令、工具权限和策略由平台控制，不能被文档覆盖。

### 6.15.2 Tool injection

工具返回结果也可能诱导模型调用其他工具或泄露信息。平台要把工具输出作为 observation，而不是更高优先级指令。

### 6.15.3 越权访问

权限必须由 Permission Service 决定。模型不能自己决定“用户应该有权限”。

### 6.15.4 数据泄漏

Agent 可能把内部数据发到外部工具、邮件或第三方 API。要做数据分类、外发检测、DLP、审批和审计。

### 6.15.5 高风险动作控制

高风险动作要二次确认、最小权限、可回滚、强审计。对于不可逆动作，默认不自动执行。

### 6.15.6 合规边界

本章讨论的是合法授权范围内的 Agent 平台设计，不提供绕过权限、规避审计、未授权访问系统、隐蔽执行操作或泄露敏感数据的方法。

## 6.16 评估体系

Agent 评估不能只看最终回答好不好。

### 6.16.1 任务成功率

看 Agent 是否完成用户目标。例如是否成功创建工单、修复测试、生成正确报表。

### 6.16.2 工具调用准确率

看 Agent 是否选对工具、参数是否正确、调用顺序是否合理。

### 6.16.3 安全指标

看越权调用率、高风险动作未确认率、敏感数据外泄率、prompt injection 成功率。

### 6.16.4 成本指标

看平均步数、平均 token、平均工具调用次数、超时率和失败任务成本。

### 6.16.5 稳定性指标

看多次运行同一任务结果是否一致，是否容易陷入循环，是否对工具错误鲁棒。

## 6.17 成本控制

Agent 成本容易失控。

### 6.17.1 Budget

每个任务可以设置：

1. 最大步骤数。
2. 最大模型调用次数。
3. 最大工具调用次数。
4. 最大 token。
5. 最大执行时间。
6. 最大外部 API 成本。

### 6.17.2 分层模型

简单步骤用小模型，复杂规划或最终总结用大模型。工具参数提取可以用小模型或规则。

### 6.17.3 缓存

可缓存工具结果、检索结果、计划模板和中间摘要。但缓存要考虑权限和时效性。

### 6.17.4 早停

如果连续失败、置信度低或预算快耗尽，Agent 应停止并请求用户或人工帮助，而不是继续消耗资源。

## 6.18 高可用与故障恢复

Agent 平台常见故障包括：

1. 模型服务超时。
2. 工具 API 失败。
3. 权限服务不可用。
4. 状态存储写入失败。
5. 任务执行进入循环。
6. 审批流程卡住。
7. 外部系统返回异常数据。

恢复策略包括：

1. 状态持久化，支持任务恢复。
2. 工具调用超时和熔断。
3. 只读工具可重试，写工具保证幂等。
4. 权限服务失败时默认拒绝高风险动作。
5. 循环检测和强制终止。
6. 人工接管。
7. 任务失败后生成可读错误报告。

## 6.19 面试答题模板

如果面试官问“设计 Agent 平台”，可以这样回答。

第一，澄清范围。

我先按企业内部通用 Agent 平台设计，支持用户提交任务，Agent 在受控工具集内规划、调用工具、观察结果、更新状态，并对高风险动作做人工确认和审计。

第二，定义指标。

核心指标包括任务成功率、平均步数、工具调用准确率、越权拦截率、安全事件率、人工确认率、P95 延迟和平均任务成本。

第三，做容量估算。

根据外部任务 QPS、平均步骤数、每步模型调用次数、工具调用次数和日志大小，估算内部模型 QPS、工具 QPS、token 吞吐和状态存储规模。

第四，画架构。

系统包括 Client/API、API Gateway、Task Service、Agent Orchestrator、LLM Service、Tool Registry、Tool Executor、Permission Service、State Store、Sandbox、Approval Service、Audit Service、Monitoring 和 Evaluation Service。

第五，讲核心链路。

用户提交任务后，Task Service 创建任务，Orchestrator 生成计划，模型提出工具调用，Permission Service 校验权限，Tool Executor 执行工具，结果写入 State Store，Agent 根据观察继续、停止或请求审批，最终输出结果并记录审计。

第六，展开关键难点。

重点讲工具注册、权限、状态机、幂等、人工确认、沙箱、审计、prompt injection 和成本控制。

第七，讲安全边界。

模型只能提出动作，系统决定是否执行。权限、审批、审计和高风险动作控制必须在平台层实现，不能依赖模型自觉。

## 6.20 常见面试追问

### 6.20.1 Agent 和普通 ChatGPT 服务有什么区别？

普通 ChatGPT 服务主要生成文本回答；Agent 平台可以调用工具、读写外部系统并完成多步任务。因此 Agent 更需要权限、状态管理、审计、人工确认和失败恢复。

### 6.20.2 为什么不能让模型直接调用所有工具？

因为模型可能误判、被 prompt injection 影响或生成错误参数。工具权限和执行必须由平台控制，Agent 只能看到当前任务授权范围内的工具。

### 6.20.3 如何防止 Agent 无限循环？

设置最大步数、最大时间、最大 token、最大连续失败次数；做状态相似性检测；模型多次重复同一动作时终止；必要时请求人工接管。

### 6.20.4 写操作如何保证安全？

写操作要权限检查、参数校验、幂等 key、高风险审批、影响范围展示、审计日志和回滚方案。不可逆高风险动作默认不自动执行。

### 6.20.5 Agent 状态为什么要持久化？

因为任务可能多步、长时间运行、等待审批或遇到故障。状态持久化后可以暂停、恢复、审计和调试。

### 6.20.6 如何评估 Agent？

看任务成功率、工具调用准确率、参数正确率、平均步数、成本、安全事件率、人工确认率、循环率和失败类型。不能只看最终回答是否流畅。

### 6.20.7 如何处理 prompt injection？

把外部文档、网页、邮件和工具输出都视为不可信数据，不允许它们覆盖系统指令和权限策略；工具执行由平台权限控制；高风险动作需要确认和审计。

### 6.20.8 Agent 什么时候应该转人工？

当风险高、权限不足、证据冲突、连续失败、预算快耗尽、用户意图不清或动作不可逆时，应转人工或请求用户确认。

## 6.21 本章小结

Agent 平台的核心价值，是把模型从“会说”扩展为“能在受控工具环境中做事”。但越能做事，越需要系统边界。

本章要记住几句话：

1. 生产级 Agent 不是 prompt demo，而是工具、权限、状态、执行、审计和安全的完整平台。
2. 模型只能提出动作，系统决定是否执行。
3. 工具必须注册 schema、权限、风险等级、版本和审计要求。
4. 权限控制不能靠模型，必须由平台层 Permission Service 实现。
5. Agent 任务要用状态机管理，支持暂停、恢复、失败和审批。
6. 写操作要幂等、可审计、可确认、可回滚。
7. Prompt injection 和 tool injection 是 Agent 的核心风险。
8. Agent 评估要看任务成功率、工具调用、成本、安全和稳定性，而不只是回答是否好看。

如果面试中被问到 Agent 平台，最好的回答是强调“受控自主”：Agent 可以规划和调用工具，但所有权限、安全、审计、预算和高风险动作都必须由系统平台控制。
