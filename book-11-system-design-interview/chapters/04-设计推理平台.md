# 第四章：设计推理平台

这一章讨论大模型系统设计中的核心基础设施：推理平台。

如果说训练平台解决“模型如何被训练出来”，推理平台解决的就是“模型如何稳定、低延迟、低成本地服务线上请求”。ChatGPT 服务、RAG 系统、Agent 平台、多模态助手、语音助手和安全审核系统，底层都依赖推理平台。

推理平台不是简单把模型加载起来开一个 HTTP 接口。它要管理模型注册、版本发布、推理引擎、请求调度、batching、KV cache、量化、多 GPU、自动扩缩容、SLA、监控、计费、安全和灰度回滚。

本章重点：模型注册、推理引擎、batching、KV Cache、量化、多 GPU、自动扩缩容。

## 4.1 先明确题目边界

面试官问“设计推理平台”，可能有三种范围。

第一，设计一个单模型在线服务。

例如把某个 7B 或 70B 模型部署成 API，重点关注延迟、吞吐、流式输出和 GPU 利用率。

第二，设计多模型推理平台。

平台支持多个模型、多个版本、多个租户、自动扩缩容、灰度发布和计费，类似内部模型服务平台。

第三，设计大规模商业 API 推理平台。

除了多模型，还要支持全球流量、SLA、开发者 API、限流、配额、账单、安全审核和滥用防护。

面试开头可以这样澄清：

我先按企业内部或云服务的大模型推理平台设计，支持多个模型和版本，为上层 Chat、RAG、Agent 等应用提供统一推理 API。重点关注模型注册、部署、请求路由、推理引擎、batching、KV cache、多 GPU、自动扩缩容、监控和成本。如果需要，我再展开商业 API 的计费和多租户。

## 4.2 功能需求

推理平台的功能可以分成九类。

| 模块 | 功能 |
| --- | --- |
| 模型管理 | 模型注册、版本、元数据、权重存储、审批 |
| 部署管理 | 创建服务、选择实例规格、灰度、回滚 |
| 请求入口 | API、鉴权、限流、配额、租户隔离 |
| 路由调度 | 模型路由、实例选择、队列、优先级 |
| 推理执行 | prefill、decode、batching、streaming |
| 资源管理 | GPU、显存、KV cache、节点拓扑、多 GPU |
| 优化能力 | 量化、speculative decoding、prefix cache |
| 观测计费 | 延迟、吞吐、GPU 利用率、token 计费 |
| 发布治理 | A/B 测试、自动回滚、安全策略、审计 |

第一版平台可以先支持文本生成模型和 embedding 模型，后续再支持 reranker、多模态模型、语音模型和安全审核模型。

## 4.3 非功能需求

推理平台的非功能需求比普通后端服务更复杂。

### 4.3.1 低延迟

在线推理要关注首 token 延迟、tokens/s、端到端延迟和排队时间。

用户请求可能很短，也可能带几万 token 上下文。平台必须避免长请求拖垮短请求。

### 4.3.2 高吞吐

GPU 很贵，平台要尽量提高 GPU 利用率和 token 吞吐。吞吐低意味着成本高。

### 4.3.3 高可用

推理服务通常直接面向线上业务。模型实例故障、GPU OOM、节点重启、流量突增、某个模型版本异常，都要能快速降级或回滚。

### 4.3.4 多租户隔离

不同业务共享推理平台时，要隔离配额、权限、日志、模型访问和成本归属。

### 4.3.5 成本可控

平台要支持按 token、按模型、按租户统计成本，并通过模型路由、batching、量化、缓存和自动扩缩容降低成本。

## 4.4 容量估算

推理平台容量估算要看请求、token、模型和 GPU。

假设平台服务三个上层业务：Chat、RAG 和 Agent。

1. Chat 峰值 1000 QPS，平均输入 1000 token，平均输出 500 token。
2. RAG 峰值 300 QPS，平均输入 4000 token，平均输出 800 token。
3. Agent 峰值 100 QPS，每个任务平均调用模型 5 次，每次输入 2000 token，输出 300 token。

估算 token 吞吐：

1. Chat 峰值约 150 万 token/s。
2. RAG 峰值约 144 万 token/s。
3. Agent 内部放大后约 115 万 token/s。

合计峰值约 409 万 token/s。

这个数字说明两个问题。

第一，平台不能只按外部 QPS 估算，Agent 和 RAG 会放大内部模型调用和输入 token。

第二，平均输入输出长度差异会导致 prefill 和 decode 压力不同。RAG 输入长，prefill 压力大；Chat 输出长，decode 压力大；Agent 多次调用，调度和成本压力大。

## 4.5 高层架构

推理平台可以分成以下组件。

| 组件 | 职责 |
| --- | --- |
| API Gateway | 鉴权、限流、协议、流式连接 |
| Tenant Service | 租户、配额、权限、计费 |
| Model Registry | 模型版本、元数据、权重、审批状态 |
| Deployment Controller | 创建部署、扩缩容、灰度、回滚 |
| Router | 选择模型、版本、区域和实例 |
| Queue Manager | 排队、优先级、长短请求隔离 |
| Inference Engine | 执行 prefill、decode、batching、streaming |
| Cache Manager | KV cache、prefix cache、response cache |
| GPU Scheduler | GPU 分配、拓扑、实例放置 |
| Monitoring | 延迟、吞吐、GPU、错误、成本 |
| Billing | token 统计、租户账单、成本归因 |
| Admin Console | 发布、回滚、配置、审计 |

在线请求链路是：请求进入 API Gateway，完成鉴权、限流和参数校验；Router 根据模型、租户、SLA、负载和灰度策略选择部署；Queue Manager 进入合适队列；Inference Engine 合批执行 prefill 和 decode；token 流式返回；Monitoring 和 Billing 异步记录指标和成本。

## 4.6 模型注册与版本管理

推理平台不应该直接加载某个随意路径下的模型权重。模型必须先进入 Model Registry。

### 4.6.1 Model Registry 记录什么

Model Registry 需要记录：

1. model_id。
2. model_version。
3. 模型类型，例如 causal LM、embedding、reranker、多模态。
4. 参数规模和架构。
5. tokenizer 和 chat template。
6. 权重位置和校验和。
7. 支持的上下文长度。
8. 推荐推理配置。
9. 评估报告和安全报告。
10. 来源训练任务。
11. 审批状态。

Tokenizer 和 chat template 非常重要。模型权重一样，但模板不同，输出质量可能明显变化。

### 4.6.2 版本发布

模型版本发布流程可以是：

1. 训练平台产出模型。
2. 自动评估和安全评估通过。
3. 注册到 Model Registry。
4. 转换为推理格式。
5. 在推理平台创建 canary 部署。
6. 小流量灰度。
7. 指标稳定后逐步扩大流量。
8. 异常时回滚旧版本。

推理平台要保留旧版本一段时间，方便回滚和复现线上问题。

### 4.6.3 模型格式转换

训练 checkpoint 往往包含 optimizer state 和分布式切分信息，不能直接用于高效推理。

推理平台或发布流程需要完成：

1. 合并或转换权重分片。
2. 去掉 optimizer state。
3. 转换 dtype，例如 FP16、BF16、INT8、INT4。
4. 生成推理引擎需要的格式。
5. 校验 tokenizer、config 和权重一致性。

## 4.7 请求路由

请求路由决定一个请求由哪个模型、哪个版本、哪个区域、哪个实例处理。

### 4.7.1 路由依据

路由可以考虑：

1. 用户指定模型。
2. 租户权限和套餐。
3. 请求类型。
4. 上下文长度。
5. SLA 要求。
6. 当前负载。
7. 模型健康状态。
8. 灰度实验配置。
9. 成本预算。

例如简单分类或摘要请求可以走小模型；复杂推理走大模型；超长上下文走支持长上下文的部署；高优先级租户走低延迟队列。

### 4.7.2 长短请求隔离

长请求会占用更多 prefill 时间和 KV cache。如果所有请求混在一个队列里，短请求可能被长请求拖慢。

常见策略包括：

1. 按输入长度分队列。
2. 按输出长度预估分队列。
3. 按租户优先级分队列。
4. 为短请求保留专用实例。
5. 对超长请求限流或转异步。

### 4.7.3 负载均衡

普通负载均衡按请求数分配不够，因为一个 100 token 请求和一个 10000 token 请求成本差异巨大。

更合理的是按估计 token 工作量、当前队列长度、KV cache 占用和实例健康状态来路由。

## 4.8 推理引擎

推理引擎是平台的核心执行层。

### 4.8.1 Prefill 和 decode

大模型生成分为两个阶段。

Prefill 阶段处理输入 prompt，计算初始 KV cache。输入越长，prefill 越重。

Decode 阶段逐 token 生成输出，每生成一个 token 都要读取 KV cache。输出越长，decode 越重。

区分这两个阶段很重要，因为优化方法不同。长上下文 RAG 主要卡 prefill，长回答聊天主要卡 decode，高并发短请求主要卡 batching 和调度。

### 4.8.2 Continuous batching

传统 batching 要等一批请求一起开始，适合固定长度任务。大模型生成长度不一，如果等整批都结束，会浪费 GPU。

Continuous batching 允许新请求动态加入，完成的请求动态退出，提高 GPU 利用率。

面试中可以这样解释：大模型 decode 是逐步生成，不同请求完成时间不同。Continuous batching 像流水线调度，每一步都尽量把还在生成的请求组成 batch，让 GPU 不空转。

### 4.8.3 Streaming

推理引擎要把生成 token 增量返回给上层服务。

Streaming 需要处理：

1. token buffer。
2. flush 频率。
3. 客户端取消。
4. finish reason。
5. 连接断开。
6. 输出审核。

对高风险场景，可能需要先生成后审核，而不是直接流式展示。

### 4.8.4 多推理引擎支持

平台可以支持 vLLM、TensorRT-LLM、TGI、SGLang 或内部引擎。

不同引擎适合不同场景。平台最好把上层 API 与底层引擎解耦，通过统一 serving interface 管理。

## 4.9 KV Cache 管理

KV cache 是推理平台最重要的资源之一。

### 4.9.1 KV cache 为什么重要

生成时，每个 token 都会产生 key 和 value 缓存。上下文越长、batch 越大、层数越多、head 越多，KV cache 占用越大。

如果 KV cache 管不好，就会出现显存碎片、OOM、吞吐下降和请求失败。

### 4.9.2 Paged attention

Paged attention 的思想是把 KV cache 像分页内存一样管理，避免为每个请求预留连续大块显存，从而减少碎片并支持更灵活的 batching。

面试中不需要实现细节，但要说明它解决的是 KV cache 动态分配和显存碎片问题。

### 4.9.3 Prefix cache

很多请求共享相同前缀，例如相同 system prompt、相同工具说明、相同企业知识库提示。Prefix cache 可以复用这些前缀的 KV，减少 prefill 成本。

但 prefix cache 要注意：

1. 前缀必须完全一致。
2. 不同租户不能错误共享敏感上下文。
3. cache invalidation 要正确。
4. 要统计命中率和收益。

### 4.9.4 KV cache 驱逐

当显存紧张时，需要驱逐低优先级或已完成请求的 cache。对于长时间空闲的会话，不能无限保留 KV cache。

## 4.10 Batching 与队列设计

Batching 是提高吞吐的关键，但会影响延迟。

### 4.10.1 Batching 的 trade-off

batch 越大，GPU 利用率通常越高，但排队等待时间也可能变长。低延迟业务不能为了吞吐无限等待合批。

常见策略是设置最大等待时间和最大 batch token 数，而不是只设置最大请求数。

### 4.10.2 Token-based batching

大模型请求长度差异巨大，所以按请求数 batching 不合理。更合理的是按 token 数控制 batch，例如 max prefill tokens、max decode tokens、max sequences。

### 4.10.3 优先级队列

可以按租户 SLA、请求类型、长度和是否交互式设置优先级。

例如在线聊天优先级高于离线批处理，付费租户高于免费租户，短请求可走低延迟队列。

## 4.11 多 GPU 与并行

大模型单张 GPU 放不下或吞吐不够时，需要多 GPU。

### 4.11.1 Tensor parallel

Tensor parallel 把单层矩阵计算切到多张 GPU 上，适合单模型太大或需要提高吞吐的场景。缺点是通信开销高，对网络拓扑敏感。

### 4.11.2 Pipeline parallel

Pipeline parallel 把不同层放到不同 GPU 上。它能降低单卡显存压力，但可能有 pipeline bubble，尤其在小 batch 或低并发下效率不高。

### 4.11.3 Expert parallel

MoE 模型可能需要 expert parallel，把不同 expert 放到不同 GPU 或节点上。调度要考虑 expert 负载不均和跨节点通信。

### 4.11.4 放置策略

多 GPU 部署要尽量把通信密集的 GPU 放在同节点或高速互联拓扑内。跨节点推理可能导致延迟和稳定性问题。

## 4.12 量化与模型压缩

量化是降低推理成本的重要手段。

### 4.12.1 为什么量化

量化可以减少显存占用、提高吞吐、降低成本，使更大模型能部署在更少 GPU 上。

常见格式包括 FP16、BF16、INT8、INT4、FP8 等。

### 4.12.2 量化风险

量化可能导致质量下降，尤其在数学、代码、长上下文、多语言和安全拒答上。

所以量化版本必须经过评估，不能只看困惑度或少量样例。

### 4.12.3 量化发布策略

可以先对低风险业务或简单请求使用量化模型，对高价值或高风险请求保留高精度模型。通过路由策略控制质量和成本。

## 4.13 自动扩缩容

推理平台要根据流量变化扩缩容。

### 4.13.1 扩缩容指标

普通服务常用 CPU 或 QPS 扩缩容，但大模型服务更适合看：

1. 队列长度。
2. 排队时间。
3. GPU 利用率。
4. 显存占用。
5. KV cache 使用率。
6. token 吞吐。
7. P95/P99 延迟。

### 4.13.2 冷启动问题

大模型实例冷启动慢，因为要拉取权重、加载到 GPU、初始化推理引擎和 warmup。

因此不能等流量已经打满再扩容。需要预测扩容、保留 warm pool，或按时间规律提前扩容。

### 4.13.3 缩容策略

缩容要谨慎，不能杀掉正在生成的请求。可以先停止接收新请求，等待已有请求完成，再释放实例。

### 4.13.4 多区域扩展

如果服务全球用户，要考虑区域就近路由、模型权重分发、数据合规、跨区域故障转移和成本差异。

## 4.14 缓存设计

推理平台可以有多种缓存。

### 4.14.1 Response cache

对完全相同的请求，可以缓存最终回答。但生成模型有随机性，且用户上下文和权限不同，所以 response cache 适用范围有限。

适合缓存的是确定性、低风险、高重复的请求，例如固定 FAQ、分类、模板化摘要。

### 4.14.2 Prefix cache

前面讲过，prefix cache 缓存共享前缀的 KV，适合相同 system prompt 或相同长文档前缀。

### 4.14.3 Embedding cache

虽然 embedding 更常在 RAG 系统中讨论，但推理平台如果也服务 embedding 模型，可以缓存高频文本的 embedding。

### 4.14.4 缓存安全

缓存必须带租户、权限、模型版本和参数信息。不能把 A 用户的上下文或回答缓存后返回给 B 用户。

## 4.15 监控与告警

推理平台监控要覆盖服务、模型、GPU、质量和成本。

### 4.15.1 服务指标

1. QPS。
2. TTFT。
3. 端到端延迟。
4. tokens/s。
5. 排队时间。
6. 错误率。
7. 超时率。
8. streaming 中断率。

### 4.15.2 GPU 指标

1. GPU 利用率。
2. 显存使用。
3. KV cache 使用率。
4. batch size。
5. prefill 吞吐。
6. decode 吞吐。
7. OOM 次数。
8. 节点健康。

### 4.15.3 模型质量指标

推理平台本身不一定负责完整模型评估，但要能按模型版本关联上层反馈、错误率、安全命中率和回归报告。

### 4.15.4 成本指标

成本要按模型、租户、业务、区域和请求类型拆分。关键指标包括 token 用量、GPU 小时、缓存命中率、单位请求成本和单位成功任务成本。

## 4.16 高可用与降级

推理平台常见故障包括：

1. 某个模型实例 OOM。
2. 某个节点 GPU 故障。
3. 某个模型版本质量异常。
4. 流量突增导致排队。
5. 权重存储不可用。
6. 网络抖动导致 streaming 中断。
7. 安全审核服务超时。

对应策略包括：

1. 多副本部署。
2. 健康检查和自动摘除。
3. 模型版本快速回滚。
4. 限流和排队。
5. 降级到小模型或旧模型。
6. 缩短最大输出长度。
7. 非核心请求转异步。
8. 保守安全策略。

降级要按业务风险设计。普通闲聊可以降级到小模型，高风险任务不能跳过安全控制。

## 4.17 安全与隔离

推理平台虽然主要做模型服务，但也有安全责任。

### 4.17.1 租户隔离

不同租户的请求、日志、缓存、配额和模型权限要隔离。尤其是 prefix cache 和 response cache，必须防止跨租户泄漏。

### 4.17.2 模型权限

不是所有用户都能调用所有模型。高成本模型、内部模型、安全敏感模型需要权限控制和审批。

### 4.17.3 输入输出日志

推理日志可能包含隐私和商业机密。平台要支持脱敏、采样、加密、访问控制和保留期限。

### 4.17.4 滥用防护

推理平台要支持 API key 限流、异常调用检测、配额、风控和审计。

这里的滥用防护目标是保护服务、用户和数据安全，不提供绕过其他平台或第三方限制的方法。

## 4.18 成本优化

推理平台成本优化是面试重点。

常见手段包括：

1. 模型路由：简单请求走小模型，复杂请求走大模型。
2. Batching：提高 GPU 利用率。
3. 量化：降低显存和计算成本。
4. Prefix cache：减少重复 prefill。
5. Response cache：缓存确定性高频请求。
6. Speculative decoding：提升 decode 速度。
7. 自动扩缩容：减少空闲 GPU。
8. 长度限制：控制最大输入和输出。
9. 离线批处理：非实时任务用批量推理。

成本优化要看质量影响。例如 INT4 量化可能节省成本，但如果代码任务准确率下降，整体业务收益可能变差。

## 4.19 面试答题模板

如果面试官问“设计推理平台”，可以这样回答。

第一，澄清范围。

我先按多模型大模型推理平台设计，支持 Chat、RAG、Agent 等上层业务调用，提供统一 API、模型版本管理、在线部署、路由、batching、流式输出、监控和计费。

第二，定义指标。

核心指标包括 TTFT、tokens/s、P95/P99 延迟、QPS、GPU 利用率、KV cache 使用率、错误率、SLA、单位 token 成本和模型质量回归。

第三，容量估算。

根据不同业务的 QPS、输入输出 token、内部调用放大倍数估算峰值 token 吞吐，并区分 prefill 和 decode 压力。

第四，画架构。

系统包括 API Gateway、Tenant Service、Model Registry、Deployment Controller、Router、Queue Manager、Inference Engine、Cache Manager、GPU Scheduler、Monitoring、Billing 和 Admin Console。

第五，讲请求链路。

请求进入后鉴权、限流、配额检查，Router 选择模型和实例，Queue Manager 做优先级和合批，Inference Engine 执行 prefill/decode 并流式返回，监控和计费异步记录。

第六，展开关键难点。

重点讲 batching、KV cache、长短请求隔离、多 GPU、量化、自动扩缩容和灰度发布。

第七，讲安全和成本。

租户隔离、模型权限、日志脱敏、缓存隔离是安全重点；成本通过模型路由、batching、量化、缓存和扩缩容优化。

## 4.20 常见面试追问

### 4.20.1 推理平台和 ChatGPT 服务是什么关系？

ChatGPT 服务是面向用户的产品系统，包含会话、上下文、安全和交互；推理平台是底层模型服务基础设施，提供模型部署、路由、推理执行、batching、监控和计费。ChatGPT 服务通常调用推理平台。

### 4.20.2 为什么不能按请求数做负载均衡？

因为大模型请求成本由 token 决定。一个 100 token 请求和一个 10000 token 请求对 prefill、KV cache 和延迟的影响完全不同。路由要看 token、队列、显存和实例状态。

### 4.20.3 Continuous batching 解决什么问题？

它解决生成长度不一致导致 GPU 空转的问题。新请求可以动态加入，已完成请求可以退出，让 decode 过程持续保持较高 batch 利用率。

### 4.20.4 KV cache 为什么会成为瓶颈？

KV cache 随上下文长度、batch size、层数和 head 数增长。长上下文和高并发会占用大量显存，管理不好会 OOM 或产生显存碎片。

### 4.20.5 如何降低 TTFT？

可以减少排队时间、优化 prefill、使用 prefix cache、缩短 prompt、做长短请求隔离、就近路由、模型 warmup，并避免同步调用太多外部服务。

### 4.20.6 如何做自动扩缩容？

不能只看 CPU，要看队列长度、排队时间、GPU 利用率、KV cache 使用率、token 吞吐和 P95 延迟。还要考虑模型冷启动慢，使用预测扩容或 warm pool。

### 4.20.7 量化模型如何上线？

先离线评估质量和安全，再小流量灰度，监控不同任务的效果、延迟和成本。可以只让低风险或简单请求走量化模型，高风险任务继续走高精度模型。

### 4.20.8 推理平台如何支持回滚？

模型版本、tokenizer、chat template、推理配置和路由策略都要版本化。发布时保留旧版本部署，异常时 Router 快速切回旧版本。

## 4.21 本章小结

推理平台的核心价值，是把昂贵、不稳定、资源敏感的大模型推理，变成统一、可控、低延迟、高吞吐、可观测、可计费的在线基础设施。

本章要记住几句话：

1. 推理平台不是一个模型 HTTP 服务，而是模型注册、部署、路由、调度、执行、监控和计费的完整系统。
2. 容量估算要按 token 和业务调用放大倍数估算，不能只看 QPS。
3. Prefill 和 decode 是两类不同瓶颈，优化方法不同。
4. Continuous batching 提高 GPU 利用率，但要平衡延迟。
5. KV cache 是长上下文和高并发推理的关键资源。
6. 多 GPU 部署要考虑 tensor parallel、pipeline parallel、拓扑和通信开销。
7. 自动扩缩容要考虑模型冷启动，不能只按 CPU 指标。
8. 安全重点是租户隔离、模型权限、日志脱敏、缓存隔离和滥用防护。

如果面试中被问到推理平台，最好的回答是从模型如何注册上线开始，讲到请求如何路由、排队、合批、执行、流式返回，再讲如何监控、扩缩容、降级、回滚和控制成本。
