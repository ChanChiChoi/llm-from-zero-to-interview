# 第一章：系统设计方法论

这一章是第十一册的总方法论。前面几册更多关注模型、训练、推理、数据、评估、安全和论文研究；从这一册开始，我们把这些知识组合成系统设计面试中的完整方案。

大模型系统设计和传统后端系统设计有相同点，也有明显不同。相同点是都要讲需求、容量、架构、存储、缓存、监控、容灾、成本和扩展性。不同点是大模型系统多了模型推理、GPU 资源、上下文窗口、token 成本、延迟抖动、流式输出、RAG、Agent 工具调用、安全审核、评估闭环和模型版本管理。

本章重点：需求澄清、指标、容量估算、核心架构、瓶颈、安全、监控、成本、扩展性。

## 1.1 为什么大模型系统设计单独成为一类面试

早期系统设计面试常问短链接、微博 feed、秒杀、搜索、聊天系统、网盘、推荐系统。这些题主要围绕传统互联网架构：请求入口、业务服务、数据库、缓存、消息队列、离线任务、监控和容灾。

大模型应用出现后，系统设计题发生了变化。面试官不再只问“如何支撑高并发”，还会问：

1. 如何设计一个 ChatGPT 类服务？
2. 如何设计一个多租户推理平台？
3. 如何设计 RAG 系统，让答案可引用、可更新、低幻觉？
4. 如何设计 Agent 平台，支持工具调用和权限控制？
5. 如何设计训练平台，支持大规模分布式训练和故障恢复？
6. 如何设计评估平台，持续发现模型回归？
7. 如何设计安全审核系统，降低有害输出和隐私泄漏？
8. 如何设计模型路由与缓存系统，兼顾成本、质量和延迟？

这些题的共同点是：系统里有一个或多个模型，而且模型不是普通函数。模型调用有高成本、高延迟、高不确定性和高风险。

传统服务调用一个普通函数，输入相同通常输出相同，延迟比较稳定，成本主要来自 CPU、内存和数据库。大模型调用不同：输出可能受 decoding 参数影响；同一个请求的 token 长度差异很大；GPU 推理容易出现排队；长上下文会显著增加成本；模型可能幻觉、拒答、泄露敏感信息或调用错误工具。

所以大模型系统设计要回答两个问题：

1. 如何把大模型能力稳定、低成本、可监控地封装成服务？
2. 如何围绕模型的不确定性设计防护、评估、回滚和治理闭环？

## 1.2 系统设计面试的基本流程

系统设计面试最忌讳一上来画架构图。成熟的回答应该按流程推进。

推荐流程如下：

1. 澄清需求。
2. 定义指标。
3. 做容量估算。
4. 画核心架构。
5. 设计关键数据流。
6. 分析瓶颈和 trade-off。
7. 补充安全、监控、容灾和成本。
8. 总结扩展方向。

这八步不是死板模板，而是帮你避免漏掉关键点。

例如面试官问“设计一个 RAG 系统”。如果你直接说 embedding、向量库、top-k、prompt 拼接，就只回答了局部。更完整的回答要先问：面向什么业务？文档规模多大？更新频率多高？是否需要权限控制？是否要求引用来源？延迟预算是多少？能不能离线建索引？对错误答案的容忍度如何？

## 1.3 第一步：需求澄清

需求澄清决定后面所有设计。很多候选人觉得澄清是在拖时间，其实它是系统设计的核心能力。

大模型系统的需求可以分成五类。

| 需求类型 | 要问什么 | 例子 |
| --- | --- | --- |
| 用户需求 | 谁用，解决什么问题 | 面向 C 端聊天、企业知识库、客服、代码助手 |
| 功能需求 | 系统必须做什么 | 多轮对话、上传文档、引用来源、工具调用、语音输入 |
| 非功能需求 | 延迟、吞吐、可用性、成本 | P95 延迟、QPS、SLA、GPU 成本上限 |
| 安全需求 | 什么不能发生 | 泄露隐私、越权访问、有害输出、工具误调用 |
| 运营需求 | 如何上线、监控、迭代 | A/B 测试、模型版本、灰度发布、回滚 |

### 1.3.1 用户是谁

同样是聊天系统，C 端助手、企业内部助手和客服助手的设计完全不同。

C 端助手通常关注体验、成本、内容安全和大规模并发。企业内部助手更关注权限、知识库更新、审计和数据隔离。客服助手更关注准确率、工单系统集成、人工接管和 SLA。

所以开头可以问：这个系统面向个人用户、企业用户、内部员工还是开发者？不同用户的任务和风险不同。

### 1.3.2 系统边界是什么

系统边界决定你设计到哪里为止。

例如“设计 ChatGPT 服务”可以包含账号、计费、对话、模型推理、插件、文件上传、安全审核、监控和训练反馈。如果面试时间有限，需要和面试官确认重点：是设计端到端产品，还是重点设计推理服务和对话管理？

一个好的回答可以说：我先按端到端聊天服务设计，包括用户请求、会话管理、模型推理、安全审核、流式返回、日志和反馈；如果时间有限，我会重点展开推理链路、上下文管理和可靠性。

### 1.3.3 成功标准是什么

没有成功标准，就无法判断架构好坏。

例如 RAG 系统的成功标准可能是：答案准确率、引用支持率、不可回答问题的拒答率、P95 延迟、索引更新延迟、单位查询成本。Agent 系统的成功标准可能是：任务成功率、平均工具调用次数、失败恢复率、越权调用率、单位任务成本。

在面试中，主动定义成功标准会让你显得更像系统负责人，而不是只会拼组件。

## 1.4 第二步：定义指标

大模型系统指标要同时覆盖用户体验、模型质量、系统性能、安全和成本。

### 1.4.1 用户体验指标

常见指标包括：

1. 首 token 延迟，也就是 TTFT。
2. 输出 token 速度，也就是 tokens per second。
3. 端到端延迟。
4. 流式输出稳定性。
5. 对话中断率和重试率。
6. 用户满意度、点赞率、继续追问率。

大模型服务通常不能只看整体响应时间。用户对首 token 延迟很敏感，因为流式输出只要尽快开始，用户会感觉系统在工作；但如果首 token 很慢，即使总时间不长，体验也会差。

### 1.4.2 模型质量指标

不同系统质量指标不同。

ChatGPT 类服务可以看 helpfulness、事实性、指令遵循、拒答质量、多轮一致性。RAG 系统可以看答案准确率、引用正确率、faithfulness、检索 recall。代码助手可以看 pass@k、编译率、单测通过率。安全审核系统可以看 precision、recall、漏报率、误杀率。

质量指标一定要和业务任务绑定。只说“模型效果好”没有意义。

### 1.4.3 系统性能指标

常见指标包括：

1. QPS 和并发会话数。
2. P50、P95、P99 延迟。
3. GPU 利用率。
4. 显存利用率。
5. 请求排队时间。
6. prefill 时间和 decode 时间。
7. KV cache 命中和占用。
8. 错误率和超时率。

大模型推理要特别区分 prefill 和 decode。Prefill 处理输入上下文，长 prompt 会让 prefill 变慢；decode 逐 token 生成，输出越长越占时间。不同瓶颈对应不同优化方法。

### 1.4.4 安全指标

安全指标包括：

1. 有害输出率。
2. 越权访问率。
3. PII 泄漏率。
4. Prompt injection 成功率。
5. 工具误调用率。
6. 过度拒答率。
7. 审核误杀率和漏报率。

安全系统不能只追求“拦得多”。如果误杀太高，正常用户体验会很差；如果漏报太高，风险不可控。因此要同时看 precision 和 recall，还要看不同风险等级下的策略。

### 1.4.5 成本指标

大模型系统成本通常是面试追问重点。

成本指标包括：

1. 每千 token 成本。
2. 每次请求平均成本。
3. 每个成功任务成本。
4. GPU 小时成本。
5. 缓存命中节省成本。
6. 不同模型路由带来的成本变化。

系统设计要承认一个事实：最强模型不一定适合所有请求。很多生产系统会用模型路由，把简单请求交给小模型，把复杂请求交给大模型。

## 1.5 第三步：容量估算

容量估算不是为了算得精确，而是为了暴露系统规模和主要瓶颈。

传统系统估算常看用户数、QPS、读写比例、存储容量、带宽。大模型系统还要估算 token。

### 1.5.1 请求规模估算

假设设计一个企业智能助手，可以这样估算：

1. 日活用户 10 万。
2. 每人每天 20 次请求。
3. 每日请求数 200 万。
4. 平均分摊到全天约 23 QPS。
5. 峰值按 5 倍估算约 115 QPS。

这只是入口请求 QPS。对于 RAG 或 Agent 系统，一个用户请求可能触发多次检索、多次模型调用和多次工具调用，所以内部 QPS 会放大。

### 1.5.2 Token 规模估算

大模型系统的成本和延迟更接近 token 驱动，而不是请求驱动。

例如：

1. 平均输入 1500 token。
2. 平均输出 500 token。
3. 每次请求 2000 token。
4. 峰值 115 QPS。
5. 峰值 token 吞吐约 23 万 token/s。

这个估算会直接影响需要多少 GPU、是否需要模型分层、是否要缓存、是否限制上下文长度。

### 1.5.3 存储估算

不同系统存储对象不同。

ChatGPT 类服务要存对话历史、用户配置、反馈、日志、审核记录。RAG 系统要存原文、切分 chunk、embedding、索引、权限元数据。训练平台要存数据集、checkpoint、实验日志、评估结果。

估算时要关注两点：

1. 热数据和冷数据分层。
2. 原始数据、派生数据和索引数据的倍数关系。

例如一个 1TB 文档库，切分后 chunk 元数据、embedding 和索引可能额外占数百 GB。日志如果保留完整 prompt 和 response，也可能快速膨胀，还会带来隐私风险。

## 1.6 第四步：核心架构

大模型应用的通用架构可以分成八层。

| 层级 | 作用 | 常见组件 |
| --- | --- | --- |
| 接入层 | 接收请求、鉴权、限流 | API Gateway、WebSocket、身份认证 |
| 编排层 | 管理任务流程 | Orchestrator、workflow engine、Agent runtime |
| 上下文层 | 组织 prompt 和记忆 | conversation store、prompt builder、memory manager |
| 知识层 | 提供外部知识 | retriever、vector DB、reranker、document store |
| 模型层 | 执行推理或训练 | inference server、training cluster、model registry |
| 安全层 | 输入输出审核和权限控制 | moderation、policy engine、PII detector |
| 观测层 | 监控质量、性能和成本 | metrics、tracing、logging、eval pipeline |
| 反馈层 | 收集反馈并迭代 | user feedback、labeling、A/B testing |

不是每个系统都需要完整八层，但面试中可以用它作为总图。

### 1.6.1 接入层

接入层负责鉴权、限流、配额、路由、请求校验和协议转换。

大模型服务常见协议包括 HTTP API、SSE、WebSocket 和 gRPC。聊天产品常用流式输出，所以 SSE 或 WebSocket 很常见。

接入层还要做租户隔离。企业客户通常需要不同 API key、配额、模型权限、数据访问权限和审计日志。

### 1.6.2 编排层

编排层决定一个用户请求如何被处理。

简单聊天可能只需要 prompt builder 加一次模型调用。RAG 请求需要 query rewrite、检索、rerank、上下文拼接、生成和引用校验。Agent 请求可能需要规划、工具调用、观察结果、再推理和终止判断。

编排层的设计重点是可控性。不能让流程完全散落在 prompt 里，否则难以监控、调试和回滚。

### 1.6.3 模型层

模型层包括模型加载、batching、KV cache、并行策略、调度、降级和版本管理。

面试中要主动提到模型服务不是普通 RPC。它需要处理长短请求混部、prefill/decode 分离、continuous batching、显存碎片、模型热更新、灰度发布和 GPU 故障。

### 1.6.4 安全层

安全层不能只放在最后。更合理的是多点防护：输入审核、检索权限过滤、工具调用授权、模型输出审核、日志脱敏和人工升级。

例如企业 RAG 系统中，即使模型本身无害，如果检索层越权返回了用户无权查看的文档，最终答案也会泄露信息。所以权限控制必须在检索前或检索时完成，而不是只靠模型“不要泄露”。

## 1.7 第五步：关键数据流

系统设计面试中，画完组件还不够，要讲清请求怎么流动。

以一个通用大模型应用为例，在线请求链路可以是：

1. 用户请求进入 API Gateway。
2. 鉴权、限流、配额检查。
3. 输入安全审核和 PII 检测。
4. 会话服务读取历史上下文。
5. 编排服务判断是否需要检索、工具或直接回答。
6. 如果需要 RAG，先做 query rewrite，再检索、rerank 和权限过滤。
7. Prompt builder 组装 system prompt、用户问题、历史摘要和外部证据。
8. 模型路由选择合适模型。
9. 推理服务执行 prefill 和 decode，流式返回 token。
10. 输出审核和引用校验。
11. 返回给用户，同时异步写日志、指标和反馈事件。

这个链路里有同步路径和异步路径。

同步路径影响用户延迟，必须尽量短。异步路径包括日志写入、离线评估、用户反馈处理、数据标注和训练样本构建，可以通过消息队列解耦。

## 1.8 第六步：瓶颈分析

大模型系统的瓶颈通常来自六类。

### 1.8.1 GPU 推理瓶颈

GPU 推理瓶颈包括显存不够、batching 不充分、decode 阶段吞吐低、KV cache 占用大、长请求拖慢短请求。

常见优化包括：

1. Continuous batching。
2. Paged attention。
3. KV cache 管理。
4. Tensor parallel 和 pipeline parallel。
5. Speculative decoding。
6. 量化。
7. 模型路由。
8. 限制最大输入和输出长度。

### 1.8.2 长上下文瓶颈

上下文越长，prefill 成本越高，KV cache 越大，延迟越高。

优化方式包括：

1. 历史对话摘要。
2. 只保留相关上下文。
3. RAG 替代盲目塞全文。
4. Prompt cache。
5. 长短请求分队列。
6. 对超长文档做分段处理。

### 1.8.3 检索瓶颈

RAG 系统中，检索瓶颈可能来自 embedding 计算、向量索引查询、rerank 模型、权限过滤和跨库检索。

优化方式包括：

1. 预计算 embedding。
2. ANN 索引。
3. 分层召回。
4. 缓存热门 query。
5. 轻量 reranker 加重 reranker 分级。
6. 权限元数据前置过滤。

### 1.8.4 Agent 工具瓶颈

Agent 系统的瓶颈不只是模型延迟，还包括工具调用慢、外部 API 不稳定、循环调用、错误恢复和状态管理。

优化方式包括：限制最大步数、工具超时、并行工具调用、结果缓存、明确终止条件、人工接管。

### 1.8.5 数据和日志瓶颈

大模型系统日志很大，因为 prompt、response、检索证据、工具调用轨迹都可能很长。

要避免把所有内容无脑同步写入主链路。更合理的是：在线链路写轻量事件，详细日志异步落地；敏感字段脱敏；按策略保留；训练可用数据单独治理。

### 1.8.6 评估瓶颈

大模型系统迭代快，如果没有自动评估和回归测试，很容易上线一个更差的版本。

评估瓶颈包括数据集维护、LLM judge 成本、人工标注成本、线上反馈噪声和评估污染。

## 1.9 第七步：安全设计

大模型系统的安全设计要从“模型会不会乱说”扩展到“系统会不会让模型接触不该接触的东西，执行不该执行的动作”。

### 1.9.1 输入安全

输入安全包括有害请求识别、prompt injection 检测、PII 检测、恶意文件扫描、速率限制和异常行为检测。

对于 RAG 和 Agent，prompt injection 尤其重要。用户或文档中可能包含指令，诱导模型忽略系统规则、泄露上下文或调用工具。防护不应只靠 prompt，要结合权限、工具白名单和输出审计。

### 1.9.2 权限安全

权限要在系统层实现，而不是交给模型记住。

典型原则是：

1. 检索前做身份鉴权。
2. 检索时按权限过滤文档。
3. 工具调用前做权限检查。
4. 高风险动作需要二次确认。
5. 所有敏感操作写审计日志。

### 1.9.3 输出安全

输出安全包括有害内容审核、隐私泄漏检测、版权风险、医疗法律金融等高风险建议控制、引用校验和拒答策略。

输出审核要注意误杀率。过度拒答会让用户觉得系统不可用。更好的策略是按风险等级分层：低风险直接回答，中风险加免责声明或限制，高风险拒答或转人工。

### 1.9.4 日志安全

日志里可能包含用户隐私、商业机密、认证信息和模型输出。日志系统要做脱敏、访问控制、保留期限、审计和训练数据隔离。

不能因为要“收集训练数据”就无限制保存用户输入。

## 1.10 第八步：监控与可观测性

大模型系统监控要覆盖四层。

### 1.10.1 基础设施监控

包括 GPU 利用率、显存、温度、网络、磁盘、节点健康、容器重启、驱动错误。

### 1.10.2 服务监控

包括 QPS、延迟、错误率、超时、排队时间、batch size、token 吞吐、模型实例健康。

### 1.10.3 质量监控

包括用户反馈、自动评估分、抽样人工审核、幻觉率、引用错误率、任务成功率、拒答率。

质量监控比传统系统难，因为输出质量不是简单 200 状态码能反映的。

### 1.10.4 成本监控

包括每个租户、每个模型、每类请求的 token 用量和 GPU 成本。

成本监控要能定位异常。例如某个 Agent workflow 进入循环，可能导致单个请求调用几十次模型，成本迅速放大。

## 1.11 成本优化

大模型系统的成本优化通常比传统系统更重要，因为模型推理单次成本高。

常见策略包括：

1. 模型分层：简单任务用小模型，复杂任务用大模型。
2. Prompt 压缩：减少无关上下文。
3. 缓存：缓存系统 prompt、热门问答、检索结果和 prefix KV。
4. 批处理：提高 GPU 利用率。
5. 量化：降低显存和计算成本。
6. Speculative decoding：用小模型辅助大模型解码。
7. 限额：对租户、用户和任务设置 token budget。
8. 离线化：非实时任务批量处理。

成本优化不能只看便宜，还要看质量损失。例如把所有请求路由到小模型可能成本低，但复杂任务失败率高，最终每个成功任务成本反而更高。

## 1.12 扩展性设计

扩展性包括流量扩展、模型扩展、数据扩展和功能扩展。

### 1.12.1 流量扩展

流量增长时，需要横向扩展 API 服务、编排服务、检索服务和模型服务。

模型服务扩展最难，因为 GPU 资源昂贵且调度复杂。要考虑不同模型池、优先级队列、租户配额、热点隔离和降级策略。

### 1.12.2 模型扩展

系统不能假设永远只有一个模型。生产中常有 base model、chat model、code model、embedding model、reranker、moderation model、small routing model。

因此需要 model registry、版本管理、灰度发布、A/B 测试和回滚机制。

### 1.12.3 数据扩展

RAG 和训练系统会遇到数据规模增长。要考虑文档增量更新、embedding 重算、索引重建、数据血缘、权限变更和删除请求。

### 1.12.4 功能扩展

从普通聊天扩展到文件问答、语音、多模态、Agent、插件和企业集成时，如果早期架构完全写死，会很难演进。

因此编排层和工具层要有清晰接口，但也不要过度抽象。面试中可以说：第一版先支持核心链路，接口上保留扩展点，例如工具注册、模型路由、评估事件和安全策略。

## 1.13 高可用与降级

大模型系统的故障模式包括：模型服务不可用、GPU 节点故障、队列积压、检索系统超时、外部工具失败、安全审核服务故障、成本异常。

常见高可用策略包括：

1. 多副本模型服务。
2. 不同模型池隔离。
3. 请求超时和重试。
4. 熔断和限流。
5. 降级到小模型或缓存答案。
6. 非关键步骤异步化。
7. 工具失败时返回部分结果。
8. 安全服务故障时采用保守策略。

降级策略要和业务风险匹配。普通闲聊可以降级到小模型；医疗、金融、法律或高风险工具调用不能简单降级成未经审核的回答。

## 1.14 常见 trade-off

系统设计面试中，面试官很看重 trade-off。下面是大模型系统常见取舍。

| 取舍 | 一边 | 另一边 |
| --- | --- | --- |
| 质量 vs 成本 | 大模型、长上下文、多轮推理 | 小模型、短上下文、缓存 |
| 延迟 vs 吞吐 | 小 batch、快速返回 | 大 batch、高 GPU 利用率 |
| 安全 vs 体验 | 严格审核、保守拒答 | 自由回答、低摩擦 |
| 新鲜度 vs 稳定性 | 高频更新索引和模型 | 慢更新、强验证 |
| 自动化 vs 可控性 | Agent 自主执行 | 固定 workflow 和人工确认 |
| 召回 vs 精度 | 检索更多候选 | 更严格 rerank 和过滤 |
| 日志完整性 vs 隐私 | 保留详细上下文 | 脱敏、采样、短保留 |

好的回答不是永远选择某一边，而是根据业务场景解释为什么。

例如企业知识库问答中，权限和准确性通常比极致流畅更重要；C 端闲聊产品中，首 token 延迟和互动体验可能更关键；代码生成系统中，单测通过率和安全扫描比回答风格更重要。

## 1.15 面试中的答题结构

如果遇到任意大模型系统设计题，可以用下面模板回答。

第一，复述并澄清需求。

“我先确认一下，我们要设计的是面向什么用户、支持哪些核心功能、规模大概多大、是否有安全和成本约束。”

第二，定义目标和指标。

“我会用 P95 延迟、TTFT、tokens/s、质量指标、风险指标和单请求成本衡量系统。”

第三，做粗略估算。

“假设 DAU、请求数、输入输出 token 和峰值倍数，可以估算峰值 token 吞吐和存储规模。”

第四，画高层架构。

“整体分为接入层、编排层、上下文层、知识层、模型层、安全层、观测层和反馈层。”

第五，展开关键链路。

“用户请求经过鉴权、审核、上下文构造、检索或工具、模型路由、推理、输出审核和日志反馈。”

第六，分析瓶颈。

“主要瓶颈可能在 GPU 推理、长上下文、检索、工具调用、日志和评估。”

第七，补充安全和监控。

“权限控制不能交给模型，安全要覆盖输入、检索、工具、输出和日志；监控要覆盖系统、质量、安全和成本。”

第八，讲扩展和 trade-off。

“后续可以用模型路由、缓存、量化、RAG、灰度发布和评估闭环优化，但要在质量、成本、延迟和安全之间取舍。”

## 1.16 小白例子：开一家 AI 咨询前台

可以把大模型系统想象成一家 AI 咨询前台。

用户进门提问，相当于请求进入接入层。前台先确认身份和权限，相当于鉴权。然后判断问题类型：普通咨询直接回答，专业问题查资料，复杂问题找专家，危险问题拒绝或转人工。

资料室相当于 RAG 知识库，专家相当于大模型或工具。前台不能随便把公司机密资料拿给外人看，所以资料室要按权限检索。前台回答前还要检查是否说错、是否泄露隐私、是否需要引用来源。

老板还会关心：每天来了多少人，平均等多久，哪些问题答错，哪些专家成本太高，哪些员工越权查资料。这个就是监控、评估、成本和审计。

这个例子说明：大模型系统不是“用户问题直接丢给模型”，而是一个有接待、权限、资料、专家、安全、记录和复盘的完整组织。

## 1.17 常见面试题

### 1.17.1 系统设计一开始为什么不能直接画图？

因为没有需求和指标，架构图没有评价标准。不同用户、规模、延迟、安全和成本约束会导致完全不同的架构。先澄清需求才能做合理取舍。

### 1.17.2 大模型系统和传统后端系统最大的区别是什么？

大模型系统的核心服务是高成本、高延迟、高不确定性的模型调用。它需要关注 token、GPU、上下文、模型质量、安全审核、评估闭环和模型版本，而不仅是数据库、缓存和微服务。

### 1.17.3 为什么要估算 token，而不只估算 QPS？

因为大模型推理成本和延迟主要由输入输出 token 决定。同样 100 QPS，如果平均 200 token 和平均 8000 token，GPU 需求完全不同。

### 1.17.4 首 token 延迟和总延迟哪个更重要？

都重要。首 token 延迟影响用户感知和交互启动，总延迟影响任务完成时间。聊天系统通常非常关注 TTFT；批处理任务可能更关注总吞吐和单位成本。

### 1.17.5 权限控制能不能靠 prompt？

不能。Prompt 可以辅助提醒模型，但不能作为安全边界。权限控制必须在系统层做，包括鉴权、检索过滤、工具授权和审计。

### 1.17.6 如何降低大模型服务成本？

可以用模型路由、小模型处理简单请求、缓存、prompt 压缩、batching、量化、speculative decoding、限制 token budget、离线批处理等方法。但要同时监控质量和失败率。

### 1.17.7 如何设计监控？

分四层：基础设施监控、服务监控、质量监控和成本监控。大模型系统还要监控幻觉率、拒答率、引用正确率、工具调用成功率、安全违规率和 token 成本。

### 1.17.8 系统上线后如何持续变好？

通过用户反馈、日志采样、自动评估、人工标注、A/B 测试、失败案例分析、数据回流和模型版本迭代形成闭环。没有评估闭环的系统很难稳定改进。

## 1.18 本章小结

大模型系统设计的核心，不是把所有热门组件都堆进架构图，而是在明确需求和约束后，设计一条稳定、可控、可观测、可扩展的模型能力交付链路。

本章要记住几句话：

1. 系统设计先澄清需求，再画架构。
2. 大模型系统要同时看用户体验、质量、性能、安全和成本。
3. Token 规模比单纯 QPS 更能决定推理资源。
4. 模型服务不是普通 RPC，要考虑 GPU、batching、KV cache、长上下文和版本管理。
5. 权限、安全和审计必须在系统层实现，不能只靠 prompt。
6. 监控不能只看错误率，还要看质量、幻觉、安全和成本。
7. 成本优化要以每个成功任务的质量和成本为目标，而不是盲目用小模型。
8. 好的大模型系统最终要形成评估、反馈、数据和模型迭代闭环。

如果面试中遇到任何大模型系统设计题，可以先用本章方法论搭骨架，再根据题目展开具体模块。
