# 第十章：统一多模态模型

前面几章分别讲了图文对齐、VLM、图像生成、视频生成和语音生成。现实中的智能助手不会只处理一种模态：用户可能说一句话、发一张截图、上传一段视频、要求生成图片，再继续用语音追问。统一多模态模型的目标，是用尽量统一的架构、表示和接口处理文本、图像、语音、视频等多种输入和输出。

统一多模态不是简单把多个模型串起来。它涉及模态 token 化、共享表示空间、跨模态对齐、统一训练目标、Any-to-Any 输入输出、跨模态 reasoning、实时交互、评估和安全。本章建立这条路线的整体图景。

## 10.1 什么是统一多模态模型

统一多模态模型希望一个系统能够处理多种模态输入和输出。

输入可以是：

1. 文本。
2. 图片。
3. 语音。
4. 视频。
5. 文档。
6. 截图。

输出可以是：

1. 文本回答。
2. 图片。
3. 语音。
4. 视频。
5. 工具调用。
6. 操作动作。

Any-to-Any 是理想目标：任意模态输入，任意模态输出。

例如：

```text
语音 + 图片 -> 文本回答
文本 -> 图片
视频 -> 语音总结
截图 -> 工具操作
```

面试回答：

```text
统一多模态模型的目标是用统一架构或统一接口处理文本、图像、语音、视频等多种模态，并支持跨模态理解、推理和生成。它可以通过统一 token 化、共享 embedding 空间、模态专用 encoder/decoder 加中心 LLM，或 Any-to-Any 生成框架实现。
```

## 10.2 统一的三种层次

“统一”可以有不同层次。

### 10.2.1 接口统一

系统层统一输入输出接口，但内部仍是多个模型级联。

例如：

```text
ASR -> LLM -> TTS
Vision encoder -> LLM
Text-to-image model
```

用户感觉是一个助手，但内部是多个模块。

优点：工程可控，模块成熟。

缺点：误差会传递，端到端优化困难。

### 10.2.2 表示统一

不同模态被编码到相近或共享的 embedding 空间。

例如 CLIP 的图文对齐，或者统一 embedding 检索系统。

优点：支持跨模态检索和融合。

缺点：不一定能生成所有模态。

### 10.2.3 架构和目标统一

把不同模态都表示成 token，用统一 Transformer 或类似架构建模。

例如：

```text
text tokens + image tokens + audio tokens + video tokens -> unified transformer
```

优点：理论上最统一。

缺点：token 序列长、训练复杂、模态差异大、工程成本高。

## 10.3 统一 Token 化

LLM 成功的重要原因之一是文本 token 化。统一多模态希望把其他模态也 token 化。

### 10.3.1 文本 token

```text
text -> tokenizer -> token ids
```

### 10.3.2 图像 token

图像可以通过 VQ-VAE、VQGAN、image tokenizer 或 patch embedding 变成 tokens。

```text
image -> image tokenizer -> image tokens
```

### 10.3.3 语音 token

语音可以通过 audio codec 变成离散 tokens。

```text
waveform -> codec encoder -> audio tokens
```

### 10.3.4 视频 token

视频可以被切成 spatiotemporal tokens。

```text
video -> spatiotemporal tokenizer -> video tokens
```

统一 token 化的好处是能用类似语言模型的 next-token prediction 训练多模态序列。

难点是：图像、语音、视频 token 数远多于文本，序列长度和计算成本非常高。

### 10.3.5 关键公式与统一多模态速查

**统一输入序列**

统一多模态模型常把不同模态编码成一个可被中心模型消费的序列：

```math
S=
[S_{\mathrm{text}},S_{\mathrm{image}},S_{\mathrm{audio}},S_{\mathrm{video}},S_{\mathrm{special}}]
```

总输入 token 数：

```math
N_{\mathrm{in}}=
N_t+N_i+N_a+N_v+N_s
```

其中 `N_t` 是文本 token，`N_i` 是图像 token，`N_a` 是音频 token，`N_v` 是视频 token，`N_s` 是 role、separator、modality tag、image placeholder 等特殊 token。

**各模态 token 数估算**

图像 patch token：

```math
N_i=
\left\lceil\frac{H_i}{P_i}\right\rceil
\left\lceil\frac{W_i}{P_i}\right\rceil
```

音频 codec token：

```math
N_a=
\left\lceil D_a q_a\right\rceil K_a
```

视频时空 token：

```math
N_v=
\left\lceil\frac{T_v}{P_t}\right\rceil
\left\lceil\frac{H_v}{P_v}\right\rceil
\left\lceil\frac{W_v}{P_v}\right\rceil
```

这些公式解释了统一多模态为什么难：文本 token 常是几十到几千，图像、长音频、多帧视频很容易把上下文推爆。

**统一 embedding 或 projector**

如果不同模态先用专用 encoder，再投影到共享 hidden size `d`：

```math
H_m=E_m(X_m)W_m+b_m,
\qquad
H_m\in\mathbb{R}^{B\times N_m\times d}
```

其中 `m` 可以是 image、audio、video 等模态。这个公式对应“中心 LLM 路线”：每个模态先被翻译成 LLM 能读的 hidden tokens。

**Any-to-Any 条件生成**

Any-to-Any 可以抽象为给定输入模态集合 `M_in`，生成输出模态集合 `M_out`：

```math
p(Y_{M_{\mathrm{out}}}\mid X_{M_{\mathrm{in}}})
```

如果输出也离散化为 tokens，可以写成自回归形式：

```math
p(y_1,\ldots,y_U\mid S)=
\prod_{u=1}^{U}
p(y_u\mid y_{1:u-1},S)
```

其中 `S` 是统一输入序列，`y_u` 可以是文本 token、audio token、image token、tool token 或 action token。真实系统通常不会所有输出都用同一个 decoder，而是混合使用文本 decoder、TTS、image generator、tool executor 等模块。

**注意力和解码成本**

如果把所有输入 token 放进同一个 self-attention 上下文，注意力 cell 数约为：

```math
C_{\mathrm{self}}\approx N_{\mathrm{in}}^2
```

如果输出 token 数为 `N_out`，decoder cross 或条件生成成本的一个粗略代理是：

```math
C_{\mathrm{decode}}\approx N_{\mathrm{in}}N_{\mathrm{out}}
```

这解释了为什么多模态系统常需要 resampler、video sampler、audio compression、RAG、memory 压缩和路由，而不是把所有原始媒体都塞进上下文。

**多任务训练目标 mixture**

统一多模态训练往往不是单一 loss，而是多个目标的加权和：

```math
L=
\sum_{k=1}^{K}
\lambda_k L_k,
\qquad
\sum_{k=1}^{K}\lambda_k=1
```

例如文本 next-token、图文对齐、VQA SFT、ASR、TTS audio token、video understanding、安全拒答等。面试时要强调：`lambda_k` 不是数学装饰，而是能力配比、稳定性和灾难性遗忘的关键旋钮。

**路由和安全门禁**

统一助手常需要先路由：

```math
r=\mathrm{Route}(S)
```

再决定调用 OCR、ASR、video sampler、retriever、image generator、TTS 或 tool executor。一个简化门禁可以写成：

```math
G_{\mathrm{omni}}=
\mathbf{1}[
N_{\mathrm{in}}\le B_{\mathrm{ctx}}
\land
N_{\mathrm{out}}\le B_{\mathrm{out}}
\land
R_{\mathrm{unsafe}}=0
\land
A_{\mathrm{route}}=1
]
```

其中 `R_unsafe` 表示不可信媒体指令、未授权 voice cloning、未审核生成输出、工具越权等风险标记数，`A_route` 表示路由到的模块和任务需求一致。

## 10.4 Encoder-Decoder 式统一

一种现实路线是保留模态专用 encoder 和 decoder，中间用 LLM 或统一表示连接。

结构：

```text
image encoder  ->
audio encoder  -> shared reasoning model -> text/audio/image decoder
text tokenizer ->
```

优点：

1. 每种模态使用最适合的 encoder。
2. 可复用已有模型。
3. 工程实现更稳。

缺点：

1. 模块多，系统复杂。
2. 端到端一致性不强。
3. 不同模态之间可能存在信息瓶颈。

很多实际产品采用这种路线，因为它比完全统一 token 模型更可控。

## 10.5 中心 LLM 路线

当前很多多模态助手以 LLM 为中心。

结构：

```text
vision encoder -> projector -> LLM
audio encoder  -> projector -> LLM
LLM -> text output
LLM -> tool / image generator / TTS
```

LLM 负责：

1. 指令理解。
2. 多轮对话。
3. 推理。
4. 工具调度。
5. 输出规划。

其他模态模块负责感知或生成。

这种路线的优点是可以继承 LLM 的语言能力和推理能力；缺点是非文本模态信息可能被压缩成文本或少量 tokens，导致细节丢失。

## 10.6 Any-to-Any 模型

Any-to-Any 指任意输入模态到任意输出模态。

例如：

```text
text -> image
image -> text
speech -> speech
video -> text
text + image -> audio
```

Any-to-Any 系统需要：

1. 多模态输入编码。
2. 跨模态推理。
3. 多模态输出解码。
4. 统一任务格式。
5. 多任务训练数据。

难点：

1. 不同模态数据规模差异大。
2. 训练目标不统一。
3. 输出空间完全不同。
4. 延迟和成本高。
5. 评估困难。

## 10.7 跨模态 Reasoning

跨模态 reasoning 不是简单识别图像或转录语音，而是需要组合多个模态证据推理。

例子：

1. 看图表，结合问题计算增长率。
2. 看视频，判断动作因果。
3. 听语音语气，结合文本内容判断情绪。
4. 看截图，推断下一步操作。
5. 结合图片和说明书回答故障排查问题。

难点：

1. 模态证据是否被正确读取。
2. 证据是否被 LLM 使用。
3. 推理过程是否可靠。
4. 错误来自感知还是推理。

评估时要区分：模型是看错了、听错了，还是推理错了。

## 10.8 多模态 Memory 和 Context

统一多模态助手需要记住多轮上下文。

上下文可能包括：

1. 用户之前说过的话。
2. 上传过的图片。
3. 视频片段。
4. 音频转写。
5. 工具调用结果。
6. 文档证据。

挑战：

1. 多模态上下文占用大量 token。
2. 图片和视频不能无限放入上下文。
3. 需要检索和压缩历史。
4. 需要区分当前图片和历史图片。

这会引出多模态 RAG、多模态 memory 和上下文管理问题。

## 10.9 统一训练目标

不同模态常用不同训练目标：

1. 文本：next-token prediction。
2. 图文对齐：contrastive loss。
3. 图像生成：diffusion loss 或 image token loss。
4. 语音识别：sequence loss。
5. TTS：audio token loss 或 acoustic loss。
6. 视频生成：video diffusion loss。

统一多模态模型要处理这些目标如何组合。

常见方案：

1. 多阶段训练。
2. 多任务 mixture。
3. 共享主干 + 模态特定 head。
4. 统一 token 自回归目标。
5. 指令微调统一接口。

问题是不同目标之间可能冲突，需要仔细调数据比例和训练阶段。

## 10.10 数据混合

统一多模态训练需要多种数据：

1. 纯文本数据。
2. 图文 caption。
3. VQA。
4. OCR 和文档。
5. 图文检索对。
6. 文生图数据。
7. 视频字幕。
8. ASR/TTS 数据。
9. 多轮多模态对话。
10. 安全拒答数据。

数据混合难点：

1. 数据质量不均。
2. 模态比例不平衡。
3. 低质量数据污染能力。
4. 某模态过强导致其他能力退化。
5. 训练成本高。

## 10.11 模态冲突和幻觉

多模态模型可能遇到模态冲突。

例如：

1. 图片里是猫，文本说这是狗。
2. 视频画面和字幕不一致。
3. 用户语音和转写结果冲突。
4. OCR 读到恶意指令。

模型应该学会：

1. 区分模态证据来源。
2. 不盲信某个模态。
3. 在冲突时说明不确定。
4. 对不可信外部内容做隔离。

多模态幻觉可能来自任一环节：视觉看错、语音转错、检索错、LLM 推理错或生成模型编造。

## 10.12 工程架构：统一助手

一个实际统一多模态助手可能是模块化系统：

```text
input router
  -> ASR
  -> vision encoder / OCR
  -> document parser
  -> video sampler
  -> LLM planner
  -> tools / retriever / generator
  -> TTS / image generator / text response
```

这种系统未必是单一端到端模型，但对用户呈现统一体验。

工程关注点：

1. 路由。
2. 延迟。
3. 上下文管理。
4. 模态转换。
5. 权限和安全。
6. 失败回退。
7. 成本控制。

## 10.13 端到端统一 vs 模块化系统

端到端统一模型：

优点：

1. 理论上信息损失更少。
2. 交互更自然。
3. 可以联合优化。

缺点：

1. 数据和算力需求巨大。
2. 可控性和可解释性差。
3. 安全评估复杂。
4. 工程迭代慢。

模块化系统：

优点：

1. 可复用成熟模块。
2. 易调试。
3. 易做权限和安全控制。
4. 易替换升级。

缺点：

1. 误差传播。
2. 模态信息可能丢失。
3. 体验不如端到端自然。

现实产品通常在两者之间折中。

## 10.14 统一多模态评估

评估要覆盖：

1. 文本能力。
2. 图像理解。
3. OCR。
4. 图表理解。
5. 视频理解。
6. 语音识别和生成。
7. 图像/视频生成。
8. 跨模态推理。
9. 多轮上下文。
10. 安全和鲁棒性。

还要做错误归因：

1. 感知错误。
2. 对齐错误。
3. 推理错误。
4. 生成错误。
5. 工具错误。
6. 安全策略错误。

如果不做错误归因，很难知道应该改 vision encoder、ASR、LLM、数据还是 prompt。

## 10.15 安全问题

统一多模态模型的安全面更大。

风险包括：

1. 图片 prompt injection。
2. 语音指令欺骗。
3. Deepfake。
4. 隐私泄露。
5. 文档恶意内容注入。
6. 高风险图像误判。
7. 工具越权调用。
8. 生成违法内容。

安全原则：

1. 不可信外部内容不能当系统指令。
2. 高风险工具调用需要权限和确认。
3. 多模态内容需要输入输出双向审核。
4. 模型应表达不确定性。
5. 需要日志和审计。

## 10.16 最小可运行统一多模态样本 / token budget 审计 demo

下面这个 demo 不实现真实模型，而是审计一个统一多模态助手样本：输入含文本、图片、音频、视频，输出含文本和语音。脚本检查统一 token budget、输出 token budget、路由模块、loss mixture 和安全门禁。

```python
from math import ceil


def ceil_div(a, b):
    return (a + b - 1) // b


def normalize(weights):
    total = sum(weights.values())
    return {key: round(value / total, 3) for key, value in weights.items()}


def unified_multimodal_audit(sample, config):
    text_tokens = sample["text_tokens"]
    image_tokens = sum(
        ceil_div(img["height"], config["image_patch"])
        * ceil_div(img["width"], config["image_patch"])
        for img in sample["images"]
    )
    audio_tokens = sum(
        ceil(audio["seconds"] * config["audio_codec_hz"])
        * config["audio_codebooks"]
        for audio in sample["audios"]
    )
    video_tokens = sum(
        ceil_div(video["frames"], config["video_temporal_patch"])
        * ceil_div(video["height"], config["video_patch"])
        * ceil_div(video["width"], config["video_patch"])
        for video in sample["videos"]
    )
    input_tokens = {
        "text": text_tokens,
        "image": image_tokens,
        "audio": audio_tokens,
        "video": video_tokens,
        "special": config["special_tokens"],
    }
    total_input_tokens = sum(input_tokens.values())

    output_tokens = {
        "text": sample["output_text_tokens"],
        "audio": ceil(sample["output_audio_seconds"] * config["audio_codec_hz"])
        * config["audio_codebooks"],
        "image": sample["output_image_tokens"],
    }
    total_output_tokens = sum(output_tokens.values())

    modality_coverage = sum(
        1 for value in [text_tokens, image_tokens, audio_tokens, video_tokens]
        if value > 0
    )
    attention_cells = total_input_tokens * total_input_tokens
    decoder_cells = total_input_tokens * max(1, total_output_tokens)

    routed_tools = []
    if image_tokens > 0 and sample["needs_ocr"]:
        routed_tools.append("ocr")
    if audio_tokens > 0:
        routed_tools.append("asr_or_audio_encoder")
    if video_tokens > 0:
        routed_tools.append("video_sampler")
    if output_tokens["image"] > 0:
        routed_tools.append("image_generator")
    if output_tokens["audio"] > 0:
        routed_tools.append("tts_or_audio_decoder")

    loss_mix = normalize(config["loss_weights"])
    risk_flags = []
    if total_input_tokens > config["context_limit"]:
        risk_flags.append("context_over_budget")
    if sample["untrusted_media_has_instruction"]:
        risk_flags.append("untrusted_media_instruction")
    if sample["requires_voice_clone"] and not sample["speaker_authorized"]:
        risk_flags.append("voice_clone_unauthorized")
    if sample["requires_generation"] and not sample["output_safety_checked"]:
        risk_flags.append("missing_output_safety_check")

    report = {
        "input_tokens": input_tokens,
        "total_input_tokens": total_input_tokens,
        "output_tokens": output_tokens,
        "total_output_tokens": total_output_tokens,
        "modality_coverage": modality_coverage,
        "attention_cells": attention_cells,
        "decoder_cells": decoder_cells,
        "routed_tools": routed_tools,
        "loss_mix": loss_mix,
        "risk_flags": risk_flags,
    }
    checks = {
        "context_fits": total_input_tokens <= config["context_limit"],
        "output_fits": total_output_tokens <= config["output_limit"],
        "coverage_at_least_3_modalities": modality_coverage >= 3,
        "routes_expected_tools": set(routed_tools) == set(config["expected_tools"]),
        "loss_mix_sums_to_1": abs(sum(loss_mix.values()) - 1.0) <= 0.002,
        "no_risk_flags": not risk_flags,
    }
    report["checks"] = checks
    report["gate_pass"] = all(checks.values())
    return report
```

## 10.17 面试官会怎么问

### 问题一：什么是统一多模态模型？

回答模板：

```text
统一多模态模型是能用统一架构、表示或接口处理文本、图像、语音、视频等多种模态的模型或系统。目标是支持多模态理解、推理和生成，例如图文问答、语音对话、文生图、视频理解和工具操作。
```

### 问题二：统一多模态有哪些技术路线？

回答模板：

```text
可以分成接口统一、表示统一和架构统一。接口统一是多个模块组成一个助手；表示统一是把不同模态映射到共享 embedding 空间；架构统一是把文本、图像、语音、视频都 token 化，用统一 Transformer 或类似模型建模。
```

### 问题三：为什么统一 token 化很难？

回答模板：

```text
因为不同模态的信息密度和时间空间结构不同。图像、音频、视频 token 数远大于文本，序列很长，训练和推理成本高；同时 tokenizer 质量会限制生成质量，语义、细节、音色和运动信息也很难统一表示。
```

### 问题四：端到端统一模型和模块化系统怎么取舍？

回答模板：

```text
端到端模型信息流更自然，可以联合优化，但训练成本高、可控性和安全评估难。模块化系统能复用成熟 ASR、OCR、VLM、TTS 和生成模型，易调试、易上线，但有误差传播和信息损失。实际产品通常折中。
```

### 问题五：跨模态 reasoning 的难点是什么？

回答模板：

```text
难点在于模型要先正确感知各模态证据，再把证据融合起来推理。错误可能来自看错、听错、OCR 错、检索错或语言推理错。因此评估时要做错误归因，区分感知错误和推理错误。
```

## 10.19 本章总结

统一多模态模型是多模态发展的长期方向。它希望模型不仅能看、听、说、生成图像和视频，还能在多种模态之间推理、记忆、调用工具并保持安全。

需要记住：

1. 统一可以发生在接口、表示或架构层面。
2. 统一 token 化是重要路线，但成本和信息密度问题很难。
3. 中心 LLM 路线把 LLM 作为多模态推理和调度核心。
4. Any-to-Any 是理想目标，但训练和评估复杂。
5. 跨模态 reasoning 要区分感知错误和推理错误。
6. 模块化系统更容易工程落地，端到端系统更自然但更难控。
7. 安全问题会随着模态数量增加而扩大。
