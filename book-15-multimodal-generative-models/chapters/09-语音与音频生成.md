# 第九章：语音与音频生成

语音和音频是多模态模型的重要组成部分。文本模型只能处理文字，视觉模型能看图，而语音模型让 AI 能听、能说、能理解声音环境。语音方向包含 ASR、TTS、speech-to-speech、voice cloning、音乐生成、音效生成和统一音频 token 建模。随着大模型发展，语音不再只是单独的识别或合成模块，而正在和 LLM、VLM、Agent 系统融合成实时多模态交互能力。

本章目标是讲清语音与音频生成的主线：波形和声谱图、ASR 与 Whisper、TTS、voice cloning、audio codec、speech token、codec language model、speech-to-speech、音频生成评估和安全风险。

## 0. 本讲资料边界与第二轮精修口径

本讲第二轮精修前，重点核对了 Whisper、wav2vec 2.0、SoundStream、EnCodec、AudioLM、VALL-E、SpeechT5、NaturalSpeech 2、AudioPaLM 以及 WER / CER / MOS 等语音评估资料。正文只吸收适合面试和工程理解的稳定结论，不追闭源实时语音系统的未公开训练数据、模型规模、延迟优化和安全策略。

本章聚焦：

1. waveform、采样率、log-mel spectrogram、声学帧和音频 token 的基本表示。
2. ASR / Whisper 的 encoder-decoder 口径、转写指标和噪声鲁棒性边界。
3. TTS、vocoder、voice cloning、speaker prompt、说话人相似度和安全门禁。
4. neural audio codec、residual vector quantization、codec token rate、codec language model 和 speech-to-speech。
5. 级联语音助手 `VAD -> ASR -> LLM -> TTS` 与端到端 speech-to-speech 的 trade-off。
6. 可手算的音频 token / WER / codec 压缩 / 延迟审计 demo。

本章不展开完整声学模型训练、真实音频数据采集和授权治理、语音合成产品安全策略、音乐生成版权判定、声纹识别系统实现，也不提供声音冒充、未授权 voice cloning 或绕过语音安全审核的方法。

## 9.1 语音任务总览

语音和音频任务可以分为几类：

1. ASR：Automatic Speech Recognition，语音转文本。
2. TTS：Text-to-Speech，文本转语音。
3. Speech-to-speech：语音输入、语音输出。
4. Voice cloning：根据少量参考音频克隆说话人音色。
5. Speech translation：语音到另一种语言的文本或语音。
6. Audio captioning：描述一段音频内容。
7. Music generation：音乐生成。
8. Sound effect generation：音效生成。

面试回答：

```text
语音多模态主要包括 ASR、TTS、speech-to-speech 和音频生成。ASR 把语音转成文本，TTS 把文本合成语音，speech-to-speech 直接从语音输入生成语音输出。现代系统常结合 audio encoder、LLM 和 audio decoder，并使用 audio codec 或 speech tokens 统一建模。
```

## 9.2 音频信号如何表示

原始音频是 waveform，也就是一维连续信号。

```text
waveform: [num_samples]
```

采样率决定每秒多少个采样点。例如 16 kHz 表示每秒 16000 个点。

直接处理 waveform 成本较高，很多模型会先转成时频表示，例如 mel spectrogram。

```text
waveform -> spectrogram / log-mel features
```

直觉：

1. 横轴是时间。
2. 纵轴是频率。
3. 数值表示能量。

语音识别常用 log-mel spectrogram 作为输入，因为它比原始 waveform 更接近人耳感知，也更适合模型处理。

### 9.2.1 关键公式与语音 / 音频生成速查

**Waveform、采样率和时长**

单通道 waveform 可以写成：

```math
x\in\mathbb{R}^{N}
```

若采样率为 `r`，音频时长为 `D` 秒，则采样点数约为：

```math
N=rD
```

例如 `16 kHz`、`3.2` 秒音频约有 `51200` 个采样点。采样率越高，高频信息保留越多，但计算和存储成本也越高。

**Log-mel spectrogram 帧数**

ASR 常把 waveform 转为 log-mel spectrogram。设 window 长度为 `w` 秒，hop 长度为 `h` 秒，mel bins 数为 `F`，则帧数可粗略估计为：

```math
T_{\mathrm{mel}}\approx
\left\lceil
\frac{D}{h}
\right\rceil
```

log-mel 特征 shape：

```math
M\in\mathbb{R}^{F\times T_{\mathrm{mel}}}
```

面试时要说明：window 决定每帧看多长局部音频，hop 决定帧移和时间分辨率；真实实现还会受 padding、center、STFT 参数和模型预处理影响。

**ASR 条件生成目标**

ASR 可以看成给定音频特征 `M` 后生成文本 token 序列 `y_1,...,y_U`：

```math
p(y_1,\ldots,y_U\mid M)=
\prod_{u=1}^{U}
p(y_u\mid y_{1:u-1},M)
```

训练时常用 token-level negative log likelihood：

```math
L_{\mathrm{asr}}=
-
\sum_{u=1}^{U}
\log p_\theta(y_u\mid y_{1:u-1},M)
```

Whisper 类 encoder-decoder 模型可以理解为：audio encoder 编码 `M`，text decoder 自回归生成 `y`。

**WER / CER**

ASR 常用 WER 衡量词级转写错误：

```math
\mathrm{WER}=
\frac{S+D+I}{N_{\mathrm{ref}}}
```

其中 `S` 是 substitution 数，`D` 是 deletion 数，`I` 是 insertion 数，`N_ref` 是参考文本词数。CER 把词换成字符：

```math
\mathrm{CER}=
\frac{S_c+D_c+I_c}{N_{\mathrm{char}}}
```

WER 适合看英文等空格分词明显的语言；CER 对中文、短命令和拼写错误分析常更直观。

**Audio codec token 数**

neural audio codec 把连续音频压缩成离散 token。若 codec 帧率为 `q` Hz，时长为 `D` 秒，每帧有 `K` 个 codebooks，则 codec token 数约为：

```math
N_{\mathrm{codec}}=
\left\lceil Dq\right\rceil K
```

若每个 code 是 `b` bit，原始 PCM 每采样点 `b_pcm` bit，则粗略压缩比为：

```math
R_{\mathrm{codec}}=
\frac{
N_{\mathrm{codec}}b
}{
rD b_{\mathrm{pcm}}
}
```

这个比例只解释离散 token 层的码率直觉，不等价于最终文件大小，也不直接代表听感质量。

**Codec language model**

codec language model 把 audio tokens 当成语言模型序列。若文本条件为 `u`，音频 token 为 `a_1,...,a_M`：

```math
p(a_1,\ldots,a_M\mid u)=
\prod_{m=1}^{M}
p(a_m\mid u,a_{1:m-1})
```

TTS 可以看成 `text tokens -> codec tokens -> codec decoder -> waveform`。Speech-to-speech 可以进一步把输入语音 tokens、文本 tokens 和输出语音 tokens 放进统一序列或 encoder-decoder 框架中。

**级联语音助手延迟**

级联系统的端到端延迟可以先粗略写为：

```math
L_{\mathrm{cascade}}=
L_{\mathrm{vad}}+
L_{\mathrm{asr}}+
L_{\mathrm{llm}}+
L_{\mathrm{tts}}+
L_{\mathrm{network}}
```

实时语音助手的优化通常不是单独优化某个模块，而是用 streaming ASR、增量 LLM decoding、streaming TTS、barge-in 和缓存共同降低首包延迟和交互中断成本。

**语音上线门禁**

一个简化上线门禁可以写成：

```math
G_{\mathrm{speech}}=
\mathbf{1}[
\mathrm{WER}\le \tau_w
\land
L_{\mathrm{cascade}}\le \tau_l
\land
N_{\mathrm{codec}}\le B
\land
A_{\mathrm{speaker}}=1
]
```

其中 `A_speaker` 表示说话人授权或安全校验通过。真实系统还需要隐私、deepfake、声音水印、高风险语音拒绝和人工评估。

## 9.3 ASR：语音转文本

ASR 的目标是把语音转成文字。

输入：

```text
audio waveform / spectrogram
```

输出：

```text
transcript text
```

常见挑战：

1. 口音。
2. 背景噪声。
3. 多说话人。
4. 语速变化。
5. 专有名词。
6. 标点和大小写。
7. 低质量录音。

ASR 是很多语音大模型系统的第一步。例如语音助手可以先 ASR 转文本，再把文本交给 LLM。

## 9.4 Whisper 的基本结构

Whisper 是典型的 encoder-decoder 语音识别模型。

流程：

```text
audio -> log-mel spectrogram -> audio encoder -> text decoder -> transcript
```

特点：

1. 输入是音频特征。
2. encoder 编码语音。
3. decoder 自回归生成文本 token。
4. 支持多语言和翻译任务。

Whisper 的强大来自大规模弱监督训练数据。它不仅做 ASR，还能处理一定程度的语种识别、翻译和噪声鲁棒性。

面试回答：

```text
Whisper 可以理解为语音 encoder-decoder 模型。它先把音频转换成 log-mel spectrogram，用 encoder 编码声学信息，再用 decoder 自回归生成文本 token。大规模多语言弱监督数据让它具备较强鲁棒性和多语言能力。
```

## 9.5 TTS：文本转语音

TTS 的目标是把文本合成自然语音。

输入：

```text
text
```

输出：

```text
speech waveform
```

一个 TTS 系统通常要决定：

1. 读什么内容。
2. 用什么音色。
3. 语速如何。
4. 语调和情感如何。
5. 停顿在哪里。
6. 最终 waveform 如何生成。

传统 TTS 常分成 acoustic model 和 vocoder。现代大模型路线可能使用 codec tokens 和语言模型来生成语音。

## 9.6 Vocoder 是什么

Vocoder 负责把中间声学表示转换成 waveform。

例如：

```text
mel spectrogram -> vocoder -> waveform
```

常见 vocoder 路线包括：

1. WaveNet 类自回归 vocoder。
2. GAN vocoder。
3. Diffusion vocoder。
4. Neural codec decoder。

Vocoder 质量会直接影响语音自然度、噪声、音色和细节。

## 9.7 Voice Cloning

Voice cloning 是根据参考语音生成相同或相似音色的新语音。

输入：

```text
target text + reference speaker audio
```

输出：

```text
目标说话人音色的语音
```

关键能力：

1. 说话人音色保持。
2. 文本内容准确。
3. 情感和语调控制。
4. 少样本适配。

风险也很高：

1. 冒充他人。
2. 电话诈骗。
3. 未授权声音克隆。
4. 深度伪造。

因此真实产品中必须做授权、检测、水印和滥用防护。

## 9.8 Audio Codec 和 Speech Token

为了让语言模型处理语音，可以把连续音频离散化成 tokens。

Audio codec 的作用是压缩和重建音频。

流程：

```text
waveform -> codec encoder -> discrete audio tokens -> codec decoder -> waveform
```

这些离散 tokens 类似文本 token，可以被 Transformer 建模。

Speech token 的好处：

1. 让语音可以用 next-token prediction 建模。
2. 方便和文本 token 混合。
3. 支持统一多模态语言模型。
4. 可用于 TTS、speech-to-speech 和音频生成。

难点：

1. token rate 高，序列很长。
2. codec 质量影响最终音质。
3. 多码本结构增加建模复杂度。
4. 语义、音色、韵律可能纠缠。

## 9.9 Codec Language Model

Codec language model 把音频 codec tokens 当成序列来建模。

类似文本语言模型：

```text
previous audio tokens -> next audio token
```

也可以条件化于文本：

```text
text tokens -> audio tokens
```

优点：

1. 训练目标统一。
2. 可以和 LLM 架构结合。
3. 支持长音频生成。

缺点：

1. 音频 token 序列长。
2. 推理速度压力大。
3. 韵律和情感控制复杂。

## 9.10 Speech-to-Speech

Speech-to-speech 直接从语音输入生成语音输出。

有两类路线：

### 9.10.1 Cascade 路线

级联系统：

```text
speech -> ASR -> text -> LLM -> text response -> TTS -> speech
```

优点：

1. 模块清晰。
2. 可复用成熟 ASR、LLM、TTS。
3. 容易调试。

缺点：

1. 延迟较高。
2. ASR 错误会传递。
3. 语气、情绪和非语言信息丢失。

### 9.10.2 End-to-end 路线

端到端系统：

```text
speech tokens -> model -> speech tokens
```

优点：

1. 有机会保留语气、情绪和韵律。
2. 延迟可能更低。
3. 更接近自然对话。

缺点：

1. 数据和训练难度大。
2. 可控性和可解释性更弱。
3. 安全评估更复杂。

## 9.11 语音助手系统结构

一个常见语音助手可以是：

```text
microphone -> VAD -> ASR -> LLM -> TTS -> speaker
```

其中：

1. VAD 检测用户是否在说话。
2. ASR 转写语音。
3. LLM 理解和生成回复。
4. TTS 合成语音。

实时系统还要考虑：

1. 端到端延迟。
2. 打断能力。
3. 流式 ASR。
4. 流式 TTS。
5. 噪声鲁棒性。
6. 多轮对话状态。

## 9.12 音乐和音效生成

音频生成不只包括语音，还包括音乐和环境音。

音乐生成关注：

1. 旋律。
2. 和声。
3. 节奏。
4. 乐器。
5. 长程结构。
6. 风格一致性。

音效生成关注：

1. 事件声音。
2. 环境氛围。
3. 与视频动作同步。
4. 真实感。

文本到音频生成类似文生图，但输出是时间信号，对长程结构和采样率要求不同。

## 9.13 语音评估

语音评估维度包括：

1. ASR 准确率。
2. TTS 自然度。
3. 说话人相似度。
4. 情感和韵律。
5. 延迟。
6. 稳定性。
7. 噪声鲁棒性。
8. 安全性。

ASR 常用 WER，Word Error Rate。

TTS 常用：

1. MOS 人工评分。
2. 说话人相似度。
3. 内容一致性。
4. 音质指标。

语音评估很依赖人工主观体验，因为自然度、情感和舒适度很难完全自动评估。

## 9.14 语音安全

语音系统有独特安全风险：

1. 声音克隆诈骗。
2. 未授权模仿真人。
3. 伪造录音证据。
4. 语音指令注入。
5. 隐私泄露。
6. 录音数据合规。

安全措施：

1. 说话人授权。
2. 声音水印。
3. 深伪检测。
4. 高风险语音拒绝。
5. 敏感信息脱敏。
6. 录音加密和权限控制。

## 9.15 和 LLM 的结合方式

语音和 LLM 可以有几种结合方式：

1. ASR -> LLM -> TTS。
2. Audio encoder -> LLM -> text。
3. Text + speech tokens 混合建模。
4. Speech-to-speech 端到端模型。
5. 统一多模态模型同时处理文本、图像和语音。

当前工程中 cascade 路线最清晰可靠，端到端路线更自然但训练和评估难度更高。

## 9.16 最小可运行音频 token / WER / codec 审计 demo

下面这个 demo 不做真实 ASR 或 TTS，而是审计语音系统中最常被面试追问的数字：采样点数、log-mel 帧数、WER / CER、codec token 数、压缩比、级联延迟、TTS 输出 token 数和 voice cloning 授权门禁。

```python
from math import ceil


def edit_distance(a, b):
    dp = [[0] * (len(b) + 1) for _ in range(len(a) + 1)]
    for i in range(len(a) + 1):
        dp[i][0] = i
    for j in range(len(b) + 1):
        dp[0][j] = j
    for i in range(1, len(a) + 1):
        for j in range(1, len(b) + 1):
            cost = 0 if a[i - 1] == b[j - 1] else 1
            dp[i][j] = min(
                dp[i - 1][j] + 1,
                dp[i][j - 1] + 1,
                dp[i - 1][j - 1] + cost,
            )
    return dp[-1][-1]


def word_error_rate(reference, hypothesis):
    ref_words = reference.split()
    hyp_words = hypothesis.split()
    return edit_distance(ref_words, hyp_words) / max(1, len(ref_words))


def char_error_rate(reference, hypothesis):
    ref_chars = list(reference.replace(" ", ""))
    hyp_chars = list(hypothesis.replace(" ", ""))
    return edit_distance(ref_chars, hyp_chars) / max(1, len(ref_chars))


def audio_audit(config):
    samples = int(config["sample_rate"] * config["duration_s"])
    window_samples = int(config["sample_rate"] * config["window_ms"] / 1000)
    hop_samples = int(config["sample_rate"] * config["hop_ms"] / 1000)
    log_mel_frames = ceil(config["duration_s"] * 1000 / config["hop_ms"])
    mel_shape = (config["n_mels"], log_mel_frames)

    codec_frames = ceil(config["duration_s"] * config["codec_hz"])
    codec_tokens = codec_frames * config["codebooks"]
    raw_bits = samples * config["pcm_bits"]
    codec_bits = codec_tokens * config["bits_per_code"]
    compression_ratio = codec_bits / raw_bits

    wer = word_error_rate(config["reference"], config["hypothesis"])
    cer = char_error_rate(config["reference"], config["hypothesis"])
    cascade_latency_ms = sum(config["latency_ms"].values())

    tts_tokens = ceil(config["target_speech_s"] * config["codec_hz"]) * config["codebooks"]
    codec_lm_context = (
        config["text_tokens"]
        + config["speaker_prompt_tokens"]
        + tts_tokens
    )

    report = {
        "samples": samples,
        "window_samples": window_samples,
        "hop_samples": hop_samples,
        "mel_shape": mel_shape,
        "codec_frames": codec_frames,
        "codec_tokens": codec_tokens,
        "raw_kbits": round(raw_bits / 1000, 2),
        "codec_kbits": round(codec_bits / 1000, 2),
        "compression_ratio": round(compression_ratio, 6),
        "wer": round(wer, 3),
        "cer": round(cer, 3),
        "cascade_latency_ms": cascade_latency_ms,
        "tts_tokens": tts_tokens,
        "codec_lm_context": codec_lm_context,
    }
    checks = {
        "mel_frames_expected": mel_shape == (80, 320),
        "codec_tokens_expected": codec_tokens == 640,
        "compression_under_1pct": compression_ratio < 0.01,
        "wer_under_30pct": wer < 0.30,
        "cer_under_20pct": cer < 0.20,
        "latency_under_700ms": cascade_latency_ms < 700,
        "context_fits": codec_lm_context <= config["context_limit"],
        "voice_clone_authorized": config["speaker_authorized"],
    }
    report["checks"] = checks
    report["gate_pass"] = all(checks.values())
    return report


config = {
    "sample_rate": 16000,
    "duration_s": 3.2,
    "window_ms": 25,
    "hop_ms": 10,
    "n_mels": 80,
    "codec_hz": 50,
    "codebooks": 4,
    "bits_per_code": 10,
    "pcm_bits": 16,
    "reference": "turn on the hallway lights",
    "hypothesis": "turn on the hallway light",
    "latency_ms": {"vad": 30, "asr": 180, "llm": 220, "tts": 190},
    "target_speech_s": 2.4,
    "text_tokens": 18,
    "speaker_prompt_tokens": 120,
    "context_limit": 1024,
    "speaker_authorized": True,
}

report = audio_audit(config)
for key, value in report.items():
    print(f"{key}={value}")
```

这个 demo 对面试很有用：

1. `3.2` 秒、`16 kHz` 单通道音频有 `51200` 个采样点。
2. `25 ms` window 和 `10 ms` hop 对应 `400` / `160` 个采样点，`80 x 320` 的 log-mel shape 可以手算。
3. WER / CER 来自编辑距离，能明确解释 substitution、deletion、insertion 的含义。
4. `50 Hz` codec、`4` 个 codebooks 会把 `3.2` 秒音频变成 `640` 个 codec tokens。
5. 语音助手上线不能只看识别准确率，还要同时看 latency、context、codec token 和说话人授权。

## 9.17 面试官会怎么问

### 问题一：ASR、TTS 和 speech-to-speech 有什么区别？

回答模板：

```text
ASR 是语音转文本，TTS 是文本转语音，speech-to-speech 是语音输入直接生成语音输出。工程上 speech-to-speech 可以用 ASR -> LLM -> TTS 的级联系统，也可以用端到端语音 token 模型。
```

### 问题二：Whisper 的基本结构是什么？

回答模板：

```text
Whisper 是 encoder-decoder 语音模型。输入音频先转换成 log-mel spectrogram，由 audio encoder 编码声学信息，再由 text decoder 自回归生成文本 token。它通过大规模多语言弱监督数据获得较强鲁棒性。
```

### 问题三：Audio codec 和 speech token 的作用是什么？

回答模板：

```text
Audio codec 把连续 waveform 编码成离散 audio tokens，并能解码回音频。Speech token 让语音可以像文本 token 一样被 Transformer 建模，从而支持 TTS、speech-to-speech 和统一多模态语言模型。
```

### 问题四：级联语音助手和端到端语音模型有什么区别？

回答模板：

```text
级联系统把语音助手拆成 ASR、LLM、TTS，模块清晰、容易调试，但 ASR 错误会传递，且语气情绪可能丢失。端到端模型直接从语音到语音，有机会更自然和低延迟，但数据、训练、可控性和安全评估更难。
```

### 问题五：Voice cloning 有哪些风险？

回答模板：

```text
Voice cloning 可能被用于冒充他人、诈骗、伪造录音和侵犯声音权益。因此需要说话人授权、声音水印、深伪检测、高风险请求拒绝和数据权限管理。
```

## 9.19 本章总结

语音与音频模型让 AI 具备听和说的能力。ASR 把语音转文本，TTS 把文本转语音，speech-to-speech 进一步把语音交互端到端化。Audio codec 和 speech token 让连续音频可以离散化并被 Transformer 建模，是统一语音语言模型的重要基础。

需要记住：

1. 原始音频是 waveform，常转成 spectrogram 或 audio tokens。
2. Whisper 是典型 ASR encoder-decoder 模型。
3. TTS 需要解决内容、音色、韵律和 waveform 生成。
4. Voice cloning 强大但安全风险高。
5. Audio codec 把音频离散化为 speech tokens。
6. 语音助手可以走 ASR -> LLM -> TTS，也可以走端到端 speech-to-speech。
7. 语音评估要覆盖准确率、自然度、相似度、延迟和安全。
