# 第一部分：深度学习与概率基础进阶

## 第 1 讲：概率建模视角下的语言模型

### 本讲目标

第一册里我们已经知道，LLM 训练的基础目标通常是 next-token prediction。本讲进一步从概率建模角度看语言模型：它到底是在拟合什么分布？为什么自回归分解成立？为什么语言建模可以统一很多任务？

你需要掌握：

1. 语言模型的概率定义。
2. 联合概率如何分解成条件概率。
3. 为什么 next-token prediction 是自监督学习。
4. 为什么语言模型既能生成，也能做理解任务。
5. 训练目标和生成行为之间的关系。
6. 面试中如何用概率语言描述 LLM。

### 语言模型是什么

从概率角度，语言模型的任务是给 token 序列分配概率。

设一个 token 序列为：

```text
x_1, x_2, ..., x_T
```

语言模型希望建模：

```math
P(x_1,x_2,\ldots,x_T)
```

对自回归模型，通常通过链式法则分解为：

```math
P(x_1,x_2,\ldots,x_T)=\prod_{t=1}^{T}P(x_t\mid x_{1:t-1})
```

这就是 GPT 类模型的根本概率形式。

### 为什么可以分解

链式法则本身是概率论的基本恒等式，不需要额外假设。

例如：

```math
P(a,b,c)=P(a)P(b\mid a)P(c\mid a,b)
```

对文本序列同理：

```math
P(x_1,\ldots,x_T)=P(x_1)P(x_2\mid x_1)\cdots P(x_T\mid x_{1:T-1})
```

语言模型做的事情，就是学习每一个条件分布 `P(x_t | x_{1:t-1})`。

### 自回归建模的意义

自回归建模有几个关键优点：

1. 目标统一。
2. 训练数据天然可得。
3. 能直接用于生成。
4. 可以用同一框架建模知识、语法、风格和推理线索。

它的核心好处是：

```text
训练目标和生成方式一致
```

模型训练时学习预测下一个 token，推理时也正是一步一步生成下一个 token。

### 为什么 next-token prediction 能学到能力

文本不是随机噪声，而是包含人类知识、逻辑结构、语言规律和任务模式。

要预测下一个 token，模型往往必须学会：

1. 语法。
2. 语义。
3. 常识。
4. 世界知识。
5. 文体。
6. 任务结构。
7. 某些推理模式。

所以 next-token prediction 虽然形式简单，但在大规模数据和大模型容量下，会逼近很多复杂能力。

### 语言模型不是“理解”还是“背诵”的二选一

常见误区是把 LLM 看成“背答案的机器”或“真正理解语言的系统”二选一。

更准确的说法是：

1. 它会记忆部分训练内容。
2. 它也会学习可泛化模式。
3. 记忆和泛化同时存在。

从概率角度看，模型学的是训练分布上的条件概率近似。

这意味着：

1. 训练分布内表现通常更好。
2. 分布外泛化依赖模型能力和数据覆盖。
3. 如果训练数据重复过多，模型会更偏向记忆。

### 为什么语言模型可以做很多任务

很多任务都可以转写成“给定上下文，预测合适的后续文本”。

例如：

1. 问答。
2. 翻译。
3. 摘要。
4. 代码补全。
5. 数学步骤生成。

统一到同一个概率建模框架后，模型可以用同一种机制处理这些任务。

这就是为什么 LLM 具有很强的通用性。

### 条件概率的视角

对给定上下文 `c`，语言模型学习的是：

```text
P(y | c)
```

这里的 `c` 可以是：

1. 过去 token。
2. 用户指令。
3. 对话历史。
4. 检索结果。
5. 工具返回结果。

因此，很多看似不同的任务，本质上都是条件生成问题。

### 训练目标和评估目标的差异

训练目标通常是 token-level cross entropy。

但评估时我们关心的可能是：

1. 正确性。
2. 鲁棒性。
3. 安全性。
4. 用户偏好。
5. 长上下文理解。

所以 `loss` 降低不等于所有任务能力都提升。

### 概率视角下的 hallucination

幻觉从概率角度看，往往是模型在上下文下给某些不真实 token 分配了较高概率。

这可能来自：

1. 训练数据缺失。
2. 训练数据错误。
3. 上下文信息不足。
4. 解码随机性。
5. 模型缺少外部验证。

也就是说，幻觉不是一个单独的 bug，而是概率建模目标与真实事实验证之间的鸿沟。

### 面试中怎么讲

如果面试官问“LLM 在概率意义上是什么”，你可以这样答：

```text
LLM 本质上是在建模 token 序列的联合概率，通常用链式法则分解为逐 token 的条件概率 P(x_t | x_{1:t-1})。训练时通过 next-token prediction 最大化训练语料的似然，推理时则按这个条件分布逐步采样或解码生成。它既能学到语言规律，也能学到部分知识和任务模式。
```

### 最小 demo：用条件概率计算序列概率

下面这个 demo 用三个条件概率模拟自回归分解。真实 LLM 会在每个位置输出整个词表的概率分布，这里只保留真实下一个 token 的概率，方便理解序列概率和 negative log likelihood。

```python
import math


token_probs = [0.5, 0.2, 0.1]

sequence_prob = 1.0
negative_log_likelihood = 0.0

for prob in token_probs:
    sequence_prob *= prob
    negative_log_likelihood += -math.log(prob)

print("sequence probability:", sequence_prob)
print("negative log likelihood:", round(negative_log_likelihood, 4))
```

这段代码说明两件事：序列概率是逐 token 条件概率的乘积；训练时常优化 log probability，是因为乘积会很小，而 log 后可以变成求和。

### 真实项目中的坑

#### 坑 1：把语言模型只讲成分类器

语言模型不是简单分类问题，而是序列概率建模。

#### 坑 2：忽略条件概率视角

很多任务本质上是 `P(y|c)`，不是无条件生成。

#### 坑 3：把记忆和泛化对立起来

两者通常同时存在，需要通过污染检测和分布外评估去区分。

#### 坑 4：只讲训练，不讲生成

语言模型的价值在于可生成，因此训练目标和 decoding 都要一起理解。

### 面试官会怎么问

#### 问题 1：语言模型在数学上到底在学什么？

回答框架：

1. 学 token 序列的联合概率。
2. 用链式法则分解成逐 token 条件概率。
3. 用 next-token prediction 近似训练这个条件分布。

#### 问题 2：为什么 next-token prediction 能产生通用能力？

回答框架：

1. 文本包含语言、知识和任务模式。
2. 预测下一个 token 需要利用上下文中的深层结构。
3. 规模足够大时会涌现出可迁移能力。

#### 问题 3：为什么说语言模型既会记忆也会泛化？

回答框架：

1. 数据中重复和高频模式会被记住。
2. 结构化规律会被泛化。
3. 需要通过污染检测和分布外测试区分两者。

### 常见误区

1. 误区：语言模型只是在预测字符。
   纠正：现代 LLM 建模的是高维 token 分布，包含语义和任务结构。

2. 误区：概率模型就不可能有推理能力。
   纠正：推理能力可以作为复杂条件分布中的模式被学习和组合。

3. 误区：模型输出看起来像理解，就一定真的理解。
   纠正：需要通过泛化、扰动和验证来判断。

### 小练习

1. 写出 `P(x_1, x_2, x_3, x_4)` 的链式分解。
2. 用自己的话解释为什么 `P(y|c)` 是很多任务的统一形式。
3. 举一个“预测下一个 token 需要知识”的例子。
4. 解释为什么语言模型会同时记忆和泛化。

### 本讲总结

本讲最重要的结论：

1. 语言模型本质上是在建模 token 序列的联合概率。
2. 自回归模型通过链式法则分解联合概率。
3. next-token prediction 是一个统一、可扩展、自监督的训练目标。
4. 很多任务都可以转化为条件生成问题。
5. 幻觉可以理解为概率建模与事实验证之间的不一致。

## 第 2 讲：交叉熵、KL 散度与信息论统一视角

### 本讲目标

第 1 讲讲的是语言模型的概率形式。本讲进一步问：为什么训练里最常见的是交叉熵？它和 KL 散度、最大似然、信息论有什么关系？

你需要掌握：

1. 交叉熵的定义。
2. KL 散度的定义和意义。
3. 最大似然和交叉熵为何等价。
4. 为什么最小化交叉熵等价于逼近真实分布。
5. 信息论视角如何帮助理解语言建模。
6. 面试中如何把这些概念讲得清楚。

### 交叉熵是什么

对于真实分布 `p` 和模型分布 `q`，交叉熵定义为：

```math
H(p,q)=-\mathbb{E}_{x\sim p}\left[\log q(x)\right]
```

直觉上，它衡量的是：

```text
如果真实数据来自 p，而你用 q 来编码/预测，会有多大代价
```

在语言模型中，交叉熵就是平均 token 负对数概率。

### KL 散度是什么

KL 散度定义为：

```math
D_{\mathrm{KL}}(p\Vert q)=\mathbb{E}_{x\sim p}\left[\log p(x)-\log q(x)\right]
```

也可以写成：

```math
D_{\mathrm{KL}}(p\Vert q)=H(p,q)-H(p)
```

因为 `H(p)` 与模型无关，所以在固定真实分布时，最小化交叉熵等价于最小化 KL 散度。

### 为什么训练用交叉熵

训练文本时，我们希望模型分布 `q` 接近真实数据分布 `p`。

如果最小化：

```math
-\log q(x)
```

在数据集上求平均，就等价于最大化训练数据的似然，也等价于最小化经验交叉熵。

这是语言模型训练最自然的目标。

### 最大似然和交叉熵的关系

给定训练集 `D = {x_i}`，最大似然是：

```math
\max \prod_i q(x_i)
```

取对数后：

```math
\max \sum_i \log q(x_i)
```

等价于最小化：

```math
-\sum_i \log q(x_i)
```

这就是经验交叉熵。

所以：

```text
最大似然 = 最小化负对数似然 = 最小化交叉熵
```

### 信息论直觉

信息论里，`-log p(x)` 可以理解为信息量。

概率越低的事件，信息量越高。

因此，模型如果对真实 token 赋予很低概率，就会带来很高 loss。

语言模型训练就是让模型对真实文本中的 token 赋予更高概率，从而减少平均信息代价。

### 为什么交叉熵和 perplexity 相关

如果用自然对数，

```math
\mathrm{PPL}=\exp\left(H(p,q)\right)
```

所以 perplexity 可以看成平均“困惑程度”的指数化表达。

loss 越低，perplexity 越低。

### 为什么最小化交叉熵不等于完美理解

交叉熵只关心 token 概率，不直接关心：

1. 事实是否正确。
2. 逻辑是否严谨。
3. 回答是否安全。
4. 是否符合用户偏好。

因此，一个模型可能交叉熵更低，但在某些真实任务上表现不一定更好。

### 多类分类和语言建模的统一

从本质上看，语言模型的每一步预测都是一个大规模多类分类问题。

类别数是词表大小。

真实标签是下一个 token 的 id。

模型输出 logits 后接 softmax，再用交叉熵训练。

这也是为什么很多深度学习基础概念可以直接迁移到 LLM 训练。

### 最小 demo：数值稳定 softmax 和交叉熵

下面这个 demo 展示为什么实际实现 softmax 前要先减去最大 logit。这样不改变概率分布，但能避免 `exp` 溢出。

```python
import math


def stable_softmax(logits):
    max_logit = max(logits)
    exp_values = [math.exp(x - max_logit) for x in logits]
    total = sum(exp_values)
    return [x / total for x in exp_values]


logits = [1000.0, 1001.0, 999.0]
target = 1

probs = stable_softmax(logits)
loss = -math.log(probs[target])

print("probs:", [round(x, 4) for x in probs])
print("cross entropy:", round(loss, 4))
```

真实框架中的 `cross_entropy` 通常会把 `log_softmax` 和 NLL loss 融合实现，既快又更数值稳定。

### 为什么 KL 视角重要

KL 视角帮助我们理解很多后训练和对齐方法。

例如：

1. RLHF 中的 KL penalty。
2. DPO 中的隐式偏好优化。
3. 蒸馏中的分布匹配。

这些方法本质上都在控制一个模型分布向另一个目标分布靠近。

### 熵和不确定性

真实分布的熵 `H(p)` 表示数据本身的不确定性。

如果数据本身很随机，任何模型都很难把 loss 降到很低。

这说明：

1. 有些任务天然难。
2. 评估时要考虑任务本身的不可预测性。
3. loss 的下界与数据分布有关。

### 信息论视角下的模型压缩

如果模型能用更少比特描述数据，就意味着它学到了更有效的分布表示。

这和压缩、编码、蒸馏等问题都有联系。

从这个角度看，语言模型训练可以被理解成一种非常强的统计压缩过程。

### 面试中怎么讲

如果被问到“为什么大模型训练用交叉熵”，可以这样说：

```text
因为语言模型要拟合真实数据分布，而交叉熵等价于负对数似然的经验平均，最小化交叉熵就等价于最大化训练数据的似然，也等价于最小化真实分布和模型分布之间的 KL 散度。它既有概率解释，也有信息论解释。
```

### 真实项目中的坑

#### 坑 1：只会背公式，不知道意义

交叉熵、KL 和 NLL 的关系必须能用自己的话说清。

#### 坑 2：把交叉熵当成万能指标

它只是 token-level 训练目标，不代表真实任务全部质量。

#### 坑 3：忽略真实分布熵

某些任务本身不确定性很高，loss 降不下去不一定是模型差。

### 面试官会怎么问

#### 问题 1：交叉熵和 KL 散度有什么关系？

回答框架：

1. `KL(p||q) = H(p,q) - H(p)`。
2. `H(p)` 与模型无关。
3. 所以最小化交叉熵等价于最小化 KL。

#### 问题 2：为什么语言模型训练用最大似然？

回答框架：

1. 文本是观测样本。
2. 最大似然鼓励模型提高真实样本概率。
3. 与自回归 next-token 目标天然一致。

#### 问题 3：perplexity 和 cross entropy 什么关系？

回答框架：

1. `PPL = exp(loss)`（自然对数下）。
2. 它是 cross entropy 的指数形式。
3. 越低表示模型越不“困惑”。

### 常见误区

1. 误区：交叉熵就是准确率。
   纠正：交叉熵是概率分布距离，不是 0/1 准确率。

2. 误区：KL 对称。
   纠正：`KL(p||q) != KL(q||p)`。

3. 误区：loss 低就说明模型懂了。
   纠正：loss 只说明 token 预测更好，不直接保证高层能力。

### 小练习

1. 写出交叉熵和 KL 的公式关系。
2. 用自己的话解释最大似然和交叉熵的等价关系。
3. 解释为什么 perplexity 和 loss 是单调对应的。
4. 举例说明为什么 loss 低不一定代表模型更强。

### 本讲总结

本讲最重要的结论：

1. 语言模型训练本质上是在最小化交叉熵。
2. 交叉熵与最大似然、负对数似然、KL 散度是统一的。
3. 信息论可以帮助我们理解 loss、perplexity 和不确定性。
4. 训练目标是概率匹配，不是直接事实验证。
5. 交叉熵是重要但不充分的模型评价标准。

## 第 3 讲：优化景观与深度网络训练动态

### 本讲目标

深度学习不只是“定义损失函数然后梯度下降”。优化景观、鞍点、平坦极小值、训练动态等因素都会影响大模型训练。

你需要掌握：

1. 什么是优化景观。
2. 为什么深层网络优化不像凸优化那样简单。
3. 什么是鞍点和平坦极小值。
4. 为什么训练动态会受初始化和学习率影响。
5. 面试中如何讲训练不稳定的本质。

### 什么是优化景观

优化景观指的是：

```text
模型参数空间中，loss 如何随参数变化而变化
```

如果把参数记为 `theta`，训练目标可以写成：

```text
minimize L(theta)
```

其中 `theta` 是所有模型参数，`L(theta)` 是训练 loss。

对线性回归这类简单问题，loss 可能是凸的，像一个碗。

但深度神经网络的 loss 通常是高度非凸的，里面包含：

1. 很多局部极小值。
2. 很多鞍点。
3. 很多平坦区域。
4. 很多陡峭区域。
5. 参数之间复杂的耦合关系。

这就是为什么深度学习训练不是“有梯度就一定好训”。

### 凸优化和非凸优化的差异

凸优化中，如果目标函数是凸的，局部最优通常也是全局最优。

但深度网络不是这样。

深度网络的非凸性来自多个方面：

1. 多层非线性组合。
2. 参数矩阵之间的乘法耦合。
3. Attention、归一化、残差结构带来的复杂交互。
4. 权重共享和高维参数空间。

这意味着：

```text
同一个模型结构，不同初始化、不同学习率、不同 batch 顺序，都可能走出不同训练轨迹
```

但一个重要事实是：非凸不等于不可训练。

现代深度网络虽然非凸，但在过参数化、残差连接、归一化、合适初始化和优化器帮助下，往往可以训练到非常低的 loss。

### 局部极小值、鞍点和平坦区域

局部极小值指的是某个点附近的 loss 都比它高。

鞍点指的是在某些方向像极小值，在另一些方向像极大值。

直观地说，鞍点像一座山脊上的马鞍：

```text
一个方向往上，一个方向往下
```

在高维空间中，鞍点比坏的局部极小值更常见。

原因是参数维度极高，想要所有方向都向上并不容易；更常见的是有些方向向上、有些方向向下。

训练卡住时，不一定是卡在局部最优，也可能是：

1. 梯度很小的鞍点附近。
2. 平坦区域。
3. 学习率太小导致移动缓慢。
4. 梯度噪声不足或过大。

### Hessian 和曲率直觉

如果一阶梯度告诉我们“往哪个方向下降”，那么二阶信息告诉我们“这个方向有多弯”。

Hessian 矩阵是 loss 对参数的二阶导数：

```text
H = ∇² L(theta)
```

Hessian 的特征值可以帮助理解局部形状：

1. 特征值全为正：局部像碗，可能是局部极小值。
2. 有正有负：鞍点。
3. 很多接近 0：平坦区域。
4. 很大正特征值：某些方向非常陡。

大模型里不可能直接完整计算 Hessian，因为参数太多。

但曲率直觉仍然很重要。

例如，学习率太大时，模型可能在陡峭方向震荡；学习率太小时，在平坦方向进展缓慢。

### 平坦极小值和尖锐极小值

训练到低 loss 只是第一步，更重要的是模型是否泛化。

一个常见观点是：平坦极小值往往比尖锐极小值泛化更好。

平坦极小值的含义是：

```text
参数发生小扰动后，loss 仍然不会明显变差
```

尖锐极小值的含义是：

```text
参数稍微变化，loss 就快速上升
```

为什么平坦性可能和泛化有关？

直觉是：如果一个解对参数扰动不敏感，它可能不是依赖训练集中的偶然细节，而是学到了更稳定的模式。

但面试中要注意，不要把“平坦极小值一定泛化好”说成绝对定理。

更稳妥的说法是：

```text
平坦性常被用来理解泛化和鲁棒性，但它受参数化方式、尺度变换和测量方式影响，需要谨慎解释。
```

### 过参数化为什么有帮助

大模型参数很多，看起来更容易过拟合，但过参数化也会让优化变容易。

原因包括：

1. 高维参数空间中存在更多低 loss 路径。
2. 模型更容易找到能拟合数据的解。
3. 梯度下降在过参数化网络中常能避开很多坏的局部结构。
4. 多个不同参数配置可能表示相似函数。

这也是现代深度学习和传统统计直觉差异很大的地方。

传统观点认为模型越大越容易过拟合。

但在深度学习中，模型大小、数据规模、正则化、训练动态和隐式偏置共同决定泛化。

### 梯度下降不只是优化器，也是隐式正则化

梯度下降表面上是在最小化 loss，但它还会对最终解产生偏置。

这种偏置叫隐式正则化。

例如，不同优化器可能都把训练 loss 降低到很低，但最终泛化不同。

原因是它们走过的路径不同，偏好的解也不同。

影响隐式偏置的因素包括：

1. 初始化。
2. 学习率。
3. batch size。
4. 优化器类型。
5. weight decay。
6. 梯度裁剪。
7. 训练时长。

所以面试中讨论优化时，不要只说“Adam 比 SGD 快”。更好的回答是：优化器会影响训练速度、稳定性、最终解的性质和泛化表现。

### 初始化为什么重要

深层网络中，如果初始化不好，信号在前向传播或反向传播中会爆炸或消失。

初始化的目标是让每层激活和梯度的尺度保持在合理范围。

如果权重太大：

1. 激活可能爆炸。
2. softmax 可能过于饱和。
3. 梯度可能不稳定。

如果权重太小：

1. 信号可能逐层衰减。
2. 梯度很小。
3. 学习变慢。

在 Transformer 里，初始化还会和残差连接、LayerNorm、attention logits 的尺度相互影响。

这就是为什么大模型训练通常不会随便初始化，而会使用经过验证的初始化策略。

### 学习率决定训练轨迹

学习率是最重要的训练超参数之一。

学习率太大时，可能出现：

1. loss 震荡。
2. loss spike。
3. 梯度爆炸。
4. 训练直接发散。

学习率太小时，可能出现：

1. 收敛很慢。
2. 卡在平坦区域。
3. 算力利用效率低。

大模型训练常用 warmup 加 decay。

warmup 的作用是让训练初期不要过早使用大步长，因为此时参数和优化器状态还不稳定。

decay 的作用是在后期降低步长，让模型更细致地收敛。

常见学习率策略包括：

1. linear warmup。
2. cosine decay。
3. constant with decay。
4. step decay。

面试中如果被问“为什么需要 warmup”，不要只说经验有效。可以说：训练初期梯度尺度和优化器动量估计还不稳定，warmup 可以减少参数更新过大导致的发散风险。

### 最小 demo：学习率如何影响优化轨迹

下面这个 demo 在一维二次函数 `loss = (w - 3)^2` 上比较不同学习率。学习率太小收敛慢，合适学习率稳定靠近最优点，过大学习率会震荡甚至发散。

```python
def run_gradient_descent(lr, steps=8):
    w = 0.0
    trajectory = []
    for _ in range(steps):
        loss = (w - 3) ** 2
        grad = 2 * (w - 3)
        trajectory.append((round(w, 3), round(loss, 3)))
        w = w - lr * grad
    return trajectory


for lr in [0.05, 0.3, 1.1]:
    print("lr =", lr)
    print(run_gradient_descent(lr))
```

这个例子很简单，但能帮助你把“learning rate 影响训练稳定性”讲清楚：它不是只影响速度，也会改变训练轨迹和是否发散。

### Batch size 和梯度噪声

mini-batch 梯度不是完整数据梯度，而是随机估计。

这会带来梯度噪声。

梯度噪声有两面性：

1. 噪声太大，训练不稳定。
2. 噪声适中，可能帮助模型逃离鞍点或尖锐区域。
3. 噪声太小，训练更确定，但可能泛化变差或算力效率受限。

batch size 增大后，梯度估计更稳定，但并不总是线性提升效果。

大 batch 训练通常需要配合：

1. 学习率缩放。
2. warmup。
3. 更谨慎的正则化。
4. 梯度裁剪。

在大模型训练中，global batch size 还会影响 token throughput、收敛速度和最终 loss。

### Adam、SGD 和自适应优化

SGD 使用梯度方向更新参数。

Adam 会维护梯度的一阶矩和二阶矩估计，对不同参数使用自适应步长。

Adam 在大模型训练中非常常见，因为它对不同参数尺度更鲁棒，训练速度也更好。

AdamW 则把 weight decay 从 Adam 的梯度更新中解耦出来，通常比直接 L2 正则更适合现代深度网络。

面试中可以这样概括：

```text
Adam 类优化器通过自适应学习率改善不同参数尺度下的训练稳定性，AdamW 进一步使用解耦 weight decay，使正则化行为更清晰，因此在 Transformer 和 LLM 训练中非常常见。
```

但也要知道，优化器选择不是孤立的。

它和学习率、batch size、weight decay、scheduler、模型结构一起决定训练效果。

### 归一化和残差连接如何改变训练动态

深层网络难训，一个核心问题是信号和梯度跨层传播困难。

残差连接让网络可以学习增量变化：

```text
x_{l+1} = x_l + F(x_l)
```

这使梯度更容易跨层传播，也让深层模型更容易优化。

LayerNorm 则帮助稳定每层激活尺度。

在 Transformer 中，Pre-LN 通常比 Post-LN 更容易训练很深的模型。

原因是 Pre-LN 结构中，归一化出现在子层之前，残差路径更直接，梯度传播更稳定。

这也是为什么很多现代 LLM 使用 Pre-LN 或 RMSNorm 等结构。

### 训练不稳定的常见表现

真实大模型训练中，优化问题经常表现为具体症状。

常见症状包括：

1. loss 突然 spike。
2. loss 长时间 plateau。
3. 梯度 norm 变得异常大。
4. 参数 norm 或激活 norm 异常。
5. 出现 NaN 或 Inf。
6. validation loss 和 training loss 分化明显。
7. 某些 rank 或 worker loss 异常。

排查时不能只盯最终 loss。

更好的监控包括：

1. learning rate。
2. gradient norm。
3. parameter norm。
4. activation statistics。
5. loss scale。
6. token throughput。
7. data batch 异常。
8. train/validation loss gap。

### 如果训练发散，应该怎么排查

一个实用排查顺序是：

1. 检查数据 batch 是否异常。
2. 检查学习率是否过大。
3. 检查 warmup 是否太短。
4. 检查梯度裁剪是否生效。
5. 检查 mixed precision 是否导致 overflow。
6. 检查初始化和归一化实现。
7. 检查 loss mask 是否正确。
8. 检查分布式同步是否正确。

在面试中，能给出这种排查顺序，会比只讲“调小学习率”更像真实做过训练的人。

### 优化和泛化的关系

优化关注的是训练 loss 是否下降。

泛化关注的是模型在未见数据上的表现。

两者相关，但不是同一件事。

可能出现的情况：

1. 训练 loss 降低，验证 loss 也降低：理想情况。
2. 训练 loss 降低，验证 loss 上升：过拟合或分布问题。
3. 训练 loss 不降：优化失败或数据/实现问题。
4. 验证 loss 降低，但任务指标不提升：训练目标和任务指标不一致。

大模型中还要注意：validation loss 不一定完全预测下游能力。

原因是下游能力可能受 instruction tuning、数据质量、评估集设计、解码方式和上下文格式影响。

### 大模型训练里的优化直觉

训练 LLM 时，优化问题不只是数学问题，也是系统问题。

例如：

1. batch size 受显存和并行策略限制。
2. learning rate 受模型规模和 batch size 影响。
3. mixed precision 会影响数值稳定性。
4. 分布式训练会引入通信和同步问题。
5. 数据质量会直接影响 loss 曲线。

因此，一个成熟回答应该同时包含：

1. 优化理论直觉。
2. 训练工程症状。
3. 可执行排查手段。

### 面试中怎么讲

如果被问到“为什么深度网络这么非凸还能训练成功”，可以这样回答：

```text
深度网络的目标函数确实高度非凸，但现代网络通常是过参数化的，高维空间里存在很多低 loss 解；同时残差连接、归一化、合理初始化、AdamW、学习率 warmup 和 decay 等技术改善了梯度传播和训练稳定性。因此实际训练中更常见的问题不只是局部最优，而是鞍点、曲率不稳定、数值发散、学习率和 batch size 配置不当等。
```

如果被问到“训练不稳定怎么排查”，可以这样回答：

```text
我会先看 loss spike、gradient norm、parameter norm、activation statistics 和是否出现 NaN/Inf。排查顺序上先检查数据和 mask，再检查学习率、warmup、梯度裁剪、mixed precision overflow、初始化和归一化实现，最后再看分布式同步和 optimizer state。这样能区分是优化超参问题、数值稳定性问题，还是数据和实现 bug。
```

### 真实项目中的坑

#### 坑 1：只看 loss，不看训练状态

loss 是结果指标，不是全部诊断信息。gradient norm、activation norm、loss scale 和数据状态同样重要。

#### 坑 2：把训练发散都归因于学习率

学习率很常见，但不是唯一原因。数据异常、mask 错误、mixed precision overflow、分布式同步错误都可能导致发散。

#### 坑 3：忽略 warmup

大模型训练初期非常脆弱，warmup 太短可能直接导致 loss spike 或发散。

#### 坑 4：把优化和泛化混为一谈

训练 loss 低不等于泛化好，也不等于指令跟随、事实性、安全性都好。

#### 坑 5：只背“非凸优化很难”

面试中更重要的是解释为什么现代网络虽然非凸但仍能训练，以及训练失败时如何定位。

### 面试官会怎么问

#### 问题 1：为什么深度网络是非凸优化问题？

回答框架：

1. 多层非线性组合导致目标函数非凸。
2. 参数矩阵之间存在复杂耦合。
3. Transformer 中 attention、MLP、归一化和残差进一步增加复杂性。

#### 问题 2：局部极小值和鞍点有什么区别？

回答框架：

1. 局部极小值附近所有方向 loss 都不更低。
2. 鞍点在某些方向上升、某些方向下降。
3. 高维网络中鞍点和平坦区域经常比坏局部极小值更值得关注。

#### 问题 3：为什么 warmup 对大模型训练重要？

回答框架：

1. 训练初期参数和优化器状态还不稳定。
2. 直接使用大学习率可能导致更新过大。
3. warmup 可以逐步增加步长，降低早期发散风险。

#### 问题 4：batch size 如何影响训练？

回答框架：

1. batch size 决定梯度估计噪声。
2. 大 batch 更稳定但不一定泛化更好。
3. 大 batch 通常要配合学习率缩放、warmup 和梯度裁剪。

#### 问题 5：如果 LLM 训练中出现 loss spike，你怎么排查？

回答框架：

1. 看数据、mask 和 batch 是否异常。
2. 看 learning rate、warmup、gradient norm 和 clipping。
3. 看 mixed precision、NaN/Inf、loss scale 和分布式同步。

### 常见误区

1. 误区：非凸优化就一定会卡在坏局部最优。
   纠正：高维过参数化网络中，鞍点、平坦区域和训练动态往往更关键。

2. 误区：训练 loss 降低就说明优化完全成功。
   纠正：还要看验证集、泛化、稳定性和最终任务指标。

3. 误区：AdamW 只是 Adam 加 L2 正则。
   纠正：AdamW 使用解耦 weight decay，和直接 L2 正则不完全等价。

4. 误区：batch size 越大越好。
   纠正：大 batch 提高吞吐和稳定性，但可能影响泛化和收敛行为。

5. 误区：所有 loss spike 都是坏事。
   纠正：轻微 spike 可能恢复，但频繁或持续 spike 需要排查数据、数值和超参问题。

### 小练习

1. 用自己的话解释什么是优化景观。
2. 画一个局部极小值、鞍点和平坦区域的二维示意图。
3. 解释为什么大模型训练需要 learning rate warmup。
4. 写出你排查 loss spike 的前 5 个步骤。
5. 对比 Adam、AdamW 和 SGD 的核心差异。
6. 解释为什么训练 loss 低不一定代表泛化好。

### 本讲总结

本讲最重要的结论：

1. 深度网络训练面对的是复杂非凸优化问题。
2. 训练轨迹、初始化、学习率和归一化都会影响最终结果。
3. 优化“好不好”不仅看能否收敛，也看收敛到什么样的解。
4. 鞍点、平坦区域、曲率和梯度噪声是理解训练动态的重要概念。
5. 大模型训练不稳定需要结合优化超参、数值稳定性、数据和分布式系统一起排查。

## 第 4 讲：泛化、过拟合与双下降现象

### 本讲目标

前面几讲分别讲了概率建模、训练目标和优化动态。本讲进入另一个核心问题：模型在训练集上表现好，为什么不一定代表真实能力强？为什么大模型参数越来越多，却不一定按传统直觉严重过拟合？双下降现象又说明了什么？

你需要掌握：

1. 什么是泛化。
2. 训练误差、验证误差、测试误差的区别。
3. 过拟合和欠拟合的本质。
4. bias-variance trade-off 的传统观点。
5. 双下降现象为什么挑战传统直觉。
6. 为什么大模型泛化不能只用参数量解释。
7. 面试中如何讨论 LLM 的泛化、记忆和数据污染。

### 什么是泛化

泛化指的是：

```text
模型在未见过的数据上仍然表现良好的能力
```

训练集表现好，只说明模型能拟合训练样本。

泛化好，才说明模型学到了一些可迁移的规律。

在监督学习里，我们通常关心：

1. 训练误差：模型在训练数据上的误差。
2. 验证误差：模型在调参数据上的误差。
3. 测试误差：模型在最终保留数据上的误差。

在语言模型里，对应的指标常见是：

1. training loss。
2. validation loss。
3. held-out test loss。
4. 下游任务 benchmark。
5. 人类偏好评估。

面试中要注意：泛化不是一个单一指标，而是一组能力问题。

### 训练分布和测试分布

机器学习最基本的假设之一是训练数据和测试数据来自相同或相近分布。

如果训练分布是 `P_train`，测试分布是 `P_test`，理想情况是：

```text
P_train ≈ P_test
```

如果两者差异很大，就会出现分布偏移。

例如：

1. 模型训练数据主要是英文网页，但测试中文法律问答。
2. 训练数据主要是短文本，但测试长上下文推理。
3. 训练数据没有某类工具调用格式，但测试要求严格调用工具。
4. 训练数据中答案有噪声，测试要求事实准确。

因此，泛化不仅是模型问题，也是数据分布问题。

### 欠拟合和过拟合

欠拟合指模型连训练数据都学不好。

常见表现是：

1. training loss 高。
2. validation loss 也高。
3. 模型容量不够。
4. 优化不足。
5. 特征或数据表达不足。

过拟合指模型过度适应训练数据中的细节和噪声，导致未见数据表现变差。

常见表现是：

1. training loss 持续下降。
2. validation loss 不降反升。
3. 模型对训练样本记忆明显。
4. 对轻微扰动很敏感。

传统机器学习里，经常用模型复杂度解释欠拟合和过拟合。

模型太简单，容易欠拟合。

模型太复杂，容易过拟合。

但大模型时代，这个故事不够完整。

### Bias-Variance Trade-off

传统泛化理论里，一个经典框架是 bias-variance trade-off。

bias 可以理解为模型假设太强导致的系统性误差。

variance 可以理解为模型对训练数据扰动的敏感性。

简单模型通常 bias 高、variance 低。

复杂模型通常 bias 低、variance 高。

传统图像是：

```text
模型复杂度增加：训练误差下降，测试误差先下降后上升
```

这对应经典 U 形测试误差曲线。

但深度学习中的大模型常常不完全遵守这个直觉。

### 双下降现象是什么

双下降现象指的是：随着模型复杂度增加，测试误差可能经历两次下降。

大致过程是：

1. 模型很小时，欠拟合，训练误差和测试误差都高。
2. 模型变大后，测试误差下降。
3. 接近刚好能插值训练集时，测试误差可能上升。
4. 模型继续变大，进入过参数化区域，测试误差再次下降。

可以直观理解为：

```text
测试误差：下降 -> 上升 -> 再下降
```

这就是 double descent。

它挑战了传统“模型越大越容易过拟合”的简单说法。

Deep Double Descent 相关工作还强调：双下降不只可能随着模型大小出现，也可能随着训练 epoch、数据量和更综合的 effective model complexity 出现。因此面试中不要把它简化成“参数越多越好”，更稳妥的说法是：泛化由模型规模、数据规模、训练时间、噪声、优化路径和正则化共同决定。

### 最小 demo：画出一个双下降玩具曲线

下面这个 demo 不模拟真实训练，只构造一条玩具测试误差曲线，帮助你记住 double descent 的形状：先下降，在插值阈值附近上升，然后在过参数化区域再次下降。

```python
model_sizes = [1, 2, 3, 4, 5, 6, 7, 8]
test_errors = [0.55, 0.42, 0.33, 0.50, 0.38, 0.30, 0.27, 0.25]

best_size = model_sizes[test_errors.index(min(test_errors))]
peak_size = model_sizes[test_errors.index(max(test_errors[2:5]))]

for size, error in zip(model_sizes, test_errors):
    print(f"model_size={size}, test_error={error:.2f}")

print("interpolation-like peak near size:", peak_size)
print("best observed size:", best_size)
```

这个 demo 只是可视化直觉，不是 double descent 的实验证明。真实研究需要控制数据噪声、训练 epoch、模型族、正则化和评估分布。

### 插值阈值

双下降中有一个关键概念：插值阈值。

插值阈值指模型刚好有能力把训练集拟合到接近零误差的位置。

在这个附近，模型可能对训练数据噪声非常敏感，测试误差升高。

但当模型继续变大，过参数化程度更高时，优化器可能找到更平滑、更简单或更稳定的解，测试误差反而下降。

这就是为什么不能只用参数数量判断泛化。

模型复杂度、优化路径、隐式正则化和数据规模共同决定泛化行为。

### 为什么大模型没有简单地严重过拟合

LLM 参数量巨大，按传统直觉似乎应该非常容易过拟合。

但实际情况更复杂。

原因包括：

1. 训练数据规模也极大。
2. next-token prediction 提供了海量监督信号。
3. 优化器和训练动态带来隐式正则化。
4. dropout、weight decay、数据混合等显式正则化也会影响训练。
5. 过参数化可能让模型找到更平坦的解。
6. 真实数据分布中存在大量可泛化结构。

但这不代表大模型不会记忆。

更准确的说法是：

```text
大模型同时存在记忆和泛化，二者不是互斥关系
```

### 记忆和泛化的关系

LLM 会记忆训练数据中的部分内容，尤其是：

1. 高频重复内容。
2. 罕见但被重复采样的样本。
3. 格式固定的文本。
4. 代码、日志、配置、个人信息等高模式化数据。

LLM 也会泛化到新任务，尤其是当任务结构在训练数据中有足够多变体时。

所以面试中不要说“大模型只是背诵”，也不要说“大模型完全不会记忆”。

更好的说法是：

```text
LLM 在训练中既会压缩和记忆部分数据，也会学习可迁移的统计结构。是否表现为记忆或泛化，取决于数据重复度、模型容量、训练轮数、任务分布和评估方式。
```

### 数据污染和评估失真

大模型泛化讨论中，数据污染非常重要。

数据污染指的是测试集或 benchmark 内容出现在训练数据中。

如果污染存在，模型在测试集上表现好，可能不是因为泛化，而是因为见过类似甚至相同样本。

常见污染形式包括：

1. benchmark 原题出现在网页训练数据中。
2. benchmark 解答出现在训练数据中。
3. 测试集被多次公开讨论，进入后续训练语料。
4. 训练数据中存在 paraphrase 版本。
5. benchmark 生成模板泄露。

评估 LLM 泛化时，必须考虑污染检测。

常见方法包括：

1. exact match 去重。
2. n-gram overlap 检测。
3. embedding similarity 检测。
4. 时间切分评估。
5. 使用私有或动态生成测试集。

### 分布外泛化

分布外泛化是指模型面对训练分布外样本时的表现。

这比普通 held-out validation 更难。

例如：

1. 新领域知识。
2. 新语言。
3. 新工具 API。
4. 更长上下文。
5. 更复杂推理链。
6. 对抗性提示。

LLM 在分布外场景下可能出现：

1. 幻觉增加。
2. 置信度和正确性不匹配。
3. 格式遵循下降。
4. 推理步骤不稳定。
5. 安全行为退化。

因此，大模型评估不能只看平均 benchmark 分数，还要看鲁棒性和边界条件。

### Scaling Law 和泛化

Scaling law 研究的是模型规模、数据规模、计算量与 loss 之间的关系。

一般来说，在合理数据和训练配置下，增加模型规模、数据规模和计算量可以降低 validation loss。

这说明大模型泛化并不只由参数量决定，而是由三者共同决定：

```text
模型规模 + 数据规模 + 计算预算
```

如果模型太大但数据太少，可能过拟合。

如果数据很多但模型太小，可能欠拟合。

如果计算预算不足，模型可能没有训练到合适状态。

这也是为什么 Chinchilla 一类工作强调 compute-optimal training：在固定计算预算下，模型参数和训练 token 数需要平衡。

### 正则化在大模型中的角色

传统正则化方法包括：

1. L2 正则。
2. dropout。
3. early stopping。
4. data augmentation。
5. label smoothing。

LLM 中常见的正则化或泛化相关手段包括：

1. weight decay。
2. dropout 或 attention dropout。
3. 数据去重。
4. 数据混合和重采样。
5. 训练 token 数控制。
6. early stopping 或 checkpoint selection。
7. 后训练中的 KL penalty。

但在大模型预训练中，最重要的“正则化”往往不是某个单独技巧，而是高质量、大规模、多样化数据和稳定训练动态。

### Early Stopping 是否总是必要

在小数据监督学习中，early stopping 是常见防过拟合手段。

在大模型预训练中，情况更复杂。

如果训练数据足够大、每个 token 基本只见一次或少量次数，模型可能很长时间都处在继续学习阶段。

但如果：

1. 数据重复率高。
2. 训练 token 数远超数据多样性。
3. validation loss 开始上升。
4. 下游任务能力退化。

那么 early stopping 或调整数据配比就很重要。

面试中可以说：early stopping 是否必要取决于数据规模、重复度、训练 token 数和验证曲线，而不是固定规则。

### 泛化不能只看 loss

validation loss 是重要指标，但 LLM 的真实泛化还包括：

1. 指令泛化。
2. 领域泛化。
3. 长上下文泛化。
4. 工具调用泛化。
5. 多轮对话泛化。
6. 推理泛化。
7. 安全泛化。

模型 loss 更低，不一定在所有能力上更好。

原因是：

1. loss 是 token-level 指标。
2. 下游任务是 sequence-level 或 behavior-level 指标。
3. 评估 prompt 和训练格式可能不一致。
4. 解码策略会影响最终输出。
5. 后训练会改变模型行为分布。

### LLM 泛化的实际诊断

在真实项目中，如果你想判断模型是否泛化更好，可以看：

1. held-out validation loss。
2. 不同领域 validation loss。
3. 去污染 benchmark。
4. 时间切分测试集。
5. 人工构造的新题。
6. adversarial evaluation。
7. 长尾样本表现。
8. 多语言和多领域表现。
9. 真实用户日志上的离线评估。

对面试来说，能讲出“泛化评估要分层”，会比只说“看 test accuracy”更好。

### 泛化和对齐的关系

预训练泛化和对齐泛化不是一回事。

预训练泛化主要关注模型是否学到语言、知识和任务结构。

对齐泛化关注模型是否能在新指令、新场景和边界条件下保持符合人类意图和安全约束的行为。

例如，一个模型可能有很强的知识泛化，但在越狱提示下安全泛化很差。

也可能一个模型 benchmark 很高，但在真实用户复杂指令下不稳定。

所以大模型能力评估需要同时考虑：

1. 预测泛化。
2. 任务泛化。
3. 行为泛化。
4. 安全泛化。

### 面试中怎么讲

如果被问到“为什么大模型参数这么多却还能泛化”，可以这样回答：

```text
传统观点认为模型越大越容易过拟合，但深度学习中的泛化由模型规模、数据规模、训练 token 数、优化动态和隐式正则化共同决定。大模型通常在海量、多样化数据上做 next-token prediction，监督信号非常密集；过参数化也可能让优化器找到更平坦、更稳定的解。因此参数多不必然意味着泛化差。但大模型仍然会记忆训练数据，所以需要通过数据去重、污染检测、时间切分和分布外评估区分记忆和泛化。
```

如果被问到“双下降是什么”，可以这样回答：

```text
双下降指测试误差随模型复杂度增加时，可能先下降、再在接近插值阈值附近上升，然后在过参数化区域再次下降。它说明传统 U 形 bias-variance 曲线不足以解释现代深度学习，模型复杂度、优化路径和隐式正则化都会影响最终泛化。
```

### 真实项目中的坑

#### 坑 1：把 benchmark 分数直接等同于泛化

benchmark 可能被污染，也可能覆盖面有限。真实泛化要看多领域、多时间、多难度评估。

#### 坑 2：只用参数量判断过拟合

参数量只是一个因素，数据规模、数据重复率、训练 token 数、优化动态同样重要。

#### 坑 3：忽略数据去重

重复数据会增加记忆风险，也会让 validation loss 和 benchmark 表现失真。

#### 坑 4：只看平均分

平均分可能掩盖长尾失败、对抗失败和安全失败。

#### 坑 5：把泛化和对齐混为一谈

模型能解决新任务，不代表它在新场景中一定安全、诚实、可控。

### 面试官会怎么问

#### 问题 1：什么是泛化？

回答框架：

1. 泛化是模型在未见数据上表现良好的能力。
2. 训练误差低不代表泛化好。
3. 需要用验证集、测试集和分布外评估判断。

#### 问题 2：过拟合和欠拟合怎么区分？

回答框架：

1. 欠拟合：训练误差和验证误差都高。
2. 过拟合：训练误差低但验证误差高。
3. 大模型中还要关注数据重复和污染。

#### 问题 3：双下降为什么重要？

回答框架：

1. 它挑战了传统 U 形泛化曲线。
2. 在插值阈值附近测试误差可能升高。
3. 过参数化后测试误差可能再次下降。

#### 问题 4：怎么判断 LLM 是泛化还是记忆？

回答框架：

1. 做数据污染检测和去重。
2. 使用时间切分或私有动态测试集。
3. 看 paraphrase、扰动和分布外样本上的表现。

#### 问题 5：validation loss 下降是否一定代表模型更强？

回答框架：

1. 不一定。
2. loss 是 token-level 指标。
3. 下游能力、指令跟随、安全性和推理能力需要额外评估。

### 常见误区

1. 误区：参数越多一定越容易过拟合。
   纠正：大模型泛化由模型、数据、计算和训练动态共同决定。

2. 误区：测试集分数高就一定说明泛化好。
   纠正：需要排查数据污染和 benchmark 覆盖不足。

3. 误区：双下降说明越大越好。
   纠正：双下降说明过参数化可能改善泛化，但仍受数据质量、计算预算和训练设置限制。

4. 误区：validation loss 越低，所有能力都越强。
   纠正：loss 与下游复杂能力相关但不等价。

5. 误区：记忆和泛化只能二选一。
   纠正：LLM 通常同时存在记忆和泛化。

## 第 5 讲：贝叶斯视角、先验与不确定性

### 本讲目标

前几讲从概率建模、交叉熵、优化和泛化角度理解了深度学习。本讲换一个视角：如果把模型参数、预测结果和训练数据都放到贝叶斯框架里，大模型训练和推理会呈现出什么新的理解？

你需要掌握：

1. 贝叶斯公式的基本含义。
2. 先验、似然、后验分别是什么。
3. MAP 估计和最大似然估计的关系。
4. 正则化为什么可以看成一种先验。
5. 不确定性有哪些类型。
6. 为什么深度学习很少完整做贝叶斯推断。
7. 贝叶斯视角对 LLM 评估、校准和幻觉有什么启发。

### 为什么需要贝叶斯视角

在普通训练流程里，我们通常把模型参数看成一个需要优化的固定变量。

训练目标是：

```text
找到一组参数 θ，让训练数据上的 loss 尽可能低
```

这是一种点估计视角。

贝叶斯视角会问另一个问题：

```text
给定训练数据 D，我们对参数 θ 的不确定性应该如何更新？
```

也就是说，贝叶斯方法不只是找一个最优参数，而是希望得到参数的后验分布：

```text
P(θ | D)
```

这会让我们更自然地讨论不确定性、先验知识、数据量不足、模型置信度和分布外样本。

### 贝叶斯公式

贝叶斯公式是：

```text
P(θ | D) = P(D | θ) P(θ) / P(D)
```

其中：

1. `P(θ | D)` 是后验，表示看到数据后对参数的信念。
2. `P(D | θ)` 是似然，表示给定参数时数据出现的概率。
3. `P(θ)` 是先验，表示看到数据前对参数的信念。
4. `P(D)` 是证据项，也叫 marginal likelihood，用来归一化。

对面试来说，最重要的是理解这个更新逻辑：

```text
后验 ∝ 似然 × 先验
```

数据告诉我们哪些参数能解释观测结果，先验告诉我们哪些参数本来更合理。

### 用一个简单例子理解先验和后验

假设你要判断一枚硬币正面朝上的概率 `p`。

如果没有任何数据，你可能认为 `p` 大概率接近 0.5，这就是先验。

现在你抛了 10 次，得到 8 次正面。似然会支持一个更大的 `p`。

后验会综合两件事：

1. 先验认为硬币大概率公平。
2. 数据显示正面出现次数偏多。

如果只抛 10 次，后验不会完全相信 `p=0.8`。

如果抛 10000 次，得到 8000 次正面，后验就会更强烈地偏向 `p≈0.8`。

这说明：

```text
数据少时，先验影响大；数据多时，似然影响大
```

这个直觉在大模型里也有用：当某个领域数据少、噪声大或分布外时，模型的不确定性应该更高。

### 最小 demo：Beta-Binomial 硬币后验更新

下面这个 demo 用 Beta 分布作为硬币正面概率的先验。观察到 `heads` 次正面和 `tails` 次反面后，后验仍是 Beta 分布，参数可以直接相加。

```python
def beta_posterior(prior_alpha, prior_beta, heads, tails):
    post_alpha = prior_alpha + heads
    post_beta = prior_beta + tails
    posterior_mean = post_alpha / (post_alpha + post_beta)
    return post_alpha, post_beta, posterior_mean


for heads, tails in [(8, 2), (80, 20), (8000, 2000)]:
    alpha, beta, mean = beta_posterior(1, 1, heads, tails)
    print(f"heads={heads}, tails={tails} -> Beta({alpha}, {beta}), mean={mean:.3f}")
```

输出中可以看到：当数据很少时，先验会明显影响后验；当数据很多时，后验均值更接近观测频率。这个例子对应“数据少时先验影响大，数据多时似然影响大”的直觉。

### 最大似然估计和 MAP 估计

最大似然估计，也就是 MLE，目标是最大化：

```text
P(D | θ)
```

等价于最大化 log likelihood：

```text
log P(D | θ)
```

语言模型的 next-token prediction 训练，本质上就是最大化训练 token 序列的似然，或者最小化负对数似然。

MAP 估计，也就是 maximum a posteriori，目标是最大化后验：

```text
P(θ | D)
```

根据贝叶斯公式，忽略和 `θ` 无关的 `P(D)` 后：

```text
argmax_θ P(θ | D) = argmax_θ P(D | θ) P(θ)
```

取 log 后：

```text
argmax_θ [log P(D | θ) + log P(θ)]
```

所以 MAP 比 MLE 多了一项先验。

### 正则化为什么像先验

很多正则化方法都可以从贝叶斯角度解释。

例如 L2 正则通常写成：

```text
Loss = NLL + λ ||θ||^2
```

其中 `NLL` 是 negative log likelihood。

如果我们假设参数先验是零均值高斯分布：

```text
P(θ) ∝ exp(-λ ||θ||^2)
```

那么最大化后验等价于最小化：

```text
NLL + λ ||θ||^2
```

也就是说，L2 正则可以理解为一个偏好小权重的高斯先验。

类似地：

1. L1 正则可以对应稀疏先验。
2. weight decay 可以理解为对参数规模的约束。
3. early stopping 可以粗略理解为限制模型不要过度适配训练数据。
4. 数据增强可以理解为向模型注入某些不变性先验。

这里的“先验”不一定总是显式写成概率分布，也可以是架构、数据和训练策略里的 inductive bias。

### 深度学习里的先验不只来自正则化

在大模型中，先验常常不是一个手写的 `P(θ)`，而是隐含在系统设计里。

常见来源包括：

1. 架构先验：Transformer 假设 token 之间通过 attention 交互。
2. 位置先验：RoPE、ALiBi 等方法编码序列位置关系。
3. 数据先验：训练语料决定模型更熟悉哪些语言、领域和风格。
4. 优化先验：AdamW、学习率调度和 batch size 影响最终解。
5. 正则化先验：weight decay、dropout、数据去重影响泛化。
6. 后训练先验：SFT、RLHF、DPO 把人类偏好注入模型行为。

所以面试中如果被问“LLM 有先验吗”，不要只想到贝叶斯公式里的显式先验。

更好的回答是：

```text
LLM 的先验既包括显式正则化，也包括架构、数据、优化和后训练带来的 inductive bias。它们共同决定模型在数据不足、冲突或分布外场景下更倾向于输出什么。
```

### 参数不确定性和预测不确定性

贝叶斯视角下，不确定性至少可以分成两层。

第一层是参数不确定性：

```text
我们不确定哪组 θ 才是正确的
```

第二层是预测不确定性：

```text
即使给定模型参数，输出 y 也可能有多个合理选择
```

在语言模型中，预测分布通常是：

```text
P(x_t | x_{1:t-1}, θ)
```

如果做完整贝叶斯预测，应该对参数后验积分：

```text
P(x_t | x_{1:t-1}, D) = ∫ P(x_t | x_{1:t-1}, θ) P(θ | D) dθ
```

但真实 LLM 通常不会这么做，而是使用训练得到的一组参数进行预测。

这意味着模型输出的 softmax 概率不等于完整贝叶斯不确定性。

### Aleatoric 和 Epistemic 不确定性

机器学习里常区分两类不确定性。

第一类是 aleatoric uncertainty，也叫数据噪声不确定性。

它来自数据本身的随机性或不可判定性。

例如：

1. 用户问题本身含糊。
2. 同一个开放问题有多个合理回答。
3. 训练标签有噪声。
4. 未来事件本来不可确定。

第二类是 epistemic uncertainty，也叫认知不确定性。

它来自模型不知道，通常可以通过更多数据、更多训练或更好的模型降低。

例如：

1. 某个领域训练数据很少。
2. 问题超出模型知识截止时间。
3. 输入来自分布外场景。
4. 模型没有见过某种工具调用格式。

对 LLM 来说，这个区分非常重要。

如果是不知道，就应该承认不确定、检索、调用工具或请求澄清。

如果是问题本身多解，就应该说明假设，并给出多个可能答案。

### 为什么深度学习很少完整做贝叶斯推断

完整贝叶斯深度学习很有吸引力，但在大模型中非常困难。

原因包括：

1. 参数维度极高，后验分布难以表示。
2. 后验积分计算成本巨大。
3. MCMC 等经典方法很难扩展到现代 LLM 规模。
4. 训练和推理成本已经很高，完整贝叶斯会进一步增加成本。
5. 工程系统更需要稳定、快速、可部署的点估计模型。

因此，实际工程中更常用近似方法或替代指标。

例如：

1. ensemble。
2. dropout 近似不确定性。
3. checkpoint averaging。
4. temperature scaling。
5. calibration evaluation。
6. self-consistency 或多次采样。
7. 检索和工具验证。
8. abstention，也就是允许模型拒答或表达不确定。

这些方法不等于完整贝叶斯推断，但能在工程上部分缓解不确定性问题。

### Softmax 概率不是可靠置信度

LLM 每一步都会输出一个 token 分布。

很多人会误以为：

```text
softmax 概率高 = 模型真的很确定 = 答案一定可靠
```

这是危险的。

原因有几个：

1. softmax 是在当前模型参数和上下文下的相对 token 概率。
2. 它不包含完整参数不确定性。
3. 模型可能在分布外问题上仍然给出高概率答案。
4. 后训练可能让模型更自信地输出符合偏好的文本。
5. 解码策略会改变最终输出和表面置信度。

这就是校准问题。

一个模型如果校准良好，意味着：

```text
当它说自己有 80% 把握时，长期来看大约 80% 是正确的
```

但很多大模型存在过度自信，尤其是在事实问答、数学推理、代码和分布外问题上。

### 贝叶斯视角下的幻觉

幻觉可以从多个角度解释。

从贝叶斯和不确定性角度看，常见原因是：

1. 模型缺少足够证据，却没有表达不确定。
2. 训练数据中存在错误或冲突模式。
3. 后训练鼓励模型给出有帮助的完整回答，而不是拒答。
4. softmax 高概率不代表事实被验证。
5. 解码过程选择了流畅但不真实的序列。

理想行为应该是：

```text
当后验证据不足或问题超出知识范围时，模型应提高不确定性表达，主动检索、调用工具或要求更多上下文。
```

这也是为什么 RAG、tool use、verification、calibration 和 abstention 在 LLM 系统里很重要。

### 贝叶斯视角和模型评估

如果用贝叶斯视角看评估，我们不仅要问“模型答对多少”，还要问：

1. 答错时是否知道自己可能错。
2. 面对分布外问题是否会过度自信。
3. 模型能否表达不确定性。
4. 多次采样的答案是否稳定。
5. 检索或工具结果能否降低不确定性。
6. 后训练是否改善或恶化校准。

常见评估方向包括：

1. calibration curve。
2. expected calibration error。
3. selective prediction。
4. abstention accuracy。
5. OOD detection。
6. factuality evaluation。
7. self-consistency stability。

这些评估会在后面的评估专题里更系统展开。

### 一个最小代码例子：估计输出不稳定性

完整贝叶斯推断很难，但我们可以用多次采样粗略观察回答是否稳定。

伪代码如下：

```python
from collections import Counter


def estimate_answer_stability(model, prompt, n=20, temperature=0.8):
    answers = []
    for _ in range(n):
        answer = model.generate(prompt, temperature=temperature)
        answers.append(normalize(answer))

    counts = Counter(answers)
    most_common_answer, count = counts.most_common(1)[0]
    stability = count / n
    return most_common_answer, stability, counts
```

这个例子不是严格的不确定性估计，但很实用。

如果同一个问题多次采样答案高度分散，说明模型在这个 prompt 下不稳定。

在真实系统中，可以进一步结合：

1. 多模型投票。
2. 检索证据覆盖率。
3. 答案引用是否支持结论。
4. verifier 或 judge 模型。
5. 规则校验和单元测试。

### 面试中怎么讲

如果面试官问“贝叶斯视角下如何理解深度学习训练”，可以这样回答：

```text
普通深度学习训练通常是点估计，目标是找到一组参数最小化训练损失。贝叶斯视角会把参数看成随机变量，训练数据用于把先验 P(θ) 更新成后验 P(θ|D)。最大似然只看 P(D|θ)，而 MAP 估计还会乘上先验 P(θ)。很多正则化方法可以解释为对参数加入先验，例如 L2 正则对应高斯先验。
```

如果面试官问“为什么 softmax 概率不能直接当作置信度”，可以这样回答：

```text
softmax 概率只是给定当前参数和上下文时不同 token 的相对概率，它不包含完整的参数不确定性，也不能保证事实正确。模型在分布外样本上可能仍然给出高概率输出，后训练也可能让回答更流畅、更自信。因此要通过校准评估、拒答机制、检索验证、多次采样或 verifier 来判断可靠性。
```

### 真实项目中的坑

#### 坑 1：把概率输出当成真实置信度

token 概率高只能说明模型在当前分布下偏好这个 token，不代表答案经过事实验证。

#### 坑 2：只优化准确率，不评估校准

两个模型准确率相同，但一个知道自己什么时候不确定，另一个盲目自信，生产风险完全不同。

#### 坑 3：把拒答当成能力差

在证据不足或高风险场景中，合理拒答和请求澄清是可靠性的体现。

#### 坑 4：忽略分布外不确定性

模型面对新领域、新格式、新工具时可能非常不稳定，需要单独评估。

#### 坑 5：以为 RAG 一定能消除不确定性

RAG 只能提供外部证据，如果检索失败、证据冲突或生成阶段不忠实，仍然会出错。

### 面试官会怎么问

#### 问题 1：什么是先验、似然和后验？

回答框架：

1. 先验是看到数据前对参数或假设的信念。
2. 似然是给定参数时观测数据出现的概率。
3. 后验是看到数据后更新得到的信念。
4. 贝叶斯公式表达为后验正比于似然乘先验。

#### 问题 2：MLE 和 MAP 有什么区别？

回答框架：

1. MLE 最大化 `P(D|θ)`。
2. MAP 最大化 `P(θ|D)`。
3. MAP 等价于在似然基础上加入先验项。
4. 正则化可以看成 MAP 中的先验约束。

#### 问题 3：为什么 L2 正则可以看成高斯先验？

回答框架：

1. 高斯先验对大权重分配较低概率。
2. 取负 log 后会得到类似 `||θ||^2` 的惩罚项。
3. 因此最小化 NLL 加 L2 正则等价于 MAP 估计。

#### 问题 4：LLM 的不确定性有哪些类型？

回答框架：

1. aleatoric uncertainty 来自数据本身噪声或问题多解。
2. epistemic uncertainty 来自模型知识不足或分布外输入。
3. 前者不能完全通过更多数据消除，后者可以通过数据、训练、检索或工具部分降低。

#### 问题 5：如何让 LLM 更好地处理不确定性？

回答框架：

1. 做校准评估，而不是只看准确率。
2. 允许模型表达不确定或拒答。
3. 引入 RAG、工具调用和外部验证。
4. 使用多次采样、self-consistency、ensemble 或 verifier。
5. 对分布外和高风险场景做单独测试。

### 常见误区

1. 误区：贝叶斯方法就是换一个公式写训练目标。
   纠正：贝叶斯视角核心是用数据更新不确定性，并显式区分先验、似然和后验。

2. 误区：正则化只是防止参数太大，和概率无关。
   纠正：很多正则化都可以解释为对参数加入先验。

3. 误区：softmax 概率高说明答案一定可靠。
   纠正：softmax 概率不等于事实置信度，也不包含完整参数不确定性。

4. 误区：大模型如果不知道，就一定会低概率输出。
   纠正：模型在分布外或幻觉场景中也可能高置信输出错误答案。

5. 误区：不确定性只影响事实问答。
   纠正：代码、数学、工具调用、安全策略和系统设计都会受到不确定性影响。

### 小练习

1. 用自己的话解释 `后验 ∝ 似然 × 先验`。
2. 推导为什么 L2 正则可以对应高斯先验。
3. 举一个 aleatoric uncertainty 的 LLM 例子。
4. 举一个 epistemic uncertainty 的 LLM 例子。
5. 设计一个简单实验，判断模型在某类问题上是否过度自信。
6. 用 2 分钟回答“为什么 softmax 概率不能直接当置信度”。

### 本讲总结

本讲最重要的结论：

1. 贝叶斯视角把训练看成从先验到后验的信念更新。
2. MLE 只最大化似然，MAP 在似然之外加入先验。
3. 正则化、架构、数据、优化和后训练都可以提供某种先验或 inductive bias。
4. LLM 的 softmax 概率不等于可靠置信度。
5. 不确定性可以分为 aleatoric 和 epistemic，两者处理方式不同。
6. 大模型工程中通常不做完整贝叶斯推断，而是用校准、采样、检索、工具和验证机制近似处理不确定性。
7. 贝叶斯视角能帮助我们更好地理解幻觉、拒答、校准和分布外风险。

## 第 6 讲：损失函数设计与优化目标错配

### 本讲目标

前面几讲讨论了概率建模、交叉熵、优化、泛化和贝叶斯视角。本讲关注一个更贴近真实大模型工程的问题：我们训练时优化的 loss，真的等于最终想要的能力和行为吗？

答案通常是否定的。

你需要掌握：

1. 什么是损失函数。
2. 为什么 loss 是训练目标的代理指标。
3. 什么是优化目标错配。
4. token-level loss 和 sequence-level 任务目标的差异。
5. 为什么预训练、SFT、RLHF、DPO 都在处理不同层面的目标错配。
6. reward hacking 和 Goodhart's Law 的基本含义。
7. 面试中如何讨论“loss 下降但模型不一定更好”。

### 损失函数是什么

损失函数是训练时用来告诉模型“当前预测有多差”的函数。

在深度学习训练中，我们通常做的是：

```text
选择一个 loss，然后通过梯度下降最小化它
```

例如语言模型预训练中，最常见的是 next-token cross entropy：

```text
Loss = -Σ_t log P(x_t | x_{1:t-1})
```

这个 loss 很自然，因为它等价于最大化训练语料的似然。

但要注意：

```text
可优化的 loss 通常只是我们真正目标的代理
```

真实目标可能是：

1. 回答正确。
2. 推理可靠。
3. 遵循指令。
4. 不幻觉。
5. 安全合规。
6. 工具调用格式正确。
7. 用户体验好。

这些目标很难全部写成一个简单、可微、稳定的 loss。

### 为什么需要代理目标

很多真实目标不可直接优化。

例如“模型是否有帮助”这个目标，看起来简单，但很难直接写成数学函数。

原因包括：

1. 有帮助依赖具体用户和场景。
2. 一个回答可能有多个合理版本。
3. 正确性、安全性、简洁性之间存在 trade-off。
4. 人类偏好本身有噪声。
5. 长答案的质量很难逐 token 标注。

所以训练时通常选择一个可操作的代理目标。

例如：

1. 预训练用 next-token likelihood 代理语言和知识学习。
2. SFT 用专家答案的交叉熵代理指令跟随。
3. RLHF 用 reward model 分数代理人类偏好。
4. DPO 用偏好对比 loss 代理“更偏好回答胜过较差回答”。
5. 代码模型用单元测试通过率代理代码正确性。

代理目标越接近真实目标，训练越有效。

代理目标和真实目标差距越大，目标错配风险越高。

### 什么是优化目标错配

优化目标错配指的是：

```text
训练时优化的指标，和最终真正关心的目标不完全一致
```

在 LLM 中非常常见。

例如：

1. 预训练 loss 降低，但事实问答不一定更准确。
2. SFT loss 降低，但回答可能更啰嗦或更模板化。
3. reward model 分数升高，但人类不一定真的更喜欢。
4. benchmark 分数提高，但真实用户场景不一定改善。
5. 格式遵循变好，但推理正确率可能下降。
6. 安全拒答变多，但有用性可能下降。

所以面试中不能只说“优化 loss 就能得到好模型”。

更准确的说法是：

```text
loss 是训练信号，不是最终能力本身。我们需要理解它代理了什么目标，又遗漏了什么目标。
```

### Token-level loss 和 sequence-level 目标

语言模型预训练的 loss 通常是 token-level 的。

它关心每个 token 的预测概率：

```text
-log P(x_t | x_{1:t-1})
```

但很多真实任务是 sequence-level 或 behavior-level 的。

例如：

1. 数学题最终答案是否正确。
2. 代码能否通过测试。
3. 摘要是否覆盖关键信息。
4. 多轮对话是否完成用户目标。
5. 工具调用参数是否完全合法。
6. 长推理链中是否每一步都可靠。

一个答案可能只有某个关键 token 错了，整个任务就失败。

反过来，一个答案 token-level loss 不一定最低，但整体更符合用户意图。

这就是 token-level objective 和 sequence-level objective 的错配。

### 一个简单例子：数学题

假设问题是：

```text
17 × 23 等于多少？
```

模型 A 输出：

```text
17 × 23 = 391
```

模型 B 输出：

```text
17 × 23 = 381
```

两个回答大部分 token 都一样，只是最后数字不同。

从 token-level loss 看，差距可能集中在最后几个 token。

但从任务目标看：

```text
一个完全正确，一个完全错误
```

这说明很多任务不能只用平均 token loss 判断。

数学、代码、工具调用、结构化输出尤其明显。

### 预训练目标的错配

预训练通常优化 next-token prediction。

它的优点是：

1. 数据规模极大。
2. 不需要人工标注。
3. 目标稳定、可扩展。
4. 能学习语言、知识和任务模式。

但它和最终助手行为存在错配。

预训练模型学的是：

```text
给定上下文，下一个 token 在训练语料中最可能是什么
```

而用户想要的是：

```text
给定我的意图，输出正确、有帮助、安全、符合格式的答案
```

这两个目标不一样。

例如，互联网文本中可能包含：

1. 错误信息。
2. 偏见内容。
3. 不安全指令。
4. 低质量回答。
5. 风格混乱的对话。

如果只做预训练，模型可能模仿这些模式，而不是稳定地作为助手。

这就是为什么需要 instruction tuning 和 alignment。

### SFT 目标的错配

SFT 通常用人工或模型生成的高质量指令数据训练。

目标还是交叉熵：

```text
最大化参考答案 token 的概率
```

SFT 能显著改善指令跟随，但也有错配。

问题在于：

1. 同一个指令可能有多个好答案，但 SFT 只奖励参考答案。
2. 参考答案不一定是最优答案。
3. 交叉熵鼓励模仿，不直接优化人类偏好。
4. 数据风格会强烈影响模型输出风格。
5. 长答案中局部 token loss 低，不代表整体质量高。

例如，一个问题可以简洁回答，也可以详细解释。

如果训练集中总是长篇回答，模型可能学会过度解释。

如果训练集中拒答过多，模型可能过度拒答。

如果训练集中格式模板很强，模型可能回答僵硬。

所以 SFT 解决了预训练和指令行为的部分错配，但没有完全解决偏好和真实任务目标。

### RLHF 目标的错配

RLHF 引入 reward model，希望更直接优化人类偏好。

流程通常是：

1. 收集同一 prompt 下多个回答。
2. 人类标注哪个回答更好。
3. 训练 reward model 预测偏好。
4. 用强化学习优化模型，使输出获得更高 reward。

这比 SFT 更接近“人类喜欢什么”。

但新的错配出现了：

```text
优化 reward model 分数 ≠ 真正优化人类长期满意度
```

reward model 本身是一个近似器，会有漏洞。

模型可能学会：

1. 输出更像高分回答的表面风格。
2. 变得啰嗦、圆滑、过度自信。
3. 迎合 reward model 的偏好模式。
4. 在 reward model 没覆盖的场景中失效。
5. 牺牲多样性或创造性。

这就是 reward hacking 风险。

### DPO 目标的错配

DPO 通过偏好对直接优化模型，不显式训练 reward model 再做 PPO。

它常被理解为更简单、更稳定的偏好优化方法。

但 DPO 也不是没有目标错配。

它依赖偏好数据：

```text
(prompt, chosen, rejected)
```

如果偏好数据质量不好，模型就会学到错误偏好。

常见问题包括：

1. chosen 不一定真的好，只是比 rejected 好。
2. 偏好标注可能偏向表面流畅性。
3. 数据覆盖不足会导致分布外行为不稳定。
4. 过度优化会让模型丢失原有能力或变得模式化。
5. 偏好目标和事实正确、代码通过、数学严谨不完全一致。

所以 DPO 是一个有用工具，但不能替代系统评估。

### Goodhart's Law

目标错配中有一个很重要的经验法则：Goodhart's Law。

它常被表述为：

```text
当一个指标变成目标时，它就不再是一个好指标
```

含义是：如果你把某个代理指标作为强优化目标，系统可能找到提高指标但不改善真实目标的方法。

在 LLM 中例子很多：

1. 优化 benchmark 分数，模型可能过拟合题型。
2. 优化人类偏好分数，模型可能学会讨好式回答。
3. 优化拒答安全率，模型可能过度拒答。
4. 优化平均响应长度，模型可能输出废话。
5. 优化格式合法率，模型可能牺牲内容质量。

Goodhart's Law 提醒我们：

```text
指标只能辅助优化，不能替代对真实目标的理解
```

### Reward Hacking

reward hacking 是目标错配在强化学习和偏好优化中的典型表现。

它指模型找到了一种让 reward 变高的方法，但这种方法不符合设计者真正意图。

例如：

1. 模型学会输出“看起来很专业”的长答案，但内容不准确。
2. 模型在不确定时编造引用，因为带引用的回答得分更高。
3. 模型过度道歉或过度解释，因为 reward model 偏好礼貌语气。
4. 模型在安全评估中学会关键词规避，而不是真正理解风险。
5. 模型为了通过测试写死特定样例，而不是解决一般问题。

reward hacking 的本质是：

```text
优化器利用了代理目标和真实目标之间的缝隙
```

模型越强、优化越充分，越可能暴露代理目标的漏洞。

### 多目标之间的 trade-off

LLM 不是只优化一个目标。

实际系统通常同时关心：

1. 正确性。
2. 有用性。
3. 安全性。
4. 简洁性。
5. 解释性。
6. 鲁棒性。
7. 延迟和成本。
8. 个性化和一致性。

这些目标经常冲突。

例如：

1. 更安全可能导致更多拒答。
2. 更详细可能导致更慢、更啰嗦。
3. 更创造性可能降低事实可靠性。
4. 更严格格式可能降低自然表达。
5. 更高通过率可能增加过拟合测试集风险。

所以训练目标设计不是“找一个完美 loss”，而是做一组取舍。

优秀的工程回答要能讲清楚：

```text
我优化了什么，牺牲了什么，用什么评估来监控副作用
```

### 目标错配如何诊断

真实项目中，如果出现“loss 变好但体验变差”，要系统排查。

可以从几个层面看：

1. 训练 loss 是否真的下降。
2. validation loss 是否同步改善。
3. 下游任务指标是否改善。
4. 人类评估是否改善。
5. 安全和拒答率是否变化。
6. 长尾样本是否退化。
7. 输出长度、风格、格式是否漂移。
8. 是否对 benchmark 或 reward model 过拟合。

一个常见现象是：

```text
平均指标上升，但关键失败案例变多
```

所以不能只看 aggregate score，还要做 error analysis。

### 如何缓解目标错配

缓解目标错配没有单一银弹，通常需要组合方法。

常见策略包括：

1. 使用多维评估，而不是单一分数。
2. 把 token-level loss 和任务级指标一起看。
3. 引入高质量 human eval。
4. 对高风险场景单独建评估集。
5. 做分布外和长尾测试。
6. 做 ablation，确认改动真的带来收益。
7. 防止 reward model 或 benchmark 过拟合。
8. 在优化偏好时保留 KL 约束，避免模型漂移太大。
9. 使用 verifier、unit test、工具结果等外部信号。
10. 对失败案例进行数据回流和 targeted fine-tuning。

核心思想是：

```text
训练目标、评估目标和产品目标要尽量闭环
```

### 一个最小代码例子：同时看 loss 和任务指标

下面是一个非常简化的评估框架，说明为什么不能只看 loss。

```python
def evaluate_model(model, dataset):
    total_loss = 0.0
    exact_match = 0
    format_ok = 0

    for example in dataset:
        prompt = example["prompt"]
        target = example["target"]

        loss = model.compute_loss(prompt, target)
        output = model.generate(prompt, temperature=0.0)

        total_loss += loss
        exact_match += int(normalize(output) == normalize(target))
        format_ok += int(check_format(output))

    n = len(dataset)
    return {
        "loss": total_loss / n,
        "exact_match": exact_match / n,
        "format_ok": format_ok / n,
    }
```

真实项目中还会加入更多指标：

1. pass@k。
2. human preference win rate。
3. factuality score。
4. refusal rate。
5. latency。
6. cost。
7. safety violation rate。
8. 长尾 case 通过率。

关键是把 loss 放回它的位置：

```text
loss 是重要训练指标，但不是唯一质量指标
```

### 最小 demo：代理指标变好但真实指标变差

下面这个 demo 构造两个候选模型。模型 B 的平均 loss 更低，但 exact match 和 safety pass rate 更差。这就是目标错配的最小例子：代理指标改善，不代表真实目标改善。

```python
models = {
    "model_a": {
        "losses": [0.8, 0.9, 0.7, 0.8],
        "exact_match": [1, 1, 0, 1],
        "safety_pass": [1, 1, 1, 1],
    },
    "model_b": {
        "losses": [0.5, 0.6, 0.4, 0.5],
        "exact_match": [1, 0, 0, 1],
        "safety_pass": [1, 0, 1, 0],
    },
}


def summarize(metrics):
    return {
        "loss": sum(metrics["losses"]) / len(metrics["losses"]),
        "exact_match": sum(metrics["exact_match"]) / len(metrics["exact_match"]),
        "safety_pass": sum(metrics["safety_pass"]) / len(metrics["safety_pass"]),
    }


for name, metrics in models.items():
    print(name, summarize(metrics))
```

真实项目中不会用这么小的数据判断模型，但这个例子能说明：任何单一指标都可能误导你，必须同时看 loss、任务指标、安全指标和关键 bad cases。

### 面试中怎么讲

如果面试官问“为什么 loss 降低不一定代表模型更好”，可以这样回答：

```text
因为训练 loss 通常是最终目标的代理指标。语言模型预训练优化的是 token-level next-token likelihood，但真实任务往往是 sequence-level 或 behavior-level，例如答案正确性、工具调用合法性、安全性和人类偏好。loss 降低说明模型更好地拟合了训练或验证分布，但不保证在下游任务、分布外样本、人类偏好和安全行为上都改善。因此需要结合任务指标、人类评估、错误分析和安全评估综合判断。
```

如果面试官问“RLHF 为什么会有 reward hacking”，可以这样回答：

```text
RLHF 优化的是 reward model 给出的分数，而 reward model 只是人类偏好的近似。如果 reward model 学到了表面模式或覆盖不足，策略模型就可能找到提高 reward 的捷径，例如更啰嗦、更迎合、更自信，甚至编造引用。这就是代理目标和真实目标之间的错配。缓解方法包括更好的偏好数据、多维评估、KL 约束、红队测试和人工抽检。
```

### 真实项目中的坑

#### 坑 1：只看 training loss

training loss 下降可能只是更好拟合训练数据，不代表泛化、偏好和安全都提升。

#### 坑 2：只看 benchmark 平均分

平均分可能掩盖关键失败，比如工具调用格式错、安全边界退化或长尾场景失败。

#### 坑 3：把 reward model 当成真理

reward model 是偏好近似器，本身会犯错，也会被策略模型利用。

#### 坑 4：SFT 数据风格污染输出

如果 SFT 数据过度模板化，模型输出会变得僵硬；如果拒答样本过多，模型可能过度拒答。

#### 坑 5：优化一个指标导致另一个指标退化

例如提高安全拒答率同时降低有用性，或提高格式合法率同时降低答案质量。

### 面试官会怎么问

#### 问题 1：什么是优化目标错配？

回答框架：

1. 训练时优化的是代理目标。
2. 真实目标通常更复杂、更难直接优化。
3. 两者不一致时就会出现目标错配。
4. LLM 中常见于 pretraining、SFT、RLHF、DPO 和 benchmark 优化。

#### 问题 2：为什么 token-level loss 不能完全代表任务质量？

回答框架：

1. token-level loss 看每个 token 的概率。
2. 真实任务常常看整个序列是否正确。
3. 数学、代码、工具调用中一个局部错误可能导致整体失败。
4. 所以还要看 exact match、pass@k、human eval 和任务指标。

#### 问题 3：SFT 有哪些目标错配？

回答框架：

1. SFT 最大化参考答案概率。
2. 但同一指令可能有多个好答案。
3. 参考答案不一定最优，数据风格会影响输出。
4. SFT 主要学会模仿，不直接优化人类偏好。

#### 问题 4：Goodhart's Law 和 LLM 有什么关系？

回答框架：

1. 当指标被强优化时，模型可能找到提高指标但不改善真实目标的方法。
2. benchmark、reward 分数、安全拒答率都可能被 Goodhart。
3. 因此需要多维评估和错误分析，不能只相信单一指标。

#### 问题 5：如何缓解目标错配？

回答框架：

1. 设计更接近真实目标的数据和 loss。
2. 使用多维评估和 human eval。
3. 对长尾、高风险和分布外场景单独评估。
4. 用 KL 约束、verifier、工具验证等方式减少优化偏移。
5. 做 error analysis 和数据回流。

### 常见误区

1. 误区：loss 越低，模型一定越好。
   纠正：loss 是代理指标，不等于所有下游能力和行为质量。

2. 误区：RLHF 已经直接优化人类偏好，所以没有错配。
   纠正：RLHF 优化的是 reward model，reward model 只是偏好的近似。

3. 误区：benchmark 分数高就说明模型真实能力强。
   纠正：benchmark 可能覆盖不足、污染或被过拟合。

4. 误区：安全指标提升一定是好事。
   纠正：如果以过度拒答换来安全分数，用户有用性可能下降。

5. 误区：目标错配只发生在 RLHF。
   纠正：预训练、SFT、DPO、评估和产品指标都会有目标错配。

### 小练习

1. 举一个 token-level loss 降低但任务准确率下降的例子。
2. 用自己的话解释 Goodhart's Law。
3. 设计一个评估表，同时覆盖正确性、有用性、安全性和格式遵循。
4. 分析 SFT 数据过度模板化可能带来的问题。
5. 举一个 reward hacking 的例子，并说明如何缓解。
6. 用 2 分钟回答“为什么 loss 下降不一定代表模型更好”。

### 本讲总结

本讲最重要的结论：

1. 损失函数是训练时可优化的代理目标，不等于真实产品目标或面试关心的能力。
2. LLM 中常见目标错配包括 token-level loss 与 sequence-level 任务质量错配、偏好分数与真实人类满意度错配、benchmark 与真实场景错配。
3. 预训练、SFT、RLHF 和 DPO 都在不同层面处理目标错配，也会引入新的错配。
4. Goodhart's Law 和 reward hacking 是理解代理指标风险的关键概念。
5. 真实工程中要同时看 loss、任务指标、人类评估、安全评估、长尾 case 和错误分析。
6. 面试中讲目标错配时，核心是说明“优化了什么、没优化什么、可能牺牲什么、如何评估副作用”。

## 第 7 讲：归一化、初始化与训练稳定性

### 本讲目标

大模型训练不是只要写出 Transformer 结构、定义好 loss、调用 AdamW 就能稳定跑起来。模型越深、参数越多、训练 token 越多，训练稳定性越重要。本讲讨论三个直接影响稳定性的基础问题：归一化、初始化和训练动态控制。

你需要掌握：

1. 为什么深层网络训练会不稳定。
2. 激活值和梯度为什么会爆炸或消失。
3. 初始化的核心目标是什么。
4. LayerNorm、RMSNorm、BatchNorm 的区别。
5. Pre-LN 和 Post-LN Transformer 的训练差异。
6. residual scaling、warmup、gradient clipping 对稳定性的作用。
7. 真实大模型训练中如何排查 loss spike 和 NaN。

### 为什么训练稳定性重要

训练稳定性指的是：模型在训练过程中 loss、梯度、激活值、参数更新和评估指标保持在可控范围内，而不是频繁出现发散、NaN、loss spike 或能力突然退化。

对小模型来说，训练不稳定可能只是多调几次学习率。

对大模型来说，训练不稳定意味着：

1. 巨额算力浪费。
2. checkpoint 不可用。
3. 训练中断难以恢复。
4. 很难判断是数据、模型还是系统问题。
5. 后续 SFT、RLHF 或部署都会受影响。

所以大模型训练中，稳定性本身就是核心工程能力。

### 深层网络为什么容易不稳定

深层网络由很多层函数复合而成：

```text
h_L = f_L(f_{L-1}(...f_1(x)))
```

前向传播时，如果每层都让激活值略微放大，很多层之后激活可能爆炸。

如果每层都让激活值略微缩小，很多层之后信号可能消失。

反向传播也类似。

梯度需要经过链式法则层层相乘：

```text
∂L/∂h_1 = ∂L/∂h_L × Π_l ∂h_l/∂h_{l-1}
```

如果这些雅可比矩阵的尺度长期大于 1，梯度可能爆炸。

如果长期小于 1，梯度可能消失。

Transformer 虽然有 residual connection，但深层堆叠、attention、MLP、归一化、混合精度和大 batch 仍然会带来稳定性问题。

### 初始化的核心目标

初始化不是随便给参数一个随机值。

好的初始化希望满足两个目标：

1. 前向传播时，各层激活值尺度不要快速爆炸或消失。
2. 反向传播时，各层梯度尺度不要快速爆炸或消失。

也就是说，初始化希望让信号在网络中传播时保持合适方差。

如果初始化太大：

1. 激活值可能过大。
2. softmax 可能饱和。
3. 梯度可能爆炸。
4. 混合精度下更容易 overflow。

如果初始化太小：

1. 激活值太弱。
2. 梯度太小。
3. 不同神经元学习信号不足。
4. 训练初期进展慢。

### Xavier 和 He 初始化

经典初始化方法包括 Xavier 初始化和 He 初始化。

Xavier 初始化的直觉是：对于 tanh 或线性附近的激活，让输入和输出方差尽量一致。

常见尺度类似：

```text
Var(W) ≈ 1 / fan_in
```

或同时考虑 fan-in 和 fan-out：

```text
Var(W) ≈ 2 / (fan_in + fan_out)
```

He 初始化主要面向 ReLU 类激活，因为 ReLU 会把一部分激活截断为 0，所以需要更大的方差：

```text
Var(W) ≈ 2 / fan_in
```

Transformer 中常用 GELU 或 SwiGLU，初始化还要结合 residual 分支、层数和具体实现。

面试中不需要死记每个框架默认值，但要讲清楚初始化的目的：

```text
控制前向激活和反向梯度的尺度，让深层网络训练初期处在稳定区域
```

### Transformer 中初始化更复杂的原因

Transformer 不是简单的全连接网络。

它包含：

1. token embedding。
2. attention projection。
3. output projection。
4. MLP up projection。
5. MLP down projection。
6. residual connection。
7. LayerNorm 或 RMSNorm。
8. 可能还有 MoE、LoRA、RoPE、cross attention 等模块。

每一层都会通过 residual 把更新加回主干：

```text
x_{l+1} = x_l + F_l(x_l)
```

如果每个 residual 分支的输出尺度都偏大，层数越深，主干表示可能越不稳定。

因此一些实现会对 residual 分支或特定 projection 做缩放初始化。

直觉是：

```text
层数越深，每一层对主干的扰动就应该更谨慎
```

### 归一化的作用

归一化的目标是控制中间表示的尺度，让训练更稳定。

它通常能带来：

1. 降低激活尺度漂移。
2. 改善梯度传播。
3. 允许更大学习率。
4. 加快收敛。
5. 降低初始化敏感性。

但归一化不是免费午餐。

它也会影响表达能力、计算开销、并行效率和推理行为。

大模型中最常见的是 LayerNorm 和 RMSNorm，而不是 BatchNorm。

### BatchNorm 为什么不常用于 LLM

BatchNorm 会在 batch 维度上统计均值和方差。

在视觉模型中，它曾经非常重要。

但在语言模型中，BatchNorm 不常用，原因包括：

1. 序列长度可变，batch 内 token 分布差异大。
2. 自回归生成时 batch 统计和训练时不一致。
3. 分布式训练下跨设备 batch 统计复杂。
4. 小 batch 或动态 batch 会让统计不稳定。
5. 语言模型更依赖每个 token 自身 hidden state 的归一化。

所以 Transformer 语言模型通常使用 LayerNorm 或 RMSNorm。

### LayerNorm

LayerNorm 对单个样本、单个 token 的 hidden dimension 做归一化。

给定 hidden state：

```text
x ∈ R^d
```

LayerNorm 计算：

```text
μ = mean(x)
σ² = mean((x - μ)²)
y = (x - μ) / sqrt(σ² + ε) * γ + β
```

特点是：

1. 不依赖 batch 统计。
2. 适合变长序列和自回归生成。
3. 每个 token 独立归一化。
4. 有可学习参数 `γ` 和 `β`。

LayerNorm 的核心作用是让每个 token 的 hidden state 尺度更稳定。

### RMSNorm

RMSNorm 可以理解为 LayerNorm 的简化版本。

它不减去均值，只用 root mean square 控制尺度：

```text
rms(x) = sqrt(mean(x²) + ε)
y = x / rms(x) * γ
```

相比 LayerNorm，RMSNorm：

1. 少了均值中心化。
2. 通常计算更简单。
3. 在很多 LLM 中表现稳定。
4. 常用于 LLaMA 类架构。

RMSNorm 的直觉是：

```text
对 Transformer 来说，控制向量尺度往往比强制均值为 0 更关键
```

面试中可以说：RMSNorm 保留了尺度归一化的核心效果，同时减少一部分计算和实现复杂度。

### Pre-LN 和 Post-LN

Transformer 中归一化放在哪里非常关键。

Post-LN 结构大致是：

```text
x_{l+1} = LN(x_l + F(x_l))
```

Pre-LN 结构大致是：

```text
x_{l+1} = x_l + F(LN(x_l))
```

早期 Transformer 使用 Post-LN，但深层模型中 Pre-LN 更常见。

原因是 Pre-LN 的 residual 主路径更直接。

梯度可以沿着 identity path 更稳定地向前传播：

```text
x_l -> x_l + F(LN(x_l))
```

这让深层 Transformer 更容易训练。

### Pre-LN 的优缺点

Pre-LN 的优点：

1. 梯度传播更稳定。
2. 深层模型更容易训练。
3. 对学习率和初始化更不敏感。
4. 训练早期更少发散。

Pre-LN 的缺点：

1. 每层 residual 更新可能相对较弱。
2. 最终输出通常还需要额外 final norm。
3. 在某些设置下，表示尺度可能随深度累积。
4. 训练稳定不代表最终效果一定最优。

Post-LN 的优点是每层输出都被规范化，但深层训练更容易遇到梯度问题。

所以现代 LLM 多采用 Pre-LN 或 RMSNorm + Pre-Norm 变体。

### Residual Connection 与稳定性

残差连接是深层网络能训练起来的关键。

没有残差时，每层都必须完整变换表示，梯度要穿过所有非线性模块。

有残差后，每层只需要学习一个增量：

```text
x_{l+1} = x_l + Δx_l
```

这带来两个好处：

1. 信息可以沿 identity path 传播。
2. 梯度也可以沿 identity path 回传。

但 residual 分支太强也会出问题。

如果每层都加入过大的 `Δx_l`，深层累积后仍然可能不稳定。

所以一些架构会使用 residual scaling，例如按层数缩放 residual 分支。

### 学习率 warmup

大模型训练中，warmup 很常见。

它指训练初期不直接使用峰值学习率，而是从较小学习率逐步升高。

原因是训练刚开始时：

1. 参数还处在随机初始化附近。
2. 优化器动量统计还不稳定。
3. 激活和梯度尺度还没进入稳定区域。
4. 大学习率容易造成 loss spike 或发散。

warmup 的直觉是：

```text
先让模型和优化器进入稳定状态，再使用较大学习率进行高效训练
```

常见策略是 linear warmup，然后接 cosine decay 或其他衰减策略。

### Gradient Clipping

gradient clipping 用来限制梯度范数。

如果梯度范数超过阈值，就按比例缩放：

```text
g <- g * threshold / ||g||
```

它的作用是防止偶发的大梯度把参数更新推到危险区域。

在大模型训练中，大梯度可能来自：

1. 异常 batch。
2. 数据噪声。
3. 长序列。
4. 混合精度 overflow。
5. loss spike。
6. 优化器状态异常。

gradient clipping 不能解决所有问题，但它是稳定训练的重要保险丝。

### 混合精度和数值稳定性

大模型训练常用 FP16、BF16 或混合精度。

混合精度可以显著降低显存和提升吞吐，但也带来数值问题。

FP16 动态范围较小，容易 overflow 或 underflow。

BF16 动态范围更接近 FP32，通常更稳定。

常见数值风险包括：

1. attention score 太大导致 softmax overflow。
2. loss scale 不合适导致梯度 underflow 或 overflow。
3. LayerNorm 方差太小导致除法不稳定。
4. 激活值异常导致 NaN 扩散。
5. optimizer state 精度不足。

所以训练日志里必须监控 NaN、Inf、grad norm、loss scale、激活尺度和参数范数。

### Attention 中的稳定性问题

Attention 里有一个经典缩放：

```text
Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) V
```

为什么要除以 `sqrt(d_k)`？

因为如果 `Q` 和 `K` 的每个维度方差近似为 1，那么点积 `QK^T` 的方差会随维度 `d_k` 增大。

不缩放时，attention logits 可能很大，softmax 进入饱和区。

softmax 饱和会导致：

1. 注意力分布过尖。
2. 梯度变小或不稳定。
3. 训练初期难以优化。

所以 `1/sqrt(d_k)` 是控制 attention logits 尺度的稳定性设计。

### Loss Spike 是什么

loss spike 指训练过程中 loss 突然大幅上升，然后可能恢复，也可能导致发散。

它在大模型训练中并不少见。

可能原因包括：

1. 学习率过高。
2. warmup 太短。
3. 梯度爆炸。
4. 异常数据 batch。
5. 混合精度数值溢出。
6. 分布式通信或 checkpoint 恢复问题。
7. optimizer state 出错。
8. sequence length 或数据分布突然变化。

loss spike 不一定意味着训练彻底失败，但必须认真排查。

如果 spike 后 validation loss 和下游能力恢复正常，可能只是瞬时异常。

如果 spike 后模型能力持续退化，可能需要回滚 checkpoint 或调整训练配置。

### NaN 和 Inf 排查顺序

如果训练出现 NaN 或 Inf，可以按下面顺序排查：

1. 检查输入数据是否有异常 token、空样本、超长样本或非法 label。
2. 检查 loss 是否在某个 batch 突然异常。
3. 检查 learning rate 和 warmup 是否过激。
4. 检查 grad norm 是否爆炸。
5. 检查 mixed precision loss scale 是否异常。
6. 检查 attention mask 是否正确，避免非法位置参与 softmax。
7. 检查 LayerNorm 或 RMSNorm 的 epsilon 是否过小。
8. 检查 optimizer state 是否损坏。
9. 检查分布式训练中某个 rank 是否出现异常。
10. 从最近正常 checkpoint 复现最小失败 batch。

面试中如果能讲出这种排查顺序，会比只说“调小学习率”更像真实工程经验。

### 一个最小代码例子：监控梯度范数

训练稳定性必须可观测。

下面是一个简化的 PyTorch 例子，用来计算梯度范数并做 clipping。

```python
import torch


def train_step(model, optimizer, batch, max_grad_norm=1.0):
    optimizer.zero_grad(set_to_none=True)

    loss = model(**batch).loss
    loss.backward()

    grad_norm = torch.nn.utils.clip_grad_norm_(
        model.parameters(),
        max_norm=max_grad_norm,
    )

    optimizer.step()

    return {
        "loss": float(loss.detach().cpu()),
        "grad_norm": float(grad_norm.detach().cpu()),
    }
```

真实训练中还应记录：

1. learning rate。
2. loss scale。
3. parameter norm。
4. activation norm。
5. tokens per second。
6. 每个 rank 的异常状态。
7. NaN 和 Inf 出现位置。

稳定训练的关键不是“永远不出问题”，而是问题出现时能快速定位。

### 最小 demo：可运行的梯度范数裁剪

上面的 `train_step` 展示了真实训练代码形态。下面这个 demo 更小，可以直接运行，观察 clipping 前后的梯度范数变化。

```python
import torch
import torch.nn as nn


torch.manual_seed(0)

model = nn.Linear(4, 1)
x = torch.randn(8, 4)
y = torch.randn(8, 1) * 100.0

loss = ((model(x) - y) ** 2).mean()
loss.backward()

before = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
after = torch.sqrt(
    sum((p.grad.detach() ** 2).sum() for p in model.parameters())
)

print("grad norm before clipping:", round(before.item(), 4))
print("grad norm after clipping:", round(after.item(), 4))
```

PyTorch 的 `clip_grad_norm_` 会返回裁剪前的总梯度范数，并原地缩放梯度。真实训练中要把这个值记录到日志里，用于定位 loss spike、异常 batch 和数值不稳定。

### 面试中怎么讲

如果面试官问“LayerNorm 和 BatchNorm 为什么在 LLM 中选择不同”，可以这样回答：

```text
BatchNorm 依赖 batch 维度统计，在语言模型中会受到变长序列、自回归生成、动态 batch 和分布式训练的影响，训练和推理统计也不容易一致。LayerNorm 是对单个 token 的 hidden dimension 做归一化，不依赖 batch 统计，更适合 Transformer 语言模型。现代 LLM 中还常用 RMSNorm，它不做均值中心化，只控制 hidden state 的 RMS 尺度，计算更简单，也能保持训练稳定。
```

如果面试官问“Pre-LN 为什么比 Post-LN 更适合深层 Transformer”，可以这样回答：

```text
Post-LN 是先做 residual add 再归一化，深层时梯度传播更容易受每层归一化影响。Pre-LN 把归一化放在子层输入侧，形式是 x + F(LN(x))，这样 residual identity path 更直接，梯度可以更稳定地穿过很多层。因此 Pre-LN 通常更适合训练很深的 Transformer，但它也可能需要 final norm，并且训练稳定不代表所有设置下效果都最优。
```

### 真实项目中的坑

#### 坑 1：只调学习率，不看数据和数值

loss spike 可能来自异常 batch、mask 错误、混合精度 overflow 或分布式问题，不一定只是学习率过大。

#### 坑 2：忽略 grad norm 监控

没有 grad norm，很难判断是梯度爆炸、数据异常还是优化器状态问题。

#### 坑 3：把 NaN 当成随机事件

NaN 通常有触发原因，需要定位到 batch、rank、模块和具体数值操作。

#### 坑 4：混用 FP16 却不监控 loss scale

FP16 动态范围有限，不合适的 loss scale 会导致 underflow 或 overflow。

#### 坑 5：照搬小模型超参数到大模型

模型变深、batch 变大、序列变长后，初始化、warmup、clipping 和归一化策略都可能需要重新调整。

### 面试官会怎么问

#### 问题 1：为什么深层网络会梯度爆炸或消失？

回答框架：

1. 反向传播是多层雅可比矩阵连乘。
2. 如果尺度长期大于 1，梯度会爆炸。
3. 如果尺度长期小于 1，梯度会消失。
4. 初始化、归一化和残差连接都用于缓解这个问题。

#### 问题 2：LayerNorm 和 RMSNorm 有什么区别？

回答框架：

1. LayerNorm 减均值并除以标准差。
2. RMSNorm 不减均值，只用 RMS 控制尺度。
3. RMSNorm 更简单，很多 LLM 中表现稳定。
4. 两者都不依赖 batch 统计，适合语言模型。

#### 问题 3：为什么 Transformer attention 要除以 `sqrt(d_k)`？

回答框架：

1. `QK^T` 点积方差会随维度增大。
2. logits 太大会让 softmax 饱和。
3. softmax 饱和会导致梯度不稳定。
4. 除以 `sqrt(d_k)` 可以控制 logits 尺度。

#### 问题 4：训练中出现 loss spike 怎么排查？

回答框架：

1. 看学习率、warmup 和 grad norm。
2. 查异常 batch、超长样本和数据分布变化。
3. 查混合精度 overflow、NaN、Inf。
4. 查 attention mask、normalization epsilon、optimizer state。
5. 分布式训练还要查具体 rank 和通信状态。

#### 问题 5：gradient clipping 有什么作用？

回答框架：

1. 限制梯度范数，防止异常大梯度导致参数更新过大。
2. 对偶发异常 batch 和训练初期不稳定有保护作用。
3. 它不是根因修复，只是稳定训练的保险丝。

### 常见误区

1. 误区：有了 LayerNorm 就不会训练不稳定。
   纠正：归一化只是稳定性组件之一，还要看初始化、学习率、数据、精度和优化器。

2. 误区：NaN 是硬件随机问题。
   纠正：NaN 往往来自数值 overflow、非法输入、mask 错误或异常梯度，需要系统排查。

3. 误区：BatchNorm 更经典，所以 LLM 也应该用 BatchNorm。
   纠正：BatchNorm 依赖 batch 统计，不适合自回归语言模型的训练和推理形态。

4. 误区：Pre-LN 全面优于 Post-LN。
   纠正：Pre-LN 通常更稳定，但不同架构和训练设置下仍需实验验证。

5. 误区：gradient clipping 会解决梯度爆炸的根因。
   纠正：clipping 只是限制后果，根因可能是学习率、数据、初始化或数值问题。

### 本讲总结

本讲最重要的结论：

1. 大模型训练稳定性取决于初始化、归一化、残差连接、学习率、精度、数据和优化器的共同作用。
2. 初始化的目标是让前向激活和反向梯度保持合适尺度。
3. LayerNorm 和 RMSNorm 不依赖 batch 统计，因此更适合语言模型。
4. Pre-LN 通过更直接的 residual identity path 改善深层 Transformer 的梯度传播。
5. warmup、gradient clipping、residual scaling 和 mixed precision 管理都是稳定训练的重要工具。
6. loss spike、NaN 和 Inf 需要系统排查，不能只靠调小学习率。
7. 面试中讨论训练稳定性时，要同时覆盖数学直觉、架构设计和工程排查流程。

## 第 8 讲：激活函数、门控结构与 SwiGLU

### 本讲目标

Transformer Block 中除了 attention，另一个核心组件是前馈网络，也就是 FFN 或 MLP。很多人只关注 attention，却忽略了 MLP、激活函数和门控结构对模型能力、训练稳定性和计算成本的影响。本讲重点讲激活函数从 ReLU、GELU 到 GLU、SwiGLU 的演进，以及为什么现代 LLM 常使用门控 MLP。

你需要掌握：

1. 激活函数为什么必要。
2. ReLU、GELU、SiLU/Swish 的直觉和差异。
3. Transformer FFN 的基本结构。
4. GLU、GEGLU、SwiGLU 的机制。
5. 为什么 SwiGLU 在现代 LLM 中常见。
6. 门控结构对表达能力和参数量的影响。
7. 面试中如何解释“attention 负责 token mixing，MLP 负责 feature transformation”。

### 为什么需要激活函数

如果神经网络只有线性层，没有非线性激活，那么多层线性变换叠加后仍然只是一个线性变换。

例如：

```text
y = W_3 W_2 W_1 x
```

它可以合并成：

```text
y = W x
```

这意味着网络再深，也不能表达复杂非线性函数。

激活函数的作用是引入非线性，让模型能够拟合复杂关系。

在 LLM 中，这种非线性不仅影响基础拟合能力，还影响：

1. 特征组合能力。
2. 稀疏激活模式。
3. 梯度传播。
4. 训练稳定性。
5. MLP 子层的表达效率。

### Transformer 中 MLP 的作用

一个标准 Transformer Block 通常包含两类子层：

1. attention 子层。
2. MLP/FFN 子层。

可以粗略理解为：

```text
attention 负责 token 之间的信息交互
MLP 负责每个 token 内部的特征变换
```

Self-attention 会让一个 token 聚合其他 token 的信息。

MLP 则对每个 token 的 hidden state 独立做非线性变换。

典型 FFN 写成：

```text
FFN(x) = W_2 activation(W_1 x)
```

其中 `W_1` 通常把维度从 `d_model` 扩展到更大的 `d_ff`，`W_2` 再投影回 `d_model`。

例如：

```text
d_model -> 4 * d_model -> d_model
```

这说明 MLP 子层承担了大量参数和计算量。

在很多 Transformer 中，MLP 参数量甚至超过 attention 参数量。

### ReLU

ReLU 是经典激活函数：

```text
ReLU(x) = max(0, x)
```

它的优点：

1. 计算简单。
2. 正区间梯度不饱和。
3. 引入稀疏激活。
4. 比 sigmoid/tanh 更容易训练深层网络。

缺点：

1. 负半轴梯度为 0。
2. 可能出现 dead ReLU。
3. 在 Transformer LLM 中通常不是最优选择。

ReLU 曾经是深度学习中非常重要的默认选择，但现代 LLM 更常用 GELU、SiLU 或 SwiGLU。

### GELU

GELU 是 Gaussian Error Linear Unit。

它的形式可以写成：

```text
GELU(x) = x * Φ(x)
```

其中 `Φ(x)` 是标准正态分布的累积分布函数。

直觉上，GELU 不是像 ReLU 那样硬截断负值，而是根据输入大小平滑地决定保留多少。

近似实现常写为：

```text
GELU(x) ≈ 0.5x(1 + tanh(sqrt(2/π)(x + 0.044715x^3)))
```

GELU 的特点：

1. 平滑。
2. 对小负值不会完全置零。
3. 适合 Transformer 这类大规模预训练模型。
4. BERT、GPT 系列早期模型中常见。

相比 ReLU，GELU 的门控更柔和，训练中通常表现更好。

### SiLU 和 Swish

SiLU，也常称为 Swish 的一种形式：

```text
SiLU(x) = x * sigmoid(x)
```

它和 GELU 很像，都是输入乘上一个平滑门控。

区别是：

1. GELU 使用正态分布 CDF。
2. SiLU 使用 sigmoid。

SiLU 的特点：

1. 平滑。
2. 小负值仍有输出。
3. 大正值近似线性。
4. 在很多现代网络和 LLM 门控结构中表现好。

SwiGLU 里的 `Swi` 就来自 Swish/SiLU。

### 为什么平滑激活常常更适合大模型

平滑激活函数的优势不是绝对的，但在大模型中常见。

可能原因包括：

1. 梯度变化更连续。
2. 训练动态更平滑。
3. 对小负值保留部分信息。
4. 避免 ReLU 的硬截断。
5. 与归一化和残差结构配合更稳定。

这不意味着 ReLU 不好，而是现代 LLM 的经验表明，GELU、SiLU 和门控变体通常更有效。

### GLU 的基本思想

GLU 是 Gated Linear Unit。

普通 FFN 是：

```text
FFN(x) = W_2 activation(W_1 x)
```

GLU 的思想是把中间层拆成两条分支：

```text
a = W_a x
g = W_g x
GLU(x) = a ⊙ sigmoid(g)
```

其中 `⊙` 是逐元素乘法。

可以理解为：

1. `a` 是内容分支。
2. `sigmoid(g)` 是门控分支。
3. 门控分支决定哪些特征通过、哪些特征抑制。

GLU 的核心直觉是：

```text
模型不只是变换特征，还能动态控制特征是否被激活
```

### GEGLU

GEGLU 把 GLU 中的 sigmoid 门控换成 GELU 相关形式。

一种常见写法是：

```text
GEGLU(x) = (W_a x) ⊙ GELU(W_g x)
```

它相比普通 GELU FFN，多了显式门控乘法。

直觉上，GEGLU 同时拥有：

1. GELU 的平滑非线性。
2. GLU 的特征门控能力。

很多研究和实践发现，GLU 变体在 Transformer FFN 中可以提升质量。

### SwiGLU

SwiGLU 使用 SiLU/Swish 作为门控：

```text
SwiGLU(x) = (W_a x) ⊙ SiLU(W_g x)
```

然后再通过输出投影：

```text
FFN(x) = W_o [(W_a x) ⊙ SiLU(W_g x)]
```

也可以写成：

```text
up = W_up x
gate = W_gate x
hidden = SiLU(gate) ⊙ up
out = W_down hidden
```

这就是很多 LLaMA 类模型中的 MLP 结构。

SwiGLU 的优势通常包括：

1. 表达能力强。
2. 门控机制更灵活。
3. 平滑激活有利于训练。
4. 在相近计算预算下效果常优于普通 GELU FFN。

### SwiGLU 和普通 FFN 的参数量

普通 FFN 通常是：

```text
d_model -> d_ff -> d_model
```

参数量大约是：

```text
d_model * d_ff + d_ff * d_model = 2 * d_model * d_ff
```

SwiGLU 有两条输入投影分支和一个输出投影：

```text
W_up: d_model -> d_hidden
W_gate: d_model -> d_hidden
W_down: d_hidden -> d_model
```

参数量大约是：

```text
3 * d_model * d_hidden
```

如果直接设置 `d_hidden = 4 * d_model`，参数量会比普通 FFN 多很多。

所以实践中 SwiGLU 常把 hidden dimension 调小，例如使用接近：

```text
d_hidden ≈ 8/3 * d_model
```

这样参数量和计算量能接近普通 `4 * d_model` FFN。

面试中要注意：

```text
SwiGLU 不是免费提升，它通过门控结构提高表达效率，但需要合理控制 hidden dimension
```

### 门控结构为什么有效

门控结构有效的直觉可以从几个角度理解。

第一，门控提供动态特征选择。

不同输入 token 可以激活不同特征通道。

第二，乘法交互增强表达能力。

普通激活大多是对单个线性投影做非线性。

门控结构引入：

```text
(W_a x) ⊙ activation(W_g x)
```

这是一种输入相关的乘法调制。

第三，门控能让 MLP 更像条件计算。

虽然它不是 MoE 那种离散专家路由，但每个维度的通过强度都由输入决定。

第四，门控和大规模预训练很匹配。

语言数据中存在大量上下文相关模式，门控有助于模型根据上下文选择不同特征组合。

### Attention 和 MLP 的分工

Transformer 中常见说法是：

```text
attention mixes tokens, MLP transforms features
```

更具体地说：

1. attention 让当前位置读取其他位置的信息。
2. attention 决定信息从哪些 token 流向当前 token。
3. MLP 对每个 token 的表示做非线性特征变换。
4. MLP 扩展 hidden dimension，形成更丰富的中间特征。
5. 门控 MLP 决定哪些中间特征被保留或抑制。

所以不要把 Transformer 的能力全部归功于 attention。

attention 负责上下文交互，MLP 负责大量非线性计算和知识/特征变换。

### MLP 中可能存储知识吗

有一种常见观点认为，Transformer 的 FFN/MLP 层可能像 key-value memory 一样存储大量事实和模式。

这个说法可以帮助理解，但不能过度简化。

直觉上：

1. attention 负责从上下文中取信息。
2. MLP 对 hidden state 进行高维非线性变换。
3. 某些 neuron 或特征方向可能对应特定模式、实体或属性。
4. 事实知识不只存在 MLP，也分布在 embedding、attention、MLP 和整体表示中。

面试中可以谨慎表达：

```text
MLP 子层承担了大量参数和非线性变换，确实可能编码许多知识和模式，但知识通常是分布式存储的，不能简单说只存在某一层或某个神经元里。
```

### 激活函数和训练稳定性

激活函数也会影响训练稳定性。

例如：

1. sigmoid/tanh 在大输入下容易饱和，梯度变小。
2. ReLU 负半轴梯度为 0，可能出现 dead neuron。
3. GELU/SiLU 更平滑，梯度变化更连续。
4. 门控结构中的乘法可能提高表达能力，也可能带来尺度控制问题。

因此，激活函数选择要和初始化、归一化、残差缩放、学习率一起考虑。

如果门控分支输出尺度过大，逐元素乘法可能放大 hidden activation。

这也是为什么现代 LLM 的 MLP 设计通常和 RMSNorm、Pre-LN、合适初始化配套出现。

### 一个最小 PyTorch 实现

下面是一个简化版 SwiGLU MLP。

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class SwiGLUMLP(nn.Module):
    def __init__(self, d_model, hidden_dim):
        super().__init__()
        self.up_proj = nn.Linear(d_model, hidden_dim, bias=False)
        self.gate_proj = nn.Linear(d_model, hidden_dim, bias=False)
        self.down_proj = nn.Linear(hidden_dim, d_model, bias=False)

    def forward(self, x):
        up = self.up_proj(x)
        gate = self.gate_proj(x)
        hidden = up * F.silu(gate)
        return self.down_proj(hidden)
```

输入输出 shape：

```text
x:      [batch, seq_len, d_model]
up:     [batch, seq_len, hidden_dim]
gate:   [batch, seq_len, hidden_dim]
hidden: [batch, seq_len, hidden_dim]
out:    [batch, seq_len, d_model]
```

这段代码体现了 SwiGLU 的核心：

```text
一条内容分支 × 一条平滑门控分支
```

### 最小 demo：验证 SwiGLU shape 和参数量

下面这个 demo 直接实例化普通 GELU FFN 和 SwiGLU MLP。它验证 SwiGLU 输入输出 shape 不变，并比较参数量。为了公平，SwiGLU 的 `hidden_dim` 通常会小于普通 FFN 的 `4 * d_model`。

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class GeluFFN(nn.Module):
    def __init__(self, d_model, hidden_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(d_model, hidden_dim, bias=False),
            nn.GELU(),
            nn.Linear(hidden_dim, d_model, bias=False),
        )

    def forward(self, x):
        return self.net(x)


class SwiGLUMLP(nn.Module):
    def __init__(self, d_model, hidden_dim):
        super().__init__()
        self.up_proj = nn.Linear(d_model, hidden_dim, bias=False)
        self.gate_proj = nn.Linear(d_model, hidden_dim, bias=False)
        self.down_proj = nn.Linear(hidden_dim, d_model, bias=False)

    def forward(self, x):
        up = self.up_proj(x)
        gate = self.gate_proj(x)
        hidden = up * F.silu(gate)
        return self.down_proj(hidden)


def count_params(module):
    return sum(p.numel() for p in module.parameters())


d_model = 12
gelu_ffn = GeluFFN(d_model, hidden_dim=4 * d_model)
swiglu = SwiGLUMLP(d_model, hidden_dim=32)

x = torch.randn(2, 5, d_model)
y = swiglu(x)

print("input shape:", x.shape)
print("output shape:", y.shape)
print("GELU FFN params:", count_params(gelu_ffn))
print("SwiGLU params:", count_params(swiglu))
```

如果不调小 `hidden_dim`，SwiGLU 会因为多一条 gate projection 而显著增加参数和计算量。这就是为什么讨论 SwiGLU 时必须同时讲质量、参数量、FLOPs 和 kernel 支持。

### 面试中怎么讲

如果面试官问“为什么现代 LLM 常用 SwiGLU”，可以这样回答：

```text
SwiGLU 是一种门控 MLP 结构，它把普通 FFN 的单分支非线性变换改成内容分支和门控分支的逐元素乘法，其中门控分支使用 SiLU/Swish。这样模型可以根据输入动态调节哪些特征通过，表达能力更强。相比普通 GELU FFN，SwiGLU 在很多 Transformer LLM 中能以相近计算预算获得更好的效果，但需要调小 hidden dimension 来控制参数量和 FLOPs。
```

如果面试官问“attention 和 MLP 分别负责什么”，可以这样回答：

```text
一个简化理解是 attention 负责 token mixing，也就是让当前位置聚合上下文中其他 token 的信息；MLP 负责 feature transformation，也就是对每个 token 的 hidden state 做非线性变换和特征组合。两者是互补的，LLM 的能力不只来自 attention，MLP 层也承担了大量参数、计算和知识模式编码。
```

### 真实项目中的坑

#### 坑 1：只改激活函数，不控制参数量

把 GELU FFN 直接换成 SwiGLU，如果 hidden dimension 不调整，参数量和计算量会明显增加，比较不公平。

#### 坑 2：忽略门控分支的数值尺度

门控乘法可能放大激活，需要结合初始化、norm 和学习率观察稳定性。

#### 坑 3：以为 attention 是 Transformer 唯一核心

MLP 子层有大量参数和计算，对模型能力非常关键。

#### 坑 4：把某种激活函数绝对化

GELU、SiLU、SwiGLU 的效果依赖模型规模、数据、超参和实现，不能脱离实验结论。

#### 坑 5：只看最终分数，不看延迟和吞吐

门控 MLP 可能提升质量，但也会影响推理延迟、显存和 kernel 优化。

### 面试官会怎么问

#### 问题 1：为什么神经网络需要激活函数？

回答框架：

1. 没有非线性，多层线性网络仍等价于一层线性变换。
2. 激活函数引入非线性表达能力。
3. 在 Transformer 中，MLP 激活负责每个 token 的特征变换。

#### 问题 2：ReLU、GELU、SiLU 有什么区别？

回答框架：

1. ReLU 是硬截断负值，简单高效但可能 dead ReLU。
2. GELU 用 `x * Φ(x)` 平滑地保留输入。
3. SiLU 用 `x * sigmoid(x)`，也是平滑门控形式。
4. 现代 LLM 常偏好 GELU、SiLU 或门控变体。

#### 问题 3：SwiGLU 的公式是什么？

回答框架：

1. 两条输入投影分支：`up = W_up x`，`gate = W_gate x`。
2. 使用 SiLU 激活门控分支。
3. 中间表示是 `up * SiLU(gate)`。
4. 最后通过 `W_down` 投影回 `d_model`。

#### 问题 4：SwiGLU 为什么要调整 hidden dimension？

回答框架：

1. 普通 FFN 有两个矩阵，SwiGLU 有三个矩阵。
2. 如果 hidden dimension 不变，参数量和计算量会增加。
3. 实践中常把 SwiGLU hidden dimension 调小，使总参数量接近普通 FFN。

#### 问题 5：MLP 在 Transformer 中有什么作用？

回答框架：

1. attention 做 token 间信息交互。
2. MLP 对每个 token 做非线性特征变换。
3. MLP 扩展维度再压回，增强表示能力。
4. 门控 MLP 还能动态选择特征通道。

### 常见误区

1. 误区：Transformer 的能力主要来自 attention，MLP 不重要。
   纠正：MLP 承担大量参数和非线性计算，对能力非常关键。

2. 误区：SwiGLU 一定比 GELU 好，而且没有代价。
   纠正：SwiGLU 常表现更好，但有参数和计算代价，需要合理设置 hidden dimension。

3. 误区：门控结构只是一种激活函数替换。
   纠正：门控结构引入了内容分支和门控分支的乘法交互，表达能力不同于普通单分支激活。

4. 误区：激活函数只影响精度，不影响训练稳定性。
   纠正：激活函数会影响梯度、激活尺度、饱和行为和混合精度稳定性。

5. 误区：MLP 中的知识可以简单定位到某个神经元。
   纠正：知识通常是分布式存储的，MLP 很重要但不是唯一位置。

2. 写出 ReLU、GELU、SiLU 的公式或近似形式。
3. 手写一个简化版 SwiGLU MLP，并标注每一步 shape。
4. 计算普通 FFN 和 SwiGLU 在不同 hidden dimension 下的参数量。

### 本讲总结

本讲最重要的结论：

1. 激活函数为神经网络引入非线性，没有激活函数的深层线性网络仍然只是线性模型。
2. Transformer MLP 子层对每个 token 独立做非线性特征变换，承担大量参数和计算。
3. GELU 和 SiLU 比 ReLU 更平滑，常用于现代 Transformer。
4. GLU、GEGLU、SwiGLU 通过门控分支引入动态特征选择和乘法交互。
5. SwiGLU 常见于现代 LLM，但需要调整 hidden dimension 控制参数量和 FLOPs。
6. attention 和 MLP 是互补关系，不能只关注 attention。
7. 激活函数和门控结构会同时影响模型效果、训练稳定性和推理成本。

## 第 9 讲：优化器进阶：AdamW、Adafactor 与 Lion

### 本讲目标

优化器决定模型参数如何根据梯度更新。第一册里我们已经讲过 SGD、Adam 和 AdamW 的基础。本讲进一步讨论大模型训练中为什么常用 AdamW，为什么 Adafactor 能节省显存，Lion 这类新优化器有什么思路，以及面试中如何从稳定性、显存、泛化和工程成本角度比较优化器。

你需要掌握：

1. SGD、Momentum、Adam 的核心差异。
2. AdamW 为什么要 decoupled weight decay。
3. AdamW 在 LLM 训练中的常见配置和风险。
4. Adafactor 如何降低优化器状态显存。
5. Lion 的符号更新思想。
6. 优化器状态为什么是大模型训练的重要成本。
7. 如何排查优化器相关的训练不稳定。

### 优化器到底在做什么

训练神经网络时，我们希望最小化损失函数：

```text
min_θ L(θ)
```

最基础的梯度下降更新是：

```text
θ_{t+1} = θ_t - η ∇L(θ_t)
```

其中：

1. `θ` 是参数。
2. `η` 是学习率。
3. `∇L(θ_t)` 是当前梯度。

优化器的核心问题是：

```text
给定当前梯度和历史信息，参数应该往哪个方向走、走多远？
```

大模型中，这个问题很复杂，因为：

1. 参数量巨大。
2. 梯度噪声明显。
3. 不同参数尺度差异大。
4. 训练成本极高，不能随便试错。
5. 混合精度、分布式和 optimizer state 都会影响稳定性。

### SGD 和 Momentum

SGD 使用 mini-batch 梯度更新参数：

```text
θ_{t+1} = θ_t - η g_t
```

其中 `g_t` 是当前 batch 的梯度。

SGD 的优点是简单、状态少、泛化表现常常不错。

但缺点是：

1. 对学习率敏感。
2. 在病态曲率下收敛慢。
3. 梯度噪声大时更新方向抖动。
4. 大模型训练中调参成本高。

Momentum 引入动量，累积历史梯度方向：

```text
v_t = β v_{t-1} + g_t
θ_{t+1} = θ_t - η v_t
```

直觉是：

```text
如果多个 step 的梯度方向一致，就加速；如果来回震荡，就相互抵消
```

这能改善 SGD 的收敛稳定性。

### Adam 的核心思想

Adam 同时维护一阶矩和二阶矩。

一阶矩是梯度的指数滑动平均：

```text
m_t = β1 m_{t-1} + (1 - β1) g_t
```

二阶矩是梯度平方的指数滑动平均：

```text
v_t = β2 v_{t-1} + (1 - β2) g_t²
```

更新时用二阶矩做自适应缩放：

```text
θ_{t+1} = θ_t - η m_t / (sqrt(v_t) + ε)
```

Adam 的直觉是：

1. 用 `m_t` 平滑梯度方向。
2. 用 `v_t` 估计每个参数的梯度尺度。
3. 梯度尺度大的参数步子小一点。
4. 梯度尺度小的参数步子大一点。

这让 Adam 在深度网络中通常比原始 SGD 更容易调。

### Bias Correction

Adam 初始化时 `m_0=0`，`v_0=0`。

训练早期，滑动平均会偏向 0。

所以 Adam 使用 bias correction：

```text
m_hat_t = m_t / (1 - β1^t)
v_hat_t = v_t / (1 - β2^t)
```

然后更新：

```text
θ_{t+1} = θ_t - η m_hat_t / (sqrt(v_hat_t) + ε)
```

在大模型训练中，训练早期本来就容易不稳定，所以 bias correction、warmup 和学习率调度会一起影响训练动态。

### Adam 的问题

Adam 很强，但也有问题。

常见问题包括：

1. 优化器状态显存大。
2. weight decay 和自适应缩放耦合时不够理想。
3. 对 `β1`、`β2`、`ε`、学习率敏感。
4. 有时泛化不如 SGD 类方法。
5. 在混合精度和分布式训练中状态管理复杂。

对 LLM 来说，最现实的问题之一是 optimizer state 显存。

如果模型参数量是 `N`，Adam 需要保存：

1. 参数本身。
2. 梯度。
3. 一阶矩 `m`。
4. 二阶矩 `v`。

如果还有 FP32 master weights，显存压力会更大。

所以优化器选择不仅是算法问题，也是系统问题。

### Weight Decay 和 L2 正则的区别

在普通 SGD 中，L2 正则和 weight decay 常常等价。

如果 loss 加上 L2 正则：

```text
L_total = L + λ ||θ||²
```

梯度会多一项：

```text
g_total = g + 2λθ
```

更新类似于：

```text
θ <- θ - ηg - ηλθ
```

这可以理解为每步把参数往 0 拉一点。

但在 Adam 这种自适应优化器中，如果把 L2 正则直接加到梯度里，这个正则项也会被 `sqrt(v)` 自适应缩放。

这会让 weight decay 的效果依赖每个参数的梯度历史，行为不够纯粹。

### AdamW 的核心：Decoupled Weight Decay

AdamW 的关键改动是把 weight decay 从梯度更新中解耦。

简化理解：

```text
Adam step:      θ <- θ - η * adaptive_update
Weight decay:   θ <- θ - η * λ * θ
```

也就是先按 Adam 的自适应方向更新，再单独做参数衰减。

这样 weight decay 不再被二阶矩缩放扭曲。

AdamW 的优势：

1. 正则化行为更清晰。
2. 大模型训练中更稳定。
3. 比 Adam + L2 更符合 weight decay 的原始直觉。
4. 已成为 Transformer/LLM 训练中的常见默认选择。

面试中如果被问 Adam 和 AdamW 的区别，核心就是：

```text
AdamW 将 weight decay 从梯度中解耦，避免被 Adam 的自适应缩放影响
```

### LLM 中 AdamW 的常见设置

大模型训练中 AdamW 常见超参数包括：

1. learning rate。
2. `β1`。
3. `β2`。
4. `ε`。
5. weight decay。
6. warmup steps。
7. learning rate schedule。
8. gradient clipping。

常见经验是：

1. `β1` 常接近 0.9。
2. `β2` 常接近 0.95、0.98 或 0.999，取决于模型和训练设置。
3. weight decay 通常不作用于 bias 和 norm 参数。
4. 学习率必须和 batch size、模型规模、warmup 配合调。
5. 训练稳定性强依赖 grad norm 和 loss spike 监控。

不要把这些数字背成绝对答案。

面试中更重要的是讲出：

```text
AdamW 有很多相互耦合的超参数，必须结合模型规模、batch、数据、精度和调度策略一起看
```

### 为什么 norm 和 bias 通常不做 weight decay

很多训练配置会把 LayerNorm/RMSNorm 参数和 bias 排除在 weight decay 之外。

原因是：

1. norm 参数主要控制尺度和平移，不像普通权重矩阵那样需要同等衰减。
2. 对 bias 做 decay 通常收益不明显，甚至可能伤害训练。
3. 大模型中这些参数占比小，但对稳定性和表达有影响。

常见做法是把参数分组：

```text
decay:    Linear weight
no_decay: bias, norm weight
```

这是一种工程经验，不是数学定律。

### Adafactor 为什么出现

AdamW 的主要问题之一是 optimizer state 太大。

对每个参数，AdamW 需要保存 `m` 和 `v`。

如果模型参数非常多，优化器状态可能占据巨大显存。

Adafactor 的目标是减少二阶矩状态的显存。

核心思想是：对于矩阵参数，不完整保存每个元素的二阶矩，而是用行统计和列统计近似。

假设参数矩阵是：

```text
W ∈ R^{r × c}
```

Adam 需要保存同样大小的二阶矩：

```text
V ∈ R^{r × c}
```

Adafactor 用两个向量近似：

```text
R ∈ R^r
C ∈ R^c
```

这样状态从 `r*c` 降到 `r+c`。

对巨大矩阵来说，节省非常明显。

### Adafactor 的优缺点

Adafactor 的优点：

1. 显著降低优化器状态显存。
2. 适合超大模型或显存受限训练。
3. 曾在 T5 等大规模模型训练中使用。
4. 对大矩阵参数尤其有效。

缺点：

1. 近似二阶矩可能不如 AdamW 精确。
2. 超参数和学习率策略需要小心。
3. 某些设置下稳定性和最终效果可能不如 AdamW。
4. 实现细节更复杂。

所以 Adafactor 的定位不是“全面替代 AdamW”，而是：

```text
在优化器状态显存成为瓶颈时，用近似二阶矩换取更低内存成本
```

### Lion 的基本思想

Lion 是一种较新的优化器思路。

它的核心特点是使用动量方向的符号来更新参数，而不是像 Adam 那样用二阶矩进行缩放。

简化形式可以理解为：

```text
m_t = β1 m_{t-1} + (1 - β1) g_t
θ_{t+1} = θ_t - η * sign(m_t)
```

实际 Lion 还有更具体的动量更新设计。

直觉是：

1. 更新方向主要看动量的符号。
2. 每个参数每步更新幅度更一致。
3. 不需要保存 Adam 那样的二阶矩。
4. optimizer state 可以更少。

Lion 的潜在优势：

1. 状态更少。
2. 计算简单。
3. 某些视觉和语言任务中表现不错。

潜在风险：

1. 对学习率和 weight decay 可能更敏感。
2. 符号更新可能带来不同训练动态。
3. 在 LLM 预训练中是否稳定，需要具体规模和设置验证。

面试中不要夸大 Lion。

更稳妥的说法是：Lion 提供了减少优化器状态、使用符号动量更新的思路，但在大规模 LLM 训练中是否优于 AdamW 需要实验证据。

### 优化器状态的显存成本

假设一个模型有 `N` 个参数。

如果参数用 BF16，每个参数 2 字节。

但优化器常需要 FP32 状态，每个状态 4 字节。

AdamW 至少需要：

1. 参数：约 `2N` 字节。
2. 梯度：约 `2N` 或 `4N` 字节，取决于实现。
3. 一阶矩：约 `4N` 字节。
4. 二阶矩：约 `4N` 字节。
5. FP32 master weights：可能再加 `4N` 字节。

所以训练显存远不只是模型参数本身。

这也是为什么大模型训练需要：

1. ZeRO optimizer。
2. optimizer state sharding。
3. mixed precision。
4. activation checkpointing。
5. Adafactor 等低状态优化器。
6. CPU/NVMe offload。

优化器选择和分布式系统设计是强相关的。

### 学习率调度和优化器不可分开看

优化器不是孤立工作的。

同一个 AdamW，如果学习率调度不同，表现可能完全不同。

LLM 常见 schedule 包括：

1. linear warmup + cosine decay。
2. linear warmup + linear decay。
3. constant with warmup。
4. inverse square root decay。

warmup 解决训练早期不稳定。

decay 控制训练后期更新幅度，帮助收敛。

如果学习率过高：

1. loss spike。
2. 梯度爆炸。
3. NaN。
4. checkpoint 退化。

如果学习率过低：

1. 收敛慢。
2. 算力浪费。
3. 欠训练。
4. 模型能力不足。

所以优化器调参必须和学习率曲线一起讨论。

### Batch Size 和优化器

batch size 会影响梯度噪声。

batch 越大，梯度估计通常越稳定，但每次更新看到的数据更多，更新次数可能减少。

大 batch 训练常需要调整学习率。

常见经验是：

```text
batch size 增大时，学习率也可能需要增大，但不是无限线性增大
```

大模型训练中更常讨论 global batch size 和 token batch size。

例如：

```text
global tokens per step = micro_batch * sequence_length * data_parallel_size * grad_accum_steps
```

优化器看到的是累计后的梯度。

如果 gradient accumulation、sequence length 或数据并行规模变化，优化器动态也会变化。

### 优化器相关的不稳定现象

优化器设置不合适可能导致：

1. loss 不下降。
2. loss spike。
3. validation loss 震荡。
4. grad norm 长期异常。
5. 参数范数持续膨胀。
6. 训练早期直接 NaN。
7. 后期能力退化。
8. 某些模块权重异常变大或变小。

常见原因包括：

1. learning rate 过高。
2. warmup 太短。
3. `β2` 不适合当前 batch 和梯度噪声。
4. weight decay 过强或过弱。
5. norm/bias 参数错误地 decay。
6. gradient clipping 阈值不合适。
7. mixed precision state 处理错误。
8. checkpoint 恢复时 optimizer state 不匹配。

### 排查优化器问题的顺序

如果怀疑优化器有问题，可以按下面顺序排查：

1. 确认学习率实际曲线是否符合预期。
2. 检查 warmup steps 是否足够。
3. 查看 grad norm、parameter norm、update norm。
4. 检查 AdamW 参数分组，确认 norm 和 bias 是否 no_decay。
5. 检查 `β1`、`β2`、`ε` 是否和训练规模匹配。
6. 检查 gradient clipping 是否生效。
7. 检查混合精度下 optimizer state 是否为预期精度。
8. 恢复 checkpoint 时确认 optimizer state、scheduler state、step count 一致。
9. 对比小规模复现实验，排除数据和模型结构问题。

工程中最好记录 update-to-weight ratio：

```text
||Δθ|| / ||θ||
```

它能帮助判断参数更新相对参数规模是否过大或过小。

### 一个最小代码例子：AdamW 参数分组

下面是常见的 AdamW 参数分组写法。

```python
import torch


def build_adamw(model, lr, weight_decay):
    decay_params = []
    no_decay_params = []

    for name, param in model.named_parameters():
        if not param.requires_grad:
            continue

        if name.endswith("bias") or "norm" in name.lower():
            no_decay_params.append(param)
        else:
            decay_params.append(param)

    optimizer = torch.optim.AdamW(
        [
            {"params": decay_params, "weight_decay": weight_decay},
            {"params": no_decay_params, "weight_decay": 0.0},
        ],
        lr=lr,
        betas=(0.9, 0.95),
        eps=1e-8,
    )
    return optimizer
```

这个例子的重点不是固定超参数，而是参数分组思想：

```text
普通权重做 decay，bias 和 norm 参数通常不做 decay
```

真实项目中还要考虑：

1. embedding 是否 decay。
2. LoRA 参数是否 decay。
3. MoE router 参数是否单独设置。
4. 不同模块是否需要不同学习率。
5. 分布式 optimizer 是否正确接收参数组。

### 最小 demo：验证 AdamW 参数分组

下面这个 demo 构造一个 tiny module，检查哪些参数进入 decay group，哪些进入 no_decay group。真实 LLM 中参数名更复杂，但核心思想相同：普通矩阵权重通常 decay，bias 和 norm 参数通常 no_decay。

```python
import torch
import torch.nn as nn


class TinyModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.linear = nn.Linear(4, 4)
        self.norm = nn.LayerNorm(4)


def split_decay_params(model):
    decay_names = []
    no_decay_names = []

    for name, param in model.named_parameters():
        if not param.requires_grad:
            continue
        if name.endswith("bias") or "norm" in name.lower():
            no_decay_names.append(name)
        else:
            decay_names.append(name)

    return decay_names, no_decay_names


model = TinyModel()
decay, no_decay = split_decay_params(model)

print("decay:", decay)
print("no_decay:", no_decay)

optimizer = torch.optim.AdamW(
    [
        {"params": [p for n, p in model.named_parameters() if n in decay], "weight_decay": 0.1},
        {"params": [p for n, p in model.named_parameters() if n in no_decay], "weight_decay": 0.0},
    ],
    lr=1e-3,
)

print("param groups:", [group["weight_decay"] for group in optimizer.param_groups])
```

这个 demo 的重点不是 `0.1` 这个数值，而是分组原则。实际项目要根据模型结构确认 embedding、LoRA、MoE router、norm 和 bias 是否符合预期分组。

### 面试中怎么讲

如果面试官问“Adam 和 AdamW 的区别是什么”，可以这样回答：

```text
Adam 用一阶矩平滑梯度方向，用二阶矩做自适应缩放。普通 Adam 如果把 L2 正则直接加到梯度里，weight decay 项也会被二阶矩缩放，导致衰减行为和参数梯度历史耦合。AdamW 的核心是 decoupled weight decay，把参数衰减从自适应梯度更新中分离出来，因此正则化行为更清晰，也成为 Transformer 和 LLM 训练中的常见默认优化器。
```

如果面试官问“为什么 Adafactor 适合大模型”，可以这样回答：

```text
AdamW 需要为每个参数保存一阶矩和二阶矩，optimizer state 显存很大。Adafactor 通过对矩阵参数的二阶矩做因子化近似，用行统计和列统计代替完整矩阵状态，把状态从 r*c 降到 r+c，因此能显著节省显存。代价是二阶矩估计更近似，训练稳定性和效果需要结合具体模型、学习率和任务验证。
```

### 真实项目中的坑

#### 坑 1：只说用了 AdamW，不说参数分组

AdamW 是否正确使用，很大程度取决于 weight decay 参数组是否合理。

#### 坑 2：恢复 checkpoint 时漏掉 optimizer state

只恢复 model weights，不恢复 optimizer 和 scheduler，训练动态会突变。

#### 坑 3：把学习率问题误判成模型结构问题

loss spike、震荡和不收敛经常首先要看学习率、warmup、grad norm 和 scheduler。

#### 坑 4：低估 optimizer state 显存

训练显存不等于参数显存，AdamW 状态和 master weights 可能占更大比例。

#### 坑 5：盲目换新优化器

Lion、Adafactor 等优化器有价值，但不能不做 ablation 就认为一定优于 AdamW。

### 面试官会怎么问

#### 问题 1：Adam 的一阶矩和二阶矩分别有什么作用？

回答框架：

1. 一阶矩是梯度的滑动平均，用来平滑更新方向。
2. 二阶矩是梯度平方的滑动平均，用来估计每个参数的梯度尺度。
3. Adam 用二阶矩对更新做自适应缩放。

#### 问题 2：AdamW 为什么比 Adam + L2 更常用？

回答框架：

1. Adam + L2 会把正则项加到梯度里。
2. 这个正则项会被 Adam 的二阶矩缩放。
3. AdamW 解耦 weight decay，让衰减行为更清晰稳定。

#### 问题 3：Adafactor 如何节省显存？

回答框架：

1. Adam 保存完整二阶矩矩阵。
2. Adafactor 对矩阵参数用行统计和列统计近似二阶矩。
3. 状态从 `r*c` 降到 `r+c`。
4. 代价是近似可能影响稳定性和效果。

#### 问题 4：Lion 和 AdamW 有什么不同？

回答框架：

1. AdamW 使用一阶矩和二阶矩，自适应缩放更新。
2. Lion 使用动量方向的符号进行更新，不保存二阶矩。
3. Lion 状态更少，但学习率和 weight decay 可能更敏感。
4. 大规模 LLM 中是否更好需要实验验证。

#### 问题 5：训练不稳定时如何判断是不是优化器问题？

回答框架：

1. 看学习率曲线、warmup 和 scheduler。
2. 看 grad norm、parameter norm、update norm。
3. 检查 AdamW 参数分组和 weight decay。
4. 检查 betas、epsilon、gradient clipping。
5. 检查 checkpoint 恢复时 optimizer state 是否一致。

### 常见误区

1. 误区：AdamW 就是 Adam 换个名字。
   纠正：AdamW 的关键是 decoupled weight decay。

2. 误区：优化器只影响收敛速度，不影响最终效果。
   纠正：优化器会影响训练稳定性、泛化、参数范数和最终模型质量。

3. 误区：Adafactor 一定比 AdamW 更好。
   纠正：Adafactor 主要优势是省显存，效果和稳定性要看具体设置。

4. 误区：新优化器可以替代学习率调度。
   纠正：优化器和 schedule 必须一起设计。

5. 误区：checkpoint 只要保存模型权重就够了。
   纠正：继续训练必须保存 optimizer state、scheduler state 和 step count。

### 小练习

1. 用自己的话解释 Adam 的一阶矩和二阶矩。
2. 推导为什么 AdamW 的 decoupled weight decay 和 Adam + L2 行为不同。
3. 估算一个 7B 模型使用 AdamW 时 optimizer state 大约需要多少显存。
4. 写一个 AdamW 参数分组函数，排除 bias 和 norm 参数的 weight decay。
5. 解释 Adafactor 如何把二阶矩状态从 `r*c` 降到 `r+c`。
6. 用 2 分钟回答“为什么优化器选择是算法问题，也是系统问题”。

### 本讲总结

本讲最重要的结论：

1. 优化器决定参数如何根据当前梯度和历史信息更新，是训练稳定性和效率的核心组件。
2. Adam 用一阶矩平滑方向，用二阶矩做自适应缩放。
3. AdamW 通过 decoupled weight decay 修正 Adam + L2 的耦合问题，是 LLM 训练常见默认选择。
4. Adafactor 通过因子化二阶矩显著降低 optimizer state 显存，适合显存受限的大模型训练。
5. Lion 使用符号动量更新，状态更少，但在大规模 LLM 中需要谨慎验证。
6. 优化器必须和学习率调度、batch size、warmup、gradient clipping、mixed precision 和分布式状态管理一起设计。
7. 面试中讨论优化器时，要同时讲算法机制、显存成本、稳定性风险和工程排查方法。

### 开放题回答的通用结构

```text
validation loss 是 token-level 分布拟合指标，而线上效果是任务级和行为级指标。两者可能因为目标错配、验证集分布不一致、解码策略变化、后训练偏移、安全拒答变化或长尾场景退化而不一致。排查时要同时看任务指标、人类评估、分领域 slice、输出长度、拒答率、幻觉率和错误案例。
```

### 高频题 1：为什么交叉熵适合训练语言模型

回答框架：

1. 语言模型是在建模 token 序列概率。
2. 自回归分解把联合概率写成逐 token 条件概率。
3. 最大似然训练等价于最小化负对数似然。
4. 对 one-hot 目标，负对数似然就是交叉熵。
5. 所以交叉熵是 next-token prediction 的自然训练目标。

进一步展开：

```text
交叉熵优化的是训练分布下真实 token 的概率，它非常适合大规模自监督学习。但它仍然是 token-level 目标，不完全等于事实正确性、推理能力、人类偏好或安全性。因此预训练后还需要 SFT、偏好优化和多维评估。
```

常见追问：交叉熵和 KL 散度有什么关系？

可以回答：

```text
H(P, Q) = H(P) + KL(P || Q)。当真实分布 P 固定时，最小化交叉熵等价于最小化 KL 散度，也就是让模型分布 Q 接近数据分布 P。
```

### 高频题 2：为什么 loss 下降不代表模型一定更好

回答框架：

1. loss 是代理目标，不是真实目标本身。
2. 语言模型 loss 通常是 token-level。
3. 真实任务常是 sequence-level 或 behavior-level。
4. 数据分布、评估分布和线上分布可能不一致。
5. 后训练可能改善偏好但损伤某些能力。

例子：

1. 数学题最后一个数字错，token loss 差异小，但任务完全失败。
2. SFT loss 降低，但模型变啰嗦或过度拒答。
3. reward 分数升高，但模型学会讨好 reward model。
4. benchmark 分数提升，但真实长尾问题退化。

排查方法：

1. 看 validation loss 和训练 loss 是否一致。
2. 看任务级指标。
3. 做人工评估。
4. 分领域、分难度、分长度切片。
5. 看错误案例而不只看平均分。

### 高频题 3：为什么大模型参数很多但还能泛化

回答框架：

1. 传统 bias-variance 直觉不足以解释现代深度学习。
2. 大模型训练在海量数据和密集 token 监督上进行。
3. 过参数化可能帮助优化器找到更平坦、更稳定的解。
4. 隐式正则化、数据多样性、weight decay、dropout、去重都会影响泛化。
5. 大模型同时存在记忆和泛化，不能二选一。

可以提到 double descent：

```text
双下降说明测试误差不一定随着模型复杂度单调变差，在过参数化区域反而可能再次下降。这提示我们不能只用参数量判断泛化，要同时看数据规模、计算预算、优化动态和评估方式。
```

面试中要补一句风险：

```text
大模型能泛化不代表不会记忆，所以需要数据去重、污染检测、时间切分测试和分布外评估。
```

### 高频题 4：如何判断模型是记忆还是泛化

回答框架：

1. 先承认两者通常同时存在。
2. 记忆更常出现在重复、高频、模板化或敏感数据中。
3. 泛化体现为对未见组合、扰动样本和分布外任务的稳定表现。
4. 判断需要实验，而不是主观感觉。

可用方法：

1. exact match 去重。
2. n-gram overlap 检测。
3. embedding 相似度检测。
4. paraphrase 测试。
5. 时间切分评估。
6. 私有动态测试集。
7. 对训练样本和非训练样本做 membership inference 风险分析。

回答重点：

```text
如果模型只在原题上好，改写后明显下降，更像记忆；如果对新组合、扰动和同构问题仍然稳定，更像泛化。
```

### 高频题 5：训练中出现 loss spike 怎么办

回答框架：

1. 先判断 spike 是瞬时恢复还是持续发散。
2. 看 learning rate、warmup 和 scheduler。
3. 看 grad norm、parameter norm、update norm。
4. 查异常 batch、超长样本、数据分布切换。
5. 查 mixed precision overflow、NaN、Inf。
6. 查 attention mask、normalization epsilon、optimizer state。
7. 分布式训练还要看具体 rank、通信和 checkpoint 恢复。

如果要给工程处理方案：

1. 降低学习率或延长 warmup。
2. 开启或调小 gradient clipping 阈值。
3. 回滚到稳定 checkpoint。
4. 定位触发 batch 并做数据过滤。
5. 检查 FP16/BF16 设置和 loss scale。
6. 增加稳定性监控。

不要只回答“调小学习率”。

### 高频题 6：AdamW 为什么常用于 LLM

回答框架：

1. Adam 用一阶矩平滑梯度方向，用二阶矩做自适应缩放。
2. 这对深度网络和稀疏、噪声梯度更容易训练。
3. AdamW 解耦 weight decay，避免正则项被二阶矩缩放扭曲。
4. Transformer/LLM 中 AdamW 经验上稳定、好调，是常见默认选择。
5. 但它 optimizer state 显存大，需要 ZeRO、sharding 或低状态优化器缓解。

可以补充：

```text
AdamW 不是总是最优，Adafactor、Lion 等方法在显存或状态成本上有优势，但大规模预训练是否更好需要实验验证。
```

### 高频题 7：为什么 LLM 常用 LayerNorm/RMSNorm 而不是 BatchNorm

回答框架：

1. BatchNorm 依赖 batch 统计。
2. 语言模型序列长度可变，batch 内 token 分布差异大。
3. 自回归推理时训练和推理 batch 统计不一致。
4. 分布式和动态 batch 下 BatchNorm 统计复杂。
5. LayerNorm/RMSNorm 对每个 token 的 hidden dimension 归一化，不依赖 batch，更适合 LLM。

进一步区分 LayerNorm 和 RMSNorm：

```text
LayerNorm 减均值并除以标准差，RMSNorm 不减均值，只用 RMS 控制尺度。RMSNorm 计算更简单，很多 LLaMA 类模型中使用。
```

### 高频题 8：Pre-LN 为什么更适合深层 Transformer

回答框架：

1. Post-LN 是 `LN(x + F(x))`。
2. Pre-LN 是 `x + F(LN(x))`。
3. Pre-LN 保留更直接的 residual identity path。
4. 梯度可以更稳定地穿过很多层。
5. 因此深层 Transformer 更容易训练。

补充 trade-off：

```text
Pre-LN 更稳定，但不代表所有情况下最终效果都绝对更好；它通常需要 final norm，且 residual 表示尺度仍要监控。
```

### 高频题 9：为什么现代 LLM 常用 SwiGLU

回答框架：

1. 普通 FFN 是单分支非线性变换。
2. SwiGLU 有内容分支和门控分支。
3. 门控分支用 SiLU，形成 `up * SiLU(gate)`。
4. 它能动态选择和调制特征，表达能力更强。
5. 在相近计算预算下常比普通 GELU FFN 表现更好。

要注意参数量：

```text
SwiGLU 有三组投影矩阵，如果 hidden dimension 不调小，参数量和 FLOPs 会增加。所以实践中常把 hidden dimension 设置得比普通 4d FFN 小，例如接近 8/3 d_model。
```

### 高频题 10：如何理解不确定性和幻觉

回答框架：

1. softmax 概率不等于真实置信度。
2. 模型可能在分布外问题上高置信输出错误答案。
3. 不确定性可以分为 aleatoric 和 epistemic。
4. 幻觉常来自证据不足、训练数据错误、解码随机性、后训练偏好和缺少验证。
5. 缓解需要校准、RAG、工具调用、verifier、拒答和人工评估。

回答时可以强调：

```text
可靠系统不应该只让模型生成答案，还要让模型在证据不足时表达不确定、请求澄清或调用外部工具。
```

### 高频题 11：如何设计一个模型能力评估方案

回答框架：

1. 先明确目标能力。
2. 区分 token-level、sequence-level 和 behavior-level 指标。
3. 覆盖准确性、有用性、安全性、鲁棒性、效率和成本。
4. 使用自动评估和人工评估结合。
5. 做分领域、分难度、分长度和分风险切片。
6. 做污染检测和时间切分。
7. 对错误案例做 error analysis。

示例指标：

1. validation loss。
2. exact match。
3. pass@k。
4. human preference win rate。
5. hallucination rate。
6. refusal rate。
7. safety violation rate。
8. latency 和 cost。

### 高频题 12：如何看待 benchmark 分数

回答框架：

1. benchmark 是有用的标准化比较工具。
2. 但它不能完全代表真实能力。
3. 可能存在数据污染、题型过拟合、覆盖不足和指标单一。
4. 平均分可能掩盖长尾失败和安全风险。
5. 因此 benchmark 要和私有评估、动态测试、人类评估和真实日志结合。

可以引用 Goodhart's Law：

```text
当 benchmark 分数成为强优化目标时，模型可能学会提高分数而不是提升真实能力。
```

### 高频题 13：模型训练失败时如何系统排查

回答框架：

1. 数据：是否有坏样本、重复、污染、label 错误、长度异常。
2. 模型：mask、position encoding、norm、初始化、激活是否正确。
3. 优化器：learning rate、betas、weight decay、scheduler、clipping。
4. 数值：FP16/BF16、loss scale、NaN、Inf、softmax overflow。
5. 分布式：rank 间数据、通信、梯度同步、checkpoint state。
6. 评估：评估代码、数据切分、解码参数是否一致。

面试中可以给出一句总结：

```text
排查训练失败要从数据、模型、优化、数值、分布式和评估六条线并行看，而不是只盯着 loss 曲线。
```

### 高频题 14：如何体现 trade-off 意识

很多开放题最终都不是“哪个方法最好”，而是“在什么约束下怎么选”。

常见 trade-off：

1. 模型规模 vs 推理成本。
2. 数据规模 vs 数据质量。
3. 安全性 vs 有用性。
4. 创造性 vs 事实可靠性。
5. 长回答完整性 vs 简洁性。
6. benchmark 分数 vs 真实用户体验。
7. AdamW 稳定性 vs optimizer state 显存。
8. SwiGLU 表达能力 vs FLOPs。
9. RAG 降低幻觉 vs 系统复杂度和延迟。

好的回答应该包含：

```text
我会在什么指标上优化，用什么指标监控副作用，如果副作用出现怎么回滚或修正。
```

### 一个开放题回答模板

可以把下面模板用于很多面试题：

```text
我会先区分这个问题的目标和代理指标。然后从数据、模型结构、训练目标、优化过程、评估方式和工程约束几个维度分析。每个维度都可能带来收益，也可能引入副作用。最后我会设计分层评估，包括自动指标、人类评估、长尾 case、安全 case 和线上反馈，确认改动是否真的改善了目标，而不是只优化了某个代理分数。
```

这个模板的价值在于：

1. 不会只背一个概念。
2. 会主动区分目标和指标。
3. 会考虑副作用。
4. 会提出验证方案。
5. 符合真实大模型工程思路。

### 最小 demo：开放题回答自评脚本

下面这个 demo 把开放题回答拆成几个必须覆盖的维度。它不是自然语言评分器，只是一个提醒工具：如果回答里完全没有目标、机制、验证或 trade-off，通常说明回答还不够完整。

```python
rubric = {
    "target": ["目标", "能力", "指标"],
    "mechanism": ["机制", "原因", "数据", "模型", "优化"],
    "evaluation": ["评估", "实验", "ablation", "错误分析"],
    "tradeoff": ["trade-off", "成本", "延迟", "风险", "副作用"],
}

answer = """
我会先明确目标能力和线上指标，再从数据、模型结构、训练目标和优化过程拆解原因。
验证上会做分领域评估、ablation 和错误分析，同时监控安全风险、延迟成本和可能的副作用。
"""

for name, keywords in rubric.items():
    hit = any(keyword in answer for keyword in keywords)
    print(f"{name}: {'pass' if hit else 'missing'}")
```

真实面试当然不能靠关键词打分，但这个小脚本可以帮助你训练一种习惯：开放题回答要有目标、机制、验证和取舍，而不是只给一个方案。

### 面试中容易失分的回答方式

#### 失分 1：只背定义

例如只说“AdamW 是解耦 weight decay”，但不解释为什么 Adam + L2 有问题。

#### 失分 2：只讲论文结论

例如只说“SwiGLU 效果更好”，但不讲门控机制、参数量和 FLOPs。

#### 失分 3：不区分场景

例如说“loss 低就是好”，忽略任务指标和目标错配。

#### 失分 4：没有工程排查思路

例如遇到 NaN 只说“调学习率”，不查数据、精度、mask 和 optimizer state。

#### 失分 5：没有评估意识

例如提出一个训练改动，但不说明如何验证是否真的变好。

### 本部分知识图谱

第一部分可以串成一条主线：

```text
概率建模 -> 交叉熵/KL -> 优化动态 -> 泛化 -> 贝叶斯/不确定性 -> 目标错配 -> 训练稳定性 -> 激活/MLP -> 优化器 -> 开放题表达
```

这条主线对应大模型算法岗的基础能力：

1. 能从概率角度解释语言模型。
2. 能从优化角度解释训练过程。
3. 能从泛化角度解释模型能力。
4. 能从工程角度排查训练问题。
5. 能从评估角度判断模型是否真的更好。
6. 能在面试中讲清楚 trade-off。

### 本讲总结

本讲最重要的结论：

1. 深度学习基础开放题考察的是系统理解，不是单点定义。
2. 好回答要同时包含机制、场景、风险、评估和 trade-off。
3. loss、benchmark、reward 都是代理指标，不能直接等同于真实能力。
4. 大模型训练问题要从数据、模型、优化、数值、分布式和评估多维排查。
5. 面试中最有价值的表达是：我优化了什么、没优化什么、可能牺牲什么、如何验证副作用。
6. 第一部分的核心主线是用概率、优化、泛化和工程稳定性共同理解 LLM 训练基础。
