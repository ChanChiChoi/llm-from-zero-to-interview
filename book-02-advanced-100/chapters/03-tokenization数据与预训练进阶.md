# 第三部分：Tokenization、数据与预训练进阶

## 第 26 讲：Tokenizer 进阶：BPE、Unigram 与 SentencePiece

### 本讲目标

Tokenizer 是大模型训练的第一道入口。它决定原始文本如何被切成 token，也会影响多语言能力、代码能力、上下文利用率、训练效率和生成质量。本讲从 BPE、Unigram、SentencePiece 入手，理解 tokenizer 设计为什么不是简单预处理，而是模型能力的一部分。

你需要掌握：

1. Tokenizer 为什么重要。
2. BPE 的基本思想。
3. Unigram tokenizer 的基本思想。
4. SentencePiece 解决了什么问题。
5. vocab size、压缩率和多语言能力的 trade-off。
6. tokenizer 对代码、中文、多语言和长上下文的影响。
7. 面试中如何回答 tokenizer 设计问题。

### Tokenizer 为什么重要

LLM 不是直接处理原始字符串，而是处理 token id 序列。

流程通常是：

```text
文本 -> tokenizer -> token ids -> embedding -> Transformer
```

所以 tokenizer 决定了模型看到的基本单位。

如果 tokenizer 设计不好，会带来：

1. 序列变长，上下文浪费。
2. 低资源语言被切得很碎。
3. 代码符号和缩进处理差。
4. 特殊 token 冲突。
5. 训练和推理分布不一致。
6. 生成时更容易出现乱码或格式错误。

Tokenizer 是数据、模型和系统之间的接口。

### 字符、词和子词

最简单的切分方式有三类：

1. 字符级。
2. 词级。
3. 子词级。

字符级优点是不会 OOV，但序列很长。

词级优点是语义单位清晰，但词表巨大，遇到新词容易 OOV。

子词级是折中方案。

现代 LLM 大多使用子词 tokenizer。

常见方法包括 BPE、WordPiece、Unigram 和 SentencePiece 系列实现。

### BPE 的基本思想

BPE 是 Byte Pair Encoding。

它的核心思想是从小单位开始，不断合并最常见的相邻 pair。

简化流程：

1. 初始词表是字符或 byte。
2. 统计训练语料中最频繁的相邻 token pair。
3. 把这个 pair 合并成新 token。
4. 重复直到达到目标 vocab size。

例如频繁出现：

```text
l + o -> lo
lo + w -> low
```

BPE 会逐步学出常见子词。

### BPE 的优点和问题

BPE 的优点：

1. 简单高效。
2. 能处理未见词。
3. 高频词会变成短 token 序列。
4. 被很多 GPT 类模型使用。

BPE 的问题：

1. 合并是贪心的。
2. 不一定得到概率意义上最优切分。
3. 对不同语言的公平性依赖训练语料。
4. 可能对空格、大小写、标点非常敏感。

所以 BPE 好用，但不是没有偏置。

### Byte-level BPE

很多 GPT 类 tokenizer 使用 byte-level BPE。

它从 byte 而不是 Unicode 字符开始。

优点是几乎不会出现无法编码的字符。

任何文本都可以转成 byte 序列。

缺点是：

1. 某些非英文语言可能被切得更碎。
2. token 可读性较差。
3. 对多语言压缩率有影响。

Byte-level BPE 非常适合开放互联网文本，但对多语言公平性需要额外关注。

### Unigram Tokenizer

Unigram tokenizer 的思想不同。

它不是从小单位不断合并，而是先构造一个较大的候选子词词表，然后学习每个子词的概率。

给定一个句子，可以有多种切分方式。

Unigram 会选择概率较高的切分，并通过训练逐步删除不重要的子词。

直觉是：

```text
BPE 是合并式，Unigram 是从候选词表中做概率选择和剪枝
```

### Unigram 的特点

Unigram 的优点：

1. 有概率建模视角。
2. 可以支持多种可能切分。
3. 常和 SentencePiece 一起使用。
4. 对某些多语言场景较灵活。

缺点：

1. 训练过程比 BPE 更复杂。
2. 实现和调参成本更高。
3. 具体效果依赖语料和 vocab 设置。

### SentencePiece 是什么

SentencePiece 不是单一 tokenizer 算法，而是一个 tokenizer 训练和编码框架。

它支持 BPE 和 Unigram。

它的重要设计是把输入当作原始字符流处理，而不是先按空格切词。

这对中文、日文、没有明显空格分词边界的语言很重要。

SentencePiece 常用特殊符号表示空格，例如 `▁`。

这让 tokenizer 能统一处理多语言文本。

### Vocab Size 的 trade-off

vocab size 是 tokenizer 的关键超参数。

词表太小：

1. 序列变长。
2. 训练和推理成本增加。
3. 长上下文可用信息减少。
4. 某些语言或代码被切得很碎。

词表太大：

1. embedding 参数增加。
2. output softmax 参数增加。
3. 低频 token 学习不足。
4. tokenizer 可能过度记忆语料表面形式。

所以 vocab size 是压缩率、参数量和泛化之间的 trade-off。

### Tokenizer 和多语言

多语言模型中，tokenizer 非常关键。

如果训练语料英文占比很高，tokenizer 可能更偏向英文。

结果是：

1. 英文平均每词 token 数少。
2. 中文、阿拉伯语、印地语等语言可能被切得更碎。
3. 同样 context length 下，非英文能表达的信息更少。
4. 训练计算也更偏向高压缩语言。

这会造成语言间的不公平。

因此多语言 tokenizer 需要关注各语言 tokenization fertility，也就是每段文本平均切成多少 token。

### Tokenizer 和中文

中文没有天然空格分词。

如果 tokenizer 设计不好，中文可能被切成很多细碎 byte 或字符片段。

这会影响：

1. 上下文利用率。
2. 生成流畅度。
3. 训练效率。
4. 术语和专名处理。

SentencePiece 这类不依赖空格预分词的方法，对中文更友好。

但最终效果仍取决于中文语料比例和 vocab 分配。

### Tokenizer 和代码

代码模型对 tokenizer 有特殊要求。

代码中重要元素包括：

1. 缩进。
2. 空格。
3. 换行。
4. 下划线。
5. 括号和标点。
6. 变量名拆分。
7. 常见关键字和库名。

如果 tokenizer 把代码符号切得太碎，会增加序列长度。

如果不能稳定保留空白和缩进，代码生成质量会受影响。

所以代码 LLM 往往需要在 tokenizer 训练语料中加入足够代码数据。

### Tokenizer 和长上下文

context length 是按 token 计算的，不是按字符计算的。

如果 tokenizer 压缩率更好，同样 32K token 可以容纳更多文本。

如果某种语言被切得更碎，它实际可用上下文更短。

所以长上下文能力也受 tokenizer 影响。

面试中可以说：

```text
max context length 不能只看 token 数，还要看目标语言和数据类型的 tokenization 压缩率
```

### 特殊 Token 设计

LLM tokenizer 通常包含特殊 token。

例如：

1. BOS。
2. EOS。
3. PAD。
4. UNK。
5. system/user/assistant 分隔符。
6. tool call 相关 token。
7. image/audio placeholder token。

特殊 token 设计会影响对话模板、工具调用、多模态输入和安全策略。

如果训练和推理使用不同模板，模型行为可能变差。

### Tokenizer 迁移风险

训练好模型后，不能随便换 tokenizer。

原因是 embedding 和 output head 都和 token id 对齐。

换 tokenizer 意味着 token id 语义变了。

这通常需要重新训练或做复杂适配。

所以 tokenizer 是模型架构的一部分，不是可随意替换的外部预处理。

### 面试中怎么讲

如果面试官问“Tokenizer 为什么重要”，可以这样回答：

```text
Tokenizer 决定原始文本如何变成 token 序列，直接影响序列长度、上下文利用率、多语言公平性、代码格式处理、embedding 参数和输出 softmax。它不是简单预处理，而是模型输入输出空间的一部分。训练好模型后也不能随便替换 tokenizer，因为 token id 和 embedding、lm head 都已经绑定。
```

如果面试官问“BPE 和 Unigram 有什么区别”，可以这样回答：

```text
BPE 是从字符或 byte 开始，不断合并语料中最频繁的相邻 pair，属于合并式贪心方法。Unigram 通常先构造候选子词词表，然后学习子词概率，并通过概率模型选择和剪枝子词。BPE 简单高效，Unigram 更偏概率建模，SentencePiece 框架可以支持这两类方法。
```

### 真实项目中的坑

#### 坑 1：只看 vocab size

同样 vocab size，不同语料和算法训练出来的 tokenizer 表现可能差很多。

#### 坑 2：忽略多语言 fertility

如果某些语言平均 token 数过多，会造成实际上下文和训练计算不公平。

#### 坑 3：代码空白处理不好

缩进、换行和空格对代码语义很重要，不能随便丢。

#### 坑 4：训练和推理模板不一致

特殊 token 和 chat template 不一致会导致模型行为异常。

#### 坑 5：训练后想换 tokenizer

这通常不可行，除非重新训练 embedding/lm head 或做复杂迁移。

### 面试官会怎么问

#### 问题 1：BPE 的训练过程是什么？

回答框架：

1. 从字符或 byte 初始化词表。
2. 统计相邻 pair 频率。
3. 合并最高频 pair。
4. 重复直到达到目标 vocab size。

#### 问题 2：SentencePiece 解决什么问题？

回答框架：

1. 把文本当原始字符流处理。
2. 不依赖空格预分词。
3. 适合中文、日文等语言。
4. 支持 BPE 和 Unigram。

#### 问题 3：vocab size 怎么选？

回答框架：

1. 太小序列长、压缩率差。
2. 太大 embedding 和 softmax 参数多。
3. 低频 token 学习不足。
4. 要结合语言覆盖、代码比例和上下文成本。

#### 问题 4：为什么 tokenizer 会影响长上下文？

回答框架：

1. context length 按 token 计数。
2. tokenizer 压缩率决定同样 token 数能放多少文本。
3. 非英文或代码被切碎会浪费上下文。
4. 所以长上下文要看实际文本容量。

#### 问题 5：能不能训练后替换 tokenizer？

回答框架：

1. 通常不能随便换。
2. token id 和 embedding/lm head 绑定。
3. 换 tokenizer 会改变输入输出空间。
4. 需要重新训练或复杂迁移。

### 常见误区

1. 误区：Tokenizer 只是预处理。
   纠正：它定义了模型输入输出空间，是架构和训练的一部分。

2. 误区：vocab 越大越好。
   纠正：过大词表会增加参数和低频 token 学习难度。

3. 误区：英文 tokenizer 对所有语言都公平。
   纠正：多语言 fertility 差异会造成上下文和计算不公平。

4. 误区：代码只是普通文本。
   纠正：代码对空白、标点、缩进和变量名拆分非常敏感。

5. 误区：特殊 token 不重要。
   纠正：对话模板、工具调用和多模态占位都依赖特殊 token 设计。

### 小练习

1. 手动模拟几步 BPE merge。
2. 比较同一句中文和英文被 tokenizer 切分后的 token 数。
3. 设计一个多语言 tokenizer 评估表。
4. 分析代码 tokenizer 应该保留哪些特殊结构。
5. 用 2 分钟回答“为什么 tokenizer 会影响模型能力”。

### 本讲总结

本讲最重要的结论：

1. Tokenizer 决定文本如何进入模型，是模型能力和效率的重要组成部分。
2. BPE 是高频 pair 合并式方法，Unigram 是概率子词选择方法。
3. SentencePiece 提供不依赖空格预分词的 tokenizer 框架，适合多语言。
4. vocab size、压缩率、多语言公平性、代码格式和特殊 token 都需要权衡。
5. 训练后 tokenizer 不能随便替换，因为它和 embedding、lm head 绑定。

## 第 27 讲：数据清洗 pipeline 设计

### 本讲目标

预训练数据不是“越多越好”，而是“规模、质量、多样性和安全性”共同决定模型能力。数据清洗 pipeline 的目标是把原始海量数据变成可训练、可追踪、可控的数据集。本讲从工程视角讲解大模型预训练数据清洗的核心流程。

你需要掌握：

1. 数据清洗 pipeline 为什么重要。
2. 原始数据到训练数据的基本步骤。
3. 质量过滤、语言识别、去重、安全过滤分别解决什么问题。
4. 规则过滤和模型过滤如何配合。
5. 数据清洗如何影响模型能力和偏差。
6. 面试中如何设计一个预训练数据 pipeline。

### 为什么数据清洗重要

大模型会从数据中学习语言、知识、风格、偏见和错误。

如果数据质量差，模型可能学到：

1. 垃圾网页格式。
2. 重复模板。
3. 低质量问答。
4. 错误事实。
5. 有害内容。
6. benchmark 泄露。
7. 乱码和编码错误。

所以数据清洗不是简单 ETL，而是模型能力建设的一部分。

### 数据 pipeline 总览

一个典型预训练数据 pipeline 包括：

1. 数据采集。
2. 格式解析。
3. 文本抽取。
4. 语言识别。
5. 质量过滤。
6. 安全和合规过滤。
7. 去重。
8. 数据分类和打标签。
9. 数据配比和采样。
10. tokenizer 统计和 tokenization。
11. 数据打包和 shard。
12. 训练时 streaming 或 dataloader。

每一步都会影响最终模型。

### 格式解析和文本抽取

原始数据可能来自网页、PDF、代码仓库、论坛、书籍、论文、问答网站等。

不同来源需要不同解析方式。

常见问题包括：

1. HTML boilerplate。
2. 导航栏和广告文本。
3. PDF 断行错误。
4. 表格错乱。
5. 代码和注释混合。
6. 编码异常。

文本抽取阶段的目标是尽量保留正文，去掉噪声结构。

### 语言识别

语言识别用于判断文本属于哪种语言。

它的作用包括：

1. 控制多语言配比。
2. 移除未知语言或乱码。
3. 为不同语言设置不同过滤规则。
4. 分析 tokenizer fertility。

语言识别也会出错。

短文本、混合语言、代码和专有名词会让识别变难。

所以真实 pipeline 中常需要置信度阈值和人工抽检。

### 质量过滤

质量过滤用于移除低质量文本。

常见规则包括：

1. 文本太短或太长。
2. 字符重复率过高。
3. 标点或数字比例异常。
4. 乱码比例过高。
5. 词汇多样性太低。
6. HTML 残留太多。
7. 模板化内容太强。

也可以使用质量分类器判断文本是否像高质量语料。

规则过滤便宜、可解释；模型过滤更灵活，但需要训练数据和校准。

### 安全和合规过滤

安全过滤用于处理有害、违法、隐私和不适合训练的内容。

常见类别包括：

1. 个人隐私信息。
2. 密钥、token、密码。
3. 恶意软件代码。
4. 极端暴力或仇恨内容。
5. 低质量成人内容。
6. 版权风险内容。

安全过滤不是简单全部删除。

某些安全研究和拒答能力也需要适量安全相关数据，但必须在合规、标注和使用方式上谨慎设计。

### 去重

去重是大模型数据处理中非常关键的一步。

重复数据会带来：

1. 训练计算浪费。
2. 记忆风险增加。
3. 数据分布被重复样本扭曲。
4. benchmark contamination 风险增加。
5. validation loss 失真。

去重可以分为：

1. exact dedup。
2. near dedup。
3. document-level dedup。
4. paragraph-level dedup。
5. benchmark 去污染。

下一讲会更系统讲去重和污染。

### 数据分类和打标签

清洗后的数据还需要分类。

常见标签包括：

1. 语言。
2. 来源。
3. 领域。
4. 质量等级。
5. 是否代码。
6. 是否数学。
7. 是否对话。
8. 是否合成数据。

这些标签用于后续数据配比和采样。

没有标签的数据湖很难精细控制训练。

### 数据配比和采样

不同来源数据价值不同。

高质量书籍、代码、数学、论文、网页、问答、对话数据对模型能力影响不同。

数据配比决定模型更擅长什么。

例如：

1. 代码比例高，代码能力更强，但可能影响自然语言风格。
2. 数学数据多，推理和公式能力可能提升。
3. 低质量网页过多，模型输出可能变差。
4. 多语言数据不足，非英文能力弱。

数据配比是训练策略的一部分，不是清洗后的附属步骤。

### 数据质量评估

数据清洗必须可评估。

常见评估方式包括：

1. 抽样人工检查。
2. 统计过滤前后 token 数。
3. 各语言和领域分布。
4. 重复率。
5. PII 命中率。
6. 毒性或安全风险分布。
7. 用小模型训练做 ablation。
8. 下游 benchmark 对比。

只写规则不评估，pipeline 很容易过度过滤或漏过滤。

### 面试中怎么讲

如果面试官问“如何设计大模型预训练数据清洗 pipeline”，可以这样回答：

```text
我会把 pipeline 分成采集、解析、语言识别、质量过滤、安全合规过滤、去重、分类打标签、配比采样、tokenization 和打包几个阶段。每个阶段都要有可观测指标，例如过滤率、语言分布、重复率、PII 命中率、质量分数和人工抽样结果。最后通过小规模训练 ablation 和下游评估验证清洗策略是否真的提升模型能力，而不是只凭规则判断。
```

### 真实项目中的坑

#### 坑 1：过度过滤

规则太严会删除长尾知识、少数语言和特殊领域文本。

#### 坑 2：过滤不足

低质量、重复和污染数据进入训练，会影响模型质量和评估可信度。

#### 坑 3：只做规则不做抽检

规则命中不等于质量改善，必须人工抽样和统计验证。

#### 坑 4：忽略数据来源追踪

没有 lineage，后续发现问题无法定位和回滚。

#### 坑 5：清洗策略不版本化

数据 pipeline 必须版本化，否则实验不可复现。

### 面试官会怎么问

#### 问题 1：数据清洗为什么影响模型能力？

回答框架：

1. 模型从数据中学习语言、知识和行为模式。
2. 垃圾数据会带来噪声和错误模式。
3. 重复数据会增加记忆风险。
4. 数据配比会改变能力分布。

#### 问题 2：质量过滤有哪些常见规则？

回答框架：

1. 长度过滤。
2. 重复字符过滤。
3. 乱码比例过滤。
4. 标点数字比例过滤。
5. 模板和 boilerplate 过滤。
6. 质量分类器过滤。

#### 问题 3：规则过滤和模型过滤怎么取舍？

回答框架：

1. 规则过滤便宜、可解释。
2. 模型过滤更灵活但可能有偏差。
3. 两者通常结合使用。
4. 需要人工抽检和 ablation 验证。

#### 问题 4：为什么要记录数据 lineage？

回答框架：

1. 方便追踪数据来源。
2. 支持问题定位和回滚。
3. 支持复现实验。
4. 满足合规和审计需求。

### 常见误区

1. 误区：数据越多越好。
   纠正：低质量和重复数据会浪费算力并伤害模型。

2. 误区：清洗规则越严格越好。
   纠正：过度过滤会损失长尾知识和多样性。

3. 误区：清洗是一次性工作。
   纠正：数据 pipeline 需要持续迭代、评估和版本化。

4. 误区：安全过滤就是全部删除敏感内容。
   纠正：需要结合合规、用途、标注和安全训练目标设计。

2. 列出 10 个低质量文本过滤规则。
3. 设计一个人工抽检表，评估清洗前后质量。
4. 说明如何判断过滤规则是否过严。
5. 用 2 分钟回答“如何设计预训练数据清洗 pipeline”。

### 本讲总结

本讲最重要的结论：

1. 数据清洗是模型能力建设的一部分，不是简单预处理。
2. 预训练数据 pipeline 包括解析、语言识别、质量过滤、安全过滤、去重、打标签、配比和打包。
3. 规则过滤和模型过滤需要结合，并通过人工抽检和训练 ablation 验证。
4. 数据质量、重复率、配比和安全合规都会影响最终模型。
5. 数据 pipeline 必须可观测、可追踪、可版本化。

## 第 28 讲：数据去重与 Benchmark Contamination

### 本讲目标

数据去重和 benchmark contamination 是大模型预训练中非常关键但容易被低估的问题。重复数据会浪费算力、增加记忆风险、扭曲数据分布；benchmark 污染会让评估分数虚高，导致我们误判模型能力。本讲系统讲解去重和污染检测的核心方法。

你需要掌握：

1. 为什么重复数据有害。
2. exact dedup 和 near dedup 的区别。
3. 文档级、段落级、n-gram 级去重的适用场景。
4. benchmark contamination 是什么。
5. 如何检测训练数据和评估集污染。
6. 面试中如何设计去重和去污染 pipeline。

### 为什么重复数据有害

重复数据带来的问题包括：

1. 浪费训练 token 预算。
2. 增加模型记忆风险。
3. 放大某些来源或模板的权重。
4. 让 validation loss 失真。
5. 让 benchmark 分数不可信。
6. 降低数据多样性。

如果模型反复看到同一段文本，它可能不是学会了泛化规律，而是记住了样本。

### Exact Dedup

exact dedup 指完全相同文本的去重。

做法通常是对文档规范化后计算 hash。

例如：

1. 去除多余空白。
2. 统一大小写或 Unicode 规范化。
3. 计算 hash。
4. 相同 hash 只保留一份。

优点：简单、快速、误删风险低。

缺点：无法处理轻微改写、模板变化、插入广告或格式差异。

### Near Dedup

near dedup 处理的是近似重复。

例如：

1. 同一篇文章被不同网站转载。
2. 只改了标题或时间。
3. 插入了少量导航或模板。
4. 句子顺序略有变化。

常见方法包括：

1. MinHash。
2. SimHash。
3. n-gram overlap。
4. embedding similarity。
5. locality-sensitive hashing。

near dedup 更强，但误删风险也更高。

### 文档级去重

文档级去重把整篇文档作为单位。

适合处理网页转载、重复 PDF、镜像站点。

优点是简单，能大幅减少重复文档。

缺点是：

1. 同一文档中重复段落仍可能保留。
2. 两篇文档部分重叠不容易处理。
3. 长文档中少量污染内容可能被忽略。

### 段落级去重

段落级去重把文档拆成 paragraph 或 chunk。

它能处理：

1. 网页模板。
2. 重复免责声明。
3. 版权声明。
4. 常见页脚。
5. 部分转载。

段落级去重更细，但需要注意不要破坏上下文结构。

### N-gram 去重

n-gram 去重通过比较连续 token 片段判断重复。

例如，如果两个文档共享大量 13-gram，就可能高度重复。

优点是能发现局部复制。

缺点是计算量大，需要高效索引。

n-gram 方法也常用于 benchmark contamination 检测。

### Benchmark Contamination 是什么

benchmark contamination 指评估集内容进入训练数据。

例如：

1. benchmark 原题出现在网页语料。
2. benchmark 答案出现在论坛讨论。
3. 测试集被公开后被爬入训练数据。
4. paraphrase 版本进入训练。
5. 数据合成时不小心使用了测试题。

污染会导致模型分数虚高。

这时模型可能不是泛化，而是见过题。

### 污染为什么严重

污染会影响模型评估可信度。

如果一个模型在 benchmark 上表现很好，但训练中见过原题或答案，那么这个分数不能代表真实泛化能力。

污染尤其影响：

1. 数学题。
2. 代码题。
3. 多选题。
4. 标准问答集。
5. 小规模高知名 benchmark。

越公开、越热门的 benchmark，越容易污染。

### Contamination 检测方法

常见检测方法包括：

1. exact match。
2. n-gram overlap。
3. fuzzy match。
4. embedding similarity。
5. paraphrase detection。
6. 时间切分。
7. 私有测试集。

exact match 能抓完全泄露。

n-gram overlap 能抓局部重叠。

embedding similarity 能抓语义近似，但误报风险更高。

### 时间切分

时间切分是很重要的防污染方法。

例如训练数据只使用某个日期之前的语料。

评估集使用该日期之后的新任务或新题。

这样可以降低训练集见过评估内容的概率。

但时间切分也不是万能的。

原因是：

1. 旧知识仍可能覆盖新题。
2. 数据时间戳可能不准。
3. 评估题可能基于旧材料改写。

所以时间切分要和相似度检测结合。

### 去重的副作用

去重也可能带来副作用。

如果过度去重，可能删除：

1. 合法引用。
2. 常见教学材料。
3. 标准代码模板。
4. 多语言平行语料。
5. 必要的高频格式。

所以去重不是越狠越好。

要根据数据类型和训练目标设置阈值。

### 去重 pipeline 设计

一个实用 pipeline 可以是：

1. 文本规范化。
2. exact dedup。
3. document-level near dedup。
4. paragraph-level dedup。
5. benchmark contamination 检测。
6. 抽样人工审核。
7. 统计重复率和删除率。
8. 训练 ablation 验证效果。

不同阶段要记录被删除原因，支持回溯。

### 面试中怎么讲

如果面试官问“如何处理训练数据去重和 benchmark 污染”，可以这样回答：

```text
我会先做文本规范化和 exact dedup，去掉完全重复文档；再用 MinHash、SimHash 或 n-gram overlap 做 near dedup，处理转载和模板重复。对于 benchmark contamination，我会把评估集和训练语料做 exact match、n-gram overlap、embedding similarity 和时间切分检测，并对高风险样本人工抽查。去重策略不能只看删除率，还要通过数据统计、人工抽检和训练 ablation 验证是否提升泛化而不是误删有价值数据。
```

### 真实项目中的坑

#### 坑 1：只做 exact dedup

完全重复容易去掉，但转载、轻微改写和模板重复仍会保留。

#### 坑 2：near dedup 阈值过严

可能误删合法相似内容，例如教程、代码模板、多语言平行文本。

#### 坑 3：只检测题目，不检测答案

benchmark 污染可能是答案或解析泄露，而不是题目原文泄露。

#### 坑 4：忽略 paraphrase 污染

语义改写后的测试题也可能进入训练数据。

#### 坑 5：没有记录删除原因

后续无法分析去重策略是否误删或漏删。

### 面试官会怎么问

#### 问题 1：exact dedup 和 near dedup 有什么区别？

回答框架：

1. exact dedup 删除完全相同文本。
2. near dedup 删除高度相似文本。
3. exact dedup 简单可靠。
4. near dedup 能处理转载和轻微改写，但误删风险更高。

#### 问题 2：benchmark contamination 如何影响评估？

回答框架：

1. 训练数据中出现评估题或答案。
2. 模型可能记住内容。
3. benchmark 分数虚高。
4. 评估不再代表真实泛化。

#### 问题 3：如何检测 contamination？

回答框架：

1. exact match。
2. n-gram overlap。
3. fuzzy match。
4. embedding similarity。
5. 时间切分和私有评估集。

#### 问题 4：去重会不会伤害模型？

回答框架：

1. 会，过度去重可能删除有价值重复。
2. 教学材料、代码模板、引用内容可能被误删。
3. 需要阈值调节和抽样审核。
4. 最终要用训练 ablation 验证。

### 常见误区

1. 误区：重复数据只是浪费一点空间。
   纠正：重复会影响训练分布、记忆风险和评估可信度。

2. 误区：benchmark 公开不代表污染。
   纠正：公开 benchmark 很容易进入网页语料和合成数据。

3. 误区：exact match 没命中就没有污染。
   纠正：还有 paraphrase、答案泄露和局部重叠。

4. 误区：去重越多越好。
   纠正：过度去重会损失长尾和有价值重复模式。

### 小练习

1. 设计一个 exact dedup 流程。
2. 解释 MinHash 或 SimHash 适合 near dedup 的原因。
3. 设计一个 benchmark contamination 检测方案。
4. 举一个答案泄露但题目不泄露的例子。
5. 用 2 分钟回答“如何判断模型 benchmark 分数是否可信”。

### 本讲总结

本讲最重要的结论：

1. 重复数据会浪费计算、增加记忆风险、扭曲分布并污染评估。
2. 去重需要结合 exact dedup、near dedup、文档级、段落级和 n-gram 方法。
3. benchmark contamination 会让评估分数虚高，必须主动检测。
4. 污染检测要覆盖题目、答案、解析、局部重叠和语义改写。
5. 去重不是越严越好，需要抽检、统计和训练 ablation 验证。

## 第 29 讲：数据配比与 Curriculum Learning

### 本讲目标

预训练不是把所有数据混在一起随机喂给模型。不同来源、语言、领域和质量等级的数据会塑造模型的能力分布。数据配比决定模型更擅长什么，curriculum learning 则关注训练过程中数据难度和类型如何随时间变化。本讲讲清楚数据配比和课程学习在大模型训练中的作用。

你需要掌握：

1. 数据配比为什么重要。
2. 不同数据类型对能力的影响。
3. oversampling 和 undersampling 的 trade-off。
4. curriculum learning 的直觉。
5. 大模型训练中 curriculum 是否总是有效。
6. 如何评估数据配比策略。
7. 面试中如何设计数据 mixture。

### 数据配比是什么

数据配比指不同数据来源在训练 token 中所占比例。

例如一个预训练 mixture 可以包含：

1. 网页文本。
2. 书籍。
3. 论文。
4. 代码。
5. 数学题。
6. 问答数据。
7. 多语言数据。
8. 对话数据。

如果代码数据占比高，模型通常会更擅长代码。

如果高质量长文占比高，模型可能更擅长长篇表达和知识组织。

如果低质量网页占比过高，模型输出风格可能变差。

### 为什么不能只按原始数据比例训练

原始互联网数据分布不是理想训练分布。

它可能存在：

1. 英文占比过高。
2. 重复网页模板很多。
3. 低质量 SEO 内容很多。
4. 代码、数学等高价值数据相对稀缺。
5. 某些语言或领域严重不足。

如果完全按原始比例训练，模型会继承这些偏差。

所以需要人工设计数据 mixture。

### 高质量数据的作用

高质量数据通常包括：

1. 结构清晰的书籍。
2. 高质量技术文档。
3. 论文和教材。
4. 高质量代码。
5. 经过筛选的问答和解释数据。

高质量数据的作用不只是降低 loss。

它还会影响：

1. 模型表达风格。
2. 推理步骤质量。
3. 事实密度。
4. 回答结构。
5. 长文一致性。

### 专项数据和能力塑造

不同专项数据会塑造不同能力。

代码数据提升：

1. 代码生成。
2. 调试。
3. API 使用。
4. 结构化推理。

数学数据提升：

1. 符号推理。
2. 多步解题。
3. 公式理解。
4. 精确计算习惯。

多语言数据提升：

1. 非英文理解。
2. 翻译。
3. 跨语言迁移。
4. 地域知识覆盖。

数据配比本质上是在决定模型能力预算如何分配。

### Oversampling 和 Undersampling

如果某类高价值数据很少，可以 oversample。

例如代码数据只有 5%，但希望模型代码能力强，可以采样到 15%。

oversampling 的好处是增强目标能力。

风险是：

1. 重复暴露增加记忆风险。
2. 过拟合小数据域。
3. 影响其他能力。
4. 输出风格偏向该领域。

undersampling 则用于降低低质量或过多数据的影响。

例如减少低质量网页占比。

### 数据配比的动态调整

数据配比不一定全程固定。

可以在训练不同阶段调整。

例如：

1. 早期使用更广泛数据学习基础语言和知识。
2. 中期增加高质量数据比例。
3. 后期增加代码、数学或指令风格数据。

这种动态数据策略和 curriculum learning 有关。

但动态调整也会引入新的变量，需要仔细做实验。

### Curriculum Learning 是什么

Curriculum learning 的直觉是：

```text
先学简单或基础数据，再学复杂或高价值数据
```

类似人类学习课程。

在大模型中，课程可以按多种维度设计：

1. 从短文本到长文本。
2. 从高质量通用文本到专业文本。
3. 从简单代码到复杂项目代码。
4. 从基础数学到复杂推理。
5. 从自然语料到指令格式。

### Curriculum 是否总是有效

不一定。

大模型训练中，curriculum 的效果依赖：

1. 模型规模。
2. 数据规模。
3. 训练步数。
4. 任务目标。
5. 数据质量。
6. schedule 设计。

如果课程设计不好，可能导致：

1. 早期分布太窄。
2. 后期分布切换造成 loss spike。
3. 某些能力被遗忘。
4. 训练分布和评估分布不匹配。

所以 curriculum 需要实验验证，而不是默认有效。

### 数据配比和遗忘

如果训练后期大幅增加某类数据，模型可能向该领域偏移。

例如后期大量加入代码，模型代码能力提高，但自然语言回答风格可能变化。

这类似一种预训练阶段的分布偏移。

因此需要监控多维评估，而不是只看目标领域指标。

### 如何评估数据配比

评估数据配比需要看多个维度：

1. validation loss。
2. 分领域 validation loss。
3. 代码 benchmark。
4. 数学 benchmark。
5. 多语言 benchmark。
6. 常识和事实问答。
7. 安全和偏见指标。
8. 生成风格和人工评估。

如果只看总 loss，很容易看不到专项能力变化。

### 数据配比实验设计

比较不同数据配比时，要尽量控制变量。

例如：

1. 固定模型大小。
2. 固定训练 token 数。
3. 固定 tokenizer。
4. 固定优化器和学习率。
5. 只改变数据 mixture。
6. 用相同评估集比较。

否则很难判断提升来自数据配比还是其他因素。

### 面试中怎么讲

如果面试官问“如何设计预训练数据配比”，可以这样回答：

```text
我会先明确目标能力，例如通用对话、代码、数学、多语言或专业领域。然后把数据按来源、语言、领域和质量打标签，设计初始 mixture。对高价值但稀缺的数据可以适度 oversample，对低质量网页做 undersample。训练中要监控总 validation loss 和分领域指标，通过 ablation 比较不同 mixture。必要时可以做动态 curriculum，但要防止分布切换导致 loss spike 或能力遗忘。
```

### 真实项目中的坑

#### 坑 1：只看总 token 数

同样 token 数，不同配比会产生完全不同的能力分布。

#### 坑 2：过度 oversample 小数据集

可能导致记忆、过拟合和风格偏移。

#### 坑 3：只优化目标能力

提升代码或数学时，可能损伤通用语言、多语言或安全能力。

#### 坑 4：课程切换太剧烈

训练分布突然变化可能造成 loss spike 或模型行为漂移。

#### 坑 5：没有分领域验证集

没有分领域指标，就无法知道配比改变影响了哪些能力。

### 面试官会怎么问

#### 问题 1：数据配比为什么重要？

回答框架：

1. 数据配比决定训练 token 预算分配。
2. 不同数据塑造不同能力。
3. 原始互联网分布不是理想训练分布。
4. 配比会影响能力、风格和偏差。

#### 问题 2：oversampling 有什么风险？

回答框架：

1. 增加记忆风险。
2. 小数据域过拟合。
3. 影响其他能力。
4. 输出风格偏移。

#### 问题 3：curriculum learning 在 LLM 中如何设计？

回答框架：

1. 可以按长度、质量、难度、领域设计。
2. 先基础后复杂。
3. 需要监控分布切换。
4. 效果必须通过 ablation 验证。

#### 问题 4：如何评估数据 mixture？

回答框架：

1. 总 validation loss。
2. 分领域 validation loss。
3. 下游专项 benchmark。
4. 多语言和安全指标。
5. 人工评估和错误分析。

### 常见误区

1. 误区：高质量数据越多越好。
   纠正：要看多样性、覆盖面和目标能力，过窄数据会造成偏移。

2. 误区：curriculum 一定提升训练。
   纠正：课程设计不好可能造成分布不匹配和能力遗忘。

3. 误区：代码数据只影响代码能力。
   纠正：代码数据也可能影响推理、格式和自然语言风格。

4. 误区：数据配比是训练前一次性决定的。
   纠正：可以动态调整，但需要监控和实验验证。

### 小练习

1. 设计一个通用 LLM 的数据 mixture。
2. 设计一个偏代码模型的数据 mixture。
3. 解释 oversampling 数学数据的收益和风险。
4. 设计一个 curriculum schedule。
5. 用 2 分钟回答“数据配比如何影响模型能力”。

### 本讲总结

本讲最重要的结论：

1. 数据配比决定训练 token 预算如何分配，是能力塑造的核心手段。
2. 不同数据类型会影响代码、数学、多语言、事实和表达风格等能力。
3. oversampling 可以增强稀缺高价值数据，但也有记忆和过拟合风险。
4. curriculum learning 可以动态安排训练数据，但不是必然有效。
5. 数据配比必须通过分领域评估、ablation 和人工分析验证。

## 第 30 讲：合成数据的价值与风险

### 本讲目标

合成数据是当前大模型训练和后训练中的重要手段。它可以补齐稀缺能力、扩展指令数据、生成推理轨迹，也可能带来偏差放大、数据污染、模式坍缩和评估失真。本讲讲清楚合成数据什么时候有价值，什么时候有风险，以及如何在真实项目中使用和评估。

你需要掌握：

1. 什么是合成数据。
2. 合成数据在预训练和后训练中的用途。
3. 合成数据为什么能提升能力。
4. 合成数据的主要风险。
5. 如何过滤、验证和配比合成数据。
6. 面试中如何讨论 synthetic data 的 trade-off。

### 什么是合成数据

合成数据指由模型、程序、规则或模拟器生成的数据。

在 LLM 中，常见合成数据包括：

1. 模型生成的指令问答。
2. 模型生成的推理步骤。
3. 模型生成的代码题和测试用例。
4. 数据增强产生的 paraphrase。
5. 规则模板生成的结构化样本。
6. 模拟环境生成的交互轨迹。

合成数据不是天然高质量，也不是天然低质量。

它的价值取决于生成方法、过滤标准、配比和评估。

### 为什么需要合成数据

真实高质量数据往往稀缺。

例如：

1. 高质量人工指令数据贵。
2. 数学推理数据难收集。
3. 工具调用轨迹需要构造环境。
4. 安全红队数据难覆盖长尾。
5. 某些专业领域公开数据少。

合成数据可以低成本扩展这些稀缺场景。

### 合成数据在预训练中的用途

预训练中，合成数据可以用于：

1. 扩展数学和代码语料。
2. 生成领域文本。
3. 增强低资源语言。
4. 构造格式化数据。
5. 补齐长尾知识表达。

但预训练合成数据要特别注意分布质量。

如果大量低质量模型输出进入预训练，模型可能学习到合成味、模板化表达和错误事实。

### 合成数据在后训练中的用途

后训练中，合成数据更常见。

用途包括：

1. instruction tuning。
2. chain-of-thought 蒸馏。
3. preference data 扩展。
4. tool-use 轨迹生成。
5. safety refusal 样本构造。
6. 多轮对话模拟。

例如用强模型生成指令和答案，再用过滤器筛选，训练较小模型。

这是一种常见的蒸馏思路。

### 合成数据为什么有效

合成数据有效的原因包括：

1. 可以定向覆盖稀缺任务。
2. 可以控制格式和难度。
3. 可以批量生成多样样本。
4. 可以把强模型能力蒸馏给弱模型。
5. 可以构造真实数据中很少出现的边界场景。

本质上，合成数据是一种可控数据生成工具。

它把“找数据”部分转化为“设计生成器和过滤器”。

### 主要风险 1：错误放大

生成模型会犯错。

如果错误答案、错误推理或错误代码进入训练数据，学生模型会学习这些错误。

尤其是：

1. 数学推理步骤看似合理但结论错误。
2. 代码能编译但逻辑错。
3. 事实问答编造来源。
4. 安全样本标签不一致。

所以合成数据必须验证。

### 主要风险 2：模式坍缩

模型生成的数据可能风格单一。

例如：

1. 开头总是类似表达。
2. 推理步骤模板化。
3. 答案长度分布单一。
4. 问题类型缺乏多样性。

如果大量使用这种数据，训练出的模型也可能变得模板化。

这叫 synthetic data 的模式坍缩风险。

### 主要风险 3：偏差继承

合成数据通常来自一个 teacher model。

teacher model 的偏见、盲点和风格会传给 student model。

如果 teacher 在某些语言、文化或领域表现差，合成数据也会继承这些问题。

所以合成数据不是自动更公平，也不是自动更安全。

### 主要风险 4：污染评估

如果用 benchmark 题生成合成训练数据，就会污染评估。

即使不是原题，也可能生成高度相似题。

这会让模型在 benchmark 上表现虚高。

因此合成数据也要做 contamination 检测。

尤其是数学、代码和标准问答任务。

### 合成数据过滤

常见过滤方法包括：

1. 规则过滤。
2. 质量模型打分。
3. 多模型一致性检查。
4. verifier 验证。
5. 单元测试。
6. 人工抽检。
7. 去重和相似度检测。
8. 毒性和安全过滤。

不同任务需要不同 verifier。

代码数据可以跑测试。

数学数据可以用 symbolic checker 或答案校验。

事实数据需要检索证据支持。

### 合成数据配比

合成数据不能无限加。

如果合成数据占比过高，可能导致模型分布偏向 teacher。

常见策略是：

1. 和真实数据混合。
2. 控制合成数据比例。
3. 按任务类型设置不同权重。
4. 对合成数据做更严格过滤。
5. 用 ablation 找最佳比例。

合成数据比例是数据 mixture 的一部分。

### 合成数据的评估

评估合成数据不能只看训练 loss。

需要看：

1. 目标任务指标是否提升。
2. 非目标任务是否退化。
3. 输出是否模板化。
4. 错误类型是否变化。
5. 安全性是否变化。
6. benchmark 是否被污染。
7. 人类偏好是否提升。

最可靠的方法是做数据 ablation。

### 面试中怎么讲

如果面试官问“合成数据有什么价值和风险”，可以这样回答：

```text
合成数据的价值在于可以低成本、可控地覆盖稀缺任务，例如数学推理、代码、工具调用、安全红队和多轮对话，也可以把强模型能力蒸馏给小模型。但风险是生成数据可能包含错误、偏差和模板化模式，过多使用会导致模式坍缩或继承 teacher 的盲点，还可能污染 benchmark。因此合成数据必须经过 verifier、规则过滤、去重、人工抽检和 ablation 评估，并控制在合适配比中使用。
```

### 真实项目中的坑

#### 坑 1：把 teacher 输出当真值

强模型也会犯错，尤其在事实、数学和代码边界场景。

#### 坑 2：只生成不过滤

没有过滤的合成数据很容易把错误和模板化风格注入学生模型。

#### 坑 3：合成数据比例过高

模型可能过度模仿 teacher 风格，丢失真实数据多样性。

#### 坑 4：忽略 contamination

合成题可能和 benchmark 高度相似，导致评估虚高。

#### 坑 5：只看目标任务提升

合成数据可能提升某个能力，同时损伤其他能力和安全性。

### 面试官会怎么问

#### 问题 1：合成数据为什么能提升模型？

回答框架：

1. 定向覆盖稀缺任务。
2. 控制格式和难度。
3. 扩展训练样本多样性。
4. 蒸馏强模型能力。

#### 问题 2：合成数据有哪些风险？

回答框架：

1. 错误放大。
2. 模式坍缩。
3. 偏差继承。
4. benchmark contamination。
5. 非目标能力退化。

#### 问题 3：如何过滤合成数据？

回答框架：

1. 规则过滤。
2. 质量模型打分。
3. verifier 或单元测试。
4. 多模型一致性。
5. 人工抽检。
6. 去重和污染检测。

#### 问题 4：如何决定合成数据比例？

回答框架：

1. 根据目标能力和真实数据稀缺程度。
2. 从小比例开始做 ablation。
3. 监控目标和非目标任务。
4. 防止模板化和分布偏移。

### 常见误区

1. 误区：合成数据越多越好。
   纠正：低质量合成数据会放大错误和模板化。

2. 误区：强模型生成的数据一定正确。
   纠正：强模型也会幻觉，需要验证。

3. 误区：合成数据不会污染 benchmark。
   纠正：合成过程可能生成与测试题相似的样本。

4. 误区：合成数据只用于后训练。
   纠正：预训练、后训练、工具调用和安全训练都可能使用合成数据。

### 本讲总结

本讲最重要的结论：

1. 合成数据可以低成本覆盖稀缺任务，并支持能力蒸馏。
2. 合成数据风险包括错误放大、模式坍缩、偏差继承和评估污染。
3. 高质量合成数据依赖生成器、过滤器、verifier 和人工抽检。
4. 合成数据需要控制配比，并通过 ablation 验证收益和副作用。
5. 面试中讨论合成数据，要同时讲价值、风险、过滤和评估闭环。

## 第 31 讲：Scaling Law 经典形式

### 本讲目标

Scaling Law 是理解大模型训练预算、模型规模、数据规模和 loss 关系的核心工具。它不是一个神秘公式，而是一组经验规律：在合理训练设置下，模型越大、数据越多、计算越多，loss 通常会按幂律下降。本讲先讲经典 scaling law 的形式和直觉，下一讲再讲 Chinchilla 对 compute-optimal training 的修正。

你需要掌握：

1. Scaling Law 研究什么。
2. 参数量、数据量、计算量和 loss 的关系。
3. 幂律下降的直觉。
4. 为什么 scaling law 对训练预算规划重要。
5. scaling law 能预测什么，不能预测什么。
6. 面试中如何解释“大模型为什么可预测地变强”。

### Scaling Law 研究什么

Scaling Law 关心的问题是：

```text
当模型规模、数据规模和计算量增加时，模型 loss 如何变化？
```

这里的 loss 通常指 validation loss 或 cross entropy loss。

它不直接等于所有下游能力，但和模型整体能力有较强相关性。

Scaling Law 的价值在于，它让大模型训练从“拍脑袋试规模”变成更可预测的工程规划。

### 三个核心变量

经典 scaling law 常关注三个变量：

1. `N`：模型参数量。
2. `D`：训练数据 token 数。
3. `C`：训练计算量。

粗略来说：

```text
C ≈ 6ND
```

这里 `6ND` 是 dense Transformer 训练 FLOPs 的常用近似估算。

实际系数会因架构和实现不同而变化。

### Loss 随参数量变化

当数据足够多、训练充分时，增加模型参数量通常会降低 loss。

经验形式类似：

```text
L(N) = A / N^α + L_inf
```

其中：

1. `L(N)` 是 loss。
2. `N` 是参数量。
3. `α` 是 scaling exponent。
4. `L_inf` 是不可约 loss 或极限项。

这说明 loss 不是线性下降，而是幂律下降。

模型越大，收益仍然存在，但边际收益会递减。

### Loss 随数据量变化

类似地，当模型足够大时，增加训练 token 数也会降低 loss。

形式类似：

```text
L(D) = B / D^β + L_inf
```

数据越多，模型越能覆盖语言、知识和任务模式。

但数据收益也会递减。

而且数据质量非常关键。

低质量 token 增加不一定等价于高质量 token 增加。

### Loss 随计算量变化

计算量 `C` 可以理解为训练预算。

在合理分配模型参数和数据 token 的情况下，loss 会随计算量增加而下降。

形式类似：

```text
L(C) = K / C^γ + L_inf
```

这对训练规划很重要。

如果你有固定算力预算，需要决定：

1. 训练多大模型。
2. 训练多少 token。
3. 是否多训练小模型还是少训练大模型。

### 幂律下降的直觉

Scaling law 中常出现 power law。

直觉是：

```text
每增加一倍资源，loss 会按相对稳定但递减的比例下降
```

这给大模型训练带来可预测性。

例如小规模实验可以拟合 scaling curve，再外推到更大规模。

当然，外推不是无限可靠。

一旦数据质量、训练设置、架构或分布发生变化，曲线可能偏移。

### 为什么 Scaling Law 重要

Scaling law 的工程价值包括：

1. 估算目标 loss 需要多少算力。
2. 决定模型参数量和训练 token 数。
3. 比较不同训练方案性价比。
4. 判断模型是否 undertrained。
5. 预测继续训练是否值得。
6. 指导预算分配。

对大模型团队来说，训练一次大模型成本很高，scaling law 可以降低盲目试错。

### Undertraining 和 Overtraining

如果模型很大，但训练 token 太少，模型可能 undertrained。

表现是：

1. 参数量很大。
2. loss 还没充分下降。
3. 小模型训练更多 token 可能更划算。

如果模型较小但训练 token 很多，也可能进入收益很低的区域。

这就引出 compute-optimal training 问题。

下一讲会讲 Chinchilla scaling law 如何重新平衡参数量和 token 数。

### Scaling Law 和下游能力

Validation loss 下降通常和很多下游能力提升相关。

但不能简单等同。

原因是：

1. 下游任务可能是 sequence-level 指标。
2. loss 是 token-level 指标。
3. 后训练会改变模型行为。
4. reasoning、tool use、safety 需要额外评估。
5. benchmark 可能受污染。

所以 scaling law 预测的是预训练分布上的 loss 趋势，不是所有能力的完整替代指标。

### Scaling Law 和 Emergence

有些能力看起来像在某个规模后突然出现。

这常被称为 emergent abilities。

Scaling law 提供一种连续视角：底层 loss 可能平滑下降，但某些离散任务指标在达到阈值后才明显变好。

例如：

1. 多步推理。
2. 少样本学习。
3. 代码任务通过率。
4. 指令跟随。

面试中可以说：loss 趋势可能平滑，但能力指标可能非线性显现。

### 如何做 Scaling 实验

一个 scaling 实验通常包括：

1. 训练多个不同规模模型。
2. 使用相似数据分布和训练 recipe。
3. 控制 batch、学习率、token 数等变量。
4. 记录 training loss 和 validation loss。
5. 拟合 power law。
6. 外推更大规模表现。

关键是控制变量。

如果每个实验的数据质量、优化器和架构都不同，很难拟合可靠曲线。

### Scaling Law 的边界

Scaling law 不是自然定律。

它是特定条件下的经验规律。

可能失效的情况包括：

1. 数据分布变化。
2. 数据质量变化。
3. 架构变化。
4. tokenizer 变化。
5. 优化器和训练 recipe 变化。
6. 训练不稳定。
7. 评估集污染。

所以面试中要避免把 scaling law 讲成万能预测器。

### 面试中怎么讲

如果面试官问“Scaling Law 是什么”，可以这样回答：

```text
Scaling Law 是大模型训练中的经验规律，描述模型 loss 如何随参数量、训练 token 数和计算量增加而下降。经典形式通常是幂律关系，例如 loss 随 N、D 或 C 增大按 power law 下降。它的价值在于帮助我们预测扩大模型和数据是否值得，并在固定算力预算下规划模型参数量和训练 token 数。但它主要预测预训练 loss，不等价于所有下游能力。
```

### 真实项目中的坑

#### 坑 1：用低质量数据拟合 scaling law

数据质量变化会改变曲线，低质量 token 不能简单等价于高质量 token。

#### 坑 2：只看参数量

模型大小、训练 token 数和计算量必须一起看。

#### 坑 3：把 loss 当成所有能力

reasoning、安全、工具调用和偏好需要额外评估。

#### 坑 4：小规模外推过远

外推跨度太大时，架构和优化问题可能导致预测失真。

#### 坑 5：忽略训练不稳定

loss spike、NaN 或数据问题会破坏 scaling 曲线。

### 面试官会怎么问

#### 问题 1：Scaling Law 里的 N、D、C 分别是什么？

回答框架：

1. `N` 是模型参数量。
2. `D` 是训练 token 数。
3. `C` 是计算量。
4. dense Transformer 中 `C` 常近似和 `ND` 成正比。

#### 问题 2：为什么 loss 随资源增加是幂律下降？

回答框架：

1. 经验上在很多规模区间拟合良好。
2. 每增加一倍资源都有稳定但递减收益。
3. 这说明规模扩大有可预测性。
4. 但它是经验规律，不是万能定律。

#### 问题 3：Scaling Law 能指导什么决策？

回答框架：

1. 目标 loss 需要多少算力。
2. 参数量和训练 token 数怎么分配。
3. 是否继续扩大模型。
4. 是否模型 undertrained。

#### 问题 4：Scaling Law 有什么局限？

回答框架：

1. 依赖数据分布和质量。
2. 依赖架构和训练 recipe。
3. 主要预测 loss，不等同于所有能力。
4. 外推过远风险高。

### 常见误区

1. 误区：模型越大一定越好。
   纠正：还要看训练 token、数据质量和计算预算。

2. 误区：Scaling Law 能预测所有 benchmark。
   纠正：它主要预测 loss 趋势，下游能力要单独评估。

3. 误区：只要加数据就能提升。
   纠正：数据质量、去重和配比同样关键。

4. 误区：小模型拟合曲线可以无限外推。
   纠正：外推范围有限，规模变化会引入新问题。

### 小练习

1. 用自己的话解释 `N`、`D`、`C` 的关系。
2. 画出 loss 随参数量幂律下降的曲线。
3. 举一个 undertrained 大模型的例子。
4. 设计一个小规模 scaling 实验。
5. 用 2 分钟回答“Scaling Law 为什么对训练预算重要”。

### 本讲总结

本讲最重要的结论：

1. Scaling Law 描述 loss 随参数量、数据量和计算量增加的经验规律。
2. 经典形式通常是幂律下降，带来可预测的规模收益。
3. 它能帮助规划训练预算和模型尺寸。
4. Scaling Law 主要预测预训练 loss，不等同于所有下游能力。
5. 数据质量、架构、训练 recipe 和评估污染都会影响 scaling 规律。

## 第 32 讲：Chinchilla Scaling Law

### 本讲目标

上一讲讲了经典 Scaling Law：模型参数量、训练 token 数和计算量增加时，loss 通常按幂律下降。

但真实训练中还有一个更关键的问题：

```text
在固定训练算力下，应该训练更大的模型，还是用更多 token 训练较小的模型？
```

Chinchilla Scaling Law 的核心贡献，就是重新回答这个 compute-optimal training 问题。

你需要掌握：

1. Chinchilla 解决的是什么问题。
2. 为什么很多早期大模型是 undertrained。
3. compute-optimal 的参数量和 token 数如何平衡。
4. Chinchilla 结论对训练预算规划有什么影响。
5. Chinchilla Law 和经典 Scaling Law 的区别。
6. 面试中如何解释“为什么小一些但训练更充分的模型可能更强”。

### Chinchilla 背景

在 Chinchilla 之前，很多大模型训练范式更偏向扩大参数量。

典型思路是：

```text
参数越多，模型越强。
```

这当然有道理，但不完整。

因为大模型不只是由参数量决定，还由训练 token 数决定。

如果一个模型参数非常多，但只看了相对少的数据，它可能没有被充分训练。

这就是 undertrained large model。

Chinchilla 的重要发现是：在相同计算预算下，许多大模型参数太多、训练 token 太少；把模型做小一些，但训练更多 token，反而能得到更低 loss 和更好下游表现。

### Compute-Optimal Training 是什么

Compute-optimal training 指的是：

```text
在固定计算预算 C 下，选择最合适的模型参数量 N 和训练 token 数 D，使最终 loss 最低。
```

这里不是单独问：

1. 模型越大越好吗？
2. 数据越多越好吗？

而是问：

```text
在同样 FLOPs 下，N 和 D 怎么分配最划算？
```

因为 dense Transformer 的训练计算量可以粗略写成：

```text
C ≈ 6ND
```

当 `C` 固定时，`N` 和 `D` 之间存在 trade-off：

1. 增大 `N`，通常就要减少 `D`。
2. 增大 `D`，通常就要减少 `N`。

Chinchilla 研究的就是这个最优平衡点。

### Chinchilla 的核心结论

Chinchilla 的结论可以简化为一句话：

```text
在 compute-optimal 条件下，模型参数量和训练 token 数应该近似同比例增长。
```

更通俗地说：

```text
模型变大时，训练数据也要同步变多。
```

一个常见经验表达是：

```text
每 1 个参数大约对应 20 个训练 token
```

例如：

1. 10B 参数模型，可能需要约 200B token。
2. 70B 参数模型，可能需要约 1.4T token。
3. 175B 参数模型，可能需要约 3.5T token。

这不是绝对定律，而是一个重要工程参考。

真实项目中还要考虑数据质量、重复训练 epoch、模型架构、tokenizer、优化器和目标任务。

### 为什么早期大模型会 Undertrained

早期很多大模型强调参数量扩张。

原因包括：

1. 大参数量更容易带来显著能力提升。
2. 模型规模是直观可见的指标。
3. 训练数据 pipeline 和高质量 token 储备不足。
4. 当时对 compute-optimal token 数的认识不充分。
5. 训练超大 token 量需要更强的数据工程和集群稳定性。

结果是，一些模型参数很大，但训练 token 数相对不足。

这种模型看起来很大，但 loss 还没有降到同等计算预算下可以达到的位置。

如果把同样算力用于稍小模型和更多 token，可能会更优。

### Chinchilla 模型案例

Chinchilla 论文中的代表性对比是：

1. Gopher：约 280B 参数，训练 token 相对较少。
2. Chinchilla：约 70B 参数，训练 token 更多。

在相近计算预算下，Chinchilla 参数更少，但训练更充分。

结果是 Chinchilla 在很多任务上超过了更大的 Gopher。

这个案例传递的关键不是“70B 一定比 280B 好”，而是：

```text
在固定算力下，参数量和 token 数要匹配，不能只堆参数。
```

### 从公式角度理解

设训练计算量近似为：

```text
C ≈ 6ND
```

如果预算 `C` 固定，有很多组合：

```text
大 N，小 D
中等 N，中等 D
小 N，大 D
```

经典 scaling law 告诉我们 `N` 和 `D` 都能降低 loss。

Chinchilla 进一步问：

```text
二者的边际收益应该怎么平衡？
```

当模型太大、数据太少时，增加 token 的边际收益大于继续增加参数。

当模型太小、数据太多时，增加参数的边际收益可能更大。

Compute-optimal 点就是二者边际收益相对平衡的位置。

### 参数量和 Token 数的 Trade-off

在工程上可以这样理解：

#### 方案 A：大模型，少 token

优点：

1. 模型容量大。
2. 推理时可能有更强表达能力。
3. 在少量训练后也可能出现一些规模能力。

缺点：

1. 容易 undertrained。
2. 训练 loss 没有充分下降。
3. 推理成本高。
4. 同等预算下性价比可能差。

#### 方案 B：小一些模型，多 token

优点：

1. 训练更充分。
2. validation loss 可能更低。
3. 推理成本更低。
4. 同等预算下性价比更高。

缺点：

1. 模型容量有限。
2. 极高难度任务可能受参数上限限制。
3. 继续增加 token 后可能进入数据边际收益下降区间。

Chinchilla 的结论不是永远选小模型，而是在给定训练预算下选 compute-optimal 的 `N` 和 `D`。

### 对训练预算规划的影响

Chinchilla Law 对大模型团队最直接的影响是预算规划。

以前可能会这样问：

```text
我们能训练多大的模型？
```

Chinchilla 之后更合理的问题是：

```text
在我们的算力、数据和时间预算下，哪个 N-D 组合最优？
```

规划时通常要同时估算：

1. 总 FLOPs 预算。
2. 可用高质量 token 数。
3. 目标模型参数量。
4. 训练时长和集群稳定性。
5. 推理成本和部署场景。
6. 是否需要继续训练或后训练。

如果高质量数据不足，即使有很多算力，也不能盲目提高 token 数。

如果推理成本受限，也不能只按预训练 loss 选择超大模型。

### Chinchilla 与数据质量

Chinchilla 讨论的是 token 数和参数量的关系，但现实中 token 不是同质的。

同样是 1T token，质量可能差异巨大。

高质量 token 包括：

1. 信息密度高。
2. 去重充分。
3. 噪声低。
4. 覆盖目标领域。
5. benchmark 污染少。

低质量 token 可能包括：

1. 重复网页模板。
2. 垃圾文本。
3. 低质量爬虫内容。
4. 机器生成污染。
5. 和目标能力无关的内容。

所以不能把 Chinchilla 简化成“只要 token 数够多就行”。

更准确的说法是：

```text
compute-optimal training 需要足够多的高质量 token。
```

### Chinchilla 与重复训练 Epoch

如果高质量数据有限，一个自然问题是：

```text
能不能把同一批数据多训几遍，凑够 token 数？
```

可以，但要谨慎。

重复训练有收益，也有风险。

收益是：

1. 数据利用更充分。
2. 小数据领域可以提高拟合程度。
3. 特定领域继续预训练可能有效。

风险是：

1. 过拟合训练数据。
2. 泛化能力下降。
3. 记忆和泄露风险增加。
4. benchmark contamination 更难控制。

因此训练 token 数不完全等于 unique token 数。

面试中要区分：

```text
训练 token budget、unique token、epoch 数、数据质量
```

### Chinchilla 与推理成本

Chinchilla 的另一个隐含优势是推理成本。

如果一个较小模型通过更多 token 训练达到类似甚至更好效果，那么它部署时更便宜。

推理成本主要受参数量、上下文长度、batch size、KV cache 和硬件利用率影响。

因此，在真实产品中，compute-optimal pretraining 不只是训练阶段省钱，也可能带来推理阶段收益。

但也要注意：

1. 参数更小不一定所有任务都更强。
2. 长上下文和复杂推理仍可能需要更大容量。
3. 后训练、蒸馏和工具使用会改变最终产品效果。

### Chinchilla 与后训练

Chinchilla 主要讨论预训练 loss 和 compute-optimal pretraining。

它不直接解决：

1. 指令跟随。
2. RLHF 或 DPO。
3. 安全对齐。
4. 工具调用。
5. 长链路推理。

但预训练底座更好，通常会让后训练更容易。

如果 base model undertrained，后训练可能只能修饰行为，难以补齐底层知识和语言建模能力。

所以可以把 Chinchilla 理解为底座模型训练阶段的重要预算原则。

### 与经典 Scaling Law 的区别

经典 Scaling Law 重点描述：

```text
loss 如何随 N、D、C 增加而变化
```

Chinchilla 更关注：

```text
固定 C 时，N 和 D 如何分配最优
```

二者不是矛盾关系，而是递进关系。

经典 scaling law 给出规模增长的总体规律。

Chinchilla 则把这个规律用于训练预算优化。

### 面试中怎么讲

如果面试官问“Chinchilla Scaling Law 是什么”，可以这样回答：

```text
Chinchilla Scaling Law 研究的是固定训练计算预算下，模型参数量和训练 token 数应该如何分配。它指出很多早期大模型参数量偏大、训练 token 偏少，处于 undertrained 状态。在 compute-optimal 条件下，参数量和训练 token 数应该更平衡地增长，常见经验是每个参数大约对应 20 个训练 token。因此，同样算力下，一个较小但训练更充分的模型可能比更大的 undertrained 模型效果更好。
```

如果要更工程化，可以补一句：

```text
但实际训练还要考虑数据质量、unique token、重复 epoch、推理成本和目标任务，不能机械套比例。
```

### 真实项目中的坑

#### 坑 1：机械套 20 token per parameter

这个比例是经验参考，不是不可变规则。

数据质量、架构和训练目标都会影响最优点。

#### 坑 2：只算 token 数，不看数据质量

低质量 token 不能简单替代高质量 token。

为了凑 token 数加入大量噪声数据，可能让模型变差。

#### 坑 3：忽略推理成本

训练阶段 compute-optimal 不等于产品阶段最优。

部署时还要看延迟、吞吐、显存和服务成本。

#### 坑 4：把预训练 loss 等同于最终助手能力

最终助手能力还受 SFT、偏好对齐、安全策略、工具调用和系统设计影响。

#### 坑 5：重复数据多训当成新数据

重复 epoch 可以增加训练 token，但不能完全等价于增加 unique high-quality token。

### 面试官会怎么问

#### 问题 1：Chinchilla Law 解决了什么问题？

回答框架：

1. 解决固定算力下参数量和 token 数如何分配的问题。
2. 指出很多大模型参数过大、训练 token 不足。
3. 强调 compute-optimal training。
4. 结论是 N 和 D 应该更平衡地增长。

#### 问题 2：为什么小模型训练更多 token 可能超过大模型？

回答框架：

1. 大模型如果 token 不足会 undertrained。
2. 在固定 FLOPs 下，大模型消耗更多计算在每个 token 上。
3. 较小模型可以看更多数据，loss 下降更充分。
4. 如果达到更优 N-D 平衡，最终效果可能更好。

#### 问题 3：20 token per parameter 是什么意思？

回答框架：

1. 这是 Chinchilla 相关结论中常用的经验比例。
2. 表示 compute-optimal 训练中，每个参数大约需要对应一定数量训练 token。
3. 它帮助估算模型大小和训练数据量。
4. 但不是硬规则，要结合数据质量和训练设置。

#### 问题 4：Chinchilla Law 有什么局限？

回答框架：

1. 主要基于特定架构、数据和训练设置的经验规律。
2. token 数不等于 token 质量。
3. 预训练 loss 不等于所有下游能力。
4. 真实系统还要考虑推理成本和后训练。

#### 问题 5：如果你有固定预算，会怎么选模型大小？

回答框架：

1. 先估算总 FLOPs 和可用高质量 token。
2. 用 scaling law 做小规模实验。
3. 拟合不同 N-D 组合的 loss 曲线。
4. 结合推理成本和目标任务选择参数量。
5. 留出预算给验证、checkpoint、失败重训和后训练。

### 常见误区

1. 误区：Chinchilla 说明模型越小越好。
   纠正：它说明固定算力下要平衡参数量和 token 数，不是永远选小模型。

2. 误区：只要满足 20 token per parameter 就一定最优。
   纠正：这个比例受数据质量、架构和训练 recipe 影响。

3. 误区：训练 token 越多越好。
   纠正：低质量或重复 token 的边际收益有限，甚至有害。

4. 误区：Chinchilla 能决定最终 Chatbot 能力。
   纠正：它主要指导预训练预算，最终能力还依赖后训练和系统设计。

### 小练习

1. 用自己的话解释 compute-optimal training。
2. 假设模型有 7B 参数，按 20 token per parameter 粗略估算训练 token 数。
3. 分析一个 100B 参数但只训练 300B token 的模型可能有什么问题。
4. 设计一个比较 3B、7B、13B 模型在固定 FLOPs 下的 scaling 实验。
5. 用 2 分钟回答“为什么 Chinchilla 之后大家更重视 token 数和数据质量”。

### 本讲总结

本讲最重要的结论：

1. Chinchilla Scaling Law 关注固定计算预算下 `N` 和 `D` 的最优分配。
2. 很多早期大模型参数量偏大、训练 token 偏少，容易 undertrained。
3. Compute-optimal training 要让模型参数量和训练 token 数更平衡地增长。
4. 常见经验是每个参数约对应 20 个训练 token，但不能机械套用。
5. 真实训练还必须考虑数据质量、unique token、重复 epoch、推理成本和后训练目标。

## 第 33 讲：Scaling Law 的局限与新趋势

### 本讲目标

学完这一讲，你应该能够回答：

1. Scaling Law 为什么有用，但为什么不能被当成“万能公式”。
2. 经典 Scaling Law 和 Chinchilla Scaling Law 在真实大模型训练中有哪些局限。
3. 为什么 loss 下降不一定等价于下游能力、对话能力、推理能力或产品体验提升。
4. 新一代 Scaling 研究正在从哪些方向扩展：数据质量、后训练、推理计算、合成数据、能力涌现和系统级 scaling。
5. 面试中如何既讲清 Scaling Law 的价值，又能体现工程判断和研究视野。

本讲非常适合面试开放题，例如：

> Scaling Law 是否还能指导今天的大模型训练？它有哪些不足？未来 scaling 的重点会是什么？

很多候选人会回答：“Scaling Law 说模型越大、数据越多、算力越多，loss 越低。”

这个回答只说对了一半。

更好的回答是：Scaling Law 是做大模型训练预算规划的重要工具，但它主要描述某些条件下预训练 loss 与参数量、数据量、算力之间的统计关系。真实模型能力还受到数据质量、任务分布、架构、优化稳定性、后训练、推理时计算和系统设计影响。所以今天的趋势不是抛弃 Scaling Law，而是把 scaling 从“参数和 token 的 scaling”扩展到“数据、后训练、推理、工具、多模态和系统”的 scaling。

### 一、先回顾：Scaling Law 到底在告诉我们什么

在前两讲中，我们已经讲过两条主线。

第一条是经典 Scaling Law：

模型越大、数据越多、计算越多，语言建模 loss 通常会按幂律下降。

第二条是 Chinchilla Scaling Law：

在固定训练计算量下，不能只盲目增大参数量，还要让训练 token 数同步增长。很多早期模型参数很大，但训练 token 不够，属于 undertrained。

用最简单的话说，Scaling Law 的价值是：

1. 帮助估算不同规模模型的预期 loss。
2. 帮助决定预算应该投给参数量、数据量还是训练步数。
3. 帮助在小规模实验上外推大规模训练结果。
4. 帮助避免拍脑袋式训练，例如“先训一个 100B 试试看”。

对大模型团队来说，这非常重要。

因为一次大模型预训练可能消耗数百万甚至上千万美元级别的算力。如果没有 scaling 规律，团队很难回答这些问题：

1. 是训 7B、13B，还是 34B？
2. 训练 1T token 够不够？
3. 如果预算翻倍，应该增加模型还是增加数据？
4. 小模型实验是否能预测大模型收益？
5. 继续训练是否还有足够边际收益？

所以 Scaling Law 不是“论文里的漂亮曲线”，而是大规模训练的预算语言。

但是问题也在这里：它主要是预算语言，不是完整的能力理论。

### 二、局限一：Scaling Law 通常预测的是 loss，不直接预测能力

经典 Scaling Law 最常研究的目标是语言建模 loss，也就是 next-token prediction 的交叉熵。

如果一个模型在验证集上的 loss 更低，通常说明它对文本分布建模更好。

但面试中一定要强调：

> loss 更低，不等于所有能力都同比例更强。

原因很简单。

预训练 loss 是一个平均指标，它把很多 token 的预测难度混在一起。模型在大量普通文本上预测得更好，会让 loss 下降；但用户真正关心的能力可能是：

1. 数学推理。
2. 代码生成。
3. 多轮对话。
4. 指令遵循。
5. 工具调用。
6. 长上下文信息检索。
7. 事实可靠性。
8. 安全拒答。

这些能力未必与整体预训练 loss 一一对应。

举一个直观例子。

假设两个模型 A 和 B：

1. A 的通用网页文本 loss 略低。
2. B 的代码、数学、指令数据占比更合理。

最终在聊天、代码和数学 benchmark 上，B 可能更好。

这不是违反 Scaling Law，而是说明你优化的 loss 分布和你评估的任务分布不完全一致。

工程上常见的现象是：

1. 预训练 loss 持续下降，但某些下游任务提升很小。
2. loss 差异很小，但 benchmark 差异明显。
3. loss 更低的模型，在人类偏好评估中未必更好。
4. base model loss 很好，但 instruction-following 很差。
5. chat model 体验很好，但 base LM loss 不是最低。

所以面试中不要把 Scaling Law 说成“能力曲线”。更准确的说法是：

> Scaling Law 主要预测某个训练分布和验证分布上的平均建模误差。它和能力相关，但不是能力的完整代理指标。

### 三、局限二：数据质量很难被一个 token 数 `D` 表示

Chinchilla 公式里常用 `D` 表示训练 token 数。

但真实训练中，1T token 和 1T token 的价值可能完全不同。

因为 token 数只描述数量，不描述质量。

高质量数据可能包含：

1. 结构清晰的教材。
2. 高质量代码仓库。
3. 数学推导。
4. 专业问答。
5. 高质量多轮对话。
6. 经过过滤的网页内容。
7. 多语言平衡语料。

低质量数据可能包含：

1. 重复模板。
2. 垃圾网页。
3. 机器翻译噪声。
4. 错误答案。
5. 过时内容。
6. 抓取失败的 HTML 片段。
7. 污染 benchmark 的样本。

如果只看 token 数，它们都是 token。

但对模型能力的贡献差异很大。

这就是 Scaling Law 的一个核心局限：

> 它把数据压缩成数量变量，但真实数据有质量、领域、难度、重复率、时效性和污染风险。

举一个训练方案对比：

方案 A：

1. 训练 2T token。
2. 数据清洗一般。
3. 去重不足。
4. 高质量代码和数学占比低。

方案 B：

1. 训练 1.2T token。
2. 数据质量高。
3. 去重严格。
4. 领域配比合理。
5. 有高质量代码、数学和指令风格数据。

从 `D` 看，A 更大。

但从最终能力看，B 可能更强。

这也是为什么今天很多团队会强调 data-centric scaling。

也就是：

> 不只是扩大数据量，而是提高每个 token 的平均训练价值。

常见做法包括：

1. 更强的数据过滤器。
2. 基于模型打分的数据选择。
3. 领域分类和动态配比。
4. 更严格的近重复去重。
5. 高质量合成数据。
6. 针对数学、代码、工具调用的专项数据。
7. 对低质量网页数据降权。

面试中可以总结为一句话：

> Chinchilla 告诉我们 token 数很重要，但没有告诉我们哪些 token 更值钱。

### 四、局限三：Scaling Law 依赖训练 recipe 和架构假设

Scaling Law 不是脱离上下文的自然定律。

它通常是在某些架构、优化器、数据分布和训练 recipe 下拟合出来的经验规律。

如果这些条件变化，曲线参数可能变化。

影响因素包括：

1. Transformer 架构细节。
2. 是否使用 RoPE、GQA、SwiGLU、RMSNorm。
3. 优化器选择，如 AdamW、Adafactor、Lion。
4. learning rate schedule。
5. batch size 和 gradient accumulation。
6. warmup 长度。
7. weight decay。
8. tokenizer。
9. 数据混合比例。
10. 上下文长度。
11. 是否使用 MoE。

如果一个团队基于旧架构拟合 scaling curve，然后直接用于新架构，可能会产生偏差。

例如：

1. MoE 模型的 activated parameters 和 total parameters 不同。
2. 长上下文模型的训练 token 分布和注意力成本不同。
3. 代码模型的数据分布与通用文本差异很大。
4. 多模态模型不仅有文本 token，还有图像 patch、音频 token 或视频 token。
5. 推理模型可能在预训练后还经过大规模强化学习或搜索式训练。

所以真实团队更常做的是：

1. 在自己的数据和 recipe 上做小规模 scaling 实验。
2. 拟合内部 scaling curve。
3. 用内部曲线指导下一轮训练。
4. 随着架构和数据变化持续校准。

不要把公开论文中的系数机械照搬到自己的项目。

公开 Scaling Law 更适合作为方向性参考，而不是精确预算表。

### 五、局限四：小规模外推到大规模并不总是可靠

Scaling Law 的一个重要用途是从小模型外推大模型。

例如先训练 100M、300M、1B，再预测 7B 或 70B。

这个思路很合理，但有几个风险。

第一个风险是小模型和大模型的瓶颈不同。

小模型可能主要受容量限制，大模型可能主要受数据质量限制。

第二个风险是训练稳定性在大规模下发生变化。

小模型训练很稳定，大模型可能出现：

1. loss spike。
2. 梯度爆炸。
3. NaN。
4. optimizer state 异常。
5. 分布式通信故障。
6. 数据 shard 问题。

第三个风险是能力可能存在非线性变化。

某些能力在小模型上几乎看不到，到一定规模后才明显出现。

这常被称为 emergence，也就是涌现。

虽然“涌现”这个词需要谨慎使用，因为有些现象可能来自评估指标的离散化或阈值效应，但工程上确实经常看到能力曲线不是完全平滑的。

第四个风险是数据重复和数据耗尽。

小规模实验可能只跑少量 token，重复率问题不明显。大规模训练时，如果高质量数据不足，就必须重复 epoch 或引入更多低质量数据，导致 scaling 曲线变化。

所以外推时不能只问“loss 曲线是否拟合得很好”，还要问：

1. 小模型实验的数据分布和大模型一致吗？
2. batch size 和学习率是否按规则缩放？
3. 大模型训练稳定性是否验证过？
4. 高质量数据是否足够？
5. 评估任务是否覆盖目标能力？
6. 外推范围是否过远？

如果只用 100M 到 1B 的曲线去预测 100B，风险很高。

### 六、局限五：预训练 Scaling Law 不覆盖后训练

今天的大模型产品通常不是直接把 base model 给用户用。

完整链路通常包括：

1. 预训练。
2. SFT。
3. 偏好优化，如 RLHF、DPO、IPO、KTO。
4. 安全对齐。
5. 工具调用训练。
6. 长上下文适配。
7. 领域微调。
8. 系统提示词和推理框架。

经典 Scaling Law 主要讨论预训练阶段。

但用户最终感知的是 chat model 或 agent system 的表现。

这中间有很长的后训练链路。

后训练会显著改变模型行为，例如：

1. 更会遵循指令。
2. 更偏好结构化回答。
3. 更安全。
4. 更少输出有害内容。
5. 更符合人类偏好。
6. 更会使用工具。
7. 更适合多轮对话。

但后训练也可能带来副作用：

1. 过度拒答。
2. 变得啰嗦。
3. 丢失一部分 base model 能力。
4. 出现 reward hacking。
5. 在某些 benchmark 上过拟合。

因此，一个预训练 loss 很强的 base model，如果后训练做得不好，最终产品体验仍然可能很差。

反过来，一个 base model 不一定最强，但后训练、工具系统和评估闭环做得好，也可能有很强的实际体验。

这说明：

> 预训练 scaling 是基础，但不是最终产品 scaling 的全部。

未来更重要的问题包括：

1. SFT 数据量和质量如何 scale？
2. preference data 如何 scale？
3. RL 训练中的 compute 如何 scale？
4. reward model 或 judge model 如何 scale？
5. reasoning trace 数据如何 scale？
6. tool-use 数据如何 scale？

这些问题已经超出了传统预训练 Scaling Law 的范围。

### 七、局限六：推理时计算越来越重要

传统 Scaling Law 主要关注训练计算。

但近年来，一个越来越重要的趋势是 inference-time scaling。

也就是：

> 不只在训练时投入更多计算，也在推理时投入更多计算，让模型有更多时间思考、搜索、验证和调用工具。

典型方式包括：

1. Chain-of-Thought。
2. Self-consistency。
3. Best-of-N sampling。
4. Tree search。
5. verifier reranking。
6. tool calling。
7. code execution。
8. retrieval augmented generation。
9. test-time compute scaling。

对于数学、代码和复杂推理任务，推理时多花计算可能显著提高结果质量。

例如同一个模型：

1. 直接回答一次，可能答错。
2. 生成 32 条思路，用 verifier 选最优，准确率可能明显提升。
3. 允许调用 Python 执行中间计算，复杂题表现又会提升。
4. 允许检索外部文档，事实类问题更可靠。

这说明能力不只来自参数和预训练 token，也来自推理策略。

面试中可以这样说：

> 过去 scaling 更关注 train-time compute，现在越来越多任务需要同时考虑 train-time compute 和 test-time compute 的分配。

一个重要开放问题是：

如果总预算固定，应该把钱花在更大模型、更长训练，还是推理时更多采样和搜索？

不同任务答案不同。

对于普通聊天，低延迟很重要，不能无限增加推理计算。

对于数学竞赛、代码修复、科研助手，推理时多花几秒甚至几十秒可能可以接受。

所以未来 scaling 可能不再只是：

```text
能力 = f(参数量, 训练 token, 训练 FLOPs)
```

而更像：

```text
能力 = f(参数量, 数据质量, 训练 FLOPs, 后训练质量, 推理计算, 工具系统, 评估反馈)
```

### 八、局限七：能力、可靠性和安全不一定同向 scaling

模型变大通常会带来更强能力。

但能力增强不等于可靠性和安全性自动增强。

例如更大的模型可能：

1. 更会编程，但也更会生成攻击代码。
2. 更会推理，但也更会为错误答案编造解释。
3. 更会遵循指令，但也可能更容易执行恶意指令。
4. 更懂知识，但幻觉仍然存在。
5. 更会使用工具，但工具误用风险更高。

所以安全、可靠性、可控性也需要单独 scaling。

这包括：

1. 更大规模 red teaming。
2. 更强安全数据。
3. 更可靠拒答边界。
4. 更好的不确定性表达。
5. 更强事实核查。
6. 更好的 tool-use sandbox。
7. 更系统的在线监控。

面试中如果被问“模型继续变大是不是就能解决幻觉”，不要简单回答“是”。

更准确的回答是：

> 规模增大会提升知识覆盖和推理能力，可能缓解一部分幻觉，但不能从根本上保证事实正确。幻觉还与训练目标、数据时效性、解码策略、检索能力、校验机制和不确定性建模有关。

这类回答会比单纯说“scale solves everything”更成熟。

### 九、新趋势一：Data Scaling 从数量转向价值密度

第一个新趋势是数据 scaling 的重心变化。

过去常见思路是：抓更多网页，清洗后训练更多 token。

现在高质量数据越来越稀缺，简单扩大 token 数的边际收益下降。

因此团队会更关注 value per token。

也就是每个 token 对能力提升的贡献。

常见方向包括：

1. 用小模型或大模型给数据打质量分。
2. 对不同领域建立 classifier。
3. 按能力目标调整数据配比。
4. 对高价值数据重复采样。
5. 对低价值数据降权或剔除。
6. 利用合成数据补足稀缺能力。
7. 构造更难、更有学习信号的数据。

例如数学数据 scaling 不只是收集更多题目，还要考虑：

1. 题目难度分布。
2. 解题过程是否正确。
3. 是否包含多种解法。
4. 是否覆盖不同知识点。
5. 是否有可验证答案。
6. 是否避免 benchmark 泄漏。

代码数据 scaling 也不只是更多 GitHub token，还要考虑：

1. repo 质量。
2. license。
3. 测试是否存在。
4. 代码是否可运行。
5. issue 和 PR 是否可用于修复任务。
6. 多语言覆盖。

所以未来的数据 scaling 更像数据工程、数据评估和主动数据生成的结合。

### 十、新趋势二：Synthetic Data Scaling

第二个趋势是合成数据规模化。

当高质量人类数据不足时，可以用强模型生成训练数据。

合成数据可以用于：

1. 指令微调。
2. 数学推理。
3. 代码修复。
4. 工具调用。
5. 多轮对话。
6. 安全拒答。
7. 长上下文问答。
8. 多语言低资源补齐。

它的价值是：

1. 成本低。
2. 可控性强。
3. 可以定向补齐能力短板。
4. 可以生成标准格式和中间过程。
5. 可以通过 verifier 或执行器自动过滤。

但它也有明显风险：

1. 错误会被放大。
2. 风格会变窄。
3. 模型可能学习到 teacher model 的偏差。
4. 过多合成数据可能降低多样性。
5. 如果过滤不严，可能让模型学会虚假的推理过程。

所以合成数据 scaling 的关键不是“生成越多越好”，而是：

1. 生成。
2. 验证。
3. 过滤。
4. 去重。
5. 配比。
6. 训练后评估。
7. 根据错误再生成。

这形成一个闭环。

在面试中可以说：

> 合成数据让数据 scaling 从被动收集转向主动构造，但它需要 verifier、执行环境和评估闭环，否则容易把噪声规模化。

### 十一、新趋势三：Post-training Scaling

第三个趋势是后训练 scaling。

早期大家更关注 base model 规模。

但现在，后训练对最终模型体验影响越来越大。

Post-training scaling 包括：

1. 更大规模 SFT 数据。
2. 更高质量偏好数据。
3. 更强 reward model 或 judge model。
4. 更长的 RL 训练。
5. 更复杂的任务环境。
6. 更大规模红队和安全数据。
7. 面向工具使用和代理任务的数据。

一个直观例子：

同一个 base model，经过不同后训练，可能变成完全不同的产品：

1. 通用聊天助手。
2. 代码助手。
3. 数学推理模型。
4. 搜索问答模型。
5. 企业知识库助手。
6. 工具调用 agent。

这些差异不是仅靠预训练 loss 解释的。

未来面试中可能会出现这样的问题：

> 如果你有一个强 base model，如何通过 post-training scaling 提升数学推理能力？

可以从以下角度回答：

1. 构造高质量数学 SFT 数据，包括题目、推理过程和最终答案。
2. 使用可验证答案的数据，减少错误 chain-of-thought。
3. 训练 reward model 或 verifier 判断解答正确性。
4. 使用 rejection sampling 或 RL 提升正确轨迹概率。
5. 引入 self-consistency 或 verifier-guided decoding。
6. 用分难度评估集监控能力提升。
7. 检查是否牺牲通用能力和安全性。

这个回答体现的是：预训练之后还有很多可 scale 的空间。

### 十二、新趋势四：Reasoning 和 Test-Time Compute Scaling

第四个趋势是 reasoning scaling。

很多复杂任务的瓶颈不是模型是否“知道”某个事实，而是能否进行多步推理、检查中间步骤、修正错误。

因此 test-time compute 变得重要。

常见方法包括：

1. 让模型生成更长推理过程。
2. 采样多条解法。
3. 使用 verifier 选择答案。
4. 把问题分解为子问题。
5. 调用代码执行器验证结果。
6. 使用搜索树探索不同路径。
7. 让模型反思和修正。

从 scaling 的角度看，这相当于增加推理阶段计算。

一个简单类比：

人类考试时，10 秒内回答和 10 分钟内回答表现不同。同一个人，多花时间检查也可能答得更好。

大模型也是类似。

当然，推理时计算不是免费午餐。

它会带来：

1. 更高延迟。
2. 更高推理成本。
3. 更复杂系统。
4. verifier 错误风险。
5. 多样本选择偏差。

所以工程上需要根据场景权衡。

普通聊天场景可能更关注低延迟。

复杂数学、代码、科研场景更能接受高 test-time compute。

### 十三、新趋势五：System Scaling 和 Agent Scaling

第五个趋势是系统级 scaling。

当模型接入工具、检索、代码执行、浏览器、数据库、工作流和多智能体协作后，能力不再只由单个模型 checkpoint 决定。

系统能力取决于：

1. 模型本身。
2. Prompt 和 system instruction。
3. 工具定义。
4. 函数调用协议。
5. 检索质量。
6. 记忆系统。
7. 任务规划。
8. 错误恢复。
9. 安全边界。
10. 评估和监控。

例如一个代码 agent 的能力取决于：

1. 是否能理解用户需求。
2. 是否能搜索代码库。
3. 是否能编辑文件。
4. 是否能运行测试。
5. 是否能读懂错误日志。
6. 是否能回滚或修复自己的错误。
7. 是否遵守权限和安全限制。

这已经不是简单的 LM loss scaling 能解释的。

因此未来可能出现更多 system-level scaling law，例如：

1. 工具数量与任务成功率的关系。
2. 检索质量与问答正确率的关系。
3. agent 轨迹长度与成功率的关系。
4. verifier 质量与推理能力的关系。
5. 多 agent 协作成本与收益的关系。

这些问题还没有像预训练 Scaling Law 那样成熟，但方向非常重要。

### 十四、工程上如何正确使用 Scaling Law

真实项目中，可以按下面方式使用 Scaling Law。

第一步：明确目标。

你要优化的是 base LM loss、代码能力、数学能力、对话体验、低延迟，还是综合产品体验？

不同目标对应不同 scaling 策略。

第二步：做小规模实验。

训练多个小模型，控制变量，例如：

1. 参数量。
2. token 数。
3. 数据配比。
4. learning rate。
5. batch size。

第三步：拟合内部曲线。

不要只看公开论文系数，要在自己的数据和 recipe 上拟合。

第四步：同时评估 loss 和能力。

至少包括：

1. validation loss。
2. 通用 benchmark。
3. 目标领域 benchmark。
4. 人类偏好或 LLM-as-a-judge。
5. 安全评估。
6. 长上下文评估。
7. 代码或数学可执行评估。

第五步：做预算决策。

决定增加参数、增加 token、提升数据质量、加强后训练还是增加推理计算。

第六步：训练中持续监控。

关注：

1. loss 曲线是否符合预期。
2. 是否出现 loss spike。
3. 下游评估是否同步提升。
4. 数据质量是否有异常。
5. 是否存在 benchmark contamination。
6. 是否需要调整数据配比。

第七步：训练后复盘。

把实际结果加入下一轮 scaling 预测。

这才是工程上健康的 scaling loop。

### 十五、一个面试回答模板

如果面试官问：

> 你怎么看 Scaling Law 的局限和未来趋势？

可以这样回答：

Scaling Law 对大模型训练非常重要，因为它提供了参数量、数据量、计算量和预训练 loss 之间的经验关系，可以指导预算分配和小规模外推。Chinchilla 进一步说明，在固定算力下不能只增大模型，还要匹配足够训练 token，避免 undertrained。

但它也有明显局限。第一，它主要预测 loss，不直接预测所有下游能力；loss 更低不一定代表指令遵循、数学推理、工具调用或人类偏好一定更好。第二，token 数不能代表数据质量，真实数据有质量、领域、重复率和污染风险。第三，Scaling Law 依赖具体架构、优化器和训练 recipe，公开曲线不能机械照搬。第四，小规模外推到大规模时会遇到训练稳定性、数据耗尽和能力非线性问题。第五，传统 Scaling Law 主要覆盖预训练，不能完整解释后训练、推理时计算和系统能力。

未来趋势我认为有几个方向：一是 data-centric scaling，从更多 token 转向更高 value per token；二是 synthetic data scaling，用生成、验证、过滤闭环构造高价值数据；三是 post-training scaling，包括 SFT、偏好优化、RL 和安全对齐；四是 test-time compute scaling，通过多样本、verifier、搜索和工具调用提升推理能力；五是 system-level scaling，也就是模型、检索、工具、agent harness 和评估监控共同决定最终能力。

所以我不会把 Scaling Law 理解成“模型越大越好”的口号，而会把它当成训练决策工具，并结合数据质量、后训练、推理成本和目标能力做综合判断。

### 十六、常见追问与回答

问题 1：既然 loss 不完全等于能力，那 Scaling Law 还有意义吗？

回答：有意义。loss 是预训练阶段最稳定、最可量化的指标之一，通常和很多能力正相关。问题不在于 loss 没用，而在于不能只看 loss。工程上应该同时看 validation loss、目标 benchmark、人类偏好、安全评估和推理成本。

问题 2：Scaling Law 是否说明只要继续加算力就能解决所有问题？

回答：不能。增加算力通常会提升能力，但数据质量、后训练、评估污染、安全性、推理策略和系统设计也很关键。很多问题不是单靠参数量增加就能解决，例如事实可靠性、工具安全、长期记忆和可控性。

问题 3：为什么数据质量会改变 scaling 结果？

回答：因为 token 数只是数量。高质量 token 提供更密集的学习信号，低质量或重复 token 的边际收益低，甚至会引入错误模式。相同 token 数下，数据质量不同会导致 loss 和下游能力差异很大。

问题 4：推理时计算为什么算一种 scaling？

回答：因为它通过增加 test-time compute 改变模型解决问题的过程。例如多采样、搜索、verifier、代码执行和工具调用都能用更多计算换更高准确率。它和训练时扩大模型不同，但同样是用计算换能力。

问题 5：你会如何设计一个 scaling 实验？

回答：我会先固定目标和评估集，然后训练多个规模的小模型，控制参数量、token 数和数据配比，记录 validation loss 和目标任务指标。接着拟合内部 scaling curve，并用中等规模模型验证外推是否可靠。最后再决定大规模训练时的模型大小、token 数、数据质量策略和后训练计划。

问题 6：如果 loss 下降但 benchmark 不提升，你会怎么排查？

回答：我会检查 benchmark 是否覆盖目标能力、是否存在评估噪声、数据分布是否错配、训练数据是否缺少相关领域、是否有过度重复、后训练是否影响能力，以及模型是否在平均 token prediction 上变好但没有学到任务所需技能。

### 十七、常见误区

1. 误区：Scaling Law 等于模型越大越好。
   纠正：Scaling Law 强调参数、数据和计算的关系，不是无条件扩大参数。

2. 误区：loss 越低，所有能力一定越强。
   纠正：loss 与能力相关，但不同任务有分布差异，后训练和评估方式也会影响结果。

3. 误区：token 数越多越好。
   纠正：低质量、重复或污染 token 的边际收益有限，甚至会伤害模型。

4. 误区：公开 Scaling Law 可以直接指导任何团队训练。
   纠正：不同数据、架构和 recipe 会改变曲线，需要内部实验校准。

5. 误区：预训练 Scaling Law 可以解释最终产品体验。
   纠正：最终体验还取决于 SFT、偏好优化、安全对齐、推理策略、工具系统和产品约束。

6. 误区：推理时多采样一定更好。
   纠正：多采样需要 verifier 或选择机制，也会增加成本和延迟。

### 十八、小练习

1. 用自己的话解释为什么 Scaling Law 主要是预算工具，而不是完整能力理论。
2. 举一个 loss 下降但下游能力不提升的可能原因。
3. 对比“更多 token”和“更高质量 token”在训练中的差异。
4. 设计一个小规模实验，验证数据配比对 scaling 曲线的影响。
5. 解释 test-time compute scaling 和 train-time scaling 的区别。
6. 用 2 分钟回答：“Scaling Law 未来会如何发展？”

### 本讲总结

本讲最重要的结论：

1. Scaling Law 是大模型训练预算和规模规划的重要工具，但不是完整的能力理论。
2. 经典 Scaling Law 主要预测预训练 loss，而 loss 不等价于所有下游能力和产品体验。
3. token 数不能代表数据质量，data-centric scaling 会越来越重要。
4. Scaling Law 依赖架构、数据和训练 recipe，公开曲线不能机械照搬。
5. 小规模外推到大规模时，要警惕训练稳定性、数据耗尽和能力非线性。
6. 未来 scaling 会扩展到合成数据、后训练、推理时计算、工具系统、Agent 和安全评估。
7. 面试中最成熟的回答不是“scale solves everything”，而是说明 scaling 有价值，但必须和数据、训练、评估、后训练、推理和系统设计结合使用。

## 第 34 讲：训练 loss、验证 loss 与下游能力的关系

### 本讲目标

学完这一讲，你应该能够回答：

1. 训练 loss、验证 loss、perplexity 和下游 benchmark 分别代表什么。
2. 为什么训练 loss 下降不一定说明模型真的更强。
3. 为什么验证 loss 比训练 loss 更重要，但也不能完全代表最终能力。
4. 如何判断 loss 曲线是否正常，如何识别过拟合、欠拟合、数据问题和训练不稳定。
5. 面试中如何解释“loss 降了，但 benchmark 没涨”或“benchmark 涨了，但 loss 变化不大”。

这一讲非常贴近真实大模型训练。

在训练模型时，团队每天都会看 loss 曲线。但如果只会说“loss 越低越好”，就还没有进入工程状态。

更成熟的理解是：

> 训练 loss 是模型在训练数据上的优化信号，验证 loss 是模型在 held-out 分布上的泛化信号，下游能力是模型在具体任务和交互场景中的表现。三者通常相关，但不是同一个东西。

面试中，很多关于训练监控、Scaling Law、实验分析、模型评估的问题，最后都会落到这一点。

### 一、先分清三个概念

大模型训练里最常见的三个指标是：

1. 训练 loss。
2. 验证 loss。
3. 下游能力指标。

训练 loss 指模型在训练 batch 上的 next-token prediction loss。

也就是模型看到训练样本后，对下一个 token 的预测有多准。

验证 loss 指模型在没有参与训练的验证集上的 next-token prediction loss。

它用于衡量模型是否真的学到了可泛化的语言规律，而不是只记住训练样本。

下游能力指标则是具体任务表现，例如：

1. MMLU 准确率。
2. GSM8K 数学准确率。
3. HumanEval 代码通过率。
4. MT-Bench 或 Arena 偏好胜率。
5. 长上下文检索准确率。
6. 多轮对话满意度。
7. 工具调用成功率。
8. 安全拒答准确率。

三者的关系可以粗略理解为：

```text
训练 loss：模型是否在训练数据上变好
验证 loss：模型是否在相似但未见数据上变好
下游能力：模型是否在目标任务上真的有用
```

它们之间通常有正相关。

如果训练完全失败，loss 很高，下游能力一般也不会好。

如果验证 loss 长期下降，很多通用能力通常会提升。

但它们不是严格等价关系。

这就是本讲的核心。

### 二、训练 loss：优化过程的直接信号

训练 loss 是训练过程中最直接的信号。

语言模型预训练通常使用交叉熵损失：

```text
L = - sum_t log p(x_t | x_{1:t-1})
```

意思是：对每个位置 `t`，模型根据前面的 token `x_{1:t-1}` 预测当前 token `x_t`，如果真实 token 的概率越高，loss 越低。

训练 loss 的作用包括：

1. 判断训练是否在正常优化。
2. 观察学习率是否合理。
3. 发现 loss spike、NaN、梯度爆炸等问题。
4. 评估模型是否欠拟合训练数据。
5. 监控不同数据 shard 或不同 domain 的难度。

例如正常训练时，训练 loss 通常应该整体下降。

它可能有 batch 级别波动，但滑动平均应该下降。

如果训练 loss 完全不降，可能说明：

1. learning rate 太低或太高。
2. 数据输入有 bug。
3. label shift 有问题。
4. attention mask 错误。
5. optimizer 没有正常 step。
6. loss 没有正确除以 token 数。
7. mixed precision 导致数值异常。

如果训练 loss 先下降后突然爆炸，可能说明：

1. learning rate 过高。
2. warmup 不足。
3. batch 中有异常样本。
4. 梯度裁剪不足。
5. 分布式训练状态异常。
6. optimizer state 损坏。

所以训练 loss 是 debug 训练过程的第一信号。

但它有一个天然缺陷：

> 训练 loss 只说明模型越来越适应训练数据，不说明它一定能泛化。

如果模型记住训练集，训练 loss 也会下降。

如果训练数据中混入重复样本，训练 loss 也可能下降得很好。

如果数据被污染，模型在训练集上也会表现很好。

因此，训练 loss 不能单独用来判断模型质量。

### 三、验证 loss：泛化能力的基础信号

验证 loss 是在 held-out validation set 上计算的 loss。

所谓 held-out，就是这些样本没有参与参数更新。

验证 loss 的作用是判断模型是否能泛化到未见数据。

在大模型预训练中，验证集通常要满足：

1. 与训练分布相近。
2. 不参与训练。
3. 不被去重误伤或泄漏。
4. 规模足够大，减少噪声。
5. 最好按 domain 分桶，例如 web、code、math、wiki、books。

验证 loss 比训练 loss 更能说明模型质量。

如果训练 loss 下降，验证 loss 也下降，通常说明模型确实学到了更通用的模式。

如果训练 loss 下降，但验证 loss 不降甚至上升，就可能出现过拟合或分布问题。

经典过拟合图像是：

```text
训练 loss：持续下降
验证 loss：先下降，后持平或上升
```

在小模型或小数据训练中，这很常见。

在 web-scale 大模型预训练中，由于数据量极大，传统意义上的严重过拟合不一定常见，但仍然会遇到类似问题，例如：

1. 高质量数据被重复训练太多遍。
2. domain 配比过窄。
3. benchmark 或验证集污染。
4. 训练后期加入低质量数据导致验证分布变差。
5. 某些 domain 的验证 loss 上升。

因此，大模型训练不应该只看一个总验证 loss。

更好的做法是看分 domain validation loss。

例如：

```text
web loss
code loss
math loss
wiki loss
books loss
multilingual loss
long-context loss
```

如果总验证 loss 下降，但 code loss 上升，说明模型整体变好，但代码能力可能受损。

如果英文 loss 下降，但低资源语言 loss 上升，说明多语言能力可能退化。

这类细分监控在真实训练中非常重要。

### 四、Perplexity：loss 的另一种表达

面试中经常会问 perplexity。

Perplexity，简称 PPL，通常定义为：

```text
PPL = exp(cross_entropy_loss)
```

如果使用自然对数，PPL 就是 `e^loss`。

直觉上，PPL 可以理解为模型在每个位置上的平均困惑程度。

例如：

1. PPL 越低，说明模型越不困惑。
2. PPL 越高，说明模型越难预测下一个 token。

如果 loss 是 2，那么：

```text
PPL = exp(2) ≈ 7.39
```

可以粗略理解为模型平均在约 7 个候选 token 之间困惑。

但这个解释只是直觉，不要过度物理化。

PPL 的问题和 loss 类似：

1. 它依赖 tokenizer。
2. 它依赖数据分布。
3. 它不能直接比较不同 tokenizer 的模型。
4. 它不能完全代表下游能力。

例如两个模型 tokenizer 不同，一个把中文切得更细，一个切得更粗，PPL 不能简单横向比较。

所以面试中可以说：

> Perplexity 是 cross-entropy loss 的指数形式，适合比较同 tokenizer、同验证集下的语言建模能力，但不适合跨 tokenizer、跨数据分布随意比较。

### 五、为什么 loss 下降不一定带来下游能力提升

这是本讲最重要的问题。

原因一：训练分布和下游任务分布不同。

预训练数据可能主要是网页、书籍、代码、百科、论坛。

下游任务可能是考试题、数学推理、多轮对话、工具调用。

如果目标任务在训练分布中占比很低，整体 loss 的下降可能主要来自普通文本，而不是目标能力。

例如模型在大量网页文本上预测更准，会降低总 loss，但 GSM8K 数学能力未必明显提升。

原因二：loss 是 token-level 平均，benchmark 是 task-level 成败。

语言模型 loss 按 token 平均。

但很多 benchmark 是整题答对才算对。

数学题中，一个中间步骤错了，最终答案就错。

代码题中，一个 token 错误可能导致测试不通过。

所以很小的 token-level 改善，不一定立刻反映在 task-level accuracy 上。

反过来，一些关键能力的提升可能让 benchmark 明显上涨，但平均 loss 变化很小。

原因三：下游任务需要解码策略。

预训练 loss 衡量的是 teacher forcing 下真实 token 的概率。

但实际生成时，模型根据自己的输出继续生成。

这会产生 exposure bias。

同时，下游表现还受解码参数影响：

1. temperature。
2. top-p。
3. top-k。
4. beam search。
5. repetition penalty。
6. max tokens。
7. stop sequence。

同一个模型，不同解码策略可能导致 benchmark 差异。

原因四：后训练改变行为。

SFT 和偏好优化会显著影响指令遵循和对话体验。

一个 base model 的 validation loss 很好，但没做 SFT 时可能不会按用户指令回答。

另一个模型 base loss 稍差，但 post-training 很好，用户体验可能更强。

原因五：benchmark 可能有噪声或污染。

如果 benchmark 样本进入训练集，模型可能记住答案，benchmark 虚高。

如果 benchmark 太小或评估模板不稳定，分数也会有噪声。

因此 benchmark 和 loss 都需要谨慎解释。

### 六、为什么 benchmark 提升但 loss 变化不大

另一个常见现象是：

> 某个能力 benchmark 提升明显，但整体 validation loss 几乎没变。

这也很正常。

原因是整体 loss 是在大规模混合分布上平均的。

如果你通过少量高质量数学数据提升了数学能力，数学 token 在总验证集里可能占比很小，所以总 loss 变化不明显。

举个例子：

一个验证集有 100B token 的分布概念，其中普通网页占 80%，代码占 10%，数学占 2%。

如果数学能力提升很多，但数学只占 2%，总 loss 的变化可能被稀释。

这就是为什么要看 domain-specific loss 和 task-specific eval。

同理：

1. 代码能力提升，看 code eval 和 code validation loss。
2. 数学能力提升，看 math eval 和 math validation loss。
3. 多语言能力提升，看各语言分桶 loss。
4. 长上下文能力提升，看长上下文任务，而不是普通短文本 loss。

面试中可以总结：

> 总 loss 是宏观信号，能力评估是局部信号。局部能力显著提升，不一定会改变宏观平均 loss。

### 七、如何读懂 loss 曲线

训练中常见的 loss 曲线有几种。

第一种：健康下降。

```text
训练 loss：稳定下降
验证 loss：稳定下降
下游指标：整体提升
```

这是理想情况。

第二种：欠拟合。

```text
训练 loss：较高且下降慢
验证 loss：也较高
```

可能原因：

1. 模型太小。
2. 训练步数不够。
3. learning rate 不合适。
4. optimizer 配置不合理。
5. 数据太难或噪声太多。

第三种：过拟合。

```text
训练 loss：持续下降
验证 loss：不降或上升
```

可能原因：

1. 数据量太少。
2. 重复训练太多。
3. 正则不足。
4. 数据分布太窄。
5. 模型容量相对数据过大。

第四种：训练不稳定。

```text
训练 loss：突然尖峰
验证 loss：波动异常
```

可能原因：

1. learning rate 太高。
2. batch 中有异常数据。
3. FP16 溢出。
4. 梯度裁剪不足。
5. 分布式训练通信异常。
6. checkpoint 恢复错误。

第五种：训练 loss 降，验证 loss 降，但下游不涨。

可能原因：

1. 下游任务分布不在验证集中。
2. benchmark 太难，需要 threshold 才体现。
3. 解码策略不合适。
4. 缺少后训练。
5. 数据配比没有覆盖目标能力。

第六种：总验证 loss 降，但某些能力退化。

可能原因：

1. 总体数据配比变化。
2. 某些 domain 数据被稀释。
3. 训练对某些语言或领域产生遗忘。
4. 后训练让模型行为偏向某种风格。

这些模式是训练 debug 的基础。

### 八、分桶验证：不要只看一个平均 loss

一个平均验证 loss 会隐藏很多问题。

假设你看到：

```text
overall validation loss: 1.85 -> 1.82
```

看起来模型变好了。

但分桶后可能是：

```text
web loss: 1.90 -> 1.84
code loss: 1.50 -> 1.58
math loss: 1.70 -> 1.75
multilingual loss: 2.10 -> 2.25
```

这说明整体提升主要来自 web，而代码、数学、多语言反而退化。

如果你的目标是通用聊天，可能还能接受。

如果目标是代码助手或数学模型，这就是严重问题。

所以真实训练中常用的 validation buckets 包括：

1. 通用网页。
2. 百科。
3. 书籍。
4. 代码。
5. 数学。
6. 学术论文。
7. 多语言。
8. 对话。
9. 长上下文。
10. 安全相关文本。

更进一步，还可以按数据质量分桶：

1. 高质量人工文本。
2. 低质量网页。
3. 合成数据。
4. 去重前后数据。
5. 不同来源数据。

这样才能知道模型到底在哪些地方变好。

### 九、训练 loss 和验证 loss 的工程坑

坑一：loss 统计口径不一致。

例如：

1. 有的 loss 按 token 平均。
2. 有的 loss 按 sequence 平均。
3. 有的包含 padding token。
4. 有的忽略 prompt 部分。
5. 有的只算 answer 部分。

如果统计口径不同，loss 就不能直接比较。

特别是在 SFT 中，经常有两种做法：

1. prompt 和 answer 都算 loss。
2. 只对 answer 算 loss。

这两个 loss 的数值不可直接比较。

坑二：tokenizer 不同。

不同 tokenizer 会改变 token 粒度。

同一段文本，如果 tokenizer 切得更细，token 数更多，loss 和 PPL 都会变化。

所以不要随便比较不同 tokenizer 模型的 PPL。

坑三：验证集泄漏。

如果验证集样本进入训练数据，验证 loss 会虚低。

这会让团队误以为模型泛化很好。

坑四：验证集太小。

小验证集噪声大，容易误判。

尤其是下游 benchmark，如果样本量很小，几道题变化就会带来明显分数波动。

坑五：数据顺序和 batch 难度导致短期波动。

训练 loss 是 batch-level 指标，会受当前 batch 难度影响。

不要对单个 step 的 loss 过度解读，要看滑动平均。

坑六：混合精度和 loss scale 问题。

FP16/BF16 训练时，如果数值溢出或 loss scale 异常，loss 曲线可能出现尖峰或 NaN。

坑七：继续预训练和 SFT 的 loss 不能直接比较。

预训练 loss 是 next-token prediction，SFT loss 往往只关注回答部分，数据分布也不同。

所以它们的数值没有直接可比性。

### 十、下游能力如何和 loss 联合分析

比较成熟的实验分析不是只看一个指标，而是建立指标矩阵。

例如训练一个通用 LLM，可以同时看：

1. overall validation loss。
2. domain validation loss。
3. MMLU。
4. GSM8K 或 MATH。
5. HumanEval 或 MBPP。
6. 多语言 benchmark。
7. 长上下文检索。
8. 指令遵循评估。
9. 人类偏好评估。
10. safety eval。

然后按现象分析。

现象一：loss 降，所有 benchmark 涨。

说明训练整体健康。

现象二：loss 降，只有通用 benchmark 涨，代码数学不涨。

说明通用建模能力提升，但专项能力数据不足或配比不够。

现象三：overall loss 不变，代码 benchmark 涨。

说明局部能力提升被整体平均稀释，应该看 code bucket。

现象四：benchmark 涨，验证 loss 变差。

可能是过拟合 benchmark、评估污染、后训练改变输出风格，或优化目标从语言建模转向任务表现。

现象五：loss 很好，人类偏好差。

可能是缺少 SFT/RLHF，模型会预测文本但不会按用户意图组织回答。

现象六：人类偏好好，知识问答差。

可能是后训练让回答风格变好，但事实能力或检索能力不足。

这种联合分析比单独讨论 loss 更接近真实工作。

### 十一、代码示例：计算交叉熵 loss 和 perplexity

下面是一个简化 PyTorch 示例，展示如何计算语言模型 loss 和 PPL。

```python
import math
import torch
import torch.nn.functional as F


def lm_loss_and_ppl(logits: torch.Tensor, input_ids: torch.Tensor) -> tuple[torch.Tensor, float]:
    """Compute next-token cross entropy and perplexity.

    logits: [batch, seq_len, vocab_size]
    input_ids: [batch, seq_len]
    """
    shift_logits = logits[:, :-1, :].contiguous()
    shift_labels = input_ids[:, 1:].contiguous()

    loss = F.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)),
        shift_labels.view(-1),
        reduction="mean",
    )
    ppl = math.exp(loss.item())
    return loss, ppl
```

这里有几个细节很重要：

1. `logits[:, :-1, :]` 用前面的 token 预测后面的 token。
2. `input_ids[:, 1:]` 是预测目标。
3. loss 是所有 token 的平均交叉熵。
4. PPL 是 loss 的指数。

如果有 padding，需要忽略 padding label：

```python
loss = F.cross_entropy(
    shift_logits.view(-1, shift_logits.size(-1)),
    shift_labels.view(-1),
    ignore_index=-100,
    reduction="mean",
)
```

在 SFT 中，常见做法是把 prompt 部分 label 设为 `-100`，只训练 answer 部分。

这就是为什么 SFT loss 和预训练 loss 不能直接比较。

### 十二、一个真实项目中的分析例子

假设你训练了两个模型。

模型 A：

1. overall validation loss = 1.72。
2. MMLU = 68。
3. GSM8K = 42。
4. HumanEval = 28。
5. Arena 风格偏好一般。

模型 B：

1. overall validation loss = 1.75。
2. MMLU = 67。
3. GSM8K = 55。
4. HumanEval = 38。
5. Arena 风格偏好更好。

如果只看 overall loss，A 更好。

但如果目标是代码和数学助手，B 可能更适合。

为什么？

可能原因是：

1. B 的数学和代码数据质量更高。
2. B 的后训练更贴近目标任务。
3. B 的解码策略更适合推理。
4. A 在普通文本上更强，但专项能力弱。

因此模型选择不能只看 loss，要看目标场景。

另一个例子。

你发现某次继续预训练后：

```text
overall loss: 1.80 -> 1.76
code loss: 1.45 -> 1.52
HumanEval: 32 -> 25
```

这说明整体变好了，但代码能力退化。

可能是新数据中代码占比下降，或者代码数据质量变差。

解决方法可能是：

1. 增加高质量代码数据比例。
2. 对代码 domain 单独监控。
3. 在训练后期做 code-heavy continued pretraining。
4. 使用代码 benchmark 做 early stopping 或 checkpoint selection。

### 十三、Checkpoint 选择：选最低 loss 还是最高 benchmark？

真实训练中经常要选择 checkpoint。

最简单的做法是选 validation loss 最低的 checkpoint。

但这不一定总是最优。

如果目标是 base model，验证 loss 是很重要的选择依据。

如果目标是 chat model 或代码模型，还要看下游能力。

例如：

1. checkpoint-100k 的 overall loss 最低。
2. checkpoint-90k 的数学 benchmark 最高。
3. checkpoint-95k 的人类偏好最好。
4. checkpoint-100k 出现轻微安全退化。

应该选哪个？

答案取决于产品目标。

更稳妥的方式是建立 checkpoint selection policy：

1. base model：优先看 validation loss 和基础 benchmark。
2. code model：重点看代码验证集和可执行 benchmark。
3. chat model：重点看指令遵循、人类偏好和安全评估。
4. reasoning model：重点看数学、代码、推理和 verifier 结果。
5. online product：还要看延迟、成本、稳定性和安全约束。

不要机械地说“选 loss 最低”。

### 十四、面试回答模板

如果面试官问：

> 训练 loss、验证 loss 和下游能力是什么关系？

可以这样回答：

训练 loss 是模型在训练 batch 上的优化信号，主要用于判断训练是否正常、是否在拟合训练数据。验证 loss 是在未参与训练的数据上计算的 loss，更能反映语言建模泛化能力。下游能力是模型在具体任务上的表现，比如数学、代码、指令遵循、安全和人类偏好。

三者通常相关，但不等价。训练 loss 下降不一定代表泛化变好，因为可能过拟合或数据重复；验证 loss 下降通常是好信号，但它主要衡量验证分布上的平均 token prediction，不一定直接代表所有下游任务。下游任务可能和验证分布不同，而且很多任务是 task-level 成败，受后训练、解码策略和评估方式影响。

所以真实训练中我会联合看训练 loss、overall validation loss、domain-specific validation loss 和下游 benchmark。如果 loss 降但 benchmark 不涨，我会检查任务分布是否被训练数据覆盖、是否需要后训练、是否解码策略不合适、是否评估噪声或 benchmark 太难。如果 benchmark 涨但 loss 不变，我会考虑局部能力提升被总体平均稀释，需要看对应 domain 的 loss。

总结来说，loss 是训练和泛化的基础信号，但最终模型选择要结合目标场景和多维评估。

### 十五、常见追问与回答

问题 1：训练 loss 和验证 loss 哪个更重要？

回答：训练 loss 对 debug 优化过程更重要，验证 loss 对判断泛化更重要。如果只判断模型质量，验证 loss 通常更关键；但训练 loss 可以帮助发现学习率、数据、mask、数值稳定性等训练问题。

问题 2：为什么 validation loss 低，但聊天体验不好？

回答：validation loss 衡量的是 next-token prediction，聊天体验还需要指令遵循、回答结构、人类偏好、安全边界和多轮对话能力。这些主要依赖 SFT、偏好优化和对话数据，不是 base model validation loss 能完全决定的。

问题 3：能否用 perplexity 比较两个模型谁更强？

回答：只有在同 tokenizer、同验证集、同统计口径下，PPL 才有较强可比性。不同 tokenizer 或不同数据分布下，PPL 不能直接比较。即使 PPL 更低，也不一定代表所有下游能力更强。

问题 4：如果训练 loss 下降，验证 loss 上升，说明什么？

回答：可能是过拟合，也可能是训练数据和验证数据分布不一致，或者验证集统计有问题。需要检查数据重复、训练步数、数据配比、正则化、验证集质量和分 domain loss。

问题 5：为什么有时总 loss 下降，但代码能力下降？

回答：因为总 loss 是混合分布平均。普通文本占比可能很高，代码占比很低。模型在普通文本上提升可以拉低总 loss，但如果代码数据被稀释或质量下降，代码能力仍可能退化。

问题 6：SFT loss 越低越好吗？

回答：不一定。SFT loss 太低可能表示模型强拟合示范答案，但不一定人类偏好更好。SFT 还要看指令遵循、泛化、回答风格、安全性和偏好评估。尤其不同 mask 策略下的 SFT loss 不可直接比较。

问题 7：如何判断 loss spike 是不是严重问题？

回答：要看 spike 的幅度、持续时间、是否伴随梯度异常、是否影响验证 loss、是否出现 NaN，以及是否能从 checkpoint 恢复。如果只是单个难 batch 引起的小尖峰，可能不严重；如果持续扩大或伴随数值异常，就需要立即排查。

### 十六、常见误区

1. 误区：训练 loss 越低，模型越好。
   纠正：训练 loss 只说明训练集拟合更好，模型质量还要看验证 loss 和下游评估。

2. 误区：验证 loss 越低，所有任务都越强。
   纠正：验证 loss 是平均语言建模指标，不等于所有具体能力。

3. 误区：PPL 可以跨 tokenizer 随便比较。
   纠正：PPL 强依赖 tokenizer 和验证集，跨 tokenizer 比较风险很大。

4. 误区：loss 降但 benchmark 不涨，说明训练没用。
   纠正：可能是 benchmark 不覆盖、任务太局部、解码不合适，或能力还没越过任务阈值。

5. 误区：benchmark 涨就一定说明模型真的更强。
   纠正：需要排查污染、评估噪声、模板过拟合和统计显著性。

6. 误区：checkpoint 一定选 validation loss 最低的。
   纠正：要根据目标场景联合选择，chat、code、reasoning 和 base model 的标准不同。

### 十七、小练习

1. 用自己的话解释训练 loss 和验证 loss 的区别。
2. 为什么 validation loss 下降不一定代表数学能力提升？
3. 举一个 benchmark 提升但 overall loss 几乎不变的例子。
4. 设计一个分 domain validation loss 面板，至少包含 6 个 bucket。
5. 写出 PPL 和 cross-entropy loss 的关系。
6. 用 2 分钟回答：“loss 降了但下游能力没涨，你会怎么排查？”

### 本讲总结

本讲最重要的结论：

1. 训练 loss 是训练优化信号，验证 loss 是泛化信号，下游能力是任务表现信号。
2. 三者通常相关，但不是同一个东西。
3. 训练 loss 下降可能只是更好拟合训练数据，不一定代表模型泛化更强。
4. 验证 loss 更重要，但它仍然只是验证分布上的平均 token-level 指标。
5. 下游能力受任务分布、后训练、解码策略、评估噪声和 benchmark 污染影响。
6. 真实训练中要看 overall loss、domain loss、benchmark、人类偏好和安全评估的组合。
7. 面试中成熟的回答是：loss 是基础信号，但模型能力判断必须结合目标任务和多维评估。

## 第 35 讲：训练稳定性：loss spike、NaN 与梯度爆炸

### 本讲目标

学完这一讲，你应该能够回答：

1. 什么是 loss spike、NaN、Inf 和梯度爆炸。
2. 大模型训练为什么比普通深度学习更容易出现稳定性问题。
3. 常见稳定性问题来自哪些层面：数据、优化器、学习率、混合精度、模型结构、分布式训练和 checkpoint。
4. 训练中遇到 loss spike 或 NaN 时，应该按什么顺序排查。
5. 面试中如何系统回答“大模型训练不稳定怎么办”。

这一讲是训练工程面试高频内容。

很多候选人知道 AdamW、warmup、gradient clipping，但一遇到真实问题就只会说“调小学习率”。

调小学习率确实经常有效，但不是完整答案。

真实的大模型训练稳定性问题，需要从数据、数值、优化、结构和分布式系统一起看。

### 一、什么是训练不稳定

训练不稳定不是一个单一问题，而是一组现象。

常见现象包括：

1. loss spike：loss 突然出现尖峰。
2. loss divergence：loss 持续上升，训练发散。
3. NaN loss：loss 变成 not a number。
4. Inf loss：loss 变成 infinity。
5. gradient explosion：梯度范数突然变得极大。
6. activation explosion：中间激活值异常放大。
7. optimizer state 异常：Adam 的二阶矩估计出现异常。
8. 某些 rank loss 异常：分布式训练中只有部分 GPU 出问题。
9. checkpoint 恢复后 loss 跳变。

其中最常见、也最容易在面试中被问到的是 loss spike、NaN 和梯度爆炸。

可以先用直觉理解。

正常训练像是在下山。

loss spike 像是路上突然踩到坑，loss 瞬间跳高，但可能还能恢复。

loss divergence 像是方向完全错了，越走越高。

NaN 像是地图坏了，已经不知道当前位置。

梯度爆炸像是一步迈得太大，直接跳出合理区域。

### 二、loss spike 不一定等于训练失败

很多初学者看到 loss spike 会立刻认为训练崩了。

其实不一定。

loss spike 可以分为几类。

第一类是短暂尖峰。

例如某个 step 的 loss 突然升高，但后面很快恢复。

这可能只是一个困难 batch、异常长文本、特殊 domain 或统计波动。

第二类是持续尖峰。

loss 上升后没有恢复，后续继续变差。

这更危险，可能说明优化已经偏离稳定区域。

第三类是周期性尖峰。

每隔一段时间出现，可能和数据 shard、学习率 schedule、checkpoint 保存、评估切换或分布式通信有关。

第四类是伴随梯度异常的尖峰。

如果 loss spike 同时伴随 grad norm 暴涨、activation max 暴涨、NaN 或 Inf，就需要立即处理。

工程上判断 spike 是否严重，要看：

1. 持续了多少 step。
2. 是否自动恢复。
3. validation loss 是否受影响。
4. grad norm 是否异常。
5. 是否出现 NaN/Inf。
6. 是否只发生在某些数据 shard。
7. 是否只发生在某些 GPU rank。
8. checkpoint 恢复后是否可复现。

所以面试中不要简单说“loss spike 就调学习率”。更好的回答是先分类，再定位原因。

### 三、NaN 和 Inf 为什么危险

NaN 是 not a number。

Inf 是 infinity。

它们通常来自数值计算异常。

常见来源包括：

1. 除以 0。
2. `log(0)`。
3. `exp` 溢出。
4. FP16 数值范围不够。
5. attention score 过大。
6. softmax 输入异常。
7. 梯度爆炸导致参数更新过大。
8. optimizer state 中出现 NaN。

NaN 的麻烦在于它会传播。

一旦某个参数变成 NaN，后续 forward、loss、gradient 很可能都会变成 NaN。

在分布式训练中，如果一个 rank 出现 NaN，通过 all-reduce 后可能污染所有 rank。

所以训练中通常要监控：

1. loss 是否 finite。
2. grad norm 是否 finite。
3. parameters 是否 finite。
4. optimizer state 是否 finite。
5. activation 是否 finite。

遇到 NaN 时，第一原则是：

> 不要继续盲训，要从最近的健康 checkpoint 恢复，并定位 NaN 首次出现的位置。

如果继续训练，通常只是浪费算力。

### 四、梯度爆炸的直觉和表现

梯度表示 loss 对参数的变化方向和幅度。

如果梯度很大，一次参数更新就可能把模型推到很差的位置。

梯度爆炸常见表现：

1. grad norm 突然增大几个数量级。
2. loss 出现 spike。
3. 参数范数突然变化。
4. activation 值异常变大。
5. mixed precision 溢出。
6. 后续出现 NaN 或 Inf。

为什么大模型容易遇到梯度爆炸？

原因包括：

1. 网络很深，误差传播路径长。
2. 参数量巨大，小问题容易被放大。
3. batch 中可能出现极端样本。
4. learning rate 和 batch size 较大。
5. mixed precision 数值范围有限。
6. attention logits 可能非常大。
7. 分布式训练中不同 rank 数据差异大。

梯度爆炸最常见的防线是 gradient clipping。

例如按全局范数裁剪：

```text
if ||g|| > threshold:
    g = g * threshold / ||g||
```

PyTorch 中常用：

```python
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
```

但 gradient clipping 不是万能的。

它能限制更新幅度，但不能解决所有根因。

如果数据有 bug、loss 计算错、attention mask 错，裁剪只能掩盖问题。

### 五、根因一：学习率和 warmup 不合理

学习率是训练稳定性最重要的因素之一。

learning rate 太高，会导致参数更新过大，训练发散。

warmup 太短，会导致训练初期模型还没稳定，就被大步长冲坏。

大模型训练常用 warmup 的原因是：

1. 初始参数随机，梯度统计还不稳定。
2. Adam 的一阶、二阶矩估计还没充分建立。
3. 大 batch 训练初期更容易震荡。
4. 混合精度训练初期更容易溢出。

典型学习率策略包括：

1. linear warmup + cosine decay。
2. linear warmup + linear decay。
3. constant LR with warmup。

如果训练初期频繁 spike，可以考虑：

1. 降低 peak learning rate。
2. 增加 warmup steps。
3. 减小 batch size 或调整 global batch。
4. 开启或加强 gradient clipping。
5. 检查初始化和 normalization。

如果训练中后期 spike，要检查：

1. schedule 是否有突变。
2. 数据分布是否切换。
3. 是否从 checkpoint 恢复后 LR 状态错误。
4. optimizer state 是否正确加载。

### 六、根因二：数据异常

数据问题是 loss spike 的常见来源。

异常数据包括：

1. 超长样本截断错误。
2. 全是特殊 token 的样本。
3. 空样本。
4. tokenizer 失败样本。
5. 重复异常字符串。
6. 非常罕见的字符分布。
7. HTML、二进制、乱码。
8. label 和 input 错位。
9. attention mask 错误。

一个异常 batch 可能导致 loss 明显高于平均值。

如果异常样本刚好触发数值极端情况，还可能导致梯度爆炸。

工程上应该记录：

1. 当前 step。
2. 数据 shard id。
3. sample id。
4. sequence length。
5. token 分布。
6. loss top-k 样本。
7. 是否包含异常特殊 token。

如果 loss spike 可复现，最好的排查方式是回放该 step 的 batch。

如果回放同一个 batch 必然 spike，问题大概率在数据或该 batch 的数值路径。

如果回放不复现，可能是分布式、随机性、硬件或 optimizer 状态问题。

### 七、根因三：混合精度数值溢出

大模型训练通常使用 FP16、BF16 或 FP8 来节省显存和提升吞吐。

混合精度带来效率，也带来数值风险。

FP16 的动态范围较小，容易出现 overflow。

BF16 的指数范围接近 FP32，通常比 FP16 更稳定，但精度较低。

FP8 更激进，需要更复杂的 scaling 和校准。

常见问题包括：

1. attention logits 太大，softmax 前溢出。
2. loss scale 不合适。
3. gradient underflow 或 overflow。
4. LayerNorm/RMSNorm 统计异常。
5. optimizer state 精度不足。

常见处理方式：

1. 优先使用 BF16 而不是 FP16，如果硬件支持。
2. 对 FP16 使用 dynamic loss scaling。
3. 关键计算保持 FP32，例如 loss、norm、optimizer state。
4. 使用稳定的 softmax 实现。
5. 对 attention logits 做合理 scaling。
6. 监控 overflow 次数。

面试中可以说：

> 混合精度不是简单把所有 tensor 转成半精度，而是要明确哪些计算可以低精度，哪些状态必须高精度保存。

### 八、根因四：模型结构和初始化问题

模型结构也会影响稳定性。

常见因素包括：

1. Pre-LN 比 Post-LN 更容易训练深层 Transformer。
2. RMSNorm 通常比复杂归一化更简洁稳定。
3. residual scaling 可以缓解深层网络激活放大。
4. 初始化过大可能导致初期 activation 爆炸。
5. attention logits scale 不合理会导致 softmax 饱和。
6. MoE gating 不稳定会导致 expert 负载不均。

为什么 Pre-LN 更稳定？

因为它在子层前做归一化，让进入 attention 和 FFN 的输入分布更稳定，梯度路径也更顺畅。

Post-LN 在很深的 Transformer 中更容易训练困难，需要额外技巧，例如 DeepNorm 或 carefully tuned initialization。

对于超深模型，还可能使用：

1. residual scaling。
2. μParam 或类似参数化方法。
3. 更保守初始化。
4. attention output scaling。
5. layer-wise learning rate 调整。

如果一个架构在小模型上稳定，不代表放大到 70B、100B 后仍然稳定。

这也是为什么大模型训练前要做 scaling experiment。

### 九、根因五：优化器状态异常

AdamW 不只是保存参数，还保存一阶矩和二阶矩。

如果 optimizer state 异常，训练会出现奇怪问题。

常见场景：

1. checkpoint 恢复时只恢复了 model weights，没有恢复 optimizer state。
2. learning rate scheduler step 数恢复错误。
3. ZeRO/FSDP 分片状态恢复错位。
4. optimizer state 中已经有 NaN。
5. weight decay 错误应用到 bias 或 norm 参数。

例如从 checkpoint 恢复后 loss 突然跳变，可能不是模型参数问题，而是 optimizer 或 scheduler 状态问题。

排查时要确认：

1. model state 是否完整加载。
2. optimizer state 是否完整加载。
3. scheduler state 是否完整加载。
4. random seed 和 dataloader state 是否恢复。
5. gradient scaler state 是否恢复。
6. global step 是否正确。

大模型训练中，checkpoint 不是一个简单的 `.pt` 文件，而是模型、优化器、调度器、随机状态、数据进度和分布式元数据的组合。

### 十、根因六：分布式训练问题

大模型训练通常使用多 GPU、多节点。

分布式系统会引入额外不稳定来源。

常见问题包括：

1. 某个 rank 数据异常。
2. 某个 rank 出现 NaN 后 all-reduce 污染全局梯度。
3. gradient accumulation step 不一致。
4. ZeRO/FSDP 参数分片恢复错误。
5. 通信超时或 silent error。
6. 不同 rank 的 batch size 不一致。
7. padding 和 mask 处理不一致。
8. pipeline parallel 的 microbatch 调度错误。

排查分布式问题时，要避免只看 rank 0 的日志。

应该记录每个 rank 的：

1. loss。
2. grad norm。
3. activation max。
4. data shard id。
5. NaN/Inf 检查结果。
6. step time。

如果只有某个 rank 先异常，根因可能在该 rank 的数据或硬件。

如果所有 rank 同时异常，可能是全局学习率、模型结构、同步后的梯度或 optimizer 状态问题。

### 十一、遇到 NaN 的排查流程

面试中很可能被问：

> 训练中突然出现 NaN，你怎么排查？

可以按下面流程回答。

第一步：停止继续训练，保存现场。

记录 step、loss、grad norm、LR、batch id、rank id、overflow 信息。

第二步：从最近健康 checkpoint 恢复。

不要从 NaN checkpoint 继续训练。

第三步：尝试复现。

用相同 seed、相同数据顺序、相同 batch 回放 NaN 前几个 step。

第四步：检查数值 finite。

检查 loss、logits、activation、gradient、parameters、optimizer state 是否出现 NaN/Inf。

第五步：定位层和操作。

通过 hooks 或 debug log 找到最早出现 NaN 的层，例如 attention、MLP、norm 或 loss。

第六步：检查数据 batch。

看是否有空样本、异常 token、超长样本、mask 错误或 label 错位。

第七步：检查超参数。

重点看 learning rate、warmup、batch size、gradient clipping、weight decay。

第八步：检查 mixed precision。

看是否 FP16 overflow、loss scale 异常，是否需要 BF16 或更稳定 kernel。

第九步：检查分布式状态。

看是否某个 rank 先异常，是否 all-reduce 后传播。

第十步：修复后从健康 checkpoint 重训。

常见修复包括降低 LR、增加 warmup、开启 clipping、过滤异常数据、切换 BF16、修复 mask、修复 checkpoint 恢复逻辑。

### 十二、监控指标：不只看 loss

为了提前发现训练不稳定，需要监控多类指标。

基础指标：

1. train loss。
2. validation loss。
3. learning rate。
4. tokens per second。
5. step time。

梯度指标：

1. global grad norm。
2. per-layer grad norm。
3. grad max。
4. grad NaN/Inf count。
5. clipping ratio。

参数指标：

1. parameter norm。
2. update norm。
3. update-to-weight ratio。
4. per-layer parameter max。

激活指标：

1. activation mean。
2. activation std。
3. activation max。
4. attention logits max。
5. norm output statistics。

混合精度指标：

1. overflow count。
2. loss scale。
3. skipped steps。

数据指标：

1. sequence length distribution。
2. token distribution。
3. special token ratio。
4. data shard id。
5. domain bucket。

分布式指标：

1. per-rank loss。
2. per-rank grad norm。
3. communication time。
4. straggler rank。

这些指标能让你在 loss 变成 NaN 之前发现风险。

### 十三、代码示例：检查 NaN 和梯度裁剪

下面是一个简化训练 step，展示 finite 检查和 gradient clipping 的位置。

```python
import torch


def train_step(model, batch, optimizer, scaler=None, max_grad_norm=1.0):
    optimizer.zero_grad(set_to_none=True)

    outputs = model(**batch)
    loss = outputs.loss

    if not torch.isfinite(loss):
        raise RuntimeError(f"Non-finite loss detected: {loss.item()}")

    loss.backward()

    grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_grad_norm)

    if not torch.isfinite(grad_norm):
        raise RuntimeError(f"Non-finite grad norm detected: {grad_norm.item()}")

    optimizer.step()
    return loss.item(), float(grad_norm)
```

如果使用 AMP，逻辑会更复杂，因为要先 unscale 再 clip：

```python
def amp_train_step(model, batch, optimizer, scaler, max_grad_norm=1.0):
    optimizer.zero_grad(set_to_none=True)

    with torch.cuda.amp.autocast(dtype=torch.float16):
        outputs = model(**batch)
        loss = outputs.loss

    if not torch.isfinite(loss):
        raise RuntimeError(f"Non-finite loss detected: {loss.item()}")

    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)

    grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_grad_norm)
    if not torch.isfinite(grad_norm):
        raise RuntimeError(f"Non-finite grad norm detected: {grad_norm.item()}")

    scaler.step(optimizer)
    scaler.update()
    return loss.item(), float(grad_norm)
```

注意：

1. AMP 下要先 `unscale_`，再做 gradient clipping。
2. 否则裁剪的是放大后的梯度，数值不对。
3. 大规模训练中还要配合分布式框架的 grad norm 统计。

### 十四、常见稳定性手段

常用手段可以分为几类。

优化侧：

1. 降低 peak learning rate。
2. 增加 warmup steps。
3. 使用 gradient clipping。
4. 调整 batch size。
5. 调整 AdamW 的 beta 和 epsilon。
6. 合理设置 weight decay。

数值侧：

1. 使用 BF16 替代 FP16。
2. 使用 dynamic loss scaling。
3. 关键操作用 FP32。
4. 使用稳定 softmax 和 fused kernel。
5. 检查 NaN/Inf。

模型侧：

1. 使用 Pre-LN 或 RMSNorm。
2. 使用合理初始化。
3. 使用 residual scaling。
4. 控制 attention logits scale。
5. MoE 中加入负载均衡和 router 稳定技巧。

数据侧：

1. 过滤异常样本。
2. 限制最大长度。
3. 检查 tokenizer 输出。
4. 检查 attention mask 和 label shift。
5. 记录 spike 对应的数据 batch。

系统侧：

1. 完整保存和恢复 checkpoint。
2. 监控每个 rank。
3. 检查通信和硬件错误。
4. 保证 dataloader 状态可恢复。
5. 做 deterministic replay 或近似 replay。

### 十五、面试回答模板

如果面试官问：

> 大模型训练中出现 loss spike 或 NaN，你怎么处理？

可以这样回答：

我会先区分现象。如果只是单 step loss spike，并且后面自动恢复，可能是困难 batch 或正常波动；如果 spike 持续、伴随 grad norm 暴涨、NaN/Inf 或 validation loss 变差，就说明训练不稳定，需要停止排查。

具体排查我会按层次来做。第一，记录出现问题的 step、LR、grad norm、rank、data shard、overflow 和 batch id。第二，从最近健康 checkpoint 恢复，尝试用相同 seed 和数据回放复现。第三，检查 loss、logits、activation、gradient、参数和 optimizer state 里最早出现 NaN/Inf 的位置。第四，检查数据 batch 是否有空样本、异常 token、mask 错误、label shift 或超长样本。第五，检查学习率、warmup、batch size、gradient clipping 和 AdamW 参数。第六，检查混合精度，比如 FP16 overflow、loss scale 是否异常，必要时切 BF16 或让关键操作保持 FP32。第七，检查分布式训练中是否某个 rank 先异常，是否 checkpoint 或 optimizer state 恢复错误。

常见修复包括降低 learning rate、增加 warmup、开启或调紧 gradient clipping、过滤异常数据、修复 mask 和 label、使用 BF16、恢复完整 optimizer/scheduler/scaler 状态，以及从健康 checkpoint 重新训练。核心原则是不盲目继续训练 NaN checkpoint，而是定位最早异常点。

### 十六、常见追问与回答

问题 1：loss spike 一定要停止训练吗？

回答：不一定。如果 spike 很短、没有 NaN/Inf、grad norm 正常，并且后续 loss 恢复，可以继续观察。但如果 spike 持续、扩大、伴随梯度异常或验证 loss 变差，就要停止排查。

问题 2：gradient clipping 能解决所有训练不稳定吗？

回答：不能。它能限制梯度范数，防止单次更新过大，但如果根因是数据 bug、mask 错误、loss 计算错误、FP16 overflow 或 checkpoint 恢复错误，clipping 只能缓解或掩盖问题。

问题 3：为什么 BF16 通常比 FP16 更稳定？

回答：BF16 的指数位更多，动态范围接近 FP32，不容易 overflow。FP16 精度更高一些但动态范围小，大模型训练中更容易溢出。所以支持 BF16 的硬件上，大模型训练常优先用 BF16。

问题 4：为什么 warmup 能提升稳定性？

回答：训练初期参数和 Adam 统计都不稳定，如果一开始就用 peak LR，更新可能过大。warmup 让学习率逐渐升高，给优化器状态和激活分布一个稳定过程。

问题 5：checkpoint 恢复后 loss 突然跳变，可能是什么原因？

回答：可能是 optimizer state、scheduler state、gradient scaler、random state、dataloader state 或 global step 没有正确恢复。也可能是分布式分片状态错位。不能只检查 model weights。

问题 6：如何定位 NaN 最早出现在哪一层？

回答：可以对关键模块注册 forward/backward hook，检查 activation 和 gradient 是否 finite；也可以分层记录 activation max、grad norm 和参数统计。目标是找到第一个从 finite 变成 NaN/Inf 的操作。

### 十七、常见误区

1. 误区：出现 loss spike 就说明训练失败。
   纠正：短暂 spike 可能是正常波动，要结合持续时间、grad norm、NaN/Inf 和验证 loss 判断。

2. 误区：NaN 后继续训练看看能不能恢复。
   纠正：NaN 通常会污染参数和 optimizer state，应该从健康 checkpoint 恢复。

3. 误区：训练不稳定只需要调小学习率。
   纠正：学习率很重要，但数据、mask、混合精度、优化器状态、架构和分布式系统也可能是根因。

4. 误区：gradient clipping 越小越好。
   纠正：过强 clipping 会压制正常梯度，影响收敛速度和最终效果。

5. 误区：只看 rank 0 日志就够了。
   纠正：分布式训练中可能是某个 rank 先异常，需要 per-rank 监控。

6. 误区：checkpoint 只保存模型参数就够。
   纠正：大模型训练恢复还需要 optimizer、scheduler、scaler、随机状态、数据进度和分布式元数据。

### 十八、小练习

1. 用自己的话解释 loss spike、NaN 和梯度爆炸的区别。
2. 为什么大模型训练初期需要 warmup？
3. 举三个可能导致 NaN 的数据问题。
4. 解释为什么 AMP 训练中要先 unscale 再做 gradient clipping。
5. 设计一个训练稳定性 dashboard，列出至少 10 个监控指标。
6. 用 2 分钟回答：“训练到一半 loss 突然变 NaN，你怎么排查？”

### 本讲总结

本讲最重要的结论：

1. 训练不稳定包括 loss spike、loss divergence、NaN、Inf、梯度爆炸和分布式异常等多种现象。
2. loss spike 不一定等于训练失败，要看是否持续、是否可恢复、是否伴随 grad norm 和 NaN/Inf 异常。
3. NaN 通常会传播并污染参数和 optimizer state，应该从健康 checkpoint 恢复。
4. 梯度爆炸常用 gradient clipping 缓解，但 clipping 不是万能修复。
5. 稳定性问题要从学习率、warmup、数据、混合精度、模型结构、优化器状态和分布式系统共同排查。
6. 大模型训练必须监控 loss 之外的指标，包括 grad norm、activation、参数范数、overflow、per-rank loss 和数据 shard。
7. 面试中成熟的回答不是单点技巧，而是系统化定位：先保存现场和复现，再定位最早异常，再按根因修复。

## 第 36 讲：混合精度训练：FP16、BF16 与 FP8

### 本讲目标

学完这一讲，你应该能够回答：

1. 为什么大模型训练需要混合精度。
2. FP32、FP16、BF16、FP8 的核心区别是什么。
3. 为什么 FP16 容易溢出，为什么 BF16 在大模型训练中更稳定。
4. AMP、loss scaling、master weights、FP32 optimizer state 分别解决什么问题。
5. FP8 训练为什么更难，需要哪些额外机制。
6. 面试中如何回答“混合精度训练如何保证稳定性”。

混合精度训练是大模型工程的基础能力。

如果只用 FP32 训练，显存占用高、吞吐低、训练成本大。

如果盲目把所有东西都改成 FP16，又容易出现 NaN、Inf 和训练发散。

因此真实训练通常是混合精度：

> 能低精度的计算低精度做，必须高精度的状态和关键操作保留高精度。

这句话就是本讲主线。

### 一、为什么需要混合精度训练

大模型训练的主要瓶颈包括：

1. 显存。
2. 计算吞吐。
3. 通信带宽。
4. 训练成本。

以一个 Transformer 模型为例，训练时显存不仅要存参数，还要存：

1. model weights。
2. gradients。
3. optimizer states。
4. activations。
5. temporary buffers。
6. communication buffers。

如果全部用 FP32，每个数占 4 bytes。

如果用 FP16 或 BF16，每个数占 2 bytes。

如果用 FP8，每个数占 1 byte。

这会直接影响能训练多大的模型、能用多大的 batch、能否放下更长 context。

同时，现代 GPU 对低精度矩阵乘有专门加速，例如 Tensor Core。

所以低精度不仅省显存，还能提高吞吐。

但问题是：

> 精度越低，数值范围和表示能力越有限，训练越容易不稳定。

混合精度的目标就是在效率和稳定性之间平衡。

### 二、浮点数的两个关键：范围和精度

理解 FP16、BF16、FP8，先要理解浮点数的两个关键属性。

第一是动态范围。

它决定能表示多大或多小的数。

如果数太大，可能 overflow，变成 Inf。

如果数太小，可能 underflow，变成 0。

第二是精度。

它决定能区分多细的数值差异。

如果精度不足，很多接近的数会被舍入成相同值。

浮点数通常由三部分组成：

1. sign：符号位。
2. exponent：指数位，主要决定范围。
3. mantissa：尾数位，主要决定精度。

可以粗略理解为：

```text
value = sign * mantissa * 2^exponent
```

不同格式的核心区别就是指数位和尾数位怎么分配。

### 三、FP32：稳定但昂贵

FP32 是传统深度学习训练最常用的格式。

它有较大的动态范围和较高精度。

优点：

1. 稳定。
2. 不容易 overflow。
3. 梯度和 optimizer state 精度较好。
4. debug 简单。

缺点：

1. 显存占用大。
2. 计算吞吐不如低精度。
3. 大模型训练成本高。

在今天的大模型训练中，纯 FP32 已经很少作为主训练格式。

但 FP32 仍然常用于关键状态和关键计算，例如：

1. optimizer state。
2. master weights。
3. loss accumulation。
4. norm 统计。
5. 某些 reduction。

也就是说，FP32 没有消失，而是从“全部训练格式”变成“关键稳定性格式”。

### 四、FP16：快但容易溢出

FP16 每个数占 16 bit，也就是 2 bytes。

它相比 FP32 节省一半显存，并且能利用 GPU 的低精度加速。

FP16 的问题是动态范围较小。

这会带来两个风险。

第一是 overflow。

例如某些中间值太大，FP16 表示不了，就会变成 Inf。

第二是 underflow。

某些梯度太小，FP16 表示不了，就可能变成 0。

在大模型中，容易出问题的位置包括：

1. attention logits。
2. softmax。
3. loss。
4. gradient。
5. LayerNorm/RMSNorm。
6. optimizer update。

如果直接用纯 FP16 训练，很容易出现 NaN 或收敛异常。

因此 FP16 通常需要 AMP 和 loss scaling。

### 五、Loss Scaling 解决什么问题

FP16 训练的一个核心问题是梯度 underflow。

很多梯度很小，用 FP32 可以表示，但用 FP16 可能变成 0。

梯度变成 0，模型就学不到细微更新。

Loss scaling 的思路很简单：

> 先把 loss 放大，再反向传播，让梯度也被放大，避免太小的梯度在 FP16 中下溢；更新前再把梯度缩回去。

流程是：

1. `scaled_loss = loss * scale`
2. 对 `scaled_loss` 做 backward。
3. 得到放大的 gradients。
4. optimizer step 前把 gradients 除以 `scale`。
5. 如果发现 overflow，就跳过 step 并降低 scale。

这就是 dynamic loss scaling 的核心。

如果 scale 太小，保护不够。

如果 scale 太大，可能导致 overflow。

所以现代框架通常自动调整 scale。

PyTorch AMP 中常见流程是：

```python
scaler.scale(loss).backward()
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
scaler.step(optimizer)
scaler.update()
```

注意顺序：

1. 先 scale loss。
2. backward 得到 scaled gradients。
3. `unscale_` 把梯度缩回真实尺度。
4. 再做 gradient clipping。
5. 最后 optimizer step。

如果先 clipping 再 unscale，裁剪的是错误尺度的梯度。

### 六、BF16：大模型训练的常用选择

BF16 也是 16 bit，但它和 FP16 的位分配不同。

BF16 的特点是：

1. 指数位更多。
2. 动态范围接近 FP32。
3. 尾数位更少。
4. 精度比 FP16 粗一些。

这意味着 BF16 更不容易 overflow。

对大模型训练来说，动态范围往往比尾数精度更关键。

因为训练中最致命的问题通常是 Inf、NaN、梯度爆炸，而不是某个数最后几位不够精确。

所以在支持 BF16 的硬件上，大模型训练常优先使用 BF16。

BF16 的优点：

1. 比 FP16 更稳定。
2. 通常不需要 loss scaling。
3. 显存占用和 FP16 一样低。
4. 吞吐也很好。

BF16 的缺点：

1. 尾数精度较低。
2. 旧硬件可能不支持或支持不好。
3. 某些小模型或精细数值任务中可能不如 FP32。

面试中可以这样总结：

> FP16 的问题主要是动态范围小，所以需要 loss scaling；BF16 保留了接近 FP32 的动态范围，因此更适合大模型训练，通常更稳定。

### 七、AMP：自动混合精度做了什么

AMP 是 Automatic Mixed Precision。

它不是简单地把所有计算改成 FP16 或 BF16。

AMP 的核心是：

> 根据操作类型自动选择低精度或高精度。

适合低精度的操作通常包括：

1. matrix multiplication。
2. convolution。
3. linear layer。
4. attention 中的大矩阵乘。

需要更谨慎的操作包括：

1. softmax。
2. normalization。
3. loss computation。
4. reduction。
5. optimizer update。

PyTorch 中常见写法：

```python
with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
    outputs = model(**batch)
    loss = outputs.loss

loss.backward()
optimizer.step()
```

如果用 FP16 AMP，则通常配合 GradScaler：

```python
scaler = torch.cuda.amp.GradScaler()

with torch.cuda.amp.autocast(dtype=torch.float16):
    outputs = model(**batch)
    loss = outputs.loss

scaler.scale(loss).backward()
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
scaler.step(optimizer)
scaler.update()
```

工程上要注意：

1. AMP 不代表所有 tensor 都是低精度。
2. 某些 op 会自动保持 FP32。
3. 自定义 CUDA kernel 或自定义 op 需要确认精度行为。
4. 不同框架和硬件的 autocast 策略可能不同。

### 八、Master Weights 和 Optimizer State

混合精度训练中，经常听到 master weights。

它的意思是：

> 前向和反向可以用低精度权重，但参数更新维护一份 FP32 主权重。

为什么需要？

因为低精度权重更新可能丢失小更新。

例如一个参数是 1.0，某次更新是 1e-7。

FP16 可能无法精确表达这个微小变化。

如果长期丢失小更新，会影响收敛。

所以典型做法是：

1. FP16/BF16 权重用于 forward/backward。
2. FP32 master weights 用于 optimizer update。
3. 更新后再把 FP32 权重 cast 回低精度用于计算。

AdamW 的 optimizer state 通常也保留 FP32，因为它包含：

1. first moment。
2. second moment。
3. 有时还有其他统计量。

这些状态对优化非常关键。

如果用低精度保存，可能导致收敛质量下降或不稳定。

这也是为什么即使用 FP16/BF16 训练，optimizer state 仍然消耗大量显存。

ZeRO、FSDP 等技术就是为了分片这些状态，降低单卡显存压力。

### 九、哪些计算应该保持高精度

混合精度的关键不是“越低越好”，而是知道哪些地方不能随便低精度。

通常建议保持更高精度的部分包括：

1. loss computation。
2. softmax 的数值稳定部分。
3. LayerNorm/RMSNorm 统计。
4. gradient norm 计算。
5. optimizer update。
6. Adam moments。
7. all-reduce accumulation。
8. logits 到 loss 的计算。

原因是这些操作往往涉及 reduction、指数、归一化或小差值。

它们对数值误差更敏感。

例如 softmax 如果直接对很大的 attention score 做 `exp`，可能溢出。

稳定 softmax 通常会先减去最大值：

```text
softmax(x_i) = exp(x_i - max(x)) / sum_j exp(x_j - max(x))
```

这样可以避免 `exp` 输入过大。

所以稳定性不仅取决于 dtype，也取决于具体 kernel 的实现。

### 十、FP8：更省但更难

FP8 每个数只占 8 bit。

它可以进一步降低显存和提升吞吐，尤其适合超大模型训练和推理。

但 FP8 的难度也明显更高。

因为 8 bit 能表达的信息非常有限。

常见 FP8 格式包括：

1. E4M3：指数 4 位，尾数 3 位。
2. E5M2：指数 5 位，尾数 2 位。

E4M3 精度稍好，但范围较小。

E5M2 范围更大，但精度更粗。

FP8 训练通常不能简单把所有 tensor 转成 FP8。

它需要额外机制：

1. per-tensor scaling。
2. per-channel scaling。
3. amax tracking。
4. delayed scaling。
5. FP16/BF16 accumulation。
6. 关键操作保留高精度。
7. 硬件和 kernel 支持。

可以把 FP8 理解为：

> 真正存储和矩阵乘时更低精度，但必须通过 scaling 把数值压到 FP8 能表示的合适范围。

如果 scaling 选得不好：

1. 太大，容易 overflow。
2. 太小，很多值 underflow 或量化成 0。
3. 动态变化太快，训练不稳定。

所以 FP8 通常比 BF16 更依赖框架、硬件和工程细节。

面试中不要把 FP8 简单说成“比 FP16 更省显存”。更准确的是：

> FP8 有更高效率潜力，但需要精细的 scaling、校准和高精度 accumulation，否则训练稳定性和精度都会受影响。

### 十一、FP16、BF16、FP8 怎么选

实际选择取决于硬件、模型规模和稳定性要求。

如果硬件支持 BF16，大模型预训练通常优先 BF16。

原因是：

1. 稳定性好。
2. 不太需要 loss scaling。
3. 效率和显存接近 FP16。
4. 工程复杂度适中。

如果硬件只支持 FP16，通常使用 FP16 AMP + dynamic loss scaling。

同时要更关注：

1. overflow。
2. loss scale。
3. gradient clipping。
4. FP32 master weights。
5. 稳定 kernel。

如果追求极致吞吐，且硬件和框架成熟，可以考虑 FP8。

但 FP8 更适合有成熟基础设施的团队，而不是随便打开一个开关。

一个粗略选择表：

```text
FP32：最稳定，最贵，常用于关键状态或 debug
FP16：高效，但动态范围小，需要 loss scaling
BF16：高效且稳定，大模型训练常用选择
FP8：效率潜力更高，但需要 scaling 和成熟 kernel
```

### 十二、混合精度训练的常见故障

故障一：FP16 overflow。

表现：loss spike、Inf、NaN、GradScaler 频繁 skip step。

处理：降低 LR、使用 BF16、调整 loss scale、检查 attention logits、启用 clipping。

故障二：gradient underflow。

表现：训练变慢、loss 降不动、部分层梯度接近 0。

处理：使用 loss scaling、检查梯度统计、保留关键梯度高精度。

故障三：norm 层数值异常。

表现：activation std 异常、某些层输出变大或变小。

处理：norm 统计用 FP32、检查 epsilon、检查输入分布。

故障四：softmax 溢出。

表现：attention 中出现 NaN 或 Inf。

处理：使用稳定 softmax、检查 mask 是否正确、检查 attention score scale。

故障五：optimizer state 精度不足。

表现：收敛质量差或训练不稳定。

处理：optimizer state 保持 FP32 或使用成熟低精度 optimizer。

故障六：自定义 kernel 精度行为不清楚。

表现：换 kernel 后 loss 曲线变化或出现 NaN。

处理：对比 reference implementation，检查 accumulation dtype 和边界条件。

### 十三、代码示例：BF16 与 FP16 AMP

BF16 训练通常更简单：

```python
import torch


def bf16_train_step(model, batch, optimizer):
    optimizer.zero_grad(set_to_none=True)

    with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
        outputs = model(**batch)
        loss = outputs.loss

    if not torch.isfinite(loss):
        raise RuntimeError(f"Non-finite loss: {loss.item()}")

    loss.backward()
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    optimizer.step()
    return loss.item()
```

FP16 训练通常需要 GradScaler：

```python
import torch


def fp16_train_step(model, batch, optimizer, scaler):
    optimizer.zero_grad(set_to_none=True)

    with torch.cuda.amp.autocast(dtype=torch.float16):
        outputs = model(**batch)
        loss = outputs.loss

    if not torch.isfinite(loss):
        raise RuntimeError(f"Non-finite loss: {loss.item()}")

    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    scaler.step(optimizer)
    scaler.update()
    return loss.item()
```

这两个例子只展示核心顺序。

真实大模型训练还要考虑：

1. gradient accumulation。
2. distributed data parallel。
3. ZeRO/FSDP。
4. activation checkpointing。
5. optimizer state sharding。
6. logging 和 anomaly detection。

### 十四、如何验证混合精度没有伤害训练

从 FP32 切到 BF16、FP16 或 FP8 后，不能只看能不能跑。

需要做对照实验。

常见验证方法：

1. 小规模训练对比 loss 曲线。
2. 比较 validation loss。
3. 比较 grad norm 分布。
4. 比较 activation statistics。
5. 比较下游 benchmark。
6. 检查 NaN/Inf 和 overflow 次数。
7. 检查吞吐和显存收益。

如果 BF16 loss 曲线和 FP32 接近，且下游评估一致，同时吞吐更高、显存更低，说明迁移成功。

如果 FP16 出现频繁 skipped step 或 loss spike，就要检查 loss scale、LR、softmax、norm 和 clipping。

如果 FP8 下 loss 明显变差，要检查 scaling 策略和 accumulation 精度。

一个成熟团队不会直接在最大规模上第一次尝试新精度格式。

更合理流程是：

1. 单卡小模型验证。
2. 多卡小模型验证。
3. 中等规模训练验证。
4. 目标规模短跑。
5. 再进行完整训练。

### 十五、面试回答模板

如果面试官问：

> FP16、BF16、FP8 有什么区别？混合精度训练怎么保证稳定？

可以这样回答：

FP32 最稳定但显存和计算成本高。FP16 每个数 16 bit，显存更省、吞吐更高，但动态范围小，容易 overflow 和 underflow，所以通常需要 AMP、dynamic loss scaling、FP32 master weights 和 FP32 optimizer state。BF16 也是 16 bit，但指数位更多，动态范围接近 FP32，因此比 FP16 更不容易溢出，通常不需要 loss scaling，是大模型训练中更常用的选择。FP8 每个数 8 bit，效率潜力更高，但数值表示更有限，需要 per-tensor 或 per-channel scaling、amax tracking、高精度 accumulation 和成熟 kernel 支持。

混合精度训练的核心不是把所有计算都变成低精度，而是让矩阵乘等计算密集操作用低精度加速，同时把 loss、softmax、norm、reduction、optimizer state、master weights 等关键部分保持高精度。工程上还要监控 loss、grad norm、overflow、loss scale、activation statistics 和 NaN/Inf。如果出现不稳定，需要检查学习率、loss scaling、gradient clipping、attention logits、norm 层和自定义 kernel 的精度行为。

### 十六、常见追问与回答

问题 1：为什么 BF16 比 FP16 更稳定？

回答：BF16 有更多指数位，动态范围接近 FP32，不容易 overflow。FP16 动态范围小，大模型中 attention logits、梯度或激活容易超出范围，导致 Inf 或 NaN。

问题 2：FP16 为什么需要 loss scaling？

回答：FP16 可能表示不了很小的梯度，导致 underflow。loss scaling 先放大 loss 和梯度，避免小梯度变成 0，optimizer step 前再 unscale 回真实尺度。

问题 3：为什么 AMP 下要先 unscale 再 gradient clipping？

回答：因为 backward 得到的是 scaled gradients。如果不先 unscale，clip 的是放大后的梯度范数，裁剪阈值就没有真实意义。

问题 4：BF16 是否完全不需要高精度状态？

回答：不是。BF16 更稳定，但 optimizer state、某些 reduction、norm 统计、loss 等关键部分仍然常用 FP32，以保证收敛质量和数值稳定。

问题 5：FP8 训练最大的难点是什么？

回答：FP8 表示范围和精度都很有限，必须通过 scaling 把数值映射到合适范围，并使用高精度 accumulation。scaling 不好会 overflow、underflow 或量化误差过大。

问题 6：混合精度会不会影响最终模型效果？

回答：可能会，所以需要小规模对照实验。成熟的 BF16 训练通常能接近 FP32 效果，但 FP16 或 FP8 如果 scaling、kernel 或高精度状态处理不好，会影响收敛和最终能力。

### 十七、常见误区

1. 误区：混合精度就是把所有 tensor 都改成 FP16。
   纠正：混合精度要区分计算、参数、梯度、optimizer state 和关键操作，不是全低精度。

2. 误区：FP16 和 BF16 都是 16 bit，所以差不多。
   纠正：两者指数位和尾数位不同，BF16 动态范围更大，通常更稳定。

3. 误区：用了 AMP 就不会有 NaN。
   纠正：AMP 只能自动选择部分 op 的精度，仍然可能因为 LR、数据、softmax、loss scale 或 kernel 问题出现 NaN。

4. 误区：BF16 不需要任何 FP32 计算。
   纠正：BF16 训练中很多关键状态仍应保留 FP32。

5. 误区：FP8 只是更小的 FP16。
   纠正：FP8 需要 scaling、校准和高精度 accumulation，工程复杂度更高。

6. 误区：低精度只影响速度，不影响模型质量。
   纠正：低精度可能影响稳定性和收敛质量，必须通过对照实验验证。

2. 为什么 FP16 容易 overflow 和 underflow？
3. 解释 loss scaling 的完整流程。
4. 为什么 BF16 大模型训练中通常比 FP16 更受欢迎？
5. 列出 5 个混合精度训练中应该保持高精度的操作或状态。

### 本讲总结

本讲最重要的结论：

1. 混合精度训练的目标是在效率和稳定性之间平衡。
2. FP32 稳定但昂贵，常用于关键状态和关键计算。
3. FP16 高效但动态范围小，容易 overflow/underflow，通常需要 dynamic loss scaling。
4. BF16 动态范围接近 FP32，显存和吞吐接近 FP16，是大模型训练常用选择。
5. FP8 效率潜力更高，但需要 scaling、amax tracking、高精度 accumulation 和成熟 kernel。
6. AMP 不是全低精度，而是按操作选择合适 dtype。
7. 混合精度稳定性要靠高精度关键状态、loss scaling、gradient clipping、稳定 kernel 和系统化监控共同保证。

## 第 37 讲：Checkpoint 策略与训练恢复

### 本讲目标

学完这一讲，你应该能够回答：

1. 大模型训练为什么必须设计 checkpoint 策略。
2. 一个可恢复的 checkpoint 里应该保存哪些内容。
3. 只保存模型权重和保存完整训练状态有什么区别。
4. checkpoint 保存频率、保留策略、异步保存和分片保存如何设计。
5. 从 checkpoint 恢复后 loss 跳变，应该如何排查。
6. 面试中如何回答“大规模训练中断后如何恢复”。

Checkpoint 是大模型训练工程中非常关键但容易被低估的一环。

很多初学者以为 checkpoint 就是保存 `model.state_dict()`。

这在小实验里可能够用，但在大模型预训练里远远不够。

真实大模型训练一次可能持续几周甚至几个月，期间可能遇到：

1. GPU 故障。
2. 节点宕机。
3. 网络中断。
4. 存储故障。
5. 作业被抢占。
6. loss spike 后需要回滚。
7. 需要选择某个中间 checkpoint 做评估或继续训练。

如果 checkpoint 策略做得不好，轻则浪费几个小时算力，重则损失数百万美元训练成本。

### 一、Checkpoint 的核心目的

Checkpoint 主要有四个目的。

第一，故障恢复。

训练中断后，从最近 checkpoint 恢复，避免从头训练。

第二，实验回溯。

当发现某个时间点后训练变坏，可以回到之前状态分析和重训。

第三，模型选择。

训练过程中不同 checkpoint 可能在不同任务上表现不同，需要保存若干候选用于评估。

第四，继续训练和后训练。

预训练 checkpoint 可以作为 continued pretraining、SFT、RLHF、DPO 或领域适配的起点。

因此 checkpoint 不是简单备份文件，而是训练系统的时间机器。

它必须回答两个问题：

1. 我能不能恢复到某个训练时刻？
2. 恢复后能不能像没中断一样继续训练？

第二个问题比第一个更难。

### 二、模型权重 checkpoint 与完整训练 checkpoint

Checkpoint 可以分两类。

第一类是模型权重 checkpoint。

它主要保存：

1. model weights。
2. model config。
3. tokenizer config。
4. generation config。

这种 checkpoint 适合：

1. 推理部署。
2. 下游微调。
3. 公开发布。
4. 评估模型能力。

第二类是完整训练 checkpoint。

它除了保存模型权重，还要保存完整训练状态，包括：

1. optimizer state。
2. learning rate scheduler state。
3. gradient scaler state。
4. global step。
5. consumed tokens。
6. dataloader state。
7. random number generator state。
8. distributed state。
9. mixed precision metadata。
10. training config。

完整训练 checkpoint 的目标是恢复训练。

面试中一定要强调：

> 用于推理的 checkpoint 和用于恢复训练的 checkpoint 不是一回事。

只保存模型权重，通常不能做到严格恢复训练。

### 三、完整 checkpoint 应该保存什么

一个严肃的大模型训练 checkpoint 至少应包含以下内容。

第一，模型状态。

包括：

1. 所有参数。
2. buffer。
3. tied weights 的关系。
4. model config。
5. tokenizer 版本。

第二，优化器状态。

以 AdamW 为例，需要保存：

1. first moment，也就是 `m`。
2. second moment，也就是 `v`。
3. step count。
4. 参数分组信息。
5. weight decay、betas、epsilon 等配置。

如果不恢复 optimizer state，即使模型权重一样，后续更新轨迹也会不同。

第三，学习率调度器状态。

包括：

1. 当前 step。
2. warmup 进度。
3. decay 进度。
4. 当前 learning rate。

如果 scheduler step 错了，恢复后可能使用错误 LR，导致 loss 跳变或训练不稳定。

第四，混合精度状态。

如果使用 FP16 AMP，需要保存 GradScaler 状态，例如：

1. loss scale。
2. growth tracker。
3. overflow 相关状态。

如果使用 FP8，可能还要保存 scaling factors、amax history 等元数据。

第五，随机状态。

包括：

1. Python random state。
2. NumPy random state。
3. PyTorch CPU RNG state。
4. PyTorch CUDA RNG state。
5. 每个 GPU rank 的 RNG state。

随机状态影响 dropout、数据 shuffle、采样、mask 等。

第六，数据状态。

包括：

1. 当前 epoch。
2. 当前 shard。
3. 当前 sample offset。
4. consumed tokens。
5. dataloader shuffle state。
6. 数据版本和 manifest。

大模型预训练通常按 token 数推进，恢复时必须知道已经消费了哪些 token，避免重复或跳过。

第七，分布式训练状态。

如果使用 ZeRO、FSDP、pipeline parallel、tensor parallel，需要保存：

1. 参数分片信息。
2. optimizer state 分片。
3. rank mapping。
4. parallel group 配置。
5. checkpoint shard metadata。

这些信息错了，可能无法加载，或者加载后 silent corruption。

第八，实验元数据。

包括：

1. git commit。
2. training config。
3. dependency version。
4. hardware topology。
5. dataset version。
6. command line args。
7. checkpoint schema version。

这些信息决定 checkpoint 是否可解释、可复现、可迁移。

### 四、为什么只保存模型权重不够

假设你训练到 step 100000，中断了。

你只保存了模型权重，没有保存 optimizer state。

恢复后会发生什么？

模型参数看起来一样，但 AdamW 的 `m` 和 `v` 不见了。

AdamW 会重新从空状态开始估计梯度一阶矩和二阶矩。

这相当于训练轨迹突然改变。

可能后果包括：

1. loss 短期跳变。
2. 收敛速度变化。
3. 训练稳定性下降。
4. 最终模型略有不同。

如果 scheduler state 也没保存，问题更严重。

例如恢复后 learning rate 又从 warmup 起点开始，或者错误地跳到更高 LR，可能直接造成 loss spike。

如果 dataloader state 没保存，模型可能重复消费一段数据，或跳过一段数据。

如果 GradScaler 没保存，FP16 训练可能重新寻找 loss scale，出现额外 skipped steps 或 overflow。

所以严格恢复训练需要完整状态，而不是单独权重。

### 五、保存频率如何设计

checkpoint 保存频率是成本和风险的权衡。

保存太频繁：

1. 存储压力大。
2. I/O 开销高。
3. 可能拖慢训练。
4. checkpoint 管理复杂。

保存太少：

1. 故障后回滚太远。
2. 浪费大量算力。
3. 难以定位训练异常。
4. 可选模型点太少。

常见策略包括：

1. 按 step 保存，例如每 1000 step。
2. 按 token 保存，例如每 50B token。
3. 按时间保存，例如每 2 小时。
4. 按评估节点保存，例如每次 validation 后。
5. 异常前后保存，例如检测到 spike 时保存现场。

大模型训练中，更常用的是按 token 或时间设计，因为不同 batch size 和 sequence length 下 step 不一定代表相同训练量。

一个实用策略是：

1. 最近 checkpoint 保存得密一些，便于故障恢复。
2. 远期 checkpoint 稀疏保留，便于回溯和模型选择。
3. 重要评估点永久保留。
4. 最佳 checkpoint 单独标记。

例如：

```text
最近 24 小时：每 1 小时保留一个
最近 7 天：每 6 小时保留一个
历史长期：每天保留一个
关键评估点：永久保留
```

这类似日志系统中的 retention policy。

### 六、保存内容如何分层

为了降低成本，可以把 checkpoint 分层。

第一层：轻量模型 checkpoint。

只保存模型权重，用于评估和推理。

第二层：完整训练 checkpoint。

保存模型、optimizer、scheduler、scaler、数据状态，用于恢复训练。

第三层：debug checkpoint。

在异常附近保存更多信息，例如 batch id、grad norm、activation statistics、数据样本引用。

第四层：release checkpoint。

清理训练专用状态，只保留推理所需内容，并转换成标准格式。

这种分层可以避免所有 checkpoint 都巨大无比。

例如完整训练 checkpoint 可能非常大，因为 AdamW state 通常是参数量的数倍。

如果每个评估点都保存完整训练状态，存储成本会非常高。

所以工程上常常是：

1. 高频保存完整 checkpoint，但只保留最近若干个。
2. 中频保存模型权重 checkpoint。
3. 关键节点长期保留完整 checkpoint。
4. 训练结束后导出部署 checkpoint。

### 七、分布式 checkpoint：为什么更复杂

单卡训练保存一个文件就够了。

大模型训练通常不同。

如果使用 FSDP 或 ZeRO，模型参数、梯度和 optimizer state 会分片存储在不同 rank 上。

保存 checkpoint 时有两种思路。

第一种是 gather 后保存完整权重。

优点：

1. 文件容易使用。
2. 适合推理和发布。
3. 加载逻辑简单。

缺点：

1. 需要额外内存。
2. 单点 I/O 压力大。
3. 大模型可能无法 gather 到单机。

第二种是 sharded checkpoint。

每个 rank 保存自己的分片。

优点：

1. 不需要集中 gather。
2. 适合超大模型。
3. I/O 可以并行。

缺点：

1. 元数据复杂。
2. rank 数变化后恢复更难。
3. 推理部署前通常要转换。

成熟系统通常会支持：

1. sharded training checkpoint。
2. consolidated model checkpoint。
3. checkpoint format conversion。
4. resharding。

Resharding 是指训练时用一种并行配置保存，恢复或推理时转换到另一种并行配置。

例如从 128 GPU 训练 checkpoint 恢复到 64 GPU，或者从 ZeRO checkpoint 转成 Hugging Face 格式。

这需要可靠的 metadata。

### 八、异步保存与 I/O 瓶颈

大模型 checkpoint 可能有数百 GB 甚至 TB 级。

如果同步保存，训练可能停很久。

这会降低 GPU 利用率。

因此常见做法是异步 checkpoint。

核心思路是：

1. 训练进程生成一致的 checkpoint snapshot。
2. 后台线程或独立进程负责写入存储。
3. 训练尽快继续。
4. 写入完成后再标记 checkpoint 为可用。

异步保存要注意一致性。

不能出现文件写了一半就被认为是完整 checkpoint。

常见做法包括：

1. 写入临时目录。
2. 完成后原子 rename。
3. 写 manifest 文件。
4. 校验 checksum。
5. 记录 checkpoint status。

例如：

```text
checkpoint_100000.tmp/   正在写入
checkpoint_100000/       写入完成后原子切换
manifest.json            记录 shard 列表、大小、checksum、schema
```

如果 checkpoint 写入失败，恢复系统应该忽略这个不完整 checkpoint，自动选择上一个健康 checkpoint。

### 九、训练恢复的一致性要求

恢复训练时，目标是让训练尽量像没中断一样继续。

需要检查：

1. 模型权重是否一致。
2. optimizer state 是否一致。
3. scheduler state 是否一致。
4. scaler state 是否一致。
5. RNG state 是否一致。
6. dataloader state 是否一致。
7. consumed tokens 是否一致。
8. distributed rank mapping 是否一致。
9. config 是否一致。
10. 代码版本是否一致。

如果追求 bitwise reproducibility，即逐 bit 一致，要求非常高。

需要完全相同硬件、kernel、随机数、数据顺序和并行策略。

很多大模型训练只追求 statistical reproducibility，也就是恢复后整体曲线和最终效果接近，不要求每个 bit 相同。

但无论如何，不能出现明显 loss 跳变或重复数据严重偏差。

### 十、恢复后 loss 跳变如何排查

这是高频面试题。

如果从 checkpoint 恢复后 loss 突然变化，可以按以下顺序排查。

第一，确认 checkpoint 是否完整。

检查 manifest、shard 数量、checksum、保存是否完成。

第二，确认 model weights 是否正确加载。

检查 missing keys、unexpected keys、shape mismatch、dtype mismatch。

第三，确认 optimizer state 是否恢复。

如果 Adam moments 丢失，loss 曲线可能变化。

第四，确认 scheduler state。

检查当前 learning rate 是否和中断前一致。

第五，确认 mixed precision state。

FP16 要检查 GradScaler，FP8 要检查 scaling metadata。

第六，确认 dataloader state。

是否重复了一段数据？是否跳过了数据？当前 shard 是否正确？

第七，确认 RNG state。

dropout、数据 shuffle、mask 是否变了。

第八，确认训练代码和 config。

是否改了 batch size、sequence length、loss mask、weight decay、gradient accumulation？

第九，确认分布式配置。

rank mapping、tensor parallel、pipeline parallel、ZeRO stage 是否一致。

第十，做短跑对比。

从 checkpoint 前后的日志中选取相同 batch，比较 forward loss、grad norm 和参数更新。

面试中可以总结：

> 恢复后 loss 跳变通常不是单纯权重问题，而要检查 optimizer、scheduler、数据进度、随机状态、混合精度和分布式元数据。

### 十一、Checkpoint 与训练稳定性

checkpoint 不只是故障恢复工具，也和训练稳定性相关。

当训练出现 loss spike 或 NaN 时，checkpoint 决定你能回滚到哪里。

一个好的策略应该支持：

1. 自动检测 NaN 后停止训练。
2. 回滚到最近健康 checkpoint。
3. 降低 LR 或过滤异常 batch 后重试。
4. 保留异常现场用于 debug。
5. 避免覆盖最后一个健康 checkpoint。

如果 checkpoint 保存策略不合理，可能发生：

1. 最新 checkpoint 已经被 NaN 污染。
2. 健康 checkpoint 被清理掉。
3. 只能回滚很远，浪费大量训练。
4. 无法定位引发 spike 的数据。

因此生产训练中常会标记 checkpoint 状态：

```text
healthy
suspect
failed
best_eval
release_candidate
```

只有通过基本验证的 checkpoint 才进入健康候选。

### 十二、Checkpoint 选择与评估

训练过程中会产生很多 checkpoint。

最终选择哪个，不一定只看最后一个。

选择依据包括：

1. validation loss。
2. domain validation loss。
3. benchmark。
4. safety eval。
5. human preference。
6. downstream fine-tuning 表现。
7. 推理成本和稳定性。

对于 base model，通常更重视 validation loss 和基础能力。

对于 chat model，更重视指令遵循、人类偏好和安全性。

对于 code model，更重视可执行代码 benchmark。

对于 reasoning model，更重视数学、代码和 verifier 评估。

因此 checkpoint 策略要和评估系统联动。

例如每保存 N 个 checkpoint，就触发一轮轻量评估。

对候选 checkpoint，再触发完整评估。

最后把评估结果写回 checkpoint metadata。

### 十三、代码示例：保存和恢复训练状态

下面是一个简化 PyTorch checkpoint 示例。

```python
import random
import numpy as np
import torch


def save_checkpoint(path, model, optimizer, scheduler, step, dataloader_state=None, scaler=None):
    state = {
        "model": model.state_dict(),
        "optimizer": optimizer.state_dict(),
        "scheduler": scheduler.state_dict() if scheduler is not None else None,
        "step": step,
        "rng_state": {
            "python": random.getstate(),
            "numpy": np.random.get_state(),
            "torch_cpu": torch.get_rng_state(),
            "torch_cuda": torch.cuda.get_rng_state_all() if torch.cuda.is_available() else None,
        },
        "dataloader_state": dataloader_state,
        "scaler": scaler.state_dict() if scaler is not None else None,
    }
    torch.save(state, path)


def load_checkpoint(path, model, optimizer, scheduler=None, scaler=None):
    state = torch.load(path, map_location="cpu")
    model.load_state_dict(state["model"])
    optimizer.load_state_dict(state["optimizer"])

    if scheduler is not None and state["scheduler"] is not None:
        scheduler.load_state_dict(state["scheduler"])

    if scaler is not None and state["scaler"] is not None:
        scaler.load_state_dict(state["scaler"])

    random.setstate(state["rng_state"]["python"])
    np.random.set_state(state["rng_state"]["numpy"])
    torch.set_rng_state(state["rng_state"]["torch_cpu"])
    if torch.cuda.is_available() and state["rng_state"]["torch_cuda"] is not None:
        torch.cuda.set_rng_state_all(state["rng_state"]["torch_cuda"])

    return state["step"], state.get("dataloader_state")
```

这个示例没有覆盖大模型分片 checkpoint，但体现了关键思想：

1. 不只保存 model。
2. optimizer、scheduler、scaler、RNG、数据状态都要保存。
3. 恢复时要按相同语义恢复。

真实生产训练还需要：

1. sharded state dict。
2. async save。
3. checksum。
4. manifest。
5. versioning。
6. distributed barrier。
7. storage retry。

### 十四、Checkpoint 的存储与命名规范

好的命名和元数据能减少很多事故。

建议 checkpoint 名称包含：

1. run id。
2. global step。
3. consumed tokens。
4. date time。
5. checkpoint type。

例如：

```text
run_042_step_120000_tokens_950B_full/
run_042_step_120000_tokens_950B_model_only/
```

metadata 中建议记录：

1. model size。
2. dataset version。
3. tokenizer version。
4. code commit。
5. config hash。
6. validation loss。
7. benchmark summary。
8. checkpoint status。
9. parent checkpoint。

parent checkpoint 很有用。

它能构成训练谱系，记录这个 checkpoint 是从哪个 checkpoint 继续训练来的。

这对于 continued pretraining、SFT 和多阶段训练很关键。

### 十五、面试回答模板

如果面试官问：

> 大模型训练中 checkpoint 应该怎么设计？训练中断后如何恢复？

可以这样回答：

我会先区分 model checkpoint 和 training checkpoint。model checkpoint 主要用于评估、推理和下游微调，只需要模型权重、config、tokenizer 等。training checkpoint 用于恢复训练，必须保存完整状态，包括模型参数、optimizer state、scheduler state、mixed precision scaler、global step、consumed tokens、dataloader state、RNG state、分布式分片元数据和训练 config。

保存策略上，我会根据训练成本和故障风险设计频率。最近 checkpoint 保留密一些，用于快速故障恢复；历史 checkpoint 稀疏保留，用于回溯；关键评估点和最佳 checkpoint 长期保留。大模型通常使用 sharded checkpoint 和异步保存，以避免单点内存和 I/O 瓶颈。写入时用临时目录、manifest、checksum 和原子切换，避免半写入 checkpoint 被误用。

恢复时，我会校验 checkpoint 完整性，然后恢复模型、optimizer、scheduler、scaler、RNG、数据进度和分布式状态。恢复后检查 learning rate、global step、consumed tokens、loss 和 grad norm 是否与中断前一致。如果恢复后 loss 跳变，我会重点排查 optimizer state、scheduler state、dataloader state、mixed precision state、config 变化和 rank mapping，而不是只看模型权重。

### 十六、常见追问与回答

问题 1：只保存模型权重可以恢复训练吗？

回答：可以继续训练，但不是严格恢复。因为 optimizer state、scheduler、scaler、RNG 和数据进度丢失后，训练轨迹会变化，可能出现 loss 跳变或收敛差异。

问题 2：为什么 AdamW 的 optimizer state 很重要？

回答：AdamW 依赖一阶矩和二阶矩估计来调整更新幅度。如果恢复时这些状态丢失，相当于优化器重新开始，参数虽然一样，但后续更新不同。

问题 3：checkpoint 保存越频繁越好吗？

回答：不是。保存太频繁会带来巨大存储和 I/O 开销，甚至影响 GPU 利用率。要在故障回滚成本和保存成本之间权衡。

问题 4：什么是 sharded checkpoint？

回答：在 ZeRO/FSDP 等分布式训练中，模型和 optimizer state 被分片在多个 rank 上。sharded checkpoint 让每个 rank 保存自己的分片，适合超大模型，但需要元数据支持恢复和转换。

问题 5：恢复后 loss 变高，最可能是什么原因？

回答：常见原因包括 optimizer state 没恢复、scheduler step 错误、learning rate 不一致、dataloader 位置错、GradScaler 状态丢失、config 改变或分布式分片加载错误。

问题 6：如何避免保存到一半的 checkpoint 被加载？

回答：可以写入临时目录，完成所有 shard 和 manifest 后再原子 rename；同时记录 checksum 和 status，恢复时只加载标记为 complete 且校验通过的 checkpoint。

### 十七、常见误区

1. 误区：checkpoint 就是 `model.state_dict()`。
   纠正：恢复训练需要 optimizer、scheduler、scaler、RNG、数据状态和分布式元数据。

2. 误区：只要模型权重一样，恢复后训练就一样。
   纠正：优化器状态、学习率、随机数和数据顺序都会影响后续轨迹。

3. 误区：永远保存最后一个 checkpoint 就够。
   纠正：最后一个 checkpoint 可能已被 NaN 污染，需要保留多个健康 checkpoint。

4. 误区：checkpoint 越多越安全。
   纠正：过多 checkpoint 会带来存储、I/O 和管理成本，需要 retention policy。

5. 误区：分布式 checkpoint 可以像单机文件一样处理。
   纠正：分片状态、rank mapping、并行策略和元数据都必须一致。

6. 误区：恢复后 loss 跳变一定是模型权重加载错。
   纠正：更常见的是 optimizer、scheduler、数据状态、scaler 或 config 问题。

### 十八、小练习

1. 列出完整 training checkpoint 至少应该保存的 8 类状态。
2. 解释 model checkpoint 和 training checkpoint 的区别。
3. 为什么恢复 AdamW 训练时必须保存一阶矩和二阶矩？
4. 设计一个 30 天大模型训练的 checkpoint retention policy。
5. 解释 sharded checkpoint 和 consolidated checkpoint 的区别。
6. 用 2 分钟回答：“恢复 checkpoint 后 loss 跳变，你会怎么排查？”

### 本讲总结

本讲最重要的结论：

1. Checkpoint 是大模型训练的时间机器，用于故障恢复、实验回溯、模型选择和继续训练。
2. model checkpoint 和 training checkpoint 不同，前者用于推理评估，后者用于恢复训练。
3. 完整训练 checkpoint 必须保存模型、optimizer、scheduler、mixed precision、RNG、数据进度、分布式元数据和训练配置。
4. 只保存模型权重无法严格恢复训练，可能导致 loss 跳变和训练轨迹变化。
5. checkpoint 保存频率要平衡故障风险、I/O 开销和存储成本。
6. 大模型常用 sharded checkpoint、异步保存、manifest、checksum 和原子切换来保证可靠性。
7. 恢复后 loss 跳变时，要系统排查 optimizer、scheduler、scaler、数据状态、RNG、config 和分布式状态。

## 第 38 讲：评估集设计与 early stopping

### 本讲目标

学完这一讲，你应该能够回答：

1. 大模型预训练为什么不能只看一个 overall validation loss。
2. 如何设计 validation set、benchmark set、held-out set 和 canary set。
3. 为什么评估集污染会让模型能力判断失真。
4. early stopping 在小模型训练和大模型预训练中的差异。
5. 如何用评估结果选择 checkpoint，而不是机械选最后一个或 loss 最低的。
6. 面试中如何回答“怎么判断模型该不该继续训练”。

这一讲承接前面几讲。

第 34 讲讲了训练 loss、验证 loss 与下游能力的关系。

第 37 讲讲了 checkpoint 策略。

本讲要把两者连接起来：

> 评估集决定你如何理解训练进展，early stopping 和 checkpoint selection 决定你何时停止、选哪个模型继续投入。

很多训练事故不是模型没训好，而是评估设计错了。

例如：

1. 验证集被训练集污染，loss 虚低。
2. 只看英文网页 loss，忽略代码和数学退化。
3. benchmark 太少，模型选择不稳定。
4. 频繁根据测试集调参，导致评估过拟合。
5. 机械 early stopping，错过后期能力提升。

所以评估集设计是训练工程的核心部分。

### 一、先区分几类评估集

大模型训练中常见的评估集合至少有四类。

第一类是 training validation set。

它用于训练过程中定期计算 validation loss。

特点是：

1. 不参与训练。
2. 分布通常接近训练数据。
3. 规模较大。
4. 用于监控泛化和训练稳定性。

第二类是 domain validation set。

它把验证集按领域拆分，例如：

1. web。
2. books。
3. wiki。
4. code。
5. math。
6. academic。
7. multilingual。
8. dialogue。
9. long context。

它用于发现局部能力变化。

第三类是 downstream benchmark set。

它用于评估具体任务能力，例如 MMLU、GSM8K、HumanEval、MBPP、长上下文检索、安全评估等。

第四类是 final test set 或 hidden eval set。

它应该尽量少用，用于最终报告和模型选择后的确认。

如果频繁用 final test set 调参，它就不再是真正的 final test。

这四类集合的关系可以这样理解：

```text
validation loss：训练健康和泛化趋势
domain validation：具体数据领域是否变好
downstream benchmark：具体能力是否变好
hidden test：最终可信确认
```

不要把它们混成一个指标。

### 二、为什么不能只看 overall validation loss

overall validation loss 是一个平均值。

平均值的优点是稳定，缺点是掩盖结构。

例如一个模型继续训练后：

```text
overall loss: 1.86 -> 1.83
```

看起来变好了。

但分桶后可能是：

```text
web loss: 1.90 -> 1.82
code loss: 1.45 -> 1.55
math loss: 1.70 -> 1.78
multilingual loss: 2.20 -> 2.35
```

这说明模型主要在 web 文本上变好，但代码、数学、多语言退化。

如果目标是通用网页生成，这可能还可以接受。

如果目标是 OpenAI 大模型算法岗关注的通用能力、代码能力和 reasoning，这就是危险信号。

overall loss 还有一个问题：它受验证集组成比例影响很大。

如果验证集中 web 占 80%，code 只占 5%，那么 code loss 的变化对 overall loss 影响很小。

所以工程上要同时看：

1. overall validation loss。
2. per-domain validation loss。
3. per-language validation loss。
4. per-length validation loss。
5. per-difficulty benchmark。
6. safety 和 alignment 指标。

面试中可以总结为：

> overall loss 是宏观温度计，domain eval 是局部体检。只看体温，可能看不出某个器官出了问题。

### 三、如何设计 validation set

一个好的 validation set 应该满足几个条件。

第一，严格 held-out。

验证集不能参与训练，也不能在数据处理过程中被泄漏进训练集。

第二，数据版本固定。

如果每次训练使用不同验证集，就难以比较实验。

第三，规模足够。

太小的验证集噪声大，loss 波动会误导判断。

第四，领域覆盖合理。

验证集要覆盖目标模型的重要能力领域，而不是只覆盖最容易收集的网页文本。

第五，保留分桶信息。

每个样本应该有 metadata，例如：

1. source。
2. domain。
3. language。
4. length bucket。
5. quality score。
6. timestamp。
7. difficulty。

这样才能做分桶评估。

第六，去重和污染检测。

验证集需要和训练数据做 exact dedup 和 near dedup，避免验证 loss 虚低。

第七，稳定但可扩展。

核心验证集应该固定，便于长期比较；同时可以增加新的专项验证集，用于新能力跟踪。

一个实用设计是：

```text
core_validation：长期固定，用于跨实验比较
domain_validation：按领域分桶，用于能力分析
fresh_validation：新近数据，用于检测时效性和泛化
stress_validation：困难样本，用于压力测试
```

### 四、Benchmark set 和 validation set 的区别

validation set 主要用于语言建模 loss。

benchmark set 主要用于任务表现。

两者不同。

例如：

1. validation set 问题：给定文本前缀，预测下一个 token。
2. benchmark set 问题：给定数学题，输出正确答案。

validation loss 是 token-level 指标。

benchmark 通常是 task-level 指标。

数学题只要最终答案错，就算错。

代码题只要测试不通过，就算错。

安全题可能要同时满足拒答正确性和有用性。

所以 benchmark 更贴近用户能力，但也更容易有噪声。

benchmark 设计要注意：

1. 样本量足够。
2. 题目难度覆盖。
3. 评分规则稳定。
4. prompt 模板固定。
5. 解码参数固定。
6. 避免训练污染。
7. 报告置信区间或多次运行结果。

尤其是生成式 benchmark，解码随机性会影响结果。

如果使用 temperature sampling，最好多次运行或固定 seed。

如果使用 greedy decoding，结果更稳定，但可能低估某些需要探索的能力。

### 五、评估集污染：最危险的评估陷阱

评估集污染指 benchmark 或 validation 样本进入训练数据。

污染会导致模型看起来很强，但实际泛化能力没那么强。

污染类型包括：

1. exact contamination：完全相同样本进入训练集。
2. near contamination：轻微改写、格式变化、答案保留。
3. template contamination：题型模板泄漏。
4. solution contamination：题目没泄漏，但答案或解析泄漏。
5. benchmark-derived synthetic data：用 benchmark 风格生成合成数据，过度贴近测试分布。

评估污染的危险在于，它会影响关键决策：

1. 错误判断模型能力。
2. 错误选择 checkpoint。
3. 错误决定是否继续训练。
4. 错误发布模型结果。

污染检测常用方法：

1. exact match。
2. n-gram overlap。
3. MinHash / SimHash。
4. embedding similarity。
5. prompt-answer pair matching。
6. 时间切分。
7. benchmark 关键词和 ID 检测。

但污染检测很难做到完美。

所以还需要设计 fresh eval。

fresh eval 指训练数据截止时间之后的新评估集，模型不可能提前见过。

例如训练数据截止到 2024 年 1 月，可以用 2024 年 6 月之后产生的新题、新闻、代码 issue 做评估。

### 六、Canary set 和 stress test

除了常规验证集，还可以设计 canary set。

Canary set 是一组专门用于检测异常的小集合。

它不一定用于主指标，而是用于报警。

例如：

1. 是否出现重复输出。
2. 是否突然丢失某种格式能力。
3. 是否安全拒答退化。
4. 是否长上下文检索失效。
5. 是否代码生成格式异常。
6. 是否多语言能力突然下降。

Stress test 则是压力测试。

它可以包含：

1. 超长上下文样本。
2. 噪声输入。
3. 对抗 prompt。
4. 罕见语言。
5. 难数学题。
6. 多工具调用任务。
7. 易混淆事实问题。

Canary set 和 stress test 的作用不是给模型打总分，而是帮助快速发现训练或后训练中的退化。

### 七、Early stopping 在小模型中的经典含义

在传统机器学习中，early stopping 通常指：

> 当验证集指标不再提升，或者开始变差时，停止训练，防止过拟合。

典型情况是：

```text
训练 loss：持续下降
验证 loss：先下降，后上升
```

此时继续训练会让模型更拟合训练集，但泛化变差。

所以可以在验证 loss 最低点附近停止。

常见参数包括：

1. patience：允许多少轮不提升。
2. min_delta：提升至少多少才算有效。
3. monitor metric：监控哪个指标。
4. restore best weights：恢复最佳 checkpoint。

例如：

```text
如果 validation loss 连续 5 次评估没有下降超过 0.001，就停止训练。
```

这在小数据、小模型和监督学习中很常见。

### 八、Early stopping 在大模型预训练中更复杂

大模型预训练不能机械照搬传统 early stopping。

原因有几个。

第一，大模型通常没有明显过拟合。

因为训练数据极大，模型可能很长时间内 validation loss 都在下降。

第二，训练预算通常预先规划。

根据 Scaling Law 和算力预算，团队会提前决定训练 token 数。

第三，下游能力可能滞后于 loss。

有些能力需要训练到一定规模后才显现，过早停止可能错过能力提升。

第四，validation loss 的微小变化不一定对应产品能力。

如果 loss 下降变慢，但关键 benchmark 还在提升，可能仍值得继续训练。

第五，数据 curriculum 或配比变化会影响曲线。

训练后期如果加入更难数据，loss 可能短期变差，但长期能力更强。

因此，大模型中的 early stopping 更像 budget-aware stopping 或 checkpoint selection，而不是简单“验证 loss 不降就停”。

可以关注：

1. loss 下降边际收益。
2. 下游 benchmark 边际收益。
3. 训练成本。
4. 推理成本目标。
5. 数据是否耗尽。
6. 是否出现能力退化。
7. 是否出现训练不稳定。

### 九、如何判断是否继续训练

真实项目中，判断是否继续训练可以从五个角度看。

第一，validation loss 是否仍在下降。

如果 overall 和关键 domain loss 都持续下降，继续训练通常有价值。

第二，下游 benchmark 是否仍在提升。

如果 loss 降得慢，但数学、代码、指令遵循还在涨，可能值得继续。

第三，边际收益是否值得成本。

如果继续训练 100B token 只带来极小提升，而成本很高，可能不划算。

第四，是否出现目标能力退化。

如果 overall loss 下降，但代码、安全或多语言退化，要重新评估数据配比，而不是盲目继续。

第五，训练是否稳定。

如果频繁 loss spike、NaN、硬件故障，继续训练风险上升，可能需要回滚或调整 recipe。

一个简单决策框架：

```text
继续训练：关键指标提升，成本可接受，稳定性正常
调整训练：overall 提升但目标能力退化
停止训练：边际收益很低，成本高，且关键指标不再提升
回滚重训：出现污染、严重不稳定或错误数据配比
```

### 十、Checkpoint selection：选哪个模型

很多时候不是“何时停止”，而是“选哪个 checkpoint”。

训练结束后可能有多个候选：

```text
ckpt_900B_tokens
ckpt_1T_tokens
ckpt_1.1T_tokens
ckpt_1.2T_tokens
```

最后一个 checkpoint 不一定最好。

选择需要看目标。

如果是 base model：

1. validation loss。
2. domain loss。
3. 基础 benchmark。
4. 稳定性。

如果是 chat model 起点：

1. instruction following potential。
2. 知识和推理能力。
3. 安全风险。
4. SFT 后表现。

如果是 code model：

1. code validation loss。
2. HumanEval/MBPP。
3. 真实 repo 任务。
4. 编译和测试通过率。

如果是 multilingual model：

1. per-language loss。
2. 低资源语言 benchmark。
3. 翻译和跨语言 QA。

所以 checkpoint selection 应该是多指标决策。

可以设置 gate：

1. validation loss 不能明显退化。
2. safety eval 必须达标。
3. 目标 benchmark 必须超过阈值。
4. 不允许明显训练异常。

在 gate 之后，再按目标指标排序。

### 十一、评估频率如何设计

评估太频繁会浪费算力。

评估太少会错过问题。

常见策略是分层评估。

第一层：高频轻量评估。

例如每隔固定 step 计算 validation loss 和少量 canary。

第二层：中频 benchmark。

例如每几个 checkpoint 跑一次 MMLU、GSM8K、HumanEval 子集。

第三层：低频完整评估。

例如对关键 checkpoint 跑完整 benchmark、安全评估、人类偏好评估。

第四层：最终 hidden eval。

只对候选最终模型运行，避免过度调参。

一个例子：

```text
每 1000 step：validation loss
每 10000 step：light benchmark suite
每 50000 step：full benchmark suite
最终候选：hidden eval + safety eval + human eval
```

评估频率也要考虑模型规模。

70B 模型跑一次完整评估可能很贵，不能像小模型一样频繁评估。

### 十二、评估指标的统计可靠性

评估不是跑一个数字就结束。

要考虑统计可靠性。

常见问题包括：

1. benchmark 样本太少。
2. 随机解码导致方差。
3. prompt 模板影响结果。
4. judge model 不稳定。
5. 多次比较导致偶然最优。
6. 差异没有统计显著性。

例如某 benchmark 只有 100 道题。

模型 A 答对 71 道，模型 B 答对 73 道。

能不能说 B 明显更强？

不一定。

这可能只是统计波动。

更可靠做法包括：

1. 增大样本量。
2. 固定 prompt 和解码。
3. 多 seed 运行。
4. 报告置信区间。
5. 使用 paired comparison。
6. 对重要任务做人类复核。

面试中可以强调：

> 评估系统本身也有噪声，checkpoint selection 要避免被小样本波动误导。

### 十三、评估过拟合

评估过拟合指团队不断根据同一套 benchmark 调参，最终模型在这套 benchmark 上很好，但真实泛化不一定好。

它和训练集过拟合类似，只是发生在实验决策层面。

常见表现：

1. 反复优化 MMLU prompt。
2. 根据某几个 benchmark 调数据配比。
3. 根据测试集选择 checkpoint。
4. 用公开 benchmark 生成大量相似训练数据。
5. 报告最优一次运行而不是稳定结果。

缓解方法：

1. 区分 dev set 和 hidden test set。
2. 限制 final test 使用次数。
3. 增加 fresh eval。
4. 使用多维评估，不只追一个榜单。
5. 对关键结论做人工和线上验证。

### 十四、一个实用评估面板设计

一个大模型预训练评估面板可以包含：

训练健康：

1. train loss。
2. validation loss。
3. grad norm。
4. learning rate。
5. loss spike count。

分桶 loss：

1. web。
2. code。
3. math。
4. multilingual。
5. long context。
6. dialogue。

基础能力：

1. MMLU。
2. ARC。
3. HellaSwag。
4. TruthfulQA。

推理和代码：

1. GSM8K。
2. MATH。
3. HumanEval。
4. MBPP。

对话和安全：

1. instruction following。
2. pairwise preference。
3. refusal accuracy。
4. jailbreak robustness。

系统指标：

1. eval latency。
2. token cost。
3. checkpoint size。
4. serving memory。

这样的面板能支持更稳健的训练决策。

### 十五、面试回答模板

如果面试官问：

> 大模型训练中如何设计评估集？early stopping 怎么做？

可以这样回答：

我会先区分不同评估集。训练过程中需要 held-out validation set 来监控语言建模 loss，还要按 web、code、math、多语言、长上下文等 domain 做分桶 loss，避免 overall loss 掩盖局部退化。除此之外，还需要 downstream benchmark 评估具体能力，例如数学、代码、指令遵循、安全和长上下文。最终还应该有 hidden test 或 fresh eval，用于避免频繁调参导致评估过拟合。

评估集设计上，我会保证严格 held-out、数据版本固定、规模足够、领域覆盖合理，并和训练集做 exact/near dedup，防止 benchmark contamination。对于生成式 benchmark，要固定 prompt、解码策略，并考虑随机性和置信区间。

early stopping 在大模型预训练里不能机械照搬小模型。小模型常根据 validation loss 不再下降来停止，但大模型通常根据 Scaling Law 和预算预先规划训练 token，validation loss 可能长期下降。实际决策更像 budget-aware stopping 和 checkpoint selection：看 validation loss、domain loss、下游 benchmark、边际收益、训练成本和稳定性。如果 overall loss 下降但目标能力退化，应调整数据配比；如果关键指标边际收益很低且成本高，可以停止；如果出现污染或严重不稳定，应该回滚或重训。

最终 checkpoint 选择也不一定选最后一个或 loss 最低的，而要根据目标场景建立多指标 gate，例如 loss 不退化、安全达标、目标 benchmark 超阈值，再选择最适合产品或后训练的 checkpoint。

### 十六、常见追问与回答

问题 1：validation set 和 test set 有什么区别？

回答：validation set 用于训练过程监控和实验选择，test set 用于最终可信评估。test set 使用次数应少，否则会被实验决策过拟合。

问题 2：为什么要做 domain validation？

回答：overall loss 是平均值，会掩盖局部退化。domain validation 可以发现代码、数学、多语言、长上下文等特定能力是否变好或变差。

问题 3：大模型预训练是否需要 early stopping？

回答：需要停止决策，但不一定是传统意义的 validation loss early stopping。大模型更常根据预算、边际收益、loss 曲线、benchmark、稳定性和数据情况综合决定。

问题 4：如果 validation loss 还在下降，但 benchmark 不涨，要不要继续训练？

回答：要看目标。如果 benchmark 对应核心能力，需要检查数据配比、后训练需求、评估噪声和任务难度。如果只是通用 loss 下降但目标能力没有收益，继续训练的价值可能有限。

问题 5：如何防止评估集污染？

回答：要做训练集和评估集的 exact/near dedup，检查题目、答案、解析和模板泄漏；使用时间切分和 fresh eval；避免用 benchmark 原题或高度相似数据做训练。

问题 6：checkpoint selection 应该看哪个指标？

回答：取决于目标。base model 更看 validation loss 和基础能力，code model 更看代码评估，chat model 更看指令遵循和偏好，reasoning model 更看数学代码推理。最好建立多指标 gate，而不是单指标选择。

### 十七、常见误区

1. 误区：overall validation loss 下降就说明模型所有能力都变强。
   纠正：overall loss 会掩盖 domain 退化，需要分桶评估。

2. 误区：公开 benchmark 分数越高，模型一定越强。
   纠正：可能存在污染、prompt 过拟合、评估噪声和统计不显著。

3. 误区：test set 可以反复用于调参。
   纠正：反复使用会导致评估过拟合，test set 失去最终验证意义。

4. 误区：大模型 early stopping 就是 validation loss 不降就停。
   纠正：大模型要结合预算、scaling、benchmark、稳定性和边际收益。

5. 误区：最后一个 checkpoint 一定最好。
   纠正：后期可能出现能力退化、过拟合某些分布或安全下降，需要多指标选择。

6. 误区：评估越频繁越好。
   纠正：评估有计算成本，也可能导致过度调参，应分层评估。

### 十八、小练习

1. 设计一个包含 overall、domain、benchmark、safety 的评估面板。
2. 为什么 validation loss 不能替代 HumanEval 或 GSM8K？
3. 举三个 benchmark contamination 的例子。
4. 解释 traditional early stopping 和大模型 budget-aware stopping 的区别。
5. 为 code model 设计 checkpoint selection 的多指标 gate。
6. 用 2 分钟回答：“validation loss 还在下降，但下游能力不涨，你怎么决策？”

### 本讲总结

本讲最重要的结论：

1. 评估集设计决定训练决策质量，不能只看一个 overall validation loss。
2. validation set、domain validation、downstream benchmark 和 hidden test 各有不同用途。
3. 好的验证集需要严格 held-out、版本固定、规模足够、分桶明确，并做污染检测。
4. benchmark 更接近能力，但也有污染、噪声、prompt 敏感和统计不显著等风险。
5. 大模型 early stopping 更像 budget-aware stopping，不应机械套用小模型规则。
6. checkpoint selection 应根据目标场景做多指标决策，而不是只选最后一个或 loss 最低的。
7. 面试中成熟的回答要同时覆盖评估设计、污染防护、边际收益、训练成本和模型目标。

## 第 39 讲：多语言预训练

### 本讲目标

学完这一讲，你应该能够回答：

1. 为什么现代大模型通常需要多语言能力。
2. 多语言预训练相比单语言预训练多了哪些问题。
3. 数据配比、tokenizer、语言识别、去重和质量过滤如何影响多语言能力。
4. 高资源语言和低资源语言之间为什么会产生迁移、竞争和干扰。
5. 如何评估多语言模型，避免只看英文 benchmark。
6. 面试中如何回答“如何训练一个中英双语或多语言大模型”。

多语言预训练不是简单地把不同语言数据混在一起训练。

它涉及数据、tokenizer、模型容量、采样策略、评估和对齐的一整套权衡。

一个英文模型加一点中文数据，不一定就会变成好的中英双语模型。

一个多语言模型如果数据配比不合理，也可能出现：

1. 英文能力强，中文能力弱。
2. 高资源语言强，低资源语言几乎不可用。
3. 翻译腔严重。
4. 多语言混杂输出。
5. 某些语言 tokenizer 效率很差。
6. benchmark 看起来不错，但真实对话很差。

所以多语言预训练是训练工程中的重要专题。

### 一、为什么需要多语言预训练

大模型服务真实用户时，很难只面对一种语言。

即使主要产品是英文，也会遇到：

1. 多语言用户输入。
2. 跨语言搜索和问答。
3. 翻译。
4. 国际化客服。
5. 多语言代码注释。
6. 多语言文档理解。
7. 跨语言知识迁移。

对于中文用户，多语言能力尤其重要。

因为很多高质量技术资料、论文、代码注释和社区讨论是英文的，而用户可能用中文提问。

模型需要做到：

1. 理解中文问题。
2. 利用英文知识。
3. 用中文解释。
4. 必要时保留英文术语。

这就是跨语言迁移能力。

多语言预训练的目标不是每种语言都完全一样强，而是根据产品目标设计能力覆盖。

例如：

1. 全球通用助手：需要广泛多语言能力。
2. 中英技术助手：重点中英双语和代码。
3. 翻译模型：重点语言对质量。
4. 低资源语言助手：重点低资源语言补齐。
5. 企业内部模型：重点企业文档中的语言分布。

目标不同，数据配比和评估也不同。

### 二、多语言预训练的核心挑战

多语言预训练主要有六类挑战。

第一，数据量不均衡。

英文网页、代码、论文数据很多。

很多低资源语言数据很少。

如果按原始数据量采样，英文会压倒其他语言。

第二，数据质量差异大。

不同语言的网页质量不同，有些语言数据中机器翻译、模板站、重复内容、乱码比例更高。

第三，tokenizer 效率不同。

同样一句话，不同语言可能被切成不同数量的 token。

如果某种语言 token 化效率低，训练和推理成本都会更高。

第四，模型容量竞争。

模型参数有限，不同语言共享容量。

高资源语言可能占用更多模型能力，低资源语言被边缘化。

第五，跨语言干扰。

训练某些语言可能帮助另一些语言，也可能干扰另一些语言。

第六，评估复杂。

英文 benchmark 很多，其他语言 benchmark 不足，而且翻译 benchmark 可能引入偏差。

这些问题决定了多语言训练不能只靠“多抓数据”。

### 三、语言数据配比：最核心的工程决策

多语言预训练最关键的问题之一是：

> 每种语言采多少数据？训练时按什么概率采样？

如果按原始数据量采样，通常英文占比会非常高。

低资源语言几乎没有训练信号。

如果强行把低资源语言采样到很高比例，又可能导致：

1. 重复训练低质量数据。
2. 过拟合低资源语料。
3. 牺牲高资源语言能力。
4. 总体 loss 变差。

常见采样策略包括：

第一，按原始比例采样。

优点：简单，接近真实数据分布。

缺点：高资源语言支配训练。

第二，均匀按语言采样。

每种语言概率相近。

优点：低资源语言得到更多训练。

缺点：低资源语言可能被重复过多，高资源语言利用不足。

第三，温度采样。

这是多语言训练常用方法。

假设语言 `i` 的原始数据比例是 `p_i`，采样比例可以设为：

```text
q_i = p_i^(1/T) / sum_j p_j^(1/T)
```

当 `T = 1` 时，接近原始比例。

当 `T > 1` 时，分布变平，低资源语言占比提高。

当 `T` 很大时，接近均匀采样。

温度采样的直觉是：

> 不让英文完全压倒其他语言，也不让低资源语言被过度重复。

第四，目标驱动采样。

根据产品目标设定语言比例。

例如中英双语助手可以设置英文、中文、代码为核心，其它语言少量覆盖。

第五，动态采样。

训练中根据 per-language validation loss 或 benchmark 变化调整采样比例。

例如某语言 loss 长期不降，可以提高该语言比例；某语言出现过拟合，可以降低比例。

但动态采样更复杂，容易引入训练不稳定。

### 四、数据质量：不同语言不能用同一把尺子

多语言数据清洗比单语言更难。

因为不同语言的文本特征不同。

英文网页清洗规则不能直接用于中文、阿拉伯语、泰语或印地语。

常见问题包括：

1. 语言识别错误。
2. 混合语言文本。
3. 编码错误。
4. OCR 噪声。
5. 机器翻译痕迹。
6. 模板重复。
7. 低质量论坛灌水。
8. 分词和标点差异。
9. 方向性文字处理，例如阿拉伯语、希伯来语。

语言识别是第一步。

如果语言 ID 错了，后续配比和评估都会错。

例如中文网页中夹杂大量英文代码，不能简单判定为英文或中文。

真实系统可能会使用：

1. document-level language ID。
2. paragraph-level language ID。
3. sentence-level language ID。
4. token-level language mixture statistics。

质量过滤也要分语言校准。

例如英文中连续没有空格可能是乱码，但中文本来就没有空格。

英文 stopword 比例规则不能直接用于中文。

不同语言的平均字符长度、标点分布、Unicode 范围都不同。

所以多语言清洗需要 language-aware filters。

### 五、Tokenizer 对多语言能力的影响

Tokenizer 是多语言预训练中的关键因素。

一个 tokenizer 如果主要由英文数据训练，可能对中文、日文、韩文、阿拉伯语等语言效率很差。

所谓 tokenizer 效率，可以用每个字符或每个词对应多少 token 来衡量。

例如同样长度的一句话：

1. 英文被切成 20 个 token。
2. 中文被切成 60 个 token。

那么中文训练和推理成本会更高。

这会带来几个问题：

1. 中文 context 实际可容纳信息更少。
2. 中文生成更慢。
3. 中文 loss 可能不可直接与英文比较。
4. 同样 token budget 下，中文语义信息更少。

设计多语言 tokenizer 要考虑：

1. tokenizer 训练语料的语言配比。
2. vocab size。
3. 字符覆盖。
4. byte fallback。
5. 是否支持 CJK。
6. 是否保护代码 token。
7. 是否保留常见符号和数字模式。

常见策略是使用 SentencePiece、BPE 或 Unigram，并在 tokenizer 训练数据中加入多语言平衡采样。

但 tokenizer 也有权衡。

如果 vocab 太大：

1. embedding 和 output head 参数增加。
2. softmax 成本增加。
3. 低频 token 学习不足。

如果 vocab 太小：

1. 多语言切分过碎。
2. sequence 变长。
3. 推理成本增加。

所以 tokenizer 不是越大越好，而是要结合语言覆盖和训练成本。

### 六、跨语言迁移：为什么多语言可以互相帮助

多语言模型的一个优势是跨语言迁移。

例如模型在英文中学到数学、代码、科学知识，可能帮助它用中文回答相关问题。

这是因为不同语言共享一些抽象能力：

1. 世界知识。
2. 逻辑推理。
3. 数学结构。
4. 代码语义。
5. 对话模式。
6. 文档结构。

如果模型学会了“链式推理”这种能力，它不一定只限于英文。

但迁移不是自动完美发生。

它依赖：

1. 模型容量是否足够。
2. tokenizer 是否能高效表示目标语言。
3. 目标语言中是否有足够锚点数据。
4. 是否有平行语料或跨语言对齐信号。
5. 后训练是否覆盖目标语言。

例如一个模型英文数学很强，但中文数学题表现差，可能不是不会数学，而是：

1. 中文题目理解弱。
2. 中文术语映射弱。
3. 中文推理数据少。
4. 中文后训练不足。
5. 解码时语言风格不稳定。

因此提升中文数学能力，可以不只增加中文预训练文本，也可以加入：

1. 中文数学题。
2. 中英平行数学解释。
3. 中文 SFT reasoning 数据。
4. 可验证答案过滤。

### 七、跨语言干扰和容量竞争

多语言训练也可能产生干扰。

模型容量有限，不同语言共享参数。

如果语言数量很多，而模型不够大，可能出现所有语言都一般。

高资源语言和低资源语言之间也会竞争。

例如提高低资源语言采样比例后，英文能力可能略降。

这不是因为低资源语言“有害”，而是固定训练预算下资源被重新分配。

常见干扰包括：

1. 语言混杂输出。
2. 翻译腔。
3. 低资源语言过拟合。
4. 高资源语言能力下降。
5. 相似语言之间混淆。
6. 不同文字系统之间表示效率差异。

缓解方式包括：

1. 更合理的数据配比。
2. 增大模型容量。
3. language-aware sampling。
4. 多语言 tokenizer 改进。
5. 后训练中加入目标语言指令数据。
6. 使用 language tag 或 system prompt 控制输出语言。
7. 对混杂输出做专门评估和修正。

### 八、中英双语模型的实际设计

如果面试官问：

> 你要训练一个中英双语大模型，怎么设计？

可以从以下方面展开。

第一，明确目标。

是通用中英助手、中文助手、技术问答模型、代码模型，还是翻译模型？

第二，数据配比。

中英双语模型通常不应简单按原始网页数量采样，否则英文可能过多。

可以根据目标设置中文、英文、代码、数学、对话数据比例。

第三，tokenizer。

需要保证中文 tokenization 效率，避免中文被切得过碎。

同时要保留英文和代码能力。

第四，质量过滤。

中文数据要处理乱码、模板站、采集噪声、繁简体、低质量问答、机器翻译痕迹。

英文数据要处理网页模板、重复、低质量 SEO 文本。

第五，去重和污染检测。

中英文都要做 exact 和 near dedup，并特别检查中文 benchmark、翻译 benchmark、代码 benchmark 泄漏。

第六，评估。

不能只看英文 MMLU，也不能只看中文 C-Eval。

需要组合：

1. 英文通用能力。
2. 中文通用能力。
3. 中英翻译。
4. 中文数学。
5. 英文数学。
6. 代码。
7. 中英混合问答。
8. 中文对话偏好。
9. 安全评估。

第七，后训练。

如果最终是 chat model，需要中文和英文 SFT、偏好数据、安全数据。

否则 base model 可能有双语知识，但不会稳定按用户语言回答。

### 九、低资源语言：数据少怎么办

低资源语言的核心问题是高质量数据少。

直接重复少量数据会过拟合。

常见方法包括：

1. 温度采样提高低资源语言比例。
2. 使用高质量平行语料。
3. 用强模型生成合成数据。
4. 翻译高质量指令数据。
5. 跨语言迁移。
6. 使用相关语言数据辅助。
7. 人工构造小而高质量的评估集。

但每种方法都有风险。

机器翻译数据可能有翻译腔。

合成数据可能引入错误。

相关语言可能导致混淆。

重复采样可能导致过拟合。

所以低资源语言训练更强调质量和评估闭环。

一个实用策略是：

1. 预训练阶段保证基本覆盖。
2. 后训练阶段用高质量指令数据增强可用性。
3. 用本地母语评估或人工评估检查真实质量。
4. 持续收集用户反馈改进。

### 十、多语言评估怎么做

多语言评估不能只翻译英文 benchmark。

翻译 benchmark 有价值，但有局限：

1. 翻译可能改变题意。
2. 翻译文本不符合母语表达习惯。
3. 文化背景不同。
4. 术语翻译可能不稳定。
5. 模型可能利用英文原题记忆。

更好的评估组合包括：

1. 原生语言 benchmark。
2. 翻译 benchmark。
3. 跨语言问答。
4. 多语言阅读理解。
5. 多语言数学和代码。
6. 翻译质量。
7. 语言一致性。
8. 人类偏好评估。
9. 真实用户任务。

还要看 per-language loss。

例如：

```text
English validation loss
Chinese validation loss
Spanish validation loss
Arabic validation loss
Hindi validation loss
```

但不同语言 loss 不能简单横向比较，因为 tokenizer 效率和语言结构不同。

更适合看同一语言内部不同 checkpoint 的趋势。

例如中文 loss 从 2.1 降到 1.9，说明中文建模变好；但不能简单说中文 loss 2.1 比英文 loss 1.7 差多少，因为 tokenization 不同。

### 十一、语言一致性和输出控制

多语言模型常见问题是语言不一致。

例如用户用中文问，模型英文回答。

或者回答中中英文混杂。

这可能来自：

1. 预训练中英文占比过高。
2. 后训练中文指令数据不足。
3. system prompt 不明确。
4. tokenizer 对中文不友好。
5. 模型学到英文回答更常见。

解决方法包括：

1. 在 SFT 中加入多语言指令和回答。
2. 明确要求按用户语言回答。
3. 训练语言保持任务。
4. 对语言混杂输出做 reward 或过滤。
5. 评估 language consistency。

语言一致性评估可以看：

1. 回答主要语言是否与用户一致。
2. 是否正确保留专业英文术语。
3. 是否出现不必要翻译。
4. 是否混入其他语言。

对于技术问答，完全禁止英文不一定合理。

例如中文回答中保留 `Transformer`、`attention`、`loss` 是正常的。

所以评估要区分合理术语和不合理混杂。

### 十二、代码、数学与多语言的关系

代码和数学是多语言模型中的特殊领域。

代码天然包含英文关键字和符号。

中文用户可能用中文描述需求，让模型生成英文代码。

这需要跨语言 grounding。

例如：

```text
用户：写一个函数，输入两个列表，返回它们的交集。
模型：def intersection(a, b): ...
```

模型要理解中文需求，并生成 Python 代码。

数学也是类似。

中文数学题可能包含中文表述、阿拉伯数字、公式符号和英文变量。

模型需要把自然语言、符号和推理结构对齐。

所以中英技术模型通常要重视：

1. 中文代码指令数据。
2. 英文代码语料。
3. 中英混合技术文档。
4. 中文数学题和英文数学题。
5. 可执行评估。

这类能力不是单纯提高中文网页比例就能解决的。

### 十三、工程坑：多语言训练常见问题

坑一：语言识别错误。

导致数据配比统计失真。

坑二：tokenizer 对某些语言极低效。

导致该语言训练成本高、context 利用率低。

坑三：低资源语言过采样导致重复。

模型可能记住数据，泛化差。

坑四：只看英文 benchmark。

模型看起来很强，但多语言体验差。

坑五：翻译 benchmark 误导。

翻译质量、文化背景和原题泄漏都会影响评估。

坑六：后训练语言覆盖不足。

base model 有多语言知识，但 chat model 不按目标语言回答。

坑七：繁简体、地区词汇和术语没有处理。

例如中文里大陆、台湾、香港用词可能不同。

坑八：安全策略只覆盖英文。

模型可能在非英文语言中安全能力明显下降。

### 十四、面试回答模板

如果面试官问：

> 如何训练一个多语言大模型？

可以这样回答：

我会先明确目标语言和产品场景，因为全球多语言助手、中英双语技术助手和低资源语言模型的数据配比完全不同。预训练阶段需要收集多语言高质量语料，先做 language ID、质量过滤、去重和污染检测，并保留 language、domain、quality 等 metadata。数据采样不能简单按原始数据量，否则英文等高资源语言会占主导；也不能完全均匀采样，否则低资源语言可能重复过多。常用做法是温度采样或目标驱动配比，在高资源和低资源语言之间折中。

Tokenizer 也很关键，要保证不同语言的 tokenization 效率，特别是中文、日文、韩文、阿拉伯语等非英文语言，避免某些语言被切得过碎。模型训练中要监控 per-language validation loss 和 domain loss，同时注意跨语言迁移和干扰。高资源语言可能帮助低资源语言学习知识和推理，但也可能因为容量竞争导致低资源语言被压制。

评估上不能只看英文 benchmark，需要原生多语言 benchmark、翻译任务、跨语言问答、语言一致性、代码数学任务和安全评估。对于最终 chat model，还需要多语言 SFT、偏好数据和安全数据，否则 base model 有多语言知识也不一定能稳定按用户语言回答。

### 十五、常见追问与回答

问题 1：多语言数据配比怎么定？

回答：取决于目标。可以从原始比例、均匀采样、温度采样和目标驱动采样中选择。真实项目常用温度采样或人工设定核心语言比例，并用 per-language eval 反馈调整。

问题 2：为什么按原始数据比例采样不好？

回答：因为高资源语言数据量远大于低资源语言，按原始比例会让英文等语言主导训练，低资源语言几乎学不到。

问题 3：为什么不能完全均匀采样？

回答：低资源语言数据少，均匀采样可能导致重复训练和过拟合，也会牺牲高资源语言的数据利用效率。

问题 4：Tokenizer 为什么影响多语言能力？

回答：tokenizer 决定不同语言被切成多少 token。切得过碎会降低 context 利用率、增加推理成本，并让同样 token budget 下该语言获得更少语义信息。

问题 5：英文能力能迁移到中文吗？

回答：部分可以，尤其是世界知识、代码、数学和推理结构。但迁移依赖中文理解、tokenizer、中文锚点数据和中文后训练。不能认为英文强就自动中文强。

问题 6：如何评估多语言模型？

回答：要结合 per-language validation loss、原生语言 benchmark、翻译 benchmark、跨语言问答、语言一致性、人类偏好和安全评估。同一语言内部的 loss 趋势更有意义，不同语言之间的 loss 不能简单横比。

### 十六、常见误区

1. 误区：把多语言数据混在一起训练就能得到好的多语言模型。
   纠正：还需要合理配比、tokenizer、质量过滤、评估和后训练。

2. 误区：英文模型加少量中文数据就会变成强中文模型。
   纠正：中文理解、中文生成、中文推理和中文对话都需要足够数据和评估闭环。

3. 误区：低资源语言采样越多越好。
   纠正：过采样会导致重复和过拟合，需要质量控制和温度采样。

4. 误区：多语言 loss 可以直接横向比较。
   纠正：不同语言 tokenizer 和结构不同，loss 更适合同语言内比较趋势。

5. 误区：翻译 benchmark 就能代表多语言能力。
   纠正：翻译 benchmark 有价值，但不能替代原生语言任务和真实用户评估。

6. 误区：多语言安全能力会自动跟随英文安全能力。
   纠正：安全对齐也需要多语言数据和评估，否则非英文安全可能退化。

### 十七、小练习

1. 解释温度采样如何提高低资源语言比例。
2. 设计一个中英双语技术助手的预训练数据配比方案。
3. 列出 5 个多语言 tokenizer 需要关注的指标。
4. 为什么同一模型的中文 loss 和英文 loss 不能简单横向比较？
5. 设计一个多语言评估面板，至少包含 8 类指标。
6. 用 2 分钟回答：“英文强的模型为什么中文不一定强？”

### 本讲总结

本讲最重要的结论：

1. 多语言预训练不是简单混合数据，而是数据配比、tokenizer、质量过滤、迁移、干扰和评估的系统工程。
2. 高资源语言和低资源语言存在严重数据不均衡，常用温度采样或目标驱动采样折中。
3. Tokenizer 效率会直接影响不同语言的训练成本、context 利用率和推理效率。
4. 跨语言迁移可以让模型把英文知识、代码和推理能力迁移到其他语言，但需要目标语言锚点数据和后训练支持。
5. 多语言训练存在容量竞争和跨语言干扰，需要通过配比、模型容量、采样策略和评估监控缓解。
6. 多语言评估不能只看英文 benchmark，也不能只看翻译 benchmark，要结合原生任务、跨语言任务、语言一致性和安全评估。
7. 面试中成熟的回答要覆盖目标定义、数据配比、tokenizer、清洗去重、训练监控、评估和后训练。

## 第 40 讲：代码模型预训练

### 本讲目标

学完这一讲，你应该能够回答：

1. 代码模型预训练和通用文本预训练有什么不同。
2. 代码数据从哪里来，如何清洗、去重和做许可风险控制。
3. 为什么代码模型需要考虑 repository-level context、文件结构、测试和执行反馈。
4. 代码 tokenizer、数据配比、语言配比和自然语言混合如何设计。
5. 如何评估代码模型，为什么 HumanEval 远远不够。
6. 面试中如何回答“如何训练一个代码大模型”。

代码模型是大模型能力体系中的重要方向。

它不仅用于代码补全，还用于：

1. 代码生成。
2. bug 修复。
3. 单元测试生成。
4. 代码解释。
5. repo 级别问答。
6. 代码迁移和重构。
7. 自动化 agent。
8. 软件工程任务执行。

相比普通文本，代码有更强的结构、更严格的正确性要求和更明确的执行反馈。

一句自然语言回答可能表达略有不同也能接受。

但一段代码少一个括号、变量名错一个字符、缩进错一层，就可能完全不能运行。

所以代码模型预训练不是“多加一些 GitHub 数据”这么简单。

### 一、代码数据和自然语言数据有什么不同

代码数据有几个明显特点。

第一，语法结构强。

代码必须符合编程语言语法。

例如 Python 依赖缩进，C++ 依赖括号和类型声明，JavaScript 有异步和模块系统。

第二，长程依赖强。

一个函数可能依赖文件开头的 import、类定义、全局变量、配置文件和其他文件。

第三，正确性可执行。

代码可以运行测试，这比普通文本评估更客观。

第四，重复和模板多。

开源代码中有大量 boilerplate、vendor code、生成代码、复制粘贴代码。

第五，许可和安全风险高。

代码数据涉及 license、敏感信息、密钥、漏洞、恶意代码等问题。

第六，自然语言和代码混合。

代码仓库中有 README、docstring、注释、issue、commit message、测试说明。

这些自然语言对代码理解非常重要。

因此代码模型预训练要同时处理：

1. code tokens。
2. comments。
3. docs。
4. tests。
5. repository metadata。
6. natural language instructions。

### 二、代码数据来源

常见代码数据来源包括：

1. GitHub、GitLab 等开源仓库。
2. package registry，例如 npm、PyPI、Maven、Crates。
3. 官方文档和教程。
4. Stack Overflow 和技术问答。
5. issue、pull request、commit message。
6. 单元测试和 benchmark。
7. 合成代码数据。
8. 企业内部代码库。

不同来源价值不同。

开源仓库提供真实代码结构。

官方文档提供高质量解释和 API 用法。

技术问答提供问题和解决方案。

issue 和 PR 提供 bug 修复、代码 review 和开发过程信号。

测试提供可验证正确性的监督信号。

企业内部代码提供特定业务领域能力，但有权限和隐私要求。

代码模型训练不能只看文件内容，还要尽量保留仓库结构。

例如：

```text
repo/
  README.md
  src/model.py
  src/train.py
  tests/test_model.py
  requirements.txt
```

这些文件之间有关系。

如果训练时完全打散文件，模型就难以学习 repo-level 依赖。

### 三、代码数据清洗

代码数据清洗比普通文本更复杂。

需要过滤：

1. 空文件。
2. 超大文件。
3. minified JavaScript。
4. 自动生成代码。
5. vendor dependencies。
6. 二进制或编码错误文件。
7. 重复文件。
8. 配置生成物。
9. 含密钥或 token 的文件。
10. 明显恶意代码。

还要识别编程语言。

文件扩展名不一定可靠。

例如 `.h` 可能是 C，也可能是 C++。

`.m` 可能是 Objective-C，也可能是 MATLAB。

有些文件没有扩展名，例如 `Dockerfile`、`Makefile`。

常见做法是结合：

1. file extension。
2. shebang。
3. syntax parser。
4. language classifier。
5. repository metadata。

代码清洗还要保留格式。

对自然语言来说，多余空格影响不大。

但对 Python 代码，缩进就是语法。

所以不能随便 normalize whitespace。

对 Markdown 文档、代码块、注释也要谨慎处理。

### 四、去重：代码模型中特别重要

代码数据重复非常严重。

常见重复来源包括：

1. fork 仓库。
2. vendored dependencies。
3. 模板项目。
4. 自动生成代码。
5. 复制粘贴代码片段。
6. 同一库的多个版本。
7. benchmark 泄漏。

如果不去重，会有几个问题。

第一，浪费训练预算。

模型反复看相同代码，边际收益低。

第二，增加记忆风险。

模型更容易逐字输出训练代码。

第三，污染评估。

HumanEval、MBPP 或真实 benchmark 代码可能出现在训练数据中。

第四，影响数据配比。

热门库和模板可能被过度放大。

代码去重通常需要多层方法：

1. exact file hash。
2. normalized hash。
3. line-level dedup。
4. token-level n-gram dedup。
5. MinHash / SimHash。
6. repo-level fork detection。
7. benchmark contamination check。

Normalized hash 的意思是先去掉一些不重要差异，例如空行、注释或格式，再计算 hash。

但要谨慎，因为注释和格式对代码理解也可能有价值。

### 五、License、隐私和安全风险

代码数据有强烈的合规风险。

主要包括：

1. license 限制。
2. 版权风险。
3. 密钥和凭证泄漏。
4. 私有代码泄漏。
5. 恶意代码。
6. 漏洞模式学习。
7. 个人信息。

训练前需要识别 repository license。

不同团队的策略不同，有些只使用 permissive license，例如 MIT、Apache-2.0、BSD。

有些会排除 GPL 类 copyleft license。

有些还会根据司法辖区和产品用途做更细策略。

密钥检测也很重要。

代码仓库中可能包含：

1. API key。
2. access token。
3. private key。
4. password。
5. database URL。
6. cloud credential。

这些必须过滤或脱敏。

安全方面，模型可能学习到不安全代码模式。

例如 SQL injection、命令注入、硬编码密码、不安全反序列化。

因此训练和后训练都需要安全数据与安全评估。

### 六、代码 tokenizer 的特殊要求

代码 tokenizer 要同时服务自然语言和编程语言。

它需要有效表示：

1. 标识符。
2. 缩进和空白。
3. 操作符。
4. 括号。
5. 字符串。
6. 数字。
7. 注释。
8. 多语言关键字。

代码中很多标识符是组合词，例如：

```text
get_user_profile
HTTPRequestHandler
maxSequenceLength
```

如果 tokenizer 处理不好，标识符会被切得很碎，模型学习变量命名和 API 使用会更困难。

代码 tokenizer 常见关注点：

1. 是否保留换行和缩进。
2. 是否高效处理 snake_case 和 camelCase。
3. 是否覆盖常见操作符。
4. 是否支持多种编程语言。
5. 是否兼顾自然语言文档。
6. vocab size 是否过大。

对于 Python，缩进和换行尤其重要。

对于 C/C++/Java，括号、分号、类型声明很重要。

对于 HTML/JS/CSS，符号密集且常有嵌套结构。

所以代码 tokenizer 不只是文本 tokenizer 的附属品。

### 七、数据配比：代码、文本、数学如何混合

训练代码模型时，一个关键问题是代码数据占比多少。

如果只训练代码，模型可能代码能力强，但自然语言理解、指令遵循和解释能力弱。

如果代码占比太低，模型生成代码能力不足。

常见混合数据包括：

1. 纯代码文件。
2. README 和文档。
3. docstring 和注释。
4. issue 和 PR。
5. Stack Overflow 风格问答。
6. 数学和逻辑题。
7. 通用自然语言。
8. 代码指令数据。

不同目标对应不同配比。

如果是通用 LLM 增强代码能力，可以加入一定比例高质量代码数据。

如果是专门代码模型，代码比例会更高，并加入更多 repo-level 数据、测试和任务数据。

如果目标是 coding agent，还要加入：

1. 修改已有代码。
2. 读取错误日志。
3. 根据测试修复。
4. 多文件编辑。
5. 工具调用轨迹。

训练代码模型时还要注意编程语言配比。

Python、JavaScript、Java、C++、Go、Rust、SQL、Shell 等语言数据量和需求不同。

按原始 GitHub 量采样不一定符合产品目标。

例如面向算法面试和 ML 工程，Python、C++、SQL、Shell、CUDA 可能更重要。

面向前端开发，JavaScript、TypeScript、HTML、CSS 更重要。

### 八、文件级、仓库级和任务级训练

代码模型预训练有不同粒度。

第一，file-level。

把单个代码文件作为训练样本。

优点是简单。

缺点是缺少跨文件依赖。

第二，repo-level。

把同一仓库中的多个文件按结构组织起来训练。

这能帮助模型学习：

1. import 关系。
2. API 定义和调用。
3. 测试与实现的对应。
4. 配置文件和代码关系。
5. README 与实现关系。

第三，task-level。

构造具体任务，例如：

1. 根据 docstring 补全函数。
2. 根据测试生成代码。
3. 根据错误日志修复 bug。
4. 根据 issue 修改 repo。
5. 补全中间代码片段。

早期代码模型更多是 file-level completion。

现代代码助手和 coding agent 更需要 repo-level 和 task-level 数据。

因为真实开发不是从空白文件写函数，而是在已有代码库里理解上下文并修改。

### 九、代码预训练目标

最基本的目标仍然是 next-token prediction。

给定前面的代码 token，预测下一个 token。

这可以学到语法、API、常见模式。

但代码任务还有其他目标。

第一，Fill-in-the-Middle，简称 FIM。

很多代码补全场景不是只在文件末尾续写，而是在中间插入代码。

FIM 会把文本拆成 prefix、middle、suffix，让模型根据前后文补中间。

第二，span corruption。

遮住一段代码，让模型恢复。

第三，docstring-to-code。

根据文档生成函数。

第四，code-to-docstring。

根据代码生成解释。

第五，test-to-code。

根据单元测试生成实现。

第六，bug-fix objective。

输入错误代码和报错信息，输出修复代码。

这些目标可以在预训练、继续预训练或后训练中加入。

对于代码助手，FIM 特别重要。

因为 IDE 中用户经常在已有文件中间补代码。

### 十、代码模型评估：HumanEval 不够

HumanEval 很常见，但它只是一个小型函数级 benchmark。

它主要评估根据 docstring 写 Python 函数，并用单元测试判断通过率。

它有价值，但远远不够。

原因是：

1. 样本数量有限。
2. 主要是 Python。
3. 多为独立函数。
4. 不覆盖 repo-level 修改。
5. 不覆盖真实依赖和环境问题。
6. 可能存在污染。

更完整的代码评估应包括：

1. HumanEval、MBPP 等函数级生成。
2. 多语言代码生成。
3. 代码补全。
4. FIM 补全。
5. bug 修复。
6. 单元测试生成。
7. repo-level issue fixing。
8. 编译通过率。
9. 测试通过率。
10. 安全漏洞评估。
11. 代码解释和重构。
12. 真实开发任务。

Pass@k 是代码评估中常见指标。

它表示生成 `k` 个候选中至少一个通过测试的概率。

如果 `pass@1` 高，说明模型一次生成就可靠。

如果 `pass@k` 高但 `pass@1` 低，说明模型有能力但需要采样和筛选。

在 coding agent 中，还要看：

1. 能否定位文件。
2. 能否理解测试失败。
3. 能否正确编辑多文件。
4. 能否运行测试并迭代修复。
5. 最终 patch 是否最小且正确。

这已经超出单纯语言模型评估。

### 十一、执行反馈和 verifier

代码模型有一个优势：代码可以执行。

这让我们可以用 verifier 或 execution feedback 提升质量。

例如：

1. 生成多个候选。
2. 运行单元测试。
3. 选择通过测试的候选。
4. 如果失败，把错误信息反馈给模型修复。

这种方式比纯文本任务更客观。

训练中也可以利用执行反馈：

1. 过滤不通过测试的合成代码。
2. 构造 bug-fix 数据。
3. 训练 verifier 判断代码正确性。
4. 做 rejection sampling。
5. 做 RL 或偏好优化。

但执行反馈也有局限。

测试通过不代表代码完全正确。

测试可能不完整。

代码可能通过测试但风格差、效率差或有安全风险。

所以代码评估要结合测试、静态分析、人类 review 和安全扫描。

### 十二、代码模型的安全问题

代码模型可能生成不安全代码。

常见风险包括：

1. SQL injection。
2. XSS。
3. 命令注入。
4. 硬编码密钥。
5. 不安全反序列化。
6. 弱密码算法。
7. 路径遍历。
8. SSRF。
9. 资源泄漏。
10. 恶意代码生成。

因此代码模型训练需要加入安全数据和安全评估。

例如：

1. 安全代码示例。
2. 漏洞修复数据。
3. 不安全代码识别。
4. 安全建议生成。
5. 拒绝恶意请求。
6. 静态分析工具反馈。

对 coding agent 来说，还要限制工具权限。

模型不应该随意删除文件、上传密钥、执行危险命令。

代码能力越强，安全边界越重要。

### 十三、合成代码数据

合成数据在代码训练中很有价值。

可以用强模型生成：

1. 编程题。
2. 单元测试。
3. 多语言实现。
4. bug-fix pair。
5. 代码解释。
6. API 使用示例。
7. repo-level 任务。

代码合成数据的优势是可验证。

例如生成 Python 函数和测试，可以运行测试过滤。

生成 SQL 查询，可以在数据库样例上执行。

生成修复 patch，可以运行原测试。

但合成代码数据也有风险：

1. 题目模板化。
2. 解法单一。
3. 测试覆盖不足。
4. teacher model 错误被学生学习。
5. 代码风格单调。
6. 过度拟合 synthetic distribution。

所以合成代码数据要配合：

1. execution filtering。
2. dedup。
3. 多样化生成。
4. 难度分层。
5. 人工抽检。
6. 与真实代码数据混合。

### 十四、面试回答模板

如果面试官问：

> 如何训练一个代码大模型？

可以这样回答：

我会先明确目标，是通用 LLM 增强代码能力、专门代码补全模型，还是 coding agent。数据上会收集开源仓库、官方文档、README、docstring、issue、PR、commit message、测试和技术问答，并保留 repository 结构。清洗时要过滤空文件、超大文件、minified 文件、生成代码、vendor 依赖、重复 fork、密钥和不合规 license。代码去重特别重要，需要 file hash、normalized hash、MinHash、repo-level fork 检测和 benchmark contamination 检查。

Tokenizer 要能高效处理标识符、缩进、操作符、括号、注释和多语言代码。数据配比上要平衡代码、自然语言、数学、文档和指令数据，并根据目标调整编程语言比例。训练目标除了 next-token prediction，还可以加入 FIM、docstring-to-code、test-to-code、bug-fix 等目标。对于 repo-level 和 agent 场景，还需要多文件上下文、测试失败日志和 patch 数据。

评估不能只看 HumanEval，还要看 MBPP、多语言代码生成、FIM、补全、bug 修复、单元测试生成、repo-level issue fixing、编译通过率、测试通过率和安全漏洞评估。代码可以执行，所以可以用 execution feedback、verifier 和 rejection sampling 提升质量，但测试通过不代表完全正确，还需要安全和人类 review。

### 十五、常见追问与回答

问题 1：代码模型和通用语言模型训练最大的区别是什么？

回答：代码有严格语法、长程依赖和可执行反馈。训练时要处理 repo 结构、测试、依赖、license、安全和去重。评估也不能只看文本相似度，而要看编译和测试是否通过。

问题 2：为什么代码数据去重特别重要？

回答：开源代码中 fork、模板、vendor、生成代码和复制粘贴很多。不去重会浪费训练预算、增加记忆风险，并污染代码 benchmark。

问题 3：什么是 FIM，为什么代码模型需要它？

回答：FIM 是 fill-in-the-middle，让模型根据前后文补中间代码。IDE 场景中用户经常在已有文件中间编辑，不只是从末尾续写，所以 FIM 对代码补全很重要。

问题 4：HumanEval 能否代表代码能力？

回答：不能完全代表。它主要是 Python 函数级生成，样本少，不覆盖 repo-level 修改、多文件依赖、真实环境、bug 修复和安全问题。需要更完整的代码评估体系。

问题 5：如何利用代码可执行性提升模型？

回答：可以生成多个候选并运行测试选择，通过 execution filtering 过滤合成数据，训练 verifier，构造 bug-fix 数据，或者用测试反馈进行迭代修复和偏好优化。

问题 6：代码模型有哪些安全风险？

回答：可能生成 SQL injection、XSS、命令注入、硬编码密钥、不安全反序列化、弱加密和恶意代码。需要安全训练数据、静态分析、红队评估和工具权限控制。

### 十六、常见误区

1. 误区：代码模型就是在 GitHub 上多训一点。
   纠正：还需要清洗、去重、license 控制、repo 结构、测试、FIM、评估和安全。

2. 误区：HumanEval 高就说明代码模型强。
   纠正：HumanEval 只是小型函数级 Python benchmark，不能覆盖真实软件工程任务。

3. 误区：测试通过就说明代码完全正确。
   纠正：测试可能不完整，代码还可能有性能、风格、边界条件和安全问题。

4. 误区：代码 tokenizer 不重要。
   纠正：标识符、缩进、符号和操作符切分会影响代码理解和生成效率。

5. 误区：只训练代码就能得到好的代码助手。
   纠正：代码助手还需要自然语言理解、文档、指令遵循、错误日志理解和多轮交互能力。

6. 误区：代码模型越会写代码越安全。
   纠正：能力增强也可能增强生成不安全或恶意代码的能力，必须单独做安全对齐。

### 本讲总结

本讲最重要的结论：

1. 代码模型预训练不只是加入代码数据，而是代码数据工程、repo 结构建模、训练目标、执行反馈和安全评估的系统工程。
2. 代码数据清洗要处理语言识别、重复、生成代码、vendor 依赖、密钥、license 和恶意代码。
3. 代码去重非常关键，因为 fork、模板和 benchmark 污染会严重影响训练和评估。
4. 代码 tokenizer 要高效处理标识符、缩进、符号、注释和多种编程语言。
5. 训练目标除了 next-token prediction，还可以包括 FIM、docstring-to-code、test-to-code 和 bug-fix。
6. 代码评估不能只看 HumanEval，需要函数级、补全、FIM、repo-level、测试通过率、安全和真实开发任务评估。
7. 代码可执行性提供了强反馈信号，可以用于 verifier、rejection sampling、合成数据过滤和 agent 迭代修复。
8. 代码能力越强，越需要安全边界、合规控制和工具权限设计。
