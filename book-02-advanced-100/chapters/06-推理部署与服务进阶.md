# 第六部分：推理、部署与服务进阶

## 第 66 讲：Prefill 与 Decode 阶段性能模型

### 本讲目标

学完本讲，你应该能回答五个问题：

1. 大模型推理为什么要拆成 Prefill 和 Decode 两个阶段。
2. Prefill 和 Decode 的计算、显存、访存瓶颈分别是什么。
3. TTFT、TPOT、吞吐、并发、batch size 之间有什么关系。
4. 为什么长 prompt 和长 output 对服务性能的影响不同。
5. 面试中如何建立一个简单的 LLM inference 性能模型。

从这一讲开始，我们进入第二册第六部分：推理、部署与服务进阶。

训练模型只是第一步。

真正上线时，问题会变成：

```text
用户请求来了，模型多久能开始回答？
每秒能生成多少 token？
一张 GPU 能服务多少并发？
长上下文为什么这么贵？
为什么 batch 大了吞吐上升但延迟也上升？
```

这些问题都绕不开两个阶段：

```text
Prefill：处理输入 prompt，建立 KV cache。
Decode：逐 token 生成输出，复用 KV cache。
```

理解这两个阶段，是理解 vLLM、TensorRT-LLM、TGI、SGLang、PagedAttention、Continuous Batching、Speculative Decoding 的基础。

---

### 一、一次大模型推理发生了什么

假设用户输入：

```text
请解释 Transformer 的 attention 机制。
```

模型要生成：

```text
Attention 的核心思想是...
```

推理过程可以分成两步。

第一步，处理完整 prompt。

这一步模型一次性读入所有输入 token，计算每一层 hidden state，并为每一层保存 key/value。

这叫 Prefill。

第二步，逐 token 生成回答。

每生成一个新 token，模型只输入上一个 token，但 attention 会访问之前所有 token 的 KV cache。

这叫 Decode。

简化流程：

```text
输入 prompt tokens
-> Prefill 计算 prompt 的 hidden states 和 KV cache
-> 采样第一个输出 token
-> Decode step 1，生成第二个输出 token
-> Decode step 2，生成第三个输出 token
-> ...
-> 直到 stop token 或长度上限
```

这两个阶段的性能特征完全不同。

---

### 二、Prefill 阶段是什么

Prefill 也叫 prompt processing。

输入长度为 `S`，batch size 为 `B`。

Prefill 阶段处理的 token 数是：

```text
B * S
```

它会对 prompt 中所有 token 做完整 Transformer 前向。

在 self-attention 中，每个 token 可以看见之前的 token。

对于一个长度为 `S` 的序列，attention 计算大致包含：

```text
QK^T：S * S
Attention * V：S * S
```

所以 attention 部分随序列长度有二次项。

但在大模型里，MLP 和 projection 也很重。

粗略理解：

```text
Prefill 更像一个大矩阵批处理任务，GPU 利用率通常较高。
```

它的特点：

1. 输入 token 多。
2. 矩阵乘法规模大。
3. 并行度高。
4. 容易吃满 GPU 算力。
5. prompt 越长，Prefill 越贵。
6. 会产生 KV cache，为后续 Decode 使用。

Prefill 主要影响一个用户指标：

```text
TTFT：Time To First Token，首 token 延迟。
```

用户感受到的是：

```text
我发出问题后，模型多久开始吐第一个字。
```

长 prompt 会显著增加 TTFT。

---

### 三、Decode 阶段是什么

Decode 是生成阶段。

模型每次生成一个 token。

假设已经有 `S` 个 prompt token，已经生成了 `t` 个 output token。

第 `t+1` 个 token 的 decode step 会：

1. 输入上一个 token。
2. 计算当前 token 的 Q/K/V。
3. 把新的 K/V 追加到 KV cache。
4. 当前 Q attend 到所有历史 K/V。
5. 得到 logits。
6. 采样下一个 token。

每个 decode step 只处理一个新 token，但要读历史 KV cache。

所以 Decode 的特点是：

1. 单步矩阵乘法小。
2. batch 不够大时 GPU 利用率低。
3. 每步都要访问 KV cache。
4. 对显存带宽和 KV cache 读写敏感。
5. output 越长，Decode step 越多。
6. 每个请求生成长度不同，调度更复杂。

Decode 主要影响两个指标：

```text
TPOT：Time Per Output Token，每个输出 token 的耗时。
Tokens/s：每秒生成 token 数。
```

用户感受到的是：

```text
模型开始回答后，吐字速度快不快。
```

长 output 会显著增加总生成时间。

---

### 四、Prefill 和 Decode 的核心区别

可以用一张表总结：

```text
维度              Prefill                         Decode
输入              完整 prompt                      上一个生成 token
主要工作          建立 prompt KV cache             逐 token 生成并追加 KV
并行度            高                               低，依赖 batch 聚合
瓶颈              算力和长序列 attention             显存带宽、KV cache、调度
影响指标          TTFT                             TPOT、生成吞吐
序列长度影响       prompt 越长越慢                  历史上下文越长，每步越贵
输出长度影响       不直接影响                       输出越长 step 越多
```

一句话：

```text
Prefill 像一次大批量计算；Decode 像很多次小步迭代。
```

这也是为什么推理服务优化往往要分别优化两个阶段。

### 五、性能指标：TTFT、TPOT、吞吐

推理服务常见指标有四个。

#### 1. TTFT

Time To First Token。

从请求到达，到第一个输出 token 返回的时间。

组成包括：

```math
T_{\mathrm{first}}=T_{\mathrm{queue}}+T_{\mathrm{tok}}+T_{\mathrm{pre}}+T_{\mathrm{sample}}+T_{\mathrm{net}}
```

其中 `T_queue` 是排队时间，`T_tok` 是 tokenizer 和 chat template 处理时间，`T_pre` 是 prefill 时间，`T_sample` 是第一次采样时间，`T_net` 是网络和流式返回开销。面试中可以先用这个加法模型解释 TTFT，再说明线上会看 P50/P95/P99，而不是只看平均值。

TTFT 对聊天体验很关键。

用户通常能接受后续慢一点，但很难接受长时间没有任何响应。

#### 2. TPOT

Time Per Output Token。

生成阶段平均每个 token 的耗时。

```math
\mathrm{TPOT}=\frac{1}{O-1}\sum_{t=1}^{O-1}T_{\mathrm{dec}}(S+t)
```

这里 `S` 是输入长度，`O` 是输出 token 数。上式把第一个输出 token 计入 TTFT，因此后续流式阶段通常统计剩余 `O-1` 个 token 的平均间隔。实际系统里也会直接统计 inter-token latency 的分位数。

TPOT 决定流式输出速度。

#### 3. Latency

端到端延迟。

```math
T_{\mathrm{e2e}}=T_{\mathrm{first}}+\sum_{t=1}^{O-1}T_{\mathrm{dec}}(S+t)+T_{\mathrm{tail}}
```

其中 `T_tail` 包括最后一次流式 flush、日志、计费、资源释放等尾部开销。如果粗略近似每个 decode step 时间相同，可以写成：

```math
T_{\mathrm{e2e}}\approx T_{\mathrm{first}}+(O-1)\cdot \mathrm{TPOT}+T_{\mathrm{tail}}
```

更准确地说，TPOT 会随上下文变长、batch 变化和 KV cache 读写压力变化，但这个公式足够用于面试解释。

#### 4. Throughput

吞吐通常用：

```text
tokens/s
requests/s
```

tokens/s 又可以分为：

```text
prefill tokens/s
decode tokens/s
total tokens/s
```

可以写成：

```math
R_{\mathrm{pre}}=\frac{N_{\mathrm{pre}}}{T_{\mathrm{pre}}},\qquad R_{\mathrm{dec}}=\frac{N_{\mathrm{dec}}}{T_{\mathrm{dec}}}
```

`N_pre` 是一段时间内处理的 input token 数，`N_dec` 是同一段时间内生成的 output token 数。两者不能直接混在一起比较，因为 prefill 和 decode 的资源瓶颈不同。

面试中一定要说明：

```text
只说 tokens/s 不够，要区分 prefill 和 decode，因为二者性能特征不同。
```

### 六、一个简单的端到端延迟模型

假设：

```text
输入长度 = S
输出长度 = O
请求排队时间 = T_queue
Prefill 时间 = T_pre(S)
第 t 个后续输出 token 的 decode 时间 = T_dec(S+t)
```

端到端延迟可以写成：

```math
T_{\mathrm{e2e}}=T_{\mathrm{queue}}+T_{\mathrm{tok}}+T_{\mathrm{pre}}(S)+T_{\mathrm{sample}}+\sum_{t=1}^{O-1}T_{\mathrm{dec}}(S+t)+T_{\mathrm{tail}}
```

如果粗略近似 decode 每步时间相同：

```math
T_{\mathrm{e2e}}\approx T_{\mathrm{first}}+(O-1)\cdot \mathrm{TPOT}+T_{\mathrm{tail}}
```

TTFT 近似为：

```math
T_{\mathrm{first}}\approx T_{\mathrm{queue}}+T_{\mathrm{tok}}+T_{\mathrm{pre}}(S)+T_{\mathrm{sample}}+T_{\mathrm{net}}
```

这解释了两个现象：

1. prompt 很长，首 token 慢。
2. answer 很长，总耗时长。

如果用户输入 20k token，但只生成 50 token：

```text
瓶颈主要是 Prefill。
```

如果用户输入 100 token，但生成 2000 token：

```text
瓶颈主要是 Decode。
```

如果输入和输出都很长，两者都贵。

### 七、计算量的粗略估算

对一个 decoder-only Transformer，参数量为 `P`。

很多推理估算会用一个粗略规则：

```math
F_{\mathrm{linear}}\approx 2P
```

也就是说，每处理一个 token，线性层前向大约需要 `2P` FLOPs 级别的计算。这是非常粗略的估算，主要来自矩阵乘法的 multiply-add，不包含 attention 二次项、采样、通信、kernel launch 和调度开销。

Prefill 处理 `B*S` 个 token，因此线性层计算大约是：

```math
F_{\mathrm{pre,linear}}\approx 2BSP
```

Decode 每步处理 `B` 个新 token，因此每步线性层计算大约是：

```math
F_{\mathrm{dec,linear}}\approx 2BP
```

但 attention 还要考虑上下文长度。

Prefill attention 大致有：

```math
F_{\mathrm{pre,attn}}=O(BLS^2H)
```

Decode attention 每步大致有：

```math
F_{\mathrm{dec,attn}}=O(BLS_tH)
```

其中：

```text
L = 层数
H = hidden size 或 attention 相关维度
S_t = 第 t 个 decode step 的当前上下文长度
```

面试时不需要精确到常数。

关键是说清：

```text
Prefill 对 prompt 长度更敏感，attention 有二次项；
Decode 每步对当前上下文长度敏感，且要重复 O 次。
```

### 八、KV Cache 的作用

如果没有 KV cache，Decode 每生成一个 token 都要重新计算所有历史 token 的 K/V。

那会非常浪费。

KV cache 的作用是：

```text
Prefill 时保存每层每个历史 token 的 K/V；
Decode 时复用历史 K/V，只计算新 token 的 K/V。
```

这样 Decode 从“重复计算完整上下文”变成“只追加新 token”。

但 KV cache 也带来显存开销。

KV cache 大小大致与以下因素成正比：

```math
M_{\mathrm{kv}}=2\cdot B\cdot S\cdot L\cdot H_{\mathrm{kv}}\cdot D_h\cdot b
```

其中 `B` 是活跃请求数，`S` 是每个请求的当前上下文长度，`L` 是层数，`H_kv` 是 KV head 数，`D_h` 是每个 head 的维度，`b` 是每个元素的字节数，前面的 `2` 表示同时保存 K 和 V。真实系统中还要考虑不同请求长度不一致、block 对齐、碎片、prefix cache 共享、临时 buffer 和框架开销。

所以：

1. batch 越大，KV cache 越大。
2. 上下文越长，KV cache 越大。
3. 层数越多，KV cache 越大。
4. FP16/BF16 比 FP8/INT8 KV cache 更占显存。
5. MQA/GQA 可以减少 KV heads，从而减少 KV cache。

这就是为什么长上下文服务贵。

不是只因为计算多，也因为 KV cache 占显存、占带宽。

下面这个 demo 用非常粗的吞吐参数估算三个请求的 TTFT、TPOT、端到端延迟和 KV cache 显存。它不模拟真实 GPU kernel，只帮助你建立“长 prompt 主要打 TTFT，长 output 主要打总延迟和持续占用”的直觉。

```python
from dataclasses import dataclass


@dataclass
class ModelConfig:
    layers: int
    kv_heads: int
    head_dim: int
    dtype_bytes: int
    prefill_tokens_per_s: float
    decode_tokens_per_s: float
    base_queue_s: float = 0.03
    tokenizer_s: float = 0.01
    sample_s: float = 0.005
    network_s: float = 0.005


def kv_cache_mib(tokens: int, cfg: ModelConfig) -> float:
    bytes_used = 2 * tokens * cfg.layers * cfg.kv_heads * cfg.head_dim * cfg.dtype_bytes
    return bytes_used / 1024**2


def estimate_request(name: str, prompt_tokens: int, output_tokens: int, cfg: ModelConfig) -> dict:
    prefill_s = prompt_tokens / cfg.prefill_tokens_per_s
    # 用平均上下文长度粗略反映 decode 越往后 KV 读得越长。
    avg_context = prompt_tokens + max(output_tokens - 1, 0) / 2
    context_factor = avg_context / 2048
    tpot_s = (1 / cfg.decode_tokens_per_s) * (1 + 0.08 * context_factor)
    decode_s = max(output_tokens - 1, 0) * tpot_s
    ttft_s = cfg.base_queue_s + cfg.tokenizer_s + prefill_s + cfg.sample_s + cfg.network_s
    total_s = ttft_s + decode_s
    final_tokens = prompt_tokens + output_tokens
    return {
        "name": name,
        "ttft_s": ttft_s,
        "tpot_ms": tpot_s * 1000,
        "total_s": total_s,
        "kv_cache_mib": kv_cache_mib(final_tokens, cfg),
    }


cfg = ModelConfig(
    layers=32,
    kv_heads=8,      # GQA 后的 KV heads，不是 query heads
    head_dim=128,
    dtype_bytes=2,   # FP16/BF16
    prefill_tokens_per_s=18000,
    decode_tokens_per_s=90,
)

requests = [
    ("short_chat", 256, 128),
    ("long_prompt", 16000, 80),
    ("long_output", 512, 3000),
]

for req in requests:
    row = estimate_request(*req, cfg)
    print(
        row["name"],
        "TTFT=", round(row["ttft_s"], 3), "s",
        "TPOT=", round(row["tpot_ms"], 2), "ms",
        "total=", round(row["total_s"], 2), "s",
        "KV=", round(row["kv_cache_mib"], 1), "MiB",
    )
```

典型输出类似：

```text
short_chat TTFT= 0.064 s TPOT= 11.25 ms total= 1.49 s KV= 48.0 MiB
long_prompt TTFT= 0.939 s TPOT= 18.07 ms total= 2.37 s KV= 2010.0 MiB
long_output TTFT= 0.078 s TPOT= 11.98 ms total= 36.02 s KV= 439.0 MiB
```

这里 `long_prompt` 的 TTFT 和初始 KV cache 明显更大，`long_output` 的首 token 不慢，但端到端耗时很长，因为 decode step 数太多。真实压测时，应把这个玩具模型换成实际采样的 input/output token 分布、P50/P95/P99 TTFT、P50/P95/P99 TPOT、prefix cache 命中率和 OOM 率。

### 九、Batch Size 与并发

推理服务中，batch size 和并发密切相关。

如果一次只服务一个请求，Decode 阶段每步只处理一个 token。

GPU 利用率很低。

如果把多个请求合成 batch，Decode 每步处理多个 token，吞吐上升。

但 batch 变大也有代价：

1. 排队时间增加。
2. KV cache 占用增加。
3. 不同请求长度不同，调度复杂。
4. 高并发下 TTFT 可能变差。
5. 长请求可能拖慢短请求。

所以服务系统要在延迟和吞吐之间权衡。

```text
低延迟：小 batch，少排队，GPU 利用率低，成本高。
高吞吐：大 batch，多排队，GPU 利用率高，延迟可能变差。
```

这也是 Continuous Batching 出现的原因。

它试图在 Decode 过程中动态加入和移除请求，提高 GPU 利用率。

第 68 讲会专门讲。

### 十、Prefill 和 Decode 的资源瓶颈

推理性能通常受两类资源限制：

```text
计算 FLOPs
显存带宽和容量
```

Prefill 通常更容易 compute-bound。

原因是它有大矩阵乘法，batch 和 sequence 维度较大，GPU 能吃满。

Decode 通常更容易 memory-bandwidth-bound。

原因是每步计算的新 token 少，但要读大量权重和 KV cache。

尤其 batch 小时，矩阵乘法规模小，算力利用不高。

可以用 roofline 的直觉解释：

```text
如果每读一字节数据能做很多计算，偏 compute-bound。
如果每读一字节数据只能做少量计算，偏 memory-bound。
```

Decode 的 arithmetic intensity 往往较低，因此容易受带宽限制。

### 十一、长 Prompt 和长 Output 的不同影响

长 prompt 主要影响：

1. Prefill 时间。
2. TTFT。
3. KV cache 初始大小。
4. Decode 每步 attention 读 KV 的长度。

长 output 主要影响：

1. Decode step 数量。
2. 总延迟。
3. KV cache 逐步增长。
4. GPU 长时间占用。

举例：

```text
请求 A：输入 16000 token，输出 100 token。
请求 B：输入 200 token，输出 4000 token。
```

A 的首 token 会慢，因为 Prefill 很重。

B 的首 token可能快，但总生成时间很长，因为 Decode step 很多。

这两个请求不能用同一个“平均 tokens/s”简单比较。

### 十二、工程优化方向

针对 Prefill，可以优化：

1. FlashAttention。
2. 高效矩阵乘 kernel。
3. Tensor Parallel。
4. Prompt caching。
5. Chunked prefill。
6. 长上下文稀疏注意力。
7. 更好的 tokenizer 和模板控制。

针对 Decode，可以优化：

1. KV cache 管理。
2. PagedAttention。
3. Continuous Batching。
4. Speculative Decoding。
5. MQA/GQA。
6. KV cache 量化。
7. 权重量化。
8. 多 token 预测。

针对服务整体，可以优化：

1. 请求调度。
2. 动态 batch。
3. 优先级队列。
4. 预估输出长度。
5. prefix cache。
6. 多副本负载均衡。
7. SLA 分层。

### 十三、面试问答

#### 问题 1：Prefill 和 Decode 的区别是什么？

可以这样回答：

```text
Prefill 是处理完整 prompt 的阶段，会计算 prompt token 的 hidden state 并建立 KV cache，主要影响 TTFT；
Decode 是逐 token 生成阶段，每步只输入新 token，但要读取历史 KV cache，主要影响 TPOT 和生成吞吐。
Prefill 并行度高，更像大矩阵批处理；Decode 单步小、迭代多，通常更受显存带宽和调度影响。
```

#### 问题 2：为什么长 prompt 会让首 token 慢？

可以这样回答：

```text
因为模型必须先对完整 prompt 做 Prefill，计算所有输入 token 的前向并构建每层 KV cache。
prompt 越长，Prefill 计算和 attention 开销越大，KV cache 初始化也越大，所以 TTFT 上升。
```

#### 问题 3：为什么 Decode 阶段 GPU 利用率容易低？

可以这样回答：

```text
Decode 每步只处理每个请求的一个新 token，矩阵乘法规模小。
如果 batch 不够大，GPU 算力吃不满，同时还要频繁读取权重和 KV cache，所以容易 memory-bandwidth-bound。
```

#### 问题 4：如何估算端到端延迟？

可以这样回答：

```text
可以拆成排队时间、Prefill 时间和 Decode 时间：
T_e2e = T_queue + T_tok + T_pre(input_len) + T_sample + sum T_dec(current_context_len) + T_tail。
粗略时可写成 T_e2e 约等于 TTFT 加上后续输出 token 数乘以 TPOT。
```

#### 问题 5：吞吐和延迟为什么冲突？

可以这样回答：

```text
增大 batch 可以提高 GPU 利用率和 tokens/s，但请求需要等待凑 batch，KV cache 占用也增加，可能导致 TTFT 和 tail latency 变差。
服务系统要在成本、吞吐和 SLA 延迟之间做权衡。
```

#### 问题 6：为什么要区分 prefill tokens/s 和 decode tokens/s？

可以这样回答：

```text
因为 Prefill 和 Decode 的计算模式不同。Prefill 并行度高、偏计算密集，Decode 逐 token 迭代、偏带宽和调度瓶颈。
一个系统 prefill 很快不代表 decode 吐字快，反之亦然。
```

---

### 十四、常见误区

1. 误区：推理性能只看 tokens/s。
   纠正：要区分 TTFT、TPOT、prefill throughput、decode throughput、tail latency。

2. 误区：长上下文只影响 Prefill。
   纠正：长上下文还会增大 KV cache，使每步 Decode 读取更多历史 KV。

3. 误区：batch 越大越好。
   纠正：batch 大提升吞吐，但增加排队、显存和尾延迟。

4. 误区：Decode 每步只处理一个 token，所以很便宜。
   纠正：每步小但要重复很多次，并且要读权重和 KV cache。

5. 误区：输出短的请求一定便宜。
   纠正：如果 prompt 很长，Prefill 仍然很贵。

6. 误区：Prefill 和 Decode 可以用同一种优化思路解决。
   纠正：Prefill 更偏大矩阵和长序列 attention，Decode 更偏 KV cache、调度和带宽。

---

### 本讲总结

本讲最重要的结论：

1. LLM 推理分为 Prefill 和 Decode 两个阶段。
2. Prefill 处理完整 prompt，建立 KV cache，主要影响 TTFT。
3. Decode 逐 token 生成，复用和追加 KV cache，主要影响 TPOT 和生成吞吐。
4. Prefill 并行度高，常偏 compute-bound；Decode 单步小、迭代多，常偏 memory-bandwidth-bound。
5. 端到端延迟可以粗略写成 `T_e2e 约等于 TTFT + 后续输出 token 数 * TPOT + tail overhead`。
6. 长 prompt 主要增加 TTFT 和 KV cache，长 output 主要增加 Decode step 数和总延迟。
7. batch size 提升吞吐但可能增加排队、显存占用和尾延迟。
8. 后续推理优化技术基本都围绕 Prefill、Decode、KV cache 和调度展开。

## 第 67 讲：KV Cache 管理与分页注意力

### 本讲目标

学完本讲，你应该能回答六个问题：

1. KV Cache 为什么会成为大模型推理服务的核心显存瓶颈。
2. 传统连续显存分配在多请求、变长生成场景下有什么问题。
3. PagedAttention 借鉴了操作系统分页的什么思想。
4. block/page 级 KV Cache 管理如何降低碎片、提升并发。
5. PagedAttention 和 Continuous Batching、prefix cache、beam search 有什么关系。
6. 面试中如何讲清楚 vLLM 的 PagedAttention 解决了什么工程问题。

上一讲我们讲了 Prefill 和 Decode 的性能模型。

核心结论是：

```text
Prefill 主要影响 TTFT，Decode 主要影响 TPOT 和生成吞吐。
```

但真正做推理服务时，很快会遇到另一个问题：

```text
不是 GPU 算力先不够，而是 KV Cache 显存先不够。
```

尤其在高并发、长上下文、长输出场景下，KV Cache 可能比权重更难管理。

PagedAttention 的出现，就是为了解决这个问题。

---

### 一、为什么 KV Cache 是推理服务的核心状态

自回归生成时，模型每生成一个 token，都需要访问历史上下文。

如果没有 KV Cache，第 `t` 步生成时要重新计算前面所有 token 的 K/V。

这样会导致大量重复计算。

KV Cache 的做法是：

```text
Prefill：为 prompt 的每个 token 计算并保存每层 K/V。
Decode：每步只计算新 token 的 K/V，并把它追加到 cache 中。
```

所以 KV Cache 本质上是推理过程中的“会话状态”。

它不是模型参数，而是每个请求自己的运行时状态。

对于一个请求，KV Cache 大小大致为：

```math
M_{\mathrm{kv,one}}=2\cdot L\cdot S\cdot H_{\mathrm{kv}}\cdot D_h\cdot b
```

其中 `L` 是层数，`S` 是当前上下文 token 数，`H_kv` 是 KV head 数，`D_h` 是 head dim，`b` 是每个元素的字节数，前面的 `2` 表示 K 和 V。

如果有 batch 或多个并发请求，可以粗略写成：

```math
M_{\mathrm{kv,total}}=2\cdot L\cdot H_{\mathrm{kv}}\cdot D_h\cdot b\cdot \sum_{i=1}^{B}S_i
```

这里 `S_i` 是第 `i` 个活跃请求的当前上下文长度。这个式子强调：同一模型下，KV cache 压力主要跟所有活跃 token 总数走，而不是只跟 request 数走。

这说明一个关键事实：

```text
KV Cache 显存不是只由模型大小决定，而是由并发数、上下文长度、输出长度和 KV 头数共同决定。
```

---

### 二、一个简单的 KV Cache 显存估算

假设一个模型：

```text
layer_num = 32
kv_head_num = 32
head_dim = 128
dtype = FP16，即 2 bytes
```

单个 token 的 KV Cache 大小约为：

```math
m_{\mathrm{token}}=2\cdot 32\cdot 32\cdot 128\cdot 2=524288\ \mathrm{bytes}
```

换算后大约是 `512 KB`。

这意味着：

```text
1 个 token 的 KV Cache 大约 512 KB。
```

如果一个请求有 4096 token：

```math
M_{\mathrm{one}}=4096\cdot 512\ \mathrm{KB}\approx 2\ \mathrm{GB}
```

如果同时有 16 个类似请求：

```math
M_{\mathrm{total}}=16\cdot 2\ \mathrm{GB}\approx 32\ \mathrm{GB}
```

这还只是 KV Cache，不包括模型权重、激活临时 buffer、CUDA workspace、通信 buffer 和框架开销。

所以部署时常见现象是：

```text
模型权重能放下，但并发一高就 OOM。
```

这就是 KV Cache 管理重要的原因。

---

### 三、传统连续分配的问题

最直观的 KV Cache 管理方式是为每个请求预留一段连续显存。

例如：

```text
请求 A：最多生成 2048 token，预留 2048 个 token 的 KV 空间
请求 B：最多生成 2048 token，预留 2048 个 token 的 KV 空间
请求 C：最多生成 2048 token，预留 2048 个 token 的 KV 空间
```

这样实现简单，但有三个问题。

第一个问题是内部碎片。

如果请求 A 实际只生成 200 token，但系统预留了 2048 token 空间，那么剩下空间在请求结束前不能给别人用。

这叫内部碎片。

第二个问题是外部碎片。

当很多请求陆续开始和结束，显存里会出现很多不连续的小空洞。

即使总空闲显存足够，也可能找不到一段足够大的连续空间给新请求。

这叫外部碎片。

第三个问题是变长生成难管理。

LLM 请求的输入长度和输出长度差异很大：

```text
有的请求输入 100 token，输出 50 token。
有的请求输入 16000 token，输出 200 token。
有的请求输入 500 token，输出 4000 token。
```

如果每个请求都按最大长度预留，会浪费显存。

如果预留太少，又需要扩容和搬迁。

搬迁 KV Cache 代价很高，因为它包含所有层、所有历史 token 的 K/V。

---

### 四、PagedAttention 的核心直觉

PagedAttention 借鉴了操作系统的分页思想。

操作系统不会要求一个进程的虚拟内存在物理内存中完全连续。

它会把内存拆成固定大小的 page，然后通过页表把虚拟地址映射到物理 page。

PagedAttention 做了类似事情。

它不要求一个请求的 KV Cache 在显存里连续存放。

它把 KV Cache 切成固定大小的 block，也可以类比成 page。

```text
逻辑上：请求 A 有连续的 token 序列。
物理上：这些 token 的 KV blocks 可以分散在显存不同位置。
```

系统维护一张映射表：

```text
request_id + logical_block_id -> physical_block_id
```

Decode 时 attention kernel 根据映射表找到对应历史 token 的 K/V block。

一句话：

```text
PagedAttention 把“请求的连续上下文”映射到“不一定连续的物理 KV blocks”。
```

---

### 五、block 级 KV Cache 管理

假设 block size 是 16 token。

一个长度为 50 token 的请求，需要：

```math
n_{\mathrm{block}}=\left\lceil\frac{50}{16}\right\rceil=4
```

其中前 3 个 block 满，最后 1 个 block 只用 2 个 token。

浪费最多发生在最后一个 block。

所以内部碎片上界大致是：

```math
0\leq w_i=n_i C-S_i\leq C-1
```

这里 `C` 是 block size，`S_i` 是第 `i` 个请求实际占用的 token 数，`n_i` 是它分到的 block 数，`w_i` 是最后一个 block 里的浪费 token 数。

如果看一批请求的整体碎片率，可以写成：

```math
r_{\mathrm{waste}}=\frac{\sum_i(n_i C-S_i)}{\sum_i n_i C}
```

相比按最大长度预留，浪费大幅下降。

当请求继续生成 token 时：

1. 如果当前 block 还没满，就继续写入当前 block。
2. 如果当前 block 满了，就从空闲 block 池中分配一个新 block。
3. 请求结束后，把它占用的 block 归还给空闲池。

这个过程类似操作系统管理物理页。

好处是：

```text
显存以 block 为单位复用，不要求大段连续空间。
```

这可以显著减少碎片，并提高同一张 GPU 上可承载的并发请求数。

---

### 六、PagedAttention 如何服务 attention 计算

普通 attention 会认为历史 K/V 是连续张量。

例如某一层的 K cache 逻辑上可以看成：

```text
[seq_len, kv_heads, head_dim]
```

但 PagedAttention 下，这个逻辑序列被拆到多个物理 block 中。

所以 attention kernel 不能再简单地按连续地址读取。

它需要：

1. 根据请求找到 block table。
2. 根据 token 位置计算 logical block id 和 block 内 offset。
3. 通过 block table 找到 physical block id。
4. 从对应物理 block 读取 K/V。
5. 完成当前 query 对历史 K/V 的 attention。

可以用伪代码理解：

```python
def locate_kv(block_table, token_pos, block_size):
    logical_block_id = token_pos // block_size
    offset = token_pos % block_size
    physical_block_id = block_table[logical_block_id]
    return physical_block_id, offset
```

下面这个纯 Python demo 把三件事放在一起看：连续预留会浪费多少 token 空间、paged allocation 的最后 block 碎片是多少、共享前缀能少占多少物理 block。

```python
from math import ceil


BLOCK_SIZE = 16
MAX_RESERVED_TOKENS = 2048

requests = {
    "chat_short": 220,
    "rag_long_prompt": 1370,
    "code_long_output": 2410,
}


def blocks_needed(tokens: int, block_size: int = BLOCK_SIZE) -> int:
    return ceil(tokens / block_size)


def paged_stats(lengths: dict[str, int]) -> dict:
    used_tokens = sum(lengths.values())
    allocated_tokens = sum(blocks_needed(n) * BLOCK_SIZE for n in lengths.values())
    return {
        "used_tokens": used_tokens,
        "allocated_tokens": allocated_tokens,
        "wasted_tokens": allocated_tokens - used_tokens,
        "waste_ratio": (allocated_tokens - used_tokens) / allocated_tokens,
    }


def contiguous_stats(lengths: dict[str, int]) -> dict:
    used_tokens = sum(lengths.values())
    allocated_tokens = len(lengths) * MAX_RESERVED_TOKENS
    return {
        "used_tokens": used_tokens,
        "allocated_tokens": allocated_tokens,
        "wasted_tokens": allocated_tokens - used_tokens,
        "waste_ratio": (allocated_tokens - used_tokens) / allocated_tokens,
    }


def allocate_blocks(lengths: dict[str, int]) -> dict[str, list[int]]:
    next_block_id = 0
    tables = {}
    for request_id, tokens in lengths.items():
        count = blocks_needed(tokens)
        tables[request_id] = list(range(next_block_id, next_block_id + count))
        next_block_id += count
    return tables


def locate_kv(block_table: list[int], token_pos: int) -> tuple[int, int]:
    logical_block_id = token_pos // BLOCK_SIZE
    offset = token_pos % BLOCK_SIZE
    return block_table[logical_block_id], offset


def prefix_sharing_blocks(prefix_tokens: int, suffix_lengths: list[int]) -> tuple[int, int]:
    no_share = sum(blocks_needed(prefix_tokens + s) for s in suffix_lengths)
    with_share = blocks_needed(prefix_tokens) + sum(blocks_needed(s) for s in suffix_lengths)
    return no_share, with_share


for name, stats in [
    ("contiguous", contiguous_stats(requests)),
    ("paged", paged_stats(requests)),
]:
    print(name, stats)

tables = allocate_blocks(requests)
print("block_table", tables["rag_long_prompt"][:5], "...")
print("token_37_location", locate_kv(tables["rag_long_prompt"], 37))

no_share, with_share = prefix_sharing_blocks(prefix_tokens=512, suffix_lengths=[64, 96, 128])
print("prefix_blocks_no_share=", no_share)
print("prefix_blocks_with_share=", with_share)
print("prefix_blocks_saved=", no_share - with_share)
```

典型输出如下：

```text
contiguous {'used_tokens': 4000, 'allocated_tokens': 6144, 'wasted_tokens': 2144, 'waste_ratio': 0.3489583333333333}
paged {'used_tokens': 4000, 'allocated_tokens': 4016, 'wasted_tokens': 16, 'waste_ratio': 0.00398406374501992}
block_table [14, 15, 16, 17, 18] ...
token_37_location (16, 5)
prefix_blocks_no_share= 114
prefix_blocks_with_share= 50
prefix_blocks_saved= 64
```

这个例子故意使用很小的 block size，方便你手算。真实系统还要把 token 空间乘以每个 token 的 KV bytes，并考虑 block table、引用计数、CUDA kernel 访存、prefix cache hash 和异常释放。

真实实现会在 CUDA kernel 里做高效访存、向量化读取和调度优化。

面试时不需要写 kernel。

你需要讲清楚：

```text
PagedAttention 的难点不只是分配 block，还要让 attention kernel 能高效读取非连续 KV blocks。
```

---

### 七、为什么它能提升并发和吞吐

推理服务中，GPU 显存通常同时放三类东西：

1. 模型权重。
2. 计算临时 buffer。
3. 所有活跃请求的 KV Cache。

权重基本固定。

临时 buffer 会随 batch 和 kernel 变化。

KV Cache 则随请求动态增长和释放。

如果 KV Cache 管理低效，很多显存会浪费在预留空间和碎片里。

PagedAttention 通过 block 级管理带来三个收益：

1. 降低内部碎片，不再为每个请求预留最大长度。
2. 降低外部碎片，不要求每个请求占用连续大块显存。
3. 更容易支持动态加入、退出和变长生成请求。

这些收益会转化成：

```text
同样显存下可以容纳更多活跃 token。
更多活跃 token 意味着可以支持更高并发或更长上下文。
更高并发有助于提高 decode 阶段 GPU 利用率和吞吐。
```

注意这里不是说 PagedAttention 让每一次 attention 的数学计算量变少。

它主要降低的是 KV Cache 显存浪费和服务调度成本。

---

### 八、PagedAttention 与 Continuous Batching 的关系

Continuous Batching 是第 68 讲的主题。

这里先建立联系。

传统 batching 可能是静态的：

```text
凑一批请求 -> 一起跑到结束 -> 再处理下一批
```

但 LLM 请求输出长度不同。

有的请求很快结束，有的请求还要生成很久。

如果一直等同一 batch 全部结束，会浪费 GPU。

Continuous Batching 的思想是：

```text
每个 decode step 后，完成的请求退出，新请求可以加入。
```

这要求 KV Cache 能动态分配和释放。

PagedAttention 正好提供了这样的基础设施。

请求加入时分配 block。

请求生成时追加 block。

请求结束时释放 block。

所以可以这样理解：

```text
PagedAttention 解决 KV Cache 内存管理问题，Continuous Batching 解决请求调度问题。
两者结合，才能高效服务变长、高并发的 LLM 请求。
```

---

### 九、PagedAttention 与 prefix cache

很多线上请求有共享前缀。

例如系统 prompt、工具说明、产品规则、多轮对话中的历史前缀。

如果每个请求都重复 prefill 这些相同前缀，会浪费计算和 KV Cache。

Prefix cache 的思想是：

```text
相同前缀的 KV Cache 可以复用。
```

PagedAttention 的 block 管理使这种复用更自然。

如果多个请求共享前几个 block，它们可以指向同一组物理 KV blocks。

后续各自生成时，再为不同后缀分配新的 block。

这类似 copy-on-write 的直觉：

```text
共享前缀只读复用，分叉后再分配新 block。
```

这对系统 prompt 很长、请求模板固定的服务尤其有价值。

它可以降低 TTFT，也可以减少重复 KV 存储。

---

### 十、PagedAttention 与 beam search

Beam search 会在生成过程中维护多个候选序列。

这些候选序列通常共享相同 prompt 和前面一部分生成 token。

如果为每个 beam 完整复制 KV Cache，会非常浪费。

PagedAttention 可以让多个 beam 共享前缀 block，只在分叉后分配不同 block。

例如：

```text
beam 1: A B C D
beam 2: A B C E
beam 3: A B F G
```

它们可以共享 `A B` 对应的 KV blocks。

分叉后的部分再各自分配。

这会降低 beam search 或 best-of-n 场景的 KV Cache 复制成本。

不过在开放式聊天中，beam search 本身未必是常用解码策略。

这里重要的是理解：

```text
PagedAttention 不只是减少碎片，也让共享前缀、分叉序列和动态调度更容易实现。
```

---

### 十一、block size 的 trade-off

block size 不是越大越好，也不是越小越好。

如果 block size 太大：

1. 最后一个 block 的内部碎片更大。
2. 短请求浪费更多 KV 空间。
3. 细粒度复用能力变弱。

如果 block size 太小：

1. block table 更大。
2. 映射和调度开销更高。
3. attention kernel 访问更碎，可能影响访存效率。
4. 管理大量小 block 会增加系统复杂度。

所以 block size 是系统权衡。

面试中可以这样说：

```text
较小 block 降低碎片，较大 block 提高管理和访存效率。实际系统需要结合模型结构、kernel 实现、典型请求长度和硬件特性选择。
```

---

### 十二、PagedAttention 不解决什么问题

很多人会把 PagedAttention 误解成“让 attention 复杂度从 O(T^2) 变低”。

这是不准确的。

PagedAttention 不改变标准 attention 的数学定义。

它也不是 sparse attention 或 linear attention。

它主要解决：

```text
LLM serving 中 KV Cache 的显存分配、碎片、复用和动态调度问题。
```

它不能直接消除长上下文的所有成本。

上下文越长，decode 时仍然要访问更多历史 K/V。

它也不能凭空增加 GPU 显存。

它只是让已有显存被更有效地使用。

一句话：

```text
PagedAttention 是内存管理和 kernel 实现优化，不是新的注意力建模方法。
```

---

### 十三、真实项目中的坑

#### 1. 只看权重显存，忽略 KV Cache

很多初学者部署模型时只估算权重：

```text
7B * 2 bytes ≈ 14 GB
```

然后认为 24GB GPU 足够。

但线上服务还要考虑 KV Cache。

上下文长、并发高时，KV Cache 很快吃掉剩余显存。

#### 2. 只看平均延迟，忽略尾延迟

KV Cache 管理不稳定会导致某些请求排队、等待 block 或触发 OOM。

平均延迟可能看起来还可以，但 P95/P99 很差。

服务系统必须关注尾延迟。

#### 3. 忽略输入输出长度分布

不同业务的长度分布差异很大。

客服问答、代码生成、长文总结、Agent 工具调用的 KV Cache 压力完全不同。

上线前要统计：

1. prompt token 分布。
2. output token 分布。
3. 并发分布。
4. 多轮对话历史长度。
5. 系统 prompt 和工具 schema 长度。

#### 4. 以为 PagedAttention 总能提升速度

PagedAttention 的主要收益是显存利用率和并发能力。

如果场景是低并发、短上下文、显存不紧张，收益可能不明显。

如果 kernel 实现不好，非连续访问还可能带来额外开销。

#### 5. 释放和复用策略不当

请求结束后必须及时释放 block。

如果引用计数、共享前缀或异常中断处理不正确，可能出现显存泄漏或错误复用。

---

### 十四、面试问答

#### 问题 1：KV Cache 为什么会成为推理服务瓶颈？

可以这样回答：

```text
KV Cache 是每个活跃请求自己的运行时状态，大小与层数、上下文长度、KV head 数、head dim、精度和并发数成正比。
模型权重是固定的，但 KV Cache 会随着请求数和生成长度动态增长。长上下文和高并发时，KV Cache 很容易成为显存容量和带宽瓶颈。
```

#### 问题 2：传统连续 KV Cache 分配有什么问题？

可以这样回答：

```text
如果为每个请求按最大长度预留连续空间，会产生大量内部碎片；请求动态开始和结束又会造成外部碎片。
LLM 请求长度高度不均匀，连续分配很难同时做到高显存利用率和高并发。
```

#### 问题 3：PagedAttention 解决了什么问题？

可以这样回答：

```text
PagedAttention 借鉴操作系统分页思想，把 KV Cache 拆成固定大小的 blocks。逻辑上一个请求的上下文是连续的，但物理 KV blocks 可以不连续，通过 block table 做映射。
这样可以减少显存碎片，按需分配和释放 KV Cache，提高高并发、变长请求下的显存利用率。
```

#### 问题 4：PagedAttention 会降低 attention 的理论复杂度吗？

可以这样回答：

```text
不会。PagedAttention 不改变标准 attention 的数学定义，也不是 sparse attention 或 linear attention。
它主要优化 KV Cache 的显存管理和访问方式，让 serving 系统更高效地支持变长和高并发请求。
```

#### 问题 5：PagedAttention 和 Continuous Batching 的关系是什么？

可以这样回答：

```text
Continuous Batching 需要请求在 decode 过程中动态加入和退出，这要求 KV Cache 能灵活分配、追加和释放。
PagedAttention 提供 block 级 KV Cache 管理，是 Continuous Batching 高效运行的重要基础之一。
```

#### 问题 6：如何估算 KV Cache 显存？

可以这样回答：

```text
单请求 KV Cache 可以粗略估算为 2 * L * S * H_kv * D_h * bytes。
多请求时把 S 换成所有活跃请求的 token 总数，也就是 sum(S_i)。GQA/MQA 可以减少 H_kv，从而显著降低 KV Cache 显存。
```

---

### 十五、常见误区

1. 误区：PagedAttention 是一种新的 attention 模型。
   纠正：它主要是 KV Cache 内存管理和 kernel 实现优化，不改变 attention 数学定义。

2. 误区：PagedAttention 能让长上下文免费。
   纠正：它降低显存浪费，但长上下文仍然带来更大的 KV Cache 和更多 attention 访问。

3. 误区：只要用了 PagedAttention，就不会 OOM。
   纠正：它提高显存利用率，但总显存仍有限。并发、上下文和输出长度过大仍会 OOM。

4. 误区：block size 越小越好。
   纠正：小 block 降低碎片，但增加映射、调度和访存开销。

5. 误区：KV Cache 只影响显存容量。
   纠正：KV Cache 还影响显存带宽、decode TPOT、batch 调度和尾延迟。

6. 误区：Prefill 优化和 KV Cache 管理无关。
   纠正：Prefill 会创建初始 KV Cache，prompt cache 和 prefix cache 都依赖 KV 管理。

### 本讲总结

本讲最重要的结论：

1. KV Cache 是每个请求的运行时状态，大小随上下文长度、输出长度和并发数增长。
2. 高并发和长上下文下，KV Cache 往往是 LLM serving 的核心显存瓶颈。
3. 传统连续分配容易产生内部碎片和外部碎片，不适合变长生成请求。
4. PagedAttention 借鉴分页思想，把 KV Cache 拆成固定大小 blocks，通过 block table 映射逻辑 token 到物理 block。
5. PagedAttention 主要提升显存利用率、并发能力和动态调度能力，不改变 attention 的数学复杂度。
6. PagedAttention 是 Continuous Batching、prefix cache、beam search 共享前缀等机制的重要底层基础。
7. block size 需要在碎片率、映射开销、访存效率和实现复杂度之间权衡。
8. 面试时要把 PagedAttention 讲成一个 LLM serving 内存管理问题，而不是新的模型结构。

## 第 68 讲：Continuous Batching

### 本讲目标

学完本讲，你应该能回答六个问题：

1. 为什么大模型推理不能简单照搬传统静态 batching。
2. Decode 阶段为什么特别需要动态调度。
3. Continuous Batching 的核心机制是什么。
4. 它如何影响 TTFT、TPOT、吞吐和尾延迟。
5. 它和 PagedAttention、KV Cache 管理、chunked prefill 有什么关系。
6. 面试中如何设计一个支持高并发 LLM serving 的调度器。

上一讲讲了 PagedAttention。

它解决的是：

```text
KV Cache 如何高效分配、增长、释放和复用。
```

这一讲讲 Continuous Batching。

它解决的是：

```text
请求如何在推理过程中动态进入、执行和退出。
```

两者经常一起出现，因为高并发 LLM serving 同时需要：

```text
高效内存管理 + 高效请求调度
```

---

### 一、为什么需要 batching

单个请求的 Decode 阶段通常 GPU 利用率很低。

原因是每一步只为每个请求生成一个 token。

如果只有一个请求：

```text
decode step 1：处理 1 个 token
decode step 2：处理 1 个 token
decode step 3：处理 1 个 token
...
```

GPU 上的大矩阵乘法规模太小，很多算力吃不满。

如果把多个请求放到一起：

```text
decode step 1：同时处理 32 个请求的下一个 token
decode step 2：同时处理 32 个请求的下一个 token
decode step 3：同时处理 32 个请求的下一个 token
...
```

矩阵计算规模变大，吞吐通常会上升。

这就是 batching 的基本动机：

```text
用更多并发请求填满 GPU，提高 tokens/s，降低单位 token 成本。
```

但 LLM serving 的 batching 比普通模型推理复杂很多。

---

### 二、传统静态 batching 的问题

传统推理服务常用静态 batch。

流程大致是：

```text
收集一批请求
-> 拼成 batch
-> 一起执行
-> 全部完成后释放 batch
-> 再收集下一批
```

这对图像分类这类固定输入、固定输出任务比较自然。

例如一次处理 64 张图片，每张图片只需要一次 forward。

但 LLM 有三个特点。

第一个特点是输入长度不同。

```text
请求 A prompt 100 token
请求 B prompt 8000 token
请求 C prompt 500 token
```

Prefill 时间差异很大。

第二个特点是输出长度未知。

```text
请求 A 生成 20 token 就结束
请求 B 生成 2000 token 才结束
请求 C 中途遇到 stop token 提前结束
```

Decode step 数差异很大。

第三个特点是请求会持续到达。

线上服务不是一批请求全部来了再开始，而是不断有新请求进入队列。

如果使用静态 batch，会出现两个问题。

第一个问题是短请求等待长请求。

```text
同一个 batch 中，短请求已经结束，但系统还在等长请求生成完。
```

第二个问题是新请求不能及时加入。

```text
GPU 正在跑一个长 batch，新来的请求只能排队等待下一轮。
```

这会降低 GPU 利用率，也会增加 TTFT 和尾延迟。

---

### 三、Continuous Batching 的核心直觉

Continuous Batching 也叫 in-flight batching 或 iteration-level batching。

核心思想是：

```text
不要把 batch 看成固定的一组请求，而是在每个 decode iteration 动态更新 batch。
```

每个 decode step 后，调度器会做三件事：

1. 把已经结束的请求移出 batch。
2. 为还没结束的请求保留状态，下一步继续 decode。
3. 从等待队列中加入新请求，补充 batch。

可以用流程表示：

```text
初始 batch：A B C
step 1 后：A 结束，B C 继续，新请求 D 加入
step 2 后：B 结束，C D 继续，新请求 E 加入
step 3 后：C D E 继续，新请求 F 加入
```

所以 batch 是流动的。

这就是 continuous 的含义。

一句话：

```text
Continuous Batching 让 GPU 每个 decode step 都尽量有足够多的活跃请求，而不是被固定 batch 的最长请求拖住。
```

---

### 四、静态 batching 与 Continuous Batching 对比

可以用表格理解：

```text
维度              静态 batching                  Continuous Batching
batch 组成        一开始固定                      每个 iteration 可变化
请求完成          等 batch 整体结束或填充空位困难   完成后立即退出
新请求加入        通常等下一批                     可在 step 间加入
适合任务          固定长度、单次 forward            变长自回归生成
GPU 利用率        容易被长请求拖累                  更容易保持高利用率
调度复杂度        低                              高
KV 管理要求       相对简单                         需要动态分配和释放
```

核心差异是：

```text
静态 batching 以“请求批次”为单位调度；Continuous Batching 以“decode iteration”为单位调度。
```

---

### 五、一个最小调度器例子

可以用简化伪代码理解 Continuous Batching。

真实系统会复杂很多，但核心循环类似：

```python
waiting = Queue()
running = []

while True:
    # 1. 尽量填充 running batch
    while len(running) < max_batch_size and not waiting.empty():
        req = waiting.pop()
        allocate_kv_cache(req)
        run_prefill(req)
        running.append(req)

    if not running:
        continue

    # 2. 对当前 running 请求执行一个 decode step
    next_tokens = model_decode_one_step(running)

    # 3. 更新每个请求状态
    finished = []
    for req, token in zip(running, next_tokens):
        req.append(token)
        append_kv_cache(req, token)
        if req.is_finished():
            finished.append(req)

    # 4. 移除完成请求并释放资源
    for req in finished:
        running.remove(req)
        free_kv_cache(req)
        stream_response_done(req)
```

这个伪代码只表达调度直觉。

真实系统还要处理：

1. prefill 和 decode 的混合调度。
2. KV Cache block 分配和回收。
3. 请求优先级。
4. 最大上下文和最大输出长度。
5. 流式返回。
6. abort、timeout、异常释放。
7. 多 GPU 和多副本负载均衡。

面试时只要能说出核心状态机，就已经比只背名词强很多。

下面这个 demo 把上面的伪代码改成一个可运行的 toy scheduler。它用很小的 `TOKEN_BUDGET`、`KV_BLOCK_BUDGET` 和 `PREFILL_CHUNK`，模拟 waiting queue、running batch、chunked prefill、decode step 和 KV block 分配释放。

```python
from collections import deque
from dataclasses import dataclass
from math import ceil


BLOCK_SIZE = 16
MAX_RUNNING = 3
TOKEN_BUDGET = 80
KV_BLOCK_BUDGET = 32
PREFILL_CHUNK = 64


@dataclass
class Request:
    name: str
    prompt_tokens: int
    output_tokens: int
    prompt_done: int = 0
    generated: int = 0
    blocks: int = 0

    @property
    def prompt_left(self) -> int:
        return self.prompt_tokens - self.prompt_done

    @property
    def cache_tokens(self) -> int:
        return self.prompt_done + self.generated


def blocks_needed(tokens: int) -> int:
    return ceil(tokens / BLOCK_SIZE)


def ensure_blocks(req: Request, new_cache_tokens: int, used_blocks: int) -> int | None:
    needed = blocks_needed(new_cache_tokens)
    extra = needed - req.blocks
    if used_blocks + extra > KV_BLOCK_BUDGET:
        return None
    req.blocks = needed
    return used_blocks + extra


arrivals = {
    0: [Request("A", 40, 3), Request("B", 80, 5)],
    1: [Request("C", 16, 2)],
    2: [Request("D_long", 180, 4)],
    4: [Request("E", 24, 2)],
}

waiting = deque()
ready = deque()
running = []
used_blocks = 0

for step in range(12):
    for req in arrivals.get(step, []):
        waiting.append(req)

    events = []
    budget = TOKEN_BUDGET

    for req in list(running):
        new_used = ensure_blocks(req, req.cache_tokens + 1, used_blocks)
        if new_used is None:
            events.append(f"{req.name}:decode_blocked")
            continue
        used_blocks = new_used
        req.generated += 1
        budget -= 1
        events.append(f"{req.name}:dec{req.generated}/{req.output_tokens}")
        if req.generated >= req.output_tokens:
            running.remove(req)
            used_blocks -= req.blocks
            events.append(f"{req.name}:done_free{req.blocks}")

    while ready and len(running) < MAX_RUNNING:
        req = ready.popleft()
        running.append(req)
        events.append(f"{req.name}:start_decode")

    while waiting and budget > 0:
        req = waiting[0]
        chunk = min(req.prompt_left, budget, PREFILL_CHUNK)
        new_used = ensure_blocks(req, req.prompt_done + chunk, used_blocks)
        if new_used is None:
            events.append(f"{req.name}:prefill_blocked")
            break
        used_blocks = new_used
        req.prompt_done += chunk
        budget -= chunk
        events.append(f"{req.name}:pre{req.prompt_done}/{req.prompt_tokens}")
        if req.prompt_left == 0:
            waiting.popleft()
            if len(running) < MAX_RUNNING:
                running.append(req)
                events.append(f"{req.name}:start_decode")
            else:
                ready.append(req)
                events.append(f"{req.name}:ready")
        else:
            break

    if not events and not running and not waiting and not ready:
        break

    print(
        f"iter={step}",
        f"budget_left={budget}",
        f"kv_blocks={used_blocks}",
        f"running={[r.name for r in running]}",
        f"events={events}",
    )
```

典型输出如下：

```text
iter=0 budget_left=0 kv_blocks=6 running=['A'] events=['A:pre40/40', 'A:start_decode', 'B:pre40/80']
iter=1 budget_left=23 kv_blocks=9 running=['A', 'B', 'C'] events=['A:dec1/3', 'B:pre80/80', 'B:start_decode', 'C:pre16/16', 'C:start_decode']
iter=2 budget_left=13 kv_blocks=15 running=['A', 'B', 'C'] events=['A:dec2/3', 'B:dec1/5', 'C:dec1/2', 'D_long:pre64/180']
iter=3 budget_left=13 kv_blocks=14 running=['B'] events=['A:dec3/3', 'A:done_free3', 'B:dec2/5', 'C:dec2/2', 'C:done_free2', 'D_long:pre128/180']
iter=4 budget_left=3 kv_blocks=20 running=['B', 'D_long', 'E'] events=['B:dec3/5', 'D_long:pre180/180', 'D_long:start_decode', 'E:pre24/24', 'E:start_decode']
iter=5 budget_left=77 kv_blocks=20 running=['B', 'D_long', 'E'] events=['B:dec4/5', 'D_long:dec1/4', 'E:dec1/2']
iter=6 budget_left=77 kv_blocks=12 running=['D_long'] events=['B:dec5/5', 'B:done_free6', 'D_long:dec2/4', 'E:dec2/2', 'E:done_free2']
iter=7 budget_left=79 kv_blocks=12 running=['D_long'] events=['D_long:dec3/4']
iter=8 budget_left=79 kv_blocks=0 running=[] events=['D_long:dec4/4', 'D_long:done_free12']
```

注意 `D_long` 的 prefill 被切成了多次执行，A/C/E 这类短请求完成后会立刻释放 KV blocks。这个 toy demo 不模拟真实 attention kernel，但它能帮助你把 Continuous Batching 理解成“每个 iteration 的预算分配 + 状态迁移 + KV 生命周期管理”。

### 六、Prefill 和 Decode 如何混合调度

Continuous Batching 最难的部分之一是：

```text
新请求需要 prefill，老请求需要 decode。
```

Prefill 和 Decode 的资源特征不同。

Prefill 通常是大矩阵计算，输入 token 多，偏 compute-bound。

Decode 每步 token 少，但要读权重和 KV Cache，偏 memory-bandwidth-bound。

如果一次性塞入很多长 prompt 做 prefill，可能会阻塞已有请求的 decode。

用户感受到的是：

```text
正在生成的回答突然变慢，TPOT 变差。
```

如果完全优先 decode，不给新请求做 prefill，新请求的 TTFT 会变差。

所以调度器要在两类任务之间权衡：

```text
让老请求持续吐 token，同时让新请求尽快开始吐第一个 token。
```

常见做法包括：

1. 限制每轮可加入的新请求数。
2. 限制每轮 prefill token budget。
3. 把长 prefill 切成多个 chunk。
4. 根据 SLA 或优先级决定 prefill/decode 配额。
5. 在吞吐和延迟之间做动态调节。

这就引出了 chunked prefill。

---

### 七、Chunked Prefill 的直觉

假设一个新请求有 32000 token prompt。

如果一次性 prefill 完，可能长时间占用 GPU。

这会影响其他正在 decode 的请求。

Chunked prefill 的做法是把长 prompt 拆成多个 chunk。

例如每次处理 2048 token：

```text
prefill chunk 1：token 1-2048
decode existing requests
prefill chunk 2：token 2049-4096
decode existing requests
...
```

这样可以避免超长 prefill 独占 GPU。

它的收益是：

1. 降低长 prompt 对其他请求 TPOT 的冲击。
2. 让调度更平滑。
3. 更容易控制每轮 GPU 工作量。

代价是：

1. 调度复杂度更高。
2. prefill 被切分后可能增加额外开销。
3. TTFT 不一定总是最优，因为长 prompt 仍需全部 prefill 后才能生成第一个 token。

所以 chunked prefill 是一种系统权衡，不是免费午餐。

---

### 八、Continuous Batching 如何影响 TTFT、TPOT 和吞吐

Continuous Batching 不是单纯“让所有指标都变好”。

它会改变多个指标的权衡。

#### 1. 对吞吐的影响

通常提升吞吐。

原因是 running batch 更容易保持较高活跃请求数，Decode 阶段 GPU 利用率更高。

```text
更多活跃请求 -> 更大的有效 batch -> 更高 decode tokens/s
```

#### 2. 对 TPOT 的影响

在合理负载下，TPOT 可能改善，因为 GPU 利用率提升。

但在高负载下，TPOT 也可能变差。

原因是 batch 过大、KV Cache 读写压力上升、调度开销增加。

#### 3. 对 TTFT 的影响

Continuous Batching 可以让新请求更早加入服务，不必等整批结束。

这有助于降低排队导致的 TTFT。

但如果系统过度优先已有 decode，新请求 prefill 仍可能排队。

如果系统过度插入 prefill，又可能影响老请求 TPOT。

#### 4. 对尾延迟的影响

尾延迟取决于调度策略。

如果没有优先级和 token budget 控制，长请求、超长 prompt 或大输出可能影响其他请求。

所以线上系统不仅要看平均 tokens/s，还要看：

```text
P50/P95/P99 TTFT
P50/P95/P99 TPOT
P50/P95/P99 end-to-end latency
```

---

### 九、调度器需要控制哪些资源

一个 LLM serving 调度器不是只控制 batch size。

它至少要控制四类资源。

#### 1. Token budget

每个 iteration 允许处理多少 token。

包括 prefill token 和 decode token。

例如：

```text
本轮最多处理 8192 个 token
其中 decode 请求 128 个 token
剩余预算给 prefill
```

可以写成：

```math
N_{\mathrm{dec}}+N_{\mathrm{pre}}\leq B_{\mathrm{tok}}
```

其中 `B_tok` 是当前 iteration 的 token budget，`N_dec` 是本轮 decode token 数，`N_pre` 是本轮允许插入的 prefill token 数。调度器本质上是在每一轮决定这个预算怎么分。

#### 2. KV Cache budget

新请求加入前，要确认是否有足够 KV Cache block。

否则可能中途 OOM。

调度器要估算：

```text
prompt KV + 预计生成 KV + 安全余量
```

按 block 估算可以写成：

```math
K_{\mathrm{need}}=\sum_i \left\lceil \frac{S_i+O_i^{\max}}{C}\right\rceil + K_{\mathrm{margin}}
```

其中 `S_i` 是第 `i` 个请求的 prompt token 数，`O_i^max` 是保守估计的最大输出 token 数，`C` 是 block size，`K_margin` 是安全余量。只有 `K_need` 不超过可用 block 数，新请求才应该进入执行。

#### 3. Running batch size

活跃请求数太小，GPU 利用率低。

活跃请求数太大，KV Cache、带宽和尾延迟压力上升。

所以最大 batch size 通常要和显存、模型、业务 SLA 一起调。

#### 4. 请求优先级

线上服务可能有不同用户或不同任务等级。

例如：

1. 交互式聊天要求低 TTFT。
2. 后台总结任务可以接受更高延迟。
3. 付费用户可能有更高优先级。
4. 短请求可以用 shortest-job-first 类策略降低平均延迟。

但优先级策略要防止低优先级请求长期饥饿。

---

### 十、调度策略的常见选择

#### 1. FCFS

First-Come, First-Served。

先来先服务，实现简单，公平性直观。

缺点是长请求可能阻塞后面的短请求。

#### 2. Shortest Job First 近似

如果能估计请求长度，可以优先处理短请求。

它可能降低平均延迟。

但 LLM 输出长度难准确预测，而且过度偏向短请求会让长请求饥饿。

#### 3. Priority Scheduling

按业务等级、用户等级或 SLA 优先级调度。

适合线上多租户服务。

缺点是需要配套限流、配额和公平性控制。

#### 4. Token Budget Scheduling

每轮设置 token budget，在 prefill 和 decode 之间分配预算。

这是 LLM serving 中很常见的调度直觉。

它比只限制 request 数更合理，因为真实成本更接近 token 数和 KV Cache 占用。

---

### 十一、与 PagedAttention 的配合

Continuous Batching 需要请求动态加入和退出。

这意味着 KV Cache 必须动态管理。

如果 KV Cache 必须为每个请求连续预留最大长度，那么每次加入新请求都很难。

PagedAttention 提供 block 级分配后，调度器可以：

1. 新请求 prefill 时分配一批 blocks。
2. 每个 decode step 追加 token，必要时分配新 block。
3. 请求结束时释放 blocks。
4. 共享 prefix 时复用 blocks。
5. batch 中请求顺序变化时，通过 block table 访问各自 KV。

所以：

```text
PagedAttention 让 KV Cache 的物理布局更灵活，Continuous Batching 让请求执行顺序更灵活。
```

二者结合，才适合真实的高并发变长生成服务。

---

### 十二、一个服务端时间线例子

假设系统正在服务三个请求：

```text
A：已经 decode 10 步，还需继续生成
B：已经 decode 3 步，即将结束
C：刚完成 prefill，准备生成
```

等待队列中有两个新请求：

```text
D：短 prompt
E：超长 prompt
```

一个合理调度可能是：

```text
iteration 1：A/B/C decode 一步，B 结束释放 KV
iteration 2：A/C decode 一步，同时插入 D 的 prefill
iteration 3：A/C/D decode 一步，开始 E 的 chunked prefill 第一段
iteration 4：A/C/D decode 一步，继续 E 的 chunked prefill 第二段
iteration 5：C 结束，E prefill 完成，A/D/E decode
```

这个例子说明：

1. 请求不是固定 batch。
2. decode 和 prefill 可能交错。
3. 长 prompt 可能被切块。
4. 完成请求会释放 KV。
5. 新请求可以在运行过程中加入。

这就是 Continuous Batching 的系统视角。

---

### 十三、真实项目中的坑

#### 1. 只追求最大吞吐，牺牲交互体验

大 batch 可以提高 tokens/s。

但如果排队时间和 TTFT 变差，聊天产品体验会明显下降。

线上调度不能只看吞吐。

#### 2. 长 prompt 抢占 decode 资源

超长 prompt prefill 可能让已有流式输出卡顿。

需要 prefill token budget 或 chunked prefill 控制。

#### 3. KV Cache 预算估计不足

只看当前 prompt KV，不考虑未来生成 KV，会导致生成过程中 OOM。

调度器要为最大输出长度或保守估计留余量。

#### 4. 请求取消后资源没有及时释放

用户断开连接、超时、客户端取消时，服务端必须释放 KV Cache 和队列状态。

否则会出现隐性显存泄漏。

#### 5. 过度依赖平均长度

真实线上流量通常长尾明显。

平均 prompt 长度和平均 output 长度无法反映 P99 风险。

容量规划要看分位数。

#### 6. 忽略流式返回的稳定性

对用户来说，平均 TPOT 不如“吐字是否稳定”直观。

如果调度导致 token 间隔忽快忽慢，体验会变差。

### 十四、面试问答

#### 问题 1：为什么 LLM serving 需要 Continuous Batching？

可以这样回答：

```text
LLM 推理是变长自回归生成，请求输入长度、输出长度和到达时间都不同。静态 batching 会让短请求等待长请求，新请求也要等当前 batch 完成。
Continuous Batching 在每个 decode iteration 动态移除完成请求并加入新请求，可以提高 GPU 利用率、降低排队浪费，更适合高并发变长生成。
```

#### 问题 2：Continuous Batching 和普通 dynamic batching 有什么区别？

可以这样回答：

```text
普通 dynamic batching 多是在请求开始前收集一批请求做一次 batch；Continuous Batching 是在自回归生成过程中，以 decode iteration 为粒度持续更新 batch。
它允许请求在生成过程中加入和退出，而不是一批请求固定跑到结束。
```

#### 问题 3：Continuous Batching 如何影响 TTFT 和 TPOT？

可以这样回答：

```text
它通常能提高吞吐，并让新请求不必等整批结束，从而降低排队导致的 TTFT。但调度策略很关键：如果过度插入 prefill，已有请求 TPOT 会变差；如果过度优先 decode，新请求 TTFT 会变差。
```

#### 问题 4：为什么 Continuous Batching 需要好的 KV Cache 管理？

可以这样回答：

```text
因为请求会在每个 iteration 动态加入、生成和结束，KV Cache 也要动态分配、追加和释放。PagedAttention 这种 block 级 KV 管理能减少碎片，使动态调度更可行。
```

#### 问题 5：什么是 chunked prefill？

可以这样回答：

```text
chunked prefill 是把长 prompt 的 prefill 拆成多个 token chunk，穿插在 decode iteration 中执行。它可以避免超长 prompt 长时间独占 GPU，降低对已有流式输出 TPOT 的冲击，但会增加调度复杂度。
```

#### 问题 6：设计 LLM 调度器时要控制哪些预算？

可以这样回答：

```text
至少要控制 token budget、KV Cache budget、running batch size 和请求优先级。不能只按请求数调度，因为真实成本与 token 数、上下文长度、生成长度和 KV Cache 占用更相关。
```

---

### 十五、常见误区

1. 误区：Continuous Batching 就是普通 batch size 调大。
   纠正：它的关键是 iteration-level 动态加入和退出，不只是 batch 更大。

2. 误区：Continuous Batching 一定降低所有请求延迟。
   纠正：它主要提升吞吐和利用率，延迟取决于调度策略和负载。

3. 误区：只要 GPU 利用率高，服务就好。
   纠正：交互式服务还要看 TTFT、TPOT 稳定性、P95/P99 和取消释放。

4. 误区：prefill 和 decode 可以无脑混在一起。
   纠正：二者资源特征不同，长 prefill 可能影响 decode，需要 token budget 或 chunked prefill。

5. 误区：调度只需要限制请求数量。
   纠正：请求成本差异很大，更应该关注 token budget、KV Cache budget 和长度分布。

6. 误区：请求结束只要停止生成即可。
   纠正：还要释放 KV Cache、清理队列状态、关闭流式响应，并处理异常取消。

### 本讲总结

本讲最重要的结论：

1. LLM 请求输入长度、输出长度和到达时间都不固定，静态 batching 容易浪费和阻塞。
2. Continuous Batching 以 decode iteration 为粒度动态更新 batch，让完成请求退出、新请求加入。
3. 它通常能提高 GPU 利用率和 decode 吞吐，但延迟表现取决于调度策略。
4. Prefill 和 Decode 资源特征不同，混合调度时要控制 token budget。
5. Chunked prefill 可以降低超长 prompt 对流式 decode 的冲击，但增加调度复杂度。
6. PagedAttention 提供动态 KV Cache 管理，Continuous Batching 提供动态请求调度，两者相互配合。
7. 线上服务不能只看平均 tokens/s，还要看 TTFT、TPOT、P95/P99、取消释放和流式稳定性。
8. 面试中要把 Continuous Batching 讲成 LLM serving 调度问题，而不是简单的 batch size 参数。

## 第 69 讲：Speculative Decoding

### 本讲目标

学完本讲，你应该能回答六个问题：

1. Speculative Decoding 想解决 Decode 阶段的什么瓶颈。
2. draft model 和 target model 分别负责什么。
3. 为什么一次 target forward 可以验证多个候选 token。
4. speculative sampling 如何在加速的同时保持目标模型分布不变。
5. 接受率、draft 长度、draft 模型质量如何影响加速比。
6. 面试中如何分析 Speculative Decoding 的优点、局限和工程落地条件。

前两讲我们讲了 KV Cache 管理和 Continuous Batching。

它们优化的是 serving 系统层面：

```text
KV Cache 怎么放得下，batch 怎么调度得更满。
```

这一讲进入 Decode 加速本身。

Decode 的核心瓶颈是：

```text
大模型每次只能生成一个 token，而且每个 token 都要跑一次大模型 forward。
```

Speculative Decoding 的目标是：

```text
让小模型先“猜”多个 token，再让大模型一次性“验”多个 token。
```

如果猜得对，就用一次大模型 forward 接受多个 token，从而减少大模型调用次数。

---

### 一、Decode 为什么慢

自回归生成有强依赖关系。

第 `t+1` 个 token 依赖前面已经生成的 token。

普通 decode 流程是：

```text
大模型 forward -> 采样 token 1
大模型 forward -> 采样 token 2
大模型 forward -> 采样 token 3
...
```

即使每一步只生成一个 token，也要读取模型权重、访问 KV Cache、做 attention 和 MLP。

对于大模型，单步 decode 常常受显存带宽、KV Cache 和调度影响。

这导致 TPOT 难以下降。

如果一个回答要生成 1000 token，普通 decode 至少需要大模型执行 1000 次 decode step。

Speculative Decoding 的想法是：

```text
能不能不要让大模型一步一步猜，而是让便宜模型先猜一段，大模型批量检查？
```

---

### 二、核心角色：draft model 和 target model

Speculative Decoding 通常包含两个模型。

第一个是 target model。

这是最终要服务的高质量大模型。

例如 70B 模型。

第二个是 draft model。

这是一个更小、更快的模型。

例如 7B 模型，或者 target model 的轻量变体。

它负责快速提出候选 token。

```text
draft model：便宜、快、质量略低，用来猜。
target model：昂贵、慢、质量高，用来验。
```

流程可以简化成：

```text
1. draft model 连续生成 k 个候选 token。
2. target model 一次 forward 计算这 k 个位置的概率。
3. 按规则接受前面若干个候选 token。
4. 如果某个候选被拒绝，就用 target model 的分布重新采样。
5. 重复这个过程。
```

如果 draft model 猜得准，target model 一次 forward 可以接受多个 token。

这样大模型的 decode step 数就减少了。

---

### 三、为什么 target model 能一次验证多个 token

这是理解 Speculative Decoding 的关键。

假设当前上下文是：

```text
x_1, x_2, ..., x_t
```

draft model 猜了 4 个 token：

```text
y_1, y_2, y_3, y_4
```

普通生成时，target model 要逐步计算：

```text
p(y_1 | x)
p(y_2 | x, y_1)
p(y_3 | x, y_1, y_2)
p(y_4 | x, y_1, y_2, y_3)
```

这些条件概率看起来是串行的。

但如果候选序列 `y_1...y_4` 已经给定，就可以把它们作为一段输入送给 target model。

Transformer 在一次 forward 中可以并行计算每个位置的 logits。

也就是说，target model 可以一次性得到：

```text
位置 t+1 的 logits
位置 t+2 的 logits
位置 t+3 的 logits
位置 t+4 的 logits
```

每个位置仍然遵守 causal mask，只看自己之前的 token。

所以这不是违反自回归因果性。

它只是利用了一个事实：

```text
给定候选 token 后，验证这些 token 的概率可以并行完成。
```

这和训练时 teacher forcing 的并行计算有相似直觉。

---

### 四、最简单的接受直觉

先讲一个不严格但容易理解的版本。

draft model 猜出：

```text
今 天 天 气 很 好
```

target model 检查后发现前几个 token 也很认可：

```text
今：接受
天：接受
天：拒绝
```

那么系统可以一次性提交前两个 token：

```text
今 天
```

在拒绝位置，用 target model 自己的分布重新采样一个 token。

然后进入下一轮。

如果每轮 draft 4 个 token，平均能接受 3 个 token，那么 target model 大致每次 forward 推进 3 个 token。

相比普通 decode 每次 forward 只推进 1 个 token，就有潜在加速。

但真实算法要更严格。

因为我们不能只是“target 觉得差不多就接受”。

如果接受规则不正确，最终生成分布会偏离 target model。

---

### 五、为什么需要保持目标分布不变

部署大模型时，用户期待输出来自 target model。

如果 Speculative Decoding 改变了生成分布，就可能带来质量下降或行为变化。

理想目标是：

```text
加速生成，但最终采样分布与直接从 target model 采样一致。
```

这叫无偏或分布保持。

设下一个 token 的候选词表为 `V`。在某个上下文 `c` 下：

```math
p(v)=P_{\mathrm{target}}(v\mid c),
\qquad
q(v)=P_{\mathrm{draft}}(v\mid c)
```

为了避免把代码变量写进复杂公式里，可以记成：`p` 是 target model 的下一个 token 分布，`q` 是 draft model 的下一个 token 分布，`v` 是词表里的一个候选 token。

draft 先从 `q` 采样一个 token `y`。

如果 `q(y)` 很小而 `p(y)` 很大，说明 draft 偶然猜到了 target 很喜欢的 token，应该容易接受。

如果 `q(y)` 很大而 `p(y)` 很小，说明 draft 很喜欢但 target 不喜欢，应该经常拒绝。

常见接受概率可以写成：

```math
a(y)=\min\left(1,\frac{p(y)}{q(y)}\right)
```

其中 `a(y)` 是候选 token `y` 的接受概率。因为 `y` 是从 `q` 里采样出来的，理论上 `q(y)` 应该大于 0；工程实现中仍然会做概率裁剪、归一化和数值保护。

如果接受失败，就需要从修正后的分布中采样。

直觉上，修正分布补偿的是：

```text
target 想生成、但 draft 过度或不足覆盖后剩下的概率质量。
```

一个常见写法是：

```math
r(v)=\frac{\max(p(v)-q(v),0)}{\sum_{u\in\mathcal{V}}\max(p(u)-q(u),0)}
```

这里 `r(v)` 是拒绝 draft token 后用于重新采样的修正分布。分子只保留 target 分布相对 draft 分布“多出来”的概率质量，分母负责重新归一化。这个公式的直觉是：draft 已经过度提案的区域不再补，target 仍然缺的区域要补回来。

面试中不一定要完整推导，但要知道：

```text
Speculative Decoding 不是简单用小模型替大模型，而是用严格的接受-拒绝采样保证最终分布仍等价于 target model。
```

---

### 六、多 token speculative sampling 流程

假设每轮 draft 长度为 `k`。

第 `i` 个候选 token 的条件分布可以写成：

```math
p_i(v)=P_{\mathrm{target}}(v\mid x,y_1,\ldots,y_{i-1}),
\qquad
q_i(v)=P_{\mathrm{draft}}(v\mid x,y_1,\ldots,y_{i-1})
```

对应的接受概率为：

```math
a_i=\min\left(1,\frac{p_i(y_i)}{q_i(y_i)}\right)
```

这个写法强调：每个候选 token 都是在“已经接受的前缀”条件下验证的，不是只比较一个固定的全局分布。

流程如下：

```text
当前上下文 x
1. draft model 自回归生成 y_1, ..., y_k
2. target model 对 x + y_1 + ... + y_k 做一次 forward
3. 从 i=1 到 k 依次检查 y_i
4. 如果 y_i 被接受，加入输出上下文
5. 如果 y_i 被拒绝，用 target 修正分布采样一个 token，结束本轮
6. 如果 y_1...y_k 全部接受，通常还可以额外从 target 的最后位置采样一个 token
```

为什么拒绝后本轮要停止？

因为后面的 draft token 是基于被拒绝 token 继续生成的。

一旦某个 token 被替换，后面的上下文就变了。

后续候选 token 不再对应正确条件分布。

所以必须结束本轮，基于新的上下文重新 draft。

---

### 七、一个简化伪代码

下面是教学版伪代码，只表达机制，不覆盖所有数值细节。

```python
def speculative_decode(target, draft, prompt, max_new_tokens, k):
    tokens = list(prompt)

    while len(tokens) < len(prompt) + max_new_tokens:
        # 1. 小模型快速猜 k 个 token
        draft_tokens = []
        draft_probs = []
        for _ in range(k):
            q = draft.next_token_distribution(tokens + draft_tokens)
            y = sample(q)
            draft_tokens.append(y)
            draft_probs.append(q)

        # 2. 大模型一次性验证这些候选 token
        target_probs = target.distributions_for_positions(tokens, draft_tokens)

        # 3. 逐个接受或拒绝
        accepted_all = True
        for i, y in enumerate(draft_tokens):
            p_y = target_probs[i][y]
            q_y = draft_probs[i][y]
            accept_prob = min(1.0, p_y / q_y)

            if random_uniform() < accept_prob:
                tokens.append(y)
            else:
                corrected = corrected_distribution(target_probs[i], draft_probs[i])
                tokens.append(sample(corrected))
                accepted_all = False
                break

        # 4. 如果全部接受，可额外从 target 最后分布采样一个 token
        if accepted_all:
            p_next = target_probs[-1]
            tokens.append(sample(p_next))

    return tokens
```

真实实现会更复杂。

比如要处理 KV Cache、batch、EOS、temperature、top-p、数值稳定和并发调度。

但核心思想就是：

```text
小模型提案，大模型验证，接受则多走几步，拒绝则回到大模型分布。
```

下面这个 demo 演示两个关键点：第一，draft 越接近 target，接受率越高；第二，即使 draft 很差，只要使用接受-拒绝和修正分布，最终单步采样的经验分布仍然接近 target 分布。最后几行用粗略公式估算接受率、draft 长度和加速比的关系。

```python
import random
from collections import Counter


TARGET = {"A": 0.50, "B": 0.25, "C": 0.20, "D": 0.05}
DRAFT_GOOD = {"A": 0.45, "B": 0.30, "C": 0.20, "D": 0.05}
DRAFT_BAD = {"A": 0.10, "B": 0.20, "C": 0.30, "D": 0.40}


def sample(dist: dict[str, float]) -> str:
    r = random.random()
    cumulative = 0.0
    last = None
    for token, prob in dist.items():
        cumulative += prob
        last = token
        if r <= cumulative:
            return token
    return last


def corrected_distribution(p: dict[str, float], q: dict[str, float]) -> dict[str, float]:
    residual = {token: max(p[token] - q[token], 0.0) for token in p}
    total = sum(residual.values())
    if total == 0:
        return p.copy()
    return {token: value / total for token, value in residual.items()}


def speculative_one_step(p: dict[str, float], q: dict[str, float]) -> tuple[str, bool]:
    proposal = sample(q)
    accept_prob = min(1.0, p[proposal] / q[proposal])
    if random.random() < accept_prob:
        return proposal, True
    return sample(corrected_distribution(p, q)), False


def simulate(name: str, q: dict[str, float], trials: int = 50_000) -> None:
    accepted = 0
    counts = Counter()
    for _ in range(trials):
        token, ok = speculative_one_step(TARGET, q)
        counts[token] += 1
        accepted += int(ok)
    empirical = {token: counts[token] / trials for token in TARGET}
    max_error = max(abs(empirical[token] - TARGET[token]) for token in TARGET)
    print(name, "accept_rate=", round(accepted / trials, 3), "dist=", empirical, "max_error=", round(max_error, 3))


def expected_tokens_per_round(alpha: float, k: int) -> float:
    return sum(alpha**i for i in range(k + 1))


def speedup(alpha: float, k: int, target_one_ms: float, target_verify_ms: float, draft_one_ms: float) -> float:
    tokens = expected_tokens_per_round(alpha, k)
    baseline_ms = tokens * target_one_ms
    speculative_ms = target_verify_ms + k * draft_one_ms
    return baseline_ms / speculative_ms


random.seed(7)
simulate("good_draft", DRAFT_GOOD)
simulate("bad_draft", DRAFT_BAD)

for alpha in [0.4, 0.7, 0.9]:
    tokens = expected_tokens_per_round(alpha, k=4)
    print("alpha=", alpha, "expected_tokens=", round(tokens, 2), "speedup=", round(speedup(alpha, 4, 12.0, 15.0, 1.2), 2))
```

典型输出类似：

```text
good_draft accept_rate= 0.95 dist= {'A': 0.50162, 'B': 0.25038, 'C': 0.19872, 'D': 0.04928} max_error= 0.002
bad_draft accept_rate= 0.551 dist= {'A': 0.49652, 'B': 0.25312, 'C': 0.20108, 'D': 0.04928} max_error= 0.003
alpha= 0.4 expected_tokens= 1.65 speedup= 1.0
alpha= 0.7 expected_tokens= 2.77 speedup= 1.68
alpha= 0.9 expected_tokens= 4.1 speedup= 2.48
```

这里 `good_draft` 的接受率明显更高，但 `bad_draft` 的最终分布仍然接近 `TARGET`。这说明 speculative sampling 的质量关键不是“draft 一定正确”，而是接受-拒绝和修正分布必须严格实现。

---

### 八、加速比由什么决定

Speculative Decoding 不是一定加速。

它的加速取决于几个因素。

#### 1. 接受率

接受率越高，一次 target forward 平均推进的 token 越多。

如果 draft 生成的 token 经常被 target 接受，加速明显。

如果经常第一个 token 就被拒绝，几乎没有收益。

为了粗略估算，可以假设每个 draft token 的平均接受概率都是 `alpha`，每轮最多 draft `k` 个 token。如果前面都接受，target 还可以额外采样一个 token，那么每轮平均推进 token 数近似为：

```math
E[N_{\mathrm{advance}}]\approx \sum_{i=0}^{k}\alpha^i
```

当 `alpha` 小于 1 时，也可以写成：

```math
E[N_{\mathrm{advance}}]\approx \frac{1-\alpha^{k+1}}{1-\alpha}
```

这个式子只用于直觉估算。真实系统里，接受概率不是独立同分布的，越靠后的 token 往往越难接受，并且还受 temperature、top-p、prompt 类型和 draft 模型质量影响。

#### 2. draft model 的速度

draft model 必须足够快。

如果 draft 本身很慢，即使 target 调用次数减少，总耗时也可能不降。

粗略加速比可以写成：

```math
S_{\mathrm{speedup}}\approx \frac{E[N_{\mathrm{advance}}]\cdot T_{\mathrm{target,one}}}{T_{\mathrm{target,verify}}(k)+k\cdot T_{\mathrm{draft,one}}+T_{\mathrm{overhead}}}
```

其中 `T_target,one` 是普通 decode 生成一个 token 的 target step 时间，`T_target,verify(k)` 是 target 一次验证 `k` 个候选 token 的时间，`T_draft,one` 是 draft 生成一个 token 的时间，`T_overhead` 包含采样、同步、调度和 KV 管理等额外开销。这个式子能解释为什么“接受率高”只是必要条件，不是充分条件。

#### 3. draft 长度 k

`k` 越大，理论上每轮最多接受更多 token。

但 `k` 太大也有问题：

1. draft 自己生成更久。
2. 后面 token 接受概率通常下降。
3. target 验证更长候选序列也有额外成本。
4. 拒绝后后续候选都作废。

#### 4. target 验证的并行效率

target 一次验证多个 token 要能高效利用 GPU。

如果候选长度太短或 batch 很小，收益有限。

#### 5. 任务分布

格式固定、低随机性的任务更容易高接受率。

开放创意写作、高温采样、多样性要求高时，draft 和 target 分布可能差异更大。

---

### 九、什么时候最有效

Speculative Decoding 通常更适合：

1. target model 很大、decode 单步很贵。
2. draft model 明显更小、更快。
3. draft 与 target 分布足够接近。
4. 输出较长，有足够多 decode step 可以摊销开销。
5. 采样温度不太高，候选更容易一致。
6. 系统能高效维护两个模型的 KV Cache。

例如代码补全、格式化回答、结构化输出、低温问答，可能有较高接受率。

它不太适合：

1. 输出很短，额外调度开销难摊销。
2. draft 模型与 target 差距太大。
3. 高温开放生成，分布差异大。
4. GPU 显存放不下额外 draft model。
5. 系统 bottleneck 不在 target decode，而在网络、排队或 tokenizer。

---

### 十、与 greedy decoding 的关系

如果使用 greedy decoding，问题会简单一些。

draft 先生成若干 token。

target 检查这些位置的 argmax 是否和 draft token 一致。

一致就接受，不一致就从 target argmax 开始修正。

这更容易理解，也更容易实现。

但真实线上常常需要 temperature、top-p 等采样。

采样场景下要保持 target 分布，就需要前面提到的接受-拒绝规则。

所以面试中要区分：

```text
greedy 下可以用匹配 argmax 的直觉解释；sampling 下需要用概率校正保证分布不变。
```

---

### 十一、与 KV Cache 和 batching 的关系

Speculative Decoding 会让系统状态更复杂。

普通 decode 只有 target model 的 KV Cache。

Speculative Decoding 至少可能需要：

1. target model 的 KV Cache。
2. draft model 的 KV Cache。
3. 候选 token 的临时状态。
4. 被拒绝后回滚或丢弃的状态。

如果多个请求同时 speculative decode，还要和 Continuous Batching 配合。

这会带来几个工程问题：

1. draft 和 target 的 batch 如何对齐。
2. 每个请求接受 token 数不同，batch 状态如何更新。
3. 拒绝后的候选 KV 是否需要丢弃。
4. target 验证阶段如何复用 KV Cache。
5. draft 模型占用额外显存是否值得。

所以 Speculative Decoding 是算法和系统的联合优化。

它不是在生成代码里加几行循环就能稳定上线。

---

### 十二、为什么小模型不能直接替代大模型

一个常见疑问是：

```text
既然 draft model 很快，为什么不直接用 draft model 输出？
```

原因是 draft model 质量通常不如 target model。

Speculative Decoding 的目标不是让小模型替代大模型。

它的目标是让小模型提出候选，再由大模型保证最终分布和质量。

可以类比成：

```text
draft model 是草稿作者，target model 是审稿人。
草稿写得好，审稿人很快通过；草稿不对，审稿人改回自己的版本。
```

所以它的质量上限仍然由 target model 决定。

这也是它相比直接蒸馏小模型更吸引人的地方。

---

### 十三、真实项目中的坑

#### 1. draft 模型太弱

如果 draft 和 target 分布差异大，接受率低。

系统多跑了一个 draft model，却没有减少多少 target step。

结果可能更慢。

#### 2. draft 模型太大

如果 draft 足够强但也很慢，额外成本抵消收益。

draft 需要在质量和速度之间平衡。

#### 3. 只看平均接受率

不同任务、不同 prompt、不同温度下接受率差异很大。

平均值可能掩盖长尾场景。

上线前要按业务类型分桶看接受率和加速比。

#### 4. 忽略显存成本

额外 draft model 会占显存。

如果因此降低可承载并发，整体吞吐未必提升。

#### 5. 采样实现不严格

如果接受-拒绝和修正分布实现错误，最终分布会偏离 target model。

可能出现质量退化、重复、格式异常或安全策略偏移。

#### 6. 与 serving 调度冲突

在高并发下，每个请求接受长度不同。

这会让 batch 更不规则。

如果调度器和 KV Cache 管理没有配合好，可能带来额外尾延迟。

---

### 十四、面试问答

#### 问题 1：Speculative Decoding 的核心思想是什么？

可以这样回答：

```text
它用一个更小更快的 draft model 先生成多个候选 token，再用 target model 一次 forward 并行验证这些候选。
如果候选被接受，一次 target forward 就能推进多个 token，从而减少大模型 decode step 数。
```

#### 问题 2：为什么 target model 可以一次验证多个 token？

可以这样回答：

```text
因为候选 token 已经由 draft model 给定。把 prompt 加候选序列一起输入 target model，Transformer 可以在 causal mask 下并行计算每个候选位置的 logits。
这类似训练时 teacher forcing 的并行计算，不违反自回归因果性。
```

#### 问题 3：Speculative Decoding 会改变 target model 的输出分布吗？

可以这样回答：

```text
严格的 speculative sampling 不应该改变。它用接受-拒绝采样和修正分布，使最终采样分布等价于直接从 target model 采样。
如果实现只是简单接受高概率 token，就可能改变分布。
```

#### 问题 4：加速比由哪些因素决定？

可以这样回答：

```text
主要取决于 draft 模型速度、draft 和 target 的分布接近程度、候选 token 接受率、draft 长度 k、target 批量验证效率以及任务输出长度。
接受率高且 draft 足够快时收益明显；接受率低或 draft 太慢时可能反而变慢。
```

#### 问题 5：Speculative Decoding 适合什么场景？

可以这样回答：

```text
适合 target model 很大、decode 昂贵、输出较长、draft 模型足够快且分布接近 target 的场景。低温问答、代码补全、格式较稳定的任务通常更容易获得较高接受率。
```

#### 问题 6：它和模型蒸馏有什么区别？

可以这样回答：

```text
蒸馏是直接训练或使用小模型替代大模型，质量上限受小模型影响。Speculative Decoding 中小模型只负责提案，最终接受和修正由 target model 决定，因此目标是保持大模型分布的同时加速。
```

---

### 十五、常见误区

1. 误区：Speculative Decoding 就是直接用小模型生成。
   纠正：小模型只负责 draft，最终分布由 target model 的验证和修正保证。

2. 误区：draft 长度越长越好。
   纠正：长度太长会增加 draft 成本，后面 token 接受率可能下降，被拒绝后也会浪费更多候选。

3. 误区：只要加一个 draft model 就一定更快。
   纠正：如果接受率低、draft 太慢或显存压力变大，可能没有收益甚至更慢。

4. 误区：Speculative Decoding 降低了单个 token 的模型计算质量。
   纠正：严格算法保持 target model 分布，不是用低质量模型替代高质量模型。

5. 误区：它只适用于 greedy decoding。
   纠正：greedy 更容易理解，但 speculative sampling 可以支持采样并保持分布，只是实现更复杂。

6. 误区：接受率高就一定线上效果好。
   纠正：还要看 draft 显存、batch 调度、P95/P99 延迟、不同任务分桶和系统瓶颈。

### 本讲总结

本讲最重要的结论：

1. Speculative Decoding 目标是减少 target model 的 decode step 数。
2. draft model 负责快速提出多个候选 token，target model 负责批量验证和修正。
3. 给定候选序列后，target model 可以在 causal mask 下并行计算多个位置的 logits。
4. 严格 speculative sampling 通过接受-拒绝和修正分布保持 target model 采样分布不变。
5. 加速效果取决于接受率、draft 速度、draft 长度、任务分布和 target 批量验证效率。
6. 它适合大模型、长输出、低温或格式稳定、draft 与 target 分布接近的场景。
7. 工程落地要考虑额外显存、双模型 KV Cache、batch 调度、拒绝回滚和尾延迟。
8. 面试中要强调它不是“小模型替代大模型”，而是“小模型提案、大模型验证”的无偏加速框架。

## 第 70 讲：Medusa、EAGLE 与多 token 预测

### 本讲目标

学完本讲，你应该能回答六个问题：

1. 多 token 预测为什么能加速自回归解码。
2. Medusa 的多头预测直觉是什么。
3. EAGLE 为什么选择在特征层面预测未来状态。
4. 多 token 预测和 Speculative Decoding 有什么关系。
5. tree attention、候选树和验证机制解决什么问题。
6. 面试中如何分析 Medusa/EAGLE 的收益、局限和落地难点。

上一讲讲了 Speculative Decoding。

它的典型形态是：

```text
额外准备一个 draft model，让它先生成多个候选 token，再由 target model 验证。
```

但这带来一个工程问题：

```text
还要部署、调度和维护一个 draft model。
```

Medusa、EAGLE 等方法尝试走另一条路线：

```text
不一定引入完整小模型，而是在 target model 附近增加轻量预测模块，让一次 forward 产生多个未来 token 候选。
```

它们都属于多 token 预测或 speculative decoding 的扩展方向。

---

### 一、为什么自回归解码天然慢

标准自回归生成每次只输出一个 token。

流程是：

```text
forward -> token t+1
forward -> token t+2
forward -> token t+3
...
```

每个 token 都依赖前一个 token。

这导致解码长度为 `O` 时，大模型至少要做 `O` 次 decode forward。

如果能让一次 forward 推进多个 token，就有机会降低 TPOT。

多 token 预测的共同目标是：

```text
一次模型调用，不只预测下一个 token，而是提出未来多个 token 的候选。
```

但困难在于：

```text
未来 token 之间有条件依赖，不能把它们当成完全独立的分类任务。
```

所以这些方法通常还需要验证机制，确保质量不明显下降，或者尽量保持原模型分布。

---

### 二、从 Speculative Decoding 到多 token 预测

Speculative Decoding 的基本模式是：

```text
proposal -> verification
```

proposal 阶段提出候选 token。

verification 阶段由 target model 验证候选。

传统 speculative decoding 用一个外部 draft model 做 proposal。

Medusa/EAGLE 的区别是 proposal 来源不同。

可以粗略对比：

```text
方法                    proposal 来源
Speculative Decoding     独立 draft model
Medusa                   target hidden state 上的多个预测头
EAGLE                    特征层面的未来状态预测模块
```

它们共同追求：

```text
减少 target model 串行 decode 次数，让一次验证接受多个 token。
```

---

### 三、Medusa 的核心直觉

Medusa 的名字来自多头。

它的核心想法是：

```text
在原始 LLM 顶部增加多个轻量 prediction heads，让不同 head 分别预测未来不同位置的 token。
```

普通 LM head 预测的是：

```text
第 1 个未来 token：x_{t+1}
```

Medusa heads 可以预测：

```text
head 1 -> x_{t+1}
head 2 -> x_{t+2}
head 3 -> x_{t+3}
head 4 -> x_{t+4}
```

如果当前最后一层 hidden state 记为 `h_t`，第 `i` 个 Medusa head 可以写成一个轻量投影：

```math
z_i=W_i h_t+b_i,
\qquad
q_i(v)=\mathrm{softmax}(z_i)_v
```

其中 `i` 表示预测第 `i` 个未来位置，`q_i(v)` 是第 `i` 个 head 对候选 token `v` 的 proposal 概率。这里的 `q_i` 是提案分布，不等同于原模型在真实自回归上下文下的 target 分布。

当然，这只是直觉。

真实生成时，未来 token 不是彼此独立的。

`x_{t+3}` 应该依赖 `x_{t+1}` 和 `x_{t+2}`。

所以 Medusa 不会简单把每个 head 的 top-1 拼起来直接输出。

它通常会从多个 head 产生候选组合，形成一棵候选树，再用原模型验证。

一句话：

```text
Medusa 用多个轻量头从当前 hidden state 提出未来 token 候选，再通过树状验证接受其中一段。
```

---

### 四、为什么不能直接相信多头输出

假设当前上下文是：

```text
我 今天 想 去
```

Medusa heads 预测：

```text
head 1：公园
head 2：吃饭
head 3：天气
```

直接拼起来可能变成：

```text
我今天想去 公园 吃饭 天气
```

这不一定通顺。

原因是 head 2、head 3 的预测不一定充分条件化在 head 1 的具体选择上。

未来 token 的合理性依赖前面已经生成的 token。

所以多 token 预测需要验证。

验证的目标是：

```text
确认这些候选 token 在原模型自回归条件下是否仍然合理。
```

这和 Speculative Decoding 中 target model 验证 draft token 是同一个思想。

---

### 五、候选树与 tree attention

Medusa 通常不是只生成一条候选路径。

它会从多个 head 的 top candidates 组合出多条候选路径。

例如：

```text
第 1 步候选：A, B
第 2 步候选：C, D
第 3 步候选：E, F
```

可以形成多条路径：

```text
A C E
A C F
A D E
B C E
...
```

这些路径组成一棵候选树。

如果第 `i` 个 head 保留 `m_i` 个候选，朴素组合出的候选路径数是：

```math
N_{\mathrm{path}}=\prod_{i=1}^{K}m_i
```

如果每个位置都保留 `m` 个候选，就变成：

```math
N_{\mathrm{path}}=m^K
```

这说明候选数增长很快，不能无脑扩大每个 head 的 top-k。候选树越大，覆盖正确路径的概率可能越高，但验证、显存和调度成本也会增长。

如果逐条路径验证，会浪费很多计算。

Tree attention 的目标是：

```text
把候选树中的多个分支组织成一次或少数几次并行验证，复用共享前缀计算。
```

例如两条候选路径共享前缀 `A C`，验证时不应该重复计算共享部分。

这和前面讲 PagedAttention、prefix cache、beam search 共享前缀有类似直觉。

核心思想是：

```text
候选不再是一条线，而是一棵树；验证时要利用树结构复用计算。
```

下面这个 demo 用纯 Python 模拟候选树验证。`HEAD_CANDIDATES` 表示 3 个 Medusa heads 各自保留两个候选；`TARGET_NEXT` 表示原模型在不同前缀下会接受的下一个 token。demo 展示候选路径数、逐条验证的 token 检查数、候选树共享前缀后的唯一节点数，以及最佳可接受前缀。

```python
from itertools import product


HEAD_CANDIDATES = [
    ["A", "B"],
    ["C", "D"],
    ["E", "F"],
]

TARGET_NEXT = {
    (): "A",
    ("A",): "C",
    ("A", "C"): "E",
    ("A", "D"): "G",
    ("B",): "D",
    ("B", "C"): "H",
    ("B", "D"): "F",
}


def build_paths(head_candidates: list[list[str]]) -> list[tuple[str, ...]]:
    return list(product(*head_candidates))


def unique_prefix_count(paths: list[tuple[str, ...]]) -> int:
    prefixes = set()
    for path in paths:
        for depth in range(1, len(path) + 1):
            prefixes.add(path[:depth])
    return len(prefixes)


def accepted_depth(path: tuple[str, ...]) -> int:
    for pos, token in enumerate(path):
        prefix = path[:pos]
        if TARGET_NEXT.get(prefix) != token:
            return pos
    return len(path)


paths = build_paths(HEAD_CANDIDATES)
ranked = sorted(((accepted_depth(path), path) for path in paths), reverse=True)
best_depth, best_path = ranked[0]

naive_checks = len(paths) * len(HEAD_CANDIDATES)
tree_nodes = unique_prefix_count(paths)

print("num_paths=", len(paths))
print("naive_token_checks=", naive_checks)
print("tree_unique_prefix_nodes=", tree_nodes)
print("best_path=", best_path)
print("accepted_depth=", best_depth)
print("accepted_prefix=", best_path[:best_depth])

for depth, path in ranked[:4]:
    print("candidate=", path, "accepted=", depth)

baseline_ms = max(best_depth, 1) * 12.0
speculative_ms = 15.0 + 1.5 + 0.5
print("rough_speedup=", round(baseline_ms / speculative_ms, 2))
```

典型输出类似：

```text
num_paths= 8
naive_token_checks= 24
tree_unique_prefix_nodes= 14
best_path= ('A', 'C', 'E')
accepted_depth= 3
accepted_prefix= ('A', 'C', 'E')
candidate= ('A', 'C', 'E') accepted= 3
candidate= ('A', 'C', 'F') accepted= 2
candidate= ('A', 'D', 'F') accepted= 1
candidate= ('A', 'D', 'E') accepted= 1
rough_speedup= 2.12
```

这个 toy demo 只模拟“候选树 + 验证 + 接受前缀”的状态机。真实 Medusa 需要 tree attention mask、位置编码、KV Cache 写入/回滚和采样策略配合；但从这个例子能看出，tree attention 的核心价值是减少重复验证共享前缀，而不是改变原模型的自回归条件依赖。

### 六、Medusa 的训练方式

Medusa 通常会冻结原始 LLM 主干，只训练新增的 Medusa heads。

这样做的好处是：

1. 不改变原模型主体能力。
2. 训练成本相对低。
3. 可以把 Medusa 当成推理加速插件。
4. 原模型权重不用大规模重训。

训练目标可以理解为：

```text
给定当前位置 hidden state，让第 i 个 head 预测第 i 个未来 token。
```

例如：

```text
head 1 预测 x_{t+1}
head 2 预测 x_{t+2}
head 3 预测 x_{t+3}
```

如果训练序列是 `x_1,...,x_T`，第 `i` 个 head 的监督目标是第 `t+i` 个 token。一个简化训练损失可以写成：

```math
L_{\mathrm{Medusa}}=-\frac{1}{K}\sum_{i=1}^{K}\log q_i(x_{t+i})
```

如果对多个位置和多个样本求平均，就是把上式再对 batch 和时间位置求平均。这里 `K` 是 Medusa heads 数，`q_i` 是第 `i` 个 head 的 proposal 分布。Medusa-1 常见做法是冻结 backbone，只训练新增 heads；Medusa-2 会联合微调 backbone 和 heads，可能提高接受率，但也更依赖训练 recipe 来保护原模型能力。

它类似给模型加了多个未来 token 辅助预测任务。

但要注意：

```text
训练 heads 能提高 proposal 质量，最终生成质量仍依赖验证机制。
```

---

### 七、EAGLE 的核心直觉

EAGLE 也是为了提高 speculative decoding 的 proposal 质量。

但它不只是直接在 token 概率上预测未来 token。

更核心的直觉是：

```text
在特征层面预测未来 hidden states，再由 LM head 得到未来 token 分布。
```

为什么要在特征层面做？

因为 token 空间很离散，直接预测多个未来 token 很难。

hidden state 是连续表示，包含语义、语法和上下文信息。

如果能预测未来一步或多步的特征状态，再映射到 token 分布，可能比直接猜 token 更稳定。

可以粗略理解为：

```text
Medusa 更像“从当前 hidden state 直接多头猜未来 token”。
EAGLE 更像“先预测未来 hidden state，再从未来 hidden state 产生 token 候选”。
```

这两者都是 proposal 模型，只是 proposal 的建模空间不同。

---

### 八、EAGLE 为什么可能更准

直接预测未来第 2、第 3 个 token 有一个问题：

```text
未来 token 依赖前一个 token 的选择。
```

如果只从当前 hidden state 出发，越往后的 token 越难预测。

EAGLE 类方法尝试利用模型内部特征演化信息。

它希望学习：

```text
当前 hidden state + 候选 token -> 下一步 hidden state 的近似
```

可以把这个思想写成一个简化形式：

```math
\hat h_{t+1}=g_{\phi}(h_t,e(y_t)),
\qquad
q(v)=\mathrm{softmax}(W_{\mathrm{lm}}\hat h_{t+1})_v
```

其中 `h_t` 是当前特征，`e(y_t)` 是已知 token 的 embedding，`g_phi` 是轻量特征预测模块，`W_lm` 是复用或对齐到 LM head 的输出投影。训练时可以用原模型的下一步特征 `h_{t+1}` 做监督：

```math
L_{\mathrm{feat}}=\left\|\hat h_{t+1}-h_{t+1}\right\|_2^2
```

这不是 EAGLE 论文所有实现细节的完整复刻，而是帮助你理解“在 feature space 里做 proposal”的数学骨架。真实方法还要处理 feature uncertainty、token 序列前移、采样、验证和硬件开销。

这样可以更接近原模型逐步 decode 时的内部状态变化。

如果未来 hidden state 预测得更准，proposal token 的接受率就可能更高。

接受率越高，speculative decoding 的加速越明显。

所以 EAGLE 的目标可以理解为：

```text
提高 draft proposal 的质量，同时比完整 draft model 更轻。
```

---

### 九、与普通 Speculative Decoding 的关系

Medusa/EAGLE 不应该被理解成完全独立的另一个世界。

它们本质上仍可以放进 speculative decoding 框架：

```text
proposal module 提出多个候选 token
target model 验证候选 token
接受一段候选，拒绝时回退或修正
```

区别主要在 proposal module：

1. 普通 speculative decoding：proposal 是独立 draft model。
2. Medusa：proposal 是多个轻量 token prediction heads。
3. EAGLE：proposal 是特征预测模块加 token 映射。

它们都要面对同一个问题：

```text
proposal 越准，接受率越高，加速越明显；proposal 越贵，额外开销越大。
```

这是核心 trade-off。

---

### 十、质量、速度和显存的 trade-off

多 token 预测方法不是越复杂越好。

它们要在三件事之间平衡。

#### 1. Proposal 质量

候选越接近 target model，接受率越高。

接受率高，target forward 平均推进 token 数更多。

可以用平均接受长度衡量 proposal 质量：

```math
\bar A=\frac{1}{N}\sum_{j=1}^{N}a_j
```

其中 `a_j` 是第 `j` 个请求或第 `j` 轮 speculative step 实际接受的 token 数，`N` 是统计样本数。线上评估时还应分任务、长度、temperature 和业务场景统计，而不是只看一个全局平均值。

#### 2. Proposal 成本

新增 heads 或特征预测模块越重，额外计算越多。

如果 proposal 成本太高，会抵消加速收益。

粗略延迟收益可以写成：

```math
S_{\mathrm{speedup}}\approx \frac{\bar A\cdot T_{\mathrm{base}}}{T_{\mathrm{verify}}+T_{\mathrm{proposal}}+T_{\mathrm{tree}}+T_{\mathrm{overhead}}}
```

其中 `T_base` 是普通 decode 一个 token 的 target step 时间，`T_verify` 是一次验证候选树的 target 时间，`T_proposal` 是 heads 或特征模块产生候选的时间，`T_tree` 是 tree attention 相关开销，`T_overhead` 是采样、调度、KV 管理和同步开销。这个式子解释了为什么候选接受数增加不一定等于端到端加速。

#### 3. 系统复杂度

候选树、tree attention、验证、KV Cache 更新、batch 调度都会增加实现复杂度。

如果线上服务已经被调度、显存或网络限制，算法加速未必能转化成端到端收益。

所以面试中可以总结为：

```text
多 token 预测的核心不是“预测越多越好”，而是“便宜地产生足够准的候选，并且高效验证”。
```

---

### 十一、和多 token 训练目标的关系

有些模型训练时会加入 multi-token prediction objective。

也就是除了预测下一个 token，还预测未来多个 token。

这和 Medusa 的直觉有联系：

```text
都希望模型表示中包含对未来多个 token 的信息。
```

但要区分两件事：

1. 训练目标层面的 multi-token prediction：用于提升表示学习或训练效率。
2. 推理加速层面的 multi-token prediction：用于一次提出多个候选 token，加速 decode。

二者相关，但不完全相同。

推理加速更关注：

```text
接受率、验证成本、延迟、显存、调度复杂度。
```

训练目标更关注：

```text
表示质量、优化信号、预训练效率和最终模型能力。
```

---

### 十二、真实项目中的坑

#### 1. 只看理论 token 接受数

论文中可能报告平均每步接受多个 token。

但线上还要看端到端延迟。

候选生成、tree attention、验证、KV 更新和调度都有成本。

#### 2. 候选树太大

候选树越大，覆盖率可能越高。

但验证成本和内存开销也越大。

树太大可能让系统变慢。

#### 3. 不同任务接受率差异大

代码、数学、闲聊、创意写作的接受率可能完全不同。

不能只用一个平均值判断上线收益。

#### 4. 与采样策略耦合

temperature、top-p、top-k 会影响候选分布和接受率。

低温下更容易接受，高温下候选分歧更大。

#### 5. 训练 heads 的数据分布不匹配

如果 Medusa heads 在一种数据分布上训练，却用于另一种业务，proposal 质量可能下降。

#### 6. 工程集成复杂

新增 heads、特征预测模块、tree attention kernel、KV Cache 更新和 continuous batching 都要适配。

如果只在离线脚本里有效，不代表线上 serving 有收益。

### 十三、面试问答

#### 问题 1：Medusa 的核心思想是什么？

可以这样回答：

```text
Medusa 在原 LLM 顶部增加多个轻量 prediction heads，让它们从当前 hidden state 预测未来多个 token 的候选。
这些候选通常组成候选树，再由原模型验证并接受其中一段，从而减少串行 decode 次数。
```

#### 问题 2：为什么 Medusa 不能直接把多个 head 的输出拼起来？

可以这样回答：

```text
因为未来 token 之间有自回归依赖。第 2、第 3 个未来 token 应该条件化在前面实际接受的 token 上。
多个 head 从当前状态独立预测未来位置，可能不一致，所以需要候选树和 target model 验证。
```

#### 问题 3：EAGLE 和 Medusa 的区别是什么？

可以这样回答：

```text
Medusa 更像在 token 层面用多个 heads 直接预测未来 token；EAGLE 更强调在特征层面预测未来 hidden states，再映射到 token 候选。
EAGLE 希望通过更接近模型内部状态演化的 proposal，提高接受率。
```

#### 问题 4：Medusa/EAGLE 和 Speculative Decoding 是什么关系？

可以这样回答：

```text
它们都可以看成 speculative decoding 框架下的 proposal 方法。普通 speculative decoding 用独立 draft model 提案；Medusa 用额外 heads 提案；EAGLE 用特征预测模块提案；最终仍需要 target model 验证。
```

#### 问题 5：多 token 预测的加速收益由什么决定？

可以这样回答：

```text
主要由候选接受率、proposal 模块成本、候选长度或树大小、target 验证效率、任务分布和系统集成成本决定。不是预测越多越好，而是要便宜地产生高质量候选。
```

#### 问题 6：线上落地 Medusa/EAGLE 要注意什么？

可以这样回答：

```text
要按任务分布评估接受率、TPOT、TTFT、P95/P99 和显存开销，还要确认 tree attention、KV Cache 更新、batch 调度和采样策略都正确。离线加速不等于线上端到端收益。
```

---

### 十四、常见误区

1. 误区：多 token 预测就是一次直接输出多个 token。
   纠正：通常还需要候选验证，否则容易破坏自回归条件依赖和输出质量。

2. 误区：Medusa heads 预测的 token 可以无条件全部接受。
   纠正：heads 只是 proposal，仍需要原模型验证。

3. 误区：EAGLE 和 Medusa 完全无关。
   纠正：它们都属于提高 speculative proposal 质量的多 token 解码方向，只是建模空间不同。

4. 误区：候选树越大越好。
   纠正：候选树大可能提高覆盖率，但也增加验证、显存和调度开销。

5. 误区：论文加速比可以直接等于线上收益。
   纠正：线上还受 batching、KV Cache、网络、流式返回、尾延迟和业务长度分布影响。

6. 误区：多 token 预测只看接受率。
   纠正：还要看 proposal 成本、验证成本、系统复杂度和质量稳定性。

### 本讲总结

本讲最重要的结论：

1. 多 token 预测试图让一次推理提出多个未来 token 候选，从而减少串行 decode 次数。
2. Medusa 在原模型顶部增加多个轻量 heads，从当前 hidden state 预测未来 token 候选。
3. Medusa 通常需要候选树和 target model 验证，不能直接拼接多个 head 的输出。
4. EAGLE 更强调在特征层面预测未来状态，再产生 token 候选，以提高 proposal 质量。
5. Medusa/EAGLE 都可以看作 speculative decoding 框架中的 proposal 方法。
6. 加速收益取决于候选接受率、proposal 成本、候选树大小、验证效率、任务分布和工程实现。
7. 线上落地要同时看质量、TPOT、TTFT、P95/P99、显存和调度复杂度。
8. 面试中要强调“便宜地产生高质量候选，并由原模型验证”，而不是简单说“一次生成多个 token”。

## 第 71 讲：量化感知训练与后训练量化

### 本讲目标

学完本讲，你应该能回答六个问题：

1. 大模型推理为什么需要量化。
2. INT8、INT4、weight-only quantization 分别在解决什么问题。
3. scale、zero-point、per-tensor、per-channel、group-wise quantization 是什么。
4. PTQ 和 QAT 的核心区别是什么。
5. GPTQ、AWQ 这类方法的直觉是什么。
6. 量化模型上线前应该如何评估质量、性能和风险。

前几讲我们讨论了 decode 加速。

这一讲进入另一个部署核心主题：量化。

量化的目标很直接：

```text
用更低 bit 的数表示模型权重、激活或 KV Cache，从而降低显存、带宽和成本。
```

但量化不是简单把 FP16 改成 INT4。

真正困难的是：

```text
如何在显著压缩模型的同时，尽量不损失质量，并且真的在硬件上跑得更快。
```

---

### 一、为什么大模型需要量化

一个 7B 模型，如果用 FP16 存权重：

```text
7B * 2 bytes ≈ 14 GB
```

一个 70B 模型：

```text
70B * 2 bytes ≈ 140 GB
```

这还只是权重，不包括 KV Cache、临时 buffer、通信 buffer 和框架开销。

如果把权重量化到 INT8：

```text
每个参数 1 byte，权重显存约减半。
```

如果量化到 INT4：

```text
每个参数 0.5 byte，权重显存约变成 FP16 的四分之一。
```

量化带来的收益包括：

1. 降低权重显存。
2. 降低显存带宽压力。
3. 允许更大模型放到同一张卡上。
4. 提高 batch 或并发空间。
5. 降低部署成本。
6. 在合适 kernel 和硬件上提升吞吐。

但量化也有代价：

1. 数值误差增加。
2. 模型质量可能下降。
3. 某些层或 outlier 特别敏感。
4. 需要硬件和 kernel 支持。
5. 评估和 debug 更复杂。

---

### 二、量化的基本公式

量化的核心是把浮点数映射到整数。

常见形式是：

```math
q=\mathrm{round}\left(\frac{x}{s}\right)+z
```

反量化时：

```math
\hat x=s\cdot(q-z)
```

其中：

```text
x：原始浮点值
q：量化后的整数值
s：缩放系数，也就是 scale
z：零点，也就是 zero-point，用于处理非对称范围
x_hat：反量化后的近似浮点值
```

如果已知浮点范围 `[x_min, x_max]` 和整数范围 `[q_min, q_max]`，非对称 affine quantization 常用：

```math
s=\frac{x_{\max}-x_{\min}}{q_{\max}-q_{\min}}
```

```math
z=\mathrm{round}\left(q_{\min}-\frac{x_{\min}}{s}\right)
```

实际实现还会把 `q` clamp 到整数范围内：

```math
q_{\mathrm{clip}}=\min\left(q_{\max},\max(q_{\min},q)\right)
```

如果是对称量化，通常没有 zero-point 或 zero-point 为 0：

```math
q=\mathrm{round}\left(\frac{x}{s}\right),
\qquad
\hat x=s\cdot q
```

对 `b` bit signed symmetric quantization，整数范围常写成：

```math
q_{\max}=2^{b-1}-1,
\qquad
q_{\min}=-q_{\max}
```

对应 scale 可以取：

```math
s=\frac{\max_i |x_i|}{q_{\max}}
```

直觉是：

```text
用有限个整数刻度，近似连续的浮点数范围。
```

量化误差来自：

```text
x 和 x_hat 之间的差距。
```

常见误差指标包括平均绝对误差和均方根误差：

```math
\mathrm{MAE}=\frac{1}{n}\sum_{i=1}^{n}|x_i-\hat x_i|
```

```math
\mathrm{RMSE}=\sqrt{\frac{1}{n}\sum_{i=1}^{n}(x_i-\hat x_i)^2}
```

---

### 三、per-tensor、per-channel 与 group-wise

scale 可以按不同粒度设置。

#### 1. Per-tensor quantization

整个 tensor 共用一个 scale。

优点是实现简单、元数据少。

缺点是如果 tensor 内不同通道数值范围差异很大，一个 scale 很难兼顾所有值。

#### 2. Per-channel quantization

每个输出通道或每个列/行使用不同 scale。

优点是更精细，误差更小。

缺点是 scale 元数据更多，kernel 实现更复杂。

#### 3. Group-wise quantization

把权重按 group 分组，每组一个 scale。

例如每 128 个权重一个 group。

它是 per-tensor 和 per-channel 之间的折中。

很多 LLM INT4 weight-only 量化会使用 group-wise scale。

如果把一个向量按 group size `G` 分成若干组，第 `g` 组的 scale 可以写成：

```math
s_g=\frac{\max_{i\in\mathcal{G}_g}|w_i|}{q_{\max}}
```

其中 `G_g` 是第 `g` 个 group 的下标集合。group 越小，scale 越能贴近局部数值范围，但 scale 元数据越多。

可以总结为：

```text
粒度越细，量化误差通常越小，但元数据和 kernel 复杂度越高。
```

---

### 四、weight-only quantization

LLM 推理中很常见的是 weight-only quantization。

它只量化权重，激活仍用 FP16/BF16。

例如：

```text
W：INT4 或 INT8
x：FP16
计算时：把 W 反量化或在 kernel 内完成低精度乘法，再得到 FP16 输出
```

为什么 weight-only 常见？

因为 Decode 阶段常常受权重读取带宽影响。

每生成一个 token，都要读大量权重。

把权重压小，可以降低带宽压力和显存占用。

同时激活保持 FP16，可以减少激活量化带来的质量风险。

缺点是：

1. 速度收益依赖高效 kernel。
2. 反量化本身有开销。
3. INT4 权重可能带来质量下降。
4. 不是所有硬件都能高效支持。

面试时可以说：

```text
weight-only quantization 是 LLM 推理中常见折中：压缩最大头的权重显存和带宽，同时避免激活量化带来的更大不稳定性。
```

下面这个 demo 对比 per-tensor INT4 和 group-wise INT4。矩阵里故意放了一个 outlier，你会看到 per-tensor scale 被 outlier 拉大，导致普通小权重的误差变大；group-wise 能降低误差，但要额外存每个 group 的 scale。

```python
from math import sqrt


W = [
    [0.12, -0.08, 0.20, 0.05],
    [0.10, -0.07, 0.18, 0.06],
    [0.11, -0.09, 2.40, 0.04],  # 一个 outlier 会拉大 per-tensor scale
    [0.09, -0.06, 0.17, 0.03],
]
X = [
    [1.0, 0.5, -0.2, 0.1],
    [0.2, -1.0, 0.3, 0.7],
    [-0.4, 0.1, 1.2, -0.5],
]


def flatten(matrix):
    return [v for row in matrix for v in row]


def unflatten(values, rows, cols):
    return [values[i * cols:(i + 1) * cols] for i in range(rows)]


def quantize_symmetric(values, bits=4):
    qmax = 2 ** (bits - 1) - 1
    max_abs = max(abs(v) for v in values) or 1.0
    scale = max_abs / qmax
    q = [max(-qmax, min(qmax, round(v / scale))) for v in values]
    deq = [scale * qi for qi in q]
    return q, deq, scale


def groupwise_quantize(values, group_size, bits=4):
    q_all, deq_all, scales = [], [], []
    for start in range(0, len(values), group_size):
        q, deq, scale = quantize_symmetric(values[start:start + group_size], bits)
        q_all.extend(q)
        deq_all.extend(deq)
        scales.append(round(scale, 5))
    return q_all, deq_all, scales


def matmul(a, b):
    rows, inner, cols = len(a), len(b), len(b[0])
    return [[sum(a[i][k] * b[k][j] for k in range(inner)) for j in range(cols)] for i in range(rows)]


def rmse(a, b):
    diffs = [a[i][j] - b[i][j] for i in range(len(a)) for j in range(len(a[0]))]
    return sqrt(sum(d * d for d in diffs) / len(diffs))


rows, cols = len(W), len(W[0])
full_output = matmul(X, W)
flat = flatten(W)

_, tensor_deq, tensor_scale = quantize_symmetric(flat, bits=4)
W_tensor = unflatten(tensor_deq, rows, cols)
out_tensor = matmul(X, W_tensor)

_, group_deq, group_scales = groupwise_quantize(flat, group_size=4, bits=4)
W_group = unflatten(group_deq, rows, cols)
out_group = matmul(X, W_group)

fp16_bytes = len(flat) * 2
int4_bytes = len(flat) * 0.5
metadata_bytes = len(group_scales) * 2

print("per_tensor_scale=", round(tensor_scale, 5))
print("group_scales=", group_scales)
print("per_tensor_weight_rmse=", round(rmse(W, W_tensor), 5))
print("groupwise_weight_rmse=", round(rmse(W, W_group), 5))
print("per_tensor_output_rmse=", round(rmse(full_output, out_tensor), 5))
print("groupwise_output_rmse=", round(rmse(full_output, out_group), 5))
print("fp16_weight_bytes=", fp16_bytes)
print("int4_payload_bytes=", int4_bytes)
print("group_scale_metadata_bytes=", metadata_bytes)
```

典型输出类似：

```text
per_tensor_scale= 0.34286
group_scales= [0.02857, 0.02571, 0.34286, 0.02429]
per_tensor_weight_rmse= 0.09748
groupwise_weight_rmse= 0.0373
per_tensor_output_rmse= 0.11418
groupwise_output_rmse= 0.0538
fp16_weight_bytes= 32
int4_payload_bytes= 8.0
group_scale_metadata_bytes= 8
```

这个例子说明：更细粒度的 scale 可以降低误差，但元数据和 kernel 复杂度也会增加。真实 LLM 部署还要考虑权重打包格式、group size、反量化融合、outlier 保护和硬件指令支持。

### 五、激活量化为什么更难

激活是输入相关的。

不同 batch、不同 token、不同 prompt 下激活分布会变化。

权重是固定的，离线量化后不变。

激活则在推理时动态产生。

激活量化难点包括：

1. 分布随输入变化。
2. outlier 可能很大。
3. 不同层敏感度不同。
4. 校准数据不一定覆盖线上分布。
5. 激活量化错误会逐层传播。

所以许多部署方案先做权重量化，再谨慎考虑 activation quantization。

如果做 weight + activation quantization，通常需要更充分校准，甚至需要 QAT。

---

### 六、PTQ：后训练量化

PTQ 是 Post-Training Quantization。

它的意思是：

```text
模型训练完成后，不重新训练或只做少量校准，就把模型量化。
```

PTQ 的优点：

1. 成本低。
2. 流程简单。
3. 不需要大规模训练数据。
4. 适合已有 checkpoint 快速部署。

PTQ 的缺点：

1. 低 bit 下质量可能下降。
2. 对校准数据敏感。
3. 难处理 outlier 和敏感层。
4. 极低精度时可能不稳定。

常见 PTQ 流程：

```text
准备校准数据
-> 跑一小批 forward 收集统计量
-> 计算 scale / zero-point 或其他量化参数
-> 量化权重或激活
-> 在评估集上验证质量和性能
```

PTQ 适合快速试验和部署迭代。

---

### 七、QAT：量化感知训练

QAT 是 Quantization-Aware Training。

它的意思是：

```text
训练或微调时模拟量化误差，让模型提前适应低精度表示。
```

QAT 通常会在 forward 中插入 fake quantization。

也就是：

```text
浮点权重/激活 -> 模拟量化 -> 反量化回浮点 -> 继续计算
```

这样模型在训练中能感受到量化噪声。

反向传播时，由于 round 不可导，通常用 STE：Straight-Through Estimator。

直觉是：

```text
前向按量化效果走，反向近似当作可导。
```

fake quantization 可以写成：

```math
\tilde x=s\cdot\left(\mathrm{round}\left(\frac{x}{s}\right)-z\right)
```

因为 `round` 的真实导数几乎处处为 0，STE 通常在反向中近似使用：

```math
\frac{\partial \tilde x}{\partial x}\approx 1
```

这不是严格数学等价，而是训练工程中的近似，让梯度能穿过 fake quantization 节点。

QAT 的优点：

1. 低 bit 下质量通常比 PTQ 更稳。
2. 模型能适应量化误差。
3. 对激活量化更有帮助。
4. 更适合极低精度或高质量要求场景。

QAT 的缺点：

1. 需要训练资源。
2. 流程复杂。
3. 需要训练数据。
4. 超参数和量化配置更难调。
5. 可能影响原模型行为，需要重新评估。

---

### 八、PTQ 和 QAT 的核心区别

可以用一张表总结：

```text
维度          PTQ                         QAT
时机          训练后                       训练或微调中
成本          低                           高
是否需要训练   通常不需要或很少               需要
质量          高 bit 较稳，低 bit 风险大       低 bit 更稳
适合场景       快速部署、已有模型压缩           极低精度、高质量要求、激活量化
复杂度         较低                         较高
```

一句话：

```text
PTQ 是训练后补救，QAT 是训练中适应。
```

---

### 九、GPTQ 的直觉

GPTQ 是一种常见的 LLM 后训练权重量化方法。

它的核心直觉是：

```text
量化一个权重时，不只看这个权重本身的误差，还要考虑它对层输出的影响，并用近似二阶信息做误差补偿。
```

如果把线性层写成：

```math
Y=XW
```

量化 `W` 后，希望：

```math
X\hat W\approx XW
```

更明确地说，GPTQ 类方法关心的是校准激活加权后的输出重构误差：

```math
L_{\mathrm{GPTQ}}=\left\|XW-X\hat W\right\|_F^2
```

其中 `X` 是校准数据收集到的输入激活，`W` 是原始权重，`W_hat` 是量化后的权重，`||.||_F` 是矩阵 Frobenius norm。近似二阶信息的作用是估计哪些权重量化误差会更强地影响输出，并把已量化权重造成的误差补偿到后续权重上。

也就是说，重要的不是每个权重数值都完全接近，而是层输出尽量接近。

GPTQ 会利用校准数据中的激活 `X` 信息，按一定顺序量化权重，并把误差补偿到剩余未量化权重上。

面试中不用推导矩阵公式。

需要说清：

```text
GPTQ 是 PTQ 方法，利用校准激活和近似二阶信息，尽量让量化后线性层输出接近原输出。
```

---

### 十、AWQ 的直觉

AWQ 是 Activation-aware Weight Quantization。

它的核心直觉是：

```text
不是所有权重同等重要。那些与大激活相乘、对输出影响更大的权重更需要保护。
```

AWQ 会利用激活统计找到重要权重或重要通道。

一个简单的激活敏感度分数可以写成：

```math
r_j=\mathbb{E}_{x\sim\mathcal{D}_{\mathrm{calib}}}|x_j|
```

其中 `r_j` 是第 `j` 个输入通道在校准数据上的平均激活幅度。通道激活越大，对应权重列的误差越容易放大到输出里。因此 AWQ 会优先保护这些 salient channels。

然后通过缩放或保护策略，让这些重要部分在量化后误差更小。

可以粗略理解为：

```text
GPTQ 更强调误差补偿和近似二阶信息。
AWQ 更强调激活感知，保护对输出更关键的权重。
```

二者都属于 PTQ 方向，都常用于 LLM weight-only quantization。

---

### 十一、哪些层更敏感

LLM 中不同层对量化敏感度不同。

常见敏感位置包括：

1. embedding 层。
2. lm head。
3. attention output projection。
4. MLP down projection。
5. norm 附近的数值路径。
6. 有明显 outlier 的层或通道。

有些部署方案会混合精度：

```text
大部分线性层 INT4，少数敏感层保留 FP16 或 INT8。
```

这叫 mixed-precision quantization 的直觉。

它牺牲一点压缩率，换取质量稳定。

---

### 十二、量化不一定等于加速

很多人以为 INT4 一定比 FP16 快 4 倍。

这是错误的。

量化能否加速取决于：

1. 硬件是否支持低精度高效计算。
2. kernel 是否优化得好。
3. 是否有反量化开销。
4. 当前瓶颈是显存带宽还是计算。
5. batch size 和序列长度。
6. 是否引入额外数据搬运。

如果系统本来不是权重带宽瓶颈，量化收益可能很小。

如果 INT4 kernel 不成熟，甚至可能比 FP16 慢。

所以面试中要强调：

```text
量化首先确定能省显存，但是否加速要看硬件、kernel 和瓶颈位置。
```

---

### 十三、量化模型怎么评估

量化上线前不能只看模型能不能加载。

至少要评估四类指标。

#### 1. 质量指标

包括：

1. perplexity。
2. 通用 benchmark。
3. 代码、数学、推理、长上下文专项评估。
4. 指令遵循。
5. 事实性和安全性。
6. 业务真实样本人工评估。

#### 2. 格式和稳定性

量化可能导致：

1. JSON 格式更容易坏。
2. 代码括号不匹配。
3. 多轮对话角色混乱。
4. 重复输出增加。
5. 长文本后半段退化。

这些问题不一定在平均 benchmark 中明显。

#### 3. 性能指标

包括：

1. 权重显存。
2. 峰值显存。
3. TTFT。
4. TPOT。
5. tokens/s。
6. P95/P99 延迟。
7. 最大并发。

#### 4. 回归和分桶

要按任务类型分桶看：

1. 短问答。
2. 长上下文总结。
3. 代码生成。
4. 数学推理。
5. RAG 问答。
6. 工具调用。
7. 安全拒答。

量化可能对某些类别影响很小，对另一些类别影响明显。

---

### 十四、真实项目中的坑

#### 1. 校准数据不匹配

PTQ 依赖校准数据。

如果校准数据都是短问答，线上却大量长代码生成，量化参数可能不适合。

#### 2. 只看 perplexity

perplexity 变化小不代表用户体验不变。

格式、推理、工具调用和安全行为都要测。

#### 3. 忽略 tokenizer 和模板

chat template、system prompt、工具 schema 会影响激活分布。

校准和评估最好使用真实推理模板。

#### 4. 低 bit 下 outlier 处理不足

INT4 对 outlier 很敏感。

如果不保护重要通道或敏感层，可能出现明显质量下降。

#### 5. kernel 不匹配

某种量化格式离线指标很好，但线上推理框架没有高效 kernel。

结果显存省了，速度没提升。

#### 6. 忽略长上下文

短上下文质量正常，不代表长上下文正常。

长上下文下误差可能累积，KV Cache 和 attention 路径也更敏感。

### 十五、面试问答

#### 问题 1：PTQ 和 QAT 有什么区别？

可以这样回答：

```text
PTQ 是训练后量化，通常用少量校准数据计算量化参数，成本低、部署快，但低 bit 下质量风险更大。
QAT 是训练或微调时模拟量化误差，让模型适应低精度表示，成本更高，但极低精度或激活量化时通常更稳。
```

#### 问题 2：为什么 LLM 推理常用 weight-only quantization？

可以这样回答：

```text
LLM decode 阶段经常受权重读取带宽和权重显存限制。只量化权重可以显著降低显存和带宽，同时保留激活为 FP16/BF16，避免激活量化带来的更大质量风险。
```

#### 问题 3：GPTQ 的核心直觉是什么？

可以这样回答：

```text
GPTQ 是后训练权重量化方法。它利用校准激活和近似二阶信息，按一定顺序量化权重并补偿误差，目标是让量化后的线性层输出尽量接近原输出。
```

#### 问题 4：AWQ 的核心直觉是什么？

可以这样回答：

```text
AWQ 是 activation-aware weight quantization。它认为与大激活相乘、对输出影响大的权重更重要，因此利用激活统计保护重要权重或通道，降低量化误差对模型输出的影响。
```

#### 问题 5：INT4 一定比 FP16 快吗？

可以这样回答：

```text
不一定。INT4 一定能显著减少权重显存，但速度收益依赖硬件、kernel、反量化开销和当前瓶颈。如果低精度 kernel 不成熟，或者瓶颈不在权重带宽，可能不快甚至更慢。
```

#### 问题 6：量化模型上线前怎么评估？

可以这样回答：

```text
要同时评估质量、格式稳定性、安全、业务样本、显存、TTFT、TPOT、吞吐和 P95/P99 延迟。还要按任务类型分桶比较 FP16 baseline 和量化模型，不能只看模型能否加载或 perplexity。
```

---

### 十六、常见误区

1. 误区：量化只是把 dtype 改成 INT4。
   纠正：还涉及 scale、zero-point、量化粒度、校准、kernel 和评估。

2. 误区：INT4 一定带来 4 倍加速。
   纠正：显存约四分之一不等于速度四倍，速度取决于硬件和 kernel。

3. 误区：perplexity 不变就说明量化没问题。
   纠正：还要看代码、数学、格式、安全、长上下文和业务样本。

4. 误区：PTQ 永远够用。
   纠正：低 bit、激活量化、高质量要求时可能需要 QAT 或混合精度。

5. 误区：所有层都应该同样量化。
   纠正：敏感层可以保留更高精度，混合精度常常更稳。

6. 误区：校准数据随便选一小批就行。
   纠正：校准数据要尽量覆盖真实推理模板和业务分布。

### 本讲总结

本讲最重要的结论：

1. 量化用低 bit 表示权重、激活或 KV Cache，目标是降低显存、带宽和成本。
2. 基本量化依赖 scale、zero-point 和反量化近似，误差来自低精度表示。
3. per-tensor 简单但误差大，per-channel/group-wise 更精细但实现更复杂。
4. LLM 推理常用 weight-only quantization，因为它压缩权重显存和带宽，同时降低激活量化风险。
5. PTQ 成本低、部署快；QAT 成本高但低 bit 或激活量化时更稳。
6. GPTQ 强调利用校准激活和近似二阶信息做误差补偿；AWQ 强调保护激活敏感的重要权重。
7. 量化不一定自动加速，速度收益依赖硬件、kernel 和系统瓶颈。
8. 上线前必须评估质量、格式、安全、业务样本、显存、TTFT、TPOT、吞吐和尾延迟。

## 第 72 讲：KV Cache 量化

### 本讲目标

学完本讲，你应该能回答六个问题：

1. KV Cache 为什么会成为长上下文和高并发推理的显存瓶颈。
2. KV Cache 量化和权重量化有什么不同。
3. K cache 与 V cache 的量化误差分别会影响什么。
4. per-token、per-channel、per-head、group-wise KV 量化各有什么 trade-off。
5. KV Cache 量化为什么要特别关注长上下文、attention 分布和累计误差。
6. KV Cache 量化上线前应该如何评估质量、延迟和稳定性。

上一讲讲了权重和激活量化。

这一讲聚焦另一个在 LLM serving 中越来越重要的对象：KV Cache。

在长上下文和高并发场景下，KV Cache 可能比权重更先成为瓶颈。

第 67 讲我们已经估算过 KV Cache 大小：

```math
M_{\mathrm{kv}}=2\cdot L\cdot H_{\mathrm{kv}}\cdot D_h\cdot b_{\mathrm{elem}}\cdot \sum_{i=1}^{B} S_i
```

其中 `L` 是层数，`H_kv` 是 KV head 数，`D_h` 是每个 head 的维度，`b_elem` 是每个 KV 元素的字节数，`S_i` 是第 `i` 个活跃请求当前已经占用的 token 数，前面的 `2` 表示同时保存 K 和 V。

如果 dtype 从 FP16 变成 INT8，KV Cache 理论上可以约减半。

如果变成 INT4，理论上可以约变成四分之一。

这就是 KV Cache 量化的直接动机。

资料边界上，KIVI 和 KVQuant 都把 KV cache 定位为长上下文、大 batch 推理中的重要内存瓶颈，并强调 K/V 的分布和敏感性不同；vLLM 文档也提供了 FP8 KV cache 与 scale calibration 的工程接口线索。本讲只抽象这些公开资料中的共性机制，不把某个论文或框架的具体支持范围写成所有系统都天然具备的能力。

---

### 一、为什么 KV Cache 需要量化

权重量化解决的是模型参数显存和权重读取带宽。

KV Cache 量化解决的是运行时状态显存和 decode 阶段 KV 访问带宽。

对于一个固定模型，权重大小基本固定。

但 KV Cache 会随请求动态增长：

```text
并发越高，KV Cache 越大。
上下文越长，KV Cache 越大。
输出越长，KV Cache 越大。
```

在长上下文服务中，KV Cache 可能占用大量显存。

例如多轮对话、长文档问答、代码仓库分析、Agent 工具调用，都可能带来很长上下文。

如果 KV Cache 保持 FP16/BF16：

1. 最大并发受限。
2. 最大上下文受限。
3. PagedAttention block 池压力增大。
4. Decode 阶段读取 KV 的带宽压力增大。
5. 服务成本上升。

所以 KV Cache 量化的目标是：

```text
用更低精度保存历史 K/V，在尽量不损失质量的情况下降低显存和带宽。
```

---

### 二、KV Cache 量化和权重量化的区别

权重是静态的。

模型加载后，权重数值不随输入变化。

所以权重量化可以离线完成，也可以用校准数据提前优化。

KV Cache 是动态的。

它在推理过程中产生，随每个请求的 token 不断追加。

这带来几个区别：

```text
权重量化：主要处理固定参数。
KV Cache 量化：处理每个请求、每层、每个 token 动态产生的 K/V。
```

具体差异包括：

1. KV 分布随输入变化。
2. KV 会随 decode 动态增长。
3. KV 量化和 attention 计算强耦合。
4. KV 误差会影响后续所有 token 的 attention。
5. KV Cache 还要和 PagedAttention、Continuous Batching 配合。

所以 KV Cache 量化通常比单纯权重量化更像系统问题。

它不只是把一个 tensor 存成 INT8。

---

### 三、K 和 V 的作用不同

KV Cache 里保存的是每层历史 token 的 Key 和 Value。

在 attention 中：

```math
s_j=\frac{q^\top k_j}{\sqrt{D_h}}
```

```math
a_j=\frac{\exp(s_j)}{\sum_{u=1}^{S}\exp(s_u)}
```

```math
o=\sum_{j=1}^{S}a_j v_j
```

这里 `q` 是当前 token 的 query，`k_j` 和 `v_j` 是历史第 `j` 个 token 的 key 和 value，`s_j` 是 attention score，`a_j` 是 softmax 后的注意力权重，`o` 是 attention 输出。

K 影响的是 attention score。

也就是当前 token 应该关注哪些历史 token。

V 影响的是被汇总的内容。

也就是关注到历史 token 后，取回什么信息。

所以 K 和 V 的量化误差影响不同。

K 的误差可能改变 attention 分布。

例如原本应该关注第 100 个 token，量化后分数排序变了，模型可能关注错误位置。

V 的误差更多影响内容表示。

即使关注位置没变，取回的 value 向量也有噪声。

如果量化后的 key 写成：

```math
\hat k_j=k_j+\epsilon^{K}_j
```

那么 score 误差是：

```math
\Delta s_j=\frac{q^\top\epsilon^{K}_j}{\sqrt{D_h}}
```

这个误差进入 softmax，可能改变 `a_j` 的排序和权重分布。尤其当多个候选 token 的 score 很接近时，一个很小的 `Delta s_j` 也可能让 top attention 位置变化。

如果量化后的 value 写成：

```math
\hat v_j=v_j+\epsilon^{V}_j
```

在 attention 权重近似不变时，value 误差对输出的直接影响可以粗略写成：

```math
\Delta o_V\approx\sum_{j=1}^{S}a_j\epsilon^{V}_j
```

这个公式只表示直接误差项。真实模型里 K 的误差还会改变 `a_j`，V 的误差会继续经过后续层，因此长上下文和多步 decode 中影响会更复杂。

可以这样理解：

```text
K 错了，可能找错地方。
V 错了，可能找到地方但取回的信息变粗糙。
```

这也是为什么有些方案会对 K/V 使用不同量化策略。

---

### 四、KV Cache 量化的基本方式

最简单的做法是对 K/V 做线性量化：

```math
q_x=\mathrm{clip}\left(\mathrm{round}\left(\frac{x}{s}\right)+z, q_{\mathrm{min}}, q_{\mathrm{max}}\right)
```

```math
\hat x=s\cdot(q_x-z)
```

其中 `s` 是 scale，`z` 是 zero-point，`q_min` 和 `q_max` 是低 bit 整数的取值范围，`x_hat` 是反量化后的近似值。对称量化时通常令 `z = 0`，非对称量化则用 `z` 处理浮点范围不以 0 为中心的情况。

但关键问题是 scale 怎么选。

KV Cache 的 shape 通常可以抽象成：

```text
[num_layers, batch_or_blocks, num_kv_heads, seq_len, head_dim]
```

不同系统存储布局会不同，但核心维度类似。

scale 可以按不同粒度设置。

#### 1. Per-tensor KV quantization

整个 K 或 V tensor 共用一个 scale。

```math
s=\frac{\max_i |x_i|}{q_{\mathrm{max}}}
```

优点是简单。

缺点是误差大，难以适应不同 token、head、channel 的数值范围。

#### 2. Per-token quantization

每个 token 的 K/V 使用自己的 scale。

```math
s_t=\frac{\max_c |x_{t,c}|}{q_{\mathrm{max}}}
```

这里 `t` 是 token 位置，`c` 是 head_dim 内的 channel 下标。

优点是能适应不同 token 的激活范围。

缺点是 scale 数量随 token 增长，元数据和读取更复杂。

#### 3. Per-channel quantization

按 head_dim 的 channel 设置 scale。

```math
s_c=\frac{\max_t |x_{t,c}|}{q_{\mathrm{max}}}
```

优点是能处理不同维度范围差异。

缺点是 kernel 要支持更细粒度 scale。

KIVI 和 KVQuant 的公开摘要都指出 key cache 更适合关注 channel 方向的量化粒度，这是因为 key 的某些 channel 可能存在更明显的范围差异或 outlier；这不等于所有模型都必须照搬同一个实现，而是提示面试中不能把 K 和 V 当成完全同分布的张量。

#### 4. Per-head quantization

每个 attention head 或 KV head 使用不同 scale。

```math
s_h=\frac{\max_{t,c}|x_{h,t,c}|}{q_{\mathrm{max}}}
```

它比 per-tensor 更细，比 per-channel 更粗。

#### 5. Group-wise quantization

把 head_dim 或 token 维度分组，每组一个 scale。

```math
s_g=\frac{\max_{i\in\mathcal{G}_g}|x_i|}{q_{\mathrm{max}}}
```

其中 `G_g` 表示第 `g` 个 group 的元素集合。

这是常见折中。

总结：

```text
粒度越细，误差越小，但 scale 元数据、kernel 复杂度和访存开销越高。
```

如果只估算 payload，不考虑元数据，`n` 个 KV 元素用 `b_q` bit 保存时：

```math
M_{\mathrm{payload}}=\frac{n\cdot b_q}{8}
```

如果共有 `G` 个 scale，每个 scale 用 `b_s` bytes 保存，额外元数据是：

```math
M_{\mathrm{scale}}=G\cdot b_s
```

最终节省比例不能只看 `b_q / 16`，还要加上 scale、zero-point、padding、block table 和 kernel 临时 buffer。

下面这个 toy demo 只模拟一个 attention head。K cache 里故意放了一个 channel outlier，V cache 里故意放了一个 token outlier。你会看到：K 用 per-channel INT4 能更好保持 attention score，V 用 per-token INT4 能更好保持最终输出，但更细粒度的 scale 也会增加元数据。

```python
from math import exp, sqrt


def dot(a, b):
    return sum(x * y for x, y in zip(a, b))


def softmax(values):
    m = max(values)
    exps = [exp(v - m) for v in values]
    return [v / sum(exps) for v in exps]


def attention(q, k_cache, v_cache):
    scores = [dot(q, k) / sqrt(len(q)) for k in k_cache]
    weights = softmax(scores)
    out = [sum(w * v[d] for w, v in zip(weights, v_cache)) for d in range(len(v_cache[0]))]
    return scores, weights, out


def rmse(a, b):
    diffs = [x - y for x, y in zip(a, b)]
    return sqrt(sum(d * d for d in diffs) / len(diffs))


def flatten(matrix):
    return [v for row in matrix for v in row]


def quantize_vector(values, bits=4):
    qmax = 2 ** (bits - 1) - 1
    scale = (max(abs(v) for v in values) or 1.0) / qmax
    q = [max(-qmax, min(qmax, round(v / scale))) for v in values]
    return [scale * v for v in q], scale


def quantize_matrix(matrix, mode):
    if mode == "tensor":
        rows, cols = len(matrix), len(matrix[0])
        deq, scale = quantize_vector(flatten(matrix))
        return [deq[i * cols:(i + 1) * cols] for i in range(rows)], [scale]

    if mode == "token":
        rows, scales = [], []
        for row in matrix:
            deq, scale = quantize_vector(row)
            rows.append(deq)
            scales.append(scale)
        return rows, scales

    if mode == "channel":
        cols, scales = [], []
        for col in zip(*matrix):
            deq, scale = quantize_vector(list(col))
            cols.append(deq)
            scales.append(scale)
        return [list(row) for row in zip(*cols)], scales

    raise ValueError(mode)


def kv_bytes(num_elements, data_bits, num_scales, scale_bytes=2):
    return num_elements * data_bits / 8 + num_scales * scale_bytes


q = [0.75, -0.40, 0.05, 0.30]
k_cache = [
    [0.34, -0.12, 0.18, 0.04],
    [0.31, -0.08, 0.16, 0.05],
    [0.36, -0.11, 2.40, 0.06],  # key channel outlier
    [0.30, -0.10, 0.21, 0.04],
    [0.33, -0.09, 0.19, 0.05],
]
v_cache = [
    [0.70, 0.10, 0.00, 0.20],
    [0.76, 0.12, 0.02, 0.18],
    [0.35, 1.10, 0.30, -0.10],
    [0.72, 0.08, 0.01, 0.21],
    [3.20, 0.16, 0.03, 0.19],  # value token outlier
]

base_scores, base_weights, base_out = attention(q, k_cache, v_cache)
k_tensor, k_tensor_scales = quantize_matrix(k_cache, "tensor")
k_channel, k_channel_scales = quantize_matrix(k_cache, "channel")
v_tensor, v_tensor_scales = quantize_matrix(v_cache, "tensor")
v_token, v_token_scales = quantize_matrix(v_cache, "token")

scores_kt, weights_kt, _ = attention(q, k_tensor, v_cache)
scores_kc, weights_kc, _ = attention(q, k_channel, v_cache)
_, _, out_vt = attention(q, k_cache, v_tensor)
_, _, out_vtok = attention(q, k_cache, v_token)

elements = len(flatten(k_cache)) + len(flatten(v_cache))
fp16_bytes = kv_bytes(elements, data_bits=16, num_scales=0)
int4_tensor_bytes = kv_bytes(elements, 4, len(k_tensor_scales) + len(v_tensor_scales))
int4_mixed_bytes = kv_bytes(elements, 4, len(k_channel_scales) + len(v_token_scales))

print("fp16_top=", max(range(len(base_weights)), key=base_weights.__getitem__), "weights=", [round(x, 3) for x in base_weights])
print("K_per_tensor_score_rmse=", round(rmse(base_scores, scores_kt), 5))
print("K_per_channel_score_rmse=", round(rmse(base_scores, scores_kc), 5))
print("K_per_tensor_weight_rmse=", round(rmse(base_weights, weights_kt), 5))
print("K_per_channel_weight_rmse=", round(rmse(base_weights, weights_kc), 5))
print("V_per_tensor_output_rmse=", round(rmse(base_out, out_vt), 5))
print("V_per_token_output_rmse=", round(rmse(base_out, out_vtok), 5))
print("toy_fp16_bytes=", fp16_bytes)
print("toy_int4_tensor_bytes=", int4_tensor_bytes)
print("toy_int4_channel_token_bytes=", int4_mixed_bytes)
```

典型输出类似：

```text
fp16_top= 2 weights= [0.199, 0.196, 0.212, 0.196, 0.198]
K_per_tensor_score_rmse= 0.0227
K_per_channel_score_rmse= 0.00688
K_per_tensor_weight_rmse= 0.00212
K_per_channel_weight_rmse= 0.00131
V_per_tensor_output_rmse= 0.11483
V_per_token_output_rmse= 0.02677
toy_fp16_bytes= 80.0
toy_int4_tensor_bytes= 24.0
toy_int4_channel_token_bytes= 38.0
```

这个例子不要解读成“per-channel K + per-token V 永远最优”。它只展示 K/V 误差路径和 scale 粒度 trade-off：更细粒度通常误差更小，但 scale 元数据和 kernel 复杂度会上升，真实部署还要看模型分布、硬件、attention backend 和长上下文评估。

---

### 五、INT8 KV Cache 与 INT4 KV Cache

INT8 KV Cache 通常相对稳。

它把 FP16/BF16 的 KV 降到 8 bit。

显存约减半，误差相对可控。

INT4 KV Cache 更激进。

显存理论上约变成 FP16 的四分之一，但误差更大。

INT4 更容易遇到：

1. attention 分布偏移。
2. 长上下文退化。
3. 事实细节丢失。
4. 代码和数学稳定性下降。
5. 多轮对话后半段质量下降。

所以实践中常见策略是：

```text
先尝试 INT8 KV Cache，再根据质量和成本需求评估 INT4 或混合精度。
```

也可以采用混合策略：

1. K 用更高精度，V 用更低精度。
2. 最近 token 保留 FP16，较旧 token 量化。
3. 敏感层保留高精度。
4. 长上下文场景使用更保守配置。

---

### 六、为什么长上下文更敏感

KV Cache 量化在短上下文下可能看起来没问题。

但长上下文会放大量化风险。

原因有三个。

第一个原因是 attention 候选位置更多。

上下文越长，Q 要和更多 K 做相似度计算。

K 的小误差可能影响 top attention positions。

第二个原因是远距离信息本来就难检索。

长上下文任务经常依赖稀疏关键证据。

如果关键 token 的 K/V 量化误差较大，模型可能更容易漏掉证据。

第三个原因是误差会影响后续生成。

Decode 每一步都使用历史 KV。

如果历史 KV 有噪声，后续 token 的 hidden state 也可能受影响。

所以评估 KV Cache 量化时，不能只测短问答。

必须测长上下文任务。

---

### 七、最近 token 和远端 token 可以不同处理

一个常见直觉是：

```text
最近 token 对下一步生成通常更重要，远端 token 更多承担背景信息。
```

因此可以设计分层策略：

1. 最近 `N` 个 token 的 KV 保留 FP16/BF16。
2. 更早的 token 的 KV 量化到 INT8 或 INT4。
3. 被频繁访问的 prefix cache 使用更高精度。
4. 普通历史上下文使用低精度。

这种策略希望在显存和质量之间折中。

但它也更复杂。

系统要维护不同精度 KV blocks，并让 attention kernel 正确读取和混合计算。

这会增加实现难度。

---

### 八、与 PagedAttention 的关系

PagedAttention 把 KV Cache 拆成 blocks。

KV Cache 量化可以和 block 管理结合。

例如：

```text
每个 KV block 存 INT8 数据 + scale 元数据。
```

或者：

```text
每个 block 内按 head/group 存 scale。
```

这样做的好处是：

1. block 是天然管理单元。
2. 请求结束时可以按 block 释放。
3. prefix cache 可以按 block 复用。
4. 不同 block 可以有不同量化参数。

但也有挑战：

1. block table 访问后还要读 scale。
2. kernel 要支持反量化或低精度 attention。
3. scale 元数据会增加额外带宽。
4. 不同精度 block 混用更复杂。

所以 KV Cache 量化不是孤立优化。

它和 PagedAttention、attention kernel、batch scheduler 都有关。

---

### 九、反量化放在哪里

KV Cache 量化后，attention 计算有几种实现方式。

#### 1. 先反量化再 attention

把 K/V 从 INT8/INT4 反量化成 FP16，再执行普通 attention。

优点是实现简单。

缺点是反量化后中间数据可能增加显存和带宽，收益打折。

#### 2. Kernel 内融合反量化

attention kernel 读取量化 K/V 和 scale，在计算时即时反量化。

优点是减少中间写回，更可能保留带宽收益。

缺点是 kernel 实现复杂。

#### 3. 低精度 attention 计算

尽量在低精度下完成更多计算。

这对硬件和数值稳定要求更高。

实际系统通常会选择 kernel 内融合反量化或硬件支持的低精度路径。

否则量化虽然省了存储，但可能没有速度收益。

---

### 十、KV Cache 量化如何影响指标

KV Cache 量化主要影响三类指标。

#### 1. 显存容量

这是最直接收益。

KV 从 FP16 到 INT8，理论上 KV 数据约减半。

从 FP16 到 INT4，理论上约四分之一。

但要扣除 scale、zero-point、padding 和 block 元数据。

更稳妥的估算是：

```math
M_{\mathrm{kv,quant}}=M_{\mathrm{payload}}+M_{\mathrm{scale}}+M_{\mathrm{block}}+M_{\mathrm{temp}}
```

这里 `M_payload` 是低 bit K/V 数据本体，`M_scale` 是 scale/zero-point 元数据，`M_block` 是分页或 block 管理相关开销，`M_temp` 是 kernel 反量化、临时 buffer 或对齐带来的额外开销。真实压测时要比较的是这个总量，而不是只比较 INT4 payload 和 FP16 payload。

如果同一张卡可用于 KV 的剩余显存是 `M_free`，每个请求平均上下文长度是 `S_avg`，粗略最大并发可以写成：

```math
B_{\mathrm{max}}\approx\left\lfloor\frac{M_{\mathrm{free}}}{2\cdot L\cdot S_{\mathrm{avg}}\cdot H_{\mathrm{kv}}\cdot D_h\cdot b_{\mathrm{eff}}}\right\rfloor
```

其中 `b_eff` 是把 payload 和元数据折算后的每元素平均字节数。这个公式只适合做容量估算，真实系统还要预留权重、激活临时 buffer、CUDA workspace、碎片和流量峰值余量。

#### 2. Decode TPOT

如果 decode 受 KV 读取带宽限制，KV 量化可能降低 TPOT。

但如果反量化开销很大，或者瓶颈在权重读取、调度、网络，TPOT 可能改善不明显。

#### 3. 最大并发和最大上下文

KV 显存减少后，同样 GPU 可以容纳更多活跃 token。

这可能提高最大并发或支持更长上下文。

但并发上升后，调度和带宽压力也可能上升。

所以要看端到端效果。

---

### 十一、质量风险有哪些

KV Cache 量化可能带来的质量问题包括：

1. 长上下文检索能力下降。
2. lost in the middle 更严重。
3. 多轮对话记忆变差。
4. 代码生成前后不一致。
5. 数学推理中间步骤漂移。
6. RAG 中引用证据不稳定。
7. 工具调用参数更容易出错。
8. 输出重复或提前结束。

这些问题有时不会反映在短文本 benchmark 上。

所以 KV Cache 量化评估必须更偏场景化。

尤其要覆盖：

```text
长 prompt + 长 output
长 prompt + 短 answer
短 prompt + 长 output
多轮对话
RAG 引用
代码长文件
```

---

### 十二、评估 KV Cache 量化的实验设计

一个比较完整的评估方案可以包含四组对比。

#### 1. 精度对比

```text
FP16 KV baseline
INT8 KV
INT4 KV
mixed precision KV
```

#### 2. 长度分桶

```text
1k tokens
4k tokens
16k tokens
32k tokens
更长上下文
```

#### 3. 任务分桶

```text
普通问答
长文总结
needle-in-a-haystack
RAG 问答
代码理解
数学推理
多轮对话
工具调用
```

#### 4. 系统指标

```text
KV 显存占用
最大并发
TTFT
TPOT
tokens/s
P95/P99 latency
OOM 率
```

最终要和 FP16 KV baseline 对比。

不能只报告单个平均分。

---

### 十三、真实项目中的坑

#### 1. 只测短上下文

短上下文正常，不代表长上下文正常。

KV Cache 量化最容易在长上下文场景暴露问题。

#### 2. 只看显存节省

显存省了，但如果反量化开销大，TPOT 可能不降。

甚至在某些 kernel 下可能变慢。

#### 3. 忽略 scale 元数据

细粒度量化需要更多 scale。

如果 scale 读取开销大，会影响带宽收益。

#### 4. K/V 使用同一策略

K 和 V 的作用不同。

有些场景下 K 对 attention 分布更敏感，应该考虑不同精度或不同量化粒度。

#### 5. 与 prefix cache 复用冲突

如果 prefix cache 的量化参数和请求后续策略不一致，复用、拼接和验证会复杂。

#### 6. 忽略多轮对话

多轮对话的历史 KV 很长，用户还会追问前文细节。

这类场景对 KV 量化很敏感。

---

### 十四、面试问答

#### 问题 1：KV Cache 量化和权重量化有什么区别？

可以这样回答：

```text
权重是静态参数，可以离线量化；KV Cache 是每个请求在推理过程中动态生成的运行时状态，会随上下文、输出和并发增长。KV 量化还和 attention kernel、PagedAttention、调度和长上下文质量强相关。
```

#### 问题 2：为什么 KV Cache 量化对长上下文特别重要？

可以这样回答：

```text
KV Cache 大小与所有活跃 token 数成正比。长上下文下 KV 显存和读取带宽都会显著增加，量化可以降低显存并可能改善 decode 带宽瓶颈。但长上下文也更容易放大量化误差，需要谨慎评估。
```

#### 问题 3：K 和 V 的量化误差有什么不同？

可以这样回答：

```text
K 参与 QK^T，影响 attention score 和关注位置；K 误差可能让模型关注错误 token。V 是被 attention 权重汇总的内容；V 误差更多影响取回信息的表示质量。
```

#### 问题 4：KV Cache 量化粒度怎么选？

可以这样回答：

```text
per-tensor 简单但误差大，per-token/per-channel 更准确但 scale 元数据和 kernel 复杂度更高，group-wise 是常见折中。实际要结合质量、显存、带宽、kernel 和长上下文评估选择。
```

#### 问题 5：KV Cache INT4 一定可以上线吗？

可以这样回答：

```text
不一定。INT4 显存收益大，但误差也更大，可能影响长上下文检索、RAG、代码、数学和多轮对话。通常要先和 FP16/INT8 baseline 对比，并按长度和任务分桶评估。
```

#### 问题 6：KV Cache 量化上线前怎么评估？

可以这样回答：

```text
要比较 FP16、INT8、INT4 和混合精度 KV，在不同上下文长度、任务类型和并发下评估质量、长上下文检索、RAG、代码、数学、多轮对话、KV 显存、TTFT、TPOT、P95/P99 和 OOM 率。
```

---

### 十五、常见误区

1. 误区：KV Cache 量化和权重量化一样简单。
   纠正：KV 是动态运行时状态，和 attention、调度、PagedAttention、长上下文强耦合。

2. 误区：INT4 KV 一定比 INT8 KV 更好。
   纠正：INT4 显存更省，但质量和稳定性风险更高。

3. 误区：只要短问答正常，就说明 KV 量化没问题。
   纠正：必须测长上下文、多轮、RAG、代码和数学。

4. 误区：scale 粒度越细越好。
   纠正：细粒度降低误差，但增加元数据、访存和 kernel 复杂度。

5. 误区：KV 量化一定降低 TPOT。
   纠正：如果反量化开销大或瓶颈不在 KV 读取，TPOT 未必下降。

6. 误区：K 和 V 可以永远使用同一量化策略。
   纠正：K 影响注意力位置，V 影响内容表示，敏感性可能不同。

### 本讲总结

本讲最重要的结论：

1. KV Cache 大小随活跃 token 数增长，是长上下文和高并发 serving 的核心瓶颈。
2. KV Cache 量化降低运行时状态显存和 KV 读取带宽，区别于静态权重量化。
3. K 影响 attention score 和关注位置，V 影响被取回的内容表示，二者量化敏感性可能不同。
4. KV 量化粒度越细，误差通常越小，但 scale 元数据、访存和 kernel 复杂度越高。
5. INT8 KV 通常更稳，INT4 KV 显存收益更大但长上下文质量风险更高。
6. KV Cache 量化要和 PagedAttention、attention kernel、prefix cache、continuous batching 一起设计。
7. 评估必须覆盖长上下文、RAG、代码、数学、多轮对话、TTFT、TPOT、P95/P99 和 OOM 率。
8. 面试中要把 KV Cache 量化讲成“动态运行时状态压缩 + attention 数值稳定 + serving 系统优化”的综合问题。

## 第 73 讲：Serving 系统设计

### 本讲目标

学完本讲，你应该能回答六个问题：

1. 一个 LLM 在线服务从请求入口到 GPU worker 的完整链路是什么。
2. Serving 系统为什么不只是“把模型放到 API 后面”。
3. 路由、排队、调度、KV Cache、流式返回分别解决什么问题。
4. 如何设计低延迟、高吞吐、高可靠的大模型服务。
5. 线上需要监控哪些指标，如何定位 TTFT、TPOT、OOM、尾延迟问题。
6. 面试中如何系统回答“设计一个 ChatGPT 类推理服务”。

前面几讲分别讲了 Prefill/Decode、KV Cache、PagedAttention、Continuous Batching、Speculative Decoding 和量化。

这一讲把这些技术串成一个完整系统。

真正的 LLM serving 不是：

```text
HTTP 请求 -> model.generate() -> 返回结果
```

而是一个复杂在线系统：

```text
请求入口 -> 鉴权限流 -> 路由 -> 队列 -> 调度 -> tokenizer -> prefill/decode -> 流式返回 -> 监控评估 -> 容错恢复
```

面试中如果问 serving system design，考察的不是某一个 kernel，而是你能否把模型、系统、用户体验和成本放在一起考虑。

---

### 一、整体架构图

一个典型 LLM serving 架构可以抽象成：

```text
Client
  -> API Gateway
  -> Auth / Rate Limit / Quota
  -> Request Router
  -> Scheduler / Queue
  -> Tokenizer
  -> Inference Worker Pool
       -> Model Engine
       -> KV Cache Manager
       -> Batching Manager
       -> GPU Kernels
  -> Streamer
  -> Logging / Metrics / Tracing
```

如果是企业级或多模型服务，还会有：

```text
Model Registry
Replica Manager
Load Balancer
Fallback Model
Safety Filter
RAG / Tool Service
Evaluation Pipeline
Cost Dashboard
```

你可以把 serving 系统分成四层：

1. 接入层：处理用户请求、鉴权、限流、协议和流式连接。
2. 调度层：决定请求去哪台机器、何时进入 batch、占用多少资源。
3. 推理层：执行 prefill、decode、KV Cache 管理、kernel 和多 GPU 通信。
4. 运维层：监控、告警、日志、trace、灰度、回滚和成本分析。

---

### 二、请求生命周期

一次请求的生命周期大致如下。

#### 1. 请求进入 API Gateway

用户发送请求：

```text
prompt / messages / model / temperature / max_tokens / stream
```

Gateway 负责：

1. 协议解析。
2. 鉴权。
3. 参数校验。
4. 请求大小限制。
5. 速率限制。
6. 记录 request_id。

#### 2. 路由到合适模型副本

Router 根据模型名、负载、队列长度、显存状态、地域和 SLA 选择副本。

例如：

```text
gpt-large -> GPU pool A
gpt-small -> GPU pool B
long-context request -> long-context pool
enterprise user -> priority pool
```

#### 3. 进入等待队列

请求不会立刻执行。

它会进入 scheduler 的队列。

调度器会根据：

1. 当前 running batch。
2. token budget。
3. KV Cache budget。
4. 优先级。
5. prefill/decode 配额。
6. timeout。

决定何时让请求开始 prefill。

#### 4. Tokenization 和 chat template

请求被转成 token ids。

这里要处理：

1. system/user/assistant 角色。
2. tool schema。
3. special tokens。
4. max context 截断策略。
5. stop tokens。

Tokenization 会影响 prompt length，从而影响 TTFT、KV Cache 和成本。

#### 5. Prefill

模型处理完整 prompt，生成初始 KV Cache。

Prefill 主要影响 TTFT。

长 prompt、RAG 文档、工具 schema、长历史对话都会让 prefill 变重。

#### 6. Decode

模型逐 token 生成。

Decode 阶段要依赖：

1. KV Cache。
2. Continuous Batching。
3. 解码参数。
4. stop 条件。
5. 流式返回。

Decode 主要影响 TPOT 和用户感受到的吐字速度。

#### 7. 流式返回与结束

每生成一个或一小批 token，服务端通过 SSE、WebSocket 或 gRPC stream 返回给客户端。

请求结束后要：

1. 释放 KV Cache。
2. 清理 scheduler 状态。
3. 记录 usage。
4. 写日志和指标。
5. 处理计费。

---

### 三、路由设计

路由层要解决：

```text
这个请求应该送到哪个模型、哪个副本、哪个 GPU worker？
```

常见路由依据包括：

1. 模型名称。
2. 上下文长度。
3. 用户等级和 SLA。
4. 当前队列长度。
5. 当前 KV Cache 剩余容量。
6. GPU 利用率。
7. 地域和网络延迟。
8. 灰度版本。
9. 失败重试和 fallback。

一个简单路由策略可能是 least-loaded：

```text
选择当前排队 token 数最少的副本。
```

但只看请求数不够。

一个长 prompt 请求比十个短请求还重。

更合理的是看：

```text
queued_prefill_tokens
running_decode_tokens
available_kv_blocks
estimated_TTFT
```

面试中要强调：

```text
LLM 路由最好按 token 和资源估算，而不是只按 request 数。
```

#### 一个可解释的路由打分公式

工程里不一定真的用一个线性公式做最终决策，但面试和设计文档中可以先用它说明思路。

对候选副本 `r` 和新请求 `x`，可以定义一个资源感知打分：

```math
C_r = w_p P_r + w_d D_r + w_k K_r + w_q Q_r - w_s S_r
```

其中：

1. `P_r` 表示 prefill 压力。
2. `D_r` 表示 decode 压力。
3. `K_r` 表示 KV Cache 压力。
4. `Q_r` 表示等待队列压力。
5. `S_r` 表示优先级或 SLA 收益，优先级越高，分数越应该降低。
6. `w_p,w_d,w_k,w_q,w_s` 是业务根据目标延迟、吞吐和成本设置的权重。

可以用下面几个近似量构造这些压力：

```math
P_r = (N_{p,r} + S_{in}) / R_{p,r}
```

```math
D_r = (N_{d,r} + S_{out}) / R_{d,r}
```

```math
K_r = (B_{used,r} + B_{req}) / B_{cap,r}
```

```math
Q_r = Q_{wait,r} / Q_{ref}
```

这里 `N_{p,r}` 是副本 `r` 已排队的 prefill token 数，`S_{in}` 是新请求输入 token 数，`R_{p,r}` 是该副本的 prefill tokens/s；`N_{d,r}` 是正在 decode 的输出 token 预算，`S_{out}` 是新请求预计输出 token 数，`R_{d,r}` 是 decode tokens/s；`B_{used,r}` 是已占用 KV blocks，`B_{req}` 是新请求需要预留的 KV blocks，`B_{cap,r}` 是副本 KV block 容量；`Q_{wait,r}` 是等待队列长度，`Q_{ref}` 是归一化用的参考队列长度。

路由时先过滤掉 `K_r > 1` 的副本，再选择 `C_r` 最小的副本。这个公式的意义不是追求绝对精确，而是把“请求数、token 数、KV Cache、队列和 SLA”放进同一个可解释框架。

#### 最小 Python demo：按请求数路由为什么会出错

下面的 demo 用纯 Python 模拟两个 GPU 副本。`gpu-a` 的请求数少，但 KV Cache 已经很紧张；`gpu-b` 的请求数多，但 token 队列和 KV 余量更适合接长请求。

```python
from copy import deepcopy
from dataclasses import dataclass
from math import ceil

BLOCK_SIZE = 16
GATEWAY_S = 0.015
TOKENIZER_S = 0.020


@dataclass
class Request:
    name: str
    prompt_tokens: int
    max_new_tokens: int
    priority: int = 0


@dataclass
class Replica:
    name: str
    prefill_tps: float
    decode_tps: float
    kv_capacity_blocks: int
    active_requests: int
    queued_prefill_tokens: int
    running_decode_tokens: int
    used_kv_blocks: int


def kv_blocks(req):
    return ceil((req.prompt_tokens + req.max_new_tokens) / BLOCK_SIZE)


def feasible(replica, req):
    return replica.used_kv_blocks + kv_blocks(req) <= replica.kv_capacity_blocks


def naive_count_route(replicas):
    return min(replicas, key=lambda r: r.active_requests)


def token_aware_score(replica, req):
    prefill_wait_s = (replica.queued_prefill_tokens + req.prompt_tokens) / replica.prefill_tps
    decode_pressure_s = (replica.running_decode_tokens + req.max_new_tokens) / replica.decode_tps
    kv_pressure = (replica.used_kv_blocks + kv_blocks(req)) / replica.kv_capacity_blocks
    priority_bonus = 0.50 * req.priority
    return prefill_wait_s + 0.25 * decode_pressure_s + 2.0 * kv_pressure - priority_bonus


def token_aware_route(replicas, req):
    candidates = [r for r in replicas if feasible(r, req)]
    if not candidates:
        return None
    return min(candidates, key=lambda r: token_aware_score(r, req))


def estimated_ttft(replica, req):
    prefill_wait_s = (replica.queued_prefill_tokens + req.prompt_tokens) / replica.prefill_tps
    return GATEWAY_S + TOKENIZER_S + prefill_wait_s


def admit(replica, req):
    replica.active_requests += 1
    replica.queued_prefill_tokens += req.prompt_tokens
    replica.running_decode_tokens += req.max_new_tokens
    replica.used_kv_blocks += kv_blocks(req)


base_replicas = [
    Replica("gpu-a", prefill_tps=26000, decode_tps=7000, kv_capacity_blocks=1200,
            active_requests=3, queued_prefill_tokens=12000, running_decode_tokens=8000,
            used_kv_blocks=900),
    Replica("gpu-b", prefill_tps=22000, decode_tps=6500, kv_capacity_blocks=1500,
            active_requests=7, queued_prefill_tokens=2500, running_decode_tokens=900,
            used_kv_blocks=700),
]
requests = [
    Request("long_rag", prompt_tokens=8400, max_new_tokens=1600, priority=0),
    Request("vip_short", prompt_tokens=220, max_new_tokens=420, priority=2),
    Request("code_agent", prompt_tokens=2100, max_new_tokens=900, priority=1),
]

replicas = deepcopy(base_replicas)
for req in requests:
    naive = naive_count_route(replicas)
    aware = token_aware_route(replicas, req)
    print(f"request= {req.name} kv_blocks= {kv_blocks(req)}")
    print(f"  naive_count= {naive.name} feasible= {feasible(naive, req)}")
    if aware is None:
        print("  token_aware= reject")
        continue
    print(
        f"  token_aware= {aware.name} "
        f"score= {token_aware_score(aware, req):.3f} "
        f"est_ttft_s= {estimated_ttft(aware, req):.3f}"
    )
    admit(aware, req)

print("final_replicas=")
for replica in replicas:
    free_kv = replica.kv_capacity_blocks - replica.used_kv_blocks
    print(
        f"  {replica.name} active= {replica.active_requests} "
        f"queued_prefill= {replica.queued_prefill_tokens} "
        f"running_decode= {replica.running_decode_tokens} free_kv= {free_kv}"
    )
```

一组输出如下：

```text
request= long_rag kv_blocks= 625
  naive_count= gpu-a feasible= False
  token_aware= gpu-b score= 2.358 est_ttft_s= 0.530
request= vip_short kv_blocks= 40
  naive_count= gpu-a feasible= True
  token_aware= gpu-a score= 1.337 est_ttft_s= 0.505
request= code_agent kv_blocks= 188
  naive_count= gpu-a feasible= True
  token_aware= gpu-a score= 2.264 est_ttft_s= 0.586
final_replicas=
  gpu-a active= 5 queued_prefill= 14320 running_decode= 9320 free_kv= 72
  gpu-b active= 8 queued_prefill= 10900 running_decode= 2500 free_kv= 175
```

第一条长 RAG 请求按请求数会被送到 `gpu-a`，但 `gpu-a` 的 KV blocks 不够，真实系统里可能只能排队、拒绝或转发。token-aware 路由会先做可行性过滤，再综合 prefill、decode、KV 和优先级压力选择副本。

---

### 四、调度器设计

调度器是 serving 系统的核心。

它要在多个目标之间权衡：

1. 低 TTFT。
2. 低 TPOT。
3. 高吞吐。
4. 低 P95/P99。
5. 高 GPU 利用率。
6. 不 OOM。
7. 公平性和优先级。

调度器通常要管理三类队列：

```text
waiting queue：等待 prefill 的请求
running queue：正在 decode 的请求
finished / aborted：已完成或取消的请求
```

每个 iteration 做：

1. 释放完成请求的 KV。
2. 选择若干 running 请求做 decode。
3. 根据 token budget 插入新请求 prefill。
4. 为新请求分配 KV blocks。
5. 更新流式输出。

这就是第 68 讲 Continuous Batching 的系统落点。

调度器每轮可以粗略理解成同时满足三个约束：

```math
N_{prefill,admit} \le B_{p,iter}
```

```math
N_{decode,step} \le B_{d,iter}
```

```math
B_{used} + B_{new} \le B_{cap}
```

其中 `N_{prefill,admit}` 是本轮允许进入 prefill 的 token 数，`B_{p,iter}` 是本轮 prefill token budget；`N_{decode,step}` 是本轮要推进的 decode token 数，`B_{d,iter}` 是 decode budget；`B_{used}` 是已经占用的 KV blocks，`B_{new}` 是新 admitted 请求需要预留的 KV blocks，`B_{cap}` 是总 KV block 容量。

如果系统一味扩大 `B_{p,iter}`，短期 TTFT 可能下降，但 decode 会被长 prefill 挤压，TPOT 和流式稳定性可能变差；如果一味偏向 decode，新请求会排队太久，TTFT 会变差。调度器的核心就是在这两个阶段之间做动态配额。

---

### 五、KV Cache Manager

KV Cache Manager 负责：

1. 分配 KV blocks。
2. 释放 KV blocks。
3. 维护 block table。
4. 支持 prefix cache。
5. 处理请求取消和异常释放。
6. 统计剩余 KV 容量。
7. 可能支持 KV Cache 量化。

它和 scheduler 紧密配合。

新请求能不能进入 prefill，不只看 batch size，还要看 KV Cache 是否够。

如果 KV 预算估计不足，可能出现：

```text
prefill 成功，但 decode 到一半 OOM。
```

这在服务中是严重问题。

因此调度器通常需要为每个请求估计：

```text
prompt_len + max_new_tokens 或保守输出长度
```

并预留足够 KV 空间，或设置中途扩容和拒绝策略。

如果使用固定大小的 KV block，可以先估计每个请求需要多少 blocks：

```math
B_{req} = ceil((S_{prompt} + S_{max}) / B_{block})
```

进一步估算这个请求在完整生命周期中可能占用的 KV 字节数：

```math
M_{kv,req} = 2 * L * H_{kv} * D_h * b_{elem} * (S_{prompt} + S_{max})
```

这里 `S_{prompt}` 是输入 token 数，`S_{max}` 是最大输出 token 预算，`B_{block}` 是每个 KV block 包含的 token 数；`L` 是层数，`H_{kv}` 是 KV heads 数，`D_h` 是每个 head 的维度，`b_{elem}` 是每个 KV 元素的字节数。前面的 `2` 来自 Key 和 Value 两份缓存。

---

### 六、流式返回设计

聊天产品通常要求 stream。

流式返回影响用户体验。

用户关注的不只是总延迟，还有：

1. 多久出第一个 token。
2. 后续 token 是否稳定。
3. 是否突然卡住。
4. 取消请求是否立即生效。
5. 网络断开后资源是否释放。

常见流式协议包括：

```text
SSE
WebSocket
gRPC streaming
```

服务端要处理：

1. token 到文本的增量 detokenization。
2. UTF-8 边界。
3. stop sequence。
4. heartbeat。
5. client disconnect。
6. backpressure。

如果客户端网络慢，服务端不能无限缓存输出。

否则会造成内存堆积。

用户体验上，TTFT 和 TPOT 可以拆成两个近似公式。

第一个 token 的延迟大致为：

```math
T_{first} = T_{gateway} + T_{auth} + T_{route} + T_{queue} + T_{tok} + T_{prefill} + T_{sample} + T_{stream}
```

后续每个 token 的间隔大致为：

```math
T_{per,token} = T_{schedule} + T_{decode} + T_{sample} + T_{detok} + T_{network}
```

这两个公式能帮助排障：TTFT 主要看网关、排队、tokenizer 和 prefill；TPOT 主要看 decode、KV 读取、调度、采样、detokenization 和网络 backpressure。

---

### 七、限流和配额

LLM 服务很容易被少数长请求打满。

限流不能只按 QPS。

因为请求成本差异很大。

更合理的限流维度包括：

1. requests per minute。
2. input tokens per minute。
3. output tokens per minute。
4. concurrent requests。
5. max context length。
6. max output length。
7. 每个用户的 KV Cache 占用。

例如一个用户每分钟只发 2 个请求，但每个请求 100k prompt + 20k output，成本非常高。

所以面试中要说：

```text
LLM rate limit 应该按 token、并发和资源预算设计，而不是只按请求数。
```

一个简单的资源账户可以写成：

```math
A_u = c_q N_{req,u} + c_i S_{in,u} + c_o S_{out,u} + c_k B_{kv,u}
```

```math
A_u \le A_{limit,u}
```

其中 `u` 是用户或租户，`N_{req,u}` 是请求数，`S_{in,u}` 是输入 token 用量，`S_{out,u}` 是输出 token 用量，`B_{kv,u}` 是 KV block 占用，`c_q,c_i,c_o,c_k` 是不同资源的计费或限流权重。这样可以避免少数超长请求用很低 QPS 打满服务。

---

### 八、多模型与 fallback

生产系统通常不只有一个模型。

可能有：

1. 大模型。
2. 小模型。
3. 长上下文模型。
4. 代码模型。
5. embedding 模型。
6. reranker。
7. safety classifier。

路由器可以根据任务选择模型。

如果大模型繁忙，可以 fallback 到小模型。

如果长上下文模型不可用，可以降级为摘要后再回答。

但 fallback 要谨慎。

需要明确：

1. 质量是否可接受。
2. 用户是否需要知道降级。
3. 安全策略是否一致。
4. 输出格式是否兼容。
5. 计费是否变化。

不能因为服务压力大，就无声切到行为差异很大的模型。

---

### 九、安全与输入输出治理

Serving 系统还要处理安全问题。

包括：

1. 输入内容审核。
2. prompt injection 检测。
3. 越权数据访问防护。
4. 输出安全过滤。
5. PII 检测和脱敏。
6. 工具调用权限控制。
7. 审计日志。

如果是 RAG 或 Agent 服务，还要特别注意：

```text
模型输出可能触发工具调用，工具调用可能产生真实副作用。
```

所以 serving 不只是生成文本。

它也是一个安全边界。

4. P50/P95/P99。
5. stream 中断率。

1. GPU utilization。

3. KV Cache used blocks。
4. KV Cache free blocks。
5. HBM bandwidth。
6. CPU tokenizer load。
7. network throughput。

#### 4. 质量和错误指标

### 十一、常见故障定位

#### 1. TTFT 变高

可能原因：

1. waiting queue 变长。
2. prompt 变长。
3. prefill token budget 太小。
4. tokenizer CPU 瓶颈。
5. 路由到繁忙副本。
6. prefix cache 命中率下降。

#### 2. TPOT 变高

可能原因：

1. running batch 过大。
2. KV Cache 读取带宽瓶颈。
3. 长上下文比例上升。
4. kernel 退化。
5. speculative decoding 接受率下降。
6. GPU 降频或资源争抢。

#### 3. OOM 增加

可能原因：

1. KV Cache 预算估计不足。
2. max_new_tokens 设置过大。
3. 长上下文请求变多。
4. block 释放不及时。
5. prefix cache 占用过高。
6. batch size 过大。

#### 4. 用户觉得卡但 tokens/s 很高

可能原因：

1. 平均吞吐高，但 P99 很差。
2. TTFT 高。
3. stream token 间隔不稳定。
4. 长请求影响短请求。
5. 网络 backpressure。
6. 前端渲染或客户端处理慢。

### 十二、容量规划

容量规划不能只问“多少 QPS”。

LLM 服务要看 token 维度。

需要估计：

1. 平均 input tokens。
2. P95/P99 input tokens。
3. 平均 output tokens。
4. P95/P99 output tokens。
5. 峰值并发。
6. 目标 TTFT/TPOT。
7. 模型权重显存。
8. KV Cache 显存。
9. 量化配置。
10. 冗余和故障切换。

一个粗略估算过程是：

```text
业务流量 -> token 分布 -> 每请求 prefill/decode 成本 -> 每 GPU 容量 -> 所需 GPU 数 -> 冗余系数 -> 成本
```

如果一个容量规划窗口长度为 `T_{window}`，窗口内请求数为 `N_{req}`，平均输入 token 为 `E_{in}`，平均输出 token 为 `E_{out}`，则需要的 prefill 和 decode 吞吐可以先估为：

```math
R_{p,need} = N_{req} * E_{in} / T_{window}
```

```math
R_{d,need} = N_{req} * E_{out} / T_{window}
```

如果单 GPU 可提供的 prefill 吞吐为 `R_{p,gpu}`，decode 吞吐为 `R_{d,gpu}`，所需 GPU 数可以粗略估为：

```math
N_{gpu} = ceil(max(R_{p,need} / R_{p,gpu}, R_{d,need} / R_{d,gpu}))
```

这只是平均负载估算。线上还要按 P95/P99 token 长度、峰值流量、故障冗余和灰度容量放大。

KV Cache 容量也要单独估算。若单卡显存为 `M_{gpu}`，权重、运行时和安全预留分别为 `M_{weight}`、`M_{runtime}`、`M_{reserve}`，则可用于 KV Cache 的显存为：

```math
M_{free} = M_{gpu} - M_{weight} - M_{runtime} - M_{reserve}
```

若每个 KV block 包含 `B_{block}` 个 token，可用 KV blocks 约为：

```math
B_{cap} = floor(M_{free} / (2 * L * H_{kv} * D_h * b_{elem} * B_{block}))
```

如果平均每个请求需要 `E_{blocks,req}` 个 KV blocks，多卡总 KV 容量为 `B_{cap,total}`，最大并发可以先估为：

```math
N_{conc,max} = floor(B_{cap,total} / E_{blocks,req})
```

这个估算最容易被两个因素打破：第一，长上下文和长输出会拉高 `E_{blocks,req}`；第二，prefix cache、fragmentation、量化元数据和运行时临时 buffer 会占用额外空间。因此容量规划中要留出保守余量，而不是把理论显存用满。

这也是第 74 讲成本模型的基础。

### 十四、真实项目中的坑

#### 1. 只按 request 数做负载均衡

两个请求成本可能差 100 倍。

LLM 服务必须关注 token 数和 KV Cache 占用。

#### 2. 没有隔离长请求

少数超长请求可能拖慢大量短请求。

可以用长上下文专用池、优先级队列或 token budget 控制。

#### 3. 忽略取消请求

用户关掉页面后，如果服务端继续生成，会浪费 GPU 和 KV Cache。

#### 4. 只看平均延迟

线上体验通常被 P95/P99 决定。

平均值好不代表服务稳定。

#### 5. tokenizer 成为 CPU 瓶颈

高 QPS 下 tokenizer 和 detokenizer 也会成为瓶颈。

不要只盯 GPU。

#### 6. fallback 行为不一致

降级模型如果安全策略、格式或质量差异太大，可能造成业务事故。

#### 问题 1：设计 LLM serving 系统时最核心的模块有哪些？

可以这样回答：

```text
核心模块包括 API gateway、鉴权限流、router、scheduler、tokenizer、GPU inference workers、KV Cache manager、batching manager、streamer、observability 和安全治理模块。
```

#### 问题 2：为什么 LLM 负载均衡不能只按请求数？

可以这样回答：

```text
LLM 请求成本主要由 input tokens、output tokens、上下文长度和 KV Cache 占用决定。两个请求数量相同，但一个是 100 token，另一个是 100k token，成本完全不同，所以路由和限流要按 token 和资源预算做。
```

#### 问题 3：Scheduler 要优化哪些指标？

可以这样回答：

```text

```

#### 问题 4：线上 TTFT 突然变高怎么排查？

可以这样回答：

```text
先看 waiting queue 和路由负载，再看 prompt length、prefill tokens/s、tokenizer CPU、prefix cache 命中率和 prefill token budget。TTFT 主要由排队和 prefill 决定。
```

#### 问题 5：为什么 tokens/s 高但用户仍然觉得卡？

可以这样回答：

```text
平均 tokens/s 高不代表交互体验好。可能 TTFT 高、P99 很差、stream token 间隔不稳定、长请求阻塞短请求、网络 backpressure 或客户端渲染慢。
```

#### 问题 6：如何做 LLM serving 容量规划？

可以这样回答：

```text
先统计 input/output token 分布、峰值并发和目标 TTFT/TPOT，再估算 prefill/decode 成本、KV Cache 显存、每 GPU tokens/s 和最大并发，最后加上冗余、灰度和故障切换容量。
```

### 十六、常见误区

1. 误区：Serving 就是把模型包成 HTTP API。
   纠正：真正 serving 包括路由、调度、KV Cache、batching、流式返回、限流、监控、安全和容错。

2. 误区：负载均衡按 request 数就够。
   纠正：LLM 成本更接近 token 数、上下文长度、输出长度和 KV Cache 占用。

3. 误区：GPU 利用率越高越好。
   纠正：利用率高可能牺牲 TTFT、TPOT 和 P99，需要结合用户体验看。

4. 误区：只要模型推理快，系统就快。
   纠正：tokenizer、队列、网络、streamer、日志和安全过滤都可能成为瓶颈。

5. 误区：长请求和短请求放一起没问题。
   纠正：长请求可能拖慢短请求，需要隔离、优先级或 token budget。

6. 误区：fallback 只要能返回文本就行。
   纠正：fallback 还要保证质量、安全、格式、计费和用户预期一致。

### 本讲总结

本讲最重要的结论：

1. LLM serving 是请求接入、路由、调度、推理、流式返回、监控和安全治理的完整系统。
2. 请求生命周期包括 gateway、router、queue、tokenization、prefill、decode、stream 和资源释放。

## 第 74 讲：成本模型：训练成本与推理成本

### 本讲目标

学完本讲，你应该能回答六个问题：

1. 训练成本和推理成本分别由哪些因素决定。
2. 为什么训练是一次性大成本，推理是持续性边际成本。
3. 如何用 token、FLOPs、GPU 小时粗略估算训练成本。
4. 如何用 input tokens、output tokens、并发、TTFT、TPOT 估算推理成本。
5. 为什么模型越大不一定业务上越划算。
6. 面试中如何讨论大模型服务的成本优化策略。

前一讲讲了 serving 系统设计。

真正上线后，系统设计还必须回答一个问题：

```text
这个服务要花多少钱？
```

大模型成本可以分成两类：

```text
训练成本：把模型训练出来的成本。
推理成本：每次服务用户请求的成本。
```

两者都和算力有关，但性质完全不同。

训练更像一次性资本投入。

推理更像持续运营成本。

---

### 一、训练成本和推理成本的区别

训练成本通常发生在模型开发阶段。

它包括：

1. 预训练。
2. 后训练。
3. 对齐训练。
4. 实验试错。
5. checkpoint 评估。
6. 失败重跑。

推理成本发生在模型上线后。

它包括：

1. 用户请求的 prefill 成本。
2. decode 生成成本。
3. KV Cache 显存成本。
4. serving 冗余成本。
5. 监控、安全、RAG、工具调用成本。
6. 峰值容量和闲置容量成本。

可以用一句话区分：

```text
训练成本主要和训练 token 数、模型规模、训练轮数、硬件效率有关；推理成本主要和线上 token 流量、输出长度、并发、延迟 SLA、服务架构有关。
```

---

### 二、训练成本的粗略公式

对 decoder-only Transformer，常见粗略估算是：

```math
F_{train} = k_{train} * P * T
```

这里 `P` 是参数量，`T` 是训练 token 数，`k_{train}` 是训练 FLOPs 系数。对常见 dense decoder-only Transformer，面试中经常用 `k_{train}=6` 做粗略估算。它不是严格物理常数，而是把 forward、backward 和梯度更新折成一个便于心算的系数。

结合 Scaling Laws 和 Chinchilla 这类工作时，要注意两点：第一，`F_{train}` 只是 compute 预算；第二，固定 compute 下还要决定参数量和训练 token 数怎么分配，不能只看模型参数越大越好。

假设：

```math
P = 7 * 10^9,
\qquad
T = 10^{12},
\qquad
k_{train}=6
```

如果模型是 7B，训练 1T token：

```math
F_{train} = 6 * 7 * 10^9 * 10^{12} = 4.2 * 10^{22}
```

要把 FLOPs 转成 GPU 时间，需要知道有效算力。

假设 GPU 数为 `G`，单卡峰值算力为 `F_{peak}`，训练有效利用率为 `U_{mfu}`：

```math
R_{eff} = G * F_{peak} * U_{mfu}
```

```math
H_{train} = F_{train} / (R_{eff} * 3600)
```

如果单 GPU 每小时成本是 `c_{gpu}`，final run 的 GPU 成本可以写成：

```math
C_{final} = H_{train} * G * c_{gpu}
```

实际中 MFU 不会等于 100%。

通信、数据加载、checkpoint、pipeline bubble、kernel 效率都会降低利用率。

---

### 三、训练成本不只是一次正式训练

很多人估算训练成本时只算 final run。

真实研发成本还包括：

1. 小规模 ablation。
2. 数据配比实验。
3. tokenizer 实验。
4. 学习率和 batch size 调参。
5. 训练失败重启。
6. loss spike 排查。
7. checkpoint 评估。
8. 后训练多轮迭代。
9. 安全和红队评估。

所以真实训练成本可以写成：

```math
C_{train,total} = C_{final} + C_{exp} + C_{fail} + C_{eval} + C_{data} + C_{people}
```

面试中如果只说 `6PT`，是不完整的。

`6PT` 是核心计算成本，不是完整项目成本。

---

### 四、推理成本的组成

推理成本通常按请求或 token 计。

一个请求可以拆成：

```text
input tokens = S
output tokens = O
```

推理过程包括：

```text
Prefill：处理 S 个输入 token
Decode：生成 O 个输出 token
```

Prefill 成本和输入长度相关。

Decode 成本和输出长度相关，也和当前上下文长度相关。

粗略端到端计算可以写成：

```math
T_{total} = T_{queue} + T_{prefill}(S) + O * T_{pot} + T_{overhead}
```

成本也可以类似拆分：

```math
C_{req} = C_{prefill}(S) + C_{decode}(O,S) + C_{kv}(S+O) + C_{sys}
```

其中 `S` 是 input tokens，`O` 是 output tokens，`T_{pot}` 是每个输出 token 的平均间隔，`C_{kv}` 表示 KV Cache 相关显存和带宽成本，`C_{sys}` 包括 tokenizer、网络、安全过滤、日志、RAG、工具调用和监控等系统开销。

---

### 五、为什么 output token 通常更贵

在商业 API 中，output token 往往比 input token 贵。

原因是 output token 需要逐步生成。

每生成一个 token，都要经过 decode step。

Decode 的特点是：

1. 串行依赖强。
2. 每步要读权重。
3. 每步要访问 KV Cache。
4. 会占用 GPU 时间直到生成结束。
5. 输出越长，占用 worker 越久。

Input token 在 prefill 阶段可以更并行地处理。

虽然长 prompt 也很贵，但 output token 对在线服务的持续占用更明显。

所以定价和成本模型中常见：

```text
output token price > input token price
```

---

### 六、推理成本的简单容量估算

假设一个 GPU worker 在某个配置下可以提供：

```text
prefill throughput = A input tokens/s
decode throughput = B output tokens/s
```

业务流量是：

```text
每秒 input tokens = I
每秒 output tokens = O
```

粗略需要的 GPU 数可以估算为：

```math
G_p = I / (A * U_{target})
```

```math
G_d = O / (B * U_{target})
```

```math
G_{need} = ceil(max(G_p,G_d) * f_{red})
```

这里 `U_{target}` 是目标利用率上限，`f_{red}` 是冗余系数。不要把 `U_{target}` 设成 1，因为线上还要给 P95/P99、突发流量、重试、灰度和故障切换留余量。

这只是粗略估计。

真实系统还要考虑：

1. P95/P99 延迟。
2. 峰值流量。
3. 请求长度分布。
4. KV Cache 容量。
5. 多模型路由。
6. 灰度和故障冗余。
7. GPU 利用率不能长期打满。

但这个估算能帮助面试中快速建立成本直觉。

---

### 七、GPU 利用率和成本

GPU 成本通常按时间计。

如果 GPU 已经租用或购买，空闲也是成本。

所以成本优化不只是让单个请求更快。

还要提高整体利用率。

但利用率不是越高越好。

如果 GPU 利用率长期接近 100%，可能导致：

1. 排队时间增加。
2. TTFT 变差。
3. P99 延迟变差。
4. 突发流量无缓冲。
5. OOM 和失败率增加。

所以线上服务通常需要在成本和 SLA 之间折中：

```text
低成本：高利用率，少冗余，但尾延迟风险高。
高 SLA：保留冗余，低排队，但单位成本更高。
```

---

### 八、影响推理成本的关键因素

#### 1. 模型大小

模型越大，权重显存、带宽和计算成本通常越高。

但大模型质量更好，可能减少重试、减少人工修正，业务价值更高。

#### 2. 上下文长度

长 prompt 增加 prefill 时间和 KV Cache。

长上下文还会增加 decode 每步读取 KV 的成本。

#### 3. 输出长度

输出越长，decode step 越多。

这是推理成本的核心变量。

#### 4. 并发和 SLA

高并发需要更多 KV Cache 和更复杂调度。

严格 TTFT/TPOT SLA 需要更多冗余容量。

#### 5. 硬件和 kernel

同一模型在不同 GPU、不同推理引擎、不同 kernel 下成本差异很大。

#### 6. 量化和压缩

量化可以降低显存和带宽，但可能带来质量风险和 kernel 依赖。

#### 7. Cache 命中率

Prefix cache、prompt cache、RAG cache、embedding cache 都能降低重复计算。

---

### 九、降本策略

#### 1. 模型侧降本

包括：

1. 使用更小模型。
2. 蒸馏。
3. MoE 或路由模型。
4. 量化。
5. MQA/GQA 降低 KV Cache。
6. 更短 tokenizer 表示。
7. 训练专用小模型处理高频任务。

#### 2. 推理引擎降本

包括：

1. PagedAttention。
2. Continuous Batching。
3. FlashAttention。
4. Speculative Decoding。
5. KV Cache 量化。
6. 高效 kernel。
7. Tensor Parallel 和 Pipeline Parallel 调优。

#### 3. 产品和请求侧降本

包括：

1. 控制 max_tokens。
2. 压缩 prompt。
3. 摘要历史对话。
4. RAG 只检索必要文档。
5. 缓存常见回答。
6. 对简单任务路由到小模型。
7. 对后台任务放宽延迟 SLA。

#### 4. 系统侧降本

包括：

1. 提高 batch 利用率。
2. 分离长短请求。
3. 峰谷调度。
4. 多区域容量规划。
5. 预留实例和弹性实例组合。
6. 灰度发布减少失败重算。

---

### 十、质量和成本的 trade-off

成本优化不能只看便宜。

每个优化都可能影响质量。

例如：

1. 小模型成本低，但复杂任务质量下降。
2. INT4 省显存，但可能影响代码和数学。
3. 缩短 prompt 省成本，但可能丢上下文。
4. 限制 output length 省成本，但回答可能不完整。
5. RAG top-k 减少省成本，但可能漏证据。
6. 高 batch 提升吞吐，但可能增加 TTFT。

所以更合理的目标是：

```text
在满足质量和 SLA 的前提下，最小化单位有效任务成本。
```

不是单纯最小化 token 成本。

---

### 十一、单位有效任务成本

一个有用指标是单位有效任务成本。

例如：

```text
Cost per successful answer
Cost per resolved ticket
Cost per accepted code completion
Cost per correct math solution
```

如果小模型便宜 10 倍，但成功率低很多，需要多次重试或人工介入，最终未必便宜。

可以粗略写成：

```math
C_{success} = C_{attempt} / p_{success}
```

如果考虑重试：

```math
C_{success} = (C_{attempt} * (1 + r_{retry}) + C_{manual} + C_{fail}) / p_{success}
```

这里 `C_{attempt}` 是一次尝试的请求成本，`p_{success}` 是一次任务成功率，`r_{retry}` 是平均重试率，`C_{manual}` 是人工修正成本，`C_{fail}` 是失败带来的业务损失或补救成本。

这就是为什么业务上不一定选最便宜模型。

要选性价比最高的模型和系统方案。

#### 最小 Python demo：训练成本、推理容量和有效任务成本

下面的 demo 使用假设数字，不代表任何云厂商真实价格。它演示三件事：final run 只是训练成本的一部分；推理容量要同时看 prefill 和 decode；单次请求便宜的模型，如果成功率低、重试多，单位成功任务成本可能反而更高。

```python
from dataclasses import dataclass
from math import ceil


@dataclass
class TrainRun:
    params_b: float
    tokens_t: float
    gpu_count: int
    peak_tflops_per_gpu: float
    mfu: float
    gpu_hour_usd: float


@dataclass
class ServingPlan:
    name: str
    prefill_tps_per_gpu: float
    decode_tps_per_gpu: float
    gpu_hour_usd: float
    success_rate: float
    retry_rate: float
    manual_fix_rate: float


def train_cost(run):
    params = run.params_b * 1e9
    tokens = run.tokens_t * 1e12
    flops = 6 * params * tokens
    effective_flops_s = run.gpu_count * run.peak_tflops_per_gpu * 1e12 * run.mfu
    hours = flops / effective_flops_s / 3600
    cost = hours * run.gpu_count * run.gpu_hour_usd
    return flops, hours, cost


def required_gpus(plan, req_s, input_tokens, output_tokens, peak_factor, target_util, redundancy):
    peak_input_tps = req_s * input_tokens * peak_factor
    peak_output_tps = req_s * output_tokens * peak_factor
    prefill_gpus = peak_input_tps / (plan.prefill_tps_per_gpu * target_util)
    decode_gpus = peak_output_tps / (plan.decode_tps_per_gpu * target_util)
    return ceil(max(prefill_gpus, decode_gpus) * redundancy)


def request_cost(plan, gpu_count, avg_req_s, overhead_usd=0.00025):
    gpu_hour = gpu_count * plan.gpu_hour_usd
    hardware_per_request = gpu_hour / (avg_req_s * 3600)
    attempt_cost = hardware_per_request + overhead_usd
    total_before_success = attempt_cost * (1 + plan.retry_rate) + plan.manual_fix_rate * 0.03
    effective_cost = total_before_success / plan.success_rate
    return hardware_per_request, attempt_cost, effective_cost


train = TrainRun(
    params_b=7,
    tokens_t=1,
    gpu_count=64,
    peak_tflops_per_gpu=312,
    mfu=0.35,
    gpu_hour_usd=2.5,
)
flops, hours, cost = train_cost(train)
print("training_flops=", f"{flops:.2e}")
print("final_run_hours=", round(hours, 1))
print("final_run_cost_usd=", round(cost))
print("with_30pct_experiment_overhead_usd=", round(cost * 1.30))

plans = [
    ServingPlan("small_7b", 24000, 5500, 2.5, success_rate=0.86, retry_rate=0.18, manual_fix_rate=0.040),
    ServingPlan("large_13b", 15000, 3200, 3.2, success_rate=0.93, retry_rate=0.08, manual_fix_rate=0.015),
]

req_s = 12
avg_in = 900
avg_out = 350
for plan in plans:
    gpus = required_gpus(plan, req_s, avg_in, avg_out,
                         peak_factor=2.0, target_util=0.65, redundancy=1.25)
    hardware_req, attempt_req, effective = request_cost(plan, gpus, req_s)
    print(f"model= {plan.name}")
    print("  required_gpus=", gpus)
    print("  hardware_cost_per_request_usd=", round(hardware_req, 6))
    print("  attempt_cost_per_request_usd=", round(attempt_req, 6))
    print("  cost_per_successful_answer_usd=", round(effective, 6))
```

一组输出如下：

```text
training_flops= 4.20e+22
final_run_hours= 1669.3
final_run_cost_usd= 267094
with_30pct_experiment_overhead_usd= 347222
model= small_7b
  required_gpus= 3
  hardware_cost_per_request_usd= 0.000174
  attempt_cost_per_request_usd= 0.000424
  cost_per_successful_answer_usd= 0.001977
model= large_13b
  required_gpus= 6
  hardware_cost_per_request_usd= 0.000444
  attempt_cost_per_request_usd= 0.000694
  cost_per_successful_answer_usd= 0.00129
```

这个例子里，`small_7b` 的单次请求成本更低，但因为成功率低、重试和人工修正更多，单位成功回答成本反而高于 `large_13b`。真实系统中还要把缓存命中、峰值容量、失败率、质量指标和业务价值一起放进 dashboard。

### 十二、训练 vs 推理：什么时候自己训练划算

一个常见业务问题是：

```text
应该调用外部 API，还是自己训练/部署模型？
```

需要比较：

1. 外部 API 单 token 价格。
2. 自部署 GPU 成本。
3. 研发和运维人力。
4. 数据和合规要求。
5. 流量规模。
6. 延迟要求。
7. 定制化需求。
8. 模型质量差距。

如果流量小、需求变化快，API 可能更划算。

如果流量大、任务稳定、对数据和延迟有强要求，自部署或微调小模型可能更划算。

面试中不要绝对回答。

应该说：

```text
看流量规模、质量要求、数据合规、延迟 SLA 和团队运维能力。
```

### 十三、成本监控 dashboard

线上要有成本 dashboard。

至少包括：

1. input tokens/day。
2. output tokens/day。
3. 每模型 tokens/s。
4. 每模型 GPU 小时。
5. 每模型单位 token 成本。
6. 每业务线成本。
7. cache hit rate。
8. 平均 prompt length。
9. 平均 output length。
10. P95/P99 output length。
11. GPU 利用率。
12. 闲置容量。
13. 失败和重试成本。

成本监控要和质量监控一起看。

如果成本下降但失败率上升，可能只是把成本转移到用户或人工修正上。

### 十四、真实项目中的坑

#### 1. 只看模型单 token 价格

真实成本还包括 RAG、工具调用、安全过滤、重试、日志、监控和人力。

#### 2. 忽略输出长度

输出 token 往往更贵。

没有 max_tokens 和 stop 策略，很容易成本失控。

#### 3. 过度压缩 prompt

prompt 太短可能降低质量，导致重试和人工修正增加。

#### 4. GPU 利用率过高

短期成本低，但 P99 变差，用户体验下降。

#### 5. 只算成功请求成本

失败、超时、取消、重试也消耗资源。

#### 6. 没有按业务线分账

不知道哪个产品、客户、功能最耗钱，就无法有效优化。

### 十五、面试问答

#### 问题 1：训练成本怎么粗略估算？

可以这样回答：

```text
decoder-only Transformer 训练 FLOPs 可粗略估算为 6 * 参数量 * 训练 token 数。再除以集群有效算力得到训练时间。真实成本还要加实验、失败重跑、评估、数据和人力成本。
```

#### 问题 2：推理成本由哪些因素决定？

可以这样回答：

```text
主要由 input tokens、output tokens、模型大小、上下文长度、并发、TTFT/TPOT SLA、KV Cache、硬件、推理引擎、量化配置和系统冗余决定。
```

#### 问题 3：为什么 output token 通常比 input token 贵？

可以这样回答：

```text
input token 在 prefill 阶段可以并行处理，而 output token 要逐 token decode，每个 token 都占用一次模型 forward、读取权重和 KV Cache，并持续占用 GPU worker，所以边际成本更高。
```

#### 问题 4：如何降低 LLM 推理成本？

可以这样回答：

```text
可以从模型、引擎、产品和系统四层降本：小模型或蒸馏、量化和 MQA/GQA；PagedAttention、continuous batching、speculative decoding；压缩 prompt、限制 max_tokens、缓存和模型路由；峰谷调度、长短请求隔离和容量规划。
```

#### 问题 5：为什么最便宜模型不一定最划算？

可以这样回答：

```text
要看单位有效任务成本。如果便宜模型成功率低、需要重试或人工修正，最终成本可能更高。应该比较 cost per successful answer，而不是只看单 token 价格。
```

#### 问题 6：什么时候自部署比 API 更划算？

可以这样回答：

```text
取决于流量规模、任务稳定性、延迟 SLA、数据合规、定制化需求和团队运维能力。小流量或快速试错通常 API 更划算；大流量、强合规、稳定任务和低延迟要求下，自部署可能更划算。
```

---

### 十六、常见误区

1. 误区：训练成本只等于 final run 成本。
   纠正：实验、失败、评估、数据和人力都是成本。

2. 误区：推理成本只看模型参数量。
   纠正：还要看 input/output tokens、KV Cache、并发、SLA、冗余和系统开销。

3. 误区：GPU 利用率越高成本越好。
   纠正：利用率过高会牺牲 P99 和突发缓冲。

4. 误区：压缩 prompt 一定降本。
   纠正：如果质量下降导致重试，整体成本可能更高。

5. 误区：小模型一定更划算。
   纠正：要看成功率、重试率、人工修正和业务价值。

6. 误区：只优化平均成本。
   纠正：P95/P99 长请求和异常重试往往贡献大量成本。

---

### 十七、小练习

1. 用 `6 * P * T` 估算一个 7B 模型训练 1T token 的 FLOPs。
2. 解释为什么真实训练成本高于 final run 计算成本。
3. 设计一个推理成本公式，包含 input tokens、output tokens、模型、KV Cache 和系统 overhead。
4. 比较 input token 和 output token 成本差异。
5. 给定 prefill tokens/s 和 decode tokens/s，估算需要多少 GPU 承载某个流量。
6. 列出 10 个降低推理成本的方法，并说明可能的质量风险。
7. 设计一个 cost dashboard，按模型、业务线和 token 类型统计成本。
8. 比较 API 调用和自部署在不同流量规模下的成本结构。
9. 用单位有效任务成本解释为什么小模型不一定更便宜。
10. 用 3 分钟回答：“如何为一个 LLM 产品做成本优化？”

### 本讲总结

本讲最重要的结论：

1. 训练成本主要由参数量、训练 token 数、训练效率、实验次数和失败重跑决定。
2. 推理成本主要由 input/output tokens、模型大小、上下文长度、并发、SLA、KV Cache 和系统冗余决定。
3. 训练是一次性大成本，推理是持续性边际成本，流量越大推理成本越关键。
4. Output token 通常比 input token 贵，因为 decode 是逐 token 串行生成并持续占用 GPU。
5. 降本可以从模型、推理引擎、产品请求和系统容量四层做。
6. 成本优化必须和质量、成功率、P95/P99 和用户体验一起评估。
7. 单位有效任务成本比单 token 价格更接近业务真实成本。
8. 面试中讨论成本要体现 trade-off：质量、延迟、吞吐、合规、运维能力和业务价值。

## 第 75 讲：端侧与小模型部署

### 本讲目标

学完本讲，你应该能回答六个问题：

1. 端侧部署和云端部署的核心差异是什么。
2. 小模型为什么在真实产品中仍然重要。
3. 端侧部署受到哪些硬件、内存、功耗和延迟约束。
4. 量化、蒸馏、剪枝、LoRA 合并在小模型部署中分别起什么作用。
5. 端云协同如何在隐私、成本、质量和延迟之间折中。
6. 面试中如何设计一个端侧 LLM 或端云混合 AI 助手。

前面几讲主要讨论云端 GPU serving。

但真实产品中，很多场景并不适合所有请求都走云端大模型。

原因包括：

1. 网络不稳定。
2. 隐私要求高。
3. 延迟要求极低。
4. 云端成本高。
5. 设备需要离线可用。
6. 高频轻量任务不值得调用大模型。

这就引出端侧与小模型部署。

---

### 一、什么是端侧部署

端侧部署指模型直接运行在用户设备或边缘设备上。

例如：

1. 手机。
2. 笔记本。
3. 浏览器。
4. 智能音箱。
5. 汽车座舱。
6. AR/VR 设备。
7. 工业边缘盒子。
8. 企业内网机器。

端侧部署的目标通常不是追求最大模型能力。

而是在受限资源下提供可接受的智能能力。

它强调：

```text
低延迟、低成本、隐私保护、离线可用、稳定功耗。
```

---

### 二、端侧和云端的核心差异

可以用一张表总结：

```text
维度            云端部署                         端侧部署
算力            GPU 集群，可扩展                   手机/NPU/CPU/GPU，受限
内存            数十 GB 到数百 GB                  几 GB 甚至更少
网络            用户依赖网络访问                    可离线或弱网运行
隐私            数据上传到服务端                    数据可留在本地
成本            按云资源和 token 持续计费            单设备本地运行，边际云成本低
延迟            受网络和排队影响                    本地响应快，但模型较小
更新            服务端统一更新                      端侧版本碎片化
质量            可用大模型                          通常使用小模型或压缩模型
```

一句话：

```text
云端胜在能力和弹性，端侧胜在隐私、离线、低延迟和边际成本。
```

---

### 三、小模型为什么重要

大模型能力强，但不是所有任务都需要大模型。

小模型在很多场景很有价值：

1. 意图识别。
2. 文本分类。
3. 简单问答。
4. 输入法补全。
5. 本地摘要。
6. 邮件分类。
7. 端侧语音指令。
8. 安全过滤。
9. 工具路由。
10. 大模型调用前的预处理。

小模型的优势是：

1. 延迟低。
2. 成本低。
3. 易部署。
4. 隐私好。
5. 可离线。
6. 可针对特定任务优化。

所以真实系统常常不是一个大模型打天下，而是：

```text
小模型处理高频简单任务，大模型处理低频复杂任务。
```

---

### 四、端侧部署的硬件约束

端侧设备通常有严格限制。

#### 1. 内存限制

模型权重、KV Cache、运行时 buffer 都要占内存。

手机上即使有 8GB 或 16GB 内存，也不可能全部给模型使用。

#### 2. 算力限制

端侧可能使用：

1. CPU。
2. 移动 GPU。
3. NPU。
4. DSP。
5. WebGPU。

不同硬件支持的算子和精度不同。

#### 3. 功耗和发热

模型持续运行会耗电和发热。

端侧产品要考虑：

1. 电池续航。
2. 机身温度。
3. 降频。
4. 长时间使用稳定性。

#### 4. 存储大小

模型包体不能太大。

一个几 GB 的模型可能影响下载安装和更新。

#### 5. 系统碎片化

不同设备、系统版本、芯片、驱动、NPU SDK 差异很大。

端侧部署要面对更复杂的兼容性测试。

#### 资源预算公式

端侧部署首先要问模型能不能装得下、加载得起、跑得稳。

模型权重内存可以先估为：

```math
M_w = P * b_w / 8
```

其中 `P` 是参数量，`b_w` 是每个权重的 bit 数。比如 1.5B 参数用 INT4 权重，权重本体约为 `1.5e9 * 4 / 8` bytes，还没有算量化 scale、runtime、KV Cache 和临时 buffer。

LLM 生成时还要估 KV Cache。单 token KV Cache 字节数可以写成：

```math
M_{kv,tok} = 2 * L * H_{kv} * D_h * b_{kv}
```

完整请求的端侧峰值内存可以粗略写成：

```math
M_{edge} = M_w + (S+O) * M_{kv,tok} + M_{act} + M_{runtime}
```

```math
M_{edge} \le M_{budget}
```

这里 `S` 是输入 token 数，`O` 是输出 token 数，`L` 是层数，`H_{kv}` 是 KV heads 数，`D_h` 是 head 维度，`b_{kv}` 是 KV 元素字节数，`M_{act}` 是激活和临时 buffer，`M_{runtime}` 是 runtime 和执行环境额外开销。

包体大小也要单独约束：

```math
D_{pkg} = D_{app} + D_{runtime} + D_{model} + D_{assets}
```

```math
D_{pkg} \le D_{limit}
```

这就是为什么端侧模型经常要做 INT4/INT8、GGUF、Core ML compression、LiteRT/ONNX Runtime mobile 优化和自定义 runtime build。

---

### 五、端侧模型压缩手段

#### 1. 量化

端侧最常见的是 INT8、INT4 甚至更低精度量化。

它可以降低模型大小、内存和带宽。

端侧量化还要考虑硬件是否支持对应精度。

例如某些 NPU 对 INT8 支持好，但对 INT4 不一定高效。

#### 2. 蒸馏

用大模型作为 teacher，训练小模型作为 student。

蒸馏目标是让小模型学会大模型在特定任务上的行为。

适合高频、明确、任务边界清楚的场景。

#### 3. 剪枝

删除不重要的权重、通道、head 或层。

剪枝可以减少计算和参数，但结构化剪枝更容易获得真实加速。

非结构化稀疏如果没有硬件和 kernel 支持，未必加速。

#### 4. LoRA 合并

如果小模型通过 LoRA 微调，部署时通常可以把 LoRA 权重合并回 base model。

这样推理时不需要额外 LoRA 分支，系统更简单。

#### 5. 架构选择

端侧可以选择更适合小模型的架构：

1. GQA/MQA 降低 KV Cache。
2. 较小 hidden size。
3. 较少层数。
4. 较短上下文。
5. 更高效 tokenizer。

#### 压缩收益和蒸馏目标

量化后的权重大小可以先按 bit 数估算：

```math
M_{q} = P * b_q / 8 + M_{meta}
```

压缩比可以写成：

```math
R_{comp} = M_{fp} / M_q
```

其中 `b_q` 是量化 bit 数，`M_{meta}` 是 scale、zero point、group metadata 等额外开销。端侧不能只看 `R_{comp}`，还要看硬件是否有对应 kernel；如果 runtime 需要先解压成 FP16，包体变小不一定代表延迟也下降。

知识蒸馏可以用一个简单目标表达：

```math
L_{distill} = \alpha L_{label} + (1-\alpha) * \tau^2 * D_{\mathrm{KL}}(p_t^{\tau} || p_s^{\tau})
```

其中 `p_t` 是 teacher 分布，`p_s` 是 student 分布，`tau` 是温度，`alpha` 控制真实标签损失和蒸馏损失的权重。端侧小模型通常不追求覆盖所有通用能力，而是追求在明确任务集合上的质量、延迟和稳定性。

结构化剪枝后的参数量和计算量可以粗略写成：

```math
P_{new} = (1-r) * P
```

```math
C_{new} = (1-r_c) * C
```

`r` 是参数剪枝比例，`r_c` 是真实计算削减比例。非结构化稀疏可能让 `r` 很大，但如果硬件和 kernel 不支持，`r_c` 可能很小。

### 六、端侧推理框架

端侧部署通常依赖专门 runtime。

常见方向包括：

1. llama.cpp / GGUF。
2. ONNX Runtime。
3. TensorRT / TensorRT-LLM。
4. Core ML。
5. Android NNAPI。
6. MediaPipe。
7. WebGPU / WebNN。
8. 厂商 NPU SDK。

选择 runtime 时要看：

1. 支持哪些算子。
2. 支持哪些量化格式。
3. 是否支持 KV Cache。
4. 是否支持流式生成。
5. CPU/GPU/NPU fallback。
6. 跨平台能力。
7. 模型转换工具链是否稳定。

端侧常见问题不是模型本身，而是转换和 runtime 兼容性。

---

### 七、端侧 LLM 的性能指标

云端常看 TTFT、TPOT、throughput、P99。

端侧也看这些，但还要看：

1. 模型包体大小。
2. 峰值内存。
3. 首次加载时间。
4. 冷启动时间。
5. tokens/s。
6. 电量消耗。
7. 发热和降频。
8. 不同设备分位表现。
9. 离线可用率。
10. 崩溃率。

例如一个模型在高端手机上 20 tokens/s，不代表在中端设备上可用。

端侧评估必须按设备分层。

端侧延迟可以拆成：

```math
T_{edge} = T_{load} + S / R_{p,edge} + O / R_{d,edge} + T_{detok} + T_{ui}
```

其中 `T_{load}` 是模型加载和冷启动时间，`R_{p,edge}` 是端侧 prefill tokens/s，`R_{d,edge}` 是端侧 decode tokens/s，`T_{detok}` 是 detokenization 时间，`T_{ui}` 是 UI 渲染和应用线程开销。

能耗可以粗略估算为：

```math
E_{edge} = P_{avg} * T_{edge}
```

如果一个交互窗口的能耗预算是 `E_{budget}`，则需要满足：

```math
E_{edge} \le E_{budget}
```

这也是端侧和云端不同的地方：云端主要付 GPU 账单，端侧还要付电量、温度和用户体验账单。

---

### 八、端云协同

很多实际产品采用端云协同。

基本思想是：

```text
简单、隐私、低延迟任务在端侧处理；复杂、高质量任务走云端大模型。
```

典型策略包括：

1. 端侧意图识别，云端复杂回答。
2. 端侧安全过滤，云端生成。
3. 端侧摘要压缩历史，云端长回答。
4. 端侧小模型先答，不确定时升级云端。
5. 端侧离线模式，联网后云端增强。
6. 端侧处理隐私字段，云端只看脱敏内容。

端云协同的关键是路由策略。

路由可以根据：

1. 任务难度。
2. 用户网络状态。
3. 隐私等级。
4. 设备能力。
5. 云端负载。
6. 成本预算。
7. 置信度。

决定端侧还是云端执行。

可以把端云协同路由写成一个带约束的打分问题。

端侧候选必须先满足：

```math
M_{edge} \le M_{budget}
```

```math
S+O \le C_{ctx,edge}
```

```math
Q_{edge} \ge Q_{min}
```

然后比较端侧和云端的代价：

```math
C_{edge} = w_l T_{edge} + w_e E_{edge} + w_m M_{edge}/M_{budget} - w_p P_{priv}
```

```math
C_{cloud} = w_l T_{cloud} + w_c C_{api} + w_n N_{risk} + w_p P_{priv}
```

这里 `Q_{edge}` 是端侧模型质量估计，`Q_{min}` 是最低可接受质量，`C_{ctx,edge}` 是端侧上下文上限，`P_{priv}` 是隐私敏感度，`N_{risk}` 是网络不稳定风险，`C_{api}` 是云端调用成本。隐私敏感时端侧得分降低、云端得分升高；复杂任务或端侧质量不足时，云端更容易胜出。

#### 最小 Python demo：端侧资源预算和端云路由

下面的 demo 用假设数字模拟一个 1.5B INT4 端侧模型。它会计算权重内存、每 token KV Cache、端侧峰值内存、端侧延迟、能耗、端侧质量估计，并根据隐私、网络和质量约束决定走端侧、云端或离线降级。

```python
from dataclasses import dataclass

MB = 1024 * 1024


@dataclass
class EdgeModel:
    params_b: float
    weight_bits: int
    layers: int
    kv_heads: int
    head_dim: int
    kv_bytes: int
    max_context: int
    prefill_tps: float
    decode_tps: float
    load_s: float
    runtime_mb: float
    activation_mb: float
    avg_power_w: float
    base_quality: float


@dataclass
class Request:
    name: str
    input_tokens: int
    output_tokens: int
    privacy: float
    complexity: float
    network_ok: bool
    cold_start: bool = False


def weight_mb(model):
    return model.params_b * 1e9 * model.weight_bits / 8 / MB


def kv_mb_per_token(model):
    bytes_per_token = 2 * model.layers * model.kv_heads * model.head_dim * model.kv_bytes
    return bytes_per_token / MB


def edge_memory_mb(model, req):
    tokens = req.input_tokens + req.output_tokens
    return weight_mb(model) + tokens * kv_mb_per_token(model) + model.runtime_mb + model.activation_mb


def edge_latency_s(model, req):
    load = model.load_s if req.cold_start else 0.0
    return load + req.input_tokens / model.prefill_tps + req.output_tokens / model.decode_tps


def edge_energy_j(model, latency_s):
    return model.avg_power_w * latency_s


def edge_quality(model, req):
    length_penalty = max(0.0, (req.input_tokens + req.output_tokens - 1024) / 4096) * 0.04
    return model.base_quality - 0.22 * req.complexity - length_penalty


def cloud_latency_s(req):
    network_s = 0.22
    queue_s = 0.18
    prefill_tps = 18000
    decode_tps = 130
    return network_s + queue_s + req.input_tokens / prefill_tps + req.output_tokens / decode_tps


def route(model, req, memory_budget_mb, q_min):
    total_tokens = req.input_tokens + req.output_tokens
    mem = edge_memory_mb(model, req)
    latency = edge_latency_s(model, req)
    energy = edge_energy_j(model, latency)
    quality = edge_quality(model, req)
    feasible = (
        total_tokens <= model.max_context
        and mem <= memory_budget_mb
        and quality >= q_min
    )
    if not req.network_ok:
        decision = "edge" if feasible else "offline_degrade"
    elif req.privacy >= 0.80 and feasible:
        decision = "edge"
    elif not feasible:
        decision = "cloud"
    else:
        edge_score = latency + 0.025 * energy - 0.60 * req.privacy
        cloud_score = cloud_latency_s(req) + 0.45 * req.privacy + 0.15 * req.complexity
        decision = "edge" if edge_score <= cloud_score else "cloud"
    return decision, mem, latency, energy, quality, feasible


model = EdgeModel(
    params_b=1.5,
    weight_bits=4,
    layers=24,
    kv_heads=8,
    head_dim=64,
    kv_bytes=2,
    max_context=2048,
    prefill_tps=2200,
    decode_tps=24,
    load_s=0.45,
    runtime_mb=320,
    activation_mb=160,
    avg_power_w=5.5,
    base_quality=0.86,
)

requests = [
    Request("private_note", 900, 120, privacy=0.95, complexity=0.25, network_ok=True, cold_start=True),
    Request("code_debug", 1600, 480, privacy=0.25, complexity=0.90, network_ok=True),
    Request("offline_intent", 60, 20, privacy=0.70, complexity=0.10, network_ok=False),
    Request("long_contract", 3600, 500, privacy=0.90, complexity=0.55, network_ok=True),
]

print("edge_weight_mb=", round(weight_mb(model), 1))
print("edge_kv_mb_per_token=", round(kv_mb_per_token(model), 4))
for req in requests:
    decision, mem, latency, energy, quality, feasible = route(model, req, memory_budget_mb=1800, q_min=0.72)
    print(f"request= {req.name} decision= {decision}")
    print(
        "  edge_feasible=", feasible,
        "edge_mem_mb=", round(mem, 1),
        "edge_latency_s=", round(latency, 2),
        "edge_energy_j=", round(energy, 1),
        "edge_quality=", round(quality, 3),
        "cloud_latency_s=", round(cloud_latency_s(req), 2),
    )
```

一组输出如下：

```text
edge_weight_mb= 715.3
edge_kv_mb_per_token= 0.0469
request= private_note decision= edge
  edge_feasible= True edge_mem_mb= 1243.1 edge_latency_s= 5.86 edge_energy_j= 32.2 edge_quality= 0.805 cloud_latency_s= 1.37
request= code_debug decision= cloud
  edge_feasible= False edge_mem_mb= 1292.8 edge_latency_s= 20.73 edge_energy_j= 114.0 edge_quality= 0.652 cloud_latency_s= 4.18
request= offline_intent decision= edge
  edge_feasible= True edge_mem_mb= 1199.0 edge_latency_s= 0.86 edge_energy_j= 4.7 edge_quality= 0.838 cloud_latency_s= 0.56
request= long_contract decision= cloud
  edge_feasible= False edge_mem_mb= 1387.4 edge_latency_s= 22.47 edge_energy_j= 123.6 edge_quality= 0.709 cloud_latency_s= 4.45
```

这个例子说明：隐私高的 `private_note` 即使云端更快，也可能为了隐私走端侧；`offline_intent` 在断网时只能走端侧；`code_debug` 和 `long_contract` 虽然能估出端侧内存，但质量或上下文约束不满足，应该走云端或降级处理。

### 九、隐私与安全

端侧部署的一个重要价值是隐私。

例如：

1. 用户输入不出设备。
2. 本地文件不上传。
3. 通讯录、位置、日历等敏感数据可本地处理。
4. 企业内网数据不出边界。

但端侧也有安全问题：

1. 模型权重可能被提取。
2. jailbreak 和 prompt injection 仍然存在。
3. 本地工具调用可能有权限风险。
4. 端侧日志可能泄露隐私。
5. 离线模型难以及时更新安全策略。

所以端侧不是天然安全。

它只是减少数据上传风险，同时增加本地模型保护和版本管理问题。

---

### 十、模型更新与版本管理

云端模型可以集中更新。

端侧模型更新更难。

问题包括：

1. 包体大。
2. 用户不一定及时升级。
3. 不同设备支持不同模型。
4. 回滚困难。
5. A/B 测试周期长。
6. 安全补丁下发慢。

因此端侧需要版本管理策略：

1. 分设备下发。
2. 增量更新。
3. 灰度发布。
4. 强制安全更新。
5. 模型能力和 runtime 版本匹配。
6. 本地 fallback。

---

### 十一、端侧产品设计 trade-off

端侧部署要在多个目标之间平衡：

```text
质量 vs 模型大小
速度 vs 功耗
隐私 vs 云端能力
离线能力 vs 更新难度
低成本 vs 设备兼容性
本地工具能力 vs 安全风险
```

例如：

1. 模型更大，质量更好，但更耗电、更慢、更占包体。
2. INT4 更省内存，但可能降低质量。
3. 完全本地更隐私，但复杂任务能力不足。
4. 云端增强能力强，但有网络和隐私成本。

端侧产品通常需要明确边界：

```text
哪些任务端侧必须能做，哪些任务可以降级，哪些任务必须走云端。
```

### 十二、端侧部署实验清单

上线前至少要测试：

1. 不同设备档位。
2. 冷启动时间。
3. 首 token 延迟。
4. tokens/s。
5. 峰值内存。
6. 电量消耗。
7. 连续运行发热。
8. 降频后的性能。
9. 离线场景。
10. 弱网切换云端场景。
11. 崩溃率。
12. 模型更新和回滚。
13. 隐私数据是否出端。
14. 本地安全策略。

端侧不是只跑通一个 demo。

真实上线要面对设备矩阵和长时间稳定性。

### 十三、面试问答

#### 问题 1：端侧部署和云端部署有什么区别？

可以这样回答：

```text
云端部署算力强、模型大、易统一更新，但有网络、隐私和持续成本；端侧部署资源受限，但低延迟、可离线、隐私更好、云端边际成本低。二者通常需要端云协同。
```

#### 问题 2：为什么小模型仍然重要？

可以这样回答：

```text
很多高频任务不需要大模型，例如意图识别、分类、简单补全、安全过滤和工具路由。小模型延迟低、成本低、易部署、可离线，适合处理简单任务或作为大模型前置模块。
```

#### 问题 3：端侧部署最重要的约束有哪些？

可以这样回答：

```text
主要约束包括内存、算力、功耗、发热、包体大小、硬件算子支持、系统碎片化和模型更新。端侧不仅要看 tokens/s，还要看冷启动、电量、温度、崩溃率和设备分层表现。
```

#### 问题 4：端侧模型如何压缩？

可以这样回答：

```text
常用方法包括 INT8/INT4 量化、知识蒸馏、结构化剪枝、LoRA 合并和选择更小更高效的架构。压缩要和端侧 runtime、硬件支持和质量评估一起考虑。
```

#### 问题 5：什么是端云协同？

可以这样回答：

```text
端云协同是把简单、隐私、低延迟任务放在端侧，把复杂、高质量任务交给云端大模型。路由可以根据任务难度、隐私等级、网络状态、设备能力、成本和置信度决定。
```

#### 问题 6：设计端侧 AI 助手时如何做取舍？

可以这样回答：

```text
先明确端侧必须支持的核心任务和设备约束，再选择小模型和量化方案；简单任务本地执行，复杂任务云端增强；同时评估质量、延迟、内存、功耗、隐私、安全、更新和 fallback。
```

### 十四、常见误区

1. 误区：端侧模型越大越好。
   纠正：端侧要平衡质量、速度、内存、功耗、发热和包体。

2. 误区：端侧部署天然安全。
   纠正：数据不上传降低隐私风险，但模型权重、本地工具权限和日志仍有安全问题。

3. 误区：云端大模型可以完全替代端侧小模型。
   纠正：弱网、离线、隐私和低延迟场景仍需要端侧能力。

4. 误区：量化后能跑就可以上线。
   纠正：还要测设备分层、功耗、发热、崩溃率、质量和长时间稳定性。

5. 误区：端侧 demo 跑通等于产品可用。
   纠正：真实产品要处理设备碎片化、更新、回滚、安全和异常场景。

6. 误区：端云协同只是失败时 fallback。
   纠正：端云协同是主动路由策略，要综合任务难度、隐私、网络、成本和质量。

### 本讲总结

本讲最重要的结论：

1. 端侧部署强调低延迟、离线、隐私和低云端边际成本，但受内存、算力、功耗和兼容性约束。
2. 小模型适合处理高频、简单、隐私敏感或离线任务，也可以作为大模型前置模块。
3. 端侧压缩常用量化、蒸馏、剪枝、LoRA 合并和高效架构选择。
4. 端侧 runtime 选择要看算子、量化格式、KV Cache、流式生成和硬件支持。
5. 端云协同是在隐私、质量、延迟、成本和设备能力之间做动态路由。
6. 端侧评估必须覆盖设备分层、冷启动、tokens/s、内存、功耗、发热、崩溃和更新回滚。
7. 端侧并非天然安全，仍要考虑模型权重、本地权限、日志和安全策略更新。
8. 面试中要把端侧部署讲成产品、系统、模型压缩和硬件约束共同作用的问题。
