数据并行是分布式训练的入门，也是理解后续 ZeRO、FSDP、Tensor Parallel、Pipeline Parallel 的基础。

很多人用过 PyTorch DDP，但面试时只会说“多卡训练，每张卡算一部分数据，然后同步梯度”。

这句话是对的，但不够。

更好的回答需要讲清：

1. 同步的是什么。
2. 为什么同步后等价于大 batch 训练。
3. 通信开销在哪里。
4. 显存为什么没有随着卡数增加而降低。
5. 为什么大模型训练不能只靠 DDP。

### 一、为什么需要数据并行

假设你只有一张 GPU。

每一步训练流程是：

1. 取一个 batch。
2. forward 得到 loss。
3. backward 得到梯度。
4. optimizer step 更新参数。

如果模型和 batch 都能放进一张卡，这样可以工作。

但大模型训练会遇到两个问题。

第一，训练数据太多。

一张卡处理 token 太慢，训练时间不可接受。

第二，需要更大的 global batch。

大模型通常使用较大的 batch 来提高吞吐和稳定梯度估计。

数据并行的直觉是：

> 把同一个模型复制到多张 GPU 上，每张 GPU 处理不同数据，最后把梯度同步，使所有 GPU 得到相同更新。

例如有 8 张 GPU，每张 GPU 处理 batch size 4。

那么一次 step 的 global batch size 是：

```text
global_batch_size = 8 * 4 = 32
```

每张卡看到不同数据，但模型参数开始时相同。

反向传播后，每张卡得到自己 batch 上的梯度。

通过 all-reduce 求平均梯度后，每张卡用相同梯度更新参数。

这样每张卡的参数仍然保持一致。

### 二、Data Parallel 和 Distributed Data Parallel 的区别

PyTorch 里有两个容易混淆的概念。

第一个是 `DataParallel`，简称 DP。

第二个是 `DistributedDataParallel`，简称 DDP。

`DataParallel` 通常在一个进程里控制多张 GPU。

它会把输入切分到多张卡，计算后把结果聚合回主卡。

问题是：

1. 主卡容易成为瓶颈。
2. Python 线程和 GIL 开销较大。
3. 通信和调度效率较差。
4. 不适合大规模训练。

`DistributedDataParallel` 通常是每张 GPU 一个进程。

每个进程只负责自己的 GPU。

梯度同步通过高效通信后端完成，例如 NCCL。

DDP 的优点是：

1. 扩展性更好。
2. 通信效率更高。
3. 每个进程独立执行，减少主卡瓶颈。
4. 支持多机多卡。

现代分布式训练一般优先使用 DDP，而不是老式 `DataParallel`。

面试中可以直接说：

> DP 是单进程多卡、主卡聚合，扩展性差；DDP 是多进程分布式，每个进程一张卡，通过 all-reduce 同步梯度，是主流数据并行实现。

### 三、DDP 的基本流程

DDP 一步训练可以拆成六个阶段。

第一，初始化。

所有 rank 加载相同模型参数。

rank 是分布式训练中进程的编号。

例如 8 张 GPU 就有 8 个 rank。

第二，数据切分。

每个 rank 从数据集中读取不同 batch。

通常使用 `DistributedSampler` 保证不同 rank 不重复读相同样本。

第三，forward。

每个 rank 用自己的 batch 计算 loss。

第四，backward。

每个 rank 计算自己 batch 上的梯度。

第五，gradient all-reduce。

DDP 自动把各 rank 的梯度做 all-reduce，通常求平均。

第六，optimizer step。

每个 rank 使用同步后的相同梯度更新参数。

因为每个 rank 更新前参数相同，梯度也相同，所以更新后参数仍然相同。

这个过程可以写成：

```text
rank 0: grad_0
rank 1: grad_1
...
rank N-1: grad_N-1

average_grad = (grad_0 + grad_1 + ... + grad_N-1) / N

每个 rank 用 average_grad 更新参数
```

所以 DDP 等价于用更大的 batch 在单个逻辑模型上训练。

更形式化地说，设第 `r` 个 rank 的本地 micro batch 为 `B_r`，world size 为 `R`，模型参数为 `theta`，单样本损失为 `ell(x; theta)`。如果每个 rank 使用平均 loss，那么本地梯度是：

```math
g_r = \frac{1}{|B_r|}\sum_{x\in B_r}\nabla_\theta \ell(x;\theta)
```

DDP all-reduce 后得到的平均梯度是：

```math
g = \frac{1}{R}\sum_{r=0}^{R-1}g_r
```

当每个 rank 的本地 batch 大小相同，也就是 `|B_r| = m` 时：

```math
g = \frac{1}{R\cdot m}\sum_{r=0}^{R-1}\sum_{x\in B_r}\nabla_\theta \ell(x;\theta)
```

这就等价于在所有 rank 拼起来的 global batch 上计算平均梯度。这里有两个边界要注意：第一，如果每个 rank 的样本数不一样，简单平均 rank 梯度不再等价于按样本加权平均；第二，如果 loss 使用的是 sum reduction 而不是 mean reduction，学习率和梯度缩放也要对应调整。

### 四、All-reduce 是什么

All-reduce 是分布式通信操作。

它的作用是：

> 每个节点都有一份 tensor，all-reduce 对这些 tensor 做聚合，并把聚合结果发回每个节点。

例如 4 个 rank 各有一个梯度值：

```text
rank0: 1.0
rank1: 2.0
rank2: 3.0
rank3: 4.0
```

做 sum all-reduce 后，每个 rank 都得到：

```text
10.0
```

如果再除以 4，每个 rank 得到平均梯度：

```text
2.5
```

DDP 梯度同步的核心就是 all-reduce。

在 GPU 训练中，常用 NCCL 实现高效 all-reduce。

通信拓扑可能使用 ring、tree 或其他算法。

你不需要在面试中推导 NCCL 细节，但要知道：

1. all-reduce 是 DDP 的主要通信开销。
2. 模型参数越多，梯度越大，通信量越大。
3. 网络带宽和 GPU 拓扑会影响 DDP 扩展效率。

### 五、为什么 DDP 能加速训练

DDP 加速来自并行处理不同数据。

如果单卡每秒处理 1000 tokens，理想情况下 8 卡可以处理 8000 tokens/s。

但真实加速不会完全线性。

原因包括：

1. gradient all-reduce 通信开销。
2. dataloader 开销。
3. 不同 GPU 负载不均衡。
4. 网络拓扑限制。
5. 小 batch 下计算不够饱和。
6. 同步等待慢 rank。

DDP 是同步训练。

每一步所有 rank 都要完成 backward 并同步梯度后才能 optimizer step。

如果某个 rank 很慢，其他 rank 要等它。

这个慢 rank 叫 straggler。

所以分布式训练中要关注：

1. 每个 rank 的 step time。
2. 数据读取是否均衡。
3. GPU 利用率。
4. 通信时间。
5. all-reduce overlap 是否有效。

### 六、DDP 的显存瓶颈

DDP 的一个关键特点是：

> 每张 GPU 都保存完整模型参数、完整梯度和完整 optimizer state。

这意味着增加 GPU 数量可以提高吞吐，但不能降低单卡模型显存。

例如一个模型需要：

1. 参数 20GB。
2. 梯度 20GB。
3. Adam optimizer state 80GB。
4. activation 30GB。

那么单卡需要的模型相关显存仍然很大。

即使用 8 张卡做 DDP，每张卡仍然要放完整模型和 optimizer state。

这就是为什么大模型训练不能只靠 DDP。

如果模型本身单卡放不下，DDP 没法解决。

这时需要：

1. ZeRO。
2. FSDP。
3. Tensor Parallel。
4. Pipeline Parallel。
5. Activation Checkpointing。

DDP 主要解决数据吞吐，不解决模型切分。

### 七、Global Batch Size 与 Gradient Accumulation

DDP 中有几个 batch 概念必须分清。

micro batch size：

每张 GPU 一次 forward/backward 处理的样本数。

data parallel size：

数据并行进程数，也就是多少张卡并行处理不同数据。

gradient accumulation steps：

累积多少个 micro batch 后再 optimizer step。

global batch size：

一次 optimizer step 实际使用的总样本数。

公式是：

```math
B_{\mathrm{global}} = B_{\mathrm{micro}} \cdot R_{\mathrm{dp}} \cdot A
```

其中 `B_micro` 是单 rank 单次 forward/backward 的 micro batch size，`R_dp` 是 data parallel rank 数，`A` 是 gradient accumulation steps。

例如：

```text
micro_batch_size = 2
data_parallel_size = 8
gradient_accumulation_steps = 4
```

那么：

```text
global_batch_size = 2 * 8 * 4 = 64
```

在大模型训练中，单卡显存有限，micro batch size 可能很小。

为了达到目标 global batch，就需要 gradient accumulation。

Gradient accumulation 的意思是：

1. 连续做多次 forward/backward。
2. 梯度累积在参数上。
3. 累积到指定步数后再同步或更新。

DDP 中为了减少通信，可以在累积的前几步使用 `no_sync()`，最后一步再同步梯度。

否则每个 micro batch 都 all-reduce，会浪费通信。

下面这个 demo 不需要多卡，它只模拟 DDP 中三个关键量：

1. 多个 rank 的本地梯度如何 all-reduce 成平均梯度。
2. global batch size 如何由 micro batch、rank 数和 accumulation steps 决定。
3. 使用 `no_sync()` 时，通信次数如何从每个 micro batch 一次降到每个 optimizer step 一次。

```python
def average_grad(local_grads):
    world_size = len(local_grads)
    return sum(local_grads) / world_size


def global_batch_size(micro_batch_size, data_parallel_size, accumulation_steps):
    return micro_batch_size * data_parallel_size * accumulation_steps


def all_reduce_calls(accumulation_steps, use_no_sync):
    if use_no_sync:
        return 1
    return accumulation_steps


local_grads = [1.0, 2.0, 3.0, 4.0]
print("average gradient:", average_grad(local_grads))

micro_batch_size = 2
data_parallel_size = 4
accumulation_steps = 8
print(
    "global batch:",
    global_batch_size(micro_batch_size, data_parallel_size, accumulation_steps),
)
print("all-reduce calls without no_sync:", all_reduce_calls(accumulation_steps, False))
print("all-reduce calls with no_sync:", all_reduce_calls(accumulation_steps, True))
```

典型输出是：

```text
average gradient: 2.5
global batch: 64
all-reduce calls without no_sync: 8
all-reduce calls with no_sync: 1
```

这个 demo 的重点是机制直觉。真实 PyTorch DDP 的梯度同步发生在 backward 过程中，并且会按 bucket 启动通信；`no_sync()` 的作用是让中间累积步不触发 DDP 的梯度同步。

### 八、DDP 的 bucket 与通信计算重叠

DDP 不是等所有梯度都算完才一次性同步。

为了提高效率，它会把参数梯度分成 bucket。

当某个 bucket 中的梯度都计算完成后，就可以立即启动 all-reduce。

与此同时，反向传播还在继续计算其他层的梯度。

这叫 communication-computation overlap。

直觉是：

> 一边算后面的梯度，一边同步前面已经算好的梯度。

这样可以隐藏一部分通信时间。

bucket size 会影响效率。

bucket 太小：

1. 通信启动次数多。
2. overhead 高。
3. 带宽利用率差。

bucket 太大：

1. 等待 bucket 填满时间长。
2. overlap 机会少。
3. 通信更难隐藏。

实际训练中，框架会有默认 bucket 策略，但大规模训练可能需要调优。

### 九、DistributedSampler 与数据不重复

DDP 中每个 rank 应该看到不同数据。

如果所有 rank 都读同一个 batch，那等于重复计算，没有扩大 batch。

PyTorch 常用 `DistributedSampler`。

它会根据 rank 和 world size 把数据切分。

关键点包括：

1. 每个 epoch 要调用 `sampler.set_epoch(epoch)`。
2. 否则 shuffle 顺序可能每个 epoch 一样。
3. 数据总数不能整除 world size 时，要处理 padding 或 drop last。
4. 多机恢复训练时要保存数据进度。

一个常见 bug 是忘记 `set_epoch`。

结果每个 epoch 数据顺序不变，影响训练随机性。

另一个 bug 是不同 rank 数据数量不一致，导致某些 rank 先结束，训练 hang 住。

### 十、代码示例：最小 DDP 训练结构

下面是一个简化 PyTorch DDP 结构。

```python
import os
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler


def setup_ddp():
    dist.init_process_group(backend="nccl")
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)
    return local_rank


def train(model, dataset, optimizer, epochs):
    local_rank = setup_ddp()
    model = model.to(local_rank)
    model = DDP(model, device_ids=[local_rank])

    sampler = DistributedSampler(dataset)
    dataloader = DataLoader(dataset, sampler=sampler, batch_size=4)

    for epoch in range(epochs):
        sampler.set_epoch(epoch)
        for batch in dataloader:
            batch = {k: v.to(local_rank) for k, v in batch.items()}
            optimizer.zero_grad(set_to_none=True)
            outputs = model(**batch)
            loss = outputs.loss
            loss.backward()
            optimizer.step()

    dist.destroy_process_group()
```

这个例子省略了很多生产细节，例如：

1. gradient accumulation。
2. mixed precision。
3. checkpoint。
4. logging。
5. fault tolerance。
6. validation。

但它展示了 DDP 的基本结构：

1. 初始化 process group。
2. 每个进程绑定一张 GPU。
3. 用 DDP 包装模型。
4. 用 DistributedSampler 切分数据。
5. 正常 forward/backward/step。

梯度同步由 DDP 在 backward 中自动完成。

### 十一、DDP 常见工程坑

坑一：不同 rank 执行路径不一致。

如果某些 rank 参与了某些参数的计算，另一些 rank 没有，可能导致梯度同步 hang 或报错。

坑二：unused parameters。

如果模型有些参数在某次 forward 没用到，DDP 可能需要 `find_unused_parameters=True`。

但这个选项会增加开销。

坑三：batch 数不一致。

某个 rank 数据少，先结束，其他 rank 还在同步，可能 hang。

坑四：随机数没有按 rank 正确设置。

可能导致数据增强、dropout 或采样行为异常。

坑五：logging 重复。

所有 rank 都写日志或保存 checkpoint，会产生重复文件。

通常只让 rank 0 做全局日志和保存入口，但分片 checkpoint 例外。

坑六：梯度累积时重复同步。

没有使用 `no_sync()`，导致每个 micro batch 都 all-reduce，通信成本过高。

坑七：学习率没有随 global batch 调整。

增加 data parallel size 后 global batch 变大，如果 LR 不调整，训练动态会变化。

坑八：验证 loss 聚合错误。

多 rank eval 时要正确 all-reduce loss 和样本数，不能只看 rank 0。

### 十二、DDP 与大模型训练的关系

DDP 是分布式训练的基础，但不是大模型训练的全部。

它适合的场景：

1. 模型单卡能放下。
2. 希望增加数据吞吐。
3. optimizer state 单卡能承受。
4. 通信带宽足够。

它不适合单独解决：

1. 模型参数单卡放不下。
2. Adam state 单卡放不下。
3. activation 单卡放不下。
4. 超长上下文显存爆炸。

所以大模型训练常把 DDP 和其他并行技术组合。

例如：

1. DDP + ZeRO：分片 optimizer/gradient/parameter。
2. DDP + Tensor Parallel：切分矩阵计算。
3. DDP + Pipeline Parallel：切分模型层。
4. DDP + Activation Checkpointing：降低 activation 显存。
5. 3D Parallelism：数据、张量、流水线并行组合。

理解 DDP 是理解这些技术的第一步。

### 十三、面试回答模板

如果面试官问：

> DDP 是怎么工作的？

可以这样回答：

DDP 是同步数据并行。每个 GPU 进程持有一份完整模型参数，处理不同的数据 batch。forward 和 backward 在每个 rank 上独立执行，得到本地梯度。backward 过程中 DDP 会把梯度按 bucket 做 all-reduce，通常求平均，使每个 rank 得到相同的全局平均梯度。然后每个 rank 用相同 optimizer 和相同梯度更新参数，所以各 rank 的模型保持一致。

DDP 的优势是实现简单、扩展数据吞吐，并且通过 bucket 可以让梯度通信和反向计算重叠。但它的局限是每张卡仍然保存完整参数、梯度和 optimizer state，所以不能降低单卡模型显存。如果模型或 Adam state 单卡放不下，就需要 ZeRO、FSDP、Tensor Parallel 或 Pipeline Parallel。

实际使用中还要注意 global batch size 等于 micro batch size 乘以 data parallel size 再乘以 gradient accumulation steps；要用 DistributedSampler 保证不同 rank 读不同数据；梯度累积时要避免每个 micro batch 都同步；多 rank eval 和 checkpoint 也要正确聚合和保存。

### 十四、常见追问与回答

问题 1：DDP 为什么等价于大 batch 训练？

回答：每个 rank 计算自己子 batch 的梯度，然后 all-reduce 求平均。这个平均梯度等价于在所有 rank 的数据组成的大 batch 上计算平均梯度。

问题 2：DDP 和 DataParallel 有什么区别？

回答：DataParallel 通常是单进程多卡，主卡负责聚合，扩展性差。DDP 是多进程分布式，每个进程负责一张卡，通过高效 all-reduce 同步梯度，适合多机多卡。

问题 3：DDP 能解决模型太大单卡放不下吗？

回答：不能。DDP 每张卡都有完整模型、梯度和 optimizer state。它提高吞吐，但不切分模型。模型太大需要 ZeRO/FSDP、张量并行或流水线并行。

问题 4：gradient accumulation 和 DDP 的关系是什么？

回答：gradient accumulation 用多个 micro batch 累积梯度后再 optimizer step。global batch 等于 micro batch 乘以 data parallel size 乘以 accumulation steps。DDP 中可以用 `no_sync()` 避免累积中间步重复 all-reduce。

问题 5：DDP 的主要性能瓶颈是什么？

回答：主要是梯度 all-reduce 通信、慢 rank 等待、dataloader 不均衡和小 batch 计算不饱和。大模型还会受到梯度通信量和网络拓扑影响。

问题 6：为什么 DDP 要用 DistributedSampler？

回答：为了让不同 rank 读取不同数据。如果所有 rank 读同一批样本，就只是重复计算，不能扩大有效 batch。

### 十五、常见误区

1. 误区：DDP 会把模型切到多张卡上。
   纠正：DDP 是复制模型，每张卡都有完整模型，不是模型切分。

2. 误区：多卡 DDP 后单卡显存会明显下降。
   纠正：参数、梯度和 optimizer state 仍然完整保存在每张卡，单卡显存不一定下降。

3. 误区：DDP 只需要同步 loss。
   纠正：DDP 同步的是梯度，不是 loss。loss 可以用于日志，但参数更新依赖同步后的梯度。

4. 误区：rank 0 的验证结果就是全局结果。
   纠正：多 rank eval 要聚合所有 rank 的 loss、样本数和指标。

5. 误区：gradient accumulation 时每步同步无所谓。
   纠正：每个 micro batch 都同步会增加通信开销，应在中间步使用 `no_sync()`。

6. 误区：DDP 卡数越多加速越线性。
   纠正：通信、straggler、dataloader 和网络拓扑都会降低扩展效率。

### 十六、小练习

1. 用自己的话解释 DDP 为什么同步梯度而不是同步参数。
2. 给定 `micro_batch_size=2`、`data_parallel_size=16`、`gradient_accumulation_steps=8`，计算 global batch size。
3. 解释为什么 DDP 不能解决模型单卡放不下的问题。
4. 画出 DDP 一步训练中的 forward、backward、all-reduce、optimizer step 流程。
5. 列出 5 个 DDP 训练中可能导致 hang 的原因。
6. 用 2 分钟回答：“DDP 和 ZeRO/FSDP 的关系是什么？”

### 本讲总结

本讲最重要的结论：

1. DDP 是同步数据并行，每个 rank 持有完整模型并处理不同数据。
2. DDP 在 backward 中通过 all-reduce 同步梯度，使每个 rank 使用相同平均梯度更新参数。
3. DDP 可以提高训练吞吐，并通过 bucket 实现通信和计算重叠。
4. global batch size 由 micro batch、data parallel size 和 gradient accumulation steps 共同决定。
5. DDP 不切分模型，每张卡仍保存完整参数、梯度和 optimizer state，因此无法单独解决大模型显存问题。
6. 大模型训练通常把 DDP 与 ZeRO/FSDP、Tensor Parallel、Pipeline Parallel 和 Activation Checkpointing 组合使用。
7. 面试中讲 DDP 要同时覆盖工作流程、通信机制、batch 概念、性能瓶颈和局限性。

## 第 42 讲：ZeRO 与 FSDP

### 本讲目标

学完这一讲，你应该能够回答：

1. 为什么 DDP 不能解决大模型训练的显存瓶颈。
2. ZeRO-1、ZeRO-2、ZeRO-3 分别分片了什么。
3. FSDP 的核心思想是什么，和 ZeRO-3 有什么关系。
4. 参数 all-gather、gradient reduce-scatter、optimizer state sharding 的流程是什么。
5. ZeRO/FSDP 的收益和代价分别是什么。
6. 面试中如何回答“模型单卡放不下，怎么训练”。

上一讲我们讲了 DDP。

DDP 的问题是：每张 GPU 都保存完整模型参数、完整梯度和完整 optimizer state。

这对中小模型可以接受，但对大模型很快会成为瓶颈。

ZeRO 和 FSDP 的核心目标是：

> 不再让每张 GPU 都保存完整训练状态，而是把参数、梯度和优化器状态分片到不同 GPU 上。

它们是现代大模型训练的核心技术。

### 一、先回顾 DDP 的显存问题

以 AdamW 训练为例，一个参数在训练中不只占一份显存。

通常需要保存：

1. 参数本身。
2. 梯度。
3. Adam 一阶矩 `m`。
4. Adam 二阶矩 `v`。
5. 有时还有 FP32 master weights。

如果使用混合精度训练，参数可能是 BF16/FP16，但 optimizer state 往往是 FP32。

粗略估算，一个参数可能需要十几个 bytes 的训练显存。

以常见 mixed precision AdamW 为例，可以用一个简化口径估算每个参数的训练状态：

```math
M_{\mathrm{param}} = N \cdot b_p
```

```math
M_{\mathrm{grad}} = N \cdot b_g
```

```math
M_{\mathrm{optim}} = N \cdot (b_m + b_v + b_{\mathrm{master}})
```

其中 `N` 是参数量，`b_p` 是训练参数副本的字节数，`b_g` 是梯度字节数，`b_m` 和 `b_v` 是 Adam 一阶矩和二阶矩的字节数，`b_master` 是可选 FP32 master weight 的字节数。若参数和梯度用 BF16/FP16，各 2 bytes，Adam `m`、`v` 和 master weight 用 FP32，各 4 bytes，那么每个参数大约是：

```math
2 + 2 + 4 + 4 + 4 = 16\ \mathrm{bytes}
```

这个 16 bytes 只是训练状态的教学估算，不包含 activation、临时通信 buffer、CUDA workspace、padding 和显存碎片。

对于 7B、13B、70B 模型，这会非常夸张。

DDP 中每张卡都有完整一份这些状态。

所以 DDP 增加 GPU 数量，只提高数据吞吐，不减少单卡模型状态显存。

ZeRO 和 FSDP 就是为了解决这个问题。

### 二、ZeRO 的核心思想

ZeRO 是 Zero Redundancy Optimizer 的缩写。

名字里的 zero redundancy 指减少冗余。

在 DDP 中，训练状态在每张卡上重复保存。

ZeRO 的想法是：

> 既然每张卡最终只需要参与一部分计算，就不要让每张卡都长期保存所有训练状态。

ZeRO 把训练状态分成三类：

1. optimizer states。
2. gradients。
3. parameters。

然后逐步分片。

对应三个阶段：

1. ZeRO-1：分片 optimizer states。
2. ZeRO-2：分片 optimizer states 和 gradients。
3. ZeRO-3：分片 optimizer states、gradients 和 parameters。

分片的意思是：

如果有 8 张 GPU，一个大 tensor 不再每张卡保存完整副本，而是每张卡只保存 1/8。

这样单卡显存显著下降。

### 三、ZeRO-1：只分片 optimizer state

ZeRO-1 分片 optimizer state。

也就是说：

1. 参数仍然每张卡完整保存。
2. 梯度仍然每张卡完整保存。
3. Adam 的 `m`、`v` 等 optimizer state 被切分到不同卡。

为什么先分 optimizer state？

因为 AdamW 的 optimizer state 很大。

通常 `m` 和 `v` 都是 FP32，每个参数对应两个 FP32 状态。

如果还有 master weights，显存更大。

ZeRO-1 的优点：

1. 实现相对简单。
2. 显存收益明显。
3. 通信开销相对较小。

缺点：

1. 参数和梯度仍然完整复制。
2. 模型本身太大时仍然放不下。

ZeRO-1 适合 optimizer state 是主要瓶颈，但模型参数本身还能放下的场景。

### 四、ZeRO-2：再分片 gradients

ZeRO-2 在 ZeRO-1 基础上进一步分片 gradients。

状态变成：

1. 参数每张卡完整保存。
2. gradients 被分片。
3. optimizer states 被分片。

DDP 中梯度同步通常是 all-reduce。

ZeRO-2 常用 reduce-scatter。

Reduce-scatter 可以理解为：

1. 先把不同 rank 的梯度求和或求平均。
2. 然后每个 rank 只保留结果的一片。

这样每张卡不需要保存完整梯度。

ZeRO-2 的优点：

1. 比 ZeRO-1 进一步降低显存。
2. 参数仍完整保存，forward/backward 相对简单。
3. 适合很多中大型模型训练。

缺点：

1. 参数仍然完整复制。
2. 模型参数本身太大时仍然受限。
3. 通信和实现复杂度比 ZeRO-1 更高。

### 五、ZeRO-3：参数也分片

ZeRO-3 是最激进的一层。

它分片：

1. optimizer states。
2. gradients。
3. parameters。

也就是说，每张卡只长期保存模型参数的一部分。

这能大幅降低单卡显存，使训练更大模型成为可能。

但问题来了：

forward 时某一层计算需要完整参数，参数被切开了怎么办？

答案是按需 all-gather。

在某一层 forward 前，各 rank 通信，把这一层需要的完整参数临时聚合出来。

计算完成后，可以释放不再需要的完整参数，只保留自己的分片。

backward 时也类似。

需要参数参与梯度计算时临时 gather，梯度计算后再 reduce-scatter，把梯度分片保存。

所以 ZeRO-3 的核心流程是：

```text
参数长期分片保存
某层计算前 all-gather 参数
完成计算后释放完整参数
backward 后 reduce-scatter 梯度
每个 rank 用自己的梯度分片更新自己的参数分片
```

ZeRO-3 的优点：

1. 显存节省最大。
2. 可以训练单卡放不下的模型。
3. optimizer、gradient、parameter 都去冗余。

缺点：

1. 通信开销更高。
2. 实现复杂。
3. 对网络带宽和 overlap 要求高。
4. checkpoint 和恢复更复杂。
5. 小模型上可能不划算。

### 六、FSDP 的核心思想

FSDP 是 Fully Sharded Data Parallel。

它是 PyTorch 生态中常用的全分片数据并行方案。

从思想上看，FSDP 和 ZeRO-3 非常接近：

> 参数、梯度和 optimizer state 都可以分片保存，需要计算时临时 all-gather 参数，计算后释放完整参数。

FSDP 通常按模块包裹模型。

例如可以把每个 Transformer block 包成一个 FSDP unit。

每个 unit 的参数平时是 sharded 的。

当执行到这个 unit 时：

1. all-gather 该 unit 的完整参数。
2. 做 forward 或 backward。
3. 释放完整参数，回到 sharded 状态。

FSDP 的关键配置包括：

1. auto wrap policy。
2. sharding strategy。
3. mixed precision policy。
4. CPU offload。
5. activation checkpointing。
6. state dict type。

这些配置会显著影响显存、速度和 checkpoint 复杂度。

### 七、ZeRO-3 和 FSDP 的关系

面试中经常会问：

> ZeRO-3 和 FSDP 有什么区别？

可以这样理解。

ZeRO 是一套思想和分阶段优化方案，最早来自 DeepSpeed。

FSDP 是 PyTorch 中实现 fully sharded data parallel 的具体机制。

从训练状态分片角度，FSDP 很接近 ZeRO-3。

两者都希望分片参数、梯度和 optimizer state。

差异更多在：

1. 框架生态。
2. API 和配置方式。
3. checkpoint 格式。
4. wrap 粒度。
5. 通信调度。
6. offload 支持。
7. 与其他并行方式集成。

工程上选择哪一个，取决于团队栈。

如果使用 DeepSpeed 生态，ZeRO 很常见。

如果使用原生 PyTorch 生态，FSDP 很常见。

但面试中更重要的是讲清分片逻辑，而不是背框架名字。

### 八、通信操作：all-gather 与 reduce-scatter

理解 ZeRO/FSDP 要理解两个通信操作。

第一个是 all-gather。

每个 rank 有一个 tensor 分片，all-gather 后每个 rank 都得到完整 tensor。

例如：

```text
rank0: A
rank1: B
rank2: C
rank3: D

all-gather 后每个 rank 都有 ABCD
```

ZeRO-3/FSDP 用 all-gather 临时恢复某层完整参数。

第二个是 reduce-scatter。

每个 rank 有完整或部分待聚合 tensor，reduce-scatter 做聚合后，每个 rank 只得到结果的一片。

例如梯度同步时：

```text
先对所有 rank 梯度求和
再把求和结果切成分片
每个 rank 只保留一片
```

ZeRO-2/3 和 FSDP 用 reduce-scatter 保存梯度分片。

这两个操作替代了 DDP 中简单的 all-reduce。

DDP 是每张卡都有完整梯度。

ZeRO/FSDP 是同步后只保留自己负责的分片。

### 九、显存收益怎么理解

假设有 `N` 张 GPU。

DDP 中每张卡都保存完整状态。

ZeRO/FSDP 分片后，部分状态可以约减少到 `1/N`。

粗略理解：

```math
M_{\mathrm{DDP}} = M_{\mathrm{param}} + M_{\mathrm{grad}} + M_{\mathrm{optim}}
```

```math
M_{\mathrm{ZeRO1}} = M_{\mathrm{param}} + M_{\mathrm{grad}} + \frac{M_{\mathrm{optim}}}{R}
```

```math
M_{\mathrm{ZeRO2}} = M_{\mathrm{param}} + \frac{M_{\mathrm{grad}}}{R} + \frac{M_{\mathrm{optim}}}{R}
```

```math
M_{\mathrm{ZeRO3}} = \frac{M_{\mathrm{param}}}{R} + \frac{M_{\mathrm{grad}}}{R} + \frac{M_{\mathrm{optim}}}{R}
```

其中 `R` 是 data parallel world size。FSDP 的长期训练状态显存可以近似按 ZeRO-3 理解，但实际峰值还会受到当前 FSDP unit all-gather 完整参数、prefetch、通信 buffer 和 wrap 粒度影响。

这只是训练状态显存的粗略估算。

实际显存还包括：

1. activations。
2. temporary all-gather buffers。
3. communication buffers。
4. fragmentation。
5. CUDA kernel workspace。

所以即使 ZeRO-3 参数显存大幅下降，也不能忽略 activation。

如果序列很长，activation 可能成为主要瓶颈。

这时还需要 activation checkpointing、sequence parallel 或 context parallel。

下面这个 demo 用参数量、world size 和每类状态字节数估算 DDP 与 ZeRO/FSDP 的单卡训练状态显存。它的目的不是精确容量规划，而是帮你快速解释为什么 ZeRO-1/2/3 的节省对象不同。

```python
def gib(bytes_value):
    return bytes_value / (1024 ** 3)


def memory_by_stage(num_params, world_size, bytes_param=2, bytes_grad=2, bytes_optim=12):
    param = num_params * bytes_param
    grad = num_params * bytes_grad
    optim = num_params * bytes_optim
    return {
        "DDP": param + grad + optim,
        "ZeRO-1": param + grad + optim / world_size,
        "ZeRO-2": param + grad / world_size + optim / world_size,
        "ZeRO-3/FSDP": param / world_size + grad / world_size + optim / world_size,
    }


num_params = 7_000_000_000
world_size = 8

for name, bytes_value in memory_by_stage(num_params, world_size).items():
    print(name, round(gib(bytes_value), 2), "GiB per GPU")
```

典型输出类似：

```text
DDP 104.31 GiB per GPU
ZeRO-1 35.86 GiB per GPU
ZeRO-2 24.45 GiB per GPU
ZeRO-3/FSDP 13.04 GiB per GPU
```

这个例子说明：在 7B 模型、8 卡、mixed precision AdamW 的粗略口径下，DDP 单卡仅训练状态就可能超过常见 GPU 显存；ZeRO-3/FSDP 可以把长期状态显存近似降到 `1/R`。但真实训练还要留出 activation、临时完整参数、通信 buffer 和 fragmentation，所以不能按这个数字把显存用满。

### 十、性能代价：显存不是免费的

ZeRO/FSDP 节省显存，但不是免费。

主要代价是通信增加。

DDP 主要在 backward 同步梯度。

ZeRO-3/FSDP 在 forward/backward 中需要频繁 all-gather 参数。

如果网络带宽不足，训练速度可能明显下降。

常见性能问题包括：

1. all-gather 太频繁。
2. wrap 粒度太小，通信启动开销大。
3. wrap 粒度太大，临时完整参数占显存高。
4. 通信和计算 overlap 不充分。
5. CPU offload 导致 PCIe 传输瓶颈。
6. checkpoint 保存和加载慢。

所以使用 FSDP/ZeRO 需要调优。

例如 FSDP wrap 粒度通常不能太细，也不能太粗。

对 Transformer 来说，一个常见选择是按 Transformer block wrap。

### 十一、CPU/NVMe Offload

ZeRO 和 FSDP 有时支持 offload。

Offload 指把部分状态从 GPU 显存转移到 CPU 内存或 NVMe 磁盘。

常见 offload 对象：

1. optimizer state。
2. parameters。
3. gradients。

好处是进一步节省 GPU 显存。

坏处是速度变慢。

因为 CPU/GPU 或 NVMe/GPU 之间传输带宽远低于 GPU 显存带宽。

Offload 适合：

1. 显存极度不足。
2. 训练速度不是最优先。
3. 微调或小规模训练。
4. 用较少 GPU 训练较大模型。

大规模预训练中，如果追求高吞吐，过度 offload 往往不理想。

### 十二、FSDP 配置中的 wrap 粒度

FSDP 需要决定哪些模块被单独 wrap。

wrap 粒度影响显存和通信。

如果 wrap 很细，比如每个 Linear 都 wrap：

1. 每次 all-gather 很小。
2. 通信启动次数很多。
3. overhead 大。

如果 wrap 很粗，比如整个模型一个 FSDP unit：

1. all-gather 一次参数很多。
2. 临时显存高。
3. 可能失去分片收益。

Transformer 模型常见策略是按 block wrap：

```text
Embedding
Block 1
Block 2
...
Block N
LM Head
```

每个 block 作为一个 FSDP unit。

这样可以在通信次数和临时显存之间折中。

### 十三、Checkpoint 复杂度

ZeRO/FSDP 的 checkpoint 比 DDP 更复杂。

因为参数和 optimizer state 是分片保存的。

常见 checkpoint 类型包括：

1. full state dict。
2. sharded state dict。
3. local state dict。

full state dict：

聚合成完整模型，适合推理和发布，但保存时内存压力大。

sharded state dict：

按分片保存，适合恢复训练和大模型，但需要元数据。

local state dict：

每个 rank 保存本地状态，和当前并行配置绑定更强。

工程上要考虑：

1. 是否支持不同 GPU 数恢复。
2. 是否支持转换成 Hugging Face 格式。
3. optimizer state 是否保存。
4. checkpoint 是否完整。
5. rank mapping 是否变化。

这也是为什么上一讲 checkpoint 策略非常重要。

### 十四、ZeRO/FSDP 与其他并行方式

ZeRO/FSDP 主要解决数据并行下的训练状态冗余。

它们不等同于 Tensor Parallel 或 Pipeline Parallel。

Tensor Parallel 切分的是单层矩阵计算。

Pipeline Parallel 切分的是模型层。

ZeRO/FSDP 切分的是训练状态。

实际大模型训练可能组合使用：

```text
FSDP/ZeRO + Tensor Parallel + Pipeline Parallel + Activation Checkpointing
```

例如：

1. Tensor Parallel 解决单层矩阵太大或单层计算分摊。
2. Pipeline Parallel 解决层数多、模型太深。
3. ZeRO/FSDP 解决参数、梯度、optimizer state 冗余。
4. Activation Checkpointing 解决 activation 显存。

不要把这些技术混为一谈。

### 十五、代码示例：FSDP 基本结构

下面是一个简化 FSDP 使用结构。

```python
import torch
import torch.distributed as dist
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy


def setup():
    dist.init_process_group("nccl")
    local_rank = torch.distributed.get_rank() % torch.cuda.device_count()
    torch.cuda.set_device(local_rank)
    return local_rank


def build_fsdp_model(model, transformer_layer_cls):
    auto_wrap_policy = transformer_auto_wrap_policy(
        transformer_layer_cls={transformer_layer_cls},
    )
    return FSDP(
        model,
        auto_wrap_policy=auto_wrap_policy,
        device_id=torch.cuda.current_device(),
    )
```

这个示例只展示结构。

真实项目还要配置：

1. mixed precision。
2. activation checkpointing。
3. sharding strategy。
4. state dict 保存方式。
5. CPU offload。
6. gradient accumulation。
7. 与 dataloader 和 checkpoint 系统集成。

FSDP 的难点不是把模型包起来，而是把显存、吞吐、通信和恢复训练调到稳定可用。

### 十六、什么时候选 DDP、ZeRO-2、ZeRO-3/FSDP

可以用一个简单决策框架。

如果模型和 optimizer state 单卡能放下，只是想提高吞吐：

选择 DDP。

如果 optimizer state 和 gradient 占显存很大，但参数还能完整放下：

选择 ZeRO-1 或 ZeRO-2。

如果参数本身也很大，单卡放完整模型压力很大：

选择 ZeRO-3 或 FSDP。

如果单层矩阵本身计算太大，或者隐藏维度很大：

需要 Tensor Parallel。

如果模型层数很多，需要跨设备切层：

需要 Pipeline Parallel。

如果 sequence 很长，activation 爆炸：

需要 Activation Checkpointing、Sequence Parallel 或 Context Parallel。

真实系统通常不是单选，而是组合。

### 十七、面试回答模板

如果面试官问：

> ZeRO 和 FSDP 是什么？它们解决什么问题？

可以这样回答：

DDP 中每张 GPU 都保存完整模型参数、梯度和 optimizer state，所以虽然能提升数据吞吐，但不能降低单卡模型状态显存。ZeRO 和 FSDP 的核心思想是把训练状态分片到多个 GPU 上，减少冗余。

ZeRO 分三个阶段。ZeRO-1 分片 optimizer state，ZeRO-2 进一步分片 gradients，ZeRO-3 连 parameters 也分片。ZeRO-3 中参数平时只保存分片，计算某层时通过 all-gather 临时恢复完整参数，反向后通过 reduce-scatter 保存梯度分片，每个 rank 只更新自己负责的参数分片。

FSDP 是 PyTorch 中的 fully sharded data parallel，实现思想和 ZeRO-3 很接近，也是分片参数、梯度和 optimizer state。它通常按 Transformer block 等模块 wrap，进入模块时 all-gather 参数，计算后释放完整参数。FSDP/ZeRO 能显著降低显存，但代价是更多通信、更复杂 checkpoint 和更多调优。它们主要解决训练状态冗余，不等同于张量并行或流水线并行，真实大模型训练中经常和 TP、PP、activation checkpointing 组合使用。

### 十八、常见追问与回答

问题 1：ZeRO-1、ZeRO-2、ZeRO-3 分别分片什么？

回答：ZeRO-1 分片 optimizer state；ZeRO-2 分片 optimizer state 和 gradients；ZeRO-3 分片 optimizer state、gradients 和 parameters。

问题 2：ZeRO-3 为什么需要 all-gather？

回答：因为参数平时是分片保存的，但某一层计算需要完整参数，所以在 forward/backward 前要临时 all-gather 出该层完整参数。

问题 3：FSDP 和 DDP 的核心区别是什么？

回答：DDP 每张卡保存完整模型状态，只同步梯度；FSDP 把参数、梯度和 optimizer state 分片保存，需要计算时临时 all-gather 参数。

问题 4：ZeRO/FSDP 是否一定比 DDP 更快？

回答：不一定。它们主要节省显存，但增加通信和调度开销。模型不大时 DDP 可能更快；模型大到 DDP 放不下或显存压力很大时，ZeRO/FSDP 才更有价值。

问题 5：FSDP wrap 粒度为什么重要？

回答：wrap 太细通信启动次数多，overhead 大；wrap 太粗临时 all-gather 参数太多，显存高。通常按 Transformer block wrap 是一种折中。

问题 6：ZeRO/FSDP 能替代 Tensor Parallel 吗？

回答：不能完全替代。ZeRO/FSDP 分片训练状态，Tensor Parallel 切分单层矩阵计算。如果单层本身太大或计算需要分摊，仍需要 TP。

### 十九、常见误区

1. 误区：ZeRO/FSDP 只是更快的 DDP。
   纠正：它们主要目标是省显存，速度可能更快也可能更慢，取决于通信和模型规模。

2. 误区：ZeRO-3 没有完整参数，所以不能正常 forward。
   纠正：ZeRO-3 会在需要时 all-gather 当前层完整参数，计算后再释放。

3. 误区：FSDP 和 Tensor Parallel 是一回事。
   纠正：FSDP 分片训练状态，TP 切分矩阵计算。

4. 误区：用了 FSDP 就不用 activation checkpointing。
   纠正：FSDP 主要省参数和 optimizer state，长序列下 activation 仍可能是瓶颈。

5. 误区：分片越细越好。
   纠正：过细会增加通信启动开销，降低吞吐。

6. 误区：checkpoint 仍然像 DDP 一样简单保存模型即可。
   纠正：分片 checkpoint 需要处理 state dict 类型、metadata、rank mapping 和格式转换。

2. 列出 ZeRO-1、ZeRO-2、ZeRO-3 分别分片的状态。
3. 画出 ZeRO-3 中参数 all-gather 和梯度 reduce-scatter 的流程。
4. 解释 FSDP wrap 粒度如何影响显存和通信。

### 本讲总结

本讲最重要的结论：

1. DDP 的主要问题是每张卡保存完整参数、梯度和 optimizer state，无法解决大模型单卡显存瓶颈。
2. ZeRO 通过逐步分片 optimizer state、gradients 和 parameters 来减少训练状态冗余。
3. ZeRO-1 分片 optimizer state，ZeRO-2 再分片 gradients，ZeRO-3 再分片 parameters。
4. FSDP 与 ZeRO-3 思想接近，通常按模块分片参数，需要计算时 all-gather，计算后释放。
5. ZeRO/FSDP 显著节省显存，但增加通信、checkpoint 和配置复杂度。
6. ZeRO/FSDP 解决训练状态分片问题，不等同于 Tensor Parallel、Pipeline Parallel 或 Activation Checkpointing。
7. 面试中要能从显存构成、分片对象、通信流程、收益代价和适用场景系统解释 ZeRO/FSDP。

## 第 43 讲：Tensor Parallel

### 本讲目标

学完这一讲，你应该能够回答：

1. Tensor Parallel 解决什么问题，和 DDP、ZeRO/FSDP 有什么不同。
2. 为什么大模型中的 Linear、Attention 和 MLP 可以按矩阵维度切分。
3. Column Parallel 和 Row Parallel 分别是什么。
4. Tensor Parallel 中常见通信操作 all-reduce、all-gather、reduce-scatter 用在哪里。
5. 为什么 Tensor Parallel 通常放在高速互联 GPU 内部使用。
6. 面试中如何回答“一个 Transformer block 怎么做张量并行”。

Tensor Parallel，简称 TP，中文常叫张量并行或模型张量并行。

它和前两讲的 DDP、ZeRO/FSDP 关注点不同。

DDP 是复制模型、切分数据。

ZeRO/FSDP 是切分训练状态。

Tensor Parallel 是切分单层计算。

也就是说：

> 当一个 Linear、Attention 或 MLP 太大，单张 GPU 不适合完整计算时，把这个张量运算拆到多张 GPU 上共同完成。

Tensor Parallel 是训练超大 Transformer 的关键技术之一。

### 一、为什么需要 Tensor Parallel

先看一个 Transformer 中最常见的 Linear 层。

输入 `X` 的形状是：

```text
[batch, seq_len, hidden_size]
```

权重 `W` 的形状是：

```text
[hidden_size, output_size]
```

输出是：

```text
Y = XW
```

当 hidden size 很大时，`W` 会非常大，矩阵乘计算也非常重。

例如 hidden size 是 12288，FFN 中间维度可能是 49152。

一个 FFN up projection 权重就有：

```text
12288 * 49152 ≈ 604M parameters
```

单层就可能有数亿参数。

如果让一张 GPU 独自保存和计算，会遇到：

1. 参数显存压力大。
2. 激活显存压力大。
3. 单卡计算时间长。
4. 单层矩阵乘太大，无法充分并行扩展。

Tensor Parallel 的思路是把这个矩阵乘拆开。

多张 GPU 各算一部分，最后通过通信合并结果。

### 二、Tensor Parallel 和 ZeRO/FSDP 的区别

这是面试高频点。

ZeRO/FSDP 分片参数，但计算某层时通常会 all-gather 当前层完整参数，再在每个 rank 上进行完整计算，或者按 FSDP unit 执行。

Tensor Parallel 不只是保存时分片，而是计算本身就分片。

区别可以这样看：

```text
DDP：每卡完整模型，切数据
ZeRO/FSDP：训练状态分片，需要时恢复参数
Tensor Parallel：单层矩阵计算按维度切开，多卡共同算一层
Pipeline Parallel：按层切模型，不同卡负责不同层
```

如果模型单层很宽，Tensor Parallel 很重要。

如果模型层数很多，Pipeline Parallel 很重要。

如果 optimizer state 太大，ZeRO/FSDP 很重要。

真实大模型训练常常三者组合。

### 三、矩阵乘如何切分

核心问题是：如何切 `Y = XW`。

有两种基本方式。

第一种，按 `W` 的列切分。

这叫 Column Parallel。

假设：

```text
W = [W1, W2]
```

其中 `W1` 和 `W2` 是按输出维度切开的两部分。

那么：

```text
Y = XW = [XW1, XW2]
```

每张 GPU 计算一部分输出，最后输出在 hidden/output 维度上拼接。

第二种，按 `W` 的行切分。

这叫 Row Parallel。

假设输入 `X` 也按 hidden 维度切开：

```text
X = [X1, X2]
W = [W1; W2]
```

那么：

```text
Y = XW = X1W1 + X2W2
```

每张 GPU 计算一个 partial output，最后需要求和。

这个求和通常通过 all-reduce 完成。

所以直觉是：

```text
Column Parallel：切输出维度，结果拼接
Row Parallel：切输入维度，结果求和
```

### 四、Column Parallel Linear

Column Parallel Linear 把权重按输出维度切。

假设有 2 张 GPU。

原始权重：

```text
W: [hidden, 4 * hidden]
```

切分后：

```text
GPU0: W0 [hidden, 2 * hidden]
GPU1: W1 [hidden, 2 * hidden]
```

每张 GPU 都拿到完整输入 `X`。

然后各自计算：

```text
Y0 = XW0
Y1 = XW1
```

完整输出是：

```text
Y = concat(Y0, Y1)
```

如果下一层也能接受分片输出，就不一定马上 all-gather。

这很重要。

高效 TP 会设计相邻层的切分方式，尽量减少通信。

### 五、Row Parallel Linear

Row Parallel Linear 把权重按输入维度切。

假设原始权重：

```text
W: [4 * hidden, hidden]
```

输入 `X` 已经被切成两份：

```text
GPU0: X0 [..., 2 * hidden]
GPU1: X1 [..., 2 * hidden]
```

对应权重：

```text
GPU0: W0 [2 * hidden, hidden]
GPU1: W1 [2 * hidden, hidden]
```

每张 GPU 计算 partial output：

```text
Y0 = X0W0
Y1 = X1W1
```

完整输出是：

```text
Y = Y0 + Y1
```

这个求和需要 all-reduce。

所以 Row Parallel 常在输出处引入一次通信。

### 六、MLP 中如何组合 Column 和 Row

Transformer MLP 通常是两层线性变换：

```text
X -> up projection -> activation -> down projection -> output
```

例如：

```text
hidden -> 4 * hidden -> hidden
```

常见 TP 设计是：

1. up projection 用 Column Parallel。
2. activation 在每张 GPU 的分片上本地计算。
3. down projection 用 Row Parallel。
4. 最后 all-reduce 得到完整 hidden 输出。

为什么这样设计？

因为 up projection 的输出维度很大，按列切后每张卡得到一部分中间激活。

activation 是逐元素操作，可以本地完成，不需要通信。

down projection 正好接收分片输入，按行切权重，每张卡计算 partial output，最后求和。

通信只需要在 down projection 后做一次 all-reduce。

这比每一层都 gather 完整激活更高效。

### 七、Attention 中如何做 Tensor Parallel

Self-Attention 中有几个关键线性层：

1. Q projection。
2. K projection。
3. V projection。
4. output projection。

对于多头注意力，head 本身就是天然可切分的。

假设有 32 个 attention heads，tensor parallel size 是 4。

可以每张 GPU 负责 8 个 heads。

Q/K/V projection 通常可以按 head 或输出维度切分。

每张 GPU 只计算自己负责的 heads。

attention softmax 也可以在本地 heads 上完成。

然后 output projection 再把多卡结果合并。

直觉是：

> Attention 的 head 维度天然适合做 Tensor Parallel。

但有一些细节。

如果使用 MQA 或 GQA，Q heads 和 KV heads 数量不同。

这会影响切分方式。

例如 GQA 中多个 Q head 共享较少 K/V head。

TP 切分时要保证 K/V heads 在各 rank 上分布合理，否则会出现通信或负载不均衡。

### 八、通信在哪里发生

Tensor Parallel 的通信主要来自跨 GPU 合并 partial results。

常见通信包括：

1. all-reduce。
2. all-gather。
3. reduce-scatter。

在经典 Megatron-style TP 中，MLP 和 Attention 可以通过 Column/Row Parallel 组合，把通信控制在少数位置。

例如：

1. Column Parallel 输出分片，不立即通信。
2. 中间 activation 本地计算。
3. Row Parallel 输出 partial result。
4. all-reduce 合并输出。

每个 Transformer block 通常会有若干次 TP 通信。

通信量和序列长度、batch、hidden size、TP size 有关。

因为 TP 通信发生在每层内部，频率很高。

所以 Tensor Parallel 对互联带宽要求非常高。

这就是为什么 TP 通常放在单机多 GPU 或 NVLink/NVSwitch 高速互联域内，而不是跨慢速网络随便扩展。

### 九、Tensor Parallel Size 怎么选

Tensor Parallel Size 指一个层被多少张 GPU 切分。

例如 TP=8 表示每层矩阵计算切到 8 张 GPU 上。

TP size 越大：

优点：

1. 单卡参数和计算更少。
2. 可以训练更宽的层。
3. 单层计算被更多 GPU 分担。

缺点：

1. 通信更多。
2. 每张卡计算粒度变小，可能不够饱和。
3. 要求 hidden size、head 数能被 TP size 整除。
4. 跨节点 TP 会很慢。

常见选择会受硬件拓扑影响。

如果一台机器有 8 张 GPU，且 NVLink/NVSwitch 互联很好，TP size 常选 2、4、8。

如果跨节点网络较慢，通常不希望 TP 跨节点。

一个实用原则是：

> TP 尽量放在高速互联内部，跨节点更多使用 DP 或 Pipeline Parallel。

### 十、Tensor Parallel 与显存

TP 可以减少单卡参数显存。

因为每张卡只保存某些权重的一部分。

它也能减少部分中间激活显存。

但 TP 不是万能省显存工具。

原因包括：

1. 有些激活仍需要复制。
2. 通信 buffer 需要显存。
3. embedding、norm、small layers 可能不切或复制。
4. optimizer state 是否切分取决于是否结合 ZeRO/FSDP。

如果只做 Tensor Parallel，不做 ZeRO，optimizer state 可能仍按 TP 分片参数保存，但数据并行维度上仍会复制。

真实大模型训练常同时使用 TP 和 ZeRO/FSDP。

例如：

```text
TP 负责切单层矩阵
ZeRO/FSDP 负责切数据并行维度上的训练状态
PP 负责切模型层
```

### 十一、Tensor Parallel 与 DDP 的组合

假设有 16 张 GPU。

可以设置：

```text
TP size = 4
DP size = 4
```

这表示每 4 张 GPU 组成一个 TP group，共同保存和计算一个模型副本。

一共有 4 个这样的模型副本，形成 DP group。

每个 DP group 处理不同数据。

TP group 内部做张量并行通信。

DP group 之间做梯度同步或 ZeRO/FSDP 分片。

这就是并行维度组合的基础。

再加 Pipeline Parallel，就会形成 3D parallelism。

### 十二、代码示例：理解列切分和行切分

Megatron-LM 论文把这类方法称为 intra-layer model parallelism：不改变 Transformer 数学结构，而是在 Linear、MLP 和 Attention 这些层内部插入少量通信操作，把单层计算拆到多张 GPU 上。下面这个 demo 不需要多卡，只用 PyTorch tensor 切片模拟两张 GPU 上的 Column Parallel 和 Row Parallel，并验证它们和完整矩阵乘结果一致。

```python
import torch


torch.manual_seed(0)

batch = 3
hidden = 4
ffn_hidden = 8
tp_size = 2

x = torch.randn(batch, hidden)
w_up = torch.randn(hidden, ffn_hidden)
w_down = torch.randn(ffn_hidden, hidden)

# Baseline: full MLP on one logical device.
full_hidden = torch.relu(x @ w_up)
full_output = full_hidden @ w_down


def column_parallel_linear(x, weight, tp_size):
    # Split output dimension: W = [W0, W1].
    weight_shards = torch.chunk(weight, tp_size, dim=1)
    output_shards = [x @ shard for shard in weight_shards]
    return output_shards


def row_parallel_linear(input_shards, weight, tp_size):
    # Split input dimension: X = [X0, X1], W = [W0; W1].
    weight_shards = torch.chunk(weight, tp_size, dim=0)
    partial_outputs = [
        x_shard @ w_shard
        for x_shard, w_shard in zip(input_shards, weight_shards)
    ]
    # Simulate all-reduce sum across TP ranks.
    return sum(partial_outputs)


# Megatron-style MLP: column-parallel up projection, local activation,
# row-parallel down projection, then all-reduce sum.
up_shards = column_parallel_linear(x, w_up, tp_size)
activated_shards = [torch.relu(shard) for shard in up_shards]
tp_output = row_parallel_linear(activated_shards, w_down, tp_size)

print("full output shape:", tuple(full_output.shape))
print("tp output shape:", tuple(tp_output.shape))
print("max error:", (full_output - tp_output).abs().max().item())
print("outputs match:", torch.allclose(full_output, tp_output, atol=1e-6))
```

典型输出是：

```text
full output shape: (3, 4)
tp output shape: (3, 4)
max error: 2.384185791015625e-07
outputs match: True
```

这个 demo 对应前面的直觉：up projection 按输出维度切，得到分片中间激活；ReLU 这种逐元素 activation 可以在每个分片本地完成；down projection 按输入维度切，每张卡得到 partial output，最后用 all-reduce sum 合并。真实 Megatron-style TP 还要处理 backward 中的通信、Q/K/V head 切分、layout、fused kernel、混合精度和 checkpoint 分片，但核心等价性就是这两个矩阵恒等式。

真实框架会处理：

1. forward 通信。
2. backward 通信。
3. tensor layout。
4. mixed precision。
5. fused kernels。
6. 与 optimizer 和 checkpoint 集成。

但核心数学就是切矩阵、局部计算、通信合并。

### 十三、Tensor Parallel 的常见工程坑

坑一：hidden size 或 head 数不能整除 TP size。

例如 32 个 heads 可以被 8 切，但不能被 6 均匀切。

坑二：跨节点 TP 导致通信瓶颈。

TP 通信频繁，跨慢速网络会严重拖慢训练。

坑三：通信没有和计算重叠。

导致 GPU 等通信，利用率低。

坑四：切分方式和模型结构不匹配。

例如 GQA/MQA、MoE、特殊 attention 结构需要特殊处理。

坑五：checkpoint 转换复杂。

TP checkpoint 是按张量分片保存的，推理或微调时可能需要 merge 或 reshard。

坑六：数值结果和单卡不完全一致。

不同通信和 reduction 顺序可能带来微小数值差异。

坑七：小模型使用过大 TP size。

每张 GPU 计算太少，通信开销超过收益。

### 十四、Tensor Parallel 与推理

TP 不只用于训练，也常用于推理。

对于大模型推理，如果单卡放不下模型，或单卡算不动大矩阵，可以用 TP 把模型切到多卡。

推理中的 TP 关注点有些不同。

训练时需要 forward、backward 和梯度通信。

推理时只有 forward，但有 KV Cache。

TP 推理要考虑：

1. 权重分片。
2. attention head 分布。
3. KV Cache 分布。
4. batch 调度。
5. token latency。
6. all-reduce 通信延迟。

对于低延迟服务，TP size 过大可能增加通信延迟。

对于大 batch 高吞吐服务，TP 可以提高单模型吞吐。

所以推理中 TP size 也要结合延迟和吞吐权衡。

### 十五、面试回答模板

如果面试官问：

> Tensor Parallel 是什么？Transformer 中怎么做？

可以这样回答：

Tensor Parallel 是把单层张量计算按维度切到多张 GPU 上，而不是像 DDP 那样复制完整模型。它主要解决单层矩阵太大、hidden size 很宽或单卡计算显存压力过大的问题。和 ZeRO/FSDP 不同，ZeRO/FSDP 主要分片训练状态，Tensor Parallel 是计算本身分片。

在 Linear 层中有两种基本切法。Column Parallel 按输出维度切权重，每张卡拿完整输入，计算一部分输出，输出可以保持分片。Row Parallel 按输入维度切权重和输入，每张卡计算 partial output，最后通过 all-reduce 求和。Transformer MLP 常用 up projection 做 Column Parallel，activation 本地计算，down projection 做 Row Parallel，最后 all-reduce。Attention 中可以按 head 切分 Q/K/V，每张卡负责一部分 heads，再在 output projection 处合并。

Tensor Parallel 通信发生在每层内部，频率很高，所以通常要求高速互联，最好放在同一节点的 NVLink/NVSwitch 内。TP size 越大，单卡计算和参数越少，但通信越多，也要求 hidden size 和 head 数可整除。真实大模型训练中，TP 常和 DP、ZeRO/FSDP、Pipeline Parallel 组合使用。

### 十六、常见追问与回答

问题 1：Tensor Parallel 和 DDP 的区别是什么？

回答：DDP 是复制模型、切分数据，并同步梯度；Tensor Parallel 是切分单层矩阵或 attention head，多卡共同完成同一层计算。

问题 2：Column Parallel 和 Row Parallel 的区别是什么？

回答：Column Parallel 按输出维度切权重，得到分片输出；Row Parallel 按输入维度切权重和输入，各卡得到 partial output，最后需要 all-reduce 求和。

问题 3：为什么 MLP 中常用 Column + Row 组合？

回答：up projection 用 Column Parallel 后中间激活天然分片，activation 可以本地做；down projection 用 Row Parallel 接收分片输入，最后 all-reduce 得到完整输出，减少中间通信。

问题 4：Attention 为什么适合 TP？

回答：多头注意力的 head 维度天然可切分。每张 GPU 可以负责一部分 heads，Q/K/V 和 attention 计算在本地完成，再在输出投影处合并。

问题 5：为什么 TP 不适合随便跨节点？

回答：TP 通信发生在每层内部，频率高、延迟敏感。跨节点网络通常比节点内 NVLink/NVSwitch 慢很多，会导致通信瓶颈。

问题 6：TP 能替代 ZeRO/FSDP 吗？

回答：不能完全替代。TP 切分单层计算，ZeRO/FSDP 分片训练状态。大模型训练经常同时需要两者。

### 十七、常见误区

1. 误区：Tensor Parallel 就是把不同 batch 放到不同 GPU。
   纠正：那是数据并行。TP 是把同一层计算切到多张 GPU。

2. 误区：TP 只省显存，没有通信代价。
   纠正：TP 每层内部都有通信，通信代价很高，尤其跨节点。

3. 误区：TP size 越大越好。
   纠正：TP size 过大可能导致通信超过计算收益，还要求维度可整除。

4. 误区：Column Parallel 和 Row Parallel 都需要立即 gather 完整输出。
   纠正：高效设计会让相邻层接受分片输出，尽量延迟通信。

5. 误区：Attention head 随便切都可以。
   纠正：GQA/MQA、head 数和 KV head 分布会影响切分方式。

6. 误区：训练用 TP，推理就不需要 TP。
   纠正：大模型推理也常用 TP，但要重新权衡 latency、throughput 和 KV Cache。

### 本讲总结

本讲最重要的结论：

1. Tensor Parallel 是切分单层张量计算，和 DDP 的数据切分、ZeRO/FSDP 的状态分片不同。
2. Column Parallel 按输出维度切权重，Row Parallel 按输入维度切权重并需要对 partial output 求和。
3. Transformer MLP 常用 Column Parallel up projection + Row Parallel down projection 的组合来减少通信。
4. Attention 的 head 维度天然适合 TP，但 GQA/MQA 等结构需要特殊处理。
5. TP 通信发生在层内部，频率高，通常需要 NVLink/NVSwitch 等高速互联。
6. TP size 要结合 hidden size、head 数、硬件拓扑、通信开销和计算饱和度选择。
7. 真实大模型训练通常将 TP 与 DP、ZeRO/FSDP、Pipeline Parallel 和 Activation Checkpointing 组合使用。

## 第 44 讲：Pipeline Parallel

### 本讲目标

学完这一讲，你应该能够回答：

1. Pipeline Parallel 解决什么问题，和 DDP、Tensor Parallel、FSDP 有什么不同。
2. 为什么可以把 Transformer 按层切到不同 GPU 上。
3. micro-batch、pipeline stage、pipeline bubble 分别是什么。
4. GPipe 和 1F1B 调度的核心区别是什么。
5. Pipeline Parallel 的显存、通信、吞吐和负载均衡问题在哪里。
6. 面试中如何回答“一个超深模型如何跨多卡训练”。

Pipeline Parallel，简称 PP，中文常叫流水线并行。

上一讲 Tensor Parallel 是切单层矩阵。

Pipeline Parallel 是切模型层。

也就是说：

> 把模型的不同层放到不同 GPU 上，前面的 GPU 负责前几层，后面的 GPU 负责后几层，数据像流水线一样依次经过这些 stage。

如果模型很深，或者单卡放不下所有层，PP 是一个自然选择。

### 一、为什么需要 Pipeline Parallel

Transformer 是层堆叠结构。

例如一个 48 层模型可以看成：

```text
Embedding -> Block 1 -> Block 2 -> ... -> Block 48 -> LM Head
```

如果一张 GPU 放不下全部层，可以把模型切成几段。

例如 4 个 pipeline stage：

```text
Stage 0: Embedding + Block 1-12
Stage 1: Block 13-24
Stage 2: Block 25-36
Stage 3: Block 37-48 + LM Head
```

每个 stage 放在一张或一组 GPU 上。

forward 时，输入先经过 Stage 0，再把激活传给 Stage 1，再传给 Stage 2，最后到 Stage 3。

backward 时，梯度从后往前传。

这就是流水线并行。

PP 的核心目标是：

1. 把模型层切开，降低单卡参数显存。
2. 让多个 stage 同时处理不同 micro-batch，提高设备利用率。
3. 支持比单卡或单个 TP group 更深的模型。

### 二、PP 和其他并行方式的区别

四种并行方式可以这样区分：

```text
Data Parallel：复制模型，切数据
ZeRO/FSDP：切训练状态
Tensor Parallel：切单层矩阵计算
Pipeline Parallel：切模型层
```

例如一个 96 层 Transformer：

1. DDP 会让每张卡都有完整 96 层。
2. FSDP 会分片参数状态，但按模块计算时仍围绕完整层执行。
3. TP 会把每层内部的 Linear/Attention 切开。
4. PP 会让不同 GPU 负责不同层段。

PP 特别适合模型很深的情况。

TP 特别适合模型很宽的情况。

ZeRO/FSDP 特别适合训练状态太大的情况。

真实大模型训练通常组合使用三者。

### 三、Pipeline Stage 和 Micro-batch

Pipeline Parallel 中有两个基础概念。

第一个是 pipeline stage。

一个 stage 是模型的一段连续层。

例如 4 stage pipeline，每个 stage 负责 12 层。

第二个是 micro-batch。

如果把一个大 batch 一次性送进 pipeline，会发生什么？

Stage 0 先工作，Stage 1 等着。

Stage 0 完成后，Stage 1 开始工作，Stage 0 又闲着。

这样利用率很差。

解决方法是把 batch 切成多个 micro-batch。

例如一个 global batch 被切成 8 个 micro-batch。

Stage 0 处理 micro-batch 1 后，马上处理 micro-batch 2。

同时 Stage 1 可以处理 micro-batch 1。

这样不同 stage 可以同时工作。

流水线像工厂生产线：

```text
Stage 0 做第一个工序
Stage 1 做第二个工序
Stage 2 做第三个工序
```

不同产品在不同工序上同时加工。

### 四、Pipeline Bubble 是什么

Pipeline bubble 指流水线中设备空闲的时间。

流水线开始时，后面的 stage 没有输入，只能等待。

流水线结束时，前面的 stage 已经没有新 micro-batch，也会等待后面完成。

这些空闲就是 bubble。

例如 4 个 stage，只有 1 个 micro-batch。

那就几乎没有并行收益，因为一次只能从 Stage 0 走到 Stage 3。

如果有很多 micro-batch，流水线更容易填满，bubble 比例下降。

粗略来说：

> micro-batch 数越多，pipeline bubble 越小，但调度和显存压力可能增加。

Pipeline bubble 是 PP 的核心效率问题。

面试中一定要能解释。

GPipe 论文的核心思路就是把 batch 切成多个 micro-batch，让顺序层网络可以像流水线一样跨 accelerator 执行。一个常用的粗略估算是：如果有 `P` 个 pipeline stages、`M` 个 micro-batches，并假设每个 stage 的每个 micro-batch 耗时接近，那么流水线需要大约 `M + P - 1` 个时间片完成 forward。理想情况下没有 bubble 只需要 `M` 个时间片，所以 forward 阶段的利用率可以粗略写成：

```math
U \approx \frac{M}{M + P - 1}
```

对应的 bubble 比例可以粗略理解为：

```math
B \approx \frac{P - 1}{M + P - 1}
```

这里 `P` 是 pipeline stage 数，`M` 是 micro-batch 数。这个公式只是用于建立直觉：stage 越多，warmup/cooldown 空闲越多；micro-batch 越多，bubble 被摊薄。但真实训练还会受到 backward 调度、stage 负载不均、通信、activation checkpointing 和 kernel 效率影响，不能只靠这个公式做容量规划。

下面这个最小 demo 用 ASCII 时间线模拟 forward-only pipeline，并计算上面的粗略利用率。它不模拟 backward，只用于理解为什么 micro-batch 数太少会让 bubble 很大。

```python
def pipeline_timeline(num_stages, num_micro_batches):
    total_slots = num_stages + num_micro_batches - 1
    rows = []
    for stage in range(num_stages):
        row = []
        for t in range(total_slots):
            micro_batch = t - stage
            if 0 <= micro_batch < num_micro_batches:
                row.append(f"F{micro_batch}")
            else:
                row.append(".")
        rows.append(row)
    return rows


def print_timeline(num_stages, num_micro_batches):
    rows = pipeline_timeline(num_stages, num_micro_batches)
    total_slots = len(rows[0])
    busy_slots = num_stages * num_micro_batches
    utilization = busy_slots / (num_stages * total_slots)
    bubble = 1.0 - utilization

    print(f"stages={num_stages}, micro_batches={num_micro_batches}")
    for stage, row in enumerate(rows):
        print(f"stage {stage}:", " ".join(row))
    print("utilization:", round(utilization, 3))
    print("bubble:", round(bubble, 3))


print_timeline(num_stages=4, num_micro_batches=2)
print()
print_timeline(num_stages=4, num_micro_batches=8)
```

典型输出是：

```text
stages=4, micro_batches=2
stage 0: F0 F1 . . .
stage 1: . F0 F1 . .
stage 2: . . F0 F1 .
stage 3: . . . F0 F1
utilization: 0.4
bubble: 0.6

stages=4, micro_batches=8
stage 0: F0 F1 F2 F3 F4 F5 F6 F7 . . .
stage 1: . F0 F1 F2 F3 F4 F5 F6 F7 . .
stage 2: . . F0 F1 F2 F3 F4 F5 F6 F7 .
stage 3: . . . F0 F1 F2 F3 F4 F5 F6 F7
utilization: 0.727
bubble: 0.273
```

这个结果和公式直觉一致：`P=4` 不变时，`M=2` 的 bubble 很大；`M=8` 时流水线更容易填满，利用率明显更高。

### 五、GPipe 调度

GPipe 是一种经典 pipeline 调度。

它的思想是：

1. 先让所有 micro-batch 完成 forward。
2. 再让所有 micro-batch 完成 backward。

也就是：

```text
Forward phase:  F1, F2, F3, ...
Backward phase: B1, B2, B3, ...
```

优点：

1. 概念简单。
2. 实现相对容易。
3. 梯度等价性清晰。

缺点：

1. forward 激活要保存到 backward，显存压力大。
2. pipeline bubble 明显。
3. 设备利用率不一定最好。

因为所有 micro-batch 都 forward 完后才 backward，中间需要保存很多激活。

这会增加 activation memory。

所以 GPipe 常结合 activation checkpointing。

### 六、1F1B 调度

1F1B 是 one-forward-one-backward。

它的思想是：

流水线 warmup 后，每个 stage 尽量交替执行一个 forward 和一个 backward。

也就是说，不是等所有 forward 完再 backward，而是让 backward 尽早开始。

这样可以减少激活保存时间，降低显存。

1F1B 通常包括三个阶段：

1. warmup：先填充流水线。
2. steady state：交替执行 forward/backward。
3. cooldown：处理剩余 backward。

优点：

1. 激活显存更低。
2. 设备利用率更好。
3. 更适合大模型训练。

缺点：

1. 调度更复杂。
2. 实现和 debug 更难。
3. 不同 stage 的依赖关系更复杂。

现代大模型 pipeline 训练中，1F1B 或其变体很常见。

### 七、Pipeline 并行中的通信

PP 的通信主要发生在相邻 stage 之间。

forward 时，Stage i 把 activation 发送给 Stage i+1。

backward 时，Stage i+1 把 activation gradient 发送回 Stage i。

通信对象不是完整参数，而是边界激活和梯度。

通信量和以下因素有关：

1. micro-batch size。
2. sequence length。
3. hidden size。
4. pipeline stage 边界数量。
5. 是否使用 tensor parallel。

如果 sequence length 很长，activation 通信会变大。

如果 stage 放在不同节点，跨节点通信可能成为瓶颈。

因此 pipeline stage 的放置要考虑硬件拓扑。

常见策略是：

1. 节点内做 TP。
2. 节点间做 PP。
3. 更外层做 DP。

这不是绝对规则，但很常见。

### 八、负载均衡：每个 stage 要差不多忙

Pipeline Parallel 很怕负载不均衡。

如果 Stage 0 很快，Stage 2 很慢，那么 Stage 0 会经常等待 Stage 2。

整个 pipeline 的吞吐由最慢 stage 决定。

负载不均衡来源包括：

1. 每个 stage 层数不一样。
2. embedding 或 LM head 额外开销大。
3. 某些层结构不同。
4. MoE 层负载不均。
5. activation checkpointing 配置不同。
6. 跨节点通信慢。

对于标准 Transformer，可以按层数平均切。

但 embedding 和 LM head 可能让首尾 stage 更重。

常见做法是：

1. 首尾 stage 少分一些 Transformer blocks。
2. 把 embedding 和 LM head 权重 tie 或特殊处理。
3. 用 profiler 测每个 stage 时间后重新切分。
4. 对 MoE 做更精细负载均衡。

### 九、Pipeline Parallel 的显存特点

PP 可以减少单卡参数显存，因为每个 stage 只保存一部分层。

但它也有新的显存压力。

主要来自 activation。

对于每个 micro-batch，stage 需要保存 forward activation，供 backward 使用。

micro-batch 越多，需要保存的激活可能越多。

GPipe 中这个问题尤其明显，因为所有 forward 先完成。

1F1B 可以缓解，因为 backward 更早发生，activation 保存时间更短。

此外，activation checkpointing 也常用于 PP。

它通过不保存部分中间激活，在 backward 时重新计算，换显存为计算。

所以 PP 显存优化通常要同时考虑：

1. stage 切分。
2. micro-batch 数。
3. 调度方式。
4. activation checkpointing。
5. sequence length。

### 十、Micro-batch 数怎么选

micro-batch 数，也叫 chunks，影响 pipeline 效率。

如果 micro-batch 太少：

1. pipeline bubble 大。
2. stage 空闲多。
3. 设备利用率低。

如果 micro-batch 太多：

1. 调度开销增加。
2. activation 管理复杂。
3. per-micro-batch 太小，计算不饱和。
4. 可能影响 kernel 效率。

所以 micro-batch 数要折中。

通常希望 micro-batch 数明显大于 pipeline stage 数，以减少 bubble。

但也不能让每个 micro-batch 小到 GPU 算不满。

这里和 global batch size 也有关：

```text
global_batch_size = micro_batch_size * micro_batch_count * data_parallel_size
```

如果还使用 gradient accumulation，概念会更复杂，但本质仍是把大 batch 拆成更小块流过 pipeline。

### 十一、PP 与梯度累积

PP 中通常一个 global batch 会拆成多个 micro-batch。

这些 micro-batch 的梯度需要累积后再做 optimizer step。

也就是说，pipeline 内部处理多个 micro-batch，并不是每个 micro-batch 都单独更新参数。

否则不同 micro-batch 会看到不同版本参数，训练语义会变复杂。

常规同步 pipeline 训练中，一个 batch 的所有 micro-batch 完成 forward/backward 后，再统一 optimizer step。

这保证等价于对大 batch 累积梯度。

但一些高级 pipeline 调度会涉及权重版本问题。

例如异步 pipeline 可能存在 stale weights。

面试中如果不是专门问异步训练，可以重点讲同步 PP。

### 十二、Interleaved Pipeline Parallel

普通 PP 中，每个 GPU 负责一段连续层。

Interleaved PP 会让一个 GPU 负责多个非连续或更小的 virtual pipeline stages。

例如 4 张 GPU、16 层模型。

普通切法：

```text
GPU0: layers 1-4
GPU1: layers 5-8
GPU2: layers 9-12
GPU3: layers 13-16
```

Interleaved 可能把每个 GPU 分成多个 virtual chunks，用更细粒度调度减少 bubble。

优点：

1. 减少 pipeline bubble。
2. 改善负载均衡。
3. 提高吞吐。

缺点：

1. 调度更复杂。
2. 通信模式更复杂。
3. 实现和 debug 更难。

大型训练框架中常提供 interleaved schedule 作为性能优化。

### 十三、PP 与 checkpoint

Pipeline Parallel 的 checkpoint 也有复杂性。

每个 stage 只保存自己负责的层。

如果还结合 TP 和 ZeRO/FSDP，checkpoint 会同时包含：

1. pipeline stage 分片。
2. tensor parallel 分片。
3. optimizer state 分片。
4. data parallel 状态。

恢复时要确保：

1. stage mapping 一致。
2. layer assignment 一致。
3. rank mapping 一致。
4. TP/PP/DP 配置一致或支持 reshard。
5. optimizer state 和 scheduler state 正确恢复。

如果改变 pipeline size，例如从 PP=4 改成 PP=8，需要重新切分层和转换 checkpoint。

这叫 reshard 或 repartition。

工程上并不简单。

### 十四、PP 的常见工程坑

坑一：pipeline bubble 太大。

micro-batch 数太少，设备空闲严重。

坑二：stage 负载不均。

某个 stage 成为瓶颈，整个 pipeline 等它。

坑三：micro-batch 太小。

GPU 计算不饱和，kernel 效率低。

坑四：跨节点激活通信慢。

stage 放置不合理，通信延迟高。

坑五：调度实现错误。

forward/backward 依赖错，可能 deadlock 或梯度错误。

坑六：activation 显存爆炸。

micro-batch 多、GPipe 调度、长序列会增加 activation 压力。

坑七：checkpoint 和 rank mapping 错。

恢复后层加载到错误 stage，训练直接错或 silent failure。

坑八：与 TP/FSDP 组合复杂。

多维并行 group 设置错会导致通信 hang。

### 十五、代码示例：流水线直觉伪代码

下面是一个极简伪代码，只展示 PP 的直觉。

```python
class PipelineStage0:
    def __init__(self, embedding, blocks):
        self.embedding = embedding
        self.blocks = blocks

    def forward(self, input_ids):
        x = self.embedding(input_ids)
        for block in self.blocks:
            x = block(x)
        return x  # send to next stage


class PipelineStage1:
    def __init__(self, blocks, lm_head):
        self.blocks = blocks
        self.lm_head = lm_head

    def forward(self, hidden_states):
        x = hidden_states
        for block in self.blocks:
            x = block(x)
        return self.lm_head(x)
```

真实 PP 框架会负责：

1. stage 间 send/recv。
2. micro-batch 切分。
3. forward/backward 调度。
4. activation 保存和释放。
5. gradient accumulation。
6. 与 TP/DP/FSDP 通信组组合。

用户通常不会手写完整 pipeline 调度，因为非常容易出错。

### 十六、什么时候用 Pipeline Parallel

可以用一个简单判断。

如果模型很宽，单层矩阵很大：

优先考虑 Tensor Parallel。

如果模型很深，层数很多，单机放不下所有层：

考虑 Pipeline Parallel。

如果训练状态太大：

考虑 ZeRO/FSDP。

如果 batch 数据量大，模型副本能放下：

使用 Data Parallel。

实际大模型训练中，常见组合是：

```text
TP within node
PP across groups/nodes
DP across replicas
ZeRO/FSDP for state sharding
```

但具体组合取决于：

1. 模型层数。
2. hidden size。
3. sequence length。
4. GPU 显存。
5. 节点内互联。
6. 节点间网络。
7. batch size。
8. 框架支持。

### 十七、面试回答模板

如果面试官问：

> Pipeline Parallel 是什么？为什么会有 pipeline bubble？

可以这样回答：

Pipeline Parallel 是把模型按层切成多个 stage，不同 GPU 或 GPU 组负责不同层段。forward 时激活从前一个 stage 传到后一个 stage，backward 时梯度从后一个 stage 传回前一个 stage。它主要解决模型层数多、完整模型单卡或单组 GPU 放不下的问题，和 Tensor Parallel 切单层矩阵、DDP 切数据不同。

为了让多个 stage 同时工作，会把一个 batch 切成多个 micro-batch，让不同 micro-batch 像流水线一样在不同 stage 上并行处理。pipeline bubble 是流水线开始和结束时 stage 空闲的时间，micro-batch 数太少时 bubble 比例很大。增加 micro-batch 数、使用 1F1B 或 interleaved schedule 可以减少 bubble，但会带来调度复杂度和显存/计算效率权衡。

GPipe 通常先跑完所有 micro-batch 的 forward，再跑 backward，简单但 activation 显存压力大。1F1B 在 warmup 后交替执行 forward 和 backward，可以降低激活保存时间、提升利用率。实际使用 PP 时还要关注 stage 负载均衡、跨 stage 通信、checkpoint 分片和与 TP/DP/ZeRO 的组合。

### 十八、常见追问与回答

问题 1：Pipeline Parallel 和 Tensor Parallel 的区别是什么？

回答：PP 按模型层切分，不同 stage 负责不同层；TP 按单层矩阵或 attention head 切分，多卡共同计算同一层。

问题 2：为什么需要 micro-batch？

回答：如果整个 batch 一次通过 pipeline，很多 stage 会空闲。micro-batch 可以让不同 stage 同时处理不同小批次，提高流水线利用率。

问题 3：pipeline bubble 是什么？

回答：流水线 warmup 和 cooldown 阶段中某些 stage 没有工作可做的空闲时间。micro-batch 数越少，bubble 占比越大。

问题 4：GPipe 和 1F1B 的区别是什么？

回答：GPipe 先完成所有 micro-batch forward，再统一 backward，简单但激活显存高。1F1B 在 warmup 后交替执行 forward 和 backward，显存更低、利用率更好，但调度更复杂。

问题 5：PP 中最容易出现什么性能问题？

回答：pipeline bubble 大、stage 负载不均、micro-batch 太小、跨 stage 通信慢、activation 显存高。

问题 6：PP 能否替代 ZeRO/FSDP？

回答：不能完全替代。PP 切层，ZeRO/FSDP 切训练状态。PP 能减少每个 stage 的参数，但 optimizer state、activation 和数据并行冗余仍可能需要 ZeRO/FSDP 处理。

### 十九、常见误区

1. 误区：Pipeline Parallel 就是 DDP 的另一种名字。
   纠正：DDP 复制完整模型切数据，PP 按层切模型。

2. 误区：PP stage 越多越好。
   纠正：stage 越多 bubble 和通信调度越复杂，负载均衡也更难。

3. 误区：micro-batch 越多越好。
   纠正：micro-batch 多能减少 bubble，但过小会导致计算不饱和，调度开销增加。

4. 误区：PP 自动解决所有显存问题。
   纠正：PP 降低参数显存，但 activation、optimizer state 和通信 buffer 仍可能是瓶颈。

5. 误区：按层数平均切一定负载均衡。
   纠正：embedding、LM head、MoE 层、通信位置都会影响 stage 时间。

6. 误区：PP checkpoint 很简单。
   纠正：PP checkpoint 和 stage mapping、rank mapping、TP/DP 配置强相关。

2. 假设 48 层 Transformer 切成 4 个 stage，设计一个基础层分配方案。
3. 解释 pipeline bubble 为什么会出现。

### 本讲总结

本讲最重要的结论：

1. Pipeline Parallel 是按模型层切分，不同 stage 负责不同层段。
2. PP 主要解决模型很深、完整模型单卡或单组 GPU 放不下的问题。
3. micro-batch 让不同 stage 同时处理不同小批次，是提高流水线利用率的关键。
4. pipeline bubble 是 stage 空闲时间，micro-batch 数不足和 stage 数过多都会让 bubble 更明显。
5. GPipe 简单但激活显存高，1F1B 更复杂但通常显存和利用率更好。
6. PP 的工程难点包括 stage 负载均衡、跨 stage 通信、activation 显存、checkpoint 和多维并行组合。
7. 真实大模型训练常把 PP 与 TP、DP、ZeRO/FSDP 和 Activation Checkpointing 组合使用。

## 第 45 讲：Sequence Parallel 与 Context Parallel

### 本讲目标

学完这一讲，你应该能够回答：

1. 为什么长序列训练会带来巨大的 activation 显存和 attention 计算压力。
2. Sequence Parallel 和 Context Parallel 都沿序列维度切分，但解决的问题有什么不同。
3. Sequence Parallel 在 Tensor Parallel 体系中如何减少 activation 冗余。
4. Context Parallel 如何把长上下文切到多张 GPU 上共同处理。
5. 沿序列维度切分时，LayerNorm、MLP、Attention 分别有什么通信需求。
6. 面试中如何回答“长上下文大模型训练显存不够怎么办”。

前几讲我们讲了 DDP、ZeRO/FSDP、Tensor Parallel 和 Pipeline Parallel。

这些并行方式分别切数据、训练状态、单层矩阵和模型层。

本讲关注另一个维度：序列维度。

也就是把 token 序列切开。

对于长上下文模型，这一点越来越重要。

当上下文长度从 4K 增加到 32K、128K、1M 时，问题不只是参数多，而是激活和注意力计算爆炸。

### 一、为什么序列维度会成为瓶颈

Transformer 输入通常可以写成：

```text
X: [batch, seq_len, hidden_size]
```

当 `seq_len` 增大时，很多东西都会变大。

第一，activation 显存线性增长。

每层都要保存中间激活用于 backward。

序列越长，activation 越大。

第二，attention 计算和显存更糟。

标准 self-attention 需要计算 `QK^T`。

它的形状大致是：

```text
[batch, heads, seq_len, seq_len]
```

所以注意力复杂度和 `seq_len^2` 相关。

如果 seq_len 从 4K 变成 32K，长度增加 8 倍，attention matrix 理论上增加 64 倍。

第三，通信量也会增加。

如果在多卡间切分序列，需要交换 token、KV、attention partial result 或归一化统计。

因此长上下文训练需要多种技术组合：

1. FlashAttention。
2. activation checkpointing。
3. sequence parallel。
4. context parallel。
5. ring attention。
6. block-sparse attention。
7. sliding window attention。
8. memory efficient kernels。

本讲重点讲 SP 和 CP。

### 二、Sequence Parallel 和 Context Parallel 的直觉区别

Sequence Parallel，简称 SP。

Context Parallel，简称 CP。

它们都和序列维度有关，但动机不同。

Sequence Parallel 常出现在 Tensor Parallel 体系中，目的是：

> 减少 TP 中某些操作对 activation 的重复保存，把 sequence 维度上的激活分片给不同 GPU。

它更多是一个 activation memory optimization。

Context Parallel 的目标更直接：

> 当上下文太长，单张 GPU 或单个 TP group 放不下完整序列时，把完整 context 切到多张 GPU 上共同计算 attention。

它更多是长上下文训练和推理的并行方式。

可以粗略理解：

```text
Sequence Parallel：主要服务 TP，减少 activation 冗余
Context Parallel：主要服务长上下文，把 sequence/context 本身切开
```

不同框架中的术语可能略有差异，但面试中讲清这个直觉很重要。

### 三、Sequence Parallel 解决什么问题

在 Tensor Parallel 中，很多线性层的权重被切分。

但某些操作并不容易按 hidden 维度切，例如：

1. LayerNorm。
2. Dropout。
3. Residual add。
4. 某些 elementwise operation。

这些操作可能导致每个 TP rank 都保存完整 activation。

如果序列很长，activation 冗余很浪费。

Sequence Parallel 的想法是：

> 对不能按 hidden 维度高效分片的部分，把 activation 沿 sequence 维度切开，让每个 rank 只保存一部分 tokens 的 activation。

例如输入形状：

```text
[batch, seq_len, hidden]
```

如果有 4 个 SP rank，可以按 sequence 维度切成：

```text
rank0: tokens 0-1023
rank1: tokens 1024-2047
rank2: tokens 2048-3071
rank3: tokens 3072-4095
```

这样每个 rank 只保存 1/4 tokens 的某些 activation。

显存下降明显。

可以用一个很粗略的 activation 估算建立直觉。假设某个 activation 形状是 `[B, T, H]`，每个元素占 `b` bytes。如果不沿 sequence 分片，每个 TP rank 都保存完整 activation，则单 rank 需要：

```math
M_{\mathrm{full}} = B \cdot T \cdot H \cdot b
```

如果用 `S` 个 sequence parallel ranks 沿 `T` 维切分，理想情况下单 rank 保存：

```math
M_{\mathrm{sp}} = \frac{B \cdot T \cdot H \cdot b}{S}
```

这个公式只描述某类 activation 的长期保存量，不包含 layout 转换通信、临时 buffer、attention score、Q/K/V、checkpointing 和 fragmentation。它的价值是说明：当 `T` 很大时，沿 sequence 维度分片可以直接降低某些 activation 的单卡占用。

### 四、Sequence Parallel 中哪些操作容易切

沿 sequence 维度切分时，有些操作天然容易处理。

例如 LayerNorm。

LayerNorm 通常在 hidden 维度上对每个 token 独立做归一化。

每个 token 的归一化不依赖其他 token。

所以如果按 sequence 切分，每个 rank 拿到一部分 token，就可以本地做 LayerNorm。

Residual add 也是逐元素操作。

如果两个 tensor 在相同 sequence 分片上，本地相加即可。

Dropout 也是逐元素操作，只要随机数处理正确，也可以本地完成。

MLP 中的逐 token 线性变换也不依赖其他 token。

但如果它已经用了 Tensor Parallel，可能需要在 hidden 维度和 sequence 维度之间做 layout 转换。

因此 SP 的难点不是单个 elementwise 操作，而是：

> 如何在不同并行布局之间高效转换，避免通信过多。

### 五、Sequence Parallel 中的通信

SP 常和 TP 配合。

在 TP 中，某些时候 tensor 按 hidden 维度分片。

在 SP 中，某些时候 tensor 按 sequence 维度分片。

这就需要 layout 转换。

常见通信包括：

1. all-gather。
2. reduce-scatter。
3. all-to-all。

例如某个操作需要完整 hidden，但当前 hidden 被 TP 切开，就要 gather。

某个操作后想把 activation 沿 sequence 分片保存，就要 scatter。

高效实现会把这些通信和已有 TP 通信结合，尽量不增加额外通信。

Sequence Parallel 的价值在于显存收益通常比较明确，尤其长序列下 activation 很大。

但如果实现不佳，通信开销可能抵消收益。

### 六、Context Parallel 解决什么问题

Context Parallel 更关注超长上下文。

假设要训练 128K context 的模型。

单个 GPU 或单个 TP group 保存完整序列 activation 很困难。

于是可以把 sequence/context 切给多个 GPU。

例如 128K token，CP size=4：

```text
rank0: token 0-32K
rank1: token 32K-64K
rank2: token 64K-96K
rank3: token 96K-128K
```

每个 rank 只持有一段 context。

问题是 self-attention 中每个 token 理论上要 attend 到前面的所有 token。

对于 causal attention，后面的 token 需要看到前面 token 的 K/V。

如果 K/V 分布在其他 rank 上，就必须通信。

这就是 CP 的核心挑战。

### 七、Context Parallel 中 Attention 的难点

标准 attention：

```text
Attention(Q, K, V) = softmax(QK^T / sqrt(d)) V
```

如果 sequence 被切开，每个 rank 只有一部分 Q/K/V。

对于本 rank 的 Q，它可能需要其他 rank 的 K/V。

尤其是 causal attention 中：

1. 前段 token 只需要看更前面的 token。
2. 后段 token 需要看所有前面段的 token。

这会形成跨 rank 依赖。

解决思路包括：

1. all-gather K/V。
2. ring attention。
3. blockwise attention。
4. reduce-scatter partial results。
5. 分块计算 softmax，保持数值稳定。

最简单的方式是 all-gather 所有 K/V。

但这会让每个 rank 又拿到完整 context 的 K/V，显存和通信压力很大。

更高效的方法是让 K/V 块在 rank 之间流动，逐块计算 attention。

这类方法常被称为 ring attention 或 context parallel attention 的变体。

### 八、Ring Attention 的直觉

Ring attention 的直觉是：

> 不一次性 gather 所有 K/V，而是让每个 rank 持有一块 K/V，然后沿 ring 传给下一个 rank，每个 rank 边接收边计算 partial attention。

例如有 4 个 rank。

每个 rank 初始有自己的 K/V block。

第一轮，每个 rank 用本地 K/V 计算一部分 attention。

第二轮，每个 rank 把自己的 K/V 发给下一个 rank，同时接收上一个 rank 的 K/V。

继续计算。

经过若干轮后，每个 rank 都处理过需要的 K/V block。

这样避免一次性保存全部 K/V。

但 attention 的 softmax 不是简单可加的。

因为 softmax 需要对所有 key 的分数一起归一化。

所以 blockwise attention 要维护数值稳定的 running max 和 running sum。

FlashAttention 中也有类似思想：分块计算 attention，不显式存完整 attention matrix。

CP 的高效实现通常离不开这种 blockwise stable softmax。

下面这个 demo 展示两件事：第一，sequence parallel 能把 `[B,T,H]` activation 按 token 切成多个 shard，降低每个 rank 的保存量；第二，blockwise softmax 不能简单把每个 block 的 softmax 输出相加，而要维护全局 running max 和 running sum，才能和完整 softmax attention 对齐。

```python
import math
import torch


def activation_mib(batch, seq_len, hidden, bytes_per_elem, shards=1):
    return batch * seq_len * hidden * bytes_per_elem / shards / (1024 ** 2)


def full_attention(q, k, v):
    scores = q @ k.T / math.sqrt(q.shape[-1])
    probs = torch.softmax(scores, dim=-1)
    return probs @ v


def blockwise_attention(q, k, v, block_size):
    # Online softmax over key blocks for one query block.
    running_max = None
    running_sum = None
    running_out = None

    for start in range(0, k.shape[0], block_size):
        k_block = k[start:start + block_size]
        v_block = v[start:start + block_size]
        scores = q @ k_block.T / math.sqrt(q.shape[-1])

        block_max = scores.max(dim=-1, keepdim=True).values
        new_max = block_max if running_max is None else torch.maximum(running_max, block_max)

        exp_scores = torch.exp(scores - new_max)
        block_sum = exp_scores.sum(dim=-1, keepdim=True)
        block_out = exp_scores @ v_block

        if running_sum is None:
            running_sum = block_sum
            running_out = block_out
        else:
            old_scale = torch.exp(running_max - new_max)
            running_sum = running_sum * old_scale + block_sum
            running_out = running_out * old_scale + block_out

        running_max = new_max

    return running_out / running_sum


batch, seq_len, hidden = 2, 8192, 4096
print("full activation MiB:", round(activation_mib(batch, seq_len, hidden, 2), 1))
print("SP=4 activation MiB per rank:", round(activation_mib(batch, seq_len, hidden, 2, shards=4), 1))

torch.manual_seed(0)
q = torch.randn(3, 4)
k = torch.randn(10, 4)
v = torch.randn(10, 6)

full = full_attention(q, k, v)
blocked = blockwise_attention(q, k, v, block_size=4)
print("max attention error:", (full - blocked).abs().max().item())
print("outputs match:", torch.allclose(full, blocked, atol=1e-6))
```

典型输出是：

```text
full activation MiB: 128.0
SP=4 activation MiB per rank: 32.0
max attention error: 2.384185791015625e-07
outputs match: True
```

这个 demo 不是完整 ring attention，也没有模拟真实跨卡通信；它只演示关键数值点：blockwise attention 想和 full attention 等价，就必须用稳定的 online softmax 合并所有 K/V block 的贡献。真实 CP/ring attention 还要处理 causal mask、block 调度、K/V 传输、负载均衡和通信计算 overlap。

### 九、SP 与 CP 的区别再总结

可以从几个维度对比。

目标：

```text
SP：减少 TP 中 activation 冗余
CP：支持超长 context，把长序列切给多卡
```

主要切分对象：

```text
SP：部分 activation 沿 sequence 维度分片
CP：完整 context/QKV/attention 计算沿 sequence 维度分片
```

常见适用场景：

```text
SP：Transformer TP 训练中的显存优化
CP：长上下文训练和推理，例如 32K、128K、1M context
```

通信复杂度：

```text
SP：layout 转换通信
CP：attention 跨 context block 通信，通常更复杂
```

面试中如果说“SP 和 CP 都是切 sequence”，还不够。

要进一步说清：SP 更偏 activation sharding，CP 更偏 long-context attention parallelism。

### 十、长上下文训练中的其他技术

SP 和 CP 往往不是单独使用。

长上下文训练常结合：

1. FlashAttention。
2. activation checkpointing。
3. sequence packing。
4. sliding window attention。
5. block-sparse attention。
6. RoPE scaling 或位置外推。
7. KV cache optimization。
8. gradient checkpointing。
9. mixed precision。

例如 128K context 训练中，可能需要：

1. FlashAttention 降低 attention 显存。
2. CP 切 context。
3. SP 减少 activation 冗余。
4. activation checkpointing 进一步省显存。
5. TP 切宽层。
6. DP 扩大吞吐。

这就是大规模训练系统的复杂性。

### 十一、通信和硬件拓扑

沿 sequence 切分后，通信可能非常频繁。

尤其 CP 中 attention 需要跨 rank 交换 K/V 或 partial results。

因此硬件拓扑很关键。

如果 CP 通信在慢速跨节点网络上进行，可能非常慢。

常见原则：

1. 高频通信尽量放在高速互联内。
2. TP/SP/CP 通常更依赖低延迟高带宽。
3. DP 相对更适合跨节点扩展。
4. PP 可以跨节点，但要控制 stage 边界通信。

当然具体取决于模型和框架实现。

面试中可以说：

> 并行维度的映射要和硬件拓扑匹配，高频小粒度通信不适合放在慢网络上。

### 十二、长上下文中的负载均衡

Context Parallel 还要考虑负载均衡。

如果是 causal attention，不同位置 token 的可见历史长度不同。

前面的 token attend 的内容少，后面的 token attend 的内容多。

如果简单按连续区间切 sequence，后面的 rank 可能更重。

一些实现会通过算法设计或调度缓解不均衡。

例如：

1. blockwise attention。
2. zig-zag partition。
3. ring-based schedule。
4. 分块均衡分配。

核心目标是让每个 rank 计算量和通信量尽量接近。

否则整个训练会被最慢 rank 拖住。

### 十三、SP/CP 与推理

长上下文推理也可能使用 CP。

训练时关注 forward/backward 和 activation。

推理时关注：

1. prefill 阶段长上下文 attention。
2. decode 阶段 KV cache。
3. 首 token 延迟。
4. token-by-token 生成吞吐。

CP 对 prefill 特别有价值，因为 prefill 要处理完整 prompt。

如果 prompt 很长，单卡 prefill 压力大。

把 context 切到多卡可以降低单卡显存和计算压力。

但 decode 阶段每步只生成一个 token，通信模式不同。

所以训练中的 CP 策略不一定直接等同于推理部署策略。

### 十四、工程坑

坑一：把 SP 和 CP 混为一谈。

两者都切 sequence，但目标和 attention 通信复杂度不同。

坑二：只看参数显存，忽略 activation。

长上下文下 activation 往往是主要瓶颈。

坑三：简单 all-gather 全部 K/V。

实现简单但可能让显存和通信爆炸。

坑四：忽略 causal mask。

按 sequence 切分后，mask 必须正确处理跨 rank 可见性。

坑五：softmax 分块数值不稳定。

blockwise attention 必须维护稳定的 max 和 sum。

坑六：通信拓扑不匹配。

把高频 CP 通信放到慢速网络上会严重拖慢训练。

坑七：评估只看短上下文。

训练长上下文模型后，要评估长上下文检索、长文问答和 needle-in-a-haystack 等任务。

### 十五、面试回答模板

如果面试官问：

> Sequence Parallel 和 Context Parallel 是什么？有什么区别？

可以这样回答：

Sequence Parallel 和 Context Parallel 都和序列维度切分有关，但目标不同。Sequence Parallel 常和 Tensor Parallel 配合，主要用于减少 activation 冗余。因为 TP 中某些操作会让每个 rank 保存完整 sequence activation，SP 会把 activation 沿 sequence 维度分片，让每个 rank 只保存一部分 token，从而降低长序列下的 activation 显存。

Context Parallel 更面向长上下文训练或推理。当 context 长度很大，例如 32K、128K 甚至更长，单个 GPU 或 TP group 难以保存和计算完整序列，就把 context 本身按 sequence 维度切给多个 rank。难点在 attention，因为每个 token 可能需要 attend 到其他 rank 上的 K/V。简单 all-gather K/V 会带来很大显存和通信压力，因此常用 ring attention、blockwise attention 或类似方法，边通信边计算，并维护数值稳定的 softmax。

总结来说，SP 更偏 TP 体系中的 activation sharding，CP 更偏 long-context attention parallelism。长上下文训练通常还要结合 FlashAttention、activation checkpointing、TP、DP 和合适的硬件拓扑。

### 十六、常见追问与回答

问题 1：为什么长上下文训练显存压力这么大？

回答：activation 随 seq_len 线性增长，标准 attention 的分数矩阵随 seq_len 平方增长。即使使用 FlashAttention，长序列下 Q/K/V、激活和通信仍然很大。

问题 2：LayerNorm 按 sequence 切分是否容易？

回答：相对容易。LayerNorm 通常对每个 token 的 hidden 维度归一化，不依赖其他 token，所以按 sequence 切分后可以本地计算。

问题 3：Context Parallel 为什么比 Sequence Parallel 更复杂？

回答：CP 要处理 attention 的跨 token 依赖。一个 rank 的 Q 可能需要其他 rank 的 K/V，还要正确处理 causal mask 和全局 softmax 归一化。

问题 4：为什么不直接 all-gather 全部 K/V？

回答：简单但代价大。长上下文下 K/V 很大，all-gather 会增加显存和通信，可能抵消 CP 的意义。

问题 5：Ring attention 的核心直觉是什么？

回答：让 K/V block 在 rank 之间循环传递，每个 rank 边接收边计算 partial attention，避免一次性保存完整 K/V，同时用 blockwise stable softmax 合并结果。

问题 6：SP/CP 能替代 FlashAttention 吗？

回答：不能完全替代。FlashAttention 优化单设备或局部 attention 计算的显存和 I/O，SP/CP 是跨设备序列并行。长上下文训练常同时使用。

### 十七、常见误区

1. 误区：Sequence Parallel 和 Context Parallel 是同一个东西。
   纠正：SP 更偏 activation sharding，CP 更偏长上下文 attention 并行。

2. 误区：只要用了 TP，就不需要 SP。
   纠正：TP 可能仍有完整 sequence activation 冗余，SP 可以进一步降低激活显存。

3. 误区：Context Parallel 只是把 tokens 平均分给 GPU。
   纠正：关键难点是 attention 跨 rank 依赖、causal mask 和 softmax 合并。

4. 误区：长上下文问题只靠 RoPE scaling 就能解决。
   纠正：RoPE scaling 解决位置泛化问题，不解决训练系统中的显存和计算瓶颈。

5. 误区：CP 通信可以随便跨节点。
   纠正：CP 通信频繁且数据量大，需要考虑硬件拓扑和带宽。

6. 误区：短上下文评估通过就说明 CP 训练成功。
   纠正：必须评估长上下文检索、长文问答和实际长 prompt 任务。

### 十八、小练习

1. 用自己的话解释为什么长上下文会导致 activation 和 attention 压力变大。
2. 对比 Sequence Parallel 和 Context Parallel 的目标、切分对象和通信难点。
3. 解释为什么 LayerNorm 可以比较自然地按 sequence 切分。
4. 画出 4 个 rank 上 context parallel 的 K/V block 交换过程。
5. 列出 5 个训练 128K context 模型可能需要的系统技术。
6. 用 2 分钟回答：“长上下文训练显存不够，你会怎么处理？”

### 本讲总结

本讲最重要的结论：

1. 长上下文训练的瓶颈不只是参数，而是 activation、attention 计算、K/V 和跨设备通信。
2. Sequence Parallel 通常和 Tensor Parallel 配合，主要用于减少沿 sequence 维度的 activation 冗余。
3. Context Parallel 面向超长 context，把完整序列切到多张 GPU 上共同处理。
4. CP 的核心难点在 attention 跨 rank 依赖，需要处理 K/V 通信、causal mask 和稳定 softmax。
5. Ring attention 等方法通过分块传递 K/V，避免一次性 all-gather 完整 context。
6. SP/CP 需要和 FlashAttention、activation checkpointing、TP、DP、硬件拓扑共同设计。
7. 面试中要能说清 SP 与 CP 的区别，而不是只说“都按 sequence 切分”。

## 第 46 讲：3D Parallelism

### 本讲目标

学完这一讲，你应该能够回答：

1. 什么是 3D Parallelism，为什么大模型训练常需要多维并行。
2. Data Parallel、Tensor Parallel、Pipeline Parallel 三个维度如何组合。
3. `world_size = DP * TP * PP` 这类并行配置如何理解。
4. rank group、通信组、硬件拓扑映射为什么重要。
5. 如何根据模型大小、层数、hidden size、集群规模选择并行策略。
6. 面试中如何回答“给你 1024 张 GPU，如何训练一个 70B 模型”。

前面几讲分别讲了 DDP、ZeRO/FSDP、Tensor Parallel、Pipeline Parallel、Sequence/Context Parallel。

真实大模型训练不是只选其中一种。

大多数超大模型训练都会组合多种并行方式。

3D Parallelism 通常指三维并行：

```text
Data Parallel + Tensor Parallel + Pipeline Parallel
```

也就是：

1. 用 TP 切单层矩阵。
2. 用 PP 切模型层。
3. 用 DP 复制多个模型副本处理不同数据。

再加上 ZeRO/FSDP、Sequence Parallel、Activation Checkpointing，就变成更完整的大规模训练系统。

### 一、为什么需要多维并行

单一并行方式总有局限。

只用 Data Parallel：

1. 每张卡要放完整模型。
2. 模型太大时单卡放不下。
3. optimizer state 冗余严重。

只用 Tensor Parallel：

1. 可以切单层矩阵。
2. 但 TP 通信频繁，不适合无限扩大。
3. 跨节点 TP 可能很慢。

只用 Pipeline Parallel：

1. 可以切模型层。
2. 但会有 pipeline bubble。
3. stage 负载均衡困难。
4. batch 和 micro-batch 调度复杂。

只用 ZeRO/FSDP：

1. 可以切训练状态。
2. 但单层计算仍可能太大。
3. 超深模型仍可能需要按层切。

因此真实训练需要组合。

一个直观比喻：

1. TP 解决“每一层太宽”。
2. PP 解决“模型太深”。
3. DP 解决“数据太多，需要吞吐”。
4. ZeRO/FSDP 解决“训练状态太冗余”。
5. SP/CP 解决“序列太长”。

### 二、3D Parallelism 的基本公式

如果总 GPU 数是 `world_size`，三维并行配置通常满足：

```math
W = D \cdot T \cdot P
```

其中：

1. `W` 是 world size，也就是总 GPU/rank 数。
2. `D` 是 data parallel size。
3. `T` 是 tensor parallel size。
4. `P` 是 pipeline parallel size。

例如有 64 张 GPU。

一种配置是：

```text
TP = 4
PP = 4
DP = 4
world_size = 4 * 4 * 4 = 64
```

含义是：

1. 每个模型副本由 `TP * PP = 16` 张 GPU 组成。
2. 一共有 `DP = 4` 个这样的模型副本。
3. 这 4 个副本处理不同数据，并同步梯度或状态。

另一个配置：

```text
TP = 8
PP = 2
DP = 4
world_size = 8 * 2 * 4 = 64
```

两者 GPU 数相同，但通信模式和性能可能很不同。

选择哪种配置取决于模型结构和硬件拓扑。

这个公式容易被误解：`TP * PP` 通常构成一个模型副本内部的切分，`DP` 才表示有多少个这样的副本处理不同数据。也就是说：

```math
G_{\mathrm{replica}} = T \cdot P
```

```math
D = \frac{W}{T \cdot P}
```

其中 `G_replica` 是承载一个模型副本所需的 rank 数。增加 `T` 或 `P` 会让单个副本占用更多 GPU；在 `W` 固定时，`D` 会下降，global batch 也可能随之下降。

### 三、一个具体例子：16 张 GPU

假设有 16 张 GPU，配置：

```text
TP = 2
PP = 4
DP = 2
```

这意味着一个模型副本需要：

```text
TP * PP = 2 * 4 = 8 张 GPU
```

总共有：

```text
DP = 2 个模型副本
```

可以理解为：

```text
Replica 0: GPU 0-7
Replica 1: GPU 8-15
```

每个 replica 内部：

```text
4 个 pipeline stages
每个 stage 由 2 张 GPU 做 tensor parallel
```

两个 replica 之间做 data parallel 同步。

如果再加 ZeRO，它可能在 DP 维度上分片 optimizer state 或参数状态。

这就是多维并行的基本结构。

下面这个 demo 用一个简单 rank mapping 展示 `TP=2, PP=4, DP=2` 时 16 个 rank 如何组成 replica、TP group、PP group 和 DP group。真实框架的 rank 排布可能不同，但这个例子能帮助你理解：多维并行本质是为不同通信模式构造不同 rank group。

```python
def build_rank_table(tp_size, pp_size, dp_size):
    rows = []
    for dp in range(dp_size):
        for pp in range(pp_size):
            for tp in range(tp_size):
                rank = ((dp * pp_size) + pp) * tp_size + tp
                rows.append({"rank": rank, "dp": dp, "pp": pp, "tp": tp})
    return rows


def group_by(rows, keys):
    groups = {}
    for row in rows:
        key = tuple(row[k] for k in keys)
        groups.setdefault(key, []).append(row["rank"])
    return groups


tp_size, pp_size, dp_size = 2, 4, 2
rows = build_rank_table(tp_size, pp_size, dp_size)
world_size = tp_size * pp_size * dp_size

print("world size:", world_size)
print("ranks:", rows)

print("TP groups within each dp,pp stage:")
for key, ranks in group_by(rows, ["dp", "pp"]).items():
    print(key, ranks)

print("PP groups within each dp,tp lane:")
for key, ranks in group_by(rows, ["dp", "tp"]).items():
    print(key, ranks)

print("DP groups across replicas for each pp,tp position:")
for key, ranks in group_by(rows, ["pp", "tp"]).items():
    print(key, ranks)
```

典型输出片段是：

```text
world size: 16
TP groups within each dp,pp stage:
(0, 0) [0, 1]
(0, 1) [2, 3]
...
PP groups within each dp,tp lane:
(0, 0) [0, 2, 4, 6]
(0, 1) [1, 3, 5, 7]
...
DP groups across replicas for each pp,tp position:
(0, 0) [0, 8]
(0, 1) [1, 9]
...
```

这个输出说明：同一个 rank 会同时属于多个通信组。例如 rank 0 既在某个 TP group 中参与层内矩阵通信，也在某个 PP group 中参与 stage 间通信，还在某个 DP group 中和另一个 replica 的对应位置同步梯度或训练状态。rank group 配错，轻则性能很差，重则通信 hang。

### 四、Rank Group 是什么

在多维并行中，不是所有 GPU 都参加同一种通信。

需要建立不同通信组。

常见 group 包括：

1. Tensor Parallel Group。
2. Pipeline Parallel Group。
3. Data Parallel Group。
4. ZeRO/FSDP Group。
5. Context Parallel Group。

Tensor Parallel Group 内部做矩阵切分通信，例如 all-reduce。

Pipeline Parallel Group 内部做 stage 间 send/recv。

Data Parallel Group 内部同步梯度或分片状态。

这些 group 的 rank 关系必须正确。

否则会出现：

1. 通信 hang。
2. 梯度同步错。
3. 参数加载错。
4. checkpoint 无法恢复。
5. 性能极差。

面试中不需要手写完整 rank mapping，但要知道多维并行本质上是构造多个通信子群。

### 五、硬件拓扑映射

并行维度要和硬件拓扑匹配。

不同通信频率不同。

TP 通信发生在每层内部，频率很高，延迟敏感。

因此 TP 最好放在节点内高速互联上，例如 NVLink/NVSwitch。

PP 通信发生在 stage 边界，传 activation 和 gradient。

频率也高，但通常比 TP 通信粒度更粗。

DP 通信通常在 backward 梯度同步或 optimizer state 分片时发生。

相对更适合跨节点扩展。

一个常见映射策略是：

```text
节点内：Tensor Parallel
节点间：Pipeline Parallel 和 Data Parallel
大规模副本间：Data Parallel / ZeRO
```

但这不是绝对规则。

如果节点内有 8 张 GPU，常见做法是 TP=8 或 TP=4 放在节点内。

如果 PP stage 跨节点，要确保 stage 间通信带宽足够。

如果 DP 跨节点，主要看 all-reduce 或 reduce-scatter 性能。

硬件拓扑错误会导致理论上可行的并行配置实际很慢。

### 六、如何选择 TP size

TP size 主要由模型宽度和硬件互联决定。

考虑因素包括：

1. hidden size 是否能被 TP size 整除。
2. attention head 数是否能被 TP size 整除。
3. 单层参数是否太大。
4. 单层矩阵乘是否需要分摊。
5. 节点内高速互联规模。
6. TP 通信开销是否可接受。

如果 hidden size 很大，TP size 可以大一些。

如果模型较小，TP size 太大会导致每张卡计算太少，通信反而主导。

所以 TP size 不是越大越好。

一个经验是：

> TP size 通常优先限制在单节点高速互联范围内，并保证 head 数和 hidden size 可整除。

### 七、如何选择 PP size

PP size 主要由模型层数和单 stage 显存决定。

考虑因素包括：

1. 模型有多少层。
2. 每层参数和 activation 大小。
3. 每个 stage 显存是否足够。
4. stage 负载是否均衡。
5. pipeline bubble 是否可接受。
6. micro-batch 数是否足够。

PP size 太小，单个 stage 层数多，显存压力大。

PP size 太大，stage 数多，bubble 增大，调度复杂。

例如 80 层模型可以切成 8 个 stage，每个 10 层。

但如果首尾 stage 有 embedding 和 LM head，可能要减少首尾 block 数。

选择 PP size 时还要看 micro-batch 数。

如果 PP=16，但 micro-batch 只有 4，pipeline 利用率会很差。

### 八、如何选择 DP size

DP size 决定有多少模型副本并行处理不同数据。

当 TP 和 PP 确定后：

```text
DP = world_size / (TP * PP)
```

DP size 越大：

优点：

1. 数据吞吐更高。
2. global batch 更大。
3. 更容易利用大量 GPU。

缺点：

1. 梯度同步或 ZeRO 通信增加。
2. global batch 过大可能影响优化。
3. 需要调整 learning rate 和 warmup。

DP size 不是越大越好。

如果 global batch 太大，训练可能收敛变差，或者需要更复杂的优化 recipe。

所以 DP size 还受目标 batch size 约束。

### 九、并行配置如何影响 Global Batch

在 3D 并行中，global batch 通常与 DP、micro-batch、gradient accumulation、pipeline micro-batch 有关。

一个简化公式：

```text
global_batch = micro_batch_size * gradient_accumulation_steps * DP
```

注意 TP 和 PP 通常不直接扩大数据 batch。

TP 和 PP 是一个模型副本内部的切分。

DP 才是复制多个模型副本处理不同数据。

所以如果你增加 TP 或 PP，而总 GPU 数固定，DP 可能变小，global batch 也可能变小。

例如 world_size=64。

配置 A：

```text
TP=4, PP=4, DP=4
```

配置 B：

```text
TP=8, PP=4, DP=2
```

配置 B 的模型副本更大，单副本切分更细，但 DP 更小。

如果其他不变，global batch 可能变小。

这会影响训练动态。

### 十、3D 并行与 ZeRO/FSDP

严格说，经典 3D parallelism 是 DP、TP、PP。

但现代训练常把 ZeRO/FSDP 加进来。

ZeRO/FSDP 通常作用在 data parallel 维度上。

例如多个 DP replica 之间，不再简单复制完整 optimizer state，而是按 DP group 分片。

这样可以进一步减少显存。

组合后可能叫：

1. 3D parallel + ZeRO。
2. 4D parallel。
3. hybrid parallelism。

名字不重要，关键是知道每个维度切什么。

一个复杂配置可能是：

```text
TP = 8
PP = 4
DP = 16
ZeRO stage = 1 or 2
Sequence Parallel = on
Activation Checkpointing = on
```

这已经是典型大规模训练配置。

### 十一、一个 70B 模型配置示例

假设要训练一个 70B dense Transformer，有 512 张 GPU，每节点 8 张 GPU。

一种可能思路：

1. TP=8，放在单节点内。
2. PP=4，把模型层切成 4 段。
3. 每个模型副本需要 `8 * 4 = 32` 张 GPU。
4. DP=512/32=16。

配置为：

```text
TP=8, PP=4, DP=16
```

含义：

1. 每个 pipeline stage 内用 8 张 GPU 做 TP。
2. 4 个 stage 组成一个模型副本。
3. 有 16 个副本处理不同数据。
4. DP 维度上同步梯度或使用 ZeRO 分片。

这只是示例，不是唯一答案。

如果网络拓扑不同，或者模型层数、hidden size、seq_len 不同，配置也会变化。

面试中重要的是展示决策逻辑，而不是背一个固定配置。

### 十二、性能调优看哪些指标

3D 并行配置是否好，要看指标。

关键指标包括：

1. tokens per second。
2. GPU utilization。
3. MFU，Model FLOPs Utilization。
4. pipeline bubble ratio。
5. TP communication time。
6. DP all-reduce time。
7. stage time balance。
8. activation memory。
9. optimizer memory。
10. dataloader throughput。

如果 TP communication time 很高，可能 TP size 过大或跨节点。

如果 pipeline bubble 很高，可能 PP size 太大或 micro-batch 太少。

如果某个 stage 时间明显更长，说明负载不均。

如果 GPU utilization 低，可能被通信、数据加载或调度卡住。

所以大规模训练不是配置完就结束，而是要 profile 和迭代。

### 十三、常见配置错误

错误一：TP 跨慢网络。

TP 每层通信频繁，跨节点慢网络会严重拖慢训练。

错误二：PP size 太大但 micro-batch 太少。

pipeline bubble 过大，GPU 空闲。

错误三：DP 太大导致 global batch 过大。

优化变差，loss 曲线异常。

错误四：head 数不能被 TP size 整除。

attention 切分困难或负载不均。

错误五：stage 切分只按层数平均。

忽略 embedding、LM head、MoE 或特殊层开销。

错误六：rank group 设置错。

通信组不匹配，可能 hang 或 silent bug。

错误七：checkpoint 没记录并行配置。

恢复时无法正确加载或转换。

### 十四、3D 并行的工程流程

真实项目中，可以按下面流程设计并行策略。

第一步，估算模型显存。

包括参数、梯度、optimizer state、activation。

第二步，确定是否需要 TP。

看单层矩阵是否太大，hidden size 和 head 数是否适合切分。

第三步，确定是否需要 PP。

看层数和单副本显存，决定是否按层切。

第四步，计算剩余 DP。

根据总 GPU 数和 `DP = world_size / (TP * PP)`。

第五步，检查 global batch。

确认 batch size、gradient accumulation、learning rate 是否合理。

第六步，映射硬件拓扑。

把 TP 放在高速互联内，尽量减少高频通信跨慢网络。

第七步，跑小规模 profile。

观察吞吐、MFU、stage balance、通信时间。

第八步，调整配置。

优化 TP/PP/DP、micro-batch、activation checkpointing 和 ZeRO。

### 十五、面试回答模板

如果面试官问：

> 什么是 3D Parallelism？给你一批 GPU，怎么设计大模型训练并行策略？

可以这样回答：

3D Parallelism 通常指 Data Parallel、Tensor Parallel、Pipeline Parallel 的组合。TP 切单层矩阵和 attention head，适合模型很宽或单层计算太大；PP 按层切模型，适合模型很深或完整模型单卡放不下；DP 复制多个模型副本处理不同数据，提高吞吐。总 GPU 数通常满足 `world_size = DP * TP * PP`。

设计时我会先估算模型参数、optimizer state、activation 显存，再根据 hidden size 和 head 数决定 TP size，通常把 TP 放在节点内高速互联中。然后根据层数和单 stage 显存决定 PP size，同时注意 pipeline bubble 和 stage 负载均衡。剩余 GPU 用作 DP，并根据 DP、micro-batch 和 gradient accumulation 确认 global batch 是否合理。ZeRO/FSDP 可以叠加在 DP 维度上减少 optimizer 和参数状态冗余，activation checkpointing 用来降低激活显存。

最后要根据硬件拓扑和 profiler 调整。TP 通信频繁，不适合跨慢网络；PP 要看 stage 边界通信和 bubble；DP 要看 all-reduce 或 reduce-scatter 开销。好的并行配置不是只让模型跑起来，而是让显存、吞吐、通信和优化动态都平衡。

### 十六、常见追问与回答

问题 1：`world_size = DP * TP * PP` 怎么理解？

回答：TP 和 PP 组成一个模型副本内部的切分，`TP * PP` 张 GPU 共同承载一个模型副本；DP 表示有多少个这样的副本并行处理不同数据。总 GPU 数就是三者乘积。

问题 2：为什么 TP 通常放在节点内？

回答：TP 通信发生在每层内部，频率高、延迟敏感，需要 NVLink/NVSwitch 等高速互联。跨节点慢网络会严重降低效率。

问题 3：增加 PP size 有什么代价？

回答：可以减少每个 stage 的层数和显存，但会增加 pipeline bubble、调度复杂度和 stage 负载均衡难度。

问题 4：增加 TP size 有什么代价？

回答：单卡计算和参数减少，但通信增加，计算粒度变小，且 hidden size 和 head 数需要可整除。

问题 5：增加 DP size 有什么代价？

回答：吞吐增加，但 gradient synchronization 增加，global batch 变大，可能需要调整学习率和优化策略。

问题 6：ZeRO/FSDP 和 3D Parallelism 是什么关系？

回答：经典 3D 是 DP/TP/PP。ZeRO/FSDP 通常叠加在 DP 维度上分片训练状态，进一步省显存，是现代 hybrid parallel training 的重要组成。

### 十七、常见误区

1. 误区：3D 并行就是随便把三种并行打开。
   纠正：需要根据模型结构、显存、通信和硬件拓扑系统设计。

2. 误区：TP、PP、DP 越大越好。
   纠正：每个维度都有代价，TP 增加层内通信，PP 增加 bubble，DP 增大全局 batch 和同步通信。

3. 误区：TP 和 PP 会直接增加 global batch。
   纠正：TP/PP 是模型副本内部切分，主要是 DP 和 gradient accumulation 影响 global batch。

4. 误区：按 GPU 数平均分配就能高效训练。
   纠正：必须考虑 rank group、硬件拓扑、stage balance 和通信频率。

5. 误区：配置能跑就代表配置好。
   纠正：还要看 tokens/s、MFU、通信占比、bubble、显存余量和 loss 曲线。

6. 误区：checkpoint 不需要记录并行配置。
   纠正：TP/PP/DP 配置决定参数分片和 rank mapping，恢复和转换必须依赖这些元数据。

1. 给定 128 张 GPU，设计一个 `TP * PP * DP = 128` 的配置，并解释理由。
2. 为什么 TP 更适合放在节点内高速互联？
3. 如果 PP bubble 很大，你会从哪些方面优化？
4. 如果 global batch 太大导致 loss 变差，你会如何调整 DP、micro-batch 或 accumulation？
5. 列出 8 个评估 3D 并行配置好坏的指标。
6. 用 2 分钟回答：“如何为一个 70B 模型选择 TP、PP、DP？”

### 本讲总结

本讲最重要的结论：

1. 3D Parallelism 通常指 Data Parallel、Tensor Parallel、Pipeline Parallel 的组合。
2. TP 切单层计算，PP 切模型层，DP 切数据和模型副本。
3. 总 GPU 数通常满足 `world_size = DP * TP * PP`，其中 `TP * PP` 构成一个模型副本，DP 表示副本数。
4. 并行配置必须匹配硬件拓扑，尤其 TP 这类高频通信通常应放在高速互联内。
5. TP、PP、DP 每个维度都有收益和代价，不能盲目增大。
6. ZeRO/FSDP、Sequence Parallel、Activation Checkpointing 常与 3D 并行组合，形成 hybrid parallel training。
7. 面试中要展示从模型结构、显存估算、通信拓扑、global batch、profile 指标到调参的完整设计思路。

## 第 47 讲：Activation Checkpointing

### 本讲目标

学完这一讲，你应该能够回答：

1. Activation Checkpointing 解决什么问题。
2. 为什么它本质上是“用额外计算换显存”。
3. Transformer 中哪些 activation 占显存，为什么长上下文时特别严重。
4. checkpoint 粒度如何影响显存、速度和实现复杂度。
5. Activation Checkpointing 如何与 DDP、FSDP、TP、PP、长上下文训练结合。
6. 面试中如何回答“显存不够时除了模型并行还能怎么办”。

Activation Checkpointing 也常叫 gradient checkpointing。

它是大模型训练中非常常用的显存优化技术。

前几讲我们讨论的并行方式主要在切参数、切层、切矩阵或切序列。

Activation Checkpointing 的思路不同：

> forward 时不保存所有中间激活，只保存少量 checkpoint；backward 时需要中间激活，就重新计算一遍。

它不改变模型数学结果，但改变训练过程中的计算和显存开销。

### 一、为什么 activation 会占很多显存

训练神经网络时，forward 不只是算出 loss。

为了 backward，需要保存很多中间结果。

例如一个 Transformer block 中可能保存：

1. LayerNorm 输入或统计。
2. Q、K、V。
3. attention logits 或 softmax 相关中间量。
4. attention output。
5. MLP hidden activation。
6. dropout mask。
7. residual 分支输入。
8. 每层输出 hidden states。

这些中间结果就是 activations。

参数显存只和模型大小有关。

activation 显存还和 batch size、sequence length、hidden size、层数有关。

可以粗略理解为：

```math
M_{\mathrm{act}} \propto B \cdot T \cdot H \cdot L
```

其中 `B` 是 batch size，`T` 是 sequence length，`H` 是 hidden size，`L` 是层数。这个式子只描述很多逐 token activation 的线性部分；标准 attention 如果显式保存 attention score 或 softmax 矩阵，还会出现与 `T^2` 相关的项。

例如某类 per-layer activation 如果形状近似是 `[B, T, H]`，每个元素 `b` bytes，每层保存 `c` 份类似中间量，那么粗略显存是：

```math
M_{\mathrm{act}} \approx B \cdot T \cdot H \cdot L \cdot c \cdot b
```

这个估算故意忽略了 kernel workspace、通信 buffer、显存碎片和框架 metadata，但足够说明一个面试重点：当 `T` 或 `B` 增大时，activation 显存可能比参数显存更快成为瓶颈。

如果是标准 attention，还会有与 `seq_len^2` 相关的中间量。

虽然 FlashAttention 可以避免显式存完整 attention matrix，但长序列下 activation 仍然很大。

所以训练长上下文模型时，activation 往往比参数更先成为瓶颈。

### 二、Activation Checkpointing 的核心思想

普通训练中，forward 会保存大量中间激活。

backward 直接使用这些激活计算梯度。

Activation Checkpointing 改变这个流程。

它只保存某些关键点的激活。

其他中间激活不保存。

等 backward 需要它们时，从最近的 checkpoint 重新 forward 计算出来。

这就是 recomputation。

直觉例子：

假设一个模块由 4 个操作组成：

```text
x -> f1 -> f2 -> f3 -> f4 -> y
```

普通训练会保存 f1、f2、f3 的中间结果。

checkpointing 可能只保存输入 `x` 和输出 `y`。

backward 时如果需要 f2 的输出，就从 `x` 重新执行 f1、f2 得到它。

所以它节省显存，但增加计算。

### 三、用计算换显存

Activation Checkpointing 的本质权衡是：

```text
显存下降，计算增加
```

优点：

1. 可以训练更大模型。
2. 可以使用更长 sequence。
3. 可以增加 batch size。
4. 和并行策略兼容性强。

缺点：

1. backward 会重新计算部分 forward。
2. 训练速度下降。
3. 实现复杂度增加。
4. 随机操作和状态管理要小心。

训练速度下降多少取决于 checkpoint 粒度。

如果 checkpoint 很粗，需要重算很多内容，计算开销大。

如果 checkpoint 很细，节省显存少，调度开销也可能增加。

### 四、Transformer 中常见 checkpoint 粒度

在 Transformer 中，常见 checkpoint 粒度有几种。

第一，按 Transformer block checkpoint。

每个 block 作为一个 checkpoint 单元。

forward 时只保存 block 输入，block 内部中间激活不保存。

backward 时重新计算整个 block。

优点：简单，显存收益明显。

缺点：重算整个 block，计算开销较大。

第二，按 attention 和 MLP 分别 checkpoint。

把一个 block 内部拆成 attention 子模块和 MLP 子模块。

优点：粒度更细，可以更精细控制显存和计算。

缺点：实现更复杂。

第三，只 checkpoint 部分层。

例如每隔几层 checkpoint，或者只对显存最重的层启用。

优点：折中速度和显存。

缺点：需要 profile 决定策略。

第四，selective activation checkpointing。

只对某些中间量选择性重算，不是一整个模块都重算。

优点：更高效。

缺点：框架和 kernel 支持要求高。

### 五、PyTorch 中的基本用法

PyTorch 提供了 checkpoint 工具。

简化示例：

```python
import torch
from torch.utils.checkpoint import checkpoint


class Block(torch.nn.Module):
    def forward(self, x):
        # attention + mlp + residual
        return x


def forward_with_checkpoint(block, x):
    return checkpoint(block, x, use_reentrant=False)
```

在模型中可以这样使用：

```python
for block in self.layers:
    if self.gradient_checkpointing:
        hidden_states = checkpoint(block, hidden_states, use_reentrant=False)
    else:
        hidden_states = block(hidden_states)
```

真实大模型框架会更复杂，因为要处理：

1. attention mask。
2. position ids。
3. KV cache。
4. dropout。
5. mixed precision。
6. FSDP/TP/PP 包装。
7. custom kernels。

但核心思想就是对某个模块启用 checkpoint，backward 时自动重算。

下面这个 demo 更直观：用一个计数器记录模块 forward 被调用了几次。不开 checkpoint 时，一次训练 step 中模块 forward 只执行一次；开启 checkpoint 后，forward 阶段执行一次，backward 为了恢复中间激活会再执行一次。

```python
import torch
from torch import nn
from torch.utils.checkpoint import checkpoint


class CountingBlock(nn.Module):
    def __init__(self):
        super().__init__()
        self.linear = nn.Linear(4, 4)
        self.forward_calls = 0

    def forward(self, x):
        self.forward_calls += 1
        return torch.relu(self.linear(x))


def run_step(use_checkpoint):
    torch.manual_seed(0)
    block = CountingBlock()
    x = torch.randn(2, 4, requires_grad=True)

    if use_checkpoint:
        y = checkpoint(block, x, use_reentrant=False)
    else:
        y = block(x)

    loss = y.square().mean()
    loss.backward()
    return block.forward_calls, x.grad.norm().item()


plain_calls, plain_grad_norm = run_step(use_checkpoint=False)
ckpt_calls, ckpt_grad_norm = run_step(use_checkpoint=True)

print("plain forward calls:", plain_calls)
print("checkpoint forward calls:", ckpt_calls)
print("plain grad norm:", round(plain_grad_norm, 6))
print("checkpoint grad norm:", round(ckpt_grad_norm, 6))
```

典型输出是：

```text
plain forward calls: 1
checkpoint forward calls: 2
plain grad norm: 0.148282
checkpoint grad norm: 0.148282
```

这个 demo 说明：checkpointing 没有改变这个确定性模块的梯度结果，但它确实让 backward 期间多执行了一次 forward。真实训练中，显存收益来自“不保存 block 内部中间激活”；代价就是这类重算，以及在 TP/FSDP 场景下可能重复触发通信。

### 六、随机性和 Dropout 问题

Activation Checkpointing 中一个重要细节是随机性。

如果模块里有 dropout，forward 第一次计算和 backward 重算时必须使用一致的 dropout mask。

否则 backward 对应的计算图就不一致，梯度会错。

框架通常会保存和恢复 RNG state 来保证一致性。

但这会带来额外开销。

如果你使用自定义随机操作、自定义 CUDA kernel 或分布式随机数，要特别小心。

常见问题包括：

1. 重算时 dropout mask 不一致。
2. 不同 rank RNG state 不一致。
3. checkpoint 区域中有副作用操作。
4. 模块 forward 修改了全局状态。

因此 checkpoint 的模块最好是纯函数式的：输入确定，输出确定，没有不可控副作用。

### 七、与 FlashAttention 的关系

FlashAttention 和 Activation Checkpointing 都能省显存，但省的是不同部分。

FlashAttention 主要优化 attention 计算，避免显式存完整 attention matrix，并通过 IO-aware 分块提高效率。

Activation Checkpointing 更通用，它可以减少整个模块中间激活保存。

两者可以一起使用。

例如：

1. FlashAttention 降低 attention 内部显存。
2. Activation Checkpointing 降低 block 级中间激活。
3. Sequence Parallel 降低 activation 冗余。
4. FSDP/ZeRO 降低参数和 optimizer state。

长上下文训练中，通常需要组合多种技术。

### 八、与 FSDP/ZeRO 的关系

FSDP/ZeRO 主要减少参数、梯度和 optimizer state 的显存。

Activation Checkpointing 主要减少 activation 显存。

所以它们互补。

例如模型训练显存可以粗略分为：

```text
total_memory = parameters + gradients + optimizer_states + activations + buffers
```

FSDP/ZeRO 主要降低前三项。

Activation Checkpointing 主要降低 activations。

当 sequence length 很长时，即使用 FSDP，activation 也可能爆炸。

这时 checkpointing 仍然必要。

但组合时要注意 FSDP all-gather 和 recomputation 的交互。

如果 backward 重算时又触发参数 all-gather，会增加通信。

因此 FSDP + checkpointing 需要合理 wrap 粒度和 prefetch 策略。

### 九、与 Pipeline Parallel 的关系

PP 中 activation 保存也很重要。

尤其 GPipe 调度中，会先完成多个 micro-batch forward，再 backward。

这意味着每个 stage 要保存多个 micro-batch 的 activation。

Activation Checkpointing 可以显著降低这个压力。

1F1B 调度已经能减少 activation 保存时间。

但对于长序列或大 batch，仍然可能需要 checkpointing。

PP + checkpointing 的权衡是：

1. 显存下降。
2. stage 计算时间增加。
3. 可能影响 stage 负载均衡。
4. pipeline bubble 和吞吐可能变化。

如果某些 stage 开启 checkpointing、某些不开启，stage 时间可能不均衡。

因此要结合 profiler 调整。

### 十、与 Tensor Parallel 的关系

TP 中每层内部计算被切到多个 GPU。

Activation Checkpointing 可以作用在 TP 包裹后的模块上。

但要注意：

1. 重算时会重新触发 TP 通信。
2. 如果 checkpoint 粒度跨越多个 TP 通信点，重算成本更高。
3. 与 Sequence Parallel 结合时，layout 转换要正确。

例如一个 Transformer block 中包含 TP attention 和 TP MLP。

如果整个 block checkpoint，backward 重算时 attention 和 MLP 的 TP all-reduce 都会再执行。

这会增加通信和计算。

所以大规模训练中 checkpoint 粒度也影响通信开销。

### 十一、选择 checkpoint 粒度的原则

可以从以下几个问题判断。

第一，显存瓶颈在哪里？

如果 activation 是主要瓶颈，checkpointing 很有价值。

如果参数或 optimizer state 是主要瓶颈，优先考虑 FSDP/ZeRO。

第二，训练速度能接受下降多少？

checkpointing 会增加重算，吞吐下降是正常的。

第三，哪些模块 activation 最大？

长 sequence 下 attention 和 MLP 都可能很重。

第四，通信是否会被重复触发？

在 TP/FSDP 下，checkpointing 可能增加通信。

第五，是否影响 pipeline balance？

不同 stage checkpoint 策略不同可能导致负载不均。

常见经验是：

1. 先按 Transformer block checkpoint。
2. 观察显存和吞吐。
3. 如果太慢，改成 selective checkpoint。
4. 如果显存仍不够，增加 checkpoint 范围或结合 SP/CP。

### 十二、显存收益怎么估算

Activation checkpointing 的收益取决于哪些激活不再保存。

如果每层都保存完整中间激活，显存随层数线性增长。

如果按 block checkpoint，只保存每个 block 的输入，block 内部激活在 backward 重算。

显存会显著下降。

但不是降到零。

仍然需要保存：

1. checkpoint 输入。
2. loss 相关激活。
3. 某些必须保存的 tensor。
4. 通信 buffer。
5. framework metadata。

所以不要把 checkpointing 理解成“不存激活”。

更准确是：

> 少存中间激活，只存重算所需的边界激活。

### 十三、训练速度影响

Activation Checkpointing 会增加训练时间。

原因是 backward 中要重算 forward。

如果 checkpoint 整个 Transformer block，那么 backward 会额外执行一次 block forward。

训练开销可能增加 20% 到 50%，具体取决于模型和实现。

但这只是粗略范围。

如果原本因为显存不足只能用很小 batch，checkpointing 允许更大 batch 或更长 context，整体训练效率可能反而更好。

所以评估 checkpointing 不能只看单 step 时间。

要看：

1. 是否能训练目标模型。
2. tokens/s 变化。
3. MFU 变化。
4. batch size 是否提高。
5. sequence length 是否提高。
6. 是否减少 OOM 和重启。

### 十四、工程坑

坑一：checkpoint 区域有副作用。

例如 forward 修改全局状态、计数器、缓存，会导致重算不一致。

坑二：dropout RNG 不一致。

重算时随机 mask 不同，梯度错误。

坑三：和 KV cache 混用。

训练 checkpointing 和推理 KV cache 逻辑不同，不能混淆。

坑四：checkpoint 粒度太粗。

显存省得多，但重算和通信开销过大。

坑五：checkpoint 粒度太细。

显存收益有限，调度 overhead 增加。

坑六：FSDP/TP 下重复通信太多。

重算触发额外 all-gather 或 all-reduce，吞吐下降明显。

坑七：不同 pipeline stage 策略不一致。

导致 stage 时间不均衡，pipeline bubble 增加。

坑八：只在小模型上测试。

小模型看不出显存收益和通信代价，放大后表现不同。

### 十五、代码示例：按 block 开启 checkpointing

下面是一个简化示例。

```python
import torch
from torch import nn
from torch.utils.checkpoint import checkpoint


class TransformerStack(nn.Module):
    def __init__(self, layers, use_checkpointing: bool):
        super().__init__()
        self.layers = nn.ModuleList(layers)
        self.use_checkpointing = use_checkpointing

    def forward(self, hidden_states, attention_mask=None):
        for layer in self.layers:
            if self.use_checkpointing and self.training:
                hidden_states = checkpoint(
                    layer,
                    hidden_states,
                    attention_mask,
                    use_reentrant=False,
                )
            else:
                hidden_states = layer(hidden_states, attention_mask)
        return hidden_states
```

真实模型中还要考虑：

1. keyword arguments。
2. position ids。
3. rotary embeddings。
4. attention implementation。
5. mixed precision。
6. distributed wrappers。

但这个例子体现了核心用法：把层 forward 包进 checkpoint。

### 十六、面试回答模板

如果面试官问：

> Activation Checkpointing 是什么？为什么能省显存？代价是什么？

可以这样回答：

Activation Checkpointing 是一种用额外计算换显存的技术。普通训练在 forward 时会保存大量中间 activation，用于 backward 计算梯度。Checkpointing 只保存少量边界激活，例如每个 Transformer block 的输入，而不保存 block 内部中间结果。backward 时需要这些中间结果，就从保存的边界激活重新 forward 计算出来。

它能显著降低 activation 显存，尤其在大 batch、长 sequence、多层 Transformer 中很有价值。但代价是 backward 期间会重复计算部分 forward，因此训练速度下降。checkpoint 粒度越粗，显存省得越多，但重算越多；粒度越细，速度影响小，但显存收益有限。

它和 FSDP/ZeRO 是互补关系。FSDP/ZeRO 主要省参数、梯度和 optimizer state，activation checkpointing 主要省 activation。长上下文训练中，即使用了 ZeRO/FSDP，也常需要 checkpointing。和 TP/PP 结合时要注意重算会重复触发通信，可能影响吞吐和 pipeline stage balance。

### 十七、常见追问与回答

问题 1：Activation Checkpointing 会改变模型训练结果吗？

回答：理想情况下不改变数学结果，只改变计算过程。但如果 dropout RNG、随机操作或副作用处理不当，可能导致结果不一致。

问题 2：为什么它会让训练变慢？

回答：因为 backward 时要重新执行部分 forward 来恢复中间激活，增加了计算量。

问题 3：它和 ZeRO/FSDP 的区别是什么？

回答：ZeRO/FSDP 主要分片参数、梯度和 optimizer state；activation checkpointing 主要减少 forward 中间激活保存。两者解决的显存项不同。

问题 4：checkpoint 粒度怎么选？

回答：通常先按 Transformer block checkpoint，观察显存和吞吐。如果太慢，可以改细粒度或 selective checkpoint；如果显存仍不够，可以扩大 checkpoint 范围或结合 SP/CP。

问题 5：长上下文训练为什么经常需要 checkpointing？

回答：activation 显存随 sequence length 增长，attention 相关开销更大。长上下文下即使参数能放下，activation 也可能 OOM。

问题 6：checkpointing 和 dropout 有什么关系？

回答：重算时必须使用和原 forward 一致的 dropout mask，因此需要保存和恢复 RNG state，或者确保随机性可复现。

### 十八、常见误区

1. 误区：Activation Checkpointing 会减少参数显存。
   纠正：它主要减少 activation 显存，不直接减少参数或 optimizer state。

2. 误区：checkpointing 是免费省显存。
   纠正：它用重计算换显存，会降低训练速度。

3. 误区：开启 checkpointing 后就不会 OOM。
   纠正：仍可能被参数、optimizer state、通信 buffer 或剩余 activation 卡住。

4. 误区：checkpoint 粒度越粗越好。
   纠正：过粗会导致重算太多，吞吐明显下降。

5. 误区：它和 FlashAttention 是同一种技术。
   纠正：FlashAttention 优化 attention kernel 和 IO，checkpointing 是通用重计算策略。

6. 误区：分布式训练中直接打开 checkpointing 就一定高效。
   纠正：和 FSDP/TP/PP 组合时可能重复通信，需要 profile 和调优。

3. 为什么 checkpointing 能帮助长上下文训练？

### 本讲总结

本讲最重要的结论：

1. Activation Checkpointing 是用额外重计算换 activation 显存的技术。
2. 它不主要减少参数或 optimizer state，而是减少 forward 中间激活保存。
3. Transformer 中 activation 显存随 batch、sequence length、hidden size 和层数增长，长上下文下尤其严重。
4. checkpoint 粒度决定显存收益和重算代价，常见做法是按 Transformer block 或子模块 checkpoint。
5. 它和 ZeRO/FSDP、TP、PP、SP/CP、FlashAttention 是互补关系，不是替代关系。
6. 分布式训练中要注意 RNG、一致性、副作用、重复通信和 pipeline 负载均衡。
7. 面试中要能说明它省哪部分显存、为什么会变慢、如何选择粒度以及如何和并行策略组合。

## 第 48 讲：通信与计算重叠

### 本讲目标

学完这一讲，你应该能够回答：

1. 为什么大模型分布式训练中通信会成为瓶颈。
2. 什么是 communication-computation overlap，为什么它能提高吞吐。
3. DDP bucket overlap、FSDP all-gather prefetch、ZeRO reduce-scatter overlap 分别是什么。
4. Tensor Parallel 和 Pipeline Parallel 中有哪些通信可以被隐藏。
5. 如何用 profiling 判断通信是否被计算隐藏。
6. 面试中如何回答“多卡训练通信太慢怎么办”。

前面几讲介绍了各种并行方式。

并行的代价是通信。

DDP 要同步梯度。

FSDP 要 all-gather 参数、reduce-scatter 梯度。

TP 每层内部要 all-reduce 或 all-gather。

PP 要在 stage 之间传 activation 和 gradient。

CP 要交换 K/V block。

如果每次通信时 GPU 都停下来等待，训练吞吐会很差。

通信与计算重叠的核心思想是：

> 不要等计算全部结束后才通信，也不要通信时让 GPU 闲着，而是尽量一边计算一边通信。

### 一、为什么通信会成为瓶颈

单卡训练主要瓶颈是计算和显存。

多卡训练还多了通信。

常见通信包括：

1. DDP gradient all-reduce。
2. ZeRO/FSDP reduce-scatter gradients。
3. ZeRO/FSDP all-gather parameters。
4. TP all-reduce partial outputs。
5. PP send/recv activations。
6. CP K/V block exchange。
7. checkpoint 写入存储。

通信性能受几个因素影响：

1. tensor 大小。
2. 通信频率。
3. 网络带宽。
4. 网络延迟。
5. GPU 拓扑。
6. collective 算法。
7. 是否与计算重叠。

如果通信不能被隐藏，GPU 会出现等待。

表现是：

1. GPU utilization 下降。
2. step time 增加。
3. tokens/s 不随 GPU 数线性增长。
4. profiler 中 NCCL 时间占比高。

### 二、什么是通信与计算重叠

假设一个训练 step 有两部分：

```text
compute: 100 ms
communication: 40 ms
```

如果完全串行，总时间是：

```text
140 ms
```

如果通信可以完全隐藏在计算中，总时间接近：

```text
100 ms
```

这就是 overlap 的价值。

更一般地，可以把一个 step 的时间写成：

```math
T_{\mathrm{serial}} = T_{\mathrm{comp}} + T_{\mathrm{comm}}
```

```math
T_{\mathrm{overlap}} = T_{\mathrm{comp}} + T_{\mathrm{exposed}}
```

其中 `T_comp` 是不能省掉的计算时间，`T_comm` 是总通信时间，`T_exposed` 是没有被计算隐藏、最终暴露在 step time 里的通信尾巴。被隐藏掉的通信量可以写成：

```math
T_{\mathrm{hidden}} = T_{\mathrm{comm}} - T_{\mathrm{exposed}}
```

如果依赖关系允许且计算足够长，`T_exposed` 可以接近 0；如果通信结果马上被后续计算需要，`T_exposed` 就会很大。一个简单的 overlap 效率指标是：

```math
E_{\mathrm{overlap}} = \frac{T_{\mathrm{hidden}}}{T_{\mathrm{comm}}}
```

`E_overlap` 越接近 1，说明越多通信被隐藏；越接近 0，说明通信基本暴露在 step time 中。

下面这个 demo 用纯 Python 模拟 backward compute 和 DDP bucket all-reduce 的时间线。它不调用多卡通信，只回答一个问题：bucket 在 backward 过程中越早 ready，通信越可能被后面的梯度计算隐藏；最后一个 bucket 往往会形成 tail。

```python
def simulate_backward_overlap(layer_compute_ms, bucket_comm_ms, bucket_ready_after):
    compute_time = 0.0
    comm_stream_free = 0.0
    events = []

    for layer_id, compute_ms in enumerate(layer_compute_ms, start=1):
        start = compute_time
        compute_time += compute_ms
        events.append((start, compute_time, f"compute layer {layer_id} backward"))

        for bucket_id, ready_after in enumerate(bucket_ready_after, start=1):
            if ready_after == layer_id:
                comm_start = max(compute_time, comm_stream_free)
                comm_end = comm_start + bucket_comm_ms[bucket_id - 1]
                comm_stream_free = comm_end
                events.append((comm_start, comm_end, f"all-reduce bucket {bucket_id}"))

    total_comm = sum(bucket_comm_ms)
    serial_time = compute_time + total_comm
    overlap_time = max(compute_time, comm_stream_free)
    exposed_comm = overlap_time - compute_time
    hidden_comm = total_comm - exposed_comm
    overlap_efficiency = hidden_comm / total_comm if total_comm else 0.0

    return {
        "events": sorted(events),
        "compute_time": compute_time,
        "total_comm": total_comm,
        "serial_time": serial_time,
        "overlap_time": overlap_time,
        "exposed_comm": exposed_comm,
        "hidden_comm": hidden_comm,
        "overlap_efficiency": overlap_efficiency,
    }


result = simulate_backward_overlap(
    layer_compute_ms=[25, 25, 25, 25],
    bucket_comm_ms=[35, 35, 35],
    bucket_ready_after=[1, 2, 4],
)

for start, end, name in result["events"]:
    print(f"{start:6.1f}-{end:6.1f} ms | {name}")

print("compute time:", result["compute_time"], "ms")
print("total communication:", result["total_comm"], "ms")
print("serial step time:", result["serial_time"], "ms")
print("overlapped step time:", result["overlap_time"], "ms")
print("exposed communication:", result["exposed_comm"], "ms")
print("hidden communication:", result["hidden_comm"], "ms")
print("overlap efficiency:", round(result["overlap_efficiency"], 3))
```

典型输出是：

```text
   0.0-  25.0 ms | compute layer 1 backward
  25.0-  50.0 ms | compute layer 2 backward
  25.0-  60.0 ms | all-reduce bucket 1
  50.0-  75.0 ms | compute layer 3 backward
  60.0-  95.0 ms | all-reduce bucket 2
  75.0- 100.0 ms | compute layer 4 backward
 100.0- 135.0 ms | all-reduce bucket 3
compute time: 100.0 ms
total communication: 105 ms
serial step time: 205.0 ms
overlapped step time: 135.0 ms
exposed communication: 35.0 ms
hidden communication: 70.0 ms
overlap efficiency: 0.667
```

这个结果说明：105 ms 的通信里有 70 ms 被 backward compute 覆盖，最终只暴露 35 ms 的通信尾巴。真实 DDP 的 bucket ready 顺序由 autograd 图、参数顺序、bucket size 和 unused parameter 处理共同决定，但这个时间线足够说明为什么 bucket 粒度、反向计算长度和尾部通信会影响 tokens/s。

当然现实中很难完全隐藏。

更常见的是部分隐藏，例如总时间从 140 ms 降到 115 ms。

通信与计算重叠依赖几个条件：

1. 通信任务可以提前启动。
2. 后续还有足够计算可以覆盖通信时间。
3. 通信和计算使用的硬件资源不完全冲突。
4. 框架调度支持异步通信。
5. tensor 依赖关系允许并行。

如果某个通信结果马上被下一步计算需要，就很难隐藏。

如果通信结果很晚才需要，就更容易隐藏。

### 三、DDP 中的 bucket overlap

DDP 的典型 overlap 来自 gradient bucket。

反向传播是从最后一层往前计算梯度。

当某些参数的梯度已经计算完成时，不必等所有层梯度都完成。

DDP 会把参数梯度分成 bucket。

当一个 bucket 中所有梯度就绪，就启动 all-reduce。

与此同时，backward 继续计算更前面层的梯度。

这样 all-reduce 可以和 backward compute 重叠。

流程是：

```text
计算后面层梯度
bucket A ready -> all-reduce bucket A
继续计算前面层梯度
bucket B ready -> all-reduce bucket B
...
```

bucket size 很关键。

bucket 太小：

1. 通信启动次数多。
2. latency overhead 高。
3. 带宽利用率差。

bucket 太大：

1. 要等更久才能 ready。
2. overlap 机会减少。
3. 最后可能有长尾通信。

所以 DDP bucket size 是性能调优点。

### 四、ZeRO/FSDP 中的 overlap

ZeRO/FSDP 的通信更复杂。

主要有两类。

第一，forward/backward 前的 parameter all-gather。

参数平时是分片的，计算某个模块前要 all-gather 完整参数。

第二，backward 后的 gradient reduce-scatter。

梯度计算完后，不保存完整梯度，而是 reduce-scatter 成分片。

Overlap 思路包括：

1. forward prefetch：提前 all-gather 下一层参数。
2. backward prefetch：backward 时提前准备下一个需要的参数。
3. gradient reduce-scatter overlap：某个模块梯度就绪后立即 reduce-scatter，同时继续计算其他模块 backward。
4. optimizer step overlap：某些分片梯度 ready 后提前更新或准备更新。

FSDP 中常见配置会涉及 forward prefetch、backward prefetch、limit all-gathers 等。

这些配置影响显存和吞吐。

Prefetch 太激进：

1. 通信更早开始。
2. overlap 可能更好。
3. 但临时参数显存更高。

Prefetch 太保守：

1. 显存更省。
2. 但计算可能等参数 all-gather。

这就是显存和速度的权衡。

### 五、Tensor Parallel 中的 overlap

TP 中通信发生在层内部。

例如 Row Parallel Linear 后需要 all-reduce partial output。

这类通信通常在下一步计算前就需要结果，所以不一定容易隐藏。

但仍有一些优化空间。

第一，使用 fused kernel 减少通信前后的 overhead。

第二，把 reduce-scatter 和后续计算组合，避免完整 all-reduce。

第三，利用异步 collective，让通信尽早启动。

第四，在 attention 或 MLP 内部重排计算，让部分通信和其他 head 或其他 micro-batch 计算重叠。

第五，结合 sequence parallel，把某些 all-reduce 替换为 reduce-scatter/all-gather 组合。

TP 通信频率高，所以它更依赖高速互联。

如果 TP 放在慢速跨节点网络上，即使 overlap 做得好，也可能很难救。

### 六、Pipeline Parallel 中的 overlap

PP 本身就是一种让不同 stage 同时工作的调度。

它天然包含计算和通信重叠的思想。

例如 Stage 0 处理 micro-batch 2 时，Stage 1 可以处理 micro-batch 1。

同时 Stage 0 和 Stage 1 之间传输 activation。

PP overlap 的关键是：

1. micro-batch 数足够。
2. stage 负载均衡。
3. send/recv 不阻塞过久。
4. 1F1B 调度减少空闲。
5. stage 边界通信尽量隐藏在计算中。

如果某个 stage 很慢，其他 stage 会等待。

如果 micro-batch 太少，pipeline bubble 大。

如果 stage 间通信太慢，激活传输会成为瓶颈。

所以 PP 性能调优常看：

1. pipeline bubble ratio。
2. per-stage step time。
3. activation send/recv time。
4. micro-batch 计算时间。

### 七、Context Parallel 中的 overlap

CP 和 ring attention 中，通信和计算重叠尤其重要。

例如 K/V block 在 rank 之间循环传递。

理想情况下，每个 rank 在计算当前 K/V block attention 的同时，预取下一个 K/V block。

如果做得好，K/V 传输可以被 attention 计算部分隐藏。

如果做得不好，每个 block 都要等通信完成，长上下文性能会很差。

CP overlap 的难点是：

1. attention softmax 有全局依赖。
2. blockwise 计算要保持数值稳定。
3. 通信和计算粒度要匹配。
4. causal mask 影响哪些 block 需要计算。
5. 负载不均会造成等待。

因此 CP 的性能高度依赖 kernel 和调度实现。

### 八、异步通信和 CUDA Stream

通信与计算重叠通常依赖异步执行。

GPU 上可以有不同 CUDA stream。

计算 kernel 在一个 stream 上执行。

通信 kernel 或 NCCL collective 可以在另一个 stream 上执行。

只要依赖关系允许，它们可以重叠。

但要注意：

1. 如果后续计算马上需要通信结果，就必须 wait。
2. 如果 stream 同步过早，会破坏 overlap。
3. 如果通信和计算争用同一硬件资源，重叠效果有限。
4. 如果 tensor 生命周期管理错误，可能出现数据 race。

框架通常会帮用户管理大部分细节。

但性能调优时，要能读懂 profiler 中的 compute stream、NCCL stream 和同步点。

### 九、Overlap 的前提：依赖关系允许

不是所有通信都能重叠。

能不能 overlap，取决于依赖关系。

例如 DDP backward 中，某个 bucket 梯度 ready 后，它的 all-reduce 可以和其他层 backward 重叠。

因为其他层 backward 不依赖这个 bucket 的 all-reduce 结果。

但 optimizer step 必须等所有梯度同步完成。

所以最后一个 bucket 的通信通常难以隐藏，会形成 tail。

FSDP forward 中，如果当前层参数还没 all-gather 完，当前层计算就不能开始。

但下一层参数的 all-gather 可以在当前层计算时提前进行。

所以 overlap 的关键是提前启动未来需要的通信。

这叫 prefetch。

### 十、如何诊断 overlap 是否有效

可以用 profiler 查看时间线。

常见工具包括：

1. PyTorch Profiler。
2. Nsight Systems。
3. Nsight Compute。
4. DeepSpeed Flops Profiler。
5. Megatron 或训练框架自带日志。

有效 overlap 的时间线特点：

1. NCCL 通信和 compute kernel 在时间上重叠。
2. GPU idle gap 少。
3. step time 小于 compute time + communication time 的简单相加。
4. backward 中 bucket all-reduce 逐步启动，而不是集中在最后。

无效 overlap 的表现：

1. backward 后出现一大段纯 NCCL 时间。
2. forward 中计算等待 all-gather。
3. NCCL 之间有大量空洞。
4. 某个 rank 明显慢，其他 rank 等待。
5. GPU utilization 周期性下降。

### 十一、通信重叠的调优手段

常见调优手段包括：

1. 调整 DDP bucket size。
2. 开启或调整 FSDP prefetch。
3. 调整 FSDP wrap 粒度。
4. 减少不必要的同步点。
5. 使用 `no_sync()` 做 gradient accumulation。
6. 把 TP group 放在高速互联内。
7. 调整 PP micro-batch 数。
8. 平衡 pipeline stage 负载。
9. 使用 fused kernels。
10. 使用更合适的 collective 算法。
11. 优化 dataloader，避免数据输入打断训练。

其中很多调优都有副作用。

例如增大 bucket size 可以提高带宽利用率，但可能降低 overlap。

开启 aggressive prefetch 可以提高 overlap，但增加显存。

增加 micro-batch 可以降低 bubble，但可能让单个 micro-batch 太小，计算不饱和。

所以性能优化不是单开关问题，而是权衡。

### 十二、Overlap 和显存的权衡

通信提前意味着数据要更早存在显存里。

例如 FSDP forward prefetch 下一层参数。

这可以隐藏 all-gather，但会让当前层参数和下一层参数同时占用显存。

所以 overlap 常常增加峰值显存。

类似地，PP 中更多 micro-batch 可以减少 bubble，但可能增加 activation 保存。

因此训练配置经常要在三者之间平衡：

```text
显存
吞吐
通信隐藏程度
```

如果显存很紧，可能不得不牺牲 overlap。

如果显存余量充足，可以更激进 prefetch 来提升吞吐。

### 十三、Overlap 和 Straggler

同步训练中，最慢 rank 决定 step 时间。

如果某个 rank 数据加载慢、计算慢或通信慢，其他 rank 会等待。

这会破坏 overlap。

常见 straggler 原因：

1. 数据 shard 不均衡。
2. 某个 GPU 性能异常。
3. 网络链路拥塞。
4. pipeline stage 负载不均。
5. MoE expert 负载不均。
6. checkpoint 或日志 I/O 干扰。

所以通信计算重叠不只是 kernel 层问题，也和系统整体均衡有关。

### 十四、代码示例：DDP 梯度累积中的 no_sync

下面是一个简化示例，说明如何避免每个 micro-batch 都同步梯度。

```python
for step, batch in enumerate(dataloader):
    is_accumulating = (step + 1) % grad_accum_steps != 0

    if is_accumulating:
        context = model.no_sync()
    else:
        context = nullcontext()

    with context:
        outputs = model(**batch)
        loss = outputs.loss / grad_accum_steps
        loss.backward()

    if not is_accumulating:
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)
```

这里的关键是：

1. 中间累积步不做 DDP all-reduce。
2. 最后一个累积步再同步梯度。
3. 减少不必要通信。

这不是 overlap 本身，但它是通信优化的重要基础。

如果每个 micro-batch 都同步，通信量会明显增加，也更难获得好的 overlap。

### 十五、面试回答模板

如果面试官问：

> 分布式训练中通信很慢，怎么优化？通信和计算重叠怎么做？

可以这样回答：

我会先区分通信来源。DDP 主要是梯度 all-reduce，FSDP/ZeRO 有参数 all-gather 和梯度 reduce-scatter，TP 有层内 all-reduce/all-gather，PP 有 stage 间 activation 和 gradient send/recv。优化时首先用 profiler 看 NCCL 时间、GPU idle、通信是否集中在 step 尾部，以及不同 rank 是否有 straggler。

通信计算重叠的核心是让通信尽早启动，并在后续计算期间完成。DDP 中通过 gradient bucket，在某些梯度 ready 后立即 all-reduce，同时继续 backward。FSDP 中可以 prefetch 下一层参数，在当前层计算时做 all-gather；backward 时梯度 ready 后立即 reduce-scatter。PP 中通过 micro-batch 和 1F1B 调度让不同 stage 同时工作。CP/ring attention 中可以边传 K/V block 边计算 attention。

具体调优包括调整 DDP bucket size、FSDP wrap 和 prefetch、减少不必要同步点、gradient accumulation 中使用 no_sync、把 TP 放在高速互联内、平衡 pipeline stage、增加合适 micro-batch、使用 fused kernels 和优化数据加载。但 overlap 也有代价，例如 prefetch 会增加峰值显存，micro-batch 太多会影响计算饱和，所以最终要根据 tokens/s、MFU、通信占比、显存余量和 loss 稳定性综合选择。

### 十六、常见追问与回答

问题 1：DDP 为什么能在 backward 中 overlap 通信？

回答：反向传播逐层产生梯度。某个 bucket 的梯度都 ready 后就可以 all-reduce，而其他层 backward 还在继续，因此通信可以和后续梯度计算重叠。

问题 2：bucket size 越小越好吗？

回答：不是。小 bucket 可以更早启动通信，但通信启动次数多、带宽利用率差。大 bucket 带宽利用好，但 ready 晚、overlap 少。需要折中。

问题 3：FSDP prefetch 为什么会增加显存？

回答：提前 all-gather 下一层参数时，当前层和下一层完整参数可能同时存在，峰值显存会上升。

问题 4：为什么 TP 通信更难隐藏？

回答：TP 通信发生在层内部，很多结果马上被下一步计算需要，依赖关系紧，通信频率高，所以比 DDP bucket 更难完全隐藏。

问题 5：如何判断通信没有被隐藏？

回答：profiler 中会看到计算后有长时间纯 NCCL 段、GPU idle gap、step 尾部长尾通信，或者 step time 接近 compute time 加 communication time。

问题 6：通信优化是否只看 NCCL？

回答：不是。还要看数据加载、rank 负载均衡、pipeline stage 时间、checkpoint I/O、GPU 拓扑和同步点。系统任何 straggler 都会破坏 overlap。

### 十七、常见误区

1. 误区：通信和计算一定能完全重叠。
   纠正：只有依赖关系允许且计算足够覆盖通信时才可能重叠，很多通信只能部分隐藏。

2. 误区：NCCL 时间高就一定是网络差。
   纠正：也可能是 bucket 策略、同步点、rank 不均衡、TP 跨节点或通信无法 overlap。

3. 误区：prefetch 越多越好。
   纠正：prefetch 会增加峰值显存，显存紧张时可能导致 OOM。

4. 误区：bucket 越大越好。
   纠正：bucket 太大 ready 晚，会减少 overlap。

5. 误区：只优化通信就能提升吞吐。
   纠正：dataloader、kernel 效率、stage balance、activation checkpointing 都会影响吞吐。

6. 误区：gradient accumulation 中每个 micro-batch 同步也没关系。
   纠正：会增加不必要通信，应使用 `no_sync()` 或框架等价机制。

3. 解释 FSDP forward prefetch 的收益和显存代价。
4. 为什么 TP 通信通常比 DDP 通信更难隐藏？

### 本讲总结

本讲最重要的结论：

1. 分布式训练中的通信来自 DDP、FSDP/ZeRO、TP、PP、CP 等多个维度。
2. 通信与计算重叠的核心是让通信尽早启动，并在后续计算期间完成。
3. DDP 通过 gradient bucket 在 backward 中 overlap all-reduce。
4. FSDP/ZeRO 通过 parameter prefetch 和 gradient reduce-scatter overlap 降低等待。
5. TP 通信发生在层内部，依赖紧、频率高，更依赖高速互联和 kernel 优化。
6. PP 通过 micro-batch 和调度让 stage 并行工作，本质上也在隐藏等待。
7. overlap 优化要结合 profiler、硬件拓扑、bucket/prefetch 策略、stage balance、显存余量和整体吞吐判断。

## 第 49 讲：训练吞吐指标与性能分析

### 本讲目标

学完这一讲，你应该能够回答：

1. 大模型训练中常见吞吐指标有哪些。
2. tokens/s、samples/s、step time、TFLOPs、MFU 分别代表什么。
3. 为什么只看 GPU utilization 不够。
4. 如何定位训练慢是计算、通信、显存、数据加载还是调度问题。
5. 如何用 profiler 和日志建立性能分析闭环。
6. 面试中如何回答“训练吞吐低，你怎么排查”。

前面几讲讲了各种并行和通信优化。

但在真实工程中，你不能只说“用了 TP、PP、FSDP”。

你必须能判断：

1. 训练到底快不快。
2. GPU 有没有被充分利用。
3. 瓶颈在哪里。
4. 调整配置后是否真的变好。

这就需要吞吐指标和性能分析。

### 一、为什么性能指标很重要

大模型训练成本极高。

如果一个训练任务使用 1024 张 GPU 跑 30 天，吞吐提升 10% 就可能节省大量成本。

性能指标的作用包括：

1. 评估训练系统效率。
2. 比较不同并行配置。
3. 判断优化是否有效。
4. 发现通信、数据、kernel、显存瓶颈。
5. 估算训练完成时间和成本。
6. 支撑 scaling 和预算决策。

没有指标，性能优化就是猜。

有了指标，才能建立闭环：

```text
观测 -> 定位 -> 修改 -> 对比 -> 复盘
```

### 二、Step Time

Step time 是最基础的指标。

它表示一次 optimizer step 花了多长时间。

例如：

```text
step_time = 2.5 seconds
```

如果使用 gradient accumulation，一个 optimizer step 可能包含多个 micro step。

这时要区分：

1. micro step time。
2. optimizer step time。
3. iteration time。

step time 的优点是简单直观。

缺点是不能直接比较不同 batch size、sequence length、模型大小的训练。

例如一个模型 step time 2 秒，另一个 4 秒，不代表前者更高效。

因为后者可能每步处理更多 tokens。

所以 step time 必须和 tokens/s、FLOPs 一起看。

### 三、Tokens per Second

大模型训练最常用吞吐指标是 tokens/s。

它表示每秒处理多少训练 token。

计算公式：

```math
N_{\mathrm{tok}} = B_{\mathrm{global}} \cdot L
```

```math
R_{\mathrm{tok}} = \frac{N_{\mathrm{tok}}}{T_{\mathrm{step}}}
```

其中 `B_global` 是一次 optimizer step 对应的 global batch size，`L` 是 sequence length，`N_tok` 是每个 step 处理的 token 数，`T_step` 是 step time，`R_tok` 是 global tokens/s。

如果要比较不同 GPU 数下的效率，还可以看每卡平均吞吐：

```math
R_{\mathrm{gpu}} = \frac{R_{\mathrm{tok}}}{G}
```

其中 `G` 是 GPU 数。

如果：

```text
global_batch_size = 1024
sequence_length = 2048
step_time = 2 seconds
```

那么：

```text
tokens_per_step = 1024 * 2048 = 2,097,152
tokens/s = 2,097,152 / 2 ≈ 1.05M tokens/s
```

tokens/s 可以分为：

1. global tokens/s：整个集群总吞吐。
2. per-GPU tokens/s：每张 GPU 平均吞吐。

per-GPU tokens/s 便于比较不同规模集群效率。

例如：

```text
global tokens/s = 1,000,000
num_gpus = 100
per GPU tokens/s = 10,000
```

但 tokens/s 也有局限。

不同模型大小不同，处理同样 token 的计算量不同。

所以还要看 FLOPs 和 MFU。

### 四、Samples per Second 和 Tokens per Second 的区别

samples/s 表示每秒处理多少样本。

在图像或固定长度任务中，samples/s 很常用。

但在语言模型中，不同样本长度可能差异很大。

一个样本可能 128 tokens，另一个可能 4096 tokens。

所以 samples/s 不如 tokens/s 稳定。

如果使用 packing，把多个短文本拼成固定长度 sequence，那么 samples 的概念更模糊。

因此 LLM 训练更常用 tokens/s。

但在 SFT 或偏好训练中，samples/s 仍有价值，因为样本可能代表一个对话或一个 preference pair。

面试中可以说：

> 预训练更关注 tokens/s，监督或偏好训练可以同时看 samples/s 和 tokens/s。

### 五、FLOPs 与训练计算量

FLOPs 表示浮点运算次数。

大模型训练中常估算每个 token 需要多少 FLOPs。

一个常见粗略估算是：

```math
C_{\mathrm{train}} \approx 6ND
```

这个 `6N` 经验来自 forward 和 backward 的计算量近似。

它是粗略估算，不包括所有细节，例如 attention 的 `seq_len^2` 部分、embedding、激活函数、通信、重计算等。

但在 scaling law 和预算估算中很常用。

其中 `N` 是模型参数量，`D` 是训练 token 数，`C_train` 是训练总 FLOPs。

这个估算默认 dense decoder-only Transformer，并且把 forward、backward 和参数梯度相关计算合并成一个教学口径。对于 MoE、长序列 attention、activation checkpointing、embedding、optimizer step 和不同 FLOPs 统计口径，需要单独说明边界。

例如 7B 模型训练 1T tokens：

```text
6 * 7e9 * 1e12 = 4.2e22 FLOPs
```

FLOPs 帮助回答：

1. 训练总计算量多少。
2. 需要多少 GPU 天。
3. 硬件利用率如何。
4. 不同模型训练成本如何比较。

### 六、TFLOPs/s 与 GPU 理论峰值

TFLOPs/s 表示每秒万亿次浮点运算。

硬件厂商会给 GPU 理论峰值，例如某 GPU 在 BF16 Tensor Core 下峰值多少 TFLOPs。

但训练中实际达到的通常低于理论峰值。

原因包括：

1. 内存访问。
2. kernel launch overhead。
3. 通信等待。
4. 小矩阵效率低。
5. 数据加载。
6. pipeline bubble。
7. activation checkpointing 重算。
8. 非矩阵操作。

所以理论峰值不是实际速度。

实际性能要通过 profiler 或估算得到。

### 七、MFU：Model FLOPs Utilization

MFU 是大模型训练中非常重要的指标。

它表示模型实际有效计算占硬件理论峰值的比例。

粗略公式：

```math
\mathrm{MFU} = \frac{F_{\mathrm{model}}}{F_{\mathrm{peak}}}
```

其中 `F_model` 是按模型有效 FLOPs 口径估算的每秒 FLOPs，`F_peak` 是集群在对应精度下的理论峰值 FLOPs/s。

如果用 `6N` 估算每 token 训练 FLOPs：

```math
F_{\mathrm{model}} \approx 6N \cdot R_{\mathrm{tok}}
```

再除以集群理论峰值：

```math
F_{\mathrm{peak}} = G \cdot F_{\mathrm{gpu}}
```

其中 `F_gpu` 是单张 GPU 在对应 dtype 下的理论峰值 FLOPs/s，`G` 是 GPU 数。

MFU 越高，说明训练系统越有效地利用了硬件。

但 MFU 也要谨慎解释。

它依赖 FLOPs 估算方法。

不同论文、框架、模型结构可能计算口径不同。

Attention 的长序列部分、MoE activated parameters、activation checkpointing 的重算，都会影响口径。

所以比较 MFU 时要确认公式一致。

### 八、HFU 与 MFU 的区别

有时还会看到 HFU：Hardware FLOPs Utilization。

HFU 可能把重计算等实际硬件执行的 FLOPs 也算进去。

MFU 更关注模型有效训练 FLOPs。

例如使用 activation checkpointing 时，硬件实际做了更多计算，因为 backward 期间重算 forward。

HFU 可能变高，但 MFU 不一定变高。

因为重算不是额外有效模型进展，只是为了省显存。

面试中可以这样说：

> MFU 更关注对模型训练有用的 FLOPs 利用率，HFU 更接近硬件实际忙碌程度。两者都要结合场景解释。

### 九、GPU Utilization 为什么不够

很多人只看 `nvidia-smi` 里的 GPU utilization。

这不够。

因为 GPU utilization 高，不代表训练效率高。

可能情况包括：

1. GPU 在做重计算，MFU 不高。
2. GPU 忙于低效小 kernel。
3. GPU 被内存带宽限制。
4. GPU 等通信但 utilization 仍显示不直观。
5. 不同 rank 利用率不均。

GPU utilization 低也不一定说明模型代码慢，可能是：

1. dataloader 慢。
2. CPU 预处理慢。
3. 通信等待。
4. checkpoint I/O。
5. pipeline bubble。

所以要结合：

1. tokens/s。
2. MFU。
3. step time breakdown。
4. memory bandwidth。
5. NCCL time。
6. kernel timeline。
7. dataloader time。

### 十、Step Time Breakdown

性能分析的核心是拆解 step time。

一个训练 step 可以拆成：

1. data loading。
2. host-to-device copy。
3. forward compute。
4. loss compute。
5. backward compute。
6. gradient communication。
7. optimizer step。
8. checkpoint/logging。
9. synchronization wait。

如果不知道时间花在哪，就很难优化。

例如：

```text
step time = 2.0s
forward = 0.5s
backward = 0.9s
communication = 0.4s
dataloader = 0.1s
optimizer = 0.1s
```

这说明通信是重要瓶颈。

另一个例子：

```text
step time = 2.0s
dataloader = 0.8s
compute = 1.0s
communication = 0.2s
```

这时优化 NCCL 没用，应该先优化数据加载。

### 十一、常见瓶颈类型

瓶颈一：计算瓶颈。

表现：矩阵乘占比高，GPU 计算饱和，通信和数据不是主要问题。

优化方向：使用更高效 kernel、混合精度、FlashAttention、fused ops、合适 batch size。

瓶颈二：通信瓶颈。

表现：NCCL 时间高，GPU 等待通信，多卡扩展效率差。

优化方向：通信计算重叠、调整 TP/DP/PP、bucket、prefetch、硬件拓扑。

瓶颈三：内存带宽瓶颈。

表现：GPU utilization 可能高，但 TFLOPs 不高，大量 elementwise、norm、数据搬运。

优化方向：fused kernel、减少 layout 转换、优化 tensor shape。

瓶颈四：显存瓶颈。

表现：OOM 或 batch/seq_len 上不去。

优化方向：FSDP/ZeRO、activation checkpointing、SP/CP、offload、mixed precision。

瓶颈五：数据加载瓶颈。

表现：GPU 有空洞，dataloader time 高，CPU 忙。

优化方向：预处理、缓存、并行 dataloader、数据格式、异步拷贝。

瓶颈六：pipeline bubble。

表现：PP stage 空闲多，某些 stage 等待。

优化方向：增加 micro-batch、平衡 stage、使用 1F1B/interleaving。

### 十二、Profiler 工具

常见工具包括：

1. PyTorch Profiler。
2. TensorBoard trace viewer。
3. Nsight Systems。
4. Nsight Compute。
5. NCCL debug logs。
6. DeepSpeed profiler。
7. Megatron-LM timers。
8. 自定义 timers。

PyTorch Profiler 适合看框架级 operator、CPU/GPU timeline。

Nsight Systems 适合看 CUDA kernel、NCCL、stream、CPU 调度。

Nsight Compute 适合深入分析单个 kernel。

NCCL logs 适合排查通信拓扑和 collective 问题。

大规模训练中，不能对所有 step 全量 profile，开销太大。

常见做法是：

1. 只 profile 少量 warmup 后的 step。
2. 选择代表性 rank。
3. 同时收集全局 timers。
4. 异常时再深入到 Nsight。

### 十三、性能分析闭环

性能优化要形成闭环。

第一步，建立 baseline。

记录当前配置、tokens/s、MFU、step time、显存、loss。

第二步，拆解瓶颈。

用 timers 和 profiler 看时间花在哪。

第三步，提出假设。

例如“TP 跨节点导致通信慢”或“bucket 太大导致 overlap 不好”。

第四步，做单变量实验。

一次只改一个关键配置，避免无法归因。

第五步，对比指标。

看 tokens/s、MFU、显存、loss 是否改善。

第六步，记录结论。

把配置和结果写入实验日志，避免重复踩坑。

### 十四、代码示例：简单吞吐统计

下面先给一个完全自包含的指标计算 demo。它不依赖真实模型，适合用来检查自己是否理解 tokens/s、MFU、扩展效率和训练成本之间的关系。

```python
def format_large(value):
    units = ["", "K", "M", "B", "T", "P"]
    value = float(value)
    unit_id = 0
    while abs(value) >= 1000 and unit_id < len(units) - 1:
        value /= 1000
        unit_id += 1
    return f"{value:.2f}{units[unit_id]}"


def training_metrics(
    num_params,
    global_batch_size,
    sequence_length,
    step_time_s,
    num_gpus,
    peak_tflops_per_gpu,
    baseline_tokens_per_s=None,
):
    tokens_per_step = global_batch_size * sequence_length
    tokens_per_s = tokens_per_step / step_time_s
    per_gpu_tokens_per_s = tokens_per_s / num_gpus
    model_flops_per_s = 6 * num_params * tokens_per_s
    peak_flops_per_s = num_gpus * peak_tflops_per_gpu * 1e12
    mfu = model_flops_per_s / peak_flops_per_s

    result = {
        "tokens_per_step": tokens_per_step,
        "tokens_per_s": tokens_per_s,
        "per_gpu_tokens_per_s": per_gpu_tokens_per_s,
        "model_flops_per_s": model_flops_per_s,
        "mfu": mfu,
    }

    if baseline_tokens_per_s is not None:
        actual_speedup = tokens_per_s / baseline_tokens_per_s
        result["scaling_efficiency"] = actual_speedup / num_gpus

    return result


metrics = training_metrics(
    num_params=7_000_000_000,
    global_batch_size=1024,
    sequence_length=2048,
    step_time_s=20.0,
    num_gpus=64,
    peak_tflops_per_gpu=312,
    baseline_tokens_per_s=2_000,
)

print("tokens/step:", format_large(metrics["tokens_per_step"]))
print("global tokens/s:", format_large(metrics["tokens_per_s"]))
print("per-GPU tokens/s:", format_large(metrics["per_gpu_tokens_per_s"]))
print("model FLOPs/s:", format_large(metrics["model_flops_per_s"]))
print("MFU:", round(metrics["mfu"], 3))
print("scaling efficiency:", round(metrics["scaling_efficiency"], 3))
```

典型输出是：

```text
tokens/step: 2.10M
global tokens/s: 104.86K
per-GPU tokens/s: 1.64K
model FLOPs/s: 4.40P
MFU: 0.221
scaling efficiency: 0.819
```

这个 demo 里的 `peak_tflops_per_gpu=312` 只是教学假设，真实计算必须使用对应 GPU、dtype、稀疏/稠密模式和厂商给出的峰值口径。MFU 也不是越高越好；如果为了提高 MFU 改变了 batch、精度或训练 recipe，导致 loss 变差，就不是有效优化。

真实训练循环中可以在每个 step 后记录 tokens/s。下面是一个简化结构：

```python
import time


def train_loop(model, dataloader, optimizer, seq_len, global_batch_size):
    for step, batch in enumerate(dataloader):
        start = time.perf_counter()

        outputs = model(**batch)
        loss = outputs.loss
        loss.backward()
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)

        elapsed = time.perf_counter() - start
        tokens_per_step = global_batch_size * seq_len
        tokens_per_second = tokens_per_step / elapsed

        if step % 10 == 0:
            print(
                f"step={step} loss={loss.item():.4f} "
                f"step_time={elapsed:.3f}s tokens/s={tokens_per_second:.0f}"
            )
```

真实分布式训练中要注意：

1. 统计 global batch，而不是单卡 batch。
2. CUDA 操作异步，需要同步后计时。
3. 多 rank 取最大 step time，而不是只看 rank 0。
4. 跳过 warmup step。
5. 区分训练 step 和 eval step。

更严格计时需要：

```python
torch.cuda.synchronize()
start = time.perf_counter()
# training step
torch.cuda.synchronize()
elapsed = time.perf_counter() - start
```

否则 CPU 计时可能不准确。

### 十五、扩展效率

当 GPU 数增加时，希望吞吐近似线性增长。

扩展效率可以定义为：

```math
E_{\mathrm{scale}} = \frac{S_{\mathrm{actual}}}{S_{\mathrm{ideal}}}
```

例如 8 张 GPU tokens/s 是单卡的 6.4 倍：

```math
E_{\mathrm{scale}} = \frac{6.4}{8} = 0.8
```

扩展效率下降的原因包括：

1. 通信增加。
2. global batch 变化。
3. 小 batch 计算不饱和。
4. straggler。
5. dataloader 不扩展。
6. pipeline bubble。

评估大规模训练配置时，扩展效率非常重要。

一个配置在 8 卡上很好，不代表 1024 卡上也好。

### 十六、成本指标

训练性能最终还要转化为成本。

常见成本指标：

1. GPU hours。
2. GPU days。
3. tokens per dollar。
4. cost per 1B tokens。
5. time to target loss。
6. energy consumption。

例如：

```math
C_{\mathrm{gpu\_day}} = G \cdot D_{\mathrm{day}}
```

如果 512 张 GPU 训练 20 天：

```text
GPU days = 512 * 20 = 10240 GPU-days
```

如果吞吐提升 10%，达到同样 token 数所需时间会减少。

这直接影响成本。

面试中如果能把性能指标和成本联系起来，会显得更工程化。

### 十七、面试回答模板

如果面试官问：

> 大模型训练吞吐低，你怎么分析？

可以这样回答：

我会先建立 baseline，记录 step time、global tokens/s、per-GPU tokens/s、MFU、显存占用、GPU utilization、NCCL 时间和 dataloader 时间。只看 GPU utilization 不够，因为 GPU 忙不代表有效训练 FLOPs 高。

然后我会拆解 step time，看时间主要花在 forward/backward compute、通信、optimizer、数据加载、pipeline bubble 还是 checkpoint/logging。通信瓶颈会看 NCCL timeline、all-reduce/reduce-scatter/all-gather 是否集中在尾部，是否有 overlap。计算瓶颈会看 matmul、attention、norm、elementwise kernel 占比。数据瓶颈会看 GPU idle gap 和 dataloader time。Pipeline 瓶颈会看每个 stage 的时间和 bubble。

定位后再做单变量优化。例如通信慢就调 bucket、prefetch、TP/DP 映射和 overlap；计算慢就用 fused kernel、FlashAttention、混合精度；显存卡住就用 FSDP、activation checkpointing 或 SP/CP；数据慢就优化数据格式、缓存和 dataloader。最后用 tokens/s、MFU、scaling efficiency、显存和 loss 曲线验证优化是否真的有效。

### 十八、常见追问与回答

问题 1：tokens/s 和 samples/s 哪个更适合 LLM 预训练？

回答：tokens/s 更适合，因为语言样本长度不固定，训练成本主要和 token 数相关。SFT 或偏好训练中可以同时看 samples/s。

问题 2：MFU 为什么比 GPU utilization 更有意义？

回答：MFU 衡量模型有效 FLOPs 占硬件峰值的比例，更接近训练效率。GPU utilization 只表示 GPU 忙，不说明是否在做高效有用计算。

问题 3：activation checkpointing 会如何影响 MFU？

回答：它会增加重算，硬件可能更忙，但有效模型 FLOPs 不一定增加，所以 HFU 可能变高，MFU 未必提高。

问题 4：怎么判断是 dataloader 瓶颈？

回答：profiler 或 timers 显示 GPU 等数据，dataloader time 高，GPU timeline 有空洞。增加 dataloader workers、缓存或预处理后 tokens/s 提升，也能验证。

问题 5：多卡扩展效率低怎么排查？

回答：看通信时间、NCCL overlap、rank straggler、global batch 变化、dataloader 是否扩展、pipeline bubble 和硬件拓扑。逐步从小规模到大规模比较。

问题 6：优化后 tokens/s 提升但 loss 变差，算成功吗？

回答：不一定。性能优化不能破坏训练语义。比如 global batch 改变、LR 没调整、精度变化导致 loss 变差，都需要重新评估。

### 十九、常见误区

1. 误区：GPU utilization 高就说明训练效率高。
   纠正：还要看 tokens/s、MFU、kernel 效率和是否在做有效计算。

2. 误区：step time 越低模型训练越高效。
   纠正：还要看每步处理多少 tokens、模型大小和 FLOPs。

3. 误区：tokens/s 可以跨任意模型直接比较。
   纠正：不同模型参数量和 sequence length 不同，每 token 计算量不同，需要结合 MFU。

4. 误区：profile 一次就能得出永久结论。
   纠正：不同 batch、seq_len、并行配置、训练阶段都可能改变瓶颈。

5. 误区：只优化最大耗时项就一定最好。
   纠正：要考虑依赖关系、overlap、显存、训练稳定性和实现复杂度。

6. 误区：吞吐提升一定等于成本下降。
   纠正：如果需要更多 GPU、更高失败率或 loss 变差，综合成本可能不降。

### 二十、小练习

1. 给定 global batch size、sequence length 和 step time，计算 tokens/s。
2. 用 `6ND` 粗略估算一个 7B 模型训练 1T token 的 FLOPs。
3. 解释 MFU 和 GPU utilization 的区别。
4. 列出一个训练 step 的时间拆解项。
5. 设计一个性能分析 dashboard，至少包含 10 个指标。
6. 用 2 分钟回答：“训练吞吐比预期低 30%，你怎么排查？”

### 本讲总结

本讲最重要的结论：

1. 大模型训练性能不能只看 step time 或 GPU utilization，要综合 tokens/s、MFU、显存、通信和成本。
2. tokens/s 是 LLM 训练最常用吞吐指标，per-GPU tokens/s 便于比较效率。
3. MFU 衡量有效模型 FLOPs 利用率，比单纯 GPU utilization 更接近训练效率。
4. 性能分析要拆解 step time，定位计算、通信、数据加载、显存、pipeline bubble 或 I/O 瓶颈。
5. profiler 和 timers 是性能优化闭环的基础，优化必须做单变量对比和记录。
6. 多卡扩展要看 scaling efficiency，不同规模下瓶颈可能变化。
7. 面试中成熟的回答要把吞吐指标、瓶颈定位、优化手段和训练成本联系起来。

## 第 50 讲：大规模训练工程事故分析

### 本讲目标

学完这一讲，你应该能够回答：

1. 大规模训练常见工程事故有哪些类型。
2. 遇到 loss spike、NaN、hang、OOM、吞吐骤降、checkpoint 损坏时如何排查。
3. 如何区分数据问题、数值问题、通信问题、硬件问题和代码变更问题。
4. 如何设计监控、报警、自动恢复和回滚机制。
5. 训练事故后如何做复盘，避免重复发生。
6. 面试中如何回答“大模型训练到一半出事故，你怎么处理”。

这一讲是第四部分的收束。

前面我们讲了并行、显存、通信、吞吐。

但真实大模型训练一定会遇到事故。

大规模训练不是写好代码然后等模型变强。

它更像运行一个长周期、高成本、高并发的分布式系统。

任何一个环节出问题，都可能浪费大量算力。

### 一、为什么大规模训练事故很常见

大模型训练事故常见，是因为系统太复杂。

一个训练任务可能包含：

1. 数百到数千张 GPU。
2. 多机网络通信。
3. PB 级数据管道。
4. 分布式 checkpoint。
5. 混合精度。
6. 多维并行。
7. 自定义 kernel。
8. 长周期运行。
9. 动态学习率和数据配比。
10. 频繁评估和日志系统。

任何一个部分都有故障概率。

小模型训练中，一个 bug 可能只是浪费几分钟。

大模型训练中，一个 bug 可能浪费几千 GPU hours。

所以训练工程的核心不是“永远不出事故”，而是：

1. 尽早发现。
2. 快速定位。
3. 安全恢复。
4. 控制损失。
5. 复盘后防止复发。

### 二、事故类型总览

常见事故可以分为几类。

第一，数值事故。

例如：

1. loss spike。
2. NaN。
3. Inf。
4. 梯度爆炸。
5. FP16 overflow。

第二，数据事故。

例如：

1. 数据 shard 损坏。
2. 样本格式错误。
3. tokenizer 版本不一致。
4. label shift 错误。
5. benchmark 泄漏。
6. 数据配比配置错误。

第三，分布式事故。

例如：

1. NCCL hang。
2. rank timeout。
3. 某个 rank OOM。
4. process group 配置错误。
5. TP/PP/DP group mismatch。

第四，硬件和基础设施事故。

例如：

1. GPU XID error。
2. 节点宕机。
3. 网络抖动。
4. 文件系统不可用。
5. 存储写入失败。

第五，checkpoint 事故。

例如：

1. checkpoint 写到一半。
2. shard 缺失。
3. optimizer state 丢失。
4. 恢复后 loss 跳变。
5. rank mapping 错误。

第六，性能事故。

例如：

1. tokens/s 突然下降。
2. GPU utilization 下降。
3. dataloader 变慢。
4. 通信时间暴涨。
5. pipeline bubble 变大。

第七，代码和配置事故。

例如：

1. 改了 loss mask。
2. 改了数据路径。
3. 学习率配置错。
4. checkpoint resume 参数错。
5. dependency 版本变化。

### 三、事故处理的第一原则

遇到训练事故，第一原则是：

> 先保留现场，再决定是否继续训练。

不要看到异常就直接重启覆盖日志。

也不要在 NaN 后继续盲训。

应该先记录：

1. global step。
2. consumed tokens。
3. learning rate。
4. loss。
5. grad norm。
6. overflow count。
7. rank id。
8. data shard id。
9. checkpoint id。
10. 最近代码 commit。
11. 最近配置变更。

这些信息是后续定位根因的证据。

很多事故定位困难，不是因为问题复杂，而是因为现场没有保存。

### 四、Loss Spike 事故分析

loss spike 是最常见事故之一。

处理流程：

第一，判断 spike 类型。

是单 step 短暂 spike，还是持续升高？

是否自动恢复？

是否伴随 grad norm 暴涨？

第二，检查数据。

记录 spike step 对应 batch、shard、sample id。

检查是否有异常长文本、空样本、乱码、特殊 token、label 错位。

第三，检查数值。

看是否 FP16 overflow、grad norm 异常、activation max 异常。

第四，检查学习率。

是否刚好在 warmup、schedule 切换或 resume 后。

第五，检查分布式。

是否只有某些 rank 先 spike。

如果 spike 后恢复，可能只需标记和继续观察。

如果 spike 持续恶化，应从健康 checkpoint 回滚。

### 五、NaN 事故分析

NaN 比普通 spike 更严重。

因为 NaN 会传播，污染参数、梯度和 optimizer state。

处理流程：

1. 立即停止训练。
2. 标记当前 checkpoint 为 suspect 或 bad。
3. 从最近健康 checkpoint 恢复。
4. 用相同数据和 seed replay NaN 前若干 step。
5. 检查 loss、logits、activation、gradient、optimizer state 的 finite 状态。
6. 定位第一个出现 NaN/Inf 的层或操作。

常见根因：

1. learning rate 过高。
2. FP16 overflow。
3. attention logits 溢出。
4. softmax mask 错误。
5. 除以 0。
6. 数据 batch 异常。
7. optimizer state 已损坏。
8. 自定义 kernel bug。

不要从 NaN checkpoint 继续训练。

这通常只会浪费更多算力。

### 六、OOM 事故分析

OOM 是 out of memory。

大模型训练 OOM 可能来自：

1. batch size 增大。
2. sequence length 变长。
3. activation checkpointing 没生效。
4. FSDP prefetch 太激进。
5. 临时 all-gather buffer 变大。
6. dataloader 产生异常长样本。
7. memory fragmentation。
8. checkpoint 保存或评估阶段额外占显存。

排查步骤：

1. 看 OOM 发生在 forward、backward、optimizer step 还是 evaluation。
2. 比较 OOM 前后的 batch shape。
3. 检查 sequence length、padding、packing。
4. 查看 peak memory 和 reserved memory。
5. 检查最近是否改了 checkpointing、FSDP prefetch、micro-batch。
6. 检查是否有 tensor 被意外保存到 list 或日志。

修复方式包括：

1. 减小 micro-batch。
2. 开启或加强 activation checkpointing。
3. 调整 FSDP/ZeRO 策略。
4. 降低 prefetch 激进程度。
5. 使用 FlashAttention。
6. 限制最大 sequence length。
7. 修复内存泄漏。

### 七、Hang 和 Timeout 事故分析

分布式训练中，hang 很常见。

表现是训练卡住，没有继续 step。

常见原因：

1. 某个 rank OOM 或崩溃，其他 rank 等通信。
2. 某些 rank 进入 collective，另一些没有。
3. 数据 batch 数不一致。
4. PP send/recv 顺序不匹配。
5. process group 配置错。
6. NCCL 网络问题。
7. 某个 dataloader worker 卡住。
8. checkpoint I/O 卡住。

排查 hang 要看所有 rank，而不是只看 rank 0。

关键问题：

1. 哪个 rank 最后打印日志？
2. 哪个 collective 卡住？
3. 是否有 rank 提前退出？
4. 是否所有 rank batch 数一致？
5. 是否有节点硬件错误？

常用手段：

1. 设置合理 timeout。
2. 打印 per-rank heartbeat。
3. 记录每个 rank 当前阶段。
4. 开启 NCCL debug。
5. 对 dataloader 和 checkpoint I/O 设置超时。

### 八、吞吐骤降事故分析

训练没有失败，但 tokens/s 突然下降，也是一种事故。

可能原因：

1. 数据加载变慢。
2. 某个节点成为 straggler。
3. 网络拥塞。
4. checkpoint 或评估占用资源。
5. 数据 sequence length 分布变化。
6. pipeline stage 负载变化。
7. thermal throttling。
8. 存储系统抖动。

排查方法：

1. 对比 step time breakdown。
2. 查看 dataloader time。
3. 查看 NCCL time。
4. 查看 per-rank step time。
5. 查看 GPU clock 和温度。
6. 查看数据 shard 和 sequence length。
7. 查看是否发生 checkpoint/eval。

吞吐事故要和 loss 指标一起看。

如果吞吐下降同时 loss 异常，可能是数据或训练逻辑问题。

如果 loss 正常但吞吐下降，更多是系统性能问题。

### 九、Checkpoint 事故分析

checkpoint 是训练恢复的生命线。

常见事故：

1. checkpoint 写入中断。
2. manifest 缺失。
3. shard 数量不完整。
4. optimizer state 没保存。
5. scheduler state 错误。
6. dataloader state 丢失。
7. 恢复后 loss 跳变。
8. 并行配置变化导致加载失败。

预防措施：

1. 写临时目录，完成后原子 rename。
2. 保存 manifest 和 checksum。
3. 区分 healthy、suspect、bad checkpoint。
4. 定期做恢复演练。
5. 保存完整训练状态。
6. 记录并行配置和代码 commit。

如果恢复后 loss 跳变，优先排查：

1. optimizer state。
2. scheduler state。
3. GradScaler state。
4. data position。
5. RNG state。
6. config diff。
7. rank mapping。

### 十、数据事故分析

数据事故有时比模型事故更隐蔽。

例如：

1. 某批数据重复过多。
2. 某个 shard 是乱码。
3. tokenizer 版本变了。
4. 文本和 label 错位。
5. attention mask 错。
6. 数据配比配置错。
7. benchmark 泄漏进训练集。
8. 低质量数据突然增多。

数据事故的表现可能是：

1. loss spike。
2. 某个 domain validation loss 变差。
3. 下游 benchmark 异常。
4. 模型输出风格变化。
5. 训练吞吐变化。

预防数据事故，需要：

1. 数据版本管理。
2. shard checksum。
3. 数据 schema 校验。
4. sample-level logging。
5. domain loss 监控。
6. contamination 检测。
7. 数据变更灰度。

### 十一、硬件事故分析

硬件事故不可避免。

常见包括：

1. GPU XID error。
2. ECC error。
3. 节点宕机。
4. 网络链路异常。
5. 磁盘或文件系统错误。
6. 电源或散热问题。

训练系统要假设硬件会坏。

这意味着：

1. checkpoint 必须可靠。
2. 训练要支持重启。
3. 作业调度要能替换坏节点。
4. 监控要能发现硬件异常。
5. 日志要能定位具体节点和 rank。

大规模训练中，硬件稳定性本身就是算法工程的一部分。

### 十二、监控和报警体系

好的训练系统应该监控：

训练指标：

1. train loss。
2. validation loss。
3. grad norm。
4. learning rate。
5. loss scale。
6. overflow count。

性能指标：

1. tokens/s。
2. step time。
3. MFU。
4. GPU utilization。
5. NCCL time。
6. dataloader time。

系统指标：

1. GPU memory。
2. GPU temperature。
3. GPU clock。
4. network throughput。
5. disk throughput。
6. node health。

分布式指标：

1. per-rank loss。
2. per-rank step time。
3. pipeline stage time。
4. rank heartbeat。

报警应该覆盖：

1. NaN/Inf。
2. loss spike。
3. grad norm 异常。
4. tokens/s 下降。
5. rank hang。
6. checkpoint 失败。
7. GPU error。

报警规则不能只靠一句“指标异常”。工程上通常会把当前值和最近健康窗口的基线比较。几个简单但有用的指标是：

```math
r_{\mathrm{loss}} = \frac{l_t}{\mathrm{median}(l_{t-k:t-1})}
```

```math
r_{\mathrm{tok}} = \frac{R_t}{\mathrm{median}(R_{t-k:t-1})}
```

```math
u_{\mathrm{mem}} = \frac{M_{\mathrm{used}}}{M_{\mathrm{limit}}}
```

```math
s_{\mathrm{rank}} = \frac{\max_i T_i}{\mathrm{median}_i(T_i)}
```

其中 `l_t` 是当前 loss，`R_t` 是当前 tokens/s，`M_used` 和 `M_limit` 是显存已用量和限制，`T_i` 是第 `i` 个 rank 的 step time。`r_loss` 明显大于 1 可能表示 loss spike，`r_tok` 明显小于 1 可能表示吞吐下降，`u_mem` 接近 1 表示 OOM 风险，`s_rank` 过高表示存在 straggler。

下面这个 demo 用一条当前训练记录和最近健康窗口做对比，自动给出事故标签。它只用标准库，适合把“事故分类”变成一个可以练习的最小脚本。

```python
import math
from statistics import median


def safe_median(values):
    clean = [v for v in values if v is not None and math.isfinite(v)]
    return median(clean) if clean else None


def classify_incident(record, history):
    issues = []

    loss = record["loss"]
    if not math.isfinite(loss):
        issues.append("nan_or_inf: stop training and rollback to a healthy checkpoint")
        return issues

    loss_base = safe_median(history["loss"])
    grad_base = safe_median(history["grad_norm"])
    tok_base = safe_median(history["tokens_per_s"])

    loss_ratio = loss / loss_base if loss_base else 1.0
    grad_ratio = record["grad_norm"] / grad_base if grad_base else 1.0
    tok_ratio = record["tokens_per_s"] / tok_base if tok_base else 1.0
    mem_util = record["memory_used_gib"] / record["memory_limit_gib"]

    rank_times = record["rank_step_ms"]
    rank_slowdown = max(rank_times) / median(rank_times)

    if loss_ratio > 1.25 and grad_ratio > 1.5:
        issues.append(f"loss_spike: loss_ratio={loss_ratio:.2f}, grad_ratio={grad_ratio:.2f}")
    if tok_ratio < 0.75:
        issues.append(f"throughput_drop: tokens/s ratio={tok_ratio:.2f}")
    if mem_util > 0.95:
        issues.append(f"oom_risk: memory utilization={mem_util:.2f}")
    if rank_slowdown > 1.25:
        issues.append(f"straggler: rank slowdown={rank_slowdown:.2f}")
    if not record["checkpoint_complete"] or not record["checkpoint_checksum_ok"]:
        issues.append("checkpoint_bad: isolate this checkpoint before resume")
    if record["missing_heartbeats"]:
        issues.append(f"hang_risk: missing heartbeats from ranks {record['missing_heartbeats']}")

    return issues or ["healthy: keep monitoring"]


history = {
    "loss": [1.82, 1.80, 1.79, 1.81, 1.80],
    "grad_norm": [0.72, 0.74, 0.71, 0.73, 0.72],
    "tokens_per_s": [101_000, 99_500, 100_200, 100_800, 99_900],
}

current = {
    "loss": 2.50,
    "grad_norm": 2.40,
    "tokens_per_s": 61_000,
    "memory_used_gib": 78.5,
    "memory_limit_gib": 80.0,
    "rank_step_ms": [2000, 2030, 2010, 3100],
    "checkpoint_complete": False,
    "checkpoint_checksum_ok": True,
    "missing_heartbeats": [7],
}

for issue in classify_incident(current, history):
    print(issue)
```

典型输出是：

```text
loss_spike: loss_ratio=1.39, grad_ratio=3.33
throughput_drop: tokens/s ratio=0.61
oom_risk: memory utilization=0.98
straggler: rank slowdown=1.53
checkpoint_bad: isolate this checkpoint before resume
hang_risk: missing heartbeats from ranks [7]
```

这个 demo 的价值不是阈值本身，而是分类思路：同一个 step 可以同时出现数值异常、性能异常、显存风险、rank straggler 和 checkpoint 风险。真实系统会用更复杂的滑动窗口、分位数、规则引擎和告警抑制，但面试回答至少要能说清“我如何把异常从日志变成可操作标签”。

### 十三、自动恢复策略

大规模训练不能所有事故都靠人手动处理。

常见自动恢复策略：

1. 进程失败后自动重启。
2. 从最近 healthy checkpoint 恢复。
3. 检测 NaN 后自动停止并回滚。
4. 节点故障后重新调度。
5. checkpoint 写入失败时重试。
6. 数据 shard 读取失败时跳过或切换副本。
7. 训练 hang 后触发 timeout 和诊断 dump。

但自动恢复也要谨慎。

不能无限自动重启同一个错误。

应该设置：

1. 最大重试次数。
2. 错误分类。
3. 自动降级策略。
4. 人工介入条件。
5. bad checkpoint 隔离。

否则系统可能在错误状态中反复浪费算力。

### 十四、事故复盘

事故后要做 postmortem。

复盘不是追责，而是建立系统记忆。

复盘应回答：

1. 发生了什么？
2. 什么时候开始？
3. 影响范围多大？
4. 发现方式是什么？
5. 根因是什么？
6. 为什么监控没有更早发现？
7. 为什么自动恢复没有避免损失？
8. 如何防止复发？

复盘产物包括：

1. 时间线。
2. 根因分析。
3. 影响评估。
4. 修复措施。
5. 预防措施。
6. owner 和 deadline。

成熟团队会把事故转化为测试、监控和流程改进。

### 十五、一个事故案例：恢复后 loss 跳变

假设训练从 checkpoint 恢复后，loss 从 1.8 跳到 2.4。

排查过程：

第一，检查模型权重是否正确。

确认 missing keys、unexpected keys、dtype、shape。

第二，检查 optimizer state。

如果 Adam moments 丢失，恢复后更新会异常。

第三，检查 scheduler。

学习率是否恢复到正确 step。

第四，检查 data state。

是否重复或跳过大量数据。

第五，检查 GradScaler。

FP16 loss scale 是否恢复。

第六，检查代码和配置 diff。

是否改了 loss mask、batch size、sequence length、数据配比。

第七，回放相同 batch。

比较恢复前后 forward loss 是否一致。

这个案例说明：checkpoint 恢复问题不能只看 model weights。

### 十六、面试回答模板

如果面试官问：

> 大模型训练到一半出事故，比如 NaN、hang 或吞吐骤降，你怎么处理？

可以这样回答：

我会先保留现场，而不是直接重启覆盖证据。记录 global step、consumed tokens、loss、grad norm、LR、data shard、rank、checkpoint、代码 commit 和最近配置变更。然后按事故类型分类处理。

如果是 NaN，我会立即停止训练，标记当前 checkpoint 为 suspect，从最近 healthy checkpoint 恢复，并 replay NaN 前后的 batch，检查 loss、logits、activation、gradient、optimizer state 中最早出现 NaN/Inf 的位置。重点排查学习率、FP16 overflow、attention mask、异常数据、自定义 kernel 和 optimizer state。

如果是 hang，我会看所有 rank 的 heartbeat 和最后日志，判断是否某个 rank OOM、数据 batch 数不一致、collective 不匹配、PP send/recv 顺序错或 NCCL 网络问题。不能只看 rank 0。

如果是吞吐骤降，我会拆解 step time，看 dataloader、NCCL、compute、pipeline bubble、checkpoint/eval 和 per-rank step time，判断是数据、通信、硬件还是调度问题。

恢复上，我会优先从健康 checkpoint 回滚，确保 optimizer、scheduler、scaler、RNG 和 dataloader state 都恢复。事故后做复盘，把根因转化为监控、测试、数据校验或自动恢复策略，防止复发。

### 十七、常见追问与回答

问题 1：NaN 后能不能继续训练？

回答：通常不应该。NaN 可能污染参数和 optimizer state，应从最近 healthy checkpoint 恢复，并定位首个 NaN 来源。

问题 2：训练 hang 你首先看什么？

回答：看所有 rank 的最后日志和 heartbeat，确定哪个 rank 最先异常、卡在哪个 collective 或 send/recv，是否有 rank 退出、OOM 或数据耗尽。

问题 3：吞吐下降但 loss 正常，可能是什么？

回答：更可能是系统性能问题，例如 dataloader 慢、网络拥塞、straggler、checkpoint I/O、sequence length 分布变化、pipeline bubble 增大。

问题 4：如何避免 checkpoint 损坏？

回答：写临时目录，完成后原子 rename；保存 manifest 和 checksum；只加载 complete 且 healthy 的 checkpoint；定期做恢复演练。

问题 5：数据事故怎么提前发现？

回答：做数据版本管理、schema 校验、shard checksum、sample logging、domain validation loss、数据配比监控和 contamination 检测。

问题 6：事故复盘最重要的是什么？

回答：找到系统性改进措施，而不是只修一次。要把事故转化为监控、测试、流程和自动恢复能力。

### 十八、常见误区

1. 误区：训练事故就是模型代码 bug。
   纠正：数据、通信、硬件、checkpoint、配置和基础设施都可能是根因。

2. 误区：NaN 后重启当前 checkpoint 就行。
   纠正：当前 checkpoint 可能已被污染，应回到 healthy checkpoint。

3. 误区：hang 只看 rank 0 日志。
   纠正：分布式 hang 往往由某个非 0 rank 引起，需要 per-rank 视角。

4. 误区：吞吐下降不影响训练质量，可以不管。
   纠正：吞吐下降会增加成本，也可能暴露数据或系统异常。

5. 误区：checkpoint 能保存就一定能恢复。
   纠正：必须定期做恢复演练，验证 optimizer、scheduler、data state 等完整恢复。

6. 误区：事故复盘是追责。
   纠正：复盘的目标是改进系统，减少同类事故复发。

### 本讲总结

本讲最重要的结论：

1. 大规模训练是长周期分布式系统，事故不可避免，关键是早发现、快定位、可恢复。
2. 常见事故包括数值、数据、分布式、硬件、checkpoint、性能、代码和配置问题。
3. 事故发生后要先保留现场，记录 step、tokens、rank、data shard、checkpoint、loss、grad norm 和配置变更。
4. NaN 通常需要停止并从 healthy checkpoint 恢复，不能盲目继续训练。
5. Hang 需要 per-rank 视角，重点排查 collective 不匹配、rank 退出、数据不一致和网络问题。
6. 吞吐骤降也是事故，需要拆解 dataloader、通信、compute、stage balance、I/O 和硬件状态。
7. 成熟训练系统需要监控、报警、自动恢复、checkpoint 健康管理和事故复盘机制。
