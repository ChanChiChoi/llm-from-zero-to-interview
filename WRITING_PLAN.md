# 二十四本书写作规划与自查规范

## 文件目的

本文件用于约束后二十四本书的写作顺序、章节模板、同步维护策略和质量自查标准，防止内容跑偏、堆砌概念或偏离面试目标。

`background.md` 是最高写作规范，本文件是具体执行规划。

## 总体写作策略

二十四本书不能按顺序一本一本从头写到尾，否则周期太长，也容易前后风格不统一。

整套书的写作宗旨是：不做孤立知识点罗列，而是帮助读者建立可迁移的大模型知识图谱。每个重要概念都要尽量放回它的历史背景、问题来源、方法谱系和后续演化中讲清楚，让小白知道“为什么有这个东西”，让中级读者知道“它和其他方法怎么连起来”，让高级读者看到“它的边界、trade-off 和可能的下一步”。

后续采用三层推进：

1. 第一层：先完成主干学习路径。
2. 第二层：同步补齐工具书、实战、题库和项目。
3. 第三层：再扩展进阶专题、研究专题和资深经验。

核心原则：

1. `background.md` 是最高写作规范。
2. 每章都要围绕“面试能力 + 实战理解”写。
3. 每个技术点都要讲“为什么提出、解决什么问题、历史背景、前人方法、核心特点、优缺点、适用场景、后来者和改进方向”。
4. 复杂点必须有代码、例子、debug 清单或面试题。
5. 不因篇幅限制压缩解释。章节长度应服务于读者理解，必要时一章拆多个文件，但不能为了让文件变短而省略背景、脉络、例子、优缺点和专家层解释。
6. 强规则：后续任何正式章节扩写、第二版重写、专题大纲设计、关键知识点补全、题库补充、练习设计和项目路线更新前，都必须先联网搜索相关资料，再基于权威资料、既有章节和项目目标共同写作。
7. 联网检索的目的不是复制资料，而是扩展写作广度和深度：补充遗漏概念、最新进展、官方定义、工程实现、论文结论、失败案例、真实系统 trade-off 和面试深挖点。
8. 对优化器、DPO/GRPO/PPO、主流大模型架构创新、推理框架、Agent 协议、多模态模型等高时效主题，扩写前必须先联网检索最新公开资料，优先参考论文、官方技术报告、官方文档和高可信工程博客。
9. 网络资料必须做可信度筛选，不能全信。来源不明、营销化、无法复现、与官方资料冲突、只有单一社区帖子支持的信息，不能直接写成确定结论；重要结论应尽量交叉验证，必要时标注“待核验”“社区说法”或不纳入正文。
10. 资料优先级为官方论文、官方技术报告、官方 GitHub 仓库、官方文档、标准或协议原文、主流框架文档、知名研究者或工程团队博客、可信社区讨论。低可信来源只能作为线索，不能直接当作结论。

## 正式章节模板

每个正式章节默认采用：

1. 本章目标
2. 问题背景
3. 这个问题为什么重要
4. 核心直觉
5. 技术原理
6. 关键公式或机制
7. 最小代码或例子
8. 真实项目中的坑
9. 优点、缺点和适用场景
10. 常见改进方向
11. 面试官会怎么问
12. 标准回答模板
13. 常见误区
14. 小练习
15. 本章总结

不是所有章节都必须机械包含全部小节，但重要技术点至少要覆盖背景、动机、直觉、机制、例子或代码、优缺点、适用场景和面试问题。

对于有明显发展脉络、概念容易被小白误解、或面试中经常追问来龙去脉的知识点，正文应酌情增加两个小节：

1. 来龙去脉：解释这个概念为什么出现、它之前的方法是什么、前人方法有什么问题、它解决了哪些问题、它的核心特点是什么、优缺点是什么、后来者或可能的后来者针对哪些点做了改进，以及它和相邻概念是什么关系。例如 AdamW、Muon、RLHF、DPO、FlashAttention、PagedAttention、MoE、Mamba、Constitutional AI、LLM-as-a-Judge 等都适合写来龙去脉。
2. 面向专家：在小白友好解释之后，补充更深入的机制、公式、边界条件、实现细节、工程 trade-off、论文差异或前沿争议，帮助读者从“能听懂”升级到“能被专家追问”。

这两个小节不是所有知识点都必须加。稳定、简单、低频或只作背景铺垫的概念可以不加。判断标准是：如果不讲历史发展会让读者不知道它为什么重要，或者不讲专家层细节会导致回答停留在表面，就应该加。

后续写第五册训练方法、优化器、后训练算法、推理框架、安全对齐、架构演进和高时效专题时，默认优先考虑是否需要加入“来龙去脉”和“面向专家”小节，避免内容过于简洁或只适合已有基础的读者。

## 文件拆分规范

文件拆分只服务于阅读和维护，不服务于压缩内容。整套书是写给人看的，不能因为担心篇幅过长而把知识点写得过密、过短或过于跳跃。

1. 不设置固定行数或字数上限；以读者能否顺畅理解为第一标准。
2. 如果章节很长，可以按主题自然拆成多个文件，例如 `01-xxx-上.md`、`01-xxx-下.md`、`01-xxx-code.md`、`01-xxx-interview.md`。
3. 拆分时保持上下文连续，拆出去的文件也要有清楚的入口、目标和前后关系。
4. 代码较多时可以拆到 `code/` 或章节子文件，但正文仍要解释输入、输出、shape、机制和 debug 思路。
5. 不把长正文堆到 `目录.md`；目录只承担索引、阅读方式和章节导航。
6. 根目录只放总控文件，不放长章节正文。
7. 如果“拆文件”和“讲清楚”冲突，优先讲清楚，再通过目录和链接改善可读性。

## 代码规范

复杂技术点优先配代码：

1. 基础数学：纯 Python。
2. 深度学习：PyTorch。
3. 微调和部署：允许 Hugging Face、PEFT、TRL、vLLM。
4. 多模态：允许 Pillow、torchaudio、transformers。
5. 每段代码必须说明输入、输出、shape。
6. 面试高频代码要尽量手写，不直接调用高级封装。
7. 工程代码要配 debug 说明，而不是只给 happy path。

优先写的代码：cross entropy、attention、causal mask、multi-head attention、RoPE、top-k/top-p、KV Cache、LoRA、DPO loss、miniGPT、RAG 检索评估、CLIP loss、简单 diffusion 加噪、推理延迟压测脚本。

## 第二版全书写作计划

第二版目标不是简单润色第一版，而是在完整书系主线已经打通后，基于强联网检索和权威资料复核，对二十四本书做一次系统升级：扩大知识覆盖面，补齐前沿变化，增强小白友好解释，提升专家层深度，并让每章都更能服务真实面试和工程落地。

### 第二版总原则

1. 每次进入任何章节、专题、题库、练习、项目或术语表修改前，必须先联网搜索相关资料。
2. 联网资料只能作为辅助材料，不能替代本项目自己的判断、结构和表达。
3. 写作时要同时参考当前章节、相邻章节、根目录总控文件和联网资料，避免局部更新导致全书体系割裂。
4. 每个重要结论都要尽量确认来源类型：官方文档、论文结论、源码实现、工程经验、社区推测或待核验信息。
5. 对存在争议或版本差异的内容，不写成单一绝对结论，要说明适用条件、版本边界和可能的不同实现。
6. 第二版要优先补“来龙去脉”和“面向专家”，让章节既能让小白看懂，也能支撑高阶面试追问。
7. 第二版不能变成资料堆砌。网络资料要被消化成历史背景、问题动机、方法谱系、机制解释、优缺点、工程 trade-off、面试问答和练习。

### 每次写作前的联网流程

1. 明确本次要写或改的主题，例如章节名、知识点、项目任务或面试题方向。
2. 搜索 3 类资料：官方资料、论文或技术报告、工程实现或高可信技术解读。
3. 对高时效主题额外搜索最新版本、近期论文、官方 release note、框架文档和真实工程实践。
4. 记录资料中的关键增量：新增概念、主流做法、争议点、版本变化、常见误区、工程坑和可面试的问题。
5. 过滤低可信资料：广告稿、营销文、单一社区传闻、无法复现内容、与官方资料冲突但无证据的说法。
6. 将可用资料映射到本项目章节结构，而不是直接复制链接或原文。
7. 写作完成后检查是否仍然符合 `background.md`、本文件和 `写作注意事项.md`。

### 资料可信度使用规则

1. 第一优先级：论文原文、官方技术报告、官方文档、官方 GitHub 仓库、标准或协议原文。
2. 第二优先级：主流框架文档、核心开发者博客、知名研究者文章、工程团队复盘、公开 benchmark 或评测报告。
3. 第三优先级：高质量社区讨论、源码阅读笔记、课程材料、个人博客，可用于发现线索，但需要交叉验证。
4. 低可信来源：营销稿、搬运文、无来源总结、标题党、无法复现实验、单一社交媒体说法，只能作为搜索线索，不能作为正文依据。
5. 如果资料之间冲突，优先相信官方材料、论文、源码和可复现实验；仍无法判断时，应在正文中降低确定性，标注“待核验”或不写入正文。
6. 对闭源模型、商业系统和未公开训练细节，必须区分官方披露、外部测评、社区推测和个人判断。
