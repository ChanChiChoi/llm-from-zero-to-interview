# 第六章：混合精度与显存优化

大模型训练最常见的两个资源瓶颈是显存和算力。本章讨论如何在尽量不损失模型质量的前提下，降低显存占用、提高吞吐并避免数值不稳定。核心手段包括混合精度训练、梯度缩放、activation checkpointing、梯度累积、参数和激活的显存规划，以及对 OOM 的系统性排查。

很多训练脚本不是模型不对，而是显存吃不下：batch size 太大、序列太长、激活太多、优化器状态太重、没有用 AMP、梯度没有及时释放、保存了多余中间 tensor、DataLoader 把 batch 预取得过多、验证时忘了关梯度。本章的目标是建立一个对显存有感知的训练思维。

## 6.1 显存到底花在什么地方

训练时显存通常分成几类：

1. 模型参数。
2. 梯度。
3. 优化器状态。
4. 前向激活。
5. 临时 workspace 和 kernel buffer。
6. 输入 batch 和中间缓存。

粗略理解：

1. 参数是模型本体。
2. 梯度是 backward 的结果。
3. 优化器状态通常比参数还大，尤其是 Adam 系列。
4. 激活和序列长度、batch size、层数强相关。
5. 临时 buffer 取决于算子实现和 kernel。

在大模型训练里，很多时候真正的瓶颈不是参数，而是激活和优化器状态。

面试回答：

```text
训练显存主要花在参数、梯度、优化器状态、激活和临时 buffer 上。参数是模型本体，梯度是反向传播结果，Adam 类优化器还要保存一阶矩和二阶矩，激活则和 batch size、序列长度、层数直接相关。很多 OOM 并不是因为模型参数太大，而是激活或 optimizer state 太大。
```

## 6.2 为什么需要混合精度

传统上很多训练用 fp32。fp32 精度高，但显存和带宽开销大。

混合精度训练的思路是：

1. 用较低精度存储和计算大部分张量。
2. 保留少数关键位置使用较高精度。
3. 在数值敏感环节保持稳定。

常见组合：

1. fp16。
2. bf16。
3. fp32 + amp。

它的目标不是“全部变低精度”，而是找到质量、性能和显存之间的平衡点。

优势：

1. 显存更省。
2. 带宽压力更小。
3. 通常训练更快。

代价：

1. 数值更脆弱。
2. 某些算子容易溢出或下溢。
3. 需要更多工程控制。

## 6.3 FP16 和 BF16 的区别

fp16 和 bf16 都是 16 位浮点，但表示方式不同。

### 6.3.1 FP16

fp16 的优点是：

1. 计算和存储都省。
2. 在很多 GPU 上支持成熟。

缺点是：

1. 指数范围较小。
2. 更容易溢出、下溢。
3. 训练时通常要配合 GradScaler。

### 6.3.2 BF16

bf16 的指数范围接近 fp32，动态范围更大。

优点是：

1. 更不容易数值溢出。
2. 大模型训练通常更稳。
3. 在很多现代硬件上支持很好。

缺点是：

1. 尾数精度比 fp16 低。
2. 某些硬件或旧环境支持不如 fp16 普遍。

工程上常见判断：

1. 如果硬件和框架支持良好，bf16 往往更省心。
2. 如果只能用 fp16，通常要配合 GradScaler。

## 6.4 autocast：自动混合精度

PyTorch 的 AMP 通过 `autocast` 自动把合适的算子放到低精度执行。

示例：

```python
from torch.cuda.amp import autocast

with autocast():
    logits = model(input_ids)
    loss = criterion(logits, labels)
```

直觉：

1. 某些算子适合低精度，比如大部分矩阵乘法。
2. 某些算子更适合保留高精度，比如 softmax、归一化、loss 计算中的敏感部分。
3. `autocast` 负责自动选择。

工程意义：

1. 减少手动 dtype 管理。
2. 更容易获得性能收益。
3. 降低写错精度策略的概率。

但要记住：`autocast` 不是万能的。它只是帮助管理前向中的算子精度，仍然需要结合模型结构和具体异常去判断。

## 6.5 GradScaler：解决 fp16 的数值问题

fp16 训练容易遇到梯度太小导致下溢。`GradScaler` 的做法是先放大 loss，再反向传播，再把梯度缩回去。

典型结构：

```python
from torch.cuda.amp import GradScaler, autocast

scaler = GradScaler()

for batch in train_loader:
    optimizer.zero_grad(set_to_none=True)

    with autocast():
        outputs = model(**batch)
        loss = outputs["loss"]

    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(optimizer)
    scaler.update()
```

关键步骤：

1. `scale(loss)` 放大 loss。
2. `backward()` 在放大后的尺度上计算梯度。
3. `unscale_(optimizer)` 把梯度缩回来。
4. 必要时再做梯度裁剪。
5. `step` 和 `update` 完成参数更新和缩放因子调整。

为什么要先 `unscale_` 再裁剪？因为裁剪的是实际梯度，不是被放大的临时梯度。

bf16 通常不那么依赖 GradScaler，但代码结构上仍可以保持一致。

## 6.6 mixed precision 训练的推荐模板

一个常见训练模板：

```python
scaler = torch.cuda.amp.GradScaler(enabled=use_fp16)

for batch in train_loader:
    batch = move_to_device(batch, device)
    optimizer.zero_grad(set_to_none=True)

    with torch.cuda.amp.autocast(enabled=use_amp):
        outputs = model(**batch)
        loss = outputs["loss"]

    if use_fp16:
        scaler.scale(loss).backward()
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        scaler.step(optimizer)
        scaler.update()
    else:
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()
```

这里要注意：

1. `autocast` 控制前向精度。
2. `GradScaler` 主要服务 fp16。
3. `bf16` 训练通常不需要缩放，但也可以保留统一结构。

## 6.7 activation checkpointing

activation checkpointing 的核心思想是：前向时少存一部分激活，反向时再重新计算这部分前向，从而用计算换显存。

直觉：

1. 普通训练保留很多中间激活，方便 backward。
2. checkpointing 只保留关键节点。
3. backward 时重新跑部分 forward 来恢复中间值。

PyTorch 里可以这样用：

```python
from torch.utils.checkpoint import checkpoint


def forward_block(x, block):
    return checkpoint(block, x)
```

或者在模型里包住某些子模块。

优点：

1. 明显省显存。
2. 可以支持更长序列或更大 batch。

缺点：

1. backward 更慢，因为要重算前向。
2. 代码结构更复杂。

适合：

1. Transformer block 很深的模型。
2. 长上下文训练。
3. 显存非常紧张的实验。

## 6.8 梯度累积和显存

梯度累积不仅是优化 batch size 的办法，也是显存优化手段。

如果一次只能放很小的 batch，就把多个 micro-batch 的梯度累积起来再更新。

示意：

```python
accum_steps = 8
optimizer.zero_grad(set_to_none=True)

for step, batch in enumerate(train_loader):
    with torch.cuda.amp.autocast(enabled=use_amp):
        outputs = model(**batch)
        loss = outputs["loss"] / accum_steps

    loss.backward()

    if (step + 1) % accum_steps == 0:
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)
```

优点：

1. 降低单次 forward/backward 的激活峰值。
2. 模拟更大 batch。

缺点：

1. 更新频率变低。
2. 训练吞吐不一定线性提升。

## 6.9 影响显存的关键因素

显存不是只由 batch size 决定，还包括：

1. 序列长度。
2. 模型层数。
3. hidden size。
4. 注意力头数。
5. 精度。
6. optimizer 类型。
7. 是否保留中间激活。
8. 是否开启 checkpointing。
9. 是否存了额外的日志或缓存。

尤其对 Transformer 来说，序列长度增加往往会显著推高激活和 attention 相关开销。

一个实用判断：如果显存爆了，先问自己是参数太大、batch 太大、序列太长、还是优化器状态太重。

## 6.10 optimizer 状态为什么也占很多显存

以 Adam 为例，除了参数本身，还要保存一阶矩和二阶矩。

直觉上，每个参数可能对应：

1. 参数值。
2. 梯度。
3. 一阶矩。
4. 二阶矩。

所以 Adam 类优化器的总显存压力明显高于 SGD。

这也是为什么在超大模型训练里，经常会考虑：

1. 参数分片。
2. 优化器状态分片。
3. 更低精度状态。
4. 更省显存的训练策略。

本章重点不是展开分布式分片，而是先建立“optimizer 状态本身也很贵”的意识。

## 6.11 训练中常见的显存优化顺序

当你遇到 OOM，可以按下面顺序试：

1. 减小 batch size。
2. 减小 sequence length。
3. 开启 mixed precision。
4. 开启 gradient accumulation。
5. 开启 activation checkpointing。
6. 检查是否保留了多余 tensor。
7. 检查 validation 是否忘了关梯度。
8. 检查 DataLoader 预取和 batch 缓存。

这个顺序的逻辑是：

1. 先改最直接影响显存的变量。
2. 再上混合精度和 checkpointing。
3. 最后排查代码层面的泄漏。

## 6.12 显存泄漏的常见来源

不是所有显存增长都是“模型太大”。很多是代码把图保留下来了。

常见来源：

1. 把带计算图的 loss 存进 list。
2. 把中间 activation 长期缓存。
3. 验证时没用 `no_grad` 或 `inference_mode`。
4. hook 里保存了未 detach 的 tensor。
5. 每步都把大 batch 的输出堆到内存里。
6. 重复创建没有释放的大 tensor。

如果怀疑泄漏，可以每隔几步打印：

```python
print(torch.cuda.memory_allocated() / 1024**2)
print(torch.cuda.memory_reserved() / 1024**2)
```

还可以更系统地用 PyTorch 的显存统计接口排查。

## 6.13 OOM 排查方法

当出现 OOM 时，不要只盯着最后一行报错，要分层排查。

### 6.13.1 先看是训练还是验证

1. 如果训练 OOM，关注 batch、seq len、激活、梯度和 optimizer state。
2. 如果验证 OOM，重点看是否忘记关梯度、是否缓存了输出、是否 batch 过大。

### 6.13.2 先缩小 batch 和 sequence length

这是最直接的验证手段。若缩小后能跑，说明问题和激活/序列长度强相关。

### 6.13.3 检查是否在累积图

例如：

```python
losses.append(loss)
```

如果长期保存带图对象，显存会持续增长。

应该改成：

```python
losses.append(loss.item())
```

### 6.13.4 检查验证流程

验证时应：

1. `model.eval()`。
2. `torch.inference_mode()`。
3. 不做 backward。
4. 不把大输出长期保存在 GPU 上。

## 6.14 一个完整 AMP + checkpointing 的训练骨架

```python
from torch.cuda.amp import autocast, GradScaler
from torch.utils.checkpoint import checkpoint


def train_step(model, batch, optimizer, scaler, use_amp=True):
    optimizer.zero_grad(set_to_none=True)

    with autocast(enabled=use_amp):
        outputs = model(**batch)
        loss = outputs["loss"]

    if use_amp:
        scaler.scale(loss).backward()
        scaler.unscale_(optimizer)
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        scaler.step(optimizer)
        scaler.update()
    else:
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()

    return loss.item()
```

如果模型内部某些 block 需要 checkpointing，可以把对应子模块包装起来。重点是要理解：

1. AMP 解决的是精度和带宽问题。
2. checkpointing 解决的是激活显存问题。
3. 梯度累积解决的是单步 batch 太大的问题。

这三者经常一起用。

## 6.15 工程中怎么选 fp16 和 bf16

简单经验：

1. 如果硬件支持良好且你希望更稳，优先 bf16。
2. 如果环境主要支持 fp16，使用 fp16 + GradScaler。
3. 如果发现训练频繁 NaN，先看学习率、初始化、loss、mask，再看换 bf16 是否更稳。

不要把“混合精度”理解成固定 recipe。它仍然要和模型结构、数据分布和硬件一起权衡。

## 6.16 面试官会怎么问

### 问题一：FP16 和 BF16 有什么区别？

回答模板：

```text
两者都是 16 位浮点，但 BF16 的指数位更多，动态范围更接近 FP32，因此更不容易溢出，通常训练更稳；FP16 的精度和效率也很好，但更容易数值不稳定，所以常常要配合 GradScaler。实际选择要看硬件和训练稳定性。
```

### 问题二：autocast 和 GradScaler 分别做什么？

回答模板：

```text
autocast 负责自动把适合的算子放到低精度执行，减少手动 dtype 管理；GradScaler 主要用于 FP16，通过放大 loss、反向后再缩回梯度，避免下溢并提升训练稳定性。它们解决的是不同问题。
```

### 问题三：activation checkpointing 为什么能省显存？

回答模板：

```text
因为它不保存所有前向激活，而是在 backward 时重新计算一部分前向，从而用额外计算换取更少的激活显存。它适合深层模型和长序列训练，但会让反向更慢。
```

### 问题四：为什么 Adam 比 SGD 更占显存？

回答模板：

```text
因为 Adam 不只保存参数和梯度，还要为每个参数保存一阶矩和二阶矩状态，所以 optimizer state 很大。对超大模型来说，optimizer state 常常是显存大头之一。
```

### 问题五：训练 OOM 时你怎么排查？

回答模板：

```text
我会先区分是训练还是验证阶段，再从 batch size、sequence length、精度、梯度累积、activation checkpointing、optimizer state 和是否保留了带图 tensor 这些方向排查。通常先缩小 batch 和序列长度验证是否和激活相关，再检查是否有显存泄漏或错误缓存。
```

## 6.17 常见误区

1. 以为 fp16 一定比 bf16 更好。实际上 bf16 往往更稳。
2. 以为 AMP 等于自动省一切显存。它主要解决精度和部分带宽问题，不是万能。
3. 以为 GradScaler 对所有混合精度都必须。bf16 通常不一定需要。
4. 以为 checkpointing 没有代价。它会增加 backward 计算量。
5. 以为 OOM 都是 batch 太大。也可能是 optimizer state、保存图或验证没关梯度。
6. 以为 loss 降低说明显存问题消失。数值稳定和显存问题是两个不同维度。

## 6.19 本章总结

混合精度与显存优化的核心不是背几个 API，而是建立资源感知的训练思维。

要记住的主线：

1. 显存主要花在参数、梯度、优化器状态、激活和临时 buffer。
2. fp16 和 bf16 都是混合精度方案，但 bf16 通常更稳。
3. `autocast` 管前向精度，`GradScaler` 主要解决 fp16 的数值问题。
4. activation checkpointing 用计算换显存。
5. 梯度累积可以模拟更大 batch 并缓解单步显存压力。
6. OOM 排查要区分训练、验证、显存泄漏和配置错误。
7. 一个好的训练脚本要同时考虑数值稳定、吞吐和显存峰值。
