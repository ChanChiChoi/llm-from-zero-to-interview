# 第十章：工程面试题

本章把第十四册的 PyTorch 与深度学习工程知识整理成面试问答。重点不是背答案，而是训练一种工程表达方式：先讲核心机制，再讲常见坑，最后给出排查路径或实现要点。很多面试问题表面是 PyTorch API，实际考的是你是否真的写过训练代码、是否能定位训练异常、是否理解大模型组件的 shape 和系统约束。

建议使用方法：先自己口述，再对照参考回答。每题尽量控制在 1-3 分钟内，如果面试官追问，再展开代码细节或 debug 清单。

## 10.0.1 工程面试公式与排查速查

第一，语言模型 batch 的 shape 主线是：

```math
I\in\mathbb{Z}^{B\times T},\qquad H\in\mathbb{R}^{B\times T\times d},\qquad Z\in\mathbb{R}^{B\times T\times V}
```

其中 `I` 是 `input_ids`，`H` 是 hidden states，`Z` 是 logits，`B` 是 batch size，`T` 是序列长度，`d` 是 hidden size，`V` 是词表大小。

第二，causal LM 的 shift loss 展平后行数应一致：

```math
N_{\mathrm{row}}=B(T-1),\qquad Z_{\mathrm{shift}}\in\mathbb{R}^{N_{\mathrm{row}}\times V},\qquad Y_{\mathrm{shift}}\in\mathbb{Z}^{N_{\mathrm{row}}}
```

面试里如果只说“算 cross entropy”，但没有说清 `logits[:, :-1, :]` 和 `labels[:, 1:]`，通常还不够。

第三，梯度累积要缩放 loss：

```math
g_{\mathrm{acc}}=\frac{1}{K}\sum_{i=1}^{K}g_i
```

其中 `K` 是 accumulation steps。实现上常见做法是每个 micro-batch 的 loss 除以 `K` 后再 backward。

第四，global batch size 是：

```math
B_{\mathrm{global}}=B_{\mathrm{device}}\cdot W\cdot K
```

其中 `B_device` 是单进程 batch size，`W` 是 world size，`K` 是 accumulation steps。从单卡切到多卡时，学习率、warmup、日志步数和评估频率都要重新核对。

第五，训练显存可粗略拆成：

```math
M_{\mathrm{train}}\approx M_{\mathrm{param}}+M_{\mathrm{grad}}+M_{\mathrm{opt}}+M_{\mathrm{act}}+M_{\mathrm{tmp}}
```

其中 AdamW optimizer state 常包含一阶矩和二阶矩，长序列训练中 activation 也可能成为主要瓶颈。

第六，DDP 梯度同步的核心是平均梯度：

```math
\bar g=\frac{1}{W}\sum_{r=0}^{W-1}g_r,\qquad \theta_{t+1}=\theta_t-\eta\bar g
```

如果不同 rank 没有同步到同一个 `\bar g`，参数会逐步漂移。

第七，多头 attention 的 score shape 是：

```math
Q,K,V\in\mathbb{R}^{B\times H\times T\times D_h},\qquad S=QK^\top\in\mathbb{R}^{B\times H\times T\times T}
```

面试中要同时说清 shape、mask 方向、softmax 维度和合并多头时的 `contiguous()` / `reshape()` 问题。

第八，KV Cache 单层显存量级是：

```math
M_{\mathrm{kv,layer}}=2 B H_{\mathrm{kv}} T_{\mathrm{cache}} D_h b
```

其中因子 2 来自 K 和 V，`b` 是每个元素字节数。长上下文和高并发服务中，KV Cache 往往是显存和带宽瓶颈。

第九，一个 PyTorch 工程面试回答可以用门禁检查：

```math
G_{\mathrm{eng}}=\mathbf{1}[
C_{\mathrm{topic}}\ge \tau_c,
C_{\mathrm{formula}}\ge \tau_f,
C_{\mathrm{debug}}\ge \tau_d,
R_{\mathrm{flag}}=0
]
```

其中 `C_topic` 是主题覆盖率，`C_formula` 是公式和 shape 覆盖率，`C_debug` 是排查路径覆盖率，`R_flag` 是空泛、绝对化或夸大表述的风险数。这个门禁不是招聘标准，而是自我复盘工具。

## 10.1 Tensor 与 Shape

### 问题 1：PyTorch tensor 最重要的元信息有哪些？

参考回答：

```text
我会优先看 shape、dtype、device、requires_grad 和 contiguous 状态。shape 决定矩阵乘法、loss 和 mask 是否对齐；dtype 决定精度和模块输入要求；device 决定是否会 CPU/GPU mismatch；requires_grad 决定是否参与 autograd；contiguous 和 stride 会影响 view、transpose 后的 reshape 行为。
```

### 问题 2：`view` 和 `reshape` 有什么区别？

参考回答：

```text
view 要求 tensor 的内存布局能按目标 shape 解释，通常要求 contiguous 或 stride 兼容；reshape 更宽松，如果不能返回 view，可能会创建拷贝。工程里如果只是想安全改 shape，常用 reshape；如果想明确避免拷贝，需要确认 tensor 是否 contiguous 后再 view。
```

### 问题 3：为什么 `transpose` 后经常要 `contiguous()`？

参考回答：

```text
transpose 通常只是改变 stride，不会真的重排底层内存，因此结果可能不是 contiguous。后面如果直接 view 合并维度，可能因为 layout 不兼容报错。常见写法是 transpose 后 contiguous().view(...)，或者直接用 reshape。
```

### 问题 4：Broadcasting 规则是什么？

参考回答：

```text
Broadcasting 从右往左对齐维度，每一维要么相等，要么其中一个是 1，要么其中一个不存在。它常用于 bias、LayerNorm 参数、attention mask 等场景。调试时不仅要看能否广播，还要看广播方向是否符合语义。
```

## 10.2 Autograd 与梯度

### 问题 5：PyTorch autograd 是怎么工作的？

参考回答：

```text
PyTorch 使用动态计算图。每次 forward 时，如果输入或参数需要梯度，相关操作会记录反向函数和必要中间值。调用 loss.backward() 后，autograd 从 loss 开始按链式法则反向遍历计算图，把梯度累积到叶子参数的 .grad 上。默认 backward 后计算图会释放。
```

### 问题 6：为什么每个 step 要 `zero_grad()`？

参考回答：

```text
因为 PyTorch 的梯度默认累积，不会在 backward 前自动清零。普通训练中如果不清梯度，当前 batch 的梯度会叠加到历史梯度上，导致更新方向和尺度错误。梯度累积场景会利用这个特性，但普通 step 要主动 zero_grad。
```

### 问题 7：为什么中间 tensor 的 `.grad` 是 None？

参考回答：

```text
默认只有叶子 tensor 的 .grad 会被保存，模型参数就是叶子 tensor。中间 tensor 虽然有梯度流过，但为了省显存，PyTorch 不默认保留它的 .grad。如果调试需要，可以调用 retain_grad() 或注册 hook。
```

### 问题 8：`detach()` 和 `no_grad()` 有什么区别？

参考回答：

```text
detach 是对某个 tensor 切断它和当前计算图的连接，常用于日志、缓存或 stop-gradient。no_grad 是上下文管理器，在作用域内不记录新的 autograd 图，常用于验证和推理。detach 更像切断一条边，no_grad 更像让一段代码不建图。
```

## 10.3 nn.Module 与模型组织

### 问题 9：`nn.Module` 的作用是什么？

参考回答：

```text
nn.Module 是 PyTorch 组织模型的核心抽象。它不仅封装 forward，还负责注册 Parameter、子模块和 buffer，并支持递归遍历参数、切换 train/eval、移动 device、保存和加载 state_dict。optimizer、DDP、AMP、checkpoint 都依赖这个结构。
```

### 问题 10：为什么子模块不能放普通 list？

参考回答：

```text
普通 Python list 里的 Module 不会被 nn.Module 自动注册，因此 parameters()、state_dict()、to(device)、train/eval、DDP 都无法递归处理它们。应该使用 ModuleList 或 ModuleDict 保存可训练子模块。
```

### 问题 11：Parameter 和 buffer 的区别是什么？

参考回答：

```text
Parameter 是模型可训练权重，通常会被 optimizer 更新；buffer 是不训练但属于模型状态的 tensor，比如 BatchNorm running statistics、固定位置编码或 causal mask。buffer 会随 model.to(device) 移动，并默认保存在 state_dict 中，但不会出现在 parameters() 里。
```

### 问题 12：`state_dict` 保存什么？

参考回答：

```text
state_dict 保存注册的 Parameter 和持久化 buffer，不保存 Python 类定义、forward 代码和 optimizer 状态。训练恢复时还要额外保存 optimizer、scheduler、step、config 和随机状态。
```

## 10.4 Dataset、DataLoader 与训练数据

### 问题 13：Dataset 和 DataLoader 有什么区别？

参考回答：

```text
Dataset 定义单个样本如何读取和返回，DataLoader 负责把样本组织成 batch，并支持 shuffle、sampler、多进程加载和 collate_fn。Dataset 偏样本级，DataLoader 偏 batch 级和加载工程。
```

### 问题 14：为什么变长序列需要自定义 collate_fn？

参考回答：

```text
默认 collate 会尝试把样本 stack 成规则张量，但变长 token 序列长度不同，不能直接 stack。自定义 collate_fn 可以在 batch 级做 padding、attention_mask、labels=-100 和其他复杂字段整理。
```

### 问题 15：padding token 为什么不能直接参与 loss？

参考回答：

```text
padding 只是为了把变长样本组成 batch，不代表真实训练目标。如果 padding token 参与 loss，模型会被训练去预测无意义的 pad，影响优化目标。通常把 padding label 设为 -100，并在 cross entropy 中用 ignore_index 跳过。
```

## 10.5 训练循环

### 问题 16：一个标准训练 step 包含哪些步骤？

参考回答：

```text
标准训练 step 包括 batch 到 device、zero_grad、forward、loss 计算、backward、可选梯度裁剪、optimizer.step、scheduler.step 和日志记录。顺序上先 backward 得到梯度，再裁剪，最后更新参数。
```

### 问题 17：梯度累积为什么要除以 accumulation steps？

参考回答：

```text
因为梯度默认累积。如果累积 N 个 micro-batch 但不缩放 loss，最终梯度尺度大约是平均梯度的 N 倍，相当于放大学习率。把 loss 除以 accumulation steps 后，累积梯度更接近真实大 batch 的平均梯度。
```

### 问题 18：验证循环和训练循环有什么区别？

参考回答：

```text
验证时要 model.eval()，并用 no_grad 或 inference_mode 关闭梯度，不做 backward 和 optimizer step。验证结束后要切回 model.train()。训练循环则需要保留计算图、反向传播并更新参数。
```

### 问题 19：checkpoint 应该保存什么？

参考回答：

```text
如果只是推理，保存模型 state_dict 就够了；如果要恢复训练，应该保存 model、optimizer、scheduler、epoch、global_step、config 和随机状态。否则恢复后学习率、动量和二阶矩丢失，训练轨迹会改变，严重时可能 loss spike。
```

## 10.6 混合精度与显存

### 问题 20：训练显存主要花在哪里？

参考回答：

```text
训练显存主要花在模型参数、梯度、优化器状态、前向激活、临时 buffer 和输入 batch 上。Adam 类优化器还要保存一阶矩和二阶矩，激活则和 batch size、序列长度、层数强相关。很多 OOM 是激活或 optimizer state 太大，而不是参数本身太大。
```

### 问题 21：FP16 和 BF16 有什么区别？

参考回答：

```text
两者都是 16 位浮点，但 BF16 的指数范围更接近 FP32，因此动态范围更大，更不容易溢出，通常大模型训练更稳。FP16 在很多硬件上也很高效，但更容易下溢或溢出，常常需要 GradScaler。
```

### 问题 22：autocast 和 GradScaler 分别做什么？

参考回答：

```text
autocast 负责在前向中自动选择合适算子的低精度执行，减少手动 dtype 管理；GradScaler 主要用于 FP16，通过放大 loss、反向后再缩回梯度，降低梯度下溢风险。它们解决的是不同问题。
```

### 问题 23：activation checkpointing 为什么能省显存？

参考回答：

```text
它在 forward 时不保存所有中间激活，而是在 backward 时重新计算一部分前向，用额外计算换显存。适合深层模型和长序列训练，但会增加反向计算时间。
```

## 10.7 分布式训练

### 问题 24：DDP 的工作原理是什么？

参考回答：

```text
DDP 是数据并行。每个进程通常绑定一张 GPU，并持有一份模型副本。每个进程处理不同 mini-batch，backward 后通过 all-reduce 同步梯度，保证每张卡得到一致的平均梯度，然后各自 optimizer step，使参数保持一致。
```

### 问题 25：rank、local rank、world size 分别是什么？

参考回答：

```text
world size 是总训练进程数，rank 是全局进程编号，local rank 是当前机器内的进程或 GPU 编号。通常用 local rank 绑定 GPU，用 rank 判断是否是主进程，比如只让 rank 0 打日志和保存 checkpoint。
```

### 问题 26：为什么 DDP 要用 DistributedSampler？

参考回答：

```text
因为每个 rank 应该处理不同数据子集，否则多张卡会重复训练同样样本。DistributedSampler 会按 rank 和 world size 切分数据，并通过 set_epoch 在每个 epoch 刷新 shuffle 顺序。
```

### 问题 27：DDP 和 FSDP 有什么区别？

参考回答：

```text
DDP 每张卡通常持有完整模型副本，主要通过数据并行提升吞吐；FSDP 会把参数、梯度和优化器状态分片到多张卡上，降低单卡显存，适合更大模型，但通信、checkpoint 和 debug 更复杂。
```

## 10.8 Debug 与 Profiling

### 问题 28：loss 不下降怎么排查？

参考回答：

```text
我会先检查数据和 label 是否正确，再看 logits 和 labels 的 shape 是否对齐，loss 是否有限且依赖参数。然后检查梯度是否为 None、0、NaN，参数是否进入 optimizer，学习率是否合理，mask 是否把有效 token 忽略。最后再看模型容量、初始化和数据分布。
```

### 问题 29：训练出现 NaN 怎么排查？

参考回答：

```text
先定位 NaN 出现在输入、logits、loss、梯度还是参数。常见原因包括学习率过大、FP16 溢出、softmax 或 mask 产生异常、loss 分母为 0、数据异常和梯度爆炸。我会记录 loss、lr、grad norm，并用 check_finite 或 anomaly detection 缩小范围。
```

### 问题 30：OOM 怎么排查？

参考回答：

```text
先确认是训练还是验证 OOM，再看 batch size、sequence length、是否开启 AMP、是否保存了带图 tensor、验证是否关闭梯度、是否输出 hidden states 或 attentions。然后考虑 activation checkpointing、梯度累积和减少缓存。
```

### 问题 31：GPU 利用率低怎么排查？

参考回答：

```text
先区分是 DataLoader 慢、CPU-GPU 拷贝慢、模型计算太小、频繁同步还是分布式通信慢。可以单独测试 DataLoader 速度，用 cuda synchronize 做分段计时，再用 torch.profiler 看 CPU/CUDA 时间分布。
```

## 10.9 Transformer 实现

### 问题 32：Self-attention 的 shape 如何变化？

参考回答：

```text
输入 hidden states 是 [B,T,d]。经过 Q/K/V 投影后仍是 [B,T,d]，再 reshape 成 [B,H,T,D]，其中 D=d/H。Q @ K.transpose(-2,-1) 得到 scores [B,H,T,T]，softmax 后乘 V 得到 context [B,H,T,D]，最后合并多头回到 [B,T,d]。
```

### 问题 33：为什么需要 causal mask？

参考回答：

```text
Decoder-only 语言模型按自回归方式预测下一个 token，当前位置不能看到未来 token。训练时虽然整段序列并行输入，但 causal mask 会屏蔽未来位置，防止信息泄漏和 loss 虚低。
```

### 问题 34：RoPE 作用在哪里？

参考回答：

```text
RoPE 通常作用在 attention 的 Q 和 K 上，对它们按位置做旋转。这样 Q 和 K 的点积会包含相对位置信息，模型可以在 attention score 中感知 token 之间的相对距离。
```

### 问题 35：KV Cache 为什么能加速推理？

参考回答：

```text
自回归推理每步只新增一个 token，历史 token 的 K/V 不变。KV Cache 缓存历史 K/V，后续 decode 只计算新 token 的 K/V，并和历史 cache 拼接，避免每一步重复计算完整上下文。
```

## 10.10 系统性开放题

### 问题 36：如何设计一个可靠的训练脚本？

参考回答：

```text
我会把训练脚本分成配置、数据、模型、optimizer/scheduler、训练循环、验证、checkpoint 和日志几部分。训练循环要支持 device 搬运、梯度累积、梯度裁剪、AMP、异常检查和恢复训练。日志至少记录 loss、lr、grad norm、吞吐、显存和验证指标。checkpoint 要保存 model、optimizer、scheduler、step、config 和随机状态。
```

### 问题 37：从单卡训练扩展到多卡训练要改什么？

参考回答：

```text
需要用 torchrun 启动多进程，初始化 process group，用 local_rank 绑定 GPU，把模型包装成 DDP，DataLoader 使用 DistributedSampler，每个 epoch 调用 set_epoch，只让 rank 0 写日志和保存 checkpoint。同时要重新计算 global batch size、学习率和 scheduler steps。
```

### 问题 38：如何把一个教学 Transformer 改得更接近真实 LLM？

参考回答：

```text
可以从架构和工程两侧升级。架构上加入 RMSNorm、RoPE、SwiGLU、GQA/MQA、weight tying 和更合理初始化。工程上加入高效 attention、KV Cache、AMP、gradient checkpointing、DDP/FSDP、可靠训练循环、checkpoint 和 profiler。还要处理 tokenizer、chat template、loss mask 和评估。
```

def audit_keywords(required, answers):
    missing = {}
    covered_topics = 0
    for topic, keywords in required.items():
        answer = answers.get(topic, "")
        missing_keywords = [kw for kw in sorted(keywords) if kw.lower() not in answer.lower()]
        if not missing_keywords:
            covered_topics += 1
        else:
            missing[topic] = missing_keywords
    return covered_topics / len(required), missing


def detect_red_flags(answers):
    risky_phrases = ["一定不会", "完全不需要", "只要", "肯定最优", "不用验证"]
    hits = []
    for topic, answer in answers.items():
        for phrase in risky_phrases:
            if phrase in answer:
                hits.append((topic, phrase))
    return hits


topic_coverage, missing = audit_keywords(required_keywords, mock_answers)
red_flags = detect_red_flags(mock_answers)
rubric_counts = Counter()
for answer in mock_answers.values():
    lower = answer.lower()
    rubric_counts["shape_or_formula"] += int(any(x in lower for x in ["shape", "global batch", "q/k/v", "shift"]))
    rubric_counts["debug_path"] += int(any(x in lower for x in ["check", "checking", "profiler", "resume", "finite"]))
    rubric_counts["pitfall"] += int(any(x in lower for x in ["oom", "padding", "detach", "contiguous", "rank"]))

rubric = {
    key: round(value / len(mock_answers), 3)
    for key, value in sorted(rubric_counts.items())
}

    and formula_checks["shift_logits_shape"] == (14, 32)
    and formula_checks["ddp_avg_grad"] == 0.2,
    "debug_coverage": rubric["debug_path"] >= 0.55,
    "risk_free": not red_flags,
}

revision_plan = {
    topic: [f"add keyword: {kw}" for kw in kws]
    for topic, kws in missing.items()
}

print("topic_coverage=", round(topic_coverage, 3))
print("missing=", missing)
print("formula_checks=", formula_checks)
print("rubric=", rubric)
print("red_flags=", red_flags)
print("revision_plan=", revision_plan)

print("gate_pass=", all(gates.values()))

运行后应看到 `gate_pass=True`。如果 `missing` 不为空，不要机械补关键词，而要回到对应章节补三件事：机制、shape / 公式、排查路径。面试复盘的价值不在于给自己打高分，而是把“我觉得会了”变成可检查的证据。

## 10.12 本章总结

PyTorch 工程面试的核心不是背 API，而是能把机制、代码和排查路径连起来。回答时建议遵循：

1. 先说概念定义。
2. 再说关键机制。
3. 再说常见坑。
4. 最后说 debug 或工程实践。

本册已经覆盖 tensor、autograd、Module、DataLoader、训练循环、混合精度、分布式、debug/profiling 和 Transformer 组件实现。掌握这些内容后，你应该能阅读大多数 PyTorch 训练代码，手写核心组件，并在训练异常时给出清晰排查路径。
