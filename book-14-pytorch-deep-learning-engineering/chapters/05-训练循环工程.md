# 第五章：训练循环工程

训练循环是把模型、数据、损失函数和优化器连成一个可持续实验系统的地方。前面几章分别讲了 tensor、autograd、`nn.Module` 和 DataLoader，但真实训练不是把它们简单拼起来就结束。一个可靠的训练循环还要处理 device 搬运、loss 计算、反向传播、梯度裁剪、梯度累积、学习率调度、日志、验证、checkpoint、恢复训练、异常检测和可复现性。

很多新手训练脚本能跑，但不稳定：loss 记录不准、验证时忘记 `no_grad`、scheduler step 时机错、梯度累积没有缩放 loss、checkpoint 只保存模型没有保存 optimizer、恢复训练后学习率不对、训练中断无法续跑、NaN 出现后不知道从哪里查。本章目标是建立一套从最小训练 step 到工程化训练循环的完整框架。

## 0. 本讲资料边界与第二轮精修口径

本讲第二轮精修前，已核对 PyTorch 官方 `Optimizer.zero_grad`、`AdamW`、`LRScheduler` / `StepLR`、`clip_grad_norm_`、`no_grad`、`inference_mode`、`Module.train()` / `eval()`、模型保存加载教程和 checkpoint 相关文档口径。

本章聚焦大模型训练最常见的训练循环工程问题：forward、loss、backward、optimizer step、scheduler step、`zero_grad(set_to_none=True)`、梯度累积、梯度裁剪、语言模型 shift loss、batch 到 device、日志、验证循环、完整 checkpoint、恢复训练、随机状态和非有限 loss 检查。

本章不展开 AMP / GradScaler、FSDP / DeepSpeed / ZeRO、DDP 梯度同步、activation checkpointing、编译优化、生产级实验追踪平台或分布式 checkpoint shard。这些内容会分别放到混合精度、分布式训练、AI Infra 和训练平台章节中展开。

## 5.1 训练循环解决什么问题

一个训练循环的本质是反复执行：

1. 从 DataLoader 取一个 batch。
2. 把 batch 移到正确 device。
3. 执行 forward。
4. 计算 loss。
5. 执行 backward。
6. 更新参数。
7. 清理梯度。
8. 记录指标。
9. 定期验证和保存 checkpoint。

最小形式看起来很简单：

```python
for batch in train_loader:
    optimizer.zero_grad()
    outputs = model(batch["input_ids"])
    loss = criterion(outputs, batch["labels"])
    loss.backward()
    optimizer.step()
```

但真实工程会多出很多约束：

1. batch 里可能有多个字段。
2. 不同 tensor 要移动到 GPU。
3. loss 可能来自模型内部，也可能来自外部 criterion。
4. 需要梯度裁剪防止 loss spike。
5. 大 batch 可能要用梯度累积模拟。
6. 学习率 scheduler 要和 optimizer step 对齐。
7. 验证必须关闭梯度并切换 eval 模式。
8. checkpoint 要能恢复完整训练状态。

面试回答：

```text
训练循环不是只有 forward、backward 和 optimizer.step。工程上还要处理 batch 到 device、loss mask、梯度累积、梯度裁剪、scheduler、日志、验证、checkpoint 和恢复训练。一个好的训练循环应该能稳定训练、可观测、可复现，并且训练中断后能恢复到接近原来的状态。
```

## 5.1.1 关键公式与训练循环调试速查

第一，普通单 step 的参数更新可以抽象为：

```math
g_t=\nabla_{\theta_t} L_t(\theta_t),\qquad
\theta_{t+1}=U(\theta_t,g_t,\eta_t,s_t)
```

其中 `theta_t` 是第 `t` 次 optimizer step 前的参数，`g_t` 是当前 step 使用的梯度，`eta_t` 是学习率，`s_t` 是 optimizer 内部状态，例如 AdamW 的一阶矩和二阶矩。训练循环的目标是保证 `g_t` 来自正确 batch、正确 loss、正确 scale，并且 `s_t` 和 `eta_t` 能随 checkpoint 恢复。

第二，梯度累积时，真实 optimizer step 的梯度通常是多个 micro-batch 梯度的平均：

```math
g_t=\frac{1}{K}\sum_{k=1}^{K}\nabla_{\theta} L_{t,k}(\theta)
```

其中 `K` 是 `accum_steps`。代码里常见做法是每个 micro-step 使用 `loss / K` 再 `backward()`。如果不除以 `K`，梯度尺度会接近放大 `K` 倍。

第三，梯度裁剪常见口径是 global norm clipping：

```math
\hat{g}=
\begin{cases}
g, & \lVert g\rVert_2 \le c \\
g\cdot \frac{c}{\lVert g\rVert_2}, & \lVert g\rVert_2 > c
\end{cases}
```

其中 `c` 是 `max_norm`。PyTorch 的 `clip_grad_norm_` 会在已有梯度上原地裁剪，并返回裁剪前的总范数。

第四，按 optimizer step 更新的 scheduler 应跟随真实更新次数，而不是 micro-step 次数：

```math
\eta_t=S(t_{\mathrm{opt}})
```

其中 `t_opt` 是已经执行的 optimizer step 数。如果梯度累积 `K=4`，每个 micro-step 都调用 scheduler，学习率会比预期快 4 倍变化。

第五，语言模型训练常见 loss 是 shift 后的 token-level 平均：

```math
L=\frac{1}{N_{\mathrm{valid}}}\sum_{i=1}^{B}\sum_{j=1}^{T-1}
\mathbf{1}[y_{i,j+1}\ne -100]\cdot
\mathrm{CE}(p_{i,j},y_{i,j+1})
```

其中 `p_{i,j}` 是第 `j` 个位置预测下一个 token 的词表分布，`y_{i,j+1}` 是下一个位置的 label，`N_valid` 是未被 ignore 的有效 label 数。

第六，完整训练 checkpoint 可以抽象为：

```math
C_t=(\theta_t,s_t,h_t,e_t,r_t,c)
```

其中 `theta_t` 是模型权重，`s_t` 是 optimizer state，`h_t` 是 scheduler state，`e_t` 是 epoch / global step，`r_t` 是随机状态，`c` 是训练 config。只保存 `theta_t` 不能严格恢复训练轨迹。

## 5.2 最小训练 step

先看一个最小分类训练 step。

```python
import torch
from torch import nn


model = nn.Linear(10, 3)
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

x = torch.randn(32, 10)
labels = torch.randint(0, 3, (32,))

optimizer.zero_grad(set_to_none=True)
logits = model(x)
loss = criterion(logits, labels)
loss.backward()
optimizer.step()
```

关键顺序：

1. `zero_grad` 清掉旧梯度。
2. forward 得到 logits。
3. loss 连接 logits 和 labels。
4. `backward` 计算梯度。
5. `optimizer.step` 更新参数。

为什么通常先 `zero_grad` 再 forward？因为 PyTorch 梯度默认累积。只要确保每个 step 的 backward 前旧梯度被清掉，放在 step 末尾也能工作，但工程里更常见的是放在 forward 前，语义更清楚。

推荐：

```python
optimizer.zero_grad(set_to_none=True)
```

`set_to_none=True` 会把梯度设为 `None`，通常更省内存，性能也更好。

## 5.3 batch 到 device

模型和数据必须在同一个 device 上。

```python
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model = model.to(device)
```

训练循环中：

```python
def move_to_device(batch, device):
    out = {}
    for key, value in batch.items():
        if torch.is_tensor(value):
            out[key] = value.to(device, non_blocking=True)
        else:
            out[key] = value
    return out
```

使用：

```python
for batch in train_loader:
    batch = move_to_device(batch, device)
```

如果 DataLoader 使用 `pin_memory=True`，配合 `non_blocking=True` 通常可以让 CPU 到 GPU 的搬运更高效。

常见错误：

```text
Expected all tensors to be on the same device
```

排查时先打印：

```python
print(next(model.parameters()).device)
print(batch["input_ids"].device)
```

## 5.4 loss 在哪里计算

loss 可以在模型内部计算，也可以在训练循环外部计算。

### 5.4.1 模型外部计算 loss

```python
logits = model(input_ids)
loss = nn.functional.cross_entropy(
    logits.view(-1, logits.size(-1)),
    labels.view(-1),
)
```

优点：

1. 模型只负责 forward，职责清楚。
2. 便于复用同一个模型做不同任务。
3. loss 逻辑在 trainer 里更明显。

缺点：

1. 每个训练脚本都要写 loss 细节。
2. 对语言模型 shift logits/labels 容易重复出错。

### 5.4.2 模型内部计算 loss

```python
outputs = model(input_ids=batch["input_ids"], labels=batch["labels"])
loss = outputs["loss"]
```

优点：

1. 训练脚本更简洁。
2. loss shift、mask 等逻辑封装在模型里。
3. 接近 Hugging Face 风格。

缺点：

1. 模型和训练目标绑定更紧。
2. 多任务或研究实验中可能不够灵活。

工程建议：一个项目内保持一致。教学代码可以外部算 loss，便于理解；较完整的语言模型项目可以让 `forward(labels=...)` 返回 loss，降低训练循环复杂度。

## 5.5 语言模型 loss 的标准形态

因果语言模型通常用当前位置预测下一个 token。假设：

```text
logits: [B, T, V]
labels: [B, T]
```

需要错位：

```python
shift_logits = logits[:, :-1, :]
shift_labels = labels[:, 1:]
```

再展平：

```python
loss = nn.functional.cross_entropy(
    shift_logits.reshape(-1, shift_logits.size(-1)),
    shift_labels.reshape(-1),
    ignore_index=-100,
)
```

这里的关键点：

1. `logits[:, :-1, :]` 用第 `0` 到第 `T-2` 个位置预测后一个 token。
2. `labels[:, 1:]` 是目标 token。
3. padding 或不需要训练的位置应设为 `-100`。
4. `ignore_index=-100` 会跳过这些位置。

常见 bug：

1. 忘记 shift，导致模型预测当前位置 token。
2. labels padding 用了 pad token id 而不是 `-100`。
3. logits 和 labels 展平维度不一致。
4. attention_mask 和 loss mask 混淆。

## 5.6 梯度裁剪

大模型训练中，loss spike 和梯度爆炸并不少见。梯度裁剪是常见稳定手段。

```python
loss.backward()
grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()
```

顺序很重要：

1. 先 `backward()` 得到梯度。
2. 再 clip grad。
3. 最后 optimizer step。

如果使用混合精度和 GradScaler，需要先 unscale 再裁剪，这会在下一章详细展开。

记录梯度范数很有用：

```python
grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
logger["grad_norm"] = float(grad_norm)
```

如果训练突然发散，优先查看：

1. loss 是否突然变大。
2. grad norm 是否突然变大。
3. 学习率是否在异常区间。
4. batch 是否包含异常样本。

## 5.7 梯度累积

显存有限时，常用梯度累积模拟更大的 batch。

假设每次只能放 `micro_batch_size=4`，但希望等效 batch 是 16，就可以累积 4 次再 step。

```python
accum_steps = 4
optimizer.zero_grad(set_to_none=True)

for step, batch in enumerate(train_loader):
    outputs = model(**batch)
    loss = outputs["loss"]
    loss = loss / accum_steps
    loss.backward()

    if (step + 1) % accum_steps == 0:
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)
```

为什么要除以 `accum_steps`？如果不除，每次 backward 的梯度会直接相加，累积 4 次后梯度尺度约等于平均梯度的 4 倍，相当于放大学习率。

还要注意最后不足一个 accum window 的 batch。如果训练循环允许最后一步也更新，可以写：

```python
is_accum_step = (step + 1) % accum_steps == 0
is_last_step = (step + 1) == len(train_loader)

if is_accum_step or is_last_step:
    optimizer.step()
    optimizer.zero_grad(set_to_none=True)
```

## 5.8 optimizer 和 scheduler 的 step 时机

优化器更新参数，scheduler 更新学习率。

常见结构：

```python
loss.backward()
optimizer.step()
scheduler.step()
optimizer.zero_grad(set_to_none=True)
```

大多数按 step 更新的 scheduler，`scheduler.step()` 放在 `optimizer.step()` 后。某些特殊 scheduler 或旧版本示例可能不同，要看文档。

梯度累积时，scheduler 通常应该只在真实 optimizer step 后更新，而不是每个 micro-step 都更新。

```python
if should_step:
    optimizer.step()
    scheduler.step()
    optimizer.zero_grad(set_to_none=True)
```

常见错误：

1. 每个 micro-step 都 scheduler step，导致学习率衰减过快。
2. 恢复训练时没加载 scheduler state，学习率从头开始。
3. warmup steps 按 batch 数算错，没有考虑梯度累积。

## 5.9 日志记录：不要只看 loss

训练日志至少应包含：

1. step 或 epoch。
2. train loss。
3. learning rate。
4. grad norm。
5. tokens/s 或 samples/s。
6. GPU 显存。
7. validation loss 或任务指标。

最小日志：

```python
if global_step % log_every == 0:
    lr = optimizer.param_groups[0]["lr"]
    print({
        "step": global_step,
        "loss": loss.item(),
        "lr": lr,
    })
```

注意，梯度累积时 `loss.item()` 可能是除以 `accum_steps` 后的值。如果想记录原始 loss，可以在缩放前保存：

```python
raw_loss = outputs["loss"]
loss = raw_loss / accum_steps
```

不要把带计算图的 loss tensor 长期保存到 list 里：

```python
losses.append(loss)  # 风险：保留计算图
```

应保存数值：

```python
losses.append(loss.item())
```

## 5.10 validation 循环

验证循环和训练循环不同。验证时应该：

1. `model.eval()`。
2. 使用 `torch.no_grad()` 或 `torch.inference_mode()`。
3. 不调用 backward。
4. 不调用 optimizer step。
5. 验证结束后切回 `model.train()`。

示例：

```python
def evaluate(model, val_loader, device):
    model.eval()
    total_loss = 0.0
    total_count = 0

    with torch.inference_mode():
        for batch in val_loader:
            batch = move_to_device(batch, device)
            outputs = model(**batch)
            loss = outputs["loss"]

            batch_size = batch["input_ids"].size(0)
            total_loss += loss.item() * batch_size
            total_count += batch_size

    model.train()
    return total_loss / max(total_count, 1)
```

为什么用 batch size 加权？因为最后一个 batch 可能更小，直接平均每个 batch 的 loss 会给小 batch 过高权重。

如果是语言模型，更严谨的做法是按有效 token 数加权，而不是按 batch size 加权：

```python
valid_tokens = (batch["labels"] != -100).sum().item()
```

## 5.11 checkpoint 保存什么

一个完整训练 checkpoint 不应只保存模型权重。

推荐保存：

```python
checkpoint = {
    "model": model.state_dict(),
    "optimizer": optimizer.state_dict(),
    "scheduler": scheduler.state_dict() if scheduler is not None else None,
    "epoch": epoch,
    "global_step": global_step,
    "config": config,
    "rng_state": torch.get_rng_state(),
}

torch.save(checkpoint, "checkpoint.pt")
```

如果使用 CUDA，还可以保存：

```python
checkpoint["cuda_rng_state"] = torch.cuda.get_rng_state_all()
```

完整保存的好处：

1. 参数能恢复。
2. optimizer 动量和二阶矩能恢复。
3. scheduler 学习率位置能恢复。
4. step 和 epoch 能恢复。
5. 随机状态尽量一致。

如果只保存模型权重，恢复后虽然能继续训练，但 optimizer 和 scheduler 状态丢失，训练轨迹会改变，严重时会导致 loss spike。

## 5.12 checkpoint 加载和恢复训练

恢复训练：

```python
checkpoint = torch.load("checkpoint.pt", map_location="cpu")

model.load_state_dict(checkpoint["model"])
optimizer.load_state_dict(checkpoint["optimizer"])

if scheduler is not None and checkpoint["scheduler"] is not None:
    scheduler.load_state_dict(checkpoint["scheduler"])

start_epoch = checkpoint["epoch"]
global_step = checkpoint["global_step"]

torch.set_rng_state(checkpoint["rng_state"])
```

如果 optimizer state 加载后在 CPU，而模型在 GPU，通常 optimizer 会在下一次 step 时处理，但更稳妥的做法是确保模型先移动到 device，再加载或迁移 optimizer state。真实项目中要根据框架行为检查。

推理只需要模型：

```python
state = torch.load("model.pt", map_location="cpu")
model.load_state_dict(state)
model.eval()
```

训练恢复和推理加载是两个不同目标，不要混在一起。

## 5.13 保存 best checkpoint 和 last checkpoint

工程里常保存两类 checkpoint：

1. `last.pt`：最近一次状态，便于中断恢复。
2. `best.pt`：验证指标最好的模型，便于最终评估或上线。

示例：

```python
best_val_loss = float("inf")

val_loss = evaluate(model, val_loader, device)
save_checkpoint("last.pt")

if val_loss < best_val_loss:
    best_val_loss = val_loss
    save_checkpoint("best.pt")
```

注意：

1. `best.pt` 如果用于继续训练，也要保存完整训练状态。
2. 如果只用于推理，可以只保存模型权重。
3. 文件名里可以带 step，便于回滚。

## 5.14 可复现性

可复现性不是绝对的，但可以尽量提高。

常见设置：

```python
import random
import numpy as np
import torch


def set_seed(seed):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)
```

还可以设置：

```python
torch.backends.cudnn.benchmark = False
torch.backends.cudnn.deterministic = True
```

但要知道：

1. 完全确定性可能降低性能。
2. 某些 CUDA 算子本身可能非确定。
3. 分布式训练、DataLoader 多 worker、混合精度都会增加复现难度。
4. 面试中更重要的是知道哪些因素会影响复现。

## 5.15 训练中的异常检测

训练循环要主动检查异常。

### 5.15.1 NaN 和 Inf

```python
if not torch.isfinite(loss):
    print("non-finite loss", loss.item())
    break
```

检查梯度：

```python
for name, param in model.named_parameters():
    if param.grad is not None and not torch.isfinite(param.grad).all():
        print("bad grad", name)
```

### 5.15.2 loss spike

如果 loss 突然变大，排查顺序：

1. 当前 batch 是否异常。
2. 学习率是否过大或 warmup 配置错误。
3. grad norm 是否突然变大。
4. 混合精度是否溢出。
5. mask 是否出现全忽略或全 `-inf`。

### 5.15.3 OOM

如果出现 CUDA OOM：

1. 减小 batch size。
2. 减小 sequence length。
3. 使用梯度累积。
4. 使用混合精度。
5. 使用 gradient checkpointing。
6. 检查是否保存了带图 tensor。

## 5.16 一个工程化训练循环骨架

下面给出一个较完整但仍然可读的训练循环。

```python
def train(
    model,
    train_loader,
    val_loader,
    optimizer,
    scheduler,
    device,
    num_epochs,
    accum_steps=1,
    log_every=10,
    eval_every=100,
):
    model.to(device)
    model.train()

    global_step = 0
    optimizer.zero_grad(set_to_none=True)

    for epoch in range(num_epochs):
        for micro_step, batch in enumerate(train_loader):
            batch = move_to_device(batch, device)

            outputs = model(**batch)
            raw_loss = outputs["loss"]
            loss = raw_loss / accum_steps

            if not torch.isfinite(raw_loss):
                raise RuntimeError(f"non-finite loss: {raw_loss.item()}")

            loss.backward()

            should_step = (micro_step + 1) % accum_steps == 0
            is_last = (micro_step + 1) == len(train_loader)

            if should_step or is_last:
                grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
                optimizer.step()
                if scheduler is not None:
                    scheduler.step()
                optimizer.zero_grad(set_to_none=True)

                global_step += 1

                if global_step % log_every == 0:
                    lr = optimizer.param_groups[0]["lr"]
                    print({
                        "epoch": epoch,
                        "step": global_step,
                        "loss": raw_loss.item(),
                        "lr": lr,
                        "grad_norm": float(grad_norm),
                    })

                if val_loader is not None and global_step % eval_every == 0:
                    val_loss = evaluate(model, val_loader, device)
                    print({"step": global_step, "val_loss": val_loss})
                    model.train()
```

这个骨架覆盖了：

1. device 搬运。
2. loss 缩放。
3. 梯度累积。
4. 非有限 loss 检查。
5. 梯度裁剪。
6. optimizer 和 scheduler step。
7. 日志和验证。

真实项目还会继续加入 checkpoint 保存、分布式同步、混合精度、进度条、实验追踪、异常恢复和配置管理。

## 5.17 最小可运行训练循环审计 demo

下面这个 demo 用一个极小 causal LM 串起训练循环的关键动作：shift loss、`ignore_index=-100`、梯度累积、梯度裁剪、scheduler 跟随 optimizer step、token 加权验证、完整 checkpoint 和恢复验证。

```python
import math
import tempfile
import warnings
from pathlib import Path

import torch
import torch.nn.functional as F
from torch import nn

warnings.filterwarnings("ignore", message="CUDA initialization.*")
IGNORE_INDEX = -100


def set_seed(seed=0):
    torch.manual_seed(seed)


class TinyCausalLM(nn.Module):
    def __init__(self, vocab_size=16, hidden_size=12):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, hidden_size)
        self.lm_head = nn.Linear(hidden_size, vocab_size)

    def forward(self, input_ids, labels=None):
        hidden = self.embed(input_ids)
        logits = self.lm_head(hidden)
        out = {"logits": logits}
        if labels is not None:
            shift_logits = logits[:, :-1, :].reshape(-1, logits.size(-1))
            shift_labels = labels[:, 1:].reshape(-1)
            out["loss"] = F.cross_entropy(
                shift_logits,
                shift_labels,
                ignore_index=IGNORE_INDEX,
            )
        return out


def make_batches():
    batches = []
    raw = [
        [[1, 2, 3, 4, 5, 6], [1, 2, 3, 4, 5, 0]],
        [[2, 3, 4, 5, 6, 7], [2, 3, 4, 5, 0, 0]],
        [[3, 4, 5, 6, 7, 8], [3, 4, 5, 0, 0, 0]],
        [[4, 5, 6, 7, 8, 9], [4, 5, 0, 0, 0, 0]],
    ]
    for pair in raw:
        input_ids = torch.tensor(pair, dtype=torch.long)
        labels = input_ids.clone()
        labels[input_ids == 0] = IGNORE_INDEX
        attention_mask = input_ids.ne(0)
        batches.append({
            "input_ids": input_ids,
            "labels": labels,
            "attention_mask": attention_mask,
        })
    return batches


def move_to_device(batch, device):
    return {k: v.to(device) if torch.is_tensor(v) else v for k, v in batch.items()}


def evaluate(model, batches, device):
    was_training = model.training
    model.eval()
    token_loss_sum = 0.0
    token_count = 0
    with torch.inference_mode():
        for batch in batches:
            batch = move_to_device(batch, device)
            out = model(batch["input_ids"], labels=batch["labels"])
            valid = int(batch["labels"][:, 1:].ne(IGNORE_INDEX).sum().item())
            token_loss_sum += out["loss"].item() * valid
            token_count += valid
    if was_training:
        model.train()
    return token_loss_sum / max(token_count, 1)


def train_one_pass(model, batches, optimizer, scheduler, device, accum_steps=2):
    model.train()
    optimizer.zero_grad(set_to_none=True)
    optimizer_steps = 0
    raw_losses = []
    grad_norms = []

    for micro_step, batch in enumerate(batches):
        batch = move_to_device(batch, device)
        out = model(batch["input_ids"], labels=batch["labels"])
        raw_loss = out["loss"]
        if not torch.isfinite(raw_loss):
            raise RuntimeError("non-finite loss")

        (raw_loss / accum_steps).backward()
        raw_losses.append(round(raw_loss.item(), 4))

        should_step = (micro_step + 1) % accum_steps == 0
        is_last = (micro_step + 1) == len(batches)
        if should_step or is_last:
            grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=0.5)
            optimizer.step()
            scheduler.step()
            optimizer.zero_grad(set_to_none=True)
            optimizer_steps += 1
            grad_norms.append(round(float(grad_norm), 4))

    return raw_losses, grad_norms, optimizer_steps


set_seed(123)
device = torch.device("cpu")
model = TinyCausalLM().to(device)
optimizer = torch.optim.AdamW(model.parameters(), lr=0.08, weight_decay=0.01)
scheduler = torch.optim.lr_scheduler.StepLR(optimizer, step_size=1, gamma=0.8)
batches = make_batches()

initial_val = evaluate(model, batches, device)
losses, grad_norms, optimizer_steps = train_one_pass(
    model, batches, optimizer, scheduler, device, accum_steps=2
)
final_val = evaluate(model, batches, device)

with tempfile.TemporaryDirectory() as tmpdir:
    ckpt_path = Path(tmpdir) / "last.pt"
    torch.save({
        "model": model.state_dict(),
        "optimizer": optimizer.state_dict(),
        "scheduler": scheduler.state_dict(),
        "global_step": optimizer_steps,
        "rng_state": torch.get_rng_state(),
    }, ckpt_path)

    restored = TinyCausalLM().to(device)
    restored_optimizer = torch.optim.AdamW(restored.parameters(), lr=0.08, weight_decay=0.01)
    restored_scheduler = torch.optim.lr_scheduler.StepLR(restored_optimizer, step_size=1, gamma=0.8)
    checkpoint = torch.load(ckpt_path, map_location="cpu", weights_only=False)
    restored.load_state_dict(checkpoint["model"])
    restored_optimizer.load_state_dict(checkpoint["optimizer"])
    restored_scheduler.load_state_dict(checkpoint["scheduler"])
    torch.set_rng_state(checkpoint["rng_state"])
    restored_val = evaluate(restored, batches, device)

print("initial_val_loss=", round(initial_val, 4))
print("micro_losses=", losses)
print("grad_norms=", grad_norms)
print("optimizer_steps=", optimizer_steps)
print("scheduler_last_epoch=", scheduler.last_epoch)
print("final_lr=", round(optimizer.param_groups[0]["lr"], 5))
print("final_val_loss=", round(final_val, 4))
print("loss_decreased=", final_val < initial_val)
print("restored_val_matches=", math.isclose(final_val, restored_val, rel_tol=0, abs_tol=1e-7))
print("restored_global_step=", checkpoint["global_step"])
print("model_training_after_eval=", model.training)
```

期望输出类似：

```text
initial_val_loss= 2.7991
micro_losses= [2.7366, 2.5374, 1.8882, 2.2651]
grad_norms= [1.7233, 1.478]
optimizer_steps= 2
scheduler_last_epoch= 2
final_lr= 0.0512
final_val_loss= 1.1386
loss_decreased= True
restored_val_matches= True
restored_global_step= 2
model_training_after_eval= True
```

这段输出要检查几件事：

1. `micro_losses` 有 4 个值，但 `optimizer_steps=2`，说明 `accum_steps=2` 生效。
2. `scheduler_last_epoch=2`，说明 scheduler 只在真实 optimizer step 后更新。
3. `final_lr=0.0512`，对应 `0.08 * 0.8^2`。
4. `loss_decreased=True`，说明训练循环确实更新了参数。
5. `restored_val_matches=True`，说明模型、optimizer、scheduler 和随机状态可以从 checkpoint 恢复到一致验证结果。
6. `model_training_after_eval=True`，说明验证函数结束后恢复了原本的训练模式。

## 5.18 训练循环常见 bug

### 5.18.1 loss 不下降

排查顺序：

1. 数据和 labels 是否正确。
2. loss 是否真的依赖模型参数。
3. 参数是否进入 optimizer。
4. 学习率是否过大或过小。
5. 梯度是否为 None、0、NaN 或 Inf。
6. 是否忘记 `optimizer.step()`。
7. 是否在 forward 中误用了 `no_grad()` 或 `detach()`。

### 5.18.2 训练速度很慢

可能原因：

1. DataLoader 太慢。
2. batch 到 device 阻塞。
3. 频繁打印日志。
4. 验证太频繁。
5. 保存 checkpoint 太频繁。
6. Python 逻辑太重。

### 5.18.3 恢复训练后 loss spike

可能原因：

1. 没加载 optimizer state。
2. 没加载 scheduler state。
3. 学习率从初始值重新开始。
4. 混合精度 scaler 状态丢失。
5. 数据顺序或随机状态变化。

### 5.18.4 验证结果不稳定

可能原因：

1. 忘记 `model.eval()`。
2. 忘记 `torch.no_grad()` 或 `inference_mode()`。
3. 验证集 shuffle 或采样不稳定。
4. 生成评估使用了随机采样。
5. 指标聚合方式错误。

## 5.19 面试官会怎么问

### 问题一：一个标准训练 step 包含哪些步骤？

回答模板：

```text
通常包括 zero_grad、forward、loss 计算、backward、可选梯度裁剪、optimizer.step、scheduler.step 和日志记录。顺序上要先 backward 得到梯度，再裁剪梯度，最后更新参数。梯度默认累积，所以每个 step 要清梯度。
```

### 问题二：梯度累积为什么要把 loss 除以 accum_steps？

回答模板：

```text
因为 PyTorch 梯度默认累积。如果累积 N 个 micro-batch 但不缩放 loss，最终梯度尺度会接近平均梯度的 N 倍，相当于放大学习率。把 loss 除以 accum_steps 后，累积梯度更接近真实大 batch 的平均梯度。
```

### 问题三：验证循环和训练循环有什么区别？

回答模板：

```text
验证时要 model.eval()，并使用 no_grad 或 inference_mode 关闭梯度，不执行 backward 和 optimizer.step。验证结束后要切回 model.train()。训练循环则需要保留计算图、反向传播并更新参数。
```

### 问题四：checkpoint 应该保存什么？

回答模板：

```text
如果只是推理，保存模型 state_dict 就够了；如果要恢复训练，应该保存 model、optimizer、scheduler、epoch、global_step、config 和随机状态。否则恢复后学习率、动量、二阶矩等状态丢失，训练轨迹会变化，甚至出现 loss spike。
```

### 问题五：训练 loss 出现 NaN 怎么排查？

回答模板：

```text
先确认是 loss 本身 NaN 还是梯度 NaN，再看当前 batch、学习率、grad norm、混合精度 loss scaling、mask 是否出现全 -inf、输入是否有异常值。工程上我会记录 loss、lr、grad norm，并在非有限 loss 出现时保存问题 batch 或中断排查。
```

## 5.20 常见误区

1. 以为训练循环只要能跑就行。工程训练循环必须可观测、可恢复、可排查。
2. 忘记 `optimizer.zero_grad()`，导致梯度跨 step 累积。
3. 梯度累积时没有缩放 loss。
4. 每个 micro-step 都 scheduler step，导致学习率变化过快。
5. 验证时只写 `model.eval()`，忘记关闭梯度。
6. checkpoint 只保存模型，不保存 optimizer 和 scheduler。
7. 日志里保存带图 tensor，导致显存增长。
8. 只看平均 loss，不看 grad norm、lr、tokens/s 和验证指标。
9. 恢复训练后不检查学习率，导致训练轨迹异常。

## 5.22 本章总结

训练循环是深度学习工程的主干。一个合格的训练循环不仅要完成 forward、loss、backward 和 optimizer step，还要处理 device、梯度累积、梯度裁剪、scheduler、日志、验证、checkpoint、恢复训练和异常检测。

需要记住的主线：

1. 每个 step 都要明确清梯度、前向、反向和更新参数的顺序。
2. 语言模型 loss 要正确 shift，并用 `ignore_index` 忽略不该训练的位置。
3. 梯度裁剪放在 backward 后、optimizer step 前。
4. 梯度累积要缩放 loss，并且 scheduler 通常只跟随真实 optimizer step。
5. 验证需要 `eval()` 和 `no_grad` 或 `inference_mode()`。
6. 完整 checkpoint 应保存模型、optimizer、scheduler、step、config 和随机状态。
7. 稳定训练依赖可观测性：loss、lr、grad norm、吞吐、显存和验证指标都要看。
