# 第七章：分布式训练入门

单卡训练能帮助你理解模型和训练循环，但大模型工程很快会遇到单卡装不下、训练太慢、数据太多的问题。分布式训练的目标是把计算、数据、参数、梯度或优化器状态拆到多张 GPU、多台机器上，让训练规模和速度突破单卡限制。

本章是入门章节，不追求覆盖所有 Megatron、DeepSpeed、ZeRO、FSDP 细节，而是先建立分布式训练的核心概念：rank、world size、process group、DDP、all-reduce、DistributedSampler、torchrun、FSDP 的基本直觉，以及常见 debug 方法。只要这些概念清楚，后续学习训练系统和大模型训练框架会容易很多。

## 0. 本讲资料边界与第二轮精修口径

本讲第二轮精修前，已核对 PyTorch 官方 `torch.distributed`、`DistributedDataParallel`、`torchrun` / elastic launch、`DistributedSampler`、collective communication、`all_reduce` 和 Fully Sharded Data Parallel 文档口径。

本章聚焦 PyTorch 分布式训练入门中最常用、最容易在面试中被追问的内容：rank / local rank / world size、process group、DDP 梯度同步、all-reduce、torchrun 环境变量、DistributedSampler 数据切分、global batch size、DDP 与 DataParallel / FSDP 的边界、`no_sync()`、rank 0 checkpoint 和分布式 debug。

本章不展开 ZeRO / DeepSpeed、Megatron-LM、tensor parallel、pipeline parallel、sequence parallel、MoE expert parallel、NCCL 拓扑调优、RDMA / InfiniBand、分布式 checkpoint shard 格式或生产级集群调度。这些内容会在大模型训练系统、AI Infra 和性能优化章节中继续展开。

## 7.1 为什么需要分布式训练

单卡训练有两个主要限制：

1. 显存不够。
2. 速度不够。

显存不够时，一个 GPU 放不下模型参数、梯度、优化器状态和激活。速度不够时，即使能放下模型，训练一个 epoch 或一轮预训练也可能太慢。

分布式训练试图解决这些问题：

1. 数据并行：每张卡放一份模型，处理不同 batch，最后同步梯度。
2. 参数分片：把参数、梯度或优化器状态拆到不同卡上。
3. 张量并行：把单层大矩阵计算拆到多卡。
4. 流水线并行：把模型层切到多卡，像流水线一样执行。

本章重点先讲数据并行和 DDP，因为它是最常用、最容易入门的分布式训练方式。

面试回答：

```text
分布式训练主要解决单卡显存和速度限制。最基础的是数据并行，每张 GPU 持有一份模型，处理不同 mini-batch，反向传播后通过 all-reduce 同步梯度。更大规模时会进一步使用参数分片、张量并行和流水线并行来降低单卡显存和提升训练吞吐。
```

## 7.1.1 关键公式与分布式调试速查

第一，`rank`、`local_rank` 和机器编号的关系可以粗略写成：

```math
r = n_{\mathrm{node}} G + r_{\mathrm{local}}
```

其中 `r` 是全局 rank，`n_node` 是机器编号，`G` 是每台机器的 GPU 数，`r_local` 是本机内 GPU 或进程编号。`world_size=W` 表示总进程数。

第二，DDP 数据并行的梯度同步可以写成：

```math
\bar{g}_t=\frac{1}{W}\sum_{r=0}^{W-1} g_{t,r}
```

其中 `g_t,r` 是第 `r` 个 rank 在第 `t` 次更新前得到的本地梯度，`\bar{g}_t` 是 all-reduce 后每个 rank 都拿到的平均梯度。

第三，每个 rank 用同一个平均梯度更新参数：

```math
\theta_{t+1}=\theta_t-\eta \bar{g}_t
```

因此只要初始参数一致、梯度同步一致、optimizer step 顺序一致，各 rank 的参数会保持一致。

第四，global batch size 是：

```math
B_{\mathrm{global}}=B_{\mathrm{device}}\cdot W\cdot A
```

其中 `B_device` 是每个 rank 的 micro-batch size，`W` 是 world size，`A` 是 gradient accumulation steps。从单卡切到多卡时，如果每卡 batch 不变，global batch 会变大，学习率、warmup、训练步数和日志统计都要重新核对。

第五，DistributedSampler 的目标是让每个 rank 看到不同索引子集：

```math
I_r \cap I_s=\varnothing,\quad r\ne s
```

在不丢样本的普通 map-style dataset 场景中，希望所有 rank 的索引集合覆盖当前 epoch 的训练样本。真实实现还要处理 shuffle、补齐或 `drop_last`。

第六，DDP 单卡状态显存不因 world size 自动下降：

```math
M_{\mathrm{DDP,rank}}\approx M_{\mathrm{param}}+M_{\mathrm{grad}}+M_{\mathrm{opt}}+M_{\mathrm{act}}
```

DDP 主要提升吞吐；如果单卡装不下完整参数、梯度和 optimizer state，需要进一步考虑 FSDP / ZeRO 等分片方法。

第七，梯度累积配合 `no_sync()` 的通信次数可以近似理解为：

```math
C_{\mathrm{sync,no\_sync}}\approx \frac{C_{\mathrm{sync,naive}}}{A}
```

也就是 `A` 个 micro-step 只在最后一次同步，而不是每个 micro-step 都 all-reduce。

## 7.2 rank、local rank 和 world size

分布式训练里最先要理解几个概念。

### 7.2.1 world size

`world_size` 表示总进程数。通常一个 GPU 对应一个训练进程。

如果一台机器有 8 张 GPU，并且每张 GPU 一个进程：

```text
world_size = 8
```

如果 2 台机器，每台 8 张 GPU：

```text
world_size = 16
```

### 7.2.2 rank

`rank` 是全局进程编号，范围是：

```text
0, 1, ..., world_size - 1
```

通常 `rank=0` 负责打印日志、保存 checkpoint、汇总指标等，避免所有进程重复写文件。

### 7.2.3 local rank

`local_rank` 是当前机器内的 GPU 编号。

单机 8 卡时，local rank 通常是：

```text
0, 1, ..., 7
```

训练时通常用它设置当前进程使用哪张 GPU：

```python
torch.cuda.set_device(local_rank)
device = torch.device("cuda", local_rank)
```

核心区别：

1. `rank` 是全局编号。
2. `local_rank` 是本机编号。
3. `world_size` 是总进程数。

## 7.3 process group 是什么

process group 可以理解为一组参与通信的进程。

初始化分布式环境：

```python
import torch.distributed as dist

dist.init_process_group(backend="nccl")
```

常见 backend：

1. `nccl`：GPU 训练最常用。
2. `gloo`：CPU 或某些调试场景常用。

初始化后，进程之间就可以执行通信操作，例如 all-reduce、broadcast、barrier。

训练结束时通常销毁：

```python
dist.destroy_process_group()
```

## 7.4 数据并行的核心直觉

数据并行的流程：

1. 每张 GPU 上放一份相同模型。
2. 每张 GPU 处理不同 mini-batch。
3. 每张 GPU 独立 forward 和 backward。
4. backward 后每张 GPU 得到一份梯度。
5. 通过 all-reduce 对梯度求平均。
6. 每张 GPU 用相同平均梯度更新参数。

这样更新后，每张 GPU 的模型参数仍然保持一致。

如果没有梯度同步，每张 GPU 会基于不同数据更新出不同模型，训练就不再是一个统一模型。

## 7.5 all-reduce 是什么

all-reduce 是分布式训练中最核心的通信操作之一。

假设有 4 张 GPU，每张 GPU 上某个参数的梯度分别是：

```text
g0, g1, g2, g3
```

all-reduce sum 后，每张 GPU 都得到：

```text
g0 + g1 + g2 + g3
```

如果要平均梯度，就再除以 `world_size`。

DDP 内部会自动处理这个过程。用户通常不需要手写 all-reduce 梯度，但要理解它的代价：通信开销会随着参数量、网络带宽、GPU 拓扑和同步频率变化。

面试回答：

```text
all-reduce 会把每个进程上的张量做归约，并把归约结果返回给所有进程。DDP 中每张卡 backward 后得到本地梯度，通过 all-reduce 求和或平均后，每张卡拿到一致的梯度，再执行 optimizer step，所以参数保持同步。
```

## 7.6 DDP 的基本用法

PyTorch 推荐使用 DistributedDataParallel，也就是 DDP。

基本结构：

```python
import os
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP


def setup_distributed():
    dist.init_process_group(backend="nccl")
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)
    return local_rank


local_rank = setup_distributed()
device = torch.device("cuda", local_rank)

model = MyModel().to(device)
model = DDP(model, device_ids=[local_rank])
```

训练循环中：

```python
for batch in train_loader:
    batch = move_to_device(batch, device)
    optimizer.zero_grad(set_to_none=True)
    outputs = model(**batch)
    loss = outputs["loss"]
    loss.backward()
    optimizer.step()
```

DDP 会在 backward 过程中自动同步梯度。

保存 checkpoint 时常用：

```python
if dist.get_rank() == 0:
    torch.save(model.module.state_dict(), "model.pt")
```

因为 DDP 包装后的原始模型在 `model.module` 里。

## 7.7 torchrun 启动训练

现代 PyTorch 常用 `torchrun` 启动分布式训练。

单机 4 卡：

```bash
torchrun --nproc_per_node=4 train.py
```

`torchrun` 会为每个进程设置环境变量，例如：

1. `RANK`
2. `LOCAL_RANK`
3. `WORLD_SIZE`

训练脚本通过这些变量初始化分布式环境。

读取示例：

```python
rank = int(os.environ["RANK"])
local_rank = int(os.environ["LOCAL_RANK"])
world_size = int(os.environ["WORLD_SIZE"])
```

不要手动让多个进程抢同一张 GPU。最常见的正确方式是：每个进程绑定一个 `local_rank` 对应的 GPU。

## 7.8 DistributedSampler

DDP 里每个进程应该看到不同数据子集。否则每张卡都训练同样样本，就浪费了多卡。

使用 `DistributedSampler`：

```python
from torch.utils.data import DataLoader, DistributedSampler

sampler = DistributedSampler(dataset, shuffle=True)
loader = DataLoader(
    dataset,
    batch_size=batch_size,
    sampler=sampler,
    num_workers=4,
)
```

注意：使用 sampler 时通常不要再设置 `shuffle=True` 给 DataLoader，因为 shuffle 由 sampler 管理。

每个 epoch 要调用：

```python
sampler.set_epoch(epoch)
```

否则每个 epoch 的 shuffle 顺序可能一样。

完整结构：

```python
for epoch in range(num_epochs):
    sampler.set_epoch(epoch)
    for batch in loader:
        ...
```

## 7.9 global batch size 怎么算

分布式训练里要区分：

1. 每卡 batch size。
2. world size。
3. gradient accumulation steps。
4. global batch size。

公式：

```text
global_batch_size = per_device_batch_size * world_size * accumulation_steps
```

例如：

```text
per_device_batch_size = 4
world_size = 8
accumulation_steps = 2
global_batch_size = 4 * 8 * 2 = 64
```

这会影响学习率、warmup steps、训练步数和日志统计。

常见错误：从单卡切到 8 卡后，没有意识到 global batch size 变大，导致优化行为变化。

## 7.10 DDP 和 DataParallel 的区别

PyTorch 早期有 `DataParallel`，但现在训练更推荐 DDP。

区别：

1. `DataParallel` 通常单进程多线程，主卡负担重。
2. DDP 是多进程，每个 GPU 一个进程，性能和扩展性更好。
3. DDP 梯度同步更高效。
4. DDP 更适合多机多卡。

面试中可以简单说：真实训练优先用 DDP，不建议用 DataParallel 做严肃多卡训练。

## 7.11 DDP 的通信时机

DDP 不是等所有 backward 完成后才一次性同步全部梯度。它会把参数分成 bucket，并在某些梯度 ready 后触发通信。

这样可以让通信和反向计算重叠：

1. 后面层的梯度先算出来。
2. DDP 开始同步这些梯度 bucket。
3. 同时 backward 继续计算前面层梯度。

这叫 communication overlap。

理解这个机制有助于解释：

1. 为什么 DDP 比朴素手写同步更高效。
2. 为什么参数使用路径不一致会导致卡住。
3. 为什么梯度同步与 autograd hook 有关。

## 7.12 梯度累积下的 no_sync

梯度累积时，如果每个 micro-step 都同步梯度，会浪费通信。

DDP 提供 `no_sync()`：

```python
for step, batch in enumerate(loader):
    should_sync = (step + 1) % accum_steps == 0

    if should_sync:
        context = nullcontext()
    else:
        context = model.no_sync()

    with context:
        outputs = model(**batch)
        loss = outputs["loss"] / accum_steps
        loss.backward()

    if should_sync:
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)
```

这里 `no_sync()` 会跳过当前 micro-step 的梯度同步，只在真正更新前同步一次。

需要导入：

```python
from contextlib import nullcontext
```

## 7.13 FSDP 的基本直觉

DDP 每张卡都有一份完整模型参数、梯度和优化器状态。模型变大后，单卡显存仍然可能不够。

FSDP 的直觉是把参数、梯度和优化器状态分片到不同 GPU 上。

可以这样理解：

1. DDP：每张卡都有完整模型副本。
2. FSDP：每张卡只长期持有一部分参数状态，需要计算某层时再临时 gather。

优点：

1. 降低单卡显存。
2. 支持更大模型。

代价：

1. 通信更复杂。
2. 对模块边界更敏感。
3. checkpoint、调试和性能优化更复杂。

本章只要求建立直觉：DDP 主要解决速度扩展，FSDP 进一步解决大模型显存分片问题。

## 7.14 分布式 checkpoint

DDP 场景通常只让 `rank=0` 保存：

```python
if dist.get_rank() == 0:
    torch.save({
        "model": model.module.state_dict(),
        "optimizer": optimizer.state_dict(),
        "step": global_step,
    }, "checkpoint.pt")
```

原因：所有 rank 的模型参数同步后相同，没必要每个 rank 都写一份。

但 FSDP 或分片训练场景下，checkpoint 可能是 sharded checkpoint，保存和加载要复杂很多。入门阶段先掌握 DDP 的保存方式。

注意：只让 rank 0 写日志和 checkpoint，避免多个进程同时写同一个文件。

## 7.15 分布式训练常见 bug

### 7.15.1 程序卡住

常见原因：

1. 某些 rank 没走到同一个 collective 操作。
2. 条件分支导致不同 rank 执行路径不同。
3. 某些参数在部分 rank 没参与 forward。
4. 数据集长度或 batch 数在不同 rank 不一致。
5. 网络或 NCCL 通信异常。

### 7.15.2 某些参数没有梯度

常见原因：

1. 模型分支没被当前 batch 使用。
2. 参数 `requires_grad=False`。
3. loss 不依赖该参数。
4. 误用了 `detach()`。

DDP 中如果存在未使用参数，可以尝试设置：

```python
DDP(model, device_ids=[local_rank], find_unused_parameters=True)
```

但这通常有额外开销，不应作为长期默认修复。更好的方式是理解为什么参数未使用。

### 7.15.3 每张卡数据重复

常见原因：

1. 没用 DistributedSampler。
2. sampler 配置错。
3. iterable dataset 没按 rank 切分。

### 7.15.4 只有 rank 0 正常，其他 rank 报错

可能是：

1. 只有 rank 0 创建了某些文件或目录。
2. 数据路径在不同机器不可见。
3. 某些初始化逻辑只在 rank 0 执行，但其他 rank 也依赖结果。

## 7.16 分布式 debug 清单

排查分布式问题时，先打印基础信息：

```python
print(
    "rank", dist.get_rank(),
    "world_size", dist.get_world_size(),
    "device", torch.cuda.current_device(),
)
```

检查：

1. 每个 rank 是否绑定不同 GPU。
2. 每个 rank 的 batch shape 是否合理。
3. 每个 rank 是否都进入训练循环。
4. 每个 rank 的 loss 是否有限。
5. 是否所有 rank 都参与同样 collective。
6. sampler 是否 set_epoch。
7. 只有 rank 0 是否写 checkpoint。

有时可以加 barrier 定位卡住位置：

```python
dist.barrier()
```

但不要滥用 barrier。barrier 可以帮助定位同步点，也可能掩盖真实性能问题。

## 7.17 一个最小 DDP 训练骨架

```python
import os
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler


def setup():
    dist.init_process_group(backend="nccl")
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)
    return local_rank


def cleanup():
    dist.destroy_process_group()


def main():
    local_rank = setup()
    device = torch.device("cuda", local_rank)

    dataset = MyDataset()
    sampler = DistributedSampler(dataset, shuffle=True)
    loader = DataLoader(dataset, batch_size=8, sampler=sampler, num_workers=4)

    model = MyModel().to(device)
    model = DDP(model, device_ids=[local_rank])
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

    for epoch in range(3):
        sampler.set_epoch(epoch)
        model.train()

        for batch in loader:
            batch = move_to_device(batch, device)
            optimizer.zero_grad(set_to_none=True)
            outputs = model(**batch)
            loss = outputs["loss"]
            loss.backward()
            optimizer.step()

        if dist.get_rank() == 0:
            torch.save(model.module.state_dict(), "model.pt")

    cleanup()


if __name__ == "__main__":
    main()
```

启动：

```bash
torchrun --nproc_per_node=4 train.py
```

这个骨架包含了 DDP 的最小关键点：初始化、local rank、DistributedSampler、DDP 包装、rank 0 保存和 cleanup。

## 7.18 最小可运行分布式训练审计 demo

真实 DDP 需要多进程和 GPU 环境，下面这个 demo 用纯 Python 模拟分布式训练中最重要的检查点：rank 数据切分、global batch size、all-reduce 平均梯度、同步前后参数是否一致、`no_sync()` 减少通信次数、collective 是否所有 rank 都参与，以及是否只有 rank 0 写 checkpoint。

```python
from itertools import combinations


def distributed_indices(num_samples, world_size, rank, epoch=0):
    indices = list(range(num_samples))
    offset = (epoch * 3) % num_samples
    indices = indices[offset:] + indices[:offset]
    return indices[rank::world_size]


def find_overlaps(parts):
    overlaps = []
    for left, right in combinations(parts, 2):
        common = sorted(set(parts[left]) & set(parts[right]))
        if common:
            overlaps.append((left, right, common))
    return overlaps


def audit_collective(traces, collective_name):
    missing = [rank for rank, events in traces.items() if collective_name not in events]
    return {
        "ok": not missing,
        "missing": missing,
    }


world_size = 4
per_device_batch_size = 2
accum_steps = 3
dataset_size = 16

epoch0_parts = {
    rank: distributed_indices(dataset_size, world_size, rank, epoch=0)
    for rank in range(world_size)
}
epoch1_parts = {
    rank: distributed_indices(dataset_size, world_size, rank, epoch=1)
    for rank in range(world_size)
}

all_epoch0_indices = sorted(index for part in epoch0_parts.values() for index in part)
sampler_overlap = find_overlaps(epoch0_parts)
global_batch_size = per_device_batch_size * world_size * accum_steps

rank_grads = [0.2, 0.4, -0.1, 0.3]
avg_grad = sum(rank_grads) / world_size
theta0 = 1.0
lr = 0.5
synced_params = [round(theta0 - lr * avg_grad, 3) for _ in rank_grads]
unsynced_params = [theta0 - lr * grad for grad in rank_grads]
unsynced_param_drift = round(max(unsynced_params) - min(unsynced_params), 3)

micro_steps = 6
sync_without_no_sync = micro_steps
sync_with_no_sync = sum(
    1 for step in range(micro_steps)
    if (step + 1) % accum_steps == 0
)

ok_trace = {
    rank: ["forward", "backward", "all_reduce", "optimizer_step"]
    for rank in range(world_size)
}
bad_trace = {
    0: ["forward", "backward", "all_reduce", "optimizer_step"],
    1: ["forward", "backward", "all_reduce", "optimizer_step"],
    2: ["forward", "backward", "optimizer_step"],
    3: ["forward", "backward", "all_reduce", "optimizer_step"],
}
ok_collective = audit_collective(ok_trace, "all_reduce")
bad_collective = audit_collective(bad_trace, "all_reduce")

rank0_writers = [rank for rank in range(world_size) if rank == 0]
checks = {
    "sampler_no_overlap": sampler_overlap == [],
    "sampler_full_coverage": all_epoch0_indices == list(range(dataset_size)),
    "sampler_epoch_changed": epoch0_parts[0] != epoch1_parts[0],
    "synced_params_equal": len(set(synced_params)) == 1,
    "unsynced_would_drift": unsynced_param_drift > 0,
    "no_sync_reduces_calls": sync_with_no_sync < sync_without_no_sync,
    "detects_missing_collective": not bad_collective["ok"],
    "only_rank0_writes": rank0_writers == [0],
}

print("global_batch_size=", global_batch_size)
print("sampler_epoch0=", epoch0_parts)
print("sampler_overlap=", sampler_overlap)
print("sampler_coverage=", len(all_epoch0_indices))
print("sampler_epoch_changed=", checks["sampler_epoch_changed"])
print("all_reduce_avg_grad=", round(avg_grad, 3))
print("synced_params=", synced_params)
print("unsynced_param_drift=", unsynced_param_drift)
print("no_sync_calls=", {
    "without_no_sync": sync_without_no_sync,
    "with_no_sync": sync_with_no_sync,
})
print("collective_audit=", {
    "ok": ok_collective["ok"],
    "bad": bad_collective["ok"],
    "bad_missing": bad_collective["missing"],
})
print("rank0_writers=", rank0_writers)
print("checks=", checks)
```

这段 demo 要抓住几个点：

1. `sampler_overlap=[]` 说明同一个 epoch 内不同 rank 没有重复读样本。
2. `global_batch_size=24` 说明多卡和梯度累积会改变真实 batch。
3. all-reduce 平均后所有 rank 的参数更新一致；如果各自用本地梯度更新，`unsynced_param_drift` 会大于 0。
4. `no_sync()` 的直觉是多个 micro-step 只同步一次，demo 中通信次数从 6 次降到 2 次。
5. collective audit 能发现 rank 2 没有进入 `all_reduce`，这类不一致是分布式训练卡住的常见原因。
6. checkpoint 写入者只有 rank 0，避免多进程同时写同一个文件。

## 7.19 面试官会怎么问

### 问题一：DDP 的工作原理是什么？

回答模板：

```text
DDP 是数据并行。每个进程通常绑定一张 GPU，并持有一份模型副本。每个进程处理不同 mini-batch，backward 后通过 all-reduce 同步梯度，保证每张卡拿到一致的平均梯度，然后各自执行 optimizer step，所以参数保持一致。
```

### 问题二：rank、local rank、world size 分别是什么？

回答模板：

```text
world size 是总训练进程数，rank 是全局进程编号，local rank 是当前机器内的进程或 GPU 编号。通常用 local rank 绑定当前 GPU，用 rank 判断是否是主进程，比如只有 rank 0 打日志和保存 checkpoint。
```

### 问题三：为什么 DDP 要用 DistributedSampler？

回答模板：

```text
因为每个 rank 应该处理不同数据子集，否则多张卡会重复训练同样数据。DistributedSampler 会按 rank 和 world size 切分数据，并在每个 epoch 通过 set_epoch 刷新 shuffle 顺序。
```

### 问题四：DDP 和 FSDP 有什么区别？

回答模板：

```text
DDP 每张卡通常持有完整模型副本，主要通过数据并行提升吞吐；FSDP 会把参数、梯度和优化器状态分片到多张卡上，降低单卡显存，适合更大模型，但通信、checkpoint 和调试更复杂。
```

### 问题五：分布式训练卡住怎么排查？

回答模板：

```text
我会先确认每个 rank 是否启动、绑定正确 GPU、进入同一训练阶段，再检查是否所有 rank 都参与同样的 collective 操作，数据 batch 数是否一致，是否有某些 rank 报错退出，是否存在条件分支导致部分 rank 没走到同步点。必要时用 rank 日志和 barrier 定位卡住位置。
```

## 7.20 常见误区

1. 以为多卡就是自动变快。通信开销、数据加载和 batch 配置都会影响加速比。
2. 以为 DDP 会自动处理数据切分。数据切分要用 DistributedSampler 或自己实现。
3. 以为每张卡 batch size 不变时训练完全等价。global batch size 变大后优化行为会变化。
4. 以为所有 rank 都可以写同一个 checkpoint。通常只让 rank 0 写。
5. 以为 `find_unused_parameters=True` 是万能修复。它有开销，应该理解参数为什么没被使用。
6. 以为 FSDP 只是更快的 DDP。FSDP 主要解决显存分片问题，复杂度也更高。

## 7.22 本章总结

分布式训练的入门主线是：多进程、多 GPU、每个进程一个 rank，DDP 负责梯度同步，DistributedSampler 负责数据切分，torchrun 负责启动和环境变量设置。

需要记住：

1. `rank` 是全局进程编号，`local_rank` 是本机 GPU 编号，`world_size` 是总进程数。
2. DDP 是最常见的数据并行方式，每张卡处理不同数据，backward 后 all-reduce 梯度。
3. DistributedSampler 防止多卡重复读同样数据。
4. global batch size 等于每卡 batch、卡数和梯度累积步数的乘积。
5. 只有 rank 0 通常负责日志和 checkpoint。
6. FSDP 进一步把参数、梯度和优化器状态分片，用于更大模型。
7. 分布式 debug 要关注 rank 是否一致进入同步点、数据是否切分正确、通信是否卡住。
