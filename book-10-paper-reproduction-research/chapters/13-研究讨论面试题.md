# 第十三章：研究讨论面试题

这一章是第十册的收束章。前面十二章讲了如何读论文、判断贡献、做复现、设计消融、处理负结果，以及 Transformer、Scaling、Alignment、RAG、Agent、多模态、Safety 等主要论文线。本章把这些能力转成面试中的回答方式。

研究讨论类面试和普通知识问答不一样。普通知识问答可能问“Transformer 的 QKV 是什么”“DPO 公式是什么”“LoRA 为什么省显存”。研究讨论更像真实组会：面试官会给你一篇论文、一个方向、一个实验现象或一个开放问题，让你判断它是否重要、如何复现、如何改进、哪里可能有坑。

本章重点：最近哪篇论文重要？如何复现 DPO？如何判断一个新 attention 方法是否值得？如何评价一篇论文贡献？如何设计 ablation？如何解释负结果？如何把论文变成可落地项目？

## 13.1 研究讨论面试到底考什么

研究讨论面试不是考你背了多少论文名，而是考五种能力。

| 能力 | 面试官想看什么 | 常见表现 |
| --- | --- | --- |
| 问题建模能力 | 能不能把论文放进正确问题里 | 先讲任务、瓶颈和约束 |
| 贡献判断能力 | 能不能区分真贡献和包装 | 看 baseline、控制变量、适用边界 |
| 实验设计能力 | 能不能复现实验并验证结论 | 讲数据、模型、指标、消融、误差来源 |
| 批判思维 | 能不能发现论文漏洞 | 指出 confounder、评估偏差、成本未计入 |
| 研究延展能力 | 能不能提出合理下一步 | 给出低风险复现和高价值扩展 |

很多候选人回答研究问题时容易犯两个错误。

第一个错误是堆名词。例如回答“最近哪篇论文重要”时，直接说 FlashAttention、RoPE、MoE、DPO、RAG、Sora、SAE，但没有解释为什么重要。

第二个错误是只讲结论。例如说“DPO 比 PPO 简单”“RAG 可以缓解幻觉”“MoE 可以省计算”，但没有讲清假设、实验条件、失败模式和替代方案。

更好的回答方式是：先把问题放进历史脉络，再讲论文解决了哪个具体瓶颈，然后讲方法机制、实验是否支持结论、有什么边界，最后讲如果自己复现或改进会怎么做。

## 13.2 通用回答模板：一篇论文怎么讲

面试中被问“请介绍一篇你熟悉的论文”，可以按下面七步回答。

第一步，讲背景。

不要一上来背公式，先说这篇论文之前大家怎么做，以及遇到了什么问题。

例如讲 DPO，不要直接写 DPO loss。可以先说：RLHF 早期常用 reward model 加 PPO，但 PPO 工程复杂、训练不稳定、超参敏感、需要维护 policy、reference、reward model 和 value model。DPO 的出现，是为了把 preference optimization 改写成更简单的监督式目标。

第二步，讲核心问题。

一篇好论文通常不是“提出一个新模块”，而是抓住一个瓶颈。

例如 FlashAttention 的核心问题不是“写了一个 CUDA kernel”，而是标准 attention 在长序列下受 HBM 读写瓶颈影响很大，实际速度不只由 FLOPs 决定，还由 IO 决定。

第三步，讲方法机制。

方法机制要讲到“为什么可能有效”。

例如 RoPE 的机制不是“加了旋转位置编码”，而是把绝对位置注入到 query 和 key 的旋转变换中，使 attention score 中自然带有相对位置信息，同时能较好支持长度外推。

第四步，讲实验设计。

要说清数据、模型规模、baseline、指标和消融。

例如评价一个 RAG 方法，要看它是否只在闭卷 QA 上提升，还是在多跳问答、长文档、拒答、时效性、引用准确性上都有评估。

第五步，讲贡献边界。

任何论文都有适用条件。讲边界不是否定论文，而是说明你真的理解它。

例如 DPO 简化了 preference optimization，但它仍依赖偏好数据质量、reference model、beta 超参和 chosen/rejected 对的覆盖范围。它不是自动解决所有 alignment 问题。

第六步，讲可复现性。

面试官很喜欢追问“你会怎么复现”。这时要讲最小复现路径，而不是说“照着论文跑”。

最小复现路径包括：选一个小模型、一个公开数据集、一个可控 baseline、少量关键指标和两三个必要消融。

第七步，讲下一步。

可以提出一个保守扩展和一个探索扩展。

保守扩展是提高可信度，比如更多数据集、更强 baseline、更严格消融。探索扩展是提出新问题，比如把方法迁移到长上下文、多模态、Agent 或安全场景。

## 13.3 高频题一：最近哪篇论文重要

这是研究讨论面试中最常见的问题之一。面试官不一定关心你选的是哪篇，而是看你如何定义“重要”。

一个成熟回答可以这样组织。

第一，先定义重要性标准。

可以说：我会从四个维度判断论文重要性：是否抓住了真实瓶颈，是否改变了后续研究范式，是否有可复现的实验证据，是否能在工程系统中产生影响。

第二，选一篇你真正理解的论文。

不要选自己只看过摘要的论文。宁可讲一篇经典论文讲深，也不要讲最新但讲不清的论文。

第三，讲它改变了什么。

下面给几个可用例子。

### 13.3.1 如果选择 FlashAttention

可以这样回答：

我认为 FlashAttention 是重要论文，因为它把 attention 加速问题从单纯 FLOPs 视角推进到 IO-aware 视角。标准 attention 需要显式构造较大的 attention matrix，在长序列场景下 HBM 读写成为瓶颈。FlashAttention 通过 tiling、online softmax 和重计算策略减少 HBM 访问，在保持精确 attention 的同时显著提高速度和降低显存。

它的重要性不只是一个 kernel 更快，而是改变了大家看待 Transformer 系统瓶颈的方式：算法复杂度、显存访问、kernel fusion 和硬件特性必须一起考虑。后来长上下文训练、推理优化和高性能 attention 变体都受到了这种思路影响。

边界是：它主要解决精确 attention 的实现效率，不直接改变 attention 的二次复杂度理论上限；在不同硬件、序列长度、batch size、head dim 下收益不同；工程实现也需要非常谨慎的数值稳定性处理。

### 13.3.2 如果选择 DPO

可以这样回答：

我认为 DPO 重要，是因为它把 RLHF 中较复杂的 reward model 加 PPO 流程，改写成一个更接近监督学习的偏好优化目标。它降低了对齐训练的工程门槛，使更多团队可以稳定复现 preference optimization。

它解决的问题是 PPO 式 RLHF 的训练复杂性和不稳定性。DPO 从 KL-constrained reward maximization 的最优解关系出发，把偏好概率和 policy/reference log probability ratio 联系起来，最后得到只需要 chosen/rejected pair 的优化目标。

它的边界是：DPO 并不意味着不需要高质量偏好数据，也不意味着没有超参敏感性。偏好对的覆盖范围、噪声、长度偏差、reference model 选择、beta 设置都会影响效果。它更像是让 preference optimization 更简单，而不是彻底解决 alignment。

### 13.3.3 如果选择 Transformer

可以这样回答：

Transformer 当然重要，但面试里不能只说“它提出了 self-attention”。更好的说法是：Transformer 把序列建模从强递归结构转向基于 attention 的并行建模，让长距离依赖、训练并行性和规模扩展变得更容易。它不仅提升了机器翻译，还成为后来预训练语言模型、多模态模型和扩散 Transformer 的基础结构。

它的重要性来自三个方面：结构上去掉 RNN 的顺序瓶颈，计算上更适合 GPU 并行，表达上用 attention 建模 token 间动态关系。

边界是：标准 attention 在长序列下计算和显存开销高；Transformer 本身不自带数据、目标函数和对齐能力；后来的 GPT、BERT、T5、ViT、MoE、RAG 都是在它的基础上继续解决不同瓶颈。

### 13.3.4 如果选择 Scaling Laws

可以这样回答：

Scaling Laws 重要，是因为它把大模型发展从经验堆规模变成更可预测的资源分配问题。它研究 loss 随模型参数、数据量和计算量变化的规律，使训练大模型前可以估算不同资源配置下的收益。

Chinchilla 进一步强调，在固定 compute 下，模型参数和训练 token 需要平衡。过大的模型配太少 token 会 undertrain，较小模型配更多数据可能更 compute-optimal。这影响了后续大模型训练策略。

边界是：scaling law 主要是经验规律，依赖数据分布、模型结构、训练 recipe 和损失定义；它预测的是统计趋势，不保证某个具体能力或安全行为按同样规律增长。

## 13.4 高频题二：如何复现 DPO

这个问题非常典型，因为它同时考论文理解、训练工程、数据处理和评估设计。

不要只回答“用 TRL 跑一下 DPOTrainer”。更好的回答要分层。

### 13.4.1 先讲目标

可以说：复现 DPO 的目标不是盲目追求论文完全同分，而是验证三个核心结论：DPO 是否能用 pairwise preference 数据提升偏好指标；是否比 SFT baseline 更符合人类偏好；是否在相近成本下比 PPO 更简单稳定。

### 13.4.2 最小复现配置

一个合理的最小复现可以这样设计。

| 组件 | 选择 |
| --- | --- |
| Base model | 1B 到 7B 级开源模型，资源有限可选更小模型 |
| SFT 数据 | 指令微调数据或已有 SFT checkpoint |
| Preference 数据 | HH-RLHF、UltraFeedback、HelpSteer 或同类 chosen/rejected 数据 |
| Baseline | SFT only、可能再加一个 IPO/KTO/ORPO 或 PPO baseline |
| 指标 | win rate、reward model score、长度分布、MT-Bench 类评估、人工抽检 |
| 消融 | beta、reference model、数据量、长度控制、偏好噪声 |

### 13.4.3 讲清 DPO 的训练逻辑

DPO 的输入是一组 prompt、chosen response 和 rejected response。训练时分别计算 policy model 和 reference model 对 chosen/rejected 的 log probability。

直观理解是：如果 chosen 比 rejected 更好，训练应该提高 policy 相对 reference 对 chosen 的偏好，同时降低对 rejected 的偏好。beta 控制偏离 reference 的强度。

面试中不一定要把完整公式写得很复杂，但要讲清三件事：

1. DPO 不是普通二分类，它比较的是 policy 相对 reference 的 log probability ratio。
2. Reference model 提供 KL 约束意义上的锚点，避免 policy 无限制漂移。
3. beta 太小或太大都会影响训练，太小可能更新弱，太大可能过度偏离或产生副作用。

### 13.4.4 复现中最容易踩的坑

第一，数据格式错误。

chosen/rejected 必须共享同一个 prompt，否则偏好比较就不成立。很多数据集还存在多轮对话模板差异，chat template 处理错误会直接影响 log probability。

第二，长度偏差。

如果 chosen 普遍比 rejected 更长，模型可能学到长度偏好，而不是真正质量偏好。因此评估时要看输出长度分布，也可以做长度归一化或分桶分析。

第三，reference model 用错。

DPO 通常以 SFT model 作为 reference。如果 reference 和 policy 初始化、tokenizer 或模板不一致，实验会变得难以解释。

第四，只看 reward score。

Reward model score 可能被 reward hacking。需要结合人工抽检、多评估器、任务指标和拒答率。

第五，忽略能力退化。

偏好优化可能提升 helpfulness，但降低数学、代码、事实性或安全拒答质量。复现时要有 general capability eval。

### 13.4.5 可以如何回答面试官

一个完整回答可以是：

如果让我复现 DPO，我会先选一个已有 SFT checkpoint 和公开 preference 数据集，建立 SFT-only baseline。然后用同一个 tokenizer 和 chat template 构造 prompt/chosen/rejected 三元组，以 SFT model 作为 reference model 和 policy 初始化。训练时重点 sweep beta、learning rate、数据量，并监控 chosen/rejected log probability margin、KL proxy、输出长度和 loss 曲线。

评估时我不会只看 reward model score，而会看 pairwise win rate、MT-Bench 或 AlpacaEval 类自动评估、少量人工抽检，以及数学、代码、事实性和安全拒答上的能力回归。必要消融包括无 reference、不同 beta、不同偏好数据质量、长度控制和不同 decoding 设置。这样才能判断 DPO 提升来自真正偏好学习，还是来自长度偏差、评估器偏差或数据泄漏。

## 13.5 高频题三：如何判断一个新 attention 方法是否值得

这是算法岗很常见的问题。很多论文会提出新的 attention 变体，声称更快、更长、更省显存、更强效果。面试官想看你能不能识别“真改进”和“纸面改进”。

可以从六个维度判断。

### 13.5.1 它解决的是哪个瓶颈

新 attention 方法通常瞄准以下瓶颈之一：

1. 训练显存：attention matrix 太大。
2. 训练速度：标准 attention IO 或计算开销高。
3. 推理 KV cache：长上下文解码时缓存太大。
4. 长程建模：模型难以有效使用很长上下文。
5. 稀疏性：不是所有 token 都需要互相看。
6. 硬件效率：理论复杂度低但实际 kernel 不快。

如果论文没有说清自己主要解决哪个瓶颈，贡献就很可疑。

### 13.5.2 复杂度是不是只在纸面上好看

很多方法把复杂度从二次降到线性或近线性，但实际不一定快。

原因包括：

1. kernel 不友好，无法高效利用 GPU。
2. 需要额外排序、聚类、路由或索引。
3. 稀疏模式导致内存访问不连续。
4. 常数项太大，小中等长度没有收益。
5. 训练和推理收益不一致。

因此判断 attention 方法不能只看 big-O，还要看 wall-clock time、显存峰值、吞吐、硬件平台和实现成熟度。

### 13.5.3 是否保持模型质量

如果一个方法更快但 perplexity、下游任务、长上下文检索、代码能力明显下降，就要看 trade-off 是否可接受。

尤其要区分两类评估：

1. 短上下文语言建模 loss。
2. 长上下文能力，例如 needle-in-a-haystack、多文档 QA、长代码理解、长链路推理。

有些方法在短文本上 loss 接近，但长上下文使用能力不足。有些方法能找到 needle，但不能做真实多文档综合。

### 13.5.4 是否能接入现有训练和推理栈

一个方法在论文里有效，不代表工程上值得用。

要看它是否兼容：

1. FlashAttention 或现有高性能 kernel。
2. Tensor parallel、sequence parallel、pipeline parallel。
3. KV cache、paged attention、continuous batching。
4. RoPE、ALiBi、GQA、MQA 等常见结构。
5. 混合精度、量化、speculative decoding。

如果接入成本很高，收益必须足够大。

### 13.5.5 baseline 是否公平

很多新方法的实验会弱化 baseline。

检查点包括：

1. 是否和 FlashAttention 等强实现比较，而不是朴素 attention。
2. 是否在同等参数量、训练 token、数据和 compute 下比较。
3. 是否调过 baseline 超参。
4. 是否只在小模型上有效。
5. 是否报告训练成本而不只是最终指标。

### 13.5.6 最终判断模板

可以这样回答：

我会先判断这个 attention 方法解决的是计算、显存、KV cache 还是长程建模问题。然后看它的理论复杂度是否能转化为真实硬件收益，尤其是 wall-clock、显存峰值和吞吐。接着看质量是否保持，包括短上下文 loss 和真实长上下文任务。再看它是否兼容现有训练推理栈，比如 FlashAttention、RoPE、GQA、KV cache 和并行策略。最后看实验是否公平，是否和强 baseline 在同等 compute 下比较。如果一个方法只在小模型、小数据或弱 baseline 下提升，而没有工程效率和能力保持证据，我会认为它还不值得投入生产复现。

## 13.6 高频题四：如何评价一篇论文贡献

评价论文贡献，不能只看作者说自己贡献是什么。要自己拆。

可以分成五类贡献。

| 贡献类型 | 例子 | 判断标准 |
| --- | --- | --- |
| 新问题 | 提出新的风险、任务或评估维度 | 是否真实重要，是否可测 |
| 新方法 | 新架构、新 loss、新训练流程 | 是否优于强 baseline，是否可复现 |
| 新系统 | 高效训练、推理、数据流水线 | 是否有真实吞吐、成本和稳定性收益 |
| 新数据 | 高质量数据集或 benchmark | 是否覆盖关键场景，是否低污染 |
| 新分析 | scaling law、mechanistic insight、失败模式 | 是否解释现象，是否能预测或指导后续工作 |

评价时可以用下面问题。

第一，它解决的问题是不是重要？

重要问题通常有明确痛点，例如训练成本太高、长上下文不可用、偏好优化不稳定、模型会泄露隐私、评估被污染。

第二，现有方法为什么不够？

如果论文没有公平解释前人方法的不足，就容易是“重新包装”。

第三，新方法的核心假设是什么？

例如 MoE 的假设是不同 token 可以路由到不同 expert，从而在增加参数容量时控制每 token 计算量。这个假设带来收益，也带来负载均衡、通信和 expert collapse 风险。

第四，实验是否支持核心假设？

很多论文的实验只能证明“结果变好了”，不能证明“作者声称的机制导致了变好”。这时需要看 ablation 和 controlled experiment。

第五，有没有隐藏成本？

隐藏成本包括训练不稳定、推理延迟、显存峰值、数据构造成本、标注成本、部署复杂度、安全副作用和维护成本。

## 13.7 高频题五：如何设计 ablation

Ablation 的目标不是把表格填满，而是验证因果问题：到底是哪个因素带来了提升？

一个好 ablation 至少要覆盖四类。

第一，组件消融。

去掉某个模块、替换某个 loss、关闭某个训练阶段，看性能变化。

第二，超参消融。

例如 DPO 的 beta、LoRA rank、learning rate、context length、retrieval top-k、MoE expert 数量。

第三，数据消融。

改变数据量、数据质量、数据来源、去重策略、合成数据比例，观察影响。

第四，评估消融。

换 benchmark、换 judge、换 decoding 参数、换 prompt template，判断结论是否稳定。

面试中可以强调一个原则：每个 ablation 都要对应一个假设。

例如论文声称 RAG 的提升来自更好的 reranker，那么 ablation 应该比较无 reranker、旧 reranker、新 reranker，并控制 retriever、top-k、生成模型和上下文长度。如果同时改变多个因素，就不能证明提升来自 reranker。

## 13.8 高频题六：如何解释负结果

研究面试中，能讲清负结果很加分。真实研究里，大量实验不会按预期提升。

负结果不等于失败，关键是能否定位原因。

可以按下面顺序排查。

第一，检查实现。

包括数据读取、label、mask、loss、梯度、学习率、混合精度、分布式同步、eval pipeline。

第二，检查复现条件。

论文是否有未公开细节？数据版本、tokenizer、prompt template、预训练 checkpoint、训练 token 数是否一致？

第三，检查 baseline。

也可能不是新方法差，而是你的 baseline 比论文更强，导致提升空间变小。

第四，检查规模。

有些方法存在 scale-dependent behavior，小模型无效，大模型有效；也有方法小模型有效，大模型收益消失。

第五，检查指标。

方法可能提升了一个维度，但你看的指标没有覆盖。例如 RAG 提升事实性，但通用聊天分数变化不大。

第六，形成结论。

好的负结果报告应该说明：实验假设是什么，做了哪些排查，哪些可能原因被排除，最终剩下哪些解释，下一步如何验证。

## 13.9 高频题七：如何把论文变成项目

算法岗面试经常会问：如果让你把这篇论文落地，你怎么做？

回答时不要说“复现代码，然后上线”。要讲从研究到工程的路径。

第一，定义业务或系统目标。

例如要降低推理延迟、提升长文档问答准确率、减少幻觉、提升偏好对齐、降低训练成本。

第二，做最小可行复现。

先用小模型、小数据、离线 benchmark 验证方向，不要一开始就上全量系统。

第三，和强 baseline 比较。

生产里最重要的 baseline 往往不是论文 baseline，而是当前线上系统。

第四，做成本评估。

包括训练成本、推理成本、延迟、显存、标注成本、数据维护成本和工程复杂度。

第五，做风险评估。

包括安全风险、隐私风险、稳定性风险、过度拒答、评估污染、回滚方案。

第六，灰度上线。

先离线评估，再小流量 A/B，监控关键指标，保留回滚。

## 13.10 高频题八：如何判断一篇 alignment 论文是否可靠

Alignment 论文尤其容易出现评估偏差，因为很多指标依赖 reward model、LLM judge 或人工偏好。

判断时重点看六点。

第一，偏好数据来源。

是人工标注、模型生成、规则构造，还是混合数据？数据是否有噪声、偏见和长度偏差？

第二，baseline 是否强。

是否和 SFT、PPO、DPO、IPO、KTO、ORPO 等合理 baseline 比较？是否使用同等数据和 compute？

第三，评估器是否可靠。

LLM judge 是否有位置偏差、长度偏差、自我偏好？是否做了人工校验？

第四，是否报告能力回归。

偏好优化可能让模型更会讨好评估器，但数学、代码、事实性或安全边界下降。

第五，是否区分 harmlessness 和 helpfulness。

更安全不等于更有帮助，过度拒答也不是好模型。

第六，是否有分布外测试。

只在训练同源数据上有效，说明不了泛化对齐。

## 13.11 高频题九：如何判断一篇 RAG 论文是否可靠

RAG 论文常见问题是把多个因素混在一起：检索器变了、切分策略变了、上下文长度变了、prompt 变了、生成模型也变了。这样很难知道提升来自哪里。

判断 RAG 论文，可以看以下维度。

第一，问题类型。

是 open-domain QA、企业知识库、多跳问答、长文档总结、代码检索，还是带时效性的问答？不同任务需要不同评估。

第二，检索指标和生成指标是否分开。

检索要看 recall、MRR、nDCG；生成要看准确率、引用支持率、faithfulness、拒答质量和 hallucination rate。

第三，是否控制上下文预算。

如果新方法只是塞了更多文本进去，提升不一定来自检索质量。

第四，是否评估失败模式。

例如检索不到、检索到冲突证据、证据过长、证据包含错误、问题不可回答。

第五，是否有端到端成本。

RAG 系统要考虑索引构建、embedding 成本、召回延迟、rerank 延迟、上下文 token 成本和更新成本。

## 13.12 高频题十：如何判断一篇 Agent 论文是否可靠

Agent 论文很容易被 demo 迷惑。一个 demo 能跑通，不代表方法可靠。

判断 Agent 论文要看：

1. 任务是否真实，还是为方法定制的 toy environment。
2. 是否报告成功率、成本、步数、时间、错误类型。
3. 是否和简单 baseline 比较，例如单次调用、固定 workflow、检索增强、工具约束版本。
4. 是否控制工具权限和环境状态。
5. 是否有多次随机种子或多轮采样稳定性。
6. 是否评估安全边界、权限隔离和失败恢复。

一个很实用的判断是：如果一个 Agent 方法需要大量 prompt trick、人工干预和挑选成功案例，却没有系统错误分析，就不能认为它已经可靠。

## 13.13 高频题十一：如果论文结果和你的直觉相反怎么办

不要急着否定论文，也不要盲信论文。

可以回答：

如果论文结果和我的直觉相反，我会先检查它的实验设定是否和我的直觉适用场景一致。很多直觉只在某个规模、数据或系统条件下成立。然后我会看 baseline 是否公平、指标是否合理、是否有消融支持机制解释。接着尝试做一个最小复现，优先复现核心现象而不是全量结果。如果复现成功，我会更新自己的假设；如果复现失败，我会从实现、数据、规模和评估差异排查，并把负结果整理成可解释报告。

这个回答体现的是科学态度：既不迷信论文，也不迷信直觉。

## 13.14 高频题十二：如何提出一个改进点

提出改进点时，最忌讳说“可以结合多模态”“可以加 RAG”“可以用更大模型”。这些太空泛。

好的改进点应该满足三个条件：

1. 针对明确失败模式。
2. 有可验证实验。
3. 成本和收益可比较。

例如针对 DPO，可以提出：如果发现 chosen response 系统性更长，可以做长度分桶、长度归一化评估，或构造长度匹配的 preference subset，验证 DPO 是否仍提升。

例如针对 RAG，可以提出：如果 hallucination 主要来自检索到冲突证据，可以增加 evidence conflict detection，并设计包含冲突文档的数据集评估拒答和证据归因。

例如针对 long context，可以提出：如果模型 needle task 很强但多文档综合弱，可以构造需要跨文档证据合成的 benchmark，而不是只做单点检索。

## 13.15 高频题十三：如何读一篇你完全没见过的论文

面试中有时会让你现场读摘要或方法图，然后问判断。

可以按快速阅读法。

第一，看标题和摘要，判断问题域。

它是架构、训练、对齐、评估、数据、推理系统、多模态还是安全？

第二，看 introduction 的 problem statement。

作者声称现有方法哪里不够？这个痛点是否真实？

第三，看方法图。

找出到底新增了什么变量、模块、loss、数据流程或系统组件。

第四，看实验表。

先看和谁比，再看在哪些数据上比，最后看提升幅度是否稳定。

第五，看 ablation。

如果没有 ablation，要谨慎；如果 ablation 不能验证核心假设，也要谨慎。

第六，看 limitations。

好论文通常会诚实说明边界；完全没有 limitations 的论文不一定更强。

## 13.16 高频题十四：研究实习中如何选题

如果面试官问“你来了以后想做什么方向”，不要只说热点。要把个人能力、团队需求和研究价值连起来。

可以这样回答：

我会优先选择同时满足三个条件的题：第一，问题对团队当前模型或系统有真实价值；第二，可以在实习周期内做出最小闭环；第三，即使结果不如预期，也能沉淀数据、评估或分析资产。

例如，如果团队做长上下文模型，我不会一开始就提出重新设计完整架构，而会先分析模型在真实长文档任务中的失败模式，区分检索失败、位置偏置、上下文压缩失败和推理失败。然后选择一个最可控的点做改进，比如更可靠的长上下文评估、训练数据构造或推理时上下文选择策略。

这个回答比“我想做 Agent”更像能落地的研究者。

## 13.17 高频题十五：如何展示自己的论文复现项目

展示复现项目时，不要只说“我复现了某某论文”。要讲清楚你做了哪些判断。

建议结构如下：

第一，为什么选这篇论文。

说明它对应的真实瓶颈和你希望验证的核心结论。

第二，你复现了什么。

说明模型、数据、代码框架、训练配置和评估指标。

第三，哪些结果复现了，哪些没有。

诚实讲差异，并解释可能原因。

第四，你做了哪些额外实验。

比如更强 baseline、额外消融、成本分析、失败案例分析。

第五，你得到什么结论。

结论最好不是“论文是对的”，而是更具体：在什么条件下有效，主要收益来自哪里，部署还差什么。

## 13.18 高频追问与简短回答

### 13.18.1 你怎么判断一篇论文是不是 incremental？

看它是否只替换一个模块但没有解决新瓶颈，是否只在弱 baseline 上提升，是否缺少机制解释和消融，是否没有改变问题理解或工程 trade-off。Incremental 不一定没价值，但需要证明收益稳定、成本低、适用范围清楚。

### 13.18.2 论文没有开源代码，你还会复现吗？

会，但会先评估风险。我会从最小核心机制开始复现，而不是追求完整系统。先确认数据、模型、loss 和评估是否足够清楚；如果关键细节缺失太多，就把目标改成验证核心现象，而不是复现完整分数。

### 13.18.3 自动评估和人工评估冲突时信谁？

先查冲突原因。自动评估可能有长度偏差、格式偏差、judge 偏差；人工评估可能样本少、一致性低、标注标准不清。更好的做法是抽样复核，计算一致性，按错误类型拆分，而不是简单信某一个。

### 13.18.4 如果老板要求一周内复现一篇大论文怎么办？

我会先定义一周内可完成的最小复现目标：复现核心机制和一个关键表格或现象。第一天读论文和列依赖，第二天跑通 baseline，第三到五天实现方法和主实验，第六天做必要消融，第七天整理误差和结论。不会承诺复现所有大规模结果。

### 13.18.5 你怎么看 leaderboard？

Leaderboard 有价值，但不能等同于研究质量。它容易受数据污染、prompt tuning、评估器偏差和过拟合影响。研究中要看多指标、多场景、成本和失败模式；生产中更要看线上任务指标和稳定性。

### 13.18.6 你怎么看“更大模型是否总会更好”？

更大模型通常有更强容量，但不是所有问题都靠变大解决。数据质量、训练 compute 分配、推理成本、延迟、安全风险、对齐质量都会影响最终效果。有些场景用 RAG、工具、数据改进或小模型蒸馏更划算。

### 13.18.7 如果你只能做一个消融，会做哪个？

我会选最能验证核心机制的消融，而不是最容易做的消融。例如论文声称提升来自某个新 loss，就固定数据、模型和训练预算，只替换 loss；如果论文声称提升来自数据质量，就固定训练流程，只改变数据过滤策略。

### 13.18.8 如何判断论文适不适合投资源复现？

看四点：是否解决团队真实瓶颈，是否有足够清楚的实现细节，是否有可在小规模验证的核心假设，潜在收益是否大于复现成本。如果四点都满足，值得做最小复现；否则先做阅读报告或等待更多证据。

## 13.19 面试中的表达细节

研究讨论不仅看内容，也看表达。

第一，先给结论，再展开。

例如：“我认为这篇论文重要，但主要重要在系统视角，不是单个模块。”然后再解释。

第二，主动说边界。

主动说边界会显得更可信。例如：“这个方法在训练阶段收益明显，但推理部署还要看 kernel 和 KV cache 兼容性。”

第三，不要假装看过。

如果没读过某篇论文，可以说：“这篇我没有完整读过，但从标题和我了解的方向看，我会重点检查它的 baseline、核心假设和评估设置。”然后给出分析框架。

第四，避免绝对化。

少说“肯定”“一定”“完全解决”。多说“在这个设定下”“可能来自”“需要进一步验证”。

第五，能落到实验。

任何观点最后都最好能落到一个实验：怎么验证，控制什么变量，看什么指标。

## 13.20 本章小结

研究讨论面试考的不是论文背诵，而是研究判断力。

本章要记住几句话：

1. 讲论文要讲背景、问题、机制、实验、边界、复现和下一步。
2. 判断重要性要看真实瓶颈、范式影响、证据强度和工程价值。
3. 复现 DPO 不能只跑脚本，要控制数据、reference、beta、长度偏差和评估器偏差。
4. 判断 attention 方法不能只看复杂度，要看真实硬件收益、质量保持和系统兼容。
5. Ablation 要对应假设，否则只是表格装饰。
6. 负结果要能定位原因，负结果报告也是研究能力。
7. Alignment、RAG、Agent 论文尤其要警惕评估偏差和 demo bias。
8. 好的研究回答不是“我知道很多论文”，而是“我知道如何判断、验证和推进一个问题”。

如果面试中被问研究讨论题，最好的状态是像在组会里和同事讨论：能给判断，能讲证据，能说边界，也能提出下一步实验。
