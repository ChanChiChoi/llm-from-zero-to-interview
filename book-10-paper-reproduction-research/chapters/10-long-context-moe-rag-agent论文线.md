# 第十章：Long Context、MoE、RAG、Agent 论文线

这一章讨论几条在大模型时代非常重要、但容易被混在一起的论文线：长上下文、稀疏专家模型、检索增强生成、工具使用和 Agent Planning。

它们看起来属于不同方向：长上下文像是模型架构和系统优化问题，MoE 像是参数规模扩展问题，RAG 像是知识接入问题，Tool Use 和 Agent 像是应用层能力问题。但如果从“模型如何突破单次前向传播的限制”来看，它们其实共享同一个主线：

模型参数不是唯一的能力来源。上下文窗口、外部检索、外部工具、专家路由、环境反馈，也可以成为模型能力的一部分。

传统 Transformer 的默认设定是：给模型一段 token，模型在固定参数内完成计算，输出下一个 token。这个设定非常优雅，也非常受限。当任务需要更长记忆、更大容量、更新知识、调用计算器、查询数据库、分步执行计划时，单纯扩大 dense 模型并不总是最优解。本章要讲的论文线，就是围绕这些限制逐步展开的。

## 10.1 先建立总图：这些方向分别解决什么问题

先用一个小白友好的例子。

假设你要让一个人完成复杂研究任务：写一份关于某个行业的投资分析报告。

如果这个人只能凭脑子记忆回答，那对应纯参数模型。它可能知道很多常识，但知识会过期，也可能编造。

如果你给他一大摞资料，让他在同一次工作里全部阅读，那对应 long context。上下文越长，他能参考的原始材料越多。

如果你不把所有资料都塞给他，而是让他需要时去图书馆检索相关资料，那对应 RAG。检索系统负责找资料，语言模型负责读资料和组织答案。

如果这个人不是一个全能专家，而是有法律、财务、技术、市场等多个专家小组，任务来了先路由给合适专家，那对应 MoE。

如果这个人还能打开表格、调用搜索引擎、运行代码、访问数据库，那对应 Tool Use。

如果这个人还能先制定计划、执行一步、观察结果、修正计划、继续执行，那对应 Agent。

这几个方向的区别可以这样记：

| 方向 | 核心问题 | 典型能力来源 | 主要瓶颈 |
| --- | --- | --- | --- |
| Long Context | 如何处理更长输入 | 更大的上下文窗口 | attention 成本、位置外推、有效利用率 |
| MoE | 如何增加模型容量而不等比例增加计算 | 稀疏激活专家 | 路由、负载均衡、通信、训练稳定性 |
| RAG | 如何接入可更新外部知识 | 检索库和非参数记忆 | 召回质量、重排、上下文压缩、引用一致性 |
| Tool Use | 如何让模型使用外部能力 | API、搜索、计算器、代码执行器 | 工具选择、参数生成、错误恢复、安全边界 |
| Agent Planning | 如何完成多步任务 | 计划、行动、观察、反思 | 长程可靠性、状态管理、评估困难 |

面试时最容易犯的错误，是把这些方向都说成“提升模型能力”。更好的说法是：它们提升能力的路径不同，所以工程 trade-off 也完全不同。

## 10.2 Long Context：从“能塞进去”到“真正用得上”

### 10.2.1 为什么长上下文重要

早期 Transformer 的上下文窗口相对较短。BERT 常见长度是 512，早期 GPT 系列也远没有今天动辄几十万 token 的窗口。短上下文在很多任务上够用，比如句子分类、短问答、摘要片段生成。但大模型进入真实应用后，短上下文很快变成瓶颈。

典型场景包括：

1. 长文档问答：法律合同、财报、论文、代码仓库都可能远超几千 token。
2. 多轮对话：用户希望模型记住很久之前的约束和偏好。
3. 代码生成：跨文件依赖、函数调用链、项目结构需要更长上下文。
4. Agent 任务：计划、执行记录、工具返回结果会不断累积。
5. 多模态任务：视频、音频、高分辨率图像切成 token 后序列很长。

所以 long context 不是单纯为了营销里的“窗口更大”，而是为了让模型能在一次推理中访问更多任务证据。

### 10.2.2 原始 self-attention 的瓶颈

Transformer 的 self-attention 需要让每个 token 看其他 token。序列长度为 n 时，attention score 矩阵规模大约是 n 乘 n。长度翻倍，attention 相关计算和中间存储通常会显著增加。

这带来两个问题。

第一个是训练和推理成本。长序列下，attention 可能成为主要瓶颈。

第二个是显存占用。尤其训练时需要保存中间激活用于反向传播，长上下文很容易把显存打爆。

早期很多论文尝试近似 attention，比如局部窗口、稀疏 attention、低秩 attention、kernel attention。这些方法的共同思路是：不要让所有 token 两两交互，而是用某种近似结构降低复杂度。

但近似方法遇到一个问题：理论复杂度降低不等于实际速度提升。GPU 上真正慢的未必只是 FLOPs，还包括显存读写、kernel launch、并行度和内存布局。

这正是 FlashAttention 论文线的重要背景。

### 10.2.3 FlashAttention：IO-aware exact attention

FlashAttention 的核心贡献不是提出新的注意力形式，而是重新审视标准 attention 的实现方式。

标准 attention 计算通常会显式形成 attention score 矩阵，再做 softmax，再乘 V。这个中间矩阵很大，需要在 GPU 高带宽显存 HBM 和片上 SRAM 之间频繁读写。FlashAttention 的观点是：attention 的瓶颈不只是计算量，也包括 IO。

FlashAttention 使用 tiling，把 Q、K、V 分块加载到更快的片上存储中，并通过在线 softmax 技术避免显式存储完整 attention 矩阵。它仍然计算精确 attention，不是近似 attention。

可以这样理解：

普通实现像是每做一步都把草稿纸搬到仓库里存起来，下一步再从仓库搬回来。FlashAttention 像是把必要的小块草稿留在桌面上，边算边汇总，减少来回搬运。

它解决的问题是：在不改变模型数学结果的情况下，降低显存读写，提升速度，支持更长序列。

FlashAttention-2 进一步优化并行策略和工作划分。它关注的问题更偏系统：如何让 attention kernel 更接近矩阵乘法那样高效，减少非矩阵乘法 FLOPs，提高 occupancy，改善线程块和 warp 之间的分工。

面试中如果被问“FlashAttention 是不是降低了 attention 的理论复杂度”，推荐回答：

FlashAttention 对 dense exact attention 的 token 两两交互关系没有做近似，所以不是从数学上把二次复杂度变成线性复杂度。它主要降低的是显存 IO 和中间存储开销，并通过更好的 GPU kernel 实现提升实际吞吐。因此它是 long context 工程化里非常关键的一步，但不等同于 sparse attention 或 linear attention。

### 10.2.4 PagedAttention：长上下文推理的 KV cache 管理

训练时关注 attention 激活和反向传播，推理服务时还有另一个核心瓶颈：KV cache。

自回归生成时，模型每生成一个新 token，都需要访问之前 token 的 K 和 V。如果每次都重新计算全部历史，会非常浪费。所以推理系统会缓存历史 KV。上下文越长、batch 越大、模型层数越多，KV cache 就越大。

传统连续内存分配容易产生浪费。不同请求长度不同，有的请求结束早，有的还在生成；有的 beam search 会复制前缀；有的多轮对话会共享很长历史。显存碎片和重复存储会限制 batch size。

PagedAttention 借鉴操作系统虚拟内存分页思想，把 KV cache 切成块，用非连续物理块承载逻辑连续的 token 序列。这样可以减少碎片，方便共享前缀，也方便动态扩展。

vLLM 基于 PagedAttention 构建高吞吐服务系统。它的贡献不是让模型“更聪明”，而是让相同模型在服务场景下更高效，尤其长序列、大 batch、复杂解码时效果明显。

这里有一个重要区分：

FlashAttention 主要解决 attention kernel 的计算和 IO 效率；PagedAttention 主要解决推理时 KV cache 的内存管理和服务吞吐。两者都和长上下文有关，但层级不同。

### 10.2.5 位置编码与长度外推

即使 attention kernel 和 KV cache 都优化好了，长上下文还有另一个问题：模型是否真的能理解超过训练长度的序列。

如果模型训练时只看过 4K 长度，推理时直接塞 64K，位置编码、注意力分布和数据分布都可能失配。常见方法包括：

1. 训练时直接扩展上下文长度，但成本高。
2. 使用 RoPE scaling、位置插值等方法做长度外推。
3. 采用分段、滑窗、全局 token、递归摘要等结构降低有效长度压力。
4. 结合 RAG，只把相关片段放入上下文，而不是盲目塞满。

面试中要避免只说“窗口扩大了”。更深入的问题是：扩大窗口之后，模型能不能在长距离上准确检索、组合和推理。

### 10.2.6 Long context 的评估陷阱

长上下文评估不能只看“最大支持长度”。更应该看：

1. Needle-in-a-haystack：在长文中找一个特定事实。
2. 多证据聚合：答案需要结合多个远距离片段。
3. 顺序敏感任务：信息出现位置不同是否影响答案。
4. 干扰鲁棒性：大量无关文本是否误导模型。
5. 真实任务收益：长合同审查、代码仓库问答、长对话是否真正变好。

很多模型能在长上下文中找到“针”，但不一定能完成复杂长文推理。长上下文能力至少包含三层：放得下、找得到、用得对。

## 10.3 MoE：用稀疏激活扩大模型容量

### 10.3.1 MoE 的基本动机

Dense Transformer 的一个特点是：每个 token 都经过相同参数。参数越多，通常能力越强，但每次前向计算也越贵。

MoE 的目标是突破这个绑定关系：让模型拥有很多参数，但每个 token 只激活其中一小部分。

这叫 conditional computation。不是所有输入都走同一条路径，而是根据输入选择不同专家。这样总参数量可以很大，单 token 计算量相对可控。

### 10.3.2 Sparsely-Gated MoE：现代 MoE 的关键起点

Shazeer 等人的 Sparsely-Gated Mixture-of-Experts 是现代大规模 MoE 的重要论文。它把很多前馈子网络作为 experts，用一个可训练 gate 为每个输入选择少数专家。

在 Transformer 语境下，MoE 通常替换的是 FFN 层，而不是 attention 层。也就是说，原来每个 token 都经过同一个 FFN，现在每个 token 经过被路由选中的专家 FFN。

一个简化流程是：

1. token hidden state 输入 router。
2. router 输出每个专家的分数。
3. 选择 top-k 个专家。
4. token 被发送给这些专家计算。
5. 专家输出按 router 权重加权合并。

MoE 带来的收益是容量增加。不同专家可以吸收不同模式、语言、领域或任务的知识。

但 MoE 也带来工程问题：

1. 路由不均衡：很多 token 可能挤到少数专家，其他专家闲置。
2. 通信成本高：分布式训练时 token 要跨设备发送到专家所在位置。
3. 训练不稳定：router 学不好会导致专家塌缩或负载震荡。
4. 推理部署复杂：虽然单 token 激活少，但总参数需要存储，专家并行调度也复杂。

### 10.3.3 Switch Transformer：把 MoE 简化到 top-1

Switch Transformer 的核心思想是进一步简化 MoE 路由：每个 token 只选择一个专家，也就是 top-1 routing。

这样做的好处是：

1. 减少通信和计算。
2. 简化实现。
3. 更容易扩展到非常大的参数量。

但 top-1 也有代价。如果路由错了，token 没有第二个专家兜底；如果负载不均衡，专家容量限制会导致 token 被丢弃或退化处理。因此 Switch Transformer 需要负载均衡损失、容量因子、稳定训练技巧等配套设计。

面试中经常问：“MoE 为什么能更强？是不是因为每个 token 用了更多参数？”

推荐回答：

MoE 的优势主要来自总模型容量增加，而不是每个 token 激活更多参数。每个 token 通常只激活少量专家，计算量不一定比同级 dense 模型高很多。但模型整体可以容纳更多知识和模式。代价是路由、通信、负载均衡和部署复杂度上升。

### 10.3.4 MoE 与 dense scaling 的关系

MoE 不是 dense scaling 的简单替代，而是一种不同扩展路径。

Dense 模型优点是结构简单、训练稳定、推理部署直观。MoE 优点是参数容量大、训练计算效率高。选择 MoE 通常需要足够大的训练规模和工程能力，否则通信开销和不稳定性可能抵消收益。

可以从三个维度比较：

| 维度 | Dense | MoE |
| --- | --- | --- |
| 参数使用 | 每个 token 使用全部层参数 | 每个 token 使用部分专家 |
| 训练稳定性 | 通常更稳定 | router 和专家负载更复杂 |
| 部署 | 相对简单 | 需要专家并行和路由调度 |
| 容量扩展 | 计算随参数增加明显上升 | 可增加总参数但控制激活计算 |
| 适用场景 | 通用模型、中小规模工程 | 超大规模预训练、多域多语言 |

### 10.3.5 复现 MoE 论文时看什么

复现 MoE 论文不能只复现网络结构。关键观察点包括：

1. router entropy：路由是否过于集中。
2. expert load：每个专家处理 token 数是否均衡。
3. dropped tokens：容量限制导致多少 token 没被正常处理。
4. communication overhead：all-to-all 通信占比。
5. quality versus activated FLOPs：同等激活计算下是否优于 dense baseline。
6. stability：loss spike、专家塌缩、训练中断是否频繁。

如果论文只报告参数量巨大，但没有和同等计算量 dense baseline 比较，贡献说服力会不足。

## 10.4 RAG：把知识从参数里搬到外部记忆

### 10.4.1 RAG 解决什么问题

语言模型参数里存了大量知识，但参数记忆有明显缺点：

1. 更新困难：新知识需要继续训练或微调。
2. 来源不透明：很难知道答案来自哪里。
3. 幻觉风险：不知道时也可能编造。
4. 长尾知识不足：冷门领域、私有文档、企业知识库不一定在训练数据中充分出现。

RAG 的基本思路是：不要要求模型把所有知识都背在参数里，而是在生成前或生成中检索相关文档，把文档作为上下文提供给模型。

RAG 论文提出 retrieval-augmented generation，把 parametric memory 和 non-parametric memory 结合起来。parametric memory 是模型参数，non-parametric memory 是外部文档索引。

### 10.4.2 RAG 的基本流程

一个典型 RAG 系统包含：

1. 文档切分：把知识库切成 chunk。
2. 向量化：用 embedding 模型把 query 和 chunk 映射到向量空间。
3. 召回：用向量检索、关键词检索或混合检索找到候选片段。
4. 重排：用 cross-encoder 或 LLM reranker 重新排序。
5. 上下文构造：把高相关片段拼入 prompt。
6. 生成：模型基于检索内容回答。
7. 引用和校验：输出来源、检查答案是否被证据支持。

RAG 的关键不只是“把资料塞给模型”，而是让检索、排序、压缩和生成形成闭环。

### 10.4.3 RAG 与 long context 的关系

很多人会问：如果模型有 1M 上下文，是不是不需要 RAG？

答案通常是否定的。长上下文和 RAG 是互补关系。

Long context 解决“能放多少”的问题；RAG 解决“该放什么”的问题。

即使上下文很长，盲目塞入整个知识库仍然浪费计算，而且可能引入干扰。RAG 可以先筛选相关证据，再让 long context 承载更多候选、更多来源和更复杂证据链。

反过来，如果上下文很短，RAG 也会受限，因为召回到的证据放不下，或者放进去后没有足够空间让模型推理。

### 10.4.4 RAG 的失败模式

RAG 系统常见失败不一定来自生成模型，而可能来自链路前段。

典型失败模式包括：

1. 召回失败：正确文档没有被找出来。
2. 切分失败：答案证据被切碎，单个 chunk 不完整。
3. 排序失败：相关文档被排在后面，没进入上下文。
4. 上下文污染：检索到过期、冲突或低质量资料。
5. 生成不忠实：模型忽略证据，用参数记忆或猜测回答。
6. 引用错配：答案和引用文档并不一致。

因此评估 RAG 时要分层：检索 recall、rerank quality、answer correctness、citation faithfulness、end-to-end latency 都要看。

### 10.4.5 RAG 论文复现怎么做

复现 RAG 不建议一上来做巨大企业知识库。更好的最小闭环是：

1. 选一个公开 QA 数据集或自己构造小型文档集。
2. 建 BM25 baseline。
3. 加 dense retrieval。
4. 加 hybrid retrieval。
5. 加 reranker。
6. 比较不用检索、只用 top-1、top-k、rerank 后 top-k 的差异。
7. 分析错误：没有召回、召回但没用、用了但推理错。

面试中如果说自己做过 RAG，面试官往往会追问：chunk size 怎么定？top-k 怎么定？召回和生成哪个是瓶颈？如何处理冲突文档？如何评估 citation faithfulness？这些比“用了某个向量数据库”更重要。

## 10.5 Tool Use：让模型调用外部能力

### 10.5.1 为什么需要 Tool Use

语言模型擅长语言理解、模式匹配和自然语言生成，但不擅长所有事情。比如：

1. 精确算术。
2. 实时搜索。
3. 查询私有数据库。
4. 执行代码。
5. 操作业务系统。
6. 获取环境反馈。

Tool Use 的思想是：让模型在需要时调用外部工具，而不是把所有能力都压进参数。

这和人类工作方式类似。一个工程师不靠记忆完成所有任务，而是会查文档、跑测试、看日志、写脚本、问数据库。

### 10.5.2 Tool Use 的基本形式

工具调用通常包括三个环节：

1. 工具选择：判断是否需要工具，需要哪个工具。
2. 参数生成：把自然语言意图转换成结构化参数。
3. 结果整合：读取工具返回结果，继续推理或回答。

例如用户问：“统计过去七天新增用户数，并和上周同比。”模型不能凭空回答。它应该调用数据库查询工具，生成 SQL 或 API 参数，拿到结果后再组织解释。

Tool Use 的核心评价不是模型会不会输出函数名，而是能否在多步任务中稳定选择正确工具、生成正确参数、处理错误返回，并把结果忠实整合进答案。

### 10.5.3 Tool Use 与 RAG 的关系

RAG 可以看成一种特殊工具使用：检索器是工具，返回文档片段。但 Tool Use 范围更广，它可以调用计算器、浏览器、代码解释器、数据库、日历、机器人控制接口等。

区别是：

RAG 的工具通常是读知识；Tool Use 可以读、算、写、执行。

这也带来更高安全要求。写操作、外部系统操作、代码执行都需要权限控制、参数校验、审计和回滚机制。

本章只讨论合法、授权、受控环境中的工具调用和 Agent 研究，不讨论绕过权限、攻击系统或执行未授权操作的方法。

## 10.6 Agent Planning：从一次回答到多步行动

### 10.6.1 Agent 和普通 Chatbot 的区别

普通 Chatbot 多数时候是一次输入、一次输出。Agent 则更像一个循环系统：

1. 观察当前状态。
2. 制定或更新计划。
3. 选择动作。
4. 调用工具或与环境交互。
5. 接收反馈。
6. 继续下一步，直到完成任务或停止。

所以 Agent 的关键不是“模型会说自己要做什么”，而是能否稳定执行多步过程。

### 10.6.2 ReAct：Reasoning and Acting 交替

ReAct 是 Agent 论文线中的经典工作。它把 reasoning traces 和 actions 交替生成。推理轨迹帮助模型跟踪目标、更新计划、解释当前动作；动作让模型访问外部知识源或环境反馈。

一个简化模式是：

```text
Thought: 我需要找到问题中的关键实体。
Action: Search[实体]
Observation: 搜索结果...
Thought: 结果表明...
Action: Lookup[...]
Observation: ...
Answer: ...
```

ReAct 的价值在于，它不是单纯 chain-of-thought，也不是单纯动作生成，而是让“想”和“做”互相修正。对于开放域 QA，它可以通过检索减少幻觉；对于交互任务，它可以根据环境反馈调整下一步。

面试中可以这样概括：

ReAct 把语言模型从一次性推理器变成了可交互的问题求解器。它的关键机制是 interleaved reasoning and acting。优点是可解释、能利用外部反馈；缺点是轨迹长、成本高、容易在某一步错误后级联失败。

### 10.6.3 Tree of Thoughts：把推理变成搜索

Chain-of-thought 通常是一条线：模型生成一条推理路径，然后到达答案。如果早期一步走错，后面可能一路错下去。

Tree of Thoughts 把中间推理步骤看成“thought”，允许模型生成多个候选 thought，对它们进行评估，然后用搜索策略继续探索。这更接近传统 AI 中的搜索和规划。

它适合那些需要探索、回溯、全局选择的任务，比如 24 点、填字游戏、复杂规划、创意写作结构设计。

但 ToT 也有成本问题。生成多个分支、评估多个分支会显著增加 token 和调用次数。它适合高价值、复杂、需要 deliberation 的任务，不适合所有简单问答。

### 10.6.4 Agent 的工程难点

Agent 看起来很强，但工程落地非常难。常见难点包括：

1. 长程任务可靠性：十步里错一步就可能失败。
2. 状态管理：历史、工具结果、计划、约束需要结构化维护。
3. 停止条件：模型可能过早停止，也可能无限循环。
4. 错误恢复：工具失败、结果为空、权限不足时要能调整。
5. 成本控制：多轮调用会放大 token、延迟和费用。
6. 安全边界：外部动作需要权限、沙箱、审批和日志。
7. 评估困难：真实任务开放度高，很难用单一准确率评估。

所以 Agent 论文复现不能只展示几个 demo。更有价值的是构建可重复 benchmark，记录成功率、步骤数、工具错误率、恢复率、成本和失败类型。

## 10.7 这些论文线之间的组合

真实系统往往不是单独使用某一条线，而是组合使用。

### 10.7.1 Long Context + RAG

RAG 负责筛选候选证据，long context 负责容纳更多证据和上下文。适合长文档问答、代码仓库分析、法律审查。

关键问题是：检索到的内容如何排序，是否需要压缩，是否保留引用，如何处理相互矛盾的证据。

### 10.7.2 RAG + Tool Use

检索只能读静态知识，工具可以读动态系统。企业问答里常见组合是：先用 RAG 找制度文档，再调用数据库查用户状态，最后综合回答。

### 10.7.3 Agent + Tool Use + Memory

Agent 需要工具执行动作，也需要 memory 保存状态。短期 memory 可以放上下文，长期 memory 可以放向量库或结构化数据库。

但 memory 不是越多越好。错误记忆、过期记忆和无关记忆都会污染决策。

### 10.7.4 MoE + Long Context

MoE 提供大容量，long context 提供大输入。两者组合会带来更强模型，但也叠加系统复杂度：专家并行、attention kernel、KV cache、负载均衡、通信都要同时优化。

## 10.8 如何判断这类论文的贡献

读 long context、MoE、RAG、Agent 论文时，可以用一组共同问题判断贡献。

### 10.8.1 它解决的是模型问题、系统问题还是产品问题

FlashAttention 和 PagedAttention 更偏系统，但对模型能力落地非常关键。RAG 同时涉及模型和系统。Agent 论文往往涉及推理框架、提示、工具环境和评估。

不要用同一个标准评价所有论文。系统论文要看吞吐、延迟、显存、可扩展性；算法论文要看任务质量和泛化；Agent 论文要看多步成功率和失败分析。

### 10.8.2 baseline 是否公平

长上下文论文要比较相同模型、相同训练数据、相同上下文长度下的差异。MoE 要比较同等激活 FLOPs 或同等训练预算。RAG 要比较无检索、不同检索器、不同 top-k。Agent 要比较同等工具、同等调用预算和同等环境信息。

### 10.8.3 指标是否覆盖真实瓶颈

只看准确率不够。还要看成本、延迟、显存、稳定性、错误类型、安全边界。

例如 RAG 系统答案准确率高，但引用经常错，在法律或医疗场景就不可接受。Agent 成功率不错，但平均调用 100 次工具，线上成本可能不可接受。

### 10.8.4 是否有清晰 ablation

这类系统往往由多个模块组成。好的论文应该拆开验证每个模块贡献：没有 reranker 会怎样，没有 memory 会怎样，不同 top-k 会怎样，不同 router loss 会怎样，不同上下文长度会怎样。

没有 ablation，就很难知道提升来自核心方法还是工程细节。

## 10.9 面试高频问题

### 问题 1：Long context 和 RAG 谁会取代谁？

不会简单取代。Long context 解决可容纳信息量，RAG 解决信息选择和知识更新。长上下文越强，RAG 可以放入更多证据；RAG 越强，长上下文可以减少无关内容干扰。真实系统常常组合使用。

### 问题 2：FlashAttention 的本质贡献是什么？

FlashAttention 是 IO-aware exact attention。它没有改变 dense attention 的数学定义，也不是通过近似降低二次交互关系，而是通过 tiling、在线 softmax 和更好的 GPU 内存访问减少 HBM 读写，降低显存占用并提升实际速度。

### 问题 3：PagedAttention 和 FlashAttention 有什么区别？

FlashAttention 关注 attention kernel 内部如何高效计算；PagedAttention 关注推理服务中 KV cache 如何高效管理。前者更像算子优化，后者更像内存管理和服务系统优化。

### 问题 4：MoE 为什么参数多但计算不一定同等增加？

因为 MoE 采用稀疏激活。模型总参数可以很大，但每个 token 只路由到少量专家，所以激活计算量受控。代价是 router、专家负载均衡、跨设备通信和部署复杂度。

### 问题 5：Switch Transformer 为什么重要？

它把 MoE 路由简化为 top-1 routing，降低通信和计算复杂度，使稀疏专家模型更容易扩展到超大规模。同时它也强调了稳定训练、负载均衡和低精度训练的重要性。

### 问题 6：RAG 的主要失败来源是什么？

可能来自召回、切分、重排、上下文构造、生成忠实性或引用对齐。不能只看最终答案，要分层诊断。如果正确文档没召回，生成模型再强也很难答对；如果召回了但模型忽略证据，那是生成和上下文利用问题。

### 问题 7：ReAct 相比 chain-of-thought 多了什么？

Chain-of-thought 主要是内部推理轨迹，ReAct 把推理和行动交替起来。模型可以边想边调用工具或环境，再根据 observation 更新下一步。因此 ReAct 更适合开放域问答和交互决策任务。

### 问题 8：Tree of Thoughts 适合什么任务？

适合需要探索、回溯和全局选择的复杂任务。不适合所有任务，因为它需要生成和评估多个分支，成本更高。简单事实问答通常没必要用 ToT。

### 问题 9：Agent 为什么 demo 看起来强，线上落地难？

因为线上任务需要长程可靠性、状态管理、权限控制、错误恢复、成本控制和可评估性。demo 可以容忍人工筛选和偶然成功，生产系统必须稳定处理大量边界情况。

### 问题 10：如果让你复现一个 Agent 论文，你会怎么做？

先选可重复环境和明确指标，比如 WebShop、ALFWorld 或小型工具调用 benchmark。固定模型、工具、prompt、调用预算和随机种子。记录成功率、平均步骤数、工具错误率、超时率、成本和失败类型。然后做 ablation：去掉 reasoning、去掉工具、限制搜索深度、改变示例数量，观察差异。

## 10.10 本章小结

Long context、MoE、RAG、Tool Use 和 Agent Planning 是大模型从“单次文本生成器”走向“可扩展智能系统”的关键论文线。

Long context 让模型在一次推理中接收更多证据，但要解决 attention 成本、KV cache、位置外推和长程利用问题。

MoE 通过稀疏激活扩大总参数容量，但要处理路由、负载均衡、通信和稳定性。

RAG 把外部知识接入生成过程，解决知识更新、来源透明和长尾知识问题，但依赖检索质量和生成忠实性。

Tool Use 让模型调用外部能力，把语言模型变成系统接口的一部分，但需要结构化参数、错误恢复和安全控制。

Agent Planning 让模型从一次回答扩展到多步行动，但长程可靠性和评估仍是难点。

读这类论文时，不要只看“能力更强”的表面结论。更重要的是问：它把能力来源从哪里引入？代价转移到了哪里？有没有公平 baseline？有没有模块 ablation？真实系统中的瓶颈是否被覆盖？

能这样读论文，才算真正理解了这条研究线。
