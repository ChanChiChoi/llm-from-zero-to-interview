# 第七章：分布式训练

## 本章目标

理解大模型训练为什么必须分布式，以及各种并行方式解决什么瓶颈。

## 核心议题

1. Data Parallelism 与 DDP。
2. ZeRO 与 FSDP。
3. Tensor Parallelism。
4. Pipeline Parallelism。
5. Sequence Parallelism 与 Context Parallelism。
6. 3D parallelism。
7. 通信开销与计算重叠。

## 面试重点

面试中不要只背名词，要能说明每种并行方式切分了什么、节省了什么、增加了什么通信成本。

## 为什么大模型必须分布式训练

大模型训练通常无法只靠单张 GPU 完成。

原因有三类：

1. 参数太多，单卡放不下模型权重、梯度和 optimizer state。
2. batch 太大，单卡放不下激活和中间结果。
3. token 太多，单卡训练时间不可接受。

以 AdamW 为例，训练时不只需要保存模型参数，还需要梯度、一阶动量、二阶动量。再加上激活、临时 buffer、通信 buffer，实际显存开销远大于模型权重本身。

分布式训练的核心目标是：

```text
把参数、数据、激活或序列切到多张 GPU 上，同时尽量保持计算效率。
```

面试表达：分布式训练不是简单“多用几张卡”，而是在显存、计算、通信和工程复杂度之间做切分设计。

### 分布式训练的资料边界

按照 `WRITING_PLAN.md` 的要求，本章第二轮精修前核对了 PyTorch DistributedDataParallel / FSDP 文档、DeepSpeed ZeRO 论文与文档、Megatron-LM 的 tensor / pipeline parallelism 资料，以及 PyTorch 性能调优中关于梯度同步和通信重叠的说明。

本章聚焦分布式训练的架构口径：Data Parallel、DDP、ZeRO、FSDP、Tensor Parallel、Pipeline Parallel、Sequence / Context Parallel、Expert Parallel、通信开销、global batch 和 activation checkpointing。它不展开 NCCL 内部算法、完整 Megatron / DeepSpeed / FSDP 配置文件、真实集群拓扑调优、checkpoint 分片格式和生产训练平台实现。

### 训练状态显存口径

设模型参数量为 `P`，参数字节数为 `b_w`，梯度字节数为 `b_g`，AdamW 一阶和二阶状态合计字节数为 `b_o`。DDP 每张卡都保存完整训练状态，单卡训练状态显存近似是：

```math
M_{\mathrm{DDP}}\approx P(b_w+b_g+b_o)
```

如果数据并行规模是 `N_dp`，ZeRO / FSDP 分片后可以粗略写成：

```math
M_{\mathrm{ZeRO1}}\approx P(b_w+b_g)+\frac{Pb_o}{N_{\mathrm{dp}}}
```

```math
M_{\mathrm{ZeRO2}}\approx Pb_w+\frac{P(b_g+b_o)}{N_{\mathrm{dp}}}
```

```math
M_{\mathrm{ZeRO3}}\approx \frac{P(b_w+b_g+b_o)}{N_{\mathrm{dp}}}
```

真实显存还要加 activation、temporary buffer、communication bucket、fragmentation 和 framework overhead，但这些公式能帮助你解释为什么 ZeRO-3 / FSDP 更省显存，也为什么它们会带来更多 gather / scatter 通信。

## 1. 分布式训练要解决哪些瓶颈

大模型训练常见瓶颈包括：

| 瓶颈 | 表现 | 常见解决方式 |
| --- | --- | --- |
| 参数显存 | 模型权重、梯度、optimizer state 放不下 | ZeRO、FSDP、Tensor Parallel |
| 激活显存 | sequence length 或 batch 太大导致 OOM | activation checkpointing、sequence parallel |
| 计算时间 | 单卡训练太慢 | data parallel、tensor parallel、pipeline parallel |
| 通信开销 | 多卡同步慢，GPU 等待 | overlap、拓扑优化、通信压缩 |
| 数据加载 | GPU 空转 | dataloader 优化、预取、数据 shard |

不同并行方式解决的问题不同。不能只说“用了分布式”，要说清楚切分对象是什么。

## 2. Data Parallelism

Data Parallelism 是最直观的并行方式。

每张 GPU 都复制一份完整模型，但处理不同 mini-batch 数据。

```text
GPU 0: model copy + batch 0
GPU 1: model copy + batch 1
GPU 2: model copy + batch 2
GPU 3: model copy + batch 3
```

每张卡独立 forward/backward，然后同步梯度，再更新参数。

Data Parallel 的 global batch 可以写成：

```math
B_{\mathrm{global}}=
B_{\mathrm{micro}}
\cdot
A
\cdot
N_{\mathrm{dp}}
```

其中 `B_micro` 是每张卡的 micro-batch size，`A` 是 gradient accumulation steps，`N_dp` 是 data parallel rank 数。对于 LLM，更常用 token 口径：

```math
B_{\mathrm{tok}}=
B_{\mathrm{micro}}
\cdot
T_{\mathrm{seq}}
\cdot
A
\cdot
N_{\mathrm{dp}}
```

### 2.1 优点

1. 概念简单。
2. 工程成熟。
3. 适合模型能放进单卡的场景。
4. 能有效扩大 global batch size。

### 2.2 缺点

1. 每张卡都要保存完整模型。
2. 不能解决单卡放不下模型的问题。
3. 每步需要梯度 all-reduce。

面试表达：Data Parallel 切的是数据，不切模型。它提高吞吐，但不降低单卡模型显存。

## 3. DDP

PyTorch DistributedDataParallel，简称 DDP，是常见数据并行实现。

DDP 的基本流程：

```text
1. 每个 rank 持有一份模型副本
2. 每个 rank 读取不同数据 shard
3. forward 计算 loss
4. backward 得到梯度
5. 对梯度做 all-reduce
6. 每个 rank 用相同梯度更新参数
```

### 3.1 all-reduce 是什么

all-reduce 会把多个 GPU 上的梯度求和或求平均，并把结果广播回每个 GPU。

这样每张 GPU 的模型参数保持一致。

如果第 `r` 个 rank 的梯度是 `g_r`，DDP 同步后的平均梯度是：

```math
\bar g=
\frac{1}{N_{\mathrm{dp}}}
\sum_{r=1}^{N_{\mathrm{dp}}}g_r
```

ring all-reduce 的单 rank 通信量常用粗略估算是：

```math
C_{\mathrm{allreduce}}\approx
2\frac{N_{\mathrm{dp}}-1}{N_{\mathrm{dp}}}S_g
```

其中 `S_g` 是梯度 bucket 大小。这个公式的重点是：DDP 通信量和梯度大小同阶，GPU 数越多，通信和拓扑越重要。

### 3.2 DDP 的瓶颈

DDP 的主要瓶颈是梯度同步通信。

当模型很大、网络带宽不足或 GPU 数量很多时，all-reduce 可能成为训练吞吐瓶颈。

DDP 还不能解决模型本身放不下单卡的问题。

## 4. ZeRO

ZeRO 的核心思想是：不要让每张 GPU 都保存完整训练状态，而是把训练状态切分到不同 GPU 上。

训练状态包括：

1. optimizer state。
2. gradients。
3. parameters。

ZeRO 常分为三个阶段。

| 阶段 | 切分内容 | 解决什么 |
| --- | --- | --- |
| ZeRO-1 | optimizer state | 减少 AdamW 状态显存 |
| ZeRO-2 | optimizer state + gradients | 进一步减少梯度显存 |
| ZeRO-3 | optimizer state + gradients + parameters | 参数也切分，支持更大模型 |

### 4.1 ZeRO-1

ZeRO-1 只切 optimizer state。

对于 AdamW，这很有价值，因为一阶动量和二阶动量很占显存。

### 4.2 ZeRO-2

ZeRO-2 进一步切 gradient。

每张卡不需要保存完整梯度，只保存自己负责的部分。

### 4.3 ZeRO-3

ZeRO-3 连参数也切分。

这可以训练更大模型，但通信更复杂，因为 forward/backward 时需要按需 gather 参数。

面试表达：ZeRO 本质是把训练状态做 shard，换取显存下降，代价是通信和工程复杂度上升。

## 5. FSDP

FSDP，Fully Sharded Data Parallel，是 PyTorch 生态中常用的全切分数据并行方案。

它和 ZeRO-3 思想接近：参数、梯度、optimizer state 都可以 shard 到不同 GPU 上。

FSDP 的典型过程：

```text
forward 前：gather 当前层参数
forward 后：可释放不再需要的完整参数
backward 时：再次 gather / reduce-scatter 梯度
optimizer step：只更新本 rank 负责的 shard
```

### 5.1 FSDP 的优势

1. 支持训练超过单卡显存的大模型。
2. 和 PyTorch 生态结合较好。
3. 可以结合 activation checkpointing。
4. 适合中大型模型训练和微调。

### 5.2 FSDP 的代价

1. 通信更频繁。
2. 配置复杂。
3. wrap 策略影响性能。
4. checkpoint 保存和加载更复杂。

面试表达：FSDP 用更复杂的参数 gather 和 shard 机制换取显存节省，适合模型状态显存成为瓶颈的场景。

可以把 FSDP 的一个 layer 级过程抽象成：

```text
all-gather 参数 shard -> 计算 forward/backward -> reduce-scatter 梯度 -> 释放完整参数
```

所以它的收益主要来自“常驻显存低”，代价是每层附近的 gather / scatter 通信更频繁。wrap 粒度太细会导致通信碎片化，wrap 粒度太粗又可能显存峰值过高。

## 6. Tensor Parallelism

Tensor Parallelism 切的是模型内部的大矩阵计算。

例如一个线性层：

```text
Y = X W
```

如果 `W` 很大，可以把 `W` 按列或按行切到多张 GPU 上。

列切分可以写成：

```math
W=[W_1,W_2,\ldots,W_{N_{\mathrm{tp}}}]
```

```math
Y_i=XW_i
```

最后再按特征维拼接 `Y_i`。行切分则通常需要对部分结果做 reduce 或 all-reduce。TP 的核心是切层内矩阵，不是切样本。

### 6.1 为什么需要 TP

当单层矩阵太大，或者单卡算力不足时，TP 可以把一层的计算拆到多卡。

它常用于非常大的 dense model 或 MoE model。

### 6.2 TP 的通信

TP 通常需要在层内做 all-reduce、all-gather 或 reduce-scatter。

这意味着 TP 对 GPU 间带宽要求很高，通常更适合在同一节点内使用，例如 NVLink 连接的 GPU。

### 6.3 面试表达

Tensor Parallel 切的是层内矩阵和 hidden dimension，不是切数据。它能解决单层太大或单卡算力不足的问题，但会增加频繁的层内通信。

## 7. Pipeline Parallelism

Pipeline Parallelism 把模型按层切成多个 stage。

例如：

```text
GPU 0: layer 1-10
GPU 1: layer 11-20
GPU 2: layer 21-30
GPU 3: layer 31-40
```

一个 batch 会按 micro-batch 流过这些 stage。

### 7.1 为什么需要 PP

如果模型层数很多，整模型放不进单卡，可以按层切分。

PP 适合很深的模型。

### 7.2 Pipeline bubble

Pipeline 的问题是会有 bubble，也就是某些 GPU 等待其他 stage 的空闲时间。

micro-batch 越少，bubble 越明显。

常见做法是把 batch 切成多个 micro-batch，提高 pipeline 利用率。

如果 pipeline stage 数是 `N_pp`，micro-batch 数是 `M`，一个简化 bubble 比例是：

```math
r_{\mathrm{bubble}}\approx
\frac{N_{\mathrm{pp}}-1}{M+N_{\mathrm{pp}}-1}
```

micro-batch 越多，bubble 越小，但调度、激活缓存和通信也会更复杂。

### 7.3 面试表达

Pipeline Parallel 切的是层。它降低单卡模型显存，但会引入 pipeline bubble、调度复杂度和跨 stage 激活通信。

## 8. Sequence Parallelism 与 Context Parallelism

当 sequence length 很长时，激活和 attention 相关显存会成为瓶颈。

Sequence Parallelism 或 Context Parallelism 尝试沿序列维度切分计算。

### 8.1 Sequence Parallelism

Sequence Parallelism 通常把某些 activation 按 sequence dimension 分散到不同 GPU 上，减少每卡激活显存。

它经常和 Tensor Parallel 结合使用。

### 8.2 Context Parallelism

Context Parallelism 更关注长上下文训练，把长序列切到多张 GPU 上处理。

难点是 attention 需要跨序列位置交互，所以必须设计高效通信。

面试表达：序列并行解决的是长序列激活和 attention 显存问题，代价是跨序列通信和实现复杂度。

## 9. 3D Parallelism

真实大模型训练常常组合多种并行方式。

所谓 3D parallelism 通常指：

```text
Data Parallel + Tensor Parallel + Pipeline Parallel
```

也可以再叠加 ZeRO/FSDP、sequence parallel、expert parallel。

一个例子：

```text
总 GPU = DP * TP * PP
```

如果有 64 张 GPU，可以设计：

```text
DP = 8
TP = 4
PP = 2
总数 = 8 * 4 * 2 = 64
```

其中：

1. TP 负责切单层矩阵。
2. PP 负责切模型层。
3. DP 负责复制并行组并处理不同数据。

总 world size 可以写成：

```math
N_{\mathrm{world}}=
N_{\mathrm{dp}}
\cdot
N_{\mathrm{tp}}
\cdot
N_{\mathrm{pp}}
```

如果再叠加 expert parallel 或 context parallel，需要把对应维度也纳入 rank mapping。真实工程里 rank mapping 要和节点内 NVLink、节点间 InfiniBand 等拓扑配合。

面试表达：大规模训练通常不是单一并行策略，而是根据模型大小、节点拓扑和通信瓶颈组合多维并行。

## 10. Expert Parallelism

MoE 模型还会涉及 Expert Parallelism。

MoE 中不同 expert 可以分布在不同 GPU 上。每个 token 经过 router 后被发送到对应 expert。

这会带来 all-to-all 通信。

如果一共有 `E` 个 expert，每个 token 选择 `k` 个 expert，一个 batch 内 token 数是 `T_tok`，capacity factor 是 `alpha`，每个 expert 的容量可以粗略写成：

```math
C_{\mathrm{expert}}=
\left\lceil
\alpha
\frac{T_{\mathrm{tok}}k}{E}
\right\rceil
```

如果 router 分配不均，某些 expert 超过容量，就会发生 token drop、padding 浪费或负载不均。

### 10.1 MoE 的分布式难点

1. token 路由不均衡。
2. expert 负载不均衡。
3. all-to-all 通信昂贵。
4. batch size 太小时 expert 利用率差。
5. checkpoint 和推理部署更复杂。

面试表达：MoE 的分布式训练难点不只是参数多，而是 token 到 expert 的动态路由会带来负载均衡和 all-to-all 通信问题。

## 11. 通信开销与计算重叠

分布式训练的效率往往受通信影响。

常见通信操作：

| 操作 | 常见场景 |
| --- | --- |
| all-reduce | DDP 梯度同步 |
| reduce-scatter | ZeRO/FSDP 梯度切分 |
| all-gather | FSDP 参数 gather |
| all-to-all | MoE expert routing |
| send/recv | Pipeline stage 之间传激活 |

为了提高效率，需要尽量让通信和计算重叠。

例如在 backward 过程中，前面层的梯度算出来后可以先开始通信，同时继续计算后面层的梯度。

如果纯计算时间是 `T_comp`，通信时间是 `T_comm`，完全不重叠时：

```math
T_{\mathrm{step}}\approx T_{\mathrm{comp}}+T_{\mathrm{comm}}
```

如果可以重叠，理想情况下接近：

```math
T_{\mathrm{step}}\approx \max(T_{\mathrm{comp}},T_{\mathrm{comm}})
```

真实系统通常介于两者之间。面试里要能说明：bucket size、backward 顺序、网络拓扑和 collective 调度都会影响 overlap 效果。

面试表达：分布式训练的目标不是通信为零，而是减少通信瓶颈并让通信尽量和计算重叠。

## 12. Global batch size

分布式训练中，global batch size 的计算非常重要。

```text
global_batch = micro_batch_size * gradient_accumulation_steps * data_parallel_size
```

如果还使用 sequence packing，可以进一步按 token 数统计：

```text
tokens_per_step = global_batch * sequence_length
```

学习率、warmup、训练 token 数和吞吐都要基于 global batch 或 tokens per step 理解。

常见错误是只看单卡 batch size，忽略 data parallel 和 gradient accumulation 后的实际 global batch。

训练总 token 数为 `D` 时，optimizer step 数近似是：

```math
S_{\mathrm{opt}}=
\left\lceil
\frac{D}{B_{\mathrm{tok}}}
\right\rceil
```

这会影响学习率 schedule、checkpoint 间隔、validation 间隔和训练时长估算。

## 13. Activation checkpointing

Activation checkpointing 也叫 gradient checkpointing。

它的思路是 forward 时不保存所有中间激活，backward 时重新计算一部分 forward。

优点：

1. 显著降低激活显存。
2. 支持更大 batch 或更长 sequence。

代价：

1. backward 更慢。
2. 计算量增加。
3. checkpoint 粒度影响性能。

如果原始 activation 显存是 `M_act`，切成 `K` 段 checkpoint，一个粗略直觉是：

```math
M_{\mathrm{act,ckpt}}\approx \frac{M_{\mathrm{act}}}{K}
```

同时 backward 会重算部分 forward，额外计算因子可以粗略理解为：

```math
R_{\mathrm{compute}}>1
```

实际节省和额外计算取决于 checkpoint 粒度、模型结构和框架实现。

面试表达：activation checkpointing 是用额外计算换激活显存，常和 FSDP、TP、长上下文训练一起使用。

## 14. 分布式训练常见故障

### 14.1 吞吐不随 GPU 数增加

可能原因：

1. 通信成为瓶颈。
2. batch 太小。
3. dataloader 跟不上。
4. pipeline bubble 太大。
5. 节点间网络慢。

### 14.2 某些 GPU 显存明显更高

可能原因：

1. pipeline stage 切分不均。
2. MoE expert 负载不均。
3. rank 0 保存额外对象。
4. checkpoint 或 logging 集中在某个 rank。

### 14.3 训练 hang 住

可能原因：

1. 某个 rank 报错但其他 rank 等待通信。
2. collective 调用顺序不一致。
3. 数据 shard 长度不一致。
4. 网络故障。

### 14.4 loss 不一致

可能原因：

1. 随机种子不一致。
2. 数据 shard 重叠或缺失。
3. gradient accumulation 逻辑错误。
4. mixed precision 配置不一致。

## 15. 如何选择并行策略

可以按下面顺序思考：

```text
1. 模型能不能放进单卡？
   能：优先 DDP。
   不能：考虑 FSDP/ZeRO 或 TP/PP。

2. 单层矩阵是否太大？
   是：考虑 Tensor Parallel。

3. 层数是否很多？
   是：考虑 Pipeline Parallel。

4. 序列是否很长？
   是：考虑 activation checkpointing、sequence/context parallel。

5. 是否是 MoE？
   是：考虑 expert parallel 和 all-to-all 优化。

6. 通信拓扑如何？
   节点内适合高通信 TP，跨节点更谨慎。
```

面试中可以强调：并行策略要根据瓶颈选择，而不是所有技术都堆上去。

下面这个 demo 用标准库审计一个 64 GPU 的分布式训练配置。它不启动真实分布式进程，而是检查 world size、DDP/ZeRO 显存、ring all-reduce 流量、global token batch、pipeline bubble、MoE expert capacity 和 activation checkpointing 取舍。

```python
import math

cfg = {
    "params": 6_000_000_000,
    "weight_bytes": 2,
    "grad_bytes": 2,
    "adam_state_bytes": 8,
    "dp": 8,
    "tp": 4,
    "pp": 2,
    "micro_batch": 2,
    "seq_len": 4096,
    "accum_steps": 4,
    "grad_bucket_gib": 4.0,
    "bandwidth_gbps": 200,
    "micro_batches_for_pp": 8,
    "tokens": 8192,
    "experts": 8,
    "top_k": 2,
    "capacity_factor": 1.25,
}

GIB = 1024**3


def gib(bytes_value):
    return bytes_value / GIB


def memory_estimates(c):
    p = c["params"]
    w = p * c["weight_bytes"]
    g = p * c["grad_bytes"]
    o = p * c["adam_state_bytes"]
    n = c["dp"]
    return {
        "ddp_gib": gib(w + g + o),
        "zero1_gib": gib(w + g + o / n),
        "zero2_gib": gib(w + (g + o) / n),
        "zero3_gib": gib((w + g + o) / n),
        "weight_gib": gib(w),
        "grad_gib": gib(g),
        "adam_state_gib": gib(o),
    }


def ring_all_reduce_traffic_gib(payload_gib, ranks):
    return 2 * (ranks - 1) / ranks * payload_gib


def transfer_seconds(gib_value, gbps):
    bytes_value = gib_value * GIB
    return bytes_value / (gbps * 1e9)


def pipeline_bubble_fraction(pp, micro_batches):
    return (pp - 1) / (micro_batches + pp - 1)


def expert_capacity(tokens, top_k, experts, capacity_factor):
    return math.ceil(capacity_factor * tokens * top_k / experts)


mem = memory_estimates(cfg)

capacity = expert_capacity(
    cfg["tokens"],
    cfg["top_k"],
    cfg["experts"],
    cfg["capacity_factor"],
)

activation_gib = 24.0
checkpoint_segments = 4
checkpointed_activation_gib = activation_gib / checkpoint_segments
extra_compute_factor = 1.0 + 1.0 / checkpoint_segments

checks = {
    "world_size_matches": world_size == 64,
    "zero3_saves_memory": mem["zero3_gib"] < mem["zero2_gib"] < mem["zero1_gib"] < mem["ddp_gib"],
    "effective_tokens_ok": effective_tokens == 262_144,
    "bubble_under_15pct": bubble < 0.15,
    "expert_capacity_enough": capacity * cfg["experts"] >= cfg["tokens"] * cfg["top_k"],
    "checkpoint_saves_activation": checkpointed_activation_gib < activation_gib,
}

print("world_size=", world_size)
print("memory_gib=", {k: round(v, 2) for k, v in mem.items()})
print("zero3_vs_ddp_ratio=", round(mem["zero3_gib"] / mem["ddp_gib"], 3))
print("effective_tokens=", effective_tokens)

print("all_reduce_time_s=", round(all_reduce_time_s, 4))

print("expert_capacity=", capacity)
print("expert_capacity_total=", capacity * cfg["experts"])

print("checks=", checks)
print("gate_pass=", all(checks.values()))
```

## 16. 面试官会怎么问

### 问法 1：DDP、ZeRO、FSDP 有什么区别？

可以这样答：

```text
DDP 是数据并行，每张卡保留完整模型和 optimizer state，只同步梯度，所以简单但不能解决单卡放不下模型的问题。ZeRO 把 optimizer state、gradient、parameter 分阶段 shard，降低显存。FSDP 类似 ZeRO-3，把参数、梯度和优化器状态分片，forward/backward 时按需 gather 参数，显存省但通信更复杂。
```

### 问法 2：Tensor Parallel 和 Pipeline Parallel 切分什么？

可以这样答：

```text
Tensor Parallel 切的是层内矩阵计算，比如 attention 或 MLP 的大矩阵，所以通信频繁但适合单层很大的模型。Pipeline Parallel 切的是模型层，把不同层放到不同 GPU 上，适合很深的模型，但会有 pipeline bubble 和 stage 间激活通信。
```

### 问法 3：ZeRO-3 为什么省显存但通信更大？

可以这样答：

```text
ZeRO-3 连参数也分片，每张 GPU 只保存部分参数、梯度和 optimizer state，所以显存显著下降。但 forward/backward 需要按层 gather 当前计算所需参数，计算后再释放或 reduce-scatter，因此通信更频繁。
```

### 问法 4：为什么 MoE 分布式训练难？

可以这样答：

```text
MoE 中 token 会被 router 动态分配到不同 expert。不同 expert 负载可能不均衡，而且 token 需要跨 GPU 发送到对应 expert，带来 all-to-all 通信。训练时还要处理 load balance loss、expert capacity、drop token 和 checkpoint 等问题。
```

### 问法 5：训练吞吐低怎么排查？

可以这样答：

```text
我会先分解时间：dataloader、forward、backward、optimizer、communication、checkpoint。再看 GPU utilization、tokens/sec、通信耗时、网络拓扑、batch size、pipeline bubble 和是否发生频繁同步。如果 GPU 空转，可能是数据加载或通信瓶颈；如果显存接近上限，可能需要 checkpointing、FSDP 或调整 batch。
```

## 17. 本章小结

本章核心结论：

1. 分布式训练解决的是显存、计算时间和吞吐瓶颈。
2. Data Parallel 切数据，不切模型，不能解决单卡放不下模型的问题。
3. DDP 通过梯度 all-reduce 保持模型副本一致。
4. ZeRO/FSDP 通过切分 optimizer state、gradient 和 parameter 降低显存。
5. Tensor Parallel 切层内矩阵，适合单层很大的模型。
6. Pipeline Parallel 切模型层，适合很深的模型，但有 pipeline bubble。
7. Sequence/Context Parallel 面向长序列训练的激活和 attention 瓶颈。
8. MoE 需要 expert parallel，并面对 all-to-all 和负载均衡问题。
9. 分布式训练效率高度依赖通信、拓扑、batch size 和计算通信重叠。
10. 面试中要围绕“切了什么、省了什么、增加了什么通信”来回答。
