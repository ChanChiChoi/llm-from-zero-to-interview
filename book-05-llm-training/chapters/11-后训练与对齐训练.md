# 第十一章：后训练与对齐强化训练

## 本章定位

预训练得到的是 base model，它具备语言建模能力和世界知识，但通常还不能直接作为可靠助手使用。

后训练的目标是把 base model 变成能遵循指令、能对话、能推理、能调用工具、能遵守安全边界，并且符合人类偏好的 assistant model 或 reasoning model。

过去很多教材会把后训练简化成：

```text
SFT -> RLHF/PPO -> DPO
```

这个说法已经不够用了。

当前真实训练路线更接近：

```text
SFT / cold start
-> reward modeling / rule-based reward / verifiable reward
-> online RL or offline preference optimization
-> long-CoT / reasoning RL
-> safety alignment
-> regression eval and iterative data refresh
```

因此，本章不能只讲 DPO。DPO 是重要方法，但它更多是离线偏好优化路线的一部分，不代表当前主流 RL 后训练全貌。

本章会系统梳理 PPO、GRPO、GSPO、DAPO、DCPO、DPO 及相关方法，重点讲清它们分别解决什么问题、依赖什么数据、优化什么目标、优缺点是什么、适合什么场景，以及面试中如何表达。

## 本章目标

学完本章，你需要掌握：

1. 后训练为什么不等于简单 SFT。
2. SFT、RM、PPO、DPO、GRPO、GSPO、DAPO、DCPO 的位置关系。
3. 为什么早期 RLHF 常用 PPO。
4. 为什么 reasoning model 训练中会出现 GRPO、DAPO 等方法。
5. DPO 为什么方便，但为什么不能代表所有主流对齐强化训练。
6. 各类方法对数据、reward、reference model、critic、采样和训练稳定性的要求。
7. 面试中如何讲清现代后训练 pipeline。

## 为什么后训练是模型变成助手的关键阶段

预训练得到的 base model 主要学到的是语言建模能力。

它知道很多文本模式、知识模式和推理模式，但通常还不适合直接面对用户。

原因包括：

1. 不一定遵循指令。
2. 不一定会拒答危险请求。
3. 不一定知道输出格式要求。
4. 不一定会优先给出帮助性、简洁性和安全性兼顾的回答。
5. 不一定会根据反馈持续改善。

后训练的目标是把 base model 变成 assistant model 或 reasoning model。

面试表达：预训练负责“学会语言和知识”，后训练负责“学会怎么当助手”。

## 1. 后训练的总体分层

后训练可以分为四个大阶段：

```text
1. SFT / cold start
2. Preference / reward learning
3. RL or offline preference optimization
4. Safety alignment and regression refresh
```

不同团队的名字不一样，但本质都在围绕这四类能力：

1. 指令遵循。
2. 偏好优化。
3. 可验证推理和探索。
4. 安全边界和稳定性。

## 2. SFT：监督微调

SFT 是最基础、最常见的后训练方式。

它使用 instruction-response 数据，让模型学会按指令回答。

### 2.1 SFT 解决什么问题

base model 的问题是“会续写，但不一定会当助手”。

SFT 要解决的是：

```text
用户问什么，模型就按要求回答什么。
```

### 2.2 SFT 的训练目标

SFT 通常仍然是 next-token cross entropy，但输入格式变成指令对话或多轮对话。

```text
maximize log p_theta(answer | prompt)
```

### 2.3 SFT 数据格式

SFT 数据常见结构：

```json
{
  "messages": [
    {"role": "system", "content": "你是一个严谨的助手。"},
    {"role": "user", "content": "解释什么是过拟合。"},
    {"role": "assistant", "content": "过拟合是指..."}
  ]
}
```

训练时只对 assistant 输出部分计算 loss。

### 2.4 SFT 的优点

1. 简单稳定。
2. 工程门槛低。
3. 适合作为所有后训练的起点。
4. 能快速建立格式和指令遵循能力。

### 2.5 SFT 的局限

1. 只能模仿示范。
2. 不直接比较多个回答谁更好。
3. 很容易过拟合格式。
4. 数据质量差会导致模型变啰嗦或模板化。

面试表达：SFT 负责把 base model 拉到“能对话、能按格式回答”的起点，但它不直接解决偏好排序和更优策略优化。

## 3. Reward Model

Reward Model 学习把人类偏好变成一个标量分数。

输入通常是：

```text
prompt + response
```

输出是一个分数 `r`。

### 3.1 训练数据

偏好对：

```text
prompt
chosen
rejected
```

模型要学到：

```text
r(prompt, chosen) > r(prompt, rejected)
```

### 3.2 Reward Model 的作用

1. 给回答打分。
2. 支持 PPO 这类在线 RL。
3. 帮助把复杂偏好压缩成可优化信号。

### 3.3 局限

1. 标注偏差会被学进去。
2. 容易被 reward hacking。
3. 与真实用户偏好可能不完全一致。

面试表达：Reward Model 把相对偏好变成可优化信号，但它本身也会被偏好噪声和分布偏移影响。

## 4. PPO-RLHF

PPO 是早期 RLHF 的经典算法。

基本流程：

```text
SFT model -> sample responses -> reward model score -> PPO update policy
```

还会加上 KL 约束，防止策略偏离 reference model 太远。

### 4.1 PPO 为什么重要

它能在线采样，优化模型自己真实生成出来的分布。

这使得它比纯监督学习更接近实际 rollout 行为。

### 4.2 PPO 的问题

1. 需要 policy、reference、reward，有时还要 critic。
2. 工程复杂。
3. 成本高。
4. 稳定性差。
5. 容易出现 reward hacking 或 KL 崩掉。

### 4.3 什么时候适合 PPO

1. reward 信号明确。
2. 需要在线探索。
3. 需要强约束控制策略偏移。

面试表达：PPO-RLHF 是经典在线对齐路线，但工程复杂度和训练成本高，所以后来出现了很多更简单的离线偏好优化方法。

## 5. DPO：直接偏好优化

DPO 的核心思想是：能不能绕开显式 reward model 和复杂 PPO，直接用偏好数据优化语言模型？

DPO 使用：

```text
(prompt, chosen, rejected)
```

直接让 chosen 比 rejected 更可能。

### 5.1 DPO 的直觉

可以把它理解为一种偏好排序训练。

模型不需要先单独学一个 reward model，而是直接在 policy 上优化偏好差异。

### 5.2 DPO 的优点

1. 工程简单。
2. 不需要显式 reward model。
3. 训练更稳定。
4. 成本比 PPO 低。

### 5.3 DPO 的局限

1. 依赖离线偏好数据。
2. 不能很好处理复杂探索。
3. 对长链 reasoning 和可验证任务不一定最优。
4. 受数据质量影响很大。

面试表达：DPO 很重要，但它解决的是离线偏好优化问题，不代表所有后训练问题都能用 DPO 解决。

## 6. IPO、KTO、ORPO 等偏好优化变体

在 DPO 之后，社区又发展出很多偏好优化方法。

它们大多围绕以下目标：

1. 更稳定。
2. 更简单。
3. 更少超参。
4. 更好的样本效率。
5. 更好的对齐行为。

你不一定要背所有缩写，但要知道它们属于离线偏好优化谱系。

面试表达：如果面试官问 DPO 之外的偏好优化方法，可以先说明它们大多是为了提高稳定性、降低工程复杂度或改善偏好学习效果。

## 7. Reasoning RL

在数学、代码、逻辑、工具调用等任务上，越来越多训练开始使用可验证 reward。

例如：

1. 数学题答案是否正确。
2. 代码是否通过测试。
3. 工具调用结果是否满足约束。

这类任务适合做 reasoning RL。

### 7.1 为什么 reasoning 任务适合 RL

因为 reward 可以比传统人类主观偏好更明确。

如果答案是对就是对，错就是错，就可以直接给 reward。

### 7.2 Reasoning RL 的困难

1. 奖励稀疏。
2. 采样成本高。
3. 训练不稳定。
4. 容易过度优化某一类题。

### 7.3 相关方法

GRPO、GSPO、DAPO、DCPO 等方法通常都出现在 reasoning RL 或偏好强化训练讨论里。

这些方法的共性是：试图比传统 PPO 更稳定、更高效，或者更适合长链推理和组内比较。

面试表达：reasoning RL 关注的是可验证任务上的策略优化，不只是让回答更像人类偏好，而是让答案真的更正确。

## 8. GRPO、GSPO、DAPO、DCPO 怎么理解

你不一定要死记每个缩写的全部公式，但要理解它们的设计动机。

### 8.1 GRPO

GRPO 常见于 group-based 或 group-relative 的训练思路。

直觉上是：

```text
同一个 prompt 采样一组回答
在组内比较好坏
用组内相对信号更新策略
```

它常用于 reasoning 场景。

### 8.2 GSPO

GSPO 常常强调 sequence-level 或 group-level 的稳定优化。

你可以把它理解为：

```text
把整段序列作为优化对象，而不是只看单 token。
```

### 8.3 DAPO

DAPO 这类方法通常是在 policy optimization 上做更稳定、更高效的改造。

它们常见目标包括：

1. 提高训练稳定性。
2. 降低采样成本。
3. 提升长链推理能力。

### 8.4 DCPO

DCPO 往往强调 contrastive 或 constrained preference 视角。

可以理解为：

```text
不仅要学会偏好排序，还要把约束、安全或行为边界纳入优化。
```

### 8.5 面试时怎么讲

不要只说缩写，要讲：

```text
这些方法主要是在解决 PPO 的成本高、稳定性差，或 DPO 不适合在线探索和复杂 reasoning 的问题。
```

## 9. 安全对齐

安全对齐和一般偏好优化不同。

它更关注：

1. 越狱。
2. 危险内容。
3. 隐私泄漏。
4. 过度拒答。
5. 偏见。
6. 工具误用。
7. 高风险领域的幻觉。

### 9.1 安全对齐不是单一 loss

真正的安全对齐通常是组合拳：

1. 安全 SFT。
2. 拒答样本。
3. 红队数据。
4. policy 约束。
5. safety eval。
6. 上线门禁。

### 9.2 过度拒答问题

如果安全数据配比过重，模型可能对正常问题也拒答。

所以安全和有用性之间要平衡。

面试表达：安全对齐不是简单把模型变得更保守，而是在有用性和风险之间找到合适的边界。

## 10. 后训练数据质量

后训练效果非常依赖数据质量。

SFT 数据如果太模板化，模型就会变啰嗦。

偏好数据如果标准不一致，模型就会学到不稳定偏好。

reward data 如果偏向某类风格，reward model 会被带偏。

### 10.1 常见问题

1. chosen 只是更长，不是真的更好。
2. rejected 质量太差，偏好信号太容易。
3. 偏好标签不一致。
4. 安全数据和有用性数据比例失衡。
5. 数据分布太窄。

### 10.2 质量控制

1. 人工复核抽样。
2. 做 pairwise 统计。
3. 按 category 划分。
4. 检查重复和冲突。
5. 与评估集隔离。

## 11. 后训练的评估

后训练不能只看训练 loss。

更重要的是：

1. 指令遵循是否提升。
2. 格式是否稳定。
3. 偏好胜率是否提升。
4. 安全拒答是否合理。
5. 通用能力是否退化。
6. reasoning 能力是否真的变强。

常见指标：

| 指标 | 作用 |
| --- | --- |
| win rate | pairwise 偏好胜率 |
| response length | 检查是否过长或过短 |
| refusal rate | 安全拒答是否合理 |
| jailbreak success rate | 越狱是否降低 |
| hallucination rate | 幻觉是否增加 |
| regression cases | 是否退化 |

面试表达：后训练评估要同时看 helpfulness、harmlessness 和 capability regression。

## 12. 一个典型现代后训练 pipeline

可以把现代后训练理解为：

```text
base model
  -> SFT 冷启动
  -> 偏好数据构造
  -> RM 或直接 preference optimization
  -> reasoning / verifiable reward training
  -> safety alignment
  -> regression eval
  -> 数据刷新和再训练
```

这个 pipeline 不是一次性完成，而是循环迭代。

评估发现问题后，往往要返回数据层重新补数据。

## 13. 常见误区

### 13.1 误区：后训练就是 DPO

不对。DPO 只是离线偏好优化的重要方法之一。

### 13.2 误区：PPO 太旧了就没用了

不对。PPO 在某些需要在线探索和 reward model 的场景仍然有价值。

### 13.3 误区：偏好数据越多越好

不对。偏好数据的质量、覆盖和一致性更重要。

### 13.4 误区：安全训练越多越安全

不对。过度安全化会导致有用性下降和过度拒答。

### 13.5 误区：训练目标和评估目标天然一致

不对。很多时候需要专门做 regression eval 和人工评测。

## 14. 面试官会怎么问

### 问法 1：后训练和预训练有什么区别？

可以这样答：

```text
预训练主要通过 next-token prediction 学语言、知识和通用模式，得到 base model。后训练则在 base model 上继续做 SFT、偏好优化、RLHF 或安全对齐，让模型变成更会遵循指令、更符合偏好、更安全的 assistant model 或 reasoning model。后训练关注的是行为塑形，而不仅是语言建模。
```

### 问法 2：PPO 和 DPO 的区别是什么？

可以这样答：

```text
PPO-RLHF 是在线 RL，通常需要 reward model、reference model，甚至 critic，通过采样和 KL 约束来更新 policy，优点是能直接优化当前模型的生成分布，缺点是工程复杂、成本高、稳定性差。DPO 是离线偏好优化，直接用 prompt/chosen/rejected 对优化模型，不需要显式 reward model，也不需要复杂 online rollout，工程更简单更稳定，但对离线偏好数据依赖更强。
```

### 问法 3：为什么 reasoning 任务常用 RL 或 group-based 方法？

可以这样答：

```text
因为数学、代码和工具调用这类任务往往有可验证 reward，比如答案对错、测试是否通过、约束是否满足。相比纯 SFT，这类任务更适合通过在线采样和 reward 优化来提升真正的正确率。GRPO、DAPO、GSPO 这类方法通常就是为了解决 reasoning RL 的稳定性、效率和组内比较问题。
```

### 问法 4：安全对齐怎么做？

可以这样答：

```text
安全对齐不是单一 loss，而是数据、reward、拒答样本、红队评估和上线门禁的组合。我会先在 SFT 阶段加入安全示例和拒答格式，再通过偏好优化或 RL 强化安全边界，同时用 jailbreak、安全和 regression eval 检查是否过度拒答或能力退化。
```

### 问法 5：为什么不能只做 SFT？

可以这样答：

```text
因为 SFT 只能模仿示范，不能直接比较多个回答谁更好，也不擅长在线探索和复杂约束优化。它能把 base model 拉到可用助手起点，但要进一步优化偏好、安全和 reasoning，通常还需要 reward learning、偏好优化或 RL。
```

## 15. 本章小结

本章核心结论：

1. 后训练的目标是把 base model 变成可用助手或 reasoning model。
2. SFT 解决指令遵循，但不能直接优化“哪个回答更好”。
3. Reward Model 把偏好转成标量信号，但也会受偏好噪声影响。
4. PPO-RLHF 是经典在线对齐路线，工程复杂但能在线探索。
5. DPO 是重要的离线偏好优化方法，简单稳定，但不覆盖所有后训练场景。
6. Reasoning RL 适合可验证任务，如数学、代码和工具调用。
7. GRPO、GSPO、DAPO、DCPO 等方法主要在解决稳定性、效率和约束优化问题。
8. 安全对齐需要数据、reward、拒答、红队和门禁组合。
9. 后训练数据质量对最终行为影响极大。
10. 面试时要把后训练讲成一个从 SFT 到偏好、RL、安全和回归的迭代系统。

## 1. 后训练总体流程

一个典型 LLM 后训练流程可以拆成六层。

### 1.1 SFT：建立基本指令遵循能力

SFT 使用人工或模型生成的高质量 instruction-response 数据，让模型学会按照用户指令回答。

它解决的问题是：

```text
base model 只会续写，不一定会按用户意图回答。
```

SFT 的目标通常是 next-token cross entropy：

```text
max log p_theta(y | x)
```

其中 `x` 是 prompt，`y` 是标准回答。

SFT 的优点是简单稳定。

缺点是它只能模仿数据，不会主动优化“哪个回答更好”。

### 1.2 Reward Modeling：学习偏好信号

Reward Model 使用偏好数据训练一个打分模型。

偏好数据通常长这样：

```text
prompt
chosen response
rejected response
```

Reward Model 学的是：

```text
r(prompt, chosen) > r(prompt, rejected)
```

它解决的问题是：

```text
如何把人类偏好变成可优化的标量 reward。
```

### 1.3 RLHF：用 RL 优化策略模型

有了 reward 后，可以把语言模型看作 policy。

模型生成回答，reward model 给分，训练算法更新 policy，使它更倾向于生成高 reward 回答。

经典路线是 PPO-RLHF。

### 1.4 Offline Preference Optimization：离线偏好优化

DPO、IPO、KTO、ORPO 等方法不一定显式训练 reward model，也不一定在线 rollout。

它们直接用偏好对优化模型。

优点是工程简单、稳定、成本低。

缺点是受限于离线偏好数据，对复杂 reasoning、长链条探索和 verifiable reward 场景不一定够。

### 1.5 Reasoning RL：面向可验证任务的强化训练

数学、代码、逻辑推理、工具调用等任务可以用规则或程序验证答案。

例如：

```text
数学题答案是否正确
代码是否通过单元测试
工具调用结果是否满足目标
```

这类任务可以使用 rule-based reward 或 verifiable reward，进行在线采样和 RL 优化。

GRPO、DAPO、GSPO、DCPO 等方法通常出现在这一类讨论中。

### 1.6 Safety Alignment：安全与行为边界

后训练还必须控制：

```text
危险请求
隐私泄露
越狱
偏见
过度拒答
事实幻觉
工具误用
```

安全对齐不是单独一个 loss 就能解决，而是数据、reward、拒答策略、红队评估和上线门禁的组合。

## 2. 方法谱系总览

可以先用一张表建立全局视角。

| 方法 | 类型 | 是否需要 Reward Model | 是否在线采样 | 是否需要 Critic | 典型用途 |
|---|---|---|---|---|---|
| SFT | 监督微调 | 否 | 否 | 否 | 指令遵循、冷启动 |
| PPO-RLHF | 在线 RL | 通常需要 | 是 | 通常需要 | 经典人类偏好 RLHF |
| DPO | 离线偏好优化 | 否 | 否 | 否 | 偏好对齐、风格优化 |
| IPO/KTO/ORPO | 离线偏好优化 | 否 | 否 | 否 | DPO 替代或变体 |
| GRPO | 在线 RL / group-based RL | 可用规则 reward | 是 | 通常不需要 critic | reasoning RL、数学代码任务 |
| GSPO | sequence-level policy optimization | 可用 reward | 是 | 依实现而定 | 序列级稳定优化、长输出对齐 |
| DAPO | decoupled / advanced policy optimization | 可用 reward | 是 | 依实现而定 | 长链推理 RL、稳定性和效率优化 |
| DCPO | contrastive / constrained preference optimization | 可用偏好或约束信号 | 可在线或离线 | 依实现而定 | 偏好、约束、安全和行为控制 |

注意：不同论文或团队对 GSPO、DAPO、DCPO 的全称和实现细节可能不同，面试中不要只背缩写。更重要的是讲清：它们是在 PPO/DPO 之外，为了解决 reasoning RL、长序列优化、稳定性、样本效率和约束对齐问题而发展出的后训练方法。

## 3. PPO-RLHF

PPO 是早期 RLHF 的代表方法。

整体流程如下：

```text
1. 用 SFT 得到初始 policy model。
2. 训练 reward model。
3. policy model 对 prompt 采样回答。
4. reward model 给回答打分。
5. PPO 根据 reward 更新 policy。
6. 使用 KL penalty 防止 policy 偏离 reference model 太远。
```

PPO 的目标大致可以理解为：

```text
maximize reward(response) - beta * KL(policy || reference)
```

其中 KL 项非常重要。

如果没有 KL 约束，模型可能为了骗 reward model 而生成奇怪回答，也就是 reward hacking。

PPO 的优点：

1. 能在线采样，优化当前模型真实生成分布。
2. 可以直接使用 reward model 或规则 reward。
3. 适合需要探索和迭代优化的任务。
4. 是经典 RLHF 路线，理论和工程经验较多。

PPO 的缺点：

1. 工程复杂，需要 policy、reference、reward、value/critic 多个模型或模块。
2. 显存和计算成本高。
3. 训练不稳定，容易 reward hacking、KL 崩掉、长度偏置。
4. 对 reward model 质量高度敏感。

面试中可以这样说：

```text
PPO-RLHF 是经典在线 RLHF 方法，优势是能在模型自己的采样分布上直接优化 reward，缺点是系统复杂、成本高、稳定性差，需要 reward model、reference model、critic 和 KL 控制。它适合 reward 明确且需要在线探索的场景，但不是所有后训练都必须用 PPO。
```

## 4. DPO：直接偏好优化

DPO 的核心动机是：能不能不显式训练 reward model，也不跑复杂 PPO，直接用偏好对训练模型？

DPO 使用数据：

```text
(prompt, chosen, rejected)
```

它直接优化模型，让 chosen 的相对概率高于 rejected。

直觉上：

```text
如果人更喜欢 chosen，就提高模型生成 chosen 的概率，降低 rejected 的概率。
```

DPO 的优点：

1. 工程简单。
2. 不需要在线 rollout。
3. 不需要显式 reward model。
4. 训练稳定性通常好于 PPO。
5. 适合偏好数据充足、目标相对明确的场景。

DPO 的缺点：

1. 依赖离线偏好数据质量。
2. 不能充分探索当前 policy 的新输出。
3. 对复杂 long-CoT reasoning 的优化能力有限。
4. 容易学到风格偏好，而不一定提升真实任务正确率。
5. 对 verifiable reward、代码执行 reward、数学答案 reward 这类场景不如在线 RL 灵活。

为什么说“DPO 目前不是全部主流”？

因为很多 reasoning model 后训练关注的是：

```text
模型自己采样多条推理链
用规则或程序验证结果
根据正确性、格式、长度、过程质量做 reward
继续优化 policy
```

这类训练天然更接近在线 RL，而不是只用固定偏好对的 DPO。

## 5. GRPO：Group Relative Policy Optimization

GRPO 可以理解为一种面向 group sampling 的 RL 优化方法。

它常见于 reasoning RL 场景。

基本思想是：对同一个 prompt，模型采样多条回答，形成一个 group：

```text
prompt -> response_1, response_2, ..., response_k
```

然后用 reward 给每条回答打分：

```text
r_1, r_2, ..., r_k
```

再用组内相对优势更新模型。

直觉是：

```text
同一道题里，奖励更高的回答应该被增强，奖励更低的回答应该被压低。
```

GRPO 相比 PPO 的一个重要特点是：通常不需要单独训练 critic/value model，而是用 group 内 reward 做 baseline 或 advantage 估计。

它适合：

1. 数学推理。
2. 代码生成。
3. 可验证 QA。
4. 有明确对错 reward 的任务。
5. 同一 prompt 可以采样多条候选的场景。

优点：

1. 比 PPO 工程上更轻，减少 critic 训练复杂度。
2. 适合 verifiable reward。
3. 能利用同一问题多条采样之间的相对质量。
4. 对 reasoning RL 比纯 DPO 更自然。

缺点：

1. 需要在线采样，成本仍然不低。
2. 对 reward 设计很敏感。
3. group size、KL、长度控制都会影响稳定性。
4. 如果 reward 太稀疏，训练效率会下降。

面试中可以这样说：

```text
GRPO 的关键是 group relative advantage。它对同一个 prompt 采样多条输出，用组内 reward 差异构造优化信号，通常不需要像 PPO 那样训练单独 critic，因此更适合数学、代码这类可验证任务上的 reasoning RL。
```

## 6. GSPO：序列级策略优化

GSPO 可以放在“sequence-level policy optimization”这类方法中理解。

很多 token-level RL 更新会遇到一个问题：最终 reward 是对整条回答给的，但更新时要分摊到每个 token。

对于长 CoT，回答可能有几百到几千 token。

如果处理不好，会出现：

```text
1. token-level credit assignment 噪声大。
2. 长回答被系统性奖励或惩罚。
3. 局部 token 概率变化破坏整体推理结构。
4. KL 控制在 token 级和 sequence 级之间不一致。
```

GSPO 这类方法强调在序列级别进行更稳定的 policy optimization。

核心关注点包括：

1. 把完整 response 作为优化单元。
2. 更直接地优化 sequence-level reward。
3. 控制长序列训练中的方差和 KL。
4. 避免 token 级更新导致的局部不稳定。

适用场景：

```text
长推理链
长代码生成
多步骤工具调用
多轮任务规划
```

面试中不建议死背 GSPO 公式，而要能表达：

```text
随着 reasoning model 输出越来越长，后训练不只关心单 token 的概率提升，而要关心整条推理序列的 reward、KL 和稳定性。GSPO 类方法就是在这个背景下出现的序列级策略优化思路。
```

## 7. DAPO：面向长链推理的高级策略优化

DAPO 可以作为 PPO/GRPO 之后的进一步优化路线来理解，重点通常在于提升长链 reasoning RL 的稳定性、效率和有效样本利用。

在 reasoning RL 中，常见痛点是：

```text
1. reward 稀疏：只有最终答案对错。
2. 输出很长：一次 rollout 成本高。
3. 正样本少：模型早期答对率低。
4. 长度偏置：模型可能学会输出更长或更短来投机。
5. KL 不稳定：模型容易偏离初始策略。
6. 难题和易题混在一起，训练信号质量不均衡。
```

DAPO 类方法通常会围绕这些问题做改进，例如：

1. 更细的 advantage 设计。
2. 更合理的 token/sequence level loss 解耦。
3. 动态采样或难度过滤。
4. 对过长、过短、格式错误输出做约束。
5. 改进 KL 或 clipping 机制。
6. 提高有效 rollout 比例。

它适合：

```text
数学推理强化
代码强化
竞赛题训练
复杂多步任务
长 CoT 模型后训练
```

面试中可以这样概括：

```text
DAPO 这类方法关注的是在长链推理 RL 中进一步提升训练稳定性和样本效率。相比只说 PPO 或 DPO，它更贴近 reasoning model 后训练的实际问题，比如稀疏 reward、长序列 credit assignment、长度偏置、动态采样和 KL 控制。
```

## 8. DCPO：偏好、对比与约束优化

DCPO 可以放在“contrastive / constrained preference optimization”这类方法中理解。

后训练不仅要让模型生成更好的答案，还要满足一些约束：

```text
不能泄露隐私
不能输出危险指导
不能过度拒答
不能破坏格式
不能牺牲基础能力
不能为了 reward 过度投机
```

这类问题只用一个简单 reward 有时不够。

DCPO 类方法通常关注：

1. chosen 和 rejected 之间的对比优化。
2. reward 最大化与约束满足之间的平衡。
3. 安全、格式、事实性、长度等多目标控制。
4. 防止模型在某个目标上过优化导致其他能力退化。

可以把它理解成：

```text
不是只问“哪个回答更好”，还要问“这个回答是否满足约束”。
```

适用场景：

```text
安全对齐
格式严格任务
工具调用约束
企业助手行为规范
多目标偏好优化
```

面试中可以这样说：

```text
DCPO 类方法体现了后训练从单一偏好优化走向多目标和约束优化的趋势。实际助手模型不仅要 reward 高，还要满足安全、格式、事实性和能力保持等约束，因此需要把对比偏好和约束信号结合起来。
```

## 9. PPO、DPO、GRPO、GSPO、DAPO、DCPO 怎么选

可以按任务和资源选择。

### 9.1 如果目标是通用聊天偏好

可以考虑：

```text
SFT -> DPO/IPO/KTO/ORPO -> safety tuning -> eval
```

原因是通用聊天偏好通常有大量离线 preference 数据，DPO 类方法成本低、稳定。

### 9.2 如果目标是数学和代码 reasoning

可以考虑：

```text
SFT/cold start -> online sampling -> rule-based reward -> GRPO/DAPO/GSPO -> eval
```

原因是数学和代码任务可以自动验证，在线 RL 能让模型探索新的推理路径。

### 9.3 如果目标是安全和格式约束

可以考虑：

```text
SFT safety data -> preference/constraint optimization -> red-team eval
```

可使用 DPO/DCPO 类思路，也可以结合 rule-based reward。

### 9.4 如果资源有限

优先路线通常是：

```text
高质量 SFT -> 小规模 DPO -> 重点评估和数据迭代
```

不建议一开始就上复杂 online RL。

### 9.5 如果资源充足且任务可验证

可以做：

```text
大规模采样 -> verifiable reward -> GRPO/DAPO -> 多轮迭代
```

这更接近 reasoning model 的训练思路。

## 10. 数据要求对比

| 方法 | 数据形态 | 关键质量要求 |
|---|---|---|
| SFT | prompt + answer | 答案高质量、格式统一、覆盖目标任务 |
| Reward Model | prompt + chosen + rejected | 偏好可靠、差异明确、标注一致 |
| PPO | prompt + reward signal | reward 可用、KL 稳定、采样质量可控 |
| DPO | prompt + chosen + rejected | chosen/rejected 质量差异真实，避免虚假偏好 |
| GRPO | prompt + 多条 sampled responses + reward | reward 可验证，group 内有区分度 |
| GSPO | prompt + sequence reward | 长序列 reward 可靠，长度和格式受控 |
| DAPO | prompt + rollout + reward/constraint | 有效样本比例高，难度和长度分布合理 |
| DCPO | preference + constraints | 偏好和约束标签可靠，多目标不冲突 |

## 11. 常见训练风险

### 11.1 Reward Hacking

模型学会骗 reward，而不是解决真实问题。

例如：

```text
输出很长，看起来很有逻辑，但答案是错的。
```

### 11.2 Length Bias

reward 偏好长答案或短答案，导致模型输出长度异常。

解决方式包括长度归一化、长度惩罚、按任务切片评估。

### 11.3 KL Collapse

policy 偏离 reference 太远，语言能力或安全行为退化。

解决方式是 KL penalty、clip、参考模型约束和 checkpoint 回滚。

### 11.4 Over-Optimization

模型在训练 reward 上变好，但真实用户体验下降。

这在 reward model 质量不足时尤其常见。

### 11.5 Catastrophic Forgetting

后训练过强导致 base model 的通用能力下降。

需要通用能力回归评估和 replay 数据。

### 11.6 Safety Regression

推理能力提升后，安全拒答或边界控制变差。

reasoning RL 后尤其需要重新做 safety eval。

## 12. 现代后训练 pipeline 示例

一个更完整的现代后训练 pipeline 可以这样设计：

```text
1. Base model checkpoint 选择。
2. 高质量 SFT，建立指令遵循和基本回答格式。
3. 构造偏好数据，训练 reward model 或直接做 DPO 类优化。
4. 对数学、代码、工具调用任务构造 verifiable reward。
5. 使用 GRPO/DAPO/GSPO 类方法做 reasoning RL。
6. 加入 safety、拒答、隐私和格式约束数据。
7. 使用 DCPO 或约束优化思路做多目标行为控制。
8. 做能力回归、安全回归、风格评估、长度评估和真实任务评估。
9. 根据失败样例刷新 SFT、preference 和 RL prompt pool。
10. 多轮迭代后选择上线 checkpoint。
```

这比“跑一个 DPO”更接近真实系统。

## 13. 面试高频问法

### 问法 1：SFT、DPO 和 PPO 的区别是什么？

可以这样答：

```text
SFT 是监督学习，直接模仿高质量答案，主要建立指令遵循能力。DPO 是离线偏好优化，使用 chosen/rejected 偏好对直接优化模型，不需要显式 reward model 和在线 rollout，工程简单稳定。PPO 是在线 RLHF 方法，模型采样回答后由 reward model 或规则 reward 打分，再用 RL 更新 policy，能力更灵活但成本和稳定性挑战更大。
```

### 问法 2：为什么 DPO 不能代表当前所有主流后训练？

可以这样答：

```text
DPO 适合离线偏好对齐，工程简单，但它主要受限于已有偏好数据，缺少在线探索。现在很多 reasoning model 后训练依赖模型自己采样多条推理链，再用数学答案、代码测试、工具执行结果等 verifiable reward 进行强化学习，这类场景更适合 GRPO、DAPO、GSPO 等在线 RL 或序列级优化方法。因此 DPO 是重要方法，但不是后训练全貌。
```

### 问法 3：GRPO 相比 PPO 的核心差异是什么？

可以这样答：

```text
PPO 通常需要 value/critic 来估计 advantage，而 GRPO 对同一个 prompt 采样多条回答，利用 group 内 reward 的相对差异构造 advantage，通常可以减少对 critic 的依赖。它特别适合数学、代码这类可以自动验证答案的 reasoning RL 场景。
```

### 问法 4：为什么 reasoning RL 需要在线采样？

可以这样答：

```text
因为推理任务的关键不只是模仿已有答案，而是让模型探索不同推理路径。在线采样可以得到当前 policy 真实会生成的解法，再用 verifiable reward 判断哪些路径正确，从而优化模型自己的生成分布。离线偏好数据很难覆盖所有新出现的推理路径。
```

### 问法 5：后训练如何避免能力退化？

可以这样答：

```text
需要控制 KL 或 reference 约束，避免 policy 偏离 base/SFT 模型太远；加入通用能力回归评估和 safety 回归评估；必要时混入 replay 数据；对 reward hacking、长度偏置、拒答异常和格式退化做切片监控；上线前比较 base、SFT、RL checkpoint 在多类任务上的变化。
```

## 14. 本章小结

本章扩展了后训练和对齐强化训练的方法谱系。

核心结论如下：

1. 后训练不等于 SFT，也不等于只做 DPO。
2. PPO 是经典 RLHF 方法，适合在线 reward 优化，但工程复杂、成本高、稳定性挑战大。
3. DPO 是重要的离线偏好优化方法，简单稳定，但不适合覆盖所有 reasoning RL 场景。
4. GRPO 用 group 内相对 reward 构造优化信号，适合数学、代码等可验证任务。
5. GSPO 类方法关注长序列和 sequence-level reward 的稳定优化。
6. DAPO 类方法关注长链推理 RL 的样本效率、稳定性、长度控制和动态采样。
7. DCPO 类方法体现了偏好优化向多目标约束优化扩展的趋势。
8. 现代后训练 pipeline 往往结合 SFT、偏好优化、online RL、verifiable reward、安全约束和回归评估。
9. 方法选择要看任务类型、reward 形式、资源预算和稳定性要求。
10. 面试中要讲清方法谱系、适用场景和 trade-off，而不是只背某一个 loss。
