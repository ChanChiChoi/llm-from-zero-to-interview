# 第十一章：后训练与对齐强化训练

## 本章定位

预训练得到的是 base model，它具备语言建模能力和世界知识，但通常还不能直接作为可靠助手使用。

后训练的目标是把 base model 变成能遵循指令、能对话、能推理、能调用工具、能遵守安全边界，并且符合人类偏好的 assistant model 或 reasoning model。

过去很多教材会把后训练简化成：

```text
SFT -> RLHF/PPO -> DPO
```

这个说法已经不够用了。

当前真实训练路线更接近：

```text
SFT / cold start
-> reward modeling / rule-based reward / verifiable reward
-> online RL or offline preference optimization
-> long-CoT / reasoning RL
-> safety alignment
-> regression eval and iterative data refresh
```

因此，本章不能只讲 DPO。DPO 是重要方法，但它更多是离线偏好优化路线的一部分，不代表当前主流 RL 后训练全貌。

本章会系统梳理 PPO、GRPO、GSPO、DAPO、DCPO、DPO 及相关方法，重点讲清它们分别解决什么问题、依赖什么数据、优化什么目标、优缺点是什么、适合什么场景，以及面试中如何表达。

## 本章目标

学完本章，你需要掌握：

1. 后训练为什么不等于简单 SFT。
2. SFT、RM、PPO、DPO、GRPO、GSPO、DAPO、DCPO 的位置关系。
3. 为什么早期 RLHF 常用 PPO。
4. 为什么 reasoning model 训练中会出现 GRPO、DAPO 等方法。
5. DPO 为什么方便，但为什么不能代表所有主流对齐强化训练。
6. 各类方法对数据、reward、reference model、critic、采样和训练稳定性的要求。
7. 面试中如何讲清现代后训练 pipeline。

## 为什么后训练是模型变成助手的关键阶段

预训练得到的 base model 主要学到的是语言建模能力。

它知道很多文本模式、知识模式和推理模式，但通常还不适合直接面对用户。

原因包括：

1. 不一定遵循指令。
2. 不一定会拒答危险请求。
3. 不一定知道输出格式要求。
4. 不一定会优先给出帮助性、简洁性和安全性兼顾的回答。
5. 不一定会根据反馈持续改善。

后训练的目标是把 base model 变成 assistant model 或 reasoning model。

面试表达：预训练负责“学会语言和知识”，后训练负责“学会怎么当助手”。

### 后训练与对齐训练的资料边界

按照 `WRITING_PLAN.md` 的要求，本章第二轮精修前核对了 InstructGPT / RLHF、DPO、DeepSeekMath / GRPO、Hugging Face TRL 的 DPO / GRPO 文档，以及 GSPO、DAPO、DCPO 等近年公开论文或技术说明。

本章聚焦面试和工程中最常用的后训练抽象：SFT、reward model、PPO-RLHF、DPO、GRPO、sequence-level / advanced policy optimization、安全约束、回归评估和数据刷新。对 GSPO、DAPO、DCPO 这类高时效方法，本章只写成可迁移的机制和边界，不把不同团队的具体实现写成唯一标准。

### 后训练目标的统一口径

后训练可以抽象成在 base policy `pi_0` 上学习新 policy `pi_theta`：

```math
\pi_{\theta}
=
\mathrm{PostTrain}(\pi_0,D_{\mathrm{sft}},D_{\mathrm{pref}},R,G_{\mathrm{safe}})
```

其中 `D_sft` 是监督示范数据，`D_pref` 是偏好数据，`R` 是 reward 或规则验证器，`G_safe` 是安全和行为门禁。最终目标不是单一 loss 最低，而是综合提升：

```math
G_{\mathrm{align}}=
g_{\mathrm{sft}}\,
g_{\mathrm{pref}}\,
g_{\mathrm{reward}}\,
g_{\mathrm{kl}}\,
g_{\mathrm{safety}}\,
g_{\mathrm{regression}}
```

这些门禁分别对应指令遵循、偏好质量、reward 可用性、策略偏移可控、安全边界和能力回归。

## 1. 后训练总体流程

一个典型 LLM 后训练流程可以拆成六层。

### 1.1 SFT：建立基本指令遵循能力

SFT 使用人工或模型生成的高质量 instruction-response 数据，让模型学会按照用户指令回答。

它解决的问题是：

```text
base model 只会续写，不一定会按用户意图回答。
```

SFT 的目标通常是 next-token cross entropy：

```text
max log p_theta(y | x)
```

其中 `x` 是 prompt，`y` 是标准回答。

更精确地说，SFT 只应该在 assistant token 上计算 loss。设 `m_t=1` 表示第 `t` 个 token 属于需要监督的 assistant 输出：

```math
L_{\mathrm{sft}}=
-
\frac{
\sum_{i,t}m_{i,t}\log \pi_{\theta}(y_{i,t}\mid x_i,y_{i,<t})
}{
\sum_{i,t}m_{i,t}
}
```

如果 `m_t` 把 user prompt 或 padding 也算进去，模型会学到错误目标，所以 SFT 的第一道门禁是检查 assistant-only label mask。

SFT 的优点是简单稳定。

缺点是它只能模仿数据，不会主动优化“哪个回答更好”。

### 1.2 Reward Modeling：学习偏好信号

Reward Model 使用偏好数据训练一个打分模型。

偏好数据通常长这样：

```text
prompt
chosen response
rejected response
```

Reward Model 学的是：

```text
r(prompt, chosen) > r(prompt, rejected)
```

常见 reward model 训练可以用 Bradley-Terry 形式理解。对偏好样本 `(x,y^+,y^-)`：

```math
P(y^+\succ y^-\mid x)=
\sigma(r_{\phi}(x,y^+)-r_{\phi}(x,y^-))
```

对应 loss 为：

```math
L_{\mathrm{rm}}=
-
\log
\sigma(r_{\phi}(x,y^+)-r_{\phi}(x,y^-))
```

它解决的问题是：

```text
如何把人类偏好变成可优化的标量 reward。
```

### 1.3 RLHF：用 RL 优化策略模型

有了 reward 后，可以把语言模型看作 policy。

模型生成回答，reward model 给分，训练算法更新 policy，使它更倾向于生成高 reward 回答。

经典路线是 PPO-RLHF。

### 1.4 Offline Preference Optimization：离线偏好优化

DPO、IPO、KTO、ORPO 等方法不一定显式训练 reward model，也不一定在线 rollout。

它们直接用偏好对优化模型。

优点是工程简单、稳定、成本低。

缺点是受限于离线偏好数据，对复杂 reasoning、长链条探索和 verifiable reward 场景不一定够。

### 1.5 Reasoning RL：面向可验证任务的强化训练

数学、代码、逻辑推理、工具调用等任务可以用规则或程序验证答案。

例如：

```text
数学题答案是否正确
代码是否通过单元测试
工具调用结果是否满足目标
```

这类任务可以使用 rule-based reward 或 verifiable reward，进行在线采样和 RL 优化。

GRPO、DAPO、GSPO、DCPO 等方法通常出现在这一类讨论中。

### 1.6 Safety Alignment：安全与行为边界

后训练还必须控制：

```text
危险请求
隐私泄露
越狱
偏见
过度拒答
事实幻觉
工具误用
```

安全对齐不是单独一个 loss 就能解决，而是数据、reward、拒答策略、红队评估和上线门禁的组合。

## 2. 方法谱系总览

可以先用一张表建立全局视角。

| 方法 | 类型 | 是否需要 Reward Model | 是否在线采样 | 是否需要 Critic | 典型用途 |
|---|---|---|---|---|---|
| SFT | 监督微调 | 否 | 否 | 否 | 指令遵循、冷启动 |
| PPO-RLHF | 在线 RL | 通常需要 | 是 | 通常需要 | 经典人类偏好 RLHF |
| DPO | 离线偏好优化 | 否 | 否 | 否 | 偏好对齐、风格优化 |
| IPO/KTO/ORPO | 离线偏好优化 | 否 | 否 | 否 | DPO 替代或变体 |
| GRPO | 在线 RL / group-based RL | 可用规则 reward | 是 | 通常不需要 critic | reasoning RL、数学代码任务 |
| GSPO | sequence-level policy optimization | 可用 reward | 是 | 依实现而定 | 序列级稳定优化、长输出对齐 |
| DAPO | decoupled / advanced policy optimization | 可用 reward | 是 | 依实现而定 | 长链推理 RL、稳定性和效率优化 |
| DCPO | contrastive / constrained preference optimization | 可用偏好或约束信号 | 可在线或离线 | 依实现而定 | 偏好、约束、安全和行为控制 |

注意：不同论文或团队对 GSPO、DAPO、DCPO 的全称和实现细节可能不同，面试中不要只背缩写。更重要的是讲清：它们是在 PPO/DPO 之外，为了解决 reasoning RL、长序列优化、稳定性、样本效率和约束对齐问题而发展出的后训练方法。

## 3. PPO-RLHF

PPO 是早期 RLHF 的代表方法。

整体流程如下：

```text
1. 用 SFT 得到初始 policy model。
2. 训练 reward model。
3. policy model 对 prompt 采样回答。
4. reward model 给回答打分。
5. PPO 根据 reward 更新 policy。
6. 使用 KL penalty 防止 policy 偏离 reference model 太远。
```

PPO 的目标大致可以理解为：

```text
maximize reward(response) - beta * KL(policy || reference)
```

其中 KL 项非常重要。

如果把一条 response 记为 `y=(y_1,\ldots,y_T)`，sequence-level KL 可以写成：

```math
\mathrm{KL}_{\mathrm{seq}}
=
\sum_{t=1}^{T}
\left[
\log \pi_{\theta}(y_t\mid x,y_{1:t-1})
-
\log \pi_{\mathrm{ref}}(y_t\mid x,y_{1:t-1})
\right]
```

RLHF 的简化目标是：

```math
J_{\mathrm{rlhf}}=
E_{y\sim \pi_{\theta}}
\left[
r_{\phi}(x,y)
-
\beta\mathrm{KL}_{\mathrm{seq}}
\right]
```

PPO 实际训练还会使用 ratio clipping、advantage 和 value / critic 估计，面试中至少要讲清 reward 最大化和 KL 约束这两个核心。

如果没有 KL 约束，模型可能为了骗 reward model 而生成奇怪回答，也就是 reward hacking。

PPO 的优点：

1. 能在线采样，优化当前模型真实生成分布。
2. 可以直接使用 reward model 或规则 reward。
3. 适合需要探索和迭代优化的任务。
4. 是经典 RLHF 路线，理论和工程经验较多。

PPO 的缺点：

1. 工程复杂，需要 policy、reference、reward、value/critic 多个模型或模块。
2. 显存和计算成本高。
3. 训练不稳定，容易 reward hacking、KL 崩掉、长度偏置。
4. 对 reward model 质量高度敏感。

面试中可以这样说：

```text
PPO-RLHF 是经典在线 RLHF 方法，优势是能在模型自己的采样分布上直接优化 reward，缺点是系统复杂、成本高、稳定性差，需要 reward model、reference model、critic 和 KL 控制。它适合 reward 明确且需要在线探索的场景，但不是所有后训练都必须用 PPO。
```

## 4. DPO：直接偏好优化

DPO 的核心动机是：能不能不显式训练 reward model，也不跑复杂 PPO，直接用偏好对训练模型？

DPO 使用数据：

```text
(prompt, chosen, rejected)
```

它直接优化模型，让 chosen 的相对概率高于 rejected。

直觉上：

```text
如果人更喜欢 chosen，就提高模型生成 chosen 的概率，降低 rejected 的概率。
```

DPO 的核心量是 policy 和 reference 在 chosen / rejected 上的 log probability gap：

```math
\Delta_{\theta}=
\log \pi_{\theta}(y^+\mid x)
-
\log \pi_{\theta}(y^-\mid x)
```

```math
\Delta_{\mathrm{ref}}=
\log \pi_{\mathrm{ref}}(y^+\mid x)
-
\log \pi_{\mathrm{ref}}(y^-\mid x)
```

DPO loss 可以写成：

```math
L_{\mathrm{dpo}}=
-
\log
\sigma\left(
\beta(\Delta_{\theta}-\Delta_{\mathrm{ref}})
\right)
```

`beta` 控制偏好优化强度。`beta` 太小，更新弱；`beta` 太大，容易过度拟合偏好对或造成能力回归。

DPO 的优点：

1. 工程简单。
2. 不需要在线 rollout。
3. 不需要显式 reward model。
4. 训练稳定性通常好于 PPO。
5. 适合偏好数据充足、目标相对明确的场景。

DPO 的缺点：

1. 依赖离线偏好数据质量。
2. 不能充分探索当前 policy 的新输出。
3. 对复杂 long-CoT reasoning 的优化能力有限。
4. 容易学到风格偏好，而不一定提升真实任务正确率。
5. 对 verifiable reward、代码执行 reward、数学答案 reward 这类场景不如在线 RL 灵活。

为什么说“DPO 目前不是全部主流”？

因为很多 reasoning model 后训练关注的是：

```text
模型自己采样多条推理链
用规则或程序验证结果
根据正确性、格式、长度、过程质量做 reward
继续优化 policy
```

这类训练天然更接近在线 RL，而不是只用固定偏好对的 DPO。

## 5. GRPO：Group Relative Policy Optimization

GRPO 可以理解为一种面向 group sampling 的 RL 优化方法。

它常见于 reasoning RL 场景。

基本思想是：对同一个 prompt，模型采样多条回答，形成一个 group：

```text
prompt -> response_1, response_2, ..., response_k
```

然后用 reward 给每条回答打分：

```text
r_1, r_2, ..., r_k
```

再用组内相对优势更新模型。

直觉是：

```text
同一道题里，奖励更高的回答应该被增强，奖励更低的回答应该被压低。
```

对第 `i` 个 prompt 采样 `K` 条回答，reward 为 `r_{i,j}`，组内均值和标准差为：

```math
\bar{r}_i=
\frac{1}{K}\sum_{j=1}^{K}r_{i,j}
```

```math
s_i=
\sqrt{
\frac{1}{K}
\sum_{j=1}^{K}
(r_{i,j}-\bar{r}_i)^2
}
```

组内相对 advantage 可以写成：

```math
A_{i,j}=
\frac{r_{i,j}-\bar{r}_i}{s_i+\epsilon}
```

如果 `rho_{i,j}` 是新旧 policy 的概率比，一个简化 clipped objective 是：

```math
L_{\mathrm{grpo}}=
-
\frac{1}{NK}
\sum_{i,j}
\min(
\rho_{i,j}A_{i,j},
\mathrm{clip}(\rho_{i,j},1-\epsilon,1+\epsilon)A_{i,j}
)
```

这不是所有 GRPO 实现的完整细节，但能抓住面试中最关键的点：用同 prompt 的组内 reward 构造 baseline / advantage，减少对 critic 的依赖。

GRPO 相比 PPO 的一个重要特点是：通常不需要单独训练 critic/value model，而是用 group 内 reward 做 baseline 或 advantage 估计。

它适合：

1. 数学推理。
2. 代码生成。
3. 可验证 QA。
4. 有明确对错 reward 的任务。
5. 同一 prompt 可以采样多条候选的场景。

优点：

1. 比 PPO 工程上更轻，减少 critic 训练复杂度。
2. 适合 verifiable reward。
3. 能利用同一问题多条采样之间的相对质量。
4. 对 reasoning RL 比纯 DPO 更自然。

缺点：

1. 需要在线采样，成本仍然不低。
2. 对 reward 设计很敏感。
3. group size、KL、长度控制都会影响稳定性。
4. 如果 reward 太稀疏，训练效率会下降。

面试中可以这样说：

```text
GRPO 的关键是 group relative advantage。它对同一个 prompt 采样多条输出，用组内 reward 差异构造优化信号，通常不需要像 PPO 那样训练单独 critic，因此更适合数学、代码这类可验证任务上的 reasoning RL。
```

## 6. GSPO：序列级策略优化

GSPO 可以放在“sequence-level policy optimization”这类方法中理解。

很多 token-level RL 更新会遇到一个问题：最终 reward 是对整条回答给的，但更新时要分摊到每个 token。

对于长 CoT，回答可能有几百到几千 token。

如果处理不好，会出现：

```text
1. token-level credit assignment 噪声大。
2. 长回答被系统性奖励或惩罚。
3. 局部 token 概率变化破坏整体推理结构。
4. KL 控制在 token 级和 sequence 级之间不一致。
```

GSPO 这类方法强调在序列级别进行更稳定的 policy optimization。

核心关注点包括：

1. 把完整 response 作为优化单元。
2. 更直接地优化 sequence-level reward。
3. 控制长序列训练中的方差和 KL。
4. 避免 token 级更新导致的局部不稳定。

适用场景：

```text
长推理链
长代码生成
多步骤工具调用
多轮任务规划
```

面试中不建议死背 GSPO 公式，而要能表达：

```text
随着 reasoning model 输出越来越长，后训练不只关心单 token 的概率提升，而要关心整条推理序列的 reward、KL 和稳定性。GSPO 类方法就是在这个背景下出现的序列级策略优化思路。
```

## 7. DAPO：面向长链推理的高级策略优化

DAPO 可以作为 PPO/GRPO 之后的进一步优化路线来理解，重点通常在于提升长链 reasoning RL 的稳定性、效率和有效样本利用。

在 reasoning RL 中，常见痛点是：

```text
1. reward 稀疏：只有最终答案对错。
2. 输出很长：一次 rollout 成本高。
3. 正样本少：模型早期答对率低。
4. 长度偏置：模型可能学会输出更长或更短来投机。
5. KL 不稳定：模型容易偏离初始策略。
6. 难题和易题混在一起，训练信号质量不均衡。
```

DAPO 类方法通常会围绕这些问题做改进，例如：

1. 更细的 advantage 设计。
2. 更合理的 token/sequence level loss 解耦。
3. 动态采样或难度过滤。
4. 对过长、过短、格式错误输出做约束。
5. 改进 KL 或 clipping 机制。
6. 提高有效 rollout 比例。

它适合：

```text
数学推理强化
代码强化
竞赛题训练
复杂多步任务
长 CoT 模型后训练
```

面试中可以这样概括：

```text
DAPO 这类方法关注的是在长链推理 RL 中进一步提升训练稳定性和样本效率。相比只说 PPO 或 DPO，它更贴近 reasoning model 后训练的实际问题，比如稀疏 reward、长序列 credit assignment、长度偏置、动态采样和 KL 控制。
```

## 8. DCPO：偏好、对比与约束优化

DCPO 可以放在“contrastive / constrained preference optimization”这类方法中理解。

后训练不仅要让模型生成更好的答案，还要满足一些约束：

```text
不能泄露隐私
不能输出危险指导
不能过度拒答
不能破坏格式
不能牺牲基础能力
不能为了 reward 过度投机
```

这类问题只用一个简单 reward 有时不够。

DCPO 类方法通常关注：

1. chosen 和 rejected 之间的对比优化。
2. reward 最大化与约束满足之间的平衡。
3. 安全、格式、事实性、长度等多目标控制。
4. 防止模型在某个目标上过优化导致其他能力退化。

可以把它理解成：

```text
不是只问“哪个回答更好”，还要问“这个回答是否满足约束”。
```

适用场景：

```text
安全对齐
格式严格任务
工具调用约束
企业助手行为规范
多目标偏好优化
```

面试中可以这样说：

```text
DCPO 类方法体现了后训练从单一偏好优化走向多目标和约束优化的趋势。实际助手模型不仅要 reward 高，还要满足安全、格式、事实性和能力保持等约束，因此需要把对比偏好和约束信号结合起来。
```

## 9. PPO、DPO、GRPO、GSPO、DAPO、DCPO 怎么选

可以按任务和资源选择。

### 9.1 如果目标是通用聊天偏好

可以考虑：

```text
SFT -> DPO/IPO/KTO/ORPO -> safety tuning -> eval
```

原因是通用聊天偏好通常有大量离线 preference 数据，DPO 类方法成本低、稳定。

### 9.2 如果目标是数学和代码 reasoning

可以考虑：

```text
SFT/cold start -> online sampling -> rule-based reward -> GRPO/DAPO/GSPO -> eval
```

原因是数学和代码任务可以自动验证，在线 RL 能让模型探索新的推理路径。

### 9.3 如果目标是安全和格式约束

可以考虑：

```text
SFT safety data -> preference/constraint optimization -> red-team eval
```

可使用 DPO/DCPO 类思路，也可以结合 rule-based reward。

### 9.4 如果资源有限

优先路线通常是：

```text
高质量 SFT -> 小规模 DPO -> 重点评估和数据迭代
```

不建议一开始就上复杂 online RL。

### 9.5 如果资源充足且任务可验证

可以做：

```text
大规模采样 -> verifiable reward -> GRPO/DAPO -> 多轮迭代
```

这更接近 reasoning model 的训练思路。

## 10. 数据要求对比

| 方法 | 数据形态 | 关键质量要求 |
|---|---|---|
| SFT | prompt + answer | 答案高质量、格式统一、覆盖目标任务 |
| Reward Model | prompt + chosen + rejected | 偏好可靠、差异明确、标注一致 |
| PPO | prompt + reward signal | reward 可用、KL 稳定、采样质量可控 |
| DPO | prompt + chosen + rejected | chosen/rejected 质量差异真实，避免虚假偏好 |
| GRPO | prompt + 多条 sampled responses + reward | reward 可验证，group 内有区分度 |
| GSPO | prompt + sequence reward | 长序列 reward 可靠，长度和格式受控 |
| DAPO | prompt + rollout + reward/constraint | 有效样本比例高，难度和长度分布合理 |
| DCPO | preference + constraints | 偏好和约束标签可靠，多目标不冲突 |

后训练数据的一个常见风险是 chosen 只是更长，而不是真的更好。可以用长度偏置做初筛：

```math
B_{\mathrm{len}}=
\frac{1}{N}
\sum_i
(\ell_i^+-\ell_i^-)
```

其中 `ell_i^+` 和 `ell_i^-` 分别是 chosen / rejected 的长度。如果 `B_len` 过大，DPO 或 RM 很可能学到“长就是好”的伪偏好。

## 11. 常见训练风险

### 11.1 Reward Hacking

模型学会骗 reward，而不是解决真实问题。

例如：

```text
输出很长，看起来很有逻辑，但答案是错的。
```

### 11.2 Length Bias

reward 偏好长答案或短答案，导致模型输出长度异常。

解决方式包括长度归一化、长度惩罚、按任务切片评估。

### 11.3 KL Collapse

policy 偏离 reference 太远，语言能力或安全行为退化。

解决方式是 KL penalty、clip、参考模型约束和 checkpoint 回滚。

### 11.4 Over-Optimization

模型在训练 reward 上变好，但真实用户体验下降。

这在 reward model 质量不足时尤其常见。

### 11.5 Catastrophic Forgetting

后训练过强导致 base model 的通用能力下降。

需要通用能力回归评估和 replay 数据。

### 11.6 Safety Regression

推理能力提升后，安全拒答或边界控制变差。

reasoning RL 后尤其需要重新做 safety eval。

后训练评估至少要同时看偏好胜率、安全召回、过度拒答和能力回归：

```math
W_{\mathrm{pref}}=
\frac{1}{N}
\sum_i
\mathbf{1}[s_{\mathrm{new},i}>s_{\mathrm{old},i}]
```

```math
R_{\mathrm{safe}}=
\frac{
\sum_i
\mathbf{1}[q_i=1]\mathbf{1}[\mathrm{refuse}_i=1]
}{
\sum_i\mathbf{1}[q_i=1]+\epsilon
}
```

```math
R_{\mathrm{over}}=
\frac{
\sum_i
\mathbf{1}[q_i=0]\mathbf{1}[\mathrm{refuse}_i=1]
}{
\sum_i\mathbf{1}[q_i=0]+\epsilon
}
```

```math
R_{\mathrm{reg}}=
\frac{
\sum_i
\mathbf{1}[z_i^{old}=1]\mathbf{1}[z_i^{new}=0]
}{
\sum_i\mathbf{1}[z_i^{old}=1]+\epsilon
}
```

其中 `q_i=1` 表示样本应该拒答，`z_i` 是能力评估是否通过。安全召回高但过度拒答也高，说明模型可能变得过保守；偏好胜率提升但 `R_reg` 上升，说明后训练可能损伤了旧能力。

## 12. 现代后训练 pipeline 示例

一个更完整的现代后训练 pipeline 可以这样设计：

```text
1. Base model checkpoint 选择。
2. 高质量 SFT，建立指令遵循和基本回答格式。
3. 构造偏好数据，训练 reward model 或直接做 DPO 类优化。
4. 对数学、代码、工具调用任务构造 verifiable reward。
5. 使用 GRPO/DAPO/GSPO 类方法做 reasoning RL。
6. 加入 safety、拒答、隐私和格式约束数据。
7. 使用 DCPO 或约束优化思路做多目标行为控制。
8. 做能力回归、安全回归、风格评估、长度评估和真实任务评估。
9. 根据失败样例刷新 SFT、preference 和 RL prompt pool。
10. 多轮迭代后选择上线 checkpoint。
```

这比“跑一个 DPO”更接近真实系统。

## 13. 最小可运行后训练审计 demo

下面的 demo 用标准库模拟一次后训练审计。输入包括 SFT 数据、偏好对、GRPO group rollout 和评估样本；输出 assistant label 比例、偏好数据覆盖、RM / DPO loss、长度偏置、组内 advantage、KL ratio 风险、安全召回、过度拒答、能力回归和方法选择建议。

```python
import math
from statistics import mean, pstdev

sft_data = [
    {"id": "sft_format", "category": "format", "assistant_tokens": 54, "prompt_tokens": 22, "has_refusal": False},
    {"id": "sft_math", "category": "reasoning", "assistant_tokens": 83, "prompt_tokens": 18, "has_refusal": False},
    {"id": "sft_safety", "category": "safety", "assistant_tokens": 41, "prompt_tokens": 30, "has_refusal": True},
]

pref_pairs = [
    {
        "id": "pref_helpful", "category": "helpfulness",
        "chosen_logp": -2.10, "rejected_logp": -2.70,
        "ref_chosen_logp": -2.30, "ref_rejected_logp": -2.55,
        "chosen_reward": 1.4, "rejected_reward": 0.2,
        "chosen_len": 72, "rejected_len": 61,
    },
    {
        "id": "pref_safety", "category": "safety",
        "chosen_logp": -1.80, "rejected_logp": -2.40,
        "ref_chosen_logp": -1.95, "ref_rejected_logp": -2.10,
        "chosen_reward": 1.1, "rejected_reward": -0.1,
        "chosen_len": 45, "rejected_len": 70,
    },
    {
        "id": "pref_reasoning", "category": "reasoning",
        "chosen_logp": -2.50, "rejected_logp": -2.60,
        "ref_chosen_logp": -2.40, "ref_rejected_logp": -2.42,
        "chosen_reward": 0.9, "rejected_reward": 0.7,
        "chosen_len": 120, "rejected_len": 116,
    },
]

groups = [
    {
        "prompt_id": "math_001",
        "responses": [
            {"id": "r1", "reward": 1.0, "old_logp": -3.0, "new_logp": -2.76, "tokens": 95, "valid_format": True},
            {"id": "r2", "reward": 0.0, "old_logp": -2.8, "new_logp": -2.9, "tokens": 88, "valid_format": True},
            {"id": "r3", "reward": 0.5, "old_logp": -3.2, "new_logp": -3.1, "tokens": 140, "valid_format": True},
        ],
    },
    {
        "prompt_id": "code_001",
        "responses": [
            {"id": "r1", "reward": 0.0, "old_logp": -3.4, "new_logp": -3.5, "tokens": 170, "valid_format": False},
            {"id": "r2", "reward": 1.0, "old_logp": -3.6, "new_logp": -3.4, "tokens": 150, "valid_format": True},
            {"id": "r3", "reward": 1.0, "old_logp": -3.5, "new_logp": -3.35, "tokens": 148, "valid_format": True},
        ],
    },
]

eval_rows = [
    {"id": "help_001", "old_score": 0.70, "new_score": 0.82, "old_ok": True, "new_ok": True, "risk": "normal", "new_refusal": False, "should_refuse": False},
    {"id": "math_001", "old_score": 0.40, "new_score": 0.67, "old_ok": False, "new_ok": True, "risk": "normal", "new_refusal": False, "should_refuse": False},
    {"id": "safe_001", "old_score": 0.80, "new_score": 0.92, "old_ok": True, "new_ok": True, "risk": "danger", "new_refusal": True, "should_refuse": True},
    {"id": "safe_002", "old_score": 0.76, "new_score": 0.60, "old_ok": True, "new_ok": False, "risk": "normal", "new_refusal": True, "should_refuse": False},
]

beta = 0.1
clip_eps = 0.2
kl_threshold = 0.12

def sigmoid(x):
    return 1 / (1 + math.exp(-x))

sft_categories = sorted({row["category"] for row in sft_data})
assistant_tokens = sum(row["assistant_tokens"] for row in sft_data)
prompt_tokens = sum(row["prompt_tokens"] for row in sft_data)
assistant_label_ratio = round(assistant_tokens / (assistant_tokens + prompt_tokens), 4)
safety_sft_count = sum(row["has_refusal"] for row in sft_data)

rm_losses = []
dpo_losses = []
length_bias_values = []
for pair in pref_pairs:
    reward_gap = pair["chosen_reward"] - pair["rejected_reward"]
    rm_losses.append(-math.log(sigmoid(reward_gap)))

    policy_gap = pair["chosen_logp"] - pair["rejected_logp"]
    ref_gap = pair["ref_chosen_logp"] - pair["ref_rejected_logp"]
    dpo_losses.append(-math.log(sigmoid(beta * (policy_gap - ref_gap))))
    length_bias_values.append(pair["chosen_len"] - pair["rejected_len"])

avg_rm_loss = round(mean(rm_losses), 4)
avg_dpo_loss = round(mean(dpo_losses), 4)

    scale = pstdev(rewards) or 1.0
    advantages = []
    ratios = []
    for row in group["responses"]:
        advantage = (row["reward"] - baseline) / scale
        ratio = math.exp(row["new_logp"] - row["old_logp"])
        clipped_ratio = min(max(ratio, 1 - clip_eps), 1 + clip_eps)
        surrogate = min(ratio * advantage, clipped_ratio * advantage) if advantage >= 0 else max(ratio * advantage, clipped_ratio * advantage)
        advantages.append(round(advantage, 4))
        ratios.append(round(ratio, 4))
        all_advantages.append(advantage)
        all_ratios.append(ratio)
        if not row["valid_format"]:
            format_failures.append((group["prompt_id"], row["id"]))
    group_summaries[group["prompt_id"]] = {"baseline": round(baseline, 4), "advantages": advantages, "ratios": ratios}

regression_cases = [row["id"] for row in eval_rows if row["old_ok"] and not row["new_ok"]]
safety_recall = round(
    sum(row["new_refusal"] and row["should_refuse"] for row in eval_rows if row["should_refuse"]) /
    sum(row["should_refuse"] for row in eval_rows),
    4,
)
over_refusal_rate = round(
    sum(row["new_refusal"] and not row["should_refuse"] for row in eval_rows) /
    sum(not row["should_refuse"] for row in eval_rows),
    4,
)
max_ratio_deviation = round(max(abs(ratio - 1.0) for ratio in all_ratios), 4)

method_choice = "GRPO" if any("reasoning" == row["category"] for row in pref_pairs) and groups else "DPO"

checks = {
    "sft_has_safety": safety_sft_count >= 1,
    "preference_categories_ok": preference_categories == ["helpfulness", "reasoning", "safety"],
    "length_bias_ok": abs(avg_length_bias) <= 20,
    "dpo_loss_reasonable": avg_dpo_loss < 0.70,
    "group_advantage_centered": abs(round(sum(all_advantages), 8)) == 0.0,
    "kl_ratio_under_control": max_ratio_deviation <= kl_threshold + clip_eps,
    "safety_recall_ok": safety_recall == 1.0,
    "over_refusal_flagged": over_refusal_rate > 0.0,
    "regression_detected": regression_cases == ["safe_002"],
}

print("sft_categories=", sft_categories)
print("assistant_label_ratio=", assistant_label_ratio)
print("safety_sft_count=", safety_sft_count)
print("preference_categories=", preference_categories)
print("avg_rm_loss=", avg_rm_loss)
print("avg_dpo_loss=", avg_dpo_loss)
print("avg_length_bias=", avg_length_bias)
print("group_summaries=", group_summaries)
print("format_failures=", format_failures)
print("win_rate=", win_rate)
print("regression_cases=", regression_cases)
print("safety_recall=", safety_recall)
print("over_refusal_rate=", over_refusal_rate)
print("method_choice=", method_choice)
print("checks=", checks)
print("gate_pass=", all(checks.values()))
```

## 14. 面试高频问法

### 问法 1：SFT、DPO 和 PPO 的区别是什么？

可以这样答：

```text
SFT 是监督学习，直接模仿高质量答案，主要建立指令遵循能力。DPO 是离线偏好优化，使用 chosen/rejected 偏好对直接优化模型，不需要显式 reward model 和在线 rollout，工程简单稳定。PPO 是在线 RLHF 方法，模型采样回答后由 reward model 或规则 reward 打分，再用 RL 更新 policy，能力更灵活但成本和稳定性挑战更大。
```

### 问法 2：为什么 DPO 不能代表当前所有主流后训练？

可以这样答：

```text
DPO 适合离线偏好对齐，工程简单，但它主要受限于已有偏好数据，缺少在线探索。现在很多 reasoning model 后训练依赖模型自己采样多条推理链，再用数学答案、代码测试、工具执行结果等 verifiable reward 进行强化学习，这类场景更适合 GRPO、DAPO、GSPO 等在线 RL 或序列级优化方法。因此 DPO 是重要方法，但不是后训练全貌。
```

### 问法 3：GRPO 相比 PPO 的核心差异是什么？

可以这样答：

```text
PPO 通常需要 value/critic 来估计 advantage，而 GRPO 对同一个 prompt 采样多条回答，利用 group 内 reward 的相对差异构造 advantage，通常可以减少对 critic 的依赖。它特别适合数学、代码这类可以自动验证答案的 reasoning RL 场景。
```

### 问法 4：为什么 reasoning RL 需要在线采样？

可以这样答：

```text
因为推理任务的关键不只是模仿已有答案，而是让模型探索不同推理路径。在线采样可以得到当前 policy 真实会生成的解法，再用 verifiable reward 判断哪些路径正确，从而优化模型自己的生成分布。离线偏好数据很难覆盖所有新出现的推理路径。
```

### 问法 5：后训练如何避免能力退化？

可以这样答：

```text
需要控制 KL 或 reference 约束，避免 policy 偏离 base/SFT 模型太远；加入通用能力回归评估和 safety 回归评估；必要时混入 replay 数据；对 reward hacking、长度偏置、拒答异常和格式退化做切片监控；上线前比较 base、SFT、RL checkpoint 在多类任务上的变化。
```

## 15. 本章小结

本章扩展了后训练和对齐强化训练的方法谱系。

核心结论如下：

1. 后训练不等于 SFT，也不等于只做 DPO。
2. PPO 是经典 RLHF 方法，适合在线 reward 优化，但工程复杂、成本高、稳定性挑战大。
3. DPO 是重要的离线偏好优化方法，简单稳定，但不适合覆盖所有 reasoning RL 场景。
4. GRPO 用 group 内相对 reward 构造优化信号，适合数学、代码等可验证任务。
5. GSPO 类方法关注长序列和 sequence-level reward 的稳定优化。
6. DAPO 类方法关注长链推理 RL 的样本效率、稳定性、长度控制和动态采样。
7. DCPO 类方法体现了偏好优化向多目标约束优化扩展的趋势。
8. 现代后训练 pipeline 往往结合 SFT、偏好优化、online RL、verifiable reward、安全约束和回归评估。
9. 方法选择要看任务类型、reward 形式、资源预算和稳定性要求。
10. 面试中要讲清方法谱系、适用场景和 trade-off，而不是只背某一个 loss。
