# 第四章：模型架构与初始化

## 本章目标

理解训练前如何选择模型架构、规模和初始化策略。

## 核心议题

1. Decoder-only Transformer 的训练优势。
2. hidden size、layer 数、head 数、FFN size 的权衡。
3. RoPE、RMSNorm、SwiGLU、GQA、MoE。
4. 参数初始化和深层稳定性。
5. 从 dense model 到 MoE model 的训练差异。

## 面试重点

架构设计要结合训练稳定性、推理成本、数据规模和目标能力，而不是孤立比较模型结构。

## 为什么训练前要先定架构

训练大模型之前，必须先确定模型结构和规模。这个决策一旦开始大规模训练后就很难回头。

如果模型太小，数据再好也可能容量不够。如果模型太大，预算不足会导致训练不充分，甚至不如一个更小但训练充分的模型。如果架构不适合目标场景，后面再做 SFT、DPO 或推理优化也很难彻底弥补。

架构设计要同时考虑：

1. 目标能力。
2. 数据规模。
3. 训练预算。
4. 推理成本。
5. 分布式训练可行性。
6. 后续微调和部署生态。

面试表达：训练前的架构选择不是论文模型对比，而是目标、数据、算力和服务约束之间的系统取舍。

### 模型架构与初始化的资料边界

按照 `WRITING_PLAN.md` 的要求，本章第二轮精修前核对了 Transformer 原论文、LLaMA 架构报告、RoPE、RMSNorm、SwiGLU、Grouped-Query Attention、Switch Transformer / MoE 和 Chinchilla compute-optimal training 相关公开资料。

本章聚焦训练前的架构决策口径：decoder-only、层数、宽度、head、FFN、RoPE、RMSNorm、SwiGLU、GQA、MoE、上下文长度、参数初始化、训练稳定性和推理成本。它不展开完整模型源码、所有架构家族差异、长上下文扩展细节、MoE 并行系统和分布式训练实现，这些会在架构专题、训练系统和推理部署章节继续展开。

### 架构设计的关键变量

设 hidden size 为 `d`，层数为 `N_L`，query head 数为 `H_q`，K/V head 数为 `H_kv`，每个 query head 的维度为 `d_h`：

```math
d_h=\frac{d}{H_q}
```

词表大小为 `V`，SwiGLU FFN 中间维度为 `d_ff`。embedding 参数量是：

```math
P_{\mathrm{emb}}=Vd
```

GQA 下 attention projection 参数量近似是：

```math
P_{\mathrm{attn}}=
d^2+2dH_{\mathrm{kv}}d_h+d^2
```

这里四项分别对应 Q、K、V、O projection。SwiGLU FFN 参数量近似是：

```math
P_{\mathrm{ffn}}\approx 3dd_{\mathrm{ff}}
```

忽略 bias 和很小的 norm 参数后，dense decoder block 参数量可以估算为：

```math
P_{\mathrm{block}}\approx P_{\mathrm{attn}}+P_{\mathrm{ffn}}
```

总参数量粗略估算为：

```math
P_{\mathrm{model}}\approx P_{\mathrm{emb}}+N_LP_{\mathrm{block}}
```

这些公式不是替代真实配置导出的参数统计，而是训练前做量级审计：模型是不是接近目标规模，GQA 是否真的降低 K/V 成本，FFN 是否成为主要参数来源。

## 1. 为什么 decoder-only 成为 LLM 主流

现代通用大语言模型最常见的是 decoder-only Transformer。

它的核心训练目标是自回归 next token prediction：

```text
给定前面的 token，预测下一个 token。
```

### 1.1 训练目标统一

文本续写、问答、代码生成、对话、工具调用，都可以统一成“给定上下文生成后续 token”。

这让预训练和后训练之间的接口非常自然：

```text
预训练：根据前文预测后文
SFT：根据指令生成回答
偏好优化：让生成回答更符合偏好
推理：根据 prompt 逐 token 生成
```

### 1.2 工程实现简单

decoder-only 架构只有 causal self-attention，不需要 encoder-decoder 之间的 cross-attention。

这带来几个工程好处：

1. 模型结构统一。
2. 训练和推理代码路径更简单。
3. KV Cache 机制清晰。
4. 适合长文本生成和多轮对话。

### 1.3 什么时候不是最佳选择

decoder-only 不是所有任务的唯一答案。

Encoder-only 仍适合很多理解类任务，encoder-decoder 仍适合某些翻译、摘要和条件生成任务。但通用 LLM 追求统一生成接口，decoder-only 的训练和服务闭环最简单。

面试表达：decoder-only 的优势是把预训练、指令微调、对话和推理统一到同一个自回归生成框架里。

## 2. 模型规模怎么定

模型规模不是只看参数量。

一个模型的主要规模参数包括：

| 超参 | 含义 |
| --- | --- |
| `num_layers` | Transformer block 层数 |
| `hidden_size` | hidden state 维度 |
| `num_attention_heads` | attention head 数 |
| `head_dim` | 每个 head 的维度 |
| `ffn_size` | MLP/FFN 中间层维度 |
| `vocab_size` | 词表大小 |
| `max_position_embeddings` | 最大上下文长度或位置范围 |

这些参数共同决定模型容量、训练成本、显存、通信和推理延迟。

### 2.1 层数和宽度

增加层数会增强逐层抽象能力，但也增加训练深度和稳定性挑战。

增加 hidden size 会增强每层表示容量，但会显著增加矩阵乘法成本。

粗略看，Transformer 参数主要来自 attention projection、FFN 和 embedding，其中 FFN 通常占很大比例。

面试中不需要精确背参数公式，但要知道：

```text
hidden size 增大通常会让参数和计算快速增加。
层数增加会线性增加 block 数量和激活显存。
```

### 2.2 head 数和 head_dim

attention head 不是越多越好。

head 数增加可以让模型在不同子空间做信息路由，但如果 hidden size 固定，head 数越多，每个 head 的维度越小。

常见设计会保持合理的 `head_dim`，例如 64、128 等量级，同时让 hidden size 能被 head 数整除。

基本约束是：

```math
d=H_qd_h
```

如果要使用 GQA，还需要 query head 可以被 K/V head 分组：

```math
H_q=gH_{\mathrm{kv}}
```

其中 `g` 是每组 query head 数。若这个整除关系不成立，工程实现通常要改 head 数、hidden size 或 K/V head 数。

### 2.3 FFN size

FFN 通常是 Transformer block 中参数和计算的重要来源。

很多 LLM 的 FFN 中间维度约为 hidden size 的数倍。使用 SwiGLU 后，参数形式和传统 ReLU/GELU FFN 不完全一样，但核心都是用更大的中间层提供非线性变换能力。

面试表达：模型规模设计不是只报一个 7B 或 70B，而是层数、宽度、head、FFN、词表和上下文长度共同决定。

## 3. 参数量、token 数和算力预算要匹配

如果训练 token 不够，大模型会 under-train。

Chinchilla 风格的结论提醒我们：在固定计算预算下，参数量和训练 token 数要匹配。盲目扩大模型而不给足训练 token，可能不如训练一个更小但更充分的模型。

训练前要回答：

1. 有多少高质量 token。
2. 有多少 GPU 和训练时间。
3. 目标模型是 base、chat、代码还是领域模型。
4. 推理端能接受多大模型。

面试表达：模型大小要从 compute budget 和 token budget 倒推，而不是越大越好。

训练 FLOPs 的常用粗估仍然可以写成：

```math
C_{\mathrm{train}}\approx 6P_{\mathrm{model}}D
```

其中 `D` 是训练 token 数。一个简单的 token / 参数比例是：

```math
r_{\mathrm{tok/param}}=\frac{D}{P_{\mathrm{model}}}
```

这个比例不能机械套用到所有模型，但它能提醒你：只报“7B”没有意义，还要说明准备用多少高质量 token 和多少算力把它训练充分。

## 4. 现代 LLM 的常见架构组件

现代 decoder-only LLM 往往不再是原始 Transformer 的直接版本，而是有一组成熟改造。

### 4.1 RoPE

RoPE 通过旋转 Q/K 向量注入位置信息，适合 decoder-only 自回归模型。

对每个二维子空间，可以把位置 `p` 上的旋转写成：

```math
\begin{bmatrix}
q'_{2i}\\
q'_{2i+1}
\end{bmatrix}
=
\begin{bmatrix}
\cos(p\theta_i)&-\sin(p\theta_i)\\
\sin(p\theta_i)&\cos(p\theta_i)
\end{bmatrix}
\begin{bmatrix}
q_{2i}\\
q_{2i+1}
\end{bmatrix}
```

K 向量做同样旋转。直觉上，RoPE 不是给 token 加一个位置 embedding，而是在 Q/K 的相似度计算里注入相对位置信息。

训练视角下要注意：

1. RoPE 参数要和最大上下文长度匹配。
2. 长上下文扩展不能只改 config。
3. RoPE scaling 可能需要继续训练或专项评估。

如果 RoPE base、scaling 策略或实现不一致，模型输出可能明显退化。

### 4.2 RMSNorm 和 Pre-Norm

现代 LLM 常用 Pre-Norm 结构，也就是在 attention/FFN 之前做 normalization。

Pre-Norm 相比 Post-Norm 通常更利于深层 Transformer 训练稳定。

RMSNorm 是 LayerNorm 的简化形式，常见于 LLaMA 风格模型。它只使用均方根进行归一化，不显式减均值，计算更简单。

RMSNorm 可以写成：

```math
\mathrm{RMS}(x)=\sqrt{\frac{1}{d}\sum_{i=1}^{d}x_i^2+\epsilon}
```

```math
y_i=\gamma_i\frac{x_i}{\mathrm{RMS}(x)}
```

训练视角下，normalization 的作用是稳定激活尺度和梯度流。

### 4.3 SwiGLU

SwiGLU 是 gated MLP 的一种形式，很多现代 LLM 用它替代传统 ReLU/GELU FFN。

直觉上，它通过门控机制让模型对不同特征通道做更灵活的选择。

一个常见写法是：

```math
\mathrm{SwiGLU}(x)=
\left[\mathrm{swish}(xW_g)\odot (xW_u)\right]W_d
```

其中 `W_g` 是 gate projection，`W_u` 是 up projection，`W_d` 是 down projection，`odot` 表示逐元素相乘。

训练视角下，SwiGLU 往往带来更好的表示能力，但也要考虑 FFN 参数量和计算成本。

### 4.4 MQA 和 GQA

MQA 和 GQA 主要是为了降低推理时 KV Cache 成本。

标准 MHA 中，每个 attention head 都有自己的 K/V。MQA 多个 query head 共享一组 K/V，GQA 则是若干 query head 共享一组 K/V。

每层每 token 的 KV cache 大小近似为：

```math
M_{\mathrm{KV,layer}}=2H_{\mathrm{kv}}d_hb
```

其中 `b` 是每个数值的字节数，例如 fp16 / bf16 通常是 2。整模型 batch KV cache 近似是：

```math
M_{\mathrm{KV}}=B_{\mathrm{batch}}C_{\mathrm{ctx}}N_LM_{\mathrm{KV,layer}}
```

GQA 的核心收益来自 `H_kv < H_q`。如果从 MHA 改为 GQA，K/V cache 近似按下面比例下降：

```math
\frac{M_{\mathrm{GQA}}}{M_{\mathrm{MHA}}}=\frac{H_{\mathrm{kv}}}{H_q}
```

训练视角下：

1. GQA 可以在效果和推理成本之间折中。
2. 它会影响 KV Cache 显存和服务吞吐。
3. 如果目标是线上推理，架构设计阶段就要考虑它。

面试表达：GQA 不是只服务推理框架的细节，而是训练前就要确定的架构选择。

## 5. 上下文长度如何设计

上下文长度会影响训练成本、显存和模型能力。

长上下文并不是把 `max_position` 改大就行。

训练长上下文要考虑：

1. 训练数据中是否真的有长样本。
2. 位置编码能否支持长位置。
3. attention 显存和计算是否可承受。
4. packing 和 truncation 策略是否合理。
5. 评估是否覆盖长文检索和长文推理。

常见错误是训练几乎都用短样本，却声称模型支持很长上下文。这种模型可能能接收长输入，但不能可靠利用远端信息。

如果上下文长度是 `C_ctx`，标准全量 attention 的注意力矩阵规模随下面量级增长：

```math
M_{\mathrm{attn}}\propto B_{\mathrm{batch}}H_qC_{\mathrm{ctx}}^2
```

decode 阶段 KV cache 则随上下文长度线性增长：

```math
M_{\mathrm{KV}}\propto B_{\mathrm{batch}}C_{\mathrm{ctx}}N_LH_{\mathrm{kv}}d_h
```

所以长上下文训练和长上下文推理是两个不同成本问题：训练主要受 attention 二次复杂度影响，推理主要受 KV cache 显存和带宽影响。

面试表达：上下文长度是数据分布、位置编码、attention 成本和评估共同决定的能力，不是单个配置项。

## 6. Dense model 和 MoE model

Dense model 每个 token 都经过同一组 FFN 参数。

MoE model 则让每个 token 路由到部分 expert，从而增加总参数量，但每个 token 激活的参数量较少。

如果共有 `E` 个 expert，每个 token 选择 `k` 个 expert，每个 expert 的 FFN 参数量是 `P_expert`，那么 MoE FFN 总参数量是：

```math
P_{\mathrm{moe,total}}=EP_{\mathrm{expert}}
```

每个 token 激活的 FFN 参数量近似是：

```math
P_{\mathrm{moe,active}}=kP_{\mathrm{expert}}
```

这就是 MoE 的基本取舍：总容量大，但每个 token 的激活计算只走一部分 expert。

### 6.1 MoE 的优势

1. 在相近激活计算下增加模型容量。
2. 对多领域、多语言、多能力可能有帮助。
3. 推理时每个 token 只激活部分 expert。

### 6.2 MoE 的训练难点

1. 路由器训练不稳定。
2. expert load balance 难。
3. 通信复杂度更高。
4. batch 和并行策略更复杂。
5. 某些 expert 可能训练不足。

### 6.3 面试表达

MoE 的核心不是“参数更多所以更强”，而是在计算预算一定时通过稀疏激活增加容量。代价是训练、路由、负载均衡和服务系统更复杂。

## 7. 初始化为什么重要

大模型训练初期非常脆弱。初始化如果不合适，可能导致激活尺度异常、梯度不稳定、loss spike 或收敛很慢。

初始化要解决的问题是：

```text
让每层输入输出的尺度在训练开始时处于合理范围。
```

### 7.1 常见初始化直觉

线性层通常使用与 fan-in / fan-out 相关的初始化，让输出方差不过大也不过小。

一个常见的 Xavier 风格标准差是：

```math
\sigma_{\mathrm{xavier}}=\sqrt{\frac{2}{f_{\mathrm{in}}+f_{\mathrm{out}}}}
```

对于输入输出维度都等于 `d` 的 projection，可以近似为：

```math
\sigma\approx \frac{1}{\sqrt{d}}
```

Embedding、attention projection、FFN projection、输出层都需要合理初始化。

对于深层网络，残差分支的尺度也很重要。如果每层残差都不断放大，深层模型容易不稳定。

一种常见直觉是让残差 projection 的初始化随层数缩小：

```math
\sigma_{\mathrm{res}}\approx \frac{\sigma}{\sqrt{2N_L}}
```

不同模型家族具体做法不同，但面试里要能说明：深层残差路径的尺度必须受控，初始化不是随便填随机数。

### 7.2 新增 token 的初始化

如果扩展 tokenizer，新增 token embedding 也要初始化。

常见做法包括：

1. 随机初始化。
2. 用已有 token 的均值初始化。
3. 用语义相关 token 的 embedding 平均初始化。

无论哪种方式，新增 token 都需要通过继续训练或微调学会稳定语义。

### 7.3 从已有 checkpoint 继续训练

如果不是从零训练，而是继续预训练或领域继续训练，初始化来自已有 checkpoint。

这时要特别注意：

1. tokenizer 是否一致。
2. 架构 config 是否一致。
3. RoPE 参数是否一致。
4. vocab size 是否一致。
5. optimizer state 是否恢复。

这些不一致会导致加载失败、训练异常或效果退化。

## 8. 训练稳定性相关架构选择

架构设计和训练稳定性直接相关。

### 8.1 Pre-Norm 更稳

深层 Transformer 中，Pre-Norm 通常比 Post-Norm 更容易训练。

原因是梯度更容易沿残差路径传播。

### 8.2 残差尺度要受控

层数越深，残差累积越明显。很多架构会通过初始化、norm 或 residual scaling 控制激活尺度。

### 8.3 Attention logits 尺度

`QK^T / sqrt(d_head)` 的缩放是为了防止 softmax 过于尖锐。

标准 scaled dot-product attention logits 可以写成：

```math
A=\frac{QK^\top}{\sqrt{d_h}}
```

如果 attention logits 尺度异常，训练可能出现梯度不稳定。

### 8.4 激活函数和数值精度

SwiGLU、GELU 等激活会影响激活分布。混合精度训练中，某些激活和归一化实现也会影响 NaN 风险。

面试表达：训练稳定性不是只靠调学习率，架构中的 norm、residual、初始化和 attention 尺度都会影响稳定性。

## 9. 架构选择和推理成本

训练时做的架构选择会影响未来服务成本。

几个典型例子：

| 架构选择 | 对推理的影响 |
| --- | --- |
| hidden size 更大 | 每 token 计算更重 |
| layer 更多 | 解码每步延迟更高 |
| vocab 更大 | logits 和采样成本更高 |
| context 更长 | KV Cache 更大 |
| MHA vs GQA | KV Cache 成本差异明显 |
| dense vs MoE | 路由和 expert 并行更复杂 |

所以训练前要和部署目标联动。

如果目标是端侧模型，参数量、KV Cache、量化友好性和延迟很关键。如果目标是服务端高吞吐模型，GQA、batching、并行方式和显存占用更关键。如果目标是研究模型，可以更重视能力上限。

## 10. 一个训练前架构决策 checklist

训练前可以按下面顺序检查：

```text
1. 模型目标：通用、代码、数学、领域、聊天还是多模态？
2. 训练 token：有多少高质量 token？
3. 预算：GPU 数量、训练时长、存储和失败容忍度？
4. 参数规模：模型大小是否和 token/compute 匹配？
5. 架构组件：RoPE、RMSNorm、SwiGLU、GQA、MoE 是否必要？
6. 上下文长度：训练数据和位置编码是否支持？
7. 推理约束：延迟、吞吐、显存、KV Cache 是否可接受？
8. 初始化：从零训练还是加载已有 checkpoint？
9. 生态兼容：是否支持主流训练和推理框架？
10. 风险预案：如果不稳定，如何缩小规模复现实验？
```

这比单纯讨论“哪个架构更先进”更接近真实训练项目。

下面这个 demo 用标准库做一个训练前架构审计。它检查 hidden/head 整除关系、GQA 的参数和 KV cache 收益、参数量、训练 FLOPs、token/参数比例、初始化标准差、RMSNorm / RoPE 的最小数值行为，以及 MoE 总参数和激活参数差异。

```python
import math

cfg = {
    "vocab_size": 32000,
    "hidden_size": 4096,
    "num_layers": 32,
    "query_heads": 32,
    "kv_heads": 8,
    "ffn_size": 11008,
    "context_length": 4096,
    "batch_size": 4,
    "bytes_per_value": 2,
    "train_tokens": 120_000_000_000,
    "moe_experts": 8,
    "moe_top_k": 2,
}


def dense_arch_stats(c):
    d = c["hidden_size"]
    layers = c["num_layers"]
    hq = c["query_heads"]
    hkv = c["kv_heads"]
    dh = d // hq
    f = c["ffn_size"]
    vocab = c["vocab_size"]

    q_params = d * d
    k_params = d * hkv * dh
    v_params = d * hkv * dh
    o_params = d * d
    attn_params = q_params + k_params + v_params + o_params
    mha_attn_params = 4 * d * d
    ffn_params = 3 * d * f
    norm_params = 2 * d
    block_params = attn_params + ffn_params + norm_params
    embedding_params = vocab * d
    total_params = embedding_params + layers * block_params

    kv_bytes_per_token_layer = 2 * hkv * dh * c["bytes_per_value"]
    kv_cache_gib = (
        kv_bytes_per_token_layer
        * layers
        * c["context_length"]
        * c["batch_size"]
        / (1024**3)
    )
    mha_kv_cache_gib = kv_cache_gib * (hq / hkv)
    train_flops = 6 * total_params * c["train_tokens"]

    return {
        "head_dim": dh,
        "attn_params": attn_params,
        "mha_attn_params": mha_attn_params,
        "gqa_attn_ratio": attn_params / mha_attn_params,
        "ffn_params": ffn_params,
        "block_params": block_params,
        "embedding_params": embedding_params,
        "total_params": total_params,
        "kv_cache_gib": kv_cache_gib,
        "mha_kv_cache_gib": mha_kv_cache_gib,
        "train_flops": train_flops,
    }


def rms_norm(values, eps=1e-6):
    rms = math.sqrt(sum(x * x for x in values) / len(values) + eps)
    return [x / rms for x in values]


def rope_pair(x0, x1, position, theta):
    angle = position * theta
    c = math.cos(angle)
    s = math.sin(angle)
    return [c * x0 - s * x1, s * x0 + c * x1]


def init_stds(c):
    d = c["hidden_size"]
    base = math.sqrt(2.0 / (d + d))
    residual = base / math.sqrt(2 * c["num_layers"])
    return base, residual


stats = dense_arch_stats(cfg)
base_std, residual_std = init_stds(cfg)
moe_ffn_total = cfg["moe_experts"] * 3 * cfg["hidden_size"] * cfg["ffn_size"]
moe_ffn_active = cfg["moe_top_k"] * 3 * cfg["hidden_size"] * cfg["ffn_size"]
tokens_per_param = cfg["train_tokens"] / stats["total_params"]

checks = {

    "ffn_large_enough": cfg["ffn_size"] >= 2 * cfg["hidden_size"],
    "kv_cache_under_16gib": stats["kv_cache_gib"] < 16,
    "tokens_per_param_reasonable": 10 <= tokens_per_param <= 40,
    "residual_std_smaller": residual_std < base_std,
    "moe_topk_valid": 1 <= cfg["moe_top_k"] <= cfg["moe_experts"],
}

print("head_dim=", stats["head_dim"])
print("attn_params_m=", round(stats["attn_params"] / 1e6, 2))

print("block_params_m=", round(stats["block_params"] / 1e6, 2))

print("gqa_attn_ratio=", round(stats["gqa_attn_ratio"], 4))

print("mha_kv_cache_gib=", round(stats["mha_kv_cache_gib"], 3))
print("tokens_per_param=", round(tokens_per_param, 2))
print("train_flops=", f"{stats['train_flops']:.2e}")
print("base_init_std=", round(base_std, 6))
```

## 11. 常见坑

### 11.1 参数越大越好

不对。参数量要和 token 数、算力、目标和推理成本匹配。

### 11.2 只改上下文长度配置

只改配置不等于模型真正具备长上下文能力。

### 11.3 忽略推理成本

训练出一个很大的模型，如果线上不可服务，工程价值会大幅下降。

### 11.4 MoE 只看总参数量

MoE 要看激活参数、路由、负载均衡、通信和服务复杂度。

### 11.5 初始化和 checkpoint 不匹配

继续训练时，如果 tokenizer、RoPE、vocab 或 config 不一致，可能直接失败或暗中退化。

## 12. 面试官会怎么问

### 问法 1：如果让你设计一个 7B 模型，你会考虑什么？

可以这样答：

```text
我会先从目标和预算倒推，而不是直接固定 7B。需要确定目标能力、训练 token 数、GPU 预算、上下文长度和推理约束。架构上通常选择 decoder-only Transformer，再确定层数、hidden size、head 数、FFN size、RoPE、RMSNorm、SwiGLU 和是否使用 GQA。最后还要看训练稳定性、评估计划和部署成本。
```

### 问法 2：为什么现代 LLM 多用 decoder-only？

可以这样答：

```text
因为 decoder-only 可以把预训练、指令微调、对话和推理统一成自回归 next token prediction。结构上只有 causal self-attention，训练和推理路径简单，KV Cache 机制清晰，适合长文本生成和多轮对话。
```

### 问法 3：GQA 解决什么问题？

可以这样答：

```text
GQA 让多个 query head 共享一组 K/V head，在效果和 KV Cache 成本之间折中。相比 MHA，它能降低推理时 K/V 缓存显存和带宽压力；相比 MQA，它保留更多 K/V 组，通常效果更稳。
```

### 问法 4：MoE 和 dense model 的核心 trade-off 是什么？

可以这样答：

```text
MoE 通过稀疏激活在相近每 token 计算下增加总参数容量，可能提升多领域能力。但代价是路由器、负载均衡、expert 并行、通信和服务系统更复杂。Dense model 更简单稳定，但同等容量下计算成本更高。
```

### 问法 5：初始化为什么会影响训练稳定性？

可以这样答：

```text
初始化决定训练开始时各层激活和梯度的尺度。如果尺度过大，可能出现 loss spike、梯度爆炸或 NaN；如果过小，学习很慢。深层 Transformer 还要考虑残差分支、normalization 和 attention logits 尺度，所以初始化和架构稳定性是绑定的。
```

## 13. 本章小结

本章核心结论：

1. 架构设计要从目标、数据、算力和部署约束倒推。
2. Decoder-only Transformer 统一了预训练、SFT、对话和推理的自回归接口。
3. 模型规模由层数、宽度、head、FFN、词表和上下文长度共同决定。
4. 参数量、训练 token 数和算力预算必须匹配。
5. RoPE、RMSNorm、SwiGLU、GQA 是现代 LLM 常见关键组件。
6. 长上下文能力不是单个配置项，而是位置编码、训练数据和 attention 成本共同决定。
7. MoE 用稀疏激活换取更大总容量，但训练和服务复杂度更高。
8. 初始化、norm、residual 和 attention 尺度会影响训练稳定性。
9. 训练前架构选择会直接影响未来推理成本。
10. 面试中要把架构讲成系统决策，而不是孤立罗列模型组件。
