# 第八章：显存优化与数值精度

## 本章目标

理解训练显存由哪些部分构成，以及如何通过精度和重计算优化显存。

## 核心议题

1. 参数、梯度、优化器状态、激活显存。
2. FP32、FP16、BF16、FP8。
3. mixed precision training。
4. activation checkpointing。
5. gradient accumulation。
6. optimizer state sharding。

## 面试重点

能估算训练显存，并解释为什么 Adam 优化器状态会显著增加显存占用。

## 为什么显存是训练中的硬约束

大模型训练经常不是先被算力限制，而是先被显存限制。

如果显存不够，训练直接 OOM；如果显存刚好够但没有余量，稍微增加 batch、sequence length、模型层数或打开某些日志，就可能崩掉。

训练显存主要由四类组成：

1. 模型参数。
2. 梯度。
3. 优化器状态。
4. 激活和临时 buffer。

很多初学者只估算模型权重大小，例如“7B 参数，BF16 大约 14GB”，但这远远不等于训练显存。训练时 AdamW 还要保存梯度、一阶动量、二阶动量，反向传播还要保存激活。

面试表达：推理只需要模型权重和 KV Cache，训练还需要梯度、optimizer state 和激活，因此训练显存远大于推理显存。

## 1. 训练显存由哪些部分构成

假设模型参数量为 `N`。

如果用 BF16 保存模型权重，每个参数约 2 bytes。

但训练中还要考虑：

| 部分 | 作用 | 典型大小 |
| --- | --- | --- |
| parameters | 模型权重 | `N * bytes` |
| gradients | 每个参数的梯度 | `N * bytes` |
| optimizer states | AdamW 的 m/v 等状态 | 通常很大 |
| activations | forward 保存给 backward 用 | 与 batch、seq、hidden、layers 相关 |
| temporary buffers | 通信、kernel、中间计算 | 与实现有关 |

AdamW 尤其占显存，因为它维护：

```text
m: 一阶动量
v: 二阶动量
```

如果 optimizer state 用 FP32，每个参数的 `m` 和 `v` 各 4 bytes，仅 optimizer state 就是：

```text
N * 8 bytes
```

这就是为什么训练显存比权重显存大很多。

## 2. 一个粗略显存估算

以 `N` 个参数为例，假设：

1. 权重用 BF16：2 bytes。
2. 梯度用 BF16：2 bytes。
3. AdamW m/v 用 FP32：8 bytes。
4. 可能还有 FP32 master weights：4 bytes。

那么仅模型状态就可能接近：

```text
2 + 2 + 8 + 4 = 16 bytes / parameter
```

如果是 7B 参数：

```text
7e9 * 16 bytes ≈ 112GB
```

这还没有算激活显存。

所以单卡 80GB 训练 7B full fine-tuning 或预训练并不轻松，需要 ZeRO/FSDP、gradient checkpointing、较小 batch 或参数高效微调。

面试表达：估算训练显存时，不能只看权重，还要把 gradient、optimizer state、activation 和 buffer 算进去。

## 3. 激活显存为什么大

反向传播需要用到 forward 的中间结果，所以训练时要保存 activation。

激活显存大致与下面因素相关：

```text
batch_size * sequence_length * hidden_size * num_layers
```

sequence length 一变长，激活显存会明显上升。

attention 相关中间结果还可能与 `T^2` 有关，取决于实现是否使用 FlashAttention 等优化。

### 激活显存的特点

1. 和 batch size 强相关。
2. 和 sequence length 强相关。
3. 和层数强相关。
4. 可以通过重计算减少。

这也是为什么长上下文训练显存压力很大。

## 4. FP32、FP16、BF16、FP8

训练数值精度直接影响显存、吞吐和稳定性。

| 精度 | bytes | 特点 | 常见用途 |
| --- | --- | --- | --- |
| FP32 | 4 | 精度高，稳定，显存大 | optimizer state、部分累积计算 |
| FP16 | 2 | 显存省，吞吐高，动态范围小 | mixed precision 训练 |
| BF16 | 2 | 动态范围接近 FP32，更稳 | 大模型训练常用 |
| FP8 | 1 | 更省显存和带宽，要求硬件和 recipe | 新一代大规模训练探索 |

### 4.1 FP32

FP32 最稳定，但显存和带宽开销大。

现代大模型训练通常不会全程 FP32 训练，但 optimizer state、某些归一化、梯度累积或 master weights 可能仍使用 FP32。

### 4.2 FP16

FP16 的优点是显存省、速度快。

问题是动态范围较小，容易 overflow 或 underflow。

因此 FP16 训练常需要 loss scaling。

### 4.3 BF16

BF16 也是 2 bytes，但指数位更多，动态范围接近 FP32。

它通常比 FP16 更适合大模型训练，因为不容易溢出。

代价是尾数精度更低，但在大模型训练中通常可以接受。

### 4.4 FP8

FP8 进一步降低显存和带宽，但训练 recipe 更复杂，需要硬件支持、scale 管理和更谨慎的数值稳定性验证。

面试中可以说：FP8 是重要方向，但不能简单认为把 dtype 改成 FP8 就能稳定训练。

## 5. Mixed precision training

混合精度训练不是所有计算都用低精度，而是在不同部分使用不同精度。

典型做法：

```text
forward/backward: FP16 或 BF16
optimizer state: FP32
部分归一化/累积: FP32 或更稳定实现
```

这样既能降低显存和提升吞吐，又能保留关键计算的稳定性。

### 5.1 Loss scaling

FP16 中小梯度可能 underflow 成 0。

loss scaling 的做法是先把 loss 放大，再 backward，更新前再把梯度缩回来。

```text
scaled_loss = loss * scale
backward(scaled_loss)
grad = grad / scale
```

BF16 因为动态范围更大，通常不需要像 FP16 那样依赖 loss scaling。

### 5.2 常见数值风险

1. loss 变 NaN。
2. gradient norm 变 Inf。
3. attention logits 过大。
4. softmax 溢出。
5. optimizer state 出现异常值。

这些问题经常和学习率、warmup、初始化、数据异常、精度配置一起出现。

## 6. Activation checkpointing

Activation checkpointing 的核心是用计算换显存。

普通训练：

```text
forward 保存大量 activation
backward 直接使用保存的 activation
```

checkpointing：

```text
forward 只保存部分边界 activation
backward 时重新计算中间 activation
```

优点：

1. 显著降低激活显存。
2. 支持更长 sequence length。
3. 支持更大 batch size。

代价：

1. 训练变慢。
2. 计算量增加。
3. checkpoint 粒度需要调。

面试表达：activation checkpointing 不是减少总计算，而是牺牲计算时间来降低激活显存。

## 7. Gradient accumulation 与显存

Gradient accumulation 不能降低单个 micro-batch 的激活显存，但可以在不增加单卡 batch 的情况下扩大 effective batch size。

```text
global_batch = micro_batch * accumulation_steps * data_parallel_size
```

如果 OOM 来自 batch 太大，可以减小 micro-batch，并增加 accumulation steps 保持 global batch 不变。

注意：

1. accumulation 会增加每次 optimizer step 前的 forward/backward 次数。
2. loss 要除以 accumulation steps。
3. logging 时要区分 micro step 和 optimizer step。

## 8. Optimizer state sharding

AdamW 的 optimizer state 很占显存，所以 sharding 很重要。

ZeRO 和 FSDP 都可以切分 optimizer state。

例如 ZeRO-1 只切 optimizer state，就能明显降低 AdamW 显存压力。

ZeRO-2 继续切 gradient。

ZeRO-3 / FSDP 进一步切 parameter。

面试表达：optimizer state sharding 主要解决训练状态显存，不直接减少每个 micro-batch 的激活显存。

## 9. 参数高效微调的显存优势

如果不是从零预训练，而是微调已有模型，可以使用 LoRA、QLoRA 等参数高效方法。

LoRA 的思想是冻结原模型，只训练低秩增量矩阵。

显存优势：

1. 不需要保存全量参数梯度。
2. optimizer state 只针对 LoRA 参数。
3. 训练状态显存大幅下降。

但注意：forward/backward 仍然要经过原模型，激活显存仍然存在。长上下文和大 batch 下仍可能 OOM。

## 10. FlashAttention 与 attention 显存

标准 attention 需要显式构造或保存较大的 attention matrix，显存和序列长度关系很强。

FlashAttention 的核心价值是通过 IO-aware 的 kernel 设计，减少显存读写和中间矩阵保存。

训练视角下，它可以：

1. 降低 attention 中间显存。
2. 提高训练速度。
3. 支持更长序列。

但它不改变 attention 理论复杂度，也不能解决所有长上下文问题。

面试表达：FlashAttention 主要优化 attention kernel 的显存访问和中间存储，而不是把 full attention 变成线性复杂度。

## 11. OOM 排查顺序

遇到 OOM，不要盲目换机器。

可以按下面顺序排查：

```text
1. 确认 OOM 发生在 forward、backward 还是 optimizer step。
2. 降低 micro-batch size。
3. 降低 sequence length。
4. 打开 activation checkpointing。
5. 使用 BF16 / mixed precision。
6. 使用 FlashAttention。
7. 使用 ZeRO/FSDP 切 optimizer state、gradient、parameter。
8. 检查是否有 tensor 被日志或 list 持有导致显存泄漏。
9. 检查 checkpoint、evaluation 是否额外占显存。
```

不同阶段 OOM 暗示不同原因：

| OOM 位置 | 可能原因 |
| --- | --- |
| forward | batch/seq 太大，activation 太多 |
| backward | activation 和 gradient 同时占用，checkpointing 不足 |
| optimizer step | optimizer state 太大 |
| evaluation | 没有 no_grad，缓存了 logits 或 hidden states |

## 12. NaN / Inf 与精度问题

NaN 和 Inf 通常说明数值已经失控。

常见原因：

1. 学习率过大。
2. warmup 太短。
3. FP16 overflow。
4. loss scaling 不合适。
5. attention logits 太大。
6. 输入数据异常。
7. 梯度爆炸。

排查顺序：

```text
1. 找到第一次出现 NaN 的 step。
2. 检查该 step 的 batch 是否异常。
3. 检查 loss、grad norm、lr。
4. 检查 mixed precision 配置。
5. 降低 lr 或延长 warmup。
6. 开启或调小 gradient clipping。
7. 从上一个健康 checkpoint 恢复。
```

面试表达：NaN 不是单一问题，要从数据、学习率、梯度、精度和模型实现一起排查。

## 13. 显存优化策略对比

| 方法 | 主要节省 | 代价 |
| --- | --- | --- |
| BF16/FP16 | 参数、激活、梯度显存 | 数值稳定性风险 |
| activation checkpointing | 激活显存 | 增加计算时间 |
| gradient accumulation | 保持 global batch，降低 micro-batch | step 时间变长 |
| ZeRO/FSDP | optimizer、gradient、parameter | 通信增加 |
| FlashAttention | attention 中间显存和 IO | 依赖 kernel 支持 |
| LoRA/QLoRA | 微调训练状态显存 | 不适合从零预训练 |
| sequence parallel | 长序列激活显存 | 通信和实现复杂 |

选择策略时要先判断瓶颈在哪里。

如果是 optimizer state 太大，用 ZeRO/FSDP。若是 activation 太大，用 checkpointing、减小 micro-batch 或 sequence length。若是 attention 中间结果大，用 FlashAttention。若是微调显存不够，用 LoRA/QLoRA。

## 14. 面试官会怎么问

### 问法 1：为什么 AdamW 训练显存这么大？

可以这样答：

```text
因为训练时不只保存模型权重，还要保存梯度和 optimizer state。AdamW 对每个参数维护一阶动量和二阶动量，通常是 FP32，所以 optimizer state 本身就可能是参数量的 8 bytes/parameter。再加上梯度、master weights 和激活，训练显存远大于模型权重显存。
```

### 问法 2：FP16 和 BF16 有什么区别？

可以这样答：

```text
FP16 和 BF16 都是 2 bytes，但 BF16 指数位更多，动态范围接近 FP32，因此大模型训练中更不容易 overflow。FP16 尾数精度更高一些，但动态范围小，常需要 loss scaling。大模型训练中 BF16 通常更稳定。
```

### 问法 3：activation checkpointing 解决什么问题？

可以这样答：

```text
它解决激活显存过大的问题。forward 时不保存所有中间 activation，只保存部分 checkpoint，backward 时重新计算中间 activation。这样用额外计算时间换显存，适合长序列、大 batch 或深模型训练。
```

### 问法 4：OOM 你怎么排查？

可以这样答：

```text
我会先判断 OOM 发生在 forward、backward、optimizer step 还是 evaluation。forward OOM 多半是 batch、sequence 或 activation 问题；backward OOM 可以考虑 checkpointing；optimizer step OOM 多半是 optimizer state，需要 ZeRO/FSDP；evaluation OOM 要检查 no_grad 和是否缓存 logits。然后按减 micro-batch、开 mixed precision、FlashAttention、checkpointing、sharding 的顺序处理。
```

### 问法 5：NaN 怎么排查？

可以这样答：

```text
先定位第一次出现 NaN 的 step，再检查该 batch 数据、loss、grad norm、learning rate、mixed precision 和 optimizer state。常见原因包括学习率过大、warmup 太短、FP16 overflow、梯度爆炸或异常数据。处理上可以降低 lr、延长 warmup、使用 BF16、调整 loss scaling、开启 gradient clipping，并从健康 checkpoint 恢复。
```

## 15. 本章小结

本章核心结论：

1. 训练显存由参数、梯度、optimizer state、激活和临时 buffer 共同组成。
2. AdamW 的 m/v 状态会显著增加显存占用。
3. 激活显存和 batch size、sequence length、hidden size、layers 强相关。
4. BF16 通常比 FP16 更适合大模型训练稳定性。
5. Mixed precision 需要在显存、吞吐和数值稳定之间平衡。
6. Activation checkpointing 用重计算换激活显存。
7. ZeRO/FSDP 主要通过 sharding 降低训练状态显存。
8. FlashAttention 降低 attention 中间显存和 IO 开销。
9. OOM 要按发生阶段定位，不要盲目调参。
10. NaN/Inf 要从数据、学习率、梯度、精度和实现一起排查。
