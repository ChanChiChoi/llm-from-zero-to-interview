# 第一章：训练总览

## 本章目标

建立大模型训练的全局视角，理解一个模型从原始数据到可用 checkpoint 的完整路径。

## 核心议题

1. 大模型训练包含哪些阶段。
2. Base model、instruct model、chat model、reasoning model 的训练差异。
3. 训练流程中的数据、模型、系统、评估、成本如何互相影响。
4. OpenAI 风格岗位为什么重视训练全流程理解。

## 典型训练流程

1. 明确模型目标和能力边界。
2. 收集、清洗、去重和混合训练数据。
3. 训练或选择 tokenizer。
4. 设计模型架构和规模。
5. 进行大规模预训练。
6. 评估 base model。
7. 做 SFT、PPO-RLHF、DPO、GRPO、GSPO、DAPO、DCPO、偏好优化和安全对齐。
8. 做能力、安全、鲁棒性和成本评估。
9. 选择 checkpoint 并进入部署阶段。

## 面试重点

如果面试官问“你如何训练一个大模型”，不要只回答 Transformer 和 loss。更好的回答结构是：目标、数据、架构、训练系统、优化策略、评估、后训练、风险和成本。

## 为什么训练总览很重要

大模型训练不是“写一个 Transformer，然后跑 cross entropy loss”这么简单。真实训练项目是一条很长的链路：数据决定模型见过什么，tokenizer 决定文本如何被切分，模型架构决定容量和计算路径，优化器决定如何收敛，分布式系统决定能不能跑起来，评估决定 checkpoint 能不能用，后训练决定模型是否能成为助手，部署成本决定模型是否能被业务承受。

很多候选人面试时会把训练回答成几个孤立概念：AdamW、cosine learning rate、DDP、loss curve、SFT、DPO。这些概念都重要，但如果不能把它们串成闭环，面试官会觉得你只会背点，不具备负责训练项目的能力。

训练总览要建立的是一张地图：你知道每个阶段解决什么问题，输入输出是什么，风险在哪里，出了问题应该先看哪一层。

面试表达：训练大模型的核心能力不是记住某个技巧，而是把目标、数据、模型、系统、评估和成本组织成一个可执行、可监控、可迭代的工程闭环。

## 一句话看完整训练链路

一个大模型从零到可用，大致经历下面这条路径：

```text
目标定义
-> 数据收集、清洗、去重、配比
-> tokenizer 训练或选择
-> 模型架构和规模设计
-> 预训练
-> base model 评估
-> 指令微调 SFT
-> 偏好优化和安全对齐
-> 综合评估和 checkpoint 选择
-> 压缩、部署、监控和持续迭代
```

每一步都会影响后面所有环节。

如果数据里代码比例不足，后面再做少量代码 SFT 很难补出强代码能力。如果 tokenizer 对中文、代码或数学符号切分很差，训练效率和下游能力都会受影响。如果预训练 checkpoint 本身推理能力弱，后训练只能改善表达和偏好，很难凭空创造深层能力。如果评估集被污染，模型看似很强，真实使用会暴露问题。

所以训练项目的核心不是某一步做到完美，而是全链路协同。

## 训练阶段总览

大模型训练可以分为预训练、后训练、评估和部署前优化几大阶段。

| 阶段 | 输入 | 输出 | 解决的问题 |
| --- | --- | --- | --- |
| 数据工程 | 原始网页、书籍、代码、论文、对话、多模态数据 | 高质量训练语料 | 决定模型能学到什么 |
| Tokenizer | 清洗后的文本语料 | tokenizer vocab 和编码规则 | 决定文本如何变成 token |
| 架构设计 | 目标能力、算力预算、推理约束 | 模型配置 | 决定容量、速度和可扩展性 |
| 预训练 | 大规模通用语料 | base model | 学语言、知识、代码、数学和世界模式 |
| Base 评估 | checkpoint 和 benchmark | 能力报告 | 判断预训练是否健康 |
| SFT | 指令-回答数据 | instruct model | 学会按指令回答 |
| 偏好优化 | chosen/rejected、reward、规则或反馈 | aligned model | 更符合人类偏好和安全要求 |
| 安全对齐 | 安全数据、红队样本、策略规则 | safer model | 降低有害输出和越权行为 |
| 部署前优化 | 目标模型 | 量化、蒸馏、推理配置 | 降低延迟和成本 |

这些阶段不是严格一次性完成。真实项目经常反复迭代：评估发现数学弱，回到数据配比；部署发现延迟高，回到架构和推理优化；红队发现安全漏洞，回到安全 SFT 或偏好优化。

## 从目标定义开始

训练项目第一步不是选模型结构，而是定义目标。

目标至少要回答以下问题：

| 问题 | 示例 |
| --- | --- |
| 模型用途是什么 | 通用助手、代码模型、数学模型、企业知识助手、语音助手 |
| 目标用户是谁 | 普通用户、开发者、客服、研究员、内部员工 |
| 主要能力是什么 | 对话、代码、数学、长上下文、多语言、多模态、工具调用 |
| 约束是什么 | 训练预算、推理延迟、显存、部署环境、安全要求 |
| 成功标准是什么 | benchmark、人工评测、业务指标、成本指标 |

如果目标是训练通用 base model，重点是大规模数据覆盖、稳定预训练和通用能力评估。如果目标是企业助手，可能更重视 RAG、指令跟随、安全拒答和私有知识可靠性。如果目标是代码模型，需要提高代码数据比例、设计代码评估和执行反馈。如果目标是端侧模型，需要从一开始考虑参数规模、量化和推理延迟。

面试表达：训练方案必须从目标倒推。不同目标会改变数据配比、模型规模、上下文长度、后训练方式和评估指标。

## Base Model、Instruct Model、Chat Model、Reasoning Model

面试中经常会问这些模型有什么区别。

| 类型 | 训练重点 | 行为特点 | 常见用途 |
| --- | --- | --- | --- |
| Base Model | 预训练 next token prediction | 会补全文本，但不一定听指令 | 继续训练、研究、作为后训练起点 |
| Instruct Model | SFT 指令微调 | 能按指令完成任务 | 问答、摘要、抽取、格式化输出 |
| Chat Model | 多轮对话和安全对齐 | 更像助手，能维持对话状态 | 通用聊天、客服、Copilot |
| Reasoning Model | 推理数据和强化学习/偏好优化 | 更擅长复杂问题分解和验证 | 数学、代码、科学推理、规划 |

Base model 学到的是语言和知识分布，它可能能续写论文、代码和网页，但未必会回答“请一步步解释”。Instruct model 通过指令数据学会输入输出格式和任务意图。Chat model 进一步强调多轮交互、拒答、安全和语气。Reasoning model 通过推理轨迹、可验证奖励或强化学习等方式强化复杂问题求解能力。

常见误区：把 instruct model 当成 base model。很多公开模型已经做过 SFT 或对齐，不再是纯 base。面试中如果讨论预训练 loss、困惑度和 scaling law，要注意你说的是 base model；如果讨论 helpfulness、harmlessness 和多轮对话，要注意你说的是 chat model。

面试表达：base model 解决“学会语言和知识”，instruct/chat model 解决“按人类意图交互”，reasoning model 进一步解决“复杂问题求解和自我验证”。

## 数据、模型、系统、评估、成本的联动

训练项目里没有孤立决策。任何一个选择都会牵动其他部分。

### 数据影响模型能力

数据是能力来源。模型能否写代码、做数学、懂中文、理解法律文本，首先取决于训练中是否有足够高质量相关数据。

数据不只是越多越好。低质量网页、重复内容、模板垃圾、模型生成垃圾、污染 benchmark 的数据都会带来问题。重复数据会浪费 token 预算，低质量数据会降低学习效率，污染数据会让评估失真。

### 模型规模影响训练策略

模型越大，容量越强，但训练更不稳定，显存和通信成本更高。小模型可能受容量限制，大模型可能受数据和系统限制。

如果算力有限，与其盲目扩大参数，不如合理选择模型大小、训练 token 数和数据质量。Chinchilla 风格结论提醒我们：参数量和训练 token 数要匹配，训练不足的大模型不一定比充分训练的小模型好。

### 系统影响可执行性

一个方案理论上可行，不代表系统上能跑。大模型训练需要考虑 GPU 数量、显存、网络带宽、存储吞吐、checkpoint 保存、容错恢复和监控。

如果数据加载跟不上，GPU 会空转。如果通信瓶颈严重，增加 GPU 不一定提升吞吐。如果 checkpoint 保存太慢，训练会频繁暂停。如果没有良好监控，loss spike 和 NaN 可能很晚才被发现。

### 评估影响方向判断

没有可靠评估，就不知道训练是否变好。预训练阶段要看 validation loss、benchmark、per-domain loss 和下游任务。后训练阶段要看指令跟随、偏好、安全、事实性、格式稳定和人工评测。

评估也会误导。如果 benchmark 被污染，分数会虚高。如果只看平均分，会掩盖某些关键能力退化。如果只看自动指标，会漏掉风格、安全和幻觉问题。

### 成本影响方案边界

训练成本包括 GPU 时间、数据处理、存储、工程人力、失败实验和推理成本。很多方案论文上有效，但在真实项目中因为成本不可接受而不能采用。

面试表达：训练方案要考虑端到端成本。算法选择不能只看效果，还要看训练吞吐、显存、通信、checkpoint、推理延迟和维护复杂度。

## 一个训练项目的关键产物

如果你负责训练一个模型，最终不是只交一个 checkpoint。你需要交付一组工程产物。

| 产物 | 作用 |
| --- | --- |
| 数据说明 | 数据来源、清洗规则、配比、去重和污染检测 |
| Tokenizer 方案 | vocab、特殊 token、覆盖语言和兼容性 |
| 模型配置 | 参数量、层数、hidden size、head、上下文长度 |
| 训练配置 | batch size、学习率、warmup、optimizer、精度、并行策略 |
| 监控面板 | loss、grad norm、学习率、吞吐、显存、NaN、数据速度 |
| Checkpoint 策略 | 保存频率、保留策略、恢复验证、最优选择标准 |
| 评估报告 | 自动评估、人工评测、安全评测、退化分析 |
| 风险记录 | 已知问题、失败实验、异常处理和回滚方案 |
| 部署建议 | 量化、推理配置、上下文长度、硬件需求和成本估算 |

这些产物体现了训练工程的可复现性。没有这些记录，即使某次训练跑出了好模型，也很难解释为什么好，下一次也难以复现。

## 预训练在做什么

预训练通常使用 next token prediction。给定前文 token，模型预测下一个 token。

```text
input:  x_1, x_2, ..., x_{t-1}
target: x_t
loss:   cross entropy(predicted distribution, true token)
```

这个目标看起来简单，但在大规模数据和大模型容量下，模型会学习语言结构、事实知识、代码模式、数学模式、推理模板和世界常识。

预训练阶段的核心问题：

| 问题 | 说明 |
| --- | --- |
| 数据够不够好 | 是否覆盖目标能力，是否去重和过滤 |
| Tokenizer 是否合适 | 是否支持多语言、代码、数学符号和特殊格式 |
| 模型规模是否匹配 | 参数量和训练 token 是否匹配预算 |
| 训练是否稳定 | loss 是否正常下降，有无 spike、NaN、梯度异常 |
| 系统吞吐是否高 | GPU 利用率、MFU、数据加载和通信是否健康 |
| checkpoint 如何选择 | 不能只看最后一步，要结合验证和下游评估 |

预训练的输出是 base model。它是后续 SFT、DPO、RLHF、工具调用和多模态扩展的基础。

面试表达：预训练的目标是用大规模通用数据训练一个能力底座。它不直接保证模型会当助手，但决定了模型的知识、语言、代码和推理上限。

## 后训练在做什么

后训练是把 base model 变成可用助手的关键阶段。

常见后训练包括 SFT、偏好优化、安全对齐和强化学习式训练。

| 方法 | 数据 | 目标 |
| --- | --- | --- |
| SFT | 指令和标准回答 | 学会按指令输出 |
| RLHF/PPO | 偏好数据、reward model | 优化人类偏好奖励 |
| DPO | chosen/rejected 偏好对 | 直接优化偏好，不显式训练 reward model |
| GRPO/GSPO/DAPO/DCPO | 可验证奖励、组内比较或改进策略目标 | 强化推理、代码、数学和对齐能力 |
| Safety Tuning | 安全指令、拒答样本、红队数据 | 降低有害和越权输出 |

后训练不能完全替代预训练。如果 base model 没学过足够代码，少量代码 SFT 很难把它变成强代码模型。如果 base model 缺乏数学基础，偏好优化可能让答案格式更像推理，但未必真正提高正确率。

后训练也可能带来能力退化。过度对齐可能降低开放生成能力，低质量 SFT 可能让模型变啰嗦，安全数据配比不当可能导致过度拒答。

面试表达：后训练的作用是把 base model 的能力组织成符合人类意图的行为，而不是凭空创造所有能力。它的风险是过拟合风格、能力退化和安全-有用性失衡。

## 训练监控要看什么

训练过程中必须持续监控。只等训练结束再评估，风险太高。

常见监控指标：

| 类别 | 指标 | 说明 |
| --- | --- | --- |
| 优化 | train loss、validation loss、learning rate | 判断收敛是否正常 |
| 梯度 | grad norm、clipping ratio | 判断是否梯度爆炸或异常 |
| 数值 | NaN、Inf、loss spike | 判断混合精度和数据异常 |
| 系统 | tokens/sec、MFU、GPU utilization | 判断吞吐是否健康 |
| 显存 | allocated/reserved、activation memory | 判断是否接近 OOM |
| 数据 | dataloader speed、样本分布、坏样本率 | 判断数据 pipeline 是否稳定 |
| checkpoint | 保存耗时、恢复成功率 | 判断容错能力 |
| 评估 | benchmark、per-domain loss | 判断能力变化 |

不同异常对应不同层面问题。loss 突然 spike 可能是坏数据、学习率过大、梯度爆炸或数值溢出。吞吐下降可能是数据加载、网络通信、checkpoint 保存或某个节点故障。validation loss 不降可能是数据分布、学习率、模型容量或 bug。

面试表达：训练监控要同时看优化健康、数值健康、系统健康和评估健康。只看 loss 不够，因为很多系统问题不会立刻反映在 loss 上。

## Checkpoint 不是越后越好

很多初学者会默认训练最后一步的 checkpoint 最好。真实训练中不一定。

预训练中，如果学习率 schedule 设计合理，后期 checkpoint 通常更好，但也可能出现过拟合某些数据、评估退化或训练异常。后训练中，训练太久可能导致过拟合风格、指令模板化、拒答过多或通用能力下降。

选择 checkpoint 需要综合：

| 维度 | 说明 |
| --- | --- |
| Validation loss | 基础收敛指标 |
| 核心 benchmark | 数学、代码、知识、多语言等目标能力 |
| 人工评测 | 指令质量、风格、事实性和安全性 |
| 回归测试 | 旧能力是否退化 |
| 安全评测 | 拒答、越狱、敏感内容 |
| 推理成本 | 延迟、显存、吞吐 |

面试表达：checkpoint 选择是多目标决策。不能只看 loss 或单个 benchmark，要看目标能力、安全、退化和部署成本。

## 训练失败常见来源

训练失败不一定是模型结构错。更常见的是系统、数据和配置问题。

| 问题 | 表现 | 常见原因 |
| --- | --- | --- |
| Loss 不下降 | 训练无效 | learning rate 错、label 错、mask 错、数据格式错 |
| Loss spike | 突然升高 | 坏 batch、学习率过大、梯度爆炸、数值溢出 |
| NaN/Inf | 训练中断 | 混合精度不稳定、除零、softmax 溢出、梯度异常 |
| 吞吐低 | GPU 利用率低 | dataloader 慢、通信瓶颈、batch 太小、checkpoint 阻塞 |
| 评估异常 | 分数不可信 | benchmark 污染、prompt 格式错、评估脚本 bug |
| 能力退化 | 某些任务变差 | 后训练数据偏、过拟合、安全数据过强 |
| 恢复失败 | checkpoint 不能用 | 保存不完整、版本不一致、optimizer state 丢失 |

排查顺序通常是：先确认数据和 label，再看 loss/mask，再看学习率和 optimizer，再看精度和梯度，再看分布式同步，再看评估脚本。

面试表达：训练 debug 要分层排查。不要一上来就改模型结构，先确认数据、目标、mask、学习率、精度和系统状态。

## 成本意识

大模型训练成本很高，所以任何训练方案都要有成本意识。

成本包括：

| 成本 | 示例 |
| --- | --- |
| 算力 | GPU 小时、集群占用、通信网络 |
| 数据 | 采集、清洗、标注、存储、版权审核 |
| 工程 | 训练框架、监控、容错、评估平台 |
| 实验 | 失败 run、消融实验、超参搜索 |
| 推理 | 显存、吞吐、延迟、KV cache |
| 维护 | 模型更新、安全修复、回归评估 |

训练前应该估算 token 数、模型参数量、batch size、训练步数、GPU 数量、预期吞吐、总训练时长和失败重跑风险。

一个简单估算思路：

```text
训练步数 = 总训练 token 数 / global batch token 数
总时长 = 训练步数 / 每秒训练 step 数
总 GPU 小时 = 总时长小时数 * GPU 数量
```

真实项目还要考虑 checkpoint、评估、故障恢复、数据处理和实验重跑带来的额外成本。

面试表达：负责训练不能只说“用更多 GPU 跑更大模型”。要能估算预算、吞吐、失败风险和最终部署成本。

## 面试回答模板：如何训练一个大模型

如果面试官问“给你资源，你如何训练一个大模型”，可以按下面结构回答。

第一步，明确目标和约束。我会先确定模型用途、目标能力、参数规模、训练预算、推理场景和安全要求。例如是通用助手、代码模型还是领域模型。

第二步，设计数据 pipeline。我会收集网页、书籍、代码、论文、多语言、数学和领域数据，做清洗、去重、质量过滤、安全过滤、benchmark contamination 检测和数据配比。

第三步，确定 tokenizer 和模型架构。我会根据语言覆盖、代码和数学符号需求选择 tokenizer，根据预算和目标选择参数量、上下文长度、RoPE、GQA、FFN 结构和初始化方案。

第四步，组织预训练。我会设置 optimizer、learning rate schedule、batch size、混合精度、gradient clipping、分布式并行策略和 checkpoint 策略，同时监控 loss、grad norm、吞吐、显存、NaN 和数据速度。

第五步，做阶段性评估。我会定期评估 validation loss、per-domain loss、知识、数学、代码、多语言、长上下文和安全基础指标，选择候选 checkpoint。

第六步，做后训练和对齐。我会先 SFT 学指令，再根据目标选择 DPO、PPO/RLHF、GRPO 或其他偏好/强化学习方法，加入安全数据和拒答策略，同时监控能力退化。

第七步，做最终评估和部署准备。我会用自动 benchmark、人工评测、红队测试、业务 golden set、推理压测和成本评估选择最终 checkpoint，并准备量化、蒸馏或部署优化。

最后，我会强调风险控制：数据污染、训练不稳定、系统故障、后训练退化、安全漏洞和成本超预算都需要提前设计监控和回滚方案。

## 一个常见错误回答

错误回答：我会用 Transformer decoder-only 架构，用 AdamW 优化器，训练 next token prediction，然后做 SFT 和 RLHF。

这个回答不是完全错，但太粗。问题在于没有说明数据怎么来、tokenizer 怎么选、模型规模如何定、分布式怎么跑、训练是否稳定、怎么评估、checkpoint 怎么选、后训练有什么风险、成本如何控制。

更好的回答是：Transformer、AdamW 和 next token prediction 只是训练方案中的一部分。完整训练方案必须覆盖目标、数据、架构、系统、优化、评估、后训练、安全和成本。

## 本章小结

大模型训练是一项全链路工程。预训练负责形成能力底座，后训练负责把能力组织成可用行为，评估负责判断方向是否正确，系统工程负责让训练稳定高效地跑完，成本管理决定方案是否现实。

本章最重要的结论如下：

1. 训练方案必须从目标和约束出发，而不是从模型结构出发。
2. 数据、tokenizer、模型、优化器、分布式系统、评估和成本互相影响。
3. Base model、instruct model、chat model 和 reasoning model 的训练目标不同。
4. 后训练能改善交互和偏好，但不能完全弥补预训练能力缺口。
5. 训练监控要同时看 loss、梯度、数值、吞吐、数据和评估。
6. checkpoint 选择是多目标决策，不是默认最后一步最好。
7. 面试中回答训练全流程，要体现工程闭环和风险意识。
