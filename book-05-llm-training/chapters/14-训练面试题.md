# 第十四章：训练面试题

## 高频问题

1. 如果让你训练一个 7B 模型，你会怎么设计？
2. 训练 loss 突然 spike，你怎么排查？
3. 为什么训练 loss 降低不代表模型能力一定提升？
4. 如何检测 benchmark contamination？
5. 如何选择 batch size 和 learning rate？
6. ZeRO、FSDP、Tensor Parallel 各自解决什么问题？
7. 如何估算训练显存？
8. SFT 后模型变笨了怎么办？
9. 如何训练一个 VLM？
10. 如何控制大规模训练成本？

## 回答原则

训练类面试题要按“目标、数据、模型、优化、系统、评估、风险、成本”的顺序回答。

## 如何回答训练类面试题

训练类面试题最忌讳只背一个点。

例如面试官问：

```text
如果让你训练一个 7B 模型，你会怎么做？
```

低质量回答是：

```text
我会用 Transformer、AdamW、数据清洗、分布式训练。
```

更好的回答应该有结构：

```text
目标 -> 数据 -> tokenizer -> 架构 -> 训练目标 -> 优化器 -> 分布式 -> 监控 -> 评估 -> 风险 -> 成本
```

面试官真正想看的是你能不能把训练当成一个系统，而不是只知道几个术语。

## 1. 如果让你训练一个 7B 模型，你会怎么设计？

### 回答框架

1. 先明确目标：通用、代码、数学、领域还是 chat。
2. 定义数据规模和数据配比。
3. 选择 tokenizer 和模型架构。
4. 设计预训练目标和训练配置。
5. 选择分布式策略。
6. 建立监控、checkpoint 和评估。
7. 规划后训练和成本。

### 标准回答

```text
我会先明确 7B 模型的目标能力，比如是通用 base model、代码模型还是领域模型。目标会决定数据配比、tokenizer、上下文长度和评估集。

数据上，我会构建多来源语料，包括高质量网页、书籍、代码、数学、多语言和领域数据，做清洗、去重、质量过滤和 benchmark contamination 检测，再按目标能力设计 data mixture。

模型上，一般会选择 decoder-only Transformer，确定层数、hidden size、head 数、FFN size、RoPE、RMSNorm、SwiGLU、GQA 等配置，并确保参数量和训练 token 数匹配。

训练上，使用 next-token prediction，优化器通常以 AdamW 为强基线，配合 warmup、cosine decay、gradient clipping、BF16 mixed precision。分布式上根据显存和吞吐选择 FSDP/ZeRO、Tensor Parallel 或 Pipeline Parallel。

监控上要看 train loss、val loss、per-domain loss、grad norm、lr、tokens/s、MFU、NaN/Inf、checkpoint 状态。评估上要看 validation loss、代码/数学/多语言 benchmark、安全初筛和污染检查。最后选择 checkpoint 进入 SFT、偏好优化和安全对齐。
```

### 常见追问

1. 你怎么定数据配比？
2. 你怎么判断训练是否健康？
3. 你怎么估算训练成本？
4. 如果训练中 loss spike 怎么办？

## 2. 训练 loss 突然 spike，你怎么排查？

### 回答框架

1. 定位第一次 spike 的 step。
2. 检查异常 batch。
3. 检查 grad norm 和 learning rate。
4. 检查 mixed precision。
5. 检查 checkpoint 恢复和分布式状态。
6. 做单变量修复。

### 标准回答

```text
我不会一上来就调学习率，而是先定位第一次 spike 的 step。然后保存该 step 的 batch，检查 input length、decoded text、特殊 token、label mask 和有效 label 数，判断是否是坏数据或格式错误。

同时看 grad norm 是否同步爆炸、learning rate 是否刚到 peak、warmup 是否太短、gradient clipping 是否生效。如果 loss 进入 NaN/Inf，还要检查 mixed precision、FP16 overflow、loss scaling 和 optimizer state。

如果发生在 checkpoint 恢复后，还要确认 optimizer、scheduler、scaler、random state 和数据位置是否完整恢复。修复时一次只改一个因素，例如过滤异常 batch、降低 peak lr、延长 warmup、开启 clipping 或改 BF16，然后对比曲线是否恢复。
```

### 常见追问

1. 如果 spike 后自己恢复，要不要管？
2. 如何判断是数据问题还是优化器问题？
3. 如果所有 rank 同时 NaN 怎么办？

## 3. 为什么训练 loss 降低不代表模型能力一定提升？

### 回答框架

1. train loss 只反映训练分布。
2. 可能存在重复、污染或过拟合。
3. 平均 loss 掩盖 domain 退化。
4. 下游能力和训练目标不完全一致。
5. 需要综合评估。

### 标准回答

```text
训练 loss 只说明模型在训练数据上的 next-token prediction 更好，不代表真实能力一定提升。比如数据重复会让 train loss 下降但泛化不一定好；benchmark contamination 会让评估虚高；整体 loss 下降可能掩盖代码、数学或多语言能力退化。

另外，预训练 loss 和指令遵循、事实性、安全性、推理能力并不完全等价。所以我会同时看 validation loss、per-domain loss、下游 benchmark、人工评测、regression eval 和 safety eval，不能只根据 train loss 判断模型变强。
```

## 4. 如何检测 benchmark contamination？

### 回答框架

1. 明确污染形式。
2. 做 exact match。
3. 做 near dedup。
4. 做 embedding 检索。
5. 对高风险样本人工抽检。

### 标准回答

```text
Benchmark contamination 指评估集或其近似版本出现在训练数据中，导致评估分数虚高。我会先对 benchmark 题目、答案、解析和代码测试样例做 exact match 检查，再用 n-gram、MinHash、SimHash 或 embedding 检索做近似匹配。

对于代码、数学和考试题，还要检查题面改写、答案解析泄漏和相似模板。高风险命中需要人工抽检，并在评估报告中标注 contamination 风险。最终要保证评估集和训练数据隔离，否则 benchmark 结论不可信。
```

## 5. 如何选择 batch size 和 learning rate？

### 回答框架

1. batch size 受显存、稳定性和吞吐限制。
2. learning rate 和 batch size 联动。
3. 需要 warmup 和 decay。
4. 用小规模实验搜索。

### 标准回答

```text
Batch size 要看 global batch，而不是单卡 batch。global batch = micro batch × gradient accumulation × data parallel size。它影响梯度噪声、吞吐和收敛稳定性。

Learning rate 要和 batch size、模型规模、数据质量和 optimizer 一起调。大模型训练通常需要 warmup，避免初期不稳定，再用 cosine decay 或其他 schedule。实际中我会先用小规模实验找稳定区间，观察 loss、grad norm、NaN、validation loss，再决定 peak lr、warmup steps 和 clipping。
```

## 6. ZeRO、FSDP、Tensor Parallel 各自解决什么问题？

### 标准回答

```text
ZeRO 和 FSDP 主要解决训练状态显存问题。ZeRO-1 切 optimizer state，ZeRO-2 进一步切 gradient，ZeRO-3 连 parameter 也切。FSDP 类似全切分数据并行，forward/backward 时按需 gather 参数，训练后再 shard 回去。

Tensor Parallel 切的是层内矩阵计算，比如 attention 和 MLP 的大矩阵，适合单层太大或单卡算力不够的模型。它通信更频繁，通常更适合同节点高速互联。

简单说，ZeRO/FSDP 切训练状态，TP 切模型层内计算。
```

### 常见追问

1. ZeRO-3 为什么通信更大？
2. FSDP 和 DDP 区别是什么？
3. TP 和 PP 怎么组合？

## 7. 如何估算训练显存？

### 回答框架

1. 参数。
2. 梯度。
3. optimizer state。
4. 激活。
5. buffer。

### 标准回答

```text
训练显存不能只看模型权重。以 AdamW 为例，除了参数，还要保存梯度、一阶动量、二阶动量，可能还有 FP32 master weights。粗略看，如果权重和梯度是 BF16，optimizer state 是 FP32，那么仅训练状态就可能接近十几 bytes per parameter。

此外还要算激活显存，它和 batch size、sequence length、hidden size、layers 强相关。长上下文训练时 activation 可能非常大。因此显存优化通常包括 BF16、activation checkpointing、减小 micro-batch、FlashAttention、ZeRO/FSDP 和参数高效微调。
```

## 8. SFT 后模型变笨了怎么办？

### 回答框架

1. 判断变笨的表现。
2. 检查 SFT 数据质量。
3. 检查数据配比和训练步数。
4. 检查学习率和 loss mask。
5. 做 regression eval。

### 标准回答

```text
我会先定义“变笨”是哪些能力退化，比如数学、代码、事实问答、多语言还是格式遵循。然后检查 SFT 数据是否质量低、模板化、答案过长、覆盖太窄，是否把 user prompt 也算了 loss，是否训练步数太多或学习率太大。

SFT 的风险是过拟合指令风格，导致 base model 的通用能力退化。处理上可以降低 lr、减少 epoch、提升数据质量、混入通用 replay 数据、调整数据配比，并用 regression eval 对比 SFT 前后能力变化。
```

## 9. 如何训练一个 VLM？

### 回答框架

1. 选择 vision encoder。
2. 设计 connector。
3. 接入 LLM。
4. 做图文对齐和 instruction tuning。
5. 做评估和安全检查。

### 标准回答

```text
VLM 通常由 vision encoder、connector 和 LLM 组成。vision encoder 提取图像特征，connector 把视觉特征映射到 LLM hidden space，LLM 根据图像和文本 prompt 生成回答。

训练上可以先做图文对齐或 connector 预训练，再做多模态 instruction tuning，覆盖 caption、VQA、OCR、图表理解、空间关系和多轮对话。根据资源可以冻结 vision encoder 或 LLM，只训练 connector，也可以联合微调。评估上要看 OCR、幻觉、计数、空间关系、图表理解和安全风险。
```

## 10. 如何控制大规模训练成本？

### 回答框架

1. 目标和规模匹配。
2. 数据和 token 预算。
3. 提升吞吐和 MFU。
4. 降低失败率。
5. 管理 checkpoint 和评估成本。

### 标准回答

```text
我会先从目标倒推模型规模和训练 token，避免模型太大但 token 不够。然后优化训练吞吐和 MFU，比如合理选择 batch、并行策略、FlashAttention、数据加载和通信重叠。

显存上用 BF16、activation checkpointing、ZeRO/FSDP 等策略。流程上先做小规模 smoke test，降低失败重跑成本；训练中做好监控和自动恢复；checkpoint 采用 latest、milestone、best 分层保留，避免存储成本失控。最终成本控制要同时看 GPU、存储、网络、人力和失败重跑成本。
```

## 11. Base model、Instruct model、Chat model 有什么区别？

### 标准回答

```text
Base model 主要通过预训练做 next-token prediction，学到语言、知识和通用模式，擅长续写但不一定会按指令回答。Instruct model 经过 SFT，学会遵循指令和输出格式。Chat model 进一步经过多轮对话、安全对齐和偏好优化，更像可用助手，能处理系统指令、多轮上下文和拒答边界。
```

## 12. DPO 和 PPO 有什么区别？

### 标准回答

```text
PPO 是在线 RLHF 方法，通常需要 reward model、reference model 和 critic，通过模型采样回答、reward 打分和 KL 约束来更新 policy。优点是能优化当前模型真实生成分布，缺点是复杂、成本高、稳定性挑战大。

DPO 是离线偏好优化，直接用 prompt/chosen/rejected 偏好对训练模型，不需要显式 reward model 和在线 rollout。它简单稳定，但依赖离线偏好数据，不一定适合需要在线探索的 reasoning RL 场景。
```

## 13. 训练类面试回答 checklist

回答训练题时，最后可以自检：

```text
1. 是否先讲目标？
2. 是否讲数据和 tokenizer？
3. 是否讲模型架构和规模？
4. 是否讲训练目标和优化器？
5. 是否讲分布式和显存？
6. 是否讲监控和 debug？
7. 是否讲 checkpoint 和评估？
8. 是否讲成本和风险？
9. 是否讲 trade-off？
```

如果一个回答覆盖这些点，就比较完整。

## 14. 本章小结

本章把第五册核心内容转化成面试回答模板。

核心结论：

1. 训练类面试要按系统链路回答，不要只背名词。
2. 设计训练方案要从目标、数据、模型、优化、系统、评估和成本展开。
3. Debug 题要先定位层级，再给修复方案。
4. 评估题要强调可信评估、污染检测和回归测试。
5. 分布式题要说清切了什么、省了什么、增加了什么通信。
6. 成本题要能做量级估算。
7. 后训练题要区分 SFT、PPO、DPO、reasoning RL 和安全对齐。
8. 多模态训练题要强调模态对齐、数据质量和评估难度。
