# 第二章：数据工程

## 本章目标

理解数据是大模型能力的核心来源，掌握数据收集、清洗、去重、过滤、配比和质量评估。

## 核心议题

1. 数据来源：网页、书籍、论文、代码、对话、数学、多语言、多模态数据。
2. 数据清洗：格式修复、乱码处理、语言识别、质量打分、安全过滤。
3. 数据去重：exact dedup、near dedup、document-level dedup、benchmark contamination 检测。
4. 数据配比：通用文本、代码、数学、多语言和专业领域数据如何混合。
5. 数据质量：高质量数据为什么可能比更多低质量数据更重要。

## 面试重点

训练大模型时，数据问题通常比模型结构问题更难。面试回答要体现数据质量、污染检测、分布覆盖和训练目标之间的联系。

## 为什么数据工程是训练起点

很多人把大模型训练理解成“把数据丢进去跑 loss”，但真实情况是：数据工程决定模型见过什么、学得什么、会不会被污染、会不会偏科。

如果数据出问题，后面再调学习率、换优化器、加更多 GPU，通常只能缓解，不能根治。

数据工程至少要回答四个问题：

1. 数据从哪里来。
2. 数据质量如何保证。
3. 数据之间如何混合。
4. 如何证明数据没有污染评估集。

面试表达：训练系统里，数据不是前置步骤，而是能力上限的主要决定因素之一。

### 数据工程的资料边界

按照 `WRITING_PLAN.md` 的要求，本章第二轮精修前核对了 T5 / C4 数据清洗、GPT-3 预训练数据混合、The Pile 数据集构建、训练数据去重论文、多语言温度采样，以及 Hugging Face Datasets 的 dataset card / DatasetInfo 资料。

本章聚焦训练数据工程的最小闭环：来源、schema、清洗、质量打分、去重、污染检测、数据配比、版本和报告。它不展开爬虫法务、版权治理、分布式数据湖、在线标注平台、数据合成系统和生产级 LSH 索引的完整实现；这些内容会在第九册数据工程和后续平台章节继续展开。

## 1. 数据来源有哪些

大模型常见训练数据大致可以分成下面几类：

| 数据类型 | 作用 | 常见风险 |
| --- | --- | --- |
| Web 文本 | 覆盖广泛知识和表达方式 | 噪声大、重复多、营销噪声多 |
| 书籍和论文 | 结构化、质量较高 | 版权、领域偏差 |
| 代码 | 提升编程能力和结构化推理 | 许可问题、测试污染 |
| 对话数据 | 提升指令遵循和多轮交互 | 模板化、低信息密度 |
| 数学数据 | 提升推理和符号计算 | 格式不统一、答案泄漏 |
| 多语言数据 | 提升跨语言能力 | 语言识别错误、配比失衡 |
| 多模态数据 | 提升图文、音频、视频能力 | 对齐错误、标注噪声 |

不同目标需要不同数据。通用 base model 更重视广覆盖和去重；代码模型要提高代码和单元测试相关数据比例；企业助手更重视高质量业务文档、FAQ、对话和安全策略；reasoning model 更重视可验证推理、数学和代码执行反馈。

## 2. 数据处理流水线

一个典型的数据 pipeline 可以写成：

```text
raw data
  -> 采集
  -> 格式统一
  -> 语言识别
  -> 清洗过滤
  -> 去重
  -> 污染检测
  -> 质量打分
  -> 数据配比
  -> shard / pack
  -> training dataloader
```

每一步都可能影响最终训练质量。

可以把原始语料写成：

```math
D_{\mathrm{raw}}=\{d_i\}_{i=1}^{n}
```

每条样本至少包含文本和元数据：

```math
d_i=(x_i,m_i)
```

其中 `x_i` 是正文内容，`m_i` 是 source、language、domain、timestamp、license、quality_score、filter_reason 等元数据。数据工程的目标不是只得到一堆文本，而是得到可追踪、可过滤、可混合、可复现的训练语料：

```math
D_{\mathrm{train}}=F_{\mathrm{mix}}(F_{\mathrm{contam}}(F_{\mathrm{dedup}}(F_{\mathrm{clean}}(D_{\mathrm{raw}}))))
```

这个写法强调一件事：每个阶段都应该留下统计和版本信息，否则训练后发现某个 benchmark 或能力退化时，很难定位到底是哪批数据造成的。

### 2.1 采集

采集阶段关注的是“拿到什么数据”。

常见来源包括爬虫、公开语料、内部文档、代码仓库、人工标注、合成数据和历史日志。

采集阶段的常见问题：

1. 来源不可靠。
2. 格式不统一。
3. 元数据缺失。
4. 采样分布偏向热门网页。

面试时不要只说“我抓了很多网页”，而要说清楚来源、许可、字段和风险。

### 2.2 格式统一

原始数据通常格式很乱，可能是 HTML、Markdown、JSON、CSV、PDF 抽取文本或对话日志。

统一格式的目标是把数据整理成训练系统可消费的 schema，例如：

```json
{
  "id": "sample_001",
  "text": "...",
  "source": "web",
  "language": "zh",
  "domain": "general",
  "quality_score": 0.87
}
```

对于对话数据，还可能包含：

```json
{
  "messages": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```

格式统一的目的不是好看，而是让后续清洗、统计、混合和追踪都能自动化。

## 3. 清洗和过滤

清洗的目标是去掉明显无效或有害的数据。

常见过滤项：

1. 乱码、空文本、重复空行。
2. 过短或过长的异常样本。
3. 营销噪声、灌水、模板页。
4. 语言不匹配。
5. 低质量自动翻译。
6. 敏感或违法内容。
7. 明显抄袭或模型生成垃圾。

### 3.1 质量打分

实际系统中，常会给样本打一个质量分，再按阈值过滤或分层采样。

质量分可能来自规则、分类器、困惑度、重复度、信息密度、结构性、长度分布、领域匹配度等特征。

一个简单思路是：

```math
q_i=
w_{\mathrm{lang}}s_{\mathrm{lang},i}
+w_{\mathrm{struct}}s_{\mathrm{struct},i}
+w_{\mathrm{domain}}s_{\mathrm{domain},i}
+w_{\mathrm{info}}s_{\mathrm{info},i}
-w_{\mathrm{noise}}s_{\mathrm{noise},i}
-w_{\mathrm{dup}}s_{\mathrm{dup},i}
```

其中 `q_i` 是样本质量分，`s_lang` 是语言识别置信度，`s_struct` 是结构完整性，`s_domain` 是领域匹配度，`s_info` 是信息密度，`s_noise` 是噪声分，`s_dup` 是重复风险分。权重 `w` 不应该拍脑袋定完就不变，而要结合目标能力、人工抽检、验证集和坏例分析迭代。

过滤门禁可以写成：

```math
g_i=
\mathbf{1}[q_i\ge \tau_q]\,
\mathbf{1}[\ell_i\in L_{\mathrm{allow}}]\,
\mathbf{1}[r_i\le \tau_r]\,
\mathbf{1}[a_i=0]
```

其中 `tau_q` 是质量阈值，`ell_i` 是语言，`L_allow` 是允许语言集合，`r_i` 是重复或模板化风险，`a_i` 是高风险内容标记。清洗后的集合是：

```math
D_{\mathrm{clean}}=\{d_i\in D_{\mathrm{raw}}\mid g_i=1\}
```

这里的重点不是把规则写成复杂公式，而是说明你知道质量评估是多维度的，并且每条被过滤的样本都应该能解释原因。

### 3.2 过滤不能过头

过滤过度也有风险。

如果规则太严，会把代码片段、表格、公式、口语表达、论坛讨论等有价值内容一起删掉。对某些领域数据来说，低频但高价值的样本尤其重要。

所以清洗不是“越干净越好”，而是“对目标任务最有帮助的分布最优先”。

面试表达：数据清洗的核心不是追求绝对纯净，而是在噪声、覆盖率和成本之间做合适 trade-off。

过滤阶段至少要记录两个比例：

```math
r_{\mathrm{keep}}=\frac{|D_{\mathrm{clean}}|}{|D_{\mathrm{raw}}|}
```

```math
r_{\mathrm{drop},c}=\frac{|\{d_i\in D_{\mathrm{raw}}\mid c_i=c\}|}{|D_{\mathrm{raw}}|}
```

第一个比例告诉你整体保留率，第二个比例告诉你每类过滤原因 `c` 的占比。若某次规则更新让某个语言、领域或长度桶突然大量下降，必须先抽检再进入训练。

## 4. 去重与近似去重

去重是训练数据工程里的关键环节。

原因很简单：重复数据会浪费 token 预算，还会让模型过度记忆某些文本模式，降低泛化。

### 4.1 Exact dedup

exact dedup 是完全相同文本的去重，例如 hash 相同就认为重复。

适合去掉完全复制粘贴、重复网页镜像和数据管线重复导入的问题。

最小形式可以写成：

```math
h_i=H(N(x_i))
```

其中 `N` 是规范化函数，例如统一大小写、空白和标点，`H` 是 hash 函数。保留集合可以写成：

```math
D_{\mathrm{exact}}=\{d_i\mid h_i\notin \{h_1,\ldots,h_{i-1}\}\}
```

面试里要补一句：hash 去重只处理“规范化后完全一样”的文本，不等价于语义去重，也不能替代污染检测。

### 4.2 Near dedup

near dedup 用于处理“内容基本相同但字面不同”的样本，例如：

1. 不同页面的转载。
2. 模板页只改了标题。
3. 同一代码仓库的 fork。
4. 轻微改写的重复问答。

常见方法包括 SimHash、MinHash、shingling、embedding 相似度和局部敏感哈希。

教学上最容易理解的是 shingling + Jaccard。把样本 `x_i` 切成 k-gram 集合：

```math
A_i=\mathrm{shingle}_k(x_i)
```

两个样本的近似相似度是：

```math
J_{ij}=\frac{|A_i\cap A_j|}{|A_i\cup A_j|}
```

如果：

```math
J_{ij}\ge \tau_{\mathrm{near}}
```

就可以把 `d_i` 和 `d_j` 视作近似重复候选，再结合来源、时间、质量分和人工抽检决定保留哪一条。大规模系统不会两两比较全部样本，通常会用 MinHash、SimHash 或 LSH 先召回候选，再做精排或抽检。

### 4.3 为什么近似去重重要

如果只做 exact dedup，很多重复内容仍会保留下来。Web 数据尤其如此，因为同一段内容会在多个站点转载、镜像或生成分页。

近似去重做得好，通常能明显改善训练 token 的有效利用率。

面试表达：去重不是数据清洁的附属步骤，而是控制训练效率、降低记忆偏差和减少 benchmark 污染的核心手段。

## 5. Benchmark contamination

污染检测是训练数据工程里最容易被低估的问题之一。

如果训练数据里包含评估集或其近似变体，模型在 benchmark 上可能“看起来很强”，但其实只是见过题目。

### 5.1 污染形式

1. 评估题原文出现在训练语料里。
2. 题目改写后仍高度相似。
3. 答案或解析泄漏到训练数据中。
4. 代码题的测试样例、题面或解答被爬入训练集。

### 5.2 怎么检测

常见办法有：

1. 字符串或 n-gram 匹配。
2. 近似相似度检索。
3. 题目与训练语料的 embedding 检索。
4. 关键词和结构模式匹配。
5. 人工抽检高风险集合。

可以把 benchmark 集合写成：

```math
B=\{b_j\}_{j=1}^{m}
```

对训练样本 `d_i`，计算它和所有 benchmark 题目的最大相似度：

```math
c_i=\max_{1\le j\le m}s(x_i,b_j)
```

污染过滤门禁是：

```math
g_{\mathrm{contam},i}=\mathbf{1}[c_i<\tau_c]
```

这里的 `s` 可以是字符串相似度、n-gram Jaccard、MinHash 估计、embedding 余弦相似度或多路召回后的综合分。高风险 benchmark 不应该只依赖单一阈值，最好保留高相似样本列表并做人工复核。

### 5.3 为什么面试官爱问

因为污染检测直接关系到你如何相信评估结果。

如果你不能说明污染控制方法，面试官会怀疑你对训练数据和 benchmark 的关系理解不够。

面试表达：数据工程不仅是准备训练集，还要确保评估集不被污染，否则所有指标都不可信。

## 6. 数据配比

数据配比决定模型“偏什么科”。

常见做法不是把所有数据混在一起，而是先按 domain、质量和任务分类，再按比例采样。

### 6.1 为什么要配比

不同数据的学习信号不同。

1. 通用网页数据提供语言和知识覆盖。
2. 代码数据提供结构化推理和程序模式。
3. 数学数据提供符号推理。
4. 对话数据提供指令跟随。
5. 专业领域数据提供垂直能力。

如果代码数据太少，模型代码能力弱；如果对话数据太多，base model 可能过早偏向助手风格；如果低质量网页比例太高，模型会学到大量噪声。

### 6.2 采样方式

常见采样方式包括：

1. 固定比例采样。
2. 温度采样。
3. 按质量分层采样。
4. 按阶段动态调整配比。

温度采样通常用于避免大数据集完全淹没小而高价值的数据集。

假设第 `k` 个数据桶有 `n_k` 个 token 或样本，温度式平滑采样可以写成：

```math
p_k=\frac{n_k^\alpha}{\sum_{j=1}^{K}n_j^\alpha}
```

其中 `0<alpha<=1`。当 `alpha=1` 时，采样比例接近原始规模；当 `alpha` 变小，小数据桶会被相对上采样，大数据桶不会完全淹没代码、数学、多语言或高质量领域数据。

如果计划训练 `D` 个 token，第 `k` 个桶的目标 token 数可以估算为：

```math
D_k=\mathrm{round}(p_kD)
```

真实训练中还要结合质量分、去重率、验证集表现和目标能力动态调整，而不是只按原始数据大小采样。

### 6.3 面试表达

训练数据配比的核心不是“越多越好”，而是“把有限 token 花在最能提升目标能力的数据上”。

## 7. 质量评估与数据版本管理

训练数据不应该是一次性产物。

一个成熟的数据工程流程应该有版本管理、数据统计和质量报告。

### 7.1 统计什么

常见统计项：

1. 样本数。
2. token 数。
3. 语言分布。
4. 领域分布。
5. 长度分布。
6. 重复率。
7. 过滤率。
8. 高风险样本比例。

一份数据质量报告至少应该能回答下面几个量：

```math
R_{\mathrm{dup}}=\frac{|D_{\mathrm{exact}}|-|D_{\mathrm{dedup}}|}{|D_{\mathrm{exact}}|}
```

```math
P_{\mathrm{lang}}(\ell)=\frac{|\{d_i\in D_{\mathrm{train}}\mid \ell_i=\ell\}|}{|D_{\mathrm{train}}|}
```

```math
P_{\mathrm{domain}}(k)=\frac{|\{d_i\in D_{\mathrm{train}}\mid k_i=k\}|}{|D_{\mathrm{train}}|}
```

`R_dup` 看重复率，`P_lang` 看语言分布，`P_domain` 看领域分布。训练 loss 或 benchmark 变化时，这些统计能帮助你判断是模型问题、优化问题，还是数据分布变了。

### 7.2 为什么要版本管理

如果没有版本号，你很难回答：

1. 这次训练和上次差在哪里。
2. 哪批数据导致指标上升或下降。
3. 某个 bug 是不是由数据变更引起的。

数据版本管理的作用，和代码版本管理一样，都是为了可复现。

面试表达：优秀的数据工程不仅有过滤规则，还要有数据版本、统计报告和可回滚能力。

一个简化的数据门禁可以写成：

```math
G_{\mathrm{data}}=
g_{\mathrm{schema}}\,
g_{\mathrm{quality}}\,
g_{\mathrm{dedup}}\,
g_{\mathrm{contam}}\,
g_{\mathrm{mix}}\,
g_{\mathrm{report}}\,
g_{\mathrm{version}}
```

这些 `g` 都是 0/1 检查项：schema 字段齐全、质量阈值通过、重复率可控、污染检测通过、配比满足目标、质量报告完整、数据版本可追踪。面试中这样回答，比只说“我会清洗数据”更有工程可信度。

```python
    text = text.lower()
    text = re.sub(r"[^a-z0-9\u4e00-\u9fff]+", " ", text)
    return re.sub(r"\s+", " ", text).strip()


def tokens(text):
    return normalize(text).split()


def shingles(text, k=2):
    ts = tokens(text)
    if len(ts) < k:
        return set(ts)
    return {tuple(ts[i : i + k]) for i in range(len(ts) - k + 1)}


def jaccard(a, b):
    if not a and not b:
        return 1.0
    if not a or not b:
        return 0.0
    return len(a & b) / len(a | b)


def quality_score(record):
    ts = tokens(record["text"])
    counts = Counter(ts)
    language_score = 1.0 if record["language"] in allowed_languages else 0.0
    structure_score = min(len(record["text"]) / 120, 1.0)
    domain_score = domain_prior.get(record["domain"], 0.70)
    unique_ratio = len(counts) / max(len(ts), 1)
    info_score = min(unique_ratio / 0.75, 1.0)
    noise_score = 1.0 if any(term in counts for term in noise_terms) else 0.0
    repeat_score = max(counts.values()) / max(len(ts), 1)
    score = (
        0.30 * language_score
        + 0.20 * structure_score
        + 0.25 * domain_score
        + 0.20 * info_score
        - 0.35 * noise_score
        - 0.15 * repeat_score
    )
    return round(max(0.0, min(score, 1.0)), 4)


for record in records:
    record["quality"] = quality_score(record)
    record["hash"] = hashlib.sha1(normalize(record["text"]).encode()).hexdigest()[:10]

quality_kept = [r for r in records if r["quality"] >= quality_threshold]
quality_drop = [r["id"] for r in records if r["quality"] < quality_threshold]

seen_hashes = set()
exact_kept = []
exact_drop = []
for record in quality_kept:
    if record["hash"] in seen_hashes:
        exact_drop.append(record["id"])
        continue
    seen_hashes.add(record["hash"])
    exact_kept.append(record)

near_kept = []
near_drop = []
for record in exact_kept:
    record_shingles = shingles(record["text"])
    best = 0.0
    best_id = None
    for kept in near_kept:
        score = jaccard(record_shingles, shingles(kept["text"]))
        if score > best:
            best = score
            best_id = kept["id"]
    if best >= near_dedup_threshold:
        near_drop.append((record["id"], best_id, round(best, 4)))
    else:
        near_kept.append(record)

bench_tokens = [(b["id"], set(tokens(b["text"]))) for b in benchmark]
contam_drop = []
final_kept = []
for record in near_kept:
    rec_tokens = set(tokens(record["text"]))
    best_id, best_sim = max(
        ((bid, jaccard(rec_tokens, btoks)) for bid, btoks in bench_tokens),
        key=lambda item: item[1],
    )
    record["contam_score"] = round(best_sim, 4)
    if best_sim >= contamination_threshold:
        contam_drop.append((record["id"], best_id, round(best_sim, 4)))
    else:
        final_kept.append(record)

domain_counts = Counter(r["domain"] for r in final_kept)
alpha = 0.5
weights = {domain: count**alpha for domain, count in domain_counts.items()}
weight_sum = sum(weights.values())
mixture_raw = {domain: weight / weight_sum for domain, weight in sorted(weights.items())}
mixture = {domain: round(prob, 4) for domain, prob in mixture_raw.items()}

remaining_hashes = [r["hash"] for r in final_kept]
remaining_near_pairs = []
for i, left in enumerate(final_kept):
    for right in final_kept[i + 1 :]:
        sim = jaccard(shingles(left["text"]), shingles(right["text"]))
        if sim >= near_dedup_threshold:
            remaining_near_pairs.append((left["id"], right["id"], round(sim, 4)))

gates = {
    "schema": all({"id", "text", "source", "language", "domain"} <= set(r) for r in records),
    "quality": not any(r["quality"] < quality_threshold for r in final_kept),
    "dedup": len(remaining_hashes) == len(set(remaining_hashes)) and not remaining_near_pairs,
    "contamination": not any(r["contam_score"] >= contamination_threshold for r in final_kept),
    "mixture": abs(sum(mixture_raw.values()) - 1.0) < 1e-12,
}

stats = {
    "raw": len(records),
    "quality_kept": len(quality_kept),
    "exact_kept": len(exact_kept),
    "near_kept": len(near_kept),
    "final_kept": len(final_kept),
    "filter_rate": round(1 - len(final_kept) / len(records), 4),
}

print("quality_by_id=", {r["id"]: r["quality"] for r in records})
print("quality_drop=", quality_drop)
print("exact_drop=", exact_drop)
print("near_drop=", near_drop)
print("contam_drop=", contam_drop)
print("kept_ids=", [r["id"] for r in final_kept])
print("domain_counts=", dict(sorted(domain_counts.items())))
print("mixture_alpha=", alpha)
print("mixture=", mixture)
print("stats=", stats)
print("gates=", gates)
print("gate_pass=", all(gates.values()))
```

这个例子体现了数据处理的思路：

1. 统一入口。
2. 明确过滤规则。
3. 输出可追踪。
4. 保留每一类丢弃原因。
5. 最后用门禁判断能否进入训练。

真实系统会复杂得多，但核心逻辑类似。

## 9. 常见坑

### 9.1 只看数量，不看质量

这是最常见错误。数量大不代表有效 token 多。

### 9.2 过滤规则过拟合某一批数据

如果规则只针对某个来源写死，换个来源就失效。

### 9.3 没有污染检测

评估集被污染后，所有 benchmark 都可能失真。

### 9.4 没有保留元数据

没有 source、language、domain、timestamp，就很难回溯问题。

### 9.5 数据配比固定不变

训练不同阶段需要不同配比。预训练、继续预训练、SFT 和偏好优化的最佳数据结构通常不一样。

## 10. 面试官会怎么问

### 问法 1：为什么数据工程这么重要？

可以这样答：

```text
因为数据决定模型学到什么。数据质量、覆盖率、重复率、污染和配比都会直接影响模型能力上限。很多训练问题看起来像模型问题，实际根因在数据。比如重复数据会浪费 token 预算，污染会让 benchmark 虚高，配比不合理会让模型偏科。
```

### 问法 2：你怎么做数据去重？

可以这样答：

```text
我会分两层做。第一层是 exact dedup，用 hash 去掉完全重复样本；第二层是 near dedup，用 n-gram、SimHash、MinHash 或 embedding 相似度去掉转载、改写和模板化重复内容。这样可以减少重复学习和 benchmark contamination。
```

### 问法 3：怎么判断训练集污染了 benchmark？

可以这样答：

```text
我会做多层检测：先做字符串和 n-gram 匹配，再做近似检索和 embedding 相似度检索，对高风险 benchmark 还会人工抽检。对于代码题，还要重点检查题面、样例、答案和解析是否泄漏到训练语料里。
```

### 问法 4：数据配比怎么定？

可以这样答：

```text
数据配比要从目标倒推。通用 base model 更重视广覆盖和质量平衡，代码模型要提高代码和执行反馈数据比例，企业助手要提高业务文档和对话数据比例。实际中我会先按 domain 和质量分层，再通过固定比例或温度采样调配，并结合验证集表现动态调整。
```

## 11. 小结

本章核心结论：

1. 数据工程是训练起点，不是辅助步骤。
2. 数据来源、清洗、去重、污染检测、配比和版本管理缺一不可。
3. 高质量数据往往比单纯更多数据更重要。
4. Benchmark contamination 会严重破坏评估可信度。
5. 数据配比决定模型偏科方向。
6. 面试中要能讲清数据如何影响能力上限、泛化和评估可信度。
