# 第十三章：训练成本与资源规划

## 本章目标

理解训练一个大模型需要如何估算计算量、显存、时间和成本。

## 核心议题

1. FLOPs 估算。
2. tokens/s 与训练时长。
3. GPU 数量和利用率。
4. MFU。
5. 数据规模和 checkpoint 存储成本。
6. 训练失败和重跑成本。

## 面试重点

能做粗略成本估算，是高级算法岗和训练系统岗位的重要能力。

## 本章资料边界

本章第二轮精修时对照了 Chinchilla / compute-optimal training、PaLM 中 MFU 口径、Megatron-LM 训练系统经验和 MLPerf Training benchmark 的公开资料。这里不写具体云厂商价格，因为价格会随区域、合约、预留实例和时间变化；本章只建立可迁移的估算框架：

1. 用模型参数量、训练 token 数和有效集群算力估算训练时长。
2. 用 MFU/HFU 区分“模型有效计算”和“硬件实际忙碌程度”。
3. 把 checkpoint、评估、故障、重跑和存储保留策略纳入预算。
4. 用量级门禁判断方案是否可执行，而不是追求精确账单。

## 为什么成本和资源规划很重要

大模型训练不是“想训就训”。

一次训练往往要消耗：

1. 大量 GPU 时间。
2. 存储和网络资源。
3. 工程和调试人力。
4. 失败实验的机会成本。

如果前期没有做粗略估算，可能会出现：

1. 模型太大，训练 token 不够。
2. GPU 不够，训练周期太长。
3. checkpoint 过大，存储和恢复成为瓶颈。
4. 失败一次损失巨大。

所以资源规划的核心不是精确到小数点后几位，而是知道量级是否可行。

面试表达：训练成本规划的目标是提前判断方案是否可执行，而不是训练结束后才发现预算不够。

## 1. 训练成本由哪些部分组成

训练成本通常分成四类：

| 成本类别 | 内容 |
| --- | --- |
| 计算成本 | GPU / TPU 时间 |
| 存储成本 | 数据、checkpoint、日志、artifact |
| 网络成本 | 多机通信、存储读写、checkpoint 传输 |
| 人力成本 | 调试、监控、实验管理、故障恢复 |

很多团队只看 GPU 成本，但实际上存储和重跑成本也很高。

如果一个训练实验经常失败，真正的成本可能远超硬件账单。

## 2. 粗略 FLOPs 估算

训练成本常先从 FLOPs 估算开始。

一个非常粗略的思路是：

```text
总计算量 ≈ 模型参数量 × 训练 token 数 × 常数因子
```

Decoder-only Transformer 预训练常用的量级估算是：

```math
C_{\mathrm{train}}\approx 6ND
```

其中 `N` 是参与前向和反向计算的模型参数量，`D` 是训练 token 数。这个公式的意义是快速估算量级：一次 forward 大约是若干倍参数量的乘加，训练还要包含 backward 和梯度计算，所以常用 `6*N*D` 做粗估。

它有几个重要边界：

1. 对极长上下文，attention 的二次项会更明显。
2. 对 MoE，`N` 应该优先理解为每个 token 激活到的参数量，而不是总参数量。
3. 对 activation checkpointing、重计算、低效 kernel 或通信等待，实际硬件消耗会更高。
4. 这个公式估的是训练主循环的模型计算，不包含数据处理、评估、checkpoint 和失败重跑。

更具体地说，Transformer 训练的总计算量和参数规模、序列长度、层数、attention/FFN 结构有关。

在面试中不需要背严格公式，但要知道：

1. 参数越大，计算量越大。
2. token 越多，计算量越大。
3. sequence length 越长，attention 代价越高。
4. MoE 的激活计算和总参数量不完全等价。

### 2.1 为什么 FLOPs 估算有用

因为它能帮助判断：

1. 需要多少 GPU。
2. 需要多久训练。
3. 训练是不是远超预算。

面试表达：FLOPs 估算是把模型规模、token 预算和硬件预算连起来的第一步。

## 3. Tokens/s 和训练时长

训练时长最实用的估算方式之一是看吞吐。

如果知道每秒训练多少 token：

```text
training_time ≈ total_tokens / tokens_per_second
```

写成公式是：

```math
T_{\mathrm{sec}}=\frac{D}{R_{\mathrm{tok}}}
```

其中 `D` 是总训练 token 数，`R_tok` 是长期平均训练吞吐。注意这里应该用长期平均值，而不是短 benchmark 里的峰值。

例如：

```text
总训练 token = 2e12
吞吐 = 200k tokens/s
```

则训练时长大约是：

```text
2e12 / 2e5 = 1e7 秒 ≈ 115 天
```

这还没有考虑恢复、评估、调试和低利用率。

### 3.1 影响 tokens/s 的因素

1. 模型大小。
2. sequence length。
3. batch size。
4. 并行策略。
5. 网络带宽。
6. FlashAttention 等 kernel 优化。
7. 数据加载速度。

### 3.2 不能只看峰值吞吐

某次 benchmark 中 tokens/s 很高，不代表整个训练过程都这么高。

真实训练中还要考虑：

1. checkpoint 保存暂停。
2. validation 停顿。
3. 节点故障重启。
4. 低利用率阶段。

面试表达：训练时长估算要基于平均 tokens/s，而不是实验里的峰值 tokens/s。

## 4. GPU 数量怎么估

GPU 数量取决于：

1. 模型能不能放下。
2. 想要多快训练完。
3. 想要多大 global batch。
4. 并行策略效率如何。

### 4.1 从单卡显存倒推

先判断单卡是否能放下：

```text
模型权重 + 梯度 + optimizer state + activation
```

如果不行，就要上 FSDP、ZeRO、TP、PP 或 checkpointing。

### 4.2 从时长倒推

如果目标是 1 个月内训练完，就必须估算所需 tokens/s，再倒推需要多少 GPU。

设目标训练时长是 `T_day` 天，运行 overhead 比例是 `rho`，则最低平均吞吐需求是：

```math
R_{\mathrm{need}}
=
\frac{D}{86400\cdot T_{\mathrm{day}}\cdot (1-\rho)}
```

其中 `rho` 可以覆盖 validation、checkpoint、重启、短暂停顿和调度空档。真实规划里不要把 `rho` 设成 0。

### 4.3 资源冗余

不要把 GPU 预算卡得太死。

因为训练中会有：

1. 节点故障。
2. 机器掉线。
3. 调试占用。
4. benchmark 和 checkpoint 评估占用。

面试表达：GPU 数量规划要同时满足显存可行性、训练时长和故障冗余。

## 5. MFU

MFU 是模型 FLOPs Utilization，衡量实际训练吞吐相对于理论峰值的利用率。

直觉上：

```text
MFU 越高，说明 GPU 算力利用越充分。
```

### 5.1 为什么 MFU 重要

因为单看 tokens/s 不容易比较不同模型或不同硬件。

MFU 能帮助判断：

1. 训练是否被通信瓶颈拖慢。
2. kernel 是否高效。
3. data loader 是否空转。
4. batch size 是否太小。

如果单个 token 的模型有效计算量是 `c_tok` FLOPs，长期平均吞吐是 `R_tok`，GPU 数量是 `G`，单卡理论峰值是 `F_peak`，那么 MFU 可以写成：

```math
\mathrm{MFU}
=
\frac{c_{\mathrm{tok}}R_{\mathrm{tok}}}
{G F_{\mathrm{peak}}}
```

对 `6ND` 粗估来说，`c_tok≈6N`。所以也可以反过来估算吞吐：

```math
R_{\mathrm{tok}}
\approx
\frac{G F_{\mathrm{peak}}\mathrm{MFU}}
{6N}
```

HFU 更偏硬件实际 FLOPs 利用率，可能把 activation recomputation 等额外计算也算进去；MFU 更偏“完成模型训练目标所需的有效模型 FLOPs”。面试里要说明口径，否则不同团队的数字不能直接比较。

### 5.2 影响 MFU 的因素

1. attention kernel。
2. 通信开销。
3. 数据加载。
4. sequence length。
5. micro-batch size。
6. 并行策略。

面试表达：MFU 是衡量训练效率的重要指标，能帮助判断硬件资源是否被真正用起来。

## 6. 数据规模和 token 规划

训练成本的核心之一是你要喂多少 token。

很多训练方案问题本质上是：

```text
参数规模和 token 规模不匹配
```

如果模型太大但 token 不够，训练就会 under-train。

如果 token 太多但模型太小，容量不够也浪费。

### 6.1 资源规划要回答

1. 目标总 token 数多少。
2. 训练阶段是否分阶段。
3. 数据配比怎么分。
4. 是否需要继续预训练。
5. 是否有多轮后训练。

### 6.2 数据成本

数据本身也有成本：

1. 采集。
2. 清洗。
3. 去重。
4. 标注。
5. 存储。
6. 版本管理。

不要只把成本理解成 GPU 消耗。

## 7. Checkpoint 存储成本

Checkpoint 会占用大量存储。

如果保存频率太高，长期存储和传输成本会非常可观。

### 7.1 需要考虑的点

1. checkpoint 大小。
2. 保存频率。
3. 保留策略。
4. 是否需要 sharded checkpoint。
5. 是否需要压缩。
6. 是否需要异步保存。

如果只保存推理权重，大小近似为：

```math
S_{\mathrm{eval}}\approx N b_w
```

其中 `b_w` 是每个参数的存储字节数，例如 BF16/FP16 通常是 2 bytes。若要恢复训练，通常还要保存梯度、optimizer state、scheduler、RNG、dataloader position 和配置。对 AdamW mixed precision 训练，一个很粗的训练恢复 checkpoint 估算是：

```math
S_{\mathrm{resume}}
\approx
N(b_w+b_g+2b_m+b_{\mathrm{master}})
```

其中 `b_g` 是梯度字节数，`b_m` 是 Adam 一阶/二阶矩每个状态的字节数，`b_master` 是 FP32 master weights 字节数。这个式子不需要死记，关键是知道 resume checkpoint 可能是推理权重的数倍。

保留策略也要估算：

```math
S_{\mathrm{retain}}
=n_{\mathrm{latest}}S_{\mathrm{resume}}
+n_{\mathrm{milestone}}S_{\mathrm{eval}}
+n_{\mathrm{archive}}S_{\mathrm{eval}}
```

保存太少会增加重跑损失，保存太多会把存储和网络打爆。

### 7.2 存储生命周期

常见策略：

```text
latest -> 短期保存，用于恢复
milestone -> 中期保存，用于评估
best -> 长期保存，用于发布
archive -> 冷存储，用于追溯
```

面试表达：checkpoint 存储规划不是技术细节，而是训练成本和故障恢复策略的一部分。

## 8. 失败和重跑成本

训练失败的代价不仅是重新跑几个小时，而是：

1. 已消耗的 GPU 时间。
2. 产出的无效 checkpoint。
3. 人工排查时间。
4. 重新同步调度资源。
5. 可能错过实验窗口。

因此需要估算失败预算。

### 8.1 重跑风险

如果训练过程很脆弱，失败率高，那么实际成本远高于理论成本。

设单次成功训练需要 `H_success` GPU-hours，完整训练失败概率是 `q`，一个粗略的期望成本放大因子是：

```math
E[H]\approx \frac{H_{\mathrm{success}}}{1-q}
```

如果训练能从 checkpoint 恢复，更实用的是估算每次失败平均丢失多少小时。设训练周期内预期失败次数为 `n_fail`，checkpoint 间隔是 `Delta_ckpt` 小时，GPU 数是 `G`，如果故障发生时间近似均匀，则平均丢失半个 checkpoint 间隔：

```math
H_{\mathrm{lost}}
\approx
n_{\mathrm{fail}}\cdot \frac{\Delta_{\mathrm{ckpt}}}{2}\cdot G
```

这就是为什么 checkpoint 间隔、自动恢复和 smoke test 都是成本规划的一部分。

### 8.2 降低重跑成本的方法

1. 定期 checkpoint。
2. 自动恢复。
3. 异常监控。
4. 单变量实验。
5. 小规模预实验。
6. 关键配置先做 smoke test。

面试表达：训练成本不只是成功一次的成本，还要考虑失败概率和重跑成本。

## 9. 资源规划流程

可以按下面顺序做规划：

```text
1. 明确目标：要做 base model、SFT、对齐、VLM 还是 reasoning model？
2. 估算参数规模和 token 数。
3. 估算显存：参数、梯度、optimizer、激活。
4. 估算 tokens/s 和训练时长。
5. 判断 GPU 数量和并行策略。
6. 估算 checkpoint 存储和恢复成本。
7. 估算失败重跑成本。
8. 给出冗余和评估时间预算。
```

这个流程比单纯问“有几张卡”更接近真实项目。

## 10. 一个粗略规划示例

假设你要训练一个中等规模模型，目标是：

```text
1. 训练 token：1e12
2. 训练时长：60 天
3. 预算：若干 GPU 集群
4. 需要中途评估和容错
```

你至少要估：

1. 每秒需要多少 token 吞吐。
2. 单卡能否放下模型。
3. 需要多少并行。
4. checkpoint 存储要多少 TB。
5. 如果失败一次，能否接受重跑。

如果实际 tokens/s 远低于目标，那就要么扩 GPU，要么降模型，要么调整序列长度、并行和优化策略。

### 10.1 最小可运行训练成本审计 demo

下面这个 0 依赖 demo 用一个 toy 训练计划做量级审计。输入包括参数量、训练 token 数、GPU 数、单卡峰值、MFU、目标训练天数、运行 overhead 和 checkpoint 策略；输出包括训练 FLOPs、长期 tokens/s、训练天数、GPU-hours、checkpoint 存储、失败重跑预期损失和门禁结果。

```python
params = 7_000_000_000
train_tokens = 1_000_000_000_000
gpus = 128
peak_tflops_per_gpu = 1_000
mfu = 0.40
target_days = 14
runtime_overhead = 0.08

train_flops = 6 * params * train_tokens
peak_flops = gpus * peak_tflops_per_gpu * 1e12
effective_flops = peak_flops * mfu
tokens_per_s = effective_flops / (6 * params)
base_train_seconds = train_flops / effective_flops
base_train_days = base_train_seconds / 86400
required_tokens_per_s = train_tokens / (
    target_days * 86400 * (1 - runtime_overhead)
)


def gib(num_bytes):
    return num_bytes / 1024**3


weight_gib = gib(params * 2)       # BF16/FP16 inference weights.
grad_gib = gib(params * 2)         # BF16/FP16 gradients, rough estimate.
adam_gib = gib(params * 8)         # FP32 m and v.
master_gib = gib(params * 4)       # FP32 master weights.
resume_ckpt_gib = weight_gib + grad_gib + adam_gib + master_gib
inference_ckpt_gib = weight_gib
retained_storage_gib = 2 * resume_ckpt_gib + 10 * inference_ckpt_gib

gpu_hours_success = gpus * base_train_seconds / 3600
failure_rate_per_day = 0.05
expected_failures = failure_rate_per_day * base_train_days
checkpoint_interval_hours = 4
expected_lost_gpu_hours = (
    expected_failures * (checkpoint_interval_hours / 2) * gpus
)
planned_wall_days = (
    base_train_seconds / 3600 * (1 + runtime_overhead)
    + expected_lost_gpu_hours / gpus
) / 24

gates = {
    "throughput_meets_target": tokens_per_s >= required_tokens_per_s,
    "storage_under_500gib": retained_storage_gib <= 500,
    "planned_under_target": planned_wall_days <= target_days,
    "mfu_reasonable": 0.25 <= mfu <= 0.6,
}

print(f"train_flops={train_flops:.2e}")
print(f"effective_cluster_pflops={effective_flops / 1e15:.1f}")
print(f"tokens_per_s={tokens_per_s:.0f}")
print(f"required_tokens_per_s={required_tokens_per_s:.0f}")
print(f"base_train_days={base_train_days:.2f}")
print(f"planned_wall_days={planned_wall_days:.2f}")
print(f"gpu_hours_success={gpu_hours_success:.0f}")
print(f"resume_ckpt_gib={resume_ckpt_gib:.2f}")
print(f"retained_storage_gib={retained_storage_gib:.2f}")
print(f"expected_failures={expected_failures:.2f}")
print(f"expected_lost_gpu_hours={expected_lost_gpu_hours:.1f}")
print(f"gates={gates}")
print(f"gate_pass={all(gates.values())}")
```

这段 demo 的关键输出应该是：

```text
train_flops=4.20e+22
effective_cluster_pflops=51.2
tokens_per_s=1219048
required_tokens_per_s=898608
base_train_days=9.49
planned_wall_days=10.29
gpu_hours_success=29167
resume_ckpt_gib=104.31
retained_storage_gib=339.00
expected_failures=0.47
expected_lost_gpu_hours=121.5
gates={'throughput_meets_target': True, 'storage_under_500gib': True, 'planned_under_target': True, 'mfu_reasonable': True}
gate_pass=True
```

读这段输出时要注意：`base_train_days` 只来自模型主循环，`planned_wall_days` 才把 checkpoint/评估 overhead 和期望故障损失纳入了计划；`resume_ckpt_gib` 远大于推理权重，是因为训练恢复需要 optimizer state 和 master weights；`gate_pass=True` 只说明这个 toy 方案在设定约束下量级可行，不代表真实集群一定能达到这个 MFU。

## 11. 常见资源规划错误

### 11.1 只看模型参数量

忽略 token 数、训练时长和激活显存。

### 11.2 只看峰值吞吐

忽略 checkpoint、评估和故障重启。

### 11.3 不考虑失败率

训练一旦不稳定，实际成本会成倍上升。

### 11.4 不考虑存储和网络

checkpoint 和数据流会成为瓶颈。

### 11.5 不留余量

所有资源卡满后，调试和容错空间很小。

## 12. 面试官会怎么问

### 问法 1：如果让你估算一个训练项目的成本，你怎么做？

可以这样答：

```text
我会先明确模型规模、训练 token 数、序列长度和目标时长，然后估算参数、梯度、optimizer state 和激活显存，再结合并行策略估算单卡吞吐和整体 tokens/s。接着估 checkpoint 存储、评估频率和失败重跑成本，最后给出 GPU 数量、训练周期和冗余预算。重点不是精确到每分钱，而是判断方案是否可执行。
```

### 问法 2：MFU 是什么？为什么重要？

可以这样答：

```text
MFU 是模型 FLOPs Utilization，用来衡量实际训练计算相对于理论峰值的利用率。它可以帮助判断训练是不是被通信、数据加载、kernel 效率或 batch 太小拖慢。相比只看 tokens/s，MFU 更适合比较不同配置的训练效率。
```

### 问法 3：checkpoint 成本为什么不能忽略？

可以这样答：

```text
因为大模型 checkpoint 很大，尤其带 optimizer state 时体积会数倍增加。保存太频繁会浪费存储和带宽，保存太少又会增加故障重跑成本。所以需要平衡 latest、milestone、best 和 archive 的策略。
```

### 问法 4：为什么训练成本不只是 GPU 费用？

可以这样答：

```text
因为还包括数据处理、存储、网络、调试、人力和失败重跑成本。一个训练系统如果频繁 OOM、NaN 或恢复失败，实际成本可能远高于表面 GPU 账单。所以要把容错和稳定性也纳入成本规划。
```

### 问法 5：怎么判断资源规划是否合理？

可以这样答：

```text
看三件事：模型能否放下、能否在目标时间内训完、失败后是否可恢复。再结合 tokens/s、MFU、checkpoint 存储、数据吞吐和冗余预算判断。如果这三件事任何一个不成立，规划就不合理。
```

## 13. 本章小结

本章核心结论：

1. 训练成本包括计算、存储、网络和人力。
2. FLOPs 和 token 数决定大致训练量级。
3. tokens/s 可以用于估算训练时长。
4. MFU 是衡量训练效率的重要指标。
5. GPU 数量要同时满足显存、时长和冗余。
6. 数据规模、checkpoint 存储和失败重跑都要计入成本。
7. 资源规划要从目标、模型、token、并行、存储和容错六个方面一起看。
8. 只看模型参数或峰值吞吐都会严重低估真实成本。
9. 面试中要能做量级估算，而不是只说“需要很多卡”。
