# 第十二章：评估面试题

重点：把第七册前 11 章的评估方法转成面试可复述、可追问、可落地的回答体系。

面试重点：评估题不是让你背 benchmark 名字，而是看你能否设计可信评估、识别评估陷阱、用实验支持决策，并把评估结果转成改进动作。

## 本章目标

学完本章，你要能回答：

1. 如何设计一个完整的大模型评估体系？
2. 如何设计 benchmark、golden set 和 regression suite？
3. Human eval、pairwise eval 和 LLM-as-a-judge 各有什么优缺点？
4. 如何避免 benchmark contamination 和评估污染？
5. 如何评估 reasoning、code、math、多模态、长上下文、安全和 RAG？
6. 如何做 A/B test、统计显著性和上线门禁？
7. 如何用 error analysis 指导模型、数据、prompt 和系统改进？

本章不是新概念堆叠，而是面试训练。

建议读法：每个问题先自己用 2 分钟回答，再看回答框架和追问点。

## 1. 评估题的通用答题框架

遇到任何评估题，先不要急着报指标。

推荐按七步回答：

```text
1. 明确评估目标
2. 定义任务和用户场景
3. 设计数据集和样本分层
4. 选择指标和评分方式
5. 控制污染、偏差和统计不确定性
6. 做 error analysis 和版本 diff
7. 把结果接入上线门禁和改进闭环
```

例如面试官问：

```text
你如何评估一个新的 chat model？
```

不要只答：

```text
跑 MMLU、GSM8K、HumanEval。
```

更好的回答是：

```text
我会先明确这个 chat model 的目标场景，例如通用助手、代码助手、企业知识库还是客服。然后构建覆盖核心任务和风险场景的 eval，包括公开 benchmark、内部 golden set、线上抽样和 regression suite。指标上同时看自动指标、pairwise preference、LLM judge、人工评估和系统指标。最后做分层分析、显著性检验、error analysis 和上线门禁，而不是只看一个平均分。
```

面试官真正想听的是：你知道评估是一个系统，而不是一个 leaderboard。

## 2. 问题 1：如何设计一个完整的大模型评估体系？

### 回答框架

完整评估体系可以分成五层。

第一层：能力评估。

1. 通用知识。
2. 指令遵循。
3. 数学推理。
4. 代码能力。
5. 多语言。
6. 长上下文。
7. 多轮对话。

第二层：行为评估。

1. Helpfulness。
2. Honesty。
3. Harmlessness。
4. Refusal quality。
5. 不确定性表达。
6. 格式稳定性。

第三层：系统评估。

1. 延迟。
2. 吞吐。
3. 成本。
4. 可用性。
5. 工具调用成功率。
6. RAG 检索和引用质量。

第四层：安全和鲁棒性评估。

1. Jailbreak。
2. Prompt injection。
3. 隐私泄露。
4. 偏见和有害内容。
5. 对抗扰动。

第五层：上线评估。

1. A/B test。
2. 用户满意度。
3. 任务完成率。
4. 事故率。
5. Regression suite。
6. 人工抽检。

### 标准回答

```text
我会把大模型评估设计成离线评估、人工评估和线上实验结合的体系。离线部分覆盖通用能力、领域能力、长上下文、代码、数学、安全和多模态等维度；人工部分用于判断开放式回答质量和用户偏好；线上部分用 A/B test 看真实用户任务完成率、满意度、延迟和成本。同时我会维护 golden set 和 regression suite，做分层指标、统计显著性、error analysis 和上线门禁，避免只用单一 benchmark 做决策。
```

### 面试官可能追问

1. 如果离线指标提升但线上用户满意度下降，你怎么办？
2. 如何选择哪些指标进入上线门禁？
3. 如何防止评估体系被模型或团队过拟合？

### 回答要点

离线指标和线上体验不一致时，优先做 error analysis 和日志抽样。

可能原因包括：

1. 离线集不代表真实流量。
2. 指标没有覆盖用户真实目标。
3. 新模型回答更长导致延迟增加。
4. Safety 策略导致 over-refusal。
5. 某些核心场景局部退化。
6. 线上 prompt、工具链或 RAG 版本和离线不同。

## 3. 问题 2：如何设计一个 benchmark？

### 回答框架

Benchmark 设计要回答六个问题。

第一，评估目标是什么。

例如：通用知识、数学推理、RAG 忠实性、长上下文定位、安全拒答。

第二，样本从哪里来。

可以来自：

1. 真实用户日志脱敏。
2. 专家构造。
3. 公开数据改写。
4. 合成数据。
5. 红队样本。
6. 历史失败样例。

第三，如何分层。

常见分层：

1. 任务类型。
2. 难度。
3. 领域。
4. 语言。
5. 输入长度。
6. 风险等级。
7. 用户意图。

第四，如何标注标准答案或评分规则。

第五，如何控制污染和泄漏。

第六，如何维护版本。

### 标准回答

```text
设计 benchmark 时，我会先定义它要评估的能力和使用场景，而不是先收集题目。然后按任务、难度、领域、语言、长度和风险等级分层采样，保证样本覆盖真实分布和关键边界场景。每个样本需要有清晰的输入、参考答案或评分 rubric、元数据标签和版本信息。最后要做污染检测、标注一致性检查、baseline 校准和定期更新，避免 benchmark 变成一次性跑分工具。
```

### 常见误区

1. 只追求样本数量，不看覆盖和质量。
2. 从公开 benchmark 直接复制，导致污染风险高。
3. 只有总分，没有分层指标。
4. 没有标注指南，人工一致性差。
5. benchmark 长期不更新，被模型和团队过拟合。

### 追问：Benchmark 和 regression suite 有什么区别？

回答：

```text
Benchmark 主要衡量模型在某类能力上的整体表现，关注代表性和可比较性。Regression suite 主要防止历史问题和关键场景退化，关注已知失败、高风险 case 和上线门禁。二者可以有重叠，但目标不同：benchmark 看能力水平，regression suite 看是否复发。
```

## 4. 问题 3：Human eval 怎么设计？

### 回答框架

Human eval 适合开放式任务。

例如：

1. 对话质量。
2. 写作。
3. 摘要。
4. 复杂问答。
5. 多轮交互。
6. 安全边界判断。

设计时要明确：

1. 标注对象：pointwise、pairwise 还是 ranking。
2. 标注维度：helpfulness、correctness、faithfulness、style、safety。
3. 标注指南：什么算好，什么算坏。
4. 盲评和随机化：避免模型名和顺序影响。
5. 一致性：多标注员和仲裁机制。
6. 质量控制：gold questions、抽检、标注时长过滤。

### 标准回答

```text
Human eval 要先定义清楚评分维度和标注指南。对于开放式回答，我通常更偏向 pairwise preference，因为让人比较两个回答哪个更好，比绝对打分更稳定。评估时要做 blind review、随机打乱顺序、多个标注员交叉标注，并计算一致性。对高风险任务，例如医疗、法律、安全，需要专家标注或人工复核。最后不能只报告胜率，还要做分层和 error analysis。
```

### 追问：为什么 pairwise eval 常比打分更稳定？

回答：

```text
绝对打分容易受标注员尺度影响，有的人偏严，有的人偏松。Pairwise eval 只要求比较两个回答哪个更好，判断更具体，一致性通常更高。但 pairwise 也有成本高、排序传递性不一定成立、对差异很小的回答不稳定等问题。
```

### 追问：Human eval 有哪些偏差？

1. 长度偏差：更长回答看起来更有帮助。
2. 位置偏差：左边或第一个回答被偏好。
3. 风格偏差：自信、流畅但错误的回答被偏好。
4. 标注员背景差异。
5. 任务理解不一致。
6. 疲劳效应。

## 5. 问题 4：LLM-as-a-judge 有什么问题？

### 回答框架

LLM-as-a-judge 是用模型评估模型输出。

优点：

1. 成本低。
2. 速度快。
3. 可扩展。
4. 能处理开放式文本。
5. 适合迭代中的初筛。

问题：

1. Judge 自身会错。
2. 可能偏好长回答。
3. 可能偏好格式好但事实错的回答。
4. 对数学、代码、专业领域不可靠。
5. 容易被 prompt 或模型输出诱导。
6. Judge 和被评模型同源时可能有偏差。
7. 不同 judge 版本结果不可直接比较。

### 标准回答

```text
LLM-as-a-judge 适合做低成本、大规模、开放式评估的辅助工具，但不能无条件当作真值。它可能有长度偏差、位置偏差、风格偏差和事实判断错误，尤其在数学、代码、安全、医疗法律等高风险任务中需要人工或可执行验证。实际使用时，我会用人工 gold set 校准 judge，固定 judge model 和 judge prompt，做 blind pairwise，要求 judge 给出理由，并定期检查它和人工标注的一致性。
```

### 追问：如何校准 LLM judge？

回答要点：

1. 构建人工标注 gold set。
2. 比较 judge 和人工一致性。
3. 分任务、分领域看一致性。
4. 调整 judge prompt 和 rubric。
5. 对不确定或高风险样本做人审。
6. 固定 judge 版本，避免评估漂移。

### 追问：什么时候不应该依赖 LLM judge？

1. 代码正确性可以运行测试时。
2. 数学答案可以符号或数值验证时。
3. 医疗、法律、安全等高风险判断。
4. 需要精确事实核查和引用验证时。
5. Judge 和被评模型可能共享同样错误时。

## 6. 问题 5：如何避免 benchmark contamination？

### 回答框架

Benchmark contamination 指评估集泄漏到训练、SFT、偏好数据、prompt 调优或模型选择过程中。

污染会导致分数虚高，无法判断真实泛化能力。

防控方法：

1. 数据层 exact match 和 near-duplicate 检测。
2. n-gram overlap 检测。
3. embedding similarity 检测。
4. 时间切分。
5. 私有 holdout。
6. 动态更新评估集。
7. 把开发集和最终测试集隔离。
8. 记录所有 eval 使用历史。

### 标准回答

```text
避免 contamination 需要从数据和流程两边控制。数据侧，我会对训练数据和评估集做 exact match、n-gram overlap、near-duplicate 和 embedding similarity 检测，尤其检查公开 benchmark、答案解析和 paraphrase。流程侧，我会区分 dev set、validation set 和 final holdout，限制最终测试集访问，记录每次使用历史，避免反复根据同一套 test set 调模型。对高价值评估，还会使用时间切分和私有动态 holdout。
```

### 追问：如果发现污染怎么办？

回答要点：

1. 不再把该分数作为可信结论。
2. 报告污染范围和检测方法。
3. 清理训练数据或剔除污染样本。
4. 使用未污染 holdout 重新评估。
5. 如果无法确认，降低结论确定性。

### 常见误区

1. 只查题目，不查答案和解析。
2. 只查完全重复，不查改写版本。
3. 只关注预训练，不关注 SFT、DPO、prompt tuning 和人工调参。
4. 用同一套 test set 反复调模型。

## 7. 问题 6：如何设计 reasoning eval？

### 回答框架

Reasoning eval 不能只看最终准确率。

要覆盖：

1. 问题理解。
2. 多步推理。
3. 组合泛化。
4. 反事实扰动。
5. 分布外题目。
6. 中间步骤可验证性。
7. 对提示词变化的鲁棒性。
8. Test-time compute 效果。

### 标准回答

```text
设计 reasoning eval 时，我会把任务按推理类型和难度分层，例如算术、符号推理、逻辑、规划、数学证明和多约束问题。指标上不仅看最终答案，还要看步骤一致性、可验证中间结果和错误类型。为了避免模型记忆 benchmark，我会构造新题、扰动题面、改变数字和实体、做反事实测试，并检查模型在不同 prompt、不同采样次数和 self-consistency 下的表现。对于可执行任务，优先用程序或 verifier 验证，而不是只靠 LLM judge。
```

### 追问：如何判断模型不是记住答案？

1. 改数字、实体和条件。
2. 换表达方式。
3. 构造同结构新题。
4. 用时间切分或私有题库。
5. 看中间步骤是否随条件正确变化。
6. 做反事实测试。

### 追问：CoT 该不该作为评分依据？

回答：

```text
要看任务。CoT 可以帮助错误分析，但不能完全依赖，因为模型可能写出看似合理但不真实的解释。对于数学和代码，最终答案、可执行验证和中间步骤一致性都重要。若使用 CoT 评分，需要明确 rubric，并避免奖励冗长但错误的推理。
```

## 8. 问题 7：如何评估 code model？

### 回答框架

代码模型评估要优先可执行。

维度包括：

1. 函数级代码生成。
2. 多文件修改。
3. Bug 修复。
4. 测试生成。
5. 代码解释。
6. 安全漏洞。
7. 依赖和 API 使用。
8. 编译、单元测试和隐藏测试。

### 标准回答

```text
评估 code model 时，我会优先使用可执行测试，而不是只看文本相似度。对于函数生成，可以看 pass@k、单元测试和隐藏测试；对于真实工程任务，要看多文件修改是否一致、是否通过现有测试、是否引入回归、是否满足 lint 和安全要求。还要做 error analysis，区分需求理解错误、API 使用错误、边界条件遗漏、复杂度问题和测试不足。
```

### 追问：pass@k 有什么问题？

回答要点：

1. 依赖测试覆盖率。
2. 可能过度奖励多次采样。
3. 不能反映代码可维护性。
4. 不能覆盖真实 repo 级修改。
5. 隐藏测试质量决定结论可信度。

### 追问：如何评估 coding agent？

需要额外看：

1. 任务分解。
2. 文件搜索。
3. 工具使用。
4. 补丁最小性。
5. 测试选择。
6. 回归风险。
7. 是否遵循仓库规范。

## 9. 问题 8：如何评估 RAG 系统？

### 回答框架

RAG 不能只评最终答案。

需要拆成四段：

```text
query understanding -> retrieval/reranking -> context construction -> grounded generation
```

指标包括：

1. Retrieval recall@k。
2. Precision@k。
3. MRR / nDCG。
4. Context precision。
5. Faithfulness。
6. Citation accuracy。
7. Unsupported claim rate。
8. Abstention accuracy。
9. End-to-end answer correctness。

### 标准回答

```text
评估 RAG 系统时，我会把检索、重排、上下文构造和生成拆开评估。首先看相关文档是否被召回，例如 recall@k、MRR、nDCG；再看进入 LLM 上下文的证据是否充分且无噪声；最后评估答案 correctness、faithfulness、citation accuracy 和 unsupported claim rate。对于资料不足的问题，还要评估模型是否能正确拒答。错误分析时要区分 retrieval miss、rerank error、context truncation、generation hallucination 和 citation error。
```

### 追问：RAG 答错了怎么定位？

对照实验：

1. 人工提供正确文档，只测生成。
2. 固定生成模型，替换 retriever。
3. 固定 retriever，替换 reranker。
4. 固定上下文，调整 prompt。
5. 检查引用是否支持答案。

如果人工提供正确文档后答对，主要问题可能在检索或上下文构造。

如果仍然答错，问题可能在生成、指令遵循或证据整合。

## 10. 问题 9：如何评估多模态 hallucination？

### 回答框架

多模态 hallucination 是模型对图像、视频、音频或文档做出没有依据的描述。

评估维度：

1. Object hallucination：说出不存在的物体。
2. Attribute hallucination：颜色、数量、材质错误。
3. Relation hallucination：空间关系或交互关系错误。
4. OCR hallucination：读错或编造文字。
5. Chart hallucination：图表数值和趋势错误。
6. Video temporal hallucination：动作顺序和事件因果错误。
7. Citation grounding：引用区域或证据不支持回答。

### 标准回答

```text
评估多模态 hallucination 时，我会把视觉内容拆成物体、属性、数量、空间关系、OCR、图表和时序事件等维度。数据上需要包含简单图片、复杂场景、小目标、干扰物、图表、截图和视频。评分时不能只看回答流畅度，而要检查每个 claim 是否被视觉证据支持。可以用人工标注、区域 grounding、问题模板和 LLM judge 辅助，但关键样例需要人工复核。
```

### 追问：如何构造测试集？

1. 有物体和无物体的成对样本。
2. 数量和颜色容易混淆的样本。
3. 空间关系反转样本。
4. OCR 中相似字符样本。
5. 图表中相近数值样本。
6. 视频中事件顺序变化样本。
7. 资料不足必须拒答的样本。

## 11. 问题 10：如何评估长上下文能力？

### 回答框架

长上下文能力不是“能塞多少 token”。

要区分三层：

1. 能放进去：context window 足够长。
2. 能看得见：模型能定位相关信息。
3. 能用得好：模型能综合、推理、引用和拒答。

评估任务：

1. Needle-in-a-haystack。
2. Lost-in-the-middle。
3. 多证据推理。
4. 长文档 QA。
5. 跨章节综合。
6. 长对话记忆。
7. 引用准确性。
8. 抗干扰和相似证据区分。

### 标准回答

```text
评估长上下文时，我不会只看最大 context length，而会测试模型能否定位、整合和忠实使用长文档中的证据。除了 needle-in-a-haystack，我会设计证据出现在开头、中间、结尾的样本，测试 lost-in-the-middle；还会设计多证据、多跳推理、相似干扰、引用准确性和资料不足拒答。指标上同时看 answer correctness、evidence recall、citation accuracy、unsupported claim rate，以及不同证据位置下的准确率曲线。
```

### 追问：Needle 测试有什么局限？

回答要点：

1. 太人工，和真实任务不同。
2. 只测定位，不测复杂推理。
3. 不一定覆盖多证据整合。
4. 模型可能适配 needle 格式。
5. 不能代表真实长文档理解能力。

## 12. 问题 11：如何评估 safety 和 robustness？

### 回答框架

Safety eval 要同时看漏拒和误拒。

维度包括：

1. 有害内容。
2. 自伤。
3. 医疗法律金融高风险建议。
4. 隐私泄露。
5. Jailbreak。
6. Prompt injection。
7. 偏见和歧视。
8. Over-refusal。
9. 正常请求的 helpfulness。

Robustness eval 看扰动稳定性：

1. 同义改写。
2. 拼写错误。
3. 多轮诱导。
4. 格式变化。
5. 干扰上下文。
6. 跨语言攻击。

### 标准回答

```text
Safety eval 不能只看模型拒绝了多少危险请求，还要同时看正常请求是否被过度拒答。我会建立风险 taxonomy，把样本分为允许、边界和禁止类别，分别评估正确拒答率、漏拒率、误拒率和安全替代建议质量。对 jailbreak、prompt injection 和多轮诱导要做红队测试。Robustness 方面，我会对同一意图做改写、翻译、拼写扰动和上下文干扰，检查模型行为是否稳定。
```

### 追问：Safety 和 helpfulness 冲突怎么办？

回答：

```text
需要按风险等级分层，而不是用单一拒答策略。危险请求要拒绝并给安全替代，正常请求要保持有帮助，边界请求要澄清意图或提供低风险信息。评估上同时报告 harmful compliance、over-refusal 和 helpfulness，不能只优化一个指标。
```

## 13. 问题 12：如何做在线 A/B test？

### 回答框架

A/B test 用于验证真实用户环境中的效果。

关键步骤：

1. 定义实验假设。
2. 选择主指标和护栏指标。
3. 随机分流。
4. 控制实验单位。
5. 估算样本量和实验周期。
6. 监控异常和安全事件。
7. 做统计显著性分析。
8. 决定上线、灰度、回滚或继续实验。

主指标可以是：

1. 任务完成率。
2. 用户满意度。
3. 采纳率。
4. 留存。
5. Pairwise win rate。

护栏指标可以是：

1. 延迟。
2. 成本。
3. 错误率。
4. 投诉率。
5. 安全事件。
6. 过度拒答率。

### 标准回答

```text
做 LLM A/B test 时，我会先明确实验假设，例如新模型是否提升任务完成率。然后定义一个主指标和多个护栏指标，例如满意度、延迟、成本和安全投诉。分流要随机且稳定，避免同一用户在不同版本间频繁切换。实验前估算样本量，实验中监控异常，实验后看统计显著性和分层结果。即使主指标显著提升，如果安全、延迟或成本护栏明显变差，也不应直接全量上线。
```

### 追问：A/B test 有哪些坑？

1. 样本量不足。
2. 提前偷看结果并停止实验。
3. 用户分流不稳定。
4. 指标太多导致 false positive。
5. 新奇效应。
6. 干扰效应。
7. 只看平均值，不看高风险分层。
8. 忽略延迟和成本。

## 14. 问题 13：如何判断实验结果是否显著？

### 回答框架

统计显著性回答要通俗，不要只背 p-value。

需要说明：

1. 观测差异可能来自真实提升，也可能来自随机波动。
2. 显著性检验用于判断随机波动解释该差异的可能性。
3. 置信区间比单个 p-value 更有信息。
4. 显著不等于业务重要。
5. 多重比较会增加误报。

### 标准回答

```text
我会先看效果大小和置信区间，而不是只看 p-value。比如新模型 win rate 是 52%，要看样本量是否足够、置信区间是否跨过 50%、不同用户和任务分层是否一致。统计显著只说明差异不太像随机波动，并不代表业务上值得上线。还要看效果大小、成本、延迟、安全护栏和回归样例。
```

### 追问：为什么显著不等于重要？

回答：

```text
样本量很大时，很小的差异也可能统计显著，但业务收益可能不足以覆盖成本或风险。比如满意度提升 0.1%，但推理成本增加 30%，就未必值得上线。
```

### 追问：为什么不显著也不等于没效果？

回答：

```text
不显著可能是样本量不足、指标噪声大、效果集中在某些分层或实验周期太短。需要看置信区间和 power，而不是简单说没效果。
```

## 15. 问题 14：如何做 error analysis？

### 回答框架

Error analysis 的目标是把分数转成可行动结论。

流程：

1. 保存完整日志。
2. 收集失败样例。
3. 定义错误 taxonomy。
4. 人工或 LLM 标注错误类型。
5. 按任务、领域、难度、风险聚合。
6. 做失败聚类。
7. 验证 root cause。
8. 设计修复方案。
9. 加入 regression suite。

### 标准回答

```text
我会先保存每个样本的输入、输出、参考答案、judge 解释、模型版本、prompt 版本、检索和工具调用日志。然后对失败样例定义错误 taxonomy，例如 factuality、reasoning、format、safety、retrieval miss、citation error、tool error。接着按任务、领域、难度和严重程度做分层统计，找高频高风险错误簇。对 root cause 不凭直觉下结论，而是通过对照实验验证。最后把重要失败样例沉淀到 regression suite，并把分析结果转成数据、prompt、检索、工具链或训练改进。
```

### 追问：失败样例能不能加入训练？

回答：

```text
可以作为修复数据，但不能用同一批样例再证明模型变强，否则会造成评估污染。更稳妥做法是把历史失败样例用于训练或 prompt 改进，同时构造未见过的等价 holdout 样例验证泛化。
```

## 16. 问题 15：如何设计上线门禁？

### 回答框架

上线门禁要同时看主指标和护栏指标。

示例规则：

```text
上线条件：
1. 核心 golden set 不低于当前线上版本。
2. 主任务指标提升或持平。
3. 高严重度 regression 为 0。
4. Safety regression 为 0。
5. 延迟和成本不超过预算。
6. JSON/schema 等关键格式通过率达标。
7. 人工审核 P0/P1 case 通过。
8. 灰度期线上投诉和错误率不升高。
```

### 标准回答

```text
我不会只用一个总分决定上线。上线门禁应该包含核心能力 golden set、历史 regression suite、安全红队、系统指标和线上灰度指标。对于高风险业务，即使总体指标提升，只要出现严重安全回归、关键客户场景失败或成本延迟超标，也应该阻断上线或继续灰度。
```

### 追问：如果新模型总体更强但安全略差怎么办？

回答要点：

1. 看安全下降的严重程度和场景。
2. 区分轻微风格问题和高风险 harmful compliance。
3. 高严重度安全回归应阻断上线。
4. 可以通过 safety tuning、策略层、分类器或灰度限制修复。
5. 修复后重新跑 regression suite 和红队。

## 17. 综合系统设计题：设计一个 LLM Eval Platform

### 题目

```text
请设计一个大模型评估平台，支持模型版本、prompt 版本、RAG 系统和 Agent 工具链的离线评估、人工评估、回归测试和上线门禁。
```

### 回答框架

可以从六层回答。

第一，数据层。

1. Benchmark 管理。
2. Golden set。
3. Regression suite。
4. 线上抽样。
5. 标注数据。
6. 元数据标签和版本管理。

第二，执行层。

1. Batch inference。
2. 多模型、多 prompt 运行。
3. 固定 decoding 参数。
4. RAG 和 tool trace 记录。
5. 重试和失败恢复。

第三，评分层。

1. Exact match。
2. Unit test。
3. Rule-based metric。
4. LLM judge。
5. Human eval。
6. Safety classifier。

第四，分析层。

1. 聚合指标。
2. 分层指标。
3. 置信区间。
4. 样本级 diff。
5. Error clustering。
6. Regression detection。

第五，报告和门禁层。

1. Eval report。
2. Dashboard。
3. 上线门禁规则。
4. 人工审核工作流。
5. 灰度和回滚建议。

第六，治理层。

1. 权限控制。
2. 数据脱敏。
3. 防污染。
4. 审计日志。
5. 评估集生命周期管理。

### 标准回答

```text
我会把 eval platform 设计成数据管理、评估执行、评分、分析和上线门禁五个核心模块。数据管理负责 benchmark、golden set、regression suite 和线上抽样，并做版本化和污染控制。评估执行负责对不同模型、prompt、RAG 索引和工具版本进行批量推理，并记录完整 trace。评分模块支持 rule、unit test、LLM judge、human eval 和 safety classifier。分析模块输出总体指标、分层指标、置信区间、样本级 diff、error clusters 和 regression case。最后通过门禁规则决定是否上线、灰度或回滚，并把失败样例沉淀回 regression suite。
```

### 加分点

1. 强调 trace 记录：RAG 文档、tool call、prompt、decoding 参数都要记录。
2. 强调版本管理：模型、数据、prompt、judge、eval code 都要版本化。
3. 强调人工闭环：LLM judge 不能替代高风险人工审核。
4. 强调防污染：训练、调参和最终 holdout 要隔离。
5. 强调上线门禁：评估平台最终服务决策，而不是只生成图表。

## 18. 常见面试失分点

### 18.1 只报 benchmark 名字

只说 MMLU、GSM8K、HumanEval 不够。

要讲为什么选、覆盖什么能力、有什么局限、如何防污染。

### 18.2 只看平均分

平均分可能掩盖局部退化。

面试中要主动提分层分析、样本级 diff 和 regression suite。

### 18.3 把 LLM judge 当真值

LLM judge 是辅助工具，需要校准和人工复核。

### 18.4 不谈统计不确定性

小样本差异可能是噪声。

要讲置信区间、显著性、样本量和多重比较。

### 18.5 不谈业务和上线

评估最终要支持决策。

要讲主指标、护栏指标、上线门禁和灰度。

### 18.6 不谈 error analysis

评估不是为了得到分数，而是为了知道下一步怎么改。

## 19. 评估面试 10 个万能句式

1. “我不会只看单一 benchmark，而会结合离线、人工和线上评估。”
2. “这个指标需要分层看，否则平均分可能掩盖关键场景退化。”
3. “LLM-as-a-judge 可以辅助评估，但需要用人工 gold set 校准。”
4. “如果是可执行任务，我会优先用 unit test 或 verifier，而不是只靠文本评分。”
5. “评估集需要做污染检测，并区分 dev set 和 final holdout。”
6. “我会保存完整 trace，包括 prompt、模型版本、检索结果、工具调用和 decoding 参数。”
7. “上线决策需要主指标和护栏指标共同满足。”
8. “对于新版本，我会做样本级 diff，重点看 old correct, new wrong。”
9. “失败样例可以用于修复，但最终要用未见过的 holdout 验证泛化。”
10. “评估闭环的终点不是报告，而是改数据、改模型、改 prompt、改系统或阻断上线。”

## 22. 本章总结

评估面试题的核心不是背 benchmark 名字，而是展示你是否具备实验设计和系统判断能力。

一个成熟回答通常包括：目标、数据、指标、评分、污染控制、统计显著性、error analysis、regression suite 和上线门禁。

Human eval、LLM-as-a-judge、自动指标和线上 A/B test 各有适用边界，不能互相简单替代。

Reasoning、code、RAG、多模态、长上下文和 safety 都需要专项评估，不能用一个总分概括。

面试中最有价值的表达是：评估不是跑分，而是把模型变化变成可信证据，并指导训练、数据、prompt、系统和上线决策。
