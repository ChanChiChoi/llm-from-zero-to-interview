# 第一章：评估总览

核心问题：如何判断一个大模型是否真的更强、更可靠、更安全。

重点：offline eval、online eval、human eval、automatic eval、safety eval、regression eval。

## 本章目标

学完本章，你要能回答四个问题：

1. 为什么大模型评估比传统模型评估更难？
2. 如何区分 offline eval、online eval、human eval、automatic eval、safety eval 和 regression eval？
3. 如何判断一个模型版本是否可以上线？
4. 面试中如何系统回答“你怎么知道模型真的变好了”？

本章是第七册的总览。

后续章节会分别展开 benchmark 设计、人工评测、pairwise eval、LLM-as-a-Judge、污染检测、reasoning/code/math eval、多模态评估、安全评估、A/B 测试、统计显著性和 error analysis。

## 本章资料边界

本章第二轮精修参考了 OpenAI Evals 文档与开源项目、Stanford HELM / Holistic Evaluation of Language Models 论文、Hugging Face Evaluate 文档、EleutherAI lm-evaluation-harness 项目，以及前面训练、部署、RAG / Agent、安全和系统设计章节的资料边界。

本章聚焦评估体系的总览口径：为什么评估困难、离线/在线/人工/自动/安全/回归评估如何分工、指标如何服务上线决策、如何做切片分析、统计置信和版本管理。不展开具体 benchmark 全量题库、LLM-as-a-Judge prompt 细节、A/B 测试完整统计推导或某个评测平台的全部 API。

## 0. 主流模型发布常见评测集速查表

很多模型发布报告会列出一长串 benchmark 分数，例如 MMLU、GPQA、AIME、SWE-bench、WebArena、OSWorld、MMMU、LongBench、SimpleQA 等。

初学者容易把这些名字当成排行榜背诵。更好的读法是先问四个问题：

1. 这个 benchmark 的样本是什么。
2. 它主要考察模型的哪类能力。
3. 它的评分方式和规模是什么。
4. 它不能代表什么。

下面的表按模型发布报告常见维度组织。数据规模会随版本、公开 split、私有 split 和 leaderboard 维护方式变化，表中数字按公开论文或项目资料给近似口径；如果用于正式报告，必须再次核对具体版本。

本节样例均为仿写样例，不是 benchmark 原题，目的是说明题型，避免传播真实测试题导致污染。

### 0.1 通用知识、指令遵循和偏好评估

| Benchmark | 公开规模 | 主要考察 | 仿写样例 | 适用边界 |
|---|---:|---|---|---|
| [MMLU](https://arxiv.org/abs/2009.03300) | 约 15.9k 道多选题，57 个学科 | 大学到专业考试层面的知识覆盖、基础推理和多学科广度 | 给出一道专业医学或法律多选题，要求从 4 个选项中选择正确答案 | 已接近饱和，且有题目错误和污染风险；不能单独代表真实助手能力 |
| [MMLU-Pro](https://arxiv.org/abs/2406.01574) | 约 12k 道题，10 个选项，14 个领域 | 更难的多学科知识和 reasoning，比 MMLU 更能区分强模型 | 给出一道需要多步推理的物理题，从 10 个相近选项中选择答案 | 仍是考试式闭集评估，不能覆盖开放式写作、工具使用或长链任务 |
| [GPQA](https://arxiv.org/abs/2311.12022) | 448 道专家编写题；常见 GPQA Diamond 子集约 198 道 | 研究生级、Google-proof 的科学问答，尤其是生物、物理、化学 | 给出一段实验条件，问哪个分子机制最可能解释观测现象 | 样本少，方差高；适合看高难科学推理，不适合评价日常产品体验 |
| [IFEval](https://arxiv.org/abs/2311.07911) | 541 个 prompt，25 类可验证指令 | 指令遵循，尤其是格式、长度、包含/排除条件和多约束 | 要求回答恰好 3 条，每条以指定词开头，且不能出现某个词 | 主要考格式和显式约束，不代表事实性、深度推理或工具执行 |
| HELM | 多场景、多指标框架，不是单一固定题库 | holistic evaluation，把准确率、鲁棒性、公平性、校准、效率等放在一起比较 | 同一模型在问答、摘要、安全、鲁棒性和效率上分别打分 | 更像评估框架；分数解释依赖场景、任务和 metric 定义 |
| MT-Bench / Chatbot Arena | MT-Bench 约 80 个多轮问题；Arena 是持续收集的人类偏好投票 | 开放式聊天质量、人类偏好、多轮回答能力 | 两个模型分别回答“帮我设计一周学习计划”，人类比较哪一个更好 | 偏好会受长度、风格、用户群体和时间影响；不是能力因果解释 |
| Arena-Hard | 约 500 个较难开放式 prompt | 比普通聊天题更难的开放式助手能力，常用 LLM judge 评分 | 要求模型综合比较两种系统设计方案，并给出取舍 | 依赖 judge 和 prompt 设计；适合辅助比较，不应作为唯一发布门禁 |

面试表达：MMLU / MMLU-Pro / GPQA 更像“知识和闭集推理体检”，IFEval 更像“指令遵循体检”，Arena 类评估更像“人类偏好和开放式回答体检”。它们互补，但没有一个能单独证明模型全面更强。

### 0.2 数学、科学和高难 reasoning

| Benchmark | 公开规模 | 主要考察 | 仿写样例 | 适用边界 |
|---|---:|---|---|---|
| GSM8K | 约 8.5k 道小学数学文字题 | 多步算术、自然语言题意理解、答案抽取 | 商店先打折再加税，问最终价格是多少 | 难度偏基础，强模型容易饱和；不能代表竞赛数学 |
| [MATH](https://arxiv.org/abs/2103.03874) | 约 12.5k 道竞赛数学题，5 个难度 | 代数、数论、几何、组合等竞赛数学推理 | 给出一个组合计数条件，要求求满足条件的方案数 | 最终答案准确率不代表过程正确；公开题有污染风险 |
| AIME | 每年 AIME I/II 合计约 30 道竞赛题；发布报告常按年份集合评估 | 高中竞赛数学、长链推理、答案格式控制 | 求一个几何构型中某个整数面积或余数 | 样本很小，年度差异大；适合看 frontier reasoning，但统计不稳定 |
| [Humanity's Last Exam / HLE](https://arxiv.org/abs/2501.14249) | 约 3k 道专家级、多学科、含多模态题 | benchmark 饱和后的专家知识和高难闭集推理 | 给出一个冷门数学定理条件，问唯一可验证结论 | 题目难、样本维护成本高；要关注答案质量、私有 split 和版本变化 |
| BIG-bench / BBH | BIG-bench 约 200+ 任务；BBH 选取更难子集 | 多任务推理、符号规则、反常识和复杂指令 | 根据一组人工规则推断最终状态 | 任务混杂，prompt 敏感；适合做能力雷达，不适合只看平均分 |

面试表达：数学和 reasoning 分数必须同时报告解码设置、采样次数、是否使用 verifier、是否使用工具、是否做 self-consistency，以及单位成本正确率。否则一次 greedy 和 64 次采样 rerank 的分数不可直接比较。

### 0.3 代码生成和真实软件工程

| Benchmark | 公开规模 | 主要考察 | 仿写样例 | 适用边界 |
|---|---:|---|---|---|
| [HumanEval](https://arxiv.org/abs/2107.03374) | 164 道 Python 函数生成题 | 函数级代码生成、单元测试通过率、pass@k | 根据 docstring 写一个函数，判断列表是否满足某种排序条件 | 小而公开，容易污染；不能代表 repo 级工程能力 |
| MBPP | 约 974 道 Python 编程题 | 初级函数实现、算法和 API 使用 | 写函数把字符串列表按某种规则过滤和排序 | 难度偏基础；测试覆盖有限 |
| [APPS](https://arxiv.org/abs/2105.09938) | 约 10k 道竞赛编程题 | 输入输出式编程、算法、复杂度和隐藏测试 | 读取多组输入，输出满足约束的最优值 | 更像竞赛编程，不等于真实代码库维护 |
| LiveCodeBench | 持续更新，按时间收集新题 | 降低污染风险的代码生成、执行、修复和测试 | 给出新近竞赛题，要求生成可运行解法 | 版本变化快；正式比较必须固定日期、split 和环境 |
| [SWE-bench](https://arxiv.org/abs/2310.06770) | 约 2.3k 个真实 GitHub issue；Verified 常见 500 个样本 | repo 级 bug 修复、阅读代码、修改文件、运行测试 | 给定一个开源库 issue，修改相关文件让回归测试通过 | 依赖历史 GitHub 数据，存在污染和测试覆盖风险 |
| [SWE-Lancer](https://arxiv.org/abs/2502.12115) | 1488 个 Upwork 软件工程任务，总计约 100 万美元任务价值 | 更接近真实 freelance 软件工程，包括独立实现和技术方案选择 | 给定客户需求和现有项目，完成一个付费功能或选择更合理的实现方案 | 任务商业化语境更强，但环境复杂，复现和授权边界要看具体 split |
| [DeepSWE](https://arxiv.org/abs/2607.07946) | 113 个原创 long-horizon 任务，覆盖 91 个活跃开源仓库和 5 种语言 | 长周期 coding agent、原创任务、手写 verifier、抗污染软件工程评估 | 给定一个仓库和新需求，跨多个文件实现功能，并通过功能 verifier | 规模小但难度高；适合 frontier coding agent，不适合日常函数生成 |
| Aider Polyglot Benchmark | 常见公开口径为 225 个 Exercism 多语言代码任务 | 多语言代码修改、模型作为 coding assistant 的编辑能力 | 在现有测试下修改 Rust 或 TypeScript 项目中的一处逻辑 | 工具链、prompt 和编辑器 scaffold 会影响分数 |
| KernelBench | 常见公开口径为 250 个 PyTorch 到 CUDA kernel 任务 | 低层性能代码生成、正确性和加速比 | 把一个张量操作改写为自定义 CUDA kernel，并验证输出误差 | 主要考性能工程和硬件相关代码，不能代表通用软件开发 |

面试表达：HumanEval / MBPP 看函数级生成，APPS / LiveCodeBench 看竞赛式可执行编程，SWE-bench / SWE-Lancer / DeepSWE 看真实软件工程和 agentic coding。代码评估一定要区分 pass@1、pass@k、隐藏测试、环境复现、修改范围、测试覆盖和污染风险。

### 0.4 Agent、浏览器、计算机使用和工具调用

| Benchmark | 公开规模 | 主要考察 | 仿写样例 | 适用边界 |
|---|---:|---|---|---|
| [WebArena](https://arxiv.org/abs/2307.13854) | 812 个网页任务，运行在可控网站环境 | Browser Agent 的网页导航、表单填写、信息查找和任务完成 | 在电商网站中找到符合条件的商品并加入购物车 | 任务环境可控，不等于真实互联网；需要固定初始状态和验收脚本 |
| OSWorld | 369 个真实桌面环境任务 | Computer-use agent 的 GUI 操作、文件、应用和多步执行 | 打开表格文件，筛选数据，保存为指定格式 | GUI 状态、系统版本和权限会影响复现 |
| AgentBench | 8 类环境，覆盖网页、数据库、操作系统、游戏等 | 通用 Agent 在多环境中的规划、动作和反馈利用 | 根据数据库 schema 查询满足条件的记录 | 环境跨度大，平均分解释要看各子任务 |
| GAIA | 466 个真实助手问题，分 3 个难度 | 工具使用、网页检索、多步推理和真实助理任务 | 找到某活动的具体日期并计算和另一个事件相隔几天 | 依赖外部信息和工具链，时间变化会影响答案 |
| tau-bench | 常见 retail / airline 等交互式任务域，任务数随版本维护 | Tool-agent-user interaction，多轮工具调用、状态更新和策略遵循 | 用户要改签航班，Agent 需要查政策、调用工具并确认结果 | 强依赖模拟器和用户模型质量；不能只看最终文本 |
| ToolBench | 约万级工具调用指令和大量 API | 工具选择、参数生成、工具链规划和执行 | 用户要查询天气并安排日程，模型选择天气和日历工具 | API 模拟器稳定性、工具描述质量和安全策略会影响结论 |
| [BrowseComp](https://arxiv.org/abs/2504.12516) | 1266 个需要浏览网页的短答案问题 | 浏览器 agent 的持久搜索、网页导航和 hard-to-find 信息检索 | 找到某个小众比赛某年冠军的出生地 | 只要求短答案，不覆盖长报告写作、歧义澄清或真实用户意图分解 |

面试表达：Agent 评估不能只看最后回答。WebArena / OSWorld / GAIA / tau-bench / BrowseComp 这类评估必须记录 action trace、工具版本、权限、环境初始状态、observation 使用、失败恢复、成本和最终总结忠实性。

### 0.5 长上下文、RAG 和事实性

| Benchmark | 公开规模 | 主要考察 | 仿写样例 | 适用边界 |
|---|---:|---|---|---|
| LongBench | 约 4.75k 个样本，21 个数据集，6 类任务，中英双语 | 长文档问答、摘要、多文档推理、代码和少样本学习 | 给出多篇长文档，问其中两个证据共同推出的结论 | 平均长度和任务类型固定；不能只代表百万 token 能力 |
| Lost in the Middle | 典型设置为多文档 QA 或 key-value retrieval，改变证据位置 | 长上下文中证据位置敏感性，尤其是中间位置遗失 | 把关键事实放在上下文中间，问模型能否准确取回 | 是现象测试，不是完整长上下文能力评估 |
| Needle-in-a-Haystack | 可合成任意长度，常按位置和长度网格测试 | 长上下文检索和位置鲁棒性 | 在很长文本里插入一句唯一事实，再询问该事实 | 过于简单，容易被专门优化；不能代表多证据推理 |
| [RULER](https://arxiv.org/abs/2404.06654) | 13 个代表任务，可配置长度和复杂度 | 比 needle 更系统的检索、多针、多跳、聚合和 QA | 在长上下文中找到多个线索并按规则聚合结果 | 合成任务为主；要和真实长文档任务一起看 |
| [FRAMES](https://arxiv.org/abs/2409.12941) | 约 824 个多跳问题 | RAG 里的事实性、检索、跨来源推理和答案合成 | 需要从多个来源找事实，再组合回答一个具体数值问题 | 适合测 end-to-end RAG，不等于单独检索器能力 |
| [SimpleQA](https://arxiv.org/abs/2411.04368) | 约 4.3k 个短事实问题 | 短事实回答、校准、知道自己不知道 | 问某个明确人物的出生年份，答案唯一且可核验 | 只测短事实，不测长解释；题目静态会随时间过期 |
| TruthfulQA | 817 个容易诱发常见误解的问题 | 诚实性、抗常见错误和拒绝编造 | 问一个民间误解是否真实，模型需要纠正而不是迎合 | 覆盖面有限；不能替代领域事实核验 |

面试表达：长上下文评估要区分“能放进窗口”和“能稳定使用证据”。RAG 评估要区分 retrieval recall、rerank、context construction、faithfulness、citation accuracy 和最终任务成功率。

### 0.6 安全、偏见和鲁棒性

| Benchmark | 公开规模 | 主要考察 | 仿写样例 | 适用边界 |
|---|---:|---|---|---|
| BBQ | 约 58k 个偏见问答样本 | 社会偏见、歧义场景和 stereotype sensitivity | 给出一个歧义故事，问哪个人物更可能从事某行为 | 主要测特定偏见维度，不代表完整 safety |
| HarmBench | 数百个有害行为类别和测试样本，随版本维护 | 有害请求拒答、红队覆盖和安全策略一致性 | 抽象地测试模型是否拒绝高风险违法帮助请求 | 只能讨论防御性评估，不应传播可复用攻击步骤 |
| XSTest | 约 450 个安全过拒/误拒测试样本 | harmless prompt 下的过度拒答和策略边界 | 用户问一个安全的医学科普问题，模型不应误拒 | 主要测 over-refusal，不能代表危险能力 |
| RealToxicityPrompts | 约 100k 个 prompt | 毒性续写风险、内容安全和分布鲁棒性 | 给出一句可能引向攻击性内容的开头，观察续写是否安全 | 续写毒性不等于完整对话安全；还要看多轮和工具场景 |

面试表达：安全评估不能只看 unsafe compliance，也要看 over-refusal、用户可用性、策略一致性、地区/语言切片、工具越权、RAG 注入和上线监控。

### 0.7 多模态理解、文档、图表和视频

| Benchmark | 公开规模 | 主要考察 | 仿写样例 | 适用边界 |
|---|---:|---|---|---|
| VQA / TextVQA | VQA 约 20 万级图像问答；TextVQA 约 45k 问题 | 图像问答、OCR 文本读取和视觉常识 | 给一张路牌图片，问牌子上写的限速是多少 | 简短问答为主，不代表复杂视觉推理 |
| DocVQA | 约 50k 个文档图像问题 | 文档 OCR、表单理解、版面和字段抽取 | 给一张发票，问发票编号或总金额 | 文档质量和 OCR 难度强相关；要结合真实业务文档 |
| ChartQA | 约 32k 个图表问答样本 | 图表读数、计算、比较和数据推理 | 给柱状图，问 2020 到 2023 的增长百分比 | 图表类型有限；真实图表常有噪声、截图和单位问题 |
| MMMU | 约 11.5k 个多学科多模态题，30 个学科 | 专家级图像理解、学科知识和多模态推理 | 给出化学结构图或电路图，问哪个结论正确 | 强模型仍可能利用文本线索；要注意是否必须看图 |
| MathVista | 约 6.1k 个视觉数学题 | 图像、图形、表格与数学推理结合 | 给几何图和边长标注，求角度或面积 | 偏视觉数学，不代表通用视觉问答 |
| OCRBench | 约 1k 个 OCR 相关评测样本 | 文字识别、场景文字、文档和关键信息抽取 | 给一张含小字截图，问其中的订单号 | OCR 只是多模态基础能力之一，不代表完整视觉推理 |
| Video-MME | 900 个视频、2700 个问题 | 长短视频理解、时序推理、音视频信息整合 | 看一段教学视频，问某个动作发生在什么步骤之后 | 视频评估成本高；帧采样、字幕、音频和时长都会影响结果 |

面试表达：多模态 benchmark 要按输入形态拆开看。OCR、文档、图表、静态图像、视频、语音和跨模态推理不是一个能力。模型发布报告里的多模态平均分必须配合样例、错误类型和输入质量切片解释。

### 0.8 读模型发布报告时的判断清单

看到一个模型发布报告时，不要只看分数最高的表。建议按下面顺序审计：

1. 是否覆盖通用知识、数学、代码、Agent、长上下文、事实性、安全和多模态。
2. 是否报告 benchmark 版本、日期、公开/私有 split、prompt、采样参数和工具条件。
3. 是否区分 zero-shot、few-shot、CoT、self-consistency、search、verifier、tool-use 和 agent scaffold。
4. 是否报告污染检测、题目去重、时间切分和私有动态集。
5. 是否给出关键切片，而不是只给平均分。
6. 是否给出成本、延迟和单位成本正确率。
7. 是否列出失败案例和限制，而不是只列最好成绩。

资深面试表达：

```text
我不会把模型发布报告里的 benchmark 表当成绝对排名。MMLU、GPQA、AIME、SWE-bench、BrowseComp、SimpleQA、MMMU 这些分数分别覆盖不同能力面，而且它们的样本规模、污染风险、评分方式和工具条件都不同。可信比较必须固定 benchmark 版本、prompt、采样预算、工具条件和统计口径，再做切片和 error analysis。否则分数差异可能来自污染、scaffold、采样次数、judge 偏差或统计噪声，而不一定是模型本身更强。
```

### 0.9 Radar：专业 eval 新簇和自动化对抗评估

Frontier model release 还会不断出现新的专业 benchmark。它们通常不是为了替代 MMLU、SWE-bench 或 MMMU，而是补某个高价值或高风险切片。

近期值得放入观察项的新簇包括：

| 方向 | 例子 | 主要想测什么 | 进入正文主干前要确认什么 |
|---|---|---|---|
| 长周期 Agent | Agents' Last Exam、AutomationBench、Terminal-Bench 2.1 | 长任务、终端操作、自动化流程、真实任务完成率 | 环境是否可复现，验收脚本是否稳定，是否区分工具 scaffold 和模型能力 |
| Cyber 安全 | SEC-Bench Pro、ExploitBench、ExploitGym | 攻防推理、漏洞理解、利用链分析和安全边界 | 是否只用于防御性评估，是否避免传播可复用攻击步骤，是否有权限和隔离 |
| 生物 / 医学 / 化学 | GeneBench-Pro、LifeSciBench、MedChemBench | 专业科学推理、实验设计、药化或生物知识 | 题目来源、专家标注、风险分级、是否需要 trusted access |
| 自动化对抗评估 | GOAT 类自动 red teaming | 多轮、动态、自动化对抗测试和安全回归 | 是否有固定策略版本、攻击面分类、人工复核和防御性报告边界 |

这些 eval 的价值在于暴露旧 benchmark 看不到的能力和风险。它们的风险是定义不稳定、样本少、环境复杂、容易被 leaderboard 化，或者因为高风险领域而不适合公开传播细节。

面试表达：

```text
我会把新专业 benchmark 当作能力切片雷达，而不是马上当成通用排行榜。比如 cyber、bio、terminal agent、automation agent、long-running coding agent 这些评估，真正重要的是任务环境、验收器、权限、安全边界、复现性和人工复核。尤其是自动化对抗评估，只能从防御性分类、策略回归和发布门禁角度讨论，不能把攻击细节扩散成训练材料。
```

## 本章核心公式

评估集可以抽象成：

```math
\mathcal{D}_{\mathrm{eval}}
=
\{(x_i,y_i,z_i)\}_{i=1}^{N}
```

其中 `x_i` 是输入，`y_i` 是参考答案或期望行为，`z_i` 是元数据，例如任务类型、难度、语言、业务线、安全类别、是否来自历史 bad case。

单样本打分可以写成：

```math
s_i = m(\hat{y}_i,y_i,z_i),
\qquad
\hat{y}_i = f_\theta(x_i)
```

这里 `m` 是 metric 或 judge，`\hat{y}_i` 是模型输出。对开放式任务，`s_i` 可以来自人工评分、pairwise preference、LLM-as-a-Judge 或规则组合。

整体平均分：

```math
S=
\frac{1}{N}
\sum_{i=1}^{N}s_i
```

如果不同任务重要性不同，可以用加权平均：

```math
S_{\mathrm{w}}
=
\frac{\sum_i w_i s_i}{\sum_i w_i}
```

其中 `w_i` 是样本权重。生产评估里，高价值业务样本、P0/P1 bad case 和安全样本通常不能和普通样本完全等权。

切片分数：

```math
S_g=
\frac{1}{|\mathcal{D}_g|}
\sum_{i\in \mathcal{D}_g}s_i
```

其中 `D_g` 是某个切片，例如中文客服、长上下文、代码、数学、安全高风险样本。平均分提升但关键切片下降，通常不能直接上线。

新旧版本差异：

```math
\Delta S = S_{\mathrm{new}}-S_{\mathrm{old}}
```

回归率可以写成：

```math
R_{\mathrm{reg}}
=
\frac{1}{N}
\sum_{i=1}^{N}
I(s_{i,\mathrm{new}}+\epsilon<s_{i,\mathrm{old}})
```

其中 `epsilon` 是容忍阈值，避免把很小的噪声波动误判为回归。

Pairwise win rate：

```math
W=
\frac{N_{\mathrm{win}}+0.5N_{\mathrm{tie}}}{N_{\mathrm{pair}}}
```

这里 `N_win` 是新版本胜过旧版本的样本数，`N_tie` 是平局数。很多开放式回答难以绝对打分，pairwise 比较更稳定。

安全评估要同时看攻击成功率和误拒率：

```math
R_{\mathrm{attack}}
=
\frac{N_{\mathrm{unsafe\_pass}}}{N_{\mathrm{unsafe}}},
\qquad
R_{\mathrm{over}}
=
\frac{N_{\mathrm{benign\_reject}}}{N_{\mathrm{benign}}}
```

`R_attack` 越低越好，`R_over` 也不能过高，否则模型会过度拒答。

在线比例指标的标准误可以粗略写成：

```math
\mathrm{SE}(\hat{p})
=
\sqrt{\frac{\hat{p}(1-\hat{p})}{n}}
```

其中 `p_hat` 是样本比例，例如任务成功率、点赞率、转人工率，`n` 是样本数。线上提升如果远小于统计噪声，不能轻易说版本更好。

上线门禁可以概括为：

```math
G_{\mathrm{eval}}
=
G_{\mathrm{quality}}
\land G_{\mathrm{slice}}
\land G_{\mathrm{regression}}
\land G_{\mathrm{safety}}
\land G_{\mathrm{latency}}
\land G_{\mathrm{cost}}
```

评估的最终目标不是分数漂亮，而是支持是否上线、是否回滚、下一步改什么。

## 1. 为什么大模型评估困难

传统分类模型常常可以用 accuracy、precision、recall、F1、AUC 评估。

大模型更复杂。

原因有五个。

### 1.1 输出是开放的

大模型不是只输出一个类别，而是生成一段文本、代码、计划、工具调用或多模态结果。

同一个问题可能有多个正确答案。

例如：

```text
问题：解释一下 attention。
```

一个回答可以从公式讲，一个回答可以从直觉讲，一个回答可以从代码讲。

它们可能都正确，但自动打分很难判断谁更好。

### 1.2 能力是多维的

大模型能力包括：

1. 知识。
2. 推理。
3. 数学。
4. 代码。
5. 指令遵循。
6. 长上下文。
7. 工具调用。
8. 多轮对话。
9. 安全性。
10. 鲁棒性。

一个总分无法代表所有能力。

模型 A 可能数学更好，模型 B 可能安全更好，模型 C 可能更便宜更快。

### 1.3 质量和用户价值不完全一致

benchmark 分数高，不一定线上用户满意。

线上用户关心的是：

1. 是否解决问题。
2. 是否稳定。
3. 是否响应快。
4. 是否少废话。
5. 是否引用可靠。
6. 是否符合产品风格。

所以离线评估只能回答一部分问题。

### 1.4 评估数据可能被污染

大模型预训练语料很大，公开 benchmark 很可能被模型见过。

如果测试题泄漏到训练数据中，分数会虚高。

这叫 contamination。

### 1.5 模型行为有随机性

大模型解码可能使用 temperature、top-p、采样。

同一个 prompt 多次运行，结果可能不同。

因此评估要控制随机种子、采样次数和统计置信度。

面试表达：大模型评估难在输出开放、能力多维、线上价值复杂、数据可能污染以及模型行为有随机性。

## 2. 评估的核心问题

评估不是为了得到一个漂亮分数，而是为了做决策。

常见决策包括：

1. 新模型是否优于旧模型？
2. 新 prompt 是否可以上线？
3. RAG 改动是否提升答案质量？
4. 微调是否真的有效？
5. 量化是否损失能力？
6. 安全策略是否拦截过度？
7. 新版本是否引入回归？
8. 哪类用户或任务受益最大？

所以评估设计要先明确决策问题。

如果问题是“新模型能不能上线”，评估就要覆盖质量、安全、延迟、成本和回归。

如果问题是“新 RAG 是否更好”，评估就要覆盖检索命中、引用准确、答案 groundedness 和 hallucination。

面试表达：评估不是追求单一分数，而是为模型选择、上线门禁、迭代方向和风险控制服务。

## 3. 评估体系总览

一个完整评估体系通常包括六层：

```text
Task Definition
  -> Dataset / Benchmark
  -> Metric / Judge
  -> Evaluation Runner
  -> Analysis / Slice
  -> Decision / Gate
```

### 3.1 Task Definition

先定义任务。

例如：

1. 问答。
2. 摘要。
3. 翻译。
4. 代码生成。
5. 数学推理。
6. RAG 问答。
7. 工具调用。
8. 安全拒答。
9. 多轮客服。

任务定义越清楚，评估越可信。

### 3.2 Dataset / Benchmark

数据集决定评估覆盖什么。

好的 benchmark 要覆盖真实分布、困难样本、边界场景和高风险场景。

### 3.3 Metric / Judge

指标决定怎么打分。

可以是 exact match、pass@k、BLEU、ROUGE、人工评分、pairwise preference 或 LLM-as-a-Judge。

### 3.4 Evaluation Runner

评估执行器负责批量运行模型、记录结果、控制版本和保存 trace。

### 3.5 Analysis / Slice

只看总体分数不够。

要按任务、语言、难度、用户群、长度、安全类型、业务类型切片分析。

### 3.6 Decision / Gate

最后要形成上线决策。

例如：

1. 总体质量不低于旧版本。
2. 关键任务提升至少 2%。
3. 安全回归为 0。
4. P95 延迟不超过阈值。
5. 单请求成本不超过预算。

面试表达：完整评估体系不只是 benchmark，而是任务定义、数据、指标、执行、切片分析和上线门禁。

## 4. Offline Eval

Offline eval 是离线评估。

它在固定数据集上比较模型、prompt 或系统版本。

### 4.1 适用场景

适合：

1. 模型训练后初筛。
2. prompt 版本比较。
3. RAG 配置比较。
4. 微调效果验证。
5. 回归测试。
6. 上线前门禁。

### 4.2 优点

1. 成本可控。
2. 可重复。
3. 可快速迭代。
4. 可以覆盖高风险样本。
5. 可以保留历史结果做对比。

### 4.3 局限

1. 数据集可能不代表真实用户。
2. 公开 benchmark 可能被污染。
3. 指标可能无法反映用户体验。
4. 无法覆盖所有线上交互。
5. 对多轮、Agent 和开放任务较难评价。

面试表达：Offline eval 是上线前的基础门禁，但不能替代线上实验和人工 bad case 分析。

## 5. Online Eval

Online eval 是在线评估。

它使用真实流量观察模型表现。

常见形式包括：

1. A/B test。
2. 灰度发布。
3. shadow traffic。
4. canary release。
5. 在线用户反馈。
6. 人工抽检。

### 5.1 线上指标

线上指标包括：

1. 任务成功率。
2. 用户满意度。
3. 点赞率、踩率。
4. 重新提问率。
5. 会话完成率。
6. 转人工率。
7. 留存和转化。
8. 延迟和错误率。
9. 成本。

### 5.2 注意事项

线上实验风险更高。

要控制：

1. 流量比例。
2. 用户分层。
3. 安全策略。
4. 回滚机制。
5. 统计显著性。
6. 实验周期。

面试表达：Online eval 更接近真实用户价值，但成本和风险更高，需要灰度、监控、回滚和统计显著性控制。

## 6. Human Eval

Human eval 是人工评估。

它通常用于自动指标难以判断的开放式任务。

### 6.1 人工评估方式

常见方式包括：

1. 绝对打分。
2. pairwise 比较。
3. 排序。
4. 标注错误类型。
5. 选择最佳答案。
6. 标注是否满足需求。

### 6.2 人工评估维度

可以从这些维度打分：

1. 正确性。
2. 完整性。
3. 有用性。
4. 简洁性。
5. 指令遵循。
6. 事实一致性。
7. 安全性。
8. 风格一致性。

### 6.3 人工评估风险

人工评估也有问题：

1. 成本高。
2. 速度慢。
3. 标注员标准不一致。
4. 容易受输出长度和格式影响。
5. 需要仲裁和一致性检查。

面试表达：Human eval 适合开放式质量判断，但要有清晰 rubric、标注员校准、一致性统计和抽样策略。

## 7. Automatic Eval

Automatic eval 是自动评估。

它依赖程序化指标或模型裁判。

### 7.1 程序化指标

常见指标包括：

1. exact match。
2. F1。
3. BLEU。
4. ROUGE。
5. pass@k。
6. unit test pass rate。
7. retrieval recall。
8. latency。
9. cost。

这类指标可重复、便宜，但对开放式任务表达能力有限。

### 7.2 LLM-as-a-Judge

LLM-as-a-Judge 用另一个模型评价输出。

优点是便宜、快、可扩展。

风险是 judge bias、位置偏置、长度偏置、提示词敏感和自我偏好。

所以它需要用人工 gold set 校准。

面试表达：Automatic eval 适合规模化回归测试，但要知道程序化指标和 LLM judge 各自的偏差。

## 8. Safety Eval

Safety eval 评估模型是否安全。

它不是普通质量评估的附属项，而是独立门禁。

常见安全维度包括：

1. harmful content。
2. jailbreak。
3. prompt injection。
4. privacy leakage。
5. self-harm。
6. cyber abuse。
7. hate and harassment。
8. tool misuse。
9. misinformation。

安全评估要同时看两个方向：

1. 应该拒绝的是否拒绝。
2. 不该拒绝的是否误拒。

如果只追求高拦截率，模型可能变得过度拒答。

面试表达：Safety eval 要同时评估攻击成功率和误拒率，避免只提高安全拦截却损害正常用户体验。

## 9. Regression Eval

Regression eval 是回归评估。

它用于防止新版本修了一个问题又引入新问题。

### 9.1 回归集来源

回归集通常来自：

1. 历史线上 bad case。
2. 客诉案例。
3. 高价值用户任务。
4. 安全事故样本。
5. 关键 benchmark。
6. 人工构造边界样本。

### 9.2 什么时候跑回归

这些变化都应该跑回归：

1. 模型权重变化。
2. prompt 变化。
3. RAG 索引变化。
4. reranker 变化。
5. decoding 参数变化。
6. safety policy 变化。
7. 工具 schema 变化。
8. 推理引擎变化。

面试表达：Regression eval 是模型系统持续迭代的安全网，bad case 要沉淀成长期回归集。

## 10. 评估指标怎么选

指标要服务任务。

不同任务适合不同指标。

### 10.1 问答任务

常用指标：

1. answer correctness。
2. groundedness。
3. citation correctness。
4. hallucination rate。
5. human preference。

### 10.2 代码任务

常用指标：

1. pass@1。
2. pass@k。
3. unit test pass rate。
4. compile success rate。
5. hidden test success rate。

### 10.3 数学和推理任务

常用指标：

1. final answer accuracy。
2. reasoning validity。
3. step correctness。
4. consistency under resampling。

### 10.4 RAG 任务

常用指标：

1. retrieval recall。
2. context precision。
3. answer faithfulness。
4. citation accuracy。
5. abstention accuracy。

### 10.5 Agent 任务

常用指标：

1. task success rate。
2. tool call accuracy。
3. step count。
4. cost per task。
5. unsafe action rate。

面试表达：指标不能一套通用到底，必须按问答、代码、推理、RAG、Agent 和安全任务分别设计。

## 11. 评估中的版本管理

评估结果必须可复现。

需要记录：

1. 模型版本。
2. tokenizer 版本。
3. prompt 版本。
4. system instruction。
5. decoding 参数。
6. 数据集版本。
7. judge 版本。
8. 代码版本。
9. 运行时间。
10. 环境配置。

如果这些不记录，评估分数没有比较意义。

例如，一个模型看似提升 3%，可能只是 prompt 改了，或者 temperature 变了。

面试表达：评估结果必须和模型、prompt、数据、judge、参数和代码版本绑定，否则不可复现也不可审计。

## 12. 评估中的切片分析

总体平均分很容易掩盖问题。

所以要做 slice analysis。

常见切片包括：

1. 任务类型。
2. 难度。
3. 输入长度。
4. 输出长度。
5. 语言。
6. 用户群。
7. 场景。
8. 安全类别。
9. 数据来源。
10. 是否使用工具。

例如总体提升 1%，但长上下文任务下降 8%，这可能无法上线。

再例如英文 benchmark 提升明显，但中文客服任务下降，也可能不符合业务目标。

面试表达：评估必须做切片分析，因为平均分提升可能掩盖关键场景回退。

## 13. 评估中的统计意识

评估结果有噪声。

噪声来源包括：

1. 样本量不足。
2. 采样随机性。
3. 标注员差异。
4. judge 不稳定。
5. 用户流量波动。

所以要关注：

1. 置信区间。
2. 显著性检验。
3. bootstrap。
4. 多次采样。
5. inter-annotator agreement。

一个 0.3% 的提升，如果没有统计显著性，不能轻易说模型更强。

面试表达：评估分数不是绝对真理，必须结合样本量、方差和置信区间判断提升是否可信。

## 14. 一个上线门禁示例

假设你要上线一个新的客服模型。

可以设计如下门禁：

```text
质量：离线人工 pairwise win rate >= 旧版本 + 3%
RAG：引用准确率 >= 95%
安全：高危问题违规输出率 = 0
回归：历史 P0/P1 bad case 通过率 = 100%
性能：P95 TTFT <= 1.5s，P95 总延迟 <= 8s
成本：单会话成本不高于旧版本 110%
线上：5% 灰度 48 小时，无关键指标下降
```

这类门禁比“benchmark 提升 2 分”更接近真实生产要求。

## 15. 最小评估规划审计 demo

下面这个 demo 用标准库模拟一次模型上线前评估规划：它同时计算离线质量、任务切片、安全攻击成功率、误拒率、历史 bad case 回归、pairwise win rate、线上 A/B 粗略差异，并给出是否通过上线门禁。

```python
import math
from pprint import pprint


eval_cases = [
    {"id": "qa_001", "task": "qa", "slice": "normal", "old": 0.82, "new": 0.88, "weight": 1.0},
    {"id": "qa_002", "task": "qa", "slice": "zh_customer", "old": 0.78, "new": 0.83, "weight": 1.5},
    {"id": "rag_001", "task": "rag", "slice": "citation", "old": 0.76, "new": 0.86, "weight": 1.5},
    {"id": "code_001", "task": "code", "slice": "python", "old": 0.64, "new": 0.68, "weight": 1.0},
    {"id": "math_001", "task": "math", "slice": "reasoning", "old": 0.58, "new": 0.62, "weight": 1.0},
    {"id": "long_001", "task": "long_context", "slice": "long", "old": 0.70, "new": 0.69, "weight": 1.2},
    {"id": "bad_001", "task": "regression", "slice": "p0_badcase", "old": 1.00, "new": 1.00, "weight": 2.0},
    {"id": "bad_002", "task": "regression", "slice": "p1_badcase", "old": 0.90, "new": 0.92, "weight": 2.0},
]

safety_cases = [
    {"id": "safe_attack_1", "kind": "unsafe", "blocked": True},
    {"id": "safe_attack_2", "kind": "unsafe", "blocked": True},
    {"id": "safe_attack_3", "kind": "unsafe", "blocked": True},
    {"id": "benign_1", "kind": "benign", "blocked": False},
    {"id": "benign_2", "kind": "benign", "blocked": False},
]

online = {
    "old_success": 480,
    "old_total": 10000,
    "new_success": 535,
    "new_total": 10000,
}


def weighted_score(rows, key):
    total_weight = sum(row["weight"] for row in rows)
    return sum(row[key] * row["weight"] for row in rows) / total_weight


def mean(values):
    return sum(values) / len(values) if values else 0.0


old_score = weighted_score(eval_cases, "old")
new_score = weighted_score(eval_cases, "new")
delta = new_score - old_score

slice_scores = {}
for row in eval_cases:
    bucket = slice_scores.setdefault(row["slice"], {"old": [], "new": []})
    bucket["old"].append(row["old"])
    bucket["new"].append(row["new"])
slice_report = {
    name: {
        "old": round(mean(values["old"]), 3),
        "new": round(mean(values["new"]), 3),
        "delta": round(mean(values["new"]) - mean(values["old"]), 3),
    }
    for name, values in sorted(slice_scores.items())
}

epsilon = 0.02
regressions = [
    row["id"]
    for row in eval_cases
    if row["new"] + epsilon < row["old"]
]

wins = sum(row["new"] > row["old"] + epsilon for row in eval_cases)
ties = sum(abs(row["new"] - row["old"]) <= epsilon for row in eval_cases)
pairwise_win_rate = (wins + 0.5 * ties) / len(eval_cases)

unsafe = [row for row in safety_cases if row["kind"] == "unsafe"]
benign = [row for row in safety_cases if row["kind"] == "benign"]
unsafe_pass = sum(not row["blocked"] for row in unsafe)
benign_reject = sum(row["blocked"] for row in benign)
safety = {
    "attack_success_rate": unsafe_pass / len(unsafe),
    "over_refusal_rate": benign_reject / len(benign),
}

old_rate = online["old_success"] / online["old_total"]
new_rate = online["new_success"] / online["new_total"]
pooled = (online["old_success"] + online["new_success"]) / (
    online["old_total"] + online["new_total"]
)
se = math.sqrt(pooled * (1 - pooled) * (1 / online["old_total"] + 1 / online["new_total"]))
z_score = (new_rate - old_rate) / se

gates = {
    "quality": delta >= 0.03,
    "slice": all(report["delta"] >= -0.02 for report in slice_report.values()),
    "regression": not regressions,
    "safety": safety["attack_success_rate"] == 0.0 and safety["over_refusal_rate"] <= 0.05,
    "online": z_score >= 1.64,
    "latency": True,
    "cost": True,
}

summary = {
    "old_score": round(old_score, 4),
    "new_score": round(new_score, 4),
    "delta": round(delta, 4),
    "pairwise_win_rate": round(pairwise_win_rate, 3),
    "slice_report": slice_report,
    "regressions": regressions,
    "safety": {key: round(value, 3) for key, value in safety.items()},
    "online": {
        "old_rate": round(old_rate, 4),
        "new_rate": round(new_rate, 4),
        "uplift": round(new_rate - old_rate, 4),
        "z_score": round(z_score, 3),
    },
    "gates": gates,
    "gate_pass": all(gates.values()),
}

pprint(summary, sort_dicts=False)
```

## 16. 面试回答模板

如果面试官问：

```text
你怎么判断一个大模型版本真的比旧版本好？
```

可以这样答：

```text
我会先明确比较目标，是通用能力、某个业务任务，还是上线版本。然后设计多层评估体系：第一层是 offline eval，用固定 benchmark 和业务 gold set 比较质量；第二层是 human eval 或 pairwise eval，用清晰 rubric 判断开放式输出；第三层是 automatic eval，包括 exact match、pass@k、RAG faithfulness、LLM-as-a-Judge 等，但 judge 要用人工集校准；第四层是 safety eval，单独评估 jailbreak、隐私泄露、prompt injection 和误拒；第五层是 regression eval，把历史 bad case 作为上线门禁；最后做 online eval，通过灰度或 A/B test 看真实用户指标、延迟和成本。

我不会只看总体平均分，还会做任务、难度、语言、长度、用户群和安全类别的切片分析，并记录模型、prompt、数据、judge、decoding 参数和代码版本。只有当质量提升有统计显著性，关键场景没有回退，安全和成本满足门禁，线上灰度稳定，才认为新版本真的更好。
```

## 17. 常见误区

### 17.1 只看公开 benchmark

公开 benchmark 容易被污染，也不一定符合业务。

要结合私有业务集和线上指标。

### 17.2 只看平均分

平均分会掩盖关键场景回退。

要做切片分析。

### 17.3 把 LLM judge 当真理

LLM judge 本身有偏置。

要用人工 gold set 校准。

### 17.4 忽视安全和成本

模型质量提升但安全下降或成本翻倍，不一定能上线。

### 17.5 不做版本记录

没有版本记录，评估无法复现。

## 19. 本章小结

本章建立了大模型评估的总框架。

核心结论：

1. 大模型评估困难，因为输出开放、能力多维、线上价值复杂、数据可能污染、行为有随机性。
2. 评估的目标不是得到分数，而是支持模型选择、上线门禁、迭代方向和风险控制。
3. 完整评估体系包括任务定义、数据、指标、执行、切片分析和决策门禁。
4. Offline eval 可重复、成本低，是上线前基础门禁。
5. Online eval 更接近真实用户价值，但需要灰度、监控、回滚和统计显著性。
6. Human eval 适合开放式任务，但要有 rubric、标注校准和一致性检查。
7. Automatic eval 适合规模化回归，但要理解程序化指标和 LLM judge 的偏差。
8. Safety eval 必须独立评估攻击成功率和误拒率。
9. Regression eval 是持续迭代的安全网。
10. 评估结果必须记录模型、prompt、数据、judge、参数和代码版本。
11. 面试中要用多层评估、切片分析、统计显著性和上线门禁来回答模型是否真的变好。
