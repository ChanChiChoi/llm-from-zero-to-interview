# 第七章：多模态与长上下文评估

重点：VQA、OCR、图表理解、视频理解、语音评估、needle-in-a-haystack、真实长文任务。

面试重点：多模态评估要关注 grounding、事实性和跨模态一致性。

## 本章目标

学完本章，你要能回答：

1. 多模态评估为什么不能只看文本答案？
2. VQA、OCR、图表理解、视频理解分别怎么评估？
3. 什么是 grounding，为什么它是多模态评估核心？
4. 语音评估要同时看哪些维度？
5. 长上下文评估为什么不能只做 needle-in-a-haystack？
6. 如何设计真实长文任务评估？
7. 多模态与长上下文评估有哪些常见陷阱？

多模态模型和长上下文模型都容易出现一个问题：看起来能力很强，但评估并没有真正测到目标能力。

多模态模型可能不是真的理解图像，只是根据语言先验猜答案。

长上下文模型可能能在几十万 token 中找到一个特殊字符串，但不能真正阅读、整合和推理长文档。

所以这一章的核心是：评估必须对齐真实任务，而不是只构造容易刷分的指标。

面试表达：多模态评估要看模型是否把回答 grounded 到输入模态上；长上下文评估要看模型是否能检索、整合、比较和推理，而不是只看上下文窗口长度。

## 1. 为什么多模态评估更难

文本评估通常只需要比较问题、答案和参考答案。

多模态评估还要处理图像、视频、音频、图表、屏幕截图、文档版面等输入。

这带来几个额外难点。

### 1.1 输入信息更复杂

一张图片里可能包含：

1. 物体。
2. 人物。
3. 动作。
4. 空间关系。
5. 文字。
6. 图表。
7. 细粒度属性。
8. 隐含场景。

模型回答错误，可能是视觉识别错，也可能是语言推理错。

### 1.2 答案可能依赖 grounding

Grounding 是指模型回答必须能对应到输入中的视觉、听觉或时序证据。

例如：

```text
问题：图中红色杯子在桌子的哪一侧？
正确回答：左侧。
```

模型不能只凭常识猜，要真的定位红色杯子和桌子。

### 1.3 语言先验会污染评估

很多 VQA 数据集存在语言捷径。

例如问题是：

```text
What color is the banana?
```

模型即使不看图，也可能回答 yellow。

如果数据集中大多数香蕉都是黄色，这个评估就无法证明模型看懂了图。

### 1.4 多模态输出更难自动打分

多模态任务可能要求输出：

1. 文本答案。
2. Bounding box。
3. 分割 mask。
4. 时间戳。
5. 表格。
6. 代码。
7. 语音。
8. 图像。

不同输出需要不同指标。

### 1.5 标注成本更高

多模态标注通常需要人工查看图像、视频或听音频。

视频和长文档标注尤其昂贵。

面试表达：多模态评估难在输入复杂、答案需要 grounding、语言先验强、输出形式多样、标注成本高。

## 2. 多模态评估的核心维度

多模态 eval 不应只问“答案对不对”。

至少要看以下维度。

### 2.1 感知能力

感知能力是识别输入中有什么。

例如：

1. 物体识别。
2. 颜色识别。
3. 文字识别。
4. 人脸或表情识别。
5. 声音事件识别。
6. 场景识别。

### 2.2 空间与结构理解

空间理解包括：

1. 左右、上下、前后。
2. 包含、遮挡、接触。
3. 物体数量。
4. 图表坐标轴。
5. 文档版面结构。

很多模型能识别物体，但空间关系容易错。

### 2.3 细粒度 grounding

细粒度 grounding 要求模型指出证据位置。

例如：

```text
请指出图片中导致你判断“有人在骑自行车”的区域。
```

如果模型能给出合理区域，可信度更高。

### 2.4 跨模态一致性

跨模态一致性是指不同模态信息之间不能互相矛盾。

例如视频中人物说“我在雨天开车”，画面却是晴天走路。

模型需要判断音频、画面和文本是否一致。

### 2.5 推理能力

多模态推理不只是识别。

例如：

```text
图片中哪个人最可能下一步接球？
```

这需要动作、位置、场景和常识推理。

### 2.6 忠实性

忠实性指回答是否基于输入，而不是幻觉。

例如图中没有狗，模型不能说“有一只狗”。

面试表达：多模态评估应拆成感知、空间结构、grounding、跨模态一致性、推理和忠实性，而不是只看最终文本分数。

## 3. VQA 评估

VQA 是 Visual Question Answering，给定图像和问题，让模型回答。

例如：

```text
Image: 一张厨房照片
Question: 桌子上有几个杯子？
Answer: 3
```

### 3.1 VQA 常见题型

VQA 题型包括：

1. 物体识别。
2. 属性识别。
3. 计数。
4. 空间关系。
5. OCR 问答。
6. 常识推理。
7. 细节查找。
8. 反事实问题。

不同题型难度差异很大。

### 3.2 VQA 指标

常见指标：

1. Accuracy。
2. Exact match。
3. Soft accuracy。
4. LLM judge。
5. 人工评估。

Soft accuracy 用于处理多个标注员答案不完全一致的情况。

例如 10 个标注员中 7 个回答 `dog`，3 个回答 `puppy`，模型回答 `puppy` 不应完全判错。

### 3.3 VQA 的语言先验问题

VQA 很容易被语言先验影响。

例如：

```text
Question: What color is the sky?
```

模型可能不看图直接回答 blue。

改进方法包括：

1. 做 image-question 反事实配对。
2. 构造答案分布均衡的数据。
3. 使用需要定位证据的问题。
4. 加入视觉遮挡或区域扰动测试。
5. 要求模型输出证据区域。

### 3.4 VQA 错误分析

VQA 错误可以分为：

1. 没识别出目标物体。
2. 识别出物体但属性错。
3. 数量错误。
4. 空间关系错误。
5. OCR 错误。
6. 推理错误。
7. 幻觉输入中不存在的对象。

面试表达：VQA 不能只看 overall accuracy，要按题型拆分，并检查模型是否真的看图而不是依赖语言先验。

## 4. OCR 与文档理解评估

OCR 是 Optical Character Recognition，即从图像中识别文字。

对多模态大模型来说，OCR 不只是读出文字，还包括理解版面、表格和文档结构。

### 4.1 OCR 基础指标

常见指标：

1. Character Error Rate，字符错误率。
2. Word Error Rate，词错误率。
3. Exact match。
4. 字段级准确率。
5. 版面区域检测 IoU。

字符错误率适合短文本和密集文字。

字段级准确率适合票据、表单和证件。

### 4.2 文档问答

文档问答输入可能是一张扫描件、PDF 页面或截图。

问题可能是：

```text
这张发票的总金额是多少？
合同的截止日期是哪一天？
表格第三行第二列是什么？
```

评估重点：

1. 是否读对文字。
2. 是否理解字段含义。
3. 是否理解版面关系。
4. 是否能跨页查找。
5. 是否能拒答缺失信息。

### 4.3 表格和版面结构

文档理解常常不是线性文本。

例如表格中，同一个数字的含义取决于行标题和列标题。

评估时要检查：

1. 行列定位。
2. 合并单元格。
3. 表头解析。
4. 脚注和单位。
5. 跨页表格。

### 4.4 OCR 评估的坑

常见问题：

1. 模型读对文字但理解错字段。
2. 模型理解字段但读错数字。
3. 对低分辨率、倾斜、遮挡图片不鲁棒。
4. 忽略小字、脚注、水印。
5. 对中英文混排、公式、表格支持差。

面试表达：OCR eval 既要看文字识别，也要看版面、字段和语义理解；发票、合同、表格这类任务最好用字段级指标和人工抽检结合。

## 5. 图表理解评估

图表理解是多模态评估中很重要的一类。

模型需要理解柱状图、折线图、饼图、散点图、流程图、科学图像等。

### 5.1 图表任务类型

常见任务：

1. 读取数值。
2. 比较大小。
3. 计算差值或比例。
4. 判断趋势。
5. 解释异常点。
6. 总结图表结论。
7. 根据图表回答业务问题。

### 5.2 为什么图表难

图表理解需要同时处理：

1. 视觉元素。
2. 坐标轴。
3. 刻度。
4. 图例。
5. 文本标签。
6. 单位。
7. 多条曲线之间的关系。

模型可能看懂标题，但读错坐标轴。

### 5.3 图表评估指标

可以按任务选择指标：

1. 数值误差。
2. Exact match。
3. 趋势判断 accuracy。
4. 表格还原准确率。
5. LLM judge 评估解释质量。
6. 人工专家评审。

对于数值答案，通常要设置容忍误差。

例如柱状图读取 `99.8`，模型回答 `100`，在很多场景下可以接受。

### 5.4 图表错误类型

常见错误：

1. 读错坐标轴。
2. 忽略单位。
3. 混淆图例。
4. 把视觉趋势说反。
5. 无法进行简单计算。
6. 对密集图表产生幻觉。

面试表达：图表 eval 要把视觉读取、坐标轴理解、数值计算和结论解释拆开评估，否则很难知道模型到底错在哪里。

## 6. 视频理解评估

视频比图像多了时间维度。

模型不仅要看见物体，还要理解动作、事件顺序和因果关系。

### 6.1 视频任务类型

常见任务：

1. 动作识别。
2. 事件检测。
3. 视频问答。
4. 时间定位。
5. 视频摘要。
6. 多人交互理解。
7. 过程步骤识别。
8. 异常行为检测。

### 6.2 时间定位

时间定位要求模型指出事件发生时间。

例如：

```text
问题：视频中人物什么时候拿起杯子？
答案：00:13 到 00:15。
```

评估可以用时间区间 IoU。

如果预测时间段和标准时间段重叠足够多，就算正确。

### 6.3 视频问答

视频问答可以分为：

1. 单帧可回答问题。
2. 多帧整合问题。
3. 时间顺序问题。
4. 因果问题。
5. 反事实问题。

真正有价值的是后几类，因为它们需要时间理解。

### 6.4 视频摘要

视频摘要没有唯一答案。

评估方式包括：

1. 人工评分。
2. LLM judge。
3. 关键事件覆盖率。
4. 事实错误率。
5. 冗余度。

### 6.5 视频评估的坑

常见坑：

1. 只抽少量帧导致漏掉关键事件。
2. 模型根据字幕猜答案，而不是看画面。
3. 只评估单帧问题，测不到视频理解。
4. 时间戳标注不一致。
5. 长视频成本高，评估样本少。

面试表达：视频 eval 的关键是时间维度，要评估事件顺序、时间定位、动作变化和跨帧推理，而不是把视频退化成几张图片。

## 7. 语音评估

语音模型评估要看输入语音理解，也要看输出语音质量。

常见场景包括 ASR、语音问答、实时语音助手和 TTS。

### 7.1 ASR 评估

ASR 是自动语音识别。

常用指标：

1. Word Error Rate。
2. Character Error Rate。
3. 关键词召回率。
4. 标点和分段准确率。
5. 说话人分离准确率。

对于中文，字符错误率常用。

对于英文，词错误率常用。

### 7.2 语音问答评估

语音问答要同时评估：

1. 是否听懂问题。
2. 是否正确回答。
3. 是否处理口音、噪声、打断。
4. 是否满足实时延迟。
5. 是否保持上下文。

如果 ASR 错了，后续回答也可能错。

所以要区分 ASR 错误和推理错误。

### 7.3 TTS 评估

TTS 是文本转语音。

常见维度：

1. 自然度。
2. 可懂度。
3. 音色相似度。
4. 情感表达。
5. 韵律和停顿。
6. 稳定性。

TTS 很多指标需要人类听评。

### 7.4 实时语音助手评估

实时语音助手还要看交互体验：

1. 首 token 延迟。
2. 端到端响应延迟。
3. 打断响应速度。
4. 噪声鲁棒性。
5. 多轮上下文保持。
6. 错误恢复能力。

面试表达：语音 eval 不能只看 WER，还要看语义正确性、噪声鲁棒性、实时延迟、打断处理和输出语音体验。

## 8. Grounding 评估

Grounding 是多模态评估的核心。

它回答：模型的答案有没有证据支持？证据在哪里？

### 8.1 图像 grounding

图像 grounding 可以要求模型输出 bounding box 或区域描述。

例如：

```text
问题：哪一处显示车辆正在转弯？
输出：车辆和转向灯所在区域。
```

指标可以用：

1. Box IoU。
2. Pointing accuracy。
3. Region recall。
4. 人工判断证据是否合理。

### 8.2 文档 grounding

文档 grounding 要求答案引用原文位置。

例如：

```text
答案：合同截止日期是 2026-06-30。
证据：第 4 页第 2.1 条。
```

评估要看答案和引用是否一致。

### 8.3 视频 grounding

视频 grounding 要求定位时间段。

例如：

```text
答案：人物在 00:31 拿起钥匙。
证据：00:30-00:33。
```

指标可以用时间 IoU 和人工复核。

### 8.4 为什么 grounding 重要

没有 grounding，模型很容易产生看似合理的幻觉。

有 grounding 后，评估者可以检查回答是否真的来自输入。

这对医疗、法律、金融、自动驾驶等场景尤其重要。

面试表达：grounding 把多模态回答从“像不像”变成“有没有证据”，是提高评估可信度的关键。

## 9. 长上下文评估

长上下文模型的核心问题不是能放多少 token，而是能不能有效使用这些 token。

一个模型支持 1M token，不代表它能在 1M token 中稳定检索、整合和推理。

### 9.1 长上下文能力类型

长上下文能力可以拆成：

1. 长距离检索。
2. 多处证据整合。
3. 跨章节推理。
4. 长文摘要。
5. 长文一致性检查。
6. 多文档比较。
7. 状态跟踪。
8. 指令保持。

不同能力不能用一个指标替代。

### 9.2 Needle-in-a-Haystack

Needle-in-a-haystack 是把一段特殊信息插入长文本，让模型找出来。

例如：

```text
在 100k token 文档中插入：The secret code is 481516.
问题：secret code 是什么？
```

这个测试可以衡量长距离检索能力。

### 9.3 Needle 的价值

它的优点：

1. 简单。
2. 可控。
3. 容易自动评分。
4. 可以测试不同位置的召回能力。
5. 可以观察 context length 增加后的退化。

### 9.4 Needle 的局限

Needle 不能代表完整长上下文能力。

原因：

1. 它通常只测检索，不测理解。
2. needle 往往很特殊，容易被模型注意到。
3. 不需要整合多处证据。
4. 不需要处理矛盾信息。
5. 不像真实业务文档。

所以 needle 分数高，只能说明模型具备某种长距离定位能力。

### 9.5 位置敏感性

长上下文模型可能存在位置偏差。

例如更容易记住开头和结尾，中间信息容易丢失。

评估时要把答案放在不同位置：

1. 开头。
2. 前 25%。
3. 中间。
4. 后 25%。
5. 结尾。

面试表达：needle-in-a-haystack 是长上下文检索测试，不是完整长文理解测试；真正评估要加入多证据整合、冲突处理和真实任务。

## 10. 真实长文任务评估

真实长文任务比 needle 更重要。

因为用户通常不是问“某个特殊字符串在哪里”，而是要求模型理解复杂材料。

### 10.1 长文摘要

长文摘要评估维度：

1. 关键事实覆盖。
2. 结构清晰度。
3. 忠实性。
4. 压缩率。
5. 冗余度。
6. 是否遗漏少数但重要信息。

自动指标很难完全评价长摘要，通常需要 LLM judge 和人工校准。

### 10.2 多文档问答

多文档问答要求模型从多个文档中找证据。

评估重点：

1. 是否找到相关文档。
2. 是否整合多个证据。
3. 是否引用正确来源。
4. 是否处理冲突信息。
5. 是否拒答没有依据的问题。

### 10.3 长文一致性检查

例如检查合同、论文或技术文档中的矛盾。

任务可能是：

```text
找出文档中关于付款期限的所有不一致描述。
```

这要求模型跨段落比较。

### 10.4 长链路状态跟踪

在对话或 Agent 任务中，模型要记住长期状态。

例如：

1. 用户偏好。
2. 已完成步骤。
3. 约束条件。
4. 待办事项。
5. 工具调用结果。

评估可以构造长对话，看模型是否在后续回合遵守早期约束。

### 10.5 长上下文代码库评估

代码库任务是长上下文能力的重要场景。

例如：

1. 找到函数定义。
2. 理解跨文件调用链。
3. 修改多个文件。
4. 通过测试。
5. 不破坏现有接口。

这类任务不能只看回答文本，最好用真实测试和代码审查评估。

面试表达：真实长文 eval 应覆盖摘要、多文档问答、冲突检测、状态跟踪和代码库任务，并要求引用证据或通过外部验证。

## 11. 多模态长上下文的交叉评估

越来越多任务同时具备多模态和长上下文特征。

例如：

1. 长视频问答。
2. 多页 PDF 理解。
3. 长会议音频总结。
4. 多张图表综合分析。
5. 屏幕录制操作理解。
6. 多轮语音视觉助手。

这些任务更接近真实应用，也更难评估。

### 11.1 多页文档

多页文档评估要看：

1. 跨页字段查找。
2. 表格延续。
3. 页眉页脚过滤。
4. 图文混排理解。
5. 引用页码和位置。

### 11.2 长视频

长视频评估要看：

1. 关键事件召回。
2. 事件顺序。
3. 时间定位。
4. 人物状态变化。
5. 多模态线索整合。

### 11.3 长会议音频

长会议音频评估要看：

1. ASR 准确率。
2. 说话人分离。
3. 议题结构。
4. 决策和 action item 提取。
5. 引用时间戳。
6. 多轮上下文一致性。

面试表达：多模态长上下文评估要同时看模态理解、时间或位置 grounding、长距离信息整合和任务完成质量。

## 12. 指标与评估方法选择

不同任务适合不同指标。

### 12.1 客观答案任务

适合：

1. Accuracy。
2. Exact match。
3. 字段级 F1。
4. 数值误差。
5. IoU。

例如 OCR 字段抽取、图表数值读取、VQA 选择题。

### 12.2 开放式生成任务

适合：

1. Rubric-based human eval。
2. LLM-as-a-judge。
3. Pairwise comparison。
4. 错误类型标注。
5. 事实性检查。

例如图像描述、视频摘要、长文总结。

### 12.3 可执行任务

适合：

1. 单元测试。
2. UI 自动化测试。
3. 工具调用成功率。
4. 端到端任务成功率。

例如 computer use、代码库修改、表格操作。

### 12.4 人工评估何时必要

以下情况通常需要人工评估：

1. 没有唯一标准答案。
2. 输出质量依赖语境。
3. 需要专业知识。
4. 需要判断证据是否充分。
5. 自动 judge 和人工不一致。

面试表达：指标选择要跟输出形式匹配；客观任务用自动指标，开放任务用 rubric 和人工校准，可执行任务用真实执行结果。

## 13. 常见错误类型

### 13.1 多模态错误

常见错误：

1. 物体识别错误。
2. 属性识别错误。
3. 空间关系错误。
4. 数量错误。
5. OCR 错误。
6. 图表读数错误。
7. 视频时序错误。
8. 音频转写错误。
9. 跨模态矛盾未发现。
10. 幻觉不存在的信息。

### 13.2 长上下文错误

常见错误：

1. 中间位置遗忘。
2. 只看开头和结尾。
3. 多证据整合失败。
4. 冲突信息处理失败。
5. 引用错误。
6. 长文摘要遗漏关键事实。
7. 后文覆盖前文约束。
8. 指令漂移。

### 13.3 评估系统错误

常见错误：

1. 标注答案不唯一。
2. 图片或视频质量差。
3. 标注框不准确。
4. 时间戳边界不一致。
5. OCR 标准答案有误。
6. judge prompt 偏好长答案。
7. 数据集存在语言捷径。

面试表达：错误分析要区分感知错误、grounding 错误、推理错误、长上下文整合错误和评估系统错误。

## 14. 真实项目中的评估流程

一个多模态或长上下文项目，可以按下面流程评估。

### 14.1 明确任务定义

先明确：

1. 输入模态是什么。
2. 输出形式是什么。
3. 是否需要引用证据。
4. 是否要求实时性。
5. 是否允许拒答。
6. 是否涉及安全和隐私。

### 14.2 构建分层评估集

评估集应覆盖：

1. 简单样本。
2. 边界样本。
3. 对抗样本。
4. 真实用户样本。
5. 私有 holdout。
6. 多语言或多场景样本。

### 14.3 选择指标

根据任务选择：

1. 自动指标。
2. 人工评审。
3. LLM judge。
4. 执行验证。
5. 线上指标。

### 14.4 做错误分析

不要只看总分。

要输出：

1. 按模态拆分。
2. 按任务类型拆分。
3. 按长度拆分。
4. 按难度拆分。
5. 按错误类型拆分。

### 14.5 回归测试

多模态系统升级后，可能出现能力回退。

例如：

1. 新视觉 encoder 提升 OCR，却降低空间关系。
2. 长上下文优化提升 needle，却降低摘要质量。
3. 新 prompt 减少幻觉，却增加拒答。

所以要建立固定回归集。

面试表达：真实项目评估要从任务定义出发，构建分层数据集，组合自动指标、人审和线上指标，并持续做回归测试。

## 15. 面试官会怎么问

### 15.1 如何评估一个 VLM？

回答要点：

1. 按任务拆分：VQA、OCR、图表、caption、grounding、文档理解。
2. 客观任务用 accuracy、字段级 F1、IoU、数值误差。
3. 开放任务用 human eval 或 LLM judge，并用人工 gold set 校准。
4. 检查语言先验和视觉 grounding。
5. 做错误类型分析和真实场景 holdout。

### 15.2 如何评估长上下文模型？

回答要点：

1. 不只看最大 context length。
2. 用 needle 测长距离检索。
3. 用多文档问答测证据整合。
4. 用长文摘要测压缩和忠实性。
5. 用冲突检测测跨段落比较。
6. 按上下文长度和答案位置拆分结果。

### 15.3 如何判断模型是真的看图，而不是靠语言猜？

回答要点：

1. 做反事实图文配对。
2. 平衡答案分布。
3. 要求输出证据区域。
4. 做遮挡或区域扰动测试。
5. 分析模型在需要视觉细节问题上的表现。

### 15.4 如何评估视频理解？

回答要点：

1. 区分单帧问题和跨帧问题。
2. 加入时间定位和事件顺序任务。
3. 使用时间 IoU、关键事件覆盖率和人工评估。
4. 检查字幕捷径。
5. 对长视频按时长和事件密度拆分。

## 16. 标准回答模板

如果面试官问：“你会如何设计多模态和长上下文评估？”

可以这样回答：

```text
我会先把任务拆开，而不是只看一个总分。多模态部分会覆盖 VQA、OCR、图表理解、视频理解、语音理解和 grounding。客观任务用 accuracy、字段级 F1、IoU、数值误差等指标；开放式 caption、摘要和解释任务用 rubric-based human eval 或 LLM judge，并用人工 gold set 校准。

我会特别检查模型是否 grounded 到输入证据上，比如图像区域、文档页码、视频时间戳或音频片段。为了避免语言先验，我会构造反事实样本、答案分布均衡样本和遮挡扰动样本。

长上下文部分不会只看 needle-in-a-haystack。Needle 可以测试长距离检索，但真实能力还包括多证据整合、冲突处理、长文摘要、状态跟踪和代码库任务。我会按上下文长度、答案位置、任务类型和错误类型拆分报告，并保留私有 holdout 和回归集。
```

## 17. 常见误区

### 17.1 把多模态评估当成文本评估

如果只看最终文本答案，不看证据位置，就很难发现模型是否真的理解输入模态。

### 17.2 只测简单 VQA

简单 VQA 可能被语言先验解决。

要加入空间关系、计数、OCR、图表和反事实问题。

### 17.3 认为 context window 越长越强

上下文窗口长只是容量，不等于有效使用能力。

### 17.4 只用 needle 代表长上下文能力

Needle 主要测检索，不能代表摘要、整合、推理和冲突处理。

### 17.5 忽略成本和延迟

多模态和长上下文评估成本很高。

真实系统还要报告 latency、吞吐、token 成本和失败率。

### 17.6 用 LLM judge 评估一切

LLM judge 对开放任务有用，但 OCR 数字、表格字段、代码执行和时间定位等任务应优先用客观指标。

## 18. 小练习

### 练习 1

设计一个 VQA 评估集，要求能区分模型是真的看图还是靠语言先验猜答案。

至少说明：题型、反事实设计、答案分布和错误分析方法。

### 练习 2

给定一个发票理解任务，你会设计哪些字段级指标？

参考字段：发票号、日期、商家名称、税额、总金额、币种。

### 练习 3

一个长上下文模型 needle 测试接近满分，但长文摘要质量很差。你会如何解释？

思考方向：检索能力、信息整合、忠实性、压缩能力、位置偏差。

### 练习 4

设计一个视频理解评估任务，要求必须用到时间顺序，不能只看单帧。

至少包含：问题、标准答案、时间定位指标和错误类型。

## 19. 本章总结

多模态评估的核心是：模型是否真的理解输入模态，并把回答 grounded 到证据上。

VQA 要防语言先验，OCR 要看文字和版面，图表要看坐标轴与数值推理，视频要看时间顺序和事件定位，语音要看识别、语义、实时性和交互体验。

长上下文评估的核心是：模型是否能有效使用长上下文。

Needle-in-a-haystack 是有用的检索测试，但不能替代真实长文任务。

真实长文评估应覆盖多文档问答、长文摘要、冲突检测、状态跟踪和代码库任务。

面试中要强调：好的评估不是把所有能力压成一个分数，而是按任务、模态、长度、证据位置和错误类型拆解能力画像。

最终目标是回答三个问题：模型看到了什么，依据是什么，能否在真实任务中稳定完成目标。
