# 第五章：污染检测与可信评估

重点：benchmark contamination、data contamination、benchmark leakage、train-test leakage、训练集泄漏、近重复检测、时间切分评估、canary 数据。

面试重点：如果评估集被训练数据污染，分数就不可信。

## 本章目标

学完本章，你要能回答：

1. 什么是 benchmark contamination / data contamination / benchmark leakage？
2. 为什么训练集泄漏会让评估分数失真？
3. 如何做 exact duplicate 和 near-duplicate 检测？
4. 什么是时间切分评估，为什么它重要？
5. Canary 数据如何帮助发现泄漏？
6. 如何写一份可信评估报告？

大模型评估的一个核心风险是：模型可能不是“会做题”，而是“见过题”。

如果评估集已经进入训练数据、微调数据、RAG 知识库或 prompt 调参过程，评估分数就会虚高。

这时再精细的指标、judge 和统计显著性都没有意义。

## 本章资料边界

本章第二轮精修参考了 [GPT-3](https://arxiv.org/abs/2005.14165)、[Deduplicating Training Data Makes Language Models Better](https://arxiv.org/abs/2107.06499)、[Detecting Pretraining Data from Large Language Models](https://arxiv.org/abs/2310.16789)、[Rephrased Samples Improve Benchmark Robustness](https://arxiv.org/abs/2305.13104)、[LiveBench](https://arxiv.org/abs/2406.19314) 和前序 benchmark / human eval / LLM-as-a-Judge 章节的资料边界。

本章聚焦污染检测和可信评估的工程口径：exact duplicate、near duplicate、答案泄漏、评估反馈泄漏、时间切分、canary、数据血缘、RAG 检索库污染、扰动测试和风险报告。不展开完整数据治理平台、商业数据合规制度、闭源模型训练数据取证、复杂隐私攻击或生产级近重复检索系统。

## 本章核心公式

一个评估样本可以抽象成：

```math
e_i=(q_i,a_i,m_i,t_i,u_i)
```

其中 `q_i` 是题目或输入，`a_i` 是参考答案、rubric 或期望行为，`m_i` 是任务、风险、来源和版本等元数据，`t_i` 是样本产生时间，`u_i` 是使用记录，例如是否用于 prompt 调参、模型选择或 RAG 知识库。

规范化后的 exact duplicate 可以用哈希检测：

```math
h_i=\mathrm{hash}(\nu(q_i,a_i))
```

其中 `\nu` 表示大小写、空白、标点、HTML、Unicode 和代码格式等规范化过程。训练或检索语料的哈希集合记作 `H_train`，则 exact 污染集合为：

```math
\mathcal{D}_{\mathrm{exact}}
=
\{e_i:h_i\in H_{\mathrm{train}}\}
```

Near duplicate 可以用字符或 token shingle 的 Jaccard similarity 粗略表示。设 `G_n(x)` 是文本 `x` 的 n-gram 集合：

```math
J_n(x,y)
=
\frac{|G_n(x)\cap G_n(y)|}{|G_n(x)\cup G_n(y)|}
```

样本 `e_i` 对训练、SFT、偏好数据、RAG 知识库和 prompt 调参日志的最大相似度为：

```math
c_i
=
\max_j J_n(\nu(q_i,a_i),\nu(d_j))
```

如果 `c_i` 超过阈值 `tau`，就要标为 near-duplicate 风险，而不是直接当作干净样本。

时间切分门禁可以写成：

```math
G_{\mathrm{time}}
=
\bigwedge_i I(t_i>T_{\mathrm{cut}})
```

这里 `T_cut` 是模型训练数据或调参数据的截止日期。时间切分只能降低预训练污染风险，不能自动排除后训练、RAG、日志回流和模板泄漏。

Canary 泄漏率可以写成：

```math
R_{\mathrm{canary}}
=
\frac{N_{\mathrm{seenCanary}}}{N_{\mathrm{canary}}}
```

如果 `R_canary` 大于 0，说明隐藏评估样本、日志、检索库或训练链路存在泄漏风险。

扰动测试可以比较原题和改写题的分数差：

```math
\Delta_{\mathrm{pert}}
=
S_{\mathrm{orig}}-S_{\mathrm{pert}}
```

如果原题高分、轻微改写后显著下降，要怀疑模型记住了题面或模板，而不是掌握了能力。

最终可信评估门禁可以写成：

```math
G_{\mathrm{trust}}
=
G_{\mathrm{exact}}
\land G_{\mathrm{near}}
\land G_{\mathrm{time}}
\land G_{\mathrm{canary}}
\land G_{\mathrm{lineage}}
\land G_{\mathrm{access}}
\land G_{\mathrm{pert}}
\land G_{\mathrm{report}}
```

也就是 exact / near duplicate 不过线、时间切分合理、canary 未泄漏、数据血缘清楚、访问隔离到位、扰动测试稳定、报告说明残余污染风险。

## 1. 什么是评估污染

评估污染也常写成 data contamination、benchmark contamination 或 benchmark leakage，指评估数据以某种形式泄漏到模型训练、调参或系统上下文中，导致评估结果不再反映真实泛化能力。

常见形式包括：

1. 测试题出现在预训练语料里。
2. 测试题出现在 SFT 数据里。
3. 测试题出现在 RLHF / preference 数据里。
4. 测试题被用于 prompt 调参。
5. 测试答案被写入 RAG 知识库。
6. 标注规范或答案模板被模型看到。
7. 线上用户反复提交 benchmark 题，进入日志再被训练。

污染的结果是：模型在 benchmark 上表现很好，但面对真正新问题时能力没有那么强。

面试表达：评估污染的本质是测试集信息泄漏到训练或调参流程，导致 benchmark 分数高估模型泛化能力。

## 2. 为什么大模型更容易污染

传统模型训练数据通常比较可控。

大模型预训练使用大规模网页、代码、书籍、论文、论坛和数据集，数据来源更复杂。

这带来几个问题。

### 2.1 公开 benchmark 容易进入预训练语料

很多 benchmark 题目、答案、解析会出现在 GitHub、论文、博客、论坛和课程资料中。

大模型预训练时可能已经见过。

### 2.2 代码数据尤其容易泄漏

代码 benchmark 的测试题、参考答案和单元测试可能出现在 GitHub。

模型可能记住了具体解法。

### 2.3 迭代过程会污染测试集

团队可能反复用同一个测试集调 prompt、调超参、调 RAG。

即使测试集没进入训练，也会被开发过程“用穿”。

### 2.4 RAG 系统有额外风险

如果 benchmark 的答案或评测集文档进入检索库，模型可能直接检索到答案。

这不是模型本身能力提升，而是评估设计泄漏。

面试表达：大模型数据规模巨大，公开 benchmark、代码仓库、开发日志、RAG 知识库和反复调参都可能造成污染。

## 3. 污染类型

可以把污染分成五类。

### 3.1 Exact Duplicate

完全重复。

测试样本原文出现在训练数据中。

例如：

```text
测试题：What is the capital of France?
训练数据：What is the capital of France? Paris.
```

### 3.2 Near Duplicate

近似重复。

文本经过轻微改写，但语义和答案基本相同。

例如题干换了措辞，答案不变。

### 3.3 Answer Leakage

题目未泄漏，但答案、解析或解法泄漏。

这在数学和代码任务中常见。

### 3.4 Distribution Leakage

模型没见过具体题目，但见过同一生成模板的大量类似题。

例如合成 benchmark 用固定模板生成，训练数据也有同模板样本。

### 3.5 Evaluation Feedback Leakage

评估结果反过来指导训练或 prompt 调整。

长期反复迭代后，模型或系统会过拟合评估集。

面试表达：污染不只有完全重复，还包括近重复、答案泄漏、模板泄漏和评估反馈泄漏。

## 4. 污染检测的基本思路

污染检测不是一个单一算法，而是一套证据链。

常见思路包括：

1. Exact match。
2. Near-duplicate search。
3. Embedding similarity。
4. MinHash / SimHash。
5. N-gram overlap。
6. 时间切分。
7. 数据血缘追踪。
8. Canary 检测。
9. 行为异常分析。

没有任何方法能保证 100% 发现污染。

所以可信评估要同时使用多种方法，并在报告里说明残余风险。

面试表达：污染检测要组合文本去重、语义相似、时间切分、数据血缘和行为分析，不能依赖单一方法。

## 5. Exact Duplicate 检测

Exact duplicate 是最简单的泄漏。

### 5.1 基本方法

对训练数据和测试数据做规范化，然后比对哈希。

规范化包括：

1. 小写化。
2. 去除多余空白。
3. Unicode 规范化。
4. 去除标点或 HTML 标签。
5. 代码格式统一。

然后计算 hash。

如果 hash 相同，就是完全重复。

可以写成：

```math
I_{\mathrm{exact}}(e_i)
=
I(h_i\in H_{\mathrm{train}})
```

其中 `H_train` 不只应该包含预训练语料，也应该覆盖 SFT、preference data、评测调参日志、RAG 文档库和公开题库快照。

### 5.2 局限

Exact match 只能发现完全一样的文本。

稍微改写、换行、翻译、变量名变化，就可能漏掉。

所以它必要但不充分。

面试表达：Exact duplicate 检测简单可靠，但只能发现完全重复，必须结合近重复和语义相似检测。

## 6. Near Duplicate 检测

Near duplicate 更常见也更难。

### 6.1 N-gram Overlap

把文本切成 n-gram，计算重叠比例。

例如 5-gram overlap 很高，说明文本可能近重复。

适合长文本和题干检测。

常用口径是 Jaccard similarity：

```math
J_n(x,y)
=
\frac{|G_n(x)\cap G_n(y)|}{|G_n(x)\cup G_n(y)|}
```

阈值不能机械固定。例如短选择题即使只有几个 n-gram 重合也可能泄漏，长文摘要则需要更高阈值和人工抽检。

### 6.2 MinHash

MinHash 用于高效估计 Jaccard similarity。

它适合大规模数据去重。

常见流程：

```text
文本规范化
  -> shingling
  -> MinHash signature
  -> LSH bucket
  -> candidate pair comparison
```

### 6.3 SimHash

SimHash 适合网页去重。

相似文本的 SimHash 汉明距离较小。

### 6.4 Embedding Similarity

Embedding similarity 能发现语义相似但表面不相似的样本。

例如题目改写、翻译或语序变化。

风险是误报较多，需要人工抽检。

### 6.5 代码近重复

代码任务要考虑：

1. 变量名替换。
2. 函数名替换。
3. 注释删除。
4. 格式变化。
5. 语义等价实现。

可以结合 AST、token 序列和测试行为检测。

面试表达：近重复检测通常结合 n-gram、MinHash/SimHash、embedding similarity 和人工抽检；代码还要考虑 AST 和变量改名。

## 7. 时间切分评估

时间切分是可信评估的重要方法。

它的核心思想是：用模型训练截止日期之后产生的数据评估模型。

例如模型预训练数据截止到 2024 年 1 月，那么评估集尽量使用 2024 年 2 月之后的数据。

### 7.1 为什么有效

如果数据产生在训练截止之后，模型预训练时不可能见过。

这能降低污染风险。

### 7.2 适用场景

适合：

1. 新闻问答。
2. 代码库新 issue。
3. 新论文理解。
4. 新产品文档问答。
5. 新政策问答。
6. 最新事实性任务。

### 7.3 局限

时间切分不能解决所有问题。

原因包括：

1. 模型真实训练截止日期可能不透明。
2. 后训练数据可能更新。
3. RAG 系统可能检索到新数据。
4. 新题可能和旧题模板相似。
5. 对纯数学和抽象推理任务不一定有效。

面试表达：时间切分能降低预训练污染风险，但要知道模型训练截止、后训练数据和 RAG 检索都会影响判断。

## 8. Canary 数据

Canary 是故意插入的特殊样本，用于检测数据泄漏或记忆。

### 8.1 Canary 的形式

Canary 可以是：

1. 独特字符串。
2. 人工构造问题。
3. 特殊格式样本。
4. 不应出现在训练数据中的伪事实。
5. 带唯一标识的测试样本。

例如：

```text
The secret evaluation phrase is: zebra-lake-7294.
```

如果模型在不该知道的情况下输出这个短语，说明可能发生泄漏或记忆。

### 8.2 使用方式

Canary 可以用于：

1. 检测训练数据泄漏。
2. 检测日志回流污染。
3. 检测 RAG 知识库污染。
4. 检测评估集是否被开发者误用。

### 8.3 注意事项

Canary 不能包含真实秘密。

Canary 本身也要严格权限管理。

如果 canary 泄漏到公开仓库，它就失去作用。

面试表达：Canary 是用于检测泄漏的特殊哨兵样本，能发现训练、日志和检索链路中的污染，但不能使用真实敏感信息。

## 9. 数据血缘和权限控制

可信评估需要数据治理。

要知道每个样本从哪里来、经过什么处理、是否进入训练。

### 9.1 数据血缘

需要记录：

1. 原始来源。
2. 采集时间。
3. 清洗过程。
4. 标注过程。
5. 数据版本。
6. 是否用于训练。
7. 是否用于调参。
8. 是否用于最终测试。

### 9.2 权限隔离

最终 holdout test set 应该限制访问。

不要让所有开发者都能反复查看和调参。

### 9.3 数据集分层

可以把评估数据分成：

1. Dev set：用于日常调试。
2. Validation set：用于模型选择。
3. Test set：用于最终报告。
4. Hidden test set：用于关键上线门禁。
5. Canary set：用于泄漏检测。

面试表达：防污染不仅是算法问题，也是数据治理问题，需要数据血缘、访问控制和 dev/validation/test/hidden 分层。

## 10. 行为异常分析

除了数据比对，还可以看模型行为是否异常。

### 10.1 异常高分

如果模型在某个公开 benchmark 上远超同等规模模型，但在私有集上一般，要怀疑污染。

### 10.2 非自然答案

模型输出和参考答案高度一致，甚至错误也一样，可能是记忆。

### 10.3 对题目扰动敏感

如果原题答对，轻微改写就答错，可能是记住了原题。

### 10.4 解析复现异常

数学或代码题中，模型能给出答案，但推理过程不合理，也要警惕记忆。

面试表达：污染不仅可以从数据查，也可以从行为查，例如异常高分、答案过度相似、轻微改写失效和推理过程不合理。

## 11. 可信评估设计

可信评估要在设计阶段就防污染。

可以采用以下策略。

### 11.1 私有 Holdout

保留一批私有样本，只用于最终评估。

不用于日常 prompt 调参和模型选择。

### 11.2 动态评估集

定期引入新样本，尤其是训练截止后的样本和线上新 bad case。

### 11.3 多源评估

不要只依赖一个公开 benchmark。

要结合：

1. 公开 benchmark。
2. 私有业务集。
3. 新时间切分集。
4. 人工专家集。
5. 线上 A/B。

### 11.4 Perturbation Test

对题目做扰动：

1. 改写题干。
2. 替换变量名。
3. 改变数字。
4. 改变上下文顺序。
5. 加入干扰信息。

如果模型只会原题，不会变体，说明泛化不足或有污染风险。

一个简单风险指标是：

```math
\Delta_{\mathrm{pert}}
=
S_{\mathrm{orig}}-S_{\mathrm{pert}}
```

如果 `Delta_pert` 很大，而且原题和变体的难度本应接近，就要复查原题是否泄漏、答案模板是否泄漏，或者模型是否只过拟合了题面格式。

### 11.5 Report Contamination Risk

评估报告要明确说明污染风险。

不要只报分数。

面试表达：可信评估要用私有 holdout、动态评估、多源评估、扰动测试和污染风险报告来降低 benchmark 过拟合。

## 12. 不同任务的污染风险

### 12.1 选择题 Benchmark

风险：题目和答案容易公开流传。

缓解：私有题、时间切分、改写变体、答案顺序随机。

### 12.2 数学推理

风险：题目和详细解析泄漏。

缓解：变换数字、生成等价题、检查过程合理性。

### 12.3 代码生成

风险：题目、函数签名、单元测试和参考解法在 GitHub 中出现。

缓解：新构造任务、隐藏测试、变换函数名和上下文、行为测试。

### 12.4 RAG

风险：答案或测试文档进入检索库。

缓解：评估库和生产库隔离、文档版本控制、检索 trace 审计。

### 12.5 安全评估

风险：攻击 prompt 被模型针对性训练，导致只会拒绝已知攻击模板。

缓解：动态红队、新攻击变体、隐藏安全集。

面试表达：不同任务污染风险不同，代码看 GitHub 泄漏，数学看解析泄漏，RAG 看检索库污染，安全看攻击模板过拟合。

## 13. 可信评估报告模板

一份可信评估报告不应该只写分数。

建议包含：

1. 评估目标。
2. 模型版本。
3. 数据集来源。
4. 数据集时间范围。
5. 是否公开。
6. 去重和近重复检测方法。
7. 与训练数据的相似度检查结果。
8. 是否使用私有 holdout。
9. 是否使用时间切分数据。
10. 是否做扰动测试。
11. 是否有 canary。
12. 残余污染风险。
13. 结论和上线建议。

示例：

```text
本次评估使用 60% 私有业务样本、20% 时间切分样本、10% 历史 bad case、10% 公开 benchmark。
公开 benchmark 仅作为参考，不作为上线门禁。
对训练数据做 exact duplicate 和 MinHash near-duplicate 检测，移除相似度高于阈值的样本。
最终 test set 未用于 prompt 调参。
残余风险：无法完全确认预训练阶段是否见过部分公开题。
```

面试表达：可信评估报告必须说明数据来源、时间范围、去重方法、防泄漏措施和残余污染风险。

## 14. 最小污染检测审计 demo

下面这个 demo 不调用模型，也不依赖外部库。它审计一个 toy 评估集是否被训练语料、RAG 文档、prompt 调参日志或 canary 泄漏污染：包含规范化 exact hash、字符 shingle Jaccard、时间切分、canary 搜索、评估反馈泄漏和扰动分数下降。

```python
from datetime import date
from pprint import pprint
import hashlib
import re


train_cutoff = date.fromisoformat("2025-12-31")
near_threshold = 0.20
perturb_threshold = 0.25

def normalize(text):
    parts = re.findall(r"[a-z0-9\u4e00-\u9fff]+", text.lower())
    return "".join(parts)


def digest(text):
    return hashlib.sha256(normalize(text).encode("utf-8")).hexdigest()


def shingles(text, n=4):
    cleaned = normalize(text)
    if len(cleaned) < n:
        return {cleaned}
    return {cleaned[i : i + n] for i in range(len(cleaned) - n + 1)}


def jaccard(left, right):
    union = left | right
    if not union:
        return 0.0
    return len(left & right) / len(union)
```

## 15. 面试回答模板

如果面试官问：

```text
你怎么判断一个 benchmark 是否可信？如何防止 contamination？
```

可以这样答：

```text
我会先看 benchmark 的数据来源、时间范围和是否公开。如果是公开 benchmark，我会默认存在预训练污染风险，不能作为唯一结论。然后检查它是否和训练数据、SFT 数据、RLHF 数据、RAG 知识库以及 prompt 调参数据有重叠。检测方法包括 exact duplicate hash、n-gram overlap、MinHash/SimHash、embedding similarity、代码 AST 或 token 相似度，以及人工抽检。

设计上，我会保留私有 holdout test set，区分 dev、validation、test 和 hidden test，不反复用最终测试集调 prompt。对于可时间切分的任务，我会使用模型训练截止日期之后的数据。还可以加入 canary 样本检测日志回流或检索库污染。最后在报告里不仅给分数，还会说明去重方法、相似度阈值、样本来源、时间范围、是否做扰动测试和残余污染风险。只有多源评估、私有集、时间切分和线上表现都支持提升，我才会认为结果可信。
```

## 16. 常见误区

### 16.1 公开 benchmark 分数高就说明模型强

公开 benchmark 可能被污染。

要结合私有集和时间切分集。

### 16.2 只做 exact match 去重

Exact match 发现不了改写和近重复。

要做 near-duplicate 和语义相似检测。

### 16.3 反复用 test set 调 prompt

这会造成评估反馈泄漏。

要区分 dev、validation 和 test。

### 16.4 忽视 RAG 污染

如果答案在检索库里，模型不是推理出来，而是检索出来。

要审计检索 trace 和知识库版本。

### 16.5 不报告残余风险

污染无法 100% 排除。

可信报告必须说明残余风险。

## 18. 本章小结

本章讲了污染检测与可信评估。

核心结论：

1. 如果评估集被训练、调参或检索系统污染，分数就不可信。
2. 污染包括 exact duplicate、near duplicate、answer leakage、distribution leakage 和 evaluation feedback leakage。
3. 大模型更容易污染，因为预训练数据规模大、公开 benchmark 容易传播、开发迭代会反复使用测试集。
4. 污染检测要组合 exact match、n-gram、MinHash/SimHash、embedding similarity、数据血缘、时间切分和 canary。
5. 时间切分能降低预训练污染风险，但不能解决所有问题。
6. Canary 是检测训练、日志和检索泄漏的哨兵样本，不能包含真实秘密。
7. 数据治理要记录样本来源、采集时间、处理过程和用途，并做好权限隔离。
8. 模型行为异常也能提示污染，例如异常高分、答案过度相似、轻微改写失效。
9. 可信评估要使用私有 holdout、动态评估、多源评估、扰动测试和污染风险报告。
10. 面试中要强调：benchmark 分数只有在数据可信、协议清晰、防泄漏充分时才有意义。
