# 第 2 章 从 naive generate 到推理框架

上一章讲了 LLM Serving Engine 的边界。本章从最简单的 `model.generate()` 出发，解释它为什么不能直接承担生产服务，以及一个推理框架是怎样一步步长出来的。

很多人第一次接触推理框架，会直接被 vLLM、SGLang 里的 scheduler、block manager、executor、prefix cache、continuous batching 等概念淹没。更稳的学习路径是先问一个朴素问题：如果我已经会写 `model.generate()`，为什么还需要 serving engine？

一句话概括：

> 推理框架不是为了把 `generate()` 包一层 HTTP 接口，而是为了在高并发、长上下文、流式输出、显存受限和稳定性约束下，持续调度 token 级别的模型执行。

## 2.1 最简单的 generate

最小推理代码大概是这样：

```python
def generate(prompt, max_new_tokens=128):
    inputs = tokenizer(prompt, return_tensors="pt").to(device)
    outputs = model.generate(
        **inputs,
        max_new_tokens=max_new_tokens,
    )
    return tokenizer.decode(outputs[0], skip_special_tokens=True)
```

这段代码完成了三件事：

1. 把文本 prompt 编码成 token。
2. 调用模型生成后续 token。
3. 把输出 token 解码成文本。

如果只是本地实验、离线 demo、低频内部脚本，这已经够用。但它的系统抽象非常粗：一次调用进去，内部循环跑完，最后一次性返回结果。

生产 serving 不能只看“能不能生成”，还要看：

1. 多个请求同时进来怎么办。
2. 用户希望边生成边返回怎么办。
3. 某个请求特别长，占满显存怎么办。
4. 请求中途取消怎么办。
5. 不同请求的生成长度不同，怎么组成 batch。
6. 如何统计 TTFT、TPOT、tokens/s 和错误率。
7. GPU 空着时为什么吞吐仍然上不去。

这些问题不是简单给 `generate()` 加一个 Web 框架就能解决的。

## 2.2 generate 背后的真实循环

大模型自回归生成不是一次性吐出完整答案，而是按 token 逐步生成。简化后可以写成：

```python
tokens = tokenizer(prompt)

for step in range(max_new_tokens):
    logits, kv_cache = model.forward(tokens, kv_cache)
    next_token = sample(logits[-1])
    tokens.append(next_token)
    if next_token == eos_token_id:
        break

return tokenizer.decode(tokens)
```

这里已经出现了几个核心概念：

1. `tokens`：当前请求的输入和已生成 token。
2. `model.forward`：一次模型前向计算。
3. `kv_cache`：历史 token 的 key/value 缓存。
4. `sample`：从 logits 里选择下一个 token。
5. `eos_token_id`：停止条件。

推理框架要做的事情，本质上就是把这个单请求循环变成多请求、可调度、可流式、可观测、可中断的系统循环。

## 2.3 Prefill 和 Decode 的分离

在真实推理里，一个请求通常分成两个阶段：

1. Prefill：处理 prompt 中已有的全部 token，建立初始 KV Cache。
2. Decode：每次基于已有 KV Cache 生成一个新 token。

naive generate 往往把这两个阶段藏在同一个接口里。serving engine 必须把它们拆出来，因为两者的资源画像不同。

Prefill 的特点是：

1. 输入 token 多。
2. 单次计算量大。
3. 更像大矩阵批量计算。
4. 对 TTFT 影响很大。

Decode 的特点是：

1. 每轮每个请求通常只生成一个 token。
2. 单轮计算较小，但会重复很多轮。
3. 更容易受内存带宽、KV Cache 读取和调度开销影响。
4. 对 TPOT 和持续吞吐影响很大。

如果不区分 prefill 和 decode，就很难解释为什么长 prompt 会拖慢首 token，为什么短请求和长请求混在一起会互相影响，也很难设计 PD 分离这类高级架构。

## 2.4 单请求到多请求

最小 `generate()` 只处理一个请求。生产服务面对的是请求流：

```text
request_1: prompt 短，生成短
request_2: prompt 长，生成长
request_3: prompt 短，生成很长
request_4: prompt 中等，用户中途取消
```

如果每个请求独占一次模型执行，GPU 利用率会很低。模型权重已经在 GPU 上，最好的方式是把多个请求组成 batch，一次 forward 同时处理。

但 LLM 生成的 batch 和传统分类模型不同。分类模型通常是固定输入、一次 forward、一次返回。LLM decode 是动态过程：有的请求很快结束，有的请求继续生成，有的新请求正在等待加入。

因此推理框架需要一个请求队列：

```text
waiting queue      等待进入模型执行的请求
running set        正在生成的请求
finished set       已完成或已取消的请求
```

还需要一个 scheduler 决定：

1. 本轮哪些 waiting 请求可以进入。
2. 本轮哪些 running 请求继续 decode。
3. 显存不够时优先保留谁。
4. 长 prompt 是否要拆成 chunk。
5. 新请求是否可以插队以降低 TTFT。

这就是从函数调用到 serving engine 的第一道分水岭：推理不再是“调用一次函数”，而是“调度一批请求的生命周期”。

## 2.5 为什么不能只做静态 batching

最容易想到的优化是静态 batching：凑够 N 个请求，一起调用 `generate()`。

静态 batching 的问题是等待和浪费。

等待体现在：如果系统必须等够 batch size 才执行，低流量时首 token 延迟会变差。

浪费体现在：一个 batch 里的请求生成长度不同。短请求已经结束，但为了对齐长请求，框架可能仍然保留 padding 或无效位置，计算和显存都被浪费。

LLM serving 更常用的是 continuous batching，也叫动态批处理或 iteration-level batching。它的核心思想是：

1. 每一轮 decode 都重新决定 batch 里有哪些请求。
2. 已完成请求立刻移出。
3. 新请求在合适时机加入。
4. batch 是随时间滚动变化的。

这会把执行方式从下面这样：

```text
batch A 开始 -> batch A 全部结束 -> batch B 开始
```

变成下面这样：

```text
iteration 1: request 1, request 2
iteration 2: request 1, request 2, request 3
iteration 3: request 2, request 3, request 4
iteration 4: request 3, request 4
```

这就是现代 LLM serving engine 的核心能力之一。

## 2.6 KV Cache Manager 为什么必要

KV Cache 是推理性能的关键。没有 KV Cache，每生成一个新 token 都要重新计算整个历史上下文，成本会随着序列长度快速上升。

有了 KV Cache，decode 阶段可以复用历史 token 的 key/value，只计算新 token 的部分。

但在多请求场景下，KV Cache 不只是一个张量，而是一套内存管理问题：

1. 每个请求需要多少 cache。
2. cache 放在 GPU 显存的哪里。
3. 请求结束后如何释放。
4. 请求被抢占后如何保存或重算。
5. 长上下文请求是否会造成碎片。
6. 多个请求能否复用相同 prefix 的 cache。

naive generate 可以让框架内部自动维护 KV Cache。serving engine 必须自己理解和管理它，因为 KV Cache 往往决定最大并发、显存占用和稳定性。

这也是 vLLM 提出 PagedAttention 和 block manager 的背景：把 KV Cache 从“连续大张量”变成类似分页内存的块管理，减少碎片，提高可调度性。

## 2.7 Streaming 不只是边打印

很多人以为 streaming 就是模型每生成一个 token 就 `print` 一下。生产系统里的 streaming 更复杂。

它至少包括：

1. token 生成后及时解码。
2. 通过 SSE、WebSocket 或其他协议返回给客户端。
3. 客户端断开后及时取消请求。
4. 处理 tokenizer 的半个字符、空格和特殊 token。
5. 记录每个 token 的时间戳。
6. 出错时返回明确的结束状态。

streaming 对用户体验非常关键。即使总生成时间不变，只要 TTFT 更低、输出更平滑，用户感知也会明显变好。

所以推理框架通常不会等完整答案生成完再返回，而是把 decode loop 和输出通道连接起来：

```text
decode one token -> detokenize -> push chunk -> continue decode
```

这要求 engine 内部维护请求状态，而不是把 `generate()` 当成一次性黑盒。

## 2.8 从 generate 到 engine 的模块拆分

一个最小推理框架通常会拆成这些模块：

```text
API Server
  -> Request Parser
  -> Waiting Queue
  -> Scheduler
  -> Model Executor
  -> KV Cache Manager
  -> Sampler
  -> Streamer
  -> Metrics / Tracing
```

每个模块解决一个具体问题。

API Server 负责接收请求、鉴权、参数校验和连接管理。

Request Parser 负责把 prompt、max tokens、temperature、stop words 等参数规范化。

Waiting Queue 负责保存尚未执行的请求。

Scheduler 负责决定每一轮执行哪些请求。

Model Executor 负责真正调用模型前向计算。

KV Cache Manager 负责分配、释放和复用 KV Cache。

Sampler 负责 greedy、top-k、top-p、temperature、repetition penalty 等采样逻辑。

Streamer 负责把 token 增量返回给客户端。

Metrics / Tracing 负责记录延迟、吞吐、错误、显存和请求生命周期。

这套拆分看起来比 `generate()` 复杂很多，但每一层都是被生产问题逼出来的。

## 2.9 一个最小 engine 循环

极简 serving engine 的主循环可以想象成：

```python
while True:
    new_requests = api_server.poll()
    waiting_queue.extend(new_requests)

    batch = scheduler.schedule(
        waiting_queue=waiting_queue,
        running_requests=running_requests,
        cache_manager=cache_manager,
    )

    model_outputs = executor.forward(batch)
    next_tokens = sampler.sample(model_outputs.logits)

    for request, token in zip(batch.requests, next_tokens):
        request.append_token(token)
        streamer.push(request, token)

        if request.is_finished():
            cache_manager.free(request)
            running_requests.remove(request)

    metrics.record(batch, model_outputs)
```

真实系统当然更复杂，但主干就是这些：收请求、排队、调度、执行、采样、流式返回、释放资源、记录指标。

看 vLLM 或 SGLang 源码时，不要一上来陷入类名和文件名。先把每段代码放回这条主链路里，就容易理解很多。

## 2.10 请求状态机

为了让请求可调度，engine 通常会维护请求状态机。一个简化版本如下：

```text
WAITING -> PREFILLING -> DECODING -> FINISHED
                    \-> ABORTED
                    \-> FAILED
```

常见状态包括：

1. WAITING：已接收，但还没有分配执行资源。
2. PREFILLING：正在处理 prompt。
3. DECODING：正在逐 token 生成。
4. FINISHED：正常结束。
5. ABORTED：客户端取消或超时取消。
6. FAILED：执行出错。

状态机非常重要。没有状态机，就很难处理取消、超时、重试、流式断开、显存回收和指标统计。

面试里如果你能从请求状态机讲起，比单纯背 “continuous batching 可以提升吞吐” 更像真正做过系统。

## 2.11 最小框架的演进路线

从教学角度，实现一个最小推理框架可以按下面顺序来：

1. 先实现单请求 generate loop。
2. 加入 greedy、top-k、top-p 和 temperature sampling。
3. 显式保存 KV Cache。
4. 实现 batched prefill。
5. 实现 batched decode。
6. 加入 waiting queue 和 running set。
7. 实现简单 scheduler。
8. 支持 token streaming。
9. 包一层 HTTP API。
10. 记录 TTFT、TPOT、tokens/s、显存占用和错误率。

这个路线和本书第二部分的章节安排基本一致。先把最小链路跑通，再逐步替换成更接近 vLLM / SGLang 的机制。

## 2.12 常见误区

误区一：推理框架就是 Web 服务。

Web 服务只是入口。真正的难点在模型执行、KV Cache 管理、请求调度和 token 级别流式返回。

误区二：batch size 越大越好。

batch 太大会增加排队延迟和显存压力，也可能拖慢短请求。生产系统要在吞吐和延迟之间做权衡。

误区三：只要 GPU 利用率高，系统就一定好。

GPU 利用率高可能是长请求堵住了队列，也可能是无效 padding 多。LLM serving 要同时看 TTFT、TPOT、tokens/s、p95/p99 和失败率。

误区四：KV Cache 是模型内部细节，不需要关心。

对单请求 demo 可以不关心；对 serving engine 来说，KV Cache 管理几乎就是并发能力和显存效率的核心。

误区五：streaming 只是前端体验问题。

streaming 会影响请求状态、连接生命周期、取消逻辑、指标统计和资源释放，是 engine 的一等能力。

## 2.13 面试追问

1. `model.generate()` 和 serving engine 的本质区别是什么？
2. 为什么 LLM 推理要区分 prefill 和 decode？
3. 静态 batching 为什么不适合高并发 LLM serving？
4. continuous batching 的核心思想是什么？
5. KV Cache Manager 需要解决哪些问题？
6. 为什么 streaming 需要和请求状态机结合？
7. 如果一个用户中途断开连接，engine 应该做什么？
8. 如何从零实现一个最小 LLM serving engine？

参考回答思路：

1. 先说 `generate()` 是单请求黑盒调用，serving engine 是多请求调度系统。
2. 再说 prefill 影响首 token，decode 影响持续输出和吞吐。
3. 然后说核心模块：queue、scheduler、executor、KV cache manager、sampler、streamer、metrics。
4. 最后补 trade-off：吞吐、延迟、显存、公平性和稳定性需要一起优化。

## 2.15 本章小结

本章从 naive generate 出发，拆出了推理框架的基本演进路径。

`model.generate()` 适合理解生成原理，但不适合直接承担生产 serving。生产系统需要处理多请求、动态 batch、KV Cache、streaming、取消、超时、指标和资源释放。

从本章开始，你应该把 LLM serving 理解为一个 token 级调度系统，而不是一个简单的模型调用接口。下一章我们继续沿着这条线，完整走一遍推理请求的生命周期。
