## 47.2 先画总架构

一个最小 LLM serving engine 可以抽象成下面这张图：

```text
Client
  |
  v
API / Server
  |
  v
Request Manager
  |
  v
Engine Loop
  |
  +--> Scheduler
  |      |
  |      +--> Request / Sequence State
  |      +--> KV Cache Manager
  |
  +--> Model Runner
  |      |
  |      +--> Token / Position / Block Table
  |      +--> Transformer Forward
  |
  +--> Sampler
  |      |
  |      +--> Logits Processor
  |      +--> Sampling Params
  |
  +--> Output Processor
         |
         +--> Stream / Return / Metrics
```

如果加入 SGLang-style runtime，还会多出：

```text
Program Runtime
  |
  +--> Prefix / Radix Cache
  +--> Structured Decoding Controller
  +--> Tool / Function Call Parser
```

但不要一开始就把系统设计得很大。

最小版本只需要回答一个问题：

```text
一批请求进入系统后，如何被调度成 batch，如何执行 prefill/decode，如何维护 KV cache，如何采样并返回 token？
```

这就是推理框架的主骨架。

## 47.3 从三个项目抽象共同模块

可以把三个教学项目中的内容映射到同一张模块表里。

| 抽象模块 | tiny-llm 提供的视角 | nano-vLLM 提供的视角 | mini-sglang 提供的视角 |
|---|---|---|---|
| 模型层 | attention、RoPE、GQA、MLP、RMSNorm | model runner 调用模型 | backend、kernel、TP、runtime 执行 |
| 请求状态 | generate loop 中的 prompt/output | `Sequence` 状态机 | request/runtime state、program state |
| 调度器 | 从单请求到 batching | waiting/running 队列 | prefix-aware、overlap、online scheduling |
| KV cache | list cache、paged attention 动机 | block manager、block table | Radix Cache + KV pool 协作 |
| prefill/decode | 从计算差异理解 | engine step 中分开处理 | chunked prefill、prefix reuse |
| sampler | temperature/top-p 等采样 | batch sampler | structured generation、tool call 约束 |
| 服务层 | 课程实验入口 | offline API | OpenAI-compatible server、shell、benchmark |
| 性能观测 | 统计 latency 和显存 | TTFT/TPOT、scheduler log | prefix hit、overlap、online p99 |

这张表说明：

```text
不同项目讲的是不同切面，但最后都会落到同一组工程模块。
```

真正学会推理框架，不是能复述某个项目的代码路径，而是能把这些模块拆出来、重新组合，并解释每个模块的职责。

## 47.4 最小推理框架的模块清单

一个可以逐步升级的 mini serving engine，至少需要这些模块：

1. `Tokenizer`：文本和 token 之间的转换。
2. `Request`：保存用户输入、采样参数、状态、输出 token。
3. `Scheduler`：决定每一轮执行哪些请求。
4. `KVCacheManager`：分配、释放、查询 KV cache。
5. `BatchBuilder`：把多个请求整理成模型 forward 需要的输入。
6. `ModelRunner`：执行模型 forward，返回 logits。
7. `Sampler`：根据 logits 和采样参数选出下一个 token。
8. `OutputProcessor`：更新请求状态、detokenize、streaming 输出。
9. `Engine`：串起整个 step loop。
10. `Metrics`：统计 TTFT、TPOT、latency、吞吐、显存使用。

如果要支持在线服务，还需要：

1. `HTTP Server`：接收请求。
2. `Request Queue`：跨线程或跨协程传递请求。
3. `Streaming Channel`：把 token 持续返回给客户端。
4. `Abort Manager`：处理取消请求。
5. `Error Handler`：处理超时、OOM、模型错误和客户端断连。

如果要支持 SGLang-style 能力，还需要：

1. `PrefixCache` 或 `RadixCache`。
2. `ProgramState`。
3. `GrammarController` 或 `StructuredDecodingController`。
4. `ToolCallParser`。
5. `RuntimeScheduler`。

但是第一版不要全做。

第一版建议只做：

```text
Request + Scheduler + KVCacheManager + ModelRunner + Sampler + Engine Loop + Metrics
```

这个组合足以支撑后面几章的升级。

## 47.5 Request：请求不是一段文本

新手最容易把请求理解成一个字符串：

```text
prompt = "请介绍一下 transformer"
```

但在推理框架里，请求是一个持续变化的执行状态。

一个 `Request` 至少包含：

| 字段 | 含义 |
|---|---|
| `request_id` | 请求唯一 ID |
| `prompt` | 原始输入文本 |
| `prompt_tokens` | tokenized prompt |
| `output_tokens` | 已生成 token |
| `sampling_params` | temperature、top_p、max_tokens 等 |
| `status` | waiting、running、finished、aborted |
| `prefill_done` | 是否完成 prefill |
| `kv_blocks` | 该请求占用的 KV block |
| `arrival_time` | 到达时间 |
| `first_token_time` | 首 token 时间 |
| `finish_time` | 完成时间 |

如果支持 streaming，还需要：

| 字段 | 含义 |
|---|---|
| `stream_channel` | 输出 token 的通道 |
| `last_streamed_index` | 上次已经输出到哪个 token |
| `client_disconnected` | 客户端是否断开 |

如果支持 prefix cache，还需要：

| 字段 | 含义 |
|---|---|
| `prefix_cache_hit_len` | 命中的前缀 token 数 |
| `computed_tokens` | 已经有 KV 的 token 数 |
| `remaining_prefill_tokens` | 还需要 prefill 的 token 数 |

所以请求状态不是静态输入，而是 engine 每一轮 step 都会更新的对象。

## 47.6 Sequence 和 Request 的关系

很多框架里会出现 `Sequence`、`SequenceGroup`、`Request`、`Session` 这些名字。

不要被名字吓到。

可以这样理解：

| 概念 | 常见含义 |
|---|---|
| `Request` | 用户提交的一次生成任务 |
| `Sequence` | 一条具体生成路径 |
| `SequenceGroup` | 同一个 request 下的一组候选序列 |
| `Session` | 多轮对话或长期状态 |

如果只支持普通单条生成：

```text
一个 Request 约等于一个 Sequence
```

如果支持 beam search：

```text
一个 Request 可能包含多个 Sequence
```

如果支持多轮对话和 prefix reuse：

```text
一个 Session 下面可能有多个 Request，共享历史前缀
```

第一版 mini engine 不需要实现复杂层级。

建议先用一个 `Request` 表示一条生成路径，后面需要 beam search 或 session 再拆。

## 47.7 Scheduler：推理框架的决策中心

Scheduler 负责回答：

```text
这一轮 engine step，哪些请求可以进入模型 forward？每个请求执行多少 token？
```

最简单的 scheduler 可以只有两个队列：

```text
waiting: 还没开始执行的请求
running: 已经进入系统、正在生成的请求
```

每一轮调度时：

1. 从 `waiting` 中取若干请求进入 `running`。
2. 对未 prefill 的请求执行 prefill。
3. 对已 prefill 的请求执行 decode。
4. 如果请求完成，把它从 `running` 移除。
5. 如果请求 abort，释放资源并移除。

伪代码如下：

```python
def schedule(waiting, running, kv_manager, token_budget):
    batch = []

    while waiting and token_budget > 0:
        req = waiting[0]
        needed = len(req.prompt_tokens)
        if needed > token_budget:
            break
        if not kv_manager.can_allocate(req, needed):
            break
        waiting.pop(0)
        running.append(req)
        batch.append((req, "prefill", needed))
        token_budget -= needed

    for req in running:
        if token_budget <= 0:
            break
        if req.prefill_done and not req.finished:
            batch.append((req, "decode", 1))
            token_budget -= 1

    return batch
```

这段逻辑很粗糙，但它已经包含 scheduler 的本质：

1. 管理 waiting/running 状态。
2. 受 token budget 限制。
3. 受 KV cache 容量限制。
4. 区分 prefill 和 decode。
5. 输出一个可执行 batch。

后面第 48 章会在这个基础上升级 continuous batching。

## 47.8 为什么要有 token budget

推理框架不是简单地“有多少请求就全塞进模型”。

原因是 prefill 和 decode 的 token 数差异很大。

例如：

```text
请求 A：prompt 4096 tokens，正在 prefill
请求 B：已经 prefill 完，每步 decode 1 token
请求 C：已经 prefill 完，每步 decode 1 token
```

如果 scheduler 每轮都无脑处理最长 prefill，请求 B 和 C 的 decode 可能被长 prompt 阻塞，TPOT 抖动会很大。

所以 scheduler 通常要控制每一轮处理的 token 总数：

```text
本轮最多处理 2048 个 token
```

这就是 token budget。

token budget 的作用是：

1. 防止超大 batch 导致 OOM。
2. 控制每轮 step 的时间。
3. 给 decode 请求保留执行机会。
4. 为 chunked prefill 提供基础。
5. 让吞吐和延迟之间可以调参。

没有 token budget，后面很难实现 continuous batching 和 chunked prefill。

## 47.9 KVCacheManager：显存不是 append list

KV cache 是推理框架中最重要的状态之一。

最简单的 KV cache 可以想象成：

```text
request_id -> list of K/V tensors
```

但这种设计很快会遇到问题：

1. 不同请求长度不同，显存碎片严重。
2. 请求结束后释放不方便复用。
3. batch attention 难以统一访问。
4. 无法高效支持 prefix sharing。
5. 难以扩展到 paged attention。

更好的抽象是 block-based KV cache：

```text
Physical KV Blocks
  block 0
  block 1
  block 2
  ...

Request Block Table
  req A -> [3, 7, 8]
  req B -> [1, 2]
  req C -> [9]
```

`KVCacheManager` 至少要支持：

| 接口 | 作用 |
|---|---|
| `can_allocate(num_tokens)` | 判断是否有足够 KV 空间 |
| `allocate(req, num_tokens)` | 给请求分配 block |
| `append(req, num_tokens)` | decode 时追加 token 空间 |
| `free(req)` | 请求结束后释放 block |
| `get_block_table(req)` | 返回模型 forward 需要的 block table |
| `num_free_blocks()` | 查询剩余 block 数 |

第一版可以先实现 list KV cache。

但接口要按 block manager 的方向设计，这样第 49 章升级 paged KV cache 时不需要重写 scheduler 和 engine。

## 47.10 ModelRunner：不要把调度逻辑塞进模型

`ModelRunner` 的职责是执行模型 forward。

它不应该决定哪些请求进 batch。

也不应该负责请求何时完成。

它只关心：

```text
给我 input_ids、positions、kv cache metadata、attention metadata，我返回 logits。
```

典型输入包括：

| 输入 | 含义 |
|---|---|
| `input_ids` | 本轮要计算的 token |
| `positions` | 每个 token 的 position |
| `slot_mapping` | token 写入 KV cache 的位置 |
| `block_tables` | 每个请求的 KV block 表 |
| `seq_lens` | 每个请求当前上下文长度 |
| `is_prefill` | prefill 或 decode 标记 |

典型输出是：

```text
logits: [num_sequences, vocab_size]
```

或者在 prefill 场景下：

```text
logits: 只取每个请求最后一个 token 的 logits
```

这里有一个重要原则：

```text
Scheduler 负责选 batch，BatchBuilder 负责组 metadata，ModelRunner 只负责跑模型。
```

如果把这些职责混在一起，后面想换 scheduler、换 KV manager、换 kernel backend 都会很痛苦。

## 47.11 BatchBuilder：容易被忽略的胶水层

很多人设计推理框架时只写：

```text
batch = scheduler.schedule()
logits = model(batch)
```

但真实系统里，scheduler 输出的通常不是模型能直接吃的 tensor。

Scheduler 输出的是逻辑计划：

```text
req A: prefill token [0:512]
req B: decode 1 token
req C: decode 1 token
```

ModelRunner 需要的是物理输入：

```text
input_ids
positions
seq_lens
block_tables
slot_mapping
attention_metadata
```

把逻辑计划转换成物理输入的模块，就是 `BatchBuilder`。

它的职责包括：

1. 拼接 input token。
2. 计算 position。
3. 查询每个请求的 block table。
4. 计算 KV 写入位置。
5. 生成 attention metadata。
6. 标记每个请求对应的 logits 位置。

为什么要单独抽出来？

因为后续升级都会影响它：

| 升级 | BatchBuilder 变化 |
|---|---|
| continuous batching | batch 中混合不同阶段请求 |
| chunked prefill | 一个 prompt 被切成多个片段 |
| paged attention | 需要 block table 和 slot mapping |
| prefix cache | 部分 token 已有 KV，只计算未命中部分 |
| speculative decoding | 一次可能处理多个 draft token |
| structured generation | 可能需要额外 logits mask metadata |

所以 `BatchBuilder` 是推理框架里最值得重视的胶水层之一。

## 47.12 Sampler：采样不是 argmax

最简单的 sampler 是：

```python
next_token = logits.argmax(dim=-1)
```

但真实框架里 sampler 需要处理很多采样参数：

1. temperature。
2. top-k。
3. top-p。
4. repetition penalty。
5. frequency penalty。
6. presence penalty。
7. stop token。
8. max tokens。
9. random seed。

如果支持 OpenAI-compatible API，还会遇到：

1. 多个 stop strings。
2. logprobs。
3. best_of 或 n。
4. response format。
5. tool call 结构。

如果支持 structured generation，sampler 还要接收 token mask：

```text
当前状态下，哪些 token 是合法的？
```

因此 sampler 的输入不应该只是 logits。

建议设计成：

```python
def sample(logits, requests, sampling_metadata):
    ...
```

其中 `sampling_metadata` 可以包含：

1. 每个请求的采样参数。
2. 每个请求的已生成 token。
3. structured decoding mask。
4. stop token 和 stop string 状态。
5. logprob 需求。

第一版可以只做 greedy 或 temperature sampling。

但接口要保留按 request 处理不同 sampling params 的能力。

## 47.13 OutputProcessor：生成 token 后还没结束

模型 forward 和 sampler 之后，会得到每个请求的新 token。

但这不代表请求已经完成。

`OutputProcessor` 需要做：

1. 把 token append 到 `request.output_tokens`。
2. 更新请求长度。
3. 检查 EOS。
4. 检查 stop token。
5. 检查 stop string。
6. 检查 max_tokens。
7. detokenize 新增文本。
8. streaming 输出。
9. 更新 TTFT、TPOT、finish_time。
10. 对完成请求触发资源释放。

如果支持 tool call，还要做：

1. 增量解析 JSON。
2. 判断 tool call 是否完整。
3. 决定是否继续生成自然语言。
4. 输出兼容 OpenAI API 的 chunk。

如果支持 structured generation，还要做：

1. 更新 grammar state。
2. 给下一轮 sampler 准备 token mask。
3. 处理非法状态。

所以生成 token 只是中间结果。

请求生命周期真正结束，需要 OutputProcessor 做状态收尾。

## 47.14 Engine Loop：推理框架的心跳

推理框架的核心通常是一个循环：

```python
while engine_has_work():
    new_requests = request_queue.pop_all()
    scheduler.add_requests(new_requests)

    batch_plan = scheduler.schedule()
    if not batch_plan:
        wait_for_new_requests()
        continue

    model_inputs = batch_builder.build(batch_plan)
    logits = model_runner.forward(model_inputs)
    next_tokens = sampler.sample(logits, batch_plan.requests)
    output_processor.process(batch_plan, next_tokens)
    scheduler.remove_finished()
```

这个 loop 里有几个关键点：

1. 新请求可以在任意时刻到达。
2. 每一轮只执行 scheduler 选中的 batch。
3. batch 可能混合 prefill 和 decode。
4. model runner 不直接修改请求状态。
5. sampler 只决定 token，不负责释放资源。
6. output processor 更新请求状态。
7. scheduler 根据最新状态进入下一轮。

这个 loop 就是后续所有升级的骨架。

## 47.15 Prefill 和 Decode 的统一抽象

推理框架里最重要的区别之一是 prefill 和 decode。

| 阶段 | 输入 | 输出 | 计算特点 | KV 行为 |
|---|---|---|---|---|
| prefill | prompt tokens | prompt 最后位置 logits | token 多，矩阵大 | 写入整段 prompt 的 K/V |
| decode | 上一个 token | 下一个 token logits | 每请求 1 token | 追加新 token 的 K/V |

在 scheduler 看来，它们可以统一成：

```text
本轮为某个请求计算一段 token
```

区别只是：

```text
prefill: token span 长度可能很大
decode: token span 长度通常是 1
```

如果引入 chunked prefill，prefill 也会被切成多个小 span：

```text
prefill chunk 0: tokens [0:1024]
prefill chunk 1: tokens [1024:2048]
prefill chunk 2: tokens [2048:3072]
```

于是 prefill 和 decode 可以进入同一个 batch 计划：

```text
req A: prefill tokens [1024:2048]
req B: decode token [17]
req C: decode token [56]
```

这就是 continuous batching 和 chunked prefill 的基础。

## 47.16 状态机视角

一个请求在 engine 内部可以用状态机表示：

```text
WAITING
  |
  v
PREFILLING
  |
  v
DECODING
  |
  +--> FINISHED
  +--> ABORTED
  +--> ERROR
```

如果支持 chunked prefill：

```text
WAITING
  |
  v
PREFILLING_CHUNK
  |
  +--> PREFILLING_CHUNK
  |
  v
DECODING
```

如果支持 prefix cache：

```text
WAITING
  |
  v
PREFIX_MATCHING
  |
  v
PREFILLING_UNCACHED_SUFFIX
  |
  v
DECODING
```

如果支持 streaming abort：

```text
DECODING
  |
  v
ABORTED
  |
  v
FREE_KV_AND_CLOSE_STREAM
```

状态机的价值是让你明确：

1. 哪些状态会占用 KV cache。
2. 哪些状态可以被 scheduler 选择。
3. 哪些状态需要输出 token。
4. 哪些状态需要释放资源。
5. 哪些状态可以转移到 finish。

很多推理框架 bug，本质都是状态机没设计清楚。

## 47.17 Metrics：没有观测就没有优化

推理框架项目一定要内置 metrics。

否则你无法证明自己的改造有意义。

最小 metrics 包括：

| 指标 | 含义 |
|---|---|
| QPS | 每秒完成请求数 |
| output TPS | 每秒输出 token 数 |
| TTFT | time to first token |
| TPOT | time per output token |
| latency | 请求总延迟 |
| running requests | 当前运行请求数 |
| waiting requests | 当前排队请求数 |
| KV usage | KV cache 使用率 |
| prefill tokens/s | prefill 吞吐 |
| decode tokens/s | decode 吞吐 |

更进一步可以统计：

1. p50/p90/p99 TTFT。
2. p50/p90/p99 TPOT。
3. scheduler 每轮 batch size。
4. 每轮 prefill/decode token 数。
5. OOM 次数。
6. abort 次数。
7. prefix cache hit length。
8. GPU utilization。

后面每做一次升级，都应该用 metrics 证明：

```text
这个改造改善了什么，牺牲了什么？
```

例如：

1. continuous batching 通常提升吞吐，但可能增加单请求延迟波动。
2. chunked prefill 通常降低 decode 抖动，但可能增加长 prompt 的 TTFT。
3. paged KV cache 通常提升显存利用率，但会引入 block table 管理复杂度。
4. streaming engine 提升用户感知体验，但增加异步状态管理难度。

## 47.18 模块边界的反模式

写 mini engine 时，最常见的坏设计有几种。

反模式一：scheduler 直接调用模型。

```text
问题：调度策略和模型执行耦合，后面无法替换 model runner 或做异步执行。
```

反模式二：model runner 修改 request 状态。

```text
问题：模型层知道太多业务状态，streaming、abort、finish 逻辑会散落各处。
```

反模式三：KV cache 直接挂在 request 的 tensor list 上。

```text
问题：后面升级 paged KV cache、prefix cache、eviction 都要大改。
```

反模式四：sampler 只支持全 batch 同一参数。

```text
问题：在线 serving 中每个请求的 temperature、top_p、max_tokens 都可能不同。
```

反模式五：没有 request id 和状态机。

```text
问题：debug 时无法追踪一个请求在 waiting、prefill、decode、finish 之间如何变化。
```

反模式六：没有 metrics。

```text
问题：所有优化都只能凭感觉，无法比较 TTFT、TPOT、吞吐和显存。
```

反模式七：一开始就设计生产级大框架。

```text
问题：还没跑通 prefill/decode，就引入多模型、分布式、grammar、spec decode，复杂度失控。
```

正确做法是：

```text
先用清晰模块边界跑通最小闭环，再逐个替换内部实现。
```

## 47.19 推荐的第一版 mini engine

如果你要从 0 做一个项目，第一版可以这样定范围。

支持：

1. 单模型。
2. 单进程。
3. offline generate API。
4. 多请求 batch。
5. 区分 prefill 和 decode。
6. naive waiting/running scheduler。
7. 简单 KV cache。
8. greedy sampling。
9. TTFT、TPOT、latency 统计。

暂不支持：

1. HTTP server。
2. streaming。
3. paged attention。
4. prefix cache。
5. tensor parallel。
6. structured generation。
7. tool call。
8. speculative decoding。
9. 多模型。
10. 分布式。

第一版的目标不是性能最好，而是结构正确。

可以把项目目录设计成：

```text

  request.py
  scheduler.py
  kv_cache.py
  batch.py
  model_runner.py
  sampler.py
  output.py
  metrics.py
  benchmark.py
```

每个文件只做一件事。

## 47.20 第一版执行流程

一个 offline generate 可以这样执行：

```text
1. 用户提交 prompts。
2. Tokenizer 把 prompt 转成 prompt_tokens。
3. 为每个 prompt 创建 Request。
4. Request 加入 scheduler waiting 队列。
5. Engine loop 开始运行。
6. Scheduler 选择 prefill 请求。
7. KVCacheManager 分配 KV 空间。
8. BatchBuilder 构造 prefill 输入。
9. ModelRunner 执行 prefill forward。
10. Sampler 采样第一个 token。
11. OutputProcessor 更新 output_tokens 和 TTFT。
12. Scheduler 下一轮选择 decode 请求。
13. ModelRunner 执行 decode forward。
14. Sampler 采样下一个 token。
15. OutputProcessor 判断 EOS 或 max_tokens。
16. 请求完成后释放 KV cache。
17. 所有请求完成后返回文本和 metrics。
```

注意这里有一个细节：

```text
prefill 之后通常会直接采样第一个输出 token。
```

也就是说，prefill 不只是写 prompt KV，它还会产生 prompt 最后一个位置的 logits，用来采样第一个生成 token。

decode 则用上一次生成的 token 作为输入，采样下一 token。

## 47.21 从第一版到后续章节的升级路线

第 47 章得到的是骨架。

后面几章会在这套骨架上替换模块。

| 后续章节 | 替换或增强的模块 | 核心变化 |
|---|---|---|
| 第 48 章 | `Scheduler` | naive batch -> continuous batching |
| 第 49 章 | `KVCacheManager`、`BatchBuilder` | list KV -> paged KV / block table |
| 第 50 章 | `Engine`、`OutputProcessor`、`API` | synchronous generate -> async streaming |
| 第 51 章 | `RequestManager`、`ModelRunner` | single model -> multi model routing |
| 第 52 章 | `ModelRunner`、`KVCacheManager`、`Scheduler` | single node -> distributed serving |

这个顺序很重要。

不要一开始就做 streaming。

不要一开始就做分布式。

不要一开始就做 prefix cache。

原因是这些能力都会依赖前面的基础模块：

1. 没有清晰 request state，就做不好 streaming abort。
2. 没有 BatchBuilder，就做不好 paged attention metadata。
3. 没有 token budget，就做不好 chunked prefill。
4. 没有 KV manager，就做不好 prefix cache 和 eviction。
5. 没有 metrics，就不知道升级是否有效。

## 47.23 面试高频问题

问题一：一个 LLM 推理框架最核心的模块有哪些？

回答要点：API/request manager、request state、scheduler、KV cache manager、batch builder、model runner、sampler、output processor、metrics。在线服务还需要 request queue、streaming channel、abort/error handling。

问题二：Scheduler 和 ModelRunner 的边界是什么？

回答要点：Scheduler 决定本轮执行哪些请求、每个请求执行多少 token，并受 token budget 和 KV 容量约束；ModelRunner 只负责根据 input ids、positions、block table、attention metadata 执行 forward，返回 logits。调度策略不应塞进模型层。

问题三：为什么需要 BatchBuilder？

回答要点：Scheduler 输出的是逻辑 batch plan，ModelRunner 需要物理 tensor 和 metadata。BatchBuilder 负责拼 input ids、positions、seq lens、slot mapping、block table、attention metadata，是 continuous batching、chunked prefill、paged attention、prefix cache 的胶水层。

问题四：prefill 和 decode 有什么区别？

回答要点：prefill 输入完整 prompt 或 prompt chunk，计算量大，写入一段 K/V，并用最后位置 logits 采样首 token；decode 每轮通常输入一个新 token，追加 K/V，并采样下一个 token。prefill 影响 TTFT，decode 影响 TPOT。

问题五：为什么第一版也要做 metrics？

回答要点：没有 metrics 就无法验证优化效果。至少要统计 TTFT、TPOT、latency、output TPS、waiting/running 请求数、KV 使用率。后续 continuous batching、chunked prefill、paged KV cache 都需要这些指标做对比。

问题六：第一版 mini engine 为什么不建议直接做分布式或 streaming？

回答要点：这些能力依赖稳定的 request state、scheduler、KV manager、output processor 和 metrics。先跑通单机 offline engine，可以把模块边界定清楚；后面再升级 streaming、paged KV、多模型、分布式，改动会更可控。

## 47.24 标准回答模板

如果面试官问“你会如何设计一个最小 LLM 推理框架”，可以这样回答：

```text
我会先把框架拆成几个核心模块：Request 状态、Scheduler、KVCacheManager、BatchBuilder、ModelRunner、Sampler、OutputProcessor 和 Metrics。API 层负责接收请求并 tokenization，Scheduler 维护 waiting/running 队列，决定每一轮哪些请求进入 batch；KVCacheManager 负责为请求分配和释放 KV 空间；BatchBuilder 把 scheduler 的逻辑计划转换成模型需要的 input_ids、positions、block table、slot mapping 和 attention metadata；ModelRunner 只负责执行 forward 返回 logits；Sampler 根据每个请求的 sampling params 采样 token；OutputProcessor 更新请求状态、检查 EOS/max_tokens/stop 条件并记录 TTFT、TPOT、latency。

第一版我会只支持单机单模型 offline generate，先跑通 prefill 和 decode。prefill 处理 prompt token，写入 prompt 的 KV，并用最后位置 logits 采样首 token；decode 每轮处理上一轮生成的 token，追加 KV 并采样下一 token。Scheduler 先用简单 waiting/running 队列和 token budget，避免长 prompt 无限制阻塞 decode。

这个设计的重点是模块边界，而不是第一版性能最好。后续可以逐步把 naive scheduler 升级为 continuous batching，把简单 KV cache 升级为 paged KV cache，把同步 generate 升级为异步 streaming，把单模型升级为多模型或分布式 serving。只要 Request 状态机、BatchBuilder、KVCacheManager 和 Metrics 设计清楚，后续升级就不需要推倒重来。
```

## 47.25 核心模块边界、状态 ownership 和可运行 demo

先把核心模块抽象成：

```math
M_j=(r_j,I_j,O_j,S_j,D_j)
```

其中 `r_j` 是职责，`I_j` 是输入集合，`O_j` 是输出集合，`S_j` 是该模块拥有的状态，`D_j` 是依赖模块。

模块覆盖率：

```math
C_{\mathrm{module}}=\frac{|M_{\mathrm{seen}}\cap M_{\mathrm{req}}|}{\max(1,|M_{\mathrm{req}}|)}
```

状态覆盖率：

```math
C_{\mathrm{state}}=\frac{|S_{\mathrm{seen}}\cap S_{\mathrm{req}}|}{\max(1,|S_{\mathrm{req}}|)}
```

边界覆盖率：

```math
C_{\mathrm{boundary}}=\frac{|B_{\mathrm{pass}}|}{\max(1,|B_{\mathrm{req}}|)}
```

最终门禁：

```math
G_{\mathrm{core}}=G_{\mathrm{request}}G_{\mathrm{scheduler}}G_{\mathrm{kv}}G_{\mathrm{batch}}G_{\mathrm{runner}}G_{\mathrm{output}}G_{\mathrm{metric}}G_{\mathrm{upgrade}}
```

这组公式强调：第一版 mini engine 的关键不是功能多，而是模块边界稳。下面的 demo 不执行真实模型，而是检查设计图本身是否可升级。

```python
from dataclasses import dataclass


@dataclass
class CoreModule:
    name: str
    responsibility: str
    inputs: tuple
    outputs: tuple
    owns_state: tuple
    depends_on: tuple
    replaceable: bool


class ToyCoreModuleAuditor:
    def __init__(self):
        self.required_modules = [
            "request",
            "scheduler",
            "kv_cache_manager",
            "batch_builder",
            "model_runner",
            "sampler",
            "output_processor",
            "engine_loop",
            "metrics",
        ]
        self.required_states = [
            "request_status",
            "waiting_queue",
            "running_set",
            "kv_blocks",
            "batch_metadata",
            "logits",
            "output_tokens",
            "ttft_tpot_latency",
        ]
        self.required_boundaries = [
            "scheduler_not_model_runner",
            "batch_builder_owns_metadata",
            "model_runner_no_request_mutation",
            "output_processor_finishes_requests",
            "kv_manager_owns_blocks",
            "metrics_observe_all_steps",
        ]

    def coverage(self, seen, required):
        return round(len(set(seen) & set(required)) / max(1, len(required)), 3)

    def audit(self, modules):
        names = [module.name for module in modules]
        states = sorted({state for module in modules for state in module.owns_state})
        outputs = sorted({item for module in modules for item in module.outputs})
        completed = set()
        rows = []
        for module in modules:
            deps_ready = all(dep in completed for dep in module.depends_on)
            row = {
                "module": module.name,
                "responsibility": module.responsibility,
                "deps_ready": deps_ready,
                "replaceable": module.replaceable,
                "owns_state": list(module.owns_state),
            }
            rows.append(row)
            if deps_ready:
                completed.add(module.name)
        boundary_checks = {
            "scheduler_not_model_runner": "logits" not in next(
                module.owns_state for module in modules if module.name == "scheduler"
            ),
            "batch_builder_owns_metadata": "batch_metadata" in next(
                module.owns_state for module in modules if module.name == "batch_builder"
            ),
            "model_runner_no_request_mutation": "request_status" not in next(
                module.owns_state for module in modules if module.name == "model_runner"
            ),
            "output_processor_finishes_requests": "request_status" in next(
                module.owns_state for module in modules if module.name == "output_processor"
            ),
            "kv_manager_owns_blocks": "kv_blocks" in next(
                module.owns_state for module in modules if module.name == "kv_cache_manager"
            ),
            "metrics_observe_all_steps": "ttft_tpot_latency" in next(
                module.owns_state for module in modules if module.name == "metrics"
            ),
        }
        summary = {
            "modules": len(rows),
            "ready_modules": sum(row["deps_ready"] for row in rows),
            "module_coverage": self.coverage(names, self.required_modules),
            "state_coverage": self.coverage(states, self.required_states),
            "boundary_coverage": self.coverage(
                [name for name, ok in boundary_checks.items() if ok],
                self.required_boundaries,
            ),
            "replaceable_modules": sum(module.replaceable for module in modules),
            "engine_outputs": outputs,
            "missing_modules": sorted(set(self.required_modules) - set(names)),
            "missing_states": sorted(set(self.required_states) - set(states)),
        }
        gates = {
            "request_state_visible": "request_status" in states,
            "scheduler_boundary_clear": boundary_checks["scheduler_not_model_runner"],
            "kv_manager_boundary_clear": boundary_checks["kv_manager_owns_blocks"],
            "batch_builder_boundary_clear": boundary_checks["batch_builder_owns_metadata"],
            "model_runner_boundary_clear": boundary_checks["model_runner_no_request_mutation"],
            "output_processor_boundary_clear": boundary_checks["output_processor_finishes_requests"],
            "metrics_ready": boundary_checks["metrics_observe_all_steps"],
            "upgrade_path_ready": summary["replaceable_modules"] >= 6,
        }
        gates["core_module_gate"] = all(gates.values())
        return rows, summary, gates


modules = [
    CoreModule(
        "request",
        "hold per-request inputs and execution state",
        ("prompt", "sampling_params"),
        ("prompt_tokens",),
        ("request_status", "output_tokens"),
        (),
        False,
    ),
    CoreModule(
        "scheduler",
        "select prefill and decode work under token and KV budgets",
        ("waiting_queue", "running_set", "kv_free"),
        ("batch_plan",),
        ("waiting_queue", "running_set"),
        ("request",),
        True,
    ),
    CoreModule(
        "kv_cache_manager",
        "allocate append query and free KV blocks",
        ("batch_plan",),
        ("block_table", "kv_free"),
        ("kv_blocks",),
        ("request",),
        True,
    ),
    CoreModule(
        "batch_builder",
        "convert logical batch plan into model runner metadata",
        ("batch_plan", "block_table"),
        ("model_inputs",),
        ("batch_metadata",),
        ("scheduler", "kv_cache_manager"),
        True,
    ),
    CoreModule(
        "model_runner",
        "run model forward and return logits",
        ("model_inputs",),
        ("logits",),
        ("logits",),
        ("batch_builder",),
        True,
    ),
    CoreModule(
        "sampler",
        "choose next token per request",
        ("logits", "sampling_params"),
        ("next_tokens",),
        (),
        ("model_runner",),
        True,
    ),
    CoreModule(
        "output_processor",
        "append tokens finish requests and trigger cleanup",
        ("next_tokens", "batch_plan"),
        ("finished_ids", "stream_chunks"),
        ("request_status", "output_tokens"),
        ("sampler",),
        True,
    ),
    CoreModule(
        "metrics",
        "record queue wait TTFT TPOT latency throughput and KV usage",
        ("batch_plan", "finished_ids"),
        ("metric_report",),
        ("ttft_tpot_latency",),
        ("output_processor",),
        True,
    ),
    CoreModule(
        "engine_loop",
        "orchestrate schedule build forward sample output cleanup",
        ("requests",),
        ("responses", "metric_report"),
        (),
        ("metrics",),
        False,
    ),
]
rows, summary, gates = ToyCoreModuleAuditor().audit(modules)
print("core_module_summary=", summary)
```

一次运行的核心输出类似：

```text
core_module_summary= {'modules': 9, 'ready_modules': 9, 'module_coverage': 1.0, 'state_coverage': 1.0, 'boundary_coverage': 1.0, 'replaceable_modules': 7, 'engine_outputs': ['batch_plan', 'block_table', 'finished_ids', 'kv_free', 'logits', 'metric_report', 'model_inputs', 'next_tokens', 'prompt_tokens', 'responses', 'stream_chunks'], 'missing_modules': [], 'missing_states': []}
core_module_gates= {'request_state_visible': True, 'scheduler_boundary_clear': True, 'kv_manager_boundary_clear': True, 'batch_builder_boundary_clear': True, 'model_runner_boundary_clear': True, 'output_processor_boundary_clear': True, 'metrics_ready': True, 'upgrade_path_ready': True, 'core_module_gate': True}
```

这个 demo 展示了第一版 mini engine 的几个硬边界：

1. Scheduler 只输出 `batch_plan`，不拥有 logits，也不直接跑模型。
2. KVCacheManager 拥有 `kv_blocks`，Request 不直接挂一串 tensor list。
3. BatchBuilder 拥有 `batch_metadata`，负责把逻辑 plan 变成物理输入。
4. ModelRunner 只返回 `logits`，不修改 `request_status`。
5. OutputProcessor 负责 append token、finish request 和触发 cleanup。
6. Metrics 贯穿全链路，至少能记录 TTFT、TPOT 和 latency。
7. 至少 6 个模块可替换，说明后续 continuous batching、paged KV、streaming、multi-model 和 distributed serving 有升级空间。

所以本章最终门禁是 `core_module_gate`：只有模块、状态、边界、依赖和可替换路径都清楚，所谓“从教学项目抽象自己的推理框架”才不是画一张大图，而是可以落到后续章节逐步替换的工程骨架。

## 47.27 本章总结

读完 tiny-llm、nano-vLLM 和 mini-sglang 后，下一步不是继续堆更多项目，而是抽象出推理框架的共同模块。

一个最小 LLM serving engine 至少需要 Request、Scheduler、KVCacheManager、BatchBuilder、ModelRunner、Sampler、OutputProcessor、Engine Loop 和 Metrics。

其中最关键的边界是：Scheduler 负责决策，BatchBuilder 负责把逻辑计划转成物理输入，ModelRunner 负责 forward，Sampler 负责选 token，OutputProcessor 负责更新请求状态和输出，KVCacheManager 负责显存状态。

第一版不要追求生产级能力，而要先跑通单机、单模型、offline、多请求、prefill/decode、KV cache 和 metrics。

有了这个骨架，后续才能有控制地升级 scheduler、KV cache、streaming、多模型和分布式 serving。
