# 第 6 章 推理框架、推理平台和 AI Infra 的边界

前五章讲的是 LLM serving engine 内部：请求生命周期、prefill、decode、KV Cache、streaming 和性能指标。本章把视角拉高，讲清三个容易混在一起的概念：推理框架、推理平台和 AI Infra。

很多团队讨论大模型推理时，会把 vLLM、Kubernetes、模型发布、灰度、监控、GPU 调度、限流、API 网关都混在一起说。这样会导致系统边界不清，设计时不知道问题应该在哪一层解决。

一句话概括：

> 推理框架负责把模型高效执行起来，推理平台负责把模型服务稳定运营起来，AI Infra 负责提供算力、存储、网络、调度和治理等基础能力。

## 6.0 本讲资料边界与第二轮精修口径

本章第二轮精修前，先用公开资料校准边界口径：vLLM 等 LLM serving engine 的公开架构通常围绕 API 入口、engine core、scheduler、KV Cache manager 和 worker 执行路径展开；Triton Inference Server 这类通用推理服务强调 model repository、scheduler、backend、metrics 和 tracing；Kubernetes 文档把资源 request / limit、Pod 调度和扩展资源管理放在集群资源治理层；SRE 体系把 SLO、error budget、容量和告警作为服务运营能力。

因此，本章只抽象稳定边界，不把某个框架版本的内部类名、Prometheus 指标名、Kubernetes 字段或云厂商产品写成标准答案。读者需要掌握的是：engine 负责 token 执行路径，platform 负责服务治理控制面，AI Infra 负责底层资源和运行环境；三层之间通过请求、指标、控制、健康检查和 trace 接口协作。

第二轮补充重点有三点：

1. 把“谁负责什么”写成可检查的职责向量和覆盖率公式。
2. 把平台和 engine、infra 之间的接口、指标交接和事故归层写成可验收门禁。
3. 增加一个 0 依赖 Python demo，让读者能审计 toy 推理系统是否把职责、接口、指标、事故和成本边界说清楚。

## 6.1 为什么边界重要

边界不清会带来很多工程问题。

例如，线上 TTFT 变差，可能原因很多：

1. engine scheduler 策略有问题。
2. KV Cache 不足导致请求等待。
3. 平台限流或网关排队。
4. Kubernetes 调度导致实例不足。
5. GPU 节点性能异常。
6. 上游请求突然变长。

如果不区分层次，就会出现错误修复方向：明明是平台扩容慢，却去调 vLLM 参数；明明是 engine KV Cache 不够，却去改 API 网关；明明是模型路由不合理，却去升级 GPU 驱动。

系统边界能帮助我们明确：

1. 谁负责模型执行。
2. 谁负责服务治理。
3. 谁负责资源供给。
4. 谁负责观测和排障。
5. 谁负责成本和容量。

## 6.2 三层结构总览

可以把大模型在线推理系统简化成三层：

```text
业务应用 / 产品
        |
推理平台 Inference Platform
        |
推理框架 Serving Engine
        |
AI Infra / GPU Infrastructure
```

更具体一些：

```text
Inference Platform:
  API gateway, auth, quota, routing, rollout, billing, observability

Serving Engine:
  request queue, scheduler, prefill/decode, KV cache, sampling, streaming

AI Infra:
  GPU cluster, Kubernetes, network, storage, image, driver, monitoring
```

三者不是完全隔离的。平台需要读取 engine 指标，engine 依赖 infra 提供 GPU，infra 的资源约束会影响 engine 调度。但职责边界要清楚。

## 6.3 推理框架负责什么

推理框架，也就是 serving engine，负责模型执行路径。

它的核心职责包括：

1. 加载模型权重。
2. 管理 tokenizer 和模型配置。
3. 接收内部格式的生成请求。
4. 维护 waiting queue 和 running set。
5. 执行 prefill 和 decode。
6. 管理 KV Cache。
7. 实现 continuous batching。
8. 执行 sampling。
9. 支持 token streaming。
10. 暴露性能指标。

典型项目包括 vLLM、SGLang、TGI，以及一些基于 TensorRT-LLM、Triton、自研 runtime 的 engine。

推理框架关注的问题是：

1. 如何提高 tokens/s。
2. 如何降低 TTFT 和 TPOT。
3. 如何减少 KV Cache 碎片。
4. 如何支持长上下文。
5. 如何实现高效 sampling 和 structured generation。
6. 如何让 GPU 在多请求下保持高效执行。

## 6.4 推理框架不应该负责什么

推理框架不应该承担所有平台治理能力。

例如：

1. 全局租户计费。
2. 跨集群容量规划。
3. 复杂灰度发布策略。
4. 多模型业务路由。
5. 企业权限系统。
6. 长期审计日志留存。
7. GPU 集群资源采购和配额。

这些能力放进 engine 会让 engine 变得臃肿，也会让执行路径不稳定。

好的 engine 应该把关键指标和控制接口暴露出来，让平台层使用，而不是自己变成一个完整业务平台。

当然，一些开源 engine 会内置 HTTP server、OpenAI-compatible API、基础 metrics。这些是为了易用性，不代表它应该承担全部平台职责。

## 6.5 推理平台负责什么

推理平台负责把模型服务稳定地对外提供。

它通常位于业务应用和 serving engine 之间。

核心职责包括：

1. API 入口和协议适配。
2. 鉴权、租户、配额和限流。
3. 模型注册和版本管理。
4. 模型路由和负载均衡。
5. 灰度发布、回滚和 A/B 实验。
6. 多模型、多实例、多区域管理。
7. 统一日志、metrics 和 trace。
8. 成本统计和计费。
9. SLO 管理和告警。
10. 降级、熔断和重试策略。

如果说 engine 关注“一个模型实例怎么跑得快”，平台关注的是“一组模型服务怎么稳定、可控、可治理地对外提供”。

## 6.6 推理平台和 engine 的接口

平台和 engine 之间通常通过几个接口协作。

请求接口：

```text
platform -> engine: generate / chat / embeddings / rerank
```

指标接口：

```text
engine -> platform: TTFT, TPOT, tokens/s, queue length, KV cache usage, errors
```

控制接口：

```text
platform -> engine: drain, reload, pause admission, update config
```

健康检查接口：

```text
platform -> engine: readiness, liveness, warmup status
```

这些接口决定了平台能否正确治理 engine。

例如，平台做扩缩容时，如果只看 GPU 利用率，可能不够；还需要看 queue length、KV Cache 使用率、TTFT p95、TPOT p95。平台做灰度发布时，需要 engine 支持优雅下线，避免正在 decode 的请求被直接杀掉。

## 6.7 AI Infra 负责什么

AI Infra 是更底层的基础设施层，负责算力、存储、网络、调度和运行环境。

核心职责包括：

1. GPU 节点管理。
2. Kubernetes 或其他集群调度。
3. 容器镜像和运行时。
4. NVIDIA driver、CUDA、NCCL 等依赖。
5. 模型权重存储和分发。
6. 网络和负载均衡基础能力。
7. 分布式存储、对象存储和缓存。
8. 节点监控和硬件故障处理。
9. 多租户资源隔离。
10. 成本、配额和容量规划。

AI Infra 更关心：GPU 有没有、放在哪里、怎么调度、怎么隔离、怎么监控、坏了怎么恢复、成本怎么控制。

第二十三册已经系统讲过 AI Infra。本书只在 serving engine 需要时引用它。

## 6.8 三层各自的指标

不同层关注的指标不同。

Serving Engine 指标：

1. TTFT。
2. TPOT。
3. prefill latency。
4. decode iteration latency。
5. tokens/s。
6. active sequences。
7. KV Cache usage。
8. scheduler queue length。

Inference Platform 指标：

1. 请求成功率。
2. p95/p99 端到端延迟。
3. 每模型 QPS 和 tokens/s。
4. 租户配额使用。
5. 路由命中率。
6. 灰度版本错误率。
7. 限流和降级次数。
8. 每百万 token 成本。

AI Infra 指标：

1. GPU 利用率。
2. GPU 显存使用率。
3. 节点健康状态。
4. 网络带宽和丢包。
5. 存储读取延迟。
6. Pod 调度等待时间。
7. 镜像拉取时间。
8. 集群资源碎片率。

排障时，要把指标放回对应层次，而不是混在一起。

## 6.9 一个例子：TTFT 变差怎么分层排查

假设线上 TTFT p99 从 2 秒变成 15 秒。

先看平台层：

1. 请求量是否突增。
2. 输入 token 长度是否变长。
3. 是否有某个租户打满配额。
4. 路由是否集中到少数实例。
5. 是否刚做灰度发布。

再看 engine 层：

1. waiting queue 是否变长。
2. prefill latency 是否升高。
3. KV Cache 是否接近满。
4. scheduler 是否过度优先 decode。
5. prefix cache 命中率是否下降。

最后看 infra 层：

1. GPU 是否降频或异常。
2. 节点是否有硬件错误。
3. 网络是否拥塞。
4. Pod 是否频繁重启。
5. 是否扩容失败或调度等待过长。

这样排查比“感觉 vLLM 慢了”有效得多。

## 6.10 一个例子：成本过高怎么分层优化

假设业务反馈每百万 token 成本太高。

平台层可以做：

1. 模型路由，把简单请求路由到小模型。
2. 结果缓存或语义缓存。
3. 限制不合理的超长输出。
4. 按租户和业务做成本报表。
5. 非实时请求转批处理或低优先级队列。

Engine 层可以做：

1. continuous batching 参数调优。
2. KV Cache block size 调优。
3. prefix cache。
4. 量化推理。
5. speculative decoding。
6. 更高效 kernel。

Infra 层可以做：

1. 选择更合适的 GPU 型号。
2. 提升资源利用率。
3. 混部低优先级任务。
4. 优化镜像和模型加载时间。
5. 做容量规划和弹性伸缩。

成本优化通常不是单层能解决的，需要平台、engine 和 infra 协同。

## 6.11 常见架构反模式

反模式一：把所有治理能力塞进 engine。

结果是 engine 越来越复杂，执行路径变重，升级困难。

反模式二：平台完全不理解 engine 指标。

结果是扩缩容、路由和限流都只能看粗粒度 QPS，无法针对 LLM serving 的真实瓶颈做决策。

反模式三：Infra 只提供 GPU，不关心模型服务特征。

结果是调度和容量规划不适配长上下文、KV Cache、模型加载和多租户隔离。

反模式四：用普通 HTTP 服务思路设计 LLM serving。

结果是忽略 token 长度、流式输出、KV Cache、动态 batch 和长尾延迟。

反模式五：所有模型共用一套固定参数。

不同模型大小、上下文长度、业务 SLO 和流量分布不同，engine 参数和平台策略也应该不同。

## 6.12 面试中如何讲边界

如果面试官让你“设计一个大模型推理服务平台”，不要一上来只说 vLLM 部署。

可以按三层回答：

1. Serving Engine：负责模型执行，包括 prefill、decode、KV Cache、scheduler、streaming、metrics。
2. Inference Platform：负责接入、鉴权、路由、限流、灰度、观测、成本和 SLO。
3. AI Infra：负责 GPU 集群、容器、调度、存储、网络、驱动和资源隔离。

然后说明三层接口：

1. 请求接口。
2. 指标接口。
3. 控制接口。
4. 健康检查接口。

最后给出排障和优化例子，例如 TTFT 变差如何分层排查，成本过高如何分层优化。

这样的回答能体现你不仅懂框架，还懂生产系统边界。

## 6.14 常见误区

误区一：会部署 vLLM 就等于懂推理平台。

部署只是开始。平台还要处理路由、灰度、限流、观测、成本、SLO 和多租户。

误区二：推理框架负责所有线上稳定性。

engine 负责执行稳定性，但平台治理和 infra 稳定性同样关键。

误区三：AI Infra 只要给 GPU 就行。

LLM serving 对显存、网络、模型加载、驱动、调度和隔离都有特殊要求。

误区四：平台层不需要理解 prefill 和 decode。

平台如果不理解这些指标，就无法做精细扩缩容、路由和 SLO 管理。

误区五：边界越清楚，系统越割裂。

正确做法是职责清楚、接口清楚、指标打通，而不是各层互不理解。

## 6.15 边界职责、接口契约和可运行 demo

可以把一个推理系统的边界对象写成：

```math
B_i=(\ell_i,r_i,I_i,M_i,K_i,S_i)
```

其中，$\ell_i$ 是所属层级，$r_i$ 是职责集合，$I_i$ 是对外接口集合，$M_i$ 是指标集合，$K_i$ 是成本和容量证据，$S_i$ 是事故排查证据。这个式子的作用不是追求形式化，而是提醒你：一个模块不能只说“我是 engine / platform / infra”，还要说明它暴露什么接口、交接什么指标、承担什么成本容量责任、事故时留下什么证据。

层级职责覆盖率可以写成：

```math
C_{\mathrm{layer}}=\frac{N_{\mathrm{covered}}}{N_{\mathrm{required}}}
```

$N_{\mathrm{required}}$ 是三层系统应该覆盖的关键职责数，$N_{\mathrm{covered}}$ 是当前设计中有明确 owner、输入、输出和验收方式的职责数。只画了架构图但没有 owner 和验收指标，不应该算 covered。

平台和 engine / infra 的接口契约覆盖率可以写成：

```math
C_{\mathrm{iface}}=\frac{N_{\mathrm{iface,ready}}}{N_{\mathrm{iface,total}}}
```

常见接口包括 request API、metrics export、control drain、health probe 和 trace context。平台要做灰度、扩容和故障切流，不能只知道一个 HTTP endpoint，还要知道 engine 是否能优雅 drain，metrics 是否能解释 TTFT / TPOT / KV pressure，健康检查是否能区分 warmup、ready 和 overload。

指标交接覆盖率可以写成：

```math
C_{\mathrm{metric}}=\frac{N_{\mathrm{metric,handoff}}}{N_{\mathrm{metric,required}}}
```

例如 engine 要向 platform 交接 TTFT、TPOT、queue length、KV pressure、tokens/s；platform 要向 engine 交接 tenant、priority、deadline、token budget；infra 要向 platform / engine 交接 GPU health、Pod pending、memory pressure、image pull 和 node readiness。指标没有交接，排障时就只能猜。

错误归层率可以写成：

```math
R_{\mathrm{misroute}}=\frac{N_{\mathrm{misrouted}}}{N_{\mathrm{incidents}}}
```

如果 TTFT p99 变差时只怀疑 engine，却没有同时检查平台流量分布、prompt 长度、扩容状态、Pod 调度和 GPU 健康，就属于高风险误归层。

最后可以把边界验收写成一个乘法门禁：

```math
G_{\mathrm{boundary}}=G_{\mathrm{role}}G_{\mathrm{iface}}G_{\mathrm{metric}}G_{\mathrm{incident}}G_{\mathrm{cost}}
```

任何一个门禁为 0，说明三层边界还不能支撑真实生产排障。面试中这比背工具名更有说服力。

```python
class MiniServingBoundaryAudit:
    def __init__(self, capabilities, incidents):
        self.capabilities = capabilities
        self.incidents = incidents

    def audit(self):
        responsibilities = {layer: set() for layer in REQUIRED_BY_LAYER}
        interfaces = set()
        metrics = set()
        layer_counts = Counter()

        for capability in self.capabilities:
            layer = capability["layer"]
            layer_counts[layer] += 1
            responsibilities[layer].update(capability["responsibilities"])
            interfaces.update(capability["interfaces"])
            metrics.update(capability["metrics"])

        required_responsibilities = set().union(*REQUIRED_BY_LAYER.values())
        covered_responsibilities = set().union(*responsibilities.values())

        missing_by_layer = {
            layer: sorted(required - responsibilities[layer])
            for layer, required in REQUIRED_BY_LAYER.items()
        }
        misrouted = [
            item["id"]
            for item in self.incidents
            if not item["root_layers"].issubset(item["investigated_layers"])
        ]

        metrics_summary = {
            "layer_coverage": ratio(covered_responsibilities, required_responsibilities),
            "interface_coverage": ratio(interfaces, REQUIRED_INTERFACES),
            "metric_handoff_coverage": ratio(metrics, REQUIRED_METRICS),
            "misroute_rate": round(len(misrouted) / len(self.incidents), 3),
        }

        gates = {
            "role": metrics_summary["layer_coverage"] >= 0.95 and all(not v for v in missing_by_layer.values()),
            "interface": metrics_summary["interface_coverage"] >= 1.0,
            "metric": metrics_summary["metric_handoff_coverage"] >= 0.95,
            "incident": metrics_summary["misroute_rate"] == 0.0,
            "cost": "cost_per_1k_tokens" in metrics,
        }

        return {
            "layer_counts": dict(layer_counts),
            "metrics_summary": metrics_summary,
            "missing_by_layer": missing_by_layer,
            "misrouted_incidents": misrouted,
            "gates": gates,
            "boundary_gate": all(gates.values()),
        }


report = MiniServingBoundaryAudit(CAPABILITIES, INCIDENTS).audit()
print("layer_counts=", report["layer_counts"])
print("metrics_summary=", report["metrics_summary"])
print("missing_by_layer=", report["missing_by_layer"])
print("misrouted_incidents=", report["misrouted_incidents"])
print("gates=", report["gates"])
print("boundary_gate=", report["boundary_gate"])
```

一组典型输出如下：

```text
layer_counts= {'engine': 1, 'platform': 1, 'infra': 1}
metrics_summary= {'layer_coverage': 1.0, 'interface_coverage': 1.0, 'metric_handoff_coverage': 1.0, 'misroute_rate': 0.333}
missing_by_layer= {'engine': [], 'platform': [], 'infra': []}
misrouted_incidents= ['ttft_p99_high_bad']
gates= {'role': True, 'interface': True, 'metric': True, 'incident': False, 'cost': True}
boundary_gate= False
```

这个 demo 的解释是：三层职责、接口和指标都覆盖了，但有一个事故只查 engine，没有查真实 root layers 里的 platform 和 infra，所以 `incident` 门禁失败，最终 `boundary_gate=False`。生产系统里，边界不是为了互相甩锅，而是为了让每个故障都能按证据走到正确层级。

## 6.18 本章小结

本章讲清了推理框架、推理平台和 AI Infra 的边界。

推理框架负责模型执行，包括 scheduler、prefill、decode、KV Cache、sampling 和 streaming。推理平台负责服务治理，包括接入、鉴权、路由、限流、灰度、观测、成本和 SLO。AI Infra 负责底层资源，包括 GPU 集群、容器、调度、存储、网络、驱动和隔离。

边界清楚不是为了割裂系统，而是为了让职责、接口、指标和排障路径更清楚。下一章开始进入第二部分，我们会从 0 实现最小 tokenizer、model wrapper 和 generate loop，正式开始动手搭一个迷你推理框架。
