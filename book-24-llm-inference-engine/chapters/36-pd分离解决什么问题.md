# 第 36 章 PD 分离解决什么问题

上一章讲了 Prefill 和 Decode 的资源画像为什么不同：Prefill 更偏大块输入计算和 KV 写入，主要影响 TTFT；Decode 更偏逐 token 生成、KV 读取和重复调度，主要影响 TPOT。

本章正式讨论 PD 分离解决什么问题。

PD 分离也叫 Prefill-Decode Disaggregation，或者 Disaggregated Prefill。它的核心思想是把 Prefill 和 Decode 从同一个统一执行引擎里拆出来，让不同资源池分别处理不同阶段。

一句话概括：

> PD 分离解决的是统一引擎中 Prefill 和 Decode 互相干扰、资源画像不匹配、扩缩容耦合和尾延迟难治理的问题；代价是必须解决 KV cache 迁移、路由、状态同步和系统复杂度。

## 36.1 本章目标

读完本章，你应该能讲清：

1. 什么是 PD 分离。
2. 统一引擎混合调度 Prefill/Decode 有哪些问题。
3. PD 分离主要改善哪些指标。
4. 为什么它可以降低 P/D 互相干扰。
5. 为什么它能让资源池独立扩缩容。
6. PD 分离带来哪些新问题。
7. 面试中如何回答“PD 分离解决什么问题”。

## 36.2 什么是 PD 分离

传统 unified serving engine 中，一个 GPU worker 同时处理 prefill 和 decode：

```text
request -> same engine
  -> prefill prompt
  -> decode tokens
```

PD 分离把它拆成：

```text
request
  -> prefill worker / prefill pool
      -> compute prompt KV cache
      -> transfer or share KV cache
  -> decode worker / decode pool
      -> generate output tokens
```

也就是：

```text
Prefill workers 负责输入阶段
Decode workers 负责输出阶段
```

注意：PD 分离不是把模型拆成前半层和后半层。

它拆的是请求生命周期阶段，不是 Transformer 层。

Prefill worker 和 Decode worker 通常都需要能运行同一个模型，只是执行阶段和调度策略不同。

## 36.3 统一引擎的问题一：Prefill interruption

SGLang 文档中提到 unified scheduling 的一个核心问题是 prefill interruption。

意思是：新来的 prefill batch 会打断正在进行的 decode batch。

时间线：

```text
t0: decode requests 正在稳定输出
t1: 一个长 prompt 到达
t2: scheduler 调度长 prefill
t3: decode 暂停或延迟
t4: 用户看到 streaming 卡顿
```

这种问题本质是 Prefill 和 Decode 争抢同一组 GPU compute。

长 prompt prefill 可能让 decode step latency 突然升高。

表现为：

1. TPOT 抖动。
2. ITL 变差。
3. streaming 一卡一卡。
4. p95/p99 输出延迟上升。

PD 分离后，prefill 进入 prefill pool，不再直接占用 decode pool 的执行时间。

## 36.4 统一引擎的问题二：Decode 让 Prefill 饥饿

反过来，如果 scheduler 为了保护 streaming，长期 decode-first，也会导致 waiting prefill 进不来。

时间线：

```text
t0: running decode 很多
t1: new request 进入 waiting queue
t2: scheduler 每轮优先 decode
t3: prefill 一直没有足够 budget
t4: 新请求 TTFT 很高
```

表现为：

1. waiting queue 增长。
2. p95/p99 TTFT 上升。
3. 新用户长时间看不到首 token。
4. GPU 似乎很忙，但新请求体验差。

PD 分离后，Prefill 有自己的资源池和调度器，Decode 压力不会直接吞掉 Prefill 的全部机会。

## 36.5 统一引擎的问题三：资源画像不匹配

Prefill 和 Decode 适合的执行策略不同。

Prefill 更希望：

1. 大 token batch。
2. 高 input tokens/s。
3. 更强计算吞吐。
4. 大 prompt chunk 处理能力。
5. 高效 KV 写入。

Decode 更希望：

1. 稳定低 step latency。
2. 高 output tokens/s。
3. 高效 KV 读取。
4. 低调度开销。
5. 平滑 streaming。

放在同一个 engine 里，配置只能折中。

例如：

```text
max_num_batched_tokens 调大 -> prefill 吞吐好，但 decode 抖动风险上升
max_num_batched_tokens 调小 -> decode 稳定，但 prefill 吞吐下降
```

PD 分离让 P/D 可以使用不同参数和策略。

## 36.6 统一引擎的问题四：扩缩容耦合

不同业务的输入输出比例不同。

长输入短输出：Prefill 压力大。

短输入长输出：Decode 压力大。

统一 engine 中，每加一个 replica，Prefill 和 Decode 能力一起增加。

但真实需求可能是：

```text
Prefill 已经够了，Decode 不够
```

或者：

```text
Decode 够了，Prefill 不够
```

统一扩容会浪费资源。

PD 分离后可以：

```text
prefill pool 独立扩容
decode pool 独立扩容
```

资源配比可以根据流量动态调整。

## 36.7 统一引擎的问题五：尾延迟难治理

统一 engine 中，p99 往往来自 P/D 互相影响。

常见长尾：

1. 超长 prompt 到来，decode 抖动。
2. decode-heavy 流量高峰，新请求 TTFT 飙升。
3. KV cache 接近满，调度和 eviction 抖动。
4. 长输出请求占用 running slots 很久。
5. batch shape 在 prefill/decode 间剧烈变化。

PD 分离的价值之一是隔离长尾来源。

Prefill pool 的 p99 TTFT 问题，可以针对 input tokens、prefix cache、chunked prefill 处理。

Decode pool 的 p99 TPOT 问题，可以针对 running batch、KV layout、speculative decoding 处理。

治理对象更清晰。

## 36.8 统一引擎的问题六：DP attention imbalance

SGLang 文档还提到 DP attention imbalance。

直觉是：在 data-parallel attention 或类似多 worker 并行场景中，不同 DP worker 同时处理的阶段可能不同。

例如：

```text
DP worker 0: 正在处理 prefill batch
DP worker 1: 正在处理 decode batch
```

由于 prefill 和 decode 的执行时间、batch shape、资源压力不同，某些 worker 会拖慢整体同步或导致 decode latency 上升。

这不是简单的“GPU 不够”，而是不同阶段混在同一并行组织里导致负载不均。

PD 分离后，prefill 和 decode 的 worker 角色更明确，可以分别组织并行策略。

## 36.9 PD 分离改善哪些指标

PD 分离可能改善：

1. p95/p99 TPOT。
2. streaming 平滑度。
3. 长 prompt 到来时的 decode 抖动。
4. p95/p99 TTFT。
5. Prefill input tokens/s。
6. Decode output tokens/s。
7. GPU 利用率稳定性。
8. 不同 workload 的资源利用率。
9. P/D 独立扩缩容效率。

但它不保证所有指标都自动变好。

特别是平均 TTFT 可能受 KV transfer 影响。

如果 KV 迁移慢，首 token 后进入 decode 的等待时间可能增加。

所以 PD 分离要看端到端收益。

## 36.10 PD 分离如何降低 Decode 抖动

统一 engine 中，decode 和 prefill 混跑。

PD 分离后，decode pool 主要处理：

```text
已完成 prefill 的请求
  -> 持续 decode
  -> streaming output
```

Decode pool 不再频繁插入大块 prefill。

因此 decode step latency 更稳定。

这对流式体验很重要。

用户不只关心平均 TPOT，也关心 inter-token latency 是否稳定。

PD 分离可以减少这种抖动来源。

## 36.11 PD 分离如何改善 Prefill 吞吐

Prefill pool 可以专门优化 input token 处理。

例如：

1. 更大的 prefill batch token budget。
2. 更适合长 prompt 的 chunk 策略。
3. 更积极的 prefix cache 复用。
4. 更适合 compute-bound 的并行配置。
5. 不被 running decode 频繁打断。

统一 engine 中，为了保护 decode，prefill 往往被限制。

独立 prefill pool 可以更大胆地做 batched prefill。

这对长上下文 RAG、文档分析、代码仓库问答等很重要。

## 36.12 PD 分离如何支持异构资源

由于 P/D 资源画像不同，可以用不同资源配置。

例如：

```text
Prefill pool:
  更强计算吞吐
  更大 prefill token budget
  更适合长 prompt

Decode pool:
  更关注低延迟
  更关注 KV cache 读取
  更稳定 running batch
```

在更复杂系统中，还可能有：

1. 不同 GPU 型号。
2. 不同 TP/DP 配置。
3. 不同 batch 策略。
4. 不同扩缩容策略。
5. 不同 QoS 等级。

但异构也带来复杂性，尤其是 KV cache layout 兼容和传输效率。

## 36.13 PD 分离的核心代价：KV cache 迁移

Prefill 结束后，Decode 需要 prompt 的 KV cache。

如果 P 和 D 在不同 worker 上，就必须解决：

```text
Prefill worker 上生成的 KV cache 如何给 Decode worker 使用？
```

常见方式：

1. 通过高速网络传输 KV。
2. 使用 RDMA、NVLink、专用 transfer engine。
3. GPU staging buffer。
4. CPU 或远端缓存中转。
5. 共享存储或统一内存抽象。

SGLang 文档中提到 Mooncake、NIXL 等 transfer backend，就是为了解决这一类 KV transfer 问题。

KV 迁移是 PD 分离的核心难点。

如果迁移太慢，收益会被抵消。

## 36.14 KV 迁移成本有多重要

KV cache 大小和这些因素有关：

1. prompt token 数。
2. 模型层数。
3. head 数和 head dimension。
4. KV dtype。
5. TP/DP 切分方式。
6. 是否 MLA、GQA、MHA。

长 prompt 的 KV 可能非常大。

如果跨节点传输，网络带宽和延迟会成为关键瓶颈。

因此 PD 分离设计必须回答：

```text
省下的 P/D 干扰成本，是否大于 KV transfer 成本？
```

这是评估 PD 分离是否值得的核心。

## 36.15 路由问题

PD 分离后，请求不再只路由到一个 worker。

它需要经历：

```text
router
  -> choose prefill worker
  -> prefill
  -> choose decode worker
  -> transfer KV
  -> decode
```

Router 要考虑：

1. Prefill pool 负载。
2. Decode pool 负载。
3. KV transfer locality。
4. 模型版本一致性。
5. TP/DP 配置兼容。
6. 故障和重试。
7. 多租户隔离。

路由不当会导致：

1. Prefill 排队。
2. Decode 排队。
3. KV transfer 跨慢链路。
4. cache locality 丢失。
5. p99 变差。

所以 PD 分离一定伴随更复杂的 router。

## 36.16 状态同步问题

统一 engine 中，一个 request 的状态通常在一个进程或一个 engine 内管理。

PD 分离后，请求状态跨多个组件：

1. Router。
2. Prefill worker。
3. Transfer backend。
4. Decode worker。
5. API server / stream connection。

需要同步：

1. request id。
2. prompt tokens。
3. KV cache metadata。
4. first token 或 prefill result。
5. decode worker 分配的 slots。
6. finish reason。
7. abort signal。
8. error state。

任何一环失败，都要清理资源。

例如 decode worker 断开，prefill worker 上暂存的 KV metadata 不能永远占着。

## 36.17 故障恢复问题

PD 分离引入更多失败点。

可能失败：

1. Prefill worker OOM。
2. Decode worker OOM。
3. KV transfer 超时。
4. Router 选错或目标不可达。
5. 网络抖动。
6. Decode worker 中途宕机。
7. 客户端断开。

需要处理：

1. 是否重试 prefill。
2. 是否换 decode worker。
3. 已传输 KV 如何清理。
4. 已输出 token 如何处理。
5. Streaming 如何返回错误。
6. Metrics 如何归因。

这就是为什么 PD 分离不适合作为初学者第一版 serving 架构。

## 36.18 什么时候 PD 分离收益明显

收益明显的场景通常有：

1. 流量规模大。
2. Prefill 和 Decode 干扰明显。
3. p99 TPOT 被长 prompt 打爆。
4. p99 TTFT 被 decode-heavy 流量拖高。
5. 输入输出比例波动大。
6. 长上下文请求多。
7. 有高性能 KV transfer 能力。
8. 需要 P/D 独立扩缩容。
9. 有成熟监控和路由系统。

例如：

```text
长上下文 RAG + 大量流式聊天混合
代码生成长输出 + 文档分析长输入混合
多租户模型服务平台
```

这些场景里，统一 engine 的折中可能已经不够。

## 36.19 什么时候不适合 PD 分离

不适合或要谨慎：

1. 流量小。
2. 请求短。
3. 单机 serving 已满足 SLO。
4. KV transfer 网络条件差。
5. 团队缺少分布式调度经验。
6. 监控不完善。
7. 模型或 KV layout 频繁变化。
8. 调试稳定性比极致性能更重要。

如果没有明确 P/D 干扰证据，先不要上 PD 分离。

先做：

1. continuous batching。
2. chunked prefill。
3. prefix cache。
4. scheduler 调优。
5. 限流和隔离。
6. 指标观测。

再判断 PD 分离是否值得。

## 36.20 和 chunked prefill 的关系

Chunked prefill 和 PD 分离都能缓解长 prefill 阻塞 decode。

但层次不同。

Chunked prefill：

```text
同一 engine 内，把长 prefill 切小，穿插 decode
```

PD 分离：

```text
不同资源池分别处理 prefill 和 decode
```

对比：

| 维度 | Chunked prefill | PD 分离 |
| --- | --- | --- |
| 改动范围 | scheduler 内策略 | 系统架构 |
| 是否分资源池 | 否 | 是 |
| 是否需要 KV 迁移 | 否 | 是 |
| 复杂度 | 中等 | 高 |
| 主要收益 | 降低长 prefill 阻塞 | 隔离 P/D 干扰，独立扩缩容 |
| 适合阶段 | 单机优化优先 | 大规模架构优化 |

工程上通常先做 chunked prefill，再评估 PD 分离。

## 36.21 和 prefix cache 的关系

Prefix cache 和 RadixAttention 可以减少 prefill 成本。

如果 prefix cache 命中率很高，prefill 压力会下降。

这可能降低 PD 分离的必要性。

但在以下情况，PD 分离仍然有价值：

1. Cache miss 仍然很多。
2. 新文档长 prompt 多。
3. Decode 长输出压力大。
4. P/D 流量比例波动大。
5. 需要独立扩缩容。

也就是说，prefix cache 是减少 prefill 工作量，PD 分离是隔离 prefill/decode 执行资源。

二者不是替代关系。

## 36.22 和 speculative decoding 的关系

Speculative decoding 主要优化 decode。

如果 decode 是瓶颈，可以先尝试 speculative decoding。

但它不能解决所有问题。

例如：

1. 长 prefill 仍可能打断 decode。
2. Prefill queue 仍可能很长。
3. P/D 扩缩容仍耦合。
4. KV cache 和 batch shape 仍混在同一 engine。

PD 分离和 speculative decoding 可以组合：

```text
Prefill pool 负责高吞吐 prompt 处理
Decode pool 使用 speculative decoding 加速输出
```

但组合后系统更复杂，需要更强监控和调度。

## 36.23 PD 分离的价值边界

可以用一句话判断：

```text
如果你的主要问题是 P/D 在同一资源池里互相干扰，并且 KV transfer 成本可控，PD 分离有价值。
```

如果主要问题是：

1. 模型太慢。
2. 单请求太短。
3. Tokenizer 慢。
4. RAG 检索慢。
5. 网络 streaming 慢。
6. GPU 数太少。

PD 分离不一定解决根因。

它是高级 serving 架构，不是通用性能按钮。

## 36.24 面试官会怎么问

问题一：PD 分离解决什么问题？

回答要点：解决统一 engine 中 prefill 和 decode 资源画像不同却混合调度导致的互相干扰、尾延迟、资源扩缩容耦合和调度折中问题。

问题二：Unified scheduling 有什么问题？

回答要点：长 prefill 会打断 decode，导致 TPOT/streaming 抖动；decode-first 又会让 prefill 饥饿，导致 TTFT 上升；不同 DP worker 混合 P/D 还可能负载不均。

问题三：PD 分离的核心代价是什么？

回答要点：KV cache 要从 prefill worker 传给 decode worker，引入网络带宽、延迟、layout 兼容、路由、状态同步、故障恢复复杂度。

问题四：PD 分离和 chunked prefill 的区别？

回答要点：chunked prefill 是同一 engine 内把长 prefill 切小，与 decode 穿插；PD 分离是系统架构上拆成 prefill/decode 资源池，需要 KV 迁移但能独立扩缩容和隔离干扰。

问题五：什么时候不该做 PD 分离？

回答要点：流量小、请求短、单机已满足 SLO、KV transfer 条件差、监控和调度能力不足时，不应过早引入。

## 36.25 标准回答模板

如果面试官问“PD 分离解决什么问题”，可以这样回答：

```text
PD 分离解决的是 Prefill 和 Decode 在统一 serving engine 中混合调度带来的资源干扰和扩缩容耦合问题。Prefill 一次处理大量输入 token，更偏 compute-bound，影响 TTFT；Decode 每轮每请求通常只生成一个 token，要频繁读取 KV cache，更偏 memory bandwidth 和调度敏感，影响 TPOT 和 streaming 平滑度。

在统一 engine 中，长 prompt prefill 可能打断正在进行的 decode，导致输出卡顿；如果 scheduler 长期 decode-first，又会让 waiting prefill 饥饿，新请求 TTFT 变高。同时 P/D 资源画像不同，却只能用同一套 batch、token budget、并行和扩缩容策略。

PD 分离把 prefill 和 decode 拆到不同资源池，让 prefill pool 专注 input tokens/s 和 prompt KV 生成，让 decode pool 专注 output tokens/s、TPOT 和 streaming 稳定性。这样可以降低 P/D 互相干扰，并支持按输入输出压力独立扩缩容。

它的代价是 KV cache 必须从 prefill worker 迁移或共享给 decode worker，因此会引入 transfer backend、网络带宽、KV layout、路由、状态同步和故障恢复复杂度。只有当 P/D 干扰明显且 KV transfer 成本可控时，PD 分离才值得做。
```

## 36.27 本章总结

PD 分离不是为了架构复杂而复杂，而是为了解决统一 engine 中 Prefill 和 Decode 混合调度带来的现实问题：prefill interruption、decode 抖动、prefill 饥饿、资源画像不匹配、扩缩容耦合和尾延迟难治理。

它的核心收益是把 P/D 放入不同资源池，各自使用更适合的调度、batch 和扩缩容策略。

它的核心代价是 KV cache 迁移和分布式状态管理。

下一章会进入 PD 分离的系统架构，具体拆解 router、prefill worker、decode worker、KV transfer backend、状态同步和请求生命周期。
