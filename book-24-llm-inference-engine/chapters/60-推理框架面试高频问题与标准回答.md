## 60.2 面试回答的基本框架

推理框架类问题可以用同一套回答骨架。

不要一上来就讲单个优化点。

先把系统边界讲清楚。

一个比较稳的回答顺序是：

```text
1. 目标：低 TTFT、低 TPOT、高吞吐、低显存、高稳定性、低成本。
2. 请求生命周期：API -> tokenizer -> scheduler -> model runner -> KV cache -> sampler -> streamer。
3. 核心瓶颈：prefill、decode、KV cache、batching、网络、调度。
4. 核心机制：continuous batching、PagedAttention、prefix cache、chunked prefill、speculative decoding、parallelism。
5. 生产能力：限流、监控、灰度、故障隔离、容量规划。
6. trade-off：延迟、吞吐、显存、质量、复杂度和成本。
```

这套框架可以覆盖大部分问题。

面试官如果问“如何设计一个推理框架”，你可以按这套展开。

面试官如果问“为什么线上 TPOT 变高”，你也可以从这套框架中抽出 decode、KV cache、batching、worker、网络和客户端消费速度来排查。

## 60.3 问题一：LLM Serving Engine 到底是什么

标准回答：

```text
LLM serving engine 是把模型 checkpoint 变成可在线服务的运行时系统。它不只是调用一次 model.generate，而是负责请求接入、tokenization、调度、batching、prefill、decode、KV cache 管理、采样、流式返回、监控和错误处理。

它的核心目标是在有限 GPU 显存和算力下，同时优化 TTFT、TPOT、吞吐、并发、稳定性和成本。相比普通深度学习推理服务，LLM serving 的特殊点是自回归生成、请求长度差异大、输出长度不可提前完全确定、KV cache 是动态运行时状态，并且 streaming 用户体验非常依赖调度策略。
```

追问要点：

1. 普通模型服务通常一次 forward 就结束，LLM 需要逐 token decode。
2. LLM 请求成本主要由 input token、output token、上下文长度和并发决定。
3. serving engine 的核心不是模型结构本身，而是调度和运行时资源管理。

## 60.4 问题二：一个请求的生命周期是什么

标准回答：

```text
一个 chat completion 请求进入系统后，通常先经过 API gateway 做鉴权、限流和参数校验。然后 tokenizer 把 messages 或 prompt 转成 input ids，并估算输入 token 数和 max output tokens。router 选择合适的 worker 或 worker pool。scheduler 把请求放入 waiting queue，并根据 token budget、KV cache 空闲量、优先级和当前 running batch 决定何时调度。

请求第一次进入模型执行 prefill，模型并行处理 prompt，生成每层 KV cache，并得到最后位置 logits。之后进入 decode 阶段，每轮生成一个或少量 token，复用历史 KV cache。sampler 根据 temperature、top-k、top-p、stop token 等规则选择下一个 token。streamer 把 token 逐步返回客户端。请求完成、取消或超时后，engine 释放 KV cache blocks，记录 metrics 和日志。
```

容易漏掉的点：

1. tokenizer 和 detokenizer 可能成为 CPU 侧瓶颈。
2. scheduler 不是只凑 batch，还要控制 token budget 和 KV cache budget。
3. streaming 请求完成前会持续占用运行时状态。
4. client cancel 后必须释放 KV cache，否则会造成泄漏。

## 60.5 问题三：Prefill 和 Decode 有什么区别

标准回答：

```text
Prefill 是处理输入 prompt 的阶段。它把完整 prompt token 送入模型，并为每层生成对应的 KV cache。prefill 通常并行度更高，输入越长计算越重，直接影响 TTFT。

Decode 是生成输出 token 的阶段。每一步通常只输入最新 token，复用历史 KV cache，生成下一个 token。decode 是自回归串行过程，更容易受到 batch size、KV cache 读取、显存带宽、调度策略和 sampling 开销影响，直接影响 TPOT。

因此 serving engine 不能把 prefill 和 decode 当成同一种任务调度。长 prefill 可能阻塞已有请求的流式输出，而 decode 过多也会让新请求迟迟拿不到首 token。
```

面试延伸：

1. TTFT 高优先看 queue time、prefill 负载、prompt length、prefix cache 命中率。
2. TPOT 高优先看 decode batch、KV cache 带宽、GPU kernel、sampling、网络和客户端 backpressure。
3. chunked prefill 的目的就是把长 prefill 切小，避免独占 GPU。

## 60.6 问题四：KV Cache 解决什么问题

标准回答：

```text
KV cache 缓存每一层历史 token 的 key 和 value。自回归生成时，新 token 需要关注历史上下文。如果没有 KV cache，每生成一个 token 都要重新计算整个上下文的 K/V，成本会随生成长度快速上升。

有了 KV cache 后，decode 阶段只需要计算新 token 的 Q/K/V，并把新 K/V append 到 cache 中。历史 K/V 可以复用，从而显著降低重复计算。

但 KV cache 不是免费优化。它把计算压力转成显存容量和显存带宽压力。长上下文、高并发和长输出都会让 KV cache 成为 serving 的核心瓶颈。
```

公式表达可以简化为：

```text
KV bytes per token = 2 * layers * kv_heads * head_dim * bytes_per_element
```

其中 `2` 分别表示 key 和 value。

追问要点：

1. GQA/MQA 可以减少 KV head 数，从而降低 KV cache。
2. KV cache 是每个活跃请求独有的运行时状态。
3. KV cache 释放、复用、迁移和量化都会影响 serving 性能。

## 60.7 问题五：PagedAttention 解决什么问题

标准回答：

```text
PagedAttention 的核心思想是借鉴操作系统分页，把逻辑上连续的 KV cache 切成固定大小 block，并通过 block table 映射到物理显存中的 blocks。这样每个请求不需要预留一整段连续 KV cache，也不需要按最大长度提前分配。

它主要解决 LLM serving 中变长请求导致的 KV cache 显存碎片和浪费问题。请求可以按需追加 block，请求结束后释放 block，调度器也更容易在高并发下管理显存。

PagedAttention 不改变 attention 的数学定义，它是 KV cache 管理和 attention kernel 访问方式上的系统优化。
```

面试中一定要强调：

1. PagedAttention 不是 sparse attention。
2. 它不是近似 attention。
3. 它解决的是 serving 中 KV cache 分配、碎片和动态增长问题。
4. block size 有 trade-off：小 block 碎片少但管理开销高，大 block 管理简单但尾部浪费更大。

## 60.8 问题六：Continuous Batching 为什么重要

标准回答：

```text
LLM 请求的输出长度不同，decode 是逐 token 迭代。如果使用静态 batch，一个 batch 里的短请求完成后，可能要等长请求结束，GPU 利用率和延迟都会变差。

Continuous batching 允许系统在每个 decode iteration 动态更新 batch：完成的请求退出，新请求加入，运行中的请求继续生成。这样可以减少空等，提高 GPU 利用率和吞吐，同时降低排队浪费。

它的关键不是简单把请求凑成 batch，而是在每一轮迭代里根据 token budget、KV cache budget、优先级和调度策略选择哪些请求执行 prefill 或 decode。
```

容易混淆：

1. dynamic batching 通常是在请求进入模型前凑批。
2. continuous batching 是生成过程中持续更新 batch。
3. continuous batching 需要 KV cache manager 支持动态追加和释放。
4. 如果调度策略不好，continuous batching 也可能造成长请求拖慢短请求。

## 60.9 问题七：Chunked Prefill 解决什么问题

标准回答：

```text
Chunked prefill 把长 prompt 的 prefill 拆成多个较小 chunk 执行，而不是让一个超长 prefill 一次占用 GPU 很久。

它解决的问题是长 prompt 对已有 decode 请求的干扰。没有 chunked prefill 时，一个很长的输入可能让正在 streaming 的请求 token 间隔变大，用户感觉输出卡顿。

chunked prefill 的代价是调度更复杂，可能引入额外 kernel launch 或状态管理开销。调得太碎会降低吞吐，调得太大又无法缓解 decode 抖动。
```

面试表达重点：

1. 它主要平衡新请求 TTFT 和老请求 TPOT。
2. 它通常和 token budget、continuous batching 一起出现。
3. 它不是提升模型质量的算法，而是 serving 调度优化。

## 60.10 问题八：Prefix Cache 和 Prompt Cache 有什么价值

标准回答：

```text
Prefix cache 或 prompt cache 用来复用相同前缀的 prefill 结果。很多业务请求有重复的 system prompt、工具说明、RAG 模板、few-shot 示例或多轮对话历史。如果每次都重新 prefill，会浪费大量计算。

命中 prefix cache 后，系统可以复用已计算的 KV cache，只对新增 token 做 prefill 或 decode，从而降低 TTFT、减少 prefill compute，并提升吞吐。

但 prefix cache 也有代价：需要缓存管理、哈希匹配、生命周期控制、显存或 CPU 内存占用，以及多租户隔离。缓存命中率低时，维护成本可能大于收益。
```

追问要点：

1. cache key 通常要和 token 序列、模型版本、LoRA adapter、chat template 等绑定。
2. 多租户场景要避免跨租户泄露上下文。
3. prefix cache 命中率应该进入 dashboard。
4. SGLang 的 RadixAttention 更强调对共享 prefix 的树状复用。

## 60.11 问题九：vLLM 的核心贡献是什么

标准回答：

```text
vLLM 的核心贡献可以概括为面向 LLM serving 的高吞吐和高显存效率运行时。它最经典的机制是 PagedAttention，用 block table 管理 KV cache，降低变长请求下的显存浪费和碎片。同时它支持 continuous batching、chunked prefill、prefix caching、OpenAI-compatible API、量化、多种并行方式、speculative decoding 和生产部署相关能力。

面试中我不会把 vLLM 简化成一个 attention kernel。更准确地说，它是一个 serving engine：scheduler、KV cache manager、model runner、worker/executor、API server 和 metrics 共同决定最终性能。
```

回答边界：

1. 不要说 vLLM 只靠 PagedAttention。
2. 不要把 PagedAttention 说成改变模型输出质量。
3. 不要忽略 scheduler 和 memory management。
4. 不要忽略版本差异，具体参数和实现细节会随 vLLM 版本演化。

## 60.12 问题十：SGLang 和 vLLM 有什么区别

标准回答：

```text
vLLM 更常被理解为高吞吐、显存高效的通用 LLM serving engine，代表性机制包括 PagedAttention、continuous batching、prefix caching 和多种推理优化。

SGLang 也是高性能 serving 框架，但它更强调 runtime 和程序化生成场景，例如 RadixAttention 复用共享 prefix、structured generation、constrained decoding、multi-turn 和 agent/tool-use serving。它也支持 continuous batching、paged attention、PD 分离、speculative decoding、并行和量化等生产能力。

二者不是简单谁替代谁，而是关注点和生态侧重不同。面试中应该从 workload 出发比较：如果大量请求共享长 prefix、结构化输出和 agent workflow 很重，SGLang 的 runtime 和 prefix sharing 设计值得重点考虑；如果是通用 OpenAI-compatible serving，vLLM 的生态和成熟度也很有吸引力。
```

注意事项：

1. 不要把 SGLang 说成只有前端 DSL。
2. 不要把 vLLM 说成不能做结构化输出。
3. 两者能力都在快速变化，回答时要强调版本和 workload 边界。

## 60.13 问题十一：PD 分离解决什么问题

标准回答：

```text
PD 分离指把 prefill 和 decode 放到不同资源池或不同节点上执行。它出现的原因是 prefill 和 decode 的资源画像不同：prefill 更偏计算密集，输入长度大时单次计算很重，主要影响 TTFT；decode 更偏迭代式生成和 KV cache 读取，持续占用请求状态，主要影响 TPOT。

把二者拆开后，可以为 prefill 和 decode 分别配置硬件、batching 策略和调度策略，减少长 prefill 对 decode streaming 的干扰，提高资源利用率。

但 PD 分离不是免费午餐。它会引入 KV cache 迁移或共享问题、网络开销、路由复杂度、跨节点一致性、故障恢复和调试复杂度。只有在模型规模、流量规模、上下文长度和硬件拓扑达到一定程度时，收益才可能覆盖复杂度。
```

面试追问：

1. KV cache 如何从 prefill 节点交给 decode 节点。
2. 网络带宽是否会成为瓶颈。
3. 短请求是否适合 PD 分离。
4. 如何做 prefill/decode 容量配比。
5. PD 分离失败时如何降级回单体 serving。

## 60.14 问题十二：TTFT 高怎么排查

标准回答：

```text
TTFT 表示用户从发出请求到收到第一个 token 的时间。TTFT 高通常和排队、tokenization、prefill、prefix cache、router 负载和 admission control 有关。

我会先拆分 TTFT：client 到 gateway、gateway 排队、router 选择、tokenization、scheduler queue time、prefill execution、first token sampling 和 streaming flush。然后看输入长度分布是否变长，waiting queue 是否增加，prefill token/s 是否下降，prefix cache hit rate 是否下降，max_num_batched_tokens 或 chunked prefill 配置是否变化，某些 worker 是否热点或故障。

如果 TTFT 只在长 prompt 上升，重点看 prefill 和 chunked prefill。如果短请求也上升，重点看队列、路由、worker capacity、CPU tokenizer 和整体过载。
```

排查顺序：

1. 是否是全局问题还是单模型、单租户、单 worker pool 问题。
2. queue time 是否升高。
3. input length P95/P99 是否升高。
4. prefill tokens/s 是否下降。
5. prefix cache hit rate 是否下降。
6. worker 是否重启、OOM 或处于 warmup。
7. tokenizer CPU 是否打满。
8. router 是否把流量打到热点 worker。

## 60.15 问题十三：TPOT 高怎么排查

标准回答：

```text
TPOT 表示每个输出 token 的平均或分位延迟，主要影响 streaming 是否流畅。TPOT 高通常发生在 decode 阶段，和 running batch size、KV cache 读取、GPU kernel、显存带宽、sampling、网络 flush、客户端消费速度和调度策略有关。

我会先确认 TPOT 是所有请求变高，还是长上下文、长输出、特定模型、特定 GPU 或特定租户变高。然后看 decode batch size、active tokens、KV cache usage、preemption rate、GPU utilization、GPU memory bandwidth、sampling 参数、structured output 约束和 client backpressure。

如果 GPU utilization 不高但 TPOT 高，可能是 memory bandwidth、KV cache 访问、调度碎片或 CPU/网络瓶颈。如果 KV cache pressure 高，要看是否有长上下文并发、block 碎片、preemption 或 swap。
```

常见原因：

1. running batch 过大导致单 token iteration 变慢。
2. 长上下文请求增多，KV cache 读取成本上升。
3. preemption 或 swap 增加。
4. structured generation 或复杂 sampling 带来 CPU/GPU 额外开销。
5. 慢客户端导致 streaming backpressure。
6. 新 kernel、新模型 revision 或量化配置引入性能回退。

## 60.16 问题十四：吞吐上去了但用户觉得更卡，为什么

标准回答：

```text
吞吐和用户体验不是同一个指标。系统总 output tokens/s 上升，可能是因为 batch 更大或长请求更多，但这可能同时提高 TTFT 或 TPOT 的尾延迟。用户感知更依赖首 token 延迟、token 间隔稳定性和 P95/P99，而不是平均 tokens/s。

例如 scheduler 为了吞吐把 batch 塞得很满，GPU 利用率上升，但短请求排队更久，streaming token 间隔更不稳定。或者系统处理了更多长输出请求，tokens/s 很高，但短 chat 的 TTFT 变差。

所以调优时必须同时看吞吐、TTFT、TPOT、queue time、输入输出长度分布和 workload 分桶，不能只看单一吞吐指标。
```

一句话总结：

```text
吞吐优化如果没有 SLO 约束，可能只是把用户延迟换成了更好看的 tokens/s。
```

## 60.17 问题十五：如何设计一个 vLLM-like 推理框架

标准回答：

```text
我会把系统拆成 API 层、router、scheduler、KV cache manager、model runner、sampler、streamer、metrics 和 worker 管理几个模块。

API 层负责 OpenAI-compatible 接口、鉴权、限流和参数校验。router 按模型、租户、负载和 KV capacity 选择 worker。scheduler 维护 waiting queue 和 running batch，在每个 iteration 根据 token budget、KV cache budget、优先级和 SLO 选择 prefill 或 decode 请求。KV cache manager 用 block/page 管理每个 sequence 的 KV blocks，支持分配、追加、释放和可能的 prefix cache。model runner 执行 prefill/decode forward，sampler 生成 token，streamer 返回 SSE。metrics 记录 TTFT、TPOT、queue time、tokens/s、KV blocks、preemption、错误率和 worker 状态。

第一版可以先实现单机单模型、greedy sampling、简单 KV cache 和 continuous batching。第二版再加入 paged KV cache、prefix cache、chunked prefill、preemption、OpenAI API、量化、多 GPU 并行和生产监控。
```

架构图可以口述为：

```text
Client
  -> API Gateway
  -> Router
  -> Engine Scheduler
  -> Model Runner
  -> KV Cache Manager
  -> Sampler
  -> Streamer
  -> Client
```

## 60.18 问题十六：Scheduler 应该怎么设计

标准回答：

```text
LLM scheduler 的核心职责是在每个 iteration 决定哪些请求进入 batch，同时控制 token budget、KV cache budget、优先级和 SLO。它不应该只按请求数凑 batch，因为不同请求的输入长度、输出长度和 KV cache 占用差异很大。

一个简单设计是维护 waiting queue 和 running set。每轮先处理 running decode 请求，再根据剩余 token budget 和 KV cache 空闲量选择部分 waiting 请求做 prefill。对于长 prompt，可以使用 chunked prefill，把一次长 prefill 拆成多个 chunk，避免影响 decode 的 TPOT。对于 KV cache 不足的情况，可以拒绝新请求、preempt 低优先级请求、swap 或 recompute。

调度策略要根据目标取舍。如果目标是最低 TTFT，可能更积极调度新 prefill；如果目标是稳定 TPOT，要保护 decode iteration；如果目标是最大吞吐，则可能增大 batch 和 token budget，但要接受更高尾延迟风险。
```

关键输入：

1. waiting queue 长度。
2. running requests 数。
3. 每个请求已生成长度和剩余预算。
4. input token 和 chunk size。
5. KV cache free blocks。
6. tenant priority。
7. deadline 或 timeout。

## 60.19 问题十七：KV Cache Manager 应该怎么设计

标准回答：

```text
KV cache manager 负责为每个 active sequence 分配、追加、查找和释放 KV cache。生产级设计通常不会给每个请求预留连续大块显存，而是把 KV cache 切成固定大小 blocks，通过 sequence 到 block table 的映射管理。

请求 prefill 时按 token 数申请 blocks，decode 每生成新 token 可能追加 block。请求结束、取消或超时后释放 blocks。如果启用 prefix cache，共享前缀的 blocks 需要引用计数或生命周期管理，避免被提前释放。KV manager 还要暴露 free blocks、used blocks、fragmentation、preemption 和 allocation failure 等指标。
```

需要说明的 trade-off：

1. block size 影响碎片和管理开销。
2. prefix cache 需要引用计数和隔离。
3. preemption 可以缓解压力，但会影响延迟。
4. swap 到 CPU 或远端缓存能扩容量，但会引入网络和恢复成本。

## 60.20 问题十八：如何做推理框架压测

标准回答：

```text
推理框架压测不能只用固定 prompt 和平均 latency。必须构造接近真实业务的输入输出长度分布，并同时测 request throughput、input tokens/s、output tokens/s、TTFT、TPOT、E2E latency、queue time、KV cache usage、preemption、GPU utilization、错误率和 timeout。

压测要覆盖短输入短输出、短输入长输出、长输入短输出、长输入长输出、混合流量、streaming、client cancel、prefix cache 命中和不命中、超长上下文、过载和 worker 故障。

最终要找到稳定容量和拐点，而不是只追求峰值吞吐。稳定容量应该是在 SLO 满足、错误率可控、没有频繁 OOM 或 preemption 的情况下测出来的。
```

面试加分点：

1. benchmark 要记录模型、GPU、dtype、parallel size、scheduler 参数和版本。
2. 结果要按 workload 分桶，而不是只给全局平均。
3. 压测要看过载行为是否优雅拒绝。
4. 新版本上线前要和 baseline 做回归对比。

## 60.21 问题十九：如何做容量规划

标准回答：

```text
我会先按模型、租户和 workload 收集流量，统计 request QPS、input token/s、output token/s、输入长度 P50/P95/P99、输出长度 P50/P95/P99 和并发数。然后用接近真实分布的 benchmark 测出单 worker 在 SLO 约束下的稳定容量，包括 TTFT、TPOT、吞吐、KV cache 占用和错误率。

容量估算时不能用 benchmark 极限值，而要乘以规划利用率，比如 60% 到 80%，再考虑峰值流量、N+1 故障冗余、发布 surge、可用区故障域和增长率。最后把 worker 数乘以 tensor parallel size 得到 GPU 数，并估算 GPU 小时、存储、网络、日志和冗余成本。
```

一句话总结：

```text
LLM serving 容量规划的单位不是 QPS，而是 token、active sequence、KV cache 和 SLO。
```

## 60.22 问题二十：量化一定能降低延迟吗

标准回答：

```text
不一定。量化通常能降低权重显存和显存带宽压力，尤其是 weight-only INT8/INT4 在 decode 阶段可能有收益。但实际延迟是否下降取决于 kernel 是否高效、硬件是否支持、反量化开销、瓶颈是否在权重带宽，以及 batch size 和模型结构。

如果系统瓶颈在 KV cache、scheduler、CPU tokenizer、网络或客户端 backpressure，权重量化不一定明显降低 TPOT。低 bit 量化还可能带来质量下降、格式稳定性变差、长上下文或代码数学能力回退。因此上线前要同时评估质量、TTFT、TPOT、吞吐、显存、错误率和业务样本。
```

面试常见误区：

1. INT4 不等于一定比 FP16 快。
2. 显存下降不等于延迟下降。
3. perplexity 小幅变化不代表业务格式稳定。
4. 量化收益强依赖 kernel 和硬件。

## 60.23 问题二十一：Speculative Decoding 为什么能加速

标准回答：

```text
Speculative decoding 用一个更小或更快的 draft model 先生成多个候选 token，再由 target model 一次 forward 并行验证这些候选。如果多个候选被接受，就相当于一次 target model 调用推进多个 token，从而减少昂贵大模型 decode step 的次数。

它的加速比取决于 draft model 速度、候选长度、接受率、target model 验证效率和任务分布。draft 太弱会导致接受率低，draft 太大又会抵消收益。严格的 speculative sampling 可以保持 target model 输出分布不变，但实现复杂度更高。
```

追问要点：

1. 它主要优化 decode，不解决长 prompt prefill 本身。
2. 高温采样、代码、数学或长上下文场景接受率可能不同。
3. 接受率高不一定总收益高，还要看 proposal 成本和系统调度开销。

## 60.24 问题二十二：Structured Generation 为什么影响 serving

标准回答：

```text
Structured generation 要求模型输出满足 JSON schema、正则、grammar 或工具调用格式。它通常通过 constrained decoding 在每一步限制可选 token 集合。

这会影响 serving，因为每个 decode step 除了模型 forward，还要维护状态机、计算合法 token mask，并和 tokenizer vocabulary 对齐。复杂 schema、长输出和大量并发时，约束逻辑可能成为 CPU 或 GPU 侧额外开销。

但它的价值也很大：可以提升工具调用、JSON 输出和自动化系统的格式可靠性，减少重试和后处理成本。
```

回答 trade-off：

1. 结构化输出提升格式稳定性。
2. 约束解码可能增加每步开销。
3. schema 太复杂会降低吞吐。
4. 评估时要同时看格式合法率、任务正确率、TTFT、TPOT 和重试率。

## 60.25 问题二十三：多 GPU serving 有哪些并行方式

标准回答：

```text
常见并行方式包括 tensor parallel、pipeline parallel、data parallel、expert parallel 和 context parallel。

Tensor parallel 把单层矩阵计算切到多张 GPU 上，适合单卡放不下或需要更高单请求吞吐的大模型，但通信频繁。Pipeline parallel 按层切分模型，通信频率较低，但可能有 pipeline bubble，serving 中调度更复杂。Data parallel 是多份模型副本处理不同请求，适合扩展 QPS。Expert parallel 常用于 MoE，把不同专家分布到不同设备。Context parallel 主要面向超长上下文，把序列维度相关计算拆分。

serving 中选择并行方式要看模型大小、单请求延迟、吞吐目标、GPU 拓扑、通信带宽、KV cache 分布和故障域。
```

面试加分点：

1. TP 增大可能降低单卡显存，但增加通信。
2. DP 扩展吞吐简单，但每份副本都要加载权重。
3. PP 对在线低延迟 serving 不一定总合适。
4. MoE serving 还要考虑 expert load balancing。

## 60.26 问题二十四：线上 OOM 怎么排查

标准回答：

```text
我会先区分 OOM 发生在模型加载、prefill、decode、KV cache 分配、prefix cache、CUDA graph capture 还是临时 workspace。然后看最近是否有模型版本、max context、max_num_batched_tokens、gpu_memory_utilization、batch size、prefix cache、quantization 或并行配置变化。

如果是请求流量导致，重点看输入长度 P99、输出长度、running requests、active tokens、KV cache free blocks 和 preemption。如果是部署导致，重点看权重显存、CUDA graph、allocator fragmentation、worker warmup 和多进程资源竞争。

止血上可以降低 max_tokens、限制长上下文、降低 batch token budget、关闭或限制 prefix cache、扩容 worker、回滚版本或提高拒绝阈值。修复后要补充容量测试和 OOM 告警。
```

排查关键词：

1. weight memory。
2. KV cache memory。
3. activation 或 workspace。
4. fragmentation。
5. prefix cache growth。
6. config regression。

## 60.27 问题二十五：如何设计监控 Dashboard

标准回答：

```text
我会把 dashboard 分成用户体验、吞吐容量、KV cache、GPU worker、错误和成本几个层次。

用户体验层看 TTFT、TPOT、E2E latency、timeout 和 stream cancel。吞吐层看 RPS、input tokens/s、output tokens/s、queue time、running/waiting requests。KV cache 层看 used/free blocks、active tokens、preemption、swap 和 prefix cache hit rate。GPU worker 层看 GPU utilization、memory usage、OOM、worker restart、NCCL error 和 model revision。错误层按 error code、tenant、model 和 route 分桶。成本层看 GPU 小时、tokens、租户和模型维度。

所有指标都要支持按 model、tenant、worker pool、GPU type、region 和 revision 切分，否则总量指标会掩盖局部故障。
```

## 60.28 问题二十六：如何回答开放系统设计题

如果面试官问：设计一个支持 OpenAI-compatible API 的 LLM serving 平台。

可以按下面结构回答：

```text
第一，我会先明确需求：支持哪些模型、上下文长度、streaming、并发、SLO、多租户、工具调用、结构化输出和成本目标。

第二，架构上分 API gateway、auth/rate limit、router、engine worker、scheduler、KV cache manager、model runner、streamer、metrics/logging 和 admin control plane。

第三，请求路径是 gateway 接入，tokenizer 处理，router 根据模型和负载选 worker，scheduler 做 continuous batching，model runner 执行 prefill/decode，KV manager 管理 blocks，sampler 生成 token，streamer 返回 SSE。

第四，性能优化包括 PagedAttention、continuous batching、chunked prefill、prefix cache、量化、speculative decoding、合理的 tensor parallel/data parallel、长短请求隔离和 admission control。

第五，稳定性包括 health check、滚动升级、灰度、drain、超时、client cancel、OOM 保护、故障演练和容量规划。

```

最后补一句 trade-off：

```text
系统设计的关键不是堆优化点，而是在目标 SLO 下平衡延迟、吞吐、显存、成本和复杂度。
```

## 60.31 最终 3 分钟回答模板

如果面试官让你整体讲一下“你如何理解 LLM 推理框架”，可以这样回答：

```text
我理解 LLM 推理框架不是简单调用 generate，而是一个面向自回归生成的 serving engine。它要把请求接入、tokenization、调度、prefill、decode、KV cache 管理、sampling、streaming、监控和错误处理串起来，在有限 GPU 资源下平衡 TTFT、TPOT、吞吐、显存、稳定性和成本。

一个请求进入系统后，会先经过 API gateway 做鉴权、限流和参数校验，然后 tokenizer 转成 token，router 选择 worker，scheduler 放入 waiting queue。第一次执行 prefill，处理 prompt 并生成 KV cache；之后每轮 decode 复用 KV cache 生成新 token，通过 sampler 选择 token，再由 streamer 返回给客户端。请求结束或取消后，要释放 KV cache 并记录 metrics。

推理框架的核心难点有三个。第一是调度，因为请求长度和输出长度差异很大，静态 batching 会浪费资源，所以需要 continuous batching 和 iteration-level scheduling。第二是 KV cache，因为长上下文和高并发会让 KV cache 成为显存瓶颈，所以需要类似 PagedAttention 的 block 管理、prefix cache、preemption 或 offload。第三是 prefill/decode 的资源画像不同，长 prefill 会影响 TTFT 和已有 decode 的 TPOT，因此需要 chunked prefill，规模更大时可以考虑 PD 分离。

以 vLLM 为例，它通过 PagedAttention、continuous batching、chunked prefill、prefix caching、多种 attention kernel、量化和并行能力提升吞吐和显存效率。以 SGLang 为例，它除了高性能 runtime，也更强调 RadixAttention、prefix sharing、structured generation 和 agent/tool-use 场景。二者都不是单一优化点，而是完整 serving runtime。

线上落地时，我会重点监控 TTFT、TPOT、queue time、input/output tokens/s、running/waiting requests、KV cache free blocks、preemption、GPU utilization、OOM、timeout 和 error rate。容量规划不能只看 QPS，要按 token、active sequence、KV cache 和 SLO 估算。调优时也不能只看吞吐，要按 workload 分桶看 P95/P99 延迟、稳定性和成本。
```

## 60.33 本章总结

推理框架面试考的不是单点名词，而是完整系统理解。

你需要能从请求生命周期讲到 scheduler，从 scheduler 讲到 KV cache，从 KV cache 讲到显存和长上下文，从 prefill/decode 讲到 TTFT/TPOT，从单机 engine 讲到分布式 serving 和生产稳定性。

vLLM、SGLang、PagedAttention、continuous batching、prefix cache、chunked prefill、speculative decoding 和 PD 分离都应该放在同一张系统图里理解。

一个好的面试回答要同时覆盖：

1. 它解决什么问题。
2. 它的核心机制是什么。
3. 它带来什么收益。
4. 它有什么代价。
5. 它适合什么 workload。
6. 它上线后看什么指标。
7. 它出问题时怎么排查。

到这里，第二十四册第一版已经完成了从最小推理框架到生产级 serving engine 的完整闭环。
