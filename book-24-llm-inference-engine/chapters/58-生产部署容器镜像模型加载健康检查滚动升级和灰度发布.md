# 第 58 章 生产部署：容器镜像、模型加载、健康检查、滚动升级和灰度发布

第 57 章我们把 inference engine 包装成了 OpenAI-compatible API。

到这里，一个 serving 系统已经具备了对外服务能力。

但能在本地跑通，不等于能在生产环境稳定运行。

生产部署会带来一组新的问题：

1. 镜像怎么构建。
2. CUDA、driver、PyTorch、NCCL 版本怎么匹配。
3. 模型权重怎么分发到节点。
4. 服务启动时什么时候算 ready。
5. 健康检查应该检查什么。
6. GPU 故障、OOM、NCCL hang 怎么处理。
7. 如何滚动升级。
8. 如何灰度发布新模型或新 engine。
9. 如何回滚。
10. 如何避免升级过程打爆线上流量。

很多推理系统的线上事故，不是 scheduler 算法错了，而是部署边界没设计好。

本章讨论如何把一个 LLM serving 系统放进生产环境。

## 58.0 本讲资料边界与第二轮精修口径

本章按第二轮精修口径，只讲教学版 LLM serving 生产部署的工程边界。

公开资料校准主要参考四类口径：

1. Kubernetes probes 文档对 startup、liveness、readiness 三类 probe 的语义说明，以及 Deployment rolling update 中 `maxSurge`、`maxUnavailable` 的公开行为。
2. NVIDIA Container Toolkit、CUDA compatibility 和 NCCL 文档对 GPU runtime、driver、CUDA、容器运行时与多卡通信健康检查的公开边界说明。
3. vLLM production serving、OpenAI-compatible server、distributed serving 和 health endpoint 相关公开文档对推理服务启动、健康检查、模型加载和多 worker 部署的工程抽象说明。
4. 本书第 55 到 57 章对 request router、parallel group、分布式 KV、API compatibility、鉴权和限流边界的教学抽象。

本章不写成 Kubernetes YAML 教程，也不绑定某个云厂商、镜像仓库、GPU 型号、driver 版本、NCCL 环境变量、真实 Helm chart、真实 service mesh、真实 autoscaler、真实灰度平台或真实生产 SLO。我们只保留一个可复盘的部署闭环：

```text
image/runtime compatibility -> model artifact manifest -> staged startup -> startup/live/ready probes -> worker register -> rolling capacity -> drain -> canary routing -> regression gate -> rollback path -> observability
```

第二轮新增 demo 的验收重点是：镜像、runtime 和模型 manifest 要可校验；worker 只有在模型、tokenizer、parallel group、KV cache 和 warmup 都完成后才 ready；滚动升级期间不能让有效 ready 容量跌破下限；drain 不能接新请求但要处理已有请求；canary 必须有稳定分流、回滚门禁和 revision 级观测。

## 58.2 为什么 LLM serving 部署更难

普通 HTTP 服务部署通常关心：

1. 镜像大小。
2. 配置注入。
3. 端口监听。
4. 健康检查。
5. 日志和 metrics。
6. 扩缩容。

LLM serving 还额外关心：

1. GPU 是否可见。
2. GPU driver 是否兼容。
3. CUDA runtime 是否匹配。
4. PyTorch 是否能加载 CUDA kernel。
5. NCCL 是否能建立通信。
6. 模型权重是否完整。
7. tokenizer、config、chat template 是否匹配。
8. 权重加载需要多久。
9. KV cache 预分配是否成功。
10. warmup 是否完成。
11. 多卡 parallel group 是否全部 ready。
12. 节点之间网络是否满足要求。

也就是说，LLM serving 的 ready 不是进程启动。

真正的 ready 是：

```text
进程已启动
+ GPU 可用
+ runtime 兼容
+ 模型权重加载完成
+ tokenizer/config 加载完成
+ parallel group 初始化完成
+ KV cache 分配完成
+ warmup 通过
+ API 层可以接收请求
```

少任何一步，都可能导致流量进入后失败。

## 58.3 生产部署的基本形态

一个典型部署形态是：

```text
Client
  |
  v
API Gateway / Ingress
  |
  v
OpenAI-compatible API Service
  |
  v
Request Router
  |
  +--> Worker Group A: model=v1, engine=old
  |
  +--> Worker Group B: model=v1, engine=new
  |
  +--> Worker Group C: model=v2, engine=new
```

在简单系统里，API service、router、worker 可能在同一个进程。

在复杂系统里，它们会拆成多个组件。

生产部署要回答几个问题：

1. 每个组件如何打包。
2. 每个组件如何配置。
3. 每个组件如何发现其他组件。
4. worker 如何注册到 router。
5. router 如何判断 worker 可用。
6. worker 下线前如何处理已有请求。
7. 升级时如何保证新老版本共存。

这些问题比“能不能 forward 一次模型”更接近真实工程。

## 58.4 镜像构建的核心原则

LLM serving 镜像通常很大。

原因包括：

1. CUDA runtime。
2. cuDNN、NCCL、TensorRT 等 GPU 库。
3. PyTorch 或其他深度学习框架。
4. 自定义 CUDA kernel。
5. tokenizer 相关依赖。
6. Python 包。
7. 可能还包含模型权重。

镜像构建的核心原则是：

1. runtime 版本固定。
2. 依赖版本固定。
3. 构建过程可复现。
4. 镜像职责清晰。
5. 权重是否进镜像要明确取舍。

一个常见分层是：

```text
base image:
  cuda runtime
  system libraries

framework image:
  pytorch
  transformers
  tokenizers
  nccl-related dependencies

engine image:
  serving engine code
  custom kernels
  api server

model artifact:
  weights
  config
  tokenizer
  chat template
```

不要把所有东西都混在一个不可解释的镜像里。

否则一旦线上出问题，很难判断是代码变了、依赖变了，还是模型变了。

## 58.5 CUDA、Driver、PyTorch 和 NCCL

GPU serving 最容易踩的坑之一是版本矩阵。

需要同时考虑：

1. 宿主机 NVIDIA driver。
2. 容器内 CUDA runtime。
3. PyTorch 编译使用的 CUDA 版本。
4. 自定义 kernel 编译使用的 CUDA 版本。
5. NCCL 版本。
6. GPU 架构，比如 A100、H100、L40S。

一个典型错误是：

```text
本地 A100 + CUDA 11.8 跑通
线上 H100 + CUDA 12.x 失败
```

或者：

```text
单卡跑通
多卡 NCCL 初始化 hang
```

生产环境应该维护一张明确的兼容矩阵：

```text
GPU: A100
Driver: 535.xx
Container CUDA: 12.1
PyTorch: 2.x cu121
NCCL: 2.x
Engine: commit abc123
Kernel build: sm80

GPU: H100
Driver: 550.xx
Container CUDA: 12.4
PyTorch: 2.x cu124
NCCL: 2.x
Engine: commit def456
Kernel build: sm90
```

不要依赖“应该兼容”。

要用启动自检和 smoke test 证明兼容。

## 58.6 镜像里要不要放模型权重

这是一个重要设计选择。

方案一：权重放进镜像。

优点：

1. 镜像即完整交付物。
2. 部署时不需要额外下载模型。
3. 版本对应关系清晰。
4. 启动路径更少依赖外部系统。

缺点：

1. 镜像非常大。
2. 构建和推送很慢。
3. 每次模型更新都要重建镜像。
4. 镜像仓库存储压力大。
5. 多模型场景不灵活。

方案二：权重不放进镜像，启动时从对象存储或共享文件系统加载。

优点：

1. 镜像较小。
2. 代码和模型可以独立发布。
3. 多模型管理更灵活。
4. 节点本地缓存可以复用。

缺点：

1. 首次启动慢。
2. 依赖对象存储或文件系统可用性。
3. 权重完整性校验更重要。
4. 需要处理并发下载和缓存淘汰。

方案三：节点预热，把权重提前同步到本地盘。

优点：

1. worker 启动快。
2. 不需要每个 pod 重复下载。
3. 适合大模型和固定模型池。

缺点：

1. 节点运维复杂。
2. 本地盘容量管理复杂。
3. 模型版本清理需要额外机制。

生产中经常采用方案二或三。

模型权重太大时，直接放进镜像通常会拖慢整个发布链路。

## 58.7 模型目录必须版本化

模型不是只有 `.safetensors` 或 `.bin`。

一个可服务模型至少包括：

1. 权重文件。
2. `config.json`。
3. tokenizer 文件。
4. tokenizer config。
5. generation config。
6. chat template。
7. special tokens 配置。
8. quantization config。
9. tensor parallel 切分配置。

生产部署时必须把这些东西作为一个整体版本管理。

错误做法是：

```text
权重来自 model-v2
tokenizer 来自 model-v1
chat template 是手写旧版本
```

这种错误不一定会导致服务启动失败。

更危险的是它可能只导致输出质量下降。

所以模型目录应该有 manifest：

```json
{
  "model_id": "chat-model-v2",
  "revision": "2026-05-31-001",
  "base_model": "example/base-model",
  "weight_format": "safetensors",
  "dtype": "bfloat16",
  "tensor_parallel_size": 4,
  "tokenizer_hash": "...",
  "config_hash": "...",
  "chat_template_hash": "...",
  "weights_hash": "..."
}
```

启动时至少要校验：

1. manifest 存在。
2. 必要文件存在。
3. hash 匹配。
4. dtype 和 engine 配置匹配。
5. tensor parallel size 和部署配置匹配。
6. tokenizer 能正常 encode/decode。
7. chat template 能渲染一个最小 messages。

## 58.8 启动流程应该分阶段

不要让 worker 一启动就对外 ready。

一个合理启动流程是：

```text
1. 读取配置
2. 检查 GPU 可见性
3. 检查 CUDA / torch / NCCL 基本可用性
4. 拉取或定位模型权重
5. 校验模型 manifest 和文件 hash
6. 加载 tokenizer / config / chat template
7. 加载模型权重到 CPU 或 GPU
8. 初始化 tensor parallel / pipeline parallel group
9. 初始化 KV cache block manager
10. 执行 warmup prefill
11. 执行 warmup decode
12. 注册到 router
13. readiness probe 返回成功
```

其中任何一步失败，都不应该让流量进入。

这和普通 Web 服务非常不同。

普通服务可能监听端口后就 ready。

LLM worker 必须等模型真的可用。

## 58.9 Cold Start 为什么重要

LLM worker cold start 可能非常慢。

原因包括：

1. 下载几十 GB 到几百 GB 的权重。
2. 从磁盘读取权重。
3. 权重搬到 GPU。
4. 初始化 CUDA context。
5. 编译或加载 custom kernel。
6. 初始化 NCCL communicator。
7. 预分配 KV cache。
8. warmup kernel。

普通服务扩容可能几十秒。

LLM serving 扩容可能几分钟甚至更久。

这会影响：

1. 自动扩缩容策略。
2. 滚动升级速度。
3. 故障恢复时间。
4. 峰值流量承载能力。

所以 LLM serving 不能只依赖“流量来了再扩容”。

需要提前预热容量。

## 58.10 Warmup 要做什么

warmup 不是形式主义。

它可以提前暴露很多问题。

最小 warmup 包括：

1. tokenizer encode 一段最小 prompt。
2. chat template 渲染一段 messages。
3. 执行一次短 prefill。
4. 执行几步 decode。
5. 验证输出 token 合法。
6. 验证 streaming 路径可用。
7. 验证 usage 统计不崩。

多 GPU worker 还要验证：

1. all-reduce 可用。
2. all-gather 可用。
3. NCCL communicator 没有 hang。
4. rank 配置正确。
5. parallel group 中所有 rank 都完成 warmup。

一个 warmup 失败的 worker 不应该注册到 router。

如果它已经注册，则必须立刻摘除。

## 58.11 Health Check 分层

健康检查通常至少分三类：

1. liveness。
2. readiness。
3. startup。

### 58.11.1 Liveness

liveness 表示进程是否还活着。

例如：

```text
GET /health/live
```

它可以检查：

1. event loop 是否响应。
2. 主进程是否未进入死锁。
3. HTTP server 是否还能处理请求。

liveness 不应该做很重的模型 forward。

否则健康检查本身会消耗 GPU。

如果 liveness 失败，平台通常会重启进程。

### 58.11.2 Readiness

readiness 表示是否可以接收业务流量。

例如：

```text
GET /health/ready
```

readiness 应该检查：

1. 模型已加载。
2. tokenizer 已加载。
3. KV cache 已初始化。
4. warmup 已完成。
5. worker 已连接 router。
6. 当前不在 drain 状态。
7. GPU 没有进入不可恢复错误状态。

如果 readiness 失败，平台应该停止给它转发新流量。

但不一定要重启。

### 58.11.3 Startup Probe

startup probe 用来给慢启动服务更多时间。

LLM worker 加载模型可能需要很久。

如果只用 liveness，平台可能误以为它卡死，然后不断重启。

这会形成循环：

```text
加载模型 3 分钟
平台 1 分钟未通过健康检查
重启
再次加载模型
再次重启
```

startup probe 的作用是告诉平台：

```text
启动期间请耐心等待，不要过早杀掉进程。
```

## 58.12 Readiness 不是简单返回 200

readiness 的核心是状态机。

可以定义 worker 状态：

```text
STARTING
DOWNLOADING_MODEL
LOADING_MODEL
INITIALIZING_PARALLEL_GROUP
WARMING_UP
READY
DRAINING
UNHEALTHY
STOPPING
```

只有 `READY` 可以接收新请求。

`DRAINING` 表示：

```text
不接收新请求，但继续处理已有请求。
```

`UNHEALTHY` 表示：

```text
不接收新请求，并且可能需要重启或人工介入。
```

router 不应该只看 HTTP 200。

更好的方式是 worker 周期性上报：

1. state。
2. model id。
3. model revision。
4. engine version。
5. parallel group id。
6. available KV blocks。
7. running requests。
8. queue length。
9. recent error rate。
10. last heartbeat time。

## 58.13 Worker 注册和摘除

worker 启动完成后，需要注册到 router。

注册信息可以包括：

```json
{
  "worker_id": "worker-a-001",
  "model_id": "chat-model-v1",
  "model_revision": "2026-05-31-001",
  "engine_version": "abc123",
  "endpoint": "10.0.1.23:9000",
  "tensor_parallel_size": 4,
  "max_context_length": 32768,
  "supported_features": ["stream", "json_mode"],
  "state": "READY"
}
```

router 收到注册后，不应该立刻永久相信它。

还要持续依赖 heartbeat。

如果 heartbeat 超时，router 应该把 worker 标记为不可用。

摘除 worker 有两种：

1. 非优雅摘除。
2. 优雅 drain。

非优雅摘除用于 crash、GPU error、节点失联。

优雅 drain 用于发布、缩容、维护。

## 58.14 Drain 的正确语义

drain 是生产部署里非常重要的概念。

它的语义是：

```text
停止接收新请求，等待已有请求完成或超时，然后退出。
```

LLM serving 的 drain 比普通服务更难。

原因是生成请求可能持续很久。

特别是：

1. stream 请求。
2. 长上下文请求。
3. max_tokens 很大的请求。
4. agent 多轮调用。

一个 worker 进入 drain 后：

1. readiness 应该变为失败。
2. router 不再分配新请求。
3. 已有请求继续运行。
4. 新到达该 worker 的请求返回 retryable error 或被 router 重试。
5. 超过 drain timeout 后，剩余请求被 cancel。
6. 释放 KV cache 和并发 slot。
7. worker 安全退出。

如果没有 drain，滚动升级会直接中断线上 stream。

用户体验会很差。

## 58.15 滚动升级的基本流程

滚动升级目标是：

```text
逐步替换老版本 worker，同时保持服务可用。
```

一个基本流程：

```text
1. 构建新镜像
2. 部署少量新 worker
3. 新 worker 启动、加载模型、warmup
4. 新 worker 通过 readiness
5. router 开始给新 worker 少量流量
6. 观察 metrics 和 error rate
7. 老 worker 进入 drain
8. 老 worker 处理完已有请求后退出
9. 继续替换下一批 worker
10. 全部替换完成
```

关键点是：

1. 新 worker ready 后才减少老 worker。
2. 老 worker drain 完成前不要强杀。
3. 任何阶段发现异常都可以暂停。
4. 保留足够冗余容量。

LLM serving 升级尤其要注意容量。

如果同时启动新 worker 和保留老 worker，GPU 资源可能不够。

如果先杀老 worker 再启动新 worker，容量会下降。

所以滚动升级需要和资源调度一起设计。

## 58.16 Max Surge 和 Max Unavailable

在 Kubernetes 里，滚动升级常见参数是：

1. max surge。
2. max unavailable。

含义可以简化理解为：

```text
max surge: 升级过程中允许额外创建多少新实例
max unavailable: 升级过程中允许多少老实例不可用
```

对普通服务，可以设置较大的 max surge。

但对 GPU worker，资源通常很紧。

你可能没有额外 GPU 创建新 worker。

这时升级策略要更谨慎。

常见选择：

1. 预留一部分 GPU 作为升级缓冲。
2. 在低峰期升级。
3. 先缩小部分低优先级模型副本。
4. 使用独立灰度集群。
5. 按节点逐批替换。

不要假设 GPU 集群像 CPU Web 服务一样可以随时 surge。

## 58.17 灰度发布是什么

灰度发布是让一小部分流量先使用新版本。

新版本可能是：

1. 新模型权重。
2. 新 tokenizer。
3. 新 chat template。
4. 新推理 engine。
5. 新 scheduler 参数。
6. 新 CUDA kernel。
7. 新 API 行为。

灰度发布不是简单“部署一个新 pod”。

关键是 router 要能按策略分流。

例如：

```text
model=chat-model-v1:
  old revision: 95%
  new revision: 5%
```

或者：

```text
tenant=A: 100% new
tenant=B: 0% new
internal users: 100% new
public users: 1% new
```

灰度的本质是把风险控制在可观测、可回滚的范围内。

## 58.18 灰度维度

常见灰度维度包括：

1. 按请求比例。
2. 按租户。
3. 按 API key。
4. 按用户 id hash。
5. 按模型名 alias。
6. 按 region。
7. 按 endpoint。
8. 按请求类型。

按比例灰度简单，但可能不稳定。

同一个用户的请求有时走新版本，有时走旧版本，会导致体验不一致。

按 user hash 灰度更稳定：

```text
hash(user_id) % 100 < 5 -> new version
else -> old version
```

按租户灰度适合企业客户。

按模型 alias 灰度适合模型发布：

```text
chat-latest -> chat-v2 for 5% traffic
chat-latest -> chat-v1 for 95% traffic
```

## 58.19 灰度时要观察什么

灰度不是只看服务有没有 500。

LLM serving 要观察：

1. HTTP error rate。
2. engine error rate。
3. timeout rate。
4. TTFT。
5. TPOT。
6. output tokens per second。
7. GPU utilization。
8. GPU memory usage。
9. KV cache block usage。
10. preemption 次数。
11. OOM 次数。
12. NCCL error。
13. client disconnect rate。
14. 平均输出长度。
15. finish_reason 分布。
16. 内容安全拦截率。
17. 质量评测分数。
18. 用户反馈。

只看技术指标不够。

新模型可能没有报错，但回答质量下降。

所以模型灰度还应该接入离线评测和线上采样评测。

## 58.20 回滚必须提前设计

回滚不是事故发生后临时想办法。

上线前就要回答：

1. 如何把 router 流量切回旧版本。
2. 旧版本 worker 是否还保留。
3. 旧模型权重是否还在节点本地缓存。
4. 旧镜像是否还可拉取。
5. 配置是否可以回滚。
6. 数据结构是否兼容。
7. API 行为是否兼容。

对 serving 系统来说，最常见的回滚是流量回滚：

```text
new version weight: 5% -> 0%
old version weight: 95% -> 100%
```

这比重新部署旧版本更快。

所以灰度期间不要太早删除旧 worker。

如果旧 worker 已经全部下线，回滚就变成重新冷启动。

冷启动可能需要几分钟，事故窗口会被拉长。

## 58.21 配置发布也要灰度

很多事故不是代码导致的，而是配置导致的。

LLM serving 常见配置包括：

1. max batch size。
2. max num batched tokens。
3. max context length。
4. GPU memory utilization。
5. block size。
6. scheduler policy。
7. preemption mode。
8. prefix cache 开关。
9. stream flush interval。
10. timeout。
11. rate limit。
12. tenant quota。

这些配置改错，会直接影响容量和稳定性。

例如：

```text
max_num_batched_tokens 从 8192 改到 65536
```

可能导致吞吐提升，也可能导致 TTFT 飙升或 OOM。

配置发布应该具备：

1. 版本号。
2. 审计记录。
3. 灰度范围。
4. 自动校验。
5. 一键回滚。
6. 与 metrics 联动的告警。

不要把配置当成无风险修改。

## 58.22 模型加载失败怎么处理

模型加载失败很常见。

原因包括：

1. 权重文件缺失。
2. 文件 hash 不匹配。
3. tokenizer 文件缺失。
4. dtype 不支持。
5. GPU 显存不足。
6. tensor parallel size 不匹配。
7. quantization kernel 不支持当前 GPU。
8. safetensors 文件损坏。
9. 对象存储超时。
10. 本地磁盘空间不足。

处理原则：

1. 启动失败时不要 ready。
2. 错误日志要明确具体文件和原因。
3. 不要反复高速重启打爆对象存储。
4. 对可重试错误做 backoff。
5. 对不可重试错误快速失败。
6. 暴露启动阶段 metrics。

例如对象存储超时可以重试。

但 manifest hash 不匹配不应该无限重试。

## 58.23 GPU OOM 的生产处理

GPU OOM 分两类：

1. 启动期 OOM。
2. 运行期 OOM。

启动期 OOM 通常说明：

1. 模型太大。
2. dtype 配错。
3. tensor parallel size 太小。
4. GPU memory utilization 配太高。
5. 预分配 KV cache 过大。

运行期 OOM 通常说明：

1. admission control 不严。
2. max context length 太大。
3. max batch tokens 太大。
4. KV block manager 有泄漏。
5. preemption 或 eviction 失效。
6. 某些请求绕过了限流。

生产处理原则：

1. 尽量在 admission 阶段拒绝超容量请求。
2. OOM 后把 worker 标记 unhealthy。
3. 记录触发 OOM 前的 batch 和请求摘要。
4. 不要记录完整 prompt。
5. 必要时重启 worker。
6. 调低相关配置后再恢复流量。

GPU OOM 后进程不一定还能处于可靠状态。

不要假设 catch 到异常就万事大吉。

## 58.24 NCCL Hang 怎么办

多 GPU serving 中，NCCL hang 是很麻烦的问题。

它可能表现为：

1. 请求一直不返回。
2. GPU utilization 异常。
3. 某个 rank 卡住。
4. 健康检查还能返回，但模型 forward 卡住。

常见原因：

1. rank 配置错误。
2. 某个 rank 进程崩溃。
3. 网络异常。
4. NCCL 版本和 driver 不匹配。
5. collective 调用顺序不一致。
6. 某个请求路径只在部分 rank 执行。

生产上需要：

1. forward 超时。
2. rank 级 heartbeat。
3. NCCL error 日志。
4. hang detector。
5. worker group 整体摘除。
6. 必要时重启整个 parallel group。

对于 tensor parallel group，不要只重启一个 rank。

通常要把整个 group 当成一个 failure domain。

## 58.25 日志、Metrics 和 Trace

生产部署必须让系统可观测。

日志要回答：

1. 哪个请求失败。
2. 失败在哪个阶段。
3. 进入了哪个 worker。
4. 使用哪个模型版本。
5. 是否 stream。
6. 是否被 cancel。
7. 是否触发 rate limit 或 admission control。

metrics 要回答：

1. 系统现在是否健康。
2. 容量是否足够。
3. 延迟是否变差。
4. GPU 是否成为瓶颈。
5. KV cache 是否紧张。
6. 新版本是否比旧版本差。

trace 要回答：

1. 请求在 API 层花了多久。
2. router 排队多久。
3. prefill 花了多久。
4. decode 每步花了多久。
5. stream flush 是否及时。
6. 哪个 worker 或 rank 慢。

没有这些信息，灰度和回滚只能靠猜。

## 58.26 生产告警

告警不要只配 GPU utilization。

LLM serving 常见告警包括：

1. 5xx error rate 高。
2. 429 突增。
3. 503 overload 突增。
4. TTFT P95/P99 升高。
5. TPOT P95/P99 升高。
6. queue waiting time 升高。
7. running requests 接近上限。
8. KV cache free blocks 过低。
9. GPU memory 接近上限。
10. worker heartbeat 超时。
11. readiness 失败数量过多。
12. model load failure。
13. NCCL error。
14. OOM。
15. client disconnect rate 突增。

告警应该带上维度：

1. model id。
2. model revision。
3. engine version。
4. tenant。
5. region。
6. worker group。

否则你只知道“系统坏了”，不知道“哪个版本坏了”。

## 58.27 自动扩缩容的难点

普通服务可以按 CPU 或 QPS 扩缩容。

LLM serving 更复杂。

因为请求成本差异很大。

扩容指标可以包括：

1. queue waiting time。
2. TTFT。
3. tokens per second。
4. running requests。
5. pending requests。
6. KV cache pressure。
7. GPU utilization。
8. admission rejection rate。

但即使触发扩容，新 worker 也需要 cold start。

所以自动扩容适合处理缓慢变化的流量。

对于突发峰值，更依赖：

1. 预留容量。
2. 限流。
3. 降级。
4. 请求排队。
5. 按租户优先级调度。

自动缩容也要谨慎。

缩容前必须 drain。

而且要考虑模型权重缓存是否会被清理。

## 58.28 多模型部署

生产环境通常不只一个模型。

可能有：

1. chat 模型。
2. code 模型。
3. embedding 模型。
4. rerank 模型。
5. small 模型。
6. large 模型。
7. 多个 revision。

多模型部署有几种方式：

1. 每个 worker 固定一个模型。
2. 一个 worker 支持多个小模型。
3. 动态加载和卸载模型。
4. 按模型池分集群。

大模型 serving 通常采用固定模型 worker。

原因是模型加载太慢，动态切换成本太高。

小模型或 embedding 模型可以考虑多模型共享 worker。

router 必须知道每个 worker 支持哪些模型和 revision。

不要把请求发给不支持该模型的 worker。

## 58.29 Blue-Green 部署

Blue-Green 是另一种发布方式。

假设当前生产环境是 Blue。

新版本部署到 Green。

流程是：

```text
1. Blue 承接 100% 流量
2. Green 完整部署并 warmup
3. 对 Green 做内部验证
4. router 或 gateway 把流量切到 Green
5. 观察一段时间
6. 如果有问题，切回 Blue
```

优点：

1. 回滚很快。
2. 新环境可以完整验证。
3. 新老版本隔离清晰。

缺点：

1. 需要接近双倍资源。
2. GPU 成本高。
3. 大模型权重需要两套容量。

对于关键模型或重大版本升级，Blue-Green 很有价值。

对于资源紧张的常规升级，滚动升级和灰度更常见。

## 58.30 Canary 部署

Canary 更强调小流量试探。

流程是：

```text
1. 部署少量 canary worker
2. 路由 1% 流量到 canary
3. 对比 canary 和 stable 指标
4. 指标正常后扩大到 5%、10%、25%、50%
5. 最终全量
```

Canary 需要强依赖指标对比。

对 LLM serving 来说，最好同时比较：

1. 技术指标。
2. 生成长度分布。
3. 错误分布。
4. 安全策略触发率。
5. 抽样质量评测。

如果只比较 HTTP 500，canary 的价值会大幅下降。

## 58.31 模型 Alias 和版本路由

生产 API 通常不希望用户直接绑定内部 revision。

可以提供 alias：

```text
chat-latest -> chat-v2-20260531
chat-stable -> chat-v1-20260401
chat-fast -> chat-small-v3
```

router 根据 alias 解析到具体 revision。

灰度时 alias 可以映射到多个 revision：

```text
chat-latest:
  chat-v1-20260401: 90%
  chat-v2-20260531: 10%
```

这样用户请求仍然是：

```json
{
  "model": "chat-latest",
  "messages": [...]
}
```

内部可以逐步迁移。

注意：

1. 日志必须记录 alias 和 resolved revision。
2. usage 计费要基于实际 revision 或产品策略。
3. 回放和 debug 时必须知道当时解析到哪个版本。

## 58.32 部署前检查清单

每次发布前可以检查：

1. 镜像 digest 是否固定。
2. 代码 commit 是否明确。
3. CUDA/PyTorch/NCCL 版本是否在兼容矩阵内。
4. 模型 manifest 是否完整。
5. 权重 hash 是否校验通过。
6. tokenizer 和 chat template 是否匹配。
7. warmup case 是否通过。
8. benchmark 是否没有明显退化。
9. 配置 diff 是否审查。
10. 灰度策略是否设置。
11. 回滚策略是否确认。
12. 告警是否启用。
13. dashboard 是否能区分新老版本。
14. drain timeout 是否合理。
15. 资源容量是否足够。

这类清单很朴素，但能避免大量事故。

## 58.34 最小 Worker 状态机

一个 worker 可以用这样的状态机管理生命周期：

```text
STARTING
  -> DOWNLOADING_MODEL
  -> LOADING_MODEL
  -> INITIALIZING
  -> WARMING_UP
  -> READY
  -> DRAINING
  -> STOPPING
```

失败路径：

```text
any state -> UNHEALTHY
```

readiness 逻辑：

```text
if state == READY:
    return 200
else:
    return 503
```

router 分流逻辑：

```text
eligible_workers = workers.filter(
    state == READY
    and model_id matches
    and revision allowed by traffic policy
    and capacity available
)
```

这个状态机比简单的布尔变量更可靠。

## 58.35 常见错误

错误一：进程启动就 readiness 返回成功。

```text
结果：流量进入后发现模型还没加载完。
```

错误二：没有 startup probe。

```text
结果：大模型加载时被平台反复重启。
```

错误三：没有 drain。

```text
结果：滚动升级直接中断线上 stream 请求。
```

错误四：新 worker 没 ready 就杀老 worker。

```text
结果：升级过程中容量骤降，TTFT 和 503 飙升。
```

错误五：模型权重、tokenizer、chat template 没有整体版本化。

```text
结果：服务能跑，但输出质量异常。
```

错误六：灰度只看 500。

```text
结果：质量退化和延迟退化没有及时发现。
```

错误七：旧 worker 过早删除。

```text
结果：发现问题后只能重新 cold start 回滚。
```

错误八：多卡 worker 只重启单个 rank。

```text
结果：parallel group 状态不一致，继续 hang 或错误。
```

错误九：配置直接全量发布。

```text
结果：一个 batch 参数改错导致全站 OOM。
```

错误十：日志没有记录模型 revision。

```text
结果：线上问题无法判断是哪个模型版本造成的。
```

## 58.36 面试高频问题

问题一：LLM serving 的 readiness 应该检查什么？

回答要点：不能只检查进程和端口。应该检查模型权重加载完成、tokenizer/config/chat template 加载完成、GPU 和 CUDA 可用、parallel group 初始化完成、KV cache 初始化完成、warmup 通过、worker 已注册到 router，并且当前不处于 drain 或 unhealthy 状态。

问题二：为什么 LLM worker 需要 startup probe？

回答要点：模型加载和 warmup 可能需要几分钟。如果只有 liveness probe，平台可能在启动期间误判服务卡死并反复重启，导致永远无法完成加载。startup probe 用来给慢启动过程足够时间。

问题三：滚动升级 LLM serving 有哪些风险？

回答要点：新 worker cold start 慢，GPU 资源无法随意 surge；如果新 worker 没 ready 就下线老 worker，会造成容量下降；如果没有 drain，会中断已有 stream；如果灰度和回滚没有设计好，新版本异常会影响大量请求。

问题四：模型灰度要观察哪些指标？

回答要点：除了 HTTP error rate，还要看 TTFT、TPOT、timeout、GPU memory、KV cache pressure、preemption、OOM、client disconnect、finish_reason 分布、输出长度分布、安全拦截率，以及离线或线上抽样质量评测。

问题五：模型权重应该放进镜像吗？

回答要点：小模型或强版本绑定场景可以放进镜像，优点是交付物完整；大模型通常不适合，因为镜像过大、构建和分发慢。更常见的是权重放对象存储或节点本地缓存，启动时校验 manifest 和 hash，但要处理 cold start、缓存和外部依赖可用性。

问题六：为什么回滚要保留旧 worker？

回答要点：LLM worker cold start 很慢。如果旧 worker 已经全部删除，回滚需要重新拉权重、加载模型、warmup，事故恢复时间会很长。灰度期间保留旧 worker，可以通过 router 快速把流量切回旧版本。

## 58.37 标准回答模板

如果面试官问“你会如何把一个 LLM serving engine 部署到生产环境”，可以这样回答：

```text
我会先把交付物拆清楚：镜像里固定 CUDA runtime、PyTorch、NCCL、自定义 kernel 和 serving 代码；模型权重、tokenizer、config、chat template 作为一个带 manifest 的模型版本管理，启动时校验 hash 和配置兼容性。CUDA、driver、PyTorch、NCCL 和 GPU 型号会维护明确的兼容矩阵，并用启动自检和 warmup 验证。

worker 的启动会分阶段：读取配置、检查 GPU、加载或下载模型、校验 manifest、加载 tokenizer 和权重、初始化 TP/PP group、初始化 KV cache，然后执行 prefill 和 decode warmup。只有全部成功后才注册到 router，并让 readiness 返回成功。liveness 只检查进程和 event loop，startup probe 给大模型加载足够时间，readiness 表示是否能接收业务流量。

发布上我会支持滚动升级和灰度。新 worker 必须先 ready，再逐步接流量；老 worker 下线前进入 drain，停止接收新请求但继续处理已有 stream，超过 drain timeout 再 cancel。router 支持按模型 alias、revision、租户或 user hash 做 canary 分流。灰度期间观察 error rate、TTFT、TPOT、KV cache、OOM、NCCL error、输出长度和质量评测。

回滚会提前设计。灰度期间保留旧 worker 和旧模型缓存，如果新版本异常，router 直接把流量权重切回旧 revision，而不是重新 cold start。配置发布也要版本化、灰度和可回滚，尤其是 max batch tokens、context length、GPU memory utilization、scheduler policy 这类会影响容量的参数。
```

## 58.38 Production Deployment 公式、发布门禁和可运行 demo

把生产部署压缩成公式，可以先看 readiness 门禁：

```math
G_{\mathrm{ready}}=G_{\mathrm{proc}}G_{\mathrm{gpu}}G_{\mathrm{runtime}}G_{\mathrm{model}}G_{\mathrm{tok}}G_{\mathrm{pg}}G_{\mathrm{kv}}G_{\mathrm{warm}}
```

cold start 时间可以粗略拆成：

```math
T_{\mathrm{cold}}=T_{\mathrm{image}}+T_{\mathrm{weights}}+T_{\mathrm{load}}+T_{\mathrm{init}}+T_{\mathrm{warm}}
```

滚动升级时刻 `t` 的有效容量：

```math
C_{\mathrm{roll}}(t)=C_{\mathrm{ready,new}}(t)+C_{\mathrm{ready,old}}(t)-C_{\mathrm{drain}}(t)
```

容量门禁：

```math
C_{\mathrm{roll}}(t)\ge C_{\min}
```

canary 流量比例：

```math
R_{\mathrm{canary}}=\frac{N_{\mathrm{canary}}}{\max(1,N_{\mathrm{total}})}
```

canary 不触发回滚的门禁：

```math
G_{\mathrm{rollback}}=\mathbf{1}[E_{\mathrm{canary}}\le E_{\max}]\mathbf{1}[L_{\mathrm{canary}}\le L_{\max}]
```

最终生产部署门禁：

```math
G_{\mathrm{deploy}}=G_{\mathrm{artifact}}G_{\mathrm{ready}}G_{\mathrm{roll}}G_{\mathrm{canary}}G_{\mathrm{rollback}}G_{\mathrm{obs}}
```

    canary_errors: int
    canary_latency_p95_ms: int


class ToyProductionDeploymentAuditor:
    def __init__(self, image: ImageArtifact, model: ModelArtifact, config: DeploymentConfig):
        self.image = image
        self.model = model
        self.config = config
        self.router: dict[str, WorkerState] = {}
        self.metrics = {
            "startup_failure_total": 0,
            "readiness_fail_total": 0,
            "drain_total": 0,
            "canary_requests_total": 0,
            "rollback_total": 0,
        }

    def runtime_compatible(self, node_driver: int, gpu_arch: str) -> bool:
        return (
            node_driver >= self.image.min_driver
            and gpu_arch in self.image.gpu_arches
            and self.image.cuda_runtime == self.model.expected_cuda
            and {"/health/live", "/health/ready", "/health/startup"} <= self.image.health_endpoints
        )

    def artifact_ready(self) -> bool:
        required = {
            "model_hash": self.model.model_hash,
            "tokenizer_hash": self.model.tokenizer_hash,
            "config_hash": self.model.config_hash,
            "chat_template_hash": self.model.chat_template_hash,
        }
        return all(value and self.model.manifest.get(key) == value for key, value in required.items())

    def start_worker(self, worker: WorkerState, runtime_ok: bool, artifact_ok: bool) -> WorkerState:
        checks = {
            "process": True,
            "gpu": runtime_ok,
            "runtime": runtime_ok,
            "model": artifact_ok,
            "tokenizer": artifact_ok,
            "parallel_group": runtime_ok,
            "kv_cache": artifact_ok,
            "warmup": runtime_ok and artifact_ok,
        }
        worker.phases = checks
        if all(checks.values()):
            worker.status = "READY"
            worker.accepts_new = True
        else:
            worker.status = "UNHEALTHY"
            self.metrics["startup_failure_total"] += 1
        return worker

    def probes(self, worker: WorkerState) -> dict[str, bool]:
        live = worker.status in {"STARTING", "READY", "DRAINING"}
        startup = bool(worker.phases) and all(worker.phases.values())
        ready = worker.status == "READY" and worker.accepts_new and worker.registered
        if not ready:
            self.metrics["readiness_fail_total"] += 1
        return {"live": live, "startup": startup, "ready": ready}

    def register_worker(self, worker: WorkerState):
        if worker.status == "READY":
            worker.registered = True
            self.router[worker.worker_id] = worker

    def drain_worker(self, worker: WorkerState) -> list[str]:
        events = [f"{worker.worker_id}:stop_accepting"]
        worker.status = "DRAINING"
        worker.accepts_new = False
        while worker.active_requests > 0:
            worker.active_requests -= 1
            events.append(f"{worker.worker_id}:finished_existing")
        worker.status = "DRAINED"
        self.metrics["drain_total"] += 1
        events.append(f"{worker.worker_id}:drained")
        return events

    def rolling_capacity(self, workers: list[WorkerState]) -> dict[str, int | bool]:
        ready_new = sum(w.capacity for w in workers if w.revision == self.config.new_revision and w.status == "READY")
        ready_old = sum(w.capacity for w in workers if w.revision == self.config.old_revision and w.status in {"READY", "DRAINING"})
        draining = sum(w.capacity for w in workers if w.status == "DRAINING")
        effective = ready_new + ready_old - draining
        return {
            "ready_new": ready_new,
            "ready_old": ready_old,
            "draining": draining,
            "effective": effective,
            "min": self.config.min_ready_capacity,
            "capacity_ok": effective >= self.config.min_ready_capacity,
        }

    def route_canary(self, users: list[str]) -> dict[str, int]:
        counts = {self.config.old_revision: 0, self.config.new_revision: 0}
        for user in users:
            bucket = sum(ord(ch) for ch in user) % 100
            revision = self.config.new_revision if bucket < self.config.canary_percent else self.config.old_revision
            counts[revision] += 1
        self.metrics["canary_requests_total"] = counts[self.config.new_revision]
        return counts

    def canary_regression_gate(self, metrics: CanaryMetrics) -> dict[str, float | int | bool]:
        error_rate = metrics.canary_errors / max(1, metrics.canary_requests)
        canary_ratio = metrics.canary_requests / max(1, metrics.total_requests)
        passed = error_rate <= self.config.max_error_rate and metrics.canary_latency_p95_ms <= self.config.max_latency_p95_ms
        if not passed:
            self.metrics["rollback_total"] += 1
        return {
            "error_rate": round(error_rate, 3),
            "latency_p95_ms": metrics.canary_latency_p95_ms,
            "canary_ratio": round(canary_ratio, 3),
            "pass": passed,
        }

这个 demo 证明了几个关键点：

1. 镜像 runtime、GPU driver、GPU arch、health endpoint 和模型 manifest 在启动前就能被审计。
2. readiness 必须晚于模型、tokenizer、parallel group、KV cache 和 warmup。
3. rolling update 的有效容量要显式计算，不能只看新 Pod 是否创建成功。
4. drain 语义是停止接新请求，同时让已有请求自然结束。
5. canary 不是随机上线，而是有稳定分流、指标门禁、旧 revision 保留和可回滚路径。
6. 日志和 metrics 必须带 revision，事故复盘才知道是镜像、模型还是配置造成的问题。

## 58.40 本章总结

生产部署是 LLM serving 从实验系统走向可用系统的关键一步。

LLM worker 的部署复杂度远高于普通 Web 服务。

它不仅要启动进程，还要保证 GPU runtime、模型权重、tokenizer、parallel group、KV cache 和 warmup 都正确。

镜像构建要固定 CUDA、PyTorch、NCCL 和自定义 kernel 版本。

模型权重、config、tokenizer 和 chat template 必须整体版本化。

readiness 表示能否接收业务流量，不等于进程是否存活。

liveness、readiness 和 startup probe 要分清。

滚动升级必须先让新 worker ready，再 drain 老 worker。

灰度发布要通过 router 控制流量比例，并观察技术指标和质量指标。

回滚路径必须提前设计，最好能通过流量切换快速恢复。

配置发布也要灰度和可回滚，因为 serving 参数会直接影响容量、延迟和稳定性。
