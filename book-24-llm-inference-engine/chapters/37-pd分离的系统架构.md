# 第 37 章 PD 分离的系统架构

上一章讲了 PD 分离解决什么问题：统一 engine 里 Prefill 和 Decode 混合调度，会带来 prefill interruption、decode 抖动、prefill 饥饿、资源画像不匹配和扩缩容耦合。

本章进入系统架构。

PD 分离不是简单地把代码里的 `prefill()` 和 `decode()` 拆成两个函数。它是把一个请求的生命周期拆到多个服务组件里执行：router 负责接入和调度，prefill worker 负责 prompt 计算和 KV 生成，decode worker 负责持续生成，KV transfer backend 负责把 prefill 产生的 KV cache 交给 decode 使用。

一句话概括：

> PD 分离的系统架构核心是：用 router 串起 prefill pool、decode pool 和 KV transfer backend，把请求从“单 engine 内部状态机”升级成“跨组件分布式状态机”。

## 37.2 从 unified engine 到 PD 架构

Unified engine 的请求路径通常是：

```text
client
  -> api server
  -> scheduler
  -> prefill
  -> decode loop
  -> stream output
```

所有阶段通常在同一个 serving engine 或同一组 worker 内完成。

PD 分离后，请求路径变成：

```text
client
  -> router
  -> choose prefill worker
  -> run prefill
  -> transfer KV cache
  -> choose decode worker
  -> run decode loop
  -> stream output
```

架构上至少多了三类东西：

1. Prefill pool。
2. Decode pool。
3. KV transfer / sharing 机制。

同时 router 不再只是一个普通负载均衡器。

它需要理解请求阶段、worker 类型、KV locality、transfer 状态和故障清理。

## 37.3 总体架构图

一个典型 PD 分离架构可以画成：

```text
                         metrics / health / control
                                  ^
                                  |
client -> api gateway -> PD router+----------------------+
                                  |                      |
                                  v                      v
                         prefill worker pool      decode worker pool
                         +---------------+        +--------------+
                         | prefill w0    |        | decode w0    |
                         | prefill w1    |        | decode w1    |
                         | prefill w2    |        | decode w2    |
                         +-------+-------+        +------+-------+
                                 |                       ^
                                 | KV cache + metadata   |
                                 v                       |
                         KV transfer backend ------------+
                         RDMA / NVLink / NIXL / Mooncake
```

这里有几条关键路径：

1. 控制路径：router 注册 worker、检查健康、更新负载、处理扩缩容。
2. 请求路径：client 请求经过 router，依次进入 prefill 和 decode。
3. 数据路径：prompt KV cache 从 prefill worker 转移到 decode worker。
4. 返回路径：decode worker 生成 token，router 或 API 层把 token stream 返回给 client。

不要把它理解成普通微服务拆分。

普通微服务传的是 JSON 或 protobuf。

PD 分离最重的东西是 KV cache，它在 GPU memory 里，体积大，layout 复杂，传输成本高。

## 37.4 核心组件一：PD Router

Router 是 PD 分离的中枢。

在普通 serving 中，router 可能只做：

1. 选择一个 worker。
2. 转发请求。
3. 做健康检查。
4. 做简单重试。

PD 分离中，router 至少要做：

1. 维护 prefill worker 列表。
2. 维护 decode worker 列表。
3. 分别跟踪 P/D worker 的负载。
4. 为每个请求选择 prefill worker。
5. 为每个请求选择 decode worker。
6. 协调 KV transfer。
7. 维护 request state。
8. 处理 streaming 返回。
9. 处理 abort、timeout、retry 和资源清理。

SGLang Model Gateway 的 PD 模式就是这一类 router 的例子。它支持 `--pd-disaggregation`，并通过 `--prefill` 和 `--decode` 注册不同类型的 worker。

示意命令：

```bash
python -m sglang_router.launch_router \
  --pd-disaggregation \
  --prefill http://prefill1:30001 9001 \
  --decode http://decode1:30011 \
  --prefill-policy cache_aware \
  --decode-policy power_of_two
```

这里 `--prefill` 后面的额外端口通常用于 prefill side 的 bootstrap 或 transfer 协调。

## 37.5 Router 为什么不能只随机选 worker

PD 分离里，如果 router 随机选择 prefill 和 decode worker，系统很容易变差。

原因是它要同时考虑：

1. Prefill worker 当前排队 token 数。
2. Decode worker 当前 running sequences。
3. Decode worker 剩余 KV cache capacity。
4. Prefill 到 decode 的网络距离。
5. KV transfer backend 是否可用。
6. 模型版本是否一致。
7. TP/DP 配置是否兼容。
8. Prefix cache 或 KV cache locality。
9. Worker 是否处于健康状态。

一个错误选择可能导致：

```text
prefill 很快完成
  -> 但 decode worker 没有 slot
  -> KV 等待或 transfer 超时
  -> TTFT / p99 变差
```

或者：

```text
选中的 decode worker 很空
  -> 但它和 prefill worker 跨慢链路
  -> KV 传输时间很长
  -> 端到端收益被抵消
```

所以 PD router 本质上是一个阶段感知、负载感知、数据位置感知的调度器。

## 37.6 核心组件二：Prefill Worker

Prefill worker 负责处理输入 prompt。

它的主要职责是：

1. 接收 prompt tokens。
2. 执行 prefill forward。
3. 生成每层 KV cache。
4. 可能生成 first token logits。
5. 暴露 KV cache metadata。
6. 和 transfer backend 协作发送 KV。
7. 在请求失败或超时时清理临时状态。

Prefill worker 更关注：

1. input tokens/s。
2. 大 batch token budget。
3. 长 prompt 处理能力。
4. prefix cache 命中。
5. KV 写入效率。
6. transfer 准备时间。

它通常不负责长时间 decode。

也就是说，prefill worker 的请求生命周期应该尽量短：完成 prompt 计算、交付 KV、释放临时资源。

## 37.7 Prefill Worker 的内部流程

Prefill worker 收到请求后，大致流程是：

```text
receive prefill request
  -> validate model / tokenizer / request id
  -> allocate KV cache blocks
  -> run prefill forward
  -> produce KV cache
  -> prepare transfer metadata
  -> wait for decode destination info
  -> transfer or expose KV
  -> return prefill result
```

其中最重要的是 metadata。

Prefill 不能只告诉 decode：“我算完了”。

它还要告诉 decode：

1. KV cache 对应哪个 request。
2. prompt 长度是多少。
3. 每层 KV 的地址或 block 映射是什么。
4. dtype 和 layout 是什么。
5. tensor parallel 切分方式是什么。
6. decode worker 应该把 KV 放到哪些目标 slot。
7. transfer 是否完成。

这些 metadata 一旦错了，轻则输出错误，重则进程崩溃或读写非法显存。

## 37.8 核心组件三：Decode Worker

Decode worker 负责持续生成输出 token。

它的主要职责是：

1. 为请求分配 decode slot。
2. 接收或拉取 prefill 阶段生成的 KV cache。
3. 把 KV cache 放入本地 KV cache manager。
4. 从 first token 或 next token 开始 decode。
5. 参与 continuous batching。
6. 执行 sampling。
7. 持续 streaming token。
8. 请求完成后释放 KV cache。

Decode worker 更关注：

1. output tokens/s。
2. TPOT。
3. inter-token latency。
4. running batch 稳定性。
5. KV cache 读取效率。
6. decode slot 和显存容量。

Decode worker 的核心目标是让输出阶段稳定。

它应该尽量避免被大块 prefill 打断。

## 37.9 Decode Worker 的内部流程

Decode worker 的流程可以理解成：

```text
receive decode assignment
  -> reserve decode slots / KV blocks
  -> provide destination metadata to prefill side
  -> receive KV cache
  -> mark request ready for decode
  -> join decode scheduler
  -> generate tokens iteratively
  -> stream output
  -> release resources
```

注意一个关键点：decode worker 不能等 KV 到了才思考放哪里。

它通常要先预留目标 KV block 或 slot，然后把目标信息告诉 prefill side 或 transfer backend。

否则 prefill side 不知道 KV 应该写到 decode side 的哪个位置。

这也是为什么很多 PD 系统会有 bootstrap 机制。

## 37.10 核心组件四：KV Transfer Backend

KV transfer backend 是 PD 分离最核心、也最难的组件。

没有高效 KV transfer，PD 分离就只是把问题从调度干扰变成网络瓶颈。

KV transfer backend 要解决：

1. KV 从哪个 GPU 地址读。
2. KV 写到哪个目标 GPU 地址。
3. 跨节点时用什么网络协议。
4. 是否使用 RDMA。
5. 是否需要 CPU staging。
6. 是否需要 GPU staging buffer。
7. 如何处理不同 TP layout。
8. 如何报告完成或失败。
9. 如何处理 timeout 和资源清理。

SGLang 文档中提到的 Mooncake、NIXL、Ascend backend，都是这一层的实现选择。

可以把 transfer backend 理解成：

```text
prefill GPU memory
  -> transfer engine
  -> decode GPU memory
```

它不是普通 HTTP body 传输。

它处理的是高吞吐、低延迟、大体积、layout 敏感的张量数据。

## 37.11 KV Transfer 的几种形态

常见形态包括：

1. 同机多 GPU：通过 NVLink、PCIe 或 CUDA IPC 传输。
2. 跨节点 GPU：通过 RDMA、InfiniBand、RoCE、UCX 等能力传输。
3. CPU staging：先从 GPU 拷到 CPU，再从 CPU 发到目标侧。
4. GPU staging buffer：先在 GPU 上整理成连续 buffer，再批量传输。
5. 远端 KV cache：把 KV 写入独立缓存层，再由 decode 拉取。
6. 共享内存或统一内存抽象：让 P/D 通过统一地址或对象访问。

不同形态的权衡不同：

| 方式 | 优点 | 代价 |
| --- | --- | --- |
| 同机 NVLink | 延迟低、带宽高 | 受单机 GPU 拓扑限制 |
| RDMA 跨节点 | 可扩展到多机 | 部署和调试复杂 |
| CPU staging | 实现相对简单 | 多一次拷贝，延迟高 |
| GPU staging buffer | 对异构 TP 更友好 | 需要额外显存和 layout 转换 |
| 远端 KV cache | 可做复用和分层 | 一致性和延迟更难控制 |

第 38 章会专门展开 KV Cache 迁移、共享和路由。

本章先建立系统架构视角。

## 37.12 Bootstrap 机制是什么

很多 PD 系统中会出现 bootstrap server 或 bootstrap port。

它解决的是：prefill side 和 decode side 如何交换 transfer 前的控制信息。

例如：

```text
decode worker:
  我已经为 request-123 分配了目标 KV blocks
  目标地址 / block ids / rank 信息如下

prefill worker:
  我已经生成 request-123 的源 KV blocks
  准备按你的目标信息传过去
```

Bootstrap 通常不传大块 KV 本体。

它传的是控制信息和 metadata。

包括：

1. request id。
2. source worker id。
3. destination worker id。
4. source KV block 信息。
5. destination KV block 信息。
6. sequence length。
7. layer/rank mapping。
8. transfer session id。
9. timeout 和状态。

如果 bootstrap 失败，KV transfer 就无法安全开始。

## 37.13 请求生命周期：从接入到首 token

一个 PD 请求的前半段可以拆成：

```text
1. client 发送请求
2. router 接收请求并生成 request id
3. router 选择 prefill worker
4. router 选择或预选择 decode worker
5. decode worker 预留 KV blocks / decode slot
6. prefill worker 执行 prompt prefill
7. prefill worker 生成 KV cache
8. transfer backend 把 KV 传到 decode worker
9. decode worker 标记 request ready
10. decode worker 生成第一个或下一个 token
11. router / API server 返回首 token
```

在这个链路里，TTFT 变成：

```text
TTFT = 接入排队
     + prefill 排队
     + prefill 执行
     + decode slot 等待
     + KV transfer
     + 首轮 decode / sampling
     + streaming 返回
```

这说明 PD 分离虽然降低了 P/D 干扰，但也新增了 KV transfer 和跨组件协调成本。

所以它不是免费优化。

## 37.14 请求生命周期：持续 decode

首 token 之后，请求进入 decode 阶段。

流程是：

```text
decode worker scheduler
  -> select running requests
  -> run one decode step
  -> sample next token
  -> append KV for new token
  -> stream token
  -> check finish condition
  -> repeat
```

这部分和 unified engine 的 decode loop 类似。

区别在于：

1. Decode worker 的 running set 主要由已完成 prefill 的请求组成。
2. 它不需要频繁插入长 prompt prefill。
3. 它可以使用更适合 decode 的 batch 和调度策略。
4. 它的资源瓶颈更清晰，主要是 KV cache capacity、memory bandwidth、decode batch 和 sampling。

这就是 PD 分离改善 TPOT 和 streaming 稳定性的主要来源。

## 37.15 跨组件状态机

Unified engine 中，请求状态可以很简单：

```text
WAITING -> PREFILLING -> DECODING -> FINISHED
```

PD 分离后，状态会更细：

```text
RECEIVED
  -> PREFILL_QUEUED
  -> PREFILL_RUNNING
  -> PREFILL_DONE
  -> DECODE_ASSIGNED
  -> KV_TRANSFER_PREPARING
  -> KV_TRANSFERRING
  -> DECODE_READY
  -> DECODING
  -> FINISHED
```

还需要错误状态：

```text
PREFILL_FAILED
DECODE_ASSIGN_FAILED
KV_TRANSFER_FAILED
DECODE_FAILED
CLIENT_ABORTED
TIMEOUT
```

每个状态都要回答三个问题：

1. 当前资源在哪里。
2. 如果失败，谁负责清理。
3. 是否可以重试。

这是 PD 分离比单 engine 难很多的根本原因。

## 37.16 必须维护的 Metadata

PD 请求至少需要维护这些 metadata：

1. request id。
2. client stream id。
3. model id 和 model version。
4. tokenizer / chat template 信息。
5. prompt token ids。
6. prompt length。
7. sampling parameters。
8. prefill worker id。
9. decode worker id。
10. prefill queue time。
11. decode assignment time。
12. KV source block mapping。
13. KV destination block mapping。
14. TP/DP rank mapping。
15. transfer session id。
16. transfer start/end time。
17. output token count。
18. finish reason。
19. error code。
20. abort signal。

这些 metadata 不只是为了日志。

它们直接参与路由、transfer、重试、清理和指标归因。

## 37.17 Prefill Pool 和 Decode Pool 的扩缩容

PD 分离的一个核心收益是 P/D 可以独立扩缩容。

例如长输入短输出场景：

```text
input tokens/s 压力大
output tokens/s 压力小
```

可以增加 prefill worker：

```text
prefill replicas: 8 -> 16
decode replicas: 8
```

短输入长输出场景：

```text
input tokens/s 压力小
output tokens/s 压力大
```

可以增加 decode worker：

```text
prefill replicas: 4
decode replicas: 8 -> 16
```

但扩缩容不能只看请求数。

应该看：

1. prefill waiting tokens。
2. prefill input tokens/s。
3. decode running sequences。
4. decode output tokens/s。
5. decode KV cache utilization。
6. KV transfer bandwidth utilization。
7. TTFT 和 TPOT 的 p95/p99。

如果只扩 prefill，不扩 transfer 网络，可能只是把瓶颈推到 KV transfer。

如果只扩 decode，但 prefill queue 已经爆了，用户仍然等不到首 token。

## 37.18 路由策略

PD router 通常需要两类策略。

第一类是 prefill routing：

1. round robin。
2. least queued tokens。
3. cache aware。
4. locality aware。
5. tenant aware。

第二类是 decode routing：

1. least running sequences。
2. least KV cache usage。
3. power of two choices。
4. locality aware。
5. SLO aware。

SGLang Model Gateway 中也区分了 `--prefill-policy` 和 `--decode-policy`。

这是合理的，因为 prefill 和 decode 的负载指标不同。

Prefill 看的是输入 token 排队和 prompt 计算。

Decode 看的是 running set、KV cache 和 step latency。

## 37.19 Cache-Aware Routing 在 PD 中的含义

Cache-aware routing 在普通多 replica serving 中，通常是为了把相似前缀路由到有 prefix cache 的 worker。

在 PD 分离中，它更复杂。

Router 可能要考虑：

1. Prefill worker 是否有可复用 prefix cache。
2. Decode worker 是否接近相关 KV locality。
3. P/D 是否在同机或近网络拓扑。
4. 是否值得为了 cache 命中选择一个更忙的 worker。
5. Cache 命中节省的 prefill 时间是否大于排队和 transfer 成本。

一个简单判断公式是：

```text
如果 cache 命中节省时间 > 额外排队时间 + 额外传输时间
  则值得 cache-aware routing
否则选择更空或更近的 worker
```

真实系统不会这么简单，但这个公式能帮你理解权衡。

## 37.20 健康检查和故障隔离

PD 架构中，健康检查要分层。

Router 至少要知道：

1. Prefill worker 是否可接新请求。
2. Decode worker 是否可接新请求。
3. Transfer backend 是否可用。
4. Bootstrap 通道是否可用。
5. Worker 的模型版本是否匹配。
6. Worker 的 load 是否异常。
7. KV cache 使用率是否过高。

健康状态也不只是 alive/dead。

更实用的状态包括：

```text
READY
DRAINING
OVERLOADED
TRANSFER_DEGRADED
UNHEALTHY
```

例如某个 decode worker 还活着，但 KV cache 已经接近满，router 就不应该继续给它分配新 decode 请求。

## 37.21 故障处理：Prefill 失败

Prefill 失败可能发生在：

1. prefill worker OOM。
2. prompt 太长。
3. model forward 出错。
4. prefix cache 或 KV allocation 失败。
5. worker 重启。

处理方式通常是：

1. 如果还没有输出 token，可以换一个 prefill worker 重试。
2. 如果是请求本身非法，直接返回错误。
3. 如果 decode worker 已经预留 slot，要通知它释放资源。
4. 记录 prefill failure metrics。

Prefill 失败相对容易处理，因为通常还没有向用户输出 token。

## 37.22 故障处理：KV Transfer 失败

KV transfer 失败更麻烦。

可能原因：

1. 网络超时。
2. RDMA 连接失败。
3. decode worker 目标 slot 失效。
4. prefill worker 清理过早。
5. metadata 不一致。
6. transfer backend 内部错误。

处理方式包括：

1. 中止当前请求。
2. 清理 prefill side 临时 KV。
3. 清理 decode side 预留 slot。
4. 必要时重新 prefill 并选择新 decode worker。
5. 对异常 worker 降权或熔断。

如果首 token 还没返回，重试空间较大。

如果已经开始 streaming，再切换 decode worker 就很困难，因为新 worker 需要完整上下文和已生成 token 的一致状态。

## 37.23 故障处理：Decode 失败

Decode 失败可能发生在生成中途。

这时最难处理，因为客户端可能已经收到了部分 token。

选择通常有：

1. 直接结束 stream 并返回错误。
2. 尝试从已生成 token 重新构造 prompt，重新 prefill，再继续生成。
3. 对非流式请求隐藏失败并重试。

方案 2 很复杂，而且可能导致输出不一致。

生产系统通常更倾向于：

```text
streaming 已输出后，decode worker 失败 -> 返回错误并清理资源
```

除非业务明确接受重试带来的语义变化。

## 37.24 Client Abort 和资源清理

用户取消请求时，PD 系统要清理多个地方。

可能存在的资源包括：

1. Router 中的 request state。
2. Prefill worker waiting queue 条目。
3. Prefill worker 已分配 KV blocks。
4. Transfer session。
5. Decode worker 预留 slot。
6. Decode worker running request。
7. Streaming connection。

如果只清理 router，不通知 worker，就会泄漏显存或让后台继续生成。

所以 abort 必须广播到相关组件。

可以理解为：

```text
client abort
  -> router marks request aborted
  -> cancel prefill if running
  -> cancel transfer if running
  -> cancel decode if running
  -> release KV blocks
```

## 37.25 观测指标

PD 分离必须有更细的 metrics。

只看总 latency 不够。

建议至少拆分：

1. end-to-end TTFT。
2. prefill queue time。
3. prefill execution time。
4. decode assignment time。
5. KV transfer wait time。
6. KV transfer duration。
7. first decode step time。
8. TPOT。
9. inter-token latency p95/p99。
10. prefill input tokens/s。
11. decode output tokens/s。
12. transfer bandwidth。
13. prefill failure count。
14. decode failure count。
15. transfer timeout count。
16. KV cache utilization。

最好把一次请求的时间线打出来：

```text
request-123:
  router_queue: 5ms
  prefill_queue: 20ms
  prefill_compute: 180ms
  decode_assign: 3ms
  kv_transfer: 40ms
  first_decode: 12ms
  ttfb/ttft: 260ms
```

没有这种拆分，就很难判断 PD 分离到底是提升还是劣化。

## 37.26 部署形态一：单机多 GPU PD

最简单的 PD 形态是单机多 GPU。

例如：

```text
GPU 0-3: prefill pool
GPU 4-7: decode pool
```

优点：

1. 网络拓扑简单。
2. KV transfer 延迟较低。
3. 调试比跨节点容易。
4. 适合验证收益。

缺点：

1. 扩展能力有限。
2. P/D 比例受单机 GPU 数限制。
3. 故障隔离粒度有限。

如果团队第一次做 PD 分离，通常应该先从单机或小规模集群验证。

## 37.27 部署形态二：跨节点 PD

跨节点 PD 更接近大规模生产场景。

示意：

```text
prefill nodes:
  node p0: GPU 0-7
  node p1: GPU 0-7

decode nodes:
  node d0: GPU 0-7
  node d1: GPU 0-7
  node d2: GPU 0-7
```

优点：

1. P/D 可以大规模独立扩容。
2. 可以按业务流量调整资源比例。
3. 可以做多租户隔离。
4. 可以按硬件类型组织资源池。

缺点：

1. KV transfer 依赖网络能力。
2. RDMA / IB / RoCE 调试复杂。
3. 拓扑感知路由更重要。
4. 故障面更大。
5. 观测和资源清理更难。

跨节点 PD 的核心问题是：

```text
网络是否足够快、足够稳定，能让 KV transfer 成本小于 P/D 隔离收益。
```

## 37.28 部署形态三：异构 P/D 资源池

更高级的部署会让 prefill 和 decode 使用不同硬件或不同并行配置。

例如：

```text
Prefill pool:
  更适合大 batch compute
  更大的 input token budget
  可能使用更高 TP

Decode pool:
  更关注稳定 latency
  更大 KV cache 容量
  可能使用 DP attention
```

异构资源池的价值是把硬件用在更适合的阶段。

但它会带来 KV layout 问题。

例如 prefill TP=4，decode attention TP=1，则 KV head 的切分方式不同。

这时需要 layout 转换或 staging buffer。

SGLang 文档中提到的 heterogeneous TP with GPU staging buffer，就是为了解决这类问题。

## 37.29 和普通 Load Balancer 的区别

面试里容易把 PD router 说成 Nginx 或普通 LB。

这是不准确的。

普通 LB 只需要回答：

```text
这个请求发给哪个后端？
```

PD router 要回答：

```text
这个请求的 prefill 给谁？
decode 给谁？
KV 从哪里到哪里？
目标 decode 是否有 slot？
transfer 是否完成？
失败后谁清理？
streaming 谁负责？
```

所以 PD router 更接近一个分布式 serving coordinator。

它既在数据面上转发请求，也在控制面上维护 worker、状态和资源。

## 37.30 和 Pipeline Parallel 的区别

PD 分离也容易和 pipeline parallel 混淆。

Pipeline parallel 是按模型层切分：

```text
GPU 0: layer 0-9
GPU 1: layer 10-19
GPU 2: layer 20-29
```

PD 分离是按请求生命周期阶段切分：

```text
prefill worker: 跑完整模型的 prefill 阶段
decode worker: 跑完整模型的 decode 阶段
```

两者不是一回事。

Prefill worker 和 decode worker 通常都需要完整模型能力。

它们只是执行不同阶段。

## 37.31 和 Expert Parallel / MoE 并行的区别

MoE expert parallel 是模型内部专家路由。

例如 DeepSeek、Mixtral 这类 MoE 模型中，不同 token 会路由到不同 expert。

PD 分离不是 expert routing。

它不决定 token 去哪个 expert。

它决定一个请求的 prefill 阶段和 decode 阶段分别在哪些 serving worker 上执行。

当然，在大模型生产系统中，PD 分离可以和 TP、DP、EP、MoE all-to-all 同时存在。

这也是高级 serving 架构复杂的地方。

## 37.32 一个最小 PD Router 伪代码

下面用伪代码描述 PD router 的主流程：

```python
def handle_request(req):
    request_id = new_request_id()
    state = RequestState(request_id, req)

    prefill = choose_prefill_worker(req)
    decode = choose_decode_worker(req)

    try:
        decode_reservation = decode.reserve_slots(request_id, req.max_tokens)

        prefill_result = prefill.run_prefill(
            request_id=request_id,
            tokens=req.prompt_tokens,
            decode_target=decode_reservation,
        )

        transfer_result = transfer_kv(
            source=prefill_result.kv_source,
            target=decode_reservation.kv_target,
        )

        decode.start_decode(
            request_id=request_id,
            kv_ready=transfer_result,
            sampling_params=req.sampling_params,
        )

        for token in decode.stream(request_id):
            yield token

    except ClientAbort:
        cancel_everywhere(state)
    except Exception as err:
        cleanup_everywhere(state)
        raise err
```

真实系统会复杂得多。

但主线就是：选 P、选 D、预留 D、跑 P、传 KV、启动 D、stream、清理。

## 37.33 架构设计中的关键取舍

设计 PD 架构时，最重要的取舍包括：

1. 是否预先选择 decode worker。
2. KV transfer 是 push 还是 pull。
3. Router 是否在数据面中转 token stream。
4. Prefill 是否生成 first token。
5. Decode worker 是否支持从 prefill result 直接接管。
6. Transfer metadata 放在 router、worker 还是独立 coordinator。
7. 失败后是否自动重试。
8. 是否允许异构 TP/DP。
9. 是否做 cache-aware routing。
10. 是否引入远端 KV cache。

没有唯一正确答案。

要看业务目标：低延迟、低成本、高吞吐、可扩展性、调试复杂度，哪个更重要。

## 37.34 Prefill 是否应该生成第一个 token

一个细节问题是：first token 到底由 prefill worker 生成，还是由 decode worker 生成？

可能方案一：prefill worker 计算 prompt 后直接采样 first token。

优点：

1. 首 token 可能更快。
2. prefill logits 已经在手边。

缺点：

1. Decode worker 接管时要同步 first token 状态。
2. KV cache 需要包含 first token 之后的状态。
3. 状态机更复杂。

可能方案二：prefill worker 只生成 prompt KV，decode worker 负责 first token 和后续 token。

优点：

1. 职责更清晰。
2. Decode 状态更一致。
3. Streaming 更容易归一。

缺点：

1. TTFT 可能多一次 decode step。

很多系统会根据实现复杂度和延迟目标选择不同方案。

面试中可以说明这个 trade-off，而不是武断说只有一种实现。

## 37.35 Push KV 还是 Pull KV

KV transfer 有两种思路。

Push：

```text
prefill worker 主动把 KV 推到 decode worker
```

Pull：

```text
decode worker 从 prefill worker 或远端缓存拉取 KV
```

Push 的优点：

1. Prefill 完成后可以立即发送。
2. 流程直观。

Push 的缺点：

1. Decode 目标必须提前确定。
2. 如果 decode worker 忙或失败，处理复杂。

Pull 的优点：

1. Decode worker 控制接收节奏。
2. 更适合远端 KV cache 或多级 cache。

Pull 的缺点：

1. 可能增加等待。
2. 需要更复杂的 source discovery。

实际系统可能是混合模式：metadata 由 router 协调，数据传输由 transfer backend 执行。

## 37.36 PD 分离的反压机制

PD 系统必须有 backpressure。

否则 prefill 太快，decode 接不住，会导致大量 KV 暂存和内存压力。

常见反压信号包括：

1. Decode worker 可用 slot 数。
2. Decode KV cache 剩余容量。
3. Transfer queue 长度。
4. Prefill queue 长度。
5. Router pending request 数。
6. Per-tenant token budget。

一种简单策略是：

```text
如果没有 decode capacity
  不启动新的 prefill
```

这样可以避免 prefill 算完后无处可去。

但代价是 prefill 资源可能空转。

另一种策略是允许少量 ahead-of-time prefill，但限制 pending transfer 数量。

这就是吞吐和稳定性的权衡。

## 37.37 多租户和 QoS

生产环境中，PD 分离通常服务多个业务或租户。

QoS 要同时作用在 P 和 D 两侧：

1. 某租户最多占用多少 prefill token budget。
2. 某租户最多占用多少 decode running slots。
3. 长 prompt 是否需要降优先级。
4. 长输出是否需要限制 max tokens。
5. 高优先级请求是否可以抢占队列。
6. 不同租户是否共享 prefix cache。
7. 不同租户是否允许共享 KV cache 基础设施。

如果只在入口限流，不在 P/D 阶段分别限流，仍可能出现一个租户把 decode pool 占满，导致其他租户 TTFT 正常但 TPOT 爆炸。

## 37.38 安全和版本一致性

PD 架构必须保证 prefill 和 decode 使用兼容的模型。

至少要校验：

1. model id。
2. model weights version。
3. tokenizer version。
4. rope scaling 配置。
5. max context length。
6. KV dtype。
7. attention layout。
8. TP/DP 配置。
9. quantization 配置。

如果 prefill 和 decode 模型版本不一致，KV cache 即使能传过去，也不代表语义正确。

最危险的是 silent wrong output。

所以生产系统中，router 注册 worker 时必须记录能力和版本信息。

## 37.39 什么时候 Router 应该拒绝请求

PD router 不应该无限接收请求。

应该拒绝或排队受控的情况包括：

1. 没有健康 prefill worker。
2. 没有健康 decode worker。
3. Transfer backend 不可用。
4. Decode KV cache 全局接近满。
5. 请求 prompt 超过限制。
6. 请求 max_tokens 超过租户配额。
7. 当前 pending transfer 过多。
8. 模型版本没有匹配的 P/D worker。

拒绝请求比把请求塞进系统然后超时更好。

因为后者会浪费 prefill 计算和 KV 资源。

## 37.40 面试官会怎么问

问题一：请画一下 PD 分离的系统架构。

回答要点：client 到 router，router 分别管理 prefill pool 和 decode pool；prefill worker 生成 prompt KV；KV transfer backend 把 KV 交给 decode worker；decode worker 做持续生成和 streaming；router 负责路由、状态、健康检查和失败清理。

问题二：PD router 和普通 load balancer 有什么区别？

回答要点：普通 LB 只选一个后端；PD router 要分别选择 prefill/decode worker，协调 KV transfer，维护 request state，感知 P/D 负载、KV locality、decode slot、模型版本和故障清理。

问题三：为什么 KV transfer backend 是核心？

回答要点：因为 prefill 产生的 KV cache 必须被 decode 使用，KV 体积大、在 GPU memory 中、layout 和 TP/DP 切分敏感。传输成本决定 PD 分离收益是否能覆盖系统复杂度。

问题四：PD 分离中最难的故障是什么？

回答要点：KV transfer 失败和 decode 中途失败最麻烦。前者涉及两侧资源清理和可能重新 prefill；后者可能已经向客户端输出 token，很难无感迁移。

问题五：如何判断 PD 架构是否设计合理？

回答要点：要看分阶段指标，包括 prefill queue、prefill compute、decode assignment、KV transfer、first decode、TPOT、transfer timeout、KV cache utilization，而不是只看总 QPS。

## 37.41 标准回答模板

如果面试官问“PD 分离的系统架构是什么样”，可以这样回答：

```text
PD 分离架构通常由 router、prefill worker pool、decode worker pool 和 KV transfer backend 组成。Router 接收请求后，不是简单选一个后端，而是分别选择 prefill worker 和 decode worker，并维护请求在两个阶段之间的状态。

Prefill worker 负责处理 prompt，执行 prefill forward，生成每层 KV cache，并准备 KV metadata。Decode worker 负责预留 decode slot 和目标 KV blocks，接收或拉取 prefill 生成的 KV cache，然后进入 continuous batching 的 decode loop，持续生成 token 并 streaming 返回。

中间最关键的是 KV transfer backend。因为 prefill 和 decode 不在同一个 worker 上，prompt KV 必须从 prefill side 迁移或共享给 decode side。这个过程涉及 GPU memory、RDMA/NVLink/NIXL/Mooncake 等传输能力、KV layout、TP/DP rank mapping、timeout 和资源清理。

所以 PD 分离本质上是把单 engine 内部的 WAITING -> PREFILLING -> DECODING 状态机，升级成跨 router、prefill worker、transfer backend、decode worker 的分布式状态机。它能隔离 P/D 干扰并支持独立扩缩容，但代价是路由、KV 迁移、状态同步、故障恢复和观测复杂度显著增加。
```

## 37.42 架构状态机、清理门禁和可运行 demo

PD 架构图容易画，难的是证明它的状态机和资源清理真的闭环。一个请求跨 router、prefill worker、decode worker 和 transfer backend 后，任何一步失败都可能留下 KV blocks、decode slot、transfer session 或 stream 状态。

可以把请求抽象成：

```math
R_i=(r_i,P_i,O_i,m_i,s_i)
```

其中 `r_i` 是 request id，`P_i` 是 prompt token 数，`O_i` 是最大输出 token 数，`m_i` 是模型版本，`s_i` 是客户端 stream 状态。

Router 的 worker 选择至少要满足：

```math
G_{\mathrm{route},i}=H_{\mathrm{prefill}}H_{\mathrm{decode}}M_{\mathrm{prefill}}M_{\mathrm{decode}}C_{\mathrm{decode}}T_{\mathrm{backend}}
```

含义是：prefill / decode worker 都健康，模型版本匹配，decode worker 有足够 KV / slot 容量，transfer backend 可用。任何一个条件不满足，router 都不应该盲目启动 prefill。

KV transfer metadata 可以抽象成：

```math
K_i=(r_i,w_i^{\mathrm{src}},w_i^{\mathrm{dst}},B_i^{\mathrm{src}},B_i^{\mathrm{dst}},L_i,d_i)
```

其中 source / destination worker、source / destination block、序列长度、dtype / layout 都必须一致。metadata 缺失比 transfer 慢更危险，因为它可能导致 silent wrong output。

一个 PD 请求的 TTFT 可以拆成：

```math
T_{\mathrm{ttft}}=T_{\mathrm{router}}+T_{\mathrm{pq}}+T_{\mathrm{prefill}}+T_{\mathrm{reserve}}+T_{\mathrm{transfer}}+T_{\mathrm{decode1}}+T_{\mathrm{stream}}
```

最终架构门禁可以写成：

```math
G_{\mathrm{pdarch}}=G_{\mathrm{route}}G_{\mathrm{reserve}}G_{\mathrm{meta}}G_{\mathrm{transfer}}G_{\mathrm{abort}}G_{\mathrm{version}}G_{\mathrm{metric}}
```

下面的 demo 用纯 Python 模拟三条路径：

1. 正常请求：router 选择 P/D worker，decode 先预留，prefill 生成 KV metadata，transfer 成功，decode streaming 完成。
2. transfer 失败请求：transfer 失败后清理 prefill side 和 decode side。
3. client abort 请求：decode 已预留但 prefill 尚未开始时取消，必须释放预留 slot。

```python
from dataclasses import dataclass


@dataclass
class Worker:
    name: str
    role: str
    model_version: str
    zone: str
    healthy: bool = True
    queued_tokens: int = 0
    running_sequences: int = 0
    kv_free_tokens: int = 0


@dataclass
class PDRequest:
    request_id: str
    prompt_tokens: int
    output_tokens: int
    model_version: str
    fail_transfer: bool = False
    abort_after_reserve: bool = False


class ToyPDArchitectureAuditor:
    def __init__(self, prefill_workers, decode_workers, kv_mib_per_token=0.015625):
        self.prefill_workers = prefill_workers
        self.decode_workers = decode_workers
        self.kv_mib_per_token = kv_mib_per_token
        self.cleanup_events = []
        self.transfer_sessions = []

    def _choose_prefill(self, request):
        candidates = [
            worker for worker in self.prefill_workers
            if worker.healthy and worker.model_version == request.model_version
        ]
        if not candidates:
            return None
        return min(candidates, key=lambda worker: worker.queued_tokens)

    def _choose_decode(self, request):
        need_tokens = request.prompt_tokens + request.output_tokens
        candidates = [
            worker for worker in self.decode_workers
            if worker.healthy
            and worker.model_version == request.model_version
            and worker.kv_free_tokens >= need_tokens
        ]
        if not candidates:
            return None
        return min(candidates, key=lambda worker: (worker.running_sequences, -worker.kv_free_tokens))

    def _transfer_rate(self, prefill, decode):
        return 1200.0 if prefill.zone == decode.zone else 300.0

    def _cleanup(self, request, prefill=None, decode=None, reason="unknown"):
        if prefill:
            prefill.queued_tokens = max(0, prefill.queued_tokens - request.prompt_tokens)
        if decode:
            decode.running_sequences = max(0, decode.running_sequences - 1)
            decode.kv_free_tokens += request.prompt_tokens + request.output_tokens
        self.cleanup_events.append({"request_id": request.request_id, "reason": reason})

    def process(self, requests):
        rows = []
        max_decode_running = 0
        stream_events = 0
        total_transfer_mib = 0.0
        for request in requests:
            states = ["RECEIVED"]
            prefill = self._choose_prefill(request)
            decode = self._choose_decode(request)
            if prefill is None or decode is None:
                rows.append({
                    "request_id": request.request_id,
                    "status": "ROUTE_FAILED",
                    "states": states,
                })
                continue

            states.extend(["PREFILL_ASSIGNED", "DECODE_RESERVED"])
            decode.running_sequences += 1
            decode.kv_free_tokens -= request.prompt_tokens + request.output_tokens
            max_decode_running = max(max_decode_running, decode.running_sequences)

            if request.abort_after_reserve:
                states.append("CLIENT_ABORTED")
                self._cleanup(request, decode=decode, reason="client_abort_after_reserve")
                rows.append({
                    "request_id": request.request_id,
                    "status": "CLIENT_ABORTED",
                    "prefill": prefill.name,
                    "decode": decode.name,
                    "states": states,
                    "cleanup": True,
                })
                continue

            prefill.queued_tokens += request.prompt_tokens
            states.extend(["PREFILL_RUNNING", "PREFILL_DONE", "KV_TRANSFERRING"])
            kv_mib = request.prompt_tokens * self.kv_mib_per_token
            transfer_ms = kv_mib / self._transfer_rate(prefill, decode)
            session = {
                "request_id": request.request_id,
                "source": prefill.name,
                "target": decode.name,
                "source_blocks": (request.prompt_tokens + 127) // 128,
                "target_reserved_tokens": request.prompt_tokens + request.output_tokens,
                "kv_mib": round(kv_mib, 1),
                "transfer_ms": round(transfer_ms, 3),
            }
            total_transfer_mib += kv_mib
            self.transfer_sessions.append(session)

            if request.fail_transfer:
                states.append("KV_TRANSFER_FAILED")
                self._cleanup(request, prefill=prefill, decode=decode, reason="kv_transfer_failed")
                rows.append({
                    "request_id": request.request_id,
                    "status": "KV_TRANSFER_FAILED",
                    "prefill": prefill.name,
                    "decode": decode.name,
                    "states": states,
                    "kv_metadata": session,
                    "cleanup": True,
                })
                continue

            states.extend(["DECODE_READY", "DECODING", "FINISHED"])
            stream_count = min(4, max(1, request.output_tokens // 100))
            stream_events += stream_count
            self._cleanup(request, prefill=prefill, decode=decode, reason="finished")
            rows.append({
                "request_id": request.request_id,
                "status": "FINISHED",
                "prefill": prefill.name,
                "decode": decode.name,
                "states": states,
                "kv_metadata": session,
                "stream_events": stream_count,
                "cleanup": True,
            })

        summary = {
            "requests": len(rows),
            "finished": sum(row["status"] == "FINISHED" for row in rows),
            "client_aborted": sum(row["status"] == "CLIENT_ABORTED" for row in rows),
            "transfer_failed": sum(row["status"] == "KV_TRANSFER_FAILED" for row in rows),
            "transfer_sessions": len(self.transfer_sessions),
            "total_transfer_mib": round(total_transfer_mib, 1),
            "cleanup_events": len(self.cleanup_events),
            "stream_events": stream_events,
            "max_decode_running": max_decode_running,
        }
        gates = {
            "router_selects_phase_workers": all(
                row.get("prefill") and row.get("decode")
                for row in rows if row["status"] != "ROUTE_FAILED"
            ),
            "decode_reserved_before_transfer": all(
                "DECODE_RESERVED" in row["states"]
                and row["states"].index("DECODE_RESERVED") < row["states"].index("KV_TRANSFERRING")
                for row in rows if "KV_TRANSFERRING" in row["states"]
            ),
            "kv_metadata_complete": all(
                {
                    "request_id",
                    "source",
                    "target",
                    "source_blocks",
                    "target_reserved_tokens",
                    "kv_mib",
                }.issubset(row["kv_metadata"])
                for row in rows if "kv_metadata" in row
            ),
            "transfer_failure_cleanup": any(
                row["status"] == "KV_TRANSFER_FAILED" and row.get("cleanup")
                for row in rows
            ),
            "client_abort_cleanup": any(
                row["status"] == "CLIENT_ABORTED" and row.get("cleanup")
                for row in rows
            ),
            "model_version_checked": all(
                row.get("status") == "ROUTE_FAILED" or row.get("prefill") != "prefill_v2"
                for row in rows
            ),
            "metrics_ready": summary["transfer_sessions"] == 2
            and summary["cleanup_events"] == 3,
        }
        gates["pd_architecture_gate"] = all(gates.values())
        return rows, summary, gates


prefill_workers = [
    Worker("prefill_a", "prefill", "v1", "rack_a", queued_tokens=5000),
    Worker("prefill_b", "prefill", "v1", "rack_b", queued_tokens=1000),
    Worker("prefill_v2", "prefill", "v2", "rack_a", queued_tokens=0),
]
decode_workers = [
    Worker("decode_a", "decode", "v1", "rack_a", running_sequences=22, kv_free_tokens=40000),
    Worker("decode_b", "decode", "v1", "rack_b", running_sequences=6, kv_free_tokens=18000),
    Worker("decode_bad", "decode", "v1", "rack_b", healthy=False, kv_free_tokens=90000),
]
requests = [
    PDRequest("req_ok", prompt_tokens=8000, output_tokens=300, model_version="v1"),
    PDRequest("req_transfer_fail", prompt_tokens=16000, output_tokens=120, model_version="v1", fail_transfer=True),
    PDRequest("req_abort", prompt_tokens=1200, output_tokens=400, model_version="v1", abort_after_reserve=True),

rows, summary, gates = ToyPDArchitectureAuditor(prefill_workers, decode_workers).process(requests)
```

一次运行的核心输出类似：

```text
pd_architecture_summary= {'requests': 3, 'finished': 1, 'client_aborted': 1, 'transfer_failed': 1, 'transfer_sessions': 2, 'total_transfer_mib': 375.0, 'cleanup_events': 3, 'stream_events': 3, 'max_decode_running': 7}

```

这个 demo 的关键不是数值大小，而是架构证据：

1. `decode_reserved_before_transfer=True` 说明 decode worker 先预留目标 KV / slot，再让 P 侧开始安全 transfer。
2. `kv_metadata_complete=True` 说明 transfer 不只是“传一坨 tensor”，而是带 request id、source、target、source blocks、target reserved tokens 和 transfer metrics。
3. `transfer_failure_cleanup=True` 说明 transfer 失败会清理 P/D 两侧资源，不让预留 slot 或临时 KV 泄漏。
4. `client_abort_cleanup=True` 说明 client 取消不能只清理 router，还要释放 decode reserve。
5. `model_version_checked=True` 说明 router 没有把 v1 请求误发给 `prefill_v2` 这类不兼容 worker。

所以本章最终门禁是 `pd_architecture_gate`：只有 router、reserve、metadata、transfer failure、client abort、version check 和 metrics 都能闭环，PD 分离才不是一张架构图，而是可运行、可清理、可观测的分布式状态机。

## 37.44 本章总结

PD 分离的系统架构不是把一个 engine 拆成两个函数，而是把请求生命周期拆成跨组件协作。

它的核心组件是 router、prefill worker、decode worker 和 KV transfer backend。

Router 负责阶段感知路由和状态协调；prefill worker 负责 prompt 计算和 KV 生成；decode worker 负责持续生成和 streaming；KV transfer backend 负责把最重、最敏感的 KV cache 从 P 侧交给 D 侧。

PD 分离的主要收益来自 P/D 资源隔离、独立扩缩容和更稳定的 decode latency。

它的主要复杂度来自 KV transfer、metadata、跨组件状态机、故障恢复和观测。
