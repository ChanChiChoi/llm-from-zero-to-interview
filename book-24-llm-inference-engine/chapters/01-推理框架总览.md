# 第 1 章 LLM Serving Engine 到底是什么

第二十三册我们从 AI Infra 角度讲了训练平台、推理平台、调度、可观测性和治理。本册继续下钻到推理框架内部，专门讲 LLM Serving Engine。

很多人会用 `model.generate()`，也会启动 vLLM 或 SGLang 服务，但不一定真正理解 serving engine 内部发生了什么。面试和真实工程中，真正难的问题往往不是“怎么启动服务”，而是“为什么吞吐上不去”“为什么 TTFT 抖动”“为什么 KV cache 爆了”“scheduler 应该怎么设计”。

先记住一句话：

> LLM Serving Engine 是把大模型从离线 generate 函数变成高并发、低延迟、可流式、可调度、可观测在线服务的执行引擎。

## 1.0 本讲资料边界与第二轮精修口径

本章是第二十四册的总览，不把某个框架版本的内部类名写成唯一标准答案。第二轮精修时，本章按稳定抽象来写：

1. vLLM 参考其公开架构文档中 API server、engine core、GPU worker、scheduler、KV cache manager、model execution 和 online serving 的边界；PagedAttention 只作为 KV cache block 管理思想，不把它写成新的 attention 数学公式。
2. SGLang 参考其论文和项目公开描述中的结构化生成、RadixAttention、prefix / KV 复用和复杂 generation program 执行边界；本章只讲 serving engine 总体能力，不提前展开 SGLang runtime 细节。
3. Hugging Face TGI 参考其文档中 streaming、continuous batching、Prometheus metrics、OpenTelemetry tracing 和 tensor parallelism 的公开口径；本章不把 TGI 作为唯一推荐 runtime。
4. NVIDIA Triton 参考其 model repository、per-model scheduler、batcher、backend、metrics、tracing 和 model management API 的通用 serving 抽象；本章只用它帮助区分“通用模型服务框架”和“LLM 专用 serving engine”。
5. 所有指标统一回到请求生命周期、prefill / decode、KV cache、scheduler、streaming、SLO、成本和可观测性，不用单一 QPS 或单一 GPU utilization 判断推理框架好坏。

所以，本章的重点不是背 vLLM、SGLang、TGI、Triton 的名字，而是能说清一个 LLM serving engine 到底要管理哪些状态、暴露哪些指标、用哪些门禁证明它可上线。

## 1.1 从 model.generate 说起

最简单的大模型推理代码可能是：

```python
inputs = tokenizer(prompt, return_tensors="pt").to(device)
outputs = model.generate(**inputs, max_new_tokens=128)
text = tokenizer.decode(outputs[0])
```

这段代码适合：

1. 本地实验。
2. Notebook 演示。
3. 单用户测试。
4. 简单 benchmark。

但它不适合直接支撑生产服务。

原因是生产服务要处理很多额外问题：

1. 多用户并发请求。
2. 请求排队。
3. 动态 batching。
4. KV cache 管理。
5. streaming 输出。
6. 超时和取消。
7. 指标统计。
8. 资源隔离。
9. 故障处理。
10. 多模型和多版本。

Serving engine 就是为了解决这些问题。

## 1.2 LLM Serving Engine 的定义

LLM Serving Engine 可以理解为一个运行时系统。

它接收用户请求，组织请求进入 batch，调用模型执行 prefill 和 decode，管理 KV cache，持续返回 token，并记录指标和状态。

它通常包含：

1. API Server。
2. Request Queue。
3. Scheduler。
4. Tokenizer。
5. Model Executor。
6. KV Cache Manager。
7. Sampler。
8. Streaming Manager。
9. Metrics Exporter。
10. Worker / Executor Runtime。

从外部看，它是一个模型服务。

从内部看，它是一个高性能并发执行系统。

## 1.3 Serving Engine 和推理平台的区别

推理平台是更上层的平台系统。

它负责：

1. API 网关。
2. 鉴权。
3. 模型路由。
4. 限流。
5. 灰度发布。
6. 成本归因。
7. 多租户治理。
8. 自动扩缩容。

Serving engine 负责更底层的执行：

1. 请求调度。
2. prefill。
3. decode。
4. KV cache。
5. batching。
6. token streaming。
7. GPU 执行。

简单说：推理平台管“服务治理”，serving engine 管“模型执行”。

## 1.4 一次请求在 Engine 内部的生命周期

一个请求进入 engine 后，大致流程是：

```text
HTTP Request
  -> Parse Request
  -> Tokenize Prompt
  -> Enter Waiting Queue
  -> Scheduler Selects Request
  -> Prefill
  -> Allocate KV Cache
  -> Decode Loop
  -> Sample Next Token
  -> Stream Token
  -> Finish / Cancel / Timeout
  -> Free KV Cache
```

这条链路就是本书后续所有章节的主线。

理解它，就能理解 vLLM、SGLang、PD 分离和各种性能优化。

## 1.5 Prefill 和 Decode 是核心阶段

LLM 推理不是一次性算完，而是分为 prefill 和 decode。

Prefill 阶段：

1. 输入完整 prompt。
2. 处理所有输入 token。
3. 生成第一步所需状态。
4. 构建 KV cache。
5. 主要影响 TTFT。

Decode 阶段：

1. 每次生成一个或少量 token。
2. 重复读取 KV cache。
3. 持续执行 sampling。
4. 逐步 streaming 输出。
5. 主要影响 TPOT。

Serving engine 的调度、缓存和性能优化，基本都围绕这两个阶段展开。

## 1.6 KV Cache 为什么是核心

KV cache 保存历史 token 在 attention 中的 Key 和 Value。

它的作用是避免每次 decode 都重新计算全部历史上下文。

没有 KV cache，生成第 1000 个 token 时还要重新处理前 999 个 token，成本会非常高。

但 KV cache 也会占用大量显存。

它和这些因素相关：

1. batch size。
2. sequence length。
3. layer 数。
4. hidden size。
5. dtype。
6. 并发请求数。

Serving engine 的一个核心工作就是管理 KV cache。

vLLM 的 PagedAttention，本质上就是对 KV cache 管理方式的一次重要工程改造。

## 1.7 为什么需要 Scheduler

如果一次只服务一个请求，GPU 利用率很低。

如果把所有请求都塞进一个 batch，又会导致长请求拖慢短请求，显存爆掉，p99 变差。

Scheduler 要决定：

1. 哪些请求进入 prefill。
2. 哪些请求进入 decode。
3. 每轮 batch 放多少 token。
4. 谁先执行。
5. 谁等待。
6. 谁被取消或拒绝。
7. KV cache 不够时怎么处理。

LLM serving scheduler 不是普通 Web 请求队列。

它要理解 token、KV cache、prefill、decode、优先级和 streaming。

## 1.8 为什么需要 Continuous Batching

普通 batching 是固定一批请求一起跑到结束。

但大模型生成中，每个请求输出长度不同。

有的请求 20 个 token 结束，有的请求 2000 个 token 才结束。

如果固定 batch，短请求会被长请求拖住。

Continuous batching 的思想是：每个 decode step 都动态更新 batch。

完成的请求退出，新请求加入。

这样可以提高 GPU 利用率，也更适合 streaming 服务。

## 1.9 Streaming 为什么重要

大模型生成可能需要几秒甚至几十秒。

如果等全部输出生成完再返回，用户体验很差。

Streaming 让模型生成一个 token 或一小段 token 就返回给用户。

Serving engine 要处理：

1. token 增量返回。
2. 客户端断开。
3. 请求取消。
4. backpressure。
5. finish reason。
6. 多请求输出顺序。

Streaming 不是简单 `print(token)`，而是 runtime、网络和状态管理共同作用的能力。

## 1.10 Engine 需要暴露哪些指标

一个生产级 engine 至少要暴露：

1. request count。
2. active requests。
3. waiting requests。
4. TTFT。
5. TPOT。
6. input tokens/s。
7. output tokens/s。
8. queue latency。
9. prefill latency。
10. decode latency。
11. KV cache usage。
12. GPU memory usage。
13. batch size。
14. error rate。
15. timeout count。

没有这些指标，就很难回答“为什么变慢”。

## 1.11 vLLM、SGLang、TGI、Triton 的位置

这些系统都和 serving engine 有关，但侧重点不同。

vLLM：强调高吞吐 LLM serving，核心是 PagedAttention、KV cache block 管理和 continuous batching。

SGLang：强调结构化 LLM 程序执行、RadixAttention、prefix sharing、structured generation 和复杂 serving 场景。

TGI：Hugging Face 生态中的 text generation inference 服务。

Triton：通用推理服务框架，可结合 TensorRT-LLM 等后端服务模型。

本书会重点讲 vLLM 和 SGLang 的内部思想，也会从最小实现角度讲 serving engine 的通用结构。

## 1.12 从 0 实现推理框架要实现什么

一个最小推理框架可以按阶段实现：

1. 单请求 generate loop。
2. sampling。
3. KV cache。
4. batched prefill。
5. batched decode。
6. 请求队列。
7. scheduler。
8. token streaming。
9. HTTP API。
10. 基础 metrics。

这也是本书第二部分的主线。

先实现 naive 版本，再逐步升级到 vLLM-like 或 SGLang-like 的设计。

## 1.13 推理框架核心指标和最小 demo

推理框架总览不能只停留在模块图上。面试中更稳的方式，是把 serving engine 抽象成一组请求样本、阶段耗时、显存预算和门禁。

可以把一个请求样本记为：

```math
R_i=(a_i,p_i,q_i,d_i,k_i,s_i,o_i,m_i,z_i)
```

其中 `a` 是到达时间，`p` 是 prompt token 数，`q` 是排队状态，`d` 是 decode 输出 token 数，`k` 是 KV cache 占用，`s` 是 streaming 状态，`o` 是观测指标，`m` 是模型和 runtime 配置，`z` 是请求最终状态。

端到端延迟可以拆成：

```math
T_{\mathrm{e2e}}=T_{\mathrm{parse}}+T_{\mathrm{tokenize}}+T_{\mathrm{queue}}+T_{\mathrm{prefill}}+T_{\mathrm{first}}+T_{\mathrm{decode}}+T_{\mathrm{stream}}
```

TTFT 主要覆盖首 token 前的路径：

```math
T_{\mathrm{ttft}}=T_{\mathrm{parse}}+T_{\mathrm{tokenize}}+T_{\mathrm{queue}}+T_{\mathrm{prefill}}+T_{\mathrm{first}}
```

TPOT 衡量后续输出 token 的平均间隔：

```math
T_{\mathrm{tpot}}=\frac{T_{\mathrm{decode}}}{\max(N_{\mathrm{out}}-1,1)}
```

其中 `N_out` 是输出 token 数。注意这里把第一个 token 归入 TTFT，后续 token 才进入 TPOT。

KV cache 的粗略显存估算可以写成：

```math
M_{\mathrm{kv}}=2LBSH_{\mathrm{kv}}D_hb
```

其中 `L` 是层数，`B` 是并发序列数，`S` 是平均上下文 token 数，`H_kv` 是 KV head 数，`D_h` 是 head dimension，`b` 是每个元素字节数，前面的 `2` 来自 Key 和 Value 两份缓存。

KV pressure 用来判断显存是否先于算力成为瓶颈：

```math
P_{\mathrm{kv}}=\frac{M_{\mathrm{kv}}}{M_{\mathrm{budget}}}
```

token 吞吐和单位成本也要按 token 口径计算：

```math
X_{\mathrm{in}}=\frac{\sum_i N_{\mathrm{in},i}}{T_{\mathrm{window}}},\qquad
X_{\mathrm{out}}=\frac{\sum_i N_{\mathrm{out},i}}{T_{\mathrm{window}}}
```

```math
K_{\mathrm{1k}}=\frac{1000K_{\mathrm{total}}}{\sum_i (N_{\mathrm{in},i}+N_{\mathrm{out},i})}
```

最终可以用一个总门禁收束：

```math
G_{\mathrm{engine}}=\mathbf{1}\left[
G_{\mathrm{lifecycle}}\land
G_{\mathrm{phase}}\land
G_{\mathrm{kv}}\land
G_{\mathrm{scheduler}}\land
G_{\mathrm{stream}}\land
G_{\mathrm{metrics}}\land
G_{\mathrm{runtime}}\land
G_{\mathrm{cost}}\land
P_0=0
\right]
```

下面这个 0 依赖 demo 演示如何把“推理框架总览”变成一个可运行审计。它不模拟真实 transformer 计算，只审计 serving engine 是否把请求生命周期、prefill / decode、KV cache、scheduler、streaming、metrics、runtime 边界和成本容量讲清楚。

```python
from copy import deepcopy


class MiniServingEngineOverviewAudit:
    def __init__(self):
        self.gates = [
            "request_lifecycle_coverage",
            "prefill_decode_contract",
            "kv_cache_formula_fit",
            "scheduler_batching_readiness",
            "streaming_state_management",
            "metrics_observability",
            "runtime_boundary_clarity",
            "cost_capacity_model",
            "serving_engine_gate",
        ]
        self.actions = {
            "request_lifecycle_coverage": "trace parse, tokenize, queue, prefill, decode, stream, finish, and cleanup",
            "prefill_decode_contract": "separate TTFT-critical prefill from TPOT-critical decode accounting",
            "kv_cache_formula_fit": "estimate KV bytes from layers, active sequences, context, KV heads, head dim, and dtype",
            "scheduler_batching_readiness": "define token budget, continuous batching, admission, and cancellation cleanup",
            "streaming_state_management": "handle client disconnect, backpressure, timeout, finish reason, and partial output",
            "metrics_observability": "export queue, prefill, decode, TTFT, TPOT, tokens/s, KV pressure, errors, and timeouts",
            "runtime_boundary_clarity": "separate API server, scheduler, executor, KV manager, sampler, streamer, and metrics",
            "cost_capacity_model": "compute per-1k-token cost and capacity from token throughput and KV budget",
            "serving_engine_gate": "block launch when any lifecycle, phase, KV, scheduler, stream, metric, runtime, or cost gate fails",
        }

    def percentile(self, values, pct):
        values = sorted(values)
        if not values:
            return 0.0
        index = int(round((len(values) - 1) * pct))
        return values[index]

    def request_times(self, request):
        ttft = (
            request["parse_ms"]
            + request["tokenize_ms"]
            + request["queue_ms"]
            + request["prefill_ms"]
            + request["first_token_ms"]
        )
        decode_tokens = max(request["output_tokens"] - 1, 1)
        tpot = request["decode_ms"] / decode_tokens
        e2e = ttft + request["decode_ms"] + request["stream_flush_ms"]
        return {"ttft_ms": ttft, "tpot_ms": tpot, "e2e_ms": e2e}

    def kv_cache_mib(self, case):
        cfg = case["model_config"]
        active = max(request["active_sequences"] for request in case["requests"])
        avg_context = sum(
            request["prompt_tokens"] + request["output_tokens"]
            for request in case["requests"]
        ) / len(case["requests"])
        bytes_total = (
            2
            * cfg["layers"]
            * active
            * avg_context
            * cfg["kv_heads"]
            * cfg["head_dim"]
            * cfg["dtype_bytes"]
        )
        return round(bytes_total / (1024 * 1024), 2)

    def example_metrics(self, case):
        times = [self.request_times(request) for request in case["requests"]]
        total_input = sum(request["prompt_tokens"] for request in case["requests"])
        total_output = sum(request["output_tokens"] for request in case["requests"])
        window_s = case["traffic"]["window_s"]
        kv_mib = self.kv_cache_mib(case)
        total_cost = case["cost"]["gpu_cost"] + case["cost"]["cpu_cost"] + case["cost"]["runtime_overhead"]
        return {
            "ttft_p95_ms": round(self.percentile([item["ttft_ms"] for item in times], 0.95), 2),
            "tpot_p95_ms": round(self.percentile([item["tpot_ms"] for item in times], 0.95), 2),
            "e2e_p99_ms": round(self.percentile([item["e2e_ms"] for item in times], 0.99), 2),
            "input_tokens_per_s": round(total_input / window_s, 2),
            "output_tokens_per_s": round(total_output / window_s, 2),
            "kv_cache_mib": kv_mib,
            "kv_pressure": round(kv_mib / case["model_config"]["kv_budget_mib"], 3),
            "cost_per_1k_tokens": round(1000 * total_cost / (total_input + total_output), 4),
            "continuous_batching_gain": round(
                case["scheduler"]["static_batch_time_ms"]
                / case["scheduler"]["continuous_batch_time_ms"],
                2,
            ),
        }

    def gate_request_lifecycle_coverage(self, case):
        required = {
            "parse",
            "tokenize",
            "queue",
            "prefill",
            "decode",
            "stream",
            "finish",
            "cleanup",
        }
        return required.issubset(set(case["lifecycle"]))

    def gate_prefill_decode_contract(self, case):
        phase = case["phase_contract"]
        return (
            phase["prefill_metric"]
            and phase["decode_metric"]
            and phase["ttft_budget_ms"] >= self.example_metrics(case)["ttft_p95_ms"]
            and phase["tpot_budget_ms"] >= self.example_metrics(case)["tpot_p95_ms"]
        )

    def gate_kv_cache_formula_fit(self, case):
        cfg = case["model_config"]
        metrics = self.example_metrics(case)
        return (
            cfg["layers"] > 0
            and cfg["kv_heads"] > 0
            and cfg["head_dim"] > 0
            and cfg["dtype_bytes"] in {1, 2, 4}
            and metrics["kv_pressure"] <= cfg["max_kv_pressure"]
            and case["kv_manager"]["admission_control"]
            and case["kv_manager"]["release_on_finish"]
        )

    def gate_scheduler_batching_readiness(self, case):
        scheduler = case["scheduler"]
        return (
            scheduler["continuous_batching"]
            and scheduler["token_budget"]
            and scheduler["max_active_sequences"]
            and scheduler["prefill_decode_policy"]
            and scheduler["cancel_cleanup"]
            and scheduler["continuous_batch_time_ms"] < scheduler["static_batch_time_ms"]
        )

    def gate_streaming_state_management(self, case):
        stream = case["streaming"]
        return (
            stream["sse_or_chunked"]
            and stream["client_disconnect"]
            and stream["backpressure"]
            and stream["finish_reason"]
            and stream["partial_output_trace"]
            and stream["timeout_cleanup"]
        )

    def gate_metrics_observability(self, case):
        required = {
            "request_count",
            "waiting_requests",
            "active_requests",
            "queue_ms",
            "prefill_ms",
            "decode_ms",
            "ttft_ms",
            "tpot_ms",
            "input_tokens_per_s",
            "output_tokens_per_s",
            "kv_pressure",
            "gpu_memory",
            "error_rate",
            "timeout_count",
        }
        return required.issubset(set(case["metrics"]))

    def gate_runtime_boundary_clarity(self, case):
        modules = case["runtime_modules"]
        required = {
            "api_server",
            "request_queue",
            "scheduler",
            "tokenizer",
            "model_executor",
            "kv_cache_manager",
            "sampler",
            "streaming_manager",
            "metrics_exporter",
            "worker_runtime",
        }
        return required.issubset(set(modules)) and case["platform_boundary_clear"]

    def gate_cost_capacity_model(self, case):
        metrics = self.example_metrics(case)
        cost = case["cost"]
        return (
            metrics["cost_per_1k_tokens"] <= cost["budget_per_1k_tokens"]
            and metrics["output_tokens_per_s"] >= case["traffic"]["min_output_tokens_per_s"]
            and cost["cost_attribution"]
            and cost["capacity_plan"]
        )

    def gate_serving_engine_gate(self, case):
        return case["final_gate"] and not case["p0_risk"]

    def audit_case(self, case):
        failed = []
        for gate in self.gates:
            if not getattr(self, "gate_" + gate)(case):
                failed.append(gate)
        return failed

    def audit(self, cases):
        failed_cases = {}
        hard_blockers = []
        for case in cases:
            failed = self.audit_case(case)
            if failed:
                failed_cases[case["name"]] = failed
                hard_blockers.extend((case["name"], gate) for gate in failed)

        metrics = {}
        for gate in self.gates:
            passed = sum(1 for case in cases if gate not in self.audit_case(case))
            metrics[gate] = round(passed / len(cases), 3)

        remediation_sample = {
            name: [self.actions[gate] for gate in gates]
            for name, gates in list(failed_cases.items())[:3]
        }
        failed_gate_names = sorted({gate for gates in failed_cases.values() for gate in gates})
        return {
            "metrics": metrics,
            "hard_blockers": hard_blockers,
            "failed_cases": failed_cases,
            "failed_gate_names": failed_gate_names,
            "remediation_sample": remediation_sample,
            "serving_engine_gate_pass": not failed_cases,
        }
```

## 1.14 常见误区

误区一：推理框架就是封装 `model.generate()`。

真实 serving engine 要管理并发、batch、KV cache、scheduler、streaming 和 metrics。

误区二：吞吐只取决于模型大小。

吞吐还取决于 batch、KV cache、输入输出长度、scheduler、runtime kernel 和硬件。

误区三：QPS 是最重要指标。

LLM serving 更应该看 TTFT、TPOT、tokens/s、KV cache 和成本。

误区四：KV cache 只是优化细节。

KV cache 是 serving engine 的核心状态，直接决定并发能力和显存压力。

误区五：vLLM 快只是因为用了更快 kernel。

vLLM 的关键价值之一是 KV cache block 管理和调度方式，而不只是 kernel。

## 1.15 面试常见追问

问题一：LLM Serving Engine 和普通模型服务有什么区别？

可以回答：普通模型服务通常一次请求一次前向，输出固定；LLM serving 是自回归生成，有 prefill、decode、KV cache、streaming、continuous batching 和 token 级调度，资源消耗与输入输出 token 强相关。

问题二：为什么 naive generate 不能支撑生产？

可以回答：它缺少并发请求管理、动态 batching、KV cache 管理、流式返回、取消超时、指标统计和资源调度，GPU 利用率和线上稳定性都难以保证。

问题三：Serving engine 的核心模块有哪些？

可以回答：API server、request queue、scheduler、tokenizer、model executor、KV cache manager、sampler、streaming manager、metrics exporter 和 worker runtime。

问题四：为什么 scheduler 是推理框架核心？

可以回答：scheduler 决定请求何时 prefill、何时 decode、如何组成 batch、如何控制 token budget、如何处理 KV cache 和优先级，直接影响 TTFT、TPOT、吞吐和显存。

## 1.17 本章小结

本章回答了 LLM Serving Engine 到底是什么。

你需要记住：

1. LLM Serving Engine 是高并发大模型在线执行引擎，不是简单 `generate()` 封装。
2. 请求生命周期包括解析、tokenize、排队、调度、prefill、decode、streaming 和释放 KV cache。
3. Prefill 影响 TTFT，decode 影响 TPOT，KV cache 影响显存和并发。
4. Scheduler、continuous batching 和 KV cache manager 是 serving engine 的核心模块。
5. vLLM、SGLang 等框架的许多设计，本质上都是围绕请求调度和 KV cache 管理展开。
