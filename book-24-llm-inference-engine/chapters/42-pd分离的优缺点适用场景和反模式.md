到这里，应该回到一个工程决策问题：

```text
到底什么时候应该做 PD 分离？
什么时候不该做？
哪些做法看起来先进，实际是反模式？
```

一句话概括：

> PD 分离不是推理系统的默认终态，而是当 P/D 干扰、资源扩缩容耦合和尾延迟问题足够严重，并且 KV transfer、路由、监控、故障恢复能力足够成熟时，才值得引入的高级架构。

## 42.2 先给结论

PD 分离适合解决的问题是：

```text
Prefill 和 Decode 在同一个资源池中互相干扰，导致 TTFT、TPOT、尾延迟和扩缩容策略都难以同时优化。
```

它不适合解决的问题是：

```text
模型本身太慢、GPU 太少、网络太差、请求太短、监控不足、工程团队还没有能力维护分布式状态机。
```

判断标准可以先记一句：

```text
P/D 隔离收益 > KV transfer 成本 + 系统复杂度成本 + 故障恢复成本
```

如果这个不等式不成立，PD 分离就是负优化。

## 42.3 PD 分离的主要优点

PD 分离的优点可以分成六类。

第一，降低 Prefill 对 Decode 的干扰。

长 prompt prefill 不再直接插入 decode worker 的关键循环，decode TPOT 更稳定。

第二，降低 Decode 对 Prefill 的饥饿。

如果 unified scheduler 长期 decode-first，新请求可能一直等不到 prefill。PD 分离后 prefill pool 有自己的调度机会。

第三，资源池独立扩缩容。

输入压力大时扩 prefill pool，输出压力大时扩 decode pool，而不是所有 worker 都按同一种比例扩容。

第四，调度策略可以分化。

Prefill 可以追求 input tokens/s、chunked prefill、prefix cache 命中；Decode 可以追求稳定 TPOT、continuous batching、streaming 平滑度。

第五，隔离不同 batch shape。

Prefill 的 batch 往往 token 数大、shape 变化大；Decode 的 batch 每步 token 少但频率高。拆开后两个阶段不用互相迁就。

第六，为高级系统能力打基础。

例如：

1. 独立 prefill pool。
2. decode-optimized pool。
3. KV-aware routing。
4. remote KV cache。
5. long-context serving。
6. 多租户资源隔离。
7. topology-aware scheduling。

## 42.4 PD 分离的主要缺点

PD 分离不是免费午餐。

它的缺点也很明确。

第一，KV transfer 成本。

Prefill worker 生成的 KV 必须被 Decode worker 使用。KV 体积大，传输慢了会直接拉高 TTFT。

第二，分布式状态机复杂。

一个请求不再只是一个 engine 内部状态，而是跨 router、prefill worker、transfer backend、decode worker。

第三，故障恢复更难。

Prefill 成功但 transfer 失败怎么办？Decode worker 接收一半失败怎么办？Streaming 已经输出部分 token 后 worker 崩溃怎么办？这些都比单机复杂。

第四，观测要求更高。

只看总 QPS 或总 latency 不够，必须分阶段看 prefill、transfer、decode、routing、KV capacity、网络。

第五，调参空间变大。

Prefill pool 大小、decode pool 大小、chunk size、transfer 并发、router 策略、KV 预留策略、backpressure 阈值都会影响结果。

第六，资源可能空转。

如果流量形态变化大，prefill pool 可能闲着，decode pool 忙；或者反过来。资源池拆开后，利用率治理更复杂。

第七，系统尾延迟来源更多。

以前尾延迟主要来自排队、prefill、decode。现在还会来自 KV transfer、metadata lookup、remote cache、网络拥塞、worker 协调。

## 42.5 收益不是看平均值

PD 分离的收益通常体现在尾延迟和稳定性，而不是平均吞吐一定大幅提升。

例如 unified engine：

```text
TTFT p50 = 300 ms
TTFT p99 = 8 s
TPOT p50 = 35 ms
TPOT p99 = 800 ms
```

PD 分离后：

```text
TTFT p50 = 350 ms
TTFT p99 = 2 s
TPOT p50 = 38 ms
TPOT p99 = 120 ms
```

平均值可能没有明显变好，甚至 p50 稍微变差，因为多了 transfer。但 p99 明显改善，streaming 更稳定。

所以评估 PD 分离时要看：

1. TTFT p95/p99。
2. TPOT p95/p99。
3. inter-token latency tail。
4. 长 prompt 到达时 decode 是否卡顿。
5. prefill queue 是否被 decode 压制。
6. 多租户是否互相干扰。
7. 输入输出比例变化时是否能独立扩缩容。

只看平均 QPS，可能误判 PD 分离价值。

## 42.6 成本模型

可以用一个简单模型判断是否值得做。

PD 分离收益：

```text
benefit
  = reduced_decode_tail_latency
  + reduced_prefill_queue_delay
  + independent_scaling_gain
  + better_isolation_gain
```

PD 分离成本：

```text
cost
  = kv_transfer_latency
  + transfer_queue_delay
  + extra_routing_latency
  + resource_fragmentation_cost
  + failure_recovery_cost
  + engineering_complexity_cost
```

值得做的条件是：

```text
benefit > cost
```

这不是数学上精确的公式，而是工程上防止拍脑袋的检查框架。

更具体一点：

```text
PD is useful if:
  prefill_interruption is frequent
  and decode_tail_latency is SLO problem
  and kv_transfer_p95 is acceptable
  and traffic has enough scale
  and ops can debug distributed failures
```

## 42.7 适用场景一：长 Prompt 和长上下文

PD 分离最典型的适用场景是长 prompt。

例如：

1. RAG 长文档问答。
2. 代码库问答。
3. 合同、论文、病历分析。
4. 多轮对话上下文很长。
5. Agent 带大量工具描述和历史轨迹。

这些请求的特点是：

```text
prefill 很重，decode 需要稳定。
```

在 unified engine 中，一个 64000-token prompt 可能明显打断正在 streaming 的 decode 请求。

PD 分离后，长 prompt 进入 prefill pool，decode pool 可以继续稳定输出。

但要注意：长 prompt 也意味着 KV transfer 更大。只有当 transfer backend 足够强，或者可以通过 chunked prefill、KV locality、同机 PD、多级 KV cache 降低成本时，PD 分离才更容易成立。

## 42.8 适用场景二：输入输出比例波动大

有些业务流量不是稳定的。

例如白天大量短问答，晚上大量长文档总结；或者某个租户突然提交大批长 prompt。

统一 engine 的问题是：

```text
所有 worker 都同时承担 prefill 和 decode。
```

当输入压力上升时，decode 体验也被影响。

PD 分离后可以：

```text
increase prefill pool
keep decode pool stable
```

或者输出压力上升时：

```text
increase decode pool
keep prefill pool smaller
```

这种独立扩缩容在大规模多租户平台里很有价值。

## 42.9 适用场景三：Streaming SLO 很严格

如果业务强依赖 streaming 体验，TPOT 和 inter-token latency 的 p95/p99 很重要。

例如：

1. 对话助手。
2. 语音实时助手。
3. IDE coding assistant。
4. 在线客服。
5. 交互式 agent。

用户对首 token 慢可能还能接受一点，但输出过程中一卡一卡，体验非常差。

PD 分离的价值是让 decode pool 尽量不被长 prefill 打断。

如果观察到：

```text
long prefill arrives
  -> decode step latency spike
  -> streaming stalls
```

PD 分离就值得评估。

## 42.10 适用场景四：多租户隔离

多租户平台里，不同租户的 workload 可能完全不同。

一个租户提交大量长文档请求，可能拖慢其他租户的 streaming。

PD 分离可以配合资源隔离：

1. 给高优先级租户保留 decode capacity。
2. 限制低优先级租户 prefill 并发。
3. 为长 prompt 租户使用独立 prefill queue。
4. 对 decode pool 做更严格 SLO 保护。

这里 PD 分离的价值不只是性能，还有资源治理。

## 42.11 适用场景五：Prefill 和 Decode 适合不同硬件

有时 prefill 和 decode 对硬件偏好不同。

Prefill 更偏大 batch、大矩阵吞吐。

Decode 更偏低延迟、KV 读取、调度频率和显存带宽。

如果集群有异构资源，例如不同 GPU 型号、不同互联拓扑，可以考虑：

```text
prefill pool: 更强 compute 或更适合大 batch 的资源
decode pool: 更稳定低延迟和充足 KV cache 的资源
```

但异构资源会增加调度和调参复杂度，不能只因为“听起来合理”就上。

## 42.12 不适合场景一：请求都很短

如果请求大多是：

```text
prompt = 50 tokens
output = 50 tokens
```

prefill 本身不重，P/D 干扰不明显。

这时引入 PD 分离可能只会增加：

1. router 开销。
2. KV transfer 开销。
3. 状态同步开销。
4. 故障恢复复杂度。

结果是 p50/p95 都变差。

短请求场景更应该先优化：

1. batching。
2. scheduler。
3. tokenizer。
4. HTTP/gRPC overhead。
5. prefix cache。
6. replica 数量。

## 42.13 不适合场景二：流量规模不够

PD 分离需要足够规模才能摊薄复杂度。

如果一天只有少量请求，或者单机 vLLM/SGLang 已经满足 SLO，强行做 PD 分离没有意义。

因为你会付出：

1. 更多服务组件。
2. 更多监控指标。
3. 更多故障模式。
4. 更复杂部署。
5. 更高维护成本。

小规模系统更适合：

```text
single engine + continuous batching + prefix cache + chunked prefill
```

先把这些做好，再考虑 PD 分离。

## 42.14 不适合场景三：KV Transfer 条件差

如果网络或同机互联无法支撑 KV transfer，PD 分离会很难。

典型情况：

1. 跨节点只有普通以太网。
2. RDMA 不稳定。
3. GPU-GPU direct copy 能力不足。
4. KV transfer p99 很高。
5. transfer 经常和其他流量抢带宽。
6. decode worker 经常等 KV。

这时 PD 分离可能把问题从：

```text
prefill 打断 decode
```

变成：

```text
decode 等待 KV transfer
```

用户看到的仍然是慢，只是瓶颈换了位置。

## 42.15 不适合场景四：主要瓶颈不在 P/D 干扰

如果系统慢是因为：

1. 模型太大，单 token compute 慢。
2. GPU 利用率很低，batching 没做好。
3. tokenizer 或预处理很慢。
4. RAG 检索很慢。
5. 后处理或工具调用慢。
6. 客户端网络慢。
7. speculative decoding 接受率低。
8. KV cache block manager 有 bug。

PD 分离不一定解决根因。

工程上最忌讳的是没有定位瓶颈，就用复杂架构覆盖症状。

## 42.16 不适合场景五：缺少可观测性和运维能力

PD 分离需要更强的可观测性。

至少要能看到：

1. prefill queue latency。
2. prefill execution latency。
3. decode assignment latency。
4. KV transfer latency。
5. transfer queue length。
6. decode step latency。
7. TTFT breakdown。
8. TPOT breakdown。
9. worker failure rate。
10. cleanup failure count。

如果这些都没有，PD 分离上线后出问题会很难排查。

没有观测的分布式系统，不是高级架构，而是黑盒事故源。

## 42.17 反模式一：为了架构先进而 PD 分离

最常见反模式是：

```text
因为大厂论文或开源系统在做 PD 分离，所以我们也做。
```

这是错误的。

PD 分离应该由指标驱动：

1. 是否有 prefill interruption。
2. 是否有 decode p99 抖动。
3. 是否有 prefill 饥饿。
4. 是否有 P/D 扩缩容耦合。
5. KV transfer 是否可控。
6. 团队是否能维护复杂系统。

没有这些证据，不该把 PD 分离作为默认架构。

## 42.18 反模式二：只拆服务，不拆资源

有些系统把 API 上拆成 prefill service 和 decode service，但底层仍然抢同一批 GPU 或同一套调度资源。

例如：

```text
prefill service and decode service
  -> same worker pool
  -> same GPU queue
```

这只是服务名拆开了，资源干扰还在。

真正的 PD 分离至少要做到：

1. prefill 和 decode 有可区分的资源池或调度策略。
2. decode 的关键循环能被保护。
3. P/D 负载能分别观测。
4. 扩缩容或配额能分别控制。

否则收益很有限。

## 42.19 反模式三：只看 Prefill 加速，不看 Decode 等待

PD 分离后，prefill pool 可能很快完成 prompt 计算。

但如果 decode worker 没有 slot，或者 KV transfer 排队，首 token 仍然慢。

错误指标是：

```text
prefill latency decreased
```

正确指标是：

```text
TTFT breakdown improved
decode TPOT tail improved
```

一个请求的路径是：

```text
queue -> prefill -> transfer -> first decode -> streaming
```

只优化中间一段，不代表端到端体验变好。

## 42.20 反模式四：无脑跨节点 PD

跨节点 PD 看起来扩展性好，但 KV transfer 可能成为瓶颈。

如果同机资源足够，或者可以先做同机 PD，不一定要一开始跨节点。

演进路径可以是：

```text
unified engine
  -> chunked prefill
  -> same-node P/D separation
  -> topology-aware cross-node PD
```

不要直接跳到：

```text
global prefill pool + global decode pool + remote KV cache everywhere
```

这会把调试难度拉满。

## 42.21 反模式五：Remote KV Cache 当万能药

有些设计认为：只要把 KV 放进远端缓存，decode worker 随时拉，就解决了迁移问题。

问题是 remote KV fetch 仍然有成本。

如果每次首 token 都要：

```text
metadata lookup
  -> remote read
  -> network transfer
  -> CPU staging
  -> GPU promote
```

TTFT 可能比直接 prefill 还差。

Remote KV Cache 应该是 cost-based 的：

```text
use remote KV only if fetch_cost < recompute_cost and latency SLO allows it
```

## 42.22 反模式六：没有 Backpressure

PD 分离后，如果 prefill pool 只管生产 KV，decode pool 消费不过来，就会堆积。

表现为：

1. pending transfer bytes 上升。
2. decode worker KV blocks 不够。
3. transfer queue 变长。
4. prefill 计算浪费。
5. 大量请求超时。

必须有 backpressure：

```text
decode pressure -> transfer pressure -> prefill admission -> router admission
```

否则 PD 分离会在高峰期自我放大拥塞。

## 42.23 反模式七：忽略失败清理

PD 分离会产生很多中间状态：

1. prefill 已完成但 decode 未接管。
2. KV transfer 进行中。
3. decode worker 已预留 blocks。
4. router 状态已更新但 worker 失败。
5. remote KV 写入一半。

如果失败清理不完整，会造成：

1. GPU blocks 泄漏。
2. CPU staging buffer 泄漏。
3. metadata 指向不存在的 KV。
4. 请求永远卡在中间状态。
5. retry 产生重复计算。

生产系统里，清理逻辑和 happy path 一样重要。

## 42.24 反模式八：忽略租户和优先级

PD 分离后，资源池更多，队列更多。

如果没有租户隔离，一个租户的大量长 prompt 可能占满 prefill pool、transfer backend 或 remote KV cache。

需要考虑：

1. tenant quota。
2. priority queue。
3. max long-prefill concurrency。
4. per-tenant KV cache budget。
5. decode slot reservation。
6. admission control。

否则 PD 分离只是把原来的 GPU 争抢，变成多个组件上的资源争抢。

## 42.25 替代方案：先做哪些优化

在上 PD 分离之前，通常应该先评估更简单的方案。

方案一：Chunked Prefill。

把长 prompt prefill 切小，在 unified engine 中和 decode 更好地穿插。

方案二：Prefix Cache / RadixAttention。

减少重复 prefill，降低 TTFT 和 prefill 负载。

方案三：Scheduler 调参。

调整 `max_num_batched_tokens`、decode-first 策略、waiting/running 比例、preemption 策略。

方案四：增加 DP replicas。

如果模型单副本能放下，横向扩 replica 可能比 PD 分离简单。

方案五：Speculative Decoding。

如果瓶颈在长输出 decode，speculative decoding 可能比 PD 分离更直接。

方案六：多级 KV Cache。

如果问题是 idle session 或 prefix 复用，可以先做 GPU/CPU KV 管理和 local prefix cache。

PD 分离应该是这些方案仍不足以解决 P/D 干扰和扩缩容问题后的选择。

## 42.26 分阶段落地路线

不要一步到位做最复杂形态。

一个更稳妥的路线是：

```text
阶段 0: unified engine baseline
阶段 1: 加强指标，拆 TTFT/TPOT 阶段耗时
阶段 2: chunked prefill + prefix cache
阶段 3: 同机内 prefill/decode 资源隔离实验
阶段 4: 明确 KV transfer backend 和 block metadata
阶段 5: 小规模 PD pool，上线灰度流量
阶段 6: topology-aware routing 和 backpressure
阶段 7: 跨节点 PD 和 remote KV cache
```

每个阶段都要验证：

1. 哪个指标变好。
2. 哪个指标变差。
3. 是否引入新故障。
4. 是否值得进入下一阶段。

如果阶段 2 已经满足 SLO，就不一定要继续做 PD 分离。

## 42.27 判断清单

上线 PD 分离前，可以问这些问题。

Workload：

1. 长 prompt 占比是多少？
2. 长输出占比是多少？
3. P/D 比例是否波动？
4. 是否存在明显 prefill interruption？
5. 是否存在 decode-first 导致的 prefill 饥饿？

性能：

1. 当前 TTFT p95/p99 根因是什么？
2. 当前 TPOT p95/p99 根因是什么？
3. KV transfer p95/p99 预计是多少？
4. transfer 是否比 recompute 更便宜？
5. decode worker 是否会频繁等 KV？

资源：

1. Prefill 和 Decode 是否需要不同扩缩容比例？
2. GPU 资源是否足够拆池？
3. 网络是否能承受 KV transfer？
4. KV cache capacity 是否足够？
5. 是否需要多租户隔离？

工程：

1. 是否有分阶段指标？
2. 是否有失败清理？
3. 是否有 backpressure？
4. 是否有 topology-aware routing？
5. 团队是否能排查分布式故障？

如果这些问题大多答不上来，先不要做完整 PD 分离。

## 42.28 一个具体决策例子

场景 A：企业 RAG 平台。

```text
prompt p50 = 2K tokens
prompt p95 = 64K tokens
output p50 = 512 tokens
streaming SLO strict
long prompt arrives -> TPOT p99 spikes
```

这里 PD 分离值得评估，因为长 prefill 明显影响 decode。

但还要继续看：

1. KV transfer 是否可控。
2. 是否可以先 chunked prefill。
3. prefix cache 命中率如何。
4. P/D pool 比例怎么定。
5. 网络 p99 是否稳定。

场景 B：短问答客服。

```text
prompt p95 = 512 tokens
output p95 = 128 tokens
single engine TTFT/TPOT already meets SLO
```

这里不应该优先做 PD 分离。

更适合做：

1. 多 replica DP。
2. scheduler 调参。
3. prefix cache。
4. HTTP/gRPC 优化。
5. tokenizer 并行。

场景 C：代码生成。

```text
prompt medium
output very long
decode is bottleneck
```

PD 分离可能不是第一选择。应先评估 speculative decoding、decode batch、KV cache capacity、TPOT 优化。

## 42.29 和其他优化的选择关系

可以按瓶颈选择优化。

| 主要瓶颈 | 优先考虑 |
| --- | --- |
| 长 prefill 打断 decode | Chunked prefill，PD 分离 |
| 重复 prompt 多 | Prefix cache，RadixAttention |
| 长输出 decode 慢 | Speculative decoding，decode batching |
| 模型单卡放不下 | TP/PP |
| 总 QPS 不够 | DP replicas，负载均衡 |
| KV 显存不够 | PagedAttention，KV block manager，多级 KV |
| 多租户互相干扰 | Admission control，quota，PD 分离 |
| 跨节点通信慢 | 拓扑优化，减少跨节点 TP，KV locality |

PD 分离通常不是第一个按钮，而是在长 prefill、decode 稳定性、扩缩容和资源隔离问题同时出现时的系统级方案。

## 42.30 评估实验设计

评估 PD 分离时，要设计对照实验。

Baseline：

```text
unified engine + current scheduler
```

候选方案：

```text
unified + chunked prefill
unified + prefix cache
same-node PD
cross-node PD
PD + remote KV cache
```

流量要覆盖：

1. 短输入短输出。
2. 长输入短输出。
3. 短输入长输出。
4. 长输入长输出。
5. 多租户混合流量。
6. burst 流量。
7. cache hit / cache miss 混合。

指标要覆盖：

1. TTFT p50/p95/p99。
2. TPOT p50/p95/p99。
3. input tokens/s。
4. output tokens/s。
5. GPU utilization。
6. KV transfer latency。
7. pending transfer bytes。
8. prefill queue latency。
9. decode queue latency。
10. error / timeout rate。

只有当 PD 分离在目标 workload 和目标 SLO 上稳定胜出，才值得继续推进。

## 42.31 面试官会怎么问

问题一：PD 分离的优点是什么？

回答要点：隔离 Prefill 和 Decode 的资源干扰，降低长 prefill 对 decode TPOT 的影响，缓解 decode-first 导致的 prefill 饥饿，支持 P/D 独立扩缩容，并让两个阶段使用不同调度策略和资源配置。

问题二：PD 分离的缺点是什么？

回答要点：核心代价是 KV transfer 和分布式状态管理。还包括路由复杂、故障恢复复杂、观测和调参复杂、资源池可能空转、网络和 transfer p99 可能抵消收益。

问题三：什么时候适合做 PD 分离？

回答要点：长 prompt 或长上下文比例高，prefill interruption 明显，decode TPOT p99 受影响，P/D 输入输出比例波动大，需要独立扩缩容或多租户隔离，并且 KV transfer 成本可控、监控和故障恢复能力足够。

问题四：什么时候不适合做 PD 分离？

回答要点：请求短、流量小、单机 unified engine 已满足 SLO、主要瓶颈不在 P/D 干扰、KV transfer 条件差、缺少可观测性和运维能力时，不适合过早引入。

问题五：PD 分离有哪些反模式？

回答要点：为了架构先进而做；只拆服务不拆资源；只看 prefill 变快不看端到端；无脑跨节点；remote KV cache 当万能药；没有 backpressure；失败清理不完整；忽略租户隔离和优先级。

## 42.32 标准回答模板

如果面试官问“你会在什么情况下引入 PD 分离”，可以这样回答：

```text
我不会把 PD 分离当成默认架构，而会先看 workload 和指标。如果系统里长 prompt 或长上下文请求较多，unified engine 中经常出现 prefill interruption，导致 decode TPOT p95/p99 抖动，或者 decode-first 策略又让 prefill queue 饥饿，说明 P/D 混合调度已经成为问题。

这时 PD 分离的收益是把 prefill 和 decode 放到不同资源池，让 prefill pool 专注 input tokens/s 和 prompt KV 生成，让 decode pool 专注稳定 TPOT 和 streaming。它还能支持 P/D 独立扩缩容，比如输入压力大时扩 prefill pool，输出压力大时扩 decode pool。

但我会同时评估成本。核心成本是 KV transfer：prefill 生成的 KV 必须迁移或共享给 decode worker，如果 transfer p95/p99 太高，或者 decode worker 经常等 KV，PD 分离的收益会被抵消。除此之外，还要考虑 router、metadata、分布式状态机、失败清理、backpressure、观测和多租户隔离。

所以我的判断标准是：P/D 隔离收益是否大于 KV transfer 成本和系统复杂度成本。如果请求都很短、流量不大、单机 continuous batching + chunked prefill + prefix cache 已经满足 SLO，或者瓶颈其实在 tokenizer、RAG、模型 compute、网络 streaming，那我不会优先做 PD 分离。
```

## 42.33 PD 分离收益/代价门禁、反模式识别和可运行 demo

先把一个 workload case 抽象成：

```math
W_i=(P_i,O_i,T_i^{\mathrm{ttft}},T_i^{\mathrm{tpot}},K_i,C_i,R_i)
```

其中 `P_i` 是 prompt p95，`O_i` 是 output p95，`T_i^{\mathrm{ttft}}` 和 `T_i^{\mathrm{tpot}}` 是 unified engine 的尾延迟，`K_i` 是 KV transfer p99，`C_i` 是系统复杂度惩罚，`R_i` 是运维 readiness。

PD 分离的教学版收益可以写成：

```math
B_i=a\max(0,T_i^{\mathrm{ttft}}-S_{\mathrm{ttft}})+b\max(0,T_i^{\mathrm{tpot}}-S_{\mathrm{tpot}})+D_i
```

其中 `S_{\mathrm{ttft}}` 和 `S_{\mathrm{tpot}}` 是目标 SLO，`D_i` 是独立扩缩容或资源隔离收益。

代价可以写成：

```math
C_i^{\mathrm{pd}}=K_i+C_i^{\mathrm{state}}+C_i^{\mathrm{ops}}
```

一个最小决策分数：

```math
S_i^{\mathrm{pd}}=B_i-C_i^{\mathrm{pd}}
```

但 `S_i^{\mathrm{pd}}>0` 仍不够。还要过 readiness 门：

```math
G_{\mathrm{ready}}=G_{\mathrm{metric}}G_{\mathrm{cleanup}}G_{\mathrm{backpressure}}
```

最终门禁：

```math
G_{\mathrm{pdtrade}}=G_{\mathrm{positive}}G_{\mathrm{short}}G_{\mathrm{decode}}G_{\mathrm{transfer}}G_{\mathrm{ops}}G_{\mathrm{alternative}}G_{\mathrm{anti}}
```

下面的 demo 覆盖五类 case：

1. `enterprise_rag`：长 prompt、TPOT p99 抖动、transfer 可控、运维能力齐备，推荐先做 same-node PD。
2. `short_qa`：短请求，unified 已经满足，拒绝 PD，建议 DP replicas。
3. `code_generation`：主要瓶颈是长输出 decode，不优先 PD，建议 speculative decoding。
4. `cross_node_bad_network`：有长 prompt 干扰，但跨节点 KV transfer 太慢，拒绝跨节点 PD。
5. `long_prompt_no_ops`：分数为正，但缺 metrics / cleanup / backpressure，先补观测和治理。

```python
from dataclasses import dataclass


@dataclass
class PDCase:
    name: str
    prompt_p95: int
    output_p95: int
    unified_ttft_p99_ms: float
    unified_tpot_p99_ms: float
    target_ttft_ms: float
    target_tpot_ms: float
    kv_transfer_p99_ms: float
    scaling_saving_ms: float
    complexity_penalty_ms: float
    prefill_interference: bool
    decode_bottleneck: bool
    metrics_ready: bool
    cleanup_ready: bool
    backpressure_ready: bool


class ToyPDTradeoffAuditor:
    def benefit_score(self, case):
        ttft_tail_saving = max(0.0, case.unified_ttft_p99_ms - case.target_ttft_ms) * 0.05
        tpot_tail_saving = max(0.0, case.unified_tpot_p99_ms - case.target_tpot_ms) * 0.2
        if not case.prefill_interference:
            ttft_tail_saving *= 0.25
            tpot_tail_saving *= 0.25
        cost = case.kv_transfer_p99_ms + case.complexity_penalty_ms
        return round(ttft_tail_saving + tpot_tail_saving + case.scaling_saving_ms - cost, 3)

    def classify(self, case):
        score = self.benefit_score(case)
        reasons = []
        alternatives = []
        if case.prompt_p95 < 1024 and case.output_p95 < 512:
            reasons.append("short_requests")
            alternatives.append("dp_replicas")
        if case.decode_bottleneck and not case.prefill_interference:
            reasons.append("decode_bottleneck_not_pd")
            alternatives.append("speculative_decoding")
        if case.kv_transfer_p99_ms > 250:
            reasons.append("kv_transfer_too_slow")
            alternatives.append("same_node_pd_or_chunked_prefill")
        if not (case.metrics_ready and case.cleanup_ready and case.backpressure_ready):
            reasons.append("ops_readiness_missing")
            alternatives.append("instrument_first")
        if not case.prefill_interference:
            reasons.append("no_prefill_interference_evidence")
        decision = "adopt_pd" if score > 0 and not reasons else "do_not_adopt_pd_yet"
        if decision == "adopt_pd":
            reasons.append("benefit_exceeds_transfer_and_complexity")
            alternatives.append("same_node_pd_first")
        return {
            "case": case.name,
            "score": score,
            "decision": decision,
            "reasons": reasons,
            "alternatives": alternatives,
        }

    def audit(self, cases):
        rows = [self.classify(case) for case in cases]
        summary = {
            "cases": len(rows),
            "adopt_pd": sum(row["decision"] == "adopt_pd" for row in rows),
            "rejected": sum(row["decision"] != "adopt_pd" for row in rows),
            "anti_patterns": sum(
                any(reason in row["reasons"] for reason in [
                    "short_requests",
                    "decode_bottleneck_not_pd",
                    "kv_transfer_too_slow",
                    "ops_readiness_missing",
                    "no_prefill_interference_evidence",
                ])
                for row in rows
            ),
            "positive_cases": [row["case"] for row in rows if row["decision"] == "adopt_pd"],
            "blocked_cases": [row["case"] for row in rows if row["decision"] != "adopt_pd"],
        }
        gates = {
            "positive_case_visible": summary["positive_cases"] == ["enterprise_rag"],
            "short_request_rejected": any("short_requests" in row["reasons"] for row in rows),
            "decode_bottleneck_not_pd": any("decode_bottleneck_not_pd" in row["reasons"] for row in rows),
            "slow_transfer_rejected": any("kv_transfer_too_slow" in row["reasons"] for row in rows),
            "ops_readiness_required": any("ops_readiness_missing" in row["reasons"] for row in rows),
            "cost_model_used": all("score" in row for row in rows),
            "alternatives_visible": all(row["alternatives"] for row in rows),
            "anti_patterns_visible": summary["anti_patterns"] >= 4,
        }
        gates["pd_tradeoff_gate"] = all(gates.values())
        return rows, summary, gates


cases = [
    PDCase("enterprise_rag", 64000, 512, 8000, 800, 2000, 120, 80, 120, 150, True, False, True, True, True),
    PDCase("short_qa", 512, 128, 600, 70, 1000, 100, 3, 5, 120, False, False, True, True, True),
    PDCase("code_generation", 4096, 4096, 1600, 650, 1800, 200, 20, 20, 140, False, True, True, True, True),
    PDCase("cross_node_bad_network", 64000, 512, 8000, 800, 2000, 120, 700, 120, 160, True, False, True, True, True),
    PDCase("long_prompt_no_ops", 32000, 512, 6000, 500, 2000, 120, 60, 80, 150, True, False, False, False, False),
]
rows, summary, gates = ToyPDTradeoffAuditor().audit(cases)
print("pd_tradeoff_rows=", rows)
print("pd_tradeoff_summary=", summary)
print("pd_tradeoff_gates=", gates)
```

一次运行的核心输出类似：

```text
pd_tradeoff_summary= {'cases': 5, 'adopt_pd': 1, 'rejected': 4, 'anti_patterns': 4, 'positive_cases': ['enterprise_rag'], 'blocked_cases': ['short_qa', 'code_generation', 'cross_node_bad_network', 'long_prompt_no_ops']}
pd_tradeoff_gates= {'positive_case_visible': True, 'short_request_rejected': True, 'decode_bottleneck_not_pd': True, 'slow_transfer_rejected': True, 'ops_readiness_required': True, 'cost_model_used': True, 'alternatives_visible': True, 'anti_patterns_visible': True, 'pd_tradeoff_gate': True}
```

这个 demo 展示了几个关键点：

1. `enterprise_rag` 是正例：长 prompt 干扰明显、transfer 可控、运维能力具备，才进入 `adopt_pd`。
2. `short_qa` 被拒绝：短请求不应为了架构先进而 PD 分离。
3. `code_generation` 被拒绝：瓶颈在长输出 decode，优先 speculative decoding 或 decode batching。
4. `cross_node_bad_network` 被拒绝：KV transfer p99 太高，跨节点 PD 会把收益吃掉。
5. `long_prompt_no_ops` 被拒绝：即使分数为正，metrics、cleanup 和 backpressure 不齐也不能上线。
6. `alternatives_visible=True`：每个 rejected case 都给出替代路径，避免把 PD 当成唯一按钮。

所以本章最终门禁是 `pd_tradeoff_gate`：只有正例、短请求反例、decode 瓶颈反例、慢 transfer 反例、运维 readiness、成本模型、替代方案和反模式识别都能闭环，PD 分离决策才是工程判断，而不是架构偏好。

## 42.35 本章总结

PD 分离的核心收益是隔离 Prefill 和 Decode 的资源干扰，让两个阶段可以使用不同资源池、调度策略和扩缩容策略，从而改善 TTFT/TPOT 尾延迟和多租户资源治理。

它的核心成本是 KV transfer 和分布式状态管理，并且会引入路由、metadata、故障恢复、backpressure、观测和调参复杂度。

PD 分离适合长 prompt、长上下文、streaming SLO 严格、P/D 比例波动大、多租户隔离要求强，并且 KV transfer 条件可控的场景。

它不适合短请求、小流量、单机已满足 SLO、瓶颈不在 P/D 干扰、网络条件差或缺少可观测性与运维能力的场景。

下一章会进入更落地的问题：如何把一个单机 serving engine 逐步升级成 PD 分离架构。重点不是重新写一个系统，而是识别哪些模块要抽象、哪些状态要外置、哪些接口要先稳定下来。
