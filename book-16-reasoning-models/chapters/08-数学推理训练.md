# 第八章：数学推理训练

数学推理是 reasoning model 最重要的训练场景之一。原因很直接：数学题目标明确、步骤可分析、答案常常可验证，适合研究长思维链、过程监督、verifier、搜索和 test-time compute scaling。很多 reasoning 模型能力的提升，都会先在数学基准上体现出来。

本章系统讲数学推理训练：数学数据从哪里来，为什么只看最终答案不够，如何构造步骤数据和合成题，verifier 如何参与训练与筛选，难度 curriculum 如何设计，以及数学推理训练常见风险和面试表达。

## 0. 本讲资料边界与第二轮精修口径

本讲第二轮精修参考公开数学 reasoning 资料，包括 [GSM8K / Training Verifiers to Solve Math Word Problems](https://arxiv.org/abs/2110.14168)、[MATH dataset](https://arxiv.org/abs/2103.03874)、[Let's Verify Step by Step / PRM800K](https://arxiv.org/abs/2305.20050)、[DeepSeekMath](https://arxiv.org/abs/2402.03300)、[OpenWebMath](https://arxiv.org/abs/2310.06786)、[AlphaGeometry](https://www.nature.com/articles/s41586-023-06747-5) 和 [NuminaMath](https://huggingface.co/datasets/AI-MO/NuminaMath-CoT) 等资料。它们共同提示：数学推理能力不是单一 prompt 技巧，而是由数学语料、步骤数据、合成数据、答案验证、过程监督、verifier、curriculum、RL / rejection sampling 和严格评估共同组成的训练闭环。

本章不把数学 benchmark 分数写成通用 reasoning 能力的充分证据。数学题常常更封闭、更可验证，也更容易被公开题库、题解和合成模板污染。第二轮精修重点是把“训练数学推理”拆成可审计对象：样本来源是否干净、答案是否可验证、步骤是否真的正确、错误样本是否能定位第一处错误、合成题是否模板化、难题切片是否改善、训练成本和 test-time compute 是否值得。

## 8.1 为什么数学适合训练 Reasoning

数学任务有几个天然优势：

1. 答案通常明确。
2. 推理过程有结构。
3. 错误步骤较容易定位。
4. 可以构造大量合成题。
5. 很多题可以自动验证。
6. 难度可以分层。

例如一道代数题，模型不仅要给出最终数值，还要完成变形、代入、化简、检查条件等步骤。这些步骤非常适合训练长链条推理。

面试回答：

```text
数学是 reasoning model 的重要训练场景，因为数学问题有明确答案、推理链条较清晰、部分任务可以程序化验证，还能通过合成题控制难度。它适合训练 CoT、process supervision、verifier 和 test-time search。但数学能力不能完全代表通用推理能力，因为现实任务往往更开放、约束更模糊。
```

### 8.1.1 关键公式与数学推理训练指标速查

把第 `i` 道数学训练样本写成：

```math
m_i=(x_i,y_i^\star,z_i,q_i,d_i,t_i,s_i,r_i)
```

其中 `x_i` 是题目，`y_i^\star` 是标准答案，`z_i=(z_{i1},\ldots,z_{iL_i})` 是解题步骤，`q_i=(q_{i1},\ldots,q_{iL_i})` 是步骤标签，`d_i` 是难度，`t_i` 是题型，`s_i` 是来源或模板标识，`r_i` 是污染、重复、验证失败等风险标记。

只做最终答案监督时，简化目标可以写成：

```math
L_{\mathrm{ans}}(\theta)=
-\frac{1}{N}
\sum_{i=1}^{N}
\log P_\theta(y_i^\star\mid x_i,z_i)
```

如果训练时只给题目和最终答案，就把条件中的 `z_i` 去掉。这个目标容易落地，但它不知道哪一步错，也无法区分“正确推理后算错”和“乱推但猜对”。

过程监督把第 `j` 步是否正确建模为 `q_{ij}`：

```math
L_{\mathrm{proc}}(\phi)=
-\frac{1}{M}
\sum_{i,j}
\left[
q_{ij}\log \hat q_{ij}
+
(1-q_{ij})\log(1-\hat q_{ij})
\right]
```

其中 `\hat q_{ij}` 是 verifier 或 PRM 对第 `j` 步正确性的预测，`M` 是被标注的步骤总数。训练时要单独记录自动标注和人工标注比例，因为过程标签很贵。

最终答案准确率：

```math
A_{\mathrm{ans}}=
\frac{1}{N}
\sum_{i=1}^{N}
\mathbb{1}[\hat y_i=y_i^\star]
```

步骤准确率：

```math
A_{\mathrm{step}}=
\frac{
\sum_{i,j}\mathbb{1}[\hat q_{ij}=q_{ij}]
}{
\sum_i L_i
}
```

第一处错误定位准确率：

```math
A_{\mathrm{first}}=
\frac{1}{N_{\mathrm{err}}}
\sum_{i\in \mathcal{E}}
\mathbb{1}[\hat e_i=e_i^\star]
```

其中 `\mathcal{E}` 是存在错误步骤的样本集合，`e_i^\star` 是人工或高可信 verifier 标出的第一处错误位置。

自动验证覆盖率：

```math
C_{\mathrm{verify}}=
\frac{1}{N}
\sum_{i=1}^{N}
\mathbb{1}[v_i=1]
```

这里 `v_i=1` 表示这道题至少可以用答案归一化、数值代入、符号化简、穷举、单元测试或规则检查中的一种方法验证。覆盖率高不代表证明完整，只表示能拿到一个较硬的自动检查信号。

训练污染率：

```math
R_{\mathrm{contam}}=
\frac{1}{N}
\sum_{i=1}^{N}
\mathbb{1}[o_i=1]
```

其中 `o_i=1` 表示样本和评测集题面、答案、解析或高度相似变体存在重叠风险。数学推理训练尤其要监控这个指标，因为公开题库和题解非常容易进入预训练或 SFT 数据。

合成题模板多样性：

```math
D_{\mathrm{temp}}=
\frac{|\{s_i\}_{i=1}^{N}|}{N}
```

如果大量样本来自同一个模板，模型可能学会模板模式，而不是稳健推理。模板多样性要和题型覆盖、难度覆盖一起看。

某个难度切片的准确率：

```math
A_g=
\frac{
\sum_{i\in \mathcal{G}_g}
\mathbb{1}[\hat y_i=y_i^\star]
}{
|\mathcal{G}_g|
}
```

其中 `\mathcal{G}_g` 可以是 easy、medium、hard，也可以是 algebra、geometry、proof、combinatorics 等题型切片。面试中要强调：数学训练最怕平均分提升，但 hard slice、证明题或几何题完全没改善。

一个简化数学推理训练门禁：

```math
G_{\mathrm{math}}=
\mathbb{1}[
A_{\mathrm{ans}}\ge \alpha
\land
A_{\mathrm{step}}\ge \beta
\land
A_{\mathrm{first}}\ge \gamma
\land
R_{\mathrm{contam}}\le \rho
\land
D_{\mathrm{temp}}\ge \delta
\land
A_{\mathrm{hard}}\ge \eta
]
```

这个门禁表达的是训练数据和模型效果的共同要求：答案要对，步骤要可靠，错误要能定位，评测不能污染，合成题不能过度模板化，难题切片不能被平均分掩盖。

## 8.2 数学数据类型

数学训练数据可以分为几类：

1. 题目和最终答案。
2. 题目、解题步骤和最终答案。
3. 多种解法数据。
4. 错误解法和纠错数据。
5. 难度分级数据。
6. 合成题数据。
7. 工具验证数据。

只包含最终答案的数据可以训练模型输出答案，但对推理路径帮助有限。包含步骤的数据更适合训练模型如何思考。包含错误步骤的数据可以训练模型识别和修正错误。

## 8.3 最终答案监督的局限

Outcome supervision 只看答案是否正确。

它的问题是：

1. 不知道哪一步错。
2. 无法奖励部分正确推理。
3. 可能奖励碰巧猜对的答案。
4. 难以训练中间步骤质量。
5. 对长链条问题信号稀疏。

例如模型前面推理全对，最后算错一个符号，最终答案错误。只看最终答案会把整条链都当成失败样本。反过来，模型中间推理乱写，但最后猜对答案，也可能被错误奖励。

这就是为什么数学推理训练通常需要 CoT 数据、process supervision 或 verifier。

## 8.4 解题步骤数据

高质量数学步骤数据通常包含：

1. 明确的问题陈述。
2. 分步推导。
3. 每一步的理由。
4. 最终答案。
5. 必要的条件检查。
6. 可选的替代解法。

一个好的步骤不是越长越好，而是每一步都承担推理功能。

常见坏步骤包括：

1. 跳步严重。
2. 只写结论不写理由。
3. 格式很漂亮但逻辑错误。
4. 中间变量未定义。
5. 最终答案和推导不一致。
6. 过度冗长，堆砌无用解释。

训练 reasoning model 时，步骤质量比步骤数量更重要。

## 8.5 合成题

合成题是数学推理训练的重要数据来源。它可以通过程序、模板、模型生成或人机协作构造。

优点：

1. 数量大。
2. 难度可控。
3. 答案可自动计算。
4. 可以覆盖稀有题型。
5. 便于构造 curriculum。

风险：

1. 模板痕迹明显。
2. 分布和真实考试题不一致。
3. 题目过于机械。
4. 模型学会题型模式而不是推理。
5. 合成答案或步骤可能有错。

因此合成题通常需要过滤、去重、难度分层和验证。

## 8.6 难度 Curriculum

Curriculum learning 的思想是从简单到困难训练。

数学推理中可以按以下维度分难度：

1. 步骤长度。
2. 概念数量。
3. 是否需要分类讨论。
4. 是否需要构造辅助变量。
5. 是否需要证明而不是计算。
6. 是否需要跨章节知识。
7. 是否容易被自动验证。

一个简单 curriculum：

```text
基础计算 -> 代数变形 -> 应用题 -> 几何证明 -> 组合计数 -> 奥赛难题
```

课程学习的价值是让模型先学稳定的基础推理模式，再逐步接触更长、更抽象、更难验证的问题。

## 8.7 Verifier 在数学训练中的作用

Verifier 可以用于多个环节：

1. 数据清洗：过滤错误答案。
2. 步骤评分：判断中间推理是否合理。
3. 候选选择：从多个解法中选更好的一条。
4. RL reward：作为强化学习奖励信号。
5. 推理时 reranking：选择最终输出。
6. 搜索剪枝：提前丢弃错误路径。

数学 verifier 有几种形态：

1. 规则验证器。
2. 符号计算验证器。
3. 程序执行验证器。
4. LLM verifier。
5. Process reward model。

程序化 verifier 最可靠，但覆盖面有限。LLM verifier 覆盖面广，但可能被表面合理的错误推导欺骗。

## 8.8 自动验证和符号计算

很多数学问题可以用工具验证，例如：

1. 数值计算。
2. 方程求解。
3. 表达式化简。
4. 代入检查。
5. 穷举验证。
6. 随机测试。

例如代数恒等式可以随机代入多个数检查；组合题可以在小规模上穷举验证；函数题可以用采样检查反例。

但自动验证不能替代完整证明。一个表达式在若干点上成立，不代表恒等成立；一个数值答案正确，不代表推导过程正确。数学训练需要区分“答案验证”和“过程证明”。

## 8.9 多解法训练

同一道题可能有多种解法。多解法数据有助于模型学习不同思路。

例如：

1. 代数法。
2. 几何法。
3. 构造法。
4. 反证法。
5. 归纳法。
6. 穷举和观察模式。

多解法训练能提升模型的探索能力，也有利于 test-time search。因为搜索时模型需要生成不同方向，而不是多次生成同一种解法。

## 8.10 错误解法和纠错数据

只训练正确解法不够。Reasoning model 还需要学会发现错误。

错误数据可以包括：

1. 符号错误。
2. 条件遗漏。
3. 除以可能为零的量。
4. 分类讨论不完整。
5. 偷换概念。
6. 证明中使用未证结论。
7. 计算中间步骤错误。

训练形式可以是：

```text
给定题目和错误解法 -> 找出第一处错误 -> 解释原因 -> 给出修正
```

这类数据对 verifier、process supervision 和反思修正都很有价值。

## 8.11 RL 在数学推理中的使用

数学任务适合做 RL 的原因是 reward 相对容易定义。最终答案正确可以给 reward，步骤正确也可以给 reward，工具验证通过也可以给 reward。

常见做法：

1. 用 SFT 训练基础解题能力。
2. 采样多条解法。
3. 用 verifier 或答案检查打分。
4. 用 RL 提高高 reward 解法概率。
5. 结合长度、格式和安全约束。

风险是模型可能 reward hacking。例如学会输出 verifier 偏好的格式，或者通过猜测最终答案拿分，而不是形成真实推理能力。

## 8.12 数学推理评估

评估不能只看一个 benchmark 分数。常见维度包括：

1. 最终答案准确率。
2. 步骤正确率。
3. 第一处错误位置。
4. 不同难度层级表现。
5. 不同题型表现。
6. 多次采样稳定性。
7. 推理长度和成本。
8. 工具验证通过率。
9. 对变体题的泛化能力。

如果一个模型在训练分布题型上很强，但换个表述就失败，说明它可能学到了模式匹配，而不是稳健推理。

## 8.13 数据泄漏风险

数学 benchmark 常有数据泄漏风险。模型可能在预训练中见过题目、答案或相似解析。

风险来源：

1. 公开题库。
2. 论坛解析。
3. 竞赛题解。
4. 训练集和测试集重叠。
5. 合成题模板重复。

缓解方式：

1. 去重。
2. 使用新题。
3. 改写题面并检查稳定性。
4. 构造参数化变体。
5. 分析错误类型而不只看准确率。

## 8.14 常见失败模式

1. 只会套模板，不会处理变体。
2. 推理步骤很长，但核心逻辑错误。
3. 答案正确，过程不可靠。
4. 过程看似漂亮，但跳过关键证明。
5. Verifier 被格式欺骗。
6. 合成题分布太窄。
7. Curriculum 太简单，迁移不到难题。
8. 过度优化 benchmark。

数学推理训练的目标不是让模型背题，而是让模型掌握可迁移的推理操作。

```python
by_topic = defaultdict(lambda: [0, 0])
for sample in SAMPLES:
    group = bucket(sample["difficulty"])
    by_bucket[group][0] += int(sample["answer_correct"])
    by_bucket[group][1] += 1
    by_topic[sample["topic"]][0] += int(sample["answer_correct"])
    by_topic[sample["topic"]][1] += 1

accepted = [
    sample for sample in SAMPLES
    if sample["answer_correct"]
    and sample["verifier_pass"]
    and not sample["train_overlap"]
]
accepted = sorted(accepted, key=lambda item: (item["difficulty"], item["id"]))
accepted_topics = Counter(sample["topic"] for sample in accepted)
training_mix = {
    topic: ratio(count, len(accepted))
    for topic, count in sorted(accepted_topics.items())
}

summary = {
    "answer_accuracy": ratio(answer_correct, total),
    "verifier_pass_rate": ratio(verified, total),
    "contamination_rate": ratio(len(contaminated), total),
    "unique_template_ratio": ratio(len(templates), total),
    "topic_coverage": ratio(len(covered_topics & TARGET_TOPICS), len(TARGET_TOPICS)),
}
process_metrics = {
    "step_accuracy": ratio(step_correct, len(all_steps)),
    "relevant_step_ratio": ratio(step_relevant, len(all_steps)),
    "auto_label_coverage": ratio(auto_labeled, len(all_steps)),
    "human_label_cost": human_cost,
    "first_error_accuracy": ratio(first_error_hits, len(error_cases)),
}
slice_accuracy = {
    "difficulty": {
        name: ratio(values[0], values[1])
        for name, values in sorted(by_bucket.items())
    },
    "topic": {
        name: ratio(values[0], values[1])
        for name, values in sorted(by_topic.items())
    },
}
risks = {
    "contaminated": contaminated,
    "verified_wrong": [
        sample["id"] for sample in SAMPLES
        if sample["verifier_pass"] and not sample["answer_correct"]
    ],
    "hard_failures": [
        sample["id"] for sample in SAMPLES
        if bucket(sample["difficulty"]) == "hard" and not sample["answer_correct"]
    ],
    "repeated_templates": repeated_templates,
}
gates = {
    "answer_quality": summary["answer_accuracy"] >= 0.75,
    "process_quality": (
        process_metrics["step_accuracy"] >= 0.8
        and process_metrics["first_error_accuracy"] >= 0.8
    ),
    "contamination_ok": summary["contamination_rate"] == 0.0,
    "coverage_ok": summary["topic_coverage"] >= 0.8,
    "hard_slice_ok": slice_accuracy["difficulty"].get("hard", 0.0) >= 0.5,
}

print(f"data_summary={summary}")
print(f"process_metrics={process_metrics}")
print(f"slice_accuracy={slice_accuracy}")
print(f"curriculum_order={[sample['id'] for sample in accepted]}")
print(f"training_mix={training_mix}")
print(f"risks={risks}")
print(f"gates={gates}")
print(f"gate_pass={all(gates.values())}")
```

## 8.16 面试题：如何构造数学推理训练数据

回答要点：

```text
我会把数学数据分成题目答案数据、步骤解法数据、多解法数据、错误纠错数据和合成题数据。数据构造时需要保证答案正确、步骤清晰、难度分层，并用规则、符号计算或 LLM verifier 做过滤。对合成题要注意模板化和分布偏差，对 benchmark 要注意数据泄漏。最终还要按题型、难度和步骤正确率评估，而不只看最终答案准确率。
```

## 8.17 面试题：数学 verifier 有什么用

回答要点：

```text
数学 verifier 可以用于数据清洗、候选 reranking、过程评分、RL reward 和推理时搜索剪枝。程序化 verifier 在可计算任务上更可靠，但覆盖范围有限；LLM verifier 覆盖范围更广，但可能被表面合理的错误推导欺骗。因此实际系统通常会结合规则、工具、符号计算和模型评分。
```

## 8.19 本章小结

数学推理训练是 reasoning model 的核心训练场景。它提供了明确目标、结构化步骤、可验证答案和可控难度，非常适合训练 CoT、process supervision、verifier、search 和 test-time compute scaling。

但数学训练也有风险：合成题可能模板化，benchmark 可能泄漏，模型可能学会格式而不是推理，verifier 可能被欺骗。真正可靠的数学 reasoning 需要高质量步骤数据、多解法、错误纠错、自动验证、难度 curriculum 和严格评估共同支撑。下一章会进入代码推理与执行反馈，讨论代码任务中如何利用测试、编译器和运行时反馈训练 reasoning model。
