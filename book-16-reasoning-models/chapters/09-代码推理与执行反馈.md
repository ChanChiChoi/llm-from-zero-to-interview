# 第九章：代码推理与执行反馈

代码任务是 reasoning model 的另一个核心训练场景。和数学一样，代码有明确目标、结构化过程和可验证结果；但代码比数学更工程化，因为模型可以运行程序、读取报错、修改代码、再运行测试。这种“生成 -> 执行 -> 反馈 -> 修正”的闭环，是训练和评估可验证推理能力的重要方式。

本章系统讲代码推理与执行反馈：代码生成为什么适合 reasoning，单元测试和解释器反馈如何提供 reward，self-debug 如何工作，程序合成和搜索如何结合，训练数据如何构造，以及工程上如何处理沙箱、安全和成本。

## 0. 本讲资料边界与第二轮精修口径

本讲第二轮精修参考公开代码 reasoning 资料，包括 [Codex / HumanEval](https://arxiv.org/abs/2107.03374)、[AlphaCode](https://arxiv.org/abs/2203.07814)、[CodeRL](https://arxiv.org/abs/2207.01780)、[Self-Debugging](https://arxiv.org/abs/2304.05128)、APPS、MBPP 和 SWE-bench 等代码生成、程序合成、执行反馈与真实仓库修复评测资料。这些资料共同提示：代码 reasoning 不能只看代码文本相似度，而要看 functional correctness、测试覆盖、pass@k、公开 / 隐藏测试差距、修复闭环、执行环境和安全沙箱。

本章不提供绕过沙箱、逃逸执行环境或访问真实系统资源的技巧。所有代码示例只用于防御性审计和教学，强调静态拦截、资源限制、测试隔离和执行日志。真实系统不能直接运行模型生成的任意代码；必须用隔离容器、最小权限、超时、内存限制、网络隔离、文件系统隔离和审计日志。

## 9.1 为什么代码适合训练 Reasoning

代码任务有几个特点：

1. 输入目标通常明确。
2. 输出可以被编译或运行。
3. 测试可以判断行为是否正确。
4. 错误信息提供中间反馈。
5. 可以迭代修复。
6. 代码结构天然适合分解。

数学题通常验证最终答案，代码题可以验证行为。例如一个函数是否正确，不只看文本是否像代码，而是看它能不能通过测试。

面试回答：

```text
代码任务适合 reasoning model，因为代码可以执行，编译器、解释器和单元测试能提供可验证反馈。模型可以生成代码、运行测试、读取错误信息并迭代修复。这让代码任务非常适合训练 self-debug、tool-use reasoning、verifier 和执行反馈驱动的强化学习。
```

### 9.1.1 关键公式与代码执行反馈指标速查

把第 `i` 个代码任务写成：

```math
c_i=(x_i,\sigma_i,T_i^{\mathrm{pub}},T_i^{\mathrm{hid}},\mathcal{P}_i,e_i,b_i)
```

其中 `x_i` 是需求，`\sigma_i` 是函数签名或接口约束，`T_i^{pub}` 是公开测试，`T_i^{hid}` 是隐藏测试，`\mathcal{P}_i` 是候选程序集合，`e_i` 是执行环境，`b_i` 是时间、内存、候选数和迭代轮数预算。

候选程序 `p_{ij}` 在测试集合 `T` 上的通过率：

```math
S(p_{ij},T)=
\frac{1}{|T|}
\sum_{\tau\in T}
\mathbb{1}[\mathrm{run}(p_{ij},\tau)=1]
```

公开测试通过率和隐藏测试通过率分别是 `S(p,T_pub)` 和 `S(p,T_hid)`。公开测试可以用于反馈和筛选，隐藏测试只能用于最终评估或离线审计。

代码生成常用 pass@k 衡量“生成 `k` 个候选时至少有一个正确候选”的概率。给定 `n` 个候选，其中 `c` 个通过隐藏测试，不放回估计为：

```math
\mathrm{pass@}k=
1-
\frac{\binom{n-c}{k}}
{\binom{n}{k}}
```

注意 pass@k 不等于线上单次调用可靠性。它更像候选池潜力，仍然需要选择器、测试覆盖和预算控制。

公开 / 隐藏泛化差距：

```math
\Delta_{\mathrm{pub-hid}}=
A_{\mathrm{pub}}-A_{\mathrm{hid}}
```

如果 `A_pub` 很高但 `A_hid` 很低，说明模型或 reranker 可能过拟合公开样例、边界条件覆盖不足，或者候选选择器偏向“样例投机”代码。

一次 self-debug 轨迹可以写成：

```math
\tau_i=(p_i^{(0)},f_i^{(0)},p_i^{(1)},f_i^{(1)},\ldots,p_i^{(R_i)},f_i^{(R_i)})
```

其中 `p_i^{(r)}` 是第 `r` 轮代码，`f_i^{(r)}` 是编译器、解释器、单元测试、类型检查或静态分析反馈。

修复成功率：

```math
A_{\mathrm{repair}}=
\frac{1}{N_{\mathrm{fail}}}
\sum_{i\in \mathcal{F}}
\mathbb{1}[
S(p_i^{(R_i)},T_i^{hid})=1
]
```

其中 `\mathcal{F}` 是初始候选没有通过公开测试或验证反馈明确失败的任务集合。

平均首次通过轮数：

```math
R_{\mathrm{first}}=
\frac{1}{N}
\sum_{i=1}^{N}
\min\{r:S(p_i^{(r)},T_i^{pub})=1\}
```

如果很多任务需要多轮才能过公开测试，说明初始生成质量或需求理解可能不足；如果公开测试很快过但隐藏测试差，说明测试覆盖和泛化有问题。

沙箱违规率：

```math
R_{\mathrm{sandbox}}=
\frac{1}{M}
\sum_{i,j}
\mathbb{1}[u_{ij}=1]
```

其中 `u_ij=1` 表示候选代码触发导入禁用模块、文件访问、网络访问、无限循环、子进程、危险调用或资源超限等拦截规则。这个指标不是为了教模型如何绕过限制，而是为了证明执行系统没有把危险候选放进真实运行环境。

一个简化代码 reasoning 上线门禁：

```math
G_{\mathrm{code}}=
\mathbb{1}[
A_{\mathrm{hid}}\ge \alpha
\land
\Delta_{\mathrm{pub-hid}}\le \delta
\land
A_{\mathrm{repair}}\ge \beta
\land
R_{\mathrm{sandbox}}\le \rho
\land
C_{\mathrm{exec}}\le C_{\max}
]
```

其中 `C_exec` 是候选生成、执行、测试和修复迭代的总成本。面试里要强调：代码 reasoning 的可靠性来自可执行反馈闭环，但上线门禁必须同时看隐藏泛化、修复能力、沙箱安全和成本。

## 9.2 代码推理的基本闭环

一个典型代码推理闭环是：

```text
理解需求 -> 设计方案 -> 生成代码 -> 执行测试 -> 读取反馈 -> 修改代码 -> 再测试
```

这个闭环比一次性代码生成更可靠。因为模型第一次生成的代码可能有语法错误、边界条件遗漏、性能问题或接口不匹配。执行反馈可以把这些错误暴露出来。

Reasoning model 的价值在于，它不只是写代码，还能根据反馈定位错误并制定修复策略。

## 9.3 执行反馈有哪些类型

常见执行反馈包括：

1. 语法错误。
2. 编译错误。
3. 运行时异常。
4. 单元测试失败。
5. 超时。
6. 内存超限。
7. 静态检查错误。
8. 类型检查错误。
9. 覆盖率不足。

不同反馈提供不同信号。语法错误通常容易修；测试失败更难，因为它只说明行为不符合预期；超时和内存超限则常常意味着算法复杂度不合适。

## 9.4 单元测试作为 Verifier

在代码任务中，单元测试是最常见的 verifier。

它可以用于：

1. 判断候选代码是否正确。
2. 给不同候选排序。
3. 作为 RL reward。
4. 作为搜索剪枝依据。
5. 作为 self-debug 的反馈。

例如：

```text
candidate A: 通过 3/10 tests
candidate B: 通过 8/10 tests
candidate C: 通过 10/10 tests
```

系统可以优先保留 candidate C，或者让模型分析 candidate B 的失败用例继续修复。

测试的优势是客观、可执行、自动化。局限是测试覆盖不一定完整，通过测试不等于完全正确。

## 9.5 Self-Debug

Self-debug 指模型根据错误反馈自我修复。

基本流程：

```text
生成代码
运行测试
读取报错
解释失败原因
修改代码
再次运行
```

一个好的 self-debug 过程不是盲目改代码，而是先定位失败类型。

例如：

1. 是接口错误还是算法错误。
2. 是边界条件遗漏还是复杂度不够。
3. 是数据结构选错还是状态更新错。
4. 是测试理解错误还是代码实现错误。

训练 self-debug 时，可以构造“错误代码 + 报错信息 + 修复后代码”的数据。

## 9.6 程序合成

程序合成是从规格或示例中生成满足要求的程序。它和 search reasoning 关系密切，因为一个程序可以通过多种方式构造。

程序合成常见输入：

1. 自然语言需求。
2. 函数签名。
3. 输入输出样例。
4. 隐藏测试。
5. 类型约束。
6. 性能约束。

程序合成中的搜索空间非常大，因此需要约束、测试和剪枝。LLM 可以生成候选程序，测试系统提供反馈，搜索控制器决定继续修改哪个候选。

## 9.7 搜索和代码修复

代码修复可以看成搜索问题。

```text
state = 当前代码和测试结果
action = 一次代码修改
transition = 应用修改并运行测试
score = 测试通过率、错误减少程度、代码质量
termination = 测试通过或预算耗尽
```

常见策略：

1. 生成多个候选补丁。
2. 运行测试评分。
3. 保留通过更多测试的补丁。
4. 针对失败测试继续修复。
5. 达到最大轮数后返回最佳候选。

这种方法适合 bug fixing、竞赛题、API 使用修复和小型代码生成任务。

## 9.8 训练数据构造

代码推理训练数据可以包括：

1. 需求到代码。
2. 需求到解题思路再到代码。
3. 错误代码到修复代码。
4. 测试失败信息到修复解释。
5. 多候选代码和测试结果。
6. 代码审查和缺陷定位。
7. 性能优化过程。
8. 工具调用轨迹。

高质量数据应该包含任务、环境、依赖、测试、错误反馈和最终修复，而不只是孤立代码片段。

## 9.9 执行反馈作为 Reward

代码任务适合 RL，因为 reward 可以来自执行结果。

常见 reward：

1. 编译通过。
2. 单元测试通过数量。
3. 隐藏测试通过率。
4. 无超时。
5. 无内存超限。
6. 代码风格或安全检查通过。

但 reward 设计要小心。只奖励公开测试通过，模型可能过拟合测试；只奖励短代码，模型可能牺牲可读性；只奖励最终通过，训练信号可能过于稀疏。

更稳健的做法是结合过程反馈、测试覆盖、静态检查和最终结果。

## 9.10 隐藏测试和泛化

公开测试容易被过拟合。隐藏测试用于评估泛化能力。

一个模型可能通过所有公开样例，但在隐藏边界条件上失败。例如：

1. 空输入。
2. 单元素输入。
3. 极大输入。
4. 重复元素。
5. 负数。
6. Unicode 或特殊字符。
7. 并发或状态污染。

训练和评估时，要关注模型是否真的理解规格，而不是只拟合样例。

## 9.11 安全沙箱

执行模型生成的代码有安全风险。必须使用沙箱。

需要限制：

1. 文件系统访问。
2. 网络访问。
3. CPU 时间。
4. 内存。
5. 子进程。
6. 系统调用。
7. 环境变量和密钥。

工程系统中，代码执行服务通常要隔离运行，并设置超时、资源限制和日志审计。不能直接在生产主机上无保护地运行模型生成代码。

## 9.12 多语言和环境问题

代码任务的难点还包括环境一致性。

常见问题：

1. Python 版本不同。
2. 依赖缺失。
3. 编译器版本不同。
4. 操作系统差异。
5. 随机数和时间依赖。
6. 测试顺序依赖。
7. 文件路径差异。

如果环境不稳定，执行反馈会变成噪声。训练和评估系统必须记录语言、版本、依赖和运行配置。

## 9.13 常见失败模式

1. 代码看起来合理，但没有通过测试。
2. 模型误解需求，修错方向。
3. 只修公开测试，泛化失败。
4. 反复修改同一处，无法跳出局部错误。
5. 报错信息很长，模型抓不到关键点。
6. 执行环境不一致导致反馈不稳定。
7. 生成不安全代码。
8. 通过测试但代码不可维护。

代码 reasoning 的关键不是“会写代码”，而是能把需求、反馈、测试和修复策略连接起来。

## 9.14 评估指标

代码推理评估可以看：

1. Pass@1。
2. Pass@k。
3. 修复成功率。
4. 平均迭代轮数。
5. 测试通过率提升。
6. 隐藏测试通过率。
7. 超时率。
8. 安全违规率。
9. 代码可读性和维护性。

Pass@k 衡量多样本生成能力；修复成功率衡量执行反馈利用能力；隐藏测试通过率衡量泛化能力。

## 9.15 最小可运行代码推理 / 执行反馈审计 demo

这个 demo 用 4 个 toy 函数任务模拟代码生成、公开测试筛选、隐藏测试评估、self-debug 修复和静态安全拦截。它只执行正文里硬编码的候选代码，并用简化 AST guard 拦截 `import`、危险调用和无限循环；这只是教学 demo，不是生产级沙箱。

```python
import ast
import math


TASKS = [
    {
        "id": "count_positive",
        "public": [(([1, -2, 3],), 2), (([-1, 5],), 1)],
        "hidden": [(([0, 1, -1],), 1), (([0, 0],), 0)],
        "candidates": [
            "def solve(nums):\n    return sum(1 for x in nums if x >= 0)",
            "def solve(nums):\n    return sum(1 for x in nums if x > 0)",
            "import os\ndef solve(nums):\n    return len(nums)",
        ],
    },
    {
        "id": "left_middle",
        "public": [(("abc",), "b"), (("hello",), "l")],
        "hidden": [(("abcd",), "b"), (("xy",), "x")],
        "candidates": [
            "def solve(s):\n    return s[len(s) // 2]",
            "def solve(s):\n    return s[(len(s) - 1) // 2]",
        ],
    },
    {
        "id": "clamp_values",
        "public": [(([-2, 0, 3], 0, 5), [0, 0, 3]), (((1, 2), 0, 5), [1, 2])],
        "hidden": [(([9, -1, 4], 0, 5), [5, 0, 4])],
        "candidates": [
            "def solve(nums, lo, hi):\n    return [min(max(x, lo), hi) for x in nums]",
            "def solve(nums, lo, hi):\n    return [max(x, lo) for x in nums]",
        ],
    },
    {
        "id": "parse_ints",
        "public": [(("1 2 -3",), [1, 2, -3])],
        "hidden": [(("4, 5 -6",), [4, 5, -6])],
        "candidates": [
            "def solve(text):\n    return [int(x) for x in text.split(',')]",
            "def solve(text):\n    return [int(x) for x in text.replace(',', ' ').split()]",
            "def solve(text):\n    while True:\n        pass",
        ],
    },
]

ALLOWED_BUILTINS = {
    "sum": sum,
    "len": len,
    "int": int,
    "str": str,
    "range": range,
    "list": list,
    "set": set,
    "sorted": sorted,
    "enumerate": enumerate,
    "min": min,
    "max": max,
}
BANNED_CALLS = {"open", "eval", "exec", "compile", "input", "__import__"}


def guard_code(src):
    try:
        tree = ast.parse(src)
    except SyntaxError as exc:
        return False, f"syntax:{exc.__class__.__name__}"
    for node in ast.walk(tree):
        if isinstance(node, (ast.Import, ast.ImportFrom)):
            return False, "blocked_import"
        if isinstance(node, ast.While):
            return False, "blocked_loop"
        if isinstance(node, ast.Attribute) and node.attr.startswith("__"):
            return False, "blocked_dunder"
        if isinstance(node, ast.Name) and node.id.startswith("__"):
            return False, "blocked_dunder"
        if isinstance(node, ast.Call) and isinstance(node.func, ast.Name):
            if node.func.id in BANNED_CALLS:
                return False, f"blocked_call:{node.func.id}"
    return True, "ok"


def load_solve(src):
    allowed, reason = guard_code(src)
    if not allowed:
        return None, reason
    env = {"__builtins__": ALLOWED_BUILTINS}
    local = {}
    exec(compile(src, "<candidate>", "exec"), env, local)
    fn = local.get("solve") or env.get("solve")
    if not callable(fn):
        return None, "missing_solve"
    return fn, "ok"


def run_tests(fn, tests):
    passed = 0
    errors = []
    for args, expected in tests:
        try:
            got = fn(*args)
            ok = got == expected
        except Exception as exc:
            got = f"error:{exc.__class__.__name__}"
            ok = False
        passed += int(ok)
        if not ok:
            errors.append({"args": args, "expected": expected, "got": got})
    return passed, len(tests), errors


def pass_at_k(n, correct, k):
    if correct <= 0:
        return 0.0
    if n - correct < k:
        return 1.0
    return round(1 - math.comb(n - correct, k) / math.comb(n, k), 3)


reports_by_task = {}
blocked = []
selected = []
public_fail_repairs = []
pass1_values = []
pass2_values = []
attempts_to_public = []
all_reports = []

for task in TASKS:
    task_reports = []
    for idx, src in enumerate(task["candidates"]):
        fn, status = load_solve(src)
        report_id = f"{task['id']}:c{idx}"
        if fn is None:
            report = {
                "id": report_id,
                "status": status,
                "public_pass": 0,
                "public_total": len(task["public"]),
                "hidden_pass": 0,
                "hidden_total": len(task["hidden"]),
            }
            blocked.append(report_id)
        else:
            pub_pass, pub_total, pub_errors = run_tests(fn, task["public"])
            hid_pass, hid_total, hid_errors = run_tests(fn, task["hidden"])
            report = {
                "id": report_id,
                "status": "ok",
                "public_pass": pub_pass,
                "public_total": pub_total,
                "hidden_pass": hid_pass,
                "hidden_total": hid_total,
                "first_public_error": pub_errors[:1],
                "first_hidden_error": hid_errors[:1],
            }
        task_reports.append(report)
        all_reports.append(report)

    reports_by_task[task["id"]] = task_reports
    executable = [report for report in task_reports if report["status"] == "ok"]
    hidden_correct = sum(
        report["hidden_pass"] == report["hidden_total"]
        for report in executable
    )
    pass1_values.append(pass_at_k(len(executable), hidden_correct, 1))
    pass2_values.append(pass_at_k(len(executable), hidden_correct, min(2, len(executable))))

    best = max(
        executable,
        key=lambda report: (
            report["public_pass"] / max(1, report["public_total"]),
            -int(report["id"].split(":c")[-1]),
        ),
    )
    selected.append(best)

    first_success = None
    for pos, report in enumerate(task_reports, start=1):
        if report["status"] == "ok" and report["public_pass"] == report["public_total"]:
            first_success = pos
            break
    attempts_to_public.append(first_success or len(task_reports) + 1)

    greedy = task_reports[0]
    if greedy["status"] == "ok" and greedy["public_pass"] < greedy["public_total"]:
        public_fail_repairs.append(
            {
                "task": task["id"],
                "repaired": best["hidden_pass"] == best["hidden_total"],
                "chosen": best["id"],
            }
        )

greedy_accuracy = sum(
    reports_by_task[task["id"]][0]["status"] == "ok"
    and reports_by_task[task["id"]][0]["hidden_pass"] == reports_by_task[task["id"]][0]["hidden_total"]
    for task in TASKS
) / len(TASKS)
selected_public_accuracy = sum(
    report["public_pass"] == report["public_total"]
    for report in selected
) / len(selected)
selected_hidden_accuracy = sum(
    report["hidden_pass"] == report["hidden_total"]
    for report in selected
) / len(selected)
repair_success_rate = (
    sum(item["repaired"] for item in public_fail_repairs) / len(public_fail_repairs)
    if public_fail_repairs else 0.0
)

summary = {
    "greedy_accuracy": round(greedy_accuracy, 3),
    "public_rerank_accuracy": round(selected_hidden_accuracy, 3),
    "selected_public_pass_rate": round(selected_public_accuracy, 3),
    "public_hidden_gap": round(selected_public_accuracy - selected_hidden_accuracy, 3),
    "pass_at_1_est": round(sum(pass1_values) / len(pass1_values), 3),
    "pass_at_2_est": round(sum(pass2_values) / len(pass2_values), 3),
    "repair_success_rate": round(repair_success_rate, 3),
    "avg_attempts_to_public_success": round(sum(attempts_to_public) / len(attempts_to_public), 3),
    "blocked_candidate_rate": round(len(blocked) / len(all_reports), 3),
}
hidden_failures_after_public = [
    report["id"].split(":c")[0]
    for report in selected
    if report["public_pass"] == report["public_total"]
    and report["hidden_pass"] < report["hidden_total"]
]
gates = {
    "public_signal_ok": summary["selected_public_pass_rate"] >= 0.9,
    "hidden_generalization_ok": summary["public_rerank_accuracy"] >= 0.75,
    "gap_ok": summary["public_hidden_gap"] <= 0.25,
    "repair_ok": summary["repair_success_rate"] >= 0.8,
    "sandbox_blocks_unsafe": len(blocked) == 2,
}

print(f"summary={summary}")
print(f"selected={[report['id'] for report in selected]}")
print(f"hidden_failures_after_public={hidden_failures_after_public}")
print(f"public_fail_repairs={public_fail_repairs}")
print(f"blocked={blocked}")
print(f"gates={gates}")
print(f"gate_pass={all(gates.values())}")
```

selected=['count_positive:c0', 'left_middle:c0', 'clamp_values:c0', 'parse_ints:c1']
hidden_failures_after_public=['count_positive', 'left_middle']
public_fail_repairs=[{'task': 'parse_ints', 'repaired': True, 'chosen': 'parse_ints:c1'}]
blocked=['count_positive:c2', 'parse_ints:c2']

## 9.16 面试题：执行反馈为什么重要

回答要点：

```text
执行反馈把代码任务从纯文本生成变成可验证闭环。模型生成代码后，可以通过编译器、解释器、单元测试、类型检查和静态分析得到反馈，再根据反馈定位和修复错误。这种反馈可以作为 verifier、search 剪枝依据和 RL reward，比只看代码文本更可靠。但测试覆盖、环境一致性和安全沙箱也很重要。
```

## 9.17 面试题：如何设计代码 self-debug 系统

回答要点：

```text
我会让系统先生成初始代码，然后在沙箱中运行测试，收集编译错误、运行异常和失败用例。模型根据反馈解释失败原因并生成补丁，再运行测试验证。控制器需要设置最大迭代轮数、超时和资源限制，并保留历史修改，避免反复震荡。最终按隐藏测试、公开测试、静态检查和代码质量选择候选。
```

## 9.19 本章小结

代码推理与执行反馈是 reasoning model 落地能力的重要来源。代码可以运行，测试可以验证，错误信息可以指导修复，这使得代码任务天然适合 tool-use reasoning、self-debug、search、verifier 和 RL。

但代码任务也有工程复杂性：测试覆盖不完整、隐藏用例难、环境差异大、安全沙箱必须可靠、模型可能过拟合公开测试。真正强的代码 reasoning 系统，不只是生成代码，而是能在受控环境中根据反馈持续修正，并在质量、成本、安全和泛化之间取得平衡。下一章会讨论 reasoning 评估，系统梳理如何判断一个模型是否真的具备推理能力。
