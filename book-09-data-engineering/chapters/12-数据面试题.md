# 第十二章：数据面试题

本章是第九册《大模型数据工程与数据智能》的收束章。前面 11 章已经系统讲过 web-scale 采集、清洗过滤、去重污染、data mixture、代码数学领域数据、合成与蒸馏、偏好与安全、多模态数据、data attribution、dataset governance。本章把这些内容组织成面试答题框架。

大模型数据面试和普通数据工程面试不一样。普通数据工程更关注 ETL、存储、调度、数据仓库和实时链路；大模型数据工程还要关注训练效果、模型行为、安全边界、评估污染、数据配比、合规治理和训练可复现。

面试官真正想听的不是“我会用某个工具清洗数据”，而是你是否理解：数据如何塑造模型能力，数据问题如何变成模型问题，数据工程如何和训练、评估、安全、产品形成闭环。

本章高频问题：如何构建预训练数据？如何做去重？如何检测数据污染？如何设计 reasoning 数据？

合规边界：本章只讨论合法合规的数据工程、风险治理和面试表达，不提供绕过访问控制、抓取未授权数据、复原训练数据、规避安全策略或生成有害内容的方法。

## 0. 本讲资料边界与第二轮精修口径

按照 `WRITING_PLAN.md` 的要求，本讲精修前核对了 C4 / T5、The Pile、RefinedWeb、FineWeb、Dolma、DCLM 等公开数据集和数据配方工作，也复核了 Datasheets for Datasets、Model Cards、W3C PROV、OpenLineage、Hugging Face Dataset Cards、NIST AI RMF / Generative AI Profile 和 OWASP LLM Top 10 中与数据污染、数据治理、文档透明度相关的公开资料。

本讲是第九册收束章，不再展开每个算法细节，而是把前 11 章压缩成面试可迁移框架：如何回答、如何量化、如何证明、如何说清边界，以及如何复盘自己的回答。

```text
面试题 -> 目标澄清 -> 数据来源 -> 处理门禁 -> 配比实验 -> 评估证据 -> 治理边界 -> 复盘改进
```

本讲不把任何公开数据集的具体比例写成唯一标准。真实训练数据配方会随模型目标、语言覆盖、授权边界、数据质量、评估集和训练阶段变化；面试中更重要的是讲清可审计的数据工程方法，而不是背某个固定比例。

## 1. 面试总框架：大模型数据题怎么答？

遇到大模型数据题，建议用一个固定框架：目标、来源、处理、配比、评估、治理。

1. 目标：模型要解决什么问题？通用 base model、代码模型、数学模型、行业模型、多模态模型还是安全对齐模型？
2. 来源：数据从哪里来？网页、书籍、代码、数学、领域文档、合成数据、偏好数据、多模态数据。
3. 处理：如何清洗、过滤、去重、脱敏、对齐、质量评分和污染检测？
4. 配比：不同数据源如何混合？哪些上采样，哪些下采样？不同训练阶段如何变化？
5. 评估：如何判断数据策略有效？小模型 ablation、validation loss、benchmark、人工评测、安全评估、线上反馈。
6. 治理：数据版本、lineage、license、权限、删除请求、datasheet、model card。

这个框架适用于绝大多数数据面试题。

### 1.1 关键公式与指标速查

数据面试不要求你现场推复杂公式，但优秀回答应该能把“我会清洗、去重、配比、治理”落到可量化指标。下面这些公式覆盖本章大多数追问。

训练候选集可以抽象为：

```math
D=\bigcup_{m=1}^{M}D_m,\qquad x_i=(u_i,b_i,q_i,r_i,a_i,v_i)
```

其中 `D_m` 是第 `m` 个数据池，`u_i` 是来源或样本 id，`b_i` 是 token 数，`q_i` 是质量分，`r_i` 是风险分或风险标记，`a_i` 是授权和访问状态，`v_i` 是数据版本。

Token 保留率：

```math
R_{\mathrm{keep}}=\frac{\sum_i b_i g_i}{\sum_i b_i}
```

其中 `g_i=1` 表示样本通过准入门禁，`g_i=0` 表示被过滤或隔离。回答清洗题时，只说删掉多少不够，还要解释保留下来的 token 是否覆盖目标语言、领域和能力。

某类风险命中率：

```math
R_{\mathrm{risk}}(k)=\frac{\sum_i b_i h_{ik}}{\sum_i b_i}
```

其中 `h_ik=1` 表示样本命中第 `k` 类风险，例如 PII、secret、license、contamination 或 safety risk。

Near dedup 常用 Jaccard 直觉：

```math
J(A,B)=\frac{|A\cap B|}{|A\cup B|}
```

其中 `A` 和 `B` 是两段文本的 shingle 集合。MinHash / LSH 的面试重点是用签名近似 Jaccard，并减少全量两两比较。

Benchmark contamination 命中率：

```math
R_{\mathrm{contam}}=\frac{N_{\mathrm{train\ eval\ overlap}}}{N_{\mathrm{query}}}
```

这里的 query 不只包括题目原文，还应包括答案、选项、解析、代码签名、测试样例和近重复变体。

Data mixture 对应的训练目标可以写成：

```math
L_{\mathrm{mix}}(\theta)=\sum_{m=1}^{M}\alpha_m\mathbb{E}_{x\sim D_m}[L_{\theta}(x)],\qquad \sum_{m=1}^{M}\alpha_m=1
```

其中 `alpha_m` 是第 `m` 个数据池的采样权重。这个公式帮助你解释：data mixture 不是拼文件，而是在改变模型实际优化的训练分布。

一种可解释的采样权重草案：

```math
p_m=\frac{b_m^{\tau}q_m(1-r_m)}{\sum_j b_j^{\tau}q_j(1-r_j)}
```

其中 `b_m` 是清洗后 token 数，`q_m` 是池级质量分，`r_m` 是风险惩罚，`tau` 控制大池和小池之间的平滑。真实系统还要叠加 license、语言覆盖、目标能力和人工审计结果。

Effective epoch：

```math
e_m=\frac{T p_m}{b_m}
```

其中 `T` 是计划训练 token 总数。如果小数据池 `e_m` 很高，要警惕记忆、过拟合和污染放大。

偏好数据标注一致率和长度偏置：

```math
A_{\mathrm{pair}}=\frac{N_{\mathrm{agree}}}{N_{\mathrm{pair}}},\qquad
B_{\mathrm{len}}=\frac{1}{N}\sum_i(\ell_i^{+}-\ell_i^{-})
```

其中 `ell_i+` 和 `ell_i-` 分别表示 chosen / rejected 的长度。长度偏置不能单独证明数据坏，但它是偏好数据审计必须检查的信号。

安全数据的漏拒率和误拒率：

```math
R_{\mathrm{leak}}=\frac{N_{\mathrm{unsafe\ accepted}}}{N_{\mathrm{high\ risk}}},\qquad
R_{\mathrm{over}}=\frac{N_{\mathrm{safe\ refused}}}{N_{\mathrm{safe\ allowed}}}
```

高质量安全模型不是拒答率最高，而是在漏拒和误拒之间保持稳定边界。

多模态对齐分的基本形式：

```math
s_{\mathrm{align}}=\frac{u_{\mathrm{image}}\cdot u_{\mathrm{text}}}{\lVert u_{\mathrm{image}}\rVert_2\lVert u_{\mathrm{text}}\rVert_2}
```

这个分数只能辅助判断图文相关性，不能替代 OCR、时序、grounded QA、隐私和版权审计。

数据源价值的面试级表达：

```math
V_s=\Delta U_s-\lambda_r R_s-\lambda_c C_s
```

其中 `Delta U_s` 是目标效用提升，`R_s` 是风险，`C_s` 是清洗、标注、存储和治理成本。数据价值不是绝对属性，而是相对于目标、阶段和约束的边际贡献。

治理门禁：

```math
G_{\mathrm{data}}=G_{\mathrm{license}}\land G_{\mathrm{pii}}\land G_{\mathrm{contam}}\land G_{\mathrm{lineage}}\land G_{\mathrm{delete}}\land G_{\mathrm{doc}}
```

如果没有 license、PII、污染、lineage、删除请求和文档门禁，就不能声称这个数据版本已经可用于正式训练或发布说明。

最后，面试回答本身也可以量化复盘：

```math
S_{\mathrm{ans}}=w_c C+w_e E+w_m M+w_g G+w_b B-P
```

其中 `C` 是框架覆盖，`E` 是工程证据，`M` 是指标和公式，`G` 是治理边界，`B` 是 trade-off 深度，`P` 是错误表述和超时惩罚。准备面试时，不要只问“我记住了吗”，要问“我的回答有没有目标、证据、指标、边界和取舍”。

## 2. 高频题一：如何构建大模型预训练数据集？

### 标准回答框架

我会把预训练数据建设看成一个端到端数据系统，而不是简单拼接文本。

第一步，明确模型目标和覆盖范围。比如是通用中文/英文 base model，还是偏代码、数学、多语言或行业模型。目标不同，数据来源和配比不同。

第二步，设计数据源。通用文本包括网页、书籍、百科、新闻、论坛、问答和技术文档；专项数据包括代码、数学、论文、领域文档；后续还要考虑合成数据、偏好数据和安全数据，但预训练阶段要控制比例。

第三步，建立采集合规流程。记录来源、license、时间、robots/ToS、授权状态和风险等级。不采集未授权、访问受限或高隐私风险数据。

第四步，做解析和清洗。HTML、PDF、代码、文档、多模态数据分别使用不同 parser；去乱码、模板、广告、低质量文本、PII 和高风险内容。

第五步，做语言识别、领域分类和质量评分。不要对所有数据用同一阈值，代码、数学、多语言和专业领域要单独规则。

第六步，做去重和污染检测。包括 exact dedup、near dedup、段落级重复、代码 fork、benchmark contamination。

第七步，设计 data mixture。按目标能力设置采样权重，控制通用文本、代码、数学、多语言、领域和合成数据比例。

第八步，做评估闭环。用小模型训练、数据 ablation、validation loss、下游 benchmark、安全评估和人工样例验证数据策略。

第九步，做版本治理。保存 manifest、lineage、schema、过滤规则、数据版本、删除请求和审计记录。

### 面试加分点

不要只说“爬网页 + 清洗 + 去重”。要强调数据目标、合规、配比、污染检测和版本可复现。

## 3. 高频题二：如何做 web-scale 数据采集？

### 标准回答框架

Web-scale 采集要先确定合法来源和目标语料范围。采集不是越多越好，而是要保证来源合规、可追溯、可清洗和可评估。

我会先做 source planning，把来源分为通用网页、百科、新闻、技术文档、论坛、问答、论文、代码和多语言来源，并记录来源元数据。

然后建立采集 pipeline：source registry、采集任务、原始数据存储、解析、正文抽取、语言识别、质量过滤、去重、版本化。

合规上，要记录来源 URL、采集时间、license、使用限制和删除机制；对于用户数据、登录后内容、访问受限内容和隐私风险高的数据，不能随意进入训练集。

最后用统计和抽样审计监控来源质量，包括有效文本比例、语言分布、重复率、PII 命中率、低质量比例和下游训练收益。

### 常见追问

面试官可能问：“如果某个来源规模很大但质量一般，怎么办？”

回答：先单独建池，不直接混入主训练集；做质量评分、去重、抽样审计和小模型 ablation；如果有价值则低权重采样或只保留高质量子集。

## 4. 高频题三：如何做数据清洗和质量评分？

### 标准回答框架

清洗过滤分层做。

第一层是基础有效性过滤：空文本、乱码、异常字符、极端长度、重复行、编码错误。

第二层是正文抽取和模板去除：去导航栏、页脚、广告、cookie 提示、推荐链接和 boilerplate。

第三层是语言和领域识别：按语言、领域、来源类型分桶，不同桶使用不同规则。

第四层是规则质量过滤：长度、符号比例、链接比例、重复 n-gram、低信息密度等。

第五层是模型质量评分：训练质量分类器或用 LLM judge 辅助，对可读性、信息密度、结构、来源可信度和安全风险打分。

第六层是安全和隐私过滤：PII、毒性、高风险内容、版权和敏感数据处理。

最后要做人工抽样审计和小模型训练验证，评估误删、漏删和下游收益。

### 面试加分点

强调清洗不是越狠越好。过度过滤会损失口语、多语言、低资源语言、专业符号、代码和数学数据。

## 5. 高频题四：如何做去重？

### 标准回答框架

去重要按粒度分层。

第一，文档级 exact dedup。对规范化文本做 hash，相同 hash 只保留一份。

第二，段落级重复统计。识别网页模板、免责声明、页脚、广告段落和高频重复片段。

第三，near dedup。用 shingling、MinHash、SimHash、LSH 或 embedding 检索识别转载、改标题、加页脚、轻微改写的重复。

第四，代码专项去重。处理 fork、vendor、自动生成文件、函数级重复、题库答案和 benchmark 泄漏。

第五，去重后重新统计 data mixture，因为去重会改变来源、语言和领域分布。

### 常见追问

面试官问：“去重是不是越彻底越好？”

回答：不是。目标是控制不合理重复，而不是删除所有相似内容。常见概念、API 用法、数学标准题型和用户意图多样表达都需要一定 exposure。过度去重会损害泛化和常见模式学习。

## 6. 高频题五：如何检测 benchmark contamination？

### 标准回答框架

污染检测要区分训练集内部重复和 train-eval overlap。benchmark contamination 重点是训练集与评测集重叠。

我会把评测集题目、答案、选项、解析、代码签名、测试样例都作为 query，而不只查题目原文。

检测方法包括 exact match、n-gram overlap、MinHash/LSH、embedding 检索、代码函数签名匹配、答案泄漏检测和人工审计。

对于高相似候选，要判断污染程度：只有题目、题目加答案、完整解析、改写版本、同源题库变体，风险等级不同。

处理方式包括从训练集中移除、隔离、降低权重、重新划分评测集或在评测报告中标注风险。

### 面试加分点

强调污染检测是评估可信度问题，不只是数据清洗问题。公开 benchmark 越流行，污染风险越高。

## 7. 高频题六：如何设计 data mixture？

### 标准回答框架

Data mixture 是训练目标的一部分。模型最终优化的是 mixture 分布上的 loss，所以配比会影响能力、风格和安全表现。

我会先定义模型目标，然后把数据按来源、语言、领域、能力、质量和风险打标签。通用文本提供世界知识和语言能力；代码增强结构化和程序能力；数学增强推理；多语言增强语言覆盖；领域数据增强行业能力；合成数据补长尾和格式。

初始配比可以参考经验，但必须通过小规模 ablation 验证。实验时要看多维评估：通用能力、代码、数学、多语言、安全、事实性、风格和人工样例。

配比不是固定不变的。预训练、继续预训练、SFT、DPO、安全训练的 mixture 应该不同。

### 常见追问

面试官问：“代码数据比例越高越好吗？”

回答：不一定。代码比例高通常提升代码和结构化能力，但会占用 token budget，可能影响自然语言、多语言和对话风格，还会增加 license、secrets、重复和题库污染风险。

## 8. 高频题七：如何建设代码数据？

### 标准回答框架

代码数据不能只收 `.py`、`.java` 源文件，还应包括 README、文档、issue、PR、commit message、测试、API reference 和编程问答。

清洗上要做语言识别、语法解析、license 检查、secrets 过滤、fork 去重、vendor 处理、自动生成文件过滤、benchmark contamination 检测。

结构化上可以按仓库、文件、函数、类、AST、测试粒度组织样本。训练样本可以构造 docstring-code、test-code、bug-fix、code-explanation、API usage。

评估上不能只看文本相似度，要看功能正确性、编译运行、单测通过率、安全和依赖版本。

### 面试加分点

强调单测价值：测试提供可执行约束，能训练和评估模型是否真正写出能运行的代码。

## 9. 高频题八：如何设计数学和 reasoning 数据？

### 标准回答框架

数学数据要按题目、答案、步骤、难度、领域和验证方式组织。来源包括教材、题库、竞赛题、课程、论坛、合成题、人工 step-by-step 解答和 verifier 数据。

清洗重点是题解对齐、公式解析、答案校验、步骤质量、重复题检测和 benchmark 污染。

Reasoning 数据不能只要最终答案。高质量数据应包含过程、约束、检查和错误候选。对于代码和数学，可以用测试、计算器、CAS 或 verifier 辅助验证。

配比上要谨慎，过多低质量 reasoning 数据会让模型学会伪推理和模板化解释。

### 常见追问

面试官问：“CoT 数据越多越好吗？”

回答：不是。CoT 质量比数量重要。错误过程、跳步过程和模板化过程会伤害模型。要用答案校验、过程审计、难度分桶和下游评估控制质量。

## 10. 高频题九：如何建设 synthetic data？

### 标准回答框架

Synthetic data 要先定义目标能力，而不是为了生成而生成。常见目标包括指令跟随、数学、代码、工具调用、多语言、安全和领域问答。

Pipeline 包括：种子设计、生成策略、teacher 选择、多样性控制、格式检查、去重、质量评分、正确性验证、安全过滤、配比实验和版本记录。

不同任务验证方式不同：数学可校验答案，代码可跑测试，事实问答要检索证据，领域问答要专家审计，安全数据要检查边界。

合成数据风险包括 teacher 偏差、错误放大、同质化、数据退化、污染和过拟合。不能无追踪地大规模混入。

### 面试加分点

说清 synthetic data 和 distillation data 区别：synthetic 强调生成来源，distillation 强调 teacher-student 能力迁移。

## 11. 高频题十：如何建设 preference data？

### 标准回答框架

Preference data 常见形式是 prompt、chosen、rejected，也可以是多个回答排序和标注理由。

构造方式包括同模型多采样、不同模型对比、人工写 chosen、模型生成 rejected、LLM judge 辅助和专家标注。

标注规范要明确 helpful、honest、harmless 的优先级，以及事实性、完整性、简洁性、语气、专业边界和安全性。

数据质量要检查标注一致性、长度偏置、拒答偏置、迎合偏置、rejected 是否太弱、多语言一致性和专业标注质量。

Preference data 可用于 reward model、RLHF、DPO 等方法。算法不同，但都依赖偏好数据质量。

### 常见追问

面试官问：“DPO 数据随便构造 chosen/rejected 可以吗？”

回答：不可以。噪声大、rejected 太弱或标注标准不一致，会让模型学到错误偏好。DPO 简化优化，不降低对数据质量的要求。

---

## 12. 高频题十一：如何建设安全数据？

### 标准回答框架

安全数据不只是拒答数据。它包括风险分类、拒答、边界允许、误拒修复、漏拒修复、安全替代建议、红队回归、多语言安全和专业高风险场景。

我会按风险分层：明确安全、敏感但可答、高风险、边界模糊、紧急风险、专业高风险。

拒答数据要训练模型不提供危险细节，同时提供安全替代方向。边界允许数据要训练模型在教育、防御、科普、治理场景中安全帮助，避免看到关键词就拒绝。

评估上同时看误拒和漏拒。好的安全模型不是拒答最多，而是边界最稳定。

### 合规提醒

回答红队数据时，强调用于发现失败模式、防御训练和回归测试，不传播可操作风险细节。

---

## 13. 高频题十二：如何建设多模态数据？

### 标准回答框架

多模态数据工程的核心是对齐，而不是拼接。

图文数据要处理 image-caption、alt text、周围文本、OCR、图文相似度、图片质量、版权和隐私。

语音数据要处理音频质量、VAD、speaker diarization、ASR 转录、forced alignment、采样率、口音、噪声和声纹隐私。

视频数据要处理切片、关键帧、字幕、音频、OCR、镜头变化、事件对齐和时间戳。

多模态指令数据要保证回答 grounded in modality，不能只凭语言先验猜测。

### 常见追问

面试官问：“图文相似度高就说明数据好吗？”

回答：不一定。相似度只是信号。新闻图、商品图、图表、文档截图和 OCR 场景都需要结合结构和人工审计。

---

## 14. 高频题十三：如何评估数据价值？

### 标准回答框架

数据价值不是固定属性，而是相对于目标模型、训练阶段、已有数据分布和评估指标的边际贡献。

我会从源级、簇级和样本级做估值。预训练更适合源级和簇级；SFT、偏好、安全数据可以做样本级分析。

方法包括数据消融、上采样/下采样实验、小模型 proxy、validation loss、embedding 相似度、gradient similarity、influence 思路、Data Shapley 近似和人工审计。

最终要输出多维价值：目标能力收益、通用能力副作用、安全风险、隐私风险、版权成本、清洗成本和治理成本。

### 面试加分点

不要说“直接用 Shapley 算”。大模型全量 Shapley 不现实，应该讲近似和工程证据链。

---

## 15. 高频题十四：如何做 dataset versioning 和 governance？

### 标准回答框架

大模型训练数据必须有不可变快照、manifest、lineage、schema 和 pipeline version。

每条样本应记录 source、license、language、domain、quality_score、safety_label、pii_label、dedup_cluster、contamination_flag、dataset_version、deletion_status。

Manifest 记录 shard、hash、token 数、样本数、来源分布、语言分布、质量统计和风险摘要。

Governance 包括权限控制、访问日志、删除请求、license 审计、PII 处理、benchmark 保护、datasheet 和 model card 支撑。

训练日志要记录数据版本、mixture、sampler、tokenizer、评测集版本和 pipeline 版本，否则模型不可复现。

### 常见追问

面试官问：“删除请求为什么难？”

回答：因为数据可能存在于原始数据、清洗数据、shard、缓存、索引、embedding、旧版本和已训练模型中，需要 lineage、hash、source_id 和版本记录支持。

---

## 16. 系统设计题：给你 100TB 原始 web 数据，如何做成训练集？

### 推荐回答

我会按“数据湖 + 处理 pipeline + 质量评估 + 版本治理”的方式设计。

第一层是 raw data lake，保存原始数据和元信息，记录来源、时间、授权和 hash。

第二层是解析层，对 HTML、PDF、代码、多模态分别解析，抽取正文、结构、OCR、语言和元数据。

第三层是清洗过滤层，做基础有效性过滤、模板去除、质量评分、PII、安全过滤和领域分类。

第四层是去重污染层，做 exact dedup、near dedup、段落重复、代码 fork、benchmark contamination。

第五层是数据分桶层，按语言、领域、来源、质量、安全和模态分桶，形成可采样数据池。

第六层是 mixture sampler，根据训练目标设置采样权重。

第七层是评估闭环，小模型 ablation、validation loss、benchmark、安全评估和人工审计。

第八层是治理，manifest、lineage、schema、license、权限、删除请求和 datasheet。

### 深挖回答

如果训练结果异常，我会先看数据版本和 mixture 是否变化，再看对应评估集相关数据源、污染检测、质量分布和小规模 ablation，而不是直接改模型。

---

## 17. 系统设计题：如何构建一个代码大模型的数据体系？

### 推荐回答

代码模型数据要分成源码、文档、测试、issue/PR、问答、教程和合成任务几个池。

采集时记录 license、仓库元数据、语言、star、更新时间、依赖和路径结构。

清洗时过滤二进制、自动生成文件、vendor、fork、secrets、过时或低质量项目。

结构化时按 repo-file-function-class-test 组织，构造 docstring-code、test-code、bug-fix、explain-code、API usage 样本。

污染检测要重点查 HumanEval、MBPP、竞赛题、题解、测试样例和参考答案。

评估要看功能正确性、单测通过率、代码安全、API 时效、多语言编程语言覆盖和自然语言能力副作用。

---

## 18. 系统设计题：如何构建一个医疗领域模型的数据体系？

### 推荐回答

医疗领域数据要优先考虑安全、权威性、时效性和隐私。

数据来源包括医学教材、指南、药品说明、综述、公开医学知识库、经授权的专业问答和专家标注数据。真实病历必须严格授权、脱敏和审计。

数据要记录来源、发布日期、适用地区、证据等级、版本和过期时间。清洗时处理 PII、错误 OCR、过期指南、低质量论坛和不可靠建议。

训练策略上，继续预训练用于学习医学术语和表达；SFT 用于学习问答格式和安全边界；RAG 用于最新指南和可引用知识；安全数据用于急症提示、免责声明和避免越权诊断。

评估上要有专家审计、事实性、引用正确性、拒答边界、风险提示和多语言能力。

---

## 19. 行为面试题：数据策略和模型效果冲突怎么办？

### 推荐回答

我会先明确冲突指标。比如某个数据策略提升数学 benchmark，但降低通用问答或增加安全误拒。

然后检查是否存在污染、过拟合或评估偏差。接着做分桶分析，看影响来自哪个来源、语言、领域或质量区间。

再通过小模型 ablation、采样权重调整和人工样例评估确认因果方向。

最后给出取舍方案：调整 mixture、隔离数据池、提高质量阈值、补边界数据、改变训练阶段或把部分能力交给 RAG/工具处理。

核心是不要只追单一 benchmark，而要看多目标 trade-off。

---

## 20. 专家追问：为什么数据质量比数据规模更重要？

严格说不是“质量一定比规模重要”，而是二者共同决定有效训练信号。

规模提供覆盖和多样性，质量提供有效信息密度和可靠监督。低质量大规模数据会浪费 token budget、引入噪声、增加记忆和污染风险；高质量但过窄的数据会导致覆盖不足和风格单一。

所以更准确的回答是：在固定 compute budget 下，我们关心的是有效 token，而不是原始 token。有效 token 由质量、去重、多样性、配比和目标相关性共同决定。

---

## 21. 专家追问：为什么数据问题会表现成模型问题？

因为模型是在训练分布上学习统计规律。数据中的重复会表现为记忆，低质量网页会表现为幻觉和差风格，污染会表现为 benchmark 虚高，偏好数据噪声会表现为对齐不稳定，安全数据过度保守会表现为误拒，多语言过滤误伤会表现为语言能力差。

因此，排查模型问题时不能只看架构、优化器和 prompt，也要看数据来源、版本、清洗、配比和后训练数据。

---

## 22. 专家追问：小模型数据实验能外推到大模型吗？

不能保证，但有参考价值。

小模型实验适合快速筛选明显好坏的数据策略，比如去重、清洗阈值、数据源质量和合成数据版本。但大模型可能出现不同 scaling behavior，某些数据在小模型上无效，在大模型上有效，反之亦然。

所以成熟做法是多尺度验证：小模型快速探索，中模型确认趋势，大模型谨慎放大，并持续监控训练曲线和下游评估。

---

## 23. 专家追问：如何避免数据工程过度优化 benchmark？

第一，做污染检测，确保 benchmark 原题、答案和解析不进入训练。

第二，评估集多样化，不只看单个公开 benchmark。

第三，保留人工评测、产品样例、holdout set 和红队评估。

第四，数据选择不要只按与 benchmark 相似度，而要保留多样性和真实分布。

第五，评估报告中记录数据版本、污染检测方法和残余风险。

---

## 24. 专家追问：数据治理会不会拖慢训练迭代？

短期看，治理需要额外成本；长期看，它会加快迭代。

没有治理时，模型变好或变坏都无法解释，删除请求无法处理，实验无法复现，安全事件无法追溯。团队会在问题出现后花更多时间补救。

好的治理不是繁文缛节，而是把 manifest、lineage、schema、权限、审计和版本记录自动化，让训练迭代可追踪、可回滚、可比较。

---

## 25. 开放题：如果让你从零搭建数据团队，你怎么分工？

可以分为：

1. 数据采集与平台：source registry、数据湖、调度、存储、权限。
2. 数据清洗与质量：解析、过滤、质量评分、去重、污染检测。
3. 数据策略与实验：data mixture、ablation、数据估值、训练反馈。
4. 专项数据：代码、数学、领域、多模态、合成数据。
5. 后训练数据：SFT、preference、安全、红队回归。
6. 治理与合规：license、PII、删除请求、datasheet、model card。
7. 标注与评估：标注规范、专家审计、人工评测、线上反馈。

这个回答体现你理解数据工程不是单一 ETL，而是跨训练、评估、安全和治理的系统。

---

## 26. 面试中的常见错误表达

错误一：我们把网上数据爬下来清洗一下就能训。

问题：忽略合规、质量、去重、污染、配比和治理。

错误二：数据越多越好。

问题：忽略有效 token、重复、噪声、风险和 compute budget。

错误三：去重就是 hash 去重。

问题：忽略 near dedup、段落重复、代码 fork 和 benchmark contamination。

错误四：安全数据就是拒答数据。

问题：忽略边界允许、误拒修复、多语言和专业高风险场景。

错误五：合成数据可以无限扩。

问题：忽略 teacher 偏差、同质化、数据退化和验证。

错误六：数据治理是上线后的事。

问题：没有 lineage 和版本，训练阶段就已经不可复现。

---

## 27. 最后一套万能回答模板

如果你遇到没准备过的数据题，可以用这个模板：

第一，我会先明确模型目标和评估指标，因为数据质量和价值都是目标相关的。

第二，我会把数据按来源、语言、领域、模态、质量和风险分池，而不是混成一个大文件。

第三，我会建立清洗过滤流程，包括解析、正文抽取、质量评分、PII、安全过滤、去重和污染检测。

第四，我会设计 data mixture 和采样权重，通过小模型 ablation 和多维评估验证。

第五，我会关注专项数据，比如代码、数学、领域、多模态、合成、偏好和安全数据，它们要单独治理。

第六，我会建立版本和治理体系，包括 manifest、lineage、license、权限、删除请求和 datasheet。

第七，我会用训练结果、评估结果、人工审计和线上反馈反向迭代数据策略。

这套模板足以覆盖大多数大模型数据工程面试题。

### 27.1 最小可运行数据面试复盘 demo

下面这个 demo 不依赖外部库，也不读写文件。输入是一次 toy mock interview 的 4 道数据工程题回答记录；输出包括每题得分、平均分、指标覆盖率、缺失框架、缺失风险和下一步修正计划。

它演示的不是自动评分面试官，而是把复盘从“感觉没答好”变成可执行清单。真实面试准备时，可以把 `covered`、`metrics`、`evidence`、`risks`、`tradeoffs` 换成你自己的录音复盘结果。

```python
from collections import Counter


FRAMEWORK = {"target", "source", "processing", "mixture", "evaluation", "governance"}
RISK_KEYS = {"license", "pii", "contamination", "safety", "deletion"}
WEIGHTS = {"framework": 0.30, "metrics": 0.18, "evidence": 0.18, "risk": 0.16, "depth": 0.18}

answers = [
    {
        "id": "q_pretrain_dataset",
        "topic": "pretraining data",
        "covered": {"target", "source", "processing", "mixture", "evaluation", "governance"},
        "metrics": {"token_retention", "risk_hit_rate", "contamination_rate", "lineage_coverage"},
        "evidence": {"small_model_ablation", "manual_audit", "version_manifest"},
        "risks": {"license", "pii", "contamination", "deletion"},
        "tradeoffs": {"quality_vs_coverage", "dedup_vs_diversity"},
        "red_flags": set(),
        "time_sec": 165,
    },
    {
        "id": "q_near_dedup",
        "topic": "dedup and contamination",
        "covered": {"processing", "evaluation"},
        "metrics": {"jaccard", "duplicate_rate"},
        "evidence": {"toy_minhash_demo"},
        "risks": {"contamination"},
        "tradeoffs": {"dedup_vs_diversity"},
        "red_flags": {"said_dedup_more_is_always_better"},
        "time_sec": 150,
    },
    {
        "id": "q_preference_safety",
        "topic": "preference and safety data",
        "covered": {"target", "source", "processing", "evaluation", "governance"},
        "metrics": {"annotator_agreement", "length_bias", "unsafe_leak_rate", "over_refusal_rate"},
        "evidence": {"rubric", "red_team_regression"},
        "risks": {"pii", "safety", "contamination"},
        "tradeoffs": {"helpfulness_vs_safety", "judge_cost_vs_human_review"},
        "red_flags": set(),
        "time_sec": 178,
    },
    {
        "id": "q_governance",
        "topic": "dataset versioning",
        "covered": {"source", "processing", "mixture", "governance"},
        "metrics": {"lineage_coverage", "license_completion", "datasheet_completion"},
        "evidence": {"manifest"},
        "risks": {"license", "deletion"},
        "tradeoffs": {"iteration_speed_vs_traceability"},
        "red_flags": set(),
        "time_sec": 205,
    },
]


def score_answer(item):
    framework_score = len(item["covered"] & FRAMEWORK) / len(FRAMEWORK)
    metric_score = min(len(item["metrics"]) / 3, 1.0)
    evidence_score = min(len(item["evidence"]) / 2, 1.0)
    risk_score = len(item["risks"] & RISK_KEYS) / len(RISK_KEYS)
    depth_score = min(len(item["tradeoffs"]) / 2, 1.0)
    penalty = 0.10 * len(item["red_flags"])
    if item["time_sec"] > 180:
        penalty += 0.05
    total = (
        WEIGHTS["framework"] * framework_score
        + WEIGHTS["metrics"] * metric_score
        + WEIGHTS["evidence"] * evidence_score
        + WEIGHTS["risk"] * risk_score
        + WEIGHTS["depth"] * depth_score
        - penalty
    )
    return round(max(total, 0.0), 3)


scores = {item["id"]: score_answer(item) for item in answers}
missing_framework = Counter()
missing_risks = Counter()
revision_plan = []

for item in answers:
    missing = sorted(FRAMEWORK - item["covered"])
    for name in missing:
        missing_framework[name] += 1
    for risk in sorted(RISK_KEYS - item["risks"]):
        missing_risks[risk] += 1
    if scores[item["id"]] < 0.72 or item["red_flags"]:
        revision_plan.append(
            {
                "id": item["id"],
                "add_framework": missing[:3],
                "add_metrics": max(0, 3 - len(item["metrics"])),
                "fix_red_flags": sorted(item["red_flags"]),
            }
        )

average_score = round(sum(scores.values()) / len(scores), 3)
metric_coverage = round(sum(min(len(item["metrics"]), 3) for item in answers) / (3 * len(answers)), 3)
readiness_gate = average_score >= 0.78 and not revision_plan

report = {
    "scores": scores,
    "average_score": average_score,
    "metric_coverage": metric_coverage,
    "missing_framework": dict(sorted(missing_framework.items())),
    "missing_risks": dict(sorted(missing_risks.items())),
    "revision_plan": revision_plan,
    "readiness_gate": readiness_gate,
}

for key, value in report.items():
    print(f"{key}=", value)

assert scores == {
    "q_pretrain_dataset": 0.968,
    "q_near_dedup": 0.332,
    "q_preference_safety": 0.886,
    "q_governance": 0.574,
}
assert average_score == 0.69
assert metric_coverage == 0.917
assert readiness_gate is False
```

运行后会看到类似输出：

```text

average_score= 0.69
metric_coverage= 0.917
missing_framework= {'evaluation': 1, 'governance': 1, 'mixture': 2, 'source': 1, 'target': 2}
```

这里最重要的不是分数，而是 `revision_plan`。它指出 near dedup 题答得太局部，缺少 source、mixture、governance，还出现了“去重越多越好”的错误表述；governance 题虽然讲了 manifest 和 lineage，但缺少目标澄清、评估证据，并且超时。下一轮复盘应该优先修这两类问题。

## 28. 本章小结

大模型数据面试的核心不是背工具，而是建立系统视角。

你要让面试官相信你理解：

1. 数据决定模型能力和行为。
2. 数据工程必须和训练、评估、安全、产品闭环。
3. 预训练、SFT、DPO、安全、多模态和 RAG 需要不同数据策略。
4. 去重、污染、配比、合成、偏好和治理都是一体化问题。
5. 好的数据系统不仅能训练模型，还能解释、复现、审计和迭代模型。

如果只能记住一句话：大模型数据工程不是“收集更多数据”，而是在目标、质量、覆盖、风险、配比、评估和治理之间做系统性优化。
