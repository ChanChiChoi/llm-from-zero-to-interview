# 第一章：数据总览

重点：数据是大模型能力来源，数据规模、质量、覆盖、配比和治理共同决定训练上限。

面试重点：不要把大模型训练理解成“有足够 GPU 就行”。模型能力、行为、安全、评估可信度和上线风险，都强烈依赖数据工程。

## 本章目标

学完本章，你要能回答：

1. 为什么说数据决定大模型能力上限？
2. 数据规模、数据质量、数据覆盖、数据配比分别影响什么？
3. 从 GPT-3 到 Chinchilla，再到高质量小模型路线，数据观念发生了什么变化？
4. 预训练数据、SFT 数据、偏好数据、评估数据、RAG 数据有什么区别？
5. 大模型数据工程的完整 pipeline 是什么？
6. 数据问题如何表现为训练问题、评估问题、安全问题和产品问题？
7. 面试中如何系统回答“大模型数据工程怎么做”？

## 1. 来龙去脉：大模型为什么越来越重视数据

### 1.1 早期深度学习：数据是燃料

在传统深度学习时代，大家就知道数据重要。

ImageNet 让视觉模型发生质变。

大规模语料让词向量、语言模型和机器翻译显著进步。

但那个阶段，很多讨论仍集中在：

1. 模型结构。
2. 损失函数。
3. 优化器。
4. 正则化。
5. 训练技巧。

数据常被看成“给模型喂的材料”，但还没有形成今天这种系统工程视角。

### 1.2 GPT-3 时代：规模和 web-scale 数据

GPT-3 证明了大规模自回归语言模型在海量文本上预训练后，可以获得强 few-shot 能力。

这个阶段的重要启发是：

```text
模型规模 + 数据规模 + 计算规模，可以带来涌现式能力提升。
```

因此，web-scale 数据采集、清洗和训练变成核心能力。

但 GPT-3 时代也暴露了问题：

1. 数据来源混杂。
2. 质量差异巨大。
3. 训练集中可能有偏见、有害内容和隐私信息。
4. benchmark contamination 难控制。
5. 数据配比不透明。
6. 低质量数据会消耗大量训练 compute。

### 1.3 Chinchilla 时代：模型不是越大越好，token 也要跟上

Chinchilla 论文系统研究了给定 compute budget 下模型参数量和训练 token 数的关系。

它的重要启发是：很多早期大模型相对“数据不足”，即模型很大，但训练 token 不够。

更 compute-optimal 的做法是让模型大小和训练 token 数一起增长。

这改变了很多团队的训练观念：

```text
不是只堆参数，数据 token 数和质量同样关键。
```

### 1.4 高质量数据时代：Textbooks Are All You Need 的启发

phi-1 相关工作展示了另一条思路：在特定任务上，高质量、教材式、结构化、合成增强的数据，可能让较小模型获得很强能力。

这并不意味着“只要教材就够了”。

更准确的启发是：

```text
数据质量、任务匹配和数据组织方式，可以显著影响模型效率和能力形成。
```

### 1.5 今天的数据观念

今天的大模型数据工程不再只是“爬更多网页”。

而是综合考虑：

1. 数据规模。
2. 数据质量。
3. 数据覆盖。
4. 数据配比。
5. 去重和污染控制。
6. 合成数据。
7. 偏好数据。
8. 安全数据。
9. 多模态数据。
10. 数据版本和治理。

这也是第九册要解决的问题。

## 2. 小白例子：训练模型像培养学生

把模型想象成一个学生。

训练数据就是他的教材、作业、考试和老师反馈。

如果教材很多但质量差，学生会学到错误知识。

如果教材只覆盖一个领域，学生会偏科。

如果作业里都是简单题，学生不会难题。

如果考试题提前泄露，分数不可信。

如果老师反馈有偏，学生会学会讨好老师而不是真懂。

对应到大模型：

1. 预训练数据像教材。
2. SFT 数据像示范答案。
3. 偏好数据像老师打分。
4. 评估数据像考试。
5. RAG 数据像开卷资料。
6. 安全数据像行为规范训练。

所以数据工程不是后勤工作，而是模型能力和行为塑造的核心。

## 3. 大模型数据的五个核心维度

### 3.1 规模

规模指 token 数、样本数、文档数和模态数据量。

数据规模影响：

1. 语言覆盖。
2. 世界知识覆盖。
3. 长尾概念学习。
4. 模型泛化。
5. Scaling law。

但规模不是越大越好。

如果数据质量差，大规模只会放大噪声。

### 3.2 质量

质量包括：

1. 正确性。
2. 可读性。
3. 信息密度。
4. 结构清晰度。
5. 领域可靠性。
6. 有害内容比例。
7. 重复程度。

高质量数据能提高训练效率。

低质量数据会导致：

1. 幻觉。
2. 格式混乱。
3. 有害行为。
4. 错误知识。
5. 训练 compute 浪费。

### 3.3 覆盖

覆盖指数据是否覆盖目标能力和目标用户场景。

例如：

1. 多语言。
2. 代码。
3. 数学。
4. 科学。
5. 法律。
6. 医疗。
7. 对话。
8. 工具调用。
9. 多模态。

覆盖不足会导致模型偏科。

### 3.4 配比

配比指不同数据类型在训练中的比例。

例如：

1. Web 文本多少。
2. 代码多少。
3. 数学多少。
4. 中文多少。
5. 高质量合成数据多少。
6. 安全数据多少。

配比影响模型能力结构。

代码比例高，模型代码能力可能更强，但可能影响自然语言分布。

安全数据比例过高，可能 over-refusal。

### 3.5 治理

治理指数据是否可追溯、可审计、可删除、可复现、可合规。

包括：

1. 数据来源。
2. 许可证。
3. PII 处理。
4. 去重策略。
5. 数据版本。
6. 评估集隔离。
7. 删除请求。
8. 安全审计。

没有治理，模型越大风险越大。

## 4. 大模型数据类型全景图

### 4.1 预训练数据

目标：学习语言、知识、代码、世界模式和基础能力。

来源可能包括：

1. Web 文本。
2. 书籍。
3. 论文。
4. 代码。
5. 问答。
6. 百科。
7. 多语言语料。

关键问题：

1. 数据质量。
2. 去重。
3. PII。
4. 版权。
5. 毒性和偏见。
6. 评估污染。

### 4.2 SFT 数据

目标：教模型按照指令回答。

特点：

1. 数量比预训练小得多。
2. 质量要求高。
3. 格式和 chat template 很重要。
4. 覆盖任务类型和回答风格。

SFT 数据影响模型“像不像助手”。

### 4.3 偏好数据

目标：让模型学会人类偏好。

形式：

```text
prompt + chosen answer + rejected answer
```

用于 RLHF、DPO、RLAIF 等。

关键问题：

1. 标注偏差。
2. 长度偏差。
3. Safety/helpfulness trade-off。
4. Reward hacking。
5. 多轮偏好。

### 4.4 安全数据

目标：让模型理解风险边界。

包括：

1. 应该拒绝的请求。
2. 正常安全教育请求。
3. 边界请求。
4. 多轮诱导。
5. Prompt injection。
6. 隐私保护。
7. 安全替代回答。

安全数据质量直接影响 over-refusal 和 harmful compliance。

### 4.5 评估数据

目标：衡量模型能力和风险。

关键要求：

1. 不能污染训练。
2. 分层清晰。
3. 有标准答案或 rubric。
4. 能做 error analysis。
5. 能做 regression。

评估数据不是训练数据。

混用会导致 benchmark contamination。

### 4.6 RAG 数据

目标：给模型提供外部知识。

关键问题：

1. 文档切分。
2. 元数据。
3. 权限控制。
4. 更新机制。
5. 检索质量。
6. 引用和 faithfulness。
7. Prompt injection。

### 4.7 多模态数据

包括：

1. 图文对。
2. 图像问答。
3. OCR 数据。
4. 视频字幕。
5. 音频转写。
6. 语音对话。
7. 图表和文档。

多模态数据难点是对齐、质量和标注成本。

## 5. 大模型数据 Pipeline

一个典型 pipeline：

```text
data sourcing
-> crawling / collection
-> parsing
-> language identification
-> normalization
-> filtering
-> deduplication
-> PII and safety processing
-> quality scoring
-> domain classification
-> mixture design
-> tokenization
-> packing / sharding
-> training
-> evaluation and attribution
-> versioning and governance
```

### 5.1 数据来源

先决定从哪里来。

包括：

1. 公开网页。
2. 许可数据。
3. 代码仓库。
4. 企业文档。
5. 合成数据。
6. 人工标注数据。

### 5.2 解析和规范化

原始数据往往很脏。

需要处理：

1. HTML。
2. PDF。
3. Markdown。
4. 表格。
5. 代码。
6. 多语言编码。
7. boilerplate。

### 5.3 过滤和质量评分

过滤低质量、重复、垃圾、广告、乱码、有害和隐私数据。

质量评分可以用规则、分类器、模型或混合方法。

### 5.4 去重

去重非常关键。

重复数据会导致：

1. 训练效率下降。
2. 记忆风险增加。
3. 评估污染。
4. 某些来源过度影响模型。

### 5.5 配比和采样

不同数据源要按目标能力配比。

这不是静态问题，而是需要实验迭代。

### 5.6 版本管理

数据必须可复现。

否则训练结果无法解释。

## 6. 数据问题如何变成模型问题

### 6.1 幻觉

可能来自：

1. 训练数据错误。
2. 知识冲突。
3. 高质量事实数据不足。
4. SFT 奖励完整回答而不是不确定性。

### 6.2 过度拒答

可能来自：

1. 安全数据过于粗糙。
2. 负样本缺少正常安全教育请求。
3. 偏好数据奖励保守拒答。

### 6.3 代码能力弱

可能来自：

1. 代码数据比例不足。
2. 低质量代码多。
3. 缺少测试和解释。
4. 缺少 repo-level 数据。

### 6.4 数学推理弱

可能来自：

1. 数学数据覆盖不足。
2. 步骤错误。
3. 合成数据质量差。
4. 缺少可验证答案。

### 6.5 评估分数虚高

可能来自：

1. Benchmark 泄漏。
2. 训练集包含测试题。
3. 反复用同一评估集调参。
4. 评估集被合成数据污染。

## 7. 数据质量的层次

### 7.1 文本层质量

1. 是否乱码。
2. 是否重复。
3. 是否语法清晰。
4. 是否包含 boilerplate。
5. 是否广告或垃圾内容。

### 7.2 内容层质量

1. 是否事实正确。
2. 是否信息密度高。
3. 是否逻辑清楚。
4. 是否有专业来源。
5. 是否过时。

### 7.3 任务层质量

1. 是否适合目标任务。
2. 是否有明确输入输出。
3. 是否覆盖难度梯度。
4. 是否能验证。

### 7.4 行为层质量

1. 是否符合助手行为。
2. 是否诚实表达不确定性。
3. 是否安全。
4. 是否不过度拒答。
5. 是否风格稳定。

## 8. 数据配比为什么难

数据配比不是简单“多放好数据”。

原因：

1. 不同能力互相竞争。
2. 领域数据可能影响通用能力。
3. 安全数据过多可能 over-refusal。
4. 代码数据影响语言风格。
5. 合成数据可能模式单一。
6. 多语言数据需要平衡。

配比设计需要实验。

常见方法：

1. 小规模 proxy training。
2. Ablation。
3. 分层评估。
4. Data attribution。
5. Curriculum。
6. 动态采样。

## 9. 数据治理为什么是核心能力

没有治理，数据工程会不可控。

### 9.1 可追溯

知道数据从哪里来。

### 9.2 可复现

知道训练用了哪个版本。

### 9.3 可删除

能响应版权、隐私和合规要求。

### 9.4 可评估

知道哪些数据带来收益或风险。

### 9.5 可审计

能解释模型出了问题时数据链路在哪里。

## 10. 面向专家：数据不是静态资产，而是控制面

很多人把数据看成训练前准备好的静态资产。

更成熟的理解是：数据是控制模型能力和行为的控制面。

你可以通过数据控制：

1. 模型学什么语言。
2. 擅长什么任务。
3. 如何回答用户。
4. 是否诚实表达不确定。
5. 如何拒绝危险请求。
6. 是否引用证据。
7. 是否会调用工具。
8. 是否偏向某种风格。

因此，数据工程和 alignment、evaluation、safety、product 都是连接的。

## 11. 真实项目中的数据团队分工

一个大模型数据团队可能包括：

1. 数据采集工程。
2. 数据清洗和过滤。
3. 质量评分模型。
4. 数据混合和采样。
5. 合成数据生成。
6. 标注平台。

数据团队不是简单 ETL 团队，而是模型能力团队的一部分。

## 12. 面试官会怎么问

### 问题 1：为什么数据决定模型能力？

回答要点：

1. 预训练数据决定基础语言和知识覆盖。
2. SFT 数据决定指令跟随和助手风格。
3. 偏好数据决定对齐行为。
4. 安全数据决定风险边界。
5. 评估数据决定模型选择是否可信。
6. RAG 数据决定外部知识质量。

标准回答：

```text
大模型能力不是只由参数和算力决定。预训练数据决定模型能学到什么语言、知识和模式；数据质量决定训练效率和错误知识；数据覆盖决定模型是否偏科；数据配比决定代码、数学、多语言、安全等能力结构；SFT 和偏好数据决定模型是否像助手以及是否符合人类偏好；评估数据和治理决定我们是否能可信地衡量和复现这些能力。
```

### 问题 2：Chinchilla 对数据工程有什么启发？

回答要点：

1. 给定 compute 下，参数量和 token 数要共同 scaling。
2. 早期很多模型 undertrained。
3. 数据 token 数不是附属项，而是 compute-optimal 训练的核心变量。
4. 需要高质量、大规模、可治理的数据管线。

### 问题 3：高质量数据为什么重要？

回答要点：

1. 提高训练效率。
2. 减少错误知识和噪声。
3. 对小模型尤其关键。
4. 影响模型风格、推理和安全行为。
5. 但高质量数据通常更贵、更难规模化。

### 问题 4：预训练数据、SFT 数据、偏好数据有什么区别？

回答要点：

1. 预训练数据学基础语言和知识。
2. SFT 数据教模型按指令回答。
3. 偏好数据教模型区分好坏回答。
4. 三者数据量、格式、质量要求和风险不同。

### 问题 5：数据问题如何导致安全问题？

回答要点：

1. 有害内容进入预训练。
2. 安全样本覆盖不足。
3. 拒答数据过粗导致 over-refusal。
4. 隐私数据导致泄露。
5. Prompt injection 样本不足导致 RAG/Agent 风险。
6. 偏好数据奖励看似安全但无用的回答。

## 13. 标准回答模板

面试中可以这样回答：

```text
我会把大模型数据工程看成从数据来源到模型行为的全链路系统。预训练阶段需要 web-scale 数据采集、解析、语言识别、清洗、过滤、去重、PII 处理、质量评分和数据配比；后训练阶段需要高质量 SFT 数据、偏好数据、安全数据和多轮工具数据；评估阶段需要隔离的 benchmark、golden set 和 regression suite；产品阶段还需要 RAG 数据、权限、版本和治理。

数据的核心维度包括规模、质量、覆盖、配比和治理。规模决定上限，质量决定效率，覆盖决定能力边界，配比决定能力结构，治理决定可复现、可审计和合规。很多训练、评估和安全问题，本质上都是数据问题。
```

## 14. 常见误区

### 14.1 误区：数据越多越好

纠正：低质量、重复和污染数据会浪费 compute，增加记忆和偏差风险。

### 14.2 误区：清洗就是删垃圾文本

纠正：清洗还包括去重、PII、版权、安全、质量评分、领域分类和评估污染控制。

### 14.3 误区：SFT 数据越多越好

纠正：SFT 更看质量、覆盖和格式一致性。低质量 SFT 会毁掉模型行为。

### 14.4 误区：评估数据也是数据，可以混着用

纠正：评估数据必须隔离，否则 benchmark contamination 会让分数失真。

### 14.5 误区：数据工程只是训练前工作

纠正：数据贯穿训练、后训练、评估、RAG、上线监控和治理全周期。

## 16. 本章总结

数据是大模型能力、行为和安全的核心来源。

大模型数据工程经历了从“更多 web-scale 数据”，到“compute-optimal token 数”，再到“高质量、任务匹配、可治理数据”的演化。

数据的五个核心维度是规模、质量、覆盖、配比和治理。

预训练数据、SFT 数据、偏好数据、安全数据、评估数据、RAG 数据和多模态数据分别塑造不同能力和风险。

大模型数据 pipeline 包括采集、解析、过滤、去重、PII、安全处理、质量评分、配比、tokenization、版本和治理。

很多训练问题、评估问题、安全问题和产品问题，本质上都是数据问题。

面试中要把数据工程讲成模型能力和行为控制系统，而不是简单 ETL。
