# 第四章：去重与污染检测

上一章讨论了如何从原始数据中清理乱码、模板、低质量文本和隐私风险。本章继续往下走，讨论大模型数据工程中另一个非常关键的问题：去重与污染检测。

很多初学者会把去重理解成“删掉完全一样的文本”。这只是最简单的一层。真实的大模型训练集中，重复和污染有很多形态：同一篇文章被多个网站转载，同一个网页被不同时间采集多次，同一个代码仓库有大量 fork，同一段 benchmark 答案出现在训练数据中，同一段隐私文本被复制粘贴到多个页面，同一个模板句子重复上万次。

这些问题不是小瑕疵。重复数据会浪费训练 token、增加记忆风险、扭曲数据分布；污染数据会让评测结果虚高，让团队误判模型真实能力。

本章重点：exact dedup、near dedup、MinHash、SimHash、benchmark contamination、canary。

合规边界：本章讨论去重、污染检测、评估可信度和隐私风险治理，不提供提取、复原或利用训练数据中敏感内容的方法。

---

## 1. 先建立直觉：为什么重复数据很危险？

想象你准备考试，有一本 1000 页的复习资料。表面上看内容很多，但里面有 300 页是同一篇文章的不同排版，有 100 页是网页页脚，有 50 页是同一道题和答案反复出现。你会误以为自己读了很多知识，其实有效信息远少于纸面页数。

大模型训练也是这样。训练集的大小通常以 token 数衡量，但 token 数不等于有效信息量。如果 1T token 中有大量重复，那么模型不是在学习更丰富的世界，而是在反复强化少数文本。

重复数据主要带来五类问题：

1. 浪费算力：模型多次看到相同内容，训练预算被低效消耗。
2. 降低泛化：模型过度拟合高频重复文本，而不是学习广泛规律。
3. 增加记忆：重复出现的长文本更容易被模型原样记住。
4. 扭曲分布：某些来源、网站、模板或观点被过度放大。
5. 污染评测：训练集中含有测试题或答案，会导致 benchmark 分数虚高。

Lee 等人在 `Deduplicating Training Data Makes Language Models Better` 中指出，现有语言建模数据集包含大量近重复样本和长重复子串，去重可以减少模型输出训练数据原文的频率，并减少 train-test overlap 对评估的影响。这是数据去重从“工程清理”变成“大模型训练质量关键环节”的典型证据。

---

## 2. 来龙去脉：从数据库去重到大模型去重

传统数据工程里的去重，常常是数据库问题：同一个用户、订单、日志或文档出现多次，通过唯一 ID、时间戳、hash 或主键合并即可。

但大模型语料不同。它通常没有可靠主键，也没有统一 schema。两个文档可能：

1. URL 不同，但正文完全相同。
2. 正文基本相同，只是标题、广告、页脚不同。
3. 同一篇文章被改写、转载、翻译或摘录。
4. 同一份代码出现在原仓库、fork、vendor 目录和镜像中。
5. 同一份 benchmark 题目被博客、教程、题解网站收录。

因此，大模型去重从简单 ID matching 变成了大规模文本相似性问题。

它需要解决三个层次：

1. 完全重复：文本一模一样。
2. 近似重复：文本大部分相同，但有少量修改。
3. 语义污染：文本不一定字面相同，但包含评测题、答案或等价解法。

前两类主要靠 hash、n-gram、MinHash、SimHash、embedding 和分布式索引处理。第三类更难，通常需要 benchmark 原始数据、变体生成、相似检索、人工审计和评估解释共同完成。

## 3. exact dedup：完全去重

exact dedup 是最简单也最便宜的去重方式：对规范化后的文本计算 hash，相同 hash 只保留一份。

这里的关键词是“规范化后”。如果直接对原始文本 hash，很多本质相同的文档会因为空格、换行、HTML 实体、编码差异、大小写差异而被视为不同。

常见规范化包括：

1. 去除首尾空白。
2. 统一换行符。
3. 合并连续空格。
4. 统一 Unicode 表示。
5. 去掉明显模板字段。
6. 可选地统一大小写。
7. 可选地去除标点或 HTML 残留。

然后计算 hash，比如 SHA 系列、xxHash、MurmurHash 等。密码学安全不是这里的主要目标，吞吐、碰撞概率和工程生态更重要。

exact dedup 的优点：

1. 简单。
2. 快。
3. 可解释。
4. 容易分布式实现。
5. 非常适合文档级完全重复。

缺点也明显：

1. 不能发现轻微改写。
2. 不能发现转载时加了广告和页脚的版本。
3. 不能发现段落级重复。
4. 对 OCR、PDF 抽取和格式差异敏感。

所以 exact dedup 通常是第一层，而不是全部。

---

## 4. 文档级、段落级和子串级去重

去重的粒度很重要。

### 4.1 文档级去重

文档级去重把一整篇文章、一页网页或一个文件作为单位。它适合去除完整重复页面、镜像站、重复下载和数据版本合并中的重复。

优点是简单，缺点是粗糙。两篇文章只要有一个页脚不同，就可能无法 exact match。

### 4.2 段落级去重

段落级去重把文档拆成段落，识别重复段落。它适合处理网页模板、转载文章中的固定段落、免责声明和重复介绍。

但段落级去重会带来上下文破坏。删除一个段落后，文档可能变得不连贯。所以很多系统不直接删除段落，而是用段落重复率作为文档质量特征。

### 4.3 子串级去重

子串级去重关注长重复片段。例如某个 50 词句子在数据集中重复几万次，即使所在文档不同，也会造成记忆风险。

子串级去重更细，但计算成本更高。通常需要 n-gram 索引、后缀数组、rolling hash 或专门的大规模文本去重工具。

---

## 5. near dedup：近重复去重

近重复是大模型语料中更常见的问题。

例如：

1. 同一篇文章被多个网站转载，开头结尾略有不同。
2. 新闻稿被不同媒体改标题。
3. 教程被复制后插入广告。
4. 代码仓库 fork 后只改了 README。
5. PDF 抽取版本和 HTML 版本内容基本一致。

如果只做 exact dedup，这些样本都会留下来。

near dedup 的目标是识别“相似到不值得重复训练”的样本。关键问题是：如何定义相似？

常见相似度包括：

1. 字符级相似度。
2. token 级相似度。
3. n-gram Jaccard 相似度。
4. 编辑距离。
5. SimHash 汉明距离。
6. embedding cosine 相似度。

在 web-scale 规模下，不能两两比较所有文档。假设有 10 亿篇文档，两两比较是不可接受的。因此需要近似检索和局部敏感哈希。

---

## 6. MinHash：把集合相似度变成可扩展检索

MinHash 常用于估计两个集合的 Jaccard 相似度。对于文本，可以把一篇文档表示成 n-gram 集合。例如按词切成 5-gram 或按字符切成 shingles。

直觉上，如果两篇文档共享很多 n-gram，它们就很可能是近重复。

MinHash 的核心思想是：不用保存完整 n-gram 集合，而是用多个 hash 函数对集合做摘要。两个集合越相似，它们的 MinHash 签名越可能相同。

在工程上常见流程是：

1. 文本规范化。
2. 分词或字符切片，生成 shingles。
3. 计算 MinHash signature。
4. 使用 LSH 把相似 signature 放入同一候选桶。
5. 只在候选桶内做精确相似度计算。
6. 对超过阈值的文档聚类或删除。

MinHash 的优点：

1. 适合大规模近重复检测。
2. 对小改动较鲁棒。
3. 相比全量 n-gram 比较节省空间。
4. 可与 LSH 结合降低候选对数量。

缺点：

1. 参数敏感，如 shingle 大小、signature 长度、LSH band 设置。
2. 对短文本不稳定。
3. 对语义改写无能为力。
4. 多语言和代码场景需要调整切片策略。

面试时，不需要推导 MinHash 数学细节，但要讲清它解决的是“大规模近重复检索不能两两比较”的问题。

---

## 7. LSH：避免全量两两比较

LSH 是 Locality-Sensitive Hashing，局部敏感哈希。普通 hash 希望相似输入也完全打散；LSH 则相反，它希望相似输入更容易落到同一个桶里。

在去重里，LSH 的作用是候选召回。

没有 LSH 时，10 亿篇文档两两比较不可行。有 LSH 后，系统只比较落入相同或相关桶的文档，候选对数量大幅下降。

但 LSH 不是最终判定器。它只是帮你找到“可能相似”的文档。最终仍然要计算 Jaccard、编辑距离、SimHash 距离或其他相似度，并根据阈值决定是否去重。

LSH 的核心 trade-off 是召回率和计算量：

1. 桶太宽：候选太多，计算量大。
2. 桶太窄：漏掉相似文档。
3. 阈值太高：只能去掉几乎完全重复。
4. 阈值太低：可能误删同主题但不同内容的文档。

因此，大规模去重要通过抽样审计和小模型实验调参，而不是只看理论阈值。

---

## 8. SimHash：适合快速近重复检测

SimHash 也是一种常见近重复方法。它把文档映射成固定长度指纹，相似文档的指纹汉明距离较小。

直觉上，SimHash 先把文本特征映射到向量空间，再压缩成一个二进制签名。两个文档越相似，签名越接近。

SimHash 常用于网页去重和重复页面检测，因为它速度快、存储小、易于索引。

相比 MinHash，SimHash 更像向量特征的随机投影摘要；MinHash 更直接对应集合 Jaccard 相似度。工程上两者都可用，选择取决于文本类型、相似定义和系统生态。

SimHash 的优点：

1. 快。
2. 指纹短。
3. 适合海量网页。
4. 对小修改有一定鲁棒性。

缺点：

1. 参数和特征选择会影响效果。
2. 对短文本不稳定。
3. 对深度语义改写能力有限。
4. 汉明距离阈值需要审计校准。

---

## 9. embedding 去重：语义相似，但更危险

近几年很多团队会考虑用 embedding 做语义去重。做法是把文档、段落或问题编码成向量，用近似最近邻检索找相似样本。

embedding 去重能发现字面不同但语义相近的内容。例如同一道数学题的不同表述、同一个 FAQ 的改写、同一段代码功能的解释。

但它也更危险，因为“语义相似”不等于“应该删除”。

例如：

1. 两篇讲 Transformer 的文章可能主题相似，但细节不同，都有价值。
2. 两道数学题方法相似，但数字和推理不同，都能训练泛化。
3. 两段法律条文解释相似，但适用场景不同。
4. 多个用户问法语义相似，但对指令泛化很有帮助。

因此，embedding 更适合用于污染检测、聚类分析、采样平衡和人工审计辅助，而不是直接大规模删除所有近邻。

如果用于删除，阈值必须很高，并且要分领域评估误删。

---

## 10. 去重不是越狠越好

去重的目标不是让训练集没有任何相似样本，而是避免不合理重复。

有些重复是有害的，比如网页模板、转载垃圾、重复 benchmark 答案、重复隐私片段。有些重复是自然的，比如常见概念的不同讲解、代码 API 的常见用法、数学定理的标准表述。

完全去掉所有相似内容会带来问题：

1. 常识知识频率被压低。
2. 常见表达模式学习不足。
3. 代码模型对常用库和 API 不熟。
4. 数学模型对标准题型 exposure 不够。
5. 对话模型对常见用户意图鲁棒性下降。

所以更专业的策略是：

1. 对明显重复删除。
2. 对近重复聚类后保留代表样本。
3. 对高质量来源重复可降权，而不是全删。
4. 对低质量来源重复严格过滤。
5. 对 benchmark 和隐私相关重复更严格。

去重本质上是在控制数据分布的频率，而不是追求集合意义上的唯一性。

---

## 11. 代码数据去重：fork、vendor 和生成文件

代码数据的重复问题非常严重。

常见来源包括：

1. fork 仓库。
2. vendor 目录。
3. 自动生成代码。
4. minified 文件。
5. 复制粘贴的模板工程。
6. 依赖包镜像。
7. 教程代码和题解代码。

如果不去重，代码模型可能大量记忆常见仓库、题解和库源码。训练时看起来 loss 很低，但真实泛化和安全性未必好。

代码去重可以按多个粒度做：

1. 文件级 hash。
2. 函数级 hash。
3. AST 结构相似。
4. token n-gram near dedup。
5. 仓库级 fork 识别。
6. license 和来源级过滤。

代码去重的难点是：代码天然有大量模板和重复模式。比如 import、main 函数、配置文件、常见算法实现、测试框架样板。如果阈值过严，会删除有用模式；如果阈值过松，又会保留大量复制代码。

工程上通常要区分：

1. 自动生成文件：尽量移除。
2. 第三方 vendor：视 license 和目标处理。
3. fork 重复：保留主源或高质量版本。
4. 常见 API 用法：不宜完全去除。
5. 题库答案：对评估污染要特别严格。

## 12. benchmark contamination：评测污染是什么？

评测污染是指训练数据中包含了评测集的问题、答案、解析或高度相似变体，导致模型评测分数不能反映真实泛化能力。

这是大模型评估中非常严重的问题。因为很多 benchmark 是公开的，而 web-scale 训练数据会抓取论文、博客、GitHub、题解网站、课程材料和论坛讨论。公开 benchmark 一旦发布，就可能被复制到各种网页中。

污染有多种形式：

1. 题目原文在训练集中。
2. 题目和答案都在训练集中。
3. 答案解析在训练集中。
4. 同一道题的改写版本在训练集中。
5. benchmark 构造代码或数据文件在训练集中。
6. 评测 leaderboard 讨论和题解在训练集中。

污染的后果是：模型看起来“会推理”，实际上可能是在记忆题目或答案模式。

BIG-bench、MMLU、HumanEval、GSM8K 等公开评测都可能面临不同程度的污染风险。越流行、越公开、越被讨论的 benchmark，污染风险越高。

## 13. 污染检测的基本方法

污染检测通常需要把评测集作为 query，在训练语料中检索相似内容。

常见方法包括：

1. exact match：题目原文 hash 匹配。
2. n-gram overlap：查找高重叠文本。
3. MinHash/LSH：大规模近重复检索。
4. embedding 检索：发现语义相似或改写题目。
5. answer leakage 检测：检查答案、解析、选项是否一起出现。
6. code benchmark 检测：查找函数签名、测试样例、题目描述和参考解法。
7. 时间切分：只使用 benchmark 发布前数据，降低污染风险。
8. 人工审计：对高相似候选做人工判断。

污染检测不能只看题目。很多时候答案和解析才是关键。如果训练集中只有题目没有答案，污染程度较轻；如果题目、选项、答案和解析一起出现，评测可信度就严重下降。

污染检测也不能只靠 exact match。许多题目会被改写、翻译、格式化或嵌入博客文章中，需要 near dedup 和语义检索辅助。

## 14. 评测污染和训练污染的区别

面试里容易混淆两个概念：训练数据重复和评测污染。

训练数据重复主要影响训练效率、记忆和分布。它发生在训练集内部。

评测污染主要影响评估可信度。它发生在训练集和测试集之间。

两者相关但不相同。一个训练集内部可以高度去重，但仍然包含 benchmark。反过来，一个训练集可能有很多重复，但不一定污染某个特定评测。

所以工程上要分别做：

1. intra-train dedup：训练集内部去重。
2. train-eval overlap detection：训练集和评测集重叠检测。
3. benchmark-specific quarantine：对评测相关数据隔离。
4. eval report annotation：评测报告中说明污染检测方法和风险。

## 15. canary：用探针检测记忆风险

canary 是一种用于检测模型记忆和数据泄漏风险的探针思想。直觉是：在训练数据中插入或标记一些特殊、罕见、可识别的字符串，然后观察模型是否容易输出这些字符串。

在研究和安全评估中，canary 可以帮助估计模型对罕见序列的记忆倾向。它不是为了泄露数据，而是为了评估风险和验证防护策略。

使用 canary 时要注意：

1. 只能使用受控、非真实敏感信息的合成字符串。
2. 不能把真实用户隐私当 canary。
3. 需要控制出现次数、上下文和训练阶段。
4. 评估结果要用于降低记忆风险，而不是复原训练内容。

canary 的价值在于建立因果对照：如果一个罕见字符串出现次数越多，模型越容易记住，那么重复数据和敏感数据的风险就更清楚。

## 16. 去重与记忆风险

大模型的记忆风险和数据重复高度相关。罕见、长、重复出现的文本更容易被记住，尤其是包含结构化信息、固定格式或低熵内容时。

去重可以降低记忆风险，原因很直接：同一段文本出现次数减少，模型在训练中反复强化它的机会也减少。

但去重不是解决记忆的全部。记忆风险还与以下因素有关：

1. 模型规模。
2. 训练轮数。
3. 数据采样策略。
4. 文本罕见程度。
5. 文本结构化程度。
6. 是否包含 PII 或 secrets。
7. 后训练是否强化了复述行为。

因此，去重应和 PII 过滤、安全训练、输出过滤、隐私评估和日志治理一起使用。

---

## 17. 去重策略会影响数据配比

去重不是孤立步骤，它会改变 data mixture。

假设某个来源重复率很高，去重后它的 token 占比大幅下降；另一个来源本来重复率低，去重后占比相对上升。这会改变模型学习到的领域分布。

例如：

1. 新闻转载很多，去重后新闻占比下降。
2. 代码 fork 很多，去重后代码占比下降。
3. 百科文本重复较少，去重后占比相对上升。
4. 论坛数据短文本多，文档级去重影响较小。
5. 数学题库重复模板多，去重策略会影响题型频率。

所以去重后必须重新统计语言、领域、来源和质量分布。否则团队可能以为 data mixture 没变，但实际训练分布已经明显改变。

---

## 18. 去重系统的工程架构

一个可落地的 web-scale 去重系统通常包括：

1. 文本规范化模块。
2. 文档 hash 和 exact dedup。
3. 段落 hash 和重复段落统计。
4. shingling 和 MinHash signature 生成。
5. LSH 候选召回。
6. 候选对精确相似度计算。
7. 连通分量或聚类。
8. 保留策略选择。
9. benchmark contamination 检索。
10. 统计报告和人工审计。
11. 数据版本和 lineage 记录。

保留策略非常重要。发现一组近重复文档后，保留哪一份？常见原则包括：

1. 保留质量分最高的版本。
2. 保留来源最可信的版本。
3. 保留解析质量最好的版本。
4. 保留时间最早或最权威的版本。
5. 对不同语言或地区版本分开处理。

如果随便保留第一份，可能保留了转载垃圾，删掉了原始高质量来源。

---

## 19. 阈值和审计

去重阈值不能拍脑袋。

如果 Jaccard 阈值过高，只能去掉几乎完全相同的文档，重复残留多。如果阈值过低，会误删同主题但有不同内容的文章。

实际做法通常是：

1. 从候选对中按相似度分桶抽样。
2. 人工判断每个桶中多少是真重复、多少是误判。
3. 按语言、领域、来源分别统计。
4. 对不同数据类型设置不同阈值。
5. 用过滤前后数据训练小模型做 ablation。
6. 观察验证 loss、下游 benchmark、记忆指标和生成质量。

特别要注意短文本。短文本的 n-gram overlap 很容易虚高。比如两个短问题都问“什么是 Transformer”，它们相似度很高，但可能都值得保留。对短文本可以用更严格的规则、聚类后采样或按意图保留多样表达。

## 20. 面向专家：去重会改变 scaling behavior

从专家视角看，去重不只是减少数据量。它会改变训练的有效样本分布，从而影响 scaling behavior。

如果重复很多，训练 loss 可能下降更快，因为模型反复看到熟悉文本。但这不代表泛化更好。去重后，训练集更难，短期 loss 可能变化，但下游泛化、记忆风险和评测可信度可能更好。

这也是为什么只看 pretraining loss 不够。一个模型可能在重复数据上 loss 很漂亮，但在真实任务中不稳。另一个模型训练 loss 稍高，但因为数据多样性更好，泛化更强。

去重还会影响 token budget 使用方式。对于固定训练 token 数，如果去重后有效多样性提高，模型在同样 token budget 下能覆盖更多模式。但如果去重过度，常见模式 exposure 不足，也会损害能力。

所以专家级回答要强调：去重要和 data mixture、训练 token budget、sampling strategy、memorization evaluation 和 downstream evaluation 一起考虑。

## 21. 一个可落地的去重与污染检测方案

如果面试官问：“你有 100TB 清洗后的预训练语料，如何做去重和污染检测？”可以按下面思路回答。

第一步，文本规范化。统一编码、空白、换行、HTML 残留和 Unicode 表示，保留原始版本和规范化版本。

第二步，做 exact dedup。对文档级规范化文本计算 hash，相同 hash 合并，记录重复次数和来源。

第三步，做段落级统计。计算段落 hash，识别高频模板段落、页脚、免责声明和重复短语。

第四步，做 near dedup。对文档生成 shingles，计算 MinHash signature，用 LSH 召回候选，再计算精确相似度。

第五步，聚类和保留。把近重复文档组成簇，按质量分、来源可信度、解析质量和时间选择代表样本。

第六步，代码专项去重。对代码做文件级、函数级、仓库级和 fork 级去重，处理自动生成文件、vendor 和题库答案。

第七步，benchmark contamination 检测。把评测集题目、选项、答案、解析、代码签名和测试样例作为 query，做 exact、n-gram、MinHash 和 embedding 检索。

第八步，隔离风险数据。对高相似 benchmark 候选，不一定永久删除，但要从训练池或评测报告中隔离和标注。

第九步，审计和评估。抽样检查误删和漏删，统计去重前后语言、领域、来源、质量分和 token 分布，训练小模型做 ablation。

第十步，版本化。记录 hash、规则、阈值、MinHash 参数、LSH 参数、评测集版本、污染检测结果和数据 lineage。

这套回答体现了系统性：先解决训练集内部重复，再解决训练集与评测集重叠，最后用审计和版本保证可信。

## 22. 常见面试题

### 22.1 exact dedup 和 near dedup 的区别是什么？

exact dedup 识别完全相同的样本，通常通过规范化文本 hash 实现。near dedup 识别高度相似但不完全相同的样本，比如转载、改标题、加广告、轻微改写，通常用 n-gram、MinHash、SimHash、LSH 或 embedding 检索。

### 22.2 为什么大模型训练必须去重？

因为重复数据会浪费训练 token、降低有效多样性、增加记忆风险、扭曲来源分布，并可能造成 train-test overlap。去重能提升训练效率、降低复述训练数据的风险，并提高评估可信度。

### 22.3 MinHash 解决什么问题？

MinHash 用紧凑签名估计集合相似度，常用于大规模文本近重复检测。它把文档表示成 n-gram 集合，通过 signature 和 LSH 召回相似候选，避免全量两两比较。

### 22.4 LSH 的作用是什么？

LSH 的作用是候选召回。它让相似文档更可能落入同一桶，从而只比较少量候选对，而不是对所有文档两两比较。LSH 不是最终判定器，最终还要计算相似度并设置阈值。

### 22.5 benchmark contamination 怎么检测？

把评测集题目、答案、选项、解析、代码签名和测试样例作为 query，在训练语料中做 exact match、n-gram overlap、MinHash/LSH、embedding 检索和人工审计。对高相似结果隔离或在评测报告中标注风险。

### 22.6 去重是不是越彻底越好？

不是。过度去重会删除常见表达、常用代码模式、标准数学题型和真实用户意图多样性。目标是控制不合理重复，而不是消灭所有相似内容。需要按来源、领域、语言和任务目标设置策略。

### 22.7 canary 是什么？

canary 是受控插入或标记的特殊合成字符串，用来评估模型记忆和泄漏风险。它应使用非真实敏感信息，并服务于风险评估和防护验证，而不是复原训练数据。

## 23. 常见误区

误区一：去重就是删除完全相同的文件。

真实系统还需要近重复、段落重复、子串重复、代码 fork 和 benchmark contamination 检测。

误区二：去重后 token 数下降就是坏事。

如果删掉的是低价值重复，虽然 token 数减少，但有效信息密度提高。

误区三：embedding 相似就应该删除。

语义相似不等于重复。很多同主题文本都有不同细节，直接删除会损害覆盖率。

误区四：评测污染只需要查题目原文。

答案、解析、选项、测试样例和题解代码同样可能污染评估。

误区五：去重不会影响 data mixture。

去重会改变来源、语言、领域和质量分布，必须重新统计和调配。

误区六：训练集内部去重做好了，评测就一定可信。

不一定。还必须检测 train-eval overlap 和 benchmark-specific contamination。

---

## 24. 本章小结

去重与污染检测是大模型数据工程中连接训练质量、隐私风险和评估可信度的关键环节。

本章要记住几句话：

1. 重复数据会浪费算力、增加记忆、扭曲分布。
2. exact dedup 只能解决完全重复，near dedup 才能处理转载、改写和模板差异。
3. MinHash、SimHash 和 LSH 是 web-scale 去重的重要工具，但阈值必须审计。
4. benchmark contamination 会让评测分数虚高，必须单独检测。
5. canary 是评估记忆风险的受控探针，不应涉及真实敏感信息。
6. 去重不是越狠越好，它会改变 data mixture 和模型归纳偏置。
7. 成熟系统要记录规则、参数、版本、污染检测结果和审计报告。

如果面试中被问到去重，最好的回答不是只说“用 hash 去重”，而是讲完整链路：规范化、exact dedup、段落重复、near dedup、MinHash/LSH、代码专项、benchmark contamination、审计、ablation 和版本治理。
