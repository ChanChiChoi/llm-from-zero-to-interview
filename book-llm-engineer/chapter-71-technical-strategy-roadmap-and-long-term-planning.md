# 第71章：技术战略、路线图与长期规划

上一章讨论了技术影响力与跨团队沟通。

当一个工程师能推动跨团队协作后，下一步会遇到更大的问题：如何规划一个方向的未来。

这时工作不再只是解决眼前需求，也不只是把某个方案讲清楚。

你需要回答：

- 这个方向未来半年应该解决什么问题？
- 哪些能力应该先做，哪些应该延后？
- 哪些技术债必须还，哪些可以暂时接受？
- 哪些平台能力值得沉淀？
- 哪些项目应该停止？
- 如何把大模型技术变化转化为团队可执行路线图？

这就是技术战略和路线图问题。

大模型领域变化很快。

新模型、新框架、新 Agent 形态、新推理优化、新评测方法不断出现。

如果没有技术战略，团队很容易陷入两种状态。

一种是追热点。

看到什么新技术都想试，结果每个方向都浅尝辄止。

另一种是只做需求。

业务提什么做什么，短期交付很多，但长期能力没有沉淀。

成熟的大模型工程师需要在两者之间找到平衡：既能响应业务，又能建设长期技术能力。

这一章讨论如何制定大模型工程方向的技术战略、路线图和长期规划。

## 71.1 技术战略不是口号

很多团队会把技术战略写成口号。

例如：

- 打造领先的大模型平台。
- 建设智能化业务中台。
- 全面拥抱 Agent。
- 降本增效，提升体验。
- 构建统一 AI 基础设施。

这些话可能方向正确，但不能指导行动。

真正有用的技术战略应该能回答：

- 当前最重要的问题是什么？
- 为什么是现在解决？
- 解决这个问题对业务和技术有什么价值？
- 我们选择哪些路径？
- 我们明确不做什么？
- 需要哪些阶段性里程碑？
- 用什么指标判断战略有效？
- 如果环境变化，什么时候调整？

技术战略不是漂亮文字。

它是一组清晰的选择。

选择做什么，也选择不做什么。

## 71.2 战略从诊断开始

制定路线图之前，先做诊断。

如果没有诊断，路线图通常只是愿望清单。

大模型方向的诊断可以从几个层面展开。

第一，业务诊断。

要问：

- 当前哪些场景最有价值？
- 哪些场景已经被验证？
- 哪些场景只是想象？
- 用户真正愿意使用什么？
- 哪些问题影响收入、效率、成本或风险？

第二，技术诊断。

要问：

- 当前最大质量瓶颈是什么？
- 最大稳定性瓶颈是什么？
- 最大成本瓶颈是什么？
- 最大安全风险是什么？
- 哪些能力只能靠人工维护？
- 哪些模块已经重复建设？

第三，组织诊断。

要问：

- 团队是否有模型、数据、平台、产品和安全协作机制？
- 是否有评测和反馈闭环？
- 是否有人能维护线上系统？
- 是否有足够的 GPU、预算和工具链？
- 哪些依赖团队是瓶颈？

第四，外部技术诊断。

要问：

- 新模型能力是否改变了方案边界？
- 推理框架是否已经足够成熟？
- 开源生态是否能替代部分自研？
- 合规和安全要求是否变化？
- 竞争对手或行业实践是否改变用户预期？

诊断越真实，路线图越不容易跑偏。

## 71.3 技术愿景和技术战略的区别

愿景回答“长期希望成为什么”。

战略回答“当前阶段如何走过去”。

例如一个企业 AI 助手方向的技术愿景可以是：

```text
让企业员工可以通过自然语言安全、可靠地获取知识、调用工具并完成跨系统任务。
```

这是愿景。

它描述长期状态。

但它还不能指导下周做什么。

技术战略需要更具体：

```text
未来 6 个月，我们不优先追求完全自主 Agent，而是先建设可信 RAG、工具调用审计和任务评测体系。原因是当前失败 case 主要来自知识不可信、权限边界不清和工具调用不可追踪。只有先把知识、权限、审计和评测打牢，后续 Agent 才能安全扩展。
```

这就是战略。

它说明了当前阶段的重点和取舍。

愿景可以相对稳定。

战略需要随着约束和证据调整。

## 71.4 路线图不是任务列表

路线图经常被写成一堆任务。

例如：

- 接入新模型。
- 支持多轮对话。
- 加 rerank。
- 加 prompt 管理。
- 加 Agent 工具。
- 做 dashboard。
- 做成本统计。

这只是 backlog，不是路线图。

真正的路线图应该表达阶段目标。

例如：

```text
阶段一：让系统可评估。
  建立核心评测集、日志字段、失败 case 分类和 baseline。

阶段二：让答案可信。
  优化文档解析、retrieval、rerank、引用校验和资料不足拒答。

阶段三：让系统可运营。
  建立反馈回流、prompt/model 版本管理、灰度、监控和成本归因。

阶段四：让能力可扩展。
  抽象工具调用、权限、审计、任务评测和多业务接入规范。
```

每个阶段都有清楚目标。

任务只是服务于阶段目标。

## 71.5 大模型路线图的常见维度

一个成熟的大模型技术路线图，通常不只包含模型能力。

可以从这些维度规划。

第一，模型能力。

- 模型选型。
- 多模型路由。
- 领域适配。
- 微调或偏好优化。
- 多模态能力。

第二，数据能力。

- 数据清洗。
- 文档解析。
- 标注体系。
- 反馈回流。
- 合成数据。
- 数据权限和治理。

第三，检索和知识能力。

- chunk 策略。
- embedding 模型。
- hybrid retrieval。
- rerank。
- 引用校验。
- 知识更新和版本。

第四，Agent 和工具能力。

- tool schema。
- 权限校验。
- 工具调用日志。
- 高风险操作确认。
- 多步任务评测。
- 人工接管。

第五，推理和平台能力。

- 模型服务。
- batching。
- KV cache。
- 量化。
- 监控告警。
- 成本归因。
- 多租户隔离。

第六，评测和实验能力。

- 离线评测集。
- LLM-as-a-judge。
- 人工评测。
- A/B 测试。
- bad case 管理。
- 回归门禁。

第七，安全和治理能力。

- prompt injection 防护。
- 权限边界。
- 敏感信息处理。
- 审计。
- red teaming。
- 合规流程。

路线图要根据当前瓶颈选择重点。

不可能所有维度同时深入。

## 71.6 用北极星指标约束路线图

路线图需要指标约束。

否则很容易变成“我们做了很多能力”，但不知道是否真的变好。

大模型项目的北极星指标不能太单一。

例如只看调用量，可能鼓励无效使用。

只看回答满意度，可能忽略安全和成本。

只看准确率，可能忽略延迟。

更好的方式是设置一组平衡指标。

以企业知识问答为例，可以有：

- 任务成功率。
- 引用准确率。
- 资料不足拒答准确率。
- 人工接管率。
- TTFT / P95 延迟。
- 单次请求成本。
- 越权召回率。

以 Agent 自动化为例，可以有：

- 任务完成率。
- 工具选择准确率。
- 参数准确率。
- 高风险误调用率。
- 平均步骤数。
- 人工接管率。
- 审计完整率。

以推理平台为例，可以有：

- TTFT。
- TPOT。
- tokens/s。
- GPU 利用率。
- KV cache 使用率。
- 错误率。
- 单 token 成本。

指标不只是验收工具。

指标会影响路线图优先级。

如果引用准确率是核心指标，文档解析和 citation eval 可能比换模型更重要。

如果单 token 成本是核心指标，推理优化和路由策略可能比新功能更重要。

## 71.7 优先级排序：影响、紧急、依赖和可逆性

路线图最难的是排序。

可以用四个维度判断优先级。

第一，影响。

这个问题解决后，能影响多少用户、多少成本、多少风险或多少团队？

第二，紧急。

它是否正在阻塞上线、客户交付、安全合规或系统稳定性？

第三，依赖。

它是否是其他能力的前置条件？

例如没有评测集，就很难判断模型切换和 prompt 改动是否有效。

没有工具调用日志，就很难做 Agent 失败分析。

第四，可逆性。

如果做错了，能否低成本回退？

可逆决策可以快速试。

不可逆或高成本决策要更谨慎。

例如换一个 prompt 是相对可逆的。

重构整个平台、迁移向量数据库、自研推理框架就是更重的决策。

排序时要避免两个极端。

一个极端是只做紧急需求，永远没有基础能力。

另一个极端是只做长期平台，业务短期看不到价值。

好的路线图通常同时包含：

- 近期业务交付。
- 中期技术补强。
- 长期平台建设。

## 71.8 技术债要翻译成业务风险

技术债如果只用工程语言表达，很难获得资源。

例如：

- prompt 没有版本管理。
- 评测脚本不统一。
- 检索日志字段缺失。
- 工具调用没有 trace。
- 模型服务没有容量规划。

这些听起来像工程内部问题。

需要翻译成业务风险：

- prompt 没有版本管理，会导致线上效果回退无法定位。
- 评测脚本不统一，会导致模型切换决策不可信。
- 检索日志字段缺失，会导致客户投诉时无法解释为什么答错。
- 工具调用没有 trace，会导致误操作无法审计。
- 模型服务没有容量规划，会导致高峰期延迟失控和成本不可预测。

技术债不是“代码不优雅”。

技术债是未来交付速度、稳定性、安全和成本的风险。

当你能把技术债翻译成业务风险，路线图中的技术建设就更容易被理解。

## 71.9 什么时候应该平台化

平台化是路线图中的常见选择。

但平台化不能太早。

过早平台化会带来：

- 抽象不稳定。
- 需求还没验证就固化。
- 平台团队被大量定制需求拖垮。
- 使用方觉得平台复杂。
- 业务价值迟迟看不到。

适合平台化的信号包括：

- 多个业务线重复接入类似能力。
- 重复问题已经稳定出现。
- 接入流程可以标准化。
- 指标和日志口径已经明确。
- 已有两个以上真实场景验证。
- 手工支持成本开始明显升高。

例如 RAG 平台化之前，最好已经知道常见文档类型、权限模型、chunk 策略、评测方式和失败模式。

Agent 平台化之前，最好已经验证工具 schema、权限、审计、高风险确认和任务评测。

推理平台化之前，最好已经明确模型类型、流量规模、SLO、成本归因和多租户隔离需求。

平台化不是把 demo 包一层 UI。

平台化是把重复成功经验抽象成可复用能力。

## 71.10 什么时候应该停止一个方向

路线图不只是增加项目。

也要决定停止什么。

大模型项目容易因为新鲜感持续投入。

但有些方向应该及时止损。

可以停止或暂停的信号包括：

- 用户真实使用率很低。
- 评测长期没有改善。
- 成本远高于收益。
- 风险无法被工程机制控制。
- 依赖数据长期无法获得。
- 业务流程本身不适合自动化。
- 需求方无法定义成功标准。
- 团队没有维护能力。

例如一个完全自主 Agent 项目，如果连续几轮评测都显示高风险工具误调用无法降到可接受范围，就应该缩小范围，退回 workflow + human confirmation，而不是继续扩大。

停止项目不是失败。

在不确定领域，及时停止低收益方向，是保护团队资源的一种能力。

## 71.11 设计 3 个月、6 个月和 12 个月路线图

路线图可以分时间尺度。

3 个月看交付和验证。

重点是：

- 明确核心场景。
- 建立 baseline。
- 完成关键实验。
- 修复最明显瓶颈。
- 形成最小闭环。

6 个月看能力沉淀。

重点是：

- 建立评测和反馈闭环。
- 抽象重复能力。
- 建立监控、灰度和回滚。
- 支持多个业务场景。
- 降低成本和维护负担。

12 个月看方向演进。

重点是：

- 平台化关键能力。
- 建立多团队协作机制。
- 形成技术资产和标准。
- 支撑更复杂场景。
- 让系统能跟随模型和业务变化持续演进。

例如企业 AI 助手路线图可以写成：

```text
3 个月：完成核心知识问答场景，建立 RAG 评测集、引用准确率指标和失败 case 回流。

6 个月：支持多业务知识库接入，统一权限过滤、文档解析、rerank、prompt 版本和灰度流程。

12 个月：扩展到工具调用和半自动任务流，建立工具审计、任务评测、人工接管和多租户成本治理。
```

这样比“做 RAG、做 Agent、做平台”清楚得多。

## 71.12 路线图要保留调整空间

大模型技术变化太快，路线图不能写死。

可能影响路线图的变化包括：

- 新模型显著提升能力。
- API 成本大幅下降。
- 开源模型满足私有化需求。
- 推理框架支持新的优化。
- 监管要求变化。
- 业务场景优先级变化。
- 线上事故暴露新风险。
- 用户反馈推翻原假设。

因此路线图应该有复盘节奏。

例如每月检查：

- 当前假设是否仍然成立？
- 指标是否朝目标变化？
- 资源投入是否匹配收益？
- 新技术是否改变了原取舍？
- 哪些任务应该提前、延后或取消？

路线图不是承诺永远不变。

路线图是当前证据下的最佳计划。

成熟团队会定期更新路线图，而不是假装环境不会变化。

## 71.13 把路线图变成团队行动

路线图如果不能落到 owner 和节奏，就只是文档。

要把路线图变成行动，需要做几件事。

第一，拆成可交付里程碑。

例如“提升 RAG 可信度”太大。

可以拆成：

- 建立 300 条评测样本。
- 补齐 retrieval 日志字段。
- 完成 rerank ablation。
- 上线 citation accuracy dashboard。
- 灰度资料不足拒答策略。

第二，明确 owner。

每个里程碑都要有人负责推进，不只是“算法团队”或“平台团队”。

第三，明确验收指标。

不是“完成优化”，而是“citation accuracy 提升到某个范围，P95 延迟不超过某个阈值”。

第四，明确依赖。

例如业务专家标注、平台日志、后端权限字段、安全评审。

第五，明确复盘时间。

路线图执行中必须定期检查偏差。

## 71.14 技术战略的面试表达

如果面试官问：“你如何为一个大模型工程方向制定技术路线图？”

可以这样回答：

```text
我不会一开始就列功能清单，而是先做诊断。诊断包括业务价值、当前失败 case、技术瓶颈、团队能力、成本和安全约束。比如在企业 RAG 场景中，我会先看失败主要来自文档解析、召回、rerank、生成、引用、权限还是产品预期。

然后我会区分技术愿景和当前阶段战略。愿景可能是构建可信企业 AI 助手，但 6 个月战略可能不是做完全自主 Agent，而是先把 RAG 可信度、权限、评测、反馈和工具审计打牢。

路线图上我会按阶段目标组织，而不是按零散任务组织。第一阶段让系统可评估，建立 baseline、评测集和日志。第二阶段让答案可信，优化解析、检索、rerank、引用和拒答。第三阶段让系统可运营，建立版本、灰度、监控、成本和反馈闭环。第四阶段再考虑平台化和 Agent 扩展。

优先级排序会看影响、紧急程度、依赖关系和可逆性。技术债也会翻译成业务风险，比如没有 prompt 版本管理会导致线上回退不可追踪，没有工具审计会导致高风险操作不可追责。路线图需要 3 个月、6 个月、12 个月不同粒度，并且定期根据指标、新技术和业务变化调整。

最后我会把路线图落到 owner、里程碑、验收指标和复盘节奏，否则路线图只是文档，不会变成团队行动。
```

## 71.15 小结

技术战略和路线图，是大模型工程师从项目负责人走向方向负责人的关键能力。

技术战略不是口号，而是基于诊断后做出的清晰选择。

路线图不是任务列表，而是围绕阶段目标组织的行动计划。

大模型方向的路线图要同时考虑模型、数据、RAG、Agent、推理平台、评测、安全、成本和组织能力。

优先级排序不能只看谁声音大，也不能只追热点。

要看影响、紧急、依赖和可逆性。

技术债要翻译成业务风险，平台化要等重复成功经验出现，低收益方向要及时停止。

一个好的技术路线图应该既能指导未来 3 个月的交付，也能支撑 6 个月的能力沉淀和 12 个月的方向演进。
