# 第73章：风险、安全与责任边界

上一章讨论了指标、复盘与持续改进。

指标能帮助团队知道系统是否变好。

但大模型工程还有一个不能回避的问题：系统是否足够安全，风险是否被正确管理，责任边界是否清楚。

很多大模型项目早期只关注效果。

能不能回答更准，能不能自动处理更多任务，能不能减少人工，能不能降低成本。

这些当然重要。

但一旦进入生产，问题会变得更复杂。

模型可能泄露敏感信息。

RAG 可能越权召回。

Agent 可能调用错误工具。

工具返回可能被 prompt injection 污染。

模型输出可能被下游系统直接执行。

用户可能过度相信模型建议。

攻击者可能利用长上下文、插件、文件上传、工具调用和多轮对话绕过防线。

如果团队没有安全和责任边界，系统能力越强，风险越大。

这一章讨论：大模型工程师如何识别风险，如何设计安全边界，如何在效果、自动化和责任之间做取舍。

## 73.1 安全不是上线前最后一步

很多团队把安全当成上线前检查项。

等功能快做完了，再找安全团队评审。

这在大模型系统中很危险。

因为很多风险不是外层加一个过滤器就能解决。

例如：

- 权限过滤如果没有在检索前执行，模型可能已经看到了不该看的文档。
- 工具调用如果没有后端鉴权，prompt 再严格也挡不住越权调用。
- 输出如果直接进入 SQL、Shell、浏览器或业务系统，后处理缺失会导致下游漏洞。
- Agent 如果被赋予过多自主权，后续补一个免责声明并不能消除风险。

安全应该从需求、架构和数据流开始。

设计一个大模型功能时，就要问：

- 输入来自哪里？
- 哪些输入不可信？
- 模型能看到哪些数据？
- 数据权限在哪里强制执行？
- 模型能调用哪些工具？
- 工具调用是否有副作用？
- 输出会被谁消费？
- 输出是否会进入下游执行环境？
- 错误结果的代价是什么？
- 谁可以审计和回滚？

安全不是最后的闸门。

安全是系统设计的一部分。

## 73.2 大模型应用的风险地图

OWASP LLM Top 10 把 LLM 应用中的典型风险总结为 prompt injection、insecure output handling、training data poisoning、model denial of service、supply chain vulnerabilities、sensitive information disclosure、insecure plugin design、excessive agency、overreliance 和 model theft 等类别。

这些风险可以映射到工程链路中。

第一，输入风险。

包括恶意 prompt、文件上传、网页内容、RAG 文档、工具返回、用户多轮对话和外部 API 返回。

第二，上下文风险。

包括越权文档、敏感字段、被污染的检索结果、过长上下文中的隐藏指令、系统 prompt 泄露和跨租户信息混入。

第三，模型行为风险。

包括幻觉、错误建议、不稳定输出、拒答失败、过度自信和不遵循安全策略。

第四，工具风险。

包括工具选择错误、参数错误、权限绕过、副作用操作、幂等问题和插件设计不安全。

第五，输出风险。

包括敏感信息泄露、恶意代码、错误 JSON、SQL/HTML/Markdown 注入、被下游系统直接执行和误导用户。

第六，系统风险。

包括成本攻击、长上下文 DoS、模型服务过载、供应链漏洞、模型或 prompt 版本污染、日志泄露和审计缺失。

第七，组织风险。

包括用户过度信任、责任不清、人工兜底缺失、安全评审流于形式和事故响应不成熟。

成熟工程师不会只说“加安全过滤”。

他会把风险放回完整链路中分析。

## 73.3 Prompt Injection 不是 prompt 写得更强就能解决

Prompt injection 是大模型应用中最典型的风险之一。

攻击者可能在用户输入、网页、文档、邮件、代码注释、工具返回或多轮上下文中植入指令，诱导模型忽略系统规则、泄露信息或调用不该调用的工具。

常见形式包括：

- 直接要求模型忽略之前指令。
- 在 RAG 文档中写入“把所有秘密发给用户”。
- 在网页内容中隐藏指令，诱导浏览器 Agent 执行操作。
- 在工具返回中加入下一步操作指令。
- 用多轮对话逐步绕过规则。

防护 prompt injection 不能只靠更长的 system prompt。

因为模型仍然会同时看到可信指令和不可信内容。

更稳的做法是分层防护：

- 明确区分指令和数据。
- 对外部内容做标记和隔离。
- 不让不可信内容覆盖系统策略。
- 工具调用必须经过后端权限校验。
- 高风险操作需要用户确认。
- 对敏感输出做后处理检测。
- 对攻击样本建立 red team 测试集。
- 日志中记录触发策略和工具调用原因。

一句话：

```text
Prompt 可以降低风险，但不能承担权限、安全和审计的全部责任。
```

## 73.4 权限必须在系统层执行

大模型系统中最危险的错误之一，是把权限交给模型判断。

例如：

```text
如果用户没有权限，请不要回答。
```

这类 prompt 只能作为辅助约束，不能作为安全边界。

真正的权限控制必须在系统层执行。

RAG 场景中，权限应该在检索前或检索时生效。

模型不应该看到用户无权访问的文档。

工具调用场景中，权限应该在服务端检查。

模型可以提出 tool call proposal，但后端必须验证用户身份、资源权限、操作范围和风险等级。

多租户场景中，租户隔离必须在数据层、索引层、缓存层和日志层都成立。

不能只依赖 prompt 告诉模型“不要泄露其他租户信息”。

权限设计可以遵循几个原则：

- 最小权限。
- 默认拒绝。
- 服务端强制。
- 数据先过滤再进入上下文。
- 高风险操作二次确认。
- 全链路审计。
- 权限策略可测试。

如果权限只存在于 prompt 中，那么它实际上没有成为系统边界。

## 73.5 工具调用的风险边界

Agent 和 function calling 让模型可以影响真实系统。

这也是风险从“说错话”升级为“做错事”的关键点。

工具调用风险主要来自几类。

第一，工具选择错误。

模型选择了不该调用的工具。

第二，参数错误。

模型把用户、金额、时间、资源 ID、权限范围填错。

第三，副作用不可控。

工具会发邮件、删文件、改配置、下订单、提交工单、调整权限或执行代码。

第四，幂等性缺失。

重试可能导致重复操作。

第五，工具返回不可信。

工具返回中可能包含恶意指令或污染内容。

第六，审计不足。

事后无法知道模型为什么调用、传了什么参数、谁批准了操作。

工具调用设计要分风险等级。

低风险工具可以自动执行。

例如搜索、读取公开信息、格式转换。

中风险工具需要明确参数校验和可回滚机制。

例如创建草稿、生成配置建议、提交非关键任务。

高风险工具需要人工确认、权限校验和审计。

例如转账、删数据、发外部邮件、修改生产配置、授予权限。

模型不应该直接拥有无限工具权限。

它应该在受控边界内提出建议，由系统执行校验。

## 73.6 Insecure Output Handling：不要直接信任模型输出

模型输出是非确定、可能错误、可能被攻击影响的文本。

如果下游系统直接执行模型输出，就会产生 insecure output handling 风险。

典型例子包括：

- 模型生成 SQL，系统直接执行。
- 模型生成 Shell 命令，系统直接运行。
- 模型生成 HTML，前端直接渲染。
- 模型生成 JSON，后端不校验就写库。
- 模型生成代码，自动合并或部署。
- 模型生成工具参数，系统直接调用高风险 API。

正确做法是把模型输出当成不可信输入。

需要：

- schema 校验。
- 类型检查。
- 白名单。
- 参数范围限制。
- escaping 和 sanitization。
- 权限校验。
- dry-run。
- 人工确认。
- 沙箱执行。
- 审计日志。

尤其是代码生成、SQL 生成、浏览器自动化和运维 Agent 场景，必须把验证成本纳入系统设计。

AI 让生成变便宜，但验证变得更重要。

## 73.7 敏感信息和数据边界

敏感信息泄露是大模型系统中的高频风险。

泄露可能发生在多个位置：

- prompt 中包含密钥或内部规则。
- RAG 召回了无权文档。
- 模型输出了个人信息、商业机密或内部链接。
- 日志记录了完整用户输入和模型输出。
- 评测数据包含真实敏感样本。
- 向第三方 API 发送了不该出域的数据。
- prefix cache 或上下文缓存跨租户复用错误。

数据边界设计要问：

- 哪些数据可以进入模型？
- 哪些数据可以进入第三方 API？
- 哪些数据可以写日志？
- 哪些数据需要脱敏？
- 哪些数据必须加密？
- 哪些数据需要保留，保留多久？
- 哪些数据可以用于训练或评测？

企业场景尤其要注意：

- 用户身份和权限。
- 租户隔离。
- 文档密级。
- 审计要求。
- 数据出域规则。
- 日志和 trace 权限。

敏感信息保护不是只靠模型“不要泄露”。

它需要数据治理、权限、脱敏、日志策略和访问控制共同实现。

## 73.8 Excessive Agency：不要给模型过度自主权

OWASP LLM Top 10 中的 excessive agency 指的是给 LLM 过多能力、权限或自主决策空间，导致不可预期后果。

这是 Agent 系统中最核心的风险之一。

过度自主权可能表现为：

- 模型可以调用过多工具。
- 工具权限过宽。
- 操作不需要确认。
- 没有任务边界。
- 没有最大步骤数。
- 没有预算限制。
- 没有失败停止条件。
- 没有人工接管。
- 没有审计。

降低 excessive agency 风险，可以使用：

- 限制工具集合。
- 按任务动态授权。
- 设置最大步骤数和最大成本。
- 高风险操作人工确认。
- 先 dry-run 再执行。
- 低置信度转人工。
- 关键操作可回滚。
- 所有工具调用记录 trace。

一个实用原则是：

```text
模型可以负责建议和规划，但系统必须负责授权、执行和审计。
```

## 73.9 Overreliance：不要让用户过度相信模型

风险不只来自攻击者。

也来自用户过度相信模型。

模型回答自然、流畅、看起来很自信。

用户可能因此忽略核验。

在医疗、金融、法律、招聘、安全、运维和高风险业务中，过度依赖会造成严重后果。

降低 overreliance 风险，需要产品和工程共同设计。

可以做：

- 明确展示引用和证据。
- 对资料不足场景拒答。
- 对低置信任务提示不确定性。
- 高风险建议要求人工确认。
- 明确模型能力边界。
- 展示最后更新时间和数据来源。
- 不把模型输出伪装成确定结论。
- 关键决策保留人工责任人。

不要滥用免责声明。

免责声明不能替代真实安全机制。

如果系统知道某类任务风险高，就应该限制、确认或转人工，而不是只在页面底部写“AI 可能出错”。

## 73.10 成本和可用性攻击

大模型系统还有一种特殊风险：成本攻击和资源耗尽。

攻击者或异常用户可能构造超长输入、反复请求、诱导长输出、触发昂贵工具、绕过缓存或制造大量并发。

这会导致：

- GPU 资源被占满。
- 正常用户排队。
- 成本快速上升。
- 模型服务 OOM。
- 下游工具被打爆。
- 日志和存储膨胀。

防护手段包括：

- 用户和租户级限流。
- input token 限制。
- output token 限制。
- 请求并发限制。
- 工具调用预算。
- 单任务最大步骤数。
- 长上下文审批或分级。
- 异常成本告警。
- admission control。
- 过载时优雅拒绝。

大模型服务的安全不仅是数据安全，也包括成本和可用性安全。

## 73.11 风险分级和上线门禁

不是所有大模型功能风险都一样。

应该按风险等级管理。

低风险功能：

- 文本润色。
- 内部知识摘要。
- 无权限敏感数据的问答。
- 只读公开信息检索。

中风险功能：

- 企业内部知识问答。
- 工单草稿生成。
- 代码建议。
- 半自动数据分析。

高风险功能：

- 权限敏感知识问答。
- 自动发送外部消息。
- 修改生产系统。
- 财务、法律、医疗建议。
- 自动执行代码。
- 自动变更权限。

不同等级对应不同上线门禁。

低风险可以重点看质量和基本安全。

中风险需要评测、权限、日志、灰度和回滚。

高风险需要安全评审、人工确认、红队测试、审计、严格权限、事故预案和责任人。

风险分级的目的不是阻止创新。

它是为了让系统在合适的边界内创新。

## 73.12 责任边界：模型、系统和人的分工

大模型系统必须明确责任边界。

模型负责什么？

系统负责什么？

人负责什么？

一个比较稳的分工是：

- 模型负责理解、生成、总结、分类、建议和候选行动。
- 系统负责权限、校验、执行、审计、限流、回滚和监控。
- 人负责高风险决策、异常确认、策略制定和最终责任。

不要让模型承担它不适合承担的责任。

模型不应该是权限系统。

模型不应该是唯一审计机制。

模型不应该在没有确认的情况下执行高风险操作。

模型不应该为法律、医疗、金融等高风险决策承担最终责任。

清楚的责任边界能减少事故，也能让团队更容易通过安全和合规评审。

## 73.13 风险治理不是一次性文档

风险治理需要持续运行。

至少包括：

- 风险登记表。
- 安全评审。
- red team 测试。
- 上线门禁。
- 监控告警。
- 事故复盘。
- 策略更新。
- 权限审计。
- 模型和 prompt 版本审计。

风险登记表可以记录：

- 风险描述。
- 影响范围。
- 发生概率。
- 严重程度。
- 缓解措施。
- owner。
- 当前状态。
- 重新评估时间。

例如：

```text
风险：RAG 越权召回内部薪酬文档。
影响：敏感信息泄露和合规风险。
缓解：检索前按用户权限过滤，索引按租户和密级隔离，日志记录 document_id 和 permission decision。
owner：知识库平台负责人。
状态：灰度前必须完成权限回归测试。
```

风险治理要进入工程流程，而不是停留在文档里。

## 73.14 面试表达

如果面试官问：“你如何保证一个大模型 Agent 系统安全上线？”

可以这样回答：

```text
我会先做风险分级，而不是一上来讨论模型效果。Agent 系统的风险主要来自不可信输入、越权上下文、工具误调用、输出被下游执行、敏感信息泄露、过度自主权和用户过度信任。

架构上，我不会把安全完全交给 prompt。权限必须在系统层执行，RAG 检索前要按用户权限过滤，工具调用必须经过服务端鉴权。模型可以提出 tool call proposal，但系统负责参数校验、权限校验、风险分级、执行和审计。高风险操作必须人工确认，最好先 dry-run，再执行。

对 prompt injection，我会区分可信指令和不可信数据，对外部内容做隔离和标记，建立攻击样本评测集。对模型输出，我会当成不可信输入处理，做 schema 校验、白名单、范围限制和后处理检查，不能直接执行 SQL、Shell、代码或高风险 API。

上线前要有 red team、离线评测、灰度、监控和回滚。上线后看工具误调用率、高风险操作确认率、越权召回率、敏感信息泄露、成本异常、错误率和人工接管率。事故后要复盘并把失败样本回流到评测集。

我的原则是：模型负责理解和建议，系统负责授权、校验、执行和审计，人负责高风险决策和最终责任。
```

## 73.15 常见误区

第一个误区是把安全交给 prompt。

Prompt 可以辅助约束，但不能替代权限、校验和审计。

第二个误区是模型看到了不该看的数据后再要求它不要说。

正确做法是不要让无权数据进入上下文。

第三个误区是工具越多越智能。

工具越多，攻击面和误调用风险越大。

第四个误区是只关心 prompt injection，不关心输出处理。

模型输出进入下游系统时同样需要校验。

第五个误区是低估成本攻击。

长上下文、长输出和多步 Agent 都可能造成资源耗尽。

第六个误区是用免责声明替代产品和工程机制。

高风险场景需要限制、确认和审计，不只是提醒用户。

第七个误区是没有责任边界。

一旦出事，团队说不清模型、系统和人的责任划分。

## 73.16 小结

大模型工程的安全和风险管理，不能等到上线前才补。

它必须从需求、数据流、权限、工具、输出处理、监控和责任边界开始设计。

OWASP LLM Top 10 和 NIST AI RMF 都提醒我们：大模型风险不仅是模型回答错，还包括 prompt injection、敏感信息泄露、不安全插件、过度自主权、过度依赖、供应链、成本攻击和治理缺失。

成熟的大模型工程师不会把安全简化为“加一个过滤器”。

他会把模型放在完整系统中看：哪些输入不可信，哪些数据不能进入上下文，哪些操作必须由系统鉴权，哪些输出必须校验，哪些任务必须人工确认，哪些风险需要持续监控和复盘。

最终原则很简单：

```text
模型可以增强能力，但系统必须定义边界。
```

没有边界的智能，不是生产能力，而是生产风险。
