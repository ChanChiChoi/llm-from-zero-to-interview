# 第68章：从执行者到技术负责人

前一章讨论了如何把项目经验沉淀成职业资产。

当一个工程师积累了一些项目经验后，下一步通常不是简单地做更多需求，而是开始负责一个方向、一个模块或一个子系统。

这时角色会发生变化。

你不再只是“把分配给你的任务做好”，而是要对一块技术区域的长期结果负责。

这就是从执行者到技术负责人的转变。

大模型工程中的技术负责人，不一定是正式管理者，也不一定有下属。

他可能负责 RAG 质量链路，可能负责 Agent 工具平台，可能负责推理服务稳定性，可能负责评测体系，可能负责数据闭环，可能负责某个业务 AI 产品的技术架构。

这一章讨论：如何完成这个角色转变，以及技术负责人真正需要负责什么。

## 68.1 执行者和技术负责人的区别

执行者关注任务完成。

技术负责人关注系统结果。

执行者常问：

- 这个需求什么时候做完？
- 接口怎么实现？
- 代码怎么改？
- 测试怎么过？
- bug 怎么修？

技术负责人还要问：

- 这个方向真正要解决什么问题？
- 当前最大瓶颈在哪里？
- 系统未来半年会不会撑不住？
- 哪些能力应该平台化？
- 哪些技术债必须尽快处理？
- 指标是否能证明业务价值？
- 团队是否有足够的评测、监控和回滚能力？
- 如果我不在，别人能否接手？

执行者可以只对局部任务负责。

技术负责人必须对端到端效果负责。

例如一个 RAG 执行者可能只负责实现 rerank 接入。

而 RAG 技术负责人要关心：文档解析是否可靠，召回是否可评估，rerank 是否值得成本，引用是否可信，权限是否正确，用户反馈是否能回流，线上失败 case 是否被纳入评测。

这就是角色差异。

## 68.2 技术负责人首先要定义问题

很多技术负责人失败，不是因为写代码能力不够，而是因为问题定义不清。

如果问题定义错了，后面做得越快，偏得越远。

大模型项目尤其容易被错误问题牵引。

例如业务说“我们要一个 Agent”。

真正的问题可能是流程自动化，而不是一定要让模型自主规划。

业务说“模型不够聪明”。

真正的问题可能是知识库召回不准、上下文缺失或工具返回错误。

业务说“希望回答更自然”。

真正的问题可能是用户不信任答案，因为没有引用和可追溯证据。

技术负责人要把模糊诉求转化成清楚问题。

可以用下面的方式定义问题：

- 现象是什么？
- 影响了谁？
- 频率有多高？
- 严重程度如何？
- 目前有什么证据？
- 根因假设是什么？
- 可以用什么指标验证？
- 解决后业务会得到什么收益？

问题定义越清楚，团队越容易形成共识。

## 68.3 技术负责人要管理技术方向，而不是追热点

大模型领域热点很多。

技术负责人如果没有判断力，很容易把团队带进频繁换方案的状态。

今天换新模型，明天换新框架，后天加 Agent，下周重构 RAG，再下周尝试微调。

结果每个方向都做了一点，但没有任何能力真正沉淀。

技术方向管理的核心是排序。

你需要判断：

- 哪些问题最值得解决？
- 哪些能力是基础设施？
- 哪些优化只是锦上添花？
- 哪些技术现在不成熟？
- 哪些项目应该止损？
- 哪些投入会带来长期复用？

例如一个企业知识库问答方向，可能同时有很多想法：换更强模型、加 rerank、做 query rewrite、支持长上下文、优化文档解析、做反馈回流、做多轮记忆、做权限过滤。

技术负责人不能简单地全部做。

他要根据当前瓶颈排序。

如果失败 case 显示大量问题来自文档解析和权限过滤，那么优先级可能不是换模型，而是先补数据链路和安全边界。

## 68.4 建立方向级指标体系

执行单个需求时，可以只看局部指标。

负责一个方向时，必须建立方向级指标体系。

没有指标，技术负责人就无法判断方向是否变好。

以 RAG 方向为例，指标可以分层：

- 检索层：召回命中率、Top-K 命中率、rerank 后命中率。
- 生成层：回答正确率、引用准确率、拒答准确率、幻觉率。
- 体验层：用户满意度、人工接管率、追问率。
- 性能层：首 token 延迟、P95 延迟、吞吐、错误率。
- 成本层：单次请求 token 成本、模型调用成本、缓存命中率。
- 安全层：越权召回率、敏感信息泄漏率、攻击拦截率。

以 Agent 方向为例，指标可以包括：

- 任务完成率。
- 工具选择准确率。
- 参数填写准确率。
- 工具调用成功率。
- 人工接管率。
- 高风险操作误触发率。
- 平均步骤数。
- 失败恢复率。
- 审计完整率。

指标体系不需要一开始完美。

但必须逐步建立。

技术负责人要推动团队从“感觉变好”转向“有证据地变好”。

## 68.5 设计架构时要考虑演进，而不是只满足当前需求

执行者实现当前需求。

技术负责人要考虑系统演进。

这并不是要求一开始做很复杂的架构。

相反，好的负责人通常会保持架构简单，但为关键变化点留出空间。

大模型系统中常见变化点包括：

- 模型会更换。
- prompt 会频繁迭代。
- embedding 模型会升级。
- 检索策略会调整。
- 工具 schema 会变化。
- 评测集会扩展。
- 数据权限会变复杂。
- 成本策略会变化。
- 安全规则会变严格。

如果架构完全写死，后续每次调整都会变成大改。

例如模型调用层最好不要散落在业务代码中。

prompt 版本最好有记录。

检索和 rerank 最好可以独立评测。

工具调用最好有统一的参数校验和日志。

权限过滤最好不要依赖 prompt 约束。

这些不是过度设计，而是对高频变化点做基本隔离。

## 68.6 技术负责人要控制复杂度

很多人以为技术负责人要设计复杂系统。

其实更重要的是控制复杂度。

大模型项目很容易变复杂：多模型、多 prompt、多工具、多流程、多数据源、多评测集、多配置、多异常分支。

复杂度失控后，团队会遇到：

- 新人看不懂系统。
- 改一个 prompt 影响多个场景。
- 指标变差不知道原因。
- 线上问题无法复现。
- 配置互相覆盖。
- 工具调用链路不可追踪。
- 测试成本不断升高。

技术负责人要定期问：

- 这个抽象是否真的必要？
- 这个框架是否解决了核心问题？
- 这个配置是否有人理解？
- 这个分支是否可以删除？
- 这个能力是否应该合并？
- 这个流程是否可以标准化？

控制复杂度不是保守。

它是为了让系统能持续迭代。

## 68.7 建立评审机制，而不是靠个人经验

当方向变大后，不能只靠负责人一个人把关。

需要建立机制。

常见机制包括：

- 方案评审。
- prompt 变更评审。
- 评测集变更评审。
- 模型切换评审。
- 工具权限评审。
- 上线 checklist。
- 事故复盘。
- 技术债定期梳理。

机制的目的不是增加流程负担，而是减少低级风险。

例如模型切换前，应该明确：为什么切换、对比样本是什么、收益是多少、成本和延迟如何变化、是否有回滚方案。

例如 Agent 新增工具前，应该明确：工具是否有副作用、权限如何控制、参数如何校验、失败如何处理、调用日志如何记录。

好的技术负责人会把自己的经验变成团队流程。

这样即使他不参与每个细节，系统质量也不会完全依赖个人记忆。

## 68.8 技术负责人要会做取舍

负责方向时，最难的不是提出方案，而是做取舍。

取舍通常发生在这些维度之间：

- 准确率和延迟。
- 效果和成本。
- 灵活性和稳定性。
- 自动化和可控性。
- 通用平台和业务定制。
- 快速上线和长期架构。
- 模型能力和工程约束。
- 用户体验和安全边界。

例如 Agent 系统中，更高的自主性可能带来更强能力，但也带来不可控风险。

对于高风险业务，技术负责人可能要选择更保守的工作流编排，而不是完全开放的自主 Agent。

例如 RAG 中，rerank 可能提升准确率，但增加延迟和成本。

如果用户场景对实时性要求很高，就可能需要缓存、分层检索或只对低置信 query 启用 rerank。

技术负责人不是永远选择最先进方案，而是在约束下选择最合适方案。

## 68.9 推动跨团队协作

大模型方向很少能由单个工程师独立完成。

技术负责人需要和多个团队协作。

例如一个企业 AI 助手可能涉及：

- 业务团队提供场景和专家反馈。
- 数据团队提供文档、标注和清洗。
- 算法团队提供模型和评测支持。
- 平台团队提供推理服务和监控。
- 后端团队负责权限、接口和流程集成。
- 前端团队负责交互体验。
- 安全团队负责合规、审计和风险控制。

技术负责人要能把不同团队的语言翻译成共同目标。

业务方关心效率和风险。

算法方关心数据和模型效果。

平台方关心稳定性和成本。

安全方关心边界和审计。

产品方关心体验和上线节奏。

如果没有人负责对齐，项目会变成各做各的。

技术负责人需要推动共识：问题是什么，目标是什么，谁负责什么，风险在哪里，什么时候评估结果。

## 68.10 技术负责人要培养接班人

如果一个方向完全依赖一个人，说明这个方向还不健康。

技术负责人应该有意识地降低个人单点风险。

可以通过几种方式：

- 写清架构文档。
- 记录关键决策。
- 让其他人参与方案评审。
- 把排障流程写成 checklist。
- 把常见问题沉淀成文档。
- 让团队成员轮流负责小模块。
- 在 code review 中解释设计意图。
- 把实验脚本和评测流程工具化。

培养接班人不是削弱自己的价值。

相反，它说明你能把个人能力转化为组织能力。

当你能让别人接手现有系统，你才有空间负责更大的方向。

## 68.11 如何判断自己是否准备好负责一个方向

可以用下面的问题自测。

如果你已经能比较稳定地回答这些问题，说明你正在接近技术负责人角色。

- 我是否能讲清这个方向的业务目标？
- 我是否知道当前最大技术瓶颈？
- 我是否知道哪些指标最重要？
- 我是否知道线上最常见失败模式？
- 我是否能判断新技术是否值得引入？
- 我是否能设计评测和回归机制？
- 我是否能识别高风险变更？
- 我是否能推动跨团队协作？
- 我是否能让别人理解并接手我的系统？
- 我是否能把经验沉淀成流程、文档和工具？

如果这些问题大多回答不上来，说明你可能还停留在执行者阶段。

这并不可怕。

成长的关键是从现在开始主动补齐这些视角。

## 68.12 常见误区

从执行者转向技术负责人时，有几个常见误区。

第一个误区是事事亲自做。

负责人不是所有代码都自己写，而是确保关键问题被正确解决。

如果你什么都亲自做，方向规模会被你的个人时间限制住。

第二个误区是只做架构不碰细节。

大模型系统的很多问题藏在细节里：prompt、数据、异常 case、权限、日志、指标口径。

如果完全脱离细节，方案容易空泛。

第三个误区是用权威代替证据。

技术负责人不能只说“我觉得这样更好”。

要尽量用数据、实验、失败 case 和工程约束支撑判断。

第四个误区是追求大而全平台。

平台化应该来自重复需求，而不是预先想象。

没有足够业务验证前，过早平台化可能拖慢项目。

第五个误区是忽视安全和成本。

大模型应用如果只看效果，很容易在上线后被成本、延迟、权限和合规问题反噬。

## 68.13 一个方向负责人的工作样例

假设你负责一个企业 RAG 方向。

你的工作不只是接需求。

你可能需要做：

- 梳理当前知识库问答的主要失败类型。
- 建立核心业务问题评测集。
- 定义召回、引用、幻觉、延迟和成本指标。
- 评估文档解析、chunk、embedding、rerank、query rewrite 的优先级。
- 设计权限过滤和引用校验机制。
- 建立 prompt 和模型变更回归流程。
- 推动线上反馈进入失败 case 池。
- 输出 RAG 接入规范，供其他业务线复用。
- 和平台团队协作降低推理成本。
- 和安全团队确认敏感数据处理方式。

这才是方向负责人的视角。

他不是只做一个功能，而是让这个方向持续变得更可靠、更可复用、更可度量。

## 68.14 从技术负责人到更高层级

技术负责人再往上，通常会走向更大的技术影响力。

例如从负责一个模块，到负责多个系统；从负责一个业务线，到负责平台能力；从解决项目问题，到定义技术战略。

更高层级会更关注：

- 多团队协同。
- 技术路线选择。
- 资源投入优先级。
- 长期平台能力。
- 风险治理。
- 人才培养。
- 技术文化。
- 业务和技术之间的战略连接。

但这些能力不是突然出现的。

它们都从负责一个小方向开始。

如果你能把一个小方向负责清楚，下一步才有资格负责更大范围。

## 68.15 小结

从执行者到技术负责人，是大模型工程师职业成长中的关键转变。

执行者关注任务完成，技术负责人关注系统结果、方向演进和长期能力建设。

技术负责人需要定义问题、管理技术方向、建立指标体系、控制复杂度、推动评审机制、做取舍、协调跨团队合作，并把个人经验转化为团队资产。

这个角色不一定需要正式头衔。

当你开始主动对一个方向的质量、稳定性、成本、安全、评测和可维护性负责时，你就已经在承担技术负责人的工作。

长期来看，这种责任意识和系统视角，是从中级工程师走向高级、资深和技术领导者的核心能力。
