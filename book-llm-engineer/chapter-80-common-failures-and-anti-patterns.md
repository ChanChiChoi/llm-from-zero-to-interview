# 第80章：常见失败案例与反面教材

上一章讨论了模拟面试训练与刻意练习。

这一章从反面看问题。

很多大模型工程师候选人不是因为完全不会而失败，而是踩中了非常典型的坑。

这些坑包括：项目包装过度、只会讲框架、没有指标、忽视安全、不会复盘、面试中硬编经验、入职后急于重构等。

反面教材的价值在于：它能帮助你提前避坑。

## 80.1 失败一：把 demo 当生产项目讲

很多候选人做过 demo，但在简历和面试中讲成生产项目。

面试官一追问：

- 有多少真实用户？
- 评测集怎么构建？
- 如何灰度？
- 如何回滚？
- 线上失败怎么处理？
- 成本如何控制？

候选人就答不上来。

demo 可以讲，但要讲清边界。

正确表达是：

```text
这是一个个人项目或原型验证，重点验证 RAG 链路和评测方法。它没有真实线上流量，所以我主要用公开数据和自建评测集验证 retrieval、rerank 和 generation 的效果。生产化还需要补权限、监控、灰度、成本和反馈闭环。
```

诚实边界比夸大更可信。

## 80.2 失败二：简历堆满技术名词

“熟悉 RAG、Agent、LangChain、vLLM、LoRA、MCP、向量数据库”不是项目能力。

面试官关心的是：

- 你解决了什么问题？
- 你负责哪一块？
- 你如何验证效果？
- 你遇到什么失败？
- 你做了什么取舍？

堆技术名词会让候选人看起来像在追热点。

简历要从工具清单改成结果清单。

## 80.3 失败三：没有指标

大模型项目没有指标，很难证明有效。

常见空泛表达：

```text
显著提升回答质量。
大幅降低延迟。
优化用户体验。
提升系统稳定性。
```

更好的表达是：

```text
citation accuracy 从 71% 提升到 86%。
P95 TTFT 从 3.4s 降到 1.9s。
格式合法率从 92% 提升到 98%。
人工接管率下降 18%。
单次成功任务成本下降 27%。
```

如果没有精确指标，也要给出样本规模、评测方式和相对变化。

没有指标的项目，很难形成说服力。

## 80.4 失败四：只会讲框架，不会讲原理

候选人说用了 LangChain、LlamaIndex、vLLM、SGLang。

面试官继续问：

- chunk size 为什么这样选？
- rerank 解决什么问题？
- PagedAttention 不是 sparse attention 吗？
- continuous batching 和 dynamic batching 有什么区别？
- Agent 工具调用失败怎么恢复？

如果答不上来，框架名会变成扣分项。

框架是工具。

面试要讲清工具背后的机制和取舍。

## 80.5 失败五：忽视安全和权限

很多候选人讲 Agent 时只讲智能。

但不讲权限、审计、确认和回滚。

这在生产面试中很危险。

如果模型能调用工具，就必须问：

- 工具是否有副作用？
- 参数如何校验？
- 权限在哪里执行？
- 高风险操作是否确认？
- 调用日志如何审计？
- prompt injection 怎么防？

只讲“模型会自动调用工具”是不够的。

生产系统里，模型可以建议，系统必须校验。

## 80.6 失败六：不会讲失败

完美项目故事通常不真实。

成熟候选人应该能讲失败。

例如：

- 最初判断错了瓶颈。
- 某次 prompt 改动导致格式回退。
- rerank 提升质量但增加延迟。
- 长上下文方案成本不可接受。
- Agent 自主性过高导致误调用风险。

关键不是失败本身。

关键是你如何发现、定位、修正和复盘。

不会讲失败的人，面试官会怀疑你没有真实生产经验。

## 80.7 失败七：面试中硬编经验

大模型面试追问很深。

如果硬编经验，很容易露馅。

例如你说做过 vLLM 生产优化，面试官可能问：

- TTFT 和 TPOT 如何拆？
- KV cache block 如何估算？
- prefix cache 命中率是多少？
- prefill 和 decode 如何调度？
- OOM 时看哪些指标？

如果没有真实经验，很难持续回答。

正确做法是承认边界。

可以说：

```text
我没有在生产中维护过 vLLM 集群，但我做过本地 benchmark，并系统学习过 serving engine 的请求生命周期、KV cache、batching 和 TTFT/TPOT 指标。如果要上线，我会重点补容量测试、监控、灰度和故障演练。
```

诚实边界不会一定扣分。

硬编通常会严重扣分。

## 80.8 失败八：入职后急于证明自己

有些人刚入职就想重构系统。

但没有理解历史约束，就容易破坏已有系统。

正确做法是：

- 先理解业务和历史决策。
- 跑通链路。
- 看线上指标。
- 和相关方沟通。
- 选择小闭环建立可信度。
- 再提出中期改进。

前 30 天最重要的不是证明自己比前人聪明。

而是证明自己可靠、能学习、能协作、能形成闭环。

## 80.9 失败九：只追热点，不沉淀资产

今天学 RAG，明天学 Agent，后天学 MCP，大后天学推理框架。

但每个方向都没有项目复盘、指标、代码、文档和面试表达。

这会导致知识很散。

正确做法是每学一个方向，都沉淀一个资产：

- 一篇项目复盘。
- 一个可运行 demo。
- 一个评测集。
- 一个 debug checklist。
- 一个标准回答。

职业成长靠资产累积，不靠热点堆积。

## 80.10 小结

常见失败并不复杂。

它们大多来自不真实、不结构化、不复盘、不看指标、不讲边界。

大模型工程师要避免：

- 把 demo 当生产。
- 堆术语。
- 没指标。
- 只讲框架。
- 忽视安全。
- 不讲失败。
- 硬编经验。
- 入职急于重构。
- 只追热点不沉淀。

真正可靠的候选人，不是永远不犯错，而是能清楚知道边界、能用证据说话、能从失败中学习。
