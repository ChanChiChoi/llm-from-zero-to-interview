# 第74章：成本、ROI 与商业判断

上一章讨论了风险、安全与责任边界。

安全解决的是“能不能安全地做”。

这一章讨论另一个现实问题：值不值得做。

大模型工程师很容易把注意力放在技术指标上。

模型更强，prompt 更复杂，RAG 更准确，Agent 步骤更多，推理框架吞吐更高。

但在生产环境中，技术方案最终要回到业务价值和成本结构。

一个方案如果质量提升 2%，但成本增加 5 倍、延迟增加 3 倍、维护复杂度显著上升，它未必是好方案。

一个模型如果能力很强，但只在少数低价值场景使用，也未必值得大规模接入。

一个 Agent demo 如果看起来很智能，但每次任务调用 20 次模型、3 个外部工具、还需要人工复核，ROI 可能并不成立。

成熟的大模型工程师要能把 token、GPU、延迟、质量、风险、维护成本和业务收益放到同一张表里判断。

这一章讨论：如何理解大模型工程的成本结构，如何评估 ROI，如何建立商业判断，如何避免“技术上可行，但商业上不成立”的项目。

## 74.1 大模型不是免费能力

传统软件功能上线后，边际成本通常较低。

大模型功能不同。

每次调用都可能产生明显成本。

成本来自：

- 输入 token。
- 输出 token。
- cached token 或 prefix cache 资源。
- embedding 调用。
- rerank 调用。
- 工具调用。
- 向量数据库查询。
- GPU 推理。
- 日志和 trace 存储。
- 人工审核。
- 失败重试。
- 安全和评测。

如果是自建模型服务，还要考虑：

- GPU 采购或租赁。
- 机器折旧。
- 机房、电力和网络。
- 推理框架维护。
- 容量冗余。
- on-call 和故障处理。
- 模型升级和回归评测。

因此大模型功能不能只看“能不能做”。

还要看“每做一次要花多少钱”。

## 74.2 Token 使用量不是价值指标

很多团队喜欢看 token 用量。

它容易统计，也能反映大模型调用规模。

但 token 用量不是价值指标。

token 用得多，可能说明系统被广泛使用。

也可能说明 prompt 过长、上下文浪费、模型输出啰嗦、用户反复追问、Agent 规划低效。

如果把 token 用量当成生产力指标，团队可能被误导。

例如：

- 工程师为了显示 AI 使用积极，增加无意义调用。
- Agent 为了完成简单任务，反复规划和反思。
- RAG 把大量无关文档塞入上下文。
- 模型输出很长，但用户真正需要的只是一个结论。

更好的指标是单位价值成本。

例如：

- 单次成功问答成本。
- 单个有效工单处理成本。
- 单次人工节省成本。
- 单个高质量代码 review 成本。
- 单个成功 Agent 任务成本。
- 每减少 1% 人工接管率需要增加多少成本。

token 是资源消耗。

价值要从任务结果看。

## 74.3 成本要按链路拆开

大模型应用的成本不是一个数字。

应该按链路拆开。

以 RAG 问答为例，一次请求可能包括：

```text
query rewrite cost
+ embedding cost
+ vector search cost
+ rerank cost
+ prompt input token cost
+ output token cost
+ citation verification cost
+ logging and storage cost
+ human review cost if failed
```

以 Agent 任务为例，一次任务可能包括：

```text
planning model calls
+ tool selection calls
+ tool execution cost
+ observation summarization calls
+ verification calls
+ final response cost
+ retry cost
+ human confirmation cost
```

以自建推理平台为例，单位成本可能包括：

```text
GPU hourly cost
/ stable output tokens per hour under SLO
+ platform operation cost
+ redundancy cost
+ storage and network cost
+ engineering maintenance cost
```

拆开后才能知道优化点在哪里。

如果成本主要来自长 prompt，就优化上下文和缓存。

如果成本主要来自 rerank，就考虑分层 rerank 或只对低置信 query 启用。

如果成本主要来自多步 Agent，就考虑 workflow 固化、工具结果缓存或减少反思轮次。

如果成本主要来自 GPU 低利用率，就优化 batching、路由和容量规划。

## 74.4 ROI 的基本公式

ROI 不需要一开始做得非常复杂。

最基本的问题是：收益是否大于成本。

可以用一个简单公式：

```text
ROI = (business value - total cost) / total cost
```

其中 business value 可以来自：

- 节省人工时间。
- 提高转化率。
- 降低错误率。
- 减少客服成本。
- 提升用户留存。
- 加快研发交付。
- 降低事故损失。
- 支持新收入。

total cost 包括：

- 模型调用成本。
- GPU 成本。
- 存储和网络成本。
- 工程开发成本。
- 维护和 on-call 成本。
- 安全和合规成本。
- 评测和标注成本。
- 用户教育和运营成本。

ROI 计算不一定精确。

但它能迫使团队把“技术收益”翻译成“业务收益”。

## 74.5 不同阶段的 ROI 标准不同

大模型项目在不同阶段，ROI 标准不同。

Demo 阶段不需要严格 ROI。

目标是验证可行性。

可以接受手工流程、少量样本和较高成本。

试点阶段需要粗略 ROI。

要看真实用户是否愿意使用，失败类型是否可控，成本是否在可接受范围内。

生产阶段必须看稳定 ROI。

要证明系统在真实流量下长期有价值，且质量、成本、延迟、安全和维护都可持续。

平台化阶段要看规模 ROI。

要证明平台能力能服务多个业务线，减少重复建设和维护成本。

不要用 demo 阶段的标准推动生产上线。

也不要在 demo 阶段用生产 ROI 扼杀探索。

关键是知道项目处在哪个阶段。

## 74.6 质量提升要和成本一起看

模型效果提升很重要。

但不能孤立看。

例如某个更强模型让回答正确率从 82% 提升到 87%。

这看起来是好事。

但还要问：

- 成本增加多少？
- P95 延迟增加多少？
- 格式稳定性是否变化？
- 安全拒答是否变好？
- 哪些场景提升最大？
- 哪些场景没有提升？
- 用户是否感知到提升？
- 这个提升是否值得长期支付？

如果高价值场景从 70% 提升到 90%，成本翻倍也可能值得。

如果低价值场景从 95% 提升到 96%，成本翻倍可能不值得。

所以评估效果时要分桶。

按用户、场景、任务价值、风险等级和请求类型分桶。

平均正确率很容易掩盖真实商业判断。

## 74.7 什么时候该用强模型，什么时候该用小模型

大模型工程不是永远用最强模型。

更成熟的做法是模型分层。

强模型适合：

- 高价值任务。
- 复杂推理任务。
- 高风险回答前的验证。
- 低频但重要的决策支持。
- 需要强指令遵循和复杂上下文理解的场景。

小模型适合：

- 分类。
- 路由。
- 简单抽取。
- 格式转换。
- 初筛。
- 低风险高频任务。
- embedding 或 rerank 的部分替代任务。

可以设计分层策略：

```text
简单问题 -> 小模型或规则
中等问题 -> 标准模型
复杂或高价值问题 -> 强模型
高风险操作 -> 强模型 + 人工确认
```

这比所有请求都打到最强模型更经济。

模型路由的目标不是省钱本身。

而是在质量可接受的前提下，把强模型用在最值得的地方。

## 74.8 RAG、微调、长上下文和 Agent 的成本取舍

不同技术路径有不同成本结构。

RAG 的成本主要来自检索、rerank、上下文 token、引用校验和数据维护。

它适合知识更新频繁、需要引用、企业私有知识和可追溯场景。

微调的成本主要来自数据构造、训练、评测、部署和版本维护。

它适合稳定任务模式、风格一致性和高频重复任务。

长上下文的成本主要来自 input token、prefill 延迟和注意力/KV cache 压力。

它适合完整文档分析、上下文关系密集、材料数量有限的场景。

Agent 的成本主要来自多轮模型调用、工具调用、失败重试、审计和人工确认。

它适合步骤不固定、工具选择动态、用户目标多样的场景。

技术选型时要问：

- 知识是否频繁更新？
- 是否需要引用？
- 任务模式是否稳定？
- 用户请求是否高频？
- 是否需要动态规划？
- 是否有高风险工具？
- 成本随规模如何增长？

不要把所有问题都升级成最复杂方案。

复杂方案如果没有对应的业务价值，就是成本负担。

## 74.9 降本不是简单砍模型

降低成本不等于把模型换小。

换小模型可能降低调用成本，也可能导致质量下降、重试增加、人工接管增加，最终总成本更高。

更系统的降本手段包括：

- 压缩 prompt。
- 删除无用上下文。
- 提高 prefix cache 命中率。
- 对重复请求做结果缓存。
- 对简单请求使用规则或小模型。
- 对复杂请求再升级强模型。
- 减少不必要的 Agent 轮次。
- 限制 max output tokens。
- 优化 RAG chunk 和 Top-K。
- 只在必要时启用 rerank。
- 批处理 embedding 和离线任务。
- 优化推理 serving 的 batching 和并发。
- 建立租户级预算。

降本要同时看质量。

一个好的降本项目应该写清：

- 成本下降多少。
- 哪些质量指标不能下降。
- 哪些场景可能受影响。
- 如何灰度和回滚。

## 74.10 成本归因：谁在花钱，为什么花钱

如果没有成本归因，大模型平台很快会变成公共水龙头。

大家都在用，但没人知道谁在花钱、钱花在哪里、是否值得。

成本至少要按几个维度归因：

- 业务线。
- 租户。
- 用户类型。
- 模型。
- 场景。
- 请求类型。
- input / output token。
- cache hit。
- 工具调用。
- 成功和失败任务。

只知道总账单没有用。

要知道：

- 哪个业务线成本最高？
- 哪类请求 token 最多？
- 哪些失败请求消耗了大量成本？
- 哪些 Agent 任务反复重试？
- 哪些租户超过预算？
- 哪些模型性价比最低？

成本归因不是为了阻止使用。

而是为了让团队把资源投到真正有价值的场景。

## 74.11 成本预算和用量治理

生产系统需要预算机制。

常见做法包括：

- 租户级月度预算。
- 用户级限额。
- 场景级预算。
- 单请求 token 上限。
- 单任务工具调用上限。
- 超预算告警。
- 高成本请求审批。
- 异常用量自动降级。

预算机制要避免过度粗暴。

不能让高价值请求因为统一限额被误伤。

可以做分级：

- 普通请求使用标准预算。
- 高价值客户使用更高预算。
- 高风险或高成本任务需要审批。
- 过载时优先保障核心场景。

预算不是财务限制而已。

预算也是系统稳定性和产品策略的一部分。

## 74.12 自建还是 API 的商业判断

自建模型服务和调用外部 API 的技术取舍，最终也要回到商业判断。

API 适合：

- 早期验证。
- 流量不稳定。
- 团队缺少推理运维能力。
- 需要最强模型能力。
- 不希望承担 GPU 和平台复杂度。

自建适合：

- 流量稳定且规模足够大。
- 数据不能出域。
- 需要版本可控。
- 需要深度定制。
- 单位成本可以通过规模摊薄。
- 团队有推理平台能力。

判断时不要只比较单次调用单价。

还要比较：

- 工程人力。
- 运维成本。
- 可用性。
- 扩容速度。
- 模型能力差距。
- 合规成本。
- 故障责任。
- 机会成本。

很多团队适合混合策略。

高价值复杂任务用外部强模型。

高频稳定任务用自建模型。

敏感数据任务用私有部署。

这样比单一策略更灵活。

## 74.13 让工程成为业务驱动者

商业判断不是产品经理或管理者的专属能力。

高级工程师也需要商业意识。

这并不是要求工程师做销售。

而是要理解技术投入如何影响业务结果。

例如：

- 一个评测平台能减少错误上线，降低事故和回滚成本。
- 一个 prompt 版本系统能减少线上回退时间。
- 一个推理成本 dashboard 能让业务线优化高成本场景。
- 一个 RAG 引用校验机制能提升客户信任。
- 一个 Agent 审计系统能让高风险自动化可上线。

工程师如果能把这些技术能力翻译成业务价值，就更容易获得资源和影响力。

工程不是成本中心的同义词。

好的工程能让业务更快、更稳、更可控地使用 AI。

## 74.14 面试表达

如果面试官问：“你如何评估一个大模型项目是否值得投入？”

可以这样回答：

```text
我会先明确项目阶段。如果是 demo，重点是验证技术可行性；如果是试点，要看真实用户是否愿意使用、失败类型是否可控、成本是否可接受；如果是生产系统，就必须看稳定 ROI。

评估 ROI 时，我不会只看模型效果，也不会只看 token 用量。收益侧要看业务价值，比如节省人工时间、提高任务完成率、降低错误率、减少客服成本、提升转化或降低风险。成本侧要看完整链路，包括模型调用、GPU、embedding、rerank、工具调用、日志存储、工程维护、评测标注、安全合规和人工复核。

技术方案上，我会按场景分桶评估。高价值复杂任务可以用强模型，低风险高频任务可以用小模型、规则或缓存。RAG、微调、长上下文和 Agent 的成本结构不同，要根据知识更新频率、引用需求、任务稳定性和自动化价值选择。降本也不是简单换小模型，而是优化 prompt、上下文、缓存、路由、rerank 策略、Agent 步数和推理平台利用率。

最终我会看单位价值成本，比如单次成功任务成本、单次有效问答成本、每减少一次人工接管的成本，而不是只看总 token 或总调用量。一个项目值得投入，应该是核心业务指标改善，同时质量、延迟、成本、安全和维护复杂度都在可接受范围内。
```

## 74.15 常见误区

第一个误区是只看 token 用量。

token 用量是消耗指标，不是价值指标。

第二个误区是只看模型单价。

完整成本还包括工程、评测、运维、安全和失败重试。

第三个误区是默认强模型一定更划算。

强模型在高价值任务上可能值得，在低价值高频任务上可能浪费。

第四个误区是降本只靠换小模型。

如果质量下降导致重试和人工接管增加，总成本可能更高。

第五个误区是只看平均 ROI。

不同场景价值不同，必须分桶分析。

第六个误区是忽略维护成本。

复杂 Agent、私有推理平台和自研框架上线后都需要长期维护。

第七个误区是没有成本归因。

没有归因，团队无法知道谁在花钱、为什么花钱、是否值得。

## 74.16 小结

大模型工程不是免费能力。

每一次 token、每一次模型调用、每一次工具调用、每一次 GPU 推理，最终都会进入成本结构。

成熟的大模型工程师不能只看技术效果。

他要能把质量、延迟、成本、安全、维护复杂度和业务收益放在一起判断。

token 用量不是价值指标。

真正重要的是单位价值成本和业务结果。

强模型、小模型、RAG、微调、长上下文、Agent、自建服务和外部 API 都有不同成本结构，没有一种方案永远正确。

好的工程判断，是把最贵的能力用在最值得的地方，把简单任务用简单方案解决，把复杂度留给真正有回报的场景。

最终，大模型工程师的商业判断不是让技术妥协，而是让技术投入产生可持续价值。
