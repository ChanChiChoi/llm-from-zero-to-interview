# 第77章：面试复盘、Offer 选择与谈薪

上一章讨论了 30-60-90 天成长计划。

如果你已经拿到面试机会，甚至已经拿到 offer，接下来还有一组非常现实的问题：

- 面试后如何复盘？
- 怎么判断自己哪里答得好、哪里需要补？
- 反向提问应该问什么？
- 拿到多个 offer 怎么选？
- 如何判断团队是否真的适合大模型工程师成长？
- 谈薪时应该注意什么？
- 如何降低入职后的风险？

很多候选人把全部精力放在“通过面试”。

但对职业发展来说，选对团队和角色同样重要。

一个看起来光鲜的大模型岗位，可能只是包装出来的 AI 项目。

一个薪资更高的 offer，可能对应更差的成长空间、更混乱的组织、更不可控的风险。

一个名气较小的团队，反而可能给你完整负责 RAG、Agent、评测、推理或 AI Infra 的机会。

这一章讨论：面试后如何复盘，如何做 offer 选择，如何谈薪，如何在加入前识别风险。

## 77.1 面试结束不是结束

每一场面试结束后，都应该立即复盘。

原因很简单：面试记忆衰减很快。

如果当天不记录，第二天你就会忘记很多细节。

面试复盘至少记录：

- 公司和岗位。
- 面试轮次。
- 面试官角色。
- 被问到的问题。
- 自己回答得好的地方。
- 回答卡住的地方。
- 面试官追问方向。
- 暴露出的知识短板。
- 反向提问得到的信息。
- 对岗位和团队的初步判断。

建议建立一个表格。

每场面试后用 15 分钟补全。

不要等整个流程结束再回忆。

复盘的价值不只是准备下一场。

它还能帮你判断哪些公司真的理解大模型工程，哪些只是跟风招聘。

## 77.2 如何分析自己的面试表现

复盘时不要只写“答得不好”。

要具体拆分。

大模型工程面试可以按几个维度分析。

第一，原理表达。

例如 Transformer、attention、KV cache、RAG、Agent、训练、评测、推理优化是否能讲清楚。

第二，工程经验。

是否能讲清楚项目架构、数据流、线上问题、监控、回滚、成本和安全。

第三，系统设计。

是否能从需求、约束、模块、指标、故障和演进讲方案。

第四，实验思维。

是否能讲 baseline、ablation、评测集、指标、误差分析和上线验证。

第五，沟通表达。

是否能结构化回答，是否能承认不确定性，是否能和面试官形成讨论。

第六，商业和风险判断。

是否能谈质量、成本、延迟、安全、ROI 和责任边界。

如果某一轮挂了，不要简单归因为“面试官不喜欢我”。

要问：

- 是知识不会？
- 是项目讲不清？
- 是缺少指标？
- 是没有真实上线经验？
- 是系统设计太散？
- 是表达没有结构？
- 是反问暴露了兴趣不匹配？

找到原因，下一轮才会变强。

## 77.3 面试后如何补短板

面试暴露的短板要分类补。

如果是概念短板，例如 KV cache、LoRA、DPO、speculative decoding 不清楚，要回到原理材料和代码样例。

如果是项目短板，例如讲不清指标和结果，要补项目复盘文档。

如果是系统设计短板，要练习固定模板：需求、约束、模块、数据流、指标、风险、演进。

如果是表达短板，要把高频问题写成 2 分钟、5 分钟、10 分钟三个版本。

如果是经验短板，要承认边界，同时用实验、demo、开源项目和设计文档补证据。

补短板时不要无限学习。

面试准备最容易陷入“我再学一点就好了”。

更有效的是面向问题补：

```text
这次被问到 RAG 评测，我答得散。
下一步我要准备一个固定回答：评测集构建、指标、bad case 分类、线上反馈回流和灰度验证。
```

这样补才有效。

## 77.4 反向提问是双向面试

很多候选人把反向提问当成礼貌环节。

这是浪费机会。

反向提问能帮你判断团队质量。

尤其是大模型岗位，外部描述和真实工作可能差异很大。

你要问出：

- 这个团队是真的做大模型核心工程，还是只做 API 包装？
- 是否有真实用户和线上流量？
- 是否有评测和反馈闭环？
- 是否有数据、平台、安全和产品协作？
- 是否有资源做长期建设？
- 是否有技术 owner 和清晰路线图？
- 是否能给你成长空间？

反向提问不是质问。

要用专业、具体的问题判断信息。

## 77.5 大模型岗位的反向提问清单

关于业务和场景，可以问：

- 这个大模型系统服务的核心用户是谁？
- 当前最重要的业务指标是什么？
- 项目处于 demo、试点还是生产阶段？
- 目前最大的用户痛点是什么？
- 成功标准如何定义？

关于技术栈，可以问：

- 当前主要使用外部 API、自建模型，还是混合模式？
- RAG、Agent、微调、长上下文分别在哪些场景使用？
- 是否有统一评测平台或实验流程？
- 推理服务是自建还是使用现成框架？
- 是否有模型、prompt、数据和工具版本管理？

关于线上质量，可以问：

- 现在如何评估回答质量？
- 失败 case 如何收集和回流？
- 是否有灰度、回滚和线上监控？
- 最常见的线上失败类型是什么？
- 有哪些事故或复盘经验？

关于安全和成本，可以问：

- 权限过滤在哪里执行？
- Agent 工具调用如何做审计和确认？
- 是否有 prompt injection 或 red team 测试？
- 成本如何归因到业务线或租户？
- 当前最大的成本瓶颈是什么？

关于团队和成长，可以问：

- 这个岗位前三个月希望我完成什么？
- 团队内谁负责技术路线图？
- 我会负责完整链路，还是只负责某个模块？
- 是否有机会参与设计评审和线上复盘？
- 这个岗位未来 6-12 个月的成长空间是什么？

这些问题会让你看清岗位真实情况。

## 77.6 Offer 选择不要只看薪资

薪资重要。

但不是唯一因素。

尤其对大模型工程师，offer 选择至少看八个维度。

第一，岗位真实性。

是否真的做大模型核心工程，还是只是调用 API 做包装。

第二，项目阶段。

demo、试点、生产、平台化，不同阶段成长完全不同。

第三，团队能力。

是否有强技术 owner，是否有数据、平台、安全、产品协作。

第四，个人职责。

你是完整负责一个问题，还是只做边缘模块。

第五，资源和授权。

是否有模型、GPU、数据、评测、业务支持。

第六，成长空间。

是否能接触真实用户、线上系统、评测、成本和安全。

第七，组织稳定性。

公司是否持续投入 AI，团队是否稳定，项目是否有明确价值。

第八，薪资和长期收益。

包括 base、bonus、股票、期权、涨薪空间和学习价值。

一个好的 offer，不一定是钱最多的。

而是能让你在未来 1-2 年形成更强职业资产的。

## 77.7 如何识别高风险团队

面试中要注意危险信号。

例如：

- 团队说不清核心业务指标。
- 所有项目都叫 Agent，但没有真实用户。
- 没有评测集，只靠主观感觉判断效果。
- 没有线上监控和回滚。
- 安全问题只靠 prompt 解决。
- 成本完全没人管。
- 大模型项目只是领导口号，没有明确 owner。
- 团队频繁变方向。
- 面试官无法解释岗位前三个月要做什么。
- 所有问题都要求候选人“来了再说”。

这些不一定意味着不能去。

但意味着你要降低预期，或在谈 offer 前问清楚。

有些高风险团队也有高成长机会。

前提是你有足够授权和资源去改变现状。

如果团队混乱，但你只是执行者，风险很高。

如果团队混乱，但明确希望你建立评测、平台或技术方向，并给你支持，可能是机会。

## 77.8 谈薪的基本原则

谈薪不是对抗。

谈薪是把你的市场价值、岗位价值和公司预算对齐。

几个原则：

第一，尽量拿到完整 offer 结构。

包括：

- base。
- bonus。
- 股票或期权。
- 签字费。
- 年终或绩效规则。
- 涨薪周期。
- 试用期政策。
- 社保、公积金、福利。

第二，不要只比较月薪。

不同公司股票、bonus、现金流、税务和稳定性差异很大。

第三，表达要基于事实。

例如：

```text
我对团队和方向很感兴趣。结合我的大模型工程经验、目前其他机会和市场水平，我希望 total compensation 能调整到 X 区间。如果空间有限，也可以讨论签字费、职级或后续 review 节奏。
```

第四，不要编造 offer。

这会损害信任。

第五，提前明确底线。

包括最低可接受薪资、通勤、远程、工作强度、成长空间和风险接受度。

谈薪前如果自己都不知道底线，很容易被动。

## 77.9 职级比 title 更重要

大模型岗位 title 很乱。

同样叫“大模型工程师”，可能差异很大。

你要看实际职级和职责。

要问：

- 这个岗位对应什么级别？
- 职级期望是什么？
- 晋升标准是什么？
- 是否需要带方向或带人？
- 绩效如何评估？
- 入职后第一个周期看什么结果？

title 听起来高级，但实际只是执行 API 接入，成长有限。

title 普通，但能负责 RAG 质量闭环、Agent 工具平台、推理成本治理，反而更有价值。

职业成长看的是你解决问题的复杂度和影响范围，不是 title 字面。

## 77.10 入职前要确认的事项

接受 offer 前，尽量确认：

- 汇报对象是谁。
- 所属团队是什么。
- 第一个项目是什么。
- 前 90 天期望是什么。
- 是否有 mentor 或 buddy。
- 是否有明确技术 owner。
- 是否有试用期目标。
- 是否有岗位变动风险。
- 是否有竞业或保密限制。
- 是否有工作地点、远程、出差和加班要求。

这些问题越早确认，入职后越少意外。

如果对方无法回答前 90 天目标，说明岗位可能还不清晰。

如果岗位不清晰，你要判断自己是否愿意承担这种不确定性。

## 77.11 面试复盘如何变成长期资产

面试复盘不只是为了当前求职。

它也能变成长期学习资产。

你可以把所有面试问题归类：

- 原理题。
- 项目题。
- 系统设计题。
- Debug 题。
- 实验评测题。
- 安全风险题。
- 成本 ROI 题。
- 行为面试题。

每类问题写出标准回答。

经过几轮面试，你会得到一份高度真实的市场题库。

这比网上零散题库更有价值。

因为它来自真实公司、真实岗位、真实追问。

如果你把面试复盘持续沉淀，下一次求职准备会轻松很多。

## 77.12 面试表达

如果面试官问：“你选择 offer 时最看重什么？”

可以这样回答：

```text
我会综合看岗位真实性、项目阶段、团队能力、个人职责、资源支持、成长空间和薪酬。对大模型岗位来说，我不会只看 title 或是否使用最新模型，而会看团队是否有真实用户、线上系统、评测闭环、成本和安全意识，以及我是否能完整负责一个问题。

如果一个岗位能让我接触真实业务场景，参与从数据、模型、RAG 或 Agent 到评测、上线、监控和复盘的完整链路，即使短期 title 没那么夸张，也会很有成长价值。反过来，如果只是包装 API、没有评测、没有真实用户、方向频繁变化，我会比较谨慎。

薪酬当然重要，但我更关注 total compensation 和未来 1-2 年能否形成职业资产。理想的机会是技术挑战真实、团队目标清晰、我能交付结果并持续扩大影响。
```

## 77.13 常见误区

第一个误区是面试后不复盘。

错过了最真实的学习材料。

第二个误区是反向提问只问福利。

福利可以问，但更要问项目、团队、评测、上线和成长空间。

第三个误区是只看薪资最高的 offer。

高薪但低成长、低授权、高风险，长期未必划算。

第四个误区是相信所有 AI 岗位包装。

很多岗位只是普通业务加一层大模型 API。

第五个误区是谈薪时没有完整结构。

只看 base，不看 bonus、股票、签字费、试用期和涨薪规则。

第六个误区是入职前不确认前 90 天目标。

入职后才发现岗位和预期不一致。

第七个误区是忽视组织风险。

方向不稳、owner 不清、资源不足，会严重影响成长。

## 77.14 小结

通过面试不是求职链路的终点。

面试后复盘、反向提问、offer 选择、谈薪和入职前确认，同样决定你的职业收益。

大模型岗位尤其需要谨慎判断。

因为外部包装往往很相似，但真实工作可能差异巨大。

一个成熟候选人，不只是会回答问题，也会判断团队是否值得加入。

你要通过反向提问看清业务目标、项目阶段、技术链路、评测闭环、安全成本和成长空间。

选 offer 时，不只看薪资，也看未来 1-2 年能否形成可迁移的职业资产。

最终，求职不是把自己卖给一个 title，而是选择一个能让你持续成长、持续交付、持续扩大影响的环境。
