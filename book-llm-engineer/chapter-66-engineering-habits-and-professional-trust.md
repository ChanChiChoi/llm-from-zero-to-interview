# 第66章：入职后的工程习惯与长期可信度建设

前几章讨论了面试、入职、职业成长和持续学习。

但真正决定一个大模型工程师能否长期走远的，不只是知识储备，也不只是某一次项目表现，而是日常工程习惯。

在真实团队中，一个人是否值得信任，往往不是通过一次漂亮演示建立的，而是在一次次需求澄清、方案设计、代码提交、问题排查、上线验证和复盘中建立的。

大模型工程尤其如此。

因为它天然包含很多不确定性：模型输出不稳定，评测不完全可靠，用户问题长尾，成本和延迟敏感，安全边界复杂，线上问题往往难以复现。

如果工程习惯不好，一个看起来很强的模型应用也可能很快变成难维护、难排查、难迭代的系统。

这一章讨论：入职后如何通过日常工程习惯建立长期可信度。

## 66.1 可信度来自稳定交付，而不是一次爆发

很多新人入职后希望快速证明自己，于是容易追求“大动作”：重构系统、替换框架、引入新模型、设计复杂架构。

这种心态可以理解，但风险也很高。

团队真正需要的是稳定交付。

稳定交付包括：

- 需求理解准确。
- 方案范围清楚。
- 进度预期可靠。
- 代码质量可维护。
- 风险提前暴露。
- 上线前验证充分。
- 出问题能快速定位。
- 复盘后能减少同类问题。

如果你每次都能把小需求做稳，团队会逐渐把更关键的任务交给你。

如果你偶尔做出一个很炫的 demo，但经常延期、遗漏边界、上线后难排查，团队对你的信任会下降。

大模型工程师尤其不能只靠 demo 建立信誉。

demo 能证明可能性，生产系统要证明可靠性。

## 66.2 接需求时先确认问题，而不是急着给方案

很多工程问题一开始并不清楚。

产品或业务方可能会说：

- “我们想做一个智能客服。”
- “希望回答更准确。”
- “能不能加一个 Agent？”
- “这个模型效果不好，换一个更强的。”
- “用户希望系统能自动处理工单。”

如果工程师立刻进入实现，很容易做偏。

正确的第一步是确认问题。

可以问：

- 当前具体失败案例是什么？
- 用户最不能接受的问题是哪类？
- 是回答错误、证据错误、拒答错误，还是流程失败？
- 有没有线上日志、用户反馈或人工标注样本？
- 目标是提升准确率、降低成本、减少人工、提高响应速度，还是改善体验？
- 成功标准是什么？
- 有哪些不能触碰的安全、合规或业务边界？

大模型项目里，“效果不好”不是一个可执行需求。

它需要被拆成可定位、可验证、可复盘的问题。

例如知识库问答效果不好，可能来自文档解析失败、chunk 切分不合理、召回不准、rerank 排序错误、prompt 指令冲突、模型不适合、引用生成错误或评测集偏差。

如果不先定位问题，直接换模型或加 Agent，只是在增加复杂度。

## 66.3 写方案时明确范围、假设和验证方式

可靠的工程师写方案，不只是描述“我要做什么”，还会说明“我不做什么”“我基于什么假设”“我如何证明有效”。

一个简洁但有效的方案可以包含：

- 背景：为什么要做？
- 目标：要改善什么指标或体验？
- 非目标：本次明确不解决什么？
- 方案：核心改动是什么？
- 影响范围：涉及哪些服务、数据、接口和流程？
- 风险：可能出什么问题？
- 验证：如何离线评测、灰度和线上观察？
- 回滚：失败后如何恢复？

非目标非常重要。

没有非目标，需求会不断膨胀，工程师会陷入模糊承诺。

例如“优化 RAG 效果”这个需求，可以明确本次只优化文档解析和 chunk 策略，不涉及模型替换、不涉及权限系统、不涉及多轮对话记忆。

假设也很重要。

例如你认为效果差主要来自召回不足，就应该说明这个判断来自哪些失败案例或统计数据。

如果后续实验发现假设不成立，就要及时调整，而不是继续把原方案做完。

## 66.4 进度沟通要暴露不确定性

很多工程师害怕暴露不确定性。

他们觉得说“不确定”“可能有风险”“需要先验证”会显得自己不够强。

实际情况相反。

在复杂项目中，能提前识别不确定性，是成熟工程师的重要标志。

进度沟通不要只说：

- “快好了。”
- “差不多了。”
- “应该没问题。”

这些说法没有信息量。

更好的表达是：

- “接口改造已完成，离线评测还没跑完，主要风险是 P95 延迟可能超过目标。”
- “当前准确率提升明显，但引用错误率没有下降，明天会先定位引用链路。”
- “方案 A 已验证不可行，因为召回阶段丢失了关键表格信息，准备转向文档解析优化。”
- “上线前还缺压测和回滚验证，如果今天要发版，风险偏高。”

这样的沟通会让团队知道真实状态，也能更早做决策。

可信度不是来自永远报喜，而是来自状态透明。

## 66.5 代码提交要让别人容易 review

大模型工程中，代码变更往往横跨 prompt、检索、模型调用、评测、配置、日志和前后端接口。

如果提交混乱，review 成本会非常高。

好的提交习惯包括：

- 一个提交解决一个清晰问题。
- 避免把重构、格式化和功能改动混在一起。
- PR 描述说明背景、改动、验证和风险。
- 对 prompt、阈值、模型参数等变更说明原因。
- 对评测结果给出样本和指标，而不是只说“效果更好”。
- 对高风险路径给出回滚方式。

大模型系统里，prompt 改动也是代码改动。

一个 prompt 多加一句话，可能影响拒答、引用、工具调用、输出格式和安全边界。

所以 prompt 变更也要 review，也要验证，也要记录原因。

不要把 prompt 当作临时文本。

它是生产行为的一部分。

## 66.6 日志和 trace 是工程生命线

大模型系统如果没有日志和 trace，线上问题会非常难查。

用户说“它答错了”，你需要知道：

- 用户原始输入是什么？
- 系统做了哪些 query rewrite？
- 检索到了哪些文档？
- rerank 分数是多少？
- 最终传给模型的上下文是什么？
- 使用了哪个模型和参数？
- 模型原始输出是什么？
- 是否触发了工具调用？
- 工具返回了什么？
- 是否经过安全过滤或格式修复？
- 最终返回给用户的内容是什么？

没有这些信息，排查只能靠猜。

日志和 trace 的设计要注意两点。

第一，足够定位问题。

关键链路必须可追踪，尤其是检索、prompt、工具调用、模型输出、后处理和错误分支。

第二，避免泄漏敏感信息。

用户隐私、企业文档、密钥、权限数据不能随意写入日志。必要时要脱敏、采样、加权限控制和保留周期。

高级工程师不会等线上事故发生后才补日志。

他会在设计系统时就考虑可观测性。

## 66.7 评测要进入日常开发流程

很多团队的大模型项目失败，不是因为没有调 prompt，而是因为没有稳定评测。

没有评测，所有优化都靠主观感受。

今天看起来变好了，明天另一个场景可能变差。

日常开发中至少应该有三类评测。

第一，回归评测。

每次改 prompt、模型、检索策略或工具逻辑，都要检查核心样本是否退化。

第二，目标评测。

针对本次需求定义指标，例如引用准确率、工单分类准确率、工具调用成功率、拒答准确率、延迟和成本。

第三，失败案例评测。

把线上出现的典型失败 case 回收进评测集，防止同类问题反复出现。

评测不一定一开始就很复杂。

早期可以是几十个高质量样本、人工标注标准和一个可重复运行的脚本。

比起追求完美评测，更重要的是让评测成为习惯。

## 66.8 上线前要检查失败路径

上线前不要只验证成功路径。

大模型系统有很多失败路径：

- 模型超时。
- 模型限流。
- 检索无结果。
- 检索结果权限不足。
- 工具调用参数错误。
- 工具执行失败。
- 模型输出格式不合法。
- 多轮对话上下文过长。
- 用户输入包含攻击指令。
- 外部服务返回异常。
- 成本突然升高。

很多线上事故不是主流程不会跑，而是失败路径没有处理。

上线前可以准备一个简单 checklist：

- 超时怎么办？
- 重试几次？
- 是否会重复执行副作用操作？
- 用户看到什么错误提示？
- 是否记录足够日志？
- 是否会泄漏内部信息？
- 是否有降级方案？
- 是否能快速回滚？

Agent 类系统尤其要注意副作用。

如果工具会发邮件、改数据库、下订单、提交工单，就不能简单重试，也不能让模型自由决定所有高风险动作。

需要有幂等、确认、权限和审计。

## 66.9 复盘不是追责，而是改系统

线上问题发生后，团队通常会复盘。

低质量复盘会停留在：

- “下次注意。”
- “加强测试。”
- “开发不够仔细。”

这些结论很难防止问题复发。

高质量复盘关注系统性改进。

可以问：

- 问题为什么没有在开发阶段发现？
- 评测集为什么没有覆盖？
- 日志为什么不足以快速定位？
- 灰度为什么没有拦住？
- 告警为什么没有及时触发？
- 回滚为什么不够快？
- 文档或 checklist 是否缺失？
- 哪个流程可以自动化？

复盘的产出应该是具体动作。

例如新增一个回归样本、补一个日志字段、增加一个发布门禁、修改一个工具权限、完善一个异常处理分支。

真正成熟的团队，会把事故转化为系统韧性。

真正成熟的工程师，会在复盘中推动机制改进，而不是只解释自己为什么没错。

## 66.10 文档是长期协作工具

很多工程师不喜欢写文档，觉得写文档浪费时间。

但在大模型工程中，文档非常重要。

因为系统行为不仅由代码决定，还由模型、prompt、数据、评测集、配置、阈值、工具权限和业务规则共同决定。

如果这些信息只存在于某个人脑子里，系统就很难维护。

有价值的文档包括：

- 系统架构图。
- 请求链路说明。
- prompt 设计说明。
- 模型和参数选择依据。
- 数据来源和清洗规则。
- 评测集构造方法。
- 指标定义。
- 上线和回滚流程。
- 常见问题排查手册。
- 重要设计决策记录。

文档不需要一开始很长。

关键是及时、准确、可更新。

如果一个需求做完后，别人无法通过文档理解系统行为，那么交付是不完整的。

## 66.11 不要过度工程，也不要欠债不还

工程习惯的难点在于平衡。

过度工程会让项目变慢。

技术债太多会让项目变脆。

大模型项目早期需要快速试错，因此不可能一开始就设计完美架构。

但也不能把所有 demo 代码直接带到生产。

可以用一个简单判断标准：

- 探索阶段，优先验证问题是否成立。
- 试点阶段，补齐基本日志、评测、权限和异常处理。
- 生产阶段，补齐稳定性、可观测性、回滚、成本控制和安全边界。

不同阶段的工程标准不同。

问题不在于早期写了临时代码，而在于临时代码进入生产后没有被治理。

成熟工程师会明确标记技术债，并在合适时机偿还。

例如为了快速验证先写死一个 prompt 可以接受，但上线前应该把版本、变更记录、评测结果和回滚方式补齐。

## 66.12 建立跨角色信任

大模型工程师通常要和很多角色协作：产品、算法、后端、前端、数据、测试、运维、安全、法务和业务专家。

长期可信度不只来自代码，也来自跨角色协作。

对产品，要能把技术不确定性转化为可决策的信息。

例如说明某个需求需要先做数据验证，而不是直接承诺上线时间。

对算法，要能提供高质量失败案例、日志和评测结果，而不是只说“模型不行”。

对数据团队，要能说明需要什么字段、标注标准和质量检查，而不是只说“数据越多越好”。

对运维和平台，要能说明容量、限流、监控、告警和回滚要求。

对安全和法务，要能说明数据流、权限边界、日志保留和高风险操作控制。

一个高级工程师的价值，往往体现在他能让不同角色围绕同一个问题形成共识。

## 66.13 个人可信度的负面清单

有些行为会快速消耗团队信任。

需要主动避免。

例如：

- 没弄清需求就承诺。
- 明知有风险但不说。
- 用“模型就是这样”解释所有问题。
- 没有评测就宣称效果提升。
- 改 prompt 不记录原因。
- 线上问题没有日志可查。
- 把 demo 代码当生产代码。
- 遇到事故只解释责任，不推动改进。
- 经常低估工作量。
- 频繁引入新框架但不负责维护。
- 不写文档，让系统依赖个人记忆。

这些问题单次看可能不严重，但长期会形成标签。

一旦团队认为你“不稳”，后续再证明自己会更难。

## 66.14 高可信工程师的日常表现

高可信工程师并不一定每天都做惊天动地的事情。

他的日常表现通常很朴素：

- 接需求时先澄清问题。
- 设计方案时明确目标和非目标。
- 开发时保持改动可 review。
- 改模型、prompt、检索策略时有评测。
- 上线前检查失败路径。
- 线上问题能通过 trace 定位。
- 风险会提前同步。
- 复盘会推动机制改进。
- 文档能让别人接手。
- 学到的新东西会转化为团队资产。

这些习惯短期看不显眼，但长期会形成很强的职业复利。

团队会更愿意让你负责关键系统，也更愿意听你的技术判断。

## 66.15 小结

入职后的长期成长，不只取决于技术知识，也取决于工程习惯。

大模型系统充满不确定性，因此更需要需求澄清、方案边界、评测、日志、trace、失败路径处理、上线验证、复盘和文档。

可信度来自稳定交付和状态透明，而不是一次漂亮 demo。

成熟的大模型工程师，会把每次开发都当作建立信任的机会：问题定义清楚，方案可验证，风险可见，代码可维护，线上可观测，失败可复盘。

长期来看，这些习惯会比单个技术点更能决定一个工程师的上限。
