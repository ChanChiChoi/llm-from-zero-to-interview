# 第79章：模拟面试训练与刻意练习

上一章讨论了求职材料包、作品集与公开证据。

材料准备好之后，还需要把它们转化成稳定的面试表现。

很多候选人不是不会，而是在面试现场讲不出来。

原因通常不是知识完全缺失，而是缺少模拟训练。

大模型工程师面试覆盖范围很广：原理、项目、系统设计、debug、实验评测、安全、成本、行为面和英语表达。

如果只靠临场发挥，很容易出现：

- 项目讲得太散。
- 系统设计没有结构。
- 被追问时慌乱。
- 指标和细节记不清。
- 英语表达卡顿。
- 行为面只讲经历，不讲判断。

这一章给出一套可执行的模拟面试训练方法。

目标不是背答案，而是建立稳定表达能力。

## 79.1 模拟面试训练什么

模拟面试不是随便找几个题问一问。

它应该覆盖六类能力。

第一，项目表达。

能否在 2 分钟、5 分钟、15 分钟三个长度下讲清一个项目。

第二，技术原理。

能否解释 Transformer、RAG、Agent、KV cache、LoRA、推理优化、评测等核心概念。

第三，系统设计。

能否从需求、约束、架构、数据流、指标、故障和演进讲方案。

第四，debug 和排查。

能否面对“效果下降”“延迟变高”“成本飙升”“工具误调用”给出排查路径。

第五，行为面。

能否讲冲突、失败、协作、推动落地、技术取舍和成长。

第六，反向提问。

能否通过问题判断团队是否适合自己。

这六类能力共同决定面试稳定性。

## 79.2 建立题库而不是临时刷题

建议建立自己的面试题库。

结构可以是：

```text
interview-bank/
  project.md
  fundamentals.md
  system-design.md
  debugging.md
  evaluation.md
  safety-cost.md
  behavioral.md
  english.md
  reverse-questions.md
```

每个问题至少包含：

- 问题。
- 标准回答结构。
- 关键术语。
- 可展开追问。
- 自己的项目例子。
- 常见错误。

不要只收藏题目。

题目没有答案结构，训练价值有限。

## 79.3 项目表达训练

每个核心项目至少准备三个版本。

第一，30 秒版本。

用于自我介绍或快速概述。

```text
我做过一个企业知识库 RAG 系统，主要负责检索质量和引用可信度。我们从文档解析、hybrid retrieval、rerank、引用校验和 bad case 回流建立闭环，把 citation accuracy 从 71% 提升到 86%，同时把 P95 延迟控制在 2.3 秒以内。
```

第二，2 分钟版本。

用于项目简介。

要讲清：背景、问题、方案、指标、我的角色。

第三，10 分钟版本。

用于深挖。

要展开：架构、数据、评测、上线、失败、取舍、安全和成本。

训练时可以录音。

听回放检查：

- 是否有背景。
- 是否有明确问题。
- 是否有指标。
- 是否讲清我的贡献。
- 是否过度堆术语。
- 是否能自然进入追问。

## 79.4 系统设计训练

系统设计题不要一上来画架构。

先问需求和约束。

一个稳定模板是：

```text
1. 明确需求：用户、场景、规模、SLO、安全、成本。
2. 定义成功指标：质量、延迟、吞吐、成本、风险。
3. 画核心链路：输入 -> 数据/工具/模型 -> 输出 -> 日志反馈。
4. 拆模块：API、数据、模型、评测、监控、权限、运维。
5. 讨论 trade-off：RAG vs 长上下文，API vs 自建，强模型 vs 小模型。
6. 设计上线策略：离线评测、灰度、监控、回滚。
7. 说明演进路线：MVP、生产化、平台化。
```

可以重点练这些题：

- 设计企业知识库 RAG 系统。
- 设计客服 Agent 系统。
- 设计大模型评测平台。
- 设计 OpenAI-compatible serving 平台。
- 设计 prompt 和模型版本管理系统。
- 设计多租户大模型应用平台。

每题至少练一次白板讲解。

## 79.5 Debug 训练

Debug 题考的是排查顺序。

不要直接猜原因。

例如面试官问：“RAG 系统最近回答质量下降，你怎么排查？”

可以这样答：

```text
我会先确认影响范围，是全局下降还是某个业务、某类文档、某个模型版本、某个时间段下降。然后拆链路看：用户 query 分布是否变化，文档更新和解析是否异常，embedding 或索引是否变更，retrieval recall 是否下降，rerank 是否变化，上下文拼接是否截断，prompt 或模型版本是否变更，最后看生成和引用校验。排查过程中会用固定评测集、线上 bad case 和日志 trace 对齐。
```

常练 debug 题：

- RAG 回答质量下降。
- Agent 工具误调用增加。
- TTFT 变高。
- TPOT 变高。
- token 成本突然上升。
- prompt 改动后格式错误增加。
- 模型切换后用户反馈变差。
- 越权召回事故。

Debug 回答要体现：先分范围，再拆链路，再看变更，再验证假设。

## 79.6 行为面训练

行为面不要只讲故事。

要讲你的判断。

推荐使用 STAR+R：

```text
Situation：背景是什么。
Task：你承担什么责任。
Action：你做了什么。
Result：结果是什么。
Reflection：你学到了什么，下次如何改进。
```

大模型工程师常见行为题：

- 讲一次你推动跨团队项目的经历。
- 讲一次技术方案被挑战的经历。
- 讲一次线上失败或事故。
- 讲一次你做技术取舍的经历。
- 讲一次你发现自己判断错了。
- 讲一次你如何处理不确定性。
- 讲一次你如何说服团队不做某个方案。

每个故事都要准备结果和反思。

没有反思的故事，容易显得不成熟。

## 79.7 英语面试训练

如果目标是外企或英文面试，英语表达要单独练。

重点不是口音完美。

重点是结构清楚。

建议准备这些英文模板：

- Self introduction。
- Project overview。
- Technical trade-off。
- System design clarification。
- Debugging approach。
- Behavioral stories。
- Asking reverse questions。

例如项目开场：

```text
One of my most relevant projects was an enterprise RAG system for internal knowledge search. My main responsibility was improving retrieval quality and citation reliability. The key challenge was that users could get fluent answers, but the supporting evidence was often weak or incorrect. I worked on document parsing, hybrid retrieval, reranking, citation verification, and bad-case feedback. As a result, citation accuracy improved from 71% to 86%, while P95 latency stayed within 2.3 seconds.
```

英语训练要录音。

检查是否：

- 句子太长。
- 术语表达不稳定。
- 逻辑连接词不足。
- 遇到追问就卡住。

## 79.8 一周模拟面试节奏

如果有一周集中准备，可以这样安排：

第一天：整理简历和项目一页纸。

第二天：练项目表达和行为面。

第三天：练技术原理。

第四天：练系统设计。

第五天：练 debug、评测、安全和成本题。

第六天：完整 mock interview。

第七天：复盘、补短板、整理反向提问。

每次训练结束必须写复盘：

- 哪个问题答得散。
- 哪个术语不熟。
- 哪个项目指标忘了。
- 哪个追问没有接住。
- 下一次如何改。

没有复盘的模拟面试，提升很有限。

## 79.9 面试当天状态管理

面试当天不要继续大量学习新内容。

重点是稳定状态。

建议：

- 提前检查网络、摄像头、麦克风。
- 准备简历、项目一页纸和反向问题。
- 复习核心项目 30 秒和 2 分钟版本。
- 准备纸笔记录问题。
- 遇到不会的问题先澄清，不要硬编。
- 回答过长时主动总结。
- 面试结束后立即复盘。

如果不会，可以说：

```text
这个细节我没有在生产中直接实现过，但我会从问题目标和系统约束来分析。我的理解是...
```

诚实加结构，比硬装更好。

## 79.10 小结

模拟面试训练的目标，是把知识转化成稳定表达。

大模型工程师面试不是只考背概念。

它考你能不能讲清项目、设计系统、排查问题、解释取舍、处理风险、表达成长。

有效训练要覆盖项目表达、原理、系统设计、debug、行为面、英语表达和反向提问。

每次模拟都要记录、复盘、修正。

真正的面试能力不是临场发挥，而是长期刻意练习形成的稳定输出。
