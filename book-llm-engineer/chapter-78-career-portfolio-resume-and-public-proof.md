# 第78章：求职材料包、作品集与公开证据

上一章讨论了面试复盘、offer 选择与谈薪。

这一章继续往前补一环：你如何让别人相信你真的具备大模型工程能力。

很多候选人准备面试时，只准备简历。

但在大模型岗位竞争中，只有简历往往不够。

因为简历很短，无法展示完整项目细节、工程判断、实验过程、失败复盘和真实取舍。

更好的方式是准备一套求职材料包。

它包括：

- 简历。
- 项目一页纸。
- 项目复盘文档。
- GitHub 仓库或代码样例。
- 技术博客。
- 面试题库和标准回答。
- 反向提问清单。
- 30-60-90 天入职计划。

这些材料共同构成你的职业证据链。

你不是只告诉面试官“我做过 RAG”或“我懂 Agent”。

你要让对方看到：你能定义问题、设计系统、做实验、上线验证、处理风险、总结复盘。

这一章讨论如何构建这套材料。

## 78.1 简历是入口，不是全部

简历的任务不是讲完所有东西。

简历的任务是让对方愿意继续了解你。

大模型工程师简历最常见的问题是堆技术名词。

例如：

```text
熟悉 LangChain、RAG、Agent、Prompt Engineering、LoRA、vLLM、SGLang、MCP、向量数据库。
```

这类描述信息密度很低。

因为它没有告诉别人你解决了什么问题。

更好的写法是：

```text
负责企业知识库 RAG 问答系统，从文档解析、chunk、hybrid retrieval、rerank、引用校验到线上反馈闭环；构建 500 条业务评测集，将 citation accuracy 从 71% 提升到 86%，P95 延迟控制在 2.3s 内。
```

简历要突出四件事：

- 问题。
- 方案。
- 指标。
- 影响。

技术栈可以写，但不能替代结果。

## 78.2 大模型简历的项目 bullet 模板

一个好的项目 bullet 可以按这个模板写：

```text
在什么场景下，为了解决什么问题，采用什么技术方案，做了什么关键设计，最终带来什么指标或业务结果。
```

例如 RAG 项目：

```text
面向企业内部知识问答场景，设计 RAG 质量优化闭环，覆盖文档解析、chunk 策略、hybrid retrieval、rerank、引用校验和 bad case 回流；构建 500 条人工标注评测集，将 answer correctness 提升 12%，citation accuracy 提升 15%。
```

Agent 项目：

```text
负责工单自动化 Agent 的工具调用链路，设计 tool schema、权限校验、参数验证、dry-run 和人工确认机制；将低风险工单自动处理率提升到 48%，高风险误调用率保持为 0。
```

推理服务项目：

```text
参与 LLM 推理服务性能优化，分析 TTFT、TPOT、KV cache 和 batching 瓶颈，调整 prefix cache、max batched tokens 和路由策略；在 P95 TTFT 不超过 1.8s 的约束下提升 output tokens/s 约 35%。
```

评测项目：

```text
搭建模型和 prompt 回归评测流程，沉淀业务样本、LLM-as-a-judge rubric、人工复核和上线门禁；将 prompt 变更导致的线上回退问题从每月 3 次降低到 0-1 次。
```

注意：指标必须真实。

没有线上指标时，可以写离线指标、样本规模、实验设置和 demo 约束。

不要编造数据。

## 78.3 项目一页纸

项目一页纸是简历和长文档之间的材料。

它适合面试前自我准备，也适合在面试后发给面试官或内推人。

结构可以是：

```text
项目名称：
场景背景：
核心问题：
我的角色：
系统架构：
关键技术：
指标结果：
主要取舍：
失败和改进：
面试可展开点：
```

例如：

```text
项目名称：企业知识库 RAG 问答系统
场景背景：客服和售前需要从内部文档中快速回答客户问题。
核心问题：回答经常缺少可靠引用，且文档更新后效果不稳定。
我的角色：负责检索质量、引用校验和评测闭环。
系统架构：文档解析 -> chunk -> embedding/hybrid retrieval -> rerank -> context assembly -> LLM -> citation verification -> feedback。
关键技术：hybrid retrieval、rerank、citation eval、bad case 分类、prompt 版本管理。
指标结果：citation accuracy 从 71% 到 86%，P95 latency 控制在 2.3s。
主要取舍：rerank 提升质量但增加延迟，因此只对低置信 query 启用。
失败和改进：表格文档解析仍是主要失败来源，后续计划引入结构化解析。
面试可展开点：RAG 评测、失败 case 分类、引用校验、延迟成本取舍。
```

一页纸的价值是帮助你在面试中稳定表达。

它也能帮助你发现项目故事是否完整。

如果一页纸写不出来，说明项目还没有被你真正理解和沉淀。

## 78.4 项目复盘文档

项目复盘比一页纸更详细。

适合沉淀成个人资产。

结构可以是：

- 背景。
- 问题定义。
- 需求和非需求。
- 系统架构。
- 技术方案。
- 数据和评测。
- 上线过程。
- 指标结果。
- 成本和风险。
- 失败 case。
- 复盘和下一步。

项目复盘要避免只写成功。

面试官往往更关心：

- 你遇到过什么失败？
- 你如何定位问题？
- 你如何选择方案？
- 你如何判断上线是否成功？
- 你如何处理安全、成本和延迟？

真实失败比完美故事更可信。

例如：

```text
最初我们以为 RAG 错误主要来自 embedding 召回不足，但分析 200 个 bad case 后发现，42% 的错误是正确 chunk 已经进入 top-20，但没有进入最终上下文。因此我们没有马上换 embedding 模型，而是先做 rerank 和 context assembly 优化。
```

这类复盘能体现工程判断。

## 78.5 GitHub 仓库如何展示

如果你有公开项目，GitHub README 很重要。

很多候选人的仓库只有代码，没有说明。

这会让面试官很难判断价值。

一个好的 README 应该包括：

- 项目解决什么问题。
- 架构图或流程图。
- 快速开始。
- 核心模块说明。
- 示例输入输出。
- 评测方法。
- 已知限制。
- 后续计划。

大模型项目尤其要写清：

- 使用什么模型。
- 数据来源是什么。
- 是否支持本地或 API。
- prompt 在哪里。
- 如何运行评测。
- 如何复现实验。
- 安全和隐私注意事项。

不要只展示一个 demo UI。

要展示工程链路。

例如一个 RAG 仓库可以包含：

```text
README.md
docs/architecture.md
docs/evaluation.md
docs/failure_cases.md
configs/
src/ingestion/
src/retrieval/
src/rerank/
src/generation/
src/evaluation/
examples/
```

项目不一定要很大。

但要可读、可运行、可解释。

## 78.6 技术博客写什么

技术博客不是为了刷存在感。

它是把你的经验公开沉淀成证据。

适合写的主题包括：

- 一次 RAG bad case 分析。
- 一个 Agent 工具调用安全设计。
- 一次模型切换评测。
- 一次推理延迟优化复盘。
- 一个 prompt 版本管理方案。
- 一个 LLM-as-a-judge 评测模板。
- 一个成本归因 dashboard 设计。
- 一个 30-60-90 天成长复盘。

博客不需要追求“大而全”。

一篇具体、真实、有细节的小文，比泛泛而谈的“万字综述”更有价值。

好的技术博客应该回答：

- 问题是什么？
- 为什么重要？
- 你试过哪些方案？
- 哪些失败了？
- 最后选择了什么？
- 指标如何变化？
- 有什么限制？

这和面试表达是同一套能力。

## 78.7 面试材料包如何组织

建议建立一个私有目录，结构如下：

```text
career-package/
  resume/
    resume-llm-engineer.pdf
    resume-ai-infra.pdf
  projects/
    rag-one-pager.md
    agent-one-pager.md
    inference-one-pager.md
    evaluation-retrospective.md
  interview/
    project-stories.md
    system-design-answers.md
    behavior-answers.md
    reverse-questions.md
  company-notes/
    company-a.md
    company-b.md
  offer/
    offer-comparison.md
```

每次面试后更新。

每次项目完成后更新。

每次复盘后更新。

这套材料会越来越值钱。

它能让你在下一次求职时不再从零开始。

## 78.8 不同岗位的材料侧重点

如果你投大模型应用工程师，重点展示：

- RAG。
- Agent。
- 工具调用。
- 评测。
- 上线和监控。
- 用户反馈。

如果你投 AI Infra 或推理工程，重点展示：

- 推理服务。
- batching。
- KV cache。
- 性能分析。
- 容量规划。
- 成本治理。

如果你投算法工程师，重点展示：

- 模型原理。
- 训练和微调。
- 数据构造。
- 实验设计。
- 指标分析。

如果你投资深或负责人岗位，重点展示：

- 技术路线图。
- 跨团队协作。
- 风险治理。
- 机制建设。
- 业务影响。

同一个人可以准备不同版本材料。

不要用同一份简历和同一套项目讲法投所有岗位。

## 78.9 公开表达要注意边界

公开项目和博客要注意保密。

不要泄露：

- 公司内部数据。
- 客户信息。
- 业务指标的敏感细节。
- 内部 prompt。
- 生产架构细节。
- 安全漏洞细节。
- 未公开模型或项目。

可以做脱敏：

- 改写业务背景。
- 使用模拟数据。
- 使用相对指标。
- 隐去公司和客户名称。
- 只讲方法，不暴露内部实现。

公开表达的目标是展示能力，而不是泄露信息。

如果不确定能不能公开，宁可写成私有面试材料。

## 78.10 面试表达

如果面试官问：“你如何证明自己具备大模型工程能力？”

可以这样回答：

```text
我会用项目证据而不是只列技术栈来证明。比如在简历中，我会写清楚我解决的具体问题、采用的方案、我的角色、指标变化和业务影响。对于核心项目，我会准备一页纸和更详细的复盘文档，覆盖背景、架构、评测、上线、风险、失败 case 和后续改进。

如果有公开项目，我会在 GitHub README 中写清问题、架构、快速开始、评测方法和已知限制，而不是只放代码。技术博客方面，我更倾向于写具体复盘，比如一次 RAG 引用准确率优化、一次 Agent 工具调用安全设计或一次推理延迟优化，而不是泛泛而谈。

我认为大模型工程能力的证据链应该包括：能定义问题，能设计系统，能做实验，能上线验证，能处理风险成本，能总结复盘。简历只是入口，项目复盘、代码、文档和面试表达共同证明能力。
```

## 78.11 常见误区

第一个误区是简历堆术语。

熟悉很多工具，不等于解决过真实问题。

第二个误区是项目没有指标。

没有指标，面试官很难判断效果。

第三个误区是 GitHub 只有代码没有 README。

代码再好，没人看懂也很难产生价值。

第四个误区是博客只写概念综述。

真实项目复盘通常更有说服力。

第五个误区是所有岗位用同一套材料。

应用、算法、推理、Infra、负责人岗位关注点不同。

第六个误区是公开泄露敏感信息。

展示能力不能以违反保密为代价。

第七个误区是材料不更新。

项目做完不复盘，面试后不沉淀，长期会反复从零准备。

## 78.12 小结

求职材料包的核心，是把你的能力变成可验证证据。

简历是入口。

项目一页纸帮助你稳定表达。

项目复盘展示你的工程判断。

GitHub README 展示你的工程可读性。

技术博客展示你的思考和总结能力。

面试题库和复盘记录帮助你持续迭代。

大模型岗位竞争激烈，单靠“熟悉某某技术栈”已经不够。

真正有说服力的是：你能讲清楚自己解决过什么问题，如何设计系统，如何验证效果，如何处理失败，如何权衡成本、安全和业务价值。

把这些材料持续沉淀下来，你的每一次项目、每一次面试、每一次复盘，都会变成下一次机会的资产。
