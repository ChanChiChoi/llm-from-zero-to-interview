# 第76章：30-60-90 天成长计划与行动清单

上一章讨论了长期成长与职业护城河。

长期成长很重要，但真正落地时，很多人需要的是更具体的行动计划。

例如：

- 刚入职一家大模型团队，前三个月应该做什么？
- 从传统后端或算法岗位转到大模型工程，如何快速进入状态？
- 晋升为技术负责人后，如何建立影响力？
- 接手一个混乱的 RAG、Agent 或推理项目，如何不被细节淹没？
- 面试通过后，如何在试用期内证明自己？

这类问题适合用 30-60-90 天计划来回答。

30-60-90 天计划不是死板日程表。

它是一种帮助你建立节奏、对齐预期、识别风险、持续交付的工具。

这一章给出大模型工程师的 30-60-90 天成长计划。

你可以用于新入职、转岗、接手新项目、晋升技术负责人，或准备面试中的“入职后你会怎么做”问题。

## 76.1 30-60-90 天计划的目的

30-60-90 天计划有三个目的。

第一，减少混乱。

新环境中信息很多。

模型、数据、业务、代码、平台、评测、安全、组织关系都需要了解。

如果没有计划，很容易每天都很忙，但没有方向。

第二，对齐预期。

你需要和 manager、mentor、项目 owner、产品、安全、平台等相关方确认：前三个月最重要的目标是什么。

第三，建立可信度。

新加入团队时，不需要一开始就做惊天动地的大项目。

更重要的是快速学习、可靠交付、清楚沟通、暴露风险、形成闭环。

30-60-90 天计划的核心不是“我要证明我很强”。

而是：

```text
我会先理解系统，再找到可控切入点，逐步交付结果，最后形成可持续影响。
```

## 76.2 前 30 天：理解系统和建立关系

前 30 天不要急着重构。

也不要急着证明原系统很差。

你的目标是理解上下文。

大模型项目的上下文包括：

- 业务目标。
- 用户场景。
- 当前系统架构。
- 模型和数据链路。
- 评测方法。
- 线上指标。
- 已知问题。
- 历史决策。
- 团队分工。
- 安全和合规约束。

这一阶段最重要的是听、读、问、记录。

## 76.3 前 30 天的具体行动

第一，读文档。

优先读：

- 项目 PRD。
- 技术设计文档。
- 架构图。
- 模型选型记录。
- 评测报告。
- 事故复盘。
- prompt 变更记录。
- 成本和容量文档。
- 安全评审记录。

第二，跑通系统。

至少要跑通：

- 本地开发环境。
- 一个核心 API。
- 一个核心离线评测脚本。
- 一个线上日志或 trace 查询。
- 一个完整请求链路。

第三，画出请求生命周期。

例如 RAG 系统可以画：

```text
user query
  -> auth
  -> query rewrite
  -> retrieval
  -> rerank
  -> context assembly
  -> LLM generation
  -> citation verification
  -> response
  -> logging / feedback
```

Agent 系统可以画：

```text
user goal
  -> planner
  -> tool selection
  -> permission check
  -> tool execution
  -> observation
  -> reflection / next step
  -> final response
  -> audit
```

推理服务可以画：

```text
request
  -> API gateway
  -> tokenizer
  -> scheduler
  -> prefill
  -> KV cache
  -> decode
  -> sampler
  -> streaming
  -> metrics
```

第四，建立关系。

至少约 1:1：

- manager。
- mentor。
- 产品负责人。
- 项目技术 owner。
- 平台或 SRE。
- 数据负责人。
- 安全或合规负责人。
- 主要业务方。

每次 1:1 可以问：

- 当前项目最重要目标是什么？
- 最大风险是什么？
- 哪些问题反复出现？
- 哪些历史决策需要了解？
- 如果我前三个月只能做好一件事，你希望是什么？

第五，整理学习笔记。

建议形成一份 `Onboarding Notes`：

- 系统架构。
- 核心术语。
- 服务列表。
- 数据表或索引。
- 评测集。
- 常见故障。
- 常用命令。
- 待确认问题。

这份笔记未来可以变成团队 onboarding 文档。

## 76.4 前 30 天的产出

前 30 天不一定要交付大功能。

但应该有可见产出。

建议至少完成：

- 一张系统请求生命周期图。
- 一份 onboarding notes。
- 一次小 bug 修复或小功能改动。
- 一次评测脚本或日志查询跑通记录。
- 一份当前问题清单。
- 一份 60 天计划草案。

这些产出证明你不是只在“熟悉业务”。

你已经开始把混乱信息整理成可复用材料。

## 76.5 第 31-60 天：找到切入点并交付闭环

第 31-60 天，目标从理解转向交付。

但仍然不要急着做大重构。

你应该选择一个可控切入点。

好的切入点通常满足：

- 问题真实。
- 范围可控。
- 有明确 owner。
- 可以量化效果。
- 风险可回滚。
- 能帮助你理解更大系统。

例如：

- 补一个关键日志字段。
- 修复一类 RAG 引用错误。
- 为 Agent 工具调用增加参数校验。
- 给 prompt 增加版本记录。
- 建立一个小型失败 case 分类表。
- 优化一个高频请求的 token 成本。
- 给推理服务增加 TTFT/TPOT 分桶 dashboard。

这些事情看起来不大。

但它们能建立可信度。

可信度来自闭环，而不是规模。

## 76.6 第 31-60 天的具体行动

第一，选择一个小闭环项目。

写清楚：

- 背景。
- 问题。
- 目标。
- 非目标。
- 方案。
- 指标。
- 风险。
- 回滚方式。

第二，建立 baseline。

任何改动前先知道当前状态。

例如：

- 当前 citation accuracy 是多少？
- 当前 P95 TTFT 是多少？
- 当前工具参数错误率是多少？
- 当前单次请求平均 token 成本是多少？
- 当前失败 case 中哪一类最多？

第三，做小实验。

不要一开始全量上线。

可以先做离线评测、小流量灰度、内部用户试点或只覆盖某类场景。

第四，交付并复盘。

交付后写一份短总结：

- 做了什么。
- 指标如何变化。
- 哪些预期成立。
- 哪些预期不成立。
- 后续建议是什么。

第五，继续建立关系。

第 31-60 天要从“认识人”进入“协作做事”。

主动让相关方看到你的思考和交付。

## 76.7 第 31-60 天的产出

建议完成：

- 一个小闭环项目。
- 一份 baseline 和实验结果。
- 一份失败 case 或指标分析。
- 一次设计评审或方案文档。
- 一次上线后复盘。
- 一份下一阶段优先级建议。

如果能做到这些，你在团队中的定位会从“新人”变成“可靠协作者”。

## 76.8 第 61-90 天：扩大影响并建立机制

第 61-90 天，目标是从单点交付走向机制化影响。

这时你已经理解了系统，也完成了一个小闭环。

下一步要问：

- 哪些问题不是一次性修复，而需要机制？
- 哪些经验可以沉淀成文档、工具或流程？
- 哪些指标应该持续监控？
- 哪些技术债需要进入路线图？
- 哪些协作关系需要固定下来？

例如你修复了一类引用错误。

第 90 天前，你不只是修 bug。

你应该推动：

- 引用错误进入评测集。
- citation accuracy 进入 dashboard。
- prompt 或 parser 变更进入回归测试。
- 失败 case review 建立固定节奏。

这就是从交付功能走向建立机制。

## 76.9 第 61-90 天的具体行动

第一，总结前三个月观察。

写一份 `First 90 Days Review`。

包括：

- 我理解的业务目标。
- 当前系统架构。
- 核心指标。
- 最大风险。
- 已完成工作。
- 发现的问题。
- 推荐的下一步路线图。

第二，提出一个中期改进方向。

不要提出十个方向。

选择一个最重要的方向。

例如：

- 建立 RAG 质量闭环。
- 建立 Agent 工具调用安全门禁。
- 建立推理成本归因。
- 建立模型切换评测门禁。
- 建立线上失败 case 回流机制。

第三，把个人经验转成团队资产。

可以沉淀：

- onboarding 文档。
- debug checklist。
- 评测模板。
- 上线 checklist。
- prompt 变更规范。
- 工具接入规范。
- 成本分析模板。

第四，和 manager 对齐下一阶段目标。

问题包括：

- 我前三个月哪些地方做得有效？
- 哪些地方需要调整？
- 接下来我应该承担什么范围？
- 我的成长目标是什么？
- 哪些项目最能体现我的价值？

第五，明确自己的角色定位。

你可能定位为：

- RAG 质量 owner。
- Agent 工具链 owner。
- 推理服务性能 owner。
- 评测平台 owner。
- 数据质量 owner。
- 安全治理推动者。

角色定位不一定是正式头衔。

但它能帮助团队知道什么问题可以找你。

## 76.10 第 61-90 天的产出

建议完成：

- 一份 90 天复盘文档。
- 一份下一阶段路线图建议。
- 一个可复用团队资产。
- 一个机制化改进提案。
- 一次 manager 对齐会。
- 一个明确的后续 owner 方向。

这会让你从“可靠协作者”进一步变成“可以负责一个问题的人”。

## 76.11 不同角色的 30-60-90 重点

如果你是初级大模型工程师，重点是：

- 跑通环境。
- 理解链路。
- 修小问题。
- 写清楚笔记。
- 建立工程基本可信度。

如果你是中级工程师，重点是：

- 独立负责小闭环。
- 建立 baseline。
- 做实验。
- 上线和复盘。
- 理解系统 trade-off。

如果你是高级工程师，重点是：

- 识别关键瓶颈。
- 推动跨团队协作。
- 设计中期方案。
- 建立机制。
- 帮团队减少混乱。

如果你是技术负责人，重点是：

- 对齐业务目标。
- 诊断系统和组织瓶颈。
- 明确优先级。
- 建立指标和节奏。
- 形成路线图。

同样是 90 天，不同级别的目标不同。

不要用高级工程师的标准要求新人。

也不要用新人标准要求自己停留在熟悉环境。

## 76.12 面试表达

如果面试官问：“如果你加入我们团队，前三个月会怎么做？”

可以这样回答：

```text
我会按 30-60-90 天拆。前 30 天重点是理解系统和建立关系，不急着做大改动。我会阅读 PRD、设计文档、评测报告、事故复盘和安全评审，跑通本地环境、核心 API、评测脚本和线上日志查询，并画出一个请求从用户输入到模型调用、工具或检索、再到响应和日志的完整生命周期。同时和 manager、产品、平台、安全、数据和项目 owner 做 1:1，了解当前目标、历史决策和最大风险。

第 31-60 天，我会选择一个可控的小闭环项目，例如补关键日志字段、修复一类 RAG 引用错误、增加工具参数校验、优化 token 成本或建立一个失败 case 分类。这个项目要有 baseline、指标、灰度或回滚方式。目标不是一开始做很大，而是通过一个闭环建立可信度。

第 61-90 天，我会把前面的学习和交付沉淀成机制。比如写 90 天复盘，提出下一阶段路线图建议，把 debug checklist、评测模板、prompt 变更规范或工具接入规范沉淀成团队资产，并和 manager 对齐后续我应该承担的 owner 方向。

我的原则是先理解上下文，再做可控交付，最后把个人经验变成团队机制。
```

## 76.13 常见误区

第一个误区是前 30 天急着重构。

没有理解历史约束就重构，容易破坏已有系统。

第二个误区是只学习不产出。

熟悉业务也要有可见产出，比如图、笔记、小修复或问题清单。

第三个误区是接太大的项目。

新人前 60 天更适合可控闭环，而不是高风险大改造。

第四个误区是不建立 baseline。

没有 baseline，就无法证明改动有效。

第五个误区是只和直属团队沟通。

大模型项目通常跨产品、平台、数据、安全和业务。

第六个误区是交付后不复盘。

没有复盘，经验不会沉淀。

第七个误区是没有和 manager 对齐预期。

你以为重要的事情，未必是团队当前最需要的。

## 76.14 小结

30-60-90 天计划，是把长期成长落到短期行动的工具。

前 30 天，重点是理解系统、建立关系、跑通链路、整理上下文。

第 31-60 天，重点是选择一个可控切入点，建立 baseline，交付小闭环。

第 61-90 天，重点是扩大影响，把个人经验沉淀成团队机制，并明确后续 owner 方向。

大模型工程师的前三个月，不需要靠激进重构证明自己。

更可靠的路径是：理解上下文，尊重已有系统，快速形成可见产出，持续对齐预期，再逐步扩大影响。

真正的成长不是前三个月做了多少事，而是前三个月之后，团队是否更清楚问题，系统是否更可观察，风险是否更透明，下一步是否更有方向。
