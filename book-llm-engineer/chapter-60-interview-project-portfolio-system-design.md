# 第60章：面试中的项目表达、作品集与系统设计

上一章讲的是如何准备大模型工程面试：面试官真正想确认的，不只是你是否知道 RAG、Agent、微调、评测这些词，而是你是否能把一个真实问题拆开，判断约束，做出取舍，并把系统稳定地交付出来。

这一章更进一步：当你已经有项目经历、练习作品、开源仓库或课程项目时，面试中应该怎样讲，才能让对方相信你真的做过、想过、踩过坑，而不是只是在简历上堆关键词。

很多候选人并不是没有能力，而是表达方式让能力被低估。他们会说：

> 我做了一个基于 LangChain 的 RAG 问答系统，使用向量数据库进行检索，接入了 OpenAI API，并实现了多轮对话。

这句话听起来完整，但对面试官来说信息量很低。因为它没有回答这些更关键的问题：

- 这个系统解决了谁的问题？
- 为什么需要 RAG，而不是直接把内容塞进 prompt？
- 文档如何切分？检索效果如何判断？
- 答案错了怎么发现？怎么降低幻觉？
- 延迟、成本、权限、更新频率有什么约束？
- 你在其中负责了什么？做过哪些决策？

面试中的项目表达，本质上不是“报技术栈”，而是“证明工程判断力”。

## 60.1 项目表达的核心：从技术栈转向决策链

一个成熟的项目讲法，应该围绕一条决策链展开：

> 问题是什么 -> 约束是什么 -> 方案怎么选 -> 关键实现是什么 -> 怎么验证效果 -> 遇到什么问题 -> 如何改进 -> 最终结果如何

这条链条比任何技术名词都重要。

例如，不建议这样讲：

> 我用 LangChain、Chroma 和 GPT-4 做了一个知识库问答系统。

可以改成这样：

> 我做的是一个内部文档问答系统，目标是让新人能快速查询部署流程和故障处理手册。最初我们尝试直接把相关文档塞进 prompt，但文档数量增长后上下文长度和成本都不可控，所以改成 RAG。实现上我把文档按标题层级和段落长度混合切分，保留章节路径作为 metadata，检索时先用向量召回，再按文档类型和更新时间做过滤。评测上我整理了 80 个真实问题，关注命中率、答案可追溯性和拒答能力。后来发现很多错误来自 chunk 边界丢失上下文，所以增加了相邻 chunk 拼接和标题前缀，准确率明显提升。

这段表达有几个特点：

- 有明确业务场景，不是抽象 demo。
- 有方案演进，不是一上来就堆框架。
- 有约束意识，提到了上下文长度、成本、文档增长。
- 有实现细节，讲到了切分、metadata、召回、过滤。
- 有评测意识，不只是“效果不错”。
- 有问题和改进，说明你真的调过系统。

面试官听到这类表达，会更愿意继续追问。因为它显示你不是只会照教程搭一个链路，而是理解系统为什么这样设计。

## 60.2 面试中讲项目的推荐结构

一个项目可以准备三个版本：30 秒版、2 分钟版、深入追问版。

### 30 秒版：让对方知道这是什么

30 秒版适合简历开场，目标是快速建立上下文。

模板如下：

> 这个项目是为了解决【用户/业务问题】。我主要负责【你的职责】。系统采用【核心方案】，关键难点是【一个主要挑战】。最后通过【评测或结果】验证效果。

示例：

> 这个项目是一个面向客服团队的知识库问答系统，目的是减少人工查询产品文档的时间。我主要负责 RAG 检索链路和回答评测。系统采用文档切分、向量召回、规则过滤和引用溯源的方式，关键难点是文档版本多、问题表达不稳定。最后我们用真实客服问题集做评测，重点看答案命中率和引用正确率。

这个版本不需要讲太细，重点是让面试官知道后面可以追问什么。

### 2 分钟版：展示完整工程思考

2 分钟版适合面试官说“详细讲讲这个项目”。

可以按下面顺序讲：

1. 背景：为什么要做这个项目。
2. 目标：要优化什么指标或体验。
3. 约束：数据、成本、延迟、权限、准确性有什么限制。
4. 方案：为什么选择当前架构。
5. 实现：你负责的关键模块。
6. 评测：如何判断它是否有效。
7. 问题：上线或测试中遇到什么坑。
8. 改进：你如何定位和解决。

注意，不要平均用力。真正能体现水平的是“约束、取舍、评测、问题定位”。

技术栈只需要带过，除非面试官追问。

### 深入追问版：准备可展开的细节

深入追问版不是背一段长稿，而是提前准备几个可以展开的细节模块。例如 RAG 项目可以准备：

- 文档解析：PDF、Markdown、网页、表格如何处理。
- 切分策略：固定长度、标题层级、语义切分的取舍。
- 检索策略：向量检索、关键词检索、混合检索、rerank。
- 上下文构造：如何排序、去重、压缩、拼接。
- 幻觉控制：引用、拒答、答案约束、置信度判断。
- 评测方法：测试集来源、指标、人工标注、回归测试。
- 工程问题：缓存、并发、延迟、成本、权限隔离。

Agent 项目可以准备：

- 任务边界：什么任务适合 Agent，什么任务不适合。
- 工具设计：工具参数、返回格式、错误处理。
- 状态管理：短期记忆、长期记忆、任务上下文。
- 规划方式：单步工具调用、多步计划、反思机制。
- 安全控制：权限、确认步骤、危险操作拦截。
- 可观测性：日志、trace、中间步骤记录。
- 失败兜底：重试、降级、人工接管。

微调项目可以准备：

- 数据来源：如何清洗、去重、标注、划分训练集。
- 训练目标：风格对齐、任务能力、格式稳定性还是领域适配。
- 方法选择：prompt、RAG、LoRA、全量微调的取舍。
- 训练配置：模型、学习率、batch size、epoch、上下文长度。
- 评测方式：自动指标、人工评测、线上 A/B、错误分析。
- 风险控制：过拟合、遗忘、数据泄漏、安全输出。

提前准备这些模块的目的，是让你在被追问时能自然展开，而不是只能重复“我用了某某框架”。

## 60.3 作品集：不要只放代码，要放证据

大模型方向的作品集，不一定非要是很大的项目。一个小而完整的项目，只要证据链充分，也能体现能力。

作品集最常见的问题是：仓库里只有代码，没有说明。面试官打开后不知道这个项目解决什么问题，也不知道怎么运行，更看不到你的思考过程。

一个好的作品集 README 至少应该包含：

- 项目背景：这个项目解决什么问题。
- 核心功能：用户能做什么。
- 架构图或流程图：请求如何流动。
- 快速开始：如何安装、配置、运行。
- 示例输入输出：展示真实效果。
- 关键设计：为什么这样切分、检索、调用模型或设计工具。
- 评测结果：测试集、指标、错误样例。
- 已知限制：当前方案哪些地方还不够好。
- 后续改进：如果继续做，会优先改什么。

这里最重要的是“证据”。

什么是证据？

- 有真实样例，而不是只写“支持智能问答”。
- 有评测表格，而不是只写“准确率较高”。
- 有错误分析，而不是只展示成功 case。
- 有运行说明，而不是只给一堆脚本。
- 有设计取舍，而不是只列依赖包。

例如，一个 RAG 作品集可以包含这样的评测表：

| 问题类型 | 样例数量 | Top-3 命中率 | 答案可引用率 | 主要错误 |
| --- | ---: | ---: | ---: | --- |
| 概念查询 | 30 | 90% | 87% | 相似术语混淆 |
| 操作流程 | 25 | 84% | 80% | chunk 跨步骤断裂 |
| 故障排查 | 20 | 75% | 70% | 日志关键词缺失 |
| 权限相关 | 15 | 82% | 78% | 版本差异导致回答过期 |

这样的表格不一定要非常复杂，但它能说明你知道系统哪里强、哪里弱。

相比之下，如果 README 只写：

> 使用 LangChain + FAISS + Streamlit 实现本地知识库问答。

面试官很难判断你的能力边界。

## 60.4 项目不大也可以讲深

很多同学会担心：我没有大厂项目，没有线上千万级流量，也没有训练过百亿参数模型，是不是没什么可讲？

其实初中级大模型工程岗位更关注的是：你是否具备把问题做完整的能力。

一个小项目，只要做得闭环，也可以讲得很有价值。

例如，你做了一个“论文问答助手”。这听起来很普通，但如果你深入处理下面的问题，它就不再只是 demo：

- PDF 中标题、公式、表格、参考文献如何解析。
- 长论文如何切分，是否保留章节结构。
- 用户问“这篇论文的核心贡献是什么”时，如何聚合多个段落。
- 用户问“实验结果如何”时，如何定位表格和实验部分。
- 模型回答是否要求引用页码或段落。
- 如何评测摘要类问题和事实类问题。
- 多篇论文之间如何比较。

你不需要每一点都做到完美，但你需要知道这些问题存在，并且能说清楚你当时为什么优先解决其中几项。

面试官更愿意听到：

> 我这个项目规模不大，但我重点做了 PDF 解析、章节级切分和引用溯源。没有做复杂 Agent，因为任务主要是检索和总结，多步工具调用的收益不明显。后来我发现表格问答效果差，所以把表格单独抽取成 Markdown，并在 metadata 里标记 table 类型，检索时对实验类问题提高表格 chunk 的优先级。

这比泛泛地说“我做了论文助手，支持上传 PDF 和智能问答”强得多。

项目小不是问题，问题是讲不出决策。

## 60.5 系统设计题：先问清楚边界

大模型工程面试中，系统设计题越来越常见。题目可能是：

- 设计一个企业知识库问答系统。
- 设计一个代码助手。
- 设计一个客服 Agent。
- 设计一个多模态内容审核系统。
- 设计一个面向销售的自动线索分析工具。

很多候选人一听题目，就立刻开始画组件：前端、后端、向量数据库、LLM、缓存、消息队列。这样容易显得像在套模板。

正确做法是先问清楚边界。

以“设计一个企业知识库问答系统”为例，应该先确认：

- 用户是谁：员工、客服、研发、外部客户？
- 数据来源是什么：PDF、网页、飞书/Notion、数据库、工单？
- 数据规模多大：几百篇文档还是千万级文档？
- 更新频率如何：每天更新、实时同步还是手动上传？
- 权限要求如何：是否不同用户只能看不同文档？
- 准确性要求如何：允许概括，还是必须严格引用？
- 延迟要求如何：2 秒内、10 秒内，还是异步生成也可以？
- 成本限制如何：每次查询预算多少？
- 是否需要多轮对话、追问、任务执行？

这些问题不是拖延时间，而是在定义系统。

如果不问权限，可能设计出一个会泄露内部文档的系统。如果不问更新频率，可能索引同步方案完全不合适。如果不问准确性要求，可能不知道是否需要引用和拒答。

系统设计不是“把所有组件都放上去”，而是“在约束下做合理取舍”。

## 60.6 一个 RAG 系统设计的表达框架

设计 RAG 系统时，可以按六层来讲：

1. 数据接入层：文档从哪里来，如何解析、清洗、去重、权限标记。
2. 索引构建层：如何切分、生成 embedding、写入向量库和关键词索引。
3. 查询理解层：是否需要改写 query、识别意图、识别用户权限。
4. 检索排序层：向量召回、关键词召回、metadata 过滤、rerank。
5. 生成回答层：构造上下文、控制引用、拒答、格式约束。
6. 评测监控层：离线评测、线上反馈、日志、成本、延迟、错误分析。

面试时可以画成一条链路：

```text
数据源 -> 解析清洗 -> chunk + metadata -> embedding/index
用户问题 -> query 改写 -> 权限过滤 -> hybrid retrieval -> rerank
top chunks -> prompt 构造 -> LLM 生成 -> 引用/拒答 -> 日志与评测
```

但只画链路还不够，必须讲关键取舍。

例如：

- 如果文档规模小，可以先用简单向量检索，不必一开始上复杂分布式架构。
- 如果术语精确匹配很重要，应考虑 BM25 + 向量混合检索。
- 如果权限严格，应在检索前或检索中做权限过滤，而不是生成后再过滤。
- 如果答案必须可审计，应要求模型基于引用回答，并在无证据时拒答。
- 如果文档频繁更新，应设计增量索引和版本管理。
- 如果成本敏感，应控制 top-k、上下文长度、模型级别和缓存策略。

这些取舍比组件名称更能体现系统设计能力。

## 60.7 Agent 系统设计：重点不是“会调用工具”

Agent 系统设计题经常被讲得过于玄学。很多人会说：

> 我会让 Agent 自主规划任务，调用多个工具，最后完成目标。

这句话风险很高。因为在真实工程里，Agent 最大的问题不是“能不能调用工具”，而是：

- 工具调用是否可靠。
- 错误是否可恢复。
- 权限是否可控。
- 中间状态是否可观察。
- 用户是否知道 Agent 做了什么。
- 危险操作是否需要确认。
- 失败时是否有兜底方案。

设计客服 Agent 时，可以这样讲：

1. 先把任务分级：只读查询、建议生成、需要修改订单的操作、需要人工审批的操作。
2. 对只读查询，可以让 Agent 自动调用知识库和订单查询工具。
3. 对修改类操作，工具必须有明确 schema，并且执行前展示操作摘要让用户确认。
4. 所有工具调用记录 trace，包括输入、输出、耗时和错误。
5. 如果模型连续两次无法完成，或用户情绪强烈，转人工。
6. 对高风险工具设置权限和白名单，不能让模型自由拼接任意 SQL 或任意 HTTP 请求。

这类表达会让面试官觉得你有生产意识。

Agent 的核心不是“自主”，而是“可控地自动化”。

## 60.8 被问到不会的问题怎么办

面试中被问到不会的问题很正常。关键是不要假装。

不建议这样回答：

> 这个我了解过，但具体细节有点忘了。

这句话太空。

更好的方式是把你知道的边界说清楚，然后给出推理路径：

> 我没有在生产环境实现过这个方案，但我理解它要解决的问题是……如果让我设计，我会先从……开始验证。这里我不确定的点是……，需要通过实验或查文档确认。

例如被问到“你如何设计一个 reranker 的训练流程”，如果你没做过，可以说：

> 我没有完整训练过 reranker，但我理解 reranker 的目标是对召回候选进行更精细排序。数据上需要 query、正样本文档和负样本文档，负样本可以从 BM25 或向量召回的 hard negatives 里采。训练后不能只看 pairwise accuracy，还要看它接到 RAG 后对最终答案引用命中率和延迟的影响。我不确定具体 loss 和模型选型，需要根据 cross-encoder、bi-encoder 或轻量模型的成本要求再定。

这个回答承认了边界，但仍然展示了工程推理能力。

面试官通常不要求你什么都做过，但会观察你面对未知问题时是否诚实、是否能拆解、是否知道如何验证。

## 60.9 简历上的项目应该怎么写

简历项目描述不要写成技术栈清单，而要写成“动作 + 方法 + 结果”。

弱表达：

> 基于 LangChain 和 FAISS 实现智能问答系统，支持文档上传、多轮对话和知识库检索。

更好的表达：

> 设计并实现企业文档 RAG 问答系统，支持 PDF/Markdown 文档解析、章节级切分、向量召回与引用溯源；构建 100 条真实问题评测集，针对 chunk 边界导致的漏召回问题引入标题前缀和相邻片段扩展，使 Top-3 证据命中率从 72% 提升到 86%。

注意，这里有几个关键词：

- 设计并实现：说明你的动作。
- PDF/Markdown、章节级切分、引用溯源：说明关键方法。
- 100 条真实问题评测集：说明评测基础。
- Top-3 证据命中率从 72% 到 86%：说明结果。

如果没有真实线上指标，也可以写离线评测结果。但不要编造数据。可以写：

> 基于 60 个手工整理的问题进行离线评测，记录召回失败、引用错误和拒答失败三类问题，并据此调整切分策略和 prompt 约束。

这仍然比“效果良好”强很多。

## 60.10 常见扣分表达

下面这些表达在面试中很容易扣分：

- “这个很简单，就是调一下 API。”
- “我用了 LangChain，所以支持 Agent。”
- “向量数据库可以解决幻觉问题。”
- “准确率挺高的，但没有具体测。”
- “这个是模型自己生成的，我也不知道为什么。”
- “我们把所有文档都塞进 prompt。”
- “权限可以在回答后再检查。”
- “失败了就重新调用一次。”
- “微调之后肯定比 prompt 好。”
- “Agent 可以自己决定调用什么工具，所以不用写太多规则。”

这些话的问题不在于完全错误，而在于暴露出工程边界感不足。

更好的表达方式是：

- API 调用只是最外层，难点在数据、评测、稳定性和成本控制。
- LangChain 是编排工具，不等于系统能力本身。
- RAG 可以提供外部证据，但不能自动消除幻觉。
- 没有评测就不能判断效果，只能说主观体验。
- 模型输出不可完全解释，所以需要日志、引用、约束和回归测试。
- 上下文窗口再大，也需要考虑成本、噪声和信息选择。
- 权限必须尽量在数据访问阶段控制，不能依赖生成后补救。
- 重试要区分错误类型，否则可能放大成本和副作用。
- 微调、RAG、prompt 是不同手段，要看问题类型和数据条件。
- Agent 越自由，越需要边界、权限和可观测性。

## 60.11 面试前的项目复盘清单

每个写在简历上的大模型项目，都建议按下面清单复盘一遍：

- 一句话说明这个项目解决什么问题。
- 目标用户是谁。
- 输入和输出是什么。
- 为什么需要大模型，传统方法是否可行。
- 为什么选择 RAG、Agent、微调或 prompt engineering。
- 数据从哪里来，质量如何。
- 核心模块有哪些，你负责哪些。
- 最关键的技术难点是什么。
- 做过哪些取舍，放弃过哪些方案。
- 如何评测效果，指标是什么。
- 最典型的失败 case 是什么。
- 你如何定位和改进这些失败 case。
- 系统的成本、延迟和稳定性如何。
- 是否涉及权限、安全、隐私。
- 如果重做一遍，会优先改什么。

如果这些问题你都能回答，项目就很扎实。即使项目规模不大，也能讲出深度。

如果很多问题答不上来，不一定要删掉项目，而是应该回去补证据：加评测集、补 README、记录错误样例、画架构图、整理设计取舍。

## 60.12 小结

大模型工程面试中，项目表达的目标不是证明你知道多少名词，而是证明你能把一个不确定的问题做成一个可运行、可评测、可维护的系统。

讲项目时，要从技术栈转向决策链：问题、约束、方案、实现、评测、问题、改进、结果。

做作品集时，不要只放代码，要放证据：README、样例、评测、错误分析、已知限制和后续改进。

做系统设计时，不要急着堆组件，要先问清楚用户、数据、权限、准确性、延迟、成本和更新频率。

面对不会的问题，不要假装会，要诚实说明边界，并展示你的拆解和验证思路。

真正能打动面试官的，不是“我用过某某框架”，而是：

> 我知道这个系统为什么这样设计，知道它哪里可能失败，也知道应该如何验证和改进。

这就是大模型工程师最重要的面试表达能力。
