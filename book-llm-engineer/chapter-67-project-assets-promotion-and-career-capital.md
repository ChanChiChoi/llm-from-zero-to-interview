# 第67章：把项目经验沉淀成作品、晋升材料和职业资本

前面几章讨论了面试、入职、成长、持续学习和工程习惯。

这一章继续往后看一个更长期的问题：工作中的项目经验如何沉淀下来。

很多工程师做过不少项目，但几年后回头看，能讲清楚的很少。

代码交付了，需求上线了，问题解决了，但经验没有形成资产。

等到写晋升材料、准备跳槽、做技术分享或带新人时，才发现自己只记得一些零散片段。

大模型工程师尤其容易遇到这个问题。

因为项目里有大量隐性工作：失败 case 分析、prompt 调整、数据清洗、评测集构建、模型对比、成本优化、线上排障、安全边界设计、跨团队协作。这些工作如果不记录，很难在事后完整复原。

真正成熟的工程师，不只是完成项目，还会把项目沉淀成可复用资产。

这些资产会在三个场景中发挥作用：晋升、跳槽和长期职业品牌。

## 67.1 项目做完不等于经验沉淀完成

很多人以为项目上线就是结束。

但从个人成长角度看，项目上线只是第一步。

如果没有复盘和沉淀，项目经验会快速流失。

几个月后你可能只记得：

- 做过一个 RAG 系统。
- 接过一个客服 Agent。
- 优化过推理成本。
- 调过一个 prompt。
- 接入过一个模型服务。

这些描述太粗糙，无法体现工程能力。

真正有价值的是更具体的信息：

- 当时的业务问题是什么？
- 为什么旧方案不够好？
- 你负责了哪部分？
- 有哪些关键技术决策？
- 做过哪些实验？
- 哪些方案失败了？
- 最后指标提升多少？
- 成本、延迟、稳定性是否改善？
- 上线后遇到什么问题？
- 你沉淀了什么机制？

项目经验不是自然积累的。

项目经验需要被结构化记录、抽象和表达。

## 67.2 什么是职业资产

职业资产不是简历上的一句话。

它是你能反复复用、能证明能力、能迁移到新场景的成果。

对大模型工程师来说，职业资产可以包括：

- 一个完整项目复盘。
- 一套评测方法。
- 一个线上问题排查案例。
- 一个成本优化方案。
- 一个 Agent 安全控制设计。
- 一组高质量失败 case 分类。
- 一个可运行 demo。
- 一篇内部技术文档。
- 一次技术分享。
- 一个开源项目或工具。
- 一套面试项目表达模板。

职业资产有两个特点。

第一，它可以证明你做过真实工作。

不是只说“我了解 RAG”，而是能展示你如何解决召回不准、引用错误、延迟过高和权限过滤问题。

第二，它可以迁移。

你在一个知识库问答项目中沉淀的评测方法，也可以迁移到客服问答、文档助手、代码助手或销售助手中。

一个项目如果只完成交付，没有沉淀成资产，它对职业成长的贡献会小很多。

## 67.3 每个项目都应该留下项目卡片

建议为每个重要项目写一张“项目卡片”。

它不需要很长，但要结构清楚。

可以包含以下部分：

- 项目背景：为什么要做？
- 业务目标：要解决什么问题？
- 技术场景：RAG、Agent、训练、推理、评测、数据还是平台？
- 你的角色：负责什么，不负责什么？
- 核心难点：最难的 2 到 3 个问题是什么？
- 方案设计：关键架构和决策是什么？
- 实验过程：比较过哪些方案？
- 指标结果：上线前后有什么变化？
- 线上问题：遇到过什么事故或失败案例？
- 复盘沉淀：留下了什么工具、文档、流程或经验？

项目卡片最好在项目刚结束时写。

因为这时细节还清楚。

如果等半年后再写，很多关键数据、失败案例和技术取舍都会忘记。

项目卡片不是流水账，而是能力证据。

它以后可以变成简历项目、晋升材料、面试回答、技术分享或内部培训内容。

## 67.4 记录指标，不要只记录感受

大模型项目很容易用感受描述结果。

例如：

- “效果明显提升。”
- “用户反馈更好了。”
- “延迟优化了很多。”
- “成本降低不少。”

这些说法在复盘、晋升和面试中都不够有力。

更好的方式是记录指标。

指标可以包括：

- 正确率。
- 召回命中率。
- 引用准确率。
- 工具调用成功率。
- 拒答准确率。
- 幻觉率。
- 人工接管率。
- 用户满意度。
- 首 token 延迟。
- P95 延迟。
- 单次请求成本。
- 缓存命中率。
- GPU 利用率。
- 故障率。

不是每个项目都有完美指标。

但至少要记录当时能获得的证据。

例如：“在 200 条人工标注样本上，引用准确率从 71% 提升到 86%，P95 延迟从 4.8 秒增加到 5.3 秒，最终通过缓存和并发优化降到 4.1 秒。”

这比“效果提升明显”更可信。

如果没有指标，也要记录为什么没有，以及后来是否补了评测机制。

## 67.5 失败方案也是资产

很多人只记录成功方案。

但在大模型工程中，失败方案非常有价值。

因为大模型项目充满试错，真正的经验往往来自失败。

例如：

- 换更大模型后准确率提升不明显，但成本翻倍。
- 加长上下文后召回问题没有解决，反而延迟升高。
- Agent 自动规划在 demo 中很好，但线上稳定性不足。
- 微调后特定任务提升，但通用问答能力下降。
- rerank 提升证据排序，但对短 query 效果有限。
- prompt 加强格式约束后，拒答率异常升高。

这些失败经验能体现判断力。

面试官或晋升评委并不只想听你做成了什么，也想知道你如何识别错误方向、如何止损、如何调整方案。

记录失败方案时，可以写：

- 目标是什么？
- 假设是什么？
- 怎么验证？
- 结果为什么不符合预期？
- 最后如何调整？
- 对后续项目有什么启发？

一个工程师如果能清楚讲出失败方案，往往比只讲成功故事更可信。

## 67.6 把项目抽象成可迁移能力

项目本身可能过时，但能力可以迁移。

例如你做过一个企业知识库问答系统。

表面上看，这是一个 RAG 项目。

但可以抽象出很多能力：

- 文档解析和结构化能力。
- 检索链路设计能力。
- 评测集构建能力。
- 引用和证据校验能力。
- 权限过滤能力。
- 延迟和成本优化能力。
- 线上失败 case 分析能力。
- 跨业务专家协作能力。

这些能力可以迁移到合同审查、客服助手、销售知识库、代码文档问答和内部流程助手。

再比如你做过一个工单 Agent。

它可以抽象为：

- 任务拆解能力。
- 工具 schema 设计能力。
- 参数校验能力。
- 幂等和重试能力。
- 高风险操作确认能力。
- trace 和审计能力。
- 人工接管机制设计能力。

抽象能力的好处是，你不会被某个具体项目限制。

当别人问“你能不能做另一个场景”时，你可以解释哪些能力可以复用，哪些需要重新验证。

## 67.7 晋升材料要讲影响力，不只是工作量

晋升材料常见问题是堆工作量。

例如：

- 完成了 10 个需求。
- 接入了 3 个模型。
- 写了 5 个服务。
- 优化了多个 prompt。

这些信息有用，但不够。

晋升更关心影响力。

影响力包括：

- 业务结果：节省多少人工、提升多少转化、减少多少错误。
- 技术结果：提高稳定性、降低成本、提升可维护性。
- 团队结果：沉淀工具、流程、文档，让别人效率提高。
- 长期结果：建立评测、监控、发布门禁，减少重复事故。

一个更好的表达是：

“负责企业知识库问答质量优化，定位主要瓶颈来自文档解析和证据排序。通过重构解析链路、引入标题路径保留和 rerank 评测机制，在 300 条业务标注样本上将证据命中率从 68% 提升到 84%，引用错误率下降 35%。同时沉淀了 RAG 回归评测脚本和失败 case 分类，被后续 3 个业务线复用。”

这段表达包含问题、动作、结果和复用。

比“优化了 RAG 效果”强得多。

## 67.8 简历项目要讲清边界

写简历时，很多人喜欢把项目写得很大。

但如果边界不清，面试时很容易被追问穿。

可信的简历项目应该清楚说明：

- 项目规模。
- 你的职责。
- 技术难点。
- 关键方案。
- 量化结果。
- 你的独立贡献。

不要把团队成果全部写成个人成果。

例如整个团队做了模型训练、服务部署、业务接入和前端产品，而你主要负责 RAG 评测与检索优化，就应该聚焦这部分。

面试官并不要求你负责所有事情。

他们更关心你负责的部分是否真实、深入、可验证。

边界清楚反而更可信。

如果你确实参与了跨模块协作，也可以写“参与”“协同”“推动”，但不要混淆成“独立负责”。

## 67.9 面试表达要从故事变成结构

项目面试不是讲故事会。

好的项目表达应该有结构。

可以使用一个简单框架：

1. 背景：业务为什么需要这个系统？
2. 问题：当时最核心的问题是什么？
3. 方案：你设计或实现了什么？
4. 难点：遇到哪些关键挑战？
5. 验证：如何证明方案有效？
6. 结果：指标和业务结果是什么？
7. 复盘：如果重做，会改进什么？

大模型项目尤其要准备追问。

例如 RAG 项目可能被追问：

- 为什么不用长上下文直接塞全文？
- chunk size 怎么选？
- embedding 模型怎么评估？
- rerank 为什么有效？
- 如何处理权限过滤？
- 引用准确率怎么评估？
- 用户问法变化怎么处理？
- 线上幻觉怎么定位？

Agent 项目可能被追问：

- 工具 schema 如何设计？
- 模型选错工具怎么办？
- 参数错误怎么办？
- 高风险操作如何确认？
- 如何防止重复执行？
- trace 怎么记录？
- 如何评测 Agent 成功率？

提前把项目沉淀成结构，面试时就不会临场乱讲。

## 67.10 内部分享是放大影响力的方式

项目经验如果只停留在个人手里，影响力有限。

把经验整理成内部分享，可以帮助团队减少重复踩坑。

分享不一定要很宏大。

可以是：

- 一次 RAG 失败 case 复盘。
- 一次 prompt 变更事故复盘。
- 一次模型评测方法分享。
- 一次 Agent 工具调用设计分享。
- 一次推理成本优化记录。
- 一次线上排障流程说明。

好的内部分享要少讲空泛概念，多讲具体案例。

例如不要只讲“RAG 很重要”，而是讲“我们为什么把 chunk 从 1000 token 改到 400 token，为什么最初效果变差，后来通过标题路径和 rerank 才提升”。

这样的分享能体现你的实践深度，也能成为晋升材料中的团队影响力证据。

## 67.11 开源和公开写作要注意边界

很多大模型工程师希望通过开源、博客或公开分享建立个人品牌。

这是好事，但要注意边界。

不能泄漏公司代码、数据、客户信息、内部指标、未公开架构和敏感业务细节。

更安全的方式是抽象方法论。

例如可以写：

- RAG 评测集如何设计。
- Agent 工具调用有哪些失败模式。
- prompt 变更为什么需要回归测试。
- 大模型应用如何做成本评估。
- 知识库问答如何做引用校验。

不要写：

- 具体客户数据。
- 内部接口和架构细节。
- 未公开模型效果指标。
- 公司专有数据处理流程。
- 可能暴露安全边界的信息。

公开输出的价值不在于泄漏细节，而在于展示你的思考能力和工程方法。

## 67.12 建立个人项目档案库

建议长期维护一个个人项目档案库。

可以按项目建立目录，保存：

- 项目卡片。
- 设计方案。
- 关键指标。
- 实验记录。
- 失败 case。
- 复盘文档。
- 分享材料。
- 简历版本。
- 面试问答。

如果公司有保密要求，私人档案中不要保存敏感数据和内部代码。

可以保存脱敏后的结构化总结。

例如不要保存真实用户输入，可以保存失败类型和抽象示例。

不要保存内部文档内容，可以保存文档解析问题的分类和处理方法。

项目档案库的价值会随着时间增加。

当你准备晋升时，它可以快速变成材料。

当你准备跳槽时，它可以快速变成简历和面试答案。

当你带新人时，它可以变成培训内容。

当你做技术选型时，它可以帮助你避免重复踩坑。

## 67.13 不同阶段的沉淀重点

不同职业阶段，沉淀重点不一样。

初级工程师要重点沉淀：

- 自己完成了什么模块。
- 学到了哪些工程规范。
- 遇到的问题如何解决。
- 如何通过测试和 review 提升质量。

中级工程师要重点沉淀：

- 独立负责的项目。
- 技术方案和取舍。
- 指标提升。
- 线上问题处理。
- 跨角色协作。

高级工程师要重点沉淀：

- 系统性问题定义。
- 复杂项目推进。
- 技术方向判断。
- 平台化和机制建设。
- 团队影响力。
- 可复用方法论。

资深工程师或负责人要重点沉淀：

- 技术战略。
- 组织效率提升。
- 多团队协同。
- 风险治理。
- 人才培养。
- 长期技术资产建设。

沉淀内容要和你的职业目标匹配。

如果你想从中级走向高级，就不能只记录自己写了多少代码，而要记录你如何定义问题、推动方案、影响团队和形成机制。

## 67.14 一次项目复盘示例

下面给出一个简化示例。

项目背景：企业内部知识库问答上线后，用户反馈答案看似流畅但引用不准，业务专家不信任系统。

业务目标：提升回答可信度，减少人工二次核查成本。

我的职责：负责 RAG 质量分析、检索链路优化和回归评测建设。

核心问题：失败 case 分析发现，主要问题不是模型生成能力不足，而是文档解析丢失标题层级，导致 chunk 缺少上下文；同时 embedding 召回后排序不稳定，相关证据经常排在 Top-K 之外。

方案设计：重构文档解析流程，保留标题路径、表格说明和段落层级；调整 chunk size 和 overlap；引入 rerank；建立 300 条业务问题评测集，指标包括证据命中率、引用准确率和 P95 延迟。

实验结果：证据命中率从 68% 提升到 84%，引用错误率下降 35%，P95 延迟初期从 3.9 秒升至 5.2 秒，后通过缓存和并发优化降至 4.3 秒。

失败方案：曾尝试直接换更大模型，回答流畅度提升，但引用错误没有明显下降，成本增加约 2 倍，因此没有上线。

沉淀资产：形成 RAG 失败 case 分类、回归评测脚本、文档解析 checklist 和上线前引用校验流程，被后续两个知识库项目复用。

这个示例比“负责企业知识库问答优化”更有信息密度。

它展示了问题定位、方案取舍、指标结果、失败经验和团队复用。

## 67.15 小结

项目经验不会自动变成职业资本。

大模型工程师需要主动把项目沉淀成项目卡片、指标记录、失败案例、评测方法、复盘文档、内部分享和可复用工具。

职业资产的价值在于它能证明真实能力，并能迁移到晋升、跳槽、面试、分享和带人场景中。

做项目时，不仅要问“这个需求有没有上线”，还要问：我是否留下了可复用的经验？我是否记录了关键指标？我是否讲得清楚技术取舍？我是否把失败转化成判断力？

长期来看，能持续沉淀项目资产的工程师，会比只完成任务的工程师成长更快，也更容易在职业竞争中形成差异化。
