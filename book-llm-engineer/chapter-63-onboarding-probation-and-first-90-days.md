# 第63章：入职准备、试用期与前 90 天站稳岗位

拿到 offer 不是结束。

对大模型工程师来说，真正的考验往往从入职第一天才开始。

面试阶段你展示的是过往能力和潜力；入职之后，团队关心的是你能否在真实约束下持续交付：理解业务、融入协作、读懂代码、定位问题、稳定上线、推动改进。

很多候选人面试表现很好，但入职后前 90 天没有站稳，常见原因不是“不够聪明”，而是没有把新环境当成一个需要系统适应的工程问题：

- 没有快速建立业务上下文。
- 过早评价历史系统，却不了解历史约束。
- 只埋头写代码，不主动对齐优先级。
- 做了很多事，但没有让 manager 和团队感知到进展。
- 对模型效果、线上风险和业务目标之间的关系理解太慢。
- 遇到问题不及时暴露，最后拖成试用期风险。

这一章讲入职前准备、前 30 天、60 天和 90 天分别该做什么，以及如何在新团队中建立可信度。

## 63.1 入职前：把不确定性提前降下来

接受 offer 后到正式入职前，通常会有几天到几周时间。

这段时间不需要过度焦虑，也不需要把所有技术重新学一遍。更重要的是把“岗位上下文”补起来。

建议确认四类信息。

第一类是组织信息：

- 直属 manager 是谁。
- 所在团队负责哪个业务方向。
- 团队规模大概多大。
- 团队里算法、工程、产品、平台如何分工。
- 入职后是否有 mentor 或 buddy。

第二类是技术信息：

- 主要技术栈是什么。
- 使用哪些模型、推理服务、向量库、评测平台和监控系统。
- 是否有内部文档可以提前看。
- 是否需要提前准备开发环境或权限申请。

第三类是目标信息：

- 入职前三个月大致会负责什么。
- 当前团队最急的问题是什么。
- 试用期评估更看重交付、学习速度、协作还是独立负责能力。

第四类是手续信息：

- 入职材料。
- 背调进度。
- 设备领取。
- 办公地点。
- 入职时间。

这些信息不一定都能提前拿到，但你可以礼貌询问。

例如：

> 为了更快进入状态，我想提前了解一下入职后前几周大概会接触哪些业务或技术栈。如果有适合提前阅读的公开资料或内部文档目录，也可以先发我，我会提前熟悉。

这类沟通不会显得冒进，反而会让团队感觉你有准备意识。

## 63.2 不要在入职前盲目“狂补技术”

很多人拿到大模型岗位 offer 后，会开始焦虑：

- 要不要把所有 Transformer 论文再读一遍？
- 要不要重学 CUDA？
- 要不要把 LangChain、LlamaIndex、vLLM、Ray、Kubernetes 全部补完？
- 要不要提前做一个完整 Agent 项目？

这种焦虑可以理解，但效率不一定高。

入职前最值得补的，不是“大而全”的知识，而是和岗位最相关的短板。

如果你加入的是 RAG 应用团队，优先补：

- 文档解析。
- chunk 策略。
- embedding 与 rerank。
- 召回评测。
- 引用与拒答。
- 线上日志分析。

如果你加入的是 Agent 平台团队，优先补：

- function calling。
- tool schema。
- 权限和审批。
- trace 与可观测性。
- 幂等与重试。
- 工具执行沙箱。

如果你加入的是推理服务团队，优先补：

- batching。
- KV cache。
- PagedAttention。
- quantization。
- speculative decoding。
- 延迟和吞吐压测。

如果你加入的是评测团队，优先补：

- 离线评测集构造。
- LLM-as-a-judge。
- 人工标注一致性。
- 回归测试。
- A/B 实验。
- 指标与用户体验偏差。

入职前的学习目标不是变成全能专家，而是让自己在第一周听得懂团队讨论，在第一个任务中少踩低级坑。

## 63.3 第一周：少评价，多理解

入职第一周最重要的不是立刻产出惊人成果，而是建立上下文。

你需要理解：

- 团队正在解决什么业务问题。
- 当前系统为什么长成这样。
- 哪些是历史债务，哪些是有意取舍。
- 当前最重要的指标是什么。
- 哪些问题已经有人尝试过但失败了。
- 哪些系统不能随便动。

新同学常犯一个错误：刚看几天代码，就开始评价系统设计不合理。

有些评价可能是对的，但时机不对。你还不知道历史约束、业务压力、人力限制、上线事故、合规要求和组织边界。

更稳妥的方式是先问问题：

- 这个模块当初为什么这样拆？
- 这里是因为性能原因，还是因为历史兼容？
- 这个配置目前由谁维护？
- 这个指标线上有没有监控？
- 这块之前有没有尝试过其他方案？

问问题不是示弱，而是在建立系统模型。

第一周建议完成一份自己的“团队地图”：

| 维度 | 需要记录的内容 |
| --- | --- |
| 业务目标 | 团队服务的用户、场景、核心指标 |
| 系统链路 | 请求从入口到模型再到结果返回的路径 |
| 数据链路 | 数据来源、清洗、标注、评测和日志回流 |
| 模型链路 | 使用哪些模型、如何路由、如何升级 |
| 上线链路 | 开发、测试、灰度、回滚和监控流程 |
| 协作关系 | 产品、算法、平台、业务方和安全团队接口人 |
| 风险区域 | 不能轻易改的模块、历史事故、合规要求 |

这份地图不用正式提交，但它会极大提升你后续做事的准确性。

## 63.4 前 30 天：交付一个小而完整的闭环

入职前 30 天，最好的目标不是“证明自己很强”，而是交付一个小而完整的闭环。

什么叫小而完整？

不是只改一行配置，也不是接一个巨大项目。

它应该满足几个条件：

1. 范围可控。
2. 能真实解决一个问题。
3. 有明确验收标准。
4. 涉及基本开发、测试、上线或文档流程。
5. 能让你熟悉团队协作方式。

例如：

- 给 RAG 系统补一个失败 case 分析脚本。
- 修复一个线上引用格式错误。
- 为某类知识库增加标题路径 metadata。
- 给 Agent 工具调用增加 trace 字段。
- 为推理服务补一个延迟压测报告。
- 把一个人工排查流程整理成可复用 runbook。
- 给评测集增加一类高频业务问题。

这类任务不一定惊艳，但非常适合新人建立可信度。

它证明你能：

- 看懂现有系统。
- 尊重团队流程。
- 和相关人对齐需求。
- 写出可合并的代码或文档。
- 交付后能说明效果。

前 30 天不要贪多。

新人最重要的是建立“这个人靠谱”的第一印象。靠谱的含义很具体：说清楚、做得到、有反馈、遇到问题不隐瞒。

## 63.5 如何和 manager 对齐预期

试用期风险很多时候不是能力问题，而是预期错位。

你以为自己在认真学习，manager 可能觉得你产出太慢。

你以为自己做了很多优化，manager 可能觉得这些不是当前优先级。

你以为某个问题很重要，团队可能已经决定暂时不投入。

因此，入职后要主动和 manager 对齐。

建议在第一周或第二周约一次 1:1，问清楚：

- 试用期最重要的评估标准是什么？
- 前 30 天希望我完成什么？
- 哪些事情我应该优先熟悉？
- 哪些项目是当前团队最高优先级？
- 我应该多久同步一次进展？
- 如果遇到阻塞，应该找谁？

可以直接说：

> 我希望前三个月尽快进入状态，也避免方向跑偏。想和你对齐一下，前 30 天你最希望我达成的结果是什么？哪些事情如果我做到了，会说明我已经比较好地融入团队？

这不是讨好，而是降低不确定性。

对齐后，最好把结论写成简短文字发给 manager：

> 我理解下来，接下来两周优先做三件事：熟悉现有 RAG 链路，完成 XX 失败 case 的分析，补齐 YY 评测脚本。过程中我每周五同步一次进展，如果遇到数据权限或上线流程问题，会及时找 A 或 B 确认。

这样的同步能避免很多误会。

## 63.6 前 60 天：从执行任务到理解系统

前 30 天你主要是在适应环境和交付小闭环。到 60 天左右，你需要开始从“执行任务”升级为“理解系统”。

这意味着你不仅知道怎么改代码，还要知道为什么这样改。

对 RAG 系统，你要能说清楚：

- 文档从哪里来。
- 如何解析和切分。
- embedding 和 rerank 如何选型。
- 检索失败通常发生在哪些环节。
- 哪些问题应该拒答。
- 如何判断改动是否真的提升效果。

对 Agent 系统，你要能说清楚：

- 工具有哪些权限边界。
- 工具 schema 如何设计。
- 多步调用如何记录状态。
- 失败时如何重试或回滚。
- 哪些操作需要用户确认。
- trace 如何帮助排查问题。

对推理服务，你要能说清楚：

- 请求如何排队和 batching。
- KV cache 如何影响显存。
- 不同模型和 batch size 下延迟如何变化。
- P50、P95 和吞吐之间如何权衡。
- 降级和限流策略是什么。

对评测系统，你要能说清楚：

- 样例从哪里来。
- 标注口径是什么。
- 指标是否和用户体验一致。
- 评测结果如何进入发布流程。
- 如何避免只对评测集过拟合。

60 天左右，一个合格的新同学应该已经能独立负责中小任务，并能解释任务所在链路的上下游影响。

## 63.7 前 90 天：形成可见贡献

试用期结束前，你需要形成可见贡献。

可见贡献不是指天天加班，也不是写了多少行代码，而是团队能明确说出：你解决了什么问题，带来了什么变化。

可见贡献可以是：

- 一个上线功能。
- 一个稳定性改进。
- 一个评测体系补齐。
- 一个成本优化。
- 一个排障工具。
- 一个关键线上问题修复。
- 一套文档或 runbook。
- 一个跨团队流程改善。

关键是要有证据。

例如：

- 某类问题的证据命中率提升了多少。
- 某个接口 P95 延迟下降了多少。
- 某个排查流程从 2 小时缩短到 20 分钟。
- 某类线上错误告警减少了多少。
- 某个评测集覆盖了多少高频问题。
- 某个工具让多少个业务接入方复用。

如果你的工作很难量化，也要尽量留下可验证产物：文档、设计稿、代码合并记录、评测报告、上线记录、事故复盘、需求验收。

90 天时，manager 不应该只听你说“我学到了很多”，而应该能看到你已经让团队某个局部变好了。

## 63.8 新人如何避免“只会问问题”

新人当然需要问问题。

但如果所有问题都直接丢给别人，会消耗团队信任。

更好的方式是带着上下文提问。

低质量问题是：

> 这个为什么报错？

高质量问题是：

> 我在本地跑 RAG 评测脚本时，`retriever_config` 为空导致初始化失败。我查了最近两次配置变更，发现测试环境现在从配置中心读取，但本地文档里还是使用 yaml。想确认一下，本地开发推荐是手动导出配置，还是走 mock 配置？

低质量问题是：

> 这个模块是干什么的？

高质量问题是：

> 我看请求从 gateway 进入后，会先经过 query rewrite，再进入 hybrid retrieval，最后 rerank。这里的 query rewrite 是主要为了解决口语化问题，还是为了补齐业务实体？有没有评测看它对召回的影响？

提问前建议先做三件事：

1. 查文档。
2. 看代码或日志。
3. 写下自己的初步判断。

然后再问：

> 我查到的信息是 A，看到的现象是 B，我猜可能原因是 C。想确认这个判断是否正确。

这种提问方式会让别人愿意帮你，因为你不是把思考外包给对方。

## 63.9 不要过早推动“大重构”

很多工程师入职后看到历史代码，会本能想重构。

有时这确实必要，但新人阶段不适合轻易推动大重构。

原因很简单：你还没有足够上下文，也没有足够信任额度。

历史系统看起来不优雅，可能背后有这些原因：

- 兼容老客户。
- 迁移成本太高。
- 上线窗口有限。
- 依赖其他团队接口。
- 某些逻辑是事故后加的保护。
- 业务规则本身就复杂。

新人更适合做渐进式改进：

- 先补测试。
- 先补日志。
- 先补类型和配置校验。
- 先消除一个明确重复逻辑。
- 先把风险最高的路径加监控。
- 先把重构收益和风险写清楚。

如果确实认为需要重构，可以这样表达：

> 我看到这里有三类逻辑混在一起：召回配置、权限过滤和结果格式化。短期我不建议直接大改，因为这条链路影响线上问答。我的建议是先补一组回归样例和日志字段，再把结果格式化抽出来，作为第一步低风险改动。等这一步稳定后，再评估是否拆权限过滤。

这种表达比“这里代码太乱了，我们重写吧”更容易被接受。

## 63.10 试用期中的沟通节奏

试用期要避免两个极端。

一个极端是过度同步，每做一点小事都打扰别人。

另一个极端是长时间沉默，直到 deadline 前才暴露问题。

比较合适的节奏是：

- 每周给 manager 同步一次重点进展。
- 对关键任务，在开始前确认目标和验收标准。
- 遇到阻塞超过半天到一天，要及时暴露。
- 对跨团队依赖，要提前约时间，不要等最后。
- 重要结论尽量文字化，避免口头信息丢失。

周报可以很短：

```text
本周进展：
1. 完成 RAG 失败 case 标注 80 条，初步归因为召回缺失、排序错误、知识库缺失三类。
2. 修复引用展示中的段落编号丢失问题，已合并到测试环境。
3. 梳理了当前评测脚本运行流程，补充了本地运行文档。

风险/阻塞：
1. 线上日志权限还未开通，影响进一步分析真实用户问题，已联系 A 处理。

下周计划：
1. 补齐 120 条回归评测样例。
2. 对比 metadata 前缀对 Top-3 证据命中的影响。
```

这样的同步清晰、简短、可跟踪。

## 63.11 如何处理试用期负反馈

如果试用期中收到负反馈，不要本能防御，也不要立刻崩溃。

负反馈可能来自三个方面：

- 产出不够。
- 沟通不够。
- 方向不对。

你要做的是把反馈具体化。

不要只问：

> 我哪里做得不好？

要问：

> 你提到我进展偏慢，能否帮我具体看一下，是任务拆解慢、代码实现慢、还是遇到阻塞没有及时同步？如果接下来两周要改善，你最希望看到哪两个变化？

然后把改进动作写下来：

- 每两天同步一次当前任务状态。
- 每个任务开始前先写验收标准。
- 遇到阻塞超过半天就暴露。
- 请 mentor 帮忙 review 设计方向。
- 把大任务拆成可交付小任务。

负反馈不可怕。可怕的是反馈很模糊，而你没有把它变成可执行计划。

## 63.12 在新团队建立可信度的底层逻辑

可信度不是靠一句“我很负责”建立的。

它来自连续的小证据：

- 答应的事情按时完成。
- 做不到时提前说。
- 不懂的问题愿意查清楚。
- 线上风险不隐瞒。
- 代码改动有测试和回滚意识。
- 技术判断有证据，而不是只凭感觉。
- 争论时围绕目标和数据，而不是面子。

对大模型工程师来说，还有几条特别重要：

- 不把所有问题都归因于“模型不行”。
- 不用 demo 效果替代系统评测。
- 不忽视安全、权限、成本和延迟。
- 不在没有证据时承诺模型效果。
- 不把 prompt 当成唯一解决方案。

如果团队发现你每次判断都能讲清依据，每次改动都考虑上下游影响，每次遇到问题都能复盘，你的可信度会增长很快。

## 63.13 前 90 天的常见错误清单

下面这些错误很常见，最好提前避开。

第一，急于证明自己。

新人越急于证明自己，越容易跳过上下文，做出不合适的判断。

第二，只学习不交付。

学习很重要，但试用期最终要看你能否产生贡献。

第三，只交付不沟通。

你做了什么、为什么做、进展如何、风险在哪里，都需要被关键人知道。

第四，不敢暴露问题。

问题越晚暴露，解决成本越高。

第五，过度依赖 mentor。

mentor 可以帮你，但不能替你承担全部思考。

第六，忽略业务目标。

技术改动必须服务业务目标。否则再优雅也可能不是优先级。

第七，不记录。

新环境信息密度很高，不记录就会反复问同样问题，也难以形成系统理解。

第八，轻视线上风险。

大模型应用的线上风险不只是不报错，还包括胡答、越权、成本暴涨、延迟抖动和用户误用。

## 63.14 试用期结束前的复盘材料

试用期结束前，建议主动准备一份简短复盘。

内容不需要华丽，重点是清楚。

可以包含：

- 入职后主要完成了哪些工作。
- 每项工作的背景、动作和结果。
- 哪些能力已经补齐。
- 对团队系统和业务的理解。
- 当前还存在的短板。
- 接下来 1 到 2 个季度希望负责的方向。

示例结构：

```text
一、前三个月主要工作
1. RAG 失败 case 分析与评测集补齐
2. 引用格式错误修复与测试环境上线
3. 本地评测脚本文档化

二、结果
1. 标注 160 条真实问题，归因 5 类失败原因
2. 修复段落编号丢失问题，减少引用不完整 case
3. 新人本地启动评测流程从半天缩短到 30 分钟以内

三、我的理解
当前系统主要瓶颈不是生成模型，而是文档解析和 metadata 缺失导致的召回质量问题。

四、后续计划
1. 推进标题路径 metadata 的灰度实验
2. 建立每周失败 case 回流机制
3. 补齐拒答类评测样例
```

这份材料既帮助 manager 评估你，也帮助你从“新人”切换到“稳定贡献者”。

## 63.15 入职后的长期成长路线

前 90 天站稳之后，不代表可以停止思考。

接下来你要逐步扩大影响范围。

第一阶段：能独立完成明确任务。

你能根据需求完成开发、测试、上线和文档。

第二阶段：能独立负责一个模块。

你不仅能改代码，还能维护质量、处理问题、规划小版本。

第三阶段：能定义问题。

你能从线上反馈、业务目标和系统指标中发现真正值得做的事。

第四阶段：能推动跨团队协作。

你能协调产品、算法、平台、安全和业务方，让复杂项目落地。

第五阶段：能形成方法论。

你能把一次项目经验沉淀成评测流程、工程规范、平台能力或团队共识。

大模型工程师的成长，不是简单从“会调用模型”变成“会调用更多模型”。真正的成长是你越来越能处理复杂约束：效果、成本、延迟、安全、业务价值、团队协作和长期可维护性。

## 63.16 小结

拿到 offer 只是职业转换的开始。

入职前要降低信息不确定性，聚焦补齐和岗位直接相关的短板。

第一周少评价、多理解，建立业务、系统、数据、模型、上线和协作地图。

前 30 天要交付一个小而完整的闭环，建立“靠谱”的第一印象。

前 60 天要从执行任务升级为理解系统，能讲清楚上下游链路和关键取舍。

前 90 天要形成可见贡献，用结果、数据、文档或上线记录证明你让团队某个局部变好了。

试用期中最重要的不是表现得无所不能，而是持续建立可信度：目标清楚、进展透明、问题及时暴露、判断有证据、交付可验证。

真正站稳岗位之后，你的目标不只是“通过试用期”，而是在新团队中逐步成长为能定义问题、推动协作、沉淀方法的大模型工程师。
