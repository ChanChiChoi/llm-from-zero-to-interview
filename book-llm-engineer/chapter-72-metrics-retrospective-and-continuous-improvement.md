# 第72章：指标、复盘与持续改进

上一章讨论了技术战略、路线图与长期规划。

路线图回答“接下来要往哪里走”。

但还有一个更现实的问题：怎么知道自己真的走对了？

大模型工程最容易出现一种假进步。

团队做了很多事：换模型、改 prompt、加 rerank、接 Agent、做 dashboard、接入新推理框架。

每个人都很忙。

但过一段时间后发现：用户满意度没有明显提升，成本却上去了；线上失败 case 仍然反复出现；团队很难说清楚到底哪个改动有效。

原因通常不是团队不努力，而是缺少指标、复盘和持续改进闭环。

成熟的大模型工程师要能回答：

- 什么指标能证明系统变好？
- 哪些指标可能误导团队？
- 如何从失败 case 中学习？
- 如何做上线后复盘？
- 如何避免同类问题反复发生？
- 如何让质量改进成为团队机制，而不是个人救火？

这一章讨论大模型工程中的指标体系、复盘方法和持续改进机制。

## 72.1 没有指标，就没有方向感

没有指标时，团队只能靠感受判断系统好坏。

感受有价值，但不能替代度量。

大模型系统尤其需要指标，因为输出具有不确定性。

同一个模型可能在 demo 中表现很好，但在线上长尾问题中失败。

同一个 prompt 可能在 20 个样本上看起来不错，但在 1000 个真实请求中暴露格式错误、引用错误和安全问题。

指标的价值不是让所有问题都变成数字。

指标的价值是帮助团队建立方向感。

它至少能回答：

- 当前系统基线是什么？
- 改动后是否真的变好？
- 哪类问题最严重？
- 哪些场景被牺牲了？
- 成本和延迟是否可接受？
- 线上是否出现新失败模式？

没有指标，技术讨论很容易变成观点竞争。

有指标，团队才有机会从争论走向实验。

## 72.2 指标不是越多越好

很多团队一开始会建很多 dashboard。

但指标太多也会失去焦点。

一个指标体系应该分层，而不是把所有数字堆在一起。

大模型应用可以分成几层指标。

第一，业务结果指标。

例如：

- 用户任务完成率。
- 人工处理时长下降。
- 客服转人工率下降。
- 用户满意度。
- 业务转化率。
- 风险事件减少。

第二，模型质量指标。

例如：

- 回答正确率。
- 引用准确率。
- 幻觉率。
- 拒答准确率。
- 格式合法率。
- 工具调用成功率。

第三，系统性能指标。

例如：

- TTFT。
- TPOT。
- P95 / P99 latency。
- throughput。
- timeout rate。
- error rate。

第四，成本指标。

例如：

- input tokens。
- output tokens。
- cached tokens。
- 单次请求成本。
- 单任务成功成本。
- GPU 利用率。

第五，安全治理指标。

例如：

- 越权召回率。
- 敏感信息泄露率。
- prompt injection 命中率。
- 高风险工具误调用率。
- 审计日志完整率。

指标分层后，团队更容易定位问题。

如果业务指标没有改善，要继续看是质量、性能、成本、安全还是产品体验拖了后腿。

## 72.3 好指标要能驱动行动

一个指标是否有用，关键看它能不能驱动行动。

无用指标通常有几个特点：

- 看起来好看，但不知道怎么改。
- 只能反映总量，不能定位问题。
- 容易被刷高。
- 与用户价值关系弱。
- 没有 owner。
- 没有阈值和报警。

例如“总调用次数”不是没用，但它不能直接证明系统有效。

调用次数上升可能代表用户喜欢，也可能代表系统回答差导致用户反复追问。

“总 token 数”也不能直接代表价值。

token 多可能说明业务使用多，也可能说明 prompt 过长、上下文浪费或模型啰嗦。

更好的指标应该能指导行动。

例如：

```text
引用准确率下降 -> 检查文档解析、retrieval、rerank、引用抽取和生成约束。

资料不足拒答准确率下降 -> 检查知识覆盖、拒答策略和评测样本。

工具误调用率上升 -> 检查 tool schema、参数校验、prompt injection 和权限边界。

TTFT P95 上升 -> 检查 queue time、input length、prefix cache、prefill 负载和 worker 健康。
```

能指向下一步排查的指标，才是真正有用的指标。

## 72.4 警惕指标被优化坏

指标会改变团队行为。

如果指标设计不好，团队可能把指标优化好，但系统变差。

这就是常见的指标异化问题。

例如只看回答长度，模型可能输出更长但更啰嗦。

只看用户点赞率，系统可能倾向迎合用户，而不是指出资料不足。

只看自动评测分数，团队可能过拟合评测集。

只看调用量，团队可能鼓励无效使用。

只看 token 成本，团队可能过度压缩上下文，导致回答质量下降。

只看上线速度，团队可能牺牲评测和安全。

设计指标时要保留平衡。

例如 RAG 不应该只看 answer correctness。

还要看 citation accuracy、refusal accuracy、latency、cost 和越权召回。

Agent 不应该只看 task success rate。

还要看 tool misuse、人工接管、高风险误操作和审计完整性。

推理平台不应该只看 tokens/s。

还要看 TTFT、TPOT、错误率、KV cache pressure 和成本。

好的指标体系会防止团队只优化单一数字。

## 72.5 大模型系统的核心质量闭环

一个可靠的大模型系统应该有质量闭环。

最小闭环可以写成：

```text
线上请求
  -> 日志和 trace
  -> 失败 case 发现
  -> 失败类型分类
  -> 样本进入评测集
  -> 方案实验
  -> 灰度上线
  -> 指标观察
  -> 复盘和路线图调整
```

这里每一步都很重要。

如果没有日志，失败无法定位。

如果没有失败分类，团队不知道优先解决什么。

如果失败 case 不进入评测集，同类问题会反复出现。

如果实验没有 baseline，无法判断改动是否有效。

如果没有灰度，线上风险不可控。

如果没有复盘，经验不会沉淀。

质量闭环的核心不是一次性评测。

它是让系统从真实使用中持续学习。

## 72.6 失败 case 分类

失败 case 是大模型工程最重要的学习材料。

但失败 case 不能只堆在表格里。

必须分类。

常见分类包括：

- 用户问题不清楚。
- 知识库缺失。
- 文档解析错误。
- chunk 切分错误。
- embedding 召回失败。
- rerank 排序失败。
- 上下文组织错误。
- 模型理解错误。
- 模型幻觉。
- 引用不支持答案。
- 资料不足时没有拒答。
- prompt 指令冲突。
- 工具选择错误。
- 工具参数错误。
- 工具执行失败。
- 权限过滤错误。
- 输出格式错误。
- 安全策略误杀或漏杀。
- 产品交互导致误解。

分类的价值是让改进有方向。

如果 60% 失败来自文档解析和引用错误，优先级就不应该是微调。

如果 40% 失败来自工具参数错误，就应该检查 schema、参数校验和工具调用样本。

如果大量失败来自用户问题模糊，可能需要澄清问题或交互设计，而不是换模型。

## 72.7 复盘不是追责

复盘的目的不是找人背锅。

复盘的目的是让系统变强。

大模型系统的失败经常是多因素叠加。

例如一次错误工具调用可能同时涉及：

- prompt 没有限制清楚。
- tool schema 过于宽松。
- 后端没有权限校验。
- 没有高风险确认。
- 评测集没有覆盖类似场景。
- 日志缺少关键字段。

如果复盘只停留在“某个工程师没有考虑到”，团队不会进步。

好的复盘要问：

- 发生了什么？
- 用户或业务受到什么影响？
- 为什么现有机制没有提前发现？
- 哪些假设被证明是错的？
- 哪些日志或指标缺失？
- 哪些测试或评测应该补上？
- 哪些流程需要调整？
- 谁负责在什么时候完成改进？

复盘的产物应该是系统改进，而不是情绪释放。

## 72.8 上线后复盘

大模型功能上线后，应该做上线后复盘。

不是只有事故才复盘。

一次模型切换、prompt 改动、rerank 上线、Agent 工具接入、推理框架升级，都值得复盘。

上线后复盘可以看：

- 原始目标是否达成？
- 核心指标是否变化？
- 是否出现新失败类型？
- 成本和延迟是否符合预期？
- 用户反馈和离线评测是否一致？
- 哪些场景收益最大？
- 哪些场景变差？
- 是否需要扩大、回滚或调整？

例如 rerank 上线后，不能只看平均正确率提升。

还要看：

- citation accuracy 是否提升。
- P95 latency 增加多少。
- token 成本是否变化。
- 短 query 和长 query 是否都受益。
- 是否有业务场景被 rerank 误伤。

上线后复盘能防止团队把“已上线”误认为“已成功”。

## 72.9 事故复盘

大模型系统也会有事故。

事故不一定是服务不可用。

可能包括：

- 大面积幻觉。
- 越权召回。
- 敏感信息泄露。
- 工具误操作。
- 成本异常飙升。
- 模型版本回退失败。
- 推理服务延迟失控。
- prompt injection 绕过防护。
- 评测门禁失效导致坏版本上线。

事故复盘要包含时间线。

例如：

```text
10:00 新 prompt 灰度 10%。
10:20 用户反馈引用错误增加。
10:35 dashboard 显示 citation accuracy 下降。
10:50 灰度扩大被暂停。
11:10 回滚到旧 prompt。
13:00 发现新 prompt 改变了引用格式，导致引用抽取器失效。
```

时间线帮助团队理解事故是如何扩大的。

事故复盘还要分清：

- 直接原因。
- 根因。
- 检测缺口。
- 响应缺口。
- 预防措施。

例如直接原因是 prompt 改动导致引用格式变化。

根因可能是 prompt 变更没有回归引用抽取器，评测集中没有格式兼容样本，灰度指标没有设置自动暂停。

## 72.10 指标和复盘如何进入路线图

指标和复盘不是独立活动。

它们应该反过来影响路线图。

如果复盘发现大量问题来自日志缺失，那么补 trace 字段应该进入路线图。

如果指标显示成本持续上涨，而质量没有提升，那么成本治理应该提升优先级。

如果失败 case 显示 Agent 工具误调用集中在某几类工具，那么工具 schema 和权限机制应该优先于新增更多工具。

如果线上 P95 TTFT 长期超过 SLO，那么推理优化和容量规划应该进入路线图。

路线图不是管理者拍脑袋定的。

路线图应该吸收真实指标和复盘结论。

这也是持续改进的核心。

## 72.11 建立质量节奏

持续改进需要节奏。

可以建立几个固定机制。

每日或每周看线上关键指标。

包括：

- 错误率。
- 延迟。
- 成本。
- 失败 case 数量。
- 用户反馈。
- 高风险事件。

每周做失败 case review。

重点看：

- 新增失败类型。
- 高频失败类型。
- 是否需要补评测。
- 是否需要调整优先级。

每两周或每月做实验复盘。

重点看：

- 哪些实验有效。
- 哪些实验无效。
- 哪些假设被推翻。
- 哪些方案应该继续或停止。

每季度做路线图回顾。

重点看：

- 战略假设是否仍成立。
- 指标是否朝目标改善。
- 资源投入是否合理。
- 技术债是否积累过快。
- 新技术是否改变路线。

节奏不需要复杂。

关键是持续。

## 72.12 面试中的指标表达

如果面试官问：“你如何判断一个大模型应用上线后是否成功？”

可以这样回答：

```text
我不会只看调用量或 demo 效果，而会建立分层指标。第一层是业务结果，例如任务完成率、人工接管率、用户满意度或业务效率提升。第二层是模型质量，例如回答正确率、引用准确率、幻觉率、拒答准确率、格式合法率和工具调用成功率。第三层是系统性能，例如 TTFT、TPOT、P95 latency、错误率和 timeout。第四层是成本，例如 input/output tokens、单次请求成本和单成功任务成本。第五层是安全治理，例如越权召回、敏感信息泄露、高风险工具误调用和审计完整率。

上线后我会按场景和用户分桶看指标，不只看全局平均。还会持续收集失败 case，按文档解析、召回、rerank、生成、引用、工具、权限和产品交互分类。高频失败类型会进入评测集和路线图。每次模型、prompt、检索或工具改动都要有 baseline、灰度、监控和复盘。

我认为成功不是功能上线，而是核心业务指标改善，同时质量、延迟、成本和安全在可接受范围内，并且团队能从线上反馈中持续改进。
```

## 72.13 常见误区

第一个误区是只看调用量。

调用量高不代表用户成功，也可能是系统不好用导致反复尝试。

第二个误区是只看平均指标。

P95 和 P99 失败可能影响最重要用户。

第三个误区是只看离线评测。

离线评测不能完全覆盖线上真实分布和产品交互。

第四个误区是只看模型质量。

延迟、成本、安全和可运营性同样决定系统是否可用。

第五个误区是复盘只复盘事故。

正常上线也需要复盘，否则不知道哪些改动真正有效。

第六个误区是失败 case 不回流。

如果线上失败不进入评测集，同类问题会反复出现。

第七个误区是指标没有 owner。

没有 owner 的指标只会被展示，不会被改进。

## 72.14 小结

指标、复盘和持续改进，是大模型工程从试点走向生产的核心能力。

没有指标，团队没有方向感。

指标太多或设计不好，又会制造噪声甚至误导行为。

好的指标体系应该分层，覆盖业务结果、模型质量、系统性能、成本和安全治理。

失败 case 是最重要的学习材料，必须分类、回流到评测集，并影响后续路线图。

复盘不是追责，而是让系统变强。

上线后复盘、事故复盘、失败 case review 和路线图回顾，共同构成持续改进机制。

成熟的大模型工程师不只会做功能，也会建立一套让系统持续变好的机制。

长期来看，真正可靠的大模型系统不是一次性设计出来的，而是在真实使用、真实失败和持续复盘中演化出来的。
