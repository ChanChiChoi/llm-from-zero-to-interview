# 第61章：大模型工程面试中的高频问题与现场推理

上一章讲的是项目、作品集和系统设计的表达方式。那一章的重点是：不要只说你用了什么技术，而要说清楚你面对什么问题、做了什么取舍、如何验证结果。

这一章进入更具体的面试现场。

大模型工程面试里，很多问题看起来是在考知识点，例如：

- RAG 为什么会产生幻觉？
- chunk size 怎么选？
- embedding 模型怎么评估？
- Agent 为什么不稳定？
- 微调和 RAG 怎么选？
- 如何降低推理成本？
- 如何设计一个大模型应用的评测体系？

但真正优秀的回答，不是背定义，而是能把问题拆成工程条件、风险来源和验证方法。

面试官问“chunk size 怎么选”，不是想听一个固定数字。面试官真正想知道的是：你是否理解文档结构、检索召回、上下文噪声、生成质量、成本延迟之间的关系。

这一章会围绕高频问题展开，重点不是给出唯一标准答案，而是训练一种回答方式：先定义问题，再分析约束，再给出方案和验证路径。

## 61.1 回答技术问题的基本结构

大模型工程面试中的技术问题，可以用一个通用结构回答：

1. 先说明问题本质。
2. 再拆影响因素。
3. 给出常见方案。
4. 说明取舍和适用场景。
5. 最后讲如何验证。

例如面试官问：

> RAG 为什么还能产生幻觉？

弱回答是：

> 因为大模型本身会幻觉，所以需要更好的 prompt。

这个回答太泛。

更好的回答是：

> RAG 只是给模型提供外部证据，但不能保证证据一定被正确检索、正确选择和正确使用。幻觉可能来自几个环节：第一是召回失败，相关文档没有被取到；第二是召回结果有噪声，模型被无关内容干扰；第三是文档本身过期或冲突；第四是 prompt 没有限制模型必须基于证据回答；第五是用户问题超出知识库覆盖范围但系统没有拒答。改进上可以分别从 hybrid retrieval、rerank、metadata 过滤、引用约束、拒答策略和离线评测入手。验证时不能只看主观效果，应该构造问题集，分别统计证据命中率、引用正确率和无答案问题的拒答率。

这个回答的好处是：它不是一个点，而是一条链路。

面试中大部分问题都可以这样回答。

## 61.2 高频问题：RAG 和长上下文怎么选

常见问题：

> 现在模型上下文窗口越来越长，还需要 RAG 吗？

不要直接回答“需要”或“不需要”。应该先说明两者解决的问题不同。

可以这样回答：

> 长上下文降低了把更多材料放进 prompt 的难度，但没有完全替代 RAG。RAG 不只是为了解决上下文长度，它还解决知识选择、权限控制、数据更新、成本控制和引用溯源问题。如果文档规模小、问题集中、延迟和成本不敏感，长上下文直接塞材料可能足够简单有效。但如果知识库规模大、文档经常更新、用户权限不同、需要引用来源，还是需要检索和索引层。

可以进一步展开：

| 维度 | 长上下文 | RAG |
| --- | --- | --- |
| 实现复杂度 | 较低 | 较高 |
| 大规模文档 | 成本和噪声较高 | 更适合 |
| 权限控制 | 需要额外处理 | 可在检索阶段过滤 |
| 数据更新 | 每次构造上下文 | 可增量索引 |
| 引用溯源 | 可以做但不天然 | 更容易 |
| 延迟成本 | 输入越长越贵 | 可控制 top-k |

总结时可以说：

> 我会把长上下文看成一种增强能力，而不是 RAG 的替代品。真实系统里经常是组合使用：先通过检索选出候选证据，再利用较长上下文放入更多相邻片段、表格或原文内容。

这个回答体现了你没有陷入“新模型淘汰旧架构”的简单判断。

## 61.3 高频问题：chunk size 怎么选

这是 RAG 面试中非常常见的问题。

弱回答是：

> 一般设成 500 或 1000 tokens。

这个回答没有错，但没有工程含义。

更好的回答是：

> chunk size 没有固定值，要看文档结构、问题类型和模型上下文预算。chunk 太小，容易丢上下文，召回到的片段不足以回答问题；chunk 太大，embedding 表达会变得混杂，检索噪声增加，也会占用更多 prompt token。实践中我会先按文档结构切，比如标题、段落、列表、表格，再设置一个 token 上限。如果文档结构弱，再用固定长度加 overlap 兜底。最后用评测集观察召回命中率、答案引用率和上下文噪声。

可以补充几个具体判断：

- FAQ、短知识条目：chunk 可以小一些，因为一个问题通常对应一个短答案。
- 操作手册：需要保留步骤完整性，chunk 不能把连续步骤切断。
- 法务、合同、政策文档：要保留条款编号、标题路径和上下文。
- 论文、技术文档：可以按章节层级切分，并把标题路径加入 chunk。
- 表格：不要简单按文本长度切，最好整体保留或转成结构化 Markdown。

如果面试官继续问 overlap，可以回答：

> overlap 是为了缓解边界信息丢失，但不是越大越好。overlap 太大会增加索引规模和重复召回。更好的方式是优先利用结构切分；如果必须固定长度切分，再设置适度 overlap，并在检索后做去重或相邻片段扩展。

最后要落到验证：

> 我不会只凭经验定 chunk size，而是会准备一批真实问题，对不同切分策略跑离线评测，看 Top-k 证据命中、最终答案正确率、引用正确率、平均上下文长度和延迟成本。

这个回答比一个数字更有说服力。

## 61.4 高频问题：向量检索、关键词检索和混合检索怎么选

常见问题：

> 既然有 embedding，为什么还需要 BM25？

可以这样回答：

> 向量检索擅长语义相似，适合用户表达和文档表达不完全一致的情况；关键词检索擅长精确匹配，适合产品型号、错误码、函数名、配置项、法律条款编号这类信息。很多企业知识库里既有自然语言问题，也有大量精确术语，所以只用向量检索容易漏掉关键词，只用 BM25 又可能漏掉语义改写问题。混合检索通常更稳。

接着讲实现方式：

- 简单方案：分别跑向量检索和 BM25，合并候选后去重，再按规则或分数融合排序。
- 更稳方案：召回阶段扩大候选集，再用 reranker 对候选做统一排序。
- 领域方案：根据 query 类型选择检索方式，例如包含错误码、版本号、API 名称时提高关键词检索权重。

还可以补充风险：

> 混合检索不是简单把两个结果拼起来。需要处理分数不可比、重复文档、权限过滤、召回数量和延迟问题。如果有严格权限要求，metadata 过滤要在召回阶段就参与，而不是等生成后再过滤。

最后讲评测：

> 我会分别统计不同问题类型下的召回表现，比如术语型、概念型、流程型、故障型。这样才能知道向量、BM25、hybrid 分别解决了什么问题，而不是只看整体平均分。

## 61.5 高频问题：embedding 模型怎么评估

常见问题：

> 怎么判断一个 embedding 模型适不适合你的业务？

不要只回答“看 MTEB 排名”。排行榜有参考价值，但不能替代业务评测。

可以这样回答：

> 我会先看通用 benchmark、语言支持、上下文长度、维度、成本和推理速度，但最终还是要用业务数据评测。embedding 模型的目标不是向量本身好看，而是在具体检索任务中把正确证据排到前面。

业务评测可以这样做：

1. 收集真实或模拟 query。
2. 为每个 query 标注相关文档或相关 chunk。
3. 对不同 embedding 模型构建索引。
4. 统计 Recall@k、MRR、nDCG 等检索指标。
5. 接入生成链路后，再看最终答案正确率和引用正确率。

常见判断维度包括：

- 中文、英文或中英混合效果。
- 领域术语和缩写的识别能力。
- 长文本 embedding 是否稳定。
- 对相似但不同概念的区分能力。
- 向量维度对存储和检索成本的影响。
- 在线推理延迟和批量构建索引成本。

可以补一句很重要的话：

> embedding 模型不是越大越好，维度和效果也不是唯一指标。对于生产系统，还要看索引规模、召回延迟、更新成本和稳定性。

这能体现你把模型选择放在系统里考虑。

## 61.6 高频问题：rerank 是否一定需要

常见问题：

> RAG 系统是否一定要加 reranker？

标准回答不是“一定要”。

可以这样说：

> reranker 的价值是对初始召回候选做更精细的相关性排序，尤其在 top-k 里混有噪声、query 和文档需要深层匹配时有帮助。但它会增加延迟和成本，所以是否需要取决于业务要求和召回质量。

适合加 reranker 的场景：

- 文档规模大，召回候选噪声多。
- 问题和文档表达差异大。
- 答案对证据排序敏感。
- top-k 召回有相关内容，但经常排在后面。
- 混合检索后需要统一排序。

不一定需要的场景：

- 知识库很小，召回已经足够准。
- 延迟要求极高。
- 问题形式固定，可以用规则或 metadata 过滤解决大部分排序问题。
- 成本预算很紧。

评估方式：

> 我会先看不加 rerank 时的错误。如果正确 chunk 根本没召回，rerank 解决不了；如果正确 chunk 在候选里但排序靠后，rerank 才有明显价值。评测时要比较加 rerank 前后的 Recall@k、MRR、最终答案正确率、引用正确率、P95 延迟和单次成本。

这句话很关键：rerank 不是召回失败的万能药。

## 61.7 高频问题：如何降低大模型应用成本

常见问题：

> 一个 LLM 应用成本太高，怎么优化？

不要只说“换便宜模型”。成本优化要分层。

可以按链路拆：

1. 输入侧：减少无效上下文，压缩 prompt，控制 history 长度。
2. 检索侧：优化 top-k，减少重复 chunk，缓存检索结果。
3. 模型侧：根据任务难度做模型路由，简单问题用小模型，复杂问题用强模型。
4. 输出侧：限制最大输出长度，避免无意义长回答。
5. 缓存侧：对高频问题、稳定知识和中间结果做缓存。
6. 架构侧：批处理、异步任务、流式输出、限流和降级。

可以举一个模型路由例子：

> 对 FAQ 类问题，可以先用规则、检索或小模型判断是否能回答；只有需要复杂推理、总结或多步骤工具调用时才进入大模型。这样比所有请求都打到最贵模型更合理。

还要提醒风险：

> 成本优化不能只看平均 token。要同时看质量下降、延迟变化和失败重试带来的隐性成本。有些优化减少了单次调用成本，但增加了错误率和重试次数，整体未必更便宜。

验证指标可以包括：

- 单请求平均成本。
- P50/P95 延迟。
- 输出 token 分布。
- 缓存命中率。
- 不同模型路由比例。
- 答案正确率和用户满意度。
- 重试率和失败率。

成本优化的成熟回答，一定要把质量一起考虑。

## 61.8 高频问题：如何设计大模型应用评测体系

这是非常重要的问题。

弱回答是：

> 用人工打分，或者让 GPT 当裁判。

更好的回答是先拆评测对象：

> 大模型应用评测不能只评最终回答。要按链路拆开看：检索是否拿到证据，prompt 是否约束清楚，模型回答是否正确，引用是否准确，无答案时是否拒答，工具调用是否成功，延迟和成本是否达标。

对于 RAG，可以设计这些指标：

- 检索层：Recall@k、MRR、相关 chunk 命中率。
- 生成层：答案正确性、完整性、引用正确率、格式遵循率。
- 安全层：无答案拒答率、越权问题拦截率、敏感信息泄露率。
- 体验层：延迟、稳定性、用户反馈。
- 成本层：平均 token、模型调用次数、单请求成本。

对于 Agent，可以设计这些指标：

- 任务成功率。
- 工具调用准确率。
- 参数填写正确率。
- 多步任务完成率。
- 错误恢复率。
- 危险操作拦截率。
- 人工接管率。

评测集来源可以包括：

- 历史用户问题。
- 客服工单或搜索日志。
- 产品文档中的典型任务。
- 人工构造的边界 case。
- 上线后的失败样例回流。

还要说明自动评测的边界：

> LLM-as-judge 可以提高评测效率，但不能完全替代人工标注。尤其是事实正确性、引用准确性、权限安全这类问题，要有可校验标准或人工抽检。更稳的做法是规则评测、标准答案、LLM judge 和人工复核结合。

最后讲回归：

> 每次改 embedding、chunk、prompt、模型或 reranker，都应该跑同一套回归集，避免某类问题提升但另一类问题退化。

这类回答会让面试官看到你有持续迭代意识。

## 61.9 高频问题：微调、RAG 和 prompt engineering 怎么选

常见问题：

> 一个业务效果不好，你会选择 prompt、RAG 还是微调？

可以这样回答：

> 我会先判断失败原因。prompt、RAG 和微调解决的问题不同，不能把它们当成同一种优化手段。

可以按问题类型区分：

| 问题类型 | 优先方案 |
| --- | --- |
| 指令不清、格式不稳 | prompt engineering |
| 缺少外部知识或知识更新频繁 | RAG |
| 需要特定风格、格式、领域表达习惯 | 微调 |
| 需要调用工具或执行动作 | Agent / tool use |
| 模型基础能力不足 | 换模型或微调 |
| 数据权限和可追溯性要求高 | RAG + 权限控制 + 引用 |

可以举例：

> 如果客服问答答错是因为产品文档没有进入上下文，微调不是第一选择，应该先做 RAG。如果答案有证据但格式总是不符合要求，可以先优化 prompt 或做轻量格式微调。如果模型需要稳定输出某种业务 JSON，并且有大量高质量样本，微调可能更合适。

还要说明微调的风险：

- 数据质量差会把错误模式学进去。
- 微调不适合记忆频繁变化的知识。
- 微调后仍需要评测和安全控制。
- 微调可能带来过拟合、遗忘和维护成本。

一个成熟的总结是：

> 我会先用最简单可验证的方式解决问题。prompt 成本最低，RAG 适合外部知识，微调适合稳定行为和风格。最终选择要看错误来源、数据条件、更新频率、可追溯性和成本。

## 61.10 高频问题：Agent 为什么不稳定

常见问题：

> Agent 系统为什么经常不稳定？怎么改进？

可以先说明 Agent 的不稳定来自多个环节：

- 任务目标不清，模型不知道何时结束。
- 工具描述不清，模型选错工具。
- 工具 schema 设计差，参数容易填错。
- 工具返回信息太长或不结构化，模型难以继续推理。
- 多步链路中前一步错误会传播。
- 没有权限边界，模型可能尝试危险操作。
- 缺少 trace，失败后无法定位。
- 没有兜底策略，模型会在错误路径上反复尝试。

改进方法也要对应链路：

- 明确任务边界，把复杂任务拆成可控步骤。
- 工具设计使用清晰 schema，参数命名和描述要具体。
- 工具返回结构化结果，避免返回大段无关文本。
- 对高风险操作加入用户确认或人工审批。
- 设置最大步数、超时、重试策略和失败退出条件。
- 记录每一步 tool call、输入、输出、错误和耗时。
- 对常见任务使用固定 workflow，而不是完全自由规划。

可以补充一句：

> 生产里的 Agent 不应该追求无限自主，而应该追求在明确边界内可靠完成任务。越高风险的动作，越需要规则、权限和确认机制。

这句话通常很加分。

## 61.11 高频问题：如何处理模型输出不稳定

常见问题：

> 同一个问题模型每次回答不一样，怎么办？

可以先区分“不稳定”的类型：

- 内容事实不一致。
- 输出格式不一致。
- 引用来源不一致。
- 工具调用路径不一致。
- 长度和风格不一致。

不同问题对应不同处理方式。

如果是事实不一致：

- 增强检索证据。
- 要求基于引用回答。
- 降低生成自由度。
- 对无证据问题拒答。

如果是格式不一致：

- 使用更明确的 schema。
- 使用结构化输出或 constrained decoding。
- 增加格式校验和重试。
- 对稳定格式任务考虑微调。

如果是工具调用不一致：

- 收紧工具选择空间。
- 优化工具描述和参数 schema。
- 对固定流程使用 workflow。
- 增加中间状态检查。

还可以说：

> temperature 降低可以减少随机性，但它不是解决所有不稳定的办法。如果上下文噪声大、证据冲突、prompt 模糊，即使 temperature 很低也可能输出错误。

这能避免把问题简化成一个采样参数。

## 61.12 高频问题：如何做线上监控和故障排查

常见问题：

> 一个 LLM 应用上线后，你会监控什么？

可以按四类指标回答：

1. 质量指标：答案正确率、用户反馈、拒答率、引用正确率、任务成功率。
2. 性能指标：TTFT、总延迟、P95/P99、吞吐、排队时间。
3. 成本指标：输入 token、输出 token、模型调用次数、缓存命中率、单请求成本。
4. 稳定性指标：错误率、超时率、重试率、工具失败率、检索失败率。

对 RAG，还要记录：

- query。
- 检索到的 chunk id。
- chunk 分数。
- rerank 结果。
- 最终进入 prompt 的上下文。
- 答案引用。

对 Agent，还要记录：

- 每一步工具调用。
- 工具参数。
- 工具返回。
- 中间 reasoning 摘要或 trace。
- 失败位置。
- 是否人工接管。

故障排查时可以按链路定位：

> 如果回答错了，我会先看正确证据有没有被召回；如果没召回，是索引、切分、embedding、query 改写或权限过滤问题；如果召回了但没进入上下文，是排序或上下文构造问题；如果进入上下文但回答错，是 prompt、模型能力或证据冲突问题。

这条定位链非常实用，也很适合面试表达。

## 61.13 现场推理题：不要急着给结论

有些面试官会给一个开放题：

> 我们的知识库问答系统最近用户反馈准确率下降，你怎么排查？

这类题目考的是现场推理。不要一上来就说“换 embedding”或“加 rerank”。

可以按下面方式回答：

1. 先确认现象：是所有问题下降，还是某类问题下降。
2. 确认时间点：是否和模型、prompt、索引、文档、权限、流量变化有关。
3. 分层排查：检索、排序、上下文、生成、权限、评测口径。
4. 对比样例：拿成功和失败 case 做差异分析。
5. 临时止血：必要时回滚配置、降级模型或加人工兜底。
6. 长期修复：补评测集、监控和回归流程。

可以给出一个完整回答：

> 我会先看准确率下降是否集中在某些文档类型、某些用户群或某个时间段。如果是最近开始下降，会检查是否有索引更新、embedding 模型切换、prompt 修改、文档批量更新或权限规则变化。然后抽样失败 case，看正确证据是否被召回。如果没有召回，排查文档解析、chunk、embedding、query 改写和过滤；如果召回了但没排到前面，排查 rerank 或排序融合；如果证据进入 prompt 但模型答错，检查 prompt 约束、证据冲突和模型版本。线上先通过回滚最近变更或降级到稳定链路止血，同时把失败 case 加入回归集。

这类回答的关键是：先定位，再优化。

现场推理题最忌讳的是直接开药方。

## 61.14 面试回答中的“边界感”

大模型工程面试很看重边界感。

所谓边界感，就是你知道一个技术能解决什么，不能解决什么。

例如：

- RAG 能提供外部证据，但不能保证模型必然正确使用证据。
- 微调能改变模型行为模式，但不适合承载频繁变化的事实知识。
- Agent 能自动化多步任务，但越自由越需要权限和可观测性。
- 长上下文能容纳更多信息，但不等于信息选择问题消失。
- rerank 能改善排序，但不能召回根本没有进入候选集的证据。
- prompt 能快速修复一些行为问题，但不能替代数据质量和系统设计。
- LLM-as-judge 能提高评测效率，但不能完全替代可验证标准和人工抽检。

面试中可以多使用这种句式：

> 这个方法适合……但不适合……如果约束是……我会优先……如果约束变成……我会改成……

这类表达比绝对化判断更成熟。

## 61.15 小结

大模型工程面试中的高频问题，很少只有一个标准答案。

面试官真正想看的，是你能不能把一个技术问题放回系统链路里分析：问题在哪里，影响因素是什么，方案有什么取舍，如何验证效果，失败后如何定位。

回答问题时，不要只背概念。更好的结构是：

- 先说明问题本质。
- 再拆影响因素。
- 给出可选方案。
- 说明适用场景和风险。
- 最后落到评测和验证。

如果你能用这种方式回答 RAG、Agent、微调、评测、成本、监控和系统设计问题，面试官会更容易相信你具备真实工程能力。

大模型工程师不是只会调用模型的人，而是能让模型在真实约束下稳定产生价值的人。
