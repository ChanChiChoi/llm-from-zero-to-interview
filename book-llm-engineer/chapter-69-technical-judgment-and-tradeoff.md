# 第69章：技术判断与方案取舍

上一章讨论了从执行者到技术负责人。

技术负责人最核心的能力之一，就是技术判断。

所谓技术判断，不是知道更多名词，也不是永远选择最新方案，而是在具体约束下选择更合适的路径。

大模型工程里，很多问题没有标准答案。

要不要换更强模型？

要不要做微调？

要不要上 Agent？

要不要使用长上下文？

要不要引入 rerank？

要不要自建推理服务？

要不要把 demo 平台化？

这些问题都不能只靠感觉回答。

这一章讨论：大模型工程师如何形成可靠的技术判断，如何在效果、成本、延迟、安全、复杂度和业务价值之间做取舍。

## 69.1 技术判断不是偏好，而是有证据的选择

很多技术争论表面上是在讨论方案，实际是在表达偏好。

有人偏好最新模型，有人偏好开源方案，有人偏好简单架构，有人偏好平台化，有人偏好端到端自动化。

偏好本身没有问题，但偏好不能替代判断。

技术判断至少需要回答：

- 要解决的问题是什么？
- 当前约束是什么？
- 可选方案有哪些？
- 每个方案的收益是什么？
- 每个方案的成本是什么？
- 主要风险是什么？
- 如何验证？
- 如果失败，如何回退？

如果这些问题回答不清，那么所谓判断很可能只是直觉。

成熟工程师可以有直觉，但最终要把直觉转化为证据、实验和可解释的取舍。

## 69.2 先判断问题类型，再选择技术手段

大模型项目中最常见的错误，是拿技术手段直接套问题。

例如：

- 效果不好，就换大模型。
- 回答不准，就调 prompt。
- 流程复杂，就上 Agent。
- 知识太多，就上长上下文。
- 个性化不足，就微调。
- 延迟高，就换推理框架。

这些反应不一定错，但都太快了。

更好的顺序是先判断问题类型。

常见问题类型包括：

- 知识缺失：模型没有看到必要信息。
- 召回错误：相关信息没有被检索到。
- 排序错误：相关信息被检索到但位置靠后。
- 理解错误：模型看到了信息但没理解。
- 指令冲突：prompt、系统规则和上下文互相干扰。
- 数据质量差：文档、标注或工具返回本身有问题。
- 权限错误：模型看到了不该看的信息，或看不到该看的信息。
- 工具失败：参数、接口、权限、幂等或异常处理有问题。
- 评测偏差：指标不能反映真实用户体验。
- 产品问题：交互方式、预期管理或业务流程设计不合理。

不同问题对应不同技术手段。

如果是知识缺失，可能需要 RAG、工具调用或上下文补充。

如果是召回错误，应该看文档解析、chunk、embedding、query rewrite 或 hybrid search。

如果是排序错误，可以考虑 rerank。

如果是工具失败，换模型未必有用，应该检查 schema、参数校验、错误恢复和权限。

如果是评测偏差，继续优化模型可能只是优化了错误指标。

先判断问题类型，才能避免错误投入。

## 69.3 换模型不是万能解法

大模型工程中，换更强模型是最直接的想法。

它有时很有效。

但它不是万能解法。

换模型通常能改善：

- 语言理解能力。
- 复杂推理能力。
- 指令遵循能力。
- 多语言能力。
- 代码生成能力。
- 鲁棒性。

但换模型不一定能解决：

- 知识库没有召回到正确证据。
- 文档解析丢失表格结构。
- 工具接口返回错误。
- 权限过滤逻辑错误。
- prompt 要求互相冲突。
- 评测集质量差。
- 用户需求本身不清楚。
- 产品交互没有管理预期。

判断是否换模型，可以问：

- 当前失败案例中，有多少是模型能力不足？
- 如果给模型正确上下文，它能不能回答对？
- 更强模型在核心样本上提升多少？
- 成本和延迟增加多少？
- 是否会引入新的格式、风格或安全问题？
- 是否有回滚和灰度方案？

一个简单实验是 oracle context 测试。

也就是人工把正确证据放进上下文，看模型是否能回答正确。

如果有正确上下文时模型仍然答错，说明模型能力可能是瓶颈。

如果有正确上下文时模型能答对，但线上答错，瓶颈更可能在检索、上下文组织或工具链路。

## 69.4 微调、RAG 和 prompt 的取舍

很多人会问：这个问题应该用 prompt、RAG，还是微调？

没有统一答案，但可以从问题性质判断。

prompt 适合解决：

- 输出格式约束。
- 角色和语气控制。
- 简单任务步骤说明。
- 小范围行为调整。
- 已有上下文下的推理和总结。

RAG 适合解决：

- 外部知识更新快。
- 需要引用和可追溯证据。
- 企业私有知识。
- 知识库规模较大。
- 不希望把知识固化进模型参数。

微调适合解决：

- 稳定重复的任务模式。
- 特定输出风格。
- 特定领域表达习惯。
- 大量高质量标注样本可用。
- prompt 太长或成本太高。
- 基座模型具备能力但行为不稳定。

不适合微调的情况包括：

- 数据少且质量不稳定。
- 知识频繁变化。
- 主要问题是检索不到证据。
- 需要严格引用来源。
- 没有可靠评测集。
- 上线后无法监控副作用。

很多企业知识问答问题，第一优先级通常不是微调，而是文档解析、检索、rerank、引用校验和评测。

微调不是不能做，而是要在问题明确、数据质量足够、评测可靠时做。

## 69.5 Agent 不是所有自动化的答案

Agent 很吸引人，因为它看起来可以自动规划、调用工具、完成任务。

但不是所有自动化都需要 Agent。

如果业务流程清楚、步骤固定、规则明确，用普通工作流可能更稳定、更便宜、更可控。

Agent 更适合：

- 用户目标多样。
- 步骤不固定。
- 需要根据上下文选择工具。
- 需要处理开放问题。
- 需要在多个信息源之间推理。

Agent 不适合或需要谨慎的场景包括：

- 高风险不可逆操作。
- 强合规流程。
- 步骤固定且规则明确。
- 对延迟和成本极敏感。
- 缺少工具权限和审计机制。
- 没有可靠评测和人工接管。

判断是否上 Agent，可以问：

- 任务是否真的需要动态规划？
- 工具调用是否有副作用？
- 模型选错工具的代价是什么？
- 是否能限制操作空间？
- 是否有确认、回滚和审计？
- 是否能评测任务完成率？

很多时候，最合适的方案是“工作流 + 局部 LLM 能力”，而不是完全自主 Agent。

例如工单处理可以用固定流程控制状态流转，只在意图识别、信息抽取、回复生成和异常解释环节使用模型。

这样既能利用模型能力，又能保持系统可控。

## 69.6 长上下文和 RAG 不是简单替代关系

长上下文模型出现后，很多人认为 RAG 会被替代。

这个判断过于简单。

长上下文确实改变了部分应用方式。

它适合：

- 单个文档很长但数量有限。
- 用户需要围绕完整材料做分析。
- 上下文结构比较完整。
- 任务对跨段落关系敏感。

但 RAG 仍然有价值：

- 知识库很大，无法全部塞入上下文。
- 需要权限过滤。
- 需要引用和证据排序。
- 需要控制成本和延迟。
- 需要增量更新。
- 需要可解释的召回链路。

长上下文也有成本和风险：

- token 成本高。
- 延迟可能增加。
- 模型不一定真正利用远距离信息。
- 无关信息过多可能干扰回答。
- 上下文构造仍然需要策略。

更合理的判断是：长上下文和 RAG 可以互补。

RAG 负责从大规模知识库中筛选相关信息，长上下文负责承载更多结构化证据和复杂任务背景。

是否使用长上下文，应该通过真实任务评测，而不是只看模型支持多少 token。

## 69.7 自建模型服务还是调用 API

企业落地大模型时，经常面对自建模型服务和调用外部 API 的选择。

调用 API 的优势是：

- 启动快。
- 模型能力强。
- 运维成本低。
- 可以快速试错。
- 不需要管理 GPU 和推理引擎。

调用 API 的风险是：

- 数据合规限制。
- 成本不可控。
- 延迟和稳定性受外部影响。
- 模型版本变化可能影响效果。
- 定制能力有限。

自建服务的优势是：

- 数据可控。
- 成本可优化。
- 版本可控。
- 可做深度定制。
- 可与内部系统更紧密集成。

自建服务的成本是：

- GPU 资源和运维复杂。
- 推理框架、监控、扩缩容、容量规划都要自己负责。
- 模型效果可能不如顶级闭源模型。
- 团队需要 infra 能力。

判断时可以问：

- 数据是否允许出域？
- 请求量是否足够大，值得自建？
- 延迟和稳定性是否有强要求？
- 团队是否有推理工程能力？
- 模型能力差距是否可接受？
- 是否需要微调、量化或私有部署？

很多团队会采用混合策略：早期用 API 快速验证，成熟后把高频、低风险、可控任务迁移到自建服务，复杂任务继续使用能力更强的模型。

## 69.8 开源框架还是自研

大模型应用生态中有很多框架。

它们可以加速开发，但也可能引入不必要的抽象。

选择开源框架时，要看：

- 是否解决你的核心问题？
- 抽象是否容易理解？
- 是否方便调试？
- 是否能满足性能和安全要求？
- 社区是否活跃？
- 版本变化是否稳定？
- 退出成本高不高？

不要因为框架流行就引入。

也不要因为“自研更可控”就什么都重写。

一个实用原则是：

- 通用基础能力优先复用成熟方案。
- 业务核心差异化能力可以自研。
- 高风险链路要确保可观测、可替换、可回滚。

例如向量数据库、基础模型 serving、日志监控可以优先复用成熟工具。

但你的业务评测集、失败 case 分类、权限过滤策略、工具审批流程，往往需要结合业务自研。

## 69.9 Demo、试点和生产系统的标准不同

技术判断必须考虑项目阶段。

同一个方案，在 demo 阶段合理，在生产阶段可能很危险。

demo 阶段的目标是验证可能性。

可以接受临时 prompt、少量样本、手动配置、简单日志。

试点阶段的目标是验证可用性。

需要基本评测、用户反馈、异常处理、成本估算和灰度机制。

生产阶段的目标是验证可靠性。

需要稳定评测、监控告警、权限控制、审计、回滚、容量规划、安全边界和运维流程。

很多大模型项目的问题，是把 demo 代码直接推进生产。

这会导致：

- prompt 没有版本管理。
- 没有回归评测。
- 没有日志和 trace。
- 没有成本监控。
- 没有失败路径处理。
- 没有权限和安全边界。

技术负责人要明确不同阶段的工程标准。

既不能在 demo 阶段过度工程，也不能在生产阶段继续用 demo 标准。

## 69.10 用小实验降低决策风险

很多技术争论可以通过小实验解决。

不要在会议里争论三小时，不如设计一个一两天能完成的验证。

小实验要有：

- 明确假设。
- 对比方案。
- 固定样本。
- 指标定义。
- 失败标准。
- 结论记录。

例如是否引入 rerank，可以用 100 到 300 条标注 query 做对比：embedding only 与 embedding + rerank 的 Top-K 命中率、引用准确率、延迟和成本。

例如是否换模型，可以在相同上下文和 prompt 下比较不同模型的正确率、格式稳定性、拒答准确率、成本和 P95 延迟。

例如是否上 Agent，可以挑选 30 个真实任务，比较固定工作流、LLM 辅助工作流和 Agent 的任务完成率、错误类型和人工接管率。

小实验的价值不是得出永久真理，而是减少盲目决策。

## 69.11 什么时候应该保持简单

保持简单是一种重要判断。

以下情况通常应该优先简单方案：

- 问题还没有被验证。
- 用户需求还不稳定。
- 数据质量还很差。
- 评测体系还没有建立。
- 团队维护能力有限。
- 高级方案收益不明确。
- 上线风险主要来自流程和权限，而不是模型能力。

例如一个业务刚开始试点智能问答，知识库文档还没有整理好，用户问题类型也不清楚。

此时不应该一开始就构建复杂 Agent 平台。

更合理的是先做一个简单 RAG 原型，收集用户问题、失败 case 和文档质量问题。

简单方案不是低水平。

在不确定性高的时候，简单方案可以帮助团队更快学习。

## 69.12 什么时候应该增加复杂度

复杂度不是绝对坏事。

当复杂度换来明确收益时，它是必要的。

可以增加复杂度的情况包括：

- 同类需求反复出现。
- 手工流程成为瓶颈。
- 线上风险需要机制控制。
- 多业务线需要复用能力。
- 指标显示某个瓶颈长期存在。
- 简单方案已经无法满足规模。
- 技术债开始显著影响交付。

例如早期可以手动整理失败 case。

但当线上每天产生大量反馈，且多个业务线都需要分析失败类型时，就应该建设反馈回流和评测样本管理工具。

早期可以手动配置 prompt。

但当 prompt 频繁变更并影响多个场景时，就应该做版本管理、评审和回归测试。

增加复杂度的前提是看见真实重复问题，而不是想象未来可能需要。

## 69.13 决策文档：让取舍可追溯

重要技术决策应该留下记录。

决策文档不需要很长，但要说明：

- 背景。
- 问题。
- 约束。
- 可选方案。
- 对比维度。
- 决策结论。
- 风险和缓解措施。
- 验证结果。
- 未来重新评估条件。

例如决定暂不微调，可以记录：

- 当前问题主要来自召回和引用错误。
- 现有标注数据不足以支撑微调。
- 微调无法解决知识更新和引用追溯问题。
- 先投入文档解析、检索评测和 rerank。
- 如果后续稳定任务样本超过一定规模，再重新评估微调。

这样的记录能减少重复争论。

几个月后如果条件变化，团队也能知道当初为什么这样决定。

## 69.14 技术判断的常见陷阱

大模型工程中有几个常见陷阱。

第一，benchmark 迷信。

某个模型在公开榜单上强，不代表在你的业务场景中强。

第二，demo 幻觉。

一个演示效果好，不代表线上长尾稳定。

第三，新技术偏见。

新方法可能有价值，但也可能只是增加复杂度。

第四，局部最优。

某个模块指标提升，可能导致整体延迟、成本或安全风险变差。

第五，忽视维护成本。

方案上线只是开始，后续监控、排障、升级和人员交接都要成本。

第六，过度相信模型自我纠错。

模型可以辅助检查，但不能替代权限、幂等、审计和工程约束。

第七，缺少退出策略。

引入框架、模型或平台前，没有想清楚如果不好用怎么退出。

识别这些陷阱，本身就是技术判断的一部分。

## 69.15 小结

技术判断是大模型工程师从执行者走向负责人的关键能力。

可靠判断不是个人偏好，而是在明确问题、约束、方案、收益、成本、风险和验证方式之后做出的选择。

面对换模型、微调、RAG、Agent、长上下文、自建服务、开源框架和平台化等问题时，不要被热点牵引，也不要停留在抽象争论中。

先判断问题类型，再设计小实验，用指标和失败案例支撑决策。

保持简单和增加复杂度都可能正确，关键在于是否匹配当前阶段、真实瓶颈和团队能力。

最终，成熟的大模型工程师不是永远选择最先进技术的人，而是在复杂约束下持续做出可解释、可验证、可回退选择的人。
