### 2. 先说公式

手写前先写公式：

```text
Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) V
```

解释：

```text
Q 表示 query，K 表示 key，V 表示 value。
QK^T 计算每个 query 对每个 key 的相关性。
除以 sqrt(d_k) 是为了缩放点积，避免维度大时 score 方差过大，softmax 过于尖锐。
softmax 得到 attention weights。
最后乘以 V，得到每个位置聚合后的表示。
```

这段话是面试开场必须熟练说出的。

### 3. 标准输入输出形状

推荐面试中使用多头内部形状：

```text
q: [B, H, T_q, D]
k: [B, H, T_k, D]
v: [B, H, T_k, D]
mask: 可广播到 [B, H, T_q, T_k]
output: [B, H, T_q, D]
attn_weights: [B, H, T_q, T_k]
```

其中：

```text
B：batch size
H：number of heads
T_q：query length
T_k：key/value length
D：head dimension
```

self-attention 中通常 `T_q = T_k`。

cross-attention 中可以不同。

### 4. 10 分钟标准代码

```python
import math
import torch
import torch.nn.functional as F


def scaled_dot_product_attention(q, k, v, mask=None, dropout_p: float = 0.0):
    """
    q: [B, H, T_q, D]
    k: [B, H, T_k, D]
    v: [B, H, T_k, D]
    mask: broadcastable to [B, H, T_q, T_k], 1 means keep, 0 means mask out
    """
    d_k = q.size(-1)
    scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(d_k)

    if mask is not None:
        scores = scores.masked_fill(mask == 0, float("-inf"))

    attn = F.softmax(scores, dim=-1)

    if dropout_p > 0:
        attn = F.dropout(attn, p=dropout_p, training=True)

    out = torch.matmul(attn, v)
    return out, attn
```

这段代码短、清楚、可解释，是面试中最稳的版本。

### 5. 逐行解释

#### 5.1 计算 score

```python
scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(d_k)
```

形状变化：

```text
q: [B, H, T_q, D]
k.transpose(-2, -1): [B, H, D, T_k]
scores: [B, H, T_q, T_k]
```

`scores[b, h, i, j]` 表示第 `b` 个样本、第 `h` 个头中，第 `i` 个 query 对第 `j` 个 key 的注意力打分。

#### 5.2 加 mask

```python
scores = scores.masked_fill(mask == 0, float("-inf"))
```

mask 必须在 softmax 前加。

如果在 softmax 后把权重置 0，剩余权重和不再等于 1，还需要重新归一化。

在 softmax 前把被屏蔽位置设为负无穷，softmax 后这些位置权重接近 0。

#### 5.3 softmax

```python
attn = F.softmax(scores, dim=-1)
```

`dim=-1` 表示对所有 key 位置归一化。

也就是每个 query 对所有 key 的权重和为 1。

#### 5.4 加权求和

```python
out = torch.matmul(attn, v)
```

形状变化：

```text
attn: [B, H, T_q, T_k]
v: [B, H, T_k, D]
out: [B, H, T_q, D]
```

每个 query 位置得到一个由 value 加权求和后的向量。

### 6. Causal Mask 怎么写

Decoder-only GPT 需要 causal mask，防止当前位置看到未来 token。

```python
def build_causal_mask(seq_len: int, device=None):
    mask = torch.tril(torch.ones(seq_len, seq_len, device=device))
    return mask.view(1, 1, seq_len, seq_len)
```

示例：

```python
mask = build_causal_mask(T, q.device)
out, attn = scaled_dot_product_attention(q, k, v, mask)
```

`torch.tril` 得到下三角矩阵：

```text
1 0 0 0
1 1 0 0
1 1 1 0
1 1 1 1
```

第 0 个位置只能看自己，第 1 个位置能看 0 和 1，第 2 个位置能看 0、1、2。

### 7. Padding Mask 怎么写

padding mask 用于屏蔽 batch 中补齐的 padding token。

假设 `attention_mask` 形状是 `[B, T]`，其中 1 表示有效 token，0 表示 padding。

要变成可广播到 scores 的形状：

```python
padding_mask = attention_mask[:, None, None, :]
```

形状：

```text
[B, T] -> [B, 1, 1, T]
```

这样可以广播到 `[B, H, T_q, T_k]`，屏蔽所有 query 对 padding key 的关注。

### 8. Causal Mask 和 Padding Mask 合并

如果同时需要 causal mask 和 padding mask：

```python
causal_mask = build_causal_mask(T, q.device)          # [1, 1, T, T]
padding_mask = attention_mask[:, None, None, :]       # [B, 1, 1, T]
mask = causal_mask * padding_mask                     # [B, 1, T, T]
```

这个 mask 可以广播到 `[B, H, T, T]`。

面试中说清楚：

```text
causal mask 解决不能看未来的问题。
padding mask 解决不能看 padding token 的问题。
二者可以通过逻辑与合并。
```

### 9. 数值稳定性问题

`float("-inf")` 通常可以用。

但在半精度训练中，有时会使用一个很小的负数，例如：

```python
scores = scores.masked_fill(mask == 0, torch.finfo(scores.dtype).min)
```

如果整行都被 mask 掉，softmax 可能得到 NaN。

正常 attention 中应该避免某个 query 没有任何可关注 token。

### 10. Dropout 的位置

标准 Transformer 中 dropout 通常加在 attention weights 上：

```python
attn = F.dropout(attn, p=dropout_p, training=self.training)
```

面试手写函数里如果没有传 `training`，可以不写 dropout，或者说明真实模块中会在 `nn.Module` 里用 `self.dropout(attn)`。

上面的教学代码用了 `training=True`，更适合作为独立函数演示。正式模块里应使用模块状态控制。

### 11. 写成 nn.Module 版本

如果面试官要求写模块，可以写：

```python
import math
import torch
import torch.nn as nn
import torch.nn.functional as F


class ScaledDotProductAttention(nn.Module):
    def __init__(self, dropout: float = 0.0):
        super().__init__()
        self.dropout = nn.Dropout(dropout)

    def forward(self, q, k, v, mask=None):
        d_k = q.size(-1)
        scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(d_k)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, torch.finfo(scores.dtype).min)
        attn = F.softmax(scores, dim=-1)
        attn = self.dropout(attn)
        out = torch.matmul(attn, v)
        return out, attn
```

这个版本更适合真实训练代码。

### 13. 复杂度分析

Attention 的主要计算是 `QK^T` 和 `attn @ V`。

如果序列长度为 `T`，head dim 为 `D`，复杂度约为：

```text
时间复杂度：O(B * H * T^2 * D)
空间复杂度：O(B * H * T^2)
```

瓶颈来自 attention matrix `[T, T]`。

这也是长上下文模型需要 FlashAttention、稀疏 attention、linear attention、MQA/GQA、KV Cache 等优化的原因。

### 14. 面试高频追问

#### 问法 1：为什么除以 sqrt(d_k)？

可以这样答：

```text
如果 q 和 k 的每个维度近似独立且方差为 1，那么点积 q·k 的方差大约会随 d_k 增大而增大。维度越大，attention score 越容易变得很大或很小，softmax 会变得过于尖锐，梯度也不稳定。除以 sqrt(d_k) 可以把 score 的尺度拉回更稳定的范围。
```

#### 问法 2：mask 为什么要加在 softmax 前？

可以这样答：

```text
因为 softmax 前把被 mask 的位置设为负无穷，softmax 后这些位置的概率就是 0，同时剩余位置会自动重新归一化。如果 softmax 后再置 0，权重和会小于 1，除非再做一次归一化，比较麻烦也容易错。
```

#### 问法 3：padding mask 和 causal mask 有什么区别？

可以这样答：

```text
padding mask 是为了不关注 batch padding 出来的无效 token，通常和输入的 attention_mask 有关。causal mask 是为了自回归生成时不能看未来 token，通常是下三角矩阵。padding mask 解决无效位置问题，causal mask 解决信息泄漏问题，二者可以合并。
```

#### 问法 4：Attention 的复杂度为什么高？

可以这样答：

```text
因为每个 token 都要和所有 token 计算相关性，所以 attention score 是 T×T 的矩阵。序列长度翻倍，attention 矩阵大小变成四倍，因此长上下文场景下时间和显存压力都很大。
```

#### 问法 5：训练和推理中的 Attention 有什么不同？

可以这样答：

```text
训练时通常并行处理整个序列，用 causal mask 防止看到未来。推理时是自回归逐 token 生成，如果每一步都重算历史会很慢，所以会使用 KV Cache 缓存过去 token 的 key/value，只对新 token 做增量计算。
```

### 15. 常见手写错误

#### 15.1 transpose 维度写错

应该是：

```python
k.transpose(-2, -1)
```

不要把 head 维和 seq 维转错。

#### 15.2 softmax 维度写错

应该对 key 维度 softmax：

```python
F.softmax(scores, dim=-1)
```

#### 15.3 mask 语义混乱

要提前说清楚：

```text
mask=1 表示保留，mask=0 表示屏蔽。
```

不同框架中 mask 语义可能相反，面试中自己实现时要定义清楚。

#### 15.4 忘记缩放

漏掉 `/ sqrt(d_k)` 是高频错误。

#### 15.5 mask 加在 softmax 后

这会导致权重没有重新归一化。

#### 15.6 忽略 batch 和 head 维度

只写二维矩阵版本不够，最好写支持 `[B, H, T, D]` 的版本。

### 17. 本讲小结

这一讲训练了 10 分钟手写 Attention。

核心结论如下：

1. 面试手写 Attention 要先写公式，再定义 shape，再写代码。
2. 标准公式是 `softmax(QK^T / sqrt(d_k))V`。
3. 推荐输入形状使用 `[B, H, T, D]`，更贴近多头注意力内部实现。
4. mask 应在 softmax 前加入，被屏蔽位置设为负无穷。
5. causal mask 防止看未来，padding mask 防止关注无效 token。
6. softmax 应沿 key 维度，也就是 `dim=-1`。
7. Attention 的主要瓶颈是 `[T, T]` attention matrix，复杂度随序列长度平方增长。
8. 推理时使用 KV Cache 避免重复计算历史 key/value。
9. 面试中要能解释代码、形状、公式、mask、缩放因子和复杂度。
10. 高频错误包括 transpose 维度错、softmax 维度错、忘记缩放、mask 语义混乱。

### 2. 一句话定义 GPT

可以这样开场：

```text
GPT 是一种基于 Decoder-only Transformer 的自回归语言模型。它把文本离散成 token 序列，通过 causal self-attention 只能关注当前位置及之前的 token，并以 next-token prediction 为训练目标，学习根据上下文预测下一个 token。推理时从 prompt 开始逐 token 生成，直到遇到停止条件。
```

这句话要包含 5 个关键词：

```text
Decoder-only
Transformer
自回归
causal self-attention
next-token prediction
```

### 3. 输入表示

GPT 的输入不是原始字符串，而是 token id 序列。

流程：

```text
文本 -> tokenizer -> token ids -> token embedding -> position embedding -> hidden states
```

可以这样讲：

```text
首先 tokenizer 把文本切成 token，例如 BPE、SentencePiece 或 tiktoken 风格 tokenizer。每个 token 被映射成一个整数 id。模型用 token embedding table 把 id 转成向量，同时加入位置编码或位置相关机制，让模型知道 token 的顺序。早期 GPT 常用 learned absolute position embedding，现代 LLM 更多使用 RoPE 等相对位置机制。
```

如果面试官追问 tokenizer，你可以补充：

```text
tokenizer 会影响词表大小、序列长度、多语言效果和未知字符处理。中文如果按字切会序列较长，BPE/SentencePiece 能在词表规模和序列长度之间折中。
```

### 4. GPT 主干结构

GPT 主体由多层 Transformer Decoder Block 堆叠而成。

整体结构：

```text
input token ids
-> token embedding + position information
-> Transformer Block x N
-> final LayerNorm
-> LM Head
-> logits over vocabulary
```

每个 block 通常包含：

```text
LayerNorm
Causal Self-Attention
Residual Connection
LayerNorm
MLP / FFN
Residual Connection
```

现代 GPT 类模型多使用 Pre-LN：

```text
x = x + Attention(LN(x))
x = x + MLP(LN(x))
```

Pre-LN 的好处是训练更稳定，尤其在深层 Transformer 中更常见。

### 5. Causal Self-Attention

GPT 的核心是 causal self-attention。

可以这样讲：

```text
self-attention 表示 Q、K、V 都来自同一个序列。causal 表示第 t 个位置只能看到第 t 个及之前的位置，不能看到未来 token。实现上通过下三角 causal mask，把未来位置的 attention score 设为负无穷，softmax 后权重为 0。
```

公式：

```text
Attention(Q, K, V) = softmax(QK^T / sqrt(d_k) + mask) V
```

为什么必须 causal？

```text
因为 GPT 的训练目标是预测下一个 token。如果训练时当前位置能看到未来 token，模型会偷看答案，训练 loss 会虚低，但推理时没有未来信息，训练和推理不一致。
```

### 6. Multi-Head Attention

Multi-head attention 的作用是让模型在不同子空间中关注不同关系。

可以这样讲：

```text
单个 attention head 只能用一组 Q/K/V 投影建模关系。Multi-head attention 把 hidden dimension 分成多个 head，每个 head 独立做 attention，然后把结果 concat 后再线性投影。不同 head 可以关注不同模式，比如局部依赖、长程依赖、语法关系或特殊标记。
```

形状可以简要说：

```text
输入 [B, T, hidden]
投影成 Q/K/V
reshape 为 [B, H, T, D]
每个 head 做 attention
concat 回 [B, T, hidden]
```

### 7. MLP / FFN

Attention 负责 token 间信息交互，MLP 负责逐位置非线性变换。

可以这样讲：

```text
Transformer block 里的 MLP 对每个 token 位置独立作用，通常先升维到 4 倍左右，再经过激活函数，再降回 hidden size。早期常用 GELU，现代 LLM 常用 SwiGLU 或 GeGLU 变体。Attention 负责混合不同位置的信息，MLP 负责在每个位置上增强表达能力。
```

### 8. Residual 和 LayerNorm

Residual connection 的作用：

```text
缓解深层网络训练困难，让梯度更容易传播。
```

LayerNorm 的作用：

```text
稳定 hidden state 分布，改善训练稳定性。
```

Pre-LN 与 Post-LN：

```text
Post-LN 是原始 Transformer 常见结构，LayerNorm 放在残差之后。
Pre-LN 是现代大模型常见结构，LayerNorm 放在子层之前，训练深层网络更稳定。
```

### 9. LM Head 和训练目标

最终 hidden state 经过 LM head 投影到词表大小：

```text
hidden: [B, T, hidden]
logits: [B, T, vocab_size]
```

训练目标是 next-token prediction。

给定 token 序列：

```text
x = [t0, t1, t2, ..., t_{n-1}]
label = [t1, t2, t3, ..., t_n]
```

模型在每个位置预测下一个 token。

loss 是交叉熵：

```text
L = - sum log p(t_{i+1} | t_0, ..., t_i)
```

可以这样讲：

```text
GPT 预训练不需要人工标签，文本本身就是监督信号。只要有大规模文本，就能通过右移一位构造训练目标。
```

### 10. 训练阶段

训练时，GPT 可以并行处理整个序列。

虽然目标是自回归，但因为真实 token 都已知，训练时可以一次性输入完整序列，并用 causal mask 保证每个位置只看过去。

可以这样讲：

```text
训练阶段不是一个 token 一个 token 地跑，而是并行计算所有位置的 logits。causal mask 保证第 i 个位置只使用 i 之前的信息。这样训练效率高，能充分利用 GPU 并行。
```

### 11. 推理阶段

推理时从 prompt 开始逐 token 生成。

流程：

```text
输入 prompt
计算最后一个位置 logits
根据 decoding strategy 选择下一个 token
把新 token 拼回上下文
重复直到达到 max_tokens 或 stop token
```

常见 decoding：

```text
greedy search
beam search
temperature sampling
```

GPT 推理的关键特点：

```text
训练可以并行，推理必须自回归串行生成。
```

这也是推理延迟高的重要原因。

### 12. KV Cache

如果推理时每生成一个 token 都重新计算整个上下文，效率很低。

KV Cache 用于缓存历史 token 的 key/value。

可以这样讲：

```text
在自回归推理中，过去 token 的 key/value 不会变化。KV Cache 把每一层 attention 的历史 key/value 缓存起来。生成新 token 时，只计算新 token 的 Q/K/V，并让新 token 的 Q 去 attend 历史 K/V cache 和当前 K/V。这样避免重复计算历史上下文，大幅提升 decode 阶段速度。
```

代价：

```text
KV Cache 会占用显存，显存占用与 batch size、层数、head 数、head dim 和上下文长度相关。
```

### 13. 复杂度分析

对于序列长度 `T`，hidden size `d`，attention 的主要复杂度来自 attention matrix。

```text
Attention 时间复杂度约 O(T^2 d)
Attention 空间复杂度约 O(T^2)
```

序列越长，attention 成本增长越快。

MLP 复杂度通常约为：

```text
O(T d^2)
```

在长上下文场景中，attention 的 `T^2` 瓶颈尤其明显。

### 14. GPT 的优点

可以总结为：

```text
结构统一，训练目标简单。
可扩展性好，参数、数据和算力增加时效果提升明显。
生成能力强，适合开放式文本生成。
预训练任务不需要人工标注，能利用海量文本。
可以通过 SFT、RLHF、DPO 等方式对齐到助手行为。
```

### 15. GPT 的局限

也要主动讲局限：

```text
自回归生成串行，推理延迟较高。
标准 attention 对长上下文成本高。
容易产生幻觉，因为参数知识不等于事实数据库。
训练数据可能带来偏见和安全风险。
缺少外部工具时，精确计算和实时知识能力有限。
上下文窗口有限，超长文档需要 RAG 或长上下文优化。
```

主动讲局限会显得更成熟。

### 16. 现代 GPT 的常见改进

可以按模块讲：

```text
位置编码：从 absolute position embedding 到 RoPE、ALiBi 等。
Attention：从 MHA 到 MQA/GQA，降低 KV Cache 显存。
MLP：从 GELU FFN 到 SwiGLU/GeGLU。
Norm：Pre-LN、RMSNorm。
训练：混合精度、ZeRO、tensor parallel、pipeline parallel。
推理：KV Cache、FlashAttention、PagedAttention、continuous batching、量化。
对齐：SFT、RLHF、DPO、安全微调。
```

不要一次讲太细，但要让面试官知道你了解现代 LLM 不是原始 GPT-1 的简单复刻。

### 17. 20 分钟口述模板

下面是一段可以直接练习的口述版本。

```text
GPT 是 Decoder-only Transformer 的自回归语言模型。它先用 tokenizer 把文本转成 token id，然后通过 token embedding 和位置编码得到输入表示。主体是多层 Transformer Decoder Block，每个 block 通常包含 Pre-LN、causal self-attention、残差连接、MLP 和残差连接。

causal self-attention 是 GPT 的关键，它通过下三角 mask 保证当前位置只能关注自己和之前的 token，不能看到未来 token。这样训练时模型不会偷看答案，和推理时的自回归生成保持一致。attention 的公式是 softmax(QK^T / sqrt(d_k) + mask)V。

训练目标是 next-token prediction。给定一段 token 序列，模型在每个位置预测下一个 token，使用交叉熵 loss。训练时虽然是自回归目标，但可以并行输入整个序列，用 causal mask 限制信息流。

推理时从 prompt 开始逐 token 生成，每一步根据最后位置 logits 通过 greedy、top-k、top-p 或 temperature sampling 选择下一个 token，再拼回上下文继续生成。为了避免每一步重复计算历史 token，推理时会使用 KV Cache 缓存每一层历史 token 的 key/value。

GPT 的优点是结构统一、目标简单、扩展性强、生成能力好；局限是自回归推理串行、attention 长上下文成本高、容易幻觉、实时知识和精确计算能力有限。现代 LLM 在 GPT 基础上做了很多工程改进，例如 RoPE、RMSNorm、SwiGLU、GQA/MQA、FlashAttention、PagedAttention、量化和 SFT/RLHF/DPO 对齐。
```

这段话控制在 3-5 分钟。面试官如果继续追问，你再展开细节。

### 18. 面试高频追问

#### 问法 1：GPT 和 BERT 有什么区别？

可以这样答：

```text
GPT 是 Decoder-only 自回归模型，用 causal mask，只能看左侧上下文，训练目标是 next-token prediction，适合生成任务。BERT 是 Encoder-only 双向模型，可以同时看左右上下文，训练目标主要是 masked language modeling，更适合理解类任务。GPT 的优势在开放式生成，BERT 的优势在分类、匹配、抽取等理解任务。
```

#### 问法 2：GPT 为什么能训练时并行，推理时不能并行？

可以这样答：

```text
训练时完整序列已经已知，所以可以一次性输入所有 token，用 causal mask 保证每个位置只看过去，并并行计算所有位置的 loss。推理时未来 token 还不存在，必须先生成第 t+1 个 token，才能把它作为上下文生成第 t+2 个 token，所以 decode 是自回归串行的。
```

#### 问法 3：GPT 为什么容易幻觉？

可以这样答：

```text
GPT 的训练目标是根据上下文预测下一个 token，本质上学习文本分布，不等价于维护一个可验证事实数据库。当 prompt 要求它回答不确定或训练中少见的信息时，它可能生成概率上合理但事实错误的内容。缓解方法包括 RAG、工具调用、校准训练、安全对齐、引用约束和不确定性表达。
```

#### 问法 4：为什么现代 LLM 常用 GQA/MQA？

可以这样答：

```text
标准 MHA 中每个 query head 都有自己的 key/value head，KV Cache 显存较大。MQA 让多个 query head 共享一组 key/value，GQA 则让一组 query heads 共享 key/value，在效果和显存之间折中。它们主要用于降低推理阶段 KV Cache 显存和带宽压力，提高吞吐。
```

#### 问法 5：GPT 架构如何支持长上下文？

可以这样答：

```text
长上下文主要受位置编码、attention 复杂度和训练长度分布限制。改进方向包括 RoPE 扩展、ALiBi、滑动窗口 attention、稀疏 attention、FlashAttention 降低显存、PagedAttention 管理 KV Cache，以及 RAG 把外部文档以检索方式接入上下文。实际效果还要看模型是否在长上下文数据上训练过。
```

### 19. 常见错误回答

#### 19.1 只说 Decoder-only

只说“GPT 是 Decoder-only”太浅，要讲结构、目标、mask、训练和推理。

#### 19.2 把 GPT Decoder 和原始 Transformer Decoder 混淆

原始 Transformer Decoder 包含 masked self-attention 和 encoder-decoder cross-attention。GPT 是 decoder-only，没有 encoder，也没有标准 cross-attention。

#### 19.3 不讲 causal mask

causal mask 是 GPT 自回归建模的核心。

#### 19.4 不区分训练和推理

训练并行，推理串行，这是 GPT 架构和工程优化的关键点。

#### 19.5 只讲原始 GPT，不讲现代 LLM 改进

面试大模型算法岗时，应补充 RoPE、RMSNorm、SwiGLU、GQA/MQA、KV Cache 等现代改造。

### 21. 本讲小结

这一讲训练了 20 分钟讲清楚 GPT 架构。

核心结论如下：

1. GPT 是 Decoder-only Transformer 的自回归语言模型。
2. 输入流程是 tokenizer、token embedding、位置编码或位置机制。
3. 主干由多层 Transformer Decoder Block 组成，包括 causal self-attention、MLP、LayerNorm 和残差连接。
4. causal mask 保证训练时不能看到未来 token，避免信息泄漏。
5. 训练目标是 next-token prediction，使用交叉熵 loss。
6. 训练时可以并行处理完整序列，推理时必须逐 token 自回归生成。
7. KV Cache 缓存历史 key/value，提升推理 decode 效率。
8. GPT 的主要瓶颈包括自回归串行推理和 attention 长上下文成本。
9. 现代 GPT 类模型常见改进包括 RoPE、RMSNorm、SwiGLU、GQA/MQA、FlashAttention、PagedAttention 和对齐训练。
10. 面试回答要有结构、有因果、有边界，不要只背“Decoder-only”。

### 5. 模拟项目一：中文 SFT 助手

#### 面试官：你这个中文指令微调项目解决什么问题？

候选人可以答：

```text
这个项目主要解决 base model 在中文指令遵循和结构化输出上不稳定的问题。base model 虽然有一定中文能力，但在 JSON 输出、信息抽取、多步骤回答和特定领域问答上容易格式错误或回答不完整。所以我构造了中文 SFT 数据，覆盖通用问答、格式遵循、文本改写、信息抽取和领域问答，并使用 LoRA 做监督微调。
```

#### 面试官：你的 SFT 数据怎么构造？

可以答：

```text
我用 messages 格式组织数据，每条样本包含 system、user 和 assistant。数据分成几类：通用问答、格式遵循、信息抽取、文本改写、领域问答和安全边界。构造后做了去重、长度过滤和任务分布统计，并划分 train/dev/test。训练时只对 assistant answer 部分计算 loss，system 和 user 部分 label 设为 -100。
```

#### 面试官：为什么只对 assistant 部分算 loss？

可以答：

```text
因为 system 和 user 是条件，不是希望模型生成的目标。如果把 prompt 也纳入 loss，模型会被训练去复述用户输入，影响助手行为。SFT 的目标是在给定指令条件下生成 assistant answer，所以工程上通常把非 assistant 部分 mask 成 -100，让 CrossEntropyLoss 忽略。
```

#### 面试官：你怎么证明 SFT 有效果？

可以答：

```text
我做了 base model 和 SFT model 的对比。自动评估上看 dev loss、格式合法率和拒答率；人工评估上按事实正确性、完整性、指令遵循、格式正确性和安全性打分；同时做 pairwise preference。结果不是只看平均分，而是按任务切片看，例如格式遵循任务提升最明显，但开放问答里出现了一些模板化输出，这是后续需要混入更多通用对话数据解决的问题。
```

### 6. 模拟项目二：DPO 偏好优化

#### 面试官：为什么 SFT 后还要做 DPO？

可以答：

```text
SFT 主要是模仿标准答案，让模型学会基本指令响应。但很多时候我们不仅希望模型能回答，还希望它在多个可行回答中选择更符合人类偏好的回答，比如更完整、更安全、更符合格式、更少幻觉。DPO 使用 chosen/rejected 偏好对，优化同一 prompt 下模型对 chosen 相对 rejected 的偏好概率，因此适合在 SFT 后进一步对齐回答风格和质量。
```

#### 面试官：你的偏好数据怎么保证质量？

可以答：

```text
我会给每个偏好对保留 category 和 reason，确保 chosen 在明确维度上优于 rejected，例如事实正确性、完整性、格式遵循、安全边界或拒答合理性。避免 chosen 只是更长或更短这种伪偏好。构造后会抽样复查，并按任务类型统计分布，防止某类偏好过多导致模型学偏。
```

#### 面试官：DPO 后可能有什么副作用？

可以答：

```text
DPO 可能导致回答变短、过度拒答、风格模板化或通用能力退化。如果偏好数据里 chosen 普遍更短，模型会学到短就是好；如果安全拒答样本比例过高，模型可能对正常问题也拒答。所以我会评估 pairwise 胜率之外的指标，包括平均输出长度、安全拒答合格率、过度拒答率、格式遵循和通用能力回归集。
```

### 7. 模拟项目三：企业知识库 RAG

#### 面试官：你的 RAG 系统为什么不用微调？

可以答：

```text
企业知识库的特点是知识更新频繁、内容私有、需要引用来源。微调会把知识写进参数里，更新成本高，也不容易追溯来源。RAG 可以在不改模型参数的情况下接入外部文档，知识更新只需要更新索引，而且可以返回引用，便于审计。所以这个场景优先用 RAG。
```

#### 面试官：RAG 答错了你怎么排查？

可以答：

```text
我会分层排查。第一看原文档是否被正确解析和入库。第二看 chunk 切分是否把关键条件切开。第三看 retriever top K 是否召回了相关 chunk，如果没召回就是检索问题。第四看 reranker 是否把相关 chunk 排到前面。第五看正确证据是否进入 prompt，如果进入了但答案仍错，就是生成或 prompt 约束问题。最后还要检查引用是否支持答案，以及权限过滤是否正确。
```

#### 面试官：chunk size 怎么选？

可以答：

```text
chunk size 是完整性和精准性的权衡。太小会丢上下文，太大会引入噪声并占用 prompt token。我会优先按标题和段落做结构化切分，超过长度再递归切分，并设置 overlap。实验上比较不同 chunk size 和 overlap 下的 Recall@K、MRR、groundedness 和平均上下文 token，选择综合效果最好的配置。
```

### 8. 模拟项目四：推理服务

#### 面试官：你的推理服务怎么衡量性能？

可以答：

```text
我会分服务指标和模型指标。服务指标包括 QPS、并发数、P50/P95/P99 延迟、错误率和超时率。模型指标包括 TTFT、tokens/s、输入输出 token 数、显存占用和 GPU 利用率。对于聊天产品，TTFT 影响用户感知响应速度，P95/P99 比平均延迟更能反映线上体验。
```

#### 面试官：怎么降低推理延迟？

可以答：

```text
先拆分延迟来源：排队、tokenize、prefill、decode 和网络传输。优化上可以控制 prompt 长度和 max_tokens，启用 KV Cache，使用半精度或量化，接入 vLLM/TGI 这类支持 continuous batching 的后端，使用流式输出降低感知延迟，并通过限流和队列控制避免高并发下尾延迟爆炸。
```

#### 面试官：流式输出是不是让模型更快？

可以答：

```text
不一定。流式输出主要降低用户感知延迟，让用户更早看到第一个 token 或第一段文本。完整生成时间可能没有变。它改善的是交互体验，而不是必然提升模型计算速度。
```

### 9. 模拟项目五：数学推理增强

#### 面试官：你怎么证明 CoT 有效？

可以答：

```text
我会构造数学推理评测集，按题型和难度切片，然后比较 zero-shot direct、few-shot direct、CoT 和 self-consistency。指标不仅看总准确率，还看多步题、干扰信息题、单位换算题等切片表现，同时统计平均输出 token 和推理成本。如果 CoT 只在多步题提升，而在简单题成本增加明显，说明它适合按场景使用，而不是所有请求默认启用。
```

#### 面试官：Self-consistency 的成本怎么控制？

可以答：

```text
Self-consistency 需要对同一道题采样多条推理路径，成本大约随采样次数线性增长。所以我不会默认所有问题都用，而是用于高价值、复杂或低置信度问题。也可以做 n=3/5/10 的消融，找到准确率提升和成本之间的平衡点。
```

### 10. 如何回答“不知道”

项目深挖时难免遇到不会的问题。

不要硬编。

更好的回答结构是：

```text
这个点我项目里没有完整实现，但我理解它的方向是 X。如果要做，我会先 Y，再 Z，最后用指标 A/B 来验证。
```

例如：

```text
我这个项目没有真正上线到生产环境，所以没有真实用户 A/B 数据。但如果要上线，我会先做小流量灰度，按用户级或会话级分桶，监控问题解决率、负反馈率、P95 延迟、安全违规率和 token 成本，并保留快速回滚。
```

承认边界，比假装做过更可信。

### 11. 如何讲 bad case

bad case 不要只说：

```text
有些效果不好。
```

要按结构讲：

```text
现象是什么？
影响什么指标？
定位到哪一层？
根因是什么？
尝试了什么修复？
修复后效果如何？
还有什么遗留问题？
```

示例：

```text
RAG 项目里一个 bad case 是差旅报销问题答错。现象是用户问“没有纸质发票能否报销”，模型回答可以，但制度中要求电子发票或替代证明。排查发现相关条款在 PDF 表格里，解析时表格列丢失，导致 chunk 中只保留了“可报销”而丢了条件。后续把表格转成 Markdown，并在评测集中加入表格类问题，Recall@5 和 citation accuracy 都有提升。
```

### 12. 如何讲个人贡献

不要说：

```text
我们做了一个系统。
```

要说清楚你负责什么：

```text
我主要负责三块：第一是文档解析和 chunk 切分策略；第二是 hybrid retrieval + rerank 的召回链路；第三是评估集构造和 bad case 分析。模型服务部署由另一位同学负责，但我参与了 API 字段和日志设计。
```

这比“全程参与”更可信。

### 13. 如何讲指标

项目指标要满足三个要求：

```text
有 baseline
有改进值
有解释
```

不好的说法：

```text
效果提升明显。
```

好的说法：

```text
在 300 条评测集上，向量召回 baseline 的 Recall@5 是 78%，加入 BM25 混合召回和 reranker 后提升到 89%。提升主要来自制度编号、产品名和错误码这类关键词强匹配问题。
```

如果没有真实数字，也可以诚实说：

```text
这是个人复现实验，不适合声称业务指标。我主要报告离线评估结果，例如 dev loss、格式合法率、Recall@K 和人工抽检胜率。
```

### 14. 项目深挖高频雷区

#### 14.1 夸大项目规模

个人项目不要说成生产级大系统。

#### 14.2 只堆技术名词

LangChain、FAISS、LoRA、DPO、vLLM 只是工具，面试官要听选择理由和效果。

#### 14.3 没有 baseline

没有 baseline，就无法证明你的改进有效。

#### 14.4 没有 bad case

没有失败案例，说明你可能没有认真评估。

#### 14.5 讲不清数据

数据是大模型项目核心。数据来源、规模、质量和分布必须讲清。

#### 14.6 讲不清自己做了什么

团队项目一定要明确个人贡献。

面试前，每个简历项目准备一页纸。

### 16. 本讲小结

这一讲做了项目深挖模拟。

核心结论如下：

1. 项目深挖考察的是你是否真的做过、是否能解释选择、是否能复盘问题。
2. 项目介绍要按背景、目标、方案、结果、个人贡献和复盘组织。
3. 面试官常从数据、方法、实现、评估、对比、bad case 和改进方向追问。
4. RAG 项目要讲清解析、切分、召回、rerank、生成、引用和评估。
5. SFT 项目要讲清数据格式、chat template、loss mask、LoRA 配置和评估。
6. DPO 项目要讲清 chosen/rejected 数据、reference model、beta、副作用和偏好评估。
7. 推理服务项目要讲清 TTFT、P95/P99、KV Cache、流式输出、限流和监控。
8. 数学推理项目要讲清 CoT、self-consistency、答案抽取、工具校验和错误类型。
9. 不会的问题不要硬编，要说明项目边界和可行改进方案。
10. 每个项目都要准备 baseline、指标、bad cases、个人贡献和下一步改进。

## 第 60 讲：开放题模拟

开放题是大模型算法岗面试中最能体现工程判断的问题类型。

这类题通常没有唯一标准答案，例如：

```text
如果让你提升一个客服大模型的回答质量，你会怎么做？
如果模型微调后出现幻觉增加，你怎么排查？
如果业务方要求把大模型成本降低一半，你怎么设计方案？
如果让你做一个面向企业的 Agent，你会怎么落地？
```

开放题考察的不是你背了多少名词，而是你是否能把模糊问题变成可执行方案。

一个好的开放题回答应该具备：

```text
先澄清目标
再拆解问题
提出 baseline
设计方案
给出评估指标
说明风险和权衡
最后讲迭代路径
```

这一讲，我们做开放题模拟训练。

### 本讲目标

学完本讲，你需要掌握：

1. 开放题的通用答题框架。
2. 如何把模糊问题拆成数据、模型、系统、评估和业务目标。
3. 如何回答质量提升、成本优化、幻觉排查、能力退化等开放题。
4. 如何避免空泛回答。
5. 如何体现算法岗候选人的工程闭环能力。

### 1. 开放题通用框架

推荐使用下面的框架：

```text
1. 澄清目标
2. 定义指标
3. 建立 baseline
4. 拆解问题
5. 设计方案
6. 实验验证
7. 风险控制
8. 迭代上线
```

面试中可以先说：

```text
我会先澄清这个问题的目标和约束，然后从数据、模型、系统和评估四个层面拆解。
```

这句话能让面试官知道你不是随便发散。

### 2. 澄清问题

开放题第一步不是直接给方案，而是问清楚目标。

例如面试官问：

```text
如何提升大模型回答质量？
```

你应该先澄清：

```text
这是通用问答、客服、RAG、代码助手还是 Agent？
质量问题主要是事实错误、回答不完整、格式不稳定、幻觉、安全拒答，还是用户满意度低？
有没有延迟、成本、模型大小、数据隐私等约束？
是离线优化还是线上系统优化？
```

如果面试官不提供更多信息，你可以自己设定合理场景：

```text
如果没有更多限制，我先假设这是一个企业客服 RAG 问答场景，目标是提升问题解决率并降低幻觉。
```

### 3. 定义指标

开放题最忌讳只说“提升效果”。

要把目标变成指标：

```text
回答正确率
groundedness
幻觉率
问题解决率
用户满意率
负反馈率
人工转接率
格式合法率
安全违规率
P95 延迟
token 成本
```

指标要分层：

```text
主指标：决定是否成功
护栏指标：不能恶化
诊断指标：解释为什么变化
```

### 4. 建立 baseline

开放题中一定要有 baseline。

例如：

```text
当前 prompt + 当前模型
base model vs SFT model
向量召回 vs hybrid retrieval
无 rerank vs 有 rerank
普通推理 vs vLLM 推理
直接回答 vs CoT/self-consistency
```

没有 baseline，后面方案无法证明有效。

### 5. 拆解问题

大模型问题通常可以从五层拆：

```text
数据层：训练数据、评测数据、知识库、偏好数据
模型层：base model、SFT、DPO、量化、推理策略
检索层：query rewrite、embedding、BM25、rerank、chunk
系统层：服务、缓存、限流、监控、A/B、回滚
评估层：自动评测、人工评测、线上指标、bad case
```

开放题回答要体现这种分层思维。

### 6. 模拟题一：如何提升客服大模型回答质量？

#### 参考回答

```text
我会先澄清客服模型的主要问题。如果是事实错误和幻觉，我会优先从知识库和 RAG 链路入手；如果是风格和指令遵循问题，可能需要 SFT 或 prompt 优化；如果是用户偏好问题，可以考虑 DPO 或人工反馈闭环。

指标上，我会定义主指标为问题解决率和用户满意率，护栏指标包括安全违规率、人工转接率、P95 延迟和 token 成本，诊断指标包括 groundedness、检索 Recall@K、格式合法率、平均输出长度和负反馈标签分布。

方案上，第一步建立 baseline，包括当前模型、当前 prompt 和当前 RAG 链路。第二步构造评测集，覆盖高频问题、长尾问题、多轮问题、安全敏感问题和知识库缺失问题。第三步分层优化：数据层清洗 FAQ 和业务文档；检索层优化 chunk、embedding、BM25+向量混合召回和 reranker；生成层设计 grounded prompt，要求资料不足时明确说明；模型层根据 bad case 决定是否做 SFT 或 DPO；系统层增加日志、人工反馈和 A/B。

最后我会用离线评测和线上灰度结合验证。离线看 Recall@K、groundedness、人工偏好胜率和幻觉率；线上看问题解决率、负反馈率、人工转接率、安全告警和成本。如果主指标提升且护栏不退化，再逐步扩大流量。
```

### 7. 模拟题二：模型微调后幻觉增加，怎么排查？

#### 参考回答

```text
我会先确认幻觉增加是在哪些场景发生，是通用问答、领域问答、RAG 问答还是安全边界。然后对比 base model、SFT model 和不同 checkpoint，确认是微调导致，还是检索或 prompt 变化导致。

数据层面，我会检查 SFT 数据中是否存在错误答案、无依据扩写、过度自信表述，以及是否缺少“不知道”或资料不足时拒答的样本。训练层面检查 learning rate、epoch、LoRA rank、target modules 和 loss mask，判断是否过拟合或更新过强。模板层面检查 train chat template 和 inference template 是否一致。

如果是 RAG 场景，还要检查相关证据是否进入 prompt。如果证据没召回，是 retriever 问题；如果证据召回了但模型仍编造，是生成约束和数据问题。

缓解上，可以清洗错误数据，加入 grounded SFT 样本和不确定性表达样本，混入通用 replay 数据，降低学习率或训练轮数，做 early stopping，并设置幻觉评测集作为上线门禁。
```

### 8. 模拟题三：如何把推理成本降低一半？

#### 参考回答

```text
我会先拆成本来源：模型规格、输入 token、输出 token、并发调度、GPU 利用率、缓存命中率和重试率。然后定义约束：质量不能明显下降，P95 延迟不能超过阈值，安全指标不能退化。

优化可以分几层。模型层可以评估更小模型、蒸馏模型、量化 INT8/INT4、LoRA merge 后部署。输入层可以做 prompt 压缩、RAG top_k 控制、去除无关上下文和历史对话摘要。输出层可以限制 max_tokens，优化回答模板，避免冗长生成。系统层可以接入 vLLM 做 continuous batching，提高 GPU 利用率；对高频问题做缓存；按任务路由不同模型，例如简单问题用小模型，复杂问题用大模型。

验证上，我会比较每千请求成本、平均输入输出 token、P95 延迟、用户满意率、人工评测质量和安全指标。只有在质量护栏不下降的前提下，才认为成本优化成功。
```

### 9. 模拟题四：如何设计一个模型能力退化评估体系？

#### 参考回答

```text
我会把能力退化分成目标任务退化、通用能力退化、指令遵循退化、格式退化、安全退化和风格退化。评估体系要覆盖 base model、finetuned model 和多个 checkpoint。

数据上准备多套评测集：目标任务测试集、通用问答集、格式遵循集、安全拒答集、推理集和原模型能力回归集。指标上，分类任务看 accuracy/F1，生成任务看人工评测和 LLM-as-Judge，结构化任务看格式合法率，RAG 任务看 groundedness 和 citation accuracy，安全任务看违规率和过度拒答率。

分析上不只看平均分，而是按任务类型、长度、难度和风险等级切片。上线时设置硬门禁，例如安全违规率不能上升，事实正确性不能低于阈值，格式失败率不能超过阈值。如果发现退化，可以从数据分布、loss mask、学习率、LoRA 范围、训练轮数和偏好数据质量排查。
```

### 10. 模拟题五：如何让大模型更可靠地输出 JSON？

#### 参考回答

```text
我会从 prompt、解码、训练和后处理四层做。prompt 层明确 schema、字段类型、示例和禁止输出解释文本。解码层可以降低 temperature，必要时使用 constrained decoding 或 grammar decoding。训练层可以构造格式遵循 SFT 数据，让 assistant 学会稳定输出 JSON。后处理层用 JSON parser 校验，如果 parse 失败，可以触发修复 prompt 或重试。

评估指标包括 JSON parse 成功率、schema 校验通过率、字段完整率、字段内容正确率和重试率。上线时格式合法率应该是硬门禁，因为结构化输出失败会直接影响下游系统。
```

### 11. 模拟题六：如何处理用户问题超出知识库范围？

#### 参考回答

```text
在 RAG 场景中，知识库范围外问题不能让模型自由编造。我会先在检索阶段根据 top score、rerank score 或 evidence coverage 判断证据是否足够。如果证据不足，生成 prompt 要明确要求模型回答“根据当前资料无法确定”，并给出建议用户补充信息或联系人工渠道。

训练或提示上要加入资料不足的样本，让模型学会合理拒答。评估上要专门构造 out-of-scope 测试集，统计正确拒答率和过度拒答率。关键是平衡：该拒答时拒答，不该拒答时不能过度保守。
```

### 12. 模拟题七：如何评估 LLM-as-Judge 是否可靠？

#### 参考回答

```text
我不会直接相信 LLM-as-Judge。首先会构造一批人工高质量标注集作为 gold set，然后让 judge 对同一批样本打分，计算和人工的一致率、相关性或 Kappa。其次要分析 judge 的偏差，例如是否偏好长回答、是否偏好某种格式、是否对安全问题过宽或过严。还可以做 position bias 测试，交换 A/B 顺序看判断是否稳定。

如果一致性足够好，可以把 LLM-as-Judge 用于大规模初筛和回归测试，但高风险场景和最终上线决策仍然需要人工抽检。
```

### 13. 开放题回答的“强信号”

好的回答会出现这些信号：

```text
先澄清目标和场景
主动定义指标
有 baseline
分层拆解
有实验验证
有护栏指标
有 bad case 分析
有上线灰度和回滚
能讲 trade-off
能承认边界
```

这些信号说明你具备工程闭环能力。

### 14. 开放题回答的“弱信号”

弱回答通常是：

```text
直接堆名词
没有指标
没有数据
没有 baseline
没有风险
没有评估
不区分离线和线上
不区分模型问题和系统问题
不讲成本和延迟
```

例如：

```text
可以用 RAG、微调、RLHF、prompt engineering 提升效果。
```

这句话太空泛。你要说清楚什么时候用 RAG，什么时候用 SFT，什么时候用 DPO，它们解决的问题不同。

### 15. 开放题中的常见权衡

面试官喜欢听你讲 trade-off。

常见权衡包括：

```text
质量 vs 成本
质量 vs 延迟
召回率 vs 噪声
回答完整性 vs 简洁性
安全拒答 vs 过度拒答
模型微调 vs RAG
大模型效果 vs 小模型成本
人工评测质量 vs 标注成本
self-consistency 准确率 vs 推理成本
量化节省显存 vs 质量退化
```

回答开放题时，主动指出权衡，会显得更成熟。

### 17. 本讲小结

这一讲做了开放题模拟。

核心结论如下：

1. 开放题没有唯一答案，考察的是结构化拆解和工程判断。
2. 回答开放题要先澄清目标和约束，不要直接堆方案。
3. 要把“提升质量”这类模糊目标转成可评估指标。
4. 每个方案都要有 baseline，否则无法证明有效。
5. 大模型问题通常要从数据、模型、检索、系统和评估多层拆解。
6. 好回答要包含实验验证、护栏指标、bad case、风险和上线策略。
7. 成本、延迟、安全、质量之间存在 trade-off，必须主动说明。
8. RAG、SFT、DPO、prompt、推理优化各自解决的问题不同，不能混为一谈。
9. LLM-as-Judge 可以辅助评估，但需要人工标注集校准。
10. 练习开放题时固定使用“目标、指标、baseline、方案、实验、风险、上线”框架。

## 第 61 讲：系统设计模拟

大模型算法岗的系统设计题，不是传统后端面试里的“设计一个短链接系统”或“设计一个消息队列”。

它更关注你是否能把模型能力、数据链路、在线服务、评估体系、安全策略和业务指标连接起来。

常见题目包括：

```text
设计一个企业知识库问答系统。
设计一个客服大模型系统。
设计一个代码助手系统。
设计一个文档总结系统。
设计一个多轮 Agent 系统。
设计一个大模型评估平台。
设计一个高并发推理服务。
```

系统设计题没有唯一答案。面试官更看重：

```text
你是否先澄清需求。
你是否能画出清晰架构。
你是否知道关键模块怎么选型。
你是否能讲数据流和请求流。
你是否考虑评估、监控、安全、权限、成本和回滚。
你是否能讲 trade-off。
```

这一讲，我们训练大模型系统设计题的答题方法。

### 本讲目标

学完本讲，你需要掌握：

1. 大模型系统设计题的通用框架。
2. 如何澄清需求和约束。
3. 如何画出 RAG/Agent/推理服务的系统架构。
4. 如何讲清离线链路和在线链路。
5. 如何设计评估、监控、安全和权限。
6. 如何回答系统设计中的高频追问。

### 1. 系统设计通用框架

推荐使用下面的 8 步框架：

```text
1. 澄清需求
2. 定义指标
3. 画总体架构
4. 讲离线链路
5. 讲在线链路
6. 讲关键模块选型
7. 讲可靠性、安全和成本
8. 讲评估、监控和迭代
```

面试中可以先说：

```text
我会先明确业务目标、用户规模、延迟成本约束和安全要求，然后从离线数据链路、在线请求链路、模型服务、评估监控和安全权限几个部分设计。
```

### 2. 先澄清需求

系统设计题第一步必须澄清。

例如题目是：

```text
设计一个企业知识库问答系统。
```

你可以问：

```text
用户是谁？员工、客服还是外部客户？
文档类型有哪些？PDF、网页、Markdown、表格？
知识库规模多大？几百篇还是百万级文档？
是否需要权限控制？
是否要求引用来源？
延迟目标是多少？
是否支持多轮对话？
是否需要流式输出？
是否允许调用外部模型 API？
```

如果面试官不提供，你可以给出假设：

```text
我先假设这是企业内部员工使用的制度问答系统，文档规模 1 万篇以内，需要权限控制、引用来源、多轮追问和 3 秒内首 token 返回。
```

主动设定边界，能让回答更具体。

### 3. 定义指标

系统设计不是只画模块，还要定义成功标准。

指标可以分为：

```text
质量指标：答案正确率、groundedness、citation accuracy、问题解决率
检索指标：Recall@K、MRR、NDCG
服务指标：QPS、P95 延迟、TTFT、错误率、超时率
成本指标：单请求 token 成本、GPU 利用率、缓存命中率
安全指标：越权召回率、安全违规率、敏感信息泄露率
用户指标：满意率、负反馈率、人工转接率
```

要明确主指标和护栏指标。

例如：

```text
主指标：问题解决率、groundedness。
护栏指标：权限违规率为 0，P95 延迟小于 5 秒，安全违规率不升高。
```

### 4. 模拟题一：设计企业知识库问答系统

#### 4.1 总体架构

可以画成文字版：

```text
离线链路：
文档源 -> 文档解析 -> 清洗去重 -> chunk 切分 -> embedding -> 向量库/BM25 索引 -> 元数据和权限索引

在线链路：
用户问题 -> 权限识别 -> query rewrite -> hybrid retrieval -> ACL 过滤 -> rerank -> prompt 构造 -> LLM 生成 -> 引用返回 -> 日志监控
```

#### 4.2 离线链路

讲清楚：

```text
文档解析要保留标题、章节、表格、页码、来源和更新时间。
chunk 切分优先按结构切分，过长再递归切分，并设置 overlap。
每个 chunk 带 doc_id、chunk_id、source、section_path、acl、updated_at 等元数据。
embedding 后写入向量库，同时建立 BM25 索引用于关键词检索。
文档更新时按 doc_id 增量删除旧 chunk 并重建新 chunk。
```

#### 4.3 在线链路

讲清楚：

```text
用户问题进入后，先做登录态和权限识别。
多轮场景下先做 question rewrite，把上下文依赖问题改写成独立问题。
检索阶段使用向量召回和 BM25 混合召回，提升语义和关键词匹配能力。
召回后先按 ACL 过滤，避免越权文档进入 prompt。
reranker 对候选 chunk 重新排序，选择 top 5 进入 prompt。
prompt 要要求模型只基于证据回答，资料不足时说明无法确定，并返回引用来源。
```

#### 4.4 评估设计

```text
检索：Recall@K、MRR、NDCG。
生成：事实正确性、groundedness、citation accuracy、完整性、拒答合理性。
线上：满意率、负反馈率、人工转接率、P95 延迟、token 成本。
安全：权限违规率、敏感信息泄露率。
```

#### 4.5 高频追问：如何防止越权？

可以答：

```text
权限不能交给 LLM 自己判断。文档入库时要把 ACL 写入 chunk 元数据。在线检索时，根据用户身份拿到权限集合，召回后在进入 prompt 前做硬过滤。日志中也要记录 retrieved_chunk_ids 和用户权限，但敏感内容要脱敏。权限违规率应该作为硬门禁。
```

### 5. 模拟题二：设计客服大模型系统

#### 5.1 需求澄清

```text
客服场景是售前、售后还是内部客服？
是否需要接入订单、物流、退款等工具？
是否允许自动执行动作，还是只给建议？
是否需要人工转接？
安全和合规要求是什么？
```

#### 5.2 总体架构

```text
用户消息
-> 意图识别/风险识别
-> 对话状态管理
-> 知识库 RAG
-> 工具路由可选
-> LLM 生成回复
-> 安全审核
-> 返回用户或转人工
-> 日志、反馈、质检
```

#### 5.3 关键模块

```text
意图识别：区分 FAQ、订单查询、投诉、安全敏感、转人工。
RAG：回答制度、产品、流程类问题。
工具调用：查询订单、物流、库存、退款状态。
安全审核：避免违规承诺、隐私泄露和不当建议。
人工转接：低置信度、高风险、用户强烈不满时转人工。
质检系统：抽样评估回答质量和合规性。
```

#### 5.4 指标

```text
问题解决率
用户满意率
人工转接率
一次解决率
平均会话轮数
负反馈率
安全违规率
工具调用成功率
P95 延迟
单会话成本
```

#### 5.5 高频追问：什么时候转人工？

可以答：

```text
我会设置多种转人工触发条件：用户主动要求转人工；模型置信度低或检索证据不足；涉及投诉、退款、法律、医疗、金融等高风险场景；连续多轮未解决；安全审核发现风险；工具调用失败且影响用户权益。转人工不是失败，而是客服系统可靠性的一部分。
```

### 6. 模拟题三：设计大模型评估平台

#### 6.1 总体架构

```text
评测集管理 -> 模型版本管理 -> 推理任务调度 -> 自动指标计算 -> LLM-as-Judge -> 人工评测 -> 报告生成 -> bad case 管理 -> 上线门禁
```

#### 6.2 核心能力

```text
支持多模型版本对比
支持多任务评测集
支持自动指标和人工评测
支持 pairwise preference
支持安全和格式硬门禁
支持 bad case 回归
支持评测报告可视化
```

#### 6.3 指标设计

```text
分类任务：accuracy、F1
生成任务：人工评分、pairwise 胜率、LLM-as-Judge
RAG：Recall@K、groundedness、citation accuracy
结构化输出：parse success rate、schema pass rate
安全：违规率、过度拒答率
推理服务：延迟、成本、错误率
```

#### 6.4 高频追问：如何保证评估可信？

可以答：

```text
首先评测集要分层覆盖高频、长尾、困难、安全和格式约束场景。其次要固定模型版本、prompt 版本、采样参数和评测脚本，保证可复现。人工评测要有标注指南和一致性检查。LLM-as-Judge 要用人工 gold set 校准，不能直接作为最终结论。最后上线门禁要区分主指标和护栏指标。
```

### 7. 模拟题四：设计高并发推理服务

#### 7.1 总体架构

```text
API Gateway
-> 鉴权/限流
-> 请求队列
-> 调度器
-> 推理后端 vLLM/TGI/TensorRT-LLM
-> 流式返回
-> 日志监控
```

#### 7.2 关键设计

```text
请求校验：限制 max_tokens、输入长度、非法参数。
调度：continuous batching，提高 GPU 利用率。
KV Cache 管理：控制上下文长度和并发。
流式输出：降低用户感知延迟。
缓存：高频问题或系统 prompt 缓存。
限流：按用户、租户、全局并发限制。
回滚：模型版本快速切换。
```

#### 7.3 指标

```text
QPS
TTFT
P50/P95/P99 latency
tokens/s
GPU utilization
GPU memory
error rate
timeout rate
cost per 1k requests
```

#### 7.4 高频追问：如何处理长请求拖慢短请求？

可以答：

```text
长请求会占用 KV Cache 和 decode 资源，导致短请求排队。可以按输入长度和 max_tokens 做队列分级，限制单请求最大长度，使用 continuous batching 和 preemption 能力，必要时把长上下文请求路由到专门实例。还可以对历史对话做摘要，减少 prompt 长度。
```

### 8. 模拟题五：设计 Agent 系统

#### 8.1 总体架构

```text
用户任务
-> 任务理解
-> 规划器 Planner
-> 工具选择
-> 工具执行
-> 观察结果 Observation
-> 反思/修正
-> 最终回答
```

#### 8.2 关键模块

```text
工具注册表：工具名称、参数 schema、权限和风险等级。
Planner：拆解任务和选择工具。
Executor：执行工具调用。
Memory：保存短期对话状态和长期用户偏好。
Verifier：检查工具结果和最终答案。
Safety Gate：阻止高风险动作。
Human-in-the-loop：关键操作人工确认。
```

#### 8.3 风险控制

Agent 最大风险是错误行动。

必须设计：

```text
只读工具和写工具分级
高风险工具二次确认
参数校验
权限校验
执行日志
幂等设计
回滚方案
最大调用步数
超时限制
```

#### 8.4 高频追问：如何评估 Agent？

可以答：

```text
Agent 评估不能只看最终回答。要看任务成功率、工具调用准确率、参数正确率、平均步数、失败恢复率、越权调用率、用户确认通过率和成本延迟。还要构造多步任务集，覆盖正常任务、工具失败、权限不足和安全敏感任务。
```

### 9. 系统设计中的 trade-off

面试中要主动讲权衡。

```text
RAG top_k 大：召回更全，但 prompt 更长、噪声更多、成本更高。
reranker：质量更好，但增加延迟。
大模型：效果更好，但成本和延迟更高。
小模型：便宜快速，但复杂问题能力弱。
流式输出：体验更好，但服务实现和异常处理更复杂。
缓存：降低成本，但可能返回过期答案。
严格安全：风险低，但可能过度拒答。
多工具 Agent：能力强，但错误行动风险更高。
```

### 10. 系统设计回答模板

可以背下面这个模板：

```text
我先明确需求：用户是谁、数据规模多大、延迟成本要求、是否需要权限和安全合规。

然后定义指标：质量指标、服务指标、成本指标和安全指标。

架构上分离线和在线两条链路。离线侧负责数据接入、清洗、索引或模型训练；在线侧负责请求理解、检索或工具调用、模型生成、安全审核和结果返回。

关键模块我会分别说明选型和 trade-off，例如 embedding、reranker、LLM、推理后端、缓存和限流。

最后设计评估和上线：离线评测、人工评测、A/B 灰度、监控告警、bad case 回归和快速回滚。
```

### 11. 常见扣分点

没有指标，就无法判断系统是否成功。

#### 11.4 不讲权限和安全

系统设计不是离线 demo，要考虑真实上线。

#### 11.6 不讲成本和延迟

大模型系统的成本和延迟经常决定能否落地。

## 第 62 讲：论文讨论模拟

大模型算法岗面试中，论文讨论经常出现在中高级岗位、研究型岗位和大厂算法面中。

面试官可能问：

```text
你最近读过哪篇大模型论文？
Transformer 论文的核心贡献是什么？
LoRA 为什么有效？
DPO 和 RLHF 有什么区别？
FlashAttention 解决了什么问题？
Mamba 为什么被认为是 Transformer 替代路线？
```

论文讨论不是背摘要。

面试官真正想看的是：

```text
你是否理解论文提出的背景问题。
你是否能讲清核心方法。
你是否能解释为什么有效。
你是否知道实验怎么验证。
你是否能指出优缺点和适用边界。
你是否能把论文和工程实践联系起来。
```

这一讲，我们训练论文讨论模拟。

### 本讲目标

学完本讲，你需要掌握：

1. 论文讨论的标准回答框架。
2. 如何 3 分钟讲清一篇论文。
3. 如何回答论文背景、方法、实验、优缺点和工程落地。
4. 如何讨论 Transformer、LoRA、DPO、FlashAttention、RAG、Mamba 等论文。
5. 如何避免只背结论、不讲机制。

### 1. 论文讨论通用框架

推荐使用 7 步框架：

```text
1. 背景：这篇论文要解决什么问题？
2. 核心想法：一句话说清方法。
3. 方法细节：关键公式、结构或算法流程。
4. 为什么有效：机制解释。
5. 实验验证：作者怎么证明有效？
6. 优缺点：适用场景和局限。
7. 工程启发：实际项目中怎么用？
```

面试中可以这样开头：

```text
我会从问题背景、核心方法、为什么有效、实验结果和工程启发几个方面讲。
```

这能让回答有结构。

### 2. 3 分钟论文讲解模板

```text
这篇论文解决的问题是 X。之前方法的主要瓶颈是 Y。

作者提出的方法是 Z，核心思想可以概括为一句话：...

具体来说，它通过 A、B、C 三个设计实现。A 解决了什么，B 解决了什么，C 带来什么权衡。

它有效的原因是 ...

实验上，作者在 ... 任务上比较了 ... baseline，主要指标提升体现在 ...

我认为它的优点是 ...，局限是 ...。在工程中，如果遇到 ... 场景，我会考虑使用它，但需要注意 ...
```

这个模板适用于大多数论文。

### 3. 论文一：Attention Is All You Need

#### 3.1 背景问题

Transformer 之前，序列建模主要依赖 RNN、LSTM、GRU 或 CNN。

主要问题：

```text
RNN 难以并行，长距离依赖建模困难。
CNN 虽然可并行，但长距离依赖需要堆很多层。
```

Transformer 的目标是用 attention 完全替代循环和卷积结构。

#### 3.2 核心方法

一句话：

```text
Transformer 使用 self-attention 建模序列中任意位置之间的依赖，并通过 multi-head attention、position encoding、residual connection 和 feed-forward network 组成 encoder-decoder 架构。
```

核心公式：

```text
Attention(Q, K, V) = softmax(QK^T / sqrt(d_k))V
```

#### 3.3 为什么有效

```text
self-attention 让任意两个 token 之间可以直接交互，路径长度短。
整个序列可以并行计算，比 RNN 更适合 GPU。
multi-head 允许模型在不同子空间学习不同关系。
位置编码补充序列顺序信息。
```

#### 3.4 局限

```text
attention 复杂度随序列长度平方增长。
原始绝对位置编码对超长外推有限。
原始 Transformer 是 encoder-decoder，并不是现代 GPT 的全部形态。
```

#### 3.5 面试表达

```text
Transformer 最大贡献不是某个单点技巧，而是证明纯 attention 结构可以替代 RNN/CNN 成为通用序列建模架构，并且更适合并行训练。现代 GPT、BERT、T5 和大多数 LLM 都是在这个基础上演化出来的。
```

### 4. 论文二：LoRA

#### 4.1 背景问题

全参数微调大模型成本高：

```text
显存占用大
训练参数多
每个任务要保存完整模型副本
容易破坏预训练能力
```

LoRA 解决的是参数高效微调问题。

#### 4.2 核心方法

一句话：

```text
LoRA 冻结原模型权重，只在部分线性层上学习一个低秩增量矩阵，用少量参数近似全参数更新。
```

公式：

```text
W' = W + ΔW
ΔW = BA
```

其中 `A` 和 `B` 是低秩矩阵，rank 远小于原矩阵维度。

#### 4.3 为什么有效

```text
论文假设大模型针对下游任务的参数更新具有低内在秩。
也就是说，不需要在完整高维空间更新全部参数，只需要在低秩子空间中调整就能适配任务。
```

#### 4.4 工程价值

```text
显著减少训练参数和显存。
每个任务只保存小的 adapter。
可以快速切换多个任务适配器。
适合个人项目和企业低成本微调。
```

#### 4.5 局限

```text
rank 太小可能表达能力不足。
rank 太大或 target modules 太多可能过拟合。
不适合所有任务，复杂领域迁移可能仍需更强更新。
target module 选择会影响效果。
```

### 5. 论文三：DPO

#### 5.1 背景问题

RLHF 标准流程复杂：

```text
需要训练 reward model
需要 PPO 强化学习
训练不稳定
工程成本高
```

DPO 希望直接从偏好数据优化 policy。

#### 5.2 核心方法

一句话：

```text
DPO 把带 KL 约束的 reward optimization 推导成一个直接在 chosen/rejected 偏好对上优化的分类式目标，绕过显式 reward model 和 PPO。
```

数据形式：

```text
prompt
chosen response
rejected response
```

训练目标直观理解：

```text
让 policy model 相比 reference model 更偏好 chosen，而不是 rejected。
```

#### 5.3 为什么有效

```text
它利用偏好对直接提供排序信号。
reference model 起到约束作用，避免 policy 偏离 SFT 模型太远。
训练流程接近监督学习，比 PPO 简单稳定。
```

#### 5.4 局限

```text
依赖高质量偏好数据。
可能学习长度偏置、风格偏置或过度拒答。
偏好提升不等于事实能力提升。
beta 等超参数会影响偏离程度。
```

#### 5.5 面试表达

```text
DPO 的工程价值在于把偏好优化从复杂 RL 流程简化成直接优化偏好对，因此非常适合 SFT 后的轻量对齐。但它不是万能的，偏好数据质量和副作用评估非常关键。
```

### 6. 论文四：FlashAttention

#### 6.1 背景问题

标准 attention 在长序列下显存和内存访问开销大。

很多人以为瓶颈只是计算量，但 FlashAttention 强调：

```text
GPU HBM 读写 attention matrix 的 IO 开销非常大。
```

#### 6.2 核心方法

一句话：

```text
FlashAttention 通过 tiling 和 online softmax，在不显式存储完整 attention matrix 的情况下精确计算 attention，从而降低 HBM 访问和显存占用。
```

#### 6.3 为什么有效

```text
它把计算分块放入更快的 SRAM 中处理。
避免把完整 T×T attention matrix 写入和读出 HBM。
使用 online softmax 保证分块计算仍然得到精确 attention 结果。
```

#### 6.4 工程价值

```text
降低显存占用
提升 attention 速度
支持更长上下文训练
是现代 LLM 训练和推理优化的重要基础
```

#### 6.5 局限

```text
主要优化 attention 计算和 IO，不改变 attention 的数学形式。
attention 理论复杂度仍然是 O(T^2)。
对硬件和 kernel 实现依赖较强。
```

### 7. 论文五：RAG

#### 7.1 背景问题

纯参数模型存在几个问题：

```text
知识更新困难
事实可追溯性差
容易幻觉
难以接入企业私有知识
```

RAG 的目标是把检索模块和生成模型结合。

#### 7.2 核心方法

一句话：

```text
RAG 先从外部知识库检索相关文档，再把文档作为上下文输入生成模型，让模型基于检索证据回答。
```

#### 7.3 为什么有效

```text
外部知识库可以更新，不需要重新训练模型。
检索证据可以作为答案依据。
能降低模型完全依赖参数记忆带来的幻觉。
适合企业私有知识场景。
```

#### 7.4 局限

```text
检索不到正确证据时，生成模型很难答对。
检索噪声会污染上下文。
chunk 切分、embedding、rerank 和 prompt 都会影响效果。
RAG 不能天然消除幻觉，仍需 groundedness 评估。
```

### 8. 论文六：Mamba / Selective State Space Models

#### 8.1 背景问题

Transformer 在长序列上有二次复杂度瓶颈。

状态空间模型希望用更接近线性复杂度的方式建模长序列。

#### 8.2 核心方法

一句话：

```text
Mamba 使用 selective state space mechanism，让状态空间模型的参数随输入动态变化，在保持长序列高效建模的同时增强内容选择能力。
```

#### 8.3 为什么重要

```text
传统 SSM 在长序列上高效，但内容选择能力不如 attention 灵活。
Mamba 的 selective 机制让模型能根据输入决定保留或忽略哪些信息。
它在长序列、推理效率和线性复杂度方向上提供了 Transformer 替代路线。
```

#### 8.4 局限

```text
生态和工程成熟度不如 Transformer。
在通用大模型和复杂推理任务上是否全面替代 Transformer 仍需验证。
attention 的显式 token-token 交互仍然有优势。
```

### 9. 论文讨论中的高频追问

#### 问法 1：这篇论文最大的贡献是什么？

回答时不要列很多点，先说主贡献：

```text
我认为最大贡献是它把 X 问题转化为 Y，并提出 Z 方法，使得 ...
```

例如 LoRA：

```text
LoRA 最大贡献是提出用低秩增量近似下游任务的权重更新，让大模型微调从全参数更新变成小规模 adapter 训练，大幅降低训练和存储成本。
```

#### 问法 2：这篇论文有什么局限？

回答要具体：

```text
局限主要在三个方面：第一是 ...，第二是 ...，第三是 ...。所以它更适合 ... 场景，而在 ... 场景要谨慎。
```

#### 问法 3：如果让你复现，你会怎么做？

可以答：

```text
我会先复现最小版本，选一个可控数据集和小模型，跑通 baseline 和论文方法。然后固定随机种子和配置，比较核心指标。接着做关键消融，例如去掉某个模块、调整超参数、改变数据规模。最后分析 bad case，确认提升来自论文机制，而不是实现细节或数据偏差。
```

#### 问法 4：这篇论文怎么落地到业务？

可以答：

```text
我会先判断它解决的问题是否和业务痛点匹配。如果业务痛点是微调成本，LoRA 适合；如果是推理显存和长上下文训练，FlashAttention 或 GQA/MQA 更相关；如果是知识更新和可追溯，RAG 更直接。落地时还要评估质量、延迟、成本、安全和维护复杂度。
```

### 12. 常见错误

#### 12.1 只背摘要

摘要只能说明你看过，不能说明你理解。

#### 12.2 只讲优点，不讲局限

每篇论文都有边界。能讲局限才显得成熟。

#### 12.3 不讲为什么有效

方法是什么不够，要讲机制。

#### 12.4 不讲实验设计

论文贡献需要实验支持。至少要知道 baseline、指标和消融。

#### 12.5 不联系工程实践

算法岗面试不是论文复述，要能讲怎么落地。

### 14. 本讲小结

这一讲做了论文讨论模拟。

核心结论如下：

1. 论文讨论不是背摘要，而是讲清问题、方法、机制、实验、局限和工程启发。
2. 推荐使用“背景、核心想法、方法细节、为什么有效、实验验证、优缺点、工程启发”框架。
3. Transformer 的核心贡献是用 self-attention 替代循环和卷积，提升并行性和长距离依赖建模。
4. LoRA 的核心是低秩增量微调，降低大模型适配成本。
5. DPO 的核心是直接用 chosen/rejected 偏好对优化 policy，简化 RLHF 流程。
6. FlashAttention 的核心是通过 tiling 和 online softmax 降低 attention 的 IO 和显存开销。
7. RAG 的核心是外部知识检索加生成，适合知识更新和可追溯场景，但不能天然消除幻觉。
8. Mamba 代表 SSM 路线，通过 selective mechanism 提升长序列建模效率和内容选择能力。
9. 每篇论文都要准备贡献、局限、复现思路和业务落地方式。
10. 面试中最好把论文和自己的项目连接起来，体现你不只是读过，而是真的能用。
