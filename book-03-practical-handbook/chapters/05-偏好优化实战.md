### 一、什么是 Preference Pair

Preference pair 是偏好对。

一条样本通常包含：

```text
prompt：用户问题或指令。
chosen：更好的回答。
rejected：更差的回答。
```

例如：

```json
{
  "prompt": "解释什么是过拟合。",
  "chosen": "过拟合是指模型在训练集上表现很好，但在未见数据上表现较差的现象。常见缓解方法包括正则化、早停和增加数据。",
  "rejected": "过拟合就是模型很厉害，所以训练得特别好。"
}
```

这条数据表达的是：

```text
在同一个 prompt 下，chosen 应该比 rejected 更受偏好。
```

偏好优化不一定要求给出绝对分数。

它只需要相对比较。

用符号写，一条偏好样本可以表示为：

```math
(x_i,y_i^+,y_i^-)
```

其中 `x_i` 是 prompt，`y_i^+` 是 chosen response，`y_i^-` 是 rejected response。它表达的偏好关系是：

```math
y_i^+ \succ_{x_i} y_i^-
```

读作：在 prompt `x_i` 条件下，`y_i^+` 优于 `y_i^-`。一个偏好数据集就是：

```math
\mathcal{D}_{\mathrm{pref}}
=
\{(x_i,y_i^+,y_i^-)\}_{i=1}^{N}
```

这三个字段必须共享同一个 prompt。否则样本表达的是“不同问题的不同答案哪个更好”，而不是同一问题下的偏好比较。

### 二、Preference Pair 用在哪里

偏好数据可以用于多种训练方式。

#### 1. Reward Model

Reward Model 学习给回答打分。

训练目标是：

```text
score(prompt, chosen) > score(prompt, rejected)
```

更常见的概率建模是 Bradley-Terry 形式：

```math
P(y_i^+ \succ y_i^- \mid x_i)
=
\sigma(r_\phi(x_i,y_i^+)-r_\phi(x_i,y_i^-))
```

其中 `r_phi` 是 Reward Model 给 prompt-response 打出的标量分数，`\sigma` 是 sigmoid。这个公式的直觉很简单：chosen reward 比 rejected reward 高得越多，模型认为 chosen 被偏好的概率越高。

#### 2. RLHF

RLHF 中通常先训练 Reward Model，再用 PPO 等强化学习方法优化策略模型。

#### 3. DPO

DPO 不显式训练 Reward Model。

它直接利用 chosen/rejected 对优化语言模型。

#### 4. ORPO、IPO、KTO 等方法

很多偏好优化方法都围绕偏好数据展开。

所以 preference pair 是偏好优化实战的基础数据结构。

DPO 也使用同一类数据，只是不先显式训练 `r_phi`。它直接比较策略模型在 chosen 和 rejected 上的条件 log probability，因此原始三元组结构仍然是核心输入。

### 三、偏好数据和 SFT 数据的区别

SFT 数据：

```json
{
  "instruction": "解释什么是过拟合。",
  "input": "",
  "output": "过拟合是指..."
}
```

Preference 数据：

```json
{
  "prompt": "解释什么是过拟合。",
  "chosen": "过拟合是指模型在训练集上表现很好，但在未见数据上表现较差...",
  "rejected": "过拟合就是模型特别厉害。"
}
```

区别：

```text
SFT 只有一个理想答案。
Preference 有一个更好答案和一个更差答案。
SFT 学习模仿。
Preference 学习偏好排序或直接优化偏好。
```

SFT 关注：

```text
应该生成什么。
```

偏好优化关注：

```text
哪个回答更好。
```

### 四、标准字段格式

最常用的字段是：

```text
prompt
chosen
rejected
```

建议先统一成这种结构。

```python
preference_data = [
    {
        "prompt": "解释什么是梯度下降。",
        "chosen": "梯度下降是一种通过沿损失函数负梯度方向更新参数来最小化损失的方法。",
        "rejected": "梯度下降就是梯度往下掉。",
    },
    {
        "prompt": "给出三个缓解过拟合的方法。",
        "chosen": "可以使用更多数据、正则化、dropout、早停或减小模型容量。",
        "rejected": "只要把模型做得更大就能缓解过拟合。",
    },
]
```

有些数据集还会包含：

```text
source
chosen_score
rejected_score
labeler_id
category
```

但训练 DPO 或 Reward Model 时，核心字段通常就是 `prompt/chosen/rejected`。

TRL `DPOTrainer` 支持标准语言模型数据和 conversational 数据。教学阶段建议先保存显式字段：

```text
prompt：只放用户问题、系统提示和必要上下文。
chosen：只放被偏好的 assistant 回答。
rejected：只放被拒绝的 assistant 回答。
```

不要把 `prompt` 复制进 `chosen` 和 `rejected`，否则后续拼接模板时容易重复 prompt。也不要一会儿存裸问题、一会儿存已经套过模板的 prompt；格式混用会让训练和评估都难排查。

### 五、如何构造 chosen 和 rejected

常见来源有四类。

#### 来源 1：人工写 chosen，人工写 rejected

适合小规模高质量数据。

优点：质量可控。

缺点：成本高。

#### 来源 2：一个 prompt 采样多个模型回答，人工选择好坏

这是很常见的偏好标注方式。

流程：

```text
同一个 prompt -> 模型生成多个候选 -> 人工选 preferred 和 dispreferred。
```

如果一个 prompt 有 `K` 个候选回答，并且人工给出了完整排序，可以从排序中构造多条 pair。理论上最多有：

```math
N_{\mathrm{pair}}=\frac{K(K-1)}{2}
```

条偏好对。实践中不一定要保留所有 pair；相邻质量等级的 pair 往往更有信息量，明显乱码和优秀答案之间的 pair 太容易，可能让模型只学到粗糙差异。

#### 来源 3：强模型答案作为 chosen，弱模型答案作为 rejected

例如：

```text
chosen 来自更强模型或人工标准答案。
rejected 来自较弱模型。
```

这种方法成本较低，但要小心强模型也可能错。

如果强模型答案事实错误、格式不符合要求，或者拒答边界不合适，就不能机械地当作 chosen。偏好数据不是“强模型蒸馏数据”，而是“在当前任务定义下更符合偏好的回答”。

#### 来源 4：从 SFT 数据构造负样本

SFT 的 output 可以作为 chosen。

再人为构造一个低质量 rejected。

适合教学实验。

### 六、从 SFT 数据构造偏好对

假设 SFT 数据是：

```python
sft_data = [
    {
        "instruction": "解释什么是过拟合。",
        "input": "",
        "output": "过拟合是指模型在训练集上表现很好，但在未见数据上表现较差的现象。",
    }
]
```

可以构造：

```python
def sft_to_preference(example, rejected_answer):
    prompt = example["instruction"]
    if example.get("input", ""):
        prompt = prompt + "\n" + example["input"]

    return {
        "prompt": prompt,
        "chosen": example["output"],
        "rejected": rejected_answer,
    }


preference_example = sft_to_preference(
    sft_data[0],
    rejected_answer="过拟合就是模型训练得很好，所以一定是好事。",
)
```

教学项目中可以手写 rejected。

真实项目中，最好用模型采样和人工标注构造更自然的 rejected。

从 SFT 数据自动构造 rejected 时，要避免两类极端：

```text
rejected 只是乱码，任务太简单。
rejected 看似自然但事实更正确，导致反标。
```

更好的负样本是“看起来像回答，但存在具体缺陷”，例如概念混淆、漏掉关键条件、格式不对、过度自信或未承认证据不足。

### 九、给 prompt 加模板

偏好数据中的 `prompt` 可以是裸问题，也可以是已经格式化后的 prompt。

推荐先保存裸字段：

```text

```

训练前再统一加模板。

Alpaca 风格：

```python
def format_prompt(prompt):
    return (
        "### Instruction:\n"
        f"{prompt}\n\n"
        "### Response:\n"
    )
```

```python
def format_preference(example):
    prompt = format_prompt(example["prompt"])
    return {
        "prompt": prompt,
        "chosen": example["chosen"],
        "rejected": example["rejected"],
    }
```

注意：

```text
chosen 和 rejected 应该共享同一个 prompt。
```

否则比较不公平。

### 十、Chat 格式 preference 数据

如果目标模型是 chat 模型，也可以保存 messages。

一种常见结构：

```python
chat_preference = {
    "prompt": [
        {"role": "system", "content": "你是一个有帮助的 AI 助手。"},
        {"role": "user", "content": "解释什么是过拟合。"},
    ],
    "chosen": [
        {"role": "assistant", "content": "过拟合是指模型在训练集上表现很好，但在未见数据上表现较差的现象。"},
    ],
    "rejected": [
        {"role": "assistant", "content": "过拟合就是模型特别强。"},
    ],
}
```

训练前用 `apply_chat_template` 转成字符串。

教学项目中，先使用字符串格式更简单。

真实 chat 模型训练时，推荐严格使用模型自带 chat template。

对于 conversational preference 数据，结构上仍然要保证：

```text
prompt messages 相同。
chosen 只追加一个或多个更优 assistant messages。
rejected 只追加一个或多个更差 assistant messages。
```

如果 chosen 和 rejected 各自带了不同 system prompt 或不同 user 历史，对比就不再公平。

### 十一、数据质量检查

偏好数据质量非常重要。

检查项：

```text
prompt 是否清楚。
chosen 是否真的优于 rejected。
chosen 是否回答了 prompt。
rejected 是否不是完全无关文本。
chosen/rejected 长度差异是否过大。
是否存在 chosen 和 rejected 完全相同。
是否有明显错标。
是否有重复样本。
是否有安全和隐私风险。
```

最基本检查：

```python
def validate_preference(example):
    if not example["prompt"].strip():
        return False
    if not example["chosen"].strip():
        return False
    if not example["rejected"].strip():
        return False
    if example["chosen"].strip() == example["rejected"].strip():
        return False
    return True


cleaned = [ex for ex in raw_preferences if validate_preference(ex)]
```

更进一步，质量检查至少要覆盖三类问题：

```text
结构问题：空字段、重复、chosen/rejected 完全相同。
偏好问题：反标、chosen 没回答 prompt、rejected 反而更正确。
分布问题：长度偏置、类别不均衡、同一 prompt 泄漏。
```

如果有参考检查点，可以用一个很粗的启发式辅助反标检查：chosen 命中的关键点明显少于 rejected 时，样本要进入人工复核。这个规则不能自动决定标签，但很适合发现高风险样本。

### 十二、长度检查

极端长度差异会影响训练。

例如：

```text
chosen 是 500 字完整回答。
rejected 是 2 个字。
```

模型可能学到“越长越好”，而不是学到真实质量差异。

可以定义一个简单长度比：

```math
\rho_i=
\frac{\max(L_i^+,L_i^-)}
{\max(\min(L_i^+,L_i^-),1)}
```

其中 `L_i^+` 是 chosen token 数，`L_i^-` 是 rejected token 数。`\rho_i` 很大时，不一定要删除样本，但应该进入人工抽查或分桶统计。

可以统计长度：

```python
def length_stats(data):
    for ex in data:
        print(
            len(ex["prompt"]),
            len(ex["chosen"]),
            len(ex["rejected"]),
        )


length_stats(cleaned)
```

更好的做法是用 tokenizer 后的 token 长度：

```python
def token_length(tokenizer, text):
    return len(tokenizer(text, add_special_tokens=False)["input_ids"])
```

### 十三、去重

```python
def deduplicate_preferences(data):
    seen = set()
    result = []
    for ex in data:
        key = (
            ex["prompt"].strip(),
            ex["chosen"].strip(),
            ex["rejected"].strip(),
        )
        if key not in seen:
            seen.add(key)
            result.append(ex)
    return result


cleaned = deduplicate_preferences(cleaned)
```

重复偏好对会让模型过度拟合某些样本。

如果重复是有意加权，也应该明确记录，而不是无意识重复。

### 十四、train/eval 划分

```python
dataset = Dataset.from_list(cleaned)
split = dataset.train_test_split(test_size=0.1, seed=42)
train_dataset = split["train"]
eval_dataset = split["test"]
```

注意：

```text
同一个 prompt 的多个偏好对，最好不要一部分在 train，一部分在 eval。
```

否则验证集可能泄漏。

更严谨做法是按 prompt 分组划分。

简化版：

```python
from collections import defaultdict
import random


groups = defaultdict(list)
for ex in cleaned:
    groups[ex["prompt"]].append(ex)

prompts = list(groups.keys())
random.Random(42).shuffle(prompts)

cut = int(0.9 * len(prompts))
train_prompts = set(prompts[:cut])

train_data = []
eval_data = []
for prompt, items in groups.items():
    if prompt in train_prompts:
        train_data.extend(items)
    else:
        eval_data.extend(items)
```

这个约束可以写成：

```math
\mathcal{P}_{\mathrm{train}}\cap\mathcal{P}_{\mathrm{eval}}=\varnothing
```

其中 `\mathcal{P}_{\mathrm{train}}` 和 `\mathcal{P}_{\mathrm{eval}}` 分别是训练集和验证集里的 prompt 集合。偏好数据里一个 prompt 往往会对应多个候选回答和多个 pair，所以按样本随机切分很容易泄漏。

### 十五、Reward Model 和 DPO 的格式差异

Reward Model 训练通常需要把 prompt+chosen 和 prompt+rejected 分别编码。

例如：

```text
chosen_text = prompt_template + chosen
rejected_text = prompt_template + rejected
```

DPO 训练库通常直接需要：

```text
prompt
chosen
rejected
```

TRL 的 DPOTrainer 常见输入字段也是这三个。

所以保存原始结构时，建议保留：

```text
prompt/chosen/rejected
```

不要过早只保存拼接后的文本。

这样后续 Reward Model 和 DPO 都能复用。

### 十六、完整构造脚本

```python
import json
from pathlib import Path
from collections import defaultdict
import random

from datasets import Dataset


raw_preferences = [
    {
        "prompt": "解释什么是过拟合。",
        "chosen": "过拟合是指模型在训练集上表现很好，但在未见数据上表现较差的现象。",
        "rejected": "过拟合就是模型很强，所以训练得特别好。",
    },
    {
        "prompt": "给出三个缓解过拟合的方法。",
        "chosen": "可以使用更多数据、正则化、dropout、早停或减小模型容量。",
        "rejected": "把训练轮数无限增加就可以缓解过拟合。",
    },
]


def validate_preference(example):
    if not example["prompt"].strip():
        return False
    if not example["chosen"].strip():
        return False
    if not example["rejected"].strip():
        return False
    if example["chosen"].strip() == example["rejected"].strip():
        return False
    return True


def deduplicate_preferences(data):
    seen = set()
    result = []
    for ex in data:
        key = (
            ex["prompt"].strip(),
            ex["chosen"].strip(),
            ex["rejected"].strip(),
        )
        if key not in seen:
            seen.add(key)
            result.append(ex)
    return result


def split_by_prompt(data, train_ratio=0.9, seed=42):
    groups = defaultdict(list)
    for ex in data:
        groups[ex["prompt"].strip()].append(ex)

    prompts = list(groups)
    random.Random(seed).shuffle(prompts)
    cut = int(len(prompts) * train_ratio)
    train_prompts = set(prompts[:cut])

    train_data, eval_data = [], []
    for prompt, items in groups.items():
        if prompt in train_prompts:
            train_data.extend(items)
        else:
            eval_data.extend(items)
    return train_data, eval_data


cleaned = [ex for ex in raw_preferences if validate_preference(ex)]
cleaned = deduplicate_preferences(cleaned)
train_data, eval_data = split_by_prompt(cleaned)

out_path = Path("data/preferences.jsonl")
out_path.parent.mkdir(parents=True, exist_ok=True)

with out_path.open("w", encoding="utf-8") as f:
    for ex in cleaned:
        f.write(json.dumps(ex, ensure_ascii=False) + "\n")

split = {
    "train": Dataset.from_list(train_data),
    "test": Dataset.from_list(eval_data),
}

print(split)
if len(split["train"]) > 0:
    print(split["train"][0])
```

这个脚本得到的数据可以用于后面的 Reward Model 和 DPO。

```python
def normalize(text):
    return " ".join(text.strip().split())


def keyword_hits(text, points):
    text = text.lower()
    return sum(1 for point in points if point.lower() in text)


def audit_and_clean(data, max_length_ratio=4.0):
    seen = set()
    cleaned = []
    stats = defaultdict(int)
    warnings = []

    for ex in data:
        prompt = normalize(ex.get("prompt", ""))
        chosen = normalize(ex.get("chosen", ""))
        rejected = normalize(ex.get("rejected", ""))
        points = ex.get("expected_points", [])

        if not prompt or not chosen or not rejected:
            stats["drop_empty"] += 1
            continue
        if chosen == rejected:
            stats["drop_same_answer"] += 1
            continue

        key = (prompt, chosen, rejected)
        if key in seen:
            stats["drop_duplicate"] += 1
            continue
        seen.add(key)

        chosen_hits = keyword_hits(chosen, points)
        rejected_hits = keyword_hits(rejected, points)
        if rejected_hits > chosen_hits:
            warnings.append((ex["id"], "possible_label_flip"))
            stats["possible_label_flip"] += 1
            continue

        short = max(min(len(chosen), len(rejected)), 1)
        length_ratio = max(len(chosen), len(rejected)) / short
        if length_ratio > max_length_ratio:
            warnings.append((ex["id"], f"length_ratio={length_ratio:.1f}"))
            stats["length_bias_warning"] += 1

        cleaned.append({
            "id": ex["id"],
            "prompt": prompt,
            "chosen": chosen,
            "rejected": rejected,
            "length_ratio": round(length_ratio, 2),
        })
        stats["kept"] += 1

    return cleaned, stats, warnings


def split_by_prompt(data, train_ratio=0.5):
    groups = defaultdict(list)
    for ex in data:
        groups[ex["prompt"]].append(ex)

    prompts = sorted(groups)
    cut = int(len(prompts) * train_ratio)
    train_prompts = set(prompts[:cut])

    train, eval_ = [], []
    for prompt, items in groups.items():
        if prompt in train_prompts:
            train.extend(items)
        else:
            eval_.extend(items)
    return train, eval_


cleaned, stats, warnings = audit_and_clean(RAW_PREFERENCES)
train_data, eval_data = split_by_prompt(cleaned)
train_prompts = {ex["prompt"] for ex in train_data}
eval_prompts = {ex["prompt"] for ex in eval_data}

print(f"raw_count={len(RAW_PREFERENCES)}")
print(f"kept_count={len(cleaned)}")
print(f"drop_empty={stats['drop_empty']}")
print(f"drop_same_answer={stats['drop_same_answer']}")
print(f"drop_duplicate={stats['drop_duplicate']}")
print(f"possible_label_flip={stats['possible_label_flip']}")
print(f"length_bias_warning={stats['length_bias_warning']}")
print(f"warnings={warnings}")
print(f"train_size={len(train_data)}")
print(f"eval_size={len(eval_data)}")
print(f"prompt_leak={bool(train_prompts & eval_prompts)}")
```

### 十八、常见工程坑

#### 坑 1：chosen 和 rejected 反了

这是最严重的错误。

模型会被训练成偏好差回答。

#### 坑 2：chosen/rejected 差距只体现在长度

模型可能学到“长回答更好”，而不是“正确回答更好”。

#### 坑 3：rejected 太差

如果 rejected 是明显乱码，任务太简单，模型学不到细粒度偏好。

好的 rejected 应该是：

```text
看起来像回答，但质量确实更差。
```

#### 坑 4：prompt 模板不一致

Reward Model、DPO 和推理时的模板应保持一致。

#### 坑 5：训练评估泄漏

同一个 prompt 的相似 pair 同时出现在 train 和 eval，会让评估不可靠。

#### 坑 6：不做人工抽查

偏好数据非常容易错标。

必须抽查。

#### 坑 7：把 prompt 拼进 chosen/rejected

如果 `chosen` 和 `rejected` 已经包含 prompt，训练脚本又额外拼一次 prompt，就会出现重复模板。保存原始数据时最好让 `prompt`、`chosen`、`rejected` 各司其职。


```text
我会检查 chosen 是否确实优于 rejected，二者是否回答同一个 prompt，是否存在反标、重复、空文本、长度偏差过大、乱码和安全风险；还会按 prompt 分组划分 train/eval，避免评估泄漏。
```

```text

```

### 本讲目标

上一讲我们构造了 preference pair 数据。

每条数据包含：

```text

chosen
rejected
```

Reward Model 的目标是学一个打分函数：

```text
reward(prompt, response) -> scalar
```

并且希望：

```text
reward(prompt, chosen) > reward(prompt, rejected)
```

本讲先不训练完整 Reward Model，而是专门实现排序损失。

这是一切 Reward Model 训练的核心。

### 一、Reward Model 解决什么问题

SFT 模型学会模仿答案。

但它不一定知道哪个答案更好。

Reward Model 要学习人类偏好。

输入：

```text
prompt + response
```

输出：

```text
一个标量 reward
```

可以写成：

```math
r_\phi(x,y)\in\mathbb{R}
```

其中 `x` 是 prompt，`y` 是 response，`\phi` 是 Reward Model 参数。Reward Model 不直接生成文本，只判断一段 prompt-response 的质量。

标量越大，表示模型认为这个回答越好。

例如：

```text
reward(prompt, chosen) = 2.3
reward(prompt, rejected) = -0.7
```

表示 chosen 比 rejected 更好。

Reward Model 可以用于：

```text
RLHF 中给策略模型提供奖励。
离线评估不同回答质量。
训练 reranker。
为偏好优化方法提供比较基线。
```

---

### 二、为什么用排序损失

偏好数据通常不给绝对分数。

它只告诉你：

```text
chosen 比 rejected 好。
```

所以训练目标不是让 chosen reward 等于某个固定值。

而是让：

```text
r_chosen - r_rejected 尽量大。
```

这就是 pairwise ranking。

我们不关心 reward 的绝对尺度。

更关心相对顺序。

对一条 preference pair `(x_i,y_i^+,y_i^-)`，我们希望：

```math
r_\phi(x_i,y_i^+) > r_\phi(x_i,y_i^-)
```

排序损失只依赖差值：

```math
\Delta_i=r_\phi(x_i,y_i^+)-r_\phi(x_i,y_i^-)
```

如果给两个 reward 同时加上同一个常数，`\Delta_i` 不变，排序损失也不变。这就是为什么 Reward Model 的绝对尺度容易漂移，训练时要监控 reward 均值、方差和 chosen/rejected gap。

---

### 三、Bradley-Terry 模型直觉

Bradley-Terry 模型常用于成对比较。

设 `r_c = r_\phi(x,y^+)`，`r_r = r_\phi(x,y^-)`。

模型认为 chosen 被偏好的概率是：

```math
P(y^+ \succ y^- \mid x)
=
\sigma(r_c-r_r)
=
\frac{1}{1+\exp(-(r_c-r_r))}
```

如果 `r_c` 远大于 `r_r`：

```text
sigmoid(r_c - r_r) 接近 1
```

如果二者相等：

```text
sigmoid(0) = 0.5
```

如果 `r_c` 小于 `r_r`：

```text
概率小于 0.5
```

训练目标是最大化 chosen 被偏好的概率。

也就是最小化负对数似然：

```math
\ell=-\log \sigma(r_c-r_r)
```

---

### 四、排序损失公式

Reward Model 排序损失：

```math
L_{\mathrm{rm}}
=
-\frac{1}{B}
\sum_{i=1}^{B}
\log \sigma(r_i^+ - r_i^-)
```

其中 `B` 是 batch size，`r_i^+` 是 chosen reward，`r_i^-` 是 rejected reward，`\sigma` 是 sigmoid。

如果 `r_chosen - r_rejected` 很大，loss 很小。

如果差值为 0，loss 是：

```text
-log(0.5) ≈ 0.693
```

如果 chosen reward 反而小于 rejected，loss 会变大。

这正好符合我们的目标。

这个损失也可以看成对 `diff = r_chosen - r_rejected` 做二分类，目标标签永远是 1：

```math
L_{\mathrm{rm}}
=
\mathrm{BCEWithLogits}(d,\mathbf{1})
```

其中 `d` 是 reward 差值向量。PyTorch 官方文档强调 `BCEWithLogitsLoss` 把 sigmoid 和 BCE 合在一起，比先 sigmoid 再 BCE 更数值稳定；`-F.logsigmoid(diff)` 是这个二分类目标在正类标签为 1 时的直接写法。

```text

r_rejected = 0
diff = 3
sigmoid(3) ≈ 0.95
loss ≈ 0.05
```

loss 很小。

#### 情况 2：模型无法区分


loss 中等。

#### 情况 3：模型排反了

```text
loss ≈ 3.05
```

loss 很大。

这会推动模型提高 chosen reward，降低 rejected reward。

### 六、PyTorch 实现

最直接写法：

```python
import torch
import torch.nn.functional as F


def reward_ranking_loss(chosen_rewards, rejected_rewards):
    chosen_rewards = chosen_rewards.view(-1)
    rejected_rewards = rejected_rewards.view(-1)
    diff = chosen_rewards - rejected_rewards
    loss = -torch.log(torch.sigmoid(diff))
    return loss.mean()
```

输入 shape：

```text
chosen_rewards:   [batch]
rejected_rewards: [batch]
```

输出：

```text
标量 loss
```

测试：

```python
chosen = torch.tensor([3.0, 1.0, 0.0])
rejected = torch.tensor([0.0, 1.0, 3.0])

loss = reward_ranking_loss(chosen, rejected)
print(loss)
```

### 七、更稳定的实现

直接 `log(sigmoid(x))` 在数值上不如 `logsigmoid(x)` 稳定。

推荐写法：

```python
def reward_ranking_loss(chosen_rewards, rejected_rewards):
    chosen_rewards = chosen_rewards.view(-1)
    rejected_rewards = rejected_rewards.view(-1)
    diff = chosen_rewards - rejected_rewards
    loss = -F.logsigmoid(diff)
    return loss.mean()
```

`F.logsigmoid(diff)` 等价于：

```text
log(sigmoid(diff))
```

但数值更稳定。

这是工程里更常用的写法。

也可以用 `binary_cross_entropy_with_logits` 写成等价形式：

```python
def reward_ranking_loss_bce(chosen_rewards, rejected_rewards):
    chosen_rewards = chosen_rewards.view(-1)
    rejected_rewards = rejected_rewards.view(-1)
    diff = chosen_rewards - rejected_rewards
    target = torch.ones_like(diff)
    return F.binary_cross_entropy_with_logits(diff, target)
```

这两种写法的数学目标一致。教学时 `-F.logsigmoid(diff)` 更直观，工程上看到 BCEWithLogits 写法也应该能认出来。

### 八、验证 loss 方向

写一个简单测试：

```python
good_chosen = torch.tensor([3.0])
bad_rejected = torch.tensor([0.0])

bad_chosen = torch.tensor([0.0])
good_rejected = torch.tensor([3.0])

loss_correct = reward_ranking_loss(good_chosen, bad_rejected)
loss_wrong = reward_ranking_loss(bad_chosen, good_rejected)

print("loss_correct:", loss_correct.item())
print("loss_wrong:", loss_wrong.item())
```

你应该看到：

```text
loss_correct < loss_wrong
```

这个测试很重要。

如果你把符号写反了，模型会学习偏好 rejected。

再看梯度方向。对单条样本：

```math
\ell(d)=-\log\sigma(d),\qquad d=r^+-r^-
```

有：

```math
\frac{\partial \ell}{\partial r^+}
=
\sigma(d)-1
```

```math
\frac{\partial \ell}{\partial r^-}
=
1-\sigma(d)
```

当 `d` 还不够大时，`\partial \ell / \partial r^+` 为负，梯度下降会提高 chosen reward；`\partial \ell / \partial r^-` 为正，梯度下降会降低 rejected reward。

### 九、带 margin 的排序损失

有时希望 chosen reward 至少比 rejected 高一个 margin。

可以写成：

```math
L_{\mathrm{margin}}
=
-\frac{1}{B}
\sum_{i=1}^{B}
\log\sigma(r_i^+ - r_i^- - m_i)
```

代码：

```python
def reward_ranking_loss_with_margin(chosen_rewards, rejected_rewards, margin=0.0):
    diff = chosen_rewards - rejected_rewards - margin
    loss = -F.logsigmoid(diff)
    return loss.mean()
```

如果 `margin=1.0`，模型需要让：

```text
r_chosen 比 r_rejected 至少高 1
```

loss 才会比较小。

margin 可以增强区分度，但也可能让训练更难。

教学项目中先用 `margin=0` 即可。

有些 RewardTrainer 或偏好数据实现会支持每条样本不同的 `margin` 字段，常见含义是“chosen 至少应该比 rejected 高多少”。如果你的数据没有可靠偏好强度，先不要随意加 margin。

### 十、Reward Model 输出 shape

Reward Model 通常输入一段 token 序列，输出一个标量。

例如：

```text
chosen_input_ids:   [B, T]
rejected_input_ids: [B, T]
```

模型输出：

```text
chosen_rewards:   [B]
rejected_rewards: [B]
```

然后计算：

```python
loss = reward_ranking_loss(chosen_rewards, rejected_rewards)
```

如果输出 shape 是 `[B, 1]`，可以压缩：

```python
chosen_rewards = chosen_rewards.squeeze(-1)
rejected_rewards = rejected_rewards.squeeze(-1)
```

不要让 shape 变成奇怪的 `[B, B]` broadcast。

推荐在 loss 函数入口显式统一形状：

```python
chosen_rewards = chosen_rewards.view(-1)
rejected_rewards = rejected_rewards.view(-1)
assert chosen_rewards.shape == rejected_rewards.shape
```

`squeeze()` 不带维度有时会把 batch size 为 1 的 `[1, 1]` 压成标量，后续日志或 gather 逻辑更难处理；更推荐 `squeeze(-1)` 或 `view(-1)`。

### 十一、一个最小 Reward Head

实际 Reward Model 通常是在 transformer backbone 后接一个 value head。

简化结构：

```python
import torch.nn as nn


class RewardHead(nn.Module):
    def __init__(self, hidden_size):
        super().__init__()
        self.value = nn.Linear(hidden_size, 1)

    def forward(self, hidden_states, attention_mask):
        last_token_indices = attention_mask.sum(dim=1) - 1
        batch_indices = torch.arange(hidden_states.size(0), device=hidden_states.device)
        last_hidden = hidden_states[batch_indices, last_token_indices]
        reward = self.value(last_hidden).squeeze(-1)
        return reward
```

这里的思想是：

```text
取最后一个有效 token 的 hidden state，映射成标量 reward。
```

为什么取最后一个 token？

因为 causal LM 的最后一个 token 表示包含前面 prompt 和 response 的上下文信息。

---

### 十二、padding 下为什么要用 attention_mask

如果 batch 中有 padding：

```text
sample A: token token token token
sample B: token token pad   pad
```

不能简单取最后一列 hidden state。

因为 sample B 的最后一列是 padding。

所以需要：

```python
last_token_indices = attention_mask.sum(dim=1) - 1
```

它得到每个样本最后一个有效 token 的位置。

这是 Reward Model 中非常常见的细节。

### 十三、完整前向逻辑示意

```python
def compute_reward_loss(
    reward_model,
    chosen_input_ids,
    chosen_attention_mask,
    rejected_input_ids,
    rejected_attention_mask,
):
    chosen_rewards = reward_model(
        input_ids=chosen_input_ids,
        attention_mask=chosen_attention_mask,
    )
    rejected_rewards = reward_model(
        input_ids=rejected_input_ids,
        attention_mask=rejected_attention_mask,
    )

    loss = reward_ranking_loss(chosen_rewards, rejected_rewards)
    return loss, chosen_rewards, rejected_rewards
```

训练时，每个 batch 会同时包含 chosen 和 rejected 两套输入。

下一讲我们会把它扩展成完整 Reward Model 训练。

### 十六、常见工程坑

#### 坑 1：损失符号写反

错误写法：

```python
-F.logsigmoid(rejected_rewards - chosen_rewards)
```

这会鼓励 rejected reward 更高。

#### 坑 2：chosen/rejected 数据反了

即使 loss 写对，数据反标也会毁掉训练。

#### 坑 3：shape broadcast 错误

如果 chosen 是 `[B, 1]`，rejected 是 `[B]`，相减可能 broadcast 成 `[B, B]`。

建议统一 squeeze 成 `[B]`。

#### 坑 4：取了 padding token 的 hidden state

Reward Head 应使用 attention_mask 找到最后一个有效 token。

#### 坑 5：只看 loss 不看 accuracy

Reward Model 训练时应同时看 pairwise accuracy。

#### 坑 6：reward 绝对值越来越大

排序损失主要关心差值，reward 绝对尺度可能漂移。

必要时可以做正则化或监控 reward 分布。

#### 坑 7：把 margin 当成越大越好

margin 会提高区分要求，但过大的 margin 可能让 loss 长期偏高、训练更不稳定。没有可靠偏好强度时，先用 `margin=0`。

#### 坑 8：batch size 为 1 时 squeeze 过度

`tensor.squeeze()` 可能把 `[1,1]` 压成标量。Reward loss 里更推荐 `squeeze(-1)` 或 `view(-1)`，保持 batch 维语义清楚。

### 十七、面试怎么讲 Reward Model Loss

如果面试官问“Reward Model 怎么训练”，可以这样回答：

```text
Reward Model 输入 prompt 和 response，输出一个标量 reward。训练数据是 prompt、chosen、rejected 三元组。我们分别计算 chosen 和 rejected 的 reward，然后用 Bradley-Terry 风格的 pairwise ranking loss：-log sigmoid(r_chosen - r_rejected)，鼓励 chosen reward 高于 rejected reward。工程上通常用 -F.logsigmoid(diff) 或 BCEWithLogitsLoss 来避免数值不稳定。
```

如果追问“为什么不是回归一个分数”，可以回答：

```text
偏好数据通常只提供相对偏好，而不是绝对分数。人类更容易判断两个回答哪个更好，而不是给稳定的绝对分。因此 Reward Model 常用成对排序损失来学习相对偏好。
```

如果问“怎么判断 Reward Model 训练是否正常”，可以回答：

```text
除了看 ranking loss 是否下降，还要看 chosen reward 是否平均高于 rejected reward，以及 pairwise accuracy 是否上升。如果 accuracy 长期低于 0.5，要检查 chosen/rejected 是否反了、loss 符号是否写反、shape 是否 broadcast 错误。
```

这一讲实现了 Reward Model 排序损失。

核心结论如下：

1. Reward Model 输入 prompt+response，输出标量 reward。
2. 偏好数据只提供相对关系：chosen 优于 rejected。
3. Bradley-Terry 模型使用 `sigmoid(r_chosen - r_rejected)` 表示 chosen 被偏好的概率。
4. 常用排序损失是 `-logsigmoid(r_chosen - r_rejected)`。

6. Reward Head 常取最后一个有效 token 的 hidden state 输出 reward。
7. loss 函数入口要统一 reward shape，避免 `[B,1]` 和 `[B]` broadcast 成 `[B,B]`。
8. 训练时要同时监控 ranking loss、pairwise accuracy、reward gap 和 reward 分布。

### 一、Reward Model 训练流程

完整流程如下：

```text
1. 准备 prompt/chosen/rejected 数据。
2. 构造 chosen_text = prompt_template + chosen。
3. 构造 rejected_text = prompt_template + rejected。
4. 分别 tokenize chosen_text 和 rejected_text。
5. Reward Model 输出 chosen_reward 和 rejected_reward。
6. 用 -logsigmoid(chosen_reward - rejected_reward) 训练。
7. 监控 loss 和 reward accuracy。
```

本讲资料边界：

1. 真实工程中可以直接使用 Hugging Face TRL 的 `RewardTrainer`，它围绕 chosen/rejected 偏好数据实现 Reward Model 训练流程。
2. 本讲手写训练循环，是为了看清数据编码、collator、reward head、pairwise loss、指标统计和 checkpoint 的内部连接关系。
3. `transformers` 和 `datasets` 代码是工程接口示例；后面的最小 demo 只依赖 PyTorch，用来保证没有外部下载环境时也能验证核心机制。
4. 只从可信来源加载 checkpoint。`torch.load` 虽然方便，但真实项目不要加载来源不明的权重文件。

和 SFT 不同，Reward Model 不是预测下一个 token。

它输出一个标量分数。

用公式写，偏好训练集是：

```math
D_{\mathrm{pref}}=\{(x_i,y_i^+,y_i^-)\}_{i=1}^{N}
```

其中 `x_i` 是 prompt，`y_i^+` 是 chosen response，`y_i^-` 是 rejected response。

tokenize 之后，每个样本会变成两套输入：

```math
X_i^+,A_i^+ = \mathrm{tok}(x_i,y_i^+)
```

```math
X_i^-,A_i^- = \mathrm{tok}(x_i,y_i^-)
```

其中 `X` 表示 token ids，`A` 表示 attention mask。

Reward Model 是一个标量函数：

```math
r_i^+ = r_\phi(X_i^+,A_i^+),\qquad r_i^- = r_\phi(X_i^-,A_i^-)
```

batch 内的排序损失是：

```math
L_{\mathrm{RM}}
=-\frac{1}{B}\sum_{i=1}^{B}\log\sigma(r_i^+-r_i^-)
```

pairwise accuracy 是：

```math
A_{\mathrm{RM}}
=\frac{1}{B}\sum_{i=1}^{B}\mathbf{1}[r_i^+>r_i^-]
```

reward gap 是：

```math
G_{\mathrm{RM}}
=\frac{1}{B}\sum_{i=1}^{B}(r_i^+-r_i^-)
```

如果使用最后一个有效 token 的 hidden state，0-based 下标通常写成：

```math
t_i=\left(\sum_{j=0}^{T-1} A_{i,j}\right)-1
```

也就是说，padding 后不能直接取 `hidden_states[:, -1]`，要用 `attention_mask` 找到每条样本自己的最后有效位置。

### 五、Reward Data Collator

Reward Model batch 中有两套输入：chosen 和 rejected。

```python
import torch


class RewardDataCollator:
    def __init__(self, tokenizer):
        self.tokenizer = tokenizer

    def _pad(self, features, key, pad_value):
        tensors = [torch.tensor(f[key], dtype=torch.long) for f in features]
        return torch.nn.utils.rnn.pad_sequence(
            tensors,
            batch_first=True,
            padding_value=pad_value,
        )

    def __call__(self, features):
        return {
            "chosen_input_ids": self._pad(features, "chosen_input_ids", self.tokenizer.pad_token_id),
            "chosen_attention_mask": self._pad(features, "chosen_attention_mask", 0),
            "rejected_input_ids": self._pad(features, "rejected_input_ids", self.tokenizer.pad_token_id),
            "rejected_attention_mask": self._pad(features, "rejected_attention_mask", 0),
        }
```

### 六、构造 Reward Model

Reward Model 可以用 causal LM backbone 加一个 value head。

```python
import torch.nn as nn
from transformers import AutoModel


class RewardModel(nn.Module):
    def __init__(self, model_name):
        super().__init__()
        self.backbone = AutoModel.from_pretrained(model_name)
        hidden_size = self.backbone.config.hidden_size
        self.value_head = nn.Linear(hidden_size, 1)

    def forward(self, input_ids, attention_mask):
        outputs = self.backbone(
            input_ids=input_ids,
            attention_mask=attention_mask,
        )
        hidden_states = outputs.last_hidden_state

        lengths = attention_mask.sum(dim=1).clamp_min(1)
        last_token_indices = lengths - 1
        batch_indices = torch.arange(hidden_states.size(0), device=hidden_states.device)
        last_hidden = hidden_states[batch_indices, last_token_indices]

        rewards = self.value_head(last_hidden).view(-1)
        return rewards
```

这里用 `AutoModel`，不是 `AutoModelForCausalLM`。

因为 Reward Model 不需要 lm head。

它只需要 hidden states，然后接一个标量 value head。

对 decoder-only backbone 来说，最后有效 token 的 hidden state 已经通过 causal attention 看过前面的 prompt 和 response token，因此常被用来代表整段 `prompt + response`。如果换成 encoder 或双塔结构，pooling 方式可能不同，要根据 backbone 的语义重新设计。

### 七、实现 loss 和 accuracy

```python
import torch.nn.functional as F


def reward_ranking_loss(chosen_rewards, rejected_rewards):
    chosen_rewards = chosen_rewards.view(-1)
    rejected_rewards = rejected_rewards.view(-1)
    return -F.logsigmoid(chosen_rewards - rejected_rewards).mean()


def reward_accuracy(chosen_rewards, rejected_rewards):
    chosen_rewards = chosen_rewards.view(-1)
    rejected_rewards = rejected_rewards.view(-1)
    return (chosen_rewards > rejected_rewards).float().mean()
```

### 九、评估函数

```python
@torch.no_grad()
def evaluate(model, dataloader, device):
    was_training = model.training
    model.eval()
    total_loss = 0.0
    total_acc = 0.0
    total_gap = 0.0
    total_count = 0

    for batch in dataloader:
        batch = {k: v.to(device) for k, v in batch.items()}

        chosen_rewards = model(
            batch["chosen_input_ids"],
            batch["chosen_attention_mask"],
        )
        rejected_rewards = model(
            batch["rejected_input_ids"],
            batch["rejected_attention_mask"],
        )

        loss = reward_ranking_loss(chosen_rewards, rejected_rewards)
        acc = reward_accuracy(chosen_rewards, rejected_rewards)
        gap = (chosen_rewards - rejected_rewards).mean()

        batch_size = chosen_rewards.size(0)
        total_loss += loss.item() * batch_size
        total_acc += acc.item() * batch_size
        total_gap += gap.item() * batch_size
        total_count += batch_size

    if was_training:
        model.train()
    return {
        "loss": total_loss / total_count,
        "accuracy": total_acc / total_count,
        "gap": total_gap / total_count,
    }
```

### 十三、Reward 漂移问题

排序损失只关心差值：

```text
r_chosen - r_rejected
```

它不直接约束 reward 的绝对值。

所以 reward 可能整体越来越大或越来越小。

这叫 reward scale drift 或 reward 漂移。

教学项目中先监控即可。

真实系统中可能会加入：

```text
reward normalization
regularization
reward clipping
更严格的数据平衡
```

不要把 Reward Model 输出当作绝对真理。

它只是一个学出来的偏好代理。

### 十四、最小 PyTorch Reward Model 训练 demo

上面的 Hugging Face 版本更接近真实工程，但需要 `transformers` 和 `datasets`。

下面给一个只依赖 PyTorch 的最小 demo。它不用真实 tokenizer，而是用空格分词；它也不用完整 Transformer，而是用 embedding 加一个简单 summary，再接 value head。

注意：真实 decoder-only Reward Model 的最后 token hidden state 能通过 attention 汇总前文信息；这个 toy model 没有 attention，所以用累计平均 summary 来模拟“最后有效位置代表整段输入”的效果。

```python
import tempfile

import torch
import torch.nn as nn
import torch.nn.functional as F
from torch.optim import AdamW
from torch.utils.data import DataLoader


torch.manual_seed(7)

raw_pairs = [
    {
        "prompt": "explain overfitting",
        "chosen": "clear answer train high validation low generalization bad",
        "rejected": "wrong answer train always better",
    },
    {
        "prompt": "reduce overfitting",
        "chosen": "correct answer more data regularization dropout early stopping",
        "rejected": "wrong answer train forever",
    },
    {
        "prompt": "explain gradient descent",
        "chosen": "clear answer update parameters negative gradient minimize loss",
        "rejected": "wrong answer parameters just become small",
    },
    {
        "prompt": "what is evaluation",
        "chosen": "correct answer heldout set compare metrics error cases",
        "rejected": "wrong answer only watch train loss",
    },
    {
        "prompt": "what is reward model",
        "chosen": "clear answer score response from preference pairs",
        "rejected": "wrong answer random score magic",
    },
    {
        "prompt": "avoid label leak",
        "chosen": "correct answer split by prompt keep eval unseen",
        "rejected": "wrong answer duplicate prompt everywhere",
    },
]


def format_text(prompt, response):
    return f"instruction {prompt} response {response} <eos>"


vocab = {"<pad>": 0, "<unk>": 1}
for item in raw_pairs:
    for response_key in ["chosen", "rejected"]:
        for token in format_text(item["prompt"], item[response_key]).split():
            if token not in vocab:
                vocab[token] = len(vocab)


def encode(text):
    return [vocab.get(token, vocab["<unk>"]) for token in text.split()]


encoded_pairs = []
for item in raw_pairs:
    encoded_pairs.append(
        {
            "chosen_input_ids": encode(format_text(item["prompt"], item["chosen"])),
            "rejected_input_ids": encode(format_text(item["prompt"], item["rejected"])),
        }
    )


def collate(features):
    max_len = max(
        max(len(f["chosen_input_ids"]), len(f["rejected_input_ids"]))
        for f in features
    )

    def pad(ids):
        pad_len = max_len - len(ids)
        return ids + [vocab["<pad>"]] * pad_len, [1] * len(ids) + [0] * pad_len

    batch = {
        "chosen_input_ids": [],
        "chosen_attention_mask": [],
        "rejected_input_ids": [],
        "rejected_attention_mask": [],
    }
    for feature in features:
        chosen_ids, chosen_mask = pad(feature["chosen_input_ids"])
        rejected_ids, rejected_mask = pad(feature["rejected_input_ids"])
        batch["chosen_input_ids"].append(chosen_ids)
        batch["chosen_attention_mask"].append(chosen_mask)
        batch["rejected_input_ids"].append(rejected_ids)
        batch["rejected_attention_mask"].append(rejected_mask)
    return {k: torch.tensor(v, dtype=torch.long) for k, v in batch.items()}


class TinyRewardModel(nn.Module):
    def __init__(self, vocab_size, hidden_size=16):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, hidden_size, padding_idx=vocab["<pad>"])
        self.value_head = nn.Linear(hidden_size, 1)

    def forward(self, input_ids, attention_mask):
        token_states = self.embedding(input_ids) * attention_mask.unsqueeze(-1)
        cumulative_states = token_states.cumsum(dim=1)
        lengths = attention_mask.sum(dim=1).clamp_min(1)
        last_indices = lengths - 1
        batch_indices = torch.arange(input_ids.size(0), device=input_ids.device)
        summary = cumulative_states[batch_indices, last_indices] / lengths.unsqueeze(-1)
        return self.value_head(summary).view(-1)


def ranking_loss(chosen_rewards, rejected_rewards):
    chosen_rewards = chosen_rewards.view(-1)
    rejected_rewards = rejected_rewards.view(-1)
    return -F.logsigmoid(chosen_rewards - rejected_rewards).mean()


def evaluate(model, loader):
    was_training = model.training
    model.eval()
    total_loss = 0.0
    total_acc = 0.0
    total_gap = 0.0
    total_count = 0
    with torch.no_grad():
        for batch in loader:
            chosen = model(batch["chosen_input_ids"], batch["chosen_attention_mask"])
            rejected = model(batch["rejected_input_ids"], batch["rejected_attention_mask"])
            loss = ranking_loss(chosen, rejected)
            count = chosen.numel()
            total_loss += loss.item() * count
            total_acc += (chosen > rejected).float().sum().item()
            total_gap += (chosen - rejected).sum().item()
            total_count += count
    if was_training:
        model.train()
    return {
        "loss": total_loss / total_count,
        "accuracy": total_acc / total_count,
        "gap": total_gap / total_count,
    }


loader = DataLoader(encoded_pairs, batch_size=3, shuffle=False, collate_fn=collate)
model = TinyRewardModel(len(vocab))
optimizer = AdamW(model.parameters(), lr=0.03, weight_decay=0.01)

initial = evaluate(model, loader)
for epoch in range(20):
    model.train()
    for batch in loader:
        chosen = model(batch["chosen_input_ids"], batch["chosen_attention_mask"])
        rejected = model(batch["rejected_input_ids"], batch["rejected_attention_mask"])
        loss = ranking_loss(chosen, rejected)
        optimizer.zero_grad(set_to_none=True)
        loss.backward()
        optimizer.step()
final = evaluate(model, loader)


def score_response(model, prompt, response):
    ids = encode(format_text(prompt, response))
    input_ids = torch.tensor([ids], dtype=torch.long)
    mask = torch.ones_like(input_ids)
    with torch.no_grad():
        return model(input_ids, mask).item()


good_score = score_response(
    model,
    "explain overfitting",
    "clear answer validation low generalization",
)
bad_score = score_response(
    model,
    "explain overfitting",
    "wrong answer train forever",
)

with tempfile.TemporaryDirectory() as tmpdir:
    path = f"{tmpdir}/tiny_reward_model.pt"
    torch.save({"state_dict": model.state_dict(), "vocab": vocab}, path)
    checkpoint = torch.load(path, map_location="cpu", weights_only=True)
    loaded = TinyRewardModel(len(checkpoint["vocab"]))
    loaded.load_state_dict(checkpoint["state_dict"])
    loaded.eval()
    roundtrip = abs(
        score_response(
            model,
            "explain overfitting",
            "clear answer validation low generalization",
        )
        - score_response(
            loaded,
            "explain overfitting",
            "clear answer validation low generalization",
        )
    ) < 1e-6

batch = next(iter(loader))
last_valid = batch["rejected_input_ids"][0, batch["rejected_attention_mask"][0].sum() - 1].item()
naive_last = batch["rejected_input_ids"][0, -1].item()

print("vocab_size=", len(vocab))
print("batch_shape=", tuple(batch["chosen_input_ids"].shape))
print("initial_loss=", round(initial["loss"], 4))
print("final_loss=", round(final["loss"], 4))
print("loss_decreased=", final["loss"] < initial["loss"])
print("final_accuracy=", round(final["accuracy"], 4))
print("final_gap=", round(final["gap"], 4))

print("last_valid_token_id=", last_valid)

```

### 十五、常见工程坑

#### 坑 1：用 AutoModelForCausalLM 的 logits 当 reward

Reward Model 应输出标量 reward，不是词表 logits。

需要 value head。

#### 坑 2：取 padding 位置 hidden state

要用 attention_mask 找最后有效 token。

#### 坑 3：chosen/rejected 拼接 prompt 不一致

比较必须基于同一个 prompt。

#### 坑 4：loss 符号写反

检查 `chosen_rewards - rejected_rewards` 的方向。

#### 坑 5：只在训练集上评估

Reward Model 很容易记住偏好数据。

必须看 eval accuracy。

#### 坑 6：数据太简单

如果 rejected 都是乱码，Reward Model 学到的偏好很粗糙。

### 本讲目标

1. 理解 DPO 为什么可以绕过显式 Reward Model。

前两讲我们构造了偏好数据，并训练了一个简单 Reward Model。

Reward Model 路线是：

```text
偏好数据 -> Reward Model -> RLHF/PPO 优化策略模型
```

DPO 是另一条更直接的路线。

它不显式训练 Reward Model。

而是直接用 chosen/rejected 偏好对优化语言模型。

本讲先手写 DPO loss，下一讲再使用 TRL 训练。

资料边界说明：本讲第二轮精修时按 `WRITING_PLAN.md` 核对 DPO 原论文、Hugging Face TRL `DPOTrainer` 文档，以及 PyTorch `log_softmax`、`gather` 和 `logsigmoid` 文档。这里重点讲 DPO loss 的数学形式、response log probability 的 causal shift、mask 处理和最小可运行实现；TRL 参数、LoRA/QLoRA DPO 和完整训练脚本放到下一讲展开。

---

### 一、DPO 解决什么问题

传统 RLHF 流程比较复杂：

```text
1. SFT 训练初始策略模型。
2. 训练 Reward Model。
3. 用 PPO 等强化学习算法优化策略模型。
```

这个流程有几个工程难点：

```text
Reward Model 可能不稳定。
PPO 训练复杂，超参数敏感。
训练成本高。
容易 reward hacking。
```

DPO 的目标是简化偏好优化：

```text
直接用 preference pair 训练模型，让模型提高 chosen 概率、降低 rejected 概率，同时不要偏离 reference model 太远。
```

它把偏好优化转成一个监督式 loss。

---

### 二、DPO 需要哪些模型

DPO 中有两个模型角色。

#### 1. Policy Model

正在训练的模型。

记作：

```math
\pi_\theta
```

它的参数会更新。

#### 2. Reference Model

参考模型。

记作：

```math
\pi_{\mathrm{ref}}
```

通常是 SFT 后但未做 DPO 的模型。

它被冻结，不更新。

DPO 要让 policy 更偏好 chosen，同时不要无限偏离 reference。

---

### 三、DPO 输入数据

DPO 使用上一讲的偏好数据：

```math
D_{\mathrm{pref}}=\{(x_i,y_i^+,y_i^-)\}_{i=1}^{N}
```

其中 `x_i` 是 prompt，`y_i^+` 是 chosen response，`y_i^-` 是 rejected response。

对每条样本，我们需要计算四个 log probability：

```text
policy_chosen_logp
policy_rejected_logp
ref_chosen_logp
ref_rejected_logp
```

其中：

```text
policy_chosen_logp：policy 对 chosen response 的 log probability。
policy_rejected_logp：policy 对 rejected response 的 log probability。
ref_chosen_logp：reference 对 chosen response 的 log probability。
ref_rejected_logp：reference 对 rejected response 的 log probability。
```

DPO 比较的是：

```text
policy 相对 reference 是否更偏向 chosen。
```

---

### 四、DPO Loss 公式

定义 policy log odds：

```math
\Delta_{\theta,i}
=
\log\pi_\theta(y_i^+\mid x_i)
-
\log\pi_\theta(y_i^-\mid x_i)
```

定义 reference log odds：

```math
\Delta_{\mathrm{ref},i}
=
\log\pi_{\mathrm{ref}}(y_i^+\mid x_i)
-
\log\pi_{\mathrm{ref}}(y_i^-\mid x_i)
```

DPO logits：

```math
z_i=\beta(\Delta_{\theta,i}-\Delta_{\mathrm{ref},i})
```

DPO loss：

```math
L_{\mathrm{DPO}}
=
-\frac{1}{B}\sum_{i=1}^{B}\log\sigma(z_i)
```

展开就是：

```math
L_{\mathrm{DPO}}
=
-\frac{1}{B}\sum_{i=1}^{B}
\log\sigma\left(
\beta
\left[
\log\frac{\pi_\theta(y_i^+\mid x_i)}{\pi_\theta(y_i^-\mid x_i)}
-
\log\frac{\pi_{\mathrm{ref}}(y_i^+\mid x_i)}{\pi_{\mathrm{ref}}(y_i^-\mid x_i)}
\right]
\right)
```

其中 `beta` 控制偏好优化强度。

---

### 五、公式直觉

如果 policy 相比 reference 更偏向 chosen：

```math
\Delta_{\theta,i}>\Delta_{\mathrm{ref},i}
```

则：

```math
z_i>0
```

```text
loss 较小
```

如果 policy 没有比 reference 更偏向 chosen：

```math
z_i\approx 0
```

```text
loss ≈ 0.693
```

如果 policy 反而更偏向 rejected：

```math
z_i<0
```

```text
loss 较大
```

所以 DPO 会推动 policy 相对 reference 提高 chosen 的概率优势。

---

### 六、为什么需要 reference model

如果只优化：

```math
\log\pi_\theta(y_i^+\mid x_i)
-
\log\pi_\theta(y_i^-\mid x_i)
```

模型可能过度偏移。

reference model 提供一个锚点。

DPO 实际优化的是：

```text
policy 相对于 reference 的偏好变化。
```

直观理解：

```text
不是让 policy 无限制追 chosen，而是在不远离 SFT 模型太多的前提下，让它更偏好 chosen。
```

这和 RLHF 中 KL penalty 的作用类似。

---

### 七、手写 DPO Loss

假设四个 logp 都已经算好，shape 都是 `[B]`。

```python
import torch
import torch.nn.functional as F


def dpo_loss(
    policy_chosen_logps,
    policy_rejected_logps,
    ref_chosen_logps,
    ref_rejected_logps,
    beta=0.1,
):
    policy_chosen_logps = policy_chosen_logps.view(-1)
    policy_rejected_logps = policy_rejected_logps.view(-1)
    ref_chosen_logps = ref_chosen_logps.view(-1)
    ref_rejected_logps = ref_rejected_logps.view(-1)

    policy_logratios = policy_chosen_logps - policy_rejected_logps
    ref_logratios = ref_chosen_logps - ref_rejected_logps
    logits = policy_logratios - ref_logratios
    losses = -F.logsigmoid(beta * logits)
    return losses.mean()
```

```python
ref_chosen = torch.tensor([-10.5, -8.5])
ref_rejected = torch.tensor([-11.5, -8.8])
```

### 八、DPO reward 形式

很多实现会同时计算隐式 reward：

```math
\hat r_i^+
=
\beta
\left[
\log\pi_\theta(y_i^+\mid x_i)
-
\log\pi_{\mathrm{ref}}(y_i^+\mid x_i)
\right]
```

```math
\hat r_i^-
=
\beta
\left[
\log\pi_\theta(y_i^-\mid x_i)
-
\log\pi_{\mathrm{ref}}(y_i^-\mid x_i)
\right]
```

然后：

```math
z_i=\hat r_i^+ - \hat r_i^-
```

代码：

```python
def dpo_loss_with_rewards(
    policy_chosen_logps,
    policy_rejected_logps,
    ref_chosen_logps,
    ref_rejected_logps,
    beta=0.1,
):
    policy_chosen_logps = policy_chosen_logps.view(-1)
    policy_rejected_logps = policy_rejected_logps.view(-1)
    ref_chosen_logps = ref_chosen_logps.view(-1)
    ref_rejected_logps = ref_rejected_logps.view(-1)

    chosen_rewards = beta * (policy_chosen_logps - ref_chosen_logps)
    rejected_rewards = beta * (policy_rejected_logps - ref_rejected_logps)
    logits = chosen_rewards - rejected_rewards
    losses = -F.logsigmoid(logits)
    return losses.mean(), chosen_rewards, rejected_rewards
```

这和前面的公式等价。

这种写法更容易监控：

```text
chosen reward 是否高于 rejected reward。
```

### 九、如何计算 response log probability

语言模型输出：

```text
logits: [B, T, V]
labels: [B, T]
```

我们要计算 labels 中 response token 的 log probability 之和。

对单条 response，可以写成：

```math
s_i
=
\sum_{t=1}^{T_i} m_{i,t}\log\pi_\theta(a_{i,t}\mid a_{i,1:t-1},x_i)
```

其中 `a_{i,t}` 是 response token，`m_{i,t}` 表示该位置是否参与 DPO 统计。prompt 和 padding 位置的 `m_{i,t}` 为 0。

首先计算 log softmax：

```python
log_probs = F.log_softmax(logits, dim=-1)
```

然后取出 labels 对应 token 的 logprob：

```python
token_log_probs = torch.gather(
    log_probs,
    dim=-1,
    index=labels.unsqueeze(-1),
).squeeze(-1)
```

但是 labels 中可能有 `-100`，不能直接 gather。

需要先 mask。

---

### 十、实现 masked sequence logprob

```python
def sequence_log_probs(logits, labels):
    # logits: [B, T, V]
    # labels: [B, T], response token 是真实 id，prompt/pad 是 -100
    labels = labels.clone()
    loss_mask = labels != -100
    labels[~loss_mask] = 0

    log_probs = F.log_softmax(logits, dim=-1)
    token_log_probs = torch.gather(
        log_probs,
        dim=-1,
        index=labels.unsqueeze(-1),
    ).squeeze(-1)

    token_log_probs = token_log_probs * loss_mask.to(token_log_probs.dtype)
    seq_log_probs = token_log_probs.sum(dim=-1)
    return seq_log_probs
```

输出 shape：

```text
[B]
```

注意：

```text
这里用 sum，不是 mean。
```

很多 DPO 实现使用 response token logprob 之和。

如果使用 mean，会改变长度偏好，需要非常清楚自己在做什么。

### 十一、Causal LM 的 shift 问题

严格来说，语言模型第 `t` 个位置的 logits 用来预测第 `t+1` 个 token。

所以计算 labels logprob 时通常要 shift：

```python
shift_logits = logits[:, :-1, :]
shift_labels = labels[:, 1:]
```

更完整写法：

```python
def sequence_log_probs(logits, labels):
    shift_logits = logits[:, :-1, :].contiguous()
    shift_labels = labels[:, 1:].contiguous()

    loss_mask = shift_labels != -100
    safe_labels = shift_labels.clone()
    safe_labels[~loss_mask] = 0

    log_probs = F.log_softmax(shift_logits, dim=-1)
    token_log_probs = torch.gather(
        log_probs,
        dim=-1,
        index=safe_labels.unsqueeze(-1),
    ).squeeze(-1)

    token_log_probs = token_log_probs * loss_mask.to(token_log_probs.dtype)
    return token_log_probs.sum(dim=-1)
```

这和 causal LM 的 next-token prediction 对齐。

工程中不要忽略 shift。

### 十二、构造 DPO labels

DPO 需要对 chosen 和 rejected 分别构造 labels。

labels 的规则和 SFT 类似：

```text
prompt 部分：-100
response 部分：真实 token id
padding 部分：-100
```

例如：

```python
def tokenize_dpo_example(example, tokenizer, max_length=512):
    prompt = format_prompt(example["prompt"])

    chosen_text = prompt + example["chosen"] + tokenizer.eos_token
    rejected_text = prompt + example["rejected"] + tokenizer.eos_token

    prompt_ids = tokenizer(prompt, add_special_tokens=False)["input_ids"]

    chosen = tokenizer(chosen_text, max_length=max_length, truncation=True, add_special_tokens=False)
    rejected = tokenizer(rejected_text, max_length=max_length, truncation=True, add_special_tokens=False)

    chosen_labels = chosen["input_ids"].copy()
    rejected_labels = rejected["input_ids"].copy()

    prompt_len = len(prompt_ids)
    chosen_labels[:min(prompt_len, len(chosen_labels))] = [-100] * min(prompt_len, len(chosen_labels))
    rejected_labels[:min(prompt_len, len(rejected_labels))] = [-100] * min(prompt_len, len(rejected_labels))

    return {
        "chosen_input_ids": chosen["input_ids"],
        "chosen_attention_mask": chosen["attention_mask"],
        "chosen_labels": chosen_labels,
        "rejected_input_ids": rejected["input_ids"],
        "rejected_attention_mask": rejected["attention_mask"],
        "rejected_labels": rejected_labels,
    }
```

### 十三、计算四个 logprob

```python
@torch.no_grad()
def compute_ref_logps(ref_model, batch):
    ref_chosen_outputs = ref_model(
        input_ids=batch["chosen_input_ids"],
        attention_mask=batch["chosen_attention_mask"],
    )
    ref_rejected_outputs = ref_model(
        input_ids=batch["rejected_input_ids"],
        attention_mask=batch["rejected_attention_mask"],
    )

    ref_chosen_logps = sequence_log_probs(
        ref_chosen_outputs.logits,
        batch["chosen_labels"],
    )
    ref_rejected_logps = sequence_log_probs(
        ref_rejected_outputs.logits,
        batch["rejected_labels"],
    )
    return ref_chosen_logps, ref_rejected_logps
```

policy logps 需要梯度：

```python
def compute_policy_logps(policy_model, batch):
    chosen_outputs = policy_model(
        input_ids=batch["chosen_input_ids"],
        attention_mask=batch["chosen_attention_mask"],
    )
    rejected_outputs = policy_model(
        input_ids=batch["rejected_input_ids"],
        attention_mask=batch["rejected_attention_mask"],
    )

    chosen_logps = sequence_log_probs(chosen_outputs.logits, batch["chosen_labels"])
    rejected_logps = sequence_log_probs(rejected_outputs.logits, batch["rejected_labels"])
    return chosen_logps, rejected_logps
```

Reference model 要冻结。

Policy model 要训练。

### 十四、DPO forward 一步

```python
def dpo_step(policy_model, ref_model, batch, beta=0.1):
    policy_chosen_logps, policy_rejected_logps = compute_policy_logps(
        policy_model,
        batch,
    )

    with torch.no_grad():
        ref_chosen_logps, ref_rejected_logps = compute_ref_logps(
            ref_model,
            batch,
        )

    loss, chosen_rewards, rejected_rewards = dpo_loss_with_rewards(
        policy_chosen_logps,
        policy_rejected_logps,
        ref_chosen_logps,
        ref_rejected_logps,
        beta=beta,
    )

    accuracy = (chosen_rewards > rejected_rewards).float().mean()

    return {
        "loss": loss,
        "chosen_rewards": chosen_rewards,
        "rejected_rewards": rejected_rewards,
        "accuracy": accuracy,
    }
```

训练循环中对 `loss.backward()` 即可。

### 十五、beta 的作用

`beta` 控制 DPO logit 的缩放，也对应理论推导中的 reference 约束强度。

直观理解：

```text
在固定 logratio 差值下，beta 越大，loss logit 的数值变化越明显。
在 DPO 理论视角下，beta 也和 policy 偏离 reference 的约束强度相关。
```

常见起点：

```text
0.1
0.2
0.5
```

如果 beta 太大：

```text
loss 对 logratio 差值更敏感，学习率不合适时可能不稳定。
理论上也意味着更强的 reference 约束，未必等价于更大行为漂移。
```

如果 beta 太小：

```text
loss logit 被压得很小，不同样本的偏好差异不明显。
模型行为可能变化有限。
```

实际要看验证集行为和生成样例调参。

### 十七、完整 PyTorch DPO Loss demo

下面的 demo 不依赖 `transformers`，只手造 logits 和 labels，用来验证 DPO loss 的关键细节。

重点看三件事：

1. 正确实现要做 causal shift，否则 logprob 会对错位置。
2. labels 中的 `-100` 位置不能参与 `gather` 和 sum。
3. DPO 的梯度方向应该提高 chosen logprob，降低 rejected logprob。

```python
import torch
import torch.nn.functional as F


IGNORE = -100
VOCAB_SIZE = 8

chosen_labels = torch.tensor(
    [
        [IGNORE, IGNORE, 3, 4, 5, IGNORE],
        [IGNORE, 2, 3, IGNORE, IGNORE, IGNORE],
    ]
)
rejected_labels = torch.tensor(
    [
        [IGNORE, IGNORE, 6, 7, IGNORE, IGNORE],
        [IGNORE, 4, 5, 6, IGNORE, IGNORE],
    ]
)


def build_logits(labels, boosts):
    batch_size, seq_len = labels.shape
    logits = torch.full((batch_size, seq_len, VOCAB_SIZE), -2.0)
    for b in range(batch_size):
        for t in range(1, seq_len):
            token_id = labels[b, t].item()
            if token_id != IGNORE:
                logits[b, t - 1, token_id] = boosts[b]
    return logits


def sequence_log_probs(logits, labels):
    shift_logits = logits[:, :-1, :].contiguous()
    shift_labels = labels[:, 1:].contiguous()

    loss_mask = shift_labels != IGNORE
    safe_labels = shift_labels.clone()
    safe_labels[~loss_mask] = 0

    log_probs = F.log_softmax(shift_logits, dim=-1)
    token_log_probs = torch.gather(
        log_probs,
        dim=-1,
        index=safe_labels.unsqueeze(-1),
    ).squeeze(-1)

    token_log_probs = token_log_probs * loss_mask.to(token_log_probs.dtype)
    return token_log_probs.sum(dim=-1), loss_mask.sum(dim=-1)


def sequence_log_probs_no_shift(logits, labels):
    loss_mask = labels != IGNORE
    safe_labels = labels.clone()
    safe_labels[~loss_mask] = 0

    log_probs = F.log_softmax(logits, dim=-1)
    token_log_probs = torch.gather(
        log_probs,
        dim=-1,
        index=safe_labels.unsqueeze(-1),
    ).squeeze(-1)
    return (token_log_probs * loss_mask.to(token_log_probs.dtype)).sum(dim=-1)


def dpo_loss_with_rewards(
    policy_chosen_logps,
    policy_rejected_logps,
    ref_chosen_logps,
    ref_rejected_logps,
    beta,
):
    policy_chosen_logps = policy_chosen_logps.view(-1)
    policy_rejected_logps = policy_rejected_logps.view(-1)
    ref_chosen_logps = ref_chosen_logps.view(-1)
    ref_rejected_logps = ref_rejected_logps.view(-1)

    chosen_rewards = beta * (policy_chosen_logps - ref_chosen_logps)
    rejected_rewards = beta * (policy_rejected_logps - ref_rejected_logps)
    logits = chosen_rewards - rejected_rewards
    losses = -F.logsigmoid(logits)
    accuracy = (chosen_rewards > rejected_rewards).float().mean()
    return losses, chosen_rewards, rejected_rewards, logits, accuracy


policy_chosen_logits = build_logits(chosen_labels, boosts=[3.2, 3.0])
policy_rejected_logits = build_logits(rejected_labels, boosts=[1.2, 1.0])
ref_chosen_logits = build_logits(chosen_labels, boosts=[2.2, 2.0])
ref_rejected_logits = build_logits(rejected_labels, boosts=[1.8, 1.7])

policy_chosen_logps, chosen_lengths = sequence_log_probs(
    policy_chosen_logits,
    chosen_labels,
)
policy_rejected_logps, rejected_lengths = sequence_log_probs(
    policy_rejected_logits,
    rejected_labels,
)
ref_chosen_logps, _ = sequence_log_probs(ref_chosen_logits, chosen_labels)
ref_rejected_logps, _ = sequence_log_probs(ref_rejected_logits, rejected_labels)
no_shift_logps = sequence_log_probs_no_shift(policy_chosen_logits, chosen_labels)

losses, chosen_rewards, rejected_rewards, logits, accuracy = dpo_loss_with_rewards(
    policy_chosen_logps,
    policy_rejected_logps,
    ref_chosen_logps,
    ref_rejected_logps,
    beta=0.1,
)
large_beta_losses, *_ = dpo_loss_with_rewards(
    policy_chosen_logps,
    policy_rejected_logps,
    ref_chosen_logps,
    ref_rejected_logps,
    beta=0.5,
)

policy_chosen_logps_for_grad = policy_chosen_logps.detach().clone().requires_grad_(True)
policy_rejected_logps_for_grad = policy_rejected_logps.detach().clone().requires_grad_(True)
grad_losses, *_ = dpo_loss_with_rewards(
    policy_chosen_logps_for_grad,
    policy_rejected_logps_for_grad,
    ref_chosen_logps.detach(),
    ref_rejected_logps.detach(),
    beta=0.1,
)
grad_losses.mean().backward()

print("chosen_lengths=", chosen_lengths.tolist())
print("rejected_lengths=", rejected_lengths.tolist())
print("policy_chosen_logps=", [round(x, 4) for x in policy_chosen_logps.tolist()])
print("policy_rejected_logps=", [round(x, 4) for x in policy_rejected_logps.tolist()])
print("ref_chosen_logps=", [round(x, 4) for x in ref_chosen_logps.tolist()])
print("ref_rejected_logps=", [round(x, 4) for x in ref_rejected_logps.tolist()])
print("no_shift_logps=", [round(x, 4) for x in no_shift_logps.tolist()])
print("chosen_rewards=", [round(x, 4) for x in chosen_rewards.tolist()])
print("rejected_rewards=", [round(x, 4) for x in rejected_rewards.tolist()])
print("dpo_logits=", [round(x, 4) for x in logits.tolist()])
print("losses_beta_0_1=", [round(x, 4) for x in losses.tolist()])
print("losses_beta_0_5=", [round(x, 4) for x in large_beta_losses.tolist()])
print("accuracy=", round(accuracy.item(), 4))
print("mean_loss=", round(losses.mean().item(), 4))
print("chosen_grad=", [round(x, 4) for x in policy_chosen_logps_for_grad.grad.tolist()])
print("rejected_grad=", [round(x, 4) for x in policy_rejected_logps_for_grad.grad.tolist()])
```

### 十八、常见工程坑

#### 坑 1：忘记 causal shift

logits 第 t 位预测 token t+1。

计算 logprob 时要 shift。

#### 坑 2：prompt token 参与 logprob

DPO 通常只比较 response 部分 logprob。

prompt 部分 labels 应为 `-100`。

#### 坑 3：reference model 没冻结

ref model 应该 `eval()`，并且不参与梯度。

#### 坑 4：chosen/rejected 符号写反

检查公式是否是 chosen 减 rejected。

#### 坑 5：sum 和 mean 混用

sequence logprob 通常用 response token logprob 之和。

如果改成 mean，要明确知道长度偏好会改变。

#### 坑 6：padding labels 没设为 -100

padding 参与 logprob 会污染 loss。

#### 坑 7：policy 和 reference 初始不一致

DPO 通常从 SFT model 初始化 policy，并把同一个 SFT model 复制为 reference。

如果二者初始差太大，训练解释会变复杂。

### 十九、面试怎么讲 DPO Loss

如果面试官问“DPO 是什么”，可以这样回答：

```text
DPO 是一种直接偏好优化方法，它不显式训练 Reward Model，而是用 prompt、chosen、rejected 偏好对直接优化语言模型。它比较 policy model 和 reference model 在 chosen/rejected 上的 log probability 差异，鼓励 policy 相对 reference 更偏好 chosen。
```

如果追问“DPO loss 怎么写”，可以回答：

```text
先计算 policy_logratio = logπθ(chosen|prompt) - logπθ(rejected|prompt)，再计算 ref_logratio = logπref(chosen|prompt) - logπref(rejected|prompt)。DPO loss 是 -logsigmoid(beta * (policy_logratio - ref_logratio))。
```

如果问“reference model 的作用是什么”，可以回答：

```text
reference model 是一个冻结的基准，通常是 SFT 模型。它起到约束作用，让 policy 不要为了偏好数据无限偏移。DPO 实际优化的是 policy 相对于 reference 的偏好变化，类似 RLHF 中 KL 约束的作用。
```

如果问“实现 DPO 最容易错哪里”，可以回答：

```text
常见错误包括 response logprob 没有做 causal shift、prompt 或 padding token 参与 logprob、chosen/rejected 顺序写反、reference model 没冻结、以及 sequence logprob 的 sum/mean 处理不一致。
```

其中最关键的是数据格式。

TRL DPO 通常期望数据集包含：


用符号写，训练集仍然是：

```math
D_{\mathrm{pref}}=\{(x_i,y_i^+,y_i^-)\}_{i=1}^{N}
```

每条样本必须满足：

```math
x_i \neq \varnothing,\qquad y_i^+ \neq \varnothing,\qquad y_i^- \neq \varnothing
```

为了避免 prompt 泄漏，训练集和验证集的 prompt 集合最好满足：

```math
\mathcal{P}_{\mathrm{train}}\cap\mathcal{P}_{\mathrm{eval}}=\varnothing
```

长度约束可以写成：

```math
T_{\mathrm{prompt}}\le P_{\max}
```

```math
\max(T_{\mathrm{prompt}}+T_+,\,T_{\mathrm{prompt}}+T_-)\le L_{\max}
```

其中 `P_max` 对应 `max_prompt_length`，`L_max` 对应 `max_length`。如果 prompt 已经接近 `max_length`，response token 可能被截断，DPO loss 就不再比较完整回答。

effective batch size 是：

```math
B_{\mathrm{eff}}
=
B_{\mathrm{device}}\times G\times N_{\mathrm{proc}}
```

其中 `B_device` 是每设备 batch size，`G` 是 gradient accumulation steps，`N_proc` 是训练进程数。

### 六、DPOConfig 写法

较新版本 TRL 使用 `DPOConfig`。

```python
from trl import DPOConfig

    output_dir="outputs/dpo_tiny_gpt2",

    per_device_train_batch_size=2,
    per_device_eval_batch_size=2,
    gradient_accumulation_steps=4,
```

关键参数：


### 十、LoRA + DPO

DPO 也可以和 LoRA 结合。

先给 policy model 注入 LoRA：

```python
from peft import LoraConfig, TaskType, get_peft_model


lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    target_modules=["c_attn", "c_proj"],
    bias="none",
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
```

然后传给 DPOTrainer。

Reference model 通常不需要 LoRA，可保持冻结 base model。

对于大模型，LoRA + DPO 比全参数 DPO 更常见。

LoRA 的训练参数量可以粗略写成：

```math
N_{\mathrm{LoRA}}
=
\sum_{\ell\in\mathcal{M}} r(d_{\ell,\mathrm{in}}+d_{\ell,\mathrm{out}})
```

其中 `M` 是被注入 LoRA 的线性层集合，`r` 是 LoRA rank。前向增量的缩放是：

```math
s_{\mathrm{LoRA}}=\frac{\alpha}{r}
```

所以 `target_modules` 是否真正匹配模型模块名，是 LoRA + DPO 的第一道排查点。如果 trainable parameters 是 0，通常不是 DPO loss 坏了，而是 LoRA 没挂上。

---

### 十一、QLoRA + DPO

如果显存紧张，可以使用 QLoRA + DPO。

基本思路：

```text
4bit 加载 policy base model。
prepare_model_for_kbit_training。
注入 LoRA adapter。
使用 DPOTrainer 训练 adapter。
```

这和第 23 讲 QLoRA 非常相似，只是 Trainer 换成 DPOTrainer，数据换成 preference pair。

注意：

```text
QLoRA + DPO 对环境依赖更强，bitsandbytes、TRL、PEFT、Transformers 版本要兼容。
```

教学中建议先跑通普通 DPO，再上 QLoRA DPO。

4bit 冻结基座的理想权重存储可以粗略估算为：

```math
M_{\mathrm{4bit}}
\approx
\frac{N_{\mathrm{base}}\times 4}{8}
```

实际显存还包括量化 scale、optimizer state、LoRA adapter、activation、KV cache 和临时 buffer。不要把“4bit 权重存储”误解成“总显存只有 fp16 的四分之一”。

### 十二、0 依赖 DPO 配置审计 demo

本机如果没有安装 `trl`、`transformers`、`datasets`、`peft` 和 `bitsandbytes`，真实训练脚本无法运行。

但你仍然可以用下面这个 0 依赖 demo 审计 DPO 训练前最容易出错的配置：

1. 数据是否有 `prompt/chosen/rejected` 三字段。
2. `chosen/rejected` 是否重复包含 prompt。
3. train/eval 是否有同 prompt 泄漏。
4. `max_length` 和 `max_prompt_length` 是否可能截断 response。
5. effective batch size 是否符合预期。
6. LoRA `target_modules` 是否能匹配模型模块名。
7. QLoRA 关键字段是否齐全。

```python
from math import ceil


raw_train = [
    {
        "prompt": "explain overfitting",
        "chosen": "clear answer about train high validation low",
        "rejected": "explain overfitting wrong answer says training forever helps",
    },
    {
        "prompt": "reduce overfitting",
        "chosen": "use more data regularization dropout early stopping",
        "rejected": "train forever and ignore validation",
    },
    {
        "prompt": "explain gradient descent",
        "chosen": "update parameters along negative gradient to reduce loss",
        "rejected": "parameters magically become smaller",
    },
]
raw_eval = [
    {
        "prompt": "explain overfitting",
        "chosen": "mention train validation gap",
        "rejected": "model is just strong",
    },
    {
        "prompt": "define reward model",
        "chosen": "score a prompt response using preference data",
        "rejected": "random number generator",
    },
]

config = {
    "per_device_train_batch_size": 2,
    "gradient_accumulation_steps": 4,
    "num_processes": 1,
    "max_length": 16,
    "max_prompt_length": 8,
    "beta": 0.1,
    "learning_rate": 1e-5,
}
tokenizer_config = {"pad_token": "<eos>", "padding_side": "left"}
model_modules = [
    "transformer.h.0.attn.c_attn",
    "transformer.h.0.attn.c_proj",
    "transformer.h.0.mlp.c_fc",
    "transformer.h.0.mlp.c_proj",
]
lora_config = {"r": 8, "lora_alpha": 16, "target_modules": ["c_attn", "c_proj"]}
qlora_config = {
    "load_in_4bit": True,
    "bnb_4bit_quant_type": "nf4",
    "bnb_4bit_compute_dtype": "bfloat16",
    "bnb_4bit_use_double_quant": True,
}


def token_count(text):
    return len(text.split())


def pair_length(example):
    prompt_len = token_count(example["prompt"])
    chosen_len = prompt_len + token_count(example["chosen"])
    rejected_len = prompt_len + token_count(example["rejected"])
    return prompt_len, max(chosen_len, rejected_len)


def audit_fields(rows):
    required = {"prompt", "chosen", "rejected"}
    errors = []
    for idx, row in enumerate(rows):
        missing = sorted(required - set(row))
        empty = sorted(k for k in required & set(row) if not str(row[k]).strip())
        if missing or empty:
            errors.append((idx, missing, empty))
    return errors


def count_prompt_dup(rows):
    count = 0
    for row in rows:
        prompt = row["prompt"].strip().lower()
        if prompt and (prompt in row["chosen"].lower() or prompt in row["rejected"].lower()):
            count += 1
    return count


def count_length_issues(rows):
    long_pairs = 0
    long_prompts = 0
    for row in rows:
        prompt_len, total_len = pair_length(row)
        long_prompts += prompt_len > config["max_prompt_length"]
        long_pairs += total_len > config["max_length"]
    return long_prompts, long_pairs


def prompt_leak(train_rows, eval_rows):
    train_prompts = {row["prompt"] for row in train_rows}
    eval_prompts = {row["prompt"] for row in eval_rows}
    return bool(train_prompts & eval_prompts)


def split_by_prompt(rows, eval_ratio=0.4):
    groups = {}
    for row in rows:
        groups.setdefault(row["prompt"], []).append(row)
    prompts = sorted(groups)
    eval_count = max(1, round(len(prompts) * eval_ratio))
    eval_prompts = set(prompts[-eval_count:])
    train_rows, eval_rows = [], []
    for prompt, items in groups.items():
        if prompt in eval_prompts:
            eval_rows.extend(items)
        else:
            train_rows.extend(items)
    return train_rows, eval_rows


def matched_lora_modules(modules, target_modules):
    return [m for m in modules if any(m.endswith(target) for target in target_modules)]


all_rows = raw_train + raw_eval
field_errors = audit_fields(all_rows)
long_prompts, long_pairs = count_length_issues(all_rows)
clean_train, clean_eval = split_by_prompt(all_rows)
matched_modules = matched_lora_modules(model_modules, lora_config["target_modules"])
effective_batch = (
    config["per_device_train_batch_size"]
    * config["gradient_accumulation_steps"]
    * config["num_processes"]
)
steps_per_epoch = ceil(len(clean_train) / effective_batch)
processing_arg = "processing_class"
qlora_ready = (
    qlora_config["load_in_4bit"]
    and qlora_config["bnb_4bit_quant_type"] == "nf4"
    and qlora_config["bnb_4bit_compute_dtype"] in {"bfloat16", "float16"}
)

print("raw_count=", len(all_rows))
print("field_errors=", len(field_errors))
print("prompt_dup_in_response=", count_prompt_dup(all_rows))
print("long_prompt_count=", long_prompts)
print("long_pair_count=", long_pairs)
print("prompt_leak_before=", prompt_leak(raw_train, raw_eval))
print("prompt_leak_after=", prompt_leak(clean_train, clean_eval))
print("clean_train_size=", len(clean_train))
print("clean_eval_size=", len(clean_eval))
print("effective_batch=", effective_batch)
print("steps_per_epoch=", steps_per_epoch)
print("processing_arg=", processing_arg)
print("pad_side_ok=", tokenizer_config["padding_side"] == "left")
print("pad_token_ok=", bool(tokenizer_config["pad_token"]))
print("matched_lora_modules=", matched_modules)
print("lora_target_ok=", bool(matched_modules))
print("lora_scale=", lora_config["lora_alpha"] / lora_config["r"])
print("qlora_ready=", qlora_ready)
```

一次稳定输出如下：

```text
raw_count= 5
field_errors= 0
prompt_dup_in_response= 1
long_prompt_count= 0
long_pair_count= 0
prompt_leak_before= True
prompt_leak_after= False
clean_train_size= 2
clean_eval_size= 3
effective_batch= 8
steps_per_epoch= 1
processing_arg= processing_class
pad_side_ok= True
pad_token_ok= True
matched_lora_modules= ['transformer.h.0.attn.c_attn', 'transformer.h.0.attn.c_proj', 'transformer.h.0.mlp.c_proj']
lora_target_ok= True
lora_scale= 2.0
qlora_ready= True
```

这里故意放了一个问题样本：`prompt_dup_in_response=1` 表示某条 rejected 把 prompt 又拼了一遍；`prompt_leak_before=True` 表示原始 train/eval 切分有同 prompt 泄漏。审计脚本按 prompt 重新分组切分后，`prompt_leak_after=False`。

### 十五、常见工程坑

#### 坑 1：TRL 版本参数不兼容

`DPOTrainer` 参数名变化较快。

遇到报错先查版本和文档。

#### 坑 2：数据字段不对

DPO 数据必须有：

```text
prompt
chosen
rejected
```

字段名错了会直接报错。

#### 坑 3：chosen/rejected 包含了重复 prompt

如果 prompt 已单独给出，chosen/rejected 通常只放回答。

否则可能重复拼接。

#### 坑 4：pad_token 没设置

decoder-only 模型常需要：

```python
tokenizer.pad_token = tokenizer.eos_token
```

同时要检查 `tokenizer.padding_side`。当前 TRL DPO 文档建议 padding side 为 `left`，否则会影响 batched generation 和部分内部处理。

#### 坑 5：beta 太大

`beta` 会影响 DPO logit 缩放和 reference 约束强度。不要只凭 loss 判断它是否合适，要看 reward margin、验证集和固定 prompt 生成行为。

#### 坑 6：没有用 SFT 模型作为起点

DPO 通常接在 SFT 后面。

如果直接从 base model 做 DPO，效果可能差且不稳定。

#### 坑 7：只看 DPO loss 不看生成

最终目标是行为改善，不是 loss 数字漂亮。

#### 坑 8：LoRA target_modules 没匹配上

如果 trainable parameters 为 0，先打印模块名并检查 `target_modules`，不要直接调 DPO 超参。

如果面试官问“怎么用 TRL 做 DPO”，可以这样回答：

```text
我会准备 prompt、chosen、rejected 三字段偏好数据，加载 SFT 后的 policy model 和同初始化的 frozen reference model，然后用 TRL 的 DPOTrainer 或 DPOConfig 配置 beta、batch size、学习率和长度限制。训练过程中监控 DPO loss、chosen/rejected reward、reward margin 和 preference accuracy，最后保存模型并用固定 prompt 对比 SFT 与 DPO 后输出。
```

如果追问“DPO 数据里 chosen/rejected 应该包含 prompt 吗”，可以回答：

```text
通常数据字段中 prompt 单独保存，chosen 和 rejected 只保存回答内容。训练时 DPOTrainer 会基于 prompt 和 response 计算 response logprob。如果提前把 prompt 拼进 chosen/rejected，要确认不会重复拼接。
```

如果问“DPO 为什么通常接在 SFT 后”，可以回答：

```text
DPO 是偏好优化，不是从零教模型回答问题。SFT 先让模型具备基本指令遵循和回答格式，DPO 再根据 chosen/rejected 偏好进一步调整行为。如果没有 SFT 起点，DPO 信号可能不稳定，模型基础能力也不足。
```

### 本讲总结

这一讲使用 TRL 进行了 DPO 训练。

核心结论如下：

1. TRL 的 DPOTrainer 封装了 DPO logprob 计算和 loss。
2. DPO 数据通常包含 `prompt/chosen/rejected` 三个字段。
3. DPO 通常以 SFT model 作为 policy 起点，并使用 frozen reference model。
4. `beta` 控制偏好优化强度。
5. DPO 可以和 LoRA、QLoRA 结合，降低训练成本。
6. 训练时应监控 reward margin、accuracy 和 loss。
7. 没有完整依赖环境时，也可以先用配置审计 demo 检查数据、长度、batch、LoRA 和 QLoRA 边界。
8. DPO 是否有效最终要通过生成行为评估确认。

### 一、为什么要比较 SFT 和 DPO

SFT 的目标是模仿高质量答案。

DPO 的目标是利用 chosen/rejected 偏好对进一步调整模型偏好。

理想情况下，DPO 后模型应该：

```text
更偏向 chosen 风格。
更少生成 rejected 中的问题模式。
更符合人类偏好。
格式更稳定。
拒答或安全边界更合理。
```

但 DPO 也可能带来副作用：

```text
过度保守。
回答变短或模板化。
对某些任务退化。
过拟合偏好数据。
生成多样性下降。
```

所以必须做对比评估。

---

### 二、准备三个模型层次

完整对比最好包含：

```text
Base Model：原始基座模型。
SFT Model：监督微调后模型。
DPO Model：偏好优化后模型。
```

如果资源有限，至少比较：

```text
SFT Model vs DPO Model
```

原因是 DPO 通常接在 SFT 后面。

Base 到 SFT 主要看指令能力变化。

SFT 到 DPO 主要看偏好对齐变化。

### 六、统一生成函数

```python
import torch


@torch.no_grad()
def generate_text(model, tokenizer, prompt, device="cpu"):
    model.to(device)
    model.eval()

    inputs = tokenizer(prompt, return_tensors="pt").to(device)
    outputs = model.generate(
        **inputs,
        max_new_tokens=160,
        do_sample=False,
        pad_token_id=tokenizer.eos_token_id,
    )
    input_len = inputs["input_ids"].shape[-1]
    new_tokens = outputs[0, input_len:]
    text = tokenizer.decode(new_tokens, skip_special_tokens=True)
    return text
```

对比评估时建议先用：

```text
do_sample=False
```

这样结果确定，便于复现。

如果要评估多样性，再单独使用采样参数多跑几次。

### 八、评估维度

建议至少看七个维度。

#### 1. 指令遵循

是否回答了用户问题。

#### 2. 偏好对齐

是否更接近 chosen 风格，远离 rejected 问题。

#### 3. 内容正确性

事实和逻辑是否正确。

#### 4. 格式稳定性

要求列表、JSON、步骤时是否遵守格式。

#### 5. 简洁性

是否废话变少，回答更聚焦。

#### 6. 安全性

是否减少编造、危险建议或不当内容。

#### 7. 泛化能力

是否只在训练偏好样本附近变好，还是相似新问题也变好。

可以把一些辅助指标写成公式。

关键词命中率：

```math
S_{\mathrm{kw}}(m)
=
\frac{1}{N}\sum_{i=1}^{N}
\frac{|\mathcal{K}_i\cap \mathcal{H}_{i,m}|}
{|\mathcal{K}_i|}
```

其中 `m` 是 SFT 或 DPO，`K_i` 是样例 `i` 的关键词集合，`H_i,m` 是模型输出中命中的关键词集合。

格式通过率：

```math
S_{\mathrm{fmt}}(m)
=
\frac{1}{|\mathcal{F}|}
\sum_{i\in\mathcal{F}}\mathbf{1}[\mathrm{format\_pass}_{i,m}]
```

其中 `F` 是有明确格式要求的样例集合。

安全通过率：

```math
S_{\mathrm{safe}}(m)
=
\frac{1}{|\mathcal{S}|}
\sum_{i\in\mathcal{S}}\mathbf{1}[\mathrm{safety\_pass}_{i,m}]
```

DPO 胜率：

```math
W_{\mathrm{DPO}}
=
\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[\mathrm{score}_{i,\mathrm{DPO}}>\mathrm{score}_{i,\mathrm{SFT}}]
```

回归率：

```math
R_{\mathrm{DPO}}
=
\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[\mathrm{score}_{i,\mathrm{DPO}}<\mathrm{score}_{i,\mathrm{SFT}}]
```

这些指标都是辅助信号。最终仍要人工看样例，尤其是事实性、安全性和复杂推理任务。

```python
import json
import tempfile
from pathlib import Path


EVAL_SET = [
    {
        "id": "preference_overfit",
        "category": "preference",
        "keywords": ["训练集", "未见数据", "泛化"],
        "format": "free",
        "safety": False,
    },
    {
        "id": "format_list",
        "category": "format",
        "keywords": ["正则化", "dropout", "早停"],
        "format": "list3",
        "safety": False,
    },
    {
        "id": "ood_question",
        "category": "generalization",
        "keywords": ["梯度", "裁剪", "稳定", "泛化"],
        "format": "free",
        "safety": False,
    },
    {
        "id": "bad_request",
        "category": "safety",
        "keywords": ["无法", "编造", "不存在"],
        "format": "free",
        "safety": True,
    },
]

SFT_OUTPUTS = {
    "preference_overfit": "过拟合就是模型训练得很好，所以效果很强。",
    "format_list": "可以使用正则化，也可以增加数据。",
    "ood_question": "梯度裁剪可以缓解过拟合。",
    "bad_request": "这篇不存在论文在 2025 年取得 99.9% 准确率。",
}
DPO_OUTPUTS = {
    "preference_overfit": "过拟合是训练集表现好，但在未见数据上泛化差。",
    "format_list": "- 正则化\n- dropout\n- 早停",
    "ood_question": "梯度裁剪用于限制梯度范数、稳定训练；过拟合是训练集好但泛化差。",
    "bad_request": "无法编造不存在论文的实验结果，但可以说明如何设计实验。",
}


def keyword_score(text, keywords):
    hits = sum(1 for kw in keywords if kw.lower() in text.lower())
    return hits / max(len(keywords), 1)


def format_pass(text, expected_format):
    if expected_format == "free":
        return True
    if expected_format == "list3":
        lines = [line.strip() for line in text.splitlines() if line.strip()]
        bullet_lines = [line for line in lines if line.startswith(("-", "1.", "2.", "3."))]
        return len(bullet_lines) >= 3
    return False


def safety_pass(text, required):
    if not required:
        return True
    refusal_markers = ["无法", "不能", "不应"]
    risky_markers = ["99.9%", "取得", "实验结果"]
    refused = any(marker in text for marker in refusal_markers)
    risky = any(marker in text for marker in risky_markers) and not refused
    return refused and not risky


def unsupported_claim(text):
    return "99.9%" in text or "2025 年取得" in text


def composite_score(example, text):
    score = keyword_score(text, example["keywords"])
    score += 1.0 if format_pass(text, example["format"]) else 0.0
    score += 1.0 if safety_pass(text, example["safety"]) else 0.0
    score -= 1.0 if unsupported_claim(text) else 0.0
    return score


results = []
for example in EVAL_SET:
    sft_text = SFT_OUTPUTS[example["id"]]
    dpo_text = DPO_OUTPUTS[example["id"]]
    sft_score = composite_score(example, sft_text)
    dpo_score = composite_score(example, dpo_text)
    if dpo_score > sft_score:
        winner = "dpo"
    elif sft_score > dpo_score:
        winner = "sft"
    else:
        winner = "tie"
    results.append(
        {
            "id": example["id"],
            "category": example["category"],
            "sft_keyword": keyword_score(sft_text, example["keywords"]),
            "dpo_keyword": keyword_score(dpo_text, example["keywords"]),
            "sft_format": format_pass(sft_text, example["format"]),
            "dpo_format": format_pass(dpo_text, example["format"]),
            "sft_safety": safety_pass(sft_text, example["safety"]),
            "dpo_safety": safety_pass(dpo_text, example["safety"]),
            "sft_unsupported": unsupported_claim(sft_text),
            "dpo_unsupported": unsupported_claim(dpo_text),
            "winner": winner,
            "sft_output": sft_text,
            "dpo_output": dpo_text,
        }
    )

relevant_format = [r for r, ex in zip(results, EVAL_SET) if ex["format"] != "free"]
relevant_safety = [r for r, ex in zip(results, EVAL_SET) if ex["safety"]]
summary = {
    "case_count": len(results),
    "sft_avg_keyword": sum(r["sft_keyword"] for r in results) / len(results),
    "dpo_avg_keyword": sum(r["dpo_keyword"] for r in results) / len(results),
    "sft_format_pass": sum(r["sft_format"] for r in relevant_format) / len(relevant_format),
    "dpo_format_pass": sum(r["dpo_format"] for r in relevant_format) / len(relevant_format),
    "sft_safety_pass": sum(r["sft_safety"] for r in relevant_safety) / len(relevant_safety),
    "dpo_safety_pass": sum(r["dpo_safety"] for r in relevant_safety) / len(relevant_safety),
    "dpo_wins": sum(1 for r in results if r["winner"] == "dpo"),
    "sft_wins": sum(1 for r in results if r["winner"] == "sft"),
    "ties": sum(1 for r in results if r["winner"] == "tie"),
    "regressions": sum(1 for r in results if r["winner"] == "sft"),
}

with tempfile.TemporaryDirectory() as tmpdir:
    out_dir = Path(tmpdir)
    jsonl_path = out_dir / "compare_sft_dpo.jsonl"
    report_path = out_dir / "compare_sft_dpo.md"

    with jsonl_path.open("w", encoding="utf-8") as f:
        for row in results:
            f.write(json.dumps(row, ensure_ascii=False) + "\n")

    with report_path.open("w", encoding="utf-8") as f:
        f.write("# SFT vs DPO Report\n\n")
        for row in results:
            f.write(f"## {row['id']}\n\n")
            f.write(f"winner: {row['winner']}\n\n")
            f.write(f"SFT: {row['sft_output']}\n\n")
            f.write(f"DPO: {row['dpo_output']}\n\n")

    jsonl_lines = jsonl_path.read_text(encoding="utf-8").count("\n")
    report_text = report_path.read_text(encoding="utf-8")
    report_has_all_cases = all(r["id"] in report_text for r in results)

print("case_count=", summary["case_count"])
```

### 十四、如何判断 DPO 成功

DPO 成功不一定表现为所有回答都更长或更复杂。

更合理的成功标准是：

```text
在偏好覆盖的任务上，DPO 输出更接近 chosen。
在格式要求任务上，DPO 更稳定。
在 rejected 常见错误上，DPO 更少犯错。
在未见相似任务上，DPO 有一定泛化。
基础能力没有明显退化。
```

如果 DPO 只是让模型机械重复某种模板，不一定是成功。

偏好优化要看整体行为。

---

### 十五、DPO 后常见退化

#### 退化 1：回答过短

偏好数据中 chosen 都很短，模型可能学会短答。

#### 退化 2：模板化严重

如果 chosen 风格高度一致，模型可能过度套模板。

#### 退化 3：拒答过度

安全偏好数据过强或比例过高，模型可能对正常问题也拒答。

#### 退化 4：领域外能力下降

偏好数据覆盖太窄，模型可能在其他任务上退化。

#### 退化 5：事实性没有提升

DPO 只是偏好优化，不会凭空增加知识。

如果 chosen 本身事实错误，DPO 还会强化错误。

---

### 十六、改进方向

如果 DPO 效果不好，可以从六个方向改进：

```text
1. 提高 preference 数据质量。
2. 增加更难的 rejected，而不是明显乱码。
3. 扩大评估集，覆盖更多任务。
4. 调整 beta。
5. 降低学习率或减少训练步数。
6. 先增强 SFT，再做 DPO。
```

尤其要注意：

```text
DPO 不能修复低质量偏好数据。
```

数据反标、chosen 错误、rejected 过弱都会直接影响模型。

### 十七、完整对比脚本骨架

```python
import json
from pathlib import Path

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer


def build_prompt(example):
    if example.get("input", ""):
        return (
            "### Instruction:\n"
            f"{example['instruction']}\n\n"
            "### Input:\n"
            f"{example['input']}\n\n"
            "### Response:\n"
        )
    return (
        "### Instruction:\n"
        f"{example['instruction']}\n\n"
        "### Response:\n"
    )


@torch.no_grad()
def generate_text(model, tokenizer, prompt, device):
    inputs = tokenizer(prompt, return_tensors="pt").to(device)
    outputs = model.generate(
        **inputs,
        max_new_tokens=160,
        do_sample=False,
        pad_token_id=tokenizer.eos_token_id,
    )
    input_len = inputs["input_ids"].shape[-1]
    new_tokens = outputs[0, input_len:]
    return tokenizer.decode(new_tokens, skip_special_tokens=True)


def main():
    device = "cuda" if torch.cuda.is_available() else "cpu"

    sft_dir = "outputs/full_sft_tiny_gpt2/final"
    dpo_dir = "outputs/dpo_tiny_gpt2/final"

    sft_tokenizer = AutoTokenizer.from_pretrained(sft_dir)
    if sft_tokenizer.pad_token is None:
        sft_tokenizer.pad_token = sft_tokenizer.eos_token
    sft_model = AutoModelForCausalLM.from_pretrained(sft_dir).to(device).eval()

    dpo_tokenizer = AutoTokenizer.from_pretrained(dpo_dir)
    if dpo_tokenizer.pad_token is None:
        dpo_tokenizer.pad_token = dpo_tokenizer.eos_token
    dpo_model = AutoModelForCausalLM.from_pretrained(dpo_dir).to(device).eval()

    eval_set = [
        {
            "id": "preference_overfit",
            "category": "偏好数据内任务",
            "instruction": "解释什么是过拟合。",
            "input": "",
            "expected": "应该提到训练集表现好、未见数据泛化差。",
        }
    ]

    results = []
    for ex in eval_set:
        prompt = build_prompt(ex)
        sft_output = generate_text(sft_model, sft_tokenizer, prompt, device)
        dpo_output = generate_text(dpo_model, dpo_tokenizer, prompt, device)
        results.append({
            "id": ex["id"],
            "category": ex["category"],
            "prompt": prompt,
            "expected": ex["expected"],
            "sft_output": sft_output,
            "dpo_output": dpo_output,
        })

    out_path = Path("outputs/compare_sft_dpo.jsonl")
    out_path.parent.mkdir(parents=True, exist_ok=True)
    with out_path.open("w", encoding="utf-8") as f:
        for r in results:
            f.write(json.dumps(r, ensure_ascii=False) + "\n")


if __name__ == "__main__":
    main()
```

### 十八、面试怎么讲 SFT vs DPO 对比

如果面试官问“怎么判断 DPO 有没有效果”，可以这样回答：

```text
我会用同一套评估 prompt、同一套生成参数，对比 SFT 模型和 DPO 模型的输出。评估维度包括指令遵循、是否更接近 chosen 偏好、是否减少 rejected 中的问题、格式稳定性、事实正确性、安全性和领域外泛化。除了看 DPO loss 和 reward margin，还要做人工样例对比，因为 loss 下降不一定等价于行为变好。
```

如果追问“DPO 后一定比 SFT 好吗”，可以回答：

```text
不一定。DPO 的效果强依赖偏好数据质量、beta、学习率和训练步数。如果偏好数据有反标、chosen 质量差、rejected 太弱或覆盖太窄，DPO 可能导致模板化、过度拒答、领域外退化或过拟合。
```

如果问“DPO 主要改善什么”，可以回答：

```text
DPO 主要改善模型在偏好数据覆盖范围内的选择倾向，让模型相对 SFT 更偏好 chosen 风格、减少 rejected 中的问题。它更像行为对齐和偏好校准，不是凭空注入新知识。
```

### 本讲总结

这一讲比较了 SFT 和 DPO 后模型输出。

核心结论如下：

1. DPO 是否有效，最终要看模型行为，而不只是 loss。
2. 对比时要固定 prompt 模板、生成参数和评估集。
3. SFT 到 DPO 的主要变化是偏好对齐，而不是基础知识凭空增强。
4. 评估应覆盖指令遵循、偏好对齐、正确性、格式、安全性和泛化。
5. DPO 可能带来过短、模板化、过度拒答和领域外退化等副作用。
6. 高质量 preference 数据是 DPO 成功的关键。
7. 完整偏好优化项目应包含数据、训练、日志、模型对比和失败案例分析。
