## 第 25 讲：构造 Preference Pair 数据

### 本讲目标

学完本讲，你应该能做到六件事：

1. 理解偏好优化为什么需要 preference pair 数据。
2. 掌握 `prompt/chosen/rejected` 的标准格式。
3. 构造可用于 Reward Model 和 DPO 的偏好数据。
4. 从 SFT 数据、模型采样和人工标注中生成偏好对。
5. 使用 Hugging Face Dataset 保存和检查偏好数据。
6. 能排查偏好数据中的常见质量问题。

前面第四部分我们完成了 SFT。

SFT 教模型“应该怎么回答”。

但真实对齐训练还会进一步关心：

```text
两个回答中，哪一个更好？
```

这就是偏好优化的起点。

本讲先构造 preference pair 数据。

后续 Reward Model、DPO、TRL 训练都会依赖这种数据结构。

---

### 一、什么是 Preference Pair

Preference pair 是偏好对。

一条样本通常包含：

```text
prompt：用户问题或指令。
chosen：更好的回答。
rejected：更差的回答。
```

例如：

```json
{
  "prompt": "解释什么是过拟合。",
  "chosen": "过拟合是指模型在训练集上表现很好，但在未见数据上表现较差的现象。常见缓解方法包括正则化、早停和增加数据。",
  "rejected": "过拟合就是模型很厉害，所以训练得特别好。"
}
```

这条数据表达的是：

```text
在同一个 prompt 下，chosen 应该比 rejected 更受偏好。
```

偏好优化不一定要求给出绝对分数。

它只需要相对比较。

---

### 二、Preference Pair 用在哪里

偏好数据可以用于多种训练方式。

#### 1. Reward Model

Reward Model 学习给回答打分。

训练目标是：

```text
score(prompt, chosen) > score(prompt, rejected)
```

#### 2. RLHF

RLHF 中通常先训练 Reward Model，再用 PPO 等强化学习方法优化策略模型。

#### 3. DPO

DPO 不显式训练 Reward Model。

它直接利用 chosen/rejected 对优化语言模型。

#### 4. ORPO、IPO、KTO 等方法

很多偏好优化方法都围绕偏好数据展开。

所以 preference pair 是偏好优化实战的基础数据结构。

---

### 三、偏好数据和 SFT 数据的区别

SFT 数据：

```json
{
  "instruction": "解释什么是过拟合。",
  "input": "",
  "output": "过拟合是指..."
}
```

Preference 数据：

```json
{
  "prompt": "解释什么是过拟合。",
  "chosen": "过拟合是指模型在训练集上表现很好，但在未见数据上表现较差...",
  "rejected": "过拟合就是模型特别厉害。"
}
```

区别：

```text
SFT 只有一个理想答案。
Preference 有一个更好答案和一个更差答案。
SFT 学习模仿。
Preference 学习偏好排序或直接优化偏好。
```

SFT 关注：

```text
应该生成什么。
```

偏好优化关注：

```text
哪个回答更好。
```

---

### 四、标准字段格式

最常用的字段是：

```text
prompt
chosen
rejected
```

建议先统一成这种结构。

```python
preference_data = [
    {
        "prompt": "解释什么是梯度下降。",
        "chosen": "梯度下降是一种通过沿损失函数负梯度方向更新参数来最小化损失的方法。",
        "rejected": "梯度下降就是梯度往下掉。",
    },
    {
        "prompt": "给出三个缓解过拟合的方法。",
        "chosen": "可以使用更多数据、正则化、dropout、早停或减小模型容量。",
        "rejected": "只要把模型做得更大就能缓解过拟合。",
    },
]
```

有些数据集还会包含：

```text
source
chosen_score
rejected_score
labeler_id
category
```

但训练 DPO 或 Reward Model 时，核心字段通常就是 `prompt/chosen/rejected`。

---

### 五、如何构造 chosen 和 rejected

常见来源有四类。

#### 来源 1：人工写 chosen，人工写 rejected

适合小规模高质量数据。

优点：质量可控。

缺点：成本高。

#### 来源 2：一个 prompt 采样多个模型回答，人工选择好坏

这是很常见的偏好标注方式。

流程：

```text
同一个 prompt -> 模型生成多个候选 -> 人工选 preferred 和 dispreferred。
```

#### 来源 3：强模型答案作为 chosen，弱模型答案作为 rejected

例如：

```text
chosen 来自更强模型或人工标准答案。
rejected 来自较弱模型。
```

这种方法成本较低，但要小心强模型也可能错。

#### 来源 4：从 SFT 数据构造负样本

SFT 的 output 可以作为 chosen。

再人为构造一个低质量 rejected。

适合教学实验。

### 六、从 SFT 数据构造偏好对

假设 SFT 数据是：

```python
sft_data = [
    {
        "instruction": "解释什么是过拟合。",
        "input": "",
        "output": "过拟合是指模型在训练集上表现很好，但在未见数据上表现较差的现象。",
    }
]
```

可以构造：

```python
def sft_to_preference(example, rejected_answer):
    prompt = example["instruction"]
    if example.get("input", ""):
        prompt = prompt + "\n" + example["input"]

    return {
        "prompt": prompt,
        "chosen": example["output"],
        "rejected": rejected_answer,
    }


preference_example = sft_to_preference(
    sft_data[0],
    rejected_answer="过拟合就是模型训练得很好，所以一定是好事。",
)
```

教学项目中可以手写 rejected。

真实项目中，最好用模型采样和人工标注构造更自然的 rejected。

### 九、给 prompt 加模板

偏好数据中的 `prompt` 可以是裸问题，也可以是已经格式化后的 prompt。

推荐先保存裸字段：

```text
prompt/chosen/rejected
```

训练前再统一加模板。

Alpaca 风格：

```python
def format_prompt(prompt):
    return (
        "### Instruction:\n"
        f"{prompt}\n\n"
        "### Response:\n"
    )
```

构造完整 chosen/rejected 文本：

```python
def format_preference(example):
    prompt = format_prompt(example["prompt"])
    return {
        "prompt": prompt,
        "chosen": example["chosen"],
        "rejected": example["rejected"],
    }
```

注意：

```text
chosen 和 rejected 应该共享同一个 prompt。
```

否则比较不公平。

### 十一、数据质量检查

偏好数据质量非常重要。

检查项：

```text
prompt 是否清楚。
chosen 是否真的优于 rejected。
chosen 是否回答了 prompt。
rejected 是否不是完全无关文本。
chosen/rejected 长度差异是否过大。
是否存在 chosen 和 rejected 完全相同。
是否有明显错标。
是否有重复样本。
是否有安全和隐私风险。
```

最基本检查：

```python
def validate_preference(example):
    if not example["prompt"].strip():
        return False
    if not example["chosen"].strip():
        return False
    if not example["rejected"].strip():
        return False
    if example["chosen"].strip() == example["rejected"].strip():
        return False
    return True


cleaned = [ex for ex in raw_preferences if validate_preference(ex)]
```

---

### 十二、长度检查

极端长度差异会影响训练。

例如：

```text
chosen 是 500 字完整回答。
rejected 是 2 个字。
```

模型可能学到“越长越好”，而不是学到真实质量差异。

可以统计长度：

```python
def length_stats(data):
    for ex in data:
        print(
            len(ex["prompt"]),
            len(ex["chosen"]),
            len(ex["rejected"]),
        )


length_stats(cleaned)
```

更好的做法是用 tokenizer 后的 token 长度：

```python
def token_length(tokenizer, text):
    return len(tokenizer(text, add_special_tokens=False)["input_ids"])
```

### 十三、去重

```python
def deduplicate_preferences(data):
    seen = set()
    result = []
    for ex in data:
        key = (
            ex["prompt"].strip(),
            ex["chosen"].strip(),
            ex["rejected"].strip(),
        )
        if key not in seen:
            seen.add(key)
            result.append(ex)
    return result


cleaned = deduplicate_preferences(cleaned)
```

重复偏好对会让模型过度拟合某些样本。

如果重复是有意加权，也应该明确记录，而不是无意识重复。

### 十四、train/eval 划分

```python
dataset = Dataset.from_list(cleaned)
split = dataset.train_test_split(test_size=0.1, seed=42)
train_dataset = split["train"]
eval_dataset = split["test"]
```

注意：

```text
同一个 prompt 的多个偏好对，最好不要一部分在 train，一部分在 eval。
```

否则验证集可能泄漏。

更严谨做法是按 prompt 分组划分。

简化版：

```python
from collections import defaultdict
import random


groups = defaultdict(list)
for ex in cleaned:
    groups[ex["prompt"]].append(ex)

prompts = list(groups.keys())
random.Random(42).shuffle(prompts)

cut = int(0.9 * len(prompts))
train_prompts = set(prompts[:cut])

train_data = []
eval_data = []
for prompt, items in groups.items():
    if prompt in train_prompts:
        train_data.extend(items)
    else:
        eval_data.extend(items)
```

### 十五、Reward Model 和 DPO 的格式差异

Reward Model 训练通常需要把 prompt+chosen 和 prompt+rejected 分别编码。

例如：

```text
chosen_text = prompt_template + chosen
rejected_text = prompt_template + rejected
```

DPO 训练库通常直接需要：

```text
prompt
chosen
rejected
```

TRL 的 DPOTrainer 常见输入字段也是这三个。

所以保存原始结构时，建议保留：

```text
prompt/chosen/rejected
```

不要过早只保存拼接后的文本。

这样后续 Reward Model 和 DPO 都能复用。

### 十七、常见工程坑

#### 坑 1：chosen 和 rejected 反了

这是最严重的错误。

模型会被训练成偏好差回答。

#### 坑 2：chosen/rejected 差距只体现在长度

模型可能学到“长回答更好”，而不是“正确回答更好”。

#### 坑 3：rejected 太差

如果 rejected 是明显乱码，任务太简单，模型学不到细粒度偏好。

好的 rejected 应该是：

```text
看起来像回答，但质量确实更差。
```

#### 坑 4：prompt 模板不一致

Reward Model、DPO 和推理时的模板应保持一致。

#### 坑 5：训练评估泄漏

同一个 prompt 的相似 pair 同时出现在 train 和 eval，会让评估不可靠。

#### 坑 6：不做人工抽查

偏好数据非常容易错标。

必须抽查。

### 十八、面试怎么讲 Preference Pair

如果面试官问“偏好优化数据长什么样”，可以这样回答：

```text
偏好优化数据通常是 prompt、chosen、rejected 三元组。prompt 是用户问题，chosen 是更优回答，rejected 是较差回答。训练目标不是学习一个绝对答案，而是学习在同一个 prompt 下 chosen 应该优于 rejected。
```

如果追问“这种数据怎么来”，可以回答：

```text
可以来自人工标注，也可以让模型对同一 prompt 采样多个回答后由人工选择偏好；还可以用强模型答案作为 chosen、弱模型答案作为 rejected。教学场景下也可以从 SFT 数据构造 chosen，并手写或采样 rejected。
```

如果问“偏好数据质量怎么检查”，可以回答：

```text
我会检查 chosen 是否确实优于 rejected，二者是否回答同一个 prompt，是否存在反标、重复、空文本、长度偏差过大、乱码和安全风险；还会按 prompt 分组划分 train/eval，避免评估泄漏。
```

如果问“Reward Model 和 DPO 的数据格式一样吗”，可以回答：

```text
原始数据都可以保存成 prompt/chosen/rejected。Reward Model 训练时通常把 prompt+chosen 和 prompt+rejected 分别编码后做排序损失；DPO 则直接利用 prompt、chosen、rejected 计算偏好优化目标。所以保留三元组结构最灵活。
```

### 本讲总结

这一讲构造了 Preference Pair 数据。

核心结论如下：

1. 偏好数据通常是 `prompt/chosen/rejected` 三元组。
2. chosen 表示更优回答，rejected 表示较差回答。
3. Preference 数据可用于 Reward Model、RLHF、DPO 等方法。
4. 偏好数据和 SFT 数据不同，前者学习相对偏好，后者学习模仿答案。

7. train/eval 划分应避免同 prompt 泄漏。

## 第 26 讲：实现 Reward Model 排序损失

### 本讲目标

学完本讲，你应该能做到六件事：

1. 理解 Reward Model 在偏好优化中的作用。
2. 理解 pairwise ranking loss 的直觉。
3. 掌握 Bradley-Terry 偏好建模公式。
4. 从零实现 Reward Model 排序损失。
5. 能用 PyTorch 验证 chosen reward 应该高于 rejected reward。
6. 能排查 reward loss 中常见的符号、shape 和数值问题。

上一讲我们构造了 preference pair 数据。

每条数据包含：

```text
prompt
chosen
rejected
```

Reward Model 的目标是学一个打分函数：

```text
reward(prompt, response) -> scalar
```

并且希望：

```text
reward(prompt, chosen) > reward(prompt, rejected)
```

本讲先不训练完整 Reward Model，而是专门实现排序损失。

这是一切 Reward Model 训练的核心。

---

### 一、Reward Model 解决什么问题

SFT 模型学会模仿答案。

但它不一定知道哪个答案更好。

Reward Model 要学习人类偏好。

输入：

```text
prompt + response
```

输出：

```text
一个标量 reward
```

标量越大，表示模型认为这个回答越好。

例如：

```text
reward(prompt, chosen) = 2.3
reward(prompt, rejected) = -0.7
```

表示 chosen 比 rejected 更好。

Reward Model 可以用于：

```text
RLHF 中给策略模型提供奖励。
离线评估不同回答质量。
训练 reranker。
为偏好优化方法提供比较基线。
```

---

### 二、为什么用排序损失

偏好数据通常不给绝对分数。

它只告诉你：

```text
chosen 比 rejected 好。
```

所以训练目标不是让 chosen reward 等于某个固定值。

而是让：

```text
r_chosen - r_rejected 尽量大。
```

这就是 pairwise ranking。

我们不关心 reward 的绝对尺度。

更关心相对顺序。

---

### 三、Bradley-Terry 模型直觉

Bradley-Terry 模型常用于成对比较。

设：

```text
r_c = reward(prompt, chosen)
r_r = reward(prompt, rejected)
```

模型认为 chosen 被偏好的概率是：

```text
P(chosen > rejected) = sigmoid(r_c - r_r)
```

如果 `r_c` 远大于 `r_r`：

```text
sigmoid(r_c - r_r) 接近 1
```

如果二者相等：

```text
sigmoid(0) = 0.5
```

如果 `r_c` 小于 `r_r`：

```text
概率小于 0.5
```

训练目标是最大化 chosen 被偏好的概率。

也就是最小化负对数似然：

```text
loss = -log sigmoid(r_c - r_r)
```

---

### 四、排序损失公式

Reward Model 排序损失：

```text
loss = -log σ(r_chosen - r_rejected)
```

其中：

```text
σ 是 sigmoid。
```

如果 `r_chosen - r_rejected` 很大，loss 很小。

如果差值为 0，loss 是：

```text
-log(0.5) ≈ 0.693
```

如果 chosen reward 反而小于 rejected，loss 会变大。

这正好符合我们的目标。

---

### 五、手算几个例子

#### 情况 1：chosen 明显更好

```text
r_chosen = 3
r_rejected = 0
diff = 3
sigmoid(3) ≈ 0.95
loss ≈ 0.05
```

loss 很小。

#### 情况 2：模型无法区分

```text
r_chosen = 1
r_rejected = 1
diff = 0
sigmoid(0) = 0.5
loss = 0.693
```

loss 中等。

#### 情况 3：模型排反了

```text
r_chosen = 0
r_rejected = 3
diff = -3
sigmoid(-3) ≈ 0.047
loss ≈ 3.05
```

loss 很大。

这会推动模型提高 chosen reward，降低 rejected reward。

---

### 六、PyTorch 实现

最直接写法：

```python
import torch
import torch.nn.functional as F


def reward_ranking_loss(chosen_rewards, rejected_rewards):
    diff = chosen_rewards - rejected_rewards
    loss = -torch.log(torch.sigmoid(diff))
    return loss.mean()
```

输入 shape：

```text
chosen_rewards:   [batch]
rejected_rewards: [batch]
```

输出：

```text
标量 loss
```

测试：

```python
chosen = torch.tensor([3.0, 1.0, 0.0])
rejected = torch.tensor([0.0, 1.0, 3.0])

loss = reward_ranking_loss(chosen, rejected)
print(loss)
```

---

### 七、更稳定的实现

直接 `log(sigmoid(x))` 在数值上不如 `logsigmoid(x)` 稳定。

推荐写法：

```python
def reward_ranking_loss(chosen_rewards, rejected_rewards):
    diff = chosen_rewards - rejected_rewards
    loss = -F.logsigmoid(diff)
    return loss.mean()
```

`F.logsigmoid(diff)` 等价于：

```text
log(sigmoid(diff))
```

但数值更稳定。

这是工程里更常用的写法。

---

### 八、验证 loss 方向

写一个简单测试：

```python
good_chosen = torch.tensor([3.0])
bad_rejected = torch.tensor([0.0])

bad_chosen = torch.tensor([0.0])
good_rejected = torch.tensor([3.0])

loss_correct = reward_ranking_loss(good_chosen, bad_rejected)
loss_wrong = reward_ranking_loss(bad_chosen, good_rejected)

print("loss_correct:", loss_correct.item())
print("loss_wrong:", loss_wrong.item())
```

你应该看到：

```text
loss_correct < loss_wrong
```

这个测试很重要。

如果你把符号写反了，模型会学习偏好 rejected。

### 九、带 margin 的排序损失

有时希望 chosen reward 至少比 rejected 高一个 margin。

可以写成：

```text
loss = -log sigmoid(r_chosen - r_rejected - margin)
```

代码：

```python
def reward_ranking_loss_with_margin(chosen_rewards, rejected_rewards, margin=0.0):
    diff = chosen_rewards - rejected_rewards - margin
    loss = -F.logsigmoid(diff)
    return loss.mean()
```

如果 `margin=1.0`，模型需要让：

```text
r_chosen 比 r_rejected 至少高 1
```

loss 才会比较小。

margin 可以增强区分度，但也可能让训练更难。

教学项目中先用 `margin=0` 即可。

### 十、Reward Model 输出 shape

Reward Model 通常输入一段 token 序列，输出一个标量。

例如：

```text
chosen_input_ids:   [B, T]
rejected_input_ids: [B, T]
```

模型输出：

```text
chosen_rewards:   [B]
rejected_rewards: [B]
```

然后计算：

```python
loss = reward_ranking_loss(chosen_rewards, rejected_rewards)
```

如果输出 shape 是 `[B, 1]`，可以压缩：

```python
chosen_rewards = chosen_rewards.squeeze(-1)
rejected_rewards = rejected_rewards.squeeze(-1)
```

不要让 shape 变成奇怪的 `[B, B]` broadcast。

---

### 十一、一个最小 Reward Head

实际 Reward Model 通常是在 transformer backbone 后接一个 value head。

简化结构：

```python
import torch.nn as nn


class RewardHead(nn.Module):
    def __init__(self, hidden_size):
        super().__init__()
        self.value = nn.Linear(hidden_size, 1)

    def forward(self, hidden_states, attention_mask):
        last_token_indices = attention_mask.sum(dim=1) - 1
        batch_indices = torch.arange(hidden_states.size(0), device=hidden_states.device)
        last_hidden = hidden_states[batch_indices, last_token_indices]
        reward = self.value(last_hidden).squeeze(-1)
        return reward
```

这里的思想是：

```text
取最后一个有效 token 的 hidden state，映射成标量 reward。
```

为什么取最后一个 token？

因为 causal LM 的最后一个 token 表示包含前面 prompt 和 response 的上下文信息。

---

### 十二、padding 下为什么要用 attention_mask

如果 batch 中有 padding：

```text
sample A: token token token token
sample B: token token pad   pad
```

不能简单取最后一列 hidden state。

因为 sample B 的最后一列是 padding。

所以需要：

```python
last_token_indices = attention_mask.sum(dim=1) - 1
```

它得到每个样本最后一个有效 token 的位置。

这是 Reward Model 中非常常见的细节。

### 十三、完整前向逻辑示意

```python
def compute_reward_loss(
    reward_model,
    chosen_input_ids,
    chosen_attention_mask,
    rejected_input_ids,
    rejected_attention_mask,
):
    chosen_rewards = reward_model(
        input_ids=chosen_input_ids,
        attention_mask=chosen_attention_mask,
    )
    rejected_rewards = reward_model(
        input_ids=rejected_input_ids,
        attention_mask=rejected_attention_mask,
    )

    loss = reward_ranking_loss(chosen_rewards, rejected_rewards)
    return loss, chosen_rewards, rejected_rewards
```

训练时，每个 batch 会同时包含 chosen 和 rejected 两套输入。

下一讲我们会把它扩展成完整 Reward Model 训练。

### 十四、reward accuracy

除了 loss，还可以计算排序准确率。

```python
def reward_accuracy(chosen_rewards, rejected_rewards):
    return (chosen_rewards > rejected_rewards).float().mean()
```

如果 accuracy 接近 1，说明模型大多数时候给 chosen 更高分。

如果接近 0.5，说明接近随机。

如果低于 0.5，可能有问题：

```text
数据反标。
loss 符号写反。
模型没训练好。
```

### 十六、常见工程坑

#### 坑 1：损失符号写反

错误写法：

```python
-F.logsigmoid(rejected_rewards - chosen_rewards)
```

这会鼓励 rejected reward 更高。

#### 坑 2：chosen/rejected 数据反了

即使 loss 写对，数据反标也会毁掉训练。

#### 坑 3：shape broadcast 错误

如果 chosen 是 `[B, 1]`，rejected 是 `[B]`，相减可能 broadcast 成 `[B, B]`。

建议统一 squeeze 成 `[B]`。

#### 坑 4：取了 padding token 的 hidden state

Reward Head 应使用 attention_mask 找到最后一个有效 token。

#### 坑 5：只看 loss 不看 accuracy

Reward Model 训练时应同时看 pairwise accuracy。

#### 坑 6：reward 绝对值越来越大

排序损失主要关心差值，reward 绝对尺度可能漂移。

必要时可以做正则化或监控 reward 分布。

### 十七、面试怎么讲 Reward Model Loss

如果面试官问“Reward Model 怎么训练”，可以这样回答：

```text
Reward Model 输入 prompt 和 response，输出一个标量 reward。训练数据是 prompt、chosen、rejected 三元组。我们分别计算 chosen 和 rejected 的 reward，然后用 pairwise ranking loss：-log sigmoid(r_chosen - r_rejected)，鼓励 chosen reward 高于 rejected reward。
```

如果追问“为什么不是回归一个分数”，可以回答：

```text
偏好数据通常只提供相对偏好，而不是绝对分数。人类更容易判断两个回答哪个更好，而不是给稳定的绝对分。因此 Reward Model 常用成对排序损失来学习相对偏好。
```

如果问“怎么判断 Reward Model 训练是否正常”，可以回答：

```text
除了看 ranking loss 是否下降，还要看 chosen reward 是否平均高于 rejected reward，以及 pairwise accuracy 是否上升。如果 accuracy 长期低于 0.5，要检查 chosen/rejected 是否反了、loss 符号是否写反、shape 是否 broadcast 错误。
```

### 本讲总结

这一讲实现了 Reward Model 排序损失。

核心结论如下：

1. Reward Model 输入 prompt+response，输出标量 reward。
2. 偏好数据只提供相对关系：chosen 优于 rejected。
3. Bradley-Terry 模型使用 `sigmoid(r_chosen - r_rejected)` 表示 chosen 被偏好的概率。
4. 常用排序损失是 `-logsigmoid(r_chosen - r_rejected)`。
5. 工程中要用 `F.logsigmoid` 提高数值稳定性。
6. Reward Head 常取最后一个有效 token 的 hidden state 输出 reward。
7. 训练时要同时监控 ranking loss 和 pairwise accuracy。


### 五、Reward Data Collator

Reward Model batch 中有两套输入：chosen 和 rejected。

```python
import torch


class RewardDataCollator:
    def __init__(self, tokenizer):
        self.tokenizer = tokenizer

    def _pad(self, features, key, pad_value):
        tensors = [torch.tensor(f[key], dtype=torch.long) for f in features]
        return torch.nn.utils.rnn.pad_sequence(
            tensors,
            batch_first=True,
            padding_value=pad_value,
        )

    def __call__(self, features):
        return {
            "chosen_input_ids": self._pad(features, "chosen_input_ids", self.tokenizer.pad_token_id),
            "chosen_attention_mask": self._pad(features, "chosen_attention_mask", 0),
            "rejected_input_ids": self._pad(features, "rejected_input_ids", self.tokenizer.pad_token_id),
            "rejected_attention_mask": self._pad(features, "rejected_attention_mask", 0),
        }
```

### 六、构造 Reward Model

Reward Model 可以用 causal LM backbone 加一个 value head。

```python
import torch.nn as nn
from transformers import AutoModel


class RewardModel(nn.Module):
    def __init__(self, model_name):
        super().__init__()
        self.backbone = AutoModel.from_pretrained(model_name)
        hidden_size = self.backbone.config.hidden_size
        self.value_head = nn.Linear(hidden_size, 1)

    def forward(self, input_ids, attention_mask):
        outputs = self.backbone(
            input_ids=input_ids,
            attention_mask=attention_mask,
        )
        hidden_states = outputs.last_hidden_state

        last_token_indices = attention_mask.sum(dim=1) - 1
        batch_indices = torch.arange(hidden_states.size(0), device=hidden_states.device)
        last_hidden = hidden_states[batch_indices, last_token_indices]

        rewards = self.value_head(last_hidden).squeeze(-1)
        return rewards
```

这里用 `AutoModel`，不是 `AutoModelForCausalLM`。

因为 Reward Model 不需要 lm head。

它只需要 hidden states，然后接一个标量 value head。

```python
@torch.no_grad()
def evaluate(model, dataloader, device):
    model.eval()
    total_loss = 0.0
    total_acc = 0.0
    total_count = 0

    for batch in dataloader:
        batch = {k: v.to(device) for k, v in batch.items()}

        chosen_rewards = model(
            batch["chosen_input_ids"],
            batch["chosen_attention_mask"],
        )
        rejected_rewards = model(
            batch["rejected_input_ids"],
            batch["rejected_attention_mask"],
        )

        loss = reward_ranking_loss(chosen_rewards, rejected_rewards)
        acc = reward_accuracy(chosen_rewards, rejected_rewards)

        batch_size = chosen_rewards.size(0)
        total_loss += loss.item() * batch_size
        total_acc += acc.item() * batch_size
        total_count += batch_size

    model.train()
    return {
        "loss": total_loss / total_count,
        "accuracy": total_acc / total_count,
    }
```

### 十三、Reward 漂移问题

排序损失只关心差值：

```text
r_chosen - r_rejected
```

它不直接约束 reward 的绝对值。

所以 reward 可能整体越来越大或越来越小。

这叫 reward scale drift 或 reward 漂移。

教学项目中先监控即可。

真实系统中可能会加入：

```text
reward normalization
regularization
reward clipping
更严格的数据平衡
```

不要把 Reward Model 输出当作绝对真理。

它只是一个学出来的偏好代理。

---

### 十四、常见工程坑

#### 坑 1：用 AutoModelForCausalLM 的 logits 当 reward

Reward Model 应输出标量 reward，不是词表 logits。

需要 value head。

#### 坑 2：取 padding 位置 hidden state

要用 attention_mask 找最后有效 token。

#### 坑 3：chosen/rejected 拼接 prompt 不一致

比较必须基于同一个 prompt。

#### 坑 4：loss 符号写反

检查 `chosen_rewards - rejected_rewards` 的方向。

#### 坑 5：只在训练集上评估

Reward Model 很容易记住偏好数据。

必须看 eval accuracy。

#### 坑 6：数据太简单

如果 rejected 都是乱码，Reward Model 学到的偏好很粗糙。

---

### 十五、面试怎么讲 Reward Model 训练

如果面试官问“Reward Model 怎么训练”，可以这样回答：

```text
我会用 prompt/chosen/rejected 偏好数据，把 prompt+chosen 和 prompt+rejected 分别输入同一个 Reward Model。Reward Model 通常是 transformer backbone 加一个 value head，从最后一个有效 token 的 hidden state 输出标量 reward。训练损失是 -logsigmoid(r_chosen - r_rejected)，同时监控 pairwise accuracy。
```

如果追问“Reward Model 输出什么”，可以回答：

```text
输出一个标量 reward，表示模型认为该 prompt-response 的质量。训练时不要求绝对分数准确，而是要求 chosen 的 reward 高于 rejected。
```

如果问“训练 Reward Model 最容易出什么问题”，可以回答：

```text
常见问题包括 chosen/rejected 反标、loss 符号写反、取到 padding token 的 hidden state、reward 绝对尺度漂移、训练集过拟合以及偏好数据质量差。实际训练要同时看 loss、accuracy、reward gap 和人工样例检查。
```

## 第 28 讲：手写 DPO Loss

### 本讲目标

学完本讲，你应该能做到六件事：

1. 理解 DPO 为什么可以绕过显式 Reward Model。
2. 掌握 DPO 中 policy model 和 reference model 的作用。
3. 从 logits 和 labels 计算 response log probability。
4. 手写 DPO loss。
5. 理解 `beta` 控制偏好优化强度的作用。
6. 能排查 DPO 实现中的 logprob、mask、符号和 shape 问题。

前两讲我们构造了偏好数据，并训练了一个简单 Reward Model。

Reward Model 路线是：

```text
偏好数据 -> Reward Model -> RLHF/PPO 优化策略模型
```

DPO 是另一条更直接的路线。

它不显式训练 Reward Model。

而是直接用 chosen/rejected 偏好对优化语言模型。

本讲先手写 DPO loss，下一讲再使用 TRL 训练。

---

### 一、DPO 解决什么问题

传统 RLHF 流程比较复杂：

```text
1. SFT 训练初始策略模型。
2. 训练 Reward Model。
3. 用 PPO 等强化学习算法优化策略模型。
```

这个流程有几个工程难点：

```text
Reward Model 可能不稳定。
PPO 训练复杂，超参数敏感。
训练成本高。
容易 reward hacking。
```

DPO 的目标是简化偏好优化：

```text
直接用 preference pair 训练模型，让模型提高 chosen 概率、降低 rejected 概率，同时不要偏离 reference model 太远。
```

它把偏好优化转成一个监督式 loss。

---

### 二、DPO 需要哪些模型

DPO 中有两个模型角色。

#### 1. Policy Model

正在训练的模型。

记作：

```text
πθ
```

它的参数会更新。

#### 2. Reference Model

参考模型。

记作：

```text
πref
```

通常是 SFT 后但未做 DPO 的模型。

它被冻结，不更新。

DPO 要让 policy 更偏好 chosen，同时不要无限偏离 reference。

---

### 三、DPO 输入数据

DPO 使用上一讲的偏好数据：

```text
prompt
chosen
rejected
```

对每条样本，我们需要计算四个 log probability：

```text
policy_chosen_logp
policy_rejected_logp
ref_chosen_logp
ref_rejected_logp
```

其中：

```text
policy_chosen_logp：policy 对 chosen response 的 log probability。
policy_rejected_logp：policy 对 rejected response 的 log probability。
ref_chosen_logp：reference 对 chosen response 的 log probability。
ref_rejected_logp：reference 对 rejected response 的 log probability。
```

DPO 比较的是：

```text
policy 相对 reference 是否更偏向 chosen。
```

---

### 四、DPO Loss 公式

定义 policy log odds：

```text
π_logratio = logπθ(chosen|prompt) - logπθ(rejected|prompt)
```

定义 reference log odds：

```text
ref_logratio = logπref(chosen|prompt) - logπref(rejected|prompt)
```

DPO logits：

```text
logits = π_logratio - ref_logratio
```

DPO loss：

```text
loss = -log sigmoid(beta * logits)
```

也就是：

```text
loss = -log σ(β * [(logπθ_c - logπθ_r) - (logπref_c - logπref_r)])
```

其中 `beta` 控制偏好优化强度。

---

### 五、公式直觉

如果 policy 相比 reference 更偏向 chosen：

```text
π_logratio > ref_logratio
```

则：

```text
logits > 0
loss 较小
```

如果 policy 没有比 reference 更偏向 chosen：

```text
logits ≈ 0
loss ≈ 0.693
```

如果 policy 反而更偏向 rejected：

```text
logits < 0
loss 较大
```

所以 DPO 会推动 policy 相对 reference 提高 chosen 的概率优势。

---

### 六、为什么需要 reference model

如果只优化：

```text
logπθ(chosen) - logπθ(rejected)
```

模型可能过度偏移。

reference model 提供一个锚点。

DPO 实际优化的是：

```text
policy 相对于 reference 的偏好变化。
```

直观理解：

```text
不是让 policy 无限制追 chosen，而是在不远离 SFT 模型太多的前提下，让它更偏好 chosen。
```

这和 RLHF 中 KL penalty 的作用类似。

---

### 七、手写 DPO Loss

假设四个 logp 都已经算好，shape 都是 `[B]`。

```python
import torch
import torch.nn.functional as F


def dpo_loss(
    policy_chosen_logps,
    policy_rejected_logps,
    ref_chosen_logps,
    ref_rejected_logps,
    beta=0.1,
):
    policy_logratios = policy_chosen_logps - policy_rejected_logps
    ref_logratios = ref_chosen_logps - ref_rejected_logps
    logits = policy_logratios - ref_logratios
    losses = -F.logsigmoid(beta * logits)
    return losses.mean()
```

测试：

```python
policy_chosen = torch.tensor([-10.0, -8.0])
policy_rejected = torch.tensor([-12.0, -9.0])
ref_chosen = torch.tensor([-10.5, -8.5])
ref_rejected = torch.tensor([-11.5, -8.8])

loss = dpo_loss(policy_chosen, policy_rejected, ref_chosen, ref_rejected)
print(loss)
```

---

### 八、DPO reward 形式

很多实现会同时计算隐式 reward：

```text
chosen_reward = beta * (policy_chosen_logp - ref_chosen_logp)
rejected_reward = beta * (policy_rejected_logp - ref_rejected_logp)
```

然后：

```text
logits = chosen_reward - rejected_reward
```

代码：

```python
def dpo_loss_with_rewards(
    policy_chosen_logps,
    policy_rejected_logps,
    ref_chosen_logps,
    ref_rejected_logps,
    beta=0.1,
):
    chosen_rewards = beta * (policy_chosen_logps - ref_chosen_logps)
    rejected_rewards = beta * (policy_rejected_logps - ref_rejected_logps)
    logits = chosen_rewards - rejected_rewards
    losses = -F.logsigmoid(logits)
    return losses.mean(), chosen_rewards, rejected_rewards
```

这和前面的公式等价。

这种写法更容易监控：

```text
chosen reward 是否高于 rejected reward。
```

---

### 九、如何计算 response log probability

语言模型输出：

```text
logits: [B, T, V]
labels: [B, T]
```

我们要计算 labels 中 response token 的 log probability 之和。

首先计算 log softmax：

```python
log_probs = F.log_softmax(logits, dim=-1)
```

然后取出 labels 对应 token 的 logprob：

```python
token_log_probs = torch.gather(
    log_probs,
    dim=-1,
    index=labels.unsqueeze(-1),
).squeeze(-1)
```

但是 labels 中可能有 `-100`，不能直接 gather。

需要先 mask。

---

### 十、实现 masked sequence logprob

```python
def sequence_log_probs(logits, labels):
    # logits: [B, T, V]
    # labels: [B, T], response token 是真实 id，prompt/pad 是 -100
    labels = labels.clone()
    loss_mask = labels != -100
    labels[~loss_mask] = 0

    log_probs = F.log_softmax(logits, dim=-1)
    token_log_probs = torch.gather(
        log_probs,
        dim=-1,
        index=labels.unsqueeze(-1),
    ).squeeze(-1)

    token_log_probs = token_log_probs * loss_mask
    seq_log_probs = token_log_probs.sum(dim=-1)
    return seq_log_probs
```

输出 shape：

```text
[B]
```

注意：

```text
这里用 sum，不是 mean。
```

很多 DPO 实现使用 response token logprob 之和。

如果使用 mean，会改变长度偏好，需要非常清楚自己在做什么。

---

### 十一、Causal LM 的 shift 问题

严格来说，语言模型第 `t` 个位置的 logits 用来预测第 `t+1` 个 token。

所以计算 labels logprob 时通常要 shift：

```python
shift_logits = logits[:, :-1, :]
shift_labels = labels[:, 1:]
```

更完整写法：

```python
def sequence_log_probs(logits, labels):
    shift_logits = logits[:, :-1, :].contiguous()
    shift_labels = labels[:, 1:].contiguous()

    loss_mask = shift_labels != -100
    safe_labels = shift_labels.clone()
    safe_labels[~loss_mask] = 0

    log_probs = F.log_softmax(shift_logits, dim=-1)
    token_log_probs = torch.gather(
        log_probs,
        dim=-1,
        index=safe_labels.unsqueeze(-1),
    ).squeeze(-1)

    token_log_probs = token_log_probs * loss_mask
    return token_log_probs.sum(dim=-1)
```

这和 causal LM 的 next-token prediction 对齐。

工程中不要忽略 shift。

---

### 十二、构造 DPO labels

DPO 需要对 chosen 和 rejected 分别构造 labels。

labels 的规则和 SFT 类似：

```text
prompt 部分：-100
response 部分：真实 token id
padding 部分：-100
```

例如：

```python
def tokenize_dpo_example(example, tokenizer, max_length=512):
    prompt = format_prompt(example["prompt"])

    chosen_text = prompt + example["chosen"] + tokenizer.eos_token
    rejected_text = prompt + example["rejected"] + tokenizer.eos_token

    prompt_ids = tokenizer(prompt, add_special_tokens=False)["input_ids"]

    chosen = tokenizer(chosen_text, max_length=max_length, truncation=True, add_special_tokens=False)
    rejected = tokenizer(rejected_text, max_length=max_length, truncation=True, add_special_tokens=False)

    chosen_labels = chosen["input_ids"].copy()
    rejected_labels = rejected["input_ids"].copy()

    prompt_len = len(prompt_ids)
    chosen_labels[:min(prompt_len, len(chosen_labels))] = [-100] * min(prompt_len, len(chosen_labels))
    rejected_labels[:min(prompt_len, len(rejected_labels))] = [-100] * min(prompt_len, len(rejected_labels))

    return {
        "chosen_input_ids": chosen["input_ids"],
        "chosen_attention_mask": chosen["attention_mask"],
        "chosen_labels": chosen_labels,
        "rejected_input_ids": rejected["input_ids"],
        "rejected_attention_mask": rejected["attention_mask"],
        "rejected_labels": rejected_labels,
    }
```

### 十三、计算四个 logprob

```python
@torch.no_grad()
def compute_ref_logps(ref_model, batch):
    ref_chosen_outputs = ref_model(
        input_ids=batch["chosen_input_ids"],
        attention_mask=batch["chosen_attention_mask"],
    )
    ref_rejected_outputs = ref_model(
        input_ids=batch["rejected_input_ids"],
        attention_mask=batch["rejected_attention_mask"],
    )

    ref_chosen_logps = sequence_log_probs(
        ref_chosen_outputs.logits,
        batch["chosen_labels"],
    )
    ref_rejected_logps = sequence_log_probs(
        ref_rejected_outputs.logits,
        batch["rejected_labels"],
    )
    return ref_chosen_logps, ref_rejected_logps
```

policy logps 需要梯度：

```python
def compute_policy_logps(policy_model, batch):
    chosen_outputs = policy_model(
        input_ids=batch["chosen_input_ids"],
        attention_mask=batch["chosen_attention_mask"],
    )
    rejected_outputs = policy_model(
        input_ids=batch["rejected_input_ids"],
        attention_mask=batch["rejected_attention_mask"],
    )

    chosen_logps = sequence_log_probs(chosen_outputs.logits, batch["chosen_labels"])
    rejected_logps = sequence_log_probs(rejected_outputs.logits, batch["rejected_labels"])
    return chosen_logps, rejected_logps
```

Reference model 要冻结。

Policy model 要训练。

### 十四、DPO forward 一步

```python
def dpo_step(policy_model, ref_model, batch, beta=0.1):
    policy_chosen_logps, policy_rejected_logps = compute_policy_logps(
        policy_model,
        batch,
    )

    with torch.no_grad():
        ref_chosen_logps, ref_rejected_logps = compute_ref_logps(
            ref_model,
            batch,
        )

    loss, chosen_rewards, rejected_rewards = dpo_loss_with_rewards(
        policy_chosen_logps,
        policy_rejected_logps,
        ref_chosen_logps,
        ref_rejected_logps,
        beta=beta,
    )

    accuracy = (chosen_rewards > rejected_rewards).float().mean()

    return {
        "loss": loss,
        "chosen_rewards": chosen_rewards,
        "rejected_rewards": rejected_rewards,
        "accuracy": accuracy,
    }
```

训练循环中对 `loss.backward()` 即可。

### 十五、beta 的作用

`beta` 控制偏好优化强度。

直观理解：

```text
beta 越大，模型越强烈地区分 chosen/rejected。
beta 越小，更新更温和，更接近 reference。
```

常见起点：

```text
0.1
0.2
0.5
```

如果 beta 太大：

```text
训练可能不稳定。
模型可能过度偏离 reference。
```

如果 beta 太小：

```text
偏好优化信号太弱。
模型变化不明显。
```

实际要看验证集行为和生成样例调参。

### 十七、常见工程坑

#### 坑 1：忘记 causal shift

logits 第 t 位预测 token t+1。

计算 logprob 时要 shift。

#### 坑 2：prompt token 参与 logprob

DPO 通常只比较 response 部分 logprob。

prompt 部分 labels 应为 `-100`。

#### 坑 3：reference model 没冻结

ref model 应该 `eval()`，并且不参与梯度。

#### 坑 4：chosen/rejected 符号写反

检查公式是否是 chosen 减 rejected。

#### 坑 5：sum 和 mean 混用

sequence logprob 通常用 response token logprob 之和。

如果改成 mean，要明确知道长度偏好会改变。

#### 坑 6：padding labels 没设为 -100

padding 参与 logprob 会污染 loss。

#### 坑 7：policy 和 reference 初始不一致

DPO 通常从 SFT model 初始化 policy，并把同一个 SFT model 复制为 reference。

如果二者初始差太大，训练解释会变复杂。

### 十八、面试怎么讲 DPO Loss

如果面试官问“DPO 是什么”，可以这样回答：

```text
DPO 是一种直接偏好优化方法，它不显式训练 Reward Model，而是用 prompt、chosen、rejected 偏好对直接优化语言模型。它比较 policy model 和 reference model 在 chosen/rejected 上的 log probability 差异，鼓励 policy 相对 reference 更偏好 chosen。
```

如果追问“DPO loss 怎么写”，可以回答：

```text
先计算 policy_logratio = logπθ(chosen|prompt) - logπθ(rejected|prompt)，再计算 ref_logratio = logπref(chosen|prompt) - logπref(rejected|prompt)。DPO loss 是 -logsigmoid(beta * (policy_logratio - ref_logratio))。
```

如果问“reference model 的作用是什么”，可以回答：

```text
reference model 是一个冻结的基准，通常是 SFT 模型。它起到约束作用，让 policy 不要为了偏好数据无限偏移。DPO 实际优化的是 policy 相对于 reference 的偏好变化，类似 RLHF 中 KL 约束的作用。
```

如果问“实现 DPO 最容易错哪里”，可以回答：

```text
常见错误包括 response logprob 没有做 causal shift、prompt 或 padding token 参与 logprob、chosen/rejected 顺序写反、reference model 没冻结、以及 sequence logprob 的 sum/mean 处理不一致。
```

### 十四、常见工程坑

#### 坑 1：TRL 版本参数不兼容

`DPOTrainer` 参数名变化较快。

遇到报错先查版本和文档。

#### 坑 2：数据字段不对

DPO 数据必须有：

```text
prompt
chosen
rejected
```

字段名错了会直接报错。

#### 坑 3：chosen/rejected 包含了重复 prompt

如果 prompt 已单独给出，chosen/rejected 通常只放回答。

否则可能重复拼接。

#### 坑 4：pad_token 没设置

decoder-only 模型常需要：

```python
tokenizer.pad_token = tokenizer.eos_token
```

#### 坑 5：beta 太大

可能导致训练不稳定或模型过度偏移。

#### 坑 6：没有用 SFT 模型作为起点

DPO 通常接在 SFT 后面。

如果直接从 base model 做 DPO，效果可能差且不稳定。

#### 坑 7：只看 DPO loss 不看生成

最终目标是行为改善，不是 loss 数字漂亮。

### 十五、面试怎么讲 TRL DPO

如果面试官问“怎么用 TRL 做 DPO”，可以这样回答：

```text
我会准备 prompt、chosen、rejected 三字段偏好数据，加载 SFT 后的 policy model 和同初始化的 frozen reference model，然后用 TRL 的 DPOTrainer 或 DPOConfig 配置 beta、batch size、学习率和长度限制。训练过程中监控 DPO loss、chosen/rejected reward、reward margin 和 preference accuracy，最后保存模型并用固定 prompt 对比 SFT 与 DPO 后输出。
```

如果追问“DPO 数据里 chosen/rejected 应该包含 prompt 吗”，可以回答：

```text
通常数据字段中 prompt 单独保存，chosen 和 rejected 只保存回答内容。训练时 DPOTrainer 会基于 prompt 和 response 计算 response logprob。如果提前把 prompt 拼进 chosen/rejected，要确认不会重复拼接。
```

如果问“DPO 为什么通常接在 SFT 后”，可以回答：

```text
DPO 是偏好优化，不是从零教模型回答问题。SFT 先让模型具备基本指令遵循和回答格式，DPO 再根据 chosen/rejected 偏好进一步调整行为。如果没有 SFT 起点，DPO 信号可能不稳定，模型基础能力也不足。
```

### 本讲总结

这一讲使用 TRL 进行了 DPO 训练。

核心结论如下：

1. TRL 的 DPOTrainer 封装了 DPO logprob 计算和 loss。
2. DPO 数据通常包含 `prompt/chosen/rejected` 三个字段。
3. DPO 通常以 SFT model 作为 policy 起点，并使用 frozen reference model。
4. `beta` 控制偏好优化强度。
5. DPO 可以和 LoRA、QLoRA 结合，降低训练成本。
6. 训练时应监控 reward margin、accuracy 和 loss。
7. DPO 是否有效最终要通过生成行为评估确认。

## 第 30 讲：比较 SFT、DPO 后模型输出

### 本讲目标

学完本讲，你应该能做到六件事：

1. 构造 SFT 与 DPO 模型的对比评估集。
2. 使用同一套 prompt 和生成参数比较两个模型输出。
3. 从指令遵循、偏好对齐、格式稳定性和安全性分析差异。
4. 保存对比结果为 JSONL 或 Markdown 报告。
5. 理解 DPO 可能带来的改进和副作用。
6. 能在面试中讲清 SFT、RM、DPO 的完整偏好优化闭环。

前面几讲我们已经完成：

```text
Preference Pair 数据构造
Reward Model 排序损失
Reward Model 训练
手写 DPO Loss
TRL DPO 训练
```

现在要做最后一步：评估 DPO 后模型是否比 SFT 模型更符合偏好。

注意：

```text
DPO loss 下降不等于模型一定变好。
```

最终还是要看模型输出行为。

---

### 一、为什么要比较 SFT 和 DPO

SFT 的目标是模仿高质量答案。

DPO 的目标是利用 chosen/rejected 偏好对进一步调整模型偏好。

理想情况下，DPO 后模型应该：

```text
更偏向 chosen 风格。
更少生成 rejected 中的问题模式。
更符合人类偏好。
格式更稳定。
拒答或安全边界更合理。
```

但 DPO 也可能带来副作用：

```text
过度保守。
回答变短或模板化。
对某些任务退化。
过拟合偏好数据。
生成多样性下降。
```

所以必须做对比评估。

---

### 二、准备三个模型层次

完整对比最好包含：

```text
Base Model：原始基座模型。
SFT Model：监督微调后模型。
DPO Model：偏好优化后模型。
```

如果资源有限，至少比较：

```text
SFT Model vs DPO Model
```

原因是 DPO 通常接在 SFT 后面。

Base 到 SFT 主要看指令能力变化。

SFT 到 DPO 主要看偏好对齐变化。

---

### 五、统一 prompt 模板

```python
def build_prompt(example):
```

### 六、统一生成函数

```python
import torch


@torch.no_grad()
def generate_text(model, tokenizer, prompt, device="cpu"):
    model.to(device)
    model.eval()

    inputs = tokenizer(prompt, return_tensors="pt").to(device)
    outputs = model.generate(
        **inputs,
        max_new_tokens=160,
        do_sample=False,
        pad_token_id=tokenizer.eos_token_id,
    )
    text = tokenizer.decode(outputs[0], skip_special_tokens=True)
    return text
```

对比评估时建议先用：

```text
do_sample=False
```

这样结果确定，便于复现。

如果要评估多样性，再单独使用采样参数多跑几次。

### 八、评估维度

#### 2. 偏好对齐

#### 3. 内容正确性

是否减少编造、危险建议或不当内容。

#### 7. 泛化能力

是否只在训练偏好样本附近变好，还是相似新问题也变好。

### 十、自动关键词检查

可以加入简单关键词指标。

```python
keyword_rules = {
    "preference_overfit": ["训练集", "泛化", "未见数据"],
    "format_list": ["正则化", "dropout", "早停"],
    "ood_question": ["梯度", "裁剪", "泛化"],
}


def keyword_score(text, keywords):
    text_lower = text.lower()
    hits = 0
    for kw in keywords:
        if kw.lower() in text_lower:
            hits += 1
    return hits / max(len(keywords), 1)
```

应用：

```python
for r in results:
    keywords = keyword_rules.get(r["id"], [])
    r["sft_keyword_score"] = keyword_score(r["sft_output"], keywords)
    r["dpo_keyword_score"] = keyword_score(r["dpo_output"], keywords)
```

注意：

```text
关键词指标只能辅助，不能替代人工判断。
```

### 十三、如何判断 DPO 成功

DPO 成功不一定表现为所有回答都更长或更复杂。

更合理的成功标准是：

```text
在偏好覆盖的任务上，DPO 输出更接近 chosen。
在格式要求任务上，DPO 更稳定。
在 rejected 常见错误上，DPO 更少犯错。
在未见相似任务上，DPO 有一定泛化。
基础能力没有明显退化。
```

如果 DPO 只是让模型机械重复某种模板，不一定是成功。

偏好优化要看整体行为。

### 十四、DPO 后常见退化

#### 退化 1：回答过短

偏好数据中 chosen 都很短，模型可能学会短答。

#### 退化 2：模板化严重

如果 chosen 风格高度一致，模型可能过度套模板。

#### 退化 3：拒答过度

安全偏好数据过强或比例过高，模型可能对正常问题也拒答。

#### 退化 4：领域外能力下降

偏好数据覆盖太窄，模型可能在其他任务上退化。

#### 退化 5：事实性没有提升

DPO 只是偏好优化，不会凭空增加知识。

如果 chosen 本身事实错误，DPO 还会强化错误。

---

### 十五、改进方向

如果 DPO 效果不好，可以从六个方向改进：

```text
1. 提高 preference 数据质量。
2. 增加更难的 rejected，而不是明显乱码。
3. 扩大评估集，覆盖更多任务。
4. 调整 beta。
5. 降低学习率或减少训练步数。
6. 先增强 SFT，再做 DPO。
```

尤其要注意：

```text
DPO 不能修复低质量偏好数据。
```

数据反标、chosen 错误、rejected 过弱都会直接影响模型。

### 十六、完整对比脚本骨架

```python
import json
from pathlib import Path

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer


def build_prompt(example):
    if example.get("input", ""):
        return (
            "### Instruction:\n"
            f"{example['instruction']}\n\n"
            "### Input:\n"
            f"{example['input']}\n\n"
            "### Response:\n"
        )
    return (
        "### Instruction:\n"
        f"{example['instruction']}\n\n"
        "### Response:\n"
    )


@torch.no_grad()
def generate_text(model, tokenizer, prompt, device):
    inputs = tokenizer(prompt, return_tensors="pt").to(device)
    outputs = model.generate(
        **inputs,
        max_new_tokens=160,
        do_sample=False,
        pad_token_id=tokenizer.eos_token_id,
    )
    return tokenizer.decode(outputs[0], skip_special_tokens=True)


def main():
    device = "cuda" if torch.cuda.is_available() else "cpu"

    sft_dir = "outputs/full_sft_tiny_gpt2/final"
    dpo_dir = "outputs/dpo_tiny_gpt2/final"

    sft_tokenizer = AutoTokenizer.from_pretrained(sft_dir)
    sft_model = AutoModelForCausalLM.from_pretrained(sft_dir).to(device).eval()

    dpo_tokenizer = AutoTokenizer.from_pretrained(dpo_dir)
    dpo_model = AutoModelForCausalLM.from_pretrained(dpo_dir).to(device).eval()

    eval_set = [
        {
            "id": "preference_overfit",
            "category": "偏好数据内任务",
            "instruction": "解释什么是过拟合。",
            "input": "",
            "expected": "应该提到训练集表现好、未见数据泛化差。",
        }
    ]

    results = []
    for ex in eval_set:
        prompt = build_prompt(ex)
        sft_output = generate_text(sft_model, sft_tokenizer, prompt, device)
        dpo_output = generate_text(dpo_model, dpo_tokenizer, prompt, device)
        results.append({
            "id": ex["id"],
            "category": ex["category"],
            "prompt": prompt,
            "expected": ex["expected"],
            "sft_output": sft_output,
            "dpo_output": dpo_output,
        })

    out_path = Path("outputs/compare_sft_dpo.jsonl")
    out_path.parent.mkdir(parents=True, exist_ok=True)
    with out_path.open("w", encoding="utf-8") as f:
        for r in results:
            f.write(json.dumps(r, ensure_ascii=False) + "\n")


if __name__ == "__main__":
    main()
```

### 十七、面试怎么讲 SFT vs DPO 对比

如果面试官问“怎么判断 DPO 有没有效果”，可以这样回答：

```text
我会用同一套评估 prompt、同一套生成参数，对比 SFT 模型和 DPO 模型的输出。评估维度包括指令遵循、是否更接近 chosen 偏好、是否减少 rejected 中的问题、格式稳定性、事实正确性、安全性和领域外泛化。除了看 DPO loss 和 reward margin，还要做人工样例对比，因为 loss 下降不一定等价于行为变好。
```

如果追问“DPO 后一定比 SFT 好吗”，可以回答：

```text
不一定。DPO 的效果强依赖偏好数据质量、beta、学习率和训练步数。如果偏好数据有反标、chosen 质量差、rejected 太弱或覆盖太窄，DPO 可能导致模板化、过度拒答、领域外退化或过拟合。
```

如果问“DPO 主要改善什么”，可以回答：

```text
DPO 主要改善模型在偏好数据覆盖范围内的选择倾向，让模型相对 SFT 更偏好 chosen 风格、减少 rejected 中的问题。它更像行为对齐和偏好校准，不是凭空注入新知识。
```

### 本讲总结

这一讲比较了 SFT 和 DPO 后模型输出。

核心结论如下：

1. DPO 是否有效，最终要看模型行为，而不只是 loss。
2. 对比时要固定 prompt 模板、生成参数和评估集。
3. SFT 到 DPO 的主要变化是偏好对齐，而不是基础知识凭空增强。
4. 评估应覆盖指令遵循、偏好对齐、正确性、格式、安全性和泛化。
5. DPO 可能带来过短、模板化、过度拒答和领域外退化等副作用。
6. 高质量 preference 数据是 DPO 成功的关键。
7. 完整偏好优化项目应包含数据、训练、日志、模型对比和失败案例分析。
