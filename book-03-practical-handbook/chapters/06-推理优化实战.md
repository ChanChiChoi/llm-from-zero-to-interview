从这一部分开始，我们进入推理优化实战。

大模型推理优化最核心的概念之一就是 KV Cache。

没有 KV Cache，自回归生成会大量重复计算历史 token 的 key 和 value。

有了 KV Cache，每一步只需要计算新 token 的 q/k/v，并复用历史 k/v。

这会显著降低 decode 阶段延迟。

### 一、自回归生成为什么慢

自回归语言模型一次生成一个 token。

假设 prompt 是：

```text
我 喜欢 机器 学习
```

模型生成第一个新 token 时，需要处理整个 prompt。

生成第二个新 token 时，输入变成：

```text
我 喜欢 机器 学习 [新 token 1]
```

如果不做缓存，每一步都会重新计算所有历史 token 的 attention。

生成越长，重复计算越多。

这非常浪费。

KV Cache 的核心思想是：

```text
历史 token 的 key 和 value 已经算过，就不要重复算。
```

假设 prompt 长度为 `T`，后续生成 `N` 个 token。

如果不使用 cache，仅从 K/V projection 角度看，每一步都要重新处理完整历史：

```math
C_{\mathrm{no\_cache}}
=
\sum_{s=1}^{N}(T+s)
=
NT+\frac{N(N+1)}{2}
```

使用 cache 时，prefill 计算 `T` 个 prompt token 的 K/V，之后每步只计算 1 个新 token：

```math
C_{\mathrm{cache}}=T+N
```

KV Cache 省掉的是历史 token 的重复 K/V 和中间层计算。注意 attention 本身仍然要让新 query 读历史 K/V，所以 decode 每步的 attention 长度仍随上下文增长。

---

### 二、Prefill 和 Decode

大模型推理通常分成两个阶段。

#### 1. Prefill 阶段

处理完整 prompt。

例如输入长度是 `T=1024`。

模型一次性计算这 1024 个 token 的 hidden states，并为每层缓存：

```text
K cache
V cache
```

Prefill 特点：

```text
输入序列长。
可以并行计算。
主要消耗算力。
```

#### 2. Decode 阶段

每次只生成一个新 token。

新 token 的 query 会 attention 到历史所有 key/value。

Decode 特点：

```text
每步输入通常是 1 个 token。
强自回归，难并行。
主要受 KV Cache 读写和显存带宽影响。
```

KV Cache 主要优化 decode 阶段。

---

### 三、没有 KV Cache 的计算

假设已经生成到长度 `T`。

没有 cache 时，每一步都输入完整序列：

```text
x: [B, T, D]
```

attention 中会重新计算：

```text
q = Wq x
k = Wk x
v = Wv x
```

每一步都对所有历史 token 重算 k/v。

生成第 `T+1` 个 token 时，历史前 `T` 个 token 的 k/v 其实不变。

所以这是重复计算。

---

### 四、有 KV Cache 的计算

有 cache 后，decode 每一步只输入新 token：

```text
x_new: [B, 1, D]
```

只计算新 token 的：

```text
q_new, k_new, v_new
```

然后把 `k_new/v_new` 追加到历史 cache：

```text
k_cache = concat(k_cache, k_new)
v_cache = concat(v_cache, v_new)
```

attention 使用：

```text
q_new attends to all cached keys and values
```

也就是：

```text
q_new:   [B, H, 1, Dh]
k_cache: [B, H, T, Dh]
v_cache: [B, H, T, Dh]
```

输出：

```text
out_new: [B, 1, D]
```

---

### 五、KV Cache 的 shape

多头 attention 中，常见 shape 是：

```text
k_cache: [batch, num_heads, seq_len, head_dim]
v_cache: [batch, num_heads, seq_len, head_dim]
```

如果：

```text
B = 2
H = 8
T = 128
Dh = 64
```

则：

```text
k_cache: [2, 8, 128, 64]
v_cache: [2, 8, 128, 64]
```

每一层 Transformer Block 都有自己的 KV Cache。

所以完整模型 cache 通常是一个 list：

```text
past_key_values[layer_id] = (k_cache, v_cache)
```

对第 `l` 层可以写成：

```math
K_l,V_l\in\mathbb{R}^{B\times H_{\mathrm{kv}}\times T\times D_h}
```

在普通 MHA 中，`H_kv=H`。在 MQA/GQA 中，`H_kv` 可能小于 query heads 数量，这是它们降低 KV Cache 显存的主要来源之一。

### 六、先实现 split 和 merge heads

```python
import math
import torch
import torch.nn as nn
import torch.nn.functional as F


class CachedSelfAttention(nn.Module):
    def __init__(self, d_model, num_heads):
        super().__init__()
        assert d_model % num_heads == 0

        self.d_model = d_model
        self.num_heads = num_heads
        self.head_dim = d_model // num_heads

        self.q_proj = nn.Linear(d_model, d_model)
        self.k_proj = nn.Linear(d_model, d_model)
        self.v_proj = nn.Linear(d_model, d_model)
        self.out_proj = nn.Linear(d_model, d_model)

    def split_heads(self, x):
        batch_size, seq_len, _ = x.shape
        x = x.reshape(batch_size, seq_len, self.num_heads, self.head_dim)
        return x.transpose(1, 2)

    def merge_heads(self, x):
        batch_size, num_heads, seq_len, head_dim = x.shape
        x = x.transpose(1, 2).contiguous()
        return x.reshape(batch_size, seq_len, num_heads * head_dim)
```

输入 `x`：

```text
[B, T, D]
```

拆头后：

```text
[B, H, T, Dh]
```

### 七、实现带 cache 的 forward

```python
    def forward(self, x, past_key_value=None, use_cache=False):
        q = self.split_heads(self.q_proj(x))
        k = self.split_heads(self.k_proj(x))
        v = self.split_heads(self.v_proj(x))

        if past_key_value is not None:
            past_k, past_v = past_key_value
            k = torch.cat([past_k, k], dim=2)
            v = torch.cat([past_v, v], dim=2)

        scores = q @ k.transpose(-2, -1)
        scores = scores / math.sqrt(self.head_dim)

        attn_weights = F.softmax(scores, dim=-1)
        out = attn_weights @ v
        out = self.merge_heads(out)
        out = self.out_proj(out)

        new_past_key_value = (k, v) if use_cache else None
        return out, new_past_key_value
```

这里先省略 causal mask。

decode 阶段一次输入一个新 token，且 k/v cache 只包含历史和当前 token，一般不需要额外 causal mask。

但 prefill 阶段处理完整 prompt 时仍然需要 causal mask。

### 八、完整 CachedSelfAttention

```python
class CachedSelfAttention(nn.Module):
    def __init__(self, d_model, num_heads):
        super().__init__()
        assert d_model % num_heads == 0

        self.d_model = d_model
        self.num_heads = num_heads
        self.head_dim = d_model // num_heads

        self.q_proj = nn.Linear(d_model, d_model)
        self.k_proj = nn.Linear(d_model, d_model)
        self.v_proj = nn.Linear(d_model, d_model)
        self.out_proj = nn.Linear(d_model, d_model)

    def split_heads(self, x):
        batch_size, seq_len, _ = x.shape
        x = x.reshape(batch_size, seq_len, self.num_heads, self.head_dim)
        return x.transpose(1, 2)

    def merge_heads(self, x):
        batch_size, num_heads, seq_len, head_dim = x.shape
        x = x.transpose(1, 2).contiguous()
        return x.reshape(batch_size, seq_len, num_heads * head_dim)

    def forward(self, x, past_key_value=None, use_cache=False, causal_mask=None):
        q = self.split_heads(self.q_proj(x))
        k = self.split_heads(self.k_proj(x))
        v = self.split_heads(self.v_proj(x))

        if past_key_value is not None:
            past_k, past_v = past_key_value
            k = torch.cat([past_k, k], dim=2)
            v = torch.cat([past_v, v], dim=2)

        scores = q @ k.transpose(-2, -1)
        scores = scores / math.sqrt(self.head_dim)

        if causal_mask is not None:
            scores = scores.masked_fill(causal_mask == 0, torch.finfo(scores.dtype).min)

        attn_weights = F.softmax(scores, dim=-1)
        out = attn_weights @ v
        out = self.merge_heads(out)
        out = self.out_proj(out)

        new_past_key_value = (k, v) if use_cache else None
        return out, new_past_key_value
```

### 十一、为什么 decode 阶段通常不需要完整 causal mask

decode 阶段每次只处理当前位置的新 token。

它只能看到：

```text
历史 cache + 当前 token
```

不存在未来 token。

所以通常不需要像 prefill 那样构造完整 `[T, T]` causal mask。

但要注意：

```text
如果一次 decode 多个 token，或者使用复杂 batch/padding，就仍然要小心 mask。
```

教学实现中，单 token decode 可以先不加 causal mask。

---

### 十二、KV Cache 和 RoPE position offset

如果模型使用 RoPE，decode 时不能把新 token 的位置当成 0。

例如 prompt 长度是 4。

生成第一个新 token 的 position 应该是：

```text
4
```

而不是：

```text
0
```

所以 RoPE 实现通常需要 `position_ids` 或 `offset`。

简化写法：

```python
past_len = 0 if past_key_value is None else past_key_value[0].size(2)
position_ids = torch.arange(past_len, past_len + x.size(1), device=x.device)
```

公式上，如果本次输入长度为 `T_new`，历史 cache 长度为 `T_past`，则本次 token 的 position ids 是：

```math
p_j=T_{\mathrm{past}}+j,\qquad j=0,\ldots,T_{\mathrm{new}}-1
```

然后用这些位置取 RoPE 的 cos/sin。

如果 position offset 错了，模型会混淆位置关系，生成质量明显变差。

---

### 十三、KV Cache 显存占用

KV Cache 会节省计算，但会增加显存。

每层 cache 大小约为：

```math
M_{\mathrm{layer}}
=
2B H_{\mathrm{kv}} T D_h b
```

其中 2 表示 K 和 V。

整个模型还要乘以层数：

```math
M_{\mathrm{KV}}
=
2L B H_{\mathrm{kv}} T D_h b
```

因为：

```math
H_{\mathrm{kv}}D_h=d_{\mathrm{kv}}
```

普通 MHA 中 `d_kv` 等于 hidden size `D`，所以也常粗略写成：

```math
M_{\mathrm{KV}}
\approx
2LBTDb
```

所以长上下文和大 batch 会让 KV Cache 显存迅速增长。

---

### 十四、KV Cache 的收益和代价

收益：

```text
避免重复计算历史 k/v。
显著降低 decode 延迟。
是大模型在线生成的基础优化。
```

代价：

```text
占用额外显存。
长上下文下 cache 很大。
batch 内不同长度序列管理复杂。
需要正确处理 position offset。
```

这就是为什么 vLLM、PagedAttention、continuous batching 等推理系统会重点优化 KV Cache 管理。

### 十五、完整 KV Cache 验证脚本

下面这个脚本验证六件事：

1. prefill 后 K/V cache shape 正确。
2. decode 一个新 token 后 cache 长度加 1。
3. decode position id 从历史长度之后开始。
4. cache decode 的新 token 输出，与 full causal attention 最后一位输出一致。
5. KV Cache 显存估算和 shape 对齐。
6. 使用 cache 后，K/V projection 需要处理的 token 数显著减少。

```python
import math

import torch
import torch.nn as nn
import torch.nn.functional as F


torch.manual_seed(123)


class CachedSelfAttention(nn.Module):
    def __init__(self, d_model, num_heads):
        super().__init__()
        assert d_model % num_heads == 0
        self.d_model = d_model
        self.num_heads = num_heads
        self.head_dim = d_model // num_heads
        self.q_proj = nn.Linear(d_model, d_model)
        self.k_proj = nn.Linear(d_model, d_model)
        self.v_proj = nn.Linear(d_model, d_model)
        self.out_proj = nn.Linear(d_model, d_model)

    def split_heads(self, x):
        batch_size, seq_len, _ = x.shape
        x = x.reshape(batch_size, seq_len, self.num_heads, self.head_dim)
        return x.transpose(1, 2)

    def merge_heads(self, x):
        batch_size, num_heads, seq_len, head_dim = x.shape
        x = x.transpose(1, 2).contiguous()
        return x.reshape(batch_size, seq_len, num_heads * head_dim)

    def forward(self, x, past_key_value=None, use_cache=False, causal_mask=None):
        q = self.split_heads(self.q_proj(x))
        k_new = self.split_heads(self.k_proj(x))
        v_new = self.split_heads(self.v_proj(x))

        if past_key_value is None:
            k = k_new
            v = v_new
            past_len = 0
        else:
            past_k, past_v = past_key_value
            past_len = past_k.size(2)
            k = torch.cat([past_k, k_new], dim=2)
            v = torch.cat([past_v, v_new], dim=2)

        scores = q @ k.transpose(-2, -1) / math.sqrt(self.head_dim)
        if causal_mask is not None:
            scores = scores.masked_fill(~causal_mask, torch.finfo(scores.dtype).min)
        weights = F.softmax(scores, dim=-1)
        out = self.out_proj(self.merge_heads(weights @ v))
        new_cache = (k, v) if use_cache else None
        position_ids = torch.arange(past_len, past_len + x.size(1), device=x.device)
        return out, new_cache, position_ids


def causal_mask(query_len, key_len, past_len=0, device="cpu"):
    query_positions = torch.arange(past_len, past_len + query_len, device=device).view(query_len, 1)
    key_positions = torch.arange(key_len, device=device).view(1, key_len)
    return (key_positions <= query_positions).view(1, 1, query_len, key_len)


B, T, D, H = 1, 4, 16, 4
new_tokens = 3
attn = CachedSelfAttention(D, H)
prompt = torch.randn(B, T, D)
first_new = torch.randn(B, 1, D)

prefill_mask = causal_mask(query_len=T, key_len=T)
prefill_out, cache, prefill_pos = attn(prompt, use_cache=True, causal_mask=prefill_mask)

decode_out, cache_after_one, decode_pos = attn(first_new, past_key_value=cache, use_cache=True)
full_input = torch.cat([prompt, first_new], dim=1)
full_mask = causal_mask(query_len=T + 1, key_len=T + 1)
full_out, _, _ = attn(full_input, use_cache=False, causal_mask=full_mask)
max_abs_diff = (decode_out - full_out[:, -1:, :]).abs().max().item()

running_cache = cache
position_trace = []
for _ in range(new_tokens):
    token = torch.randn(B, 1, D)
    _, running_cache, pos = attn(token, past_key_value=running_cache, use_cache=True)
    position_trace.append(pos.item())

layers = 2
bytes_per_element = 2
one_layer_cache_bytes = 2 * B * H * cache_after_one[0].size(2) * (D // H) * bytes_per_element
model_cache_bytes = layers * one_layer_cache_bytes
no_cache_kv_tokens = sum(T + step + 1 for step in range(new_tokens))
with_cache_kv_tokens = T + new_tokens

print("prefill_out_shape=", tuple(prefill_out.shape))
print("prefill_k_shape=", tuple(cache[0].shape))
print("prefill_v_shape=", tuple(cache[1].shape))
print("prefill_position_ids=", prefill_pos.tolist())
print("decode_out_shape=", tuple(decode_out.shape))
print("decode_position_ids=", decode_pos.tolist())
print("new_cache_k_shape=", tuple(cache_after_one[0].shape))
print("full_vs_cache_last_close=", torch.allclose(decode_out, full_out[:, -1:, :], atol=1e-6))
print("max_abs_diff=", round(max_abs_diff, 8))
print("position_trace=", position_trace)
print("final_cache_len=", running_cache[0].size(2))
print("one_layer_cache_bytes_fp16=", one_layer_cache_bytes)
print("two_layer_cache_bytes_fp16=", model_cache_bytes)
print("no_cache_kv_tokens=", no_cache_kv_tokens)
print("with_cache_kv_tokens=", with_cache_kv_tokens)
print("saved_kv_projection_tokens=", no_cache_kv_tokens - with_cache_kv_tokens)
```

一次稳定输出如下：

```text
prefill_out_shape= (1, 4, 16)
prefill_k_shape= (1, 4, 4, 4)
prefill_v_shape= (1, 4, 4, 4)
prefill_position_ids= [0, 1, 2, 3]
decode_out_shape= (1, 1, 16)
decode_position_ids= [4]
new_cache_k_shape= (1, 4, 5, 4)
full_vs_cache_last_close= True
max_abs_diff= 6e-08
position_trace= [4, 5, 6]
final_cache_len= 7
one_layer_cache_bytes_fp16= 320
two_layer_cache_bytes_fp16= 640
no_cache_kv_tokens= 18
with_cache_kv_tokens= 7
saved_kv_projection_tokens= 11
```

最重要的是 `full_vs_cache_last_close=True`：它说明在 causal mask 和 position 处理正确时，用 cache 逐 token decode 得到的新 token 输出，和把完整序列重新跑一遍得到的最后位置输出是一致的。

### 十六、常见工程坑

#### 坑 1：每步仍然输入完整序列

如果使用 KV Cache，decode 阶段应该只输入新 token。

#### 坑 2：cache 拼接维度错了

K/V shape 是 `[B, H, T, Dh]`，应该沿 `T` 维拼接，也就是 `dim=2`。

#### 坑 3：忘记更新 cache

每一步生成后都要把新的 k/v 放回 cache。

#### 坑 4：RoPE position offset 错误

新 token 的位置应该接在历史长度之后。

#### 坑 5：prefill 阶段忘记 causal mask

完整 prompt prefill 时仍然需要 causal mask，防止位置看到未来。

#### 坑 6：KV Cache 显存估计不足

长上下文、多层、大 batch 下 cache 会非常大。

### 十七、面试怎么讲 KV Cache

如果面试官问“KV Cache 是什么”，可以这样回答：

```text
KV Cache 是自回归推理中的缓存机制。Transformer decode 每一步只新增一个 token，历史 token 的 key 和 value 不会变化，所以可以把每层历史 K/V 缓存起来。下一步只计算新 token 的 Q/K/V，并让新 token 的 Q attend 到缓存中的全部 K/V，从而避免重复计算历史 token。
```

如果追问“KV Cache 的 shape 是什么”，可以回答：

```text
常见 shape 是 [batch, num_heads, seq_len, head_dim]，每层都有一份 K cache 和 V cache，所以完整模型的 past_key_values 通常是一个按层存储的 list，每层包含 (k_cache, v_cache)。
```

如果问“KV Cache 有什么代价”，可以回答：

```text
KV Cache 能降低 decode 计算量，但会占用显存。显存大约和 layers、batch、seq_len、hidden_size 成正比，长上下文和大 batch 下 KV Cache 会成为主要显存瓶颈。
```

如果问“RoPE 和 KV Cache 有什么关系”，可以回答：

```text
使用 RoPE 时，decode 阶段新 token 的 position 不能从 0 开始，而应该接在 past length 后面。否则 q/k 的旋转位置会错，影响 attention 的相对位置关系和生成质量。
```

#### 练习 2

### 本讲总结

这一讲实现了 KV Cache。

核心结论如下：

1. KV Cache 用来缓存历史 token 的 key 和 value。
2. Prefill 阶段处理完整 prompt，并建立初始 cache。
3. Decode 阶段每次只处理新 token，并复用历史 cache。
4. K/V cache 常见 shape 是 `[B, H, T, Dh]`。
5. cache 拼接应沿序列长度维 `dim=2`。
6. KV Cache 降低 decode 计算量，但增加显存占用。
7. 使用 RoPE 时必须正确处理 position offset。
8. 正确实现下，cache decode 的新 token 输出应和 full causal attention 的最后位置输出一致。

### 一、Prefill Latency 是什么

Prefill 是处理 prompt 的阶段。

输入：

```text
prompt tokens: [B, T_prompt]
```

模型要对完整 prompt 做一次 forward，建立 KV Cache。

Prefill latency 指的是：

```text
从输入 prompt 到模型完成 prompt 处理并准备生成第一个 token 的时间。
```

通常也和 TTFT 相关。

TTFT 是 Time To First Token：

```text
从请求开始到第一个 token 输出的时间。
```

TTFT 包含 prefill，还可能包含排队、调度、网络、采样等开销。

本讲先关注模型侧 prefill。

---

### 二、Decode Latency 是什么

Decode 是逐 token 生成阶段。

每一步输入通常只有一个新 token。

模型使用 KV Cache 生成下一个 token。

Decode latency 指的是：

```text
每生成一个新 token 所需的时间。
```

常见指标：

```text
per-token latency
decode tokens/s
```

在线聊天体验中，decode latency 直接决定流式输出速度。

---

### 三、为什么要分开测

Prefill 和 decode 的瓶颈不同。

Prefill：

```text
输入长。
矩阵计算大。
并行度高。
更像大 batch 矩阵计算。
```

Decode：

```text
每次只生成一个 token。
并行度低。
频繁读写 KV Cache。
容易受显存带宽和调度开销影响。
```

所以优化方法也不同。

Prefill 常关注：

```text
FlashAttention
prompt batching
chunked prefill
长上下文优化
```

Decode 常关注：

```text
KV Cache 管理
continuous batching
PagedAttention
speculative decoding
量化
```

用符号表达，设 prompt token 数为 `T_p`，输出 token 数为 `T_o`。

模型侧 prefill 延迟记为：

```math
t_{\mathrm{prefill}}
```

如果第一个输出 token 直接来自 prefill logits，则模型侧 TTFT 可以粗略写成：

```math
t_{\mathrm{TTFT}}
=
t_{\mathrm{queue}}+t_{\mathrm{prefill}}+t_{\mathrm{sample,1}}+t_{\mathrm{network,1}}
```

本讲的本地脚本只测模型侧，因此会把排队和网络记为 0，重点看：

```math
t_{\mathrm{TTFT,model}}
\approx
t_{\mathrm{prefill}}+t_{\mathrm{sample,1}}
```

后续输出 token 的平均间隔常用 TPOT 表示：

```math
t_{\mathrm{TPOT}}
=
\frac{1}{T_o-1}
\sum_{j=2}^{T_o}t_{\mathrm{decode},j}
```

decode 吞吐可以写成：

```math
R_{\mathrm{decode}}
=
\frac{T_o-1}{\sum_{j=2}^{T_o}t_{\mathrm{decode},j}}
```

prefill 吞吐可以写成：

```math
R_{\mathrm{prefill}}
=
\frac{T_p}{t_{\mathrm{prefill}}}
```

实际线上服务要区分“单请求 latency”和“系统整体 throughput”。单请求快，不一定代表系统吞吐最高。

### 四、计时前的注意事项

GPU 操作通常是异步的。

如果直接写：


可能测不准。

正确做法是在计时前后同步：

```python

```

或者使用 CUDA Event。

### 九、完整测量生成过程

```python
@torch.no_grad()
def benchmark_generation(model, tokenizer, prompt, max_new_tokens, device):
    inputs = tokenizer(prompt, return_tensors="pt").to(device)
    input_ids = inputs["input_ids"]
    attention_mask = inputs["attention_mask"]

    start = now()
    outputs = model(
        input_ids=input_ids,
        attention_mask=attention_mask,
        use_cache=True,
    )
    prefill_end = now()

    past_key_values = outputs.past_key_values
    next_token_logits = outputs.logits[:, -1, :]
    next_token = torch.argmax(next_token_logits, dim=-1, keepdim=True)

    decode_times = []
    generated = [next_token]

    for _ in range(max_new_tokens - 1):
        attention_mask = torch.cat(
            [attention_mask, torch.ones((attention_mask.size(0), 1), device=device, dtype=attention_mask.dtype)],
            dim=1,
        )

        step_start = now()
        outputs = model(
            input_ids=next_token,
            attention_mask=attention_mask,
            past_key_values=past_key_values,
            use_cache=True,
        )
        step_end = now()

        decode_times.append(step_end - step_start)

        past_key_values = outputs.past_key_values
        next_token_logits = outputs.logits[:, -1, :]
        next_token = torch.argmax(next_token_logits, dim=-1, keepdim=True)
        generated.append(next_token)

    end = now()

    prompt_len = input_ids.shape[1]
    prefill_latency = prefill_end - start
    total_latency = end - start
    decode_latency = sum(decode_times)

    generated_ids = torch.cat([input_ids] + generated, dim=1)

    return {
        "prompt_len": prompt_len,
        "max_new_tokens": max_new_tokens,
        "prefill_latency_s": prefill_latency,
        "decode_latency_s": decode_latency,
        "total_latency_s": total_latency,
        "avg_decode_latency_s": decode_latency / max(len(decode_times), 1),
        "decode_tokens_per_s": len(decode_times) / max(decode_latency, 1e-9),
        "text": tokenizer.decode(generated_ids[0], skip_special_tokens=True),
    }
```

这里第一个新 token 是从 prefill logits 中取出来的。

后续 token 才进入循环 decode。

### 十七、常见工程坑

#### 坑 1：没有 CUDA synchronize

GPU 异步执行会导致计时偏小。

#### 坑 2：把第一次运行当正式结果

第一次运行可能包含 warmup 开销。

#### 坑 3：混淆 prefill 和 decode

prompt 长度主要影响 prefill，生成长度主要影响 decode。

#### 坑 4：采样策略影响计时

复杂采样会增加额外开销。
性能基准可以先用 greedy。

#### 坑 5：batch size 不固定

不同 batch size 的 latency 和 throughput 不可直接比较。

#### 坑 6：没有固定生成长度

如果模型遇到 EOS 提前停止，不同样本生成长度不同，结果不公平。

基准测试中可以固定 `max_new_tokens`，并记录实际生成长度。

### 十八、面试怎么讲 Prefill 和 Decode

如果面试官问“prefill 和 decode 有什么区别”，可以这样回答：

```text
Prefill 是处理输入 prompt 的阶段，模型对完整 prompt 做并行 forward 并建立 KV Cache；decode 是逐 token 生成阶段，每一步只输入新 token，并复用历史 KV Cache。Prefill 更偏计算密集，decode 更受自回归串行、KV Cache 读写和显存带宽影响。
```

如果问“怎么测推理延迟”，可以回答：

```text
我会把 prefill 和 decode 分开测。prefill 测完整 prompt forward 并返回 past_key_values 的时间；decode 测使用 past_key_values 后每步生成一个 token 的平均耗时。GPU 上计时前后要 torch.cuda.synchronize，并做 warmup，多次重复取平均。
```

如果问“TTFT 和 TPOT 是什么”，可以回答：

```text
TTFT 是 time to first token，表示用户从发起请求到看到第一个 token 的时间，主要受 prefill 和调度影响。TPOT 是 time per output token，表示后续每个输出 token 的平均生成时间，主要反映 decode 阶段速度。
```

---

### 十九、小练习

#### 练习 1

测量同一个模型在 short、medium、long prompt 下的 prefill latency。

#### 练习 2

固定 prompt，分别生成 8、32、128 个 token，比较 decode latency。

#### 练习 3

去掉 `torch.cuda.synchronize()`，观察计时是否异常。

#### 练习 4

比较 greedy decoding 和 top-p sampling 的延迟差异。

#### 练习 5

把结果保存成 JSONL，并写一个简单脚本计算平均 TPOT。

### 本讲总结

这一讲测量了 prefill 与 decode 延迟。

核心结论如下：

1. Prefill 处理完整 prompt，并建立 KV Cache。
2. Decode 逐 token 生成，并复用 KV Cache。
3. Prefill latency 主要受 prompt 长度影响。
4. Decode latency 主要受生成长度、KV Cache 长度和显存带宽影响。
5. GPU 计时必须同步，否则结果不准。
6. 基准测试应 warmup、多次重复、固定条件。
7. TTFT 和 TPOT 是在线推理体验中的关键指标。
8. 没有真实模型环境时，也可以用固定 trace 先验证指标统计逻辑。

### 一、vLLM 解决什么问题

Hugging Face `model.generate` 适合本地实验。

但线上部署会遇到问题：

```text
多个请求长度不同。
KV Cache 管理复杂。
batch 动态变化。
显存碎片和浪费。
吞吐不足。
流式输出和 API 服务需要额外实现。
```

vLLM 的核心优势包括：

```text
PagedAttention 管理 KV Cache。
高吞吐 continuous batching。
OpenAI 兼容 API Server。
支持多种 Hugging Face 模型。
支持 tensor parallel。
```

你可以把 vLLM 理解为：

```text
面向生产推理的高性能 LLM serving engine。
```

从 KV Cache 角度看，vLLM 不只是在“更快地调用 generate”，而是在管理一组动态请求：

```math
\mathcal{R}=\{r_i=(T_{p,i},T_{o,i},s_i)\}_{i=1}^{N}
```

其中 `T_p` 是 prompt token 数，`T_o` 是计划输出 token 数，`s_i` 是请求状态。serving engine 要在显存预算内动态安排这些请求。

### 九、关键参数：max_model_len

启动服务时可以设置最大上下文长度：

```bash
vllm serve facebook/opt-125m --max-model-len 2048
```

`max_model_len` 影响：

```text
最大 prompt + output token 长度。
KV Cache 预留和调度。
显存占用。
```

对单个请求，基本约束是：

```math
T_{\mathrm{prompt}}+T_{\mathrm{output}}\le L_{\max}
```

其中 `L_max` 对应 `max_model_len`。如果这个值设得很大，系统会为更长上下文规划 KV Cache 容量；如果设得太小，长 prompt 或长输出会被拒绝。

如果设置太大，可能显存不够。

如果设置太小，长 prompt 会报错或被拒绝。

---

### 十、关键参数：gpu_memory_utilization

```bash
vllm serve facebook/opt-125m --gpu-memory-utilization 0.9
```

它表示 vLLM 可以使用 GPU 显存的比例。

常见值：

```text
0.8
0.9
0.95
```

设置太低：

```text
可用 KV Cache 空间少，吞吐可能下降。
```

设置太高：

```text
更容易 OOM，影响系统稳定性。
```

线上部署要保留一定显存余量。

可以用一个粗略预算理解它：

```math
M_{\mathrm{usable}}
=uM_{\mathrm{gpu}}
```

```math
M_{\mathrm{KV,avail}}
=
M_{\mathrm{usable}}
-M_{\mathrm{weights}}
-M_{\mathrm{runtime}}
```

其中 `u` 是 `gpu_memory_utilization`，`M_gpu` 是总显存，`M_weights` 是模型权重显存，`M_runtime` 是框架、activation、临时 buffer 等预留。

---

### 十一、关键参数：tensor_parallel_size

多 GPU 部署大模型时使用 tensor parallel：

```bash
vllm serve /path/to/model --tensor-parallel-size 2
```

表示把模型切到 2 张 GPU 上。

如果 4 张 GPU：

```bash
vllm serve /path/to/model --tensor-parallel-size 4
```

注意：

```text
tensor_parallel_size 应该不超过可见 GPU 数量。
```

基本约束是：

```math
N_{\mathrm{TP}}\le N_{\mathrm{visible\_gpu}}
```

`tensor_parallel_size` 增大后，单卡权重显存压力可能下降，但通信开销会上升，且并不自动解决所有 KV Cache 容量问题。

如果只想使用某些 GPU，可以设置：

```bash
CUDA_VISIBLE_DEVICES=0,1 vllm serve /path/to/model --tensor-parallel-size 2
```

### 十五、vLLM 为什么吞吐高

vLLM 的关键设计包括：

```text
PagedAttention：像操作系统分页一样管理 KV Cache。
Continuous batching：动态把不同请求合并执行。
高效 kernel：优化 attention 和采样等操作。
OpenAI API server：方便接入服务。
```

PagedAttention 的直觉：

```text
不要为每个请求预留一整块连续 KV Cache，而是按块管理，减少碎片和浪费。
```

如果每个 KV token 的显存为：

```math
M_{\mathrm{token}}
=
2L H_{\mathrm{kv}}D_h b
```

block size 为 `B_blk`，则每个 block 大小约为：

```math
M_{\mathrm{block}}
=
B_{\mathrm{blk}}M_{\mathrm{token}}
```

在可用 KV 显存 `M_KV,avail` 下，最多可分配 block 数约为：

```math
N_{\mathrm{block}}
=
\left\lfloor
\frac{M_{\mathrm{KV,avail}}}{M_{\mathrm{block}}}
\right\rfloor
```

最大可缓存 token 数约为：

```math
T_{\mathrm{capacity}}
=
N_{\mathrm{block}}B_{\mathrm{blk}}
```

这对长上下文、多并发推理非常重要。

### 十六、0 依赖 vLLM 配置审计 demo

本机如果没有安装 vLLM 或没有合适 GPU，真实 `vllm serve` 可能无法运行。

下面这个 0 依赖脚本先审计配置和容量边界：

1. `max_model_len` 是否覆盖请求的 prompt + output。
2. `gpu_memory_utilization` 是否在合理范围。
3. `tensor_parallel_size` 是否不超过可见 GPU 数。
4. `SamplingParams` 是否合法。
5. OpenAI API 请求里的 `model` 字段是否使用服务端暴露名。
6. 粗略估算 KV block 容量是否覆盖最大请求。

```python
from math import floor


GIB = 1024 ** 3

CONFIG = {
    "model_name": "facebook/opt-125m",
    "served_model_name": "opt-125m-local",
    "max_model_len": 4096,
    "gpu_memory_utilization": 0.85,
    "tensor_parallel_size": 1,
    "visible_gpus": 1,
    "gpu_total_gib": 24,
    "model_weights_gib": 8,
    "runtime_reserve_gib": 2,
    "layers": 24,
    "kv_heads": 16,
    "head_dim": 64,
    "bytes_per_element": 2,
    "block_size": 16,
}
SAMPLING = {
    "temperature": 0.7,
    "top_p": 0.9,
    "top_k": 50,
    "max_tokens": 256,
}
REQUESTS = [
    {"id": "short", "prompt_tokens": 512, "max_tokens": 128},
    {"id": "medium", "prompt_tokens": 2048, "max_tokens": 512},
    {"id": "too_long", "prompt_tokens": 3900, "max_tokens": 256},
]


def kv_bytes_per_token(cfg):
    return 2 * cfg["layers"] * cfg["kv_heads"] * cfg["head_dim"] * cfg["bytes_per_element"]


def available_kv_bytes(cfg):
    usable = cfg["gpu_total_gib"] * cfg["gpu_memory_utilization"]
    available = usable - cfg["model_weights_gib"] - cfg["runtime_reserve_gib"]
    return max(0, int(available * GIB))


def audit_sampling(params):
    return {
        "temperature_ok": params["temperature"] >= 0,
        "top_p_ok": 0 < params["top_p"] <= 1,
        "top_k_ok": params["top_k"] == -1 or params["top_k"] >= 1,
        "max_tokens_ok": params["max_tokens"] > 0,
    }


def audit_requests(cfg, requests):
    rows = []
    for req in requests:
        total = req["prompt_tokens"] + req["max_tokens"]
        rows.append((req["id"], total, total <= cfg["max_model_len"]))
    return rows


per_token = kv_bytes_per_token(CONFIG)
per_block = per_token * CONFIG["block_size"]
available = available_kv_bytes(CONFIG)
num_blocks = floor(available / per_block)
max_cache_tokens = num_blocks * CONFIG["block_size"]
request_rows = audit_requests(CONFIG, REQUESTS)
sampling_ok = audit_sampling(SAMPLING)
config_ok = {
    "tp_size_ok": CONFIG["tensor_parallel_size"] <= CONFIG["visible_gpus"],
    "gpu_util_ok": 0 < CONFIG["gpu_memory_utilization"] < 1,
    "max_model_len_ok": CONFIG["max_model_len"] > 0,
    "served_name_ok": bool(CONFIG["served_model_name"]),
}
max_request_tokens = max(total for _, total, _ in request_rows)

print("kv_bytes_per_token=", per_token)
print("kv_bytes_per_block=", per_block)
print("available_kv_gib=", round(available / GIB, 2))
print("num_kv_blocks=", num_blocks)
print("max_cache_tokens=", max_cache_tokens)
print("max_request_tokens=", max_request_tokens)
print("capacity_ok_for_largest_request=", max_request_tokens <= max_cache_tokens)
print("request_length_ok=", {name: ok for name, _, ok in request_rows})
print("sampling_ok=", sampling_ok)
print("config_ok=", config_ok)
print("openai_base_url=", "http://localhost:8000/v1")
print("completion_model_field=", CONFIG["served_model_name"])
```

一次稳定输出如下：

```text
kv_bytes_per_token= 98304
kv_bytes_per_block= 1572864
available_kv_gib= 10.4
num_kv_blocks= 7099
max_cache_tokens= 113584
max_request_tokens= 4156
capacity_ok_for_largest_request= True
request_length_ok= {'short': True, 'medium': True, 'too_long': False}
sampling_ok= {'temperature_ok': True, 'top_p_ok': True, 'top_k_ok': True, 'max_tokens_ok': True}
config_ok= {'tp_size_ok': True, 'gpu_util_ok': True, 'max_model_len_ok': True, 'served_name_ok': True}
openai_base_url= http://localhost:8000/v1
completion_model_field= opt-125m-local
```

注意 `capacity_ok_for_largest_request=True` 不代表所有请求都合法。`too_long` 的 `prompt_tokens + max_tokens` 超过了 `max_model_len`，所以 `request_length_ok` 中它是 `False`。

```text
我会关注 max_model_len、gpu_memory_utilization、tensor_parallel_size、模型 dtype、并发请求数、max_tokens 和采样参数。max_model_len 和并发会影响 KV Cache 显存，tensor_parallel_size 影响多卡切分，gpu_memory_utilization 控制 vLLM 可使用显存比例。
```

### 一、为什么需要量化

假设一个模型有 7B 参数。

如果使用 fp16，每个参数 2 字节。

仅模型权重就大约需要：

```text
7B * 2 bytes = 14GB
```

如果使用 8bit，每个参数约 1 字节：

```text
约 7GB
```

如果使用 4bit，每个参数约 0.5 字节：

```text
约 3.5GB
```

实际还会有 scale、zero point、metadata、KV Cache、激活等额外开销。

但量化确实能显著降低权重显存。

更一般地，假设模型参数量为 `N_p`，权重存储 bit 数为 `q`，量化 metadata 额外比例为 `\alpha_q`，则权重显存近似为：

```math
M_{\mathrm{w}}(q)
\approx
\frac{N_p q}{8}(1+\alpha_q)
```

相对 fp16 的权重压缩率可以写成：

```math
C_q
=
\frac{M_{\mathrm{w}}(16)}{M_{\mathrm{w}}(q)}
```

如果忽略 metadata，8bit 相对 fp16 大约是 2 倍压缩，4bit 大约是 4 倍压缩。真实工程里还要把 scale、zero point、double quant metadata、量化 kernel workspace 和框架缓存算进去。

---

### 二、量化降低的是哪部分显存

推理显存主要包括：

```text
模型权重
KV Cache
临时激活
框架缓存
```

weight-only quantization 主要降低：

```text
模型权重显存
```

它不直接降低 KV Cache。

如果上下文很长，KV Cache 可能成为主要显存瓶颈。

这也是为什么量化后长上下文推理仍然可能 OOM。

推理总显存可以粗略拆成：

```math
M_{\mathrm{total}}
\approx
M_{\mathrm{w}}(q)
+
M_{\mathrm{KV}}
+
M_{\mathrm{act}}
+
M_{\mathrm{runtime}}
```

其中 `M_w(q)` 是量化后的权重显存，`M_KV` 是 KV Cache，`M_act` 是临时激活，`M_runtime` 是框架、kernel、allocator 等运行时开销。

对 decoder-only 模型，KV Cache 显存常用近似为：

```math
M_{\mathrm{KV}}
\approx
2LH_{\mathrm{kv}}D_hB_{\mathrm{batch}}T_{\mathrm{ctx}}b_{\mathrm{kv}}
```

其中 `L` 是层数，`H_kv` 是 KV heads 数，`D_h` 是 head dim，`B_batch` 是 batch size，`T_ctx` 是当前上下文 token 数，`b_kv` 是 KV Cache 每个元素的字节数。注意 weight-only quantization 主要改变 `M_w(q)`，通常不改变这个 `M_KV`。

### 八、统一生成函数

```python
@torch.no_grad()
def generate_once(model, tokenizer, prompt, max_new_tokens=128):
    device = next(model.parameters()).device
    inputs = tokenizer(prompt, return_tensors="pt").to(device)
    pad_token_id = tokenizer.pad_token_id
    if pad_token_id is None:
        pad_token_id = tokenizer.eos_token_id

    outputs = model.generate(
        **inputs,
        max_new_tokens=max_new_tokens,
        do_sample=False,
        pad_token_id=pad_token_id,
    )
    return tokenizer.decode(outputs[0], skip_special_tokens=True)
```

如果 `device_map="auto"` 下模型被切到多张卡，`next(model.parameters()).device` 只是最简单的单入口写法。更复杂的多卡场景，需要结合模型的 hf device map 处理输入位置。

小模型单卡时通常没问题。

### 九、测量延迟

```python
import time


def sync_device():
    if torch.cuda.is_available():
        torch.cuda.synchronize()


@torch.no_grad()
def benchmark_generate(model, tokenizer, prompt, max_new_tokens=128, repeat=3):
    times = []
    outputs = []

    for _ in range(2):
        _ = generate_once(model, tokenizer, prompt, max_new_tokens=16)

    for _ in range(repeat):
        sync_device()
        start = time.perf_counter()
        text = generate_once(model, tokenizer, prompt, max_new_tokens=max_new_tokens)
        sync_device()
        end = time.perf_counter()

        times.append(end - start)
        outputs.append(text)

    return {
        "avg_latency_s": sum(times) / len(times),
        "tokens_per_s_rough": max_new_tokens / (sum(times) / len(times)),
        "sample_output": outputs[-1],
    }
```

表里的核心指标可以写成：

```math
S_q
=
\frac{\tau_{16}}{\tau_q}
```

其中 `S_q` 是相对 fp16 的延迟速度比，`\tau_q` 是量化版本的平均生成延迟。`S_q>1` 才表示更快。

生成吞吐可以写成：

```math
R_q
=
\frac{T_{\mathrm{out}}}{\tau_q}
```

其中 `T_out` 是本次生成的新 token 数。

```math
Q_q
=
\frac{1}{N}\sum_{i=1}^{N} I_i^{(q)}
```

其中 `I_i^{(q)}` 表示第 `i` 个样例在量化版本 `q` 下是否通过。相对 fp16 的质量下降可以写成：

```math
\Delta Q_q
=
Q_{16}-Q_q
```

这类指标很粗，但能提醒你不要只报告显存节省。

### 十三、量化和 KV Cache 的关系

量化主要降低权重显存。

KV Cache 通常仍是 fp16/bf16。

```text

```


### 十七、0 依赖量化收益审计 demo

本机如果没有安装 `transformers`、`bitsandbytes` 或没有 CUDA，也可以先用固定数据审计量化收益。

下面脚本回答四个问题：

1. 权重量化后显存大约省多少。
2. KV Cache 在短上下文和长上下文下分别有多大。
3. 量化后是否能放进 16GB 显存。
4. 延迟和质量是否可能退化。

```python
GIB = 1024 ** 3

MODEL = {
    "params": 7_000_000_000,
    "layers": 32,
    "kv_heads": 32,
    "head_dim": 128,
    "kv_bytes": 2,
    "batch_size": 1,
    "runtime_gib": 1.5,
    "activation_gib": 0.8,
}
VARIANTS = {
    "fp16": {"weight_bits": 16, "metadata_ratio": 0.0, "latency_s": 2.10, "quality": [1, 1, 1, 1, 1]},
    "int8": {"weight_bits": 8, "metadata_ratio": 0.06, "latency_s": 2.30, "quality": [1, 1, 1, 1, 1]},
    "nf4": {"weight_bits": 4, "metadata_ratio": 0.12, "latency_s": 2.60, "quality": [1, 1, 1, 0, 1]},
}


def gib(num_bytes):
    return num_bytes / GIB


def weight_gib(variant):
    raw_bytes = MODEL["params"] * variant["weight_bits"] / 8
    return gib(raw_bytes * (1 + variant["metadata_ratio"]))


def kv_gib(seq_len):
    kv_bytes = (
        2
        * MODEL["layers"]
        * MODEL["kv_heads"]
        * MODEL["head_dim"]
        * MODEL["batch_size"]
        * seq_len
        * MODEL["kv_bytes"]
    )
    return gib(kv_bytes)


def total_gib(variant, seq_len):
    return (
        weight_gib(variant)
        + kv_gib(seq_len)
        + MODEL["runtime_gib"]
        + MODEL["activation_gib"]
    )


def quality_rate(variant):
    return sum(variant["quality"]) / len(variant["quality"])


output_tokens = 128
short_seq = 4096
long_seq = 32768
base_weight = weight_gib(VARIANTS["fp16"])
base_latency = VARIANTS["fp16"]["latency_s"]
base_quality = quality_rate(VARIANTS["fp16"])

weight = {name: round(weight_gib(v), 2) for name, v in VARIANTS.items()}
compression = {name: round(base_weight / weight_gib(v), 2) for name, v in VARIANTS.items()}
short_total = {name: round(total_gib(v, short_seq), 2) for name, v in VARIANTS.items()}
long_total = {name: round(total_gib(v, long_seq), 2) for name, v in VARIANTS.items()}
fits_short = {name: total_gib(v, short_seq) <= 16 for name, v in VARIANTS.items()}
fits_long = {name: total_gib(v, long_seq) <= 16 for name, v in VARIANTS.items()}
throughput = {name: round(output_tokens / v["latency_s"], 2) for name, v in VARIANTS.items()}
speedup = {name: round(base_latency / v["latency_s"], 2) for name, v in VARIANTS.items()}
quality = {name: round(quality_rate(v), 2) for name, v in VARIANTS.items()}
quality_drop = {name: round(base_quality - quality_rate(v), 2) for name, v in VARIANTS.items()}

print("kv_short_gib=", round(kv_gib(short_seq), 2))
print("kv_long_gib=", round(kv_gib(long_seq), 2))
print("weight_gib=", weight)
print("compression_vs_fp16=", compression)
print("total_short_gib=", short_total)
print("total_long_gib=", long_total)
print("short_context_fits_16g=", fits_short)
print("long_context_fits_16g=", fits_long)
print("tokens_per_s=", throughput)
print("latency_speedup_vs_fp16=", speedup)
print("quality_pass_rate=", quality)
print("quality_drop_vs_fp16=", quality_drop)
print("kv_long_gt_nf4_weight=", kv_gib(long_seq) > weight_gib(VARIANTS["nf4"]))
```

一次稳定输出如下：

```text
kv_short_gib= 2.0
kv_long_gib= 16.0
weight_gib= {'fp16': 13.04, 'int8': 6.91, 'nf4': 3.65}
compression_vs_fp16= {'fp16': 1.0, 'int8': 1.89, 'nf4': 3.57}
total_short_gib= {'fp16': 17.34, 'int8': 11.21, 'nf4': 7.95}
total_long_gib= {'fp16': 31.34, 'int8': 25.21, 'nf4': 21.95}
short_context_fits_16g= {'fp16': False, 'int8': True, 'nf4': True}
long_context_fits_16g= {'fp16': False, 'int8': False, 'nf4': False}
tokens_per_s= {'fp16': 60.95, 'int8': 55.65, 'nf4': 49.23}
latency_speedup_vs_fp16= {'fp16': 1.0, 'int8': 0.91, 'nf4': 0.81}
quality_pass_rate= {'fp16': 1.0, 'int8': 1.0, 'nf4': 0.8}
quality_drop_vs_fp16= {'fp16': 0.0, 'int8': 0.0, 'nf4': 0.2}
```

这个例子故意让短上下文下 8bit / 4bit 能放进 16GB，但长上下文下三者都放不进去。它说明量化能降低权重显存，但不能替代 KV Cache 管理。

### 十八、常见工程坑

#### 坑 1：以为量化一定加速

量化主要降低显存。
速度是否提升取决于 kernel 和硬件。

#### 坑 2：只比较显存，不比较质量

量化可能损失任务能力。
必须做输出质量评估。

#### 坑 3：bitsandbytes 环境不兼容

CUDA、PyTorch、bitsandbytes 版本不匹配会导致加载失败。

#### 坑 4：4bit compute dtype 选错

不支持 bf16 的 GPU 上使用 bf16 可能出问题。

#### 坑 5：忽略 KV Cache 显存

长上下文下 KV Cache 可能超过权重显存。

#### 坑 6：量化模型和 tokenizer 不匹配

量化不改变 tokenizer。
仍然必须使用原模型 tokenizer。

### 十九、面试怎么讲模型量化

如果面试官问“模型量化是什么”，可以这样回答：

```text
模型量化是把模型权重从 fp16/fp32 等高精度表示转换成 int8、int4、nf4 等低 bit 表示，从而降低显存和存储占用。推理时通常会在低精度权重和较高精度 compute dtype 之间做计算，具体速度收益取决于硬件和 kernel。
```

如果追问“量化主要省哪部分显存”，可以回答：

```text
weight-only quantization 主要节省模型权重显存，不直接节省 KV Cache。长上下文场景下，即使权重量化了，KV Cache 仍然可能成为显存瓶颈。
```

如果问“8bit 和 4bit 怎么选”，可以回答：

```text
8bit 通常质量更稳、损失更小，适合对效果要求较高且显存有一定余量的场景；4bit 显存更省，适合显存紧张或 QLoRA 微调，但更需要做任务级质量评估。
```

如果问“怎么评估量化效果”，可以回答：

```text
我会比较量化前后的显存占用、prefill/decode 延迟、tokens/s，以及固定评测 prompt 上的输出质量。不能只看显存，因为量化可能影响数学、代码、长上下文和格式遵循等能力。
```

### 本讲总结

这一讲量化模型并比较了效果。

核心结论如下：

1. 量化用更低 bit 表示模型权重，降低显存和存储占用。
2. 8bit 通常更稳，4bit 更省显存。
3. bitsandbytes 可以方便加载 8bit 和 4bit 模型。
4. 量化主要节省权重显存，不直接解决 KV Cache 显存问题。
5. 量化不一定加速，速度取决于硬件和 kernel。
6. 评估量化要同时看显存、延迟、吞吐和输出质量。
7. 低 bit 量化可能影响数学、代码、长上下文和格式遵循能力。
8. 量化后仍要估算 KV Cache，否则长上下文场景依然可能 OOM。

## 第 35 讲：实现简单 Speculative Decoding

### 本讲目标

学完本讲，你应该能做到七件事：

1. 理解 Speculative Decoding 为什么能加速生成。
2. 区分 draft model 和 target model。
3. 掌握“草稿生成 + 大模型验证”的基本流程。
4. 实现一个简化版 speculative decoding。
5. 理解接受率对加速效果的影响。
6. 能解释 speculative decoding 的适用场景和工程限制。
7. 用 0 依赖 toy demo 验证 target 调用次数减少的来源。

前面几讲我们讲了 KV Cache、延迟测量、vLLM 和量化。

这些都是推理优化的重要方向。

本讲讲另一个经典加速方法：Speculative Decoding。

它的核心想法是：

```text
用小模型先猜多个 token，再让大模型一次性验证这些 token。
```

如果小模型猜得准，大模型一次 forward 可以接受多个 token，从而减少大模型 decode 次数。

资料边界说明：

```text
本讲按 Speculative Decoding / Speculative Sampling 原论文、Hugging Face Transformers assisted generation 文档和 vLLM speculative decoding 文档核对。
正文手写实现是 greedy 教学版，用来理解 draft、verify、accept 和 target call reduction。
严格 speculative sampling 需要接受-拒绝采样来保持 target model 的采样分布，不能把教学版等同于完整论文算法。
```

---

### 一、自回归 decode 的瓶颈

普通自回归生成每次只生成一个 token：

```text
大模型 forward -> 得到 token 1
大模型 forward -> 得到 token 2
大模型 forward -> 得到 token 3
...
```

即使有 KV Cache，每个 token 仍然需要一次大模型 forward。

大模型很大，decode 阶段串行，延迟高。

Speculative Decoding 试图减少大模型 forward 次数。

---

### 二、Speculative Decoding 的核心思想

准备两个模型：

```text
draft model：小模型，速度快，用来猜 token。
target model：大模型，质量高，用来验证 token。
```

流程：

```text
1. draft model 连续生成 k 个候选 token。
2. target model 一次性计算这些候选 token 的概率。
3. 按规则接受 draft token。
4. 如果接受多个 token，就等于少跑了多次 target decode。
```

直观理解：

```text
小模型先打草稿，大模型批改。如果草稿正确，大模型一次批改通过多个字。
```

---

### 三、为什么不能直接用小模型输出

小模型快，但质量不如大模型。

如果直接用小模型生成，输出分布会变。

Speculative Decoding 的目标是：

```text
尽量保持 target model 的生成分布，同时减少 target model 调用次数。
```

严格 speculative sampling 有一套接受-拒绝采样公式。

本讲先实现一个教学版：

```text
draft 生成多个 token，target 用贪心方式验证，连续匹配就接受。
```

这个版本不完全等价于严格采样分布。

但非常适合理解工程直觉。

---

### 四、简化版接受规则

假设 draft 生成：

```text
[a, b, c, d]
```

target 对同一上下文验证时，逐位置贪心得到：

```text
[a, b, x, ...]
```

那么：

```text
a 接受
b 接受
c 拒绝
使用 target 的 x 作为下一个 token
```

也就是说，连续匹配多少个，就接受多少个。

第一次不匹配时停止。

这种简化规则适合 greedy decoding。

用公式表示，设 draft token 为 `d_1,...,d_k`，target 贪心验证 token 为 `t_1,...,t_k`。教学版接受长度为：

```math
A
=
\max\{a: d_j=t_j,\ 1\le j\le a\}
```

如果 `A<k`，则第 `A+1` 个位置使用 target token `t_{A+1}` 作为回退 token。这个规则能保持 greedy target 路径一致，但不是严格采样分布保持算法。

### 六、普通 greedy 生成函数

先写普通大模型 greedy decode。

```python
@torch.no_grad()
def greedy_generate(model, input_ids, max_new_tokens):
    ids = input_ids
    for _ in range(max_new_tokens):
        outputs = model(ids)
        next_logits = outputs.logits[:, -1, :]
        next_token = torch.argmax(next_logits, dim=-1, keepdim=True)
        ids = torch.cat([ids, next_token], dim=1)
    return ids
```

这个版本没有手动 KV Cache。

为了教学清晰，先保持简单。

真实部署中 speculative decoding 会结合 KV Cache。

### 七、draft 生成 k 个 token

```python
@torch.no_grad()
def draft_generate(draft_model, input_ids, num_draft_tokens):
    ids = input_ids
    drafted = []

    for _ in range(num_draft_tokens):
        outputs = draft_model(ids)
        next_logits = outputs.logits[:, -1, :]
        next_token = torch.argmax(next_logits, dim=-1, keepdim=True)
        ids = torch.cat([ids, next_token], dim=1)
        drafted.append(next_token)

    return torch.cat(drafted, dim=1)
```

输出 shape：

```text
[B, num_draft_tokens]
```

---

### 八、target 一次性验证 draft tokens

如果原始上下文是：

```text
context
```

draft tokens 是：

```text
d1, d2, d3
```

拼接成：

```python
candidate = torch.cat([context, draft_tokens], dim=1)
```

target 对 candidate 做一次 forward。

然后我们取每个 draft 位置对应的预测。

```python
@torch.no_grad()
def target_verify(target_model, context_ids, draft_tokens):
    candidate = torch.cat([context_ids, draft_tokens], dim=1)
    outputs = target_model(candidate)
    logits = outputs.logits

    context_len = context_ids.size(1)
    num_draft = draft_tokens.size(1)

    verify_logits = logits[:, context_len - 1:context_len - 1 + num_draft, :]
    target_tokens = torch.argmax(verify_logits, dim=-1)
    return target_tokens
```

为什么从 `context_len - 1` 开始？

因为 causal LM 的 logits 第 `t` 个位置预测第 `t+1` 个 token。

要验证第一个 draft token，就看 context 最后一个位置的 logits。

### 九、接受连续匹配 token

```python
def accept_tokens(draft_tokens, target_tokens):
    # draft_tokens: [B, K]
    # target_tokens: [B, K]
    assert draft_tokens.size(0) == 1, "教学版本只处理 batch=1"

    accepted = []
    rejected_token = None

    for i in range(draft_tokens.size(1)):
        draft_token = draft_tokens[0, i].item()
        target_token = target_tokens[0, i].item()

        if draft_token == target_token:
            accepted.append(draft_token)
        else:
            rejected_token = target_token
            break

    if rejected_token is None:
        return accepted, None
    return accepted, rejected_token
```

如果全部 draft tokens 都被接受，下一轮继续从新上下文开始。

如果中途拒绝，则把 target token 加入输出。

### 十、简化版 speculative generate

```python
@torch.no_grad()
def speculative_generate(
    draft_model,
    target_model,
    input_ids,
    max_new_tokens,
    num_draft_tokens=4,
):
    ids = input_ids
    generated = 0
    target_calls = 0
    drafted_count = 0
    accepted_count = 0

    while generated < max_new_tokens:
        draft_tokens = draft_generate(
            draft_model,
            ids,
            num_draft_tokens=min(num_draft_tokens, max_new_tokens - generated),
        )

        target_tokens = target_verify(target_model, ids, draft_tokens)
        target_calls += 1
        drafted_count += draft_tokens.size(1)

        accepted, rejected_token = accept_tokens(draft_tokens, target_tokens)

        if accepted:
            accepted_tensor = torch.tensor([accepted], device=ids.device, dtype=ids.dtype)
            ids = torch.cat([ids, accepted_tensor], dim=1)
            generated += len(accepted)
            accepted_count += len(accepted)

        if generated >= max_new_tokens:
            break

        if rejected_token is not None:
            rejected_tensor = torch.tensor([[rejected_token]], device=ids.device, dtype=ids.dtype)
            ids = torch.cat([ids, rejected_tensor], dim=1)
            generated += 1
        elif len(accepted) == draft_tokens.size(1):
            # 教学简化：全部接受时不额外采 target token，直接进入下一轮。
            continue

    stats = {
        "target_calls": target_calls,
        "drafted_count": drafted_count,
        "accepted_count": accepted_count,
        "acceptance_rate": accepted_count / max(drafted_count, 1),
        "tokens_per_target_call": generated / max(target_calls, 1),
    }
    return ids, stats
```

这个版本强调机制，不追求最优性能。

真实系统会使用 KV Cache、批量验证和严格采样接受规则。


如果普通 target greedy 生成 32 个 token 需要 32 次 target forward。

Speculative 只调用了 12 次 target，就有加速潜力。

### 十二、接受率为什么重要

Speculative Decoding 的加速效果取决于 acceptance rate。

本章把接受率定义为：

```math
r_{\mathrm{acc}}
=
\frac{N_{\mathrm{accepted}}}{N_{\mathrm{drafted}}}
```

其中 `N_accepted` 是被 target 接受的 draft token 数，`N_drafted` 是 draft model 提出的候选 token 数。

普通 greedy target 生成 `T_out` 个 token，需要的 target 调用次数是：

```math
C_{\mathrm{base}}=T_{\mathrm{out}}
```

speculative decoding 的 target 调用次数记为 `C_spec`，target 调用减少倍数是：

```math
G_{\mathrm{call}}
=
\frac{C_{\mathrm{base}}}{C_{\mathrm{spec}}}
```

但真实延迟加速还要扣除 draft model 成本和 target 批量验证成本：

```math
S_{\mathrm{latency}}
=
\frac{\tau_{\mathrm{base}}}{\tau_{\mathrm{spec}}}
```

所以 `G_call` 变大不等于端到端延迟一定按同样比例下降。

如果 draft model 经常猜对：

```text
一次 target forward 接受多个 token。
target 调用次数减少。
加速明显。
```

如果 draft model 经常猜错：

```text
每次只能接受 0 或 1 个 token。
还额外运行了 draft model。
可能不加速，甚至变慢。
```

所以 draft model 要满足：

```text
足够快。
和 target 分布足够接近。
使用同一 tokenizer。
```

---

### 十三、num_draft_tokens 怎么选

`num_draft_tokens` 越大，一次猜得越多。

优点：

```text
如果接受率高，可能减少更多 target 调用。
```

缺点：

```text
draft 成本增加。
后面 token 猜错概率更高。
target 验证序列更长。
```

常见做法是实验调参。

例如测试：

```text
k = 2, 4, 8
```

记录：

```text
target_calls
acceptance_rate
total_latency
```

---

### 十四、严格 Speculative Sampling 简介

本讲的简化版使用 greedy 匹配。

严格 speculative sampling 会比较 draft 分布 `q` 和 target 分布 `p`。

对 draft token `x`，接受概率大致和：

```math
a(x)
=
\min\left(1,\frac{p(x)}{q(x)}\right)
```

有关。

如果拒绝，会从修正后的分布中采样。

一种常见写法是从残差分布采样：

```math
r_{\mathrm{rej}}(x)
=
\frac{\max(p(x)-q(x),0)}
{\sum_{z\in\mathcal{V}}\max(p(z)-q(z),0)}
```

它的目标是保证最终采样分布和 target model 一致。

这个推导更复杂，属于进阶内容。

面试中先讲清 draft/target/verify/accept 的机制已经很重要。

---

### 十五、Speculative Decoding 和 KV Cache

真实 speculative decoding 必须结合 KV Cache。

原因：

```text
draft model 生成多个 token 时需要 cache。
target model 验证 candidate tokens 时也需要 cache。
接受部分 token 后，需要正确更新 target cache。
拒绝时，需要丢弃未接受部分 cache。
```

这比教学版复杂很多。

推理框架通常会在底层处理这些 cache 状态。

手写时最容易错的是：

```text
cache 长度和实际接受 token 数不一致。
```

### 十六、0 依赖 toy speculative decoding demo

本机没有 `transformers` 时，可以用一个固定 token 表模拟 greedy target 和 draft。

这个 demo 验证三件事：

1. 简化 speculative 结果和 target greedy 结果一致。
2. target 调用次数从 8 次降到 3 次。
3. 接受率高时，每次 target call 平均产出多个 token。

```python
TARGET_NEXT = {
    "<s>": "A",
    "A": "B",
    "B": "C",
    "C": "D",
    "D": "E",
    "E": "F",
    "F": "G",
    "G": "H",
    "H": "I",
    "I": "J",
}
DRAFT_NEXT = {
    "<s>": "A",
    "A": "B",
    "B": "X",
    "C": "D",
    "D": "E",
    "E": "Y",
    "F": "G",
    "G": "H",
    "H": "I",
    "I": "J",
}


def next_from(table, ids):
    return table[ids[-1]]


def greedy_target(ids, max_new_tokens):
    out = list(ids)
    calls = 0
    for _ in range(max_new_tokens):
        out.append(next_from(TARGET_NEXT, out))
        calls += 1
    return out, calls


def draft_generate(ids, num_draft_tokens):
    out = list(ids)
    drafted = []
    for _ in range(num_draft_tokens):
        token = next_from(DRAFT_NEXT, out)
        drafted.append(token)
        out.append(token)
    return drafted


def target_verify(context, draft_tokens):
    candidate = list(context) + list(draft_tokens)
    preds = []
    for i in range(len(draft_tokens)):
        prefix = candidate[: len(context) + i]
        preds.append(next_from(TARGET_NEXT, prefix))
    return preds


def accept_tokens(draft_tokens, target_tokens):
    accepted = []
    fallback = None
    for draft_token, target_token in zip(draft_tokens, target_tokens):
        if draft_token == target_token:
            accepted.append(draft_token)
        else:
            fallback = target_token
            break
    return accepted, fallback


def speculative_generate(ids, max_new_tokens, num_draft_tokens):
    out = list(ids)
    generated = 0
    target_calls = 0
    draft_calls = 0
    drafted_count = 0
    accepted_count = 0
    trace = []

    while generated < max_new_tokens:
        k = min(num_draft_tokens, max_new_tokens - generated)
        draft_tokens = draft_generate(out, k)
        target_tokens = target_verify(out, draft_tokens)
        accepted, fallback = accept_tokens(draft_tokens, target_tokens)

        target_calls += 1
        draft_calls += len(draft_tokens)
        drafted_count += len(draft_tokens)
        accepted_count += len(accepted)

        out.extend(accepted)
        generated += len(accepted)

        if generated < max_new_tokens and fallback is not None:
            out.append(fallback)
            generated += 1

        trace.append({
            "draft": draft_tokens,
            "target": target_tokens,
            "accepted": accepted,
            "fallback": fallback,
        })

    return out, {
        "target_calls": target_calls,
        "draft_calls": draft_calls,
        "drafted_count": drafted_count,
        "accepted_count": accepted_count,
        "acceptance_rate": round(accepted_count / drafted_count, 2),
        "tokens_per_target_call": round(generated / target_calls, 2),
        "target_call_reduction": round(max_new_tokens / target_calls, 2),
        "trace": trace,
    }


prompt = ["<s>"]
max_new_tokens = 8
greedy_ids, greedy_calls = greedy_target(prompt, max_new_tokens)
spec_ids, stats = speculative_generate(prompt, max_new_tokens, num_draft_tokens=3)

print("greedy_tokens=", " ".join(greedy_ids[1:]))
print("spec_tokens=", " ".join(spec_ids[1:]))
print("match_greedy=", spec_ids == greedy_ids)
print("target_calls_greedy=", greedy_calls)
print("target_calls_spec=", stats["target_calls"])
print("draft_calls_spec=", stats["draft_calls"])
print("drafted_count=", stats["drafted_count"])
print("accepted_count=", stats["accepted_count"])
print("acceptance_rate=", stats["acceptance_rate"])
print("tokens_per_target_call=", stats["tokens_per_target_call"])
print("target_call_reduction=", stats["target_call_reduction"])
print("trace=", stats["trace"])
```

一次稳定输出如下：

```text

spec_tokens= A B C D E F G H
match_greedy= True
target_calls_greedy= 8
target_calls_spec= 3
draft_calls_spec= 8
drafted_count= 8
accepted_count= 6
acceptance_rate= 0.75
tokens_per_target_call= 2.67
target_call_reduction= 2.67
trace= [{'draft': ['A', 'B', 'X'], 'target': ['A', 'B', 'C'], 'accepted': ['A', 'B'], 'fallback': 'C'}, {'draft': ['D', 'E', 'Y'], 'target': ['D', 'E', 'F'], 'accepted': ['D', 'E'], 'fallback': 'F'}, {'draft': ['G', 'H'], 'target': ['G', 'H'], 'accepted': ['G', 'H'], 'fallback': None}]
```

注意：这个 demo 模拟的是 greedy 验证，不是严格 speculative sampling。严格采样还要处理概率分布和随机接受。

### 十七、常见工程坑

#### 坑 1：draft 和 target tokenizer 不一致

token id 不一致会导致验证无意义。

#### 坑 2：draft 太弱

接受率低，无法加速。

#### 坑 3：draft 太大

draft 本身成本太高，抵消加速收益。

#### 坑 4：忽略 target 验证成本

target 一次验证多个 token 也有成本，不是免费。

#### 坑 5：cache 状态更新错

真实实现中接受几个 token，就只能保留对应 cache。

#### 坑 6：把简化 greedy 版本当严格采样版本

本讲实现适合理解机制，不保证采样分布等价。

---

### 十八、面试怎么讲 Speculative Decoding

如果面试官问“Speculative Decoding 是什么”，可以这样回答：

```text
Speculative Decoding 是一种加速自回归生成的方法。它使用一个较小较快的 draft model 先生成多个候选 token，再让大的 target model 一次性验证这些 token。如果候选 token 被接受，就能用一次 target forward 生成多个 token，从而减少大模型 decode 调用次数。
```

如果追问“为什么能保证质量”，可以回答：

```text
严格 speculative sampling 会根据 target 分布和 draft 分布设计接受-拒绝规则，目标是让最终输出分布与 target model 保持一致。工程上也有 greedy 验证的简化版本，用于确定性生成场景。
```

如果问“什么时候加速明显”，可以回答：

```text
当 draft model 足够快，并且生成分布和 target model 足够接近时，接受率高，一次 target 验证能通过多个 token，加速明显。如果 draft 太弱或太大，可能不加速甚至变慢。
```

如果问“实现难点是什么”，可以回答：

```text
主要难点是接受-拒绝采样的正确性、draft 和 target 的 tokenizer 一致性、KV Cache 的状态管理，以及如何选择 draft size 和 num_draft_tokens 来平衡接受率和额外计算成本。
```

### 本讲总结

这一讲实现了一个简化版 Speculative Decoding。

核心结论如下：

1. Speculative Decoding 用小 draft model 猜 token，大 target model 验证 token。
2. 如果 draft 猜得准，一次 target forward 可以接受多个 token。
3. 加速效果主要取决于 draft 成本和 acceptance rate。
4. draft 和 target 必须共享 tokenizer。
5. 严格 speculative sampling 需要接受-拒绝规则以保持 target 分布。
6. 真实实现必须正确管理 KV Cache。
7. Speculative Decoding 适合 decode 阶段瓶颈明显的场景。
8. greedy 教学版可以验证机制，但不能替代严格采样正确性证明。

### 一、为什么长上下文推理难

假设 prompt 长度从 1K 增加到 32K。

影响包括：

```text
Prefill 时间显著增加。
KV Cache 显存线性增加。
Decode 每步需要 attend 到更长历史。
请求更容易 OOM。
batching 更难。
```

长上下文不是简单把 `max_length` 调大。

它是系统、模型和显存共同约束的问题。

本讲先做单请求和简单批量压测。

压测长度集合可以写成：

```math
\mathcal{L}
=
\{\ell_1,\ell_2,\ldots,\ell_n\}
```

每个请求要满足模型长度边界：

```math
\ell_i+T_{\mathrm{out}}\le L_{\mathrm{max}}
```

其中 `ell_i` 是 prompt token 数，`T_out` 是固定生成 token 数，`L_max` 是模型或服务配置允许的最大长度。

模型侧 TTFT 可以近似写成：

```math
t_{\mathrm{TTFT,model}}
=
t_{\mathrm{prefill}}+t_{\mathrm{decode},1}
```

后续平均 TPOT 写成：

```math
t_{\mathrm{TPOT}}
=
\frac{1}{T_{\mathrm{decode}}}
\sum_{j=1}^{T_{\mathrm{decode}}}t_{\mathrm{decode},j}
```

prefill 和 decode 吞吐分别是：

```math
R_{\mathrm{prefill}}
=
\frac{T_{\mathrm{prompt}}}{t_{\mathrm{prefill}}},
\qquad
R_{\mathrm{decode}}
=
\frac{T_{\mathrm{decode}}}{\sum_{j=1}^{T_{\mathrm{decode}}}t_{\mathrm{decode},j}}
```

```python
def make_prompt(tokenizer, target_tokens):
```

```python
def benchmark_one(prompt, max_new_tokens=32):

    prompt_len = inputs["input_ids"].shape[1]
    pad_token_id = tokenizer.pad_token_id

    start = now()
    outputs = model.generate(

        max_new_tokens=max_new_tokens,
        do_sample=False,
        use_cache=True,
        pad_token_id=pad_token_id,

    peak_memory = get_peak_memory_gb()

    return {

        "max_new_tokens": max_new_tokens,
        "total_latency_s": total_latency,
        "rough_tokens_per_s": max_new_tokens / total_latency,
        "peak_memory_gb": peak_memory,
        "output_preview": tokenizer.decode(outputs[0][-max_new_tokens:], skip_special_tokens=True)[:200],
    }
```

### 八、分离 prefill 和 decode

如果要更精确，可以分两步测。

Prefill：

```python
@torch.no_grad()
def measure_prefill(prompt):
    inputs = tokenizer(prompt, return_tensors="pt").to(input_device(model))
    start = now()
    outputs = model(**inputs, use_cache=True)
    end = now()
    return outputs, end - start, inputs
```

Decode 单步：

```python
@torch.no_grad()
def measure_decode(outputs, inputs, steps=32):
    past_key_values = outputs.past_key_values
    next_token = torch.argmax(outputs.logits[:, -1, :], dim=-1, keepdim=True)
    attention_mask = inputs["attention_mask"]

    decode_times = []

    for _ in range(steps):
        attention_mask = torch.cat(
            [attention_mask, torch.ones((attention_mask.size(0), 1), device=attention_mask.device, dtype=attention_mask.dtype)],
            dim=1,
        )
        start = now()
        outputs = model(
            input_ids=next_token,
            attention_mask=attention_mask,
            past_key_values=past_key_values,
            use_cache=True,
        )
        end = now()
        decode_times.append(end - start)

        past_key_values = outputs.past_key_values
        next_token = torch.argmax(outputs.logits[:, -1, :], dim=-1, keepdim=True)

    return decode_times
```

```text
avg_decode_latency
```

### 九、KV Cache 显存估算

KV Cache 大小约为：

```math
M_{\mathrm{KV}}
\approx
2LH_{\mathrm{kv}}D_hB_{\mathrm{batch}}T_{\mathrm{ctx}}b_{\mathrm{kv}}
```

例如：

```text
layers = 32
kv_heads = 32
head_dim = 128
batch = 1
seq_len = 8192
bytes = 2  # fp16
```

计算：

```python
def estimate_kv_cache_gb(
    layers,
    kv_heads,
    head_dim,
    batch,
    seq_len,
    bytes_per_element=2,
):
    total_bytes = (
        2
        * layers
        * kv_heads
        * head_dim
        * batch
        * seq_len
        * bytes_per_element
    )
    return total_bytes / 1024**3


print(estimate_kv_cache_gb(32, 32, 128, 1, 8192))
```

这能帮助你在压测前估计是否可能 OOM。

如果是 MHA，通常 `H_kv * D_h` 等于 hidden size。如果是 GQA/MQA，`H_kv` 小于 query heads，KV Cache 会更省。

### 十五、常见工程坑

#### 坑 1：超过模型最大上下文

模型可能报错，或者位置编码外推效果很差。

#### 坑 2：只看 total latency

应拆分 prefill 和 decode，否则不知道瓶颈在哪。

#### 坑 3：忽略 KV Cache 显存

长上下文下 KV Cache 可能比权重还大。

#### 坑 4：压测没有固定生成长度

输出长度不同会让 latency 不可比。

#### 坑 5：没有记录 OOM 点

OOM 点是容量规划的重要信息。

#### 坑 6：只测性能不测质量

长上下文能跑，不代表模型能用远处信息。

---

### 十六、面试怎么讲长上下文压测

如果面试官问“怎么做长上下文推理压测”，可以这样回答：

```text
我会构造不同 token 长度的 prompt，例如 1K、2K、4K、8K、16K，固定生成长度，分别测 prefill latency、decode per-token latency、总延迟、峰值显存和是否 OOM。GPU 计时要 synchronize，并做 warmup 和重复测量。服务端还要测并发下的 P50/P90/P99 延迟、吞吐和错误率。
```

如果追问“长上下文主要瓶颈是什么”，可以回答：

```text
长上下文会增加 prefill 计算成本，并线性增加 KV Cache 显存；decode 阶段每步需要访问更长的 K/V cache，也会增加显存带宽压力。因此瓶颈既包括计算，也包括 KV Cache 显存和带宽。
```

如果问“怎么优化长上下文推理”，可以回答：

```text
可以使用 vLLM 的 PagedAttention 管理 KV Cache，降低 max_model_len 或使用 sliding window，做模型量化降低权重显存，使用 FlashAttention 或 chunked prefill 优化 prefill，必要时进行上下文压缩或检索增强，避免把所有信息都塞进上下文。
```

### 本讲总结

这一讲完成了长上下文推理压测。

核心结论如下：

1. 长上下文会显著增加 prefill latency。
2. KV Cache 显存随上下文长度线性增长。
3. Decode 阶段也会受长 KV Cache 读写影响。
4. 压测应记录 prompt_len、生成长度、latency、tokens/s、峰值显存和 OOM 点。
5. 服务端压测还要关注并发、P50/P90/P99、QPS 和错误率。
6. 长上下文不仅要测性能，也要测模型是否能利用远处信息。
7. vLLM、量化、PagedAttention、上下文压缩和 RAG 都是长上下文优化方向。
8. 没有真实 GPU 时，也可以先用固定 trace 验证报告统计和 OOM 点记录逻辑。
