### 4. 字符级 Tokenizer

字符级 tokenizer 简单、透明、适合教学和小项目。

```python
class CharTokenizer:
    def __init__(self, text: str):
        chars = sorted(list(set(text)))
        self.stoi = {ch: i for i, ch in enumerate(chars)}
        self.itos = {i: ch for ch, i in self.stoi.items()}

    @property
    def vocab_size(self) -> int:
        return len(self.stoi)

    def encode(self, text: str) -> list[int]:
        return [self.stoi[ch] for ch in text if ch in self.stoi]

    def decode(self, ids: list[int]) -> str:
        return "".join(self.itos[i] for i in ids)
```

这个 tokenizer 有明显局限：

1. 不支持未登录字符。
2. 中文、英文、标点都按字符切。
3. 序列会比 BPE 更长。

但它有一个优点：足够简单，方便把重点放在 GPT 结构和训练目标上。

面试中可以主动说明：

```text
这个项目第一版使用字符级 tokenizer 是为了降低依赖并突出自回归训练流程。后续可以替换为 BPE、SentencePiece 或 tiktoken 风格 tokenizer。
```

### 5. Dataset 设计

自回归语言模型的训练样本非常简单。

给定 token 序列：

```text
x = [t0, t1, t2, ..., t_{n-1}]
y = [t1, t2, t3, ..., t_n]
```

模型看到前面的 token，预测下一个 token。

```python
import torch
from torch.utils.data import Dataset


class TextDataset(Dataset):
    def __init__(self, token_ids: list[int], block_size: int):
        self.data = torch.tensor(token_ids, dtype=torch.long)
        self.block_size = block_size

    def __len__(self) -> int:
        return len(self.data) - self.block_size

    def __getitem__(self, idx: int):
        chunk = self.data[idx : idx + self.block_size + 1]
        x = chunk[:-1]
        y = chunk[1:]
        return x, y
```

这里 `block_size` 是上下文长度。

如果 `block_size=128`，模型最多看到前 128 个 token。

### 6. GPT 模型结构

miniGPT 的核心结构是：

```text
token embedding
position embedding
N 层 Transformer Block
LayerNorm
LM Head
```

每个 Transformer Block 包含：

```text
LayerNorm
Causal Self-Attention
Residual Connection
LayerNorm
MLP
Residual Connection
```

下面是一份最小实现骨架。

```python
import math
import torch
import torch.nn as nn
import torch.nn.functional as F


class CausalSelfAttention(nn.Module):
    def __init__(self, n_embd: int, n_head: int, block_size: int, dropout: float):
        super().__init__()
        assert n_embd % n_head == 0
        self.n_head = n_head
        self.head_dim = n_embd // n_head
        self.qkv = nn.Linear(n_embd, 3 * n_embd)
        self.proj = nn.Linear(n_embd, n_embd)
        self.dropout = nn.Dropout(dropout)
        mask = torch.tril(torch.ones(block_size, block_size)).view(1, 1, block_size, block_size)
        self.register_buffer("mask", mask)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        bsz, seq_len, n_embd = x.shape
        qkv = self.qkv(x)
        q, k, v = qkv.chunk(3, dim=-1)

        q = q.view(bsz, seq_len, self.n_head, self.head_dim).transpose(1, 2)
        k = k.view(bsz, seq_len, self.n_head, self.head_dim).transpose(1, 2)
        v = v.view(bsz, seq_len, self.n_head, self.head_dim).transpose(1, 2)

        att = (q @ k.transpose(-2, -1)) / math.sqrt(self.head_dim)
        att = att.masked_fill(self.mask[:, :, :seq_len, :seq_len] == 0, float("-inf"))
        att = F.softmax(att, dim=-1)
        att = self.dropout(att)

        out = att @ v
        out = out.transpose(1, 2).contiguous().view(bsz, seq_len, n_embd)
        return self.proj(out)


class Block(nn.Module):
    def __init__(self, n_embd: int, n_head: int, block_size: int, dropout: float):
        super().__init__()
        self.ln1 = nn.LayerNorm(n_embd)
        self.attn = CausalSelfAttention(n_embd, n_head, block_size, dropout)
        self.ln2 = nn.LayerNorm(n_embd)
        self.mlp = nn.Sequential(
            nn.Linear(n_embd, 4 * n_embd),
            nn.GELU(),
            nn.Linear(4 * n_embd, n_embd),
            nn.Dropout(dropout),
        )

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        x = x + self.attn(self.ln1(x))
        x = x + self.mlp(self.ln2(x))
        return x
```

注意这里使用的是 Pre-LN 结构。

Pre-LN 在训练稳定性上通常比原始 Post-LN 更友好，是现代 GPT 类模型中常见选择。

### 7. 完整 GPT 类

```python
class MiniGPT(nn.Module):
    def __init__(
        self,
        vocab_size: int,
        block_size: int,
        n_layer: int = 4,
        n_head: int = 4,
        n_embd: int = 256,
        dropout: float = 0.1,
    ):
        super().__init__()
        self.block_size = block_size
        self.token_emb = nn.Embedding(vocab_size, n_embd)
        self.pos_emb = nn.Embedding(block_size, n_embd)
        self.drop = nn.Dropout(dropout)
        self.blocks = nn.Sequential(
            *[Block(n_embd, n_head, block_size, dropout) for _ in range(n_layer)]
        )
        self.ln_f = nn.LayerNorm(n_embd)
        self.lm_head = nn.Linear(n_embd, vocab_size, bias=False)

    def forward(self, idx: torch.Tensor, targets: torch.Tensor | None = None):
        bsz, seq_len = idx.shape
        if seq_len > self.block_size:
            raise ValueError("sequence length exceeds block_size")

        pos = torch.arange(seq_len, device=idx.device)
        x = self.token_emb(idx) + self.pos_emb(pos)[None, :, :]
        x = self.drop(x)
        x = self.blocks(x)
        x = self.ln_f(x)
        logits = self.lm_head(x)

        loss = None
        if targets is not None:
            loss = F.cross_entropy(logits.view(-1, logits.size(-1)), targets.view(-1))
        return logits, loss
```

这段代码的关键点：

1. 输入 `idx` 形状是 `[batch, seq_len]`。
2. token embedding 和 position embedding 相加。
3. 输出 logits 形状是 `[batch, seq_len, vocab_size]`。
4. loss 是每个位置预测下一个 token 的交叉熵。

### 8. 自回归生成

生成时，每次取当前上下文，预测下一个 token，然后把新 token 拼回序列。

```python
@torch.no_grad()
def generate(
    model: MiniGPT,
    idx: torch.Tensor,
    max_new_tokens: int,
    temperature: float = 1.0,
    top_k: int | None = None,
) -> torch.Tensor:
    model.eval()

    for _ in range(max_new_tokens):
        idx_cond = idx[:, -model.block_size :]
        logits, _ = model(idx_cond)
        logits = logits[:, -1, :] / temperature

        if top_k is not None:
            values, _ = torch.topk(logits, top_k)
            logits[logits < values[:, [-1]]] = -float("inf")

        probs = F.softmax(logits, dim=-1)
        next_id = torch.multinomial(probs, num_samples=1)
        idx = torch.cat([idx, next_id], dim=1)

    return idx
```

这里 `idx[:, -model.block_size:]` 的作用是截断上下文，避免超过模型最大上下文长度。

### 9. 训练循环

简化训练循环如下：

```python
import torch
from torch.utils.data import DataLoader


def train(model, train_ds, val_ds, device: str = "cuda"):
    model.to(device)
    optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4, weight_decay=0.1)
    train_loader = DataLoader(train_ds, batch_size=32, shuffle=True, drop_last=True)
    val_loader = DataLoader(val_ds, batch_size=32, shuffle=False, drop_last=False)

    for epoch in range(10):
        model.train()
        for step, (x, y) in enumerate(train_loader):
            x = x.to(device)
            y = y.to(device)
            _, loss = model(x, y)

            optimizer.zero_grad(set_to_none=True)
            loss.backward()
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            optimizer.step()

            if step % 100 == 0:
                print(f"epoch={epoch} step={step} train_loss={loss.item():.4f}")

        val_loss = evaluate(model, val_loader, device)
        print(f"epoch={epoch} val_loss={val_loss:.4f}")
        torch.save(model.state_dict(), f"checkpoint_epoch_{epoch}.pt")


@torch.no_grad()
def evaluate(model, loader, device: str) -> float:
    model.eval()
    losses = []
    for x, y in loader:
        x = x.to(device)
        y = y.to(device)
        _, loss = model(x, y)
        losses.append(loss.item())
    return sum(losses) / len(losses)
```

### 13. 面试怎么讲

面试官可能问：你这个 miniGPT 和真正 GPT 有什么区别？

可以这样答：

```text
我的 miniGPT 主要用于验证 Decoder-only Transformer 的核心原理和训练流程。它和真实 GPT 在规模、数据、tokenizer、训练工程和对齐阶段上都有很大差距。真实 GPT 使用大规模 BPE tokenizer、海量语料、多机多卡训练、混合精度、并行策略、复杂数据清洗和后续 SFT/RLHF/DPO 对齐。我这个项目强调从零实现 attention、causal mask、next-token prediction、训练循环和生成策略，目的是证明我理解底层机制，并能把理论转成可运行代码。
```

面试官可能问：为什么用 causal mask？

可以这样答：

```text
自回归语言模型在第 t 个位置只能看见 t 以及之前的 token，不能看到未来 token。Causal mask 会把 attention score 中未来位置置为负无穷，softmax 后这些位置权重为 0，从而保证训练和推理目标一致。如果不加 causal mask，模型训练时会偷看答案，loss 会虚低，但生成时无法使用未来信息，效果会崩。
```

面试官可能问：为什么验证 loss 降低，但生成效果仍然一般？

可以这样答：

```text
loss 衡量的是平均 next-token prediction 能力，但生成质量还受采样策略、训练数据规模、模型容量、tokenizer 粒度和长程依赖能力影响。小语料和小模型即使 loss 下降，也可能只学到局部模式，生成时容易重复或语义漂移。因此我会同时看 val loss、perplexity、生成样例和 bad case。
```

### 15. 常见工程坑

#### 15.1 忘记 causal mask

训练 loss 可能很低，但生成完全不可靠。

#### 15.2 target 没有右移

如果输入和目标完全相同，模型学的是复制当前位置，而不是预测下一个 token。

#### 15.3 position embedding 长度不够

生成或训练序列超过 `block_size` 会报错。

#### 15.4 训练集和验证集泄漏

如果验证集来自训练文本的重叠片段，val loss 会虚高可信度低。

#### 15.5 没有保存配置

只保存 checkpoint，不保存模型参数和 tokenizer，后续无法恢复生成。

#### 15.6 夸大项目效果

miniGPT 项目价值在于机制和工程闭环，不在于声称达到商用效果。

### 16. 本讲小结

这一讲设计了一个可以写进简历的 miniGPT 项目。

核心结论如下：

1. miniGPT 的定位是验证 GPT 原理和工程闭环，不是训练商用大模型。
2. 项目应包含 tokenizer、dataset、model、train、generate、config、README 和实验记录。
3. 最小版本可以使用字符级 tokenizer，先跑通 next-token prediction。
4. GPT 结构包括 token embedding、position embedding、Causal Self-Attention、MLP、LayerNorm 和 LM Head。
5. Causal mask 是自回归建模的关键，防止模型偷看未来 token。
6. 训练时要记录 train loss、val loss、perplexity、生成样例和 bad cases。
7. 简历表达要突出“从零实现、可运行、可复现、能分析问题”，不要夸大成自研商用大模型。
8. 面试中要能讲清 miniGPT 和真实 GPT 在规模、数据、训练工程和对齐上的差距。

## 第 52 讲：中文指令微调助手

### 本讲目标

### 4. SFT 数据格式

最简单的数据格式是 Alpaca 风格：

```json
{"instruction": "解释什么是 LoRA", "input": "", "output": "LoRA 是一种参数高效微调方法..."}
```

更贴近真实聊天模型的是 messages 格式：

```json
{"messages": [{"role": "system", "content": "你是一个严谨的中文 AI 助手。"}, {"role": "user", "content": "解释什么是 LoRA"}, {"role": "assistant", "content": "LoRA 是一种参数高效微调方法..."}]}
```

推荐使用 messages 格式，因为它和现代 chat model 的训练、推理更一致。

### 7. Chat Template 的重要性

同一条多轮消息必须被转换成模型认识的文本格式。

例如 Qwen 风格可能类似：

```text
<|im_start|>system
你是一个严谨的中文 AI 助手。<|im_end|>
<|im_start|>user
解释什么是 LoRA<|im_end|>
<|im_start|>assistant
LoRA 是一种参数高效微调方法...<|im_end|>
```

不要自己随便拼一个格式，除非你明确知道模型预训练和 SFT 使用的模板。

推荐使用 tokenizer 自带的：

```python
text = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=False,
)
```

推理时则通常使用：

```python
prompt = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True,
)
```

训练和推理模板不一致，是 SFT 项目里非常高频的坑。

### 8. Loss Mask

SFT 中通常只希望 assistant 的回答部分参与 loss。

原因是 user prompt 和 system prompt 是条件，不是模型要学习生成的目标。

如果把 user prompt 也纳入 loss，模型会被训练去复述用户问题，影响助手行为。

一个简化理解：

```text
system/user tokens -> labels = -100
assistant answer tokens -> labels = token_id
```

PyTorch 的 `CrossEntropyLoss` 会忽略 label 为 `-100` 的位置。

### 9. 简化版样本预处理

下面是一个教学版预处理函数。

真实项目中要结合具体模型的 special tokens 做更严格处理。

```python
def preprocess_example(example, tokenizer, max_length: int = 2048):
    messages = example["messages"]
    full_text = tokenizer.apply_chat_template(
        messages,
        tokenize=False,
        add_generation_prompt=False,
    )

    prompt_messages = messages[:-1]
    prompt_text = tokenizer.apply_chat_template(
        prompt_messages,
        tokenize=False,
        add_generation_prompt=True,
    )

    full = tokenizer(full_text, truncation=True, max_length=max_length, add_special_tokens=False)
    prompt = tokenizer(prompt_text, truncation=True, max_length=max_length, add_special_tokens=False)

    input_ids = full["input_ids"]
    labels = input_ids.copy()
    prompt_len = min(len(prompt["input_ids"]), len(labels))
    labels[:prompt_len] = [-100] * prompt_len

    return {
        "input_ids": input_ids,
        "attention_mask": full["attention_mask"],
        "labels": labels,
    }
```

这里的关键是：先构造完整训练文本，再构造 assistant 之前的 prompt 文本，用 prompt token 长度确定 mask 区间。

### 10. LoRA 配置

LoRA 通过给模型中的线性层加入低秩增量矩阵，实现参数高效微调。

典型配置：

```python
from peft import LoraConfig


lora_config = LoraConfig(
    r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"],
)
```

不同模型的 target module 名字不一样。

如果配置错了，可能出现：

```text
没有任何参数参与训练
只训练了很少模块
训练效果几乎没有变化
```

所以训练前必须打印可训练参数比例：

```python
model.print_trainable_parameters()
```

### 11. 训练脚本骨架

使用 Hugging Face Trainer 可以快速跑通。

```python
from datasets import load_dataset
from peft import get_peft_model, LoraConfig
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer
from transformers import DataCollatorForSeq2Seq


model_name = "Qwen/Qwen2.5-1.5B-Instruct"

tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto",
    device_map="auto",
    trust_remote_code=True,
)

lora_config = LoraConfig(
    r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"],
)
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()

dataset = load_dataset("json", data_files={"train": "data/train.jsonl", "dev": "data/dev.jsonl"})
dataset = dataset.map(lambda x: preprocess_example(x, tokenizer), remove_columns=dataset["train"].column_names)

args = TrainingArguments(
    output_dir="outputs/qwen-lora-sft",
    per_device_train_batch_size=2,
    per_device_eval_batch_size=2,
    gradient_accumulation_steps=8,
    learning_rate=2e-4,
    num_train_epochs=3,
    logging_steps=10,
    eval_strategy="steps",
    eval_steps=100,
    save_steps=100,
    save_total_limit=3,
    bf16=True,
    report_to="none",
)

trainer = Trainer(
    model=model,
    args=args,
    train_dataset=dataset["train"],
    eval_dataset=dataset["dev"],
    data_collator=DataCollatorForSeq2Seq(tokenizer=tokenizer, padding=True),
)

trainer.train()
trainer.save_model()
tokenizer.save_pretrained(args.output_dir)
```

注意：不同 transformers 版本中参数名可能不同，例如有的版本使用 `evaluation_strategy`，有的版本开始改为 `eval_strategy`。项目 README 中要固定依赖版本。

### 12. 推理对比脚本

项目必须比较 base 和 SFT 后模型。

```python
import torch
from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer


def load_model(base_model: str, lora_path: str | None = None):
    tokenizer = AutoTokenizer.from_pretrained(base_model, trust_remote_code=True)
    model = AutoModelForCausalLM.from_pretrained(
        base_model,
        torch_dtype="auto",
        device_map="auto",
        trust_remote_code=True,
    )
    if lora_path:
        model = PeftModel.from_pretrained(model, lora_path)
    model.eval()
    return tokenizer, model


@torch.no_grad()
def chat(tokenizer, model, query: str) -> str:
    messages = [
        {"role": "system", "content": "你是一个严谨的中文 AI 助手。"},
        {"role": "user", "content": query},
    ]
    prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
    inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
    output_ids = model.generate(
        **inputs,
        max_new_tokens=512,
        temperature=0.7,
        top_p=0.9,
        do_sample=True,
    )
    new_tokens = output_ids[0, inputs["input_ids"].shape[1] :]
    return tokenizer.decode(new_tokens, skip_special_tokens=True)
```

推理时要固定一组 eval prompts，输出 base 和 sft 的对比结果。

### 16. 面试高频问法

#### 问法 1：SFT 的训练目标是什么？

可以这样答：

```text
SFT 本质上还是自回归语言建模，但训练数据变成了指令和回答格式。给定 system 和 user 内容作为条件，模型学习生成 assistant 的回答。工程上通常只对 assistant answer 部分计算 loss，把 system/user 部分 label 设为 -100，避免模型学习复述用户输入。
```

#### 问法 2：为什么训练和推理 chat template 必须一致？

可以这样答：

```text
因为模型不是直接理解 role 列表，而是看到 tokenizer 后的 token 序列。如果训练时用一种 role 分隔符，推理时用另一种格式，模型看到的上下文分布会变化，可能导致回答风格异常、停止符错误或无法正确进入 assistant 生成状态。所以我会优先使用 tokenizer.apply_chat_template，并区分训练时 add_generation_prompt=False 和推理时 add_generation_prompt=True。
```

#### 问法 3：LoRA 的 target modules 怎么选？

可以这样答：

```text
通常会优先覆盖 attention 中的 q_proj、k_proj、v_proj、o_proj，以及 MLP 中的 gate_proj、up_proj、down_proj。只训 attention 参数更少、风险更低，但表达能力可能不够；attention 和 MLP 都训效果通常更强，但更容易过拟合或引入风格偏移。具体选择要结合任务复杂度、数据规模、显存和验证集表现，并且训练前要打印 trainable parameters 确认配置生效。
```

#### 问法 4：SFT 后模型变差怎么办？

可以这样答：

```text
我会先区分是目标任务没提升，还是目标任务提升但通用能力退化。排查顺序包括数据质量、数据分布、chat template、loss mask、学习率、训练轮数、LoRA rank 和 target modules。然后对比 base、不同 checkpoint 和不同数据配比，查看是否过拟合或灾难性遗忘。缓解方法包括清洗数据、混入通用 replay 数据、降低学习率、减少 epoch、缩小 LoRA 更新范围或做早停。
```

### 18. 常见工程坑

#### 18.1 把所有 token 都算 loss

模型可能学习复述 prompt，助手行为变差。

#### 18.2 chat template 错误

训练 loss 看似正常，但推理格式异常。

#### 18.3 LoRA target modules 配错

可训练参数过少或为 0，训练没有效果。

#### 18.4 学习率过大

可能导致模型风格剧烈变化，甚至能力退化。

#### 18.5 数据太模板化

模型输出变得机械，通用对话能力下降。

#### 18.6 只看训练 loss

训练 loss 下降不代表用户体验提升，必须看验证集、人工评估和 bad cases。

#### 18.7 没有固定依赖版本

Transformers、PEFT、TRL 版本变化较快，不固定版本会影响复现。

## 第 53 讲：DPO 偏好优化实验

SFT 让模型学会“怎么回答”。偏好优化进一步让模型学会“什么回答更好”。

在真实大模型训练链路中，SFT 之后通常还会做偏好对齐，例如 RLHF、DPO、IPO、KTO、ORPO 等。对于个人项目和面试作品集来说，DPO 是非常适合落地的一种方法，因为它不需要训练 reward model，也不需要复杂的 PPO 强化学习流程。

这一讲，我们设计一个 DPO 偏好优化实验项目。

项目目标不是训练出最强模型，而是证明你理解偏好数据、chosen/rejected 样本、reference model、DPO loss、偏好评估和对齐副作用。

### 本讲目标

学完本讲，你需要掌握：

1. DPO 项目的合理定位。
2. 如何构造偏好数据集。
3. chosen/rejected 数据格式怎么设计。
4. 如何使用 TRL 跑 DPOTrainer。
5. 如何理解 beta、reference model 和 DPO loss。
6. 如何评估 DPO 前后效果。
7. 如何分析 DPO 带来的副作用。
8. 面试中如何讲 DPO 项目。

### 1. 项目定位

一个适合简历的 DPO 项目可以这样定义：

```text
基于已完成 SFT 的中文助手模型，构造 chosen/rejected 偏好数据集，使用 DPO 进行偏好优化，使模型在回答完整性、格式遵循、拒答边界和用户偏好上更接近人工选择，并通过 pairwise 评测、长度分析、安全评测和 bad case 对比分析 DPO 效果与副作用。
```

这个项目展示的能力包括：

```text
我知道 DPO 解决什么问题。
我知道偏好数据如何构造。
我知道 DPO 和 SFT 的区别。
我知道 reference model 的作用。
我知道如何评估偏好优化是否真的变好。
我知道 DPO 可能导致回答变短、过度拒答或奖励黑客。
```

### 2. DPO 为什么适合个人项目

RLHF 标准流程通常包括：

```text
SFT model
偏好数据收集
Reward Model 训练
PPO 强化学习
安全评估和回归
```

这对个人项目来说比较重。

DPO 的优点是：

```text
不需要单独训练 reward model
不需要 PPO rollout
训练流程接近普通 supervised fine-tuning
可以直接使用 chosen/rejected 偏好对
适合单卡或小规模实验
```

所以简历项目中，DPO 是比完整 RLHF 更现实的选择。

```text
    dpo_eval_report.md
    bad_cases.md
```

项目要体现：数据构造、训练、推理对比、评估和复盘。

### 4. 偏好数据格式

DPO 数据的核心是同一个 prompt 下有两个回答：

```text
chosen：更好的回答
rejected：更差的回答
```

一个 JSONL 样本可以这样写：

```json
{"prompt": "请解释 LoRA 和全参数微调的区别", "chosen": "LoRA 是一种参数高效微调方法...", "rejected": "LoRA 就是微调，和全参数差不多。"}
```

如果使用 messages 风格，可以写成：

```json
{"prompt": [{"role": "system", "content": "你是一个严谨的中文 AI 助手。"}, {"role": "user", "content": "请解释 LoRA 和全参数微调的区别"}], "chosen": [{"role": "assistant", "content": "LoRA 是一种参数高效微调方法..."}], "rejected": [{"role": "assistant", "content": "LoRA 就是微调，和全参数差不多。"}]}
```

具体格式取决于训练框架版本。关键是保证 prompt 相同，只有回答不同。

### 5. 偏好样本怎么来

偏好数据可以来自几种方式。

#### 5.1 人工标注

给同一个 prompt 生成多个候选回答，让人工选择更好的。

优点：质量高。

缺点：成本高。

#### 5.2 规则构造

例如构造 rejected：

```text
删除关键步骤
制造格式错误
加入明显幻觉
让回答过度冗长
让回答过度简短
让安全问题直接给危险建议
```

优点：成本低，适合个人项目。

缺点：分布可能不够真实。

#### 5.3 多模型候选

使用 base model、SFT model、不同 decoding 参数生成多个回答，再人工或 LLM-as-Judge 选择偏好。

优点：更接近真实模型错误。

缺点：需要额外生成和评审。

#### 5.4 用户反馈

真实业务中可以使用点赞、点踩、追问、投诉、人工接管等信号。

个人项目通常没有真实用户反馈，但可以在项目报告中说明业务场景下的扩展方式。

### 6. 偏好数据质量原则

偏好数据不是越多越好。

高质量偏好对应该满足：

```text
prompt 清晰
chosen 明显优于 rejected
差异原因明确
没有把错误答案标成 chosen
chosen 不只是更长，而是真的更好
rejected 不要全部是弱智错误
覆盖多个任务类型
包含安全和拒答边界
```

如果偏好对质量差，DPO 会放大错误偏好。

例如如果数据里总是“更短的回答是 chosen”，模型可能学会过度简短。

### 7. 偏好数据切片

建议按任务类型组织：

```text
完整性偏好：chosen 覆盖关键点，rejected 缺少关键点
事实性偏好：chosen 正确，rejected 有事实错误
格式偏好：chosen 符合 JSON/表格格式，rejected 格式错误
简洁性偏好：chosen 简洁清楚，rejected 冗长重复
安全偏好：chosen 合理拒答，rejected 给出危险建议
拒答边界：chosen 正常回答安全问题，rejected 过度拒答
领域偏好：chosen 符合业务规则，rejected 违反业务规则
```

这样评估时能知道 DPO 到底在哪些维度起作用。

### 8. 构造偏好样本脚本

下面是一个简单示例：从人工整理的 CSV 构造 JSONL。

```python
import csv
import json


def build_preference_jsonl(csv_path: str, out_path: str) -> None:
    with open(csv_path, newline="", encoding="utf-8") as f, open(out_path, "w", encoding="utf-8") as out:
        reader = csv.DictReader(f)
        for row in reader:
            item = {
                "prompt": row["prompt"].strip(),
                "chosen": row["chosen"].strip(),
                "rejected": row["rejected"].strip(),
                "category": row.get("category", "general"),
                "reason": row.get("reason", ""),
            }
            if item["prompt"] and item["chosen"] and item["rejected"]:
                out.write(json.dumps(item, ensure_ascii=False) + "\n")
```

建议保留 `category` 和 `reason` 字段，方便后续切片分析。

### 9. DPO 的直观理解

DPO 希望模型在同一个 prompt 下，提高 chosen 的相对概率，降低 rejected 的相对概率。

直观地说，它不是告诉模型：

```text
只要模仿 chosen。
```

而是告诉模型：

```text
相比 rejected，你应该更偏好 chosen。
```

DPO 还会参考 reference model，避免当前模型偏离原模型太远。

这就是为什么 DPO 公式里会比较 policy model 和 reference model 对 chosen/rejected 的 log probability 差异。

### 10. beta 的作用

DPO 中常见一个超参数 `beta`。

可以直观理解为控制偏好优化强度。

```text
beta 较小：更新更温和，模型不容易偏离 reference。
beta 较大：偏好信号更强，模型变化更明显，但退化风险更高。
```

实际项目中可以尝试：

```text
beta = 0.01
beta = 0.05
beta = 0.1
```

并比较偏好胜率、输出长度、安全拒答和通用能力变化。

### 11. DPO 训练脚本骨架

下面示例使用 TRL 的 `DPOTrainer`。不同 TRL 版本 API 变化较快，项目中要固定版本。

```python
from datasets import load_dataset
from peft import LoraConfig
from transformers import AutoModelForCausalLM, AutoTokenizer
from trl import DPOConfig, DPOTrainer


model_name = "Qwen/Qwen2.5-1.5B-Instruct"

tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True)

model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto",
    device_map="auto",
    trust_remote_code=True,
)
ref_model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto",
    device_map="auto",
    trust_remote_code=True,
)

peft_config = LoraConfig(
    r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"],
)

dataset = load_dataset("json", data_files={
    "train": "data/preference_train.jsonl",
    "dev": "data/preference_dev.jsonl",
})

args = DPOConfig(
    output_dir="outputs/qwen-dpo-lora",
    per_device_train_batch_size=1,
    per_device_eval_batch_size=1,
    gradient_accumulation_steps=16,
    learning_rate=5e-6,
    num_train_epochs=1,
    logging_steps=10,
    eval_steps=100,
    save_steps=100,
    beta=0.05,
    bf16=True,

trainer = DPOTrainer(
```

```text
transformers==4.x
trl==0.x
peft==0.x
```

### 12. 使用 SFT 模型作为起点

更合理的流程是：

```text
base model -> SFT model -> DPO model
```

而不是直接：

```text
base model -> DPO model
```

原因是 DPO 偏好优化默认模型已经具备基本指令遵循能力。SFT 提供基础助手行为，DPO 在此基础上调整偏好。

项目中可以把第 52 讲的 SFT LoRA 作为初始化，再做 DPO。

### 13. DPO 评估怎么做

DPO 后不能只看训练 loss。

至少要比较：

```text
SFT model
DPO model
```

评估维度包括：

```text
pairwise 人工偏好胜率
LLM-as-Judge 偏好胜率
chosen/rejected logprob gap
平均输出长度
格式遵循率
安全拒答合格率

通用能力回归集
bad case 分布
```

### 14. Pairwise 评估模板

可以复用第 49 讲的成对评测表。

重点比较：

```text

```

统计：

```text

都不好比例
```


### 15. 输出长度分析

DPO 很容易改变输出长度。

你应该单独统计：

```python
import json


def avg_length(path: str) -> float:
    lengths = []
    with open(path, encoding="utf-8") as f:
        for line in f:
            item = json.loads(line)
            lengths.append(len(item["output"]))
    return sum(lengths) / len(lengths)
```

如果 DPO 后输出明显变短，要检查偏好数据是不是偏向短回答。

如果 DPO 后输出明显变长，要检查 chosen 是否总是更长，模型是否学到了“长就是好”。

### 16. DPO 常见副作用

#### 16.1 回答变短

如果偏好数据中 chosen 往往更简洁，模型可能过度学习“短回答更好”。

#### 16.2 过度拒答

如果安全偏好样本比例过高或边界设计不好，模型可能对正常问题也拒答。

#### 16.3 奖励黑客

模型学会一些表面偏好特征，例如：

```text
总是分点
总是说“需要注意”
总是输出更长
总是加免责声明
```

但真实质量不一定提升。

#### 16.4 通用能力退化

DPO 数据分布太窄时，模型可能在其他任务上变差。

#### 16.5 风格偏移

模型可能变得过于保守、啰嗦、模板化或客服腔。

### 17. Bad Case 复盘

DPO 项目报告中一定要有 bad case。

示例：

```markdown
## Bad Case 1：回答变短导致信息不足

Prompt：请比较 LoRA、Prefix Tuning 和全参数微调。

SFT 输出：覆盖三者原理、优缺点和适用场景。

DPO 输出：只给出简短定义，缺少适用场景。

改进方向：增加“完整但不啰嗦”的 chosen 样本，避免把过短回答标为偏好。
```

### 19. 面试高频问法

#### 问法 1：DPO 和 SFT 有什么区别？

可以这样答：

```text
SFT 是让模型模仿标准答案，本质是对 assistant answer 做 next-token prediction。DPO 使用偏好对，同一个 prompt 下有 chosen 和 rejected，它优化的是模型对 chosen 相对 rejected 的偏好概率。SFT 解决“怎么回答”的基础模仿问题，DPO 更偏向解决“什么回答更符合人类偏好”的排序问题。
```

#### 问法 2：DPO 为什么需要 reference model？

可以这样答：

```text
reference model 提供一个稳定参照，通常是 SFT 模型。DPO 不只是让 policy model 提高 chosen 概率，还会比较 policy 相对于 reference 在 chosen/rejected 上的概率差异。这样可以限制模型不要为了偏好数据过度偏离原始能力，起到类似 KL 约束的作用。
```

#### 问法 3：DPO 数据怎么构造？

可以这样答：

```text
每条数据要有同一个 prompt 下的 chosen 和 rejected。chosen 应该在某个明确维度上优于 rejected，例如事实更正确、覆盖更完整、格式更合规、安全边界更合理或业务规则更符合。数据构造可以来自人工标注、多模型候选、用户反馈或规则构造，但要避免 chosen 只是更长或更短这种伪偏好。最好保留任务类型和偏好原因，方便后续切片评估。
```

#### 问法 4：DPO 后模型变差怎么办？

可以这样答：

```text
我会先看变差发生在哪个维度，是回答变短、过度拒答、事实错误、格式退化还是通用能力退化。然后检查偏好数据分布和 chosen/rejected 标注质量，尤其看是否存在长度偏置、安全样本过多或错误偏好。训练侧会对比 beta、学习率、epoch、LoRA rank 和 checkpoint。缓解方法包括清洗偏好数据、增加反例边界样本、降低 beta 或学习率、减少训练步数，并加入通用能力回归集作为门禁。
```

### 21. 常见工程坑

#### 21.1 直接用低质量偏好数据

DPO 会放大偏好数据中的错误偏好。

#### 21.2 chosen 只是更长

模型可能学会输出更长，而不是真正更好。

#### 21.3 chosen 只是更短

模型可能学会过度简短，信息不足。

#### 21.4 beta 设置过大

模型变化过猛，容易风格偏移或能力退化。

#### 21.5 不设 dev 偏好集

只看训练集偏好优化，无法判断泛化。

#### 21.6 不做安全和通用能力回归

偏好胜率提升不代表模型整体可上线。

#### 21.7 把 DPO 说成 RLHF 全流程

DPO 属于偏好优化方法，不等于完整 RLHF 工程链路。

### 22. 本讲小结

这一讲设计了一个 DPO 偏好优化实验项目。

核心结论如下：

1. DPO 适合个人项目，因为它不需要单独训练 reward model，也不需要 PPO。
2. DPO 数据的核心是同一个 prompt 下的 chosen/rejected 偏好对。
3. 偏好数据质量决定 DPO 效果，错误偏好会被模型放大。
4. chosen 不应该只是更长或更短，而应该在明确质量维度上更好。
5. DPO 通常以 SFT model 为起点，并使用 reference model 控制偏离。
6. beta 控制偏好优化强度，需要做消融对比。
7. DPO 评估不能只看 loss，要看 pairwise 胜率、输出长度、安全拒答、格式遵循和通用能力回归。
8. DPO 常见副作用包括回答变短、过度拒答、风格偏移和通用能力退化。
9. 简历表达要突出偏好数据构造、DPO 训练、评估指标和 bad case 分析。
10. 面试中要能讲清 DPO 与 SFT/RLHF 的区别、reference model 作用和退化排查。

### 4. FastAPI 服务骨架

```python
    model: str

app = FastAPI(title="LLM Inference Service")


@app.post("/v1/chat/completions")
def chat(req: ChatRequest):
    request_id = f"chatcmpl-{uuid.uuid4().hex}"
    start = time.time()

    if not req.messages or req.messages[-1].role != "user":
        raise HTTPException(status_code=400, detail="last message must be user")

    text, usage = generate_once(req)
    latency_ms = int((time.time() - start) * 1000)

    return {
        "id": request_id,
        "model": req.model,
        "choices": [
            {"message": {"role": "assistant", "content": text}, "finish_reason": "stop"}
        ],
        "usage": usage,
        "latency_ms": latency_ms,
    }
```

这里 `generate_once` 可以封装 tokenizer、chat template、model.generate 和 decode。

### 5. 非流式生成逻辑

```python
import torch


@torch.no_grad()
def generate_once(req):
    messages = [m.model_dump() for m in req.messages]
    prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
    inputs = tokenizer(prompt, return_tensors="pt").to(model.device)

    output_ids = model.generate(
        **inputs,
        max_new_tokens=req.max_tokens,
        temperature=req.temperature,
        top_p=req.top_p,
        do_sample=req.temperature > 0,
        use_cache=True,
    )

    new_ids = output_ids[0, inputs["input_ids"].shape[1] :]
    text = tokenizer.decode(new_ids, skip_special_tokens=True)
    usage = {
        "prompt_tokens": int(inputs["input_ids"].shape[1]),
        "completion_tokens": int(new_ids.shape[0]),
        "total_tokens": int(output_ids.shape[1]),
    }
    return text, usage
```

注意 `use_cache=True`，它会启用 KV Cache，避免每生成一个新 token 都重复计算全部历史上下文。

### 6. 流式输出

大模型生成可能需要几秒甚至几十秒。非流式接口要等完整回答生成后才返回，用户体验较差。

流式输出的目标是：

```text
模型每生成一小段，就立刻返回给前端。
```

常见方式是 Server-Sent Events，也就是 SSE。

```python
from fastapi.responses import StreamingResponse


@app.post("/v1/chat/completions/stream")
def chat_stream(req: ChatRequest):
    return StreamingResponse(
        stream_generate(req),
        media_type="text/event-stream",
    )
```

教学版流式生成可以使用 `TextIteratorStreamer`。

```python
import json
import threading
from transformers import TextIteratorStreamer


def stream_generate(req):
    messages = [m.model_dump() for m in req.messages]
    prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
    inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
    streamer = TextIteratorStreamer(tokenizer, skip_prompt=True, skip_special_tokens=True)

    kwargs = dict(
        **inputs,
        streamer=streamer,
        max_new_tokens=req.max_tokens,
        temperature=req.temperature,
        top_p=req.top_p,
        do_sample=req.temperature > 0,
        use_cache=True,
    )

    thread = threading.Thread(target=model.generate, kwargs=kwargs)
    thread.start()

    for token_text in streamer:
        payload = {"delta": token_text}
        yield f"data: {json.dumps(payload, ensure_ascii=False)}\n\n"
    yield "data: [DONE]\n\n"
```

真实服务中还要处理客户端断连、超时、异常、敏感词中断和 request_id 日志。

### 7. 性能指标

推理服务至少要报告以下指标：

```text
首 token 延迟 TTFT：Time To First Token
总延迟 latency：完整回答耗时
输出吞吐 tokens/s：每秒生成多少 token
请求吞吐 QPS：每秒处理多少请求
并发数 concurrency：同时处理请求数量
显存占用 GPU memory
错误率 error rate
超时率 timeout rate
P50/P90/P95/P99 延迟
```

面试中要特别区分：

```text
TTFT 影响用户是否觉得系统响应快。
tokens/s 影响长回答生成速度。
QPS 影响服务承载能力。
P95/P99 比平均延迟更能反映线上体验。
```

### 8. KV Cache

自回归生成每次只生成一个新 token。

如果没有 KV Cache，第 t 步生成时要重新计算前面所有 token 的 key/value，复杂度很高。

KV Cache 会缓存历史 token 的 key/value，新 token 只需要和历史缓存做 attention。

好处：

```text
降低重复计算
提升 decode 阶段速度
长文本生成收益明显
```

代价：

```text
占用更多显存
batch 越大、上下文越长、层数越多，KV Cache 越大
```

所以推理服务要在吞吐、上下文长度和显存之间做权衡。

### 9. 批处理与动态批处理

单请求推理通常不能充分利用 GPU。

批处理把多个请求合并成一个 batch，提高 GPU 利用率。

普通批处理的问题：

```text
请求长度不同，padding 浪费
生成长度不同，短请求要等长请求
实时请求不能等太久
```

更好的方式是动态批处理或连续批处理。

实际项目可以先实现简单队列：

```text
请求进入队列
等待最多 10-20ms
凑够 batch_size 或到达等待上限
一起送入模型
```

如果使用 vLLM、TGI、TensorRT-LLM 等后端，可以重点说明它们支持更高效的调度，例如 paged attention 和 continuous batching。

### 10. 量化

量化用于降低显存和提升吞吐。

常见方式：

```text
FP16/BF16：常规半精度推理
INT8：进一步降低显存
INT4：显存更低，但质量风险更高
GPTQ/AWQ：常见权重量化方案
```

量化评估不能只看速度，还要看质量：

```text
显存下降多少
TTFT 是否改善
tokens/s 是否提升
困惑度或评测集是否退化
人工样例是否出现明显质量下降
```

### 11. 压测脚本思路

压测要模拟并发请求。

```python
import asyncio
import time
import httpx


async def one_request(client, url, payload):
    start = time.time()
    resp = await client.post(url, json=payload, timeout=120)
    latency = time.time() - start
    return resp.status_code, latency


async def benchmark(url, payload, concurrency: int, total: int):
    async with httpx.AsyncClient() as client:
        sem = asyncio.Semaphore(concurrency)

        async def run_one():
            async with sem:
                return await one_request(client, url, payload)

        tasks = [run_one() for _ in range(total)]
        results = await asyncio.gather(*tasks)

    latencies = [x[1] for x in results if x[0] == 200]
    latencies.sort()
    print("success", len(latencies), "total", total)
    print("p50", latencies[int(0.50 * len(latencies))])
    print("p95", latencies[int(0.95 * len(latencies))])
```

### 15. 面试高频问法

#### 问法 1：如何降低大模型推理延迟？

可以这样答：

```text
我会先拆分延迟来源，包括排队时间、prefill 时间、decode 时间和网络传输时间。优化上可以控制 prompt 长度和 max_tokens，启用 KV Cache，使用 FP16/BF16 或合适量化，采用批处理或连续批处理提升 GPU 利用率，必要时接入 vLLM/TGI/TensorRT-LLM 等后端。用户体验上还可以用流式输出降低首屏等待。
```

#### 问法 2：什么是 TTFT？为什么重要？

可以这样答：

```text
TTFT 是 Time To First Token，也就是从请求发出到收到第一个生成 token 的时间。它主要受排队、tokenize、prefill 和调度影响。对聊天产品来说，TTFT 直接影响用户主观响应速度，即使完整回答还没生成完，只要首 token 快且流式输出顺畅，用户体验会明显更好。
```

#### 问法 3：KV Cache 有什么作用和代价？

可以这样答：

```text
KV Cache 缓存历史 token 在每一层 attention 中的 key/value。自回归生成时，新 token 不需要反复重算全部历史上下文，只需要和缓存交互，因此 decode 阶段更快。代价是显存占用增加，且与 batch size、上下文长度、层数、hidden size 有关，所以服务端要在并发、上下文长度和显存之间权衡。
```

#### 问法 4：如何设计推理服务的监控？

可以这样答：

```text
我会同时监控服务指标、模型指标和业务质量指标。服务指标包括 QPS、P95/P99 延迟、TTFT、错误率、超时率、GPU 显存和利用率；模型指标包括输入输出 token、tokens/s、finish_reason 和成本；质量指标包括用户反馈、安全拦截、人工抽检 bad case 和版本对比结果。所有日志都要带 request_id、model_version、prompt_version，便于排查问题。
```

### 16. 常见工程坑

#### 16.1 只测单请求延迟

单请求延迟不能代表线上并发能力。必须压测不同并发下的 P95/P99。

#### 16.2 不限制 max_tokens

用户请求可能生成超长回答，导致队列阻塞和成本失控。

#### 16.3 不记录版本信息

模型、prompt、检索器或量化配置变化后，如果日志没有版本，问题很难定位。

#### 16.4 把流式输出等同于模型更快

流式输出主要改善感知延迟，不一定降低完整生成时间。

#### 16.5 只看平均延迟

线上体验更关注 P95/P99，少量慢请求也会严重影响用户体验。

#### 16.6 量化后不做质量评估

量化可能降低显存，但也可能带来事实性、格式和推理能力退化。

### 6. Chunk 切分

chunk 是 RAG 的核心。

切得太小：

```text
上下文不完整，答案缺少条件。
```

切得太大：

```text
召回不精准，占用 prompt token，噪声变多。
```

常见策略：

```text
固定长度切分
按标题切分
按段落切分
递归切分
滑窗 overlap
语义切分
```

推荐个人项目中实现一个结构感知切分：

```text
优先按标题和段落切分
超过最大长度再递归切分
保留 section_path
相邻 chunk 保留 10%-20% overlap
```

示例代码：

```python
def chunk_text(text: str, max_chars: int = 800, overlap: int = 100) -> list[str]:
    chunks = []
    start = 0
    while start < len(text):
        end = min(start + max_chars, len(text))
        chunk = text[start:end]
        chunks.append(chunk.strip())
        if end == len(text):
            break
        start = max(0, end - overlap)
    return [c for c in chunks if c]
```

项目报告中可以比较不同 chunk size：

```text
chunk_size=300
chunk_size=600
chunk_size=1000
overlap=0/100/200
```

观察 Recall@K、上下文 token 数和答案质量变化。

```python
import faiss
import numpy as np


class FaissVectorStore:
    def __init__(self, dim: int):
        self.index = faiss.IndexFlatIP(dim)
        self.chunks = []

    def add(self, embeddings: np.ndarray, chunks: list[dict]) -> None:
        embeddings = embeddings.astype("float32")
        faiss.normalize_L2(embeddings)
        self.index.add(embeddings)
        self.chunks.extend(chunks)

    def search(self, query_embedding: np.ndarray, top_k: int = 10):
        query_embedding = query_embedding.astype("float32")
        faiss.normalize_L2(query_embedding)
        scores, indices = self.index.search(query_embedding, top_k)
        results = []
        for score, idx in zip(scores[0], indices[0]):
            if idx == -1:
                continue
            item = self.chunks[idx].copy()
            item["score"] = float(score)
            results.append(item)
        return results
```

这里使用 inner product，并对向量做 L2 normalize，等价于 cosine similarity。

### 10. 混合召回

只用向量召回不够。

向量召回擅长语义相似，但对精确词、编号、产品名、制度条款、错误码可能不稳定。

BM25 擅长关键词匹配，但不懂语义改写。

企业 RAG 常用混合召回：

```text
向量召回 top_k=20
BM25 召回 top_k=20
合并去重
reranker 重排 top_k=5
```

混合召回能提升：

```text
专业术语
编号条款
产品型号
人名地名
错误码
精确制度问题
```

### 11. Reranker

初召回通常会有噪声。

reranker 接收 query 和候选 chunk，对每个候选打相关性分数，再重新排序。

常见 reranker：

```text
bge-reranker
cross-encoder reranker
LLM reranker
```

reranker 的优点：

```text
相关性判断更准确
减少无关 chunk 进入 prompt
提升答案 groundedness
```

代价：

```text
增加延迟
增加算力成本
候选过多时速度慢
```

所以一般流程是：先高召回，再 rerank 小候选集。

### 14. 检索评估

RAG 要先评估检索，再评估生成。

检索评估集可以这样构造：


常用指标：


对于 RAG，Recall@K 通常非常重要。因为如果相关证据没召回，后面的 LLM 很难答对。

### 16. Bad Case 分类

RAG bad case 可以分为：


分类 bad case 可以帮助定位是 retriever 问题，还是 generator 问题。

企业知识库必须考虑权限。

### 21. 面试高频问法

#### 问法 1：RAG 为什么能减少幻觉？

可以这样答：

```text
RAG 在生成前先检索外部知识，把相关证据放入上下文，让模型基于证据回答，而不是完全依赖参数记忆。它尤其适合企业私有知识和频繁更新的知识。但 RAG 不能天然消除幻觉，如果检索不到、检索错了，或者 prompt 没有限制模型基于证据回答，模型仍可能编造。所以要同时优化召回、重排、prompt 和 groundedness 评估。
```

#### 问法 2：chunk size 怎么选？

可以这样答：

```text
chunk size 要在完整性和精准性之间权衡。太小会丢上下文，关键条件可能被切开；太大召回不精准，还会占用 prompt token。我通常会按文档结构切分，优先保留标题、段落和章节路径，超过长度再递归切分，并设置一定 overlap。最终通过 Recall@K、答案质量和上下文 token 成本做实验选择。
```

#### 问法 3：为什么需要 reranker？

可以这样答：

```text
向量召回阶段通常追求高召回，会取较多候选，但里面有噪声。reranker 使用 query 和 chunk 的交互式匹配，对候选重新排序，能把真正相关的证据排到前面，减少无关内容进入 prompt。代价是增加延迟，所以通常先召回 top 20 或 top 50，再 rerank 到 top 5。
```

#### 问法 4：RAG 答错了怎么排查？

可以这样答：

```text
我会先看相关证据有没有被解析和入库。如果入库没问题，再看 chunk 是否把关键条件切开。然后检查检索 top K 是否召回相关 chunk，如果没召回就是 retriever 或 query 问题；如果召回了但排序靠后，就是 rerank 问题；如果正确证据已经进入 prompt 但模型仍答错，就是 generation 或 prompt 约束问题。最后还要检查引用是否支持答案，以及用户是否有权限访问相关文档。
```

### 23. 常见工程坑

#### 23.1 只做向量检索 Demo

没有解析、切分、评估和引用，很难体现工程深度。

#### 23.2 chunk 切分随意

把标题、表格、关键条件切碎，会导致召回和生成都变差。

#### 23.3 只看最终回答，不看召回

RAG 答错时必须先定位检索是否成功。

#### 23.4 prompt 没有限制基于证据回答

模型可能用参数知识补充，产生幻觉。

#### 23.5 引用不校验

模型可能引用了文档，但引用并不支持答案。

#### 23.6 忽略权限

企业 RAG 中权限错误是严重安全问题。

#### 23.7 不支持知识更新

文档过期后，系统会持续输出旧答案。

核心结论如下：

### 5. Baseline 设计

实验至少比较四种策略。

#### 5.1 Zero-shot Direct

直接让模型回答。

```text
请直接给出答案。

问题：{question}
答案：
```

优点：简单、成本低。

缺点：多步题容易跳步。

#### 5.2 Few-shot Direct

给几个示例，但不要求推理过程。

```text
问题：小明有 2 个苹果，又买 3 个，一共有几个？
答案：5

问题：{question}
答案：
```

优点：能规范输出格式。

缺点：复杂推理提升有限。

#### 5.3 Chain-of-Thought

要求模型逐步思考。

```text
请一步一步推理，并在最后用“最终答案：”给出结果。

问题：{question}
```

优点：多步推理通常更好。

缺点：可能生成看似合理但错误的推理。

#### 5.4 Self-Consistency

对同一道题采样多个 CoT 解答，然后对最终答案投票。

```text
同一道题生成 N 个推理路径。
抽取每个路径的最终答案。
选择出现次数最多的答案。
```

优点：降低单次采样偶然错误。

缺点：推理成本成倍增加。

### 7. 答案抽取与标准化

数学评估最大的工程坑是答案格式。

模型可能输出：

```text
14
14 个
最终答案：14
所以答案是 14 个苹果
14.0
```

这些都应视为同一个答案。

简单抽取函数：

```python
import re


def extract_final_answer(text: str) -> str:
    patterns = [
        r"最终答案[:：]\s*([^\n]+)",
        r"答案[:：]\s*([^\n]+)",
    ]
    for pattern in patterns:
        match = re.search(pattern, text)
        if match:
            return normalize_answer(match.group(1))
    numbers = re.findall(r"-?\d+(?:\.\d+)?", text)
    return normalize_answer(numbers[-1]) if numbers else ""


def normalize_answer(ans: str) -> str:
    ans = ans.strip()
    ans = ans.replace("，", ",")
    ans = re.sub(r"[个元米厘米千克%％。\s]", "", ans)
    if re.fullmatch(r"-?\d+\.0+", ans):
        ans = ans.split(".")[0]
    return ans
```

这个函数只是教学版。真实项目要处理分数、百分比、单位、多个答案和表达式等情况。

### 8. 评估脚本骨架

```python
import json


def run_eval(model, tokenizer, data_path: str, prompt_template: str):
    total = 0
    correct = 0
    records = []

    with open(data_path, encoding="utf-8") as f:
        for line in f:
            item = json.loads(line)
            prompt = prompt_template.format(question=item["question"])
            output = generate_text(model, tokenizer, prompt)
            pred = extract_final_answer(output)
            gold = normalize_answer(item["answer"])
            ok = pred == gold

            total += 1
            correct += int(ok)
            records.append({
                "id": item.get("id"),
                "question": item["question"],
                "gold": gold,
                "pred": pred,
                "output": output,
                "correct": ok,
                "category": item.get("category"),
                "difficulty": item.get("difficulty"),
            })

    accuracy = correct / total if total else 0
    print("accuracy", round(accuracy, 4))
    return records
```

报告中要按 category 和 difficulty 切片，而不是只给总准确率。

### 9. Self-Consistency 实现

Self-consistency 要对同一道题采样多次。

```python
from collections import Counter


def self_consistency_answer(model, tokenizer, question: str, n: int = 5):
    answers = []
    outputs = []
    prompt = COT_PROMPT.format(question=question)

    for _ in range(n):
        output = generate_text(
            model,
            tokenizer,
            prompt,
            temperature=0.7,
            top_p=0.9,
        )
        outputs.append(output)
        answers.append(extract_final_answer(output))

    counter = Counter(answers)
    final_answer, votes = counter.most_common(1)[0]
    return final_answer, votes, outputs
```

注意：self-consistency 要使用采样，否则多次生成可能几乎一样。

### 10. 工具校验

大模型很容易在算术上犯错。

可以让模型输出表达式，再用 Python 校验。

例如 prompt 要求：

```text
请给出推理过程，并在最后输出：
表达式：...
最终答案：...
```

然后抽取表达式：

```python
import ast
import operator as op


OPS = {
    ast.Add: op.add,
    ast.Sub: op.sub,
    ast.Mult: op.mul,
    ast.Div: op.truediv,
    ast.USub: op.neg,
}


def safe_eval_expr(expr: str):
    def eval_node(node):
        if isinstance(node, ast.Constant) and isinstance(node.value, (int, float)):
            return node.value
        if isinstance(node, ast.BinOp) and type(node.op) in OPS:
            return OPS[type(node.op)](eval_node(node.left), eval_node(node.right))
        if isinstance(node, ast.UnaryOp) and type(node.op) in OPS:
            return OPS[type(node.op)](eval_node(node.operand))
        raise ValueError("unsupported expression")

    tree = ast.parse(expr, mode="eval")
    return eval_node(tree.body)
```

工具校验只能验证表达式计算，不保证表达式本身建模正确。

例如题目应该是 `12 - 3 + 5`，模型写成 `12 + 3 + 5`，Python 会准确算出错误表达式的结果。所以工具能减少计算错误，但不能完全解决推理错误。

### 11. 错误类型分析

数学推理项目最重要的不是准确率，而是错误分析。

建议分类：

```text
题意理解错误：误读条件或问题目标
步骤遗漏：少算一步
运算错误：表达式对，但算错
公式错误：使用了错误公式
单位错误：单位换算错误
干扰信息错误：把无关信息纳入计算
答案抽取错误：推理正确但最终答案格式不规范
自相矛盾：推理过程和最终答案不一致
过度猜测：没有根据题目条件补充信息
```

Bad case 示例：

```markdown
## Bad Case：干扰信息错误

题目：小明有 12 个苹果，小红有 8 个苹果。小明送给小红 3 个后，小明还有几个？

模型输出：12 + 8 - 3 = 17，所以答案是 17。

正确答案：9。

错误类型：干扰信息错误。

原因分析：模型把小红原有苹果数量错误加入了目标计算。

改进方向：增加带干扰信息的训练/示例题，prompt 强调先确定问题目标。
```

### 12. 过程质量评估

最终答案正确，不代表推理过程正确。

常见情况：

```text
答案猜对，但过程错误
过程大体正确，但最后抄错答案
过程冗长且有无关步骤
过程无法验证
```

可以人工评估：

```text
步骤是否完整
每一步是否由题目条件推出
计算是否正确
最终答案是否与过程一致
```

对于训练推理模型，过程质量非常重要。

### 15. 面试高频问法

#### 问法 1：为什么 CoT 能提升数学推理？

可以这样答：

```text
CoT 通过显式生成中间推理步骤，把一个复杂问题分解成多个简单子问题。对于多步数学题，直接输出答案容易跳步，而 CoT 让模型先识别条件、列式、计算，再给最终答案，相当于把隐式推理外显化。不过 CoT 不是保证正确，它也可能生成看似合理但错误的推理，所以还需要答案校验和 bad case 分析。
```

#### 问法 2：Self-consistency 为什么有效？

可以这样答：

```text
Self-consistency 的思想是同一道题采样多个推理路径，然后对最终答案投票。单次 CoT 可能因为采样或局部推理错误得到错误答案，但如果多数推理路径收敛到同一个答案，这个答案通常更稳。它的代价是推理成本按采样次数增加，所以适合高价值或复杂问题，不适合所有线上请求都默认使用。
```

#### 问法 3：数学推理评估有哪些坑？

可以这样答：

```text
第一是答案抽取和标准化，比如 14、14 个、14.0 可能是同一个答案。第二是最终答案正确不代表过程正确，可能猜对或过程自相矛盾。第三是不同题型难度差异很大，只看总准确率会掩盖短板。第四是 CoT 输出更长，成本更高，所以要同时看准确率、输出 token 和延迟。第五是工具校验只能减少计算错误，不能保证建模和题意理解正确。
```

#### 问法 4：如何进一步提升数学推理能力？

可以这样答：

```text
可以从数据、推理策略和工具三个方向做。数据上构造高质量带步骤的 SFT 数据，覆盖多步、干扰信息和单位换算等难点。推理策略上使用 few-shot CoT、self-consistency、问题分解或 verifier rerank。工具上让模型调用计算器或 Python 校验表达式，减少纯算术错误。最后要做细粒度错误分析，区分题意理解、公式选择、运算和答案抽取问题。
```

### 17. 常见工程坑

#### 17.1 只看几个样例

几个样例无法说明推理能力，必须构造评测集。

#### 17.2 不做答案标准化

格式差异会导致准确率统计失真。

#### 17.3 把 CoT 输出当作一定正确

CoT 可能生成合理但错误的过程。

#### 17.4 不统计成本

self-consistency 提升准确率，但成本和延迟会明显增加。

#### 17.5 错误分析太粗

只说“模型推理错了”没有价值，要区分题意、公式、运算、单位和抽取错误。

#### 17.6 工具调用边界不清

计算器能算表达式，但不能保证表达式列得对。

### 18. 本讲小结

这一讲设计了一个数学推理增强实验项目。

核心结论如下：

1. 数学推理项目展示的是模型推理、评测、prompting、工具校验和错误分析能力。
2. 评测集要覆盖多种题型和难度，不能只看简单算术。
3. 至少比较 zero-shot、few-shot、CoT 和 self-consistency。
4. CoT 能提升多步推理，但不保证过程正确。
5. Self-consistency 能提升稳定性，但成本按采样次数增加。
6. 数学评估必须做答案抽取和标准化。
7. 工具校验可以减少计算错误，但不能解决题意理解和建模错误。
8. 错误分析要细分为题意理解、步骤遗漏、运算错误、公式错误、单位错误、干扰信息错误等。
9. 简历表达要突出实验对比、切片评估、成本收益和 bad case 复盘。
10. 面试中要能讲清 CoT、self-consistency、工具校验和推理评估的局限。
