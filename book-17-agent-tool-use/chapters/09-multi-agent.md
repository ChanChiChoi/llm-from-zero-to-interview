# 第九章：Multi-Agent

Multi-Agent， 多智能体系统，是把多个 Agent 组合起来协作完成任务的架构。一个 Agent 可以负责规划，一个 Agent 负责检索，一个 Agent 负责写代码，一个 Agent 负责评审，还有一个 Agent 负责汇总。多 Agent 看起来很强，但也容易带来通信成本、责任不清、冲突、重复劳动和错误放大。

本章系统讲 Multi-Agent：为什么需要多 Agent，角色分工、通信协议、协调器、辩论与评审、任务分配、冲突解决、共享记忆、成本控制、安全边界、评估指标和面试表达。

## 9.1 为什么需要 Multi-Agent

单 Agent 的问题是能力和上下文都集中在一个模型调用或一个执行循环里。

复杂任务中，单 Agent 可能会：

1. 同时承担规划、执行、验证和总结，负担过重。
2. 忽略不同视角。
3. 无法并行处理子任务。
4. 自己检查自己的错误，容易漏掉问题。
5. 在长任务中上下文混乱。

Multi-Agent 的思路是把任务分给多个专门角色，让它们协作、互检或并行执行。

面试回答：

```text
Multi-Agent 是用多个 Agent 通过角色分工、通信和协调来完成复杂任务。它可以提升并行性、专业化和互相检查能力，例如 planner、researcher、coder、reviewer 分别负责不同环节。但它也会增加通信成本、协调复杂度和冲突解决问题，不是所有任务都值得使用。
```

## 9.2 Multi-Agent 的基本结构

常见结构包括：

1. Centralized：一个 coordinator 分配任务。
2. Decentralized：多个 Agent 彼此通信协作。
3. Hierarchical：高层 Agent 规划，低层 Agent 执行。
4. Debate：多个 Agent 给出观点，再由 judge 选择。
5. Committee：多个 Agent 独立解题后投票或汇总。

不同结构适合不同任务。工程系统最常用的是 coordinator 模式，因为更容易控制流程、权限和日志。

## 9.3 角色分工

Multi-Agent 的关键是角色明确。

常见角色：

1. Planner：拆解任务和制定计划。
2. Researcher：检索资料和收集证据。
3. Executor：调用工具或执行动作。
4. Coder：修改代码。
5. Reviewer：检查结果和发现问题。
6. Critic：专门找漏洞和反例。
7. Summarizer：汇总输出。
8. Coordinator：分配任务和控制流程。

角色不是越多越好。角色过多会增加通信和管理成本。一个实用原则是：只有当角色能减少复杂度或提升验证质量时，才拆出独立 Agent。

## 9.4 Coordinator 模式

Coordinator 是多 Agent 系统中的调度者。

它负责：

1. 理解用户目标。
2. 拆解子任务。
3. 分配给合适 Agent。
4. 收集结果。
5. 解决冲突。
6. 控制预算。
7. 生成最终回答。

Coordinator 模式的优势是可控。缺点是 coordinator 成为瓶颈，如果它分配任务错了，整个系统会偏离方向。

## 9.5 通信协议

Agent 之间不能随意聊天，最好有结构化通信协议。

一个消息可以包含：

1. sender：发送者。
2. receiver：接收者。
3. task_id：任务编号。
4. intent：消息目的。
5. content：内容。
6. evidence：证据或引用。
7. confidence：置信度。
8. status：完成、失败、需要澄清。

结构化通信有助于减少误解、支持日志回放和自动评估。

## 9.6 共享状态与共享记忆

多 Agent 需要共享部分状态。

共享内容可以包括：

1. 全局目标。
2. 当前计划。
3. 子任务状态。
4. 已收集证据。
5. 已执行动作。
6. 冲突点。
7. 最终决策。

但不是所有信息都应该共享。过度共享会造成上下文膨胀和隐私风险。工程上可以用共享 blackboard 或任务状态表，让各 Agent 只读取自己需要的信息。

## 9.7 辩论与 Critique

多 Agent 常用于辩论或互评。

典型流程：

```text
Agent A 给出答案
Agent B 找反例
Agent C 检查证据
Judge 汇总并选择最终结论
```

优势：

1. 能发现单 Agent 忽略的问题。
2. 能从不同角度评估答案。
3. 适合复杂决策和开放问题。

局限：

1. 多个 Agent 可能共享同样偏差。
2. 辩论可能变成冗长文本互相说服。
3. Judge 也可能判断错误。
4. 成本明显上升。

辩论不是万能验证器。对于可执行任务，工具验证通常比纯文本辩论更可靠。

## 9.8 并行执行

Multi-Agent 可以提升并行性。

例如研究报告任务：

1. Agent A 查背景。
2. Agent B 查竞品。
3. Agent C 查技术路线。
4. Agent D 查风险。
5. Coordinator 汇总。

并行执行可以节省时间，但需要处理结果合并和冲突。如果多个 Agent 得到矛盾结论，系统必须记录来源并解决冲突。

## 9.9 冲突解决

多 Agent 系统一定会出现冲突。

冲突类型：

1. 事实冲突。
2. 方案冲突。
3. 优先级冲突。
4. 工具结果冲突。
5. 权限冲突。
6. 资源冲突。

解决方式：

1. 要求提供证据。
2. 调用外部工具验证。
3. 由 judge Agent 判断。
4. 交给 coordinator 决策。
5. 请求用户确认。
6. 保留不确定性而不是强行统一。

高风险任务中，冲突不应由模型闭环自动决定，应该引入人工确认或程序化验证。

## 9.10 任务分配

任务分配要考虑 Agent 能力和工具权限。

例如：

1. Researcher 有检索工具，但没有写权限。
2. Coder 有文件编辑权限，但不能访问用户隐私数据。
3. Reviewer 只能读 diff 和测试结果。
4. Coordinator 可以分配任务，但不直接执行高风险动作。

权限分离可以降低事故范围。不要让所有 Agent 都拥有全部工具权限。

## 9.11 多 Agent 的成本

Multi-Agent 成本很高。

成本来源：

1. 多个模型调用。
2. Agent 之间通信。
3. 重复检索。
4. 重复验证。
5. 上下文汇总。
6. 冲突解决。

因此要判断任务是否值得多 Agent。简单任务用单 Agent 或 workflow 更合适。

## 9.12 Multi-Agent 和 Workflow

Multi-Agent 和 workflow 可以结合。

一种实用架构：

```text
固定 workflow 控制主流程
关键步骤交给专门 Agent 执行
Coordinator 汇总结果
Verifier 做最终检查
```

这样既保留 workflow 的稳定性，又利用 Agent 的灵活性。完全自由聊天式 multi-agent 在 demo 中很有趣，但生产系统通常需要强控制。

## 9.13 安全风险

Multi-Agent 增加了安全面。

风险包括：

1. Agent 之间传播错误信息。
2. 一个 Agent 被注入影响其他 Agent。
3. 权限边界不清。
4. 日志难以审计。
5. 冲突处理不透明。
6. 成本失控。
7. 多 Agent 合谋绕过限制。

安全设计：

1. 最小权限。
2. 明确角色边界。
3. 结构化消息。
4. 共享状态审计。
5. 高风险动作统一由 controller 审批。
6. 不信任其他 Agent 的未验证结论。

## 9.14 评估指标

Multi-Agent 评估可以看：

1. 任务成功率。
2. 相比单 Agent 的提升。
3. 平均成本。
4. 平均延迟。
5. 通信轮数。
6. 冲突解决成功率。
7. 重复工作比例。
8. 安全违规率。
9. 人工接管率。
10. 最终答案证据支持率。

核心问题是：多 Agent 是否真的比单 Agent 更好，还是只是更贵、更复杂。

## 9.15 常见失败模式

1. 角色分工不清。
2. Coordinator 分配错误。
3. 多个 Agent 重复做同一件事。
4. Agent 互相传递错误结论。
5. Judge 被流畅表达误导。
6. 冲突没有解决就输出答案。
7. 通信成本超过收益。
8. 权限边界混乱。
9. 共享 memory 被污染。
10. 最终结果没人负责。

Multi-Agent 的目标不是把系统变热闹，而是提高任务成功率和可验证性。

## 9.16 面试题：什么时候需要 Multi-Agent

回答要点：

```text
当任务复杂、需要多个专业角色、可以并行处理、需要独立审查或不同视角时，可以考虑 Multi-Agent。例如研究报告、代码开发加 review、复杂规划和多来源验证。但简单任务不适合 Multi-Agent，因为通信和协调成本会超过收益。是否使用多 Agent 要看它是否真正提升任务成功率、验证质量或并行效率。
```

## 9.17 面试题：如何设计 Multi-Agent 系统

回答要点：

```text
我会先明确任务是否需要多 Agent，然后设计 coordinator、角色分工、工具权限和通信协议。每个 Agent 只拿到完成任务所需的上下文和工具。共享状态通过结构化 blackboard 管理，冲突通过证据、工具验证、judge 或人工确认解决。系统还要记录 trace，评估任务成功率、成本、通信轮数、冲突解决和安全违规率。
```

## 9.18 本章小结

Multi-Agent 通过角色分工、并行执行和互相检查，让 Agent 系统有机会处理更复杂任务。但它不是免费午餐，会引入通信成本、协调复杂度、冲突解决、权限管理和安全风险。

可靠的 Multi-Agent 系统通常不是一群 Agent 自由聊天，而是有 coordinator、结构化消息、共享状态、明确权限、冲突解决和评估闭环的工程系统。下一章会进入 Agent 评估，系统讨论如何衡量 Agent 是否真的完成任务、是否安全、是否高效。
