# 第二章：Tool Use 与 Function Calling

工具调用是 Agent 能从“会说”变成“会做”的关键。模型本身只能生成文本，但通过工具可以检索资料、执行代码、查询数据库、读写文件、调用业务 API、操作浏览器或完成计算。Function calling 则是让模型按照结构化 schema 生成工具调用参数，使工具调用更可控、更可验证。

本章系统讲 tool use 与 function calling：工具 schema 如何设计，模型如何选择工具和生成参数，工具调用结果如何处理，调用失败如何恢复，如何做权限控制、日志审计和评估，以及面试中如何回答工具调用相关问题。

## 2.1 Tool Use 是什么

Tool use 指模型在生成回答之外，调用外部工具完成任务。

工具可以是：

1. 搜索引擎。
2. 数据库。
3. 计算器。
4. Python 解释器。
5. 文件系统。
6. 浏览器。
7. 邮件、日历、工单系统。
8. 内部业务 API。
9. RAG 检索器。

面试回答：

```text
Tool use 是让 LLM 通过外部工具获取信息或执行动作。模型负责理解任务、选择工具和生成参数，工具负责真实查询或执行，工具返回 observation 后模型再决定下一步。它可以弥补模型知识过期、不能计算、不能访问外部环境的问题，但也带来权限、安全、错误恢复和成本控制问题。
```

## 2.2 Function Calling 是什么

Function calling 是一种结构化工具调用方式。系统预先给模型提供工具名称、描述和参数 schema，模型不直接生成自然语言命令，而是生成符合 schema 的函数调用。

例如：

```json
{
  "name": "search_docs",
  "arguments": {
    "query": "agent tool calling error handling",
    "top_k": 5
  }
}
```

相比让模型自由写“帮我搜一下”，function calling 更容易解析、校验、执行和审计。

## 2.3 为什么需要 Schema

Schema 是工具调用的契约。它告诉模型：工具叫什么，能做什么，需要哪些参数，参数类型是什么，有哪些限制。

一个好的 schema 可以减少：

1. 参数缺失。
2. 参数类型错误。
3. 工具误用。
4. 不可解析输出。
5. 安全越权。
6. 执行失败。

一个差的 schema 会让模型猜参数、误解工具能力，甚至把不该传的内容传给工具。

## 2.4 工具 Schema 设计

工具 schema 通常包含：

1. name：工具名称。
2. description：工具用途。
3. parameters：参数结构。
4. required：必填字段。
5. enum：可选值范围。
6. constraints：长度、格式、范围限制。
7. examples：可选示例。
8. safety notes：安全限制。

示例：

```json
{
  "name": "query_order_status",
  "description": "查询订单状态，只能查询当前用户有权限访问的订单",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "订单 ID"
      }
    },
    "required": ["order_id"]
  }
}
```

description 要写清楚工具边界。不要写成“万能查询工具”，否则模型会把所有需求都塞给它。

## 2.5 工具选择

Tool selection 是模型决定是否调用工具、调用哪个工具。

常见决策：

1. 是否需要外部信息。
2. 是否需要计算。
3. 是否需要执行动作。
4. 哪个工具最适合。
5. 是否已有足够信息直接回答。
6. 是否需要用户确认。

工具选择错误是 Agent 常见失败来源。例如该查数据库时去搜索网页，该调用只读工具时调用写入工具，该直接回答时反复调用工具。

工程上可以通过工具描述、路由器、权限控制和任务类型分类来降低误选概率。

## 2.6 参数生成

参数生成是 function calling 中最容易出错的环节之一。

常见错误：

1. 参数缺失。
2. 类型错误。
3. 格式不符合要求。
4. 使用用户没有提供的信息。
5. 把自然语言解释塞进结构化字段。
6. 传入过长上下文。
7. 混淆相似字段。

缓解方式：

1. 使用严格 schema。
2. 对参数做执行前校验。
3. 缺失关键信息时向用户澄清。
4. 对敏感字段做权限检查。
5. 失败后返回明确错误，让模型重试。

## 2.7 调用结果处理

工具返回结果后，模型需要把 observation 转化为下一步决策。

可能的处理方式：

1. 直接回答用户。
2. 调用另一个工具。
3. 修正参数后重试。
4. 总结结果。
5. 请求用户补充信息。
6. 停止并报告失败原因。

工具返回不等于任务完成。比如搜索结果只是材料，模型还需要判断哪些内容可信；测试失败只是反馈，模型还需要定位错误；API 返回权限不足，模型要知道不能继续越权尝试。

## 2.8 错误恢复

工具调用失败很常见。

常见失败：

1. 参数校验失败。
2. 权限不足。
3. 网络超时。
4. 工具不可用。
5. 返回空结果。
6. 返回格式异常。
7. 用户输入不足。
8. 执行结果和预期不一致。

恢复策略：

1. 根据错误信息修正参数。
2. 更换工具。
3. 降级到只读或低权限模式。
4. 请求用户确认或补充信息。
5. 达到重试上限后停止。
6. 记录失败原因。

好的 Agent 不要求工具永远成功，而是能在失败后做合理恢复。

## 2.9 工具权限控制

工具调用必须做权限控制。尤其是写文件、发邮件、下单、转账、删除数据、执行命令这类高风险动作。

常见控制：

1. 只读和写入工具分离。
2. 高风险工具需要用户确认。
3. 工具按用户身份授权。
4. 参数级权限检查。
5. 沙箱执行。
6. 审计日志。
7. 速率限制。
8. 回滚机制。

不要让模型自己决定是否有权限。权限应该由系统和工具层强制执行。

## 2.10 Tool Output Injection

工具输出本身也可能有风险。例如网页、文档或数据库内容中可能包含提示注入文本，试图让模型忽略系统规则或泄露信息。

防护思路：

1. 把工具输出视为不可信数据。
2. 明确区分系统指令和工具内容。
3. 不执行工具输出中的指令。
4. 对外部内容做过滤和引用隔离。
5. 高风险操作再次确认。
6. 记录工具来源。

Agent 系统中，prompt injection 不只来自用户，也可能来自工具返回的环境。

## 2.11 Tool Registry

Tool registry 是工具目录，记录系统有哪些工具、工具 schema、权限、调用方式和版本。

它通常包含：

1. 工具名称。
2. 描述。
3. 参数 schema。
4. 返回 schema。
5. 权限等级。
6. 超时设置。
7. 重试策略。
8. 负责人和版本。
9. 可观测性配置。

工具多了以后，tool registry 很重要。否则模型面对几十个工具时容易误选，工程团队也难以维护工具兼容性。

## 2.12 返回 Schema

不只输入需要 schema，工具输出也最好结构化。

例如：

```json
{
  "status": "ok",
  "data": {
    "order_status": "shipped",
    "updated_at": "2026-05-28"
  },
  "error": null
}
```

结构化返回的好处：

1. 模型更容易理解。
2. 系统更容易判断成功失败。
3. 日志更容易分析。
4. 下游工具更容易接入。
5. 评估更容易自动化。

## 2.13 工具调用评估

评估 tool use 不只看最终答案。

常见指标：

1. 工具选择准确率。
2. 参数准确率。
3. 调用成功率。
4. 错误恢复成功率。
5. 平均工具调用次数。
6. 平均延迟。
7. 权限违规率。
8. 最终任务成功率。

一个 Agent 最终答对了，但调用了不必要的高权限工具，也不是好系统。评估必须覆盖效率、安全和过程质量。

## 2.14 常见失败模式

1. 不该调用工具时调用工具。
2. 该调用工具时直接编答案。
3. 选错工具。
4. 参数生成错误。
5. 工具失败后反复重试。
6. 忽略工具返回的错误。
7. 被工具输出注入误导。
8. 越权调用高风险工具。
9. 把工具返回内容无验证地当事实。
10. 最终回答没有引用工具结果。

这些失败说明，工具调用系统需要模型能力和工程约束共同配合。

## 2.15 面试题：Function Calling 解决什么问题

回答要点：

```text
Function calling 解决的是模型调用工具时的结构化和可控性问题。系统提供工具名称、描述和参数 schema，模型生成符合 schema 的调用参数，而不是自由生成自然语言命令。这样更容易解析、校验、执行、记录和评估，也能减少参数错误和工具误用。但它仍然需要权限控制、错误恢复和工具输出安全处理。
```

## 2.16 面试题：如何设计一个可靠工具调用系统

回答要点：

```text
我会先设计 tool registry，给每个工具定义清晰 schema、描述、权限等级、超时和返回格式。模型负责工具选择和参数生成，系统在执行前做参数校验和权限检查。工具返回 observation 后，Agent 决定继续调用、重试、请求用户补充还是停止。系统还要有日志、错误恢复、速率限制、prompt injection 防护和评估指标。
```

## 2.17 本章小结

Tool use 让 Agent 能获取外部信息和执行动作，function calling 让工具调用变得结构化、可验证和可审计。一个可靠工具调用系统不仅需要模型会选工具、填参数，还需要 schema 设计、权限控制、错误恢复、输出安全、日志追踪和评估体系。
