# 第十章：Agent 评估

Agent 评估比普通模型评估更难。普通问答可以看答案是否正确，Agent 还要看任务是否真正完成、工具是否调用正确、步骤是否高效、是否越权、是否能从失败中恢复、成本是否可接受，以及长期运行是否稳定。一个 Agent 最终给出漂亮总结，不代表它真的完成了任务。

本章系统讲 Agent 评估：任务成功率、步骤效率、工具调用质量、轨迹评估、长期可靠性、真实环境 benchmark、人工评估、自动评估、安全评估、成本评估和常见陷阱。

## 10.1 Agent 评估为什么难

Agent 的输出不只是文本，而是一串动作轨迹。

例如：

```text
理解目标 -> 搜索文件 -> 编辑代码 -> 运行测试 -> 修复失败 -> 总结
```

评估时要问：

1. 目标是否完成。
2. 工具是否选对。
3. 参数是否正确。
4. 是否有无关动作。
5. 是否安全。
6. 是否成本可控。
7. 是否能复现。
8. 最终总结是否忠实于实际操作。

面试回答：

```text
Agent 评估不能只看最终回答，因为 Agent 的价值来自多步执行。需要同时评估任务成功率、工具调用准确性、步骤效率、错误恢复、安全违规率、成本和 trace 可审计性。尤其是真实环境中，Agent 可能部分完成任务、误用工具或给出和实际执行不一致的总结。
```

## 10.2 任务成功率

任务成功率是最核心指标。

定义：Agent 是否完成用户目标。

例如：

1. Code Agent：测试是否通过。
2. Browser Agent：表单是否正确提交。
3. RAG Agent：答案是否由证据支持。
4. Data Agent：分析结果是否正确。
5. Workflow Agent：工单是否处理完成。

任务成功率需要明确验收标准。没有验收标准，评估会变成主观印象。

## 10.3 部分成功

Agent 任务常常不是简单成功或失败。

可能出现：

1. 完成了检索，但没有正确总结。
2. 修复了一个测试，但引入另一个失败。
3. 填写了表单，但漏了一个字段。
4. 找到了证据，但引用不准确。
5. 生成了计划，但没有执行。

因此可以设计分级评分：

```text
0 分：完全失败
1 分：部分理解目标
2 分：完成部分子任务
3 分：基本完成但有小问题
4 分：完全完成且验证通过
```

分级评分比二元成功率更适合复杂 Agent 任务。

## 10.4 工具调用评估

工具调用评估关注 Agent 如何使用工具。

指标包括：

1. 是否该调用工具。
2. 工具选择是否正确。
3. 参数是否正确。
4. 调用顺序是否合理。
5. 是否重复调用。
6. 是否忽略工具错误。
7. 是否调用高风险工具。
8. 工具结果是否被正确使用。

一个 Agent 最终答对，但中间调用了不必要的高权限工具，仍然存在风险。

## 10.5 步骤效率

Agent 不只是要做对，还要高效。

效率指标：

1. 步骤数。
2. 工具调用次数。
3. 总 token。
4. 总延迟。
5. 重试次数。
6. 无效动作比例。
7. 重复操作比例。

高步骤数不一定坏，复杂任务需要多步；但无意义重复和低效探索应该被扣分。

## 10.6 Trace 评估

Trace 是 Agent 的执行轨迹。

评估 trace 可以看：

1. 每一步是否有必要。
2. Action 是否与目标相关。
3. Observation 是否被正确理解。
4. 失败后是否合理恢复。
5. 最终结论是否忠实于 trace。
6. 是否有安全违规动作。

Trace 评估能发现最终答案看不出的错误。例如 Agent 声称测试通过，但 trace 中根本没有运行测试。

## 10.7 错误恢复能力

真实环境中工具失败很常见。

需要评估 Agent 是否能处理：

1. 参数错误。
2. 权限不足。
3. 网络超时。
4. 测试失败。
5. 搜索无结果。
6. 页面变化。
7. 工具返回格式异常。

好的 Agent 能根据错误类型调整策略，而不是机械重试。

## 10.8 长期可靠性

长期运行 Agent 会遇到更多问题。

评估维度：

1. 多轮任务状态是否保持。
2. 是否会忘记用户约束。
3. memory 是否被污染。
4. 是否出现成本漂移。
5. 是否能断点恢复。
6. 是否会累积错误。
7. 是否能处理环境变化。

短 demo 成功不代表长期可靠。Agent 上线前需要长任务和多轮回归测试。

## 10.9 真实环境 Benchmark

Agent 需要真实环境 benchmark。

例子：

1. 真实代码仓库修复任务。
2. 浏览器网页操作任务。
3. 企业知识库问答任务。
4. 数据分析任务。
5. 工单处理任务。
6. 多工具组合任务。

真实 benchmark 的难点是环境可复现。需要固定数据、初始状态、工具版本、权限和评估脚本。

## 10.10 Sandbox Benchmark

为了安全和可复现，Agent benchmark 常放在沙箱环境中。

沙箱需要提供：

1. 初始状态。
2. 可用工具。
3. 权限限制。
4. 标准答案或验收脚本。
5. 日志记录。
6. 重置机制。

代码任务可以用测试套件做验收；浏览器任务可以用页面状态检查；数据任务可以用结果文件或指标检查。

## 10.11 自动评估

自动评估适合可验证任务。

例如：

1. 测试是否通过。
2. 文件是否正确生成。
3. 数据库状态是否符合预期。
4. 页面是否到达目标状态。
5. API 调用是否成功。
6. 输出是否满足格式约束。

自动评估优势是可扩展、客观、便宜。缺点是覆盖有限，容易被过拟合，也可能漏掉安全和过程质量问题。

## 10.12 人工评估

人工评估适合开放任务和过程质量判断。

人工评估要看：

1. 任务是否真正完成。
2. 路径是否合理。
3. 是否有多余或危险操作。
4. 最终解释是否忠实。
5. 用户体验是否好。
6. 失败时是否清楚说明原因。

人工评估需要明确 rubric，否则不同评审者标准不一致。

## 10.13 LLM Judge

LLM judge 可以辅助评估 Agent trace 和最终结果。

适合：

1. 检查回答是否完整。
2. 判断证据是否支持结论。
3. 识别明显无关步骤。
4. 给失败案例分类。

风险：

1. 被流畅总结欺骗。
2. 忽略 trace 中的安全问题。
3. 偏好长答案。
4. 和被评模型共享偏差。

最好把 LLM judge 作为辅助，不要作为唯一评估依据。

## 10.14 安全评估

Agent 安全评估包括：

1. 是否越权调用工具。
2. 是否执行危险命令。
3. 是否泄露敏感信息。
4. 是否被工具输出注入影响。
5. 是否在高风险动作前请求确认。
6. 是否遵守只读和写入权限。
7. 是否记录审计日志。

Agent 的安全风险比普通聊天模型更高，因为它能执行动作。

## 10.15 成本和延迟评估

Agent 通常比普通模型调用更贵。

需要记录：

1. 模型调用次数。
2. token 消耗。
3. 工具调用次数。
4. 工具执行时间。
5. 总延迟。
6. 缓存命中率。
7. 失败重试成本。

评估时要看 quality-cost trade-off。一个成功率提升 1% 但成本增加 10 倍的方案，未必值得上线。

## 10.16 回归评估

Agent 系统更新后容易引入回归。

需要固定回归集：

1. 常见成功任务。
2. 历史失败任务。
3. 安全边界任务。
4. 工具异常任务。
5. 长上下文任务。
6. 多轮任务。

每次更新 prompt、工具 schema、模型版本或 controller，都应跑回归评估。

## 10.17 常见评估陷阱

1. 只看最终回答，不看 trace。
2. 只评估 happy path。
3. 忽略工具错误和失败恢复。
4. 不报告成本和延迟。
5. 用不稳定环境评估。
6. 人工 rubric 不清楚。
7. LLM judge 作为唯一裁判。
8. 忽略安全违规。
9. 忽略用户未提交改动或环境状态。
10. benchmark 过于简单。

Agent 评估必须贴近真实任务，否则很容易高估系统能力。

## 10.18 面试题：如何评估 Agent 系统

回答要点：

```text
我会从任务成功率、工具调用质量、步骤效率、错误恢复、安全和成本几个维度评估。首先定义每类任务的验收标准，再记录完整 trace，检查工具选择、参数、observation 使用和最终结果是否一致。可验证任务用自动评估，例如测试通过或页面状态；开放任务结合人工 rubric 和 LLM judge。还要报告 token、延迟、工具调用次数和安全违规率。
```

## 10.19 面试题：为什么 Agent 不能只看最终答案

回答要点：

```text
因为 Agent 的价值和风险都在执行过程中。最终答案看起来正确，不代表它真的运行了测试、没有越权调用工具、没有误删文件、没有忽略错误反馈。评估 Agent 必须看 trace，包括每一步 action、工具参数、observation、失败恢复和最终总结是否忠实于实际执行。
```

## 10.20 本章小结

Agent 评估要从文本答案评估扩展到任务执行评估。核心指标是任务成功率，但还要评估工具调用、步骤效率、错误恢复、长期可靠性、安全、成本和 trace 忠实性。

一个可靠的 Agent benchmark 应该有明确初始状态、可用工具、权限限制、验收标准和可复现日志。下一章会进入 Agent 安全，专门讨论工具权限、prompt injection、越权操作、数据泄露和高风险任务控制。
