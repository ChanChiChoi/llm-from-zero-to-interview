## Language Model

一句话定义：语言模型是为 token 序列分配概率的模型。

核心直觉：它学习“在某个上下文中，接下来最可能出现什么”。如果模型看到“北京是中国的”，它应该给“首都”更高概率，而不是给一个无关 token 更高概率。

概率视角：给定序列 `x_1, x_2, ..., x_T`，语言模型要估计整个序列出现的概率。自回归语言模型会把它分解成多个条件概率：

```text
P(x_1, x_2, ..., x_T) = P(x_1) * P(x_2 | x_1) * ... * P(x_T | x_{1:T-1})
```

为什么重要：文本生成、翻译、问答、摘要、代码补全、对话、工具调用都可以被统一成“根据上下文继续生成 token”的问题。大语言模型之所以通用，核心在于它把很多任务都转化成语言建模问题。

与传统分类模型的区别：分类模型通常在固定标签集合中选择答案，语言模型在整个词表上预测下一个 token。它的输出空间更大，也更灵活，因此可以生成开放式答案，但也更难控制真实性、格式和安全性。

面试表达：语言模型不是简单的词频统计表，而是一个条件概率模型。现代 LLM 用 Transformer 表示上下文，通过 next-token prediction 学到语言规律、知识关联、任务模式和推理模板。

常见追问：语言模型和搜索引擎有什么区别？搜索引擎主要检索已有文档，语言模型主要根据参数和上下文生成文本。RAG 系统会把两者结合，用检索提供外部依据，再让语言模型组织答案。

## Large Language Model / LLM

一句话定义：大语言模型是参数规模、训练数据规模和训练计算量都很大的语言模型，通常具备较强的通用文本理解、生成和任务迁移能力。

核心直觉：LLM 不是一个人工编写的规则库，也不是简单存储问答对的数据库。它通过海量 token 训练，把语言模式、世界知识、代码模式、数学表达、对话风格和任务格式压缩进参数中。

为什么“大”有意义：模型规模扩大后，表示能力提升；训练 token 增多后，覆盖的语言现象、知识和任务形式更丰富；计算量提升后，模型有机会从数据中学到更复杂的统计结构。三者共同决定 LLM 的能力上限。

典型能力：文本生成、问答、摘要、翻译、代码生成、结构化抽取、分类、改写、推理辅助、多轮对话、工具调用规划等。

典型局限：LLM 优化的基础目标仍是 token 预测，不等于直接优化真理、逻辑正确性和用户满意度。因此它可能幻觉、过度自信、对提示敏感、长上下文退化、在分布外任务上失败。

工程视角：一个完整 LLM 产品通常不只是 base model，还包括 tokenizer、prompt 模板、后训练、对齐、安全策略、RAG、工具调用、推理加速、监控评估和反馈闭环。

面试表达：LLM 的本质是大规模条件概率模型，但工程价值来自规模化预训练带来的通用表示能力，以及后训练把这种能力对齐到用户指令和业务场景。

## Next-Token Prediction

一句话定义：next-token prediction 是给定前文 token，预测下一个 token 的训练目标。

核心公式：

```text
P(x_t | x_1, x_2, ..., x_{t-1})
```

为什么提出：互联网上有大量未标注文本，如果要求人工给每条文本打标签，成本不可接受。next-token prediction 把文本自身变成监督信号：前面的 token 是输入，后一个 token 是标签。

训练过程：模型输入一段 token 序列，通过 causal mask 保证第 `t` 个位置只能看到 `t` 之前的信息，然后预测每个位置的下一个 token。对所有位置的预测计算 cross entropy loss，再反向传播更新参数。

优点：目标简单，数据便宜，扩展性强，天然适合生成任务。它不需要为每个下游任务单独设计标签，因此可以支撑大规模通用预训练。

局限：预测下一个 token 不直接保证事实正确、安全合规、推理严密或完全符合用户偏好。后续需要 instruction tuning、RLHF、DPO、RAG、工具调用和评估体系来补足。

面试表达：next-token prediction 是 LLM 预训练最核心的自监督目标。它看似简单，但在足够大模型、数据和算力下，可以让模型学到大量可迁移能力。

## Autoregressive Model

一句话定义：自回归模型按从左到右的顺序生成序列，每一步都基于已经生成的历史 token 预测下一个 token。

核心公式：

```text
P(x_1, ..., x_T) = ∏_t P(x_t | x_{1:t-1})
```

典型模型：GPT、LLaMA、Qwen、DeepSeek、Mistral 等 decoder-only LLM 通常都是自回归模型。

为什么适合生成：生成文本本来就是逐步展开的过程。自回归建模和推理方式一致，每次用已有上下文预测下一个 token，再把新 token 拼回上下文继续预测。

与 masked language model 对比：masked LM 类模型如 BERT 训练时预测被遮盖位置，可以双向看上下文，适合理解和表征任务；自回归 LM 只能看左侧上下文，天然适合开放式生成。

工程限制：自回归推理阶段无法在生成长度维度完全并行，因为第 `t+1` 个 token 依赖第 `t` 个 token。KV Cache、speculative decoding、continuous batching 等技术主要是在降低这个串行过程的成本。

面试表达：自回归模型通过链式法则建模序列概率，训练时可以并行计算所有位置的 loss，但推理时必须逐 token 生成。

## Teacher Forcing

一句话定义：teacher forcing 是训练时使用真实历史 token 作为上下文来预测下一个 token 的方法。

核心直觉：训练时模型看到的是正确前缀，而不是自己刚生成的前缀。这样每个位置的预测都可以稳定计算，并且所有位置可以并行训练。

例子：训练句子“我 喜欢 机器 学习”时，模型在“我 喜欢”之后预测“机器”，即使它当前还不会正确生成“喜欢”，训练时仍然使用真实 token 作为上下文。

优点：训练稳定、并行效率高、实现简单，是 causal LM 训练的标准做法。

局限：训练和推理存在暴露偏差。推理时模型必须依赖自己的历史输出，一旦前面生成错误，后续上下文也会被污染，可能导致错误累积。

面试表达：teacher forcing 让自回归语言模型训练可以并行化，但它也解释了为什么模型推理时会出现生成漂移和错误累积。

## Label Shift

一句话定义：label shift 是 causal language model 训练中输入 token 与监督 label 相差一个位置的做法。

例子：输入 token 是 `[x_1, x_2, x_3]`，对应目标 label 是 `[x_2, x_3, x_4]`。模型在第一个位置看到 `x_1`，要预测 `x_2`；在第二个位置看到 `x_1, x_2`，要预测 `x_3`。

为什么需要：语言模型的目标是预测下一个 token，而不是重构当前 token。如果没有正确 shift，模型可能学到复制当前位置 token，loss 看起来下降，但生成能力错误。

工程坑：不同框架对 shift 的处理位置不同。有些模型内部会自动 shift logits 和 labels，有些训练代码需要外部手动处理。重复 shift 或漏 shift 都会造成训练异常。

排查方式：用一个极小 batch 打印 input ids、labels 和 loss mask，确认每个位置的 label 是否真的是下一个 token，并确认 padding 位置是否被 `ignore_index` 屏蔽。

面试表达：label shift 是 causal LM 训练正确性的基础细节。很多微调 loss 异常、模型学会复读、输出错位，都可能和 shift 或 mask 错误有关。

## Logits

一句话定义：logits 是模型在 softmax 之前输出的未归一化分数。

形状：对于语言模型，logits 通常形状为 `[batch_size, seq_len, vocab_size]`。最后一维中每个值对应一个 token 的得分。

核心直觉：logit 越大，模型越倾向于选择该 token。但 logit 本身不是概率，不能直接解释为“70% 可能性”。需要经过 softmax 才能变成概率分布。

与解码关系：greedy decoding 选择 logit 最大的 token；sampling 会先经过 temperature、top-k、top-p 等处理，再从概率分布中采样。

数值注意：softmax 对 logit 差值敏感，而不是对绝对值敏感。所有 logits 同时加同一个常数，softmax 后概率不变。

面试表达：logits 是语言模型对整个词表的原始打分，后续的 softmax、采样策略和 loss 计算都基于它。

## Softmax

一句话定义：softmax 是把 logits 转换成概率分布的函数。

核心公式：

```text
softmax(z_i) = exp(z_i) / Σ_j exp(z_j)
```

作用：它让每个 token 的概率非负，并且所有 token 概率和为 1。

temperature 影响：temperature 会缩放 logits。温度低时分布更尖锐，模型更确定；温度高时分布更平滑，输出更多样但也更容易不稳定。

与 cross entropy：训练时通常对 logits 做 cross entropy。实现中常用 fused cross entropy，不会显式先算 softmax 再算 log，以提高数值稳定性。

面试表达：softmax 把模型对词表的原始打分归一化成概率分布，是训练 loss 和生成采样之间的桥梁。

## Causal Language Model

一句话定义：causal language model 是只能根据过去 token 预测当前或未来 token 的语言模型。

关键机制：causal mask 会屏蔽未来位置，保证第 `t` 个位置不能看到 `t+1` 之后的信息。如果没有 causal mask，训练 loss 会虚低，因为模型可以偷看答案。

典型架构：decoder-only Transformer。它没有 encoder-decoder 架构中的独立 encoder，也不使用 BERT 式双向 attention。

训练目标：通常是 next-token prediction。模型每个位置输出一个 hidden state，再通过 lm head 映射到词表 logits。

推理过程：输入 prompt，模型取最后一个位置的 logits 得到下一个 token 分布，选出新 token 后追加到上下文，再重复上述过程。

面试表达：causal LM 的“causal”指信息流方向受限，只允许从过去到未来，这正是 GPT 类模型可以做自回归生成的原因。

## Base Model

一句话定义：base model 是经过大规模预训练但尚未经过指令微调和偏好对齐的基础模型。

典型表现：它擅长续写文本，但不一定会像聊天助手一样回答问题。如果输入“请解释 self-attention”，base model 可能继续生成类似网页、论文或问答语料的文本，而不一定稳定遵循指令。

为什么需要 base model：base model 是通用能力底座。它通过预训练获得语言、知识、代码、数学和世界模式，后续 instruct model、chat model 和领域模型都通常建立在 base model 之上。

优点：分布更接近原始预训练目标，适合继续预训练、领域适配、作为研究底座，也常用于分析模型原始能力。

局限：指令遵循弱，对话格式弱，安全边界弱，输出可能不符合用户期望。直接面向用户产品通常需要后训练。

面试表达：base model 是“会续写”的模型，不是天然“会听话”的助手。instruct tuning 和 alignment 的目标就是把它变成更会按用户意图工作的模型。

## Instruct Model

一句话定义：instruct model 是经过指令微调后，能够更好理解并遵循用户指令的模型。

训练数据：常见数据形式包括 instruction、input、response 三元组，也包括问答、摘要、改写、代码解释、分类、推理等任务样本。

解决的问题：base model 只是预测文本续写，不一定知道用户希望它完成什么任务。指令微调让模型学习“看到任务描述后输出合适答案”的行为模式。

能力变化：instruct model 通常在零样本任务、格式遵循、问题回答、解释性输出方面明显优于 base model。但如果微调数据质量差，也可能导致能力退化、风格变窄或过度拒答。

与 chat model 的区别：instruct model 强调单轮或通用指令遵循；chat model 更强调多轮对话格式、角色消息、上下文承接和安全策略。

面试表达：instruct model 的核心是把预训练得到的通用能力映射到“按指令完成任务”的交互方式上。

## Chat Model

一句话定义：chat model 是面向多轮对话场景优化的语言模型，通常能处理 system、user、assistant 等角色消息。

关键特征：它不仅要回答当前问题，还要理解对话历史、遵循系统指令、保持角色一致、处理多轮追问，并在必要时拒绝不安全请求。

数据格式：chat model 通常使用聊天模板把多轮消息序列化成 token，例如 system message、user message、assistant response 加上特殊分隔符。

工程注意：chat template 非常关键。同一个模型如果使用错误聊天模板，可能出现不回答、乱续写、角色混乱、特殊 token 泄漏等问题。

与 base model 对比：base model 更像文本续写器，chat model 更像对话助手。差异不只是模型权重，也包括 tokenizer、special token、prompt 模板和对齐策略。

面试表达：chat model 是经过对话数据和偏好对齐塑形后的 LLM，重点能力是多轮交互、指令层级遵循和助手风格输出。

## Assistant Model

一句话定义：assistant model 是被设计成通用助手形态的 chat model，目标是在安全边界内帮助用户完成任务。

核心要求：有帮助性、诚实性、无害性。它需要尽量解决用户问题，不知道时避免编造，同时拒绝明显有害请求。

产品化差异：一个可用助手通常还需要工具调用、检索增强、代码执行、文件读写、记忆管理、监控告警和人工反馈闭环。模型本身只是系统的一部分。

常见问题：过度迎合用户、幻觉、拒答过度、长对话遗忘、工具调用错误、格式不稳定、对系统提示被攻击敏感。

面试表达：assistant model 不是单纯追求最大似然文本生成，而是在模型能力、用户意图、安全约束和工具环境之间做平衡。

## Prompt

一句话定义：prompt 是提供给模型的输入上下文，用来说明任务、约束、背景和期望输出。

组成：prompt 可以包含系统指令、用户问题、示例、上下文资料、输出格式要求、工具说明和约束条件。

为什么有效：LLM 具备 in-context learning 能力，可以从 prompt 中临时识别任务模式。高质量 prompt 能降低任务歧义，让模型更容易生成符合预期的答案。

局限：prompt 不是可靠的硬约束。复杂格式、强安全边界、事实性和长期一致性不能只依赖 prompt，需要训练、检索、工具、校验和评估配合。

面试表达：prompt 是运行时控制模型行为的主要接口，但它只能调动模型已有能力，不能凭空创造模型没有学到的稳定能力。

## System Prompt

一句话定义：system prompt 是对模型行为、角色、安全边界和输出风格进行高优先级约束的提示。

作用：它告诉模型应该扮演什么角色、遵循哪些规则、优先满足哪些约束，以及如何处理冲突指令。

工程场景：客服助手、代码助手、数据分析助手、医疗问答助手等都会使用 system prompt 固定身份、能力边界和输出规范。

风险：system prompt 可能被 prompt injection 攻击绕过，也可能和用户需求冲突。真实系统需要结合权限控制、工具隔离、内容审核和输出校验。

面试表达：system prompt 是 LLM 应用的行为配置层，不是安全系统的全部。它能提高一致性，但不能替代模型对齐和工程防护。

## Instruction Following

一句话定义：instruction following 是模型理解并执行用户自然语言指令的能力。

衡量维度：是否完成任务、是否遵守格式、是否覆盖约束、是否拒绝越界要求、是否在多约束冲突时合理取舍。

训练来源：指令微调、偏好对齐、高质量人工数据、合成指令数据、多轮对话数据和格式遵循数据。

常见失败：忽略部分约束、输出格式错误、回答跑题、在多轮对话中忘记先前要求、对否定指令理解错误。

面试表达：指令遵循不是简单问答能力，而是把自然语言任务描述转化成可执行生成行为的能力。

## Context Window

一句话定义：context window 是模型一次推理中能够接收和处理的最大上下文 token 长度。

包括什么：上下文窗口包含 system prompt、用户输入、对话历史、检索文档、工具结果以及模型已经生成的 token。生成越长，可用输入空间越少。

为什么重要：长上下文可以容纳更多文档、代码、对话历史和复杂任务信息。但上下文更长不等于模型一定更会利用信息。

常见现象：模型可能在长上下文中忽略中间信息，也就是 lost in the middle；也可能因为检索内容太多、噪声太大而回答变差。

工程策略：做上下文压缩、检索重排、重要信息前置或后置、分块摘要、引用定位、长上下文评估和成本控制。

面试表达：context window 决定模型“能看多少”，但有效上下文利用能力决定模型“看了多少有用信息”。

## Generation

一句话定义：generation 是模型根据已有上下文逐 token 产生新 token 的过程。

基本步骤：输入 prompt，模型 forward 得到最后位置 logits，解码策略选择下一个 token，把新 token 拼接到上下文，重复直到遇到停止条件。

停止条件：达到最大生成长度、生成 EOS token、命中 stop words、工具调用中断或系统策略终止。

解码方式：常见策略包括 greedy decoding、beam search、temperature sampling、top-k sampling、top-p sampling。不同策略会影响确定性、多样性和事实稳定性。

工程注意：生成质量不只由模型权重决定，也受 prompt、聊天模板、temperature、top-p、max tokens、重复惩罚、stop 配置和后处理影响。

面试表达：LLM 生成是自回归闭环过程。每一步的输出都会成为下一步的输入，因此早期错误可能被放大。

## Perplexity

一句话定义：perplexity 是衡量语言模型预测测试集 token 困难程度的指标，通常可以理解为 cross entropy 的指数形式。

核心公式：

```text
PPL = exp(loss)
```

其中 loss 通常是按 token 平均的 negative log likelihood。

直觉解释：perplexity 越低，说明模型对真实下一个 token 的平均概率越高，也就是越“不困惑”。

用途：预训练监控、模型规模对比、数据质量判断、领域继续预训练效果评估。若领域语料上的 PPL 下降，通常说明模型更适应该领域分布。

局限：PPL 不是万能指标。低 PPL 不一定代表指令遵循好、推理强、安全性好或用户体验好。不同 tokenizer、不同数据清洗方式、不同上下文长度下的 PPL 也不宜直接横向比较。

面试表达：perplexity 是语言建模能力的基础指标，适合看模型对文本分布的拟合程度，但不能替代下游任务评测和人工评测。

## Emergent Ability

一句话定义：emergent ability 指模型规模、数据或计算量增长到一定程度后，在某些任务上表现突然明显提升的现象。

典型例子：多步推理、少样本学习、复杂指令遵循、代码能力、数学题求解等常被讨论为规模扩大后更明显的能力。

为什么会出现：一种解释是模型容量和数据覆盖足够后，学到了更复杂的组合模式；另一种解释是评测指标可能是离散或非线性的，使连续能力提升看起来像突然出现。

争议：emergent ability 不是魔法，也不意味着只要模型变大就必然获得所有能力。数据质量、训练目标、评测方式、prompt 设计和后训练都会影响观察结果。

面试表达：涌现能力描述的是规模化后某些能力的非线性表现，但面试中要避免神秘化，应该从容量、数据、优化和评测指标四个角度解释。

## In-Context Learning

一句话定义：in-context learning 是模型在不更新参数的情况下，从 prompt 中的说明和示例临时识别任务并完成任务的能力。

核心直觉：用户在 prompt 里给几个输入输出样例，模型可以在上下文中归纳“这个任务要怎么做”，然后对新输入生成同样格式的输出。

例子：

```text
文本：这家餐厅太好吃了
情感：正面

文本：等了一个小时还没上菜
情感：负面

文本：服务员很热情
情感：
```

模型可能根据前两个样例输出“正面”。

与微调区别：in-context learning 不改变模型参数，只在当前上下文中生效；微调会更新模型权重，使行为在后续调用中持续改变。

优点：灵活、低成本、无需训练、适合快速适配新格式。局限是受上下文长度限制，对示例顺序、示例质量和 prompt 表达敏感。

面试表达：in-context learning 可以理解为模型利用上下文进行临时任务归纳，而不是在推理时真的做梯度下降更新参数。

## Few-Shot Learning

一句话定义：few-shot learning 是在 prompt 中提供少量示例，让模型根据示例完成新样本的任务方式。

作用：示例能明确任务边界、输出格式、标签空间和隐含规则，通常比单纯文字说明更稳定。

适用场景：分类标签不直观、输出格式严格、业务规则特殊、模型容易误解任务时，few-shot prompt 往往很有效。

示例选择：示例要覆盖典型情况和容易混淆的边界情况，格式保持一致，避免示例本身包含错误标签或多余噪声。

局限：示例会占用上下文窗口；示例过多可能引入冲突；模型可能模仿示例中的偏差；少样本效果不等于真实泛化能力一定强。

面试表达：few-shot 的价值是用少量上下文示例约束模型行为，尤其是约束输出格式和任务映射关系。

## Zero-Shot Learning

一句话定义：zero-shot learning 是不提供任务示例，只通过任务描述让模型直接完成任务。

例子：直接输入“请判断下面评论的情感是正面还是负面：这家店服务很好”，模型输出“正面”。

为什么可行：大模型预训练和指令微调中见过大量任务描述和解题模式，因此能把自然语言指令映射到已学到的能力上。

优点：prompt 简洁，不占示例空间，适合标准任务和模型已熟悉的任务。缺点是对复杂规则、特殊格式和业务边界不如 few-shot 稳定。

评估注意：zero-shot 表现受 prompt wording 影响很大。同一个任务换一种问法，结果可能明显变化，因此评测时要固定提示模板或做多模板评估。

面试表达：zero-shot 体现的是模型从自然语言指令直接泛化到任务的能力，是衡量 instruction following 和预训练知识迁移的重要方式。

## Chain-of-Thought

一句话定义：chain-of-thought 是让模型生成中间推理步骤，再给出最终答案的提示或训练方式。

为什么有效：复杂问题往往需要多步计算、条件判断或中间状态。让模型显式展开推理过程，可以降低一步到位输出答案的难度。

典型场景：数学应用题、逻辑推理、多跳问答、代码推演、计划生成等。

局限：模型写出的推理步骤不一定真实反映内部计算，也可能出现“步骤看起来合理但结论错误”。在生产系统中，最终答案仍需要验证器、工具或规则校验。

安全注意：有些产品会把详细隐式推理隐藏，只输出简洁解释或关键依据，以降低泄漏内部策略和增加冗长错误的风险。

面试表达：CoT 的本质是把复杂任务分解成中间 token 序列，让模型有更多计算轨迹可用，但它不是正确性的充分保证。

## Memorization vs Generalization

一句话定义：memorization 是模型复现训练数据中的具体内容，generalization 是模型把学到的规律应用到未见样本。

核心直觉：大模型既会记忆，也会泛化。高频文本、重复数据、特殊字符串、代码片段和评测答案更容易被记住；抽象语法、任务格式和推理模式更接近泛化。

如何区分：可以用去重评测集、时间切分评测、扰动测试、反事实样本、私有测试集和数据污染检查来判断模型是否只是记住答案。

为什么重要：如果模型在 benchmark 上表现好只是因为训练集中见过题目，评估结果就不可信。数据污染是大模型评测中的核心风险之一。

面试表达：不能简单说大模型“只是背题”或“完全理解”。更准确的说法是，它在参数中同时包含记忆和可泛化模式，评估设计要尽量区分二者。

## Hallucination

一句话定义：hallucination 是模型生成看似合理但不真实、无依据或与上下文冲突的内容。

根本原因：语言模型优化的是 token 概率，而不是事实数据库查询。一个答案只要在语言模式上合理，就可能被模型高置信度生成，即使事实错误。

常见类型：编造引用、编造论文、编造 API、错误归因、数字计算错误、把过时知识当成事实、在上下文没有依据时强行回答。

缓解方法：RAG 提供外部依据，工具调用进行计算和查询，训练中加入拒答和不确定性表达，评估中加入事实性检查，产品中加入引用、校验和人工审核。

局限：幻觉不能被完全消除，只能在具体场景中通过数据、模型、工具和流程降低风险。

面试表达：幻觉来自生成式目标和事实性目标之间的不一致。解决思路不是只靠更大模型，而是结合检索、工具、校验、对齐和评估闭环。

## 本章小结

本章从语言模型的概率定义出发，串起了大语言模型的基础概念。面试中要抓住一条主线：LLM 通过 next-token prediction 预训练得到 base model，再通过指令微调和对话对齐变成 instruct model 或 chat model，最终通过 prompt、上下文、解码策略和外部工具服务具体任务。

高频结论如下：

1. 语言模型本质上是在建模 token 序列概率。
2. next-token prediction 是大规模自监督预训练的核心目标。
3. 自回归模型训练时可并行算 loss，推理时必须逐 token 生成。
4. base model 擅长续写，instruct model 擅长遵循指令，chat model 擅长多轮对话。
5. perplexity 能衡量语言建模拟合程度，但不能替代任务评测。
6. in-context learning 不更新参数，只利用当前上下文临时适配任务。
7. 涌现能力要从规模、数据、优化和评测指标共同理解。
8. 幻觉是生成式语言模型的结构性问题，需要检索、工具、校验和对齐共同缓解。
