## Language Model

一句话定义：语言模型是为 token 序列分配概率的模型。

核心直觉：它学习“在某个上下文中，接下来最可能出现什么”。如果模型看到“北京是中国的”，它应该给“首都”更高概率，而不是给一个无关 token 更高概率。

概率视角：给定序列 `x_1, x_2, ..., x_T`，语言模型要估计整个序列出现的概率。自回归语言模型会把它分解成多个条件概率：

```text
P(x_1, x_2, ..., x_T) = P(x_1) * P(x_2 | x_1) * ... * P(x_T | x_{1:T-1})
```

为什么重要：文本生成、翻译、问答、摘要、代码补全、对话、工具调用都可以被统一成“根据上下文继续生成 token”的问题。大语言模型之所以通用，核心在于它把很多任务都转化成语言建模问题。

与传统分类模型的区别：分类模型通常在固定标签集合中选择答案，语言模型在整个词表上预测下一个 token。它的输出空间更大，也更灵活，因此可以生成开放式答案，但也更难控制真实性、格式和安全性。

面试表达：语言模型不是简单的词频统计表，而是一个条件概率模型。现代 LLM 用 Transformer 表示上下文，通过 next-token prediction 学到语言规律、知识关联、任务模式和推理模板。

常见追问：语言模型和搜索引擎有什么区别？搜索引擎主要检索已有文档，语言模型主要根据参数和上下文生成文本。RAG 系统会把两者结合，用检索提供外部依据，再让语言模型组织答案。

## Large Language Model / LLM

一句话定义：大语言模型是参数规模、训练数据规模和训练计算量都很大的语言模型，通常具备较强的通用文本理解、生成和任务迁移能力。

核心直觉：LLM 不是一个人工编写的规则库，也不是简单存储问答对的数据库。它通过海量 token 训练，把语言模式、世界知识、代码模式、数学表达、对话风格和任务格式压缩进参数中。

为什么“大”有意义：模型规模扩大后，表示能力提升；训练 token 增多后，覆盖的语言现象、知识和任务形式更丰富；计算量提升后，模型有机会从数据中学到更复杂的统计结构。三者共同决定 LLM 的能力上限。

典型能力：文本生成、问答、摘要、翻译、代码生成、结构化抽取、分类、改写、推理辅助、多轮对话、工具调用规划等。

典型局限：LLM 优化的基础目标仍是 token 预测，不等于直接优化真理、逻辑正确性和用户满意度。因此它可能幻觉、过度自信、对提示敏感、长上下文退化、在分布外任务上失败。

工程视角：一个完整 LLM 产品通常不只是 base model，还包括 tokenizer、prompt 模板、后训练、对齐、安全策略、RAG、工具调用、推理加速、监控评估和反馈闭环。

面试表达：LLM 的本质是大规模条件概率模型，但工程价值来自规模化预训练带来的通用表示能力，以及后训练把这种能力对齐到用户指令和业务场景。

## Next-Token Prediction

一句话定义：next-token prediction 是给定前文 token，预测下一个 token 的训练目标。

核心公式：

```text
P(x_t | x_1, x_2, ..., x_{t-1})
```

为什么提出：互联网上有大量未标注文本，如果要求人工给每条文本打标签，成本不可接受。next-token prediction 把文本自身变成监督信号：前面的 token 是输入，后一个 token 是标签。

训练过程：模型输入一段 token 序列，通过 causal mask 保证第 `t` 个位置只能看到 `t` 之前的信息，然后预测每个位置的下一个 token。对所有位置的预测计算 cross entropy loss，再反向传播更新参数。

优点：目标简单，数据便宜，扩展性强，天然适合生成任务。它不需要为每个下游任务单独设计标签，因此可以支撑大规模通用预训练。

局限：预测下一个 token 不直接保证事实正确、安全合规、推理严密或完全符合用户偏好。后续需要 instruction tuning、RLHF、DPO、RAG、工具调用和评估体系来补足。

面试表达：next-token prediction 是 LLM 预训练最核心的自监督目标。它看似简单，但在足够大模型、数据和算力下，可以让模型学到大量可迁移能力。

## Autoregressive Model

一句话定义：自回归模型按从左到右的顺序生成序列，每一步都基于已经生成的历史 token 预测下一个 token。

核心公式：

```text
P(x_1, ..., x_T) = ∏_t P(x_t | x_{1:t-1})
```

典型模型：GPT、LLaMA、Qwen、DeepSeek、Mistral 等 decoder-only LLM 通常都是自回归模型。

为什么适合生成：生成文本本来就是逐步展开的过程。自回归建模和推理方式一致，每次用已有上下文预测下一个 token，再把新 token 拼回上下文继续预测。

与 masked language model 对比：masked LM 类模型如 BERT 训练时预测被遮盖位置，可以双向看上下文，适合理解和表征任务；自回归 LM 只能看左侧上下文，天然适合开放式生成。

工程限制：自回归推理阶段无法在生成长度维度完全并行，因为第 `t+1` 个 token 依赖第 `t` 个 token。KV Cache、speculative decoding、continuous batching 等技术主要是在降低这个串行过程的成本。

面试表达：自回归模型通过链式法则建模序列概率，训练时可以并行计算所有位置的 loss，但推理时必须逐 token 生成。

## Teacher Forcing

一句话定义：teacher forcing 是训练时使用真实历史 token 作为上下文来预测下一个 token 的方法。

核心直觉：训练时模型看到的是正确前缀，而不是自己刚生成的前缀。这样每个位置的预测都可以稳定计算，并且所有位置可以并行训练。

例子：训练句子“我 喜欢 机器 学习”时，模型在“我 喜欢”之后预测“机器”，即使它当前还不会正确生成“喜欢”，训练时仍然使用真实 token 作为上下文。

优点：训练稳定、并行效率高、实现简单，是 causal LM 训练的标准做法。

局限：训练和推理存在暴露偏差。推理时模型必须依赖自己的历史输出，一旦前面生成错误，后续上下文也会被污染，可能导致错误累积。

面试表达：teacher forcing 让自回归语言模型训练可以并行化，但它也解释了为什么模型推理时会出现生成漂移和错误累积。

## Label Shift

一句话定义：label shift 是 causal language model 训练中输入 token 与监督 label 相差一个位置的做法。

例子：输入 token 是 `[x_1, x_2, x_3]`，对应目标 label 是 `[x_2, x_3, x_4]`。模型在第一个位置看到 `x_1`，要预测 `x_2`；在第二个位置看到 `x_1, x_2`，要预测 `x_3`。

为什么需要：语言模型的目标是预测下一个 token，而不是重构当前 token。如果没有正确 shift，模型可能学到复制当前位置 token，loss 看起来下降，但生成能力错误。

工程坑：不同框架对 shift 的处理位置不同。有些模型内部会自动 shift logits 和 labels，有些训练代码需要外部手动处理。重复 shift 或漏 shift 都会造成训练异常。

排查方式：用一个极小 batch 打印 input ids、labels 和 loss mask，确认每个位置的 label 是否真的是下一个 token，并确认 padding 位置是否被 `ignore_index` 屏蔽。

面试表达：label shift 是 causal LM 训练正确性的基础细节。很多微调 loss 异常、模型学会复读、输出错位，都可能和 shift 或 mask 错误有关。

## Assistant-Only Label Mask

一句话定义：assistant-only label mask 是 SFT 中只让 assistant response token 参与 loss，把 system、user、control token 和 padding 位置设为 ignore index 的做法。

为什么重要：SFT 的目标是让模型在给定 system / user 上下文后生成 assistant 答案，而不是学习复述提示词。如果 user token 参与 loss，模型可能更容易复述用户输入或污染角色边界。

工程检查：打印 input ids、decode 文本、role span、labels 和 ignore index；确认所有 assistant 内容和必要 EOS 被训练，prompt 与 PAD 不参与 loss。

面试表达：assistant-only mask 是聊天模型 SFT 的目标边界。很多“模型不听话”不是能力问题，而是 mask 把条件上下文也当成了目标。

## Prompt Loss Leak

一句话定义：prompt loss leak 是 system、user、工具 schema、控制 token 或多模态占位符等提示部分错误参与训练 loss 的问题。

表现：模型复述用户问题、生成 system / user 角色文本、工具调用前缀混乱、对话边界不稳定，或者看似 loss 很低但指令跟随变差。

排查方式：按 role 统计 label 不等于 ignore index 的比例，prompt 侧应接近 0；多轮 SFT 要逐轮检查，不只检查最后一轮。

面试表达：prompt 是条件，不是目标。除非有非常明确的预训练或格式建模设计，SFT 中 prompt 参与 loss 通常是事故。

## PAD Loss Leak

一句话定义：PAD loss leak 是 padding token 没有被 attention mask 和 label mask 正确屏蔽，导致 PAD 位置参与 loss 的事故。

表现：长短样本 loss 差异异常、batch padding 多时训练不稳定、模型生成 padding 相关 token 或结束行为异常。

排查方式：检查 `attention_mask` 中 PAD 为 0，`labels` 中 PAD 位置为 ignore index，并对不同长度样本分别统计有效 label 数。

面试表达：PAD 是 batch 对齐工具，不是语言内容。PAD 参与 loss 会把实现细节写进训练目标。

## Logits

一句话定义：logits 是模型在 softmax 之前输出的未归一化分数。

形状：对于语言模型，logits 通常形状为 `[batch_size, seq_len, vocab_size]`。最后一维中每个值对应一个 token 的得分。

核心直觉：logit 越大，模型越倾向于选择该 token。但 logit 本身不是概率，不能直接解释为“70% 可能性”。需要经过 softmax 才能变成概率分布。

与解码关系：greedy decoding 选择 logit 最大的 token；sampling 会先经过 temperature、top-k、top-p 等处理，再从概率分布中采样。

数值注意：softmax 对 logit 差值敏感，而不是对绝对值敏感。所有 logits 同时加同一个常数，softmax 后概率不变。

面试表达：logits 是语言模型对整个词表的原始打分，后续的 softmax、采样策略和 loss 计算都基于它。

## Softmax

一句话定义：softmax 是把 logits 转换成概率分布的函数。

核心公式：

```text
softmax(z_i) = exp(z_i) / Σ_j exp(z_j)
```

作用：它让每个 token 的概率非负，并且所有 token 概率和为 1。

temperature 影响：temperature 会缩放 logits。温度低时分布更尖锐，模型更确定；温度高时分布更平滑，输出更多样但也更容易不稳定。

与 cross entropy：训练时通常对 logits 做 cross entropy。实现中常用 fused cross entropy，不会显式先算 softmax 再算 log，以提高数值稳定性。

面试表达：softmax 把模型对词表的原始打分归一化成概率分布，是训练 loss 和生成采样之间的桥梁。

## Causal Language Model

一句话定义：causal language model 是只能根据过去 token 预测当前或未来 token 的语言模型。

关键机制：causal mask 会屏蔽未来位置，保证第 `t` 个位置不能看到 `t+1` 之后的信息。如果没有 causal mask，训练 loss 会虚低，因为模型可以偷看答案。

典型架构：decoder-only Transformer。它没有 encoder-decoder 架构中的独立 encoder，也不使用 BERT 式双向 attention。

训练目标：通常是 next-token prediction。模型每个位置输出一个 hidden state，再通过 lm head 映射到词表 logits。

推理过程：输入 prompt，模型取最后一个位置的 logits 得到下一个 token 分布，选出新 token 后追加到上下文，再重复上述过程。

面试表达：causal LM 的“causal”指信息流方向受限，只允许从过去到未来，这正是 GPT 类模型可以做自回归生成的原因。

## Base Model

一句话定义：base model 是经过大规模预训练但尚未经过指令微调和偏好对齐的基础模型。

典型表现：它擅长续写文本，但不一定会像聊天助手一样回答问题。如果输入“请解释 self-attention”，base model 可能继续生成类似网页、论文或问答语料的文本，而不一定稳定遵循指令。

为什么需要 base model：base model 是通用能力底座。它通过预训练获得语言、知识、代码、数学和世界模式，后续 instruct model、chat model 和领域模型都通常建立在 base model 之上。

优点：分布更接近原始预训练目标，适合继续预训练、领域适配、作为研究底座，也常用于分析模型原始能力。

局限：指令遵循弱，对话格式弱，安全边界弱，输出可能不符合用户期望。直接面向用户产品通常需要后训练。

面试表达：base model 是“会续写”的模型，不是天然“会听话”的助手。instruct tuning 和 alignment 的目标就是把它变成更会按用户意图工作的模型。

## Instruct Model

一句话定义：instruct model 是经过指令微调后，能够更好理解并遵循用户指令的模型。

训练数据：常见数据形式包括 instruction、input、response 三元组，也包括问答、摘要、改写、代码解释、分类、推理等任务样本。

解决的问题：base model 只是预测文本续写，不一定知道用户希望它完成什么任务。指令微调让模型学习“看到任务描述后输出合适答案”的行为模式。

能力变化：instruct model 通常在零样本任务、格式遵循、问题回答、解释性输出方面明显优于 base model。但如果微调数据质量差，也可能导致能力退化、风格变窄或过度拒答。

与 chat model 的区别：instruct model 强调单轮或通用指令遵循；chat model 更强调多轮对话格式、角色消息、上下文承接和安全策略。

面试表达：instruct model 的核心是把预训练得到的通用能力映射到“按指令完成任务”的交互方式上。

## Chat Model

一句话定义：chat model 是面向多轮对话场景优化的语言模型，通常能处理 system、user、assistant 等角色消息。

关键特征：它不仅要回答当前问题，还要理解对话历史、遵循系统指令、保持角色一致、处理多轮追问，并在必要时拒绝不安全请求。

数据格式：chat model 通常使用聊天模板把多轮消息序列化成 token，例如 system message、user message、assistant response 加上特殊分隔符。

工程注意：chat template 非常关键。同一个模型如果使用错误聊天模板，可能出现不回答、乱续写、角色混乱、特殊 token 泄漏等问题。

与 base model 对比：base model 更像文本续写器，chat model 更像对话助手。差异不只是模型权重，也包括 tokenizer、special token、prompt 模板和对齐策略。

面试表达：chat model 是经过对话数据和偏好对齐塑形后的 LLM，重点能力是多轮交互、指令层级遵循和助手风格输出。

## Tokenizer Format Incident

一句话定义：tokenizer format incident 是 tokenizer、special token、chat template、EOS / PAD、label mask、截断或多模态 placeholder 配置错误导致模型行为异常的事故。

典型表现：模型不停生成、复述用户输入、忽略 system prompt、角色混乱、JSON / tool call 不稳定、多模态模型像没看到图、中文或代码成本异常。

排查顺序：打印原始 messages、渲染后的 prompt、token ids、decode 文本、special token 位置、attention mask、labels、EOS / PAD id、截断前后内容和模板版本。

面试表达：格式事故的核心是训练目标和推理协议不一致。先 decode 和看 labels，再讨论模型能力或超参。

## Chat Template Drift

一句话定义：chat template drift 是训练、评估、推理或不同 checkpoint 使用的聊天模板不一致，导致 role、special token、assistant 起始位置或 EOS 语义变化。

为什么重要：模型只看到 token 序列，不知道开发者心里的 role 概念。模板一变，相当于模型协议变了，旧 checkpoint、旧评估和线上 prompt 可能都不再兼容。

工程做法：给模板版本化，训练日志记录模板 hash，评估和推理复用同一渲染函数；模板变更必须跑回归评估和灰度。

面试表达：chat template 不是普通 prompt 字符串，而是 chat model 的输入协议。

## EOS Coverage

一句话定义：EOS coverage 是 assistant 结束位置或文档边界处正确出现 EOS，并且 EOS 在需要学习停止时参与 label 的覆盖程度。

常见事故：训练数据没有 EOS，EOS 被 mask 掉，推理 `eos_token_id` 配错，PAD 被当成 EOS，或者截断把 EOS 截掉。

影响：模型可能一直生成到最大长度，生成下一轮 user 模板，或者在多轮对话中边界混乱。

面试表达：模型会不会停，不只取决于 decoding stop words，也取决于训练中是否看过正确的 EOS 监督。

## Assistant Model

一句话定义：assistant model 是被设计成通用助手形态的 chat model，目标是在安全边界内帮助用户完成任务。

核心要求：有帮助性、诚实性、无害性。它需要尽量解决用户问题，不知道时避免编造，同时拒绝明显有害请求。

产品化差异：一个可用助手通常还需要工具调用、检索增强、代码执行、文件读写、记忆管理、监控告警和人工反馈闭环。模型本身只是系统的一部分。

常见问题：过度迎合用户、幻觉、拒答过度、长对话遗忘、工具调用错误、格式不稳定、对系统提示被攻击敏感。

面试表达：assistant model 不是单纯追求最大似然文本生成，而是在模型能力、用户意图、安全约束和工具环境之间做平衡。

## Prompt

一句话定义：prompt 是提供给模型的输入上下文，用来说明任务、约束、背景和期望输出。

组成：prompt 可以包含系统指令、用户问题、示例、上下文资料、输出格式要求、工具说明和约束条件。

为什么有效：LLM 具备 in-context learning 能力，可以从 prompt 中临时识别任务模式。高质量 prompt 能降低任务歧义，让模型更容易生成符合预期的答案。

局限：prompt 不是可靠的硬约束。复杂格式、强安全边界、事实性和长期一致性不能只依赖 prompt，需要训练、检索、工具、校验和评估配合。

面试表达：prompt 是运行时控制模型行为的主要接口，但它只能调动模型已有能力，不能凭空创造模型没有学到的稳定能力。

## System Prompt

一句话定义：system prompt 是对模型行为、角色、安全边界和输出风格进行高优先级约束的提示。

作用：它告诉模型应该扮演什么角色、遵循哪些规则、优先满足哪些约束，以及如何处理冲突指令。

工程场景：客服助手、代码助手、数据分析助手、医疗问答助手等都会使用 system prompt 固定身份、能力边界和输出规范。

风险：system prompt 可能被 prompt injection 攻击绕过，也可能和用户需求冲突。真实系统需要结合权限控制、工具隔离、内容审核和输出校验。

面试表达：system prompt 是 LLM 应用的行为配置层，不是安全系统的全部。它能提高一致性，但不能替代模型对齐和工程防护。

## Instruction Following

一句话定义：instruction following 是模型理解并执行用户自然语言指令的能力。

衡量维度：是否完成任务、是否遵守格式、是否覆盖约束、是否拒绝越界要求、是否在多约束冲突时合理取舍。

训练来源：指令微调、偏好对齐、高质量人工数据、合成指令数据、多轮对话数据和格式遵循数据。

常见失败：忽略部分约束、输出格式错误、回答跑题、在多轮对话中忘记先前要求、对否定指令理解错误。

面试表达：指令遵循不是简单问答能力，而是把自然语言任务描述转化成可执行生成行为的能力。

## Context Window

一句话定义：context window 是模型一次推理中能够接收和处理的最大上下文 token 长度。

包括什么：上下文窗口包含 system prompt、用户输入、对话历史、检索文档、工具结果以及模型已经生成的 token。生成越长，可用输入空间越少。

为什么重要：长上下文可以容纳更多文档、代码、对话历史和复杂任务信息。但上下文更长不等于模型一定更会利用信息。

常见现象：模型可能在长上下文中忽略中间信息，也就是 lost in the middle；也可能因为检索内容太多、噪声太大而回答变差。

工程策略：做上下文压缩、检索重排、重要信息前置或后置、分块摘要、引用定位、长上下文评估和成本控制。

面试表达：context window 决定模型“能看多少”，但有效上下文利用能力决定模型“看了多少有用信息”。

## Generation

一句话定义：generation 是模型根据已有上下文逐 token 产生新 token 的过程。

基本步骤：输入 prompt，模型 forward 得到最后位置 logits，解码策略选择下一个 token，把新 token 拼接到上下文，重复直到遇到停止条件。

停止条件：达到最大生成长度、生成 EOS token、命中 stop words、工具调用中断或系统策略终止。

解码方式：常见策略包括 greedy decoding、beam search、temperature sampling、top-k sampling、top-p sampling。不同策略会影响确定性、多样性和事实稳定性。

工程注意：生成质量不只由模型权重决定，也受 prompt、聊天模板、temperature、top-p、max tokens、重复惩罚、stop 配置和后处理影响。

面试表达：LLM 生成是自回归闭环过程。每一步的输出都会成为下一步的输入，因此早期错误可能被放大。

## Perplexity

一句话定义：perplexity 是衡量语言模型预测测试集 token 困难程度的指标，通常可以理解为 cross entropy 的指数形式。

核心公式：

```text
PPL = exp(loss)
```

其中 loss 通常是按 token 平均的 negative log likelihood。

直觉解释：perplexity 越低，说明模型对真实下一个 token 的平均概率越高，也就是越“不困惑”。

用途：预训练监控、模型规模对比、数据质量判断、领域继续预训练效果评估。若领域语料上的 PPL 下降，通常说明模型更适应该领域分布。

局限：PPL 不是万能指标。低 PPL 不一定代表指令遵循好、推理强、安全性好或用户体验好。不同 tokenizer、不同数据清洗方式、不同上下文长度下的 PPL 也不宜直接横向比较。

面试表达：perplexity 是语言建模能力的基础指标，适合看模型对文本分布的拟合程度，但不能替代下游任务评测和人工评测。

## Emergent Ability

一句话定义：emergent ability 指模型规模、数据或计算量增长到一定程度后，在某些任务上表现突然明显提升的现象。

典型例子：多步推理、少样本学习、复杂指令遵循、代码能力、数学题求解等常被讨论为规模扩大后更明显的能力。

为什么会出现：一种解释是模型容量和数据覆盖足够后，学到了更复杂的组合模式；另一种解释是评测指标可能是离散或非线性的，使连续能力提升看起来像突然出现。

争议：emergent ability 不是魔法，也不意味着只要模型变大就必然获得所有能力。数据质量、训练目标、评测方式、prompt 设计和后训练都会影响观察结果。

面试表达：涌现能力描述的是规模化后某些能力的非线性表现，但面试中要避免神秘化，应该从容量、数据、优化和评测指标四个角度解释。

## In-Context Learning

一句话定义：in-context learning 是模型在不更新参数的情况下，从 prompt 中的说明和示例临时识别任务并完成任务的能力。

核心直觉：用户在 prompt 里给几个输入输出样例，模型可以在上下文中归纳“这个任务要怎么做”，然后对新输入生成同样格式的输出。

例子：

```text
文本：这家餐厅太好吃了
情感：正面

文本：等了一个小时还没上菜
情感：负面

文本：服务员很热情
情感：
```

模型可能根据前两个样例输出“正面”。

与微调区别：in-context learning 不改变模型参数，只在当前上下文中生效；微调会更新模型权重，使行为在后续调用中持续改变。

优点：灵活、低成本、无需训练、适合快速适配新格式。局限是受上下文长度限制，对示例顺序、示例质量和 prompt 表达敏感。

面试表达：in-context learning 可以理解为模型利用上下文进行临时任务归纳，而不是在推理时真的做梯度下降更新参数。

## Label Space Coverage

一句话定义：label space coverage 衡量 few-shot prompt 是否覆盖任务需要的候选标签或输出类型。

面试表达：示例不只提供答案，还在告诉模型“可选输出有哪些”；如果标签空间缺失，模型可能输出 prompt 中出现过的偏置标签或自由生成长句。

## Demonstration Format Consistency

一句话定义：demonstration format consistency 衡量 ICL 示例的字段、分隔符、输入输出顺序和输出格式是否一致。

面试表达：ICL 很依赖 prompt 中的临时数据结构；格式不一致会让模型难以判断哪里是输入、哪里是标签、当前 query 应该按什么 schema 输出。

## Context Evidence Use

一句话定义：context evidence use 衡量模型答案是否真正使用了上下文中的示例、文档证据或工具结果，而不是只靠参数先验。

面试表达：RAG 或长上下文系统不能只看是否检索到正确文档，还要检查最终答案是否引用并遵守上下文证据。

## Middle Evidence Risk

一句话定义：middle evidence risk 指关键信息位于长上下文中间位置时更容易被模型忽略或弱化的风险。

面试表达：长上下文的关键不是能塞进多少 token，而是首部、中部、尾部证据都能稳定被读取和使用。

## Token Retrieval Audit

一句话定义：token retrieval audit 是检查模型能否在上下文 token 中定位、读取并使用相关示例、标签、字段值和证据句的审计。

面试表达：它把 ICL、RAG 和长上下文统一到“上下文内显式读取能力”上，常用检索标签匹配率、位置分桶、冲突证据、引用检查和 bad case 复盘来验证。

## Retrieval Label Match

一句话定义：retrieval label match 衡量最相关上下文示例或证据是否支持当前样本的正确标签或结论。

面试表达：few-shot prompt 不能只看“有没有相似示例”，还要看最相似示例是否对应正确标签；否则近尾冲突样本可能覆盖真正相关证据。

## ICL Conflict Rate

一句话定义：ICL conflict rate 衡量相关上下文示例中标签或结论与当前正确答案冲突的比例。

面试表达：冲突率高说明 prompt 内部证据不一致，模型可能被位置更近、格式更强或相似度更高的错误示例带偏，应该进入 bad case 分桶而不是直接上线。

## ICL Gate

一句话定义：ICL gate 是把标签空间覆盖、示例格式一致性、上下文证据使用、中间证据风险和冲突示例检查组合起来的 prompt 上线门禁。

面试表达：few-shot prompt 不能只凭主观感觉上线，应有验证集、位置鲁棒性测试、retrieval label match、冲突率、冲突样本和输出格式检查。

## Transformer Backbone

一句话定义：Transformer backbone 是把 self-attention、FFN、residual、normalization、position encoding 和自回归目标组合成大模型主干的架构选择。

为什么重要：Transformer 不是因为单个模块神奇才成为主干，而是同时满足并行训练、内容相关路由、next-token prediction、scaling 证据、硬件友好和生态成熟。

面试表达：回答“为什么 Transformer 成为基础架构”时，要同时讲算法路由、训练并行性、统一目标、scaling law、生态和瓶颈，不能只复述 attention 公式。

## Architecture Selection Gate

一句话定义：architecture selection gate 是用路由路径、训练并行性、内容路由、目标适配、scaling 证据、生态、硬件、ICL 和部署压力评估一个新架构是否适合作为基础模型主干的门禁。

面试表达：新架构不能只看单个 benchmark 或复杂度公式，还要证明它能稳定规模化训练、支持上下文学习、接入既有训练推理生态，并在真实成本上有收益。

## Route Score

一句话定义：route score 衡量远距离 token 之间的信息通信路径是否短，self-attention 通常接近一跳，RNN 类递推会随距离增长。

## Training Parallelism

一句话定义：training parallelism 衡量训练阶段能否同时处理序列中多个位置，并高效映射到 GPU / TPU 大矩阵计算。

## Parallel Path Length

一句话定义：parallel path length 衡量两个位置之间的信息需要经过多少串行步骤才能交互。

面试表达：RNN 的远距离路径随序列长度增长，小卷积核 CNN 需要多层扩大感受野，self-attention 在可见范围内能让 token 一层直接交互。

## Content Routing

一句话定义：content routing 衡量模型是否能根据 token 内容动态选择该读取哪些上下文信息，而不是只依赖固定局部窗口或固定状态递推。

## Attention Routing Audit

一句话定义：attention routing audit 是检查 self-attention 是否在给定 Query、Key、mask 和期望证据下正确路由信息的审计方法。

面试表达：Self-Attention 不只要会写公式，还要能检查 route hit、future leak、softmax 行归一化、缩放稳定性、lost-in-the-middle 和解释边界。

## Route Hit Rate

一句话定义：route hit rate 衡量 attention 最大权重位置是否命中期望证据 token 或人工标注的路由目标。

## Future Leak Rate

一句话定义：future leak rate 衡量 causal self-attention 中未来 token 是否仍获得非零注意力权重。

## Softmax Row Sum Check

一句话定义：softmax row sum check 检查 attention weight 每一行是否归一化为 1，是 attention 实现正确性的基本数值门禁。

## Scale Stability

一句话定义：scale stability 衡量除以 `sqrt(d_k)` 后 attention logits 是否避免过尖 softmax 分布，并保持更稳定的梯度区间。

## Interpretation Boundary

一句话定义：interpretation boundary 提醒 attention weight 只能作为某层某 head 的局部信息混合线索，不能直接等同于模型最终输出的因果解释。

## QK Circuit

一句话定义：QK circuit 是 attention head 中由 Query 和 Key 投影共同决定的匹配电路，用来决定当前位置从哪些 token 读取信息。

面试表达：QK 只回答“读哪里”，不能单独说明“写入了什么信息”或“最终输出为什么这样”。

## OV Circuit

一句话定义：OV circuit 是 attention head 中由 Value 投影和输出投影共同决定的写入电路，用来决定被读取 token 的内容如何写回 residual stream。

面试表达：同样的 attention map 可能因为 OV 写入不同而产生不同功能，所以分析 head 时不能只看 attention weight。

## Head Subspace Audit

一句话定义：head subspace audit 是从 QK 路由命中、OV 写入目标、head 功能签名和 KV cache 成本审计 attention head 是否真的承担有效、可区分且成本合理的子空间路由。

面试表达：多头不是简单复制 attention，而是在多个低维子空间里并行路由；但训练后可能存在冗余，所以要同时讲表达收益、可解释性边界和推理成本。

## Head Redundancy Rate

一句话定义：head redundancy rate 粗略衡量多个 attention heads 是否呈现相似路由和写入功能签名，可用于解释 head pruning 和压缩动机。

## KV Head Ratio

一句话定义：KV head ratio 衡量 query heads 与 key/value heads 的数量比例，是理解 MHA、MQA、GQA 和 KV cache 节省的重要指标。

面试表达：GQA / MQA 的缓存收益来自减少 `H_kv`，而不是取消多个 Query 子空间。

## Multi-head Latent Attention

一句话定义：Multi-head Latent Attention / MLA 是把 K/V 相关历史信息压缩到 latent cache，再在 attention 中使用或恢复相关表示的注意力结构。

为什么重要：MLA 不是简单减少 KV head 数，而是进一步压缩 KV 表示本身，适合解释长上下文、高并发和 MoE serving 中的 KV cache 压力。

面试表达：GQA 主要减少 `H_kv`，MLA 主要减少每个历史 token 需要缓存的表示维度；二者都服务于推理效率，但压缩层级不同。

## Latent KV Cache

一句话定义：latent KV cache 是用低维 latent 表示保存历史 K/V 信息的缓存形式，目标是在保留多头表达能力的同时降低显存和带宽压力。

## KV Cache Saving Rate

一句话定义：KV cache saving rate 衡量某种 attention 架构相对 MHA baseline 减少了多少 KV cache 显存。

面试表达：只说“更省显存”不够，应说明 baseline、batch、层数、上下文长度、KV head 数、head dim、精度和是否包含 latent / RoPE 额外缓存。

## Attention Architecture Gate

一句话定义：attention architecture gate 是从质量、KV cache 节省率、吞吐延迟收益、实现成熟度和复杂度评估 MHA / MQA / GQA / MLA 是否适合上线的门禁。

面试表达：MHA、MQA、GQA、MLA 没有绝对最优，选型必须同时看质量门禁、serving 成本和工程成熟度。

## Position-wise FFN

一句话定义：position-wise FFN 是 Transformer block 中对每个 token 位置独立应用同一组两层或门控 MLP 参数的通道变换模块。

面试表达：attention 主要混合序列维度的信息，FFN 主要在 hidden/channel 维度做非线性加工和容量扩展；它不是附属模块，很多模型的大量参数和计算都集中在 FFN/MLP。

## SwiGLU

一句话定义：SwiGLU 是一种 gated MLP，把 SiLU gate branch 与 up branch 逐元素相乘，再用 down projection 投回 hidden size。

面试表达：SwiGLU 不是只换激活函数，而是多了一条 gate projection；比较参数量时要说明普通 4D FFN 约为 `2dm`，SwiGLU 约为 `3dm`，因此常把 intermediate size 调到接近 `8d/3`。

## Gated MLP

一句话定义：gated MLP 是用门控分支控制候选通道特征通过程度的 MLP 结构，SwiGLU 是现代 LLM 常见 gated MLP 形式。

面试表达：gated MLP 的核心直觉是通道级选择性激活，工程上仍主要由矩阵乘、逐元素激活和逐元素乘法组成，复杂度远低于 MoE routing。

## MoE FFN

一句话定义：MoE FFN 是把 dense FFN 替换为多个 experts，并由 router 为每个 token 选择少数 experts 计算的稀疏激活 FFN。

面试表达：MoE 的价值是扩大总参数容量，同时让每 token 只激活 top-k experts；它通常替换 FFN 而不是 attention，因为 FFN 本来就是 token-wise 独立计算且是容量主体。

## Top-k Routing

一句话定义：top-k routing 是 MoE router 根据 softmax 权重为每个 token 选择得分最高的 `k` 个 experts。

面试表达：top-1 routing 更省、更简单，top-2 routing 表达更强但通信和 combine 更复杂；无论 top-k 怎么选，都要配合负载均衡和 capacity 约束。

## Expert Capacity

一句话定义：expert capacity 是 MoE 中单个 expert 在一个 batch 或 dispatch step 中最多接收多少 token 的容量约束。

面试表达：capacity 太小会 drop 或 overflow token，capacity 太大又会浪费 padding 和通信资源；它必须和 batch size、top-k、expert 数和 capacity factor 一起估算。

## Active Parameters

一句话定义：active parameters 是一次前向计算中实际参与某个 token 计算的参数量，MoE 中通常远小于总参数量。

面试表达：说 MoE 模型大小时要区分 total parameters 和 active parameters；总参数大代表容量空间大，active parameters 才更接近每 token 计算成本。

## FFN / MoE Gate

一句话定义：FFN / MoE gate 是从 dense 参数量、SwiGLU 等参配置、MoE active parameters、capacity overflow、load balance 和 expert 利用率审计 FFN 架构是否合理的门禁。

面试表达：MoE 不是 active parameters 越低越好，还要看 router 是否 collapse、expert 是否过载、all-to-all 通信是否可控，以及质量和 serving 延迟是否过线。

## Residual Stream

一句话定义：residual stream 是 Transformer block 中被 attention、FFN/MLP 和其他子层逐层写入增量信息的主隐藏状态通道。

面试表达：Residual 不是简单相加，它让信息和梯度沿主路径流动；attention 和 FFN 更像是在同一条 residual stream 上写入不同类型的增量。

## Norm Placement

一句话定义：norm placement 是 LayerNorm / RMSNorm 相对子层和 residual add 的放置方式，典型形式包括 Pre-LN 和 Post-LN。

面试表达：norm placement 会改变残差主路径、梯度流和 warmup 依赖，不能孤立地说某个 norm 一定更好。

## Pre-LN

一句话定义：Pre-LN 是先对输入做 LayerNorm / RMSNorm，再进入 attention 或 FFN 子层，最后与原 residual stream 相加的结构。

面试表达：Pre-LN 保留更干净的 identity residual path，深层训练更稳定，因此现代 decoder-only LLM 常见；代价是 residual stream 尺度仍需 final norm、初始化和 residual scaling 控制。

## Post-LN

一句话定义：Post-LN 是先执行子层和 residual add，再对相加结果做 LayerNorm 的结构。

面试表达：Post-LN 每层输出尺度更统一，但深层训练通常更依赖 warmup；DeepNorm 等方法试图结合 Post-LN 表达和深层稳定性。

## RMSNorm

一句话定义：RMSNorm 是只按 root mean square 做 re-scaling、不显式减均值的归一化方法。

面试表达：RMSNorm 的重点不是“LayerNorm 的粗糙近似”，而是保留尺度归一化、去掉 re-centering，计算更简单，并被 LLaMA 等现代 LLM 广泛采用。

## Residual Scaling

一句话定义：residual scaling 是对每层 residual branch 写入幅度做缩放，防止深层模型中每层增量持续放大 residual stream。

面试表达：当层数很深时，norm 本身不一定足够，还要配合 residual scaling、初始化、learning rate 和 optimizer 控制每层更新幅度。

## Early Update Pressure

一句话定义：early update pressure 是教学审计中用早期学习率和最大子层更新比例粗略衡量训练初期参数扰动风险的指标。

面试表达：warmup 的价值可以理解为降低早期有效更新幅度，尤其是在 Post-LN 或初始化梯度较大的设置下。

## Normalization Stability Gate

一句话定义：normalization stability gate 是从 activation norm、update ratio、mean drift、early update pressure 和 NaN / overflow 检查 residual + norm 结构是否稳定的门禁。

面试表达：训练稳定性不是只看 LayerNorm/RMSNorm，还要同时看 residual stream 尺度、grad norm、learning rate、warmup、mixed precision、data outlier 和分布式通信异常。

## Position Encoding

一句话定义：position encoding 是把 token 的顺序、绝对位置、相对距离或位置偏置注入 Transformer 的机制。

面试表达：Self-Attention 本身只根据 Q/K/V 做内容相关路由，不天然知道先后顺序；位置编码可以加到 embedding、attention score、K/V 表示或 Q/K 几何结构中。

## Sinusoidal Position Encoding

一句话定义：sinusoidal position encoding 用不同频率的 sin/cos 函数为每个位置生成固定向量。

面试表达：它不需要学习参数，可以生成训练长度外的位置向量，但“公式能生成”不等于模型一定学会长上下文外推。

## Learned Absolute Position Embedding

一句话定义：learned absolute position embedding 为每个绝对位置学习一个向量，并与 token embedding 相加。

面试表达：它实现简单，在固定最大长度内有效，但超过训练位置后没有自然外推能力，也不直接表达“距离我多远”这种相对关系。

## Relative Position Representation

一句话定义：relative position representation 把 query token 与 key token 的相对距离表示加入 attention 交互。

面试表达：attention 是 token pair 之间的交互，相对位置方法更直接地建模 pairwise distance，但朴素实现可能增加长序列内存和计算压力。

## Relative Position Bias

一句话定义：relative position bias 不引入完整位置向量，而是在 attention score 上加一个与相对距离或距离 bucket 对应的标量 bias。

面试表达：T5 风格 relative bias 更轻量，近距离可以细分、远距离可以粗分，但表达能力通常弱于完整相对位置表示。

## Rotary Position Embedding / RoPE

一句话定义：RoPE 根据位置对 Q/K 做旋转，使旋转后点积自然包含相对位置差。

面试表达：RoPE 用绝对 position id 实现，但在 Q/K 点积中体现相对位置关系；它兼容 causal attention 和 KV cache，是 LLaMA-like decoder-only LLM 的常见位置编码路线。

## ALiBi

一句话定义：ALiBi 在 attention score 上加入与距离成比例的线性 bias，距离越远惩罚越大。

面试表达：ALiBi 不需要 position embedding，外推友好且实现简单，但会引入较强 recency bias，不一定适合所有远距离精确匹配任务。

## Position Interpolation

一句话定义：Position Interpolation 把目标长上下文中的位置压缩回训练上下文尺度，缓解 RoPE 直接外推到未训练相位区域的问题。

面试表达：插值能降低长位置相位失配风险，但会降低位置分辨率，通常还需要少量长上下文继续训练和真实长文任务评估。

## RoPE Scaling

一句话定义：RoPE scaling 是通过调整位置映射、频率尺度或不同频段缩放来扩展 RoPE 上下文窗口的一类方法。

面试表达：RoPE scaling 不能只看配置项是否存在，还要看训练长度、目标长度、短上下文退化、KV cache position id 和长上下文评测。

## Position ID Audit

一句话定义：position id audit 检查 position id 是否与真实 token 位置、KV cache 长度、packed sequence 边界和 long-context scaling 配置一致。

面试表达：很多位置编码线上问题不是公式错，而是继续生成时 position id 从 0 重启、prefix cache 偏移错误、packed sample 位置泄漏或长上下文只改 max length。

## Position Encoding Gate

一句话定义：position encoding gate 是把 cache position 连续性、packed sequence 隔离性和长上下文 scaling 配置组合成的位置编码审计门禁。

面试表达：位置编码要和 attention mask、KV cache、sliding window、prefix cache、长上下文扩展和评估任务一起审计，不能孤立看 RoPE 或 ALiBi 公式。

## Attention Mask

一句话定义：attention mask 控制每个 query 位置可以从哪些 key/value 位置读取信息。

面试表达：attention mask 不是删除 token，也不是决定哪些 token 算 loss；它是在 softmax 前改变 attention score，决定信息流可见性。

## Loss Mask

一句话定义：loss mask 控制哪些 token 参与训练损失。

面试表达：SFT 中 user token 通常不参与 loss，但 assistant token 必须能通过 attention 看见 user prompt，所以 loss mask 不能直接当 attention mask 用。

## Bidirectional Attention

一句话定义：bidirectional attention 允许每个非 padding token 同时读取左侧和右侧上下文。

面试表达：BERT 类 encoder-only 模型依赖双向 attention 学习理解表征；它适合分类、抽取、匹配、rerank 和 embedding，但不是天然 left-to-right 生成器。

## Prefix LM

一句话定义：Prefix LM 让 prefix 内部双向可见，target 可以看全部 prefix 且在 target 内部因果可见。

面试表达：Prefix LM 把条件输入和生成输出放在同一条 self-attention 序列里，通过 mask 模拟“理解输入 + 因果生成输出”。

## Encoder-Decoder Attention

一句话定义：encoder-decoder attention 是 encoder 双向编码 source、decoder 因果生成 target，并由 decoder cross-attention 读取 encoder 输出的结构。

面试表达：翻译、摘要、改写等输入输出结构差异明显的任务常适合 encoder-decoder；decoder self-attention 防未来泄漏，cross-attention 负责读取 source。

## Cross-Attention

一句话定义：cross-attention 中 Q 来自当前模块，K/V 来自另一个序列或模块输出。

面试表达：encoder-decoder 里 decoder 用 cross-attention 读取 encoder 表示；多模态模型里文本 token 也常通过 cross-attention 读取图像、音频或检索证据。

## Packed Sequence Mask

一句话定义：packed sequence mask 在把多条样本拼成一个长序列训练时，阻止不同样本之间互相可见。

面试表达：只用普通 causal mask 会让后面的样本看到前面样本，造成跨样本信息泄漏、loss 虚低和文档边界污染。

## Block-Diagonal Causal Mask

一句话定义：block-diagonal causal mask 在每个样本内部使用下三角 causal 可见性，不同样本 block 之间完全不可见。

面试表达：它是 packed training 的核心防泄漏机制，要和 position id、loss mask、EOS 边界和数据 packing 逻辑一起审计。

## Sliding Window Attention

一句话定义：sliding window attention 只允许 token 读取局部窗口内的历史或邻近 token。

面试表达：它用局部可见性降低长上下文成本，但远距离信息不能直接访问，依赖多层传播、global token 或其他机制补足。

## Mask Visibility Audit

一句话定义：mask visibility audit 检查 future leak、cross-sample leak、全 mask 行、mask 方向、query/key 长度偏移和 kernel fast path 是否符合预期。

面试表达：mask 审计要看 decoded text、attention mask、padding mask、loss mask、labels、position ids、sample boundary 和 logits/labels shift，不能只看张量 shape。

## Mask Gate

一句话定义：mask gate 是把未来泄漏率、跨样本泄漏率、每行可见 key 覆盖和高效 kernel 支持组合成的注意力可见性门禁。

面试表达：mask 数学上正确还不够；如果全 mask 行导致 NaN、packed 边界泄漏或复杂 dense mask 退化到慢 kernel，训练和推理仍会出事故。

## Scaling Evidence

一句话定义：scaling evidence 衡量一个架构是否在足够大的参数量、训练 token 数、计算预算和任务范围上表现出稳定可扩展收益。

## Attention Pair Count

一句话定义：attention pair count 是 attention score 矩阵中的 query-key pair 数，full attention 长度为 `n` 时约为 `n^2`。

面试表达：Transformer 用更多可并行 pair 计算换取短路径和动态路由，长上下文时这个平方成本会成为显存和延迟瓶颈。

## Quadratic Attention Cost

一句话定义：quadratic attention cost 指 full attention 的 query-key pair、QK^T / AV 计算和 score/probability 中间张量随序列长度平方增长。

面试表达：从 4K 到 32K 不是 8 倍成本，而是 attention pair 约 64 倍；FlashAttention 可以减少 IO 和中间显存，但 full attention 的 pair 数仍是平方级。

## Attention Score Memory

一句话定义：attention score memory 是 materialize `S` 或 `A` 这类 `[B,H,T_q,T_k]` 中间矩阵时占用的显存。

面试表达：32K、32 heads、bf16/fp16 下单个 full score tensor 已约 64 GiB；如果同时保存 scores 和 probabilities，朴素实现会更难承受。

## Prefill Attention Cost

一句话定义：prefill attention cost 是推理开始处理 prompt 时 `T_q=T_k=n` 的 attention 成本，长 prompt 下仍然接近训练阶段的平方级。

面试表达：TTFT 经常受 prefill 影响；长 prompt 的首 token 慢，不能只用 decode 单步成本解释。

## Decode Attention Cost

一句话定义：decode attention cost 是自回归生成时单个新 token 查询历史 KV cache 的成本，单步通常随历史长度线性增长。

面试表达：decode 单步不是 `n^2`，但长输出会累积，并且 KV cache 显存和内存带宽常成为瓶颈。

## Attention Cost Gate

一句话定义：attention cost gate 是把 pair count、score/probability 显存、prefill FLOPs、decode 单步成本、window ratio 和质量风险组合起来的长上下文 attention 成本门禁。

面试表达：长上下文方案不能只说支持多少 token，要同时报告 prefill、decode、显存、kernel 路径、可见模式变化和长距离证据召回风险。

## KV Cache Budget

一句话定义：KV cache budget 是 serving 系统为活跃请求历史 K/V 状态预留和动态分配的显存预算。

面试表达：KV cache 不压缩模型权重，而是新增运行时状态；它随 `L * B * T * H_kv * D_h * 2 * bytes` 线性增长，长上下文和高并发下常先卡住 batch size。

## KV Cache Per Token

一句话定义：KV cache per token 是每个新 token 给单个请求追加到所有层的 K/V cache 大小。

面试表达：长输出不是只多算几个 logits；每个 decode token 都会追加 cache，并增加后续 token 需要读取的历史 K/V。

## Decode Bandwidth Pressure

一句话定义：decode bandwidth pressure 指自回归 decode 单步需要反复读取历史 KV cache，导致显存带宽而非 FLOPs 成为瓶颈的压力。

面试表达：prefill 更像大矩阵计算，decode 更像小 batch、长历史的持续访存；MQA/GQA/MLA 的重要收益就是降低每步读取的 K/V 量。

## Paged KV Block

一句话定义：paged KV block 是 PagedAttention / vLLM 类系统中管理 KV cache 的固定大小物理块，通过 block table 映射逻辑 token 序列。

面试表达：分页不是新的 attention 公式，而是降低变长请求碎片、支持按需分配、释放、prefix sharing 和 continuous batching 的内存管理方法。

## KV Cache Gate

一句话定义：KV cache gate 是把 cache 显存、每 token 增长、decode 读取带宽、PagedAttention block 浪费、GQA/MQA/MLA 质量风险和上线 SLO 组合起来的推理门禁。

面试表达：长上下文上线不能只证明单请求能跑，还要证明在目标并发、TTFT/TPOT、显存、带宽和质量回归下能稳定服务。

## Training FLOPs Estimate

一句话定义：training FLOPs estimate 是训练计算量的粗略估计，dense Transformer 常用 `C_train` 近似正比于参数量乘训练 token 数。

面试表达：`6ND` 只是教学和早期规划估算，真实成本还要看激活重算、并行效率、通信、optimizer、硬件利用率和失败重跑。

## Compute-Optimal Training

一句话定义：compute-optimal training 是在固定训练计算预算下平衡参数量和训练 token 数，而不是只把模型做大。

面试表达：Chinchilla 的核心启发是很多大模型 undertrained；给定 compute 时，需要同时扩参数和数据，并结合数据质量与推理成本。

## Tokens per Parameter

一句话定义：tokens per parameter 是训练 token 数与参数量的比例，用来粗略观察模型是否训练充分。

面试表达：这个比例不能机械套固定数字，但过低常提示 undertraining 风险，过高则可能说明容量不足或数据重复收益递减。

## Undertrained Model

一句话定义：undertrained model 指参数量相对训练 token 数过大，模型容量没有被足够数据充分训练的情况。

面试表达：undertrained 不等于模型小，而是参数、数据和 compute 配比不合理；更小但训练更充分的模型可能实际更强。

## Serving Pressure

一句话定义：serving pressure 衡量候选模型在推理延迟、显存、吞吐、KV cache、成本和上线 SLO 上的压力。

面试表达：训练 loss 更低不等于产品可用；如果 serving pressure 超预算，可能需要更小模型、蒸馏、量化、MoE/GQA、缓存或路由策略。

## Scaling Gate

一句话定义：scaling gate 是把路径长度、compute balance、数据质量、训练稳定性和 serving pressure 组合成的 scaling 方案门禁。

面试表达：scaling 决策要同时看 loss trend、tokens per parameter、数据质量、训练稳定性、推理预算和评估切片，不能只看参数量或单个 benchmark。

## Ecosystem Readiness

一句话定义：ecosystem readiness 衡量一个架构是否有成熟 kernel、分布式训练、推理框架、模型实现、评估基准、量化压缩和部署经验。

## Deployment Pressure

一句话定义：deployment pressure 衡量一个架构在长上下文效率、KV / state 缓存、流式状态、延迟和显存上的工程压力。

## Attention-SSM Hybrid Readiness

一句话定义：attention-SSM hybrid readiness 衡量 Attention + SSM 等混合架构是否同时保留内容检索和上下文学习能力，并降低长序列成本和流式状态压力。

面试表达：混合架构的价值不在于“替代 Transformer”这个口号，而在于证明哪些层用 attention、哪些层用状态或卷积，能在质量、成本和生态迁移上形成稳定收益。

## Few-Shot Learning

一句话定义：few-shot learning 是在 prompt 中提供少量示例，让模型根据示例完成新样本的任务方式。

作用：示例能明确任务边界、输出格式、标签空间和隐含规则，通常比单纯文字说明更稳定。

适用场景：分类标签不直观、输出格式严格、业务规则特殊、模型容易误解任务时，few-shot prompt 往往很有效。

示例选择：示例要覆盖典型情况和容易混淆的边界情况，格式保持一致，避免示例本身包含错误标签或多余噪声。

局限：示例会占用上下文窗口；示例过多可能引入冲突；模型可能模仿示例中的偏差；少样本效果不等于真实泛化能力一定强。

面试表达：few-shot 的价值是用少量上下文示例约束模型行为，尤其是约束输出格式和任务映射关系。

## Zero-Shot Learning

一句话定义：zero-shot learning 是不提供任务示例，只通过任务描述让模型直接完成任务。

例子：直接输入“请判断下面评论的情感是正面还是负面：这家店服务很好”，模型输出“正面”。

为什么可行：大模型预训练和指令微调中见过大量任务描述和解题模式，因此能把自然语言指令映射到已学到的能力上。

优点：prompt 简洁，不占示例空间，适合标准任务和模型已熟悉的任务。缺点是对复杂规则、特殊格式和业务边界不如 few-shot 稳定。

评估注意：zero-shot 表现受 prompt wording 影响很大。同一个任务换一种问法，结果可能明显变化，因此评测时要固定提示模板或做多模板评估。

面试表达：zero-shot 体现的是模型从自然语言指令直接泛化到任务的能力，是衡量 instruction following 和预训练知识迁移的重要方式。

## Chain-of-Thought

一句话定义：chain-of-thought 是让模型生成中间推理步骤，再给出最终答案的提示或训练方式。

为什么有效：复杂问题往往需要多步计算、条件判断或中间状态。让模型显式展开推理过程，可以降低一步到位输出答案的难度。

典型场景：数学应用题、逻辑推理、多跳问答、代码推演、计划生成等。

局限：模型写出的推理步骤不一定真实反映内部计算，也可能出现“步骤看起来合理但结论错误”。在生产系统中，最终答案仍需要验证器、工具或规则校验。

安全注意：有些产品会把详细隐式推理隐藏，只输出简洁解释或关键依据，以降低泄漏内部策略和增加冗长错误的风险。

面试表达：CoT 的本质是把复杂任务分解成中间 token 序列，让模型有更多计算轨迹可用，但它不是正确性的充分保证。

## Memorization vs Generalization

一句话定义：memorization 是模型复现训练数据中的具体内容，generalization 是模型把学到的规律应用到未见样本。

核心直觉：大模型既会记忆，也会泛化。高频文本、重复数据、特殊字符串、代码片段和评测答案更容易被记住；抽象语法、任务格式和推理模式更接近泛化。

如何区分：可以用去重评测集、时间切分评测、扰动测试、反事实样本、私有测试集和数据污染检查来判断模型是否只是记住答案。

为什么重要：如果模型在 benchmark 上表现好只是因为训练集中见过题目，评估结果就不可信。数据污染是大模型评测中的核心风险之一。

面试表达：不能简单说大模型“只是背题”或“完全理解”。更准确的说法是，它在参数中同时包含记忆和可泛化模式，评估设计要尽量区分二者。

## Hallucination

一句话定义：hallucination 是模型生成看似合理但不真实、无依据或与上下文冲突的内容。

根本原因：语言模型优化的是 token 概率，而不是事实数据库查询。一个答案只要在语言模式上合理，就可能被模型高置信度生成，即使事实错误。

常见类型：编造引用、编造论文、编造 API、错误归因、数字计算错误、把过时知识当成事实、在上下文没有依据时强行回答。

缓解方法：RAG 提供外部依据，工具调用进行计算和查询，训练中加入拒答和不确定性表达，评估中加入事实性检查，产品中加入引用、校验和人工审核。

局限：幻觉不能被完全消除，只能在具体场景中通过数据、模型、工具和流程降低风险。

面试表达：幻觉来自生成式目标和事实性目标之间的不一致。解决思路不是只靠更大模型，而是结合检索、工具、校验、对齐和评估闭环。

## LLM Productization

一句话定义：LLM productization 是把大模型能力转成稳定用户价值、业务指标、可靠系统和持续反馈闭环的过程。

核心问题：用户是谁、痛点是否高频或高价值、为什么需要大模型、成功指标是什么、成本是否可控、风险是否可控、上线后如何监控和迭代。

面试表达：产品化不是接一个模型 API，而是把模型能力放进真实工作流，用评估、成本、延迟、安全和反馈闭环证明它能持续创造价值。

## Demo to Product

一句话定义：demo to product 是从“样例能跑通”升级到“真实用户长期可用”的过程。

关键差异：demo 输入少、边界可控、失败可接受；产品要面对长尾输入、并发、稳定性、权限、合规、成本、监控、灰度和回归测试。

常见误区：把精心挑选样例上的惊艳效果，当成真实生产环境中的可持续价值。

## Scenario Selection

一句话定义：scenario selection 是在大模型产品化前筛选“哪个用户、哪条任务链路、哪个环节最值得做”的过程。

核心维度：真实需求、用户痛点、任务频率、单次价值、数据准备度、可验证性、大模型适配度、工作流适配、剩余风险和落地难度。

面试表达：场景选择不是问“模型能不能做”，而是问“用户是否真的需要、收益是否覆盖成本、错误是否可控、能否进入受控试点”。

## Real Demand

一句话定义：real demand 是用户已经在用时间、预算、流程或替代方案解决的问题，而不是只停留在口头兴趣的大模型想象。

判断方法：看当前谁在做、多久做一次、每次成本多少、现有方案哪里痛、成功后哪个指标改善、用户是否愿意接入工作流。

反面概念：fake demand 通常没有明确用户、没有当前成本、没有可衡量指标，或者规则和搜索已经足够解决。

## User Persona

一句话定义：user persona 是对目标使用者的角色、目标、约束、痛点和工作流的结构化描述。

大模型场景：同一个企业知识库助手，对一线客服、主管、IT、法务和管理层的价值与风险完全不同，不能只写“企业用户”。

## Stakeholder Map

一句话定义：stakeholder map 是把使用者、决策者、数据负责人、合规方、运维方和业务 owner 放在同一张责任图里。

为什么重要：很多大模型项目失败不是模型不够强，而是使用者嫌麻烦、决策者只看概念、技术团队只看模型指标，目标不一致。

## Task Journey

一句话定义：task journey 是用户完成一个真实任务的完整步骤链路。

大模型场景：不要只说“帮销售写方案”，要拆成查客户、找案例、选卖点、写初稿、人工修改、审批和发送，才能判断模型介入哪一步最有价值。

## Product Metric

一句话定义：product metric 是衡量用户价值和业务效果的指标，例如任务完成率、自助解决率、采纳率、转人工率、满意度、留存、成本下降和收入提升。

和技术指标关系：召回率、引用准确率、测试通过率等技术指标需要连接到用户和业务指标，才说明项目有产品闭环。

## Model-to-Experience Mapping

一句话定义：model-to-experience mapping 是把理解、生成、检索、推理、工具调用、结构化输出和安全拒答等模型能力，映射到用户可感知的体验指标。

核心判断：模型 benchmark 提升不等于体验提升，必须继续看任务完成率、P95 延迟、引用支持、格式稳定、可控性、错误恢复、信任校准和风险门禁。

## LLM Product Experience

一句话定义：LLM product experience 是用户在真实工作流中感知到的大模型产品质量，包括快不快、准不准、稳不稳、能否控制、能否信任、出错后能否恢复。

面试表达：用户不会直接感知模型参数量或公开榜单，而会感知任务是否更快完成、答案是否有证据、输出是否能直接使用、错误是否可纠正。

## Experience Metric

一句话定义：experience metric 是衡量大模型产品体验的指标，例如任务成功率、首次成功率、采纳率、追问率、重新生成率、人工接管率、引用点击率和留存率。

和评估指标关系：离线 eval 负责证明模型或系统在样本集上过线，体验指标负责证明真实用户是否愿意持续使用。

## Structured Output Stability

一句话定义：structured output stability 是模型输出持续符合 JSON、表格、字段或业务模板要求的比例。

为什么重要：内容看起来正确但格式不稳定，会导致后续系统无法解析、审批流无法接入、批处理任务失败。

## Citation Support Rate

一句话定义：citation support rate 是被引用证据真实支持的关键结论比例。

大模型场景：企业 RAG 和知识助手不能只要求“有引用”，还要检查引用是否存在、相关、足以支持 claim，并且没有过期或越权。

## Controllability

一句话定义：controllability 是用户能否通过长度、风格、格式、受众、数据范围、保守程度和编辑入口控制模型输出。

产品化意义：可控性差会让用户反复改 prompt；可控性好则把 prompt 工程隐藏为模板、选项、字段和工作流控件。

## Error Recovery Rate

一句话定义：error recovery rate 是模型或产品出现失败后，通过追问、重新生成、修改约束、工具校验或人工接管恢复成功的比例。

面试表达：大模型产品不能假装永远正确，关键是错误可见、可解释、可反馈、可恢复。

## Trust Calibration

一句话定义：trust calibration 是让用户既不过度信任，也不过度怀疑模型输出的能力。

常见机制：展示来源、标注不确定性、区分事实和推断、对高风险结论提示复核、避免无证据自信表达。

## Experience Gate

一句话定义：experience gate 是大模型产品体验上线门禁，覆盖任务成功率、P95 延迟、引用支持、格式稳定、可控性、恢复、安全和过度拒答。

面试表达：如果体验门禁不过线，换更强模型不一定解决问题，可能需要改交互、结构化输出、引用校验、流式返回、缓存、反馈或人工接管。

## Frequency-Value Matrix

一句话定义：frequency-value matrix 是用任务频率和单次价值粗分候选场景优先级的方法。

判断口径：高频低单价场景看总量和自动化收益；低频高价值场景看专家时间、风险降低和质量提升；低频低价值通常不适合作为首个项目。

## Data Readiness

一句话定义：data readiness 是场景所需数据是否可访问、结构化、及时、干净、权限清楚、可用于评估和可持续更新的程度。

常见失败：RAG 项目答不好，往往不是模型不会答，而是知识库过期、权限混乱、缺少标准答案或没有反馈数据。

## Verifiability

一句话定义：verifiability 是模型输出能否被测试、规则、证据、人工标注或业务指标验证。

落地影响：可验证任务更适合自动化和规模化；难验证任务需要人工反馈、灰度、抽检和更保守的产品形态。

## LLM Fit

一句话定义：LLM fit 是判断一个需求是否真的需要大模型，而不是规则、搜索、模板或传统机器学习。

适合 LLM：自然语言表达多样、上下文复杂、长尾问题多、需要生成或总结、传统规则难以覆盖。

不适合 LLM：规则稳定、搜索结果已经足够、输出格式完全确定、错误成本高但缺少验证闭环。

## Automation Level

一句话定义：automation level 是产品从 assist、recommend、review、approval 到 fully automate 的自动化层级。

产品化原则：低风险高可验证任务可以逐步自动化；低容错或高风险任务应优先做人机协同、引用、审批和审计。

## Pilot Scenario

一句话定义：pilot scenario 是边界清楚、数据可得、风险可控、指标可衡量且能短周期验证价值的试点场景。

常见误区：试点不是展示型 demo，而是受控范围内验证真实用户价值、工作流适配和单位经济账。

## Scenario Selection Gate

一句话定义：scenario selection gate 是候选场景进入试点前的门禁，覆盖真实需求、收益、大模型适配度、数据、验证、风险和工作流。

面试表达：如果痛点、收益、数据、验证、风险或工作流不过线，即使模型 demo 效果好，也应该暂缓、拆分试点或放弃。

## Unit Economics

一句话定义：unit economics 是单次任务、单个用户或单个订单的收益和成本账。

大模型场景：要同时计算模型调用、检索、工具、人工审核、日志监控、运维和预期风险成本，而不是只看 token 单价。

## Total Cost of Ownership

一句话定义：total cost of ownership 是一个大模型产品从试点到长期运行的总拥有成本。

大模型场景：除了 token 或 GPU，还要算 RAG、向量库、工具调用、人审、数据治理、研发摊销、监控告警、合规安全、事故处理和模型升级。

## Variable Cost

一句话定义：variable cost 是随任务量增长而增长的可变成本。

大模型场景：模型调用、输入输出 token、检索、rerank、工具调用、重试和人审通常都是可变成本。单次可变成本为负毛利时，规模越大亏损越多。

## Fixed Cost

一句话定义：fixed cost 是短期内不随任务量线性变化的固定成本。

大模型场景：研发摊销、评估集建设、权限系统、监控、日志、合规审计和固定基础设施通常属于固定成本。固定成本高的项目需要足够任务量才能摊平。

## Model Cost

一句话定义：model cost 是模型推理调用带来的成本，通常由输入 token、输出 token、调用次数、缓存命中、批处理折扣和模型单价共同决定。

面试表达：不能只看单次回答报价，要按真实 prompt、RAG 上下文、输出长度、重试率和多轮 Agent 调用估算。

## Review Cost

一句话定义：review cost 是人在回路审核、复核、质检、投诉处理和反馈标注带来的成本。

常见误区：模型节省了 5 分钟，但人审又花 4 分钟，实际收益会很低。人机协同要让人只看关键风险点。

## ROI

一句话定义：ROI 是投入产出比，常用于判断项目收益是否覆盖成本。

常见口径：收益成本比可以写成 `benefit / cost`；净 ROI 可以写成 `(benefit - cost) / cost`。面试中要先说明自己采用哪种口径。

## Net Benefit

一句话定义：net benefit 是收益减去成本后的净收益。

大模型场景：月净收益为正只是基本条件，还要继续看回本周期、风险、质量门禁和是否损害用户体验。

## Break-Even Point

一句话定义：break-even point 是收益刚好覆盖成本时所需的任务量、用户量或收入规模。

面试表达：如果单次毛利为负，盈亏平衡点不存在；如果固定成本高，就需要足够任务量摊平固定投入。

## Sensitivity Analysis

一句话定义：sensitivity analysis 是检查 ROI 对采用率、质量提升、token 成本、人审成本、重试率和固定成本变化的敏感程度。

为什么重要：ROI 估算依赖假设，试点阶段必须知道哪些变量最容易把项目从盈利打成亏损。

## Payback Period

一句话定义：payback period 是一次性投入靠净收益收回所需的时间。

大模型场景：如果月净收益小于等于 0，就不要报一个看似精确的回本月数，而应明确当前假设下无法回本。

## ROI Gate

一句话定义：ROI gate 是产品化前的成本收益门禁，覆盖净收益、收益成本比、回本周期、单次毛利、质量、延迟和风险。

面试表达：ROI 门禁不过线时，常见修复不是“换更贵模型”，而是提高采用率、减少上下文、加缓存、降人审成本、限制 Agent 步数、拆试点或换更高价值场景。

## Enterprise LLM Application

一句话定义：enterprise LLM application 是接入企业身份、权限、数据、工作流、审计、SLO 和业务指标的大模型应用。

和普通应用区别：企业应用不能只看回答体验，还要保证用户只能访问授权数据，模型和工具动作可追溯，结果能进入真实业务流程，并且收益覆盖成本和治理投入。

## Enterprise Knowledge Base

一句话定义：enterprise knowledge base 是面向企业内部制度、产品文档、技术文档、工单经验和业务流程的知识库系统。

大模型场景：企业知识库通常通过 RAG 使用，但成败取决于文档版本、元数据、权限、引用支持、数据新鲜度和反馈闭环，而不只是 embedding 模型。

## Single Sign-On / SSO

一句话定义：SSO 是企业用户使用统一身份系统登录多个应用的机制。

大模型场景：SSO 让 LLM 应用继承企业身份和用户组，便于做权限过滤、审计日志、离职回收和租户隔离。

## Identity and Access Management / IAM

一句话定义：IAM 是管理身份、认证、授权、用户组、服务账号和访问策略的控制面。

面试表达：企业 LLM 应用要把 IAM 放在后端检索、工具调用、日志查看和管理配置中，而不是只在前端做一次登录判断。

## Role-Based Access Control / RBAC

一句话定义：RBAC 是按角色授予权限的访问控制方法，例如客服、主管、法务、管理员有不同数据和工具权限。

大模型场景：RBAC 需要贯穿 RAG 召回、引用展示、Agent 工具调用和日志查询，避免模型把无权数据通过生成结果泄露出来。

## Tenant Isolation

一句话定义：tenant isolation 是保证不同租户、部门、项目或客户的数据、索引、日志和配置不会互相泄露的隔离机制。

企业门禁：租户隔离违规率通常应接近 0；平均准确率再高，也不能覆盖跨租户数据泄露风险。

## Permission-Aware RAG

一句话定义：permission-aware RAG 是在检索、重排、上下文拼接、引用展示和日志记录中都执行权限过滤的 RAG。

常见误区：只在 UI 隐藏无权文档不够，向量检索阶段和生成上下文阶段也必须过滤，否则模型可能从无权片段中总结出敏感信息。

## Tool Permission Gate

一句话定义：tool permission gate 是 Agent 或 LLM 调用企业工具前，对用户身份、角色、动作、数据范围和风险等级做授权检查的门禁。

大模型场景：读数据、改数据、发邮件、改权限、执行代码、创建工单和退款等动作要分级处理，高风险动作需要二次确认、dry run、人审和审计。

## Audit Log Coverage

一句话定义：audit log coverage 是请求 trace 中完整记录身份、数据来源、引用、工具调用、权限判断、模型版本、输出和复核状态的比例。

为什么重要：没有审计覆盖，系统出错后无法复盘，也无法证明数据访问和工具调用符合企业流程。

## Data Freshness

一句话定义：data freshness 是知识库、指标口径、文档版本和业务数据是否足够新的程度。

大模型场景：企业 RAG 引用过期文档会造成错误决策。数据新鲜度要结合更新时间、版本号、owner、失效策略和业务阈值一起看。

## Workflow Integration

一句话定义：workflow integration 是大模型应用进入企业已有工单、审批、CRM、ERP、BI、代码仓库或办公流程的程度。

产品化意义：没有工作流集成，LLM 往往只是聊天 demo；接入工作流后，才能衡量任务完成、节省时间、转人工率、审批通过率和业务收益。

## Enterprise Gate

一句话定义：enterprise gate 是企业 LLM 应用上线前的综合门禁，覆盖权限、租户隔离、RAG 权限过滤、工具权限、审计、PII 脱敏、数据新鲜度、引用、SSO、工作流、评估、反馈、SLO、业务指标和高风险人审。

面试表达：企业门禁不是一个平均分，而是一组硬约束。权限、租户隔离、审计或高风险人审不过线时，即使 demo 效果好，也只能继续试点或改成人机协同。

## Productization Gate

一句话定义：productization gate 是大模型产品上线前的综合门禁，覆盖痛点、质量提升、单位经济账、延迟、安全合规、工作流适配、评估和反馈闭环。

面试表达：一个模型效果更好的方案，如果成本、延迟、风险或工作流不过线，也不应该直接产品化。

## Feedback Loop

一句话定义：feedback loop 是产品上线后把日志、用户反馈、人工审核、失败样本、A/B test 和线上指标转成下一轮评估与优化数据的闭环。

为什么重要：没有反馈闭环，系统很难适应真实用户长尾、知识库变化、模型升级和业务流程变化。

## Online Monitoring

一句话定义：online monitoring 是大模型产品上线后持续跟踪质量、成本、延迟、安全、反馈、trace 和业务指标的监控体系。

面试表达：大模型线上监控不能只看 QPS 和平均延迟，还要看任务成功率、采纳率、引用支持、bad case 分类、P95/P99、单位成本、事故率、反馈行动率和回归门禁。

## Canary Rollout

一句话定义：canary rollout 是先把新模型、prompt、RAG 策略、工具 schema 或安全策略暴露给小比例用户或低风险流量，再逐步扩大的灰度发布方式。

为什么重要：灰度让团队在真实流量中发现长尾问题，同时把失败影响限制在可回滚范围内。

## Regression Pass Rate

一句话定义：regression pass rate 是一次模型、prompt、检索、工具或安全策略变更后，回归评估集中通过样本的比例。

面试表达：回归通过率要按业务、风险、权限、语言、长度和高价值用户切片看；平均通过率好看但高风险切片失败时，不能直接放量。

## Trace Coverage

一句话定义：trace coverage 是应该保留可审计执行轨迹的线上请求中，实际记录了版本、检索证据、工具调用、关键指标和决策路径的比例。

为什么重要：没有 trace，很难解释 Agent、RAG 或工具调用为什么失败，也很难把线上 bad case 转成复现评估集。

## Rollback Plan

一句话定义：rollback plan 是新版本失败时回到旧模型、旧 prompt、旧检索策略、旧工具 schema 或人工兜底流程的预案。

产品化意义：没有回滚能力的上线只能算冒险试验；成熟产品通常要配 feature flag、旧版本保留、配置回滚、数据回滚、告警和事故沟通流程。

## Ops Gate

一句话定义：ops gate 是大模型产品上线或扩大灰度前的运营门禁，覆盖质量、采纳、反馈行动、bad case triage、回归、切片、延迟、成本、安全、trace 和回滚。

面试表达：ops gate 强调上线后的可运营性。一个产品即使 demo 效果不错，如果反馈没人处理、bad case 无法复现、事故不能止血、trace 缺失或无法回滚，也不应直接全量上线。

## Project Collaboration Incident

一句话定义：project collaboration incident 是大模型项目因为目标、指标、baseline、版本、责任边界、变更控制、灰度、回滚或复盘缺失而导致实验发散、上线风险、事故不可复现或跨团队扯皮的协作事故。

典型来源：目标只写“提升效果”，没有用户和场景；离线指标没有护栏；baseline 没冻结；prompt 热修无版本；RACI 不清；高风险变更未升级；bad case 没进入回归；复盘 action item 没关闭。

面试表达：协作事故不能只归因成“沟通不好”。更工程化的说法是把它拆成 objective clarity、metric tree、baseline、experiment record、version trace、RACI、change control、rollout、rollback、postmortem 和 action item closure 等可审计字段。

## Objective Clarity Rate

一句话定义：objective clarity rate 衡量项目或实验中目标是否明确写出用户、场景、任务、主指标、护栏指标和失败边界的比例。

工程意义：目标不清会直接导致实验发散。大模型项目尤其要把“模型更聪明”改写成解决率、引用准确率、P95 延迟、权限泄露率、成本等可检查目标。

## Baseline Coverage

一句话定义：baseline coverage 衡量项目、实验或上线候选中有多少样本或变更能和冻结 baseline 做公平对比。

常见误区：只展示新版本好样例，或者同时改模型、prompt、检索、数据和评估脚本后声称单点有效，都不能算有可信 baseline。

## Version Trace Coverage

一句话定义：version trace coverage 衡量线上请求、实验记录和事故样本中是否记录了模型、prompt、数据、检索索引、工具 schema、安全策略、代码和配置版本。

工程意义：大模型系统的版本不仅是模型 checkpoint。缺 prompt hash、索引版本或工具 schema 版本时，线上事故很可能无法复现。

## RACI Coverage

一句话定义：RACI coverage 衡量项目关键环节是否明确 Responsible、Accountable、Consulted 和 Informed，或者至少有等价的 DRI 责任人机制。

面试表达：责任边界不是为了甩锅，而是为了缩短决策链路。线上幻觉、权限泄露、延迟超标和数据质量问题都应该能找到直接执行人和最终负责人。

## Change Control Coverage

一句话定义：change control coverage 衡量模型、prompt、数据、检索、工具、安全策略和配置变更是否经过记录、评审、回归、灰度和可回滚流程。

工程意义：prompt、RAG 配置和安全策略都是生产逻辑。热修不记录版本，等价于把不可审计代码直接推到生产。

## Project Collaboration Gate

一句话定义：project collaboration gate 是把目标清晰度、指标树、baseline、实验复现、版本 trace、RACI、变更控制、风险升级、灰度、回滚、bad case 回归、复盘完备率和 action item 关闭率放在一起的项目协作门禁。

面试表达：项目管理不是开会汇报进度，而是把模糊协作问题变成可检查的工程门禁。如果 `project collaboration gate` 不通过，说明项目即使短期指标好看，也还没有达到可上线、可复盘、可持续迭代的状态。

## Product Interview Readiness

一句话定义：product interview readiness 是算法工程师面对技术产品类问题时，能否用用户价值、技术方案、指标公式、demo 证据、风险边界和 trade-off 组成完整回答的准备度。

为什么重要：技术产品面试不只考“懂不懂模型”，还考能否证明模型方案值得做、能上线、能治理、能持续迭代。

## Product Interview Gate

一句话定义：product interview gate 是技术产品面试复盘门禁，检查回答是否覆盖核心主题、公式、demo、风险、trade-off 和弱题修复计划。

面试表达：如果回答缺 ROI 公式、RAG / Agent / 隐私 / 运营 demo 证据，或只讲概念不讲门禁，就说明还没准备好被深挖。

## Practitioner Interview Readiness

一句话定义：practitioner interview readiness 是实战型大模型工程师能否把项目讲成目标、baseline、指标、实验、失败、debug、trade-off、上线、安全、成本和复盘闭环的准备度。

工程意义：真实面试的深挖通常不止问“你用了什么技术”，还会追问证据、失败样例、边界和取舍。准备度不足时，回答会从工程事实滑向泛泛表达。

## Project Evidence Score

一句话定义：project evidence score 衡量项目回答是否有目标、baseline、指标、bad case、debug 路径和生产化证据。

面试表达：项目证据分低时，不要先背更复杂的术语，而要补齐评估集来源、分桶指标、失败样例、版本 trace、灰度、监控、回滚、安全和成本证据。

## Trade-off Depth

一句话定义：trade-off depth 衡量回答是否同时讲清收益、代价、决策和验证方式。

面试表达：资深表达不是“用了更强方案”，而是能说明为什么没有全量启用更大模型、更长上下文、更高 top-k 或更强 reranker，以及怎样用指标证明这个取舍可上线。

## STAR Reflection Score

一句话定义：STAR reflection score 衡量行为面回答是否覆盖 Situation、Task、Action、Result 和 Reflection。

工程意义：大模型项目的 Reflection 应落到评估集、上线门禁、版本治理、权限边界、监控告警或行动项关闭，而不是空泛说“以后沟通更好”。

## Expert Follow-up Readiness

一句话定义：expert follow-up readiness 衡量回答面对专家追问时，能否回到 baseline、指标、失败样例、trade-off 和证据边界。

面试表达：专家追问的关键不是把所有问题都答满，而是诚实区分已验证结论、合理假设和下一步验证方法。

## Practitioner Interview Gate

一句话定义：practitioner interview gate 是实战项目面试准备门禁，综合检查项目证据、trade-off 深度、STAR 复盘、专家追问防守、英文清晰度和红旗表述。

面试表达：如果门禁不过线，说明项目故事还不能稳定承受深挖。修复动作应该绑定具体弱题，例如补 baseline、补失败样例、补安全边界、补 action item 验证或重写“为什么不用更大模型”的回答。

## 本章小结

本章从语言模型的概率定义出发，串起了大语言模型的基础概念。面试中要抓住一条主线：LLM 通过 next-token prediction 预训练得到 base model，再通过指令微调和对话对齐变成 instruct model 或 chat model，最终通过 prompt、上下文、解码策略和外部工具服务具体任务。

高频结论如下：

1. 语言模型本质上是在建模 token 序列概率。
2. next-token prediction 是大规模自监督预训练的核心目标。
3. 自回归模型训练时可并行算 loss，推理时必须逐 token 生成。
4. base model 擅长续写，instruct model 擅长遵循指令，chat model 擅长多轮对话。
5. perplexity 能衡量语言建模拟合程度，但不能替代任务评测。
6. in-context learning 不更新参数，只利用当前上下文临时适配任务。
7. 涌现能力要从规模、数据、优化和评测指标共同理解。
8. 幻觉是生成式语言模型的结构性问题，需要检索、工具、校验和对齐共同缓解。
