## LLM Serving

一句话定义：LLM serving 是把训练好的大模型部署成可被线上请求调用的推理服务。

它解决什么问题：训练完成的模型只是参数文件。要让用户稳定使用，还需要 tokenizer、chat template、请求队列、batch 调度、KV Cache 管理、解码策略、流式输出、监控告警、限流、容灾和成本控制。

与普通深度学习 serving 的区别：LLM 请求长度差异大，输出长度不可预知，推理过程分为 prefill 和 decode 两个阶段，decode 还存在逐 token 串行依赖。因此，LLM serving 的核心瓶颈经常不是一次 forward，而是调度、KV Cache 和显存带宽。

常见目标：低延迟、高吞吐、稳定质量、低成本、高可用、可观测、可扩展。

面试表达：LLM serving 不是简单把模型 `model.generate()` 包成接口，而是围绕长序列、动态输出和 KV Cache 做系统级优化。

## LLM Serving Engine Overview Audit

一句话定义：LLM Serving Engine Overview Audit 是检查一个推理框架是否真正覆盖请求生命周期、prefill / decode、KV cache、scheduler、streaming、metrics、runtime 边界和成本容量的总览审计方法。

它解决什么问题：很多回答会把 serving engine 说成 `model.generate()` 的 HTTP 包装，或者只背 vLLM、SGLang、TGI、Triton 的名字。总览审计要求把请求从 parse、tokenize、queue、prefill、decode、stream 到 finish / cleanup 的状态链路讲清，并能写出 TTFT、TPOT、KV cache 显存、token 吞吐和单位 token 成本。

面试表达：一个合格的 LLM serving engine 不是“能生成文本”，而是能在高并发下管理请求、token、KV、batch、streaming、错误、取消、观测和成本。

## From Scratch Mini Serving Engine

一句话定义：From Scratch Mini Serving Engine 是用最小代码把 naive generate loop 拆成 tokenizer、request state、prefill、decode、scheduler、streamer、KV cleanup 和 metrics 的教学版推理引擎。

它解决什么问题：单请求 `generate()` 把请求状态、prefill / decode、KV cache、采样和输出都藏在一个黑盒里。最小推理框架把这些状态显式化，让读者先理解“为什么需要 serving engine”，再学习 vLLM、SGLang 等复杂实现。

常见门禁：request state machine 正确流转，streaming chunks 可见，prefill / decode 指标可统计，请求结束后 KV cache 被释放，事件 trace 能复盘每轮调度和生成。

面试表达：从零实现推理框架时，不要一开始追求高性能；先把单请求 loop 拆成可调度、可观测、可取消、可清理的最小系统闭环。

## Request Lifecycle Audit

一句话定义：Request Lifecycle Audit 是检查一次 LLM 请求从接入、校验、tokenization、排队、调度、prefill、decode、streaming 到 finish / abort / timeout / cleanup 是否形成闭环的审计方法。

它解决什么问题：很多推理框架回答只背 API Server、Scheduler、Worker 和 KV Cache Manager，却说不清一个请求在状态机里怎么流转、什么时候占用 KV、什么时候释放资源、异常路径如何返回 finish reason。生命周期审计要求正常结束、客户端取消、排队超时和执行失败都进入统一清理路径。

常见门禁：request object 字段完整，WAITING / PREFILLING / DECODING / FINISHED / ABORTED / TIMEOUT / FAILED 状态可追踪，TTFT / TPOT / queue / prefill / decode 指标可拆分，streaming chunk 或 finish reason 可解释，请求结束后 KV cache 和 running slot 被释放，trace 能复盘关键事件。

面试表达：推理请求不是一次函数调用，而是带状态、资源、输出通道和异常收敛路径的生命周期对象；只有 happy path 和 exception path 都能清理资源，才算 serving engine 可靠。

## Prefill Decode KV Streaming Audit

一句话定义：Prefill Decode KV Streaming Audit 是把一次 LLM 推理拆成 prefill、decode、KV cache 和 token streaming 四个阶段，并用 TTFT、TPOT、KV pressure、stream backlog 和 cleanup 共同验收的教学审计方法。

它解决什么问题：很多人知道 prefill、decode、KV cache 和 streaming 的名字，却不能解释它们如何共同影响用户体验和系统容量。这个审计把 prompt token、output token、prefill steps、decode steps、KV MiB、stream chunks、backpressure 和 finish reason 放到一张表里，说明长 prompt 拉高 TTFT，长输出增加 decode steps，长上下文和并发推高 KV pressure，streaming 还要处理客户端消费速度和资源释放。

常见门禁：prefill accounting 不丢 prompt token，decode accounting 和输出 token 对齐，KV cache 显存估算包含层数、KV head、head dim 和 dtype bytes，streaming chunk 数和输出 token 对齐，客户端慢读或断开能触发 backpressure / cleanup，最终 Phase Gate 不只看吞吐，还看 TTFT、TPOT、KV 和 finish reason。

面试表达：Prefill 决定首 token 等多久，Decode 决定后续 token 多快，KV Cache 决定高并发显存边界，Streaming 决定用户感知和取消清理；四者必须一起看，不能只用 QPS 或 GPU utilization 判断推理系统好坏。

## Serving Metric Cost Audit

一句话定义：Serving Metric Cost Audit 是把 TTFT、TPOT、input / output tokens/s、active sequences、KV pressure、tail latency、timeout rate 和 cost per 1k tokens 放在同一张压测表里的指标成本审计方法。

它解决什么问题：传统 QPS 和平均延迟无法解释 LLM serving 的真实负载。两个请求 QPS 都是 1，但 input / output token、prefill、decode、KV cache 和 streaming 可能差几个数量级。指标成本审计要求压测报告同时给出 token 分布、阶段延迟、p95 / p99、KV 显存压力、有效吞吐和单位 token 成本。

常见门禁：TTFT p95 / p99 满足首响 SLO，TPOT p95 满足流式输出 SLO，input / output tokens/s 可解释，KV pressure 低于水位，active sequence capacity 有余量，timeout rate 不超标，cost per 1k tokens 在预算内，Metric Cost Gate 不允许只因吞吐好看就忽略长尾和成本。

面试表达：评估推理服务性能要从“请求数”升级到“token + 阶段 + 成本”视角；一个系统可能 tokens/s、KV 和成本都过线，但 TTFT p99 或 E2E p99 失败，仍然不能算用户体验达标。

## Serving Boundary Audit

一句话定义：Serving Boundary Audit 是审计推理框架、推理平台和 AI Infra 三层职责、接口、指标交接、事故归层和成本容量证据是否清楚的方法。

它解决什么问题：很多系统设计回答会把 vLLM、API gateway、Kubernetes、GPU driver、灰度、限流、计费和监控混在一起说。边界审计要求先明确 engine 负责 token 执行路径，platform 负责服务治理控制面，AI Infra 负责 GPU、容器、调度、网络、存储和隔离；再检查 request API、metrics export、control drain、health probe、trace context 是否打通。

常见门禁：layer coverage 覆盖 engine / platform / infra 的核心职责，interface contract coverage 覆盖请求、指标、控制、健康检查和 trace，metric handoff coverage 覆盖 TTFT / TPOT / queue length / KV pressure / tenant quota / route version / pod pending / GPU health，incident layer routing 能把 TTFT、TPOT、成本、扩容和硬件异常排到正确层级，Serving Boundary Gate 不允许只会背工具名。

面试表达：推理平台不是“部署一个 vLLM Pod”，而是 engine、platform 和 infra 通过接口、指标和事件证据协作；边界越清楚，排障和优化越能走到正确 owner。

## Minimal Generate Loop Audit

一句话定义：Minimal Generate Loop Audit 是审计最小 tokenizer、model wrapper、logits 选择、token 追加、停止条件和 trace 是否构成可解释生成闭环的方法。

它解决什么问题：直接调用 `model.generate()` 会把 encode、forward、last logits、token selection、append、stop、decode 和 cache 更新都藏起来。最小生成循环审计要求先用 toy tokenizer 和 toy model wrapper 跑通最小链路，让读者知道文本如何变成 token ids，模型如何输出词表 logits，greedy 如何选下一个 token，以及 naive loop 为什么会重复计算历史 token。

常见门禁：Tokenizer Contract 保证 encode / decode 类型清楚，Model Wrapper Contract 保证输入 token ids、输出最后位置 logits，Last Logits Selection 只用最后位置预测下一个 token，Stop Condition 覆盖 EOS 和最大生成长度，Naive Recompute Work 能量化重复计算，Minimal Generate Gate 要求 trace、metrics 和停止原因都可复盘。

面试表达：理解推理框架要先拆开最小 generate loop；否则后面讲 KV cache、batching、scheduler 和 streaming 时，只是在黑盒外面背名词。

## Sampling Strategy Audit

一句话定义：Sampling Strategy Audit 是审计 greedy、temperature、top-k、top-p、seeded multinomial 和采样参数 trace 是否正确控制候选分布的方法。

它解决什么问题：很多回答会说 temperature 控制随机性、top-k / top-p 控制候选集合，但说不清 logits、softmax、过滤、归一化和采样顺序。采样策略审计要求能从 logits 出发，先做稳定 softmax 或 greedy，再按请求参数应用 temperature、top-k、top-p，最后从过滤后的概率分布中采样，并记录候选集合、entropy、seed 和 selected token。

常见门禁：Stable Softmax 要减 max 防溢出，Temperature Softmax 要能解释低温和高温的 entropy 差异，Top-k Candidate Filter 固定保留高分 token，Top-p Nucleus Filter 按累计概率动态保留候选，Seeded Multinomial Sampling 在简单单请求路径可复现，Sampling Gate 不能让所有概率被过滤为空。

面试表达：sampling 不是“让模型更聪明”的按钮，而是对下一个 token 分布做可解释约束；它会影响质量、多样性、输出长度、停止命中、TPOT 和成本。

## Minimal KV Cache Audit

一句话定义：Minimal KV Cache Audit 是审计单请求 prefill / decode 是否正确使用 past key values、追加 cache、保持输出等价并减少重复计算的教学方法。

它解决什么问题：很多人知道 KV Cache 能加速生成，但说不清 cache 里保存什么、prefill 和 decode 的输入有什么区别、为什么 decode 只传最新 token、cache 为什么会增加显存。最小 KV Cache 审计要求先用 toy attention 或最小模型验证“完整重算最后 token”和“缓存路径最后 token”一致，再统计 cache length、重复投影减少量和 KV 显存估算。

常见门禁：Past Key Values Contract 要说明每层 key / value 的 shape 和生命周期；Prefill Cache Build 要用完整 prompt 建初始 cache；Decode Cache Append 要每步只计算新 token 并追加 K/V；KV Cache Equivalence Check 要验证缓存路径和完整重算路径输出一致或在可解释容差内；KV Cache Memory Formula 要包含 key/value 两份、层数、KV head、head dim、active tokens 和 dtype bytes；KV Cache Gate 要求正确性、工作量下降和显存估算都可复盘。

面试表达：KV Cache 不是减少模型权重显存，而是用额外显存换掉 decode 阶段的重复历史计算；单请求最小 cache 跑通后，才有资格继续讨论 batch decode、PagedAttention、prefix cache 和 KV 迁移。

## Batched Prefill Audit

一句话定义：Batched Prefill Audit 是审计多个 prompt 是否能被正确 padding、mask、一次 prefill、按最后真实 token 取 logits 并建立 batch KV cache 的教学方法。

它解决什么问题：从单请求走向多请求时，不能只把 prompts 放进 list。不同 prompt 长度需要 padding，padding 必须配 attention mask，首 token logits 必须来自每个请求最后一个真实 token；右 padding 下直接取 `logits[:, -1, :]` 会让短请求读到 PAD 位置。Batched Prefill Audit 要求同时验证左 / 右 padding、last real token gather、padding waste 和 token budget。

常见门禁：Padding Side Policy 要说明左 padding 和右 padding 的 logits 取法差异；Attention Mask Contract 要让 PAD 不参与有效上下文；Last Real Token Logits 要能用 mask 找到每个请求最后真实位置；Padding Waste Ratio 要量化 padded tokens 占比；Prefill Token Budget 要限制单轮真实 prompt tokens 或 padded tokens；Batched Prefill Gate 要求首 token 选择、mask、长度、KV cache 估算和 padding waste 都可复盘。

面试表达：batched prefill 不是 batch size 越大越好，而是在 GPU 并行、TTFT、padding waste、KV 显存和后续 decode batch 之间做取舍；先把最后真实 token logits 取对，再谈 continuous batching。

## Batched Decode Audit

一句话定义：Batched Decode Audit 是审计多个已完成 prefill 的请求是否能在同一 decode step 中生成新 token，并保持 request state、finished mask、position、attention mask 和 KV cache 行顺序一致的教学方法。

它解决什么问题：batched decode 不是把 last token ids 拼成 tensor 就结束。每个请求有自己的输出、停止条件、prompt length、cache length 和 sampling 参数；finished 请求如果继续留在固定 batch 里，会浪费 decode 行；active compaction 虽然更高效，但必须同步选择和回写对应的 KV cache。

常见门禁：Finished Mask 要阻止已完成请求继续追加 token；Batch Cache Alignment 要维护 batch row -> request id -> KV cache slot 映射；Decode Position Tracking 要按请求自己的长度更新 position ids 和 attention mask length；Active Batch Compaction 要只对活跃请求构造 decode batch；Decode Row Savings 要量化 fixed batch 与 compact batch 的行数差异；Batched Decode Gate 要求输出、停止原因、cache length 和 row/cache alignment 都可复盘。

面试表达：batched decode 的核心不是一次 forward，而是让每一行 token、请求状态和 KV cache slot 永远对齐；这一步做对后，才能继续实现动态加入和退出的 continuous batching。

## Simple Scheduler Audit

一句话定义：Simple Scheduler Audit 是审计教学版推理 scheduler 是否能维护 waiting queue、running set、finished set，并用 max active sequences 与 token budget 做最小准入控制的方法。

它解决什么问题：很多人会把 scheduler 误解成队列 `pop()`。推理 scheduler 的输入不只是 waiting queue，还包括正在 decode 的请求、可用 running slot、prefill token budget、decode token 数、KV 压力和完成清理。Simple Scheduler Audit 要求最小系统能解释每轮谁 decode、谁 prefill、谁 deferred、为什么 deferred，以及这些决策如何影响 queue wait 和 TTFT。

常见门禁：Waiting Queue 保存尚未 prefill 的请求；Running Set 只保存仍需 decode 的请求；Scheduler Admission 同时检查可用 slot 和 token budget；Max Active Sequences 不超过配置；Queue Wait Metric 和 TTFT Steps 可按请求计算；Scheduler Trace 能复盘每轮 arrivals、prefill、decode、waiting、running 和 scheduled tokens；Simple Scheduler Gate 要求所有请求进入终态，预算不超限，deferred 可解释。

面试表达：简单 scheduler 的价值不是性能，而是把请求流、状态机、token budget 和指标闭环显式化；这一步做对后，才能继续讨论 continuous batching、chunked prefill、PagedAttention 和多租户调度。

## Token Streaming Audit

一句话定义：Token Streaming Audit 是审计推理 engine 是否能把 prefill 首 token 和每轮 decode 结果转换成可消费的增量事件，并正确处理 finish、stop sequence、取消和 backpressure 的教学方法。

它解决什么问题：很多实现会把 streaming 简化成每个 token `print` 一下，但真实 serving 需要把模型执行和网络输出解耦。Token Streaming Audit 要求定义 Streaming Event Contract，把 delta、finish reason、usage 和 error 放入每请求 stream queue；用 Incremental Detokenization 避免单 token decode 边界问题；用 Stop Sequence Buffer 防止跨 token stop sequence 泄露；客户端取消和慢读都必须反馈给 engine，避免继续占用 GPU 与 KV cache。

常见门禁：First Token Chunk 要在 prefill 首 token 后发出；Finish Event Contract 要保证每个请求都有终态事件；Incremental Detokenization 要能重构完整文本；Stop Sequence Buffer 要隐藏 stop sequence；Client Cancellation Cleanup 要停止后续 decode 并记录 finish reason；Streaming Backpressure 要能检测慢客户端或队列积压；Streaming Gate 要求 delta、finish、取消、背压和 trace 都可复盘。

面试表达：Streaming 不是前端体验细节，而是请求生命周期的一部分；它把 TTFT、TPOT、事件队列、取消反馈和资源清理连在一起，任何一个环节缺失都会让用户体验或 GPU 资源治理失真。

## Minimal HTTP API Audit

一句话定义：Minimal HTTP API Audit 是审计教学版 LLM serving engine 是否能通过最小 HTTP API 接收 JSON 请求、校验参数、提交到 engine、返回同步或流式结果，并处理过载与取消清理的方法。

它解决什么问题：很多项目把 `model.generate()` 包进 Web handler 就叫推理服务，但这会把协议层、请求状态、队列准入、streaming 事件和 KV cleanup 混在一起。最小 HTTP API 审计要求先定义 Generate Request Contract，把 prompt、max new tokens、sampling params 和 stream flag 校验清楚；再用 Queue Admission Control 在队列满时返回 429；同步路径返回 request id、text、finish reason 和 usage；流式路径用 SSE Frame Format 返回 delta 和 finish event；客户端断开必须触发 HTTP Cancellation Cleanup，通知 engine 停止 decode 并释放 KV。

常见门禁：Request Validation Gate 要区分 400 输入错误和 500 服务端错误；Queue Admission Control 要避免高峰期无限排队；Sync Generate Response 要包含终态和 usage；Streaming Generate Response 要遵守 `text/event-stream`、`event:`、`data:` 和空行分隔；HTTP Cancellation Cleanup 要把 abort、KV release 和 finish reason 收敛到同一清理路径；HTTP API Gate 要求 validation、admission、sync、stream 和 cancel 都可复盘。

面试表达：最小 HTTP API 的价值不是多一个 endpoint，而是把外部协议层和 engine 执行层解耦；handler 负责校验、准入、连接和响应，engine 负责调度、prefill、decode、streaming 和资源释放。

## Serving Benchmark Audit

一句话定义：Serving Benchmark Audit 是用 TTFT、TPOT、E2E latency、tokens/s、queue、active requests、KV memory 和 cleanup 共同审计 LLM serving engine 性能瓶颈的方法。

它解决什么问题：LLM serving 的负载不是普通 QPS，prompt 长度、输出长度、prefill、decode、KV cache 和 streaming 都会改变成本。Serving Benchmark Audit 要求压测报告先说明模型、硬件、workload、并发和到达模式，再记录每请求 TTFT Metric、TPOT Metric、E2E Latency Metric、input / output Token Throughput Report、Queue Length Percentile、Active Request Percentile 和 KV Memory Peak。只给平均延迟或 QPS，会掩盖长 prompt、burst、queue 堆积和 decode 长尾。

常见门禁：TTFT Metric 要在客户端可见首 token 时记录；TPOT Metric 要说明是否从首 token 后开始算；Token Throughput Report 要分 input tokens/s 和 output tokens/s；Queue Length Percentile 和 Active Request Percentile 要能解释 TTFT 长尾；KV Memory Peak 要区分模型权重、KV cache、工作区和 runtime 开销；cleanup 后显存或 KV token 要回落；Benchmark Bottleneck Classification 要能把问题归到 queue / prefill、decode / streaming、KV capacity 或 workload；Benchmark Gate 要求 trace、latency、throughput、KV、cleanup 和归因都可复盘。

面试表达：推理压测不是跑一个 QPS 数字，而是证明系统在给定 workload 和硬件下，首响、后续 token、吞吐、显存和错误路径都可测、可解释、可定位。

## vLLM Motivation Audit

一句话定义：vLLM Motivation Audit 是审计为什么 naive serving engine 需要升级到 vLLM 式 block-based KV cache 和 continuous batching 的教学方法。

它解决什么问题：很多人把 vLLM 简化成“PagedAttention 更快”，但 vLLM 的动机来自系统瓶颈：Naive KV Reservation 为每个请求连续预留 prompt + max output，实际输出短时会浪费；固定 batch decode 让短请求陪长请求跑空行；finished / cancelled 请求如果不能释放和复用 KV，显存容量会被吃掉。vLLM Motivation Audit 要求用 Paged KV Block Allocation 解释逻辑 token 到物理 block 的映射，用 KV Waste Ratio 量化连续预留与块内浪费差异，用 Static Decode Rows 与 Continuous Decode Rows 解释动态调度为什么减少无效 decode 行。

常见门禁：Naive KV Reservation 要能算出预留容量和实际使用容量；Paged KV Block Allocation 要能展示 block count 和 Logical Physical Block Mapping；Block Reuse After Cleanup 要证明 finished 请求释放的 block 可复用；Static Decode Rows 和 Continuous Decode Rows 要能量化固定 batch 与 iteration-level scheduling 的差距；vLLM Motivation Gate 要求 KV waste、block reuse、continuous batching、cleanup 和 metrics 都可复盘。

面试表达：vLLM 不是“一个 attention kernel”，而是围绕 KV cache、block manager、scheduler、worker、streaming 和 metrics 组成的 serving engine；PagedAttention 是核心，但必须和 continuous batching 与请求生命周期一起讲。

## PagedAttention Core Audit

一句话定义：PagedAttention Core Audit 是审计一个推理系统是否能把请求逻辑 KV cache 映射到非连续 physical KV blocks，并正确处理地址翻译、释放、复用和 prefix sharing 的方法。

它解决什么问题：连续 `past_key_values` 适合教学，但动态请求和高并发下容易遇到预留浪费、碎片和 batch row 重排困难。PagedAttention Core Audit 要求定义 Logical Block、Physical KV Block 和 Block Table Address Translation：token position 先算 logical block 与 offset，再通过 block table 找到 physical block。请求增长时按 block 分配；请求结束时释放不再被引用的 blocks；共享 prefix 时用 Prefix Block Ref Count 避免重复存储；Freed Block Reuse 证明非连续空闲块能继续服务新请求。

常见门禁：Block Table Address Translation 要能把 token position 映射到正确 physical block 和 offset；Block Size Tradeoff 要解释小 block 降低内部浪费但增加映射和 kernel 开销；Internal Block Waste 要能统计尾块空洞；Prefix Block Ref Count 要保证共享块不会被提前释放；Freed Block Reuse 要证明 cleanup 后 block 回到 free list；PagedAttention Gate 要求地址翻译、共享、释放、复用和浪费统计都可复盘。

面试表达：PagedAttention 不只是更快的 attention，而是 KV cache 的分页式内存管理；attention kernel 的特殊之处，是能在计算时根据 block table 读取非连续的物理 KV blocks。

## KV Block Manager Audit

一句话定义：KV Block Manager Audit 是审计推理 engine 是否能把 physical KV blocks 当成可分配、可追踪、可回收资源来管理的方法。

它解决什么问题：PagedAttention 需要 block table 和 physical KV blocks，但分配、释放、复用和准入不是 attention kernel 自己完成的。KV Block Manager Audit 要求定义 Block Free List、Block Reference Count 和 Sequence block table：prefill 前用 Prefill Block Admission 判断 prompt 是否能放下；decode 时用 Decode Block Extension 在跨 block 边界时追加 block；请求 finished / aborted / failed 时走 Shared Prefix Cleanup，只在 ref count 归零时归还 block；free blocks 不足时记录 Block Allocation Failure，而不是让 scheduler 调度后再 OOM。

常见门禁：Prefill Block Admission 要按 token 数换算所需 blocks；Decode Block Extension 要能在 block 满时追加 physical block；Block Reference Count 要正确处理 prefix sharing；Shared Prefix Cleanup 要释放非共享块并保留共享块；Block Manager Metrics 要至少包含 free blocks、used blocks、shared blocks、allocation failures 和 blocks per request；Block Manager Gate 要求 prefill、decode、ref count、free、admission 和 metrics 都可复盘。

面试表达：Block Manager 不是一个简单 free list，而是 scheduler 和 PagedAttention 之间的 KV 资源控制面；没有它，continuous batching、prefix sharing、cleanup 和可靠准入都会失真。

## Continuous Batching Audit

一句话定义：Continuous Batching Audit 是审计推理 scheduler 是否能在每个模型 iteration 动态选择 prefill / decode 请求，并在 token budget 与 KV block budget 下安全运行的方法。

它解决什么问题：固定 batch 会让短请求陪长请求跑空行，新请求也要等整批完成后才能进入 prefill。Continuous Batching Audit 要求用 Iteration Level Scheduling 把 request lifetime 和 execution batch lifetime 解耦；Dynamic Request Join 允许 waiting 请求在后续 iteration 进入；Dynamic Request Exit 允许 finished / cancelled 请求移出 running set；Decode First Scheduling 保障已有 streaming 请求的 TPOT；Continuous Token Budget 控制单轮 prefill + decode tokens；Scheduler KV Budget 要求 scheduler 在调度前查询 block manager，避免能算但放不下。

常见门禁：Toy Continuous Batcher 要能输出每轮 arrived、decode、prefill、waiting、running、used tokens、free blocks 和 Deferred Scheduling Reason；Static Batch Hole 要能量化固定 batch 空洞；finished / cancelled 请求必须触发 cleanup；Continuous Batching Gate 要求 dynamic join / exit、decode-first、token budget、KV budget、cleanup 和 metrics trace 都可复盘。

面试表达：Continuous batching 不是把 batch size 开大，而是把调度粒度降到每个 iteration；它必须和 PagedAttention、Block Manager、streaming、TTFT / TPOT 指标以及异常清理一起讲，才是可上线的 serving scheduler。

## vLLM Scheduler Flow Audit

一句话定义：vLLM Scheduler Flow Audit 是审计一个 vLLM-like engine 中请求从 API 输入、engine request、waiting queue、scheduler output、model runner metadata 到 output processor 和 cleanup 的完整链路是否闭环的方法。

它解决什么问题：很多人把 vLLM 请求流程只讲成“进队列、调度、跑模型”，但真实 serving engine 要维护 Engine Request Contract、Request State Machine Trace、KV block table、Slot Mapping Metadata、采样参数、streaming buffer、finish reason 和 metrics。Scheduler Output Metadata 不能只是请求 id 列表，而要能转换成 Model Runner Execution Metadata，包括 input ids、positions、block tables、slot mapping、phase 和 sampling metadata。Output Processor Finish Gate 还要把 EOS、stop string、max tokens 和 abort 反馈给 engine core，触发 Abort Cleanup Path。

常见门禁：Engine Request Contract 要包含 request id、input ids、sampling params、状态、block table、输出 tokens 和时间戳；Request State Machine Trace 要覆盖 WAITING、PREFILL、RUNNING / DECODING、FINISHED / ABORTED；Scheduler Output Metadata 要通过 token、sequence 和 KV budget；Model Runner Execution Metadata 要能还原 positions、slot mapping 和 block table；Request Flow Metrics 要包含 queue wait、TTFT、E2E、finish reason 和 KV blocks after cleanup；Request Flow Gate 要求 input、state、scheduler、metadata、output、cleanup 和 metrics 都可复盘。

面试表达：vLLM 请求调度流程不是某个类名，而是一条状态和资源链路；scheduler 决定本轮执行计划，block manager 保障 KV 资源，model runner 消费 execution metadata，output processor 决定请求是否继续进入下一轮，终止路径必须释放资源。

## vLLM Memory Management Audit

一句话定义：vLLM Memory Management Audit 是审计 vLLM-like engine 是否能把 GPU memory 中最动态的 KV Cache 管理成可分配、可复用、可淘汰、可观测 block pool 的方法。

它解决什么问题：模型权重相对静态，但 KV Cache 会随请求数、prompt 长度、输出长度和 prefix cache 状态动态变化。vLLM Memory Management Audit 要求先做 GPU Memory Breakdown，区分 weight、KV、workspace、CUDA graph、sampling、adapter 和 reserved memory；再用 KV Page Size、block table 和 free queue 解释 KV block 如何分配；用 Prefix Cache Hash 绑定 parent hash、block tokens 和 extra hashes，避免错误复用；用 Cached Free Block 和 LRU Cache Eviction 解释为什么 free 不等于清空；用 Preemption Recompute Path 解释 KV 空间不足时如何保护系统。

常见门禁：Toy KV Memory Manager 要能展示 block pool、free queue、request block table、cached blocks、ref count、prefix hit、cached-but-free、eviction、preemption / recompute、cleanup 和 metrics；Prefix Cache Hit Rate 要区分 lookup 与 hit；Hybrid KV Cache Group 要说明 full attention、sliding window 和 state space 层的 cache 需求不同；Memory Management Gate 要求 pool、prefix、ref count、free queue、eviction、preemption、cleanup 和 metrics 都可复盘。

面试表达：vLLM 的显存管理不是只看 `nvidia-smi`，而是看 KV block pool 是否有预算、是否能复用、是否能释放、是否能在内存压力下抢占 / 重算，以及这些行为是否能通过 metrics 解释 TTFT、TPOT、OOM 和 preemption。

## vLLM Executor Architecture Audit

一句话定义：vLLM Executor Architecture Audit 是审计 vLLM-like engine 是否把 API、engine core、executor、worker、model runner 和 model 的职责边界拆清，并能把 scheduler output 可靠派发到 GPU workers 的方法。

它解决什么问题：很多人把 vLLM 架构讲成“scheduler 调模型”，但真实 serving engine 要分清 Engine Core Dispatch Loop、Executor Dispatch Layer、Worker Rank Mapping 和 Model Runner Metadata Gate。Engine core 负责请求状态、scheduler 和 KV cache；executor 负责把本轮执行计划派发给 worker group；worker 绑定 device、rank / local rank 和模型资源；model runner 把 request-level scheduler output 转成 input ids、positions、block tables、slot mapping、attention metadata 和 sampling metadata；Tensor Parallel Merge 则负责把多个 worker 的 shard 输出聚合。

常见门禁：Toy Executor Architecture 要能展示 TP/PP/DP 下 expected worker count、rank/local rank/device 映射、worker input metadata 完整性、executor dispatch trace、TP merge 结果、CPU Process Budget 和 Worker Failure Recovery；Executor Architecture Gate 要求 layer boundary、dispatch、runner metadata、rank mapping、merge、CPU budget、failure 和 metrics trace 都可复盘。

面试表达：scheduler 决定“谁该跑”，executor 决定“怎么把这轮执行计划发给哪些 worker 跑”。worker 通常一进程一 GPU，model runner 是把调度结果落成 tensor 和 metadata 的执行层。这个边界讲清后，TP、PP、DP、worker failure 和 CPU 资源瓶颈才有分析入口。

## Prefix Caching Audit

一句话定义：Prefix Caching Audit 是审计 vLLM-like runtime 是否能安全复用相同 prompt 前缀对应 KV full blocks 的方法。

它解决什么问题：prompt cache 是平台层复用前缀处理结果的宽泛概念，prefix caching 是 runtime 内部复用 KV blocks 的具体机制。Prefix Caching Audit 要求先区分 prompt cache、KV cache、result cache 和 semantic cache；再用 Full Block Prefix Reuse 说明为什么只缓存 full blocks；用 Parent Hash Chain 说明相同 block tokens 在不同前文后不能复用；用 Extra Hash Isolation 和 Cache Salt Isolation 约束 model、tokenizer、LoRA、多模态媒体 hash 和租户边界；最后用 Cached Free Touch、ref count 和 Prefix Cache Eviction 解释命中、释放和 LRU 淘汰的生命周期。

常见门禁：Toy Prefix Cache Manager 要能展示 full block 命中、partial suffix 重新 prefill、cached-free block 被 touch 出 free queue、不同 salt / LoRA / media hash miss、LRU eviction、Prefix Saved Prefill Tokens、Prefix Cache Hit Rate、cached-free blocks 和 eviction count；Prefix Cache Gate 要求 hash、full block、touch、ref count、salt isolation、eviction 和 metrics 都可复盘。

面试表达：prefix caching 主要减少重复长前缀的 prefill，所以核心收益是 TTFT 和 prefill 吞吐；它不是万能缓存，也不能绕过模型、tokenizer、LoRA、多模态输入和租户隔离约束。

## Serving Parallelism Audit

一句话定义：Serving Parallelism Audit 是审计 LLM serving 是否根据模型大小、显存、拓扑、吞吐和延迟目标正确选择 TP / PP / DP / EP 的方法。

它解决什么问题：很多回答会把“多 GPU”说成统一加速，但 serving 并行要先区分瓶颈。Tensor Parallel Topology 关注单层张量切分和高频 collective 是否留在 NVLink / NVSwitch 范围内；Pipeline Parallel Bubble 关注层切分、跨节点激活传输和 microbatch 空泡；Data Parallel Replica 关注复制模型副本扩吞吐、每个 replica 独立 KV cache 和 Prefix Cache Locality；Expert Parallel Placement 关注 MoE experts 分布、all2all 通信和 expert 负载不均。

常见门禁：Toy Serving Parallel Auditor 要能展示 Worker Group Size、总 GPU 数、Per Rank Memory Fit、TPOT 估算、Cross Node TP Risk、PP bubble、DP throughput、prefix cache locality、EP imbalance 和 Parallel Serving Gate；不能只写 `TP=8` 或 `DP=4`，必须说明为什么这个配置适合当前模型、硬件和流量目标。

面试表达：TP/PP 解决单个模型副本放不下或单副本执行分布问题，DP 解决横向吞吐，EP 解决 MoE expert 容量。并行策略必须跟 GPU 拓扑、KV cache、prefix cache、CPU 进程和 p95 / p99 延迟一起讨论。

## vLLM Performance Tuning Audit

一句话定义：vLLM Performance Tuning Audit 是把 TTFT、TPOT、吞吐、KV cache、preemption、prefix cache、CPU、streaming 和并行拓扑放到同一张证据表里做调优决策的方法。

它解决什么问题：很多调优会直接调大 batch、加 GPU 或换量化，但没有说明瓶颈在哪里。vLLM Performance Tuning Audit 要先用 TTFT Bottleneck Diagnosis 拆 queue、tokenize、prefill、prefix cache 和前置链路；用 TPOT Bottleneck Diagnosis 拆 decode step、KV 读取、TP 通信、sampling、detokenize 和 streaming；用 KV Pressure Tuning、Preemption Rate 和 allocation failure 判断 `max_num_seqs`、`max_num_batched_tokens`、`max_model_len` 与 `gpu_memory_utilization` 是否过激；用 Prefix Cache Effectiveness 判断 hit rate、saved prefill tokens、DP route locality 和 eviction 是否真的带来收益。

常见门禁：Toy vLLM Tuning Auditor 要能输出 per-request TTFT / TPOT / queue / prefill / KV / cache 指标，识别 queue-or-prefill TTFT、decode-or-streaming TPOT、KV pressure / preemption、prefix cache ineffective、DP route breaks cache locality、cross-node TP risk 和 CPU / streaming path；Config Trade-off Plan 要同时说明参数收益和副作用，Rollback Guard 要覆盖 p99、OOM、error rate 和质量回滚阈值；vLLM Tuning Gate 要求 metrics、TTFT、TPOT、KV、cache、config 和 rollback 都可验收。

面试表达：vLLM 调优不是背参数，而是用真实流量 trace 证明瓶颈、改一个主要变量、压测 p50 / p95 / p99、观察 KV 与 cache 指标，再灰度上线并保留回滚。

## SGLang Motivation Audit

一句话定义：SGLang Motivation Audit 是判断一个 workload 是否真的需要 SGLang-like 前端语言和 runtime 协同优化的方法。

它解决什么问题：很多回答会把 SGLang 说成“另一个 vLLM server”或“支持 JSON 的 OpenAI API wrapper”。SGLang Motivation Audit 要先把任务拆成 Complex LLM Program，看它是否包含多次 generation、变量引用、分支、并行、工具 / RAG / multi-turn 状态和 structured output；再用 RadixAttention Motivation 和 Radix Prefix Reuse 估算 naive prefill 与 radix prefill 的差距；最后用 Structured Decoding Retry Saving 解释 JSON / regex / EBNF constrained decoding 为什么能减少后处理失败和重试尾延迟。

常见门禁：Toy SGLang Motivation Audit 要能输出 naive prefill tokens、radix prefill tokens、saved prefill tokens、reuse ratio、branch sharing、retry tokens without constraints、grammar overhead tokens、naive work tokens、SGLang work tokens、work reduction 和 SGLang Motivation Gate；不能只说“有 RadixAttention 所以快”，必须说明 frontend language 如何让 runtime 看见共享前缀、分支和结构化输出。

面试表达：vLLM 更偏高效服务大量请求，SGLang 更偏高效执行复杂 LLM programs。是否选择 SGLang，要看 workload 是否有多步控制流、树状 prefix sharing、结构化输出和 runtime 可利用的程序结构，而不是只看框架名。

## SGLang Runtime Audit

一句话定义：SGLang Runtime Audit 是把 SGLang 的 OpenAI-compatible API、native `/generate`、offline engine 和 frontend program 统一放进 request state、scheduler、RadixAttention、memory pool、model runner、sampler、grammar backend 和 streaming 链路中验收的方法。

它解决什么问题：很多回答会把 SGLang Runtime 说成 HTTP wrapper、RadixAttention 数据结构或某个 JSON 输出功能。SGLang Runtime Audit 要先做 Runtime Entrypoint Unification，说明四类入口如何落到同一套请求执行后端；再用 Request State Contract 解释 token ids、phase、sampling params、KV slot、grammar state、streaming 和 finish reason；用 Radix Prefix Lookup 和 KV Memory Pool Budget 说明 prefill 前的 prefix hit 与显存硬约束；最后用 Prefill Decode Scheduler、Grammar Mask Step 和 Streaming Event Trace 说明 runtime 如何逐轮执行和输出。

常见门禁：Toy SGLang Runtime Audit 要能输出 entrypoints、total input tokens、run prefill tokens、saved prefill tokens、reuse ratio、prefill rounds、decode rounds、decode rows、KV peak slots、KV pressure、grammar steps、stream events、avg TTFT 和 SGLang Runtime Gate；不能只画一张 API server 到 model 的箭头图。

面试表达：SGLang Runtime 的核心是把不同入口统一到一条高性能生成链路，并在 tokenized prompt 之后、prefill 之前做 radix prefix lookup，在 scheduler 和 memory pool 约束下执行 prefill / decode，再把 grammar mask 和 sampler、streaming 输出放到 decode loop 里。

## RadixAttention Prefix Sharing Audit

一句话定义：RadixAttention Prefix Sharing Audit 是审计 SGLang-like runtime 是否能用压缩前缀树安全复用 token prefix 对应 KV cache 的方法。

它解决什么问题：很多回答会把 RadixAttention 误说成新的 attention kernel，或者只说“前缀缓存会更快”。RadixAttention Prefix Sharing Audit 要先说明它是 runtime 级 KV cache reuse 机制，不替代 FlashAttention / attention backend；再用 Compressed Prefix Tree、Longest Prefix Match 和 Radix Tree Split 解释为什么复杂 LLM programs 的树状共享前缀需要 radix tree；用 Page Aligned Prefix Hit 说明 page size / block size 会影响可复用完整页；用 Radix KV Ref Count 和 Leaf LRU Eviction 说明 active path 不能淘汰，显存紧张时优先释放 ref count 为 0 的叶子；最后用 Cache Aware Scheduling Cost 说明 scheduler 应按未命中 suffix 而不是原始 prompt 长度估算 prefill 成本。

常见门禁：Toy RadixAttention Cache 要能输出每个请求的 token hit、page hit、hit owner、run prefill tokens、new cached tokens、split count、cached tokens、pinned nodes、evicted leaf nodes、reuse ratio 和 RadixAttention Gate；不能只画 trie，必须证明 split、page 对齐、ref count、leaf eviction 和 scheduler cost 都可复盘。

面试表达：RadixAttention 的收益主要在 prefill / TTFT，不直接消除 decode TPOT。它要求 token prefix 完全一致，并且模型、tokenizer、chat template、LoRA、多模态输入、position 规则和租户隔离等 forward 条件一致。

## SGLang Scheduler Audit

一句话定义：SGLang Scheduler Audit 是审计 SGLang-like runtime 能否在每个 engine step 里同时处理 decode、prefill、RadixAttention 命中、KV cache 容量、structured output、abort cleanup 和公平性的验收方法。

它解决什么问题：很多回答会把 scheduler 简化成“把请求凑成 batch”。SGLang Scheduler Audit 要先用 Decode First Scheduling 解释 running streaming 请求为什么要稳定推进；再用 Cache Aware Admission 和 Suffix Cost Scheduling 说明 waiting 请求不能只看 prompt length，而要看 RadixAttention 命中后的未命中 suffix；用 Scheduler Token Budget、Scheduler Sequence Budget 和 Scheduler KV Admission 说明 token、请求数和 KV slots 是三类不同硬约束；最后用 Scheduler Aging Fairness、Chunked Prefill Scheduling、Scheduler Abort Cleanup 和 structured output grammar cost 说明吞吐、TTFT、TPOT、显存和公平性之间的取舍。

常见门禁：Toy SGLang Scheduler 要能输出 scheduled decode ids、scheduled prefill ids、scheduled tokens、scheduled sequences、prefill suffix tokens、naive prompt tokens、grammar steps、cleaned aborts、evicted nodes、pinned nodes、KV free after scheduling、deferred reasons 和 SGLang Scheduler Gate；不能只输出一个 request id 列表。

面试表达：SGLang scheduler 的核心是用同一轮 budget 同时约束 prefill 和 decode，并把 RadixAttention 的 hit length 转成 suffix cost。KV 不够时只能淘汰安全 cached nodes，决定复用 prefix 后要 pin 住，长 prompt 要 chunked prefill，cache hit 优先也要配合 aging 防饥饿。

## Structured Generation Audit

一句话定义：Structured Generation Audit 是审计 runtime 是否把 JSON schema、regex、EBNF、structural tag 或 choices 这类输出约束放进 token selection 过程，而不是只依赖 prompt 和生成后解析的方法。

它解决什么问题：很多应用要求模型输出能被程序消费的对象、标签、DSL 或 tool call。Structured Generation Audit 要先用 Grammar State 说明每个请求都有独立语法状态；用 Valid Token Mask 说明 sampler 每步只能从合法 token 集合中选择；用 JSON Schema Constraint、Regex Choice Constraint、EBNF Grammar Constraint 和 Structural Tag Constraint 说明不同约束适合的任务边界；再用 Empty Valid Token Set 说明约束冲突必须显式失败；最后用 Format Not Fact Gate 和 Structured Streaming Partial 说明格式合法不等于事实正确，streaming 中间 chunk 也通常不是完整对象。

常见门禁：Toy Constrained Decoder 要能输出 constrained JSON text、parseability、fact check、choice / EBNF / structural tag 结果、mask calls、blocked high-score invalid tokens、stream partial parseability、empty valid token detection 和 Structured Generation Gate；不能只给一个 prompt “请输出 JSON”。

面试表达：Constrained decoding 的关键是在采样前根据 grammar state 计算合法 token 集合，把非法 token logits 置为负无穷。它能减少格式错误和重试，但不保证事实正确，还会带来 grammar compile / mask latency，因此 scheduler 和 dashboard 要单独观测 structured output 请求。

## Speculative Decoding Audit

一句话定义：Speculative Decoding Audit 是审计 serving runtime 是否真正实现了 draft propose、target verify、accepted prefix 提交、fallback 修正、临时 KV 清理和收益指标闭环的方法。

它解决什么问题：很多回答会把 speculative decoding 说成“小模型替代大模型”或“必然加速”。Speculative Decoding Audit 要先用 Draft Source 说明 draft 可以来自 EAGLE / EAGLE3、MTP、Standalone 小模型、NGRAM 或其他机制；再用 Target Verify Call 和 Accept Length 说明 target model 才是最终裁判；用 Draft Acceptance Rate、Target Call Reduction 和 tokens per verify 说明收益来自一次 verify 提交多个 token；最后用 Speculative Fallback Token、Speculative KV Cleanup 和 Adaptive Speculative Steps 说明低接受率、grammar 阻断、rejected token 回滚、显存峰值和动态 steps 都必须进入工程证据链。

常见门禁：Toy Speculative Decoder 要能输出每个请求的 baseline target calls、spec target calls、drafted tokens、accepted tokens、fallback tokens、grammar blocked tokens、discarded temporary KV tokens、acceptance rate、tokens per target call、target call reduction、latency speedup estimate 和 Speculative Decoding Gate；不能只给一句“draft 猜多个 token，所以更快”。

面试表达：Speculative decoding 主要优化 decode / TPOT，不直接优化 prefill。它是否有效取决于 draft 成本、接受率、target verify 并行效率、scheduler overhead、structured output 约束和 KV 显存压力；只有 accepted tokens 才能进入正式上下文和 prefix cache，rejected speculative tokens 必须丢弃或回滚。

## Agent Serving Audit

一句话定义：Agent Serving Audit 是审计 runtime 和应用层是否能把 multi-turn、tool use、tool result 回灌、agent loop、多分支轨迹、prefix cache 和 task-level metrics 串成可靠执行链路的方法。

它解决什么问题：很多回答会把 agent serving 简化成“模型会调工具”。Agent Serving Audit 要先用 Multi-turn Runtime State 说明 session history、tokenized history、KV refs、grammar state 和 streaming offset 都会跨 generation 延续；用 Session Aware Routing 和 Agent Trajectory Prefix Sharing 说明同一 session 或 agent trajectory 后续轮次要尽量复用同一 replica 的 RadixAttention / prefix cache；再用 Tool Parser Fit、Tool Schema Validation 和 Tool Result Backfill 说明 parser、structured output、validator、tool executor 和 tool message 回灌是不同层；最后用 Tool Wait GPU Release、Agent Round Budget 和 task-level metrics 说明工具等待期间不能占用 GPU decode slot，agent loop 必须有轮数、token、上下文和成本上限。

常见门禁：Toy Agent Serving Audit 要能输出每次 generation 的 replica、prompt length、prefix hit length、run prefill tokens、tool name、validator result、blocked reason、session affinity、naive / run / saved prefill tokens、tool calls requested / executed / blocked、tool latency、stream events、model calls per task 和 Agent Serving Gate；不能只画一个 ReAct loop。

面试表达：Multi-turn、tool use 和 agent serving 是复杂 LLM program 的典型 workload。SGLang-like runtime 的价值在于用 RadixAttention 复用稳定历史和分支轨迹，用 structured output 和 parser 让工具调用可解析，用 scheduler 管理多次 generation 和工具等待，并用 task-level 指标拆分模型、cache、工具和 agent 逻辑的瓶颈。

## Serving Architecture Comparison Audit

一句话定义：Serving Architecture Comparison Audit 是审计候选人能否按 workload、runtime 可见结构、KV cache 复用、scheduler 成本、structured output / agent 需求和指标证据来比较 vLLM-like 与 SGLang-like 架构的方法。

它解决什么问题：很多回答会把架构对比变成“谁更快”或“谁替代谁”。Serving Architecture Comparison Audit 要先用 Common Serving Core 说明两者都要处理 API、tokenizer、prefill / decode、KV cache、scheduler、sampler、streaming 和 metrics；再用 Paged Prefix Cache Evidence 说明 vLLM-like 体系中 PagedAttention / KV block / full-block prefix cache 解决的是显存布局和通用 serving；用 Radix Program Reuse Evidence 说明 SGLang-like 体系中 RadixAttention / frontend runtime 协同更关注多步 program、分支和 trajectory 的 prefix reuse；最后用 Workload Fit Score 和 Runtime Workload Router 说明独立 chat、共享 RAG、结构化抽取、agent tree 可能适合不同 backend。

常见门禁：Toy Architecture Comparator 要能输出每类 workload 的 naive prefill tokens、vLLM saved / run prefill、SGLang saved / run prefill、vLLM fit score、SGLang fit score、recommended backend、总体 saved ratio、SGLang extra saved、vLLM Not Replaced Gate、SGLang Program Fit Gate 和 Architecture Comparison Gate；不能只列静态功能表。

面试表达：PagedAttention 和 RadixAttention 不在同一抽象层。前者主要关注 KV cache 物理块和显存碎片，后者主要关注 token prefix / program trajectory 的自动复用。vLLM 很适合通用高并发 OpenAI-compatible serving，SGLang 更能体现复杂 LLM program 的 runtime 优势；真实平台可以按 workload 做路由，而不是强行二选一。

## mini-sglang Source Path Audit

一句话定义：mini-sglang Source Path Audit 是审计源码学习是否覆盖入口、request state、scheduler、KV cache、RadixAttention、model runner、sampler、structured output、streaming 和 cleanup 主链路的方法。

它解决什么问题：很多回答会把“读过源码”停留在文件名和类名。mini-sglang Source Path Audit 要先用 Runtime Module Map 说明每个模块的输入、输出、状态和资源；用 Request Lifecycle Order 检查阅读顺序是否沿外部请求到内部 request、scheduler、KV、engine、sampler、output 的生命周期推进；再用 Source Resource Coverage 说明 token budget、sequence budget、KV allocate / write / read / release、prefix cache 都被解释过；最后用 Source Experiment Coverage、Radix Split Experiment 和 Abort Cleanup Experiment 说明必须跑最小生成、共享前缀、radix split、长短混排、streaming、structured output 和 abort cleanup 等实验。

常见门禁：Toy Source Path Auditor 要能输出 module coverage、resource coverage、experiment coverage、lifecycle order、实验触达模块比例、observable signal count、missing modules / resources / experiments 和 Source Path Gate；不能只说“先读 scheduler，再读 cache”。

面试表达：mini-sglang 的价值是用更小代码建立 SGLang Runtime 骨架。真正看懂源码要能把模块职责映射到 TTFT、TPOT、KV pressure、prefix hit、grammar mask、streaming event 和 abort cleanup，而不是背诵某个版本的文件路径。

## Prefill Decode Resource Profile Audit

一句话定义：Prefill Decode Resource Profile Audit 是把 LLM 推理请求按 prompt token、output token、prefix hit、prefill compute、decode KV read、TTFT、TPOT 和 P/D 干扰拆开审计的方法。

它解决什么问题：很多回答会把 PD 分离动机背成“prefill 偏计算、decode 偏显存”，但说不清为什么。Prefill Decode Resource Profile Audit 要先用 Request Token Profile 区分 prompt、output 和 prefix hit；用 Run Prefill Tokens 说明长 prompt 命中缓存后不一定长 prefill；再用 Prefill Compute Profile、Prefill KV Write 和 Decode KV Read 解释为什么 prefill 更容易形成大 token batch，而 decode 每步虽然 token 少，却会反复读历史 KV；最后用 TTFT TPOT Separation、Long Prefill Stall Risk、Decode-first Starvation Risk 和 PD KV Transfer Cost 说明 chunked prefill 与 PD 分离各自缓解什么、额外付出什么。

常见门禁：Toy Prefill Decode Profiler 要能输出每个请求的 run prefill tokens、decode steps、prefill compute units、decode compute units、prefill KV write MiB、decode KV read MiB、TTFT、TPOT，以及全局 prefill / decode token ratio、long prefill stall、chunked prefill stall、decode-first wait、PD KV transfer 和 Prefill Decode Profile Gate；不能只给一句资源倾向结论。

面试表达：Prefill / Decode 资源画像题要按“token 分布 -> prefix hit -> prefill compute / KV write -> decode KV read -> TTFT / TPOT -> 长 prefill 卡 decode -> decode-first 让 prefill 饥饿 -> chunked prefill 和 PD 分离取舍 -> KV transfer cost”展开。这样才能把 PD 分离讲成可观测、可压测、可权衡的系统问题。

## PD Disaggregation Motivation Audit

一句话定义：PD Disaggregation Motivation Audit 是审计系统是否真的存在 P/D 干扰、资源池扩缩容耦合和尾延迟治理困难，并且 KV transfer 成本可控的方法。

它解决什么问题：很多回答会一上来画 prefill pool、decode pool 和 KV transfer backend，但没有证明为什么需要拆。PD Disaggregation Motivation Audit 要先用 Unified Engine Interference 说明长 prefill 会打断 decode；用 Prefill Interruption Evidence 和 Decode-first Starvation Evidence 分别证明 TPOT 抖动和 TTFT 饥饿；再用 Independent P/D Scaling 说明输入输出比例不同导致统一扩容浪费；最后用 KV Transfer Cost Gate 和 Slow Transfer Counterexample 说明 PD 分离不是免费午餐，transfer 过慢时收益会被吞掉。

常见门禁：Toy PD Disaggregation Auditor 要能输出 unified engine 的 decode jitter p99、prefill wait p99、prefill / decode time，PD 架构的 prefill pool / decode pool time、KV transfer MiB、largest request transfer time、interference saving、慢 transfer 反例，以及 PD Motivation Gate；不能只说“拆开就更快”。

面试表达：PD 分离解决的是统一 engine 内 P/D 互相干扰、资源画像不匹配、独立扩缩容困难和 p99 难治理。判断是否值得做，要看 long prefill 是否打爆 TPOT、decode-first 是否拖高 TTFT、输入输出压力是否需要不同资源池，以及 KV transfer 是否足够快。没有这些证据，先做 chunked prefill、prefix cache、scheduler 调优和观测。

## PD Architecture Audit

一句话定义：PD Architecture Audit 是审计 PD 分离系统是否把 router、prefill worker、decode worker、KV transfer backend、bootstrap metadata、跨组件状态机和资源清理串成可靠闭环的方法。

它解决什么问题：很多回答会把 PD 架构画成三四个方框，但说不清请求失败时谁清理资源。PD Architecture Audit 要先用 PD Router 说明它不是普通负载均衡器，而是阶段感知、负载感知、KV locality 感知的 coordinator；再用 Prefill Worker Pool、Decode Worker Pool 和 Decode Reservation 说明 P/D worker 职责边界；用 KV Transfer Backend 和 Bootstrap Metadata 说明 source / target KV block、rank、layout 和 session 如何对齐；最后用 Cross Component State Machine、Transfer Failure Cleanup、Client Abort Cleanup 和 Model Version Compatibility 说明失败、取消和版本不一致不能静默放过。

常见门禁：Toy PD Architecture Auditor 要能输出每个请求的 states、prefill worker、decode worker、KV metadata、stream events、cleanup 标记，以及全局 finished / aborted / transfer failed、transfer sessions、total transfer MiB、cleanup events、max decode running 和 PD Architecture Gate；不能只说“router 负责调度”。

面试表达：PD 分离架构本质上是把单 engine 的 WAITING -> PREFILLING -> DECODING 状态机升级为跨 router、prefill worker、transfer backend、decode worker 的分布式状态机。合理架构必须证明 decode 先预留目标 slot、KV metadata 完整、transfer 失败会释放两侧资源、client abort 会广播清理、P/D worker 模型版本兼容，并且分阶段指标可观测。

## KV Transfer Routing Audit

一句话定义：KV Transfer Routing Audit 是审计 PD 分离中 KV cache 从 prefill 侧交给 decode 侧时，metadata、容量预留、路由成本、租户隔离、重算兜底和失败清理是否闭环的方法。

它解决什么问题：很多回答会把 KV transfer 说成“把 KV cache 复制过去”，但真正风险在正确性和治理。KV Transfer Routing Audit 要先用 KV Metadata Compatibility 检查 model version、tokenizer、dtype、layout、TP size 和 rank / block mapping 是否一致；再用 Decode Capacity Reservation 说明目标 decode worker 必须先预留 KV blocks；用 KV Routing Cost Score 把 free blocks、running sequences、pending transfers、TPOT、网络 zone 和 prefix reuse 合并到路由决策；最后用 Tenant Cache Isolation、Recompute Fallback Path 和 Transfer Failure Plan 说明跨租户 prefix 命中不能复用，短 prompt 可以重算，transfer 失败必须释放目标 blocks 并保留可重试证据。

常见门禁：Toy KV Transfer Router 要能输出每个请求的 migrate / recompute / route_failed 决策、decode worker、required blocks、reserved blocks、KV transfer MiB、transfer ms、score、prefix reuse、tenant cache blocked、failure plan，以及全局迁移数、重算数、route failed、total transfer MiB、reserved blocks、skipped incompatible 和 KV Transfer Routing Gate；不能只说“选择空闲 decode worker”。

面试表达：KV transfer 题要按“metadata 兼容 -> decode 容量预留 -> transfer 成本估算 -> prefix / locality 路由 -> 租户隔离 -> recompute fallback -> 失败释放和重试 -> metrics”展开。这样才能说明 KV cache 是带布局、版本、租户和生命周期的运行时状态，而不是普通 byte payload。

## Chunked Disaggregated Prefill Audit

一句话定义：Chunked Disaggregated Prefill Audit 是审计长 prompt prefill 是否被切成可调度 chunk，并且能和 decode interleaving、PD transfer、backpressure 与失败清理形成闭环的方法。

它解决什么问题：很多回答会把 Chunked Prefill 说成“把 prompt 切块”，但说不清它和 Disaggregated Prefill 的层级差异。Chunked Disaggregated Prefill Audit 要先用 Run Prefill Tokens 和 Prefill Chunk Count 说明 prefix hit 后真正要执行多少 prefill、会拆成几个 chunk；再用 Decode Interleaving 和 Prefill Token Budget 说明每轮调度要把 decode rows 与 prefill chunk 一起算进预算；用 Position Continuity 说明 chunk 边界不能让 position、attention mask 或 KV block table 断裂；最后用 Short Request Fairness、PD Transfer Pipeline 和 Backpressure Cleanup 说明短请求不能被长 prefill 饿死，按 chunk transfer 可以流水化，但下游拥塞和 abort 必须释放 partial KV。

常见门禁：Toy Chunked Disagg Prefill Auditor 要能输出 full prefill summary、chunked rows、chunked summary、PD transfer summary 和 gates，至少覆盖 max full prefill stall、short request wait、scheduled rounds、chunks run、decode steps interleaved、max chunk prefill ms、completion round、aborted requests、cleanup events、pending transfer chunks、backpressure events、total transfer MiB、pipeline transfer saving 和 Chunked Disagg Prefill Gate；不能只画一个“切块”时间线。

面试表达：Chunked Prefill 回答的是 how to run a long prefill，Disaggregated Prefill 回答的是 where to run prefill。前者是执行粒度和调度公平性问题，后者是 P/D 资源隔离和 KV transfer 问题。二者结合时，要同时证明 token budget、position continuity、KV block table、partial transfer、backpressure、abort cleanup 和 metrics 都可观测。

## Multi Level KV Cache Audit

一句话定义：Multi Level KV Cache Audit 是审计 KV block 在 GPU、CPU、SSD、remote 和 recompute 之间如何按 residency、成本、正确性和延迟门禁做决策的方法。

它解决什么问题：很多回答会把多级 KV Cache 讲成“GPU 放不下就放 CPU / 远端”，但没有证明何时搬、何时重算、谁不能搬。Multi Level KV Cache Audit 要先用 KV Residency Level 说明 cache hit 还要看 KV 在哪一层；用 Active GPU Block Protection 说明 active decode 的 KV 不能为了命中率被挤出 GPU；再用 CPU KV Promote、Remote KV Fetch 和 Recompute Fallback 对比 transfer cost 与 recompute cost；最后用 KV Tenant Isolation、KV Demotion Policy 和 Residency Metrics 说明跨租户命中要阻断，GPU 满时只 demote ref count 为 0 的 block，并且指标要同时包含 transfer MiB、residency、latency 和 fallback。

常见门禁：Toy Multi Level KV Cache Auditor 要能输出每个 block 的 action、cost、transfer MiB、demoted victims，以及全局 GPU hit、CPU promote、remote fetch、SSD fetch、recompute、tenant block、demotion、GPU / CPU / remote residency、total transfer MiB、estimated latency 和 Multi Level KV Gate；不能只说 cache hit rate 高。

面试表达：多级 KV Cache 的关键不是多放几层存储，而是让 scheduler 知道每个 KV block 的 residency、移动成本和正确性边界。active decode KV 优先留 GPU；CPU 和 remote 用于恢复、复用和跨 worker 共享；SSD / remote 命中如果比重算慢，就应该重算；任何 model、tokenizer、position、adapter、parallel config 或 tenant 不匹配都必须阻断。

## Cross Node Network Audit

一句话定义：Cross Node Network Audit 是审计跨节点 serving 中 TP、PP、PD KV transfer、remote KV cache、router 和 backpressure 哪些通信进入 TTFT / TPOT 关键路径的方法。

它解决什么问题：很多回答会把跨节点 serving 讲成“多机多卡扩容”，但没有区分通信模式。Cross Node Network Audit 要先用 Cross Node Link Profile 说明带宽、基础延迟和 p99 拥塞不是一回事；用 TP Cross Node Risk 说明 TP collective 是 per-token 高频通信，跨慢网络会直接打爆 TPOT；再用 Pipeline Activation Cost 和 PD KV Transfer Cost 分别计算 PP activation 与长 prompt KV transfer；用 Remote Recompute Decision 说明 remote hit 要和 recompute 比；最后用 Topology Aware Routing、Transfer Backpressure 和 Control Data Plane Separation 说明 router 不能只看 worker load，大对象 KV transfer 也不能压垮控制面。

常见门禁：Toy Cross Node Network Auditor 要能输出 decode worker scores、TP same-node / cross-rack TPOT overhead、PP activation ms、PD KV transfer MiB / ms、remote hot / cold decision、router choice、pending transfer MiB、backpressure events、data/control plane MiB 和 Cross Node Network Gate；不能只说“网络带宽要足够”。

面试表达：跨节点 serving 要按通信频率和关键路径拆开讲。DP 主要是请求分流；TP 是每层每 token 的 collective，通常应放在单节点高速互联内；PP 跨节点传 activation；PD 跨节点传 KV；remote cache 需要比较 fetch 与 recompute；router 要 topology-aware / KV-aware，并且用 admission control 和 backpressure 限制大 KV transfer。

## PD Tradeoff Audit

一句话定义：PD Tradeoff Audit 是把 PD 分离从“高级架构选择”落成收益、成本、readiness、反模式和替代方案共同验收的决策方法。

它解决什么问题：很多回答会把 PD 分离当成默认终态，但 PD Tradeoff Audit 要先用 PD Benefit Cost Score 比较 P/D 干扰收益、独立扩缩容收益、KV transfer p99 和状态机复杂度；再用 PD Positive Case 说明什么样的长 prompt / streaming workload 值得试；用 Short Request Anti Pattern、Decode Bottleneck Anti Pattern 和 Slow Transfer Anti Pattern 说明短请求、长输出 decode 瓶颈、跨节点网络差时不该上 PD；最后用 Ops Readiness Gate 和 PD Alternative Path 说明 metrics、cleanup、backpressure 不齐时要先补治理，且每个 rejected case 都要有 chunked prefill、prefix cache、DP、speculative decoding 或 same-node PD 等替代路径。

常见门禁：Toy PD Tradeoff Auditor 要能输出每个 case 的 score、decision、reasons、alternatives，以及全局 adopt / rejected、anti patterns、positive cases、blocked cases 和 PD Tradeoff Gate；不能只说“长 prompt 多就做 PD”。

面试表达：PD 分离适合长 prompt 干扰 decode、P/D 比例波动、多租户隔离和独立扩缩容收益明显，并且 KV transfer、监控、失败清理和 backpressure 成熟的场景。不适合短请求、单机已达标、主要瓶颈在 decode 或网络、运维能力不足的场景。判断标准是 P/D 隔离收益是否大于 KV transfer 与系统复杂度成本。

## Single Engine To PD Migration Audit

一句话定义：Single Engine To PD Migration Audit 是把已有单机 serving engine 升级成 PD 分离时，对阶段顺序、接口契约、状态迁移、same-node 原型、失败清理、观测和回滚能力做门禁验收的方法。

它解决什么问题：很多改造会直接把 `prefill()` 和 `decode()` 拆成两个服务，但没有先外置 request state、稳定 KV metadata、拆分 scheduler、引入 router 状态机和 cleanup / fallback。Single Engine To PD Migration Audit 要先用 Explicit Request Stage、Model Runner Interface Split 和 KV Metadata Contract 稳住单进程内部边界；再用 Prefill Decode Scheduler Split 和 PD Router State Machine 验证逻辑 PD；然后用 Same Node PD Prototype、Cleanup Cancel Path 和 Observability Fallback Readiness 证明 transfer、cancel、backpressure、metrics 和 fallback 可控；最后才进入 cross-node PD。

常见门禁：Toy PD Migration Auditor 要能输出 migration rows、ready steps、interfaces ready、state machine ready、cleanup paths、fallback ready、same-node-before-cross-node 和 PD Migration Gate；不能只画一张 router / P worker / D worker 架构图。

面试表达：单机 engine 升级 PD 的顺序是先改接口和状态，再拆 worker 和部署。先把 request stage 显式化，把 model runner 拆成 prefill / decode，把 KV manager 和 KV metadata 标准化；再拆 prefill / decode scheduler，引入 router 管 lifecycle；先做 same-node PD 验证 transfer、reservation、cleanup、cancel、metrics、backpressure 和 fallback；最后才做 cross-node PD 和拓扑路由。

## nano-vLLM Source Learning Audit

一句话定义：nano-vLLM Source Learning Audit 是用模块地图、请求路径、engine step、KV block 生命周期、model runner batch contract 和实验信号验收 nano-vLLM 源码学习是否真的可复盘的方法。

它解决什么问题：很多回答会把“读过 nano-vLLM”停在文件列表和类名层面。nano-vLLM Source Learning Audit 要先用 Source Module Map 覆盖 example、LLM、sampling params、Sequence、LLMEngine、Scheduler、BlockManager、ModelRunner、Attention、Sampler 和模型结构；用 Generate Trace Path 与 Engine Step Trace 说明请求如何从 `generate()` 进入 waiting / running 调度，再到 prefill / decode、sampler 和 append token；用 Sequence State Audit、Scheduler Decision Trace、KV Block Lifecycle 和 Model Runner Batch Contract 说明 request state、block table、free blocks、positions、slot mapping、logits 和 sampled token 如何跨模块流动；最后用 Source Experiment Signal 证明这些路径能通过最小 generate、prefill/decode step、scheduler budget、KV block reuse、prefix cache hit、finish cleanup 和 sampler params 实验观察到。

常见门禁：Toy nano-vLLM Source Auditor 要能输出 module coverage、resource coverage、experiment coverage、lifecycle order、runtime module touch、observed signal count、missing modules / resources / experiments 和 nano-vLLM Source Gate；不能只说“先读 scheduler，再读 block manager”。

面试表达：nano-vLLM 适合用来建立 vLLM-like engine 骨架。阅读顺序沿 request lifecycle：example / LLM -> sampling params -> Sequence -> LLMEngine -> Scheduler -> BlockManager -> ModelRunner -> Attention -> Sampler -> model。项目证据要用 trace、scheduler 日志、KV block 可视化、TTFT / TPOT、prefix hit 和 cleanup 实验说明，而不是只背源码文件名。

## tiny-LLM Learning Audit

一句话定义：tiny-LLM Learning Audit 是把 tiny-llm 从模型算子、最小 generate 到 KV cache、batching、chunked prefill 和 paged attention 的学习路线落成概念覆盖、实验覆盖和依赖顺序门禁的方法。

它解决什么问题：很多回答会把 tiny-llm 当作“能生成文本的小模型教程”，但 tiny-LLM Learning Audit 要先用 Operator To Model Path、Attention Shape Trace、RoPE Position Alignment 和 GQA KV Head Audit 证明 Q/K/V shape、position、num query heads / KV heads 和 decoder layer 组装可解释；再用 Generate Sampling Loop 证明自回归生成和 sampling policy 可运行；然后用 KV Cache Equivalence 证明 prefill / decode 和 naive generate 的关系；最后用 Serving Optimization Ladder、Chunk Position Continuity 和 paged block trace 说明 quantized matmul、FlashAttention、continuous batching、chunked prefill、paged attention 如何逐步把模型 forward 推向 serving engine。

常见门禁：Toy tiny-LLM Learning Auditor 要能输出 learning units、ready units、concept coverage、experiment coverage、dependency order、serving units、missing concepts / experiments 和 tiny-LLM Learning Gate；不能只说“我按课程读完了 attention、KV cache 和 batching”。

面试表达：tiny-llm 适合解释推理系统能力如何从底层长出来。先讲 attention / RoPE / GQA / RMSNorm / MLP / model load，再讲 generate 和 sampling，然后讲 KV cache 等价性、TTFT / TPOT 指标、quantized matmul、FlashAttention、continuous batching、chunked prefill 和 paged attention。它和 nano-vLLM 互补：tiny-llm 解释底层动机，nano-vLLM 解释 engine 模块边界。

## mini-sglang Learning Audit

一句话定义：mini-sglang Learning Audit 是把 mini-sglang 从普通 serving engine 学习提升到 SGLang-style runtime 学习时，对 Radix Cache、chunked prefill、overlap scheduling、online serving、structured/tool 边界、abort cleanup 和生产差距做门禁验收的方法。

它解决什么问题：很多回答会把 mini-sglang 当作另一个 vLLM clone，只说 scheduler、KV cache 和 benchmark。mini-sglang Learning Audit 要先用 SGLang Runtime Capability 说明 request state、Radix Cache、runtime scheduler、online serving、sampler 和 streaming 的协作；用 Radix Cache Runtime Evidence 证明 longest prefix match、tree split、KV ownership 和 prefix hit metrics；用 Chunked Prefill Ablation 与 Overlap Scheduling Ablation 说明长 prompt suffix chunk、position continuity、partial KV cleanup、CPU schedule next 和 GPU forward current；再用 Online Serving Trace、Structured Tool Boundary 和 Abort Cleanup Check 说明 OpenAI-compatible streaming、client abort、grammar state、valid token mask、tool parser 与应用层工具执行边界；最后用 Production Gap Map 说明 mini-sglang 与完整 SGLang 在 distributed、grammar、speculative、metrics 和 fault recovery 上的差距。

常见门禁：Toy mini-sglang Learning Auditor 要能输出 capability coverage、experiment coverage、evidence count、runtime-specific capability、dependency order、missing capabilities / experiments 和 mini-sglang Learning Gate；不能只说“mini-sglang 有 Radix Cache 和 online serving”。

面试表达：mini-sglang 应该放在 tiny-llm 和 nano-vLLM 之后读。tiny-llm 解释底层优化动机，nano-vLLM 解释 vLLM-like engine 骨架，mini-sglang 解释 SGLang runtime 特色：Radix Cache、chunked prefill、overlap scheduling、online serving、structured generation、tool/agent serving 和完整 SGLang 的能力差距。

## Teaching Project Core Module Audit

一句话定义：Teaching Project Core Module Audit 是从 tiny-llm、nano-vLLM 和 mini-sglang 中抽象自己的 mini serving engine 时，对核心模块、状态 ownership、边界契约、依赖顺序和可替换升级路径做门禁验收的方法。

它解决什么问题：很多项目会把“自研推理框架”写成一张大架构图，但模块边界不清。Teaching Project Core Module Audit 要先用 Core Module Boundary 覆盖 Request、Scheduler、KVCacheManager、BatchBuilder、ModelRunner、Sampler、OutputProcessor、EngineLoop 和 Metrics；用 Request State Ownership、KV Manager Ownership 和 BatchBuilder Metadata Contract 说明 request status、waiting/running、KV blocks、batch metadata、logits、output tokens 和 TTFT / TPOT 由谁维护；再用 Scheduler ModelRunner Boundary、Output Processor Cleanup Boundary 和 Metrics Observability Contract 检查 scheduler 不直接跑模型、model runner 不修改 request 状态、output processor 负责 finish / cleanup、metrics 贯穿全链路；最后用 Replaceable Upgrade Path 说明 naive scheduler、list KV、sync output、single model 后续可以逐个替换。

常见门禁：Toy Core Module Auditor 要能输出 module coverage、state coverage、boundary coverage、ready modules、replaceable modules、engine outputs、missing modules / states 和 Core Module Gate；不能只说“我抽象出了 API、scheduler、KV cache 和 model runner”。

面试表达：最小 serving engine 的重点是模块边界，而不是第一版功能多。第一版先做单机、单模型、offline、多请求、prefill/decode、KV cache 和 metrics；Scheduler 负责决策，BatchBuilder 负责元数据，ModelRunner 负责 forward，Sampler 负责 token，OutputProcessor 负责状态收尾，KV manager 负责显存状态，Metrics 负责证明升级效果。

## Naive Scheduler To Continuous Batching Audit

一句话定义：Naive Scheduler To Continuous Batching Audit 是把教学版 request-level scheduler 升级成 iteration-level continuous batching scheduler 时，对动态加入、动态退出、预算准入和清理路径做门禁验收的方法。

它解决什么问题：很多 mini engine 第一版只有 waiting queue 和固定 running batch，一旦进入 decode，请求集合就被最长输出锁住。Naive Scheduler To Continuous Batching Audit 要先用 Request Level Batch Lock 量化静态 batch 空洞，再用 Iteration Boundary Scheduling 证明每轮 forward 结束后可以重新选择 batch；Dynamic Request Admission 允许 D/E 这类后到请求在 A 未完成时进入 running，Dynamic Request Exit 要求 B/C 结束后移出 running 并释放 KV；Decode First Upgrade 保证已有 streaming 请求的 TPOT 不被长 prompt 打断；Bounded Prefill Budget 和 KV Capacity Admission 则防止长 prompt 或 KV 不足把 scheduler 推向 OOM。

常见门禁：Toy Scheduler Upgrade Auditor 要能输出 finished order、cancelled ids、queue wait steps、TTFT steps、static decode rows、continuous output rows、saved rows、max running、max KV blocks used、deferred reasons 和 trace tail；Scheduler Upgrade Gate 要求 request-level lock、iteration boundary、dynamic join / exit、decode-first、bounded prefill、KV admission 和 cleanup 都可复盘。

面试表达：从 naive scheduler 升级 continuous batching 的关键不是“batch size 变大”，而是把 request lifetime 和 execution batch lifetime 解耦。Scheduler 输出 BatchPlan，BatchBuilder 构造 metadata，OutputProcessor 更新状态并释放 KV；只有 token budget、running 上限、KV 准入和取消清理同时成立，才算真正完成升级。

## Paged KV Cache Upgrade Audit

一句话定义：Paged KV Cache Upgrade Audit 是把教学版 request 私有 list KV cache 升级成全局 block pool、request block table、slot mapping 和幂等 cleanup 的方法。

它解决什么问题：list KV cache 把 KV 状态挂在每个 request 私有对象上，scheduler 很难准确做 KV admission，也难以证明 finished / aborted / failed 后 GPU KV 空间可复用。Paged KV Cache Upgrade Audit 要先用 List KV Private Cache 标出旧抽象的问题，再用 Global KV Block Pool 和 Request Block Table 把逻辑 token 序列映射到非连续 physical blocks；Allocate Until Contract 要求 prefill chunk 和 decode append 前按 target token 数补齐 block；Paged Slot Mapping 证明当前输入 token 写入物理 slot；Decode Block Extension Gate 要覆盖输出跨 block 边界时追加 block；Idempotent KV Free 和 Double Free Guard 则保证 cleanup 可重复但不会把同一 block 放回 free list 两次。

常见门禁：Toy Paged KV Upgrade Auditor 要能输出 A 的 block table 增长、decode extension block、B 释放 blocks、C 复用 blocks、D admission failure、slot mapping、block tables、fragmentation ratio、allocation failures、free blocks after cleanup、idempotent free count 和 trace tail；Paged KV Upgrade Gate 要求全局 block pool、block table、slot mapping、decode extension、KV admission、reuse 和 cleanup 全部可复盘。

面试表达：paged KV cache 的价值不是“换个存储结构”，而是让 KV cache 成为 scheduler 能预算、BatchBuilder 能寻址、OutputProcessor 能释放、metrics 能观测的资源。第一版先跑通 block pool、allocate_until、block table、slot mapping、free 和 reuse，再谈 prefix cache、preemption、swap 或多 worker。

## Prefix Prompt Cache Upgrade Audit

一句话定义：Prefix Prompt Cache Upgrade Audit 是在 paged KV cache 上实现第一版 hash-based prefix cache，并澄清它与平台层 prompt cache、result cache 和 semantic cache 边界的方法。

它解决什么问题：很多实现会把 prompt cache 写成 `cache[prompt]=output`，但 serving runtime 中真正能稳定复用的是已经计算完成的 KV full blocks。Prefix Prompt Cache Upgrade Audit 要先用 Prompt Prefix Cache Boundary 区分平台层缓存与 runtime KV 复用；用 Full Block Cache Reuse 规定只复用 full blocks；用 Parent Hash Cache Chain 证明同样 block tokens 在不同前文后不能共享；用 Extra Hash Cache Isolation 把 model、tokenizer、LoRA、tenant salt 和多模态 hash 纳入 cache key；命中后用 Suffix Prefill Start Position 证明 scheduler 只计算未命中 suffix；Cached Ref Count Lifecycle 要展示 active ref、cached ref、cached-free touch 和 eviction；Full Prompt Hit Fallback 则提醒：KV 命中不等于已经缓存 logits 或最终输出。

常见门禁：Toy Prefix Prompt Cache Auditor 要能输出 A 注册的 cached block table、B 命中 blocks、saved prefill tokens、run prefill tokens、suffix start、suffix positions、suffix slot mapping、touch 后 free queue、ref count attach/free、salt miss、LoRA miss、full prompt hit fallback、cached-free blocks、evicted block、token hit ratio、metrics 和 trace tail；Prefix Prompt Cache Upgrade Gate 要求边界、full block、parent hash、extra hash、suffix prefill、ref count、eviction 和 fallback 都可复盘。

面试表达：prefix cache 的本质不是缓存字符串或答案，而是让新请求的 block table 前缀引用已经计算好的 cached physical blocks。它主要降低 TTFT 和 prefill tokens，但会占用 KV blocks；因此必须用 parent hash、extra hashes、ref count、eviction、水位线、token hit ratio 和 hit/miss TTFT 对比证明收益和风险。

## Preemption Recompute Swap Upgrade Audit

一句话定义：Preemption Recompute Swap Upgrade Audit 是在 paged KV cache 和 prefix cache 之后，为教学版 serving engine 增加 KV 压力兜底、recompute 恢复和 swap 状态机的门禁方法。

它解决什么问题：KV block pool 不足时，scheduler 不能只在 model forward 里 OOM。Preemption Recompute Swap Upgrade Audit 要先用 KV Pressure Trigger 证明 scheduler 在调度前发现 free blocks 不足；用 Cached Block Eviction Before Preemption 规定先淘汰没有 active request 的 cached-free blocks，再动 running request；Victim Selection Policy 要避免抢占高优先级或反复抢占同一请求；Recompute Context Preservation 要证明 preempt 后保留 prompt tokens 和已生成 output tokens；Recompute Resume Path 要用 prompt + output 重建 KV；Swap Failure Rollback 要求 swap in 失败时 CPU block table 不丢；Swap In State Restore 要求恢复后更新 GPU block table 和 RUNNING 状态。

常见门禁：Toy Preemption Upgrade Auditor 要能输出 free before、required blocks、evicted cache blocks、victims、freed blocks、burst blocks、recompute context、resume blocks、preempted count、swap out/in plan、failed swap in 后 CPU table、metrics 和 gates；Preemption Upgrade Gate 要求 pressure、eviction、victim、recompute、swap、metrics 全部可复盘。

面试表达：preemption 是鲁棒性机制，不是性能优化。正常顺序是先降低调度压力和淘汰 cached-free blocks，仍然不够才 preempt active request；recompute 简单但重算 prompt + output，swap 减少重算但引入 CPU memory、带宽和失败回滚复杂度。频繁 preemption 通常意味着容量、`max_num_seqs`、`max_num_batched_tokens`、上下文长度或 prefix cache 策略需要重调。

## Unified Scheduler Loop Audit

一句话定义：Unified Scheduler Loop Audit 是把 continuous batching、paged KV cache、prefix cache、chunked prefill 和 preemption 放进同一个 engine step 验证的调度循环门禁方法。

它解决什么问题：单独实现 scheduler、KV block manager、prefix cache 或 preemption 并不等于 serving engine 能稳定运行。Unified Scheduler Loop Audit 要用 Engine Step Order 规定每轮先清理状态、处理 waiting/running、做 prefix lookup、选 decode/prefill、处理 KV 压力、commit plan、构造 batch metadata、执行模型和更新输出；Prefix Lookup Once 要证明 waiting 请求不会每轮重复增加 ref count；Decode First Commit 要证明 streaming 请求先拿到 decode 预算；Suffix Prefill Plan 要证明 prefix hit 后只计算未命中 suffix；KV Budget Commit 要证明 token budget 和 KV block budget 同时生效；Memory Pressure Order 要证明先 evict cached-free block，再 preempt active request；Batch Metadata Invariant 和 Output State Update 则检查 positions、slot mapping、block table、`num_computed_tokens` 和 output tokens 是否一致。

常见门禁：Toy Unified Scheduler Loop Auditor 要能输出 prefix hits、decode items、prefill items、free before、required new blocks、evicted blocks、preempted requests、commit allocations、metadata positions、slot mapping、block tables、output tokens、computed tokens、free after 和 invariants；Unified Scheduler Loop Gate 要求 prefix、decode、prefill、budget、pressure、commit、metadata、output 和 invariant 全部可复盘。

面试表达：vLLM-like scheduler 的难点不是把请求放进一个 batch，而是在 iteration boundary 上把队列状态、prefix cache 命中、decode 优先、chunked prefill、KV block 分配、cache eviction、preemption、BatchBuilder metadata 和 OutputProcessor 状态更新串成一个一致的事务。只要 `num_computed_tokens`、block table、positions 或 ref count 任意一个错，后面看到的就是乱码输出、重复调度、KV 泄漏或尾延迟异常。

## Serving Benchmark Framework Audit

一句话定义：Serving Benchmark Framework Audit 是把 workload、benchmark config、request trace、engine step trace、summary metrics、regression comparison 和 tuning decision 串成可复现实验闭环的方法。

它解决什么问题：只跑一次 QPS 或只看 tokens/s，不能证明 scheduler、KV cache、prefix cache 或 preemption 的改动真的变好。Serving Benchmark Framework Audit 要用 Benchmark Workload Coverage 覆盖 short、long、shared prefix 和 KV pressure；用 Benchmark Experiment Fingerprint 记录关键参数、seed、commit 和 workload；用 Request Trace Summary 推导 TTFT、TPOT、E2E、success rate 和 prefix hit tokens；用 Engine Step Trace Summary 推导 waiting、active、KV peak、cleanup、eviction 和 preemption；SLO Regression Gate、Throughput Regression Check、KV Cleanup Check、Prefix Effect Check 和 Preemption Risk Check 则共同判断 candidate 是否值得接受。

常见门禁：Toy Serving Benchmark Framework 要能输出 baseline / candidate 指纹、profiles、请求数、成功率、TTFT / TPOT / E2E 分位数、output tokens/s、total tokens/s、prefix hit rate、preemption total、queue p95、active p95、KV peak、cleanup 后 KV、bottleneck classification、request rows、comparison 和 decision；Serving Benchmark Framework Gate 要求 workload、trace、SLO、throughput、KV、cache、preemption、reproducibility 和 decision 全部可复盘。

面试表达：benchmark framework 的价值不是生成一张漂亮表格，而是防止局部作弊。比如 candidate 吞吐提升，必须同时检查 TTFT、TPOT、E2E、失败率、KV peak、cleanup、prefix cache 收益和 preemption 风险；只有实验指纹一致、workload 覆盖真实场景、trace 可回放、trade-off 说清楚，调参结论才可信。

## Async Serving Architecture Audit

一句话定义：Async Serving Architecture Audit 是把同步 engine loop 拆成 API admission、tokenizer queue、engine input queue、engine core、output queue、per-client stream queue 和 cancel cleanup 的异步边界门禁方法。

它解决什么问题：教学版 engine 常把 HTTP 接入、tokenization、scheduler、model forward、detokenization、streaming 和 metrics 写在一个同步 loop 里。高并发下，tokenizer CPU、慢 client、网络发送、日志或 cancel 清理都可能阻塞 GPU 主循环。Async Serving Architecture Audit 要用 Async Boundary Design 划清 API、tokenizer、engine、output worker 和 stream worker 的职责；API Admission Boundary 只做校验、限流和入队；Tokenizer Worker Queue 解耦 tokenization；Engine Input Queue 与 Bounded Engine Drain 控制 engine 每轮接入量；Engine Output Queue 和 Per Client Stream Queue 让 OutputProcessor 不直接写网络；Stream Queue Backpressure 和 Cancel Signal Queue 则让慢 client、断连和重复 cancel 都回到 engine core 做幂等清理。

常见门禁：Toy Async Serving Runtime 要能输出 API admitted / rejected、tokenized ids、first drain added、engine input backlog、finished / cancelled、client stream emitted tokens、队列剩余长度、cleanup events 和 metrics；Async Serving Architecture Gate 要求 admission、tokenizer 解耦、bounded drain、output queue、stream backpressure、cancel signal、engine state ownership、idempotent cleanup、slow client isolation 和 queue drained 全部可复盘。

面试表达：异步 serving 的关键不是用了 `async` 关键字，而是 GPU engine core 独占调度状态，其他组件通过有界队列和消息传递交互。慢 client 只能影响自己的 stream queue，client disconnect 必须进入 cancel queue，cleanup 必须由 engine 释放 KV blocks 和 prefix refs。验证时要同时看 TTFT、TPOT、E2E、queue wait、engine step duration、stream backpressure、cancel cleanup latency 和队列积压，而不是只看吞吐是否变高。

## Multi Worker Router Audit

一句话定义：Multi Worker Router Audit 是把单机异步 serving engine 扩展成多 worker、多 GPU 和 request router 时，用能力过滤、容量估算、sticky locality、load-aware fallback、heartbeat 和 retry 边界验证路由正确性的方法。

它解决什么问题：多起几个 worker 不等于分布式 serving。LLM 请求的 prompt、output、KV block 和 prefix locality 差异很大，round-robin 容易把长请求或热门 tenant 打到同一个 worker，造成 KV 压力、preemption 和尾延迟。Multi Worker Router Audit 要先用 Router Worker Capability Filter 检查 model、adapter 和 worker health；用 Request Cost Block Estimate 估算 KV block；用 Load Aware Worker Score 综合 waiting、running、engine input queue、TTFT、TPOT 和 preemption；用 Sticky Route Fallback 在 prefix cache locality 和负载均衡之间折中；Global Router Admission 要在所有候选 worker 都过载时及时拒绝；Worker Heartbeat Timeout、Safe Retry Boundary 和 Request Worker Map 则约束 worker crash 后未输出请求可重试、已 streaming 请求显式失败。

常见门禁：Toy Multi Worker Router 要能输出 round-robin baseline、load-aware choice、sticky preferred、实际 worker、超大请求 admission、跨模型能力过滤、heartbeat timeout、request state rows、worker selection counts、selection imbalance 和 metrics；Multi Worker Router Gate 要求 capability、admission、load-aware、sticky fallback、health check、retry boundary、state boundary 和 metrics 全部可复盘。

面试表达：router 的职责是 worker 级控制面，不是直接改 worker 内部 scheduler、KV block table 或 prefix cache。一个 TP group 从 router 视角看通常是一个 worker；多个 replica 才是请求级负载均衡目标。多 worker benchmark 必须按 worker 展示 selected_total、queue、KV free、TTFT / TPOT、preemption、failure 和 retry，只看全局 tokens/s 会掩盖热点。

## Distributed Parallel KV Audit

一句话定义：Distributed Parallel KV Audit 是在 tensor parallel、pipeline parallel 和组合 parallel group 中，验证 KV cache 分片、通信成本、pipeline bubble、迁移决策和跨 rank / stage 清理一致性的方法。

它解决什么问题：多 GPU inference 不是把 GPU 数量乘到吞吐上。Tensor parallel 会把权重、attention heads 和 KV shard 切到多个 ranks，上层 router 看到的是一个 parallel group，而不是每张 GPU；Pipeline parallel 会按层切 stage，KV cache 也随 layer ownership 分散。Distributed Parallel KV Audit 要用 Parallel Group Boundary 防止把 TP ranks 当成独立 worker；用 TP Head Shard Assignment 和 TP Block Table Consistency 证明各 rank 的 head shard 不重叠但逻辑 block table 一致；用 PP Layer Ownership 证明每个 stage 只管理自己的层；Distributed KV Byte Accounting 要从 total、TP shard 和 PP stage 三个视角对齐 KV bytes；Collective Communication Cost 与 Pipeline Bubble Ratio 用来解释 TPOT / E2E 变化；KV Migration Decision 则比较远程拉取和 recompute；Cross Rank Cleanup 要求 finish、cancel、preemption 时所有 rank / stage 一起释放。

常见门禁：Toy Distributed Parallel KV Auditor 要能输出 worker unit、TP head ranges、head coverage、logical blocks、TP block table consistency、PP layer ranges、layer coverage、KV total MiB、per TP rank MiB、per PP stage MiB、collective communication report、pipeline bubble ratio、migration decision、cleanup events 和 gates；Distributed Parallel KV Gate 要求 group boundary、TP shard、PP ownership、block table、communication、bubble、migration 和 cleanup 全部可复盘。

面试表达：TP 解决单卡显存和单卡计算瓶颈，但每层 forward 会引入 all-reduce / all-gather 等通信；PP 降低层级显存压力，但在线 decode 容易有 pipeline bubble。KV cache 是 serving 的关键状态，TP 下通常按 head shard 分布，PP 下按 layer stage 分布。运行中的请求迁移很难，因为要保持 layer、head shard、block table、position、dtype、layout 和 sampling 状态一致，实际系统通常优先用 locality、sticky routing 和 worker-local cache，而不是频繁搬迁 KV。

## OpenAI Compatible API Audit

一句话定义：OpenAI Compatible API Audit 是把 serving engine 包装成 OpenAI-compatible HTTP API 时，验证请求契约、streaming frame、错误响应、鉴权、权限、限流、usage 和隐私日志边界的方法。

它解决什么问题：engine 内部有 scheduler、KV cache 和 router，并不等于用户能稳定调用。OpenAI-compatible API 看似只是 JSON 包装，但真实风险在字段语义、streaming 增量、错误码、鉴权、配额和日志。OpenAI Compatible API Audit 要用 API Compatibility Contract 明确支持、忽略和不支持字段；Chat Request Schema Validation 校验 model、messages、role、content、max_tokens 和 temperature；SSE Stream Frame Contract 要求 chunk 可解析、`data:` 帧和 `[DONE]` 结束；OpenAI Style Error Shape 要稳定输出 message、type、param、code 和 request_id；Bearer Auth Gate、Model Permission Gate、Token Rate Limit Gate 与 Concurrency Limit Gate 要在昂贵 tokenization / engine admission 前拦截；Usage Accounting Check 要统计 prompt、completion 和 total tokens；API Privacy Log Check 要防止 API key 和完整 prompt 泄露。

常见门禁：Toy OpenAI Compatible API 要能输出非流式 response、流式 chunk 数、首个 assistant role delta、最后 `[DONE]`、usage、认证错误、不支持字段错误、模型权限错误、context 超限、RPM 限流、并发限流和 metrics；OpenAI Compatible API Gate 要求 auth、schema、model permission、context、rate limit、non-stream、stream、usage、error shape 和 privacy 全部可复盘。

面试表达：OpenAI-compatible 不等于完全复制官方服务，而是诚实实现一个稳定子集。API 层要和 engine 层解耦：先做 auth、schema、model permission、context、rate limit 和 concurrency，再构造内部 EngineRequest 交给 router / engine。streaming 用 SSE 时，HTTP status 一旦开始就不能随意改，client disconnect 要 cancel engine request。rate limit 是租户额度语义，admission control 是系统容量语义，二者错误码和排查方向不同。

## Production Deployment Audit

一句话定义：Production Deployment Audit 是把 LLM serving 从本地可运行推进到生产可发布时，验证镜像 runtime、模型 manifest、startup/readiness、worker 注册、滚动容量、drain、canary、回滚和观测闭环的方法。

它解决什么问题：LLM serving 的 ready 不是进程启动。GPU driver、CUDA runtime、PyTorch、NCCL、模型权重、tokenizer、config、chat template、parallel group、KV cache 和 warmup 任一环节失败，都会在流量进入后变成高延迟、OOM、NCCL hang、质量异常或 5xx。Production Deployment Audit 要用 Runtime Compatibility Matrix 检查 driver、GPU arch、CUDA runtime 和健康端点；用 Model Artifact Manifest 校验权重、tokenizer、config 和 chat template hash；用 Startup Readiness Gate 区分 startup、liveness 和 readiness；用 Worker Registration Gate 保证只有真正 ready 的 worker 才进入 router；用 Rolling Update Capacity Gate 计算升级中的有效容量；用 Drain Completion Gate 保证老 worker 停止接新请求但处理已有 stream；用 Canary Traffic Split、Canary Regression Gate 和 Rollback Readiness Gate 让新 revision 先小流量验证，并保留旧 revision 可快速切回。

常见门禁：Toy Production Deployment Auditor 要能输出 runtime compatibility、artifact manifest、startup phases、probe report、registered workers、rolling capacity before/after drain、drain events、canary routes、canary regression report、rollback readiness、revision 级观测日志、metrics 和 gates；Production Deployment Gate 要求 artifact、readiness、rolling capacity、drain、canary、rollback 和 observability 全部可复盘。

面试表达：生产部署要从交付物、启动、健康检查和发布策略四层讲。交付物层固定镜像 runtime 和模型 manifest；启动层按 GPU/runtime/model/tokenizer/parallel group/KV/warmup 分阶段；健康检查层用 startup 防止慢加载被误杀，用 liveness 看进程和 event loop，用 readiness 表示能接业务流量；发布层先让新 worker ready 再接流量，老 worker drain，canary 观察 error、TTFT、TPOT、OOM、KV pressure、质量指标和 revision 日志，异常时通过 router 直接切回旧 revision。

## Capacity SLO Fault Drill Audit

一句话定义：Capacity SLO Fault Drill Audit 是把 LLM serving 的线上容量、成本、SLO、admission 和故障演练放到同一张可复算表里的方法。

它解决什么问题：LLM serving 容量不能只按 QPS 规划。短问答、RAG 总结和代码生成在 input tokens、output tokens、active tokens 和 KV cache 占用上差异很大，单看请求数会低估 prefill、decode 和并发显存压力。Capacity SLO Fault Drill Audit 要用 Workload Token Profile 统计峰值 RPS、input token/s、output token/s 和 active tokens；用 Benchmark Stable Capacity 把单 worker 极限压测值转换成规划容量；用 GPU Count Estimate 叠加利用率、安全余量、N+1 和发布容量；用 KV Concurrency Limit 防止长上下文把 KV cache 打满；用 Cost Attribution Model 按 GPU 小时、存储、网络、观测和 token 粗分摊；用 SLO Error Budget 把 availability、TTFT、TPOT 和 timeout 变成可计算预算；用 Admission Overload Policy 在容量不足时拒绝或降级；用 Fault Drill Scenario 和 Runbook Coverage Check 验证 worker 丢失、依赖不可用、长上下文突增和新版本退化都有 detect、mitigate、verify。

常见门禁：Toy Capacity SLO Fault Drill Auditor 要能输出 traffic profile、required workers by request / prefill / decode / KV、base / safe / N+1 / release workers、planned GPUs、cost report、SLO report、admission decision、fault drill rows 和 gates；Capacity SLO Fault Drill Gate 要求 workload profile、benchmark capacity、KV capacity、cost、SLO、admission、fault drill 和 runbook 全部可复盘。

面试表达：容量规划先按 workload 拆流量，不要只报 QPS；再用真实分布 benchmark 得到 SLO 约束下的稳定单 worker 容量；worker 数从 request、prefill、decode 和 KV 四个轴取最大值，并加安全余量、N+1、滚动发布和可用区冗余。SLO 要按 workload 定义 TTFT、TPOT、E2E、timeout 和 availability，告警要看 queue、KV pressure、preemption、OOM 和 worker health。故障演练要覆盖 worker、GPU、模型加载、依赖系统、长上下文突增和新版本性能退化，并把 runbook 变成可执行检查项。

## Inference Engine Interview Readiness Audit

一句话定义：Inference Engine Interview Readiness Audit 是把推理框架面试准备从“背名词”转成题目、概念覆盖、debug 路径、项目证据和 trade-off 覆盖率审计的方法。

它解决什么问题：推理框架面试很少只考某个参数名，更多是在追问候选人能否从请求生命周期、KV cache、调度、指标、生产治理和项目证据形成完整解释。Inference Engine Interview Readiness Audit 要用 Interview Question Rubric 定义每类题的必答概念；用 Concept Coverage Score 检查回答是否覆盖 Request Lifecycle Answer、KV Cache Answer Evidence、Scheduler Answer Evidence 和 Production Governance Answer；用 Serving Metrics Coverage 检查 TTFT、TPOT、queue time、KV pressure、preemption、tokens/s 和 error rate；用 Ordered Debug Path 检查排查是否按 scope、reproduce、segment、measure、hypothesize、fix、verify 展开；用 Project Evidence Portfolio 检查是否有 demo、metrics、incident 和 artifact；用 Tradeoff Coverage 检查 latency vs throughput、memory vs batch、quality vs speed 和 cost vs reliability。

常见门禁：Toy Inference Engine Interview Auditor 要能输出 concept rows、metric gate、debug gate、tradeoff gate、project gate、weak questions、revision plan 和 gates；Inference Engine Interview Gate 要求 lifecycle、KV、scheduler、metrics、debug、production、project evidence 和 trade-off 全部可复盘。

面试表达：一个好的推理框架回答要先讲系统边界，再讲请求生命周期，然后落到数据结构和调度策略，最后用指标、debug 路径、生产治理和项目证据收束。说 vLLM、SGLang、PagedAttention 或 prefix cache 时，都要说明它解决什么问题、核心机制、收益、代价、适用 workload、线上指标和故障排查路径。

## Throughput

一句话定义：throughput 是单位时间内系统处理的请求数或 token 数。

常见单位：requests/sec、tokens/sec、output tokens/sec、total tokens/sec。

为什么重要：吞吐决定同样硬件能服务多少用户，也直接影响推理成本。对于批量离线生成，吞吐往往比单请求延迟更重要。

影响因素：模型大小、batch size、输入输出长度、GPU 利用率、KV Cache 显存、调度策略、量化、并行方式和推理框架实现。

注意：吞吐不能脱离延迟讨论。极端增大 batch 可以提升吞吐，但可能让单个用户等待更久。

面试表达：吞吐衡量系统整体产能，线上服务通常要在吞吐、延迟和成本之间做平衡。

## Latency

一句话定义：latency 是单个请求从提交到得到结果所经历的时间。

LLM 中的拆分：常见拆成 TTFT 和 TPOT。TTFT 影响用户看到第一个 token 的速度，TPOT 影响后续流式输出速度。

为什么复杂：同一个模型在不同输入长度、输出长度、并发量、batch 调度策略下，延迟会有很大差异。

线上指标：除了平均延迟，更要关注 P95、P99。大模型服务经常出现长尾请求，平均值好看不代表体验稳定。

面试表达：LLM 延迟不是单个数字，要拆成排队、prefill、decode 和网络传输等部分分析。

## TTFT

一句话定义：TTFT 是 Time To First Token，从请求进入系统到生成第一个 token 的时间。

主要来源：请求排队、tokenization、prompt 拼接、调度等待、prefill 计算和首步 decode。

优化方向：缩短 prompt、缓存公共前缀、优化调度、减少冷启动、提升 prefill 并行效率、使用更小模型或量化模型。

典型场景：对话产品非常重视 TTFT，因为用户会把首 token 速度感知为“系统是否响应”。

面试表达：TTFT 主要受输入长度和 prefill 阶段影响，是交互式 LLM 服务的重要首响指标。

## TPOT

一句话定义：TPOT 是 Time Per Output Token，decode 阶段平均生成每个输出 token 的时间。

主要来源：每步 decode 的模型计算、KV Cache 读取、batch 调度、采样逻辑和流式传输。

优化方向：提高 decode batch 效率、优化 KV Cache 管理、使用量化、使用高效 kernel、减少不必要的长输出。

与用户体验关系：长答案中，TPOT 往往比 TTFT 更影响总生成时间。如果每个 token 都很慢，流式输出会显得卡顿。

面试表达：TPOT 是生成速度指标，主要与输出长度、KV Cache、显存带宽和并发调度有关。

## QPS

一句话定义：QPS 是 Queries Per Second，表示系统每秒处理的请求数。

在 LLM 中的局限：请求之间长度差异很大。一个短分类请求和一个长文生成请求的成本可能相差几十倍，只看 QPS 容易误判容量。

更细指标：tokens/sec、prefill tokens/sec、decode tokens/sec、平均输入长度、平均输出长度、P95 输出长度。

面试表达：LLM 服务不能只看 QPS，token 级吞吐更能反映真实计算成本。

## Batching

一句话定义：batching 是把多个请求合并到同一次模型计算中，以提高硬件利用率。

为什么有效：GPU 适合并行计算。单个请求可能无法充分利用 GPU，合并多个请求可以提高吞吐。

难点：LLM 请求输入和输出长度不一样，生成过程中有的请求先结束，有的请求还在继续。如果 batch 固定不变，会产生 padding 浪费和等待浪费。

面试表达：batching 是提高吞吐的基础，但 LLM 的动态输出让 batching 比普通分类模型更复杂。

## Dynamic Batching

一句话定义：dynamic batching 是在短时间窗口内收集多个请求，组成 batch 后一起推理。

价值：提高 GPU 利用率和整体吞吐，适合并发请求较多的在线服务。

权衡：等待窗口越长，batch 越容易凑大，吞吐越好，但单请求排队延迟也越高。

适用场景：请求到达较密集、延迟要求不是极端严格、需要提高 GPU 利用率的服务。

面试表达：dynamic batching 用少量等待换更高吞吐，核心是控制 batch 聚合窗口。

## Continuous Batching

一句话定义：continuous batching 是 LLM serving 中动态把新请求加入正在运行的 batch，并把已完成请求及时移出的调度方式。

为什么提出：传统 batch 要等一组请求全部结束后才能处理下一组。LLM 输出长度差异大，短请求可能被长请求拖住，GPU 也可能因为 padding 和空槽浪费而利用率下降。

核心机制：每个 decode step 后，系统检查哪些请求完成，释放它们的 batch slot 和 KV Cache；同时把队列中的新请求加入下一步计算。

优点：提高吞吐，降低队头阻塞，更适合流式生成和不同长度请求混合的场景。

代价：调度和 KV Cache 管理更复杂，需要推理引擎支持动态 batch 和高效内存管理。

面试表达：continuous batching 是 LLM serving 的关键优化，因为 LLM decode 是逐 token 迭代，天然适合在每一步动态调整 batch。

## Request Scheduling

一句话定义：request scheduling 是决定哪些请求何时进入 prefill、何时参与 decode、如何分配显存和计算资源的策略。

常见目标：减少排队、提高吞吐、控制 P95/P99 延迟、避免长请求拖垮短请求、保证不同优先级用户的服务质量。

调度挑战：prefill 计算重但可并行，decode 单步轻但持续时间长；长上下文请求占用大量 KV Cache；高并发时显存可能先于算力成为瓶颈。

常见策略：按到达顺序、按优先级、按 token budget、区分 prefill/decode 队列、限制超长请求、对长请求单独调度。

面试表达：LLM 调度要同时考虑计算和 KV Cache 显存，不只是把请求排队先进先出。

## KV Cache Management

一句话定义：KV Cache management 是管理推理过程中每个请求的历史 key/value 缓存的显存分配、复用和释放。

为什么重要：KV Cache 随 batch、层数、head 数、序列长度和精度增长。长上下文和高并发下，KV Cache 可能占掉大部分显存。

核心问题：不同请求长度不同，生成过程中还不断增长。如果按最大长度连续分配，容易造成显存浪费和碎片。

优化方向：分页管理、块式分配、及时释放完成请求、共享公共前缀、KV Cache 量化、限制最大上下文和最大输出长度。

面试表达：LLM serving 的显存瓶颈很大一部分来自 KV Cache，而不是模型权重本身。

## PagedAttention

一句话定义：PagedAttention 是用分页思想管理 KV Cache blocks，减少显存碎片并支持高效 continuous batching 的注意力实现思想。

典型来源：vLLM。

为什么提出：传统连续显存分配要求每个序列的 KV Cache 连续存放，遇到动态长度和高并发时容易浪费显存。PagedAttention 把 KV Cache 切成固定大小 blocks，类似操作系统虚拟内存分页。

核心收益：按需分配 blocks，减少预留浪费；请求结束后可回收 blocks；不同请求可以更灵活地组织 batch。

注意：PagedAttention 主要解决 KV Cache 内存管理和访问效率问题，不改变模型数学语义。

面试表达：PagedAttention 的核心不是新的注意力公式，而是用分页式 KV Cache 管理提升 serving 吞吐和显存利用率。

## Prompt Cache

一句话定义：prompt cache 是缓存可复用 prompt 前缀的计算结果或 KV Cache，以降低重复 prefill 成本。

适用场景：大量请求共享 system prompt、固定业务模板、相同知识库前缀、长上下文对话历史。

收益：减少重复 prefill，降低 TTFT，节省计算成本。

风险：缓存命中依赖前缀完全一致或可稳定识别；不同 tokenizer、chat template、系统提示版本变化都会导致缓存失效；权限隔离不当可能引入数据泄漏风险。

面试表达：prompt cache 优化的是重复前缀的 prefill 成本，特别适合固定系统提示和模板化业务场景。

## Prefix Cache

一句话定义：prefix cache 是对输入前缀的 KV Cache 进行复用的机制，常被视为 prompt cache 的一种具体形式。

核心条件：要复用缓存，前缀 token 序列必须一致。文本看起来相似但 token 不一致，通常不能直接复用。

工程重点：缓存 key 的构造、版本管理、权限隔离、淘汰策略、内存占用控制。

面试表达：prefix cache 的本质是复用公共前缀 KV，减少重复读 prompt 的开销。

## FlashAttention

一句话定义：FlashAttention 是一种 IO-aware 的精确 attention 实现，通过 tiling、online softmax 和 fused kernel 减少 HBM 读写和中间矩阵存储。

它解决什么问题：标准 attention 如果显式物化 `QK^T` 注意力矩阵，会产生大量显存读写，长序列下成本很高。FlashAttention 通过分块计算避免完整存储 attention matrix。

关键点：FlashAttention 不是近似 attention，不改变 softmax attention 的数学结果。它优化的是计算实现和内存访问路径。

适用场景：训练和推理中的长序列 attention，尤其是 attention 受显存带宽和中间矩阵存储限制时。

面试表达：FlashAttention 的核心是 IO-aware，不是减少理论 FLOPs，而是减少 HBM 访问和中间结果物化。

## IO-Aware Optimization

一句话定义：IO-aware optimization 是设计算法时不仅考虑 FLOPs，也考虑显存层级和数据搬运成本。

为什么重要：GPU 算力很强，但从 HBM 读写数据比在 SRAM/register 中复用数据更慢。很多深度学习算子实际瓶颈是 memory bandwidth，而不是算术计算。

在 FlashAttention 中：通过 tiling 把小块数据放到更快的 SRAM 中处理，减少对 HBM 的反复读写。

面试表达：IO-aware 的观点是，大模型性能优化不能只看算力峰值，还要看数据怎么移动。

## Tiling / Blocking

一句话定义：tiling 或 blocking 是把大矩阵计算切成小块，在更快的 SRAM/register 中分块处理。

作用：提高数据复用，减少中间结果写回 HBM，改善 cache locality。

例子：attention 中不一次性计算完整 `QK^T`，而是按 block 读取 Q、K、V，边算边更新结果。

面试表达：tiling 是把大问题拆成适合硬件缓存层级的小块，以减少慢速显存访问。

## Online Softmax

一句话定义：online softmax 是在分块计算中维护最大值和归一化分母，从而得到与完整 softmax 等价结果的方法。

为什么需要：softmax 需要全局归一化。如果 attention 分块计算，就不能简单对每个块单独 softmax 后拼接。

核心思想：逐块更新当前最大值和指数和，保证数值稳定，并能在不物化完整注意力矩阵的情况下得到正确 softmax 输出。

面试表达：online softmax 是 FlashAttention 能边分块边保持精确 softmax 的关键技术之一。

## HBM

一句话定义：HBM 是 High Bandwidth Memory，GPU 上容量较大但相对 SRAM/register 更慢的高带宽显存。

特点：容量大，带宽高，但访问延迟和能耗仍明显高于片上存储。

在大模型中的角色：模型权重、激活、KV Cache 和中间结果主要存放在 HBM 中。

面试表达：HBM 是 GPU 的主显存，大模型推理优化经常围绕减少 HBM 读写展开。

## SRAM

一句话定义：SRAM 是 GPU 片上更快但容量更小的存储，常用于缓存计算中的小块数据。

价值：访问速度快，适合在 kernel 内复用数据。

限制：容量小，不可能容纳完整大矩阵或完整模型参数。

面试表达：FlashAttention 通过把分块数据放入 SRAM 重复使用，减少对 HBM 的依赖。

## Fused Kernel

一句话定义：fused kernel 是把多个计算步骤融合到一个 GPU kernel 中执行，以减少中间结果写回和读出。

为什么有效：多个小算子分开执行会产生 kernel launch 开销和中间 tensor 的 HBM 读写。融合后可以在寄存器或 SRAM 中完成连续计算。

例子：bias、activation、dropout、residual、layer norm 等操作在某些实现中可以融合。

风险：融合 kernel 对硬件、shape 和框架版本更敏感，调试和维护成本更高。

面试表达：fused kernel 通过减少 kernel launch 和中间内存访问提升推理效率。

## Memory Bandwidth Bottleneck

一句话定义：memory bandwidth bottleneck 是性能主要受数据读写速度限制，而不是受算术计算能力限制的瓶颈。

在 LLM decode 中常见：每生成一个 token 都要读取大量权重和 KV Cache，但 batch 较小时计算密度不足，GPU 算力可能吃不满。

优化方向：量化降低数据量，增大 batch 提高复用，优化 KV Cache 布局，使用高效 kernel，减少不必要的内存读写。

面试表达：LLM 推理不总是算力瓶颈，decode 阶段经常是显存带宽瓶颈。

## Quantization

一句话定义：quantization 是用更低精度的数据格式表示模型权重或激活，以降低显存、带宽和部署成本。

为什么有效：参数和 KV Cache 占用的字节数下降后，可以部署更大模型、增大 batch、降低显存带宽压力，并可能提升吞吐。

主要对象：权重量化、激活量化、KV Cache 量化。LLM 中常见的是 weight-only quantization，因为它相对更容易保持质量。

风险：低精度会引入量化误差，可能导致困惑度上升、生成质量下降、数学和代码能力退化、长尾任务不稳定。

面试表达：量化用精度换成本和速度，但必须用校准集和业务评测验证质量损失。

## FP32

一句话定义：FP32 是 32 位浮点格式，精度高但显存和带宽成本大。

特点：数值稳定性好，常用于传统训练基线和部分高精度计算。

在 LLM 中：完整 FP32 训练和推理成本太高，实际大模型通常使用 BF16、FP16 或更低精度。

面试表达：FP32 稳但贵，现代大模型很少全程用 FP32 做主要计算。

## FP16

一句话定义：FP16 是 16 位浮点格式，显存约为 FP32 的一半，常用于训练和推理。

优点：显存和带宽成本低，硬件支持成熟。

风险：动态范围较小，训练中可能出现 overflow 或 underflow，需要 loss scaling 等技巧。

面试表达：FP16 快且省，但数值范围比 BF16 小，训练稳定性要特别关注。

## BF16

一句话定义：BF16 是 Brain Floating Point 16，指数位接近 FP32，动态范围大，常用于稳定训练和推理。

优点：比 FP16 更不容易溢出，很多大模型训练默认使用 BF16。

缺点：尾数精度比 FP16 更低，但大模型训练中通常更看重动态范围。

面试表达：BF16 的核心优势是动态范围接近 FP32，因此大模型训练更稳定。

## FP8

一句话定义：FP8 是 8 位浮点格式，可进一步降低显存和带宽成本，但依赖硬件、框架和校准策略支持。

价值：在支持 FP8 的新硬件上，可用于训练或推理加速。

挑战：数值范围、scale 管理、算子支持和质量评估更复杂。

面试表达：FP8 是更激进的低精度方向，收益大但工程和数值风险也更高。

## INT8

一句话定义：INT8 是 8 位整数量化格式，常用于较稳健的推理压缩。

优点：显存占用和带宽约为 FP16 的一半，质量通常比 INT4 更容易保持。

常见方式：权重量化、权重加激活量化、per-channel scale、per-token activation scale。

风险：异常激活值和分布漂移会影响量化效果。

面试表达：INT8 通常是质量和压缩率之间较稳健的折中。

## INT4

一句话定义：INT4 是 4 位整数量化格式，比 INT8 更省显存，但质量和 kernel 风险更高。

优点：大幅降低权重显存，适合单卡部署更大模型。

风险：量化误差更大，对校准、分组大小、重要权重保护和硬件 kernel 依赖更强。

适用场景：成本敏感、显存紧张、可接受少量质量损失的推理部署。

面试表达：INT4 很适合压缩权重，但不是无损加速，必须看任务质量和 kernel 实现。

## Weight-Only Quantization

一句话定义：weight-only quantization 是只量化模型权重，activation 仍使用较高精度的量化方案。

为什么常见：LLM 推理中权重占用大，decode 阶段频繁读取权重。只量化权重能降低显存和带宽，同时避免激活量化带来的额外误差。

优点：实现相对简单，质量更容易保持，适合大模型推理部署。

缺点：不是所有硬件和 kernel 都能充分利用低比特权重带来的速度收益；有时主要收益是省显存，而不是显著降延迟。

面试表达：weight-only quantization 是 LLM 推理最常见的量化路线之一，核心收益是减少权重显存和带宽。

## Post-Training Quantization / PTQ

一句话定义：PTQ 是模型训练完成后进行量化，不重新训练或只用少量校准数据。

优点：成本低、流程简单、适合已有模型快速部署。

缺点：质量依赖校准数据和量化算法，极低比特量化时容易损失能力。

适用场景：线上部署前压缩模型，快速评估不同精度和不同量化方法的效果。

面试表达：PTQ 的优势是便宜快速，风险是没有充分训练适应量化误差。

## Quantization-Aware Training / QAT

一句话定义：QAT 是训练或微调时模拟量化误差，让模型适应低精度表示。

优点：低比特场景下质量通常优于纯 PTQ，模型能学会抵抗量化噪声。

缺点：需要训练成本，流程更复杂，可能需要原始训练数据或高质量微调数据。

适用场景：对质量要求高、量化比特很低、PTQ 损失不可接受的部署。

面试表达：QAT 用额外训练成本换更好的低精度质量，适合 PTQ 无法满足要求的场景。

## GPTQ

一句话定义：GPTQ 是使用近似二阶信息进行逐层量化误差补偿的 LLM PTQ 方法。

核心思想：量化某些权重会引入误差，GPTQ 利用 Hessian 近似信息在逐列或逐块量化时补偿后续权重，尽量降低输出误差。

优点：在 4bit weight-only 量化中应用广泛，能在较低比特下保持较好质量。

限制：校准数据、分组大小、实现细节和 kernel 支持都会影响最终效果。

面试表达：GPTQ 是典型 PTQ 方法，重点是用近似二阶信息补偿量化误差。

## AWQ

一句话定义：AWQ 是 Activation-aware Weight Quantization，利用激活信息保护重要权重的量化方法。

核心观察：并非所有权重同等重要。对激活影响大的权重更敏感，需要更好保护。

做法直觉：根据校准数据中的激活分布识别重要通道，通过缩放等方式降低重要权重量化误差。

优点：常用于高质量低比特推理部署，尤其是 4bit 权重量化场景。

面试表达：AWQ 的关键词是 activation-aware，即用激活统计判断哪些权重更值得保护。

## Calibration Data

一句话定义：calibration data 是用于估计量化 scale、激活分布或敏感权重的小规模代表性数据。

为什么重要：量化参数来自数据分布。如果校准数据和真实业务分布差异很大，量化后模型可能在线上任务中明显退化。

选择原则：覆盖真实输入长度、语言、格式、领域、任务类型和高频业务模式。

常见坑：只用短问答校准长上下文模型，只用英文校准中文业务，只用通用数据校准专业领域服务。

面试表达：校准数据决定量化是否贴近真实分布，是 PTQ 质量的关键变量。

## Distillation

一句话定义：distillation 是用大模型或强模型作为 teacher，训练较小 student 模型模仿其行为的方法。

在部署中的价值：小模型推理更快、成本更低、延迟更小，适合高频或边缘场景。

常见信号：teacher 的输出文本、logits、偏好排序、推理过程或工具调用轨迹。

风险：student 容量有限，可能继承 teacher 的错误，也可能在复杂推理和长尾场景上能力不足。

面试表达：蒸馏是用训练成本换推理成本，适合把大模型能力迁移到更便宜的小模型上。

## TensorRT-LLM

一句话定义：TensorRT-LLM 是 NVIDIA 面向大语言模型推理优化的高性能部署框架。

常见能力：高效 kernel、量化支持、张量并行、pipeline 并行、KV Cache 优化、in-flight batching 等。

适用场景：NVIDIA GPU 上追求高性能、低延迟和工程可控的生产部署。

注意：框架收益依赖模型结构、硬件型号、batch、输入输出长度和具体配置，不是接入后必然所有场景都变快。

面试表达：TensorRT-LLM 是偏底层性能优化的 LLM 推理框架，重点服务 NVIDIA GPU 生态。

## vLLM

一句话定义：vLLM 是高吞吐 LLM serving 框架，代表性特性是 PagedAttention 和高效 continuous batching。

核心价值：提升显存利用率和在线 serving 吞吐，适合多请求并发和动态输出长度场景。

常见能力：OpenAI-compatible API、continuous batching、prefix caching、多种模型支持、分布式推理支持。

注意：实际性能取决于模型、硬件、并发、上下文长度和采样配置。

面试表达：vLLM 的代表性贡献是用 PagedAttention 管理 KV Cache，从而更好支持高吞吐 serving。

## Triton Inference Server

一句话定义：Triton Inference Server 是 NVIDIA 的通用模型推理服务框架，支持多后端、多模型和动态 batching。

价值：提供模型管理、HTTP/gRPC 接口、批处理、监控指标和多框架后端能力。

在 LLM 中：可以作为部署基础设施的一部分，但 LLM 还需要结合专门的生成推理后端或自定义后端处理 KV Cache 和流式输出。

面试表达：Triton 是通用推理服务框架，LLM 场景还需要关注生成式推理特有的调度和缓存管理。

## Model Parallel Inference

一句话定义：model parallel inference 是把一个模型切分到多张 GPU 上进行推理。

为什么需要：模型权重或 KV Cache 太大，单卡放不下；或者单卡吞吐不足，需要多卡协同。

常见方式：tensor parallel、pipeline parallel、expert parallel。推理中最常见的是 tensor parallel，把矩阵计算切到多卡。

代价：跨卡通信增加，部署复杂度上升，小 batch 或短请求下通信开销可能抵消收益。

面试表达：模型并行推理解决单卡放不下或吞吐不足的问题，但要付出通信和工程复杂度成本。

## CPU Offloading

一句话定义：CPU offloading 是把部分权重、KV Cache 或中间状态放到 CPU 内存，在需要时再搬到 GPU。

价值：缓解 GPU 显存不足，让更大模型或更长上下文在有限显存上运行。

代价：CPU-GPU 传输带宽和延迟远弱于 GPU HBM，频繁 offload 会显著拖慢推理。

适用场景：显存极度受限、低并发、离线任务、成本优先而非低延迟的场景。

面试表达：CPU offloading 用延迟换显存，不是高性能 serving 的默认选择。

## Inference Performance Incident

一句话定义：inference performance incident 是线上推理服务在延迟、吞吐、KV Cache、成本、错误率或质量回归上违反门禁的事故。

常见信号：P95 TTFT 升高、P95 TPOT 升高、P99 端到端延迟升高、KV block 分配失败、cache 命中率低、prompt token 膨胀、取消率上升、量化或调度导致关键业务退化。

排查方法：先把请求 trace 拆成 queue、tokenize、prefill、decode、streaming、postprocess 和客户端接收，再按业务线、输入长度、输出长度、模型版本和调度配置分桶。

面试表达：推理性能事故不能只看 GPU tokens/s，要把 TTFT、TPOT、尾延迟、KV、cache、prompt 成本和质量回归放在同一张审计表里。

## TTFT Regression

一句话定义：TTFT regression 是首 token 延迟相对 baseline 或 SLO 变差。

常见原因：排队时间增加、prompt 变长、prefill token budget 不合理、prefix cache 失效、tokenizer CPU 瓶颈、RAG / rerank / tool call 前置链路变慢、router 把请求打到拥塞实例。

定位方式：比较 `queue_ms`、`tokenize_ms`、`prefill_ms`、`first_chunk_ms` 和 input token 分布，而不是直接判断模型本体慢。

面试表达：TTFT 退化通常先看 queue 和 prefill，再看 prompt 长度、prefix cache、tokenizer 和上游链路。

## TPOT Regression

一句话定义：TPOT regression 是每输出 token 延迟变差，用户会感到流式输出卡顿。

常见原因：decode batch 过大、KV Cache 访存压力高、显存带宽瓶颈、流式链路缓冲、长短请求混跑、speculative decoding 接受率低或 proposal 成本过高。

定位方式：同时比较 worker 生成 token 时间、网关发送 chunk 时间和客户端接收时间，区分模型 decode 慢和传输 / 渲染慢。

面试表达：TPOT 退化不一定是算力不够，也可能是 KV 访存、调度、网络 backpressure 或客户端渲染造成的。

## KV Pressure

一句话定义：KV pressure 是活跃请求的 KV Cache 占用接近或超过可用 KV 显存预算的程度。

粗略口径：`rho_kv = M_kv / B_kv`，其中 `M_kv` 是当前或峰值 KV Cache 显存，`B_kv` 是可用于 KV Cache 的预算。

风险：`rho_kv` 过高时，系统可能出现 block 分配失败、抢占、排队、OOM、长请求拖垮短请求或取消请求释放不及时。

面试表达：KV pressure 是 serving 容量规划里的硬约束，不能只看模型权重是否能放进显存。

## Prompt Cost Drift

一句话定义：prompt cost drift 是新版 system prompt、RAG 上下文、工具 schema 或对话历史让输入 token 相对 baseline 膨胀。

典型指标：`gamma_prompt = new_input_tokens / baseline_input_tokens`。这个比例升高会同时推高 TTFT、KV Cache、token 成本和长上下文干扰风险。

常见治理：压缩 system prompt、按需注入工具 schema、提高 RAG context precision、做多轮摘要、设置 max input tokens 和业务线 token budget。

面试表达：很多推理成本事故不是模型突然变慢，而是 prompt 被产品和工具链不断加长。

## Cache Effectiveness

一句话定义：cache effectiveness 是缓存真实节省 prefill、延迟和成本的程度，而不是单纯命中次数。

关键指标：cache hit rate、命中 token 数、节省 prefill 时间、缓存占用、错误复用率、权限隔离和失效策略。

风险：cache key 过粗可能错误复用，过细则几乎没有命中；企业 RAG 还必须把租户、权限、文档版本和时间纳入缓存治理。

面试表达：缓存是成本优化工具，不是质量保证工具。是否有效要看节省了多少 token 和延迟，以及有没有错误复用。

## Serving Gate

一句话定义：serving gate 是推理服务上线或配置变更前的性能、容量、成本和质量硬门禁。

常见门禁：P95 TTFT、P95 TPOT、P99 E2E latency、P95 queue time、KV pressure、cache effectiveness、prompt cost drift、错误 / 取消率、关键切片质量回归和回滚准备度。

价值：它把“更高 tokens/s”转成可上线判断，避免优化吞吐却牺牲短请求体验、成本和质量。

面试表达：推理 serving gate 应该同时约束用户感知延迟、系统容量、单位成本和质量回归，而不是只看平均 latency。

## Inference Platform Overview Audit

一句话定义：Inference Platform Overview Audit 是把大模型推理平台从“一个模型 HTTP 服务”升级为请求生命周期、SLO、Model Registry、Router、Runtime、Batching、KV cache、扩缩容、发布、安全、成本、观测和开发者体验的可审计系统设计方法。

核心直觉：推理平台和单模型服务的差别，不在于有没有 API Gateway，而在于是否能稳定回答“请求从哪里来、走到哪个模型、排队多久、prefill / decode 各耗时多少、KV cache 是否够、能不能流式返回、出问题如何限流 / 熔断 / 降级 / 回滚、成本归到谁、审计能不能追踪”。vLLM、TGI、Triton、TensorRT-LLM、SGLang 等 runtime 负责高性能执行，平台控制面负责治理、SLO、发布和多租户边界。

常见审计维度：Request Lifecycle Coverage 看 auth、rate limit、input safety、routing、queue、prefill、decode、streaming、output safety、metrics、cost 和 audit 是否全链路覆盖；SLO Metric Contract 看 TTFT、TPOT、E2E、queue、input / output tokens、error、timeout 和 SLO 文档是否明确；Model Registry Readiness 看 model name、version、weight URI、tokenizer、runtime、context limit、quantization、eval report、stage 和 rollback version；Model Router Policy Fit 看租户、模型版本、延迟 SLO、成本预算、负载、fallback 和安全等级；Runtime Prefill Decode Fit、Continuous Batching Readiness 和 KV Cache Capacity Governance 看 runtime 是否区分 prefill / decode、支持动态加入 / 退出请求、管理 KV block、prefix cache、eviction、max context、admission 和指标；Autoscaling Warm Pool Fit、Release Rollback Control、Rate Limit Degrade Protection、Safety Governance Fit、Cost Attribution Optimization、Observability Trace Coverage、Multi Model Governance、Developer API Self Service 和 Inference Platform Gate 则检查扩缩容、灰度回滚、保护机制、安全、成本、trace、多模型策略和开发者自助是否闭环。

常见公式：推理平台审计样本可以写成

```math
u_i=(a_i,r_i,s_i,m_i,p_i,e_i,b_i,k_i,h_i,w_i,g_i,c_i,o_i,z_i)
```

统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(u_i)=1]
```

端到端延迟可以拆成

```math
T_{\mathrm{e2e}}=T_{\mathrm{auth}}+T_{\mathrm{route}}+T_{\mathrm{queue}}+T_{\mathrm{tokenize}}+T_{\mathrm{prefill}}+T_{\mathrm{decode}}+T_{\mathrm{stream}}+T_{\mathrm{safety}}
```

TTFT 与 TPOT 可以写成

```math
T_{\mathrm{ttft}}=T_{\mathrm{auth}}+T_{\mathrm{route}}+T_{\mathrm{queue}}+T_{\mathrm{tokenize}}+T_{\mathrm{prefill}}+T_{\mathrm{first}}
```

```math
T_{\mathrm{tpot}}=\frac{T_{\mathrm{decode}}}{N_{\mathrm{out}}}
```

KV cache 显存可以写成

```math
M_{\mathrm{kv}}=2LBSH_{\mathrm{kv}}D_hB_{\mathrm{elem}}
```

单位成本可以写成

```math
K_{\mathrm{1k}}=\frac{1000K_{\mathrm{total}}}{N_{\mathrm{in}}+N_{\mathrm{out}}}
```

推理平台门禁可以写成

```math
G_{\mathrm{infer}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{ttft,p95}}\le B_{\mathrm{ttft}} \land T_{\mathrm{tpot,p95}}\le B_{\mathrm{tpot}} \land R_{\mathrm{kv}}\le \rho_{\mathrm{kv}} \land R_{\mathrm{err}}\le \rho_{\mathrm{err}} \land P_0=0\right]
```

面试表达：推理平台总览题可以按“需求澄清 -> 请求生命周期 -> Model Registry -> Model Router -> Runtime -> batching / KV cache -> SLO 与扩缩容 -> 灰度回滚 -> 限流熔断降级 -> 安全治理 -> 成本归因 -> 观测 trace -> 多模型治理 -> 开发者自助 -> Inference Platform Gate”展开。不要只回答“用 K8s 部署模型服务”。

## Request Lifecycle Coverage

一句话定义：Request Lifecycle Coverage 是检查推理请求从 auth、rate limit、safety、routing、queue、prefill、decode、streaming 到 metrics、cost、audit 是否有完整阶段记录。

## SLO Metric Contract

一句话定义：SLO Metric Contract 是明确 TTFT、TPOT、E2E、queue time、tokens/s、error rate、timeout rate 和成本等推理平台指标的口径和阈值。

## Model Registry Readiness

一句话定义：Model Registry Readiness 是检查模型名称、版本、权重路径、tokenizer、runtime、上下文长度、量化、评估报告、发布状态和回滚版本是否可追踪。

扩展口径：推理平台接入模型仓库时，不应只读一个权重 URI，还要校验 Model Artifact Registry Audit 中的权重 manifest、safetensors 安全格式、tokenizer / config hash、adapter / merge lineage、量化 eval、runtime smoke test、安全权限、发布审批和 rollback alias。

## Model Router Policy Fit

一句话定义：Model Router Policy Fit 是检查路由是否根据租户、模型版本、延迟 SLO、成本预算、负载、fallback 和安全等级做决策。

## Runtime Prefill Decode Fit

一句话定义：Runtime Prefill Decode Fit 是检查推理 runtime 是否明确区分模型加载、tokenizer、prefill、decode、KV cache、streaming 和 runtime metrics。

## Continuous Batching Readiness

一句话定义：Continuous Batching Readiness 是检查 runtime 是否支持请求动态加入、动态退出、max wait、max batch tokens、优先级队列和长请求隔离。

## KV Cache Capacity Governance

一句话定义：KV Cache Capacity Governance 是把 KV block 管理、eviction、prefix cache、max context、admission control、KV pressure 和指标纳入推理平台容量门禁。

## Autoscaling Warm Pool Fit

一句话定义：Autoscaling Warm Pool Fit 是用 queue、TTFT、KV pressure、tokens/s、冷启动时间、warm pool 和 scale-down 策略共同判断推理扩缩容。

## Release Rollback Control

一句话定义：Release Rollback Control 是让模型发布支持 canary、A/B、traffic split、质量 / 延迟 / 成本护栏和快速回滚。

## Rate Limit Degrade Protection

一句话定义：Rate Limit Degrade Protection 是用租户配额、模型配额、最大输入 / 输出、排队超时、执行超时、熔断和缓存降级保护推理平台。

## Safety Governance Fit

一句话定义：Safety Governance Fit 是把鉴权、输入策略、输出策略、PII 脱敏、日志脱敏、留存策略和审计纳入在线推理治理。

## Cost Attribution Optimization

一句话定义：Cost Attribution Optimization 是按 tenant、model、version、input tokens、output tokens、GPU seconds 和 cache savings 归因推理成本并触发预算告警。

## Observability Trace Coverage

一句话定义：Observability Trace Coverage 是让 metrics、logs、traces、request id、route / queue / prefill / decode / stream spans、dashboard 和 alerts 串成一条排障证据链。

## Multi Model Governance

一句话定义：Multi Model Governance 是用 model、model version、endpoint、deployment、route policy、scaling policy、quota policy、safety policy 和 cost policy 管理多模型平台。

## Developer API Self Service

一句话定义：Developer API Self Service 是提供 OpenAPI、SDK、playground、usage dashboard、SLO dashboard、错误报告和配额申请，降低业务接入推理平台的摩擦。

## Inference Platform Gate

一句话定义：Inference Platform Gate 是判断大模型推理平台是否达到低延迟、高吞吐、高可用、可回滚、可观测、可限流、可审计和可归因最低要求的综合门禁。

典型硬阻断：只有一个 HTTP 服务没有请求生命周期 trace；只看 QPS 不看 TTFT / TPOT / tokens；Model Registry 没有 tokenizer、eval report 或 rollback version；Router 不看租户、SLO、成本和负载；runtime 不区分 prefill / decode；没有 continuous batching；KV cache 没有 admission 和 eviction；扩容只看 GPU utilization；模型发布无灰度和回滚；没有限流、超时、熔断和降级；日志不脱敏；成本不能按租户 / 模型 / 版本归因；只有 worker metrics 没有 request trace；多模型没有 route / quota / safety / cost policy；开发者无法自助查看用量和错误；没有最终推理平台门禁。

## Model Serving Runtime Selection Audit

一句话定义：Model Serving Runtime Selection Audit 是把 Triton、vLLM、TGI、SGLang、TensorRT-LLM 或自研 runtime 放进同一套 profile，按模型、硬件、权重格式、prefill / decode、batching、KV cache、streaming、量化、分布式、指标、benchmark、发布、维护状态、成本和自研门槛做选型审计。

核心直觉：runtime 选型不是排行榜问题。Triton 更偏通用多模型推理服务框架，vLLM / SGLang 更偏 LLM serving engine，TGI 对 Hugging Face 生态和历史部署有价值但要考虑当前维护状态，TensorRT-LLM 更偏 NVIDIA 生态深度优化，自研 runtime 则要求流量规模、团队能力和 fallback 方案同时成立。

常见审计维度：Model Hardware Fit 看模型类型、权重大小、上下文长度、GPU / CPU / 加速器、单机 / 多机拓扑是否匹配；Runtime Boundary Clarity 看平台层治理和 runtime 执行边界是否清楚；Model Format Tokenizer Fit 看权重格式、tokenizer、chat template、stop token 和 generation config 是否绑定；Prefill Decode Scheduler Fit、Continuous Batching Fit 和 KV Cache Memory Manager Fit 看 runtime 是否真正理解生成式推理；Streaming API Fit 看 SSE / gRPC、backpressure、取消和 finish reason；Quantization Accuracy Fit、Distributed Inference Fit、Observability Metrics Fit、Benchmarking Method Fit、Deployment Rollback Fit、Ecosystem Maintenance Fit、Cost Capacity Fit、Self Development Threshold 和 Runtime Selection Gate 则检查质量、分布式、指标、测评、发布、维护、成本、自研和最终上线门禁。

常见公式：runtime 选型样本可以写成

```math
r_i=(m_i,h_i,f_i,t_i,p_i,b_i,k_i,s_i,q_i,d_i,o_i,e_i,c_i,a_i,u_i,z_i)
```

统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(r_i)=1]
```

runtime 内部延迟可以拆成

```math
T_{\mathrm{runtime}}=T_{\mathrm{tokenize}}+T_{\mathrm{queue}}+T_{\mathrm{prefill}}+T_{\mathrm{decode}}+T_{\mathrm{sample}}+T_{\mathrm{stream}}
```

TTFT 与 TPOT 可以写成

```math
T_{\mathrm{ttft}}=T_{\mathrm{tokenize}}+T_{\mathrm{queue}}+T_{\mathrm{prefill}}+T_{\mathrm{first}}
```

```math
T_{\mathrm{tpot}}=\frac{T_{\mathrm{decode}}+T_{\mathrm{sample}}+T_{\mathrm{stream}}}{N_{\mathrm{out}}}
```

单请求 KV cache 显存和 KV 并发上限可以写成

```math
M_{\mathrm{kv,req}}=2LSH_{\mathrm{kv}}D_hB_{\mathrm{elem}}
```

```math
N_{\mathrm{kv}}=\left\lfloor \frac{B_{\mathrm{kv}}}{M_{\mathrm{kv,req}}}\right\rfloor
```

runtime 选型门禁可以写成

```math
G_{\mathrm{runtime}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{ttft,p95}}\le B_{\mathrm{ttft}} \land T_{\mathrm{tpot,p95}}\le B_{\mathrm{tpot}} \land S_{\mathrm{feature}}\ge \tau_{\mathrm{feature}} \land R_{\mathrm{compat}}=1 \land P_0=0\right]
```

面试表达：runtime 选型题可以按“需求澄清 -> 候选 runtime 定位 -> 模型 / 硬件 / tokenizer 兼容 -> prefill / decode -> batching / KV cache -> streaming -> 量化 -> 分布式 -> 指标 -> benchmark -> 发布回滚 -> 维护状态 -> 成本容量 -> 自研门槛 -> Runtime Selection Gate”展开。不要只回答“vLLM 更快”或“Triton 更稳定”。

## Model Hardware Fit

一句话定义：Model Hardware Fit 是检查模型结构、权重大小、上下文长度、目标吞吐、GPU 显存、互联拓扑和 runtime 支持是否匹配。

## Runtime Boundary Clarity

一句话定义：Runtime Boundary Clarity 是区分平台层的鉴权、路由、配额、发布治理与 runtime 内部的模型加载、prefill、decode、KV cache、sampling 和 streaming。

## Model Format Tokenizer Fit

一句话定义：Model Format Tokenizer Fit 是检查权重格式、tokenizer 版本、chat template、special token、stop token 和 generation config 是否随模型版本绑定。

## Prefill Decode Scheduler Fit

一句话定义：Prefill Decode Scheduler Fit 是检查 runtime 是否把 prefill、decode、queue、token budget、长短请求隔离和 phase metrics 纳入调度。

## Continuous Batching Fit

一句话定义：Continuous Batching Fit 是检查 runtime 是否支持请求动态加入、动态退出、max batch tokens、max wait 和长请求隔离，而不是只做静态 batch。

## KV Cache Memory Manager Fit

一句话定义：KV Cache Memory Manager Fit 是检查 KV block、admission control、prefix cache、eviction、KV pressure 和 OOM 保护是否闭环。

## Streaming API Fit

一句话定义：Streaming API Fit 是检查 runtime 的 SSE / gRPC streaming、backpressure、cancel、timeout、finish reason 和 usage 统计是否能支撑线上体验。

## Quantization Accuracy Fit

一句话定义：Quantization Accuracy Fit 是检查 FP8 / INT8 / INT4 / weight-only / KV cache quantization 是否有质量评测、切片回归和回滚阈值。

## Distributed Inference Fit

一句话定义：Distributed Inference Fit 是检查 tensor parallel、pipeline parallel、expert parallel、data parallel serving、PD 分离、拓扑和 rank 健康是否满足模型和 SLO。

## Observability Metrics Fit

一句话定义：Observability Metrics Fit 是检查 TTFT、TPOT、queue、prefill、decode、KV、OOM、error、version 和 per-rank 指标是否可观测。

## Benchmarking Method Fit

一句话定义：Benchmarking Method Fit 是检查 runtime benchmark 是否固定同一模型、硬件、精度、输入输出长度分布、并发、SLO 和 streaming 设置。

## Deployment Rollback Fit

一句话定义：Deployment Rollback Fit 是检查 runtime 镜像 digest、版本 pin、health probe、canary、回滚和兼容性测试是否完整。

## Ecosystem Maintenance Fit

一句话定义：Ecosystem Maintenance Fit 是检查 runtime 的维护状态、release tracking、兼容矩阵、社区或团队 owner 和安全更新责任。

## Cost Capacity Fit

一句话定义：Cost Capacity Fit 是把 GPU-hour、input / output tokens、KV 预算、并发上限、冷启动和每 1k token 成本纳入 runtime 选型。

## Self Development Threshold

一句话定义：Self Development Threshold 是判断自研 runtime 是否真的具备规模收益、系统能力、CUDA / kernel 能力、分布式经验和 fallback runtime。

## Runtime Selection Gate

一句话定义：Runtime Selection Gate 是判断模型服务 runtime 是否达到功能、性能、质量、维护、发布、安全和成本最低上线要求的综合门禁。

典型硬阻断：候选 runtime 不支持目标模型或硬件；平台治理和 runtime 执行边界混乱；tokenizer / chat template 未绑定；不区分 prefill / decode；没有 continuous batching；KV cache 无 admission 和 eviction；streaming 不能取消或 backpressure；量化没有质量回归；多 GPU 拓扑未知；缺 TTFT / TPOT / decode 指标；benchmark 输入输出分布不一致；版本不可回滚；维护状态未知；成本容量未估算；自研理由只有“想更快”；没有最终 Runtime Selection Gate。

## Prefill Decode KV Resource Profile Audit

一句话定义：Prefill Decode KV Resource Profile Audit 是把大模型在线推理请求拆成输入 token、输出 token、TTFT、TPOT、prefill、decode、KV cache、prefix cache、长上下文、多租户、streaming 和成本的可审计资源画像。

核心直觉：推理慢不是一个平均 latency 或 QPS 能解释的问题。长输入主要推高 prefill 和 TTFT，长输出主要拉长 decode 和 TPOT，高并发和长上下文会放大 KV cache 显存压力，prefix cache 可能节省 prefill，但也要受租户隔离、权限和错误复用门禁约束。

常见审计维度：Request Token Profile 看输入 / 输出 token 分布、上下文长度和 token budget；Prefill Phase Accounting、Decode Phase Accounting 和 TTFT TPOT Contract 看 prefill、decode、queue、TTFT、TPOT 和 E2E 是否分开记录；KV Cache Formula Fit 和 KV Capacity Admission 看 KV cache 公式、预算、admission 和 KV pressure 是否正确；Paged Block Management、Prefix Cache Reuse、Long Context Policy 和 Tenant KV Isolation 检查 block 管理、cache 复用、长上下文治理和租户隔离；Continuous Batching Policy、PD Disaggregation Fit、Streaming Backpressure Fit、Observability Phase Metrics、Cost Capacity Model 和 Resource Profile Gate 则检查调度、PD 分离、流式返回、观测、成本容量和最终门禁。

常见公式：资源画像样本可以写成

```math
p_i=(x_i,y_i,c_i,q_i,f_i,d_i,k_i,b_i,r_i,s_i,t_i,o_i,z_i)
```

统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(p_i)=1]
```

TTFT、TPOT 和端到端延迟可以写成

```math
T_{\mathrm{ttft}}=T_{\mathrm{gateway}}+T_{\mathrm{route}}+T_{\mathrm{queue}}+T_{\mathrm{tokenize}}+T_{\mathrm{prefill}}+T_{\mathrm{first}}
```

```math
T_{\mathrm{tpot}}=\frac{T_{\mathrm{decode}}+T_{\mathrm{sample}}+T_{\mathrm{stream}}}{N_{\mathrm{out}}}
```

```math
T_{\mathrm{e2e}}\approx T_{\mathrm{ttft}}+N_{\mathrm{out}}T_{\mathrm{tpot}}
```

KV cache 显存、KV pressure 和 prefix cache 收益可以写成

```math
M_{\mathrm{kv}}=2LBSH_{\mathrm{kv}}D_hB_{\mathrm{elem}}
```

```math
R_{\mathrm{kv}}=\frac{M_{\mathrm{kv}}}{B_{\mathrm{kv}}}
```

```math
R_{\mathrm{prefix}}=\frac{N_{\mathrm{hit\_tokens}}}{N_{\mathrm{input\_tokens}}}
```

资源画像门禁可以写成

```math
G_{\mathrm{resource}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{ttft,p95}}\le B_{\mathrm{ttft}} \land T_{\mathrm{tpot,p95}}\le B_{\mathrm{tpot}} \land R_{\mathrm{kv}}\le \rho_{\mathrm{kv}} \land P_0=0\right]
```

面试表达：Prefill / Decode / KV 资源画像题可以按“token 分布 -> TTFT / TPOT -> prefill -> decode -> KV cache 公式 -> KV pressure -> PagedAttention / block -> prefix cache -> 长上下文 -> 多租户 -> continuous batching -> PD 分离 -> streaming -> observability -> cost -> Resource Profile Gate”展开。不要只说“QPS 高”或“GPU 利用率低”。

## Request Token Profile

一句话定义：Request Token Profile 是记录输入 token、输出 token、上下文长度、token budget、长短请求分布和业务切片的推理请求画像。

## Prefill Phase Accounting

一句话定义：Prefill Phase Accounting 是把 tokenization、queue、prompt prefill、chunked prefill、prefix cache 命中和 prefill tokens/s 单独记录。

## Decode Phase Accounting

一句话定义：Decode Phase Accounting 是把每步 decode、采样、stream flush、output tokens/s、长输出和取消释放单独记录。

## TTFT TPOT Contract

一句话定义：TTFT TPOT Contract 是明确 TTFT、TPOT、E2E、P95 / P99、queue 和 streaming 指标口径、阈值与报警策略。

## KV Cache Formula Fit

一句话定义：KV Cache Formula Fit 是检查 KV cache 显存估算是否包含层数、活跃序列数、上下文长度、KV head、head dim 和 dtype bytes。

## KV Capacity Admission

一句话定义：KV Capacity Admission 是在请求进入 runtime 前按 KV 预算、max context、并发、block 余量和优先级做准入控制。

## Paged Block Management

一句话定义：Paged Block Management 是用固定大小 KV block、block table、free list、碎片指标和回收策略管理非连续 KV cache。

## Prefix Cache Reuse

一句话定义：Prefix Cache Reuse 是复用相同或可共享前缀的 KV cache，以减少重复 prefill、降低 TTFT 和节省成本。

## Long Context Policy

一句话定义：Long Context Policy 是对长 prompt、长文档、RAG 上下文和多轮历史设置 max context、预算、截断 / 压缩、长短队列和专用实例策略。

## Tenant KV Isolation

一句话定义：Tenant KV Isolation 是按租户、权限、模型版本和数据分类隔离 KV cache、prefix cache、配额和成本归因。

## Continuous Batching Policy

一句话定义：Continuous Batching Policy 是定义请求动态加入 / 退出、max batch tokens、max wait、长请求隔离和 decode step 调度的策略。

## PD Disaggregation Fit

一句话定义：PD Disaggregation Fit 是判断 prefill / decode 分离是否值得引入，包括 KV transfer、资源池独立扩缩容、故障边界和复杂度成本。

## Streaming Backpressure Fit

一句话定义：Streaming Backpressure Fit 是检查 SSE / gRPC streaming 的 flush、cancel、客户端背压、timeout 和 usage 统计是否完整。

## Observability Phase Metrics

一句话定义：Observability Phase Metrics 是让 queue、tokenize、prefill、decode、sample、stream、KV、prefix hit、OOM、tenant 和 cost 指标可按请求 trace 关联。

## Cost Capacity Model

一句话定义：Cost Capacity Model 是按 input tokens、output tokens、GPU seconds、KV MiB、prefix savings、冷启动和租户维度估算推理成本容量。

## Resource Profile Gate

一句话定义：Resource Profile Gate 是判断推理资源画像是否足以支撑低延迟、高吞吐、长上下文、多租户、可观测和成本可控上线的综合门禁。

典型硬阻断：只记录 QPS 不记录 input / output tokens；TTFT 和 TPOT 混成一个 latency；prefill / decode 没有 phase metrics；KV cache 显存公式缺 KV head 或 dtype；没有 KV admission；PagedAttention 只知道名字没有 block 指标；prefix cache 不做租户隔离；长上下文无限制；多租户没有 cache 配额；continuous batching 无动态退出；PD 分离不估 KV transfer；streaming 无 backpressure；trace 里缺 prefill / decode span；成本模型只按请求数算；没有 Resource Profile Gate。

## Inference Scheduling Audit

一句话定义：Inference Scheduling Audit 是检查推理调度是否把请求到达分布、continuous batching、token budget、prefill/decode 平衡、KV block、准入控制、长短请求隔离、多租户公平、取消释放、streaming 背压和最终门禁放进同一张审计表。

常见审计维度：Request Arrival Profile 看 input / output token 分布、到达时间、租户和优先级；Continuous Batching Core 看 dynamic join、dynamic exit、iteration-level scheduling 和 active sequence 限制；Token Budget Policy 看 max batch tokens、max prefill chunk 和 decode token budget；Prefill Decode Balance 看 decode-first、chunked prefill、prefill queue 和 decode queue；Paged KV Block Management 看 block size、block table、free list 和碎片指标；Scheduler KV Admission、Long Short Isolation、Tenant Priority Fairness、Cancellation Cleanup、Streaming Backpressure、Scheduler Observability Metrics 和 Scheduler Gate 则检查准入、长短队列、租户公平、取消释放、流式背压、指标和最终上线门禁是否闭环。

面试表达：推理调度题可以按“请求画像 -> dynamic join / exit -> token budget -> prefill/decode phase policy -> PagedAttention block -> KV admission -> 长短请求隔离 -> 多租户公平 -> cancel / timeout cleanup -> streaming backpressure -> metrics -> Scheduler Gate”展开。不要只说“开 continuous batching 就能提升吞吐”。

## Request Arrival Profile

一句话定义：Request Arrival Profile 是记录请求到达时间、租户、优先级、输入 token、最大输出 token、上下文长度和业务切片的调度输入画像。

## Continuous Batching Core

一句话定义：Continuous Batching Core 是支持请求在 decode iteration 之间动态加入、动态退出，并维护 waiting queue、running set 和 finished set 的核心调度能力。

## Token Budget Policy

一句话定义：Token Budget Policy 是用每轮 max batch tokens、max prefill chunk、decode token budget 和 max active sequences 控制吞吐、TTFT、TPOT 和 KV 压力。

## Prefill Decode Balance

一句话定义：Prefill Decode Balance 是在长 prompt prefill 和已有请求 decode 之间分配 GPU 时间，避免长 prefill 拉高 TPOT 或过度偏向 decode 拉高 TTFT。

## Paged KV Block Management

一句话定义：Paged KV Block Management 是用 block table、free list、block size、引用计数和碎片指标支持 KV cache 动态分配、追加、释放和复用。

## Scheduler KV Admission

一句话定义：Scheduler KV Admission 是在请求进入 running set 前按 KV block 预算、水位线、上下文长度、租户配额和优先级做准入控制。

## Long Short Isolation

一句话定义：Long Short Isolation 是把长 prompt、长输出或长上下文请求和短请求分队列、分预算或分实例，降低 head-of-line blocking 和 p99 抖动。

## Tenant Priority Fairness

一句话定义：Tenant Priority Fairness 是按租户配额、优先级、token share、GPU seconds、KV 占用和成本归因控制多租户推理资源分配。

## Cancellation Cleanup

一句话定义：Cancellation Cleanup 是在用户取消、客户端断开、超时或 stop sequence 触发后及时释放 running slot、KV blocks、streaming 连接和指标状态。

## Scheduler Observability Metrics

一句话定义：Scheduler Observability Metrics 是把 queue wait、TTFT、TPOT、scheduled tokens、active sequences、KV blocks、KV waste、rejects、cancellations 和 tenant tokens 纳入可观测闭环。

## Scheduler Gate

一句话定义：Scheduler Gate 是判断推理调度是否满足低 TTFT、低 TPOT、高吞吐、KV 可控、公平性、低拒绝率和无 P0 风险的综合门禁。

## Model Routing Audit

一句话定义：Model Routing Audit 是检查多模型平台是否把请求意图、模型能力、权限、成本、延迟 SLO、质量安全、fallback、灰度、实时健康、trace、降级、配置版本和最终门禁放进同一套路由审计表。

常见审计维度：Request Intent Profile 看任务类型、租户、输入 / 输出 token、工具调用、多模态、SLO、预算和风险等级；Model Capability Profile 看模型任务能力、上下文长度、tool calling、结构化输出、多模态、版本、质量分和安全等级；Permission Candidate Filter 看租户权限、合规区域、模型访问范围和 endpoint 可用性；Cost Budget Policy 和 Latency SLO Policy 看 token 成本、GPU seconds、TTFT、TPOT、排队和总延迟是否先硬过滤；Quality Safety Policy 看任务质量、风险等级、安全拦截和评估回归；Fallback Chain Governance、Canary Routing Stability、Realtime Health Load、Route Trace Coverage、Degrade Policy Control、Route Config Versioning 和 Model Routing Gate 则检查 fallback 预算、稳定灰度、健康负载、trace 完整性、降级边界、配置审计和最终上线门禁。

面试表达：模型路由题可以按“请求画像 -> 模型能力画像 -> 权限和能力过滤 -> 成本预算 -> 延迟 SLO -> 质量安全打分 -> fallback chain -> 灰度分桶 -> 实时健康负载 -> route trace -> 降级策略 -> 配置版本 -> Model Routing Gate”展开。不要只说“按模型名字转发”或“便宜请求用小模型”。

## Request Intent Profile

一句话定义：Request Intent Profile 是路由器对一次请求的结构化画像，覆盖任务类型、租户等级、输入 / 输出 token、上下文长度、工具调用、多模态、延迟 SLO、成本预算、风险等级和是否允许降级。

## Model Capability Profile

一句话定义：Model Capability Profile 是模型注册表中的能力画像，覆盖支持任务、最大上下文、多模态、tool calling、结构化输出、质量评估、安全等级、成本、延迟基线、版本和回滚状态。

## Permission Candidate Filter

一句话定义：Permission Candidate Filter 是在模型打分前先按租户权限、合规区域、模型访问范围、endpoint 健康和业务策略过滤候选，避免把不合规模型靠高分选中。

## Cost Budget Policy

一句话定义：Cost Budget Policy 是根据输入 / 输出 token、模型单价、GPU seconds、cache savings、租户预算和优先级控制一次请求是否可以进入某个模型候选。

## Latency SLO Policy

一句话定义：Latency SLO Policy 是根据 TTFT、TPOT、queue time、route overhead、输出 token 预算和 endpoint 实时负载判断候选模型是否满足请求延迟目标。

## Quality Safety Policy

一句话定义：Quality Safety Policy 是把任务质量评估、安全等级、风险等级、在线反馈、回归指标和拦截策略纳入路由排序和上线门禁。

## Fallback Chain Governance

一句话定义：Fallback Chain Governance 是为主模型失败后的备用模型链设置最大尝试次数、错误类型、总时间预算、总成本预算、幂等控制和熔断规则。

## Canary Routing Stability

一句话定义：Canary Routing Stability 是用稳定哈希、租户分桶、版本权重和快速回滚策略控制灰度模型流量，避免同一用户在新旧版本间抖动。

## Realtime Health Load

一句话定义：Realtime Health Load 是把 endpoint 健康、队列长度、错误率、timeout、TTFT、TPOT、KV pressure、admission reject 和区域状态输入路由决策。

## Route Trace Coverage

一句话定义：Route Trace Coverage 是检查 route trace 是否记录 request id、tenant、task、tokens、selected model、version、route reason、fallback、degrade、cost、latency 和 config version。

## Degrade Policy Control

一句话定义：Degrade Policy Control 是定义哪些业务、租户、风险等级和错误类型允许降级到小模型、缓存、文本-only 或简化流程，并统计降级率和质量影响。

## Route Config Versioning

一句话定义：Route Config Versioning 是让模型路由策略、权重、灰度、fallback、权限和降级规则可版本化、可审计、可 dry-run 和可回滚。

## Model Routing Gate

一句话定义：Model Routing Gate 是判断多模型路由是否满足权限正确、能力匹配、预算可控、SLO 可达、质量安全、fallback 有界、灰度稳定、trace 完整和无 P0 风险的综合门禁。

## Inference Cache System Audit

一句话定义：Inference Cache System Audit 是检查 prompt / prefix cache、KV cache、semantic cache 和 result cache 是否在收益、正确性、租户隔离、一致性、随机性、streaming 和可观测性上形成闭环。

常见审计维度：Request Cache Profile 看请求 hash、模型、版本、tokenizer、参数、租户、权限、安全策略、数据版本、TTL 和 trace 是否完整；Cache Layer Boundary 看 result cache、semantic cache、prompt / prefix cache、KV cache 的命中条件和生命周期是否分清；Key Version Fingerprint 看模型、tokenizer、参数、安全策略和数据版本是否进入 key；Prompt Prefix Reuse 和 KV Lifecycle Management 看前缀 token 复用、KV block、release、retention、eviction 和显存水位；Semantic Similarity Guard 看相似度阈值、任务等价、权限、数据版本和误命中率；Result Cache Determinism、Tenant Permission Isolation、TTL Staleness Control、Eviction Quota Policy、Streaming Cache Policy、Cache Observability Metrics、Cost Latency Savings 和 Cache Governance Gate 则检查确定性、隔离、过期、驱逐、流式返回、指标、收益和最终门禁。

面试表达：缓存体系题可以按“请求画像 -> 缓存层边界 -> key 和版本指纹 -> prompt / prefix cache -> KV 生命周期 -> semantic cache 阈值 -> result cache 确定性 -> 租户权限隔离 -> TTL 与失效 -> 驱逐与配额 -> streaming 策略 -> 指标 -> 成本延迟收益 -> Cache Governance Gate”展开。不要只说“加 Redis 缓存”。

## Request Cache Profile

一句话定义：Request Cache Profile 是把一次缓存查找的模型、版本、tokenizer、参数、prompt hash、prefix hash、租户、权限域、安全策略、数据版本、TTL、采样随机性和 trace 字段结构化。

## Cache Layer Boundary

一句话定义：Cache Layer Boundary 是区分 result cache、semantic cache、prompt / prefix cache、KV cache 和 tokenizer cache 的缓存对象、命中条件、生命周期、收益和风险边界。

## Key Version Fingerprint

一句话定义：Key Version Fingerprint 是把模型版本、tokenizer、生成参数、prompt 模板、安全策略、数据版本、租户和权限域纳入 cache key 或 cache fingerprint，防止跨版本误命中。

## Prompt Prefix Reuse

一句话定义：Prompt Prefix Reuse 是复用相同 system prompt、工具说明、RAG 文档或代码上下文前缀，减少重复 prefill token 和 TTFT。

## KV Lifecycle Management

一句话定义：KV Lifecycle Management 是管理 KV cache 从 prefill 创建、decode 追加、prefix retention、取消释放、超时释放、eviction 到显存水位控制的完整生命周期。

## Semantic Similarity Guard

一句话定义：Semantic Similarity Guard 是语义缓存命中前检查 embedding 相似度、任务等价、租户权限、数据版本、敏感等级和误命中风险。

## Result Cache Determinism

一句话定义：Result Cache Determinism 是要求完整结果缓存只在确定性或近似确定性任务中使用，并把 temperature、top_p、max_tokens、stop sequence 和输出策略纳入 key。

## Tenant Permission Isolation

一句话定义：Tenant Permission Isolation 是确保缓存不能跨租户、跨用户、跨权限域或跨数据分类返回私有上下文、检索结果、KV state 或完整响应。

## TTL Staleness Control

一句话定义：TTL Staleness Control 是用 TTL、数据版本、模型版本、主动失效、purge 和二次校验控制过期缓存命中。

## Eviction Quota Policy

一句话定义：Eviction Quota Policy 是按 LRU、LFU、TTL、成本收益、租户配额、安全等级、模型版本和显存水位驱逐缓存，防止缓存挤占关键资源。

## Streaming Cache Policy

一句话定义：Streaming Cache Policy 是定义流式响应下是否缓存完整结果、是否模拟分片返回、如何处理取消、计费、trace 和 cache-hit 标记。

## Cache Observability Metrics

一句话定义：Cache Observability Metrics 是记录 hit / miss、saved tokens、saved TTFT、saved GPU time、cache latency、eviction、stale hit、semantic false hit、tenant hit rate 和 net cost saved。

## Cost Latency Savings

一句话定义：Cost Latency Savings 是扣除缓存系统自身存储、检索、网络和运维成本后，衡量缓存实际节省的 token、GPU time、TTFT、E2E latency 和成本。

## Cache Governance Gate

一句话定义：Cache Governance Gate 是判断缓存体系是否同时满足收益为正、误命中可控、过期命中可控、租户隔离正确、trace 完整和无 P0 风险的综合门禁。

## Inference Autoscaling Audit

一句话定义：Inference Autoscaling Audit 是检查 LLM 推理扩缩容是否把 token 负载、SLO、队列、GPU / KV、冷启动、warm pool、缩容 draining、路由准入、租户配额、成本和 trace 放进同一套控制闭环。

常见审计维度：Traffic Token Profile 看 QPS、input tokens/s、output tokens/s、active sequences、context tokens 和租户流量；SLO Latency Contract 看 TTFT、TPOT、E2E、p95 / p99、timeout 和错误率目标；Queue Backlog Signal 看 waiting queue、queue wait、pending prefill、pending decode 和 admission reject；GPU KV Capacity Signal 看 GPU utilization、显存、KV pressure、free KV blocks、eviction、OOM 和 tokens/s；Cold Start Lead Time 和 Warm Pool Readiness 看调度、镜像、权重加载、runtime 初始化、预热、endpoint 注册和 warm pool 可用副本；Multi Metric Recommendation、Scale Up Down Policy、Draining Streaming Safety、Router Admission Coupling、Tenant Quota Priority、Cost Budget Guard、Autoscaling Trace Coverage 和 Autoscaling Gate 则检查多指标推荐、扩缩容策略、draining、路由 / 准入联动、租户配额、成本、trace 和最终门禁。

面试表达：自动扩缩容题可以按“流量 token 画像 -> SLO -> 队列 -> GPU / KV -> 冷启动 -> warm pool -> 多指标副本推荐 -> 扩容快缩容慢 -> draining / streaming -> router / admission 联动 -> 租户配额 -> 成本预算 -> autoscaling trace -> Autoscaling Gate”展开。不要只说“用 HPA 看 GPU 利用率”。

## Traffic Token Profile

一句话定义：Traffic Token Profile 是把 QPS 拆成 input tokens/s、output tokens/s、上下文长度、active sequences、长短请求比例、租户流量和优先级画像。

## SLO Latency Contract

一句话定义：SLO Latency Contract 是为 TTFT、TPOT、E2E、queue wait、p95 / p99、timeout rate 和 error rate 定义明确扩缩容目标。

## Queue Backlog Signal

一句话定义：Queue Backlog Signal 是用 waiting queue、queue wait、pending prefill、pending decode、active requests 和 reject count 提前发现容量不足。

## GPU KV Capacity Signal

一句话定义：GPU KV Capacity Signal 是把 GPU utilization、GPU memory、KV pressure、free KV blocks、eviction、OOM、input / output tokens/s 和 per-GPU throughput 作为容量信号。

## Cold Start Lead Time

一句话定义：Cold Start Lead Time 是从触发扩容到新实例真正接流量的总时间，覆盖调度、镜像、权重加载、runtime 初始化、显存分配、kernel 预热和路由注册。

## Warm Pool Readiness

一句话定义：Warm Pool Readiness 是评估常驻副本、预拉镜像、权重本地缓存、预加载模型、预热 kernel 和可快速接流量实例是否足以覆盖冷启动窗口。

## Multi Metric Recommendation

一句话定义：Multi Metric Recommendation 是按 QPS、input tokens/s、output tokens/s、queue wait、TTFT、TPOT、KV pressure、GPU utilization 和错误率取最大需求副本，而不是依赖单一指标。

## Scale Up Down Policy

一句话定义：Scale Up Down Policy 是定义扩容阈值、缩容阈值、滑动窗口、cooldown、最小 / 最大副本、预算上限、预测扩容和异常指标平滑。

## Draining Streaming Safety

一句话定义：Draining Streaming Safety 是缩容前摘除路由、不接新请求、等待 active requests 和 streaming 结束、设置最长等待和 cooldown，避免直接中断用户请求。

## Router Admission Coupling

一句话定义：Router Admission Coupling 是让 autoscaler、router、admission controller、缓存、限流和降级共享容量指标，避免单独扩容或单独转移流量造成故障扩散。

## Tenant Quota Priority

一句话定义：Tenant Quota Priority 是按租户 token/s、队列、SLO、预算、优先级和突发配额决定扩缩容、限流、降级和成本归因。

## Cost Budget Guard

一句话定义：Cost Budget Guard 是在扩容决策中加入 GPU-hour、warm pool 空闲成本、预算上限、租户成本归因和单位成功请求成本。

## Autoscaling Trace Coverage

一句话定义：Autoscaling Trace Coverage 是记录触发指标、观测窗口、当前副本、建议副本、warm pool、冷启动、draining、成本、路由变更和配置版本。

## Autoscaling Gate

一句话定义：Autoscaling Gate 是判断推理扩缩容是否满足 SLO、队列、KV、水位、有效容量、冷启动、成本、租户公平、trace 完整和无 P0 风险的综合门禁。

## Inference Protection Policy Audit

一句话定义：Inference Protection Policy Audit 是检查 LLM 推理平台是否把限流、准入、熔断、重试、超时、降级、租户公平、streaming 清理、成本和 trace 放进同一条请求保护链路。

常见审计维度：Request Token Rate Limit 看请求数、input tokens、output tokens、上下文长度和成本额度是否共同受控；Concurrency Quota Control 看租户、用户、API key、模型和 endpoint 的并发是否有上限；Admission Capacity Guard 看队列、KV pressure、剩余 timeout、runtime 健康和 SLO 是否决定是否接收请求；Circuit Breaker State Machine 看 closed / open / half-open、滑动窗口错误率、慢调用率和探测恢复；Retry Error Classification、Retry Budget Bound 和 Idempotency Stage Guard 看是否只对未开始执行的临时错误重试，并受 retry budget、幂等键和执行阶段约束；Timeout Budget Allocation 和 Degradation Policy Control 看总 deadline 是否分配到 gateway、queue、prefill、decode、streaming 和 fallback；Priority Tenant Fairness、Streaming Cancellation Cleanup、Protection Trace Coverage、Cost Budget Guard 和 Protection Gate 则检查优先级、公平性、取消释放、审计 trace、成本和最终门禁。

面试表达：限流、熔断、重试、超时和降级题可以按“租户 / token / 并发限流 -> admission control -> endpoint 熔断 -> retry 分类和预算 -> timeout budget -> 业务允许的降级 -> streaming / cancel 清理 -> trace / cost / policy version -> Protection Gate”展开。不要只说“失败就重试”或“QPS 高了限流”。

## Request Token Rate Limit

一句话定义：Request Token Rate Limit 是把普通 QPS 限流扩展到 input tokens、output tokens、上下文长度、请求权重和成本额度。

## Concurrency Quota Control

一句话定义：Concurrency Quota Control 是限制租户、用户、API key、模型、endpoint 和任务类型的同时在途请求数，防止长请求占满连接、KV cache 和 GPU slot。

## Admission Capacity Guard

一句话定义：Admission Capacity Guard 是在请求进入 runtime 前检查队列、KV pressure、剩余 timeout、endpoint 健康、租户优先级和 SLO，决定 admit、reject、queue 或 degrade。

## Circuit Breaker State Machine

一句话定义：Circuit Breaker State Machine 是用 closed、open 和 half-open 管理异常 endpoint 或依赖，避免错误率、慢调用、队列爆炸或 OOM 继续扩散。

## Retry Error Classification

一句话定义：Retry Error Classification 是区分 connect reset、尚未开始执行的临时不可达、权限失败、配额失败、安全拒绝、用户取消和已开始 streaming 等错误，决定哪些可以重试。

## Retry Budget Bound

一句话定义：Retry Budget Bound 是限制重试流量占初始请求流量的比例，避免多层重试在过载时形成 retry storm。

## Idempotency Stage Guard

一句话定义：Idempotency Stage Guard 是要求重试必须绑定幂等键、执行阶段、attempt id 和 partial output 状态，避免已经开始生成或产生副作用的请求被盲目重放。

## Timeout Budget Allocation

一句话定义：Timeout Budget Allocation 是把总 deadline 拆到 gateway、queue、prefill、decode、streaming、工具调用、后处理和 buffer，并用剩余预算决定继续、拒绝或降级。

## Degradation Policy Control

一句话定义：Degradation Policy Control 是把小模型、缓存、截断上下文、减少输出、关闭工具调用、异步处理和转人工等降级动作绑定到业务风险、租户权限、质量下限和用户可见说明。

## Priority Tenant Fairness

一句话定义：Priority Tenant Fairness 是在过载时按租户等级、业务 criticality、SLO、预算、配额和历史使用量做公平保护，而不是让单个突发租户拖垮全局。

## Streaming Cancellation Cleanup

一句话定义：Streaming Cancellation Cleanup 是在客户端断开、用户取消、timeout 或降级时释放 KV block、连接、队列项、partial output 状态和 trace，避免资源泄漏。

## Protection Trace Coverage

一句话定义：Protection Trace Coverage 是记录 rate limit、admission、circuit、retry、timeout、degrade、fallback、成本、policy version 和最终 decision reason 的 trace 覆盖率。

## Protection Gate

一句话定义：Protection Gate 是判断推理保护策略是否满足限流正确、准入可靠、熔断隔离、重试有界、超时受控、降级可解释、trace 完整和无 P0 风险的综合门禁。

## Inference Release Governance Audit

一句话定义：Inference Release Governance Audit 是检查模型发布是否把 release bundle、版本包完整性、离线评估、契约测试、稳定分桶、canary、A/B、guardrail、rollback、audit trace 和上线后监控放进同一套发布治理闭环。

常见审计维度：Release Bundle Trace 看模型权重、tokenizer、prompt、runtime config、安全策略、eval report、owner、approver 和 rollback target 是否完整；Model Artifact Integrity 看权重 checksum、镜像 digest、tokenizer checksum 和 artifact lineage；Tokenizer Prompt Runtime Binding 看线上输出是否能追溯到完整版本包；Offline Eval Quality Gate、Safety Eval Gate 和 Contract Test Compatibility 看质量、安全、结构化输出、tool call schema 和下游 parser 是否过线；Stable Bucket Assignment、Canary Ramp Control、AB Experiment Design 和 Sample Ratio Health 看分桶、放量、实验设计和样本比例是否可信；Guardrail Metric Control、Rollback Readiness、Release Audit Trace、Post Release Monitoring 和 Release Governance Gate 则检查安全、延迟、成本、业务护栏、回滚目标、审计和最终门禁。

面试表达：模型发布题可以按“release bundle -> artifact 完整性 -> tokenizer / prompt / runtime 绑定 -> offline eval -> safety eval -> contract test -> stable bucket -> canary ramp -> A/B design -> sample ratio -> guardrail -> rollback -> audit trace -> post-release monitoring -> Release Governance Gate”展开。不要只说“灰度 1% 然后慢慢加流量”。

## Release Bundle Trace

一句话定义：Release Bundle Trace 是把模型权重、tokenizer、prompt、runtime config、安全策略、eval report、推理参数、路由策略、owner、approver 和 rollback target 绑定成可审计发布包。

## Model Artifact Integrity

一句话定义：Model Artifact Integrity 是用 checksum、digest、manifest、lineage 和不可变 URI 证明上线模型、tokenizer、adapter、量化包和镜像没有被替换或篡改。

扩展口径：生产发布时还要和 Artifact Management Audit 对齐，把 deployment package、release manifest、runtime image digest、health check、safety policy、route policy、rollback target 和删除依赖检查一起纳入发布证据。

推理侧关联：线上模型发布和灰度实验也应和 Experiment Tracking Audit 对齐，把 release candidate 的 run id、offline eval run、canary 指标、A/B 配置、prompt / runtime / safety policy 版本、sample failures、成本和回滚证据写入可追踪记录，避免只知道线上模型名而无法复现发布判断。

评估平台关联：发布侧还要和 Evaluation Platform Audit 对齐，把离线批量评测、人评、LLM judge、pairwise win rate、切片回归、在线 guardrail、eval report artifact、release decision 和 rollback evidence 串成证据链。否则 release dashboard 只能看到“已通过评估”，却无法解释通过的是哪个数据版本、哪个指标口径、哪个 judge、人评质量是否稳定、线上分桶是否健康。

## Tokenizer Prompt Runtime Binding

一句话定义：Tokenizer Prompt Runtime Binding 是记录每次线上请求实际使用的 tokenizer、prompt 模板、runtime config、sampling 参数和安全策略版本，避免只知道模型名而无法复现输出。

## Offline Eval Quality Gate

一句话定义：Offline Eval Quality Gate 是候选模型发布前必须通过固定评估集、关键切片、bad case 回归、人工偏好或业务 proxy 指标的离线质量门禁。

## Safety Eval Gate

一句话定义：Safety Eval Gate 是候选模型发布前必须通过安全拒答、越权、隐私、敏感信息、jailbreak、tool misuse 和高风险场景评估的安全门禁。

## Contract Test Compatibility

一句话定义：Contract Test Compatibility 是检查 JSON schema、tool call 参数、结构化输出、错误格式、拒答格式、max context 和下游 parser 是否与生产契约兼容。

## Stable Bucket Assignment

一句话定义：Stable Bucket Assignment 是用稳定 hash 把用户、租户或会话绑定到实验分桶，保证同一实体在同一实验中持续看到同一版本。

## Canary Ramp Control

一句话定义：Canary Ramp Control 是按 1%、5%、10%、25%、50% 等步骤逐步放量，并在每一步用质量、安全、延迟、成本和业务 guardrail 决定 promote、pause、abort 或 rollback。

## AB Experiment Design

一句话定义：AB Experiment Design 是定义 baseline、treatment、随机单位、样本量、主指标、guardrail、停止规则、切片分析和显著性判断的在线实验设计。

## Sample Ratio Health

一句话定义：Sample Ratio Health 是检查 A/B 实验实际样本比例是否接近期望比例，避免分流、日志、过滤或埋点问题污染实验结论。

## Guardrail Metric Control

一句话定义：Guardrail Metric Control 是把安全、p99 延迟、成本、工具失败率、结构化输出解析失败率、投诉率和人工接管率设成发布硬护栏。

## Rollback Readiness

一句话定义：Rollback Readiness 是确认旧模型、prompt、runtime、路由、安全策略和配置仍可快速恢复，并且回滚不会破坏状态或下游契约。

## Release Audit Trace

一句话定义：Release Audit Trace 是记录谁创建版本、谁审批、放量步骤、指标、实验配置、回滚原因、policy version 和最终决策的发布审计链。

## Post Release Monitoring

一句话定义：Post Release Monitoring 是全量后继续监控质量、安全、延迟、成本、业务指标、投诉、bad case 和回归样本，防止灰度窗口外才暴露问题。

## Release Governance Gate

一句话定义：Release Governance Gate 是判断模型发布是否满足版本包完整、离线评估、安全评估、契约测试、分桶实验、guardrail、回滚、trace 和无 P0 风险的综合门禁。

## Inference Platform System Design Audit

一句话定义：Inference Platform System Design Audit 是把一份推理平台系统设计答案放进需求、请求生命周期、SLO、模型注册、路由、runtime、KV、缓存、扩缩容、保护、发布、多租户、安全、观测、成本和 trade-off 的统一审计表。

常见审计维度：Requirement Clarification 看模型类型、QPS、input / output token 分布、上下文长度、TTFT、TPOT、p99、streaming、多租户、安全、成本和可用性是否澄清；SLO Token Contract 看答案是否用 TTFT、TPOT、E2E、token load 和容量预算约束设计；Model Registry Release、Router Policy Fit、Runtime Prefill Decode 和 KV Scheduler Capacity 看模型版本、路由、prefill / decode、continuous batching、KV cache 和调度是否连成请求链路；Cache Isolation Strategy、Autoscaling Capacity Plan、Protection Policy Design、Release Governance Design、Tenant Security Governance、Observability Trace Readiness、Cost Capacity Governance 和 Tradeoff Boundary Reasoning 则检查缓存隔离、容量、稳定性、发布、多租户、安全、trace、成本和工程取舍是否闭环。

面试表达：推理平台系统设计题可以按“需求澄清 -> 目标与 SLO -> 总体架构 -> 请求链路 -> Model Registry / Router -> Cache -> Runtime / scheduler / KV -> autoscaling -> protection -> release -> tenant / security -> observability -> cost -> bottleneck and trade-off -> Inference Platform Design Gate”展开。不要只回答“API Gateway + K8s + vLLM”。

## SLO Token Contract

一句话定义：SLO Token Contract 是把 TTFT、TPOT、p99、input tokens/s、output tokens/s、active sequences、上下文长度、timeout 和成本预算写成推理平台设计的硬约束。

## KV Scheduler Capacity

一句话定义：KV Scheduler Capacity 是用活跃请求数、上下文 token、层数、KV heads、head dimension、元素字节数、KV 显存水位和调度策略判断平台是否接得住当前并发。

## Inference Platform Design Gate

一句话定义：Inference Platform Design Gate 是判断推理平台系统设计是否满足覆盖率、SLO、KV 容量、trace、发布回滚、多租户安全、成本归因和无 P0 风险的综合门禁。

## Inference Fault Diagnosis Audit

一句话定义：Inference Fault Diagnosis Audit 是把 TTFT、TPOT、p99、吞吐、错误率、timeout、KV cache、模型加载、streaming、cache、route、tool、质量和发布 diff 放进同一张推理故障排查表的方法。

核心直觉：推理故障是在线用户直接感知的问题，不能只看 QPS 或 GPU utilization。成熟平台要能从 request profile、scope slice、阶段 trace、KV cache、model runtime、route、cache、error taxonomy、tool trace、release diff 和应急动作串出根因。

常见审计维度：Inference Fault Evidence Coverage 检查证据是否齐全；Request Scope Slice 和 Trace Stage Coverage 检查 model、tenant、region、endpoint、runtime、streaming、输入 / 输出长度、版本和 gateway / router / queue / prefill / decode / stream span；TTFT Decomposition、TPOT Decode Health、Tail Latency Attribution 和 Throughput Token Capacity 检查首响、生成速度、尾延迟和 token 产能；Error Taxonomy Timeout Stage、KV Cache Pressure Guard、Model Artifact Load Readiness、Streaming Reliability、Cache Hit Key Governance、Route Trace Correctness、Tool Dependency Trace、Quality Release Diff Readiness 和 Inference Fault Diagnosis Gate 则检查错误、容量、加载、streaming、cache、路由、工具、质量、回滚和复盘。

面试表达：推理故障定位题要先判断影响范围，再拆请求链路；TTFT 看 gateway、router、cache、queue、tokenize、prefill 和首 token；TPOT 看 decode、active sequences、KV cache、显存带宽和 runtime；p99 看慢请求 trace 和切片；错误率先分类，再决定限流、降级、回滚、摘除实例或扩容。

## Inference Fault Evidence Coverage

一句话定义：Inference Fault Evidence Coverage 衡量请求画像、范围切片、阶段 trace、KV、模型 runtime、路由、cache、错误、工具、发布 diff、应急动作和诊断是否齐全。

## Request Scope Slice

一句话定义：Request Scope Slice 是按 model、tenant、region、endpoint、runtime、streaming、input length bucket、output length bucket 和 release version 切分推理故障影响范围。

## Trace Stage Coverage

一句话定义：Trace Stage Coverage 是检查 gateway、auth、router、cache、queue、tokenize、prefill、first decode、decode、stream、safety 和 downstream 是否都有 span。

## TTFT Decomposition

一句话定义：TTFT Decomposition 是把首 token 延迟拆成前置服务、cache、queue、tokenize、prefill、first decode 和 first flush，避免把所有首响变慢都归因于模型。

## TPOT Decode Health

一句话定义：TPOT Decode Health 是用 decode time、output tokens、active sequences、KV cache、GPU 负载、sampling 和 runtime 版本解释每 token 生成速度。

## Tail Latency Attribution

一句话定义：Tail Latency Attribution 是用 p99 / p50、慢请求 trace、长上下文、租户、endpoint、instance、tool call 和网络切片定位尾延迟。

## Throughput Token Capacity

一句话定义：Throughput Token Capacity 是同时检查 QPS、input tokens/s、output tokens/s、per GPU output tokens/s 和有效吞吐，而不是只看请求数。

## Error Taxonomy Timeout Stage

一句话定义：Error Taxonomy Timeout Stage 是把错误分成 4xx、429、5xx、timeout、OOM、model load、tool failure、safety block，并要求 timeout 记录发生阶段。

## KV Cache Pressure Guard

一句话定义：KV Cache Pressure Guard 是用 KV pressure、free blocks、eviction、admission reject、fragmentation 和 active sequences 判断 KV 是否成为瓶颈。

## Model Artifact Load Readiness

一句话定义：Model Artifact Load Readiness 是检查 manifest、weight checksum、tokenizer、config、runtime compatibility、quantization compatibility、权限和 warmup 是否完整。

## Streaming Reliability

一句话定义：Streaming Reliability 是观测 stream open、server abort、client disconnect、idle timeout、tokens before abort 和 cancellation cleanup。

## Cache Hit Key Governance

一句话定义：Cache Hit Key Governance 是检查 cache hit rate、cache key 是否绑定 model、tokenizer、prompt、安全策略、tenant、permission 和 TTL / eviction 策略。

## Route Trace Correctness

一句话定义：Route Trace Correctness 是检查候选模型、选择模型、路由原因、灰度比例、租户策略、endpoint 健康、fallback chain 和 trace 是否完整。

## Tool Dependency Trace

一句话定义：Tool Dependency Trace 是检查 tool schema、权限决策、timeout、retry、下游状态、错误消息和 tool call trace 是否完整。

## Quality Release Diff Readiness

一句话定义：Quality Release Diff Readiness 是把模型、prompt、runtime、route、cache、safety、sampling diff 和质量指标、线上反馈、回滚证据关联起来。

## Inference Fault Diagnosis Gate

一句话定义：Inference Fault Diagnosis Gate 是判断推理故障定位是否有完整证据、owner、runbook、应急动作、回滚、postmortem 和无 P0 风险的综合门禁。

典型硬阻断：只看 QPS 和 GPU utilization；没有 request trace；TTFT 不拆 queue / prefill；TPOT 不看 decode 和 KV；p99 没有慢请求样本；吞吐只看请求数；错误没有分类；timeout 没有阶段；KV pressure 不可见；模型加载不校验 checksum / tokenizer / runtime；streaming abort 不区分客户端和服务端；cache key 不绑定版本；route trace 缺失；tool 失败被归因成模型差；质量下降没有 release diff；没有最终 Inference Fault Diagnosis Gate。

## 部署排障常见问题

一句话定义：部署排障是从质量、延迟、吞吐、显存和稳定性多个维度定位线上推理异常。

常见质量问题：tokenizer 不一致、chat template 错误、EOS 配置错误、量化退化、采样参数不合理、prompt 被截断。

常见性能问题：batch 太小导致 GPU 利用率低，batch 太大导致排队延迟高，KV Cache 占满显存，长请求阻塞短请求，prompt cache 命中率低。

常见稳定性问题：OOM、显存碎片、请求超时、流式连接断开、不同版本模型和 tokenizer 混用。

面试表达：LLM 部署排障要先区分质量问题和系统问题，再沿 tokenizer、模板、解码、KV Cache、调度和硬件指标逐层排查。

## 本章小结

本章围绕 LLM 推理优化和部署展开。第 8 章解释了推理和解码本身，本章进一步回答：如何让模型在真实服务中更快、更省、更稳。

高频结论如下：

1. LLM serving 的难点来自动态输入输出、prefill/decode 分离、KV Cache 和高并发调度。
2. throughput 衡量整体产能，latency 衡量单请求体验，线上要同时看平均值和 P95/P99。
3. TTFT 主要受排队和 prefill 影响，TPOT 主要受 decode、KV Cache 和显存带宽影响。
4. dynamic batching 用等待窗口换吞吐，continuous batching 在 decode 过程中动态加入和移出请求。
5. KV Cache 是推理显存大户，PagedAttention 用分页式管理减少浪费并支持高吞吐 serving。
6. prompt cache 和 prefix cache 复用公共前缀，主要优化重复 prefill 成本。
7. FlashAttention 是精确 attention 的 IO-aware 实现，核心是减少 HBM 读写和中间矩阵物化。
8. decode 阶段经常是 memory bandwidth bottleneck，不只是算力瓶颈。
9. 量化能降低显存和带宽成本，但必须用校准数据和业务评测验证质量损失。
10. GPTQ 和 AWQ 都是常见 LLM PTQ 方法，前者强调近似二阶误差补偿，后者强调 activation-aware 保护重要权重。
11. vLLM、SGLang、TensorRT-LLM、TGI、Triton 等框架各有定位，选型要结合硬件、模型、并发、维护状态和延迟目标。
12. 生产排障要同时检查 tokenizer、chat template、EOS、解码参数、量化、KV Cache、调度和硬件指标。
13. 推理性能事故要把 TTFT regression、TPOT regression、KV pressure、prompt cost drift、cache effectiveness 和 serving gate 放在同一张表里判断。
14. runtime 选型要用 Model Serving Runtime Selection Audit 统一比较候选，而不是只看框架名、单点 tokens/s 或历史经验。
15. Prefill / Decode / KV 资源画像要把 token 分布、phase latency、KV 显存、prefix cache、长上下文、多租户、streaming、observability 和成本放进同一套上线门禁。
