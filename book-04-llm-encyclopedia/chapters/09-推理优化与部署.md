## LLM Serving

一句话定义：LLM serving 是把训练好的大模型部署成可被线上请求调用的推理服务。

它解决什么问题：训练完成的模型只是参数文件。要让用户稳定使用，还需要 tokenizer、chat template、请求队列、batch 调度、KV Cache 管理、解码策略、流式输出、监控告警、限流、容灾和成本控制。

与普通深度学习 serving 的区别：LLM 请求长度差异大，输出长度不可预知，推理过程分为 prefill 和 decode 两个阶段，decode 还存在逐 token 串行依赖。因此，LLM serving 的核心瓶颈经常不是一次 forward，而是调度、KV Cache 和显存带宽。

常见目标：低延迟、高吞吐、稳定质量、低成本、高可用、可观测、可扩展。

面试表达：LLM serving 不是简单把模型 `model.generate()` 包成接口，而是围绕长序列、动态输出和 KV Cache 做系统级优化。

## Throughput

一句话定义：throughput 是单位时间内系统处理的请求数或 token 数。

常见单位：requests/sec、tokens/sec、output tokens/sec、total tokens/sec。

为什么重要：吞吐决定同样硬件能服务多少用户，也直接影响推理成本。对于批量离线生成，吞吐往往比单请求延迟更重要。

影响因素：模型大小、batch size、输入输出长度、GPU 利用率、KV Cache 显存、调度策略、量化、并行方式和推理框架实现。

注意：吞吐不能脱离延迟讨论。极端增大 batch 可以提升吞吐，但可能让单个用户等待更久。

面试表达：吞吐衡量系统整体产能，线上服务通常要在吞吐、延迟和成本之间做平衡。

## Latency

一句话定义：latency 是单个请求从提交到得到结果所经历的时间。

LLM 中的拆分：常见拆成 TTFT 和 TPOT。TTFT 影响用户看到第一个 token 的速度，TPOT 影响后续流式输出速度。

为什么复杂：同一个模型在不同输入长度、输出长度、并发量、batch 调度策略下，延迟会有很大差异。

线上指标：除了平均延迟，更要关注 P95、P99。大模型服务经常出现长尾请求，平均值好看不代表体验稳定。

面试表达：LLM 延迟不是单个数字，要拆成排队、prefill、decode 和网络传输等部分分析。

## TTFT

一句话定义：TTFT 是 Time To First Token，从请求进入系统到生成第一个 token 的时间。

主要来源：请求排队、tokenization、prompt 拼接、调度等待、prefill 计算和首步 decode。

优化方向：缩短 prompt、缓存公共前缀、优化调度、减少冷启动、提升 prefill 并行效率、使用更小模型或量化模型。

典型场景：对话产品非常重视 TTFT，因为用户会把首 token 速度感知为“系统是否响应”。

面试表达：TTFT 主要受输入长度和 prefill 阶段影响，是交互式 LLM 服务的重要首响指标。

## TPOT

一句话定义：TPOT 是 Time Per Output Token，decode 阶段平均生成每个输出 token 的时间。

主要来源：每步 decode 的模型计算、KV Cache 读取、batch 调度、采样逻辑和流式传输。

优化方向：提高 decode batch 效率、优化 KV Cache 管理、使用量化、使用高效 kernel、减少不必要的长输出。

与用户体验关系：长答案中，TPOT 往往比 TTFT 更影响总生成时间。如果每个 token 都很慢，流式输出会显得卡顿。

面试表达：TPOT 是生成速度指标，主要与输出长度、KV Cache、显存带宽和并发调度有关。

## QPS

一句话定义：QPS 是 Queries Per Second，表示系统每秒处理的请求数。

在 LLM 中的局限：请求之间长度差异很大。一个短分类请求和一个长文生成请求的成本可能相差几十倍，只看 QPS 容易误判容量。

更细指标：tokens/sec、prefill tokens/sec、decode tokens/sec、平均输入长度、平均输出长度、P95 输出长度。

面试表达：LLM 服务不能只看 QPS，token 级吞吐更能反映真实计算成本。

## Batching

一句话定义：batching 是把多个请求合并到同一次模型计算中，以提高硬件利用率。

为什么有效：GPU 适合并行计算。单个请求可能无法充分利用 GPU，合并多个请求可以提高吞吐。

难点：LLM 请求输入和输出长度不一样，生成过程中有的请求先结束，有的请求还在继续。如果 batch 固定不变，会产生 padding 浪费和等待浪费。

面试表达：batching 是提高吞吐的基础，但 LLM 的动态输出让 batching 比普通分类模型更复杂。

## Dynamic Batching

一句话定义：dynamic batching 是在短时间窗口内收集多个请求，组成 batch 后一起推理。

价值：提高 GPU 利用率和整体吞吐，适合并发请求较多的在线服务。

权衡：等待窗口越长，batch 越容易凑大，吞吐越好，但单请求排队延迟也越高。

适用场景：请求到达较密集、延迟要求不是极端严格、需要提高 GPU 利用率的服务。

面试表达：dynamic batching 用少量等待换更高吞吐，核心是控制 batch 聚合窗口。

## Continuous Batching

一句话定义：continuous batching 是 LLM serving 中动态把新请求加入正在运行的 batch，并把已完成请求及时移出的调度方式。

为什么提出：传统 batch 要等一组请求全部结束后才能处理下一组。LLM 输出长度差异大，短请求可能被长请求拖住，GPU 也可能因为 padding 和空槽浪费而利用率下降。

核心机制：每个 decode step 后，系统检查哪些请求完成，释放它们的 batch slot 和 KV Cache；同时把队列中的新请求加入下一步计算。

优点：提高吞吐，降低队头阻塞，更适合流式生成和不同长度请求混合的场景。

代价：调度和 KV Cache 管理更复杂，需要推理引擎支持动态 batch 和高效内存管理。

面试表达：continuous batching 是 LLM serving 的关键优化，因为 LLM decode 是逐 token 迭代，天然适合在每一步动态调整 batch。

## Request Scheduling

一句话定义：request scheduling 是决定哪些请求何时进入 prefill、何时参与 decode、如何分配显存和计算资源的策略。

常见目标：减少排队、提高吞吐、控制 P95/P99 延迟、避免长请求拖垮短请求、保证不同优先级用户的服务质量。

调度挑战：prefill 计算重但可并行，decode 单步轻但持续时间长；长上下文请求占用大量 KV Cache；高并发时显存可能先于算力成为瓶颈。

常见策略：按到达顺序、按优先级、按 token budget、区分 prefill/decode 队列、限制超长请求、对长请求单独调度。

面试表达：LLM 调度要同时考虑计算和 KV Cache 显存，不只是把请求排队先进先出。

## KV Cache Management

一句话定义：KV Cache management 是管理推理过程中每个请求的历史 key/value 缓存的显存分配、复用和释放。

为什么重要：KV Cache 随 batch、层数、head 数、序列长度和精度增长。长上下文和高并发下，KV Cache 可能占掉大部分显存。

核心问题：不同请求长度不同，生成过程中还不断增长。如果按最大长度连续分配，容易造成显存浪费和碎片。

优化方向：分页管理、块式分配、及时释放完成请求、共享公共前缀、KV Cache 量化、限制最大上下文和最大输出长度。

面试表达：LLM serving 的显存瓶颈很大一部分来自 KV Cache，而不是模型权重本身。

## PagedAttention

一句话定义：PagedAttention 是用分页思想管理 KV Cache blocks，减少显存碎片并支持高效 continuous batching 的注意力实现思想。

典型来源：vLLM。

为什么提出：传统连续显存分配要求每个序列的 KV Cache 连续存放，遇到动态长度和高并发时容易浪费显存。PagedAttention 把 KV Cache 切成固定大小 blocks，类似操作系统虚拟内存分页。

核心收益：按需分配 blocks，减少预留浪费；请求结束后可回收 blocks；不同请求可以更灵活地组织 batch。

注意：PagedAttention 主要解决 KV Cache 内存管理和访问效率问题，不改变模型数学语义。

面试表达：PagedAttention 的核心不是新的注意力公式，而是用分页式 KV Cache 管理提升 serving 吞吐和显存利用率。

## Prompt Cache

一句话定义：prompt cache 是缓存可复用 prompt 前缀的计算结果或 KV Cache，以降低重复 prefill 成本。

适用场景：大量请求共享 system prompt、固定业务模板、相同知识库前缀、长上下文对话历史。

收益：减少重复 prefill，降低 TTFT，节省计算成本。

风险：缓存命中依赖前缀完全一致或可稳定识别；不同 tokenizer、chat template、系统提示版本变化都会导致缓存失效；权限隔离不当可能引入数据泄漏风险。

面试表达：prompt cache 优化的是重复前缀的 prefill 成本，特别适合固定系统提示和模板化业务场景。

## Prefix Cache

一句话定义：prefix cache 是对输入前缀的 KV Cache 进行复用的机制，常被视为 prompt cache 的一种具体形式。

核心条件：要复用缓存，前缀 token 序列必须一致。文本看起来相似但 token 不一致，通常不能直接复用。

工程重点：缓存 key 的构造、版本管理、权限隔离、淘汰策略、内存占用控制。

面试表达：prefix cache 的本质是复用公共前缀 KV，减少重复读 prompt 的开销。

## FlashAttention

一句话定义：FlashAttention 是一种 IO-aware 的精确 attention 实现，通过 tiling、online softmax 和 fused kernel 减少 HBM 读写和中间矩阵存储。

它解决什么问题：标准 attention 如果显式物化 `QK^T` 注意力矩阵，会产生大量显存读写，长序列下成本很高。FlashAttention 通过分块计算避免完整存储 attention matrix。

关键点：FlashAttention 不是近似 attention，不改变 softmax attention 的数学结果。它优化的是计算实现和内存访问路径。

适用场景：训练和推理中的长序列 attention，尤其是 attention 受显存带宽和中间矩阵存储限制时。

面试表达：FlashAttention 的核心是 IO-aware，不是减少理论 FLOPs，而是减少 HBM 访问和中间结果物化。

## IO-Aware Optimization

一句话定义：IO-aware optimization 是设计算法时不仅考虑 FLOPs，也考虑显存层级和数据搬运成本。

为什么重要：GPU 算力很强，但从 HBM 读写数据比在 SRAM/register 中复用数据更慢。很多深度学习算子实际瓶颈是 memory bandwidth，而不是算术计算。

在 FlashAttention 中：通过 tiling 把小块数据放到更快的 SRAM 中处理，减少对 HBM 的反复读写。

面试表达：IO-aware 的观点是，大模型性能优化不能只看算力峰值，还要看数据怎么移动。

## Tiling / Blocking

一句话定义：tiling 或 blocking 是把大矩阵计算切成小块，在更快的 SRAM/register 中分块处理。

作用：提高数据复用，减少中间结果写回 HBM，改善 cache locality。

例子：attention 中不一次性计算完整 `QK^T`，而是按 block 读取 Q、K、V，边算边更新结果。

面试表达：tiling 是把大问题拆成适合硬件缓存层级的小块，以减少慢速显存访问。

## Online Softmax

一句话定义：online softmax 是在分块计算中维护最大值和归一化分母，从而得到与完整 softmax 等价结果的方法。

为什么需要：softmax 需要全局归一化。如果 attention 分块计算，就不能简单对每个块单独 softmax 后拼接。

核心思想：逐块更新当前最大值和指数和，保证数值稳定，并能在不物化完整注意力矩阵的情况下得到正确 softmax 输出。

面试表达：online softmax 是 FlashAttention 能边分块边保持精确 softmax 的关键技术之一。

## HBM

一句话定义：HBM 是 High Bandwidth Memory，GPU 上容量较大但相对 SRAM/register 更慢的高带宽显存。

特点：容量大，带宽高，但访问延迟和能耗仍明显高于片上存储。

在大模型中的角色：模型权重、激活、KV Cache 和中间结果主要存放在 HBM 中。

面试表达：HBM 是 GPU 的主显存，大模型推理优化经常围绕减少 HBM 读写展开。

## SRAM

一句话定义：SRAM 是 GPU 片上更快但容量更小的存储，常用于缓存计算中的小块数据。

价值：访问速度快，适合在 kernel 内复用数据。

限制：容量小，不可能容纳完整大矩阵或完整模型参数。

面试表达：FlashAttention 通过把分块数据放入 SRAM 重复使用，减少对 HBM 的依赖。

## Fused Kernel

一句话定义：fused kernel 是把多个计算步骤融合到一个 GPU kernel 中执行，以减少中间结果写回和读出。

为什么有效：多个小算子分开执行会产生 kernel launch 开销和中间 tensor 的 HBM 读写。融合后可以在寄存器或 SRAM 中完成连续计算。

例子：bias、activation、dropout、residual、layer norm 等操作在某些实现中可以融合。

风险：融合 kernel 对硬件、shape 和框架版本更敏感，调试和维护成本更高。

面试表达：fused kernel 通过减少 kernel launch 和中间内存访问提升推理效率。

## Memory Bandwidth Bottleneck

一句话定义：memory bandwidth bottleneck 是性能主要受数据读写速度限制，而不是受算术计算能力限制的瓶颈。

在 LLM decode 中常见：每生成一个 token 都要读取大量权重和 KV Cache，但 batch 较小时计算密度不足，GPU 算力可能吃不满。

优化方向：量化降低数据量，增大 batch 提高复用，优化 KV Cache 布局，使用高效 kernel，减少不必要的内存读写。

面试表达：LLM 推理不总是算力瓶颈，decode 阶段经常是显存带宽瓶颈。

## Quantization

一句话定义：quantization 是用更低精度的数据格式表示模型权重或激活，以降低显存、带宽和部署成本。

为什么有效：参数和 KV Cache 占用的字节数下降后，可以部署更大模型、增大 batch、降低显存带宽压力，并可能提升吞吐。

主要对象：权重量化、激活量化、KV Cache 量化。LLM 中常见的是 weight-only quantization，因为它相对更容易保持质量。

风险：低精度会引入量化误差，可能导致困惑度上升、生成质量下降、数学和代码能力退化、长尾任务不稳定。

面试表达：量化用精度换成本和速度，但必须用校准集和业务评测验证质量损失。

## FP32

一句话定义：FP32 是 32 位浮点格式，精度高但显存和带宽成本大。

特点：数值稳定性好，常用于传统训练基线和部分高精度计算。

在 LLM 中：完整 FP32 训练和推理成本太高，实际大模型通常使用 BF16、FP16 或更低精度。

面试表达：FP32 稳但贵，现代大模型很少全程用 FP32 做主要计算。

## FP16

一句话定义：FP16 是 16 位浮点格式，显存约为 FP32 的一半，常用于训练和推理。

优点：显存和带宽成本低，硬件支持成熟。

风险：动态范围较小，训练中可能出现 overflow 或 underflow，需要 loss scaling 等技巧。

面试表达：FP16 快且省，但数值范围比 BF16 小，训练稳定性要特别关注。

## BF16

一句话定义：BF16 是 Brain Floating Point 16，指数位接近 FP32，动态范围大，常用于稳定训练和推理。

优点：比 FP16 更不容易溢出，很多大模型训练默认使用 BF16。

缺点：尾数精度比 FP16 更低，但大模型训练中通常更看重动态范围。

面试表达：BF16 的核心优势是动态范围接近 FP32，因此大模型训练更稳定。

## FP8

一句话定义：FP8 是 8 位浮点格式，可进一步降低显存和带宽成本，但依赖硬件、框架和校准策略支持。

价值：在支持 FP8 的新硬件上，可用于训练或推理加速。

挑战：数值范围、scale 管理、算子支持和质量评估更复杂。

面试表达：FP8 是更激进的低精度方向，收益大但工程和数值风险也更高。

## INT8

一句话定义：INT8 是 8 位整数量化格式，常用于较稳健的推理压缩。

优点：显存占用和带宽约为 FP16 的一半，质量通常比 INT4 更容易保持。

常见方式：权重量化、权重加激活量化、per-channel scale、per-token activation scale。

风险：异常激活值和分布漂移会影响量化效果。

面试表达：INT8 通常是质量和压缩率之间较稳健的折中。

## INT4

一句话定义：INT4 是 4 位整数量化格式，比 INT8 更省显存，但质量和 kernel 风险更高。

优点：大幅降低权重显存，适合单卡部署更大模型。

风险：量化误差更大，对校准、分组大小、重要权重保护和硬件 kernel 依赖更强。

适用场景：成本敏感、显存紧张、可接受少量质量损失的推理部署。

面试表达：INT4 很适合压缩权重，但不是无损加速，必须看任务质量和 kernel 实现。

## Weight-Only Quantization

一句话定义：weight-only quantization 是只量化模型权重，activation 仍使用较高精度的量化方案。

为什么常见：LLM 推理中权重占用大，decode 阶段频繁读取权重。只量化权重能降低显存和带宽，同时避免激活量化带来的额外误差。

优点：实现相对简单，质量更容易保持，适合大模型推理部署。

缺点：不是所有硬件和 kernel 都能充分利用低比特权重带来的速度收益；有时主要收益是省显存，而不是显著降延迟。

面试表达：weight-only quantization 是 LLM 推理最常见的量化路线之一，核心收益是减少权重显存和带宽。

## Post-Training Quantization / PTQ

一句话定义：PTQ 是模型训练完成后进行量化，不重新训练或只用少量校准数据。

优点：成本低、流程简单、适合已有模型快速部署。

缺点：质量依赖校准数据和量化算法，极低比特量化时容易损失能力。

适用场景：线上部署前压缩模型，快速评估不同精度和不同量化方法的效果。

面试表达：PTQ 的优势是便宜快速，风险是没有充分训练适应量化误差。

## Quantization-Aware Training / QAT

一句话定义：QAT 是训练或微调时模拟量化误差，让模型适应低精度表示。

优点：低比特场景下质量通常优于纯 PTQ，模型能学会抵抗量化噪声。

缺点：需要训练成本，流程更复杂，可能需要原始训练数据或高质量微调数据。

适用场景：对质量要求高、量化比特很低、PTQ 损失不可接受的部署。

面试表达：QAT 用额外训练成本换更好的低精度质量，适合 PTQ 无法满足要求的场景。

## GPTQ

一句话定义：GPTQ 是使用近似二阶信息进行逐层量化误差补偿的 LLM PTQ 方法。

核心思想：量化某些权重会引入误差，GPTQ 利用 Hessian 近似信息在逐列或逐块量化时补偿后续权重，尽量降低输出误差。

优点：在 4bit weight-only 量化中应用广泛，能在较低比特下保持较好质量。

限制：校准数据、分组大小、实现细节和 kernel 支持都会影响最终效果。

面试表达：GPTQ 是典型 PTQ 方法，重点是用近似二阶信息补偿量化误差。

## AWQ

一句话定义：AWQ 是 Activation-aware Weight Quantization，利用激活信息保护重要权重的量化方法。

核心观察：并非所有权重同等重要。对激活影响大的权重更敏感，需要更好保护。

做法直觉：根据校准数据中的激活分布识别重要通道，通过缩放等方式降低重要权重量化误差。

优点：常用于高质量低比特推理部署，尤其是 4bit 权重量化场景。

面试表达：AWQ 的关键词是 activation-aware，即用激活统计判断哪些权重更值得保护。

## Calibration Data

一句话定义：calibration data 是用于估计量化 scale、激活分布或敏感权重的小规模代表性数据。

为什么重要：量化参数来自数据分布。如果校准数据和真实业务分布差异很大，量化后模型可能在线上任务中明显退化。

选择原则：覆盖真实输入长度、语言、格式、领域、任务类型和高频业务模式。

常见坑：只用短问答校准长上下文模型，只用英文校准中文业务，只用通用数据校准专业领域服务。

面试表达：校准数据决定量化是否贴近真实分布，是 PTQ 质量的关键变量。

## Distillation

一句话定义：distillation 是用大模型或强模型作为 teacher，训练较小 student 模型模仿其行为的方法。

在部署中的价值：小模型推理更快、成本更低、延迟更小，适合高频或边缘场景。

常见信号：teacher 的输出文本、logits、偏好排序、推理过程或工具调用轨迹。

风险：student 容量有限，可能继承 teacher 的错误，也可能在复杂推理和长尾场景上能力不足。

面试表达：蒸馏是用训练成本换推理成本，适合把大模型能力迁移到更便宜的小模型上。

## TensorRT-LLM

一句话定义：TensorRT-LLM 是 NVIDIA 面向大语言模型推理优化的高性能部署框架。

常见能力：高效 kernel、量化支持、张量并行、pipeline 并行、KV Cache 优化、in-flight batching 等。

适用场景：NVIDIA GPU 上追求高性能、低延迟和工程可控的生产部署。

注意：框架收益依赖模型结构、硬件型号、batch、输入输出长度和具体配置，不是接入后必然所有场景都变快。

面试表达：TensorRT-LLM 是偏底层性能优化的 LLM 推理框架，重点服务 NVIDIA GPU 生态。

## vLLM

一句话定义：vLLM 是高吞吐 LLM serving 框架，代表性特性是 PagedAttention 和高效 continuous batching。

核心价值：提升显存利用率和在线 serving 吞吐，适合多请求并发和动态输出长度场景。

常见能力：OpenAI-compatible API、continuous batching、prefix caching、多种模型支持、分布式推理支持。

注意：实际性能取决于模型、硬件、并发、上下文长度和采样配置。

面试表达：vLLM 的代表性贡献是用 PagedAttention 管理 KV Cache，从而更好支持高吞吐 serving。

## Triton Inference Server

一句话定义：Triton Inference Server 是 NVIDIA 的通用模型推理服务框架，支持多后端、多模型和动态 batching。

价值：提供模型管理、HTTP/gRPC 接口、批处理、监控指标和多框架后端能力。

在 LLM 中：可以作为部署基础设施的一部分，但 LLM 还需要结合专门的生成推理后端或自定义后端处理 KV Cache 和流式输出。

面试表达：Triton 是通用推理服务框架，LLM 场景还需要关注生成式推理特有的调度和缓存管理。

## Model Parallel Inference

一句话定义：model parallel inference 是把一个模型切分到多张 GPU 上进行推理。

为什么需要：模型权重或 KV Cache 太大，单卡放不下；或者单卡吞吐不足，需要多卡协同。

常见方式：tensor parallel、pipeline parallel、expert parallel。推理中最常见的是 tensor parallel，把矩阵计算切到多卡。

代价：跨卡通信增加，部署复杂度上升，小 batch 或短请求下通信开销可能抵消收益。

面试表达：模型并行推理解决单卡放不下或吞吐不足的问题，但要付出通信和工程复杂度成本。

## CPU Offloading

一句话定义：CPU offloading 是把部分权重、KV Cache 或中间状态放到 CPU 内存，在需要时再搬到 GPU。

价值：缓解 GPU 显存不足，让更大模型或更长上下文在有限显存上运行。

代价：CPU-GPU 传输带宽和延迟远弱于 GPU HBM，频繁 offload 会显著拖慢推理。

适用场景：显存极度受限、低并发、离线任务、成本优先而非低延迟的场景。

面试表达：CPU offloading 用延迟换显存，不是高性能 serving 的默认选择。

## Inference Performance Incident

一句话定义：inference performance incident 是线上推理服务在延迟、吞吐、KV Cache、成本、错误率或质量回归上违反门禁的事故。

常见信号：P95 TTFT 升高、P95 TPOT 升高、P99 端到端延迟升高、KV block 分配失败、cache 命中率低、prompt token 膨胀、取消率上升、量化或调度导致关键业务退化。

排查方法：先把请求 trace 拆成 queue、tokenize、prefill、decode、streaming、postprocess 和客户端接收，再按业务线、输入长度、输出长度、模型版本和调度配置分桶。

面试表达：推理性能事故不能只看 GPU tokens/s，要把 TTFT、TPOT、尾延迟、KV、cache、prompt 成本和质量回归放在同一张审计表里。

## TTFT Regression

一句话定义：TTFT regression 是首 token 延迟相对 baseline 或 SLO 变差。

常见原因：排队时间增加、prompt 变长、prefill token budget 不合理、prefix cache 失效、tokenizer CPU 瓶颈、RAG / rerank / tool call 前置链路变慢、router 把请求打到拥塞实例。

定位方式：比较 `queue_ms`、`tokenize_ms`、`prefill_ms`、`first_chunk_ms` 和 input token 分布，而不是直接判断模型本体慢。

面试表达：TTFT 退化通常先看 queue 和 prefill，再看 prompt 长度、prefix cache、tokenizer 和上游链路。

## TPOT Regression

一句话定义：TPOT regression 是每输出 token 延迟变差，用户会感到流式输出卡顿。

常见原因：decode batch 过大、KV Cache 访存压力高、显存带宽瓶颈、流式链路缓冲、长短请求混跑、speculative decoding 接受率低或 proposal 成本过高。

定位方式：同时比较 worker 生成 token 时间、网关发送 chunk 时间和客户端接收时间，区分模型 decode 慢和传输 / 渲染慢。

面试表达：TPOT 退化不一定是算力不够，也可能是 KV 访存、调度、网络 backpressure 或客户端渲染造成的。

## KV Pressure

一句话定义：KV pressure 是活跃请求的 KV Cache 占用接近或超过可用 KV 显存预算的程度。

粗略口径：`rho_kv = M_kv / B_kv`，其中 `M_kv` 是当前或峰值 KV Cache 显存，`B_kv` 是可用于 KV Cache 的预算。

风险：`rho_kv` 过高时，系统可能出现 block 分配失败、抢占、排队、OOM、长请求拖垮短请求或取消请求释放不及时。

面试表达：KV pressure 是 serving 容量规划里的硬约束，不能只看模型权重是否能放进显存。

## Prompt Cost Drift

一句话定义：prompt cost drift 是新版 system prompt、RAG 上下文、工具 schema 或对话历史让输入 token 相对 baseline 膨胀。

典型指标：`gamma_prompt = new_input_tokens / baseline_input_tokens`。这个比例升高会同时推高 TTFT、KV Cache、token 成本和长上下文干扰风险。

常见治理：压缩 system prompt、按需注入工具 schema、提高 RAG context precision、做多轮摘要、设置 max input tokens 和业务线 token budget。

面试表达：很多推理成本事故不是模型突然变慢，而是 prompt 被产品和工具链不断加长。

## Cache Effectiveness

一句话定义：cache effectiveness 是缓存真实节省 prefill、延迟和成本的程度，而不是单纯命中次数。

关键指标：cache hit rate、命中 token 数、节省 prefill 时间、缓存占用、错误复用率、权限隔离和失效策略。

风险：cache key 过粗可能错误复用，过细则几乎没有命中；企业 RAG 还必须把租户、权限、文档版本和时间纳入缓存治理。

面试表达：缓存是成本优化工具，不是质量保证工具。是否有效要看节省了多少 token 和延迟，以及有没有错误复用。

## Serving Gate

一句话定义：serving gate 是推理服务上线或配置变更前的性能、容量、成本和质量硬门禁。

常见门禁：P95 TTFT、P95 TPOT、P99 E2E latency、P95 queue time、KV pressure、cache effectiveness、prompt cost drift、错误 / 取消率、关键切片质量回归和回滚准备度。

价值：它把“更高 tokens/s”转成可上线判断，避免优化吞吐却牺牲短请求体验、成本和质量。

面试表达：推理 serving gate 应该同时约束用户感知延迟、系统容量、单位成本和质量回归，而不是只看平均 latency。

## Inference Platform Overview Audit

一句话定义：Inference Platform Overview Audit 是把大模型推理平台从“一个模型 HTTP 服务”升级为请求生命周期、SLO、Model Registry、Router、Runtime、Batching、KV cache、扩缩容、发布、安全、成本、观测和开发者体验的可审计系统设计方法。

核心直觉：推理平台和单模型服务的差别，不在于有没有 API Gateway，而在于是否能稳定回答“请求从哪里来、走到哪个模型、排队多久、prefill / decode 各耗时多少、KV cache 是否够、能不能流式返回、出问题如何限流 / 熔断 / 降级 / 回滚、成本归到谁、审计能不能追踪”。vLLM、TGI、Triton、TensorRT-LLM、SGLang 等 runtime 负责高性能执行，平台控制面负责治理、SLO、发布和多租户边界。

常见审计维度：Request Lifecycle Coverage 看 auth、rate limit、input safety、routing、queue、prefill、decode、streaming、output safety、metrics、cost 和 audit 是否全链路覆盖；SLO Metric Contract 看 TTFT、TPOT、E2E、queue、input / output tokens、error、timeout 和 SLO 文档是否明确；Model Registry Readiness 看 model name、version、weight URI、tokenizer、runtime、context limit、quantization、eval report、stage 和 rollback version；Model Router Policy Fit 看租户、模型版本、延迟 SLO、成本预算、负载、fallback 和安全等级；Runtime Prefill Decode Fit、Continuous Batching Readiness 和 KV Cache Capacity Governance 看 runtime 是否区分 prefill / decode、支持动态加入 / 退出请求、管理 KV block、prefix cache、eviction、max context、admission 和指标；Autoscaling Warm Pool Fit、Release Rollback Control、Rate Limit Degrade Protection、Safety Governance Fit、Cost Attribution Optimization、Observability Trace Coverage、Multi Model Governance、Developer API Self Service 和 Inference Platform Gate 则检查扩缩容、灰度回滚、保护机制、安全、成本、trace、多模型策略和开发者自助是否闭环。

常见公式：推理平台审计样本可以写成

```math
u_i=(a_i,r_i,s_i,m_i,p_i,e_i,b_i,k_i,h_i,w_i,g_i,c_i,o_i,z_i)
```

统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(u_i)=1]
```

端到端延迟可以拆成

```math
T_{\mathrm{e2e}}=T_{\mathrm{auth}}+T_{\mathrm{route}}+T_{\mathrm{queue}}+T_{\mathrm{tokenize}}+T_{\mathrm{prefill}}+T_{\mathrm{decode}}+T_{\mathrm{stream}}+T_{\mathrm{safety}}
```

TTFT 与 TPOT 可以写成

```math
T_{\mathrm{ttft}}=T_{\mathrm{auth}}+T_{\mathrm{route}}+T_{\mathrm{queue}}+T_{\mathrm{tokenize}}+T_{\mathrm{prefill}}+T_{\mathrm{first}}
```

```math
T_{\mathrm{tpot}}=\frac{T_{\mathrm{decode}}}{N_{\mathrm{out}}}
```

KV cache 显存可以写成

```math
M_{\mathrm{kv}}=2LBSH_{\mathrm{kv}}D_hB_{\mathrm{elem}}
```

单位成本可以写成

```math
K_{\mathrm{1k}}=\frac{1000K_{\mathrm{total}}}{N_{\mathrm{in}}+N_{\mathrm{out}}}
```

推理平台门禁可以写成

```math
G_{\mathrm{infer}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{ttft,p95}}\le B_{\mathrm{ttft}} \land T_{\mathrm{tpot,p95}}\le B_{\mathrm{tpot}} \land R_{\mathrm{kv}}\le \rho_{\mathrm{kv}} \land R_{\mathrm{err}}\le \rho_{\mathrm{err}} \land P_0=0\right]
```

面试表达：推理平台总览题可以按“需求澄清 -> 请求生命周期 -> Model Registry -> Model Router -> Runtime -> batching / KV cache -> SLO 与扩缩容 -> 灰度回滚 -> 限流熔断降级 -> 安全治理 -> 成本归因 -> 观测 trace -> 多模型治理 -> 开发者自助 -> Inference Platform Gate”展开。不要只回答“用 K8s 部署模型服务”。

## Request Lifecycle Coverage

一句话定义：Request Lifecycle Coverage 是检查推理请求从 auth、rate limit、safety、routing、queue、prefill、decode、streaming 到 metrics、cost、audit 是否有完整阶段记录。

## SLO Metric Contract

一句话定义：SLO Metric Contract 是明确 TTFT、TPOT、E2E、queue time、tokens/s、error rate、timeout rate 和成本等推理平台指标的口径和阈值。

## Model Registry Readiness

一句话定义：Model Registry Readiness 是检查模型名称、版本、权重路径、tokenizer、runtime、上下文长度、量化、评估报告、发布状态和回滚版本是否可追踪。

扩展口径：推理平台接入模型仓库时，不应只读一个权重 URI，还要校验 Model Artifact Registry Audit 中的权重 manifest、safetensors 安全格式、tokenizer / config hash、adapter / merge lineage、量化 eval、runtime smoke test、安全权限、发布审批和 rollback alias。

## Model Router Policy Fit

一句话定义：Model Router Policy Fit 是检查路由是否根据租户、模型版本、延迟 SLO、成本预算、负载、fallback 和安全等级做决策。

## Runtime Prefill Decode Fit

一句话定义：Runtime Prefill Decode Fit 是检查推理 runtime 是否明确区分模型加载、tokenizer、prefill、decode、KV cache、streaming 和 runtime metrics。

## Continuous Batching Readiness

一句话定义：Continuous Batching Readiness 是检查 runtime 是否支持请求动态加入、动态退出、max wait、max batch tokens、优先级队列和长请求隔离。

## KV Cache Capacity Governance

一句话定义：KV Cache Capacity Governance 是把 KV block 管理、eviction、prefix cache、max context、admission control、KV pressure 和指标纳入推理平台容量门禁。

## Autoscaling Warm Pool Fit

一句话定义：Autoscaling Warm Pool Fit 是用 queue、TTFT、KV pressure、tokens/s、冷启动时间、warm pool 和 scale-down 策略共同判断推理扩缩容。

## Release Rollback Control

一句话定义：Release Rollback Control 是让模型发布支持 canary、A/B、traffic split、质量 / 延迟 / 成本护栏和快速回滚。

## Rate Limit Degrade Protection

一句话定义：Rate Limit Degrade Protection 是用租户配额、模型配额、最大输入 / 输出、排队超时、执行超时、熔断和缓存降级保护推理平台。

## Safety Governance Fit

一句话定义：Safety Governance Fit 是把鉴权、输入策略、输出策略、PII 脱敏、日志脱敏、留存策略和审计纳入在线推理治理。

## Cost Attribution Optimization

一句话定义：Cost Attribution Optimization 是按 tenant、model、version、input tokens、output tokens、GPU seconds 和 cache savings 归因推理成本并触发预算告警。

## Observability Trace Coverage

一句话定义：Observability Trace Coverage 是让 metrics、logs、traces、request id、route / queue / prefill / decode / stream spans、dashboard 和 alerts 串成一条排障证据链。

## Multi Model Governance

一句话定义：Multi Model Governance 是用 model、model version、endpoint、deployment、route policy、scaling policy、quota policy、safety policy 和 cost policy 管理多模型平台。

## Developer API Self Service

一句话定义：Developer API Self Service 是提供 OpenAPI、SDK、playground、usage dashboard、SLO dashboard、错误报告和配额申请，降低业务接入推理平台的摩擦。

## Inference Platform Gate

一句话定义：Inference Platform Gate 是判断大模型推理平台是否达到低延迟、高吞吐、高可用、可回滚、可观测、可限流、可审计和可归因最低要求的综合门禁。

典型硬阻断：只有一个 HTTP 服务没有请求生命周期 trace；只看 QPS 不看 TTFT / TPOT / tokens；Model Registry 没有 tokenizer、eval report 或 rollback version；Router 不看租户、SLO、成本和负载；runtime 不区分 prefill / decode；没有 continuous batching；KV cache 没有 admission 和 eviction；扩容只看 GPU utilization；模型发布无灰度和回滚；没有限流、超时、熔断和降级；日志不脱敏；成本不能按租户 / 模型 / 版本归因；只有 worker metrics 没有 request trace；多模型没有 route / quota / safety / cost policy；开发者无法自助查看用量和错误；没有最终推理平台门禁。

## Model Serving Runtime Selection Audit

一句话定义：Model Serving Runtime Selection Audit 是把 Triton、vLLM、TGI、SGLang、TensorRT-LLM 或自研 runtime 放进同一套 profile，按模型、硬件、权重格式、prefill / decode、batching、KV cache、streaming、量化、分布式、指标、benchmark、发布、维护状态、成本和自研门槛做选型审计。

核心直觉：runtime 选型不是排行榜问题。Triton 更偏通用多模型推理服务框架，vLLM / SGLang 更偏 LLM serving engine，TGI 对 Hugging Face 生态和历史部署有价值但要考虑当前维护状态，TensorRT-LLM 更偏 NVIDIA 生态深度优化，自研 runtime 则要求流量规模、团队能力和 fallback 方案同时成立。

常见审计维度：Model Hardware Fit 看模型类型、权重大小、上下文长度、GPU / CPU / 加速器、单机 / 多机拓扑是否匹配；Runtime Boundary Clarity 看平台层治理和 runtime 执行边界是否清楚；Model Format Tokenizer Fit 看权重格式、tokenizer、chat template、stop token 和 generation config 是否绑定；Prefill Decode Scheduler Fit、Continuous Batching Fit 和 KV Cache Memory Manager Fit 看 runtime 是否真正理解生成式推理；Streaming API Fit 看 SSE / gRPC、backpressure、取消和 finish reason；Quantization Accuracy Fit、Distributed Inference Fit、Observability Metrics Fit、Benchmarking Method Fit、Deployment Rollback Fit、Ecosystem Maintenance Fit、Cost Capacity Fit、Self Development Threshold 和 Runtime Selection Gate 则检查质量、分布式、指标、测评、发布、维护、成本、自研和最终上线门禁。

常见公式：runtime 选型样本可以写成

```math
r_i=(m_i,h_i,f_i,t_i,p_i,b_i,k_i,s_i,q_i,d_i,o_i,e_i,c_i,a_i,u_i,z_i)
```

统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(r_i)=1]
```

runtime 内部延迟可以拆成

```math
T_{\mathrm{runtime}}=T_{\mathrm{tokenize}}+T_{\mathrm{queue}}+T_{\mathrm{prefill}}+T_{\mathrm{decode}}+T_{\mathrm{sample}}+T_{\mathrm{stream}}
```

TTFT 与 TPOT 可以写成

```math
T_{\mathrm{ttft}}=T_{\mathrm{tokenize}}+T_{\mathrm{queue}}+T_{\mathrm{prefill}}+T_{\mathrm{first}}
```

```math
T_{\mathrm{tpot}}=\frac{T_{\mathrm{decode}}+T_{\mathrm{sample}}+T_{\mathrm{stream}}}{N_{\mathrm{out}}}
```

单请求 KV cache 显存和 KV 并发上限可以写成

```math
M_{\mathrm{kv,req}}=2LSH_{\mathrm{kv}}D_hB_{\mathrm{elem}}
```

```math
N_{\mathrm{kv}}=\left\lfloor \frac{B_{\mathrm{kv}}}{M_{\mathrm{kv,req}}}\right\rfloor
```

runtime 选型门禁可以写成

```math
G_{\mathrm{runtime}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{ttft,p95}}\le B_{\mathrm{ttft}} \land T_{\mathrm{tpot,p95}}\le B_{\mathrm{tpot}} \land S_{\mathrm{feature}}\ge \tau_{\mathrm{feature}} \land R_{\mathrm{compat}}=1 \land P_0=0\right]
```

面试表达：runtime 选型题可以按“需求澄清 -> 候选 runtime 定位 -> 模型 / 硬件 / tokenizer 兼容 -> prefill / decode -> batching / KV cache -> streaming -> 量化 -> 分布式 -> 指标 -> benchmark -> 发布回滚 -> 维护状态 -> 成本容量 -> 自研门槛 -> Runtime Selection Gate”展开。不要只回答“vLLM 更快”或“Triton 更稳定”。

## Model Hardware Fit

一句话定义：Model Hardware Fit 是检查模型结构、权重大小、上下文长度、目标吞吐、GPU 显存、互联拓扑和 runtime 支持是否匹配。

## Runtime Boundary Clarity

一句话定义：Runtime Boundary Clarity 是区分平台层的鉴权、路由、配额、发布治理与 runtime 内部的模型加载、prefill、decode、KV cache、sampling 和 streaming。

## Model Format Tokenizer Fit

一句话定义：Model Format Tokenizer Fit 是检查权重格式、tokenizer 版本、chat template、special token、stop token 和 generation config 是否随模型版本绑定。

## Prefill Decode Scheduler Fit

一句话定义：Prefill Decode Scheduler Fit 是检查 runtime 是否把 prefill、decode、queue、token budget、长短请求隔离和 phase metrics 纳入调度。

## Continuous Batching Fit

一句话定义：Continuous Batching Fit 是检查 runtime 是否支持请求动态加入、动态退出、max batch tokens、max wait 和长请求隔离，而不是只做静态 batch。

## KV Cache Memory Manager Fit

一句话定义：KV Cache Memory Manager Fit 是检查 KV block、admission control、prefix cache、eviction、KV pressure 和 OOM 保护是否闭环。

## Streaming API Fit

一句话定义：Streaming API Fit 是检查 runtime 的 SSE / gRPC streaming、backpressure、cancel、timeout、finish reason 和 usage 统计是否能支撑线上体验。

## Quantization Accuracy Fit

一句话定义：Quantization Accuracy Fit 是检查 FP8 / INT8 / INT4 / weight-only / KV cache quantization 是否有质量评测、切片回归和回滚阈值。

## Distributed Inference Fit

一句话定义：Distributed Inference Fit 是检查 tensor parallel、pipeline parallel、expert parallel、data parallel serving、PD 分离、拓扑和 rank 健康是否满足模型和 SLO。

## Observability Metrics Fit

一句话定义：Observability Metrics Fit 是检查 TTFT、TPOT、queue、prefill、decode、KV、OOM、error、version 和 per-rank 指标是否可观测。

## Benchmarking Method Fit

一句话定义：Benchmarking Method Fit 是检查 runtime benchmark 是否固定同一模型、硬件、精度、输入输出长度分布、并发、SLO 和 streaming 设置。

## Deployment Rollback Fit

一句话定义：Deployment Rollback Fit 是检查 runtime 镜像 digest、版本 pin、health probe、canary、回滚和兼容性测试是否完整。

## Ecosystem Maintenance Fit

一句话定义：Ecosystem Maintenance Fit 是检查 runtime 的维护状态、release tracking、兼容矩阵、社区或团队 owner 和安全更新责任。

## Cost Capacity Fit

一句话定义：Cost Capacity Fit 是把 GPU-hour、input / output tokens、KV 预算、并发上限、冷启动和每 1k token 成本纳入 runtime 选型。

## Self Development Threshold

一句话定义：Self Development Threshold 是判断自研 runtime 是否真的具备规模收益、系统能力、CUDA / kernel 能力、分布式经验和 fallback runtime。

## Runtime Selection Gate

一句话定义：Runtime Selection Gate 是判断模型服务 runtime 是否达到功能、性能、质量、维护、发布、安全和成本最低上线要求的综合门禁。

典型硬阻断：候选 runtime 不支持目标模型或硬件；平台治理和 runtime 执行边界混乱；tokenizer / chat template 未绑定；不区分 prefill / decode；没有 continuous batching；KV cache 无 admission 和 eviction；streaming 不能取消或 backpressure；量化没有质量回归；多 GPU 拓扑未知；缺 TTFT / TPOT / decode 指标；benchmark 输入输出分布不一致；版本不可回滚；维护状态未知；成本容量未估算；自研理由只有“想更快”；没有最终 Runtime Selection Gate。

## Prefill Decode KV Resource Profile Audit

一句话定义：Prefill Decode KV Resource Profile Audit 是把大模型在线推理请求拆成输入 token、输出 token、TTFT、TPOT、prefill、decode、KV cache、prefix cache、长上下文、多租户、streaming 和成本的可审计资源画像。

核心直觉：推理慢不是一个平均 latency 或 QPS 能解释的问题。长输入主要推高 prefill 和 TTFT，长输出主要拉长 decode 和 TPOT，高并发和长上下文会放大 KV cache 显存压力，prefix cache 可能节省 prefill，但也要受租户隔离、权限和错误复用门禁约束。

常见审计维度：Request Token Profile 看输入 / 输出 token 分布、上下文长度和 token budget；Prefill Phase Accounting、Decode Phase Accounting 和 TTFT TPOT Contract 看 prefill、decode、queue、TTFT、TPOT 和 E2E 是否分开记录；KV Cache Formula Fit 和 KV Capacity Admission 看 KV cache 公式、预算、admission 和 KV pressure 是否正确；Paged Block Management、Prefix Cache Reuse、Long Context Policy 和 Tenant KV Isolation 检查 block 管理、cache 复用、长上下文治理和租户隔离；Continuous Batching Policy、PD Disaggregation Fit、Streaming Backpressure Fit、Observability Phase Metrics、Cost Capacity Model 和 Resource Profile Gate 则检查调度、PD 分离、流式返回、观测、成本容量和最终门禁。

常见公式：资源画像样本可以写成

```math
p_i=(x_i,y_i,c_i,q_i,f_i,d_i,k_i,b_i,r_i,s_i,t_i,o_i,z_i)
```

统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(p_i)=1]
```

TTFT、TPOT 和端到端延迟可以写成

```math
T_{\mathrm{ttft}}=T_{\mathrm{gateway}}+T_{\mathrm{route}}+T_{\mathrm{queue}}+T_{\mathrm{tokenize}}+T_{\mathrm{prefill}}+T_{\mathrm{first}}
```

```math
T_{\mathrm{tpot}}=\frac{T_{\mathrm{decode}}+T_{\mathrm{sample}}+T_{\mathrm{stream}}}{N_{\mathrm{out}}}
```

```math
T_{\mathrm{e2e}}\approx T_{\mathrm{ttft}}+N_{\mathrm{out}}T_{\mathrm{tpot}}
```

KV cache 显存、KV pressure 和 prefix cache 收益可以写成

```math
M_{\mathrm{kv}}=2LBSH_{\mathrm{kv}}D_hB_{\mathrm{elem}}
```

```math
R_{\mathrm{kv}}=\frac{M_{\mathrm{kv}}}{B_{\mathrm{kv}}}
```

```math
R_{\mathrm{prefix}}=\frac{N_{\mathrm{hit\_tokens}}}{N_{\mathrm{input\_tokens}}}
```

资源画像门禁可以写成

```math
G_{\mathrm{resource}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{ttft,p95}}\le B_{\mathrm{ttft}} \land T_{\mathrm{tpot,p95}}\le B_{\mathrm{tpot}} \land R_{\mathrm{kv}}\le \rho_{\mathrm{kv}} \land P_0=0\right]
```

面试表达：Prefill / Decode / KV 资源画像题可以按“token 分布 -> TTFT / TPOT -> prefill -> decode -> KV cache 公式 -> KV pressure -> PagedAttention / block -> prefix cache -> 长上下文 -> 多租户 -> continuous batching -> PD 分离 -> streaming -> observability -> cost -> Resource Profile Gate”展开。不要只说“QPS 高”或“GPU 利用率低”。

## Request Token Profile

一句话定义：Request Token Profile 是记录输入 token、输出 token、上下文长度、token budget、长短请求分布和业务切片的推理请求画像。

## Prefill Phase Accounting

一句话定义：Prefill Phase Accounting 是把 tokenization、queue、prompt prefill、chunked prefill、prefix cache 命中和 prefill tokens/s 单独记录。

## Decode Phase Accounting

一句话定义：Decode Phase Accounting 是把每步 decode、采样、stream flush、output tokens/s、长输出和取消释放单独记录。

## TTFT TPOT Contract

一句话定义：TTFT TPOT Contract 是明确 TTFT、TPOT、E2E、P95 / P99、queue 和 streaming 指标口径、阈值与报警策略。

## KV Cache Formula Fit

一句话定义：KV Cache Formula Fit 是检查 KV cache 显存估算是否包含层数、活跃序列数、上下文长度、KV head、head dim 和 dtype bytes。

## KV Capacity Admission

一句话定义：KV Capacity Admission 是在请求进入 runtime 前按 KV 预算、max context、并发、block 余量和优先级做准入控制。

## Paged Block Management

一句话定义：Paged Block Management 是用固定大小 KV block、block table、free list、碎片指标和回收策略管理非连续 KV cache。

## Prefix Cache Reuse

一句话定义：Prefix Cache Reuse 是复用相同或可共享前缀的 KV cache，以减少重复 prefill、降低 TTFT 和节省成本。

## Long Context Policy

一句话定义：Long Context Policy 是对长 prompt、长文档、RAG 上下文和多轮历史设置 max context、预算、截断 / 压缩、长短队列和专用实例策略。

## Tenant KV Isolation

一句话定义：Tenant KV Isolation 是按租户、权限、模型版本和数据分类隔离 KV cache、prefix cache、配额和成本归因。

## Continuous Batching Policy

一句话定义：Continuous Batching Policy 是定义请求动态加入 / 退出、max batch tokens、max wait、长请求隔离和 decode step 调度的策略。

## PD Disaggregation Fit

一句话定义：PD Disaggregation Fit 是判断 prefill / decode 分离是否值得引入，包括 KV transfer、资源池独立扩缩容、故障边界和复杂度成本。

## Streaming Backpressure Fit

一句话定义：Streaming Backpressure Fit 是检查 SSE / gRPC streaming 的 flush、cancel、客户端背压、timeout 和 usage 统计是否完整。

## Observability Phase Metrics

一句话定义：Observability Phase Metrics 是让 queue、tokenize、prefill、decode、sample、stream、KV、prefix hit、OOM、tenant 和 cost 指标可按请求 trace 关联。

## Cost Capacity Model

一句话定义：Cost Capacity Model 是按 input tokens、output tokens、GPU seconds、KV MiB、prefix savings、冷启动和租户维度估算推理成本容量。

## Resource Profile Gate

一句话定义：Resource Profile Gate 是判断推理资源画像是否足以支撑低延迟、高吞吐、长上下文、多租户、可观测和成本可控上线的综合门禁。

典型硬阻断：只记录 QPS 不记录 input / output tokens；TTFT 和 TPOT 混成一个 latency；prefill / decode 没有 phase metrics；KV cache 显存公式缺 KV head 或 dtype；没有 KV admission；PagedAttention 只知道名字没有 block 指标；prefix cache 不做租户隔离；长上下文无限制；多租户没有 cache 配额；continuous batching 无动态退出；PD 分离不估 KV transfer；streaming 无 backpressure；trace 里缺 prefill / decode span；成本模型只按请求数算；没有 Resource Profile Gate。

## Inference Scheduling Audit

一句话定义：Inference Scheduling Audit 是检查推理调度是否把请求到达分布、continuous batching、token budget、prefill/decode 平衡、KV block、准入控制、长短请求隔离、多租户公平、取消释放、streaming 背压和最终门禁放进同一张审计表。

常见审计维度：Request Arrival Profile 看 input / output token 分布、到达时间、租户和优先级；Continuous Batching Core 看 dynamic join、dynamic exit、iteration-level scheduling 和 active sequence 限制；Token Budget Policy 看 max batch tokens、max prefill chunk 和 decode token budget；Prefill Decode Balance 看 decode-first、chunked prefill、prefill queue 和 decode queue；Paged KV Block Management 看 block size、block table、free list 和碎片指标；Scheduler KV Admission、Long Short Isolation、Tenant Priority Fairness、Cancellation Cleanup、Streaming Backpressure、Scheduler Observability Metrics 和 Scheduler Gate 则检查准入、长短队列、租户公平、取消释放、流式背压、指标和最终上线门禁是否闭环。

面试表达：推理调度题可以按“请求画像 -> dynamic join / exit -> token budget -> prefill/decode phase policy -> PagedAttention block -> KV admission -> 长短请求隔离 -> 多租户公平 -> cancel / timeout cleanup -> streaming backpressure -> metrics -> Scheduler Gate”展开。不要只说“开 continuous batching 就能提升吞吐”。

## Request Arrival Profile

一句话定义：Request Arrival Profile 是记录请求到达时间、租户、优先级、输入 token、最大输出 token、上下文长度和业务切片的调度输入画像。

## Continuous Batching Core

一句话定义：Continuous Batching Core 是支持请求在 decode iteration 之间动态加入、动态退出，并维护 waiting queue、running set 和 finished set 的核心调度能力。

## Token Budget Policy

一句话定义：Token Budget Policy 是用每轮 max batch tokens、max prefill chunk、decode token budget 和 max active sequences 控制吞吐、TTFT、TPOT 和 KV 压力。

## Prefill Decode Balance

一句话定义：Prefill Decode Balance 是在长 prompt prefill 和已有请求 decode 之间分配 GPU 时间，避免长 prefill 拉高 TPOT 或过度偏向 decode 拉高 TTFT。

## Paged KV Block Management

一句话定义：Paged KV Block Management 是用 block table、free list、block size、引用计数和碎片指标支持 KV cache 动态分配、追加、释放和复用。

## Scheduler KV Admission

一句话定义：Scheduler KV Admission 是在请求进入 running set 前按 KV block 预算、水位线、上下文长度、租户配额和优先级做准入控制。

## Long Short Isolation

一句话定义：Long Short Isolation 是把长 prompt、长输出或长上下文请求和短请求分队列、分预算或分实例，降低 head-of-line blocking 和 p99 抖动。

## Tenant Priority Fairness

一句话定义：Tenant Priority Fairness 是按租户配额、优先级、token share、GPU seconds、KV 占用和成本归因控制多租户推理资源分配。

## Cancellation Cleanup

一句话定义：Cancellation Cleanup 是在用户取消、客户端断开、超时或 stop sequence 触发后及时释放 running slot、KV blocks、streaming 连接和指标状态。

## Scheduler Observability Metrics

一句话定义：Scheduler Observability Metrics 是把 queue wait、TTFT、TPOT、scheduled tokens、active sequences、KV blocks、KV waste、rejects、cancellations 和 tenant tokens 纳入可观测闭环。

## Scheduler Gate

一句话定义：Scheduler Gate 是判断推理调度是否满足低 TTFT、低 TPOT、高吞吐、KV 可控、公平性、低拒绝率和无 P0 风险的综合门禁。

## Model Routing Audit

一句话定义：Model Routing Audit 是检查多模型平台是否把请求意图、模型能力、权限、成本、延迟 SLO、质量安全、fallback、灰度、实时健康、trace、降级、配置版本和最终门禁放进同一套路由审计表。

常见审计维度：Request Intent Profile 看任务类型、租户、输入 / 输出 token、工具调用、多模态、SLO、预算和风险等级；Model Capability Profile 看模型任务能力、上下文长度、tool calling、结构化输出、多模态、版本、质量分和安全等级；Permission Candidate Filter 看租户权限、合规区域、模型访问范围和 endpoint 可用性；Cost Budget Policy 和 Latency SLO Policy 看 token 成本、GPU seconds、TTFT、TPOT、排队和总延迟是否先硬过滤；Quality Safety Policy 看任务质量、风险等级、安全拦截和评估回归；Fallback Chain Governance、Canary Routing Stability、Realtime Health Load、Route Trace Coverage、Degrade Policy Control、Route Config Versioning 和 Model Routing Gate 则检查 fallback 预算、稳定灰度、健康负载、trace 完整性、降级边界、配置审计和最终上线门禁。

面试表达：模型路由题可以按“请求画像 -> 模型能力画像 -> 权限和能力过滤 -> 成本预算 -> 延迟 SLO -> 质量安全打分 -> fallback chain -> 灰度分桶 -> 实时健康负载 -> route trace -> 降级策略 -> 配置版本 -> Model Routing Gate”展开。不要只说“按模型名字转发”或“便宜请求用小模型”。

## Request Intent Profile

一句话定义：Request Intent Profile 是路由器对一次请求的结构化画像，覆盖任务类型、租户等级、输入 / 输出 token、上下文长度、工具调用、多模态、延迟 SLO、成本预算、风险等级和是否允许降级。

## Model Capability Profile

一句话定义：Model Capability Profile 是模型注册表中的能力画像，覆盖支持任务、最大上下文、多模态、tool calling、结构化输出、质量评估、安全等级、成本、延迟基线、版本和回滚状态。

## Permission Candidate Filter

一句话定义：Permission Candidate Filter 是在模型打分前先按租户权限、合规区域、模型访问范围、endpoint 健康和业务策略过滤候选，避免把不合规模型靠高分选中。

## Cost Budget Policy

一句话定义：Cost Budget Policy 是根据输入 / 输出 token、模型单价、GPU seconds、cache savings、租户预算和优先级控制一次请求是否可以进入某个模型候选。

## Latency SLO Policy

一句话定义：Latency SLO Policy 是根据 TTFT、TPOT、queue time、route overhead、输出 token 预算和 endpoint 实时负载判断候选模型是否满足请求延迟目标。

## Quality Safety Policy

一句话定义：Quality Safety Policy 是把任务质量评估、安全等级、风险等级、在线反馈、回归指标和拦截策略纳入路由排序和上线门禁。

## Fallback Chain Governance

一句话定义：Fallback Chain Governance 是为主模型失败后的备用模型链设置最大尝试次数、错误类型、总时间预算、总成本预算、幂等控制和熔断规则。

## Canary Routing Stability

一句话定义：Canary Routing Stability 是用稳定哈希、租户分桶、版本权重和快速回滚策略控制灰度模型流量，避免同一用户在新旧版本间抖动。

## Realtime Health Load

一句话定义：Realtime Health Load 是把 endpoint 健康、队列长度、错误率、timeout、TTFT、TPOT、KV pressure、admission reject 和区域状态输入路由决策。

## Route Trace Coverage

一句话定义：Route Trace Coverage 是检查 route trace 是否记录 request id、tenant、task、tokens、selected model、version、route reason、fallback、degrade、cost、latency 和 config version。

## Degrade Policy Control

一句话定义：Degrade Policy Control 是定义哪些业务、租户、风险等级和错误类型允许降级到小模型、缓存、文本-only 或简化流程，并统计降级率和质量影响。

## Route Config Versioning

一句话定义：Route Config Versioning 是让模型路由策略、权重、灰度、fallback、权限和降级规则可版本化、可审计、可 dry-run 和可回滚。

## Model Routing Gate

一句话定义：Model Routing Gate 是判断多模型路由是否满足权限正确、能力匹配、预算可控、SLO 可达、质量安全、fallback 有界、灰度稳定、trace 完整和无 P0 风险的综合门禁。

## Inference Cache System Audit

一句话定义：Inference Cache System Audit 是检查 prompt / prefix cache、KV cache、semantic cache 和 result cache 是否在收益、正确性、租户隔离、一致性、随机性、streaming 和可观测性上形成闭环。

常见审计维度：Request Cache Profile 看请求 hash、模型、版本、tokenizer、参数、租户、权限、安全策略、数据版本、TTL 和 trace 是否完整；Cache Layer Boundary 看 result cache、semantic cache、prompt / prefix cache、KV cache 的命中条件和生命周期是否分清；Key Version Fingerprint 看模型、tokenizer、参数、安全策略和数据版本是否进入 key；Prompt Prefix Reuse 和 KV Lifecycle Management 看前缀 token 复用、KV block、release、retention、eviction 和显存水位；Semantic Similarity Guard 看相似度阈值、任务等价、权限、数据版本和误命中率；Result Cache Determinism、Tenant Permission Isolation、TTL Staleness Control、Eviction Quota Policy、Streaming Cache Policy、Cache Observability Metrics、Cost Latency Savings 和 Cache Governance Gate 则检查确定性、隔离、过期、驱逐、流式返回、指标、收益和最终门禁。

面试表达：缓存体系题可以按“请求画像 -> 缓存层边界 -> key 和版本指纹 -> prompt / prefix cache -> KV 生命周期 -> semantic cache 阈值 -> result cache 确定性 -> 租户权限隔离 -> TTL 与失效 -> 驱逐与配额 -> streaming 策略 -> 指标 -> 成本延迟收益 -> Cache Governance Gate”展开。不要只说“加 Redis 缓存”。

## Request Cache Profile

一句话定义：Request Cache Profile 是把一次缓存查找的模型、版本、tokenizer、参数、prompt hash、prefix hash、租户、权限域、安全策略、数据版本、TTL、采样随机性和 trace 字段结构化。

## Cache Layer Boundary

一句话定义：Cache Layer Boundary 是区分 result cache、semantic cache、prompt / prefix cache、KV cache 和 tokenizer cache 的缓存对象、命中条件、生命周期、收益和风险边界。

## Key Version Fingerprint

一句话定义：Key Version Fingerprint 是把模型版本、tokenizer、生成参数、prompt 模板、安全策略、数据版本、租户和权限域纳入 cache key 或 cache fingerprint，防止跨版本误命中。

## Prompt Prefix Reuse

一句话定义：Prompt Prefix Reuse 是复用相同 system prompt、工具说明、RAG 文档或代码上下文前缀，减少重复 prefill token 和 TTFT。

## KV Lifecycle Management

一句话定义：KV Lifecycle Management 是管理 KV cache 从 prefill 创建、decode 追加、prefix retention、取消释放、超时释放、eviction 到显存水位控制的完整生命周期。

## Semantic Similarity Guard

一句话定义：Semantic Similarity Guard 是语义缓存命中前检查 embedding 相似度、任务等价、租户权限、数据版本、敏感等级和误命中风险。

## Result Cache Determinism

一句话定义：Result Cache Determinism 是要求完整结果缓存只在确定性或近似确定性任务中使用，并把 temperature、top_p、max_tokens、stop sequence 和输出策略纳入 key。

## Tenant Permission Isolation

一句话定义：Tenant Permission Isolation 是确保缓存不能跨租户、跨用户、跨权限域或跨数据分类返回私有上下文、检索结果、KV state 或完整响应。

## TTL Staleness Control

一句话定义：TTL Staleness Control 是用 TTL、数据版本、模型版本、主动失效、purge 和二次校验控制过期缓存命中。

## Eviction Quota Policy

一句话定义：Eviction Quota Policy 是按 LRU、LFU、TTL、成本收益、租户配额、安全等级、模型版本和显存水位驱逐缓存，防止缓存挤占关键资源。

## Streaming Cache Policy

一句话定义：Streaming Cache Policy 是定义流式响应下是否缓存完整结果、是否模拟分片返回、如何处理取消、计费、trace 和 cache-hit 标记。

## Cache Observability Metrics

一句话定义：Cache Observability Metrics 是记录 hit / miss、saved tokens、saved TTFT、saved GPU time、cache latency、eviction、stale hit、semantic false hit、tenant hit rate 和 net cost saved。

## Cost Latency Savings

一句话定义：Cost Latency Savings 是扣除缓存系统自身存储、检索、网络和运维成本后，衡量缓存实际节省的 token、GPU time、TTFT、E2E latency 和成本。

## Cache Governance Gate

一句话定义：Cache Governance Gate 是判断缓存体系是否同时满足收益为正、误命中可控、过期命中可控、租户隔离正确、trace 完整和无 P0 风险的综合门禁。

## Inference Autoscaling Audit

一句话定义：Inference Autoscaling Audit 是检查 LLM 推理扩缩容是否把 token 负载、SLO、队列、GPU / KV、冷启动、warm pool、缩容 draining、路由准入、租户配额、成本和 trace 放进同一套控制闭环。

常见审计维度：Traffic Token Profile 看 QPS、input tokens/s、output tokens/s、active sequences、context tokens 和租户流量；SLO Latency Contract 看 TTFT、TPOT、E2E、p95 / p99、timeout 和错误率目标；Queue Backlog Signal 看 waiting queue、queue wait、pending prefill、pending decode 和 admission reject；GPU KV Capacity Signal 看 GPU utilization、显存、KV pressure、free KV blocks、eviction、OOM 和 tokens/s；Cold Start Lead Time 和 Warm Pool Readiness 看调度、镜像、权重加载、runtime 初始化、预热、endpoint 注册和 warm pool 可用副本；Multi Metric Recommendation、Scale Up Down Policy、Draining Streaming Safety、Router Admission Coupling、Tenant Quota Priority、Cost Budget Guard、Autoscaling Trace Coverage 和 Autoscaling Gate 则检查多指标推荐、扩缩容策略、draining、路由 / 准入联动、租户配额、成本、trace 和最终门禁。

面试表达：自动扩缩容题可以按“流量 token 画像 -> SLO -> 队列 -> GPU / KV -> 冷启动 -> warm pool -> 多指标副本推荐 -> 扩容快缩容慢 -> draining / streaming -> router / admission 联动 -> 租户配额 -> 成本预算 -> autoscaling trace -> Autoscaling Gate”展开。不要只说“用 HPA 看 GPU 利用率”。

## Traffic Token Profile

一句话定义：Traffic Token Profile 是把 QPS 拆成 input tokens/s、output tokens/s、上下文长度、active sequences、长短请求比例、租户流量和优先级画像。

## SLO Latency Contract

一句话定义：SLO Latency Contract 是为 TTFT、TPOT、E2E、queue wait、p95 / p99、timeout rate 和 error rate 定义明确扩缩容目标。

## Queue Backlog Signal

一句话定义：Queue Backlog Signal 是用 waiting queue、queue wait、pending prefill、pending decode、active requests 和 reject count 提前发现容量不足。

## GPU KV Capacity Signal

一句话定义：GPU KV Capacity Signal 是把 GPU utilization、GPU memory、KV pressure、free KV blocks、eviction、OOM、input / output tokens/s 和 per-GPU throughput 作为容量信号。

## Cold Start Lead Time

一句话定义：Cold Start Lead Time 是从触发扩容到新实例真正接流量的总时间，覆盖调度、镜像、权重加载、runtime 初始化、显存分配、kernel 预热和路由注册。

## Warm Pool Readiness

一句话定义：Warm Pool Readiness 是评估常驻副本、预拉镜像、权重本地缓存、预加载模型、预热 kernel 和可快速接流量实例是否足以覆盖冷启动窗口。

## Multi Metric Recommendation

一句话定义：Multi Metric Recommendation 是按 QPS、input tokens/s、output tokens/s、queue wait、TTFT、TPOT、KV pressure、GPU utilization 和错误率取最大需求副本，而不是依赖单一指标。

## Scale Up Down Policy

一句话定义：Scale Up Down Policy 是定义扩容阈值、缩容阈值、滑动窗口、cooldown、最小 / 最大副本、预算上限、预测扩容和异常指标平滑。

## Draining Streaming Safety

一句话定义：Draining Streaming Safety 是缩容前摘除路由、不接新请求、等待 active requests 和 streaming 结束、设置最长等待和 cooldown，避免直接中断用户请求。

## Router Admission Coupling

一句话定义：Router Admission Coupling 是让 autoscaler、router、admission controller、缓存、限流和降级共享容量指标，避免单独扩容或单独转移流量造成故障扩散。

## Tenant Quota Priority

一句话定义：Tenant Quota Priority 是按租户 token/s、队列、SLO、预算、优先级和突发配额决定扩缩容、限流、降级和成本归因。

## Cost Budget Guard

一句话定义：Cost Budget Guard 是在扩容决策中加入 GPU-hour、warm pool 空闲成本、预算上限、租户成本归因和单位成功请求成本。

## Autoscaling Trace Coverage

一句话定义：Autoscaling Trace Coverage 是记录触发指标、观测窗口、当前副本、建议副本、warm pool、冷启动、draining、成本、路由变更和配置版本。

## Autoscaling Gate

一句话定义：Autoscaling Gate 是判断推理扩缩容是否满足 SLO、队列、KV、水位、有效容量、冷启动、成本、租户公平、trace 完整和无 P0 风险的综合门禁。

## Inference Protection Policy Audit

一句话定义：Inference Protection Policy Audit 是检查 LLM 推理平台是否把限流、准入、熔断、重试、超时、降级、租户公平、streaming 清理、成本和 trace 放进同一条请求保护链路。

常见审计维度：Request Token Rate Limit 看请求数、input tokens、output tokens、上下文长度和成本额度是否共同受控；Concurrency Quota Control 看租户、用户、API key、模型和 endpoint 的并发是否有上限；Admission Capacity Guard 看队列、KV pressure、剩余 timeout、runtime 健康和 SLO 是否决定是否接收请求；Circuit Breaker State Machine 看 closed / open / half-open、滑动窗口错误率、慢调用率和探测恢复；Retry Error Classification、Retry Budget Bound 和 Idempotency Stage Guard 看是否只对未开始执行的临时错误重试，并受 retry budget、幂等键和执行阶段约束；Timeout Budget Allocation 和 Degradation Policy Control 看总 deadline 是否分配到 gateway、queue、prefill、decode、streaming 和 fallback；Priority Tenant Fairness、Streaming Cancellation Cleanup、Protection Trace Coverage、Cost Budget Guard 和 Protection Gate 则检查优先级、公平性、取消释放、审计 trace、成本和最终门禁。

面试表达：限流、熔断、重试、超时和降级题可以按“租户 / token / 并发限流 -> admission control -> endpoint 熔断 -> retry 分类和预算 -> timeout budget -> 业务允许的降级 -> streaming / cancel 清理 -> trace / cost / policy version -> Protection Gate”展开。不要只说“失败就重试”或“QPS 高了限流”。

## Request Token Rate Limit

一句话定义：Request Token Rate Limit 是把普通 QPS 限流扩展到 input tokens、output tokens、上下文长度、请求权重和成本额度。

## Concurrency Quota Control

一句话定义：Concurrency Quota Control 是限制租户、用户、API key、模型、endpoint 和任务类型的同时在途请求数，防止长请求占满连接、KV cache 和 GPU slot。

## Admission Capacity Guard

一句话定义：Admission Capacity Guard 是在请求进入 runtime 前检查队列、KV pressure、剩余 timeout、endpoint 健康、租户优先级和 SLO，决定 admit、reject、queue 或 degrade。

## Circuit Breaker State Machine

一句话定义：Circuit Breaker State Machine 是用 closed、open 和 half-open 管理异常 endpoint 或依赖，避免错误率、慢调用、队列爆炸或 OOM 继续扩散。

## Retry Error Classification

一句话定义：Retry Error Classification 是区分 connect reset、尚未开始执行的临时不可达、权限失败、配额失败、安全拒绝、用户取消和已开始 streaming 等错误，决定哪些可以重试。

## Retry Budget Bound

一句话定义：Retry Budget Bound 是限制重试流量占初始请求流量的比例，避免多层重试在过载时形成 retry storm。

## Idempotency Stage Guard

一句话定义：Idempotency Stage Guard 是要求重试必须绑定幂等键、执行阶段、attempt id 和 partial output 状态，避免已经开始生成或产生副作用的请求被盲目重放。

## Timeout Budget Allocation

一句话定义：Timeout Budget Allocation 是把总 deadline 拆到 gateway、queue、prefill、decode、streaming、工具调用、后处理和 buffer，并用剩余预算决定继续、拒绝或降级。

## Degradation Policy Control

一句话定义：Degradation Policy Control 是把小模型、缓存、截断上下文、减少输出、关闭工具调用、异步处理和转人工等降级动作绑定到业务风险、租户权限、质量下限和用户可见说明。

## Priority Tenant Fairness

一句话定义：Priority Tenant Fairness 是在过载时按租户等级、业务 criticality、SLO、预算、配额和历史使用量做公平保护，而不是让单个突发租户拖垮全局。

## Streaming Cancellation Cleanup

一句话定义：Streaming Cancellation Cleanup 是在客户端断开、用户取消、timeout 或降级时释放 KV block、连接、队列项、partial output 状态和 trace，避免资源泄漏。

## Protection Trace Coverage

一句话定义：Protection Trace Coverage 是记录 rate limit、admission、circuit、retry、timeout、degrade、fallback、成本、policy version 和最终 decision reason 的 trace 覆盖率。

## Protection Gate

一句话定义：Protection Gate 是判断推理保护策略是否满足限流正确、准入可靠、熔断隔离、重试有界、超时受控、降级可解释、trace 完整和无 P0 风险的综合门禁。

## Inference Release Governance Audit

一句话定义：Inference Release Governance Audit 是检查模型发布是否把 release bundle、版本包完整性、离线评估、契约测试、稳定分桶、canary、A/B、guardrail、rollback、audit trace 和上线后监控放进同一套发布治理闭环。

常见审计维度：Release Bundle Trace 看模型权重、tokenizer、prompt、runtime config、安全策略、eval report、owner、approver 和 rollback target 是否完整；Model Artifact Integrity 看权重 checksum、镜像 digest、tokenizer checksum 和 artifact lineage；Tokenizer Prompt Runtime Binding 看线上输出是否能追溯到完整版本包；Offline Eval Quality Gate、Safety Eval Gate 和 Contract Test Compatibility 看质量、安全、结构化输出、tool call schema 和下游 parser 是否过线；Stable Bucket Assignment、Canary Ramp Control、AB Experiment Design 和 Sample Ratio Health 看分桶、放量、实验设计和样本比例是否可信；Guardrail Metric Control、Rollback Readiness、Release Audit Trace、Post Release Monitoring 和 Release Governance Gate 则检查安全、延迟、成本、业务护栏、回滚目标、审计和最终门禁。

面试表达：模型发布题可以按“release bundle -> artifact 完整性 -> tokenizer / prompt / runtime 绑定 -> offline eval -> safety eval -> contract test -> stable bucket -> canary ramp -> A/B design -> sample ratio -> guardrail -> rollback -> audit trace -> post-release monitoring -> Release Governance Gate”展开。不要只说“灰度 1% 然后慢慢加流量”。

## Release Bundle Trace

一句话定义：Release Bundle Trace 是把模型权重、tokenizer、prompt、runtime config、安全策略、eval report、推理参数、路由策略、owner、approver 和 rollback target 绑定成可审计发布包。

## Model Artifact Integrity

一句话定义：Model Artifact Integrity 是用 checksum、digest、manifest、lineage 和不可变 URI 证明上线模型、tokenizer、adapter、量化包和镜像没有被替换或篡改。

扩展口径：生产发布时还要和 Artifact Management Audit 对齐，把 deployment package、release manifest、runtime image digest、health check、safety policy、route policy、rollback target 和删除依赖检查一起纳入发布证据。

推理侧关联：线上模型发布和灰度实验也应和 Experiment Tracking Audit 对齐，把 release candidate 的 run id、offline eval run、canary 指标、A/B 配置、prompt / runtime / safety policy 版本、sample failures、成本和回滚证据写入可追踪记录，避免只知道线上模型名而无法复现发布判断。

评估平台关联：发布侧还要和 Evaluation Platform Audit 对齐，把离线批量评测、人评、LLM judge、pairwise win rate、切片回归、在线 guardrail、eval report artifact、release decision 和 rollback evidence 串成证据链。否则 release dashboard 只能看到“已通过评估”，却无法解释通过的是哪个数据版本、哪个指标口径、哪个 judge、人评质量是否稳定、线上分桶是否健康。

## Tokenizer Prompt Runtime Binding

一句话定义：Tokenizer Prompt Runtime Binding 是记录每次线上请求实际使用的 tokenizer、prompt 模板、runtime config、sampling 参数和安全策略版本，避免只知道模型名而无法复现输出。

## Offline Eval Quality Gate

一句话定义：Offline Eval Quality Gate 是候选模型发布前必须通过固定评估集、关键切片、bad case 回归、人工偏好或业务 proxy 指标的离线质量门禁。

## Safety Eval Gate

一句话定义：Safety Eval Gate 是候选模型发布前必须通过安全拒答、越权、隐私、敏感信息、jailbreak、tool misuse 和高风险场景评估的安全门禁。

## Contract Test Compatibility

一句话定义：Contract Test Compatibility 是检查 JSON schema、tool call 参数、结构化输出、错误格式、拒答格式、max context 和下游 parser 是否与生产契约兼容。

## Stable Bucket Assignment

一句话定义：Stable Bucket Assignment 是用稳定 hash 把用户、租户或会话绑定到实验分桶，保证同一实体在同一实验中持续看到同一版本。

## Canary Ramp Control

一句话定义：Canary Ramp Control 是按 1%、5%、10%、25%、50% 等步骤逐步放量，并在每一步用质量、安全、延迟、成本和业务 guardrail 决定 promote、pause、abort 或 rollback。

## AB Experiment Design

一句话定义：AB Experiment Design 是定义 baseline、treatment、随机单位、样本量、主指标、guardrail、停止规则、切片分析和显著性判断的在线实验设计。

## Sample Ratio Health

一句话定义：Sample Ratio Health 是检查 A/B 实验实际样本比例是否接近期望比例，避免分流、日志、过滤或埋点问题污染实验结论。

## Guardrail Metric Control

一句话定义：Guardrail Metric Control 是把安全、p99 延迟、成本、工具失败率、结构化输出解析失败率、投诉率和人工接管率设成发布硬护栏。

## Rollback Readiness

一句话定义：Rollback Readiness 是确认旧模型、prompt、runtime、路由、安全策略和配置仍可快速恢复，并且回滚不会破坏状态或下游契约。

## Release Audit Trace

一句话定义：Release Audit Trace 是记录谁创建版本、谁审批、放量步骤、指标、实验配置、回滚原因、policy version 和最终决策的发布审计链。

## Post Release Monitoring

一句话定义：Post Release Monitoring 是全量后继续监控质量、安全、延迟、成本、业务指标、投诉、bad case 和回归样本，防止灰度窗口外才暴露问题。

## Release Governance Gate

一句话定义：Release Governance Gate 是判断模型发布是否满足版本包完整、离线评估、安全评估、契约测试、分桶实验、guardrail、回滚、trace 和无 P0 风险的综合门禁。

## Inference Platform System Design Audit

一句话定义：Inference Platform System Design Audit 是把一份推理平台系统设计答案放进需求、请求生命周期、SLO、模型注册、路由、runtime、KV、缓存、扩缩容、保护、发布、多租户、安全、观测、成本和 trade-off 的统一审计表。

常见审计维度：Requirement Clarification 看模型类型、QPS、input / output token 分布、上下文长度、TTFT、TPOT、p99、streaming、多租户、安全、成本和可用性是否澄清；SLO Token Contract 看答案是否用 TTFT、TPOT、E2E、token load 和容量预算约束设计；Model Registry Release、Router Policy Fit、Runtime Prefill Decode 和 KV Scheduler Capacity 看模型版本、路由、prefill / decode、continuous batching、KV cache 和调度是否连成请求链路；Cache Isolation Strategy、Autoscaling Capacity Plan、Protection Policy Design、Release Governance Design、Tenant Security Governance、Observability Trace Readiness、Cost Capacity Governance 和 Tradeoff Boundary Reasoning 则检查缓存隔离、容量、稳定性、发布、多租户、安全、trace、成本和工程取舍是否闭环。

面试表达：推理平台系统设计题可以按“需求澄清 -> 目标与 SLO -> 总体架构 -> 请求链路 -> Model Registry / Router -> Cache -> Runtime / scheduler / KV -> autoscaling -> protection -> release -> tenant / security -> observability -> cost -> bottleneck and trade-off -> Inference Platform Design Gate”展开。不要只回答“API Gateway + K8s + vLLM”。

## SLO Token Contract

一句话定义：SLO Token Contract 是把 TTFT、TPOT、p99、input tokens/s、output tokens/s、active sequences、上下文长度、timeout 和成本预算写成推理平台设计的硬约束。

## KV Scheduler Capacity

一句话定义：KV Scheduler Capacity 是用活跃请求数、上下文 token、层数、KV heads、head dimension、元素字节数、KV 显存水位和调度策略判断平台是否接得住当前并发。

## Inference Platform Design Gate

一句话定义：Inference Platform Design Gate 是判断推理平台系统设计是否满足覆盖率、SLO、KV 容量、trace、发布回滚、多租户安全、成本归因和无 P0 风险的综合门禁。

## Inference Fault Diagnosis Audit

一句话定义：Inference Fault Diagnosis Audit 是把 TTFT、TPOT、p99、吞吐、错误率、timeout、KV cache、模型加载、streaming、cache、route、tool、质量和发布 diff 放进同一张推理故障排查表的方法。

核心直觉：推理故障是在线用户直接感知的问题，不能只看 QPS 或 GPU utilization。成熟平台要能从 request profile、scope slice、阶段 trace、KV cache、model runtime、route、cache、error taxonomy、tool trace、release diff 和应急动作串出根因。

常见审计维度：Inference Fault Evidence Coverage 检查证据是否齐全；Request Scope Slice 和 Trace Stage Coverage 检查 model、tenant、region、endpoint、runtime、streaming、输入 / 输出长度、版本和 gateway / router / queue / prefill / decode / stream span；TTFT Decomposition、TPOT Decode Health、Tail Latency Attribution 和 Throughput Token Capacity 检查首响、生成速度、尾延迟和 token 产能；Error Taxonomy Timeout Stage、KV Cache Pressure Guard、Model Artifact Load Readiness、Streaming Reliability、Cache Hit Key Governance、Route Trace Correctness、Tool Dependency Trace、Quality Release Diff Readiness 和 Inference Fault Diagnosis Gate 则检查错误、容量、加载、streaming、cache、路由、工具、质量、回滚和复盘。

面试表达：推理故障定位题要先判断影响范围，再拆请求链路；TTFT 看 gateway、router、cache、queue、tokenize、prefill 和首 token；TPOT 看 decode、active sequences、KV cache、显存带宽和 runtime；p99 看慢请求 trace 和切片；错误率先分类，再决定限流、降级、回滚、摘除实例或扩容。

## Inference Fault Evidence Coverage

一句话定义：Inference Fault Evidence Coverage 衡量请求画像、范围切片、阶段 trace、KV、模型 runtime、路由、cache、错误、工具、发布 diff、应急动作和诊断是否齐全。

## Request Scope Slice

一句话定义：Request Scope Slice 是按 model、tenant、region、endpoint、runtime、streaming、input length bucket、output length bucket 和 release version 切分推理故障影响范围。

## Trace Stage Coverage

一句话定义：Trace Stage Coverage 是检查 gateway、auth、router、cache、queue、tokenize、prefill、first decode、decode、stream、safety 和 downstream 是否都有 span。

## TTFT Decomposition

一句话定义：TTFT Decomposition 是把首 token 延迟拆成前置服务、cache、queue、tokenize、prefill、first decode 和 first flush，避免把所有首响变慢都归因于模型。

## TPOT Decode Health

一句话定义：TPOT Decode Health 是用 decode time、output tokens、active sequences、KV cache、GPU 负载、sampling 和 runtime 版本解释每 token 生成速度。

## Tail Latency Attribution

一句话定义：Tail Latency Attribution 是用 p99 / p50、慢请求 trace、长上下文、租户、endpoint、instance、tool call 和网络切片定位尾延迟。

## Throughput Token Capacity

一句话定义：Throughput Token Capacity 是同时检查 QPS、input tokens/s、output tokens/s、per GPU output tokens/s 和有效吞吐，而不是只看请求数。

## Error Taxonomy Timeout Stage

一句话定义：Error Taxonomy Timeout Stage 是把错误分成 4xx、429、5xx、timeout、OOM、model load、tool failure、safety block，并要求 timeout 记录发生阶段。

## KV Cache Pressure Guard

一句话定义：KV Cache Pressure Guard 是用 KV pressure、free blocks、eviction、admission reject、fragmentation 和 active sequences 判断 KV 是否成为瓶颈。

## Model Artifact Load Readiness

一句话定义：Model Artifact Load Readiness 是检查 manifest、weight checksum、tokenizer、config、runtime compatibility、quantization compatibility、权限和 warmup 是否完整。

## Streaming Reliability

一句话定义：Streaming Reliability 是观测 stream open、server abort、client disconnect、idle timeout、tokens before abort 和 cancellation cleanup。

## Cache Hit Key Governance

一句话定义：Cache Hit Key Governance 是检查 cache hit rate、cache key 是否绑定 model、tokenizer、prompt、安全策略、tenant、permission 和 TTL / eviction 策略。

## Route Trace Correctness

一句话定义：Route Trace Correctness 是检查候选模型、选择模型、路由原因、灰度比例、租户策略、endpoint 健康、fallback chain 和 trace 是否完整。

## Tool Dependency Trace

一句话定义：Tool Dependency Trace 是检查 tool schema、权限决策、timeout、retry、下游状态、错误消息和 tool call trace 是否完整。

## Quality Release Diff Readiness

一句话定义：Quality Release Diff Readiness 是把模型、prompt、runtime、route、cache、safety、sampling diff 和质量指标、线上反馈、回滚证据关联起来。

## Inference Fault Diagnosis Gate

一句话定义：Inference Fault Diagnosis Gate 是判断推理故障定位是否有完整证据、owner、runbook、应急动作、回滚、postmortem 和无 P0 风险的综合门禁。

典型硬阻断：只看 QPS 和 GPU utilization；没有 request trace；TTFT 不拆 queue / prefill；TPOT 不看 decode 和 KV；p99 没有慢请求样本；吞吐只看请求数；错误没有分类；timeout 没有阶段；KV pressure 不可见；模型加载不校验 checksum / tokenizer / runtime；streaming abort 不区分客户端和服务端；cache key 不绑定版本；route trace 缺失；tool 失败被归因成模型差；质量下降没有 release diff；没有最终 Inference Fault Diagnosis Gate。

## 部署排障常见问题

一句话定义：部署排障是从质量、延迟、吞吐、显存和稳定性多个维度定位线上推理异常。

常见质量问题：tokenizer 不一致、chat template 错误、EOS 配置错误、量化退化、采样参数不合理、prompt 被截断。

常见性能问题：batch 太小导致 GPU 利用率低，batch 太大导致排队延迟高，KV Cache 占满显存，长请求阻塞短请求，prompt cache 命中率低。

常见稳定性问题：OOM、显存碎片、请求超时、流式连接断开、不同版本模型和 tokenizer 混用。

面试表达：LLM 部署排障要先区分质量问题和系统问题，再沿 tokenizer、模板、解码、KV Cache、调度和硬件指标逐层排查。

## 本章小结

本章围绕 LLM 推理优化和部署展开。第 8 章解释了推理和解码本身，本章进一步回答：如何让模型在真实服务中更快、更省、更稳。

高频结论如下：

1. LLM serving 的难点来自动态输入输出、prefill/decode 分离、KV Cache 和高并发调度。
2. throughput 衡量整体产能，latency 衡量单请求体验，线上要同时看平均值和 P95/P99。
3. TTFT 主要受排队和 prefill 影响，TPOT 主要受 decode、KV Cache 和显存带宽影响。
4. dynamic batching 用等待窗口换吞吐，continuous batching 在 decode 过程中动态加入和移出请求。
5. KV Cache 是推理显存大户，PagedAttention 用分页式管理减少浪费并支持高吞吐 serving。
6. prompt cache 和 prefix cache 复用公共前缀，主要优化重复 prefill 成本。
7. FlashAttention 是精确 attention 的 IO-aware 实现，核心是减少 HBM 读写和中间矩阵物化。
8. decode 阶段经常是 memory bandwidth bottleneck，不只是算力瓶颈。
9. 量化能降低显存和带宽成本，但必须用校准数据和业务评测验证质量损失。
10. GPTQ 和 AWQ 都是常见 LLM PTQ 方法，前者强调近似二阶误差补偿，后者强调 activation-aware 保护重要权重。
11. vLLM、SGLang、TensorRT-LLM、TGI、Triton 等框架各有定位，选型要结合硬件、模型、并发、维护状态和延迟目标。
12. 生产排障要同时检查 tokenizer、chat template、EOS、解码参数、量化、KV Cache、调度和硬件指标。
13. 推理性能事故要把 TTFT regression、TPOT regression、KV pressure、prompt cost drift、cache effectiveness 和 serving gate 放在同一张表里判断。
14. runtime 选型要用 Model Serving Runtime Selection Audit 统一比较候选，而不是只看框架名、单点 tokens/s 或历史经验。
15. Prefill / Decode / KV 资源画像要把 token 分布、phase latency、KV 显存、prefix cache、长上下文、多租户、streaming、observability 和成本放进同一套上线门禁。
