## Distributed Training

一句话定义：distributed training 是把模型训练过程拆到多张 GPU、多个节点甚至多个集群上协同完成的方法。

为什么需要：大模型训练需要巨量参数、激活、梯度和 optimizer state。单卡通常放不下模型，也无法在可接受时间内完成训练，因此必须把数据、模型、计算和状态拆分到多卡。

核心问题：怎么拆参数，怎么拆 batch，怎么同步梯度，怎么减少显存，怎么隐藏通信，怎么让每张卡尽量不空转。

常见并行维度：数据并行、张量并行、流水线并行、专家并行、序列并行，以及 ZeRO/FSDP 这类状态分片方法。真实大模型训练通常是多种并行方式组合使用。

面试表达：分布式训练的本质是用通信换容量和速度。所有方案都在计算、显存、通信和工程复杂度之间做权衡。

## Data Parallelism

一句话定义：data parallelism 是每张 GPU 保存一份完整模型副本，处理不同数据 batch，并在反向传播后同步梯度的并行方式。

工作流程：每张卡拿到不同 mini-batch；各自前向和反向；得到本地梯度；通过 all-reduce 聚合梯度；每张卡用相同梯度更新参数。

优点：概念简单，工程成熟，适合模型能放进单卡但数据和训练时间压力大的场景。

局限：每张 GPU 都要保存完整参数、梯度和 optimizer state。模型很大时，数据并行无法解决单卡显存放不下的问题。

面试表达：数据并行提升的是样本吞吐，不拆模型本身；如果单卡放不下模型，需要张量并行、流水并行或参数分片。

## DDP

一句话定义：DDP 是 Distributed Data Parallel，PyTorch 中常用的数据并行训练框架。

工作方式：每个进程通常绑定一张 GPU，每个进程持有一份模型副本；反向传播时，DDP 把梯度按 bucket 聚合，并通过 all-reduce 同步。

为什么高效：DDP 可以在反向传播过程中边计算梯度边启动通信，让部分通信和计算重叠，减少等待时间。

适用场景：模型、梯度和 optimizer state 能放入单卡，但希望多卡加速训练。

常见坑：不同 rank 数据不一致、随机种子不一致、某些参数未参与 loss 导致同步异常、batch norm 或梯度累积配置错误。

面试表达：DDP 是成熟的数据并行方案，核心是每卡完整模型副本加梯度 all-reduce。

## Gradient Synchronization

一句话定义：gradient synchronization 是多卡训练中把不同设备上的梯度聚合成一致结果的过程。

为什么需要：数据并行中每张卡看到不同样本，本地梯度不同。如果不同步，各卡参数会逐渐分叉，训练就不再等价于一个全局 batch。

常见方式：all-reduce 对梯度求和或求平均；reduce-scatter 把聚合后的梯度分片留在不同设备；parameter server 架构集中聚合。

影响：梯度同步是分布式训练的重要通信开销，模型越大、通信带宽越低，同步成本越明显。

面试表达：数据并行的关键通信就是梯度同步，它保证各个模型副本更新一致。

## All-Reduce

一句话定义：all-reduce 是聚合多个设备上的张量，并把聚合结果同步回所有设备的通信操作。

典型用途：DDP 中同步梯度。每张卡都有一份梯度，all-reduce 后每张卡都得到全局聚合梯度。

常见算法：ring all-reduce、tree all-reduce。不同算法在带宽利用、延迟和节点拓扑上有不同表现。

成本来源：通信量与张量大小相关，也受 GPU 间互联、跨节点网络、bucket 大小和通信库实现影响。

面试表达：all-reduce 是数据并行的核心通信原语，用来让所有 rank 拿到一致的聚合结果。

## Reduce-Scatter

一句话定义：reduce-scatter 是先对多个设备上的张量做聚合，再把聚合结果按分片分发到不同设备。

与 all-reduce 的关系：all-reduce 可以看成 reduce-scatter 加 all-gather。先聚合并分片，再把分片收集回所有设备。

用途：ZeRO、FSDP 等分片训练中常用 reduce-scatter 同步梯度，因为每张卡只需要保留自己负责的梯度分片。

面试表达：reduce-scatter 适合状态分片场景，它不像 all-reduce 那样让每张卡都保存完整结果。

## All-Gather

一句话定义：all-gather 是从多个设备收集张量分片，并让每个设备都得到完整张量的通信操作。

用途：FSDP 或 ZeRO-3 中，参数被分片存储。某一层前向计算前，需要 all-gather 收集该层完整参数，计算结束后再释放。

代价：all-gather 会引入额外通信和显存峰值，因此需要和计算重叠、按层调度和及时释放。

面试表达：all-gather 常用于参数分片训练中临时恢复完整参数，是 FSDP/ZeRO-3 的关键通信。

## Tensor Parallelism

一句话定义：tensor parallelism 是把单层内部的大矩阵计算切分到多张 GPU 上并行完成。

为什么需要：当单层矩阵太大或单卡算力不足时，可以把矩阵乘法拆到多卡。典型用于 Transformer 的 attention 和 MLP 层。

常见切法：按输出维度切的 column parallel，按输入维度切的 row parallel。Megatron-LM 中常把 MLP 和 attention 投影按这种方式拆分。

优点：能拆单层参数和计算，适合超大模型。缺点：层内通信频繁，对 GPU 互联带宽要求高。

面试表达：张量并行拆的是层内矩阵，解决单层太大或算力不足的问题，但代价是频繁通信。

## Column Parallel

一句话定义：column parallel 是按权重矩阵输出维度切分，让不同 GPU 计算不同输出通道。

例子：线性层 `Y = XW` 中，如果按 `W` 的列切分，每张 GPU 得到一部分输出 `Y_i`，最后可以拼接成完整输出。

适用位置：MLP 的上投影、attention 的 QKV 投影等经常可以使用 column parallel。

通信特点：有些位置可以延迟 gather，让后续 row parallel 消化分片输出，从而减少不必要通信。

面试表达：column parallel 切输出通道，天然产生分片输出。

## Row Parallel

一句话定义：row parallel 是按权重矩阵输入维度切分，让不同 GPU 处理输入的不同部分，再把结果求和。

例子：线性层 `Y = XW` 中，如果按 `W` 的行切分，输入 `X` 也要对应切分。每张卡计算局部输出，最后通过 all-reduce 得到完整 `Y`。

适用位置：MLP 的下投影、attention 的输出投影等常用 row parallel。

通信特点：通常需要一次 all-reduce 聚合局部输出。

面试表达：row parallel 切输入通道，局部结果需要通信求和。

## Pipeline Parallelism

一句话定义：pipeline parallelism 是把模型不同层放到不同 GPU，并用 micro-batch 形成流水线执行。

为什么需要：当模型层数多、整模型放不进单卡时，可以让每张 GPU 只保存一部分层。

基本流程：把一个 global batch 切成多个 micro-batch。第一个 micro-batch 在 stage 1 完成后传给 stage 2，同时 stage 1 开始处理第二个 micro-batch。

优点：显著降低单卡参数显存。缺点：有 pipeline bubble，前后向调度复杂，stage 之间负载不均会拖慢整体速度。

面试表达：流水并行拆的是层，核心问题是如何用 micro-batch 填满流水线，减少 bubble。

## Micro-batch

一句话定义：micro-batch 是把一个较大的 batch 切成多个小批次，用于梯度累积或流水线并行调度。

在流水并行中：micro-batch 数越多，流水线越容易填满，bubble 占比越小。但 micro-batch 太小会降低单卡计算效率。

在梯度累积中：多个 micro-batch 的梯度累积后再执行一次 optimizer step，可以模拟更大的 global batch。

面试表达：micro-batch 是流水并行和梯度累积的基本调度单位。

## Pipeline Bubble

一句话定义：pipeline bubble 是流水线并行中某些 stage 因等待数据或等待梯度而空闲的时间。

为什么出现：流水线开始时后面的 stage 没有数据，结束时前面的 stage 已经无事可做。前后向阶段切换也会造成空转。

影响因素：pipeline stage 数、micro-batch 数、调度策略、各 stage 计算负载是否均衡。

优化方向：增加 micro-batch 数、使用 1F1B 调度、合理切分层、平衡每个 stage 的计算量和参数量。

面试表达：pipeline bubble 是流水并行的主要效率损失，micro-batch 越多通常 bubble 比例越低。

## 1F1B

一句话定义：1F1B 是 one-forward-one-backward 的流水线调度方式，稳定阶段中每个设备交替执行一个前向和一个反向 micro-batch。

为什么提出：朴素调度可能先跑完所有前向再跑所有反向，激活显存高，等待时间也大。1F1B 能降低激活峰值并提高流水线利用率。

特点：有 warmup、steady 和 cooldown 阶段。稳定阶段尽量让各 stage 持续工作。

面试表达：1F1B 是减少流水线空转和激活显存的常见调度策略。

## ZeRO

一句话定义：ZeRO 是通过分片 optimizer states、gradients 和 parameters，降低数据并行显存冗余的方法。

为什么提出：普通数据并行中，每张卡都保存完整参数、梯度和 optimizer state。以 Adam 为例，optimizer state 往往比参数本身还占显存。

核心思想：数据并行保留全局 batch 扩展能力，但把冗余训练状态切到不同 GPU 上，让每张卡只保存一部分。

三个阶段：ZeRO-1 分片 optimizer states；ZeRO-2 进一步分片 gradients；ZeRO-3 进一步分片 parameters。

面试表达：ZeRO 不是新的优化器，而是消除数据并行中训练状态冗余的显存优化方法。

## ZeRO-1

一句话定义：ZeRO-1 是只分片 optimizer states 的 ZeRO 阶段。

保存什么：参数和梯度仍然每卡完整保存，但 Adam 的一阶、二阶动量等 optimizer state 被切分到不同卡。

收益：实现相对简单，对通信模式影响较小，能减少 optimizer state 的冗余显存。

局限：参数和梯度仍完整复制，超大模型下节省不够。

面试表达：ZeRO-1 先切 optimizer state，是 ZeRO 中侵入性较低的一步。

## ZeRO-2

一句话定义：ZeRO-2 是在 ZeRO-1 基础上进一步分片 gradients。

保存什么：optimizer states 和 gradients 分片，参数仍然每卡完整保存。

通信特点：反向传播后通过 reduce-scatter 等方式只保留每张卡负责的梯度分片。

收益：比 ZeRO-1 进一步降低显存，适合参数仍能复制但梯度和 optimizer state 压力较大的场景。

面试表达：ZeRO-2 切 optimizer state 和梯度，但参数仍完整复制。

## ZeRO-3

一句话定义：ZeRO-3 是同时分片 optimizer states、gradients 和 parameters 的 ZeRO 阶段。

保存什么：每张卡只长期保存部分参数、梯度和 optimizer state。前向或反向需要某层参数时，通过 all-gather 临时收集，计算后释放。

收益：显存节省最大，可以训练远超单卡容量的模型。

代价：通信复杂度和通信量增加，性能依赖调度、预取、通信计算重叠和网络带宽。

面试表达：ZeRO-3 显存最省，但通信最重，是用通信换显存的典型方案。

## FSDP

一句话定义：FSDP 是 Fully Sharded Data Parallel，把参数、梯度和 optimizer state 分片到多张 GPU 的训练方式。

与 ZeRO-3 的关系：FSDP 可以理解为 PyTorch 生态中类似 ZeRO-3 思想的全分片数据并行实现。

工作方式：参数平时以分片形式存在；进入某个模块前 all-gather 完整参数；完成前向/反向后释放完整参数；梯度通过 reduce-scatter 回到分片形式。

优势：显著降低单卡显存，支持更大模型训练；与 PyTorch 生态结合较深。

挑战：wrap 策略、prefetch、混合精度、checkpoint 保存、通信重叠都会影响效果。

面试表达：FSDP 的核心是按模块动态 all-gather 参数，用后释放，从而降低长期显存占用。

## Activation Checkpointing

一句话定义：activation checkpointing 是少保存中间激活，在反向传播时重新计算，用计算换显存的方法。

为什么需要：训练中除了参数和 optimizer state，激活也会占大量显存，尤其是长序列、大 batch 和深层模型。

工作方式：前向时只保存部分 checkpoint 激活；反向时从最近 checkpoint 重新执行部分前向，恢复需要的中间激活。

优点：显著降低激活显存。缺点：增加额外计算，训练时间变长。

面试表达：activation checkpointing 是典型的 compute-memory tradeoff，用重算换显存。

## Gradient Accumulation

一句话定义：gradient accumulation 是连续处理多个 micro-batch，累积梯度后再执行一次参数更新。

为什么需要：显存放不下大 batch 时，可以用多个小 micro-batch 模拟较大的 global batch。

等价条件：如果 loss 缩放、随机性、batch norm 等处理一致，梯度累积近似等价于更大的 batch。但 dropout、数据顺序和归一化层可能带来细微差异。

代价：optimizer step 频率下降，单次更新前需要更多前反向计算，wall-clock 可能变长。

面试表达：梯度累积解决的是显存不足下的有效 batch size 问题，不直接提高吞吐。

## Mixed Precision Training

一句话定义：mixed precision training 是训练中混合使用 FP32、FP16、BF16 等精度，以降低显存、提高速度并保持数值稳定。

常见方式：大部分矩阵计算使用 FP16/BF16，部分敏感操作或主权重使用 FP32。

收益：减少显存和带宽，利用 Tensor Core 加速。

风险：低精度可能导致 overflow、underflow 或梯度精度不足。BF16 因动态范围大，通常比 FP16 更稳定。

面试表达：混合精度不是简单全低精度，而是在速度、显存和数值稳定之间折中。

## Loss Scaling

一句话定义：loss scaling 是在 FP16 训练中放大 loss 或梯度，避免小梯度下溢为 0 的技巧。

为什么需要：FP16 动态范围有限，小梯度可能无法表示。把 loss 乘以 scale 后反向传播，再在更新前把梯度除回去，可以减少 underflow。

类型：静态 loss scaling 使用固定 scale；动态 loss scaling 根据是否出现 overflow 自动调整。

注意：BF16 动态范围接近 FP32，通常不太需要 loss scaling。

面试表达：loss scaling 主要是 FP16 训练的数值稳定技巧，用于缓解梯度下溢。

## Training Stability

一句话定义：training stability 是训练过程在数据、学习率、梯度、精度、mask、checkpoint 和分布式状态上保持可观测、可复现、可恢复的能力。

典型信号：loss spike、NaN / Inf、gradient norm 突增、AMP overflow、有效 label token 异常、学习率曲线跳变、train / eval 背离和 rank 间 loss 差异。

工程做法：每 step 记录真实 lr、loss、grad norm、overflow、label token 数、batch id、rank loss 和 checkpoint / resume 状态；异常时先保存 batch 和日志，再做最小复现。

面试表达：训练稳定性不是单个超参问题，而是数据、优化、数值和分布式系统的联合可观测性问题。

## Loss Spike Audit

一句话定义：loss spike audit 是对 loss 突然升高的 step 进行 batch、学习率、梯度、overflow、mask 和 rank 日志定位的审计流程。

关键证据：第一次 spike step、对应 batch id、样本来源、sequence length、有效 label token、grad norm、真实 lr、overflow 状态和 per-rank loss。

面试表达：loss spike 的第一原则是保存证据。没有 step 和 batch id，就很难判断是数据、数值还是系统问题。

## Non-Finite Loss Rate

一句话定义：non-finite loss rate 是训练 step 中 loss、gradient 或参数出现 NaN / Inf 的比例。

为什么重要：一旦出现非有限值，后续状态通常会被污染。排查重点是第一次出现的位置，而不是 NaN 扩散后的所有 step。

常见原因：学习率过大、FP16 overflow、attention mask 全 `-inf`、loss 分母为 0、梯度爆炸、异常数据和 optimizer state 损坏。

面试表达：NaN / Inf 是硬门禁。定位顺序通常是输入、logits、loss、grad、param 和 optimizer state。

## Effective Label Tokens

一句话定义：effective label tokens 是一个 batch 中真正参与 loss 的 label token 数。

为什么重要：SFT、packing 和 padding 场景下，有效 label token 数直接决定 loss 分母和训练信号。它突然过低或为 0，通常说明 mask、padding、截断或数据格式出错。

工程做法：按 batch、rank、任务类型统计有效 label token 数，并把异常低值写入训练日志和门禁。

面试表达：训练 loss 不能只看数值，还要看这个 loss 是由多少有效 token 计算出来的。

## LR Continuity

一句话定义：LR continuity 是实际生效 learning rate 与 scheduler 根据 global step 预期 learning rate 的连续一致性。

常见事故：checkpoint resume 后 scheduler step 错、global step 错、gradient accumulation 改了但 warmup / decay 没改、optimizer state 没恢复。

面试表达：不要只看配置里的 lr，要记录每个 optimizer step 的真实 lr。resume 后第一步尤其要检查连续性。

## Rank Loss Skew

一句话定义：rank loss skew 是分布式训练中不同 rank 的 loss 差异过大。

常见原因：某个 rank 的数据 shard 异常、随机种子不同、mask 错、batch 长度极端、硬件或通信问题。

工程做法：不要只记录 rank 0。训练日志至少要能抽样记录 per-rank loss、batch id、样本数量和有效 label token 数。

面试表达：rank loss skew 高时，排查方向不是全局调参，而是先找哪个 rank、哪个 batch、哪个 shard 出了问题。

## Training Stability Gate

一句话定义：training stability gate 是把 non-finite、loss spike、gradient norm、overflow、effective label tokens、LR continuity、rank loss skew 和 checkpoint resume 状态合并成训练稳定性门禁。

用途：在大训练前的小规模 dry run、resume 演练、混合精度切换、数据版本变更和分布式扩容后，快速判断训练链路是否可继续放大。

面试表达：训练稳定性门禁要输出 failed gates 和 first bad step。它不是为了让训练永远不波动，而是让异常可定位、可复现、可回滚。

## Distributed Incident

一句话定义：distributed incident 是分布式训练中由 rank 状态、数据 shard、collective 调用、通信、checkpoint、resume 或并行调度不一致引发的训练事故。

典型现象：训练 hang、NCCL timeout、吞吐骤降、某个 rank loss 异常、checkpoint 恢复后曲线跳变、pipeline stage 利用率低或多节点扩容后性能反而下降。

工程做法：不要只看 rank 0。每个 rank 至少记录 step、phase、batch id、token 数、loss、step time、data time、communication time、collective 序列、checkpoint shard 和 resume 状态。

面试表达：分布式事故的核心是系统一致性。排查要先证明所有 rank 是否处在同一个 step、同一个 collective 序列和同一个训练状态上。

## Rank Step Alignment

一句话定义：rank step alignment 是检查所有 rank 在同一时间窗口内是否到达同一个训练 step。

常见事故：某个 rank 卡在 dataloader、某个 rank 提前结束、异常处理只在部分 rank 执行、resume 后 global step 不一致。

面试表达：训练卡死时，先看 per-rank step 和 phase。rank 0 正常不代表其他 rank 正常。

## Collective Mismatch

一句话定义：collective mismatch 是不同 rank 的 all-reduce、all-gather、reduce-scatter、broadcast 等 collective 调用顺序或次数不一致。

常见来源：按 rank 写条件分支、某些 rank batch 为空、只在 rank 0 做保存或评估但同步不完整、异常分支没有让所有 rank 共同退出。

面试表达：collective 是分布式训练的同步契约，某个 rank 少一次 collective 就可能让其他 rank 永久等待。

## Token Shard Imbalance

一句话定义：token shard imbalance 是不同 rank 分到的 token 数或序列长度差异过大。

为什么重要：大模型训练的真实工作量更接近 token 数和序列长度，而不是样本条数。按样本均匀可能仍然让某些 rank 分到超长 batch。

面试表达：数据并行要检查 per-rank token 数、平均长度、来源分布和 batch sampler，而不是只检查样本数。

## Straggler Ratio

一句话定义：straggler ratio 是最慢 rank 的 step time 与典型 rank step time 的比值。

常见原因：数据读取慢、网络差、GPU 性能异常、batch 长度极端、共享存储抖动或某个节点环境不同。

面试表达：straggler 会拖慢全局同步。排查时要把 step time 拆成 data、compute 和 communication。

## Communication Ratio

一句话定义：communication ratio 是通信时间占单步训练时间的比例。

用途：判断吞吐瓶颈是否主要来自 all-reduce、all-gather、reduce-scatter、跨节点通信、bucket 配置或计算通信重叠不足。

面试表达：通信占比高时，不要只调 NCCL 参数。还要看 batch size、bucket、拓扑、并行切分、overlap 和 straggler。

## Checkpoint Shard Coverage

一句话定义：checkpoint shard coverage 是实际保存成功的分布式 checkpoint shard 占预期 shard 的比例。

常见事故：只保存 rank 0 权重、漏掉 optimizer state、FSDP / ZeRO shard 缺失、scheduler / scaler / sampler 状态丢失、保存过程中部分 rank 失败。

面试表达：没有定期 restore test 的 checkpoint 不可靠。分布式 checkpoint 要验证 shard、状态和 global step，而不是只看目录是否存在。

## Resume Continuity

一句话定义：resume continuity 是 checkpoint 加载后 global step、learning rate、optimizer state、scheduler、AMP scaler 和数据位置是否与保存前连续。

常见事故：只恢复模型权重、scheduler 从头开始、optimizer 动量丢失、sampler 位置错、global step 偏移。

面试表达：resume 后 loss 跳变时，先检查训练状态完整性和固定 batch loss，而不是立刻改学习率或换数据。

## Pipeline Bubble Ratio

一句话定义：pipeline bubble ratio 是 pipeline parallel 中 stage 空转时间占总调度窗口的近似比例。

影响因素：stage 数、micro-batch 数、stage 计算量是否均衡和 pipeline schedule。micro-batch 太少或 stage 切分不均都会让 bubble 吃掉收益。

面试表达：pipeline parallel 不是切得越细越好。要用 micro-batch 填充流水线，并平衡每个 stage 的计算时间。

## Distributed Incident Gate

一句话定义：distributed incident gate 是把 rank step alignment、collective mismatch、token shard imbalance、straggler ratio、communication ratio、checkpoint shard coverage、resume continuity 和 pipeline bubble ratio 合并成分布式训练事故门禁。

用途：在单机多卡到多机多卡扩容、FSDP / ZeRO 切换、checkpoint 改造、resume 演练和 pipeline 并行调整后，快速判断训练是否可以继续放大。

面试表达：分布式事故门禁要输出 failed gates 和对应 rank / shard / stage。它的价值是把“我感觉卡了”变成可复现的 per-rank 证据链。

## Offloading

一句话定义：offloading 是把部分参数、梯度、optimizer state 或激活转移到 CPU 内存或 NVMe，以缓解 GPU 显存压力。

价值：让有限 GPU 显存训练更大模型。

代价：CPU/GPU 或 NVMe/GPU 数据传输慢，会显著增加训练时间。offload 的收益取决于能否把传输和计算重叠。

适用场景：显存极度紧张、预算有限、可以接受训练变慢的场景。

面试表达：offloading 用速度换容量，是显存不够时的兜底方案，不是最高性能训练的首选。

## Parameter Sharding

一句话定义：parameter sharding 是把模型参数分片存储到不同设备上，避免每张 GPU 保存完整参数。

用途：ZeRO-3、FSDP 和部分模型并行方案都会使用参数分片。

挑战：计算某一层时通常需要完整参数或对应分片参与，因此要设计 all-gather、预取和释放策略。

面试表达：参数分片降低长期显存，但会带来参数收集和通信调度问题。

## Optimizer State Sharding

一句话定义：optimizer state sharding 是把优化器状态切分到不同设备上，减少 Adam 等优化器带来的显存冗余。

为什么重要：Adam 通常为每个参数维护一阶和二阶动量，再加上 master weights，训练状态显存可能是参数显存的数倍。

典型方法：ZeRO-1 就是分片 optimizer states；FSDP 也会配合优化器状态分片保存和加载。

面试表达：优化器状态是训练显存大户，切它通常能带来明显收益。

## Communication Overlap

一句话定义：communication overlap 是让通信和计算同时进行，以减少分布式训练中的等待时间。

例子：DDP 在反向传播时，某些层梯度刚算完就启动 all-reduce，同时继续计算更前面层的梯度。FSDP 可以预取下一层参数，隐藏 all-gather 延迟。

关键条件：通信依赖关系、bucket 大小、网络带宽、计算耗时和框架调度都要合适。

面试表达：通信重叠不是减少通信量，而是把通信隐藏到计算背后，提升整体效率。

## NCCL

一句话定义：NCCL 是 NVIDIA Collective Communications Library，用于 GPU 间高效集合通信。

常见操作：all-reduce、all-gather、reduce-scatter、broadcast。

为什么重要：多 GPU 训练效率高度依赖通信库。NCCL 会根据硬件拓扑、NVLink、PCIe、InfiniBand 等选择通信策略。

常见排障：rank 卡住、网络配置错误、拓扑性能差、超时、不同节点环境不一致。

面试表达：NCCL 是 NVIDIA GPU 分布式训练的核心通信库，很多多卡性能问题最终都要看通信链路。

## Node

一句话定义：node 是分布式训练中的一台机器，通常包含多张 GPU、CPU、内存和网络接口。

单机多卡：通信通常依赖 NVLink、NVSwitch 或 PCIe。

多机多卡：还需要跨节点网络，例如 InfiniBand 或 RoCE，跨节点通信往往比节点内通信更慢。

面试表达：分布式训练要区分节点内通信和节点间通信，后者通常是更难优化的瓶颈。

## Rank

一句话定义：rank 是分布式训练中每个进程的唯一编号。

常见概念：global rank 表示全局编号；local rank 表示当前节点内编号；rank 0 常用于日志、保存 checkpoint 或协调任务。

为什么重要：数据切分、设备绑定、通信分组和日志保存都依赖 rank。

面试表达：rank 是分布式训练进程身份，理解 rank 才能理解多进程如何协同。

## World Size

一句话定义：world size 是参与同一个分布式训练作业的总进程数，通常等于总 GPU 数。

例子：4 台机器，每台 8 张 GPU，如果每张 GPU 一个进程，则 world size 为 32。

影响：world size 决定数据并行规模、通信组大小和 global batch size。

面试表达：world size 是全局并行进程数量，不是单机 GPU 数。

## Global Batch Size

一句话定义：global batch size 是一次参数更新实际使用的总样本数。

常见公式：

```text
global batch size = micro batch size × gradient accumulation steps × data parallel world size
```

重要性：global batch size 影响训练稳定性、学习率设置、收敛速度和泛化。扩大并行规模时，通常要同步调整 batch 和 learning rate schedule。

面试表达：多卡训练中要区分 micro batch、per-device batch 和 global batch，很多训练不稳定来自 batch 配置理解错误。

## MFU

一句话定义：MFU 是 Model FLOPs Utilization，衡量训练中硬件理论算力有多少真正用于模型计算。

为什么比 GPU 利用率更有意义：GPU 利用率高只说明 GPU 忙，不代表忙在有效模型计算上。MFU 更关注实际完成的模型 FLOPs 与硬件理论峰值的比例。

影响因素：模型结构、batch size、序列长度、kernel 效率、通信开销、重算、padding 浪费和数据加载。

面试表达：MFU 是大模型训练效率的关键指标，比单纯看 `nvidia-smi` 的 GPU utilization 更有参考价值。

## GPU Utilization

一句话定义：GPU utilization 是 GPU 在采样窗口内有多少时间处于忙碌状态的指标。

局限：GPU utilization 高不代表训练效率高。可能是低效 kernel、通信等待、内存搬运或小 batch 导致的低效忙碌。

如何结合使用：排查性能时同时看 MFU、tokens/sec、step time、通信时间、显存占用、数据加载耗时和网络指标。

面试表达：GPU utilization 是粗粒度指标，不能单独用来判断大模型训练效率。

## Training Efficiency Audit

一句话定义：Training Efficiency Audit 是把 GPU utilization、tokens/s、step time、MFU、HFU、通信、I/O、checkpoint、rank skew、padding、重计算和 loss 正确性放进同一张训练效率检查表的方法。

核心直觉：GPU utilization 只说明 GPU 在采样窗口内有没有 kernel 在跑，不说明这些 kernel 是否高效，也不说明计算是否转化成有效模型训练。训练效率要同时回答三件事：训练快不快，硬件是否被有效使用，训练是否仍然正确。

常见维度：Tokens Throughput Accounting 看 global tokens/s 是否按真实非 padding token 统计；Step Time Breakdown Coverage 看 dataloader、H2D、forward、backward、communication、optimizer、checkpoint 和 misc 是否拆开；GPU Utilization Interpretation 避免把 GPU 忙碌当成最终结论；MFU Estimation 和 HFU Estimation 区分模型有效 FLOPs 与硬件实际 FLOPs；Model FLOPs Accounting 和 Hardware Peak Accounting 说明 FLOPs 与峰值口径；Communication Ratio Tracking、IO Dataloader Tracking 和 Checkpoint Overhead Tracking 检查通信、数据供给和保存尖刺；Rank Skew Detection 看最慢 rank 是否拖住全局 step；Scaling Efficiency Tracking 看多卡是否接近线性扩展；Padding Waste Awareness、Recompute Overhead Awareness 和 Loss Correctness Coupling 则防止“跑得快但浪费多或训练错”。

常见公式：端到端 token 吞吐可以写成

```math
Q_{\mathrm{tok}}=\frac{B_{\mathrm{global}}S_{\mathrm{eff}}}{T_{\mathrm{step}}}
```

MFU 可以写成

```math
U_{\mathrm{mfu}}=\frac{F_{\mathrm{model}}/T_{\mathrm{step}}}{N_{\mathrm{gpu}}F_{\mathrm{peak}}}
```

HFU 可以写成

```math
U_{\mathrm{hfu}}=\frac{F_{\mathrm{hw}}/T_{\mathrm{step}}}{N_{\mathrm{gpu}}F_{\mathrm{peak}}}
```

其中，`F_model` 是模型有效 FLOPs，`F_hw` 是硬件实际执行 FLOPs，`F_peak` 是单 GPU 理论峰值 FLOPs/s。

面试表达：训练效率题要先说 GPU utilization 只是入口，再用 tokens/s、step time 分解、MFU/HFU、通信占比、I/O 占比、rank skew、扩展效率和 loss 曲线给出证据链。

## Training Efficiency Gate

一句话定义：Training Efficiency Gate 是判断一个训练任务是否达到效率、扩展性和正确性最低要求的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{eff}}=\mathbf{1}\left[U_{\mathrm{mfu}}\ge \tau_{\mathrm{mfu}} \land E_n\ge \tau_{\mathrm{scale}} \land R_{\mathrm{comm}}\le \rho_{\mathrm{comm}} \land R_{\mathrm{io}}\le \rho_{\mathrm{io}} \land P_0=0\right]
```

其中，`E_n` 是多卡扩展效率，`R_comm` 是通信占比，`R_io` 是数据与 checkpoint I/O 占比，`P_0` 是 P0 级训练风险数量。

典型硬阻断：只报告 GPU utilization；不统计真实 token 数；没有 step time 分解；MFU / HFU 口径不清；硬件峰值按错误精度计算；通信或 I/O 占比过高；rank skew 严重；padding 或重计算浪费未量化；checkpoint 周期性阻塞；loss、梯度或数值稳定性异常但仍把吞吐当成功。

面试表达：训练效率门禁的目标不是追求单一指标最高，而是证明“有效 token 吞吐、模型有效 FLOPs、硬件利用、通信/I/O、扩展效率和训练正确性”同时过线。

## Task Profile Audit

一句话定义：Task Profile Audit 是把预训练、SFT、RLHF、评估、推理、RAG、Agent 和多模态任务的资源、调度、观测、成本、血缘和治理差异转成可检查指标表的方法。

核心直觉：AI Infra 服务的是不同工作负载，而不是抽象的“模型”。预训练重点是大规模训练吞吐、通信、数据供给和 checkpoint；SFT 重点是高频实验、数据版本和评估；RLHF 重点是 SFT、reward model、policy、reference model、rollout 和偏好数据的多阶段编排；评估重点是 benchmark、模型版本、样本级结果和可复现性；推理重点是 TTFT、TPOT、KV cache、batching、扩缩容和 SLO；RAG 重点是知识库、embedding、索引、retrieval、rerank、权限和引用；Agent 重点是工具、状态、trace、权限、预算和失败恢复。

常见公式：工作负载样本可以写成

```math
a_i=(w_i,s_i,r_i,c_i,m_i,n_i,d_i,l_i,o_i,g_i,z_i)
```

资源画像可以写成

```math
\mathbf{v}_i=(C_i,M_i,N_i,D_i,L_i,R_i,K_i,S_i)
```

其中，`C_i` 是 compute，`M_i` 是 memory，`N_i` 是 network，`D_i` 是 data / storage，`L_i` 是 latency，`R_i` 是 reliability，`K_i` 是 cost，`S_i` 是 security / governance。

面试表达：任务画像题不要只背“预训练重、推理快、Agent 复杂”，而要说明每类任务该进入什么资源池、用什么调度策略、看什么指标、如何算成本、失败后如何恢复。

## Workload Type Classification

一句话定义：Workload Type Classification 是先判断任务属于预训练、SFT、RLHF、评估、推理、RAG、Agent、多模态或混合链路，再决定资源和平台策略的步骤。

常见坑：把所有任务都标成 generic AI job，会导致预训练缺 gang scheduling、推理缺 SLO、Agent 缺 trace 和权限边界。

## Resource Shape Completeness

一句话定义：Resource Shape Completeness 是检查任务是否同时描述 compute、memory、network、data / storage、latency、reliability、cost 和 security / governance。

面试表达：资源画像不完整时，平台很容易只按 GPU 数量调度，忽略 KV cache、checkpoint、数据更新、人工评估和权限审计。

## Pretraining Profile Accuracy

一句话定义：Pretraining Profile Accuracy 是检查预训练画像是否覆盖 tokens/s、MFU、step time、通信、数据供给、checkpoint、拓扑感知和容错。

典型硬阻断：没有 checkpoint 策略、没有数据版本、没有拓扑感知、只报告 GPU 数量或 GPU utilization。

## SFT Iteration Profile Accuracy

一句话定义：SFT Iteration Profile Accuracy 是检查 SFT 是否覆盖数据集版本、prompt / template 版本、adapter artifact、自动评估和可复现实验。

面试表达：SFT 不是缩小版预训练，核心诉求是高频、可比较、可回滚的实验迭代。

## RLHF Pipeline Profile Accuracy

一句话定义：RLHF Pipeline Profile Accuracy 是检查 SFT model、reward model、policy model、reference model、rollout、偏好数据和安全评估是否被统一追踪。

常见坑：只写 PPO / DPO 算法名，不统计 rollout 吞吐、reward 评分成本、多模型显存和阶段间 artifact。

## Evaluation Reproducibility Profile

一句话定义：Evaluation Reproducibility Profile 是检查评估任务是否记录 benchmark 版本、模型版本、judge 口径、样本级输出、随机种子和回放条件。

面试表达：评估平台的关键不是跑完，而是可比较、可复现、可解释。

## Serving SLO Profile

一句话定义：Serving SLO Profile 是检查在线推理是否覆盖 TTFT、TPOT、p95 / p99、错误率、队列、KV cache、batching、自动扩缩容和降级。

常见公式：SLO 通过率可以写成

```math
R_{\mathrm{slo}}=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[TTFT_i\le \tau_{\mathrm{ttft}}\land TPOT_i\le \tau_{\mathrm{tpot}}\land E_i=0]
```

## RAG Freshness Retrieval Profile

一句话定义：RAG Freshness Retrieval Profile 是检查 RAG 是否覆盖文档更新延迟、embedding 版本、索引版本、召回、rerank、引用、权限过滤和 RAG trace。

常见坑：只估算 LLM token 成本，忽略文档解析、索引构建、检索质量和引用可信度。

## Agent Tool Runtime Profile

一句话定义：Agent Tool Runtime Profile 是检查 Agent 是否覆盖工具调用、任务状态、trace、最大步数、工具预算、权限、人审和失败恢复。

面试表达：Agent 成本和可靠性不能只看一次模型调用，要看完整调用链和工具执行结果。

## Multimodal Resource Profile

一句话定义：Multimodal Resource Profile 是检查图像、音频、视频、OCR 等多模态任务是否覆盖输入体积、解码预处理、batch 组装、显存、存储吞吐和质量评估。

常见坑：把视频或文档任务当普通文本请求，只算 LLM token，不算文件解析、帧采样、OCR / ASR 和存储吞吐。

## Scheduler Policy Fit

一句话定义：Scheduler Policy Fit 是检查不同任务是否使用匹配的调度策略，例如预训练 gang scheduling、SFT 配额队列、评估批处理、推理 SLO 队列和 Agent deadline / budget 控制。

## Observability Metric Fit

一句话定义：Observability Metric Fit 是检查 metrics、logs、traces、events 是否覆盖当前任务真正的瓶颈，而不是只画通用 GPU utilization。

## Cost Model Fit

一句话定义：Cost Model Fit 是检查成本模型是否拆开 GPU、存储、网络、人工、API / judge / tool 调用和失败重试。

成本可以写成：

```math
K_i=K_{\mathrm{gpu},i}+K_{\mathrm{storage},i}+K_{\mathrm{network},i}+K_{\mathrm{human},i}+K_{\mathrm{api},i}
```

## Artifact Lineage Fit

一句话定义：Artifact Lineage Fit 是检查数据、代码、镜像、模型、adapter、checkpoint、eval report 和 deployment package 是否能从任务画像中追溯。

## Safety Governance Fit

一句话定义：Safety Governance Fit 是检查任务是否有 owner、权限、PII 策略、风险评审、审计日志和高风险动作治理。

## Task Profile Gate

一句话定义：Task Profile Gate 是判断一个任务画像是否达到平台设计和上线讨论最低门槛的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{task}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{slo}}\ge \tau_{\mathrm{slo}} \land P_0=0\right]
```

其中，`C_j` 是第 `j` 个审计指标覆盖率，`R_slo` 是在线请求 SLO 通过率，`P_0` 是 P0 级风险数量。

典型硬阻断：任务类型泛化、资源画像缺维度、预训练缺 checkpoint、SFT 缺数据版本、RLHF 缺 rollout、评估不可复现、推理缺 TTFT / TPOT、RAG 忽略新鲜度、Agent 缺 trace、多模态 payload 无上限、调度只有一个队列、成本只按 token 估算、artifact lineage 缺失或安全治理缺失。

## GPU Cluster Topology Audit

一句话定义：GPU Cluster Topology Audit 是把单机多卡、多机多卡和机柜级 GPU 集群从“资源数量”升级为可审计拓扑、通信、故障域、资源池和调度门禁的方法。

核心直觉：GPU 集群不是 GPU 数量越多越好，而是要让并行策略、通信路径、GPU-NIC 亲和性、机柜位置、存储路径、电力散热和资源池隔离匹配任务画像。tensor parallel 这类通信密集维度应优先放在同机 NVLink / NVSwitch 或其他 scale-up 高速互联域内；data parallel 可以跨节点扩展，但要看机间 fabric、NCCL、rack locality 和 oversubscription。

常见公式：集群候选资源组可以写成

```math
g_i=(n_i,u_i,p_i,\ell_i,a_i,r_i,b_i,s_i,f_i,c_i,o_i,z_i)
```

总 GPU 和总显存可以写成

```math
G_{\mathrm{total}}=N_{\mathrm{node}}G_{\mathrm{node}}
```

```math
M_{\mathrm{total}}=G_{\mathrm{total}}M_{\mathrm{gpu}}
```

通信耗时可以用简化模型表达为

```math
T_{\mathrm{comm}}\approx \alpha H+\frac{V}{B_{\mathrm{eff}}}
```

其中，`\alpha` 是通信启动或单跳延迟，`H` 是路径跳数，`V` 是通信量，`B_eff` 是有效带宽。

面试表达：GPU 集群架构题要先区分 scale-up 和 scale-out，再讲单机 PCIe / NUMA / NVLink / NVSwitch、GPU-NIC 亲和性、多机 fabric、rack locality、存储 checkpoint、故障域、电力散热、资源池隔离和拓扑感知调度。

## Scale-Up Domain Fit

一句话定义：Scale-Up Domain Fit 是检查通信密集并行组是否能放进同机或同一高速 GPU 互联域内。

常见坑：把 tensor parallel 随机跨节点放置，会把每层通信放到慢链路上，扩展效率明显下降。

## PCIe NUMA Locality

一句话定义：PCIe NUMA Locality 是检查 GPU、CPU socket、PCIe Root Complex、系统内存和 NIC 是否有明确亲和性映射。

面试表达：不是所有 8 卡机器都一样，PCIe 层级和 NUMA 亲和性会影响 CPU-GPU 拷贝、GPU-NIC 通信和多任务切分。

## NVLink NVSwitch Locality

一句话定义：NVLink NVSwitch Locality 是检查 GPU 间高速互联域是否存在、是否均匀、是否被调度器和并行策略正确使用。

## GPU NIC Affinity

一句话定义：GPU NIC Affinity 是检查 GPU 到网卡的 PCIe 路径、NUMA 亲和性和 GPUDirect RDMA 条件是否清楚。

常见坑：多机训练只看网卡型号，不看哪个 GPU 到哪个 NIC 更近，导致跨 socket 或跨 PCIe 路径额外开销。

## Inter-Node Fabric Readiness

一句话定义：Inter-Node Fabric Readiness 是检查节点之间的 InfiniBand、RoCE 或 Ethernet fabric 是否满足带宽、延迟、拥塞控制、RDMA 和运维要求。

## Rack Locality Awareness

一句话定义：Rack Locality Awareness 是检查任务是否优先放在同一机柜或网络距离较近的机柜内，同时避免不必要的跨全局集群通信。

## Oversubscription Awareness

一句话定义：Oversubscription Awareness 是检查下行接入带宽与上行汇聚带宽的比例是否被记录，并评估它对集合通信和并发任务的影响。

常见公式：

```math
O_{\mathrm{net}}=\frac{B_{\mathrm{down}}}{B_{\mathrm{up}}}
```

## Collective Communication Fit

一句话定义：Collective Communication Fit 是检查 AllReduce、AllGather、ReduceScatter、Broadcast 和 point-to-point 等通信模式是否与拓扑和 NCCL 配置匹配。

## Parallel Group Placement

一句话定义：Parallel Group Placement 是检查 tensor parallel、pipeline parallel、data parallel、expert parallel 等并行组是否按通信密度放在合适拓扑范围内。

## Storage Checkpoint Locality

一句话定义：Storage Checkpoint Locality 是检查训练数据缓存、checkpoint 写入和恢复路径是否靠近计算资源，避免共享存储或远端对象存储成为瓶颈。

## Fault Domain Isolation

一句话定义：Fault Domain Isolation 是检查单卡、单机、单机柜、单交换机或共享存储故障会影响多少 GPU 和任务。

常见公式：

```math
F_{\mathrm{rack}}=\frac{G_{\mathrm{rack}}}{G_{\mathrm{total}}}
```

## Power Cooling Capacity Fit

一句话定义：Power Cooling Capacity Fit 是检查机柜电力、供电冗余、散热能力和运维空间是否能支撑目标 GPU 密度。

## Resource Pool Isolation

一句话定义：Resource Pool Isolation 是检查训练、推理、评估、SFT、交互式开发和低优先级任务是否有资源池或优先级隔离。

## Topology-Aware Scheduling

一句话定义：Topology-Aware Scheduling 是调度器不只找够 GPU 数量，还要按 GPU 互联、GPU-NIC、rack、fault domain、storage locality 和任务画像选择资源。

## Observability Topology Coverage

一句话定义：Observability Topology Coverage 是检查 PCIe、NVLink、NIC、NCCL、rack、storage、checkpoint 和调度决策是否有 metrics、logs、traces 或事件记录。

## GPU Cluster Gate

一句话定义：GPU Cluster Gate 是判断 GPU 集群资源组是否满足拓扑、通信、资源池、故障域和观测最低要求的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{cluster}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{slow}}\le \rho_{\mathrm{slow}} \land O_{\mathrm{net}}\le \rho_{\mathrm{over}} \land P_0=0\right]
```

其中，`R_slow` 是跨慢链路通信比例，`O_net` 是网络 oversubscription，`P_0` 是 P0 级风险数量。

典型硬阻断：scale-up 域不匹配、PCIe / NUMA 未知、NVLink / NVSwitch 拓扑未用、GPU-NIC 亲和性缺失、机间 fabric 未就绪、机柜 locality 缺失、oversubscription 未知、collective 未建模、并行组随机放置、checkpoint 路径过远、故障域过大、电力散热超配、训练推理混池、调度器不拓扑感知或缺拓扑观测指标。

## Network Communication Audit

一句话定义：Network Communication Audit 是把 InfiniBand、RoCE、以太网、RDMA、GPUDirect RDMA、NCCL 和集合通信从“网络很快/很慢”的描述升级为可检查指标表的方法。

核心直觉：多机大模型训练的网络在每个 step 参与梯度、参数分片、activation 或 optimizer state 同步。网络排查不能只看纸面带宽，要同时看 bit / byte 单位、有效带宽、延迟、抖动、丢包、重传、RDMA、GDR、fabric、collective、NCCL 拓扑、rank straggler 和调度放置。

常见公式：带宽单位换算可以写成

```math
B_{\mathrm{GiB/s}}=\frac{B_{\mathrm{Gbps}}\cdot 10^9}{8\cdot 2^{30}}
```

通信耗时可以粗略拆成

```math
T_{\mathrm{msg}}\approx \alpha H+\frac{D}{B_{\mathrm{eff}}}+T_{\mathrm{queue}}+T_{\mathrm{retx}}
```

Ring AllReduce 每个 rank 的近似通信量可以写成

```math
V_{\mathrm{ring}}\approx 2\frac{P-1}{P}D
```

面试表达：多机训练扩展效率差时，先看 step time 和通信占比，再看 NCCL / profiler、rank 差异、网卡、错误包、重传、RDMA / GDR、交换机拥塞、跨机柜路径和调度放置。

## Bandwidth Unit Accounting

一句话定义：Bandwidth Unit Accounting 是检查 Gbps、GB/s、GiB/s、理论带宽和有效带宽是否被清楚区分。

常见坑：把 400 Gbps 当成 400 GB/s，或者只看链路峰值不看有效带宽系数。

## Latency Jitter Tracking

一句话定义：Latency Jitter Tracking 是检查 p50、p95、p99 延迟和 step time 抖动是否被记录。

常见公式：

```math
J_{\mathrm{lat}}=\frac{L_{\mathrm{p99}}}{L_{\mathrm{p50}}}
```

## RDMA Capability Fit

一句话定义：RDMA Capability Fit 是检查网络是否支持远程直接内存访问，以及训练栈是否实际启用和验证。

## GPUDirect RDMA Path

一句话定义：GPUDirect RDMA Path 是检查 NIC 是否能直接访问 GPU 显存，并确认 GPU-NIC PCIe / NUMA 亲和性。

## InfiniBand Fabric Readiness

一句话定义：InfiniBand Fabric Readiness 是检查 InfiniBand fabric 的链路、子网、RDMA、带宽、延迟和运维健康是否满足训练任务。

## RoCE Congestion Losslessness

一句话定义：RoCE Congestion Losslessness 是检查 RoCE 网络的 PFC、ECN、拥塞控制、低丢包和运维配置是否满足 RDMA 通信。

常见坑：把 RoCE 当普通以太网使用，不配置 lossless 或拥塞控制，导致丢包和重传放大成 NCCL timeout。

## Ethernet Fallback Scope

一句话定义：Ethernet Fallback Scope 是明确普通以太网适合小规模训练、评估、批量推理或低通信任务，不应默认承载大规模高通信预训练。

## Collective Operation Modeling

一句话定义：Collective Operation Modeling 是检查 AllReduce、AllGather、ReduceScatter、Broadcast 和 point-to-point 的通信量、频率和消息大小是否被建模。

## NCCL Topology Runtime Fit

一句话定义：NCCL Topology Runtime Fit 是检查 NCCL 是否识别预期 NIC、GPU 拓扑、GDR 路径、ring / tree 选择和实际通信路径。

## AllReduce Cost Estimation

一句话定义：AllReduce Cost Estimation 是估算数据并行梯度同步的通信量和耗时，避免只看模型参数量或 GPU 数量。

## AllGather ReduceScatter Cost

一句话定义：AllGather ReduceScatter Cost 是估算 FSDP / ZeRO 等分片策略带来的额外参数拉取和梯度分发通信。

## Rank Straggler Detection

一句话定义：Rank Straggler Detection 是检查最慢 rank 是否拖住同步训练全局 step。

## Packet Error Retransmit Tracking

一句话定义：Packet Error Retransmit Tracking 是检查网卡、交换机和协议层错误包、丢包、重传和错误计数。

常见公式：

```math
R_{\mathrm{retx}}=\frac{N_{\mathrm{retx}}}{N_{\mathrm{packet}}}
```

## Topology Congestion Awareness

一句话定义：Topology Congestion Awareness 是检查任务路径是否跨拥塞交换机、跨机柜、oversubscription 链路或与其他大任务共享瓶颈链路。

## Scheduler Network Locality

一句话定义：Scheduler Network Locality 是检查调度系统是否根据 rack、fabric、GPU-NIC 亲和性和通信密集并行组放置任务。

## Network Communication Gate

一句话定义：Network Communication Gate 是判断一个多机训练或推理任务的网络通信是否达到最低可用要求的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{net}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{comm}}\le \rho_{\mathrm{comm}} \land J_{\mathrm{lat}}\le \rho_{\mathrm{jitter}} \land R_{\mathrm{retx}}\le \rho_{\mathrm{retx}} \land P_0=0\right]
```

典型硬阻断：带宽单位混淆、延迟抖动未记录、RDMA 未启用、GDR 路径缺失、InfiniBand fabric 未验收、RoCE 无 PFC / ECN / 拥塞控制、普通以太网承载大规模预训练、collective 未建模、NCCL 拓扑未知、AllReduce / AllGather / ReduceScatter 成本缺失、rank straggler 被隐藏、错误包和重传未追踪、拓扑拥塞未知、调度器不看网络 locality 或缺网络通信门禁。

## Storage System Audit

一句话定义：Storage System Audit 是把本地 NVMe、共享文件系统、对象存储、数据湖、checkpoint、模型权重和 artifact 从“存得下”升级为可审计的吞吐、metadata、缓存、恢复、血缘、权限和成本门禁。

核心直觉：AI 存储不只是放文件。训练数据要稳定供给 GPU，checkpoint 要能写出且恢复，模型权重要能快速加载，artifact 要能追踪血缘，数据湖要能治理权限和质量，生命周期要能控制成本。

常见公式：有效吞吐可以写成

```math
B_{\mathrm{eff}}=\frac{D}{T}
```

训练 I/O 占比可以写成

```math
R_{\mathrm{io}}=\frac{T_{\mathrm{dataloader}}+T_{\mathrm{storage}}+T_{\mathrm{h2d}}}{T_{\mathrm{step}}}
```

checkpoint 保存时间可以写成

```math
T_{\mathrm{ckpt}}=\frac{S_{\mathrm{ckpt}}}{\eta B_{\mathrm{write}}}+T_{\mathrm{meta}}+T_{\mathrm{commit}}
```

面试表达：存储体系题要按访问模式分层回答：对象存储 / 数据湖作权威源，本地 NVMe / 分布式缓存作热数据加速，共享文件系统提供易用性但要防 metadata 瓶颈，checkpoint 和模型权重要有恢复测试、checksum、commit、缓存和生命周期策略。

## Capacity Tier Fit

一句话定义：Capacity Tier Fit 是检查 hot、warm、cold、archive 等存储层级是否匹配训练数据、checkpoint、日志和 artifact 的增长。

## Throughput IOPS Fit

一句话定义：Throughput IOPS Fit 是检查读写吞吐、IOPS、延迟和并发是否按真实训练、checkpoint 和模型加载流量测过。

## Local NVMe Cache Fit

一句话定义：Local NVMe Cache Fit 是检查训练节点本地 NVMe 是否用于热点 shard、解压数据、模型权重和临时文件缓存，并有容量和淘汰策略。

## Shared FS Metadata Fit

一句话定义：Shared FS Metadata Fit 是检查共享文件系统是否能承受 open、stat、list、rename、commit 和小文件 metadata 压力。

## Object Store Authority Fit

一句话定义：Object Store Authority Fit 是检查对象存储是否作为权威源、归档和 artifact 后端使用，而不是被误当成高频 POSIX 文件系统直接训练读取。

## Data Lake Governance Fit

一句话定义：Data Lake Governance Fit 是检查数据湖是否记录数据血缘、质量、权限标签、清洗规则、版本和合规状态。

## Training Shard Format Fit

一句话定义：Training Shard Format Fit 是检查训练数据是否被预处理成适合顺序读取和并发读取的 shard，例如 WebDataset、Parquet、Arrow 或自定义二进制格式。

## Small File Amplification Control

一句话定义：Small File Amplification Control 是检查小文件数量是否被 shard、索引、合并、缓存和预取控制。

常见公式：

```math
A_{\mathrm{meta}}=\frac{N_{\mathrm{file}}}{D_{\mathrm{TiB}}}
```

## Dataloader Cache Hit Tracking

一句话定义：Dataloader Cache Hit Tracking 是检查 dataloader 是否记录本地缓存、远端读取、等待时间和 cache hit rate。

常见公式：

```math
H_{\mathrm{cache}}=\frac{D_{\mathrm{cache}}}{D_{\mathrm{total}}}
```

## Checkpoint Write Recovery Fit

一句话定义：Checkpoint Write Recovery Fit 是检查 checkpoint 是否有写入吞吐、分片策略、异步保存、checksum、commit metadata 和恢复演练。

## Model Weight Load Cache Fit

一句话定义：Model Weight Load Cache Fit 是检查模型权重、tokenizer、config、adapter 和量化版本是否有本地缓存、分层缓存、预热和完整性校验。

## Artifact Lineage Metadata Fit

一句话定义：Artifact Lineage Metadata Fit 是检查 dataset version、training config、checkpoint、final model、eval report 和 deployment package 是否能互相追溯。

## Consistency Commit Integrity

一句话定义：Consistency Commit Integrity 是检查大文件写入是否先写临时对象，校验 checksum 后再用 manifest 或 commit 标记为可用。

## Security Compliance Fit

一句话定义：Security Compliance Fit 是检查身份认证、访问控制、加密、审计日志、数据分类、留存删除和租户隔离是否覆盖存储链路。

## Lifecycle Cost Governance

一句话定义：Lifecycle Cost Governance 是检查 checkpoint、数据集、日志、trace 和中间 artifact 是否有保留、归档、删除、去重和成本归因策略。

常见公式：

```math
K_{\mathrm{storage}}=K_{\mathrm{capacity}}+K_{\mathrm{request}}+K_{\mathrm{egress}}+K_{\mathrm{metadata}}+K_{\mathrm{ops}}
```

## Storage System Gate

一句话定义：Storage System Gate 是判断 AI 存储体系是否达到训练、恢复、部署和治理最低要求的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{storage}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{io}}\le \rho_{\mathrm{io}} \land T_{\mathrm{restore}}\le \tau_{\mathrm{restore}} \land P_0=0\right]
```

典型硬阻断：容量不分层、吞吐和 IOPS 未实测、本地缓存缺失、共享文件系统 metadata 过载、对象存储被当 POSIX 高频训练源、数据湖无血缘治理、训练格式仍是 raw JSON、小文件未 shard、cache hit 未追踪、checkpoint 没恢复测试、模型权重冷启动无缓存、artifact 血缘缺失、commit / checksum 缺失、安全审计缺失、生命周期策略缺失或没有存储门禁。

## Checkpoint Lifecycle Audit

一句话定义：Checkpoint Lifecycle Audit 是把训练状态 checkpoint 从“保存目录”升级为可审计的状态对象、分片布局、异步写入、manifest、checksum、恢复演练、保留策略、成本、权限和 SLO 门禁。

核心直觉：checkpoint 的价值不是保存成功，而是故障、抢占、扩缩容、代码回滚和模型发布时能被可靠恢复和解释。训练状态 checkpoint 至少要覆盖模型参数、优化器、scheduler、global step、AMP scaler、RNG、dataloader 位置、分布式 rank / partition metadata、代码版本、数据版本和训练配置。

常见公式：分片大小可以写成

```math
S_{\mathrm{shard}}=\frac{S_{\mathrm{ckpt}}}{N_{\mathrm{shard}}}
```

保存耗时可以写成

```math
T_{\mathrm{save}}=\frac{S_{\mathrm{ckpt}}}{\eta_w B_{\mathrm{write}}}+T_{\mathrm{sync}}+T_{\mathrm{meta}}+T_{\mathrm{commit}}
```

恢复耗时可以写成

```math
T_{\mathrm{restore}}=\frac{S_{\mathrm{ckpt}}}{\eta_r B_{\mathrm{read}}}+T_{\mathrm{validate}}+T_{\mathrm{rebuild}}+T_{\mathrm{requeue}}
```

面试表达：checkpoint 系统设计题要先区分模型权重 checkpoint 和训练状态 checkpoint，再讲分片、manifest、checksum、commit 语义、异步保存、恢复演练、RNG / dataloader / rank state、一致性、保留策略、成本归因、跨区复制、权限审计和 SLO。

## Checkpoint Object Completeness

一句话定义：Checkpoint Object Completeness 是检查 checkpoint 是否保存完整训练状态，而不只是模型权重。

## Shard Layout Fit

一句话定义：Shard Layout Fit 是检查 checkpoint shard 数量、大小、rank 映射、manifest 记录和并行配置是否匹配。

## Async Save Overlap

一句话定义：Async Save Overlap 是检查异步 checkpoint 是否真正降低训练可见阻塞，并且没有破坏快照一致性或造成后台队列积压。

## Write Bandwidth Fit

一句话定义：Write Bandwidth Fit 是检查 checkpoint 写入带宽、metadata 开销、commit 开销和保存 SLO 是否实测过。

## Metadata Commit Integrity

一句话定义：Metadata Commit Integrity 是检查 checkpoint 是否先写临时对象，所有 shard 校验成功后再把 manifest 标记为 committed。

## Checksum Manifest Validation

一句话定义：Checksum Manifest Validation 是检查每个 shard 是否有 digest，恢复时是否按 manifest 校验完整性。

## Restore Replay Readiness

一句话定义：Restore Replay Readiness 是检查 checkpoint 是否通过真实恢复演练，而不是只验证保存流程。

## Dataloader RNG State Capture

一句话定义：Dataloader RNG State Capture 是检查 Python、NumPy、PyTorch、CUDA、sampler 和 dataloader 位置是否可恢复。

## Distributed Rank State Capture

一句话定义：Distributed Rank State Capture 是检查 tensor parallel、pipeline parallel、data parallel、ZeRO / FSDP partition 和 gradient accumulation 状态是否可恢复。

## Retention Policy Fit

一句话定义：Retention Policy Fit 是检查最近 checkpoint、milestone、best eval、release 版本和失败临时版本是否有清晰保留策略。

## Cross-Region Replication Fit

一句话定义：Cross-Region Replication Fit 是检查关键 checkpoint、发布权重和灾备副本是否有跨故障域复制和恢复测试。

## Security Access Control

一句话定义：Security Access Control 是检查 checkpoint 存储是否有 IAM / RBAC、加密、审计日志和租户隔离。

## Resume SLO Tracking

一句话定义：Resume SLO Tracking 是检查 save p95、restore p95、time since last checkpoint 和最大丢失进度是否进入训练平台 SLO。

## Failure Drill Coverage

一句话定义：Failure Drill Coverage 是检查是否自动演练节点故障、shard 缺失、checksum 错误、抢占恢复和恢复后 loss 连续性。

## Checkpoint Lifecycle Gate

一句话定义：Checkpoint Lifecycle Gate 是判断 checkpoint 生命周期是否达到训练容错和成本治理最低要求的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{ckpt}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{restore}}\le \tau_{\mathrm{restore}} \land W_{\mathrm{lost}}\le \omega \land P_0=0\right]
```

典型硬阻断：只保存模型权重、单大文件、异步保存没有一致快照、写入带宽未知、manifest 先标 committed、checksum 缺失、从未恢复演练、RNG / dataloader / rank state 丢失、无限保留、冷归档不用、无灾备副本、权限开放、恢复 SLO 缺失、故障演练缺失或没有 checkpoint 生命周期门禁。

## Container Environment Audit

一句话定义：Container Environment Audit 是把 Docker 镜像、GPU runtime、CUDA / driver / framework、依赖锁定、镜像安全、registry 权限和运行 metadata 放进同一套可复现环境门禁。

核心直觉：容器化不是“把代码塞进镜像”，而是让训练和推理环境可复现、可分发、可调度、可审计。GPU 容器仍依赖宿主机驱动、内核、设备挂载、网络和多租户策略，因此镜像 digest、CUDA 兼容、GPU runtime、NCCL / RDMA、依赖锁定和安全扫描必须一起看。

常见公式：镜像拉取时间可以写成

```math
T_{\mathrm{pull}}=\frac{S_{\mathrm{img}}}{\eta B_{\mathrm{reg}}}+T_{\mathrm{unpack}}
```

冷启动时间可以写成

```math
T_{\mathrm{start}}=T_{\mathrm{pull}}+T_{\mathrm{init}}+T_{\mathrm{smoke}}
```

依赖锁定覆盖率可以写成

```math
C_{\mathrm{lock}}=\frac{N_{\mathrm{pinned}}}{N_{\mathrm{dep}}}
```

面试表达：容器化设计题要先说镜像分层和 digest，再讲 GPU 容器依赖宿主机 driver、CUDA 用户态库和框架兼容矩阵；随后补依赖锁定、训练 / 推理镜像分离、镜像体积和冷启动、CI smoke test、漏洞扫描 / 签名 / SBOM、registry RBAC、非 root / 非 privileged、多租户挂载边界和环境 metadata。

## Image Digest Reproducibility

一句话定义：Image Digest Reproducibility 是检查训练和推理任务是否记录不可变 image digest，而不是只记录可覆盖的 tag。

## Base Layer Cache Fit

一句话定义：Base Layer Cache Fit 是检查基础镜像层是否可复用，节点是否预拉取常用 layer，避免任务启动被镜像拉取拖慢。

常见公式：

```math
R_{\mathrm{layer}}=\frac{S_{\mathrm{cached}}}{S_{\mathrm{layer}}}
```

## CUDA Driver Framework Compatibility

一句话定义：CUDA Driver Framework Compatibility 是检查宿主机 driver、容器 CUDA 用户态库、PyTorch / TensorFlow、cuDNN、NCCL、Triton 和自定义 kernel 是否形成兼容矩阵。

## GPU Runtime Visibility

一句话定义：GPU Runtime Visibility 是检查容器是否通过 runtime / CDI、device plugin、device file 和 driver capabilities 正确看到 GPU。

## Dependency Lock Coverage

一句话定义：Dependency Lock Coverage 是检查 Python 包、系统库、私有 wheel、CUDA 扩展和 serving runtime 是否锁版本或 hash。

## Training Serving Image Separation

一句话定义：Training Serving Image Separation 是检查训练镜像和推理镜像是否按目标分离，避免把调试工具、编译工具和实验依赖直接带到线上推理。

## Image Size Startup Fit

一句话定义：Image Size Startup Fit 是检查镜像大小、拉取时间、解压时间、初始化时间和冷启动 SLO 是否匹配训练任务提交和推理扩容。

## Build Pipeline Smoke Test

一句话定义：Build Pipeline Smoke Test 是检查镜像构建流水线是否自动跑 Python、CUDA、框架、NCCL、关键 import 和小样本推理 / 训练 smoke test。

## Security Scan Signing

一句话定义：Security Scan Signing 是检查镜像是否完成漏洞扫描、签名、SBOM 生成和基础镜像更新治理。

## Secret Data Exclusion

一句话定义：Secret Data Exclusion 是检查镜像中是否误打入 API key、token、密钥、训练数据、敏感样本或大模型权重。

## Runtime Hardening Fit

一句话定义：Runtime Hardening Fit 是检查容器是否避免 root、privileged、过宽 Linux capabilities 和可写 root filesystem 等高风险运行方式。

## Registry Access Governance

一句话定义：Registry Access Governance 是检查镜像仓库是否有私有访问、RBAC、审计日志、retention、immutable tag 或 digest policy。

## Environment Metadata Capture

一句话定义：Environment Metadata Capture 是检查训练和推理任务是否自动记录 image digest、driver、CUDA、Python、框架、NCCL、命令、环境变量、代码 commit 和数据版本。

## Multi-Tenant Mount Isolation

一句话定义：Multi-Tenant Mount Isolation 是检查容器挂载是否限制在租户和任务授权范围内，避免 wildcard hostPath、敏感卷误挂和环境变量泄露。

## NCCL RDMA Runtime Fit

一句话定义：NCCL RDMA Runtime Fit 是检查分布式训练容器是否挂载 RDMA / IB 设备、网络模式和 NCCL smoke test 是否通过。

## Container Environment Gate

一句话定义：Container Environment Gate 是判断 AI 训练和推理容器环境是否达到复现、启动、安全和多租户治理最低要求的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{container}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{start}}\le \tau_{\mathrm{start}} \land V_{\mathrm{crit}}=0 \land P_0=0\right]
```

典型硬阻断：只记录 latest tag、基础 layer 缓存差、宿主机 driver 太旧、容器看不到 GPU、依赖未锁定、训练镜像直接上推理、镜像过大导致冷启动超 SLO、缺 smoke test、镜像未签名或有 critical 漏洞、密钥或模型权重打进镜像、root / privileged 运行、registry 公开、环境 metadata 缺失、wildcard hostPath、RDMA / IB 设备不可见或没有容器环境门禁。

## Kubernetes GPU Resource Management Audit

一句话定义：Kubernetes GPU Resource Management Audit 是把 device plugin、GPU extended resource、request / limit、gang scheduling、碎片治理、拓扑放置、MIG / 共享、namespace / quota、多租户、监控和排障放进同一套 Kubernetes GPU 资源管理门禁。

核心直觉：Kubernetes 是 AI Infra 的常见底座，但默认调度器并不自动理解所有大模型训练和推理语义。GPU 资源管理要同时回答：GPU 是否被 kubelet 识别，Pod 是否正确申请资源，分布式训练是否能整体启动，空闲 GPU 是否被碎片化卡住，调度是否理解 NVLink / GPU-NIC / rack / storage locality，MIG 或 time slicing 是否匹配工作负载，租户配额和成本是否可治理，监控是否能把 Pod 映射到具体 GPU。

常见公式：GPU 分配率可以写成

```math
U_{\mathrm{gpu}}=\frac{G_{\mathrm{allocated}}}{G_{\mathrm{total}}}
```

碎片化比例可以写成

```math
R_{\mathrm{frag}}=1-\frac{\sum_n \lfloor g_n / g_{\mathrm{job}}\rfloor g_{\mathrm{job}}}{\sum_n g_n}
```

租户配额占用率可以写成

```math
Q_{\mathrm{tenant}}=\frac{G_{\mathrm{used}}}{G_{\mathrm{quota}}}
```

Kubernetes GPU 资源管理门禁可以写成

```math
G_{\mathrm{k8s\_gpu}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{frag}}\le \rho \land Q_{\mathrm{tenant}}\le 1 \land P_0=0\right]
```

面试表达：Kubernetes GPU 管理题要先说 device plugin 把 GPU 注册成 `nvidia.com/gpu` 这类 extended resource，再讲 Pod resource limit、request / limit 一致性、GPU Operator、MIG / time slicing 的适用边界；随后补 gang scheduling、碎片化、拓扑感知、node label / affinity / taint / toleration、namespace / ResourceQuota、多租户隔离、训练 CRD、推理 SLO、DCGM / Prometheus 监控和 Pending 排障。核心观点是：K8s 是底座，不是完整 AI 调度系统。

## Device Plugin Readiness

一句话定义：Device Plugin Readiness 是检查 GPU device plugin 是否就绪，是否把节点 GPU capacity / allocatable 注册给 kubelet，并让 Pod 能通过标准资源名申请 GPU。

## GPU Extended Resource Fit

一句话定义：GPU Extended Resource Fit 是检查 Pod / Job / CRD 是否用 extended resource 表达 GPU 需求，并且总申请量不超过目标资源池可用量。

## GPU Request Limit Integrity

一句话定义：GPU Request Limit Integrity 是检查 GPU request / limit 口径是否一致，避免平台层把整数 GPU 资源写成容易误解或不可调度的声明。

## Gang Scheduling Readiness

一句话定义：Gang Scheduling Readiness 是检查分布式训练 worker 是否具备一起调度、一起失败释放和按完整 world size 启动的调度语义。

## Fragmentation Control

一句话定义：Fragmentation Control 是检查调度系统是否通过 bin packing、资源池划分、大任务预留或抢占重调度控制 GPU 碎片化。

## Topology Aware Placement

一句话定义：Topology Aware Placement 是检查调度是否考虑节点内 GPU 互联、GPU-NIC 亲和性、rack locality、storage locality 和任务通信画像。

## Node Label Affinity Fit

一句话定义：Node Label Affinity Fit 是检查 node label、node selector、node affinity 和标签治理是否能把任务放到正确 GPU 型号、网络和资源池。

## Taint Toleration Fit

一句话定义：Taint Toleration Fit 是检查专用 GPU 节点、推理池、高优先级资源池等 taint 是否有对应 toleration，且不会被普通任务误用。

## MIG Sharing Policy Fit

一句话定义：MIG Sharing Policy Fit 是检查 MIG、time slicing、MPS 或整卡独占是否和工作负载匹配，避免把大模型预训练放到不稳定共享模式。

## Quota Namespace Governance

一句话定义：Quota Namespace Governance 是检查 namespace、ResourceQuota、按 GPU 型号配额、成本归因和租户使用上限是否一起生效。

## Training Operator Readiness

一句话定义：Training Operator Readiness 是检查 PyTorchJob、MPIJob、RayJob 或自研 TrainingJob CRD 是否能处理 rank、service discovery、checkpoint 和 worker 生命周期。

## Inference Service Readiness

一句话定义：Inference Service Readiness 是检查 K8s 推理服务是否具备自动扩缩容、SLO 指标、健康检查、冷启动预算和灰度 / 回滚能力。

## GPU Monitoring Mapping

一句话定义：GPU Monitoring Mapping 是检查 DCGM / Prometheus 等 GPU 指标是否能映射到 team、job、pod、container 和具体 GPU。

## Multi-Tenant Isolation

一句话定义：Multi-Tenant Isolation 是检查 namespace、RBAC、network policy、secret scope、registry scope 和挂载边界是否能隔离不同租户。

## Pending Troubleshooting Coverage

一句话定义：Pending Troubleshooting Coverage 是检查平台是否能把 Pod Pending 拆成 GPU 不足、quota、node affinity、taint、image pull、gang waiting 等可定位原因。

## Kubernetes GPU Gate

一句话定义：Kubernetes GPU Gate 是判断 Kubernetes GPU 资源池是否达到可调度、可隔离、可观测和可排障最低要求的综合门禁。

典型硬阻断：device plugin 缺失、没有声明 `nvidia.com/gpu`、GPU request / limit 混乱、没有 gang scheduling、GPU 碎片化严重、拓扑被忽略、label / affinity 缺失、taint 无 toleration、预训练误用 time slicing、namespace / quota 缺失、训练 CRD 不完整、推理 SLO 缺失、GPU 指标无法映射到 Pod、多租户隔离缺失、Pending 原因不可解释或没有 Kubernetes GPU 门禁。

## Training Scheduler Governance Audit

一句话定义：Training Scheduler Governance Audit 是把训练任务资源画像、队列、优先级、配额、公平性、gang scheduling、backfilling、抢占、checkpoint、碎片、拓扑、成本、观测和失败重排放进同一套调度治理门禁。

核心直觉：训练调度不是 FIFO，也不是单纯提高 GPU utilization。大模型训练调度要在稀缺 GPU 上同时平衡大任务启动、小任务等待、业务优先级、租户公平性、资源借用、抢占损失、checkpoint 恢复、拓扑效率和成本归因。只看谁先提交或谁优先级高，都会让系统在真实多团队环境里失控。

常见公式：队列等待时间可以按队列统计 P95：

```math
W_{q,\mathrm{p95}}=\mathrm{P95}(t_{\mathrm{start}}-t_{\mathrm{sub}}\mid q)
```

租户配额占用率可以写成

```math
Q_u=\frac{G_{u,\mathrm{used}}}{G_{u,\mathrm{quota}}}
```

多资源 dominant share 可以写成

```math
D_u=\max_r \frac{A_{u,r}}{C_r}
```

抢占损失可以写成

```math
L_j=G_j(t_{\mathrm{now}}-t_{\mathrm{ckpt}})
```

训练调度门禁可以写成

```math
G_{\mathrm{train\_sched}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land W_{q,\mathrm{p95}}\le \omega_q \land Q_u\le 1 \land R_{\mathrm{frag}}\le \rho \land P_0=0\right]
```

面试表达：训练调度系统设计题要先讲 TrainingJob 资源画像，再讲 queue / priority / quota / fair sharing；分布式训练要用 gang scheduling，低优先级短任务可以 backfilling，但不能阻塞已保留资源；抢占只应作用于可抢占且有 checkpoint 的任务，并记录恢复状态；调度还要结合 GPU 碎片、拓扑、成本、SLO、失败重排和观测指标，最终用调度门禁收束，而不是只说“用 Kubernetes scheduler”。

## Workload Resource Shape

一句话定义：Workload Resource Shape 是检查训练任务是否声明 GPU 类型、GPU 数、每节点 GPU、网络、存储、队列、优先级、可抢占性、预计运行时长和 checkpoint 路径。

## Queue Policy Coverage

一句话定义：Queue Policy Coverage 是检查每个队列是否有配额、并发上限、优先级集合、抢占策略、借用 / 出借规则和默认 checkpoint 要求。

## Priority Governance

一句话定义：Priority Governance 是检查 critical / high / normal / low / best-effort 等优先级是否有审批、额度、审计和成本归因，避免所有任务都申请高优先级。

## Quota Usage Control

一句话定义：Quota Usage Control 是检查团队 GPU 数、GPU 小时、GPU 型号、预算和并发任务是否在配额或可借用范围内。

## Fair Share Accounting

一句话定义：Fair Share Accounting 是检查调度系统是否按权重、历史使用、dominant share 或队列策略解释多团队资源分配。

## Gang Scheduling Fit

一句话定义：Gang Scheduling Fit 是检查分布式训练 worker 是否具备整体 admission、整体启动和失败释放语义，避免部分 worker 占住资源。

## Backfilling Safety

一句话定义：Backfilling Safety 是检查短任务使用空闲资源时是否不会推迟已保留的大任务，并且具备截止时间、可抢占或运行时长约束。

## Preemption Checkpoint Safety

一句话定义：Preemption Checkpoint Safety 是检查抢占只作用于低优先级、可抢占、最近有 checkpoint、恢复成本可控且记录原因的任务。

## Scheduler Fragmentation Control

一句话定义：Scheduler Fragmentation Control 是检查调度系统是否用 bin packing、小任务池、大任务池、空间预留、抢占或重排降低 GPU 碎片。

## Scheduler Topology Awareness

一句话定义：Scheduler Topology Awareness 是检查调度是否使用 GPU 型号、NVLink / NVSwitch、GPU-NIC、rack、存储 locality 和故障域信息。

## Checkpoint Scheduler Coupling

一句话定义：Checkpoint Scheduler Coupling 是检查调度器是否知道 checkpoint URI、最近 checkpoint 时间、最大丢失进度和恢复耗时，从而做理性抢占和恢复。

## Cost Attribution Control

一句话定义：Cost Attribution Control 是检查 GPU 小时、队列成本中心、超额资源申请、闲置 GPU 和失败重跑是否能归因到团队和项目。

## Scheduler Observability

一句话定义：Scheduler Observability 是检查 queue length、pending P95、running jobs、GPU allocated、fragmentation、preemption count、fair share 和 success rate 是否可观测。

## Failure Requeue Readiness

一句话定义：Failure Requeue Readiness 是检查任务失败、抢占或节点故障后是否有重排、backoff、event reason、terminal status 和恢复路径。

## Anti Starvation Control

一句话定义：Anti Starvation Control 是检查低优先级或小团队任务是否有最大等待时间、aging boost、队列 SLO 或人工升级路径，避免长期饥饿。

## Training Scheduler Gate

一句话定义：Training Scheduler Gate 是判断训练调度系统是否达到可排队、可公平、可抢占、可恢复、可观测和可成本治理最低要求的综合门禁。

典型硬阻断：任务资源画像缺网络或 checkpoint、队列无策略、优先级无边界、租户超配额、公平性不可解释、gang scheduling 缺失、backfilling 可能阻塞保留任务、抢占无 checkpoint、碎片治理缺失、调度不看拓扑、checkpoint 信息未进入调度器、成本不可归因、调度指标缺失、失败后不能重排、低优任务长期饥饿或没有训练调度门禁。

## Multi Tenant GPU Scheduler Design Audit

一句话定义：Multi Tenant GPU Scheduler Design Audit 是把多租户队列、配额借用、公平性、gang scheduling、拓扑、碎片、抢占、低优先级、推理 SLO、异构 GPU、可解释性、成本和观测放进同一套 GPU 集群调度系统设计门禁。

核心直觉：多租户 GPU 调度不是“用 Kubernetes scheduler 调 Pod”，也不是只做 GPU 利用率优化。高分系统设计要证明每个租户通过 queue / quota / priority 被治理，借用资源可以回收，分布式训练具备整体 admission，拓扑放置能匹配通信画像，抢占和 checkpoint 绑定，低优先级任务可回收，推理服务有 SLO 隔离，异构 GPU 有成本和适配策略，并且每次 pending、placement 和 preemption 都能解释给用户和平台 owner。

常见公式：队列配额占用率可以写成

```math
U_q=\frac{G_{\mathrm{used},q}}{G_{\mathrm{quota},q}}
```

借用 GPU 数可以写成

```math
B_q=\max\left(0,G_{\mathrm{used},q}-G_{\mathrm{guaranteed},q}\right)
```

Dominant share 可以写成

```math
D_q=\max_{r\in\mathcal{R}}\frac{u_{q,r}}{c_r}
```

带权公平偏差可以写成

```math
H_q=\frac{D_q}{w_q},\qquad \Delta_{\mathrm{fair}}=\max_q H_q-\min_q H_q
```

Gang scheduling 可行性可以写成

```math
G_{\mathrm{gang}}=\mathbf{1}\left[\sum_{n=1}^{N}\left\lfloor\frac{a_n}{g_{\mathrm{pernode}}}\right\rfloor \ge n_{\mathrm{worker}}\right]
```

碎片率可以写成

```math
R_{\mathrm{frag}}=1-\frac{\sum_n \left\lfloor a_n/g_{\mathrm{job}}\right\rfloor g_{\mathrm{job}}}{\sum_n a_n}
```

多租户 GPU 调度门禁可以写成

```math
G_{\mathrm{mtgpu}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land \Delta_{\mathrm{fair}}\le \delta \land R_{\mathrm{frag}}\le \rho \land L_{\mathrm{preempt}}\le \lambda \land P_0=0\right]
```

面试表达：先澄清集群规模、GPU 异构性、训练 / 推理 / 评估任务比例、租户和优先级规则；再讲 Queue Manager、Resource Snapshot、Policy Engine、Placement Engine、Preemption Engine 和 Observability。调度策略要按 queue quota、borrow / lend、dominant share、gang、topology-aware placement、fragmentation control、checkpoint-safe preemption、low-priority reclaim 和 inference SLO isolation 分层。最后用 Multi Tenant GPU Scheduler Gate 收束，说明 pending 原因、抢占原因、成本归因和公平性证据都可审计。

## Tenant Queue Contract

一句话定义：Tenant Queue Contract 是检查每个租户队列是否声明 owner、quota、guaranteed / burst / borrowed 资源、priority、最大并发、可抢占策略、允许 GPU 型号和成本中心。

## Quota Borrowing Policy

一句话定义：Quota Borrowing Policy 是检查队列借用空闲 GPU 时是否有上限、cohort、回收规则、补偿策略、victim 选择和审计事件。

## Dominant Fairness Accounting

一句话定义：Dominant Fairness Accounting 是检查公平性是否同时考虑 GPU、CPU、内存、网络、存储和稀缺 GPU 型号，而不是只看 GPU 张数。

## Low Priority Reclaim

一句话定义：Low Priority Reclaim 是检查低优先级任务是否只使用可回收余量，并且具备 checkpoint、graceful shutdown、重新入队、通知和抢占频率限制。

## Inference SLO Isolation

一句话定义：Inference SLO Isolation 是检查低延迟推理服务是否从批训练抢占、碎片整理和低优先级填充中隔离出来，至少有独立资源池、优先级或 SLO guardrail。

## Heterogeneous GPU Policy

一句话定义：Heterogeneous GPU Policy 是检查调度是否根据 GPU 型号、显存、互联、CUDA / driver、任务适配性和成本，把训练、推理、评估、embedding 与数据处理放到合适加速器上。

## Resource Snapshot Health

一句话定义：Resource Snapshot Health 是检查调度器资源视图是否包含 GPU 空闲量、健康状态、拓扑、MIG / sharing、队列占用、租户借用和节点故障，并且有 freshness 约束。

## Placement Explainability

一句话定义：Placement Explainability 是检查每次调度、排队和抢占是否记录候选节点、过滤原因、打分项、最终选择、失败原因和用户可读解释。

## Cost Utilization Attribution

一句话定义：Cost Utilization Attribution 是检查 GPU-hours、idle hours、failed GPU-hours、preempted GPU-minutes、borrowed quota、低优先级资源和推理 SLO 资源是否能归因到 tenant、project、queue、job type 和 owner。

## Scheduler Observability Audit

一句话定义：Scheduler Observability Audit 是检查 queue wait、pending reason、quota usage、dominant share、fragmentation、topology score、preemption、gang admission、placement latency 和调度决策事件是否可观测。

## Tradeoff Boundary Reasoning

一句话定义：Tradeoff Boundary Reasoning 是检查调度系统回答是否讲清公平性 vs 利用率、紧凑放置 vs 容错、抢占效率 vs 进度浪费、资源预留 vs 空闲成本、异构成本优化 vs 用户体验复杂度。

## Multi Tenant GPU Scheduler Gate

一句话定义：Multi Tenant GPU Scheduler Gate 是判断多租户 GPU 集群调度系统是否达到可公平、可借用、可回收、可整体调度、可拓扑优化、可抢占恢复、可解释、可归因和可审计最低要求的综合门禁。

典型硬阻断：队列没有 owner 和 quota、借用不可回收、公平性只看 FIFO、gang 任务能半启动、topology ignored、碎片率不可观测、抢占无 checkpoint、低优先级任务不可回收、推理服务被批训练抖动影响、异构 GPU 策略缺失、资源快照陈旧、placement 无解释、成本不可归因、调度事件缺失、trade-off 边界缺失或没有最终调度门禁。

## Multi-Tenant Isolation Audit

一句话定义：Multi-Tenant Isolation Audit 是把资源、身份、权限、数据、网络、运行时、镜像、密钥、日志、成本、环境、共享资源、审计和故障域放进同一套 AI Infra 多租户隔离门禁。

核心直觉：多租户隔离不是“每个团队一个 namespace”。Namespace 只是治理入口，真正的隔离要证明：租户资源不会互相抢，身份不会串租户，权限遵守最小授权，数据和 artifact 不越界，网络默认拒绝，运行时不能绕过容器边界，镜像和密钥可信，日志 / trace 脱敏，成本能归因，生产和实验不会互相影响，跨租户共享有 owner、版本、权限和审计。

常见公式：租户配额占用率可以写成

```math
Q_u=\frac{G_{u,\mathrm{used}}}{G_{u,\mathrm{quota}}}
```

未授权访问拦截率可以写成

```math
B_{\mathrm{auth}}=\frac{N_{\mathrm{blocked}}}{N_{\mathrm{unauth}}}
```

数据等级传播可以写成

```math
L_{\mathrm{artifact}}\ge \max_{d\in D_{\mathrm{train}}} L_d
```

密钥暴露率可以写成

```math
R_{\mathrm{secret}}=\frac{N_{\mathrm{secret\_exposed}}}{N_{\mathrm{secret}}}
```

多租户隔离门禁可以写成

```math
G_{\mathrm{tenant}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land Q_u\le 1 \land B_{\mathrm{auth}}=1 \land R_{\mathrm{secret}}=0 \land C_{\mathrm{cost}}\ge \gamma \land P_0=0\right]
```

面试表达：多租户隔离题要先明确 namespace 只是入口，再按资源、身份、权限、数据、网络、运行时、镜像、密钥、日志、成本和审计分层。AI 场景还要强调数据等级会传播到 checkpoint、模型权重、embedding index、eval report、日志和 trace；生产推理和实验训练要隔离；跨租户共享要产品化、版本化、权限化、审计化。最后用 Multi-Tenant Isolation Gate 收束，而不是只背 RBAC 或 NetworkPolicy。

## Resource Quota Isolation

一句话定义：Resource Quota Isolation 是检查租户 GPU、CPU、内存、存储、队列并发、推理 QPS 和成本预算是否有保障、上限和超额策略。

## Identity Namespace Binding

一句话定义：Identity Namespace Binding 是检查 IdP group、tenant label、namespace、service account 和 workload identity 是否一致绑定，避免身份串租户。

## RBAC ABAC Permission Fit

一句话定义：RBAC ABAC Permission Fit 是检查角色权限和属性策略是否共同覆盖 submit job、read data、read logs、deploy model、access secret 和 change quota 等动作。

## Data Access Boundary

一句话定义：Data Access Boundary 是检查训练数据、RAG 文档、checkpoint、模型权重、日志、trace、prompt 和用户请求是否按租户、项目、用途和数据等级授权。

## Data Classification Lineage

一句话定义：Data Classification Lineage 是检查数据分类和血缘是否传播到 checkpoint、模型权重、adapter、embedding index、eval report 和发布状态。

## Network Policy Isolation

一句话定义：Network Policy Isolation 是检查租户间 Pod 访问、外网 egress、生产数据 endpoint、调试端口和私有服务访问是否默认拒绝、按需放行。

## Runtime Security Boundary

一句话定义：Runtime Security Boundary 是检查 privileged、hostPath、root、共享 namespace、GPU 共享、本地缓存和系统调用限制是否符合租户风险等级。

## Image Supply Chain Boundary

一句话定义：Image Supply Chain Boundary 是检查租户镜像是否来自可信 registry，是否 digest pin、扫描、签名、基础镜像白名单和高危漏洞阻断。

## Secret Scope Rotation

一句话定义：Secret Scope Rotation 是检查密钥是否由 Secret Manager 或等价系统管理，是否租户化、短期化、最小权限、自动轮换、日志脱敏和访问审计。

## Logs Trace Redaction

一句话定义：Logs Trace Redaction 是检查日志、trace、prompt、模型输出、工具调用结果和错误堆栈是否按租户权限查看并脱敏。

## Cost Attribution Coverage

一句话定义：Cost Attribution Coverage 是检查 GPU 小时、存储、网络、checkpoint、推理 token、评估和 Agent 工具调用成本是否归因到 tenant、project、user、job 和 model。

## Prod Experiment Separation

一句话定义：Prod Experiment Separation 是检查生产推理和实验训练是否在 namespace、节点池、网络、权限、镜像审核、发布流程和 SLO 上分离。

## Cross Tenant Sharing Governance

一句话定义：Cross Tenant Sharing Governance 是检查基础模型、公共数据集、benchmark、基础镜像和工具平台等共享资源是否有 owner、版本、只读默认、复制审批、成本分摊和使用审计。

## Audit Evidence Readiness

一句话定义：Audit Evidence Readiness 是检查 policy decision、data access、secret access、image admission、cost ledger 和 trace 是否可查询、可追责、可回放。

## Blast Radius Control

一句话定义：Blast Radius Control 是检查单租户事故、镜像污染、密钥泄露、网络误配或存储权限错误会影响多少租户，并是否有 kill switch 和故障域标签。

## Multi-Tenant Isolation Gate

一句话定义：Multi-Tenant Isolation Gate 是判断 AI 平台多租户是否达到资源、权限、数据、网络、运行时、供应链、密钥、审计和成本隔离最低要求的综合门禁。

典型硬阻断：namespace 无 quota、身份和 namespace 不一致、权限通配、跨租户数据读未阻断、restricted 数据产物被降级、网络默认互通、容器 privileged / hostPath / root、未签名镜像放行、长期共享高权限密钥、日志 / trace 未脱敏、成本无法归因、生产和实验混池、共享资源无版本和 owner、审计日志缺失、单事故影响多个租户或没有多租户隔离门禁。

## Cluster Capacity Planning Audit

一句话定义：Cluster Capacity Planning Audit 是把任务画像、GPU 数量、GPU 型号、训练队列、推理峰值、网络、存储、利用率余量、故障冗余、增长预测、成本预算、资源池隔离和观测回填放进同一套集群容量门禁。

核心直觉：AI 集群容量规划不是“预算能买多少 GPU”，而是先确认要支撑哪些训练、推理、评估、RAG、Agent 和交互式任务，再把 GPU-hours、峰值 QPS、token 长度、网络通信、checkpoint 写入、模型加载、故障冗余、增长场景和成本上限一起审计。GPU 数量算对但网络、存储、推理峰值或预测回填缺失，仍然是不可上线的容量计划。

常见公式：训练 GPU-hours 可以写成

```math
H_{\mathrm{gpu}}=\sum_{k=1}^{K} n_k g_k t_k r_k
```

考虑峰值、故障、碎片和增长后的 GPU 需求可以写成

```math
G_{\mathrm{need}}=\left\lceil \frac{H_{\mathrm{gpu}}}{D\cdot 24\cdot u_{\mathrm{eff}}}\left(1+h_{\mathrm{peak}}+h_{\mathrm{fail}}+h_{\mathrm{frag}}+h_{\mathrm{growth}}\right) \right\rceil
```

推理实例需求可以写成

```math
I_{\mathrm{serve}}=\left\lceil \frac{\lambda_{\mathrm{peak}}(L_{\mathrm{in}}+L_{\mathrm{out}})}{\mu_{\mathrm{inst}}\eta_{\mathrm{slo}}} \right\rceil
```

容量规划门禁可以写成

```math
G_{\mathrm{capacity}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land G_{\mathrm{actual}}\ge G_{\mathrm{need}} \land I_{\mathrm{actual}}\ge I_{\mathrm{serve}} \land B_{\mathrm{net}}\ge \beta_{\mathrm{net}} \land B_{\mathrm{store}}\ge \beta_{\mathrm{store}} \land K_{\mathrm{est}}\le K_{\mathrm{budget}} \land P_0=0\right]
```

面试表达：容量规划题要先问任务画像和 SLO，再用 GPU-hours 估算训练容量，用峰值 QPS、输入输出 token、TTFT / TPOT 和单实例吞吐估算推理容量；随后补 GPU 型号分层、网络 collective、存储吞吐、checkpoint 生命周期、利用率余量、N+1、增长预测、成本预算和 dashboard 回填。最后用 Cluster Capacity Planning Gate 收束，而不是只报一个 GPU 数字。

## Workload Forecast Coverage

一句话定义：Workload Forecast Coverage 是检查预训练、SFT、RLHF / RLAIF、评估、推理、RAG、Agent、多模态和交互式开发等任务是否有数量、时长、SLO、截止时间和增长假设。

## GPU Count Capacity Fit

一句话定义：GPU Count Capacity Fit 是检查实际可用 GPU 数是否覆盖 GPU-hours 折算后的训练需求，并额外考虑峰值、故障、碎片和增长余量。

## GPU Type Mix Fit

一句话定义：GPU Type Mix Fit 是检查高端大显存 GPU、中端 GPU、低优或可抢占资源的比例是否匹配预训练、大模型 SFT、推理、评估、embedding 和开发任务。

## Training Queue SLO Fit

一句话定义：Training Queue SLO Fit 是检查训练队列 P95 / P99 等待时间是否满足团队和任务优先级约束，而不是只看整体 GPU utilization。

## Serving Peak SLO Fit

一句话定义：Serving Peak SLO Fit 是检查推理资源是否覆盖峰值 QPS、输入输出 token、TTFT、TPOT、P99 延迟、KV cache 和自动扩缩容延迟。

## Network Bandwidth Capacity Fit

一句话定义：Network Bandwidth Capacity Fit 是检查训练 collective、checkpoint 路径、模型加载和跨机柜流量是否被实测带宽、RDMA、oversubscription、重传和拓扑约束覆盖。

## Storage Throughput Capacity Fit

一句话定义：Storage Throughput Capacity Fit 是检查训练数据读取、checkpoint 写入与恢复、模型权重加载和多任务并发读写是否有实测吞吐余量。

## Storage Capacity Lifecycle Fit

一句话定义：Storage Capacity Lifecycle Fit 是检查原始数据、清洗数据、训练 shard、checkpoint、模型权重、eval report、日志、trace 和索引是否有容量、保留、归档和删除策略。

## Utilization Headroom Fit

一句话定义：Utilization Headroom Fit 是检查目标有效利用率是否保留 burst、维护、故障、扩容和高优任务弹性，避免长期压到无恢复空间。

## Failure Redundancy Fit

一句话定义：Failure Redundancy Fit 是检查 N+1、跨机柜、推理多副本、checkpoint 多副本、维护窗口和故障恢复容量是否进入容量预算。

## Growth Forecast Fit

一句话定义：Growth Forecast Fit 是检查 3 个月、6 个月、12 个月预测、保守 / 中性 / 激进场景、触发扩容阈值和预测偏差回填是否完整。

## Cost Budget Fit

一句话定义：Cost Budget Fit 是检查 GPU、网络、存储、checkpoint、日志、推理 token、云上弹性、运维和闲置成本是否落在预算与归因模型内。

## Pool Separation Fit

一句话定义：Pool Separation Fit 是检查预训练、推理、评估、交互式开发和低优任务是否按 SLO、抢占性、GPU 型号和安全边界分池。

## Quota Burst Governance

一句话定义：Quota Burst Governance 是检查租户配额、burst credit、借用 / 出借、超额抢占、成本归因和人工审批是否配套。

## Observability Forecast Feedback

一句话定义：Observability Forecast Feedback 是检查 GPU 分配率、有效利用率、队列等待、碎片、推理延迟、网络、存储、checkpoint 增长、成本和预测偏差是否进入 dashboard 并周期复盘。

## Cluster Capacity Planning Gate

一句话定义：Cluster Capacity Planning Gate 是判断 AI 集群容量计划是否达到可支撑训练、推理、网络、存储、成本、增长和故障恢复的最低上线门禁。

典型硬阻断：任务画像缺失、GPU 数低于需求、GPU 型号混配错误、训练队列 SLO 缺失、推理峰值未建模、网络带宽不足、存储吞吐未实测、checkpoint 生命周期容量不足、目标利用率过高、没有 N+1 或故障余量、增长预测缺失、成本超过预算、训练推理混池、quota / burst 无治理、dashboard 不含预测回填或没有集群容量规划门禁。

## Training Platform Lifecycle Audit

一句话定义：Training Platform Lifecycle Audit 是把训练平台从脚本启动器升级为覆盖提交、校验、调度、启动、数据、checkpoint、观测、实验追踪、artifact、权限、成本、恢复和状态机的生命周期门禁。

核心直觉：训练平台不是让用户少写几行 `bash train.sh`，而是把一次训练任务的关键决策变成可验证对象。Kubernetes Job、队列系统、训练 operator、`torchrun`、实验追踪和模型仓库分别解决不同层的问题；平台要把它们串成 TrainingJob 生命周期，而不是把任一工具单独说成完整平台。

常见公式：训练任务审计样本可以写成

```math
x_i=(j_i,c_i,r_i,e_i,d_i,l_i,k_i,o_i,m_i,a_i,s_i,u_i)
```

其中，`j_i` 是 TrainingJob 契约，`c_i` 是配置校验，`r_i` 是资源和配额，`e_i` 是镜像、代码和环境，`d_i` 是数据版本和权限，`l_i` 是 launcher 与分布式配置，`k_i` 是 checkpoint 策略，`o_i` 是观测信号，`m_i` 是实验追踪和模型产物，`a_i` 是权限审计，`s_i` 是失败恢复和状态机，`u_i` 是用户自助体验。

可复现字段覆盖率可以写成

```math
R_{\mathrm{repr}}=\frac{K_{\mathrm{captured}}}{K_{\mathrm{required}}}
```

训练平台综合门禁可以写成

```math
G_{\mathrm{train\_platform}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{repr}}\ge \gamma \land A_{\mathrm{lineage}}\ge \eta \land C_{\mathrm{cost}}\ge \kappa \land P_0=0\right]
```

面试表达：训练平台设计题先把 TrainingJob 作为核心对象，说明它要绑定 image digest、code commit、config version、dataset version、resources、distributed config、checkpoint 和 queue；再讲配置校验、权限配额、调度、launcher、数据接入、checkpoint、logs / metrics / events、experiment tracking、artifact registry、失败恢复、成本归因和审计。最后用 Training Platform Gate 收束，而不是只说“支持提交任务、看日志和保存 checkpoint”。

## TrainingJob Contract

一句话定义：TrainingJob Contract 是训练任务的平台化契约，描述 owner、project、image、code commit、entrypoint、config、dataset、resource request、distributed config、checkpoint、queue、priority 和 tags。

## Config Validation

一句话定义：Config Validation 是在任务进入队列前校验 schema、镜像、代码、数据、权限、资源、checkpoint 路径、launcher 参数和 dry run，避免排队后才失败。

## Resource Quota Binding

一句话定义：Resource Quota Binding 是把训练任务的 GPU 类型、GPU 数、每节点 GPU、queue、priority 和 tenant quota 绑定，防止任务绕过配额或抢占策略。

## Image Code Reproducibility

一句话定义：Image Code Reproducibility 是要求训练镜像使用 digest、代码使用 commit、依赖锁定并捕获环境 metadata，保证训练环境可复现。

## Dataset Lineage Permission

一句话定义：Dataset Lineage Permission 是同时检查数据集版本、数据血缘、数据分类和访问授权，避免训练数据只表现为一个无审计路径。

## Launcher Distributed Fit

一句话定义：Launcher Distributed Fit 是检查 torchrun、DeepSpeed、Ray、MPI 或自研 launcher 的节点数、进程数、master、端口和资源 shape 是否一致。

## Checkpoint Resume Policy

一句话定义：Checkpoint Resume Policy 是检查 checkpoint output、保存频率、manifest、恢复演练和最大丢失 step 是否满足恢复 SLO。

## Observability Events Metrics

一句话定义：Observability Events Metrics 是检查日志、指标和生命周期事件是否覆盖 rank 日志、loss、tokens/s、step time、GPU、通信、dataloader、checkpoint 和状态流转。

## Experiment Tracking Lineage

一句话定义：Experiment Tracking Lineage 是把参数、指标、随机种子、artifact、eval report、代码、数据和环境记录到 experiment / run 中，支持比较和复现。

## Artifact Registry Linkage

一句话定义：Artifact Registry Linkage 是把 final / best checkpoint、tokenizer、config、adapter、eval report、training metadata 和权限状态注册为可发布资产。

## Failure Recovery Classification

一句话定义：Failure Recovery Classification 是区分节点故障、网络抖动、存储短暂失败、OOM、配置错误、代码 bug 和权限错误，决定重试、恢复、快速失败或人工介入。

## Security Audit Control

一句话定义：Security Audit Control 是检查提交任务、读取数据、读取日志、取消任务、恢复训练、注册模型和发布模型的权限与审计是否完整。

## Cost Attribution Control

一句话定义：Cost Attribution Control 是把 GPU-hours、CPU / memory、存储、checkpoint、网络、评估推理、日志和 trace 成本归因到 team、project、user、job、model 和 queue。

## Developer Self Service

一句话定义：Developer Self Service 是训练平台为用户提供 CLI、SDK、Portal、模板、状态页、失败提示和诊断报告，让训练能力产品化而不是依赖平台同学人工代跑。

## Lifecycle State Machine

一句话定义：Lifecycle State Machine 是把 submitted、validated、queued、scheduled、running、checkpointing、failed、retrying、completed 和 cancelled 等状态及合法迁移显式化。

## Training Platform Gate

一句话定义：Training Platform Gate 是判断训练平台是否达到可提交、可复现、可观测、可恢复、可治理和可审计最低门槛的综合门禁。

典型硬阻断：TrainingJob 契约缺字段、未做配置 dry run、资源申请未绑定 queue / quota、镜像只用 tag、代码或依赖未锁定、数据权限缺失、launcher 进程数和 GPU 数不一致、checkpoint 无 manifest 或未做恢复演练、缺 rank 日志和生命周期事件、实验追踪缺 seed / eval report、artifact 未注册、失败盲目重试、日志未脱敏、成本无法归因、用户无状态页或状态机允许非法跳转。

## Training Submission Audit

一句话定义：Training Submission Audit 是把训练任务提交入口从“命令提交器”升级为覆盖 schema、镜像、代码、数据、命令、配置、资源、队列、幂等、模板、状态机和 trace 的控制面门禁。

核心直觉：训练任务提交系统是训练平台的第一道治理关口。它要在任务进入队列前绑定 image digest、code commit、dataset version、final config snapshot、structured command、distributed shape、checkpoint URI、queue / priority、client request id 和 audit trace，避免任务排队后才发现无法复现、无权限、配额越界或重复创建。

常见公式：提交审计样本可以写成

```math
z_i=(m_i,s_i,e_i,c_i,d_i,u_i,r_i,l_i,k_i,o_i,q_i,a_i,h_i,t_i,v_i)
```

其中，`m_i` 是 metadata，`s_i` 是 schema，`e_i` 是镜像环境，`c_i` 是代码版本，`d_i` 是数据版本和权限，`u_i` 是启动命令，`r_i` 是资源申请，`l_i` 是 launcher，`k_i` 是 checkpoint，`o_i` 是日志指标，`q_i` 是队列优先级，`a_i` 是 admission / dry run，`h_i` 是幂等信息，`t_i` 是模板版本，`v_i` 是状态机和审计 trace。

必填字段覆盖率可以写成

```math
R_{\mathrm{required}}=\frac{K_{\mathrm{present}}}{K_{\mathrm{required}}}
```

训练提交综合门禁可以写成

```math
G_{\mathrm{train\_submission}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{required}}\ge \gamma \land R_{\mathrm{binding}}\ge \eta \land R_{\mathrm{dup}}=0 \land P_0=0\right]
```

面试表达：训练任务提交系统设计题先讲 TrainingJob schema，再按镜像 digest、代码 commit / diff hash、数据版本权限、结构化 command / args、最终配置快照、资源与分布式一致性、checkpoint、日志指标、queue / priority、ResourceQuota / dry run、client request id 幂等、模板版本和状态机展开。最后用 Training Submission Gate 收束，而不是只说“提供 CLI 提交任务”。

## TrainingJob Schema Completeness

一句话定义：TrainingJob Schema Completeness 是检查 TrainingJob 是否包含 metadata、image、code、data、command、config、resources、distributed、checkpoint、observability、queue、admission、idempotency、template、state 和 audit 等结构化对象。

## Required Field Coverage

一句话定义：Required Field Coverage 是检查 owner、project、job name、image reference、repo、dataset id、entrypoint、config schema、GPU 数、launcher、checkpoint output、日志指标、queue、priority、client request id 和 actor 等必填字段是否完整。

## Image Digest Binding

一句话定义：Image Digest Binding 是要求训练镜像使用不可变 digest，并检查可信 registry、安全扫描和 CUDA / driver 兼容性，避免只记录可漂移 tag。

## Code Commit Diff Binding

一句话定义：Code Commit Diff Binding 是要求训练代码绑定不可变 commit；如果允许本地 diff，必须把 diff 打包并记录 hash。

## Dataset Version Permission

一句话定义：Dataset Version Permission 是同时检查 dataset id、dataset version、数据血缘、访问权限、数据等级和 artifact 权限继承。

## Structured Command Safety

一句话定义：Structured Command Safety 是把启动命令拆成 entrypoint、args、env、working directory 和 launcher 字段，避免任意 shell 字符串绕过校验和审计。

## Final Config Snapshot

一句话定义：Final Config Snapshot 是记录合并 YAML、命令行参数、模板默认值和配置中心覆盖后的最终生效配置快照。

## Distributed Resource Consistency

一句话定义：Distributed Resource Consistency 是检查 `gpu_count = nnodes * nproc_per_node` 且 `world_size = gpu_count`，并确认 rendezvous 和端口策略已经声明。

## Checkpoint URI Resume Validation

一句话定义：Checkpoint URI Resume Validation 是检查 checkpoint output 是否可写、resume path 是否存在、格式是否兼容、保留策略是否明确。

## Logging Metrics Destination

一句话定义：Logging Metrics Destination 是检查日志 URI、指标目标、rank 日志策略、保留天数和脱敏策略是否在提交时已经声明。

## Queue Priority Policy Binding

一句话定义：Queue Priority Policy Binding 是检查用户是否有 queue 和 priority 使用权限，任务是否满足 max runtime、cost budget 和可抢占 checkpoint 策略。

## Quota Dry Run Admission

一句话定义：Quota Dry Run Admission 是在进入队列前用服务端 dry run、ResourceQuota、resource flavor 和策略检查确认任务可被平台接收。

## Idempotency Duplicate Control

一句话定义：Idempotency Duplicate Control 是使用 `client_request_id`、submit token 或幂等窗口，保证网络重试返回同一个 job id，而不是重复创建训练任务。

## Template Version Governance

一句话定义：Template Version Governance 是要求训练模板有 template id、template version、锁定默认值和变更记录，避免模板更新后历史任务不可解释。

## Submit State Machine Validity

一句话定义：Submit State Machine Validity 是检查 draft、submitted、validating、rejected、queued、scheduled、starting、running、completed、failed、retrying 和 cancelled 等状态迁移是否合法。

## Submission Audit Trace

一句话定义：Submission Audit Trace 是记录提交事件、提交人、validator 版本、最终 spec hash、admission 决策和审计日志保留状态。

## Training Submission Gate

一句话定义：Training Submission Gate 是判断训练任务提交系统是否达到可校验、可复现、可幂等、可排队和可审计最低要求的综合门禁。

典型硬阻断：schema 缺结构、必填字段缺失、镜像只用 tag、代码只写 branch、数据用 latest 或权限缺失、启动命令是任意 shell 字符串、最终配置未快照、分布式资源 shape 不一致、checkpoint output 不可写、日志指标缺失、priority 未审批、ResourceQuota / dry run 未通过、缺 client request id、模板使用 latest、状态机非法跳转、audit trace 缺 final spec hash 或没有提交门禁。

## Distributed Launcher Audit

一句话定义：Distributed Launcher Audit 是把 torchrun、DeepSpeed、Megatron、Ray 或 MPI 的启动过程从“拼命令”升级为覆盖资源、rank、rendezvous、GPU 绑定、框架配置、网络、日志、失败恢复和审计 trace 的门禁。

核心直觉：分布式启动器连接调度系统和训练框架。调度系统给出节点、GPU、队列、网络和挂载信息；launcher adapter 要把这些信息转换成 `RANK`、`LOCAL_RANK`、`WORLD_SIZE`、`MASTER_ADDR`、`MASTER_PORT`、`CUDA_VISIBLE_DEVICES`、NCCL 环境和实际启动命令。只要其中一个维度不一致，任务就可能 hang 在初始化、某个 rank 独立 OOM、日志不可查或 checkpoint 无法安全恢复。

常见公式：world size 和资源形状应满足

```math
W_{\mathrm{world}}=N_{\mathrm{node}}P_{\mathrm{node}}
```

Megatron 类模型并行配置应满足

```math
W_{\mathrm{world}}=T_{\mathrm{tp}}P_{\mathrm{pp}}D_{\mathrm{dp}}
```

global batch size 可写成

```math
B_{\mathrm{global}}=B_{\mathrm{micro}}A_{\mathrm{grad}}D_{\mathrm{dp}}
```

启动器综合门禁可以写成

```math
G_{\mathrm{launcher}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land W_{\mathrm{cfg}}=W_{\mathrm{env}} \land P_0=0\right]
```

面试表达：分布式启动器设计题要先讲统一 launcher interface，再讲 torchrun、DeepSpeed、Megatron 和 Ray 的边界；最后用 rank / world size、rendezvous、GPU 绑定、NCCL、日志、失败分类、elastic checkpoint、调度交接和审计 trace 证明平台不是只会拼启动命令。

## Resource World Size Consistency

一句话定义：Resource World Size Consistency 是检查 `world_size = node_count * nproc_per_node`，并确认该值和资源分配、launcher 参数、worker 环境变量一致。

## Rank Local Rank Mapping

一句话定义：Rank Local Rank Mapping 是检查全局 rank 连续唯一、节点内 local rank 与每节点进程数和 GPU 绑定一致。

## Rendezvous Endpoint Readiness

一句话定义：Rendezvous Endpoint Readiness 是检查 master address、master port、端口范围、DNS / IP 可达性和冲突策略是否就绪。

## GPU Binding Visibility

一句话定义：GPU Binding Visibility 是检查 `CUDA_VISIBLE_DEVICES`、local rank、每节点 GPU 数、MIG / 共享策略和训练进程可见设备是否一致。

## Launcher Adapter Fit

一句话定义：Launcher Adapter Fit 是检查 TrainingJob 声明的 launcher 类型与平台选择的 adapter 一致，并且 adapter 支持当前训练代码生态。

## Torchrun Argument Fit

一句话定义：Torchrun Argument Fit 是检查 `nnodes`、`nproc_per_node`、`node_rank`、rendezvous 后端和 master 信息是否与资源分配一致。

## DeepSpeed Config Fit

一句话定义：DeepSpeed Config Fit 是检查 DeepSpeed config、ZeRO stage、hostfile 或节点声明、offload、batch 和 launcher 参数是否彼此一致。

## Megatron Parallel Consistency

一句话定义：Megatron Parallel Consistency 是检查 tensor parallel、pipeline parallel、data parallel、micro batch、gradient accumulation 和 global batch 的乘法关系是否成立。

## Ray Runtime Fit

一句话定义：Ray Runtime Fit 是检查 Ray workers、每 worker GPU、scaling config、runtime env 和外部调度系统资源形状是否匹配。

## Network NCCL Readiness

一句话定义：Network NCCL Readiness 是检查训练网卡选择、NCCL 环境变量、RDMA / GPUDirect 路径、master 可达性和通信日志是否可验证。

## Log Rank Aggregation

一句话定义：Log Rank Aggregation 是检查所有 rank 日志是否被采集、按 node / rank 索引，并在 worker 失败时能定位非 rank 0 的真实错误。

## Failure Stage Classification

一句话定义：Failure Stage Classification 是区分 preflight、container start、distributed init、training runtime 等阶段，避免把网络或 rank 错误误判为训练代码 bug。

## Elastic Training Safety

一句话定义：Elastic Training Safety 是检查弹性训练是否具备 checkpoint 兼容、数据重新分片、global batch 变化处理和恢复策略。

## Checkpoint Launcher Coupling

一句话定义：Checkpoint Launcher Coupling 是检查 checkpoint output、恢复演练、rank 状态、launcher 状态和重启命令是否能一起恢复。

## Scheduler Launcher Handoff

一句话定义：Scheduler Launcher Handoff 是检查调度系统输出的 node list、node rank、GPU 绑定、网络和挂载信息是否完整交给 launcher adapter。

## Launcher Audit Trace

一句话定义：Launcher Audit Trace 是记录最终启动命令 hash、环境变量快照、节点和 rank 拓扑、adapter 版本、失败阶段和恢复决策。

## Distributed Launcher Gate

一句话定义：Distributed Launcher Gate 是判断分布式启动器是否达到可启动、可解释、可恢复和可审计最低要求的综合门禁。

典型硬阻断：world size 与资源分配不一致；rank 重复或 local rank 错绑 GPU；master 地址不可达；launcher adapter 选错；torchrun 参数缺 node rank；DeepSpeed config 与资源不一致；Megatron 并行维度乘积不等于 world size；Ray worker 资源和外部调度不一致；NCCL 网卡缺失；只采集 rank 0 日志；失败阶段不分类；elastic 训练没有 checkpoint 和数据重分片；调度到 launcher 的 node rank / GPU 绑定缺失；没有最终命令和环境快照。

## Training Config Management Audit

一句话定义：Training Config Management Audit 是把训练配置从“保存 YAML”升级为覆盖配置来源、最终快照、版本绑定、校验、复现、恢复、发布、审批、diff 和 trace 的控制面门禁。

核心直觉：训练结果由最终生效配置决定，而不是由某一个 YAML 文件决定。最终配置可能来自默认值、模板、用户 YAML、命令行 override、环境变量、配置中心和平台强制策略。平台必须记录每个字段的最终值、来源、覆盖顺序、merge 规则版本和配置 hash，并把它与 image digest、code commit、dataset version、环境版本、随机种子、checkpoint state、实验 run 和发布配置绑定。

常见公式：最终生效配置可以写成

```math
c^{\star}=M(c_{\mathrm{default}},c_{\mathrm{template}},c_{\mathrm{yaml}},c_{\mathrm{cli}},c_{\mathrm{env}},c_{\mathrm{policy}})
```

batch 和并行配置应满足

```math
B_{\mathrm{global}}=B_{\mathrm{micro}}A_{\mathrm{grad}}D_{\mathrm{dp}}
```

可复现证据覆盖率可以写成

```math
R_{\mathrm{repr}}=\frac{K_{\mathrm{captured}}}{K_{\mathrm{required}}}
```

训练配置门禁可以写成

```math
G_{\mathrm{config}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{required}}\ge \gamma \land R_{\mathrm{repr}}\ge \eta \land G_{\mathrm{resume}}=1 \land P_0=0\right]
```

面试表达：训练配置管理题不要停在“保存 YAML”。要讲清配置来源和覆盖顺序、最终配置快照、字段来源、config hash、代码 / 数据 / 镜像 / 环境 / seed / checkpoint / run 绑定、schema 和跨字段校验、resume 兼容、发布配置耦合、审批和 diff，最后用 Training Config Gate 收束。

## Config Source Order

一句话定义：Config Source Order 是显式定义 default、template、YAML、CLI override、environment 和 platform policy 的覆盖顺序，并记录每个字段的来源。

## Final Config Snapshot Freeze

一句话定义：Final Config Snapshot Freeze 是在任务启动前生成不可变最终配置快照，记录 hash、生成时间、字段来源和 merge 规则版本。

## Required Config Field Coverage

一句话定义：Required Config Field Coverage 是检查模型、数据、优化器、batch、并行、checkpoint、环境、随机种子、实验追踪和审计字段是否完整。

## Config Schema Type Validation

一句话定义：Config Schema Type Validation 是检查字段类型、枚举、范围和必填项，例如 learning rate 为正数、precision 在允许集合内、world size 为正整数。

## Batch Parallel Consistency

一句话定义：Batch Parallel Consistency 是检查 micro batch、gradient accumulation、data parallel、global batch、tensor parallel、pipeline parallel 和 world size 的乘法关系是否一致。

## Immutable Version Binding

一句话定义：Immutable Version Binding 是要求 image digest、code commit、dataset version、tokenizer version 和最终配置 hash 使用不可变引用，避免 latest、branch 或漂移路径。

## Environment Version Capture

一句话定义：Environment Version Capture 是记录 Python、CUDA、PyTorch、NCCL、driver、GPU 型号、launcher 和 platform 版本，解释性能和数值差异。

## Reproducibility Seed Boundary

一句话定义：Reproducibility Seed Boundary 是记录 Python、NumPy、PyTorch、CUDA、dataloader seed 和确定性策略，同时说明严格复现与统计复现边界。

## Experiment Tracking Linkage

一句话定义：Experiment Tracking Linkage 是把最终配置、参数、指标、artifact、代码、数据、环境和 eval report 绑定到同一个 experiment / run。

## Hyperparameter Sweep Trace

一句话定义：Hyperparameter Sweep Trace 是记录 sweep id、trial id、父配置 hash、变化字段和结果，避免超参搜索只留下一堆不可解释任务。

## Checkpoint Resume Compatibility

一句话定义：Checkpoint Resume Compatibility 是检查模型结构、tokenizer、并行策略、optimizer、scheduler、RNG、dataloader cursor 和数据位置是否与 checkpoint 兼容。

## Release Config Coupling

一句话定义：Release Config Coupling 是把训练配置中的模型结构、tokenizer、generation config、eval config、serving config、量化配置和数据 lineage 绑定到模型发布资产。

## Config Permission Approval

一句话定义：Config Permission Approval 是检查高敏数据、生产资源池、高优先级、关闭日志、禁用 checkpoint、未审核镜像和超长运行时间等配置是否经过策略或审批。

## Config Diff Readiness

一句话定义：Config Diff Readiness 是支持两个实验或两次运行之间的配置 diff，帮助解释效果变化来自 learning rate、batch、数据版本、precision、环境还是平台策略。

## Config Audit Trace

一句话定义：Config Audit Trace 是记录 actor、validator version、merge rule、final config hash、policy decision、审批记录和审计日志保留状态。

## Training Config Gate

一句话定义：Training Config Gate 是判断训练配置管理是否达到可校验、可复现、可恢复、可比较、可审批和可审计最低要求的综合门禁。

典型硬阻断：只保存原始 YAML；命令行 override 没记录；最终配置未冻结；配置 hash 缺失；dataset 使用 latest；image 只用 tag；code 只用 branch；batch / parallel 乘法关系不一致；环境版本缺失；只记录 seed 但没有随机性边界；实验 run 没有最终配置；超参 sweep 没有父配置和 diff；resume checkpoint 时模型结构或 tokenizer 不匹配；发布配置和训练配置脱节；高敏数据或高优先级无审批；没有配置 diff；审计 trace 缺 final config hash。

## Training Observability Audit

一句话定义：Training Observability Audit 是把大模型训练任务的 logs、metrics、events、traces、状态机、attempt、实验追踪、告警、权限脱敏、留存成本和成本归因放进同一张可审计表的方法。

核心直觉：训练可观测性不是“多收日志”，而是能回答任务在哪、是否健康、为什么慢、为什么失败、是否可恢复、成本花在哪里，以及训练过程是否能被复盘和复现。日志提供具体上下文，指标提供趋势和聚合，事件提供生命周期变化，trace 串起跨组件阶段耗时，experiment tracking 连接配置、指标、artifact 和最终结果。

常见审计维度：Structured Log Context 看日志是否包含 job、attempt、node、pod、rank、timestamp、level 和 component；Rank Log Coverage 看所有 rank 是否能查询并能定位第一个失败 rank；Metric Taxonomy Coverage 看模型、性能、资源和平台指标是否完整；Metric Dimension Scope 看指标是否能按 job、attempt、rank、node、step 和 timestamp 下钻；Event Schema Lifecycle 看生命周期事件是否结构化；State Machine Validity 看状态迁移是否合法；Attempt Retry Traceability 看多次运行尝试和 checkpoint 恢复是否可区分；Trace Phase Coverage 看 submit、validate、queue、schedule、startup、distributed init、training、checkpoint、eval 和 artifact registration 是否有 span；Dashboard Diagnostic Readiness 看 dashboard 是否覆盖训练曲线、资源、诊断和成本；Alert Rule Readiness 和 Anomaly Detection Rules 看高风险异常是否能触发；Retention Cost Governance、Privacy Redaction Access 和 Cost Attribution Signal 分别检查留存、脱敏权限和成本归因。

常见公式：rank 日志覆盖率可以写成

```math
R_{\mathrm{rank}}=\frac{N_{\mathrm{logged}}}{N_{\mathrm{rank}}}
```

状态迁移合法性可以写成

```math
V_{\mathrm{state}}=\frac{1}{K}\sum_{k=1}^{K}\mathbf{1}[(s_{k-1},s_k)\in E_{\mathrm{allowed}}]
```

trace 阶段覆盖率可以写成

```math
R_{\mathrm{span}}=\frac{K_{\mathrm{observed}}}{K_{\mathrm{required}}}
```

训练可观测性门禁可以写成

```math
G_{\mathrm{obs}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{rank}}\ge \gamma \land V_{\mathrm{state}}=1 \land R_{\mathrm{span}}\ge \eta \land A_{\mathrm{ckpt}}\le \tau_{\mathrm{ckpt}} \land P_0=0\right]
```

面试表达：训练可观测性题不要停在“Prometheus + Grafana”或“看 stdout”。要讲清结构化日志、rank 下钻、模型 / 性能 / 资源 / 平台指标、生命周期事件、状态机、attempt、trace、实验追踪、告警、异常检测、留存成本、权限脱敏和成本归因，最后用 Training Observability Gate 收束。

## Structured Log Context

一句话定义：Structured Log Context 是要求每条训练日志至少带 job、attempt、node、pod、rank、timestamp、level 和 component 等上下文字段，避免分布式排障时只能看无归属文本。

## Rank Log Coverage

一句话定义：Rank Log Coverage 是衡量所有训练 rank 是否都能被查询、过滤和聚合的覆盖率，尤其要发现非 rank 0 的第一个失败或慢 rank。

## Metric Taxonomy Coverage

一句话定义：Metric Taxonomy Coverage 是检查模型指标、性能指标、资源指标和平台指标是否完整，避免只看 loss 曲线就判断训练健康。

## Metric Dimension Scope

一句话定义：Metric Dimension Scope 是检查指标是否带 job、attempt、rank、node、GPU、step 和 timestamp 等维度，支持从全局趋势下钻到具体 rank 或节点。

## Event Schema Lifecycle

一句话定义：Event Schema Lifecycle 是把 submitted、validated、queued、scheduled、started、checkpoint saved、worker failed、retry started、completed 等生命周期变化写成结构化事件。

## State Machine Validity

一句话定义：State Machine Validity 是检查训练任务状态迁移是否符合允许图，防止出现 running 回到 queued、completed 后又 running 或 failed 后 completed 等非法状态。

## Attempt Retry Traceability

一句话定义：Attempt Retry Traceability 是区分逻辑 job、多次 attempt 和具体 run，让日志、指标、事件和 checkpoint 恢复过程不会混在一起。

## Trace Phase Coverage

一句话定义：Trace Phase Coverage 是检查提交、校验、排队、调度、镜像拉取、启动、分布式初始化、训练、checkpoint、评估和 artifact 注册等阶段是否有 trace span。

## Dashboard Diagnostic Readiness

一句话定义：Dashboard Diagnostic Readiness 是要求训练 dashboard 不只展示 loss，还要覆盖任务概览、训练曲线、资源使用、rank 差异、通信、数据加载、checkpoint、错误事件和成本。

## Alert Rule Readiness

一句话定义：Alert Rule Readiness 是检查 loss NaN、GPU 长时间低利用、step time 突增、checkpoint 长时间失败、worker failure、重试过多和预算超限等规则是否就绪。

## Anomaly Detection Rules

一句话定义：Anomaly Detection Rules 是用固定阈值、滑动窗口或同类任务基线发现 loss spike、tokens/s 下降、rank straggler、checkpoint 变慢和数据吞吐下降。

## Retention Cost Governance

一句话定义：Retention Cost Governance 是给高频原始指标、降采样指标、失败日志、发布模型关联日志、审计事件和归档数据设置不同留存周期和成本策略。

## Privacy Redaction Access

一句话定义：Privacy Redaction Access 是对训练日志、trace、工具输出和样本片段做 RBAC、脱敏、敏感字段检测、导出审计和保留控制，防止排障日志变成泄露通道。

## Cost Attribution Signal

一句话定义：Cost Attribution Signal 是把 GPU hours、存储、网络、日志指标存储、owner、project、queue 和 experiment run 关联起来，解释每次训练的成本归属。

## Training Observability Gate

一句话定义：Training Observability Gate 是判断训练任务是否达到可定位、可下钻、可关联、可告警、可恢复、可复盘、可脱敏和可归因最低要求的综合门禁。

典型硬阻断：日志缺 job / attempt / rank；只采 rank 0 日志；指标没有 rank / node / step 维度；只有 loss 没有 step time、GPU、通信、数据加载和 checkpoint 指标；事件不是结构化 schema；状态机非法；attempt 混在一起；trace 缺分布式初始化或 checkpoint span；dashboard 不能诊断慢点；loss NaN 没告警；异常检测缺失；实验 run 无法跳到日志和 checkpoint；日志无脱敏权限；高频指标无限期保留；成本无法归到 owner 或 project；没有最终观测门禁。

## Checkpoint Strategy Audit

一句话定义：Checkpoint Strategy Audit 是把 checkpoint 保存频率、保留策略、异步保存、断点恢复、抢占、评估、发布、成本、权限、监控和恢复演练放进同一张可审计策略表的方法。

核心直觉：Checkpoint 策略不是“每隔多少 step 保存一次”的单参数问题，而是在训练阻塞、最大丢失进度、恢复时间、存储成本、评估需求、发布需求和可抢占资源效率之间做工程权衡。第 10 章的 Checkpoint Lifecycle Audit 更关注 checkpoint 对象是否完整、可恢复、可治理；Checkpoint Strategy Audit 更关注何时保存、保存哪类、保留多久、从哪里恢复以及和平台生命周期如何联动。

常见审计维度：RPO RTO Budget 看最大丢失进度和恢复耗时是否有目标；Save Trigger Policy 看是否支持按 step、按时间、milestone 和 preemption notice 保存；Checkpoint Scope Separation 看 resume、eval、release 三类 checkpoint 是否分清；Async Snapshot Consistency 和 Manifest Commit Integrity 看异步保存是否有一致 snapshot、临时目录、checksum 和 committed-only 恢复语义；Shard Metadata Resharding 看分片元数据是否支持恢复和可能的 resharding；Restore Selection Policy 看是否只从健康、已提交、配置兼容、数据可访问、权限允许的 checkpoint 恢复；Preemption Checkpoint Coupling 看可抢占队列是否绑定 checkpoint 新鲜度；Evaluation Best Linkage 和 Release Checkpoint Readiness 看 checkpoint 是否能驱动 best eval 和发布注册；Retention Tier Lifecycle、Storage Cost Budget、Checkpoint Monitoring Alerts、Restore Drill Coverage 和 Permission Release Protection 则检查保留、成本、告警、恢复演练和发布权限。

常见公式：按 step 保存的最大丢失时间可以写成

```math
A_{\mathrm{step}}=\frac{I_{\mathrm{step}}T_{\mathrm{step}}}{60}
```

混合保存触发下的最大丢失时间可以写成

```math
A_{\mathrm{lost}}=\min(A_{\mathrm{step}},A_{\mathrm{time}})
```

保存阻塞比例可以写成

```math
R_{\mathrm{block}}=\frac{T_{\mathrm{block}}}{I_{\mathrm{step}}T_{\mathrm{step}}+T_{\mathrm{block}}}
```

保留成本可以写成

```math
K_{\mathrm{retain}}=\sum_{k=1}^{K}N_kS_kD_kP_k+K_{\mathrm{request}}+K_{\mathrm{ops}}
```

Checkpoint 策略门禁可以写成

```math
G_{\mathrm{ckpt\_strategy}}=\mathbf{1}\left[A_{\mathrm{lost}}\le \mathrm{RPO} \land T_{\mathrm{restore}}\le \mathrm{RTO} \land R_{\mathrm{block}}\le \rho_{\mathrm{block}} \land K_{\mathrm{retain}}\le B_{\mathrm{storage}} \land \min_j C_j\ge \tau_j \land P_0=0\right]
```

面试表达：checkpoint 策略题不要只回答“每 1000 step 保存”。要先讲 RPO/RTO，再讲保存触发、resume / eval / release 分层、异步 snapshot 和 manifest commit、分片元数据、恢复选择、抢占联动、评估 best、发布注册、保留成本、监控告警、恢复演练和权限保护，最后用 Checkpoint Strategy Gate 收束。

## RPO RTO Budget

一句话定义：RPO RTO Budget 是用可接受的最大丢失进度和恢复耗时约束 checkpoint 保存频率、保存大小、加载路径和恢复流程。

## Save Trigger Policy

一句话定义：Save Trigger Policy 是把按 step、按时间、milestone、评估点、发布候选和 preemption notice 等触发条件统一纳入平台策略。

## Checkpoint Scope Separation

一句话定义：Checkpoint Scope Separation 是区分 resume、eval、release checkpoint 的内容和用途，避免所有 checkpoint 都保存完整训练状态或发布版本缺 tokenizer / config / eval report。

## Async Snapshot Consistency

一句话定义：Async Snapshot Consistency 是异步保存前先在一致边界冻结需要保存的状态，避免模型参数、optimizer、scheduler、RNG 和 dataloader cursor 来自不同 step。

## Manifest Commit Integrity

一句话定义：Manifest Commit Integrity 是先写临时目录和 shard checksum，全部完成后再提交 manifest，并且恢复时只读取 committed checkpoint。

## Shard Metadata Resharding

一句话定义：Shard Metadata Resharding 是在分片 checkpoint 中记录 rank map、tensor shards、global step、format version 和 reshard policy，支撑恢复和并行策略变化。

## Restore Selection Policy

一句话定义：Restore Selection Policy 是规定从 latest committed、用户指定、最近健康、best eval 或 milestone checkpoint 恢复时需要执行的完整性、配置、数据、权限和事件记录校验。

## Preemption Checkpoint Coupling

一句话定义：Preemption Checkpoint Coupling 是把可抢占任务、preemption notice、checkpoint 新鲜度和调度器抢占选择联动，降低抢占造成的训练进度损失。

## Evaluation Best Linkage

一句话定义：Evaluation Best Linkage 是在 checkpoint 后触发版本化评估，把 eval report、best checkpoint、模型指标和训练 run 关联起来。

## Release Checkpoint Readiness

一句话定义：Release Checkpoint Readiness 是发布前检查权重转换、tokenizer、config、generation config、推理加载测试、评估报告、安全审核、metadata 注册和权限标记。

## Retention Tier Lifecycle

一句话定义：Retention Tier Lifecycle 是把最近 resume、周期 milestone、best eval、release 和 failed temporary checkpoint 放到不同保留周期和存储层。

## Storage Cost Budget

一句话定义：Storage Cost Budget 是用 checkpoint 大小、保留个数、保留天数、存储层价格和请求运维成本约束 checkpoint 策略的月度成本。

## Checkpoint Monitoring Alerts

一句话定义：Checkpoint Monitoring Alerts 是监控 save / load duration、checkpoint age、失败率、临时目录堆积、存储用量、恢复成功率和抢占无新鲜 checkpoint 等指标并触发告警。

## Restore Drill Coverage

一句话定义：Restore Drill Coverage 是定期演练从 checkpoint 恢复训练和加载发布模型，证明保存成功真的等于可恢复。

## Permission Release Protection

一句话定义：Permission Release Protection 是对 release checkpoint、best eval、归档版本和高敏数据关联 checkpoint 做 RBAC、审计、保护标记和归档策略控制。

## Checkpoint Strategy Gate

一句话定义：Checkpoint Strategy Gate 是判断 checkpoint 策略是否达到低损失恢复、低训练阻塞、可控存储成本、可抢占、可评估、可发布、可审计和可演练最低要求的综合门禁。

典型硬阻断：没有 RPO/RTO；只按 step 保存但 step time 变化导致 RPO 失控；所有 checkpoint 都保存完整训练状态；只保留 latest；异步保存没有 snapshot / temp path / checksum / manifest commit；恢复读取半成品；分片 metadata 缺失；抢占不看 checkpoint 新鲜度；best eval 没有关联 eval report；release checkpoint 缺 tokenizer、config 或加载测试；保留策略无限增长；没有 checkpoint save / load / age / failure 告警；从未做恢复演练；release checkpoint 权限开放；没有最终策略门禁。

## Training Fault Tolerance Audit

一句话定义：Training Fault Tolerance Audit 是把节点失败、GPU 故障、通信 hang、OOM、数据权限、checkpoint 恢复、数值异常、自动重试、诊断报告和用户动作放进同一张可审计容错表的方法。

核心直觉：训练容错不是无限重试，而是先判断故障是否可恢复，再用有边界的 attempt、backoff、checkpoint 恢复和节点 / GPU 隔离控制损失。可恢复故障要自动恢复，不可恢复故障要快速失败并给出诊断，否则平台会把 GPU 花在重复失败上。

常见审计维度：Fault Taxonomy Coverage 看故障分类是否覆盖 node、GPU、network、OOM、data、checkpoint、config 和 numeric；Retryability Classification 看 retryable / non-retryable 是否和错误类型一致；Retry Budget Bounded 和 Retry Backoff Policy 看 attempt、timeout、backoff 和 jitter 是否有边界；Checkpoint Resume Readiness 看 committed checkpoint、manifest、checksum、配置兼容和训练状态是否完整；Node Failure Reschedule、GPU Health Isolation 和 Communication Hang Detection 看节点替换、坏 GPU 隔离、rank heartbeat、首个失败 rank、NCCL 日志和 worker 退出状态是否可见；OOM Handling Policy 看 OOM 是否快速失败、给出显存峰值和调参建议，而不是原样重试；Data Permission Fast Fail、Checkpoint Commit Safety 和 Numeric Instability Stop Rollback 看权限错误、半成品 checkpoint 和 NaN 是否被阻断；Diagnosis Report Evidence、Blacklist Precision、User Action Clarity 和 Training Fault Tolerance Gate 则检查报告、黑名单误伤、用户可见动作和最终门禁。

常见公式：故障样本可以写成

```math
f_i=(c_i,r_i,a_i,b_i,k_i,n_i,g_i,h_i,o_i,d_i,p_i,s_i,u_i,t_i,z_i)
```

统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(f_i)=1]
```

失败后的最大丢失 GPU-hours 可以写成

```math
L_{\mathrm{gpu}}=G\frac{t_{\mathrm{fail}}-t_{\mathrm{ckpt}}}{60}
```

无效重试率可以写成

```math
R_{\mathrm{waste}}=\frac{N_{\mathrm{bad\_retry}}}{N_{\mathrm{retry}}}
```

OOM 显存余量可以写成

```math
H_{\mathrm{mem}}=1-\frac{M_{\mathrm{peak}}}{M_{\mathrm{limit}}}
```

训练容错门禁可以写成

```math
G_{\mathrm{fault}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land L_{\mathrm{gpu}}\le B_{\mathrm{loss}} \land R_{\mathrm{waste}}\le \rho_{\mathrm{waste}} \land A_{\mathrm{hb}}\le \tau_{\mathrm{hb}} \land V_{\mathrm{resume}}=1 \land P_0=0\right]
```

面试表达：训练容错题不要只说“失败自动重试”。要先讲故障分类和 retryability，再讲 max attempts、backoff、checkpoint committed 恢复、节点替换、GPU 隔离、NCCL hang 诊断、OOM 快速失败、数据权限快速失败、NaN 回滚、诊断报告和用户动作，最后用 Training Fault Tolerance Gate 收束。

## Fault Taxonomy Coverage

一句话定义：Fault Taxonomy Coverage 是检查训练平台是否覆盖节点、GPU、网络、OOM、数据、checkpoint、配置、代码和数值稳定性等故障类型。

## Retryability Classification

一句话定义：Retryability Classification 是把故障分成可自动重试、需要调整配置后重试、必须快速失败和需要人工介入几类，避免对权限、配置、持续 OOM 和 NaN 盲目重试。

## Retry Budget Bounded

一句话定义：Retry Budget Bounded 是为每个训练任务设置最大 attempt、总重试时长、失败次数阈值和不可恢复错误短路规则。

## Retry Backoff Policy

一句话定义：Retry Backoff Policy 是用指数退避、固定退避或带 jitter 的等待策略控制自动重试节奏，避免任务集体重启压垮控制面或存储。

## Checkpoint Resume Readiness

一句话定义：Checkpoint Resume Readiness 是确认恢复点是 committed checkpoint，且 manifest、checksum、配置兼容、optimizer、scheduler、RNG 和 dataloader cursor 都可用。

## Node Failure Reschedule

一句话定义：Node Failure Reschedule 是节点或 worker 失败后重新选择健康节点，重启 worker group，并从最新健康 checkpoint 恢复。

## GPU Health Isolation

一句话定义：GPU Health Isolation 是用 Xid、ECC、GPU reset、掉卡和多任务失败证据隔离坏 GPU 或坏节点，避免同一硬件反复拖垮训练任务。

## Communication Hang Detection

一句话定义：Communication Hang Detection 是用 rank heartbeat、首个失败 rank、NCCL 日志、worker 退出状态和网络指标区分网络问题、rank OOM、配置错误和节点失联。

## OOM Handling Policy

一句话定义：OOM Handling Policy 是区分 GPU OOM、CPU OOM 和 container OOM，并给出 batch、sequence length、activation checkpointing、FSDP / ZeRO 和显存峰值建议，而不是原样重试。

## Data Permission Fast Fail

一句话定义：Data Permission Fast Fail 是在启动前和读取时快速识别数据权限、路径、shard 和格式错误，对权限类错误直接失败并生成诊断。

## Checkpoint Commit Safety

一句话定义：Checkpoint Commit Safety 是只允许恢复 committed checkpoint，禁止读取临时目录、半成品 shard 或 checksum / manifest 不完整的 checkpoint。

## Numeric Instability Stop Rollback

一句话定义：Numeric Instability Stop Rollback 是对 loss NaN、Inf、梯度爆炸和 AMP overflow 停止训练、保存诊断并回滚到健康 checkpoint，而不是盲目重试。

## Diagnosis Report Evidence

一句话定义：Diagnosis Report Evidence 是把失败时间、阶段、首个失败 rank、错误类型、日志、指标、checkpoint、attempt 和建议动作放进可读报告。

## Blacklist Precision

一句话定义：Blacklist Precision 是用硬件和系统证据决定 node / GPU blacklist，同时避免把用户代码 OOM、配置错误或数据错误误判成坏节点。

## User Action Clarity

一句话定义：User Action Clarity 是让用户看到当前状态、重试次数、恢复 checkpoint、损失进度、是否需要操作和下一步建议。

## Training Fault Tolerance Gate

一句话定义：Training Fault Tolerance Gate 是判断训练任务是否达到可分类、可恢复、可短路、低损失、可隔离、可诊断、可解释和可审计最低要求的综合门禁。

典型硬阻断：故障分类缺失；权限、配置、持续 OOM 和 NaN 被自动重试；attempt 无上限；没有 backoff；checkpoint 未 committed 就恢复；节点失败后仍调度到原节点；Xid / ECC 多次失败但不隔离；NCCL hang 无 rank heartbeat 和日志；OOM 无显存峰值和建议；数据权限错误被重试；数值异常继续训练；诊断报告缺日志、指标和建议；用户只看到 failed；黑名单把用户错误误伤成硬件故障；没有最终训练容错门禁。

## Training Data Supply Audit

一句话定义：Training Data Supply Audit 是把大模型训练数据从 manifest、shard、流式读取、缓存、shuffle、rank 切分、DataLoader、H2D copy、坏样本、观测和 checkpoint 数据状态放进同一张可审计表的方法。

核心直觉：训练数据供给不是“路径能打开”，而是在保证数据版本、权限、shuffle 和可复现性的前提下，把数据以足够高、足够稳定的吞吐送到 GPU。GPU 利用率低时，根因可能是对象存储、缓存、解析、预处理、DataLoader worker、H2D copy、rank 不均衡或坏样本，而不一定是模型算子慢。

常见审计维度：Dataset Manifest Integrity 看 dataset id、版本、shard、checksum、token 数、预处理版本和权限元数据是否完整；Shard Format Size Fit 和 Small File Amplification Control 看格式、索引、shard 大小、小文件数量和 metadata 压力是否适合训练；Streaming Prefetch Retry、Local Cache Policy 和 Distributed Cache Policy 看流式读取、预取、重试、本地缓存、分布式缓存、版本隔离和容量治理；Shuffle Reproducibility 与 Rank Shard Assignment 看 shard shuffle、shuffle buffer、seed、`set_epoch`、rank 不重不漏和恢复 cursor；Data Locality Fit 看训练节点是否靠近数据和缓存；Dataloader Parallelism Fit 与 H2D Copy Readiness 看 worker、prefetch、CPU 预处理、pinned memory 和异步拷贝；Bad Sample Threshold、Data Supply Observability、Checkpoint Dataloader State、Permission Version Governance 和 Training Data Supply Gate 则检查坏样本阈值、指标、断点恢复数据状态、权限版本和最终门禁。

常见公式：训练数据供给样本可以写成

```math
d_i=(m_i,s_i,a_i,u_i,c_i,q_i,r_i,\ell_i,p_i,h_i,b_i,o_i,k_i,v_i,z_i)
```

数据供给吞吐可以写成

```math
Q_{\mathrm{data}}=\frac{N_{\mathrm{tok}}}{T_{\mathrm{load}}}
```

GPU 等数据比例可以写成

```math
R_{\mathrm{wait}}=\frac{T_{\mathrm{data\_wait}}}{T_{\mathrm{step}}}
```

缓存命中率可以写成

```math
R_{\mathrm{cache}}=\frac{B_{\mathrm{cache}}}{B_{\mathrm{cache}}+B_{\mathrm{remote}}}
```

坏样本率可以写成

```math
R_{\mathrm{bad}}=\frac{N_{\mathrm{bad}}}{N_{\mathrm{seen}}}
```

训练数据供给门禁可以写成

```math
G_{\mathrm{data}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land Q_{\mathrm{data}}\ge \gamma Q_{\mathrm{train}} \land R_{\mathrm{wait}}\le \rho_{\mathrm{wait}} \land R_{\mathrm{cache}}\ge \rho_{\mathrm{cache}} \land R_{\mathrm{bad}}\le \rho_{\mathrm{bad}} \land P_0=0\right]
```

面试表达：训练数据供给题不要只回答“用 DataLoader 读数据”。要先讲离线预处理、shard 和 manifest，再讲 streaming / cache、shuffle 和 rank 切分，接着讲 DataLoader、H2D、坏样本、监控和 checkpoint 数据状态，最后用 Training Data Supply Gate 收束。

## Dataset Manifest Integrity

一句话定义：Dataset Manifest Integrity 是检查数据集版本、shard 列表、token 数、checksum、预处理版本、来源和权限元数据是否完整且不可变。

## Shard Format Size Fit

一句话定义：Shard Format Size Fit 是检查训练数据格式、shard 大小、索引、压缩和可并行读取能力是否适合大规模训练。

## Small File Amplification Control

一句话定义：Small File Amplification Control 是控制小文件数量、metadata 请求和打开文件开销，避免对象存储或共享文件系统被小文件放大效应拖垮。

## Streaming Prefetch Retry

一句话定义：Streaming Prefetch Retry 是为远端流式读取配置预取、超时、重试和吞吐监控，避免网络抖动直接变成 GPU 等数据。

## Local Cache Policy

一句话定义：Local Cache Policy 是把热点 shard 按数据版本预热或缓存到本地 NVMe，并用容量、LRU 和命中率治理缓存。

## Distributed Cache Policy

一句话定义：Distributed Cache Policy 是用共享缓存层缓解对象存储压力，同时保证版本一致性、容量规划和缓存层不成为新瓶颈。

## Shuffle Reproducibility

一句话定义：Shuffle Reproducibility 是在 shard shuffle、shuffle buffer、epoch seed、`set_epoch` 和 rank 切分之间平衡随机性、吞吐和可复现。

## Rank Shard Assignment

一句话定义：Rank Shard Assignment 是给每个 distributed rank 分配不重不漏且负载均衡的 shard，并记录恢复 cursor。

## Data Locality Fit

一句话定义：Data Locality Fit 是让训练节点尽量靠近数据或缓存，减少跨区域读取、网络拥塞、读取延迟和不必要流量成本。

## Dataloader Parallelism Fit

一句话定义：Dataloader Parallelism Fit 是检查 DataLoader worker、persistent worker、prefetch、CPU 预处理和 batch packing 是否支撑训练吞吐。

## H2D Copy Readiness

一句话定义：H2D Copy Readiness 是检查 CPU 到 GPU 拷贝是否使用 pinned memory、异步 copy 和合理 copy 预算，避免最后一段传输卡住。

## Bad Sample Threshold

一句话定义：Bad Sample Threshold 是记录坏样本位置、坏样本率和失败阈值，避免无限跳过坏样本导致训练分布静默变化。

## Data Supply Observability

一句话定义：Data Supply Observability 是采集 dataloader time、data wait、storage throughput、cache hit、shard latency、bad sample、tokens loaded、CPU preprocess、H2D 和 per-rank lag 等指标。

## Checkpoint Dataloader State

一句话定义：Checkpoint Dataloader State 是在训练 checkpoint 中记录 epoch、global step、shard index、sample offset、shuffle seed 和 worker state，避免恢复后重复或跳过数据。

## Permission Version Governance

一句话定义：Permission Version Governance 是把数据权限、数据分类、dataset version 和 preprocessing version 绑定到训练任务和 manifest。

## Training Data Supply Gate

一句话定义：Training Data Supply Gate 是判断训练数据供给是否达到版本正确、高吞吐、低等待、缓存有效、shuffle 可复现、rank 不重不漏、坏样本可控和可恢复最低要求的综合门禁。

典型硬阻断：没有 manifest；数据版本可变；checksum 缺失；shard 过小或无索引；小文件数量过多；流式读取无预取和重试；缓存无版本隔离；分布式缓存变成瓶颈；shuffle 无 seed；rank 读重或漏读；跨区域读训练数据；DataLoader worker 不足；H2D copy 阻塞；坏样本无限跳过；没有 data wait / cache hit / per-rank lag 指标；checkpoint 不保存 dataloader state；数据权限未校验；没有最终数据供给门禁。

## Data Platform Supply Chain Audit

一句话定义：Data Platform Supply Chain Audit 是把大模型数据从 source registry、raw lake、cleaning、dedup、PII redaction、quality scoring、dataset builder、manifest、version、permission、lineage、observability 到训练读取吞吐放进同一张可审计表的方法。

核心直觉：数据平台不是文件目录，也不只是训练时 DataLoader 的上游。它要证明某批数据从哪里来、为什么能用、经过哪版清洗和去重、是否阻断 PII 和污染、manifest 是否完整、版本是否不可变、用途权限是否匹配、血缘是否覆盖，以及训练平台能否以足够吞吐稳定消费。

常见审计维度：Source Registry Coverage 看来源、owner、license、用途、分类、接入时间和 checksum；Raw Data Lake Integrity 看原始对象是否不可变、checksum 和审计日志是否完整；Cleaning Policy Versioning、Dedup Contamination Guard 和 PII Redaction Coverage 看清洗、去重、污染隔离和隐私阻断是否版本化；Quality Scoring Calibration 和 Dataset Builder Recipe 看质量分、slice 统计、配方和采样权重是否可解释；Dataset Manifest Completeness 和 Dataset Version Immutability 看 manifest 与版本是否完整且冻结；Streaming Shard Readiness、Read Throughput Fit 和 Data Wait Fit 看 shard、streaming、cache、预取、重试、读取吞吐和 GPU 等数据比例；Permission Purpose Binding、Data Lineage Coverage、Data Platform Supply Observability 和 Data Platform Supply Gate 则检查用途权限、血缘、指标告警和最终门禁。

常见公式：数据平台样本可以写成

```math
d_i=(s_i,r_i,c_i,v_i,m_i,q_i,p_i,g_i,l_i,t_i,h_i,o_i,z_i)
```

来源注册覆盖率可以写成

```math
C_{\mathrm{source}}=\frac{K_{\mathrm{source,ok}}}{K_{\mathrm{source,total}}}
```

Manifest 完整率可以写成

```math
C_{\mathrm{manifest}}=\frac{K_{\mathrm{manifest,filled}}}{K_{\mathrm{manifest,required}}}
```

PII 泄露率可以写成

```math
R_{\mathrm{pii,leak}}=\frac{N_{\mathrm{pii,unblocked}}}{N_{\mathrm{pii,total}}}
```

血缘覆盖率可以写成

```math
C_{\mathrm{lineage}}=\frac{N_{\mathrm{samples,linked}}}{N_{\mathrm{samples,total}}}
```

有效读取吞吐和等数据比例可以写成

```math
Q_{\mathrm{read}}=\frac{N_{\mathrm{tok}}}{T_{\mathrm{read}}}
```

```math
R_{\mathrm{wait}}=\frac{T_{\mathrm{wait,data}}}{T_{\mathrm{step}}}
```

数据平台供给门禁可以写成

```math
G_{\mathrm{data}}=\mathbf{1}\left[\min_j C_j\ge\tau_j \land R_{\mathrm{pii,leak}}=0 \land C_{\mathrm{lineage}}\ge\rho \land Q_{\mathrm{read}}\ge\gamma Q_{\mathrm{train}} \land R_{\mathrm{wait}}\le\rho_{\mathrm{wait}} \land P_0=0\right]
```

面试表达：数据平台题不要直接跳到“Parquet + 对象存储 + DataLoader”。先讲来源和 raw lake，再讲清洗、去重、PII、质量、builder 和 manifest，然后讲版本、权限、血缘和观测，最后用读取吞吐、等数据比例和 Data Platform Supply Gate 证明它能被训练平台可靠消费。

## Source Registry Coverage

一句话定义：Source Registry Coverage 是检查数据源的来源、owner、许可证、用途、分类、接入时间、原始路径和 checksum 是否完整登记。

## Raw Data Lake Integrity

一句话定义：Raw Data Lake Integrity 是检查原始数据对象是否不可变、可校验、可审计、可按来源和时间回溯，并且不会被清洗产物覆盖。

## Cleaning Policy Versioning

一句话定义：Cleaning Policy Versioning 是把解析、语言识别、低质过滤、格式规范化、安全过滤和保留率统计绑定到明确策略版本。

## Dedup Contamination Guard

一句话定义：Dedup Contamination Guard 是同时检查重复样本、近似重复、train / eval overlap、benchmark 污染和去重规则版本的门禁。

## PII Redaction Coverage

一句话定义：PII Redaction Coverage 是检查检测到的个人可识别信息是否被删除、脱敏、隔离或审批，不允许未阻断 PII 直接进入训练集。

## Quality Scoring Calibration

一句话定义：Quality Scoring Calibration 是检查质量评分策略、校准报告、slice 统计和用途边界是否可解释，避免把所有数据混成一个不可追溯总分。

## Dataset Builder Recipe

一句话定义：Dataset Builder Recipe 是记录数据来源、清洗策略、采样权重、split、权限约束和输出格式的可复现数据集构建配方。

## Dataset Manifest Completeness

一句话定义：Dataset Manifest Completeness 是检查 manifest 是否包含 dataset id、version、sources、recipe、清洗 / 去重 / PII 策略版本、shard、样本数、token 数、checksum、权限版本、lineage run id 和质量统计。

## Dataset Version Immutability

一句话定义：Dataset Version Immutability 是保证同一个 dataset version 指向不可变 manifest、shard checksum、配方和权限快照。

## Streaming Shard Readiness

一句话定义：Streaming Shard Readiness 是检查 shard 格式、大小、checksum、索引、streaming、shuffle、预取、重试和缓存版本隔离是否适合训练读取。

## Permission Purpose Binding

一句话定义：Permission Purpose Binding 是把 actor、project、dataset version、classification、allowed purpose 和训练任务提交绑定，避免“有路径就能读”。

## Data Lineage Coverage

一句话定义：Data Lineage Coverage 是检查样本、shard、dataset version、清洗任务、去重任务、builder run、训练任务、checkpoint 和 eval report 是否能连成血缘链。

## Data Platform Supply Observability

一句话定义：Data Platform Supply Observability 是让数据平台暴露来源覆盖、manifest 完整、PII 泄露、血缘覆盖、读取吞吐、等数据比例、权限拒绝和质量分布等指标与告警。

## Read Throughput Fit

一句话定义：Read Throughput Fit 是判断数据平台提供给训练任务的有效 token 读取吞吐是否高于训练侧消耗速度。

## Data Wait Fit

一句话定义：Data Wait Fit 是判断训练 step 中等待数据的时间比例是否低于阈值，避免 GPU 因数据供给链路空转。

## Data Platform Supply Gate

一句话定义：Data Platform Supply Gate 是判断数据平台是否达到来源完整、原始数据可回溯、清洗去重可复现、PII 零泄露、manifest 完整、版本冻结、权限用途绑定、血缘覆盖、观测可用、吞吐足够和 P0 风险清零的综合门禁。

典型硬阻断：来源缺 owner、license 或 checksum；raw data 可覆盖；清洗策略未版本化；train / eval overlap 未隔离；PII 未阻断；质量分无校准；builder 配方缺失；manifest 缺 shard checksum 或 lineage run id；dataset version 可变；streaming shard 不可读；权限不绑定用途；血缘覆盖不足；缺读取吞吐和等数据指标；`Q_read` 低于训练消耗速度；`R_wait` 超阈值；没有最终数据平台供给门禁。

## Data Version Lineage Quality Audit

一句话定义：Data Version Lineage Quality Audit 是把 dataset version、manifest checksum、不可变 snapshot、diff report、lineage graph、影响分析、质量指标、漂移监控、泄漏监控、告警路由和训练 / 模型版本联动放进同一张审计表的方法。

核心直觉：数据版本治理不是给目录起名字，数据血缘也不是只服务合规。大模型训练要能解释“为什么这次训练变好或变差”，就必须知道用的是哪个不可变数据版本、它和上一个版本差在哪里、哪些 source / rule / shard 参与了构建、哪些训练任务和模型消费了它，以及新版本是否通过质量和安全门禁。

常见审计维度：Dataset Version Contract 看 dataset id、版本、父版本、owner、schema、tokenizer、recipe 和 split seed 是否完整；Manifest Checksum Integrity 看 manifest、shard checksum、样本数和 token 数是否一致；Immutable Snapshot Policy 看发布后是否冻结；Delta Version Diff 看新增、删除、token delta、来源比例和规则变化是否解释清楚；Lineage Graph Completeness 与 Impact Analysis Readiness 看 source、job、dataset、training run、checkpoint、eval、model 和 deployment 是否可达；Schema Quality Gate、Content Quality Gate、Data Safety Compliance Gate、Distribution Drift Monitoring、Train Eval Leakage Monitoring、Annotation Quality Monitoring 和 Synthetic Data Monitoring 检查 schema、重复、低质、PII、license、分布漂移、评估泄漏、标注和合成数据；Quality Alert Routing 和 Training Model Version Linkage 检查告警上下文、TrainingJob 和 ModelVersion 是否绑定；最后用 Data Version Quality Gate 收束。

常见公式：dataset version 可以写成

```math
V_k=(D_k,M_k,R_k,S_k,P_k,Q_k,L_k,H_k,A_k)
```

血缘图可以写成

```math
G_{\mathrm{lin}}=(N,E)
```

血缘覆盖率可以写成

```math
C_{\mathrm{lineage}}=\frac{N_{\mathrm{linked}}}{N_{\mathrm{expected}}}
```

离散分布漂移可以写成

```math
D_{\mathrm{tv}}(p,q)=\frac{1}{2}\sum_{c\in C}|p_c-q_c|
```

数据质量门禁可以写成

```math
G_{\mathrm{quality}}=\mathbf{1}\left[C_{\mathrm{manifest}}=1 \land C_{\mathrm{checksum}}=1 \land C_{\mathrm{lineage}}\ge\rho \land R_{\mathrm{schema}}\ge\tau_{\mathrm{schema}} \land R_{\mathrm{dup}}\le\rho_{\mathrm{dup}} \land R_{\mathrm{pii,open}}=0 \land R_{\mathrm{overlap}}\le\rho_{\mathrm{overlap}} \land D_{\mathrm{tv}}\le\rho_{\mathrm{drift}} \land P_0=0\right]
```

面试表达：这类题先讲 version contract 和 immutable snapshot，再讲 manifest checksum 和 diff report，然后用 lineage graph 做删除、撤权和污染影响分析，接着列 schema、重复、PII、license、drift、overlap、annotation、synthetic data 等质量门禁，最后说明告警必须带 dataset、version、source、job 和 rule version，并把 TrainingJob / ModelVersion 绑定到数据版本。

## Dataset Version Contract

一句话定义：Dataset Version Contract 是 dataset id、version、parent version、owner、schema、tokenizer、recipe、split seed、权限和状态机组成的数据版本契约。

## Manifest Checksum Integrity

一句话定义：Manifest Checksum Integrity 是检查 manifest checksum、shard checksum、样本数、token 数和文件列表是否一致。

## Immutable Snapshot Policy

一句话定义：Immutable Snapshot Policy 是保证已发布 dataset version 不被覆盖，修复或变更必须产生新版本。

## Delta Version Diff

一句话定义：Delta Version Diff 是记录两个数据版本之间新增、删除、token delta、来源比例、规则版本和质量指标变化的报告。

## Lineage Graph Completeness

一句话定义：Lineage Graph Completeness 是检查 source、ingestion、raw dataset、cleaning、dedup、dataset version、training job、checkpoint、eval report、model version 和 deployment 是否连成可追踪图。

## Impact Analysis Readiness

一句话定义：Impact Analysis Readiness 是在数据删除、撤权、许可证风险或污染发现后，能沿血缘图定位受影响 dataset、训练任务、checkpoint、模型和 deployment。

## Schema Quality Gate

一句话定义：Schema Quality Gate 是用 schema 合法率、解析成功率、空字段比例、标签缺失率和多模态文件存在率拦截结构异常数据版本。

## Content Quality Gate

一句话定义：Content Quality Gate 是用重复率、低质量比例、乱码比例、长度分布、语言 / 领域比例和质量分分布拦截内容异常数据版本。

## Data Safety Compliance Gate

一句话定义：Data Safety Compliance Gate 是用 PII 未阻断率、secret 命中率、许可证风险比例、敏感等级和 train / eval overlap 拦截安全合规风险。

## Distribution Drift Monitoring

一句话定义：Distribution Drift Monitoring 是持续比较新旧数据版本在语言、领域、来源、长度、合成比例和质量分上的分布差异。

## Train Eval Leakage Monitoring

一句话定义：Train Eval Leakage Monitoring 是对 train 和 eval 之间的 exact duplicate、near duplicate、benchmark contamination 和来源交叉进行持续监控。

## Annotation Quality Monitoring

一句话定义：Annotation Quality Monitoring 是监控标注员一致性、gold set 准确率、标签分布、冲突样本、审核通过率和标注指南版本。

## Synthetic Data Monitoring

一句话定义：Synthetic Data Monitoring 是监控合成数据生成模型、prompt 版本、采样参数、合成比例、多样性、事实性、难度分布和偏差。

## Quality Alert Routing

一句话定义：Quality Alert Routing 是让质量告警带上 dataset、version、source、job、rule version 和责任 owner，避免只有指标没有可执行定位。

## Training Model Version Linkage

一句话定义：Training Model Version Linkage 是把 TrainingJob 的 dataset version、manifest checksum、tokenizer、sampling seed、代码和环境记录到 ModelVersion 血缘中。

## Data Version Quality Gate

一句话定义：Data Version Quality Gate 是判断数据版本是否达到契约完整、manifest 和 checksum 一致、snapshot 不可变、diff 可解释、血缘完整、影响分析可用、质量和安全指标过线、告警可路由、训练 / 模型版本可追溯的综合门禁。

典型硬阻断：dataset version 缺 owner、schema 或 tokenizer；manifest checksum 缺失；shard 数和样本数不一致；发布后覆盖 snapshot；diff report 缺失；lineage graph 断边；删除请求无法定位影响模型；schema 失败率升高；重复率异常；PII 或许可证风险未阻断；语言 / 来源分布漂移无解释；train / eval overlap 非零；标注一致性过低；合成数据比例过高；告警无 owner；TrainingJob 只记录路径；ModelVersion 无数据版本；没有最终数据版本质量门禁。

## Model Artifact Registry Audit

一句话定义：Model Artifact Registry Audit 是把 ModelVersion、权重 manifest、safetensors、tokenizer、config、adapter、merge、量化版本、runtime、评估、安全权限、发布状态、回滚、lineage、缓存和生命周期放进同一张模型仓库审计表的方法。

核心直觉：模型仓库不是对象存储目录，也不是“能下载权重就能上线”。大模型线上行为由权重、tokenizer、chat template、model config、adapter、merge 过程、量化配置、runtime 支持和评估报告共同决定；任何一个产物错配，都可能导致加载失败、输出格式异常、质量退化、越权下载或无法回滚。

常见审计维度：Model Version Contract 看 model id、version、architecture、owner、stage、训练任务、数据版本、评估报告、安全报告、发布 alias 和审批是否完整；Weight Manifest Integrity 与 Safe Weight Format 看权重分片、checksum、manifest、safetensors、安全扫描和 pickle 风险；Tokenizer Config Binding 看 tokenizer hash、config hash、chat template、special tokens 和 stop tokens 是否与 ModelVersion 绑定；Adapter Base Compatibility 与 Merge Lineage Completeness 看 adapter base model、rank、target modules、merge 工具、merge 后 eval 和 lineage；Quantization Eval Gate 与 Runtime Compatibility Fit 看量化方法、校准数据、质量退化、目标 runtime、上下文长度、tensor parallel 和加载 smoke test；Eval Report Linkage、Safety Permission Gate、Release Status Governance、Rollback Alias Readiness、Artifact Lineage Completeness、Load Cache Readiness、Lifecycle Retention Policy 和 Model Registry Gate 则检查发布证据、安全权限、回滚、血缘、冷启动、保留策略和最终门禁。

常见公式：模型版本可以写成

```math
a_i=(m_i,v_i,w_i,t_i,c_i,b_i,p_i,q_i,r_i,e_i,s_i,l_i,o_i,z_i)
```

元数据完整率可以写成

```math
C_{\mathrm{artifact}}=\frac{K_{\mathrm{filled}}}{K_{\mathrm{required}}}
```

权重 checksum 覆盖率可以写成

```math
C_{\mathrm{weight}}=\frac{N_{\mathrm{checked}}}{N_{\mathrm{weight,total}}}
```

tokenizer 和 config 绑定门禁可以写成

```math
B_{\mathrm{tok}}=\mathbf{1}[h_{\mathrm{tok}}=h_{\mathrm{tok,expected}} \land h_{\mathrm{cfg}}=h_{\mathrm{cfg,expected}}]
```

量化质量退化可以写成

```math
\Delta Q=Q_{\mathrm{base}}-Q_{\mathrm{quant}}
```

模型仓库门禁可以写成

```math
G_{\mathrm{registry}}=\mathbf{1}[\min_j C_j\ge\tau_j \land C_{\mathrm{weight}}=1 \land B_{\mathrm{tok}}=1 \land \Delta Q\le\rho_Q \land T_{\mathrm{load}}\le\tau_{\mathrm{load}} \land P_0=0]
```

面试表达：这类题先区分 Model、ModelVersion 和 Artifact，再讲权重 manifest / checksum / safetensors，然后讲 tokenizer、config、adapter、merge 和量化版本必须绑定到同一个模型契约；接着讲 runtime 兼容、评估报告、安全权限、发布状态、灰度和回滚；最后用 Model Registry Gate 收束，说明模型仓库连接训练平台、评估平台和推理平台，是模型生命周期控制面。

## Model Version Contract

一句话定义：Model Version Contract 是 model id、version、architecture、参数规模、owner、stage、训练任务、数据版本、权重 manifest、tokenizer、config、eval report、安全报告、发布 alias 和审批组成的模型版本契约。

## Weight Manifest Integrity

一句话定义：Weight Manifest Integrity 是检查权重文件列表、分片大小、manifest checksum、每个 shard checksum、重复路径和总大小是否一致。

## Safe Weight Format

一句话定义：Safe Weight Format 是要求生产权重使用 safetensors 或经过安全扫描的格式，并避免不可信 pickle 反序列化风险。

## Tokenizer Config Binding

一句话定义：Tokenizer Config Binding 是把 tokenizer hash、model config hash、chat template、special tokens、stop tokens 和上下文长度绑定到具体 ModelVersion。

## Adapter Base Compatibility

一句话定义：Adapter Base Compatibility 是检查 LoRA / QLoRA 等 adapter 的 base model version、rank、target modules、dtype、tokenizer 和训练配置是否与预期基座兼容。

## Merge Lineage Completeness

一句话定义：Merge Lineage Completeness 是记录哪个 base model 加哪个 adapter、用哪个工具和配置生成哪个 merged model，并要求 merge 后重新评估。

## Quantization Eval Gate

一句话定义：Quantization Eval Gate 是检查量化方法、bits、校准数据、工具版本、目标 runtime、质量退化、延迟吞吐和 eval report 是否满足上线阈值。

## Runtime Compatibility Fit

一句话定义：Runtime Compatibility Fit 是检查目标 runtime 是否支持当前权重格式、tokenizer、上下文长度、量化格式、tensor parallel、CUDA / driver 和加载 smoke test。

## Eval Report Linkage

一句话定义：Eval Report Linkage 是让 ModelVersion 关联通用能力、领域能力、安全、工具调用、结构化输出、长上下文、延迟吞吐和成本评估报告。

## Safety Permission Gate

一句话定义：Safety Permission Gate 是检查模型安全评估、许可证、敏感等级、授权项目、下载角色、导出限制和审计日志是否满足发布策略。

## Release Status Governance

一句话定义：Release Status Governance 是用 registered、validated、evaluated、staged、production、deprecated 和 archived 等状态机治理模型发布和下线。

## Rollback Alias Readiness

一句话定义：Rollback Alias Readiness 是保证生产 alias、回滚 alias、回滚版本、回滚加载测试和路由策略已经预先准备好。

## Artifact Lineage Completeness

一句话定义：Artifact Lineage Completeness 是让 dataset version、training job、checkpoint、model version、adapter、merged model、quantized model、eval report 和 deployment 组成可追踪血缘图。

## Load Cache Readiness

一句话定义：Load Cache Readiness 是检查模型产物是否有本地缓存、预热、checksum 校验、加载耗时 SLO 和冷启动回退策略。

## Lifecycle Retention Policy

一句话定义：Lifecycle Retention Policy 是定义 production、candidate、deprecated、archived 和实验 checkpoint 的保留、冷存储、删除依赖检查和审计要求。

## Model Registry Gate

一句话定义：Model Registry Gate 是判断模型仓库是否达到版本契约完整、权重安全可校验、tokenizer / config 绑定、adapter / merge / quantization 可追踪、runtime 可加载、评估和安全过线、发布可回滚、生命周期可治理的综合门禁。

典型硬阻断：ModelVersion 缺 owner、训练任务、数据版本或 eval report；权重分片没有 checksum；使用不可信 pickle 权重；tokenizer 或 config hash 与记录不一致；adapter 绑定错 base model；merge 后没有 lineage 和 eval；量化质量退化超阈值；目标 runtime 不支持该格式或量化；安全评估、许可证或权限未过线；production 发布缺审批；没有 rollback alias；lineage graph 断边；冷启动加载超过 SLO；生产模型可被误删；没有最终模型仓库门禁。

## Artifact Management Audit

一句话定义：Artifact Management Audit 是把 dataset、checkpoint、eval report、deployment package、release manifest、日志、配置和模型产物统一纳入元数据、checksum、权限、血缘、promotion、回滚、删除依赖和生命周期门禁的方法。

核心直觉：Artifact 不是普通文件路径，而是 AI 研发和生产流程中的可复现证据。训练任务读了什么数据、产出了哪个 checkpoint，评估报告绑定了哪个模型和 eval 数据，部署包用了哪个 runtime image 和安全策略，回滚依赖哪些旧产物，这些都应该能从 artifact 系统里查到、校验和审计。

常见审计维度：Artifact Metadata Contract 看 artifact id、type、name、version、URI、checksum、size、producer job、依赖、权限、lifecycle 和 tags 是否完整；Dataset Artifact Manifest 看 dataset version、split、schema、文件列表、样本数、token 数、来源组成、清洗 / 去重规则、质量指标和权限；Checkpoint Artifact Recoverability 看 model、optimizer、scheduler、dataloader、RNG、distributed metadata、shard checksum 和 restore test；Eval Report Reproducibility 看模型、eval 数据、评估代码、metric definition、prompt、inference config、runtime、样本输出和 failure cases；Deployment Package Completeness 与 Release Manifest Integrity 看权重、tokenizer、config、runtime image digest、安全策略、路由策略、health check、eval report 和 rollback target；Checksum Integrity Gate、Artifact Lineage Completeness、Artifact Store Metadata Split、Permission Access Control、Lifecycle Retention Policy、Promotion Gate Readiness、Experiment Tracking Linkage、Rollback Artifact Readiness、Deletion Dependency Safety 和 Artifact Management Gate 则检查完整性、血缘、存储分层、权限、保留、晋级、实验追踪、回滚、删除和最终门禁。

常见公式：artifact 记录可以写成

```math
A_i=(u_i,\tau_i,n_i,v_i,r_i,h_i,s_i,o_i,d_i,p_i,l_i,g_i,e_i,z_i)
```

元数据完整率可以写成

```math
C_{\mathrm{meta}}=\frac{K_{\mathrm{filled}}}{K_{\mathrm{required}}}
```

checksum 覆盖率可以写成

```math
C_{\mathrm{checksum}}=\frac{N_{\mathrm{checked}}}{N_{\mathrm{files,total}}}
```

artifact 血缘图可以写成

```math
G_{\mathrm{art}}=(V_{\mathrm{art}},E_{\mathrm{dep}})
```

deployment package 完整率可以写成

```math
C_{\mathrm{release}}=\frac{N_{\mathrm{artifact,present}}}{N_{\mathrm{artifact,required}}}
```

删除安全门禁可以写成

```math
G_{\mathrm{delete}}=\mathbf{1}[N_{\mathrm{dependents}}=0 \land T_{\mathrm{age}}\ge T_{\mathrm{retention}} \land H_{\mathrm{legal}}=0 \land A_{\mathrm{audit}}=1]
```

最终 artifact 门禁可以写成

```math
G_{\mathrm{artifact}}=\mathbf{1}[\min_j C_j\ge\tau_j \land C_{\mathrm{checksum}}=1 \land C_{\mathrm{lineage}}\ge\rho \land C_{\mathrm{release}}=1 \land G_{\mathrm{delete}}=1 \land P_0=0]
```

面试表达：这类题先说明 artifact 管理比模型仓库更通用，再按 dataset、checkpoint、eval report、deployment package 和 release manifest 分类型讲元数据和门禁；然后讲 artifact store 存大文件、metadata store 存索引，checksum service 做完整性，lineage graph 做追溯，access control 做权限，lifecycle manager 做保留和删除依赖检查；最后用 Artifact Management Gate 收束，证明训练、评估、部署、回滚、审计和成本治理都能靠 artifact 系统复现。

## Artifact Metadata Contract

一句话定义：Artifact Metadata Contract 是 artifact id、type、name、version、URI、checksum、size、created time、producer job、dependencies、permissions、lifecycle state 和 tags 组成的产物契约。

## Dataset Artifact Manifest

一句话定义：Dataset Artifact Manifest 是把 dataset version、split、schema、file list、sample count、token count、source composition、rule version、quality metrics、permissions 和 checksum 冻结成训练输入证据。

## Checkpoint Artifact Recoverability

## Eval Report Reproducibility

一句话定义：Eval Report Reproducibility 是检查评估报告是否绑定 model / checkpoint、eval dataset、eval code、metric definitions、prompt template、inference config、runtime、sample outputs、failure cases 和 evaluator version。

## Deployment Package Completeness

一句话定义：Deployment Package Completeness 是检查部署包是否包含权重、tokenizer、model config、runtime config、image digest、health check、generation config、安全策略、路由策略和 release manifest。

## Release Manifest Integrity

## Checksum Integrity Gate

## Artifact Store Metadata Split

一句话定义：Artifact Store Metadata Split 是大文件进入 artifact store，索引、权限、状态、checksum 和依赖进入 metadata store，避免把大模型权重塞进元数据库。

## Permission Access Control

一句话定义：Permission Access Control 是对 dataset、checkpoint、eval sample outputs、deployment package、删除和跨租户使用做 RBAC / ABAC、审批和审计。

## Promotion Gate Readiness

一句话定义：Promotion Gate Readiness 是把 checkpoint -> candidate model -> evaluated model -> release candidate -> production package 的每次晋级绑定完整性、评估、安全、runtime 和审批门禁。

## Experiment Tracking Linkage

一句话定义：Experiment Tracking Linkage 是让 experiment run 的参数、指标、日志和输入 / 输出 artifact 能互相跳转和复现。

## Rollback Artifact Readiness

一句话定义：Rollback Artifact Readiness 是保证旧权重、旧 tokenizer、旧 runtime image、旧配置、旧安全策略、旧 route policy 和旧 release manifest 都仍可加载。

## Deletion Dependency Safety

一句话定义：Deletion Dependency Safety 是删除 artifact 前检查下游依赖、retention、legal hold、deployment、rollback、审计和删除记录，避免破坏复现与回滚。

## Artifact Management Gate

一句话定义：Artifact Management Gate 是判断 artifact 系统是否达到元数据完整、manifest 可校验、checkpoint 可恢复、eval 可复现、部署包完整、release manifest 不可变、权限可控、血缘可追踪、promotion / rollback / deletion 安全和生命周期可治理的综合门禁。

典型硬阻断：artifact 只有路径没有元数据；dataset manifest 样本数或 token 数不一致；checkpoint 不能恢复；eval report 缺 eval 数据或 prompt 版本；deployment package 缺 tokenizer、runtime image 或安全策略；release manifest 可变；checksum 未验证；artifact lineage 断边；大文件混进 metadata DB；eval sample outputs 权限开放；retention policy 缺失；promotion 绕过 eval / safety；experiment run 和 artifact 脱节；rollback artifact 被删；删除前未检查依赖；没有最终 artifact 管理门禁。

## Experiment Tracking Audit

一句话定义：Experiment Tracking Audit 是把一次训练、评估、prompt 实验或压测 run 的参数、指标、日志、样本、代码、数据、模型、环境、artifact、成本、状态、搜索索引、血缘和结论放进同一套可复现门禁的方法。

核心直觉：Experiment tracking 不是曲线截图，也不是只记录最终分数。它要回答三类问题：第一，结果从哪里来，能否比较；第二，输入、代码、数据、环境和 artifact 是否能复现；第三，实验结论、成本、状态和后续动作是否能被团队协作使用。训练平台、评估平台、artifact 系统和成本系统都应该围绕同一个 run id 打通。

常见审计维度：Run Metadata Contract 看 run id、experiment id、owner、status、created / ended time、tags 和 notes 是否完整；Parameter Config Capture 看 learning rate、global batch size、optimizer、seed、sequence length、precision、parallel config 和 ZeRO stage 是否记录；Metric Curve Step Capture 看 train / eval 指标是否带 step 或 timestamp；Log Run Linkage 看 stdout、framework log、dataloader log、distributed log、checkpoint log 和 eval log 是否绑定 run id；Sample Result Diagnosis 看 input hash、prediction、reference、score、error type、prompt version 和 evaluator version；Code Version Reproducibility、Data Version Reproducibility、Model Checkpoint Linkage 与 Environment Capture 检查 commit、dirty flag、image digest、dataset manifest、base model、checkpoint、Python / framework / CUDA / driver / NCCL / GPU；Prompt Eval Config Capture、Artifact Output Linkage、Cost Attribution、Run State Transition、Search Index Readiness、Lineage Graph Readiness、Decision Note Capture 和 Experiment Tracking Gate 则检查 prompt / eval 口径、输出产物、成本归因、状态机、可搜索性、血缘、实验结论和最终门禁。

常见公式：run 记录可以写成

```math
R_i=(e_i,u_i,a_i,t_i,p_i,m_i,l_i,s_i,c_i,d_i,q_i,o_i,z_i)
```

参数覆盖率可以写成

```math
C_{\mathrm{param}}=\frac{1}{N_{\mathrm{param}}}\sum_{j=1}^{N_{\mathrm{param}}}\mathbf{1}[p_j\ \mathrm{recorded}]
```

指标曲线可以写成

```math
M_k=\{(r_i,k,t,v)\mid r_i\in \mathcal{R},\ t\in \mathcal{T}\}
```

样本诊断覆盖率可以写成

```math
C_{\mathrm{sample}}=\frac{N_{\mathrm{samplediag}}}{N_{\mathrm{sample}}}
```

代码可复现门禁可以写成

```math
G_{\mathrm{code}}=\mathbf{1}[C_{\mathrm{commit}}=1 \land C_{\mathrm{dirty}}=0 \land C_{\mathrm{image}}=1 \land C_{\mathrm{entry}}=1]
```

数据可复现门禁可以写成

```math
G_{\mathrm{data}}=\mathbf{1}[C_{\mathrm{dataset}}=1 \land C_{\mathrm{manifest}}=1 \land C_{\mathrm{recipe}}=1 \land C_{\mathrm{seed}}=1]
```

最终实验追踪门禁可以写成

```math
G_{\mathrm{track}}=\mathbf{1}[\min_j C_j\ge\tau_j \land G_{\mathrm{code}}=1 \land G_{\mathrm{data}}=1 \land P_0=0]
```

面试表达：这类题先说明 experiment / run 的对象模型，再分层讲 parameters、metrics、logs、sample results、code / data / environment、model / checkpoint / artifact、cost、state、tags / notes 和 lineage；随后讲存储分层，metadata store 存 run 索引，time series store 存指标，log store 存日志，artifact store 存产物，sample store 存诊断样本；最后用 Experiment Tracking Gate 收束，证明实验结果可比较、可复现、可解释、可审计。

## Run Metadata Contract

一句话定义：Run Metadata Contract 是 run id、experiment id、owner、status、created time、ended time、tags、notes、hypothesis、decision 和 next action 组成的实验记录契约。

## Parameter Config Capture

一句话定义：Log Run Linkage 是把训练日志、框架日志、分布式日志、checkpoint 日志、eval 日志和错误栈统一绑定到 run id。

## Code Version Reproducibility

一句话定义：Code Version Reproducibility 是记录 Git commit、branch、dirty flag、diff、container image digest、entrypoint 和依赖版本，证明实验代码可复现。

## Data Version Reproducibility

一句话定义：Data Version Reproducibility 是记录 dataset id / version、manifest checksum、split、sampling recipe、shuffle seed 和 tokenizer version，避免路径漂移。

## Model Checkpoint Linkage

## Environment Capture

## Prompt Eval Config Capture

一句话定义：Prompt Eval Config Capture 是把 prompt template、system prompt、tool schema、generation config、eval dataset、judge model、metric definition 和 evaluator version 绑定到 run。

## Artifact Output Linkage

一句话定义：Artifact Output Linkage 是让 checkpoint、final model、eval report、log artifact 和 sample output artifact 都能从 run id 跳转并校验 checksum。

## Cost Attribution

一句话定义：Cost Attribution 是把 GPU hours、GPU type、storage、network、training tokens、inference tokens、eval calls 和 total cost 归因到 experiment / run / project。

## Run State Transition

一句话定义：Run State Transition 是用 created、running、succeeded、failed、killed、evaluated、release candidate、rejected 和 archived 等状态管理实验生命周期。

## Search Index Readiness

一句话定义：Search Index Readiness 是保证 owner、experiment、model、dataset、code commit、metric threshold、tag、status、created time 和 artifact id 可检索。

## Lineage Graph Readiness

一句话定义：Lineage Graph Readiness 是让 dataset、code、config、run、checkpoint、eval report、model version 和 deployment package 能在图上双向追踪。

## Decision Note Capture

一句话定义：Decision Note Capture 是记录实验 hypothesis、观察、结论、next action 和 rejection reason，让实验结论能被团队复用。

## Experiment Tracking Gate

一句话定义：Experiment Tracking Gate 是判断实验追踪系统是否达到元数据完整、参数指标日志样本齐全、代码数据环境可复现、artifact 和 lineage 可追踪、成本状态结论可审计且无 P0 风险的综合门禁。

典型硬阻断：run 缺 owner 或状态；参数缺 seed 或并行配置；metric 没有 step；日志和 run id 脱节；只存聚合分数不存样本诊断；代码 dirty 且没有 diff；dataset 只记录可变路径；checkpoint 和 parent run 断开；环境版本不全；eval prompt、judge model 或 metric definition 缺失；artifact 没有 checksum；成本不可归因；状态从 debug 直接跳到 release candidate；搜索索引缺关键字段；lineage graph 断边；没有实验结论和下一步动作。

## Evaluation Platform Audit

一句话定义：Evaluation Platform Audit 是把模型版本、评估数据、prompt、推理配置、指标、人评、LLM judge、pairwise、在线实验、报告、缓存、可观测性、artifact 和发布门禁放进同一套质量控制审计表的方法。

核心直觉：评估平台不是 benchmark 脚本，而是模型质量控制系统。它要回答候选模型是否真的更好、哪些切片退化、judge 和人评是否可信、线上放量是否安全、报告能否复现、发布和回滚证据是否完整。训练平台、实验追踪、模型仓库、artifact 管理、推理发布和安全治理都应该围绕同一份评估证据链协作。

常见审计维度：Eval Dataset Contract 看数据集 id、版本、任务、样本数、切片、标签、来源、构造方式、训练隔离和污染检查是否完整；Offline Batch Reproducibility 看模型、数据、prompt、推理配置、评估代码、runtime 和指标定义是否固定；Metric Definition Versioning 看指标名称、方向、权重、阈值和版本；LLM Judge Calibration、Human Review Quality Control 与 Pairwise Blinding Randomization 检查 judge 校准、人评 gold set / 复审 / 仲裁、盲评随机顺序和 tie 记录；Online Eval Guardrail 与 Release Gate Readiness 检查 A/B 分桶、样本比例、质量、延迟、成本、安全事件、回滚和审批；Eval Report Completeness、Slice Regression Detection、Regression Suite Coverage、Eval Job Scheduler Reliability、Eval Cache Key Integrity、Eval Observability、Artifact Tracking Linkage 和 Evaluation Platform Gate 则检查报告、切片退化、回归集、调度、缓存、监控、血缘和最终门禁。

常见公式：评估任务可以写成

```math
J_i=(m_i,d_i,p_i,c_i,q_i,g_i,h_i,o_i,r_i,z_i)
```

数据集切片覆盖率可以写成

```math
C_{\mathrm{dataset}}=\frac{1}{N_{\mathrm{slice}}}\sum_{s=1}^{N_{\mathrm{slice}}}\mathbf{1}[n_s\ge n_{\min}]
```

训练 / 评估污染率可以写成

```math
R_{\mathrm{leak}}=\frac{N_{\mathrm{overlap}}}{N_{\mathrm{eval}}}
```

指标加权汇总可以写成

```math
S_{\mathrm{agg}}=\frac{\sum_{k=1}^{K}w_k \tilde{M}_k}{\sum_{k=1}^{K}w_k}
```

pairwise win rate 可以写成

```math
W=\frac{N_{\mathrm{win}}+0.5N_{\mathrm{tie}}}{N_{\mathrm{win}}+N_{\mathrm{loss}}+N_{\mathrm{tie}}}
```

在线护栏门禁可以写成

```math
G_{\mathrm{online}}=\mathbf{1}[\Delta_Q\ge -\epsilon_Q \land \Delta_L\le \epsilon_L \land \Delta_K\le \epsilon_K \land P_0=0]
```

最终评估平台门禁可以写成

```math
G_{\mathrm{eval}}=\mathbf{1}[\min_j C_j\ge\tau_j \land G_{\mathrm{offline}}=1 \land G_{\mathrm{human}}=1 \land G_{\mathrm{online}}=1 \land G_{\mathrm{release}}=1 \land P_0=0]
```

面试表达：这类题先讲对象模型，说明一次 eval 必须绑定 ModelVersion、EvalDataset、PromptVersion、InferenceConfig、MetricDefinition、JudgeConfig、HumanReviewTask、PredictionArtifact、EvalReport 和 ReleaseDecision；然后按离线评测、人评、judge、pairwise、在线评估、切片回归、缓存调度和可观测性展开；最后用 Evaluation Platform Gate 收束，证明评估结果可复现、可比较、可解释、可发布、可回滚。

## Eval Dataset Contract

一句话定义：Eval Dataset Contract 是评估数据集的 id、版本、任务类型、样本数、切片分布、标签、来源、构造方式、训练隔离、污染检查和权限组成的数据契约。

## Offline Batch Reproducibility

一句话定义：Offline Batch Reproducibility 是保证离线评测固定模型版本、数据版本、prompt 版本、推理配置、评估代码、runtime 和指标定义，使不同 run 的分数可比较。

## Metric Definition Versioning

一句话定义：Metric Definition Versioning 是记录每个指标的名称、版本、方向、权重、阈值、计算代码和适用任务，避免同名指标口径漂移。

## LLM Judge Calibration

一句话定义：LLM Judge Calibration 是用人工标注、gold set、bias check 和版本化 rubric 校准 judge model，避免把 judge 偏差当成真实模型质量。

## Human Review Quality Control

一句话定义：Human Review Quality Control 是用标注指南、培训、gold set、多标注员重复标注、复审、仲裁和耗时监控保证人评结论稳定。

## Pairwise Blinding Randomization

一句话定义：Pairwise Blinding Randomization 是在 A/B 输出偏好评测中隐藏模型身份、随机展示顺序、记录 tie 并按切片统计 win rate。

## Online Eval Guardrail

一句话定义：Online Eval Guardrail 是在线 A/B 或灰度中检查稳定分桶、样本比例、质量、延迟、成本、安全事件和回滚准备的放量护栏。

## Release Gate Readiness

一句话定义：Release Gate Readiness 是检查离线、人评、在线、安全、性能、成本、审批和回滚是否都通过，避免只凭一个总分上线。

## Eval Report Completeness

一句话定义：Eval Report Completeness 是检查评估报告是否包含模型版本、数据版本、代码版本、prompt、推理配置、指标定义、总分、切片、样本、失败案例、baseline 对比和决策。

## Slice Regression Detection

一句话定义：Slice Regression Detection 是按语言、领域、难度、输入长度、安全类别、工具类型和多轮轮数等切片比较候选模型和 baseline，防止平均分掩盖局部退化。

## Regression Suite Coverage

一句话定义：Regression Suite Coverage 是检查回归集是否覆盖历史事故、关键业务、安全边界、工具调用、格式约束和高价值客户场景，并持续加入新失败样本。

## Eval Job Scheduler Reliability

一句话定义：Eval Job Scheduler Reliability 是评估调度是否支持批量推理、多模型对比、judge、人评、安全扫描、依赖、重试、优先级、预算和失败分类。

## Eval Cache Key Integrity

一句话定义：Eval Cache Key Integrity 是保证评估缓存 key 包含模型、数据、prompt、推理配置、评估代码和 runtime 等会影响输出与指标的版本字段。

## Eval Observability

一句话定义：Eval Observability 是评估任务具备状态、吞吐、推理错误率、judge 错误率、成本、样本失败原因、队列等待、结果写入和 trace id 等观测信号。

## Artifact Tracking Linkage

一句话定义：Artifact Tracking Linkage 是把 prediction outputs、sample table、judge run、人评结果、eval report、release decision 和 lineage edges 注册为可追踪 artifact。

## Evaluation Platform Gate

一句话定义：Evaluation Platform Gate 是判断评估平台是否达到数据、指标、judge、人评、在线、报告、回归、调度、缓存、观测、artifact、发布和回滚证据完整且无 P0 风险的综合门禁。

典型硬阻断：eval dataset 缺版本、切片或污染检查；离线评测未固定 prompt / 推理配置 / 评估代码；指标无版本或方向；LLM judge 未做人类校准；人评缺 gold set、复审或仲裁；pairwise 未盲评或未随机顺序；在线 A/B 分桶不稳定或样本比例异常；发布门禁不含安全、成本、延迟或回滚；报告缺样本级结果；切片退化被总分掩盖；回归集没有历史事故样本；评估调度无重试和失败分类；缓存 key 缺 prompt 或 runtime；评估任务不可观测；eval report 和 release decision 没有 artifact / lineage；没有最终评估平台门禁。

## Feature Embedding Index Infrastructure Audit

一句话定义：Feature Embedding Index Infrastructure Audit 是把 Feature Store、Embedding Store、Vector Index、metadata filter、retrieval trace、lineage、multi-tenant、cost 和 RAG / Agent 集成放进同一套数据访问层审计表的方法。

核心直觉：Feature Store 解决结构化特征的定义、版本、离线训练和在线服务一致性；Embedding Store 解决向量资产的模型版本、chunk、权限和索引状态；Vector Index 解决大规模相似度检索的召回、延迟和成本。生产系统不能只说“我们有向量库”，而要证明特征没有 training-serving skew、embedding 没有版本混用、索引切换可回滚、权限过滤不会越权、检索结果可追踪到 source 和 chunk。

常见审计维度：Feature Definition Contract 看特征名称、版本、实体键、来源、transform、窗口、TTL、owner 和权限；Offline Online Consistency 和 Point in Time Correctness 检查 offline / online 特征值、默认值、类型、transform 版本和历史回放是否正确；Embedding Version Contract 和 Chunk Embedding Lineage 检查 embedding model、维度、pooling、normalize、预处理、chunking、offset、checksum、tenant 和 ACL；Vector Index Build Readiness、ANN Quality Latency Gate 与 Metadata Permission Filter 检查索引版本、维度、metric、HNSW / IVF / PQ 参数、recall@k、P95/P99、空结果率、tenant / ACL / 数据等级过滤；Shadow Index Switch Readiness、Retrieval Trace Completeness、Feature Embedding Lineage Graph、Multi Tenant Isolation、Quality Monitoring Metrics、Cost Capacity Governance、RAG Agent Integration 和 Feature Embedding Index Gate 则检查双索引灰度回滚、检索 trace、血缘图、租户隔离、质量监控、成本预算、RAG / Agent 引用和最终门禁。

常见公式：数据访问链可以写成

```math
A_i=(f_i,e_i,c_i,v_i,s_i,r_i,p_i,l_i,z_i)
```

training-serving skew 可以写成

```math
R_{\mathrm{skew}}=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}\left[|x^{\mathrm{off}}_i-x^{\mathrm{on}}_i|>\epsilon_i\right]
```

point-in-time 泄漏率可以写成

```math
R_{\mathrm{pit\_leak}}=\frac{1}{N_{\mathrm{join}}}\sum_{i,j}\mathbf{1}[t_{\mathrm{feature\_available}}(i,j)>t_{\mathrm{event}}(i)]
```

embedding 版本门禁可以写成

```math
G_{\mathrm{emb}}=\mathbf{1}[C_{\mathrm{model}}=1 \land C_{\mathrm{dim}}=1 \land C_{\mathrm{chunk}}=1 \land C_{\mathrm{norm}}=1]
```

ANN recall@k 可以写成

```math
R@k=\frac{|S_{\mathrm{ann}}(k)\cap S_{\mathrm{exact}}(k)|}{|S_{\mathrm{exact}}(k)|}
```

权限泄露率可以写成

```math
R_{\mathrm{leak}}=\frac{N_{\mathrm{unauth\_returned}}}{N_{\mathrm{returned}}}
```

双索引切换门禁可以写成

```math
G_{\mathrm{switch}}=\mathbf{1}[C_{\mathrm{shadow}}=1 \land R@k\ge\tau_R \land L_{\mathrm{p95}}\le\tau_L \land C_{\mathrm{filter}}=1 \land C_{\mathrm{rollback}}=1]
```

最终门禁可以写成

```math
G_{\mathrm{feidx}}=\mathbf{1}[\min_j C_j\ge\tau_j \land R_{\mathrm{skew}}\le\rho \land R_{\mathrm{pit\_leak}}=0 \land R_{\mathrm{leak}}=0 \land G_{\mathrm{switch}}=1 \land P_0=0]
```

面试表达：这类题先区分 Feature Store、Embedding Store 和 Vector Index 的职责；然后讲特征定义、offline / online、point-in-time、embedding model / chunking / normalize 版本、ANN 索引和 metadata filter；再讲 shadow index、retrieval trace、lineage、多租户、质量监控和成本；最后用 Feature Embedding Index Gate 收束，说明它支撑 RAG、Agent、搜索、推荐和模型路由的数据访问层。

## Feature Definition Contract

一句话定义：Feature Definition Contract 是特征名称、版本、owner、entity key、source、transform version、dtype、window、TTL、默认值和权限组成的结构化契约。

## Offline Online Consistency

一句话定义：Offline Online Consistency 是检查训练读取的 offline feature 和线上服务读取的 online feature 是否使用同一语义定义、transform、默认值、类型和版本。

## Point in Time Correctness

一句话定义：Point in Time Correctness 是训练样本只能读取事件时间之前已经可获得的特征，避免未来信息泄漏。

## Embedding Version Contract

一句话定义：Embedding Version Contract 是 embedding model name / version、dimension、pooling、normalize、preprocess、chunking 和生成时间组成的向量版本契约。

## Chunk Embedding Lineage

一句话定义：Chunk Embedding Lineage 是记录 document、chunk id、offset、chunking version、embedding id、checksum、tenant、ACL 和 source pointer 的向量血缘。

## Vector Index Build Readiness

一句话定义：Vector Index Build Readiness 是检查索引 id、版本、dimension、metric、index type、index params、embedding model version、chunking version、构建状态和输入 manifest 是否完整。

## ANN Quality Latency Gate

一句话定义：ANN Quality Latency Gate 是用 recall@k、precision@k、MRR、NDCG、P95/P99 latency、empty result rate 和 index size 判断近似检索是否满足上线阈值。

## Metadata Permission Filter

一句话定义：Metadata Permission Filter 是在向量检索阶段强制执行 tenant、ACL、数据等级、文档类型、时间范围和安全等级过滤，避免只在 UI 层隐藏无权结果。

## Shadow Index Switch Readiness

一句话定义：Shadow Index Switch Readiness 是新索引先作为 shadow index 构建、验证、canary，再切 active index，并保留旧索引用于 rollback。

## Retrieval Trace Completeness

一句话定义：Retrieval Trace Completeness 是每次检索记录 query hash、embedding model、index version、filters、candidate ids、scores、rerank ids、permission decision、latency 和 source pointers。

## Feature Embedding Lineage Graph

一句话定义：Feature Embedding Lineage Graph 是把 raw event、feature transform、feature version、document、chunk、embedding、index、retrieval result 和 RAG answer 连成可追溯图。

## Multi Tenant Isolation

一句话定义：Multi Tenant Isolation 是特征、embedding、索引、缓存和检索结果按租户、权限和数据等级做逻辑或物理隔离，并保留审计证据。

## Quality Monitoring Metrics

一句话定义：Quality Monitoring Metrics 是持续监控 feature skew、point-in-time leak、embedding stale rate、recall@k、latency、empty result、permission leak 和 RAG citation support。

## Cost Capacity Governance

一句话定义：Cost Capacity Governance 是把 embedding 生成成本、向量内存、索引构建成本、查询成本、存储、增长容量和 owner / budget 统一治理。

## RAG Agent Integration

一句话定义：RAG Agent Integration 是让 retrieval trace、rerank、citation source pointer、Agent memory scope、permission context 和最终回答证据链打通。

## Feature Embedding Index Gate

一句话定义：Feature Embedding Index Gate 是判断特征、embedding、chunk、索引、过滤、trace、血缘、租户、质量、成本、RAG / Agent 集成和回滚证据是否完整且无 P0 风险的综合门禁。

典型硬阻断：特征缺版本或 owner；offline / online transform 不一致；训练样本使用未来特征；embedding model、dimension、normalize 或 chunking 缺版本；chunk 没有 offset、checksum 或 ACL；索引维度和 embedding 不一致；ANN recall 不达标或 P99 超阈值；metadata filter 没有强制 tenant / ACL；shadow index 没有 canary 或 rollback；检索 trace 缺 index version 或 candidate ids；lineage graph 断边；跨租户命中未阻断；质量监控缺 permission leak；构建成本不可归因；RAG citation 无 source pointer；没有最终 Feature Embedding Index Gate。

## AI Infra Observability Audit

一句话定义：AI Infra Observability Audit 是把 metrics、logs、traces、events、SLO、error budget、label cardinality、training / inference / data / RAG / Agent / cost / privacy / alert 信号放进同一套可排障、可告警、可复盘的审计表。

核心直觉：AI Infra 可观测性不是收集更多曲线，而是让训练、推理、数据、模型、RAG、Agent、GPU、存储、网络、调度和成本问题能够被快速发现、定位、解释和复盘。

常见审计维度：Signal Inventory Coverage 看 metrics、logs、traces、events、costs、alerts、dashboards 和 versions 是否都有入口；Metric Contract Completeness 看 metric name、unit、labels、aggregation、owner 和 threshold；SLO Error Budget Readiness 与 Latency Quantile Guard 检查 availability、error budget、burn rate、p95 / p99、TTFT 和 TPOT；Trace Span Coverage、Log Event Structure、Correlation ID Coverage 和 Cardinality Budget Control 检查 gateway / router / queue / prefill / decode span、结构化日志、事件时间线、统一 ID 和 label 基数预算；Training Observability、Inference Observability、Data Quality Observability、RAG Agent Observability、Cost Observability、Privacy Redaction Retention、Alert Actionability 和 Observability Platform Gate 则检查训练、推理、数据、RAG / Agent、成本、隐私、告警、dashboard 和最终门禁。

面试表达：这类题先区分 metrics、logs、traces、events 的职责；然后讲 AI Infra 专属信号：GPU、tokens/s、TTFT、TPOT、KV cache、data wait、eval、RAG trace、tool trace、cost；再讲统一 ID、标签基数、SLO、告警和隐私；最后用 Observability Platform Gate 收束，说明平台不是有 dashboard，而是能支撑事故发现、定位、复盘和治理。

## Signal Inventory Coverage

一句话定义：Signal Inventory Coverage 衡量关键工作负载是否都有 metrics、logs、traces、events、costs、alerts、dashboards 和 versions。

常见问题：只有 GPU metrics，没有模型版本、数据版本、trace 和事件，仍然无法定位训练或推理质量退化。

## Metric Contract Completeness

一句话定义：Metric Contract Completeness 是 metric name、unit、labels、aggregation、owner、threshold、SLO 绑定和保留策略的完整性。

工程价值：没有单位、聚合方式和 owner 的指标，很难用于告警、对比和责任归属。

## SLO Error Budget Readiness

一句话定义：SLO Error Budget Readiness 是可用性 SLI、延迟 SLI、SLO target、窗口、error budget 和 burn rate 是否可计算。

面试表达：AI 服务不能只看当前错误率，还要知道错误预算是否被快速消耗，以及是否需要暂停发布或扩容。

## Latency Quantile Guard

一句话定义：Latency Quantile Guard 是用 p95 / p99、TTFT、TPOT、queue、prefill、decode 和 streaming 分位延迟判断用户体验是否过线。

常见问题：平均延迟稳定，但 p99 或 TTFT 退化，线上用户仍然会明显感知变慢。

## Trace Span Coverage

一句话定义：Trace Span Coverage 是关键链路 span 覆盖 gateway、auth、router、cache、queue、prefill、decode、safety、response，以及训练 submit、schedule、data load、train step、checkpoint 和 eval。

工程价值：没有阶段 span，就无法区分慢在排队、prefill、decode、工具调用还是下游存储。

## Log Event Structure

一句话定义：Log Event Structure 是日志和事件都具备 timestamp、level、service、trace id、event type、entity、version、actor、status 和 redaction 状态。

常见问题：非结构化日志只能人工搜索，无法和 trace、SLO、dashboard 自动关联。

## Correlation ID Coverage

一句话定义：Correlation ID Coverage 衡量 metrics、logs、traces、events、costs、model version、dataset version 和 run id 是否能串联。

工程价值：统一 ID 是跨训练、推理、RAG、Agent、成本和发布事件做根因定位的前提。

## Cardinality Budget Control

一句话定义：Cardinality Budget Control 是控制 metrics label 的取值规模，避免 request id、user id、prompt hash、document id 等高基数字段进入 time series label。

面试表达：高基数字段应该进入 logs 或 traces，不应该成为 metrics label。

## Training Observability

一句话定义：Training Observability 是同时观测 train loss、eval loss、learning rate、gradient norm、step time、tokens/s、GPU、NCCL、data wait、checkpoint、retry 和 node failure。

常见问题：只看 loss 会漏掉 data loader、NCCL、checkpoint 和 GPU 空转瓶颈。

## Inference Observability

一句话定义：Inference Observability 是观测 QPS、TTFT、TPOT、p95 / p99、error、timeout、input / output tokens/s、queue、active sequences、KV cache、GPU、cache hit、fallback、degradation 和 admission reject。

工程价值：推理体验由 queue、prefill、decode 和 streaming 共同决定，不能只看总延迟或 QPS。

## Data Quality Observability

一句话定义：Data Quality Observability 是观测 ingestion、schema、PII、duplicate、corrupt shard、quality gate、data loader wait 和 dataset version 对训练 / eval 的影响。

常见问题：数据质量问题常表现为 loss 异常或模型效果退化，必须能和训练指标关联。

## RAG Agent Observability

一句话定义：RAG Agent Observability 是观测 retrieval trace、prompt assembly trace、tool call trace、execution trace、step count、tool success、loop rate、groundedness、citation correctness 和 cost per run。

工程价值：RAG / Agent 是多步系统，不看 trace 基本无法排查错误来源。

## Cost Observability

一句话定义：Cost Observability 是把 GPU、storage、network、LLM token、embedding、rerank、tool API、trace storage 和 idle GPU 成本按 tenant、model、endpoint、run 和 component 归因。

面试表达：总账单无法治理成本，必须知道哪条链路、哪个模型、哪个租户和哪个 run 消耗异常。

## Privacy Redaction Retention

一句话定义：Privacy Redaction Retention 是 prompt、用户输入、检索片段、tool 参数、模型输出、日志和 trace 的字段级脱敏、TTL、访问控制、加密和审计。

常见问题：完整保存所有 prompt 和输出会提升排障能力，也会扩大隐私和合规风险。

## Alert Actionability

一句话定义：Alert Actionability 衡量告警是否有 owner、runbook、severity、影响面、去重和可能根因，而不是只报告曲线波动。

工程价值：告警噪音会让值班人员忽略真正事故，AI Infra 告警要围绕用户影响和可执行动作设计。

## Observability Platform Gate

一句话定义：Observability Platform Gate 是判断 signals、SLO、trace、日志、事件、成本、隐私、告警、dashboard、incident process 和 postmortem 是否完整且无 P0 风险的综合门禁。

典型硬阻断：只有 GPU metrics；metric 无 owner 或单位；没有 SLO / error budget；只看平均延迟；trace 缺 queue / prefill / decode span；日志未结构化；事件没有 actor / version；ID 无法关联；request id 进入 metrics label；训练缺 data wait / checkpoint 指标；推理缺 TTFT / TPOT；数据质量不可观测；RAG / Agent trace 缺失；成本只看总账单；日志和 trace 未脱敏；告警无 runbook；没有最终 Observability Platform Gate。

## Training Fault Diagnosis Audit

一句话定义：Training Fault Diagnosis Audit 是把 loss 异常、NaN / Inf、训练 hang、OOM、通信慢、I/O 慢、checkpoint、resume、数据异常、配置 diff、最小复现和事故复盘放进同一张可审计排障表的方法。

核心直觉：训练故障定位不能靠最后一行报错和经验猜测。成熟平台要能从 TrainingJob 生命周期事件、所有 rank 日志、step time 分解、loss / gradient 曲线、显存、NCCL、data loader、checkpoint、配置和数据版本 diff 串出证据链，最后给出根因、修复动作和复盘项。

常见审计维度：Training Fault Evidence Coverage 看事件、rank 日志、loss、梯度、显存、通信、数据 I/O、checkpoint、配置 diff、最小复现和诊断是否齐全；Lifecycle Event Timeline 与 Rank Log Completeness 检查最早异常阶段和所有 rank 状态；Loss Anomaly Diagnosis、NaN Inf Guard 和 Update Health Check 检查数值与更新；Hang Straggler Detection、OOM Phase Memory Evidence、Communication Bottleneck Diagnosis 和 IO Data Bottleneck Diagnosis 检查资源、通信与数据瓶颈；Checkpoint Integrity、Training Resume Continuity、Data Fault Isolation、Config Code Diff Readiness、Minimal Reproduction Readiness 和 Training Fault Diagnosis Gate 检查恢复、复现、修复和复盘闭环。

面试表达：这类题先按现象分类：loss 异常、NaN、hang、OOM、通信慢、I/O 慢、resume 异常；再按证据层排查：事件、日志、指标、profile、数据、配置、checkpoint 和最小复现；最后用 Training Fault Diagnosis Gate 收束，说明没有证据链的根因判断不能进入事故结论。

## Training Fault Evidence Coverage

一句话定义：Training Fault Evidence Coverage 衡量训练事故是否同时保存事件、rank 日志、loss、梯度、显存、通信、数据 I/O、checkpoint、配置 diff、复现和诊断结论。

常见问题：只保存 rank 0 stdout 或最后一个 traceback，会漏掉另一个 rank 更早的 OOM、dataloader 阻塞、NCCL 错误或数据异常。

## Lifecycle Event Timeline

一句话定义：Lifecycle Event Timeline 是按时间记录 job submitted、scheduled、started、last progress、checkpoint、node lost、retry、failed 等事件。

工程价值：时间线能判断故障发生在调度、镜像、数据挂载、训练 step、checkpoint、resume 还是平台重试阶段。

## Rank Log Completeness

一句话定义：Rank Log Completeness 是所有 rank 都有 node、local rank、last step、last event、status、heartbeat 和错误上下文。

常见问题：分布式训练最后 timeout 的 rank 未必是根因，缺 rank 日志会把首个失败 rank 隐藏掉。

## Loss Anomaly Diagnosis

一句话定义：Loss Anomaly Diagnosis 是把 loss spike、不下降、突然 NaN 或 train / eval 背离绑定到 step、batch、学习率、precision、梯度、tokenizer、mask 和 resume 状态。

面试表达：loss 异常要先看是否真的在更新，再看异常 batch、LR、grad norm、label / mask、tokenizer、precision 和 checkpoint state，而不是直接怀疑模型结构。

## NaN Inf Guard

一句话定义：NaN Inf Guard 是检查 loss、logits、gradient、loss scale 和自定义算子是否出现非有限值，并在触发时停止、回滚或保存最小复现。

典型硬阻断：出现 NaN 后只跳过 batch；没有保存触发 batch；没有检查 logits / gradient finite；没有记录 precision 和 loss scale。

## Update Health Check

一句话定义：Update Health Check 是用非零梯度比例、optimizer step、learning rate、参数变化和有效 token 数证明训练确实在更新。

常见问题：loss 不下降可能是参数被冻结、LoRA target 错、有效 label 被 mask 掉、optimizer 没 step 或 LR 为 0。

## Hang Straggler Detection

一句话定义：Hang Straggler Detection 是用 step 停止推进、rank heartbeat、collective wait、dataloader 状态、GPU 利用率和首个异常 rank 定位训练卡住原因。

工程价值：hang 往往是一个 rank 先崩溃、阻塞或变慢，其他 rank 在 collective、barrier 或 dataloader 上等待。

## OOM Phase Memory Evidence

一句话定义：OOM Phase Memory Evidence 是记录 OOM 发生在 forward、backward、optimizer、eval 还是 checkpoint，并保存显存峰值、上限、显存拆分和修复建议。

面试表达：forward OOM、backward OOM、optimizer step OOM 和 eval OOM 的修复方向不同，不能只回答“减 batch size”。

## Communication Bottleneck Diagnosis

一句话定义：Communication Bottleneck Diagnosis 是用通信占比、rank step time 分布、NCCL 日志、网络吞吐、错误包和首个错误 rank 判断通信是否拖慢训练。

常见问题：GPU 利用率低不一定是数据慢，也可能是 all-reduce、reduce-scatter、all-gather 或跨节点网络在等待。

## IO Data Bottleneck Diagnosis

一句话定义：IO Data Bottleneck Diagnosis 是用 data wait ratio、cache hit rate、worker queue、存储吞吐、CPU preprocessing 和坏样本隔离判断数据链路是否拖慢训练。

工程价值：I/O 慢常表现为 GPU 利用率周期性下降，需要和通信等待、checkpoint 阻塞、CPU preprocessing 区分。

## Checkpoint Integrity

一句话定义：Checkpoint Integrity 是 checkpoint manifest、checksum、model、optimizer、scheduler、RNG、dataloader state 和 committed 状态是否完整。

常见问题：目录里有文件不等于 checkpoint 可恢复。未 committed、缺 optimizer / scheduler / RNG 或 checksum 不通过，都可能导致 resume 后 loss 跳变。

## Training Resume Continuity

一句话定义：Training Resume Continuity 是检查 resume 后 global step、learning rate、loss、optimizer、scheduler、RNG、dataloader cursor 和配置是否连续。

面试表达：只加载模型权重不是完整 resume。大模型训练恢复要保证训练动力学连续，否则看似恢复成功，实际已经改变实验。

## Data Fault Isolation

一句话定义：Data Fault Isolation 是把空样本、超长样本、非法 JSON、标签错位、tokenizer 异常、分布漂移和权限问题定位到具体 batch、shard 和数据版本。

工程价值：训练故障中数据问题很常见，平台要能保存触发 batch、生成 dataset diff report，并支持隔离或回滚数据版本。

## Config Code Diff Readiness

一句话定义：Config Code Diff Readiness 是把 learning rate、batch、sequence length、precision、并行策略、tokenizer、dataset、checkpoint、代码 commit 和环境版本变化放进同一张 diff 表。

常见问题：没有 final config snapshot、code diff、dataset diff 和 environment diff 时，排查只能靠人工记忆，事故复盘不可复现。

## Minimal Reproduction Readiness

一句话定义：Minimal Reproduction Readiness 是固定 seed、异常 batch、配置、数据版本、单机试验和复现命令，用更小环境复现训练故障。

面试表达：单机小数据能复现，优先怀疑代码、数据、配置或数值；只有多机复现，才更可能是通信、拓扑、调度或环境问题。

## Training Fault Diagnosis Gate

一句话定义：Training Fault Diagnosis Gate 是判断训练故障定位是否有完整证据链、owner、runbook、最小复现、修复动作、postmortem 和 P0 风险阻断的综合门禁。

典型硬阻断：只有最后一行报错；只采 rank 0 日志；loss spike 没有 batch；NaN 后继续训练；hang 没有 rank heartbeat；OOM 不记录阶段和峰值；通信慢没有 NCCL 日志；I/O 慢没有 data wait / cache hit；checkpoint 缺 checksum；resume 不校验 LR / step / dataloader；数据异常没有 batch 和 shard；配置 diff 缺失；没有最小复现；没有最终 Training Fault Diagnosis Gate。

## SLO Oncall Audit

一句话定义：SLO Oncall Audit 是把 SLI、SLO、SLA、错误预算、burn rate、告警、on-call、事故分级、runbook、变更、止损、复盘和成本取舍放进同一张生产可靠性治理表的方法。

核心直觉：AI Infra 可靠性不是追求永不失败，而是定义哪些失败可接受、如何计算预算、什么时候告警、谁来响应、能做哪些止损动作、事后如何把事故沉淀成行动项和上线门禁。没有错误预算和值班闭环，SLO 只是 dashboard 上的百分比。

常见审计维度：SLI Contract Completeness、SLO Target Measurability 和 SLA Boundary Clarity 检查指标、目标和承诺边界；Inference SLO Coverage、Training SLO Coverage、Data Eval SLO Coverage 检查推理、训练、数据和评估平台的用户体验指标；Error Budget Accounting 与 Burn Rate Alerting 检查预算和快慢窗口告警；Alert Actionability、Oncall Ownership Escalation、Incident Severity Routing、Runbook Executability 检查告警和值班可执行性；Change Event Linkage、Mitigation Rollback Authority、Postmortem Action Closure 和 SLO Cost Tradeoff Gate 检查变更、止损、复盘和成本收益边界。

常见公式：错误预算比例是

```math
B=1-S_{\mathrm{slo}}
```

预算消耗比例可以写成

```math
U_{\mathrm{budget}}=\frac{N_{\mathrm{bad}}}{\max(1,N_{\mathrm{bad,allowed}})}
```

窗口 burn rate 可以写成

```math
\beta_w=\frac{R_{\mathrm{bad},w}}{B}
```

综合门禁可以写成

```math
G_{\mathrm{slo\_oncall}}=\prod_{j=1}^{16}G_j
```

面试表达：SLO / SLA / 值班体系题要先区分 SLI 是实际观测指标、SLO 是内部目标、SLA 是对外承诺；再讲推理、训练、数据和评估各自的 SLO；随后用错误预算和 burn rate 驱动发布、告警和冻结策略；最后补 on-call、runbook、权限、变更时间线、事故分级、postmortem action item 和成本取舍。

## SLI Contract Completeness

一句话定义：SLI Contract Completeness 是检查关键服务的指标是否有名称、定义、数据源、窗口、owner 和坏事件口径。

## SLO Target Measurability

一句话定义：SLO Target Measurability 是检查 SLO 是否能用实际查询计算、能回放、能解释用户体验，并能触发明确工程动作。

## SLA Boundary Clarity

一句话定义：SLA Boundary Clarity 是检查对外承诺、客户范围、排除项、赔偿或业务责任、owner 是否清楚。

## Inference SLO Coverage

一句话定义：Inference SLO Coverage 是检查推理服务是否覆盖 availability、TTFT、TPOT、p99、error、timeout 和质量 / 安全 guardrail。

## Training SLO Coverage

一句话定义：Training SLO Coverage 是检查训练平台是否区分用户错误和平台错误，并覆盖提交、调度、平台失败率、checkpoint 保存和恢复。

## Data Eval SLO Coverage

一句话定义：Data Eval SLO Coverage 是检查数据构建、manifest 查询、质量门禁、eval report、RAG / Agent run success 是否有可观测目标。

## Error Budget Accounting

一句话定义：Error Budget Accounting 是按服务、窗口和坏事件规则计算允许失败空间，并把剩余预算绑定到发布和风险策略。

## Burn Rate Alerting

一句话定义：Burn Rate Alerting 是用错误预算消耗速度做告警，通常同时看快窗口和慢窗口，兼顾突发事故和慢性退化。

## Alert Actionability

一句话定义：Alert Actionability 是衡量告警是否有 owner、runbook、下一步动作和低噪音，而不是只看触发次数。

## Oncall Ownership Escalation

一句话定义：Oncall Ownership Escalation 是检查值班排班、主备、ACK SLO、升级链路和状态沟通是否完整。

## Incident Severity Routing

一句话定义：Incident Severity Routing 是按用户影响、持续时间、服务重要性和风险权重决定 SEV 等级、通知范围和复盘要求。

## Runbook Executability

一句话定义：Runbook Executability 是检查 runbook 是否真的能指导值班完成影响判断、根因路径、止损、回滚和升级。

## Change Event Linkage

一句话定义：Change Event Linkage 是事故排查时能关联模型、prompt、runtime、route、data、cluster、cache 和 owner 变更时间线。

## Mitigation Rollback Authority

一句话定义：Mitigation Rollback Authority 是检查值班是否有可审计的限流、降级、扩容、摘除、回滚和暂停队列权限。

## Postmortem Action Closure

一句话定义：Postmortem Action Closure 是检查复盘行动项是否有 owner、截止时间、关闭证据和回归验证，而不是只写事故文档。

## SLO Cost Tradeoff Gate

一句话定义：SLO Cost Tradeoff Gate 是判断更高可靠性目标是否值得额外基础设施成本和值班成本的综合门禁。

典型硬阻断：SLI 只有指标名没有坏事件口径；SLO 不可测量；SLA 承诺和内部目标混淆；推理只看可用性不看 TTFT / TPOT；训练不区分用户错误和平台错误；数据和评估没有质量 / 产出 SLO；错误预算不驱动发布；burn rate 告警缺失；告警无 runbook；值班无升级链路；事故分级只看错误率；runbook 从未演练；事故无法关联最近变更；值班无回滚权限；复盘行动项长期不关闭；SLO 目标不考虑成本。

## AI Infra Cost Governance Audit

一句话定义：AI Infra Cost Governance Audit 是把 GPU、训练、推理、缓存、存储、网络、artifact、日志 trace、预算、配额、成本归因、dashboard 和优化建议放进同一张成本治理审计表的方法。

核心直觉：成本治理不是月底看账单，也不是简单砍资源，而是把资源消耗变成可观测、可归因、可预算、可优化的工程指标。好的成本治理要能回答谁花了钱、花在什么资源上、产生了什么有效产出、哪些成本是浪费、哪些优化不会伤害质量和 SLO。

常见审计维度：Usage Metering Coverage 与 Cost Attribution Labels 检查 usage 和标签；GPU Cost Efficiency、Training Waste Control、Inference Unit Cost 和 Cache Savings Accounting 检查 GPU、训练、推理和缓存单位成本；Storage Lifecycle Governance、Network Egress Governance 和 Artifact Dependency Safety 检查存储、网络和删除安全；Budget Quota Enforcement、Cost Anomaly Alerting、Tenant Model Chargeback、Dashboard Drilldown Readiness、Optimization Recommendation Trace、SLO Quality Cost Tradeoff 和 Cost Governance Gate 检查预算、告警、showback / chargeback、dashboard、优化建议和最终门禁。

常见公式：GPU 任务成本可以写成

```math
K_{\mathrm{gpu}}=G\cdot T\cdot P_{\mathrm{gpu}}\cdot \eta_{\mathrm{bill}}
```

每 1k token 推理成本可以写成

```math
K_{\mathrm{1k}}=\frac{1000K_{\mathrm{infer}}}{N_{\mathrm{in}}+N_{\mathrm{out}}}
```

预算使用率可以写成

```math
U_{\mathrm{budget}}=\frac{K_{\mathrm{actual}}}{K_{\mathrm{budget}}}
```

面试表达：成本治理题要先讲 usage metering 和 attribution，再分 GPU、训练浪费、推理单位成本、缓存节省、存储生命周期、网络出站和 artifact 删除安全；随后讲预算、配额、成本告警和 dashboard 下钻；最后用 SLO / quality / risk trade-off 收束，说明降本不能牺牲用户体验、质量、安全和回滚能力。

## Usage Metering Coverage

一句话定义：Usage Metering Coverage 是检查 GPU、CPU、memory、storage、network、request、token、artifact、logs 和 traces 是否都有准实时用量记录。

## Cost Attribution Labels

一句话定义：Cost Attribution Labels 是检查 tenant、project、team、owner、model、job type、environment 和 cost center 等标签是否在资源创建时强制存在。

## GPU Cost Efficiency

一句话定义：GPU Cost Efficiency 是用 GPU-hours、tokens per GPU hour、MFU / utilization、idle GPU hours 和有效产出判断 GPU 成本是否健康。

## Training Waste Control

一句话定义：Training Waste Control 是定位失败任务、无效实验、抢占丢失、checkpoint / resume 缺失和 failure owner 缺失造成的训练浪费。

## Inference Unit Cost

一句话定义：Inference Unit Cost 是按 request、1k tokens、output token、tenant、model 和 endpoint 计算推理单位成本。

## Cache Savings Accounting

一句话定义：Cache Savings Accounting 是量化 result cache、prompt cache、prefix / KV cache 和 semantic cache 的节省成本、维护成本、权限风险和质量影响。

## Storage Lifecycle Governance

一句话定义：Storage Lifecycle Governance 是对 checkpoint、dataset、model、eval output、logs、traces 和 vector index 设置保留、归档、删除和冷热分层策略。

## Network Egress Governance

一句话定义：Network Egress Governance 是计量和优化跨地域数据、模型权重、checkpoint、日志 trace 和远端索引访问的网络成本。

## Artifact Dependency Safety

一句话定义：Artifact Dependency Safety 是删除或降冷 artifact 前检查部署、回滚、训练、评估、审计和血缘依赖。

## Budget Quota Enforcement

一句话定义：Budget Quota Enforcement 是用预算控制花费、用配额控制资源，并为例外审批和硬限制建立审计记录。

## Cost Anomaly Alerting

一句话定义：Cost Anomaly Alerting 是对日成本、单位成本、idle、failed job、storage growth、egress 和 agent run 异常告警到 owner。

## Tenant Model Chargeback

一句话定义：Tenant Model Chargeback 是把成本按租户、模型、模型版本、endpoint、job、experiment 和 cost center 做 showback / chargeback。

## Dashboard Drilldown Readiness

一句话定义：Dashboard Drilldown Readiness 是 dashboard 能从总成本下钻到 tenant、model、job、artifact、endpoint、request slice 和 recent change。

## Optimization Recommendation Trace

一句话定义：Optimization Recommendation Trace 是每条降本建议都带估算节省、证据、owner、质量 / SLO 风险和回滚条件。

## SLO Quality Cost Tradeoff

一句话定义：SLO Quality Cost Tradeoff 是判断降本动作是否在质量、SLO、安全、风险和用户体验边界内。

## Cost Governance Gate

一句话定义：Cost Governance Gate 是判断成本治理是否具备用量、归因、预算、配额、告警、dashboard、优化 backlog、审批、审计和 P0 风险阻断的综合门禁。

典型硬阻断：usage 缺 token 或 network；成本标签缺 owner / cost center；只看 GPU utilization 不看有效产出；失败训练任务成本无人负责；推理没有 cost per 1k tokens；缓存节省不扣维护成本或忽略权限；checkpoint 永久保留；跨地域 egress 无理由；删除 artifact 不查回滚依赖；预算和配额只提醒不阻断；成本异常无 owner；成本不能按模型版本下钻；dashboard 只有总账单；优化建议没有风险评估；为降本牺牲 SLO 或质量；没有最终 Cost Governance Gate。

## Resource Utilization Optimization Audit

一句话定义：Resource Utilization Optimization Audit 是把 GPU allocation、GPU utilization、有效产出、碎片、混部、抢占、低优先级队列、弹性训练、推理容量、warm pool、配额公平、调度观测和可靠性 headroom 放进同一张资源治理审计表的方法。

核心直觉：资源利用率优化不是把 GPU 打满，而是在 SLO、隔离、公平性、checkpoint、恢复成本和业务价值约束下减少空闲、减少碎片、减少失败并提高有效产出。GPU 被分配不等于 GPU 有效工作，GPU 忙也不等于产出可用。

常见审计维度：Resource Metric Contract 和 Effective Utilization Accounting 先确认指标口径与有效产出；Fragmentation Control、Topology Binpacking Readiness、Colocation SLO Guard、Preemption Checkpoint Safety、Low Priority Reclaim Policy 和 Elastic Training Safety 检查调度动作是否安全；Inference Capacity Utilization、Warm Pool Headroom Policy、Quota Fairness Borrowing、Scheduler Observability、Dashboard Drilldown Readiness、Reliability Headroom Guard、Cost SLO Tradeoff Control 和 Resource Utilization Gate 检查推理容量、预留、公平性、观测、可靠性和最终门禁。

常见公式：GPU 分配率可以写成

```math
U_{\mathrm{alloc}}=\frac{G_{\mathrm{allocated}}}{G_{\mathrm{total}}}
```

有效利用率可以写成

```math
U_{\mathrm{eff}}=\frac{\sum_i h_i v_i s_i}{\sum_i h_i}
```

碎片比例可以写成

```math
R_{\mathrm{frag}}(g)=1-\frac{\sum_{n=1}^{N}\lfloor f_n/g\rfloor g}{\sum_{n=1}^{N}f_n}
```

面试表达：资源利用率优化题要先区分 allocation、utilization 和 effective utilization，再讲碎片、拓扑和 bin packing；然后讲混部要受 SLO guardrail 约束，抢占要绑定 checkpoint 和 requeue，低优先级队列只能吃可回收余量；最后讲弹性训练、推理容量、warm pool、quota borrowing、dashboard 下钻和 headroom，说明为什么不能追求 100% GPU 利用率。

## Resource Metric Contract

一句话定义：Resource Metric Contract 是统一 GPU allocation、GPU utilization、显存、MFU、tokens、idle、pending、queue wait、preemption、failed GPU-hours 和 fragmentation 的指标口径。

## Effective Utilization Accounting

一句话定义：Effective Utilization Accounting 是把失败任务、无效实验、SLO 失败请求和不可发布产物从资源利用率中扣除，只统计有效产出。

## Fragmentation Control

一句话定义：Fragmentation Control 是按节点、GPU 型号、拓扑和任务规格计算碎片，并通过 bin packing、预留、队列重排和可抢占任务减少碎片。

## Topology Binpacking Readiness

一句话定义：Topology Binpacking Readiness 是检查 bin packing 是否同时考虑 GPU、NIC、rack、storage、gang scheduling 和调度 score trace。

## Colocation SLO Guard

一句话定义：Colocation SLO Guard 是训练 / 推理、在线 / 离线、高 / 低优先级混部时，用 p99、step time、驱逐、低优先级上限和隔离策略保护关键任务。

## Preemption Checkpoint Safety

一句话定义：Preemption Checkpoint Safety 是抢占前检查 checkpoint 新鲜度、graceful shutdown、requeue、频率限制、owner 通知和丢失 GPU-minutes。

## Low Priority Reclaim Policy

一句话定义：Low Priority Reclaim Policy 是低优先级队列只使用可回收余量，并在高优先级需求出现时可抢占、可归还、可补偿。

## Elastic Training Safety

一句话定义：Elastic Training Safety 是控制 world size 变化带来的 global batch、learning rate、optimizer state、dataloader、checkpoint 和 rendezvous 风险。

## Inference Capacity Utilization

一句话定义：Inference Capacity Utilization 是在 TTFT、TPOT、p99、KV pressure、continuous batching 和 autoscaling 约束下统计推理有效吞吐。

## Warm Pool Headroom Policy

一句话定义：Warm Pool Headroom Policy 是用模型热度、冷启动损失、常驻副本成本、权重缓存、headroom 和业务优先级决定 warm pool 策略。

## Quota Fairness Borrowing

一句话定义：Quota Fairness Borrowing 是允许租户借用空闲配额，但用 dominant share、公平性、可回收和补偿机制控制挤占风险。

## Scheduler Observability

一句话定义：Scheduler Observability 是让 gang、queue、priority、preemption、quota、topology、elastic 和 event trace 都能被查询、告警和复盘。

## Reliability Headroom Guard

一句话定义：Reliability Headroom Guard 是在提高利用率时保留故障恢复、突发流量、高优先级任务和线上 SLO 所需的安全余量。

## Cost SLO Tradeoff Control

一句话定义：Cost SLO Tradeoff Control 是检查资源优化建议是否有节省估算、质量 / SLO / 风险评估、owner 和回滚条件。

## Resource Utilization Gate

一句话定义：Resource Utilization Gate 是判断资源利用率优化是否具备 owner、资源策略、例外审批、审计证据和 P0 风险阻断的最终门禁。

典型硬阻断：只有 GPU allocation 没有 utilization；只看 GPU utilization 不看有效产出；失败任务和 SLO 失败请求仍被算入收益；碎片无法解释；bin packing 不看拓扑；训练推理混部没有 p99 guardrail；抢占无 checkpoint；低优先级任务不可回收；弹性训练不处理 global batch；推理只看 tokens/s 不看 TTFT / TPOT；warm pool 只有固定副本；quota borrowing 无归还机制；调度事件不可观测；dashboard 不能下钻；headroom 被挤没；优化建议没有质量、SLO 和回滚证据。

## AI Infra Security Governance Audit

一句话定义：AI Infra Security Governance Audit 是把身份、权限、密钥、数据合规、PII、多租户、模型访问、模型输出、RAG、Agent、供应链、model artifact、运行时、日志 trace、审计和安全响应放进同一张安全治理审计表的方法。

核心直觉：AI Infra 安全不是给推理 API 加鉴权，而是让训练、数据、模型、RAG、Agent、artifact、runtime、日志和事故响应都受统一策略、证据和门禁约束。模型权重、数据集、prompt、trace、embedding、工具参数和 checkpoint 都可能是敏感资产。

常见审计维度：Identity Workload Binding、RBAC ABAC Policy Fit 和 Secret Lifecycle Management 检查身份、权限和密钥；Data Compliance Classification、PII Sensitive Data Control、Tenant Isolation Boundary、Model Access Control 和 Model Output Safety Gate 检查数据、租户、模型和输出；RAG Permission Enforcement、Agent Tool Safety Gate、Supply Chain Integrity、Model Artifact Integrity、Runtime Security Boundary、Log Trace Privacy Governance、Audit Incident Response Readiness 和 Security Governance Gate 检查 RAG、Agent、供应链、运行时、日志、审计和响应闭环。

常见公式：身份绑定覆盖率可以写成

```math
C_{\mathrm{id}}=\frac{N_{\mathrm{bound}}}{N_{\mathrm{workload}}}
```

密钥暴露率可以写成

```math
R_{\mathrm{secret}}=\frac{N_{\mathrm{secret,exposed}}}{N_{\mathrm{secret,total}}}
```

RAG 权限阻断率可以写成

```math
B_{\mathrm{rag}}=\frac{N_{\mathrm{unauthorized\_chunk\_blocked}}}{N_{\mathrm{unauthorized\_chunk}}}
```

面试表达：安全治理题要先画资产和身份边界，再讲 RBAC + ABAC、secret manager、数据分级、PII、租户隔离、模型访问、RAG 权限、Agent 工具确认、供应链 provenance、artifact checksum / signature、runtime hardening、日志脱敏、审计留存和响应预案；最后强调这是防御性平台门禁，不是靠人工规范或提示词承诺。

一句话定义：RAG Permission Enforcement 是在检索和上下文组装前强制 metadata / ACL 权限过滤，并保证 citation 和 trace 不泄露无权限来源。

## Supply Chain Integrity

一句话定义：Supply Chain Integrity 是为镜像、依赖、runtime、数据集、模型权重、tokenizer 和工具记录 provenance、checksum、签名、扫描和许可证证据。

## Model Artifact Integrity

## Audit Incident Response Readiness

一句话定义：Audit Incident Response Readiness 是检查关键事件审计、防篡改留存、检测、隔离、吊销、影响定界、修复和复盘是否就绪。

## Security Governance Gate

一句话定义：Security Governance Gate 是判断 AI Infra 是否具备 owner、安全策略、例外审批、审计证据、响应预案和 P0 风险阻断的最终安全门禁。

## Audit Change Governance Audit

一句话定义：Audit Change Governance Audit 是把审计事件、日志留存、Trace 隐私、事故复盘、行动项关闭、变更记录、灰度回滚、冻结策略和治理闭环放进同一张平台审计表的方法。

核心直觉：审计告诉我们发生了什么，日志留存决定证据能保存多久，事故复盘把事故变成系统改进，变更管理降低下一次事故概率。AI Infra 的变更不仅是代码发布，还包括模型版本、prompt、runtime、路由、数据集、权限策略、工具定义、RAG 知识库和 GPU 集群配置。

常见审计维度：Audit Event Schema Completeness、Critical Operation Audit Coverage 和 Audit Log Integrity Retention 检查审计事件是否结构化、关键操作是否覆盖、日志是否可靠写入和防篡改；Log Retention Policy Fit、Log Minimization Redaction 和 Trace Privacy Sampling 检查 TTL、脱敏、字段级权限和采样；Incident Timeline Completeness、Impact Detection Response Metrics、Root Cause Systemic Analysis 和 Postmortem Action Item Closure 检查事故时间线、MTTD / MTTA / MTTR、系统根因和行动项关闭；Change Record Completeness、High Risk Change Approval、Rollout Rollback Readiness、Change Freeze Error Budget Policy、Audit Postmortem Change Linkage 和 Governance Loop Gate 检查变更记录、审批、灰度、回滚、冻结、错误预算和跨对象跳转。

常见公式：审计 schema 完整率可以写成

```math
C_{\mathrm{schema}}=\frac{N_{\mathrm{event\_complete}}}{N_{\mathrm{audit\_event}}}
```

关键操作审计覆盖率可以写成

```math
C_{\mathrm{audit}}=\frac{N_{\mathrm{critical\_audited}}}{N_{\mathrm{critical\_operation}}}
```

```math
R_{\mathrm{change\_fail}}=\frac{N_{\mathrm{change\_incident}}}{N_{\mathrm{change}}}
```

综合门禁可以写成

```math
G_{\mathrm{audit\_change}}=\prod_{j=1}^{16}G_j
```

## Audit Log Integrity Retention

一句话定义：Audit Log Integrity Retention 是检查审计日志是否可靠写入、防篡改、加密、可查询、可导出和长期留存。

## Log Retention Policy Fit

一句话定义：Log Retention Policy Fit 是按日志类型、敏感等级、合规要求、删除请求和成本预算制定 TTL 与访问策略。

## Log Minimization Redaction

一句话定义：Log Minimization Redaction 是在写入前脱敏敏感字段，并避免长期保存原始 prompt、工具参数、检索片段和用户数据。

## Trace Privacy Sampling

一句话定义：Trace Privacy Sampling 是按错误、慢请求、普通成功和高敏请求分层采样、脱敏和留存 RAG / Agent trace。

## Incident Timeline Completeness

一句话定义：Incident Timeline Completeness 是让事故记录覆盖发生、检测、告警、响应、缓解、恢复和根因确认等关键时间点。

## Impact Detection Response Metrics

一句话定义：Impact Detection Response Metrics 是用 MTTD、MTTA、MTTR、影响范围、用户切片和业务影响衡量事故响应能力。

## Root Cause Systemic Analysis

一句话定义：Root Cause Systemic Analysis 是把根因追到系统防线、流程门禁、监控、测试或设计缺陷，而不是停在个人操作。

## Postmortem Action Item Closure

一句话定义：Postmortem Action Item Closure 是检查复盘行动项是否有 owner、deadline、关闭证据、验证结果和逾期跟踪。

## Change Record Completeness

一句话定义：Change Record Completeness 是检查变更记录是否包含 owner、对象、原因、diff、风险、审批、灰度、回滚和观测指标。

## High Risk Change Approval

一句话定义：High Risk Change Approval 是让高风险模型、数据、权限、路由、runtime、工具和集群配置变更经过审批、通知和回滚验证。

## Rollout Rollback Readiness

一句话定义：Rollout Rollback Readiness 是检查变更是否具备 canary、自动观测、暂停、回滚和回滚演练能力。

## Change Freeze Error Budget Policy

一句话定义：Change Freeze Error Budget Policy 是用变更冻结和错误预算约束高风险变更，并保留紧急修复例外路径。

## Audit Postmortem Change Linkage

一句话定义：Audit Postmortem Change Linkage 是让 audit log、change record、incident、postmortem、action item 和 regression test 能互相跳转。

## Governance Loop Gate

一句话定义：Governance Loop Gate 是判断审计、留存、复盘和变更管理是否具备 owner、治理策略、例外审批、dashboard、审计导出和 P0 风险阻断的最终门禁。

典型硬阻断：审计事件缺 actor 或 trace id；关键操作没有 deny 事件；审计日志可被普通管理员删除；日志 TTL 没有按敏感等级区分；prompt、检索片段或工具参数原文长期落盘；Trace 没有采样和字段级权限；事故没有完整时间线；MTTD / MTTA / MTTR 不可计算；复盘停在“人为失误”；行动项没有验证证据；变更记录缺 diff、owner 或 rollback；高风险变更无审批；canary 无自动观测；错误预算耗尽仍继续放量；postmortem 行动项没有进入回归测试。

## Training Platform Security Audit

一句话定义：Training Platform Security Audit 是把训练平台身份、RBAC / ABAC、数据权限、产物等级传播、资源权限、镜像供应链、代码运行时、密钥、网络、日志、模型产物、高风险审批、审计日志和事件响应放进同一张安全门禁表的方法。

核心直觉：训练平台安全不是给入口加登录，而是控制谁能用什么身份运行任务、读什么数据、用什么资源、运行什么镜像和代码、产生什么模型、查看什么日志，并且能事后追踪全过程。训练脚本本质是任意代码执行，安全控制必须进入任务提交、运行时和产物注册全链路。

常见审计维度：Identity Runtime Binding 看用户身份、项目身份、TrainingJob 运行身份和 service account 是否绑定且没有共享高权限身份；RBAC ABAC Policy Fit 看角色和属性策略是否覆盖项目、数据等级、资源池和用途，并满足最小权限；Dataset Permission Purpose 与 Artifact Classification Propagation 看数据使用目的和产物等级传播是否正确；Resource Quota Priority Governance 看高价值 GPU、队列和优先级是否受控；Image Supply Chain Trust 与 Runtime Code Isolation 看可信 registry、digest、签名、扫描、provenance、非 privileged、hostPath、root 和 Pod 安全边界；Secret Scope Rotation、Network Egress Control 和 Log Redaction Access 看短期凭证、最小权限、出网白名单和日志脱敏；Model Artifact Permission、High Risk Approval、Audit Log Completeness、Audit Integrity Retention、Incident Response Readiness 和 Training Platform Security Gate 则检查模型权重、审批、审计、防篡改留存和事故响应是否闭环。

常见公式：训练平台安全样本可以写成

```math
s_i=(u_i,r_i,d_i,a_i,q_i,m_i,c_i,e_i,n_i,l_i,w_i,h_i,o_i,t_i,z_i)
```

越权阻断率可以写成

```math
R_{\mathrm{block}}=\frac{N_{\mathrm{denied\_unauth}}}{N_{\mathrm{unauth}}}
```

过度授权率可以写成

```math
R_{\mathrm{over}}=\frac{N_{\mathrm{granted}}-N_{\mathrm{used}}}{N_{\mathrm{granted}}}
```

数据等级传播可以写成

```math
V_{\mathrm{artifact}}=\mathbf{1}\left[\mathrm{class}(A)\ge \max_k \mathrm{class}(D_k)\right]
```

审计覆盖率可以写成

```math
R_{\mathrm{audit}}=\frac{N_{\mathrm{audited}}}{N_{\mathrm{required}}}
```

训练平台安全门禁可以写成

```math
G_{\mathrm{security}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{block}}=1 \land R_{\mathrm{over}}\le \rho_{\mathrm{over}} \land V_{\mathrm{artifact}}=1 \land R_{\mathrm{secret}}=0 \land R_{\mathrm{audit}}\ge \rho_{\mathrm{audit}} \land P_0=0\right]
```

面试表达：训练平台安全题不要只讲登录和 RBAC。要从身份、任务运行身份、数据权限、用途授权、资源权限、镜像供应链、运行时隔离、密钥、网络、日志、artifact、高风险审批、审计日志和事件响应逐层展开，最后用 Training Platform Security Gate 收束。

## Identity Runtime Binding

一句话定义：Identity Runtime Binding 是把用户身份、项目身份、TrainingJob 运行身份、service account 和 workload identity 绑定起来，避免所有任务共享平台管理员身份。

## RBAC ABAC Policy Fit

一句话定义：RBAC ABAC Policy Fit 是检查角色权限和项目、团队、数据等级、资源池、用途、环境等属性策略是否共同覆盖训练平台访问控制。

## Dataset Permission Purpose

一句话定义：Dataset Permission Purpose 是检查用户和任务身份是否被允许以当前用途读取指定 dataset version，而不是只看文件路径是否可读。

## Artifact Classification Propagation

一句话定义：Artifact Classification Propagation 是让 checkpoint、模型权重、adapter、eval report、日志和样本输出继承训练数据的最高敏感等级。

## Resource Quota Priority Governance

一句话定义：Resource Quota Priority Governance 是用队列、配额、优先级审批和 GPU-hour 预算控制高价值资源使用。

## Image Supply Chain Trust

一句话定义：Image Supply Chain Trust 是检查镜像来源、digest、签名、漏洞扫描、SBOM / provenance 和 secret exclusion，降低供应链风险。

## Runtime Code Isolation

一句话定义：Runtime Code Isolation 是限制训练代码的 privileged、hostPath、root、外联和 Pod 运行时权限，降低任意代码执行带来的逃逸和越权风险。

## Secret Scope Rotation

一句话定义：Secret Scope Rotation 是用 Secret Manager、短期凭证、最小权限、轮换和日志脱敏管理训练任务访问凭证。

## Network Egress Control

一句话定义：Network Egress Control 是用默认拒绝、egress allowlist、生产服务阻断和外联日志控制训练代码向外发送数据。

## Log Redaction Access

一句话定义：Log Redaction Access 是对训练日志查看、下载、保留和脱敏做权限控制，防止日志成为敏感数据泄露通道。

## Model Artifact Permission

一句话定义：Model Artifact Permission 是对 checkpoint、final model、adapter、tokenizer、config、eval report 和 deployment package 做 registry RBAC、下载限制、阶段策略和 lineage 绑定。

## High Risk Approval

一句话定义：High Risk Approval 是对 restricted 数据训练、模型导出、生产发布、高优先级队列、关闭日志、未审核镜像、外部网络和删除关键 checkpoint 等动作做审批。

## Audit Log Completeness

一句话定义：Audit Log Completeness 是检查任务提交、数据访问、镜像使用、checkpoint、模型下载、权限变更、配额变更、审批和日志查看是否都有 actor、action、resource、decision、reason 和 request id。

## Audit Integrity Retention

一句话定义：Audit Integrity Retention 是要求审计日志防篡改、可查询、有留存周期，并能支撑合规、事故响应和复盘。

## Incident Response Readiness

一句话定义：Incident Response Readiness 是准备冻结任务、撤销密钥、隔离产物、查询审计、通知 owner、修复策略和复盘的安全响应能力。

## Training Platform Security Gate

一句话定义：Training Platform Security Gate 是判断训练平台是否达到身份可信、权限最小化、数据和产物不越权、运行时隔离、密钥安全、网络受控、审计完整和事故可响应最低要求的综合门禁。

典型硬阻断：所有任务共享高权限身份；RBAC 通配；数据用途未授权；restricted 数据训练产物被降级；高优先级资源无审批；镜像无 digest / 签名 / 扫描；privileged 或 hostPath 放开；密钥写入镜像、代码或日志；默认允许外联；日志无脱敏和访问审计；模型权重可随意下载；高风险动作无审批；审计日志缺 actor / action / resource；审计可被篡改或留存不足；没有密钥撤销和产物隔离预案；没有最终安全门禁。

## Training Platform System Design Audit

一句话定义：Training Platform System Design Audit 是把“大模型训练平台怎么设计”这类系统设计题，拆成需求、TrainingJob 契约、状态机、准入、调度、launcher、血缘、checkpoint、观测、容错、安全、成本、自助、接口、trade-off 和最终门禁的可审计回答方法。

核心直觉：训练平台系统设计不是说“用 Kubernetes 跑 Job”，也不是把 Portal、Scheduler、Logger、Registry 这些名词排成架构图。高分回答要证明训练任务从提交到模型产物注册的每个阶段都有清楚输入、输出、状态、指标、失败处理和权限边界。Kubernetes Job、Kueue、Kubeflow Trainer、`torchrun`、OpenTelemetry 和 MLflow 等工具可以是实现选项，但面试中更重要的是抽象边界和生命周期闭环。

常见审计维度：Requirement Clarification 看是否问清训练类型、集群规模、多租户、敏感数据、抢占、恢复、发布和成本预算；TrainingJob Contract 看 owner、project、image digest、code commit、dataset version、final config、resources、distributed、checkpoint、observability、security 和 queue 是否完整；Lifecycle State Machine 看 submitted、validated、queued、scheduled、starting、running、checkpointing、retrying、failed、completed 和 cancelled 是否覆盖且迁移合法；Admission Quota Policy、Scheduler Resource Fit 和 Distributed Launcher Fit 检查准入、配额、fair share、gang scheduling、拓扑感知、rank、world size、rendezvous、GPU binding 和 NCCL；Data Config Lineage、Checkpoint Recovery Coupling、Observability Experiment Linkage 和 Fault Recovery Flow 检查可复现、可恢复、可排障和可重试；Security Audit Governance、Cost Capacity Governance、Developer Self Service、Interface Artifact Contract、Tradeoff Boundary Reasoning 和 Training Platform Design Gate 则检查治理、体验、契约、取舍和最终上线讨论门槛。

常见公式：训练平台系统设计样本可以写成

```math
p_i=(r_i,j_i,s_i,a_i,q_i,l_i,d_i,c_i,o_i,f_i,g_i,k_i,u_i,m_i,b_i,t_i)
```

统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(p_i)=1]
```

world size 和 effective batch 可以写成

```math
W=n_{\mathrm{node}}n_{\mathrm{proc}},\qquad B_{\mathrm{eff}}=B_{\mathrm{micro}}A_{\mathrm{grad}}W
```

P95 排队时间可以写成

```math
T_{\mathrm{queue,p95}}=Q_{0.95}\left(\{t_{\mathrm{start},i}-t_{\mathrm{submit},i}\}_{i=1}^{N}\right)
```

最大丢失 GPU-hours 可以写成

```math
L_{\mathrm{lost}}=G\frac{t_{\mathrm{fail}}-t_{\mathrm{ckpt}}}{60}
```

trace 覆盖率可以写成

```math
R_{\mathrm{trace}}=\frac{N_{\mathrm{linked}}}{N_{\mathrm{required}}}
```

```math
G_{\mathrm{design}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land T_{\mathrm{queue,p95}}\le B_{\mathrm{queue}} \land L_{\mathrm{lost}}\le B_{\mathrm{loss}} \land R_{\mathrm{trace}}\ge \rho_{\mathrm{trace}} \land P_0=0\right]
```

面试表达：训练平台系统设计题可以按“需求澄清 -> TrainingJob 契约 -> 控制面准入 -> 调度与资源 -> launcher -> 数据与配置血缘 -> checkpoint 与容错 -> 可观测性与实验追踪 -> artifact registry -> 安全审计 -> 成本容量 -> 开发者体验 -> trade-off 与门禁”展开。不要把 Kubernetes 当完整答案，也不要漏掉数据、checkpoint、实验、模型产物、安全和成本。

## Requirement Clarification

一句话定义：Requirement Clarification 是训练平台系统设计回答的需求澄清层，明确 workload scope、集群规模、多租户、数据敏感度、抢占、恢复 SLO、artifact 发布和成本预算。

## TrainingJob Contract

一句话定义：TrainingJob Contract 是训练平台的核心对象契约，冻结一次训练任务的 owner、project、image digest、code commit、dataset version、final config、resources、distributed、checkpoint、observability、security 和 queue。

## Lifecycle State Machine

一句话定义：Lifecycle State Machine 是把训练任务从 submitted 到 completed / failed / cancelled 的状态和合法迁移显式化，避免平台只剩不透明的 running / failed。

## Admission Quota Policy

一句话定义：Admission Quota Policy 是在任务入队前完成 schema、镜像、数据权限、资源配额、优先级审批、幂等和 dry run 准入校验。

## Scheduler Resource Fit

一句话定义：Scheduler Resource Fit 是检查 queue、quota、fair share、gang scheduling、topology-aware placement、backfilling、preemption 和 fragmentation 治理是否匹配训练资源需求。

## Distributed Launcher Fit

一句话定义：Distributed Launcher Fit 是检查 launcher adapter 是否把调度结果正确转成 rank、local rank、world size、rendezvous、GPU binding 和 NCCL 配置。

## Data Config Lineage

一句话定义：Data Config Lineage 是绑定镜像 digest、代码 commit、dataset version、manifest、final config、环境版本、seed 和 experiment run，回答模型怎么训出来。

## Checkpoint Recovery Coupling

一句话定义：Checkpoint Recovery Coupling 是让 checkpoint 策略和调度、抢占、容错、恢复演练、manifest、checksum、最大 stale 时间相互绑定。

## Observability Experiment Linkage

一句话定义：Observability Experiment Linkage 是把 logs、metrics、events、traces、rank 维度、dashboard、alerts 和 experiment run 连成同一条排障证据链。

## Fault Recovery Flow

一句话定义：Fault Recovery Flow 是用故障分类、retry budget、backoff、checkpoint resume、不可重试快速失败和诊断报告替代盲目自动重试。

## Security Audit Governance

一句话定义：Security Audit Governance 是把身份、RBAC / ABAC、数据用途、运行时隔离、镜像供应链、审计日志和事件响应纳入训练平台设计。

## Cost Capacity Governance

一句话定义：Cost Capacity Governance 是按 owner、project、job、queue 归因 GPU-hours、存储、网络和利用率，并提供预算告警和容量反馈。

## Developer Self Service

一句话定义：Developer Self Service 是让用户能通过 Portal、CLI、SDK、模板、状态页、错误报告和从 checkpoint 重跑自助完成常见训练流程。

## Interface Artifact Contract

一句话定义：Interface Artifact Contract 是显式定义 API、scheduler handoff、launcher handoff、data contract、artifact registry 和 audit event schema，避免模块之间只靠口头约定。

## Tradeoff Boundary Reasoning

一句话定义：Tradeoff Boundary Reasoning 是在系统设计回答中讲清 Kubernetes vs training operator、公平性 vs 利用率、拓扑 vs 排队、抢占 vs 丢失进度、自助 vs 护栏、成本 vs 可靠性。

## Training Platform Design Gate

一句话定义：Training Platform Design Gate 是判断训练平台系统设计是否达到可运行、可恢复、可复现、可治理、可排障、可审计和可面试深挖最低要求的综合门禁。

典型硬阻断：需求假设不清；TrainingJob 契约缺关键字段；状态机只有 running / failed；准入绕过配额和安全策略；分布式训练没有 gang scheduling；launcher 的 world size 和资源不一致；数据和配置无法复现；checkpoint 未做恢复演练；只看 stdout 或 loss 曲线；失败全部盲目重试；安全审计缺失；成本无人归因；用户只能找平台同学查状态；artifact registry 和实验记录断链；只背工具名没有 trade-off；没有最终设计门禁。

## AI Infra Overview Audit

一句话定义：AI Infra Overview Audit 是把 AI Infra 总览从“模块清单”升级为可检查指标表的审计方法。

核心直觉：AI Infra 不是单独的 Kubernetes、Slurm、GPU 采购、监控工具或实验平台，而是把计算、网络、存储、调度、训练、推理、数据、artifact、评估、可观测性、安全、成本和开发者体验连成可运行、可治理、可复现的基础设施底座。

关键检查项：Compute Accelerator Readiness 看 GPU/加速器、驱动、拓扑和资源暴露是否就绪；Network Communication Readiness 看多机通信、拓扑、RDMA、AllReduce 和跨节点瓶颈；Storage Data Checkpoint Readiness 看数据供给、checkpoint、缓存和恢复；Scheduler Resource Governance 看队列、配额、优先级、抢占和 gang scheduling；Training Platform Reproducibility 看代码、数据、镜像、超参、日志和 checkpoint 是否可复现；Inference Platform SLO Readiness 看 TTFT、TPOT、吞吐、KV cache、限流、灰度和回滚；Data Platform Lineage Quality 看数据来源、版本、权限、质量和污染隔离；Model Artifact Registry Governance 看权重、tokenizer、adapter、量化版本、eval report 和发布状态；Eval Experiment Tracking Coverage 看离线评估、人评、A/B、实验参数和 artifact 追踪；Observability Signal Coverage 看 metrics、logs、traces、events 和 profiles；Security Governance Coverage 看身份、权限、密钥、租户隔离、镜像和审计；Cost Capacity Governance 看 GPU、存储、网络、推理 token、容量和闲置资源；Developer Self-Service Readiness 看 Portal、CLI、SDK、模板和自助排障；AI Infra Boundary Clarity 看是否避免把 AI Infra 简化成某个工具；MLOps LLMOps Platform Boundary Clarity 看生命周期管理、大模型应用运维和平台工程边界；Algorithm Infra Collaboration Readiness 看算法团队和平台团队是否能用指标、SLO、成本和失败样例协作。

常见公式：统一覆盖率可以写成

```math
C_j=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[g_j(x_i)=1]
```

其中，`N` 是审计样本数，`g_j(x_i)=1` 表示样本通过第 `j` 个检查。

面试表达：AI Infra 总览题不要只背“GPU、网络、存储、调度、训练平台、推理平台”这些词，而要说明每一层解决什么瓶颈、如何被观测、如何被治理，以及算法岗为什么需要理解这些边界。

## AI Infra Overview Gate

一句话定义：AI Infra Overview Gate 是判断一个 AI Infra 总览方案是否达到上线讨论门槛的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{ai\_infra}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{p0}}=0\right]
```

其中，`\tau_j` 是每个维度最低通过阈值，`R_p0` 是 P0 级硬阻断数量。

典型硬阻断：GPU 资源有但网络或存储无法支撑训练；推理平台没有 SLO、限流和回滚；模型权重和 tokenizer 没有 artifact 版本；实验无法复现；数据没有 lineage；没有 metrics / logs / traces；跨租户权限和审计缺失；成本和容量没有归因；算法团队只能提工单，无法自助查看任务、指标和失败原因。

常见误区：把 AI Infra 等同于 Kubernetes、等同于买 GPU、等同于 MLOps 平台、等同于监控 dashboard，都会漏掉跨层边界和治理问题。

## AI Infra MLOps LLMOps Boundary Audit

一句话定义：AI Infra MLOps LLMOps Boundary Audit 是把 AI Infra、MLOps、LLMOps、Platform Engineering、Data Platform、Model Platform、DevOps 和 SRE 的职责边界转成可检查表的方法。

核心直觉：名字不是关键，主责层、协作层、交付物、接口和门禁才是关键。GPU 队列排队、prompt 变更回归、模型版本回滚、RAG 文档过期、推理 SLO 下降和开发者无法自助提交任务，虽然都可能发生在同一个 AI 平台里，但主责层不同。

常见维度：AI Infra Scope Accuracy 看资源、调度、训练和推理运行时是否被归到 AI Infra；MLOps Lifecycle Accuracy 看数据版本、训练 pipeline、实验追踪、模型注册、部署和回滚是否被归到模型生命周期；LLMOps Application Accuracy 看 prompt、RAG、Agent、工具调用、模型路由、LLM eval、安全和 token 成本是否被归到大模型应用生命周期；Platform Engineering DX Accuracy 看 Portal、CLI、SDK、模板、自助流程和 golden path 是否被当成内部开发者平台能力；DevOps SRE Boundary Accuracy 看 CI/CD、SLO、错误预算、告警和事故响应是否和 AI 工作负载可靠性正确连接；Data Platform Boundary Accuracy 看数据湖、数据质量、血缘、权限和 RAG 文档新鲜度；Model Platform Boundary Accuracy 看权重、tokenizer、adapter、量化版本、评估报告和发布状态；Primary Owner Clarity、Interface Contract Coverage、Artifact Lineage Handoff、Observability SLO Handoff、Security Cost Governance Handoff、Lifecycle Stage Mapping、Incident Routing Accuracy 和 Collaboration Handoff Readiness 则检查跨团队协作是否可落地。

典型公式：边界路由错误率可以写成

```math
R_{\mathrm{route}}=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}[h_i\ne y_i]
```

其中，`y_i` 是样本应该归属的主责层，`h_i` 是当前回答或系统设计给出的主责层。

## Boundary Gate

一句话定义：Boundary Gate 是判断 AI 平台职责边界是否足够清楚、可交接、可审计的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{boundary}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{route}}=0 \land P_0=0\right]
```

其中，`C_j` 是第 `j` 个边界维度覆盖率，`\tau_j` 是最低阈值，`R_route` 是主责层路由错误率，`P_0` 是 P0 级边界风险数量。

常见硬阻断：上线事故没人主责；模型权重、数据版本、prompt 版本和评估报告无法串起来；监控有曲线但没有 SLO；平台能力只有工单没有自助接口；安全和成本治理只写在文档里；RAG 质量问题被误判为 GPU 问题；MLOps 被说成 AI Infra，LLMOps 被说成 prompt 管理工具，Platform Engineering 被说成 DevOps 改名。

面试表达：边界题不要追求把每个工具永久归类，而要先讲“这个问题主要是谁主责”，再讲“需要哪些协作层”，最后讲“交付物、接口、指标和门禁是什么”。

## Accelerator Selection Audit

一句话定义：Accelerator Selection Audit 是把 GPU、NPU、TPU 或其他 AI 加速器选型从“看峰值算力”升级为容量、带宽、互联、软件生态、成本和风险的可检查表。

核心直觉：大模型硬件选型要同时看工作负载、模型规模、batch、context length、训练或推理 runtime、显存容量、显存带宽、多卡互联、低精度支持、kernel / 通信库 / profiler、云上或自建约束、供给和迁移风险。峰值 TFLOPS 只回答“理论上能算多快”，不能回答“模型能不能放下、decode 会不会被带宽卡住、多机训练会不会被通信卡住、团队能不能调优和排障”。

常见维度：Peak Compute Fit 看峰值算力和低精度矩阵单元是否匹配目标吞吐；Memory Capacity Fit 看权重、梯度、optimizer state、activation、临时 buffer 和 KV cache 是否放得下；Memory Bandwidth Fit 看 decode、embedding、optimizer update 和低 arithmetic intensity 算子是否被 HBM 带宽限制；Interconnect Bandwidth Fit 看张量并行、流水并行、数据并行和 MoE 通信是否匹配 NVLink、PCIe、InfiniBand 或 Pod 拓扑；Low Precision Support 看 FP16、BF16、FP8、INT8 等精度是否被硬件和框架稳定支持；Software Stack Maturity、Kernel Library Readiness、Distributed Communication Readiness 看 CUDA / ROCm / TPU runtime / Neuron 等栈、算子库、通信库和 profiler 是否能覆盖目标模型；Training Memory Budget 和 Inference KV Cache Budget 分别检查训练状态显存和长上下文推理 KV cache；Workload Hardware Fit、Cloud Self Build Decision、Cost Power Capacity Awareness、Fallback Portability Plan 和 Selection Risk Governance 则检查场景、成本、电力、供应、厂商锁定和迁移预案。

常见公式：训练状态显存的粗略下限可以写成

```math
M_{\mathrm{train}}\approx N(B_{\mathrm{param}}+B_{\mathrm{grad}}+B_{\mathrm{opt}})+M_{\mathrm{act}}+M_{\mathrm{tmp}}
```

推理 KV cache 显存可以写成

```math
M_{\mathrm{kv}}=2LBSH_{\mathrm{kv}}D_hB_{\mathrm{elem}}
```

其中，`N` 是参数量，`L` 是层数，`B` 是 batch 或并发序列数，`S` 是序列长度，`H_kv` 是 KV head 数，`D_h` 是 head dimension，`B_elem` 是每个元素字节数。

面试表达：GPU、NPU、TPU 选型题不要只背厂商和型号，而要把训练显存、推理 KV cache、roofline 带宽上限、通信暴露比例、软件生态、成本和迁移风险讲成同一套审计口径。

## Accelerator Selection Gate

一句话定义：Accelerator Selection Gate 是判断一个加速器选型方案是否达到工程落地讨论门槛的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{accelerator}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land P_0=0\right]
```

其中，`C_j` 是第 `j` 个加速器选型维度覆盖率，`\tau_j` 是最低阈值，`P_0` 是 P0 级硬件选型风险数量。

典型硬阻断：只看峰值 TFLOPS 而不算训练状态显存；权重能放下但 AdamW 状态放不下；长上下文推理没有估算 KV cache；decode 场景忽略 HBM 带宽；多机训练忽略互联和通信库；选择新硬件但缺 kernel、profiler、框架和排障经验；云上 / 自建只看单价不看利用率、电力、供给、迁移和厂商锁定。

面试表达：高分回答要能解释“为什么这类工作负载适合这类硬件”，再用容量估算、带宽瓶颈、通信瓶颈、软件栈和风险门禁证明方案可落地。

## Bandwidth Bottleneck Audit

一句话定义：Bandwidth Bottleneck Audit 是把显存、HBM、PCIe、NVLink、NVSwitch、机间网络、存储和 checkpoint I/O 统一放进性能排查表的审计方法。

核心直觉：大模型性能不是只看峰值算力。GPU 计算单元需要从 HBM 读取权重、activation、KV cache 和临时张量；CPU、GPU、网卡和存储之间会走 PCIe 或相关 peer 路径；单机多卡依赖 NVLink / NVSwitch 或 PCIe 拓扑；多机训练依赖 InfiniBand / RoCE、网卡位置、NCCL 路由和网络拥塞。任何一层数据搬运跟不上，GPU 都可能处于“很忙但不高效”或“等数据”的状态。

常见维度：VRAM Capacity Accounting 看权重、梯度、optimizer state、activation、KV cache、临时 buffer 和通信 buffer 是否都进入预算；HBM Bandwidth Model 看算子是否 memory-bound；PCIe Transfer Awareness 看 CPU-GPU 拷贝、offload、网卡和存储路径是否被量化；NVLink Topology Awareness 和 NVSwitch All-to-All Awareness 看单机 GPU 互联是否真实可用、是否均匀；Inter-Node Network Awareness 看机间网络、RDMA、拥塞和拓扑；KV Cache Growth Accounting、Training State Memory Accounting 和 Communication Volume Accounting 分别检查推理 KV cache、训练状态显存和 collective 通信量；Topology-Aware Parallel Group 看 tensor parallel、pipeline parallel、data parallel group 是否按拓扑分组；Dataloader Storage IO Awareness、Checkpoint IO Awareness 和 Offload Penalty Awareness 看数据供给、checkpoint 阻塞和 offload 惩罚；Overlap Fusion Optimization 看通信计算重叠和 kernel fusion；Observability Metric Coverage 看 HBM、PCIe、NVLink、网络、存储、NCCL 和 step time 分解是否可观测。

常见公式：显存总预算可以写成

```math
M_{\mathrm{vram}}=M_{\mathrm{weight}}+M_{\mathrm{grad}}+M_{\mathrm{opt}}+M_{\mathrm{act}}+M_{\mathrm{kv}}+M_{\mathrm{tmp}}+M_{\mathrm{comm}}
```

有效带宽可以写成

```math
B_{\mathrm{eff}}=\frac{D}{T}
```

其中，`D` 是实际搬运数据量，`T` 是耗时。多卡通信是否暴露，可以看 `R_comm`；PCIe / offload 是否拖慢 step，可以看 `R_pcie`。

面试表达：带宽瓶颈题要先分层：HBM、PCIe、GPU 间互联、机间网络、存储 / checkpoint 和软件 kernel；再用 step time 分解、通信占比、有效带宽、拓扑和失败样例定位，而不是只看 GPU utilization。

## Bandwidth Bottleneck Gate

一句话定义：Bandwidth Bottleneck Gate 是判断一个训练或推理系统是否已经把主要数据搬运链路纳入容量、性能和观测门禁的综合门禁。

门禁形式可以写成：

```math
G_{\mathrm{bandwidth}}=\mathbf{1}\left[\min_j C_j\ge \tau_j \land R_{\mathrm{comm}}\le \rho_{\mathrm{comm}} \land R_{\mathrm{pcie}}\le \rho_{\mathrm{pcie}} \land P_0=0\right]
```

其中，`C_j` 是第 `j` 个带宽审计维度覆盖率，`\tau_j` 是最低阈值，`R_comm` 是通信暴露比例，`R_pcie` 是 PCIe / offload 暴露比例，`P_0` 是 P0 级带宽风险数量。

典型硬阻断：没有完整显存预算；只看权重不看 KV cache；decode 被 HBM 带宽卡住但只调 batch；CPU-GPU 拷贝阻塞 step；offload 让 PCIe 成为主瓶颈；把 tensor parallel group 跨慢链路切分；以为 8 卡机器默认都有 NVSwitch；多机训练没有看网卡拓扑、NCCL 路由和网络拥塞；checkpoint 写入阻塞训练；没有 HBM / PCIe / NVLink / 网络 / 存储的指标。

面试表达：带宽门禁的价值是把“慢”拆成可测的链路。先量化容量、有效带宽、通信暴露和 I/O 暴露，再决定用 kernel fusion、量化、拓扑分组、通信重叠、并行策略调整、数据预取或 checkpoint 异步化。

## 分布式训练排障常见问题

一句话定义：分布式训练排障是定位多卡训练中显存、通信、数值、数据和性能异常的过程。

常见显存问题：batch 过大、序列过长、激活未 checkpoint、ZeRO/FSDP wrap 不合理、gradient accumulation 配置错误。

常见通信问题：NCCL 卡住、跨节点网络慢、rank 配置错误、某些进程提前退出、collective 调用顺序不一致。

常见数值问题：FP16 overflow、loss scaling 不合适、学习率随 global batch 改变后未调整、梯度裁剪缺失。

常见性能问题：pipeline bubble 大、tensor parallel 通信过重、数据加载跟不上、checkpoint 保存阻塞训练、padding 浪费严重。

面试表达：分布式训练排障要先判断是 OOM、hang、loss 异常还是吞吐低，再分别沿显存、通信、数值和数据链路排查。

## 本章小结

本章围绕大模型分布式训练系统展开。核心问题只有四个：模型能不能放下，训练能不能跑快，通信能不能承受，训练结果能不能稳定复现。

高频结论如下：

1. 数据并行拆 batch，不拆模型；张量并行拆层内矩阵；流水并行拆模型层。
2. DDP 的核心是每卡完整模型副本，并通过 all-reduce 同步梯度。
3. all-reduce 让所有设备获得完整聚合结果，reduce-scatter 保留聚合分片，all-gather 收集完整张量。
4. 张量并行适合超大矩阵，但层内通信频繁，对高速互联依赖强。
5. 流水并行通过 micro-batch 填充多 stage，主要问题是 pipeline bubble 和负载均衡。
6. ZeRO 通过切 optimizer state、gradient 和 parameter 消除数据并行冗余。
7. ZeRO-3 和 FSDP 显存节省大，但需要频繁 all-gather 和 reduce-scatter。
8. activation checkpointing 用重算换显存，是大模型训练常用手段。
9. gradient accumulation 用多个 micro-batch 模拟大 global batch，但不直接提高训练速度。
10. mixed precision 用低精度加速和省显存，但 FP16 需要关注 loss scaling 和数值稳定。
11. MFU 比 GPU utilization 更能衡量大模型训练效率。
12. 分布式训练排障要分清显存、通信、数值、数据和调度问题。
