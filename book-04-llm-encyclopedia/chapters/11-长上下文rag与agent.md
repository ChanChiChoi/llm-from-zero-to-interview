# K. 长上下文、RAG 与 Agent

条目：Long Context、Context Window、Position Extrapolation、RAG、Dense Retrieval、Reranker、Vector Database、Agent、Tool Use、Function Calling、ReAct、Memory。

## Long Context

一句话定义：long context 是指模型能够处理远超普通对话长度的输入，例如几十万 token 甚至百万 token 的文档、代码库、会议记录或多轮任务轨迹。

为什么提出：真实业务中，用户经常不是问一句孤立问题，而是希望模型理解整份合同、整篇论文、完整工单历史、多个文件或一段长时间交互。传统上下文窗口太短，会导致模型只能看到局部信息，无法做跨段落、跨文件、跨时间的综合判断。

解决什么问题：长上下文主要解决信息无法完整放入模型输入的问题。它让模型可以直接读取更多原始材料，减少人工摘要、切块检索和上下文拼接带来的信息丢失。

核心挑战：注意力计算复杂度高、显存消耗大、位置编码外推困难、长距离依赖难学、有效信息容易被海量无关文本稀释，以及模型可能出现 lost in the middle，也就是中间位置的信息利用率低。

典型方法：扩大训练长度、改造位置编码、使用稀疏注意力或线性注意力、引入 sliding window attention、使用 chunked prefill、KV cache 分页管理，以及用 RAG 先筛选再放入上下文。

优点：实现直观，用户体验好，适合长文档阅读、长代码理解、长会议总结、法律审查和多轮交互记忆。

缺点：成本高，延迟高，输入越长越容易引入噪声；窗口变长不代表模型一定能稳定利用远距离信息。

面试表达：长上下文解决的是“看得见更多”的问题，但不等于“用得好更多”。面试中要强调上下文长度、有效注意力、位置外推、检索增强和成本之间的权衡。

## Context Window

一句话定义：context window 是模型一次推理可接收的最大 token 数，包括用户输入、系统提示、历史对话、工具结果和模型已生成内容。

为什么重要：模型不会天然记住当前窗口外的信息。如果关键信息没有放进 context window，模型就无法基于它回答，只能依赖参数记忆或猜测。

组成部分：系统提示占一部分，用户问题占一部分，检索证据占一部分，历史对话占一部分，模型输出也会消耗窗口。实际工程中，窗口经常被 prompt 模板、工具返回和冗余历史快速占满。

常见策略：历史摘要、滑动窗口、重要消息保留、按需检索、证据压缩、去重、模板精简，以及把结构化状态外置到数据库中。

工程坑：窗口很大时不能简单把所有材料塞进去。无关信息会干扰模型，长输入会增加费用和延迟，重复内容还可能让模型过度相信某些错误信息。

面试表达：context window 是推理时的信息带宽。窗口越大，信息容量越高，但成本、延迟和噪声管理压力也越高。

## Position Extrapolation

一句话定义：position extrapolation 是让模型在超过训练时最大长度的位置上仍能理解 token 顺序和相对距离的能力。

为什么需要：很多模型预训练时只见过固定长度，例如 4K、8K 或 32K。如果直接在 128K 上推理，位置编码可能落到训练分布之外，导致注意力模式不稳定。

常见背景：Transformer 本身没有天然顺序感，需要位置编码提供位置信息。绝对位置编码、相对位置编码、RoPE、ALiBi 等方案对外推能力的影响不同。

典型方法：RoPE scaling、NTK-aware scaling、YaRN、位置插值、继续长上下文训练，以及在长序列数据上做专门微调。

优点：可以在不完全重训模型的情况下扩展上下文长度，工程成本相对可控。

局限：位置外推只解决位置表示问题，不保证模型学会长距离推理。很多模型即使能接收长输入，也可能只关注开头和结尾。

面试表达：position extrapolation 是长上下文的底层条件之一，但不是全部。长上下文能力还依赖训练数据、注意力机制、推理系统和评估设计。

## Lost in the Middle

一句话定义：lost in the middle 是指模型更容易利用上下文开头和结尾的信息，而对中间位置的关键信息利用不足。

为什么出现：长上下文中注意力分布、训练数据模式、位置偏置和 prompt 结构都会影响模型对不同位置内容的关注。很多任务中答案线索如果被放在中间，准确率会下降。

解决思路：把关键证据放在靠近问题的位置，对检索结果做 rerank，把最重要证据放在前后显著位置，要求模型先列证据再回答，或使用分段阅读和聚合策略。

评估方法：构造 needle-in-a-haystack 测试，把同一条关键信息插入不同位置，观察模型召回和回答是否稳定。

面试表达：长窗口不等于长上下文理解。lost in the middle 说明我们要评估“有效上下文长度”，而不是只看模型宣称支持多少 token。

## RAG

一句话定义：RAG 是 Retrieval-Augmented Generation，即先从外部知识库检索相关信息，再把检索结果交给大模型生成答案。

为什么提出：大模型参数知识可能过时、不完整、不可追溯，且容易幻觉。RAG 把外部知识库接入生成过程，让模型基于可更新、可引用的证据回答。

解决什么问题：RAG 主要解决知识更新、私有知识问答、事实依据、长文档问答和降低幻觉的问题。它常用于企业知识库、客服、法律、医疗、金融研报和代码问答。

基本流程：文档清洗，切分 chunk，向量化，写入向量库；用户提问后生成 query embedding；召回候选 chunk；可选 rerank；把证据拼入 prompt；模型基于证据回答；最后做引用和质量评估。

核心难点：检索不到、检索到但排序错、chunk 切分不合理、证据互相冲突、prompt 塞入过多噪声、模型不忠实于证据、引用和结论不匹配。

优点：知识可更新，可接入私有数据，答案可追溯，训练成本低于重新预训练或微调。

缺点：系统链路更长，延迟更高，依赖检索质量；如果检索错了，生成模型可能基于错误证据给出更有迷惑性的答案。

面试表达：RAG 不是简单的“向量库加大模型”，而是一条检索、排序、上下文构造、生成、引用校验和评估的完整链路。RAG 的上限经常由检索和证据组织决定。

## Chunking

一句话定义：chunking 是把长文档切成适合检索和放入上下文的小片段。

为什么需要：文档通常很长，不能整体作为一个向量单元。切块后可以更细粒度召回相关内容，也便于控制 prompt 长度。

常见方法：固定长度切分、按段落切分、按标题层级切分、按语义边界切分、滑动窗口重叠切分，以及针对代码、表格、FAQ 的结构化切分。

关键参数：chunk size、overlap、元数据、标题继承、文档层级、是否保留表格结构和引用位置。

常见坑：chunk 太小会丢上下文，chunk 太大召回不精准；overlap 太小可能切断答案，overlap 太大则增加重复和成本。

面试表达：chunking 决定 RAG 的基本检索粒度。好的切分不是追求固定长度，而是尽量让一个 chunk 表达完整语义单元。

## Dense Retrieval

一句话定义：dense retrieval 是把 query 和文档都编码成稠密向量，并用向量相似度召回相关内容的检索方法。

为什么提出：传统关键词检索依赖词面匹配，难以处理同义表达、改写问题和语义相关内容。稠密检索可以根据语义相似性召回文本。

工作流程：用 embedding model 把文档 chunk 编码成向量并建索引；查询时把用户问题编码成向量；用 cosine similarity、dot product 或 L2 距离寻找最近邻。

优点：语义匹配能力强，适合自然语言问答和改写问题。

缺点：对数字、专有名词、代码符号、精确字段和否定条件可能不如关键词检索稳定；embedding 模型如果领域不匹配，召回效果会下降。

常见改进：使用领域 embedding、query rewrite、多路召回、hybrid retrieval、hard negative 训练和 reranker 重排。

面试表达：dense retrieval 提升语义召回，但不能完全替代 BM25。真实 RAG 系统常用 dense retrieval 加 sparse retrieval 的混合召回。

## Sparse Retrieval

一句话定义：sparse retrieval 是基于关键词、倒排索引和词项权重的检索方法，典型代表是 BM25。

为什么仍然重要：很多业务问题依赖精确匹配，例如错误码、合同条款编号、函数名、产品型号、人名、日期和金额。向量检索可能把语义相近但细节不一致的内容召回。

优点：可解释性强，精确匹配好，速度快，工程成熟。

缺点：对同义词、自然语言改写和跨语言语义匹配能力有限。

面试表达：RAG 检索不是只看向量。关键词检索在精确字段和高置信召回上仍然很关键。

## Hybrid Retrieval

一句话定义：hybrid retrieval 是把 dense retrieval 和 sparse retrieval 结合起来，提高召回覆盖率和稳定性。

常见做法：并行使用向量召回和 BM25 召回，再做分数融合、候选合并、去重和 rerank。

适用场景：企业知识库、代码搜索、日志问答、法律条款、产品手册等既需要语义理解又需要精确匹配的场景。

优点：兼顾语义召回和关键词精确匹配。

缺点：系统复杂度更高，分数归一化、候选合并和权重调参更麻烦。

面试表达：hybrid retrieval 的核心不是简单拼两个结果，而是用多路召回提升候选覆盖，再用 reranker 统一排序。

## Reranker

一句话定义：reranker 是对初步召回的候选文档重新排序的模型或规则模块。

为什么需要：第一阶段检索追求快和召回率，可能返回很多弱相关或只表面相关的 chunk。reranker 用更精细的 query-document 交互判断相关性，把真正有用的证据排到前面。

常见形式：cross-encoder reranker、LLM reranker、规则重排、基于元数据和时间的新鲜度重排。

优点：显著改善 top-k 证据质量，降低无关上下文进入 prompt 的概率。

缺点：增加延迟和计算成本；如果候选召回阶段没有召回正确文档，reranker 也无法补救。

面试表达：retriever 决定候选池，reranker 决定最终证据顺序。RAG 质量差时，要区分是召回问题还是排序问题。

## Vector Database

一句话定义：vector database 是存储、索引和检索向量的数据库，常用于 RAG 中的 embedding 检索。

核心能力：向量写入、近似最近邻搜索、metadata filter、分片、持久化、增量更新、删除、权限隔离和混合检索。

常见索引：HNSW、IVF、PQ 等。它们通过近似搜索在速度、内存和召回率之间做权衡。

工程关注点：索引构建时间、查询延迟、召回率、更新频率、多租户隔离、权限过滤、embedding 版本管理和数据删除一致性。

常见误区：向量库不是 RAG 的全部。即使向量库性能很好，embedding、切块、rerank、prompt 和评估做不好，系统仍然会失败。

面试表达：vector database 解决的是大规模向量近邻搜索和管理问题，不直接保证问答正确性。

## Query Rewrite

一句话定义：query rewrite 是在检索前改写用户问题，让查询更适合召回目标文档。

为什么需要：用户问题可能口语化、信息缺失、含多轮指代或包含多个子问题。直接拿原问题检索可能召回不到正确文档。

常见方式：补全上下文指代、生成多个改写 query、拆解子问题、生成关键词 query、把自然语言转成结构化过滤条件。

风险：改写可能引入用户没有表达的意图，导致检索偏离原问题。因此需要保留原 query，并对改写结果做约束。

面试表达：query rewrite 是提升 RAG 召回率的重要手段，但必须避免语义漂移。

## Grounding

一句话定义：grounding 是让模型输出建立在给定证据、工具结果或外部事实上的过程。

为什么需要：没有 grounding 的回答容易依赖参数记忆和语言流畅性，产生无法验证的幻觉。

在 RAG 中的体现：模型应优先根据检索证据回答，不能把证据没有支持的内容说成事实；最好给出引用并说明答案来自哪些片段。

常见检查：答案中的关键断言是否能被证据支持，引用位置是否正确，是否混合了证据外知识，是否对缺失信息做了不确定性表达。

面试表达：RAG 的目标不是让答案更长，而是让答案可追溯、可验证、忠实于证据。

## Agent

一句话定义：agent 是能够围绕目标进行规划、调用工具、观察结果并迭代行动的大模型系统。

为什么提出：普通大模型一次生成只适合回答或简单任务。复杂任务需要查资料、调用 API、写代码、运行测试、分析结果和多步修正，因此需要模型具备行动循环。

基本循环：接收目标，形成计划，选择工具，调用工具，观察返回，更新状态，继续行动，直到完成任务或达到停止条件。

关键能力：任务分解、工具选择、参数生成、状态跟踪、错误恢复、结果验证和终止判断。

适用场景：数据分析、代码修改、自动化运维、信息检索、办公流程、客服工单、软件测试和多系统协同。

主要风险：工具误用、无限循环、状态污染、错误累积、越权调用、成本失控和难以评估。

面试表达：agent 不是单独一个模型，而是模型、工具、记忆、规划、执行和安全控制组成的系统。

## Tool Use

一句话定义：tool use 是模型根据任务需要调用外部工具，例如搜索、数据库、计算器、代码执行器、文件系统或业务 API。

为什么需要：模型擅长语言理解和生成，但不擅长精确计算、实时信息、私有数据库查询和实际执行动作。工具弥补模型能力边界。

工作方式：系统向模型描述可用工具、参数 schema 和使用约束；模型判断是否需要工具；生成工具调用；系统执行工具；模型读取结果并继续回答。

优点：增强实时性、准确性和可执行性。

风险：参数生成错误、工具返回误读、调用顺序错误、权限控制不足、把工具失败结果当成真实结果。

面试表达：tool use 的关键不是“能调用工具”，而是能否在正确时机调用正确工具，并正确解释工具结果。

## Function Calling

一句话定义：function calling 是让模型按预定义函数 schema 生成结构化调用参数的一种工具调用机制。

为什么提出：自然语言调用不稳定，容易格式错误。function calling 通过 schema 约束输出字段、类型和必填项，提高工具调用可靠性。

典型字段：函数名称、参数名、参数类型、枚举值、描述、必填字段和返回格式。

优点：结构化强，便于系统解析和执行，适合 API 调用、数据库查询、订单操作和工作流自动化。

局限：schema 不能保证语义正确。模型可能生成格式合法但业务含义错误的参数，因此仍需要权限校验、参数校验和执行前确认。

面试表达：function calling 解决的是工具调用的结构化接口问题，不等于解决工具使用决策和业务安全问题。

## ReAct

一句话定义：ReAct 是 Reasoning and Acting 的缩写，让模型在推理和行动之间交替进行。

基本形式：模型先思考当前问题，决定下一步动作，调用工具，观察结果，再继续思考和行动。

为什么有效：很多任务不能一次回答，需要根据外部观察不断修正。ReAct 把推理链和工具调用结合起来，使模型能处理动态任务。

优点：过程可解释，适合检索、问答、网页操作和多步工具任务。

缺点：容易冗长、成本高，错误步骤会传播；如果没有停止条件，可能陷入循环。

面试表达：ReAct 的核心是把“想”和“做”闭环，而不是只让模型输出一段 chain-of-thought。

## Planning

一句话定义：planning 是 agent 在执行前或执行中把复杂目标拆成子任务和行动顺序的能力。

为什么需要：复杂任务往往不能靠一步工具调用完成。模型需要决定先查什么、再算什么、何时验证、失败后如何回退。

常见策略：一次性计划、边执行边计划、任务树、检查清单、反思修正和基于约束的工作流编排。

工程取舍：自由规划灵活但不可控，固定工作流稳定但泛化差。生产系统常把关键流程工作流化，把开放问题交给模型处理。

面试表达：planning 的价值在于降低复杂任务的混乱度，但需要配合状态管理、工具约束和结果验证。

## Memory

一句话定义：memory 是 agent 或对话系统保存、检索和使用历史信息的机制。

类型：短期记忆通常指当前 context window 内的对话和状态；长期记忆通常指外部数据库中保存的用户偏好、历史任务、项目知识或环境状态。

为什么需要：多轮任务中，模型需要记住用户目标、已经尝试过的方法、工具返回结果和长期偏好。仅靠上下文窗口会受长度限制。

实现方式：对话摘要、事件日志、向量记忆、结构化状态表、用户画像、任务检查点和文件化工作记录。

风险：记忆过期、错误记忆污染、隐私泄露、权限混淆和把历史偏好过度泛化到新任务。

面试表达：memory 不是把所有历史塞进 prompt，而是对历史进行筛选、压缩、结构化和按需检索。

## Agent Evaluation

一句话定义：agent evaluation 是评估 agent 是否能稳定完成多步任务的过程。

为什么难：agent 的结果不仅取决于最终回答，还取决于工具选择、调用参数、执行顺序、错误恢复、成本、耗时和安全性。

评估维度：任务成功率、步骤正确率、工具调用准确率、无效调用率、平均成本、平均步数、失败恢复能力、权限合规性和用户满意度。

常见方法：构造端到端任务集，记录 action trace，做人工审查，使用 sandbox 验证执行结果，比较不同 agent 策略的成功率和成本。

面试表达：评估 agent 不能只看最终文本，要看执行轨迹是否正确、可控、低成本且安全。

## 长上下文、RAG 与 Agent 的关系

长上下文解决的是一次输入能看多少信息，RAG 解决的是从外部知识中找哪些信息放进上下文，Agent 解决的是模型如何围绕目标多步行动。

三者经常组合使用：RAG 为 agent 提供外部知识，agent 调用检索、数据库和业务工具，长上下文保存任务轨迹和关键证据。真实系统不是三选一，而是根据任务需求组合。

典型对比：如果材料固定且必须完整审阅，长上下文更直接；如果知识库很大且需要动态查询，RAG 更经济；如果任务需要多步操作和外部执行，Agent 更合适。

工程设计原则：先明确任务类型，再决定是扩大窗口、做检索增强，还是引入 agent。不要因为 agent 概念热门，就把简单问答做成复杂多步系统。

面试表达：长上下文、RAG 和 Agent 分别对应信息容量、知识获取和行动能力。优秀的大模型应用设计通常是在三者之间做成本、准确率、延迟和可控性的权衡。

## 本章小结

本章围绕长上下文、RAG 和 Agent 建立了系统词汇表。

核心结论如下：

1. 长上下文提升模型可见信息量，但不保证模型能稳定使用全部信息。
2. context window 是推理时的信息带宽，需要管理成本、噪声和关键证据位置。
3. position extrapolation 解决超长位置表示问题，但不等于完整长文本理解能力。
4. RAG 的关键是检索、重排、上下文构造、grounding 和评估，不只是向量数据库。
5. dense retrieval、sparse retrieval 和 hybrid retrieval 各有优势，真实系统常组合使用。
6. reranker 可以显著改善进入 prompt 的证据质量。
7. agent 是模型、工具、记忆、规划和安全控制组成的系统，而不是单次回答。
8. tool use 和 function calling 提升模型外部执行能力，但必须配合参数校验和权限控制。
9. memory 要筛选、压缩、结构化和按需检索，不能无脑保存全部历史。
10. 长上下文、RAG 和 Agent 分别解决信息容量、知识获取和多步行动问题。
