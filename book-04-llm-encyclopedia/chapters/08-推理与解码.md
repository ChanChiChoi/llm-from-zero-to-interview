# H. 推理与解码

条目：Inference、Autoregressive Generation、Decoding、Greedy Decoding、Sampling、Temperature、Top-k Sampling、Top-p Sampling、Beam Search、Repetition Penalty、Presence Penalty、Frequency Penalty、Stop Token、EOS Token、Max New Tokens、Best-of-N、Self-Consistency、Log Probability、Token Probability、KV Cache、Prefill、Decode、Time to First Token / TTFT、Time per Output Token / TPOT、Speculative Decoding。

## Inference

一句话定义：inference 是使用训练好的模型，根据输入上下文生成输出的过程。

在 LLM 中：推理通常不是一次性生成完整答案，而是自回归地逐 token 生成。模型先读取 prompt，输出下一个 token 的概率分布，再根据解码策略选出 token，追加到上下文中继续生成。

基本流程：tokenizer 把文本转成 token ids；模型 forward 得到 logits；解码策略从 logits 中选择下一个 token；把新 token 拼回输入；重复直到遇到停止条件。

与训练的区别：训练时使用 teacher forcing，可以并行计算一段序列中所有位置的 loss；推理时下一个 token 依赖上一个已生成 token，因此输出长度维度存在串行依赖。

主要指标：首 token 延迟、每 token 延迟、吞吐、显存占用、生成质量、稳定性和成本。算法岗面试中，既要讲清模型侧机制，也要理解系统侧约束。

面试表达：LLM 推理的核心是自回归生成。它看似只是 forward，但真实系统中还涉及 tokenizer、chat template、KV Cache、解码策略、停止条件和延迟吞吐权衡。

## Autoregressive Generation

一句话定义：autoregressive generation 是模型每次生成一个 token，并把该 token 作为后续生成上下文的一部分。

核心公式：

```text
P(y_1, ..., y_T | x) = ∏_t P(y_t | x, y_{1:t-1})
```

其中 `x` 是 prompt，`y_t` 是第 `t` 个输出 token。

为什么串行：第 `t+1` 个 token 的分布依赖第 `t` 个 token。如果第 `t` 个 token 还没确定，就无法准确计算下一步上下文。

影响：生成长答案时，延迟会随输出 token 数增加；早期生成错误会污染后续上下文；解码策略会显著影响最终答案。

面试表达：自回归生成让 LLM 能开放式输出，但也带来推理串行、错误累积和长输出成本高的问题。

## Decoding

一句话定义：decoding 是从模型输出的下一个 token 概率分布中选择 token 的策略。

输入是什么：模型 forward 后得到词表维度 logits。经过 temperature、top-k、top-p、重复惩罚等处理后，可以得到一个候选 token 分布。

为什么重要：同一个模型、同一个 prompt，使用不同 decoding 参数可能得到完全不同的输出。解码直接影响稳定性、多样性、创造性、重复率、事实性和格式遵循。

常见目标：问答和代码生成通常希望稳定、准确，可以用较低 temperature；创意写作希望多样，可以提高 sampling 随机性；分类和结构化抽取通常希望确定性，可以使用 greedy 或低温采样。

面试表达：decoding 不改变模型参数，但会改变模型行为。调生成效果时，除了换模型和改 prompt，也要检查 decoding 参数。

## Greedy Decoding

一句话定义：greedy decoding 是每一步都选择概率最高的 token。

核心规则：

```text
y_t = argmax_i P(token_i | context)
```

优点：确定、简单、速度快、复现性好。相同模型、相同输入、相同环境下通常输出一致。

缺点：容易陷入局部最优，输出可能保守、死板、重复；在开放式写作中缺少多样性；在需要探索多条推理路径的问题上可能错过正确答案。

适用场景：分类、信息抽取、格式转换、代码补全、低随机性问答、评测基线。

面试表达：greedy 的优点是稳定，缺点是没有探索。它每一步局部最优，不保证整段序列全局最优。

## Sampling

一句话定义：sampling 是按照下一个 token 概率分布随机抽取 token 的解码方式。

核心直觉：模型不是永远选概率最高的 token，而是让高概率 token 更容易被选中，低概率 token 也有一定机会出现。

优点：多样性更强，适合开放式对话、创意写作、头脑风暴和需要多个候选答案的场景。

缺点：随机性更高，可能采到低质量 token，事实性和格式稳定性可能下降。temperature、top-k、top-p 设置不当时，幻觉和跑题风险会上升。

面试表达：sampling 用随机性换多样性。生成任务越开放，sampling 越有价值；任务越强调准确和稳定，随机性越需要收敛。

## Temperature

一句话定义：temperature 是缩放 logits、控制概率分布尖锐程度的参数。

常见形式：

```text
softmax(logits / temperature)
```

低温效果：temperature 小于 1 时，概率分布更尖锐，高概率 token 更突出，输出更稳定、更确定。

高温效果：temperature 大于 1 时，概率分布更平滑，更多 token 有机会被采样，输出更多样，但也更容易不稳定。

特殊情况：temperature 接近 0 时行为接近 greedy decoding。实际系统中通常会对温度取值做边界限制，避免数值异常。

面试表达：temperature 控制生成随机性。低温适合准确任务，高温适合创意任务，但高温会增加幻觉和格式错误风险。

## Top-k Sampling

一句话定义：top-k sampling 是只保留概率最高的 `k` 个 token，再在这些 token 中重新归一化采样。

核心作用：截断长尾低概率 token，避免模型采到明显不合理的候选。

例子：如果 `k=50`，每一步只在概率最高的 50 个 token 中采样，其他 token 概率置零。

优点：简单直接，能减少低质量尾部 token。缺点是候选集合大小固定，不管当前分布是否已经很确定，都保留同样数量的候选。

面试表达：top-k 用固定候选数量控制采样范围，适合限制低概率噪声，但不如 top-p 自适应。

## Top-p Sampling

一句话定义：top-p sampling 又称 nucleus sampling，是保留累计概率达到 `p` 的最小 token 集合，再从中采样。

核心直觉：当模型很确定时，只需要少量 token 就能覆盖大部分概率；当模型不确定时，候选集合会自动变大。

例子：如果 `p=0.9`，先按概率从高到低排序，取累计概率达到 90% 的最小集合，再重新归一化采样。

优点：候选集合大小自适应，通常比 top-k 更灵活。缺点是 `p` 太高会保留过多长尾 token，太低会让输出过于保守。

面试表达：top-p 控制的是概率质量，而不是固定 token 数。它会随模型不确定性动态调整候选集合。

## Beam Search

一句话定义：beam search 是维护多个候选序列，每一步保留累计得分最高的若干条路径。

核心参数：beam size。beam size 越大，搜索路径越多，计算成本越高。

适用场景：机器翻译、摘要、语音识别等目标较明确、答案空间较受限的任务。

在开放式对话中的问题：beam search 常生成平庸、模板化、低多样性的答案，也可能偏向短句或高频表达。聊天模型通常更常用 sampling 或 greedy。

面试表达：beam search 是序列级近似搜索，不是简单逐步贪心。但在开放式 LLM 对话中，它不一定比采样更好。

## Repetition Penalty

一句话定义：repetition penalty 是降低已出现 token 再次被生成概率的机制，用于缓解重复输出。

为什么需要：自回归模型可能陷入重复循环，例如反复生成同一句话、同一段格式或同一个词。

作用方式：对上下文中已经出现过的 token 调整 logits，使它们再次出现的概率降低。

风险：过强的重复惩罚会伤害正常表达。代码、数学公式、列表编号、专有名词和引用中本来就需要重复 token。

面试表达：重复惩罚可以缓解复读，但不是越强越好。它解决的是生成策略问题，不是模型能力本身的根治。

## Presence Penalty

一句话定义：presence penalty 是只要 token 出现过，就对其再次出现施加惩罚。

核心特点：它关注“是否出现过”，不太关心出现次数。

适用场景：鼓励模型引入新话题、新词汇或减少重复短语。

风险：过强会让模型回避必要重复，导致术语不一致或表达绕远。

面试表达：presence penalty 更像鼓励新内容的开关，适合提升多样性，但会牺牲一致性。

## Frequency Penalty

一句话定义：frequency penalty 是 token 出现次数越多，惩罚越强的机制。

与 presence penalty 区别：presence penalty 看是否出现过；frequency penalty 看出现了多少次。

适用场景：减少长文本中的高频复读，让模型避免重复同一表达。

风险：对代码、结构化文本、法律条款、数学符号这类需要重复的场景不友好。

面试表达：frequency penalty 用出现次数控制重复程度，比 presence penalty 更强调频率累积。

## Stop Token

一句话定义：stop token 是触发生成结束的 token 或字符串。

常见形式：EOS token、聊天模板中的 assistant 结束标记、用户指定 stop words、工具调用结束标记。

为什么重要：停止条件配置错误会导致回答被截断、停不下来、泄漏角色标记、继续生成下一轮 user 内容，或输出多余模板符号。

工程注意：不同模型的 EOS、BOS、PAD、role token 可能不同。推理框架、tokenizer 和 chat template 必须一致。

面试表达：stop 配置是 LLM 推理的基础工程细节。很多生成异常不是模型本身差，而是 EOS 或 stop words 配置错。

## EOS Token

一句话定义：EOS token 是 end-of-sequence token，用来表示序列结束。

训练作用：预训练或 SFT 中，EOS 告诉模型一个文档、回答或对话轮次在何处结束。

推理作用：当模型生成 EOS 时，系统通常停止继续生成。

常见问题：如果 SFT 数据没有正确加入 EOS，模型可能学不会停止；如果 EOS 被错误 mask 或 tokenizer 不一致，模型可能输出异常结束行为。

面试表达：EOS 不只是特殊符号，而是模型学习“何时结束”的训练信号。

## Max New Tokens

一句话定义：max new tokens 是限制模型最多生成多少个新 token 的参数。

作用：防止模型无限生成，控制延迟、成本和输出长度。

与 context length 区别：context length 限制输入加输出的总 token 容量；max new tokens 只限制本次生成的新 token 数量。

风险：设太小会截断答案，设太大会增加延迟和成本，也可能让模型生成冗余内容。

面试表达：max new tokens 是推理成本和答案完整性的显式控制参数，不等同于模型上下文窗口。

## Best-of-N

一句话定义：best-of-N 是生成 N 个候选答案，再用规则、reward model、评审模型或人工方式选择最佳答案的方法。

为什么有效：采样可以探索不同解答路径，reranker 可以从多个候选中选出质量更高的答案。

适用场景：数学推理、代码生成、复杂问答、创意生成和高价值请求。

成本：推理成本约随候选数增加。N 越大，质量潜力越高，但延迟和算力成本也越高。

风险：如果选择器本身有偏差，best-of-N 会放大选择器偏好；如果 reward model 有漏洞，也可能引入 reward hacking。

面试表达：best-of-N 用计算换质量，本质是“多生成再筛选”。它适合高价值任务，不适合所有请求默认开启。

## Self-Consistency

一句话定义：self-consistency 是对推理问题采样多个解题路径，再选择多数一致答案或最可信答案的方法。

为什么有效：复杂推理中，单条链路可能偶然出错。多次采样可以让模型探索不同路径，如果多个路径得到同一答案，该答案更可能可靠。

适用场景：数学题、逻辑题、多跳问答、程序推理和需要中间步骤的问题。

局限：如果模型系统性误解题意，多数投票也会一致错误；如果答案形式复杂，聚合和判等也不容易。

面试表达：self-consistency 用多路径采样提升推理稳健性，但代价是更多推理成本，并且需要可靠的答案聚合方式。

## Log Probability

一句话定义：log probability 是 token 或序列概率的对数形式。

为什么使用：多个 token 的概率相乘容易数值下溢，取 log 后可以把乘法变成加法，便于比较序列分数。

序列分数：一个回答的 log probability 通常是各 token log probability 的和，必要时会做长度归一化。

用途：候选排序、困惑度计算、DPO loss、beam search、置信度分析和拒答策略。

注意：高 log probability 不等于答案事实正确。它只表示模型认为该序列在当前上下文中概率高。

面试表达：logprob 是模型自评分数的重要接口，但它衡量的是模型概率，不是客观真实性。

## Token Probability

一句话定义：token probability 是模型在当前上下文下对某个下一个 token 分配的概率。

来源：logits 经 softmax 之后得到概率分布。

解释限制：token 概率是局部的下一个 token 置信度，不等同于整段回答置信度。一个答案可能每个 token 都很流畅，但事实仍然错误。

工程用途：可视化模型不确定性、做 constrained decoding、构建分类器、分析模型偏好和调试异常输出。

面试表达：token probability 能解释模型下一步倾向，但不能直接当作事实置信度。

## KV Cache

一句话定义：KV Cache 是缓存自回归推理中每一层历史 token 的 Key 和 Value，避免每生成一个 token 都重复计算全部历史注意力。

为什么需要：没有 KV Cache 时，生成第 `t` 个 token 需要重新计算前面所有 token 的 K/V，成本很高。有了 cache，新 token 只需要计算自己的 Q/K/V，并与历史 K/V 做 attention。

缓存内容：每一层 attention 的 key 和 value，通常形状与 batch、head 数、序列长度、head dim 有关。序列越长、层数越多、batch 越大，KV Cache 显存占用越高。

收益：显著加速 decode 阶段，让每步生成不必重复跑完整 prompt。

代价：占用大量显存，并带来显存带宽访问压力。长上下文和高并发场景中，KV Cache 往往是推理系统的关键瓶颈。

面试表达：KV Cache 用空间换时间，是 LLM 自回归推理加速的基础。它加速 decode，但也让显存管理变得重要。

## Prefill

一句话定义：prefill 是推理时处理完整 prompt 并建立初始 KV Cache 的阶段。

做什么：模型并行处理输入 prompt 的所有 token，计算每层 hidden states，并为这些 prompt token 写入 KV Cache。

特点：prefill 能在序列维度并行，因此算力利用率相对高；长 prompt 会显著增加 prefill 成本。

影响指标：prefill 主要影响首 token 延迟，也就是 TTFT。用户输入越长、检索文档越多、system prompt 越复杂，首 token 通常越慢。

面试表达：prefill 是“读 prompt”的阶段，成本主要和输入长度相关，决定用户多久看到第一个 token。

## Decode

一句话定义：decode 是在 KV Cache 基础上逐 token 生成输出的阶段。

做什么：每一步只输入最新 token，读取历史 KV Cache，计算下一个 token 分布，选出 token 后更新 cache。

特点：输出 token 之间串行依赖，无法像训练一样在生成长度维度完全并行。decode 通常更受显存带宽和 KV Cache 访问影响。

影响指标：decode 主要影响每 token 延迟和总生成时间。输出越长，总 decode 成本越高。

面试表达：decode 是“写答案”的阶段，成本主要和输出长度、batch、KV Cache 和模型大小相关。

## Time to First Token / TTFT

一句话定义：TTFT 是从请求进入系统到生成第一个 token 的时间。

组成：排队时间、调度时间、tokenization、prompt 构造、prefill 计算和首步 decode。

为什么重要：TTFT 决定用户是否感觉系统“反应快”。即使总生成时间较长，只要首 token 快，流式输出体验也会更好。

优化方向：减少 prompt 长度、缓存公共前缀、优化调度、降低排队、提升 prefill 吞吐、使用更快模型或更好硬件。

面试表达：TTFT 主要受输入侧和 prefill 影响，是交互体验中的关键首响指标。

## Time per Output Token / TPOT

一句话定义：TPOT 是 decode 阶段平均生成每个输出 token 所需时间。

影响因素：模型大小、batch size、KV Cache 长度、显存带宽、并发数、解码策略和推理框架实现。

与吞吐的关系：系统可能通过 batching 提高总体吞吐，但单请求 TPOT 可能上升。线上服务经常需要在吞吐和延迟之间平衡。

面试表达：TPOT 衡量生成速度。长回答场景下，TPOT 往往比 TTFT 更决定总耗时。

## Speculative Decoding

一句话定义：speculative decoding 是用小模型或草稿模型先快速生成多个候选 token，再由大模型并行验证，从而加速自回归推理的方法。

核心动机：大模型逐 token decode 成本高，但很多 token 较容易预测。让小模型先猜一段，大模型一次验证多个 token，可以减少大模型调用轮数。

基本流程：draft model 生成若干候选 token；target model 对这些 token 进行验证；通过接受-拒绝机制保留被大模型认可的 token；被拒绝处再由大模型采样或继续生成。

关键要求：最终采样分布应尽量保持与原 target model 一致，不能为了速度明显改变输出质量。

适用条件：draft model 足够快，并且与 target model 分布足够接近。若草稿模型猜得很差，接受率低，加速收益会下降。

面试表达：speculative decoding 用“小模型猜、大模型验”减少大模型串行 decode 次数，是推理加速方法，不是训练方法。

## 本章小结

本章围绕 LLM 推理和解码展开。推理的主线是：prompt 经 prefill 建立 KV Cache，随后模型在 decode 阶段逐 token 生成；decoding 策略决定每一步如何从概率分布中选择 token。

高频结论如下：

1. LLM 推理通常是自回归逐 token 生成，训练可并行，推理输出长度维度串行。
2. greedy 稳定但缺少探索，sampling 多样但风险更高。
3. temperature 控制概率分布尖锐程度，top-k 控制候选数量，top-p 控制累计概率质量。
4. beam search 适合目标明确的序列任务，但开放式对话中不一定好。
5. repetition、presence、frequency penalty 可以缓解重复，但过强会伤害正常表达。
6. EOS、stop token 和 max new tokens 决定生成何时停止，是常见工程坑。
7. best-of-N 和 self-consistency 用更多计算换更高质量或更稳健的答案。
8. KV Cache 用空间换时间，是 decode 加速基础，但会增加显存压力。
9. prefill 主要影响 TTFT，decode 主要影响 TPOT 和总生成时间。
