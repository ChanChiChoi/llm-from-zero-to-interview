# 主流大模型架构创新时间线

## 本章定位

学习 Transformer 架构演进时，不能只按模块学习：RoPE、RMSNorm、GQA、MoE、MLA、长上下文、稀疏注意力、推理优化。

真实的大模型演进更像一条时间线：一个模型提出某个设计，另一个模型吸收其中一部分，再结合自己的训练数据、系统约束和产品目标继续改造。

所以本章按时间维度梳理主流大模型的架构创新，帮助读者理解：

```text
这些模型不是孤立出现的，而是在不断借鉴、取舍和工程化。
```

重点关注：

1. GPT 系列如何奠定 decoder-only + scaling 主线。
2. PaLM、Chinchilla、LLaMA 如何改变训练配比和开源架构范式。
3. Mistral、Mixtral 如何推动 GQA、滑窗注意力和开源 MoE。
4. Gemini、Claude、Qwen、DeepSeek 等模型如何在多模态、长上下文、MoE、MLA、推理强化上继续演进。
5. DeepSeek 系列中的 MLA、DeepSeekMoE、MTP、GRPO 等设计为什么重要。
6. 面试中如何从时间线讲“主流大模型如何一点点升级”。

说明：公开资料会持续变化，很多闭源模型不会完整披露架构细节。本章对公开论文、技术报告中明确披露的信息会直接描述；对社区传闻、未正式公开或命名不稳定的实现细节会标注为“待核验”，不把它当作确定事实。

## 1. 总体演进主线

先给出一个简化时间线。

| 时间 | 代表模型 | 关键创新 | 对后续模型的影响 |
|---|---|---|---|
| 2017 | Transformer | Self-Attention、Encoder-Decoder、并行训练 | 奠定大模型架构基础 |
| 2018-2019 | GPT / GPT-2 / BERT | Decoder-only 自回归、Encoder-only 预训练 | 形成生成式 LLM 和理解式 PLM 两条路线 |
| 2020 | GPT-3 | 大规模 decoder-only、in-context learning | 证明 scaling 后的 few-shot 能力 |
| 2021-2022 | Gopher / Chinchilla / PaLM | 训练 token 与参数配比、Pathways、规模化训练 | 推动 compute-optimal training 思路 |
| 2022-2023 | InstructGPT / ChatGPT | SFT + RLHF、PPO、对话对齐 | 让 base model 变成 assistant model |
| 2023 | LLaMA | RoPE、RMSNorm、SwiGLU、开源高效 decoder-only | 成为开源 LLM 架构基线 |
| 2023 | Falcon / MPT / Baichuan / ChatGLM | MQA/GQA、长上下文、本地化训练 | 推动开源生态多样化 |
| 2023 | Mistral 7B | GQA、Sliding Window Attention、强小模型 | 证明结构和数据效率可弥补参数规模 |
| 2023 | Mixtral 8x7B | Sparse MoE、开源专家模型 | 推动 MoE 从研究走向实用开源 |
| 2023-2024 | Gemini / GPT-4V / Claude 3 | 多模态、长上下文、工具和安全系统 | 模型从文本生成器走向通用助手 |
| 2024 | LLaMA 3 / Qwen2 / Yi / Gemma | 更强 tokenizer、GQA、长上下文、多语言 | 开源模型训练工程成熟化 |
| 2024 | DeepSeek-V2 | MLA、DeepSeekMoE、低成本长上下文推理 | 强化 KV cache 压缩和 MoE 成本优化方向 |
| 2024-2025 | DeepSeek-V3 / R1 | MLA、MoE、MTP、GRPO/reasoning RL | 推动低成本训练和 reasoning RL 热潮 |
| 2025+ | 新一代模型 | MoE、MLA、长上下文、推理强化、多模态 Agent | 架构、训练、推理和对齐越来越耦合 |

这条线说明一个趋势：

```text
早期重点是模型变大。
中期重点是训练更高效、推理更便宜。
后期重点是长上下文、多模态、MoE、推理能力和系统级优化。
```

## 2. 2017：Transformer 奠定基础

Transformer 的关键创新是 Self-Attention。

它解决了 RNN/CNN 在长距离依赖和并行训练上的问题。

核心特点：

1. 用 attention 让任意两个 token 直接交互。
2. 训练时可以并行处理整个序列。
3. 使用多头注意力学习不同子空间关系。
4. 使用 FFN 提供 token-wise 非线性变换。
5. 用残差和 LayerNorm 提高训练稳定性。

它最早是 encoder-decoder 架构，用于机器翻译。

但后来大模型主流逐渐转向 decoder-only。

原因是 decoder-only 更适合统一成一个目标：

```text
next-token prediction
```

这个目标简单、可扩展、数据来源广，并且天然适合生成。

## 3. 2018-2020：GPT 系列确立 decoder-only scaling 主线

GPT 系列的重要性不只在于规模，而在于它把生成式语言模型的训练目标和架构路线固定下来。

### 3.1 GPT-1 / GPT-2

GPT-1 和 GPT-2 证明：decoder-only Transformer 可以通过大规模无监督文本预训练获得强语言能力。

核心特点：

1. Causal self-attention。
2. 自回归 next-token prediction。
3. 预训练 + 下游任务适配。
4. 生成能力随数据和参数规模提升。

### 3.2 GPT-3

GPT-3 的关键影响是 in-context learning。

它说明当模型规模足够大时，不一定需要每个任务都微调，可以通过 prompt 和 few-shot examples 激发能力。

GPT-3 之后，大模型研究开始高度关注：

```text
规模、数据、上下文学习、prompt、涌现能力
```

它也让 decoder-only 成为通用 LLM 的默认路线。

## 4. 2021-2022：Chinchilla、PaLM 与 compute-optimal 训练

早期 scaling 往往强调参数变大。

Chinchilla 之后，一个重要观念变得清晰：

```text
在固定计算预算下，模型参数和训练 token 数要匹配。
```

如果模型参数很大但训练 token 不够，模型可能 undertrained。

这影响了后来的开源模型训练策略：

1. 不盲目堆参数。
2. 更重视高质量 token 数。
3. 更重视数据清洗和数据配比。
4. 用较小模型训练更多 token，也可能很强。

PaLM 则展示了大规模训练系统和 Pathways 思路。

这一阶段的核心不是某一个 attention 改造，而是训练范式变化：

```text
模型能力 = 架构 × 参数 × 数据 × 训练 token × 系统效率
```

## 5. 2022-2023：InstructGPT、ChatGPT 与对齐训练

Base model 不等于 assistant。

InstructGPT 和 ChatGPT 的关键贡献是把后训练流程推到主流视野：

```text
预训练 base model
-> SFT
-> Reward Model
-> PPO-RLHF
-> 对话助手
```

这一阶段的架构变化不一定体现在 Transformer block 内部，而是体现在“模型行为塑造”上。

它让大家意识到：

1. 预训练决定基础能力。
2. SFT 决定指令遵循和回答格式。
3. RLHF 决定偏好、安全和人类可用性。
4. 评估和红队决定是否能上线。

后续 DPO、GRPO、DAPO 等方法都可以看作对这个后训练体系的简化、替代或增强。

## 6. 2023：LLaMA 把现代开源 LLM 架构标准化

LLaMA 对开源生态影响很大。

它不是第一个 decoder-only 模型，但它把一组现代 LLM 常用组件组合成了开源基线。

典型设计包括：

1. Decoder-only Transformer。
2. RoPE 位置编码。
3. RMSNorm。
4. SwiGLU。
5. 更高质量训练数据和更长训练 token。

这些设计后来被大量开源模型吸收。

### 6.1 RoPE

RoPE 把位置信息编码到 Q/K 旋转中，适合 decoder-only attention，并且后续可以配合 NTK scaling、YaRN 等方法扩展上下文。

### 6.2 RMSNorm

RMSNorm 相比 LayerNorm 更简单，省去均值中心化，常用于提高训练效率和稳定性。

### 6.3 SwiGLU

SwiGLU 是 gated FFN 变体，通常比普通 ReLU/GELU FFN 表达能力更强。

LLaMA 之后，很多模型可以被描述为：

```text
LLaMA-like decoder-only + RoPE + RMSNorm + SwiGLU + GQA/MQA + 更好数据
```

## 7. 2023：Mistral 推动小模型高效架构

Mistral 7B 的影响在于：小模型也可以通过架构和数据做到很强。

它的典型设计包括：

1. Grouped-Query Attention。
2. Sliding Window Attention。
3. 更高效的推理和长上下文处理。

### 7.1 GQA

GQA 介于 MHA 和 MQA 之间。

```text
MHA：每个 head 有自己的 K/V，效果好但 KV cache 大。
MQA：多个 Q head 共享一组 K/V，KV cache 小但可能损失表达。
GQA：多个 Q head 分组共享 K/V，在效果和效率之间折中。
```

GQA 后来成为很多大模型的标准配置，因为推理阶段 KV cache 成本非常关键。

### 7.2 Sliding Window Attention

滑窗注意力让每个 token 主要关注局部窗口，降低长上下文成本。

它牺牲一部分全局注意力能力，换取更高效率。

这类设计后来影响了长上下文模型的 attention pattern 设计。

## 8. 2023：Mixtral 推动开源 MoE 实用化

Mixtral 8x7B 是开源 MoE 模型的重要代表。

MoE 的核心思想是：

```text
模型总参数很多，但每个 token 只激活一部分专家。
```

例如 top-2 routing：

```text
每个 token 选择两个专家参与计算。
```

优点：

1. 增大总参数容量。
2. 每个 token 的计算量可控。
3. 训练和推理可以在容量与成本之间折中。

缺点：

1. routing 复杂。
2. 专家负载均衡困难。
3. 分布式通信成本高。
4. serving 比 dense 模型更复杂。

Mixtral 之后，开源社区对 MoE 的兴趣明显上升。

DeepSeekMoE、Qwen-MoE 等路线都可以放在这个大趋势下理解。

## 9. 2023-2024：Gemini、GPT-4V、Claude 走向多模态和长上下文

这一阶段的主流闭源模型不再只是文本模型。

它们开始强调：

```text
多模态
长上下文
工具调用
安全对齐
复杂推理
```

### 9.1 多模态

多模态模型要解决图片、音频、视频和文本如何进入统一模型的问题。

常见路线：

```text
vision encoder -> projector -> LLM
```

或者更原生的多模态 token 融合。

### 9.2 长上下文

长上下文带来三类问题：

1. 训练时如何让模型见过长序列。
2. 推理时 KV cache 如何承受。
3. 模型是否真的能使用中间信息，而不是 lost-in-the-middle。

因此后续模型会结合：

```text
RoPE scaling
GQA/MQA/MLA
attention kernel 优化
检索增强
上下文压缩
```

## 10. 2024：Qwen、LLaMA 3、Gemma 等开源模型工程成熟化

2024 年开源模型的特点是：不只追赶架构，也追赶完整训练工程。

常见升级点包括：

1. 更强 tokenizer，改善多语言和代码效率。
2. 更高质量数据清洗和配比。
3. GQA 降低推理 KV cache。
4. 更长上下文。
5. 更完善的 instruction tuning。
6. 更系统的安全评估。
7. 更友好的开源生态和工具链。

Qwen 系列尤其重视多语言、代码、数学、工具调用和长上下文能力。

LLaMA 3 系列强调更大规模高质量训练数据、更强 tokenizer 和开源生态影响力。

Gemma 则体现了小模型、开放权重和高效部署方向。

这一阶段说明：

```text
模型强不强，不只看 block 结构，还看 tokenizer、数据、训练 recipe、后训练和评估。
```

## 11. 2024：DeepSeek-V2 的 MLA 和 DeepSeekMoE

DeepSeek-V2 是架构演进中非常值得关注的模型。

它公开强调了两个重要方向：

```text
MLA：Multi-head Latent Attention
DeepSeekMoE：更细粒度、更低成本的 MoE 设计
```

### 11.1 MLA：压缩 KV cache 的注意力改造

标准 attention 推理时需要缓存每层每个 token 的 K/V。

上下文越长，KV cache 越大。

GQA/MQA 的思路是减少 K/V head 数。

MLA 的思路更进一步：把 K/V 相关信息压缩到 latent 表示中，再在需要时恢复或投影。

直觉上可以理解为：

```text
不要为每个 token 存完整 K/V，而是存一个更紧凑的 latent cache。
```

它解决的问题是：

1. 长上下文推理 KV cache 成本高。
2. MoE 模型 serving 时显存压力大。
3. GQA/MQA 虽然省 cache，但可能有表达折中。

MLA 的价值在于把注意力机制和推理系统成本直接联系起来。

这也是现代架构的重要趋势：

```text
不是只设计训练时好看的结构，而是设计推理时真的便宜的结构。
```

### 11.2 DeepSeekMoE：专家更细粒度，激活更稀疏

DeepSeekMoE 的核心方向是提高 MoE 参数效率。

MoE 的关键问题是：

```text
如何让更多专家提供容量，同时每个 token 只激活少量计算。
```

DeepSeekMoE 强调更细粒度专家和共享专家等设计，以改进专家利用率和知识分工。

可以从三个角度理解：

1. 细粒度专家：把专家拆得更细，让 routing 更灵活。
2. 共享专家：保留一部分通用能力路径，缓解知识碎片化。
3. 稀疏激活：控制每个 token 的实际计算成本。

它延续了 Mixtral 的 MoE 大方向，但更强调专家组织方式和成本效率。

## 12. 2024-2025：DeepSeek-V3/R1 与 reasoning RL

DeepSeek-V3/R1 之后，社区对低成本训练、MoE、MLA 和 reasoning RL 的关注显著上升。

公开信息中，值得关注的点包括：

1. MLA 继续作为高效 attention / KV cache 方向。
2. MoE 用于提高总参数容量和训练效率。
3. MTP 等多 token prediction 思路用于提高训练效率或增强预测信号。
4. GRPO 等 RL 方法用于 reasoning 能力提升。
5. 通过可验证任务推动数学、代码和长链推理能力。

### 12.1 MTP：Multi-Token Prediction

传统语言模型训练每个位置预测下一个 token。

MTP 的直觉是：除了预测下一个 token，也预测更后面的多个 token。

潜在好处包括：

1. 提供更密集训练信号。
2. 促进模型学习更长程的未来规划。
3. 可能改善推理或生成效率。

MTP 不是完全新概念，但在大模型训练 recipe 中重新受到关注。

### 12.2 GRPO 与 reasoning RL

DeepSeek-R1 相关讨论让 GRPO 进入更多人的视野。

它的关键不是“又一个 DPO 变体”，而是 reasoning RL 的代表思路之一。

对于数学和代码任务，可以做：

```text
同一个题目采样多条推理链
用答案验证或单元测试给 reward
在组内比较哪些回答更好
用相对优势更新模型
```

这和离线 DPO 的差别很大。

DPO 主要依赖已有 chosen/rejected 对；GRPO 类方法强调当前模型在线采样和可验证 reward。

### 12.3 DeepSeek V4 与 tid2eid：从专家路由到工程索引

据苏剑林博客《DeepSeek V4的tid2eid是怎么来的？》介绍，`tid2eid` 可以作为理解 DeepSeek V4 一类 MoE 工程改造的一个切入点。

从命名上看：

```text
tid2eid = token id to expert id
```

它关注的是 MoE 中 token 和 expert 之间的映射关系。

普通 MoE 的路由过程可以简单理解为：

```text
每个 token -> router 打分 -> 选择 top-k expert -> 送入对应 expert 计算
```

这个过程在概念上很清楚，但在大规模训练和推理中会带来工程问题：

1. 每个 token 选择的 expert 不同，导致数据需要重排。
2. 不同 expert 负载可能不均衡。
3. 跨卡 dispatch 和 combine 会带来通信成本。
4. serving 时需要快速知道 token 应该发给哪个 expert。
5. 专家并行、token 排序、bucket、all-to-all 通信都会影响吞吐。

`tid2eid` 这类映射的意义在于把“路由结果”变成一个更明确的工程索引：

```text
token index -> expert index
```

有了这个映射，就可以围绕它做 token 重排、专家分组、dispatch、combine 和通信优化。

可以用一个简化例子理解：

```text
token 0 -> expert 3
token 1 -> expert 7
token 2 -> expert 3
token 3 -> expert 1
```

那么 `tid2eid` 可以帮助系统快速构造：

```text
expert 1 receives: token 3
expert 3 receives: token 0, token 2
expert 7 receives: token 1
```

这背后反映的是 MoE 从“算法结构”走向“系统实现”的关键问题。

对于 MoE 模型，架构论文里常说的是：

```text
top-k routing
load balancing
shared expert
fine-grained expert
```

但真正训练和部署时，还必须关心：

```text
token 如何分发到 expert
expert 输出如何还原到原 token 顺序
跨 GPU 通信如何组织
路由索引如何压缩和复用
负载不均衡如何缓解
```

所以 `tid2eid` 的价值不只是一个变量名，而是提醒读者：现代 MoE 创新不只发生在 loss 或 routing 公式里，也发生在 token-expert 映射、通信调度和推理引擎实现里。

写进本书时需要注意边界：这里根据苏剑林博客进行概念性解释，具体到 DeepSeek V4 的完整实现仍应以官方技术报告、源码或可复现实验为准。

## 14. 从时间线看互相借鉴

可以把主流大模型的互相借鉴关系理解为几条线。

### 14.1 Decoder-only 主线

```text
GPT -> GPT-2 -> GPT-3 -> LLaMA -> Mistral/Qwen/DeepSeek 等
```

这条线说明 decoder-only + next-token prediction 是最强主干。

### 14.2 归一化和 FFN 改造线

```text
LayerNorm -> RMSNorm
GELU FFN -> SwiGLU / Gated MLP
Dense FFN -> MoE FFN
```

这条线说明模型内部模块不断朝训练稳定、表达更强、计算更高效演进。

### 14.3 Attention 推理效率线

```text
MHA -> MQA -> GQA -> MLA
```

这条线的核心是 KV cache。

上下文越长，KV cache 越成为瓶颈，所以注意力结构不断向更低缓存成本演进。

### 14.4 长上下文线

```text
RoPE -> NTK scaling / YaRN -> sliding window / sparse attention -> long-context training -> retrieval/memory
```

这条线说明长上下文不是只改位置编码，还涉及训练数据、attention pattern、推理系统和评估。

### 14.5 MoE 线

```text
Switch/GShard -> GLaM -> Mixtral -> DeepSeekMoE -> Qwen-MoE 等
```

MoE 的核心是容量与计算解耦。

但是 MoE 的难点也很明确：routing、负载均衡、通信、serving 和稳定训练。

从工程实现看，还可以继续细分为：

```text
token -> expert routing
tid2eid 映射
expert bucket 构造
all-to-all dispatch
expert compute
combine 回原 token 顺序
```

这说明 MoE 架构演进和 AI Infra、推理框架高度耦合。

### 14.6 后训练与 reasoning RL 线

```text
SFT -> PPO-RLHF -> DPO/IPO/KTO/ORPO -> GRPO/DAPO/GSPO -> verifiable reward RL
```

这条线说明模型能力不只来自预训练架构，也来自后训练目标。

特别是 reasoning model，往往需要在线采样、可验证 reward 和多轮 RL。

## 15. 面试中怎么讲这条时间线

如果面试官问：

```text
你怎么看近几年主流大模型架构的演进？
```

可以这样回答：

```text
我会把演进分成几条主线。第一条是 decoder-only scaling 主线，从 GPT-3 到 LLaMA，证明 next-token prediction 加大规模数据和参数能形成通用能力。第二条是现代 Transformer block 改造，比如 RoPE、RMSNorm、SwiGLU、GQA，这些让训练更稳定、推理更便宜。第三条是长上下文和 KV cache 优化，从 GQA/MQA 到 DeepSeek 的 MLA，本质是在降低长上下文推理成本。第四条是 MoE，从 Switch/GShard 到 Mixtral、DeepSeekMoE，通过稀疏激活扩大总容量。第五条是后训练和 reasoning RL，从 RLHF/DPO 走向 GRPO、verifiable reward，让模型不仅会聊天，还能在数学、代码等任务上通过强化学习提升推理能力。
```

如果继续追问 DeepSeek，可以这样答：

```text
DeepSeek 系列比较值得关注的是把架构效率和训练方法结合起来。DeepSeek-V2 公开强调 MLA 和 DeepSeekMoE，前者针对长上下文 KV cache 成本，后者针对 MoE 参数效率和激活计算。DeepSeek-V3/R1 相关工作又把 MLA、MoE、MTP 和 reasoning RL 放在一起看，说明现代大模型创新已经不是单点模块，而是架构、训练、推理和后训练共同优化。
```

如果被问到 `tid2eid`，可以这样答：

```text
tid2eid 可以理解成 token id 到 expert id 的映射，是 MoE 路由结果的一种工程索引表达。MoE 不只是让 router 选 top-k expert，还要把大量 token 高效 dispatch 到对应 expert，再把输出 combine 回原顺序。tid2eid 这类设计体现了 MoE 从算法结构走向工程实现的关键问题：token 重排、专家分组、all-to-all 通信、负载均衡和 serving 吞吐。具体实现细节需要结合官方报告或源码核验。
```

## 16. 本章小结

本章按时间线梳理了主流大模型架构创新。

核心结论如下：

1. Transformer 奠定了 attention + 并行训练的基础。
2. GPT 系列确立了 decoder-only + next-token prediction + scaling 主线。
3. Chinchilla 改变了大家对参数和训练 token 配比的理解。
4. InstructGPT/ChatGPT 让 SFT + RLHF 成为 assistant model 的关键流程。
5. LLaMA 把 RoPE、RMSNorm、SwiGLU 等现代 LLM 组件标准化到开源生态。
6. Mistral 推动 GQA 和滑窗注意力等高效小模型设计。
7. Mixtral 推动开源 MoE 实用化。
8. Gemini、GPT-4V、Claude 等模型推动多模态、长上下文和系统能力。
9. Qwen、LLaMA 3、Gemma 等模型体现了开源训练工程成熟化。
10. DeepSeek 系列的 MLA、DeepSeekMoE、MTP、GRPO/reasoning RL 展示了架构效率和后训练强化结合的趋势。
11. DeepSeek V4 相关 `tid2eid` 讨论说明 MoE 创新不只在 routing 公式，也在 token-expert 映射、dispatch 和通信优化。
12. 面试中最好按时间线和技术主线讲演进，而不是孤立背模型名。
